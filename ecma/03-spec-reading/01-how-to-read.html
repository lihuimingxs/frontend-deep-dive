怎么读规范 —— 元能力训练

ECMA-262 第 16 版（ES2025）正文超过 800 页。直接打开 262.ecma-international.org/15.0 翻第一页就放弃，是几乎所有工程师的共同经验。但规范不是"文档"，是"形式化伪代码"——一旦掌握它的"语法"（算法步骤怎么读、? / ! 简写什么意思、Let _x_ be 是局部变量声明），800 页规范就变成了一份可以逐行 trace 执行的精确程序。本章训练这套元能力，是 P4-P7 规范精读的入门基础。

⚠️ 跳过本章读 P4-P7 一定会迷路

后续章节会大量出现 "? OrdinaryGet(O, P, Receiver)"、"Let _proto_ be ? O.[[GetPrototypeOf]]()"、"Assert: IsCallable(_F_) is true" 这种规范风格的表达。如果不先读懂 ? / ! / Assert / Let 的含义，看后面就像看天书。本章 30 分钟换 4 阶段不迷路，性价比极高。

阅读时长 ~30 分钟 前置 任意 JavaScript 工程经验 对应 ECMA-262 §5 Notational Conventions

01 速览

三句话定义本章：

  1. 规范是"伪代码 + 形式化文字"。每个抽象操作（ToObject、OrdinaryGet 等）都是一段编了号的算法步骤。把它当"一段需要在脑里执行的代码"读，速度比当文档快 5 倍。
  2. 三个核心简写撑起 80% 的规范文本：?（abrupt completion 直接返回）、!（断言不会 throw）、Assert:（不变量声明，违反是规范 bug）。读懂这三个就能流畅读 §6-§29 大部分章节。
  3. Annex B 是历史包袱集中地。规范主体（§1-§29）是"应该如此"，Annex B 是"历史让它必须如此"——HTML 注释 <!--、String.prototype.substr、__proto__ 访问器都在这。web 浏览器必须实现，非 web 环境（Node CLI、Deno 严格模式）可不实现。

看一段真实规范文本，识别其中的 4 种"语法"元素：

ECMA-262 §10.1.8.1 OrdinaryGet 节选 
// 1. Let _desc_ be ? O.[[GetOwnProperty]](P).
//                  ↑ ?: 如果调用 throw 了，整个 OrdinaryGet 直接 throw（abrupt completion 传递）

// 2. If _desc_ is undefined, then
//    a. Let _parent_ be ? O.[[GetPrototypeOf]]().
//    b. If _parent_ is null, return undefined.
//    c. Return ? _parent_.[[Get]](P, Receiver).

// 3. If IsDataDescriptor(_desc_) is true, return _desc_.[[Value]].

// 4. Assert: IsAccessorDescriptor(_desc_) is true.
//    ↑ Assert: 不变量声明。前面 3 个分支都已 return，到第 4 步必然是 accessor descriptor

// 5. Let _getter_ be _desc_.[[Get]].
// 6. If _getter_ is undefined, return undefined.
// 7. Return ? Call(_getter_, Receiver).

这一段 7 步描述了"沿原型链查找属性"的完整算法，对应你日常写的 obj.x。本章后面会拆解这段的每一行写法。

02 底层逻辑

规范的整体结构

ECMA-262（写作时点 ES2025）目录：

章节 内容
§1-§4 引言、范围、一致性要求、概述（含著名的 §4.3 "originally designed as scripting language"）
§5 Notational Conventions —— 本章主要内容来源。读规范前必读
§6 ECMAScript Data Types and Values —— 8 个 Language Types + 4 个 Specification Types（详见 B2）
§7 Abstract Operations —— ToPrimitive、ToObject、SameValueZero 等核心抽象操作（详见 B3）
§8 Syntax-Directed Operations —— 语法树到行为的映射
§9 Executable Code and Execution Contexts —— Realm/Agent/Job/Environment Record（详见 B5）
§10 Ordinary and Exotic Objects Behaviours —— 11 个 essential internal methods + 7 种 Exotic（详见 B4）
§11-§17 语言层：源文本、词法、表达式、语句、函数与类、对象、正则
§18-§19 错误处理 + 程序结构
§20-§28 内置对象详细规范：Object/Function/Array/String/Number/BigInt/Boolean/Symbol、Math、Date、TypedArray、Map/Set、Reflect、Proxy、Promise、Generator、Iterator
§29 Memory Model（共享内存的 happens-before 规则）—— ES2017 加
Annex A Grammar Summary —— 全语法 BNF 汇总，是规范"索引"
Annex B Additional ECMAScript Features for Web Browsers —— 历史包袱集中地（详细见下文）
Annex C-F strict mode 限制汇总、版本变更日志、新特性总结、宿主分层

新人最快上手姿势：从你当前最关心的章节反查依赖。比如你想搞清楚 obj?.x 怎么求值——查 §13.3.9（Optional Chains）→ 它引用 GetValue → §6.2.5.5 → 引用 GetV → §7.3.3 → 引用 ToObject → §7.1.18。一层层钻下去，每个抽象操作都有完整定义。

Normative vs Informative —— 哪些段必须遵守

规范的每一段属于以下两类之一：

  • Normative（规范性）—— 必须遵守。任何符合 ECMA-262 的实现必须按此实现。
  • Informative（资料性）—— 仅供参考。常见于 NOTE 块、举例、说明性段落。

规范用视觉提示区分两者：

规范文本中的标记示例 
// Normative 段 —— 默认就是。所有正式的算法步骤、文本、定义。
// 例：
// "10.1.8.1 OrdinaryGet ( O, P, Receiver )
//  When the abstract operation OrdinaryGet is called with arguments..."

// Informative 段 —— 用 NOTE 块标记
// 例：
// NOTE: This algorithm is designed to allow ordinary objects to share
//       the same get implementation while letting Proxy objects override it.

// 还有 EDITOR'S NOTE（编辑性注释，不进最终版本）

关键区分：NOTE 块里的解释虽然不是 normative，但常常是读懂算法意图的关键——直接读步骤可能"知其然不知其所以然"，NOTE 给的就是 why。不要因为 informative 就跳过。

Annex B —— 历史包袱集中地 ⚠️

这是 ECMA-262 内最违反"洁癖"的部分。Annex B 含大量从未在规范主体定义、但浏览器必须实现的特性，因为web 上跑了几十亿行代码依赖它们，去掉就破坏向后兼容。规范不得不"把它们写进 Annex 标记为历史"，承认现实。

Annex B 里典型的"历史病灶"：

特性 历史原因
HTML 注释 <!-- 和 --> 1995 年浏览器为兼容老浏览器（不识别 <script>），在脚本里包 HTML 注释。结果 JS 把它们当注释处理也成了事实标准
String.prototype.substr(start, length) JScript 1.0 引入，和 String.prototype.substring(start, end) 功能重叠。规范主体只定义 substring；substr 永远是 Annex B
__proto__ 访问器（getter/setter） SpiderMonkey 早期暴露的"伪属性"，Web 上代码大量使用。ES2015 不得不正式纳入 Annex B
正则 octal 转义 /\012/ 八进制转义在 ES5 strict 已禁，但老正则代码仍需支持
document.all 的 typeof === 'undefined' HTML 规范侧的"反规范"行为：DOM 集合对象 typeof 是 undefined（!）—— 因为 90 年代代码用 typeof document.all !== 'undefined' 检测 IE
String .anchor() / .bold() / .italics() 等 把字符串包成 <a> / <b> / <i> 标签——1995 年的"动态生成 HTML"用法。今天没人用，但规范无法删除
命名函数表达式（NFE）的 binding 溢出 非 strict 下 (function f(){})() 内的 f 在外层也可见，是 IE 老 bug 的延续

Annex B 的实现差异：

  • 浏览器（Chrome / Firefox / Safari / Edge）：必须全部实现 Annex B
  • Node.js：实现大部分 Annex B（继承自 V8）
  • Deno / Bun（默认模式）：可选实现，但实际都跟随 V8/JSC 默认值
  • "严格 ECMAScript 引擎"假想实现（如教学/嵌入式 JS）：可不实现 Annex B，但实测几乎不存在这种实现

工程意义：写跨平台代码（既跑浏览器又跑 Node）时，避免使用 Annex B 特性。比如用 String.prototype.substring 不用 substr，用 Object.getPrototypeOf 不用 __proto__。这样代码理论上能在任何符合 ECMA-262 主体的引擎运行。

抽象操作的命名约定

规范定义了大量内部函数（抽象操作 / Abstract Operations）。它们有命名规则，能从名字看出"是用户可见的还是规范内部的"：

命名风格 含义 例子
NamePascalCase 无前缀 规范内部抽象操作 ToObject、ToPrimitive、OrdinaryGet
Ordinary* 前缀 普通对象的默认实现，Exotic 对象覆盖时回落到这里 OrdinaryGet、OrdinaryDefineOwnProperty
Is* 前缀 谓词，返回 true / false IsCallable、IsConstructor、IsArray
Get* / Set* / Has* 属性查询/修改 Get(O, P)、HasProperty(O, P)
[[InternalSlot]] 对象的内部槽（双方括号） [[Prototype]]、[[Extensible]]、[[Get]]
%Intrinsic% Intrinsic Object 引用（每个 Realm 独立的内置对象） %Array.prototype%、%Object%、%Promise%
@@symbol 或 %Symbol.X% Well-known Symbol @@iterator、@@toPrimitive
用户可见 API（小写命名） 规范层定义的公开 API Array.prototype.map、Object.create

关键区分：Get（无前缀）和 OrdinaryGet（带前缀）不是同一个东西——Get(O, P) 调用 O.[[Get]](P)，而 O.[[Get]] 对普通对象就是 OrdinaryGet，对 Proxy 是 ProxyGet，对 Module Namespace 是 ModuleNamespaceGet。这种"调度层 + 实现层"分离是规范的核心设计模式（B4 详细展开）。

算法步骤的 4 种核心记法

这是本章最关键的一节。读懂下面 4 个记法，规范文本变得透明。

① Let _x_ be ... —— 局部变量声明

规范记法 
// 1. Let _desc_ be ? O.[[GetOwnProperty]](P).

// 等价的伪代码：
const desc = O.[[GetOwnProperty]](P);  // 含义层面
// （但 ? 是关键，见下面 ② ）

下划线包裹的标识符（_x_）是规范层局部变量。读时把它当 let x = ... 即可。be 后面是赋值表达式。

② ? —— Abrupt Completion 直接返回 ⭐

这是规范最核心的简写。理解这一个能省掉一半阅读量。

规范记法 vs 完整展开 
// 规范写：
// 1. Let _desc_ be ? O.[[GetOwnProperty]](P).

// 完整展开（规范的"语法糖"还原后）：
let descCompletion = O.[[GetOwnProperty]](P);
if (descCompletion is abrupt completion) {
  return descCompletion;          // ← 把异常一路传出去
}
let desc = descCompletion.[[Value]];

// 简写为 "? Op()" 之后：调用如果 throw（产生 abrupt completion），
// 当前抽象操作直接 return 这个 abrupt completion，传给调用方。
// 等价于 JS 的 try/throw 沿调用栈传播。

为什么要这个简写？因为规范里几乎每一步都可能 throw。如果每步都展开成"调用 → 检查 abrupt → 提取 [[Value]]"，规范文本会膨胀 3 倍。? 把这个模式压缩成 1 个字符，可读性大增。

③ ! —— 断言不会 throw

规范记法 
// Let _O_ be ! ToObject(_value_).
//             ↑ ! 表示"这一步保证不会 throw"

// 完整展开：
let oCompletion = ToObject(value);
Assert: oCompletion is normal completion;  // 断言！违反是规范 bug
let O = oCompletion.[[Value]];

// 用法场景：调用方已经验证了输入合法，知道这步绝不可能失败。
// 例：刚做完 IsObject(value) 检查，再 ToObject 必然 normal。

区分 ? 和 !：? 是"可能 throw，throw 了就传出去"；! 是"绝对不会 throw，如果 throw 了就是规范 bug"。误用 ! 会导致引擎实现冒抽象操作的失败错误。在自己写 polyfill 时，对应实现也要做同样的断言（用 console.assert 或 dev-only 检查）。

④ Assert: ... —— 不变量声明

规范记法 
// 4. Assert: IsAccessorDescriptor(_desc_) is true.

// 含义：到这一步，IsAccessorDescriptor(desc) 必然为 true。
// 如果不是，就是 ① 调用方传错了 / ② 前面步骤实现错了 / ③ 规范 bug 之一。

// Assert 不是运行时检查 —— 它是"读规范的人帮规范作者排错"的标记。
// 引擎实现可以加 debug-only assertion 检查。

Assert 是规范的"代码审查注释"——它告诉读者"这里有个不变量，到这一步必然成立"，让你不用回头去验证前面分支为什么把 desc 缩窄到 accessor 类型。实践上读 Assert 等于读"规范作者的 mental model"，比读步骤本身信息密度更高。

If/Else/Repeat 的算法控制流

规范用有缩进的编号步骤表达控制流：

规范的控制流记法 
// 1. If _desc_ is undefined, then
//    a. Let _parent_ be ? O.[[GetPrototypeOf]]().
//    b. If _parent_ is null, return undefined.
//    c. Return ? _parent_.[[Get]](P, Receiver).
// 2. If IsDataDescriptor(_desc_) is true, return _desc_.[[Value]].

// 等价 JS：
if (desc === undefined) {
  const parent = O.[[GetPrototypeOf]]();   // ? 是 throw 传播
  if (parent === null) return undefined;
  return parent.[[Get]](P, Receiver);
}
if (IsDataDescriptor(desc)) return desc.[[Value]];

// Repeat、While、For each 也类似：
// "Repeat,
//    a. Let _v_ be ? IteratorNext(_iterator_).
//    b. If _v_ is done, return _result_."
// 等价于 while (true) { ... }
//
// "For each element _e_ of _list_, do
//    a. ..."
// 等价于 for (const e of list) { ... }

Editorial vs Normative 变更

规范每天都在更新（每两个月 TC39 会议合并新提案）。变更分两类：

  • Editorial（编辑性）—— 不影响行为。重命名变量、改文字、补 NOTE、修排版。引擎不需要改实现。
  • Normative（规范性）—— 影响行为。引擎必须更新实现。可能破坏向后兼容，所以非常谨慎。

GitHub 上每个 ecma262 PR 都标 editorial-change 或 normative-change 标签。读 PR 时看标签先判断"我家引擎需不需要担心"。

Stage 流程文档（与 P1.2 / P1.3 配合阅读）

P1.2 现代生态 已详述 Stage 0-4 流程。从规范阅读视角补充几点：

  • 每个 Stage 3+ 提案有完整的 spec text（用 ecmarkup 写，跟 ECMA-262 同款语法）
  • 提案仓库结构标准：README.md（高层介绍）+ spec.html（规范文本）+ polyfill/（参考实现）+ test/（Test262 风格测试）
  • 读 Stage 3 提案是练规范阅读的最佳材料——比读 ECMA-262 主体短得多（一个提案通常 5-50 页 spec text），但用了同样的记法

推荐入门提案：tc39/proposal-explicit-resource-management（ES2025 using）—— 规范文本约 30 页，含 using 声明、Symbol.dispose、DisposableStack 等，难度中等，能让你完整经历"读 Stage 3 提案 → 看引擎实现 → 写 polyfill"的全流程。

ECMA 关联标准家族

ECMA-262 不是孤立的，它是 ECMA International 标准家族的核心成员。本主题只关心 ECMA-262，但读规范时偶尔会撞到几个亲戚标准的引用——一句话识别它们：

  • ECMA-262 —— JS 核心语言（本主题主角）
  • ECMA-402 —— Intl 国际化 API（独立编号但与 ECMA-262 同步年度发布；详见 JS · Date 与 Intl）
  • ECMA-404 —— JSON Data Interchange Format（极简标准，只有 14 页，定义了 JSON 文本语法）
  • ECMA-414 —— ECMAScript 规范元标准（"如何写一个 ES 标准"）
  • ECMA-419 —— ECMAScript 嵌入式 API（小型设备 IoT 场景，是 Moddable / XS 引擎的标准化）

关键判断：ECMA-262 之外的标准是"扩展能力"，不影响核心语言行为。本主题只关心 ECMA-262；遇到 Intl / JSON 的具体 API 用法链回 JS 主题；遇到嵌入式 / 元规范的引用，知道它们存在即可，不深究。

03 应用场景

场景 A · 读 V8 / SpiderMonkey 的 GitHub issue

V8 issue tracker（v8-bugs）和 SpiderMonkey bug tracker 上，工程师交流时大量用 §X.Y 编号引用规范。例：

真实 issue 风格 
// "This violates ECMA-262 §10.1.8.1 step 4 — IsAccessorDescriptor
//  should be asserted, but our impl returns undefined here when the
//  descriptor is malformed."

// 翻译：在 OrdinaryGet 第 4 步，规范要求 desc 必然是 accessor descriptor，
//      但我们的实现返回 undefined 了。是 bug。

读懂这种交流要 ① 知道 §10.1.8.1 是哪个抽象操作；② 知道 step 4 是什么；③ 知道 Assert 是什么意思。本章训练的就是这三件事。

场景 B · 写 polyfill 必须照规范实现 abrupt completion

写 Array.prototype.findLast polyfill（ES2023）—— 不照规范的"每一步可能 throw 都要传出去"，会出错。例如：

javascript · 简陋 polyfill 的 bug 
// 错的简陋 polyfill：
Array.prototype.findLast = function (callback) {
  for (let i = this.length - 1; i >= 0; i--) {
    if (callback(this[i], i, this)) return this[i];   // ← callback throw 时呢？
  }
};

// 规范要求按 §22.1.3.6.1 实现，关键步骤：
// 6. Repeat, while _k_ ≥ 0,
//    a. Let _Pk_ be ! ToString(F(_k_)).
//    b. Let _kValue_ be ? Get(_O_, _Pk_).      ← ? 传播 Get 的 throw（如 Proxy 的 trap）
//    c. Let _testResult_ be ToBoolean(? Call(_predicate_, _thisArg_, « _kValue_, F(_k_), _O_ »)).
//    d. If _testResult_ is true, return _kValue_.

// 关键差异：要用 ? Get(O, Pk) 而不是 this[i]，因为：
// - Proxy 对象的 [[Get]] trap 可以 throw
// - 用 [] 直接访问也会调用 [[Get]]，但中间步骤的错误可能被忽略
// - 规范保证 callback 看到的就是 ? Get 返回的值（含正确的 abrupt 处理）

生产级 polyfill（如 core-js 内部）严格按规范步骤实现。读 core-js 源码本质就是看"规范步骤的 JS 翻译"。

场景 C · 解读 TC39 proposal 的 spec text

团队想用一个 Stage 3 提案——比如 AsyncContext。读它的 spec.html，需要：

  • 识别新增的抽象操作（AsyncContextSnapshot、AsyncContextSwap）
  • 识别在哪些现有抽象操作里插入新步骤（如修改 NewPromiseReactionJob）
  • 识别哪些是新内部槽（如 [[AsyncContextMapping]]）

掌握这些后能预判：这个特性会改 V8 哪些代码、polyfill 能 polyfill 到什么程度、对现有代码是否破坏向后兼容。是评估"要不要用 Stage 3 提案"的关键能力。

04 典型代码

把 OrdinaryGet 翻译成可执行 JS

javascript · 规范步骤完整翻译 
// ECMA-262 §10.1.8.1 OrdinaryGet ( O, P, Receiver )
//
// 1. Let _desc_ be ? O.[[GetOwnProperty]](P).
// 2. If _desc_ is undefined, then
//    a. Let _parent_ be ? O.[[GetPrototypeOf]]().
//    b. If _parent_ is null, return undefined.
//    c. Return ? _parent_.[[Get]](P, Receiver).
// 3. If IsDataDescriptor(_desc_) is true, return _desc_.[[Value]].
// 4. Assert: IsAccessorDescriptor(_desc_) is true.
// 5. Let _getter_ be _desc_.[[Get]].
// 6. If _getter_ is undefined, return undefined.
// 7. Return ? Call(_getter_, Receiver).

function OrdinaryGet(O, P, Receiver) {
  // 步骤 1（? 处理 throw）
  const desc = O.[[GetOwnProperty]](P);   // 假设 [[GetOwnProperty]] throw 时整个函数也 throw

  // 步骤 2
  if (desc === undefined) {
    const parent = O.[[GetPrototypeOf]]();
    if (parent === null) return undefined;
    return parent.[[Get]](P, Receiver);
  }

  // 步骤 3
  if (IsDataDescriptor(desc)) return desc.[[Value]];

  // 步骤 4 —— Assert（实现可加 debug 断言）
  console.assert(IsAccessorDescriptor(desc), 'desc must be accessor');

  // 步骤 5-7
  const getter = desc.[[Get]];
  if (getter === undefined) return undefined;
  return Call(getter, Receiver);
}

// 这段 JS（语义上）就是 V8 / SpiderMonkey 等引擎实现 obj.x 属性查找的核心。
// 读懂这段就读懂了"原型链查找"的官方算法。

05 常见陷阱

⚠️ 陷阱 1：跳过 §5 直接读 §6 → 看不懂 ?

最常见的错误。打开规范第一反应是去查 §6 "Data Types"，结果第一段就是 "Let _value_ be ? ToPrimitive(...)"，? 是什么？读起来一脸懵。

规避：先花 30 分钟读 §5 全章（Notational Conventions）。这是规范的"说明书"，不读直接看正文必迷路。

⚠️ 陷阱 2：把 Annex B 当主规范用

写 Node 工具用 String.prototype.substr、__proto__ =，跨平台时（Deno strict、嵌入式 JS）跑不起来。

规避：写新代码避免 Annex B 特性。代码评审清单加一条："这段代码用了 Annex B 吗？" Annex B 的特性常用 ESLint no-restricted-properties 拦截。

⚠️ 陷阱 3：把 NOTE 当无关紧要跳过

正文步骤告诉你"怎么做"，NOTE 告诉你"为什么"。跳过 NOTE 就丢了规范作者的设计意图。

规避：读完每个抽象操作后必读它的 NOTE。常常一句 NOTE 解释了 10 行步骤的目的。比如 OrdinaryGet 的 NOTE 解释了"这套设计让 ordinary objects 共享 [[Get]] 实现，同时让 Proxy 能完全覆盖"——一句话讲清整个调度模式。

⚠️ 陷阱 4：在自己 polyfill 里误用 ! 简写

读到"! ToObject(_value_)" 直接抄成 const o = ToObject(value); 不做检查，结果 value 是 null / undefined 时静默失败。! 是"调用方保证已经过滤"的简写，但你的实现可能没有这层过滤。

规避：写 polyfill 时把 ! 当 dev assertion——console.assert(value != null, 'ToObject precondition failed')。生产构建可移除，但开发时帮你抓 bug。
