Reflect 与元编程谱

P6.1 讲了 Proxy 怎么替换 11 个 essential internal methods——但元编程是一对设计，Reflect 是 Proxy 的对应物：把 internal methods "暴露成静态函数"，让 trap 内可以调它们做"转发原行为"，让外部代码可以以"显式调内部方法"的姿态做反射。本章把 Reflect 完整 13 个 API + 与 Proxy 的协作模式 + Membrane 跨 Realm 隔离的经典实现 + WeakRef / FinalizationRegistry 的 GC 语义 一次讲透——让 P6 形成完整的元编程谱。

📐 本章交叉规则

Proxy / Reflect 用法在 JS · Proxy / Reflect 已有完整 API 列表与示例。本章不重复，聚焦"Reflect 为什么存在 + Proxy + Reflect 标准搭配 + Membrane 模式 + WeakRef GC 语义"——这些是 JS 主题没展开的"设计层 + 工程模式"。

阅读时长 ~30 分钟 前置 P6.1 内部槽与内部方法（必读） 对应 ECMA-262 §28.1 Reflect + §26 WeakRef

01 速览

三句话定义本章：

  1. Reflect 不是新功能，是把 internal methods "暴露成函数"。13 个静态方法分别对应 13 个 Proxy trap 和 11 个 essential internal methods。在 Proxy trap 内调 Reflect.X(target, ...args) 是"转发原行为"的标准惯用法。
  2. Membrane 模式 = "一对 Proxy 包装跨边界对象 + WeakMap 维护身份映射 + 撤销时所有跨边界引用同时失效"。是 SES（Secure ECMAScript） / ShadowRealm polyfill / Realms shim 等可信执行环境的核心基础设施。
  3. WeakRef + FinalizationRegistry（ES2021）让 JS 第一次有了"不阻止 GC 的引用" + "GC 时执行 cleanup"。但规范刻意不规定 GC 时机——不能把它当"析构函数"用。

Reflect 与 Proxy 的对应关系一图：

Reflect ↔ Proxy trap ↔ internal method 三方对应

  set(t, p, v, r) { return Reflect.set(t, p, v, r); },
  // ...
});

02 底层逻辑

为什么需要 Reflect

ES2015 之前，做"反射"操作得用一堆零散 API：

操作 ES2015 之前 ES2015 之后
读属性 obj[key] Reflect.get(obj, key)
写属性（失败抛错） Object.defineProperty 抛错 Reflect.defineProperty 返回 false
调用函数 fn.apply(thisArg, args) Reflect.apply(fn, thisArg, args)
用 new 调用 new Fn(...args)（不能展开） Reflect.construct(Fn, args, newTarget)
检查属性存在 key in obj Reflect.has(obj, key)
删属性 delete obj[key] Reflect.deleteProperty(obj, key)

三个关键改进：

  • 统一签名：所有 Reflect 方法都是 (target, ...args)，签名整齐——便于在 Proxy trap 内做"通用转发"
  • 返回 boolean 而不是抛错：Reflect.set / Reflect.defineProperty 失败返回 false，不抛 TypeError——比 Object.defineProperty 更便于处理失败
  • 支持 Receiver / newTarget：Reflect.get(t, p, r) 的 r 是 getter 内 this 的值；Reflect.construct(F, args, NT) 的 NT 是 new.target——这两个旋钮在普通语法里拿不到

Proxy + Reflect 标准搭配

Proxy trap 的默认行为应该是"不拦截，透明转发"。这时调 Reflect 是惯用法（而不是手动 target[p]，会丢 receiver 上下文）：

javascript · Vue 3 / Solid 风格响应式 
function reactive(target) {
  return new Proxy(target, {
    get(t, p, r) {
      track(t, p);                   // 拦截：收集依赖
      return Reflect.get(t, p, r);   // 转发：保持原行为（含 receiver）
    },
    set(t, p, v, r) {
      const oldValue = t[p];
      const result = Reflect.set(t, p, v, r);  // 转发：尝试写
      if (result && oldValue !== v) {
        trigger(t, p);               // 拦截：触发更新
      }
      return result;
    },
    deleteProperty(t, p) {
      const had = Reflect.has(t, p);
      const result = Reflect.deleteProperty(t, p);
      if (had && result) trigger(t, p);
      return result;
    },
  });
}

// 关键：trap 内必须用 Reflect 转发，不能用 t[p]：
//   const obj = { get name() { return this.first + ' ' + this.last; } };
//   const proxy = reactive(obj);
//   proxy.name；
//   ↑ 走 get(t='obj', p='name', r=proxy)。
//     用 t['name'] 调 getter 时 this = t（原始 obj），收集不到 proxy 的依赖。
//     用 Reflect.get(t, 'name', r) 时 this = r（proxy），依赖收集到 proxy。
//     这就是为什么 Reflect 比 t[p] 更"对"。

Membrane 模式：跨边界隔离

问题场景：你想跑一段第三方代码，希望它能拿到一些数据（"读权限"），但所有"写" 都被拦截，并且能在某一刻整体撤销整个隔离环境——所有跨边界引用立刻失效。

Membrane 的核心思想：在两个对象图（"main side" / "guest side"）之间放一对镜像 Proxy。任何引用穿越边界时自动包装，并用 WeakMap 维护身份映射（保证同一个对象多次穿越仍是同一个 Proxy）。撤销时把所有 Proxy 标记为 revoked，整片对象图同时失效。

javascript · 最小可运行 Membrane（30 行核心） 
function makeMembrane(initialTarget) {
  const wrapped = new WeakMap();    // original → proxy（避免重复包装）
  const revokers = [];               // 所有 Proxy.revoke 函数

  function wrap(value) {
    if (value === null || (typeof value !== 'object' && typeof value !== 'function')) {
      return value;  // primitive 直接传，不需要包装
    }
    if (wrapped.has(value)) return wrapped.get(value);  // 身份映射

    const { proxy, revoke } = Proxy.revocable(value, {
      get(t, p, r) {
        return wrap(Reflect.get(t, p, r));  // 出去的引用也包装
      },
      set(t, p, v, r) {
        return Reflect.set(t, p, unwrap(v), r);  // 进来的解包
      },
      apply(t, thisArg, args) {
        return wrap(Reflect.apply(t, unwrap(thisArg), args.map(unwrap)));
      },
      // ... 其他 trap 同理
    });

    wrapped.set(value, proxy);
    revokers.push(revoke);
    return proxy;
  }

  function unwrap(value) { /* 反向：把 proxy 找回原对象 */ }

  return {
    expose: wrap(initialTarget),
    revoke: () => revokers.forEach(r => r()),
  };
}

// 用法：
const { expose, revoke } = makeMembrane(realData);
plugin.run(expose);   // plugin 拿到的是 wrapped 视图
// ...
revoke();              // 一键撤销所有跨边界引用，plugin 持有的 proxy 全部失效

实战中 Membrane 还要处理：原型链（Reflect.getPrototypeOf 的返回值也要包装）、Symbol 属性、Promise 跨边界（resolve 出来的值要包装）、Error 跨边界（栈帧泄漏问题）等。完整工业实现：Agoric SES-shim、Salesforce observable-membrane。

与 ShadowRealm 的关系：ShadowRealm（P7.1 已介绍，Stage 3）原生就是语言级 Membrane——跨边界传值被强制限制为 primitive + 函数。在它落地前，社区 polyfill 用 Membrane 实现近似能力。

WeakRef + FinalizationRegistry（ES2021）

WeakRef：持有不阻止 GC 的弱引用。

javascript · WeakRef 基本用法 
let big = { /* 1MB 数据 */ };
const ref = new WeakRef(big);

ref.deref();  // 返回 big 或 undefined（如果已被 GC）

big = null;   // 解除强引用
// 此后 GC 可能回收 big；ref.deref() 何时变 undefined 由 GC 决定

// 典型用法：缓存中持有"建议保留但允许回收"的对象
const cache = new Map();
function getThumbnail(url) {
  const ref = cache.get(url);
  const cached = ref?.deref();
  if (cached) return cached;  // 还活着，复用

  const fresh = generateThumbnail(url);  // 重新生成（昂贵）
  cache.set(url, new WeakRef(fresh));
  return fresh;
}

FinalizationRegistry：注册对象 GC 时执行的 cleanup 回调。

javascript · 自动释放外部资源 
// 场景：JS 对象包了一个 WASM 内存指针，希望对象 GC 时释放 WASM 端的内存

const registry = new FinalizationRegistry((heldValue) => {
  // heldValue 是注册时给的"清理依据"
  wasmModule.free(heldValue);
  console.log('已释放 WASM 内存:', heldValue);
});

class WasmBuffer {
  constructor(size) {
    this.ptr = wasmModule.malloc(size);
    registry.register(this, this.ptr);  // 注册：当 this 被 GC 时调 cleanup(ptr)
  }
}

let buf = new WasmBuffer(1024);
buf = null;  // 一段时间后 GC 触发，cleanup 自动跑

// ⚠️ 关键约束：cleanup 触发时机不确定
//   - GC 时机由引擎决定，可能立刻、可能几分钟后、甚至程序退出前都不跑
//   - cleanup 在 microtask 里跑，但什么时机被排队 = GC 决定
//   - 不应放业务关键逻辑（比如刷新 UI、发请求、关键资源释放）

规范刻意不规定 GC 时机，以避免开发者写出依赖回收时机的代码。这是设计上的"反模式防御"——把 GC 当作"看不见的优化"而不是"可调度的析构"。

Reflect.construct + new.target 元编程

javascript · Reflect.construct 高级用法 
// 用法 1：用一个构造器构造另一个的实例（changeTo subclass 模式）
class Animal { constructor(name) { this.name = name; } }
class Dog extends Animal {}

const d = Reflect.construct(Animal, ['Rex'], Dog);
//             ↑ 用 Animal 的逻辑构造，但实例的 [[Prototype]] 链来自 Dog
d instanceof Dog;  // true！
d.name;            // "Rex"

// 用法 2：动态参数（args 是数组，spread 比 new Fn(...args) 更直接）
function build(Cls, args) {
  return Reflect.construct(Cls, args);
}

// 用法 3：写 abstract class 检测
class Shape {
  constructor() {
    if (new.target === Shape) {
      throw new Error('Shape is abstract');
    }
  }
}
class Circle extends Shape { /* OK */ }
new Shape();    // throws
new Circle();   // OK

03 应用场景

场景 A · 写 Vue 3 / Solid / Mobx 风格响应式

响应式系统的核心是 Proxy + Reflect。Proxy 拦截属性访问，Reflect 转发原行为同时保持 receiver 正确。Vue 3 的 reactive() 内部就是 30 行这个模板，加上"track 收集依赖 / trigger 触发副作用"。

场景 B · 写沙箱 / 插件系统

想跑第三方代码，但需要：① 隔离命名空间 ② 限制可见 API ③ 一键撤销整个环境。Membrane 是核心机制——比 iframe 轻、比 vm.createContext 跨平台。Salesforce LWC、Figma 插件系统、低代码引擎都用类似 Membrane。

场景 C · 缓存 + 自动 WASM 内存释放

处理大量 wasm 计算结果（图像缩略图、ML 推理结果）—— 既想缓存复用，又怕内存爆。WeakRef 让缓存"有则用、没则重建"；FinalizationRegistry 在 JS 对象被回收时释放对应 WASM 内存。注意 cleanup 时机不可靠，仍要有手动 dispose 兜底。

04 典型代码

Proxy + Reflect 模板（透明拦截器）

javascript · 标准 trap 转发 
function tracingProxy(target, name) {
  return new Proxy(target, {
    get(t, p, r) {
      console.log(`[${name}] get`, String(p));
      return Reflect.get(t, p, r);
    },
    set(t, p, v, r) {
      console.log(`[${name}] set`, String(p), '=', v);
      return Reflect.set(t, p, v, r);
    },
    deleteProperty(t, p) {
      console.log(`[${name}] delete`, String(p));
      return Reflect.deleteProperty(t, p);
    },
    has(t, p) {
      console.log(`[${name}] has`, String(p));
      return Reflect.has(t, p);
    },
    ownKeys(t) {
      console.log(`[${name}] ownKeys`);
      return Reflect.ownKeys(t);
    },
    apply(t, thisArg, args) {
      console.log(`[${name}] apply`, args);
      return Reflect.apply(t, thisArg, args);
    },
    construct(t, args, nt) {
      console.log(`[${name}] construct`, args);
      return Reflect.construct(t, args, nt);
    },
  });
}

const data = tracingProxy({ a: 1 }, 'data');
data.a;       // [data] get a
data.b = 2;   // [data] set b = 2
'a' in data;  // [data] has a

WeakRef-based LRU 缓存

javascript · 弱缓存 
class WeakLRU {
  constructor() { this.map = new Map(); }
  get(key) {
    const ref = this.map.get(key);
    const value = ref?.deref();
    if (value === undefined && ref !== undefined) {
      this.map.delete(key);  // 已被 GC，清理 stale ref
    }
    return value;
  }
  set(key, value) {
    this.map.set(key, new WeakRef(value));
  }
}

const cache = new WeakLRU();
cache.set('user:1', { name: 'Alice', /* big data */ });
// ... 内存压力下，user:1 可能被 GC 回收
cache.get('user:1');  // 可能是 undefined（已回收）或对象（还活着）

// 注意：基础类型（number / string）不能 WeakRef，要转 wrapper

05 常见陷阱

⚠️ 陷阱 1：Proxy trap 内忘了用 Reflect 转发

写 get(t, p) { /* 拦截 */ return t[p]; } —— 看起来对，但 receiver 丢了。如果属性是 getter，this 是 t（原对象）而不是 proxy，依赖收集失败、嵌套响应式失效。

规避：永远用 Reflect.get(t, p, r) 转发，把 receiver 显式传进去。这是 Vue 3、Solid、Mobx 等响应式系统的标准模板。

⚠️ 陷阱 2：用 WeakRef 当强引用替代品

const ref = new WeakRef(obj); 之后立刻 ref.deref() 一定拿得到，但过段时间未必。把 WeakRef 当"持久存"用会丢数据。

规避：只用 WeakRef 表达"建议保留但允许回收"的语义；如果一定要保留，用普通强引用。

⚠️ 陷阱 3：FinalizationRegistry 回调跑业务逻辑

cleanup 不保证触发——GC 可能永远不跑（程序退出前对象一直可达；引擎不优先回收）。把"关闭 socket / 释放 lock / 通知用户" 放进 cleanup 会导致资源泄漏或行为不一致。

规避：cleanup 只放"可有可无的优化"（清缓存、释放可重新申请的资源）。关键资源用显式 dispose / using 模式（详见 P7.3 异步演进 的 using 节）。

⚠️ 陷阱 4：Membrane 漏拦截 Symbol / 原型方法

手写 Membrane 容易漏：① Symbol 属性的访问（要在 get trap 里处理 Symbol key）② 原型链方法（getPrototypeOf 返回的也要包装）③ Promise.then 的回调参数（resolve 出来的值跨边界要包装）。漏一处就破隔离。

规避：直接用 SES-shim 等成熟实现，不要手写工业级 Membrane（除非学习目的）。

⚠️ 陷阱 5：Reflect.set 失败被忽略

Reflect.set 在写入失败（如对象 frozen）时返回 false 而不抛错。如果不检查返回值，会出现"看似写了但实际没写" 的静默失败。

规避：在 Proxy trap 里 const ok = Reflect.set(...); if (!ok) throw new TypeError(...);，或在 strict mode 下用 obj.x = v（失败抛错）替代。
