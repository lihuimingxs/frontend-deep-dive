内部槽与内部方法（§10）

ECMA-262 §10 是"对象本质"的规范层定义：每个对象有 11 个 essential internal methods（[[GetPrototypeOf]] / [[Get]] / [[Set]] / [[Delete]] 等）+ 若干内部槽（[[Prototype]] / [[Extensible]] 等）。普通对象的 internal methods 走"Ordinary 默认实现"（如 OrdinaryGet），7 种 Exotic Object（Array、String、Arguments、Module Namespace、Proxy、Bound、TypedArray）覆盖部分方法实现特殊行为。Proxy 的本质就是"替换全部 11 个 internal methods 为 trap"——理解这一层，Proxy 这个被神秘化的特性立刻变得朴素直白。

📐 本章交叉规则

原型链查找、Object.freeze 行为、Proxy 用法、TypedArray 等"用户视角"机制在 JS · 原型链、JS · Proxy / Reflect 等章节已详述。本章从规范层"对象内部方法"视角讲——把"对象的行为"还原成"11 个内部方法的实现"。

阅读时长 ~30 分钟 前置 P3 + P4 + P5 对应 ECMA-262 §10 Ordinary and Exotic Objects Behaviours

01 速览

三句话定义本章：

  1. 对象 = 内部槽 + 11 个 internal methods + 属性表。所有对象行为（obj.x / obj.x = 1 / delete obj.x / x in obj）都映射到这 11 个方法。
  2. Ordinary 是默认实现，Exotic 是覆盖。普通对象走 OrdinaryGet / OrdinarySet 等；7 种 Exotic 对象（Array / String / Arguments / Module Namespace / Proxy / Bound / TypedArray）覆盖部分内部方法实现特殊行为。
  3. Proxy 的本质就是"把全部 11 个 internal methods 重定向到 trap"。每个 trap 没定义时回落到 OrdinaryX(target)。这套机制让 Vue 3 / Solid / Mobx 等响应式系统能完全拦截属性访问。

11 个 essential internal methods 总览：

spec · §10.1 11 个 essential internal methods 
// 原型 / 可扩展性
[[GetPrototypeOf]]()      → Object | null
[[SetPrototypeOf]](V)     → boolean
[[IsExtensible]]()        → boolean
[[PreventExtensions]]()   → boolean

// 属性查询
[[GetOwnProperty]](P)     → PropertyDescriptor | undefined
[[DefineOwnProperty]](P, Desc) → boolean
[[HasProperty]](P)        → boolean
[[OwnPropertyKeys]]()     → List of (String | Symbol)

// 读写删
[[Get]](P, Receiver)      → any
[[Set]](P, V, Receiver)   → boolean
[[Delete]](P)             → boolean

// 函数对象额外（非所有对象都有）
[[Call]](thisArg, args)   → any
[[Construct]](args, newTarget) → Object

02 底层逻辑

Ordinary Object —— 默认实现（§10.1）

Ordinary（普通）就是"没有特殊行为的常规对象"——你写 { a: 1 } 创建的就是 Ordinary。它的 11 个 internal methods 都用规范定义的"Ordinary*"系列函数实现：

internal method Ordinary 实现
[[GetPrototypeOf]] OrdinaryGetPrototypeOf —— 直接返回 [[Prototype]] 内部槽
[[SetPrototypeOf]](V) OrdinarySetPrototypeOf —— 检查是否会形成循环原型链，再设
[[GetOwnProperty]](P) OrdinaryGetOwnProperty —— 在属性表里查 P，返回 PropertyDescriptor
[[DefineOwnProperty]](P, Desc) OrdinaryDefineOwnProperty —— 检查 configurable 等再写入属性表
[[HasProperty]](P) OrdinaryHasProperty —— 自有 + 沿原型链查
[[Get]](P, Receiver) OrdinaryGet —— B1 给过完整算法（沿原型链 + getter）
[[Set]](P, V, Receiver) OrdinarySet —— 找属性 → setter？data？沿原型链？
[[Delete]](P) OrdinaryDelete —— 检查 configurable 再删
[[OwnPropertyKeys]]() OrdinaryOwnPropertyKeys —— 数字 key 升序 → 字符串 key 插入序 → Symbol key 插入序

属性遍历顺序的规范保证：OrdinaryOwnPropertyKeys 明确：① 整数 key 按数值升序（"0, 1, 2, ..., 10, 11"）；② 字符串 key 按插入序；③ Symbol key 按插入序。这是 ES2015 起的规范保证——之前各引擎实现不一致。

7 种 Exotic Object（§10.2-§10.5）

Exotic Object = 至少有一个 internal method 不是 Ordinary 默认实现的对象。规范定义了 7 种：

① Array Exotic Object（§10.4.2）—— length 自动同步

数组对象的特殊行为：写 arr.length = N 时自动删除超出索引的元素，写 arr[N] = x 时自动更新 length。这是通过覆盖 [[DefineOwnProperty]] 实现的：

javascript · Array exotic 行为 
const arr = [1, 2, 3, 4, 5];
arr.length = 3;            // 触发 [[DefineOwnProperty]]("length", { value: 3, ... })
                            // Array 的特殊版本：删除 arr[3]、arr[4]
console.log(arr);          // [1, 2, 3] —— 元素真的被删了

arr[10] = 'x';             // 触发 [[DefineOwnProperty]]("10", ...)
                            // Array 的特殊版本：把 length 自动改成 11
console.log(arr.length);   // 11
console.log(arr);          // [1, 2, 3, <7 empty items>, 'x']

// 这种"写 X 影响 Y"的行为是 Ordinary 做不到的，必须 Exotic。
// "Array.isArray" 就是判断对象是不是 Array exotic（B3 已讲）。

// 一个 plain object 即使有 length 属性和数字 key，也不是 Array exotic：
const arrLike = { 0: 'a', 1: 'b', length: 2 };
arrLike.length = 1;
console.log(arrLike);      // { 0: 'a', 1: 'b', length: 1 } —— length 改了但 a/b 还在！
arrLike.map?.();           // undefined —— Array.prototype.map 不在它原型链上

② String Exotic Object（§10.4.3）—— index → char 映射

String 包装对象（new String("abc")）和原始字符串自动装箱时产生 Object：它的 [[GetOwnProperty]] 被覆盖，整数 key 直接返回对应位置的 1-字符 String：

javascript · String exotic 行为 
const s = "abc";
s[0];                       // 'a'
                            // 等价于 ToObject(s)[0] —— ToObject 装箱成 String exotic
                            // String exotic 的 [[GetOwnProperty]]("0") 返回 'a'

s[100];                     // undefined —— 越界
"abc"[1.5];                 // undefined —— 非整数 key 走 Ordinary 路径，没这属性

// String exotic 也是 frozen-like：所有 index property 是 non-writable, non-configurable
const o = Object(s);        // 显式装箱
o[0] = 'x';                 // 静默失败（strict 抛 TypeError）
o[0];                       // 仍是 'a'

③ Arguments Exotic Object（§10.4.4）—— 与命名参数同步

非 strict 函数的 arguments 对象是 Exotic：修改 arguments[0] 会同步修改第一个命名参数（"mapped arguments"）。strict 模式下退化成 Ordinary Object，不再同步：

javascript · arguments mapped vs unmapped 
function nonStrict(a, b) {
  arguments[0] = 100;
  console.log(a);            // 100 —— mapped！修改 arguments 影响命名参数
}

function strict(a, b) {
  'use strict';
  arguments[0] = 100;
  console.log(a);            // 原值 —— unmapped，arguments 是 Ordinary
}

// 现代代码已不用 arguments —— 用 ...rest 代替（ES2015）
// 因为 mapped arguments 的"行为耦合"反直觉，且 strict/sloppy 不一致

④ Module Namespace Exotic Object（§10.4.6）—— frozen + live binding

import * as ns from './m' 中的 ns 是 Module Namespace exotic。它的特殊行为：

  • 所有属性 non-writable, non-configurable（ns.foo = 1 抛错）
  • 但 [[Get]] 是"live binding"——读 ns.foo 时实际去模块 EnvironmentRecord 取最新值（A2 设计原则讲过）
  • 没有 [[Prototype]]（设为 null），obj.toString() 都没有

⑤ Proxy Exotic Object（§10.5）⭐ —— 最核心

Proxy 的本质：每个 internal method 都不是 Ordinary 实现，而是查 handler 上对应的 trap。trap 没定义时回落到 OrdinaryX(target)。

Proxy internal method handler trap
[[GetPrototypeOf]] getPrototypeOf
[[SetPrototypeOf]] setPrototypeOf
[[IsExtensible]] isExtensible
[[PreventExtensions]] preventExtensions
[[GetOwnProperty]] getOwnPropertyDescriptor
[[DefineOwnProperty]] defineProperty
[[HasProperty]] has
[[Get]] get
[[Set]] set
[[Delete]] deleteProperty
[[OwnPropertyKeys]] ownKeys
[[Call]]（仅 callable target） apply
[[Construct]]（仅 constructible target） construct
📐 一句话理解 Proxy

Proxy 不是"拦截器"（虽然常这样说）。它是一个把 11 个 essential internal methods 全部重定向到 handler 对应 trap的 Exotic Object。每个没实现的 trap 自动回落到 OrdinaryX(target)——这就是为什么 new Proxy({}, {})（空 handler）行为完全像普通对象。

因此 Proxy 是 ECMAScript 元编程能力的最高层——它能拦截语言能感知的所有对象操作。Vue 3 / Solid 用 get trap 收集依赖、set trap 触发更新；Mobx 用 get 转 observable；immer 用全套 trap 实现"看似可变但实际不可变"的 produce API。

⑥ Bound Function Exotic Object（§10.4.1）

fn.bind(thisArg, ...args) 返回的对象是 Bound Function exotic。它的特殊行为：

  • [[Call]] 调用时把 thisArg 强制设为 bind 时传的（忽略实际调用的 this）
  • [[Construct]] 时不忽略 newTarget（new boundFn() 仍然以 boundFn 为 newTarget）
  • 不会有 prototype 属性（与原函数解耦）

⑦ Integer-Indexed Exotic Object（§10.4.5）—— TypedArray

Int32Array / Float64Array 等的内部。整数 index 不是属性查找，而是直接读底层 ArrayBuffer 的字节：

javascript · TypedArray exotic 
const buf = new ArrayBuffer(8);
const view = new Int32Array(buf);
view[0] = 0x12345678;        // 触发 [[Set]] —— TypedArray 特殊：写入底层 Data Block 的字节
view[1] = -1;
view[100] = 42;              // 越界 —— 静默失败（不抛错，也不存）
view.foo = 'bar';            // 字符串 key 走 Ordinary —— 但仍能存
console.log(view.foo);       // 'bar' —— 字符串 key 用 Ordinary 行为

// TypedArray 是"整数 key 用 exotic、其他 key 用 ordinary"的混合

[[Call]] 与 [[Construct]] —— 函数对象的 internal methods

函数对象（function、class、箭头函数、bound function）都有 [[Call]]，但不一定有 [[Construct]]：

类型 有 [[Call]]？ 有 [[Construct]]？
普通 function 声明 ✅ ✅
class ❌（直接调用抛 TypeError） ✅
箭头函数 () => {} ✅ ❌
方法简写 { method() {} } ✅ ❌
Generator function* ✅ ❌
async function ✅ ❌
Bound function 看 target 看 target

[[Construct]] 调用时还接收 newTarget（B3 提过）—— 这是 ES2015 引入的概念，new.target 在函数内访问到。Reflect.construct(F, args, newTarget) 是显式传 newTarget 的入口。

PrivateElement / PrivateName（§6.1.7.1）

ES2022 加的 class #field 私有字段在规范层用 PrivateElement 类型表示——它和普通属性完全分开存储，不在属性表里：

javascript · #field 是真私有，不只是约定 
class Counter {
  #count = 0;
  inc() { this.#count++; }
  get value() { return this.#count; }
}

const c = new Counter();
c.inc();

// 完全不可访问：
c['#count'];                  // undefined（不是属性）
Object.keys(c);               // [] —— 属性枚举看不到
Object.getOwnPropertyNames(c); // [] —— 即使非可枚举也看不到
c['#count'] = 100;            // 静默成功 —— 但创建了一个新 string 属性 "#count"！
console.log(c.value, c['#count']); // 1 100

// PrivateElement 存在 ExecutionContext 的 PrivateEnvironment 上（A3 提过），
// 不是 obj 自身的属性 —— 这就是"硬隐私"的实现。

// 跨 class 访问 抛 TypeError：
class Other { test(c) { return c.#count; } }  // SyntaxError 已在 parse 阶段

关键差异：TypeScript 的 private 只是编译期约束（运行时仍是普通属性）；ES2022 的 #field 是规范级硬隐私，引擎层面隔离。详见 TS · Classes。

03 应用场景

场景 A · 写 Proxy-based 响应式系统

Vue 3 / Solid / Mobx 都用 Proxy 实现响应式。最核心的两个 trap：get 收集依赖、set 触发更新。

javascript · 极简响应式 
let activeEffect = null;
const targetMap = new WeakMap();

function track(target, key) {
  if (!activeEffect) return;
  let depsMap = targetMap.get(target);
  if (!depsMap) targetMap.set(target, (depsMap = new Map()));
  let dep = depsMap.get(key);
  if (!dep) depsMap.set(key, (dep = new Set()));
  dep.add(activeEffect);
}

function trigger(target, key) {
  const dep = targetMap.get(target)?.get(key);
  dep?.forEach(effect => effect());
}

function reactive(obj) {
  return new Proxy(obj, {
    get(target, key, receiver) {
      track(target, key);
      const result = Reflect.get(target, key, receiver);
      // 嵌套对象也变 reactive
      return typeof result === 'object' && result !== null
        ? reactive(result)
        : result;
    },
    set(target, key, value, receiver) {
      const result = Reflect.set(target, key, value, receiver);
      trigger(target, key);
      return result;
    },
  });
}

function effect(fn) {
  activeEffect = fn;
  fn();
  activeEffect = null;
}

// 用法：
const state = reactive({ count: 0 });
effect(() => console.log('count is', state.count));   // 立即跑一次：count is 0
state.count++;                                         // 触发 effect：count is 1
state.count++;                                         // count is 2

这就是 Vue 3 reactive 的简化原理。Vue 实际还要处理 Map/Set/Array 等特殊集合的 trap、ref/computed/effect scope 等——但核心是 Proxy 拦截 internal methods。

场景 B · 调试 Array-like 不能 .map

document.querySelectorAll('div') 返回 NodeList，arguments 是 Arguments exotic——它们看起来像数组但 .map 调用失败。原因：它们的原型链上没有 Array.prototype。

javascript · 转成真数组 
// ❌ 直接调用：
const nodes = document.querySelectorAll('div');
nodes.map(...);              // TypeError: nodes.map is not a function

// ✅ 1. Array.from（适用所有 iterable / array-like）
Array.from(nodes).map(...);

// ✅ 2. spread（仅 iterable）
[...nodes].map(...);

// ✅ 3. Array.prototype.map.call（借调）
Array.prototype.map.call(nodes, ...);

// 三者底层都是把 array-like 转成真 Array（含 Array.prototype 链）。
// Array.from 还接受 mapFn 参数，最常用。

场景 C · 实现深 freeze

Object.freeze 是浅冻结：嵌套对象仍可修改。要"真正不可变"需要递归冻结：

javascript · deepFreeze 
function deepFreeze(obj) {
  if (Object.isFrozen(obj)) return obj;
  // 先递归（freeze 后不能修改 child reference）
  for (const key of Reflect.ownKeys(obj)) {
    const val = obj[key];
    if (val && (typeof val === 'object' || typeof val === 'function')) {
      deepFreeze(val);
    }
  }
  return Object.freeze(obj);
}

// Object.freeze 在规范层做的事（§7.3.18 SetIntegrityLevel "frozen"）：
// 1. 调 [[PreventExtensions]]() —— 不能加新属性
// 2. 对所有自有属性调 [[DefineOwnProperty]]：
//    - data: writable=false, configurable=false
//    - accessor: configurable=false（getter/setter 仍可调用）
// 3. 不递归（这就是为什么要 deepFreeze）

const config = deepFreeze({
  api: { url: 'https://...', retries: 3 },
  features: { darkMode: true },
});
config.api.retries = 5;       // 静默失败（strict 抛）
config.features.darkMode = false;  // 同上
config.api;                   // 仍是原对象

注意：Object.freeze 不递归，structuredClone 也不复制 frozen 状态——所以"深冻结 + 深克隆"组合需要自己实现。

04 典型代码

用 Proxy 实现负数索引

javascript · 让 arr[-1] 返回最后一个元素 
function negativeIndex(arr) {
  return new Proxy(arr, {
    get(target, key, receiver) {
      // 检测 "-1", "-2" 这样的字符串 key（注意：key 总是 string 或 symbol）
      if (typeof key === 'string') {
        const n = Number(key);
        if (Number.isInteger(n) && n < 0) {
          return Reflect.get(target, target.length + n, receiver);
        }
      }
      return Reflect.get(target, key, receiver);
    },
  });
}

const arr = negativeIndex([1, 2, 3, 4, 5]);
arr[-1];    // 5
arr[-2];    // 4
arr[0];     // 1
arr.length; // 5（沿原型链查 .length，走 Reflect.get）

// 注意：这是 ES2022 Array.prototype.at(-1) 的"超集"版本 —— at() 是方法调用，
// 这里直接用 [-1] 索引语法。
// 工程上推荐用 .at()，因为 Proxy 有性能开销且行为不直观。

用 Reflect 完整调用 internal methods

javascript · Reflect 是 internal methods 的"用户入口" 
// Reflect 提供了几乎每个 essential internal method 的"显式调用"接口：
// （都在 Reflect 命名空间下，不污染 Object）

const obj = { a: 1 };

// 11 个 internal methods 对应 Reflect 方法：
Reflect.getPrototypeOf(obj);              // [[GetPrototypeOf]]
Reflect.setPrototypeOf(obj, null);        // [[SetPrototypeOf]]
Reflect.isExtensible(obj);                // [[IsExtensible]]
Reflect.preventExtensions(obj);           // [[PreventExtensions]]
Reflect.getOwnPropertyDescriptor(obj, 'a'); // [[GetOwnProperty]]
Reflect.defineProperty(obj, 'b', { value: 2 }); // [[DefineOwnProperty]]
Reflect.has(obj, 'a');                    // [[HasProperty]]
Reflect.get(obj, 'a');                    // [[Get]]
Reflect.set(obj, 'c', 3);                 // [[Set]]
Reflect.deleteProperty(obj, 'a');         // [[Delete]]
Reflect.ownKeys(obj);                     // [[OwnPropertyKeys]]

// 函数额外：
Reflect.apply(fn, thisArg, args);         // [[Call]]
Reflect.construct(F, args, newTarget);    // [[Construct]]

// 写 Proxy trap 时几乎总是用 Reflect 转发未拦截的部分：
new Proxy(target, {
  get(target, key, receiver) {
    console.log('get', key);
    return Reflect.get(target, key, receiver);  // 转发到默认行为
  }
});

// Object.* 和 Reflect.* 的区别：
// - Object.* 失败常抛错；Reflect.* 失败返回 false
// - Reflect 总是有完整签名（含 receiver），适合写 Proxy trap
// - Object.* 设计早，Reflect.* 是 ES2015 整理出来的"完整 internal method API"

05 常见陷阱

⚠️ 陷阱 1：Proxy trap 不实现就忘了它要回落 ordinary

"我只想拦 set，不实现 get"——没问题，未实现的 trap 自动回落到 Reflect.X(target, ...)。但如果你实现了但只 return undefined，会破坏正常行为。

javascript · 错误的 trap 实现 
// ❌ 拦截所有 get 但什么都不返回
new Proxy(obj, { get() { /* 啥都不写 */ } });
// 所有 obj.x 返回 undefined

// ✅ 不实现 → 自动回落
new Proxy(obj, {});

// ✅ 显式转发
new Proxy(obj, {
  get(t, k, r) { return Reflect.get(t, k, r); }
});

规避：要么不实现 trap，要么实现时显式转发不需要拦截的部分给 Reflect。这就是为什么 Vue 3 的 get trap 总是 Reflect.get(target, key, receiver) 兜底。

⚠️ 陷阱 2：用 Object.assign 复制 Array 当 deep clone

Object.assign({}, arr) 复制 Array 的属性到 Object，但目标是 Ordinary Object 不是 Array exotic——结果丢了 length 同步行为，且不是数组：

javascript · Array exotic 行为不复制 
const arr = [1, 2, 3];
const copy = Object.assign({}, arr);
console.log(copy);             // { 0: 1, 1: 2, 2: 3, length: 3 }
console.log(Array.isArray(copy));  // false ⚠️
copy.push(4);                  // TypeError: copy.push is not a function

// 复制 Array 用 [...arr] 或 arr.slice() 或 Array.from(arr)
// 这些都创建新的 Array exotic 对象

规避：复制不同类型的对象用对应的工厂——Array 用 [...arr]、Map 用 new Map(orig)、Set 用 new Set(orig)、Date 用 new Date(d)。

⚠️ 陷阱 3：忽视 Proxy 的性能开销

每次属性访问 Proxy 都要走 trap → JS 函数调用 → Reflect → 真正 internal method。比直接访问慢 5-50 倍。Vue 3 在 hot path 上做了大量缓存优化才达到生产可用性能。

规避：① Proxy 适合"少数关键对象"，不适合包装"所有对象"；② 性能敏感场景用 dirty check 或 immer 等轻量方案；③ 千万别在循环热点里 new Proxy。

⚠️ 陷阱 4：以为 Object.freeze 是深冻结

Object.freeze 只冻结自身属性。嵌套对象仍可修改。

规避：要深冻结自己写递归（上面给过 deepFreeze）。或用 immutable.js / Immer 等库。规范层有 Records & Tuples 提案目标深不可变（卡 Stage 2 至今未进，详见 P1.3 提案旅程）。

⚠️ 陷阱 5：在 strict 函数里用 arguments 期待 mapped 行为

'use strict' 函数（含所有 ES Module、class body）的 arguments 是 unmapped——修改 arguments[0] 不影响命名参数。老代码迁移到 strict 时这是"静默 break"的常见来源。

规避：现代代码用 ...rest 取代 arguments。function (a, ...rest) 行为一致、类型友好（TS 能精确推导）、性能更好（V8 对 rest 优化更狠）。
