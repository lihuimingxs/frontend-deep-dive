B3 · 抽象操作与算法记法（§5、§7）

ECMA-262 §7 定义了几十个抽象操作（Abstract Operations）—— 规范层的"内部函数"。它们没有被用户直接调用，但每个 JS 表达式背后都在执行它们：你写 x + y 就调了 ToPrimitive；写 x == y 就调了 IsLooselyEqual 的 11 步算法；写 for (const x of iter) 就调了 GetIterator + IteratorNext。本章把最高频出现的几十个抽象操作按家族讲透——读完后，规范文本里 80% 的抽象操作引用你都能秒读。

📐 本章交叉规则

typeof、+ 多义、== 反直觉、NaN 行为等"现象级"内容在 JS · 数据类型系统 已详述。本章从规范层视角重新讲——把"现象"还原成"抽象操作的 step-by-step 算法"。读完之后，你能照规范步骤逐行 trace 任何 JS 转换/比较行为。

阅读时长 ~30 分钟 前置 B1 · 怎么读规范 + B2 · 数据类型与值 对应 ECMA-262 §7 Abstract Operations

01 速览

三句话定义本章：

  1. §7 把"JS 表达式背后的算法"切成几十个具名抽象操作。每个都有精确的 step-by-step 定义。理解它们就理解了 JS 行为的源头。
  2. 核心 4 大家族：① 类型转换（ToPrimitive / ToBoolean / ToString / ToObject 等）② 比较（IsLooselyEqual / IsStrictlyEqual / SameValue / SameValueZero）③ 对象操作（Get / Set / HasProperty）④ 迭代器（GetIterator / IteratorNext）。
  3. 最反直觉的两个算法是 ToPrimitive 和 IsLooselyEqual——它们各 11 步左右，每一步都是为历史兼容服务的。本章给完整 step trace。

规范里典型的抽象操作引用样式：

规范引用样式 
// "Let _result_ be ? ToString(_value_)."
//   ↑ ? 处理 abrupt completion（B1 讲过）

// "Let _O_ be ? ToObject(! ToPropertyKey(_key_))."

// "If SameValue(_x_, _y_) is true, return ..."

类型转换族（§7.1）

类型转换是规范最频繁的抽象操作。每个数学/字符串/对象操作背后都至少调一次。

操作 spec § 触发位置

ToBoolean(arg) §7.1.2 if、!、&&、|| 等逻辑上下文
ToNumeric(arg) §7.1.3 数学运算 - * % 等（统一处理 Number 和 BigInt）

ToPrimitive —— 最复杂的转换 ⭐

ToPrimitive(input, hint) 是把对象转成原始值的统一入口。hint 是"调用方期望什么类型"的提示：

  • "number" hint —— 数学运算（obj * 2）
  • "string" hint —— 字符串拼接（String(obj)、`${obj}`）
  • "default" hint —— +、== 用（不知道想要什么类型）
spec · §7.1.1 ToPrimitive 完整算法 
// ToPrimitive ( input [, preferredType] )
// 1. If input is an Object, then
//    a. Let _exoticToPrim_ be ? GetMethod(_input_, @@toPrimitive).
//    b. If _exoticToPrim_ is not undefined, then
//       i.   If preferredType is not present, let _hint_ be "default".
//       ii.  Else if preferredType is string, let _hint_ be "string".
//       iii. Else, let _hint_ be "number".
//       iv.  Let _result_ be ? Call(_exoticToPrim_, _input_, « _hint_ »).
//       v.   If _result_ is not an Object, return _result_.
//       vi.  Throw a TypeError exception.
//    c. If preferredType is not present, let preferredType be number.
//    d. Return ? OrdinaryToPrimitive(_input_, preferredType).
// 2. Return input.

// 翻译：先看对象有没有 [Symbol.toPrimitive] 方法（自定义优先），
//       没有的话走 OrdinaryToPrimitive（默认行为）。
//       原始值直接返回。

// OrdinaryToPrimitive ( O, hint )
// 1. If hint is string, let _methodNames_ be « "toString", "valueOf" ».
// 2. Else, let _methodNames_ be « "valueOf", "toString" ».
// 3. For each name in _methodNames_, do
//    a. Let _method_ be ? Get(O, name).
//    b. If IsCallable(_method_) is true, then
//       i.  Let _result_ be ? Call(_method_, O).
//       ii. If _result_ is not an Object, return _result_.
// 4. Throw a TypeError exception.

// 翻译：默认按 [valueOf, toString] 顺序（hint=number/default）
//       或 [toString, valueOf] 顺序（hint=string）
//       依次调用，第一个返回原始值的就用。
//       都失败抛 TypeError。

关键洞察：JS 用户能通过 3 种方式控制对象怎么转原始值：

  1. 定义 [Symbol.toPrimitive] 方法（最优先，能根据 hint 返回不同值）
  2. 覆盖 valueOf（数学/比较场景）
  3. 覆盖 toString（字符串场景）

ToBoolean / ToNumber / ToString —— 一表速查

每个原始类型在每个转换的结果（§7.1.2 / §7.1.4 / §7.1.17）：

类型 ToBoolean ToNumber ToString
undefined false NaN ⚠️ "undefined"
null false 0 "null"
Boolean 原值 true→1, false→0 "true" / "false"
Number 0/NaN→false, 其他→true 原值 "123" 等
BigInt 0n→false, 其他→true TypeError ⚠️ "123" （不带 n）
String ""→false, 其他→true parsable→数字, 否则 NaN 原值
Symbol true TypeError ⚠️ TypeError（要用 String(s)）
Object true（包括 new Boolean(false)） ToPrimitive(o, "number") 后再转 ToPrimitive(o, "string") 后再转

反直觉点：

  • Number(undefined) === NaN（不是 0！）
  • Number(null) === 0（不是 NaN！）
  • Boolean(new Boolean(false)) === true（对象 always true，包装类的"假"被淹没）
  • Number(1n) 抛 TypeError（BigInt 不能隐式转 Number，要用显式 Number(1n) 才行——但跨界仍危险）

ToInt32 / ToUint32 —— 位运算的隐式转换

位运算（|、&、^、~、<<、>>、>>>）在 JS 里是 32-bit 整数运算——不管输入是什么 Number，都先 ToInt32 / ToUint32 截到 32 位整数：

javascript · 位运算的隐式转换 
// ToInt32 流程（§7.1.6）：
// 1. ToNumber(value)
// 2. NaN/±0/±∞ → 0
// 3. 取符号 + 绝对值，向 0 取整
// 4. 模 2^32，再调整到 [-2^31, 2^31) 区间

// 实际行为：
3.7 | 0;                  // 3（向 0 取整 —— 比 Math.trunc 快的写法）
"42" | 0;                 // 42（先 ToNumber 再 ToInt32）
"abc" | 0;                // 0（NaN → 0）
2 ** 32 | 0;              // 0（4294967296 mod 2^32 = 0）
2 ** 31 | 0;              // -2147483648（溢出到负数区）
true | 0;                 // 1（Boolean → Number → ToInt32）

// 这就是为什么 |0 是历史悠久的"取整"技巧 —— 比 Math.floor 快，
// 且对超大数会"环绕"（modular arithmetic）。但只能处理 ≤ 32 位整数。

// ToUint32（§7.1.7）：把负数补码后映射到 [0, 2^32) 区间
-1 >>> 0;               // 4294967295（无符号右移 0 = ToUint32 的常用技巧）

// Array.length 的范围就是 [0, 2^32 - 1]，用的就是 ToUint32 检查。

IsLooselyEqual（== 的 11 步） ⭐

这是规范最反直觉的算法。每一步都是为兼容 1995-2009 年间的真实 web 代码服务。完整流程：

spec · §7.2.13 IsLooselyEqual ( x, y ) 
// 1. If Type(x) is the same as Type(y), then
//    a. Return IsStrictlyEqual(x, y).
//    （同类型走 === 算法）
//
// 2. If x is null and y is undefined, return true.
// 3. If x is undefined and y is null, return true.
//    （null/undefined 互等，是唯二跨类型相等）
//
// 4. NOTE: This step is replaced in section B.3.7.2.
//    （Annex B 的 document.all hack —— 浏览器才有）
//
// 5. If x is a Number and y is a String, return ! IsLooselyEqual(x, ! ToNumber(y)).
// 6. If x is a String and y is a Number, return ! IsLooselyEqual(! ToNumber(x), y).
//    （String 和 Number 比 → 把 String 转 Number 再比）
//
// 7. If x is a BigInt and y is a String, then
//    a. Let _n_ be StringToBigInt(y).
//    b. If _n_ is undefined, return false.
//    c. Return ! IsLooselyEqual(x, _n_).
// 8. If x is a String and y is a BigInt, return ! IsLooselyEqual(y, x).
//    （BigInt vs String：尝试把 String 解析成 BigInt）
//
// 9. If x is a Boolean, return ! IsLooselyEqual(! ToNumber(x), y).
// 10. If y is a Boolean, return ! IsLooselyEqual(x, ! ToNumber(y)).
//     （Boolean → Number 后再比）
//
// 11. If x is either a String, a Number, a BigInt, or a Symbol and y is an Object,
//     return ! IsLooselyEqual(x, ? ToPrimitive(y)).
// 12. If x is an Object and y is either a String, a Number, a BigInt, or a Symbol,
//     return ! IsLooselyEqual(? ToPrimitive(x), y).
//     （原始值 vs Object → ToPrimitive(Object) 后再比）
//
// 13. If x is a BigInt and y is a Number, or if x is a Number and y is a BigInt, then
//     a. If x is not finite or y is not finite, return false.
//     b. If ℝ(x) = ℝ(y), return true; otherwise return false.
//     （BigInt vs Number：转成数学实数比较）
//
// 14. Return false.

"反直觉案例的根因"：

javascript · == 反直觉解析 
// "" == 0
//   step 5: ToNumber("") === 0 → IsLooselyEqual(0, 0) → true ✅

// "0" == false
//   step 10: ToNumber(false) === 0 → IsLooselyEqual("0", 0)
//   step 6: ToNumber("0") === 0 → IsLooselyEqual(0, 0) → true ✅

// [] == false
//   step 10: ToNumber(false) === 0 → IsLooselyEqual([], 0)
//   step 12: ToPrimitive([]) === "" → IsLooselyEqual("", 0)
//   step 5: ToNumber("") === 0 → IsLooselyEqual(0, 0) → true ✅

// [] == ![]
//   ![] === false（[] 是 truthy，所以 !truthy 是 false）
//   再走上面 [] == false 的路径 → true ⚠️

// null == undefined → step 2/3 直接 true
// null == 0 → 没走特殊规则，类型不同 → false（!）

// NaN == NaN
//   step 1: 同类型，走 IsStrictlyEqual → IsStrictlyEqual 第 5 条 NaN ≠ NaN → false ⚠️

实际工程：永远用 ===，唯一例外是 x == null（同时检查 null 和 undefined，依赖 step 2/3 特殊规则）。

迭代器操作族（§7.4）

每个 for...of、...spread、new Map(iter) 背后都在调这些抽象操作。规范用它们统一处理"可迭代对象"。

操作 用途
GetIterator(obj, hint) 取迭代器。hint = sync/async
IteratorNext(iter, value) 调 iter.next(value)，返回 IteratorResult Record
IteratorComplete(result) 取 result.done
IteratorValue(result) 取 result.value
IteratorClose(iter, completion) 提前退出迭代时调 iter.return() 清理
IteratorStep(iter) 组合操作：next + 检查 done

关键场景：for (const x of iter) { if (cond) break; } ——break 时会调 IteratorClose（让 generator 走到 finally / 自定义清理）。这是规范级保证，不是引擎实现细节。

类型谓词（§7.2）

谓词 检查什么
IsCallable(v) v 有 [[Call]] 内部方法（即可调用）
IsConstructor(v) v 有 [[Construct]]（即可 new）
IsArray(v) v 是 Array exotic 或 Proxy 包装的 Array（跨 Realm 工作！）
IsRegExp(v) v 是 RegExp，或有 Symbol.match 属性（鸭子类型）
IsPropertyKey(v) v 是 String 或 Symbol
IsIntegralNumber(v) v 是有限整数 Number

关键洞察：IsCallable 和 IsConstructor 是独立的——箭头函数有 [[Call]] 但没有 [[Construct]]，所以 new (() => {}) 抛 TypeError。class 有两者，普通 function 也有两者。

03 应用场景

场景 A · 写自定义 Symbol.toPrimitive 实现

写一个 Money 类，希望它在 + * / 算术里返回数字、在字符串拼接里返回带货币的格式化串：

javascript · Money 类 
class Money {
  constructor(amount, currency) {
    this.amount = amount;
    this.currency = currency;
  }
  [Symbol.toPrimitive](hint) {
    if (hint === 'number') return this.amount;
    if (hint === 'string') return `${this.amount.toFixed(2)} ${this.currency}`;
    return `${this.amount.toFixed(2)} ${this.currency}`;  // default 也用字符串
  }
}

const price = new Money(99.5, 'USD');
+price;            // 99.5 (hint='number')
`${price}`;        // "99.50 USD" (hint='string')
price + ' for VIP'; // "99.50 USD for VIP" (hint='default' → 走字符串)
price * 2;         // 199 (hint='number')

// 如果不实现 [Symbol.toPrimitive]，会走 OrdinaryToPrimitive 调 valueOf/toString —— 通常返回 "[object Object]"

这是 Date 对象支持算术（new Date() - new Date('2024-01-01')）的同款机制。

场景 B · 调试 === 不工作的根因

"明明两个值看起来一样，=== 返回 false" 的常见原因：

  1. 有 NaN：用 Number.isNaN(x) 或 Object.is(x, y) 处理
  2. 跨 Realm（iframe / vm）：两个 Object 来自不同 Realm，=== 仍可能 true（同一个引用），但 arr instanceof Array 会 false（不同 Realm 的 prototype）—— 详见 A3 执行模型
  3. +0 vs -0：Object.is(-0, 0) 是 false 但 === 是 true
  4. 对象身份 vs 结构：两个不同的 { a: 1 } === 是 false——要结构相等用 lodash _.isEqual

诊断流程：① 先看 Object.is —— 如果它和 === 不一致，说明涉及 NaN 或 +0/-0；② 再看 typeof —— 类型不同直接 false；③ 再看 x && x.constructor === y && y.constructor —— 跨 Realm 的话不同。

场景 C · 读 lodash _.isEqual 源码理解 SameValueZero

lodash 的 _.isEqual(a, b) 做深度结构比较，叶子节点用 SameValueZero（+0 = -0，NaN = NaN），中间走属性枚举递归。源码大致结构：

javascript · 简化版 isEqual 
function isEqual(a, b) {
  // SameValueZero —— 处理 NaN/+0/-0
  if (a === b) return true;
  if (a !== a && b !== b) return true;  // 都是 NaN

  // 类型不同直接 false
  if (typeof a !== typeof b) return false;
  if (typeof a !== 'object' || a === null || b === null) return false;

  // 深度递归
  const keysA = Reflect.ownKeys(a);
  const keysB = Reflect.ownKeys(b);
  if (keysA.length !== keysB.length) return false;

  return keysA.every(k => isEqual(a[k], b[k]));
}

// 实际 lodash 还要处理：
// - Date / RegExp / Map / Set / Buffer 等特殊类型
// - 循环引用（用 WeakMap 记录已访问对）
// - Symbol key
// 这些都是规范层 SameValueZero 之上的工程加固

读 lodash 这种"无数 corner case 工业级实现"前先理解规范层的 SameValueZero / Object Equality 概念，源码读起来快很多。

对照规范读 IsArray 算法

spec · §7.2.2 IsArray 完整算法 
// IsArray ( argument )
// 1. If argument is not an Object, return false.
// 2. If argument is an Array exotic object, return true.
// 3. If argument is a Proxy exotic object, then
//    a. Perform ? ValidateNonRevokedProxy(argument).
//    b. Let target be argument.[[ProxyTarget]].
//    c. Return ? IsArray(target).
// 4. Return false.

// 翻译：
// - 普通对象不是 Array → false
// - Array exotic 对象（含跨 Realm 的）→ true ✅
// - Proxy 包装的 Array → 递归看包装的 target ✅
//
// 这就是为什么 Array.isArray 能跨 Realm 工作 —— 它只看「是不是 Array exotic」，
// 不查原型链，不依赖某个 Realm 的 Array.prototype。

// 对比 instanceof 失败的原因：
// arr instanceof Array  → 检查 Array.prototype 是否在 arr 的原型链上
// 跨 Realm 的 arr 的原型链指向 otherRealm.Array.prototype，不等于本 Realm 的 Array.prototype
// 所以 false ⚠️

05 常见陷阱

⚠️ 陷阱 1：Number(undefined) 期望 0

Number(undefined) === NaN，Number(null) === 0。两者反对称。

规避：用 Number(x) || 0 或者 +(x ?? 0) 提供 fallback。

⚠️ 陷阱 2：== 期待"宽松相等"，结果跟 ToPrimitive 顺序相关

obj == 0 时，obj 怎么转成原始值？取决于它的 valueOf / toString 实现。如果它返回 string，会再走一遍 String/Number 比较。

规避：永远用 ===。如果你真要"宽松相等"，显式转换：Number(obj) === 0。

⚠️ 陷阱 3：箭头函数被 new 抛 TypeError

new (() => {})() → TypeError: () => {} is not a constructor。原因：箭头函数没有 [[Construct]] 内部方法，IsConstructor 返回 false。

规避：知道 IsCallable 和 IsConstructor 是独立的。能调用不等于能 new——箭头函数、method() {} 简写、bound function、Generator 都是 callable 但不是 constructor。

⚠️ 陷阱 4：BigInt 不能和 Number 隐式混算

1n + 1 抛 TypeError: Cannot mix BigInt and other types, use explicit conversions。规范层是故意这样设计的——避免精度丢失（Number 有精度上限，BigInt 没有）。

规避：明确转换：BigInt(1) + 1n 或 1n + BigInt(1)。但 BigInt → Number 仍可能丢精度，跨界要谨慎。

⚠️ 陷阱 5：用 instanceof Array 检查跨 Realm 数组

来自 iframe / Worker / vm 的数组，instanceof Array 失败（不同 Realm 的 prototype）。Array.isArray 用 IsArray 抽象操作，跨 Realm 工作。

规避：检查"这是不是数组"统一用 Array.isArray(x)，永远不要用 x instanceof Array。
