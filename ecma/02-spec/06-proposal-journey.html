B6 · 提案 → 标准的旅程（Case Study）

B1-B5 是"读规范的工具"。本章是综合实战——把这些工具用在三个真实提案上，看一个特性如何从"GitHub 上一个 markdown 想法"走到"ECMA-262 §X.Y.Z 写入正文"。本章选三个 case：① Top-Level Await（成功案例，跨 §9.10 + §27 多 clause 修改的复杂提案）② Records & Tuples（卡 Stage 2 三年的反例，揭示提案"进不去"的真实阻碍）③ Iterator Helpers（ES2025 顺畅推进的最近案例）。读完这章，你既能评估"要不要用 Stage 3 特性"，也能给 TC39 提 PR，更能读懂 polyfill 为什么这样写。

📐 这是 ECMAScript 主题 Track B 的收官章

读到这里你已经掌握：① B1 算法记法 ② B2 内部数据类型 ③ B3 抽象操作 ④ B4 内部槽与方法 ⑤ B5 执行上下文模型。本章是把这些 mental model 用在真实历史上——从一个个具体的 Stage 0 → Stage 4 路径中印证你已经学到的概念。

阅读时长 ~35 分钟 前置 B1-B5（必读）+ A1（TC39 流程历史）+ A5（Stage 流程详解） 权威来源 tc39/proposals

01 速览

三句话定义本章：

  1. 提案旅程不是"线性流水线"，是"共识 + 实现验证 + 反复修订"的迭代过程。Stage 卡很久的提案常常不是技术问题，是社区分歧（语义、心智、工程影响）。
  2. "成功"和"失败"提案的最大差异不是技术复杂度，是「是否动摇核心心智」。Top-Level Await 让 Module 求值变成异步——但保持了"import 顺序确定"的核心心智，所以 5 年走完。Records & Tuples 引入"按值相等的对象"——动摇了 ES "对象按引用比较" 的 30 年心智，所以 3 年没进。
  3. 实施 timeline 比规范本身更慢。Stage 4 ≠ 所有引擎已支持。Top-Level Await 进 Stage 4 后，V8 当周支持，SpiderMonkey 用了 3 个月，JSC 用了 6 个月，Node 集成又用了 4 个月。"原生支持"的实际时间线 ≈ Stage 4 + 6-12 个月。

本章三个 case 速览：

提案 起点 当前 Stage 关键复杂性
Top-Level Await 2014 Stage 4 (2022) 跨 §9.10 + §27.2 + §16 多 clause 修改、cycle + async 死锁检测
Records & Tuples 2017 Stage 2（卡 7+ 年） 新原始类型、按值相等、GC / typeof / === 心智冲突
Iterator Helpers 2017 Stage 4 (2024) 把 Iterator 升级到一等公民、Lazy evaluation、向后兼容

02 底层逻辑

提案旅程的标准模式

每个进 Stage 1+ 的提案都有同样的仓库结构（tc39 GitHub 模板）：

tc39/proposal-* · 标准仓库结构 
tc39/proposal-top-level-await/
├── README.md            ← 高层介绍 + 当前状态 + Champion 列表
├── EXPLAINER.md         ← 详细的「问题 + 方案」叙事（给非 TC39 成员看）
├── spec.html            ← ecmarkup 写的 spec text（最终会合并进 ECMA-262）
├── polyfill/            ← 参考实现（如果可 polyfill）
├── test/                ← Test262 风格测试用例
├── meetings/            ← TC39 各次会议讨论记录链接
└── implementations.md   ← 各引擎实现状态

Champion（推动者）是关键——TC39 成员中有人愿意在每两个月一次的 plenary 会议上反复推动这个提案。没有 champion 的提案永远停在 Stage 0/1。

ecmarkup（tc39.es/ecmarkup）是写 spec text 的工具——把 HTML + 自定义标签（<emu-clause>、<emu-alg> 等）渲染成规范文本。ECMA-262 全文用它生成，每个提案的 spec.html 也用同款语法——这样进 Stage 4 时直接 merge。

Case 1 · Top-Level Await ⭐（成功案例）

问题：ES2015 引入 ESM 时，模块顶层不能 await——你必须包一个 async function。这意味着"异步初始化"代码（fetch config、连接数据库、加载 WASM）必须写成额外的"调用入口"。

javascript · 没有 TLA 时的模式 
// ❌ 不能这样写（pre-ES2022）
const config = await fetch('/api/config').then(r => r.json());
export default config;

// ✅ 只能这样
let _config;
export const initConfig = async () => {
  if (_config) return _config;
  _config = await fetch('/api/config').then(r => r.json());
  return _config;
};
export const getConfig = () => _config; // 可能返回 undefined ⚠️

跨 clause 修改的复杂性

TLA 不是"加一个语法"，是修改了 §9 + §16 + §27 三大块。让我们看具体改了什么（用 B5 学到的概念）：

spec · TLA 涉及的 clause 修改 
// ① §9.10 Source Text Module Records 修改：
//    - 加 [[HasTLA]] 字段（本模块是否含 TLA）
//    - 加 [[AsyncEvaluation]] 字段（异步求值状态）
//    - 加 [[AsyncEvaluationOrder]]（cycle 中的 settle 顺序）
//    - 加 [[TopLevelCapability]]（顶层 evaluate 返回的 Promise）
//    - 改 LinkRequestedModules 算法支持 cycle + async 检测
//    - 改 Evaluate 算法变成异步（返回 Promise 而非同步完成）
//
// ② §16.2.1.5 InitializeEnvironment 修改：
//    - module evaluation 从 sync execution 改为返回 Promise
//
// ③ §13 Expressions 修改：
//    - AwaitExpression 在 Module 顶层（除 Script）也合法
//
// ④ §27.2 Promise Objects 修改：
//    - PromiseResolveThenableJob 配合 module evaluation 的新协议
//
// 共修改 ~15 处算法步骤，加 ~600 行新文本。
// 这是「单一特性跨多个 clause 修改」的典型 —— 普通提案改 1-3 处。

关键观察：从 Stage 4（2021-04）到所有主流引擎都支持，用了约 6 个月。所以"我可以写 TLA 了"的实际时间是 2021 年底，不是 ES2022 发布的 2022-06。

Case 2 · Records & Tuples ⚠️（卡 Stage 2 反例）

问题：JS 的对象 / 数组按引用比较——{ a: 1 } === { a: 1 } 是 false。但很多场景需要"结构相等"（Redux state 比较、React props memo、深 immutable）。社区方案（Immutable.js、structuredClone-based equality）都是性能或语法妥协。

提案：引入新原始类型 Record (#{}) 和 Tuple (#[])：

javascript · Records & Tuples 设计目标 
// 字面量语法（带 # 前缀）
const r1 = #{ name: 'Alice', age: 30 };
const r2 = #{ name: 'Alice', age: 30 };
r1 === r2;     // true!（按值相等）

const t1 = #[1, 2, 3];
const t2 = #[1, 2, 3];
t1 === t2;     // true!

// 深度 immutable
r1.name = 'Bob';   // TypeError（不可变）

// 嵌套保持原始性
const nested = #{ inner: #[1, 2] };
typeof nested;     // "record"
typeof nested.inner; // "tuple"

// 与 Object / Array 互转
const obj = Object(r1);    // 转成 Object
const tup = #[...arr];     // Array 转 Tuple（仅当元素都是原始值）

时间 Stage 事件

4 大阻碍详解

① GC 实现复杂

  • "结构相等"要求 hash 用结构而非引用——每次创建一个 Record 都要算结构 hash
  • 引擎要维护"已存在的相等 Record 共享内存"机制（hash-cons）—— 否则 #{a:1} === #{a:1} 实现起来很慢
  • 这种 hash-cons 表本身要 GC，且和现有 V8 / SpiderMonkey 的对象内存模型不兼容

② === 心智冲突

  • 30 年来 JS 心智："=== 检查引用相等"。突然引入"有些类型 === 检查结构相等"——这是语义分裂
  • 用户将看到 {} === {} 是 false 但 #{} === #{} 是 true。不一致带来心智成本

③ typeof 新值争议

  • Records 是新原始类型 → typeof r === "record"？还是 "object"？还是 "tuple"？
  • 30 年的 typeof === "object" 检查代码会全部不正确处理 Record
  • "向后兼容性"和"语义清晰"的两难

④ 与 React 等框架的心智冲突

  • React 用 "对象按引用变化触发 re-render"——Record 不再是引用语义，破坏 useState/useMemo 直觉
  • "setState({...state, x: 1})" 在 Record 下变成"setState(#{...state, x: 1})"，但相等性意义完全不同
  • 框架社区对此有明确反对意见（Dan Abramov 等公开质疑）

当前替代方案

  • Immer ——"用可变 API 写不可变更新"，工程上事实标准
  • Immutable.js（Facebook）—— 老牌方案，性能好但 API 不直观
  • structured cloning + deep equal（lodash _.isEqual）—— 简单但性能差
  • Object.freeze 浅冻结 + 工程纪律 —— 最务实的"假" immutability

给提案的启示：技术上可行的提案不一定能进规范。需要：① 不动摇核心心智 ② 框架社区不强烈反对 ③ 引擎实现可行 ④ 不存在更轻量的库方案。Records & Tuples 4 项都不满足，所以卡了 7 年。

Case 3 · Iterator Helpers（最近顺畅案例）

问题：ES2015 引入 Iterator 协议（A2 设计原则 详述）—— 但 Iterator 自身没有 .map / .filter / .toArray 等方法。要用必须先 spread 成数组 [...iter].map(...)，失去了懒求值（spread 已经把全部值算出来了）。

javascript · Iterator Helpers 之前 vs 之后 
// 之前：必须先 spread —— 全量内存，失去懒求值
const result = [...generator()]
  .filter(x => x > 0)
  .map(x => x * 2)
  .slice(0, 10);

// ES2025 Iterator Helpers：链式 + 懒求值
const result = generator()
  .filter(x => x > 0)
  .map(x => x * 2)
  .take(10)
  .toArray();
// take(10) 后只算前 10 个，generator() 不会全跑完

// 配套的还有 .reduce / .forEach / .some / .every / .find / .flatMap
// 还有静态方法 Iterator.from(iterable) 把任意 iterable 转 Iterator

时间线

时间 Stage 事件
2017-12 Stage 1 Gus Caplan 提案 (Node.js core 维护者)
2018-2021 Stage 2 设计反复（关键决策：是否包括 async iterator helpers？最终先做 sync 版）
2022-09 Stage 3 规范文本完整
2023-2024 Stage 3 V8、SpiderMonkey、JSC 陆续实现
2024-12 Stage 4 所有引擎齐备 + Test262 全过
2025-06 — 纳入 ES2025（第 16 版）

为什么这次顺畅

  • 不动摇任何心智：只是给现有 Iterator 加方法，老代码完全不受影响
  • polyfill 简单：纯 JS 就能 polyfill（core-js 早已支持）—— 风险可预测
  • 工程价值显著：每个 JS 工程师都用过 [...gen()].map(...)，对懒求值有直接需求
  • 跨引擎共识：V8 团队主动推动，其他引擎跟进顺利

对比 Records & Tuples：Iterator Helpers 是"给已有抽象加便利方法"，Records & Tuples 是"引入全新数据类型"。前者风险低，后者风险高。提案的"工程友好度"≈ 它对现有代码的破坏程度的反比。

03 应用场景

场景 A · 团队评估"要不要用 Stage 3 提案"

React 项目想用 AsyncContext（Stage 2，2025-04）做隐式 trace ID 传递——这值得用吗？决策框架：

  1. Stage 2 是上限警戒线——可能永远不进，或语义大改。禁止生产用 Stage 0-2 提案语法
  2. Stage 3 + 引擎都已实现—— 可以用，但配 polyfill + 关注更新
  3. Stage 4 + 1 年—— 可假设原生支持，但仍要看 baseline 浏览器
  4. 查 polyfill 风险——能 polyfill？polyfill 体积？polyfill 100% 还原原生？
  5. 查框架社区态度——React/Vue 等明确反对的提案，即使进规范也不要押注

AsyncContext 当前评估：Stage 2，未进 Stage 3，暂不用。等到进 Stage 4 + 至少 V8/SpiderMonkey 实现后再回看（预计 2027+）。同期的替代方案：用 OpenTelemetry SDK 提供的"fake AsyncContext" polyfill。

场景 B · 给 TC39 提 PR 的入门路径

想给 ECMAScript 提个想法（如 Array.prototype.partition）——怎么做？

  1. 先去 es.discourse.group（社区讨论区）发帖，看社区反应
  2. 如果有兴趣，找一个 TC39 成员 champion 你的提案（Bloomberg / Microsoft / Mozilla / Apple / Google 等公司都有 representatives）
  3. 建 GitHub repo，按 tc39 模板写 README + EXPLAINER + spec.html
  4. 在 TC39 plenary（2 个月一次）present，争取 Stage 1
  5. 反复迭代 spec text，找引擎团队反馈，争取 Stage 2/3/4

大多数提案在 Stage 0-1 就被否决——常见原因：① 已有库方案 ② 语义不清晰 ③ 不符合 ES 设计哲学。读 tc39 inactive-proposals 的退役提案是免费的"怎么避免不被接受"教材。

场景 C · 解读为什么这个 polyfill 这样写

读 core-js 源码看 Iterator Helpers polyfill：

  • 为什么要新建一个 %IteratorPrototype% 而不是改 Array.prototype？—— 因为提案明确加在 Iterator 上而非 Array，向后兼容
  • 为什么 .map 返回 lazy iterator 而不是 Array？—— 提案要求懒求值，全量执行违反语义
  • 为什么有 IteratorClose 处理 break？—— 提案要求 break 时调用 generator 的 .return() 让 finally 跑（B3 讲过）

读 polyfill 时对照原提案 spec text能解释每一行决策。这是 B 轨学完后能做到的"用规范读代码"。

读 spec text 的实战

spec · Iterator.prototype.map（ES2025）实际规范 
// 27.1.4.1.5 Iterator.prototype.map ( mapper )
// 1. Let O be the this value.
// 2. If O is not an Object, throw a TypeError exception.
// 3. If IsCallable(mapper) is false, throw a TypeError exception.
// 4. Let iterated be ? GetIteratorDirect(O).
// 5. Let counter be 0.
// 6. Let closure be a new Abstract Closure with no parameters that
//    captures iterated, mapper, and counter and performs the following
//    steps when called:
//    a. Repeat,
//       i. Let value be ? IteratorStepValue(iterated).
//       ii. If value is done, return undefined.
//       iii. Let mapped be Completion(Call(mapper, undefined, « value, F(counter) »)).
//       iv. IfAbruptCloseIterator(mapped, iterated).
//       v. Let completion be Completion(Yield(mapped)).
//       vi. IfAbruptCloseIterator(completion, iterated).
//       vii. Set counter to counter + 1.
// 7. Let result be CreateIteratorFromClosure(closure, "Iterator Helper", %IteratorHelperPrototype%).
// 8. Set result.[[UnderlyingIterator]] to iterated.
// 9. Return result.

// 用 B 轨学到的能力解读：
// - 步骤 1-3：标准的 this 检查 + IsCallable（B3）
// - 步骤 4：GetIteratorDirect 是新抽象操作（不调 Symbol.iterator，直接用 O 当迭代器）
// - 步骤 6：Abstract Closure（B2）—— 规范层的"匿名函数"，捕获 iterated/mapper/counter
// - 步骤 6.iv / 6.vi：IfAbruptCloseIterator 处理 abrupt completion + 调 IteratorClose（B3）
// - 步骤 7：返回新 Iterator exotic（B4）

// 全程使用 B1-B5 的术语和概念。
// 读懂这段 = Track B 学习成果验收。

05 常见陷阱

⚠️ 陷阱 1：以为 Stage 4 = 所有引擎已实现

不是。Stage 4 要求 ≥2 个引擎实现，不要求所有。如 ES2024 的 Promise.withResolvers Stage 4 时仅 V8 + SpiderMonkey 实现，JSC 用了 4 个月才支持。

规避：查 kangax compat-table 看具体引擎支持情况。"原生支持"的实际时间 ≈ Stage 4 + 6-12 个月。

⚠️ 陷阱 2：以为 polyfill 能 100% 还原原生行为

部分提案无法 polyfill：

  • Top-Level Await —— 影响 Module 求值，无法在 polyfill 层模拟
  • Records & Tuples —— === 行为靠引擎深度集成，polyfill 只能用方法（.equals()）模拟
  • Optional Chaining ?. —— 这是语法，必须用 Babel 等 transpile
  • Symbol.dispose / using —— 部分能 polyfill，但 using 语法仍需 transpile

规避：判断"这个提案能否 polyfill"——是新 API（如 Promise.try）就能 polyfill；是新语法（如 ?.、using）就必须 transpile；是涉及引擎深度集成（如 TLA、Records）就无法 polyfill。

⚠️ 陷阱 3：把 inactive proposals 当现存特性

tc39/proposals 仓库有 inactive-proposals.md—— 已退役、不再推进的提案（如 Number.fromString、RegExp.escape 早期版本）。读到这些提案不要以为它们存在。

规避：任何 Stage 信息都看 tc39/proposals/README.md 当前主表（含 finished + active + inactive 分类），不只看个别提案的 README。

⚠️ 陷阱 4：把 Stage 进度等同于"应该用"

Stage 4 的提案不一定是"好特性"——只是社区共识 + 引擎实现都到位了。有些 Stage 4 特性后来被发现"设计有遗憾"（如 ES5 的 __proto__ getter/setter 进了 Annex B），但已无法回退。

规避：用 Stage 4 特性时仍要判断它是否解决你的问题。不是所有"标准特性"都值得用——with 语句、arguments.callee 等已成事实禁用。
