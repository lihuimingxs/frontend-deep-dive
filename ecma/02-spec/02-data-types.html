B2 · 数据类型与值（§6）

ECMA-262 §6 把"类型"分为两大类：Language Types（用户可见的 8 种：Undefined / Null / Boolean / String / Number / BigInt / Symbol / Object）和 Specification Types（规范内部的：Reference Record / Completion Record / Property Descriptor / Environment Record / Data Block / Abstract Closure 等）。Specification Types 不是 JS 值——你写不出 typeof x === 'CompletionRecord'——但它们是规范文本里反复出现的"参数 / 返回值 / 中间结果"，不理解它们就读不懂 (0, obj.x)() 为什么丢 this，也读不懂 ? 简写的真正含义。

📐 本章交叉规则

8 个 Language Types 在 JS · 数据类型系统 已详述（含装箱/拆箱、ToPrimitive 完整链路）。本章对 Language Types 做1-3 句话短重述 + 链回，正文聚焦 Specification Types——这些是 JS 主题没有覆盖的"规范层数据结构"。

阅读时长 ~30 分钟 前置 B1 · 怎么读规范（必读 ? / ! 简写） 对应 ECMA-262 §6 ECMAScript Data Types and Values

01 速览

三句话定义本章：

  1. Language Types 8 种：Undefined / Null / Boolean / String / Number / BigInt / Symbol（7 个原始）+ Object（1 个引用）。这是 typeof 看到的那些（不过 typeof null === 'object' 是历史 bug，typeof function 是 'function' 是规范选择）。
  2. Specification Types 是"规范的内部脚手架"——你看不到，但每个表达式都用到。最重要的两个：Reference Record（描述"引用一个 binding"，是 obj.x 的中间结果，决定 this 推断和左值赋值）和 Completion Record（所有抽象操作的返回值，是规范的"异常机制"）。
  3. String 在规范里是 UTF-16 code unit 序列，不是 Unicode 序列。所以 "💩".length === 2、"💩"[0] 是孤立 surrogate；要正确处理 emoji 必须用 Intl.Segmenter 或 Array.from(str)（基于 code point 迭代）。

02 底层逻辑

§6.1 Language Types（8 种）

每种类型在规范里有一节专门定义其取值集合 + 内部表示。短重述 + 链回：

类型 spec § 核心要点
Undefined §6.1.1 唯一值 undefined。表示"未初始化" / "缺省值"
Null §6.1.2 唯一值 null。"故意为空"。typeof null === 'object' 是 1995 年实现 bug，向后兼容无法修复
Boolean §6.1.3 true / false。简单
String §6.1.4 UTF-16 code unit 序列（关键），不是 Unicode 字符串。详见下文专节
Symbol §6.1.5 唯一不可重复的值。Well-known Symbols 在 §6.1.5.1（详见下文）
Number §6.1.6.1 IEEE-754 双精度浮点。+0 / -0 / NaN / ±∞ 是规范级别的特殊值（详见下文）
BigInt §6.1.6.2 任意精度整数（ES2020）。和 Number 不能混算，1n + 1 TypeError
Object §6.1.7 属性集合 + 11 个 essential internal methods（详见 B4）+ 内部槽集合

详细机制请参考 JS · 01-fundamentals/01-types（含装箱/拆箱完整链路、ToPrimitive 用法）。下面 3 节系统补全 JS 主题没展开的"规范层细节"。

Number 内部 —— IEEE-754 的细节

ECMA-262 §6.1.6.1 把 Number 类型精确定义为 IEEE 754-2019 binary64（双精度浮点），加上规范层的几个特殊值：

  • +0 与 -0 —— IEEE-754 区分这两个值，但 === 视它们相等。Object.is(+0, -0) 返回 false。规范里很多算法会区分（如 Math.atan2、1/x 在 +0 是 +∞ 在 -0 是 -∞）。
  • NaN —— 唯一一个不等于自己的值（NaN === NaN 是 false）。规范要求所有 NaN bit pattern 视为同一值（IEEE-754 允许多个 NaN bit pattern 但 ECMAScript 认为它们都是 NaN）。
  • +∞ 与 -∞ —— 算术运算溢出时产生（1/0、Math.exp(1000)）。
  • safe integer 范围：Number.MAX_SAFE_INTEGER === 2^53 - 1 === 9007199254740991。超过这个范围的整数无法精确表示（2^53 + 1 === 2^53），需要用 BigInt。
  • EPSILON：Number.EPSILON ≈ 2.22e-16，是 1 与下一个可表示的双精度浮点数之间的差。比较浮点相等用 Math.abs(a - b) < Number.EPSILON（仅小数值附近精确）。

对应不同相等比较算法：

算法 用在 +0 vs -0 NaN vs NaN
=== (IsStrictlyEqual) 大部分场景 相等 不等
Object.is (SameValue) 需精确区分 不等 相等
SameValueZero Map/Set 的 key 比较、Array.includes 相等 相等

这就是为什么 [NaN].includes(NaN) 是 true（用 SameValueZero）但 [NaN].indexOf(NaN) 是 -1（用 IsStrictlyEqual）。历史包袱。

String 内部 —— UTF-16 而非 Unicode ⚠️

这是 JS 字符串最常被误解的事实。ECMA-262 §6.1.4 规定：

⚠️ "JavaScript String 是 UTF-16 code unit 序列"

不是 Unicode 字符序列，不是 UTF-8 字节序列。每个 String 是一个不可变的 16-bit code unit 数组。str.length 返回 code unit 数，str[i] 返回单个 code unit（字符串）。

code unit vs code point：Unicode 把每个字符（grapheme）映射到一个 code point（U+0000 到 U+10FFFF 共 1114112 个）。UTF-16 用 16 bit 编码：

  • BMP 内字符（U+0000 - U+FFFF，65536 个）：1 个 code unit 表示，str.length === 1 ✅
  • BMP 外字符（U+10000 - U+10FFFF，emoji / 罕见汉字 / 部分 CJK 扩展）：用一对 surrogate pair（高代理 + 低代理），2 个 code unit，str.length === 2 ⚠️
javascript · code unit 陷阱 
"a".length;             // 1（BMP）
"中".length;            // 1（BMP）
"💩".length;            // 2（surrogate pair！）
"👨‍👩‍👧".length;       // 8（家庭 emoji 是 4 字符 + 3 个零宽连接符）

"💩"[0];                // '\uD83D'（孤立的高代理，无意义）
"💩".charCodeAt(0);     // 55357（高代理）
"💩".codePointAt(0);    // 128169（完整 code point）✅

// 切割 emoji 的不同方式
[..."💩"];               // ['💩']  ← spread/iterator 按 code point
Array.from("💩");        // ['💩']  ← 也是按 code point
"💩".split("");          // ['\uD83D', '\uDCA9']  ← 按 code unit ⚠️

// 复杂 emoji 仍要用 Intl.Segmenter（grapheme cluster 级别）
const seg = new Intl.Segmenter("en", { granularity: "grapheme" });
[...seg.segment("👨‍👩‍👧")];   // [{ segment: "👨‍👩‍👧", index: 0, ... }]  ← 1 个！

历史原因：JavaScript 1995 年起就用 UCS-2（16-bit 固定宽度），UCS-2 在 1996 年被 UTF-16 取代——但 ECMA-262 规范没有跟着升级，至今 String 类型本质仍是"UCS-2 + 程序员自己处理 surrogate pair"。这是 30 年来无法弥补的设计债务。

规范层补救：

  • ES2015：codePointAt、String.fromCodePoint、for...of 按 code point 迭代、正则 u flag
  • ES2018：正则命名捕获 + Unicode property escape（\p{Emoji}）
  • ES2024：正则 v flag（Unicode 集合操作，更强的 emoji 支持）
  • ES2024：String.prototype.isWellFormed() / toWellFormed() 检测/修复孤立 surrogate

Symbol & Well-known Symbols

短重述：Symbol 是唯一不可重复的原始值，每次 Symbol() 调用产生新 Symbol。Well-known Symbols 是规范预定义的"协议钩子"，让用户对象参与语言行为（如 Symbol.iterator 让对象可被 for...of 迭代）。

ECMA-262 §6.1.5.1 列出 14 个 Well-known Symbols：asyncIterator / hasInstance / isConcatSpreadable / iterator / match / matchAll / replace / search / species / split / toPrimitive / toStringTag / unscopables / dispose（ES2025）/ asyncDispose（ES2025）。

详细用法请参考 JS · 01-fundamentals/11-symbol。A2 设计原则 也讲了 Symbol 作为"扩展点"的设计哲学。

§6.2 Specification Types ⭐

这是本章最关键的部分。Specification Types 不是 JS 值，是规范用来描述算法的"内部数据结构"。规范定义了 7 种：

类型 spec § 用途
List §6.2.2 有序的值序列。规范层的"数组"
Record §6.2.3 命名字段的集合。"数据类"。[[Field]] 双方括号访问
Completion Record ⭐ §6.2.4 所有抽象操作的返回类型。规范的"异常机制"
Reference Record ⭐ §6.2.5 表达式的"未求值的引用"中间结果。决定 this 推断、左值赋值
Property Descriptor §6.2.6 对象属性的元信息（writable / enumerable / configurable / value 或 get/set）
Environment Record §9.1 变量绑定的存储单元（详见 A3 / B5）
Abstract Closure §6.2.7 规范层的"匿名函数"，捕获若干规范层变量
Data Block / Shared Data Block §6.2.9 ArrayBuffer / SharedArrayBuffer 的规范表达（连续字节序列）

下面专门拆 Reference Record 和 Completion Record——这两个是规范文本里出现频率最高的。

Reference Record（最重要的"看不见类型"）⭐

问题：当你写 obj.x，规范是怎么处理的？答：obj.x 不是值，是一个 Reference Record——它"记录了一个指向某 binding 的引用"，必须通过 GetValue 才能拿到实际值。

Reference Record 含 4 个字段（§6.2.5）：

字段 含义
[[Base]] 引用的目标对象（或 unresolvable / Environment Record）
[[ReferencedName]] 属性名（字符串或 Symbol）
[[Strict]] 当前引用是否在 strict 代码里
[[ThisValue]] "super.x" 引用时的 this 绑定（普通引用为 empty）

关键概念：表达式 obj.x 求值时不立刻取值，先产生 Reference Record { Base: obj, ReferencedName: "x", ... }。然后取决于怎么用它：

  • 读取场景：let v = obj.x → 调 GetValue(ref) → 返回 obj.[[Get]]("x", obj)
  • 赋值场景：obj.x = 1 → 调 PutValue(ref, 1) → obj.[[Set]]("x", 1, obj)。左值赋值之所以可能，就是因为 Reference Record 保留了 base + name
  • 方法调用场景：obj.x() → 关键：调用规范知道 obj.x 是 Reference Record 且 Base 是对象 → 调用时自动把 this 设为 ref.[[Base]]（即 obj）
  • 普通调用场景：const f = obj.x; f() → const f = obj.x 时已 GetValue，f 是普通值 → 调用时 this 是 undefined（strict）或 globalThis（sloppy）
🎯 经典之谜：(0, obj.x)() 为什么丢 this

理解 Reference Record 之后这个谜题秒解：

javascript · Reference Record 的语义 
const obj = {
  name: 'Alice',
  greet() { console.log('Hi, ' + this.name); }
};

obj.greet();           // Hi, Alice
//   ↑ obj.greet 是 Reference Record { Base: obj, Name: 'greet' }
//     调用时检测到是 Reference 且 Base 是对象 → this = obj

(0, obj.greet)();      // Hi, undefined （strict 模式抛错）
//   ↑ 逗号表达式：先求值 0（丢弃），再求值 obj.greet（这一步对 Reference 调 GetValue）
//     → 逗号表达式的结果是 GetValue 后的"普通函数值"，不再是 Reference
//     → 调用时 this = undefined（strict）

const f = obj.greet;
f();                   // 同样丢 this（赋值时 GetValue）

// 这是 minifier 的常用技巧 —— "(0, fn)()" 强制丢 this，
// 在某些场景比 fn.call(undefined) 更紧凑。

核心规则：this 来自 Reference Record 的 [[Base]]。任何让表达式从"Reference"退化成"值"的操作（赋值、传参、逗号、解构）都会丢 this。这就是为什么 React 函数组件用 const onClick = this.handleClick 然后传给 onClick 时丢 this（class 组件经典坑）。

Completion Record（异常机制的规范化）

每个抽象操作的返回值都是 Completion Record——规范用它统一表达"正常完成 + 中断（异常）"。

Completion Record 含 3 个字段（§6.2.4）：

字段 含义
[[Type]] 5 种之一：normal（正常）/ throw（抛错）/ break（循环跳出）/ continue（循环继续）/ return（函数返回）
[[Value]] 携带的值（normal 是返回值，throw 是异常对象，return 是返回值，break/continue 通常是 empty）
[[Target]] break/continue 的标签（用于跳出多层循环），其他类型为 empty

"abrupt completion"：除 normal 之外的 4 种统称。这就是 B1 讲的 ? 简写处理的对象——? 检查返回的 Completion Record，如果是 abrupt（throw/break/continue/return）就把它原样向上传播。

规范记法 vs Completion Record 完整展开 
// 规范写：
// 1. Let _v_ be ? Get(_O_, "x").

// 完整展开：
const completion = Get(O, "x");
//   completion 类型是 Completion Record { [[Type]], [[Value]], [[Target]] }
if (completion.[[Type]] !== 'normal') {
  return completion;  // ← abrupt 直接传出去
}
const v = completion.[[Value]];

// "?" 简写省掉了 Completion Record 的 wrap/unwrap 样板，
// 直接对成功值取 [[Value]]，失败时整个抽象操作 return 这个 Completion。

// 这就是为什么 try/throw 的传播是"自动的" —— 规范层每一步都在传 Completion。

break / continue / return 的特殊性：它们是 abrupt completion 但不是异常——规范用同一个机制把"循环跳转"和"函数返回"也建模成 abrupt。break 沿语法节点向上传播，遇到对应的循环就停；return 遇到函数边界就停。这是规范统一"非线性控制流"的做法。

Property Descriptor（4 种形态）

Property Descriptor 是对象属性的元信息——你写 obj.x = 1 时，规范层在 obj 内部存储一个 PropertyDescriptor 描述这个属性。Object.getOwnPropertyDescriptor(obj, "x") 暴露的就是它（部分字段）。

4 种形态（§6.2.6）：

形态 含字段 用例
Data Descriptor [[Value]] + [[Writable]] + [[Enumerable]] + [[Configurable]] 普通属性 obj.x = 1
Accessor Descriptor [[Get]] + [[Set]] + [[Enumerable]] + [[Configurable]] get x() {} / set x(v) {}
Generic Descriptor 仅 [[Enumerable]] + [[Configurable]]，不指定 value 或 get/set "只改可枚举性，不改值" 场景
Fully Populated Data 或 Accessor 之一，所有字段都填（无 absent） 规范内部产生的"完整描述符"

"absent" 字段：descriptor 字段可以是 absent（缺省），不等于 undefined。Object.defineProperty(obj, "x", { writable: false }) 时 value 是 absent，意思是"保留原 value 不变，只改 writable"。{ value: undefined, writable: false } 是不同的——明确把 value 设为 undefined。

Data Block / Shared Data Block

ArrayBuffer / SharedArrayBuffer 的规范层表达——一段连续的字节序列，长度不可变。

  • Data Block（§6.2.9）：单 Agent 内的字节数组，ArrayBuffer 的内部
  • Shared Data Block：多 Agent 共享的字节数组，SharedArrayBuffer 的内部。访问需要 Atomics 保证原子性（详见 A4 异步演进）

这两个类型支撑 TypedArray（Int32Array 等）在规范层的实现——TypedArray 是 Exotic Object（详见 B4），它的 [[GetOwnProperty]] 等内部方法直接操作底层 Data Block。

03 应用场景

场景 A · 写 emoji-safe 字符串切割

做截断显示（如 username 最长 8 字符）时，str.slice(0, 8) 在 emoji 上会切坏（切到 surrogate pair 中间）。正确做法：

javascript · 三种粒度的截断 
const s = "Hello 👨‍👩‍👧 World";

// ❌ code unit 级（错的）
s.slice(0, 9);                            // "Hello 👨‍👩"  会切坏家庭 emoji

// ✅ code point 级（适合大多数 emoji）
[...s].slice(0, 8).join("");              // "Hello 👨‍👩‍👧"（部分情况）
Array.from(s).slice(0, 8).join("");

// ✅ grapheme cluster 级（最准确，但需要 Intl.Segmenter）
const seg = new Intl.Segmenter("zh-CN", { granularity: "grapheme" });
const graphemes = [...seg.segment(s)].map(x => x.segment);
graphemes.slice(0, 7).join("");           // "Hello 👨‍👩‍👧"

// 何时用哪个：
// - 简单 ASCII / 中文 → slice 够用
// - 一般 emoji → spread / Array.from
// - 复杂 emoji（家庭、肤色变种、国旗）→ Intl.Segmenter

这背后就是"String 是 UTF-16 序列"的代价。

场景 B · 实现 deep clone 处理 PropertyDescriptor

写一个 deepClone，要保留 getter/setter、不可枚举属性、Symbol key。这就要求逐属性复制 Descriptor，不能只用 Object.assign 或 spread（它们只复制可枚举的 Data 属性）。

javascript · descriptor-aware deep clone 
function deepClone(obj, seen = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj;
  if (seen.has(obj)) return seen.get(obj);

  const proto = Object.getPrototypeOf(obj);
  const clone = Object.create(proto);
  seen.set(obj, clone);

  // 拿到所有自有属性（含不可枚举 + Symbol key）
  const keys = [...Object.getOwnPropertyNames(obj), ...Object.getOwnPropertySymbols(obj)];

  for (const key of keys) {
    const desc = Object.getOwnPropertyDescriptor(obj, key);
    // desc 是 Data 还是 Accessor？
    if ('value' in desc) {
      desc.value = deepClone(desc.value, seen);   // 递归复制值
    }
    // get/set 不递归（函数不深克隆）
    Object.defineProperty(clone, key, desc);
  }
  return clone;
}

// 这就是 lodash _.cloneDeep / structuredClone 内部的简化版（实际更复杂）。

理解 Property Descriptor 4 种形态后，能解释为什么浅 clone 会丢 getter、为什么 freeze 不可逆——它们都在操作 descriptor。

场景 C · 调试"this 丢失"

React class 组件经典坑："事件 handler 里的 this 是 undefined"。根因是 Reference Record 退化：

javascript · 三种 this 丢失场景 
class Button extends React.Component {
  handleClick() {
    console.log(this.props);
  }

  render() {
    // ❌ 1. 赋值丢 this：onClick 拿到的是 GetValue(ref) 的普通值
    return <button onClick={this.handleClick} />;

    // ✅ 2. bind 显式绑：返回新函数，this 锁死
    return <button onClick={this.handleClick.bind(this)} />;

    // ✅ 3. 箭头函数包：this 沿外层 lexical 找
    return <button onClick={() => this.handleClick()} />;
    // 这里 onClick 收到的是箭头函数，调用时 this 是箭头函数定义时的外层 this（即 Button 实例）

    // ✅ 4. class field（最优）：直接定义箭头方法
    // handleClick = () => { console.log(this.props); }
  }
}

理解了 Reference Record 后这些场景都有清晰的解释：① 是 GetValue 退化；② 是手动重建一个绑定 this 的函数；③/④ 是箭头函数没自己的 this。诊断 this 丢失的根问题就是"这里有没有把 Reference 转成 value"。

04 典型代码

用 Object.is 区分 +0 和 -0 的实战

javascript · 容易被忽略的浮点细节 
+0 === -0;                  // true（IsStrictlyEqual 视为相等）
Object.is(+0, -0);          // false（SameValue 区分）

// 实际意义：1 / x 在 +0 是 +∞，在 -0 是 -∞
1 / +0;                     // Infinity
1 / -0;                     // -Infinity

// Math.atan2 也区分
Math.atan2(0, +0);          // 0
Math.atan2(0, -0);          // π （3.14159...）

// 写归一化函数处理用户传入的 -0
function normalize(n) {
  return Object.is(n, -0) ? 0 : n;
}

// 写 deep equal 时处理 NaN
function dataEquals(a, b) {
  // === 处理大部分 case，但 NaN 失败
  if (a === b) return true;
  // SameValueZero 处理 NaN
  return Number.isNaN(a) && Number.isNaN(b);
}

// Map / Set 用 SameValueZero 比较 key
const m = new Map();
m.set(NaN, 'a');
m.set(NaN, 'b');             // 覆盖（NaN === NaN under SameValueZero）
m.size;                      // 1
m.get(NaN);                  // 'b'

对照规范读 ToObject 算法

spec · §7.1.18 ToObject ( argument ) 
// 规范文本：
// "1. If argument is undefined or null, throw a TypeError exception.
//  2. If argument is a Boolean, return a new Boolean object whose [[BooleanData]] internal slot is set to argument.
//  3. If argument is a Number, return a new Number object whose [[NumberData]] internal slot is set to argument.
//  4. If argument is a String, return a new String object whose [[StringData]] internal slot is set to argument.
//  5. If argument is a Symbol, return a new Symbol object whose [[SymbolData]] internal slot is set to argument.
//  6. If argument is a BigInt, return a new BigInt object whose [[BigIntData]] internal slot is set to argument.
//  7. Assert: argument is an Object.
//  8. Return argument."

// 用 B1 + B2 学到的能力解读：
// - 步骤 1 throw 一个 Completion Record { Type: throw, Value: TypeError }
// - 步骤 2-6 创建一个 Object 类型值，含一个内部槽存储原始值（装箱）
// - 步骤 7 是 Assert（不变量）—— 经过前 6 步，剩下的必然是 Object
// - 步骤 8 直接 return（已经是 Object 不需要包装）

// JS 等价：
function ToObject(argument) {
  if (argument == null) throw new TypeError('Cannot convert undefined or null to object');
  if (typeof argument === 'boolean') return new Boolean(argument);
  if (typeof argument === 'number') return new Number(argument);
  if (typeof argument === 'string') return new String(argument);
  if (typeof argument === 'symbol') return Object(argument);
  if (typeof argument === 'bigint') return Object(argument);
  // assert(typeof argument === 'object' || typeof argument === 'function')
  return argument;
}

// 这就是 obj.x 中"obj 是原始值时怎么取属性"的桥梁
// "abc".length 工作：内部先 ToObject("abc") → new String("abc") → 拿 .length

05 常见陷阱

⚠️ 陷阱 1：以为 String.length 是字符数

"💩".length === 2，"𠮷".length === 2（罕见汉字也是 surrogate pair）。这是常见的"字数限制"实现错误——后端按字符校验、前端按 length 校验，导致用户能输入但提交失败。

规避：要数字符用 [...str].length（code point 计数）或 Intl.Segmenter（grapheme 计数）。前后端校验对齐到同一个粒度。

⚠️ 陷阱 2：用 === 区分 +0 / -0

+0 === -0 是 true，区分要用 Object.is。常见错误：归一化代码 if (n === -0) return 0; 永远不进。

规避：明确知道这两个相等比较算法的差异——大部分场景用 ===，有特殊需求（区分 +0 / -0、把 NaN 视为相等）用 Object.is。

⚠️ 陷阱 3：不知道 Reference Record，看不懂 (0, fn)() 的语义

读 minifier 输出会看到大量 (0, m.fn)()、(void 0, fn).call(...) 等模式，不理解就觉得是混淆。

规避：理解 Reference Record——任何让表达式从 Reference 退化成 value 的操作都会丢 this。读源码看到这种模式时，直接判断为"明确丢 this"。这在某些场景比 fn.call(undefined) 更紧凑（少一次调用）。

⚠️ 陷阱 4：以为 Property Descriptor 没有 absent 字段

Object.defineProperty(obj, "x", { writable: false })——这里没传 value，意思是"保持 value 不变"。如果你以为 absent === undefined，会以为它把 value 设为 undefined。结果是你看不到的 absent 状态。

规避：写 PropertyDescriptor 操作时，区分 absent（缺省）和 undefined（明确设为 undefined）。Reflect.getOwnPropertyDescriptor 返回的 descriptor 总是"完整"的（无 absent），是排查时的黄金工具。

⚠️ 陷阱 5：把 Number 范围当无限大整数

2 ** 53 + 1 === 2 ** 53 —— 超过 Number.MAX_SAFE_INTEGER 的整数会丢精度。常见踩坑场景：后端返回 64-bit 整数 ID（如 Twitter snowflake、银行账号、订单号），前端 JSON.parse 后已经丢了精度。

规避：① 后端返回大整数时用 string；② 或前端用 BigInt 处理（JSON.parse 不会自动转，需自定义 reviver）；③ Postgres / MySQL 的 BIGINT 类型务必序列化为字符串发给前端。
