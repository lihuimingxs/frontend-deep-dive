编写约定

每章遵循统一的 6 段结构：速览 → 底层逻辑 → 应用场景 → 典型代码 → 常见陷阱 → 延伸阅读。但 ECMA 主题相比 JS / TS 有以下额外约定：

  • 「短重述 + 链回」 —— 当一个机制在 JavaScript 主题已详述时（如 Promise 状态机、闭包内存模型、TDZ），本主题用 1-3 句话点到本质，紧跟一句"详细机制请参考 X"，避免读者反复跳跃
  • 规范字段精确名称 —— Track B 严格使用规范术语：OrdinaryGet 而非"属性查找"；[[Prototype]] 而非 __proto__（除非讨论可见 API）
  • 历史决策标注动机 —— Track A 每个设计决策都标"当时有哪些选项、TC39 怎么选的、根因是什么"；不只罗列特性
  • ⚠️ 标注 —— ES4 永远标"abandoned 2008"；Annex B 永远标"web 兼容性遗留"；Stage 0-2 提案永远标"可能永远进不了标准"

与既有主题的关系

ECMA 主题不重复 JS 主题已经讲透的运行时机制，但会在每个交叉点提供"短重述 + 链接"。常见对应：

  • A2 设计原则 → JS 类型系统 / 原型链 / 闭包
  • A3 执行模型 → JS 作用域与 TDZ / this 四规则
  • A4 异步演进 → JS P3 · 异步 & 错误处理（全 8 章）
  • B 轨的规范术语反查 → JS / TS 章节中提到的 §X.Y.Z 都能在 Track B 找到对应章节
