标准发展史 ——「为什么存在？」

ECMAScript 不是某个公司的产品，是一份从 1997 年发布至今共 16 版的国际标准（ECMA-262）。它的演化分三个阶段：1995-2004 标准化与僵局（出生即僵局，ES4 大爆炸式失败）→ 2004-2015 重生（Chrome+V8 引爆性能、HTML5 围攻 Flash、ES5 救场）→ 2015-至今 年度小步快跑（TC39 流程改革，每年一版）。理解这条线之后，许多"JS 为什么这么古怪"的问题就有了根因——不是设计糟糕，是历史让它别无选择。

阅读时长 ~30 分钟 前置 任意 JavaScript 经验 对应 ECMA-262 §4 Overview

01 速览

三句话定义本章：

  1. ECMAScript 是 JavaScript 的核心语言规范。两者不是同义词：JavaScript 是 Netscape 的商标 + 浏览器实现 + DOM/BOM 等宿主 API；ECMAScript 只是其中"语言核心"那一份标准化文档。
  2. 从 1997 至今共 16 版正式发布，但没有 ES4——第 4 版编辑号被官方废弃（2003-2008 年的 ES4 提案因厂商分裂被放弃）。这次失败是 TC39 流程改革的根因。
  3. 2015 是分水岭：ES2015（也叫 ES6）从语言基础上重新塑造了 JS（class、let/const、模块、Promise、Symbol、Proxy）；同时 TC39 把发布节奏从"大爆炸"切到"年度小步快跑 + Stage 流程"，从此每年 6 月一版。

16 版完整目录（写作时点 2026-04）：

02 底层逻辑

史前史（1995）—— 10 天写出的脚本

1995 年 5 月，Netscape 决定给浏览器加一种"轻量胶水语言"——用来给页面加个时钟、做个表单校验，不打算和 Java 抢编程语言地位。Brendan Eich 在 10 天内写出第一版原型，叫 Mocha。9 月改名 LiveScript，12 月又改名 JavaScript——纯粹是营销决策：当时 Java 正热，Netscape 与 Sun 达成协议，借 Java 的名字蹭流量。

这个改名的代价持续了 30 年：JavaScript 在语义上和 Java 几乎没关系（原型 vs 类、动态 vs 静态、弱类型 vs 强类型），但 Java 风格的 C 系语法外衣（{} 块、; 结尾、function 声明、new、throw）让无数初学者误以为它是"前端的 Java"——事实上它是 Self（原型）+ Scheme（一等函数）+ 一层 Java 语法皮肤。这个设计基因在 A2 设计原则 章节会详细展开。

第一阶段 · 1995-2004 · 标准化与僵局

1996 年微软发布 IE 3.0 + JScript（不能用 "JavaScript" 这个名字，避开 Sun 商标）。两个实现差异迅速扩大，开发者必须写"if (Netscape) ... else ..."的双版本代码——这就是后来"浏览器战争（Browser Wars）"的开端。

1996 年 11 月，Netscape 把 JavaScript 1.1 提交给 Ecma International（一个总部在日内瓦的国际标准组织，前身 ECMA = European Computer Manufacturers Association）。Ecma 立刻成立 TC39（Technical Committee 39）专门起草这门语言的标准。命名上不能叫 "JavaScript"（商标在 Sun 手里），不能叫 "JScript"（微软的私货），妥协选了 ECMAScript——一个所有人都嫌丑、所有人都能接受的中性名字。

1997-06 发布 ES1（92 页），1998-08 发布 ES2（仅文字校对）。1999-12 发布 ES3，加了正则、try/catch、do-while、严格相等 ===——这一版稳定了大约 10 年成为浏览器实现的事实基线。

但 TC39 内部从 2003 年开始酝酿一个极度激进的 ES4：要加 class、模块、namespace、强类型注解、generics，本质上想把 JS 改造成"小型 Java"。这个方向从一开始就有分裂——Mozilla / Adobe（ActionScript 3 已经实现了类似设计）支持，微软强烈反对（认为变化太大、破坏向后兼容、削弱 IE 在 Web 上的话语权），Yahoo 的 Douglas Crockford 也持保留态度。

⚠️ ES4 的失败（2003-2008）

ES4 的失败不是技术问题，是共识问题。提案过于庞大，厂商无法独立验证实现可行性；微软不愿冒"全网兼容性破裂"的风险。2008-07 在 Oslo 会议上，TC39 正式宣布放弃 ES4，编辑号永久跳过，下一版直接是 ES5。同时形成新流程共识："Harmony"——后续提案必须按小步快跑、有共识才推进。

这段经历是 TC39 至今所有流程设计的"创始创伤"。今天的 Stage 0-4 流程、"必须有 ≥2 个引擎独立实现"才能进 Stage 4 的硬性规定，都是 ES4 教训的结晶。任何一个想"一次升级整个语言"的提案都会被这个机制拦下。

第二阶段 · 2004-2015 · JS 重生

2004 年 Firefox 发布，结束了 IE 在浏览器上的实质垄断。同年 Gmail / Google Maps 上线，AJAX 模式（XMLHttpRequest + 异步 UI 更新）从此成为前端默认架构——JS 第一次承担"做完整应用"的责任，不再是"加个时钟"的玩具。

但真正改变 JS 命运的是 2008-09 Chrome + V8 发布。V8 引入 JIT 编译（hidden class、inline caching）把 JS 性能提升 10-100 倍。这之前 Mozilla / Apple / 微软的 JS 实现都把"性能差"视为天然限制，V8 用工程证明这是错的。从此各厂商进入 JS 性能竞赛——Mozilla 推出 SpiderMonkey + TraceMonkey + IonMonkey，Apple 把 JavaScriptCore 升级到 SquirrelFish + Nitro，Microsoft 的 Chakra 完全重写。

2009-12 ES5 发布，没有 ES4 的激进改动，但加了几个"解锁现代 JS"的关键能力：

  • 'use strict' —— 第一次允许"显式排除掉历史遗留的危险语义"
  • JSON 内置对象 —— 之前都靠手写 eval('(' + str + ')')，是 XSS 重灾区
  • Object.create / Object.defineProperty —— 把原型操作和属性描述符暴露给用户
  • Function.prototype.bind —— 解决 this 丢失
  • 属性描述符（Property Descriptor） —— writable/enumerable/configurable，从此对象有了"细颗粒度"控制

2010 年 HTML5 进入主流（Apple 推 iPad，刚好不支持 Flash；Jobs 公开发表 Thoughts on Flash，断言 HTML5 才是 Web 的未来），Apple + Google 联手围攻 Adobe Flash。HTML5 + CSS3 + JS 成为"前端三件套"的最终形态，至此 JS 的角色从"页面胶水"升级为"Web 平台的事实主语言"。

第三阶段 · 2015-至今 · 年度小步快跑

2015-06 发布 ES2015（仍叫 ES6 是因为它跨越了 ES4 的废弃编辑号）。这是有史以来最大的一次升级，引入：

  • class（原型语法糖）+ let / const（块作用域 + TDZ）
  • ES Modules（import/export）—— JS 第一次有原生模块
  • Promise + Iterator/Generator —— 异步编程的语言级抽象
  • 箭头函数（词法 this）+ 解构 + 模板字符串 + 默认参数 + 剩余/展开
  • Symbol + Map / Set / WeakMap / WeakSet + Proxy / Reflect

但 ES2015 太大，编写时间长达 6 年（2009→2015）。TC39 从这次起改革流程——再也不做"大版本"，改为：

  1. 每年 6 月发布一版，包含所有当年达到 Stage 4 的提案
  2. Stage 流程（0-4）让每个特性独立推进
  3. 提案进 Stage 4 必须有 ≥2 个引擎独立实现 + Test262 测试通过——即"实现先行，规范后写"
📐 TC39 Stage 0-4 速查
  • Stage 0 · Strawperson（任何想法或讨论）—— 没门槛，任何人可提
  • Stage 1 · Proposal（正式提案）—— 有 champion、问题描述清楚、初版 API 草图
  • Stage 2 · Draft（首选方案）—— 初始规范文本（ecmarkup 写）、关键决策定下
  • Stage 2.7 · Candidate（候选实现）—— 完整规范文本、Test262 测试案例
  • Stage 3 · 推荐实现 —— 各引擎可以开始实现 + 反馈
  • Stage 4 · Finished —— ≥2 引擎实现、Test262 全过、纳入下一年度标准

Stage 3 ≠ "可以放心用"。Stage 3 时 polyfill 才靠谱，但浏览器原生还没；Stage 4 才能假设原生支持。详细案例见 提案旅程 case study。

这套机制运行至今 10 年（2015-2025），交付了 10 个年度版本。每年提案数量稳定在 5-15 个之间，有些"坐冷板凳"多年（如 Records & Tuples 卡 Stage 2 三年至今未动），有些则迅速推进（Optional Chaining 从 Stage 1 到 Stage 4 只用了 2 年）。这种稳定的、可预测的、共识驱动的节奏正是 ES4 失败的反义词。

三个绕不过去的根因

把上述时间线压缩成三个高密度的洞察：

① ES4 失败是 TC39 流程的"创始创伤"

今天的 Stage 流程、"≥2 个引擎实现"硬性规定、"小步快跑+共识驱动"的节奏，全部是 ES4 教训的产物。任何一个"想一次性升级整个语言"的提案都会被这个机制拦下——这是好事，是 JS 生态对历史的诚实回应。Records & Tuples（深 immutable 数据结构）从 2019 提到现在卡 Stage 2 没进，正是这个机制在发挥作用：还没有 2 个引擎独立实现 + 社区争议未平。

② "标准滞后实现"是常态，不是缺陷

很多人误以为"规范是先发明特性、引擎是后实现"。事实正相反——绝大多数特性是先在引擎或社区里普及，再追认到规范：

  • Promise 在 2012 年已被 jQuery $.Deferred、Q、Bluebird 等库广泛使用 → 2015 年才进入 ES2015 规范
  • async/await 在 2015 年 TypeScript 已实现 + Babel 已能 transpile → 2017 年才进入 ES2017 规范
  • Optional Chaining ?. 在 CoffeeScript / Babel-plugin / TypeScript 早已有 → 2020 年进入 ES2020
  • Array.prototype.includes lodash _.includes 早就存在 → 2016 年进入 ES2016

这个模式叫 "standardize what's already common practice"。规范的角色是事后追认社区共识，不是事先发明特性。这也解释了为什么提案要进 Stage 4 必须有 ≥2 个引擎独立实现——TC39 不接受"纸上设计"，必须有社区使用证据。

③ JS 没有"版本号"，只有"宿主支持的 ES 子集"

这是工程师最容易踩坑的认知。"我写的是 ES2020 代码"——这句话本身没意义。真正决定代码能不能跑的是运行时引擎支持哪些语法 + Runtime API：

  • 你写的 ?.（ES2020 语法）在 Node 14- 跑不起来 —— 不是因为"代码是 ES2020 的"，是因为 V8 早期版本没实现这个语法的 parser
  • 你写的 Array.prototype.includes（ES2016 API）在 Node 4- 跑不起来 —— 不是版本不匹配，是 Array.prototype 上没有这个方法
  • 同一份代码可以在 Bun 上原生跑（最新 V8），同时被 Babel 降级成 ES2015 给老 Safari 跑

所以"目标版本"决策只能基于实际运行环境——浏览器市场份额（用 caniuse / kangax / MDN BCD 查）+ Node 版本 + bundle 体积权衡。A5 现代生态（编写中）会给出"客户端保守 / 客户端激进 / 服务端"三种打包目标版本策略。

03 应用场景

场景 A · 老项目 polyfill 决策

你接手一个老 React 项目，browserslist 配的是 "> 1%, last 2 versions"，bundle 大小 800KB（其中 200KB 是 core-js）。要不要升级 browserslist 来减小 bundle？

理解了"语法 vs API"的区别就好做决策——?. / ?? / class fields 是语法，由 Babel/SWC transpile 处理（无运行时开销）；Array.prototype.includes / Object.fromEntries 是 API，必须 polyfill（占 bundle 体积）。把 baseline 从 ES2015 升到 ES2020，能砍掉 60% 的 core-js 内容，因为可选链、空值合并、动态 import 都已是浏览器原生支持的语法。

场景 B · 团队定 baseline

新项目启动时定 tsconfig.target 和 browserslist。给这两个值的依据是用户群分布，不是"我喜欢的版本":

  • To B SaaS / 内部系统：用户都用现代 Chrome/Firefox/Safari → ES2022 起步，激进可到 ES2024
  • To C 大众产品：长尾老设备多 → ES2020 是甜蜜点（覆盖率 99.9% + 主要现代特性都在）
  • 电商 / 政府类：可能还要兼容 IE 11 / 老 Android WebView → 必须 ES2015 + 大量 polyfill
  • Node 服务端：你能控制 Node 版本 → 直接 ES-Next（Node 22 LTS / Bun 全部支持）

具体决策树和打包配置见 A5 现代生态（编写中）。

场景 C · 给新人讲"为什么 JS 这么古怪"

新同事问"为什么 typeof null === 'object'"、"为什么 0.1 + 0.2 !== 0.3"、"为什么 this 这么难懂"——单独解释每个特性都是"语言古怪"，但放回历史框架就是"10 天写出来的脚本被全世界用了 30 年"。每个古怪都有一个历史原因：

  • typeof null === 'object' 是 1995 年 V1 实现的 bug，到 ES1 时已无法修复（向后兼容）
  • 浮点精度问题继承自 IEEE-754（这不是 JS 独有的，只是因为 JS 没有 int 类型，问题特别显眼）
  • this 难懂是因为它从 Java 借了语法但底层是 Self 的"无类对象"——语法层和语义层错配

把"语言古怪"重新框定为"历史决策"，新人对 JS 的态度会从"避雷"变成"理解"。这章是任何"前端 onboarding 文档"都该有的开篇。

04 典型代码

看一段代码穿越 30 年

同样是"取一个嵌套对象的字段并加 1"这个操作，在 ES3 / ES5 / ES2015 / ES2020 / ES2025 各自的写法演化：

ES3 · 1999 
// 没有 const、没有解构、没有可选链
var user = window.user;
if (user && user.profile && user.profile.age) {
  user.profile.age = user.profile.age + 1;
}
ES5 · 2009 —— 加了 strict mode 和属性描述符 
'use strict';

var user = window.user;
if (user && user.profile && typeof user.profile.age === 'number') {
  user.profile.age += 1;
}

// 还多了 Object.keys / forEach 等
Object.keys(user).forEach(function (key) {
  console.log(key, user[key]);
});
ES2015 · 现代 JS 起点 
const user = window.user;
const { profile: { age } = {} } = user ?? {};
if (typeof age === 'number') {
  user.profile.age = age + 1;
}

// for...of 替代 Object.keys + forEach
for (const [key, value] of Object.entries(user)) {
  console.log(key, value);
}
ES2020 · 可选链 + 空值合并 + 逻辑赋值 
// 一行解决
if (typeof user?.profile?.age === 'number') {
  user.profile.age += 1;
}

// 或者用空值合并初始化
const age = user?.profile?.age ?? 0;
ES2025 · Iterator Helpers + Set 集合方法 
// 不再需要 [...arr.entries()].filter(...).map(...)
const ages = Object.values(users)
  .map(u => u.profile?.age)
  .filter(a => typeof a === 'number')
  .toArray(); // Iterator Helpers

// Set 也有了集合运算
const adults = new Set(ages.filter(a => a >= 18));
const seniors = new Set(ages.filter(a => a >= 65));
const adultNonSeniors = adults.difference(seniors); // ES2025

感受 TC39 提案的 spec 化路径

以 Optional Chaining ?. 为例，看一个特性怎么从想法到标准：

演化时间线 
// 2014：CoffeeScript 已支持 a?.b
// 2015：TypeScript 实现 (PR #1144) 但未合并
// 2016：Babel plugin 出现 (@babel/plugin-proposal-optional-chaining)
// 2017-09：TC39 Stage 1 (Claude Pache, Gabriel Isenberg, Daniel Rosenwasser)
// 2018-05：Stage 2，确定语义（短路、不抛错）
// 2019-04：Stage 3，V8 + JSC 开始实现
// 2019-09：V8 7.7 / Chrome 80 / Node 14 原生支持
// 2020-04：Stage 4
// 2020-06：进入 ES2020 标准

注意时间分布：从 Stage 1 到 Stage 4 总共不到 3 年，但社区使用早在 2014 年（用 CoffeeScript / Babel）。规范走的是"追认共识"路线，不是"发明语法"。

05 常见陷阱

⚠️ 陷阱 1：以为存在 "ES4"

不存在 ES4。第 4 版编辑号在 2008 年被官方废弃，没有任何符合 ES4 标准的实现。如果你看到老书或文章提到 "ES4 features"，那指的是 已放弃的提案集合，不是任何真实存在的标准。这些提案中有一部分（class、模块、Promise）后来被改造成更小的提案，分散进了 ES5/ES2015/ES2017，但形式已和 ES4 时期完全不同。

规避：写文档/注释时永远不用 "ES4"；引用历史只说"ES4 abandoned 2008"。这是和 TC39 社区交流的标准说法。

⚠️ 陷阱 2：以为 ES2015 ≠ ES6

ES2015 和 ES6 是同一个东西，编号是 ECMA-262 第 6 版（因为跳过了 ES4，所以 ES5 之后是第 6 版）。命名混乱来自 TC39 在 2015 年同时切换了命名风格：

  • ES6 是按编辑号的旧称呼
  • ES2015 是按发布年份的新称呼
  • 两者并行使用至今，TC39 官方文档更倾向于年份命名，但口语和老资料里 "ES6" 仍然普遍

规避：写新文档统一用年份命名（ES2015 / ES2020 / ES2025），引用规范文档时用"ECMA-262 第 N 版"。"ES6" 仅在和老人交流或读老资料时用。

⚠️ 陷阱 3：把 "JavaScript" 和 "ECMAScript" 当同义词

不是同义词。它们的关系：

  • JavaScript = 商标（Sun → Oracle 持有，Netscape 授权 Mozilla 使用）+ 完整运行时（语言核心 + DOM + BOM + Web API）+ 通常指浏览器中的实现
  • ECMAScript = ECMA-262 标准文档定义的语言核心（语法 + 内置对象 + 抽象操作），不包括 DOM、不包括 fetch、不包括 setTimeout

setTimeout 不在 ECMAScript 规范里，它在 HTML 规范（WHATWG）里。fetch 在 Fetch 规范里。document.getElementById 在 DOM 规范里。ECMAScript 只管 Array.prototype.map 这种"语言层"的东西。

规避：讨论"语言特性"用 ECMAScript（或 ES2020 这种版本）；讨论"运行时能力"用 JavaScript 或具体平台名（Node.js、Bun、Chrome 122）。

⚠️ 陷阱 4：以为 "Stage 3 = 可以放心用"

Stage 3 ≠ 生产可用。Stage 3 意味着规范文本完整、各引擎可以开始实现，但 不一定所有引擎都已实现。Stage 3 提案最终也可能因实现反馈而被回退或修改（少数情况）。

规避：要用 Stage 3 提案，务必 polyfill + 测试，并准备好"提案变化时改代码"。Stage 4 才能假设原生支持，且原生支持还要看你的 baseline 浏览器。最稳妥的做法是用 Stage 4 + 至少 1 年缓冲（让浏览器市场份额追上）。
