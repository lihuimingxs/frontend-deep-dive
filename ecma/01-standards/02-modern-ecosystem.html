现代生态与目标版本 ——「我今天该用什么版本？」

A1-A4 讲了 ECMAScript 的"历史"和"机器"。本章把它们落到现代工程实战：TC39 Stage 流程怎么走、ES2016-ES2025 各版本加了什么、团队该用什么 ES 版本作为 baseline、polyfill 和 transpile 各处理什么、Stage 3 提案现在卡在哪。这是一份产品工程师的"把 ECMA 历史用起来"的速查指南——读完就能回答"我们项目该把 tsconfig.target 设成什么"这种实际问题。

📐 本章是 P1「标准与演化」的"怎么用"

前一章（标准发展史）告诉你 ECMA 是怎么演化的；下一章（提案旅程）告诉你新特性怎么进来的。本章夹在中间回答"我今天该用什么版本？"——读完之后你应该能给团队定一份合理的 ES baseline 方案，知道 polyfill 和 transpile 的边界在哪里。

阅读时长 ~25 分钟 前置 01 标准发展史（理解 16 版编辑号 + ES4 历史） 权威来源 tc39/proposals + kangax compat-table

01 速览

三句话定义本章：

  1. "目标版本"决策不是技术问题，是产品问题。它取决于用户群分布（caniuse 数据），不是"我喜欢哪个版本"。To B 内部系统可以激进 ES2024；toC 大众产品保守 ES2020。
  2. 语法 vs API 是必须分清的两件事。?. / ?? / class fields 是语法，由 Babel/SWC transpile 处理；Array.prototype.includes / Object.fromEntries 是 API，必须 polyfill（占 bundle 体积）。混淆这两个会做出错的优化。
  3. Stage 3 ≠ 生产可用。Stage 3 提案可能被回退，可能等 2 年才有第二个引擎实现。Stage 4 才能假设原生支持，且原生支持还要看你的 baseline 浏览器。

02 底层逻辑

TC39 Stage 流程详解

TC39 自 2014 年起用Stage 流程管理提案推进。每个 Stage 有明确的入口标准（提案达到什么状态可以晋级）和产出物（这个 Stage 必须有什么材料）。这是 ES4 失败之后的"共识驱动"治理机制（A1 详述）：

Stage 名称 入口标准 + 产出物
Stage 0 Strawperson
稻草人		 任何 TC39 成员可提出的"想法"。无门槛，可以是 Markdown 草稿。
Stage 1 Proposal
正式提案		 需要 champion（推动者）+ 问题描述清楚 + 初版 API 草图 + GitHub 仓库（基于 tc39 模板）
Stage 2 Draft
草案		 初始 规范文本（用 ecmarkup 写，跟规范同款语法）+ 关键决策定下 + 至少 1 个 polyfill 验证
Stage 2.7 Candidate
候选实现 ⭐ 2024 加		 完整规范文本 + 全套 Test262 测试案例 + 各引擎可以开始原型实现。新加的中间 stage，让"测试覆盖完整"成为强制门槛
Stage 3 推荐实现 规范文本无重大变更 + 各引擎全力实现 + 收集 implementer feedback。但仍可能因 feedback 回退到 Stage 2
Stage 4 Finished ≥2 个独立引擎已实现 + Test262 全过 + champion 提交合并请求。纳入下一年 6 月发布的标准

关键点：Stage 4 不等于"已纳入正式标准"，是"已通过所有门槛、待下个 6 月发布合并"。同时 Stage 4 ≠ 所有引擎已实现——只要 ≥2 个引擎实现就行，所以理论上 V8 + JSC 实现了，但 SpiderMonkey 还没——Firefox 用户用不了。这要查 kangax 或 MDN BCD。

ES2016-ES2025 全特性表

每年 6 月一版，特性按 Stage 4 时间排进对应年度。下表覆盖 10 个版本的核心特性：

版本 核心特性
ES2016
第 7 版		 幂运算 **、Array.prototype.includes —— 改革后第一个"小版本"，仅 2 个特性，象征"每年一版可以很轻"
ES2017
第 8 版		 async/await（最重磅）、SharedArrayBuffer + Atomics、Object.values / Object.entries、Object.getOwnPropertyDescriptors、String.prototype.padStart/padEnd
ES2018
第 9 版		 对象 Rest/Spread { ...a }、异步迭代 for await...of + async function*、Promise.finally、正则 dotAll s flag、命名捕获组、lookbehind
ES2019
第 10 版		 Array.flat/flatMap、Object.fromEntries、String.prototype.trimStart/trimEnd、Optional Catch Binding（catch {} 不写 binding）、Symbol.description、稳定排序保证
ES2020
第 11 版		 可选链 ?.、空值合并 ??、BigInt、dynamic import()、globalThis、Promise.allSettled、String.prototype.matchAll、import.meta、export * as ns
ES2021
第 12 版		 逻辑赋值 ??= ||= &&=、String.prototype.replaceAll、Promise.any + AggregateError、WeakRef + FinalizationRegistry、数字分隔符 1_000_000
ES2022
第 13 版		 Class Fields（含 #private 硬隐私）、Top-Level Await、Array.at()、Object.hasOwn、Error.cause、正则 d flag（indices）、private fields in 检查
ES2023
第 14 版		 findLast/findLastIndex、不可变数组方法（toSorted/toReversed/toSpliced/with）、Hashbang #!、Symbol 作为 WeakMap 的 key
ES2024

ES2025 ⭐

读这张表的姿势：不是按年份记，是按"解决了什么问题"分组。比如"异步"族（ES2017 async/await → ES2018 async iter → ES2020 dynamic import → ES2022 TLA → ES2024 withResolvers / waitAsync → ES2025 Promise.try），每族都对应一个长期演化主线。完整异步主线见 P7.3 Event Loop 与异步演进。

打包器目标版本策略（PPT 经典页）

PPT 第一章末尾给出过三种"打包器目标版本"策略，覆盖了几乎所有 web 工程场景。本节系统化展开：

策略 1 · 客户端保守 → ES2020

适用：面向公众的 Web 应用（电商、媒体、SaaS landing page）、需要长尾覆盖率（>99%）、SEO 重要场景。

为什么是 ES2020：可选链、空值合并、动态 import、BigInt 已 99.9%+ 普及（caniuse 2026-04 数据）；老 Safari 14、Chrome 80+、Firefox 74+ 都支持。低于 ES2020 的特性（如 ES2015 的 Map/Set、ES2017 async/await）也都安全。

配置：tsconfig.target: "ES2020" + browserslist: "> 0.5%, last 2 versions, not dead, not IE 11" + Babel/SWC 自动 transpile 高版本语法 + core-js 自动 polyfill 高版本 API。

策略 2 · 客户端激进 / Electron → ES2024

适用：To B 内部工具（用户都是公司员工 + 有 IT 控管浏览器版本）、Electron 桌面应用（控制 Chromium 版本）、Tauri 应用、admin dashboard。

为什么是 ES2024：用户环境可控，能砍掉 60% 的 polyfill（不需要降 class fields、TLA、optional chaining 等已成基线的特性）；bundle 显著变小；Top-Level Await 可用，模块级异步初始化更优雅。

配置：tsconfig.target: "ES2022" 或更高 + browserslist: "Chrome >= 119, last 2 Edge"（按团队约定锁定）+ Vite 默认或 SWC + 几乎不需要 polyfill。

策略 3 · 服务端 → ES-Next（Node 22+ / Bun）

适用：Node.js 后端、Bun 服务、Cloudflare Workers / Vercel Edge / Deno。

为什么是 ES-Next：你能完全控制运行时版本（在 package.json engines 字段或部署配置里锁定 Node 22 LTS / Bun latest），无需降级。Node 22 已支持几乎所有 ES2024 特性，Bun 跟进 ES2025。

配置：tsconfig.target: "ESNext" + tsconfig.module: "NodeNext" + 不用 Babel/SWC（用 tsc 或 esbuild 仅做类型擦除 + 路径解析）。不需要 core-js（运行时已自带所有 API）。

polyfill vs transpile —— 必须分清的两件事

很多人把这两个概念混着用，做出错误优化。它们处理的是不同问题：

维度 Transpile（语法降级） Polyfill（API 补丁）
处理什么 新语法编译成老语法 缺失的运行时 API 补回来
例子 ?. → (x && x.y)
class → function + prototype
async function → generator + state machine		 Array.prototype.includes 没有？补一个
Object.fromEntries 没有？补一个
Promise 没有？补一个
工具 Babel / SWC / TypeScript / esbuild core-js / regenerator-runtime / es-shims
开销 编译时（不进 bundle 但产物冗长） 运行时（占 bundle 体积，影响首屏）
无法处理的情况 新原始类型（BigInt、Symbol）—— 必须 runtime 支持 语法层特性（?.、TLA）—— polyfill 救不了

事实标准：Babel preset-env + core-js: 3 是"按 browserslist 智能 polyfill"的最成熟方案——你只配 browserslist，preset-env 自动决定哪些语法需要 transpile、哪些 API 需要 polyfill、哪些可以原生跳过。SWC 的 swc-helpers 跟进同样模式。理解这个机制比改 Babel 配置更重要——配错了 browserslist 就什么都歪了。

Feature Detection 工具

定 baseline 时必查的 3 个权威来源：

  • caniuse.com —— 按浏览器市场份额 + 各浏览器版本支持情况。最常用的"这个特性能用吗"工具。支持自定义 browserslist 然后看综合覆盖率。
  • kangax compat-table —— 各引擎按 ES 版本的实现进度，最权威的 ES 特性矩阵。看 V8 / SpiderMonkey / JSC / Bun / Node / Deno 的对比。
  • MDN Browser Compat Data (BCD) —— MDN 文档每个 API 页面底部的"Browser compatibility" 表，数据来源是社区维护的 @mdn/browser-compat-data。最细颗粒度（精确到方法/参数）。

这 3 个工具可以互相验证。决定一个 baseline 时，至少在两个工具上确认覆盖率，避免单源数据偏差。

Stage 3 提案当前快照（写作时点 2026-04）

这一节会随时间过期——但它展示了"当前社区在讨论什么"，能帮你理解 ES2026/ES2027 可能的样子。

提案 Stage 现状与展望
Temporal Stage 3 Date 的替代品，借鉴 Java java.time 设计。即时（Instant）、日历日期（PlainDate）、带时区时间（ZonedDateTime）、Duration 等正交类型。Firefox 已实现，Safari/Chrome 进度中。polyfill 体积大（~150KB），但解决了 Date 设计糟糕 30 年的痛点。
Decorator Metadata Stage 3 给 Stage 3 标准 decorator 加 metadata 反射能力。NestJS / TypeORM 等 DI 框架等的就是这个——有了它们才能彻底从 legacy decorator 迁移到标准 decorator。详细 decorator 历史见 TS · Decorators。
Records & Tuples Stage 2 深度不可变数据结构 #{ a: 1 } / #[1, 2]，按值相等。卡 Stage 2 三年——社区争议未平（GC 实现复杂、和 React useState 心智冲突、相等语义在嵌套时反直觉）。可能永远进不了。
Pipeline Operator |> Stage 2 类似 F# / Elixir 的管道：x |> f |> g 等价 g(f(x))。设计反复——F# style vs Hack style 争论 5 年，目前选 Hack style（带 placeholder %）但仍未通过。中等概率进入未来标准。
AsyncContext Stage 2 异步上下文跟踪——OpenTelemetry / Sentry 等 APM 工具等了 5 年的特性。让"sync 代码 → async 代码"也能传递 trace ID 等隐式上下文。Stage 3 概率较高。
ShadowRealm Stage 3 用户态创建独立 Realm（A3 讲过 Realm 概念）。Stage 3 多年，安全性争议（侧信道攻击）让 Mozilla 反对。可能拆分成 Web 平台 API 而非 ECMA 规范。

从这张表能学到的元规律：① Stage 卡很久 ≠ 没希望（async/await 也卡过 2 年）；② Stage 卡很久 ≠ 必然进（Records & Tuples 可能永远进不了）；③ "用户量大 + 实现简单"的提案推进最快（Optional Chaining 只用 3 年）；④ "心智冲突大 / 实现复杂 / 安全争议"的提案最可能死（Records & Tuples / ShadowRealm / Pipeline）。

03 应用场景

场景 A · 团队定 baseline 决策树

新项目启动，要回答三个问题：① tsconfig.target 用什么？② browserslist 怎么写？③ polyfill 工具链怎么配？决策树：

decision · baseline 选型 
// 第 1 步：用户群是什么？
// → 内部 / B 端控管 → 策略 2（ES2024 激进）
// → 大众 / 长尾设备 → 策略 1（ES2020 保守）
// → 仅服务端 → 策略 3（ES-Next）

// 第 2 步：caniuse 验证目标版本覆盖率
// 把要用的最高版本特性（如 TLA = ES2022）输入 caniuse，
// 配上自己的 browserslist 看综合覆盖率。
// 目标：>= 99% （toC）或 >= 95%（toB）

// 第 3 步：tsconfig 写
{
  "compilerOptions": {
    "target": "ES2020",         // ← 决策点 1
    "module": "ESNext",         // 留给打包器决定
    "moduleResolution": "bundler",
    "lib": ["ES2024", "DOM"],   // lib 写最高版本，让 TS 知道你想用什么 API
    "strict": true
  }
}

// 第 4 步：package.json + browserslist
{
  "browserslist": [
    "> 0.5%",
    "last 2 versions",
    "not dead",
    "not IE 11"
  ]
}

// 第 5 步：让 Babel/SWC 按 browserslist 自动 transpile + polyfill
// Babel: useBuiltIns: 'usage' + corejs: 3
// SWC: env.targets 用 browserslist
// Vite: build.target 用 browserslist 或 'esnext'

注意：target（编译输出版本）和 lib（类型声明可用 API）是两件事。target: "ES2020" 但 lib: ["ES2024", "DOM"]——意思是"我可以用 ES2024 API，但你帮我编译输出降到 ES2020 兼容"。

场景 B · 老项目升级（ES5 + Webpack 4 → ES2020 + Vite）

一个 5 年前的 React 项目，browserslist: "> 1%, last 2 versions, IE 11"，bundle 800KB（含 200KB core-js）。要升级到 ES2020 + Vite。步骤：

  1. 查 caniuse：你现在的用户里 IE 11 还有多少？低于 0.5% 就直接踢掉（IE 11 已在 2022 EOL）
  2. 更新 browserslist："> 1%, last 2 versions, not dead"——丢 IE 11 + 自动跟新浏览器
  3. 切到 Vite：npm create vite@latest + 迁移 webpack 配置（多数 plugin 有对应 vite 插件）
  4. 升级 core-js：从 2.x 到 3.x，useBuiltIns: 'usage'，让 Vite 按新 browserslist 重算
  5. 实测 bundle：典型砍掉 30-50% polyfill、首屏加载快 40%+
  6. 监控 SyntaxError：Sentry / 自建错误监控，看是否有用户跑不起来——有就回退 browserslist

关键点：升级 baseline 一定要监控线上错误。理论覆盖率 99% 也意味着 1% 用户白屏，对大流量产品就是几万人。

场景 C · 写新库决定 dist 目标版本

你写一个 npm 库（如自定义 hooks 集合），要决定 dist/ 输出哪个 ES 版本。两个选项：

  • 输出 ES2020（保守）—— 最大兼容，但用户的 bundler 还要再处理一次（如果用户 baseline 更高也无法删除）
  • 输出 ES-Next + 不 transpile（激进）—— 让用户的 bundler 自己降。这是现代 npm 库的事实标准（lodash-es、zod、tanstack 等都这样）

建议输出两份 dist：① dist/esnext/ 给现代 bundler；② dist/legacy/ 给老用户。配 package.json 的 exports 字段按条件分发。或者更简单——只输出 ESNext + ESM，老用户自己处理（这是 2025 年的潮流）。

04 典型代码

验证 baseline ——「这段代码我的 baseline 跑得起来吗？」

javascript · 多代特性混用样本 
// 这段代码用了 ES2015 - ES2025 多个特性。
// 看看你的 baseline 能跑哪些。

// ES2015: const, arrow, template literal, destructuring
const greet = (name) => `Hello, ${name}`;
const { x, y } = { x: 1, y: 2 };

// ES2017: async/await
async function fetchData() {
  const r = await fetch('/api');
  return r.json();
}

// ES2018: 对象 spread
const merged = { ...a, ...b };

// ES2019: Object.fromEntries
const obj = Object.fromEntries([['a', 1], ['b', 2]]);

// ES2020: 可选链 + 空值合并
const age = user?.profile?.age ?? 0;

// ES2021: 逻辑赋值
config.timeout ??= 5000;

// ES2022: class fields + private + Top-Level Await
class Counter {
  #count = 0;
  get value() { return this.#count; }
  inc() { this.#count++; }
}
// 模块顶层（仅 type="module" 时）
const data = await import('./data.json', { with: { type: 'json' } });

// ES2023: toSorted（不可变数组）
const sorted = [3, 1, 2].toSorted();

// ES2024: Promise.withResolvers
const { promise, resolve, reject } = Promise.withResolvers();

// ES2025: Iterator Helpers + using
const sum = items.values().filter(i => i > 0).map(i => i * 2).take(10).toArray();

await using file = await open(path);

把这段贴到 kangax compat-table 或 TS Playground，能立刻看到每行属于哪一版、当前 target 哪些会被降级、哪些原生支持。

<!-- 浏览器原生支持，无需打包 -->

05 常见陷阱

⚠️ 陷阱 1：把 polyfill 全打进主 bundle

老配置：import 'core-js/stable' 写在入口顶部。这会把 core-js 全量塞进主 bundle（70-200KB），即使你只用了一两个特性。

规避：用 useBuiltIns: 'usage'（Babel）或 swc-plugin-helpers（SWC）让工具链按实际使用注入 polyfill。或者把 polyfill 拆成动态 import：检测到不支持才加载。

⚠️ 陷阱 2：target 设太高导致老用户白屏

团队按"自己机器都没问题"定 ES2024 baseline，部署后一周内 Sentry 收到 SyntaxError: Unexpected token '?'——某些用户用旧 Android 4.4 WebView 或公司分发的老 Chromium。白屏是用户感知最差的错误。

规避：① 按 caniuse 实测目标用户群覆盖率，不凭感觉；② 上线前做"低版本浏览器实测"（BrowserStack / 真机）；③ 部署后立刻看 Sentry SyntaxError 告警——这是 baseline 设错的早期信号；④ 备好回退路径（CDN 切回老 bundle 的能力）。

⚠️ 陷阱 3：用 Stage 2 提案的语法

"Records & Tuples 看起来很棒，先用着"——然后提案被 TC39 改了语义，你的代码全部要重写。Pipeline operator 改了 5 次，每次社区代码都要调整。

规避：Stage 3 + 至少 2 个引擎已实现才考虑生产用，且必须配 polyfill。Stage 2 及以下仅做实验代码、技术验证 demo。永远不要把"Stage 2 提案"当生产依赖。

⚠️ 陷阱 4：误以为 Node 版本就是 ES 版本

"Node 22 = ES2024"—— 不严格成立。Node 各版本的 V8 内核版本不同，对 ES 特性的支持时间也不同。比如 Node 18 是 V8 10.2（ES2022 完整支持），Node 20 是 V8 11.3（ES2023 大部分），Node 22 是 V8 12.4（ES2024 大部分 + ES2025 部分）。

规避：用 node.green（kangax compat-table 的 Node 子集）查具体特性的 Node 起始版本。在 package.json.engines.node 里锁版本。
