模块系统与 Top-Level Await

P2.2 讲了 ESM 静态结构的设计决策，P7.3 讲了 Top-Level Await 的用户语义。本章把它们落到规范层——ECMA-262 §9.10 Source Text Module Records 的 4 阶段加载算法（Parse → LoadRequestedModules → Link → Evaluate）、循环依赖怎么不死锁、TLA 让 Evaluate 阶段变异步后规范如何检测潜在死锁、Module Namespace 为什么是 frozen exotic、import.meta 的 host hook 设计。读完这章，你能看懂 V8 / Node 加载 ESM 时的每一步、调试 Top-Level Await 死锁、写 Node loader hook。

⚠️ 这是 ECMA-262 中最复杂的算法之一

模块加载牵动 §9.4 ExecutionContext、§9.10 Module Records、§16.2 Modules 三个 clause，再叠加 §27 Promise（因为 Evaluate 现在返回 Promise）和 HTML 规范的 fetch 流程。建议带着具体场景读：先想"我写 import x from './a'，浏览器从看到这一行到 x 可用，发生了什么 14 步？"再对照本章。

📐 本章交叉规则

ESM 语法（import / export / 三阶段加载用户视角）在 JS · ESM 已详述；ESM 设计决策（为什么静态结构）在 P2.2 设计权衡 已讲；TLA 用户视角在 P7.3 Event Loop 与异步演进 已介绍。本章只在规范层算法这一层补全。

阅读时长 ~30 分钟 前置 P7.1 执行模型基础 + P7.3 异步演进 对应 ECMA-262 §16.2 Modules + §16.2.1.6 Source Text Module Records

01 速览

三句话定义本章：

  1. ESM 加载 = 4 阶段算法：Parse（生成 Module Record）→ LoadRequestedModules（递归 fetch + parse 所有依赖）→ Link（绑定 import/export，分配 EnvironmentRecord）→ Evaluate（按拓扑顺序执行，TLA 时返回 Promise）。
  2. 循环依赖靠 [[CycleRoot]] 算法处理：进入第一个被访问的模块时记 cycle root，递归遇到正在求值的模块就跳过不再深入；这就是为什么 ESM cycle 行为可预期，CommonJS 不行。
  3. Top-Level Await 让 Evaluate 阶段变异步，所以模块求值现在是 Promise；规范加了死锁检测算法（[[CycleRoot]] 和 [[AsyncEvaluation]] 字段）来识别"cycle 中两个 async 模块互相 await 对方"的情况。

02 底层逻辑

Source Text Module Record 的字段

每个 ES 模块在规范层是一个 Source Text Module Record（§16.2.1.6），关键字段如下：

字段 用途
[[Realm]] 模块所在的 Realm
[[Environment]] Module Environment Record（含所有 import/export binding）
[[Namespace]] 该模块的 Module Namespace exotic 对象（import * as ns 拿到的就是它）
[[Status]] 模块状态：new / unlinked / linking / linked / evaluating / evaluating-async / evaluated
[[ECMAScriptCode]] parse 后的 AST
[[RequestedModules]] 本模块依赖的所有模块说明符（按 import 出现顺序）
[[ImportEntries]] 所有 import 项（含 module specifier、import name、local name）
[[LocalExportEntries]] / [[IndirectExportEntries]] / [[StarExportEntries]] 三种 export 项（直接 export、re-export 单个、export *）
[[CycleRoot]] 循环依赖根模块（用于 Evaluate 阶段判断是否进入已访问的模块）
[[HasTLA]] 本模块是否包含 Top-Level Await（影响 Evaluate 是否异步）
[[AsyncEvaluation]] 当前是否处于异步 Evaluate 中
[[TopLevelCapability]] 异步 Evaluate 完成时要 fulfill 的 PromiseCapability

4 阶段加载算法

阶段 1 · Parse

源码 → AST → 创建 Source Text Module Record。这一步只解析本模块自身的 import/export 声明（不递归依赖）。[[Status]] 设为 new。

阶段 2 · LoadRequestedModules

递归处理 [[RequestedModules]]：每个模块说明符通过 Host hook HostLoadImportedModule 解析为目标 Module Record。Host 决定怎么 fetch（浏览器走 HTTP、Node 走文件系统、Deno 可走 URL/cache）。所有依赖都 parse 完后，[[Status]] 变为 unlinked。

阶段 3 · Link（曾叫 Instantiate）

为每个模块创建 Module Environment Record，分配 import/export binding 槽位（但不赋值）。这一步关键是把 import 名字和源模块的 export 槽位连起来——实现"live binding"语义（导出值变了，导入端跟着变）。算法递归处理 cycle：进入正在 link 的模块就跳过（防止无限递归）。完成后 [[Status]] 变 linked。

阶段 4 · Evaluate ⭐

按后序遍历执行模块代码（先依赖后调用）。每个模块的 [[ECMAScriptCode]] 在它自己的 Module Environment 上运行，所有 binding 在这一阶段才被赋值。

有 TLA 的情况：如果模块 [[HasTLA]] 是 true（含 top-level await），Evaluate 不再同步——返回一个 Promise，等所有 await 完成后才 fulfill。[[Status]] 经过 evaluating → evaluating-async → evaluated。

cycle handling：进入正在 evaluate 的模块时，记下 [[CycleRoot]]，跳过不再深入；当 cycle 全部 evaluate 完后，再统一 fulfill。

Top-Level Await 死锁检测

TLA 让模块求值变异步，开了一个新的死锁口子：cycle 中两个 async 模块互相 await 对方导出的值。看一个例子：

javascript · TLA cycle 死锁示例 
// a.js
import { b } from './b.js';
export const a = await Promise.resolve('A');
console.log('a.js: b =', b);   // 永远跑不到

// b.js
import { a } from './a.js';
export const b = await Promise.resolve('B');
console.log('b.js: a =', a);   // 永远跑不到

规范的处理方式：在 Link 阶段检测 cycle 时记下所有参与 cycle 的模块；Evaluate 时如果发现 cycle 内有 [[HasTLA]] 模块，整个 cycle 用统一的 PromiseCapability等待——只有所有 cycle 模块都把自己的 await 完成后，cycle 才 fulfill。如果其中两个模块互相依赖对方还未生成的值，整个 cycle 永久挂起（不抛错，但 Promise 永不 settle）。

这是规范有意的选择：抛错会让"无害的 cycle"也被毙掉；挂起则让宿主决定怎么处理（浏览器靠 unhandledrejection / Node 靠 timeout）。

Module Namespace Exotic Object

import * as ns from './m.js' 拿到的不是普通对象，是 Module Namespace exotic（§10.4.6）。它有几个特殊行为：

  • frozen：所有属性都 non-writable + non-configurable，Object.isFrozen(ns) === true
  • live binding：访问 ns.x 实际是去源模块的 EnvironmentRecord 取值，源模块改了 x 这里立刻生效（普通对象做不到）
  • [[Get]] trap：未定义的 export 返回 undefined（不是 throw）
  • typeof ns === "object"，但 ns.constructor 是 undefined（不挂在 Object.prototype 上）
  • Symbol.toStringTag 为 "Module"，所以 Object.prototype.toString.call(ns) === "[object Module]"

这些行为是为了让 Module Namespace 看起来像一个静态命名空间，而不是普通可变对象——你不能 ns.newProp = 1，也不能 delete ns.x。详细见 P6 内部槽与元编程 的 Exotic Object 列表。

Host Hooks（宿主扩展点）

§16.2 把模块加载的几个关键步骤交给宿主实现：

Host Hook 触发时机 / 用途
HostLoadImportedModule LoadRequestedModules 阶段，宿主决定怎么 fetch / parse 一个 specifier
HostResolveImportedModule（旧 API，仍兼容） 解析 specifier 到 Module Record（同步路径）
HostFinalizeImportMeta 每个模块的 import.meta 对象由宿主填——浏览器填 url，Node 填 url + resolve()，Deno 填更多
HostImportModuleDynamically 用户调 import('./x.js')（dynamic import）时触发

这就是为什么浏览器和 Node 的模块行为不一样——规范只规定了"什么时候调 hook"，"hook 干什么"完全交给宿主。Node 的 --experimental-loader 就是给开发者一个改 HostLoadImportedModule 行为的口子。

03 应用场景

场景 A · 调试 Top-Level Await 死锁

生产环境一个模块在浏览器加载后永久挂起，没有报错。常见原因是 cycle + TLA：

  • 用 Chrome DevTools "Sources" 标签的 "Module loader" 视图查看模块依赖图
  • 找出 cycle 中所有 [[HasTLA]] 模块（含 top-level await 的）
  • 检查 cycle 内是否有"模块 A await 模块 B 导出的值，但模块 B 还在 await 模块 A 导出的值"——这就是死锁
  • 修复：把 await 移到函数里（懒求值），或重构 cycle

场景 B · 写 Node loader hook

Node 的 --experimental-loader 让你用一个 ESM 模块覆盖 HostLoadImportedModule。常见用途：

  • TypeScript on-the-fly 编译（拦截 .ts 路径）
  • 自定义协议（npm:、github:）
  • 测试时 mock 依赖（让 ./api 返回 fake 实现）

理解 HostLoadImportedModule 的输入（specifier + parent module + assertions）+ 输出（Module Record）就能写出 loader。Node 文档：Customization hooks。

场景 C · 给 jsdom / linkedom 写 ESM polyfill

jsdom 实现 DOM API 时遇到的难点之一：<script type="module"> 怎么办？答案是实现 Module Record 的所有 host hook，让 V8 / SpiderMonkey 调你的 hook 而不是浏览器的。这是为什么 jsdom 的 module 支持仍然不完整——TLA、import.meta.resolve 等较新特性需要完整实现 4 阶段算法。

04 典型代码

观察 Module Namespace 的 frozen 行为

javascript · ns 是 frozen exotic 
// m.js
export let counter = 0;
export function inc() { counter++; }

// app.js
import * as ns from './m.js';

console.log(ns.counter);  // 0
ns.inc();
console.log(ns.counter);  // 1（live binding 生效——ns 看到的是源模块当前值）

// 但你不能写它
ns.counter = 999;         // strict 下 TypeError；非 strict 下静默失败
ns.newProp = 'x';         // 同上
delete ns.counter;        // 同上
Object.isFrozen(ns);      // true

// 它甚至不在 Object.prototype 链上
Object.getPrototypeOf(ns); // null
ns.hasOwnProperty;         // undefined！
Object.prototype.toString.call(ns);  // "[object Module]"

用 dynamic import 规避 cycle 死锁

javascript · 把 await 推到运行时 
// 改写前（cycle + TLA 死锁）
// a.js
// import { b } from './b.js';
// export const a = await fetchA();   // ← 死锁原因

// 改写后（用 dynamic import 推迟到调用时）
export async function getA() {
  const { b } = await import('./b.js');  // 此时 b.js 已 evaluated
  const a = await fetchA();
  return { a, b };
}

// b.js 同理改写。两个模块的 top-level 不再 await，cycle 解开。

05 常见陷阱

⚠️ 陷阱 1：以为 import * as ns 返回普通对象

ns 是 Module Namespace exotic，不是普通对象。Object.assign({}, ns) 可以拷出一份普通对象但失去 live binding；Object.keys(ns) 能拿到 export 名字（这个支持），但 ns instanceof Object 是 false。

规避：把 ns 当"只读的属性视图"用，不 mutate、不当作普通对象传递；需要拷贝时用 { ...ns }（明确拷出一份普通对象）。

⚠️ 陷阱 2：在 cycle 中用 Top-Level Await

可能死锁。规范不会抛错，宿主也未必有超时——表现为 Promise 永久挂起，看起来像页面"卡在加载中"。

规避：① 拆 cycle（多数情况是设计问题）；② 把 await 移到函数里（不在 top-level）；③ 用 dynamic import 推迟初始化。Node 的 process.on('unhandledRejection', ...) 检测不到这种死锁——它根本没 reject。

⚠️ 陷阱 3：以为 dynamic import 能替代静态 import

动态 import 失去 tree-shaking 能力——bundler 看到 import('./x') 就把整个 x 模块作为单独 chunk 打包，无法 tree-shake 你不用的 export。

规避：dynamic import 用于明确的代码分割（如路由懒加载），不要替代普通 import。需要全模块拆分时才动态。
