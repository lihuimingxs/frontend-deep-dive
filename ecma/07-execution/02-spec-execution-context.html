§9 执行模型规范精读

P7.1 用"用户视角"讲了 ECMAScript 的执行机器（Realm / Agent / Job / ExecutionContext / EnvironmentRecord）。本章用"规范层数据结构视角"再讲一遍——把每个概念还原成 ECMA-262 §9 章里的具体记录字段、方法签名、算法步骤。读完后你能照着规范文本精确复述引擎做了什么、Host hook 在哪几个点被调用、Module evaluation 的 4 阶段算法怎么处理 cycle。这是 P7 最难的一章——但也是和引擎实现联系最紧密的，读懂之后看 V8 / SpiderMonkey 源码会快很多。

⚠️ 与 P7.1 的关系：同一台机器的两个视角

P7.1 讲"机器是什么样的"（含图、含完整生命周期 trace），适合工程师建立心智模型；本章讲"规范怎么用数据结构精确建模这台机器"，适合写 polyfill / 读引擎源码 / 追踪 TC39 提案的场景。两章不重复——P7.1 的图本章不再画，P7.1 的"完整生命周期 trace"本章不再 trace；本章只补 P7.1 没展开的规范字段细节。

阅读时长 ~40 分钟 前置 P7.1 执行模型基础（必读）+ P3 怎么读规范 + P4-P6 对应 ECMA-262 §9 全章（10 个子章节）

01 速览

三句话定义本章：

  1. §9 把"运行时机器"切成 10 个子章节：§9.1 Environment Records / §9.2 PrivateEnvironment Records / §9.3 Realms / §9.4 Execution Contexts / §9.5 Agents / §9.6 Agent Clusters / §9.7 Forward Progress / §9.8 Jobs / §9.9 InitializeHostDefinedRealm / §9.10 Source Text Module Records。
  2. EnvironmentRecord 是规范的"作用域基类"，5 种实现（Declarative / Function / Object / Module / Global）共享同一组方法签名（HasBinding / CreateMutableBinding / GetBindingValue / SetMutableBinding 等）。变量查找算法 ResolveBinding 沿 [[OuterEnv]] 链调用这些方法。

02 底层逻辑

§9.1 Environment Records —— 作用域的规范基类

所有 EnvironmentRecord 共享 9 个方法（规范层的"接口"）：

方法 用途
HasBinding(N) 是否有 binding N？返回 boolean
CreateMutableBinding(N, D) 新建可变 binding（D 表示是否可被 delete）
CreateImmutableBinding(N, S) 新建不可变 binding（S = strict 标志，const 用这个）
InitializeBinding(N, V) 把 N 绑到值 V（结束 TDZ 状态）
SetMutableBinding(N, V, S) 修改 binding 的值（S = strict，影响 readonly 失败行为）
GetBindingValue(N, S) 取 binding 的值（未 initialize 时抛 ReferenceError，这就是 TDZ）
DeleteBinding(N) 删除 binding（let/const 都不可删）
HasThisBinding() 是否有 this（仅 Function / Module / Global）
HasSuperBinding() 是否有 super（仅 method 内的 Function Env）

5 种实现各自重写部分方法，并扩展独有字段：

类型 spec § 独有字段 / 行为
Declarative §9.1.1.1 最基础。let/const/function/class 在这里。binding 表 + uninitialized 状态
Function §9.1.1.3 Declarative 的子类。多 [[ThisValue]] / [[ThisBindingStatus]]（lexical / initialized / uninitialized）/ [[FunctionObject]] / [[NewTarget]]
Object §9.1.1.2 把一个对象当 binding 表（obj.x 即 binding x）。with 用、非 strict global 用
Module §9.1.1.5 多 CreateImportBinding(N, M, N2) —— 把当前 N 绑到模块 M 的导出 N2，实现 live binding
Global §9.1.1.4 合体：Object Env（globalThis 上的属性）+ Declarative Env（顶层 let/const）。所以 var x = 1 后 window.x === 1 但 let y = 1 后 window.y === undefined

ResolveBinding 算法 —— 标识符解析

引擎解析标识符 x 时调 ResolveBinding（§9.4.2）：

spec · §9.4.2 ResolveBinding ( name [, env] ) 
// 1. If env is not present or is undefined, then
//    a. Set env to the running execution context's LexicalEnvironment.
// 2. Let strict be IsStrict() of running execution context.
// 3. Return ? GetIdentifierReference(env, name, strict).

// GetIdentifierReference ( env, name, strict ):
// 1. If env is null, return Reference Record { [[Base]]: unresolvable, ... }.
// 2. If ? env.HasBinding(name) is true, return Reference Record { [[Base]]: env, [[ReferencedName]]: name, ... }.
// 3. Else, return ? GetIdentifierReference(env.[[OuterEnv]], name, strict).

// 关键：沿 [[OuterEnv]] 链向上查 —— 这就是词法作用域的实现。
// 找到 → 产生 Reference Record（B2 讲过）。
// 找不到 → unresolvable Reference，取值时抛 ReferenceError。

§9.2 PrivateEnvironment Records —— class #field 的作用域链

ES2022 加 #field 时引入了一种独立作用域链——和 LexicalEnvironment 平行。规范叫 PrivateEnvironment：

  • 每个 class declaration 创建一个 PrivateEnvironment
  • 含 PrivateName 集合（#x / #y / #method）
  • nested class 沿 [[OuterPrivateEnvironment]] 上溯
  • 访问 this.#x 触发 PrivateFieldGet，从当前 PrivateEnvironment 解析 #x
  • 跨 class 访问 #x parse 阶段就是 SyntaxError（不是运行时）

这就是 B4 讲过的"硬隐私"实现——PrivateName 不在属性表里，而在独立的 PrivateEnvironment 上。引擎完全控制访问路径，用户代码无法绕过。

§9.3 Realms —— Intrinsics 表

每个 Realm 有 4 个内部槽：

字段 含义
[[Intrinsics]] Intrinsics 表——本 Realm 独立的几百个内置对象
[[GlobalObject]] 本 Realm 的 globalThis
[[GlobalEnv]] 本 Realm 的 Global EnvironmentRecord
[[HostDefined]] 宿主可附加的额外字段（浏览器存 origin / CSP / cookies 关联等）

Intrinsics 表是关键——它含 %Object% / %Array% / %Function.prototype% 等几百个引用，每个都对应一个独立的对象。这就是为什么 arr instanceof Array 跨 Realm 失败：每个 Realm 的 %Array.prototype% 都不同（虽然结构相同）。

规范用%名字% 语法引用 intrinsics（如 %Promise.prototype.then%）。这个表在 §6.1.7.4 有完整列表（300+ 条目）。

§9.4 Execution Contexts

A3 已详述每个 ExecutionContext 的字段。规范层补充几点：

  • execution context stack：LIFO 栈，栈顶是 "running execution context"
  • 3 种类型：script（脚本顶层）/ module（模块顶层）/ function（函数体）
  • [[Function]] 字段：脚本/模块时为 null，函数时指向被调用的函数对象
  • "suspended" 与 "active"：generator / async function 暂停时，ExecutionContext 从栈中移除但不销毁，存在 generator/async 对象的内部槽里（A3 详述过）

§9.5-§9.7 Agents / Clusters / Forward Progress

Agent（§9.5）

每个 Agent 有 4 个内部字段：

字段 含义
[[LittleEndian]] 本 Agent 用的字节序（DataView 默认值）
[[CanBlock]] 是否允许 Atomics.wait 阻塞（主线程通常 false，Worker 通常 true）
[[Signifier]] Agent 的"身份证"——一个唯一 ID，在 Atomics 操作中标识"哪个 Agent 在等"
[[IsLockFree1]] / [[IsLockFree2]] / [[IsLockFree8]] 1/2/8 字节原子操作是否是 lock-free（影响 Atomics.isLockFree 返回值）

Agent Cluster（§9.6）

一组能共享 SharedArrayBuffer 的 Agent。规范要求：

  • 同一 Cluster 内的 Agent 看到 一致的内存模型（§29 Memory Model 定义 happens-before 关系）
  • 跨 Cluster 不能共享内存（必须用 postMessage 复制）
  • 浏览器：同 Tab 主线程 + dedicated Worker 默认在同一 Cluster；跨 Tab / 跨 Origin / SharedWorker 不在
  • Node：每个 worker_threads 默认在主线程同 Cluster

Forward Progress（§9.7）

规范保证：每个非阻塞 Agent 必须周期性取得进展。这是规范层的"不能让某个 Agent 永久饿死"承诺——但没有具体调度公平性要求。Host 决定调度策略，规范只要求"不能完全停下"。

§9.8 Jobs and Host Operations ⭐

这是 §9 中最实用的子章——直接定义了 Host Hooks，宿主用它们实现 Event Loop / 模块加载 / Promise 调度。

核心 Host Hooks 一览：

Host Hook 用途
HostEnqueuePromiseJob(job, realm) 规范说"把 Promise reaction 加入 microtask queue"，宿主决定怎么投递
HostMakeJobCallback(callback) 包装一个 callback 成 Job。浏览器额外保存 incumbent realm / promise context 等元数据
HostCallJobCallback(jobCallback, V, args) 实际调用回调。V8 在这里做调试 hook、Sentry 在这里捕获错误
HostPromiseRejectionTracker(promise, operation) 未处理 reject 怎么报告（浏览器触发 unhandledrejection，Node 写 stderr）
HostFinalizeImportMeta(importMeta, moduleRecord) 设置 import.meta.url 等。Node 还设 import.meta.resolve
HostLoadImportedModule(referrer, specifier, hostDefined, payload) 处理 import 语句——宿主负责 fetch + parse 模块
HostGetImportMetaProperties(moduleRecord) import.meta 的初始属性集合
HostEnsureCanCompileStrings(callerRealm, calleeRealm) eval / new Function 是否被 CSP 允许
HostEnqueueGenericJob(job, realm) 通用 Job 投递（非 Promise 的）
HostEnqueueTimeoutJob(job, milliseconds) 带延迟的 Job —— setTimeout 可基于此实现（实际 setTimeout 是 HTML 规范，不是直接 ECMA 的）
📐 一句话总结 Host Hooks

ECMA-262 规范层只定义"什么时候调谁"，宿主（浏览器、Node、Bun、Deno、Cloudflare Workers、引擎嵌入）决定"怎么实现"。所以：

  • Promise.then / queueMicrotask 行为在所有宿主一致——这部分 ECMA-262 锁死语义
  • setTimeout / I/O / 文件系统 行为各 host 不同——这部分是 host hook + host 自己规范
  • 看到不一致行为，先想"这个 API 是不是 Host 实现"——答案常常解释一切

§9.9 InitializeHostDefinedRealm —— 启动序列

规范层定义"启动一个 ECMAScript 实例"的算法：

  1. 创建新 Realm
  2. 调 CreateIntrinsics(realm) 初始化 300+ 内置对象（每个都是新对象）
  3. 调 host hook HostDefined 让宿主附加 globalThis（浏览器加 DOM API、Node 加 process / require / Buffer、Bun 加 Bun 等）
  4. 创建 Global EnvironmentRecord
  5. 创建 Realm 的执行上下文栈
  6. 返回 Realm，可以开始跑代码了

这个算法解释了为什么启动一个 V8 isolate 大约要几 ms——300+ intrinsic 对象的创建 + host bindings 加载是固定成本。Node 的 vm.createContext() 和浏览器的 iframe 各跑一次这个序列。

§9.10 Source Text Module Records —— ESM 4 阶段算法

ESM 是 ECMA-262 中最复杂的算法。一个 Module Record 含的字段（精选）：

字段 含义
[[Realm]] 所属 Realm
[[Environment]] 本模块的 Module EnvironmentRecord
[[Namespace]] 缓存的 Module Namespace exotic（B4 讲过）
[[Status]] new / unlinked / linking / linked / evaluating / evaluating-async / evaluated
[[ImportEntries]] import 语句产生的 entry 列表
[[LocalExportEntries]] 本模块声明的 export
[[IndirectExportEntries]] export { x } from './m' 形式
[[StarExportEntries]] export * from './m' 形式
[[CycleRoot]] 循环依赖中"哪个模块作为 cycle 入口"
[[HasTLA]] 本模块是否含 Top-Level Await
[[AsyncEvaluation]] 异步求值状态（含一个 [[AsyncEvaluationOrder]] 决定 settle 顺序）
[[TopLevelCapability]] 本模块顶层 evaluate 返回的 Promise

4 阶段算法

每个模块都按这个顺序处理：

① Parse（§16.1.4 ParseModule）

  • 解析源码 → AST → 收集 import / export 声明
  • 产生 Module Record，状态 new
  • 失败：throw SyntaxError，模块作废

② Load (§16.2.1.5 LoadRequestedModules)

  • 对所有 import 的依赖，调 host hook HostLoadImportedModule 让宿主 fetch 源文件
  • 递归对每个依赖也走 Parse + Load
  • 构建依赖图，包括 cycle 检测
  • 状态：unlinked

③ Link (§16.2.1.6 InitializeEnvironment + §16.2.1.5)

  • 给每个模块创建 Module EnvironmentRecord
  • 对每个 import { x } from './m'，在当前 Env 上 CreateImportBinding(x, m, x)—— 把 x 绑到模块 m 的导出 x，这就是 live binding 的实现（导入端 x 实际去 m 的 EnvironmentRecord 取值）
  • 对每个 export 声明，在 m 的 LocalExportEntries 注册
  • 状态：linked
  • 失败：linking error

④ Evaluate (§16.2.1.5.3 / §16.2.1.7 ExecuteModule)

  • 按拓扑序执行模块顶层代码
  • 遇到 cycle 时，按 [[CycleRoot]] 协调（cycle 内的模块可能拿到 uninitialized binding，访问时抛 ReferenceError）
  • 含 Top-Level Await 时变成异步过程，详见下面
  • 状态：evaluated

Top-Level Await 死锁检测

ES2022 引入 TLA 后，模块求值变成异步过程。规范层加了：

  • 每个模块的 evaluation 现在返回 Promise（[[TopLevelCapability]]）
  • Async sibling 协调：cycle 内的多个 async 模块用 [[AsyncEvaluationOrder]] 决定 settle 顺序，避免乱序
  • 两个 async 模块互相 await import() 形成 cycle 时——不会无限挂起，规范要求宿主直接报错
  • 异步评估完成后再开始执行后续 sync 依赖（保持顺序保证）

实际工程教训：TLA 让模块依赖图变成 Promise 树。一个慢 TLA 拖累所有依赖它的下游模块——谨慎在共享模块用 TLA。Vite / Next.js 等工具会警告"未优化的 TLA 阻塞首屏"。

03 应用场景

场景 A · 写 Node loader hook

Node 的 --experimental-loader 让你钩入 HostLoadImportedModule——拦截 import 解析，做 alias、动态生成代码、TS on-the-fly 编译等。

javascript · loader.mjs 
// loader.mjs
export async function resolve(specifier, context, nextResolve) {
  // 把 #alias/* 解析成 ./src/*
  if (specifier.startsWith('#alias/')) {
    return nextResolve(specifier.replace('#alias/', './src/'), context);
  }
  return nextResolve(specifier, context);
}

export async function load(url, context, nextLoad) {
  // 拦截 .ts 文件，运行时编译
  if (url.endsWith('.ts')) {
    const source = await fs.readFile(new URL(url), 'utf8');
    const compiled = ts.transpile(source, { module: 'esnext' });
    return { format: 'module', source: compiled, shortCircuit: true };
  }
  return nextLoad(url, context);
}

// 启动：node --experimental-loader ./loader.mjs main.mjs
// 这就是 ts-node、tsx、@swc-node/register 的工作方式。

底层就是 Node 在 HostLoadImportedModule 里调用你的 resolve + load。

场景 B · 调试 Top-Level Await 死锁

两个 async 模块互相 await import() 对方时，规范保证不死锁——但你会看到 unhandled rejection 而不是预期的成功。诊断思路：

javascript · 死锁示例 
// a.mjs
const b = await import('./b.mjs');
export const a = 'a value';

// b.mjs
const a = await import('./a.mjs');
export const b = 'b value';

// 启动 node --experimental-modules a.mjs：
// 看到 "Module evaluation cycle could not be resolved"
// 或者 a 拿到的 b.mjs.b 是 undefined（取决于实现）

// 解法：避免 cycle —— 把共享代码抽到第三个模块
// 或：用动态 import 在函数里调用，不在顶层

诊断工具：Node 的 --trace-module-load、Vite 的 import graph 可视化。理解 [[CycleRoot]] 后这种问题不再玄学。

场景 C · 给 jsdom / linkedom 写 polyfill

jsdom 模拟浏览器 DOM 环境给 Node 用——它要在 Node 里实现浏览器 host 的所有 hook。如：

  • HostFinalizeImportMeta：设 import.meta.url 为 jsdom 模拟 URL
  • HostPromiseRejectionTracker：触发 jsdom 的 unhandledrejection 事件给 window
  • 不同 jsdom Realm 间的 postMessage 包装
  • setTimeout 等 timer hook（Node 已有，jsdom 重新包成浏览器 API）

理解 §9.8 的 host hooks 表后，给 DOM 实现做扩展（vue-test-utils、@testing-library 等）就有了"知道我能 hook 哪些点"的全景图。

04 典型代码

用 queueMicrotask 观察 microtask 调度

javascript · HostEnqueuePromiseJob 路径 
// queueMicrotask 是 ECMA-262 标准 API（§27.7.5），
// 内部直接调 HostEnqueuePromiseJob —— 比 Promise.then 更精简。
// 也是写"自定义微任务调度"的官方入口。

console.log('1');
queueMicrotask(() => console.log('3'));
Promise.resolve().then(() => console.log('4'));   // 也走 microtask
console.log('2');

// 输出：1 → 2 → 3 → 4
// queueMicrotask 投递的先于 Promise.then 注册的（注册顺序 FIFO）

// 实测注册顺序：
queueMicrotask(() => console.log('A'));
queueMicrotask(() => console.log('B'));
Promise.resolve().then(() => console.log('C'));
queueMicrotask(() => console.log('D'));
// 输出：A → B → C → D（严格 FIFO）

// 工程意义：
// - 写 React 18 / Vue 3 调度器时，microtask 就是"批量更新"的边界
// - 一个宏任务 + 多个 microtask 的"批"是渲染颗粒度的下限

对照规范理解 var hoisting 的本质

javascript · var 在 EnvironmentRecord 上的实际行为 
// 看似魔法的 hoisting：
console.log(x);  // undefined（不报错！）
var x = 1;

// 规范层视角（§10.2.11 FunctionDeclarationInstantiation 和
// §16.1.7 GlobalDeclarationInstantiation）：
//
// 在 ExecutionContext 创建阶段（不是运行阶段）：
// 1. 收集源码中所有 var 声明的名字
// 2. 对每个名字，调当前 VariableEnvironment.CreateMutableBinding(name, true)
// 3. 调 InitializeBinding(name, undefined) —— 直接初始化为 undefined
//
// 所以 console.log(x) 时：
// - HasBinding('x') 返回 true（已在创建阶段建好）
// - GetBindingValue('x') 返回 undefined（已 initialize 为 undefined）
// - 不抛 ReferenceError！

// let / const：
console.log(y);  // ReferenceError: Cannot access 'y' before initialization
let y = 1;

// 规范层视角：
// 1. CreateMutableBinding('y', true) —— 创建 binding
// 2. NOT InitializeBinding('y', ...)  —— 不初始化（保持 uninitialized 状态）
// 3. 运行到 let y = 1 时才 InitializeBinding('y', 1)
//
// 所以前面 console.log(y)：
// - HasBinding('y') 返回 true（已创建）
// - GetBindingValue('y') 检查 uninitialized → 抛 ReferenceError
// - 这就是 TDZ —— "binding 已存在但未初始化"

// 这两段代码的根因差异是 EnvironmentRecord 的"先 init / 不 init"决策。
// 不是"代码搬运"，是"创建阶段的 init 行为差异"。

05 常见陷阱

⚠️ 陷阱 1：把 Job 等同于 microtask

Job 是规范层抽象，包含多种类型——PromiseJob（最常见，对应 microtask）、ScriptJob（脚本初次执行）、其他 host-defined Jobs（setTimeout 回调等）。

规避：说"microtask"指浏览器/Node 的 microtask queue 实现；说"Job"指规范层抽象。在 spec 讨论里区分清楚。

⚠️ 陷阱 2：以为 Realm 之间共享 Intrinsics

"Array.prototype 是单例" 是错的。每个 Realm 都有独立的 Intrinsics 表——这是为什么跨 iframe 的 arr instanceof Array 失败、arr.constructor === Array 也失败。详见 P7.1 执行模型基础 + P5 抽象操作 的 IsArray 算法。

规避：跨 Realm 用 Array.isArray（Realm-aware）、Object.prototype.toString.call、或者结构判断。

⚠️ 陷阱 3：在 Module 顶层写 var 期待 globalThis 上有

ES Module 顶层的 var x = 1 不挂在 globalThis 上——因为 Module 用 Module EnvironmentRecord，不是 Global。这与 script 顶层 var 行为不同。

javascript 
// script: var x = 1; → window.x === 1
// module: var x = 1; → window.x === undefined（Module Env 隔离）
//
// 同理：function f() {} 在 module 顶层也不挂 globalThis

规避：要全局变量明确 globalThis.x = ...。Module 应该用 export 而不是污染全局。

⚠️ 陷阱 4：在 Top-Level Await 里阻塞所有依赖加载

一个共享 utils 模块的 TLA await fetch(...) 会让所有 import 它的模块都等—— 即使下游不需要 fetch 结果。整个应用首屏被阻塞。

规避：① TLA 仅用于"本模块自己必须的初始化"，不用在共享 utils；② 用 lazy init pattern（const data = (() => { let _; return () => _ ??= fetch(...); })()）按需触发；③ 用 dynamic import() 把异步部分隔离。

⚠️ 陷阱 5：以为 Promise.then 的 microtask 调度是 ECMA-262 定义的

不是。ECMA-262 只说"调用 HostEnqueuePromiseJob"——具体 microtask queue / Event Loop 是 HTML 规范（浏览器）或 Node 自己定义的。所以"microtask 优先于 macrotask"是 HTML 规范要求，不是 ECMA-262。

规避：写跨平台代码时，如果某行为不在 ECMA-262 里，查 HTML 规范或 Node 文档，不要假设语义统一。
