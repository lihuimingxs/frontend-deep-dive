内存模型与多线程（§24 Memory Model + Atomics）

ECMA-262 §24 是规范里"最像 C++ 标准"的一章——它形式化定义了多 Agent 之间共享内存的事件偏序、happens-before 关系、race condition 的精确语义。这是从 ES2017 加入 SharedArrayBuffer / Atomics 起，TC39 把"JS 多线程"从"偶尔的 hack"升级为"规范一等公民"的标志。本章把 §24 形式化 + Atomics 11 个 API + Lock-free 经典模式 + Spectre 缓解后的浏览器配置一次讲透——是 P7 的压轴章，也是 WebGPU compute / FFmpeg.wasm / AudioWorklet 等高性能 Web 应用的硬基础。

⚠️ 这是 ECMA-262 中最深的一章

§24 用 Candidate Execution、Synchronizes-With、happens-before 等形式化术语描述多线程语义——和 C++/Java 内存模型同构，但 JS 的好处是所有 Atomics 默认顺序一致（SC），不需要选 6 种 memory_order，心智负担小一半。代价是性能略低于精细控制。本章不要求你写 lock-free 代码——读懂 80% 就够工程使用，剩下的 20% 留给真正写并发原语的人。

📐 本章交叉规则

P7.1 Agent / Agent Cluster 已讲了多 Agent 的容器边界；P7.3 异步演进 讲了从 Promise 到多线程的演进路径。本章在它们基础上展开共享内存的形式化语义 + 同步原语 + 浏览器安全配置。

阅读时长 ~45 分钟 前置 P7.1 Agent / Agent Cluster（必读）+ P7.3 异步演进 SAB 段（推荐） 对应 ECMA-262 §24 Memory Model + §25.4 Atomics

01 速览

三句话定义本章：

  1. JS 多线程 = Agent Cluster + SharedArrayBuffer + Atomics + §24 Memory Model。Agent 边界默认隔离（postMessage 传值），但同 Cluster 的 Agent 可以共享 SAB（传引用）；Atomics 提供无撕裂的读写 + 原子读改写 + 同步原语；§24 形式化保证什么操作之间有 happens-before。
  2. Spectre 让 SAB 在 2018 全网下线，2020 配 COOP+COEP 头重启。今天用 SAB 必须满足 crossOriginIsolated === true——浏览器把 origin 完全隔离到独立进程，恢复高精度时钟（Spectre 的核心条件），代价是嵌入第三方 iframe 要小心。
  3. 所有 Atomics 默认顺序一致（SC）——比 C++ 6 种 memory_order 简单得多，但普通访问不是原子：写 arr[0]++ 在多线程下会撕裂，必须 Atomics.add(arr, 0, 1)。

02 底层逻辑

§24 Memory Model 形式化

§24 用 6 个核心概念定义多线程语义。读懂这 6 个，规范文本就读得通。

概念 含义
Event 程序执行中可观察的"动作"——每次 Atomics 操作 / 普通 SAB 读写都是一个 Event，含字节范围 + 类型（Read / Write / RMW）
Candidate Execution 所有可能的"事件偏序"集合——给定一段并发程序，规范不规定 Event 的全序，而是给一个集合，每个元素是一种合法的执行顺序
Agent Order 同一个 Agent 内 Event 的程序顺序（program order）——这是单线程内的"按代码先后"
Synchronizes-With 跨 Agent 的"同步点"关系——一个 Atomics.store 与之后的 Atomics.load 同步；Atomics.notify 与对应的 Atomics.wait 同步
Happens-Before 由 Agent Order ∪ Synchronizes-With 的传递闭包定义。如果事件 A happens-before B，那么 A 的副作用对 B 可见
Race Condition 两个事件没有 happens-before 关系（双向都不) + 至少一个是 Write + 访问相同内存——精确定义为"未同步的并发访问"

关键观察：JS 没有 C++ 6 种 memory_order（relaxed / acquire / release / acq_rel / consume / seq_cst）。所有 Atomics 操作隐含 seq_cst（Sequentially Consistent，顺序一致）语义——所有线程观察到的 Atomics 操作顺序一致。代价是性能比 relaxed 略低（CPU 不能重排）；好处是不会写错。

SharedArrayBuffer：共享内存的载体

SAB 是 ArrayBuffer 的"共享版"——同一段字节被多个 Agent 同时持有引用，任何一方写都立刻被其他方看见。规范层叫 Shared Data Block（§24.2.2）。

javascript · SAB 创建与跨 Agent 共享 
// 检查环境是否支持（必须 crossOriginIsolated）
if (typeof SharedArrayBuffer === 'undefined' || !crossOriginIsolated) {
  throw new Error('需要 COOP+COEP 头才能用 SAB');
}

// 创建 1024 字节的共享内存
const sab = new SharedArrayBuffer(1024);

// 创建 view（必须用 SAB 兼容的 TypedArray）
const i32 = new Int32Array(sab);   // 256 个 int32
const f64 = new Float64Array(sab); // 128 个 double（不能 Atomics！）

// 主线程发给 Worker
const worker = new Worker('worker.js');
worker.postMessage(sab);
//          ↑ 这是引用传递，不是值复制（与普通 ArrayBuffer 的 transfer 语义不同）

// worker.js
self.onmessage = (e) => {
  const sab = e.data;
  const i32 = new Int32Array(sab);
  Atomics.add(i32, 0, 1);  // 主线程立刻看见 i32[0] 变化
};

关键约束：只有 Int8/Uint8/Int16/Uint16/Int32/Uint32/BigInt64/BigUint64Array 支持 Atomics。Float32Array / Float64Array 没有 Atomics（IEEE 754 浮点 + 跨架构原子访问太复杂，规范没纳）。

Atomics 的 11 个核心 API

1. 原子读写（避免撕裂）

  • Atomics.load(typedArray, index) → 完整读 32 / 64 位（普通 ta[i] 在 CPU 不对齐时可能撕裂）
  • Atomics.store(typedArray, index, value) → 完整写

2. 原子读-改-写（RMW）

  • Atomics.add(ta, i, v) / Atomics.sub(ta, i, v) —— 原子加减，返回旧值
  • Atomics.and / Atomics.or / Atomics.xor —— 原子位运算
  • Atomics.exchange(ta, i, v) —— 原子交换：写新值，返回旧值（"无条件 swap"）

3. CAS（Compare-And-Swap）—— Lock-free 数据结构核心

  • Atomics.compareExchange(ta, i, expected, replacement) —— 如果当前值 === expected，则写 replacement，返回旧值；否则不变

4. 同步原语（实现 mutex / 条件变量）

  • Atomics.wait(ta, i, value, timeout?) —— 阻塞当前 Agent，直到 ta[i] !== value 或 notify 或 timeout。返回 "ok" / "not-equal" / "timed-out"。主线程禁用（会卡 UI）
  • Atomics.notify(ta, i, count?) —— 唤醒 wait 在 ta[i] 上的最多 count 个 Agent，返回实际唤醒数
  • Atomics.waitAsync(ta, i, value, timeout?)（ES2024）—— 主线程的 wait 替代，返回 { async, value: Promise<"ok" | "not-equal" | "timed-out"> }

Lock-free 经典模式

SPSC ring buffer（单生产者-单消费者无锁队列）

javascript · SPSC ring buffer 核心 
// 共享内存布局：
//   [head_index, tail_index, slot_0, slot_1, ..., slot_{N-1}]
//                  ↑ 只生产者写、消费者只读
//                          ↑ 只消费者写、生产者只读
// 因为读写各自独占，不需要 CAS，只需 Atomics.load/store

class SPSCQueue {
  constructor(sab, capacity) {
    this.meta = new Int32Array(sab, 0, 2);    // [head, tail]
    this.data = new Int32Array(sab, 8, capacity);
    this.cap = capacity;
  }
  // 生产者
  push(value) {
    const tail = Atomics.load(this.meta, 1);
    const head = Atomics.load(this.meta, 0);
    if ((tail + 1) % this.cap === head) return false;  // full
    this.data[tail] = value;                           // 普通写：同一线程后续 store 之前
    Atomics.store(this.meta, 1, (tail + 1) % this.cap);
    return true;
  }
  // 消费者
  pop() {
    const head = Atomics.load(this.meta, 0);
    const tail = Atomics.load(this.meta, 1);
    if (head === tail) return undefined;  // empty
    const value = this.data[head];
    Atomics.store(this.meta, 0, (head + 1) % this.cap);
    return value;
  }
}
// 关键：head 由 Atomics.store 同步，所以消费者读到 head 推进时，
// data[head] 的写入对消费者可见（happens-before 由 Atomics 链建立）。

spinlock with backoff（自旋锁 + 退避）

javascript · 简易 mutex 
// 共享变量 lock[0]: 0=unlocked, 1=locked
function lock(state) {
  while (Atomics.compareExchange(state, 0, 0, 1) !== 0) {
    // CAS 失败 = 已被锁住，退避
    Atomics.wait(state, 0, 1);  // 阻塞直到 unlock 后被 notify
  }
}

function unlock(state) {
  Atomics.store(state, 0, 0);
  Atomics.notify(state, 0, 1);  // 唤醒 1 个等待者
}

// 主线程不能用 Atomics.wait —— 改用 Atomics.waitAsync：
async function lockMain(state) {
  while (Atomics.compareExchange(state, 0, 0, 1) !== 0) {
    const { async, value } = Atomics.waitAsync(state, 0, 1);
    if (async) await value;
  }
}

双缓冲（double buffering）

UI 主线程读 + Worker 写交替的经典模式：两块缓冲区 + 一个"当前可读 index" 原子标记。Worker 写完一块就把 index 翻转，主线程下一帧读新块。AudioWorklet / Canvas FLIP 都用这个模式，避免读到半写的数据。

Spectre 缓解与 COOP/COEP

Spectre 攻击（2018）利用 CPU 推测执行 + 高精度时钟测出"不该读到的内存"。SAB 给 JS 带来微秒级精度计时器，于是 SAB 在 2018 年被全网下线。

2020 重启：要求页面声明跨源隔离（cross-origin isolated），浏览器把它放到独立进程，断绝跨 origin 的内存可观察性。配置：

HTTP headers · 启用 SAB 
// 主页面 HTTP 响应头
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp

// 所有子资源（image / iframe / script）必须额外标
Cross-Origin-Resource-Policy: same-site   // 或 cross-origin

// 然后页面 JS 可以检测：
console.log(crossOriginIsolated);  // true 才能用 SAB
console.log(typeof SharedArrayBuffer);  // "function"

COEP 的 require-corp 是最严格的：所有跨 origin 子资源（CDN 上的 image、第三方 iframe）都必须显式声明 CORP 头允许嵌入。这会破坏不少老页面（特别是嵌广告 / 第三方分析）——是 SAB 的"政治成本"。

主线程限制 vs Atomics.waitAsync

Atomics.wait 在主线程会抛错（"Atomics.wait cannot be called in this context"）—— HTML 规范的 host hook 限制，避免阻塞 UI。ES2024 加的 Atomics.waitAsync 返回 Promise，不阻塞，主线程可用。这是 JS 多线程心智的"异步基因"——同步阻塞总是被替换为返回 Promise。

03 应用场景

场景 A · AudioWorklet 实时音频处理

音频处理要求在每个 ~3ms 的回调周期内完成 sample 处理。主线程完全不可用（GC 暂停就掉帧）。方案：

  • 音频回调跑在 AudioWorkletProcessor（独立线程）
  • UI 通过 SAB 传递参数（音量、频率等）
  • 用 Atomics.store 更新参数，AudioWorklet Atomics.load 读取
  • 无需序列化、无需 postMessage 抖动

这是 SAB 在 Web 端的旗舰应用——延迟从毫秒级降到微秒级。

场景 B · WebGPU compute / FFmpeg.wasm

处理大型矩阵 / 视频帧：

  • 用 SAB 直接装张量 / 帧数据，避免 postMessage 的"复制 + 反序列化"开销
  • 多个 Worker 并行计算（map-reduce 模式），用 Atomics 同步进度
  • WebGPU 的 GPUBuffer 可以从 SAB 直接 mapping，零拷贝传给 GPU

FFmpeg.wasm 解码 4K 视频时，每帧 ~30MB；用 postMessage 复制需要几 ms，用 SAB 是 0ms。

场景 C · 共享状态实时协作（Figma / Excalidraw 风格）

多 Worker 共享 CRDT 文档状态，Atomics 保证一致性：UI Worker 渲染 + 输入 Worker 处理鼠标事件 + 网络 Worker 同步远端更新——三方共享同一份 SAB-backed 文档。CRDT 算法本身的 lock-free 性质 + Atomics 的 SC 语义结合，做到无锁实时协作。

04 典型代码

原子计数器 vs 普通计数器（演示 race）

javascript · 主线程 + 4 Worker 各加 100 万次 
// 期望结果：4_000_000；实际"普通版"会丢更新。

// main.js
const sab = new SharedArrayBuffer(8);
const i32 = new Int32Array(sab);

const workers = [];
for (let i = 0; i < 4; i++) {
  const w = new Worker('inc-worker.js');
  w.postMessage({ sab, useAtomics: true });
  workers.push(w);
}

// inc-worker.js
self.onmessage = (e) => {
  const { sab, useAtomics } = e.data;
  const i32 = new Int32Array(sab);
  for (let i = 0; i < 1_000_000; i++) {
    if (useAtomics) {
      Atomics.add(i32, 0, 1);   // 正确：4_000_000
    } else {
      i32[0] += 1;              // 错误：~700K，丢了大量更新
    }
  }
};
// useAtomics=false 时丢更新的根因：
//   load i32[0] → +1 → store i32[0]  这 3 步可被另一线程的 store 插入

用 Atomics.waitAsync 做 mutex（主线程版）

javascript · 主线程友好的 mutex 
class AsyncMutex {
  constructor(sab) {
    this.state = new Int32Array(sab);  // state[0]: 0=unlocked, 1=locked
  }
  async lock() {
    while (Atomics.compareExchange(this.state, 0, 0, 1) !== 0) {
      const { async, value } = Atomics.waitAsync(this.state, 0, 1, 5000);
      if (async) await value;  // 异步等待 unlock
    }
  }
  unlock() {
    Atomics.store(this.state, 0, 0);
    Atomics.notify(this.state, 0, 1);
  }
}

// 用法：
const mutex = new AsyncMutex(new SharedArrayBuffer(4));
async function criticalSection() {
  await mutex.lock();
  try {
    // 临界区代码
  } finally {
    mutex.unlock();
  }
}

05 常见陷阱

⚠️ 陷阱 1：忘了 COOP/COEP，SAB 构造器抛错

未配置头时 typeof SharedArrayBuffer 可能是 "function"（浏览器假装存在），但 new SharedArrayBuffer(1024) 抛 SecurityError。运行前必须 console.log(crossOriginIsolated) 确认。

规避：CDN / nginx / Cloudflare 上配齐 COOP: same-origin、COEP: require-corp，并把所有子资源（包括 favicon、CDN script）改成同源或加 CORP: cross-origin 头。是配置工程而不是代码工程。

⚠️ 陷阱 2：在主线程 Atomics.wait

直接抛 TypeError: Atomics.wait cannot be called in this context。规范允许（Forward Progress 满足），但宿主（浏览器）禁——会卡 UI。

规避：主线程用 Atomics.waitAsync（ES2024）。Node 主线程可以用 Atomics.wait，但仍不推荐（事件循环会卡）。

⚠️ 陷阱 3：用普通 i32[0] += 1 做并发计数

JS 的 += 是 3 步（read / add / write），中间任何步骤可被另一 Agent 打断。结果是"丢更新"——计数远小于期望。

规避：所有共享内存修改用 Atomics.add / sub / exchange / compareExchange。普通访问只能用于"只读"或"SPSC 中的单方独占字段"。

⚠️ 陷阱 4：以为 postMessage(sab) 是转移所有权

普通 ArrayBuffer 用 postMessage(buf, [buf]) 转移后原方就不能用；SAB 是共享引用，双方同时持有，谁都可以读写。语义完全不同。

规避：心里记住"SAB = 多方共享一份内存"，与 ArrayBuffer 的"transfer 转移所有权"是两套语义。

⚠️ 陷阱 5：嵌入第三方 iframe 后失去 crossOriginIsolated

COEP require-corp 会把所有跨源子资源加严要求。第三方 iframe 没声明 CORP 时，整个页面失去 crossOriginIsolated 状态，SAB 立刻失效。

规避：要么所有第三方资源同步配 CORP；要么用 credentialless COEP 模式（更宽松但有兼容性代价）；要么把第三方放独立子域 + iframe sandbox。

⚠️ 陷阱 6：用 FinalizationRegistry 释放 SAB 内存

FinalizationRegistry 的回调"不保证触发"——GC 时机由引擎决定。把它当作"SAB 内存释放保证"会泄漏。

规避：SAB 由引用计数管理，不能手动释放；要"释放"就得让所有 Agent 解除引用，等 GC 自然回收。详见 P6.2 Reflect 与元编程谱 的 WeakRef 节。
