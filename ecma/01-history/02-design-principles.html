A2 · 设计原则与语言特性 ——「它是什么？」

ECMAScript 的"基因"是三股血脉的合体：Self（原型继承）+ Scheme（一等函数 + 词法作用域）+ Java（C 风格语法外衣）。理解这一点能解释 90% 的"JS 怎么这么古怪"的疑问——它不是一门设计糟糕的语言，而是三种范式的杂交品种，外衣像 Java 但内核完全不是 Java。本章从这个根因出发，覆盖原型对象模型、动态弱类型、词法作用域闭包，以及 ES2015 之后系统补全的几个关键设计：错误处理、迭代协议、模块系统、Symbol 扩展点。

📐 本章交叉规则

本章涉及的具体机制（闭包、原型链、TDZ、Symbol、Iterator、ESM）在 JavaScript 深度学习 已有详细章节。本章对每个机制做 1-3 句话短重述 + 链回，正文集中讲"为什么这样设计、当时有哪些选项、TC39 怎么选的"。如果你对某个机制的运行细节有疑问，跟随链接去 JS 主题获得完整解释，再回来看历史决策。

阅读时长 ~35 分钟 前置 A1 标准发展史 + 任意 JavaScript 经验 对应 ECMA-262 §4 Overview

01 速览

三句话定义 ECMAScript 的设计本质：

  1. JS 不是"类的 Java"，是"原型的 Self"。class 关键字（ES2015）只是原型链 + 构造器函数的语法糖，不是真正的"基于类"系统。
  2. 动态 + 弱类型。类型绑定到值不绑定到变量；任何类型间转换都不抛错（+ 多义、== 抽象相等）；这是"脚本语言宽容度"决策的延续。
  3. 词法作用域 + 一等函数 = 必然有闭包。闭包不是"高级特性"，是这两条决策的必然副产物——规范层根本没"闭包"这个词，它是函数对象 + [[Environment]] 内部槽的组合。

对比看一眼"JS 长得像 Java 但不是 Java"：

java · "类的 Java" 
// Java
class User {
  String name;
  User(String name) { this.name = name; }
  void greet() { System.out.println("Hi, " + name); }
}
User u = new User("Alice");
u.greet();
// 这里 User 是「类型」，u 是「实例」，类型在编译期决定。
javascript · "原型的 Self" 穿着 Java 衣服 
// 表面也叫 class，但底层完全不同
class User {
  constructor(name) { this.name = name; }
  greet() { console.log("Hi, " + this.name); }
}
const u = new User("Alice");
u.greet();

// 实际等价于：
function User(name) { this.name = name; }
User.prototype.greet = function () { console.log("Hi, " + this.name); };
const u2 = new User("Alice");

// 「class」只是函数 + prototype 对象的语法糖。
// User 同时是值（构造器函数）和「类型」（拥有 prototype 的对象）。
// 修改 User.prototype.greet 会立刻影响所有已创建的 u —— 这在 Java 是不可能的。

02 底层逻辑

三大灵感来源（Eich 自述）

Brendan Eich 在 1995 年 5 月用 10 天写出 JS 原型时，参考了三种当时知名的语言。这是有据可查的设计决策，不是后人附会——Eich 在多次访谈和博客中明确列出过：

来源 借了什么 具体表现
Self 原型继承 + 对象 = 属性包 没有 class（只有原型）；对象就是 key-value 集合；动态加属性；__proto__ 链查找
Scheme 一等函数 + 词法作用域 函数是值（可赋值、可传参、可返回）；闭包；let（虽然 ES2015 才正式加）；尾调用；表达式优先
Java C 风格语法外衣 + 营销蹭名 { } 块、; 结尾、function 声明、new 关键字、throw/try/catch、原始类型大小写约定（String vs string）

关键观察：Java 只贡献了语法外衣，不贡献语义。语义全部来自 Self（对象模型）和 Scheme（求值模型）。这就是为什么从 Java/C# 转 JS 的工程师最容易踩坑——语法看起来熟悉，但每个语义细节都是反 Java 直觉的。

这种"借皮不借骨"的设计被 Eich 自己评价为"I had to make it look like Java but be different enough that it complemented Java rather than competed with it"——管理层要求"看起来像 Java"以便营销，但 Eich 把语义留给了 Self + Scheme。结果 JavaScript 在心智层面始终是分裂的："写起来像 Java，跑起来像 Self+Scheme"。

原型对象模型（Self 血脉）

JS 没有"类"，对象就是属性的集合，每个对象有一个内部槽 [[Prototype]] 指向另一个对象（或 null）。属性查找时，如果当前对象没有，沿 [[Prototype]] 链向上找——这就是原型链。

短重述：原型链查找算法叫 OrdinaryGet（ECMA-262 §10.1.8），步骤是"在自己的属性表里找 → 找不到沿 [[Prototype]] 上溯 → 直到 null 返回 undefined"。__proto__ 是访问 [[Prototype]] 的访问器糖（ES2015 才正式纳入规范，之前是事实标准）；每个函数都有 prototype 属性（用于 new 时设置实例的 [[Prototype]]），但 prototype 和 [[Prototype]] 是两个完全不同的东西。

详细机制请参考 JS · 02-advanced/05-prototype（含完整属性查找算法、Shadowing 规则、Object.create 用法）和 06-class（class 语法糖如何 desugar 成原型）。本章只回答设计层面的"为什么"：

💡 为什么是原型，不是类？

1995 年的"轻量脚本语言"语境下，原型有两个决定性优势：① 实现极简——10 天写不出完整的类系统，但能写出原型链；② 动态性强——可以运行时给对象加方法、改方法、删方法（class-based 系统通常静态绑定）。Self 的影响起源是 Eich 想"用最少的语言机制表达最多的抽象"——原型是当时最经济的对象模型。

ES2015 加 class 是为了让 Java 工程师写起来更舒服 + 让 mixin / 多继承等模式有清晰语法，但底层模型从未改变。修改 String.prototype 仍然影响所有字符串实例（monkey patch）；这件事在真正基于类的语言里是做不到的。

动态 + 弱类型

短重述：JS 有 8 个原始类型（Undefined / Null / Boolean / String / Number / BigInt / Symbol + 引用类型 Object）。"动态"指类型绑定到值不绑定到变量，let x = 1; x = "a"; 是合法的。"弱类型"指任何类型间转换都不抛错（"5" + 3 === "53"，"5" - 3 === 2），+ 按操作数类型多义，== 走 11 步抽象相等算法。

详细机制请参考 JS · 01-fundamentals/01-types（8 类型 + 装箱/拆箱机制 + ToPrimitive）。本章回答的"为什么"是：

动态 + 弱类型是"脚本语言"语境的必然选择。1995 年 JS 的目标用户是网页设计师，不是程序员。让他们写 "3" + 2 报错是反生产力的——脚本语言的优先级是"容忍输入、做点合理的事"，而不是"类型纯洁性"。这条原则贯穿 30 年：今天 ES2025 仍然不会"修复" "5" + 3 === "53"，因为修复就是破坏向后兼容。

ECMA-262 §4.3 自述："ECMAScript was originally designed to be used as a scripting language, but has become widely used as a general-purpose programming language"——这一句话解释了所有"宽容到出奇"的设计：自动分号插入（ASI）、== 抽象相等、隐式转 string、对象自动装箱、typeof null === 'object' 等等。每一个都是"脚本宽容度"决策的延续，每一个都因向后兼容无法修复。

TypeScript 的存在本质上是承认"动态弱类型对大型工程不够"，但补丁方式是"编译期叠一层静态约束，运行时仍是弱动态 JS"。详见 TS · The Basics 与类型系统本质。

词法作用域 + 闭包（Scheme 血脉）

短重述：JS 是词法作用域（Lexical Scope）——一个标识符引用哪个变量，由它在源代码中的位置决定，而不是调用栈位置。函数对象在创建时把当时的作用域链记录到内部槽 [[Environment]]，调用时把它作为外层环境——这就是闭包的本质。规范层没有"闭包"这个词，它是"函数 + 词法作用域 + 一等函数"三条决策的必然副产物。

详细机制请参考：

  • JS · 02-advanced/01-scope-tdz —— 作用域链、TDZ、let/const/var 的差异
  • JS · 02-advanced/02-closures —— 闭包内存模型、循环捕获问题、内存泄漏

本章回答的"为什么"是：

📐 PPT 标语："让先声明后使用从最佳实践变成语言强制约束"

ES2015 引入 let/const + TDZ 是 JS 设计史上最重要的"修复型决策"之一。早期 var 的 hoisting 让 console.log(x); var x = 1; 不报错（输出 undefined）——这是 Self/Scheme 都没做的"邪道"妥协，纯粹为了脚本宽容度而设。

let/const 仍然 hoist（规范层叫 HasDeclaration），但访问 binding 时引擎检查"是否已 initialize"，未初始化则抛 ReferenceError——这就是 TDZ（Temporal Dead Zone）。从规范层主动加的"安全网"，把 30 年来 lint 工具反复推荐的"先声明后使用"升级为语言级强制约束。这条决策的影响是：写 ES2015+ 代码几乎不可能再踩 hoisting 坑。

错误处理设计（PPT 未覆盖，规范层补全）

关键事实：try/catch/finally 是 ES3（1999）才加的。ES1/ES2 时期 JS 的错误处理只有"出错就停"——脚本中断，控制台报错，页面其他脚本继续跑。这是为什么老 JS 代码里看不到 try/catch 的根因。

ES3 加上 try/catch 后，又延续了"脚本宽容度"基因：throw 任意值合法（不像 Java 必须 throw new Exception()），throw "string"、throw 42 都行——但你抛了非 Error 对象就丢了 stack trace。

错误类型层级在 ES3 时定下，至今稳定：

javascript · Error 层级 
Error                  // 基类
├── EvalError          // 历史遗留，ES5 后几乎不用
├── RangeError         // 数值越界（new Array(-1)、Number.toFixed(101)）
├── ReferenceError     // 未声明变量、TDZ 访问
├── SyntaxError        // 解析失败（new Function、JSON.parse、await 在非 async）
├── TypeError          // 类型错（null.x、调用非函数、修改 const）
├── URIError           // URI 编解码失败
└── AggregateError     // ES2021 加，Promise.any 全部失败时聚合

ES2015+ 的错误处理演进：

  • ES2018: try { ... } catch { ... }——可以省略 catch 的 binding，对应"我知道会抛但不在乎是什么"的常见模式
  • ES2021: AggregateError + Promise.any——首个聚合多个 reason 的 Error 子类
  • ES2022: Error.cause——构造时传 { cause: e }，让错误链可追溯（替代 lodash 风格的 throw new Error(msg + ': ' + e.message)）

对应 JS 章节：JS · 01-fundamentals/14-errors 详细讲了 Error 子类用途、自定义 Error、async 函数中的错误传播。

迭代协议设计（PPT 未覆盖，规范层补全）

关键事实：ES2015 引入迭代协议（Iteration Protocol）不是为了"加个 for...of 循环"，而是为了用一个统一抽象解决多个特性的"取下一个"问题：

  • for (const x of iterable) —— for-of 循环
  • [...iterable] —— 数组展开
  • const [a, b] = iterable —— 数组解构
  • new Map(iterable)、new Set(iterable) —— 集合构造
  • Promise.all(iterable) —— Promise 组合器
  • yield* iterable —— Generator 委托

这 6 个特性如果各自独立设计接口，就要 6 套 API。统一抽象让任何"可被这些操作处理"的对象只需实现一个标准接口（Symbol.iterator 方法），就能同时支持所有 6 种用法。这是 OOP 经典的"program to interface, not implementation"原则的最大成功案例之一。

javascript · Iteration Protocol 最小实现 
// 一个对象想成为可迭代的，只需实现 [Symbol.iterator]() 方法
// 该方法返回一个迭代器（含 next() 方法，返回 { value, done }）
const range = {
  start: 1,
  end: 3,
  [Symbol.iterator]() {
    let current = this.start;
    const end = this.end;
    return {
      next() {
        return current <= end
          ? { value: current++, done: false }
          : { value: undefined, done: true };
      },
    };
  },
};

// 立刻获得 6 种能力，完全不用各自适配：
for (const n of range) console.log(n); // 1 2 3
[...range]; // [1, 2, 3]
const [a, b] = range; // a=1, b=2
new Set(range); // Set(3) {1, 2, 3}
Promise.all(range); // Promise resolved [1, 2, 3]

详细机制请参考 JS · 01-fundamentals/13-iterators（含 Iterator vs Iterable 区别、Generator 实现迭代器、async iterator）。

ES2025 加了 Iterator Helpers（.map/.filter/.take/.toArray 等），让迭代器有了类似数组的链式 API——这是迭代协议设计 10 年之后的"二期工程"，让懒求值（lazy evaluation）变得 ergonomic。

模块系统设计（PPT 未覆盖，规范层补全）

关键事实：JS 在 2015 年才有原生模块系统。之前 18 年（1997-2015）没有，全靠社区方案：IIFE、CommonJS（Node.js）、AMD（RequireJS）、UMD（兼容多平台）。

ES2015 ESM 的核心设计决策是「静态结构」：

维度 CommonJS（运行时） ESM（静态结构）
加载时机 require() 是函数，运行时调用 import 是声明，parse 阶段解析依赖
位置约束 任意位置（if、函数体内都行） 必须 top-level（除 dynamic import()）
导出绑定 module.exports = {...}，值复制 export，live binding（导出的值变了，导入端跟着变）
循环依赖 各方拿到"截至当前的值"，常出 undefined 规范定义"[[CycleRoot]]"算法，行为可预期
tree-shaking 不可能（运行时函数无法静态分析） 可以（静态结构能确定哪些 export 没用到）

"必须 top-level"这条约束是关键——它让 import/export 关系在 parse 阶段就能确定，从而 tree-shaking、热更新、import maps 等工程优化才成为可能。CommonJS 的 require() 因为是运行时函数，bundler 必须做复杂的静态分析（往往不准确）才能模拟同样能力。

ES2020 加了动态 import() 作为妥协：返回 Promise 的函数式 import，可在任意位置调用，用于代码分割。它不破坏静态结构原则——dynamic import 是"明确标注的运行时导入"，bundler 知道这里要切割代码。

ES2022 加了 Top-Level Await：在模块顶层可以直接 await，模块求值变成异步。这是"静态结构 + 异步初始化"的折中——规范层加了死锁检测算法（cycle + async 模块同时存在时），实现非常复杂（详见 A4 异步演进 + B5 §9 上下文）。

详细机制请参考 JS · 04-modules/01-esm（ESM 完整规则）和 02-cjs-interop（CJS/ESM 互通）。

Symbol 与扩展点设计（PPT 未覆盖）

关键事实：ES2015 引入 Symbol 不只是"unique key"，更核心的目的是给语言提供"可扩展协议钩子"。

Symbol 是一种新原始类型，每次 Symbol() 调用产生独一无二的值（Symbol("x") !== Symbol("x")）。它有两个用途：

  • 用户用法：作为对象 key，避免名字冲突（库内部状态、私有元数据）。但实际工程中 WeakMap 更常用。
  • 规范用法（核心）：Well-known Symbols——预定义的 Symbol，用作"协议钩子"，让用户对象参与语言的内置行为。
Well-known Symbol 语言内置行为
Symbol.iterator for-of / ... / 解构等如何取下一个
Symbol.asyncIterator for await...of 如何取下一个
Symbol.toPrimitive 对象在 +、==、模板字符串中如何转原始值
Symbol.hasInstance x instanceof Y 如何判断（覆盖默认原型链查找）
Symbol.toStringTag Object.prototype.toString.call(x) 输出什么
Symbol.dispose / Symbol.asyncDispose ES2025 using / await using 资源释放（A4 详述）

这是扩展性设计的范本：语言定义"何时调用什么 Symbol"，用户对象通过实现这些 Symbol 方法主动接入语言行为，而不需要修改语言本身。每加一个新协议（如 ES2025 的 dispose），只需新增一个 Well-known Symbol——增量友好，不破坏向后兼容。

详细机制请参考 JS · 01-fundamentals/11-symbol。

03 应用场景

场景 A · 给 Java/C# 工程师讲 JS

最有效的切入点是"原型 vs 类"对比，而不是从语法开始。讲清楚"class 是糖、底层是原型链"之后，许多 JS 怪现象就有根因：

  • "为什么 [].constructor === Array 而不是某个 metadata？" → 因为没有 metadata，constructor 就是 prototype 上的一个普通属性
  • "为什么修改 Array.prototype 影响所有数组？" → 因为它是单一对象，所有数组的原型链都指向它（monkey patch）
  • "为什么 instanceof 在跨 iframe 时会 false？" → 因为每个 Realm 有自己的 Array.prototype，跨 Realm 的实例原型链指向另一个对象（详见 A3 执行模型）

把"语言古怪"重构为"不同的对象模型"，团队迁移阻力会小很多。

场景 B · 写自定义可迭代对象（流式 API）

实现一个分页 API 客户端：让用户写 for await (const page of api.pages())，自动按需翻页。靠的就是 Symbol.asyncIterator 协议：

javascript · 分页客户端 
class PaginatedAPI {
  constructor(url) { this.url = url; }
  async *[Symbol.asyncIterator]() {
    let cursor = null;
    do {
      const res = await fetch(this.url + (cursor ? `?cursor=${cursor}` : ''));
      const data = await res.json();
      for (const item of data.items) yield item;
      cursor = data.nextCursor;
    } while (cursor);
  }
}

const api = new PaginatedAPI('/api/users');
for await (const user of api) {
  console.log(user.name);
  // 自动按需 fetch 下一页，不一次性拉全量
}

用户完全感知不到分页存在，但内存友好——这就是迭代协议设计的力量。

场景 C · 读懂老代码的 IIFE 模块模式

ES2015 之前没有原生模块，很多老库用 IIFE（Immediately Invoked Function Expression）模拟模块：

javascript · 老库的 IIFE 模块 
// jQuery 1.x、Underscore 1.x 都长这样
var MyLib = (function () {
  var privateState = 0;
  function privateHelper() { /* ... */ }

  return {
    publicMethod() { return ++privateState; },
    anotherMethod() { /* ... */ },
  };
})();

MyLib.publicMethod(); // 公开
MyLib.privateState;   // undefined（无法访问）

原理是立刻调用一个函数，函数内部的 var 形成私有作用域，返回的对象只暴露公开方法。这正是"词法作用域 + 闭包"的工程应用——闭包让 privateState 在函数返回后仍然存活，被返回的方法引用。

理解这一段后，老代码读得懂；同时这个模式也是为什么 ES2015 加 let/const + 模块后，IIFE 几乎绝迹——原生语法已经能做到。

04 典型代码

Symbol.toPrimitive 演示对象参与算术

javascript · 自定义对象的算术行为 
class Money {
  constructor(amount, currency) {
    this.amount = amount;
    this.currency = currency;
  }
  [Symbol.toPrimitive](hint) {
    // hint: "number" | "string" | "default"
    if (hint === 'number') return this.amount;
    if (hint === 'string') return `${this.amount} ${this.currency}`;
    return `${this.amount} ${this.currency}`; // default
  }
}

const m = new Money(100, 'USD');
+m;            // 100 (hint = number)
`${m}`;        // "100 USD" (hint = string)
m + 50;        // "100 USD50" (hint = default → string)
m * 2;         // 200 (* 强制 hint = number)

// 这就是 Date 对象支持算术的机制：
// new Date() - new Date('2024-01-01') 返回毫秒差

Class 与原型的等价转换（揭穿语法糖）

javascript · class 完全展开 
// === Class 写法 ===
class Animal {
  static species = 'unknown';
  constructor(name) { this.name = name; }
  greet() { return `Hi, ${this.name}`; }
  static create(name) { return new Animal(name); }
}

class Dog extends Animal {
  constructor(name) {
    super(name);
    this.kind = 'dog';
  }
  greet() { return super.greet() + '!'; }
}

// === 完全等价的"裸原型"写法 ===
function Animal_v2(name) { this.name = name; }
Animal_v2.species = 'unknown';
Animal_v2.prototype.greet = function () { return `Hi, ${this.name}`; };
Animal_v2.create = function (name) { return new Animal_v2(name); };

function Dog_v2(name) {
  Animal_v2.call(this, name);  // ← super(name) 的实质
  this.kind = 'dog';
}
// 把 Dog_v2.prototype 的 [[Prototype]] 设为 Animal_v2.prototype（继承链）
Object.setPrototypeOf(Dog_v2.prototype, Animal_v2.prototype);
// 静态属性继承
Object.setPrototypeOf(Dog_v2, Animal_v2);

Dog_v2.prototype.greet = function () {
  return Animal_v2.prototype.greet.call(this) + '!';  // ← super.greet() 的实质
};

// 两份代码运行行为完全相同：
new Dog('Rex').greet();    // "Hi, Rex!"
new Dog_v2('Rex').greet(); // "Hi, Rex!"

注意：class 并非纯语法糖，有几个微妙差异——class 内部默认 strict mode、不可在定义前调用、方法不可枚举、构造器没 prototype 属性。但底层模型仍是原型链。详细对比见 JS · 02-advanced/06-class。

05 常见陷阱

⚠️ 陷阱 1：把 prototype 和 [[Prototype]] 混为一谈

这两个名字看起来很像，但是完全不同的两个东西：

  • prototype（小写、可见属性）—— 只在函数对象上有。它是一个普通对象，会在 new F() 时被设为新实例的 [[Prototype]]。F.prototype.foo = 1 之后，new F().foo 是 1（沿原型链查到）。
  • [[Prototype]]（双方括号、内部槽）—— 每个对象都有。它是属性查找时的"上溯指针"。访问方式：Object.getPrototypeOf(obj)（推荐）或 obj.__proto__（访问器糖，可读可写）。

两者的关系：new F() 创建实例时把 F.prototype赋给实例的 [[Prototype]]。换句话说，F.prototype === instance.__proto__。

规避：日常用 Object.getPrototypeOf / Object.setPrototypeOf，不直接用 __proto__（虽然合法，但访问器开销 + 可被覆盖）。函数的 prototype 属性永远是普通对象赋值，不会触发原型链。

⚠️ 陷阱 2：以为 class 是新引入的对象模型

不是。class（ES2015）只是语法糖，底层模型仍是原型链 + 构造器函数。这意味着：

  • 修改 String.prototype 仍然影响所有字符串（monkey patch 仍然生效）
  • instanceof 检查的是原型链，不是"名义类型"——跨 Realm 不工作
  • 没有"类的私有方法"——ES2022 的 #field 是真私有（基于 PrivateName 内部槽），但 private 关键字（TS 才有）只是编译期约束

规避：从 Java/C# 来的工程师写 instanceof 检查时一定要想清楚 Realm 边界（详见 A3 执行模型），跨 iframe / vm context / Worker 通信场景用结构判断（鸭子类型）或 Array.isArray 这种 Realm-aware 的 API，不用 instanceof。

⚠️ 陷阱 3：用 == 比较跨类型值

== 走 ECMA-262 §7.2.13 的 11 步抽象相等算法（IsLooselyEqual），结果常违反直觉：

javascript · == 反直觉案例 
"" == 0          // true（"" → 0，再 0 == 0）
"0" == false     // true（false → 0，"0" → 0）
[] == false      // true（[] → "" → 0）
[] == ![]        // true！([] 是 truthy，但 ![] 是 false，... 长故事)
null == undefined // true（特殊规则）
null == 0        // false（特殊规则——只 null/undefined 互等）
NaN == NaN       // false（NaN 不等于自己）

规避：永远用 ===。仅在明确想"null 或 undefined"时用 x == null（这是唯一推荐的 == 用法）。详细 == 算法见 B3 抽象操作（编写中）。

⚠️ 陷阱 4：以为 let/const 不 hoist

let/const 仍然 hoist，但访问会因为 TDZ 抛 ReferenceError。这和 var 的 hoisting（访问得到 undefined）行为不同，但都是 hoisting：

javascript · TDZ 行为 
// var 的 hoisting
console.log(a); // undefined（var 已声明但未赋值）
var a = 1;

// let 的 hoisting + TDZ
console.log(b); // ReferenceError: Cannot access 'b' before initialization
let b = 1;
// 注意错误信息——不是 "b is not defined"，而是 "before initialization"。
// 这证明 b 已经被 hoist（在作用域里），只是处于 TDZ 状态。

// 一个更清晰的证明
let x = 1;
{
  console.log(x); // ReferenceError！不是外层的 1
  let x = 2;      // 因为内层 x 已 hoist，整个块从入口开始就在 TDZ
}

规避：理解"hoisting 是规范层的"作用域注册"，不是"代码搬运""。详见 JS · 02-advanced/01-scope-tdz。
