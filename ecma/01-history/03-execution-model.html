A3 · 执行模型 ——「机器怎么跑？」

A2 讲了 ECMAScript 的"语言特性"——表层用户能看到的那些。本章把它们翻译成"运行时机器结构"：当一段代码真的在跑的时候，规范层定义了哪些数据结构、它们如何嵌套、谁推谁出栈、闭包靠什么活下来、this 从哪来、setTimeout 谁在调度。这是 JS 这台抽象机器的"VM 架构图"，比"语言特性"更接近金属，比"V8 实现"更稳定（V8 重写过 4 次，规范没变）。

⚠️ 本章是最难的一章

Realm / Agent / Job / ExecutionContext / EnvironmentRecord 是规范的"机器术语"，不是"语言特性"。绝大多数 JS 工程师从没遇到过它们——但它们是后续 A4 异步演进、B5 §9 上下文、以及一切"为什么 vm 沙箱要这样写、为什么跨 iframe 的 Array 不等"问题的根因。建议至少读两遍，第二遍带着具体场景（如"我写一个 setTimeout 时发生了什么？"）回头对照。

📐 本章交叉规则

作用域、闭包、this、TDZ 在 JavaScript 主题 已详述。本章对这些机制做1-3 句话短重述 + 链回，正文聚焦"规范层怎么定义这台机器、Host 与 ECMA-262 的责任划分"。

阅读时长 ~45 分钟 前置 A2 设计原则 对应 ECMA-262 §9 Executable Code and Execution Contexts

01 速览

三句话定义 ECMAScript 的运行时机器：

  1. 三层嵌套容器：Agent（≈ "线程"，含独立 Heap / Stack / Job Queue）→ 一个 Agent 含多个 Realm（每个 Realm 有自己的 globalThis 和 Array.prototype 等 Intrinsics）→ 每个 Realm 上跑多个 Job（脚本、Promise reaction、async 函数 step）。
  2. Execution Context（执行上下文）是函数调用栈的最小单元，含 LexicalEnvironment、VariableEnvironment、PrivateEnvironment、this 绑定、[[Realm]] 等内部槽。栈顶是 "running execution context"。
  3. Environment Record（作用域链节点）是变量绑定的存储单元，[[OuterEnv]] 指针构成词法作用域链。闭包就是函数对象在 [[Environment]] 内部槽里记住的那个 Environment Record。

02 底层逻辑

三层嵌套容器

Agent（≈ "线程"）

Agent 是 ECMA-262 §9.7 定义的执行容器。一个 Agent 含：独立的 Heap（对象池）、独立的执行上下文栈、独立的 Job Queue（PromiseJobs + ScriptJobs + 其他 host-defined jobs）、独立的 GC。

实际映射：

  • 浏览器主线程 = 1 个 Agent
  • 每个 Web Worker = 1 个 Agent
  • 每个 ServiceWorker / SharedWorker = 1 个 Agent
  • Node 主线程 = 1 个 Agent
  • 每个 worker_threads.Worker = 1 个 Agent

Agent 之间默认完全隔离——独立堆，无共享变量，只能通过结构化克隆的 postMessage 通信。这就是"JS 是单线程"这句话的精确含义：单 Agent 默认，不是"全局只能有一个 Agent"。

Agent Cluster（共享内存的一组 Agent）

Agent Cluster 是能共享 SharedArrayBuffer 的一组 Agent（§9.8）。同一个 Tab 的主线程 + 所有 dedicated Worker 默认在同一 Cluster；跨 Tab、跨 Origin、ServiceWorker、SharedWorker 通常不在同一 Cluster。

这是 JS 第一次真正引入"共享内存"（ES2017）。配合 Atomics.wait / Atomics.notify，可以做出真正的多线程并发——但要付出 Spectre 攻击面 + 必须 COOP+COEP 头才在浏览器可用的代价。详细异步与并发在 A4 展开。

Realm（≈ "全局环境"）

Realm 是 ECMA-262 §9.3 定义的"独立的全局环境"。每个 Realm 包含：

  • [[Intrinsics]]：一份独立的 Array.prototype / Object.prototype / Function.prototype 等所有内置原型对象（数百个）
  • [[GlobalObject]]：本 Realm 的 globalThis（浏览器是 window，Node 是 global）
  • [[GlobalEnv]]：本 Realm 的全局 EnvironmentRecord
  • [[HostDefined]]：宿主可附加的额外字段（浏览器存 origin、CSP 等）

实际映射：

  • 每个 iframe = 一个独立 Realm（同 Agent，主线程）
  • 每个 vm.createContext()（Node）= 一个独立 Realm
  • 每个 Worker = 一个独立 Realm（也独立 Agent）
  • 同一文档的 main script + module = 共享一个 Realm
⚠️ Realm 隔离 = 跨边界 instanceof 失败

因为每个 Realm 有独立的 Array.prototype，跨 Realm 创建的数组的 [[Prototype]] 链指向另一个对象。所以：

javascript · 跨 iframe 的 instanceof 
// 主页面
const iframe = document.createElement('iframe');
document.body.appendChild(iframe);
const arrFromFrame = new iframe.contentWindow.Array(1, 2, 3);

arrFromFrame instanceof Array;  // false! 因为 iframe 有自己的 Array
Array.isArray(arrFromFrame);    // true（这个 API 是 Realm-aware 的）

// lodash _.isArray 内部就是 Array.isArray，故能正确识别。
// 自己写代码遇到跨 Realm 时也应该用 Array.isArray，不用 instanceof Array。

Job（执行单元）

Job 是 ECMA-262 §9.5 定义的异步执行单元——某段代码"稍后"在某个 Agent 上跑的最小调度单位。Job 的种类：

  • PromiseJobs（微任务）：Promise.then 回调、queueMicrotask、MutationObserver 回调、async 函数恢复点
  • ScriptJobs（脚本初次加载）：每个 <script> 标签的解析+执行被包装成一个 Job
  • HostDefined Jobs（宿主定义）：setTimeout、setImmediate、I/O 回调、UI 渲染——这些不在 ECMA-262 规范里，由 HTML 规范 / Node 各自定义

每个 Agent 跑完当前 Job 后会清空 PromiseJobs 队列（这就是"微任务"在每个宏任务后被批量执行），然后从其他 Job 队列取下一个。详细的 Event Loop 规则在 A4 异步演进 展开（PPT 经典页"Event Loop：宏任务与微任务"）。

Execution Context（执行上下文）

ExecutionContext 是 ECMA-262 §9.4 定义的函数调用栈的最小单元。每次函数调用、每段脚本/模块求值、每次 eval 调用，引擎创建一个新的 ExecutionContext 压入栈；返回时弹出。

每个 ExecutionContext 含以下内部槽（关键字段）：

字段 用途
code evaluation state 当前执行到代码哪一步（生成器/异步函数恢复时回到此）
Function 正在调用的函数对象（脚本/模块时为 null）
[[Realm]] 当前 Realm（决定使用哪份 Intrinsics）
[[ScriptOrModule]] 当前所在的脚本或模块记录（用于 import.meta 等）
LexicalEnvironment 词法作用域链顶——标识符解析从这里开始沿 [[OuterEnv]] 上溯
VariableEnvironment var 声明的存放点。函数顶层 = LexicalEnvironment；eval 在非 strict 时不同
PrivateEnvironment class #field 私有字段的环境（ES2022 新加）

注意：this 不在 LexicalEnvironment 里。this 由 ExecutionContext 自己持有的 "this binding" 提供（具体存在 Function Environment Record 上）——这就是为什么"箭头函数没有自己的 this"——它根本没创建 Function Environment，this 沿外层 ExecutionContext 找。

栈顶的 ExecutionContext 叫 "running execution context"，所有"当前 this"、"当前 Realm"、"当前作用域"的查询都基于它。

💡 "call stack" vs "execution context stack"

日常说的"call stack"（V8 / Chrome DevTools 用的术语）和规范的"execution context stack"基本是同一个东西，但视角不同：

  • call stack：V8 实现层的栈帧（含 register / 临时变量 / 栈指针等运行时数据）
  • execution context stack：规范层的抽象栈（含 LexicalEnvironment 等"逻辑信息"）

两者一一对应，但栈帧含的数据不完全相同。读 V8 源码用 call stack；读 ECMA-262 用 execution context stack。

Environment Record（作用域链节点）

EnvironmentRecord（ECMA-262 §9.1）是变量绑定的存储单元。一个 EnvironmentRecord 含：

  • 一组 binding（变量名 → 值 + 元数据如 strict/initialized/mutable）
  • 一个 [[OuterEnv]] 指针，指向外层 EnvironmentRecord（构成词法作用域链）

规范定义了 5 种 EnvironmentRecord：

类型 用途
Declarative let / const / function / class 等声明的存放点。大部分情况下用这种
Function Declarative 的子类，额外存 this binding、NewTarget、SuperBase
Object 把一个普通对象当作 binding 表（with 语句使用，不推荐；非 strict 全局也用）
Module ES Module 顶层。比 Declarative 多了"立即冻结的 import bindings"机制（live binding）
Global 合体：Object Environment（globalThis 上的属性）+ Declarative Environment（let/const 全局声明）

标识符解析算法（ResolveBinding，§9.4.2）：从当前 ExecutionContext 的 LexicalEnvironment 开始，调 HasBinding(name)；找不到沿 [[OuterEnv]] 上溯；最终到全局环境仍找不到，返回 undefined（取值时抛 ReferenceError）。

这就是闭包的实现：函数对象在创建时，把当时的 LexicalEnvironment 存到自己的 [[Environment]] 内部槽。函数调用时，把这个 [[Environment]] 作为新 ExecutionContext 的外层环境。即使外层函数已经返回（栈帧被弹出），这个 EnvironmentRecord 仍被函数对象引用——因此不会被 GC，闭包就这样"活下来"。

详细机制请参考 JS · 作用域与 TDZ 和 02-closures。本章只回答规范层视角。

一次函数调用的完整生命周期 ⭐

这是 PPT 第三章的精华页，也是本章必须啃下来的核心。下面用一段简单代码，逐行 trace 规范层发生了什么：

javascript · 待 trace 的代码 
'use strict';

let counter = 0;

function makeAdder(x) {
  return function add(y) {
    counter += 1;
    return x + y;
  };
}

const add5 = makeAdder(5);
setTimeout(() => {
  console.log(add5(3));   // 8
  console.log(counter);    // 1
}, 0);

第 1 步：Agent + Realm 建立（脚本加载之前）

  • 浏览器进程为本 Tab 的 V8 实例创建新的 Agent（含独立 Heap、Stack、Job Queue）
  • 本页面的 main frame 创建一个 Realm：分配独立的 Intrinsics（Array.prototype、Object.prototype 等数百个对象）、创建 globalThis（浏览器是 window，自动挂 DOM API）、创建 Global EnvironmentRecord
  • 把 setTimeout / console 等宿主 API 挂到 globalThis（HTML 规范定义，不在 ECMA-262 里）

第 2 步：脚本解析与全局执行

  • 引擎 parse 脚本源码 → 生成 AST → 包装成ScriptJob 投入 Job Queue
  • 事件循环取出 ScriptJob 开始执行：
  • - 创建 Script ExecutionContext 压栈，LexicalEnvironment = VariableEnvironment = Global EnvironmentRecord
  • - 执行 'use strict'：当前 ExecutionContext 标 strict
  • - 处理 let counter = 0：在 Global EnvironmentRecord 的 Declarative 部分新建 binding counter，初始 uninitialized；执行赋值后变 0
  • - 处理 function makeAdder(x) {...}：创建函数对象，[[Environment]] 设为当前 LexicalEnvironment（即 Global Env）；新建 binding makeAdder 指向它
  • - 处理 const add5 = makeAdder(5)：开始函数调用——见下一步

第 3 步：调用 makeAdder(5)

  • 创建 新 ExecutionContext（Function 类型），压栈
  • 新建 Function EnvironmentRecord，[[OuterEnv]] = makeAdder.[[Environment]] = Global Env；this binding 因 strict 设为 undefined
  • 新 ExecutionContext 的 LexicalEnvironment = VariableEnvironment = 这个 Function Env
  • 把参数 x = 5 写入 binding
  • 开始执行函数体

第 4 步：创建内层函数 add（关键 —— 闭包形成）

  • 遇到 function add(y) {...} 表达式 → 创建函数对象
  • 关键：add.[[Environment]] = 当前 LexicalEnvironment（即 makeAdder 的 Function Env）
  • 这一步把 { x: 5 } 这份 EnvironmentRecord 钉在 add 函数对象上——闭包形成
  • return add 把函数对象返回给调用方

第 5 步：makeAdder 返回，但闭包没死

  • makeAdder 的 ExecutionContext 弹出栈
  • 但 makeAdder 的 Function EnvironmentRecord 不被 GC——因为 add5（作为 add 函数对象）的 [[Environment]] 仍引用它
  • 新建全局 binding add5 = 这个 add 函数对象

第 6 步：setTimeout 调度（Host Job 投递）

  • 遇到 setTimeout(callback, 0) —— 这是 Host API，不是 ECMA-262 的
  • HTML 规范的 setTimeout：把 callback 包装成一个 HostDefined Job（task source = "DOM manipulation" 或 "timer"），延迟 0ms 后投入 Agent 的 Job Queue（实际最小延迟 ~4ms）
  • callback 函数本身的 [[Environment]] 设为当前 Lexical Environment（Global Env）
  • 立即返回 timer ID（一个数字）；callback 还没执行

第 7 步：当前脚本结束，事件循环取下一个 Job

  • Script ExecutionContext 弹出栈，栈空
  • 事件循环检查微任务队列（PromiseJobs）—— 空
  • 事件循环取出刚才投递的 setTimeout Job（已到时间），开始执行

第 8 步：执行 setTimeout 回调（箭头函数）

  • 创建新 ExecutionContext，但箭头函数不创建 Function Env——它直接复用外层环境（Global Env）
  • this binding 沿外层找，得到 Global 的 this（strict 下是 undefined）
  • 执行 add5(3)：
  • - 创建新 ExecutionContext + 新 Function Env，[[OuterEnv]] = add5.[[Environment]]（指向第 4 步钉住的那份）
  • - y = 3；执行 counter += 1：沿 OuterEnv 链上溯，在 makeAdder Function Env 找不到 counter，继续上溯到 Global Env 找到，写回 1
  • - 执行 return x + y：x = 5（在 makeAdder Function Env 找到），y = 3（当前 Function Env），返回 8
  • - ExecutionContext 弹出
  • console.log 打印 8
  • 类似地打印 counter = 1

这一段就是 PPT 经典页"一次函数调用的完整生命周期"的文字版。把每一步内化之后，几乎所有"闭包/this/this 丢失/异步执行顺序"的疑问都能用同一套机器语言推理出来。

strict mode 机制（PPT 未覆盖，规范层补全）

'use strict'（ES5 引入）不是表面的"严格模式"标志，规范层把 strict 信息同时存在多处：

  • ExecutionContext.[[Strict]]：当前是否在 strict 代码里
  • Reference Record.[[Strict]]：每个引用记录自己是否 strict（影响赋值时是否抛错）
  • Function.[[ECMAScriptCode]].[[Strict]]：函数自己的 strict（继承自定义点）

strict 影响的语言行为：

  • this 在普通函数调用时为 undefined（非 strict 是 globalThis）
  • delete x（变量）抛 SyntaxError
  • with 语句禁用
  • arguments 不再与命名参数同步（mapped arguments → unmapped）
  • 赋值给 readonly / non-writable 抛错（非 strict 静默失败）
  • eval / arguments / 保留字不能作变量名
  • 八进制字面量 0o 必须写新语法（0123 禁用）

自动 strict 的位置：

  • 所有 ES Module（含 <script type="module">）—— 不需要 'use strict'
  • 所有 class body —— 内部默认 strict
  • 使用 new Function() 创建函数时不自动 strict（需在源码里加）

自 ES2015 开始，几乎所有现代代码都在 strict 上下文里跑——但非 strict 的"sloppy mode"仍然存在，因为老网页脚本必须保持向后兼容。这是 ES1-ES2025 的"双轨制"。

生成器与协程的暂停机制（PPT 未覆盖）

function* 生成器是怎么"暂停"的？规范用一种叫 "suspended ExecutionContext" 的技巧：

  1. 调用生成器函数不立即执行，而是创建一个 Generator 对象（含 [[GeneratorContext]] 内部槽，存当前 ExecutionContext）
  2. 调用 gen.next()：恢复 [[GeneratorContext]] 压回栈顶
  3. 遇到 yield expr：暂停当前 ExecutionContext —— 把它从栈顶移除但不销毁，控制权返回给 gen.next() 的调用方
  4. 下次 gen.next(value)：再次把它压回栈顶，从 yield 处继续，value 成为 yield 表达式的值

这就是协程（coroutine）的实现——ExecutionContext 作为"可挂起的栈帧"。async function 也是基于这个机制（含 await 时挂起 ExecutionContext，Promise resolve 后恢复）。详细见 A4 异步演进。

模块加载与求值顺序（PPT 未覆盖，规范层补全）

ES Module 是 ECMA-262 中最复杂的算法之一，规范用 Source Text Module Record（§16.2）建模。每个模块有 4 个生命周期阶段：

  1. Parse：解析源码 → 收集所有 import / export 声明 → 生成 Module Record（含 [[ImportEntries]] / [[LocalExportEntries]] / [[IndirectExportEntries]] 等字段）
  2. Load (Linking)：递归 fetch 所有依赖模块 → 通过 HostResolveImportedModule hook 让宿主决定怎么找文件 → 构建依赖图
  3. Link：建立 Module EnvironmentRecord，把 import 的 binding 连接到对应导出方的 binding（这就是 live binding——导入端不是值复制，是引用同一个 binding）
  4. Evaluate：按拓扑序执行模块顶层代码 → 模块 namespace object 被冻结

循环依赖处理：规范用 [[CycleRoot]] 字段标识循环模块的"根"——cycle 中所有模块共享一个 [[CycleRoot]]，evaluate 时先深度优先把非循环依赖跑完，遇到 cycle 就接受"当前能拿到什么就拿什么"（未初始化的 binding 暂时是 uninitialized）。这比 CommonJS 的 cycle 处理（拿到截至当前的 module.exports）更严格但更可预期——你能 import 一个 binding 但访问时仍可能抛 ReferenceError（TDZ-like）。

Top-Level Await（ES2022） 让模块求值变成异步过程。规范层加了：

  • [[CycleRoot]] + [[AsyncEvaluation]] 字段共同决定 cycle 中谁等谁
  • 死锁检测：cycle 中两个模块互相 await import() 对方时，规范要求宿主直接 reject 而不是无限挂起
  • 顶层 promise：每个模块 evaluation 现在返回 Promise，整个 import 树是一棵 Promise 依赖树

详细机制请参考 JS · ES Modules（导入/导出语法、live binding 行为）。模块系统的设计决策（为什么是静态结构 vs 运行时函数）已在 A2 设计原则 讨论。

Host 与 ECMA-262 的责任划分

ECMA-262 不管很多事。它只定义"语言核心 + 抽象 Job 调度框架"，具体的 I/O、定时器、UI 渲染、网络、文件——全部由宿主（Host）实现。划分点叫 Host Hook：

Host Hook 用途
HostEnqueuePromiseJob 规范说"把这个 Promise reaction 投递到 microtask queue"，宿主决定怎么投
HostMakeJobCallback 包装一个回调成 Job（浏览器额外保存"正在等待的 promise" / "脚本来源"等元数据）
HostPromiseRejectionTracker 未处理的 reject 怎么报告（浏览器触发 unhandledrejection 事件，Node 输出到 stderr）
HostFinalizeImportMeta 设置 import.meta.url / import.meta.resolve 等
HostResolveImportedModule 处理 import 语句，宿主负责 fetch + parse 模块
HostCallJobCallback 实际调用回调（V8 在这里做调试 hook、Sentry 在这里捕获错误）

这就是为什么浏览器和 Node 行为不一样。同样一段 setTimeout(f, 0)，浏览器 4ms 最小延迟、Node 1ms 最小延迟、Bun 立即；同样一段 queueMicrotask(f)，三者行为一致（因为这是 ECMA-262 规范的，不是 host）。看到不一致行为先想"这个 API 是 ECMA-262 还是 HTML/Node 自己的"——答案常常解释一切。

03 应用场景

场景 A · 调试闭包内存泄漏

React 项目里 useEffect(() => { someBigArray; }, []) 让 someBigArray 一直被持有，明明组件已 unmount。根因：useEffect 的回调函数对象 [[Environment]] 引用了 closure 中的 someBigArray，只要 React 还持有这个回调（如 cleanup 队列），EnvironmentRecord 就不被 GC。

排查方法：Chrome DevTools → Memory → Take Heap Snapshot → 搜索这个变量的 retainers，会看到一条 (closure) 引用链回到 React 内部数据结构——这就是 [[Environment]] 在 GC graph 上的实际表现。

修复：useEffect 不引用大对象，或在 cleanup 里显式置 null，或用 useRef 间接引用。

场景 B · Node vm 模块写沙箱

Node 的 vm.createContext() + vm.runInContext() 是创建独立 Realm 的官方 API，常用于：① 多租户运行用户脚本（serverless / lowcode 平台）② 隔离测试代码 ③ Edge runtime（Vercel Edge、Cloudflare Workers 内部都依赖类似机制）。

javascript · vm 创建独立 Realm 
const vm = require('node:vm');

const sandbox = { x: 1 };
vm.createContext(sandbox);   // sandbox 变成新 Realm 的 globalThis
vm.runInContext('y = x + 1; arr = [1, 2, 3];', sandbox);

console.log(sandbox.y);                // 2
console.log(sandbox.arr);              // [1, 2, 3]
console.log(sandbox.arr instanceof Array);  // false! 跨 Realm
console.log(Array.isArray(sandbox.arr));    // true（Realm-aware）

// 沙箱里能否访问 process / require？取决于 contextify 配置。
// 默认不能，但用户脚本若构造 Function 仍可能逃逸——真正生产隔离要用 isolated-vm 等。

场景 C · jsdom / happy-dom 的 Realm 隔离

Jest / Vitest 的测试环境用 jsdom 或 happy-dom 模拟浏览器 DOM。每个 test file 跑在独立的 jsdom Realm 里——这就是为什么不同 test 的 document 不互相污染。

理解 Realm 之后，"jest.useFakeTimers() 为什么能拦 setTimeout"也好懂了——Jest 把 jsdom Realm 的 globalThis.setTimeout 替换成自己的实现。这是 Realm 隔离 + Host hook 替换的工程组合。

04 典型代码

用 Generator 演示 ExecutionContext 暂停

javascript · 协程式生成器 
function* counter() {
  console.log('  [gen] start');
  let i = 0;
  while (true) {
    const reset = yield i++;   // ← 这里 ExecutionContext 暂停
    if (reset) i = 0;
    console.log('  [gen] resumed, i=', i);
  }
}

const g = counter();           // 不执行函数体，仅创建 Generator 对象
console.log(g.next());         //   [gen] start  | { value: 0, done: false }
console.log(g.next());         //   [gen] resumed, i= 1 | { value: 1, done: false }
console.log(g.next(true));     //   [gen] resumed, i= 0 | { value: 0, done: false }
//                                  ↑ 注意：i 被 reset 了，因为 yield 的"返回值"是 next 传入的 true

// 整个过程中，counter 的 ExecutionContext 反复被压回栈、暂停、压回栈。
// 同一个 ExecutionContext，含同一个 LexicalEnvironment，含 i 这个 binding。
// 这就是协程：一个能挂起/恢复的"栈帧"。

实战观察 Realm 隔离

javascript · 浏览器中跨 iframe 的 Realm 边界 
// 主页面
const iframe = document.createElement('iframe');
document.body.appendChild(iframe);
const iWin = iframe.contentWindow;

// 1. globalThis 不同
console.log(window === iWin);        // false
console.log(globalThis === iWin);    // false

// 2. Intrinsics 不同（关键）
console.log(Array === iWin.Array);                     // false
console.log(Array.prototype === iWin.Array.prototype); // false

// 3. 跨 Realm 的对象 instanceof 失败
const arr = new iWin.Array(1, 2, 3);
console.log(arr instanceof Array);     // false（!!）
console.log(arr instanceof iWin.Array); // true（Realm-correct）
console.log(Array.isArray(arr));        // true（Realm-aware API）

// 4. 但 Realm 之间共享 Agent —— 同步执行，单线程
//    iWin 的脚本和主页面脚本在同一个 ExecutionContext 栈上交替运行。
//    所以才会有"iframe 卡死整个 Tab"的现象。

// 5. 跨 Realm 的对象 typeof 仍然正常
console.log(typeof arr);  // 'object'（typeof 不依赖 Intrinsics）

05 常见陷阱

⚠️ 陷阱 1：以为 this 在 LexicalEnvironment 里

不在。this 由 ExecutionContext 自己持有的"this binding"提供，存在 Function EnvironmentRecord 上（不是 Declarative）。这就是箭头函数没有自己的 this 的本质——箭头函数不创建 Function Env，故没有 this binding 字段，this 沿外层找。

规避：理解 this 是"调用现场决定"的，不是"声明位置决定"的。详细 4 规则见 JS · this 四规则。

⚠️ 陷阱 2：以为闭包"复制"了变量

不复制。闭包共享同一份 EnvironmentRecord 的 binding。经典坑：

javascript · 闭包循环捕获 
// 错的——所有闭包共享同一个 i
const fns = [];
for (var i = 0; i < 3; i++) {
  fns.push(() => console.log(i));
}
fns.forEach(f => f());  // 3 3 3（var 的 i 是函数级，所有闭包指向同一 binding）

// 对的——每次循环创建新 binding
for (let i = 0; i < 3; i++) {
  fns.push(() => console.log(i));
}
fns.forEach(f => f());  // 0 1 2（let 在每次迭代创建新 EnvironmentRecord）

规避：用 let / const（块作用域 + 每次迭代独立 binding），不用 var。详见 JS · 闭包。

⚠️ 陷阱 3：跨 Realm 用 instanceof

如上面"典型代码"所示，跨 iframe / vm / Worker 的对象 instanceof 会失败，因为每个 Realm 有自己的 Intrinsics。常见踩坑：

  • 主页面写 arr instanceof Array 检查从 iframe 传来的数组
  • Node 测试中跨 vm context 检查 Error 类型
  • Web Worker postMessage 传回的对象再做 instanceof 判断

规避：用 Realm-aware API：Array.isArray、typeof x === 'function'、Object.prototype.toString.call(x)（返回 "[object Array]"、"[object Date]" 等，跨 Realm 一致）。或者用结构判断（鸭子类型，看有没有 then 方法判 thenable 等）。

⚠️ 陷阱 4：把 setTimeout(fn, 0) 当"立即执行"

不是立即。setTimeout(fn, 0) 走 Host Job 调度路径，至少要等：① 当前 ExecutionContext 栈清空；② 微任务队列清空；③ 浏览器最小延迟（4ms）。所以下面代码：

javascript · 任务调度顺序 
setTimeout(() => console.log('macro'), 0);
Promise.resolve().then(() => console.log('micro'));
console.log('sync');

// 输出：sync → micro → macro
// 因为：① 同步代码先跑完
//       ② 微任务（PromiseJobs）在每个宏任务后批量清空
//       ③ 宏任务（HostDefined Jobs）下个 tick 才执行

规避：要"下个微任务"用 queueMicrotask 或 Promise.resolve().then，不用 setTimeout(fn, 0)。详见 JS · Event Loop。
