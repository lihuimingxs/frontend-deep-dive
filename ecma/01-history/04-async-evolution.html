A4 · 异步演进 ——「单线程怎么并发？」

JS 是单 Agent 默认（A3 已讲），但 30 年里它把"单线程"做出了相当丰富的并发表达力：回调地狱（ES1-2014）→ Promise（ES2015）→ async/await（ES2017）→ 异步迭代器 + AbortController（ES2018+）→ Top-Level Await + 真共享内存（ES2017+ 2022）→ 资源管理 await using（ES2025）。每一步都是"不打破单线程心智，但让异步代码看起来更像同步"的折中。本章串起这条演化路径，并补足 PPT 没讲的 6 个现代特性。

📐 本章交叉规则

Promise 机制、Event Loop 调度细节、async/await 错误传播在 JS · P3 异步 & 错误 全 8 章已详述。本章对每个机制做1-3 句话短重述 + 链回，正文聚焦"为什么这样演进、规范层怎么改、Host 与 ECMA-262 的责任划分"。

阅读时长 ~40 分钟 前置 A3 执行模型（必读，本章建立在 ExecutionContext / Job 概念上） 对应 ECMA-262 §27 Control Abstraction Objects

01 速览

三句话定义 ECMAScript 的异步基础：

  1. JS 单线程的根因不是 ECMA-262，是 HTML/DOM 设计。DOM 不是线程安全的；如果 JS 多线程，UI 就要锁。Worker 是"用 message passing 绕过共享状态"的折中，SharedArrayBuffer 是后来开的"共享内存逃生舱"。
  2. Event Loop 是 HTML 规范定义的，不是 ECMA-262 定义的。ECMA-262 只定义"Job 抽象"，HTML 规范用 Job 实现"宏任务/微任务/rendering steps"。这就是浏览器和 Node 行为不同的根因。
  3. async/await 不是"暂停函数"，是 Generator 的语法糖。底层用 A3 讲的 ExecutionContext 暂停机制——await 处把 ExecutionContext 挂起，Promise resolve 后通过 microtask 恢复。这就是为什么 async 函数 stack trace 不连续。

5 代异步模型对比：

代 ES 版本 代表 API 核心抽象
第 1 代 ES1-ES5（1997-2009） setTimeout / Node callback "callback hell"——错误处理靠约定，组合靠嵌套
第 2 代 ES2015 Promise "延迟值的容器"——状态机 + 链式 then + 错误冒泡
第 3 代 ES2017-2018 async/await + for await...of "同步看起来的异步代码"——Generator 状态机 + 异步迭代
第 4 代 ES2017-2022 SharedArrayBuffer + Atomics + Top-Level Await "真并发突破口"——共享内存 + 模块级异步
第 5 代 ES2025 using / await using + Promise.try "资源管理 + 异步同步桥"——栈式资源释放、统一同步异步入口

02 底层逻辑

第 1 代 · 回调时代（ES1-2014）

1995-2014 年 JS 的异步只有一种姿势：把回调函数传给宿主 API，等事件触发时被调用。setTimeout（ES1 时代就有但不在规范里，HTML 规范才加）、XMLHttpRequest.onload、Node 的 fs.readFile(path, callback)。

致命问题不是"难看"，是错误处理与组合极差：

  • 错误传播靠约定：Node 的 "(err, result) => ..." 第一参数是 error，但没有强制——你忘了检查 err，错误就静默丢失
  • 组合靠嵌套：3 个串行异步 = 3 层嵌套，5 个就是金字塔。社区方案 async.js / Step.js 长期未能成为事实标准
  • 没有取消机制：setTimeout 可以 clearTimeout，但其他异步操作各自有各自的取消方式（XHR.abort、流的 destroy）

这个时代留下的遗产：Node 内置 API 至今保留 callback 形式（fs.readFile + fs.promises.readFile 双轨制），所有底层 host API 仍然是 callback（DOM 事件、定时器）。Promise 是包在它们之上的，不是替代。

Event Loop 的责任划分（HTML 规范 vs ECMA-262）

这是整个异步话题最关键的一刀。Event Loop 不在 ECMA-262 里——ECMA-262 §9.5 只定义抽象的 "Job" 概念。具体的 Event Loop（含宏任务/微任务/渲染时机）由HTML 规范（WHATWG）定义，Node 自己也有一份不同的 Event Loop 实现。

责任划分清单：

职责 谁定义的 具体内容
Job 抽象 ECMA-262 §9.5 "Job 是 Agent 上的执行单元"——纯抽象，无具体队列
HostEnqueuePromiseJob ECMA-262 + Host 规范说"调用 Host hook 投递 PromiseJob"，具体投递方式由 Host 决定
microtask queue HTML 规范 定义 microtask 是 "HostEnqueuePromiseJob 投递 + queueMicrotask + MutationObserver" 的统称
macrotask（task source） HTML 规范 定义 task source（"DOM manipulation"、"timer"、"network"...），各自独立队列
rendering steps HTML 规范 每个 task 后清空 microtask → 然后浏览器决定是否 reflow/paint（约 60fps 一次）
setTimeout / setInterval HTML 规范 "timer" task source；最小延迟（浏览器 4ms、Node 1ms、Bun < 1ms）
process.nextTick Node 专属 不在任何规范里——是 Node 自己的"比 microtask 更早"队列

这就是为什么 process.nextTick 在浏览器里是 undefined，为什么 Node 18 之前 setTimeout 在某些场景比浏览器快，为什么 Bun 的 setTimeout 几乎立即执行——它们各自实现 Host 部分。但 Promise.then / queueMicrotask 行为在三者一致，因为这部分 ECMA-262 锁死了语义。

详细 Event Loop 调度规则请参考：

  • JS · 03-async/01-event-loop —— Event Loop 完整 6 步循环
  • JS · 02-microtask-macrotask —— 宏任务/微任务的精确调度顺序
  • JS · 08-node-eventloop-errors —— Node 的 6 阶段 Event Loop 与浏览器差异

第 2 代 · Promise（ES2015）

短重述：Promise 是状态机——pending 不可逆地迁移到 fulfilled 或 rejected。.then(onFulfilled, onRejected) 注册回调，错误自动沿链冒泡到最近的 onRejected。规范用 PromiseReaction Records（ECMA-262 §27.2.1.1.1）存储待执行的 reaction，settle 时通过 HostEnqueuePromiseJob 投递到 microtask queue。

详细机制请参考 JS · 03-async/03-promise。本章只回答规范设计的"为什么"：

💡 Promise 不是"异步"，是"延迟值的容器"

常见误解："Promise 让代码异步"。不对——Promise 包同步值仍然合法（Promise.resolve(42)），then 仍然异步执行（通过 HostEnqueuePromiseJob 投递到下个 microtask）。Promise 的核心抽象是"表达一个还没拿到的值"，异步只是这个抽象最常见的用法。

这个区分很重要——理解之后就懂了为什么 Promise.resolve().then(f) 比 setTimeout(f, 0) 快（前者是 microtask，后者是 macrotask + 最小延迟），为什么"包同步代码进 Promise 也是有用的"（统一错误处理路径）。

"Resolved ≠ Settled"陷阱：resolve(p) 时，如果 p 是另一个 Promise，外层 promise 进入 "resolved" 状态（已绑定到 p）但不是 settled——它要等 p settle 后才 settle。这是规范层精确区分的术语，PPT 第四章专门点出。

Promise 组合器

短重述：5 个标准组合器，按"短路策略"区分：

组合器 ES 版本 短路策略
Promise.all ES2015 任一 reject 立刻 reject；全部 fulfill 才 fulfill（结果数组）
Promise.race ES2015 任一 settle 立刻同状态 settle（最快者胜）
Promise.allSettled ES2020 不短路；全部 settle 后 fulfill（结果是 {status, value | reason} 数组）
Promise.any ES2021 任一 fulfill 立刻 fulfill；全部 reject 才 reject（AggregateError）
Promise.withResolvers ES2024 不是组合器，是构造工具：返回 { promise, resolve, reject }，避免在 executor 里捕获作用域

设计上的微妙点：Promise.all "一个失败短路其他还在跑"——未完成的 promise 不会被取消，它们的 fetch / 计算继续消耗资源直到自然完成。Promise 设计时故意没加取消机制（不可逆性是核心保证），这后来催生了 AbortController（详见下面）。

详细机制与配方请参考 JS · 03-async/04-promise-combinators。

第 3 代 · async/await（ES2017）—— Generator 的语法糖

关键事实：async function 不是"新的暂停机制"，是 ES2015 的 Generator 加 Promise 的语法糖。底层用 A3 讲的 ExecutionContext 暂停机制实现：

  1. 调用 async 函数 → 创建 ExecutionContext + 立即返回一个 pending Promise
  2. 同步执行函数体直到遇到 await expr
  3. 对 expr 求值 → 用 Await 抽象操作（§27.7.5.3）：把 ExecutionContext 挂起（如 Generator 那样）
  4. 给这个值（如果是 Promise）注册一个 reaction："resolve 时恢复 ExecutionContext，rejected 时让 await 抛错"
  5. reaction 在 microtask queue 被处理时 → 把 ExecutionContext 压回栈、继续执行

这就是为什么每个 await 都至少花一个 microtask，即使 await Promise.resolve(1)——因为恢复必须走 reaction 调度。也是为什么 async 函数的 stack trace 不连续——await 后续代码作为新 microtask 执行，新栈帧没了之前的调用上下文（V8 后来加了"async stack traces"补救，但本质上是 V8 实现的 hack，不是 JS 规范的）。

错误传播规则：

  • async 函数体抛错 → 转成 reject，函数返回的 Promise rejected
  • await 一个 rejected Promise → 抛错，可被外层 try/catch 捕获
  • 在 async 函数里抛错不会冒泡到调用方的 try/catch——必须 await 调用
  • finally 在 async 函数里异步执行——它在最后一个 await 之后的 microtask 里跑

详细机制与陷阱请参考 JS · 03-async/05-async-await。

异步迭代器（ES2018，PPT 未覆盖）

ES2015 的迭代协议（A2 讲过）解决了"同步取下一个"。ES2018 加 异步迭代协议——本质是把"取下一个"返回的值改成 Promise：

javascript · 同步 vs 异步迭代器签名 
// 同步迭代器（ES2015）
interface Iterator<T> {
  next(): { value: T, done: boolean };
}

// 异步迭代器（ES2018）
interface AsyncIterator<T> {
  next(): Promise<{ value: T, done: boolean }>;
}

// 用 Symbol.asyncIterator 标识可异步迭代
// 用 for await...of 消费
for await (const chunk of stream) {
  // 每次循环 await 一次 next()
}

ES2018 还引入async generator（async function*），可以同时用 yield（产出值）和 await（等异步资源）。这是 ECMAScript 第一次把两个独立的暂停机制（generator + async）组合在一起，规范层非常复杂（[[GeneratorContext]] + Promise reaction 双重挂起）。

典型用途：处理 SSE（Server-Sent Events）、HTTP/2 streams、数据库 cursor、文件分块读取、分页 API（A2 已演示）。配合 node:stream/web 的 ReadableStream 可直接做 for await。

AbortController 与 Promise 不可逆性的张力（PPT 未覆盖）

Promise 设计时故意没加取消——状态不可逆是它的核心保证。但实际工程中"用户切到下一页，把上一页的请求都取消"是高频需求。社区先用各种 hack 解决（Promise.race + 超时 promise、cancelable wrapper），最终方案是外加一个独立的取消信号机制：AbortController（DOM 规范，2017 引入；ES 没有，但被 Node 在 2021 也实现）。

javascript · AbortController 的核心 API 
const ctrl = new AbortController();

// signal 是一个事件源：触发 'abort' 事件
fetch('/api/data', { signal: ctrl.signal })
  .then(r => r.json())
  .catch(err => {
    if (err.name === 'AbortError') {
      // 被取消了
    }
  });

// 调用 abort() → signal 触发 'abort' → fetch 内部检测到，reject 返回 Promise
ctrl.abort();

// 多个操作共享一个 signal（取消传播）
function doWork(signal) {
  return fetch('/a', { signal })
    .then(() => fetch('/b', { signal }))
    .then(() => processData(signal));   // 自定义函数也要支持 signal
}

设计上的微妙点：

  • 取消是"信号"，不是"动作"——AbortController 不主动 reject Promise；它发信号，Promise 的执行方（fetch 内部、自定义函数）必须主动监听 signal 并 reject。这保留了 Promise 不可逆性。
  • 取消会变成 reject，不是 cancel——抛 AbortError（DOMException）。所以 try/catch 必须区分"取消"和"真正失败"。
  • ES2024 加 AbortSignal.any([...])——多个 signal 任一 abort 就 abort，方便组合超时 + 用户取消。

详细用法请参考 JS · 03-async/06-abort-controller。

第 4 代 · Top-Level Await + 共享内存

Top-Level Await（ES2022）

ES Module 的顶层可以直接 await，模块求值变成异步过程：

javascript · Top-Level Await 
// db-config.mjs
const config = await fetch('/api/config').then(r => r.json());
export default config;

// app.mjs
import config from './db-config.mjs';   // ← 这里隐式等待 config 模块求值完
console.log(config);

规范层的代价（A3 模块加载部分提过）：

  • 每个模块的 evaluation 现在返回 Promise
  • 整个 import 树是一棵 Promise 依赖树——某个模块的 TLA 阻塞所有依赖它的下游模块
  • cycle + async 模块组合 → 死锁检测：规范用 [[CycleRoot]] + [[AsyncEvaluation]] 检测两个互相 await 的循环模块，宿主直接 reject 而不是无限挂
  • 不能在普通 script 里用（必须 type="module"）；瀑布加载风险——一个慢 TLA 拖累整棵树

SharedArrayBuffer + Atomics（ES2017）

这是 JS 第一次真正引入"共享内存"——多个 Agent（主线程 + Worker）能同时读写同一段内存，不是 postMessage 的复制语义。配合 Atomics.add/sub/load/store（保证原子性）和 Atomics.wait/notify（多线程信号量），可以做出真正的多线程并发。

但 2018 年 Spectre 攻击让所有浏览器禁用了 SharedArrayBuffer——因为它配合高精度 timer 能用来构造侧信道。2020 年才回归，要求页面发送 COOP（Cross-Origin-Opener-Policy） + COEP（Cross-Origin-Embedder-Policy） 头才解锁。Node 没这个限制（不在浏览器沙箱里）。

实际用途：WebGPU compute shader 与 JS 通信、FFmpeg.wasm（视频转码）、Squoosh（图片压缩）、SQLite WASM、像素级图像处理。是计算密集型 Web 应用的关键基础设施。

第 5 代 · 资源管理 using / await using（ES2025）

ES2025 加的资源管理 API是"异步同步桥"的最新答案。基于 A2 讲过的 Symbol 扩展点机制：

  • Symbol.dispose —— 同步资源释放钩子
  • Symbol.asyncDispose —— 异步资源释放钩子
  • using x = resource() —— 同步声明，离开块作用域时自动调 x[Symbol.dispose]()
  • await using x = asyncResource() —— 异步声明，离开块作用域时 await x[Symbol.asyncDispose]()
javascript · ES2025 await using 简化资源管理 
async function processFile(path) {
  await using file = await open(path);   // 异步打开
  // ... 用 file 做事
}                                          // 自动 await file[Symbol.asyncDispose]()，不管成功还是抛错
                                           // ↑ 替代了 try/finally + await close()

// 标准库已实现：
// - FileHandle 实现 [Symbol.asyncDispose]
// - ReadableStream / WritableStream 实现
// - 数据库连接、HTTP 流、Lock 等都跟进

这是 RAII（Resource Acquisition Is Initialization）模式的 JS 版本，在 C++ / Rust / C# using 已是标配。ECMAScript 等了 30 年才引入——因为单线程心智下"不写 finally 也很少出大问题"，需求积累足够才进规范。

配套的还有 Promise.try（ES2025）—— 把"可能同步可能异步抛错的函数"统一成 Promise，去掉 new Promise(resolve => resolve(fn())) 的样板：

javascript · Promise.try 统一同步异步入口 
// 旧写法：要小心 fn 可能同步抛错
Promise.resolve().then(() => fn(args));   // 同步错也包成 reject 了，但要写一层

// ES2025 新写法
Promise.try(fn, args);                     // 同步抛错也变 reject，简洁

03 应用场景

场景 A · 排队 / 限流（手写 PQueue）

前端要批量上传 100 个文件但浏览器只允许同源 6 个并发——用 p-limit 或自己写一个并发控制器。核心是用 Promise 链把多个 await 串成"令牌池"：

javascript · 极简并发控制 
function limit(concurrency) {
  let running = 0;
  const queue = [];
  const next = () => {
    if (running >= concurrency || !queue.length) return;
    running++;
    const { fn, resolve, reject } = queue.shift();
    Promise.resolve()
      .then(fn)
      .then(resolve, reject)
      .finally(() => { running--; next(); });
  };
  return (fn) => new Promise((resolve, reject) => {
    queue.push({ fn, resolve, reject });
    next();
  });
}

const limiter = limit(6);
const results = await Promise.all(
  files.map(f => limiter(() => uploadFile(f)))
);

读懂这段就懂了 p-limit / pQueue 的 80% 实现。详细变体见 JS · 07-concurrency。

场景 B · 取消异步（路由切换、组件卸载）

React 组件切走时取消正在请求的 fetch、停止订阅 SSE、撤销长查询。用 AbortController:

javascript · React 中的 AbortController 
function useApi(url) {
  useEffect(() => {
    const ctrl = new AbortController();
    fetch(url, { signal: ctrl.signal })
      .then(r => r.json())
      .then(setData)
      .catch(err => {
        if (err.name === 'AbortError') return;
        setError(err);
      });
    return () => ctrl.abort();   // unmount 时取消
  }, [url]);
}

这个模式是 React Query / SWR / TanStack Router 内部都用的——signal 通过参数透传，每层异步操作都有机会响应取消。

场景 C · WebGPU compute（多线程 + 共享内存）

浏览器里跑 ML 推理（Transformers.js、ONNX Runtime Web）—— GPU 计算 + CPU 后处理通过 SharedArrayBuffer 通信，避免每次 GPU→CPU 复制 GB 级张量。需要：

  • 页面发 Cross-Origin-Opener-Policy: same-origin + Cross-Origin-Embedder-Policy: require-corp
  • 检测 crossOriginIsolated === true 后才 new SharedArrayBuffer(N)
  • 用 Atomics.wait/notify 实现 worker 间同步

失败回退到普通 ArrayBuffer + postMessage（性能下降 10x 但仍能跑）。

04 典型代码

用 await using 简化 try/finally

javascript · ES2025 vs 老写法 
// 旧写法：try/finally + 显式释放
async function readJSON_old(path) {
  const file = await open(path, 'r');
  try {
    const text = await file.readFile('utf8');
    return JSON.parse(text);
  } finally {
    await file.close();   // 即使抛错也保证关闭
  }
}

// ES2025 写法
async function readJSON_new(path) {
  await using file = await open(path, 'r');
  const text = await file.readFile('utf8');
  return JSON.parse(text);
  // 离开块时自动 await file[Symbol.asyncDispose]() —— 即使抛错也释放
}

// 多资源叠加更明显
async function copy_new(src, dst) {
  await using src = await open(srcPath, 'r');
  await using dst = await open(dstPath, 'w');
  // 自动按 LIFO 顺序 dispose dst, src（栈式释放）
  await pipeline(src.readable, dst.writable);
}

实测 microtask vs macrotask 调度顺序

javascript · 异步执行顺序经典面试题 
console.log('1 · sync');

setTimeout(() => console.log('5 · macro setTimeout'), 0);

Promise.resolve().then(() => {
  console.log('3 · micro promise.then');
  Promise.resolve().then(() => console.log('4 · nested micro'));
});

queueMicrotask(() => console.log('2 · micro queueMicrotask 但比上面晚注册'));

console.log('? · sync end');

// 输出顺序：
//   1 · sync
//   ? · sync end
//   3 · micro promise.then         ← Promise.then 先注册
//   2 · micro queueMicrotask 但比上面晚注册   ← 后注册
//   4 · nested micro                ← 嵌套 microtask 也在这一轮清空
//   5 · macro setTimeout             ← 所有 microtask 清完才进 macro

// 关键：每轮 macrotask 跑完 → 把 microtask queue 完全清空 → 然后 rendering steps → 下个 macrotask

05 常见陷阱

⚠️ 陷阱 1：在非 async 函数里用 await

await 在以下两种位置之外用都是 SyntaxError（解析期，不是运行期）：

  • async 函数体内
  • ES Module 的顶层（Top-Level Await，要求 type="module"）

规避：在 forEach 回调里用 await 不报错但不生效（forEach 不等 promise）——改用 for...of + await，或 Promise.all + map。

⚠️ 陷阱 2：Promise.all 短路时未完成请求继续消耗资源

Promise.all([a, b, c]) 中 a 失败立刻 reject，但 b 和 c 不会被取消——它们的 fetch / 计算继续跑到自然完成或失败。这是 Promise 不可逆性的代价。

规避：要可取消，必须配 AbortController。Promise.all([a, b, c]) 失败时 ctrl.abort() 取消其他正在跑的。或用 Promise.allSettled 不短路、自己处理结果。

⚠️ 陷阱 3：microtask 死循环把 UI 卡死

HTML 规范要求每个 macrotask 后清空整个 microtask queue 才进 rendering steps。如果 microtask 互相投递新 microtask，UI 永远不渲染：

javascript · UI 永远卡死 
function spam() {
  Promise.resolve().then(spam);   // 每个 microtask 又投递新 microtask
}
spam();
// 浏览器：UI 完全冻结，Tab 必须强制关闭
// 即使 setInterval(() => render(), 16) 也跑不到——因为永远清不完 microtask

规避：递归调度用 setTimeout 或 requestAnimationFrame（macrotask）。如果业务必须用 microtask 链，每隔 N 次让一次 macrotask（"yield to event loop"）。

⚠️ 陷阱 4：SharedArrayBuffer 跨 origin 必须 COOP/COEP

自 2020 年起浏览器默认不允许跨 origin 用 SharedArrayBuffer。要解锁必须发 HTTP 头：

  • Cross-Origin-Opener-Policy: same-origin
  • Cross-Origin-Embedder-Policy: require-corp 或 credentialless
  • 所有第三方资源（图片、iframe、脚本）也必须发 Cross-Origin-Resource-Policy，否则页面 break

规避：检测 crossOriginIsolated === true 决定走 SAB 还是回退路径。如果你的应用有大量第三方嵌入（广告、Tab 等），COOP/COEP 改造成本可能比预期高。

⚠️ 陷阱 5：async 函数里抛错不冒泡到调用方 try/catch

async 函数总是返回 Promise，函数体抛错变成 reject。所以这样写无效：

javascript · 错误丢失 
async function fetchData() {
  throw new Error('boom');
}

try {
  fetchData();   // 没 await！函数返回 rejected Promise，但 try 不知道
} catch (e) {
  // 永远不会进这里
}
// 控制台：UnhandledPromiseRejection: Error: boom

// 正确：必须 await 或显式 .catch
try {
  await fetchData();
} catch (e) {
  console.log(e.message);  // "boom"
}

规避：调用 async 函数要么 await，要么 .catch。"fire-and-forget" 也要至少 .catch(console.error) 兜底。
