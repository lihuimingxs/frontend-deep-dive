Next.js 项目史（2016-2026）

Next.js 10 年的演化主线，不是按版本号背，而是按"每次范式转移及其根因"。从 2016 年 Vercel（当时叫 Zeit）开源 1.0 那天起，每一次断代式重写都是对前一版具体痛点的回应——getInitialProps 被 getStaticProps 取代是因为"运行时 SSR 太慢"，App Router 取代 Pages Router 是因为 RSC 需要不同的渲染心智，v15 缓存反转是因为 v13-14 的 cache-by-default 在生产引爆 stale 数据事故。把这条线拉直，后面每一章的"为什么这么设计"都不再是天书。

阅读时长 ~30 分钟 前置 写过 React + 用过 Next.js（任意版本） 主要来源 nextjs.org/blog 各版本 release notes + Vercel 工程 blog + Lee Robinson / Tim Neutkens 公开演讲

01 速览

  1. Next.js 10 年由 6 次断代式重写组成 —— 1.0 getInitialProps（2016）→ 9 TS + API Routes（2019）→ 9.3 getStaticProps + 9.5 ISR（2020）→ 13 App Router + RSC + Turbopack alpha（2022）→ 15 React 19 + 缓存反转（2024）→ 16 Turbopack/PPR GA + Cache Components（2026）。每次都是回应上一版的具体生产痛点。
  2. 2016 年 Next.js 1.0 解决"React + SSR 配置地狱" —— 当时单页 React 应用上线要自己拼 webpack + babel + express + react-dom/server，配置上千行；Next.js 把这套打包成一个二进制命令 + 一组约定（pages 目录 = 路由），走红的根因是"开箱即用"。
  3. 2019 年 Next.js 9 把它扩展为全栈框架 —— TypeScript 一等支持 / API Routes（pages/api/* 直接是 HTTP handler）/ 自动静态优化。从这一版起 Next.js 不再只是"SSR 工具"。
  4. 2020 年 Next.js 9.3 + 9.5 引入 SSG + ISR —— getStaticProps（build-time 渲染）/ getServerSideProps（每请求渲染）/ Incremental Static Regeneration（按需后台再生）。"一个项目里混合三种渲染模式"成为现实，jamstack 风潮被推上顶峰。
  5. 2022 年 Next.js 13 是最大的断代式重写 —— App Router beta + RSC（React Server Components）+ Turbopack alpha + 新 image/font。Vercel 在 RSC 还是 RFC 阶段就押宝并把整个路由系统围着它建，这是 Next.js 与 Remix 路线分叉的根本点。
  6. 2023 年 Next.js 14 把 Server Actions 推到 stable —— 'use server' 函数 + 表单一体化；同时 Partial Prerendering（PPR）以 experimental 出现，预告了 v16 的 Cache Components 方向。
  7. 2024 年 Next.js 15 完成两件大事：React 19 + async request APIs（cookies/headers/params 全 Promise），以及缓存默认值反转——fetch 不写选项从 cache-by-default 改回 explicit-opt-in。这次反转是迟到的修正，也是 Next.js 历史上承认设计错误最公开的一次。
  8. 2026 年 Next.js 16 让 Turbopack 在 dev + prod 双 stable，PPR 正式 GA，Cache Components 用 'use cache' 指令重做整个缓存心智。Webpack 被完全取代；五层缓存模型被简化为"静态壳 + 动态洞"两层概念。
  9. 纵观 10 年，Next.js 重构频率高于绝大多数同代框架 —— 这不是工程混乱，是 Vercel 一贯的"每隔 2-3 年押注下一代 React 特性"策略：永远是 React RFC 的首发实现者，享受先发红利但承担生产试错成本。
  10. 2026 的稳态：新项目里 Next.js 占比 60-70%（含 Vercel 部署生态），15-20% 走 React Router v7，其余分给 TanStack Start / Astro / 纯 SPA。Next.js 未来的差异化点不再是渲染模式（PPR 已 GA），而是 AI SDK 集成 + Cache Components 心智成熟。

02 底层逻辑

前史 · 2013-2016 · React 时代的 SSR 配置地狱

2013 年 Facebook 开源 React，SPA（Single Page App）成为主流。但 SPA 有几个固有问题：首屏白屏、SEO 不友好、慢网络下感知差。解决方案是 SSR（Server-Side Rendering）—— React 0.14（2015）发布 react-dom/server 让组件能在 Node 里渲染成 HTML。

但要把 SSR 跑起来，2015-2016 年的开发者要自己组装：

  • Webpack 配置：双份（client bundle + server bundle）+ code splitting + dynamic import + CSS extraction
  • Babel 配置：JSX + ESM + Stage-X 提案 + browser target
  • 路由：自己定义 server-side route → React 组件映射
  • data fetching：自己设计 server 端 fetch + serialize 到 client + hydration 时不重复请求
  • express / koa server：handle SSR + static asset + API
  • HMR / dev server：自己装 webpack-dev-middleware + react-hot-loader

结果是每个团队都重复造轮子，配置文件动辄上千行。社区涌现各种 boilerplate（react-starter-kit / react-server / NextAlbum 等），但都不是"框架"，更像"模板"。

2016-10 · Next.js 1.0 · "约定优于配置"打破 SSR 地狱

Vercel（当时叫 Zeit）的 Tim Neutkens、Naoyuki Kanezawa、Guillermo Rauch 在 2016 年 10 月发布 Next.js 1.0。设计立场清晰：

  • pages/foo.js = URL /foo，文件系统即路由
  • 每个 page 是一个 React 组件，组件 export 一个静态方法 getInitialProps 来 fetch 数据
  • 同一份组件代码在 server 和 client 都跑（universal rendering）
  • 开箱即用：next dev / next build / next start，配置文件 next.config.js 可选
javascript 
// pages/posts/[id].js (Next.js 1-8 风格)
import React from 'react';

export default class Post extends React.Component {
  static async getInitialProps({ query }) {
    const res = await fetch(`https://api.example.com/posts/${query.id}`);
    return { post: await res.json() };
  }
  render() {
    return <article>{this.props.post.title}</article>;
  }
}

1.0 走红的根因不是技术先进，是"取消选择负担"。开发者不再需要决定用 webpack 还是 rollup、用 express 还是 koa、用什么路由库、怎么处理 SSR + hydration。Next.js 全替你定了。

但 1.x-8.x 时代也埋下了几个长期痛点：

  • getInitialProps 的运行时 SSR 模式 —— 每次请求都跑一次 fetch，CPU + 延迟成本高，scale 上限低
  • getInitialProps 同时跑在 server 和 client（hydration 时复用），错误模型复杂
  • 没有 build-time 渲染（SSG），完全做不了 jamstack 那种"build 时生成全部 HTML 上 CDN"
  • API route 概念不存在，需要自己搭 express server

2019-07 · Next.js 9 · 全栈化与自动静态优化

2019 年 7 月发布的 Next.js 9 是第一次大规模能力扩展。三个关键特性：

TypeScript 一等支持 —— 之前要装 @zeit/next-typescript 插件，9 起开箱可用。这一步把 Next.js 推向"企业级默认"。

API Routes —— pages/api/foo.js 直接是 HTTP handler，省掉自建 express server：

javascript 
// pages/api/users.js (Next.js 9 风格)
export default function handler(req, res) {
  if (req.method !== 'POST') return res.status(405).end();
  res.json({ id: 123, name: req.body.name });
}

Automatic Static Optimization —— 没有 getInitialProps 的页面会在 build 时被预渲染成静态 HTML，运行时直接从磁盘 / CDN 返回，不走 Node SSR。这是 Next.js 第一次有"build-time 渲染"概念。

从 9 起 Next.js 不再只是"SSR 工具"，而是"全栈 React 框架"。竞品 Gatsby 在内容站点领域强势（jamstack），Next.js 9 的扩展直接对标，开始侵蚀 Gatsby 的地盘。

2020-03 · Next.js 9.3 · getStaticProps + getServerSideProps 分家

2020 年 3 月的 9.3 版本是渲染模式的分水岭。getInitialProps 被两个新 API 取代：

  • getStaticProps —— build 时跑一次，生成静态 HTML + 数据 JSON；运行时从 CDN 返回
  • getServerSideProps —— 每次请求跑一次，server 渲染（与 getInitialProps 类似但只在 server 执行）

这次拆分的根本动机：同一个项目里，不同 page 适合不同渲染模式。一个电商站，首页 + 商品详情应该 SSG（CDN 秒返）、个人订单页应该 SSR（每请求新数据）、博客页应该 SSG + 后台再生。9.3 让开发者按 page 选择，不再是"全站 SSR"或"全站 SSG"。

同年 7 月 Next.js 9.5 引入 ISR（Incremental Static Regeneration）：

javascript 
// pages/products/[id].js (Next.js 9.5+)
export async function getStaticProps({ params }) {
  const product = await fetchProduct(params.id);
  return {
    props: { product },
    revalidate: 60, // 每 60 秒后台再生一次
  };
}

export async function getStaticPaths() {
  return { paths: [], fallback: 'blocking' };
}

ISR 解决了 SSG 最大问题——"build 时生成几万个 page 太慢"。开发者只 build 头部几百个，剩下的"按访问触发再生"。这是 Vercel 推 jamstack 的最后一块拼图。2020-2022 年 ISR 成为内容站点 / 电商站的事实标准方案。

2022-10 · Next.js 13 · App Router + RSC + 断代式重写

2022 年 10 月的 Next.js Conf 上 Vercel 发布 Next.js 13。这是 10 年里最大的一次断代式重写，同步推出三个新东西：

  • App Router（beta） —— 全新的路由系统，基于 app/ 目录而非 pages/
  • React Server Components（beta）—— 服务器组件作为 App Router 默认渲染模式
  • Turbopack（alpha）—— Rust 重写的 bundler，目标是取代 Webpack

App Router 与 Pages Router 的本质差异：

  • Pages Router 每个 page 是独立渲染树；layout 用 _app.tsx 全局包裹
  • App Router 嵌套布局原生，app/dashboard/layout.tsx 包裹 app/dashboard/* page，路由切换时不重新挂载（state 保留）
  • Pages Router 数据获取在 page 层（getStaticProps / getServerSideProps）
  • App Router 每个 Server Component 都可以独立 fetch，数据获取分散到组件层

RSC 的引入更激进。Vercel 在 React 团队还在 RFC 阶段就押宝并把 App Router 围绕它建。当时 Remix（现在的 React Router v7）走相反路线——继续用传统 SSR + loaders，不押 RSC。这次分叉至今未弥合。

13 发布时 App Router 标 beta，原因诚实：

  • 大量 API 还在变（Server Actions 还没 stable）
  • 缓存模型还在迭代（详见 P5）
  • RSC payload 格式还在调整
  • Turbopack alpha 远没有 production-ready

开发者社区分两派：早期采用者（Vercel 自家、Shopify、TikTok 部分团队）赌 App Router 是未来；保守派（大量企业项目）继续 Pages Router 等成熟。这种分裂持续了 13 / 14 两个大版本。

2023-10 · Next.js 14 · Server Actions stable + PPR 预览

14 主要做两件事：把 Server Actions 推到 stable、把 Partial Prerendering 推出 experimental。

Server Actions 是 React 19 Actions 的全栈实现：

javascript 
// app/todo/page.tsx (Next.js 14)
import { revalidatePath } from 'next/cache';

async function addTodo(formData: FormData) {
  'use server';
  await db.todos.create({ text: formData.get('text') });
  revalidatePath('/todo');
}

export default function TodoPage() {
  return (
    <form action={addTodo}>
      <input name="text" />
      <button>Add</button>
    </form>
  );
}

这一版让"表单 + 数据变更"变成全栈一等公民——无需自己写 fetch、无需自己管 loading、无需手动 revalidate 缓存。详见 P4。

Partial Prerendering 是新概念：一个 page 同时是静态 + 动态。静态壳从 CDN 秒返，动态部分流式补齐。当时还在 experimental，但已经预告了 v16 的方向。

同年 React 19 进入 RC，Next.js 14 要求 React 18 + 同时兼容 19 RC。这又是一次"Next.js 跟着 React 节奏走"的典型例子。

2024-10 · Next.js 15 · 缓存反转 + React 19 + Turbopack dev GA

15 是 Next.js 历史上承认设计错误最公开的一次。三个关键变化：

缓存默认值反转 —— v13-14 默认 fetch 缓存"激进"踩了大量生产 stale 数据事故。v15 把默认改回：fetch() 不写选项就是 dynamic（每次请求）；要 cache 必须显式 { cache: 'force-cache' } 或 { next: { revalidate } }。

diff 
// v13-14 默认行为（已废弃）
- await fetch('/api/data');  // 默认 cache，永久不重新拿
+ await fetch('/api/data', { cache: 'force-cache' });  // v15+ 必须显式

// v15+ 默认行为
await fetch('/api/data');  // 默认 dynamic，每次重新拿
await fetch('/api/data', { next: { revalidate: 60 } });  // 想 cache 显式标

这次反转的根因：v13-14 上线后，大量团队反馈"我没标 cache，怎么数据是 1 小时前的"。问题出在 Next.js 隐式把整个 Data Cache 默认开了，开发者无感知。v15 撤回默认，承认是设计错误。

async request APIs —— cookies() / headers() / params / searchParams 全部变成 Promise：

typescript 
// v14
import { cookies } from 'next/headers';
const cookieStore = cookies();
const token = cookieStore.get('token');

// v15+
import { cookies } from 'next/headers';
const cookieStore = await cookies();  // ← await
const token = cookieStore.get('token');

// page params
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;  // ← await
}

这次改动看起来像一次烦人的迁移，但根本目的是为 PPR 铺路——让"读 cookies"变成显式 await，PPR 才能在静态壳渲染时跳过。

Turbopack dev 进入 stable —— next dev 默认走 Turbopack；Webpack 仍可作为 fallback。45.8% 编译速度提升的官方数字基于 Vercel 自家大型项目实测。

2026-01 · Next.js 16 · Turbopack/PPR GA + Cache Components 重做

2026 年初的 Next.js 16 把 Turbopack prod 也推到 stable，PPR 正式 GA，缓存系统用 Cache Components 重做。这一版是 v15 反转的延续，也是缓存心智的最终形态（至少在 2027 之前）。

Cache Components —— 用 'use cache' 指令显式标记可缓存单元，替代 v13-15 那套五层缓存模型：

typescript 
// next.config.ts
export default { cacheComponents: true };

// app/blog/[slug]/page.tsx
async function PostContent({ slug }: { slug: string }) {
  'use cache';  // ← 这个 component 是 cacheable 单元
  const post = await db.posts.findUnique({ where: { slug } });
  return <article>{post.content}</article>;
}

async function Comments({ postId }: { postId: string }) {
  // 不带 'use cache' = 动态，每请求重新计算
  const comments = await db.comments.findMany({ where: { postId } });
  return <ul>{comments.map(c => <li>{c.body}</li>)}</ul>;
}

export default function Post({ params }: { params: { slug: string } }) {
  return (
    <>
      <PostContent slug={params.slug} />
      <Suspense fallback={<p>loading...</p>}>
        <Comments postId={params.slug} />
      </Suspense>
    </>
  );
}

PPR 默认开 —— 启用 cacheComponents 后，page 自动按"静态壳 + 动态洞"渲染。静态壳从 CDN 秒返；动态洞通过 stream 在请求里补齐。

updateTag 新增 —— 在当前请求内立即失效 + 重算，解决"提交后立刻看到新数据"的体验问题：

typescript 
'use server';
import { updateTag } from 'next/cache';

export async function addComment(postId: string, body: string) {
  await db.comments.create({ data: { postId, body } });
  updateTag(`post-${postId}-comments`);  // 当前请求就重算，不等下次访问
}

Turbopack 完全替代 Webpack —— next dev / next build 默认 Turbopack；Webpack 作为 deprecation 路径在 v17 删除。详见 P6。

纵观 10 年 · Next.js 为什么爱重构

Next.js 在同代框架里重构频率最高，原因不是工程混乱，而是 Vercel 的明确策略：

  • 永远是 React RFC 的首发实现者 —— Hooks（2018）/ Concurrent（2022）/ RSC（2022）/ Actions（2023）全是 Next.js 抢先 GA。这给 Vercel 先发红利（吸引早期采用者），也承担生产试错成本（v13-14 缓存事故、App Router 反复迭代）
  • 每 2-3 年押注下一代渲染模式 —— 1.0 SSR / 9.3 SSG+SSR / 9.5 ISR / 13 RSC / 16 PPR+Cache Components。每一代都是对前一代瓶颈的回应
  • vendor-coupled 生态 —— 新特性优先在 Vercel 平台验证（Edge Functions / Image Optimization / KV / Postgres），self-host 总是滞后
  • 对话式 release notes —— Vercel 工程 blog + Lee Robinson 视频深度解释每个变更动机；这种透明度让早期采用者愿意承担风险

这条路线的代价显著：v13-14 的生产事故让相当一部分团队对 Next.js 失去信任，转向 Remix 或 Astro。v15 的反转 + v16 的简化是修复信任的努力。但只要 Vercel 继续押注下一代 React 特性，这种"激进 → 反转"循环就会持续。

03 应用场景

场景 1 · 从 Pages Router 项目迁 App Router

2024-2026 年最常见的迁移场景。理解历史的实战价值在这里：

  • Pages Router 不会被立即废弃 —— v16 仍完全支持，可以两套并存（pages/ + app/ 同时存在）
  • 迁移路径：先把新功能放 app/，旧 page 留 pages/；逐步迁移
  • getServerSideProps 直接对应 Server Component（在 RSC 里 await）
  • getStaticProps + revalidate 对应 'use cache' + revalidateTag
  • API Routes（pages/api/*）对应 Route Handlers（app/api/*/route.ts）+ 一部分变成 Server Actions
  • 不要尝试一次性迁移整个项目，会卡在缓存 + RSC 心智上

场景 2 · 评估"我们要不要现在升 v16"

2026-Q2 的典型决策。理解 v15 → v16 的变化才能做：

  • 已经在 v15 + 用 fetch cache 的项目 —— 升 v16 风险低，主要是 Turbopack prod build 验证
  • v14 + 大量 unstable_cache 的项目 —— 升级两步：先升 v15（处理 async request APIs + 缓存反转），再升 v16（迁 unstable_cache → 'use cache'）
  • 使用 dynamicIO 实验配置的项目 —— v16 直接重命名为 cacheComponents，迁移路径官方文档清晰
  • self-host 的项目 —— v16 自定义 cache handler 接口稳定了，但要重写 cache handler 接 Redis

场景 3 · 给团队选 Next.js 主线版本

新项目起步：

  • 2026 主流：直接 Next.js 16 + App Router + Cache Components
  • 不接受 v16 试错的保守团队：Next.js 15.4 LTS 风格使用
  • 不要起 Pages Router 新项目（v17 不一定保留 Webpack 兼容路径）

04 典型代码

代码 1 · 同一功能在四个时代的写法对比

以"产品详情页"为例，看 Next.js 4 个时代怎么写：

javascript 
// 时代 1 · Next.js 1-8 (2016-2019) · getInitialProps
// pages/products/[id].js
export default class Product extends React.Component {
  static async getInitialProps({ query }) {
    const res = await fetch(`/api/products/${query.id}`);
    return { product: await res.json() };
  }
  render() { return <Detail product={this.props.product} />; }
}
// 痛点：每请求都跑 SSR，CPU 高；server / client 双跑，错误模型复杂
javascript 
// 时代 2 · Next.js 9.3-12 (2020-2022) · getStaticProps + ISR
// pages/products/[id].js
export async function getStaticProps({ params }) {
  const product = await fetchProduct(params.id);
  return { props: { product }, revalidate: 60 };
}
export async function getStaticPaths() {
  return { paths: [], fallback: 'blocking' };
}
export default function Product({ product }) { return <Detail product={product} />; }
// 痛点：page 层 fetch，组件无法独立请求；client 还是 hydration 整棵树
typescript 
// 时代 3 · Next.js 13-14 (2022-2024) · App Router + RSC
// app/products/[id]/page.tsx
export default async function Product({ params }: { params: { id: string } }) {
  const product = await fetchProduct(params.id);  // 直接 await，server 渲染
  return <Detail product={product} />;
}
// 优势：组件直接 fetch；client bundle 不含 server 代码

05 常见陷阱

  • 把 Pages Router 心智带进 App Router —— "我之前用 getServerSideProps 都没问题，App Router 怎么这么复杂"。App Router 不是 Pages Router 的语法糖，是不同的渲染模型。先读 P2 + P3 建立心智，再写代码
  • 看老博客文章学 Next.js —— 2022-2023 年的 RSC 教程很多是错的（fetch 默认行为已变 / unstable_cache 已废弃 / Server Actions 早期 API 已变）。只看 nextjs.org 当前文档 + Vercel blog 同期文章
  • v13-14 + 生产 stale 数据反复出现 —— 升级到 v15+，并审查所有 fetch 调用，把"想 cache 的"显式标 { next: { revalidate } }
  • v15 升级时忘了 await cookies/headers/params —— async request APIs 是同步 → Promise 的破坏式变更，TypeScript 会报错；老代码没类型的话会运行时崩
  • 把 v13-15 五层缓存模型背熟然后用 v16 —— v16 的 Cache Components 是新心智，不是五层叠加。详见 P5
  • "Next.js 太爱重构，我们不用它" —— 有道理，但要看团队规模。Vercel 重构频率高，但 LTS 流程相对稳定（每个 major 维护到下下个 major 出来）。中大型团队可以滞后一个 major
  • 把 Server Actions 当万能 API —— Server Actions 是给"表单 + 数据变更"，不是给所有 RPC 调用。复杂查询 / 第三方 webhook 仍然要 Route Handlers
  • 误解 PPR = SSG + SSR 的简单合并 —— PPR 是同一个 route 里"静态壳 + 动态洞"的混合，不是"有的 page 静态有的 page 动态"。v14 → v16 PPR 模型一直在演化，看 v16 文档
