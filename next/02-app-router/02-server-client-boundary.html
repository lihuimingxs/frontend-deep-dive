Server / Client 边界与序列化

App Router 默认所有组件是 RSC（Server Component），'use client' 是模块边界 opt-in，'use server' 是函数边界 opt-in。这两个指令决定了哪段代码在 server 跑、哪段在 client 跑、哪段需要序列化过桥。这一章把规则、序列化限制、混合模式（Server 包 Client 包 Server）、防御性导入（server-only / client-only）讲清，回答 90% 新手在 App Router 里 props 报错的根因。

阅读时长 ~30 分钟 前置 P2.1 文件约定 + 写过 React 19 主要来源 nextjs.org/docs/app rendering + React RFC #188 + Dan Abramov "The Two Reacts" / "RSC for two computers"

01 速览

  1. App Router 里所有组件默认是 Server Component。要让组件在 client 跑，必须显式写 'use client'。这与 Pages Router 完全相反 —— Pages Router 默认 client，SSR 是"附加效果"。
  2. 'use client' 不是"组件级"标记，是"模块级"标记。文件首行写了 'use client'，整个文件 + 它 import 的所有东西都进 client bundle（除非进一步标记 server-only）。这个边界一旦跨过去就回不来。
  3. 'use server' 是函数级标记，与 client 边界含义不同 —— 它把函数标为 Server Action，client 调用时变成一次 POST 请求。详见 P4。
  4. 跨 Server → Client 边界传 props 必须可序列化。可序列化的：plain object / 数组 / Promise（19+）/ Date / Map / Set / RegExp / BigInt / Uint8Array / null / undefined / primitives。不可序列化的：函数 / class instance（Symbol / Map iterator / 自定义类 etc）/ React element 实例（<div>）。
  5. 函数 prop 是个例外 —— 普通函数不能传，但 'use server' 标记的 Server Action 可以传（它会被替换成"调用句柄"）。
  6. 关键混合模式：Server 包 Client 包 Server。把 Server Component 作为 children prop 传给 Client Component —— children 是"已经渲染好的 React 树"，可以塞进 client component 不让 client 污染整棵子树。
  7. 防御性导入：import 'server-only' 让模块如果不小心被 client 导入立即报错；import 'client-only' 反之。这是大型项目防止边界泄露的关键工具。
  8. 常见错觉："'use client' = 渲染在 client" —— 错。'use client' 标记的组件仍然在 server 渲染一次（SSR 阶段，输出 HTML），然后在 client hydrate。它真正的语义是"这个组件可以用 useState / useEffect / 浏览器 API"，不是"只在 client 跑"。
  9. 如果组件不需要 hooks / event handler / 浏览器 API，就不要标 'use client'。RSC 的核心收益（client bundle 不含 server 代码、可以直接 await 数据库）只在不写 'use client' 时才生效。
  10. Server / Client 边界是 App Router 心智里最难建立的部分。但只要把"模块级标记 + 序列化 + children 混合"三件事建对，90% 的报错都能避免。剩下 10% 是 RSC 协议本身的边角情况，留到 P3.1 讲。

02 底层逻辑

默认是什么 · App Router vs Pages Router 的反转

Pages Router 时代的心智："React 组件默认在 client 跑，SSR 是 server 附加跑一遍生成首屏 HTML"。所有 hook / event handler / window 都直接能用，因为最终都是在 client 执行。

App Router 反转了这个默认：

tsx 
// app/posts/page.tsx
// 没有 'use client' 指令 → 默认 RSC
export default async function Posts() {
  // 可以直接 await 数据库（这段代码从不下发到 client）
  const posts = await db.posts.findMany();
  // ❌ 不能用 useState / useEffect
  // ❌ 不能用 onClick / onChange
  // ❌ 不能访问 window / localStorage
  return <ul>{posts.map(p => <li>{p.title}</li>)}</ul>;
}

要让组件能用 hooks / event / 浏览器 API，加 'use client'：

tsx 
// app/posts/_components/SearchBox.tsx
'use client';
import { useState } from 'react';

export default function SearchBox() {
  const [query, setQuery] = useState('');
  return (
    <input value={query} onChange={e => setQuery(e.target.value)} />
  );
}

'use client' 是模块边界，不是组件边界

这是 App Router 最常见的误解。'use client' 写在文件首行，不是函数前。它意味着"从这个模块开始，整棵 import 子树都在 client 打包"：

tsx 
// SearchBox.tsx
'use client';
import { someUtil } from './utils';      // utils 也进 client bundle
import { Button } from '@/ui/Button';     // Button 也进 client bundle
import OtherClientCmp from './Other';    // 不需要再写 'use client'，已在 client 区内

反过来：从 client 模块 import 一个 RSC？不行 —— RSC 不能被 client 导入：

tsx 
// SearchBox.tsx
'use client';
import PostList from './PostList';  // 如果 PostList 是 RSC（含 await db）→ 报错
                                     // 因为整个 client 子树都要打到 client bundle
                                     // db 代码不能进 client

这就是为什么 'use client' 一旦标了就回不来 —— 它向下污染整棵子树。要想让一个 client 组件包含 server 渲染的内容，必须用 children prop（看下面"混合模式"）。

序列化规则 · 哪些类型可以跨边界传

Server → Client 传 props 必须可序列化（因为要走 RSC payload）。React 19 + Next.js 16 支持的类型：

类型 可序列化？ 说明
string / number / boolean / null / undefined ✅ 原始类型
plain object / 数组 ✅ 必须递归可序列化
Date ✅ 不要先 toString，可以直接传
BigInt ✅
Map / Set ✅ 但内部 entries 要递归可序列化
RegExp ✅
Uint8Array / ArrayBuffer ✅ 二进制数据
Promise ✅（React 19+） client 用 use(promise) 读
Symbol ❌ 除了内置如 Symbol.iterator
函数 ❌（除非 'use server'） 普通函数不可，Server Action 可
class instance ❌ 需要先 toJSON / 转 plain object
React element 实例 (<div>) ✅（特殊处理） 作为 children prop 传可以

常见踩坑：从数据库 ORM 取出来的对象往往是 class instance（Prisma / TypeORM 等），传给 client 会报错。需要 JSON.parse(JSON.stringify(obj)) 或显式 toJSON。

函数 prop 例外 · Server Action 可以传

Server Action 是个特殊情况。普通函数不可序列化，但 'use server' 标记的函数会被替换成"调用句柄"（一个 ID + 类型签名），可以作为 prop 跨边界传：

tsx 
// app/posts/page.tsx (RSC)
async function deletePost(id: string) {
  'use server';
  await db.posts.delete({ where: { id } });
}

export default async function Posts() {
  const posts = await db.posts.findMany();
  return (
    <PostList posts={posts} onDelete={deletePost} />
    {/* ↑ deletePost 不是真函数，是个调用句柄；client 可以"调用"它 */}
  );
}

// app/posts/_components/PostList.tsx
'use client';
export function PostList({
  posts,
  onDelete,
}: {
  posts: Post[];
  onDelete: (id: string) => Promise<void>;
}) {
  return posts.map(p => (
    <button onClick={() => onDelete(p.id)}>Delete {p.title}</button>
  ));
}

底层机制：客户端"调用" onDelete 时实际是 POST 请求到当前路由 + 加密的 action ID，server 路由匹配后执行 deletePost 函数。详见 P4。

关键混合模式 · Server 包 Client 包 Server

App Router 最强的能力之一：让 Client Component 包 Server Component 作为 children。这让"仅一小段需要 client，大部分仍然是 server"成为可能：

tsx 
// app/_components/ThemeToggle.tsx
'use client';
import { useState } from 'react';

export default function ThemeToggle({ children }: { children: React.ReactNode }) {
  const [dark, setDark] = useState(false);
  return (
    <div className={dark ? 'dark' : ''}>
      <button onClick={() => setDark(d => !d)}>Toggle</button>
      {children}
    </div>
  );
}

// app/posts/page.tsx (RSC)
import ThemeToggle from '@/components/ThemeToggle';
import PostContent from './_components/PostContent';  // RSC

export default async function Posts() {
  const posts = await db.posts.findMany();
  return (
    <ThemeToggle>
      {/* PostContent 是 RSC，正常 server 渲染 */}
      {/* 它作为 children 传给 client component，但本身不进 client bundle */}
      <PostContent posts={posts} />
    </ThemeToggle>
  );
}

这种模式让"我有一个全局 theme provider（client）但里面包的 page 内容仍然是 RSC"成立。如果不用 children 模式，会强迫整棵子树进 client bundle。

防御性导入 · server-only / client-only

大型项目里很容易不小心把 server 代码（含 secret / 数据库连接）import 到 client component。两个 npm 包专门防御：

tsx 
// lib/db.ts
import 'server-only';  // 这一行让模块如果被 client import 立即编译报错
import { Pool } from 'pg';

export const pool = new Pool({
  connectionString: process.env.DATABASE_URL,  // secret 不会泄露到 client
});
tsx 
// lib/analytics.ts
import 'client-only';  // 反向，这个模块只能 client 用
import { track } from '@vercel/analytics';
export { track };

这两个包内部就是一个简单的 throw —— 但通过条件 export，让它在 server context 不报错、在 client context 报错（或反之）。建议所有访问 secret / 数据库 / server-only API 的模块都加 import 'server-only'。

'use client' 不等于"只在 client 跑"

这是最容易误解的点。看一个标了 'use client' 的组件：

tsx 
'use client';
import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);
  console.log('Counter renders');  // 这行会在 server 和 client 各跑一次
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

它的真实生命周期：

  1. SSR 阶段（server）：组件 render，输出 HTML（用初始 state 0），同时输出 RSC payload 描述这个 client component 的位置 + props
  2. Client 加载：浏览器收到 HTML，先显示
  3. Hydration（client）：浏览器加载组件 JS bundle，重新 render 一次（recompute initial state），attach event handlers
  4. 交互：用户点击，setState 触发 re-render，全在 client 跑

所以 'use client' 的真实语义是"这个组件可以用 hooks / event / 浏览器 API，并且会 hydrate"，不是"只在 client 跑"。理解这点对调试 hydration mismatch 错误至关重要。

什么时候应该不用 'use client'

RSC 的好处只在不标 'use client' 时生效：

  • 组件代码不下发到 client bundle（包体积小）
  • 可以直接 await 数据库 / 文件系统 / server-side API
  • 可以用 server-only 的 secret（API key / DB 密码）
  • 不参与 hydration，没有 hydration 成本

判断一个组件该不该标 'use client'：

  • 需要 useState / useReducer / useContext？→ 'use client'
  • 需要 useEffect？→ 'use client'
  • 需要 onClick / onChange / 任何 event handler？→ 'use client'
  • 需要 window / document / localStorage？→ 'use client'
  • 仅渲染数据 + 静态 markup → 保持 RSC
  • 需要 await fetch / await db → 必须 RSC

03 应用场景

场景 1 · 全局 ThemeProvider 不污染整棵 page

典型需求：root layout 加 ThemeProvider（client），但内部 page 大部分仍然是 RSC。

tsx 
// app/_components/Providers.tsx
'use client';
import { ThemeProvider } from 'next-themes';
import { Toaster } from 'sonner';

export default function Providers({ children }: { children: React.ReactNode }) {
  return (
    <ThemeProvider attribute="class" defaultTheme="system">
      {children}
      <Toaster />
    </ThemeProvider>
  );
}

// app/layout.tsx (RSC)
import Providers from './_components/Providers';
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html><body>
      <Providers>{children}</Providers>
    </body></html>
  );
}

关键点：children 作为 prop 传，不会被 ThemeProvider 的 client 边界污染。所有 page / layout 仍然是 RSC，可以 await fetch。

场景 2 · 组件内部既要 server 数据又要 client 交互

典型需求：搜索表单 —— 输入框是 client（受控），结果列表是 server（直接 fetch）。

tsx 
// app/search/page.tsx (RSC)
import SearchForm from './_components/SearchForm';
import SearchResults from './_components/SearchResults';

export default async function SearchPage({
  searchParams,
}: {
  searchParams: Promise<{ q?: string }>;
}) {
  const { q } = await searchParams;
  return (
    <>
      <SearchForm initialQuery={q} />
      {q && <SearchResults query={q} />}
    </>
  );
}

// app/search/_components/SearchForm.tsx
'use client';
import { useRouter } from 'next/navigation';
import { useState } from 'react';

export default function SearchForm({ initialQuery }: { initialQuery?: string }) {
  const [q, setQ] = useState(initialQuery || '');
  const router = useRouter();
  return (
    <form onSubmit={e => { e.preventDefault(); router.push(`/search?q=${q}`); }}>
      <input value={q} onChange={e => setQ(e.target.value)} />
    </form>
  );
}

// app/search/_components/SearchResults.tsx (RSC)
export default async function SearchResults({ query }: { query: string }) {
  const results = await db.posts.findMany({ where: { title: { contains: query } } });
  return <ul>{results.map(r => <li key={r.id}>{r.title}</li>)}</ul>;
}

分工清晰：表单交互在 client（无法 server），结果获取在 server（不需要 client）。两者通过 URL（searchParams）通信，不需要 props 序列化。

场景 3 · 把 server 数据预先 fetch，然后用 client 做交互

典型需求：商品列表初始 SSR 显示，然后 client 端"加载更多"。

tsx 
// app/products/page.tsx (RSC)
import ProductList from './_components/ProductList';

export default async function Products() {
  const initialProducts = await db.products.findMany({ take: 20 });
  // 把 server 数据传给 client 组件
  return <ProductList initialProducts={initialProducts} />;
}

// app/products/_components/ProductList.tsx
'use client';
import { useState } from 'react';

export default function ProductList({ initialProducts }: { initialProducts: Product[] }) {
  const [products, setProducts] = useState(initialProducts);

  async function loadMore() {
    const offset = products.length;
    const more = await fetch(`/api/products?offset=${offset}`).then(r => r.json());
    setProducts([...products, ...more]);
  }

  return (
    <>
      <ul>{products.map(p => <li>{p.name}</li>)}</ul>
      <button onClick={loadMore}>Load more</button>
    </>
  );
}

注意：initialProducts 必须是 plain object（不是 Prisma class instance），否则序列化报错。

04 典型代码

代码 1 · 序列化错误的修复

tsx 
// ❌ 错：传 Date 没问题，但传 class instance 报错
import { Decimal } from '@prisma/client';
async function Page() {
  const product = await db.products.findUnique({ where: { id } });
  return <Detail product={product} />;  // product.price 是 Decimal class，报错
}

// ✅ 修复 1：转换成 plain object
async function Page() {
  const product = await db.products.findUnique({ where: { id } });
  const safe = {
    ...product,
    price: product.price.toString(),  // Decimal → string
    createdAt: product.createdAt,      // Date 可以直接传
  };
  return <Detail product={safe} />;
}

// ✅ 修复 2：在 ORM 层就转换（更好）
const PrismaSafe = Prisma.defineExtension({
  result: {
    product: {
      price: { compute: (p) => p.price.toString() },
    },
  },
});

代码 2 · server-only 防御 secret 泄露

tsx 
// lib/auth.ts
import 'server-only';  // 关键
import jwt from 'jsonwebtoken';

const SECRET = process.env.JWT_SECRET!;  // 不会泄露到 client

export function verifyToken(token: string) {
  return jwt.verify(token, SECRET);
}

// ❌ 如果 client component 不小心 import 这个：
// 'use client';
// import { verifyToken } from '@/lib/auth';
// → build 时报错："This module is server-only and cannot be imported from a client component"

代码 3 · Server 包 Client 包 Server 实战

tsx 
// app/dashboard/page.tsx (RSC)
import Tabs from './_components/Tabs';
import OverviewTab from './_components/OverviewTab';     // RSC
import AnalyticsTab from './_components/AnalyticsTab';   // RSC

export default async function Dashboard() {
  return (
    <Tabs
      tabs={[
        { id: 'overview', label: 'Overview', content: <OverviewTab /> },
        { id: 'analytics', label: 'Analytics', content: <AnalyticsTab /> },
      ]}
    />
  );
}

// app/dashboard/_components/Tabs.tsx
'use client';
import { useState } from 'react';

export default function Tabs({
  tabs,
}: {
  tabs: Array<{ id: string; label: string; content: React.ReactNode }>;
}) {
  const [active, setActive] = useState(tabs[0].id);
  const activeTab = tabs.find(t => t.id === active)!;
  return (
    <>
      <nav>{tabs.map(t => (
        <button key={t.id} onClick={() => setActive(t.id)}>{t.label}</button>
      ))}</nav>
      <div>{activeTab.content}</div>
    </>
  );
}

关键：Tabs 是 client（管 state），但每个 tab 的 content 是预先在 server 渲染好的 RSC 树作为 React.ReactNode 传进来。Tabs 不需要 import OverviewTab / AnalyticsTab 模块（这两个仍然是 server-only）。

代码 4 · Promise 作为 prop（React 19+）

tsx 
// app/page.tsx (RSC)
import { Suspense } from 'react';
import LikeCount from './_components/LikeCount';

export default function Page() {
  // 不 await！直接把 Promise 传给 client component
  const likesPromise = fetch('/api/likes').then(r => r.json());
  return (
    <Suspense fallback={<p>Loading...</p>}>
      <LikeCount promise={likesPromise} />
    </Suspense>
  );
}

// app/_components/LikeCount.tsx
'use client';
import { use } from 'react';

export default function LikeCount({ promise }: { promise: Promise<{ count: number }> }) {
  const { count } = use(promise);  // React 19 use API
  return <p>{count} likes</p>;
}

这是 React 19 + Next.js 16 的新模式：让 server 触发 fetch、不等结果就把 Promise 传给 client，client 用 use() 在 Suspense 里读。比 server-side await 灵活，比 client-side useEffect fetch 高效。

05 常见陷阱

  • "Functions cannot be passed directly to Client Components" —— 90% 是给 client 传了普通函数。要么把函数改成 Server Action（'use server'），要么把整个组件改成 client（但要权衡）
  • "Only plain objects can be passed to Client Components" —— 传了 Prisma / Mongoose / Sequelize 的 class instance。需要 toJSON 或在 ORM 层就转 plain object
  • 在 RSC 里写 useState / useEffect / onClick 报错 —— 加 'use client' 在文件首行；或把这部分逻辑提取成单独 client component
  • "Cannot access Y on the server" —— 在 RSC 里访问 window / document / localStorage。这些是 client-only API，必须在 'use client' 模块里
  • "Hydration mismatch" / 服务端渲染与客户端不一致 —— 'use client' 组件在 SSR 阶段也跑一次。如果 render 依赖 Date.now() / Math.random() / localStorage，server 和 client 结果会不同。要么 useEffect 里读、要么用 suppressHydrationWarning
  • 给一个 client component 标 'use client' 然后还在里面 await db —— 编译可能不报错（如果 db 没标 server-only），但运行时崩；如果 db 标了 server-only，编译就报"server-only imported by client"
  • 整个项目都 'use client'，结果 SSR 失去意义 —— 团队迁移时常见。问题：bundle 体积没小、不能 await db、丢了 RSC 所有好处。逐步审查每个 'use client'，能去掉就去掉
  • 把 Server Component 当 Client Component 用 props.callback —— 普通函数不能传，但 Server Action 可以。如果只是 form action，用 Server Action；如果是事件 callback，把交互组件改 client 并用 children 模式
  • 在 'use client' 里 import Prisma client —— 编译可能不报错（取决于配置），但极容易让 DATABASE_URL 等 secret 泄露到 client bundle。所有 db 模块 必须 加 import 'server-only'
  • 认为 'use client' 组件不在 server 跑 —— 错。SSR 阶段也会跑一次（生成首屏 HTML）。所以 'use client' 组件代码必须能在 Node 跑，不能直接用 window（要包 useEffect 或 typeof window !== 'undefined' 检查）
  • 把 'use client' 放在函数内部 —— 不生效。必须放在文件首行，作为模块级标记
  • 层层 import 之后整棵子树进 client bundle —— bundle 体积爆炸的常见根因。审查 'use client' 的位置 —— 把它推到尽可能下层（leaf 组件），上层保持 RSC
