文件系统路由与约定

App Router 用文件夹定义路由段，用文件名约定层级行为。这一章把 7 个核心约定（page / layout / loading / error / not-found / template / default）+ 动态段 + 路由分组 + Parallel/Intercepting Routes + Multiple Root Layouts 全部讲透。最后用一个真实 dashboard 路由结构 + Twitter 风格 modal 两个案例把所有约定串起来。

阅读时长 ~30 分钟 前置 P1.1 项目史 + 写过 Next.js（Pages Router 即可） 主要来源 nextjs.org/docs/app/building-your-application/routing 全套

01 速览

  1. App Router 的核心规则：文件夹 = 路由段，文件名 = 该段的角色。app/dashboard/settings/page.tsx 即 URL /dashboard/settings，page.tsx 表示这个段是叶子页面。
  2. 7 个核心文件约定：page（页面，叶子节点）/ layout（包裹子段，state 保留）/ loading（Suspense 边界）/ error（错误边界，必须 client component）/ not-found（404 边界）/ template（每次导航重新挂载）/ default（parallel routes 的 fallback）。
  3. 动态段三种形态：[slug]（单段动态）/ [...slug]（catch-all，必须有内容）/ [[...slug]]（optional catch-all，可空）。配 generateStaticParams 决定 build 时预生成哪些。
  4. 路由分组 (group) 不影响 URL，只影响代码组织 —— app/(marketing)/about/page.tsx 仍然是 /about，但分组让"marketing 站套独立 layout、app 套另一个 layout"成为可能。
  5. 私有文件夹 _foo 不被路由系统识别，用来放 helper / component / hook 等不构成路由的代码。
  6. Parallel Routes（@slot）让一个 layout 同时渲染多个独立子路由 —— dashboard 多面板 / modal 与主内容并存的场景必备。每个 slot 独立 loading / error 边界。
  7. Intercepting Routes（(.) / (..) / (...)）让"点击列表项弹出详情 modal"这种交互成为可能 —— 客户端导航时拦截到 modal 里渲染，刷新时退回完整页面。
  8. Multiple Root Layouts —— 在同一个 app 里有"两套完全不同的根布局"（marketing site + 应用）。通过把 app/layout.tsx 删掉、放进各路由分组的 layout.tsx 实现。
  9. Route Handlers（route.ts）—— 用于 REST API / webhook / 非 React-friendly 返回（image / PDF / CSV）。注意 route.ts 与 page.tsx 在同一段不能共存。
  10. 这套约定的设计目标：让"路由结构 + 渲染层级 + 错误边界 + Suspense 边界"全部在文件结构里直接看到，不再散落在 _app.tsx + getServerSideProps + 自定义 ErrorBoundary 等多个地方。代价：约定是黑盒，跳出约定时（比如想要某个 layout 不嵌套）要用 multiple root layouts / parallel routes 这种特殊机制。

02 底层逻辑

核心规则：文件夹 = 路由段，文件名 = 角色

App Router 把"什么 URL 渲染什么组件"完全交给文件结构。看这个例子：

text 
app/
├── layout.tsx            // root layout（必需），所有 page 的最外层 wrapper
├── page.tsx              // 对应 URL: /
├── about/
│   └── page.tsx          // 对应 URL: /about
├── posts/
│   ├── page.tsx          // 对应 URL: /posts
│   └── [id]/
│       └── page.tsx      // 对应 URL: /posts/123 （[id] = 动态段）
└── dashboard/
    ├── layout.tsx        // 包裹 dashboard/* 所有 page
    ├── page.tsx          // 对应 URL: /dashboard
    ├── loading.tsx       // dashboard/* 加载时的 fallback
    └── settings/
        └── page.tsx      // 对应 URL: /dashboard/settings

渲染时层级是：root layout → dashboard layout → dashboard/settings page。每一级都是一个 React 组件，layout 用 children prop 包子级。

7 个核心文件约定

page.tsx · 叶子节点。一个段必须有 page.tsx 才能被访问。export default 一个 React 组件。

tsx 
// app/posts/[id]/page.tsx
export default async function Post({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const post = await fetchPost(id);
  return <article>{post.title}</article>;
}

layout.tsx · 包裹子段，导航时不重新挂载（state 保留）。这是 App Router 比 Pages Router 强的地方 —— Pages Router 的 _app.tsx 是全局的，而 layout 可以分段嵌套。

tsx 
// app/dashboard/layout.tsx
export default function DashboardLayout({ children }: { children: React.ReactNode }) {
  return (
    <div className="dashboard-grid">
      <Sidebar />          {/* 路由切换时不重挂载 */}
      <main>{children}</main>
    </div>
  );
}

每个段最多一个 layout。如果根需要 <html> + <body>，必须放在 app/layout.tsx（root layout）：

tsx 
// app/layout.tsx · root layout
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="zh-CN">
      <body>{children}</body>
    </html>
  );
}

loading.tsx · 等价于在 page 外包一层 <Suspense fallback={<Loading />}>。子段加载时显示。

tsx 
// app/dashboard/loading.tsx
export default function DashboardLoading() {
  return <Skeleton />;
}

error.tsx · 错误边界。必须是 client component（用 'use client'）—— 因为 ErrorBoundary 需要 catch 渲染错误，是 client 概念。

tsx 
// app/dashboard/error.tsx
'use client';
export default function DashboardError({ error, reset }: { error: Error; reset: () => void }) {
  return (
    <div>
      <p>Something went wrong: {error.message}</p>
      <button onClick={reset}>Try again</button>
    </div>
  );
}

not-found.tsx · 404 边界。当 page 内 notFound() 函数被调用时渲染。

tsx 
// app/posts/[id]/page.tsx
import { notFound } from 'next/navigation';

export default async function Post({ params }) {
  const { id } = await params;
  const post = await fetchPost(id);
  if (!post) notFound();  // 触发最近的 not-found.tsx
  return <article>{post.title}</article>;
}

// app/posts/[id]/not-found.tsx
export default function PostNotFound() {
  return <p>Post not found.</p>;
}

template.tsx · 与 layout 相反 —— 每次导航都重新挂载。常用于"页面切换 transition / animation 重新触发"或"reset 某个 state"的场景。

tsx 
// app/dashboard/template.tsx
'use client';
import { motion } from 'framer-motion';

export default function Template({ children }: { children: React.ReactNode }) {
  // 路由切换时整个 motion.div 重新挂载，触发 enter animation
  return (
    <motion.div initial={{ opacity: 0 }} animate={{ opacity: 1 }}>
      {children}
    </motion.div>
  );
}

default.tsx · 仅在 Parallel Routes 用 —— 当某个 slot 在当前 URL 没有匹配时，渲染 default。后面 Parallel Routes 段会展开。

动态段：[slug] / [...slug] / [[...slug]]

三种动态段对应不同语义：

形式 匹配 params 类型
app/posts/[id]/page.tsx /posts/123 / /posts/abc（必须有一段） { id: string }
app/docs/[...slug]/page.tsx /docs/a / /docs/a/b/c（一段或多段，必须有内容） { slug: string[] }
app/docs/[[...slug]]/page.tsx /docs / /docs/a / /docs/a/b（包括空） { slug: string[] | undefined }

动态段配 generateStaticParams 在 build 时预生成（类似 v9.3 的 getStaticPaths）：

tsx 
// app/posts/[id]/page.tsx
export async function generateStaticParams() {
  const posts = await db.posts.findMany({ select: { id: true } });
  return posts.map(p => ({ id: p.id }));
}

export default async function Post({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const post = await db.posts.findUnique({ where: { id } });
  return <article>{post.title}</article>;
}

路由分组 (group) · 不影响 URL，只组织代码

用括号包裹的文件夹不出现在 URL 里，只用来组织代码：

text 
app/
├── (marketing)/
│   ├── layout.tsx        // 仅包裹 marketing 段
│   ├── page.tsx          // URL: /
│   ├── about/page.tsx    // URL: /about
│   └── pricing/page.tsx  // URL: /pricing
└── (app)/
    ├── layout.tsx        // 仅包裹 app 段（不同 layout）
    ├── dashboard/page.tsx // URL: /dashboard
    └── settings/page.tsx  // URL: /settings

这种结构让"marketing 部分用一套头部 / 底部、app 部分用另一套"自然实现。两套 page 分组归类，URL 仍是平级。

私有文件夹 _foo · 不被路由识别

下划线开头的文件夹不参与路由。用来放 helper / component / hook：

text 
app/
├── posts/
│   ├── _components/      // 不是路由，纯组件
│   │   ├── PostCard.tsx
│   │   └── PostList.tsx
│   ├── _lib/             // 不是路由，工具函数
│   │   └── fetch-posts.ts
│   └── page.tsx

这是 App Router 的"反约定"逃生口 —— 你想把组件放路由旁边但不让它成为路由，加个下划线就行。

Parallel Routes · @slot 同时渲染多个独立子路由

核心场景：dashboard 同时有 sidebar + main + analytics 三块独立内容，每块有自己的加载状态 / 错误边界 / 缓存。Parallel Routes 用 @name 命名 slot：

text 
app/dashboard/
├── layout.tsx            // 接收多个 slot 作为 props
├── @analytics/
│   ├── page.tsx          // analytics slot
│   ├── loading.tsx       // analytics 独立加载状态
│   └── error.tsx         // analytics 独立错误边界
├── @notifications/
│   ├── page.tsx
│   └── loading.tsx
└── page.tsx              // 主 children
tsx 
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children,
  analytics,
  notifications,
}: {
  children: React.ReactNode;
  analytics: React.ReactNode;
  notifications: React.ReactNode;
}) {
  return (
    <div className="grid grid-cols-3">
      <main>{children}</main>
      <aside>{analytics}</aside>
      <aside>{notifications}</aside>
    </div>
  );
}

每个 slot 独立加载、独立失败。用户看到的是三块各自 reveal，而不是"必须等所有数据齐全"。

default.tsx 在这里出场：当用户从某个 URL 跳到另一个 URL，新 URL 没有匹配某个 slot 时，渲染该 slot 的 default.tsx：

tsx 
// app/dashboard/@analytics/default.tsx
export default function AnalyticsDefault() {
  return null;  // 或者一个占位
}

Intercepting Routes · 不离开当前页面的"插队"渲染

典型场景：Twitter / Instagram 的"点击图片弹出详情 modal、刷新页面退回完整图片页"模式。Intercepting Routes 用 (.) / (..) / (...) 表示"拦截某个路由到当前位置渲染"：

  • (.) · 同级
  • (..) · 上一级
  • (..)(..) · 上两级
  • (...) · 从 app 根
text 
app/
├── feed/
│   ├── page.tsx                       // /feed 列表
│   └── @modal/
│       ├── default.tsx                // 默认 null（不显示 modal）
│       └── (.)photo/[id]/page.tsx     // 拦截 /photo/[id]
├── photo/
│   └── [id]/
│       └── page.tsx                   // /photo/[id] 完整页（刷新时显示）
└── layout.tsx                         // 接收 children + modal 两个 slot

用户在 feed 页点 photo 链接 → URL 变为 /photo/123，但 (.)photo/[id] 拦截把它渲染到 @modal slot；刷新 /photo/123 → 拦截不命中，走到 /photo/[id]/page.tsx 完整页。

Multiple Root Layouts · 两套根布局

当你有"完全独立的两套 app"（marketing + dashboard，有不同的 fonts / global styles / SEO 策略），可以删 app/layout.tsx，把 root layout 放进各分组：

text 
app/
├── (marketing)/
│   ├── layout.tsx        // 含 <html> <body> ── 这是 root layout
│   └── page.tsx
└── (app)/
    ├── layout.tsx        // 含 <html> <body> ── 这也是 root layout
    └── dashboard/page.tsx

限制：跨 root layout 切换路由时，整页重新加载（不是 client navigation）。所以这种结构只在"两套 app 真的独立"时用。

Route Handlers · route.ts

App Router 里 REST API / webhook 用 route.ts（不是 page.tsx）。同一段 route.ts 与 page.tsx 不能共存：

tsx 
// app/api/users/route.ts
export async function GET() {
  const users = await db.users.findMany();
  return Response.json(users);
}

export async function POST(request: Request) {
  const body = await request.json();
  const user = await db.users.create({ data: body });
  return Response.json(user, { status: 201 });
}

Route Handlers 适合 webhook / 第三方集成 / 返回非 JSON（image / PDF）/ 给非 React 客户端用。日常表单 / 数据变更应该用 Server Actions（详见 P4）。

03 应用场景

场景 1 · 真实 SaaS dashboard 路由结构

text 
app/
├── (marketing)/                         // 营销站点（公开）
│   ├── layout.tsx                       // 营销头部 / 底部
│   ├── page.tsx                         // 首页
│   ├── pricing/page.tsx
│   └── docs/[[...slug]]/page.tsx        // 嵌套文档
├── (auth)/                              // 登录登出
│   ├── layout.tsx                       // 极简 layout
│   ├── login/page.tsx
│   └── signup/page.tsx
├── (app)/                               // 应用主体（登录后）
│   ├── layout.tsx                       // app 头部
│   ├── dashboard/
│   │   ├── layout.tsx                   // dashboard 侧栏 + 主区
│   │   ├── @analytics/                  // parallel slot
│   │   │   ├── default.tsx
│   │   │   └── page.tsx
│   │   ├── loading.tsx
│   │   ├── page.tsx
│   │   └── [orgId]/
│   │       ├── page.tsx
│   │       └── projects/
│   │           ├── page.tsx
│   │           └── [projectId]/
│   │               ├── layout.tsx       // 项目级 layout（tabs）
│   │               ├── page.tsx         // overview tab
│   │               ├── settings/page.tsx
│   │               └── members/page.tsx
│   └── settings/page.tsx
├── api/
│   └── webhooks/
│       └── stripe/route.ts              // webhook
└── layout.tsx                           // 全局 root layout

这个结构包含的元素：

  • 路由分组 (marketing) / (auth) / (app) 区分功能区域
  • 嵌套 layout（dashboard layout 包多个子段，state 保留）
  • 动态段 [orgId] / [projectId]
  • Optional catch-all [[...slug]] 给 docs 用
  • Parallel route @analytics 给 dashboard 多面板
  • Route handler route.ts 给 webhook

场景 2 · Twitter 风格 modal 实现

text 
app/
├── layout.tsx                           // 接收 children + modal slot
├── feed/
│   ├── page.tsx                         // /feed 列表
│   └── @modal/                          // parallel slot
│       ├── default.tsx                  // null
│       └── (.)photo/[id]/page.tsx       // 拦截 /photo/[id] 到 modal
└── photo/
    └── [id]/
        └── page.tsx                     // /photo/[id] 完整页
tsx 
// app/layout.tsx
export default function RootLayout({
  children,
  modal,
}: {
  children: React.ReactNode;
  modal: React.ReactNode;
}) {
  return (
    <html><body>
      {children}
      {modal}
    </body></html>
  );
}

// app/feed/@modal/(.)photo/[id]/page.tsx
import { Modal } from '@/components/Modal';
export default async function PhotoModal({ params }) {
  const { id } = await params;
  const photo = await fetchPhoto(id);
  return <Modal><img src={photo.url} /></Modal>;
}

// app/photo/[id]/page.tsx
export default async function PhotoFull({ params }) {
  const { id } = await params;
  const photo = await fetchPhoto(id);
  return <main><img src={photo.url} /></main>;
}

用户在 feed 点图片 → modal 弹出（URL 变 /photo/123，但拦截渲染到 modal slot）；刷新或直接打开 /photo/123 → 走完整页面。这种"客户端柔性 + URL 持久化"的 UX 是 Intercepting Routes 的标杆场景。

场景 3 · 国际化 + Pages Router 共存

大型项目从 Pages Router 迁移过程中常需要 App + Pages 共存。Next.js 16 仍完全支持：

text 
// 项目根
app/
├── layout.tsx
└── (new)/                       // 新写的功能放 App Router
    └── dashboard/page.tsx
pages/
├── _app.tsx                     // 老的 _app
├── index.tsx                    // 老的 / 仍走 Pages Router
└── api/
    └── legacy.ts                // 老的 API Routes 仍可用

同一个 URL 不能两边都有。冲突时 App Router 优先。这种共存让"逐步迁移"成为可能，不必一次性重写。

04 典型代码

代码 1 · 嵌套 layout + state 保留

tsx 
// app/dashboard/layout.tsx
'use client';
import { useState } from 'react';

export default function DashboardLayout({ children }: { children: React.ReactNode }) {
  const [sidebarCollapsed, setSidebarCollapsed] = useState(false);
  // 这个 state 在 /dashboard → /dashboard/settings → /dashboard/users
  // 路由切换时不会丢失（layout 不重新挂载）
  return (
    <div className={sidebarCollapsed ? 'collapsed' : ''}>
      <Sidebar onToggle={() => setSidebarCollapsed(c => !c)} />
      <main>{children}</main>
    </div>
  );
}

代码 2 · loading.tsx + Suspense 嵌套

tsx 
// app/dashboard/page.tsx
import { Suspense } from 'react';
import Stats from './_components/Stats';
import Chart from './_components/Chart';

export default function Dashboard() {
  return (
    <>
      <h1>Dashboard</h1>
      {/* 整个 page 加载时显示 dashboard/loading.tsx */}
      {/* 内部还可以再用 Suspense 让 stats / chart 各自独立流式 */}
      <Suspense fallback={<StatsSkeleton />}><Stats /></Suspense>
      <Suspense fallback={<ChartSkeleton />}><Chart /></Suspense>
    </>
  );
}

代码 3 · error.tsx 错误恢复

tsx 
// app/posts/[id]/error.tsx
'use client';
import { useEffect } from 'react';

export default function PostError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    // 上报错误监控（Sentry / Datadog）
    console.error('Post error:', error);
  }, [error]);

  return (
    <div>
      <h2>无法加载这篇文章</h2>
      <p>digest: {error.digest}</p>
      <button onClick={reset}>重试</button>
    </div>
  );
}

代码 4 · catch-all 段处理 docs 嵌套

tsx 
// app/docs/[[...slug]]/page.tsx
import { notFound } from 'next/navigation';
import { renderMDX } from '@/lib/mdx';

export async function generateStaticParams() {
  const allDocs = await getAllDocPaths();  // 例如 [['intro'], ['guide','setup'], ...]
  return allDocs.map(slug => ({ slug }));
}

export default async function Docs({ params }: { params: Promise<{ slug?: string[] }> }) {
  const { slug = [] } = await params;
  const path = slug.join('/') || 'index';
  const doc = await loadDoc(path);
  if (!doc) notFound();
  return <article>{renderMDX(doc.content)}</article>;
}

05 常见陷阱

  • 把 page.tsx 和 route.ts 放同一段 —— 报错。一个段只能是 page 或 route 二选一。如果想要"既渲染页面也提供 API"，把 API 放到子段（page.tsx 同级 api/route.ts 子文件夹）
  • error.tsx 忘了 'use client' —— 报错"error must be a client component"。错误边界必须是 client component，因为 ErrorBoundary 是 client API
  • error.tsx 不能 catch root layout 错误 —— root layout 出错只能用 app/global-error.tsx catch（不是普通 error.tsx）
  • layout.tsx 写 'use client' 然后 fetch 数据 —— Client component 不能 await server fetch；要么把 layout 改回 RSC（默认），要么把 fetch 移到子组件
  • 动态段类型忘了 Promise（v15+） —— v15 起 params 是 Promise，必须 await params 拿值。老代码直接 params.id 会拿到 Promise 对象
  • generateStaticParams 返回类型错 —— 必须返回 Array<{paramName: string}>，{paramName: number} 会运行时崩
  • Parallel Routes 没写 default.tsx —— 跨 URL 切换时若 slot 在新 URL 没匹配，没有 default 会报错。建议每个 @slot 都配 default.tsx（最简单返回 null）
  • Multiple Root Layouts 跨分组导航整页刷新 —— 这是设计行为不是 bug。所以 multiple root layouts 只在"两套 app 真的独立"时用，不要为"两个不同 header"过度使用
  • Intercepting routes 路径写错 —— (.) 是同级，(..) 是上一级，路径计算的是路由层级不是文件夹层级。app/feed/@modal/(..)photo/... 的 (..) 表示从 feed 路由段往上一级（即 root），所以拦截 /photo/[id]
  • private folder 也写 layout / page —— _components/page.tsx 不会成为路由。下划线开头整段被路由系统忽略
  • 用 layout 而不是 template 然后 animation 不触发 —— layout 不重新挂载，所以 enter animation 不会触发。需要 template.tsx 才行
