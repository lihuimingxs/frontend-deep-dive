Turbopack 内部 + bundler 横评

Turbopack 是 Tobias Koppers（Webpack 原作者）2022 年加入 Vercel 后用 Rust 重写的 bundler，2026 年 Next.js 16 让它在 dev + prod 双 stable，正式取代 Webpack。这一章讲清它的核心引擎 turbo-tasks（细粒度增量计算）、为什么用 Rust 而不是 Go/Zig、与 Vite-Rolldown / esbuild / Rspack / Bun build 的路线分叉，以及为什么 Vercel 不直接用 Vite。Bundler 历史短重述，详细演化看 Vite 主题。

阅读时长 ~25 分钟 前置 Vite 主题 P1（bundler 历史）+ P2（Vite 双模架构） 主要来源 turbo.build/pack docs + Tobias Koppers 公开演讲 + vercel.com/blog Turbopack 系列 + v15/v16 release notes

01 速览

  1. Turbopack 是 Vercel 自主开发的 bundler，目标是取代 Webpack 在 Next.js 中的角色。2022 年首发 alpha（Next.js 13），2024 年 dev stable（Next.js 15），2026 年 prod stable（Next.js 16）。
  2. 核心引擎是 turbo-tasks —— 一套用 Rust 写的细粒度增量计算系统。每个编译单元是一个 task，task 之间靠 macro 自动建立依赖关系；改了一个文件只重算受影响的 task，其他全部缓存命中。
  3. 编译用 SWC（Rust 写的 Babel 替代）—— 比 Babel 快 10-20×，比 esbuild（Go 写）速度接近，但增量计算能力更强。
  4. 与 Webpack 数据兼容部分 loader / plugin 接口 —— 让从 Webpack 迁移有渐进路径，但不是 100% 兼容（特别是底层 plugin hooks）。
  5. 2026 bundler 格局：Rust 是赢家，路径剩两条 —— Turbopack（vendor-locked 给 Next.js）/ Vite + Rolldown（通用，给 RR v7、TanStack Start、Astro、SvelteKit 等）。Webpack / esbuild 进入存量维护。
  6. Turbopack vs Vite-Rolldown 的根本差异：Turbopack 是单 pipeline（dev 和 prod 用同一套机制），Vite 是双模（dev 用 esbuild + native ESM、prod 用 Rolldown 打包）。两种路线都能工作，但工程哲学不同。
  7. 为什么 Vercel 不直接用 Vite —— 1) 全栈控制（dev server + prod bundle + RSC + cache 一体）；2) Vite 双模架构在 RSC 流式场景需要专门适配；3) 长期赌 Rust 增量计算比 Go/Zig 更适合大型 monorepo。这些是工程判断而非技术优劣。
  8. Turbopack 的成本是 vendor-locked —— 它仅服务 Next.js（短期内不会被其他框架采用）。生态外（Storybook / Docusaurus / 自家 monorepo build）仍要用 Webpack 或 Vite 系。
  9. 性能数字（v16 官方）：dev 启动比 Webpack 快 ~45%，HMR 更快，prod build 与 Webpack 接近（早期 prod 不如 Webpack，v16 GA 后追平）。具体数字取决于项目规模。
  10. 这一章不重写 bundler 史（已在 Vite 主题 P1 讲过）。重点是 Turbopack 自己 + 横评其他选项。如果你想深入 Webpack / Rollup / esbuild 内部，先看 Vite 主题。

02 底层逻辑

turbo-tasks · 增量计算引擎

Turbopack 的核心不是"打包"，是"知道改了什么，只重算受影响的部分"。turbo-tasks 是这套增量计算系统的实现。

核心思想：

  • 每个编译步骤是一个 task —— 解析模块 / 编译 JS / 打包 chunk 都是 task
  • task 自动建立依赖关系 —— task A 调用 task B，A 自动依赖 B 的结果
  • 结果可缓存 —— task 输入相同，结果相同；缓存命中跳过执行
  • 修改触发失效 —— 改一个文件，从 file → parse → transform → bundle 这条 task 链上的所有 task 重算；其他不相关的 task 全部缓存命中

用 Rust macro 简化 task 定义：

rust 
// 简化的 turbo-tasks 风格代码（实际更复杂）
#[turbo_tasks::function]
async fn parse_module(file: PathRef) -> Result<ModuleAst> {
  let source = read_file(file).await?;  // 自动依赖 file 内容
  parse_javascript(source).await
}

#[turbo_tasks::function]
async fn build_chunk(modules: Vec<ModuleAst>) -> Result<Chunk> {
  // 这个 task 依赖所有传入的 modules
  // 任一 module 变化，这个 chunk 重算；其他 chunk 不受影响
}

对比 Webpack：Webpack 是"全图重建"模式 —— 改一个文件触发 incremental rebuild，但内部图算法是 JS 写的，且粒度粗。Turbopack 把这套搬到 Rust + 细粒度 task。

SWC · Rust 写的编译器

SWC（Speedy Web Compiler）是 Donny（강동윤）2018 年起独立开发的 Rust 编译器，2022 年起被 Vercel 雇用专职。它的角色相当于 Babel：

  • 解析 JS / TS / JSX / TSX → AST
  • 跑 transformation（JSX → React.createElement、TS 类型擦除、装饰器、Polyfill 注入）
  • 输出最终 JS

性能：比 Babel 快 10-20×（单核），比 esbuild 略慢但增量计算更强。

SWC 也是 Next.js 不依赖 Babel 的核心 —— 自从 Next.js 12 起，TypeScript / JSX / SSR 编译全走 SWC，Babel 仅作为 fallback。

Webpack loader / plugin 兼容

Turbopack 部分兼容 Webpack 接口，让迁移有路径：

  • loader 兼容：Webpack loader（JS 函数）大多可用，特别是 markdown-loader / yaml-loader / svg-loader
  • plugin 兼容：部分 Webpack plugin 可用，但底层 hooks（compilation / optimization）大多不兼容 —— Turbopack 的内部模型与 Webpack 不同
  • resolve 配置：alias / extensions 等基础配置兼容
  • 不兼容：Webpack 5 的 Module Federation、复杂 splitChunks 配置、custom hooks

实战中：用 Webpack 标准 loader（80% 项目）的迁移大多无痛；用复杂 Webpack plugin / 自定义编译流程的项目（20%）需要重写或暂时留 Webpack。

dev + prod 单 pipeline

这是 Turbopack 与 Vite 路线最大的差异。看对比：

维度 Vite（双模） Turbopack（单 pipeline）
dev server 模块系统 原生 ESM（浏览器直接 import） 打包成 chunks（与 prod 一致）
dev 编译 esbuild（按需，单文件转换） SWC + turbo-tasks（增量打包）
prod 编译 Rolldown（独立 bundler） SWC + turbo-tasks（同一套）
dev / prod 行为差异 明显（ESM vs CJS / chunk 边界 / tree-shaking） 几乎相同（同一引擎）

Vite 双模的优势：dev 启动极快（不打包，浏览器直接 ESM 加载）。Turbopack 单 pipeline 的优势：dev 与 prod 行为一致（避免"dev 工作 prod 坏"事故）。

对 Next.js 来说，单 pipeline 重要，因为 RSC 流式渲染需要 server 端的精确 chunk 边界（详见 P3.1）。Vite 在 RSC 场景下需要专门适配（这就是为什么 RR v7 / TanStack Start 加 RSC 支持要花时间）。

横评 · 2026 bundler 格局

Bundler 语言 定位 主要场景 2026 状态
Webpack JS 万能但慢 存量项目 / 复杂配置 存量维护
Rollup JS library bundler npm 包发布 仍主流（library 场景）
esbuild Go 极快但功能有限 Vite dev / 单文件转换 Vite dev 仍用
Rolldown Rust Rollup-compat 重写 Vite 7 默认 prod bundler 2026 GA
Rspack Rust Webpack-compat 字节跳动主推 / 老 Webpack 项目迁移 稳定，生态扩张
Bun build Zig 与 Bun runtime 同源 Bun 项目首选 稳定，生态较小
Turbopack Rust vendor-locked 给 Next.js Next.js 16+ 默认 2026 prod GA

关键观察：

  • Rust 是赢家（除 esbuild Go 和 Bun build Zig 外，新 bundler 几乎全 Rust）
  • Vite-Rolldown 与 Turbopack 是两条主路 —— 通用 vs Next.js 专用
  • Webpack 不会消失（生态太大），但新项目几乎不选
  • esbuild 找到了"Vite dev 的 dependency optimizer"这个稳定生态位，不再追求成为完整 bundler

为什么 Vercel 押 Turbopack 而不直接用 Vite

这是社区争议最多的话题之一。理由：

1. 全栈控制 —— Next.js 不只是"应用层框架"，它要管 dev server、SSR pipeline、RSC payload、缓存层、Edge / Node runtime 切换、Image Optimization 等。Vite 是通用 bundler，不专门为这些场景优化；Vercel 想完全控制每一层。

2. 增量计算优先 —— Vercel 大量项目是 monorepo + 多 page。Vite 的双模架构在 dev 阶段每次改文件需要让浏览器重新 import；prod build 还要再来一遍 Rolldown。Turbopack 的 turbo-tasks 在 dev / prod 复用增量计算，monorepo 场景理论上更快。

3. RSC 流式适配 —— RSC payload 流式 + Suspense flush 需要 bundler 配合（生成 manifest、控制 chunk 边界）。Webpack 已实现 react-server-dom-webpack；Turbopack 自家实现；Vite 等其他 bundler 需要从头实现 RSC plugin。

4. 长期 Rust 押注 —— Vercel 把 Tobias Koppers（Webpack 作者）和 Donny（SWC 作者）招进来，目标是让 Rust 成为 web 工具链长期主流。Vite 也用 Rust（Rolldown），但 Vite 团队（Evan You）独立运作，Vercel 不能控制方向。

但这不是说 Vite 路线错了。Vite-Rolldown 在通用场景（不押 RSC 的项目 / 多框架支持）有优势。两条路线长期共存，不是零和。

Turbopack 性能数字（2026 v16 官方）

Vercel 公开数字（基于自家大型项目实测）：

  • dev 启动：比 Webpack 快 45.8%
  • HMR：单文件改动到浏览器更新 < 100ms（Webpack 通常 200-500ms）
  • prod build：v15 时仍慢于 Webpack；v16 GA 时基本追平
  • cold cache build（CI 第一次）：与 Webpack 相当（增量优势在二次 build 才体现）

注意：性能数字高度依赖项目规模 + 配置。小项目（< 100 模块）差距不明显；中大型项目（1000+ 模块）+ monorepo 场景 Turbopack 优势明显。

03 应用场景

场景 1 · 从 Webpack 迁 Turbopack

Next.js 16 默认 Turbopack。从 v14 / v15 项目升级时：

  1. 跑 next dev --turbo 看是否成功启动（v15+ 默认 Turbopack）
  2. 检查报错 —— 大多与 Webpack 自定义 plugin / loader 有关；标准 loader 通常无痛
  3. 常见需要替换的：next-bundle-analyzer 旧版本、自定义 webpack 配置中的 plugin
  4. 跑 next build --turbo 验证 prod build
  5. 对比性能 —— dev 启动 / HMR / build 时间
  6. v16 起：next.config 里的 webpack() 配置仍可用，但只在 fallback 时生效

场景 2 · 评估"我们项目能用 Turbopack 吗"

判断：

  • 用标准 Next.js 配置 → 直接 Turbopack（无问题）
  • 用 next.config 里的 webpack() 自定义 plugin → 检查这些 plugin 是否兼容
  • 用 Module Federation → 仍需 Webpack（Turbopack v16 还不支持）
  • 用复杂 splitChunks 配置 → 测试一下，可能行为不一致
  • monorepo 场景 → Turbopack 优势大，强烈推荐

场景 3 · 调试"Turbopack 比 Webpack 慢"

少数情况 Turbopack 反而慢。排查：

  • 是否启用了 source map（dev 默认开，prod 看配置）
  • 是否有大量自定义 loader 强制走 Webpack 兼容层
  • 是否在 monorepo 中没正确设置 transpilePackages
  • 是否第一次 build（cold cache 时增量优势不体现）
  • v16 之前 prod build 通常比 Webpack 慢，v16+ GA 后追平 —— 升级版本

04 典型代码

代码 1 · next.config.ts 启用 Turbopack（v16 默认）

typescript 
// next.config.ts (v16+)
import type { NextConfig } from 'next';

const config: NextConfig = {
  // v16 起 Turbopack 默认开启，无需 --turbo flag
  // experimental.turbo 已稳定为顶层 turbopack 配置

  turbopack: {
    // SVG → React component
    rules: {
      '*.svg': {
        loaders: ['@svgr/webpack'],
        as: '*.js',
      },
    },
    // alias
    resolveAlias: {
      '@/': './src/',
    },
  },

  // Webpack fallback（仅在 turbopack 不工作时用）
  webpack(config) {
    // ...
    return config;
  },
};

export default config;

代码 2 · 与 Webpack 共存（迁移期）

json 
// package.json
{
  "scripts": {
    "dev": "next dev --turbo",
    "dev:webpack": "next dev",
    "build": "next build --turbo",
    "build:webpack": "next build"
  }
}
// 团队迁移期可以两套共存对比

代码 3 · 用 @next/bundle-analyzer 看 chunk

javascript 
// next.config.ts
import bundleAnalyzer from '@next/bundle-analyzer';

const withBundleAnalyzer = bundleAnalyzer({
  enabled: process.env.ANALYZE === 'true',
});

export default withBundleAnalyzer({
  // ...
});

// ANALYZE=true npm run build
// 浏览器打开 chunk 依赖图，看 client bundle 包含什么

代码 4 · monorepo + transpilePackages

typescript 
// apps/web/next.config.ts
export default {
  // 让 Turbopack 也编译这些 monorepo 内的 package
  transpilePackages: ['@myorg/ui', '@myorg/utils'],
};

05 常见陷阱

  • 用旧版本 next-bundle-analyzer 不兼容 Turbopack —— 升级到最新版本（含 Turbopack 适配）
  • 自定义 Webpack plugin 在 Turbopack 不工作 —— 确认 plugin 是否在 Turbopack 兼容列表（turbo.build 文档）；不兼容则只能继续 Webpack 或重写
  • Module Federation 项目升 Turbopack —— v16 仍不支持 MF，必须留 Webpack。等 v17 / v18 看 Turbopack 是否实现
  • monorepo 内 import 不工作 —— 加 transpilePackages 列表，告诉 Turbopack 编译 workspace 内的 package
  • "为什么 dev 启动很慢" —— 可能 cold cache（第一次启动）；可能用了大量 webpack-only 配置走兼容层；可能项目结构不利于增量计算
  • "为什么 HMR 慢" —— 检查是否改了 layout / providers 等顶层组件（会触发更大范围的重算）；考虑把 'use client' 推到下层
  • 跑 next build --turbo 失败但 next build 成功 —— prod build 路径与 dev 不同，prod 兼容性可能稍差。升 v16 后大多解决；如仍失败提交 issue
  • 对比 Vite 项目跑得更快然后纠结 —— 不同项目类型不可直接对比。Vite 在 dev 启动有先天优势（双模 + ESM）；Turbopack 在 monorepo 增量构建有优势。看你项目特点
  • "Turbopack 文档不完整" —— 真实情况，2026 仍在补。复杂配置看 turbo.build/pack docs + Next.js GitHub discussions + 实际代码
  • 认为 Turbopack 在所有场景都比 Webpack 快 —— 不一定。v15 prod build 还慢；v16 后追平。dev / HMR 是 Turbopack 优势区
  • "Webpack 还没死，Turbopack 是过度承诺" —— Webpack 短期不会死（生态太大），但 Next.js 内部 v17 起 Webpack 会进 deprecation 路径。Next.js 项目最终都会走 Turbopack
