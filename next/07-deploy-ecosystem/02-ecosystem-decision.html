生态决策树 + Next.js vs 同代人收束

这是 Next.js 主题的收束章。把 P1-P6 学到的所有东西串起来，给一份能直接用到生产的决策框架：用 Next.js 的时候 image / font / metadata / auth / 状态 / 数据 / 测试 选什么；什么时候应该不选 Next.js（Astro / RR v7 / TanStack Start 何时更合适）；2027 展望（Cache Components 心智成熟、RSC 协议被其他框架采用、AI SDK 差异化）。读完这一章你应该能独立判断"我们项目该用什么"。

阅读时长 ~25 分钟 前置 P1-P7.1 全部 主要来源 nextjs.org/docs 完整 + 各生态库官方 + 2026 公开调研数据

01 速览

  1. Next.js 生态在 2026 已经成熟到"不需要花太多时间选型" —— 大多数场景有事实标准（image: next/image / font: next/font / auth: NextAuth / 测试: Vitest+Playwright / 数据: TanStack Query 或 RSC + Server Actions）。
  2. next/image 在 Vercel 部署时几乎完美（自动 sharp + LQIP + responsive + lazy）；self-host 时要权衡（自跑 sharp / 用 Cloudflare Images / 直接 unoptimized）。是 LCP / CLS 优化的标准方案。
  3. next/font 把 Google Fonts 在 build 时下载并自托管，消除 FOIT/FOUT + 第三方请求。这是 next/* 系列里 ROI 最高的工具，不用就是浪费。
  4. next/metadata API 把 SEO + OG image + sitemap 用声明式 export 描述，比传统 React Helmet 简洁得多。generateMetadata 还能基于路由参数动态生成。
  5. NextAuth（Auth.js）是 Next.js 一等公民 auth 方案 —— OAuth + email link + 自托管 + 与 Server Actions 配合好。生产中如有强自定义 auth 需求，用 better-auth 或自己写更灵活。
  6. 状态管理：客户端用 Zustand（小项目）或 Jotai（atomic 模型）；server-side state 走 RSC + Server Actions（不需要全局 store）。Redux Toolkit 仍稳定但新项目少选。
  7. 数据获取：RSC + fetch + 'use cache'（推荐）/ TanStack Query（client-side 复杂场景，特别是无限滚动 / 实时同步）。两者可以共存。
  8. 测试：Vitest（单元 + 集成）+ Playwright（E2E）是 Next.js 项目标准栈。详见 测试主题。
  9. monorepo：pnpm + Turborepo 是 Next.js + pnpm-monorepo + Vercel 三方协同的事实标准；详见 pnpm-monorepo 主题。
  10. "何时不用 Next.js" 的回答比"何时用"更重要 —— 内容站点用 Astro 性能更好、纯 SPA + 自管 backend 用 RR v7 / TanStack Router 简单清晰、不接受 RSC 心智的团队选 RR v7、强 vendor-neutral 偏好选 TanStack Start。这些场景 Next.js 是 overkill 而不是优势。

02 底层逻辑

next/image · 图片优化全栈

Next.js 给 <img> 的最佳替代。核心特性：

  • 自动 lazy loading（loading="lazy" 默认，首屏可加 priority）
  • 响应式 srcset（自动生成多尺寸 srcset）
  • format negotiation（浏览器支持 AVIF / WebP 时自动选）
  • LQIP（low-quality image placeholder，用 placeholder="blur"）
  • CLS 防御（要求 width/height 或 fill，自动占位）
tsx 
import Image from 'next/image';
import heroImg from './hero.jpg';

export default function Hero() {
  return (
    <Image
      src={heroImg}                                {/* 或 string URL */}
      alt="Hero"
      placeholder="blur"                           {/* LQIP，自动从 import 提取 */}
      priority                                     {/* 首屏不 lazy */}
      sizes="(max-width: 768px) 100vw, 50vw"      {/* 响应式 srcset */}
    />
  );
}

// 远程图片：next.config.ts 加 remotePatterns
export default {
  images: {
    remotePatterns: [
      { protocol: 'https', hostname: 'cdn.example.com' },
    ],
  },
};

next/font · 字体优化

Google Fonts / 本地字体在 build 时下载 + 自托管 + 子集化 + 预加载。零运行时第三方请求。

tsx 
// app/layout.tsx
import { Inter, Noto_Sans_SC } from 'next/font/google';
import localFont from 'next/font/local';

const inter = Inter({ subsets: ['latin'] });
const noto = Noto_Sans_SC({ subsets: ['chinese-simplified'], display: 'swap' });
const customFont = localFont({ src: './fonts/MyFont.woff2', variable: '--font-custom' });

export default function RootLayout({ children }) {
  return (
    <html lang="zh-CN" className={`${inter.className} ${noto.className} ${customFont.variable}`}>
      <body>{children}</body>
    </html>
  );
}

结果：build 时下载字体到 .next/，运行时无第三方请求 + 字体预加载 link。FOIT/FOUT 消除（display:swap 避免空白闪烁）。

next/metadata API · 声明式 SEO

静态 metadata 直接 export：

tsx 
// app/layout.tsx
import type { Metadata } from 'next';

export const metadata: Metadata = {
  title: { default: 'My Site', template: '%s | My Site' },
  description: 'Site description',
  openGraph: {
    type: 'website',
    locale: 'zh_CN',
    url: 'https://example.com',
    siteName: 'My Site',
  },
  twitter: { card: 'summary_large_image', creator: '@me' },
};

动态 metadata（基于路由）用 generateMetadata：

tsx 
// app/blog/[slug]/page.tsx
import type { Metadata } from 'next';

export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>;
}): Promise<Metadata> {
  const { slug } = await params;
  const post = await getPost(slug);
  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      images: [{ url: post.coverImage, width: 1200, height: 630 }],
    },
  };
}

OG image 还可以 动态生成（用 ImageResponse）：

tsx 
// app/blog/[slug]/opengraph-image.tsx
import { ImageResponse } from 'next/og';

export const size = { width: 1200, height: 630 };
export const contentType = 'image/png';

export default async function Image({ params }) {
  const post = await getPost(params.slug);
  return new ImageResponse(
    (
      <div style={{ display: 'flex', fontSize: 60, background: 'white' }}>
        {post.title}
      </div>
    ),
    { ...size }
  );
}

build / runtime 自动生成 OG image，不需要 design 团队提供静态图。

NextAuth (Auth.js) · 认证一等公民

2024 年 NextAuth 改名 Auth.js（支持其他框架）。设计立场：OAuth + email link + 自托管 session。

tsx 
// auth.ts
import NextAuth from 'next-auth';
import GitHub from 'next-auth/providers/github';

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [GitHub],
  callbacks: {
    authorized({ request: { nextUrl }, auth }) {
      const isLoggedIn = !!auth?.user;
      const isOnDashboard = nextUrl.pathname.startsWith('/dashboard');
      if (isOnDashboard) return isLoggedIn;
      return true;
    },
  },
});

// app/api/auth/[...nextauth]/route.ts
export { handlers as GET, handlers as POST } from '@/auth';

// middleware.ts
export { auth as middleware } from '@/auth';

// 在 RSC / Server Action 里
import { auth } from '@/auth';
const session = await auth();

2026 备选：better-auth（更灵活的 self-hosted 方案）/ Clerk（managed service）/ Lucia（轻量自管）。NextAuth 仍是开源默认，但有自定义需求时其他选项可能更合适。

状态管理 · Zustand / Jotai / Redux Toolkit

App Router 时代状态管理的核心心智：能用 RSC + URL state 解决的不要用 client state。client state 库的角色比以前小。

需求 推荐
UI 状态（modal 开关 / theme） useState / useReducer（不需要库）
跨组件共享 + 不复杂 Zustand（小、简洁、SSR 友好）
原子化粒度 / 复杂派生 Jotai
已有大型 Redux 项目 Redux Toolkit（不要新引入）
Server state（fetch / cache / mutation） RSC + Server Actions（默认）/ TanStack Query（client 复杂场景）
URL 状态（filter / pagination） nuqs / 直接 useSearchParams
表单状态 useActionState（详见 P4）/ react-hook-form（client 复杂表单）

数据获取 · RSC + Server Actions vs TanStack Query

App Router 的数据心智：

  • 读 server 数据 → 在 Server Component 里 await（不需要库）
  • 写 server 数据 → Server Actions（不需要 fetch + setState 组合）
  • 缓存 → 'use cache' / fetch options
  • 失效 → revalidatePath / revalidateTag / updateTag

什么时候仍用 TanStack Query：

  • 无限滚动（上拉加载更多）
  • 实时同步（轮询 / WebSocket）
  • 乐观更新跨多个组件
  • 复杂 mutation 队列管理
  • 已有大量 TanStack Query 代码不愿迁移

RSC + Server Actions 解决 80% 场景；剩 20% 复杂 client-side 数据用 TanStack Query。两者可以在同一项目共存。

测试栈 · Vitest + Playwright

详见 测试主题。Next.js 项目的标准栈：

  • Vitest 跑单元测试 + 集成测试（utility / Server Action 内部逻辑 / 部分组件）
  • Playwright 跑 E2E（完整页面行为 / 跨页面流程 / 缓存行为验证）
  • 不推荐：Jest（v15+ 用 Vitest 更顺）/ Cypress（Playwright 后来居上）

monorepo · pnpm + Turborepo

详见 pnpm-monorepo 主题。三方协同：

  • pnpm workspace 管 package + 高效 install
  • Turborepo 管 task 调度 + cache + remote cache
  • Vercel 自动检测 Turborepo + 仅 build 受影响 app（remote cache 共享）

何时不用 Next.js

这是本主题最重要的反向决策。Next.js 不是万能的。

1. 内容站点 / 文档站 / blog → 用 Astro

  • Astro 默认 0 JS，性能极致
  • Markdown / MDX 一等公民
  • 不需要 RSC 心智 + 缓存复杂度
  • Astro Starlight 是 React 生态 docs site 主流

2. 纯 SPA + 自管 backend → 用 Vite + React Router

  • 内部 dashboard / admin 后台 / 桌面级应用
  • 不需要 SEO，登录后才用
  • SPA 模型最简单，不需要 SSR / RSC / 缓存层

3. 不接受 RSC 心智 + 偏 Web 标准 → 用 React Router v7

  • Remix 老团队继续投入
  • loaders + actions 模型清晰
  • vendor-neutral，自由部署

4. 强类型 / vendor-neutral 偏好 → 用 TanStack Start

  • 类型安全比 Next.js 更彻底
  • 与 TanStack Query 集成无缝
  • 2026 仍属早期采用

5. 已经在 Cloudflare 生态 → 选 Next.js + OpenNext.js 或 SvelteKit / Astro（Cloudflare 一等支持）

2027 展望

基于 2026 主线状态推测：

  • Cache Components 心智成熟 —— v16 是过渡，v17 / v18 把 'use cache' / PPR 打磨完善；缓存事故大幅减少
  • RSC 协议被其他框架采用 —— RR v7、TanStack Start 都在路线图加 RSC；2027 RSC 不再是 Next.js 独有护城河
  • Turbopack 取代 Webpack —— v17 / v18 Webpack 进入 deprecation；Next.js 项目最终全 Turbopack
  • AI SDK 差异化 —— Vercel AI SDK 与 Next.js 深度集成（streaming / tool use / agent 模式），成为 Next.js 新一代差异化点（详见待写 AI 主题）
  • Component Testing 模糊单元/集成边界 —— Vitest browser mode + Playwright Component Testing 让"测试组件 in real browser"普及
  • Edge runtime 能力扩展 —— Vercel 持续扩 Edge 能力（Durable Objects-like / WebSocket server / 更多 npm 包兼容）

03 应用场景

场景 1 · 起一个新 SaaS（dashboard + 订阅）

推荐栈：

  • Framework：Next.js 16 + App Router + cacheComponents
  • UI：Tailwind + shadcn/ui + Radix
  • 状态：Zustand（少量 client state）+ RSC + Server Actions（server state）
  • 数据库：PostgreSQL + Drizzle / Prisma
  • Auth：NextAuth / better-auth（按需求）
  • 支付：Stripe + Server Actions（webhook 用 Route Handler）
  • 邮件：Resend + React Email
  • 测试：Vitest + Playwright
  • 部署：Vercel（首选）/ standalone + Redis（self-host）
  • monorepo：pnpm + Turborepo（如果多 app）
  • 错误监控：Sentry
  • 分析：PostHog / Vercel Analytics

场景 2 · 公司官网 + 文档

推荐栈：

  • Framework：Astro（不是 Next.js）
  • 文档：Astro Starlight
  • blog：Astro Content Collections + MDX
  • 交互组件（订阅表单 / 评论）：React + Astro Islands
  • 部署：Vercel / Cloudflare Pages / Netlify
  • 说明：用 Next.js 是 overkill，Astro 性能 + 维护更优

场景 3 · 内部管理后台

推荐栈：

  • Framework：Vite + React + React Router v7（pure SPA 模式）
  • UI：shadcn/ui + Tanstack Table + Tanstack Form
  • 数据：TanStack Query + 自己后端 API
  • 状态：Zustand（如需要）
  • 部署：S3 + CloudFront / Cloudflare Pages（纯静态）
  • 说明：内部用，无 SEO，SPA 简单清晰

场景 4 · 已有 Remix 项目继续投入

推荐：

  • 升级到 React Router v7（自动迁移）
  • 继续 loader + action 模型
  • 2026-2027 评估是否引入 RSC（v7 路线图加 RSC 是可选）
  • 不要为了"跟随主流"迁 Next.js（迁移成本 30-40%，业务价值不高）

代码 2 · OG image 动态生成

tsx 
// app/blog/[slug]/opengraph-image.tsx
import { ImageResponse } from 'next/og';
import { getPost } from '@/lib/posts';

export const alt = 'Blog post';
export const size = { width: 1200, height: 630 };
export const contentType = 'image/png';

export default async function Image({ params }: { params: { slug: string } }) {
  const post = await getPost(params.slug);
  return new ImageResponse(
    (
      <div style={{
        display: 'flex',
        flexDirection: 'column',

        height: '100%',

      </div>
    ),
    { ...size }
  );
}

代码 3 · 完整 Auth + middleware + Server Action 集成

tsx 
// auth.ts
import NextAuth from 'next-auth';
import GitHub from 'next-auth/providers/github';

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [GitHub],
});

// middleware.ts
export { auth as middleware } from '@/auth';
export const config = {
  matcher: ['/dashboard/:path*'],
};

// app/_actions/profile.ts
'use server';
import { auth } from '@/auth';
import { revalidatePath } from 'next/cache';

export async function updateProfile(formData: FormData) {
  const session = await auth();
  if (!session) throw new Error('Unauthorized');
  await db.users.update({
    where: { id: session.user.id },
    data: { name: formData.get('name') as string },
  });
  revalidatePath('/dashboard/profile');
}

05 常见陷阱

  • 用 Next.js 做内容站 —— overkill。Astro 性能 + 维护更优；Next.js 的 RSC + 缓存优势在内容站点用不到
  • 用 Next.js 做内部管理后台 —— 也是 overkill。SSR + RSC 不需要；Vite + React Router 简单得多
  • "先把 Next.js 全部学完再写代码" —— Next.js 太大学不完。读完 P1 + P2 + P3 + P4，然后写一个真实项目，遇到问题查 P5-P7。先动手
  • 用 React Helmet / next-seo 而不是 next/metadata —— v13+ 用 metadata API。next-seo 等老库可能仍在但已被官方 API 取代
  • "我项目用 Next.js 但不用 next/image" —— LCP / CLS 损失大。除非有强理由（用 Cloudflare Images），都应该用 next/image
  • 用 useEffect 做数据获取（在 App Router 里） —— 可以但不是最佳。Server Component await 更简洁；流式 + 缓存自动支持
  • 所有 client state 用 Redux Toolkit —— 新项目 overkill。绝大多数场景 useState + Zustand 已够
  • "Auth.js 不够灵活"然后自己写 auth —— 自己写 auth 是 known security risk。Auth.js + 自定义 callback 通常够；真不行用 better-auth / Clerk
  • 用 Next.js 但部署到 Cloudflare Workers 没用 OpenNext.js —— 直接部署不工作。OpenNext.js 是 Cloudflare 适配标准
  • "vendor-locked Vercel 是问题" —— 大多数团队不会真的迁出。但如果你真有跳走计划，从一开始用 vendor-neutral 工具（Upstash / Neon / Cloudflare）
  • 同时用 RSC + TanStack Query —— 不是错，但要清晰职责。RSC 拉取的初始数据传给 TanStack Query 作为 initialData 才能两全其美
  • 不用 monorepo 但项目复杂度上来了 —— 中等规模项目（多端 + 共享 UI）应该早 monorepo。pnpm + Turborepo 学习成本不高
  • "我担心 Next.js 又重构" —— 合理。但 v15 v16 已经把缓存反转 + Cache Components 的"修正"做完，2027-2028 大概率稳定。可以 v15 LTS 风格使用，等 v17 LTS 再升
