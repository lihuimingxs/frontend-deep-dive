Edge / Node Runtime + 部署矩阵

Next.js 代码可以跑在两种 runtime（Node.js / Edge V8 isolate）+ 部署到4 种模式（Vercel / standalone / static export / Cloudflare Workers）。每种组合的能力和限制不同。这一章讲清两个 runtime 的差异、Edge 的限制、4 种部署的取舍、self-host 的真实痛点（缓存持久化 / ISR 协调 / Image Optimization 自部署），以及一份能直接拿到生产用的 Dockerfile + Redis cache handler。

阅读时长 ~30 分钟 前置 P5 缓存系统（self-host 必读）+ V8 主题 P5.1（V8 isolate） 主要来源 nextjs.org/docs deploying + Vercel infra blog + OpenNext.js（Cloudflare 适配）+ 真实生产案例

01 速览

  1. Next.js 代码可以指定跑在 Node.js runtime（默认，完整 Node API）或 Edge runtime（V8 isolate，Web APIs 子集）。通过 export const runtime = 'edge' | 'nodejs' 在 page / layout / route handler 顶部指定。
  2. middleware.ts 永远在 Edge runtime —— 不能改。这意味着 middleware 不能用 Node API（fs / child_process / 部分 npm 包）。
  3. Edge runtime 的限制：没有 fs / child_process；部分 npm 包（特别是 native bindings）不可用；Vercel 上有内存（128MB）+ CPU 时间（50ms-30s 不等）+ 包大小（1-5MB）限制；冷启动几乎为 0。
  4. Node runtime 的特点：完整 Node API、适合 dashboard / dynamic page / 复杂 SDK；冷启动相对慢（100-500ms）；Vercel Functions 有更高时间和内存上限。
  5. 4 种部署模式：Vercel（一等支持，所有特性即开即用）/ output: standalone（Node.js 自部署，需自管 cache 持久化 + ISR 协调）/ output: export（纯静态，不支持 ISR / Server Actions / middleware）/ Cloudflare Workers / Pages（通过 OpenNext.js 适配，2026 已成熟）。
  6. self-host 三大痛点：1) 缓存持久化 —— 默认 fs cache 在容器重启会丢；要配置 cacheHandler 接 Redis；2) ISR / revalidate 协调 —— 多实例间需同步失效；3) Image Optimization —— Vercel 自带，self-host 要自部署 sharp 或禁用。
  7. v16 的好消息：自定义 cache handler 接口稳定了（v15 还是 unstable_）；多实例 ISR 协调有官方文档；OpenNext.js 在 Cloudflare 已 production-ready。
  8. 选哪个部署：初创 / 小团队 → Vercel（一键 + 零运维）；有 ops 能力 + 已有 K8s → standalone + Redis；纯营销站 → static export 上 CDN；Cloudflare 生态 → Workers + OpenNext.js。
  9. middleware 是 Edge 一等公民 —— 适合 a/b test / auth check / geo redirect / rate limit 等"路由前的轻量处理"。重逻辑不要塞 middleware（CPU 时间限制）。
  10. 不要为了 Edge 性能而把所有东西塞 Edge runtime —— Edge 限制多，业务逻辑常常需要 Node runtime。Edge 适合 middleware 和简单 route handler，Node 适合大部分 page。

02 底层逻辑

两个 runtime 的本质差异

Node.js runtime：标准 Node 进程，完整 Node API，npm 包大多可用。Vercel 上跑在 AWS Lambda（Functions）；self-host 跑在 Docker / Node server。

Edge runtime：V8 isolate（不是完整 Node 进程）。只支持 Web APIs（fetch / Request / Response / Headers / URL 等），不支持 Node API。Vercel 上跑在 Cloudflare-like 边缘网络；冷启动 ~5ms（vs Lambda 100-500ms）。

Web APIs vs Node APIs 的差异：

API Node runtime Edge runtime
fs / path ✅ ❌
child_process ✅ ❌
fetch / Request / Response ✅（v18+） ✅
crypto.subtle ✅ ✅
Buffer ✅ ❌（用 Uint8Array）
process.env ✅（完整） ✅（仅 build-time 嵌入）
WebSocket（client） 需库 ✅（原生）
WebSocket（server） 需库 ✅（v16+ Vercel）
Streams Node Streams Web Streams

选 runtime 的判断

Edge 适合：

  • middleware（强制 Edge）
  • 简单 route handler（auth check / a/b test / redirect）
  • 低延迟首响（用户在全球分布，Edge 近用户）
  • 不依赖 Node-only npm 包

Node 适合：

  • 大多数 page（用 ORM / 复杂 SDK / sharp 等 native lib）
  • 复杂业务逻辑（CPU 时间长）
  • 需要文件系统访问
  • 使用 native bindings 的 npm 包

2026 实战中：middleware Edge、大部分 page Node、热路由（如 health check / simple auth）可以挑出来 Edge。

middleware · Edge 一等公民

middleware.ts 是 Next.js 的"路由前"hook，永远在 Edge runtime。典型用例：

tsx 
// middleware.ts
import { NextRequest, NextResponse } from 'next/server';

export async function middleware(req: NextRequest) {
  // 1. Geo-based redirect
  const country = req.geo?.country || 'US';
  if (country === 'CN' && !req.url.includes('/cn/')) {
    return NextResponse.redirect(new URL('/cn' + req.nextUrl.pathname, req.url));
  }

  // 2. Auth check
  const session = req.cookies.get('session')?.value;
  if (!session && req.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', req.url));
  }

  // 3. A/B test
  const variant = req.cookies.get('ab-variant')?.value || (Math.random() < 0.5 ? 'a' : 'b');
  const res = NextResponse.next();
  res.cookies.set('ab-variant', variant);
  return res;
}

export const config = {
  matcher: ['/((?!api|_next/static|favicon.ico).*)'],
};

注意事项：

  • 只能用 Web APIs（不能 fs / child_process）
  • 不能用 ORM 直连数据库（用 fetch / 轻量 SDK 替代）
  • 有 CPU 时间限制（Vercel 50ms 软上限）
  • matcher 配置严格 —— 排除 _next / api / static asset 防止重复执行

4 种部署模式详解

1. Vercel（一等公民）

所有特性即开即用：

  • Edge Functions / Node Functions 自动调度
  • Cache Components / ISR / 'use cache' 自动持久化
  • Image Optimization 内置（自动 sharp）
  • Edge middleware 全球分布
  • git push 自动部署 + preview environment

代价：vendor lock-in。但对中小团队几乎不是问题（迁移成本低）。

2. output: 'standalone'

把 Next.js 应用打包成自包含 Node server，可在 Docker / EC2 / Kubernetes 跑：

typescript 
// next.config.ts
export default {
  output: 'standalone',
};

// build 后产物在 .next/standalone/
// node .next/standalone/server.js 即启动

痛点：

  • 缓存默认 fs，容器重启全丢；必须配 cacheHandler 接 Redis
  • ISR / revalidate 在多实例间不自动同步；需要 Redis pub/sub 或 webhook
  • Image Optimization 需要自己跑 sharp 服务（或禁用，让 client 加载原图）
  • 没有 Vercel KV / Postgres 这种 zero-config 服务，要自部署

3. output: 'export'（纯静态）

typescript 
// next.config.ts
export default {
  output: 'export',
};

// build 后产物在 out/，全是静态 HTML + 资源
// 直接上传 S3 / Netlify / GitHub Pages

限制：

  • 不支持 Server Actions
  • 不支持 Middleware
  • 不支持 ISR / on-demand revalidation
  • 不支持 next/image 默认（要 unoptimized: true 或自托管）
  • 不支持 dynamic page（catch-all / generateStaticParams 必须穷举）

适合：纯营销站 / 文档站 / blog（生成一次后不变 / 少量变化重新 build）。

4. Cloudflare Workers / Pages（通过 OpenNext.js）

OpenNext.js 是社区维护的适配器，把 Next.js 应用编译成可在 Cloudflare 跑的产物。2026 年已 production-ready。

  • 所有 server code 跑在 Cloudflare Workers（Edge）
  • cache 用 Cloudflare KV / R2
  • middleware / Server Actions / RSC 都支持
  • 限制：Workers 是 Edge runtime（没 Node API），所以 ORM 选择有限（推荐 Drizzle + neon / planetscale + http）

适合：已经在 Cloudflare 生态（用 R2 / KV / Durable Objects） + 想跳出 Vercel vendor lock。

self-host 缓存持久化 · cache handler

self-host 时配置 Redis cache handler 是必备步骤。看完整实现：

typescript 
// cache-handler.ts
import { CacheHandler, CacheHandlerValue } from 'next/dist/server/lib/incremental-cache';
import Redis from 'ioredis';

export default class RedisCacheHandler implements CacheHandler {
  private client: Redis;

  constructor(options: any) {
    this.client = new Redis(process.env.REDIS_URL!);
  }

  async get(key: string): Promise<CacheHandlerValue | null> {
    const data = await this.client.get(`next:${key}`);
    return data ? JSON.parse(data) : null;
  }

  async set(key: string, data: CacheHandlerValue, ctx: { tags?: string[] }): Promise<void> {
    const serialized = JSON.stringify(data);
    await this.client.set(`next:${key}`, serialized);
    for (const tag of ctx.tags || []) {
      await this.client.sadd(`tag:${tag}`, key);
    }
  }

  async revalidateTag(tags: string | string[]): Promise<void> {
    const tagList = Array.isArray(tags) ? tags : [tags];
    for (const tag of tagList) {
      const keys = await this.client.smembers(`tag:${tag}`);
      if (keys.length > 0) {
        await this.client.del(...keys.map(k => `next:${k}`));
        await this.client.del(`tag:${tag}`);
      }
    }
  }
}

// next.config.ts
export default {
  cacheHandler: require.resolve('./cache-handler.ts'),
  output: 'standalone',
};

关键点：

  • 所有 cache key 用 prefix（next:）避免与其他应用冲突
  • tag 用 SET 数据结构索引（O(1) 失效）
  • 多实例共享 Redis = 自动 ISR 协调
  • 生产用要加 monitoring + 自动 TTL 清理（防止无限增长）

Image Optimization · self-host 选择

Vercel 部署时 next/image 自动优化（裁剪 / 压缩 / 格式转换）。self-host 时三个选项：

  1. 禁用 Optimization：images: { unoptimized: true } —— 简单但失去 next/image 优势
  2. 自托管 sharp：在 server 安装 sharp，由 Next.js 在 runtime 处理。需要 server 有 CPU 余量
  3. 用第三方：Cloudinary / imgix / Cloudflare Images。custom loader 接入

2026 推荐：中型项目用 sharp 自托管（标准 Dockerfile 已含）；大型项目用 Cloudflare Images（成本可控 + 全球 CDN）。

场景 2 · Kubernetes 多实例 + Redis pub/sub revalidate

多实例部署时，Redis cache handler 让 cache 在实例间共享。但 ISR revalidate 触发时（比如 Server Action 调用 revalidateTag），需要让所有实例知道某个 tag 已失效。

typescript 
// cache-handler.ts (多实例版本)
export default class RedisCacheHandler implements CacheHandler {
  private client: Redis;
  private subscriber: Redis;

  constructor() {
    this.client = new Redis(process.env.REDIS_URL!);
    this.subscriber = this.client.duplicate();
    // 订阅 revalidate channel，其他实例触发的 revalidate 通知所有实例
    this.subscriber.subscribe('revalidate-tag');
    this.subscriber.on('message', (_channel, tag) => {
      // 清理本实例的内存 cache（如果有）
    });
  }

  async revalidateTag(tags: string | string[]): Promise<void> {
    // ... 失效 Redis 中的 cache
    // 然后 publish 给所有实例
    const tagList = Array.isArray(tags) ? tags : [tags];
    for (const tag of tagList) {
      await this.client.publish('revalidate-tag', tag);
    }
  }
}

场景 3 · middleware 实现 a/b test

tsx 
// middleware.ts
import { NextRequest, NextResponse } from 'next/server';

export function middleware(req: NextRequest) {
  // 检查是否已分配 variant
  let variant = req.cookies.get('ab-variant')?.value;
  if (!variant) {
    // 50/50 随机分配
    variant = Math.random() < 0.5 ? 'control' : 'experiment';
  }

  // rewrite 到对应 page（不改 URL）
  const url = req.nextUrl.clone();
  if (variant === 'experiment') {
    url.pathname = '/_experiments' + url.pathname;
  }
  const res = NextResponse.rewrite(url);

  // 设置 cookie，下次访问保持同 variant
  res.cookies.set('ab-variant', variant, { maxAge: 60 * 60 * 24 * 30 });
  return res;
}

export const config = {
  matcher: ['/landing'],
};

05 常见陷阱

  • middleware 里调用 ORM 直连数据库 —— middleware 是 Edge runtime，没 Node API。用 fetch 调用 /api 端点 / 用 Edge 兼容 ORM（Drizzle + http）
  • 给整个 page 标 runtime: 'edge' 然后崩 —— Edge 不支持 sharp / canvas / 部分 native bindings。先确认所有依赖兼容 Edge
  • self-host 容器重启后所有 cache 丢 —— 默认 fs cache 不持久。配置 cacheHandler 接 Redis
  • 多实例 self-host 但 cache 不同步 —— Redis cache handler 解决数据共享，但 revalidateTag 触发要 publish/subscribe（看场景 2）
  • output: export 然后用 Server Actions —— 编译报错。export 模式不支持 Server Actions
  • output: standalone 但 Image Optimization 不工作 —— sharp 没装。Dockerfile 加 RUN apk add --no-cache libc6-compat + 安装 sharp
  • middleware matcher 写错导致重复执行 —— matcher 应该排除 _next / api / 静态资源；否则 middleware 在静态资源请求上也跑（性能差 + bug 多）
  • middleware 跑得慢导致 Vercel CPU 超时 —— middleware 不是放任意逻辑的地方。只放路由前轻量处理（auth check / redirect / rewrite）
  • Cloudflare Workers + OpenNext.js 时用 Prisma —— Prisma binary 大，超 Workers 包大小限制。用 Drizzle / Kysely + neon-http / planetscale 替代
  • "为什么我的 ISR 在 self-host 不工作" —— 默认 fs cache + 多实例不同步。看 P5 + 本章 cache handler 部分
  • Vercel Preview environment 用 prod 数据库 —— 危险。每个 preview 都连 prod 会污染数据。用环境变量在 preview 切到 staging db
  • "Edge runtime 一定比 Node 快" —— 冷启动是。但执行复杂逻辑 Edge CPU 限制反而更紧。具体看任务类型
  • 用 Vercel KV / Postgres 然后想 self-host 时迁移痛苦 —— Vercel 服务 vendor-locked。如果可能跳走，从一开始用 vendor-neutral 替代（Upstash / Neon / Supabase）
