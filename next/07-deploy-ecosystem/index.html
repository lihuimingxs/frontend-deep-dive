P7 · 部署 + 生态决策

Next.js 主题的收束阶段。先讲清"代码到底跑在哪"——Edge runtime（V8 isolate）vs Node runtime + Vercel / standalone / static export / Cloudflare Workers 部署矩阵 + self-host 时缓存层怎么搭；再讲生态选型——image / font / metadata / auth / 测试 / monorepo + Next.js vs RR v7 vs TanStack Start 的最终决策。

章节数 2 主要来源 nextjs.org/docs deploying + Vercel infra blog + OpenNext.js（Cloudflare 适配）+ NextAuth/Auth.js docs 状态 ✅ 全部完成

本阶段覆盖

  • Node.js runtime（默认） · 完整 Node API；冷启动慢；适合 dashboard / dynamic page
  • Edge runtime · V8 isolate + Web APIs 子集；冷启动几乎 0；middleware 永远在 Edge
  • Edge 限制 · 没 fs/child_process / 部分 npm 包不可用 / 内存 + CPU 时间 + 包大小限制
  • output: standalone · Node.js 自部署；ISR + revalidate 需持久化层
  • output: export · 纯静态；不支持 ISR / Server Actions / Edge middleware
  • Cloudflare Workers / Pages · 通过 OpenNext.js 适配；2026 已成熟
  • self-host 痛点 · 缓存持久化 / ISR 协调 / Image Optimization 自部署 sharp
  • 自定义 cache handler 接 Redis · v16 文档新增的官方扩展点
  • next/image · sharp + LQIP + lazy + responsive，解决 LCP/CLS
  • next/font · build-time 字体优化，自托管 Google Fonts
  • next/metadata · 声明式 SEO + OG image + sitemap
  • NextAuth (Auth.js) · OAuth + magic link + 自托管
  • 状态 / 数据 / 测试栈选择 · Zustand/Jotai · TanStack Query · Vitest+Playwright
  • 何时不用 Next.js · SPA + 自管 backend / 不接受 vendor-locked / 内容站 / 不接受 RSC 心智
  • 2027 展望 · Cache Components 心智稳定 / RSC 协议被其他框架采用 / AI SDK 差异化
