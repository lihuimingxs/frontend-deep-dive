流式 + Selective Hydration + Loading UX

流式渲染是 Next.js 用户能直接感知的最大特性 —— 一个 dashboard 不再等所有数据齐全才显示，而是慢段独立 reveal、快段先可交互。这一章把用户视角的流式讲透：loading.tsx 怎么映射 Suspense、嵌套边界怎么分段 reveal、Selective Hydration 让快段先可点、避免 waterfall 的提前 fetch 模式、error.tsx 错误恢复、template 与 layout 的语义差。配 Network timeline 截图说话，不停留在 API 罗列。

阅读时长 ~30 分钟 前置 P3.1 RSC payload + React 18+ Suspense 基础 主要来源 nextjs.org/docs streaming + React 18 Concurrent Renderer 文档 + Andrew Clark "Why Selective Hydration matters"

01 速览

  1. loading.tsx 是文件约定，等价于在 page 外包一层 <Suspense fallback={<Loading />}>。每个段（layout / page）都可以独立配 loading.tsx。
  2. 显式 <Suspense> 比 loading.tsx 更细粒度 —— loading.tsx 是"整个段加载时显示一个 fallback"；显式 Suspense 可以包某个慢组件让"快组件已显示，慢组件继续等"。
  3. Suspense 嵌套 = 流式分段 reveal。每个 Suspense 边界独立 flush，用户看到的是"逐段填充"而不是"全有或全无"。
  4. React 18 的 Selective Hydration 让"快段先可交互" —— 假设页面有 3 段 client component，第 1 段 hydrate 完用户已可点击，不必等第 2/3 段。这与"一次性 hydrate 整棵树"的传统 SSR 完全不同。
  5. 避免 fetch waterfall 的关键：在父组件把 fetch 早早触发，子组件 await。或者用 Promise.all 并行多个 fetch。最忌讳"父等子开始 fetch"链式串行。
  6. error.tsx 是错误边界，必须 'use client'。它接 error（含 digest）和 reset（重新尝试渲染）两个 prop。global-error.tsx 仅用于 root layout 错误。
  7. template.tsx 与 layout.tsx 的关键区别：layout 路由切换不重新挂载（state 保留），template 每次都重新挂载（用于 enter animation / reset state）。
  8. 流式渲染的真实 UX 价值不是"快"（绝对时间不变），是"感知快"和"感知有进度"。骨架屏让用户知道"在加载"，独立 reveal 让用户感觉"已开始 work"。
  9. 设计 Suspense 边界的原则：把"快的"放外面、"慢的"包起来；每个独立慢段单独 Suspense；fallback 是有意义的骨架（不要空 div / 不要旋转图标）。
  10. 流式不是免费 —— 流式 HTTP response 不能 gzip 压缩到极致（chunk 边界打断），CDN 缓存策略也复杂（每个 chunk 单独 cache vs 整页 cache）。所以"是否启用流式"也是工程决策（详见 P5 + P7）。

02 底层逻辑

loading.tsx · 文件约定背后的 Suspense

App Router 看到 app/dashboard/loading.tsx 时，编译时把它转成：

tsx 
// 你写的：
// app/dashboard/page.tsx
export default function Dashboard() { return <Stats />; }
// app/dashboard/loading.tsx
export default function Loading() { return <Skeleton />; }

// Next.js 编译后等价于：
function DashboardSegment() {
  return (
    <Suspense fallback={<Loading />}>
      <Dashboard />
    </Suspense>
  );
}

这就是为什么 loading.tsx 自动生效 —— Suspense 在 React 层包好了，server 渲染时遇到 page 内部 await 不 ready 就显示 fallback。

每个段（每个文件夹）可以有自己的 loading.tsx。导航到 /dashboard/settings 时，路径上每个 loading.tsx 都可能触发：

  • app/dashboard/loading.tsx —— dashboard layout 加载时显示
  • app/dashboard/settings/loading.tsx —— settings page 加载时显示

嵌套时外层先 ready 才到内层。如果 dashboard layout 已 ready（之前导航过），切到 settings 只触发 settings 的 loading.tsx。

显式 Suspense vs loading.tsx · 何时用哪个

loading.tsx 是"段级"的 —— 整个 page 加载时显示一个 fallback。但很多场景需要"page 内部某个慢组件 fallback、其他立即显示"。这时用显式 Suspense：

tsx 
// app/dashboard/page.tsx
import { Suspense } from 'react';

export default function Dashboard() {
  return (
    <>
      {/* 标题立即显示，不等任何 fetch */}
      <h1>Dashboard</h1>

      {/* Stats 内部 await fetch，慢就显示骨架；快就立即显示 */}
      <Suspense fallback={<StatsSkeleton />}>
        <Stats />
      </Suspense>

      <Suspense fallback={<ChartSkeleton />}>
        <Chart />
      </Suspense>
    </>
  );
}

async function Stats() {
  const data = await fetchStats();  // 800ms
  return <StatsCards data={data} />;
}

async function Chart() {
  const data = await fetchChart();  // 3000ms
  return <LineChart data={data} />;
}

对比：

  • 用 loading.tsx：整个 page 不 ready 都显示骨架，3000ms 后突然全显示
  • 用嵌套 Suspense：标题立即显示（0ms）→ Stats 800ms 显示 → Chart 3000ms 显示

第二种 UX 远好。loading.tsx 适合"整段加载"的场景（比如刚导航过来时给一个全屏 skeleton），嵌套 Suspense 适合"段内细粒度流式"。

Selective Hydration · 让快段先可交互

React 18 引入 Selective Hydration，意思是"每个 Suspense 边界独立 hydrate"。这与 React 17 的"必须 hydrate 整棵树才能交互"完全不同。

看一个例子：

tsx 
// app/dashboard/page.tsx
<>
  <Header />                                  {/* RSC */}
  <Suspense>
    <FastClientWidget />                       {/* client, ready 100ms */}
  </Suspense>
  <Suspense>
    <SlowClientWidget />                       {/* client, ready 3000ms */}
  </Suspense>
</>

React 18+ 行为：

  1. 0ms：HTML 收到，浏览器显示首屏（含 Header markup + 两个 fallback）
  2. 100ms：FastClientWidget 数据 ready，server flush 它的 HTML + RSC chunk
  3. 稍后：浏览器加载 client bundle，开始 hydrate
  4. FastClientWidget 先 hydrate 完成 —— 用户立即可以与它交互，不等 SlowClientWidget
  5. 3000ms：SlowClientWidget ready，flush + hydrate

这是 React 18 + Next.js App Router 的核心工程价值之一。React 17 时代：必须等整棵树 hydrate 完用户才能点任何东西，慢页面给人"看起来好了但点不动"的体验。React 18 让 hydration 也变成"分段进行"。

更进一步：如果用户点击了 SlowClientWidget 的位置（hydration 还没完成），React 会优先 hydrate 那个位置（基于用户交互的优先级提升）。这是 Concurrent Renderer 的内部能力。

避免 waterfall · 在父组件提前触发 fetch

RSC 让"组件层 fetch"成为可能，但带来的副作用是 fetch waterfall —— 父组件 await 完才轮到子组件 await，串行而非并行。

tsx 
// ❌ Waterfall：串行 await
async function Page() {
  const user = await fetchUser();           // 200ms
  const posts = await fetchPosts(user.id);  // 300ms（必须等 user 完成）
  const comments = await fetchComments();   // 400ms（必须等 posts 完成）
  // 总耗时：900ms
  return <>...</>;
}

修复方式 1 · Promise.all 并行（如果数据无依赖）：

tsx 
// ✅ 并行
async function Page() {
  const [user, posts, comments] = await Promise.all([
    fetchUser(),
    fetchPosts(),
    fetchComments(),
  ]);
  // 总耗时：max(200,300,400) = 400ms
  return <>...</>;
}

修复方式 2 · 早早触发 fetch（数据有依赖时）：

tsx 
// ✅ 父早触发，子 await
async function Page() {
  const userPromise = fetchUser();           // 立即触发
  const commentsPromise = fetchComments();   // 立即触发，不等 user
  const user = await userPromise;
  const postsPromise = fetchPosts(user.id);  // 现在才能开始
  const [posts, comments] = await Promise.all([postsPromise, commentsPromise]);
  return <>...</>;
}

修复方式 3 · 把 promise 传给子组件，子组件 await（流式 reveal）：

tsx 
// ✅ Suspense + 提前触发
async function Page() {
  const postsPromise = fetchPosts();     // 不 await
  const commentsPromise = fetchComments(); // 不 await
  return (
    <>
      <Suspense fallback={<Skeleton />}>
        <PostsList promise={postsPromise} />
      </Suspense>
      <Suspense fallback={<Skeleton />}>
        <CommentsList promise={commentsPromise} />
      </Suspense>
    </>
  );
}

async function PostsList({ promise }: { promise: Promise<Post[]> }) {
  const posts = await promise;
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}

error.tsx · 错误边界与恢复

error.tsx 是 React Error Boundary 在 App Router 的文件约定。每个段可以独立配，错误冒泡到最近的 error.tsx 处理。

tsx 
// app/posts/[id]/error.tsx
'use client';
import { useEffect } from 'react';

export default function PostError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;  // 调用让 React 重新尝试渲染段
}) {
  useEffect(() => {
    console.error('Post page error:', error);
    // 上报 Sentry / Datadog
  }, [error]);

  return (
    <div>
      <h2>无法加载这篇文章</h2>
      {/* error.message 在 prod 被替换成通用消息（防 secret 泄露）；digest 是 server 错误 ID */}
      <p>Error ID: {error.digest}</p>
      <button onClick={reset}>重试</button>
    </div>
  );
}

关键点：

  • 必须 'use client'（React Error Boundary 是 client 概念）
  • error.digest 是 server 错误 ID，可以与 server log 对应排查
  • prod 环境 error.message 被 Next.js 替换成通用消息（防止 secret 泄露到客户端）
  • reset() 让 React 重新尝试渲染该段（适合"网络抖动后重试"）

global-error.tsx 仅用于 root layout 错误。普通 error.tsx catch 不到 root layout 内的错误（因为 error.tsx 包在 layout 内部，root layout 出错时 error.tsx 不存在）：

tsx 
// app/global-error.tsx
'use client';
export default function GlobalError({ error, reset }: any) {
  return (
    <html><body>
      <h2>Critical error</h2>
      <button onClick={reset}>Try again</button>
    </body></html>
  );
}

template.tsx vs layout.tsx · 重新挂载语义

layout.tsx 在路由切换时不重新挂载（state 保留）。template.tsx 反之 —— 每次都重新挂载。两者 file 位置可以重合（同一段同时有 layout 和 template）。

用 template 的典型场景：

  • 页面切换 enter animation 每次都触发（layout 不重新挂载，animation 不会重跑）
  • 每次进入页面 reset 某个 state（比如表单数据 / 滚动位置）
  • 每次进入页面 effect 重跑（埋点 / 上报）
tsx 
// app/dashboard/template.tsx
'use client';
import { motion } from 'framer-motion';

export default function Template({ children }: { children: React.ReactNode }) {
  return (
    <motion.div initial={{ opacity: 0, y: 12 }} animate={{ opacity: 1, y: 0 }}>
      {children}
    </motion.div>
  );
}

// app/dashboard/layout.tsx
export default function Layout({ children }: { children: React.ReactNode }) {
  return <div className="dashboard-grid">{children}</div>;
}

顺序：layout 包 template 包 page。Layout 始终挂载（提供 grid + sidebar 等持久化 UI）；Template 每次切换重新挂载（触发 fade-in 动画）。

03 应用场景

场景 1 · Dashboard 三段慢接口的 timeline 设计

典型 SaaS dashboard 页面：

  • 导航条（用户信息）—— 100ms（缓存）
  • 核心指标卡片（昨日数据）—— 800ms（数据库）
  • 实时图表（过去 24 小时）—— 3000ms（复杂查询）

设计目标：用户进入页面 100ms 内看到导航 + 骨架，3000ms 完整显示，期间逐段填充。

tsx 
// app/dashboard/page.tsx
import { Suspense } from 'react';

export default function Dashboard() {
  return (
    <div className="grid">
      <NavBar />                                    {/* 100ms，作为父级直接 await */}
      <Suspense fallback={<StatsCardsSkeleton />}>
        <StatsCards />
      </Suspense>
      <Suspense fallback={<ChartSkeleton />}>
        <RealtimeChart />
      </Suspense>
    </div>
  );
}

async function NavBar() {
  const user = await getCurrentUser();  // 100ms
  return <header>{user.name}</header>;
}

async function StatsCards() {
  const stats = await fetchStats();  // 800ms
  return <Cards data={stats} />;
}

async function RealtimeChart() {
  const data = await fetchChartData();  // 3000ms
  return <Chart data={data} />;
}

用户感知 timeline：

  • 0ms：HTML 到达，看到导航框架（无文字）+ 两个骨架
  • 100ms：NavBar 文字填充，"已加载" 感觉
  • 800ms：StatsCards 填充
  • 3000ms：Chart 填充，整页完成

场景 2 · 修复"为什么看起来很慢"

典型用户反馈："dashboard 加载要 3 秒，太慢"。但其实可以做到"用户 100ms 就感知到加载已开始 + 进度可见"。

排查步骤：

  1. 开 DevTools Network → Slow 3G 模拟
  2. 看 timeline：HTML 何时第一字节、何时 LCP（最大内容绘制）、何时 TTI（可交互）
  3. 如果 HTML 几秒后才到 → 流式没启用 / Suspense 边界设错（包了所有内容）
  4. 如果 HTML 立即到但 LCP 慢 → 大组件应该独立 Suspense + 骨架
  5. 如果 TTI 慢 → client bundle 大，audit hydration（详见 P3.1）

场景 3 · 错误恢复 + Server Action 配合

用户提交表单失败，error.tsx 触发，提供 retry 按钮：

tsx 
// app/checkout/error.tsx
'use client';
export default function CheckoutError({ error, reset }: any) {
  return (
    <div>
      <h2>支付失败</h2>
      <p>Error: {error.digest}</p>
      <button onClick={reset}>重新提交</button>
      <a href="/cart">返回购物车</a>
    </div>
  );
}

reset 让 React 重新尝试渲染该段。如果是临时网络问题，reset 可能成功；持久问题（卡号无效）则需要走"返回购物车"。

04 典型代码

代码 1 · waterfall 反例 → 修复

tsx 
// ❌ waterfall：800 + 600 + 400 = 1800ms
async function ProductPage({ params }) {
  const { id } = await params;
  const product = await fetchProduct(id);          // 800ms
  const reviews = await fetchReviews(product.id);  // 600ms（需要 product.id）
  const seller = await fetchSeller(product.sellerId); // 400ms（需要 product.sellerId）
  return <Detail {...{product, reviews, seller}} />;
}

// ✅ 修复：800 + max(600, 400) = 1400ms
async function ProductPage({ params }) {
  const { id } = await params;
  const product = await fetchProduct(id);          // 800ms
  const [reviews, seller] = await Promise.all([    // 并行 600ms 和 400ms
    fetchReviews(product.id),
    fetchSeller(product.sellerId),
  ]);
  return <Detail {...{product, reviews, seller}} />;
}

// ✅ 进一步：流式让用户先看到 product，reviews/seller 后面 reveal
async function ProductPage({ params }) {
  const { id } = await params;
  const product = await fetchProduct(id);  // 800ms（必须，因为下面要用）
  const reviewsPromise = fetchReviews(product.id);  // 触发不等
  const sellerPromise = fetchSeller(product.sellerId);
  return (
    <>
      <ProductDetail product={product} />
      <Suspense fallback={<ReviewsSkeleton />}>
        <Reviews promise={reviewsPromise} />
      </Suspense>
      <Suspense fallback={<SellerSkeleton />}>
        <Seller promise={sellerPromise} />
      </Suspense>
    </>
  );
}

代码 2 · 用 React 19 use API 让 client 读 Promise

tsx 
// app/page.tsx (RSC)
import { Suspense } from 'react';
import LikeButton from './_components/LikeButton';

export default function Page() {
  // server 触发 fetch，不 await
  const initialLikesPromise = fetchLikes('post-1');
  return (
    <>
      <article>...</article>
      <Suspense fallback={<p>Loading likes...</p>}>
        <LikeButton initialLikesPromise={initialLikesPromise} />
      </Suspense>
    </>
  );
}

// app/_components/LikeButton.tsx
'use client';
import { use, useState } from 'react';

export default function LikeButton({
  initialLikesPromise,
}: {
  initialLikesPromise: Promise<number>;
}) {
  // use() 在 Suspense 包裹下读 Promise
  const initialLikes = use(initialLikesPromise);
  const [likes, setLikes] = useState(initialLikes);
  return <button onClick={() => setLikes(l => l + 1)}>❤️ {likes}</button>;
}

代码 3 · template.tsx 触发页面切换 animation

tsx 
// app/(app)/template.tsx
'use client';
import { motion } from 'framer-motion';
import { usePathname } from 'next/navigation';

export default function Template({ children }: { children: React.ReactNode }) {
  const pathname = usePathname();
  return (
    <motion.div
      key={pathname}        // 路径变化时 key 变，强制重新挂载
      initial={{ opacity: 0, y: 16 }}
      animate={{ opacity: 1, y: 0 }}
      transition={{ duration: 0.3 }}
    >
      {children}
    </motion.div>
  );
}

代码 4 · loading.tsx 的有意义骨架

tsx 
// app/posts/loading.tsx
export default function PostsLoading() {
  return (
    <ul className="space-y-4">
      {Array.from({ length: 5 }).map((_, i) => (
        <li key={i} className="bg-gray-200 rounded h-24 animate-pulse" />
      ))}
    </ul>
  );
}
// 关键：骨架的视觉结构应该接近真实内容
// ❌ 一个旋转图标 / 空 div
// ✅ 5 个矩形条对应将出现的 5 个 post 项

05 常见陷阱

  • 整个 page 包在一个 Suspense / loading.tsx —— 流式效果失效，用户必须等最慢的才看到。每个慢段独立 Suspense
  • "流式启用了但 Network 看 response 仍然一次性返回" —— 检查 server / 反向代理 / CDN 是否禁用了 streaming（buffering）。Nginx / Cloudflare 默认行为可能 buffer chunked response，要显式配置
  • fetch waterfall —— 串行 await 父子 fetch；用 Promise.all 并行 / 提前触发 / 流式 Suspense 三种方式之一修
  • error.tsx 不工作，错误冒到 global-error —— 检查 error.tsx 是否标 'use client'；检查错误是否在 layout 而不是 page（layout 错误 error.tsx 不能 catch）
  • 骨架屏视觉与真实内容差距大，造成 layout shift —— 骨架要与真实内容尺寸匹配，否则用户看到"骨架 → 内容"突然挤压。CLS（Cumulative Layout Shift）会很差
  • 用 layout 想做 enter animation 但不触发 —— layout 不重新挂载。要用 template + 动态 key（如 pathname）触发 animation
  • "Selective Hydration 听起来很神奇但我没感觉到" —— 它在快网络下不易感知。慢网络（Slow 3G） + 大 bundle 才能看出"FastWidget 已可点而 SlowWidget 还在 hydrate"
  • 给 Suspense fallback 一个会 fetch 的组件 —— fallback 必须立即可渲染，不能 await。否则 fallback 自己也触发 Suspense，死循环
  • stream 中间错误处理不当 —— 如果 server 在 flush 后某段抛错，已 flushed 的内容不能撤回。HTML 已经走到浏览器了。Next.js 用 inline JS 在出错位置替换为 error fallback，但这要求 error.tsx 存在
  • "我用了 Suspense 但没看到流式" —— 检查内部组件是否真的 await。如果 Suspense 内部所有组件都是同步渲染（无 await fetch），Suspense 就不会触发流式 boundary
  • 把 useEffect fetch 当流式 —— useEffect fetch 是 client-side fetch，在 hydrate 后才触发；不是 Suspense 流式。流式必须用 Server Component await 或 use() + Promise prop
  • error.digest 在 dev 显示完整 stack，prod 只显示 hash —— 这是设计行为（防 secret 泄露）。用 digest 在 server log 里搜对应错误详情
