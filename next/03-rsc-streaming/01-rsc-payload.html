RSC payload 协议与 flush

RSC（React Server Components）不是"另一种 SSR"，是 React 团队定的组件树序列化协议 —— 把组件树编码成行式的 wire format，server 边渲染边推、client 边收边构建。这一章把 RSC 神话祛魅 —— 抓一段真实 RSC payload 看它长啥样、SSR 阶段 HTML + RSC 怎么双流、客户端导航的 payload-only 路径、Suspense flush 机制、Client Component 引用怎么从 manifest 解析。看完你不再觉得 RSC 是"魔法"，它就是个协议 + 几个工程组件。

阅读时长 ~30 分钟 前置 P2.2 Server/Client 边界 + React 19 基础 主要来源 React RFC #188 + react-server-dom-webpack 源码 + Next.js renderer 源码 + Sebastian Markbåge "Tour of RSC" 演讲

01 速览

  1. RSC payload 不是 HTML 也不是 JSON，是 React 团队定的行式序列化格式。每行格式 {id}:{type}{data}，可以增量解析。本质是把"已经渲染过的组件树"打包成 wire 格式让 client 重建。
  2. "渲染过"在 RSC 里的含义特殊 —— Server Component 直接执行，输出"就绪的 React element 树（含 props 已 resolve）"；Client Component 不执行，只输出"引用 + props"作为占位（client 拿到后才真正渲染）。
  3. 首次访问时（SSR），server 同时输出两条流：HTML 流（给搜索引擎 + 首屏即可见）+ RSC payload 流（给 hydration + 后续导航对账）。两条流通过 <script>self.__next_f.push(...)</script> 在同一个 HTML 响应里穿插发送。
  4. 客户端导航时（点 <Link> 跳转），server 只返回 RSC payload 一条流，不返回 HTML。Router 用新 payload 局部更新 React 树，无整页刷新。
  5. Suspense flush 是 RSC 流式的核心机制 —— 每个 Suspense boundary 就绪（内部 await 都 resolve）时，server 立即推一段 chunk 到 stream。这就是"边渲染边发"的本质。
  6. Client Component 在 payload 里只有"引用"（一个 module ID + chunk 路径 + export 名），不含真实代码。Client 加载时通过 webpack/Turbopack 生成的 manifest 把引用解析成实际 JS 模块。
  7. RSC 的核心收益：Server Component 代码不下发到 client bundle。一个 page 用 50 个 npm 包（markdown 渲染、图表库、ORM）但只有几 KB JS 下发到浏览器 —— 因为这些包都在 server 跑，不进 client bundle。
  8. 与传统 SSR 区别：传统 SSR server 渲染整棵 HTML → client hydrate 整棵；RSC server 渲染"server 部分 + client 占位"→ client 只 hydrate client 占位部分。
  9. RSC payload 不向用户暴露，所以协议格式细节随版本演化。但稳定的概念边界（Server vs Client、序列化规则、Suspense flush）写代码时关心；payload 字节细节只在调试 / 优化时关心。
  10. 看这章的实战收益：调试 hydration mismatch / 知道 fetch waterfall 怎么发生 / 理解 Network 面板里那条 RSC 请求的内容 / 设计 Suspense 边界时知道每段什么时候 reveal。

02 底层逻辑

RSC payload 长什么样

抓一个真实 Next.js 16 RSC payload（用 DevTools Network 面板筛 ?_rsc=*）。简化展示：

text 
0:["$","html",null,{"lang":"zh-CN","children":["$","body",null,{"children":["$L1",null,{}]}]}]
1:I["6789","./chunks/page.js","Page"]
2:["$","main",null,{"className":"container","children":[
  ["$","h1",null,{"children":"My Posts"}],
  ["$","$L1",null,{"posts":[{"id":"1","title":"Hello"},{"id":"2","title":"World"}]}]
]}]

这串文本里几个核心概念：

  • 每行是一个 entry —— 格式 {id}:{data}，id 用来交叉引用。第 0 行是入口（root）。
  • ["$","tagName",key,props] —— 表示一个 React element。第一项 "$" 是 marker（避免与普通字符串冲突），第二项是 type（HTML tag 或 component 引用），第三项是 key，第四项是 props。
  • I["6789","./chunks/page.js","Page"] —— Client Component 的引用声明（I = Import）。意思是 ID 1 对应 chunks/page.js 里的 Page export，module hash 是 6789。这是给 client 看的 manifest 索引。
  • "$L1" —— 引用 ID 1 的 lazy load 标记。当 client 解析到这个标记时，知道要异步加载 ID 1 对应的 client component（即 chunks/page.js 里的 Page）。
  • props 直接嵌入 —— {posts: [{id:"1",...}]} 是 server-side 已经 fetch 好的数据，作为 props 序列化进 payload。

用人话翻译：这段 payload 在告诉 client："根是个 html，里面 body 里有个 ID 1 的 client 组件（要从 chunks/page.js 加载）；这个 client 组件的 props 是 {posts: [...]}。在它外面包一个 <main> 容器、里面塞一个 h1 'My Posts' 和这个 client component 实例。"

SSR 阶段 · HTML + RSC 双流

用户首次访问（无论是直接输入 URL 还是搜索引擎打开）。server 做这些事：

  1. 路由匹配 → 找到对应 page.tsx + layouts
  2. 从外层 layout 开始，一路向内执行 RSC（直接 await fetch / db）
  3. 遇到 Client Component → 不执行，只在 payload 写一个引用
  4. 遇到 Suspense boundary 内部还在 pending → 先输出 fallback、把内部跳过
  5. 同时做两件事：用 react-dom/server 把树渲染成 HTML（给浏览器首屏 + SEO）+ 把树序列化成 RSC payload（给 hydration / 后续对账）
  6. 两条流穿插写入 HTTP response：HTML 直接写、RSC payload 通过 <script>self.__next_f.push([0,"chunk"])</script> 一段段写
  7. 每当某个 Suspense 内部 ready，server flush 一段（HTML <template> + 对应的 RSC chunk + 一段把 template 替换到 fallback 位置的 inline JS）

看真实 HTML 输出（简化）：

html 
<!DOCTYPE html>
<html lang="zh-CN">
<head>...</head>
<body>
  <main>
    <h1>My Posts</h1>
    <!-- 这里是 SSR 渲染好的 HTML，首屏即可见 -->
    <ul><li>Hello</li><li>World</li></ul>
  </main>

  <!-- 然后是 RSC payload chunks -->
  <script>self.__next_f=self.__next_f||[];self.__next_f.push([0,"0:[\"$\",\"html\",null,..."])</script>
  <script>self.__next_f.push([0,"1:I[\"6789\",\"./chunks/page.js\",\"Page\"]"])</script>
  <script>self.__next_f.push([0,"2:[\"$\",\"main\",..."])</script>

  <!-- 最后加载 client bundle，开始 hydration -->
  <script src="/_next/static/chunks/main.js"></script>
</body>
</html>

关键洞察：HTML 与 RSC payload 在同一个 HTTP response 里，HTML 给浏览器渲染首屏，RSC payload 给 React hydration + 后续 Router 状态。这就是为什么打开 Network 面板看 SSR 请求时 response 既包含 markup 又包含一堆 __next_f.push 脚本。

客户端导航 · 只发 RSC payload

用户进入站点后点击 <Link href="/posts/123">。这次不会触发整页加载，而是：

  1. Link 组件向 server 发请求 GET /posts/123?_rsc=...，accept 头标 RSC
  2. server 走和首次访问相同的渲染逻辑，但只输出 RSC payload，不渲染 HTML
  3. client Router 收到 payload，解析、对比当前树、局部更新
  4. 没有 hydration mismatch 风险（client 已经 hydrated）

看这次请求的 response（简化）：

text 
0:["$","main",null,{"children":[
  ["$","h1",null,{"children":"Post 123"}],
  ["$","$L1",null,{"id":"123"}]
]}]
1:I["6790","./chunks/post-detail.js","PostDetail"]

没有 <html> 包裹，没有 HTML 标签，纯粹是树的 wire format。Router 解析后知道："要把 main 这棵替换到 layout 的 children 位置"。

这就是为什么 Next.js 客户端导航很快 —— 不需要重新加载 HTML / CSS / JS，只是一次小 RSC payload + React 局部更新。

Suspense flush · 边渲染边发的本质

如果 page 内有 Suspense，比如：

tsx 
// app/dashboard/page.tsx (RSC)
export default function Dashboard() {
  return (
    <>
      <FastHeader />                                {/* 100ms */}
      <Suspense fallback={<Skeleton />}>
        <SlowChart />                                {/* 3000ms */}
      </Suspense>
      <Suspense fallback={<Skeleton />}>
        <MediumStats />                              {/* 800ms */}
      </Suspense>
    </>
  );
}

Server 渲染流程：

  1. t=0ms：开始渲染。FastHeader 已 ready，两个 Suspense 内部还在 pending。
  2. t=100ms：FastHeader 完成。Server 输出 HTML：FastHeader markup + 两个 fallback 占位。同时输出 RSC payload chunk（root tree + fallback 引用）。flush 一次。
  3. 用户立即看到：FastHeader + 两个骨架屏。
  4. t=800ms：MediumStats ready。Server 输出 <template> 包含 MediumStats markup + 一段 inline JS 把 template 替换到对应 fallback 位置 + 对应 RSC chunk。flush 一次。
  5. 用户看到：FastHeader + MediumStats + SlowChart 骨架（仍在等）。
  6. t=3000ms：SlowChart ready。同样的 flush 流程。
  7. 用户看到：完整页面。Stream close。

关键：每次 flush 都是真实的 HTTP chunk（不是等全部完成才发）。用户的浏览器在 t=100ms 就能看到首屏，3000ms 时整页完成。这就是流式渲染最大价值 —— TTFB 极小，FCP 接近 TTFB，LCP 由最慢的 Suspense 决定但不阻塞其他部分显示。

底层机制：React 的 renderToReadableStream（Node 用 renderToPipeableStream）把每个 Suspense 看作一个独立的"渲染任务"，等内部所有 await 完成后 emit 一段。Next.js renderer 把每段串成 HTTP stream。

Client Component 引用 · webpack/Turbopack manifest

RSC payload 里的 I["6789","./chunks/page.js","Page"] 是怎么解析成真实模块的？

build 时，bundler（Webpack 或 Turbopack）做两件特殊的事：

  1. 识别所有 'use client' 标记的模块 → 单独打包到 client chunk
  2. 生成一份 RSC manifest，把每个 client module 的 ID 映射到 chunk 路径 + export 名

manifest 大概长这样：

json 
{
  "6789": {
    "id": "6789",
    "name": "Page",
    "chunks": ["./chunks/page.js", "./chunks/page.css"],
    "async": false
  },
  "6790": {
    "id": "6790",
    "name": "PostDetail",
    "chunks": ["./chunks/post-detail.js"],
    "async": false
  }
}

Server 渲染时，遇到 client component 引用，从 manifest 拿到 ID + chunk 路径，写进 payload 的 I[...] 行。Client 解析 payload 时，遇到 I[id, path, name] 就动态 import 对应 chunk，拿到组件后用 props 渲染。

这就是为什么 RSC 必须配合特定 bundler —— 协议本身是 React 标准，但 ID/chunk 解析依赖 bundler manifest。Webpack 和 Turbopack 都实现了；其他 bundler（Vite Rolldown / esbuild / Bun build）需要自己实现 RSC plugin（这是为什么 RR v7 / TanStack Start 加 RSC 支持要花时间）。

RSC 与传统 SSR 的本质区别

看一个对比表：

维度 传统 SSR（Pages Router / RR v7 不押 RSC） RSC（Next.js App Router）
server 输出 HTML + 一份 props JSON（给 hydration） HTML + RSC payload（流式）
client bundle 含 所有 page 组件代码（用于 hydration） 只含 client component 代码（RSC 代码不下发）
hydration 范围 整棵 page 树 只 client component（RSC 不参与）
数据获取 page 层（getStaticProps 等） 组件层（每个 RSC 直接 await）
包体积 大（含数据获取库 / SDK） 小（这些都在 server 跑）
客户端导航 fetch JSON props + client 重新 render fetch RSC payload + 局部更新树

RSC 真正的价值不在"server 渲染 HTML"（传统 SSR 早就能做），而在"server 渲染的代码不下发到 client"。这让你可以在组件里直接 import 200KB 的 ORM 库 / 100KB 的 Markdown 渲染器，而 client bundle 只含浏览器必需的部分。

03 应用场景

场景 1 · 调试"为什么 client bundle 这么大"

RSC 主要价值之一是减小 client bundle。但常见 bug 是不小心把整棵子树标 client。调试方法：

  • 跑 next build，看输出的 bundle 体积报告
  • 对每个大 chunk，用 @next/bundle-analyzer 看里面有什么
  • 如果发现 ORM / 服务端 SDK 进了 client bundle → 找出是哪个 'use client' 模块 import 了它
  • 修复方式：把 'use client' 推到更下层（leaf 组件）；或用 children 模式让 RSC 包含 server 内容

场景 2 · 解释为什么客户端导航这么快

同事问"为什么 Next.js 跳转比 SPA 还快"，答案是 RSC payload：

  • 不重新加载 HTML / CSS（已在 client）
  • 不重新加载 JS bundle（已在 client）
  • 只 fetch 一次小的 RSC payload（通常 1-50KB）
  • server 端渲染 + 数据获取一并完成，client 不用再发多次 fetch

场景 3 · 设计 Suspense 边界让流式效果最大化

原则：把"快的"放在 Suspense 外面、"慢的"放在 Suspense 里面，每个慢段独立 Suspense。

tsx 
// ❌ 反例：所有内容包在一个 Suspense，必须等最慢的才显示
<Suspense fallback={<Skeleton />}>
  <FastHeader />
  <SlowChart />
  <MediumStats />
</Suspense>

// ✅ 正例：每个慢段独立 Suspense，流式 reveal
<FastHeader />                                          {/* 立即显示 */}
<Suspense fallback={<ChartSkeleton />}>
  <SlowChart />                                          {/* 慢，独立 */}
</Suspense>
<Suspense fallback={<StatsSkeleton />}>
  <MediumStats />                                        {/* 中速，独立 */}
</Suspense>

代码 3 · Suspense 流式效果对比

tsx 
// app/dashboard/page.tsx
import { Suspense } from 'react';

async function FastHeader() {
  // 100ms
  const user = await getCurrentUser();
  return <header>Hi, {user.name}</header>;
}

async function MediumStats() {
  // 800ms
  await new Promise(r => setTimeout(r, 800));
  const stats = await fetchStats();
  return <StatsCards data={stats} />;
}

async function SlowChart() {
  // 3000ms - 复杂的 BI 查询
  await new Promise(r => setTimeout(r, 3000));
  const data = await fetchChartData();
  return <Chart data={data} />;
}

export default function Dashboard() {
  return (
    <div>
      <Suspense fallback={<HeaderSkeleton />}>
        <FastHeader />
      </Suspense>
      <Suspense fallback={<StatsSkeleton />}>
        <MediumStats />
      </Suspense>
      <Suspense fallback={<ChartSkeleton />}>
        <SlowChart />
      </Suspense>
    </div>
  );
}

// 浏览器看到的时间线：
// 0ms      → 三个骨架屏
// 100ms    → FastHeader 显示 + Stats 骨架 + Chart 骨架
// 800ms    → FastHeader + MediumStats 显示 + Chart 骨架
// 3000ms   → 全部显示

代码 4 · 验证 RSC 不下发 server 代码

tsx 
// app/posts/page.tsx (RSC)
import { renderMarkdown } from '@some/heavy-markdown-lib';  // 假设 200KB
import { db } from '@/lib/db';

export default async function Posts() {
  const posts = await db.posts.findMany();
  return (
    <ul>
      {posts.map(p => (
        <li key={p.id} dangerouslySetInnerHTML={{ __html: renderMarkdown(p.content) }} />
      ))}
    </ul>
  );
}

// 跑 next build 后看 client bundle：
// - Posts 组件代码不在 client bundle
// - @some/heavy-markdown-lib 不在 client bundle
// - db 不在 client bundle
// - Client 收到的是已经渲染好的 HTML + RSC payload，不需要再跑 markdown 渲染

05 常见陷阱

  • 认为"RSC payload 是 JSON" —— 它不是。是 React 团队定的特殊行式格式。直接 JSON.parse 解析失败。如果想程序化处理，用 react-server-dom-webpack/server-edge 提供的 API
  • 把整个 page 包在一个 Suspense 里 —— 流式效果失效，必须等最慢的才显示。每个慢段独立 Suspense
  • "我的 page 没有 Suspense，应该不流式吧" —— Next.js 会隐式给 page 加 root Suspense（loading.tsx）。但效果有限 —— 整个 page ready 才 reveal。要细粒度流式必须显式加 Suspense
  • Server Component 里 setState / useEffect —— 报错。RSC 没有 state 概念。把交互部分提到 client component
  • Client Component 里 await db —— 编译可能不报错，运行时崩。所有 db / secret 模块加 'server-only'
  • RSC payload 比预期大 —— 检查传给 client component 的 props，看是否传了过多数据。RSC payload 里 props 是完整序列化的，传一个 10MB 的对象就 10MB 进 payload
  • 客户端导航变慢，Network 看 RSC 请求 5 秒 —— 通常是 Server Component 内 await 的接口慢。用 Suspense 拆分让快的部分先 reveal
  • "为什么我直接打开 RSC URL 看到的是文本而不是页面" —— RSC 端点返回 wire format，不是 HTML。直接打开看到的是协议文本（"$","html"...）。这就是它的真实样子
  • 用错 fetch 选项导致 RSC 缓存意外 —— v15+ fetch 默认 dynamic，但 RSC payload 本身可能被 Next.js 缓存层缓存。详见 P5
  • 把 RSC 当成"远程 RPC 调用" —— RSC 是组件树序列化，不是 RPC。要 RPC 用 Server Actions（详见 P4），不要把 RSC 当 GraphQL 用
  • 调试 hydration mismatch 发现 server / client 渲染不同 —— 通常是 Date.now() / Math.random() / typeof window 检查没处理好。用 useEffect 让 client-only 内容在 hydration 后才出现
  • RSC 协议字节细节背下来然后写代码 —— 没必要。协议格式随版本变（特别是 React 19 → 20 之间）。稳定的概念边界（Server vs Client、序列化、Suspense）才是要建立的心智
