五层缓存模型 → Cache Components 重构

Next.js 缓存是社区最被骂的部分，没有之一。v13 引入五层缓存（Request Memoization / Data Cache / Full Route Cache / Router Cache / CDN）+ 默认 cache-by-default，结果在生产引爆大量 stale 数据事故；v14 PPR 实验出现；v15 把默认彻底反转为 explicit-opt-in，承认是设计错误；v16 用 Cache Components（'use cache' 指令 + cacheComponents 配置）把整个心智重做一遍，简化为"静态壳 + 动态洞"两层。这一章拉直 4 年演化、解释每层的真实作用、为什么默认值反转、为什么 v16 这么做。读完 P5.1 + P5.2，能在生产代码里独立判断"这段该不该 cache、用哪个 revalidate"。

阅读时长 ~35 分钟（最难章） 前置 P3 RSC + P4 Server Actions 主要来源 nextjs.org/docs caching + Vercel "Partial Prerendering" / "Cache Components" 工程 blog + v15 / v16 release notes + Tim Neutkens 公开演讲

01 速览

  1. Next.js v13-15 缓存模型有五层，由内向外：Request Memoization（请求内 fetch 去重）→ Data Cache（跨请求 fetch 持久缓存）→ Full Route Cache（整个 route 的渲染结果缓存）→ Router Cache（client RSC payload 缓存）→ CDN/浏览器（最外层）。
  2. v13-14 默认 cache-by-default：fetch() 不写选项就被 Data Cache 永久缓存；用了 cookies/headers 自动 dynamic（整个 route 不缓存）。这套规则隐式且不直观，引爆大量"为什么数据是 1 小时前的"事故。
  3. v15 把默认反转为 explicit-opt-in：fetch 不写选项 = dynamic（每请求）；要 cache 必须显式 { cache: 'force-cache' } 或 { next: { revalidate: N } }。这是承认 v13-14 是设计错误的迟到修正。
  4. v15 同时引入 async request APIs（cookies/headers/params/searchParams 全 Promise）—— 这是为 PPR 铺路：让"读 cookies"成为显式 await，PPR 才能在静态壳渲染时跳过。
  5. v16 引入 Cache Components，用 'use cache' 指令显式标记可缓存单元，替代 v13-15 的隐式五层叠加。cacheComponents: true 启用后整个 app 走"静态壳 + 动态洞"渲染（PPR 默认）。
  6. 新心智：每个组件要么是静态壳（用 'use cache' 包，CDN 秒返）要么是动态洞（在 Suspense 内、读 cookies/headers 等 runtime API，每请求重算）。两类组件混合在一棵树里，不是"全静态 vs 全动态"的二选一。
  7. v16 同时新增 updateTag() —— 失效 + 当前请求内立即重算，解决"提交表单后立即看到新数据"的体验问题（详见 P4）。这是 revalidatePath / revalidateTag 之外的第三种工具。
  8. 从 v15 → v16 的迁移：把 unstable_cache 改 'use cache'；把 dynamicIO: true 改 cacheComponents: true；把 page 拆成"static 部分 + Suspense 包的 dynamic 部分"。官方文档有 codemod 帮忙。
  9. 新心智的优势：缓存策略显式可读（看到 'use cache' 就知道这段是 cacheable）；不再隐式触发 dynamic（不会因为某个组件读 cookies 整个 route 失效）；与 PPR 自然契合（静态壳就是 'use cache' 包的部分）。
  10. v16 不是终点。Cache Components 还在迭代，'use cache' 的cache key 推断 / 失效粒度 / multi-tenant 隔离等问题仍在打磨。但相比 v13-14 的隐式五层叠加，v16 心智简化了一个数量级。

02 底层逻辑

v13-15 五层缓存模型（理解历史用）

把每层独立讲清楚。即使在 v16 你也需要理解这个模型 —— 因为"为什么 v16 这么改"必须先看 v13-15 是怎么错的。

第 1 层 · Request Memoization

同一个 request 内，相同的 fetch URL + 选项被去重。基于 React cache() API：

tsx 
// 同一个 request 处理过程中，下面三次 fetch 实际只触发一次
async function getUser(id: string) {
  return fetch(`/api/users/${id}`).then(r => r.json());
}

// app/page.tsx
async function Header() {
  const user = await getUser('123');  // 触发真实 fetch
}
async function Sidebar() {
  const user = await getUser('123');  // hits memoization，复用上次结果
}
async function Footer() {
  const user = await getUser('123');  // hits memoization
}

这一层基本没有争议，所有版本都启用。它是 React 的能力，不是 Next.js 独有。范围：单个 request。请求结束就清空。

第 2 层 · Data Cache

跨请求持久化的 fetch 缓存。存储在文件系统（self-host）或 Vercel 的分布式 KV。这是引爆事故的层：

tsx 
// v13-14 默认行为：被 Data Cache 永久缓存（没有失效）
await fetch('/api/products');

// 想缓存 60 秒：
await fetch('/api/products', { next: { revalidate: 60 } });

// 想强制每请求重新拿（dynamic）：
await fetch('/api/products', { cache: 'no-store' });

// v15+ 默认反转：上面第一个 = no-store，第二个仍然是 60 秒缓存
await fetch('/api/products');  // 现在等价于 cache: 'no-store'

v13-14 的设计错误：默认 = 'force-cache'（永久缓存）。开发者写 fetch('/api/data') 完全没意识到自己在缓存。常见事故场景：

  • 商品价格改了，用户看到的是 1 小时前的
  • 用户提交表单后页面没刷新（其实已写入但 cache 命中旧数据）
  • 不同环境（dev / preview / prod）行为不一致（dev 默认 no-cache，prod 启用）

v15 反转后，默认 = 'no-store'。开发者要 cache 必须显式标。

第 3 层 · Full Route Cache

整个 route 渲染结果（HTML + RSC payload）的缓存。决定 page 是 static 还是 dynamic：

  • 所有 fetch 都是 cached → page 是 static，build 时预渲染，运行时从 CDN 返回
  • 有任何 fetch 是 dynamic / 用了 cookies/headers/searchParams → page 是 dynamic，每请求渲染

v13-14 的隐式触发规则：使用 cookies() / headers() / searchParams / noStore() 任何一个，整个 route 立即变成 dynamic。这导致：

  • 开发者在 layout 里读了 cookies 检查 auth → 整个 app 全部 dynamic（Full Route Cache 失效）
  • 开发者后来才知道这事，但已经写了大量代码
  • 没法做到"layout 静态、page 动态"或反过来

v16 用 PPR + Cache Components 解决：每个组件独立标 'use cache'，整个 route 不再是"全 static 或全 dynamic"。

第 4 层 · Router Cache（client）

客户端 Router 缓存最近访问过的 RSC payload，让"前进/后退"导航瞬时返回：

  • v13-14：默认 30 秒 client cache，dynamic page 也缓存（这导致用户看到旧数据）
  • v14.1 起把 dynamic page 客户端 cache 默认时间改为 0
  • v15 起 prefetched 但未访问的内容默认 cache 5 分钟、访问过的 dynamic page 默认 0、static page 默认 5 分钟
  • v16 仍保留这层但默认值更保守

这层的争议：用户在 dashboard 修改了数据，按"返回"按钮看到旧版本（client cache 命中）。修复需要 router.refresh() 或 revalidatePath。

第 5 层 · CDN / 浏览器

HTTP 标准的 Cache-Control 头管控的最外层缓存。Next.js 自动给 static page 设置正确的 Cache-Control，dynamic page 设置 no-store。这层是 HTTP 标准，所有框架都有，不是 Next.js 独有的复杂度。

v15 反转的具体内容

v15 的"缓存反转" release notes 列出 4 个具体变化：

  1. fetch 默认：force-cache → no-store。fetch('/api/x') 现在每请求重新拿
  2. Route Handlers GET 默认：cached → not cached。GET handler 不再默认缓存
  3. Client Router Cache · dynamic page：30s → 0s。dynamic page 不再 client cache
  4. async request APIs：cookies/headers/params/searchParams 全 Promise 化

从 v14 升 v15 是破坏性变更，必须做的迁移：

diff 
// 1. fetch 默认变化
- await fetch('/api/data');                           // v14: cached forever
+ await fetch('/api/data', { cache: 'force-cache' }); // v15+ 必须显式
+ // 或 { next: { revalidate: 60 } }

// 2. async params
- export default function Page({ params }: { params: { id: string } }) {
-   return <div>{params.id}</div>;
- }
+ export default async function Page({ params }: { params: Promise<{ id: string }> }) {
+   const { id } = await params;
+   return <div>{id}</div>;
+ }

// 3. async cookies / headers
- import { cookies } from 'next/headers';
- const c = cookies();
+ const c = await cookies();

v15 提供 codemod 自动改大部分代码。但语义改变（特别是默认 cache）需要逐个 fetch 审查。

v16 Cache Components · 整体重构

v16 不是 v15 的小补丁，是把缓存模型重新设计。开启需要：

typescript 
// next.config.ts
export default {
  cacheComponents: true,  // 启用 Cache Components（替代 v15 的 dynamicIO: true 实验配置）
};

启用后：

  • 所有 page 默认走 PPR（Partial Prerendering）—— "静态壳 + 动态洞"渲染
  • 'use cache' 指令替代 unstable_cache
  • 动态部分必须在 Suspense 内 —— 编译时报错提示你"这段在静态壳里读 cookies/headers，必须包 Suspense"
  • 新增 updateTag()（详见 P4 + P5.2）

核心心智转变：

v13-15 五层心智 v16 Cache Components 心智
隐式：fetch 默认行为决定 cache 显式：'use cache' 标记 cacheable 单元
读 cookies → 整个 route dynamic 读 cookies → 仅当前 Suspense 内动态
"全 static 或全 dynamic"二选一 "静态壳 + 动态洞"混合
Data Cache 是 fetch 级别 'use cache' 是 component / function 级别
需要熟记 5 层各自规则 只需要决定每段是 cache 还是 dynamic

看一个完整 v16 例子：

tsx 
// app/blog/[slug]/page.tsx (Next.js 16 + cacheComponents)
import { Suspense } from 'react';

async function PostContent({ slug }: { slug: string }) {
  'use cache';  // ← 这段是静态壳，CDN 秒返
  const post = await db.posts.findUnique({ where: { slug } });
  return <article>{post.content}</article>;
}

async function Comments({ slug }: { slug: string }) {
  // 没有 'use cache' = 动态，每请求重算
  const comments = await db.comments.findMany({ where: { postSlug: slug } });
  return <ul>{comments.map(c => <li key={c.id}>{c.body}</li>)}</ul>;
}

async function CurrentUser() {
  // 读 cookies = 必须动态
  const session = await getSession();  // 内部读 cookies
  return session ? <p>Hi, {session.name}</p> : null;
}

export default async function PostPage({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
  return (
    <>
      <Suspense fallback={<p />}>
        <CurrentUser />                                 {/* 动态洞 */}
      </Suspense>
      <PostContent slug={slug} />                       {/* 静态壳 */}
      <Suspense fallback={<p>Loading comments...</p>}>
        <Comments slug={slug} />                        {/* 动态洞 */}
      </Suspense>
    </>
  );
}

渲染流程：

  1. build 时预生成 PostContent（'use cache' + slug 作为 key）的 HTML
  2. 用户访问 → CDN 立即返回 PostContent 的 HTML 壳 + Suspense 占位
  3. server 流式 stream Comments + CurrentUser（每请求新计算）
  4. 用户先看到 article（CDN 返回，~10ms），评论 + 用户问候稍后填充（~500-1000ms）

对比 v13-15：那时候要么整个 page 是 SSG（评论也被静态化，刷新看不到新评论），要么整个 page 是 SSR（每次重新查 article + 评论 + auth，慢）。v16 让两全其美成为可能。

'use cache' 详细机制

'use cache' 可以放在不同级别：

函数级（最常用）：

tsx 
async function getProduct(id: string) {
  'use cache';
  return db.products.findUnique({ where: { id } });
}
// cache key 自动从参数推断（id）

组件级：

tsx 
async function ProductCard({ id }: { id: string }) {
  'use cache';
  const product = await db.products.findUnique({ where: { id } });
  return <div>{product.name}</div>;
}
// 整个组件渲染结果按 props 缓存

文件级：

tsx 
// lib/products.ts
'use cache';
// 这个文件 export 的所有函数 / 组件都是 cacheable
export async function getProduct(id: string) { /* ... */ }
export async function getProductReviews(id: string) { /* ... */ }

cache key 由参数自动推断（必须是可序列化的类型）。控制 cache 寿命用 cacheLife：

tsx 
import { unstable_cacheLife as cacheLife } from 'next/cache';

async function getProduct(id: string) {
  'use cache';
  cacheLife('hours');  // 'seconds' | 'minutes' | 'hours' | 'days' | 'weeks' | 'max'
  return db.products.findUnique({ where: { id } });
}

用 tag 让 revalidate 精准失效：

tsx 
import { unstable_cacheTag as cacheTag } from 'next/cache';

async function getProduct(id: string) {
  'use cache';
  cacheTag(`product-${id}`);
  return db.products.findUnique({ where: { id } });
}

// 在 Server Action 里失效
'use server';
import { revalidateTag } from 'next/cache';
export async function updateProduct(id: string, data: any) {
  await db.products.update({ where: { id }, data });
  revalidateTag(`product-${id}`);
}

从 v15 unstable_cache 迁到 v16 'use cache'

diff 
// v15 写法
-import { unstable_cache } from 'next/cache';
-
-const getProduct = unstable_cache(
-  async (id: string) => db.products.findUnique({ where: { id } }),
-  ['products'],
-  { revalidate: 3600, tags: (id) => [`product-${id}`] }
-);

// v16 写法
+import { unstable_cacheLife as cacheLife, unstable_cacheTag as cacheTag } from 'next/cache';
+
+async function getProduct(id: string) {
+  'use cache';
+  cacheLife('hours');
+  cacheTag(`product-${id}`);
+  return db.products.findUnique({ where: { id } });
+}

v16 写法的优势：

  • 不需要把函数包成 closure（unstable_cache 必须包）
  • cache key 自动从参数推断，不需要手动传 key 数组
  • 更接近 React 19 'use server' / 'use client' 的指令风格，统一心智

03 应用场景

场景 1 · v14 项目升 v16 的迁移路径

不要一步到位。两步走：

  1. 第一步：v14 → v15
    • 跑官方 codemod 自动改 async params / cookies / headers
    • 逐个审查所有 fetch()，把"原本依赖默认 cache"的显式改为 { cache: 'force-cache' } 或 { next: { revalidate } }
    • 跑测试，特别是测"提交表单后看到新数据"这类
  2. 第二步：v15 → v16
    • 把 unstable_cache 改 'use cache'
    • 逐个 page 添加 cacheComponents: true 后看哪些 Suspense boundary 缺失（编译会提示）
    • 把读 cookies/headers 的部分 wrap 在 Suspense 内

场景 2 · 排查"为什么数据是旧的"

排查路径：

  1. 检查这段数据怎么获取（fetch 还是 db / 其他）
  2. fetch 的 cache 选项是什么？v14 默认 cache、v15+ 默认 no-store；如果不确定写显式
  3. 是否在 'use cache' 函数内？如果是，cacheLife 是多久？
  4. 是否有 generateStaticParams 在 build 时预生成？
  5. CDN / 反向代理是否额外缓存？查 Cache-Control 头
  6. Router Cache（client）是否命中？开 DevTools Network 看是否真的 fetch 了
  7. 是否在 Server Action 后调用 revalidate？

场景 3 · 决定一段代码是 'use cache' 还是 dynamic

判断标准：

  • 对所有用户结果一样、可以提前生成 → 'use cache'（marketing 页 / 文档 / 商品详情）
  • 每用户不同（个性化 / dashboard）→ dynamic（在 Suspense 内、读 cookies）
  • 实时数据（股票 / 在线人数）→ dynamic
  • 偶尔变（每小时 / 每天）→ 'use cache' + cacheLife('hours' / 'days')
  • 用户写完立刻要看到 → dynamic 或 'use cache' + updateTag

04 典型代码

代码 1 · Blog 页面的标准 v16 实现

tsx 
// app/blog/[slug]/page.tsx
import { Suspense } from 'react';
import { unstable_cacheLife as cacheLife, unstable_cacheTag as cacheTag } from 'next/cache';

async function getPost(slug: string) {
  'use cache';
  cacheLife('days');
  cacheTag(`post-${slug}`);
  return db.posts.findUnique({ where: { slug } });
}

async function getRelatedPosts(slug: string) {
  'use cache';
  cacheLife('hours');
  return db.posts.findMany({ where: { tags: { hasSome: [/* ... */] } }, take: 5 });
}

async function getComments(slug: string) {
  // 不 cache：每请求新评论
  return db.comments.findMany({ where: { postSlug: slug }, orderBy: { createdAt: 'desc' } });
}

export default async function PostPage({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
  const post = await getPost(slug);
  if (!post) notFound();

  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.content }} />

      <Suspense fallback={null}>
        <Related slug={slug} />
      </Suspense>

      <Suspense fallback={<p>Loading comments...</p>}>
        <CommentsList slug={slug} />
      </Suspense>
    </article>
  );
}

async function Related({ slug }: { slug: string }) {
  const related = await getRelatedPosts(slug);
  return <aside>{related.map(p => <a key={p.id} href={`/blog/${p.slug}`}>{p.title}</a>)}</aside>;
}

async function CommentsList({ slug }: { slug: string }) {
  const comments = await getComments(slug);
  return <ul>{comments.map(c => <li key={c.id}>{c.body}</li>)}</ul>;
}

代码 2 · v15 fetch 显式 cache

tsx 
// app/products/page.tsx (v15)

// 静态：build 时预渲染，CDN 秒返
const products = await fetch('/api/products', {
  cache: 'force-cache',
}).then(r => r.json());

// 动态：每请求新拿
const flashSale = await fetch('/api/flash-sale', {
  cache: 'no-store',
}).then(r => r.json());

// ISR 风格：60 秒后台重新生成
const featured = await fetch('/api/featured', {
  next: { revalidate: 60 },
}).then(r => r.json());

// 带 tag：可被 revalidateTag 失效
const userCart = await fetch('/api/cart', {
  next: { tags: [`cart-${userId}`] },
}).then(r => r.json());

代码 3 · 自定义 cache handler（self-host）

self-host 时默认 fs cache 在容器重启会丢，需要接 Redis：

javascript 
// cache-handler.js (Next.js 16)
const Redis = require('ioredis');

class RedisCache {
  constructor() {
    this.client = new Redis(process.env.REDIS_URL);
  }

  async get(key) {
    const data = await this.client.get(`next:${key}`);
    return data ? JSON.parse(data) : null;
  }

  async set(key, data, { tags }) {
    await this.client.set(`next:${key}`, JSON.stringify(data));
    for (const tag of tags || []) {
      await this.client.sadd(`tag:${tag}`, key);
    }
  }

  async revalidateTag(tag) {
    const keys = await this.client.smembers(`tag:${tag}`);
    for (const key of keys) {
      await this.client.del(`next:${key}`);
    }
    await this.client.del(`tag:${tag}`);
  }
}

module.exports = RedisCache;

// next.config.ts
export default {
  cacheComponents: true,
  cacheHandler: require.resolve('./cache-handler.js'),
};

05 常见陷阱

  • 不知道项目用 v14、v15 还是 v16 —— 查 package.json next 版本。三个版本的默认行为完全不同
  • v13-14 默认 cache 没察觉 —— 数据看起来"莫名其妙旧"。升 v15 + 显式标
  • 升 v15 不跑测试 —— fetch 默认从 cache 改 no-store 是行为变更，可能导致新版本性能下降（每个 fetch 都打 db）。要重新审视哪些应该 cache
  • v15 cookies 没 await —— TypeScript 报错明显，但 JS 项目运行时崩
  • v16 'use cache' 函数读 cookies —— 编译报错。'use cache' 函数不能依赖 request-specific 数据
  • v16 Suspense fallback 报错"this dynamic component must be inside Suspense" —— 把读 cookies/headers 的组件包 Suspense
  • cacheLife 写错单位 —— cacheLife(60) 不工作，要 cacheLife({ stale: 60, revalidate: 60, expire: 60 }) 或字符串如 'minutes'
  • 用了 dynamicIO（v15 实验）但忘了改 cacheComponents（v16 正式） —— v16 把配置名改了，老配置仍可用但 deprecated
  • Redis cache handler 内存爆掉 —— 没设置 TTL；tag 索引不清理。生产用要加监控 + 自动清理过期 key
  • "为什么我 revalidateTag 后页面没刷" —— 检查 fetch / 'use cache' 是否真的声明了对应 tag；revalidate 是"下次访问失效"，不是"立即看到"。要立即看到用 updateTag（v16）
  • 把所有东西 'use cache' —— 个性化数据（user-specific）不能简单 cache，会让用户 A 看到用户 B 的数据。'use cache' 必须用于"对所有人结果相同"的数据
  • 认为 v16 解决了所有缓存问题 —— v16 简化了心智但仍有边角情况（multi-tenant 隔离 / 大数据集失效粒度 / Redis HA）。Cache Components 在迭代
  • 看老博客文章学缓存 —— Next.js 缓存 4 年改 4 次，2022-2023 年的文章几乎全过时。看 nextjs.org 当前版本的 docs
