同代人横评：10 工具全维度对比

横向对照 2026-05 主流构建工具：Webpack / Rollup / Parcel / Vite / Turbopack / Rspack / Rolldown / Bun build / esbuild / Farm。每个工具都有"sweet spot"——选错工具不是性能问题，是不匹配场景。本章一次讲清各自定位 + 选型决策。

阅读时长 ~25 分钟 前置 P1.1（项目史） 主要来源 各工具官方 + 实测 benchmark + 社区 case studies

01 速览

  1. 没有"通用最佳" bundler—— 每个工具都有 sweet spot：库 → Rollup；app → Vite；Next.js → Turbopack；Webpack 大型项目迁移 → Rspack；Bun 项目 → Bun build；通用快 transform → esbuild/SWC。选错 ≠ 性能问题，而是不匹配场景。
  2. 2026 默认 = Vite—— 新 app 项目 70%+ 选 Vite；Vue/Svelte/Solid/Astro 默认；React 大量从 CRA 迁移。Webpack 仍存量但新项目几乎不选。
  3. Rust bundler 5 兄弟：Turbopack（Vercel）/ Rspack（字节）/ Rolldown（Vite team）/ Farm（中国社区）/ 各自绑定不同生态。Bun build 用 Zig 不是 Rust，但同属"原生语言 bundler"阵营。
  4. 每个工具的"绑定生态"是关键—— Turbopack 绑 Next.js / Rspack 绑 Webpack 兼容 / Rolldown 绑 Vite + Rollup ecosystem / Bun build 绑 Bun runtime。独立用某个 Rust bundler 罕见。
  5. esbuild 是"工具链子组件"—— 不是完整 bundler 替代。Vite dev / Bun build 内部 / 大量 transform 工具用 esbuild。直接当 bundler 用仅适合极简单项目。
  6. Webpack 仍存活但停止增长—— Tobias Koppers 走了 / Rspack 是它的 Rust 替代 / Webpack 6 长期"路线图中" 没真正发布。但2026 仍有数百万项目用 Webpack，迁移要按需。

分工具详细解读

Webpack 5+：老牌 + 不增长

定位：老牌；Loader/Plugin 生态最大；新项目几乎不选。

优势：

  • 插件 ecosystem 最大（webpack-plugins / loader 数千个）
  • 支持几乎所有奇怪的需求（Module Federation / 复杂 chunk 策略 / 多入口）
  • 大型企业项目久经考验

劣势：

  • JS 实现性能上限（dev start 15-30s 大项目；HMR 1-3s）
  • 配置复杂（webpack.config.js 几百行常见）
  • 核心团队走了（Tobias Koppers 去 Vercel）
  • Webpack 6 长期延期

选它的场景：(1) 已有 Webpack 项目稳定运行；(2) 需要 Module Federation 等高级特性 Webpack 独家；(3) 团队已经精通 Webpack 不想换。

Rollup 4+：库打包标准

定位：库打包；ESM 第一公民；不擅长 dev server。

优势：

  • 最干净的 ESM 输出
  • Tree shaking 第一公民
  • 多输出格式（ESM/CJS/UMD/IIFE 同时产出）
  • Plugin API 是事实标准（Vite/Rolldown/Bun build 都兼容）

劣势：

  • 没 dev server / HMR（库不需要）
  • JS 实现 build 慢（Rolldown 接班）

选它的场景：库作者；2026 等 Rolldown stable 后切换。

Vite 6+：app dev 事实标准

定位：app dev/build 一站式；2026 默认选择。

优势：

  • 启动 < 1s；HMR < 50ms
  • 配置简洁（默认 OK，按需加）
  • 框架无关（Vue/React/Svelte/Solid/Astro/Lit 等都用）
  • Rollup plugin 兼容 → 生态丰富
  • 2024-11 Vite 6 加 Environment API 让 SSR/RSC/Edge 都能跑同一 Vite
  • Rolldown 接班让 build 也加速

劣势：

  • build mode 仍是 Rollup 慢（Rolldown 解决中）
  • 极复杂场景（Module Federation 等）支持不及 Webpack

选它的场景：99% 新 app 项目。

Turbopack：Next.js 强绑定

定位：Next.js 15+ 默认；Rust + 增量编译。

优势：

  • 极快（增量编译 + 共享缓存）
  • Next.js 深度集成（RSC / Server Actions / App Router 等）
  • Tobias Koppers 主导（Webpack 原作者经验）

劣势：

  • Next.js 之外用得罕见
  • plugin ecosystem 非常弱（不兼容 Webpack/Rollup plugin）
  • 独立文档少

选它的场景：Next.js 15+ 项目（不需要主动选 —— Next.js 默认）。

Rspack 1+：Webpack 大项目迁移神器

定位：Rust + Webpack API 兼容；让旧项目"换 Rspack 即加速"。

优势：

  • Webpack 配置 + plugin 几乎直接兼容（90%+ 不改）
  • 构建速度 5-10x Webpack
  • 字节跳动持续维护（抖音 / TikTok 用）

劣势：

  • 少数 Webpack plugin 不兼容（特别是用 internal API 的）
  • 新项目不会选（Vite 更适合）

选它的场景：大型 Webpack 项目，迁移到 Vite 成本太高，但启动 / 构建慢急需解决。

Rolldown：Vite team 自家 Rust bundler

定位：Rust 实现的 Rollup 兼容版；2026-Q2 stable 后接班 Vite build。

优势：

  • Rollup plugin API 兼容（切换无痛）
  • Rust 速度（10x Rollup）
  • Vite team 主导，与 Vite 深度协调

劣势：

  • 2026-05 仍 alpha；生产用要谨慎

选它的场景：等 Vite 7+ 自动用上 Rolldown；现在不需要主动切。

Bun build：Bun runtime 内置 bundler

定位：Bun 自家 bundler；Zig 实现 + JSC 集成；详见 Bun P4.3。

优势：

  • 极快（Zig + 与 Bun runtime 共享 parser）
  • Bun.serve 集成 dev server
  • 简单 API（bun build src/index.ts --outdir dist）

劣势：

  • plugin ecosystem 弱
  • 非 Bun 项目用不上
  • HMR 仍在快速发展中

选它的场景：Bun runtime 项目；需要极简 bundle 工具。

esbuild：工具链子组件

定位：极快 transformer + minimal bundler；不是完整 bundler 替代。

优势：

  • 极快（10-100x Babel）
  • Go 实现 + 单遍编译
  • 简单 API + 易嵌入

劣势：

  • Plugin 不能改 AST（只能 resolve / load hooks）
  • HMR 弱
  • Code splitting 简单

选它的场景：(1) 工具链子组件（Vite dev / 各种 transform）；(2) 极简单项目（小工具 / 单页 demo）；(3) 库构建快速预览。

Farm：中国社区 Rust bundler

定位：Vite plugin 兼容 + 声称比 Vite 快 5-10x。

优势：

  • Rust 实现速度
  • Vite plugin 兼容
  • 持续活跃（中文社区主要支持）

劣势：

  • 用户群较小（vs Vite 主流）
  • 遇到边缘问题求助资源少
  • 实测加速幅度因项目而异

选它的场景：愿意尝试新工具 + 追求极致速度 + 不怕踩坑的团队。

03 应用场景：选型决策树

2026 选型决策树

问题 1：你写的是什么？

  • library（npm 发包）→ Rollup（即将 Rolldown）
  • web app → Vite（默认）
  • Next.js app → Turbopack（Next.js 默认）
  • Bun runtime → Bun build

问题 2：你已有什么？

  • 大型 Webpack 项目，启动 / build 慢急需解决 → Rspack（兼容性最好）
  • 大型 Webpack 项目，能完全重写 → 迁 Vite
  • 稳定 Webpack 项目，速度可接受 → 不动

问题 3：性能优先级？

  • 启动最快 → Bun build / esbuild / Farm（按场景）
  • build 最快 → Rolldown / Bun build / esbuild
  • 稳态 / 兼容性 → Vite + Rollup → Rolldown 接班路径

问题 4：团队偏好？

  • 主流 + 文档丰富 → Vite
  • 愿意尝试新工具 → Bun build / Farm / Rolldown 早采用
  • 需要 Webpack 兼容 → Rspack

常见错配场景

错配 1：库作者用 Vite

Vite 的优化方向是 app dev / build；输出库格式（multi-format ESM/CJS/UMD）不是优势。库用 Rollup（直接）。Vite 的 lib mode 能用但不是首选。

错配 2：Next.js 用 Vite

Next.js 与自家 bundler（Webpack 老 / Turbopack 新）深度绑定（RSC / Server Actions / App Router 都依赖）。Next.js 项目跟随 Next.js 选择。如果一定要 Vite + RSC，看 vite-plugin-rsc / TanStack Start 等社区方案。

错配 3：简单项目用 Webpack

2026 简单 React/Vue/Svelte 项目用 Webpack 是过度工程—— 配置复杂 + 启动慢 + 体验差。Vite 是更好选择。

错配 4：用 esbuild 做完整 bundler

esbuild 设计是 compiler 子组件。除非项目极简单（单文件 demo / 小工具），完整 app 用 Vite（内部已含 esbuild）。

04 典型代码：相同任务不同工具

"打包一个 React 库"

Rollup（推荐）

// rollup.config.js
import resolve from '@rollup/plugin-node-resolve';
import commonjs from '@rollup/plugin-commonjs';
import typescript from '@rollup/plugin-typescript';
import dts from 'rollup-plugin-dts';

export default [
  {
    input: 'src/index.ts',
    output: [
      { file: 'dist/index.cjs', format: 'cjs' },
      { file: 'dist/index.mjs', format: 'es' }
    ],
    plugins: [resolve(), commonjs(), typescript()]
  },
  {
    input: 'src/index.ts',
    output: { file: 'dist/index.d.ts' },
    plugins: [dts()]
  }
];

Vite lib mode（也可以但不首选）

// vite.config.js
import { defineConfig } from 'vite';

export default defineConfig({
  build: {
    lib: {
      entry: 'src/index.ts',
      name: 'MyLib',
      fileName: (format) => `index.${format}.js`
    },
    rollupOptions: {
      external: ['react'],
      output: { globals: { react: 'React' } }
    }
  }
});

"打包一个 React app"

Vite（推荐）

// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()]
});

Webpack（旧）

// webpack.config.js
const HtmlWebpackPlugin = require('html-webpack-plugin');

module.exports = {
  entry: './src/index.tsx',
  module: {
    rules: [
      { test: /\.tsx?$/, use: 'ts-loader', exclude: /node_modules/ },
      { test: /\.css$/, use: ['style-loader', 'css-loader'] },
      { test: /\.(png|jpg)$/, type: 'asset/resource' }
    ]
  },
  resolve: { extensions: ['.tsx', '.ts', '.js'] },
  plugins: [new HtmlWebpackPlugin({ template: './public/index.html' })],
  devServer: { hot: true, port: 3000 }
};

Bun build（Bun 项目）

$ bun build src/index.tsx --outdir dist --target browser --jsx-import-source react

05 常见陷阱

陷阱 1：相信营销 benchmark

每个工具发布时都会 benchmark 自己很快。实测往往打折：

  • Turbopack 早期 alpha 声称 "Webpack 700x" → 实际 5-15x（依项目）
  • Farm 声称 "Vite 5-10x" → 部分场景接近，部分场景仅 1.5-2x
  • "快 100x" 通常是 minimal benchmark；真实复杂项目 5-20x 是合理预期

评估方法：在自家项目跑实测，不要看营销数字。

陷阱 2：过度追求"最新"

2024-2026 构建工具变化快，每月有新工具 / 大版本。但生产项目不应该追新：

  • Vite 7（2026 预期）/ Rolldown stable 等让"等等再切"是合理
  • 稳定的 Vite 6 + 项目稳定运行 = 比追 Rolldown alpha 更明智
  • Farm / Bun build 等小众工具：愿意踩坑 + 报 issue 的早期采用者才上

陷阱 3：忘记"工具链兼容性"

选 bundler 时检查：

  • 你用的 framework（React/Vue/Svelte）是否有官方 plugin？
  • 你用的 CSS 方案（Tailwind / CSS-in-JS）是否兼容？
  • 你用的 testing 工具（Vitest / Jest）是否能用？
  • 你的 deploy 平台（Vercel / Netlify / Cloudflare）是否支持？

常见问题：选 Bun build 但 Vercel 默认 Node + npm 部署；要额外配置。

陷阱 4：换 bundler 期望"100% 一致"

不同 bundler 行为不完全一致。换 Webpack → Vite 常见差异：

  • CommonJS 处理方式不同（Webpack 自动 / Vite 通过 plugin）
  • require.context 等 Webpack 独家特性 Vite 没有
  • Babel transform 链略有差异

换 bundler 必须充分测试，不要假设无缝切换。

陷阱 5：以为 Rust bundler "一律快"

实现语言只决定速度上限，不决定实际速度。早期 Turbopack alpha 在某些场景比 Vite 慢；某些 Rust bundler 因为没充分优化，实测仅快 2-3x。实测优于实现语言。

陷阱 6：以为 esbuild 能全场景替代

esbuild plugin 能力有限（不支持 AST 转换），让它不能完全替代 Vite/Webpack。常见场景：

  • Vue SFC / Svelte：esbuild 没原生支持，要 plugin
  • React Refresh：esbuild 无法
  • 复杂 chunk 策略：esbuild 简单

esbuild 是子组件不是替代品。
