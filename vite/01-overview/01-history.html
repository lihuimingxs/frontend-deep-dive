构建工具项目史（1976-2026）

把构建工具 50 年史讲成一条主线 —— 不是按工具版本，按"每次范式转移及其根因"。核心问题：构建工具怎么从 1976 Make 演化到 2026 Vite + Rust bundler？答案藏在三次范式转移：(1) Make 看文件 → Webpack 看模块（CommonJS）；(2) Webpack 全量打包 → Vite 不打包 dev + 浏览器原生 ESM；(3) JS bundler → Rust/Go bundler。每次都有具体技术原因，不是"工具流行的轮换"。

阅读时长 ~30 分钟 前置 无 主要来源 各工具官方 + 作者博客（Evan You / Rich Harris / Tobias Koppers / Evan W）

01 速览

  1. 构建工具 50 年由 3 次范式转移组成—— (1) Make 看文件 → Webpack 看模块（2014）；(2) Webpack 全量打包 → Vite 不打包 dev + 浏览器原生 ESM（2020）；(3) JS bundler → Rust/Go bundler（2020-2024）。每次转移都有具体技术原因。
  2. 2014-2022 是 Webpack 主导期—— "什么都是模块"哲学统治。但代价是配置爆炸（webpack.config.js 几百行常见）+ 启动慢（10-60s 大项目）+ 维护停滞（核心团队走了）。
  3. 2016 浏览器原生 ESM—— Chrome 61 / Safari 10.1 / Firefox 60 默认支持 <script type="module">。这奠定了 Vite 设计基础——dev mode 不打包成为可能。
  4. 2020 是构建工具大爆发年—— esbuild（Go）/ Vite（JS+esbuild）/ SWC（Rust）三连发。esbuild 证明"compiler 用 Go 写能快 100x"；Vite 证明"dev 不打包是对的"；SWC 让 Rust 进入 JS 工具链。
  5. 2022-2023 Webpack 退场不可逆—— Tobias Koppers（Webpack 原作者）跳槽 Vercel 做 Turbopack；CRA 2023 弃用；新项目几乎没人选 Webpack。Rspack（字节跳动）是给"Webpack 大型企业项目"的 Rust 兼容版替代。
  6. 2024+ 进入 Rust bundler 时代—— Turbopack（Next.js stable）/ Rspack 1.0 / Rolldown alpha / Bun build / Farm。每个绑定一个生态：Turbopack→Next.js / Rspack→Webpack 迁移 / Rolldown→Vite / Bun build→Bun。2026 趋势：Vite + Rolldown 是 app dev/build 双标准。

02 底层逻辑

前史：Make · 1976 · "基于文件依赖图编译"模式起源

1976 年 Stuart Feldman 在贝尔实验室发明 Make。原始动机：C/C++ 项目编译时不想每次都全量重编 —— 要"只重编修改过的文件"。

Make 的核心 idea：

(1) Makefile 描述依赖图—— "foo.o 依赖 foo.c bar.h"；变化时只重编受影响的部分

(2) 任务驱动—— Make 不只是编译，是"声明依赖 + 执行命令"框架；可以 build / clean / test / install

(3) timestamp-based 增量—— 用文件 mtime 判断"是否需要重做"

这套模型至今是 C/C++/Go 等系统语言的标配（Bazel / Ninja / cmake 都从 Make 演化）。对 JS bundler 的影响：所有现代 bundler 内部都有"依赖图 + 增量编译"概念，源头在 Make。

2006-2009 · 早期 JS 构建：手工与脚本

jQuery 时代（2006-2010），JS 项目通常：

  • 手写多个 .js 文件，HTML 里 <script> 引一堆
  • 用 YUI Compressor（2007，Yahoo）/ Closure Compiler（2009，Google）做压缩
  • 没有"模块"概念 —— 全局变量到处飞 + 命名空间靠 window.foo = {...}

这个阶段的"构建"是手工脚本拼接 + 压缩，没有真正的工具。

2012 · Grunt · 第一个流行的 JS 任务化构建

2012-01 Ben Alman 发布 Grunt。设计：JS 版的 Make + 配置插件化。

Grunt 的核心：

  • Gruntfile.js—— 配置驱动（不是命令式）
  • 插件 ecosystem—— grunt-contrib-uglify / grunt-contrib-concat / grunt-contrib-jshint
  • 任务（task）抽象—— grunt build / grunt test

Grunt 2013-2014 主流；之后被 Gulp（2013-10, Eric Schoffstall）替代——Gulp 的"流式 + 代码胜过配置"更符合 Node 流派。

但 Grunt 和 Gulp 都没解决"模块"问题—— 它们仍是任务编排，不是真正的 bundler。要做模块化，需要 Browserify / Webpack。

2013-07 · Browserify · 把 Node 的 CJS 带到浏览器

2013 年 7 月 James Halliday 发布 Browserify。核心 idea：让你在浏览器写 require('module')。

Node 已有 CommonJS 模块系统（module.exports / require），但浏览器没有。Browserify 做的：

(1) 静态分析 require 调用—— 找出所有依赖

(2) 把所有模块打包进单个 bundle.js—— 模块代码包装在 function(require, module, exports) {...} 里

(3) 运行时 require shim—— bundle 启动时注入 require 函数模拟 Node 行为

Browserify 启动了 "JS bundler" 时代。但有限制：(1) 仅支持 CJS 模块（不支持 AMD / 当时尚未存在的 ESM）；(2) 配置不够灵活；(3) 不擅长 CSS / image 等非 JS 资源。

2014-03 · Webpack 1 · "什么都是模块"哲学

2014 年 3 月 Tobias Koppers 发布 Webpack 1.0。核心 idea：把 Browserify 的"模块"概念扩展 —— JS / CSS / image / font 都是 module。

Webpack 的设计创新：

(1) Loader 系统—— 每种文件用对应 loader 转换（css-loader / file-loader / babel-loader）；让 import './style.css' 这种成为可能

(2) Plugin 系统—— 在编译生命周期注入逻辑（DefinePlugin / HtmlWebpackPlugin / MiniCssExtractPlugin）

(3) 多种模块语法支持—— CJS / AMD / ESM / Webpack 自家 require

(4) Code splitting—— dynamic import → 自动分 chunk

(5) Dev server + HMR—— webpack-dev-server 内置；HMR 概念在此发明

Webpack 在 2016-2018 年成为事实标准：

  • create-react-app（2016 Facebook）默认 Webpack
  • Vue CLI 默认 Webpack
  • Angular CLI 默认 Webpack
  • Next.js（早期）基于 Webpack

但 Webpack 的代价 2018 后越来越明显：

  • 配置复杂度爆炸—— 一个完整 webpack.config.js 几百行常见；entry / output / module rules / plugins / optimization / resolve / 等几十个配置项
  • 启动慢—— 大项目 dev 启动 30-60s（要全量解析依赖图）
  • 构建慢—— production build 几分钟级
  • JS 实现性能瓶颈—— V8 单线程 + JS 字符串处理 + 没有 native 优化

2015-05 · Rollup · 专注 ESM + Tree Shaking

2015 年 5 月 Rich Harris（Svelte 作者）发布 Rollup。动机：Webpack 太重，不适合库打包。库需要：(1) 输出干净 ESM；(2) tree shaking；(3) 简洁配置。

Rollup 的设计：

(1) 专注 ESM 静态分析—— 只支持 ESM 输入（CJS 通过 plugin 转）；让 tree shaking 第一公民化

(2) 范围提升（scope hoisting）—— 把所有模块的代码"提升"到同一作用域（Webpack 后来跟进，叫 "module concatenation"）

(3) 无 dev server / 无 HMR—— 库不需要

(4) 多种输出格式—— ESM / CJS / UMD / IIFE 同时产出

2015-2025 Rollup 成为库打包事实标准：

  • React / Vue / Svelte / Lodash / D3 / RxJS / Three.js 等大量主流库用 Rollup
  • Vite build mode 内置 Rollup（2020+）
  • Rolldown（2024+）是它的 Rust 兼容继任者

Rich Harris 的著名文章 Tree-shaking versus dead code elimination（2015）首次提出 "tree shaking" 概念，深刻影响 Webpack 等所有 bundler。

2016 · 浏览器原生 ESM · 为 Vite 设计奠基

2016-2017 年浏览器开始原生支持 ESM：

  • Chrome 61（2017-09）默认开启
  • Safari 10.1（2017-03）
  • Firefox 60（2018-05）

语法：

<script type="module" src="./main.js"></script>
<!-- main.js 里可以 import './foo.js' 浏览器自己解析 -->

这看似小特性颠覆了 dev server 设计：以前 dev server 必须"全量打包成一个 bundle 给浏览器"；现在浏览器自己懂 ESM，dev server 可以"不打包，让浏览器逐个请求模块"。

Evan You 在 2019-2020 年正是从这里得到 Vite 的核心 idea。但 2017-2019 大部分人没意识到这个机会 —— 还在用 Webpack 全量打包 dev mode。

2018 · Parcel · 零配置挑战

Devon Govett 发布 Parcel 1.0（2017 年首发，2018 v1 稳定）。核心卖点：零配置（不需要 webpack.config.js）。

Parcel 1 用 JS 实现；Parcel 2（2021）转 Rust 后端。

Parcel 的局限：

  • plugin ecosystem 不及 Webpack
  • "零配置" 在小项目好用，大项目仍需要配置（不如 Vite 渐进式配置）
  • 2020 后 Vite 出来抢走"零配置 + 现代化"位置

2026 Parcel 仍存活但小众，主要用户是简单项目或对配置敏感的团队。

2020-01 · esbuild · Go 实现 + 100x 速度

2020 年 1 月 Evan W（前 Figma 工程师，化名 evanw）发布 esbuild 0.1。声称：比当时主流工具快 10-100x。

esbuild 的设计创新：

(1) Go 实现—— Goroutine 让真正多核并行（V8 单线程）

(2) 自家 AST in memory—— 不依赖现有 parser 库；为速度极致优化

(3) 单遍 parse + transform + bundle—— 不是"parse → transform → bundle"三遍

(4) 零依赖—— 自己实现 minifier / tree shaker / sourcemap

实测速度（典型 React 项目）：

工具build 时间
Webpack 4 + Babel~30s
Rollup + Babel~15s
esbuild~0.5s

esbuild 不是完整 bundler 替代—— 它的 plugin 系统弱（不能改 AST）/ HMR 弱 / code splitting 简单。但它极适合做"compiler 子组件"：Vite dev mode 用 esbuild 预构建 + transform；Snowpack 用；Bun build 内部参考其设计。

Evan W 的 essay "Why is esbuild fast?" 是构建工具领域经典必读。

2020-04 · Vite 1.0 · 颠覆 dev server

2020 年 4 月 尤雨溪（Vue 作者）发布 Vite 1.0。核心 idea：

Dev mode 不打包 + Build mode 打包

具体：

  • Dev mode：浏览器原生 ESM + esbuild 预构建依赖 + on-demand transform
  • Build mode：Rollup 生产打包

启动效果对比：

项目大小Webpack dev startVite dev start
100 文件3-5s< 500ms
1000 文件15-30s< 1s
5000 文件60s+（OOM 常见）~1s

Vite 2.0（2021-02）从 Vue-only 扩展到框架无关；2022-2023 大量框架迁移：

  • Vue 3 默认 Vite
  • SvelteKit 用 Vite
  • SolidJS 用 Vite
  • Astro 用 Vite
  • Nuxt 3 用 Vite
  • React 项目大量从 CRA 迁移到 Vite

Vite 6（2024-11）引入 Environment API（让 SSR / RSC / Edge 都跑同一 Vite）+ Rolldown 实验性集成。Vite 7（预计 2026-Q2）默认 Rolldown。

2020-12 · SWC · Rust 进入 JS 工具链

2020 年底 DongYoon Kang（Donny Wong）发布 SWC 1.0。定位：Rust 实现的 JS/TS compiler，Babel 替代品。

SWC vs Babel：

  • SWC 速度：~100x Babel
  • SWC 兼容性：90%+ Babel preset 覆盖
  • SWC plugin：用 Rust 写编译成 wasm；性能损失小（详见 P6.1）

2022 年 Next.js 12 默认切换到 SWC（替代 Babel），Vercel 雇 Donny Wong 全职做 SWC。Deno / Parcel 2 / 多个 Rspack 实验也用 SWC。

SWC 把 Rust 带入 JS 工具链，奠定后续 Turbopack / Rspack / Rolldown 都是 Rust 的趋势。

2022-2023 · Webpack 退场不可逆

2022 年发生几件事让 Webpack 退场不可逆：

(1) 2022-10 Tobias Koppers 跳槽 Vercel—— Webpack 原作者去做 Turbopack；Webpack 失去核心维护者

(2) 2022-11 Turbopack alpha 发布—— 声称 Webpack 700x 快；Vercel 重金推

(3) 2023-03 Rspack 0.1—— 字节跳动发布；让 Webpack 项目"换上即加速"；大型企业开始迁移

(4) 2023-04 CRA 弃用—— React 官方不再推荐 CRA，建议用 Vite / Next.js / Remix

(5) 2023 大量项目从 Webpack 迁 Vite—— Vue / Svelte / Solid 等社区先行；React 社区也大量跟进

2024-2026：Webpack 仍有大量存量（数百万项目），但新项目几乎没人选 Webpack。Webpack 6 长期"路线图中"但没真正发布。

2023-08 · Rolldown · Vite team 自家 Rust bundler

2023-08 尤雨溪团队（Vite 团队 + 多名贡献者）启动 Rolldown。目标：Rust 实现的 Rollup 兼容 bundler，接管 Vite build mode。

设计选择：

  • Rollup plugin API 兼容—— 让用户切换时 plugin 不需改
  • Rust + 并行—— 速度目标 10x Rollup
  • 逐步替换—— Vite 6 实验性，Vite 7+ 默认

2024 Rolldown 进入 Vite 6 实验阶段；2026-Q2 预期 stable。对 Rolldown 影响：成为 app + 库的统一 build 后端。

2024-09 · Bun build · 第三种语言（Zig）的 bundler

2024-09 Bun 发布 bun build。差异：用 Zig 实现（不是 Rust），与 Bun runtime（基于 JSC）深度集成。详见 Bun P4.3。

2026 Bun build 主要服务 Bun 项目用户；非 Bun 项目用 Vite / Turbopack 等。

2024-2026 · Rust bundler 时代成型

2026-05 现状：

  • Vite 6+（JS+esbuild+Rollup → 接 Rolldown）
  • Turbopack（Vercel, Rust, Next.js 15+ stable）
  • Rspack 1+（字节跳动, Rust, Webpack 兼容）
  • Rolldown alpha（Vite team, Rust, Rollup 兼容）
  • Bun build（Bun team, Zig）
  • Farm（中国社区, Rust）
  • Webpack 5+（仍主流但增长停滞）

趋势：每个新项目几乎都用 Rust/Go 实现的 bundler；JS bundler（Webpack / Rollup）退到"维护已有项目 + 库打包"。

03 应用场景：从历史看决策

场景 A · 评估"我的项目要不要从 Webpack 迁移"

2026 看是否迁移，历史给的判据：

(1) Webpack 启动慢（> 10s）严重影响 dev 体验？ → 迁

(2) build 慢（> 1 分钟）严重影响 CI？ → 迁

(3) 配置 webpack.config.js 复杂到没人懂？ → 迁

(4) 项目仍在快速迭代 + 新人加入多？ → 迁（Vite 学习成本低）

(5) 上面都不是 + 项目稳定不变？ → 不迁（迁移成本不值得）

迁移目标选择：

  • 项目复杂 / 用了大量 Webpack 特定 loader → Rspack（Webpack 兼容，成本最低）
  • 项目相对简单 / 想现代化 → Vite（生态最好）
  • 项目是 Next.js → Turbopack（自然升级）

场景 B · 选语言：JS bundler vs Rust bundler

2026 一般规则：

  • 新 app 项目：Vite（JS+Rolldown）
  • 新库：Rollup（即将 Rolldown 接班）
  • Next.js app：Turbopack
  • 大型 Webpack 迁移：Rspack
  • Bun runtime 项目：Bun build
  • 极致快 + 简单：esbuild（直接用，不要再套 Vite）

场景 C · 长期投资判断

从历史看：

Webpack 仍存活但持续退场—— 2030 预期仍有大量 legacy 项目用 Webpack，但新项目几乎不选

Vite + Rolldown 是 2026-2030 默认—— Vue/Svelte/Solid/Astro/SvelteKit/Nuxt 都已 Vite-first；React 社区大量迁移

Turbopack 绑定 Next.js—— 不会成为通用 bundler；但 Next.js 用户多，Turbopack 影响大

Rust bundler 占新项目 90%+（包括 Vite 内的 Rolldown）—— 不可逆

04 典型代码：版本时间线对照

同样的"打包 React app"在不同时代

2014 · Browserify + Gulp（早期前端 bundler 启程）

// gulpfile.js
const gulp = require('gulp');
const browserify = require('browserify');
const source = require('vinyl-source-stream');

gulp.task('bundle', function() {
  return browserify('./src/main.js')
    .transform('babelify', { presets: ['react'] })
    .bundle()
    .pipe(source('bundle.js'))
    .pipe(gulp.dest('./dist'));
});

典型项目结构：手动 require + Babel + Gulp 任务

2017 · Webpack 4 + Babel（主流期）

// webpack.config.js
module.exports = {
  entry: './src/index.js',
  module: {
    rules: [
      { test: /\.jsx?$/, exclude: /node_modules/, use: { loader: 'babel-loader' }},
      { test: /\.css$/, use: ['style-loader', 'css-loader'] },
      { test: /\.(png|jpg)$/, use: ['file-loader'] }
    ]
  },
  resolve: { extensions: ['.js', '.jsx'] },
  devServer: { hot: true, port: 3000 },
  // ... 配置 100+ 行省略
};

典型 Webpack 4 项目：webpack.config.js 100-300 行，需要专门"webpack 工程师"

2026 · Vite + Rolldown（现代）

// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()]
});

3 行配置 + 默认起步。需要时再加配置项（build target / chunks / proxy 等）。

启动速度演化

2017 Webpack 4 + Babel:    dev start ~30s, hot reload ~3-5s
2019 Webpack 5 + esbuild:  dev start ~15s, hot reload ~1-2s
2020 Vite 1:               dev start ~1s,  hot reload ~50ms
2024 Vite 6:               dev start ~500ms, hot reload ~30ms
2026 Vite 7 + Rolldown:    dev start ~300ms, build 5x faster

05 常见陷阱

陷阱 1：以为"Webpack 死了应该立刻迁"

Webpack 没死，只是不增长。稳定运行的 Webpack 项目不要为了迁移而迁移—— 迁移成本（plugin / loader 重写 + 配置重学 + 兼容性测试）通常 1-3 周；如果 dev / build 速度还能接受，不值得迁。

典型迁移触发点：CI 太慢拖累团队 / 新人吐槽 dev 启动慢 / 大版本升级时顺势迁。

陷阱 2：用 esbuild 当完整 bundler

esbuild 设计是compiler 子组件，不是完整 bundler。它的 plugin 弱（不能改 AST）/ HMR 弱 / code splitting 简单。除非项目极简单，否则用 Vite（内置 esbuild dev + Rollup build）更合适。

陷阱 3：以为 Vite 等于 esbuild

常见误解。Vite dev mode 用 esbuild 预构建依赖 + transform；但 Vite build mode 用 Rollup（即将切到 Rolldown）不用 esbuild。这是 Vite 双模架构的精髓（详见 P2）。

陷阱 4：选 Turbopack 但不用 Next.js

Turbopack 与 Next.js 强绑定 —— 独立用 Turbopack 的项目极罕见，文档 / 生态都围绕 Next.js。不用 Next.js 不要选 Turbopack，选 Vite。

陷阱 5：以为"Rust bundler 就是快"

Rust 实现速度上限高，但实际速度还要看具体优化。早期 Turbopack alpha 版本在某些场景比 Vite 慢；Farm 声称比 Vite 快 5-10x 但部分场景实测差距小。看实测 benchmark 而不是营销。

陷阱 6：执着"零配置"

Parcel "零配置"卖点 2018 时强；2026 Vite 已"轻配置"+ 大量默认正确。真正的工程项目几乎都需要配置（build target / proxy / env / aliases / plugin 等）。Vite 的"按需配置"比 Parcel "零配置"更适合复杂项目。
