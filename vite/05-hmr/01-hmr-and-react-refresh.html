HMR + React Refresh / Vue HMR / Svelte HMR

"改一行 React 代码不丢 state" 是 modern dev 体验核心。本章讲清：HMR 怎么真正工作（文件变化 → WebSocket → 浏览器 client → fetch 新模块 → accept callback）+ 模块边界（自我边界 / 父边界 / 走到顶 full reload）+ React Refresh 怎么自动 HMR 组件 + Vue/Svelte HMR 实现 + 为什么有时 state 丢失（修改 hook 数 / 顶层全局 / 副作用注册等）。

阅读时长 ~30 分钟 前置 P2.1（Vite Dev mode） 主要来源 Vite HMR API · React Refresh docs · Vue HMR · Svelte HMR · Webpack HMR docs

01 速览

  1. HMR（Hot Module Replacement）核心思想—— 源代码改了 → 不重启浏览器（保留 state） → 热替换改动的模块。HMR 是 Webpack 2014 发明的概念；现代 bundler 都支持。
  2. HMR 协议层：1) Vite server 监听文件变化（chokidar）；2) 检测到改动 → 编译新版本；3) WebSocket push HMR update message；4) 浏览器 client 收到 → 决定 accept 或 full reload；5) accept → fetch 新模块 → 替换 + 调 accept callback。
  3. import.meta.hot API—— 模块声明自己怎么 accept HMR：import.meta.hot.accept((newMod) => {...})。Vite/Webpack/Rspack 等都用类似 API。
  4. HMR boundary（边界）—— 模块 accept 自己 = 自我边界；父模块 accept 子模块 = 父是边界；一直没 accept = 走到顶 = full reload。组件级 HMR 需要边界设置正确。
  5. React Refresh—— 自动给每个 React 组件加 HMR boundary。组件改了 → 重新挂载组件 + 保留 useState/useReducer 状态。Hook order 改变（违反 Rules of Hooks）→ full reload（无法保留 state）。
  6. Vue HMR：SFC（.vue 文件）每个 block（template / script / style）独立 HMR；Vue runtime 处理保留 reactive state。Svelte HMR：编译时 instrument；保留组件实例 state。
  7. 典型失败场景—— 修改 hook 数 / 顶层全局状态 / 副作用注册（map polyfill）/ 修改 export 名 / CSS-in-JS 样式名变化。HMR 不是全自动，依赖框架和用户配合。

02 底层逻辑

HMR 完整流程

用户在编辑器修改 src/Counter.tsx：

1. 文件保存 → fs 通知
2. Vite server (chokidar) 检测到 src/Counter.tsx 变化
3. Vite server 编译新版 Counter.tsx (TS → JS + JSX)
4. Vite server 计算 HMR 更新计划:
   - 检查 Counter.tsx 是否 self-accept (import.meta.hot.accept)
   - 如果 accept → 单文件更新
   - 如果 not accept → 检查父模块（importer）
   - 直到找到边界或走到顶
5. WebSocket push: { type: "update", path: "/src/Counter.tsx", ... }
6. 浏览器 client (vite/client.js) 收到消息:
   - import 新模块 (fetch /src/Counter.tsx?t=timestamp)
   - 调当前模块的 hot.accept callback，传入新模块
7. callback 执行:
   - React Refresh: re-render 组件，保留 hook state
   - Vue: 重新激活 component，保留 reactive
   - 自定义: 用户决定怎么处理

整个流程通常 < 50ms。

import.meta.hot API（Vite）

每个模块通过 import.meta.hot 声明 HMR 行为：

// my-module.js
export const value = 42;

if (import.meta.hot) {
  // 接受自己的更新
  import.meta.hot.accept((newModule) => {
    // newModule 是新版本的导出
    console.log('Module updated, new value:', newModule.value);
  });

  // 接受依赖的更新
  import.meta.hot.accept('./dep.js', (newDep) => { ... });

  // 接受多个依赖
  import.meta.hot.accept(['./a', './b'], ([newA, newB]) => { ... });

  // 旧 API：清理（已弃用，用 dispose）
  import.meta.hot.dispose((data) => {
    // 清理 timer / event listener / etc.
    // data 会传给新版 module 的 hot.data
  });

  // 完全主动 reload
  import.meta.hot.invalidate();
}

// 访问之前 dispose 传的数据
if (import.meta.hot) {
  const oldState = import.meta.hot.data;   // 之前 dispose 传的
}

HMR 边界（boundary）

当模块 X 改动，Vite 沿着 importer 链向上找第一个能 accept 的模块：

App.tsx (root, 没 accept)
  ├─ Header.tsx (没 accept)
  └─ Pages.tsx (没 accept)
       └─ Counter.tsx (有 import.meta.hot.accept)
            └─ utils.ts (没 accept)

修改 utils.ts → Vite 找 importer:
  utils.ts → Counter.tsx (Counter accept 自己 + 任何 import 的更新) → 替换

规则：

  • 修改的模块 self-accept → 单文件更新
  • 父模块 accept 子模块 → 父更新
  • 没人 accept → 走到顶 → full page reload

full reload 让 state 全丢。所以好的 HMR 要 accept 在合适层级。

React Refresh：自动 HMR boundary

每个 React 组件如果手动写 hot.accept 太烦。React Refresh（Facebook 出品，Vite/Next/Webpack 都支持）自动注入：

// 你写的
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

// React Refresh 编译时注入（简化）
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

if (import.meta.hot) {
  import.meta.hot.accept();
  // RefreshRuntime 处理：替换组件 + 保留 fiber 节点 + 保留 useState 值
  RefreshRuntime.register(Counter, "Counter");
}

React Refresh 怎么保留 state

核心：用 React 的 fiber tree（详见 React P4 Fiber）。

  1. 组件改了 → 收到新版本 Counter
  2. RefreshRuntime 找到所有 Counter 实例（fiber 节点）
  3. 用新版本替换 fiber.type
  4. 调度 React 重新 render；hook state 保留（hook list 还在 fiber 上）

结果：组件 UI 用新代码渲染，state 不变。

React Refresh 限制

(1) Hook 数量 / 顺序变 → full reload

React 依赖 hook 调用顺序唯一识别（详见 React P3）。改 hook 数让顺序变 → React 不知怎么 mapping → 强制 full reload。

// 改前
function Counter() {
  const [count, setCount] = useState(0);
  return <...>;
}

// 改后（多了一个 useEffect）
function Counter() {
  const [count, setCount] = useState(0);
  useEffect(() => { ... });   // ← 加新 hook
  return <...>;
}

// HMR 看到 hook count 变 1 → 2 → full reload

(2) 改非组件文件（utils 函数）→ 仍可 HMR 但 state 可能不更新

如果 Counter 用 formatCount，你改 formatCount.ts：

  • React Refresh 不直接管 utils（只管组件）
  • Vite 默认行为：父模块 (Counter) 重新加载 → 触发 React Refresh → 组件 re-render
  • 但如果 utils 顶层有 side effect（注册全局），不一定按预期

(3) 默认导出非组件 → full reload

// ❌ React Refresh 不识别 default export 是常量
export default 42;

// ❌ 也不识别 default export 是非组件函数
export default function notAComponent() { return 42; }

// ✅ 命名导出常量 OK（不影响 React Refresh）
export const SOME_CONST = 42;

(4) Anonymous default export → 无法识别

// ❌ 匿名 export，React Refresh 不能识别为组件
export default () => <div>Hello</div>;

// ✅ 给个名字
export default function Component() {
  return <div>Hello</div>;
}

Vue HMR：SFC block 级

Vue 单文件组件（.vue）有 template / script / style 三个 block。Vue HMR 把每个 block 独立处理：

<script setup>
import { ref } from 'vue';
const count = ref(0);
</script>

<template>
  <button @click="count++">{{ count }}</button>
</template>

<style>
button { color: red; }
</style>

HMR 行为：

  • 改 <template> → 只重新 render，保留 setup 内的 ref 值
  • 改 <style> → 替换 style 元素，DOM 不动
  • 改 <script> 但不改 setup 顶层结构 → 重 render，保留
  • 改 <script> 大改（加 props / 改 emit 等）→ 重新挂载，state 丢

Vue 3 Composition API 让 setup 级 HMR 更稳定（vs Options API）。
