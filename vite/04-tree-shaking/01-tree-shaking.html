Tree Shaking 真相

行业最被误解的特性之一。"我开了 tree shaking 怎么 bundle 还这么大"是常见困惑——答案是 tree shaking 不是开关，依赖整个生态共同配合：包正确标 sideEffects + bundler 实现 + 用户写 code 风格 + 编译器不破坏 ESM 结构。任何一环失效都让 tree shake 失败。Rollup 最严最干净 / Webpack 依赖 sideEffects + Babel pure annotation 常假阳性 / esbuild 中等严格。本章讲清：tree shaking 怎么真正工作 + 失效场景全谱 + 怎么诊断和修复。

阅读时长 ~30 分钟 前置 P3.1（ESM 标准）+ P3.2（CJS Interop） 主要来源 Rich Harris 经典文章 · Rollup docs · Webpack tree shaking docs · esbuild 文档

01 速览

  1. Tree shaking 定义—— 编译时分析模块图 → 标记未使用的 export → 不打包到产物。"shake 掉死代码"。
  2. 前提：必须用 ESM—— ESM 静态分析（详见 P3.1）让编译时知道哪些 export 被用。CJS 是动态 require → 不能可靠 tree shake。用 lodash-es 而不是 lodash 是 tree shake 第一规则。
  3. side effects（副作用）概念—— 模块加载时执行的代码（console.log / 注册全局 / polyfill / 初始化）有副作用。tree shaker 默认假设所有模块有副作用 → 不敢删 → tree shake 不动。
  4. package.json sideEffects 字段—— 让包作者声明哪些文件有副作用："sideEffects": false（整包纯）/ "sideEffects": ["./src/polyfill.js", "*.css"]（指定）。没标 sideEffects 的包默认保守保留。
  5. 静态分析的 4 个限制：(1) 动态 import 不能完全分析；(2) 调用未知函数（可能改全局）保守处理；(3) re-export（export *）扩展分析图；(4) destructuring import 是必要不充分（虽然 import 但用了什么不一定确定）。
  6. 三大 bundler 实现差异：Rollup 最严（"tree shaking 第一公民"，无包标注也敢 shake）/ Webpack 依赖包标注 + Babel /* @__PURE__ */，常有"假阳性"（认为有副作用其实没有）/ esbuild 中等严格（接近 Rollup 但不及）。
  7. 失效场景全谱—— 大多数 "tree shake 没生效"是用户 code 写法 + 第三方包没标 sideEffects 共同导致。诊断工具：rollup-plugin-visualizer / source-map-explorer。

02 底层逻辑

tree shaking 概念起源

2015 年 Rich Harris（Rollup 作者）写文章 Tree-shaking versus dead code elimination，区分两个概念：

  • Dead code elimination（DCE）—— 编译器把已包含的代码里 unreachable 部分删（如 if (false) {...} 块）
  • Tree shaking—— 编译器在合并模块时，决定哪些 export 不引入

类比：DCE 是"做完一桌菜后扔掉吃不完的"；tree shaking 是"做菜前算好不要做太多"。

tree shaking 是 ESM 时代的特性，CJS 时代只有 DCE。

静态分析能做什么

编译时（不运行代码）能确定：

  • 每个模块的 import 路径（必须 literal）
  • 每个模块的 export 列表（必须顶层）
  • 每个 import 用了哪些 export（语法分析）
  • 构建完整模块依赖图

有了这些信息，tree shaker 算法：

  1. 从 entry 出发遍历所有可达 export
  2. 未被到达的 export 标记为"未使用"
  3. 把未使用的 export + 它的实现代码 删除

但实现代码删除有副作用风险：删了某个函数定义，但定义里有 polyfill.register() 怎么办？这就是 sideEffects 概念出现的原因。

Side effects 详解

"副作用"在 tree shaking 上下文 = 模块加载（top-level execution）时执行的代码：

// math.js
console.log('math.js loaded');   // ← 副作用！

export function add(a, b) {
  return a + b;
}
export function subtract(a, b) {
  return a - b;
}

用户代码：

// app.js
import { add } from './math';
console.log(add(1, 2));
// 没用 subtract

如果 tree shaker 删除 subtract → 看似 OK。但整个 math.js 模块还得 load（因为 console.log 是副作用）→ 实际上 subtract 实现也保留（防止 console.log 失效）。

tree shaker 的保守原则：遇到不确定的副作用，宁可不 shake。结果：用户经常发现 "明明没用怎么还在 bundle 里"。

package.json sideEffects 字段

包作者用 sideEffects 字段告诉 bundler "我没副作用 / 副作用在哪"：

// 整包纯（无副作用）
{
  "sideEffects": false
}

// 指定哪些文件有副作用
{
  "sideEffects": [
    "./src/polyfill.js",
    "*.css",
    "./src/init/*"
  ]
}

// 默认（不写）：所有文件都有副作用 → tree shake 不动

关键：

  • 包作者必须正确标—— 错标会让 polyfill 等被错删（bug）
  • 用户能 override：项目自己 package.json 设 sideEffects
  • 2026 主流包（lodash-es / date-fns / @mui / @emotion / etc.）都正确标

用户代码 sideEffects 标注

不只是包，用户项目也可以标：

// 你的 app 的 package.json
{
  "name": "my-app",
  "sideEffects": [
    "./src/polyfills/*",       // 这些是副作用
    "*.css",
    "./src/setup-tracking.ts"
  ]
}

让 bundler 知道你的 app 哪些文件有副作用。其他文件可以激进 tree shake。

三大 bundler tree shaking 对比

Rollup：最严最干净

Rollup 把 tree shaking 视为第一公民：

  • 默认开启 + 激进
  • 不依赖 sideEffects 标注（也用 /* @__PURE__ */ 注释）
  • 能识别"调用没副作用的函数"模式（如 const x = createObj()，如果 createObj 标了 PURE，x 没用就 shake 掉 createObj）
  • scope hoisting（把所有模块代码合并到同一作用域）让 minifier 进一步删

结果：Rollup 输出通常最干净。库打包用 Rollup 是行业标准。

Webpack：依赖标注 + Babel

Webpack tree shaking：

  • 必须 production mode 才完全开启（dev 模式不 shake）
  • 严重依赖 sideEffects 标注—— 没标的包保守保留
  • 依赖 Babel /* @__PURE__ */ 注释（Babel preset-env 会丢掉这些注释，要小心）
  • concatenateModules（scope hoisting）2018 后改善

结果：常有假阳性—— 该 shake 的没 shake 掉，bundle 比 Rollup 大。

esbuild：中等严格

esbuild tree shaking：

  • 默认开启
  • 识别 sideEffects 标注
  • 识别 /* @__PURE__ */
  • 速度快但不及 Rollup 精细

结果：接近 Rollup 但不完全等价。

静态分析的 4 个限制

限制 1：动态 import 不能完全分析

// 路径完全动态
const mod = await import(getUserInput());   // bundler 不知道
// → 不能 tree shake 任何模块（保守保留所有可能性）

对策：不要用完全动态 import；用 glob pattern 让 bundler 至少能限制范围。

限制 2：调用未知函数（可能改全局）保守

// utils.js
export function init() { /* 可能改全局 */ }

// app.js
import { init } from './utils';
init();   // bundler 不确定 init 是否有副作用 → 保守保留

对策：标 /* @__PURE__ */：

/*#__PURE__*/ init();   // 告诉 tree shaker：这个调用纯，删了 OK

限制 3：re-export 扩展分析图

// utils/index.js
export * from './a';
export * from './b';
export * from './c';

// app.js
import { foo } from 'utils';   // bundler 必须分析 a/b/c 都看一遍找 foo

这本身 OK，但大型 barrel file（index.ts 一次 export 几百个）让 bundler 扫描范围大；某些 bundler / 配置下退化到"全部保留"。

限制 4：destructuring import 必要不充分

// 表面上：用了 add，没用 subtract
import { add } from 'math';

// 实际上：bundler 还要分析 math.js 内部所有代码确认无副作用
// 如果 math.js 内部有 console.log 或调用未知函数，保留所有

典型失效场景

失效 1：默认导入整个 lodash

// ❌ 整个 lodash 进来（CJS 包，~70KB）
import _ from 'lodash';
const r = _.debounce(fn);

// ✅ 只 debounce
import { debounce } from 'lodash-es';
const r = debounce(fn);

// ✅ 或单文件 import（更激进）
import debounce from 'lodash/debounce';   // 仍 CJS 但 lodash 设计支持单文件 import

失效 2：第三方包没标 sideEffects

// 包 X 没标 sideEffects
import { foo } from 'pkg-x';   // bundler 保守保留整个 pkg-x

// 临时对策：用户项目 package.json 强行标
{
  "sideEffects": ["./node_modules/pkg-x/some-side-effect-file.js"]
}

失效 3：CSS 副作用

// 这是副作用（注入 DOM），不能 tree shake
import 'global-styles.css';
import './side-effect.scss';

包作者要标：

{
  "sideEffects": ["*.css", "*.scss"]
}

失效 4：模块顶层 console.log

// foo.js
console.log('foo loaded');   // ← 让整个模块被认为有副作用

export function bar() { /* ... */ }

// app.js
import { bar } from './foo';   // bar 被保留，foo.js 整个加载

对策：删掉无意义的 console.log；或包装在条件里。

失效 5：Babel 转译破坏 ESM 静态结构

Babel preset-env 默认把 ESM 转 CJS（兼容旧浏览器）。转 CJS 后 tree shake 不动。

对策：

// babel.config.js
{
  "presets": [
    ["@babel/preset-env", {
      "modules": false   // 不转 ESM 模块
    }]
  ]
}

让 Babel 保留 ESM 给 bundler 处理。2026 用 SWC / esbuild 替代 Babel 通常默认正确。

失效 6：Class 字段 / private 等转译副作用

Babel 转 ES2022 私有字段时可能加 helper：

// 源
class Foo {
  #count = 0;
  inc() { this.#count++; }
}

// Babel 转译后（简化）
class Foo {
  constructor() {
    _classPrivateFieldInitSpec(this, _count, { writable: true, value: 0 });
  }
  inc() { /* 用 helper */ }
}
function _classPrivateFieldInitSpec() { /* ... */ }   // 全局 helper

helper 看起来副作用 → tree shake 失效。

对策：

  • 用 SWC / esbuild（不用 Babel preset-env）
  • target 现代浏览器（不需要转译）
  • Babel 的 @babel/runtime 配 sideEffects: false

关键工具：/* @__PURE__ */ 注释

这是跨 bundler 标准：标在函数调用前，告诉 tree shaker"这调用纯，删了 OK"：

// 告诉 tree shaker：createComponent 是 pure 的
const Component = /*#__PURE__*/ createComponent({ name: 'Foo' });

// 没用 Component 时整个被删（包括 createComponent 调用）

// 多个调用都标
export const cx = /*#__PURE__*/ createCx();
export const cy = /*#__PURE__*/ createCy();

注意：

  • 注释格式严格：/*#__PURE__*/ 或 /* @__PURE__ */
  • 必须紧贴函数调用前面
  • Babel preset-env 默认会丢掉注释（要 shouldPrintComment 选项保留）

诊断工具

rollup-plugin-visualizer

$ npm i -D rollup-plugin-visualizer
// vite.config.js
import { visualizer } from 'rollup-plugin-visualizer';

export default {
  plugins: [react(), visualizer({ open: true, gzipSize: true })]
};
$ vite build
# 自动打开浏览器，treemap 可视化每个 chunk 内容

source-map-explorer（Webpack 友好）

$ npm i -g source-map-explorer
$ source-map-explorer dist/assets/index-*.js
# 浏览器打开，treemap 可视化

esbuild metafile

$ esbuild src/index.js --bundle --metafile=meta.json --outfile=out.js
$ cat meta.json | jq '.outputs'
# 看每个 output 的 inputs / imports / bytes

vite build --debug bundle:

$ DEBUG="vite:*" vite build 2>&1 | grep "tree-shake"

03 应用场景

场景 A · 我开了 tree shaking 但 bundle 还很大

诊断流程：

步骤 1：可视化 bundle 内容—— 用 visualizer。看哪些大模块

步骤 2：识别意外模块—— 比如发现 lodash 整个进来 + moment.js 完整 + 等。

步骤 3：检查包是 ESM 还是 CJS：

$ cat node_modules/lodash/package.json | jq '.type'
null   # CJS（默认）

步骤 4：换 ESM 版：

  • lodash → lodash-es
  • moment → date-fns / dayjs
  • chart.js（CJS）→ chart.js（ESM build，2024+）

步骤 5：检查包 sideEffects：

$ cat node_modules/some-pkg/package.json | jq '.sideEffects'
true   # 默认（保守保留）

对策：项目 package.json 强标 sideEffects；或开 issue 让包作者标。

场景 B · 写库时 tree shake 友好

包作者最佳实践：

1. 用 ESM 输出（"type": "module" 或 exports.import）

2. 标 sideEffects：

{
  "sideEffects": false   // 大多数库没副作用
}
// 或
{
  "sideEffects": ["*.css", "./src/polyfill.js"]
}

3. 分散导出（不要 barrel file）：

// package.json
{
  "exports": {
    ".": "./dist/index.js",
    "./debounce": "./dist/debounce.js",
    "./throttle": "./dist/throttle.js"
  }
}

4. 标 PURE 函数：

export const Component = /*#__PURE__*/ createComponent({ ... });

5. 测试：用 publint / are-the-types-wrong / 实际接进 demo 项目验证 tree shake。

场景 C · 修复 Babel 破坏 tree shaking

检查 Babel 配置：

// babel.config.js (问题)
{
  "presets": ["@babel/preset-env"]   // 默认转 CJS
}

修复：

// babel.config.js
{
  "presets": [
    ["@babel/preset-env", {
      "modules": false,            // 保留 ESM
      "targets": "> 0.25%, not dead"
    }]
  ]
}

或换 SWC / esbuild（默认行为正确）。

场景 D · 用 sideEffects 让 polyfill 仍工作

// src/polyfills/index.js
import 'core-js/features/promise';
import 'core-js/features/array';

// app.js
import './polyfills';   // 仅副作用 import
// package.json
{
  "sideEffects": ["./src/polyfills/**"]
}

这让 bundler 知道：polyfills 文件夹里的代码必须保留；其他可以 tree shake。

场景 E · 测试 tree shake 是否生效

简单实验：

// test.ts
import { specificFunc } from 'lib-x';
console.log(specificFunc);
$ vite build
# 看 dist/ 里 chunk 是否只含 specificFunc 还是整个 lib-x

如果整个 lib-x 都进来 → tree shake 失败 → 调查包配置。

04 典型代码：完整示例

检测包是否 tree-shake-friendly

$ cat node_modules/some-pkg/package.json | jq '{type, sideEffects, exports, main, module}'

判定：

  • "type": "module" 或 "exports.import" 存在 → ESM ✅
  • "sideEffects": false 或 数组 → 标了 ✅
  • "sideEffects": null（默认）→ 没标 ⚠️

用 unbuild 发 tree-shake-friendly 库

$ npm i -D unbuild
// build.config.ts
import { defineBuildConfig } from 'unbuild';

export default defineBuildConfig({
  entries: ['./src/index'],
  declaration: true,
  rollup: {
    emitCJS: true,
    inlineDependencies: false
  }
});
// package.json
{
  "name": "my-pkg",
  "type": "module",
  "sideEffects": false,
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  },
  "scripts": { "build": "unbuild" }
}

用 PURE 注释优化 createSlice 等工厂函数

// store/slices/user.ts
import { createSlice } from '@reduxjs/toolkit';

// ❌ tree shaker 不确定 createSlice 是否有副作用
export const userSlice = createSlice({ ... });

// ✅ 标 PURE 让 tree shake 工作
export const userSlice = /*#__PURE__*/ createSlice({ ... });

Redux Toolkit 4+ 自动标 PURE，但旧代码 / 自定义工厂要手动。

诊断真实 bundle 内容

// vite.config.js
import { visualizer } from 'rollup-plugin-visualizer';

export default {
  plugins: [
    react(),
    visualizer({
      open: true,
      template: 'sunburst',   // 或 'treemap', 'network'
      gzipSize: true,
      brotliSize: true,
      filename: 'dist/stats.html'
    })
  ]
};
$ vite build
# 自动打开 dist/stats.html
# 浏览器看每个文件大小（gzip 后）+ tree shaking 结果

检查 Rollup 是否成功 tree shake

// vite.config.ts
export default {
  build: {
    rollupOptions: {
      onLog(level, log) {
        if (log.code === 'TREESHAKING_ANNOTATION') {
          console.log('Tree shaking note:', log.message);
        }
      }
    }
  }
};

05 常见陷阱

陷阱 1：以为 tree shaking 是开关

常见误解。不是开关，是整个生态共同努力：

  1. 包作者用 ESM + 标 sideEffects + 标 PURE
  2. 编译器（Babel/SWC）不破坏 ESM 静态结构
  3. bundler（Rollup/Webpack/esbuild）正确实现
  4. 用户写 ESM-friendly 代码（细粒度 import）

任何一环失效 → tree shake 失败。

陷阱 2：用 Babel preset-env 默认配置

preset-env 默认 modules: "auto"，环境是 Node 时转 CJS → 给 bundler 是 CJS → tree shake 不动。

修复：modules: false 让 Babel 保留 ESM。

陷阱 3：以为 lodash 单文件 import 等同 lodash-es

// 单文件 import：减少了进入的代码
import debounce from 'lodash/debounce';
// 但 lodash 内部仍是 CJS；bundler 还是要做 CJS-ESM interop
// tree shaking 比 lodash-es 略差

// lodash-es：纯 ESM
import { debounce } from 'lodash-es';
// tree shaking 最优

规则：能用 lodash-es 就用；不能（CJS 项目）退而求其次用单文件。

陷阱 4：barrel file 导致大 bundle

// utils/index.ts (barrel)
export * from './a';
export * from './b';
export * from './c';
export * from './d';
// ... 100 个

// 用户
import { something } from 'utils';

问题：bundler 要分析所有 100 个文件找 something。性能差 + 某些 bundler tree shake 失效。

对策：

  • 从源头单独 import：import { something } from 'utils/something'
  • 包导出用 exports 字段精细控制 subpath

陷阱 5：SSR / Webpack 5 magic comments 失效

Webpack 5 / Vite SSR 模式有时 tree shake 不及预期 —— 因为 SSR 输出是 CJS / 含 dynamic 路径 / 等。

对策：CI 检查 SSR bundle 大小 + 可视化分析。

陷阱 6：TS enum 比预期大

enum Status { Open, Closed, Pending }
// 编译后包含整个 enum 定义（双向映射）

// 用 const enum 让 inline
const enum Status { Open, Closed, Pending }
// Status.Open → 0（直接替换）

// 或用 union type
type Status = 'open' | 'closed' | 'pending';

陷阱 7：CSS / @import 不能 tree shake

CSS 没有"未使用 class shake"的概念（PurgeCSS / Tailwind JIT 是不同机制）。@import 引入的样式都被打入 bundle。

用 PurgeCSS / Tailwind v4 JIT 让没用的 class 不进 bundle（不是 ESM tree shaking，是 CSS-specific 优化）。

陷阱 8：Webpack mode "development" 不 shake

Webpack mode: "development" 默认不开 tree shaking（dev 不需要）。

// 检查 production 实际 shake 情况
$ NODE_ENV=production webpack --mode production

Vite 的 vite build 默认 production mode，自动开。
