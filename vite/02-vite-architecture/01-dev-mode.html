Vite Dev mode：esbuild 预构建 + 浏览器原生 ESM

Vite 设计核心。核心 idea：浏览器原生支持 ESM，dev mode 不打包 —— 让浏览器请求每个 module 文件，server on-demand 转换返回。关键洞察：Vite 快不是因为 esbuild 快，是因为不全量打包。esbuild 只做"依赖预构建"（把 node_modules 里散文件 / CJS 转换成单 ESM 文件，避免浏览器发几百个请求）和"逐文件 transform"（TS / Vue SFC / JSX 转 JS）。架构层面，1000 文件的项目和 100 文件几乎一样快。

阅读时长 ~30 分钟 前置 P1（构建工具是谁）+ 浏览器渲染 P3.1（HTML parser + 资源加载） 主要来源 Vite 官方文档 · 尤雨溪演讲 · vite/packages/vite 源码

01 速览

  1. Vite dev mode 不打包源代码—— 浏览器原生支持 ESM，让 dev server 可以让浏览器自己解析 import 链。Vite server 只做"按浏览器请求逐个转换文件"。
  2. node_modules 依赖必须预构建—— 因为：(1) node_modules 里很多包是 CJS（浏览器只懂 ESM）；(2) 一个包可能有几百个散文件（浏览器一个 import 一个 HTTP 请求会爆）。Vite 用 esbuild 把每个依赖预 bundle 成单个 ESM 文件。
  3. 缓存机制：.vite/ 目录 + hash invalidation—— 预构建结果存到 node_modules/.vite/；hash 基于 lock file + 配置；package.json 改了才重构。
  4. on-demand transform—— 源代码（src/）按浏览器请求逐个转换：TS → JS / Vue SFC → JS / JSX → JS。用 esbuild 做 transform（不做 bundle）。
  5. import 重写—— 浏览器只接受 absolute / relative URL；Vite server 把 import 'react' 重写为 /node_modules/.vite/deps/react.js?v=hash 等。同时处理 .css / .svg 等资源 import。
  6. "为什么 Vite 快"的真正答案：不全量打包整个项目。即使把 esbuild 换成更慢的 bundler，启动也仍然快——大项目 1000 文件和小项目 100 文件几乎一样快，因为不全量解析依赖图。esbuild 速度只影响"预构建那几百毫秒"。

02 底层逻辑

核心架构：dev server 不打包

传统 Webpack dev mode 流程：

用户 npm run dev
  ↓
Webpack 启动
  ↓
解析整个项目依赖图（所有 src/ + node_modules）
  ↓
全量打包成 bundle.js（可能 50-200MB）
  ↓
启动 webpack-dev-server，serve bundle.js
  ↓
浏览器加载 bundle.js（第一次几秒）
  ↓
源代码改 → Webpack 增量重 build → push 给浏览器

Vite dev mode 流程：

用户 vite
  ↓
Vite 启动（< 100ms）
  ↓
扫描 + 预构建 node_modules 依赖（esbuild，~几百 ms 首次 / 几十 ms 缓存命中）
  ↓
启动 server，监听文件变化（chokidar）
  ↓
浏览器请求 /index.html → server 返回带 <script type="module">
  ↓
浏览器解析 main.js → 发现 import './App.tsx' → 请求 /App.tsx
  ↓
Vite server: 编译 App.tsx（TS → JS） + 重写 import → 返回
  ↓
浏览器继续解析 App.tsx 里的 import → 继续请求...
  ↓
源代码改 → Vite 编译那一个文件 → WebSocket push 给浏览器 → HMR

关键差异：

  • Webpack：启动时全量解析 + 打包整个项目
  • Vite：启动时只预构建 node_modules + 等浏览器请求

依赖预构建（dependency pre-bundling）

预构建是 Vite dev 中唯一"打包"步骤。它做两件事：

原因 1：CJS → ESM 转换

node_modules 里很多包仍是 CJS：

// react 4-5 仍是 CJS（react-dom 同样）
// node_modules/react/index.js
'use strict';
if (process.env.NODE_ENV === 'production') {
  module.exports = require('./cjs/react.production.min.js');
} else {
  module.exports = require('./cjs/react.development.js');
}

浏览器只懂 ESM（import / export）。预构建时 esbuild 把 CJS 转 ESM：

// node_modules/.vite/deps/react.js（预构建后）
import { ... } from './chunk-XXX.js';
export default ReactExport;
export { useState, useEffect, ... };

原因 2：散文件 → 单文件

很多包内部有几百个散文件。没有预构建，浏览器解析每个 import 都发 HTTP 请求：

// 例：lodash-es 有 600+ 个文件
import { debounce } from 'lodash-es';
// 浏览器：发 1 个请求拿到 lodash-es/debounce.js
// debounce.js 里 import './_baseDebounce' → 发 1 个请求...
// _baseDebounce.js 里 import './...' → 继续...
// 总计可能几百个 HTTP 请求！dev 启动巨慢

预构建后：

// node_modules/.vite/deps/lodash-es.js（单个文件，所有依赖打包到一起）
浏览器：1 个请求拿到 lodash-es 全部

这是 Vite dev 启动快的关键 —— 不是 esbuild 多么快，是把"几百个 HTTP 请求" 减成 "几个 HTTP 请求"。

预构建实现

Vite 启动时：

  1. 扫描入口文件（默认 index.html）→ 发现所有 import
  2. 分类：node_modules 里的（要预构建）vs src/ 里的（不预构建）
  3. 用 esbuild 把每个 node_modules 入口 bundle 成单个 ESM 文件
  4. 输出到 node_modules/.vite/deps/
  5. 计算依赖 hash（基于 lock file + 配置）→ 写入 node_modules/.vite/deps/_metadata.json

典型预构建产物：

node_modules/.vite/deps/
├── _metadata.json
├── react.js                  ← 单文件 ESM
├── react-dom.js
├── react-dom_client.js
├── lodash-es.js
└── chunk-XXX.js              ← 共享 chunk（多个依赖共用）

缓存机制

预构建有几百毫秒成本（首次）。Vite 缓存策略：

  • cache key = lock file content + Vite config + 入口列表的 hash
  • cache 位置：node_modules/.vite/deps/_metadata.json
  • 失效条件：lock file 变 / config 变 / 入口加新依赖
  • 缓存命中：直接 mmap 读 .vite/deps/ 文件，跳过 esbuild bundling

实测：首次 ~500ms-2s（依项目）；缓存命中 < 100ms。

on-demand transform：源代码按需编译

src/ 里的文件不预构建。它们由 Vite server 按浏览器请求逐个编译：

浏览器请求 GET /src/App.tsx
  ↓
Vite server middleware 链：
  → resolve plugin: 解析路径
  → load plugin: 读文件内容
  → transform plugin: 编译 TS → JS（用 esbuild）
  → transform plugin: 重写 import 路径
  → 返回 JS 给浏览器

每个文件只在首次请求时编译；二次请求 + 内存缓存命中。

import 重写

浏览器原生 ESM 只支持绝对 URL 或相对路径：

  • import './foo.js' ✅ 相对
  • import '/foo.js' ✅ 绝对
  • import 'react' ❌ bare specifier，浏览器不知道去哪找

Vite server 在 transform 阶段重写 import：

// 源代码
import React from 'react';
import { debounce } from 'lodash-es';
import './style.css';

// Vite server 重写后返回
import React from '/node_modules/.vite/deps/react.js?v=abc123';
import { debounce } from '/node_modules/.vite/deps/lodash-es.js?v=abc123';
import '/src/style.css?direct';   // CSS 也是 module（HMR 用）

?v=abc123 是 cache busting：依赖 hash 变了 query 也变，让浏览器 invalidate cache。

CSS / 静态资源处理

Vite 把 CSS / image / font 等都当 ESM module：

import './style.css';        // 注入 CSS 到 DOM
import logo from './logo.png'; // logo 是 URL string
import data from './data.json' with { type: 'json' };  // import attributes

CSS 处理特别：

// 浏览器接收的 CSS 实际是 JS module
// vite 把 CSS 包装成：
const css = ".foo { color: red; }";
const style = document.createElement('style');
style.textContent = css;
document.head.appendChild(style);
export default css;
// HMR 时直接替换 style 元素

HTTP 请求模型与 HTTP/2

"Vite 让浏览器发几百个请求"听起来差。但HTTP/2 / HTTP/3 多路复用让小请求高效：

  • HTTP/1.1：每个请求要独立 TCP 连接（max ~6 并发）/ head-of-line blocking
  • HTTP/2：一个 TCP 连接多路复用 N 个请求；几百个请求几乎免费
  • 浏览器 cache：二次访问几乎零 HTTP 开销

这也是 Vite 设计的时代背景 —— 没有 HTTP/2 普及（~2018+），Vite 设计可能不可行。

性能数据：为什么 < 1s 启动

典型 Vite dev 启动分解：

步骤时间
Vite server 启动~50ms
读 vite.config.js + 解析配置~50ms
预构建（缓存命中）~100ms
预构建（首次，10-20 个依赖）~300-1000ms
启动 chokidar（文件监听）~50ms
浏览器加载 + 第一次解析取决于网络 / CPU

对比 Webpack：

  • Webpack 启动需要扫描 + 解析 + 编译所有源文件 + 所有依赖
  • 1000 文件的 React 项目典型 30-60s
  • 大项目（10k+ 文件）可能几分钟甚至 OOM

大型项目可扩展性

因为 Vite dev 不全量打包，启动时间几乎与项目大小无关：

项目大小Webpack dev startVite dev start
100 文件~3-5s~500ms
1000 文件~15-30s~700ms
5000 文件~60s+~1s
10000 文件OOM 常见~1.5s

Vite 启动时间增长曲线 = O(node_modules 数量)，几乎不依赖 src/ 文件数。

SSR + RSC：Environment API（Vite 6+）

Vite 6（2024-11）引入 Environment API，让 SSR / RSC / Edge / Worker 等不同环境跑同一个 Vite：

// vite.config.js
export default {
  environments: {
    client: { /* 浏览器环境 */ },
    ssr: { /* Node SSR */ },
    rsc: { /* React Server Components */ },
    edge: { /* Cloudflare Workers */ }
  }
};

每个 environment 独立编译流水线 + 不同 transform 规则。这让 React Router 7 / TanStack Start / Astro 等同时支持多 runtime 的框架更易实现。

03 应用场景

场景 A · 启动时间慢的诊断

Vite dev 启动慢通常 3 类原因：

(1) 预构建慢—— 依赖太多 / 包结构不友好（CJS / 散文件）。Vite 启动日志看：Pre-bundling dependencies (...) 那段时间。

对策：检查依赖是否必要；用 ESM 版本（lodash-es 而不是 lodash）；optimizeDeps.include / exclude 调整。

(2) 第一次浏览器加载慢—— 首页 import 链深 + 文件多 → 浏览器要请求很多。

对策：optimizeDeps.entries 显式声明入口；vite.config.js server.warmup 让 server 预热源文件。

(3) 监听文件慢—— 大项目的 chokidar 启动慢；macOS 上 fsevents 偶尔有问题。

对策：server.watch.usePolling 调整；忽略不必要目录。

场景 B · CJS-only 依赖让 dev 报错

有时第三方包是 CJS only，没正确标注 exports；预构建可能失败：

[vite] Internal server error: 'foo' is not exported by node_modules/cjs-only-pkg/index.js

解决：

// vite.config.js
export default {
  optimizeDeps: {
    include: ['cjs-only-pkg'],   // 强制预构建
  }
};

场景 C · 大型 monorepo 性能优化

monorepo 里 packages/ 多个内部包都是源代码（不在 node_modules）。Vite 默认不预构建源代码，每次都 transform。大型 monorepo 这会让浏览器请求几千文件。

解决：

// vite.config.js
export default {
  optimizeDeps: {
    // 强制预构建内部包
    include: ['@my-org/ui', '@my-org/utils']
  }
};

场景 D · proxy 后端 API（dev 期）

// vite.config.js
export default {
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3001',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, '')
      }
    }
  }
};

所有 /api/* 请求转发到 backend，避免 CORS。

场景 E · Vite 6 Environment API（SSR + 客户端）

import { createServer } from 'vite';

const server = await createServer({
  environments: {
    client: {
      build: { outDir: 'dist/client' }
    },
    ssr: {
      build: { outDir: 'dist/ssr', rollupOptions: { input: 'src/entry-server.ts' } }
    }
  }
});

await server.listen();
// dev 时同时跑 client + ssr 两个 environment

04 典型代码：完整 Vite 项目

最简 vite 项目结构

my-app/
├── package.json
├── vite.config.js
├── index.html              ← 入口（不是 src/index.html）
├── src/
│   ├── main.tsx
│   ├── App.tsx
│   └── style.css
└── public/                 ← 静态资源（不经 import 直接复制）
    └── favicon.ico

index.html（关键）

<!doctype html>
<html>
  <head>
    <meta charset="UTF-8" />
    <link rel="icon" type="image/svg+xml" href="/favicon.svg" />
    <title>My App</title>
  </head>
  <body>
    <div id="root"></div>
    <script type="module" src="/src/main.tsx"></script>
  </body>
</html>

注意：

  • type="module" 是关键 —— 让浏览器以 ESM 方式解析
  • 路径 /src/main.tsx 是绝对路径 —— Vite server 会从项目根目录 serve
  • HTML 在项目根，不在 public/（与 CRA 不同）

vite.config.ts

import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],

  // dev server
  server: {
    port: 3000,
    open: true,                  // 启动后自动打开浏览器
    proxy: {
      '/api': 'http://localhost:3001'
    }
  },

  // 优化预构建
  optimizeDeps: {
    include: ['react', 'react-dom', 'lodash-es'],
    exclude: ['some-cjs-pkg-that-breaks']
  },

  // 构建（详见 P2.2）
  build: {
    outDir: 'dist',
    target: 'es2020'
  }
});

src/main.tsx

import React from 'react';
import { createRoot } from 'react-dom/client';
import App from './App';
import './style.css';

const root = createRoot(document.getElementById('root')!);
root.render(<App />);

用 import.meta.env 区分环境

// Vite 内置 env 变量
console.log(import.meta.env.MODE);          // 'development' | 'production'
console.log(import.meta.env.DEV);           // true | false
console.log(import.meta.env.PROD);          // true | false
console.log(import.meta.env.BASE_URL);      // 配置的 base 路径

// 自定义 env：以 VITE_ 开头才暴露给客户端
// .env.local
// VITE_API_URL=https://api.example.com

console.log(import.meta.env.VITE_API_URL);  // "https://api.example.com"

预构建状态检查

# 启动后看 .vite/deps 内容
$ ls node_modules/.vite/deps/
_metadata.json
chunk-XXX.js
react.js
react-dom_client.js
react-dom.js

# 看 metadata
$ cat node_modules/.vite/deps/_metadata.json | jq .
{
  "hash": "abc123",
  "browserHash": "def456",
  "optimized": {
    "react": { "src": "...", "file": "react.js", ... },
    ...
  }
}

清理预构建缓存（debug 用）

$ rm -rf node_modules/.vite
$ vite --force      # 启动时强制重新预构建

05 常见陷阱

陷阱 1：以为 Vite 快是因为 esbuild

常见误解。Vite 快的根本是不全量打包，esbuild 只是预构建那几百毫秒用得快。即使把 esbuild 替换成更慢的 bundler 做预构建，启动时间仍比 Webpack 快 10-50x —— 因为预构建只处理依赖（数十个），Webpack 全量打包整个项目（数千文件）。

陷阱 2：用 require / __dirname / process.env 期望工作

Vite dev mode 是浏览器 ESM 环境，不是 Node：

// ❌ 这些在 dev 中报错
const fs = require('fs');                    // 浏览器没 require
console.log(__dirname);                      // 浏览器没 __dirname
console.log(process.env.NODE_ENV);           // 浏览器无 process

// ✅ 浏览器原生 ESM 用：
import.meta.url                              // 当前模块 URL
new URL('./asset.png', import.meta.url)      // 资源相对路径
import.meta.env.NODE_ENV                     // Vite 注入的 env
import.meta.env.DEV                          // dev 标志

陷阱 3：依赖里 import 浏览器不支持的 Node API

有些 npm 包（如 fs-extra / path）是 Node-only。在浏览器 dev 用时报错：Module "fs" has been externalized for browser compatibility.

对策：

  • 用浏览器版替代（path-browserify 替代 path）
  • vite.config.js resolve.alias 重定向
  • 用 SSR 模式跑 Node 代码（Vite 6 Environment API）

陷阱 4：CommonJS 默认导出问题

CJS 包用 module.exports = obj；ESM 期望 export default obj。Vite 预构建做 interop，但有时不完美：

// 旧 CJS 包
import lodash from 'lodash';     // 可能拿到 { default: { ...lodash } }
// 实际访问要：lodash.default.debounce 而不是 lodash.debounce

// ✅ 用 ESM 版本
import { debounce } from 'lodash-es';

陷阱 5：用 CRA 风格 public 路径

<!-- ❌ CRA 风格（Vite 不支持） -->
<img src="%PUBLIC_URL%/logo.png" />

<!-- ✅ Vite 风格 -->
<img src="/logo.png" />       <!-- public/ 下的资源直接 / 路径 -->

陷阱 6：忘记 @vitejs/plugin-react / vue 插件

Vite 不内置 React/Vue 支持。必须装对应 plugin：

$ npm i -D @vitejs/plugin-react
# 或
$ npm i -D @vitejs/plugin-vue

然后 vite.config.js 加 plugin。否则 JSX / SFC 不能编译。

陷阱 7：optimizeDeps.include / exclude 误用

include 强制预构建（针对自动检测漏掉的）；exclude 强制不预构建（针对预构建出错的）。误用会导致诡异 bug：

  • 把纯 ESM 包放 exclude 没意义（Vite 不预构建它）
  • 把很重的包放 exclude 让浏览器发几百请求

规则：默认让 Vite 自动判断；只在明确遇到错误才调整。

陷阱 8：缓存不更新

诡异行为时清缓存：

$ rm -rf node_modules/.vite
$ vite --force

常见诱因：lock file 没更新但 package.json 改了；deps 锁定 hash 不一致。
