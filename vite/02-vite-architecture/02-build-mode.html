Vite Build mode：Rollup → Rolldown 接班

Vite 生产构建。回答"为什么 dev 用 esbuild，build 用 Rollup"：dev 求"启动快 + 单文件转换快"，build 求"代码体积小 + 优化彻底"——两个目标方向不同，所以工具不同。Rollup 4 优势：成熟 ESM 处理 / 优秀 tree shaking / 完善 plugin ecosystem / 优秀 sourcemap。Vite 在 Rollup 之上加：智能 chunk 分包策略 / CSS code splitting / 资源处理 / 默认 minify（terser）。Rolldown（Rust 实现的 Rollup 兼容 bundler）2026-Q2 stable 后接班 Vite build mode，速度 10x+ 提升但 plugin 兼容。

阅读时长 ~30 分钟 前置 P2.1（Dev mode）+ P1.1（项目史，特别是 Rollup 部分） 主要来源 Vite build docs · Rollup docs · Rolldown roadmap

01 速览

  1. "为什么 dev/build 不统一用一个工具"—— dev 求"启动快 + 单文件转换快"；build 求"代码体积小 + 优化彻底"。两个目标不同，工具自然不同。esbuild 适合前者，Rollup 适合后者。
  2. Build mode 仍要 bundle—— 生产代码要 chunk 分包 / tree shaking / 代码压缩 / sourcemap / 资源优化。这些 esbuild 做得不够好（plugin API 弱 / chunk 策略简单）。
  3. Vite 在 Rollup 之上加：智能 chunk 策略（vendor 分割 + manualChunks）/ CSS code splitting（每个 entry 独立 .css）/ 资源处理（小图 inline base64 / 大图 file + hash）/ 默认 minify（terser 或 esbuild）/ multi-page 支持。
  4. Chunk 分包策略—— 每个 dynamic import 一个 chunk + node_modules 自动 vendor 分割 + manualChunks 自定义。分包直接影响 LCP（详见 浏览器渲染 P4.1）。
  5. Rolldown 接班—— Vite team 自家 Rust bundler，Rollup plugin API 兼容。Vite 6 实验性集成（experimental.rolldown），Vite 7（预计 2026-Q2）默认。切换无痛，因为 plugin 不需改。
  6. "Bundle 大小决定首屏"是工程化最重要原则—— production app 的 LCP 直接受 bundle 大小影响（4G 网络下载 200KB 还是 2MB 差别巨大）。chunk 分包 + tree shaking + 资源优化是 LCP 优化第一关。

02 底层逻辑

为什么 dev/build 不统一

构建工具有两套工程目标：

维度Dev 目标Build 目标
速度启动快 + HMR 即时无所谓（CI 跑一次）
体积不重要（dev 不部署）越小越好（影响 LCP）
优化不需要 minify / tree shake必须最优化
调试需要 sourcemap + 不变换需要 sourcemap + 优化变换
format每文件单独 ESMchunk 合并 + 多 format（ESM/CJS/UMD）

esbuild 速度极致 —— 适合 dev "快速 transform"。但 plugin API 弱（不能 AST 改写）/ chunk 策略简单 —— 不适合 build "复杂优化"。

Rollup 慢但精细 —— ESM 第一公民 + tree shaking 干净 + 多 format 输出 + 优秀 plugin API —— 适合 build。

所以 Vite 选 dev=esbuild + build=Rollup。这是工程务实选择，不是 ideology。

Vite 在 Rollup 之上加的

Vite 不直接暴露 Rollup 给用户。它在 Rollup 之上做了几层增强：

1. 智能 chunk 策略

默认行为：

  • 每个 dynamic import 自动一个 chunk（lazy load 的基础）
  • node_modules 里的 import 自动分到 vendor chunk（让 user code 改了不让 vendor cache 失效）
  • 共享代码自动 dedupe（多个 chunk 共用的 module 提取出来）

用户可自定义：

// vite.config.js
export default {
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          'react-vendor': ['react', 'react-dom'],
          'ui-vendor': ['@mui/material', '@emotion/react'],
          'utils': ['lodash-es', 'date-fns']
        }
      }
    }
  }
};

2. CSS code splitting

每个 entry 的 CSS 独立 .css 文件 + hash filename。让 CSS 也可缓存 + 按需加载。

原理：Vite 把 CSS import 变成 module；build 时把所有 CSS module 收集 → 按 entry 切分 → 输出 .css 文件。HTML 里自动注入 <link>。

3. 资源处理

图片 / 字体 / 视频等：

  • < 4KB 默认 inline base64（节省 HTTP 请求）
  • ≥ 4KB 输出独立 file + content hash（让 long-term cache）
  • SVG 可以选择 inline 或 file
// vite.config.js
export default {
  build: {
    assetsInlineLimit: 4096   // 默认 4096 bytes
  }
};

4. minify 默认开

Vite 默认用 esbuild minify（不是 terser）—— esbuild 比 terser 快 50x，效果接近。

用户可改成 terser（更激进的优化但慢）：

// vite.config.js
export default {
  build: {
    minify: 'terser',   // 'terser' | 'esbuild' | false
    terserOptions: {
      compress: { drop_console: true }
    }
  }
};

5. multi-page 支持

大型项目多个 HTML 入口：

// vite.config.js
export default {
  build: {
    rollupOptions: {
      input: {
        main: 'index.html',
        admin: 'admin/index.html',
        nested: 'nested/page.html'
      }
    }
  }
};

Chunk 分包策略详解

分包目标：缓存友好 + 按需加载 + 平衡大小。

策略 1：自动 vendor 分割

Vite 默认把 node_modules 里的代码放到独立 chunk：

dist/
├── assets/
│   ├── index-abc123.js    ← user code (~50KB)
│   └── vendor-def456.js   ← node_modules (~150KB)

用户改 src/ 代码 → user chunk hash 变；vendor chunk hash 不变 → 浏览器仍用 cache。缓存命中率高。

策略 2：dynamic import = 独立 chunk

典型 React lazy 路由：

const Home = lazy(() => import('./pages/Home'));
const About = lazy(() => import('./pages/About'));
const Profile = lazy(() => import('./pages/Profile'));

Vite 自动把每个 dynamic import 放到独立 chunk：

dist/assets/
├── index-abc.js
├── vendor-def.js
├── Home-ghi.js          ← lazy load
├── About-jkl.js         ← lazy load
└── Profile-mno.js       ← lazy load

首屏只下载 index + vendor + 当前 route 对应 chunk。其他 route 在用户访问时按需 fetch。

策略 3：manualChunks 精细控制

大型项目可能要更精细：

build: {
  rollupOptions: {
    output: {
      manualChunks(id) {
        if (id.includes('node_modules/react')) return 'react';
        if (id.includes('node_modules/lodash')) return 'lodash';
        if (id.includes('node_modules/@mui')) return 'mui';
        if (id.includes('node_modules')) return 'vendor';
      }
    }
  }
}

这让大依赖（react / mui / lodash）独立 chunk，change 频率不同的依赖不互相影响 cache。

策略 4：Vite 6 智能默认（experimental）

Vite 6+ 实验性引入 build.rollupOptions.output.experimentalMinChunkSize：自动合并过小的 chunk（避免 chunk 太碎产生过多 HTTP 请求）。

构建产物结构

dist/
├── index.html                       ← 入口（自动注入 chunk link）
├── assets/
│   ├── index-abc123.js              ← entry chunk
│   ├── index-abc123.js.map          ← sourcemap
│   ├── vendor-def456.js             ← node_modules
│   ├── vendor-def456.js.map
│   ├── Home-ghi789.js               ← lazy chunks
│   ├── About-jkl012.js
│   ├── index-mno345.css             ← entry CSS
│   ├── logo-pqr678.png              ← 静态资源（hash filename）
│   └── font-stu901.woff2
└── favicon.ico                      ← public/ 内容直接复制

所有 .js / .css / 资源都带 content hash —— 让 HTTP cache 永久（Cache-Control: max-age=31536000, immutable）。

Rollup → Rolldown 接班路径

Rolldown 是 Vite team 2023-08 启动的项目，目标：Rust 实现的 Rollup 兼容 bundler。设计选择：

  • Rollup plugin API 兼容—— 用户切换时 plugin 不需改
  • Rust + 并行—— 速度目标 10x Rollup
  • Vite 深度集成—— Vite team 主导，与 Vite 协调演化

接班路径：

Vite 版本状态说明
Vite 5（2024-04）仅 Rollup稳定基线
Vite 6（2024-11）Rollup 默认 + Rolldown 实验性experimental.rolldown: true 切换
Vite 7（2026-Q2 预计）Rolldown 默认 + Rollup fallback用户主动 opt-out 才回 Rollup
Vite 8+（2027+）仅 RolldownRollup 移除

对用户的影响：几乎无感切换。Vite 6 用户可以现在加 experimental.rolldown 提前体验 build 加速。

性能数据

典型 React 项目（500 components, 50 dependencies）build 时间：

工具build 时间
Webpack 5~30-60s
Vite 6 (Rollup)~15-25s
Vite 6 (Rolldown 实验)~5-8s
Turbopack~10-15s
Bun build~3-6s
esbuild 直接~1-3s（但功能不全）

注：实测随项目而异。Bun build / esbuild 速度极致但功能 / plugin 不及 Vite。

Bundle 大小对 LCP 的影响

生产 app 的 LCP（详见 浏览器渲染 P4.1）直接受 bundle 大小影响。

4G 网络（~10Mbps）下载时间：

Bundle 大小（gzip 后）下载时间
50KB~50ms
200KB~200ms
500KB~500ms
1MB~1s
2MB~2s

LCP 目标 ≤ 2.5s。如果 bundle 1MB+ 且没分包 → 用户在 4G 上等 1+ 秒只看到白屏。chunk 分包让首屏只下载必要部分。

2026 工程实践：

  • 主 entry chunk（首屏代码）目标 < 100KB gzip
  • vendor chunk（react + 核心库）目标 < 200KB gzip
  • 其他 routes 用 dynamic import 懒加载
  • 大型库（图表 / 富文本编辑器）一定 lazy load

Library mode（库打包）

Vite 6 build 也能用 lib mode 打库（不打 app）：

// vite.config.js
export default {
  build: {
    lib: {
      entry: 'src/index.ts',
      name: 'MyLib',
      formats: ['es', 'cjs', 'umd', 'iife'],   // 多 format
      fileName: (format) => `index.${format}.js`
    },
    rollupOptions: {
      external: ['react', 'react-dom'],
      output: { globals: { react: 'React', 'react-dom': 'ReactDOM' }}
    }
  }
};

但纯库打包仍推荐直接用 Rollup（不套 Vite）—— Vite lib mode 是便利，不是首选。

03 应用场景

场景 A · 优化 LCP / 首屏速度

步骤：

(1) 检查当前 bundle 大小：用 rollup-plugin-visualizer：

$ npm i -D rollup-plugin-visualizer
// vite.config.js
import { visualizer } from 'rollup-plugin-visualizer';

export default {
  plugins: [react(), visualizer({ open: true })]
};
$ vite build
# 自动打开浏览器显示 stats.html，可视化 bundle 内容

(2) 找出大依赖—— 通常是图表库（recharts / chart.js）/ 富文本编辑器（tiptap / draft-js）/ 表单库 / 国际化（i18next + 全语言文件）/ moment.js 等。

(3) lazy load 重型组件：

const Chart = lazy(() => import('./Chart'));   // 用 React.lazy
const Editor = lazy(() => import('./Editor'));

(4) Tree shake 工具库：用 lodash-es 而不是 lodash；用 date-fns（每函数独立 import）而不是 moment.js。

(5) manual chunks 分大依赖：

manualChunks: {
  'recharts': ['recharts'],
  'editor': ['@tiptap/core', '@tiptap/react'],
}

(6) 定期检查 bundle 大小—— 加 size-limit 到 CI；超阈值 build 失败。

场景 B · 多页 app（admin + user 两套入口）

my-app/
├── index.html              ← 用户端
├── admin.html              ← 管理后台
└── src/
    ├── user/main.tsx
    └── admin/main.tsx
// vite.config.js
export default {
  build: {
    rollupOptions: {
      input: {
        user: resolve(__dirname, 'index.html'),
        admin: resolve(__dirname, 'admin.html')
      }
    }
  }
};

build 后两个独立 entry，各自的 chunk 完全独立 cache。

场景 C · 库打包发 npm

// vite.config.js
import { defineConfig } from 'vite';

export default defineConfig({
  build: {
    lib: {
      entry: './src/index.ts',
      name: 'my-lib',
      fileName: 'my-lib',
      formats: ['es', 'cjs']
    },
    rollupOptions: {
      external: ['react', 'react-dom'],
      output: {
        globals: { react: 'React', 'react-dom': 'ReactDOM' }
      }
    }
  }
});
// package.json
{
  "name": "my-lib",
  "type": "module",
  "main": "./dist/my-lib.cjs",
  "module": "./dist/my-lib.js",
  "exports": {
    ".": {
      "import": "./dist/my-lib.js",
      "require": "./dist/my-lib.cjs"
    }
  },
  "files": ["dist"]
}

场景 D · 提前体验 Rolldown

# Vite 6+ 实验性
$ npm i vite@latest
# vite.config.js 加 experimental 标志
// vite.config.js (Vite 6+)
export default {
  experimental: {
    rolldown: true
  }
};
$ vite build
# 用 Rolldown 而不是 Rollup

实测 build 时间显著降低（5-10x）但要注意：实验阶段，遇到问题报 issue 给 Rolldown。

场景 E · 静态资源版本化部署

// vite.config.js
export default {
  base: '/static/',                  // 部署到 /static/* 子路径
  build: {
    assetsDir: 'assets',             // 资源放 dist/static/assets/
    rollupOptions: {
      output: {
        // 自定义文件名 pattern
        entryFileNames: 'assets/[name]-[hash].js',
        chunkFileNames: 'assets/[name]-[hash].js',
        assetFileNames: 'assets/[name]-[hash][extname]'
      }
    }
  }
};

所有产物都带 hash → CDN 上 Cache-Control: max-age=31536000, immutable 永久缓存。

04 典型代码：完整 build 配置

生产级 vite.config.ts

import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import { visualizer } from 'rollup-plugin-visualizer';
import { resolve } from 'path';

export default defineConfig(({ mode }) => ({
  plugins: [
    react(),
    // 仅在 build 时生成 stats
    mode === 'production' && visualizer({
      filename: 'dist/stats.html',
      open: false,
      gzipSize: true,
      brotliSize: true
    })
  ].filter(Boolean),

  resolve: {
    alias: {
      '@': resolve(__dirname, './src')
    }
  },

  build: {
    target: 'es2020',
    outDir: 'dist',
    sourcemap: true,                 // 生产 sourcemap（部署到 Sentry）
    minify: 'esbuild',
    cssCodeSplit: true,
    chunkSizeWarningLimit: 500,      // chunk > 500KB 警告

    rollupOptions: {
      output: {
        manualChunks: {
          'react-vendor': ['react', 'react-dom', 'react-router-dom'],
          'utils': ['lodash-es', 'date-fns', 'zod'],
          'charts': ['recharts'],
          'editor': ['@tiptap/core', '@tiptap/react']
        },
        entryFileNames: 'assets/[name]-[hash].js',
        chunkFileNames: 'assets/chunks/[name]-[hash].js',
        assetFileNames: ({ name }) => {
          if (/\.(woff2?|ttf|eot)$/.test(name ?? '')) {
            return 'assets/fonts/[name]-[hash][extname]';
          }
          if (/\.(png|jpg|svg|webp)$/.test(name ?? '')) {
            return 'assets/images/[name]-[hash][extname]';
          }
          return 'assets/[name]-[hash][extname]';
        }
      }
    }
  }
}));

05 常见陷阱

陷阱 1：以为 dev 快 build 也快

Vite dev mode < 1s 启动，但 build mode 用 Rollup 仍慢（5-30s）。Webpack 用户切到 Vite 期望 build 也"飞快"会失望。

Vite 7 + Rolldown 后 build 加速 5-10x；现在可以提前用 experimental.rolldown。

陷阱 2：忘记 lazy load 大组件

新人常见错：所有 React 组件直接 import。结果 bundle 包含所有页面 / 所有图表。

修复：

// ❌ 不好
import Dashboard from './pages/Dashboard';
import Chart from './components/Chart';   // 几百 KB

// ✅ 好
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Chart = lazy(() => import('./components/Chart'));

陷阱 3：manualChunks 配置过细

分太多 chunk 让 HTTP 请求变多 + chunk overhead 累积反而拖慢。

好的分包数量级：5-15 个 chunk（比如 main + vendor + 几个大库 + 路由 chunks）。超过 50 个 chunk 通常过度优化。

陷阱 4：source map 暴露公网

生产 sourcemap 等于暴露源代码。除非 public open source，否则不要部署 .map 文件到 public CDN。

正确做法：

  • 生成 sourcemap → 上传到 Sentry / Datadog / 等错误追踪服务
  • 不部署 .map 到 public
  • 或者 build.sourcemap: 'hidden'（生成但不在 .js 里加 //# sourceMappingURL）

陷阱 5：忘记 "type": "module" 让 Node SSR 报错

Vite 6 SSR 输出 ESM；Node ESM 需要 package.json 标 "type": "module"。否则启动 SSR server 报错：SyntaxError: Cannot use import statement outside a module。

陷阱 6：assetsInlineLimit 默认值不适合所有场景

默认 4KB inline base64。某些场景：

  • 很多小 SVG icon → 都 inline 到 JS bundle，bundle 变大
  • 需要单独 cache 的资源 → inline 后无法独立 cache

调整：

build: {
  assetsInlineLimit: 0   // 全部独立 file
}
// 或
build: {
  assetsInlineLimit: (filePath) => {
    // 只 inline 小 SVG
    return filePath.endsWith('.svg') && getFileSize(filePath) < 2048;
  }
}

陷阱 7：dev 没问题 build 报错

常见。原因：

  • dev 用 esbuild transform（宽松）；build 用 Rollup（严格）
  • tree shaking 在 dev 不开，build 才开 → 某些 side effect 被删
  • minify 在 dev 不开，build 才开 → 某些反射代码失效

对策：CI 跑 build 而不是只跑 dev。预发环境用 production build。

陷阱 8：以为 esbuild minify 等于 terser

esbuild minify 速度极快但优化没 terser 激进。某些场景（极致体积优化）用 terser 仍然有意义：

build: {
  minify: 'terser',
  terserOptions: {
    compress: {
      drop_console: true,           // 删 console.log
      pure_funcs: ['console.log', 'debug'],
      passes: 2                     // 多遍优化
    },
    mangle: { toplevel: true }
  }
}

典型差距：terser 比 esbuild minify 多压缩 5-10%；但慢 50-100x。生产关键 app 值得；其他用 esbuild 即可。
