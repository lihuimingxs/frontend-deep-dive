下一代构建工具横评 + 选型

2026 选什么 bundler。这一章把 P1.2 同代人横评升级 —— 聚焦"2024+ 下一代"5 个 Rust/Zig bundler：Turbopack（Vercel, Tobias Koppers）/ Rspack（字节跳动）/ Rolldown（Vite team）/ Bun build（Bun team）/ Farm（中国社区）。每个绑定一个生态。选型决策树 + 2026-2030 趋势：Webpack 持续退场 / Vite + Rolldown 双标准 / Rust bundlers 占 90%+ 新项目 / Component Model 让 wasm + bundler 进一步集成。

阅读时长 ~30 分钟 前置 P1.2（同代人横评）+ P2-P6 全部 主要来源 各项目官方 + 性能 benchmark + Twitter / 博客

01 速览

  1. 5 个下一代 bundler 同时存在—— Turbopack / Rspack / Rolldown / Bun build / Farm。每个绑定一个生态：Turbopack→Next.js / Rspack→Webpack 兼容 / Rolldown→Vite + Rollup / Bun build→Bun runtime / Farm→Vite 兼容。独立用某个 Rust bundler 罕见。
  2. 实现语言—— 4 个 Rust（Turbopack/Rspack/Rolldown/Farm）+ 1 个 Zig（Bun build）。都比 JS bundler 快 5-50x。
  3. Turbopack（Vercel）—— Tobias Koppers（Webpack 原作者跳槽）主导；增量编译 + 共享缓存；Next.js 15+ production stable。独立用罕见，生态围绕 Next.js。
  4. Rspack（字节跳动）—— Webpack API 兼容；让旧项目"换 Rspack 即加速"。大型企业（字节自家 / 抖音 / TikTok）用。新项目不会主动选（Vite 更适合）。
  5. Rolldown（Vite team）—— Rollup plugin API 兼容；2026-Q2 stable 后接班 Vite build mode。Vite 用户几乎无感切换。也能独立用做库打包。
  6. Bun build—— 与 Bun runtime 深度集成；Zig 实现 + JSC parser 共享。极快。非 Bun 项目用不上。详见 Bun P4.3。
  7. Farm（中国社区）—— Vite plugin 兼容 + 比 Vite 快 5-10x（声称）。用户群较小但活跃。愿意尝试新工具的团队。
  8. 2026-2030 趋势—— Webpack 持续退场（仍有大量存量但增长停滞）/ Vite + Rolldown 成为 app dev/build 双标准 / Rust bundlers 占 90%+ 新项目 / Component Model 让 wasm + bundler 进一步集成（如 wasm 作为 build cache 后端）。

02 底层逻辑

Turbopack：Tobias Koppers 的"Webpack 重写"

背景：Tobias Koppers（Webpack 原作者，2014-2022 维护 Webpack 8 年）2022-10 跳槽 Vercel。Vercel 重金支持他做 Turbopack—— 用 Rust 重写一个"现代 Webpack"。

2022-11 alpha 发布时声称 "Webpack 700x 快"—— 但实际 benchmark 受争议（被指 misleading）。Vercel 2023 年低调修正叙述："增量编译性能领先"。

2024-10 Next.js 15 production stable —— Turbopack 第一次正式 production-ready。

关键设计：

(1) Rust 实现 + 高度并行—— 类似 esbuild Go，但 Rust 让内存管理更精细

(2) 增量编译—— Tobias 在 Webpack 时代的经验：增量是关键。Turbopack 用 Turbo 引擎（Vercel 自家）做函数级 memoization—— 每个编译步骤的输入输出缓存，相同输入直接返回缓存

(3) 共享缓存—— 多个项目共享构建缓存（experimental）；让 monorepo 第二个 build 几乎免费

(4) Next.js 深度集成—— RSC / Server Actions / App Router / Turbopack 是同一团队设计。Next 15 默认 turbo dev，turbo build 也 stable

限制：

  • plugin ecosystem 弱—— 不兼容 Webpack/Rollup plugin
  • 独立用罕见 —— 文档 / 示例都围绕 Next.js
  • 不开源（部分 Vercel 闭源）—— 社区贡献困难

典型用户：Next.js 15+ 项目（默认）。

Rspack：字节跳动的"Webpack Rust 兼容版"

背景：字节跳动 2022 启动 Rspack —— Webpack 项目太多（抖音 / TikTok / Lark / 飞书等），但 Webpack 慢。不能全部迁移 Vite（项目太大），所以做"API 兼容的 Rust 替代"。

2023-03 Rspack 0.1 / 2024 v1.0 stable。

关键设计：

(1) Webpack API 90%+ 兼容—— 大部分配置 / loader / plugin 直接用

(2) Rust 实现 + 极度优化—— 5-10x Webpack 速度

(3) 内置 SWC—— 转译用 SWC（不是 Babel）

(4) Module Federation 支持—— Webpack 独家特性，Rspack 也实现

典型迁移：

# 把 webpack 命令换成 rspack
$ npm uninstall webpack webpack-cli webpack-dev-server
$ npm install -D @rspack/core @rspack/cli @rspack/dev-server

# 配置只需改：
# webpack.config.js → rspack.config.js
# 大部分内容直接复制

启动 / build 速度立刻 5-10x。

限制：

  • 少数 Webpack plugin 不兼容（特别是用 internal API 的）
  • 新项目不会选（Vite 体验更好）
  • 仍带 Webpack 的"配置复杂"问题（继承 API）

典型用户：

  • 字节跳动自家（抖音 / TikTok / Lark / Lynx）
  • 大型企业 Webpack 项目，迁移到 Vite 成本太高
  • 需要 Module Federation 等 Webpack 独家特性

Rolldown：Vite team 自家继任 Rollup

背景：尤雨溪（Vite/Vue 作者）2023-08 启动 Rolldown。动机：

  • Vite build mode 用 Rollup，慢（JS 实现）
  • 切到 esbuild 不行（plugin 弱、不兼容 Rollup ecosystem）
  • 所以做Rust 实现的 Rollup 兼容版

关键设计：

(1) Rollup plugin API 100% 兼容—— 用户切换时 plugin 不需改

(2) Rust + 并行—— 速度目标 10x Rollup

(3) Vite 深度协调—— Vite team 主导，与 Vite 演化一致

接班路径：

Vite 版本状态
Vite 5仅 Rollup（稳定）
Vite 6（2024-11）Rollup 默认 + Rolldown 实验性（experimental.rolldown 切换）
Vite 7（2026-Q2 预计）Rolldown 默认 + Rollup fallback
Vite 8+（2027+）仅 Rolldown

对用户影响：

  • Vite 用户几乎无感切换
  • build 速度 5-10x 提升
  • plugin ecosystem 不变（Rollup plugin 直接用）

独立用 Rolldown：库打包替代 Rollup 也可（Rolldown CLI / config 与 Rollup 类似）。

Bun build：与 Bun runtime 深度集成

Bun build 是 Bun 自家 bundler。详见 Bun P4.3。简短：

  • Zig 实现（不是 Rust）
  • 与 Bun JSC parser 共享 → 速度极致
  • bun build CLI + Bun.build() JS API
  • Bun.serve 集成 dev server
  • plugin ecosystem 弱

典型用户：

  • Bun runtime 项目（npm install + bun run）
  • 追求极简的 demo / 工具
  • 不需要复杂 plugin

Farm：中国社区的 Rust bundler

背景：Farm（草根项目，主要中国维护者）2022 启动。动机：尝试做"比 Vite 更快的 Vite"。

关键设计：

  • Rust 实现
  • Vite plugin 兼容（部分）
  • 声称 5-10x Vite

状态（2026-05）：

  • v1.0 已发布（2024）
  • 用户群较小（vs Vite 主流）
  • 实测加速效果因项目而异（小项目差异不大）
  • 遇到问题求助资源少

典型用户：愿意尝试新工具 + 追求极致速度 + 不怕踩坑的团队。

选型决策树（详细版）

问题 1：你写的是什么？

  • npm 库 → Rollup（即将 Rolldown 接班）
  • Web app（任何 framework）→ Vite（默认）
  • Next.js app → Turbopack（Next 默认）
  • Bun runtime 应用 → Bun build
  • 纯 wasm 命令行工具 → esbuild 或自家脚本

问题 2：你已有什么？

  • 大型 Webpack 项目，启动慢急需解决，不想改 plugin → Rspack
  • 大型 Webpack 项目，能完全重写 → 迁 Vite
  • 稳定 Webpack 项目，速度可接受 → 不动
  • 已用 Vite 6 但 build 慢 → 试 experimental.rolldown

问题 3：性能优先级？

  • 启动最快 → Bun build / esbuild
  • build 最快 → Rolldown / Bun build
  • HMR 最快 → Vite + Turbopack 接近
  • 稳态 / 兼容性优先 → Vite + Rollup 路径

问题 4：团队偏好？

  • 主流 + 文档丰富 → Vite
  • 愿意尝试新工具 → Bun build / Farm / Rolldown 早采用
  • 需要 Webpack 兼容 → Rspack
  • 跟随 Vercel → Next.js + Turbopack

2026-2030 趋势预判

趋势 1：Webpack 持续退场但仍存活

  • 2026 仍有数百万 Webpack 项目
  • 新项目几乎不选 Webpack
  • Webpack 6 长期"路线图中"，没真正发布
  • 2030 预期：Webpack 仍在 legacy 项目，但新项目几乎归零

趋势 2：Vite + Rolldown 成为双标准

  • Vite 6+ 已是 app 默认；2026-Q2 Rolldown stable 让 build 也极快
  • Vue / Svelte / Solid / Astro / SvelteKit / Nuxt 都 Vite 优先
  • React 社区从 CRA 迁到 Vite 已大规模发生（2023+）
  • 2030 预期：Vite 占新项目 70%+（剩下是 Next.js + Turbopack）

趋势 3：Rust bundlers 占 90%+ 新项目

  • 所有主流 bundler 都已或正在 Rust 化
  • JS bundler 性能上限低（V8 + 单线程）
  • Bun build 的 Zig 是少数派但不可忽视
  • 2030 预期：JS 实现 bundler 几乎只剩 Rollup（被 Rolldown 接班路径中）

趋势 4：Component Model + bundler 集成

  • WASM Component Model（详见 WASM P7）让 wasm 成为"跨语言库格式"
  • 未来 bundler 可能：
  • - 用 wasm 作为 build cache 后端（共享缓存跨平台）
  • - bundler plugin 用 wasm 写（跨语言互操作）
  • - bundler 自身编译成 wasm 跑在浏览器（dev 完全在浏览器）
  • 2027+ 预期：实验阶段；2030+ 部分主流

趋势 5：build cache 共享 / 远程构建

  • Turbopack / Vercel 推动远程 build cache
  • Nx / Turborepo 等已有类似
  • monorepo / CI 加速
  • 2026 仍偏 enterprise；2030 可能主流

趋势 6：dev 不打包成主流

  • Vite 验证了"dev 用浏览器原生 ESM"
  • Turbopack / Bun build 也走同方向
  • 未来可能完全"无 bundler dev"（直接用浏览器 ESM + import maps + esm.sh CDN）

03 应用场景

场景 A · Next.js 项目升级到 Turbopack

Next.js 14+ 默认 dev 用 turbo；build 仍用 Webpack（Next 15 起 build 也支持 turbo）。

# Next.js 15+
$ next dev --turbo        # dev 用 turbopack（默认）
$ next build --turbo      # build 用 turbopack（实验性 → stable）

不需要额外配置。Next.js 处理 turbopack 集成。

场景 B · 大型 Webpack 项目迁 Rspack

$ npm uninstall webpack webpack-cli webpack-dev-server
$ npm install -D @rspack/core @rspack/cli @rspack/dev-server
// rspack.config.js（基本是 webpack.config.js 复制粘贴）
const { rspack } = require('@rspack/core');

module.exports = {
  entry: './src/index.tsx',
  output: { ... },
  module: {
    rules: [
      // 多数 webpack loader 直接用
      { test: /\.tsx?$/, use: 'builtin:swc-loader', exclude: /node_modules/ }
    ]
  },
  plugins: [
    new rspack.HtmlRspackPlugin({ template: './src/index.html' })
  ]
};
$ npm run build   # 5-10x 快

场景 C · 提前用 Rolldown（Vite 6+）

// vite.config.js (Vite 6+)
export default {
  experimental: {
    rolldown: true   // 启用 Rolldown
  }
};
$ vite build
# build 用 Rolldown，5-10x 加速

注意：仍实验性，遇到问题报 issue 给 Rolldown。生产用要先充分测试。

场景 D · Bun 项目用 Bun build

$ bun build src/index.tsx --outdir dist --target browser
// 或 JS API
await Bun.build({
  entrypoints: ['./src/index.tsx'],
  outdir: './dist',
  target: 'browser',
  minify: true,
  sourcemap: 'external'
});

详见 Bun P4.3。

场景 E · 试用 Farm

$ npm i -D @farmfe/cli @farmfe/core
// farm.config.ts
import { defineConfig } from '@farmfe/core';
import react from '@farmfe/plugin-react';

export default defineConfig({
  plugins: [react()],
  compilation: {
    input: { index: './index.html' }
  }
});
$ farm dev
$ farm build

场景 F · 长期路线图：Webpack → Vite

大型 Webpack 项目迁移路径（2-6 个月）：

  1. 第 1 月：评估 Webpack 配置 + 列出所有 loader/plugin → 找 Vite 等价物
  2. 第 2 月：在新分支并行 Vite 配置 + 跑 dev/build → 解决兼容性问题
  3. 第 3-4 月：测试覆盖（unit / integration / e2e）/ visual regression
  4. 第 5 月：staging 环境 Vite 跑一段时间
  5. 第 6 月：production 切换 + 移除 Webpack 配置

对很多大型项目这是 6+ 月工程。Rspack 是过渡选择—— 1-2 周完成 + 立刻获得 5-10x 速度。

04 典型代码：完整选型

Vite + Rolldown 实验配置

// vite.config.ts (Vite 6+)
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],

  experimental: {
    // 启用 Rolldown（实验性）
    rolldown: true
  },

  build: {
    target: 'es2022',
    sourcemap: 'hidden',
    rollupOptions: {
      // Rolldown 兼容 Rollup options
      output: {
        manualChunks: {
          'react-vendor': ['react', 'react-dom']
        }
      }
    }
  }
});

Rspack 配置（Webpack 风格）

// rspack.config.js
const { rspack } = require('@rspack/core');
const path = require('path');

module.exports = {
  entry: './src/index.tsx',

  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: '[name]-[contenthash].js',
    publicPath: '/'
  },

  module: {
    rules: [
      {
        test: /\.tsx?$/,
        exclude: /node_modules/,
        loader: 'builtin:swc-loader',
        options: {
          jsc: {
            parser: { syntax: 'typescript', tsx: true },
            transform: { react: { runtime: 'automatic' }}
          }
        }
      },
      {
        test: /\.css$/,
        use: ['style-loader', 'css-loader']
      }
    ]
  },

  plugins: [
    new rspack.HtmlRspackPlugin({
      template: './src/index.html'
    })
  ],

  optimization: {
    splitChunks: {
      chunks: 'all',
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendor'
        }
      }
    }
  }
};

Turbopack（Next.js）

// next.config.js
module.exports = {
  experimental: {
    turbo: {
      // Turbopack 配置
      rules: {
        '*.svg': {
          loaders: ['@svgr/webpack'],
          as: '*.js'
        }
      }
    }
  }
};
$ next dev --turbo
$ next build --turbo

Bun build CLI + 集成

$ bun build src/index.tsx \
    --outdir dist \
    --target browser \
    --jsx-import-source react \
    --minify \
    --sourcemap external \
    --splitting
// build.ts
const result = await Bun.build({
  entrypoints: ['./src/index.tsx'],
  outdir: './dist',
  target: 'browser',
  minify: true,
  sourcemap: 'external',
  splitting: true
});

if (!result.success) {
  console.error('Build failed:', result.logs);
  process.exit(1);
}

console.log(`Built ${result.outputs.length} files`);

Farm 配置

// farm.config.ts
import { defineConfig } from '@farmfe/core';
import react from '@farmfe/plugin-react';

export default defineConfig({
  plugins: [react()],

  compilation: {
    input: {
      index: './index.html'
    },
    output: {
      path: './dist',
      publicPath: '/'
    },
    persistentCache: true,   // 持久缓存
    treeShaking: true
  },

  server: {
    port: 9000,
    open: true
  }
});

05 常见陷阱

陷阱 1：相信营销 benchmark

反复强调。Turbopack alpha 声称 "Webpack 700x"；Farm 声称 "Vite 5-10x"。实测往往打折。在自家项目跑 benchmark 才有意义。

陷阱 2：过度追新

2024-2026 构建工具变化快。生产项目不应该追新：

  • 稳定 Vite 6 + 项目稳定 = 比追 Rolldown alpha 更明智
  • Farm / Bun build 等小众工具：愿意踩坑 + 报 issue 的早期采用者才上
  • 2026 主流仍是 Vite 6（Rollup）/ Webpack（旧项目）/ Turbopack（Next）

陷阱 3：忽略生态匹配

选 bundler 必须和 framework / deploy / testing 兼容：

  • 选 Bun build → Vercel 默认 Node + npm；要额外配置部署
  • 选 Turbopack → 几乎只能用 Next.js
  • 选 Rspack → 可能少数 Webpack plugin 不兼容

陷阱 4：换 bundler 期望"100% 一致"

不同 bundler 行为不完全一致。换 Webpack → Vite 常见差异：

  • CommonJS 处理不同（Webpack 自动 / Vite 通过 plugin）
  • require.context 等 Webpack 独家特性 Vite 没有
  • Babel transform 链略有差异

换 bundler 必须充分测试。

陷阱 5：以为 Rust bundler "一律快"

实现语言只决定速度上限。Turbopack alpha 在某些场景比 Vite 慢；某些 Rust bundler 因为没充分优化，实测仅快 2-3x。实测优于实现语言。

陷阱 6：用 Turbopack 但不用 Next.js

Turbopack 与 Next.js 强绑定 —— 独立用罕见，文档 / 生态都围绕 Next.js。不用 Next.js 不要选 Turbopack，选 Vite。

陷阱 7：Bun build 不适合复杂场景

Bun build plugin ecosystem 弱 + HMR 仍发展中。复杂项目（Vue / Svelte SFC / 复杂 SSR / Webpack-style 配置）不适合。简单 React / 工具 / library 是 sweet spot。

陷阱 8：忘记 monorepo / 大型项目实测

小项目 benchmark 差别可能不明显。大型项目（5000+ 文件）+ 大量依赖 才能看出 5-10x 差距。选型要在真实大小项目实测，不要看 hello-world demo。
