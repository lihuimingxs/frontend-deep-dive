ESM 标准 + import.meta + import attributes + dynamic import

现代模块系统。关键特性：静态分析（import 必须顶层 + 字符串 literal）让 tree shaking 成为可能；live binding 与 CJS 复制值不同；import.meta 提供模块元数据；import attributes（2025 stage 4）让 ESM 安全 import JSON/CSS/wasm；dynamic import + top-level await 是异步特性；import maps 让浏览器原生解析 bare specifier。bundler 视角：bundler 的核心工作就是"把所有 import 解析 + 合并成产物"。每个 bundler 对 ESM spec 的覆盖度不同。

阅读时长 ~30 分钟 前置 P2（Vite 架构）+ JS 基础 主要来源 ECMA-262 §16 · TC39 import attributes 提案 · MDN ESM 文档

01 速览

  1. ESM 是 Web 标准的"唯一模块系统"—— 与 CommonJS（Node 专属）/ AMD（已死）/ UMD（兼容包装）不同。ESM 是 ECMAScript 规范一部分，浏览器和 Node 都原生支持。
  2. 静态 vs 动态的根本权衡—— ESM 是静态（import 必须在顶层 + 字符串 literal） → 编译时能分析依赖图 → tree shaking 可能。CJS 是动态（require 任意 string） → 编译时不能完全分析 → tree shaking 困难。
  3. live binding—— ESM 的 import 是引用不是复制值。import { count } from './counter'，外部能看到 count 后续变化。CJS 的 const { count } = require('./counter') 是复制，看不到后续变化。这是 ESM/CJS 互操作复杂的根源之一。
  4. import.meta—— 每个 module 有独立元数据。import.meta.url 是 module 的 URL；Vite/Webpack 等扩展 import.meta.env（环境变量）/ import.meta.hot（HMR API）。
  5. import attributes（2025 stage 4 → ECMA 2026）—— import data from './data.json' with { type: 'json' }。让 ESM 安全 import 非 JS 资源（JSON/CSS/wasm/...），替代旧的 import assertions。
  6. dynamic import + top-level await—— const mod = await import('./foo') 运行时 import；top-level await 让 module 顶层可 await。都让 ESM 更接近"应用语言"。
  7. bundler 的核心工作 = 模块解析 + 合并—— 拿到所有 import → 解析路径（node_modules / alias / relative）→ 合并成 chunk → 输出。每个 bundler 对 ESM spec 的覆盖度不同（如 import attributes 普遍支持，import maps 浏览器原生）。

02 底层逻辑

ESM 历史 + spec 来源

ESM 是 ECMAScript 2015（ES6）引入：

  • 2015-06：ECMA-262 6th Edition 发布；首次定义模块语法
  • 2015-2017：规范完善（ES2017 加 dynamic import / ES2020 加 import.meta）
  • 2017-2018：浏览器原生支持（Chrome 61 / Safari 10.1 / Firefox 60）
  • 2019-12：Node 13 默认启用 ESM（require 仍兼容）
  • 2020：Vite 等 ESM-first bundler 出现
  • 2025-03：import attributes 进入 ES2025

规范源详见 ECMA P7.4 模块系统。本章讲开发者视角 + bundler 视角。

静态分析特性

ESM 设计强约束：

规则 1：import / export 必须在顶层

// ✅ 顶层
import { foo } from './foo';

// ❌ 不行
if (cond) {
  import { foo } from './foo';   // SyntaxError
}

function loadModule() {
  import { foo } from './foo';   // SyntaxError
}

但dynamic import 可以任何位置：

if (cond) {
  const { foo } = await import('./foo');  // ✅ dynamic import 是函数调用
}

规则 2：import 路径必须是字符串 literal

// ✅
import x from './foo';

// ❌ 不行
const path = './foo';
import x from path;             // SyntaxError

// dynamic import 可以
const x = await import(path);   // ✅

规则 3：export 必须在顶层—— 同上。

这两条规则的意义：

编译时（不运行代码）就能知道 module 的所有 import / export 是什么。这让：

  • Tree shaking 成为可能—— 编译器知道每个 export 是否被用
  • 循环依赖检测—— 编译时构建依赖图
  • bundler 优化—— scope hoisting / chunk splitting / 等

CJS 的 require 是函数调用（动态），编译时不知道实际加载什么 → 所有这些优化都难。

live binding（活引用）

ESM 的 import 是引用不是复制：

// counter.js
export let count = 0;
export function increment() { count++; }

// app.js
import { count, increment } from './counter';

console.log(count);   // 0
increment();
console.log(count);   // 1   ← 看到了变化！

CJS 行为不同：

// counter.js
let count = 0;
exports.count = count;
exports.increment = function() { count++; };

// app.js
const { count, increment } = require('./counter');

console.log(count);   // 0
increment();
console.log(count);   // 0   ← 看不到变化（复制了值）

live binding 的实现：

底层 ESM 引擎为每个 import 创建一个引用槽。访问 count 时实际上是去 module 的命名空间里取当前值。

影响：

  • 测试时 mock 简单（重新 import 拿最新值）
  • 但 destructuring 后失去 live binding：const { count } = mod 复制了
  • CJS 互操作复杂 —— bundler 必须模拟 live binding

import.meta：模块元数据

每个 ESM 模块有独立的 import.meta 对象：

// 浏览器原生
console.log(import.meta.url);   // "https://example.com/main.js"

// Node ESM
console.log(import.meta.url);   // "file:///path/to/main.js"
console.log(import.meta.dirname);  // "/path/to" (Node 21+)
console.log(import.meta.filename); // "/path/to/main.js"

// Vite 扩展
console.log(import.meta.env.MODE);    // 'development' | 'production'
console.log(import.meta.env.DEV);     // true | false
console.log(import.meta.env.VITE_API_URL);  // 自定义

// HMR API（Vite/HMR runtime 注入）
if (import.meta.hot) {
  import.meta.hot.accept((newMod) => { ... });
}

// glob import（Vite 特性）
const modules = import.meta.glob('./pages/*.tsx');
// modules = { './pages/Home.tsx': () => import('./pages/Home.tsx'), ... }

import.meta.url 的实用价值：

// 解析相对当前模块的资源路径
const dataUrl = new URL('./data.json', import.meta.url);

// Worker 用相对路径
const worker = new Worker(new URL('./worker.js', import.meta.url), { type: 'module' });

import attributes（2025 ES2025）

原始问题：ESM import 默认期望 JS。要 import JSON / CSS / wasm 等怎么办？早期方案 import assertions（assert { type: 'json' }）2023 被 import attributes 替代：

// 2025+ stage 4
import data from './data.json' with { type: 'json' };
import styles from './styles.css' with { type: 'css' };
import wasmModule from './module.wasm' with { type: 'webassembly' };

实现状态：

  • JSON：浏览器 / Node 都支持（2023+）
  • CSS：实验性（CSSStyleSheet 集成）
  • WebAssembly：实验性
  • bundler 端：Vite / Webpack / esbuild 都支持 JSON

之前的方案：bundler 让你直接 import data from './data.json'（约定路径后缀），但这不是 spec。import attributes 是 spec 标准化版本。

dynamic import

// 运行时 import；返回 Promise
const mod = await import('./foo');
console.log(mod.default);

// 路径可以动态
const moduleName = condition ? './a' : './b';
const mod = await import(moduleName);

// 用于 lazy loading
async function loadHeavyModule() {
  const { heavyFunc } = await import('./heavy');
  return heavyFunc();
}

// React 用
const Page = lazy(() => import('./Page'));

bundler 处理：

  • 静态分析：找出所有 dynamic import 调用 + 路径（文字部分）
  • 每个 dynamic import 自动一个 chunk（code splitting 基础）
  • 路径动态时（如 import(\`./pages/${name}\`)）：bundler 通常 glob 匹配可能路径，所有匹配文件都打成 chunk

top-level await

ESM 模块顶层可以直接 await（不必包装在 async function 里）：

// data.js
const response = await fetch('/api/data');
export const data = await response.json();

// app.js
import { data } from './data';   // ✅ 等待 data.js 完成
console.log(data);

语义：模块的执行变成异步 —— import 它的模块也异步等待。

限制：

  • CJS 不兼容（CJS 是同步 require）
  • 循环依赖可能 deadlock
  • 不要随意用（让模块加载链变慢）

典型场景：

  • 动态 config 加载
  • WASM 模块初始化
  • 数据库 / 资源连接

import maps（浏览器原生）

浏览器原生 ESM 不支持 bare specifier（import 'react'）。import maps 解决：

<script type="importmap">
{
  "imports": {
    "react": "https://esm.sh/react@19",
    "react-dom": "https://esm.sh/react-dom@19",
    "lodash-es": "/vendor/lodash-es.js"
  }
}
</script>

<script type="module">
  import React from 'react';   // 浏览器查 import map → "https://esm.sh/react@19"
</script>

Vite dev mode 内部用类似机制（但 server 端重写而不是 import map）。

2026 浏览器普遍支持 import maps（Chrome 89+ / Safari 16.4+ / Firefox 108+）。Deno 也用 import maps 配置依赖。

module 命名空间（namespace）

ESM 有命名导出和默认导出两种：

// utils.js
export const PI = 3.14;        // 命名导出
export function add(a, b) { return a + b; }
export default class Calc { ... }   // 默认导出（每模块只一个）

// app.js
import Calc, { PI, add } from './utils';   // 默认 + 命名

// 全部导入为命名空间
import * as utils from './utils';
console.log(utils.PI, utils.add, utils.default);   // default 是默认导出

命名空间对象是不可变（不能改字段）+ 可枚举。

bundler 怎么处理 ESM

典型 bundler 处理流程：

1. Resolve（路径解析）—— 把 import 字符串解析到具体文件：

  • 相对路径（./foo）→ 直接 resolve
  • 绝对路径（/foo）→ 项目根 + foo
  • bare specifier（react）→ 从 node_modules 查找；用 package.json exports 字段
  • 带 alias（@/foo）→ 应用 alias 配置

2. Load（读文件）—— 读 source code（含 source map）。

3. Transform（编译）—— TS → JS / JSX → JS / 等。

4. Module graph 构建—— 递归处理 imports，构建完整依赖图。

5. Tree shaking—— 标记未使用 export（详见 P4）。

6. Chunk 分割—— 按 dynamic import + manualChunks 分。

7. Code generation—— 输出 ESM/CJS/UMD。

ESM 与 bundler 的虚拟解析

浏览器原生 ESM 用 URL（绝对 / 相对）。但 bundler 处理 import 'react' 时，"react" 不是 URL —— 是 bare specifier。bundler 必须做"虚拟 ESM 解析"：

用 Node Resolution Algorithm（虽然 ESM spec 不强制，但 bundler 实际遵循）：

  1. 查 node_modules/react/package.json
  2. 看 exports 字段（首选）
  3. 看 module 字段（ESM）
  4. 看 main 字段（fallback）
  5. 解析到具体文件

不同 bundler 实现细节略有差异（如对 conditional exports 的支持），导致同一个包在不同 bundler 行为不同。

03 应用场景

场景 A · 用 dynamic import 实现 lazy loading

import { lazy, Suspense } from 'react';

const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings = lazy(() => import('./pages/Settings'));

export default function App() {
  return (
    <Routes>
      <Route path="/dashboard" element={
        <Suspense fallback={<Loading />}>
          <Dashboard />
        </Suspense>
      } />
      <Route path="/settings" element={
        <Suspense fallback={<Loading />}>
          <Settings />
        </Suspense>
      } />
    </Routes>
  );
}

每个 dynamic import 自动一个 chunk，按需加载。

场景 B · 用 import.meta.glob 自动注册路由

// router.ts (Vite 特性)
const pages = import.meta.glob('./pages/**/*.tsx');
// pages = { './pages/Home.tsx': () => import(...), './pages/About.tsx': () => import(...) }

const routes = Object.entries(pages).map(([path, loader]) => ({
  path: path.replace('./pages', '').replace('.tsx', ''),
  component: lazy(loader)
}));

场景 C · 用 top-level await 加载 wasm

// wasm-mod.ts
const wasmModule = await WebAssembly.instantiateStreaming(fetch('/my.wasm'));
export const { add, multiply } = wasmModule.instance.exports;

// app.ts
import { add } from './wasm-mod';   // 自动等待 wasm 加载完
console.log(add(2, 3));

场景 D · 用 import.meta.env 区分环境

const apiUrl = import.meta.env.PROD
  ? 'https://api.example.com'
  : 'http://localhost:3001';

if (import.meta.env.DEV) {
  // 仅 dev 启用 redux devtools
  enableDevTools();
}

// 自定义 env（必须 VITE_ 前缀才暴露给客户端）
const sentryDsn = import.meta.env.VITE_SENTRY_DSN;

场景 E · 浏览器原生 ESM + import maps（无 bundler）

<!doctype html>
<html>
  <head>
    <script type="importmap">
    {
      "imports": {
        "react": "https://esm.sh/react@19?dev",
        "react-dom/client": "https://esm.sh/react-dom@19/client?dev"
      }
    }
    </script>
  </head>
  <body>
    <div id="root"></div>
    <script type="module">
      import React from 'react';
      import { createRoot } from 'react-dom/client';

      const App = () => React.createElement('h1', null, 'Hello!');
      createRoot(document.getElementById('root')).render(React.createElement(App));
    </script>
  </body>
</html>

这种"无 bundler"开发越来越流行（特别是 demo / 小工具）—— ESM CDN（esm.sh / unpkg）+ import maps 让你不需要 npm install。

04 典型代码：ESM 完整使用

命名导出 + 默认导出 + re-export

// utils.js

// 命名导出
export const VERSION = '1.0';
export function add(a, b) { return a + b; }
export class Calculator { ... }

// 默认导出
export default class App { ... }

// 重新导出
export { foo, bar } from './other';
export * from './common';
export { default as Helper } from './helper';
// 各种 import 形式

// 默认导出
import App from './utils';

// 命名导出
import { VERSION, add } from './utils';

// 默认 + 命名
import App, { VERSION, add } from './utils';

// 重命名
import { add as addNumbers } from './utils';

// 全部为命名空间
import * as Utils from './utils';
// Utils.VERSION, Utils.add, Utils.default

// 仅副作用（不获取任何 export）
import './side-effects';

dynamic import 模式

// 简单 lazy load
const mod = await import('./heavy');

// 错误处理
try {
  const mod = await import('./optional');
  mod.init();
} catch (e) {
  console.warn('Optional module failed to load:', e);
}

// 路径动态
async function loadLocale(locale: string) {
  const mod = await import(`./locales/${locale}.json`, { with: { type: 'json' } });
  return mod.default;
}

// 并行 import
const [a, b, c] = await Promise.all([
  import('./a'),
  import('./b'),
  import('./c'),
]);

// React.lazy
const HeavyComponent = lazy(() => import('./HeavyComponent'));

import attributes 实战

// 静态 import attributes
import config from './config.json' with { type: 'json' };
console.log(config.version);

// dynamic import with attributes
const data = await import('./data.json', { with: { type: 'json' } });

// CSS modules（实验性，部分浏览器）
import sheet from './styles.css' with { type: 'css' };
document.adoptedStyleSheets = [sheet];

top-level await + 错误处理

// config-loader.ts
const loadConfig = async () => {
  const response = await fetch('/api/config');
  if (!response.ok) throw new Error('Config load failed');
  return response.json();
};

let config: Config;
try {
  config = await loadConfig();
} catch (e) {
  console.error('Failed to load config, using defaults:', e);
  config = DEFAULT_CONFIG;
}

export { config };

Vite glob import 高级用法

// 同步 glob（编译时全部 import）
const modules = import.meta.glob('./pages/*.tsx', { eager: true });
// 编译时 modules = { './pages/Home.tsx': { default: HomeComponent }, ... }

// 异步 glob（lazy load）
const lazyModules = import.meta.glob('./pages/*.tsx');
// lazyModules = { './pages/Home.tsx': () => import(...), ... }

// 只 import 特定 export
const components = import.meta.glob('./components/*.tsx', {
  import: 'default',
  eager: true
});

// 自定义 query（asset import）
const images = import.meta.glob('./assets/*.png', {
  query: '?url',
  import: 'default',
  eager: true
});

05 常见陷阱

陷阱 1：以为 destructuring 后还是 live binding

// counter.js
export let count = 0;
export const increment = () => { count++; };

// app.js (live binding)
import { count, increment } from './counter';
increment();
console.log(count);   // 1 ✅

// app.js (destructuring 后 NOT live)
import * as mod from './counter';
const { count, increment } = mod;
increment();
console.log(count);   // 0 ❌ 复制了

// 用 mod.count 仍是 live
console.log(mod.count);   // 1 ✅

陷阱 2：以为 dynamic import 路径完全任意

bundler 静态分析时仍要看路径中"字符串部分"。如果完全动态（如来自 user input），bundler 不能预知 → 不能 prebundle → 运行时可能失败：

// ❌ bundler 不知道要 build 什么
const userInput = getInput();
const mod = await import(userInput);

// ✅ bundler 看到 './pages/' prefix → glob 匹配所有 pages
const mod = await import(`./pages/${name}.tsx`);

陷阱 3：top-level await 让模块加载阻塞

过度用 top-level await 让 module 加载链变慢：

// data.js
export const data = await fetch('/api/slow').then(r => r.json());
// 这个 fetch 5s 才完成

// 任何 import data.js 的模块都阻塞 5s

对策：把 fetch 包成 async function，让 caller 决定何时等：

// data.js
export async function loadData() {
  return fetch('/api/slow').then(r => r.json());
}

// caller 决定何时 await
const data = await loadData();

陷阱 4：循环依赖 + ESM

ESM 比 CJS 处理循环依赖好（live binding 让"先 import 后赋值"工作），但仍可能出问题：

// a.js
import { b } from './b';
console.log(b);   // 可能 undefined 取决于加载顺序
export const a = 'a';

// b.js
import { a } from './a';
console.log(a);   // 同样可能 undefined
export const b = 'b';

对策：

  • 避免循环依赖（架构层面）
  • 把循环部分用 dynamic import
  • 函数内访问而不是顶层访问（lazy）

陷阱 5：忘记 .js 扩展（Node ESM 严格）

Node ESM 模式（package.json "type": "module"）必须显式 .js 扩展：

// ❌ Node ESM 报错
import { foo } from './utils';

// ✅
import { foo } from './utils.js';

bundler（Vite/Webpack）通常自动补 .js 扩展（dev / build），但 Node 直接跑没自动。TS 用户特别注意：源码 import './utils'，编译后要 import './utils.js'。

陷阱 6：default export 互操作问题

当 ESM 模块 import CJS 模块的 default：

// CJS module
module.exports = { a: 1, b: 2 };

// ESM 这样 import
import obj from './cjs';
console.log(obj);   // ?
// 不同 bundler 行为不同：
//   一些 bundler: { a: 1, b: 2 }
//   一些 bundler: { default: { a: 1, b: 2 } }（要 obj.default.a）

详见 P3.2 CJS Interop。

陷阱 7：import.meta.env 暴露敏感变量

Vite 默认只暴露 VITE_ 前缀的 env 给客户端 bundle。但有时不小心：

// .env
VITE_API_URL=https://api.example.com
DATABASE_PASSWORD=secret123        ← 不会暴露（无 VITE_）
VITE_SECRET_KEY=secret456          ← 会暴露！危险

规则：VITE_ 前缀的只放 public 信息。secrets 永远不要给客户端。

陷阱 8：忘记 type: "module"

npm 包做 ESM 必须 package.json 声明：

{
  "type": "module",
  "main": "./dist/index.js",
  "exports": {
    ".": {
      "import": "./dist/index.js"
    }
  }
}

否则 Node 默认按 CJS 加载，import 报错。
