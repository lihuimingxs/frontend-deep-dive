CJS Interop + 双 Package Hazard + bundler 处理

现实世界的模块混乱。回答"为什么我 import lodash 报错"、"为什么 node-fetch 升 3 后我用不了"、"为什么这个包同时发 CJS 和 ESM 反而让我状态错乱"。核心问题：CJS / ESM 共存是 Node ecosystem 的"原罪"——Node 决定 ESM 时社区已有 100k+ CJS 包，不能强制全切 ESM。本章讲清：CJS / ESM 的 4 个不兼容点 / 双 package hazard / package.json exports 字段 / bundler 怎么处理 CJS / 现实世界混乱场景。

阅读时长 ~30 分钟 前置 P3.1（ESM 标准）+ Node.js 基础 主要来源 Node Modules docs · package.json exports 提案 · 各 bundler interop 文档

01 速览

  1. CJS / ESM 共存是 Node ecosystem 的"原罪"—— Node 决定 ESM（2017+）时社区已有 100k+ CJS 包；不能强制全切 ESM。2026 仍在管理两个系统，新代码 ESM 主流但 CJS 大量存活。
  2. 4 个根本不兼容点：(1) CJS 同步 require / ESM 异步 import + top-level await；(2) CJS 复制值 / ESM live binding；(3) CJS 动态路径 / ESM 静态；(4) CJS exports namespace / ESM named exports + default。
  3. 双 package hazard—— 一个包同时发布 CJS 和 ESM 版本；Node 加载时可能load 两次（CJS 一份 + ESM 一份） → 内部状态不一致 → bug 难调试。
  4. package.json exports 字段—— 现代约定，精细声明每个 entry 的 CJS / ESM / types："exports": { ".": { "import": "...", "require": "...", "types": "..." }}。所有新包都应该用 exports 而不是 main。
  5. bundler 处理 CJS 各家不同—— Vite (Rollup) 用 @rollup/plugin-commonjs 转 ESM AST（不完美）/ Webpack 原生支持两种 + 自动 wrapper / esbuild 原生支持两种 + 自动 interop。同一个 CJS 包不同 bundler 行为可能略不同。
  6. 现实场景痛点—— lodash vs lodash-es（tree shaking 差异）/ node-fetch 3+ 仅 ESM（CJS 用户痛）/ Vite SSR 模式 dev/build 输出不同 / CJS-only 库 lock 整个项目升级 ESM。

02 底层逻辑

CJS（CommonJS）核心

CJS 是 Node 1.0+（2009）的原生模块系统。Browserify（2013）后被带到浏览器。核心 API：

// foo.js
const x = 'hello';
function greet() { return x; }

module.exports = { x, greet };
// 或
exports.x = x;
exports.greet = greet;

// app.js
const { x, greet } = require('./foo');
console.log(greet());

关键属性：

  • 同步加载—— require 同步返回；阻塞执行直到模块加载完
  • 动态路径—— require(anyString)，运行时计算
  • 复制值—— const { x } = require(...) 复制了 x 当前值
  • module.exports 是 namespace 对象—— 一个对象代表整个模块导出

CJS vs ESM 4 个不兼容点

不兼容点 1：同步 vs 异步

// CJS：同步
const fs = require('fs');
const config = require('./config');   // 同步返回

// ESM：异步
import fs from 'fs';
import config from './config.js';     // 解析阶段异步处理

// top-level await（仅 ESM）
import { data } from './data.js';
const result = await fetch(data.url);

影响：CJS 模块不能 import ESM 模块（CJS 是同步，ESM 是异步）。Node 22+ 部分支持同步加载 ESM（如果 ESM 没 top-level await）但仍有限制。

不兼容点 2：复制值 vs live binding

详见 P3.1。简短：

// CJS 复制值
const { count } = require('./counter');
counter.increment();
console.log(count);   // 旧值

// ESM live binding
import { count, increment } from './counter';
increment();
console.log(count);   // 新值

不兼容点 3：动态 vs 静态路径

// CJS：动态完全可以
function load(name) {
  return require(`./modules/${name}`);   // 运行时算
}

// ESM：static import 必须字符串 literal
import x from `./modules/${name}`;   // ❌ SyntaxError

// ESM 用 dynamic import
async function load(name) {
  return await import(`./modules/${name}`);   // ✅
}

不兼容点 4：exports 结构

// CJS：module.exports 整体是一个对象
module.exports = { a: 1, b: 2 };
// 或
module.exports = function() { ... };
// 或
module.exports = "string";

// ESM：必须命名 export 或 default export
export const a = 1;
export const b = 2;
export default function() { ... };

问题：CJS 的 module.exports = function() {} 怎么映射到 ESM？是当 default export？还是当命名空间？这是 interop 复杂的根源。

双 Package Hazard 详解

问题场景：你的应用同时通过两个路径加载同一个包：

app.mjs (ESM)
  ↓ import 'state-lib'
  → state-lib (ESM 版本) loaded as Module A

dep-of-app.cjs (CJS)
  ↓ require('state-lib')
  → state-lib (CJS 版本) loaded as Module B

Module A 和 B 是两个独立模块实例！

如果 state-lib 内部有module-level state（单例 / 计数器 / cache）：

// state-lib internal
let counter = 0;
export function increment() { counter++; return counter; }

// 通过 ESM
import { increment as inc1 } from 'state-lib';
inc1(); inc1();   // counter = 2

// 通过 CJS
const { increment: inc2 } = require('state-lib');
inc2();           // counter = 1（不同实例！）

用户期望两次调用 counter = 3；实际是两个独立 counter。极难调试，因为代码看起来正确。

规避方案：

  • 包作者：不要同时发 CJS 和 ESM 单独包。如果发，exports 字段统一指向同一文件（用现代工具如 tsup 输出 dual format 但内部共享）。
  • 用户：检查 npm ls 看是否同包多版本；用 yarn dedupe / pnpm dedupe 合并。

package.json exports 字段（现代标准）

npm 包传统用 main 字段：

{
  "main": "./index.js"
}

缺点：只能一个 entry；不能区分 CJS / ESM；没有 conditional resolution。

2019+ Node 引入 exports 字段：

{
  "name": "my-pkg",
  "type": "module",
  "main": "./dist/index.js",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./utils": {
      "types": "./dist/utils.d.ts",
      "import": "./dist/utils.js",
      "require": "./dist/utils.cjs"
    },
    "./package.json": "./package.json"
  }
}

关键特性：

  • 条件 resolution—— Node 看 require 选 CJS；看 import 选 ESM
  • 多 subpath—— 不同子路径独立 entry
  • encapsulation—— 没列在 exports 里的内部文件外部不能访问（防止用户依赖 internal API）
  • types 优先—— TypeScript 看 types

条件键的优先级（自上而下）：

{
  "exports": {
    ".": {
      "types": "./types/index.d.ts",        // TS 优先
      "browser": "./browser/index.js",       // 浏览器环境
      "deno": "./deno/index.js",             // Deno
      "node": {
        "import": "./node/index.mjs",
        "require": "./node/index.cjs"
      },
      "default": "./fallback/index.js"
    }
  }
}

bundler 怎么处理 CJS

Vite (Rollup) 处理

Vite build 用 Rollup，依赖 @rollup/plugin-commonjs 把 CJS 转 ESM AST：

// CJS 输入
module.exports = { foo: 1, bar: 2 };

// 转换后（AST 级）
const cjsExports = { foo: 1, bar: 2 };
export default cjsExports;
export const foo = cjsExports.foo;
export const bar = cjsExports.bar;

不完美：

  • 动态 require（require(somePath)）转换困难
  • 条件 require（if (x) require('a') else require('b')）需 plugin 启发式
  • module.exports 重赋值情况复杂

Vite dev mode 用 esbuild 预构建依赖，esbuild 自带 CJS 转换（更稳定但语义略不同）。

Webpack 处理

Webpack 原生支持 CJS 和 ESM，自动 interop wrapper：

// Webpack 生成的代码（简化）
function __webpack_require__(moduleId) {
  if (moduleCache[moduleId]) return moduleCache[moduleId].exports;
  const module = moduleCache[moduleId] = { exports: {} };
  modules[moduleId](module, module.exports, __webpack_require__);
  return module.exports;
}

优势：兼容性最好（任何包基本能跑）；劣势：bundle 体积大（每模块都有 wrapper）+ tree shaking 不及 Rollup。

esbuild 处理

esbuild 原生支持两种格式 + 自动 interop：

// CJS 输入
module.exports = { foo: 1 };

// esbuild 输出（简化）
var stdin_exports = {};
__export(stdin_exports, { default: () => stdin_default });
var stdin_default = { foo: 1 };

// ESM 用户
import x from 'cjs-module';
console.log(x);   // { foo: 1 }

esbuild 的 interop 逻辑相对简单清晰，bug 少；但功能不及 Rollup（无法处理某些边缘 CJS 模式）。

bundler 输出格式

bundler 通常能输出多种格式：

格式含义使用场景
esm / esESM现代浏览器 / Node ESM
cjs / commonjsCJS旧 Node / Browserify 兼容
umdUniversal浏览器全局 + Node CJS（库常用）
iifeIIFE浏览器全局 only（脚本 inject）
amdAMDRequireJS（已死）
systemSystemJSSystemJS loader

2026 实践：

  • App build → 通常 ESM（现代浏览器）+ legacy ES5（@vitejs/plugin-legacy 支持旧浏览器）
  • Library build → ESM + CJS（兼容性）；不再发 UMD（除非 IE 10 兼容）

现实场景痛点

痛点 1：lodash vs lodash-es

// lodash 是 CJS
import _ from 'lodash';
const r = _.debounce(fn);
// 整个 lodash bundle 进来（~70KB）

// lodash-es 是 ESM
import { debounce } from 'lodash-es';
const r = debounce(fn);
// 只 debounce 进来（~5KB）—— tree shaking 生效

规则：能用 ESM 版本就用（lodash-es / date-fns / chart.js ESM build / 等）。CJS 版本通常 tree shake 不动。

痛点 2：node-fetch 3+ 仅 ESM

node-fetch 是 Node 经典 fetch polyfill。2021 年 v3 发布仅 ESM（去掉 CJS 支持）：

// node-fetch 3+
// CJS 用户：报错
const fetch = require('node-fetch');   // ❌ Error [ERR_REQUIRE_ESM]

// ESM 用户：OK
import fetch from 'node-fetch';   // ✅

大量 CJS 项目升级 node-fetch 失败。社区压力下 v3 后续提供 cjs export，但只是包装：

// 后续 hack
const fetch = (...args) => import('node-fetch').then(({ default: f }) => f(...args));

2026 解决：Node 18+ 自带 fetch（不需要 node-fetch）。但旧 Node 项目仍痛。

痛点 3：Vite SSR dev/build 输出不同

Vite SSR：

  • dev：用 ESM（Vite server 直接 transform）
  • build：可能输出 CJS（Node 旧版兼容）或 ESM（现代）

dev 跑得 OK 的 SSR build 后报错的常见原因：

  • 用了 ESM-only 包（dev OK，build CJS 时不能 require）
  • 用了 top-level await（dev OK，build CJS 时不支持）

规避：build 配置 ssr.noExternal 让 Vite 把它打进 bundle；或者输出 ESM（要求 Node 16+）。

痛点 4：CJS-only 库 lock 整个项目

你想升 ESM；但项目某个老依赖只发 CJS（包括它的 transitive deps）。升不了。

规避：

  • 找替代品（社区维护的 ESM fork）
  • 等待原作者更新
  • 用 patch-package 临时修补
  • 暂时保持 CJS

Node 22+ 同步 require ESM

2024 Node 22 引入实验性同步 require ESM（如果 ESM 没 top-level await）：

// Node 22+
const esmMod = require('./pure-esm-module.mjs');
// 如果 ESM 没 top-level await，能同步加载

这缓解了 CJS 用户用 ESM 库的痛苦。但仍有限制：

  • ESM 不能 top-level await
  • 实验性（要 --experimental-require-module flag）
  • 语义微妙（live binding 在同步 require 后变复杂）

tsx / ts-node：TypeScript ESM 痛点

TS 编译输出有几种选择：

  • module: "commonjs"（旧默认）—— TS 把 ESM 转 CJS
  • module: "esnext"—— 输出 ESM（需 .js 扩展）
  • module: "nodenext"—— 跟随 Node 行为（看 package.json type）

常见痛点：

  • TS 源码 import './utils'，编译 ESM 后要 import './utils.js'（TS 不自动加扩展）
  • tsx / ts-node 的 ESM 模式不稳（2024-2026 仍在演进）

详见 TypeScript 主题。

03 应用场景

场景 A · 发布一个 dual format（CJS + ESM）库

用 tsup（基于 esbuild）或 tshy（TS-first）：

$ npm i -D tsup
// tsup.config.ts
import { defineConfig } from 'tsup';

export default defineConfig({
  entry: ['src/index.ts'],
  format: ['esm', 'cjs'],
  dts: true,
  splitting: false,
  sourcemap: true,
  clean: true
});
// package.json
{
  "name": "my-lib",
  "version": "1.0.0",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  },
  "files": ["dist"]
}

场景 B · 解决 CJS-only 包不能在 ESM 项目用

包 X 只发 CJS。你的项目 ESM。

方案 1：default import 通常 OK：

// CJS 包 X：module.exports = { foo: 1 }
import X from 'x';        // ✅ X = { foo: 1 }
console.log(X.foo);
import { foo } from 'x';  // ❌ 不行（ESM 期望命名 export）

方案 2：用 createRequire（Node ESM）：

import { createRequire } from 'node:module';
const require = createRequire(import.meta.url);
const X = require('x');
console.log(X.foo);

方案 3：dynamic import（统一接口）：

const X = (await import('x')).default;

场景 C · 用 exports 字段做 conditional export

{
  "exports": {
    ".": {
      "types": "./types/index.d.ts",

      "browser": {
        "import": "./browser/index.js",
        "require": "./browser/index.cjs"
      },

      "node": {
        "import": "./node/index.js",
        "require": "./node/index.cjs"
      },

      "deno": "./deno/index.js",

      "default": "./fallback/index.js"
    }
  }
}

不同环境（浏览器 / Node / Deno）加载不同实现 —— 比如浏览器版用 fetch，Node 版用 axios。

场景 D · 修复 lodash tree shaking

# 替换
$ npm uninstall lodash
$ npm install lodash-es

# 类型
$ npm install -D @types/lodash-es
// 之前
import _ from 'lodash';
_.debounce(fn);
// → bundle 含整个 lodash (~70KB)

// 之后
import { debounce } from 'lodash-es';
debounce(fn);
// → bundle 只含 debounce 及其依赖 (~5KB)

场景 E · Vite SSR 打 ESM 输出

// vite.config.js
export default {
  build: {
    ssr: true,
    rollupOptions: {
      input: 'src/entry-server.ts',
      output: {
        format: 'esm',                  // 输出 ESM
        entryFileNames: '[name].js'
      }
    }
  },
  ssr: {
    // 让某些 CJS-only 包打进 bundle
    noExternal: ['cjs-only-pkg'],
    // 让某些 ESM-only 包不打包（Node 直接 import）
    external: ['node-fetch']
  }
};
// package.json
{
  "type": "module",         // 让 Node 把 .js 当 ESM
  "scripts": {
    "start": "node dist/server/entry-server.js"
  }
}

修复 dual package hazard

// 错误 pattern：CJS 和 ESM 各自有 module-level state
// state-lib/dist/index.cjs
let count = 0;
exports.increment = () => ++count;

// state-lib/dist/index.js
let count = 0;
export const increment = () => ++count;

// → 两个独立 count 变量
// 修复 pattern：CJS wrapper 引用 ESM 实现
// state-lib/dist/index.js (ESM, 真实现)
let count = 0;
export const increment = () => ++count;

// state-lib/dist/index.cjs (CJS wrapper)
const esm = require('./index.js');   // Node 22+ 同步 require ESM
module.exports = esm;

// 或用 __esModule + default 桥接技巧

05 常见陷阱

陷阱 1：仍用 main 字段不用 exports

2026 新发包必须用 exports。只用 main：

  • 不能区分 CJS / ESM
  • 用户能 import 包内任意文件（破坏封装）
  • 没有 conditional resolution

Vite / Bun / Node 等现代工具优先看 exports。

陷阱 2：exports 字段顺序错

条件键顺序敏感。Node 按从上到下匹配第一个：

// ❌ 错：default 在前面捕获所有
"exports": {
  ".": {
    "default": "./fallback.js",
    "types": "./types.d.ts"      // 永远不会匹配
  }
}

// ✅ 正：types 优先，default 兜底
"exports": {
  ".": {
    "types": "./types.d.ts",
    "node": "./node.js",
    "default": "./fallback.js"
  }
}

规则：types 永远第一；default 永远最后。

陷阱 3：dual format 但没修 dual package hazard

发 dual format 包就要考虑 dual package hazard。不用 module-level state，或共享底层实现。

实测：用 are-the-types-wrong + 测试 CJS / ESM 两个 import 路径互通。

陷阱 4：发 ESM-only 不留 CJS

node-fetch 3 教训。除非有强烈理由，新库默认发 dual format。CJS 用户仍是大多数。

用 tsup / tshy 等工具发 dual format 是几乎免费的。

陷阱 5：以为 import * 等同 default import

// CJS module: module.exports = { foo: 1 };

// 不同 import 行为可能不同（依 bundler）
import x from 'cjs-mod';                       // x = { foo: 1 } 或 x = { default: { foo: 1 } }
import * as y from 'cjs-mod';                  // y = { default: { foo: 1 }, foo: 1 }（Vite/Webpack 自动 wrap）
const z = require('cjs-mod');                  // z = { foo: 1 }（CJS 原始）

// 安全做法：测试每个 bundler 行为

陷阱 6：忘记 @types 包

很多 CJS 包没自带 types。需要 @types/X：

$ npm install -D @types/lodash @types/node-fetch
# 等

陷阱 7：CJS 库内部用 require.cache 期望单例

有些 CJS 库假设"每次 require('mylib') 都是同一实例"（用 require.cache 实现单例）。ESM 加载不走 require.cache → 单例假设破坏。

例：早期 React + react-dom 双 import 让 react 实例不一致 → "invalid hook call" 错误。React 18+ 用 exports 字段 + 强 dedupe 缓解。

陷阱 8：用户代码混 ESM / CJS

// app.mjs (ESM)
import { foo } from './utils.js';

// utils.cjs (CJS)
module.exports = { foo: 1 };

// Node ESM 加载 .cjs 需要 createRequire
import { createRequire } from 'node:module';
const require = createRequire(import.meta.url);
const { foo } = require('./utils.cjs');   // ✅

实践：项目内代码统一一种格式（要么全 ESM 要么全 CJS），别混。
