esbuild Go vs SWC Rust vs Babel

bundler 内部用什么编译器把 TS / JSX / 现代 JS → 浏览器能跑的 JS。三家鼎立：Babel（2014, JS 实现，插件最大但慢 100-1000ms/file）/ esbuild（2020, Go 实现，并行 + AST in memory，极快 < 10ms/file 但 plugin 弱不能改 AST）/ SWC（2020, Rust 实现，接近 Babel 兼容性 + 接近 esbuild 速度，wasm plugin）。性能 100x 差距。三者都做：transform 现代 JS → 老 JS / JSX → React.createElement / TS → JS / polyfill 注入。本章讲清各自定位 + 何时选谁。

阅读时长 ~30 分钟 前置 P3.1（ESM） 主要来源 Babel docs · esbuild docs · swc.rs · 各项目 README + benchmark

01 速览

  1. 三个 compiler 不是替代关系，是三家鼎立—— Babel 仍是 plugin ecosystem 最大；esbuild 是 bundler 子组件 + 速度极致；SWC 是 Babel 替代品 + 接近 esbuild 速度。各有 sweet spot。
  2. Babel（2014-）—— JS 实现 + 插件 ecosystem 最大（@babel/preset-env / preset-react / preset-typescript / 各种第三方）+ 是事实标准 + 慢（100-1000ms/file）。
  3. esbuild（2020-, Evan W）—— Go 实现 + 并行 + 单 pass + 自家 AST in memory + 不依赖第三方 parser → 极快（< 10ms/file）+ plugin 不能 AST 改写（只 resolve / load hooks）+ baseline 功能。
  4. SWC（2020-, DongYoon Kang / Vercel）—— Rust 实现 + 接近 Babel preset 兼容性 + 接近 esbuild 速度（~50x Babel）+ wasm plugin（用 Rust 写）+ Next.js 默认。
  5. 三者都做的工作：(1) 现代 JS → 老 JS（箭头 / class / async）；(2) JSX → React.createElement 或 jsx-runtime；(3) TS → JS（类型擦除）；(4) polyfill 注入（@babel/preset-env auto polyfill）。
  6. 性能数据（typical TS file）：Babel ~100-200ms / SWC ~5-10ms / esbuild ~3-5ms。1000 文件项目：Babel ~50-100s / SWC ~5-10s / esbuild ~1-5s。100x 差距。
  7. 选择规则：库作者 → Babel + preset-env（最广 target 兼容）；app → esbuild / SWC（速度优先）；Next.js → SWC 默认；Vite → dev esbuild + build esbuild 或 Rollup（含 Babel/SWC plugin）。

02 底层逻辑

Babel：JS 实现 + 插件最大

历史：2014 Sebastian McKenzie 发布 6to5（后改名 Babel）。原始动机：让 ES6 代码跑在 ES5 浏览器（IE 11 / 旧 Safari）。Babel 通过 AST transform 实现"转译现代语法到旧语法"。

架构：

  1. parse：源码 → AST（用 @babel/parser）
  2. transform：AST → AST（plugins 链式应用）
  3. generate：AST → 源码（用 @babel/generator）

plugin / preset 系统：

  • plugin：单个 transform（如 @babel/plugin-transform-arrow-functions）
  • preset：plugin 集合（如 @babel/preset-env / @babel/preset-react / @babel/preset-typescript）
  • preset-env 的特殊性：根据 browserslist target 自动选择需要的 plugin（不会过度转译）

典型 babel.config.js：

module.exports = {
  presets: [
    ['@babel/preset-env', {
      targets: '> 0.25%, not dead',
      modules: false,         // 不转 ESM 模块
      useBuiltIns: 'usage',    // 自动 polyfill
      corejs: 3
    }],
    '@babel/preset-react',
    '@babel/preset-typescript'
  ],
  plugins: [
    'babel-plugin-transform-react-remove-prop-types',
    '@babel/plugin-proposal-decorators'
  ]
};

Babel 慢的原因

  • JS 实现：V8 单线程；字符串 / object 操作；GC 压力大
  • 插件链：每个 plugin 独立 visit AST → 多次遍历
  • AST 重建：传 AST 在 plugin 间用 JS object → 内存压力
  • preset-env 复杂：根据 target 决定哪些 plugin → 大量 if-else

结果：Babel transform 一个 TS 文件 ~100-200ms 是常见。1000 文件项目 ~3-5 分钟。

esbuild：Go 实现 + 极致速度

历史：2020-01 Evan W（化名 evanw，前 Figma 工程师）发布 esbuild 0.1。设计目标：构建速度 100x。

关键设计选择：

(1) Go 实现—— Goroutine 让真正多核并行。Babel/SWC 也并行（multi-process），但 Go 的 goroutine 比启动 process 轻量。

(2) AST in memory + 自家 parser—— 不依赖现有库；为 esbuild 极致优化。Go 的 GC 比 V8 GC 对 compiler 工作负载更友好。

(3) 单 pass—— parse + transform + bundle 在一次扫描完成（不像 Babel 多次 visit AST）。

(4) 零依赖—— 自实现 minifier / tree shaker / sourcemap。不调外部 npm 包。

(5) 字符串处理优化—— Go []byte 操作 + 避免不必要 allocation。

Evan W 的 essay "Why is esbuild fast?" 详细分析这些选择。

esbuild plugin API 限制

esbuild 只暴露resolve / load hooks：

esbuild.build({
  plugins: [{
    name: 'my-plugin',
    setup(build) {
      // 1. 控制 resolve
      build.onResolve({ filter: /\.svg$/ }, (args) => ({
        path: args.path,
        namespace: 'svg-ns'
      }));

      // 2. 控制 load
      build.onLoad({ filter: /\.svg$/ }, (args) => ({
        contents: 'export default ...',
        loader: 'js'
      }));
    }
  }]
});

不能 AST transform—— 不能像 Babel/SWC 那样在 transform 阶段改 AST。这让 esbuild 不能完全替代 Babel/SWC（如果你需要复杂的 AST 操作）。

设计权衡：

  • 没 AST plugin → 速度极致（不需暴露 AST 给 plugin → 可以高度优化内部）
  • 有 AST plugin → plugin 灵活但慢（要序列化 AST + plugin 间通信）

esbuild 选了前者。

SWC：Rust 实现 + 兼容 Babel

历史：2020 DongYoon Kang（后被 Vercel 雇全职做 SWC）发布 SWC 1.0。定位：Babel 替代 + 速度接近 esbuild。

关键设计：

(1) Rust 实现—— 性能接近 Go；内存安全更强；与 Rust ecosystem 集成（Turbopack / Rspack 等）

(2) Babel-compatible API—— 大部分 Babel preset 在 SWC 有对应（@swc/preset-env / @swc/preset-react）

(3) Wasm plugin—— 让 Rust 写 plugin → 编译 wasm（详见 WASM 主题）→ SWC 加载执行。性能损失小（vs Native plugin）。

(4) Next.js 默认—— 2022 Next.js 12 默认切到 SWC（替代 Babel），Vercel 重金推。

典型 SWC 配置（.swcrc）：

{
  "jsc": {
    "parser": {
      "syntax": "typescript",
      "tsx": true,
      "decorators": true
    },
    "transform": {
      "react": {
        "runtime": "automatic"
      }
    },
    "target": "es2020"
  },
  "module": {
    "type": "es6"
  }
}

SWC 用 wasm plugin

SWC plugin 不像 Babel plugin（JS）；用 Rust 写编译 wasm：

// 我的 SWC plugin
use swc_core::ecma::{ast::*, visit::FoldWith};

#[plugin_transform]
pub fn process_transform(program: Program, _metadata: TransformPluginProgramMetadata) -> Program {
    // 自定义 transform
    program.fold_with(&mut MyVisitor {})
}
$ cargo build --target wasm32-wasi --release
# 产 my_plugin.wasm

# 配置 .swcrc
{
  "jsc": {
    "experimental": {
      "plugins": [["./my_plugin.wasm", { ... }]]
    }
  }
}

性能：wasm plugin 比 native plugin 慢 ~2x，但比 Babel plugin（JS）快 ~10x。

三家性能对比

典型 TS 项目（500 文件，含 React JSX）：

步骤BabelSWCesbuild
单文件 transform 平均~100-200ms~5-10ms~3-5ms
500 文件总（serial）~50-100s~3-5s~1-3s
500 文件总（4 核 parallel）~15-30s~1-2s~0.5-1s
Memory peak~500MB-1GB~200-300MB~100-200MB

实际差距 ~30-100x—— 不到营销宣称的 100x，但仍极显著。

三者做的工作

三者都做同样的 transformation：

1. 现代 JS → 老 JS

// 源
const greet = (name) => `Hello, ${name}!`;
class Counter {
  count = 0;
  increment() { this.count++; }
}
async function loadData() {
  const data = await fetch('/api');
  return data.json();
}

// transform 到 ES5（target old browser）
var greet = function(name) { return "Hello, " + name + "!"; };
function Counter() {
  this.count = 0;
}
Counter.prototype.increment = function() {
  this.count++;
};
function loadData() {
  return /* generator wrapper for async */ ...;
}

2. JSX → React.createElement / jsx-runtime

// 源
const App = () => <div className="root">Hello</div>;

// 旧 React (createElement)
const App = () => React.createElement('div', { className: 'root' }, 'Hello');

// React 17+ jsx-runtime
import { jsx as _jsx } from 'react/jsx-runtime';
const App = () => _jsx('div', { className: 'root', children: 'Hello' });

3. TypeScript → JS（类型擦除）

// 源
interface User { name: string; age: number; }
function greet(user: User): string {
  return `Hello, ${user.name}!`;
}

// 类型擦除后
function greet(user) {
  return `Hello, ${user.name}!`;
}

注意：esbuild / SWC / Babel 都不做类型检查—— 只是把类型注解删掉。类型检查要单独跑 tsc 或编辑器（详见 TS P5）。

4. polyfill 注入

// 源
const set = new Set([1, 2, 3]);   // Set 老浏览器没有
arr.includes(x);                   // Array.includes 老浏览器没有

// Babel preset-env useBuiltIns: 'usage' 自动加
import 'core-js/modules/es.set';
import 'core-js/modules/es.array.includes';
const set = new Set([1, 2, 3]);
arr.includes(x);

esbuild / SWC 也支持 polyfill 注入但功能不及 Babel preset-env 完善。

选谁用：决策树

问题 1：你写的是什么？

  • npm 库（要广 target 兼容）→ Babel + preset-env（最稳）
  • Web app → SWC 或 esbuild（速度优先）
  • Next.js → SWC（Next 默认）
  • Vite → 已内置 esbuild（dev/build 都用）

问题 2：你需要什么 plugin？

  • 需要复杂 AST plugin（装饰器 / 编译期 macro / styled-components）→ Babel 或 SWC（不能 esbuild）
  • 只需要标准 transform（TS / JSX / preset-env）→ esbuild / SWC 都行

问题 3：性能优先级？

  • 启动最快 → esbuild
  • 稳态 + 大项目 → SWC（与 Rust ecosystem 集成）
  • 兼容性最重要 → Babel

各场景选择

场景推荐
Vite 项目esbuild（dev）+ Rollup with esbuild plugin（build）
Next.js 项目SWC（默认）
create-react-app~~Babel~~（CRA 弃用，推荐迁 Vite）
Webpack 项目Babel（Webpack 默认 babel-loader）或 SWC（swc-loader）
Rspack 项目SWC（默认）
Bun 项目Bun 内置 transformer
npm 库（广兼容）Babel + @babel/preset-env
npm 库（modern only）tsup（用 esbuild）
极致快 monorepo buildesbuild 直用

Babel 不会消失的原因

虽然 SWC / esbuild 速度碾压，Babel 仍存活：

  • plugin ecosystem 最大（数千个 npm 包）
  • 很多 framework 特定 plugin（React Server Components / styled-components / Solid 等）
  • preset-env 的 polyfill 处理最完善
  • 新 ES proposal 通常先在 Babel 实现（让大家试用）
  • 大量库的 build pipeline 已配 Babel + preset-env，不愿改

2026 趋势：Babel 仍是"规范探索"层（新 proposal 试用）+ "老项目维护"；新 app 项目几乎都用 SWC / esbuild。

tsc / TypeScript compiler 的位置

TypeScript 自带 tsc compiler。但tsc 不适合做 bundler 的 transform：

  • tsc 慢（与 Babel 量级，~100ms/file）
  • tsc 类型检查内置（不能纯类型擦除）
  • tsc 输出格式 / 兼容性配置不够灵活

典型工程实践：

  • tsc：仅做 type checking（tsc --noEmit）
  • esbuild / SWC：做 transform（类型擦除 + 转译）

2024+ 还有 tsgo（Go 重写的 tsc，2024 stage 1）—— 让 tsc 也快 100x。但仍未替代 esbuild / SWC 做 build transform。

compiler 与 bundler 的关系

compiler（Babel/SWC/esbuild）和 bundler（Vite/Webpack/Rollup）不是替代关系：

  • compiler 做 single-file transform（TS → JS / JSX → JS / 等）
  • bundler 做 multi-file 合并 + chunk 分割 + tree shaking

典型组合：

  • Vite (bundler) + esbuild (compiler)
  • Webpack (bundler) + babel-loader (compiler)
  • Webpack + swc-loader
  • Rspack + 内置 SWC
  • Rollup + @rollup/plugin-babel 或 @rollup/plugin-swc

esbuild 自身既是 compiler 也是 bundler，但 bundler 功能不及完整的 Vite/Webpack/Rollup（详见 P1.2）。

03 应用场景

场景 A · Webpack 项目从 babel-loader 切换到 swc-loader

典型 5-10x 加速：

$ npm uninstall babel-loader @babel/core @babel/preset-env @babel/preset-react @babel/preset-typescript
$ npm install -D @swc/core swc-loader
// webpack.config.js
module.exports = {
  module: {
    rules: [
      {
        test: /\.(jsx?|tsx?)$/,
        exclude: /node_modules/,
        use: {
          loader: 'swc-loader',
          options: {
            jsc: {
              parser: { syntax: 'typescript', tsx: true },
              transform: { react: { runtime: 'automatic' }},
              target: 'es2020'
            }
          }
        }
      }
    ]
  }
};

场景 B · 写自定义 SWC plugin（Rust）

$ cargo new --lib my-swc-plugin
$ cd my-swc-plugin
# Cargo.toml
[package]
name = "my-swc-plugin"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
serde = "1"
serde_json = "1"
swc_core = { version = "0.95", features = ["ecma_plugin_transform"] }
// src/lib.rs
use swc_core::ecma::{
    ast::Program,
    visit::{as_folder, FoldWith, VisitMut},
};
use swc_core::plugin::{plugin_transform, proxies::TransformPluginProgramMetadata};

pub struct TransformVisitor;

impl VisitMut for TransformVisitor {
    // 你的 transform 逻辑
}

#[plugin_transform]
pub fn process_transform(program: Program, _metadata: TransformPluginProgramMetadata) -> Program {
    program.fold_with(&mut as_folder(TransformVisitor))
}
$ cargo build --target wasm32-wasi --release

import * as esbuild from 'esbuild';

await esbuild.build({
  entryPoints: ['src/index.ts'],
  bundle: true,
  outfile: 'dist/bundle.js',
  format: 'esm',
  platform: 'browser',

  jsx: 'automatic',

05 常见陷阱

陷阱 1：以为 esbuild 能完全替代 Babel

esbuild plugin 不能 AST transform。很多 Babel 特有 plugin 在 esbuild 不可能：

  • 装饰器（decorators）—— 复杂 transform
  • styled-components Babel plugin（自动加 displayName）
  • 编译期 macro（@babel/plugin-transform-react-constant-elements）

esbuild 适合baseline transform（TS / JSX / target）；复杂 plugin 用 SWC 或 Babel。

陷阱 2：Babel preset-env 默认让 tree shaking 失效

详见 P4 Tree Shaking。重要规则：modules: false 让 Babel 保留 ESM。

陷阱 3：tsc + esbuild 重复 type 检查

esbuild 不做 type 检查（只擦除类型）。tsc 做 type 检查。不要让 esbuild 跑 tsc：

# 错：让 esbuild 串 tsc
$ tsc && esbuild ...   # tsc 慢拖累

# 对：分开跑（CI 同时）
$ tsc --noEmit          # 只 type 检查（CI / pre-commit）
$ esbuild ...           # transform（dev / build）

陷阱 4：SWC plugin 用错 wasm 版本

SWC plugin（wasm）必须用与 SWC 兼容的 swc_core 版本。版本不匹配会报错或行为奇怪。

# Cargo.toml
[dependencies]
swc_core = { version = "0.95", features = ["..."] }   # 必须与目标 SWC 版本兼容

检查：plugins.swc.rs 兼容性表。

陷阱 5：Babel 转 ES5 让 bundle 巨大

preset-env 转 ES5（target IE 11）会加大量 polyfill + helper 代码。2026 不必要—— IE 已死。target 现代浏览器：

{
  "targets": "> 0.5%, last 2 versions, Firefox ESR, not dead, not IE 11"
}

能省 30-50% bundle 体积。

陷阱 6：用 swc-loader 但忘 SWC 配置

不配置 .swcrc 时 swc-loader 用默认（target es2015）。可能输出过老或丢失你需要的特性。显式配置：

// webpack.config.js
{
  loader: 'swc-loader',
  options: {
    // 显式 jsc 配置
  }
}

陷阱 7：esbuild minify 行为差异

esbuild minify 不及 terser 激进。如果你的代码靠 Function.name / Class.name 等元信息，esbuild 默认 mangle 让 name 变化：

// 源
class MyComponent { ... }
console.log(MyComponent.name);   // "MyComponent"

// esbuild minify 后
class A { ... }   // ← name 变了！
console.log(A.name);   // "A" 而不是 "MyComponent"

对策：minify: { keepNames: true }（esbuild）；或用 terser。

陷阱 8：忘记 react-refresh/babel plugin

用 Babel + Webpack 时，要用 react-refresh/babel 让 React Refresh 工作：

// babel.config.js
{
  plugins: [
    process.env.NODE_ENV === 'development' && 'react-refresh/babel'
  ].filter(Boolean)
}

+ webpack.config.js 加 ReactRefreshWebpackPlugin。

用 SWC 时直接 jsc.transform.react.refresh: true。

用 Vite 时 @vitejs/plugin-react 自动包含。
