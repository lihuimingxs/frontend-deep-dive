Source Maps 工程化

Source maps 是调试体验基础——没它，production 错误几乎无法 debug，dev 时 minified 代码看不到原 TS / JSX。v3 格式 = JSON + mappings (VLQ encoded) + sources + names。VLQ 变长 base64 让 sourcemap 紧凑。多层 sourcemap（composite）：source → TS → JS → bundled → minified；浏览器 devtools 自动追踪到原始 .ts。生产部署：inline 体积大不推荐 / external 浏览器按需加载 / 私有 CDN + Sentry 等错误追踪服务解析 stack trace。

阅读时长 ~25 分钟 前置 P6.1（编译器后端） 主要来源 Source Map v3 spec · Sentry 文档 · Chrome DevTools 文档

01 速览

  1. Source maps 的核心问题：编译后代码（minified / bundled）和源代码完全不同形态。如何让浏览器 devtools 显示原始源代码位置而不是编译后乱码？答案：source map 文件做位置映射。
  2. v3 格式（2011 主流）—— JSON 文件，关键字段：version / file / sources（原文件路径列表）/ names（标识符）/ mappings（VLQ 编码的位置数据）/ sourcesContent（可选，源代码 inline）。
  3. VLQ（Variable Length Quantity）编码—— 变长 base64 编码每行偏移；让 sourcemap 紧凑（vs 全文本 JSON）。每个 segment 5 个值：generated column / source index / source line / source column / name index。
  4. 多层 sourcemap（composite）—— TS → JS（map 1）→ bundled JS（map 2）→ minified（map 3）。每步必须传递 sourcemap，任何一步丢就断链。浏览器自动追踪到最原始 .ts。
  5. 生产部署 3 种策略：(1) inline（//# sourceMappingURL=data:...）—— 体积大不推荐；(2) external file（//# sourceMappingURL=foo.js.map）—— 浏览器按需加载，主流；(3) private CDN + 错误追踪服务（Sentry / Datadog 解析 stack trace）—— 不暴露公网。
  6. 常见问题：HMR 后 sourcemap 错位 / Babel + esbuild 链 sourcemap 丢失 / Production stack trace 显示 minified 名字。都是 sourcemap 链路某一步失效。
  7. 2026 工程化：sourcemap 必须上传 Sentry / Datadog 让生产 stack trace 可读；不部署 .map 到 public CDN（暴露源代码）。

02 底层逻辑

问题：编译后代码和源代码不一样

用户写：

// src/Counter.tsx
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

浏览器收到（minified + bundled）：

function p(){let[t,r]=u.useState(0);return o.jsx("button",{onClick:()=>r(t+1),children:t})}

报错时浏览器只能显示 minified 位置：chunks/index-abc123.js:1:42。无法 debug。

Source map 解决：附加一个 .js.map 文件，告诉浏览器 "minified line 1 col 42 = 原文件 src/Counter.tsx line 4 col 24"。浏览器 devtools 自动还原。

Source Map v3 格式

v3（2011 起，至今主流）是 JSON 文件：

{
  "version": 3,
  "file": "out.js",
  "sourceRoot": "",
  "sources": ["foo.ts", "bar.ts"],
  "sourcesContent": ["// foo.ts content...", "// bar.ts content..."],
  "names": ["myFunc", "x", "y"],
  "mappings": "AAAA;;;;;;;;;;;;;;;;CACO,oBAAS"
}

字段含义：

  • version: 3
  • file: 关联的输出文件名
  • sources: 原文件路径列表
  • sourcesContent:（可选）原文件内容（用于网络条件差时浏览器无需 fetch）
  • names: 标识符（变量名 / 函数名）列表
  • mappings: 核心 —— VLQ 编码的位置映射数据

VLQ 编码

VLQ = Variable-Length Quantity。变长 base64 编码整数，让 sourcemap 紧凑。

mappings 结构：

  • 用 ; 分隔每行（generated line）
  • 用 , 分隔每个 segment
  • 每个 segment 含 1, 4, 或 5 个值

每 segment 的值（差量编码 = 相对前一个 segment）：

  1. generated column（输出列）
  2. source index（sources 数组索引）
  3. source line（源代码行）
  4. source column（源代码列）
  5. name index（names 数组索引）—— 可选

例子：AAAA（1 个 segment）：

  • 4 个 base64 字符 = 4 个 VLQ 数字
  • A = 0（base64）
  • 所以是 [0, 0, 0, 0] = 输出列 0，source index 0，源 line 0，源 col 0

差量编码省空间：连续行的 segment 通常增量小（generated column +1, source line +0, source column +1）→ 编码出来 1-2 字节。

VLQ 编码 base64 字符表：

A-Z = 0-25
a-z = 26-51
0-9 = 52-61
+ = 62
/ = 63

每字符前 5 bit 是数据，第 6 bit 是 continuation flag
如 'B' (1) → binary 000001 → flag=0, value=1
如 'C' (2) → binary 000010 → flag=0, value=2 (差量 -1 用 zigzag 编码)

sourcemap 加载机制

编译器输出 .js 文件末尾加注释：

function p(){...}
//# sourceMappingURL=out.js.map

浏览器 devtools 检测到这个注释，自动 fetch out.js.map（仅在 devtools 打开时）。然后用 mappings 数据：

  1. 用户在 devtools 看到 stack trace（指向 out.js）
  2. devtools 查 sourcemap：out.js:1:42 → src/Counter.tsx:4:24
  3. devtools 显示原 src/Counter.tsx 内容（从 sourcesContent 或 fetch）

用户感觉就是直接调试源码。

多层 sourcemap（composite）

实际项目编译有多步：

src/Counter.tsx           ← 原始源
  ↓ TS compile (esbuild)
.tmp/Counter.js + Counter.js.map   ← map 1
  ↓ Bundle (Rollup)
dist/index.js + index.js.map         ← map 2 (链接到 .tmp)
  ↓ Minify (terser)
dist/index.min.js + index.min.js.map  ← map 3 (链接到 dist)

每一步都必须正确传递 sourcemap：

  • esbuild 输出 .js + .js.map（指向 .tsx）
  • Rollup 接受 input map → 合并 → 输出新 map（指向最原始 .tsx）
  • terser 接受 input map → 输出最终 map（仍指向 .tsx）

浏览器只看最终 map—— 它已经 chain 到原始 .tsx。所以浏览器 devtools 直接显示 .tsx 源码。

这种"链式 sourcemap"是 v3 设计精髓。任何一步丢 sourcemap → 链断 → 浏览器看不到原 .tsx。

build.sourcemap 选项详解

Vite/Rollup 的 sourcemap 选项：

选项值含义
true生成 .map 文件 + 加 sourceMappingURL 注释
'inline'整个 sourcemap inline 到 .js（base64 data URL）
'hidden'生成 .map 文件但不加 sourceMappingURL 注释（浏览器不自动加载）
false不生成（dev 用，体积小）

生产推荐 'hidden'—— 生成 .map 文件，上传到 Sentry / Datadog，但不部署到 public（用户浏览器不能 fetch，源代码不暴露）。

sourcemap 安全考虑

生产 sourcemap 等于暴露源代码：

  • sourcesContent 字段含原代码全文
  • 没有它仍能 fetch sources URL（如果 .ts 部署）
  • 用户直接看 minified .js + .map → 完全等同看源代码

除非是 open source 项目，否则 sourcemap不应该部署到 public。

正确做法：

  1. build 时生成 sourcemap（sourcemap: 'hidden'）
  2. CI 上传 .map 到 Sentry / Datadog（错误追踪服务）
  3. 不把 .map 部署到 public CDN
  4. 生产 stack trace 在错误追踪服务里被自动解析（用 sourcemap 把 minified 位置 → 原始位置）

Sentry / Datadog 集成

Sentry 是错误追踪标准。流程：

1. 用户浏览器报错 → Sentry SDK 上报 stack trace（minified 位置）
   "Error in chunk-abc123.js:1:42"

2. Sentry 服务端：
   - 找对应的 release / sourcemap
   - 用 sourcemap 把 minified 位置 → 原 .tsx 位置
   - 在 Sentry UI 显示：
     "Error in src/Counter.tsx:4:24"

3. 开发者打开 Sentry 看到原始位置 + 上下文（前几行 / 后几行代码）

Sentry CLI 上传 sourcemap：

$ sentry-cli sourcemaps upload \
  --org=my-org \
  --project=my-app \
  --release=v1.2.3 \
  ./dist

Sentry 用 release / commit hash 关联 sourcemap 和实际部署版本。

HMR 时 sourcemap 处理

Vite dev mode HMR 后，sourcemap 仍要保持准确：

  • esbuild dev transform 自动生成 sourcemap
  • HMR 替换模块时，浏览器收到新 .js + 新 sourcemap
  • devtools 自动用新 sourcemap 显示

常见问题：HMR 后 stack trace 偶尔错位 → 可能是 plugin 没正确传 sourcemap。

compiler 链 sourcemap 传递

多 compiler 链（Babel + esbuild + Rollup）每步都要传 sourcemap。如果某步 plugin 不正确：

src/foo.ts (line 10)
  ↓ Babel transform   (会丢 sourcemap 如果 plugin bug)
foo.js (line 5)
  ↓ Rollup bundle      (Rollup 接 input map → 新 map)
dist/index.js (line 100)

最终 sourcemap 应该指向 foo.ts:10
但如果 Babel 丢了 → sourcemap 指向 foo.js:5（无意义中间产物）

调试方法：

  • 用 source-map-cli 解析 .map 文件看实际指向
  • 检查每个 plugin 输出 RawSourceMap 是否正确

sourcemap 大小考虑

典型 sourcemap 大小：

  • 50KB JS bundle → ~80-150KB sourcemap（含 sourcesContent）
  • 500KB JS bundle → ~800KB-1.5MB sourcemap

sourcemap 比 JS 还大是常态（因为 sourcesContent 含全部源码 + mappings 数据）。不要部署 sourcemap 到 public。

devtoolModuleFilenameTemplate（Webpack 特性）

Webpack 让你自定义 sourcemap 中 sources 字段的路径格式（debug 用）：

module.exports = {
  output: {
    devtoolModuleFilenameTemplate: 'webpack:///[resource-path]?[hash]'
  }
};

结果：sourcemap 里 sources = ['webpack:///./src/Counter.tsx?abc']（自定义协议 + 路径）。

Vite 默认行为通常足够，不需要自定义。

03 应用场景

场景 A · 生产部署 sourcemap 到 Sentry（不暴露公网）

Vite 配置：

// vite.config.js
export default {
  build: {
    sourcemap: 'hidden'   // 生成 .map 但不加 sourceMappingURL
  }
};

build 后 dist/ 里有 .js 和 .js.map。部署只 .js，把 .map 上传 Sentry：

$ npm run build
$ sentry-cli sourcemaps upload \
  --org=my-org --project=my-app --release=$VERSION \
  dist/

# 部署仅 .js（不 .map）
$ rsync -av --exclude='*.map' dist/ server:/path/

# 或部署后删 .map
$ ssh server "find /path -name '*.map' -delete"

场景 B · 解析生产 stack trace

用户上报 bug："点击 button 报错，console 显示 chunk-abc123.js:1:42"。

方案 1：去 Sentry 看（自动解析）

方案 2：手动用 source-map-cli：

$ npm install -g source-map-cli
$ source-map-cli resolve dist/chunk-abc123.js.map 1 42
{
  "source": "src/Counter.tsx",
  "line": 4,
  "column": 24,
  "name": "setCount"
}

场景 C · 调试 dev 时 sourcemap 错位

典型现象：浏览器 devtools 显示原 .ts 但 line number 错（指向 .ts 第 5 行实际是源代码第 10 行）。

可能原因：

  • 某个 Vite plugin 修改代码但没传 sourcemap
  • Babel preset 配置问题（modules: "auto" 让 ESM 转 CJS）
  • HMR transform 缺 sourcemap

诊断：临时用 inline sourcemap 看实际生成的 .js + map：

// vite.config.js
export default {
  build: { sourcemap: 'inline' }   // dev 也可以试 build 看
};

用浏览器 view-source: 看 .js 末尾的 base64 sourcemap → 解 base64 → 检查 mappings。

场景 D · 大型 monorepo sourcemap path 问题

monorepo 多个 package：

my-monorepo/
├── packages/
│   ├── ui/
│   ├── utils/
│   └── app/
└── ...

app 引用 ui / utils，build 后 sourcemap 中的 sources 字段：

"sources": [
  "../ui/src/Button.tsx",        // 相对于 app/dist
  "../utils/src/format.ts"
]

路径是相对的。部署到 server 时，server 上没有这些 ../ → Sentry 找不到 source content。

对策：

  • 用 sourcesContent 字段（inline 源代码到 sourcemap）
  • Vite/Rollup 默认包含 sourcesContent
  • Sentry 优先用 sourcesContent

场景 E · 自定义 sourcemap 路径前缀（隐私）

不想 sourcemap 中暴露绝对路径（如 /Users/myname/myproject/src/Counter.tsx）：

// vite.config.js (Rollup 配置)
export default {
  build: {
    rollupOptions: {
      output: {
        sourcemapPathTransform: (relativePath) => {
          // 重写路径，去掉个人路径
          return relativePath.replace(/^\.\.\/\.\.\/Users\/.*?\//, 'webpack:///');
        }
      }
    }
  }
};

04 典型代码：sourcemap 完整工程化

Vite 生产 sourcemap 配置

// vite.config.ts
import { defineConfig } from 'vite';
import { sentryVitePlugin } from '@sentry/vite-plugin';

export default defineConfig(({ mode }) => ({
  build: {
    sourcemap: 'hidden',          // 生成但不在 .js 加 sourceMappingURL
    // 或 'inline' 用于 dev 调试
  },

  plugins: [
    react(),

    // Sentry 自动上传 sourcemap
    mode === 'production' && sentryVitePlugin({
      org: process.env.SENTRY_ORG,
      project: process.env.SENTRY_PROJECT,
      authToken: process.env.SENTRY_AUTH_TOKEN,
      sourcemaps: {
        assets: ['./dist/**/*.js', './dist/**/*.map'],
        ignore: ['node_modules'],
        filesToDeleteAfterUpload: ['./dist/**/*.map']   // 上传后删
      },
      release: {
        name: process.env.RELEASE_VERSION
      }
    })
  ].filter(Boolean)
}));

这个 plugin 会：

  1. build 完后扫 dist/*.map
  2. 上传到 Sentry（关联 release）
  3. 上传成功后删 dist/*.map
  4. 部署 dist 不含 .map

验证 sourcemap 正确

$ npx source-map-cli validate dist/assets/index-*.js.map

dist/assets/index-abc123.js.map
  Version: 3
  Sources: 156
  Has sourcesContent: true
  Has names: true
  Mappings entries: 5432
  ✓ All sources accessible

手动解析 stack trace

// resolve-stack.ts
import { SourceMapConsumer } from 'source-map';
import * as fs from 'fs';

const mapContent = fs.readFileSync('./dist/assets/index-abc123.js.map', 'utf8');
const consumer = await new SourceMapConsumer(mapContent);

const result = consumer.originalPositionFor({
  line: 1,
  column: 42
});

console.log(result);
// {
//   source: 'src/Counter.tsx',
//   line: 4,
//   column: 24,
//   name: 'setCount'
// }

consumer.destroy();

看 inline sourcemap 内容

# build with inline sourcemap
$ vite build --sourcemap inline

# 查看
$ tail -1 dist/assets/index-*.js
//# sourceMappingURL=data:application/json;base64,eyJ2ZXJzaW9uIjoz...

# 解码 base64 + 看 JSON
$ tail -1 dist/assets/index-*.js | sed 's/.*base64,//' | base64 -d | jq '.sources'
[
  "src/main.tsx",
  "src/App.tsx",
  "src/Counter.tsx"
]

测试 sourcemap chain

// test-sourcemap.ts
import { SourceMapConsumer, SourceMapGenerator } from 'source-map';

// 模拟多步 transform
const stage1Map = '...'; // TS → JS map
const stage2Map = '...'; // JS → bundled map

// 链合并
const stage1 = await new SourceMapConsumer(stage1Map);
const stage2 = await new SourceMapConsumer(stage2Map);

const merged = SourceMapGenerator.fromSourceMap(stage1);
merged.applySourceMap(stage2, '...');

console.log(merged.toString());

devtools workspace 持久化修改

Chrome devtools 支持 "workspace" —— 让 sourcemap 显示的源文件可以直接编辑保存到磁盘（dev 用）：

  1. devtools → Sources → 右键 → "Add folder to workspace"
  2. 选你的项目根目录
  3. devtools 自动 mapping 显示的 .ts 到本地文件
  4. 编辑直接保存

结合 Vite HMR 让"在 devtools 编辑代码 + 立刻 HMR" 可行。

05 常见陷阱

陷阱 1：sourcemap 部署到 public 暴露源代码

最常见 + 最危险。检查 production 部署：

$ curl -I https://my-app.com/assets/index-abc.js.map
HTTP/1.1 200 OK   # ❌ 别人能直接下载 sourcemap → 拿到完整源代码！

# 应该是
HTTP/1.1 404 Not Found   # ✅

修复：build 用 sourcemap: 'hidden' + 上传 Sentry + 部署不含 .map。

陷阱 2：plugin 链丢 sourcemap

典型：自定义 Vite plugin 做 transform 但没返回 map：

// ❌ 错：没返回 map
{
  name: 'my-plugin',
  transform(code, id) {
    return modifiedCode;   // 丢了 sourcemap
  }
}

// ✅ 正：返回 { code, map }
{
  name: 'my-plugin',
  transform(code, id) {
    const result = transform(code);   // 假设你的 transformer 返回 sourcemap
    return {
      code: result.code,
      map: result.map
    };
  }
}

陷阱 3：Babel modules: "auto" 破坏 sourcemap

Babel preset-env 默认 modules: "auto"（在 build 环境转 CJS）。这会让一些 plugin 链失效。

对策：明确 modules: false（让 bundler 处理 ESM）。

陷阱 4：HMR 后 stack trace 错位

HMR 替换模块时，浏览器旧 stack trace 引用旧 sourcemap，但页面已加载新代码 → 显示错位。

典型现象：刷新页面后正常，HMR 后偶尔错位。

对策：通常无需特别处理（HMR 是 dev 行为，刷新即可）。如果频繁出现可能是 Vite plugin bug，开 issue。

陷阱 5：sourcesContent 缺失但 .ts 文件不部署

sourcemap 默认含 sourcesContent（源代码 inline）。如果关掉 sourcesContent（节省空间）但部署时没把 .ts 放出来：

浏览器收到 sourcemap：
  sources: ['src/Counter.tsx']
  sourcesContent: null

浏览器尝试 fetch /src/Counter.tsx → 404
显示空源代码

对策：保留 sourcesContent（默认 OK）；或确保 .ts 文件能 fetch 到。

陷阱 6：自定义 sourcemap path 让 Sentry 找不到

// ❌ 用 webpack:// 协议
sourcemapPathTransform: (path) => `webpack:///${path}`

// Sentry 接收到 sourcemap，sources 是 ["webpack:///src/Counter.tsx"]
// Sentry 找不到对应文件 → stack trace 显示原始 minified 位置

对策：保持简单的相对路径。

陷阱 7：minify 后还以为有名字

// 源
function MyComponent() { ... }
console.log(MyComponent.name);   // "MyComponent"

// minified
function p() { ... }
// 即使有 sourcemap，运行时 .name 是 "p" 不是 "MyComponent"

sourcemap 让 devtools 看代码位置，但不影响运行时 name 属性。

对策：esbuild minify: { keepNames: true }；terser keep_classnames: true, keep_fnames: true。

陷阱 8：Webpack devtool 选错

Webpack devtool 选项有几十个：

  • 'eval'—— 最快 dev，但 sourcemap 弱
  • 'eval-source-map'—— 平衡
  • 'source-map'—— 标准 + 完整
  • 'hidden-source-map'—— production 推荐
  • 'nosources-source-map'—— 不含 sourcesContent

选错让 sourcemap 不准 / production 暴露源码。Vite 简化为 true / 'inline' / 'hidden' / false，更清晰。
