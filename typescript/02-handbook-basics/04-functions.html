04 · Functions

函数是 TS 类型系统最复杂的部分之一 —— 不只是「参数和返回值的类型」，还涉及 callable 与 newable 的区分、泛型延迟绑定、function overload、this 参数、void / never / unknown 三种特殊返回类型的细节差异。本章覆盖函数类型的 9 个维度，重点回答为什么 forEach 回调可以 return 任意值、为什么 overload 实现签名宽松而对外签名精确、什么时候用 never 兜底。

阅读时长 ~30 分钟 前置 02 Everyday Types 对应 typescriptlang.org Handbook · More on Functions

01 速览

TS 描述函数类型有两种语法：function type expression（短）和 call signature（在对象/接口里）：

typescript 
// 形式 A：function type expression（变量 / 参数 / 返回值类型最常用）
type Greeter = (name: string) => string;
const g: Greeter = (name) => `Hello, ${name}`;

// 形式 B：call signature（描述带额外属性的可调用对象）
interface Counter {
  (start: number): string;     // 可调用
  count: number;                // 还有自己的属性
  reset(): void;
}

// 形式 C：construct signature（描述构造器）
interface UserCtor {
  new (name: string): User;
}

函数类型的三大变量：参数（数量、可选、rest）、返回值（含 void/never）、泛型参数（约束、默认）。再叠加 overload、this 参数、解构注解，构成全谱。

02 底层逻辑

函数类型的几个非直觉决策都和 JS 的"宽松调用约定"有关。理解它们的设计动机就理解了 TS 在易用性与严格性之间的取舍。

1. void 不等于"必须返回 undefined"

表面上 void 表示"无返回值"，但 TS 的 void 返回类型有一个特殊宽松规则：调用方期望 void 时，函数实际返回任意值都允许。

typescript 
type VoidFn = () => void;

const f1: VoidFn = () => 42;          // ✓ 返回 number 也通过
const f2: VoidFn = () => "hello";     // ✓ 返回 string 也通过

// 但调用方拿到的类型仍是 void
const r = f1();   // r: void（不是 number）

为什么这样设计？最经典的例子是 Array.prototype.forEach：

typescript 
// forEach 的类型签名（简化）：
// forEach(callback: (value: T) => void): void;

const src = [1, 2, 3];
const dst: number[] = [];

src.forEach(x => dst.push(x));
//                ^^^^^^^^^^^ push 返回 number，但回调期望 void
//                          ⇒ 如果 void 严格要求 undefined，这一行会报错

TS 的设计：声明 void 表达"我不在乎你返回什么"，让调用方写得自由。但函数定义自己是 void 时，函数体内 return x（带值）仍会报错。

这条规则只对函数类型（赋值兼容性）适用，不影响函数定义本身的检查。

2. never 是"永不返回"，不是"返回 nothing"

never 描述的是函数永远到不了 return 那一行的情况：

  • 函数体一定会抛错：function fail(msg: string): never { throw new Error(msg) }
  • 死循环：function loop(): never { while (true) {} }
  • 调用 process.exit()

never 是所有类型的子类型（"底类型"），可以赋给任何位置：

typescript 
function fail(msg: string): never {
  throw new Error(msg);
}

const x: number = fail("nope");   // ✓ never 可以赋给 number
const y: string = fail("nope");   // ✓ never 可以赋给 string

// 应用：写"必须取到值"的辅助函数
function getOrThrow<T>(value: T | undefined): T {
  if (value === undefined) return fail("missing");
  return value;
}

组合：03 章已经讲过 never 在 exhaustive check 中的兜底用法。两种用法本质相通 —— never 表示"这里逻辑上不该到达"。

3. Overload signature 不参与实现

Function overload 的语法：多个签名 + 一个实现。调用方只能看签名，实现签名只是给实现者看的：

typescript 
// 对外可见的两个签名
function len(x: string): number;
function len(x: unknown[]): number;
// 实现签名 — 必须能"覆盖"所有 overload 签名，但调用方看不见
function len(x: string | unknown[]): number {
  return x.length;
}

len("abc");      // ✓ 匹配第一个签名
len([1, 2, 3]);  // ✓ 匹配第二个签名
len(123);        // ❌ 没有签名能匹配 number

关键设计决策：实现签名宽松，对外签名精确。这是为什么实现签名常写 x: string | unknown[] 这种 union，对外暴露的两个签名却是分开的。调用方拿到的类型信息是更精确的版本。

但 overload 在 TS 里属于偏复杂的特性。能用 union 解决的就用 union 解决：

typescript 
// ❌ 不必要的 overload
function len(x: string): number;
function len(x: unknown[]): number;
function len(x: string | unknown[]): number { return x.length; }

// ✅ 直接 union 更简洁，行为一样
function len(x: string | unknown[]): number {
  return x.length;
}

真正需要 overload 的场景：返回类型依赖参数类型（不同输入有不同输出），靠 union 表达不了：

typescript 
// 参数是 string → 返回 string；参数是 number → 返回 number
function double(x: string): string;
function double(x: number): number;
function double(x: string | number): string | number {
  return typeof x === "string" ? x + x : x * 2;
}

const a = double("ab");   // a: string
const b = double(3);      // b: number

但即便这种情况，conditional types（09 章）通常是更好的解：function double<T extends string | number>(x: T): T extends string ? string : number。

4. 泛型在 TS 中是延迟绑定

泛型不是 any 的代名词。T 是占位符，在调用处被某个具体类型替代，所有出现 T 的位置同步替换。

typescript 
function firstElement<T>(arr: T[]): T | undefined {
  return arr[0];
}

const a = firstElement(["a", "b"]);    // a: string | undefined
const b = firstElement([1, 2]);        // b: number | undefined
const c = firstElement([true]);        // c: boolean | undefined

调用 firstElement(["a"]) 时，TS 推断 T = "a" 的"宽化"形式（即 string），把所有 T 替换成 string。这是使用点绑定，不是定义点绑定。

约束让 T 获得"最少能力"：

typescript 
// T 必须有 length 属性，函数体才能用 .length
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

longest([1, 2], [1, 2, 3]);    // ✓ 数组有 length
longest("abc", "ab");           // ✓ 字符串有 length
longest(10, 100);               // ❌ number 没有 length

5. "如果一个类型参数只出现一次，几乎肯定不需要它"

这是 TS 官方手册的明确建议。理由是：泛型的目的是建立两个或多个位置之间的关联。只出现一次说明没有关联可建：

typescript 
// ❌ T 只用了一次 — 多此一举
function greet<T extends string>(s: T): void {
  console.log(s.toUpperCase());
}

// ✅ 直接用具体类型
function greet(s: string): void {
  console.log(s.toUpperCase());
}

// ✅ T 出现两次（参数 + 返回值）— 真正需要泛型
function identity<T>(x: T): T {
  return x;
}

单次出现的泛型常常是"看起来通用但实际没用"的样板代码。删掉换成 string 等具体类型一般是改进。

6. this 参数：编译期约束运行时绑定

JS 函数里 this 的值由调用方式决定（4 规则：默认 / 隐式 / 显式 / new；详见 JS this 四规则）。TS 提供了 this 参数语法，让函数声明它期望被怎样调用：

typescript 
interface User {
  name: string;
  greet(this: User): string;
}

const u: User = {
  name: "Alice",
  greet() {            // this 被声明为 User
    return `Hi ${this.name}`;
  },
};

u.greet();             // ✓ this 是 u（User）

// 但
const fn = u.greet;
fn();                  // ❌ The 'this' context of type 'void' is not assignable to method's 'this' of type 'User'

关键：this: User 不是真实参数，编译产物里完全消失。它只是一个特殊位置的类型注解，告诉 TS"调用此函数时 this 必须是 User"。配合 strictBindCallApply 后 fn.call(notAUser) 也会报错。

7. Function 类型相当于半个 any

有些人会写 let cb: Function = ...，看起来像"任意函数"。这是反模式：

typescript 
// ❌ Function 类型几乎等于 any
function caller(cb: Function) {
  cb();              // 通过
  cb(1, 2, 3);       // 通过
  cb.foo();          // 也通过 — 因为 Function 上没有形状约束
  return cb;         // 通过
}

// ✅ 用具体的函数类型
function caller(cb: () => void) { cb() }
function caller<A extends unknown[], R>(cb: (...args: A) => R, ...args: A): R { return cb(...args) }

typescript-eslint 的 @typescript-eslint/no-unsafe-function-type 规则会拦住 Function 类型。永远写具体签名。

03 应用场景

场景 A · Express / Hono middleware 的 void 返回

HTTP framework middleware 的 handler 类型几乎都是 void 返回，让你能写 return res.json(...) 也能写 res.json(...)：

typescript 
type Handler = (req: Request, res: Response, next: NextFunction) => void;

const auth: Handler = (req, res, next) => {
  if (!req.headers.authorization) {
    return res.status(401).json({ error: "Unauthorized" });
    //     ^^^^^^^^^^^^^^^^^^^^^^^^ 返回 Response — void 不在乎
  }
  next();
};

// 如果 Handler 改成 (...) => undefined，上面那行就要拆成两行（先调用再 return）

这是 void 设计的最大价值场景。Express、Hono、Koa、Fastify 等都用同一套 idiom。

场景 B · 类型安全的 fetch wrapper

泛型让 fetch 的返回类型由调用方决定：

typescript 
async function api<T>(url: string, init?: RequestInit): Promise<T> {
  const res = await fetch(url, init);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json() as Promise<T>;
}

// 调用方指定预期类型
interface User { id: string; name: string }
const user = await api<User>("/api/users/1");
//    ^? User

⚠️ 这种模式不安全 —— res.json() 实际类型是 unknown，但被强制 cast 成 T。生产代码应配合 zod 在边界校验：

typescript 
async function api<T>(url: string, schema: z.ZodType<T>, init?: RequestInit): Promise<T> {
  const res = await fetch(url, init);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return schema.parse(await res.json());   // 运行时校验
}

const UserSchema = z.object({ id: z.string(), name: z.string() });
const user = await api("/api/users/1", UserSchema);
//    ^? { id: string; name: string }

场景 C · React 事件 handler 类型

React 给所有 DOM 事件提供了泛型类型：

typescript 
import { ChangeEvent, FormEvent, MouseEvent } from "react";

function MyForm() {
  const onSubmit = (e: FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    const data = new FormData(e.currentTarget);
  };

  const onChange = (e: ChangeEvent<HTMLInputElement>) => {
    console.log(e.target.value);
  };

  const onClick = (e: MouseEvent<HTMLButtonElement>) => {
    console.log(e.button);
  };

  return (
    <form onSubmit={onSubmit}>
      <input onChange={onChange} />
      <button type="button" onClick={onClick}>Click</button>
    </form>
  );
}

tip：写 onChange={(e) => ...} 内联时 TS 自动从 prop 类型推 e 的类型（contextual typing，详见 17 章）。脱离 JSX 上下文时必须显式注解。

场景 D · ORM 查询的泛型函数

Repository pattern 的标配：

typescript 
interface Entity { id: string }

class Repository<T extends Entity> {
  constructor(private collection: T[]) {}

  findById(id: string): T | undefined {
    return this.collection.find(x => x.id === id);
  }

  findWhere(where: Partial<T>): T[] {
    return this.collection.filter(x =>
      Object.entries(where).every(([k, v]) => (x as any)[k] === v)
    );
  }
}

interface User extends Entity { name: string; age: number }
const users = new Repository<User>([{ id: "1", name: "A", age: 30 }]);
const adults = users.findWhere({ age: 30 });   // adults: User[]
const u = users.findById("1");                  // u: User | undefined

04 典型代码

所有参数形态

typescript 
// 必传
function f1(x: number) {}

// 可选 — 类型自动 union undefined
function f2(x?: number) {}                    // x: number | undefined
f2();                                          // ✓
f2(1);                                         // ✓

// 默认值 — 调用方仍是 number | undefined，函数体内是 number
function f3(x: number = 0) {}
f3();          // ✓
f3(undefined); // ✓
// 注意：默认参数和 ? 不能同时写

// rest
function f4(first: string, ...rest: number[]) {}
f4("a", 1, 2, 3);

// rest 用 tuple 类型表达可变签名
function f5(...args: [string, number] | [boolean]) {}
f5("a", 1);   // ✓
f5(true);     // ✓

// 参数解构 + 默认值
function f6({ name, age = 0 }: { name: string; age?: number }) {}
f6({ name: "Alice" });

泛型函数 + 多个约束

typescript 
// 类型参数互相约束：K 必须是 T 的某个 key
function getProp<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const u = { name: "Alice", age: 30 };
const n = getProp(u, "name");   // n: string
const a = getProp(u, "age");    // a: number
const x = getProp(u, "email");  // ❌ 'email' is not assignable to '"name" | "age"'

// 泛型默认值（TS 5.0+）
function create<T = string>(): T[] {
  return [];
}
const xs = create();          // xs: string[]（用了默认）
const ys = create<number>();  // ys: number[]

// const type parameter（TS 5.0）— 让传入对象自动按 readonly 字面量推断
function pair<const T extends readonly string[]>(x: T): T {
  return x;
}
const p = pair(["a", "b"]);
//    ^? readonly ["a", "b"]   不是 string[]！

function overload 的实战写法

typescript 
// jQuery 风格的 selector
function $(s: string): HTMLElement | null;
function $(s: string, parent: Element): HTMLElement | null;
function $<T extends HTMLElement>(s: string, parent?: Element): T | null {
  return (parent ?? document).querySelector<T>(s);
}

// 时间格式化
function formatDate(d: Date): string;
function formatDate(d: Date, withTime: true): string;
function formatDate(d: Date, withTime: false): string;
function formatDate(d: Date, withTime = false): string {
  return withTime ? d.toISOString() : d.toDateString();
}

Call signature + 函数对象（带属性的可调用对象）

typescript 
// 描述一个既可调用、又有属性的对象
interface Calculator {
  (x: number, y: number): number;     // call signature
  history: string[];                   // 普通属性
  clear(): void;                       // method
}

const add: Calculator = Object.assign(
  (x: number, y: number) => x + y,
  {
    history: [] as string[],
    clear() { this.history.length = 0; },
  },
);

add(1, 2);              // 3
add.history.push("1+2"); // ✓

void 返回的"宽松"演示

typescript 
// 从类型层接受任意返回值的回调
function each<T>(arr: T[], cb: (x: T) => void): void {
  for (const x of arr) cb(x);
}

// 各种返回都通过
each([1, 2], x => console.log(x));         // ✓ 隐式 undefined
each([1, 2], x => x + 1);                  // ✓ 返回 number
each([1, 2], (x) => { return x.toString() }); // ✓ 显式 return string

// 但函数声明自己是 void 时不能 return 带值
function badFn(): void {
  return 42;   // ❌ Type 'number' is not assignable to type 'void'
}

05 常见陷阱

陷阱 1 · 把 void 当 undefined 用

两者表面相似，但语义不同：

typescript 
// undefined：函数确实必须返回 undefined
type StrictFn = () => undefined;
const f1: StrictFn = () => { return; };       // ✓
const f2: StrictFn = () => 42;                  // ❌ Type 'number' is not assignable to 'undefined'

// void：调用方不在乎返回值
type LooseFn = () => void;
const f3: LooseFn = () => 42;                   // ✓
const f4: LooseFn = () => { return; };          // ✓

规则：函数类型描述"调用方期望"用 void；声明"我必须返回 undefined（不返回别的）"用 undefined。

陷阱 2 · overload 顺序错误导致永远匹配不到窄签名

TS 从上到下匹配 overload，第一个匹配的就用。宽签名放前面会"吃掉"后面的窄签名：

typescript 
// ❌ 顺序错
function format(x: string | number): string;
function format(x: string): "string-result";   // 永远匹配不到 — 上面那个先吞了
function format(x: string | number): string {
  return typeof x === "string" ? "string-result" : x.toString();
}

const r = format("abc");
//    ^? string         不是 "string-result"

// ✅ 窄签名放前面
function format(x: string): "string-result";
function format(x: string | number): string;
function format(x: string | number): string {
  return typeof x === "string" ? "string-result" : x.toString();
}

const r = format("abc");
//    ^? "string-result"   ✓

规避：overload 永远窄签名在前，宽签名在后。

陷阱 3 · 用泛型表达 union 是冗余

常见反模式：用 extends string | number 约束泛型，但函数体并不依赖参数类型在 T 上的关联：

typescript 
// ❌ T 只用了一次 + 函数体不需要保留 T 的精确性
function log<T extends string | number>(x: T): void {
  console.log(x);
}

// ✅ 等价的简洁写法
function log(x: string | number): void {
  console.log(x);
}

规避：泛型用于"在两个或多个位置之间建立类型关联"。单次出现就改成具体类型 / union。

陷阱 4 · 箭头函数赋值后 this 丢失

JS 经典陷阱在 TS 里仍然存在：method 拿出来赋值后调用，this 不再是原对象。

typescript 
class Counter {
  count = 0;
  inc() { this.count++; }
}
const c = new Counter();
const fn = c.inc;
fn();   // 运行时：this 是 undefined（strict 模式），this.count++ 炸

// ✅ 解法 1：bind
const fn = c.inc.bind(c);

// ✅ 解法 2：箭头函数 field（自动绑定 this）
class Counter {
  count = 0;
  inc = () => { this.count++; };
}
const c = new Counter();
const fn = c.inc;
fn();   // ✓ 箭头函数捕获了 c

// ✅ 解法 3：声明 this 参数让 TS 拦住
class Counter {
  count = 0;
  inc(this: Counter) { this.count++; }
}
const c = new Counter();
const fn = c.inc;
fn();   // ❌ TS 报错：The 'this' context of type 'void' is not assignable...

规避：React 类组件 / 状态机等场景优先用箭头函数 field；不能改写时用 this: 参数 + strictBindCallApply 让 TS 帮你抓 bug。

陷阱 5 · 函数类型的"参数变型"陷阱（method bivariance）

这是 17 章会展开的内容，但函数章必须先剧透：默认情况下 TS 对 method 参数的类型检查是双变（bivariant），不严格。开启 strictFunctionTypes 后才严格。

typescript 
interface Animal { name: string }
interface Dog extends Animal { bark(): void }

type AnimalCb = (x: Animal) => void;
type DogCb = (x: Dog) => void;

const dogCb: DogCb = (d) => d.bark();

// strictFunctionTypes: false（默认 strict 包含 true）
const cb: AnimalCb = dogCb;   // 不安全 — dogCb 期望 Dog 但被传 Animal
cb({ name: "cat" });            // 运行时 TypeError: bark is not a function

// strictFunctionTypes: true → 这一赋值会报错

规避：永远开 strict: true（默认包含 strictFunctionTypes）。详见 17 章。

陷阱 6 · 用 Function 类型逃避思考

Function 类型几乎等于 any，绕过所有类型检查：

typescript 
// ❌
function once(fn: Function): Function {
  let done = false;
  let result: any;
  return function (this: unknown, ...args: any[]) {
    if (done) return result;
    done = true;
    return result = fn.apply(this, args);
  };
}

// ✅ 用泛型 + tuple 表达"任意签名的函数"
function once<A extends unknown[], R>(fn: (...args: A) => R): (...args: A) => R {
  let done = false;
  let result: R;
  return function (this: unknown, ...args: A): R {
    if (done) return result;
    done = true;
    return result = fn.apply(this, args);
  };
}

规避：永远写具体函数签名。"任意函数"用 (...args: A) => R 配合泛型 A、R 表达。
