03 · Narrowing 与控制流分析

Narrowing 是 TS 比 JS 多出来的最大智能 —— 在每一行代码上，TS 都根据前面的判断自动收窄变量类型。if (x === null) return; x.foo() 中第二行的 x 已经是非空类型，无需再判空。这一章拆解 7 种 narrowing 手段、自定义 type guard 的承诺机制、discriminated union 与 never 兜底如何实现"漏一个 case 编译期就报错"，以及 TS 5.5 引入的推断式 type predicate。

阅读时长 ~35 分钟 前置 02 Everyday Types 对应 typescriptlang.org Handbook · Narrowing

01 速览

一句话定义 narrowing：TS 在 AST 的每个节点位置都维护一份「当前类型快照」。当你写 if / typeof / instanceof / x !== null 等判断时，TS 在 then 分支和 else 分支分别推出更窄的类型。

typescript · 最小例子 
function print(value: string | number | null) {
  // 此处 value: string | number | null

  if (value === null) {
    return;        // 触发 narrowing：return 后 value 不再是 null
  }
  // 此处 value: string | number

  if (typeof value === "string") {
    value.toUpperCase();    // value: string
  } else {
    value.toFixed(2);       // value: number（自动排除 string）
  }
}

没有 narrowing，你每次访问属性都得手写 type guard 或 ! 断言。Narrowing 是让 TS 能写得「像 JS 一样自然」的关键技术。

02 底层逻辑

要写出"TS 自动给力"的代码，必须理解它在每个判断点做了什么、信任什么、在哪里会失效。

七大 narrowing 手段

手段例子触发条件
typeof typeof x === "string" 判断原始类型（含 "undefined" / "object" / "function"）
truthiness if (x) / if (!x) 排除 falsy 值（0 / "" / null / undefined / NaN）
equality x === y / x == null 比较缩窄两侧；== null 同时排 null/undefined
in "foo" in x 对象有特定属性时进入分支
instanceof x instanceof Date 类的实例判断（依赖运行时构造器）
assignment x = "abc" 赋值后 x 类型收窄到赋的值类型
control flow if (...) return; ... 提前返回 / throw 后续代码自动 narrow

这 7 个不是孤立机制，是同一套 control flow analysis 的不同入口：每个判断都把变量类型分成"满足"和"不满足"两支。

control flow analysis：每个语句位置一张类型快照

TS 在 AST 上对每个变量维护一个类型快照表。每经过一个会改变快照的节点，TS 更新它。这不是"代码分析"，而是"在每个 AST 节点都有一份类型环境"。

typescript · 类型快照演示 
function process(input: string | number | null) {
  // 行 1：input: string | number | null
  if (input == null) return;
  // 行 2：input: string | number（== null 同时排 null/undefined）

  if (typeof input === "string") {
    // 行 3：input: string
    input = input.length;
    // 行 4：input: number（赋值改变类型）
  }
  // 行 5：input: number
  return input.toFixed(2);
}

注意第 4 行：赋值改变了变量类型。从此处往后，input 在 TS 看来就是 number，即使变量声明是 string | number | null。这就是 narrowing 的"流"性质。

truthiness 的 5 个 falsy 值

if (x) 排除 JS 所有 falsy 值：0 / "" / null / undefined / NaN（外加 0n / document.all）。

typescript 
function getLength(s: string | null | undefined): number {
  if (!s) return 0;       // 同时排 null/undefined/""
  return s.length;        // s: string（注意：仍可能是 ""，但 .length 没问题）
}

但 truthiness narrow 有个常被忽略的陷阱：0 是 falsy。if (count) 当 count 为 0 时进 false 分支，可能不是你想要的：

typescript 
function showBadge(count: number | undefined) {
  // ❌ count = 0 时也走默认 → 不显示 0 的徽章
  if (count) return <Badge>{count}</Badge>;
  return null;
}

// ✅ 显式判 undefined
if (count !== undefined) return <Badge>{count}</Badge>;

== null 是少数推荐使用 == 的场景

一般规则是 ===，但 == null / != null 同时排 null 和 undefined，这是它的特殊价值。TS 也理解这种 narrowing：

typescript 
function f(x: string | null | undefined) {
  if (x == null) return;
  x.toUpperCase();  // x: string

discriminated union：把"运行时分支"和"类型分支"合一

这是 TS 类型系统设计的核心模式。其想法：给 union 的每个成员加一个共享名字、不同字面量值的字段（"鉴别器" / discriminant），让一次 switch 既做运行时分支判断、又做类型 narrow。

typescript 
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; size: number }
  | { kind: "triangle"; base: number; height: number };

function area(s: Shape): number {
  switch (s.kind) {
    case "circle":
      return Math.PI * s.radius ** 2;       // s: { kind: "circle"; radius: number }
    case "square":
      return s.size ** 2;                   // s: { kind: "square"; size: number }
    case "triangle":
      return (s.base * s.height) / 2;
  }
}

没有 discriminant 的 union 也能工作（在分支里用 "radius" in s narrow），但有 discriminant 的版本表达力更强、性能更好、IDE 跳转更准。

应用范围比想象的广：

  • Redux / useReducer 的 action：{ type: "INCREMENT" } | { type: "SET", payload: number }
  • Result / Either：{ ok: true, value: T } | { ok: false, error: E }
  • 异步数据 4 态：{ status: "idle" } | { status: "loading" } | { status: "success", data: T } | { status: "error", error: E }
  • AST 节点：{ kind: "BinaryOp", left, right, op } | { kind: "Number", value } | ...

never 是兜底哨兵：让"漏一个 case"编译期就报错

never 是一个奇怪的类型：没有任何值属于 never。它在 narrowing 中的用法叫 exhaustiveness checking（穷尽检查）：

typescript 
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; size: number };

function area(s: Shape): number {
  switch (s.kind) {
    case "circle": return Math.PI * s.radius ** 2;
    case "square": return s.size ** 2;
    default:
      // 此处 s 已经被排除了所有已知 case，类型是 never
      const _exhaustive: never = s;
      throw new Error(`Unhandled shape: ${JSON.stringify(s)}`);
  }
}

// 后来加了一个 case
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; size: number }
  | { kind: "triangle"; base: number; height: number };  // 新增

// 立刻在 area() 的 default 分支报错：
//   Type '{ kind: "triangle"; base: number; height: number }' is not assignable to type 'never'
// → 编译期就提醒你忘了处理新 case

这是 TS 静态保障最强的一招。没有 never 兜底，新增 union 成员时，所有处理 union 的代码都需要手动找一遍 —— 必漏。有 never 兜底，编译就帮你抓出所有遗漏点。

type predicate：把运行时检查"承诺"给类型系统

内置 narrow 不够用时，可以写自定义函数告诉 TS"这个值确实是某个类型"：

typescript 
interface Cat { meow(): void }
interface Dog { bark(): void }

function isCat(pet: Cat | Dog): pet is Cat {
  return "meow" in pet;
}

function play(pet: Cat | Dog) {
  if (isCat(pet)) {
    pet.meow();   // pet: Cat
  } else {
    pet.bark();   // pet: Dog
  }
}

关键是返回类型 pet is Cat（type predicate）。它告诉 TS：当此函数返回 true 时，参数 pet 一定是 Cat。

但这是开发者的承诺 —— TS 不验证函数体内逻辑是否真的对应。撒谎就是埋雷：

typescript · 危险 
function isCat(pet: Cat | Dog): pet is Cat {
  return true;   // 永远返回 true 也通过编译
}

function play(pet: Cat | Dog) {
  if (isCat(pet)) {
    pet.meow();   // pet: Cat（TS 信任）→ 但 pet 真有可能是 Dog
  }                //                       → 运行时 TypeError
}

规则：写 type predicate 时，函数体的实际逻辑必须真的能保证返回 true 时确实是该类型。这是契约，不是 hint。

TS 5.5：推断式 type predicate

很常见的痛点：arr.filter(x => x != null) 后类型还是 (T | null)[]，因为 filter 的回调返回 boolean 不是 type predicate。

TS 5.5 起，编译器会从函数体推断 type predicate：

typescript 
const xs: (string | null)[] = ["a", null, "b"];

// TS < 5.5
const ys = xs.filter(x => x !== null);
//    ^? (string | null)[]   仍带 null

// TS 5.5+
const ys = xs.filter(x => x !== null);
//    ^? string[]             自动推 type predicate

// 不需要的话仍可手写
const ys = xs.filter((x): x is string => x !== null);

推断生效的条件：函数体只有一个 return，且 return 表达式是简单的 narrowing 谓词（x !== null / typeof x === "string" 等）。复杂逻辑仍需手写 x is T。

assertion function：抛错版本的 type predicate

有时不想分支 if/else，而是"如果不是预期类型就抛错"。这时用 asserts x is T：

typescript 
function assertIsString(x: unknown): asserts x is string {
  if (typeof x !== "string") {
    throw new Error(`Expected string, got ${typeof x}`);
  }
}

function process(input: unknown) {
  assertIsString(input);
  // 此处 input: string（如果 assert 没 throw，TS 信任收窄）
  return input.toUpperCase();
}

签名 asserts x is T 告诉 TS："如果此函数没有 throw，那么调用之后 x 一定是 T"。同样是开发者承诺，函数体必须真的在不匹配时 throw。

组合用法：assertion function + 通用 helper：

typescript 
function assertExists<T>(value: T | undefined | null, msg: string): asserts value is T {
  if (value == null) throw new Error(msg);
}

const env = process.env.DATABASE_URL;  // string | undefined
assertExists(env, "DATABASE_URL is required");
// env 此处类型是 string

narrowing 的失效边界

narrowing 不是无条件的。三个常见失效点：

  1. 异步回调中的闭包：narrow 后的变量被异步函数捕获时，TS 假设它在等待期间可能被改变，回到原 union 类型
    typescript 
    let value: string | null = "abc";
if (value !== null) {
  setTimeout(() => {
    value.toUpperCase();   // ❌ 'value' is possibly 'null'
    //  TS：你的 narrow 不延伸进异步闭包
  }, 100);
}
  2. 函数调用打破 narrow：调用任何函数（包括方法）后，TS 假设它可能改了变量，回到原类型
    typescript 
    class Container {
  value: string | null = null;
  process() {
    if (this.value !== null) {
      doSomething();
      // 此处 this.value 又变回 string | null（doSomething 可能改了它）
    }
  }
}
  3. 解构后无法 narrow：discriminated union 字段被解构出来后，narrow 链断裂
    typescript 
    type R = { ok: true; value: number } | { ok: false; error: string };

function f(r: R) {
  const { ok, value, error } = r;   // ❌ 解构破坏了 ok 和 value/error 的关联

  if (ok) {
    return value;    // value: number | undefined（不再精确）
  }
}

规避：长 narrow 的代码块用临时 const、不解构 discriminated union、对异步闭包提前用 const 锁定快照值。

03 应用场景

场景 A · Result 模式（替代 throw）

把可能失败的操作返回 Result 而不抛错：

typescript 
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

async function fetchUser(id: string): Promise<Result<User, "not_found" | "network">> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (res.status === 404) return { ok: false, error: "not_found" };
    if (!res.ok) return { ok: false, error: "network" };
    return { ok: true, value: await res.json() };
  } catch {
    return { ok: false, error: "network" };
  }
}

// 调用方
const r = await fetchUser("u_1");
if (!r.ok) {
  if (r.error === "not_found") return <NotFound />;
  return <NetworkError />;
}
return <Profile user={r.value} />;

优点：错误显式出现在类型里，调用方必须处理；error union 也是 literal，可以精确分支。比 try/catch 更可控。

场景 B · Redux / useReducer action

discriminated union 是 reducer 的标配：

typescript 
type State = { count: number };

type Action =
  | { type: "increment" }
  | { type: "decrement" }
  | { type: "set"; payload: number }
  | { type: "reset" };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case "increment": return { count: state.count + 1 };
    case "decrement": return { count: state.count - 1 };
    case "set":       return { count: action.payload };  // payload 自动收窄
    case "reset":     return { count: 0 };
    default:
      const _: never = action;   // 哨兵
      return state;
  }
}

将来加新 action type，TS 会在 reducer 的 default 分支立刻报错，提醒你处理。

场景 C · Array.filter 过滤掉 null/undefined

前端常见需求：从可能含 null 的列表里筛出有效元素。TS 5.5 之前需要显式 type predicate：

typescript 
const ids: (string | null)[] = ["1", null, "3"];

// TS < 5.5：手写 predicate
const valid: string[] = ids.filter((x): x is string => x !== null);

// TS 5.5+：自动推断
const valid: string[] = ids.filter(x => x !== null);

// 也可以用 Array.prototype.flatMap 替代
const valid: string[] = ids.flatMap(x => (x !== null ? [x] : []));

flatMap 配合 x ? [x] : [] 是不少团队的"安全 filter" 写法 —— 不依赖 type predicate 推断，在所有 TS 版本下都正确。

04 典型代码

typeof 全谱

typescript 
// typeof 返回的字符串是这 8 个之一
type Typeof =
  | "string"
  | "number"
  | "bigint"
  | "boolean"
  | "symbol"
  | "undefined"
  | "function"
  | "object";  // 注意：null 也是 "object"！

function describe(x: unknown) {
  switch (typeof x) {
    case "string":   return `str of length ${x.length}`;
    case "number":   return `num ${x.toFixed(2)}`;
    case "function": return `fn with ${x.length} params`;
    case "object":
      // 注意：x 此处可能是 null
      if (x === null) return "null";
      if (Array.isArray(x)) return `array of ${x.length}`;
      return "object";
    default:
      return typeof x;
  }
}

in 操作符 narrow

typescript 
type Bird = { fly(): void; sing(): void };
type Fish = { swim(): void; layEggs(): void };

function move(animal: Bird | Fish) {
  if ("fly" in animal) {
    animal.fly();    // animal: Bird
  } else {
    animal.swim();   // animal: Fish
  }
}

instanceof narrow

typescript 
function getDate(value: Date | string): number {
  if (value instanceof Date) {
    return value.getTime();   // value: Date
  }
  return Date.parse(value);   // value: string
}

// instanceof 也用于自定义类
class HttpError extends Error {
  constructor(public status: number, message: string) { super(message); }
}

function handleError(e: unknown) {
  if (e instanceof HttpError) {
    console.log(e.status);
  } else if (e instanceof Error) {
    console.log(e.message);
  } else {
    console.log("unknown error", e);
  }
}

discriminated union + exhaustive check 完整模板

typescript 
type AsyncState<T, E = Error> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: E };

function unreachable(x: never): never {
  throw new Error(`Unreachable: ${JSON.stringify(x)}`);
}

function render<T>(state: AsyncState<T>) {
  switch (state.status) {
    case "idle":    return null;
    case "loading": return <Spinner />;
    case "success": return <Data data={state.data} />;
    case "error":   return <ErrorMsg error={state.error} />;
    default:        return unreachable(state);  // 类型层兜底
  }
}

type predicate 实用模板

typescript 
// 通用 non-null
function isNonNullable<T>(x: T): x is NonNullable<T> {
  return x != null;
}

// 数组用法
const valid = items.filter(isNonNullable);  // 自动收窄

// 类型谓词配合 Object.entries
function isStringEntry(entry: [string, unknown]): entry is [string, string] {
  return typeof entry[1] === "string";
}

const stringFields = Object.entries(obj).filter(isStringEntry);
// stringFields: [string, string][]

assertion function vs throw 风格对比

typescript 
// ❌ 不带 asserts 标注：调用后类型未变窄
function check(x: unknown) {
  if (typeof x !== "string") throw new Error("not a string");
}
const s: unknown = "abc";
check(s);
s.toUpperCase();   // ❌ Object is of type 'unknown'

// ✅ 带 asserts：调用后类型自动收窄
function check(x: unknown): asserts x is string {
  if (typeof x !== "string") throw new Error("not a string");
}
const s: unknown = "abc";
check(s);
s.toUpperCase();   // ✓ s: string

05 常见陷阱

陷阱 1 · typeof null === "object"

JS 的历史包袱：typeof null 不是 "null" 而是 "object"。TS 不会替你修复：

typescript 
function f(x: object | string) {
  if (typeof x === "object") {
    // x: object — 但运行时可能是 null（如果 x 类型也允许 null 的话）
    x.toString();
  }
}

// 推荐写法
if (typeof x === "object" && x !== null) {
  // 真的非空对象
}

规避：处理 object | null 时务必显式判 null。或开 strictNullChecks 让 TS 把 null 单列出来强制处理。

陷阱 2 · narrowing 不延伸进异步闭包

最大的 narrow 失效场景。TS 假设异步回调执行时，外层变量可能已经被其他代码改了：

typescript 
class Search {
  query: string | null = null;

  async run() {
    if (this.query !== null) {
      const result = await fetch(`/api/q?s=${this.query}`);
      this.query.toUpperCase();   // ❌ 'this.query' is possibly 'null'
      //                          await 之间 query 可能被另一次调用改成 null
    }
  }
}

规避：narrow 后立刻把值复制到本地 const：

typescript 
async run() {
  const q = this.query;
  if (q !== null) {
    const result = await fetch(`/api/q?s=${q}`);
    q.toUpperCase();   // ✓ q 是本地 const，不会被改
  }
}
陷阱 3 · type predicate 撒谎

写错 predicate 函数体，TS 不会发现：

typescript 
interface User { name: string; email: string }

// ❌ 撒谎：忘了检查 email
function isUser(x: unknown): x is User {
  return typeof x === "object" && x !== null && "name" in x;
  //  没有检查 email — 但承诺了 x 是 User
}

const data = { name: "Alice" };
if (isUser(data)) {
  data.email.toLowerCase();   // 编译过 → 运行时炸（email 是 undefined）
}

规避：写 type guard 时逐字段验证；或干脆用 zod 等 schema 库自动生成 type guard。复杂对象别手写 predicate。

陷阱 4 · strictNullChecks 关闭让所有 narrow 几乎失效

strictNullChecks: false 时，所有类型隐式包含 null | undefined，TS 不会强制你处理它们：

typescript 
// strictNullChecks: false 下
function f(s: string) {
  s.toUpperCase();  // 编译过 — 但 s 可能是 null
}
f(null);  // 也通过 — 运行时炸

// strictNullChecks: true 下
function f(s: string) {
  s.toUpperCase();  // ✓
}
f(null);  // ❌ Argument of type 'null' is not assignable to 'string'

规避：永远开启 strictNullChecks（strict: true 包含它）。这是 TS 真正区别于"带类型 JS"的 90% 价值所在。

陷阱 5 · in 检查不等于"属性存在"

"foo" in obj 检查的是原型链上是否有 foo 属性，包括从 Object.prototype 继承的：

typescript 
const obj: Record<string, unknown> = {};
"toString" in obj;  // true — 来自 Object.prototype
"hasOwnProperty" in obj;  // true — 同上

规避：in 只用于 discriminated union narrow（场景明确）；想检查 own property 用 Object.hasOwn(obj, "foo")（ES2022）。

陷阱 6 · 解构 discriminated union 后 narrow 失效

TS 不能在解构后维持鉴别字段和其他字段的关联：

typescript 
type Result<T> = { ok: true; value: T } | { ok: false; error: string };

function bad<T>(r: Result<T>) {
  const { ok, value, error } = r;
  if (ok) {
    return value;   // ❌ value: T | undefined（不再精确）
  }
}

function good<T>(r: Result<T>) {
  if (r.ok) {
    return r.value;  // ✓ value: T
  }
  return r.error;
}

规避：处理 discriminated union 时不要解构，保持通过 r.field 访问。
