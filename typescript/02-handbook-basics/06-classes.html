06 · Classes

TS 在 ES class 之上叠加了一层类型系统能力：可见性修饰符、abstract、this 类型、parameter properties、字段初始化检查。本章拆解每一项 TS 增强、它们与 ES 标准（特别是 #field 硬隐私）的互动、以及 Liskov 替换原则在结构化类型下的实际行为。重点回答：private 到底"私有"到什么程度、为什么 implements 不会自动给方法加类型、为什么基类构造器调用 override 方法是危险的、abstract class 的 constructor type 为什么不能 new。

阅读时长 ~32 分钟 前置 04 Functions + JS · class 与继承 对应 typescriptlang.org Handbook · Classes

01 速览

02 底层逻辑

1. private 是软隐私，#field 是硬隐私

TS 的 private 修饰符只在编译期阻止访问，运行时和普通字段无区别：

typescript 
class Counter {
  private count = 0;
  inc() { this.count++; }
}

const c = new Counter();
// c.count;          // ❌ TS 编译错：'count' is private
(c as any).count;    // ✓ 运行时拿到 0 — TS 类型是擦的
console.log(JSON.stringify(c));   // {"count":0}  仍序列化出来！

ES2022 标准的 #field 是真正的运行时隐私，由 V8 等引擎在私有字段表里实现：

typescript 
class Counter {
  #count = 0;
  inc() { this.#count++; }
}

const c = new Counter();
// c.#count;             // ❌ TS 编译错
// (c as any).#count;    // ❌ 语法错（# 必须直接拿 class 内可见）
// c["#count"];          // ❌ undefined（# 不是字符串 key）

console.log(JSON.stringify(c));   // {}  真的看不到
private#field
编译期阻止外部访问✓✓
运行时阻止外部访问✗ (o as any).x 可绕✓
JSON.stringify 输出✓ 输出✗ 不输出
子类访问父类同名✗✗ 每个 class 独立的 # 命名空间
反射 / Object.keys看得到看不到

实用建议：

  • 新代码用 #field：真正的隐私，受 ES 标准保护，工具链一致
  • 遗留代码用 private：性能略好（无 WeakMap 实现）、调试器看得见、和 legacy decorator 兼容
  • 不要混用：一个 class 内部要么全 #，要么全 private

2. implements 不会自动给方法加类型

implements 只检查已存在的结构是否符合接口，不会反向推断方法签名：

typescript 
interface Greet {
  greet(name: string): string;
}

class A implements Greet {
  greet(name) {     // ❌ Parameter 'name' implicitly has an 'any' type
    return `Hi ${name}`;
  }
}

// ✅ 必须显式注解
class B implements Greet {
  greet(name: string): string {
    return `Hi ${name}`;
  }
}

为什么这样设计？因为 implements 仅仅是"声明这个 class 满足这个 shape"的标签。它不是 ES 标准的一部分，只是给类型系统的 hint。如果它会反向推断，就破坏了"声明独立、检查独立"的原则。

规避：用 abstract class（如果你能控制基类）、或者写 helper 类型从 interface 提取签名。多数情况老老实实写注解。

3. class 同时占据类型空间和值空间

01 章已经讲过 类型与值是两个独立命名空间。class 是同时占两边的特殊声明：

typescript 
class User {
  constructor(public name: string) {}
  greet() { return `Hi ${this.name}` }
}

// 类型空间：User 表示"实例类型"
function f(u: User) { return u.greet() }

// 值空间：User 表示"构造器函数"
const ctor: typeof User = User;
const alice = new ctor("Alice");

// 反向取类型
type UserConstructor = typeof User;             // new (name: string) => User
type UserInstance = InstanceType<typeof User>;  // User

typeof ClassName 和 ClassName 是不同的类型：前者是构造器（newable），后者是实例类型。这是 18 章 declaration files 写带 class 的库 .d.ts 时的关键。

4. Liskov 在结构化类型下的实际表现

Liskov substitution principle（LSP）：子类可以替换父类。在 TS 的结构化类型下，替换是默认安全的，因为只要形状兼容就互通：

typescript 
class Animal {
  name: string = "";
  move(distance: number) { console.log(distance); }
}

class Dog extends Animal {
  bark() { console.log("woof") }
  // override 时：可以加新字段，但已有签名要兼容
  move(distance: number) { console.log(`Dog ran ${distance}`) }
}

const a: Animal = new Dog();   // ✓ Dog 是 Animal 的子类型（结构兼容 + extends）
a.move(10);                     // 实际调 Dog.move（运行时多态）

注意 method override 默认双变（bivariant），参数可以"更窄"的不安全用法默认通过：

typescript 
class Base {
  process(x: Animal) {}
}

class Bad extends Base {
  // 方法 override 默认 bivariant：参数 Dog 比 Animal 窄不报错
  process(x: Dog) { x.bark() }
}

const b: Base = new Bad();
b.process(new Animal());   // 编译过 → 运行时 bark is not a function

开 strictFunctionTypes 也救不了 method override（仅函数类型生效，不对方法签名生效）。要严格，TS 5.0 起可用 override 关键字 + noImplicitOverride，至少防止"我以为我 override 了，其实拼错了"。

5. override 关键字与 noImplicitOverride

typescript 
class Base {
  greet() { return "Hello" }
}

class Sub extends Base {
  override greet() { return "Hi" }   // ✓ 明确表示"覆盖父类"

  override grret() { return "x" }    // ❌ 'grret' has no equivalent in 'Base'
  //                                   拼错时立刻报错
}

配合 tsconfig:

json 
{ "compilerOptions": { "noImplicitOverride": true } }

打开后所有 override 都必须加 override 关键字，否则编译错。这是中大型 OOP 项目的标配。

6. abstract class 的 constructor type 不能 new

abstract class 不能直接实例化：

typescript 
abstract class Shape {
  abstract area(): number;
  describe() { return `Area: ${this.area()}` }
}

const s = new Shape();   // ❌ Cannot create an instance of an abstract class

class Circle extends Shape {
  constructor(public radius: number) { super() }
  area() { return Math.PI * this.radius ** 2 }
}
const c = new Circle(5);   // ✓

但从 abstract class 取 constructor type 时需要用特殊工具：

typescript 
// ❌ 想写 factory 接受 Shape 类型的 constructor
function factory(C: new () => Shape) { return new C() }
factory(Shape);   // ❌ Cannot assign an abstract constructor type to a non-abstract one

// ✅ 用 abstract construct signature
function factory(C: abstract new () => Shape) { return new C() }
factory(Shape);   // 类型上能传，但仍不能 new abstract — TS 5.0 把这个矛盾点拉清晰

实际生产代码里 abstract new 几乎只在写"接受 class 但不实例化"的工具时用到（如打印继承树、生成 schema）。

7. Parameter properties：构造器参数同时声明字段

一个 TS 独有的简写：在构造器参数前加可见性修饰符或 readonly，自动声明同名字段并赋值：

typescript 
// 等价的两种写法
class A {
  constructor(public name: string, private age: number, readonly id: string) {}
}
// 编译器自动展开为：
class B {
  public name: string;
  private age: number;
  readonly id: string;
  constructor(name: string, age: number, id: string) {
    this.name = name;
    this.age = age;
    this.id = id;
  }
}

NestJS / Angular 的 DI 系统大量使用 parameter properties + 装饰器，节省样板。但有副作用：让构造器变"半声明半实现"，新人读起来略费解；不能在赋值前做计算或验证。

8. strictPropertyInitialization 与 definite assignment

严格模式下，所有非 optional 实例字段必须在构造器结束前赋值：

typescript 
class User {
  name: string;       // ❌ Property 'name' has no initializer and is not definitely assigned in the constructor
  constructor() {
    // 没赋值
  }
}

// 解决方式
class User {
  name: string = "";          // ✓ 字段初始化
  // 或
  constructor(public name: string) {}   // ✓ parameter property 自动赋值
  // 或
  name?: string;              // ✓ 显式 optional
  // 或
  name!: string;              // ✓ definite assignment assertion（"我担保会赋值"）
  init() { this.name = "..." }
}

!: 是开发者承诺，类似非空断言：告诉 TS"虽然你看不到这里赋值，但运行时一定会赋"。常见场景是 ORM Entity（外部 hydrate 字段）：

typescript 
// TypeORM 模式
@Entity()
class User {
  @PrimaryGeneratedColumn()
  id!: number;     // ! 因为 ORM 在 hydrate 时填值

  @Column()
  email!: string;
}

9. Getter / setter 类型可不同（TS 4.3+）

typescript 
class Thermostat {
  private _temp = 20;

  get temp(): number {
    return this._temp;
  }

  // setter 接受多种输入，存储时统一
  set temp(value: number | string) {
    this._temp = typeof value === "string" ? parseFloat(value) : value;
  }
}

const t = new Thermostat();
t.temp = "25.5";    // ✓ string 输入
t.temp = 30;         // ✓ number 输入
const c = t.temp;    // c: number（getter 返回 number）

TS 4.3 之前 setter 必须接受 getter 同类型。新版本允许 setter 接受更宽的输入（让外部赋值更灵活），getter 返回更窄（让读取更精确）。这是「输入位置逆变 / 输出位置协变」原则的应用。

03 应用场景

场景 A · NestJS Controller / Service / DI

class + 装饰器 + parameter properties + DI 是 NestJS 的核心套路：

typescript 
@Injectable()
class UserService {
  constructor(
    @InjectRepository(User) private userRepo: Repository<User>,
    private logger: LoggerService,
  ) {}

  async findById(id: string): Promise<User | null> {
    this.logger.debug(`Finding user ${id}`);
    return this.userRepo.findOneBy({ id });
  }
}

@Controller("users")
class UserController {
  constructor(private userService: UserService) {}

  @Get(":id")
  async getUser(@Param("id") id: string): Promise<User> {
    const u = await this.userService.findById(id);
    if (!u) throw new NotFoundException();
    return u;
  }
}

这种代码仍依赖 legacy decorator + emitDecoratorMetadata。Stage 3 标准装饰器还没补齐 metadata reflection 能力（详见 14 章）。

场景 B · 自定义 Error 层次

class 最简单也最有价值的应用：建立有类型的错误体系：

typescript 
class AppError extends Error {
  constructor(message: string, public readonly cause?: unknown) {
    super(message);
    this.name = this.constructor.name;
  }
}

class NotFoundError extends AppError {
  constructor(public readonly resource: string, public readonly id: string) {
    super(`${resource} not found: ${id}`);
  }
}

class ValidationError extends AppError {
  constructor(message: string, public readonly field: string) { super(message) }
}

// 业务代码
async function getUser(id: string): Promise<User> {
  const u = await db.users.findById(id);
  if (!u) throw new NotFoundError("User", id);
  return u;
}

// 调用方
try {
  const u = await getUser("xxx");
} catch (e) {
  if (e instanceof NotFoundError) console.log(`${e.resource}/${e.id}`);
  else if (e instanceof ValidationError) console.log(e.field);
  else if (e instanceof AppError) console.log(e.cause);
  else throw e;   // 未知错误重抛
}

场景 C · 状态机基类 + 子类

abstract class 适合表达"骨架已定，细节由子类填"的设计：

typescript 
abstract class State<Ctx> {
  constructor(protected ctx: Ctx) {}
  abstract enter(): void;
  abstract exit(): void;
  abstract handle(event: string): State<Ctx> | null;
}

class Idle extends State<App> {
  enter() { console.log("idle") }
  exit() {}
  handle(e: string) {
    if (e === "start") return new Loading(this.ctx);
    return null;
  }
}

class Loading extends State<App> {
  enter() { console.log("loading") }
  exit() {}
  handle(e: string) {
    if (e === "done") return new Idle(this.ctx);
    return null;
  }
}

场景 D · TypeORM Entity 与 definite assignment

typescript 
@Entity()
class Article {
  @PrimaryGeneratedColumn("uuid")
  id!: string;             // ! ORM hydrate

  @Column({ length: 200 })
  title!: string;

  @Column("text")
  content!: string;

  @ManyToOne(() => User)
  author!: User;

  @CreateDateColumn()
  createdAt!: Date;

  @UpdateDateColumn()
  updatedAt!: Date;
}

04 典型代码

所有可见性 + 类型增强一览

typescript 
class Item {
  // 字段
  public name: string;          // public 是默认，可省
  protected sku: string;        // 子类可见
  private cost: number;          // 仅当前类（软隐私）
  #internalId: string;            // 真正硬隐私（# field）
  readonly createdAt: Date;     // 不可重新赋值
  static count = 0;              // 静态字段

  constructor(name: string, sku: string, cost: number) {
    this.name = name;
    this.sku = sku;
    this.cost = cost;
    this.#internalId = crypto.randomUUID();
    this.createdAt = new Date();
    Item.count++;
  }

  // method
  public describe(): string {
    return `${this.name} (${this.sku})`;
  }

  // accessor
  get displayCost(): string {
    return `$${this.cost.toFixed(2)}`;
  }

  // static method
  static get total(): number {
    return Item.count;
  }
}

泛型 class

typescript 
class Stack<T> {
  private items: T[] = [];

  push(item: T): void {
    this.items.push(item);
  }

  pop(): T | undefined {
    return this.items.pop();
  }

  peek(): T | undefined {
    return this.items.at(-1);
  }

  get size(): number {
    return this.items.length;
  }
}

const s = new Stack<number>();
s.push(1);
s.push(2);
const top = s.pop();    // top: number | undefined

// ⚠️ 类型参数无法用在 static 成员上
class Foo<T> {
  // static x: T;     // ❌ Static members cannot reference class type parameters

}

this 类型与 builder pattern

typescript 
// this 类型让链式调用在子类也能精确返回子类型
class QueryBuilder {
  protected conditions: string[] = [];

  where(condition: string): this {
    this.conditions.push(condition);
    return this;
  }

  build(): string {
    return `WHERE ${this.conditions.join(" AND ")}`;
  }
}

class UserQueryBuilder extends QueryBuilder {
  byEmail(email: string): this {
    return this.where(`email = '${email}'`);
  }
}

// 链式调用类型仍精确
const q = new UserQueryBuilder()
  .where("active = true")
  .byEmail("a@b.c")        // 不会因为前面的 .where 退化成 QueryBuilder
  .build();

this is X type guard（自定义类型守卫 method）

typescript 
class FileSystemEntity {
  constructor(public path: string) {}

  isFile(): this is File {
    return this instanceof File;
  }

  isDirectory(): this is Directory {
    return this instanceof Directory;
  }
}

class File extends FileSystemEntity {
  constructor(path: string, public size: number) { super(path) }
}

class Directory extends FileSystemEntity {
  constructor(path: string, public children: FileSystemEntity[]) { super(path) }
}

function process(entry: FileSystemEntity) {
  if (entry.isFile()) {
    console.log(entry.size);     // ✓ entry 收窄为 File
  } else if (entry.isDirectory()) {
    console.log(entry.children); // ✓ entry 收窄为 Directory
  }
}

abstract class 完整示例

typescript 
abstract class Notifier {
  // abstract 方法 — 子类必须实现
  abstract send(message: string): Promise<void>;

  // 共用逻辑 — 子类继承
  async notify(user: { email: string }, message: string): Promise<void> {
    const formatted = `[${new Date().toISOString()}] ${message}`;
    await this.send(formatted);
  }
}

class EmailNotifier extends Notifier {
  async send(message: string): Promise<void> {
    await fetch("/api/email", { method: "POST", body: message });
  }
}

class SlackNotifier extends Notifier {
  async send(message: string): Promise<void> {
    await fetch("/api/slack", { method: "POST", body: message });
  }
}

const n: Notifier = Math.random() > 0.5 ? new EmailNotifier() : new SlackNotifier();
await n.notify({ email: "a@b.c" }, "hello");

05 常见陷阱

陷阱 1 · 基类构造器调用被子类 override 的方法

JS 的字段初始化在子类构造器调 super() 之后才发生。如果基类构造器调用了被子类 override 的方法，子类还没初始化的字段会是 undefined：

typescript 
class Base {
  constructor() {
    this.greet();   // 多态调用 — 调到子类的 greet
  }
  greet() { console.log("base") }
}

class Sub extends Base {
  prefix = "hi";   // 子类字段
  greet() {
    console.log(`${this.prefix} sub`);
    //          ^^^^^^^^^^^^^^^^^^ 此时 this.prefix 是 undefined！
  }
}

new Sub();   // 输出 "undefined sub"，不是 "hi sub"

根因：执行顺序是 super() → 子类字段初始化 → 子类构造器体。基类构造器跑的时候，子类字段还没被赋值。

规避：不在基类构造器调用 abstract / 可被 override 的方法。需要"模板方法"模式时，用 init() 之类显式 method 让调用方决定时机。

陷阱 2 · private 字段在 JSON.stringify 仍输出

private 是软隐私，JSON.stringify 不知情：

typescript 
class User {
  constructor(public name: string, private passwordHash: string) {}
}
const u = new User("Alice", "hash_xxx");
JSON.stringify(u);   // {"name":"Alice","passwordHash":"hash_xxx"}  泄露！

规避：

  • 用 #field：JSON.stringify 不会输出
  • 实现 toJSON() method 显式控制序列化
  • API 返回前用 zod / class-transformer 等做 DTO 转换
陷阱 3 · this 丢失绑定（method 拿出来调用）

04 章已经讲过，class method 上同样存在：

typescript 
class Counter {
  count = 0;
  inc() { this.count++ }
}

const c = new Counter();
const fn = c.inc;
fn();   // 运行时炸：this 是 undefined

// ✅ 解 1：箭头函数 field（自动绑定 this）
class Counter {
  count = 0;
  inc = () => { this.count++ };   // 编译产物：每个实例自带一个 inc 函数
}

// ✅ 解 2：bind
const fn = c.inc.bind(c);

// ✅ 解 3：声明 this 参数让 TS 拦住误用
class Counter {
  count = 0;
  inc(this: Counter) { this.count++ }
}

React 类组件大量用方案 1（箭头 field）。性能代价：每个实例多一个函数对象，无法走 prototype。绝大多数场景代价可以忽略。

陷阱 4 · static 成员引用 class type parameter

泛型参数不能用在 static 成员上：

typescript 
class Box<T> {
  static empty: T;   // ❌ Static members cannot reference class type parameters
}

原因：T 是实例泛型参数，每个实例可以是不同 T。但 static 成员属于 class 自身，只有一份 —— 没法对应到具体某个 T。

规避：让 static method 自带泛型，或者把泛型迁到 method 层：

typescript 
class Box<T> {
  static create<U>(value: U): Box<U> { return new Box<U>(value) }
  constructor(public value: T) {}
}
陷阱 5 · implements 不会推断方法参数类型
typescript 
interface Repository<T> {
  findById(id: string): Promise<T | null>;
}

class UserRepo implements Repository<User> {
  // ❌ id 隐式 any（implements 不传播类型）
  async findById(id) { return null }
}

// ✅ 必须显式注解
class UserRepo implements Repository<User> {
  async findById(id: string): Promise<User | null> { return null }
}

规避：abstract class 比 interface 更"DRY"——参数类型可以从 abstract method 自动继承到 override：

typescript 
abstract class Repository<T> {
  abstract findById(id: string): Promise<T | null>;
}

class UserRepo extends Repository<User> {
  async findById(id) { return null }   // ✓ id 自动推 string
}
陷阱 6 · readonly + class 字段并不阻止内部 mutation
typescript 
class Container {
  readonly items: number[] = [];

  add(x: number) {
    this.items.push(x);   // ✓ 通过 — readonly 只阻止重新赋值，不阻止改内部
  }

  // this.items = [];      // ❌ Cannot assign — 这才是 readonly 拦的
}

规避：要阻止数组也变，类型用 readonly number[]。深 readonly 自定义类型（10 章 mapped type）。
