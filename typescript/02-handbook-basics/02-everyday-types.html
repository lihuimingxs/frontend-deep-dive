02 · Everyday Types

本章覆盖你每天都会用的类型工具：原始类型、对象、数组、union、type alias、interface、literal type、断言。这些不是"基础"在"简单"的意思，而是基础在"用得最多" —— 写得对不对决定了项目的类型质量。重点回答四个问题：type 和 interface 怎么选 / 什么时候用 unknown 替代 any / literal type 的实际价值 / as 在哪里会让你翻车。

阅读时长 ~30 分钟 前置 01 The Basics 对应 typescriptlang.org Handbook · Everyday Types

01 速览

TS 的 11 种原始/特殊类型一览：

类型用途典型例子
string字符串原始"hello"
number数字原始（包括 NaN/Infinity）42 / 3.14
boolean布尔true / false
bigint任意精度整数9007199254740993n
symbol唯一符号Symbol("foo")
undefined未赋值let x 默认
null显式空值API 返回 null
void函数无意义返回事件 handler
never永不返回（抛错/死循环/exhaustive 兜底）throw
object非原始（极少用）{} / 数组 / 函数
unknownany 的安全替代JSON.parse 返回
any关闭类型检查⚠️ 慎用

组合工具：T[] / Array<T>（数组）、A | B（union）、{ x: number }（对象字面量）、"left" | "right"（literal）、x as T（断言）、x!（非空断言）。然后用 type 或 interface 给类型起名字。

02 底层逻辑

下面 5 条是你写 TS 不出错的地基。每一条都对应一个"为什么这样设计"。

1. type 和 interface 不是风格选择，是能力选择

很多人争论"用 type 还是 interface"，把它当代码风格问题。实际上两者能力不同：

能力interfacetype
描述对象 / 函数 / 类形状✓✓
extends 继承✓ 多继承✓ 用 & intersection
declaration merging（同名合并）✓ 这是关键差异✗ 报错"重复声明"
描述 union A | B✗✓
描述 tuple [a, b]✗✓
描述 mapped / conditional / template literal✗✓
给原始类型起别名 type Id = string✗✓
报错时保留名字✓ "Type 'X' is missing..."常被展开为字面量
性能（大型联合编译速度）更快（cache 友好）稍慢

实用决策树：

  • 描述对象形状 / 库的公共 API → 默认用 interface。报错信息友好，能 declaration merge（用户可扩展库类型）
  • 组合 union / intersection / mapped 等高级类型 → 用 type
  • 给原始类型起别名 → 必须 type

declaration merging 是 interface 的杀手锏。同名 interface 会自动合并字段，这是给"全局类型扩展"和"第三方库 patch"留的逃生口（详见 15 章）：

typescript 
// 你的项目自己加字段
interface Window {
  __APP_VERSION__: string;
}

// 或扩展第三方库（Express）
declare module "express-serve-static-core" {
  interface Request {
    user?: { id: string };  // middleware 注入
  }
}

// 这两件事 type 都做不到

2. unknown 是 any 的安全替代品

表面上 any 和 unknown 都是"我不知道是什么类型"，但使用规则完全相反：

anyunknown
能赋给它任意值✓✓
能赋给其他类型✓ 任意✗ 必须先 narrow
能直接 .property 访问✓✗
能直接调用✓✗
typescript 
function fromAny(x: any) {
  x.foo.bar.baz;        // 通过 — 任何路径都不报
  const n: number = x;  // 通过
}

function fromUnknown(x: unknown) {
  // x.foo;              // ❌ 'x' is of type 'unknown'
  // const n: number = x; // ❌

  if (typeof x === "string") {
    x.toUpperCase();    // ✓ narrow 后可用
  }
  if (x instanceof Error) {
    console.error(x.message); // ✓
  }
}

使用 unknown 的场景（凡是来自"信任边界"的数据）：

  • JSON.parse 的返回 — TS 标准库返回 any，建议手动 cast 成 unknown
  • catch (e) — TS 4.4+ 默认推 unknown（启用 useUnknownInCatchVariables）
  • 第三方 SDK 没类型时
  • 跨 worker / iframe 的消息
  • Function.prototype.call.apply 等动态调用结果

规则：项目里见到 any 都问一句"能不能换 unknown"，绝大多数情况答案是能。any 真正合理的场景只有"过渡期 + 你担保稍后会修"。

3. literal type 是"窄到只能取一个值"的类型

"left" 既是值也是类型。当它出现在类型位置时，是「只能等于字符串 "left"」的类型，集合里只有一个成员。

typescript 
type Direction = "left" | "right" | "center";

function setAlign(d: Direction) { /* ... */ }
setAlign("left");    // ✓
setAlign("middle");  // ❌ Argument of type '"middle"' is not assignable

这等价于「字符串枚举」，但比 enum 更轻 —— 没有运行时对象、tree-shaking 干净、与 JSON 序列化天然兼容。新代码应优先用 literal union 而非 enum。

关键细节：let 推断成宽类型，const 推断成 literal：

typescript 
let a = "left";    // a: string（因为 let 可重新赋值）
const b = "left";  // b: "left"（const 不可变，能精确推断）

function setAlign(d: "left" | "right") { /* ... */ }
setAlign(a);       // ❌ string 不能赋给 "left" | "right"
setAlign(b);       // ✓

这就是为什么 const 的另一个好处：让推断更精确。

对象字面量需要显式 as const 才能保留 literal：

typescript 
const cfg1 = { dir: "left" };
//    ^? { dir: string }      因为对象的字段默认可变

const cfg2 = { dir: "left" } as const;
//    ^? { readonly dir: "left" }    所有字段 readonly + literal

as const 触发 literal inference：让对象的所有 string / number 字面量保留为字面量类型，所有字段标记为 readonly。这是写"类型安全的配置对象"的标准模式。

4. excess property check 是给"立即字面量"的特别保护

一个细节经常让人困惑：直接传字面量会报"多余字段"，先赋给变量再传却通过：

typescript 
interface Options {
  width: number;
  height: number;
}

function render(o: Options) { /* ... */ }

// ❌ 直接传 — 触发 excess property check
render({ width: 100, height: 200, padding: 10 });
//                                ^^^^^^^ Object literal may only specify known properties

// ✅ 先赋值再传 — TS 假设你故意宽松
const opts = { width: 100, height: 200, padding: 10 };
render(opts);  // 通过

为什么这样设计：直接传字面量最像"我以为这个字段叫 padding 但其实是 spacing"的拼写错误。先入变量 = 你故意构造了一个超集，可能想给多处使用 —— TS 信任你。

要有意忽略 excess property check 的三种方式：先入变量、用 as Options 断言、给 interface 加 index signature [k: string]: unknown。但多数情况建议直接修字面量，因为这正是 TS 在帮你抓 bug。

5. type assertion 不是运行时转换

x as T 是编译期断言，告诉 TS"我担保 x 在运行时确实是 T"。它不做任何运行时检查，也不改变 x 的实际值。

typescript 
const s: unknown = "abc";
const n = s as number;   // 编译过 — TS 信任你
n.toFixed(2);            // 运行时 TypeError: n.toFixed is not a function

三种断言形式：

  • x as T — 推荐，.tsx 文件唯一可用形式
  • <T>x — 旧语法，与 JSX 冲突，.tsx 不可用，不推荐
  • x as unknown as T — 双重断言，"我知道直接 as T 不通过，请闭嘴"。这是危险信号，绝大多数情况说明类型设计错了

TS 会拒绝"明显矛盾"的断言："abc" as number 直接报错。但只要找到任何"中间类型可达"的桥（如 unknown），TS 就放行。所以 as unknown as T 是绕过类型系统的万能后门 —— 永远是技术债。

更好的替代是 12 章会讲的 satisfies：校验形状但保留具体推断，不放宽类型。

03 应用场景

场景 A · React 组件 Props 设计

React props 是 interface vs type 选择最高频的位置。推荐用 interface（用户可以 declaration merge 扩展），用 literal union 描述 variant：

typescript 
interface ButtonProps {
  variant: "primary" | "secondary" | "ghost";
  size: "sm" | "md" | "lg";
  disabled?: boolean;
  onClick?: (e: React.MouseEvent) => void;
  children: React.ReactNode;
}

function Button({ variant, size, disabled, onClick, children }: ButtonProps) {
  return (
    <button
      type="button"
      className={`btn btn-${variant} btn-${size}`}
      disabled={disabled}
      onClick={onClick}
    >
      {children}
    </button>
  );
}

// 用法
<Button variant="primary" size="md" onClick={() => console.log("clicked")}>
  Click me
</Button>

literal union 让 variant 在 IDE 里有自动补全 + 拼写错误立刻报错。比 enum 简洁，比纯 string 安全。

场景 B · API 响应类型 + zod 校验

类型设计两层：interface 描述编译期形状，zod 在运行时校验：

typescript 
import { z } from "zod";

// 单一真源：从 schema 反推 type
const UserSchema = z.object({
  id: z.string(),
  email: z.string().email(),
  role: z.enum(["admin", "user", "guest"]), // → literal union
  createdAt: z.coerce.date(),
});

type User = z.infer<typeof UserSchema>;
//   ^? { id: string; email: string; role: "admin" | "user" | "guest"; createdAt: Date }

async function fetchUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  const json: unknown = await res.json();    // 标记 unknown，强制校验
  return UserSchema.parse(json);              // 失败抛 ZodError
}

z.infer 把 schema 反向变成 TS type —— 一处定义，编译 + 运行时复用。这是 tRPC / Hono / Drizzle 端到端类型安全的基础套路。

场景 C · 配置对象 as const 锁定字面量

Tailwind / 路由表 / 状态机定义都需要"配置写死，类型精确"的能力：

typescript 
const ROUTES = {
  home: "/",
  user: "/user/:id",
  settings: "/settings",
} as const;

// ROUTES.home 类型是 "/", 不是 string
type RoutePath = (typeof ROUTES)[keyof typeof ROUTES];
//   ^? "/" | "/user/:id" | "/settings"

function navigate(path: RoutePath) { /* ... */ }
navigate(ROUTES.home);          // ✓
navigate("/typo");              // ❌

as const + typeof + keyof 三件套是把"对象字面量的精确形状"提到类型层的标准做法（08 章会展开 typeof / keyof）。

04 典型代码

原始类型与 union 组合

typescript 
// 多类型的输入参数
function format(value: string | number): string {
  if (typeof value === "string") return value.toUpperCase();
  return value.toFixed(2);
}

// 可选属性
interface User {
  id: string;
  name: string;
  avatar?: string;       // 可能不存在
}

// readonly 属性
interface Point {
  readonly x: number;
  readonly y: number;
}
const p: Point = { x: 1, y: 2 };
// p.x = 3;  // ❌ Cannot assign to 'x' because it is a read-only property

type alias vs interface 实战

typescript 
// ✅ 对象形状 — interface 优先
interface ApiResponse<T> {
  ok: true;
  data: T;
}

// ✅ union — 必须 type
type Result<T> = ApiResponse<T> | { ok: false; error: string };

// ✅ 原始类型别名 — 必须 type
type UserId = string;
type Timestamp = number;

// ✅ tuple — 必须 type
type Range = [number, number];

// ✅ 函数类型 — 两者都行，type 更紧凑
type Handler = (event: string) => void;
interface IHandler { (event: string): void } // 也合法但少见

// ❌ interface 不能描述
// interface BadResult = ApiResponse<T> | { ok: false };  // 语法错误

any vs unknown 的实战转换

typescript 
// ❌ 偷懒
function getValue(key: string): any {
  return localStorage.getItem(key);
}
const x = getValue("count");
const n: number = x;     // 编译过 — 实际是 string 或 null

// ✅ 安全
function getValue(key: string): unknown {
  return localStorage.getItem(key);
}
const x = getValue("count");
// const n: number = x;   // ❌ TS 拦住

if (typeof x === "string") {
  const n = Number(x);   // 显式转换
}

// ✅ 更好：直接用 zod 校验
const CountSchema = z.coerce.number().int().min(0);
const count = CountSchema.parse(localStorage.getItem("count"));

非空断言 ! 与替代写法

typescript 
function findUser(id: string): User | undefined { /* ... */ }

// ❌ ! 关闭检查 — 找不到时运行时炸
const u = findUser("xxx")!;
u.name;

// ✅ 显式处理
const u = findUser("xxx");
if (!u) throw new Error("user not found");
u.name;

// ✅ 可选链 + 默认值
const name = findUser("xxx")?.name ?? "Anonymous";

// ✅ assertion function（03 章会展开）
function assertExists<T>(x: T | undefined, msg: string): asserts x is T {
  if (x == null) throw new Error(msg);
}
const u = findUser("xxx");
assertExists(u, "user not found");
u.name;  // 类型已收窄为 User

readonly array / tuple

typescript 
// 三种数组写法
const a: number[] = [1, 2, 3];                  // 可变数组
const b: ReadonlyArray<number> = [1, 2, 3];     // 只读
const c: readonly number[] = [1, 2, 3];          // 只读（推荐写法）
// b.push(4);  // ❌ 'push' does not exist on ReadonlyArray

// tuple
const range: [number, number] = [0, 100];
const named: [name: string, age: number] = ["Alice", 30]; // labeled tuple

// 推断为 readonly tuple
const point = [1, 2] as const;
//    ^? readonly [1, 2]

05 常见陷阱

陷阱 1 · 写大写 String / Number 当原始类型

String / Number / Boolean 是 JS 的包装对象类型，不是原始类型。它们几乎永远不该出现在你的代码里：

typescript 
// ❌ 错
function trim(s: String): String {
  return s.trim();
}
const x = trim("hello");
const y: string = x;  // ❌ Type 'String' is not assignable to type 'string'

// ✅ 对
function trim(s: string): string {
  return s.trim();
}

规避：永远写小写 string / number / boolean。typescript-eslint 有 @typescript-eslint/ban-types 规则可强制。

陷阱 2 · let 推断的"宽化"陷阱

把 const 改成 let，类型从字面量自动放宽：

typescript 
function setMode(m: "dark" | "light") { /* ... */ }

// 原本 OK
const mode = "dark";
setMode(mode);  // ✓

// 改成 let 突然报错
let mode = "dark";    // mode: string
setMode(mode);  // ❌ Argument of type 'string' is not assignable to '"dark" | "light"'

根因：let 可重新赋值，TS 必须考虑"以后可能赋成任何 string"，所以推宽到 string。const 不可变，可以保留 literal。

规避：能 const 就 const；必须 let 时显式注解 let mode: "dark" | "light" = "dark"。

陷阱 3 · excess property check 被"绕过"导致拼写错误漏过

把字面量先入变量再传，excess property check 被绕过 —— 拼写错的字段不报：

typescript 
interface Options { width: number; height: number }
function render(o: Options) { /* ... */ }

// ❌ 类型上"通过"，但 padding 是 noise，可能是想写 width 拼错了
const opts = { width: 100, height: 200, paadding: 10 };
render(opts);   // 编译过

规避：保持字面量直接传；如果必须先赋值，给变量显式注解类型 const opts: Options = {...}。

陷阱 4 · as 双重断言绕过类型系统

遇到"as T 不通过"时容易翻成 as unknown as T，把检查彻底关掉：

typescript 
// 一个真实场景：HTML 元素类型断言
const input = document.querySelector(".email") as HTMLInputElement;
input.value;  // ✓

// ❌ 但有人写
const data = response as unknown as ApiData;

// ✅ 替代：用 zod 真做校验
const data = ApiDataSchema.parse(response);

规避：每写一个 as unknown as T 都问"为什么类型系统拦我？是不是类型设计错了？是不是该加 type guard 或 zod？"绝大多数情况都有更安全的做法。

陷阱 5 · readonly 是浅 readonly

readonly 只锁第一层，嵌套对象仍可改：

typescript 
interface User {
  readonly id: string;
  readonly profile: { name: string };
}
const u: User = { id: "1", profile: { name: "Alice" } };
// u.id = "2";          // ❌ 拦住
u.profile.name = "Bob"; // ✓ 通过 — 内层不 readonly

规避：要深 readonly 用工具类型 DeepReadonly<T>（11 章用 Mapped Type 实现），或用 Readonly<{...}> 嵌套手写。

陷阱 6 · Object / object / {} 三个不同类型

三个长得像但语义完全不同：

类型含义能赋什么
Object所有非 null/undefined（含原始）"a" / 1 / true / {} / []
object非原始（小写 o）{} / [] / 函数
{}所有非 null/undefined同 Object

规避：几乎从不直接用这三个。要"任意对象"用 Record<string, unknown>；要"任何值"用 unknown。
