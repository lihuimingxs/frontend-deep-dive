14 · Decorators

TS 装饰器是当前最混乱的话题之一：现存两套互不兼容的装饰器系统 —— TS 5.0 默认启用的 Stage 3 标准装饰器、以及通过 experimentalDecorators 启用的 legacy 装饰器。NestJS / TypeORM / TypeDI 等主流框架仍依赖 legacy，因为标准还没补齐元数据反射能力。本章并排讲两套系统，回答：它们怎么不兼容、什么时候用哪套、为什么 Stage 3 不支持 parameter decorator。

阅读时长 ~35 分钟 前置 06 Classes 对应 typescriptlang.org Decorators

01 速览

装饰器是函数，写在 class / method / property / parameter 前面，运行时被框架调用以"修饰"目标。例如：

typescript 
@logged           // 类装饰器
class User {
  @bind           // method 装饰器
  greet() {}
}

// 装饰器函数本身
function logged(value: any, context: ClassDecoratorContext) {
  console.log(`Class ${context.name} defined`);
}

关键现状：

Stage 3（标准）Legacy（实验性）
启用方式TS 5.0+ 默认experimentalDecorators: true
装饰器签名(value, context)(target, propertyKey?, descriptor?)
支持位置class / method / accessor / field+ parameter decorator
元数据反射📝 提案中（context.metadata）✓ reflect-metadata
NestJS 兼容❌ 仍依赖 legacy✓
TypeORM 兼容❌ 仍依赖 legacy✓
未来方向✓ 长期标准逐步淘汰

实用判断：

  • 用 NestJS / TypeORM / TypeDI / class-validator → 必须 legacy
  • 纯业务横切（日志 / 缓存 / 重试）→ Stage 3 即可
  • 新建库 → Stage 3（除非要做 DI 框架）

02 底层逻辑

1. Stage 3 标准装饰器：(value, context)

标准装饰器签名极简：每个装饰器接受两个参数：

  • value：被装饰的东西本身（class / method / accessor / field 初始化器）
  • context：富含元数据的对象，含 name、kind、static 标志、addInitializer 等
typescript · Stage 3 
// Class decorator
function logged<T extends new (...args: any[]) => any>(
  value: T,
  context: ClassDecoratorContext,
): T {
  console.log(`Class ${context.name} created`);
  return class extends value {
    constructor(...args: any[]) {
      super(...args);
      console.log(`Instance of ${context.name} created`);
    }
  };
}

@logged
class User {
  constructor(public name: string) {}
}

// Method decorator
function bound(
  value: Function,
  context: ClassMethodDecoratorContext,
) {
  context.addInitializer(function () {
    (this as any)[context.name] = value.bind(this);
  });
}

class Counter {
  count = 0;
  @bound
  inc() { this.count++; }
}
const c = new Counter();
const fn = c.inc;
fn();   // ✓ this 已绑定 — 不会丢失

关键设计：装饰器只接收上下文，不修改 class 内部细节。返回新值（class / function）替换原值；要在初始化时跑代码用 context.addInitializer。

2. Legacy 装饰器：(target, propertyKey, descriptor)

legacy 设计早于 ES 标准化。语义和 Object.defineProperty 紧密绑定：

typescript · legacy 
// experimentalDecorators: true 才能编译

// Class decorator
function logged(constructor: Function) {
  console.log(`Class ${constructor.name} defined`);
}

// Method decorator
function timed(target: any, propertyKey: string, descriptor: PropertyDescriptor) {
  const original = descriptor.value;
  descriptor.value = function (...args: any[]) {
    const start = Date.now();
    const result = original.apply(this, args);
    console.log(`${propertyKey} took ${Date.now() - start}ms`);
    return result;
  };
}

// Property decorator
function readonly(target: any, propertyKey: string) {
  Object.defineProperty(target, propertyKey, { writable: false });
}

// Parameter decorator（只有 legacy 支持！）
function inject(target: any, propertyKey: string | symbol | undefined, parameterIndex: number) {
  // 通常存到元数据里供 DI 容器后续用
}

@logged
class Service {
  @readonly
  version = "1.0";

  @timed
  process(@inject input: string) { return input.length }
}

3. reflect-metadata + emitDecoratorMetadata —— DI 的核心

NestJS / TypeORM 等 DI 框架需要在运行时知道方法参数的类型（这样才能 inject）。但 TS 类型是擦除的（01 章），运行时拿不到。

解决方案：legacy decorator + emitDecoratorMetadata: true。打开后 TS 编译器会把参数类型作为元数据 emit 到产物中，配合 reflect-metadata 库在运行时读取：

typescript 
// tsconfig: { experimentalDecorators: true, emitDecoratorMetadata: true }
import "reflect-metadata";

class Logger { log(s: string) {} }
class Database { query(sql: string) {} }

@Injectable()
class UserService {
  // 编译产物中：Reflect.metadata("design:paramtypes", [Logger, Database]) 被注入
  constructor(
    private logger: Logger,
    private db: Database,
  ) {}
}

// DI 容器读取元数据
function getDeps(target: any): any[] {
  return Reflect.getMetadata("design:paramtypes", target) || [];
}

console.log(getDeps(UserService));   // [Logger, Database]

产物大约长这样（关键的 Reflect.metadata 调用）：

javascript · 编译产物 
let UserService = class UserService {
  constructor(logger, db) {
    this.logger = logger;
    this.db = db;
  }
};
UserService = __decorate([
  Injectable(),
  __metadata("design:paramtypes", [Logger, Database]),   // ← 关键
], UserService);

这就是 NestJS / TypeORM 无法迁移到 Stage 3 的根本原因 —— Stage 3 暂时没有等价的 metadata 机制。TC39 decorator metadata 提案正在补这个洞，但还没完全到位。

4. 装饰器组合的求值顺序

多个装饰器组合时遵循 "工厂从上到下、装饰器从下到上" 的规则：

typescript 
function f(name: string) {
  console.log(`f(${name}) — factory`);
  return function (...args: any[]) {
    console.log(`f(${name}) — decorator`);
  };
}

class Test {
  @f("first")
  @f("second")
  method() {}
}

// 输出顺序：
// f(first) — factory      ← 工厂从上到下
// f(second) — factory
// f(second) — decorator   ← 装饰器从下到上（先内层）
// f(first) — decorator

类比：装饰器像数学函数组合，@a @b method 等价于 a(b(method)) —— 内层先执行。

5. 为什么 Stage 3 不支持 parameter decorator

Stage 3 的设计目标是纯函数化、无副作用。parameter decorator 的本质是"给某个参数位置打标记"，这种"非局部副作用"违反 Stage 3 设计哲学。

替代方案（Stage 3 思路）：

typescript 
// ❌ Stage 3 不允许
class Service {
  process(@inject input: string) {}   // 参数装饰器无效
}

// ✅ Stage 3 思路：method decorator + context.addInitializer + 自定义 metadata
function inject(...args: string[]) {
  return function (value: Function, context: ClassMethodDecoratorContext) {
    context.addInitializer(function () {
      (this as any).__deps = args;
    });
    return value;
  };
}

class Service {
  @inject("logger", "db")
  process(logger: any, db: any) {}
}

6. context.metadata（Stage 3 元数据 API）

TS 5.2+ 的 Stage 3 装饰器有一个简化版 metadata：通过 context.metadata 访问一个共享对象，装饰器可读写：

typescript · TS 5.2+ 
// 简单版元数据（不需 reflect-metadata）
function tagged(tag: string) {
  return function (value: any, context: ClassDecoratorContext) {
    context.metadata.tag = tag;
  };
}

@tagged("api-handler")
class UserHandler {}

// 读取元数据
const metadata = (UserHandler as any)[Symbol.metadata];
console.log(metadata.tag);   // "api-handler"

但这个 metadata 比 legacy 的 design:paramtypes 简单得多，不能自动 emit 类型信息。要做 DI 仍要手写注解。完整的"标准 metadata 配 DI"还需 TC39 提案进一步推进。

7. 装饰器 vs 高阶函数：什么时候不用装饰器

很多"装饰器"工作可以用普通高阶函数完成：

typescript 
// 装饰器风格
class Service {
  @cached
  @retry(3)
  async fetchUser(id: string) { /* ... */ }
}

// 高阶函数风格 — 等价
class Service {
  fetchUser = withCache(withRetry(3, async (id: string) => { /* ... */ }));
}

// 或更平：
async function rawFetchUser(id: string) { /* ... */ }
const fetchUser = withCache(withRetry(3, rawFetchUser));

什么时候用装饰器更好：装饰器需要访问 class / method 元信息（名字、static、kind）时；或框架/约定要求（NestJS）。否则高阶函数更简单、更显式、和泛型契合更好。

03 应用场景

场景 A · NestJS Controller / Service / DI（legacy 必需）

typescript 
// tsconfig: experimentalDecorators + emitDecoratorMetadata 必开
import { Controller, Get, Param, Inject } from "@nestjs/common";

@Injectable()
class UserService {
  constructor(@InjectRepository(User) private repo: Repository<User>) {}
  findById(id: string) { return this.repo.findOneBy({ id }) }
}

@Controller("users")
class UserController {
  constructor(private service: UserService) {}    // 自动注入

  @Get(":id")
  async getOne(@Param("id") id: string) {
    return this.service.findById(id);
  }
}

场景 B · TypeORM Entity（legacy 必需）

typescript 
@Entity()
class Article {
  @PrimaryGeneratedColumn("uuid")
  id!: string;

  @Column({ length: 200 })
  title!: string;

  @Column("text")
  body!: string;

  @ManyToOne(() => User, user => user.articles)
  author!: User;

  @CreateDateColumn()
  createdAt!: Date;
}

TypeORM 在运行时通过 reflect-metadata 读取 @Column 配的类型，自动建表、查询时反序列化。这种"零样板"体验完全依赖 legacy + emitDecoratorMetadata。

场景 C · 业务横切（Stage 3 即可）

typescript · Stage 3 
// 缓存装饰器
function cached(ttl: number) {
  return function <F extends (...args: any[]) => Promise<any>>(
    value: F,
    context: ClassMethodDecoratorContext,
  ) {
    const cache = new Map<string, { value: any; expires: number }>();
    return async function (this: any, ...args: any[]) {
      const key = JSON.stringify(args);
      const hit = cache.get(key);
      if (hit && hit.expires > Date.now()) return hit.value;
      const result = await (value as any).apply(this, args);
      cache.set(key, { value: result, expires: Date.now() + ttl });
      return result;
    } as F;
  };
}

class UserService {
  @cached(60_000)
  async findById(id: string) {
    return fetch(`/api/users/${id}`).then(r => r.json());
  }
}

场景 D · class-validator 风格的运行时校验

typescript · legacy 
// class-validator 库（依赖 legacy + reflect-metadata）
import { IsEmail, IsInt, Min, validate } from "class-validator";

class CreateUserDto {
  @IsEmail()
  email!: string;

  @IsInt()
  @Min(0)
  age!: number;
}

const dto = new CreateUserDto();
dto.email = "not-email";
dto.age = -1;

const errors = await validate(dto);
console.log(errors);   // 详细的字段错误对象

zod / valibot 等 schema 库提供了非装饰器的等价能力，且不需要 legacy decorator。新代码倾向于 schema 优先。

04 典型代码

Stage 3 全套装饰器 context 类型

typescript 
// 五种 context 类型（lib.decorators.d.ts）
type DecoratorContext =
  | ClassDecoratorContext         // 类
  | ClassMethodDecoratorContext   // method
  | ClassGetterDecoratorContext   // getter
  | ClassSetterDecoratorContext   // setter
  | ClassFieldDecoratorContext    // field
  | ClassAccessorDecoratorContext;// auto-accessor (5.0+)

// 共享字段
interface BaseContext {
  kind: string;           // "class" | "method" | "field" | ...
  name: string | symbol;
  static: boolean;        // 是否 static
  private: boolean;       // 是否 #field
  metadata: DecoratorMetadata;   // 共享元数据（5.2+）
  addInitializer(fn: () => void): void;
  access?: { has(o: any): boolean; get?(o: any): any; set?(o: any, v: any): void };
}

// 例：method decorator 类型
function timed(value: Function, context: ClassMethodDecoratorContext) {
  // value 是原方法
  // context.name 是方法名
  // context.kind === "method"
  return function (this: any, ...args: any[]) {
    const start = performance.now();
    const result = (value as any).apply(this, args);
    console.log(`${String(context.name)} took ${performance.now() - start}ms`);
    return result;
  };
}

auto-accessor decorator（TS 5.0+）

typescript 
// auto-accessor 是 ES 草案：自动生成 getter/setter 配私有字段
class User {
  accessor name = "";
  // 等价于：
  // #name = "";
  // get name() { return this.#name; }
  // set name(v) { this.#name = v; }
}

// 装饰 auto-accessor
function logged<T>(
  target: ClassAccessorDecoratorTarget<any, T>,
  context: ClassAccessorDecoratorContext,
) {
  return {
    get(this: any) {
      const v = target.get.call(this);
      console.log(`get ${String(context.name)} => ${v}`);
      return v;
    },
    set(this: any, v: T) {
      console.log(`set ${String(context.name)} = ${v}`);
      target.set.call(this, v);
    },
  };
}

class Logged {
  @logged
  accessor name = "Alice";
}

const u = new Logged();
u.name;        // get name => Alice
u.name = "B";  // set name = B

Legacy 五种装饰器签名

typescript 
// experimentalDecorators: true

// 1. Class
function classDec<T extends new (...args: any[]) => any>(constructor: T): T {
  return constructor;
}

// 2. Method
function methodDec(
  target: any,
  propertyKey: string,
  descriptor: PropertyDescriptor,
): PropertyDescriptor | void {
  return descriptor;
}

// 3. Property（无第三参数 — descriptor）
function propertyDec(target: any, propertyKey: string): void {}

// 4. Accessor（参数同 method）
function accessorDec(
  target: any,
  propertyKey: string,
  descriptor: PropertyDescriptor,
): PropertyDescriptor | void {}

// 5. Parameter（仅 legacy 有！）
function paramDec(
  target: any,
  propertyKey: string | symbol | undefined,
  parameterIndex: number,
): void {}

class Demo {
  @propertyDec
  field = 1;

  @methodDec
  method(@paramDec arg: string) {}

  @accessorDec
  get x() { return 1 }
}

装饰器工厂（最常见模式）

typescript · Stage 3 
// 接受配置参数的装饰器都是工厂：函数返回装饰器
function logged(prefix: string) {
  return function (value: Function, context: ClassMethodDecoratorContext) {
    return function (this: any, ...args: any[]) {
      console.log(`[${prefix}] ${String(context.name)}`, args);
      return (value as any).apply(this, args);
    };
  };
}

class Service {
  @logged("API")
  fetchUser(id: string) {}

  @logged("DB")
  query(sql: string) {}
}

装饰器顺序的实战影响

typescript 
// 多个装饰器组合 — 顺序决定行为
class Service {
  @cached(60_000)        // 外层：先查缓存
  @retry(3)              // 内层：失败重试
  @timed                 // 最内：计时
  fetchData() { /* ... */ }
}

// 调用顺序：
// 1. cached 检查命中
// 2. miss 时进入 retry
// 3. retry 内部计时
// 4. 实际 fetchData 执行
// 5. 结果回到 retry → cached 缓存

// 顺序换一下，行为也变：
class Service2 {
  @retry(3)              // 外层：每次都重试
  @cached(60_000)        // 内层：每次重试也走缓存（可能永远 hit）
  fetchData() { /* ... */ }
}

05 常见陷阱

⚠️ 陷阱 1 · legacy 与 Stage 3 不能混用
json 
{
  "compilerOptions": {
    "target": "ES2022",
    "experimentalDecorators": true   // 开 = legacy
    //                                关 = Stage 3
  }
}

一个项目只能选一套。NestJS 项目继续用 legacy；纯业务代码新项目用 Stage 3。Monorepo 中如果一个 app 是 NestJS 而另一个是 Vite + React，可以分别配 tsconfig。

⚠️ 陷阱 2 · Stage 3 不支持 parameter decorator
typescript 
// Stage 3 下：
class Service {
  process(@inject input: string) {}
  //      ^^^^^^^   ❌ Decorators are not valid here
}

规避：迁移到 Stage 3 时把 parameter decorator 改写成 method decorator + 元数据存储；或者继续用 legacy。

陷阱 3 · reflect-metadata 必须在入口 import
typescript 
// ❌ 错的
// main.ts
import { App } from "./app";
new App();   // 运行时炸：Reflect.getMetadata is not a function

// ✅ 对：reflect-metadata 必须在所有装饰器使用之前加载
// main.ts
import "reflect-metadata";   // ← 必须第一行
import { App } from "./app";
new App();

规避：在项目主入口（main.ts / index.ts）的第一行 import "reflect-metadata"。NestJS 自动处理这个，但自建项目要小心。

陷阱 4 · 装饰器执行顺序与直觉相反

"工厂从上到下、装饰器从下到上"是固定规则。常见踩坑：

typescript 
// 期望：先验证再缓存
class Service {
  @cached
  @validate    // 想：先验证
  fetchData() {}
}

// 实际：装饰器从下到上 — validate 在 cached 内层
// 调用：
// cached 拦截 → 先查缓存 → miss 时调 validate → validate 调 method
// 即：缓存命中时 validate 不会跑！

// 如果想"每次都先 validate"：调换顺序
class Service2 {
  @validate    // 外层：每次都 validate
  @cached      // 内层：通过校验后才查缓存
  fetchData() {}
}
陷阱 5 · @types/reflect-metadata 与运行时 reflect-metadata 不匹配

同步两个包版本：reflect-metadata（运行时）和 @types/reflect-metadata（类型，旧版本可能已过时）。新版 reflect-metadata 自带 .d.ts，不需要 @types。

json 
// package.json
{
  "dependencies": {
    "reflect-metadata": "^0.2.2"   // 新版本自带类型
  }
  // 不需要 @types/reflect-metadata
}
陷阱 6 · 装饰器在 class field 上的初始化时机
typescript · Stage 3 
function counted(value: any, context: ClassFieldDecoratorContext) {
  let count = 0;
  return function (this: any, init: any) {
    count++;
    console.log(`Field ${String(context.name)}: instance ${count}`);
    return init;
  };
}

class C {
  @counted
  x = 1;
}

new C();   // Field x: instance 1
new C();   // Field x: instance 2
// 装饰器 returned function 在每次实例化时跑，不是 class 定义时

规避：理解 field decorator 的"初始化函数"是每个实例独立调用的。如果想"每个 class 跑一次"用 class decorator。
