12 · satisfies 与组合艺术

TS 4.9 引入的 satisfies 操作符解决了一个困扰多年的两难选择：怎样让一个值既被某个类型「检查」、又保留它「字面量精确推断」。本章先把 : T / as T / satisfies T 三者本质差异讲透，然后展示 satisfies 在配置对象、路由表、状态机的实战；再演示 09/10/11 三剑客组合做 branded type 和 DSL，作为 Part 2 综合实战收官。

阅读时长 ~30 分钟 前置 11 Utility Types 对应 typescriptlang.org TS 4.9 Release Notes

01 速览

三种"约束类型"语法对比：

typescript 
type Color = "red" | "green" | "blue";
type Palette = Record<string, Color | [number, number, number]>;

// 1️⃣ 类型注解 : T — 检查 + 把变量类型变成 T
const palette1: Palette = {
  red: "red",
  green: [0, 255, 0],
};
palette1.red.toUpperCase();   // ❌ 'string' | '[number, number, number]' 上不存在

// 2️⃣ 断言 as T — 强制变成 T，无安全检查
const palette2 = {
  red: "red",
  green: [0, 255, 0],
} as Palette;
palette2.red.toUpperCase();   // ❌ 同样退化为 union

// 3️⃣ satisfies T (4.9+) — 检查但保留具体推断
const palette3 = {
  red: "red",
  green: [0, 255, 0],
} satisfies Palette;
palette3.red.toUpperCase();   // ✓ red 推断为 "red"，仍是 string
palette3.green.length;         // ✓ green 推断为 [number, number, number]，有 length

satisfies 既校验形状（违反 Palette 约束会报错）又保留具体推断（每个字段的精确类型仍可见）。这是配置对象的最佳工具。

02 底层逻辑

1. 三种语法的本质差异

关键认知：: T 把变量类型「变成」T；as T 强制「断言」是 T；satisfies T 仅「校验」是否可赋给 T，不改变量类型。

: Tas Tsatisfies T
检查值符合 T✓有限（仅"明显矛盾"）✓ 严格
excess property check✓✗✓
变量最终类型TT原始推断的精确类型
字面量保留✗ 退化✗ 退化✓ 保留
支持版本始终始终4.9+
typescript · 类型变化对比 
type Config = { mode: "dark" | "light"; padding: number };

const a = { mode: "dark", padding: 10 } as Config;
//    ^? Config         字面量丢失

const b: Config = { mode: "dark", padding: 10 };
//    ^? Config         字面量丢失

const c = { mode: "dark", padding: 10 } satisfies Config;
//    ^? { mode: "dark"; padding: number }   ← 仍精确

2. as const satisfies T：双保险模式

as const 让所有字面量保留 + 字段 readonly；satisfies T 在此基础上追加形状校验。这是写"类型化常量"的标准 idiom：

typescript 
type RouteConfig = Record<string, { method: "GET" | "POST"; auth?: boolean }>;

const ROUTES = {
  "/": { method: "GET" },
  "/login": { method: "POST" },
  "/admin": { method: "GET", auth: true },
} as const satisfies RouteConfig;

//    ^? {
//      readonly "/": { readonly method: "GET" };
//      readonly "/login": { readonly method: "POST" };
//      readonly "/admin": { readonly method: "GET"; readonly auth: true };
//    }
//
//    既精确（字面量都保留）又被校验（拼错 method 会报错）

ROUTES["/login"].method;   // ✓ 精确推为 "POST"，可做 narrow

// 派生：所有 path 的 union
type Path = keyof typeof ROUTES;   // "/" | "/login" | "/admin"

3. 没有 satisfies 之前的痛苦

为什么 satisfies 这么受欢迎？看看以前怎么解决"既要校验又要精确"：

typescript · 4.9 之前的两条死路 
// 死路 1：用 : T —— 校验 OK，但字面量丢失
type Config = { mode: "dark" | "light" };
const a: Config = { mode: "dark" };
a.mode.toUpperCase();   // mode 是 "dark" | "light"，无法收窄到 "dark"

// 死路 2：用 helper function 强制泛型推断
function makeConfig<T extends Config>(c: T): T { return c }
const b = makeConfig({ mode: "dark" });
//    ^? { mode: "dark" }   ✓ 但要写一个永远只 return 参数的函数

// satisfies 直接解决：
const c = { mode: "dark" } satisfies Config;
//    ^? { mode: "dark" }   ✓

makeConfig 的 helper trick 是社区流行的"满足约束 + 保留推断"方案。satisfies 让它成为语言原生功能，不再需要一次性 helper。

4. as 是危险的，satisfies 几乎总是更好

typescript 
interface Options { width: number; height: number }

// ❌ as：拼写错误也允许
const a = { width: 100, heigh: 200 } as Options;
//                       ^^^^^   typo 不报，运行时炸

// ✅ satisfies：拼写错误立刻报
const b = { width: 100, heigh: 200 } satisfies Options;
//                       ^^^^^   ❌ 'heigh' is not assignable to property 'height'

规则：每写一个 as T 都问"能不能换 satisfies T"。绝大多数能换。无法换的少数情况：

  • 明确想"放宽"类型（把 unknown 当 SomeType 用）—— 这种情况用 type guard / zod 替代更好
  • 类型不兼容但你担保运行时安全（如 DOM 元素 narrow）—— 这是 as 真正合理的场景

5. Branded type：用结构系统模拟名义类型

02 章讲过 TS 是结构化类型系统：两个长得一样的类型互相兼容。但有时你想要区分（比如 UserId 和 OrderId 都是 string，但不应该互传）。

解法：在 string 上叠加一个"brand 字段"，让两个类型在结构上有差异：

typescript 
// 用 unique symbol 创建运行时唯一的 brand
declare const UserIdBrand: unique symbol;
declare const OrderIdBrand: unique symbol;

type UserId = string & { readonly [UserIdBrand]: never };
type OrderId = string & { readonly [OrderIdBrand]: never };

// 构造函数（运行时只是 string，brand 字段不存在）
const UserId = (id: string) => id as UserId;
const OrderId = (id: string) => id as OrderId;

function findUser(id: UserId): User | null { /* ... */ return null; }

const uid = UserId("u_1");
const oid = OrderId("o_1");

findUser(uid);            // ✓
findUser(oid);            // ❌ OrderId not assignable to UserId
findUser("u_2");          // ❌ string not assignable to UserId（必须显式 brand）

关键观察：

  • 运行时仍是 string，零开销
  • 编译期不能互传，强制 explicit boxing
  • brand 字段类型用 never 是技巧 —— 即使你知道 brand 字段名，也无法构造一个 never 值
  • unique symbol 让 brand 之间互相不可赋值（即使写一样的字段名）

6. assertion function + 类型谓词的高级组合

03 章讲过 type predicate 和 assertion function。在 Part 2 学完 utility 后，可以组合得很精巧：

typescript 
// 通用：从 unknown 校验某字段
function hasField<K extends string>(o: unknown, k: K): o is { [P in K]: unknown } {
  return typeof o === "object" && o !== null && k in o;
}

const data: unknown = JSON.parse('{"name": "Alice"}');
if (hasField(data, "name")) {
  data.name;    // ✓ unknown
}

// 用 assertion function 做"信任边界守门"
function assertObject(x: unknown): asserts x is Record<string, unknown> {
  if (typeof x !== "object" || x === null) throw new Error("not an object");
}

const data2: unknown = JSON.parse(input);
assertObject(data2);
data2.name;    // ✓ unknown

7. 三剑客组合做 DSL：路由 + 参数 + 类型化 navigate

09 + 10 + 11 + satisfies 全套组合做"端到端类型安全的路由"：

typescript 
// Step 1: 用 satisfies 定义路由配置
const routes = {
  home: "/",
  user: "/users/:userId",
  post: "/users/:userId/posts/:postId",
} as const satisfies Record<string, string>;

// Step 2: 提取参数类型（09 章 conditional + infer）
type ExtractParams<P extends string> =
  P extends `${string}/:${infer Param}/${infer Rest}` ? Param | ExtractParams<`/${Rest}`> :
  P extends `${string}/:${infer Param}` ? Param :
  never;

// Step 3: 用 mapped type 转成参数对象（10 章）
type ParamsOf<P extends string> = {
  [K in ExtractParams<P>]: string;
};

// Step 4: 类型安全的 navigate
type RouteName = keyof typeof routes;
type RoutePath<R extends RouteName> = (typeof routes)[R];

function navigate<R extends RouteName>(
  name: R,
  params: ParamsOf<RoutePath<R>>,
): string {
  let url: string = routes[name];
  for (const [k, v] of Object.entries(params)) {
    url = url.replace(`:${k}`, v as string);
  }
  return url;
}

navigate("home", {});                                   // ✓
navigate("user", { userId: "u_1" });                     // ✓
navigate("post", { userId: "u_1", postId: "p_1" });      // ✓
navigate("post", { userId: "u_1" });                     // ❌ Property 'postId' is missing
navigate("typo", {});                                    // ❌ 'typo' not assignable

这是 TanStack Router / typed-fetch 等库的核心套路。一份配置，编译期保证路由名 + 参数对象双重精确。

03 应用场景

场景 A · Tailwind / 设计 token 配置

typescript 
type ColorScale = Record<string, string | Record<string, string>>;

const colors = {
  white: "#ffffff",
  primary: {
    50: "#f0f9ff",
    500: "#0ea5e9",
    900: "#0c4a6e",
  },
  // typo: { 100: 123 },   ❌ value 必须 string
} as const satisfies ColorScale;

// 使用时类型精确
colors.primary[500];   // ✓ "#0ea5e9"，仍是字面量
colors.white;           // ✓ "#ffffff"

type ColorKey = keyof typeof colors;            // "white" | "primary"
type ShadeKey = keyof typeof colors.primary;    // "50" | "500" | "900"

场景 B · 状态机定义

typescript 
type StateMachine<S extends string, E extends string> = {
  initial: S;
  states: Record<S, { on?: Partial<Record<E, S>> }>;
};

const trafficLight = {
  initial: "red",
  states: {
    red:    { on: { TIMER: "green" } },
    green:  { on: { TIMER: "yellow" } },
    yellow: { on: { TIMER: "red" } },
  },
} as const satisfies StateMachine<"red" | "green" | "yellow", "TIMER">;

// 类型上：每个 state 的 transitions 字段都被校验
// 同时保留字面量，可以做精确推断：
type InitialState = typeof trafficLight.initial;   // "red"
type RedTransitions = typeof trafficLight.states.red.on;
//   ^? { readonly TIMER: "green" }

场景 C · API client schema 驱动

typescript 
import { z } from "zod";

const apiSchemas = {
  getUser: {
    input: z.object({ id: z.string() }),
    output: z.object({ id: z.string(), name: z.string() }),
  },
  createPost: {
    input: z.object({ title: z.string(), body: z.string() }),
    output: z.object({ id: z.string(), createdAt: z.coerce.date() }),
  },
} as const satisfies Record<string, { input: z.ZodType; output: z.ZodType }>;

type ApiName = keyof typeof apiSchemas;
type Input<K extends ApiName> = z.infer<(typeof apiSchemas)[K]["input"]>;
type Output<K extends ApiName> = z.infer<(typeof apiSchemas)[K]["output"]>;

async function call<K extends ApiName>(name: K, input: Input<K>): Promise<Output<K>> {
  const { input: inSchema, output: outSchema } = apiSchemas[name];
  const validated = inSchema.parse(input);
  const response = await fetch(`/api/${name}`, { method: "POST", body: JSON.stringify(validated) });
  return outSchema.parse(await response.json());
}

const u = await call("getUser", { id: "1" });   // u: { id: string; name: string }
const p = await call("createPost", { title: "Hi", body: "..." });
//    ^? { id: string; createdAt: Date }

场景 D · Branded type 防止 ID 混用

typescript 
declare const UserId: unique symbol;
declare const PostId: unique symbol;
type UserId = string & { readonly [UserId]: never };
type PostId = string & { readonly [PostId]: never };

// 构造器（验证 + brand）
function userId(s: string): UserId {
  if (!/^u_\w+$/.test(s)) throw new Error("Invalid user id");
  return s as UserId;
}
function postId(s: string): PostId {
  if (!/^p_\w+$/.test(s)) throw new Error("Invalid post id");
  return s as PostId;
}

interface User { id: UserId; name: string }
interface Post { id: PostId; authorId: UserId; title: string }

async function getPosts(authorId: UserId): Promise<Post[]> { /* ... */ return [] }

const uid = userId("u_alice");
const pid = postId("p_hello");

getPosts(uid);     // ✓
getPosts(pid);     // ❌ PostId not assignable to UserId
getPosts("u_x");   // ❌ string not assignable to UserId

// 同时仍是 string —— 字符串方法可用
uid.startsWith("u_");   // ✓ 因为 UserId & string

04 典型代码

satisfies 的常见 idiom

typescript 
// idiom 1：带值字面量保留 — 最高频
const config = { theme: "dark", port: 3000 } satisfies Config;

// idiom 2：as const + satisfies — 完全冻结
const ROUTES = { home: "/" } as const satisfies Record<string, string>;

// idiom 3：函数签名校验
const handlers = {
  click: (e: MouseEvent) => e.preventDefault(),
  submit: (e: SubmitEvent) => e.preventDefault(),
} satisfies Record<string, (e: Event) => void>;

// idiom 4：枚举式 union 的精确 default
function getDefault(): "dark" | "light" {
  return "dark" satisfies "dark" | "light";
  //     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^   等价 "dark" as "dark" | "light" 但更安全
}

// idiom 5：联合类型 narrow 后断言
type Status = "ok" | "error" | "pending";
const s = "ok" satisfies Status;
//    ^? "ok"   保留窄类型

类型谓词 + utility 组合

typescript 
// 把 unknown 安全地转换为已知类型
function isType<T>(o: unknown, schema: z.ZodType<T>): o is T {
  return schema.safeParse(o).success;
}

const data: unknown = await fetch("/api").then(r => r.json());
const UserSchema = z.object({ id: z.string(), name: z.string() });

if (isType(data, UserSchema)) {
  data.name;   // ✓ data: { id: string; name: string }
}

// 用 NonNullable 配合 type predicate
function isPresent<T>(x: T): x is NonNullable<T> {
  return x != null;
}

const items: (string | null | undefined)[] = ["a", null, "b"];
const valid = items.filter(isPresent);   // string[]

组合 conditional + mapped + template literal 做 DSL

typescript 
// CSS-in-JS 风格：从 token 名生成完整 className union
type Spacing = 1 | 2 | 4 | 8;
type Direction = "t" | "r" | "b" | "l" | "x" | "y" | "";

type MarginClass = `m${Direction}-${Spacing}`;
//   ^? "mt-1" | "mr-1" | "mb-1" | "ml-1" | "mx-1" | "my-1" | "m-1"
//      | "mt-2" | ... 共 28 个

type PaddingClass = `p${Direction}-${Spacing}`;

function setClass(c: MarginClass | PaddingClass) { /* ... */ }
setClass("mt-4");    // ✓
setClass("mt-3");    // ❌ 3 不在 Spacing 里
setClass("xx-1");    // ❌

用 satisfies 替代 helper functions

typescript 
// ❌ 4.9 之前：写一个 helper 强制约束推断
function defineConfig<T extends Config>(c: T): T { return c }
const cfg1 = defineConfig({ port: 3000 });

// ✅ 4.9+：直接用 satisfies
const cfg2 = { port: 3000 } satisfies Config;

// 当库需要让用户写"被约束的字面量"时，satisfies 让 helper 不再必要
// 库只需提供类型即可：export type Config = { ... }
// 用户用：const c = { ... } satisfies Config;

Part 2 综合实战：表单生成器

typescript 
// Step 1: 字段定义
type FieldDef =
  | { type: "string"; required?: boolean }
  | { type: "number"; min?: number; max?: number }
  | { type: "select"; options: readonly string[] };

// Step 2: 字段定义 → 字段值类型
type ValueOf<F extends FieldDef> =
  F extends { type: "string" } ? string :
  F extends { type: "number" } ? number :
  F extends { type: "select"; options: infer O } ? O extends readonly string[] ? O[number] : never :
  never;

// Step 3: schema → 最终值对象类型
type FormValues<S extends Record<string, FieldDef>> = {
  [K in keyof S]: ValueOf<S[K]>;
};

// Step 4: 配合 satisfies 写 schema
const schema = {
  name: { type: "string", required: true },
  age: { type: "number", min: 0 },
  role: { type: "select", options: ["admin", "user"] as const },
} as const satisfies Record<string, FieldDef>;

// Step 5: 派生最终值类型
type Values = FormValues<typeof schema>;
//   ^? { name: string; age: number; role: "admin" | "user" }

const v: Values = { name: "Alice", age: 30, role: "admin" };
// const v2: Values = { name: "A", age: 30, role: "guest" };   ❌ "guest" not allowed

05 常见陷阱

陷阱 1 · satisfies 不改变变量类型

初学者常误以为 satisfies 等价于 : T。它不是：

typescript 
type Config = { port: number; mode?: "dark" | "light" };

const a: Config = { port: 3000 };
a.mode;        // ✓ 类型是 "dark" | "light" | undefined（来自 Config）

const b = { port: 3000 } satisfies Config;
b.mode;        // ❌ Property 'mode' does not exist on type '{ port: number }'
//             因为 b 的类型是 { port: number }，没 mode 字段

规避：要"具体推断 + 保留可选字段"时不能只用 satisfies。要么写完整的字段（{ port: 3000, mode: undefined }），要么直接用 : Config。

陷阱 2 · 把 satisfies 当 as 用
typescript 
// satisfies 严格校验 — 不兼容时报错
const x = "hello" satisfies number;   // ❌ Type 'string' does not satisfy 'number'

// as 几乎万能 — 不兼容时仍可强转
const y = "hello" as unknown as number;   // ✓ 通过编译，运行时炸

// 这是 satisfies 的优点 — 但当你"故意"想绕开类型时它不让你
// 这种情况下 as 才是正确工具

规则：要"违反"类型就用 as（少数合法场景）；要"校验"用 satisfies。它们是两个互补工具，不能互换。

陷阱 3 · branded type 失去字符串方法
typescript 
type UserId = string & { readonly __brand: "UserId" };

const uid = "u_1" as UserId;
uid.toUpperCase();   // ✓ 因为 string 的方法仍可用

// 但反向：UserId 不能直接传给 string 参数
function log(s: string) {}
log(uid);            // ✓ 这个能通过（UserId & brand 仍然是 string）

// 真正的问题：把 UserId 传给"接受任何字符串"的函数
function takeAnyString(s: string) {}
takeAnyString("hello");    // ✓
takeAnyString(uid);         // ✓ —— 这是对的，UserId 继承 string

// 但也可能不是想要的：
function processUser(id: UserId) {}
processUser("u_x" as UserId);   // ✓
processUser("anything");        // ❌ 必须显式 brand

branded type 提供"不能从 string 隐式赋给 UserId"，但反向"UserId 是 string"仍然成立。这是合理的不对称。

陷阱 4 · 滥用 branded type 造成类型噪声

不是所有 string 都需要 brand。给每个字段都 brand 一遍会让类型签名爆炸：

typescript 
// ❌ 过度 brand
type FirstName = string & { __brand: "FirstName" };
type LastName = string & { __brand: "LastName" };
type Email = string & { __brand: "Email" };
type Phone = string & { __brand: "Phone" };

// 现在所有函数签名都得到处 brand
function createUser(fn: FirstName, ln: LastName, e: Email, p: Phone) {}
// 调用方 boilerplate 增多

// ✅ 只 brand 那些"真的会被混用的"或"语义重要的"
type UserId = string & { __brand: "UserId" };
// FirstName / LastName 互传也无害，普通 string 即可

规则：brand 用在结构相同但语义不能混的地方（ID 类、Currency 单位类、加密前/后值）。普通 string 字段不要 brand。

陷阱 5 · Part 2 三剑客组合的编译性能问题

Conditional + Mapped + Template Literal 嵌套调用会让编译时间剧增：

  • 大 union × template literal 笛卡尔积
  • 深递归 conditional types
  • 复杂 mapped type 中嵌套 conditional
  • 所有 utility type 一层套一层

诊断：

bash 
tsc --extendedDiagnostics
# 看 "Check time" / "Total time"，定位是哪个文件慢

tsc --generateTrace ./trace
# 生成详细 trace，用 chrome://tracing 打开

规避：

  • 能用 satisfies 简单解决的不要硬上类型操作
  • 大型 utility 写好后缓存为命名 type，不要每次都展开
  • 避免在 hot path（每个文件都 import）放复杂类型
  • 用第三方现成的（type-fest）而不是重新发明
陷阱 6 · 期望 satisfies 报错但它通过了（结构兼容陷阱）
typescript 
type Sized = { size: number };

// 期望：报错（多了字段）
const a = { size: 10, color: "red" } satisfies Sized;
// 实际：✓ 通过 — satisfies 也走"结构兼容"

// 但 excess property check 在对象字面量直接 satisfies时会触发
const b = { size: 10, color: "red" } satisfies Sized;
//                    ^^^^^   嗯实际上这里也通过，satisfies 没有 excess property check 的"立即字面量"特别保护
// 注：表现上 satisfies 与 : T 的 excess property check 都触发，但 4.9 早期版本行为不一致

规避：要严格"形状完全等于 T"用 helper：identity<T>(x: T): T，调用 identity<Sized>({...})。
