09 · Conditional Types & infer

这是 TS 类型系统的图灵完备分水岭。Conditional types（T extends U ? X : Y）让类型有"分支判断"能力；infer 让分支条件中"打个洞"做模式匹配。两者组合能写出 ReturnType / Awaited / Parameters 这种"从已有类型抽取信息"的 utility，是后续所有高级类型操作的基础。

⚠️ 本章是 Part 2 最难一章。最关键的认知转变是理解：extends 不是 OOP 继承，distributive 是默认隐藏行为，infer 是模式匹配（不是变量声明）。请慢读，多写代码验证。

阅读时长 ~45 分钟 前置 07 Generics + 08 keyof/typeof 对应 typescriptlang.org Handbook · Conditional Types

01 速览

语法：

typescript 
// SomeType extends OtherType ? TrueBranch : FalseBranch

type IsString<T> = T extends string ? true : false;

type A = IsString<"hello">;   // true
type B = IsString<42>;         // false

// infer：在条件位置打"模式洞"，匹配到则在 then 分支拿到值
type ElementOf<T> = T extends (infer E)[] ? E : never;

type C = ElementOf<string[]>;     // string
type D = ElementOf<[1, 2, 3]>;     // 1 | 2 | 3
type E = ElementOf<number>;        // never（不匹配）

直觉对比：JS 的 x === "abc" ? "y" : "n" 是值层三元；TS 的 T extends "abc" ? "y" : "n" 是类型层三元。但有几个关键差异：

  • extends 不是 OOP 继承，读作"可赋给"
  • 当 T 是 union 时，conditional type 默认分发（distributive）
  • infer 是 conditional 内部专属，模式匹配语义

02 底层逻辑

1. extends 读作"可赋给"，不是"继承"

这是初学者最容易陷的语义陷阱。T extends U 在 conditional type 里检查的是"T 类型的值能否赋给 U 类型的变量"，不是 OOP 中"T 是 U 的子类"。

typescript 
type A = "hello" extends string ? true : false;        // true（"hello" 能赋给 string 变量）
type B = string extends "hello" ? true : false;        // false（string 不能赋给字面量 "hello"）

type C = { x: 1; y: 2 } extends { x: 1 } ? true : false; // true（结构兼容）
type D = { x: 1 } extends { x: 1; y: 2 } ? true : false; // false（少了 y）

type E = Array<string> extends Iterable<string> ? true : false;   // true（数组实现 Iterable）

// 重要：这是结构化判断，不看名字
interface Point2D { x: number; y: number }
interface Coord { x: number; y: number }
type F = Point2D extends Coord ? true : false;   // true（同形状）

一个好用的助记法："T 中的值能填进 U 形状的盒子吗"。能填就 true，不能就 false。

2. Distributive：union 输入会自动分发

这是 TS 类型系统最容易让人栽跟头的隐藏语义。当 conditional type 的 T 是"裸"类型参数（没被任何容器包裹）且实参是 union 时，TS 会自动分发到每个成员上：

typescript · distributive 演示 
type ToArray<T> = T extends any ? T[] : never;

type R = ToArray<string | number>;
//   ^? string[] | number[]
//        不是 (string | number)[] !

// 展开过程：
// ToArray<string | number>
// = (string extends any ? string[] : never) | (number extends any ? number[] : never)
// = string[] | number[]

为什么？这是 TS 给 conditional type 设计的分发律：union 在裸类型参数位置时自动展开。这样 Exclude<T, U> / Extract<T, U> 这种"对 union 逐个判断"的操作才能写出来。

严格定义触发分发的条件：

  1. conditional type 中 extends 左侧是裸的类型参数（如 T，不是 T[] / { value: T } / [T]）
  2. 实例化时 T 是 union（含 never —— 见下文）

3. [T] extends [U] —— 显式关闭分发的常见 idiom

有时分发不是想要的。例如想判断"T 是否是 union"：

typescript 
// ❌ 直觉错误的写法
type IsUnion<T> = T extends T ? "always true" : "never";
//                ^^^^^^^^^^^   分发后每个成员都 extends 自己 = true

// 实际效果：
type Test = IsUnion<"a" | "b">;   // "always true" | "always true" = "always true"

// ✅ 用 [T] 包裹强制不分发
type Equal<X, Y> = [X] extends [Y] ? ([Y] extends [X] ? true : false) : false;
//                 ^^^   ^^^   把 X / Y 用 tuple 包起来 — 不再是裸参数

关闭分发的 trick：把类型参数包裹进 tuple [T] 或对象 { value: T }。这样 T 不再是裸的，分发律不触发。

反过来，需要分发时必须保持 T 裸：

typescript 
// 想：把 union 中所有 string 成员变成 string[]，其他保留
// ❌ 错的（不分发）
type WrapStrings<T> = [T] extends [string] ? T[] : T;
type R = WrapStrings<string | number>;
//   ^? string | number   不分发 → 整体判断 [string|number] extends [string]，false → 返回原 T

// ✅ 对（分发）
type WrapStrings<T> = T extends string ? T[] : T;
type R = WrapStrings<string | number>;
//   ^? string[] | number   分发后逐个判断

4. never 在 distributive 里"消失"

never 是空 union。分发到 never 时没有任何成员可处理，结果是 never：

typescript 
type ToArray<T> = T extends any ? T[] : never;

type A = ToArray<never>;   // never （不是 never[]）

// 因为 never 分发后没有任何 case 触发 — 等于 0 个 union 成员

// 要保留 never：用 [T] 关闭分发
type ToArrayN<T> = [T] extends [any] ? T[] : never;
type B = ToArrayN<never>;   // never[]   ✓

这一点常导致"我的工具类型对 never 失效了"的困惑。规避：要处理 never 时显式关闭分发。

5. infer 是模式匹配，不是变量声明

infer R 出现在 conditional type 的 extends 右侧，作用是"在这个位置打个洞，等待 T 的形状去填"。如果形状匹配，就可以在 then 分支引用 R。

typescript 
// ReturnType: 取函数返回类型
type ReturnType<F> = F extends (...args: any[]) => infer R ? R : never;
//                                                 ^^^^^^^   "在返回值位置打个洞"

type R1 = ReturnType<() => string>;            // string
type R2 = ReturnType<(x: number) => boolean>; // boolean
type R3 = ReturnType<number>;                  // never（不是函数）

// 多个 infer：从函数类型抽参数和返回值
type Decompose<F> = F extends (...args: infer A) => infer R ? { args: A; ret: R } : never;

type D = Decompose<(x: string, y: number) => boolean>;
//   ^? { args: [x: string, y: number]; ret: boolean }

关键认知：infer 不是声明一个变量，是声明一个"待匹配的位置"。匹配成功则该位置的实际类型被绑定到名字上。匹配失败走 false 分支。

可以在多个位置 infer 同一个名字 —— 那么所有位置必须能统一到同一个类型：

typescript 
// 同一个 R 出现在两处
type SameRet<A, B> = A extends () => infer R ? B extends () => infer R ? R : never : never;
//                                          ^                      ^   两处必须能统一

// 实际更易懂的写法
type SameRet2<A, B> = A extends () => infer R1
  ? B extends () => infer R2
    ? R1 extends R2 ? R1 : never
    : never
  : never;

6. 标准库的 utility 用 distributive + infer 实现

Exclude<T, U> 的真正实现：

typescript · lib.es5.d.ts 
type Exclude<T, U> = T extends U ? never : T;

// 展开 Exclude<"a" | "b" | "c", "a">:
// = ("a" extends "a" ? never : "a")
//   | ("b" extends "a" ? never : "b")
//   | ("c" extends "a" ? never : "c")
// = never | "b" | "c"
// = "b" | "c"

// 关键：依赖分发性，每个成员独立判断

Extract<T, U> 反过来：

typescript 
type Extract<T, U> = T extends U ? T : never;

type R = Extract<string | number | boolean, string | boolean>;
//   ^? string | boolean

NonNullable：

typescript 
type NonNullable<T> = T extends null | undefined ? never : T;
type R = NonNullable<string | null | undefined>;   // string

Awaited（递归解 Promise 链）：

typescript 
type Awaited<T> =
  T extends null | undefined ? T :
  T extends object & { then(onfulfilled: infer F, ...args: any): any } ?
    F extends ((value: infer V, ...args: any) => any) ?
      Awaited<V> :       // 递归
      never :
    T;

type R1 = Awaited<Promise<string>>;            // string
type R2 = Awaited<Promise<Promise<number>>>;   // number（递归解开）

7. 递归 conditional type 与"tail recursion"

conditional type 可以递归引用自己。常见场景是处理嵌套结构：

typescript 
// 深度展开嵌套数组
type Flatten<T> = T extends (infer U)[] ? Flatten<U> : T;

type R = Flatten<number[][][]>;   // number

// 字符串拆分（TS 4.5+ 支持 tail recursion，深度可达 ~1000）
type Split<S extends string, D extends string> =
  S extends `${infer Head}${D}${infer Tail}` ? [Head, ...Split<Tail, D>] : [S];

type Parts = Split<"a,b,c,d", ",">;   // ["a", "b", "c", "d"]

限制：

  • 非 tail position 的递归限制深度约 50 层
  • tail position（递归在 conditional 的 then/else 末尾）TS 4.5+ 可达 ~1000 层
  • 超出会报 Type instantiation is excessively deep and possibly infinite

实践中 50 层够用 99% 场景。处理超长字符串需要特殊技巧（10 章会展开 template literal 处理）。

8. infer 的高级用法：variadic、constraint、from string

infer 不只是抽简单类型。可以从 tuple、字符串模板里抽：

typescript 
// 从 tuple 抽 head / tail
type Head<T extends unknown[]> = T extends [infer H, ...unknown[]] ? H : never;
type Tail<T extends unknown[]> = T extends [unknown, ...infer R] ? R : [];

type H = Head<[1, 2, 3]>;   // 1
type T = Tail<[1, 2, 3]>;   // [2, 3]

// 从字符串模板抽片段（template literal）
type RouteParam<P extends string> =
  P extends `${string}/:${infer Param}/${infer Rest}` ? Param | RouteParam<`/${Rest}`> :
  P extends `${string}/:${infer Param}` ? Param :
  never;

type R = RouteParam<"/users/:userId/posts/:postId">;
//   ^? "userId" | "postId"

// 带约束的 infer（TS 4.7+）
type FirstStringNumber<T> = T extends [infer N extends number, ...unknown[]] ? N : never;
//                                              ^^^^^^^^^^^^^^^   N 必须是 number 子类型
type R1 = FirstStringNumber<[1, "a"]>;   // 1
type R2 = FirstStringNumber<["x", "a"]>; // never（"x" 不是 number）

03 应用场景

场景 A · 路由参数提取（端到端类型安全的核心）

从路由模板字符串自动派生参数类型，是 tRPC / Tanstack Router / Hono 等的核心：

typescript 
type ExtractParams<T extends string> =
  T extends `${string}/:${infer Param}/${infer Rest}` ?
    { [K in Param | keyof ExtractParams<`/${Rest}`>]: string } :
  T extends `${string}/:${infer Param}` ?
    { [K in Param]: string } :
  {};

// 路由 -> 参数对象类型
type Params1 = ExtractParams<"/users/:id">;
//   ^? { id: string }
type Params2 = ExtractParams<"/users/:userId/posts/:postId">;
//   ^? { userId: string; postId: string }

// 类型安全的 navigate 函数
function navigate<P extends string>(path: P, params: ExtractParams<P>) {
  // 替换 :param 为 params[param]
}

navigate("/users/:userId", { userId: "u_1" });   // ✓
navigate("/users/:userId", { typo: "u_1" });      // ❌
navigate("/users/:userId", {});                    // ❌ 缺 userId

场景 B · Promise 链类型解包

typescript 
// 标准库 Awaited 的简化版
type Unwrap<T> = T extends Promise<infer U> ? Unwrap<U> : T;

async function fetchUser() { return { id: "1", name: "Alice" } }

type User = Awaited<ReturnType<typeof fetchUser>>;
//   ^? { id: string; name: string }

// 即使是嵌套 Promise 也能解开
type X = Awaited<Promise<Promise<Promise<number>>>>;   // number

场景 C · 函数柯里化的类型推导

typescript 
// 把 (a, b, c) => R 变成 (a) => (b) => (c) => R
type Curry<F> =
  F extends (head: infer H, ...rest: infer T) => infer R ?
    T extends [] ? (h: H) => R : (h: H) => Curry<(...args: T) => R> :
  never;

declare function curry<F extends (...args: any[]) => any>(f: F): Curry<F>;

const add3 = (a: number, b: number, c: number) => a + b + c;
const curried = curry(add3);

curried(1)(2)(3);   // 6  类型推导：(h: number) => (h: number) => (h: number) => number

场景 D · 自动派生 JSON 兼容类型

typescript 
// 把任意类型转换为可 JSON 序列化的形式
type ToJSON<T> =
  T extends Date ? string :                                    // Date 序列化为 string
  T extends Map<any, any> | Set<any> | Function ? never :       // 不可序列化
  T extends bigint ? never :                                     // bigint 不可序列化
  T extends Array<infer U> ? Array<ToJSON<U>> :
  T extends object ? { [K in keyof T]: ToJSON<T[K]> } :
  T;

interface User {
  id: string;
  age: number;
  createdAt: Date;
  friends: Map<string, string>;
}

type SerializedUser = ToJSON<User>;
//   ^? { id: string; age: number; createdAt: string; friends: never }
//                                  ^^^^^^             ^^^^^   危险字段被标 never，警告

04 典型代码

distributive 与 [T] 关闭分发对比表

typescript 
// 分发（默认）
type Distributive<T> = T extends string ? T[] : never;
type R1 = Distributive<"a" | 1 | "b">;   // "a"[] | "b"[]
//        ^^^^^^^^^^^^^^   "a" 和 "b" 各自包装

// 不分发（[T] 包裹）
type Atomic<T> = [T] extends [string] ? T[] : never;
type R2 = Atomic<"a" | 1 | "b">;   // never
//        ^^^^^^^^^^^^^^   union 整体不能赋给 [string]，因 1 不是 string

// 不分发 + 严格变体
type AllString<T> = [T] extends [string] ? T : never;
type R3 = AllString<"a" | "b">;        // "a" | "b"  （整体能赋给 string）
type R4 = AllString<"a" | 1>;          // never        （整体不能赋给 string）

Equal<X, Y> —— 严格类型相等的标准实现

typescript 
// 简单实现：对称兼容
type Equal<X, Y> = [X] extends [Y] ? ([Y] extends [X] ? true : false) : false;

type T1 = Equal<string, string>;        // true
type T2 = Equal<string, "abc">;          // false
type T3 = Equal<{ x: 1 }, { x: 1 }>;     // true

// 但有缺陷：any 与任何类型互相 extends
type T4 = Equal<any, string>;            // true ⚠️ 不期望

// 严格版（用 conditional 中的 inference 不变性）
type StrictEqual<X, Y> =
  (<T>() => T extends X ? 1 : 2) extends (<T>() => T extends Y ? 1 : 2) ? true : false;

type S1 = StrictEqual<any, string>;     // false  ✓
type S2 = StrictEqual<{ x: 1 }, { x: 1; y: 2 }>;   // false  ✓

严格 Equal 用了一个高级技巧：把 X / Y 放进函数签名，然后比较两个"以 X / Y 为正题的高阶函数类型是否相等"。这个比较走 TS 内部不变量检查，比 extends 严格。type-fest / utility-types 等库都采用此实现。

infer 抽数组中的元素类型 + tuple 操作

typescript 
// 数组元素
type ItemOf<T> = T extends (infer I)[] ? I : never;
type A = ItemOf<number[]>;     // number
type B = ItemOf<[1, 2, 3]>;    // 1 | 2 | 3

// tuple 长度
type Length<T extends readonly unknown[]> = T["length"];

// tuple 反转
type Reverse<T extends unknown[]> =
  T extends [infer Head, ...infer Tail] ? [...Reverse<Tail>, Head] : [];

type R = Reverse<[1, 2, 3]>;   // [3, 2, 1]

// tuple 拼接
type Concat<A extends unknown[], B extends unknown[]> = [...A, ...B];

type C = Concat<[1, 2], [3, 4]>;   // [1, 2, 3, 4]

从函数类型抽各种信息

typescript 
// 标准 Parameters / ReturnType
type Parameters<F> = F extends (...args: infer A) => any ? A : never;
type ReturnType<F> = F extends (...args: any) => infer R ? R : any;

// 还能抽 this 类型
type ThisParameterType<F> = F extends (this: infer T, ...args: any[]) => any ? T : unknown;

// 抽 first / second 参数
type FirstArg<F> = F extends (a: infer A, ...rest: any[]) => any ? A : never;

// 抽 generator 的 yield 类型
type YieldType<G> = G extends Generator<infer Y, any, any> ? Y : never;

// 抽 Promise 单层
type ResolveOf<P> = P extends Promise<infer R> ? R : never;

条件类型的常见组合 idiom

typescript 
// IsAny<T>：检测 any
type IsAny<T> = 0 extends 1 & T ? true : false;
//             ^^^^^^^^^^^^^   只有 1 & any = any 时 0 extends any 才为 true

type A = IsAny<any>;        // true
type B = IsAny<string>;      // false
type C = IsAny<unknown>;     // false

// IsNever<T>：用 [T] 关闭分发
type IsNever<T> = [T] extends [never] ? true : false;

type D = IsNever<never>;     // true
type E = IsNever<string>;     // false

// IsUnion<T>：T extends T 在 union 时拆开，与原 T 比较
type IsUnion<T, U = T> = T extends T ? [U] extends [T] ? false : true : never;

type F = IsUnion<"a">;        // false
type G = IsUnion<"a" | "b">; // true

05 常见陷阱

陷阱 1 · 忘记分发律导致结果错误

最高频的"为什么我的 conditional type 不对"。一个例子：

typescript 
// 想：当 T 是 string union 时返回 T，否则 never
type AllStrings<T> = T extends string ? T : never;

type R = AllStrings<string | number>;
//   ^? string   不是 never！
//
// 因为分发：
// = (string extends string ? string : never) | (number extends string ? number : never)
// = string | never
// = string

规避：先想清楚要不要分发。要"整体判断"用 [T] 包裹：

typescript 
type AllStrings<T> = [T] extends [string] ? T : never;
type R = AllStrings<string | number>;   // never  ✓
陷阱 2 · 把 Exclude / Extract 写错（少了 distributive）
typescript 
// ❌ 关掉了分发 — 完全失效
type WrongExclude<T, U> = [T] extends [U] ? never : T;

type R1 = WrongExclude<"a" | "b" | "c", "a">;   // "a" | "b" | "c"
//        ^^^^^^^^^^^^^^^   整体 union 不能赋给 "a" → 全部保留

// ✅ 必须分发
type Exclude<T, U> = T extends U ? never : T;
type R2 = Exclude<"a" | "b" | "c", "a">;        // "b" | "c"   ✓
陷阱 3 · never 在分发里"消失"
typescript 
type ToBox<T> = T extends any ? { value: T } : never;
type R = ToBox<never>;   // never  （不是 { value: never }）

// 因为 never 是空 union，分发后 0 个 case 触发

// 解法：用 [T] 关闭分发
type ToBoxSafe<T> = [T] extends [any] ? { value: T } : never;
type R2 = ToBoxSafe<never>;   // { value: never }
陷阱 4 · infer 不出现的位置始终走 false 分支
typescript 
// ❌ 想从对象抽 name 字段类型，但 infer 写错位置
type GetName<T> = T extends infer N ? N : never;
//                            ^^^^^^^   只是把 T 重新绑定到 N — 没意义

// ✅ infer 应该在"形状描述"里
type GetName<T> = T extends { name: infer N } ? N : never;

type A = GetName<{ name: "Alice" }>;   // "Alice"
type B = GetName<{ age: 30 }>;          // never（不匹配形状）
陷阱 5 · 递归条件类型超过深度上限
typescript 
// ❌ 非 tail position 递归 — 50 层就炸
type CountUp<N extends number, R extends number[] = []> =
  R["length"] extends N ? R : [...R, R["length"]];   // 这里递归的是返回值，不算 tail

type T = CountUp<100>;   // ❌ Type instantiation is excessively deep

// ✅ tail position 递归（递归在 conditional 末尾）— 4.5+ 可达 1000 层
type CountUpTail<N extends number, R extends number[] = []> =
  R["length"] extends N ? R : CountUpTail<N, [...R, R["length"]]>;
//                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^   tail position
陷阱 6 · 滥用 conditional types 导致编译变慢

复杂条件类型每次实例化都需要 TS 类型检查器跑一遍，深度组合会让编译速度从秒级跌到分钟级：

  • 大 union 配 distributive：union 有 100 个成员就触发 100 次 conditional 计算
  • 嵌套递归：每层都要展开
  • 跨多个文件的类型联动：一处改动触发整个项目重检

规避：

  • 能用 satisfies 简单解决的不要硬上 conditional types
  • 大型 utility 类型用 tsc --extendedDiagnostics 监控编译时间
  • 第三方库（react-hook-form / zod）的类型已经做了大量优化，自己重新发明前先看库的实现
