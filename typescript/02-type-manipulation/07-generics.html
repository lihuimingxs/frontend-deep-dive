07 · Generics

泛型不是 any 的代名词，是延迟绑定的具体类型。T 是占位符，在调用处被某个具体类型替代，所有出现 T 的位置同步替换 —— 这就是为什么"如果一个类型参数只出现一次，几乎肯定不需要它"。本章覆盖泛型 6 个维度：基础语法、约束、双参数互相约束、class type generic、defaults、TS 5.0 <const T> 与 5.4 NoInfer<T> 这两个新工具。

阅读时长 ~32 分钟 前置 04 Functions 对应 typescriptlang.org Handbook · Generics

01 速览

泛型的最小例子：

typescript 
// 不带泛型 — 类型信息丢失
function identityAny(x: any): any { return x }
const a = identityAny("hello");
//    ^? any   类型已坏

// 带泛型 — 类型信息保留
function identity<T>(x: T): T { return x }
const b = identity("hello");
//    ^? "hello"     字面量类型保留（因为 const）
const c = identity<number>(42);
//    ^? number      显式实参

关键观察：T 在定义时是占位，调用时被替换成具体类型，所有出现 T 的位置同步替换。这是"延迟绑定"的本质。

可在四种位置使用泛型：

  • function：function f<T>(x: T): T
  • type alias：type Box<T> = { value: T }
  • interface：interface Box<T> { value: T }
  • class：class Stack<T> { items: T[] = [] }

02 底层逻辑

1. 延迟绑定：T 在使用点被实例化，不是定义点

理解 T 的生命周期是写好泛型的核心。T 不是一个实际类型，是一个变量名。函数定义时，类型系统知道"这里有一个未确定类型 T"；调用时，根据实参类型推断出具体的 T，然后把所有 T 替换。

typescript · 推断过程示意 
function pair<T>(a: T, b: T): [T, T] {
  return [a, b];
}

// 调用时：
pair(1, 2);
// 1️⃣ 看到 a: T，传入 1（number），假设 T = number
// 2️⃣ 看到 b: T，传入 2（也是 number），与假设一致 ✓
// 3️⃣ 把所有 T 替换为 number：function pair(a: number, b: number): [number, number]
// 返回值类型：[number, number]

pair("a", 1);
// 1️⃣ 看到 a: T，T 推断为 string
// 2️⃣ 看到 b: T，传入 1（number），与 string 不一致
// 3️⃣ TS 取共同父类型：string | number 不兼容时降为更宽 union
// 实际：报错或推断为 string | number 取决于上下文

这个过程叫 type inference。它有几个关键性质：

  • **单调性**：每次实参都给 T 加一个"必须满足"的约束，最终 T 是所有约束的"最小满足"
  • **位置敏感**：同一个 T 出现两次时，两次的实参类型必须能"会和"成一个
  • **可显式覆盖**：用 identity<string>(x) 可以强制指定，跳过推断

2. 约束（constraint）让 T 获得能力

没有约束时，函数体内对 T 几乎什么都不能做：

typescript 
function lengthy<T>(x: T): number {
  return x.length;   // ❌ Property 'length' does not exist on type 'T'
}

为什么？因为 T 可能是任意类型 —— number、boolean 都没有 length。约束告诉 TS"我对 T 的最低要求"：

typescript 
function lengthy<T extends { length: number }>(x: T): number {
  return x.length;   // ✓ T 至少有 length 字段
}

lengthy([1, 2, 3]);     // ✓ 数组有 length
lengthy("abc");          // ✓ 字符串有 length
lengthy({ length: 5 });  // ✓ 显式 length 也行
lengthy(42);             // ❌ number 没有 length

约束读作"T 必须可以赋给 X"（T extends X），不是 OOP 意义上的"继承"。这是结构化类型的概念：T 形状必须包含 X 形状的所有要求。

3. 双类型参数互相约束（K extends keyof T）

一个类型参数可以通过 extends 引用另一个：

typescript 
function getProp<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const u = { name: "Alice", age: 30 };
const n = getProp(u, "name");   // n: string
const a = getProp(u, "age");    // a: number
const x = getProp(u, "email");  // ❌ 'email' is not assignable to '"name" | "age"'

这一组约束做了三件事：

  1. T 没约束 → 任何对象都能传
  2. K extends keyof T → key 必须是 T 实际存在的字段名
  3. 返回类型 T[K] → 用 indexed access 精确取出对应字段类型（08 章详讲）

整体效果：JS 的"动态属性访问"获得了精确类型。obj[key] 的运行时操作在类型层有完全对应的 T[K]。这种"类型层与运行时层一一对应"的能力是 TS 高级类型最迷人的部分。

4. 官方守则："如果一个类型参数只出现一次，几乎肯定不需要它"

泛型的目的是建立两个或多个位置之间的类型关联。只出现一次说明没有关联可建：

typescript 
// ❌ T 只在参数出现，没有关联
function greet<T extends string>(s: T): void {
  console.log(s.toUpperCase());
}

// ✅ 等价的简洁写法
function greet(s: string): void {
  console.log(s.toUpperCase());
}

// ✅ T 出现两次（参数 + 返回值）— 真正需要泛型
function identity<T>(x: T): T {
  return x;
}

// ✅ T 出现两次（参数 + 返回值用 T 派生）— 真正需要泛型
function box<T>(x: T): { value: T } {
  return { value: x };
}

反模式："给所有函数都加泛型让它看起来更通用"。事实是这样写让代码更难读而非更通用。

5. const type parameter（TS 5.0）：让推断保留字面量

5.0 之前的痛点：传入对象字面量给泛型函数，TS 把它推宽成可变类型，丢失精确性：

typescript · 5.0 之前 
function describe<T>(x: T): T { return x }

const r = describe(["red", "green"]);
//    ^? string[]                  字面量 "red" / "green" 丢失

解决方法是在调用处加 as const：

typescript 
const r = describe(["red", "green"] as const);
//    ^? readonly ["red", "green"]   ✓

但每次调用都加 as const 烦人。5.0 起在类型参数前加 const，让函数自己强制保留字面量推断：

typescript · 5.0+ 
function describe<const T>(x: T): T { return x }

const r = describe(["red", "green"]);
//    ^? readonly ["red", "green"]   ✓ 自动保留字面量
//                                   不用调用方加 as const

// 嵌套对象同样
const cfg = describe({ theme: "dark", size: 12 });
//    ^? { readonly theme: "dark"; readonly size: 12 }

这是写"接受配置对象"类型工具时的杀手锏。Vite / Rollup / Tailwind 等库的配置 helper 大量使用。

6. NoInfer<T>（TS 5.4）：阻止某个位置参与推断

一个常见痛点：函数有多个使用同一个 T 的位置，但有时希望只让某些位置主导推断，其他位置只做检查：

typescript · 5.4 之前的问题 
function createState<T>(initial: T, validate: (value: T) => boolean) {
  return { initial, validate };
}

// 期望：T 由 initial 决定，validate 接受同样类型
const s = createState(42, (x) => x > 0);   // 直觉上 T = number 没问题

// 但
const s = createState("hello", (x) => typeof x === "string");
//                                       ^^ x: string  ✓
// 但如果两边类型不一致：
const s = createState(42 as 42 | 100, (x: number) => x > 0);
// T 被推断为 union "42 | 100 | number" — 太宽，可能不是意图

5.4 引入 NoInfer<T>，让某个位置不参与推断，仅做检查：

typescript · 5.4+ 
function createState<T>(
  initial: T,                              // T 由这里推断
  validate: (value: NoInfer<T>) => boolean  // 这里不参与，只接受 T
) {
  return { initial, validate };
}

// 现在 T 由 initial 主导
const s = createState(42, (x) => x > 0);
//                       ^^ x: number  ✓ 由 initial 决定

典型场景：useState(initial, reducer)、configure(value, options) 这种"主参数 + 副参数"模式。React 团队在 19 之后会逐步用上。

7. 推断的"宽化"行为

字面量传给泛型时，TS 通常把它推断为更宽的类型（除非有 const T 或 const 上下文）：

typescript 
function id<T>(x: T): T { return x }

// const 上下文 → 字面量保留
const a = id("hello");        // a: "hello"

// let 上下文 → 推宽
let b = id("hello");           // b: string

// 显式给 string 类型 → 强制为 string
const c: string = id("hello"); // c: string

// 默认上下文 → 宽化
function f() { return id("hello") }
//                                 推断 T = string
const r = f();                 // r: string

要保留字面量的两种方式：用 as const、或函数定义时用 <const T>。

03 应用场景

场景 A · Generic Repository（数据访问层）

typescript 
interface Entity { id: string }

class Repository<T extends Entity> {
  constructor(
    private collection: T[],
    private idField: keyof T = "id" as keyof T,
  ) {}

  findById(id: string): T | undefined {
    return this.collection.find(x => x[this.idField] === id);
  }

  // 双参数泛型：where 必须是 T 字段的子集
  findWhere<K extends keyof T>(field: K, value: T[K]): T[] {
    return this.collection.filter(x => x[field] === value);
  }

  // 返回部分字段
  select<K extends keyof T>(keys: K[]): Pick<T, K>[] {
    return this.collection.map(x => {
      const out = {} as Pick<T, K>;
      for (const k of keys) out[k] = x[k];
      return out;
    });
  }
}

interface User extends Entity { name: string; age: number }
const repo = new Repository<User>([{ id: "1", name: "Alice", age: 30 }]);

const u = repo.findById("1");                 // User | undefined
const adults = repo.findWhere("age", 30);     // User[]
const partial = repo.select(["name", "age"]); // { name: string; age: number }[]
// 拼写错误立刻报：
repo.findWhere("ageeeeee", 30);                // ❌ '"ageeeeee"' not assignable to 'keyof User'

这是 Drizzle / Prisma / TypeORM 等 ORM 在用户层暴露的核心模式。多个泛型参数 + keyof 约束，把"动态字段名"操作类型化。

场景 B · 类型安全的 fetch wrapper

typescript 
import { z } from "zod";

async function api<T>(
  url: string,
  schema: z.ZodType<T>,
  init?: RequestInit,
): Promise<T> {
  const res = await fetch(url, init);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  const data: unknown = await res.json();
  return schema.parse(data);   // 运行时校验 + 类型推断
}

const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
});

const u = await api("/api/users/1", UserSchema);
//    ^? { id: string; name: string }  从 schema 反推

关键：z.ZodType<T> 把 T 与 schema 的输出类型绑定，调用方只要写 schema，类型自动跟随。这种"schema 驱动类型"是 tRPC / Hono / Drizzle 的核心套路。

场景 C · 类型安全的事件总线

typescript 
// 事件名 → payload 类型的映射表
type EventMap = {
  "user:login": { userId: string; at: Date };
  "user:logout": { userId: string };
  "post:created": { postId: string; authorId: string };
};

class EventBus<Events extends Record<string, unknown>> {
  private listeners: { [K in keyof Events]?: Array<(payload: Events[K]) => void> } = {};

  on<K extends keyof Events>(event: K, handler: (payload: Events[K]) => void): void {
    (this.listeners[event] ??= []).push(handler);
  }

  emit<K extends keyof Events>(event: K, payload: Events[K]): void {
    this.listeners[event]?.forEach(h => h(payload));
  }
}

const bus = new EventBus<EventMap>();

// 触发事件 — payload 类型精确
bus.emit("user:login", { userId: "u_1", at: new Date() });   // ✓
bus.emit("user:login", { userId: "u_1" });                    // ❌ 缺 at

// 监听 — handler 参数类型精确
bus.on("post:created", (p) => console.log(p.postId, p.authorId));
//                          ^? { postId: string; authorId: string }

bus.emit("user:typo", { /* ... */ });   // ❌ 'user:typo' not in keys of EventMap

这是 trpc / typed-emitter / mitt 等库的基础模式。把 event 名映射到 payload 类型，调用 emit / on 时自动联动。

场景 D · 自定义 Hook 配 const T 保留字面量

typescript 
// 接受字面量数组，返回联合类型 + 切换函数
function useToggle<const T extends readonly string[]>(options: T) {
  const [value, setValue] = useState<T[number]>(options[0]);

  const next = () => {
    const idx = options.indexOf(value);
    setValue(options[(idx + 1) % options.length]);
  };

  return [value, next] as const;
}

const [theme, toggle] = useToggle(["light", "dark", "system"]);
//      ^? "light" | "dark" | "system"   字面量自动保留

如果没有 const T，options 推断为 string[]，T[number] 退化为 string，调用方拿到的 theme 就是 string 而非精确联合。

04 典型代码

泛型函数的所有形态

typescript 
// 单参
function id<T>(x: T): T { return x }

// 多参
function pair<A, B>(a: A, b: B): [A, B] { return [a, b] }

// 约束
function lengthy<T extends { length: number }>(x: T): number { return x.length }

// 双参互相约束
function getProp<T, K extends keyof T>(obj: T, key: K): T[K] { return obj[key] }

// 类型参数有默认值（5.0+ 可加约束 + 默认）
function create<T extends string = "default">(): T[] { return [] as T[] }

// const 类型参数（5.0+）
function describe<const T>(x: T): T { return x }

// 显式实参
const a = id<string>("abc");

// 推断实参
const b = id("abc");

Generic interface / type / class

typescript 
// interface
interface Box<T> {
  value: T;
  setValue(v: T): void;
}

// type alias
type Pair<A, B> = [A, B];
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

// class
class Stack<T> {
  private items: T[] = [];
  push(x: T) { this.items.push(x) }
  pop(): T | undefined { return this.items.pop() }
  peek(): T | undefined { return this.items.at(-1) }
}

const s = new Stack<number>();
s.push(1);
s.peek();   // number | undefined

构造器类型（class as type）

typescript 
// 接受一个 class 构造器，返回它的实例
function create<T>(Ctor: { new (): T }): T {
  return new Ctor();
}

class Cat { meow() {} }
class Dog { bark() {} }

const c = create(Cat);   // c: Cat
c.meow();

// 带参数的构造器
function createWith<T, A extends unknown[]>(Ctor: { new (...args: A): T }, ...args: A): T {
  return new Ctor(...args);
}

class User { constructor(public name: string, public age: number) {} }
const u = createWith(User, "Alice", 30);   // u: User

泛型默认值

typescript 
// 没指定时使用默认
type ApiResponse<T = unknown> = { ok: boolean; data: T };

const r1: ApiResponse = { ok: true, data: { x: 1 } };
//        ^? ApiResponse<unknown>
const r2: ApiResponse<string> = { ok: true, data: "hello" };

// 默认值与约束组合
type WithDefault<T extends string = "default"> = { value: T };
const a: WithDefault = { value: "default" };       // ✓
// const b: WithDefault = { value: "other" };      // ❌ 'other' not assignable to 'default'
const c: WithDefault<"any-string"> = { value: "any-string" };  // ✓ 显式实参覆盖

TS 5.0 const T 与 5.4 NoInfer 实战

typescript 
// const T：保留字面量
function makeRoutes<const T extends Record<string, string>>(routes: T): T {
  return routes;
}
const r = makeRoutes({ home: "/", about: "/about" });
//    ^? { readonly home: "/"; readonly about: "/about" }

// NoInfer：让某位置不参与推断
declare function useAnimation<T extends string>(
  initial: T,
  states: NoInfer<T>[],
): T;

// initial 主导推断
const a = useAnimation("idle", ["idle", "running", "done"]);
//    ^? "idle"   仅由 initial 决定
//                state 数组只能含 "idle"（因为 T 已锁为 "idle"）

05 常见陷阱

陷阱 1 · 函数体内偷偷把 T 变成 any

常见反模式：clone 函数用 JSON 实现，类型上看起来通用，实际 T 内部被破坏：

typescript 
function clone<T>(x: T): T {
  return JSON.parse(JSON.stringify(x));   // 运行时丢失 Date / Map / Set / RegExp / function 等
}

const d = new Date();
const d2 = clone(d);
//    ^? Date     类型上仍是 Date — 但运行时已经是 string！
d2.getTime();   // 运行时 TypeError

根因：JSON 序列化 Date 变 string、丢 function、Map 变 {}。但类型签名仍是 T → T，编译期完全无感。

规避：用 structuredClone()（运行时支持的深拷贝）或 lodash cloneDeep，并在签名上注明限制。

陷阱 2 · 过度泛型化让代码变难读
typescript 
// ❌ 看似通用，实际样板
function logAll<T extends unknown[]>(items: T): void {
  for (const item of items) console.log(item);
}

// ✅ 简洁等价
function logAll(items: unknown[]): void {
  for (const item of items) console.log(item);
}

规避：每写一个泛型问"它出现几次？两次以上才需要"。一次出现的几乎都是冗余。

陷阱 3 · 推断失败时报"Could not infer T"

如果 T 在签名里只出现在一个不能反向推断的位置（如返回值），TS 不知道 T 应该是什么：

typescript 
function fetchData<T>(): Promise<T> {
  return fetch("/api").then(r => r.json());
}

const data = await fetchData();
//    ^? unknown   T 没法推 — 退化成 unknown

// 必须显式指定
const data = await fetchData<User>();
//    ^? User

规避：返回值含 T 但参数没 T 的函数，调用方必须显式实参。或者改设计，让参数也含 T（如传入 schema）。

陷阱 4 · 默认值与约束的顺序错
typescript 
// ❌ 默认值先于约束
type Bad<T = number, U extends T> = ...;
//             ^^^^^^^^^^^^^^^^^ Required type parameters may not follow optional type parameters

// ✅ 约束先，默认值后
type Good<T extends string = "default", U extends T = T> = ...;

规则：与函数参数同理，必填的类型参数在前，带默认的在后。

陷阱 5 · const T 与 readonly 的"传染"

<const T> 把传入对象推断成深 readonly，导致"想传给非 readonly 参数"的位置报错：

typescript 
function describe<const T>(x: T): T { return x }

const arr = describe([1, 2, 3]);
//    ^? readonly [1, 2, 3]

function takeMutable(xs: number[]) {}
takeMutable(arr);   // ❌ 'readonly [1, 2, 3]' is 'readonly' and cannot be assigned to mutable type

规避：const T 仅在需要保留字面量时使用。如果后续要交给可变参数，要么不用 const T，要么显式 [...arr] 复制成可变。

陷阱 6 · 泛型用在错误的位置（应该约束的地方放成参数）
typescript 
// ❌ T 用作 union 表达
function setStatus<T extends "active" | "inactive">(s: T): void {
  console.log(s);
}

// ✅ 直接 union
function setStatus(s: "active" | "inactive"): void {
  console.log(s);
}

// ⚠️ 真要泛型，是因为想保留字面量精确
function record<const T extends string>(s: T): { value: T } {
  return { value: s };
}
const r = record("active");
//    ^? { value: "active" }   字面量保留

规则：泛型让"使用位置的具体类型"被 captured。只要不需要 capture，就用 union / 具体类型代替。
