01 · The Basics 与类型系统本质

TypeScript 不是新语言，是 JavaScript 之上的「开发期约束系统」。所有类型注解在编译后完全消失，运行时是纯 JS。理解这一条之前，你写的 TS 都只是「带帮助的 JS」；理解之后，你才能解释为什么 tsc 通过的代码线上仍会爆炸、为什么 any 的传染如此致命、为什么 strict 是档位而非开关。

阅读时长 ~25 分钟 前置 JavaScript 基础语法 对应 typescriptlang.org Handbook · The Basics

01 速览

三句话定义 TypeScript：

  1. 类型注解只在编译期存在。编译后类型完全擦除，产物是合法 JavaScript，运行时无法 instanceof MyInterface。
  2. 类型与值是两个独立命名空间。同一个名字可同时是类型和值；class 同时占两边，interface 只占类型，const 只占值。
  3. 类型系统是结构化的，不是名义化的。两个 interface 字段一样就互相赋值兼容，不靠"名字相同"。

最小例子：

typescript · 编写时 
interface User {
  id: string;
  name: string;
}

function greet(u: User): string {
  return `Hello, ${u.name}`;
}

const alice: User = { id: "u_1", name: "Alice" };
console.log(greet(alice));
javascript · 编译产物（target: ES2022） 
function greet(u) {
  return `Hello, ${u.name}`;
}

const alice = { id: "u_1", name: "Alice" };
console.log(greet(alice));

interface 完全消失，参数注解 : User 完全消失，变量注解 : User 完全消失。运行时世界没有 TypeScript 的任何痕迹 —— 这是后续一切讨论的起点。

02 底层逻辑

要回答「编译通过为什么还会出错」，必须先理解 TypeScript 在编译期做了什么、不做什么。

tsc 三阶段：parse → check → emit

一次 tsc 调用在内部分三个阶段：

  1. Parse（解析） —— 把 .ts 源文件变成 AST（抽象语法树）。这一阶段会处理 TS 特有语法（类型注解、interface、enum、namespace、装饰器等），把它们存入 AST 节点。
  2. Check（类型检查） —— 在 AST 上跑类型推断 + 检查规则。这里是 TS 唯一的"魔法"所在：判断每处赋值/调用/返回是否符合声明的类型。所有报错都来自这一阶段。
  3. Emit（输出） —— 删除所有类型注解，把 AST 重新输出成 JavaScript。target 配置决定语法降级到哪个 ES 版本。注意：默认即使有类型错误，emit 仍会执行（除非配 noEmitOnError: true）。

这种分离意味着类型检查和代码生成是解耦的。babel / swc / esbuild 等工具只做 emit（删类型 + 语法降级），不做 check —— 这就是为什么 Vite / Next.js 等项目会用 tsc --noEmit 单独跑类型检查。

类型擦除：所有类型在运行时不存在

TS 的核心工程决策是不引入运行时类型。同一个 interface 在编译后完全消失，所以：

typescript · 反例 
interface User {
  id: string;
}
interface Admin {
  id: string;
  permissions: string[];
}

function check(x: User | Admin) {
  if (x instanceof Admin) {
    // ❌ 编译错误：'Admin' only refers to a type, but is being used as a value here
  }
}

运行时根本没有 Admin 这个值，instanceof 是 ES 运算符，需要左侧的类型在运行时存在。要想区分 User 和 Admin，必须用运行时可见的属性判断 —— 这就是后续 03 章 discriminated union 的来源。

类型擦除带来的另外几个直接后果：

  • 不能在运行时读取「这个变量的 TS 类型是什么」 —— 类型不是值，没法 typeof x 拿到 "User"
  • 泛型 T 在运行时不存在 —— function f<T>() { new T() } 不可能（T 没法构造）
  • 反射框架（NestJS、TypeORM）需要靠 reflect-metadata + emitDecoratorMetadata 把部分类型信息保留到运行时（见 14 章）

类型与值是两个独立命名空间

TS 把所有标识符分到两个命名空间：类型空间（编译期）和 值空间（运行时）。一个名字可以同时存在于两个空间，互不冲突。

声明形式 类型空间 值空间
const x = 1—✓ x
type T = string✓ T—
interface I {}✓ I—
class C {}✓ C（实例类型）✓ C（构造器）
function f() {}✓ typeof f✓ f
enum E { A }✓ E✓ E
namespace N {}✓ N✓ N（如果含值）

同名不冲突的能力是 declaration merging（15 章）和 parameter properties（06 章）的基础。也是为什么 typeof x 在类型上下文里可以用 —— 它是从值空间取出类型的桥（08 章）。

这个二分还衍生出 TS 4.5 的 import type / inline import { type X } 语法：

typescript 
// 仅在类型空间引入，编译产物完全消失
import type { User } from "./types";

// inline 标注：fn 是值，type X 是类型
import { fn, type X } from "./mod";

import type 的价值：避免循环依赖（A 引用 B 类型，B 运行时引用 A 不会再循环加载）、利于 tree-shaking、配合 verbatimModuleSyntax（19 章）让编译产物更可预测。

结构化类型（structural typing）

TS 类型系统是结构化的：看形状不看名字。这是和 Java / C# 等名义化（nominal）类型系统最大的认知差异。

typescript 
interface Point2D { x: number; y: number }
interface Coord    { x: number; y: number }

const p: Point2D = { x: 1, y: 2 };
const c: Coord = p; // ✓ 通过 — 字段一样就兼容

Java 中 Point2D 和 Coord 是两个独立类型，互不赋值兼容。TS 中只要字段形状匹配就互通 —— 因为 JS 本来就是基于「鸭子类型」的运行时。

结构化类型的好处：写 utility 函数不需要类型继承图。function distance(p: { x: number; y: number }) 接受任何「碰巧有 x 和 y」的对象，无论它叫什么名字、来自哪个库。代价：有时需要 branded type（12 章）来模拟名义化区分（如 UserId vs OrderId）。

编译通过 ≠ 运行时安全

下面这些情况 TS 都会"放行"，但运行时可能炸：

typescript · 类型系统的盲区 
// 1️⃣ as 强转 — TS 完全相信你
const x = "abc" as unknown as number;
x.toFixed(2); // 编译过 → 运行时 TypeError

// 2️⃣ any 注入
function parseConfig(s: string): any {
  return JSON.parse(s);
}
const cfg = parseConfig('{"port": "not a number"}');
const p: number = cfg.port; // 编译过 → 实际是 string

// 3️⃣ 非空断言 !
function findUser(id: string): User | undefined { /* ... */ }
const u = findUser("xxx")!; // 编译过 → 找不到时 u 实际是 undefined
u.name; // 运行时 TypeError

// 4️⃣ 第三方 .d.ts 写错
// 库说自己的函数返回 string，实际返回 null
declare function getName(): string;
const name = getName();
name.toUpperCase(); // 编译过 → 实际 null.toUpperCase() 炸

// 5️⃣ 外部数据未校验
const user: User = await fetch("/api/user").then(r => r.json()); // 编译过 → 服务端字段名变了就炸

类型系统只在「内部代码」可信。所有信任边界（fetch / JSON.parse / process.env / URL params / 用户输入 / 第三方 SDK）必须在运行时再验证一次：

  • 用 zod / valibot / arktype 做 schema 校验
  • 用 z.infer<typeof Schema> 把 schema 反向变成 TS 类型，保持单一真源

strictness 是档位，不是开关

"strict": true 不是一个独立检查，而是把一组检查全部打开的总开关。它包含：

flag作用关掉的代价
noImplicitAny禁止隐式 any到处都是 any 黑洞
strictNullChecksnull / undefined 必须显式放弃 90% null 安全收益
strictFunctionTypes函数参数严格逆变检查method bivariance 漏掉子类型违反
strictBindCallApplybind/call/apply 检查参数可以传任意参数
strictPropertyInitialization类字段必须初始化或显式 ?构造器后字段可能 undefined
alwaysStrictemit "use strict"—
useUnknownInCatchVariablescatch (e) 推 unknown 而非 anye.foo 直接通过
noImplicitThis禁止 this 隐式 anythis 传染 any

外加（不在 strict 里但常配套打开的）：

  • noUncheckedIndexedAccess — arr[i] 推断为 T | undefined，反映 JS 真实运行时行为
  • exactOptionalPropertyTypes — { foo?: string } 不允许显式赋 undefined
  • noImplicitReturns / noFallthroughCasesInSwitch

新项目应直接 strict: true，按需单独关掉暂不能满足的项；老项目从 noImplicitAny 起步，逐项打开。每打开一个 flag 都会暴露代码里的旧债 —— 这正是它的价值。

target 与 lib 是两件事

这两个常被混淆的配置项含义不同：

  • target 决定语法降级到哪个 ES 版本。如 target: "ES5" 会把 async / class / let / 箭头函数 / 模板字面量都降级。
  • lib 决定 TS 知道哪些标准库 API 存在。如 lib: ["ES2020"] 让 String.prototype.matchAll 可被识别。

关键：target 不会自动注入 polyfill。target: "ES5" + lib: ["ES2020"] 时：

  • 语法被降级到 ES5（OK，老浏览器能解析）
  • 但 Promise / Map / fetch 等 API 在 ES5 浏览器中不存在
  • 运行时 Promise is not defined 炸

解法：要么把 polyfill（core-js / regenerator-runtime）显式注入打包，要么把 target 拉高到目标浏览器实际支持的版本。TS 只管语法降级，运行时 API 兼容由 bundler / polyfill 负责。

03 应用场景

场景 A · Express / Next.js handler 的请求体类型 + zod 兜底

HTTP handler 是典型的「类型系统盲区」—— 请求体来自外部，TS 无法担保字段。正确模式是用 zod schema 既做类型又做运行时校验：

typescript 
import { z } from "zod";

const CreateUserSchema = z.object({
  email: z.string().email(),
  age: z.number().int().min(0),
});

// 类型从 schema 反向推导，单一真源
type CreateUserInput = z.infer<typeof CreateUserSchema>;

export async function POST(req: Request) {
  const body = await req.json(); // body: any，运行时不可信
  const result = CreateUserSchema.safeParse(body);
  if (!result.success) {
    return Response.json({ error: result.error.issues }, { status: 400 });
  }
  // result.data: CreateUserInput，已校验
  await db.users.insert(result.data);
  return Response.json({ ok: true });
}

schema 既约束运行时（拒绝非法 body），又约束编译时（result.data 自动有正确类型）。这是 tRPC / Hono / Drizzle 等"端到端类型安全"项目的核心套路。

场景 B · 老 JS 项目渐进式迁移

把现有 JS 项目改 TS 不必一次到底。tsconfig.json 里允许 JS / TS 共存，并给 JS 文件加类型检查：

json · tsconfig.json 
{
  "compilerOptions": {
    "allowJs": true,        // 允许 .js 文件参与编译
    "checkJs": false,       // 默认不检查所有 JS（避免噪声）
    "noImplicitAny": true,
    "strictNullChecks": true,
    "outDir": "dist"
  },
  "include": ["src/**/*"]
}

然后在愿意切的单个 JS 文件里加 // @ts-check，让该文件接受类型检查；用 JSDoc 注释提供类型：

javascript · 加 @ts-check 的 JS 
// @ts-check

/** @typedef {{ id: string, name: string }} User */

/** @param {User[]} users */
function listIds(users) {
  return users.map(u => u.id);
}

策略：第三方依赖最复杂的入口先改 .ts，纯工具函数先加 @ts-check，等覆盖率上来再统一改后缀名 + 提高 strict 档位。

场景 C · CI 中的类型守门员

生产编译用 SWC / esbuild（快），类型检查独立跑：

json · package.json 
{
  "scripts": {
    "build": "next build",            // bundler 自己 transpile
    "typecheck": "tsc --noEmit",      // 仅类型检查，不输出文件
    "ci": "bun typecheck && bun build"
  }
}

--noEmit 让 tsc 只跑 check 阶段、跳过 emit。CI 在编译之前先跑 typecheck 拦住类型错误，编译速度则由 bundler 决定。这是 Vite / Next.js / Remix 等项目的标准做法。

04 典型代码

类型 vs 值：同名不冲突

typescript 
// User 既是类型又是值
class User {
  constructor(public name: string) {}
}

// 用作类型（编译期）
function greet(u: User) { return `Hi ${u.name}`; }

// 用作值（运行时）
const alice = new User("Alice");

// 从值空间反向取类型
type UserConstructor = typeof User;        // new (name: string) => User
type UserInstance = InstanceType<typeof User>; // User

// 反例：interface 只在类型空间
interface Order { id: string }
const o: Order = { id: "1" };  // ✓ 类型用法
// const x = Order;             // ❌ 'Order' only refers to a type

编译产物对照

理解类型擦除最好的方式是看产物。下面是同一份 TS 的不同 target 输出：

typescript · 源 
interface Result<T> {
  ok: boolean;
  value: T;
}

const success = <T>(value: T): Result<T> => ({ ok: true, value });
const r = success(42);
javascript · target: ES2022 
const success = (value) => ({ ok: true, value });
const r = success(42);
javascript · target: ES5（语法降级） 
var success = function (value) { return { ok: true, value: value }; };
var r = success(42);

注意：

  • interface Result 完全不见了 —— 类型擦除
  • 泛型 <T> 完全不见了 —— 没有运行时实参
  • ES5 target 把 const / 箭头函数 / 简写属性都降级了
  • 但没有引入 polyfill —— 如果代码里用 Promise，ES5 输出仍然是 new Promise（运行时炸）

noEmitOnError 的取舍

json 
{
  "compilerOptions": {
    "noEmitOnError": true   // 默认 false：有错误仍 emit
  }
}

打开后 tsc 在有任何类型错误时不输出 .js 文件。生产构建建议打开（避免错误代码被部署）；本地开发可以关闭（边写边运行调试更顺）。Vite / Next.js 等 bundler 自己 transpile 时这个选项不生效，要用 tsc --noEmit 单独把关。

05 常见陷阱

陷阱 1 · any 的传染性

一处 any 让整条调用链失去类型保护：

typescript 
function fetchUser(id: string): any {       // 偷懒返回 any
  return fetch(`/api/users/${id}`).then(r => r.json());
}

const u = await fetchUser("123");
const name = u.profile.fullname.toUppercase();   // 拼写错也不报，运行时 TypeError

根因：any 关闭了所有类型检查。u.x.y.z 任意访问都通过。

规避：用 unknown 替代 any，强制 narrow 后再用；或用 zod 校验。"noImplicitAny": true 必须开。

陷阱 2 · target ES5 + 现代 API 的运行时炸点

target: "ES5" 配 lib: ["ES2020"]：编译过，老浏览器运行时 Promise is not defined。

根因：target 只降语法，不引入 polyfill；lib 只声明 API 存在让 TS 类型识别。两者职责分离 —— 真实兼容性由 bundler / polyfill 负责。

规避：要么用 Babel @babel/preset-env + core-js 自动注入 polyfill；要么把 target 拉到目标浏览器真实能跑的版本。Vite / Next.js 通常默认 ES2017+，几乎不用担心。

陷阱 3 · @ts-ignore 的"屏蔽腐烂"

// @ts-ignore 屏蔽下一行的类型错误，但修复后仍残留，看不出来。

typescript 
// ❌ @ts-ignore：错误修了仍残留
// @ts-ignore
const x: string = 123;

// ✅ @ts-expect-error：错误消失会主动报警
// @ts-expect-error - intentionally invalid for legacy reasons
const y: string = 123;

规避：永远用 @ts-expect-error 加注释说明原因。修复后那行变成无意义注释，TS 立刻提醒你删除。

陷阱 4 · var 在 for 循环里捕获最终值

这是 JS 历史包袱，TS 不会替你修复，但 strict 也不会拦住：

typescript 
// ❌ var：所有回调都打印 5
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 10);
}

// ✅ let：每次循环新建绑定，正确打印 0-4
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 10);
}

根因：var 是函数作用域，循环里所有迭代共享同一个 i；let 是块作用域，每次迭代新建。

规避：永远用 const / let，不写 var。可加 ESLint 规则 no-var 强制。

陷阱 5 · 第三方 .d.ts 与运行时不匹配

从 @types/foo 引入的类型可能滞后于库的真实版本：

  • 库升级了返回类型，@types 还没跟
  • 库行为改了但 .d.ts 没改
  • 有些库自带类型 + 又有 @types/foo，互相覆盖

规避：

  • 优先选自带 .d.ts 的库（package.json 有 types 字段）
  • 升级库时同步检查 @types/foo 版本
  • 关键边界（API 客户端、ORM）用 zod / schema 二次验证，别完全信赖 .d.ts
