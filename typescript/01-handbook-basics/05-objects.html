05 · Object Types

对象建模是 TS 的中心。本章覆盖 readonly / optional 修饰符、index signature、excess property check、interface 与 intersection 的差异、tuple 全谱、ReadonlyArray、as const 推断。重点解答：readonly 是不是真的不能改、index signature 为什么会让其他字段报错、tuple 与数组运行时是同一回事但类型上有何不同、interface extends 和 & 在冲突时的根本差异。

阅读时长 ~28 分钟 前置 02 Everyday Types 对应 typescriptlang.org Handbook · Object Types

01 速览

描述对象的 4 种语法：

typescript 
// 形式 A：匿名对象类型
function f(p: { x: number; y: number }) {}

// 形式 B：interface
interface Point { x: number; y: number }

// 形式 C：type alias
type Point = { x: number; y: number };

// 形式 D：从值反推（typeof）
const origin = { x: 0, y: 0 };
type Origin = typeof origin;

叠加四种修饰：

  • ?：optional，字段可不传 / 可为 undefined
  • readonly：意图标注，禁止赋值（编译期）
  • index signature [k: string]: T：动态键
  • 泛型 Box<T>：参数化形状

02 底层逻辑

1. readonly 是意图标注，不是运行时保护

readonly 在编译期阻止赋值：

typescript 
interface Point {
  readonly x: number;
  readonly y: number;
}
const p: Point = { x: 1, y: 2 };
p.x = 3;   // ❌ Cannot assign to 'x' because it is a read-only property

但有几条容易被忽略：

  • 编译产物没有 readonly：JS 中 p.x = 3 仍然可以执行 —— readonly 是给读者的契约，不是给 V8 的指令
  • readonly 是浅 readonly：嵌套对象内层不 readonly
  • 能用 as 强转绕过：(p as { x: number }).x = 3 编译过
  • 结构兼容时 readonly 不算"差异"：可变对象可以赋给 readonly 形状（反向不行）
typescript · 兼容性方向 
interface Mutable { x: number }
interface ReadOnly { readonly x: number }

const m: Mutable = { x: 1 };
const r: ReadOnly = m;   // ✓ 可变 → readonly 安全（"我承诺不改"）
m.x = 5;                  // 但 m 仍可变 — r.x 也变成 5！

const r2: ReadOnly = { x: 1 };
const m2: Mutable = r2;  // ✓ readonly → 可变 也通过（结构上 readonly 不算少东西）
m2.x = 5;                 // 编译过 — 但实际上"突破"了 readonly

这是结构化类型的代价：readonly 不构成"额外约束"，纯粹是意图。要真正不可变就用 Object.freeze() 或 immer。

2. Index signature 限制了所有显式字段必须兼容

[key: string]: T 表达"任何 string key 的值都是 T"。这是个全局约束，因此显式声明的字段类型必须是 T 的子类型：

typescript 
interface Cache {
  [key: string]: number;
  size: number;       // ✓ size 是 number，符合
  // version: string;  // ❌ Property 'version' of type 'string' is not assignable to 'string' index type 'number'
}

为什么这样设计？因为 obj["size"] 必须既符合 size: number，又符合 [key: string]: number。两者都成立的唯一方式就是显式字段是 index signature 的子类型。

解法：要"既能动态存任意字段，又能给某个字段精确类型"，把 index signature 类型放宽：

typescript 
interface Cache {
  [key: string]: number | string;
  size: number;       // ✓
  version: string;    // ✓
}

// 或者用 union：
interface Cache {
  [key: string]: unknown;
  size: number;
  version: string;
}

注意 [key: number]: T 也合法，但 JS 的对象 key 始终字符串化，所以 number 索引等价于 string 索引"中能转 number 的子集"。在 TS 里 keyof T（08 章）面对 number index 时返回 number。

3. interface extends vs type intersection 的冲突处理

这两个工具看起来都能"组合类型"，但在字段冲突时行为完全不同：

typescript 
interface A { x: string }
interface B { x: number }

// ❌ extends — 立刻报错
interface C extends A, B {}
//        ^^^^^^^^^^^^^^^^^^^^
//        Interface 'C' cannot simultaneously extend types 'A' and 'B'.

// ⚠️ intersection — 静默通过，但 x 的类型变成 string & number = never
type D = A & B;
const d: D = { x: 1 as never };  // 几乎不可用

extends 在定义时检查冲突；intersection 在使用时把冲突字段算成 never，使用时才暴露错误。extends 报错信息更友好、调试更早。

实用建议：组合"已知不冲突的对象形状"用 interface extends；组合 union / utility type 等无法 extends 的位置才用 &。

4. excess property check 是给"立即字面量"的特别保护

02 章已经讲过这个，这里补充更多细节。规则：对象字面量直接传给某个类型时，多余字段会被特别检查。先入变量绕过这层保护。

typescript 
interface Options { width: number }

function render(o: Options) {}

// ❌ 直接传字面量 — excess property check 触发
render({ width: 100, height: 200 });
//                   ^^^^^^^^^^^ Object literal may only specify known properties

// ✅ 三种"故意宽松"的方式
const o1 = { width: 100, height: 200 };
render(o1);                                    // 先入变量

render({ width: 100, height: 200 } as Options); // 断言

interface OptionsAny {
  width: number;
  [key: string]: unknown;                       // index signature 显式表达"开放"
}
function render2(o: OptionsAny) {}
render2({ width: 100, height: 200 });           // ✓

为什么专门对字面量做这个检查？官方原因：直接传字面量最像"我以为这个字段叫 padding 但其实是 spacing"的拼写错误；先赋给变量则被解读为开发者有意准备超集。

5. Tuple 是"长度和位置都被锁住的数组"

运行时层面 tuple 就是普通数组（typeof t === "object" && Array.isArray(t)）。TS 层面给它加了"长度固定 + 每个位置类型固定"的约束：

typescript 
type Pair = [string, number];
const p: Pair = ["age", 30];

p[0].toUpperCase();   // ✓ p[0] 类型是 string
p[1].toFixed(2);      // ✓ p[1] 类型是 number
p[2];                  // ❌ Tuple type 'Pair' of length '2' has no element at index '2'

// 字段名（labeled tuple，TS 4.0+）— 仅文档作用，运行时无差别
type Range = [start: number, end: number];

// 可选 element
type T1 = [string, number?];   // 第二个元素可省略
const t1: T1 = ["a"];           // ✓

// rest element（TS 4.0+）
type T2 = [string, ...number[]];
const t2: T2 = ["a", 1, 2, 3];  // ✓

// 中间 rest（TS 4.2+）
type T3 = [string, ...boolean[], number];
const t3: T3 = ["a", true, false, 99];

// readonly tuple
type ReadonlyPair = readonly [string, number];
const rp: ReadonlyPair = ["a", 1];
// rp[0] = "b";   // ❌

经典应用：React useState 返回 [T, Dispatch<SetStateAction<T>>]，调用方一行解构两个变量。

6. as const 触发字面量 + readonly tuple 推断

普通数组字面量推断为可变数组，as const 让它变成 readonly tuple 且每个元素保留 literal 类型：

typescript 
const a = ["red", "green", "blue"];
//    ^? string[]                     可变数组，元素类型 string

const b = ["red", "green", "blue"] as const;
//    ^? readonly ["red", "green", "blue"]   readonly tuple + literal

// 能从中精确取出 union
type Color = (typeof b)[number];
//   ^? "red" | "green" | "blue"

这是把"运行时常量数组"提升到精确类型的标准模式。配合 typeof + [number] 索引能拿到元素 union。

7. 数组的三种写法都等价

写法语义建议
T[]可变数组✓ 推荐
Array<T>可变数组泛型嵌套时可读性更好
readonly T[]只读数组✓ 推荐（写法简洁）
ReadonlyArray<T>只读数组同上，泛型嵌套时偶用

ESLint 的 @typescript-eslint/array-type 规则可强制风格统一。

03 应用场景

场景 A · React useState 与 readonly tuple

typescript 
// useState 的简化签名（来自 @types/react）
declare function useState<S>(
  initial: S | (() => S),
): [S, Dispatch<SetStateAction<S>>];

// 调用时一行解构
const [count, setCount] = useState(0);
//     ^? number   ^? Dispatch<SetStateAction<number>>

// 自定义 hook 模仿同样模式
function useToggle(initial = false): [boolean, () => void] {
  const [v, setV] = useState(initial);
  const toggle = useCallback(() => setV(x => !x), []);
  return [v, toggle];
}

const [isOpen, toggleOpen] = useToggle();

场景 B · 配置对象 as const + 派生类型

typescript 
const CONFIG = {
  apiUrl: "https://api.example.com",
  timeout: 5000,
  features: ["search", "export", "share"],
  theme: { primary: "#3178c6", secondary: "#f5f5f5" },
} as const;

// 全部字段都 readonly + literal
type ApiUrl = typeof CONFIG.apiUrl;          // "https://api.example.com"
type Feature = (typeof CONFIG.features)[number]; // "search" | "export" | "share"
type ThemeKey = keyof typeof CONFIG.theme;       // "primary" | "secondary"

function setTheme(key: ThemeKey) { /* ... */ }
setTheme("primary");   // ✓
setTheme("tertiary");  // ❌

这是把"配置当类型源"的标准做法。用一份 const 同时驱动运行时值和类型，零重复。

场景 C · ORM Model + Partial / Pick

typescript 
interface User {
  id: string;
  email: string;
  name: string;
  passwordHash: string;
  createdAt: Date;
  lastLogin: Date | null;
}

// 创建时不需要 id / createdAt（数据库生成）和 lastLogin
type CreateUserInput = Omit<User, "id" | "createdAt" | "lastLogin">;

// 公共字段（不能暴露 passwordHash）
type PublicUser = Omit<User, "passwordHash">;

// 更新时所有字段可选
type UpdateUserInput = Partial<Pick<User, "email" | "name">>;

async function createUser(input: CreateUserInput): Promise<PublicUser> { /* ... */ }
async function updateUser(id: string, input: UpdateUserInput): Promise<PublicUser> { /* ... */ }

Partial / Pick / Omit 都是 11 章会展开的 utility types。在 ORM / DTO 转换里用得最多。

场景 D · Express middleware augmentation

给 Express Request 加业务字段是 declaration merging 最经典的应用：

typescript 
// types/express.d.ts
import "express";

declare module "express-serve-static-core" {
  interface Request {
    user?: { id: string; email: string };  // auth middleware 注入
    requestId: string;                      // tracing middleware 注入
  }
}

// 之后所有 Express handler
app.get("/me", (req, res) => {
  if (!req.user) return res.status(401).end();
  res.json({ id: req.user.id, email: req.user.email });
});

interface 的 declaration merging 让用户能"打开"第三方库的类型。type alias 永远做不到。详见 15 章。

04 典型代码

readonly 配合 Object.freeze 实现真不可变

typescript 
function deepFreeze<T>(o: T): Readonly<T> {
  Object.freeze(o);
  Object.values(o as object).forEach(v => {
    if (v && typeof v === "object" && !Object.isFrozen(v)) deepFreeze(v);
  });
  return o;
}

const config = deepFreeze({
  api: { url: "https://...", timeout: 5000 },
  features: ["a", "b"] as const,
});

// 编译期：readonly 阻止赋值
// config.api.url = "x";   // ❌

// 运行时：frozen 阻止修改（strict 模式抛 TypeError，否则静默失败）
// config.api.url = "x";   // 运行时也阻止

Generic object types：Box / Container 模式

typescript 
interface Box<T> {
  contents: T;
}

const b1: Box<string> = { contents: "hello" };
const b2: Box<number> = { contents: 42 };

// 嵌套
type Tree<T> = {
  value: T;
  children: Tree<T>[];
};

const t: Tree<string> = {
  value: "root",
  children: [
    { value: "a", children: [] },
    { value: "b", children: [{ value: "b1", children: [] }] },
  ],
};

Tuple 完整谱

typescript 
// 固定长度
type T1 = [string, number];

// labeled（仅文档）
type T2 = [name: string, age: number];

// 可选末尾元素
type T3 = [string, number?];           // 长度 1 或 2
const a: T3 = ["alice"];                 // ✓
const b: T3 = ["alice", 30];             // ✓

// 末尾 rest
type Args = [string, ...number[]];
const c: Args = ["log", 1, 2, 3];

// 头部 rest（TS 4.2+）
type T4 = [...string[], number];
const d: T4 = ["a", "b", 99];

// 中间 rest
type T5 = [string, ...boolean[], number];
const e: T5 = ["start", true, false, 99];

// 用 tuple 表达可变函数签名
function f(...args: [string, number] | [boolean]) {}
f("a", 1);   // ✓
f(true);     // ✓

Index signature 与具体字段共存

typescript 
// HTTP headers：name 和 set-cookie 是已知字段，其余动态
interface Headers {
  [key: string]: string | string[] | undefined;
  "content-type"?: string;
  "set-cookie"?: string[];
}

const h: Headers = {
  "content-type": "application/json",
  "set-cookie": ["a=1", "b=2"],
  "x-custom": "value",
};

变型方向陷阱：readonly array 与 mutable array

typescript 
// ✓ mutable → readonly 安全（"我承诺不改"）
function readOnly(xs: readonly number[]) {}
const m: number[] = [1, 2, 3];
readOnly(m);

// ❌ readonly → mutable 不安全
function mutable(xs: number[]) {
  xs.push(4);   // 调用方传 readonly 数组就崩
}
const r: readonly number[] = [1, 2, 3];
mutable(r);    // ❌ The type 'readonly number[]' is 'readonly' and cannot be assigned to mutable type 'number[]'

05 常见陷阱

陷阱 1 · readonly 不递归（浅 readonly）

readonly 只锁第一层，嵌套对象内层可改：

typescript 
interface User {
  readonly id: string;
  readonly profile: { name: string };   // profile 不可换，但 profile.name 可改
}
const u: User = { id: "1", profile: { name: "Alice" } };
// u.id = "2";                // ❌ 拦住
u.profile.name = "Bob";       // ✓ 通过

// ✅ 要深 readonly：自定义 DeepReadonly（10 章会用 mapped type 实现）
type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
};
type DeepUser = DeepReadonly<User>;
const u2: DeepUser = { id: "1", profile: { name: "Alice" } };
// u2.profile.name = "Bob";   // ❌
陷阱 2 · index signature 让显式字段类型被强制窄化
typescript 
// ❌ 直觉错误
interface Cache {
  [key: string]: number;
  size: number;        // ✓
  version: string;     // ❌ 'string' is not assignable to index signature type 'number'
}

规避：放宽 index signature 类型，或者根本不用 index signature 用 Record：

typescript 
// 方案 A：放宽 index 类型
interface Cache {
  [key: string]: number | string;
  size: number;
  version: string;
}

// 方案 B：拆分两个对象
interface Cache {
  size: number;
  version: string;
  data: Record<string, number>;
}
陷阱 3 · intersection 冲突字段静默变 never
typescript 
// 两个 type 都声明了 status，但类型不同
type WithStatus1 = { status: "pending" | "done" };
type WithStatus2 = { status: number };

// ⚠️ 静默通过：status 变成 ("pending" | "done") & number = never
type Combined = WithStatus1 & WithStatus2;

const c: Combined = { status: ??? };
//                              ^^^ 永远找不到能赋的值，因为 status: never

// ✅ 等价的 interface extends 立刻报错
interface I1 { status: "pending" | "done" }
interface I2 { status: number }
interface ICombined extends I1, I2 {}
//        ^^^^^^^^^^^^^^^^^^^^^^^^^^^ Interface 'ICombined' cannot simultaneously extend types 'I1' and 'I2'

规避：组合明确不冲突的形状用 interface extends，错误能立刻被发现。

陷阱 4 · tuple 仍然可以 .push()（直到 TS 4.0 才收紧）
typescript 
type Pair = [string, number];
const p: Pair = ["a", 1];

// TS 4.0+：编译期拦住
p.push("b");
//  ^^^^ Tuple type 'Pair' of length '2' has no element at index '2'

// 但仍能通过类型逃逸绕过
(p as unknown as unknown[]).push("b");

// 运行时：p 实际是数组，长度变 3
console.log(p.length);   // 3

规避：tuple 仅用于"我担保不变长"的场景；长度可变就用 readonly T[] 或普通数组。

陷阱 5 · Object.keys / for...in 返回 string 而非 keyof T
typescript 
interface User { id: string; name: string }
const u: User = { id: "1", name: "Alice" };

Object.keys(u).forEach(k => {
  //                    ^? string  不是 "id" | "name"
  // u[k];   // ❌ 'string' can't be used to index type 'User'
});

// 为什么？因为 u 可能"实际有"超出声明的字段（结构化类型 + excess property）
// TS 不能担保 keys 仅限 keyof User

// ✅ 解法 1：自己断言
(Object.keys(u) as (keyof User)[]).forEach(k => u[k]);

// ✅ 解法 2：写 helper
function keysOf<T extends object>(o: T): (keyof T)[] {
  return Object.keys(o) as (keyof T)[];
}
keysOf(u).forEach(k => u[k]);   // k: "id" | "name"

这不是 bug，是结构化类型的合理保守。要安全的反射就用 satisfies（12 章）+ helper 自己保障。

陷阱 6 · {} 不是"空对象"

{} 在 TS 里是所有非 null/undefined 值的类型，不是"空对象"：

typescript 
function f(x: {}) {}
f({});       // ✓
f({ a: 1 }); // ✓
f("abc");    // ✓ — 字符串也满足
f(123);      // ✓
f(true);     // ✓
f(null);     // ❌
f(undefined); // ❌

规避：

  • "任意对象"用 Record<string, unknown> 或 object
  • "任意值"用 unknown
  • "绝对无字段的对象"用 type-fest 的 EmptyObject 或自定义 type Empty = Record<string, never>
