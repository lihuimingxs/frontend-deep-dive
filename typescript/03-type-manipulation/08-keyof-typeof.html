08 · keyof / typeof / Indexed Access

三个互相配合的类型操作符，把 JS 的"动态属性访问"在类型层做精确建模。keyof T 取出属性名 union；typeof x（type 上下文）把值反推成类型；T[K] 按 key 取属性类型。三者组合是 09/10/11 三章工具的基础 —— 没有它们，Pick / Omit / Partial 等 utility 都写不出来。

阅读时长 ~25 分钟 前置 07 Generics 对应 typescriptlang.org Handbook · keyof / typeof / Indexed Access

01 速览

三个操作符的最小例子：

typescript 
interface User {
  id: string;
  name: string;
  age: number;
}

// keyof T：取出所有属性名
type UserKeys = keyof User;     // "id" | "name" | "age"

// T[K]：按 key 取属性类型
type Name = User["name"];        // string
type Age = User["age"];           // number

// T[K1 | K2]：按多个 key 取，结果是 union
type Mixed = User["name" | "age"]; // string | number

// T[keyof T]：取所有属性值类型的 union
type Values = User[keyof User];    // string | number

// typeof x（在 type 上下文中）：从值反推类型
const config = { theme: "dark", size: 12 };
type Config = typeof config;      // { theme: string; size: number }

组合起来：从一个值（运行时常量）反推出类型 + 从类型取键 + 按键取值类型 = 类型层的"反射"能力。

02 底层逻辑

1. typeof 是"值 → 类型"的桥

注意：JS 里 typeof x 在值上下文是运行时操作（返回 "string" / "number" 等字符串）。在 TS 的类型上下文，typeof 是把值反推成类型的操作符 —— 完全不同。

typescript 
// 值上下文：JS 的运行时操作
const t = typeof "hello";        // t: string，运行时值是 "string"

// 类型上下文：TS 的反推操作
const greeting = "hello";
type T = typeof greeting;        // T: "hello"（字面量类型！因为 const）

typeof 配合 as const 是把"字面量精确性"提到类型层的标准做法：

typescript 
const COLORS = ["red", "green", "blue"] as const;
type ColorTuple = typeof COLORS;        // readonly ["red", "green", "blue"]
type Color = (typeof COLORS)[number];   // "red" | "green" | "blue"

// 应用：让运行时常量驱动类型
function setColor(c: Color) { /* ... */ }
setColor("red");        // ✓
setColor("yellow");     // ❌

为什么这个模式重要？因为运行时和类型层共用一份数据，永远不会"加了运行时枚举值忘了改类型"。

2. keyof 在不同类型上的行为

keyof T 是对象类型 T 所有 key 的 union。但 TS 的对象类型形态多样，keyof 对每种返回不同结果：

typescript 
// 普通对象 — string literal union
interface A { x: number; y: string }
type KA = keyof A;   // "x" | "y"

// string index signature — string | number（妥协见下文）
interface B { [k: string]: number }
type KB = keyof B;   // string | number

// number index signature — number
interface C { [k: number]: string }
type KC = keyof C;   // number

// 联合两种 key
interface D {
  x: string;
  [k: number]: boolean;
}
type KD = keyof D;   // "x" | number

// 类
class E { foo = 1; bar() {} }
type KE = keyof E;   // "foo" | "bar"

// 数组（注意 — 含所有 Array 原型方法）
type KArr = keyof string[];   // number | "length" | "push" | "pop" | ... 100+ keys

// tuple
type KTup = keyof [string, number];   // number | "0" | "1" | "length" | ...

// 原始类型
type KStr = keyof string;   // number | "length" | "toString" | "charAt" | ...

关键观察：keyof 包括方法和原型链上的成员。在数组 / 字符串上调用 keyof 几乎得不到有用结果。

3. 为什么 keyof [k: string]: T 返回 string | number

这看似不直觉的"妥协"来自 JS 设计：对象的 key 永远是字符串。即使你写 obj[1] = "x"，运行时 1 也被转换为 "1"。

typescript 
interface Numbers {
  [n: number]: string;
}

const o: Numbers = { 0: "zero", 1: "one" };
o[0];     // ✓ 类型是 string
o["0"];   // ❌ 类型层不允许，但运行时一样有效

// 但 TS 把 number index 桥接到 string —— "凡是能 toString 的 number 都能用"
type K = keyof Numbers;   // number（不是 string | number）

相反，[k: string]: T 的 keyof 是 string | number，因为所有 number 字符串化后都满足 string。这是 JS 的妥协反射到 TS 类型上。

4. Indexed access 的 key 必须是类型，不是值

T[K] 的 K 是类型，不是值变量。这是初学者最易混淆的：

typescript 
interface User { id: string; name: string }

// ✓ 类型字面量
type Id = User["id"];          // string

// ✓ 通过 typeof 把 const 值变成类型字面量
const k = "id";
type Id2 = User[typeof k];     // string

// ❌ 直接用值变量
const k2 = "id";
type Id3 = User[k2];           // 'k2' refers to a value, but is being used as a type
//                                Did you mean 'typeof k2'?

// ✓ keyof 给的也是类型
type Keys = keyof User;
type AllValues = User[Keys];   // string

这一规则源于 01 章讲的类型与值是两个独立命名空间。T[K] 是类型空间运算，所以 K 必须先到类型空间。

5. keyof typeof 是高频组合

一个常见模式：从一个 const 对象同时取它的 key union 和 value union：

typescript 
const STATUS = {
  pending: 0,
  active: 1,
  done: 2,
} as const;

type StatusKey = keyof typeof STATUS;
//   ^? "pending" | "active" | "done"

type StatusValue = (typeof STATUS)[StatusKey];
//   ^? 0 | 1 | 2

// 实战：替代 enum
function toStatus(s: StatusKey): StatusValue {
  return STATUS[s];
}
toStatus("active");   // ✓ 返回 1
toStatus("typo");     // ❌

这套模式可以完全替代 enum，且：

  • 没有运行时 enum 对象（更小的产物）
  • tree-shaking 干净
  • JSON 序列化天然兼容
  • 可以在 schema 里直接当字面量 union 用

新代码不要写 enum，写 const X = { ... } as const + keyof typeof X。

6. 数组 / tuple 的 indexed access 技巧

数组是对象，arr[number] 返回元素类型；tuple 用具体下标取精确类型；["length"] 取长度（tuple 是字面量数字，array 是 number）：

typescript 
// 数组元素类型
type Arr = string[];
type Item = Arr[number];          // string

// 联合数组
type Mixed = (string | number)[];
type Item2 = Mixed[number];       // string | number

// tuple 按位置
type Pair = [string, number];
type First = Pair[0];             // string
type Second = Pair[1];            // number
type Both = Pair[number];         // string | number（取所有元素的 union）

// tuple 长度
type Len = Pair["length"];         // 2（字面量！）

// 数组长度
type ArrLen = string[]["length"];  // number

// 通过 typeof 从 const 取
const COLORS = ["red", "green", "blue"] as const;
type ColorTuple = typeof COLORS;        // readonly ["red", "green", "blue"]
type Color = ColorTuple[number];        // "red" | "green" | "blue"
type Count = ColorTuple["length"];      // 3

7. typeof 的限制：只能用在标识符或属性访问

typeof 不能用在任意表达式上：

typescript 
const a = 1;
type A = typeof a;             // ✓ 标识符

const obj = { x: 1 };
type X = typeof obj.x;          // ✓ 属性访问
type Y = typeof obj["x"];       // ✓ 索引访问

// ❌ 不允许
type B = typeof (1 + 1);        // 'typeof' can only be used on identifiers and property accesses
type C = typeof someFn();       // 不允许调用
type D = typeof a + b;           // 不允许运算

要"从函数调用结果取类型"用 ReturnType<typeof fn>（11 章）。要"从复杂表达式取类型"先把它存到 const 再 typeof。

03 应用场景

场景 A · 类型安全的 pick / omit 工具

typescript 
function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K> {
  const out = {} as Pick<T, K>;
  for (const k of keys) out[k] = obj[k];
  return out;
}

const u = { id: "1", name: "Alice", age: 30, email: "a@b.c" };
const sub = pick(u, ["id", "name"]);
//    ^? { id: string; name: string }   精确

pick(u, ["typo"]);   // ❌ '"typo"' not assignable to keyof user

// omit 同理（用 Exclude，09 章）
function omit<T, K extends keyof T>(obj: T, keys: K[]): Omit<T, K> {
  const out = { ...obj } as Omit<T, K> & T;
  for (const k of keys) delete (out as Record<K, unknown>)[k];
  return out;
}

const sub2 = omit(u, ["email", "age"]);
//    ^? { id: string; name: string }

场景 B · 类型化的 i18n key

typescript 
// 把翻译表的所有 key 提到类型层
const messages = {
  hello: "Hello",
  goodbye: "Goodbye",
  user: {
    profile: "Profile",
    settings: "Settings",
  },
} as const;

// 顶层 key
type TopKey = keyof typeof messages;
//   ^? "hello" | "goodbye" | "user"

function t(key: TopKey): string {
  const v = messages[key];
  return typeof v === "string" ? v : "[nested]";
}

t("hello");      // ✓
t("missing");    // ❌

如果要支持嵌套 key（如 "user.profile"），需要 10 章的 template literal types。这里只演示扁平 key。

场景 C · Redux selector 链式取值

typescript 
interface State {
  user: {
    profile: { name: string; email: string };
    settings: { theme: "light" | "dark" };
  };
  posts: { id: string; title: string }[];
}

type UserState = State["user"];
//   ^? { profile: { name: string; email: string }; settings: ... }

type Profile = State["user"]["profile"];
//   ^? { name: string; email: string }

type Theme = State["user"]["settings"]["theme"];
//   ^? "light" | "dark"

type Post = State["posts"][number];
//   ^? { id: string; title: string }

// selector 函数返回类型自动跟随
const selectTheme = (s: State): Theme => s.user.settings.theme;
const selectPosts = (s: State): Post[] => s.posts;

场景 D · 配置驱动的 dispatcher

typescript 
// 处理器表
const handlers = {
  ping: (msg: { id: string }) => `pong:${msg.id}`,
  echo: (msg: { text: string }) => msg.text,
  add: (msg: { a: number; b: number }) => msg.a + msg.b,
} as const;

// 自动派生：能处理的 message 类型 union
type MessageType = keyof typeof handlers;
//   ^? "ping" | "echo" | "add"

// 每种 message 的 payload 类型
type PayloadOf<T extends MessageType> = Parameters<(typeof handlers)[T]>[0];

// 派发函数
function dispatch<T extends MessageType>(type: T, payload: PayloadOf<T>) {
  return handlers[type](payload as any);
}

dispatch("ping", { id: "x" });          // ✓ 返回 string
dispatch("add", { a: 1, b: 2 });        // ✓ 返回 number
dispatch("echo", { id: "x" });          // ❌ payload 不匹配 echo 的参数

这种模式让"加 handler 自动同步类型"，是 tRPC / actions 框架的核心思路。

04 典型代码

从值派生类型的全谱

typescript 
// 简单值
const port = 3000;
type Port = typeof port;          // number（const 数字仍推宽到 number）

// 字面量保留需要 as const
const env = "production" as const;
type Env = typeof env;             // "production"

// 函数
const greet = (name: string) => `Hi ${name}`;
type GreetFn = typeof greet;       // (name: string) => string

// 类
class User { name = "" }
type UserCtor = typeof User;       // typeof User（构造器类型）
type UserInstance = InstanceType<typeof User>;  // User

// 模块（命名空间）
import * as fs from "node:fs";
type FS = typeof fs;

indexed access 的复杂用法

typescript 
interface Form {
  name: { value: string; error?: string };
  age: { value: number; error?: string };
}

// 取所有字段的 value 类型
type Values = Form[keyof Form]["value"];   // string | number

// 取某个特定字段的子类型
type AgeError = Form["age"]["error"];       // string | undefined

// 通过类型参数动态取
type FieldValue<K extends keyof Form> = Form[K]["value"];
type Name = FieldValue<"name">;             // string
type Age = FieldValue<"age">;               // number

从 const 数组派生 union 的标准模式

typescript 
const ROLES = ["admin", "editor", "viewer"] as const;
type Role = (typeof ROLES)[number];
//   ^? "admin" | "editor" | "viewer"

// 现在运行时和类型一致
function checkRole(r: Role) { /* ... */ }
ROLES.forEach(checkRole);   // ✓
checkRole("admin");          // ✓
checkRole("guest");          // ❌

// 反向：从对象 values 派生 union
const HTTP_CODES = { OK: 200, NOT_FOUND: 404, SERVER_ERROR: 500 } as const;
type HttpCode = (typeof HTTP_CODES)[keyof typeof HTTP_CODES];
//   ^? 200 | 404 | 500

类型安全的 Object.keys / Object.entries

typescript 
// 标准 Object.keys 返回 string[]，丢失类型
function objectKeys<T extends object>(o: T): (keyof T)[] {
  return Object.keys(o) as (keyof T)[];
}

const u = { id: "1", name: "Alice", age: 30 };
objectKeys(u).forEach(k => {
  //                  ^? "id" | "name" | "age"
  const v = u[k];   // ✓
});

// Object.entries 同理
function objectEntries<T extends object>(o: T): [keyof T, T[keyof T]][] {
  return Object.entries(o) as [keyof T, T[keyof T]][];
}

objectEntries(u).forEach(([k, v]) => {
  // k: keyof typeof u, v: string | number
});

⚠️ 这种 helper 内部用 as 断言 —— 因为 TS 的结构化类型不能保证对象的 key 仅限声明的字段（02 章讲过）。这是开发者承诺，不是真正的类型安全。但实际写应用代码时几乎总是对的。

05 常见陷阱

陷阱 1 · keyof 数组 / 字符串包括所有原型方法
typescript 
type K1 = keyof string;   // number | "length" | "toString" | "charAt" | "concat" | ... 50+
type K2 = keyof number[];  // number | "length" | "push" | "pop" | "map" | ... 100+

// 直觉错误的用法
function getCharAt(s: string, k: keyof string) {
  return s[k];   // s 可能是 .toString / .charAt / ... 完全不是想要的

规避：要"数组下标"用 number，不用 keyof Array<T>。要"字符串字符"通过 charAt(i)，i 是 number。

陷阱 2 · T[keyof T] 在 T 有 method 时包括 method 类型
typescript 
interface User {
  name: string;
  age: number;
  greet(): string;
}

type Values = User[keyof User];
//   ^? string | number | (() => string)
//                       ^^^^^^^^^^^^^^^^ method 类型也被包括

规避：先过滤掉 method 类型（用 09 章的 conditional types）：

typescript 
type NonFunctionKeys<T> = {
  [K in keyof T]: T[K] extends Function ? never : K
}[keyof T];

type DataKeys = NonFunctionKeys<User>;        // "name" | "age"
type DataValues = User[DataKeys];              // string | number
陷阱 3 · typeof 不能直接用在表达式上
typescript 
// ❌ 不允许
type A = typeof someFn();           // typeof 不能调用
type B = typeof a.b.c.d.toString(); // 同样不行
type C = typeof (1 + 2);             // 不允许运算

// ✅ 解法：先 const 再 typeof，或用 ReturnType
declare function getUser(): User;
type GetUserReturn = ReturnType<typeof getUser>;   // User
陷阱 4 · index access 的 key 必须用类型空间值
typescript 
interface User { id: string }

const k = "id";
type Bad = User[k];    // ❌ 'k' refers to a value, but is being used as a type
//                       Did you mean 'typeof k'?

type Good = User[typeof k];   // ✓ string

// 或直接用字面量
type Good2 = User["id"];      // ✓

规避：每次写 T[xxx] 时确保 xxx 是类型，不是值。const 字符串变量需要 typeof 桥接。

陷阱 5 · 直接 typeof 数组得宽类型，不是字面量
typescript 
const colors = ["red", "green", "blue"];
type C1 = typeof colors;             // string[]
type Color1 = C1[number];             // string  ← 不是字面量

// ✅ 必须 as const
const colors2 = ["red", "green", "blue"] as const;
type C2 = typeof colors2;             // readonly ["red", "green", "blue"]
type Color2 = C2[number];              // "red" | "green" | "blue"
陷阱 6 · keyof type alias 不能 declaration merge

type 不能 declaration merge（02 章讲过）。所以基于 type alias 的 keyof 在第三方扩展时不会被新字段影响：

typescript 
// 用户库
type User = { id: string; name: string };

// 你想给它加字段 — type 做不到
// type User = { ...User, email: string };   // ❌ Duplicate identifier

// 但如果是 interface，可以
interface UserI { id: string; name: string }
interface UserI { email: string }   // ✓ 自动合并
type Keys = keyof UserI;             // "id" | "name" | "email"

规避：要被外部扩展的形状（库的公共类型）用 interface。详见 15 章 declaration merging。
