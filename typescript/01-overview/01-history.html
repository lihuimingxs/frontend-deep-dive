TypeScript 项目史 2010-2025

TypeScript 15 年的故事讲成一条主线 —— 不是按版本号流水账，而是按"每次关键决策及其根因"来组织。从 2010 年微软内部立项（Anders Hejlsberg 主导）、2012 公开 v0.8、2014 v1.0 做出"性能优先 / 不做 sound type system"决策、2017-2020 类型操作三件套（条件 / 映射 / 模板字面量）落地、到 2024 年 Microsoft 宣布用 Go 重写整个编译器（tsgo / Project Corsa）、2025 TC39 Type Annotations 提案重启——理解这条线之后，许多"TS 为什么这样设计"的问题就有了根因。

阅读时长 ~25 分钟 前置 任意 TypeScript 经验 主要来源 Anders Hejlsberg 演讲 · TS Release Notes · TC39 Type Annotations 提案文档

01 速览

  1. TypeScript 是 Microsoft 2012 年公开发布的 JavaScript 超集。"超集"意味着：任何合法 JS 是合法 TS。这条渐进采用设计是 TS 击败所有竞争对手（Flow / Hegel / Dart）的根本原因。
  2. TS 由 Anders Hejlsberg 主导设计—— 他是 Turbo Pascal / Delphi / C# 的设计者，类型系统设计经验最丰富的工程师之一。这条传承给 TS 留下了"务实优先 / 不追学术完美"的基因。
  3. TS 15 年走过 4 个阶段：2010-2014 立项与 v1.0（决定性能优先 / 不做 sound）→ 2014-2020 类型表达力爆发（泛型 / 条件 / 映射 / 模板字面量）→ 2020-2024 工程化 + 编辑器集成 → 2024+ tsgo Go 重写（10× 速度）+ TC39 Type Annotations 提案。每个阶段都解决了 JS 工程化的具体痛点。

02 底层逻辑

史前史（2008-2010）：Microsoft 的 JS 痛点

2010 年 Microsoft 内部已经在用 JS 开发大型 web 应用——Outlook Web App / Office 365 / Bing Maps 等。代码规模达到几十万行，JS 的弱类型让大型项目难以维护：重构容易破坏、跨文件函数签名变化没编译错误、新人入职难以理解 API 边界。

同一时间，Google 推出了 Closure Compiler（2009）—— 用 JSDoc 注释加类型，做静态检查 + 优化。Microsoft 内部研究了这套方案但觉得不够——JSDoc 注释式类型表达力有限、错误信息不友好、IDE 集成麻烦。2010 年 Microsoft 决定自己做—— 想要"结构化的类型系统嵌入到 JS 工具链"。

领头人选了 Anders Hejlsberg—— Turbo Pascal（1983）/ Delphi（1995）/ C#（2000）的设计者，当时业界最资深的类型系统设计者之一。Hejlsberg 是丹麦人，2007 年到微软主导 .NET Framework / C# 设计——他对"务实类型系统"有深刻理解：C# 早期就因为追求 sound 太严格遭到吐槽，后来加 dynamic 类型解决一些场景。这种"务实优先 / 兼容性优先"的基因后来直接传给了 TypeScript。

第一阶段 · 2010-2014 · 立项与 v1.0

2010 年：微软内部立项，code name "Strada"。Hejlsberg + Steve Lucco 主导。最初想法是"给 JS 加类型 + 类"——这正好对应 TC39 当时的 ES4 提案（也想给 JS 加类 + 类型）。

2012-10-01：Microsoft 在 NodeConf（？？纠正：应是 BUILD 大会）公开发布 TypeScript 0.8。GitHub 开源（Apache 2.0 协议）。第一版主要特性：类型注解（: string）+ 接口（interface）+ 类（class，提前于 ES2015）+ 模块（module）。

2013：v0.9 加泛型（generics）—— 这是 TS 区别于 Closure Compiler 等竞品的关键特性。泛型让"容器类型"（Array<T> / Promise<T>）成为可能。

2014-04-02 · v1.0 正式发布—— TypeScript 历史上最关键的几个决策都在 v1.0 期间确定：

  • 结构化类型（structural typing）而非命名类型（nominal typing）—— 两个对象只要"形状"匹配就被认为同类型，不需要显式 implements。这是为了兼容 JS 的"鸭子类型"传统：JS 写 {name: "..."} 不会显式标记类型，TS 不能强制要求 implements User。
  • 类型擦除（type erasure）—— 编译输出纯 JS，运行时没有类型信息。这对应 JVM 泛型擦除——但 TS 是更彻底的擦除（连 type / interface 都消失）。这条决策让 TS 编译产物 100% 与现有 JS 工具链兼容。
  • 不做 sound type system—— sound 意味着"类型检查通过的代码运行时不会出 type error"。TS 明确放弃 sound（如允许 any 类型 / 协变变体的"双向兼容"）—— 因为sound 会破坏对 JS 生态的兼容性。这是 TS 与 Flow / Hegel 最大的设计差异。
  • 不做 dependent types—— 不支持"类型依赖于值"（如 Array<Length=5>）。这种类型系统表达力极强但编译性能极差。TS 选了务实路线：性能优先。
⚠️ TypeScript "不 sound" 不等于"差"

很多人误以为 sound 是好的。实际上类型系统设计是权衡题：sound 太严会让 JS 生态的渐进采用不可能（JS 写法不能直接通过 TS 检查）。Hejlsberg 的选择是：用 any 留逃生口，让用户能渐进加类型，"不 sound"是 TS 取代 Flow 的根因—— Flow 太严了。

第二阶段 · 2014-2020 · 类型表达力爆发

v1.0 之后 TS 进入快速演化期。每个版本都加表达力或修体验：

  • v1.4 (2015-01): 联合类型（A | B）+ 类型守卫（typeof 缩窄）—— 让"参数能是几种类型"在类型层表达
  • v1.5 (2015-07): let / const / module→namespace 改名（避免与 ES2015 module 冲突）
  • v1.6 (2015-09): JSX 支持—— React 工程师终于不用手动写 // @flow
  • v2.0 (2016-09): strictNullChecks（null safety）+ never 类型 + readonly—— 这是 TS 类型系统第一次"认真起来"。strictNullChecks 后用户必须显式处理 null / undefined，2017 起几乎所有新 TS 项目都默认开
  • v2.1 (2016-12): keyof 操作符 + 映射类型（{[K in T]: ...}）—— 类型操作三件套之一
  • v2.8 (2018-03): 条件类型（T extends U ? X : Y）+ infer 关键字—— 类型操作三件套之二，让 TS 类型系统达到图灵完备级表达力
  • v3.0 (2018-07): 项目引用（project references）—— 大型 monorepo 工程化基础
  • v3.4 (2019-03): const assertions（as const）—— 让字面量类型保持狭窄
  • v3.7 (2019-11): 可选链 ?. + 空合并 ??—— 与 ES2020 同步落地
  • v4.0 (2020-08): variadic tuple types（变长元组）+ 各种 quality-of-life 改进
  • v4.1 (2020-11): 模板字面量类型（`${string}-${number}`）—— 类型操作三件套之三，让"字符串模式匹配"在类型层表达。这是 TS 类型系统表达力的巅峰，能在类型层做 JSON 解析、SQL 类型推导、路由参数提取等

第三阶段 · 2020-2024 · 工程化 + ESM + 装饰器

2020 年后 TS 重心从"加表达力"转向"工程化场景修补"——大型项目编译性能、ESM 支持、装饰器规范化、tsconfig 优化等。

  • v4.7 (2022-05): ESM Node 支持—— Node ESM 1.0 的复杂性（详见 Node P3.2）让 TS 必须特别支持。moduleResolution: "NodeNext" 引入
  • v5.0 (2023-03): 装饰器（Stage 3 规范）正式支持—— 之前 TS 的 experimentalDecorators 是基于 Stage 1 的旧提案，与 ES 标准不一致。v5.0 实现 Stage 3 新规范——让 TS 装饰器与未来 ES 装饰器对齐。这也意味着v5.0 起 experimentalDecorators 是过时方案，新代码应迁移
  • v5.0-5.4 (2023-2024): const 类型参数、accessor decorators、各种 inference 改进

第四阶段 · 2024-2025 · tsgo（Go 重写）+ TC39 Type Annotations

2024 年 3 月，Microsoft TypeScript 团队公布历史上最大的工程决策—— 用 Go 重写整个 TypeScript 编译器，代号 "Project Corsa"，简称 tsgo。

为什么重写？性能瓶颈

原 tsc 编译器用 TypeScript 自身写（dogfooding），跑在 Node 上。这套架构在小项目跑得快，但大型项目（VSCode 几十万行 TS / Microsoft Office 365）编译慢得离谱—— 完整 type check 一遍 60 秒+，editor 反应延迟用户能感觉到。

根本原因有几个：(1) JS 单线程，tsc 不能并行；(2) Node startup 开销不可忽略；(3) TS 自身类型系统复杂度（条件类型 / 模板字面量 / infer）让类型检查算法本身是 NP-hard 级别。

为什么选 Go？

团队评估了 Rust / Go / C# / TypeScript+优化 几个选项。最终选 Go：

  • Go 编译速度本身快—— 编译 Go 写的编译器比 Rust 快几倍
  • Go 的 GC + goroutines 适合 type checker—— 类型 checker 大量用图遍历，goroutines 让并行 type check 可能
  • Go 标准库够用—— 不需要 Rust 的零开销抽象
  • Microsoft 内部 Go 经验多—— Azure / dotnet 后端用 Go
  • 放弃 TS dogfooding—— Anders Hejlsberg 公开承认这是务实决定，"用 TS 写 TS 编译器是哲学正确但不是工程最优"

tsgo 的承诺

  • 10× 编译速度—— Microsoft 内部 demo 显示 60 秒 → 6 秒
  • 10× 内存效率
  • API 兼容—— tsgo 输出与 tsc 完全一致；CLI / tsconfig 兼容
  • Editor 集成更快—— Language Server 反应几乎实时

2025-04 当前状态

tsgo 仍在开发中，2024 后期发布 alpha 版本，2025 年内承诺发布 stable。预期影响：大型 TS 项目（特别是 Microsoft / 大厂）编译性能瓶颈消除—— 让 TS 在更大项目（百万行级别）继续主导。

📊 tsgo 反映的工程哲学

"用 TS 写 TS 编译器"是 dogfooding 的极致——TS 团队 12 年坚持。但 2024 团队公开承认：dogfooding 是哲学正确但不是工程最优。当性能瓶颈到一定程度，选最适合的工具比保持哲学纯洁性重要。这条思路适用所有大型工程——Pinterest 用 Java + Python、Microsoft 用 C# + Go + TS、Google 用 Java + Go + C++ + Python——同一组织里多语言并存是务实选择。

TC39 Type Annotations 提案（Stage 1，2022-）

2022 年 Microsoft 团队推动 TC39 启动 "Type Annotations" 提案——想把 TS 类型语法纳入 JS 标准。提案核心：

  • JS 引擎"忽略"类型注解—— 把 let x: number = 1 当 let x = 1 跑，类型不影响运行时
  • 让浏览器原生支持 TS 语法—— 不需要编译步骤就能跑 TS 代码（虽然类型不被检查）
  • TS 类型语法成 ES 标准的一部分—— 但不把 TS 类型检查器纳入引擎

这个提案目前 Stage 1（2025-04），进展缓慢——TC39 内部对"把单一公司的语法纳入标准"有顾虑。如果通过，将是TS 历史上最大的胜利——TS 类型系统成为 JS 事实标准。但即使不通过，TS 已经在生态上占了主导。

三条底层逻辑

务实优先一直是 TS 的基因

从 v1.0 选择"不 sound" / 选择"结构化类型" / 选择"类型擦除"，到 v5.0 选择 Stage 3 装饰器、2024 选 Go 重写，TS 团队一直选务实而非学术完美。这条基因来自 Hejlsberg 的 C# 经验——他在 C# 早期已经踩过"sound 太严"的坑。这就是为什么 TS 能击败 Flow（太严）/ Hegel（学术）/ ReScript（不渐进）—— 务实让用户能渐进采用。

TS 与 ES 是双向影响

TS 一直追着 ES 提案落地（class / let-const / async-await / decorators 都先在 TS 实现）。但反向也存在——装饰器 stage 3 规范受 TS 实现启发，TC39 Type Annotations 提案直接就是 TS 团队推动的。今天 TS 不仅是 JS 的"渐进类型补丁"，也是 TC39 思考"未来 JS"的实验场。

tsgo 是 JS 工具链 Rust/Go 化趋势的一环

2020 年后 JS 工具链大批用非 JS 写：esbuild（Go）/ Vite（Go + Rust）/ swc（Rust）/ Rome→Biome（Rust）/ Bun（Zig）/ tsgo（Go）。原因不是"JS 太慢不能写工具链"——而是工具链对启动速度 / 并行 / 内存极敏感，这些场景非 JS 语言确实更优。TS 用户语言层不变（TS / JS），但工具链层快速去 JS 化—— 这是 2025 年的最大趋势之一。

03 应用场景

场景 A · 给团队讲"TS 为什么这样设计"

新人或转岗工程师常问"为什么 TS 不像 Java 那样严"、"为什么 any 这么宽容"。理解 v1.0 决策（务实优先 / 兼容 JS 生态 / 不 sound）后，这些问题都有具体根因——TS 是设计来取代 JSDoc，不是取代 Java。这套背景比直接讲 strictNullChecks / noImplicitAny 等 flag 记得更深。

场景 B · 老 TS 项目升级 v5

从 TS 4.x 升级到 5.x：关键 breaking change 是装饰器（v5.0 改用 Stage 3 规范，旧 experimentalDecorators 项目要明确选择保留旧 flag 或迁移）。其他大多数升级是 quality-of-life 改进。建议读 TS 5.0 release notes。

场景 C · 评估 tsgo 何时切换

2025 年中 tsgo stable 后，大型 TS 项目（编译时间 > 30 秒）应该评估迁移。判据：(1) 编译时间是不是瓶颈？(2) CI/CD pipeline 重 type check 多久？(3) Editor 反应延迟？大型 TS monorepo 切换 tsgo 收益最大。小项目（单包 < 10k 行）不必着急——tsc 已经够快。

场景 D · 解释"为什么 TS 写工具但 tsgo 用 Go"

团队成员可能困惑——"既然 TS 这么强为什么 Microsoft 用 Go 重写编译器"。理解：TS 是用户应用语言，Go 是系统工具语言。两者解决不同问题。Microsoft 用 TS 写 VSCode（用户应用）+ 用 Go 写 tsgo（系统工具）+ 用 C# 写 .NET（基础设施）—— 大型组织多语言并存是常态。

05 常见陷阱

"TS 是带类型的 JS"——简化但有用的描述

更准确：TS 是JS 超集 + 编译期类型检查。运行时没有类型——擦除后就是普通 JS。这就是为什么 TS 不能做运行时类型 check（要用 zod / yup / class-validator 等运行时验证库）。

"TS 不 sound 是设计缺陷"——是务实选择

sound 会让 JS 渐进采用不可能。TS 选了"不 sound 但能用"，这正是它击败 Flow 的根因。

v5.0 装饰器与 v4.x experimental 装饰器不兼容

升级 v5 时要决定：保留 experimentalDecorators flag（旧代码）还是迁移到 Stage 3 装饰器（推荐）。

tsgo 不是 tsc 的替代品（短期内）

2025 上半年 tsgo 仍在 alpha；生产环境继续用 tsc 直到 stable。
