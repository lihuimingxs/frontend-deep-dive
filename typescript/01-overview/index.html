Part 1 · TypeScript 是谁

这一阶段不讲技术细节，只解决两个问题：TypeScript 怎么走到今天、它在类型系统家族里站什么位置。给后面 Part 2-5 所有"为什么 TS 这样设计"提供历史与生态坐标。本阶段是参照 V8 / ECMA / Node / React 主题的"Part 1 是谁"模式补的——它们都用第一阶段开篇讲项目史与同代人，TS 主题原本只直接进 Handbook 基础，少了这层。

章节数 2 / 2 ✅ 主要来源 Anders Hejlsberg 演讲 · TS Release Notes · TC39 Type Annotations 提案 状态 ✅ 全部完成

本阶段覆盖

  • 项目史 15 年 —— 2010 微软内部立项（Anders Hejlsberg / C# 父亲主导，code name Strada）→ 2012-10 v0.8 公开（NodeConf）→ 2014-04 v1.0 决定性能优先 / 不做 sound type system → 2017-2020 类型操作三件套（条件 / 映射 / 模板字面量类型）落地 → 2024-03 announce tsgo（Project Corsa） 用 Go 重写编译器（10× 速度）→ 2025 TC39 Type Annotations 提案 Stage 1 重启
  • 关键设计决策 —— 为什么 TS 选结构化类型而非命名类型、为什么不做 sound（兼容 JS 生态优先）、为什么不做 dependent types（性能 + 学习成本）、为什么 2024 选 Go 重写而不是 Rust
  • 类型系统家族对照 —— Flow（Facebook，相似但更 sound）/ Hegel（纯 sound 但生态死）/ JSDoc + tsc（与 TS 共生模式）/ ReScript（OCaml 后端，离 JS 较远语法）/ Reason / Closure Compiler（Google 内部，已停）
  • TC39 Type Annotations 提案（Stage 1） —— 把 TS 类型语法纳入 JS 标准的可能；浏览器原生忽略类型注解（仍由 tsc / tsgo 检查）

💡 这一阶段与 ECMA / JS 主题的关系

TS 项目史与 ECMAScript / JavaScript 项目史是平行又交织的——TS 一直追着 ES 提案落地，但也反向影响过 ES（如 decorators 的 stage 3 规范受 TS 实现启发）。详见 ECMA P1.1 · 标准发展史 + JS P1.1 · JavaScript 项目史。
