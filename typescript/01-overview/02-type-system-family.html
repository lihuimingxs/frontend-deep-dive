类型系统家族中的位置

Part 1.1 讲了 TS 怎么走到今天。本章讲 TS 站在 2025 年看是什么类型系统—— 它的4 个核心设计选择（结构化类型 / 类型擦除 / 不 sound / 渐进采用）、它的同代人（Flow / Hegel / JSDoc / ReScript / Closure Compiler 各自结局）、它与传统类型语言（Java / Haskell / Rust）的对比、TC39 Type Annotations 提案对未来的影响。读完后你能给团队讲清"TS 在 2025 年还有没有挑战者"。

阅读时长 ~25 分钟 前置 Part 1.1 项目史 主要来源 Flow / Hegel / ReScript 官方文档 · TC39 Type Annotations 提案 · Anders Hejlsberg 演讲

01 速览

  1. TS 不是"最强类型系统"——是"最务实类型系统"。它故意放弃 sound（让生态兼容）、放弃 dependent types（让性能可控），换来了 JS 生态的无缝渐进采用。
  2. 所有挑战者都因不渐进而萎缩。Flow 太严 / Hegel 太学术 / ReScript 语法太远——它们没能在 npm 生态站稳。这条规则在 2025 年仍然适用。
  3. TC39 Type Annotations 提案如果通过，TS 类型语法成 JS 标准—— 浏览器原生支持类型注解（虽然不检查）。这会进一步巩固 TS 地位。

02 底层逻辑

TS 的 4 个核心设计选择

① 结构化类型（structural typing）而非命名类型

Java / C# 用命名类型—— 两个 class 即使字段相同也不互相兼容（必须显式 implements interface）。TS 用结构化类型——只要"形状"匹配就互相兼容。

typescript 
interface Point { x: number; y: number; }
interface Vector { x: number; y: number; }
const p: Point = { x: 1, y: 2 };
const v: Vector = p;  // ✅ 合法 - 结构匹配
// Java/C# 里同样的代码会报"类型不匹配"

这个选择直接对应 JS 鸭子类型传统——JS 写 {name: "..."} 不会显式标记类型，TS 也不能强制。如果 TS 要求显式 implements User，大量 npm 包没法直接用—— 这就违反了"渐进采用"原则。

② 类型擦除（type erasure）

TS 编译输出纯 JS——所有 : string / interface / type 都消失。运行时无法反射回类型信息。

typescript-vs-js 
// TS 源码
interface User { name: string; age: number; }
function greet(u: User) { console.log(`Hi ${u.name}`); }

// 编译后 JS（类型完全消失）
function greet(u) { console.log(`Hi ${u.name}`); }

这条决策让 TS 编译产物100% 与现有 JS 工具链兼容——可以直接给 webpack / Babel / SWC 处理。代价是运行时不能做类型检查——必须用 zod / yup / class-validator 等运行时验证库（这是 2024 年极火的方向）。

③ 不做 sound type system（故意的）

sound 意味着"类型 check 通过的代码运行时不会出 type error"。TS 明确放弃 sound：

  • any 类型—— 故意提供逃生口，让用户能渐进加类型
  • 函数参数双向兼容（bivariant）—— TS 允许 (x: Animal) => void 赋给 (x: Dog) => void（理论上不安全），换 callback API 兼容
  • "index 类型"宽松—— arr[100] 在 TS 里类型是 T（即使可能 undefined），需要 noUncheckedIndexedAccess flag 才更严

这些"不 sound"决策都是为了兼容 JS 写法。Flow（Facebook 的同类型项目）尝试做更 sound 的版本，结果是很多 JS 库的写法 Flow 直接报错，用户只能不停加 $FlowFixMe 注释——最终 Flow 失败。TS 的"不 sound"设计是它击败 Flow 的核心原因。

④ 渐进采用（gradual typing）

这是 TS 最重要也最被低估的设计——JS 超集 + any 类型 + // @ts-check 在 .js 文件里启用类型检查。让用户能：

  • 把项目里 1 个文件改 .ts，其他保持 .js
  • 第三方库没 .d.ts 时用 any 兜底
  • 从 JSDoc 注释开始，慢慢迁移到 .ts

这条渐进采用路径是 TS 击败所有竞争者的根本原因。任何要求"从一开始就用全套类型系统"的语言（Dart / Elm / ReScript）都失败了——因为切换成本太高。

TS 的同代人：4 个挑战者的不同结局

同代人 · Flow（Facebook，2014-）

Facebook 2014 年推出 Flow，和 TS 几乎同时。Flow 设计上比 TS 更接近 sound——更严的类型推断、更少的"宽松边界"。React 项目内部用 Flow 几年，但 2018 后逐步迁移到 TS。

Flow 失败的原因：

  • 太严——很多 npm 包的 JS 写法 Flow 直接报错，用户疲于打补丁
  • 生态贫血——Flow 的类型定义生态远小于 TS 的 DefinitelyTyped（@types）
  • 工具链集成弱——VSCode / WebStorm 对 TS 集成更好
  • Microsoft 投入大—— TS 团队几十人持续迭代，Flow 团队规模小

2024 年 Facebook 内部 React Native 仍用 Flow，但外部生态完全是 TS。Flow 是"更 sound 但失去用户"的典型案例。

同代人 · Hegel（2018-2021，已死）

Hegel 是 2018 年俄罗斯工程师 JSMonk 团队推出的纯 sound 的 JS 类型系统——比 Flow 更严，比 TS 严格 100×。例：Hegel 永远不允许 any，要求所有类型可推导，强行类型推断错误信息超长。

Hegel 失败的原因：太学术。开发者上手 30 分钟后就放弃——错误信息看不懂、写法约束太死、生态空。2021 年项目几乎停止。Hegel 是"纯学术追求 sound"的失败案例。

同代人 · JSDoc + tsc（共生模式）

JSDoc 注释（/** @param {string} name */）从 1999 起就有，是 JS 的"注释式类型"。TS 兼容 JSDoc 注释—— 你可以在 .js 文件加 JSDoc 注释，用 // @ts-check 启用 tsc 类型检查。

这种"JS + JSDoc + tsc"模式有特殊优势：

  • 不需要构建步骤—— .js 文件直接跑（重要的是 V8 / Node 不能直接跑 .ts，必须先编译）
  • 类型注解仍是注释—— grep / git diff 友好
  • 逃生口比 TS 更宽—— 默认就是"动态类型"，用 JSDoc 是"选择性"

这种模式在V8 自身代码 / Node 自身代码 / Vite 等工具链实际使用——他们要避免编译步骤又想要类型 check，JSDoc + tsc 是最优解。但用户应用层面 .ts 仍是主流。

同代人 · ReScript（OCaml 后端，2020+）

ReScript 是 BuckleScript（OCaml → JS 编译器）2020 年改名后的项目。语法离 JS 远（OCaml 风格）但类型系统比 TS 更强—— sound + 模式匹配 + ADT。

ReScript 不会成主流的原因：(1) 语法不像 JS——切换成本高；(2) 不能渐进——一个文件不能"这部分用 ReScript，那部分用 JS"；(3) 生态小——npm 类型定义大多没有 ReScript binding。

ReScript 在小众场景（金融 / 关键安全代码）有忠实用户——它是"为坚定的强类型爱好者"的选择，但永远不会取代 TS。

历史方案 · Closure Compiler（Google，2009-2024）

Google 内部从 2009 起用 Closure Compiler——基于 JSDoc 的静态类型 + 死代码消除 + 极致优化。Gmail / Google Maps 等大型 web app 都用 Closure。

2024 年 Google 内部宣布逐步迁移到 TypeScript—— Closure 维护成本高、招聘难、与外部生态脱节。这是 TS 在最后一个大公司内部对手的胜利。

TS 与传统类型语言的对比

vs Java / C#（命名类型 + sound）

共同点：都有静态类型 + 类 + 接口。不同点：

  • Java/C# 是命名类型—— 两个相同形状的 class 不兼容；TS 是结构化—— 形状一样就兼容
  • Java/C# 是 sound（虽然 Java 泛型擦除有 ClassCastException）；TS 故意不 sound
  • Java/C# 编译后是字节码（保留类型信息）；TS 编译后是 JS（类型擦除）
  • Java/C# 用JIT 优化；TS 编译后由 V8 / 引擎优化

来自 Java / C# 的工程师初学 TS 常觉"太宽松"——这是预期的——TS 的设计哲学就是JS 优先 / 务实优先。

vs Haskell / Rust（强类型 + 函数式 / 所有权）

共同点：都有泛型 + 高阶类型操作。不同点：

  • Haskell / Rust 有 sound 类型系统 + dependent types（一定程度）；TS 都没有
  • Haskell / Rust 有显式所有权 / 内存管理（Rust）/ 纯函数（Haskell）；TS 用 JS 的 GC + 不强制纯函数
  • Haskell / Rust 学习曲线陡；TS 学习曲线相对平缓

来自 Haskell / Rust 的工程师常对 TS 表达力满意——条件类型 + 模板字面量类型让 TS 能做很多类型层操作（虽然没 dependent types 的极致表达力）。

vs JavaScript（无类型）

这是最关键的对比。TS 不是"更好的 JS"——是"在 JS 之上加类型注解"。运行时它就是 JS。

所有 TS 优势（自动补全 / 重构安全 / 文档化 API）都来自编译期类型 check。运行时 TS 与 JS 没区别。这就是为什么2025 年大型项目几乎都用 TS——零运行时开销 + 编译期保护。

TC39 Type Annotations 提案：TS 类型语法可能进 JS 标准

2022 年 Microsoft TS 团队 + Igalia 推动 TC39 启动 "Type Annotations as Comments" 提案。核心想法：让 JS 引擎忽略类型注解（不检查、不报错），把 let x: number = 1 当 let x = 1 跑。

提案动机：

  • 让浏览器原生跑 TS 文件（不需要先 transpile）—— 简化 dev / 教学场景
  • 让 TS 类型语法成事实标准—— 任何工具都能 parse
  • 不把 TS 类型 checker 纳入引擎——保持引擎简单

提案现状（2025-04）：Stage 1。进展缓慢——TC39 内部对"把单一公司的语法纳入标准"有顾虑。Brendan Eich 等人持谨慎态度。但Microsoft 投入大——他们想让 TS 更深嵌入 JS 标准。

如果通过的影响：

  • 浏览器原生支持 TS 语法（虽不 check）—— 教学 / dev 简化
  • TS 类型工具更碎片化—— 现在 tsc 是唯一权威 checker，未来可能有多个 checker（tsgo / esbuild-check / ...）
  • "JS = 弱类型"的传统印象彻底改变—— 类型成为 JS 的"一等可见"概念
📊 TS 设计哲学一句话总结

TS 不追求"最强类型系统"——追求"最适合 JS 生态的类型系统"。务实优先 / 渐进采用 / 不 sound / 类型擦除——这 4 个看似"妥协"的决策，让 TS 在 12 年里击败所有竞争对手。这种"务实击败学术"的胜利是软件工程史上的经典案例。

03 应用场景

场景 A · 解释"为什么 TS 这样妥协"

来自 Java / Haskell / Rust 的工程师初学 TS 常困惑"为什么这么宽松？为什么 any 这么宽？"。理解 TS 的 4 个设计选择（结构化 / 擦除 / 不 sound / 渐进）后，这些"妥协"都有具体根因——为了兼容 JS 生态。"妥协"不等于"差"——是务实选择。

场景 B · 选 TS 还是 JSDoc + tsc

判据：(1) 项目是用户应用还是工具链 / 库？应用用 TS 主流；工具链优先 JSDoc（不需要 build step）。(2) 项目会被嵌入吗？如 V8 / Node 自身代码、Vite / esbuild 等——倾向 JSDoc。(3) 团队偏好如何？JSDoc 写法稍冗长但可读性更高。Vite / Node 都从 TS 迁回了 JSDoc + tsc。

场景 C · 评估"换语言"诱惑（再次）

团队有人提"换 ReScript / Hegel / Flow"。大概率不应该换——所有 TS 的同代人都因不能渐进而萎缩。除非：(1) 整个项目从零开始；(2) 团队全员熟悉目标语言；(3) 不需要大量 npm 生态。这三个条件全满足才考虑——大多数情况只满足 1 个，所以保持 TS。

场景 D · 关注 Type Annotations 提案进展

如果 2026-2030 提案进 Stage 4，影响生态最大的事件——之后浏览器原生跑 TS 语法。即使提案不通过，TS 已经是事实标准。不需要因这个提案改变现有项目——但值得关注 TC39 进展。

04 典型代码

typescript 
// TS 4 个设计选择的具体体现

// ① 结构化类型 - 形状匹配即兼容
interface Named { name: string }
function greet(x: Named) { console.log(x.name); }
greet({ name: 'Alice', age: 30 });  // ✅ 多余字段也 OK（变量场景）

// ② 类型擦除 - 编译后无类型
type User = { id: number };
const u = {} as User;
console.log(typeof u);  // 'object' 不是 'User'

// ③ 不 sound - any 兜底 + 协变变体
const arr: any[] = [1, 'two', { three: 3 }];
arr.push(null);  // ✅ 完全合法

// ④ 渐进采用 - JS 文件加 @ts-check
// @ts-check
/** @param {string} name */
function hello(name) {
  console.log(name.toUpperCase());  // tsc 会 check 类型
}

05 常见陷阱

"TS 是带类型的 JS"——简化但有用

更准确：TS 是 JS 超集 + 编译期类型检查 + 类型擦除。运行时没有类型——擦除后就是普通 JS。

"any 是 TS 设计错误"——不，是设计选择

没有 any 就没法渐进采用，没法兼容 npm 生态。any 是 TS 击败 Flow / Hegel 的根本原因。但生产代码应该尽量避免 any——逃生口不是日常写法。

"运行时类型 check 用 TS 就行"——错

TS 运行时没有类型。要 runtime 验证用 zod / yup / class-validator / ts-runtime 等运行时类型库。

"Flow 比 TS 更 sound 所以更好"——理论对，实际不

更 sound 意味着更严，更严意味着更难兼容 JS 生态。Flow 已经被市场用脚投票——选 TS。

TC39 Type Annotations 提案不是"TS 进 JS 标准"

提案只让 JS 引擎"忽略"类型注解（不检查），TS checker 仍然在编译期跑。这与"TS 类型成为 JS 标准"完全不同。
