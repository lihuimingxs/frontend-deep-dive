19 · 项目配置

tsconfig.json 有 100+ 选项。本章把常用 30+ 个按家族分组讲：项目结构、strict（13 子 flag）、module（输出 vs 解析）、output、JSX、lib/types、Project References。每个选项都给出"它解决什么工程问题"。最后给出 4 套可直接复用的模板（Vite / Next.js / Node ESM / Library）。

阅读时长 ~40 分钟 前置 13 Modules + 18 Declaration Files 对应 typescriptlang.org tsconfig 参考

01 速览

tsconfig 的最小有效形态（适合 Vite / Next.js）：

json 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "noEmit": true,
    "jsx": "react-jsx",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true,
    "moduleDetection": "force"
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}

每个 tsconfig 都做三件事：选项（compilerOptions）、包含哪些文件（include / files / exclude）、继承谁（extends）。

02 底层逻辑

1. 项目结构：files / include / exclude / extends

json 
{
  "extends": "@my/tsconfig/base.json",   // 继承基础配置
  // 5.0+ 支持数组多继承（后写覆盖前写）：
  // "extends": ["@my/tsconfig/base.json", "@my/tsconfig/strict.json"],

  "files": ["src/index.ts"],            // 显式列文件（少用）

  "include": [                          // glob 模式
    "src/**/*.ts",
    "src/**/*.tsx",
    "types/**/*.d.ts"
  ],

  "exclude": [                          // 排除（默认含 node_modules）
    "node_modules",
    "dist",
    "**/*.test.ts"
  ]
}

关键细节：

  • files 和 include 至少存在一个，否则 tsc 报错
  • exclude 不影响"被代码 import 的文件"（即使在 exclude 里，import 它仍会编译）
  • extends 数组继承（5.0+）按从前到后顺序，后者覆盖前者；适合做"base + 严格度叠加"

2. strict 家族：13 个独立 flag 的总开关

"strict": true 等价于打开下面这一组：

flag作用关掉的代价
noImplicitAny 禁止隐式 any 到处都是 any 黑洞
strictNullChecks null / undefined 必须显式 放弃 90% null 安全
strictFunctionTypes 函数参数严格逆变（仅函数类型） method 不安全口子继续开
strictBindCallApply bind/call/apply 检查参数 可传任意参数
strictPropertyInitialization 类字段必须初始化或 ? 构造器后字段可能 undefined
alwaysStrict emit "use strict" —
useUnknownInCatchVariables catch (e) 推 unknown 而非 any e.x 直接通过
noImplicitThis 禁止 this 隐式 any this 传染 any
strictBuiltinIteratorReturn 5.6+：内置迭代器 return 类型严格 —

外加（不在 strict 里但常配套打开）：

  • noUncheckedIndexedAccess —— arr[i] 推 T | undefined。痛苦但收益巨大
  • exactOptionalPropertyTypes —— { foo?: string } 不允许显式 undefined
  • noImplicitReturns / noFallthroughCasesInSwitch —— 控制流检查
  • noUnusedLocals / noUnusedParameters —— ESLint 也能管，但 tsc 内置
  • allowUnreachableCode / allowUnusedLabels —— 默认 undefined（warn），设 false 严格

实用建议：新项目直接 strict: true，按需关单个 flag。老项目从 noImplicitAny + strictNullChecks 起步，再逐项叠加。

3. module 家族：target / module / moduleResolution

这三个最容易混淆 — 它们是三件不同的事：

选项决定常用值
target 降级到哪个 ES 版本（语法） ES2017 / ES2020 / ES2022 / ESNext
module 输出哪种 module 格式 CommonJS / ESNext / NodeNext / Preserve
moduleResolution 如何解析 import 找文件 Node10 / Node16 / NodeNext / Bundler

典型组合：

  • Vite / Next.js：module: "ESNext", moduleResolution: "Bundler"
  • Node 22 ESM：module: "NodeNext", moduleResolution: "NodeNext"
  • Node CJS 老库：module: "CommonJS", moduleResolution: "Node10"
  • 跨平台 lib：module: "ESNext", moduleResolution: "Bundler" + 用 tsup / unbuild 输出双格式

相关辅助选项：

  • esModuleInterop + allowSyntheticDefaultImports：CJS / ESM 互通垫片（13 章）
  • isolatedModules：让 babel/swc/esbuild 单文件 transpile 安全。用 bundler 时必开
  • verbatimModuleSyntax（5.0+）：替代旧 importsNotUsedAsValues。要求所有仅类型 import/export 显式标 type
  • moduleDetection: "force"：所有 .ts 都按 module 处理（避免 13 章讲的"全局脚本"陷阱）
  • resolveJsonModule：允许 import .json 文件
  • resolvePackageJsonExports / resolvePackageJsonImports：尊重 package.json 的 exports / imports 字段

4. paths 与 baseUrl

json 
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@app/*": ["apps/app/src/*"],
      "@shared": ["packages/shared/src/index.ts"]
    }
  }
}

⚠️ paths 只影响 tsc 类型检查，不影响运行时。Vite / webpack / Next.js 各自必须配 alias。否则编译过运行 module not found：

javascript · vite.config.js 
import { defineConfig } from "vite";
import path from "path";

export default defineConfig({
  resolve: {
    alias: {
      "@": path.resolve(__dirname, "src"),
      "@app": path.resolve(__dirname, "apps/app/src"),
    },
  },
});

有专门工具从 tsconfig 同步到 bundler：vite-tsconfig-paths / unplugin-tsconfig-paths。Next.js 自动读取 tsconfig 的 paths（不需要 next.config.js 手动配）。

5. output 家族

选项作用
outDir编译产物目录（如 "dist"）
rootDir源根目录（影响产物层级）
declaration生成 .d.ts
declarationMap生成 .d.ts.map（IDE 跳转）
sourceMap生成 .js.map（debug）
inlineSourceMapsource map 内联到 .js
noEmit不输出文件（仅 type check）
noEmitOnError错误时不输出（生产构建建议开）
removeComments产物去注释
incremental增量编译（缓存到 .tsbuildinfo）
tsBuildInfoFile缓存路径自定义
composite开启 Project References（强制 declaration）

6. lib / types：精确控制运行时类型可见

lib 决定 TS 知道哪些标准库 API 存在。默认值由 target 决定（如 target ES2020 自动含 ES2020 所有 lib）：

json 
{
  "compilerOptions": {
    // 浏览器 + ES2022（不含 Node 的 process / Buffer 等）
    "lib": ["DOM", "DOM.Iterable", "ES2022"],

    // Node 项目（不含 DOM 的 window / document）
    // "lib": ["ES2022"]
  }
}

可选的常用 lib：

  • DOM / DOM.Iterable —— 浏览器
  • ES2015 / ES2017 / ES2020 / ES2022 / ESNext —— ES 版本
  • WebWorker —— Worker 上下文（不含 window）
  • ScriptHost —— 老的 Windows Script Host（极少用）

types：白名单加载 @types/*（详见 18 章）：

json 
{
  "compilerOptions": {
    "types": ["node", "vite/client", "@testing-library/jest-dom"]
    // 仅这几个 @types 包被加载（默认是全部加载）
  }
}

7. JSX 选项

16 章已展开。tsconfig 相关：

json 
{
  "compilerOptions": {
    "jsx": "react-jsx",            // preserve | react | react-jsx | react-jsxdev | react-native
    "jsxImportSource": "react",     // 自定义 runtime（@emotion/react / preact 等）
    "jsxFactory": "h",              // jsx: react 时使用的函数（默认 React.createElement）
    "jsxFragmentFactory": "Fragment"
  }
}

8. Project References：大型 monorepo 增量构建

标准 tsc 是整体重编译。一个文件改动会触发所有依赖它的文件重检。在大型 monorepo（数百 package）这是不可接受的。

Project References 把项目拆成多个子项目，每个有独立 tsconfig + 独立产物（带 .tsbuildinfo 缓存）。tsc -b 按依赖顺序构建，未变文件直接走缓存：

json · 项目根 tsconfig.json 
{
  "files": [],
  "references": [
    { "path": "./packages/shared" },
    { "path": "./packages/agent-core" },
    { "path": "./apps/web" }
  ]
}
json · packages/shared/tsconfig.json 
{
  "compilerOptions": {
    "composite": true,         // ← 必须，开启 Project References 模式
    "declaration": true,        // 强制 — 让其他 package 能拿到类型
    "declarationMap": true,
    "outDir": "dist",
    "rootDir": "src"
  },
  "include": ["src"]
}
json · apps/web/tsconfig.json 
{
  "compilerOptions": {
    "composite": true,
    "outDir": "dist"
  },
  "references": [
    { "path": "../../packages/shared" },
    { "path": "../../packages/agent-core" }
  ],
  "include": ["src"]
}
bash 
# 构建（按依赖顺序）
tsc -b

# 仅类型检查
tsc -b --noEmit   # 5.6+ 支持
# 或
tsc -b --dry      # 显示要构建什么

# 清理缓存
tsc -b --clean

# Watch 模式
tsc -b --watch

一旦配好，构建速度可以从分钟级降到秒级。代价：每个 package 都得有独立 tsconfig，初始 setup 较麻烦。Bun / pnpm / Nx 等 monorepo 工具有自己的 incremental 机制，简单项目不一定要 Project References。

noUncheckedIndexedAccess 的实际影响

typescript 
// noUncheckedIndexedAccess: true

const arr = ["a", "b", "c"];
arr[0].toUpperCase();   // ❌ 'arr[0]' is possibly 'undefined'

// 必须显式处理
const first = arr[0];
if (first) first.toUpperCase();   // ✓
arr[0]?.toUpperCase();              // ✓ 更简洁

// Map / Record 同理
const map: Record<string, number> = { a: 1 };
const v = map["b"];                 // v: number | undefined
v.toFixed(2);                        // ❌ 必须先判 undefined

// 这个开关让数组 / 字典访问完全反映 JS 真实行为
// 痛苦但准确 — 推荐打开

05 常见陷阱

陷阱 1 · paths 在运行时不生效
typescript 
// tsconfig.paths: { "@/*": ["src/*"] }
import { foo } from "@/utils";   // ✓ tsc 识别

// 编译产物：
// const { foo } = require("@/utils");
// 运行时：Cannot find module '@/utils'
//
// 因为 tsc 不会 rewrite 路径！

规避：

  • 用 bundler（Vite / webpack / Next）配 alias，让运行时也能解析
  • 用 tsc-alias 在 build 后 rewrite
  • 用 tsconfig-paths 在 Node 启动时 register hook
  • 用 tsx / bun 直接跑 .ts，它们原生支持 paths
陷阱 2 · target ES5 + 现代 API

01 章已讲。target 只降语法不引入 polyfill：

typescript 
// tsconfig: { target: "ES5", lib: ["ES2020"] }
// 编译过：lib 让 TS 知道 Promise / Map 类型
// 但产物保留 new Promise(...) — 老浏览器没 Promise 运行时炸

规避：现代项目 target: "ES2022"（覆盖 95%+ 浏览器，2024+）。需要老浏览器用 babel + core-js 注入 polyfill。

陷阱 3 · composite + 大量循环依赖

开 composite: true 后，Project References 强制 declaration: true。循环依赖会让 tsc 编译爆炸：

bash 
error TS6310: Referenced project '...' may not disable emit

规避：拆 monorepo 时把"shared types"单独成一个零依赖 package；其他 package 仅类型层 import 它（用 import type），运行时无依赖。

陷阱 4 · skipLibCheck: false 让第三方 .d.ts 错误也报

默认 skipLibCheck: true 跳过 node_modules/**/*.d.ts 的检查。关掉后任何第三方库 .d.ts 错误都会报到你的项目里：

bash 
node_modules/some-pkg/dist/index.d.ts(42,5): error TS2403:
  Subsequent variable declarations must have the same type...
# 你的代码完全没动 — 但因为 some-pkg 的 .d.ts 有错就编译失败

规避：永远 skipLibCheck: true。第三方 .d.ts 错由库自己负责。

陷阱 5 · isolatedModules 禁止某些写法

isolatedModules: true 让 babel/swc 单文件 transpile 安全，代价是禁止某些"跨文件类型信息"的写法：

typescript 
// ❌ const enum
// 单文件 transpile 时不知道 enum 值，无法内联
const enum Color { Red, Green }
console.log(Color.Red);

// ✅ 改普通 enum 或 as const union
const Color = { Red: 0, Green: 1 } as const;
type Color = typeof Color[keyof typeof Color];

// ❌ 不带 type 的"仅类型 export"
import { Helper } from "./helper";
export { Helper };   // ⚠️ 如果 Helper 仅用作类型，这里也要标 type

// ✅
export type { Helper } from "./helper";
陷阱 6 · verbatimModuleSyntax 强制 import type
typescript 
// verbatimModuleSyntax: true 下

import { User } from "./types";   // ❌ 'User' is a type-only import; emit will preserve it
//                                   编译后会保留 import "./types"，触发副作用

// ✅
import type { User } from "./types";
// 或
import { type User } from "./types";   // inline type modifier

规避：项目级开 verbatimModuleSyntax 后用 codemod 工具（ts-morph / jscodeshift）一次性把所有仅类型 import 加 type。
