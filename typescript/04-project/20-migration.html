20 · Migration & Interop

这是整个手册的收官章。整合前 19 章语言知识 + 大量工具链经验，回答四个核心问题：怎么把老 JS 项目分阶段迁移到 TS、怎么处理 CJS / ESM 双世界互通、怎么在「编译期类型」与「运行时安全」之间画好接缝、TS 5.x 给工程化带来了什么新工具。最后给出一份覆盖 Vite / Next / Bun / Node 的工具链对比。

阅读时长 ~35 分钟 前置 前 19 章全部 对应 typescriptlang.org Migration / Tooling 综合

01 速览

四条工程化"硬规则"，每条都对应一种最常见的踩坑：

  1. 类型系统不是运行时验证器 — fetch / JSON.parse / process.env / 用户输入必须 zod 兜底
  2. 用 @ts-expect-error，不用 @ts-ignore — 错误修复后会主动告警
  3. bundler-only 项目必开 isolatedModules — babel/swc/esbuild 的安全前提
  4. CJS/ESM 互通靠 esModuleInterop + nodenext + 包的 exports 字段三件套
typescript · 信任边界示意 
// 编译期类型（TS 范围）
async function fetchUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  const data: unknown = await res.json();      // ← 跨边界 — 标 unknown
  return UserSchema.parse(data);                // ← 运行时校验（zod）
}                                                //   tsc 信任边界内，zod 守护边界外

02 底层逻辑

1. JS → TS 迁移的三个阶段

直接把所有 .js 改 .ts 风险大、收益慢。推荐分三阶段：

阶段目标tsconfig 关键设置
P1 共存 JS / TS 共存编译，不强制 .js 改 .ts allowJs: true, checkJs: false
P2 软检查 所有 JS 也按 TS 类型检查（仅 noImplicitAny） checkJs: true, noImplicitAny: true
P3 强类型 全 .ts，开 strict，逐项打开 noUncheckedIndexedAccess 等 strict: true

配合 // @ts-check 做单文件试点：

javascript · src/util.js 
// @ts-check

/** @typedef {{ id: string, name: string }} User */

/** @param {User[]} users */
function listIds(users) {
  return users.map(u => u.id);
}

// @ts-check 让单个 JS 文件接受 TS 检查，配合 JSDoc 注解。这种模式让你在不改后缀名的情况下渐进改类型 —— Vue / svelte / vscode 等大型项目都用过这一阶段。

2. @ts-expect-error 优于 @ts-ignore

两个看似等价的错误屏蔽指令，实际行为差异巨大：

@ts-ignore@ts-expect-error
下一行有错时静默静默
下一行修好后仍静默（注释残留）主动报警："不需要"
用途已不推荐明确"我知道这是错，故意保留"
typescript 
// ❌ @ts-ignore：错误修了仍残留注释
// @ts-ignore - was failing
const x: string = "abc";   // 现在已经对了 — 但 @ts-ignore 还在，永远没人发现

// ✅ @ts-expect-error：错误消失会主动报警
// @ts-expect-error - intentionally wrong for legacy reasons
const y: string = 123;     // 修好后 TS 报"Unused @ts-expect-error directive"
                            // 提醒你删除注释

规则：

  • 新代码永远用 @ts-expect-error
  • 每个 expect-error 必须有注释说明原因（"何时能修复"或"为什么必须保留"）
  • 项目里 grep @ts-ignore，逐个改成 @ts-expect-error 或修代码
  • ESLint 可强制：@typescript-eslint/ban-ts-comment（默认禁 @ts-ignore）

3. noEmit + bundler：现代前端工程的职责切分

老 TS 项目用 tsc 同时做 type check + 转译。现代前端工程把这两件事拆开：

  • 类型检查：tsc 跑 --noEmit，仅做 check 不输出
  • 转译：bundler（Vite / esbuild / SWC / Babel）负责 .ts → .js

为什么这样切？因为 bundler 比 tsc 快 10-50 倍（不做类型检查），开发体验好；类型检查可以放 CI 或 pre-commit 单独跑。

json · package.json 
{
  "scripts": {
    "dev": "vite",                          // bundler 跑，不做 typecheck
    "build": "vite build",                  // bundler build，不做 typecheck
    "typecheck": "tsc --noEmit",            // tsc 单独 typecheck
    "ci": "bun typecheck && bun build"      // CI 串起来
  }
}

dev 时怎么知道类型错？两种方案：

  • IDE（VS Code）实时类型检查 —— 90% 场景够用
  • 用 tsc-watch 或 vite 的 vite-plugin-checker 在 dev server 内同步跑 typecheck

关键 tsconfig：bundler-only 模式必须开 isolatedModules: true。这告诉 TS 你的代码会被单文件 transpile，要禁止跨文件类型依赖（如 const enum）。

4. CJS / ESM 双世界深坑

Node 生态最痛苦的双格式问题。一图看完：

CommonJSES Module
语法require / module.exportsimport / export
加载同步 / 运行时异步 / 静态
tree-shake不可可
__dirname / __filename原生需要 import.meta.url 转换
动态 importrequire(path)import(path) 返 Promise
top-level await✗✓

决定哪种 module：

  • package.json 的 "type": "module" → 整个包按 ESM
  • "type": "commonjs" 或不设 → 按 CJS
  • 文件后缀 .mjs 强制 ESM；.cjs 强制 CJS（覆盖 type）
  • TS 中：.mts → .mjs；.cts → .cjs

5. ESM-only 包在 CJS 项目里炸

很多新库（chalk@5+、node-fetch@3+、got@12+）已经是ESM-only。在 CJS 项目里 require() 它会失败：

javascript · CJS 项目 
const chalk = require("chalk");   // ❌ Error [ERR_REQUIRE_ESM]: require() of ES Module not supported

规避方案：

  • 方案 A：项目升 ESM（"type": "module" + 改 import）
  • 方案 B：用动态 import包裹（仍然 CJS 项目，但能 await import("chalk")）
  • 方案 C：用旧版本的 ESM-only 库（chalk@4 是 CJS）
typescript · CJS 项目用动态 import 
async function colorize(s: string) {
  const chalk = (await import("chalk")).default;
  return chalk.red(s);
}

6. nodenext 模式的严格规则

Node 22 + ESM 推荐的 tsconfig：module: "NodeNext" + moduleResolution: "NodeNext"。但它强制几条规则：

  • import 路径必须带 .js 后缀（即使源是 .ts）
  • 不允许 import x = require(...) 等 CJS 语法
  • 动态 import / top-level await 可用
  • 使用 import.meta.url 替代 __dirname / __filename
typescript · NodeNext 模式 
// ❌ 报错：Relative import paths need explicit file extensions
import { foo } from "./util";

// ✅ 必须加 .js（不是 .ts）
import { foo } from "./util.js";

// __dirname 替代
import { fileURLToPath } from "node:url";
import path from "node:path";
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);

7. 信任边界：类型 vs 运行时校验

这是 TS 工程最重要的元规则：类型系统在内部代码可信，所有"信任边界"必须运行时校验。

边界位置原因校验工具
fetch().json()服务端字段可能变zod / valibot
JSON.parse()外部输入zod schema
process.env.X实际是 string | undefinedzod 启动时校验
URL params / query string用户可改zod 在 router
postMessage / Worker message跨边界zod / discriminated union
第三方 SDK 返回类型可能不准关键路径加 schema
localStorage / IndexedDB 读取序列化数据zod
typescript · process.env 启动校验 
// 启动时跑一次，缺少必填环境变量直接 crash
import { z } from "zod";

const env = z.object({
  DATABASE_URL: z.string().url(),
  JWT_SECRET: z.string().min(32),
  NODE_ENV: z.enum(["development", "production", "test"]),
  PORT: z.coerce.number().int().positive().default(3000),
}).parse(process.env);

// 之后所有代码用 env 而非 process.env
export { env };

// 用法
import { env } from "./env";
const conn = await connect(env.DATABASE_URL);   // env.DATABASE_URL: string ✓

9. using declaration（5.2）— 资源自动释放

ES Stage 3 提案，TS 5.2 实现。语义类似 Python 的 with / C# 的 using：变量出作用域自动调 [Symbol.dispose]()。

typescript 
class FileHandle {
  constructor(public fd: number) {}
  [Symbol.dispose]() {
    console.log(`closing fd ${this.fd}`);
    // 实际关文件
  }
}

function readFile() {
  using f = new FileHandle(42);
  // ... 用 f
  return f.fd;
}   // 函数结束 / throw / return — 自动调 f[Symbol.dispose]()

// async 资源用 await using
class Connection {
  async [Symbol.asyncDispose]() {
    await this.close();
  }
}

async function query() {
  await using conn = await openConnection();
  return conn.query("SELECT 1");
}   // 自动 await close

解决了"打开了忘记关"这一类历史最常见 bug。Node 22+ 原生支持 Symbol.dispose。库（如数据库 driver、文件 stream）开始适配。

03 应用场景

场景 A · 老 Express 项目分阶段迁移

  1. 第一周：搭基础
    • 装 typescript、@types/node、@types/express
    • 写 tsconfig：allowJs: true, checkJs: false, outDir: "dist", target: "ES2020"
    • scripts 加 "build": "tsc"、"typecheck": "tsc --noEmit"
  2. 第二周：试点
    • 选一个独立 utility 模块，加 // @ts-check
    • 用 JSDoc 给关键函数加类型
    • 修出现的 implicit any
  3. 第三-N 周：扩面
    • 从叶子节点（utility / model）开始改 .ts
    • 逐个文件，先编译过再说
    • 第三方库无类型时写 declare module "x"; 兜底
  4. 稳定后：提升严格度
    • 开 noImplicitAny → 开 strictNullChecks → 开 strict
    • 开 noUncheckedIndexedAccess 是最痛的一步，但收益最大

场景 B · Monorepo CJS / ESM 混合

一个真实场景：你的 monorepo 里

  • shared utility 包是 ESM（"type": "module"）
  • web app 是 ESM（Vite + React）
  • legacy CLI 是 CJS（用 commander 等老依赖）

CLI 怎么用 shared 包？方案：shared 包发布双格式。package.json：

json · packages/shared/package.json 
{
  "name": "@my/shared",
  "type": "module",
  "main": "./dist/index.cjs",        // CJS 入口
  "module": "./dist/index.mjs",      // ESM 入口
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}

用 tsup 打两份产物：

bash 
tsup src/index.ts --format esm,cjs --dts --clean

场景 C · Next.js 16 端到端类型安全

场景 D · process.env 启动校验（信任边界）

typescript · src/env.ts 
import { z } from "zod";

const EnvSchema = z.object({
  NODE_ENV: z.enum(["development", "production", "test"]),
  DATABASE_URL: z.string().url(),
  JWT_SECRET: z.string().min(32),
  REDIS_URL: z.string().url().optional(),
  PORT: z.coerce.number().int().positive().default(3000),

});

if (!result.success) {
  console.error("❌ Invalid environment variables:");
  console.error(result.error.format());
  process.exit(1);

这种"启动即校验"模式让缺失的环境变量在启动期就暴露 — 而不是某个特殊代码路径才发现。生产事故防御黄金法则之一。

typescript · src/env.ts（生产推荐） 
import { z } from "zod";

// 区分 server / client 环境
const ServerEnvSchema = z.object({
  NODE_ENV: z.enum(["development", "production", "test"]),
  DATABASE_URL: z.string().url(),
  JWT_SECRET: z.string().min(32),
  STRIPE_SECRET_KEY: z.string().startsWith("sk_"),
});

const ClientEnvSchema = z.object({
  NEXT_PUBLIC_API_URL: z.string().url(),
  NEXT_PUBLIC_GA_ID: z.string().optional(),
});

function parseEnv<T>(schema: z.ZodType<T>, source: NodeJS.ProcessEnv): T {
  const result = schema.safeParse(source);
  if (!result.success) {
    console.error("❌ Invalid env:", result.error.format());
    throw new Error("Environment validation failed");
  }
  return result.data;
}

export const serverEnv = typeof window === "undefined"
  ? parseEnv(ServerEnvSchema, process.env)
  : ({} as z.infer<typeof ServerEnvSchema>);   // 浏览器侧用空对象（仅类型）

export const clientEnv = parseEnv(ClientEnvSchema, process.env);

一份覆盖大多数场景的 helper：safeFetch

typescript 
import { z } from "zod";

type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

export async function safeFetch<T>(
  url: string,
  schema: z.ZodType<T>,
  init?: RequestInit,
): Promise<Result<T, "network" | "validation" | "http">> {
  try {
    const res = await fetch(url, init);
    if (!res.ok) return { ok: false, error: "http" };
    const data: unknown = await res.json();
    const parsed = schema.safeParse(data);
    if (!parsed.success) return { ok: false, error: "validation" };
    return { ok: true, value: parsed.data };
  } catch {
    return { ok: false, error: "network" };
  }
}

// 用法
const UserSchema = z.object({ id: z.string(), name: z.string() });
const r = await safeFetch("/api/user", UserSchema);
if (!r.ok) {
  if (r.error === "validation") return "Server schema changed";
  return "Network or HTTP error";
}
return r.value.name;

using declaration 实战：自动关闭资源

typescript · TS 5.2+ 
// 实现一个"用完自动断"的 lock
class Mutex {
  private locked = false;
  async acquire() {
    while (this.locked) await new Promise(r => setTimeout(r, 1));
    this.locked = true;
    return {
      [Symbol.dispose]: () => { this.locked = false }
    };
  }
}

const mutex = new Mutex();

async function critical() {
  using lock = await mutex.acquire();
  // 锁住执行
  await doStuff();
}   // 函数结束 — 自动调 lock[Symbol.dispose]() 释放

05 常见陷阱

陷阱 1 · 升级 ESM 后忘 .js 后缀

13 章已讲。NodeNext / nodenext 模式下所有 relative import 必须 .js 后缀（即使源是 .ts）。

规避：用 tsc-alias 自动加；或全局 codemod。

陷阱 2 · @ts-ignore 在 codebase 里"积灰"

老项目里 grep 出几十个 @ts-ignore 完全正常。但每个都是定时炸弹 — 错误已修复但注释残留，下次代码改动时类型错重新出现却被屏蔽。

规避：

  • 开 ESLint @typescript-eslint/ban-ts-comment: "error"
  • 批量 codemod：把所有 @ts-ignore 换成 @ts-expect-error + 注释 "TODO: investigate"
  • 下个迭代删掉所有 expect-error 显示无效的注释（说明错误已修）
陷阱 3 · process.env.X 当 string 用
typescript 
// 默认 process.env.X 类型是 string | undefined
const dbUrl = process.env.DATABASE_URL;
//    ^? string | undefined

// ❌ 直接用
fetch(dbUrl);   // ❌ string | undefined 不能传给 string

// ❌ 强转 — 缺时运行时炸
fetch(dbUrl as string);
fetch(dbUrl!);

// ✅ 启动校验
if (!process.env.DATABASE_URL) throw new Error("DATABASE_URL required");
fetch(process.env.DATABASE_URL);

// ✅ 更好：用 zod env helper（见上文）
陷阱 4 · tsc 编译过 + bundler 编译炸

出现这种"双系统不一致"通常是 tsc 和 bundler 用了不同的 module 解析规则：

  • tsconfig moduleResolution: "Node10" 但 bundler 走 ESM 解析 — 路径解析差异
  • tsc 接受 const enum，bundler 不能跨文件分析
  • tsc 看得见 .ts，bundler 看不见（要 .js 后缀）
  • 类型与运行时 import 错配（verbatimModuleSyntax 没开）

规避：tsconfig 用 moduleResolution: "Bundler" + isolatedModules: true + verbatimModuleSyntax: true，让 tsc 行为对齐 bundler。

陷阱 5 · 老库 module.exports = function 配 esModuleInterop
typescript 
// CJS 老库
// node_modules/old-lib/index.js:
//   module.exports = function () { ... };

// 不开 esModuleInterop
import oldLib = require("old-lib");
oldLib();   // ✓

// 开 esModuleInterop
import oldLib from "old-lib";
oldLib();   // ⚠️ 取决于 TS 版本和 .d.ts 写法 — 可能 oldLib 是 { default: fn } 而不是 fn
// 实际行为：
// const oldLib = require("old-lib").default ?? require("old-lib");
// 但如果 .d.ts 没 export default，可能拿到 undefined

规避：第三方库返 function 时，看 .d.ts 是怎么 export 的。多数情况 import * as fn from "old-lib" 最稳。或者写自己的 .d.ts patch。

陷阱 6 · Bun / tsx 直接跑 .ts 跳过类型检查

bun run src/index.ts 和 tsx src/index.ts 都不做类型检查，仅快速 transpile 跑代码。这意味着类型错误在生产前不会被发现。

规避：

  • CI 必跑 tsc --noEmit
  • pre-commit hook 跑 typecheck（用 lefthook / husky）
  • IDE 实时检查（VS Code）—— 但只在打开的文件生效
