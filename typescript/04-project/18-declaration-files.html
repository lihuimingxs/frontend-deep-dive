18 · Declaration Files

.d.ts 文件是 TS 与"未类型化的 JS 世界"之间的桥。它只含类型、不含值实现，编译时不输出。本章覆盖：消费 @types/*、写 ambient declaration、module declaration（给无类型库加类型）、通配 module（给 svg/css/png import 加类型）、global augmentation（给 Window / process.env 加字段）、最后是发布带类型的 npm 包。

阅读时长 ~28 分钟 前置 13 Modules + 15 Declaration Merging 对应 typescriptlang.org Declaration Files

01 速览

.d.ts 的三大用途：

  1. 给无类型 JS 库写类型（自己写或下载 @types/foo）
  2. 给资源 import 加类型（svg / css module / png 等 webpack/Vite 资源）
  3. 扩展全局 / 第三方库类型（Window 加字段 / Express Request 加 user）
typescript · 最小例子 
// types/legacy-lib.d.ts
declare module "legacy-lib" {
  export function helper(s: string): number;
  export const VERSION: string;
}

// 之后任何代码都能 import
import { helper } from "legacy-lib";   // ✓ 类型化

关键约束：.d.ts 内不能写运行时代码。所有 declaration 都用 declare 关键字。tsc 不会输出 .d.ts（它本身就是类型）。

02 底层逻辑

1. .d.ts 的"事实承诺"性质

declare const FOO: string 告诉 TS："我担保运行时存在一个叫 FOO 的 string 变量"。TS 不验证，运行时如果不存在就炸。

typescript 
// global.d.ts
declare const __DEV__: boolean;       // 担保此变量运行时存在（如 webpack DefinePlugin 注入）
declare function gtag(...args: any[]): void;   // 担保此全局函数存在
declare const process: { env: Record<string, string | undefined> };

// 任何代码都能用：
if (__DEV__) {
  gtag("event", "loaded");
}

// 但如果 webpack 没注入 __DEV__ — 运行时 ReferenceError

这种"担保"性质是 .d.ts 的核心特征。它让 TS 能"信任"那些它无法直接验证的运行时实体（global 变量、未类型化库的 export、bundler 注入）。

2. 消费类型：@types / types / typeRoots

@types/foo 是 DefinitelyTyped 仓库提供的类型补丁包。tsc 默认会从 node_modules/@types/* 自动加载所有 @types 包：

bash 
npm install --save-dev @types/express @types/node @types/react

安装后不需要 import @types，TS 自动找到。但有时这种"全局自动加载"不是想要的，可以用以下选项控制：

选项作用
"types": ["node", "express"] 白名单：仅加载这几个 @types 包
"types": [] 禁用所有 @types 自动加载
"typeRoots": ["./types", "./node_modules/@types"] 自定义 @types 目录（默认仅 node_modules/@types）

实际项目大多数都用默认。只有 monorepo 中不同 package 需要不同 @types 时才用 types 隔离。

3. declare module "..."：给无类型库写补丁

一个 npm 包没有自带 .d.ts、@types/foo 也没有时，自己写：

typescript · types/untyped-lib.d.ts 
declare module "untyped-lib" {
  export interface Options {
    timeout?: number;
    retry?: number;
  }
  export function init(opts: Options): Promise<void>;
  export const VERSION: string;
  export default class Client {
    constructor(opts: Options);
    request(path: string): Promise<Response>;
  }
}

这个文件不需要放在 untyped-lib 包内 —— 你的项目自己 types 目录下就行。tsconfig 的 include 包含此目录后，整个项目都能用。

如果只想"勉强能用"（不写完整类型），用极简兜底：

typescript 
// 临时方案：让 import 不报错，类型全 any
declare module "any-lib";
// 等价于把 any-lib 整个模块标记为 any — 后续慢慢补

4. 通配 module：给资源 import 加类型

webpack / Vite / Next.js 让你 import logo from "./logo.svg"，但 TS 默认不知道 .svg 是什么。用通配 module 声明：

typescript · types/assets.d.ts 
// 图片资源
declare module "*.svg" {
  const content: string;     // bundler 把它转成 URL 字符串
  export default content;
}

declare module "*.png" {
  const content: string;
  export default content;
}

// CSS Module
declare module "*.module.css" {
  const styles: { readonly [key: string]: string };
  export default styles;
}

// 用法
import logo from "./logo.svg";       // logo: string ✓
import styles from "./Button.module.css";   // styles: { ... } ✓

通配 module 在资源类型 import 工程中无处不在。Vite 自带的 vite/client 已经声明了所有常见资源（svg / png / json / module.css / ?raw / ?url 等），只要 tsconfig 包含它就行：

json · tsconfig 
{ "compilerOptions": { "types": ["vite/client"] } }

5. declare global + declare module 的边界

15 章已经讲过 augmentation。这里聚焦 .d.ts 中的特殊规则：

  • 没有 import / export 的 .d.ts：被当成"全局脚本"，所有 declaration 直接进全局命名空间
  • 有 import / export 的 .d.ts：被当成模块；要扩展全局必须用 declare global { ... }
typescript · 形态 A：纯全局脚本 
// types/globals.d.ts —— 没有 import / export
declare const __APP_VERSION__: string;
declare function gtag(...args: any[]): void;

interface Window {     // 自动 declaration merge 到 global Window
  myCustomField: string;
}
typescript · 形态 B：模块 + declare global 
// types/globals.d.ts —— 含 import 或 export
import type { User } from "../src/types";

declare global {
  interface Window {
    currentUser?: User;     // 这里会引用上面 import 的类型
  }
  namespace NodeJS {
    interface ProcessEnv {
      DATABASE_URL: string;
    }
  }
}

export {};   // 让此文件是模块（如果上面没有 import 就需要）

6. 发布带类型的 npm 包

发布 TS 库时让用户拿到类型，package.json 必须配置：

json · 简单版（适合纯 CJS / ESM） 
{
  "name": "my-lib",
  "version": "1.0.0",
  "main": "dist/index.js",
  "types": "dist/index.d.ts"   // ← 关键
}

需要双格式（CJS + ESM）时用 exports 字段，types condition 必须放第一：

json · 双格式 
{
  "name": "my-lib",
  "version": "1.0.0",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",     // ← types 第一
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    },
    "./client": {
      "types": "./dist/client.d.ts",
      "import": "./dist/client.mjs",
      "require": "./dist/client.cjs"
    }
  }
}

陷阱：types condition 放在 import / require 之后会被 Node 16+ 模块解析跳过 — 用户拿不到类型。

tsconfig 输出 .d.ts 需要：

json · tsconfig.json 
{
  "compilerOptions": {
    "declaration": true,            // 生成 .d.ts
    "declarationMap": true,          // 生成 .d.ts.map（IDE 跳转源码）
    "outDir": "dist"
  }
}

7. declarationMap：让用户能跳转到源码

打开 declarationMap: true 后，用户的 IDE 在 Cmd+点击你的 export 时，能跳到原始 .ts 源而不是 .d.ts。前提是把 src/ 也打进 npm 包：

json · package.json 
{
  "files": [
    "dist",
    "src"      // ← 同时发布 src，让 declarationMap 能找源
  ]
}

8. ⚠️ 三斜线指令：仅 .d.ts 内偶用

/// <reference> 在 ES module 之前的 TS 用来声明文件级依赖。现代项目几乎不用，仅在 .d.ts 内偶尔用作显式依赖：

typescript · types/index.d.ts 
/// <reference types="node" />       // 这个 .d.ts 依赖 @types/node
/// <reference path="./other.d.ts" />  // 依赖另一个 .d.ts
/// <reference lib="ES2020" />          // 依赖某 lib

declare module "my-lib" {
  // 这里能用 NodeJS.Buffer 等（来自 @types/node）
}

一般用 tsconfig 的 types / lib 配置代替。

03 应用场景

场景 A · 给无类型 JS 工具包写 .d.ts

场景：内部 monorepo 里某个包还没改 .ts，但其他包想用它的类型：

typescript · types/internal-utils.d.ts 
declare module "@internal/utils" {
  export function formatDate(d: Date, locale?: string): string;
  export function parseJSON<T = unknown>(s: string): T | null;

  export interface Logger {
    info(msg: string): void;
    error(msg: string, err?: Error): void;
  }

  export class FileWatcher {
    constructor(path: string);
    on(event: "change", handler: (file: string) => void): void;
    close(): void;
  }
}

场景 B · webpack DefinePlugin 注入的环境变量类型化

typescript · src/global.d.ts 
// 这些变量由 bundler 在编译时静态替换
declare const __APP_VERSION__: string;
declare const __BUILD_TIME__: string;
declare const __FEATURE_FLAGS__: Record<string, boolean>;
declare const __IS_DEV__: boolean;
javascript · webpack.config.js 
new webpack.DefinePlugin({
  __APP_VERSION__: JSON.stringify(pkg.version),
  __BUILD_TIME__: JSON.stringify(new Date().toISOString()),
  __FEATURE_FLAGS__: JSON.stringify(flags),
  __IS_DEV__: process.env.NODE_ENV !== "production",
});

场景 C · 资源类型 import（Vite/webpack）

typescript · src/assets.d.ts 
// 图片
declare module "*.svg" {
  const url: string;
  export default url;
}
declare module "*.svg?react" {
  import { FC, SVGProps } from "react";
  const Component: FC<SVGProps<SVGSVGElement>>;
  export default Component;
}

declare module "*.png" {
  const url: string;
  export default url;
}

// JSON（Vite 默认支持）
declare module "*.json" {
  const json: any;
  export default json;
}

// CSS Module
declare module "*.module.css" {
  const styles: { readonly [key: string]: string };
  export default styles;
}

// 文本资源（Vite ?raw）
declare module "*?raw" {
  const content: string;
  export default content;
}

// 用法
import logo from "./logo.svg";              // logo: string
import LogoIcon from "./logo.svg?react";    // LogoIcon: FC<SVGProps>
import styles from "./Button.module.css";   // styles: { btn: string; ... }
import readmeText from "./README.md?raw";   // readmeText: string

场景 D · 发布 npm 包带类型（双格式）

json · package.json 
{
  "name": "my-utility",
  "version": "1.0.0",
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.mjs",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  },
  "files": ["dist", "src"]
}
json · tsconfig.build.json 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "outDir": "dist",
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "strict": true
  },
  "include": ["src/**/*"]
}

实际打两份产物用 tsup / unbuild / vite build 等工具自动化。tsc 单独跑 tsc -p tsconfig.build.json --noEmit 做类型检查。

04 典型代码

所有 ambient declaration 形态

typescript 
// 变量
declare const PI: number;
declare let userId: string;

// 函数
declare function loadConfig(): Promise<Config>;

// class
declare class Logger {
  constructor(name: string);
  log(msg: string): void;
  static globalLevel: "debug" | "info" | "warn";
}

// interface（与全局合并 / 在模块内独立）
declare interface Config { /* ... */ }

// type alias（与全局合并 / 在模块内独立）
declare type UserId = string;

// namespace（含子结构）
declare namespace MyApp {
  function init(): void;
  const VERSION: string;
  interface Config { /* ... */ }
}

// module
declare module "some-lib" {
  export function fn(): void;
}

// 通配 module
declare module "*.txt" {
  const content: string;
  export default content;
}

// 全局扩展（在模块内）
declare global {
  interface Window { customField: string }
}

给一个无类型库渐进加类型

typescript · types/legacy-pkg.d.ts 
// 第一阶段：兜底，让 import 不报错
declare module "legacy-pkg";
// 整个模块是 any，先解锁编译

// 第二阶段：补主要 API
declare module "legacy-pkg" {
  export function main(): void;
  export const version: string;
  // 其他还没补的字段仍按 any 走（因为整个 module 还有兜底）
}

// 第三阶段：完整类型
declare module "legacy-pkg" {
  export interface Options {
    timeout?: number;
    retries?: number;
  }
  export function main(opts?: Options): Promise<void>;
  export const version: string;
  export class Logger { /* ... */ }
}

module augmentation 给 next-auth Session 加业务字段

typescript · types/next-auth.d.ts 
import type { DefaultSession, DefaultUser } from "next-auth";

declare module "next-auth" {
  interface Session {
    user: {
      id: string;
      role: "admin" | "user";
      tenantId: string;
    } & DefaultSession["user"];
  }

  interface User extends DefaultUser {
    role: "admin" | "user";
    tenantId: string;
  }
}

declare module "next-auth/jwt" {
  interface JWT {
    role: "admin" | "user";
    tenantId: string;
  }
}

从值反向生成 .d.ts（dts-bundle-generator / tsup）

手写 .d.ts 容易和实现脱节。实际项目几乎都用工具从 .ts 源自动生成：

bash 
# tsup（最简单）
tsup src/index.ts --format esm,cjs --dts

# unbuild（rollup-based）
unbuild

# rollup + rollup-plugin-dts
# vite + vite-plugin-dts

tsc 自带的 declaration: true 也能生成，但每个 .ts 一个 .d.ts，没有 bundle。库发布时通常需要 rollup 或专门工具合并。

typeRoots / types 限定加载

json · tsconfig.json 
{
  "compilerOptions": {
    // 默认行为：自动加载 node_modules/@types/*

    // 白名单：仅加载这几个 @types
    "types": ["node", "vite/client", "@testing-library/jest-dom"],

    // 自定义 typeRoots（默认仅 node_modules/@types）
    "typeRoots": ["./types", "./node_modules/@types"],

    // 让 src 下的 .d.ts 也被加载（默认 include 已包含）
    "baseUrl": "."
  },
  "include": [
    "src/**/*",
    "types/**/*"   // 自定义类型目录
  ]
}

05 常见陷阱

陷阱 1 · .d.ts 中误写运行时代码
typescript 
// ❌ .d.ts 不能写实现
declare module "my-lib" {
  export function helper(s: string): number {
    return s.length;   // ❌ Implementation expected
  }
}

// ✅ 仅签名
declare module "my-lib" {
  export function helper(s: string): number;
}
陷阱 2 · declare global 在错误位置不生效

15 章已讲。declare global 必须在模块文件内（含 import 或 export）。否则当成全局脚本时直接写 interface Window 即可，反而不需要 declare global。

陷阱 3 · @types/foo 与库版本不匹配

DefinitelyTyped 维护的 @types 滞后于库本身：库 2.5 可能对应 @types/foo@2.4，新增 API 拿不到类型。

规避：

  • 升级库时同步检查 @types 版本
  • 关键边界（API 客户端、ORM）用 zod 二次验证，不完全信赖 .d.ts
  • 优先选自带 .d.ts 的库（package.json 有 types 字段）
陷阱 4 · 全局类型污染
typescript · src/types.d.ts 
// ❌ 在没有 import/export 的 .d.ts 里写 type
type User = { id: string };
type Post = { id: string };
// 这两个 type 进入全局 — monorepo 中其他包"无意"也能用 User
// 与其他包同名 type 冲突的概率很高

规避：

  • 普通类型放 .ts 文件 + 显式 export，不放 .d.ts
  • 真正想全局的（ambient declaration）才放 .d.ts，且名字加前缀避免冲突
  • tsconfig 开 "moduleDetection": "force" 让所有文件按模块处理
陷阱 5 · package.json exports 中 types condition 顺序
json 
// ❌ 顺序错 — Node 16+ 解析时 types 被忽略
{
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs",
      "types": "./dist/index.d.ts"   // ← 必须在 import / require 之前！
    }
  }
}

// ✅ 正确顺序
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",   // ← 第一
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}
陷阱 6 · 通配 module 与具体 module 冲突
typescript 
// 通配
declare module "*.svg" { const x: string; export default x }

// 具体声明（更优先）
declare module "./assets/special.svg" {
  import { FC } from "react";
  const Component: FC;
  export default Component;
}

// 调用方
import logo from "./assets/logo.svg";          // 走通配 — string
import Special from "./assets/special.svg";    // 走具体 — Component
//                                                ⚠️ 但具体路径 module 不太常用，bundler 行为可能不一致

规避：尽量保持通配统一。需要不同行为的资源用 query 区分（如 Vite 的 ./logo.svg?react）。
