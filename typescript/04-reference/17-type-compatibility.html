17 · Type Compatibility & Inference

这是 TS 类型系统的元层机制 —— 解释"为什么一个类型能赋给另一个类型"。本章深入协变 / 逆变 / 双变 / 不变这套 variance 概念、method bivariance 这个 TS 故意留的不安全口子、strictFunctionTypes 解决了什么、contextual typing 让回调参数能被反推、以及 best common type 在数组字面量推断中的行为。理解这些后回头看 Part 2 也会更深 —— 因为 09 章 conditional type 的 distributive、10 章 mapped type 的 modifier 都基于这套底层规则。

⚠️ 本 Part 最难一章。variance 是函数式 / 库作者必须掌握的元能力，对应用代码作者只需知道直觉法则即可。

阅读时长 ~45 分钟 前置 前 16 章的所有内容（这是综合元层） 对应 typescriptlang.org Type Compatibility + Type Inference

01 速览

一句话总结：TS 是结构化（structural）类型系统。两个类型的兼容性看形状不看名字。

typescript 
interface Named { name: string }
interface User  { name: string; age: number }

const u: User = { name: "Alice", age: 30 };
const n: Named = u;     // ✓ User 的形状包含 Named 要求的所有字段
const u2: User = n;     // ❌ Named 缺 age

但函数 / class / generic 上有更细致的规则，叫 variance（型变）。直觉法则：

  • 输入位置变窄不安全（应"逆变" contravariant）
  • 输出位置变窄安全（可"协变" covariant）

02 底层逻辑

1. 对象兼容性：源至少要有目标的所有 required 字段

typescript 
interface Point2D { x: number; y: number }
interface Point3D { x: number; y: number; z: number }

const p3: Point3D = { x: 0, y: 0, z: 0 };
const p2: Point2D = p3;    // ✓ Point3D 形状是 Point2D 的超集

const p4: Point3D = p2;    // ❌ Point2D 缺 z

可选字段 ? 不强制要求，readonly 不影响赋值方向（02 章讲过）。多余字段在立即字面量赋值时触发 excess property check（这是结构兼容的特殊保护）。

2. 函数参数：逆变（contravariance）

这是 variance 概念中最反直觉的部分。一个法则：

能接受 Animal 的函数，可以替换"期望接受 Dog 的函数"位置。

为什么？想象有一个函数类型 type Handler = (x: Dog) => void。"期望 Dog handler"的代码会传 Dog 给它。如果实际函数是 (x: Animal) => void，由于 Dog 是 Animal 子类型，传 Dog 满足 Animal —— 安全。

typescript 
interface Animal { name: string }
interface Dog extends Animal { bark(): void }

type DogHandler = (d: Dog) => void;
type AnimalHandler = (a: Animal) => void;

const animalH: AnimalHandler = a => console.log(a.name);
const dogH: DogHandler = d => d.bark();

// 参数逆变：宽参数可以放窄参数位置
const f1: DogHandler = animalH;   // ✓ AnimalHandler 比 DogHandler 更"宽"
                                    //   能接 Animal 的函数自然也能接 Dog
const f2: AnimalHandler = dogH;   // ❌ DogHandler 接不了非 Dog 的 Animal
                                    //   把它放进期望 AnimalHandler 的位置不安全

形式化：函数类型 (x: A) => B 中，参数 A 是逆变位置（输入），返回值 B 是协变位置（输出）。

3. 函数返回值：协变（covariance）

返回值更窄是安全的：调用方期望 Animal，实际给了 Dog（Dog 是 Animal 的子类型），完全没问题。

typescript 
type ReturnsAnimal = () => Animal;
type ReturnsDog = () => Dog;

const getDog: ReturnsDog = () => ({ name: "Rex", bark() {} });

// 返回值协变：窄返回值可以放宽返回位置
const f1: ReturnsAnimal = getDog;   // ✓ Dog 是 Animal — 调用方拿 Animal 当然也行
const f2: ReturnsDog = (() => ({ name: "x" } as Animal));   // ❌ Animal 不一定是 Dog

4. 完整法则：函数类型兼容（含返回值 + 参数）

位置要兼容的方向口诀
参数逆变（变宽 OK，变窄不 OK）"少要求"安全
返回值协变（变窄 OK，变宽不 OK）"多承诺"安全
typescript · 综合例子 
type FA = (x: Animal) => Dog;          // 接受 Animal，返回 Dog
type FB = (x: Dog) => Animal;           // 接受 Dog，返回 Animal

// FA 可以赋给 FB 吗？
//   参数：FA 接 Animal（宽），FB 接 Dog（窄） — 给 FB 位置一个 FA，调用方传 Dog，FA 接受 Animal 包含 Dog ✓
//   返回值：FA 返回 Dog（窄），FB 返回 Animal（宽） — 调用方拿 Animal，FA 给的 Dog 是 Animal ✓
const f1: FB = ((x: Animal) => ({ name: "x", bark() {} })) as FA;   // ✓

// 反向：FB 赋给 FA？
//   参数：FB 接 Dog（窄），FA 接 Animal（宽） — 给 FA 一个 FB，调用方传 Animal，FB 不接 ❌
const f2: FA = ((x: Dog) => ({ name: "x" } as Animal)) as FB;     // ❌

5. ⚠️ Method bivariance：TS 故意留的不安全口子

上面的"参数逆变"规则只对函数类型（(x: T) => U）成立。但 TS 对方法签名（interface / class 里的 method(x: T): U）默认采用双变（bivariance）—— 参数双向都允许：

typescript · 默认（strictFunctionTypes 关） 
interface Listener<T> {
  handle(event: T): void;   // method 签名 — bivariant
}

const dogListener: Listener<Dog> = { handle(d) { d.bark() } };

// ⚠️ 参数变窄竟然也通过 — 这不安全
const animalListener: Listener<Animal> = dogListener;
animalListener.handle({ name: "cat" });   // 编译过 → 运行时炸：bark is not a function

为什么 TS 这么设计？历史包袱：在 strictFunctionTypes 出现之前，TS 完全双变，因为这种不安全在实践中被工程接受。最经典的例子：

typescript 
// JS / TS 设计上 Array<T> 的 push 是 bivariant 的
const dogs: Dog[] = [{ name: "Rex", bark() {} }];
const animals: Animal[] = dogs;   // ⚠️ 不严格安全 — 但 TS 允许，因为大多数情况"够用"

// 严格说不安全的原因：
animals.push({ name: "cat" });    // 编译过
dogs[1].bark();                    // 运行时炸（cat 不是 Dog）

6. strictFunctionTypes 关掉双变（仅函数类型）

TS 2.6 引入这个 flag。打开后，函数类型（function type expression）按严格逆变；方法签名（method signature）仍是双变 —— 后者无法严格化（涉及 array、event handler 等大量历史代码）。

typescript · strictFunctionTypes: true 
// 函数类型 — 严格逆变
type DogCb = (d: Dog) => void;
type AnimalCb = (a: Animal) => void;

const dogCb: DogCb = d => d.bark();
const animalCb: AnimalCb = dogCb;   // ❌ 严格 mode 下不允许

// 但方法签名仍是双变（设计如此）
interface Handler { handle(d: Dog): void }
const h: Handler = { handle(d) { d.bark() } };
const h2: { handle(a: Animal): void } = h;   // ✓ 仍允许（method bivariance 未关闭）

实用建议：永远开 strictFunctionTypes（包含在 strict: true 里）。无法严格的 method 部分接受现状。要严格 method 兼容性，把 method 改成 callable property：

typescript 
// 方法签名 — bivariant
interface Listener1<T> {
  handle(event: T): void;
}

// 函数类型 callable property — strictFunctionTypes 下严格逆变
interface Listener2<T> {
  handle: (event: T) => void;   // 注意：用 = 不是 :
}

// 在严格模式下 Listener2 行为更安全

7. Class 兼容性：private / protected 创建"血缘绑定"

普通 class 走结构兼容。但有 private 或 protected 字段时，TS 把这些字段视为"来源标记"—— 只有共同继承链下的 class 才兼容：

typescript 
class A {
  private secret = 1;
}
class B {
  private secret = 1;
}
// A 和 B 的 secret 名字相同，但来源不同
const a: A = new B();   // ❌ 'private' property 'secret' has different origins

// 继承关系下兼容
class A2 {
  private secret = 1;
}
class C extends A2 {}
const a2: A2 = new C();   // ✓ C 继承自 A2，secret 是同源

这是 TS 类型系统中极少数破坏纯结构化的地方 —— 用名义化区分 class 的具体来源。设计意图：让"私有字段"真正起到"封装"作用，外部无法通过结构兼容绕过。

8. Generic 兼容性：类型参数不出现在结构里时被忽略

typescript 
// T 不出现在结构里
interface Empty<T> {}

const e1: Empty<string> = {};
const e2: Empty<number> = e1;   // ✓ T 不影响兼容性

// T 出现在结构里
interface Box<T> { value: T }
const b1: Box<string> = { value: "x" };
const b2: Box<number> = b1;   // ❌ Box<string> 不兼容 Box<number>

9. 特殊类型互通：any / unknown / never / void

类型能赋给什么什么能赋给它
any 任何（除 never） 任何
unknown 仅 unknown / any 任何
never 任何（never 是底类型，所有类型的子类） 仅 never
void 仅 void / any void / undefined（in non-strict）/ 任何（在函数返回值位置，特殊规则）

关键观察：any 是"双向洞" — 既是顶又是底。这是它危险的根源。unknown 是顶（top type） — 任何能赋给它，但它必须 narrow 后才能用。never 是底（bottom type） — 只有 never 能赋给 never，但 never 能赋给任何。

10. Contextual typing：由位置决定参数类型

通常类型推断是"从值推类型"：const x = "abc" 推 string。Contextual typing 反过来 —— "从位置推类型"：

typescript 
interface Button {
  addEventListener(type: "click", h: (e: MouseEvent) => void): void;
}
declare const btn: Button;

// 内联回调 — contextual typing 工作
btn.addEventListener("click", e => {
  e.button;   // ✓ e: MouseEvent — 从 addEventListener 签名反推
});

// 脱离上下文 — 退化成 any
const handler = e => e.button;   // ❌ Parameter 'e' implicitly has an 'any' type
//                                   handler 不在 addEventListener 上下文里
btn.addEventListener("click", handler);

这是为什么内联回调好写、独立函数难写的根因。要给独立函数加类型，要么显式注解，要么用 typeof 复用签名：

typescript 
// 显式注解
const handler1 = (e: MouseEvent) => e.button;

// 复用 addEventListener 第二参数类型
type ClickHandler = Parameters<Button["addEventListener"]>[1];
const handler2: ClickHandler = e => e.button;

11. Best common type：数组字面量推断的妥协

数组字面量含多种类型时，TS 推断为这些类型的 union，不会自动找父类：

typescript 
class Animal { name = "" }
class Dog extends Animal { bark() {} }
class Cat extends Animal { meow() {} }

const xs = [new Dog(), new Cat()];
//    ^? (Dog | Cat)[]    不是 Animal[]

// 要推 Animal[] 必须显式注解
const ys: Animal[] = [new Dog(), new Cat()];

// 数字与字符串混合
const zs = [1, "a"];   // (string | number)[]
//    ^^^^^^^^^^^^^^   不会推 unknown[]

为什么不找父类？因为 TS 没法在结构化类型系统里可靠地"找最小公共父类型" —— 任意两个对象都可能有无穷多共同父类型（例如空 interface {}、Object、unknown 等都是它们的父类）。所以 TS 取折中：保留每个具体类型的 union。

12. 推断的"宽化"

字面量类型在某些位置会自动"宽化"成原始类型，避免推断过窄：

typescript 
const a = "hello";    // a: "hello"   字面量
let b = "hello";       // b: string    宽化 — 因为 let 可重新赋值

const obj = { x: "hello" };   // { x: string }   对象字段也宽化
const obj2 = { x: "hello" } as const;   // { readonly x: "hello" }   as const 阻止宽化

// 函数返回 inference 会"立即宽化"避免过窄
function get() { return "hello" }
type R = ReturnType<typeof get>;   // string
//                                  不是 "hello"

03 应用场景

场景 A · 写函数式工具库时的 variance 选择

typescript 
// 写一个"事件 channel"：subscribe / publish
class Channel<T> {
  private subs: Array<(v: T) => void> = [];

  subscribe(fn: (v: T) => void) { this.subs.push(fn) }
  publish(v: T) { this.subs.forEach(fn => fn(v)) }
}

// 应用：能给 Channel<Animal> 订阅 Channel<Dog> 的 handler 吗？
//   handler 期望 Dog（窄）；channel publish Animal（宽）
//   实际可能传非 Dog 的 Animal 给 handler — 不安全
const ac = new Channel<Animal>();
ac.subscribe((d: Dog) => d.bark());
//                       ^   ❌ strictFunctionTypes 严格逆变拦住

// 反过来 — 安全
const dc = new Channel<Dog>();
dc.subscribe((a: Animal) => console.log(a.name));   // ✓ handler 宽，channel 窄

场景 B · 第三方库类型 patch 的兼容性

typescript 
// 第三方库声明
declare class Logger {
  log(msg: string): void;
}

// 你想 mock 它做测试 — 接受更宽参数
class TestLogger {
  log(msg: string | number): void {   // 参数变宽
    console.log(String(msg));
  }
}

const logger: Logger = new TestLogger();   // ✓
// 因为 TestLogger.log 接收的更宽（string | number），
// 调用方调 logger.log("x") 时 TestLogger.log 完全能处理 — 安全
logger.log("hello");

场景 C · React 高阶组件的泛型联动

typescript 
// HOC：给被包装组件加 props
function withAuth<P extends { user?: User }>(
  Component: ComponentType<P>,
): ComponentType<Omit<P, "user">> {
  return (props) => {
    const user = useAuth();
    return <Component {...(props as P)} user={user} />;
  };
}

interface ProfileProps { user?: User; theme: "light" | "dark" }
const Profile = (p: ProfileProps) => ...;

const AuthProfile = withAuth(Profile);
// AuthProfile 的 props: Omit<ProfileProps, "user"> = { theme }
<AuthProfile theme="dark" />   // ✓
<AuthProfile theme="dark" user={...} />   // ❌ user 已被 HOC 注入

场景 D · 写 zod / valibot 等 schema 库的类型基础

typescript 
// schema 库要把 schema 与"它能产生的类型"双向绑定
abstract class Schema<Out, In = Out> {
  abstract parse(input: In): Out;
}

// 嵌套使用时 covariance 让"窄 schema 可作为宽 schema"传
class StringSchema extends Schema<string> {
  parse(x: string) { return x }
}
class EmailSchema extends Schema<string> {
  parse(x: string) {
    if (!/.+@.+/.test(x)) throw new Error("not email");
    return x;
  }
}

// 接受任何 Schema<string>
function validate(s: Schema<string>) { return s.parse("test") }
validate(new EmailSchema());   // ✓ EmailSchema 是 Schema<string> 的子类

这种设计是 zod / valibot / arktype 等 schema 库类型系统的内核。理解 variance 后能更好读它们的源码。

04 典型代码

检验自己的"variance 直觉"

typescript · 自测题 
interface Animal { name: string }
interface Dog extends Animal { bark(): void }

// strictFunctionTypes: true

// 1. (a: Animal) => void 能赋给 (d: Dog) => void 吗？
let f1: (d: Dog) => void = (a: Animal) => { /* */ };
// ✓ 能 — 接受更宽的参数函数可以放在期望更窄参数的位置（参数逆变）

// 2. (d: Dog) => void 能赋给 (a: Animal) => void 吗？
let f2: (a: Animal) => void = (d: Dog) => { d.bark() };
// ❌ 不能 — strictFunctionTypes 拦住

// 3. () => Dog 能赋给 () => Animal 吗？
let f3: () => Animal = (() => ({ name: "Rex", bark() {} } as Dog));
// ✓ 能 — 返回更窄的函数可以放在期望更宽返回的位置（返回值协变）

// 4. () => Animal 能赋给 () => Dog 吗？
let f4: () => Dog = (() => ({ name: "x" } as Animal));
// ❌ 不能 — Animal 不一定是 Dog

// 5. Array<Dog> 能赋给 Array<Animal> 吗？
let arr1: Animal[] = [{ name: "x", bark() {} } as Dog];
// ✓ 能 — TS Array 是 covariant（虽然严格说不安全 — push Animal 后 Dog[] 拿不到 bark）

区分函数类型和方法签名

typescript 
// 函数类型语法（function type expression）
interface A {
  cb: (x: Dog) => void;       // strictFunctionTypes 严格
}

// 方法签名语法（method signature）
interface B {
  cb(x: Dog): void;             // bivariant — 即使 strict 也宽松
}

// 等价的写法但行为不同！
const a: A = { cb: (a: Animal) => { /* */ } };   // ❌ strict mode
const b: B = { cb(a: Animal) { /* */ } };          // ✓ method 双变

contextual typing 的边界

typescript 
// 内联：contextual 工作
[1, 2, 3].map(x => x * 2);
//             ^? number   从 Array<number>.map 反推

// 提取出来：失去 context
const double = x => x * 2;   // ❌ implicit any

// 解决方案 1：显式注解
const double = (x: number) => x * 2;

// 解决方案 2：用 typeof + 索引取出 callback 类型
type MapCb<T, U> = Parameters<Array<T>["map"]>[0];
//   实际太复杂，一般直接显式注解

best common type 与显式注解

typescript 
// 默认推 union
const xs = [1, "a", true];   // (string | number | boolean)[]

// 显式注解推到父类型
const xs2: unknown[] = [1, "a", true];   // unknown[]

// 数组字面量含 null/undefined — 默认推宽
const ys = [1, null, "x"];   // (string | number | null)[]

// strictNullChecks: false 时
// const ys = [1, null, "x"];   //（string | number）[] — null 被静默吞

完整 variance 表（含其他位置）

位置variance例
函数参数逆变（contravariant）(x: Animal) => void 可放 (x: Dog) => void
函数返回值协变（covariant）() => Dog 可放 () => Animal
method 参数双变（bivariant）strictFunctionTypes 不影响
readonly 数组协变readonly Dog[] 可放 readonly Animal[]
可变数组双变（不严格安全，TS 允许）Dog[] 可放 Animal[]
Promise<T>协变（仅有 .then）Promise<Dog> 可放 Promise<Animal>
类型参数（默认）双变（取决于使用位置）—
in / out modifier（4.7+）显式type F<in T> = (x: T) => void

05 常见陷阱

陷阱 1 · 关掉 strictFunctionTypes 让 callback 类型不安全

"strict": true 包含 strictFunctionTypes，但单独关掉会让所有函数类型双变：

typescript 
// strictFunctionTypes: false
type DogCb = (d: Dog) => void;
type AnimalCb = (a: Animal) => void;

const dogCb: DogCb = d => d.bark();
const animalCb: AnimalCb = dogCb;   // ✓ 不安全但通过
animalCb({ name: "cat" });           // 编译过 → 运行时炸

规避：永远 strict（默认）。

陷阱 2 · 仅 private 字段名相同的两个 class 不兼容
typescript 
class A {
  private id = "";
  name = "";
}
class B {
  private id = "";
  name = "";
}
// A 和 B 看起来一样 — 但 private id 来源不同
const a: A = new B();   // ❌ 不兼容

规避：要"接口形状一致"用 interface 而非 class；class 间共享 shape 用 abstract 父类。

陷阱 3 · 数组 inference 推到 union 而非父类
typescript 
function add(animals: Animal[]) {}

// 期望传 Animal[]，但
add([new Dog(), new Cat()]);
//   ^^^^^^^^^^^^^^^^^^^^^^   推断为 (Dog | Cat)[]
//   能赋给 Animal[] 吗？  — 是的（结构上 Dog | Cat 是 Animal 子类型）

// 但有些 utility 期望严格 Animal[]
const arr = [new Dog(), new Cat()];   // (Dog | Cat)[]
const animals: Animal[] = arr;          // ✓
const arr2: Dog[] = arr;                // ❌

规避：需要父类型时显式注解 const arr: Animal[] = [...]。

陷阱 4 · 内联回调 vs 提取函数的 contextual typing 差异
typescript 
// 1️⃣ 内联：context 工作
btn.addEventListener("click", e => e.button);   // ✓ e: MouseEvent

// 2️⃣ 提取：失去 context
const handler = e => e.button;   // ❌ implicit any
btn.addEventListener("click", handler);

// 3️⃣ 显式注解：OK
const handler1 = (e: MouseEvent) => e.button;
陷阱 5 · void 在返回值位置的"宽松"行为
typescript 
// 04 章已讲过：void 返回类型上下文允许任意返回值
type VoidFn = () => void;

const f1: VoidFn = () => 42;   // ✓ 返回 number 也通过

// 这导致一个微妙的赋值兼容性
type StrictReturn = () => undefined;
type LooseReturn = () => void;

const a: StrictReturn = () => { return };   // ✓ — 显式 return
const b: StrictReturn = () => 42;             // ❌
const c: LooseReturn = () => 42;              // ✓ — void 宽松
const d: StrictReturn = c;                     // ❌ — void 不能赋给 undefined
陷阱 6 · TS 4.7 in / out 显式 variance 修饰符 — 仅在严格库需要

TS 4.7 引入 in / out modifier 让你显式标注类型参数的 variance。但这是给库作者的工具，应用代码几乎用不到：

typescript 
// 显式逆变 in
type Sink<in T> = (x: T) => void;

// 显式协变 out
type Source<out T> = () => T;

// 显式不变（同时 in + out）
type Channel<in out T> = { send(x: T): void; receive(): T };

没有这个 flag 时，TS 自动从使用位置推 variance。99% 项目不需要显式 in/out。
