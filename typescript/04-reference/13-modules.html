13 · Modules

TS 的模块系统跨越三个时代：ES module（静态、可 tree-shake）、CommonJS（运行时 require）、namespace（旧 internal module，反模式）。本章拆解三者互通、import type 的编译期消失、esModuleInterop 和 5.0 引入的 verbatimModuleSyntax 配置含义、以及为什么"没有 import/export 的 .ts 文件会污染全局类型"。

阅读时长 ~30 分钟 前置 JS 模块基础（推荐 JS · 04-modules） 对应 typescriptlang.org Handbook · Modules

01 速览

三个核心规则：

  1. 有 top-level import 或 export = 模块。否则该文件被当成全局脚本，所有声明污染全局类型空间
  2. import type 编译产物中完全消失。仅在类型层引入，不触发运行时副作用
  3. module 决定输出格式，moduleResolution 决定如何找文件，是两件事
typescript · 最常用语法 
// 命名导入 / 导出
export const x = 1;
import { x } from "./mod";

// default 导入 / 导出
export default function f() {}
import f from "./mod";

// 仅类型 — 编译产物消失
import type { User } from "./types";

// inline type modifier（4.5+）
import { fn, type X } from "./mod";

// namespace 导入（拿到所有 export 的对象）
import * as M from "./mod";

// re-export
export { x } from "./mod";
export type { X } from "./types";

// side-effect import（仅运行 mod 顶层代码）
import "./polyfill";

// CJS 互通
import x = require("legacy-cjs-pkg");

02 底层逻辑

1. ES module 是静态结构 —— tree-shaking 的根因

ES module 的 import / export 必须在 top-level，不能在 if 里、不能在函数里（动态 import() 是另一回事）。这一约束的根本动机：让 bundler 在不执行代码的情况下，仅靠静态分析就能知道哪些 export 被使用、可以删除什么。

typescript 
// ❌ ES module 不允许
if (condition) {
  import { x } from "./mod";   // SyntaxError
}

// ❌ 也不允许
function load() {
  import { x } from "./mod";   // SyntaxError
}

// ✅ 动态 import — 运行时按需加载，返回 Promise
async function load() {
  const { x } = await import("./mod");
}

对比 CommonJS：require() 是运行时函数，可以放在任意位置、可以条件调用、可以被赋值。这让它灵活，但无法静态分析 —— 这就是为什么纯 CJS 项目无法 tree-shake。

javascript · CJS 
if (process.env.NODE_ENV === "dev") {
  const debug = require("./debug");   // 运行时条件加载
  debug.start();
}

// 这种代码 bundler 无法分析 — 必须把所有可能 require 的东西都打进去

2. import type：仅类型层引入

import type 是 TS 3.8 引入的语法，编译产物中完全消失。它解决三个工程痛点：

  • 避免循环依赖：A 引用 B 的类型，B 运行时引用 A —— 用 import type 后，A → B 的依赖只在编译期，运行时无
  • 避免 side effect 加载：仅取类型时，原模块的顶层副作用不会运行
  • 配合 isolatedModules：babel/swc 等单文件 transpiler 必须能区分"类型 vs 值"才能正确 emit
typescript · 编译对比 
// 源 .ts
import type { User } from "./types";
import { fn } from "./util";

const greet = (u: User) => fn(u.name);

// 编译后 .js（target: ES2022）
import { fn } from "./util";
//                  注意：./types 完全消失了
const greet = (u) => fn(u.name);

TS 4.5 进一步引入 inline type modifier，让类型导入和值导入混在一行：

typescript 
// 4.5+ inline type
import { fn, type Config, type User } from "./mod";
//          ^^^   ^^^^^^^^^^^^^^^^^^   编译时仅 fn 保留，类型消失

3. esModuleInterop：CJS/ESM 不兼容的运行时垫片

这是 TS 工程配置中最容易让人混淆的开关。要理解它先看历史：

  • CJS 模块：module.exports = ...，整个 module 是单值（可以是函数、对象、类）
  • ES module：export default ... 是命名为 "default" 的导出，所有命名导出在一个对象里

一个 ES 模块导入 CJS 模块时存在不兼容：

typescript · 不开 esModuleInterop 
// CJS 库 fs：module.exports = { readFileSync, ... }

// ❌ 直觉的 default 导入失败
import fs from "fs";
fs.readFileSync;   // 错 — fs 是 undefined

// 必须用 namespace 导入
import * as fs from "fs";
fs.readFileSync;   // ✓

// 或者老语法
import fs = require("fs");
fs.readFileSync;   // ✓

开 esModuleInterop: true 后，TS 在编译产物里注入一个垫片函数 __importDefault，让 import fs from "fs" 也能工作：

typescript · 开 esModuleInterop 
import fs from "fs";   // ✓ 通过
fs.readFileSync;

编译产物大约是：

javascript · 编译后 
var __importDefault = (mod) => (mod && mod.__esModule) ? mod : { default: mod };
const fs = __importDefault(require("fs"));
fs.default.readFileSync;
// （TS 通过 IDE 提示让你写得像 fs.readFileSync，但产物里是 fs.default.readFileSync）

实用建议：新项目永远开 esModuleInterop: true。它兼容性更好。allowSyntheticDefaultImports 是 esModuleInterop 的仅类型层子集（不注入运行时垫片，仅让 TS 类型层接受 default 导入），通常和 esModuleInterop 一起开。

4. verbatimModuleSyntax（TS 5.0）：严格区分 import type 和 import

TS 5.0 引入这个 flag 替代旧的 importsNotUsedAsValues / preserveValueImports。语义：编译时不删除任何 import 语句，原样保留你写的 import / import type。

json · tsconfig 
{ "compilerOptions": { "verbatimModuleSyntax": true } }

这一开关强制了几条规则：

  • 仅类型导入必须用 import type。否则编译产物里仍会保留 import "..." 触发副作用
  • 仅类型导出必须用 export type
  • 不允许 import x = require() 与 module = ... 等 CJS 语法，因为它们在 ESM 输出时无法直接保留

为什么这个 flag 重要？因为 babel / swc / esbuild 等 transpiler 是单文件处理的 —— 它们不知道某个 import 仅是类型用途。开 verbatimModuleSyntax 让 TS 的输出和这些 transpiler 输出完全一致，工具链可互换。

5. 没有 import/export 的 .ts 文件 = 全局脚本

这是最容易踩的坑：

typescript · src/utils.ts 
// 没有 import / export
function helper() { return 1 }
const x = "abc";

// 此文件被当成全局脚本：helper / x 进入全局命名空间
// 其他文件即使不 import 也能用 helper（且与其他全局名字可能冲突）

要把它变成模块（推荐）：

typescript 
export function helper() { return 1 }   // ← 加 export 即可
const x = "abc";

// 或者随便加个空 export
export {};

tsconfig 选项 "moduleDetection": "force"（TS 4.7+）可以强制所有 .ts 都按模块处理，避免这种意外。新项目建议开。

6. ⚠️ namespace 是反模式（在 ES module 时代）

TS 早期曾用 namespace 做"internal module"，把多个文件的代码合并到同一命名空间下。在 ES module 标准化之前这是合理设计，现在是反模式。

typescript · 旧时代 
namespace MyApp {
  export function start() {}
  export namespace Utils {
    export function helper() {}
  }
}

MyApp.start();
MyApp.Utils.helper();

为什么不用了：

  • 无法 tree-shake：整个 namespace 被打包成一个对象，bundler 看不出哪些没用
  • 不利于代码分割（dynamic import）
  • 和 ES module 混用容易行为诡异
  • 命名空间嵌套深时调用链冗长

什么时候仍然要用：仅在声明文件 .d.ts 中给全局对象（jQuery、$、process 等）写类型时，namespace 是合适的（详见 18 章）。生产代码避免。

7. ⚠️ Triple-slash directive：仅 .d.ts 内偶用

/// <reference> 是 TS 在 ES module 之前的"文件级依赖声明"。现代项目几乎不用。

typescript 
/// <reference path="./types.d.ts" />       // 加载某 .d.ts
/// <reference types="node" />              // 加载某 @types 包
/// <reference lib="ES2020" />               // 加载内置 lib

// 现代项目用 tsconfig 的 "types" / "lib" 配置代替

8. module vs moduleResolution

这两个 tsconfig 选项常被混淆：

选项决定什么常用值
module 输出什么模块格式 commonjs / esnext / nodenext / preserve
moduleResolution 解析 import 时如何找文件 node / node10 / node16 / nodenext / bundler

TS 5.0 引入 moduleResolution: "bundler"，对应 Vite / webpack / esbuild 等 bundler 的解析逻辑（接受 ESM 语法 + 不强制 .js 后缀 + 支持 conditional exports）。是新项目最常用的配置。

03 应用场景

场景 A · Monorepo workspace alias

json · tsconfig.json 
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@app/*": ["apps/app/src/*"],
      "@shared/*": ["packages/shared/src/*"],
      "@/lib/*": ["src/lib/*"]
    }
  }
}

⚠️ paths 只影响 tsc 的类型检查，不影响运行时。Vite / Next.js / webpack 必须各自配 alias，否则编译过运行时 module not found（详见 19 章）。

场景 B · 仅类型导入避免循环依赖

typescript 
// user.ts
import type { Post } from "./post";   // 仅类型 — 编译产物消失

export interface User {
  id: string;
  posts: Post[];
}

export const fetchUser = async () => { /* ... */ };

// post.ts
import { fetchUser } from "./user";   // 运行时引用 user

export interface Post { id: string; authorId: string }

export async function fetchAuthor(p: Post) {
  return fetchUser();   // 运行时仅 user → post 的边没循环
}

如果 user.ts 用 import { Post } 而非 import type { Post }，运行时会形成 user ↔ post 循环依赖，可能导致部分 export 是 undefined（取决于加载顺序）。

场景 C · Vite + Next.js 16 推荐配置

json · tsconfig.json 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "verbatimModuleSyntax": true,
    "isolatedModules": true,
    "moduleDetection": "force",
    "noEmit": true,
    "jsx": "preserve"
  }
}

noEmit: true + bundler 自己 transpile 是现代前端工程的标配。tsc 仅做类型检查，编译速度由 esbuild / SWC 决定（详见 19 章）。

场景 D · Node 22 + ESM 项目

json · package.json 
{
  "type": "module",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js"
    }
  }
}
json · tsconfig.json 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext"
  }
}

NodeNext 模式要求 import 路径带 .js 后缀（即使源文件是 .ts）：

typescript 
import { foo } from "./util.js";   // ✓ 必须 .js（不是 .ts）
import { bar } from "./util";       // ❌ 在 NodeNext + ESM 下报错

04 典型代码

所有 import / export 形态一览

typescript 
// === 命名导出 ===
export const x = 1;
export function fn() {}
export class C {}
export interface I {}
export type T = string;
export enum E { A }

// === 一次导出多个（已声明的） ===
const a = 1;
const b = 2;
export { a, b };
export { a as renamed };

// === default 导出（仅一个） ===
export default function () {}
// 或
const value = {};
export default value;

// === 仅类型导出（不出现在产物） ===
export type { I, T };

// === re-export ===
export { x } from "./mod";
export * from "./mod";          // 全部 re-export
export * as M from "./mod";     // re-export 为 namespace
export { default } from "./mod"; // re-export default

// === 导入 ===
import { x } from "./mod";              // 命名
import { x as renamed } from "./mod";   // 重命名
import default_ from "./mod";            // default
import * as M from "./mod";              // namespace
import default_, { x } from "./mod";    // 混合
import "./polyfill";                     // side-effect

// === 仅类型导入 ===
import type { T } from "./mod";
import { fn, type T } from "./mod";     // inline type（4.5+）

// === CJS 互通 ===
import x = require("legacy-pkg");
export = x;                              // CJS 风格 default 导出（不推荐 ESM 项目用）

动态 import 的类型化

typescript 
// 动态 import 返回 Promise<Module>
async function loadAdmin() {
  const adminModule = await import("./admin");
  // adminModule: typeof import("./admin")
  adminModule.start();
}

// 类型层取动态 import 的 module 类型
type AdminModule = typeof import("./admin");

// 配合 React.lazy
const Admin = React.lazy(() => import("./Admin"));

// dynamic import 是代码分割的 hook
async function feature() {
  const { exportedFn } = await import("./big-feature");
  return exportedFn();
}

TypeScript 的 ambient module declaration

typescript · types/global.d.ts 
// 给"无类型的 npm 包"加类型
declare module "untyped-lib" {
  export function helper(): void;
  export const version: string;
}

// 通配 module — 让 webpack/Vite 资源 import 通过
declare module "*.svg" {
  const content: string;
  export default content;
}

declare module "*.css" {
  const styles: { [k: string]: string };
  export default styles;
}

// 全局变量
declare global {
  interface Window {
    __APP_VERSION__: string;
  }
}
export {};   // 必须加，否则文件会被当全局脚本

避免循环依赖的"interface only"模式

typescript 
// types.ts — 仅类型，不被运行时依赖
export interface User { id: string; name: string }
export interface Post { id: string; authorId: string }

// user.ts
import type { Post } from "./types";   // 仅类型
import { fetchPost } from "./post";

export interface UserWithPosts extends User {
  posts: Post[];
}

// post.ts
import type { User } from "./types";   // 仅类型
import { fetchUser } from "./user";

// 运行时依赖图：user ↔ post 仍循环
// 但若 user/post 不互相运行时引用，仅类型层关联，循环消失

05 常见陷阱

陷阱 1 · 没有 import/export 的 .ts 污染全局

最易遗漏。一个 utility 文件如果只声明不导出，所有声明会进入全局空间：

typescript 
// utils/log.ts — 没 import / export
function log(msg: string) { console.log(msg) }
const PREFIX = "[app]";

// other.ts 不 import 也能用 — 和你想的不一样
log("hello");   // ✓ TS 居然不报错

规避：

  • 每个 .ts 至少写一个 export {}
  • tsconfig 开 "moduleDetection": "force"（4.7+）—— 强制所有文件都是模块
陷阱 2 · 默认导出与命名导出混用导致 tree-shaking 失败
typescript 
// ❌ 把多个 utility 集合在一个 default object 里
export default {
  formatDate,
  parseJSON,
  log,
};

// 调用方
import utils from "./utils";
utils.log("hi");

// 即使只用 utils.log，bundler 仍把整个 utils 对象打进去 — 无法删除未用的 method

// ✅ 命名导出，每个独立
export { formatDate, parseJSON, log };

// 调用方
import { log } from "./utils";   // ✓ tree-shake 友好

规则：库代码永远用命名导出。default 导出仅用于"该模块语义就是单值"的场景（如 export default function App()）。

陷阱 3 · esModuleInterop 在 ESM-only 项目下的反作用

如果你的项目所有依赖都是 ESM（不再有 CJS），esModuleInterop: true 会让 TS 给每个 default 导入加垫片，反而可能改变语义。

typescript 
// ESM 库 some-pkg 直接 export default {fn}
import x from "some-pkg";
x.fn();   // ESM 原生：x = { fn }，OK

// 开 esModuleInterop 后产物注入垫片：
// const x = __importDefault(require("some-pkg")).default
// → x = { fn }，仍然 OK
// 但有些边缘情况（CJS module.exports 是函数）行为会怪

规避：纯 ESM 项目可以试试 esModuleInterop: false，需要时显式 import * as ns from。但绝大多数项目仍开 esModuleInterop（兼容性最好）。

陷阱 4 · NodeNext 模式忘了 .js 后缀
typescript 
// tsconfig: module: NodeNext
import { foo } from "./util";        // ❌ Cannot find module './util'
import { foo } from "./util.js";     // ✓ 必须 .js（即使源是 .ts）
import { foo } from "./util.ts";     // ❌ 不允许 .ts

原因：NodeNext 严格遵循 Node 的 ESM 解析规则，要求带后缀。但 TS 不让你写 .ts（产物里是 .js）—— 所以约定写 .js，TS 自己映射回 .ts 源做检查。

规避：用 moduleResolution: "bundler"（如果用 Vite/Next）—— 不需要后缀。或者用 tsc-alias 等工具自动加后缀。

陷阱 5 · 在 .d.ts 中误用 import 引发"全局 vs 模块"切换
typescript · types/global.d.ts 
// 想给全局 Window 加字段
interface Window {
  __APP_VERSION__: string;
}
// ✓ 此时文件是"全局脚本" — interface Window 自动 declaration merge

// 加了 import 之后变成"模块"
import { something } from "./other";
interface Window { __APP_VERSION__: string }
// ❌ 不再 merge —— 在模块内的 interface Window 是局部的

规避：模块内扩展全局必须用 declare global { ... } 包裹（详见 15 章）：

typescript 
import { something } from "./other";

declare global {
  interface Window {
    __APP_VERSION__: string;
  }
}

export {};   // 让此文件是模块
陷阱 6 · 老代码用 namespace 与新代码混合
typescript 
// ❌ 混用
namespace MyApp {
  export function start() {}
}

import { other } from "./other";   // 模块导入

MyApp.start();   // 还能用 — 但产物诡异

规避：迁移老代码时把 namespace 全部改成 ES module。仅在 .d.ts（给全局 jQuery 等写类型时）保留。
