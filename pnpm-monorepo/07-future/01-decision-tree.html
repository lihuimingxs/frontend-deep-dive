决策树 + 下一代趋势

P1-P6 把 monorepo 的所有维度（包管理 / node_modules / pnpm 内部 / workspace / 编排 / 发布）讲透了。本章把它收敛成一张 4 档决策树，并展望 5 年后的格局。核心立场：中立——每条决策都给 trade-off，不当任何家广告。读完本章你能对"这个项目该用什么"有可论证的答案。

阅读时长 ~25 分钟 前置 P1-P6 全部 主要来源 各家官方公告 + 行业 trends + JSR 设计文档 + Bazel/Rush 大厂实践

01 速览

  1. 4 档决策树 —— (1) 个人 / 单包小项目 → npm 或 Bun，不必上 monorepo；(2) 中型 5-30 包 → pnpm + Turborepo（最低门槛 monorepo）；(3) 大型 30-100 包 → pnpm 或 Bun + Turborepo / Nx 二选一（按"组合 vs 全家桶"哲学）；(4) 超大规模 100+ 包 + 多语言 → Bazel / Rush / Pants（JS 工具链不够用）。
  2. 2026 主流方案 = pnpm + Turborepo + Changesets —— Vite / Vitest / Astro / typescript-eslint / shadcn/ui / Cloudflare Workers / Remix 等多数旗舰 OSS 项目用这套。它在简洁性 / 严格性 / 灵活性 / 生态成熟度上接近最佳平衡。
  3. Bun 是中型新项目的另一个明显选项 —— Bun + Turborepo + Changesets 同样能跑，速度优势明显。2026 年 Bun runtime 在 SaaS / 工具型应用份额快速增长，包管理紧随。
  4. Yarn 4 PnP 路线复盘 —— 5 年来在生态接受度上失败。技术先进 ≠ 市场胜利。它的"取消 node_modules"思想被 Bun / Deno 部分吸收，但作为独立产品的影响力衰减。
  5. JSR（Deno 主导）能否撼动 npm —— 短期不能（npm 200 万包基数 + 14 年生态）。但 JSR 的方向（原生 ESM + TS + 强签名 + cross-runtime）很可能成为 npm v2 的设计参考。5-10 年视野不可忽视。
  6. catalog 标准化是 2025-2026 最重要的 monorepo 进化 —— pnpm / Bun 兼容、Yarn 在跟进、npm 仍未支持。这是 npm 11 落后一代的具体表现，也是 monorepo 选 pnpm/Bun 的具体理由。
  7. 2026 反模式集锦 —— pnpm + Turborepo + Nx + Lerna 同时存在的"积弊型"项目；用 path mapping 替代 workspace；CI 不用 --frozen-lockfile；publish 不开 provenance；"先全家桶后悔过"——选哲学时摇摆。
  8. 5 年展望 —— (1) catalog 跨工具完全兼容；(2) JSR 在 OSS 库领域获明显份额；(3) 远程 cache 成 CI 标配（不再是付费高级功能）；(4) Bun runtime + Bun install + Bun build 一体化挤压 Node + pnpm + Vite 组合；(5) workspace 协议进 ECMA spec？（推测）

02 底层逻辑

4 档决策树详解

档 1 · 个人 / 单包小项目（< 5 包）

典型场景：

  • 个人 Side project
  • 单包 npm 库（一个 SDK，不需要拆 monorepo）
  • 学习项目 / demo
  • 小工具脚本

建议：

  • npm 11 或 Bun install（看个人偏好）
  • 不要用 monorepo（增加复杂度无收益）
  • 不需要 Turborepo / Nx
  • 不需要 Changesets（直接 npm publish 够用）

为什么不推 pnpm：单包项目下 pnpm 的严格性 / 磁盘节省价值不显著。npm 11 / Bun 装上即用，零学习成本。pnpm 留给真正需要的中大型场景。

档 2 · 中型项目（5-30 包）

典型场景：

  • 2-3 个 web app + 共享 UI 库 + utils
  • 开源库的核心 + plugins
  • 中等 SaaS 公司的前端栈
  • 10 人以下小团队

建议：

  • pnpm 10（包管理 + workspace + catalog）
  • Turborepo 2（task runner + cache）
  • Changesets（如有 lib 需要 publish；纯应用项目不用）
  • Vercel Remote Cache（免费层够用）

为什么这套：

  • pnpm 比 npm 多了 catalog + 严格性，少了幻影依赖风险——长期生产力收益显著
  • Turborepo 比手写 npm-run-all + concurrent 强（task graph + cache + dev server 协调）
  • Changesets 让发版自动化，不用每次手动 npm publish 多个包
  • 整套学习曲线 1-2 天，对中型团队 ROI 极佳

替代：

  • Bun + Turborepo + Changesets：如果团队偏好 Bun（速度 / 一体化）
  • npm + Turborepo：如果不愿换包管理器（但放弃 catalog 收益）
  • Yarn 4 nodeLinker + Turborepo：Yarn 用户老顾客（少见）

档 3 · 大型项目（30-100 包）

典型场景：

  • 大型 SaaS 公司前端栈
  • 开源 monorepo 旗舰项目（Vite / Astro / Vitest）
  • 公司内部多产品线共用 monorepo
  • 20-100 人团队

建议：

  • pnpm 10（更稳成熟，大型 monorepo 仍是首选）
  • Turborepo 2 或 Nx 20（按团队哲学选）
  • Changesets（必须）
  • provenance（必须，对抗供应链）
  • 付费 Remote Cache（Vercel 付费层 / Nx Cloud）

Turborepo vs Nx 决策（详见 P5.2）：

  • Turborepo 优势：低侵入、可逆、与已有工具组合自由、OSS 生态主流
  • Nx 优势：generator + migration 让长期一致性强、affected 更精确、DTE 分布式执行
  • 团队大 + 强需求一致性 → Nx
  • 团队偏好组合自由 → Turborepo

不建议 Bun install 在大型 monorepo：

  • Bun install 速度优势在装包阶段；大型 monorepo 的瓶颈是 CI build / test 时间，那靠 Turborepo cache 解决
  • 大型 monorepo 经常有"复杂依赖"（patch / 复杂 hooks / 老依赖），pnpm 的成熟度更稳
  • 2026 年 Bun install 在大型 monorepo 仍偶有边界 case 需要 fallback
  • 除非项目从一开始就用 Bun runtime，否则单独切 Bun install 价值不高

档 4 · 超大规模（100+ 包，多语言）

典型场景：

  • Google / Meta / Microsoft 大厂级 monorepo
  • 多语言混合（JS + Java + Go + Python + Scala）
  • 500+ 人团队
  • Office / Azure / Spotify / Uber 等量级

建议：

  • Bazel（Google 出品，工业级）
  • 或 Rush（Microsoft 自家方案，JS 专精）
  • 或 Pants（Twitter 现 OSS，Python + JS + 多语言）

为什么 Turborepo / Nx 不够：

  • 它们设计针对 JS 生态，多语言支持弱
  • 500+ 包时 cache 算法 + IO 性能成为瓶颈，需要更工业的实现
  • 大厂场景需要 SLO 级别保证：可重现 build / 严格 sandbox / 任意机器结果一致
  • Bazel 的 BUILD 文件描述 + sandbox 机制是 Google 内部 Blaze 的开源版本

2026 现状：99% 的项目永远不会到这个规模。这一档主要给"知道自己在做什么"的大厂用——错选 Bazel 给中型项目意味着 6 个月学习曲线 + 极度复杂配置 + 可能没收益。

常见误区：看到 Google 用 Bazel 就想自己上 Bazel。不要。Bazel 解决的是"10000 包 + 多语言 + 强可重现"问题，中型项目不是这个 profile。

2026 生态趋势观察

趋势 1 · catalog 协议跨工具兼容

2024-2025 关键变化。pnpm 9.5 引入 → Bun 1.1.42 跟进 → Yarn 4 RFC。子包写 "react": "catalog:" 在 pnpm / Bun 都能跑。

下一步：

  • Yarn 4 GA 支持（2026 中下半年）
  • npm 11 / 12 跟进（推测 2027）
  • workspace 协议（workspace:*）走类似路径——npm 在 v9+ 已支持，跨工具兼容已实现

趋势 2 · 二进制 vs 文本 lockfile 之争

Bun 早期用二进制 bun.lockb，2024-2025 改回 JSON bun.lock（diff 友好）。pnpm-lock.yaml 仍是 YAML（diff 友好）。2026 年趋势是文本 lockfile 主流——二进制的速度优势没有想象中大，diff / merge / review 友好性是更重要的工程特性。

趋势 3 · 远程 cache 之争

Vercel Remote Cache（Turborepo） vs Nx Cloud vs 自建（S3 / Cloudflare R2）。2026 趋势：

  • Vercel + Nx Cloud 免费层在变厚（吸引开源 + 小团队）
  • 大型团队倾向自建（控制 + 成本）
  • Cloudflare Workers + R2 成为热门自建栈
  • 5 年内远程 cache 成 CI 标配，从付费高级功能变基础设施

趋势 4 · Bun 一体化的挤压

Bun runtime + Bun install + Bun build + Bun test 是同一个产品的 4 个面（详见 Bun 主题）。这种一体化对"Node + pnpm + Vite + Vitest"组合栈的挤压在 2025-2026 年显著加速。

组合栈的反应：

  • pnpm 不会被 Bun install 替代（短期）——大型 monorepo 仍偏好 pnpm 的严格性 + 成熟
  • 但中型 + 新项目份额会被 Bun 抢去
  • Vite 通过 Rolldown（Rust bundler）追速度；Vitest 改进 watch 模式
  • 结果：2026 是组合栈与 Bun 一体化两条路线的并行期，5 年后看哪个赢更多

趋势 5 · JSR 能否撼动 npm

Deno 主导的 JSR（jsr.io）2024 发布。设计先进——原生 ESM + TS + 强签名 + cross-runtime。但npm 200 万包 + 14 年生态是巨大壁垒。

5 年视角：

  • JSR 在 OSS 库（特别是新发布的）获 5-10% 份额
  • npm 仍是历史包 + 业务代码主流
  • JSR 的方向（强签名 / 原生 TS）很可能成为 npm 自身演进参考
  • "杀死 npm"是错的目标——更可能是npm v2 = npm + JSR 设计精华

Yarn Berry / PnP 路线复盘

Yarn Berry 是 5 年前最激进的 monorepo 方案。技术上几乎完美——无 node_modules / 极速 / 严格 / zero-installs。为什么没赢？

结构性原因：

  • JS 生态有几十万个包，大多数假设 node_modules 存在。一个新方案再好，让 80% 的包要 patch 才能跑，市场不接受
  • 调试体验降级（栈跟踪指向 zip）
  • 编辑器 / TS / webpack 都要适配，每家 plugin 维护
  • Yarn classic → Berry 升级路径不无痛（事实上 Yarn classic 1.x 仍在用，Berry 没大量替代）

对 pnpm 的胜利：

  • pnpm 走"保留 node_modules 形式 + 改写其拓扑"——兼顾严格性和生态兼容
  • 幻影依赖被消除（与 PnP 等效），但 Node resolution 仍走标准路径
  • 这是"渐进改良比革命更易胜利"的工程经典案例

PnP 思想的延续：

  • Bun install 的"binary lockfile + cache 复用"是 PnP 思想的轻量版
  • Deno 的"全局 cache + URL imports"在概念上接近 PnP
  • "zero-installs"（commit cache 到 git）思路仍偶被使用

2026 现状：Yarn Berry 仍有忠实用户（特别是已经投入 PnP 适配的项目）。但"新项目选 Yarn 4 PnP"在 2026 年罕见。

历史教训

教训 1 · Lerna 退场不是 monorepo 死

2020-2022 看到 Lerna "deprecated" 标记，有人得出"monorepo 是错的"结论。事实正相反——Lerna 退场是因为monorepo 工具市场升级了（Turborepo / Nx 提供 cache + 项目图）。2026 年 monorepo 比 Lerna 时代更主流。

教训 2 · 不要在生产挂 4 套工具

"积弊型"项目状态：原本 Lerna，加了 pnpm，加了 Turborepo，又加了 Nx。结果四套配置都有，团队搞不清哪个生效。2 个工具就够——pnpm + Turborepo（组合）或 Nx + 内置包管理（全家桶）。

教训 3 · "全家桶" vs "组合工具"哲学路线无对错

Nx 和 Turborepo 都有大型成功项目背书。不要把它们当对立竞争——本质是两种 monorepo 哲学。选时按团队文化匹配，但选定后不要中途切（迁移成本高 + 团队心智成本）。

教训 4 · 技术先进 ≠ 市场胜利

Yarn Berry / PnP 是经典案例。技术上几乎完美，但生态成本被低估。"渐进改良"通常比"彻底革命"更易胜利——pnpm 接受 node_modules 形式但改其拓扑、catalog 沿用 SemVer 但加版本字典。

教训 5 · 不要选"最新的"，选"匹配的"

Bun 是 2024 年 hot tech，但大型 monorepo 切 Bun 时机未到（兼容性边界仍在补）。中型新项目或 SaaS 工具型应用 Bun 是好选择。选型应基于项目特征 + 团队能力，不是技术 hype。

5 年展望（2026-2031）

预测有风险，但根据 2024-2026 的趋势可以给出方向性判断：

(1) catalog + workspace 协议跨工具完全兼容：pnpm / Bun / Yarn / npm 都支持同一套语法。"切包管理器只改 packageManager 字段"成为现实。

(2) 远程 cache 成 CI 标配：从 Vercel / Nx 付费产品变成 GitHub Actions / GitLab CI 内置功能。每个 CI 运行都自动复用 cache。

(3) Bun runtime 在中型 SaaS / 工具应用主流化：Bun + Bun install + Bun build + Bun test 一体化对中型项目的吸引力增加。Bun 5 年内份额可能从 12% 升到 25-30%。

(4) 大型 monorepo 仍以 pnpm + Turborepo/Nx 为主：成熟度 + 严格性是大型场景的硬要求，Bun 在这一档暂不挑战 pnpm。

(5) JSR 在 OSS 库领域获明显份额：新 OSS 库优先发 JSR + npm 双重发布。npm 在 5 年后仍是主流但 JSR 不再"小众"。

(6) provenance 成为 npm 包默认：发包不开 provenance 在 5 年后被视为"不专业"。供应链安全成为生态共识。

(7) Vite + Rolldown 完全 GA：Vite 6/7 把 Rollup 替换成 Rolldown（Rust），dev / build 双模架构性能跨越。组合栈 vs Bun 之争更激烈。

(8) workspace / catalog 协议进 ECMA 或 npm spec（推测）：当前 workspace 是包管理器层约定，不是 npm spec 的一部分。5 年内可能正式化。

不确定性：

  • Yarn 5 是否会推出？路线不明
  • Bun 在大型 monorepo 能否突破？取决于复杂依赖兼容
  • JSR 是否会被 npm 收购或合并？商业可能性
  • 新 entrant？2026 年看不到（前一代已经成熟，破坏性创新需要新 paradigm）
💡 终极原则：选你能维护的，不是最炫的

monorepo 工具栈的最大成本不是"装上"，是"5 年后还能 happy 升级"。Nx migrate / Changesets 成熟 / pnpm 稳定迭代等"长期可维护性"特性比"最新最快"重要得多。选型时优先考虑团队能持续维护的方案，避免"装上后维护人离开 → 团队不敢动 → 6 年后还在跑 deprecated 工具"的常见结局。

03 应用场景

场景 1 · 新创业公司选 monorepo 栈

背景：5 人创业团队，要开发 SaaS。前端 React + Next.js，后端 Hono on Bun。共享 schema / utils。预计 1 年内 10-20 包。

建议（档 2）：

  • 包管理：Bun install（已用 Bun runtime + 速度收益）
  • 编排：Turborepo 2（与 Bun 自然兼容）
  • 发布：暂不需要（应用项目，不发 npm）；以后某 lib 要发再加 Changesets
  • cache：Vercel Remote Cache（免费层够）

理由：项目用 Bun runtime，包管理同栈一体化最大化；中型规模 Turborepo 比 Nx 简单；早期不需要 Changesets。

场景 2 · 大型 SaaS 公司从 npm 老 monorepo 升级

背景：5 年的 npm + Lerna monorepo，60 包，30 人前端团队。CI 时间 25 分钟，幻影依赖偶尔翻车。

建议（档 3）：

  • 第 1 阶段（1 周）：迁移到 pnpm + 修幻影依赖（暴露 + 修复 5-30 处）
  • 第 2 阶段（1 周）：装 Turborepo + 配 turbo.json + Vercel Remote Cache → CI 时间应降到 5 分钟
  • 第 3 阶段（1 周）：用 catalog 统一核心依赖版本（react / typescript / 内部 utils）
  • 第 4 阶段（1 周）：装 Changesets 替代 Lerna 的 publish 流程；启用 provenance
  • 第 5 阶段（持续）：评估 Nx（如果团队希望更多 framework consistency）— 通常不切

预期 ROI：CI 时间从 25 分钟到 5 分钟（CI 账单省 80%），幻影依赖根除，发版流程自动化。1 个月投入，长期收益。

场景 3 · 评估"切 Nx 还是 Turborepo"

背景：100 人团队，3 个产品线共用大型前端 monorepo（80 包）。已经用 pnpm + Turborepo 1 年，平稳。最近讨论是否切 Nx 拿 generator + migration 收益。

分析：

  • 切 Nx 的潜在收益：generator 让新人建包遵循模板；nx migrate 让大型 monorepo 升级成本低；DTE 分布式执行提速 CI
  • 切 Nx 的成本：迁移 80 包到 Nx 配置（每个包需 project.json / nx.json 适配）；团队学习新工具；团队文化"组合 vs 全家桶"切换

建议：

  • 如果团队感受 Turborepo 在具体痛点上不够用（升级成本高、新人建包不一致、CI 仍慢）→ 切 Nx 值得
  • 如果团队反馈"Turborepo 用着挺好" → 不要切。"It ain't broke, don't fix it"。投入精力到业务而不是工具迁移
  • 中间方案：保持 Turborepo + 自建少量 generator script（解决一致性问题）

关键：不要为切而切。哲学路线选定后稳定下来，长期收益最大。

04 典型代码

2026 中型 monorepo 完整初始化（pnpm + Turborepo + Changesets） 
# Step 1: 创建 monorepo 基础
mkdir my-monorepo && cd my-monorepo
pnpm init
echo "node_modules\ndist\n.turbo" > .gitignore
git init

# Step 2: 配 pnpm workspace + catalog
cat > pnpm-workspace.yaml <<EOF
packages:
  - 'apps/*'
  - 'packages/*'

catalog:
  typescript: ^5.6.0
  '@types/node': ^22.0.0
  vitest: ^1.6.0

catalogs:
  react18:
    react: ^18.3.1
    react-dom: ^18.3.1
EOF

# Step 3: 配 root package.json
cat > package.json <<EOF
{
  "name": "my-monorepo",
  "private": true,
  "packageManager": "pnpm@10.12.0",
  "engines": { "node": ">=22.0.0" },
  "scripts": {
    "build": "turbo run build",
    "dev": "turbo run dev",
    "test": "turbo run test",
    "lint": "turbo run lint",
    "typecheck": "turbo run typecheck",
    "ci": "pnpm install --frozen-lockfile && pnpm build && pnpm test && pnpm lint"
  },
  "devDependencies": {
    "turbo": "^2.0.0",
    "typescript": "catalog:"
  }
}
EOF

# Step 4: 配 turbo.json
cat > turbo.json <<EOF
{
  "\$schema": "https://turbo.build/schema.json",
  "globalDependencies": ["**/.env.*local", "tsconfig.base.json"],
  "globalEnv": ["NODE_ENV", "CI"],

05 常见陷阱

陷阱 1 · 选型用"hype 强弱"作依据

看到 Bun 火就全员切 Bun，看到 Nx 在 conference 频繁就上 Nx。选型应基于项目特征 + 团队能力。Bun 适合中型新项目；Nx 适合大型企业 + 全家桶哲学；不要逆向把工具适配项目。

陷阱 2 · 在中型项目上 Bazel

看到 Google 用 Bazel 觉得"大公司方案肯定好"。不要。Bazel 解决 10000+ 包 + 多语言 + 强可重现的问题，中型项目装 Bazel 等于"用核武打苍蝇"——6 个月学习曲线，复杂配置，最终没收益。

陷阱 3 · 频繁切工具栈

每年看到新东西就切：今年 Yarn → 明年 pnpm → 后年 Bun → 然后 Nx。每次切意味着 2 周迁移 + 团队学习成本 + 可能引入新 bug。选定后稳定 2-3 年，把精力投入业务。除非有具体痛点驱动，否则不切。

陷阱 4 · "all in 全家桶" 或 "all in 组合工具"

"必须 Nx" 或 "必须 Turborepo"——没有"必须"。两条哲学路线都有大型成功项目背书。看团队文化匹配，不要意识形态化。

陷阱 5 · 不考虑 5 年维护

选最新最快的工具，但 1 年后"维护人离职" → 团队搞不懂复杂配置 → 不敢动 → 6 年后还在跑 deprecated 工具。选型时考虑可维护性：文档质量、社区大小、迁移工具（如 nx migrate）。"能 happy 升级 5 年" 比 "装上时最快" 重要得多。
