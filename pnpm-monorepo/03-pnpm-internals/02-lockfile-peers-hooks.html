pnpm-lock.yaml + 严格 peer + hooks

上章讲了 pnpm 的"物理布局"（CAS + symlink）。本章讲它的"解析行为"——pnpm-lock.yaml 怎么记录依赖图、严格 peer 模式如何工作、.pnpmfile.cjs hooks 如何让你 patch 上游、pnpm patch 如何持久化代码修改。理解这些后你就有能力系统性管理 monorepo 依赖，不只是"pnpm install 然后祈祷"。

阅读时长 ~30 分钟 前置 P3.1（CAS + symlink 拓扑） 主要来源 pnpm.io 官方文档 + pnpm GitHub discussions + Zoltan Kochan blog

01 速览

  1. pnpm-lock.yaml 三段结构 —— (1) importers（每个 workspace package 的直接依赖 + 解析版本，monorepo 的入口）；(2) packages（所有 transitive 包的 metadata，包括 resolution / engines / peerDependencies）；(3) snapshots（每个 包@版本 在具体上下文中实际安装的依赖图——peer 解析后才确定）。
  2. YAML 不是任意选择 —— pnpm-lock 是diff 友好的：行级别变化清晰，merge conflict 容易解决。对比 package-lock.json 的嵌套 JSON（一处版本变化导致大段 diff），yaml 易读且自动化处理友好。
  3. 严格 peer dependency 模式 —— pnpm 默认对 peer dep 不自动满足，缺失会 warn。这与 npm/Yarn classic 默默装入的行为不同——pnpm 把"peer 是不是匹配"的责任放回用户，避免静默不一致。
  4. auto-install-peers —— pnpm 7+ 默认开启。如果某个 peer 缺失但有合理范围，pnpm 自动装。这是严格性 + 易用性的折衷——发包工程师有意识，普通用户不被 warn 淹没。
  5. resolution-mode —— pnpm 8+ 引入：highest（传统，取最新满足范围版本）/ time-based（取 lockfile 创建后才发布的版本以避免踩到刚发布烂包）/ lowest-direct（取最低满足范围的直接依赖，反向兼容老用户）。time-based 是新项目推荐。
  6. .pnpmfile.cjs 是 monorepo 的"系统 patch 中心" —— 通过 readPackage(pkg) hook 改写任何包的 package.json（fix 错误的 peer range、强制版本统一、添加 missing dep）。比 patch-package 更轻量——不改源码，只改 metadata。
  7. pnpm patch <pkg> 是代码级 patch —— 把包源码 checkout 到 node_modules/.pnpm/.tmp/，编辑后 pnpm patch-commit <path> 生成 .patch 文件并写入 lockfile。重装时自动应用。替代社区的 patch-package + 集成进 lockfile。
  8. side-effects-cache（P3.1 提过）—— 与 lockfile 配合：postinstall 副作用按 lockfile 中的 包@版本 缓存，重装时跳过。

02 底层逻辑

pnpm-lock.yaml 三段结构详解

用一个 monorepo 例子（apps/web 和 packages/ui，apps/web 依赖 react + @app/ui，@app/ui 依赖 react）展开。

pnpm-lock.yaml 完整骨架 
lockfileVersion: '9.0'

settings:
  autoInstallPeers: true
  excludeLinksFromLockfile: false

# Section 1: importers - 每个 workspace package 的直接依赖
importers:
  .:
    devDependencies:
      typescript:
        specifier: ^5.6.0
        version: 5.6.3

  apps/web:
    dependencies:
      '@app/ui':
        specifier: workspace:*
        version: link:../../packages/ui          # workspace 内部用 link
      react:
        specifier: ^18.3.1
        version: 18.3.1

  packages/ui:
    dependencies:
      react:
        specifier: ^18.3.1
        version: 18.3.1

# Section 2: packages - 所有外部包的 metadata
packages:
  react@18.3.1:
    resolution: {integrity: sha512-...}
    engines: {node: '>=0.10.0'}

  scheduler@0.23.2:
    resolution: {integrity: sha512-...}

  loose-envify@1.4.0:
    resolution: {integrity: sha512-...}
    hasBin: true

# Section 3: snapshots - 每个包在上下文中的实际依赖图
snapshots:
  react@18.3.1:
    dependencies:
      loose-envify: 1.4.0

  scheduler@0.23.2:
    dependencies:
      loose-envify: 1.4.0

  loose-envify@1.4.0:
    dependencies:
      js-tokens: 4.0.0

  js-tokens@4.0.0: {}

importers（最高层）：

  • . 是根 workspace（顶层 package.json）
  • monorepo 中每个 package（apps/web / packages/ui）都是一个 importer
  • 每个 importer 列出它直接依赖的包：specifier（package.json 写的范围）+ version（pnpm 解析出的具体版本）
  • workspace 内部依赖用 version: link:<path> 表示——这不会出现在 packages section

packages（中间层）：

  • 所有外部包（不是 workspace 内）的元数据
  • 每个 包@版本 一个条目：integrity hash（防篡改）+ engines（node 版本要求）+ peerDependencies（peer 范围）+ hasBin（是否有 CLI）
  • 这一段不包含依赖关系——只是包的"身份证"

snapshots（具体实例化层）：

  • 每个 包@版本 在具体上下文中的依赖图
  • "上下文"指 peer 解析结果——同一个 react@18.3.1 在依赖它的不同包看来，配套的 react-dom 可能是不同 instance，pnpm 用括号表示
  • 例如 react-dom@18.3.1(react@18.3.1) 表示"react-dom 18.3.1 在 react 18.3.1 上下文下的解析"
  • 这是 lockfile 最复杂、变化最频繁的部分

peer 解析的复杂性 · 为什么需要 snapshots

问题：同一个 包@版本（比如 react-dom@18.3.1）声明 peerDependencies: { react: "^18" }。在不同上下文下，react 的具体实例可能不同：

  • app 直接装了 react@18.3.1 + react-dom@18.3.1 → react-dom 看到的 react 是顶层那个
  • 某个 dep 装了 react@18.2.0 + react-dom@18.3.1 → react-dom 看到的 react 是 18.2.0
  • monorepo 内 packages/ui 用 react@18.3.1，packages/admin 用 react@18.3.0 → 两份 react-dom（虽然 SHA 相同）需要分别配套

pnpm 给每个 包@版本(peer 上下文) 创建独立的 sandbox 目录：

node_modules/.pnpm 中的 peer 区分 
node_modules/.pnpm/
├── react-dom@18.3.1(react@18.3.1)/      ← peer = react 18.3.1
│   └── node_modules/
│       ├── react-dom/
│       └── react → ../../react@18.3.1/...
└── react-dom@18.3.1(react@18.3.0)/      ← peer = react 18.3.0
    └── node_modules/
        ├── react-dom/                    ← 同样的物理文件（hardlink 到同 store）
        └── react → ../../react@18.3.0/...

这就是 pnpm 严格 peer 模式的物理实现——每个 peer 上下文都有独立 sandbox，让运行时拿到正确的 peer instance。npm/Yarn classic 因为扁平 hoisting，无法做到这种区分（最多 dedupe 到一个）。

严格 peer dependency · pnpm 的核心论点

peerDependencies 字段 npm 早就有，但处理行为分歧：

  • npm 7+：peer 不满足时自动装满足版本（这是 npm 对老用户的折衷）
  • Yarn classic：peer 不满足只 warn，需要用户自己装
  • pnpm 默认（< 7.x）：peer 不满足只 warn，不自动装
  • pnpm 7+ auto-install-peers=true 默认：自动装

为什么 pnpm 早期默认不装？因为 peer 自动装可能掩盖真正的问题：

  • 包作者错误声明 peer 范围（写成 react: ^17 实际兼容 18），用户被自动装了不必要的 react@17
  • monorepo 中某个内部包错误把 peer 当 dep，自动装导致版本不一致
  • 静默自动装让用户失去对依赖图的控制

pnpm 7+ 妥协：开启 auto-install-peers 但保持严格的 peer 不匹配警告——peer 满足时不打扰，不满足时显式 warn。这平衡了"易用性"和"严格性"。

strict-peer-dependencies 选项：

.npmrc 
# 默认（pnpm 8+）
auto-install-peers=true               # 自动装
strict-peer-dependencies=false        # peer 不匹配只 warn

# 严格模式（CI 推荐）
auto-install-peers=true
strict-peer-dependencies=true         # peer 不匹配直接报错，install 失败

resolution-mode · 防"刚发布烂包"

2024 年 pnpm 8 引入新选项 resolution-mode，解决一类痛点：

背景：你 pnpm install 时 npm 上某包刚发布有 bug 的 X.Y.0，pnpm 默认按 highest 模式解析 → 装上烂包。等到几小时后包作者发布 X.Y.1 修复，但你已经踩坑。

三种 mode：

highest（传统）：

  • 每次 install 取范围内最新已发布版本
  • 风险：刚发布的烂包立刻进入
  • 这是 npm/Yarn 的传统行为

time-based（pnpm 9+ 默认）：

  • 第一次装某包时记录"cutoff time"——之后只考虑 cutoff time 之前发布的版本
  • 新项目自动取最新，但已存在的 lockfile 不会被刚发布的版本污染
  • 显式 pnpm update 才升级（提供主动控制）
  • 降低踩刚发布烂包的概率

lowest-direct：

  • 取范围内最低满足版本（保守）
  • 用于"我要测试低版本兼容性"的场景
  • 不是日常推荐

实测效果：time-based 让"昨天还跑得好的项目今天 install 后崩了"几乎消失。这是 2024 年 pnpm 最受好评的改进之一。

.pnpmfile.cjs hooks · monorepo 的"metadata patch 中心"

.pnpmfile.cjs 放在 monorepo 根目录，pnpm install 时自动加载。提供两个 hooks：

.pnpmfile.cjs 
module.exports = {
  hooks: {
    // 每解析到一个包的 package.json 都调用
    readPackage(pkg, context) {
      // pkg 是这个包的 package.json 内容
      // 可以修改后返回，影响后续解析

      // 例 1: fix 上游错误的 peer range
      if (pkg.name === 'some-old-pkg' && pkg.peerDependencies?.react) {
        pkg.peerDependencies.react = '^17 || ^18 || ^19';
      }

      // 例 2: 强制把某个 dep 改成 peer
      if (pkg.name === 'wrong-pkg') {
        if (pkg.dependencies?.react) {
          pkg.peerDependencies = pkg.peerDependencies || {};
          pkg.peerDependencies.react = pkg.dependencies.react;
          delete pkg.dependencies.react;
        }
      }

      // 例 3: 删除 monorepo 中不需要的 optionalDependencies
      if (pkg.name === 'fsevents-using-pkg') {
        delete pkg.optionalDependencies;
      }

      return pkg;
    },

    // lockfile 完成解析后调用
    afterAllResolved(lockfile, context) {
      // 可以做最后的整理
      // 例如：log 出当前所有 react 版本
      const reactVersions = new Set();
      for (const pkg of Object.keys(lockfile.packages || {})) {
        if (pkg.startsWith('react@')) {
          reactVersions.add(pkg);
        }
      }
      if (reactVersions.size > 1) {
        console.warn('Multiple React versions detected:', [...reactVersions]);
      }
      return lockfile;
    }
  }
};

实战场景：

场景 A · 修上游错误的 peer 范围：

some-old-pkg 声明 peerDependencies: { react: "^17" }，但实际兼容 react@18。每次装 react@18 都报 peer 不匹配 warn。在 .pnpmfile.cjs 里 readPackage 改写 some-old-pkg 的 peer 到 ^17 || ^18。

场景 B · 把 monorepo 内部某个版本统一：

monorepo 中 50 个包用 typescript，但写了不同的范围（4.x, 5.0, 5.6）。在 .pnpmfile.cjs 里检测到 typescript 时统一改写到 5.6。当然 catalog 是更现代的解（详见 P4.1），但 .pnpmfile.cjs 给老 monorepo 一个迁移路径。

场景 C · 移除不需要的 optionalDependencies：

某些包有 fsevents（macOS 文件监听）作为 optional。在 Linux CI 上每次 fail-but-continue 浪费时间。.pnpmfile.cjs 里删除可避免。

差异于 patch-package：.pnpmfile.cjs 改的是解析阶段的 package.json metadata，不改实际代码文件。要改源码用 pnpm patch（下面）。

pnpm patch · 代码级修改持久化

2022 年 pnpm 7 引入官方 patch 子系统，替代社区的 patch-package。

完整工作流 
# Step 1: 把包 checkout 到临时目录
pnpm patch some-pkg@1.2.3
# > You can now edit the following folder:
# > /private/var/folders/.../node_modules/.tmp/some-pkg-1.2.3-X

# Step 2: 在那个目录里改代码
cd /private/var/folders/.../some-pkg-1.2.3-X
vim lib/index.js   # 修 bug

# Step 3: 提交 patch
pnpm patch-commit /private/var/folders/.../some-pkg-1.2.3-X
# 自动：
# - 生成 patches/some-pkg@1.2.3.patch 文件
# - 写入 package.json 的 pnpm.patchedDependencies 字段
# - 更新 pnpm-lock.yaml

# Step 4: commit patches/ 到 git
git add patches/ package.json pnpm-lock.yaml
git commit -m "fix: patch some-pkg buggy behavior"

# 之后任何人 / CI install 都自动应用 patch

package.json 中的存储：

json 
{
  "name": "my-monorepo",
  "pnpm": {
    "patchedDependencies": {
      "some-pkg@1.2.3": "patches/some-pkg@1.2.3.patch",
      "another@5.0.0": "patches/another@5.0.0.patch"
    }
  }
}

与 patch-package 对比：

  • pnpm patch 原生集成 lockfile，不用额外 npm install patch-package
  • patch 文件路径在 package.json，直观
  • 支持 monorepo（patch 应用到所有 workspace package 的依赖）
  • side-effects-cache 与 patch 配合（patch 后不重跑 postinstall）

使用建议：

  • patch 是临时方案——目标始终是把修复推到上游
  • 每个 patch 在 PR 中说明"为什么 + 上游 issue 链接 + 临时性"
  • 定期审查 patches/ 目录，删除已被上游修复的 patch
  • 不要把 patch 当作"分叉私有版本"——版本升级时 patch 可能失效

frozen-lockfile · CI 的安全开关

CI 场景下 pnpm install --frozen-lockfile 是必需的：

  • 不修改 pnpm-lock.yaml——任何让 lockfile 需要更新的情况都报错
  • 等价于 npm 的 npm ci、Yarn 的 yarn install --immutable
  • 防止 CI 静默升级依赖
  • 验证 lockfile 与 package.json 一致

什么时候会 fail：

  • 有人改了 package.json 但没跑 pnpm install 重新生成 lockfile
  • lockfile 与 package.json 范围不匹配
  • workspace 包 add/remove 但 lockfile 未更新

这时候 dev 应该本地 pnpm install + commit 新 lockfile，而不是在 CI 跳过此检查。

⚠️ 常见误用：在 CI 用 pnpm install 而非 --frozen-lockfile

不带 --frozen-lockfile 的 install 在 CI 上会自动更新 lockfile，但更新不会被 commit 回去——下次 CI 还会"重新更新"。这造成"lockfile 漂移"——每次 CI 装的依赖版本可能不一样，构建不可重现。CI 永远用 --frozen-lockfile。

03 应用场景

场景 1 · 修上游包错误的 peer 范围

背景：用 react@19，但某个老包 some-chart-lib peer 写 react: "^18"。每次 install 报 peer 不匹配。已经向上游提 PR 但还没 merge。

方案：在 .pnpmfile.cjs 里改写：

javascript 
module.exports = {
  hooks: {
    readPackage(pkg) {
      if (pkg.name === 'some-chart-lib') {
        pkg.peerDependencies = {
          ...pkg.peerDependencies,
          react: '^18 || ^19'   // 扩展到 19
        };
      }
      return pkg;
    }
  }
};

commit .pnpmfile.cjs。所有团队成员 install 时自动 apply。上游 PR merge 后删除这段 hook。

场景 2 · 修包的代码 bug

背景：some-pkg@1.2.3 有 bug 导致 SSR 崩溃。已经报 issue 但作者还没修。

方案：

bash 
pnpm patch some-pkg@1.2.3
# 终端会输出临时目录路径，例如：
# /private/var/folders/.../some-pkg-1.2.3-XXX

# 在编辑器打开那个目录
code /private/var/folders/.../some-pkg-1.2.3-XXX

# 修 lib/server.js 中的 bug
# 修完后保存

pnpm patch-commit /private/var/folders/.../some-pkg-1.2.3-XXX
# 生成 patches/some-pkg@1.2.3.patch
# 写入 package.json 的 pnpm.patchedDependencies

git add patches/ package.json pnpm-lock.yaml
git commit -m "fix: patch some-pkg SSR crash (upstream issue #123)"

所有团队成员 + CI install 时自动 apply patch。

场景 3 · CI 报"lockfile out of sync"

背景：开发提了一个 PR 改了 package.json，CI 报 --frozen-lockfile failed: package.json is not in sync with lockfile。

诊断 + 修复：

  • 本地 pnpm install（不带 --frozen-lockfile）
  • 看 git diff pnpm-lock.yaml——pnpm 会更新 lockfile
  • 确认更新合理（没有意外引入大版本变化）
  • git add pnpm-lock.yaml && git commit -m "chore: update lockfile"
  • push 到 PR 分支，CI 应该过

预防：让团队所有人 git pre-commit hook 强制：改 package.json 必须 commit 对应的 lockfile 变化。

读 lockfile 找单个包的所有版本 
# 看 react 装了几个版本
grep -E '^  react@' pnpm-lock.yaml | head -10

# 用 pnpm 命令更准确

# 看哪些包依赖某个版本

.pnpmfile.cjs 实战 · monorepo 版本统一 
// .pnpmfile.cjs
const REQUIRED_REACT = '^18.3.0';

module.exports = {
  hooks: {
    readPackage(pkg, context) {
      // 1. 任何 peer react 都接受 18.3+
      if (pkg.peerDependencies?.react) {
        pkg.peerDependencies.react = REQUIRED_REACT;
      }
      // 2. 任何 dep react 也用 18.3+（强制升级）
      if (pkg.dependencies?.react) {
        pkg.dependencies.react = REQUIRED_REACT;
      }
      return pkg;
    },

    afterAllResolved(lockfile, context) {
      // 验证：lockfile 中应只有一个 react 版本
      const reactVersions = Object.keys(lockfile.packages || {})
        .filter(k => k.startsWith('react@'))
        .filter(k => !k.startsWith('react-'));   // 排除 react-dom 等
      if (reactVersions.length > 1) {
        throw new Error(
          `Multiple React versions detected: ${reactVersions.join(', ')}`
        );
      }
      return lockfile;
    }
  }
};

05 常见陷阱

陷阱 1 · 手动编辑 pnpm-lock.yaml

"我把 lockfile 里某个版本改一下吧" —— 不要直接编辑 lockfile。手动改容易让 importers / packages / snapshots 三段不一致，pnpm install 时会报错或静默重生成（覆盖你的修改）。正确做法：改 package.json + 跑 pnpm install，让 pnpm 重新解析。

陷阱 2 · CI 不用 --frozen-lockfile

很多团队 CI workflow 写 pnpm install（不带 frozen），导致 CI 静默改 lockfile 但不 commit。下次 CI 又"重新改"——构建不可重现。修复：CI 永远 --frozen-lockfile，让 lockfile 与代码同步严格化。

陷阱 3 · 把 .pnpmfile.cjs 当万能补丁

有人在 .pnpmfile.cjs 里改写几十个包的 metadata，最后这个文件成"项目隐性配置中心"。新加入团队的人完全看不懂为什么"装出来就这样"。建议：每个 hook 加注释说明"为什么 + 上游 issue 链接"；定期清理已不需要的 hook；优先用 catalog / overrides / peer 自然机制。

陷阱 4 · 用 pnpm patch 修代码不更新 patch 当版本变化

patch 是针对具体版本的。包从 1.2.3 升到 1.2.4 后原 patch 可能 fail。建议：每次包升级前看 patches/ 目录，重新评估每个 patch 是否需要 / 是否要 rebase。CI 里 install 报 patch fail 时立即处理。

陷阱 5 · 关掉 strict-peer-dependencies 然后忽略所有 warn

strict-peer-dependencies=false 是 dev 友好的默认，但有人就此忽略所有 peer warning。这些 warning 反映的真实依赖图问题不会因为你不看就消失。建议：dev 用 false（不打断流程），CI 用 true（强制修），定期跑一次 strict 模式检查。
