内容寻址存储 (CAS) + symlink 拓扑

pnpm 不是"一个更快的 npm"，是用 CAS + 链接拓扑彻底重做 node_modules 的物理布局。本章把这套机制——全局 store + hardlink + 三层 symlink——讲清楚。理解了它你才知道为什么 pnpm 既能省磁盘 70-95%，又能从结构上消除幻影依赖（P2.2 的痛点）。

阅读时长 ~30 分钟 前置 P2 全部（resolution + layout + 幻影依赖） 主要来源 pnpm.io / Zoltan Kochan blog + GitHub discussions / pnpm 源码

01 速览

  1. pnpm 的设计是 2 层 —— (1) 全局 CAS store（~/.pnpm-store/，按 SHA-512 hash 切片，每个 包@版本 全机一份物理文件）；(2) 项目本地 symlink 拓扑（用 hardlink 引用 store + symlink 控制可见性）。两层组合实现"磁盘最省 + 严格性最高"双赢。
  2. 3 层链接拓扑：
    • Layer 1：~/.pnpm-store/files/<hash> → 物理文件（全机唯一）
    • Layer 2：node_modules/.pnpm/<name>@<ver>/node_modules/<name>/... → hardlink 到 store
    • Layer 3：node_modules/<name> → symlink 到 .pnpm 中的版本目录（仅声明的依赖出现在顶层）
  3. 磁盘节省真实数字 —— 同一台机器装 5 个相同 React 项目：npm 占 1.25 GB（每个 250 MB × 5）；pnpm 占 ~250 MB（store 一份 + 5 个项目仅 hardlink，几乎零额外）。多项目场景节省 70-95%。
  4. Hardlink vs Reflink vs Copy 三种引用方式 —— hardlink 是默认（同分区零成本）；reflink（APFS / Btrfs / XFS / ZFS）是 copy-on-write 的 hardlink，跨子卷可用；copy 是退路（跨设备 / 不支持 hardlink 的网络盘）。pnpm 自动检测，不用手动配。
  5. node-linker 选项 —— 默认 isolated（symlink 拓扑，原 pnpm 模式）；可选 hoisted（npm 风格扁平，兼容老工具）；pnp（实验性，类似 Yarn PnP）。除非有兼容性需求，保持默认。
  6. side-effects-cache —— pnpm 8.0+ 默认开启。缓存 postinstall 脚本副作用（如 esbuild 下载二进制、prisma generate）；reinstall 时直接复用，省时间。
  7. Windows 一级支持 —— Windows 10+ 默认支持 hardlink 和 directory junction（替代 symlink）。pnpm 在 Windows 上行为与 macOS / Linux 一致；2017 年早期的 Windows 兼容问题已基本解决。

02 底层逻辑

什么是内容寻址存储（CAS）

"内容寻址" 来自 git 等系统。原则：把文件的 hash 作为它的存储地址。

具体到 pnpm：

  • 装包时把 tarball 解压，每个文件按内容 SHA-512 hash 单独存放到 ~/.pnpm-store/files/<hash[0:2]>/<hash[2:]>
  • 同样内容的文件，无论来自哪个包哪个版本，都映射到同一个 hash 地址，物理上只存一份
  • store 还存了 metadata：包@版本 对应的文件清单（哪些 hash 组成这个包）

实际效果：跨包文件去重。例如 react@18.3.1 和 react@18.3.0 大部分文件可能内容相同（仅 1-2 个文件改了）；CAS 下相同的文件复用同一份物理存储，仅不同的部分单独存。

CAS 的另一个名字：content-addressable filesystem。这套思路在 git / IPFS / Docker layer / Nix store 等系统中都见到。pnpm 是 JS 包管理领域第一个工业级 CAS 实现。

3 层链接拓扑详解

用一个具体例子展开。my-app 直接依赖 react 和 lodash；react 传递依赖 loose-envify，loose-envify 又依赖 js-tokens。

完整 pnpm 拓扑 
# Layer 1: 全局 CAS store（每台机器只有一份）
~/.pnpm-store/
├── v3/
│   ├── files/
│   │   ├── 0a/
│   │   │   └── b1c2...  ← 物理文件（按 hash 寻址）
│   │   ├── 1f/
│   │   └── ...
│   └── index/
│       └── files/        ← 包到 hash 的索引

# Layer 2: 项目本地 .pnpm 目录（每个项目自己一份）
my-app/
└── node_modules/
    ├── .pnpm/
    │   ├── react@18.3.1/
    │   │   └── node_modules/
    │   │       ├── react/                     ← hardlink 到 ~/.pnpm-store/files/...
    │   │       │   ├── package.json
    │   │       │   ├── index.js
    │   │       │   └── lib/...
    │   │       └── loose-envify              ← symlink → ../../loose-envify@1.4.0/...
    │   ├── lodash@4.17.21/
    │   │   └── node_modules/
    │   │       └── lodash/                    ← hardlink 到 ~/.pnpm-store/files/...
    │   ├── loose-envify@1.4.0/
    │   │   └── node_modules/
    │   │       ├── loose-envify/              ← hardlink 到 store
    │   │       └── js-tokens                  ← symlink → ../../js-tokens@4.0.0/...
    │   └── js-tokens@4.0.0/
    │       └── node_modules/
    │           └── js-tokens/                 ← hardlink 到 store
    │
    # Layer 3: 项目顶层（只有声明的依赖）
    ├── react        ← symlink → .pnpm/react@18.3.1/node_modules/react
    └── lodash       ← symlink → .pnpm/lodash@4.17.21/node_modules/lodash

每层的角色：

Layer 1（CAS store）：

  • 全机唯一，所有项目共享
  • 同一 包@版本 全机只存一份
  • 更进一步：同 hash 的文件全机也只存一份（跨包去重）
  • store 不直接被 Node 读——只通过 hardlink 暴露给项目

Layer 2（.pnpm 沙箱）：

  • 每个 包@版本 在项目内有独立的 sandbox 目录
  • 包文件用 hardlink 指向 store（零拷贝）
  • 它的传递依赖在同一 sandbox 内用 symlink 引用——这样 Node resolution 找它的依赖时只能看到 .pnpm 中对应版本
  • 关键：每个包的依赖图是封闭的，不会"漂"到别的包的依赖

Layer 3（项目顶层）：

  • 只有 package.json 直接声明的依赖出现
  • 每个顶层条目是 symlink，指向 .pnpm 中对应版本
  • Node resolution 从源码出发，顶层 node_modules 看不到的包就 require 不到——幻影依赖结构性消除

从源码 require 的视角追一遍

假设 my-app/src/main.js 写：

javascript 
// my-app/src/main.js
const React = require('react');           // ✅ 声明的
const lodash = require('lodash');          // ✅ 声明的
const envify = require('loose-envify');    // ❌ 未声明（幻影）

Node resolution 走法：

  • require('react') 从 my-app/src 出发逐级向上 → my-app/src/node_modules（无）→ my-app/node_modules/react（命中 symlink）→ 跟随 symlink 到 .pnpm/react@18.3.1/node_modules/react ✅
  • require('lodash') 同理，命中顶层 symlink ✅
  • require('loose-envify') 从 my-app/src 出发逐级向上 → my-app/node_modules（顶层没有 loose-envify！）→ /node_modules（无）→ throw MODULE_NOT_FOUND ❌

这就是结构性消除幻影依赖——Node 算法不变，pnpm 通过摆放让"未声明的依赖"在 resolution 路径上不可见。

反过来，从 react 内部代码 require：

javascript 
// node_modules/.pnpm/react@18.3.1/node_modules/react/lib/foo.js
const envify = require('loose-envify');   // ✅ react 声明了 loose-envify
  • Node 从 .pnpm/react@18.3.1/node_modules/react/lib 出发逐级向上
  • 第一站 .pnpm/react@18.3.1/node_modules（命中 loose-envify symlink → .pnpm/loose-envify@1.4.0/node_modules/loose-envify）✅
  • 每个包的依赖在自己的 sandbox 内能找到，但不会跨包污染

磁盘节省真实数字

用一个典型 React + Next.js 项目（300+ 包，~250 MB node_modules）作对比。复现条件：macOS APFS / 同一台机器 / 同一个 package.json。

场景 npm 占用 pnpm 占用（项目内） pnpm 占用（store + 项目） 节省
第一次装项目 250 MB 0（全 hardlink） 250 MB（store 首装） 0
装第 2 个相同项目 +250 MB（共 500） +0（共 0） +0（store 已有） ~100%（增量）
装第 5 个相同项目 +250 MB（共 1.25 GB） +0（共 0） +0（store 已有） ~100%（增量）
装第 6 个升级的项目（30% 包升级） +250 MB +0 +~75 MB（仅新版本） ~70%（增量）

结论：

  • 第一次装某项目时 pnpm 不省（store 仍要存）
  • 第二次起省得显著（hardlink 零开销）
  • 多项目共享同版本依赖时，节省最大
  • 多版本同时存在时，仍按版本独立存（不能跨版本去重）

对个人开发者（一台机器多个项目）的实际收益：磁盘节省 50-200 GB 不夸张。对 CI（每次 fresh install）收益小（仍需下载到 store）。

Hardlink vs Reflink vs Copy 三种引用方式

Hardlink（硬链接）：

  • 同一文件系统内多个目录项指向同一个 inode
  • 零额外磁盘开销，零 IO 开销
  • 只能在同一 filesystem / 同一分区内
  • 修改 hardlink 文件等于修改 store 文件——但 pnpm 把 store 文件设为 read-only，意外修改时会复制（copy-on-write 行为）
  • 这是 pnpm 默认且最优的引用方式

Reflink（CoW 链接，APFS / Btrfs / XFS / ZFS / ReFS）：

  • 看起来是独立文件，但底层共享 block，写时才 copy（copy-on-write）
  • 跨子卷 / 跨 dataset 可用（hardlink 不行）
  • 修改一份不影响另一份（与 hardlink 不同）
  • 需要 filesystem 支持。macOS APFS / Linux Btrfs+XFS / Windows ReFS（少见）支持
  • pnpm 自动检测——APFS 上比 hardlink 更安全（因为修改不会 propagate）

Copy（复制，退路）：

  • 真实复制文件内容
  • 跨设备 / 网络盘 / 不支持 hardlink 时使用
  • 占用 N 倍磁盘
  • 速度最慢
  • pnpm 仅在前两种都不行时回落

配置选项：

.npmrc 
# 自动选择最优（默认；推荐保持）
package-import-method=auto

# 强制用 hardlink（同分区时最快）
package-import-method=hardlink

# 强制 copy（用于 store 在网络盘 / 项目在本地的情况）
package-import-method=copy

# 强制 clone（reflink）
package-import-method=clone

# Linux 上 Btrfs 使用 reflink
package-import-method=clone-or-copy

跨分区 / 跨文件系统的退路

典型场景：开发者把 ~/.pnpm-store 放在快速 NVMe SSD（系统盘），但项目放在大容量 HDD 或外置硬盘。这种情况下 hardlink 不能跨分区，pnpm 自动 fallback 到 copy（或者 reflink，如果两边都是 Btrfs/ZFS）。

解决方案：把 store 移到与项目同分区：

bash 
# 看当前 store 路径
pnpm store path
# /Users/me/.pnpm-store

# 如果项目在 /Volumes/external/projects/my-app，把 store 也移过去
pnpm config set store-dir /Volumes/external/.pnpm-store

# 或者写到 .npmrc
echo "store-dir=/Volumes/external/.pnpm-store" >> ~/.npmrc

Docker / CI 场景：CI 通常用 ephemeral container，每次都 fresh install。store 不在镜像里就没意义。常见做法：

  • 用 GitHub Actions / GitLab cache 缓存 ~/.pnpm-store 跨 build 复用
  • Docker 用 BuildKit 的 --mount=type=cache,id=pnpm-store 跨 build 复用
  • 在 Dockerfile 中显式 pnpm fetch --prod 把 store 装好作为镜像层

node-linker 选项

pnpm 8+ 引入 node-linker 配置，让用户在严格性 / 兼容性间选：

node-linker=isolated（默认）：上面讲的 symlink 拓扑。严格 + 高效，但少数老工具不兼容。

node-linker=hoisted：

  • 装出来类似 npm 的扁平 hoisting 布局
  • 顶层 node_modules 包含所有传递依赖（牺牲严格性）
  • 但 store + hardlink 仍保留（仍省磁盘）
  • 用于：兼容某些假设扁平布局的老工具（如 React Native 早期版本、某些 Native binding 工具）

node-linker=pnp（实验性）：

  • 类似 Yarn PnP，无 node_modules + .pnp.cjs loader
  • 2026 年仍 experimental，生态成熟度低于 Yarn PnP
  • 不推荐生产

选哪个：

  • 99% 项目用默认 isolated——这是 pnpm 价值的核心
  • 只在工具兼容性确实卡住时才换 hoisted，且要清楚牺牲了严格性
  • pnp 留给极客团队 + Yarn PnP 用户做对比试验

side-effects-cache · 重装时的副作用快进

很多包有 postinstall 脚本：

  • esbuild：下载平台对应的 native binary
  • sharp：编译 libvips bindings
  • prisma：generate type definitions
  • better-sqlite3：编译 native module

第一次装包跑这些脚本要几秒到一分钟。重装时 pnpm 默认缓存这些副作用到 store——下次 install 直接复用，不再跑脚本。这就是 side-effects-cache。

开关：

.npmrc 
# pnpm 8.0+ 默认开启
side-effects-cache=true

# 关闭（在 CI 上某些场景需要）
side-effects-cache=false

实际收益：典型项目的 pnpm install --frozen-lockfile（CI 模式）：

  • 第一次：~30 秒（下载 + postinstall）
  • 第二次（store 已有）：~5 秒
  • 第二次但关 cache：~15 秒（要重跑 postinstall）

Windows 一级支持

2017 年 pnpm 早期，Windows 用户报告大量"symlink 创建失败"问题——Windows 创建 symlink 默认要管理员权限。pnpm 改用 directory junction（Windows 自家的目录链接，不需要管理员权限）替代 symlink，问题大幅缓解。

2026 年的 Windows 现状：

  • Windows 10 1703+ 默认开启"开发者模式"后 symlink 不需要管理员
  • pnpm 在 Windows 上行为与 macOS / Linux 一致
  • Hardlink 和 junction 都已稳定支持
  • NTFS 上 hardlink 极快（媲美 Linux ext4）

剩余 Windows 问题：长路径（> 260 字符）。Win10+ 的 LongPathsEnabled 注册表项可启用 32K 路径，pnpm 利用此特性避免大型 monorepo 路径长度问题。

💡 关键洞见：pnpm 的"严格 + 省磁盘"是同一套机制的两面

表面上看，"严格性"和"省磁盘"是两个独立优点。但 pnpm 的 CAS + 链接拓扑设计让它们其实是同一个机制的两面——把每个 包@版本 物理上唯一存放（省磁盘）+ 用 symlink 控制可见性（严格性）。这是 pnpm 设计的精妙处：不是用复杂换两个优点，是用一个简洁机制同时拿两个。

03 应用场景

场景 1 · 个人开发者磁盘吃紧

背景：开发者笔记本 256 GB SSD，30+ 个 Node 项目。du -sh ~/code 显示 60 GB，其中 50 GB 是 node_modules。怎么省？

方案：

  • 所有项目切 pnpm（删 node_modules + package-lock.json，pnpm install）
  • 预期：第一个项目 store 装 ~250 MB，后续项目 hardlink 几乎 0
  • 30 个项目装完后总占用：1-2 GB（store）+ ~50 MB × 30（仅 node_modules 元数据）= ~3 GB
  • 从 50 GB → 3 GB，节省 47 GB

这是 pnpm 个人开发者收益最大的场景。

场景 2 · CI 速度优化（store 跨 build 缓存）

背景：GitHub Actions 上 pnpm install 每次 ~30 秒。能否更快？

方案：

yaml 
# .github/workflows/ci.yml
- uses: pnpm/action-setup@v4
- name: Get pnpm store path
  id: pnpm-cache
  run: echo "STORE_PATH=$(pnpm store path)" >> $GITHUB_OUTPUT

- uses: actions/cache@v4
  with:
    path: ${{ steps.pnpm-cache.outputs.STORE_PATH }}
    key: ${{ runner.os }}-pnpm-store-${{ hashFiles('**/pnpm-lock.yaml') }}
    restore-keys: |
      ${{ runner.os }}-pnpm-store-

- run: pnpm install --frozen-lockfile

cache 命中后 install 时间从 30 秒降到 ~5 秒。

场景 3 · 用 Docker BuildKit 缓存 store

背景：Docker 镜像构建时每次 pnpm install 都重头来。怎么用缓存？

方案：

dockerfile 
# Dockerfile
# syntax=docker/dockerfile:1.4
FROM node:22-alpine
RUN corepack enable

WORKDIR /app
COPY package.json pnpm-lock.yaml ./

# BuildKit cache mount: store 跨 build 复用
RUN --mount=type=cache,id=pnpm,target=/root/.local/share/pnpm/store \
    pnpm install --frozen-lockfile

COPY . .
RUN pnpm build

# 构建命令
# DOCKER_BUILDKIT=1 docker build -t my-app .

第二次 build 时 cache mount 复用，install 部分接近瞬时。

04 典型代码

查看 store 信息 
# store 路径
pnpm store path

# store 状态（哪些包、占用多少）
pnpm store status

# 清理未被任何项目引用的包（节省磁盘）
pnpm store prune

# 检查 store 完整性（hash 校验）
pnpm store status --verify-store-integrity
.npmrc 关键配置 
# pnpm 推荐的 .npmrc
auto-install-peers=true               # 自动装 peer 依赖（详见 P3.2）
strict-peer-dependencies=false        # 严格但不报错（warn）；true 时报错
package-import-method=auto            # 自动选 hardlink/reflink/copy
side-effects-cache=true               # 缓存 postinstall（默认 true）
node-linker=isolated                  # 默认严格拓扑

# CI 推荐
frozen-lockfile=true                  # 装包时不修改 lockfile
查看具体包的 hardlink 关系 
# Linux/macOS
ls -li node_modules/.pnpm/react@18.3.1/node_modules/react/index.js
# 1234567 -rw-r--r-- 5 me me 4096 ... index.js
#                    ^ inode + link count（5 个 hardlink 指向同 inode）

# 找出哪些项目共享了这个文件
find ~/.pnpm-store -inum 1234567

# Windows（PowerShell）
Get-Item node_modules\.pnpm\react@18.3.1\node_modules\react\index.js |
  Select-Object FullName, LinkType, Target
验证幻影依赖被消除 
# 用 pnpm 装一个仅声明 react 的项目
mkdir test-pnpm && cd test-pnpm
echo '{"dependencies":{"react":"^18"}}' > package.json
pnpm install

# 顶层只有 react
ls node_modules/
# react

# 不能 require 未声明的包
node -e "require('loose-envify')"
# Error: Cannot find module 'loose-envify'

# 但在 react 内部 require 是 OK 的
ls node_modules/.pnpm/react@18.3.1/node_modules/
# react  loose-envify   ← react 的依赖在它的 sandbox 内

05 常见陷阱

陷阱 1 · 把 store 放在不同分区导致 fallback 到 copy

用户把项目放在 D 盘 / 外置硬盘，但 store 仍在 C 盘 ~/。pnpm 自动 fallback 到 copy，磁盘占用恢复到 npm 水平。修复：把 store 也移到项目同分区（pnpm config set store-dir D:\\.pnpm-store）。

陷阱 2 · 修改 node_modules 内的代码，意外同步到所有项目

调试时直接改 node_modules 里某个包的源代码——因为是 hardlink，修改会反映到 store + 所有共享同包的项目。pnpm 默认把 store 文件设为 read-only 防止此类意外，但仍偶有用户绕过权限改写。正确做法：用 pnpm patch <pkg> 创建独立 patch（详见 P3.2）。

陷阱 3 · 在 Docker 里不挂载 store 缓存

Dockerfile 里 RUN pnpm install 不用 BuildKit cache mount，每次 build 都全量装。损失：每次 build 慢 10-20 秒；BuildKit cache 让 install 部分几乎瞬时。修复：用 --mount=type=cache（如上面 Docker 场景代码）。

陷阱 4 · 用 cp -r 备份项目然后 pnpm install

把项目目录 cp -r 到另一处，期待原地用。但 cp 默认不保留 hardlink；新位置的 node_modules 变成"假 hardlink"指向原位置文件——一旦原位置删除，引用断裂。修复：用 cp -al（保留 hardlink）或在新位置 pnpm install 重建链接。

陷阱 5 · 担心 hardlink 跨项目"互相影响"

新手怀疑："多个项目 hardlink 同一个 store 文件，A 项目改了 B 也变？" —— 不会。store 文件是 read-only，hardlink 也是 read-only。任何修改都会触发 EACCES，迫使你用 patch 而非直接改。这是 pnpm 设计的安全保护。
