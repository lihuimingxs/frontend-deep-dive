P3 · pnpm 内部

pnpm 不是"一个更快的 npm"，是用 CAS + symlink 拓扑彻底重做了"node_modules 应该长什么样"。本阶段把它的两条腿——存储模型 + 解析行为——讲透。

章节数 2 主要来源 pnpm.io / Zoltan Kochan blog + GitHub discussions / pnpm-lock 文档 状态 ⏳ 待写作

本阶段覆盖

  • ~/.pnpm-store 全局存储 · 按 SHA-512 hash 切片；同版本全机一份物理文件
  • 3 层链接拓扑 · global store → .pnpm 目录（hardlink）→ 内部 deps（symlink）→ 顶层（symlink）
  • 磁盘节省的真实数字 · 3 个相同 React 项目 npm vs pnpm 落地对比
  • 跨分区退路 · hardlink 失败时回落 copy；APFS / Btrfs reflink；node-linker 选项
  • pnpm-lock.yaml 三段结构 · importers / packages / snapshots 各自记录什么
  • 严格 peer 模式 · 默认不自动满足；auto-install-peers 开关；严格性如何防 bug
  • resolution-mode · highest / time-based / lowest-direct（保护新项目不踩刚发布烂包）
  • .pnpmfile.cjs hooks · readPackage / afterAllResolved 改写 lockfile 与 package.json
  • patches 子系统 · pnpm patch <pkg> 与 patch-package 对比
