Turborepo · task graph + remote cache

光有 workspace 不够：50 个包跑 build 半小时，每次 PR CI 都重跑就是浪费。Turborepo（2020 Jared Palmer，2021 Vercel 收购）的设计立场是：不替包管理器；建在 pnpm/Bun 之上的 task runner + cache layer。本章把它的核心机制讲透——turbo.json 任务图、input/output hash 的 cache 算法、Vercel Remote Cache 实现、以及最容易踩的"环境变量边界"陷阱。

阅读时长 ~30 分钟 前置 P4（workspace + filter） 主要来源 turbo.build/repo/docs / Jared Palmer 演讲 / Vercel 团队 blog

01 速览

  1. Turborepo 不是包管理器 —— 它建在 npm/Yarn/pnpm/Bun 之上，专做"按依赖图调度任务 + cache 复用"。这是它的核心定位差异——与 Nx 的"全家桶"路线相反，Turborepo 是"组合工具"流派。
  2. turbo.json 描述任务图 —— 每个 task 配 dependsOn（依赖谁）/ inputs（哪些文件影响 hash）/ outputs（产物路径）/ cache（是否缓存）/ env（影响 hash 的环境变量）。^build 表示"上游 workspace 的 build"。
  3. input/output hash 是 cache 的核心算法 —— Turborepo 把任务的所有输入文件 + 环境变量 + 工具版本 + 依赖任务的 output hash 全部 SHA-256 → 得到 cache key。命中即replay output 文件 + log 输出，未命中真正执行。
  4. Local cache —— 每个 task 的 cache 存到 node_modules/.cache/turbo/<hash>.tar.zst。重跑相同输入直接拿，速度 ~瞬时（解压 + 复制）。
  5. Remote Cache —— Turborepo 最大商业卖点。cache key 推到 Vercel Remote Cache（默认免费层）/ 自建 S3 / Cloudflare Workers 兼容层。团队成员 + CI 共享："Linus 编译过的，我不再编译"。
  6. 环境变量边界陷阱 —— Turborepo 默认不感知环境变量。漏配 env 字段导致跨环境 cache 命中误判（dev cache 在 prod 复用，bug）。这是最常见的 Turborepo 故障。
  7. turbo dev / turbo run dev —— 长跑任务支持（不缓存输出但智能调度）；自动并发跑所有 dev 命令（apps/web + packages/ui watcher）。这是 monorepo dev 联调的现代标准。
  8. Turborepo 2（2024-08）大改 —— Rust 重写（原 Go 实现）；filter UI 可视化 cache 命中；新的 boundaries 规则；更好的错误信息。2026 年主流是 turbo@2。

02 底层逻辑

Turborepo 在 monorepo 中的位置

把 monorepo 工具栈分层看：

层 负责什么 工具
L4 任务编排 + cache 按依赖图跑 task，cache 复用结果 Turborepo / Nx
L3 包管理器 + workspace 装依赖、symlink、catalog pnpm / Bun / npm
L2 单个 task 的实现 编译 / 测试 / lint 等 Vite / tsc / Vitest / ESLint
L1 source code — —

Turborepo 占 L4。它不管 L3（让 pnpm 处理），也不管 L2（让 Vite/Vitest 处理）。它只关心"当我跑 build 时，需要哪些 task 按什么顺序，怎么用 cache 加速"。

这就是"组合工具"哲学——每层用专门工具，互不锁定。

turbo.json 任务图详解

任意 Turborepo monorepo 的核心配置文件 turbo.json（项目根）：

turbo.json 完整示例 
{
  "$schema": "https://turbo.build/schema.json",
  "ui": "tui",

  "globalDependencies": [
    "**/.env.*local",
    "tsconfig.json"
  ],

  "globalEnv": [
    "NODE_ENV",
    "VERCEL_ENV"
  ],

  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": [
        "src/**",
        "package.json",
        "tsconfig.json",
        "vite.config.ts"
      ],
      "outputs": ["dist/**", ".next/**"],
      "env": [
        "NEXT_PUBLIC_*",
        "DATABASE_URL"
      ]
    },
    "test": {
      "dependsOn": ["build"],
      "inputs": ["src/**", "test/**"],
      "outputs": []
    },
    "lint": {
      "inputs": ["src/**", "*.json", ".eslintrc.*"],
      "outputs": []
    },
    "typecheck": {
      "dependsOn": ["^typecheck"],
      "inputs": ["src/**", "*.ts", "tsconfig.json"],
      "outputs": []
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}

关键字段：

dependsOn：

  • "^build" —— 上游 workspace 的 build（@app/web 的 build 依赖 packages/ui 的 build 完成）
  • "build"（不带 ^）—— 同 workspace 的 build（test 在跑前先跑同包的 build）
  • "^build", "build" —— 上游所有 build + 同包 build 都要先完成

inputs：

  • 哪些文件影响这个 task 的 hash
  • 关键：不写 inputs 时默认是包内所有文件（除了 .gitignore 中的）。看似省事但容易导致无关变化使 cache 失效（如 README.md 改了导致 build cache miss）
  • 显式列出能精确控制 cache 边界

outputs：

  • 这个 task 产生的文件路径（相对于 workspace 根）
  • cache 命中时，这些文件被从 cache 还原到工作区
  • 对不产生输出的 task（lint / typecheck / test）写 "outputs": []

cache：

  • 默认 true，可以缓存
  • 对长跑任务（dev / start）设 false——没有"完成"概念，cache 没意义

persistent：

  • true 表示这个 task 会持续运行（dev server）
  • Turborepo 不会等它"完成"才跑下游
  • 用于 dev / watch 类

env：

  • 哪些环境变量影响 hash
  • 通配符 NEXT_PUBLIC_* 把所有此前缀的 env 都纳入
  • 这是 Turborepo 最易出 bug 的字段（详见下面）

globalDependencies / globalEnv：

  • 所有 task 都受影响的全局依赖（root package.json / tsconfig / .env）
  • 全局 env 同理
  • 通常放"项目级"配置文件 + 跨任务共用 env

input/output hash · cache 的核心算法

Turborepo 在跑某个 task 前，先计算 cache key：

cache key 计算（伪代码） 
function computeCacheKey(task, workspace) {
  const hasher = new SHA256();

  // 1. 全部 input 文件的内容 hash
  for (const file of resolveGlobs(task.inputs, workspace)) {
    hasher.update(filePath, fileContent);
  }

  // 2. globalDependencies 文件
  for (const file of resolveGlobs(turbo.globalDependencies, root)) {
    hasher.update(filePath, fileContent);
  }

  // 3. 影响 hash 的环境变量值
  for (const key of [...task.env, ...turbo.globalEnv]) {
    hasher.update(key, process.env[key]);
  }

  // 4. 上游依赖任务的 cache key（递归）
  for (const upstream of task.dependsOn) {
    hasher.update(upstream, computeCacheKey(upstream, ...));
  }

  // 5. task 命令本身的 hash（pnpm run 的 script 内容）
  hasher.update('command', readScript(workspace, task.name));

  // 6. workspace 包名 + Turborepo 版本
  hasher.update('workspace', workspace.name);
  hasher.update('turbo-version', turbo.VERSION);

  return hasher.digest();
}

每次跑 task：

  • 计算 cache key
  • 查 cache：local 先（node_modules/.cache/turbo/），remote 后（如果配了）
  • 命中 → 解压 cache 包，复制 outputs 到工作区，replay 当时的 stdout/stderr，标 cached
  • 未命中 → 真正跑 task；成功后把 outputs + log 打成 <hash>.tar.zst 存 cache

实测 cache 行为：

bash 
# 第一次跑
pnpm turbo run build
# • Building...
# Tasks: 12 successful, 12 total
# Cached: 0 cached, 12 total
# Time: 35.2s

# 立即第二次跑（一切未变）
pnpm turbo run build
# • Building...
# Tasks: 12 successful, 12 total
# Cached: 12 cached, 12 total           ← 全命中！
# Time: 0.8s                            ← 几乎瞬时

0.8 秒的开销主要是解压 cache + 复制 outputs。真正的 build 跳过。

Local cache 与 Remote Cache

Local cache（默认）：

  • 存到 node_modules/.cache/turbo/<hash>.tar.zst
  • 用 zstd 压缩（速度比 gzip 快、压缩率好）
  • 只对当前机器有效——同事机器 / CI 上没有

Remote Cache：

  • 本地查不到时去远程拉
  • 本地跑成功后推到远程
  • 团队成员 + CI 共享——你已 build 过的，CI 可以直接拉

三种 Remote Cache 选择：

(1) Vercel Remote Cache（默认推荐）：

  • 免费层够用（团队 < 10 人）
  • 登录 turbo login + 链接项目 turbo link
  • 不要求项目部署在 Vercel——只是用 Vercel 的 cache 服务
  • 对 OSS 项目甚至商业项目这是最低门槛方案

(2) 自建 S3 / Cloudflare R2 等 S3-compatible：

  • 用社区项目 turborepo-remote-cache（开源 S3 wrapper）
  • 自己控制 cache 数据 + 不付 Vercel 费用
  • 适合大团队 / 已有云存储基建

(3) Cloudflare Workers 兼容层：

  • 把 Vercel Remote Cache API 重做在 Cloudflare Workers + R2
  • 对全球团队低延迟

实战收益：

  • 团队成员 A 改了一行 README，B 拉代码 → B 的 build cache 命中 A 的（README 不在 inputs，hash 不变）
  • CI 上跑 build → 拉 main 分支 cache 直接 replay
  • 典型 CI 时间从 8 分钟降到 1 分钟（节省 80-90%）

环境变量边界陷阱（最易踩）

Turborepo cache 要"对所有影响 output 的输入敏感"。环境变量是经典坑——它们不在文件里，但显著影响 build 输出。

默认行为：Turborepo不感知任何环境变量，除非你显式列在 env / globalEnv。

典型 bug 1 · NEXT_PUBLIC_* 漏配：

json 
// 错误的 turbo.json
{
  "tasks": {
    "build": {
      "outputs": [".next/**"]
      // ❌ 没声明 env！
    }
  }
}
  • 本地 dev: NEXT_PUBLIC_API_URL=http://localhost:3000 + 跑 turbo build
  • 本地 prod 测试: NEXT_PUBLIC_API_URL=https://api.production.com + 跑 turbo build
  • Turborepo 看 inputs（src 文件）没变 → cache 命中 → 用 dev 时 build 的 .next 文件
  • 启动 prod 测试 → 浏览器请求 localhost：bug

正确写法：

json 
{
  "tasks": {
    "build": {
      "outputs": [".next/**"],
      "env": ["NEXT_PUBLIC_*"]              // ← 任何 NEXT_PUBLIC_ 前缀的 env 都影响 hash
    }
  }
}

现在 NEXT_PUBLIC_API_URL 变化触发 cache miss，build 重跑。

典型 bug 2 · CI / dev 的全局 env 没区分：

  • CI 环境有 CI=true
  • 有些 build script 看 CI 标志改行为（不跑 dev 工具、不打开 source map 等）
  • 没声明 globalEnv 导致 dev 和 CI cache 互相污染

解：根 turbo.json 加 "globalEnv": ["CI", "NODE_ENV"]。

诊断工具：

bash 
# 看某个 task 的 cache 计算细节（dry run）
turbo run build --dry-run=json | jq '.tasks[] | select(.taskId=="@app/web#build")'

# 输出包含：
# - hashOfInputs: 哪些文件参与
# - hashOfEnv: 哪些 env 参与
# - cache: hit/miss
# - 整体 hash

turbo dev · 长跑任务的智能调度

monorepo dev 场景：要同时跑 apps/web 的 Vite dev + packages/ui 的 tsc --watch + packages/db 的 prisma generate watch + ... 手写 npm-run-all 配 8-10 个并发命令既复杂又难看 log。

turbo dev：

turbo.json 
{
  "tasks": {
    "dev": {
      "cache": false,
      "persistent": true,
      "dependsOn": ["^build"]   // 上游 build 一次再跑 dev
    }
  }
}
bash 
# 在 monorepo 根
turbo run dev

# Turborepo 自动：
# 1. 看哪些包定义了 dev script
# 2. 上游包先 build（packages/ui 的 dist 准备好）
# 3. 然后并发跑所有 dev script
# 4. 输出按包聚合（不交错）
# 5. 一个 task 失败不影响其他

filter 配合：

bash 
# 只跑 apps/web 的 dev + 它的上游依赖的 watch
turbo run dev --filter=@app/web

Turborepo 2 的关键改进（2024-08）

原 Turborepo 用 Go 实现。2024 年 8 月 Vercel 发布 turbo@2，Rust 重写，带来：

  • 性能提升：cache 处理 + hash 计算更快
  • TUI（Terminal UI）模式：交互式查看 task 进度 + cache 命中
  • boundaries 规则（实验）：声明哪些包能/不能依赖哪些包，编译时拦截违规 import
  • 更好的错误信息：cache miss 原因、失败 task 的清晰输出
  • turbo.json 的命名一致化："pipeline" 改 "tasks"（旧字段仍兼容但 deprecated）

2026 年主流项目用 turbo@2.x，这是 Turborepo 现代化的明确分界。

💡 关键洞见：Turborepo 是"cache layer"而非"monorepo 工具"

很多人把 Turborepo 当 monorepo 全方案。这是错的。Turborepo 不管包管理（pnpm 管）、不管编译（Vite/tsc 管）、不管测试（Vitest 管）。它专做"缓存任务结果 + 智能调度"。理解这一点你就明白：选 Turborepo 不是"换框架"，是"加一层 cache"——可逆、可叠加、低门槛。

03 应用场景

场景 1 · CI build 时间从 8 分钟降到 1 分钟

背景：30 包 monorepo，CI 上 pnpm -r build 每次 8 分钟。每天 50+ PR，CI 排队。

方案：

  1. 装 Turborepo: pnpm add -D -w turbo
  2. 写 turbo.json（先简单：build / test / lint 三个 task）
  3. 本地试: pnpm turbo run build 验证正常
  4. 登录 Vercel Remote Cache: turbo login && turbo link
  5. CI workflow 改：
yaml 
# .github/workflows/ci.yml
- run: pnpm install --frozen-lockfile
- run: pnpm turbo run build typecheck lint test
  env:
    TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
    TURBO_TEAM: 'my-team'
  1. 第一次 CI 跑 8 分钟（build 全部 + 推 cache）
  2. 第二次 CI（small PR）：~1 分钟（绝大部分 cache 命中）

典型 ROI：节省 6-7 分钟/PR × 50 PR/天 = 5-6 小时/天 CI 时间。月省数百美元 CI 账单 + 极大改善 dev 体验。

场景 2 · monorepo dev 命令一统

背景：dev 时要同时跑 apps/web Vite + packages/ui tsc --watch + packages/db prisma generate --watch。手写 concurrently 命令难维护。

方案：用 turbo dev

turbo.json 
{
  "tasks": {
    "dev": { "cache": false, "persistent": true, "dependsOn": ["^build"] },
    "build": { "outputs": ["dist/**", ".next/**"] }
  }
}
bash 
# 一条命令启动所有
pnpm turbo dev

# 或仅启动主 app + 它的依赖 watch
pnpm turbo dev --filter=@app/web

Turborepo 自动：

  • 跑上游 packages 的 build 一次（让 dist 准备好）
  • 然后并发跑各包的 dev / watch
  • log 按包分组聚合

场景 3 · cache 突然全 miss 的诊断

背景：turbo build 一直好用，今天突然全 miss，cache 完全不命中。

诊断顺序：

  1. turbo run build --dry-run=json 查 hash，看变化的部分
  2. 常见原因：
    • 团队升级了 turbo 版本（version 进 hash）→ 升级时全 miss 一次正常
    • 修改了 turbo.json（schema 进 hash）→ 改一次 miss 一次
    • 修改了 globalDependencies 中的文件（如 tsconfig.json）→ 全 miss
    • 新加了 globalEnv 中的 env 变量（每次值不同）
    • 升级 Node 版本（不在 hash 中但影响 build 行为，间接通过 lockfile）
  3. 修复：明确改动的目的，确认是不是预期行为
  4. 验证：再次 build 后查 cache 是否命中

04 典型代码

完整 turbo.json 模板（中型 monorepo） 
{
  "$schema": "https://turbo.build/schema.json",
  "ui": "tui",

  "globalDependencies": [
    "**/.env.*local",
    "tsconfig.base.json",
    "package.json"
  ],

  "globalEnv": ["NODE_ENV", "CI", "VERCEL_ENV"],

  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": [
        "$TURBO_DEFAULT$",
        "!**/*.md",
        "!**/*.test.*"
      ],
      "outputs": ["dist/**", ".next/**", "!.next/cache/**"],
      "env": ["NEXT_PUBLIC_*", "DATABASE_URL"]
    },
    "test": {
      "dependsOn": ["build"],
      "inputs": ["$TURBO_DEFAULT$"],
      "outputs": ["coverage/**"]
    },
    "test:watch": { "cache": false, "persistent": true },
    "lint": {
      "inputs": ["src/**", "*.json", "*.cjs", ".eslintrc.*"],
      "outputs": []
    },
    "typecheck": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "*.ts", "tsconfig.json"],
      "outputs": []
    },
    "dev": {
      "cache": false,
      "persistent": true,
      "dependsOn": ["^build"]
    },
    "clean": {
      "cache": false
    }
  }
}

注意 $TURBO_DEFAULT$：turbo@2 引入的占位符，等于"包内所有 git-tracked 文件"。配合 !**/*.md 等排除模式精确控制。

CI workflow（GitHub Actions） 
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

env:
  TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
  TURBO_TEAM: 'my-team'

jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'

      - run: pnpm install --frozen-lockfile

      # 一条命令跑所有任务，全用 cache
      - run: pnpm turbo run build typecheck lint test
        env:
          # 这里列出的 env 必须在 turbo.json 的 task.env 中声明
          # 才能让 cache 对它们敏感
          NEXT_PUBLIC_API_URL: ${{ secrets.NEXT_PUBLIC_API_URL }}
root package.json scripts 
{
  "scripts": {
    "dev": "turbo run dev",
    "build": "turbo run build",
    "test": "turbo run test",
    "lint": "turbo run lint",
    "typecheck": "turbo run typecheck",
    "ci": "pnpm install --frozen-lockfile && turbo run build test lint typecheck",
    "clean": "turbo run clean && rm -rf node_modules/.cache/turbo"
  },
  "devDependencies": {
    "turbo": "^2.0.0"
  }
}
filter + remote cache 实战 
# 只 build 受当前 PR 影响的包
turbo run build --filter='[origin/main...HEAD]'

# 只测试 ui 包及其下游
turbo run test --filter='@app/ui...'

# 看 cache 状态
turbo run build --dry-run

# 强制不用 cache（验证某个 task 真的能跑）
turbo run build --force

# 仅 local cache 不推 remote
turbo run build --no-remote-cache

05 常见陷阱

陷阱 1 · 漏配 env 字段

最常见 Turborepo bug。改 .env 后 cache 命中错的版本。修复：每个 build / publish 类 task 都列 env；CI 跑前 turbo run build --dry-run=json 验证关键 env 在 hash 中。

陷阱 2 · outputs 写错路径

把 outputs 写成 "./dist/**"（带 ./）或绝对路径——Turborepo 期待 workspace-relative 路径。结果：cache 找不到产物，每次都 miss + 复制不上。修复：用 "dist/**"（无前导斜杠或 ./）。

陷阱 3 · 把 node_modules 放进 inputs

"我的 task 依赖 node_modules 里的某些包" —— 不要！node_modules 是 lockfile + pnpm install 的产物，Turborepo 间接通过 globalDependencies 中的 lockfile 跟踪。把 node_modules 列入 inputs 会导致每次装包都全 miss + hash 计算极慢。修复：把 lockfile 列入 globalDependencies 而非 node_modules。

陷阱 4 · Remote cache 不工作但没报错

CI 上配了 TURBO_TOKEN 但没生效，cache 看似正常其实没跨机器。诊断：

bash 
# --summarize 查看 cache 来源
turbo run build --summarize
# 查 .turbo/runs/<timestamp>.json 中的 cacheState

陷阱 5 · 和 Nx 同时挂在项目里

有人跟着不同教程，先装了 Nx 后又装 Turborepo（或反之）。两套并存互不知情，cache 各自一份，配置冗余。修复：选一个。Turborepo 是"组合工具"流派，Nx 是"全家桶"流派——理念冲突，混用得不偿失。
