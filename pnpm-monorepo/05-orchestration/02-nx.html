Nx · 项目图 + affected + 横评

上章讲 Turborepo（"组合工具"流派）。本章讲 Nx（"全家桶"流派）—— 不仅是 task runner，还有项目图、plugin 生态、code generator、迁移 codemod、Nx Cloud。Nx 把 Lerna 的位置接过来后做了大量改造，2026 年是企业级 monorepo 的另一极。本章讲它的核心机制 + 与 Turborepo 的决策矩阵。

阅读时长 ~25 分钟 前置 P5.1（Turborepo） 主要来源 nx.dev 官方 / Victor Savkin blog / Nrwl 团队演讲

01 速览

  1. Nx 不只是 task runner —— 与 Turborepo 单一职责不同，Nx 提供：(1) 项目图（自动从代码推断，可视化）；(2) affected 命令（按图差分跑任务）；(3) plugin 生态（@nx/react / @nx/next / @nx/vite / @nx/jest）；(4) generator（脚手架代码）；(5) nx migrate（升级 Nx 自动改全 monorepo 配置）；(6) Nx Cloud（远程 cache + DTE 分布式执行）。
  2. 项目图（project graph） —— Nx 自动从 package.json / tsconfig.json / 源码 import 推断 monorepo 内部依赖关系。nx graph 渲染交互式 SVG，能看每个包依赖谁、被谁依赖。这是"affected"的基础。
  3. nx affected —— nx affected -t test 只跑被改影响的项目（项目图差分实现）。比 Turborepo 的 git filter 更精确——它知道哪些文件属于哪个项目。
  4. plugin 生态是 Nx 最大差异点 —— @nx/react 不只跑 React 项目，还提供 generator（nx g @nx/react:lib my-lib 自动建包 + 配 tsconfig + lint + test）+ executor（标准化跑 build/test/lint）+ migration（升级时自动改配置）。这套"框架式 monorepo"是 Nx 的灵魂。
  5. nx migrate —— Nx 18 → 19 → 20 升级时，nx migrate latest 自动改 monorepo 内所有 plugin 配置 + 跑 codemod 适配 breaking changes。这是 Turborepo 没有的能力，也是 Nx 在长期维护性上的优势。
  6. Nx Cloud + DTE —— Nx Cloud 提供远程 cache（与 Vercel Remote Cache 类似）+ DTE（Distributed Task Execution）把任务分到多 CI 节点并行跑。50 包的 monorepo CI 从 10 分钟降到 2 分钟（不仅 cache，还有真正的并行执行）。
  7. Nx vs Turborepo 哲学差异 —— Nx 是"从零搭，工具链一统"（适合大企业团队、新项目）；Turborepo 是"已有工具链，加层 cache"（适合现存项目、组合自由）。没有对错只有选择。
  8. Nx 接管 Lerna（2022） —— Lerna 6+ 由 Nrwl 维护，本质是 Nx 的"轻模式"。新项目几乎没人单独用 Lerna；存量 Lerna 项目升 Nx 6+ 是渐进路线。

02 底层逻辑

项目图（project graph）

Nx 启动时扫描整个 monorepo，构建项目依赖图。来源：

  • 每个项目的 package.json 中的 dependencies / devDependencies（找到 workspace 内部依赖）
  • tsconfig.json 中的 paths 字段
  • 源码中的 import 语句静态分析（@nx/js/typescript plugin 提供）

结果是个 DAG：

project graph 示例 
          [apps/web]
              ↓
        [apps/web-e2e]   [apps/admin]
              ↓               ↓
            ┌────────────────┘
            ↓
       [packages/ui] ──→ [packages/utils]
            ↓                  ↓
       [packages/icons]   [packages/types]

可视化：

bash 
nx graph
# 自动启动 localhost:4211 浏览器交互式 SVG
# 可以点击节点高亮 / filter / 看任务依赖

# 静态导出
nx graph --file=graph.html
nx graph --file=graph.json   # 用于 CI 分析

这是 Nx 的差异化武器。Turborepo 也有任务图（隐式从 dependsOn 推断），但没有项目级的可视化。Nx 的项目图让大型 monorepo（50+ 包）的结构清晰可见。

nx affected · 项目图驱动的差分

Turborepo 的 --filter='[main]' 用 git diff 找变化文件 → 推断变化的包。这够用但不精确——大文件改动 / 跨包重构 / git 历史复杂时会误判。

Nx 的 affected 走项目图 + 文件归属：

  1. git diff 找出变化的文件路径
  2. 用项目图把每个文件归属到 owning project（packages/ui/src/Button.tsx → packages/ui）
  3. 选中所有 owning project
  4. 沿项目图下游遍历，把所有受影响的下游 project 也选中
  5. 对结果集跑指定 task（test / build / lint）
bash 
# 自 main 以来 affected 的项目跑 test
nx affected -t test --base=main

# 跑 build（test 同样，但 build 涉及上游依赖也要 build）
nx affected -t build

# 看哪些 affected
nx show projects --affected --base=main

典型 PR 场景：

  • 改了 packages/ui/src/Button.tsx
  • nx affected：packages/ui + 它的下游（apps/web, apps/admin）
  • 跑 5-10 个项目的 test，跳过其余 40+ 项目
  • CI 时间从 15 分钟降到 3 分钟

plugin 生态 · Nx 的"框架式"基因

装一个 Nx plugin（如 @nx/react）后获得：

(1) generator · 标准化建包

bash 
# 创建一个新 React lib
nx g @nx/react:lib packages/my-component-lib

# Nx 自动：
# - 创建目录结构
# - 配置 tsconfig.json（继承根 tsconfig）
# - 配置 package.json + 依赖
# - 配置 .eslintrc.json
# - 配置 vite.config.ts（或 jest.config.ts，按 plugin）
# - 创建 src/index.ts + src/lib/my-component-lib.tsx 模板
# - 注册到 nx.json 的 projects

# 创建 React app
nx g @nx/react:app apps/my-new-app

# Nx 也自动配 .gitignore / public/ / 路由 / SSR 选项

意义：团队所有 React lib 配置一致——所有人 nx g 出来的 lib 用同一套 tsconfig / eslint / build 流程。新人加入 0 学习成本。

(2) executor · 标准化跑任务

project.json（Nx 的项目配置文件） 
{
  "name": "my-react-lib",
  "targets": {
    "build": {
      "executor": "@nx/vite:build",
      "options": {
        "outputPath": "dist/packages/my-react-lib"
      }
    },
    "test": {
      "executor": "@nx/vite:test",
      "options": {
        "config": "packages/my-react-lib/vite.config.ts"
      }
    }
  }
}

每个 task 用一个 executor 跑。executor 是 Nx 提供的标准实现——比直接调 vite build 多了：

  • 统一的输入/输出格式（让 Nx cache 知道边界）
  • 错误处理 + 报告
  • 多项目共用配置

(3) migration · 升级时自动改

bash 
# 准备升级 Nx
nx migrate latest

# 自动：
# - 升级 nx + 所有 @nx/* plugins 到最新
# - 升级 Nx 间接依赖（react / vite / vitest 等到兼容版本）
# - 生成 migrations.json，记录所有要跑的 codemod

# 应用 codemod
nx migrate --run-migrations

# 这一步自动：
# - 改 nx.json schema 适配新版本
# - 改每个 project.json 的 executor 名（如果 Nx 18 的 @nx/vite:build 改名了）
# - 改 tsconfig.json 选项（如果 TS 升级要求）
# - 改 .eslintrc.json（如果 ESLint 升级）
# - 自动 fix 一些已知 breaking changes

这是 Nx 比 Turborepo 强的地方。Turborepo 升级你要自己改 turbo.json + 自己适配各 plugin 的 breaking changes。Nx 的 migration 系统让大型 monorepo 跨版本升级成本可控。

Nx Cloud · DTE 分布式执行

Vercel Remote Cache（Turborepo）只做"cache 复用"。Nx Cloud 多了一层：Distributed Task Execution（DTE）。

什么是 DTE：

  • monorepo 50 个项目要跑 test + build
  • 传统 CI：1 个 runner 顺序跑或并发跑（受单 runner CPU 限制，通常 4-8 并发）
  • DTE：把 50 个 task 分到 N 个 CI runner 同时跑，主 runner 协调
  • 结果聚合 + cache 推送都自动
.github/workflows/ci.yml（Nx DTE） 
jobs:
  main:
    runs-on: ubuntu-latest
    steps:
      - uses: nrwl/nx-set-shas@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm exec nx-cloud start-ci-run --stop-agents-after="build"
      - run: pnpm exec nx affected -t lint test build --parallel=3
      - run: pnpm exec nx-cloud stop-agents

  agents:
    name: Nx Cloud Agents
    runs-on: ubuntu-latest
    strategy:
      matrix:
        agent: [1, 2, 3]   # 3 个 agent 协助主 runner
    steps:
      - uses: actions/checkout@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm exec nx-cloud start-agent

实测效果：

  • 50 包 monorepo：单 runner 10 分钟，4 agent DTE 2-3 分钟
  • 主 runner 不再"满负载"，可以做轻活（合 cache、检查结果）
  • 付费功能（Nx Cloud 免费层不含 DTE）

Nx 的 nx.json + project.json 配置

Nx 用两层配置：

nx.json（项目根，全局）：

nx.json 
{
  "$schema": "./node_modules/nx/schemas/nx-schema.json",
  "namedInputs": {
    "default": ["{projectRoot}/**/*", "sharedGlobals"],
    "production": [
      "default",
      "!{projectRoot}/**/*.spec.tsx",
      "!{projectRoot}/**/*.test.tsx",
      "!{projectRoot}/tsconfig.spec.json"
    ],
    "sharedGlobals": [
      "{workspaceRoot}/tsconfig.base.json",
      "{workspaceRoot}/.eslintrc.json"
    ]
  },
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["production", "^production"],
      "cache": true
    },
    "test": {
      "dependsOn": ["build"],
      "inputs": ["default", "^production"],
      "cache": true
    },
    "lint": {
      "inputs": ["default", "{workspaceRoot}/.eslintrc.json"],
      "cache": true
    }
  },
  "plugins": [
    "@nx/vite/plugin",
    "@nx/eslint/plugin",
    "@nx/react/plugin"
  ],
  "nxCloudId": "abc123-cloud-id"
}

project.json（每个项目，可选）：

  • 覆盖该项目的 task 配置
  • 大部分项目用 plugin 推断（不写 project.json）
  • 需要个性化时写

Nx 的配置比 Turborepo 复杂——这是"全家桶"哲学的代价。学习曲线陡 + 配置 surface 大。

Nx vs Turborepo · 哲学差异详解

维度 Turborepo Nx
定位 Task runner + cache layer Framework for monorepo
侵入性 低（一个 turbo.json） 高（nx.json + 各 project.json + plugin）
可逆性 高（删 turbo.json 即恢复） 低（深度集成）
学习曲线 缓（1-2 小时） 陡（1-2 周）
项目图 隐式 显式 + 可视化
plugin 生态 极少 大（@nx/react / vite / next / jest 等）
generator (脚手架) 无 核心特性
migration codemod 无 核心特性
远程 cache Vercel / 自建 Nx Cloud（含免费层）
DTE 分布式执行 无 有（Nx Cloud 付费）
主流场景 已有项目加 cache 从零搭企业级 monorepo

选 Turborepo 还是 Nx · 决策矩阵

团队特征 建议 理由
已有成熟工具链（Vite + Vitest + ESLint） Turborepo 不破坏现有工具，加层 cache 即可
从零搭新 monorepo + 团队大（20+ 人） Nx generator + migration 让长期一致
项目数 5-30 包 Turborepo 简单足够，不需要 Nx 复杂度
项目数 30-100 包 都 OK，看哲学偏好 —
项目数 100+ 包，多语言（JS + Java + Go） Nx 或 Bazel Turborepo 不擅长跨语言；Nx 有多语言 plugin
跨团队 + 强需求"affected" Nx 项目图驱动的 affected 更精确
团队已用 Vercel 部署 Turborepo Vercel Remote Cache 自然集成
OSS 项目（Vite / Vitest / Astro） Turborepo 实际：这些项目都用 Turborepo
组合工具哲学（不喜欢被锁定） Turborepo —
"全家桶让我省心" 心态 Nx —

超大规模选项 · Bazel / Rush / Pants

当 monorepo > 500 包 + 多语言（Java / Go / Python / JS 混合）时，Turborepo / Nx 都吃力。这时考虑：

Bazel（Google）：

  • 大厂工业级（Google / Spotify / Uber 都用）
  • 语言无关，全用 BUILD 文件描述
  • 极致 cache + 可重现 build
  • 学习曲线极陡（数月）
  • JS 工具链支持有限（rules_nodejs）

Rush（Microsoft）：

  • Microsoft 出品，自家产品（Azure DevOps / Office）用
  • JS 专精，但理念偏"严格 + 大企业"
  • 有自己的 Phantom Dependency 解决方案（早于 pnpm）
  • 2026 年存在感不如 Nx

Pants（Twitter / 现 Twitter Tech Spinoff）：

  • Bazel 的更易用替代
  • 多语言（Python / Go / Scala / JS）
  • JS 生态支持比 Bazel 好但仍不主流

2026 现实：超大规模 + JS-only 仍是 Nx 的天下；超大规模 + 多语言才会上 Bazel / Rush / Pants。绝大多数团队不会到那个规模。详见 P7.1 决策树。

💡 关键洞见：选哲学，不只选工具

Nx vs Turborepo 不是"哪个更好"，是"你想要什么样的 monorepo 生活"。Nx 是"我接受框架约束换来一致性"；Turborepo 是"我保留组合自由换来灵活"。两条路线在 2026 年都有大型成功项目背书，选你团队文化匹配的那种。

03 应用场景

场景 1 · 新建 React 企业级 monorepo

背景：50 人团队，要从零搭一个 monorepo。预计 50+ 包，多个 Next.js app + 共享组件库 + 后端 NestJS API。

建议：Nx。理由：

  • 项目数大 + 团队大 → 一致性比灵活性重要
  • generator 让团队成员建包遵循同一模板
  • migration 让 5 年后升级成本可控
  • nx affected 更精确（重要 PR 场景）
bash 
# 创建 Nx workspace
pnpm create nx-workspace@latest my-org --preset=react-monorepo

# 创建第一个 app
nx g @nx/next:app web

# 创建共享 lib
nx g @nx/react:lib ui-components --directory=packages/ui

# 项目图可视化（团队定期审视）
nx graph

场景 2 · 在已有项目加 cache（不愿大改）

背景：5 年的 pnpm monorepo + Vite + Vitest，CI 慢。不想改工具链，只想加 cache。

建议：Turborepo（详见 P5.1 场景 1）。Nx 的侵入性对这种项目过大——要改大量配置才能让 Nx 接管 build。

场景 3 · OSS 项目选型

背景：开源 React 库 monorepo（10 包），主要给社区贡献者用。

建议：Turborepo。理由：

  • OSS 贡献者背景多样，Nx 学习曲线高门槛
  • 简单 turbo.json 一目了然，PR 易 review
  • Vercel Remote Cache 对 OSS 免费、配置简单
  • 实际：2026 年大量旗舰 OSS 项目（Vite / Vitest / Astro / shadcn/ui）选 Turborepo，几乎没用 Nx

04 典型代码

完整 nx.json 模板 
{
  "$schema": "./node_modules/nx/schemas/nx-schema.json",
  "namedInputs": {
    "default": ["{projectRoot}/**/*", "sharedGlobals"],
    "production": [
      "default",
      "!{projectRoot}/**/?(*.)+(spec|test).[jt]s?(x)",
      "!{projectRoot}/tsconfig.spec.json"
    ],
    "sharedGlobals": [
      "{workspaceRoot}/.gitignore",
      "{workspaceRoot}/tsconfig.base.json"
    ]
  },
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["production", "^production"],
      "cache": true,
      "outputs": ["{projectRoot}/dist", "{projectRoot}/.next"]
    },
    "test": {
      "dependsOn": ["build"],
      "inputs": ["default", "^production"],
      "cache": true
    },
    "lint": {
      "cache": true,
      "inputs": ["default", "{workspaceRoot}/.eslintrc.json"]
    }
  },
  "plugins": [
    {
      "plugin": "@nx/vite/plugin",
      "options": {
        "buildTargetName": "build",
        "testTargetName": "test"
      }
    },
    {
      "plugin": "@nx/eslint/plugin",
      "options": { "targetName": "lint" }
    }
  ],
  "nxCloudAccessToken": "***"
}
nx 命令速查 
# 列出所有项目
nx show projects

# 项目图可视化
nx graph

# 跑某个项目的某个 task
nx run my-app:build
nx run my-app:test
nx run my-app:lint

# 等价简写
nx build my-app
nx test my-app

# 跑所有项目某个 task（topological 排序）
nx run-many -t build

# 仅跑 affected
nx affected -t test --base=main

# 创建项目
nx g @nx/react:lib my-lib
nx g @nx/next:app my-app

# 查看项目细节
nx show project my-app

# 升级 Nx
nx migrate latest
nx migrate --run-migrations

# 清空 cache
nx reset
从 pnpm workspace 渐进引入 Nx 
# 已有 pnpm workspace（不想破坏）
# 渐进加 Nx 作为 task runner

pnpm add -D -w nx

# 创建简单 nx.json（不强制 plugin）
cat > nx.json <<EOF
{
  "namedInputs": {
    "default": ["{projectRoot}/**/*"],
    "production": ["default", "!{projectRoot}/**/*.test.*"]
  },
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["production", "^production"],
      "cache": true
    }
  }
}
EOF

# Nx 自动从 package.json scripts 推断 task
nx run-many -t build

# 渐进迁移到 plugin（可选）
# 不必一次性切到 generator 模式

05 常见陷阱

陷阱 1 · 把 Nx 当 Turborepo 用（仅 cache）

装了 Nx 但不用 plugin / generator / migration，只用 cache 功能——本质上花了 Nx 的复杂度成本但没拿到大半收益。判断：只想要 cache → 选 Turborepo；想要 generator + migration + framework consistency → 选 Nx。不要中间状态。

陷阱 2 · Nx + Turborepo 混挂

跟着不同教程同时装两套，配置冗余 + cache 各自一份 + 团队 confused。修复：选一个清理另一个。

陷阱 3 · 不跑 nx migrate 就升级

pnpm update nx 直接升级，不跑 nx migrate。结果项目配置与新 Nx 版本不匹配，task 跑失败。修复：Nx 升级永远走 nx migrate latest --run-migrations 流程。

陷阱 4 · 期望 Nx generator 完美

Nx generator 创建项目模板已知"刻板"——它给你的 tsconfig / eslint / build config 反映 Nx 团队的偏好，未必符合你的项目风格。建议：team 决定建 lib 后立即调整 generator 输出到团队规范，或者写自定义 generator（Nx 支持）。

陷阱 5 · Nx Cloud 默认开 telemetry

Nx 装完默认开 telemetry（执行数据上报）。企业项目可能合规上有问题。修复：npx nx@latest --analytics=disable 或在 nx.json 加 "defaultBase": "main" 同时关 telemetry。
