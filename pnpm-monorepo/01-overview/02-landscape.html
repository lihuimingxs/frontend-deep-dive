同代人横评：6 家全维度对比

横向对照 2026-05 的 6 家主流 JS 包管理器：npm 11 / Yarn classic / Yarn 4 (Berry) / pnpm 10 / Bun install / Deno + JSR。每家都有"世界观"——选错工具不是性能问题，是不匹配场景。本章一次讲清各自定位 + 8 维度对比 + 选型决策。

阅读时长 ~25 分钟 前置 P1.1（项目史） 主要来源 各家官方文档 + 实测 benchmark + 社区 case studies

01 速览

  1. 没有"通用最佳"包管理器 —— 每家都有 sweet spot：大型 monorepo + 严格 → pnpm；新项目极速安装 → Bun；老 npm 项目无痛升级 → npm 11；Webpack/CRA 历史项目 → Yarn classic 维护期；Deno / 原生 TS / 跨 runtime → Deno + JSR。选错 ≠ 性能问题，而是不匹配场景。
  2. 2026 主流分布（粗估） —— pnpm ~35% / npm ~30% / Yarn (classic + 4) ~20% / Bun ~12% / 其他 ~3%。Yarn classic 仍占明显份额（百万级遗留项目），但增量市场基本不选 Yarn classic 了。
  3. 选型最重要的 3 个维度 —— (1) 严格性（要不要禁止幻影依赖）；(2) monorepo 深度（workspace + catalog + filter 怎么用）；(3) 生态兼容性（你的依赖里有没有"对包管理器挑食"的奇葩）。性能差距通常不是最重要的——除非你 CI 时间紧到秒。
  4. "快"的真相 —— Bun cold install 最快（10-20x of npm），但 cold install 通常 CI 才出现。warm install / lockfile-only 几家差距收窄到 2-3x。对日常开发，pnpm/Bun 都"足够快"。性能数字通常是营销，落地差距不那么戏剧性。
  5. Yarn 4 是"两个产品" —— 默认 PnP 模式（激进）+ nodeLinker: node-modules 模式（保守）。后者本质是把 Berry 当 classic 用，性能 + 体验介于 classic 和 pnpm 之间。在 2026 仍是"小众但忠实"路线。
  6. Deno + JSR 是另一个范式 —— 不是替代 npm 的"更快 npm"，而是从 registry 层重做。同一项目要么走 npm 生态，要么走 Deno 生态，二者目前不能混（Deno 可以 import npm 包但反向不行）。
  7. catalog 是 2026 的分水岭 —— 支持 catalog 的包管理器（pnpm 9.5+ / Bun 1.1.42+）做 monorepo 才"现代"；不支持的（npm 11、Yarn 4 PnP 默认）落后一拍。这影响中大型 monorepo 选型。

性能横评（典型中型项目，cold install）

需要先标注：包管理器性能数字依硬件 / 网络 / 项目规模 / cache 状态变化巨大。下面是一个"典型 React 项目（next.js + 50 个 npm 依赖）"在 macOS / 千兆网 / 现代 SSD 的近似数字。复现条件 + 真实 benchmark 见各家官方公告。

工具 cold install（首次，无 cache） warm install（有 cache，重装） lockfile-only（CI 场景） 磁盘（单项目）
npm 11 ~25-30s ~10-15s ~8-12s ~250 MB
Yarn classic ~20-25s ~8-10s ~6-8s ~250 MB
Yarn 4 PnP ~5-8s ~1-2s ~1-2s ~80 MB（zip 不解压）
pnpm 10 ~10-15s ~3-5s ~2-4s ~80 MB（hardlink）
Bun install ~2-4s ~0.5-1s ~0.5-1s ~80 MB

关键观察：

  • Bun install 是性能标杆，但 cold install 这个差距在日常开发中很少触发——大部分时候你装的是 1-2 个新依赖，不是从零
  • pnpm / Bun / Yarn PnP 的磁盘占用都低于 npm/Yarn classic 3 倍以上（CAS / hardlink / zip 复用各有路线）
  • lockfile-only（CI）场景 npm 仍是慢的——CI 时间成本是 npm 用户的最大日常痛点
  • 差距随项目规模扩大：100+ 包的 monorepo cold install 差距能到 10x，CI 时间差从 2 分钟到 20 分钟
⚠️ 性能数字读法

不要把这些数字当作"绝对真理"。所有包管理器都在快速演化，npm 11 vs npm 8 的速度差距已经很大。Benchmark 必须复现：相同项目 / 相同硬件 / 相同网络 / 相同 cache 状态。营销文章里的"快 30x"通常是 cherry-picked 场景。

npm 11 · "默认基线"

世界观：JS 生态需要一个"够用"的包管理器，作为 Node 的捆绑组件普及到每个 Node 用户。npm 不追求性能巅峰，追求"装上 Node 就有"。

设计立场：

  • 跟 Node LTS 绑定：每代 Node LTS 自带某版本 npm，零额外安装
  • 布局 = 扁平 hoisting（npm 3 以来不变）
  • lockfile = package-lock.json，2017 年加入，与 yarn.lock 兼容地解决"非确定性"
  • workspace = 2020 年加入（v7），但功能比 pnpm/Yarn 落后
  • 没有 catalog（2025-2026 仍未跟进）

2025-2026 进化方向：

  • provenance（npm 9.5+，2023 年）—— GitHub Actions OIDC 自动签名，对抗供应链攻击。这是 npm 近年最显著的改进
  • 性能持续改进—— npm 10 → npm 11 安装速度有 30-40% 提升，但仍落后于 pnpm/Bun
  • workspace 改进—— v9+ workspace 命令更完善，但仍不支持 catalog 这种 "monorepo 高级特性"

Sweet spot：

  • 新人 / 一次性 demo / 教学项目（"装上 Node 就有"是优势）
  • 遗留 npm 项目维护期（不想折腾迁移）
  • 对 monorepo 没需求的小项目

避坑：

  • 大型 monorepo 不推荐——workspace 实现单薄，缺 catalog
  • 幻影依赖问题严重（详见 P2.2）

Yarn classic（1.x）· "维护期遗产"

世界观：2016 年是革命者，2026 年是遗产。Yarn classic 当年解决了 npm 3 的非确定性 + 速度问题，但今天它的优势被 npm 5+ 完全追平。

2026 状态：

  • Yarn 官方主线已经走到 v4（Berry）；classic（v1.x）只接受关键 bug fix
  • 大量历史项目仍用 classic（百万级 GitHub 仓库锁定 yarn 1.x）
  • classic 与 Berry 配置 / 命令不互通——"yarn 2 升级"实际是换了一个产品

Sweet spot：

  • 历史 Yarn 1.x 项目维护期（不想动）
  • 对老项目无痛——Yarn classic 的 hoisting 行为与 npm 兼容性最好（甚至比 Yarn 4 都好）

避坑：

  • 新项目不要选 Yarn classic——选什么都比它合适
  • 不要把"yarn classic 用得好好的"当作"不需要升级"——真正决定时机是何时切 pnpm/Bun，不是何时切 Yarn 4

Yarn 4（Berry）· "PnP 信仰者 + 保守 fallback"

世界观：Maël Nison 主导，主张"node_modules 是错的，应该取消"。2026 年这个立场仍小众但坚定。

两个模式：

  • PnP 模式（默认）—— 完全无 node_modules；.pnp.cjs 劫持 require；性能 / 严格性最高，生态接受度最低
  • nodeLinker: node-modules 模式—— 退回到 hoisting + node_modules；本质把 Berry 当 classic 用；牺牲 PnP 优势但获得生态兼容

独特能力（PnP 模式）：

  • zero-installs—— commit .yarn/cache + .pnp.cjs 到 git；新 clone 的人不用 install 就能跑
  • 极致严格—— require 必须经过 .pnp.cjs 仲裁，比 pnpm 还严
  • protocol 丰富—— workspace: / portal: / patch: / git: / file: 各种依赖来源

2026 现状：

  • 大部分 Yarn 4 项目实际跑 nodeLinker mode（生态兼容性原因）
  • PnP 用户社区高度忠实但小众
  • catalog 仍在 RFC 阶段，落后 pnpm/Bun

Sweet spot：

  • 有重度 patch 需求的项目（Yarn patch protocol 强大）
  • 愿意为 zero-installs 付出生态适配成本的极客团队
  • 已用 Yarn 多年的团队迁移成本最低

避坑：

  • 新项目 + 主流栈，pnpm 几乎总是更省心的选择
  • 不要把 Yarn 4 当 Yarn classic 升级版——它是几乎完全不同的产品

pnpm 10 · "monorepo 标准答案"

世界观：Zoltan Kochan 主导。扁平化是错的，但不必取消 node_modules——用 CAS + symlink 拓扑。在严格性 + 生态兼容性之间找到了 2026 年的平衡点。

设计核心（详见 P3.1）：

  • ~/.pnpm-store 全局 CAS，每包+版本只有一份物理文件
  • 项目内 hardlink 引用 store；symlink 拓扑控制可见性
  • 顶层 node_modules 只有 package.json 显式声明的包
  • 严格 peer dependency 解析（详见 P3.2）

monorepo 优势：

  • workspace 实现成熟（pnpm-workspace.yaml）
  • catalog 协议主导者（2024-08 引入，已成事实标准）
  • --filter 选择器强大（按包名 / 路径 / git 变更）
  • .pnpmfile.cjs hooks 可以系统性 patch 上游错误

2026 旗舰用户：Vue / Vite / Astro / TypeScript-eslint / shadcn/ui / Cloudflare Workers / 大量 Vercel 旗舰开源项目。事实上的"现代 JS monorepo 标配"。

Sweet spot：

  • 大型 monorepo（30+ 包）— 几乎是默认选择
  • 需要严格依赖管理的项目
  • 跨多项目共享磁盘 cache 的开发者机器

避坑：

  • 从 npm 切 pnpm 通常会暴露幻影依赖，需要修补 5-30 处
  • 严格 peer 模式对老旧依赖不友好——可能需要 auto-install-peers 或 .pnpmfile.cjs patch

Bun install · "速度怪兽 + 一体化新人"

世界观（详见 Bun 主题）：Jarred Sumner 主导。install / run / build / test 应该是同一个产品的 4 个面，不是 4 个独立工具。

设计选择：

  • Zig 实现—— 直接调系统调用，绕过 libuv 抽象
  • 二进制 lockfile（早期）→ JSON lockfile（Bun 1.2+）—— 2024-2025 切回 JSON 形式提升 diff 友好度，仍保留快速解析
  • hardlink + hoisting 布局—— 比 npm 类似但 hardlink 节省磁盘
  • workspace + catalog—— catalog 在 1.1.42（2024-12）跟进 pnpm 协议
  • npm 兼容 registry—— 直接用 npm registry，不要求改 publish 流程

独特能力：

  • 性能 5-10x 优于 pnpm，10-20x 优于 npm（cold install 场景）
  • 与 Bun runtime 深度整合——同一个工具搞定 install / run / test / build
  • Windows 一级支持（Bun 1.1+）

2026 状态：

  • 新项目 + 中小项目份额快速增长
  • 大型 monorepo 仍以 pnpm 为主流（peer 处理 + 复杂 hooks 的边界情况 Bun 在补）
  • 生态兼容性基本到位（剩余少量"对包管理器挑食"的奇葩包仍可能踩坑）

Sweet spot：

  • 新项目（特别是 SaaS / 工具型应用）
  • 极致 CI 速度需求（每分钟都贵的场景）
  • 已用 Bun runtime 的项目（一体化最大化）
  • 个人项目 / 副业项目（"装一个 Bun 替代 Node + npm + tsx + jest"）

避坑：

  • 大型 monorepo 切 Bun 之前要先做兼容性试跑——少量复杂依赖可能需要 fallback
  • Bun runtime 与 Node 仍有兼容性边界（详见 Bun P3.2），但 bun install 本身是 npm 兼容的，可以"用 Bun 装包但用 Node 跑"——这个混合用法是常见过渡方案

Deno + JSR · "另一个范式"

世界观：Ryan Dahl 在 Deno 反思 Node 设计的延续。不是"更快的 npm"，而是从 registry 层重做 JS 包管理。

关键设计：

  • URL-based imports（早期）—— import { foo } from 'https://deno.land/x/std/foo.ts'；package.json 不必要
  • JSR registry（2024-03）—— 改进版 URL imports：import { foo } from 'jsr:@scope/name'；不打 tarball；原生 ESM + TS；强签名
  • npm 兼容—— Deno 可以 import 'npm:react' 拉 npm 包；但 npm 不能反向拉 jsr/deno.land 包
  • deno.lock—— Deno 自家 lockfile
  • workspace—— Deno 1.39+ 支持，集中在 deno.json

独特能力：

  • 原生 TS 包（不用预编译）
  • cross-runtime（同一个 JSR 包在 Deno / Node / Bun / 浏览器跑）
  • scoped slugs 强制（避免 npm 全局命名空间污染）
  • 所有发布自动签名（与 npm provenance 类似但内置）

2026 现状：

  • Deno 自家生态（Fresh / Oak）使用 JSR
  • 少数库（标准库 / Deno 团队作品）发布到 JSR
  • 主流前端 / Node 生态仍以 npm 为主
  • 能否撼动 npm 还看 5-10 年

Sweet spot：

  • 用 Deno runtime 的项目
  • 发布 cross-runtime 库（同一个包在 Node/Deno/Bun 都跑）
  • 追求"原生 TS + 强签名"的新项目

避坑：

  • 不要尝试在 Node 项目里"用 JSR 替换 npm"——大部分依赖你要的还在 npm 上
  • JSR 是补充，不是替代——至少 2026 年不是

选型决策矩阵（2026-05）

把上面所有信息收敛成一张决策树：

场景 首选 备选 避免
新建大型 monorepo（30+ 包） pnpm 10 Bun install（如有性能需求） npm / Yarn classic
新建中型项目（5-30 包） pnpm 10 Bun install Yarn classic
新建小项目（< 5 包） Bun install / npm 11 pnpm 10 Yarn 4 PnP（无必要复杂度）
极致 CI 速度（每秒钟都贵） Bun install pnpm 10 npm
遗留 npm 项目维护期 npm 11（不动） 计划切 pnpm 10 盲目切 Bun（可能踩坑）
遗留 Yarn classic 项目 Yarn classic（不动） 计划切 pnpm 10 切 Yarn 4 PnP（升级路径不是无痛）
跨 runtime 库发布 Deno + JSR（先发 JSR） 同时发 npm 仅 npm（限制使用范围）
用 Bun runtime 的项目 Bun install pnpm 10（如生态兼容问题） npm（一体化优势丧失）
用 Deno runtime 的项目 Deno + JSR — 用 npm 不一致

03 应用场景

场景 1 · 团队选型：新 monorepo 该用什么？

背景：8 人前端团队，要把 3 个 React 项目合并成 monorepo。Tech Lead 倾向 Bun（"性能好"），团队成员用过 npm/pnpm。问 Bun 还是 pnpm。

决策维度：

  • monorepo 规模：3 个项目 + 共享 UI 库 + 共享 utils ≈ 10-15 包，中型
  • 严格性需求：要的，团队希望避免幻影依赖
  • CI 时间紧迫度：不极端，目前 npm 5 分钟，能容忍
  • 生态依赖兼容性：用了一些复杂依赖（patch-package, monorepo-aware tools）

建议：pnpm 10。理由：

  • monorepo 中型规模，pnpm 的 catalog + filter 收益明显
  • 严格性是核心需求，pnpm 是"严格 + 生态友好"的最佳平衡
  • Bun 的速度优势在中型项目落地不那么戏剧（cold install 几秒差距）
  • 有复杂依赖时 pnpm 更稳，Bun 兼容性边界仍可能踩坑
  • 未来切 Bun 留余地——pnpm-lock.yaml + Bun 互操作良好

这个判断的依据：选型不是看"最快的"，是看"匹配场景的"。Bun 在小项目 + 极致 CI 速度场景是首选，但这个团队都不是。

场景 2 · CI 时间紧迫的 SaaS 团队

背景：10 人 SaaS 团队，每天 50+ PR，每个 PR CI 30 分钟（其中 10 分钟在 npm install）。每月 CI 账单上千美元。考虑切 Bun install。

决策维度：

  • 性能价值：CI 时间直接 = 钱，10 分钟 → 1 分钟相当于每月省数百美元
  • 风险容忍：SaaS 不能允许部署 break；切包管理器有兼容性风险
  • 团队带宽：能投 1-2 周做切换 + 验证

建议：分阶段切 Bun install，runtime 仍用 Node。

  • 第 1 周：在 dev 分支用 Bun install 替代 npm install，跑全套测试，看是否 break
  • 第 2 周：CI 启用 Bun install + Node runtime 混合模式（"用 Bun 装、用 Node 跑"是 Bun 团队推荐的过渡方案）
  • 切完后保留 npm 作为 fallback（如发现某依赖不兼容立即可切回）
  • 1-2 个月后看是否值得把 runtime 也切到 Bun（这是另一个决策）

预期结果：CI install 时间从 10 分钟降到 1 分钟，整体 CI 从 30 分钟降到 21 分钟。投入产出比极佳。

场景 3 · 想选 Yarn 4 PnP，要不要选？

背景：工程师看到 Yarn Berry 文档说"zero-installs + 极致严格"，技术上心动。问选 Yarn 4 PnP 还是 pnpm。

决策维度：

  • 团队对 zero-installs（commit cache 到 git）有具体需求吗？
  • 项目依赖里有"对包管理器挑食"的奇葩吗？
  • 团队成员愿意为 PnP 适配花时间吗？

建议：除非有极强 zero-installs 需求，否则选 pnpm。

  • PnP 在技术上几乎完美，但生态成本是真实的
  • pnpm 提供 90% 的严格性 + 100% 的生态兼容
  • "我想试试新东西"不是好理由——把时间花在业务而不是包管理器调适上
  • 如果就是想体验 PnP，做个 side project 试，不要在生产项目冒险

05 常见陷阱

陷阱 1 · 用"快"作为唯一选型依据

包管理器营销战激烈，"我比 X 快 30 倍"是常见话术。但实际开发中，install 速度差距通常不是决定性——你日常装新依赖每次差几秒不那么戏剧，CI 才是大头。选型应优先看严格性 + monorepo 能力 + 生态兼容性，性能是次要维度。Bun 适合极致 CI 速度场景，但对中型项目 pnpm 通常是更全面的选择。

陷阱 2 · 把 Yarn 4 当 Yarn classic 升级

Yarn 1.x 和 Yarn 4 共享名字但是几乎完全不同的产品——配置语法、命令行、布局都不互通。从 Yarn 1 升 Yarn 4 不是"升级"，是"迁移"。如果你只是想要 Yarn classic 的小修复，留在 1.x 维护期；如果想要现代特性，直接评估 pnpm 而不是默认升 Yarn 4。

陷阱 3 · 在 Deno 项目中混用 npm 和 jsr

Deno 可以 import npm 包（import 'npm:react'），看起来"两边都能用"。但npm 包的依赖图最终仍要走 npm 生态——你的 lockfile 里会出现 npm: 和 jsr: 两套。原则上：选定 runtime（Node 或 Deno），就以那个 runtime 的 native registry 为主，跨 registry 依赖只在必要时用。

陷阱 4 · 把"主流就是最好的"当作选型理由

npm 是主流，但 npm 是"装上 Node 就有"的默认值，不是"最好"的代名词。市场份额 ≠ 工程价值。pnpm/Bun 在工程价值上对中大型项目超越 npm，是有具体技术理由的。反过来：因为"pnpm 大家都在用"就盲目切，但你的项目实际上是单包 + 5 个依赖，可能没有任何收益甚至增加复杂度。选型应基于项目特征，不是大势。

陷阱 5 · 在 monorepo 选型时忽略 catalog

2024-2026 catalog 是 monorepo 的关键能力——升级 React 不用改 50 处 package.json。选包管理器时如果不支持 catalog（npm 11 / Yarn 4 PnP 默认），monorepo 用起来会比 pnpm/Bun 累。这个差距比"装包速度"对长期生产力影响更大。
