幻影依赖与双 package hazard

扁平化 hoisting 不只是"性能优化"——它引入了三个结构性问题：(1) 幻影依赖（phantom dependency，未声明却能 require）；(2) doppelganger（同包多版本同时存在导致单例失效）；(3) 双 package hazard（CJS / ESM 双产物加载两次）。本章用 3 个真实 production bug 让读者意识到这不是洁癖、是工程问题——也是 pnpm / PnP 在 2017+ 起义的根本动机。

阅读时长 ~25 分钟 前置 P2.1（resolution + 4 种 layout） 主要来源 pnpm.io / Rush 文档 / Node ESM spec / 真实开源 issue

01 速览

  1. 三个问题都是 npm 3+ 扁平化的结构性副作用 —— 不是 npm 实现 bug，是"提升所有依赖到顶层"这个决定本身的代价。它们在 2015 年 npm 3 发布时就已存在，整个 JS 生态用了 10 年才接受"必须解决"。
  2. 幻影依赖（phantom dependency） —— 你的 package.json 没有声明 X，但 X 是某个传递依赖。hoisting 把 X 提到顶层后，require('X') 也能跑。看似正常但脆弱：升级、切包管理器、修改 dep 时 X 可能消失。
  3. Doppelganger —— 同一个包的多个版本同时存在 node_modules 中（不同子路径）。看似无害，直到 instanceof / 单例 / 全局状态在两份实例间不共享，bug 难复现。React"Hook 报错 You may have more than one copy of React"是 doppelganger 的典型症状。
  4. 双 package hazard —— ESM / CJS 双产物的包在不同条件下被加载两次，状态不共享。例子：库内部的"懒初始化 singleton"在 ESM 入口和 CJS 入口分别初始化一次，副作用执行两次。
  5. pnpm + PnP 各自从结构上消除这三个问题 —— pnpm 通过 symlink 拓扑限制可见性（顶层只有声明的依赖）+ 同版本物理一份；PnP 通过 .pnp.cjs 严格仲裁 require。这是它们对 npm 扁平化的"结构性回应"，不是"性能优化"。
  6. 诊断幻影依赖的工具 —— arethetypeswrong（包发布前检查 exports/双产物）、publint（package.json lint）、depcheck（找未声明的 import）、pnpm 直接报错。建议所有发包项目 CI 集成 publint + arethetypeswrong。
  7. 修复路径 —— 显式声明所有用到的依赖（最干净）；用 pnpm/PnP 让结构问题"无处藏身"；对单例敏感的库（React / Zustand 等）用 peer dependency 声明 + 用户保证版本一致。

02 底层逻辑

幻影依赖（phantom dependency）的产生机制

回到 P2.1 的扁平 hoisting 例子。my-app 声明 react@^18，react 传递依赖 loose-envify。npm 3+ 装出来的 node_modules：

扁平后的结构 
my-app/
├── package.json   # 只声明了 react
└── node_modules/
    ├── react/         # 顶层
    └── loose-envify/  # ← 提到顶层！

现在写 my-app 的源码：

javascript 
// my-app/src/foo.js
const envify = require('loose-envify');
console.log(envify);  // ← 居然能跑！

能跑的原因：Node resolution 算法逐级向上查 node_modules，loose-envify 在 my-app/node_modules 里，命中。Node 不关心"它是不是你声明的"——这是 JS 生态没有 import 边界检查的结构性后果。

什么时候会暴雷：

  • 升级 react—— react 的新版本可能不再依赖 loose-envify（换了实现），loose-envify 不再被提到顶层；my-app 的 require 突然报错
  • 切包管理器—— pnpm 装出来的 layout 顶层只有 react，没有 loose-envify；my-app 切到 pnpm 后立即崩溃
  • removed transitive—— react 的另一个传递依赖（不是 loose-envify）被升级，触发 npm 重做 hoisting，loose-envify 可能因为新的版本冲突回退到嵌套位置；require 失败
  • 修复 audit—— npm audit fix 升级某个上游 sub-dep，连带改了 hoisting 决策

这是工程灾难的本质：bug 触发条件与代码本身无关，与表面看起来不相关的某个其他依赖的版本变化有关。debug 极难——你修着 react 升级问题，结果发现是 loose-envify 不见了。

真实数据：根据 pnpm 团队 2022 年的统计，主流 npm 项目中平均 15-30 个文件使用了未声明的依赖。Rush 团队（Microsoft）在大型 monorepo 内部审计发现，单个 monorepo 可能有数百处幻影依赖，是切 pnpm 时最大的迁移成本。

Doppelganger（同包多版本）

当 hoisting 遇到版本冲突时，"赢家"留在顶层，"输家"回退到嵌套位置：

doppelganger 示例 
my-app/
├── package.json   # 直接依赖 react@^18
└── node_modules/
    ├── react/                    # version 18.3.1（顶层）
    ├── some-old-pkg/
    │   ├── package.json          # 依赖 react@^17
    │   └── node_modules/
    │       └── react/            # version 17.0.2（嵌套）！doppelganger 出现
    └── ...

# 现在 my-app/src/foo.js 用了 react@18
# some-old-pkg 内部用了 react@17
# 它们各自 require('react') 会走不同 resolution 路径，得到不同对象

表面无害——each package gets its expected version。但对依赖单例 / 全局状态的库，这是灾难：

典型场景：React Hook 报错

错误信息 
Warning: Invalid hook call. Hooks can only be called inside of the body
of a function component. This could happen for one of the following reasons:
1. You might have mismatching versions of React and the renderer (such as React DOM)
2. You might be breaking the Rules of Hooks
3. **You might have more than one copy of React in the same app**
   ↑ doppelganger 经典症状

诊断：

bash 
# 看 react 在 node_modules 里出现了几次
find node_modules -name "react" -type d -maxdepth 3
# node_modules/react
# node_modules/some-old-pkg/node_modules/react   ← doppelganger！

# 让 npm 显示版本树
npm ls react

# Yarn 用
yarn why react

为什么 React Hook 翻车：React 18 的 Hook 实现依赖模块级 module-private 状态（"current dispatcher"等）。如果你的代码 require 的是 react@18.3.1（顶层），some-old-pkg 内部 require 的是 react@17.0.2（嵌套），两者运行时是不同的 module instance，state 不共享，Hook 调用栈对不上。

类似受害者：

  • Zustand / Jotai / Valtio（store 实例必须全局唯一）
  • styled-components / emotion（CSS-in-JS 的 sheet 注册表）
  • history（react-router 用）（路由器状态全局唯一）
  • graphql / apollo（schema 注册）

npm 的部分缓解：dedupe 和 overrides

json 
// package.json，强制把所有 react 引用拉到 ^18.3.1
{
  "overrides": {
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  }
}

npm 8+ / pnpm 都支持 overrides；Yarn 用 resolutions。但这是用户事后补救——根本问题是扁平化对"包用同版本"不能保证。

pnpm 的结构性消解：pnpm 把每个 包@版本 在 store 里物理上只放一份。如果项目两次需要 react@18.3.1（一处直接、一处传递），两处都 hardlink 指向同一份；require.cache 命中相同物理路径，module instance 自动合一。版本不同时（react@17 + react@18）pnpm 不会强制合并，但用户能立即看到（顶层只有一份显式声明的，多版本不被默认 hoist）。

双 package hazard（dual package hazard）

这是 ESM / CJS 时代特有的问题。一个包同时发 CJS（CommonJS）和 ESM（ES Module）两份产物：

json 
// some-pkg/package.json
{
  "name": "some-pkg",
  "main": "./dist/index.cjs",
  "module": "./dist/index.mjs",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}

同一个项目内：

  • app 主代码用 ESM import { foo } from 'some-pkg' → 加载 .mjs
  • 某个老依赖用 CJS require('some-pkg') → 加载 .cjs
  • 同一个包被加载了两次！两份 module instance！

当 some-pkg 有内部状态时灾难发生：

javascript 
// some-pkg/dist/index.mjs / index.cjs（同一份源码两种格式）
let counter = 0;
let initialized = false;

export function init() {
  if (initialized) return;
  initialized = true;
  // 注册全局副作用，例如启动定时器、连接数据库
  setInterval(() => counter++, 1000);
}

export function getCounter() {
  return counter;
}

// 现在 ESM 路径调用 init() → 副作用执行 1 次（启动 1 个定时器）
// CJS 路径调用 init() → 副作用又执行 1 次（启动第 2 个定时器）
// 用户从 ESM 路径调 getCounter() → 看到 ESM 的 counter
// 同时 CJS 路径的 counter 在独立增长，被忽略

真实受害者：

  • node-fetch v3（ESM-only）vs v2（CJS）—— 大量项目同时 import 两版导致 fetch polyfill 注册两次
  • uuid—— 内部 ID 计数器在双产物下不一致
  • chalk v5（ESM-only）—— 强制 ESM 不再有 dual hazard，但不少老项目装了多版本

检测：arethetypeswrong + publint

bash 
# 发包前自检
npx publint                    # 检查 package.json 的 exports / main / module 配置
npx attw --pack .              # arethetypeswrong：检查 ESM/CJS dual + types

# 输出会标出"dual package hazard"风险

避免方案（包作者侧）：

  • ESM-only—— 只发 .mjs 不发 .cjs。彻底没 dual hazard，代价是 CJS 用户用不了（除非用 dynamic import）
  • CJS wrapper around ESM—— ESM 是真身，CJS 入口 re-export ESM。技术上能做到 module instance 合一，但配置复杂
  • 无状态库—— 如果库本身没有 module-level mutable state（纯函数 / 工具函数），dual hazard 无害

避免方案（用户侧）：

  • 统一项目模块系统（要么全 ESM 要么全 CJS）
  • 用 overrides / resolutions 强制依赖某一版本，防止同时装多个
  • 检查 npm ls some-pkg 输出，确认只有一个 instance

3 个真实 production bug 案例

案例 1 · Stripe SDK 升级翻车（幻影依赖）

2023 年某 SaaS 把 stripe SDK 从 11.x 升 12.x。CI 通过，部署到 staging。结果某个支付流程 throw Cannot find module 'stripe-utils'。

  • 诊断：源码里有 require('stripe-utils')，但 package.json 从来没声明过
  • 根因：stripe@11 内部依赖 stripe-utils，hoisting 把它提到顶层；stripe@12 重构后不再依赖 stripe-utils，hoisting 不再提供
  • 修复：显式 npm install stripe-utils。但真正的修复是切 pnpm 让这种问题装包时就报错，而不是部署后翻车

案例 2 · Storybook 7 升级 React Hook 报错（doppelganger）

2024 年某 React 项目升 Storybook 7。dev 模式下打开 Storybook，立即 console 报 "You may have more than one copy of React"。

  • 诊断：npm ls react 显示项目顶层 react@18，但 storybook 的某个 internal addon 依赖 react@17，回退到嵌套位置
  • 临时方案：用 overrides 强制全部 react ≥ 18
  • 根因：Storybook 7 的某个 addon 没及时升级 peer 范围；overrides 治标
  • 切 pnpm 后：pnpm 严格 peer 模式直接报警告，迫使你在装包阶段就修

案例 3 · node-fetch dual hazard 引发的奇诡 401（双 package hazard）

2023 年某全栈项目用 node-fetch v3（ESM）做 SSR。同时某个老 SDK 内部用 node-fetch v2（CJS）。生产环境出现间歇性 401——每隔几个请求一次。

  • 诊断：node-fetch v2 / v3 各自维护一份 cookie jar；用户登录走 v3 路径设了 cookie，但后续 SDK 调用走 v2 路径，没拿到 cookie
  • 修复：统一用 native fetch（Node 18+ 内置）+ undici cookie jar；或强制 SDK 升级 node-fetch v3
  • 教训：dual package hazard 在有内部状态的库下很危险。无状态库（lodash 等）不受影响

03 应用场景

场景 1 · 切包管理器前的诊断

背景：从 npm 切 pnpm，担心生产暴雷。如何在切之前找出所有幻影依赖？

建议工作流：

  • 先在分支上 pnpm install 而不删 package-lock.json
  • 跑全套测试 + lint + build——pnpm 严格 layout 下幻影依赖会立即 module-not-found
  • 用 npx depcheck 主动扫描"未声明的 import"
  • 对每个被发现的幻影依赖，pnpm add <name> 显式声明
  • 典型项目修 5-30 处
  • 验证完后才删 package-lock.json，正式切 pnpm

场景 2 · React 项目"You may have more than one copy of React"

背景：突然报 React 多 copy 错误，dev 不能启动。

诊断步骤：

bash 
# Step 1: 看 react 装了几份
npm ls react
# 或 pnpm ls react / yarn why react

# Step 2: 找到嵌套的 react
find node_modules -name "react" -type d | grep -v ".pnpm"

# Step 3: 看是哪个包导致
npm ls --all | grep -B 5 "react@17"

修复路径（按优先级）：

  • 升级旧依赖到兼容当前 React 版本（最干净）
  • 临时用 overrides（npm 8+）/ resolutions（Yarn）强制单版本
  • 如果是 monorepo 内部：把 react 设为 peer dependency，让 app 决定版本

场景 3 · 发库前的 dual package hazard 自检

背景：维护一个 npm 库，要发新版。需要支持 ESM + CJS。

建议 CI 流程：

  • build 产 .mjs + .cjs + .d.ts + .d.cts
  • 跑 publint 检查 package.json 配置
  • 跑 arethetypeswrong --pack . 检查 dual + types 正确
  • 跑一个 minimal ESM consumer + CJS consumer 验证
  • 发版前看 npmjs.com 的"Files"标签，确认 dist 内容

这套自检对应是 npm 生态社区共识——typescript-eslint / Vite / Vitest / Astro 都在 CI 跑这两个工具。

04 典型代码

用 depcheck 找未声明的依赖 
npx depcheck

# 输出：
# Unused dependencies
# * eslint-plugin-old
# Unused devDependencies
# * jest-old
# Missing dependencies   ← 幻影依赖！
# * loose-envify: ./src/utils.js
# * lodash.merge: ./src/api.js
overrides 强制单版本 
// npm 8+ / pnpm
{
  "overrides": {
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  }
}

// Yarn classic / Berry
{
  "resolutions": {
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  }
}
peer dependency 声明（库作者） 
// my-react-lib/package.json
{
  "name": "my-react-lib",
  "peerDependencies": {
    "react": "^18.0.0",
    "react-dom": "^18.0.0"
  },
  "peerDependenciesMeta": {
    "react-dom": { "optional": true }
  }
}

// 用 peer dependency 声明：让 app 提供 react，
// 而不是 my-react-lib 自己装。这样不会出现 doppelganger。
publint + arethetypeswrong CI 集成 
# .github/workflows/release.yml（节选）
- name: Validate package
  run: |
    pnpm build
    pnpm publint
    pnpm attw --pack .

# 失败示例输出（attw）：
# .src/index.cjs is treated as ESM by Node, not CJS
# Dual package hazard: ESM and CJS both initialized
检测 doppelganger 的工具脚本 
#!/usr/bin/env bash
# 找 node_modules 中重复出现的包
find node_modules -name "package.json" -not -path "*/.bin/*" \
  | xargs -I {} cat {} 2>/dev/null \
  | jq -r '"\(.name)@\(.version)"' 2>/dev/null \
  | sort | uniq -c | sort -rn | awk '$1 > 1 {print}'

# 输出：
#   3 react@17.0.2
#   2 react@18.3.1
#   ...
# 任何 count > 1 的都是潜在 doppelganger

05 常见陷阱

陷阱 1 · "我从来没遇到过幻影依赖问题"

一个常见错觉。你没遇到不代表没有——只是"目前没暴雷"。幻影依赖是定时炸弹，会在升级、切包管理器、修依赖时突然引爆。主动诊断（depcheck 或切 pnpm）总好过事后救火。

陷阱 2 · 把 doppelganger 修复用 overrides 当根本解

overrides 治标——它强制版本统一，但问题根源是某个上游包的 peer 范围太窄。长期路径是给上游提 PR 扩 peer 范围，或换一个维护更好的替代。否则下次升级时又会出现新的不匹配。

陷阱 3 · 把 ESM-only 看成"过激"

"为什么不发 dual？dual 兼容更好" —— dual 兼容是有代价的（dual hazard 风险 + 维护两份产物）。chalk / nanoid / nanoid 等先发 ESM-only 的库迫使生态升级，效果证明可行。如果你的库本来就主要服务现代 Node + bundler 用户，ESM-only 是合理选择。

陷阱 4 · 在测试环境忽略幻影依赖

"测试都过了 —— 应该没问题吧？" —— 测试可能只测试到了 hoisting 提供的依赖，没测试到边界场景。真正可靠的检验：在 CI 用 pnpm 装一遍，跑全套测试。pnpm 的严格 layout 把幻影依赖暴露在装包阶段，CI 就能拦住。

陷阱 5 · 把"切 pnpm 后报错"看成 pnpm 的问题

切 pnpm 后报 module-not-found 不是 pnpm 错——是原本扁平化掩盖的幻影依赖暴露了。pnpm 是诊断工具不是肇事者。这种情况下加显式声明就解决，不要骂 pnpm "破坏了我能跑的项目"。
