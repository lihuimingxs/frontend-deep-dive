Node module resolution + 4 种 layout

从 require('foo') / import 'foo' 出发，把"为什么 npm 装出来这样、pnpm 装出来那样"讲透。核心问题：Node 拿到一个 bare specifier（"foo" 这样的字符串），是怎么找到它对应的物理文件的？4 种 node_modules layout（嵌套 / 扁平 hoisting / pnpm symlink / Yarn PnP）都是这个 resolution 算法的不同物化形态。理解了 resolution，下一章的"幻影依赖"就是水到渠成的结论。

阅读时长 ~30 分钟 前置 用过 require / import；P1.1 历史 主要来源 Node.js 官方 modules 文档 + ESM resolution spec + 各家包管理器实现

01 速览

  1. Node 有两套 resolution 算法 —— CJS（require，1999 CommonJS 起源，Node 自家实现）+ ESM（import，ECMA-262 spec 定义 + Node 适配）。两套核心步骤相似（都是逐级向上查 node_modules）但细节差异巨大（exports 字段、conditional exports、import attributes、文件扩展名要求）。
  2. CJS 算法 5 步 —— (1) 是 ./ / ../ 相对路径吗？直接拼；(2) 是 Node 内置（node:fs）吗？跳过 fs 查找；(3) 从当前目录开始，逐级向上查 node_modules/<name>；(4) 找到目录后，看 package.json 的 main / exports；(5) 缺扩展名时尝试 .js / .json / .node / index.js。
  3. ESM 算法更严 —— 必须带扩展名（./foo.js 不能省 .js）+ exports 字段是强制 gateway（package 没声明 exports 的子路径不能 import）+ 支持 import attributes（import json from './data.json' with { type: 'json' }）+ 支持 conditional exports（按 "node" / "browser" / "import" / "require" 条件解析）。
  4. node_modules 4 种物理 layout —— (1) 嵌套（npm 1-2，2010-2015）；(2) 扁平 hoisting（npm 3+ / Yarn classic / npm 11，2015-）；(3) pnpm symlink 拓扑（pnpm 1+，2017-）；(4) Yarn PnP（无 node_modules）（Yarn 2+, 2019-）。每种是 resolution 算法走在不同物理结构上的产物。
  5. resolution 算法本身不区分 layout —— Node 实现是同一份代码；它逐级向上查 node_modules，无论那个 node_modules 内容是嵌套还是扁平还是 symlink。包管理器只决定怎么"摆放文件"，让 Node 算法以最自然的方式找到包。
  6. PnP 是唯一例外 —— Yarn PnP 不用 node_modules；它在 Node 启动时注入一个 .pnp.cjs loader，劫持 Node 的 resolution，直接从 manifest 查找包位置。本质上是"另一套 resolution 算法"。
  7. package.json 的 exports 字段是 2026 的关键 —— 控制"哪些子路径外界可以 import"。exports 不只是文档，是强制约束——ESM 下没声明的子路径直接 throw。这影响每个发包者的 API surface 设计。

02 底层逻辑

CJS resolution 算法（require）

Node 的 require(X) 在调用时执行如下流程（来自 Node 官方文档，简化版）：

Step 1 · X 是 Node 内置模块？（如 node:fs / node:path / 早期的 fs bare name）

  • 是 → 返回内置模块，结束
  • 否 → 继续

Step 2 · X 是相对路径？（以 ./ / ../ / / 开头）

  • 是 → 走 LOAD_AS_FILE（试 X、X.js、X.json、X.node）→ 失败则 LOAD_AS_DIRECTORY（试 X/package.json#main、X/index.js）
  • 否 → 继续到 Step 3

Step 3 · 从 caller 所在目录开始，逐级向上查 node_modules

  • 例如 caller 在 /proj/src/foo.js，require('lodash')
  • 查找路径序列：/proj/src/node_modules/lodash → /proj/node_modules/lodash → /node_modules/lodash
  • 第一个匹配的赢——这是 Node 算法的核心

Step 4 · 找到目录后，看 package.json

  • 有 exports 字段（Node 12+） → 按 exports 解析，主入口是 "."
  • 有 main 字段 → 用 main 指向的文件
  • 都没有 → 试 index.js

Step 5 · 找不到 → 抛 MODULE_NOT_FOUND 错误

CJS 算法的几个隐藏特性：

  • 逐级向上是核心——只要你在祖先目录装了某包，就能 require。这是 hoisting 能"提到顶层就到处可用"的根本机制
  • cache 命中——require 第一次执行后结果缓存到 require.cache，再次 require 直接拿；这是 module singleton 的来源
  • 动态可计算——require(someVar) 合法，运行时才能确定指向；这是 CJS 比 ESM 灵活但难静态分析的原因

ESM resolution 算法（import）

ESM 是 ECMA-262 spec 定义 + Node 适配（《ESM Resolution》一节）。它和 CJS 共享"逐级向上查 node_modules"的核心，但有几个关键不同：

差异 1 · 必须显式扩展名

ESM import './foo.js' 是合法的，import './foo' 默认不会自动加 .js（除非用 --experimental-specifier-resolution=node，已弃用）。原因：HTTP / 浏览器规范都要求 URL 完整，Node 想保持兼容。

差异 2 · exports 字段是强制 gateway（不只是文档）

json 
// some-pkg/package.json
{
  "name": "some-pkg",
  "exports": {
    ".": "./dist/index.js",
    "./utils": "./dist/utils.js"
  }
}

有了上面的 exports 后：

  • import 'some-pkg' ✅ → ./dist/index.js
  • import 'some-pkg/utils' ✅ → ./dist/utils.js
  • import 'some-pkg/dist/internal.js' ❌ → throw ERR_PACKAGE_PATH_NOT_EXPORTED

为什么这个限制重要：包作者第一次拥有真正的 API 边界控制。在 CJS 时代任何用户都能 require('lodash/fp/curry') 进入未公开内部；ESM 下只有 exports 声明的路径外界可达。

差异 3 · conditional exports

json 
{
  "name": "isomorphic-fetch",
  "exports": {
    ".": {
      "node": "./dist/node.js",
      "browser": "./dist/browser.js",
      "default": "./dist/default.js"
    }
  }
}

resolver 按 condition 顺序匹配。bundler / runtime 各自定义自己的 condition：

  • Node ESM 默认：node + import + default
  • Node CJS 默认：node + require + default
  • Bun runtime：bun + node + import + default
  • Vite dev：module + browser + development + ...
  • Vite build：module + browser + production + ...

这套机制让一个包能"在不同环境暴露不同实现"，是 isomorphic 库的现代标准（详见 Vite P3.1）。

差异 4 · import attributes（旧名 import assertions）

javascript 
import data from './config.json' with { type: 'json' };
import css from './style.css' with { type: 'css' };  // 浏览器 / Bun

2024+ 的 stage 4 提案（详见 ECMA P7.4）。声明 import 类型，让 resolver / bundler 能正确处理非 JS 资源。

差异 5 · import maps（浏览器 + Deno）

ESM 在浏览器 / Deno 通过 <script type="importmap"> 或 deno.json#imports 自定义 bare specifier 映射。Node 不支持 import maps（截至 2026 仍在 RFC）。

Layout 1 · 嵌套 node_modules（npm 1-2，2010-2015）

每个包独立解析自己的依赖，物理上嵌套：

嵌套布局示例 
my-app/
├── package.json   # 声明依赖 react, lodash
├── node_modules/
│   ├── react/
│   │   ├── package.json   # 声明依赖 loose-envify
│   │   └── node_modules/
│   │       └── loose-envify/
│   │           ├── package.json   # 依赖 js-tokens
│   │           └── node_modules/
│   │               └── js-tokens/
│   └── lodash/  # 无依赖

Resolution 行为：

  • require('react') 从 my-app/src/foo.js 出发 → 查 my-app/src/node_modules（无）→ my-app/node_modules（命中 react）✅
  • require('loose-envify') 从 my-app/node_modules/react/lib/... 出发 → 查 react/lib/node_modules（无）→ react/node_modules（命中 loose-envify）✅
  • require('lodash') 从 react/lib/... 出发 → 逐级向上查 → my-app/node_modules（命中 lodash）✅—— react 没声明 lodash 但能 require 到，已经埋下幻影依赖伏笔！

优势：每个包的依赖独立 sandbox，版本冲突时各自相安无事。如 react 用 lodash@3、my-app 用 lodash@4，两者并存不冲突。

劣势：磁盘爆炸（同一个 lodash 可能装 N 份）、Windows 路径长度灾难（嵌套 7-8 层超过 260 字符）。

Layout 2 · 扁平 hoisting（npm 3+ / Yarn classic，2015-）

npm 3 引入扁平化：把所有依赖提到根 node_modules，冲突时回退到嵌套。

扁平 hoisting 示例 
my-app/
├── package.json
├── node_modules/
│   ├── react/
│   ├── lodash/
│   ├── loose-envify/   ← 提到顶层！React 的传递依赖
│   └── js-tokens/      ← 同样提到顶层

# 仅当版本冲突时才回退嵌套
# 例如 my-app/node_modules/some-pkg/node_modules/lodash（如果 some-pkg 要 lodash@3 而顶层是 lodash@4）

Resolution 行为：

  • require('lodash') 从任何子目录都能命中顶层的 lodash
  • 提升顺序按解析顺序，非确定性（同一份 package.json 不同时间装出不同布局；Yarn 1 加了 sort 保证确定，npm 5+ 也修了）
  • 遇到版本冲突，"提升获胜者"决定：通常顶层版本是 package.json 直接声明的版本；冲突的传递依赖回退到嵌套

幻影依赖核心：源码里 require('loose-envify') 能跑——尽管 my-app 的 package.json 没声明它。这是 npm 3 扁平化的结构性副作用，不是 bug 是设计本身。详见 P2.2。

Layout 3 · pnpm symlink 拓扑（pnpm 1+，2017-）

pnpm 用 3 层链接拓扑，把每个包 sandbox 到独立目录，但避免重复存储：

pnpm 拓扑（简化） 
my-app/
├── package.json
└── node_modules/
    ├── react → .pnpm/react@18.3.1/node_modules/react   ← symlink
    ├── lodash → .pnpm/lodash@4.17.21/node_modules/lodash
    └── .pnpm/
        ├── react@18.3.1/
        │   └── node_modules/
        │       ├── react/   ← hardlink 到 ~/.pnpm-store
        │       └── loose-envify → ../../loose-envify@1.4.0/node_modules/loose-envify
        ├── loose-envify@1.4.0/
        │   └── node_modules/
        │       ├── loose-envify/
        │       └── js-tokens → ../../js-tokens@4.0.0/node_modules/js-tokens
        ├── js-tokens@4.0.0/
        │   └── node_modules/
        │       └── js-tokens/
        └── lodash@4.17.21/
            └── node_modules/
                └── lodash/

# 全局 store 在 ~/.pnpm-store/ —— 每个版本物理上只有一份

Resolution 行为：

  • 顶层 node_modules/<name> 是 symlink，只有 package.json 直接声明的依赖。其他包不在顶层——require 找不到
  • 每个包的传递依赖在 .pnpm/<name>@<ver>/node_modules/ 内 sandbox；它们走 Node resolution 时只能看到自己的 transitive deps，不会"漂"到别的包的依赖
  • 所有 hardlink 指向 ~/.pnpm-store，同一版本全机一份物理文件，多项目共享磁盘

关键洞见：pnpm 不是"另一套 resolution 算法"，它仍走标准 Node resolution；只是把 node_modules 摆放成让幻影依赖结构性不可能。这是 pnpm 设计的精妙处。详见 P3.1。

Layout 4 · Yarn PnP · 没有 node_modules（Yarn 2+, 2019-）

Yarn Berry 走最激进路线：取消 node_modules，劫持 Node resolution。

Yarn PnP 布局 
my-app/
├── package.json
├── yarn.lock
├── .pnp.cjs              ← Node loader！包含全部 resolution 决策
├── .pnp.loader.mjs       ← ESM loader 版本
└── .yarn/
    └── cache/
        ├── react-npm-18.3.1-abc.zip       ← 不解压！
        ├── lodash-npm-4.17.21-def.zip
        └── ...

# 启动 Node 时：node --require .pnp.cjs your-script.js
# 或在 package.json 加 "type": "module" + import .pnp.loader.mjs

Resolution 行为：

  • require('react') → .pnp.cjs 拦截 → 查内置 manifest → 知道 react 在 cache/react-npm-18.3.1-abc.zip 内 / dist/index.js 路径 → 返回该路径（zip 内部用 zip-fs 读）
  • 没有 node_modules 目录！Node 标准 resolution 找不到，必须走 .pnp.cjs
  • 每个 require 调用都经过 manifest 仲裁——严格性绝对（你 require 的必须在 manifest 里）

优势：

  • install 极快（不解压 + 不建立目录树）
  • 磁盘最省（zip 不解压，只在执行时读）
  • 严格性最高
  • "zero-installs"——commit cache + .pnp.cjs 到 git，新 clone 不用 install

劣势：

  • 需要工具链适配（很多包假设 node_modules 存在；webpack/metro/vite 各自需要 plugin）
  • VS Code 需要 @yarnpkg/sdks 配合
  • 调试栈跟踪指向 zip 内部

2026 年 PnP 仍小众但坚定（详见 P1.2）。

4 种 layout 在同一项目下的对比

维度 嵌套 扁平 hoisting pnpm symlink Yarn PnP
磁盘占用 大（重复） 中 小（hardlink） 最小（zip）
Resolution 速度 快（局部） 快 快 极快（无 IO）
幻影依赖 少（局部范围） 严重 无 无
doppelganger 不可避免 偶发 不会（同版本物理一份） 不会
需要 Node 修改？ 不需要 不需要 不需要 需要 .pnp.cjs loader
工具链兼容性 完全兼容 完全兼容 99%（rare 工具假设扁平） ~80%（要 patch）
Windows 友好 差（路径长度） 好 好 好
💡 关键洞见：算法不变，layout 不同

Node resolution 算法 15 年来基本没变（CJS 部分；ESM 是新加的）。所有"包管理器之争"本质上是"node_modules 应该长什么样"之争。每种 layout 都是同一个 resolution 算法跑在不同物理结构上的结果。理解了这一点你就明白：选包管理器不是选"算法"，是选"对未声明依赖的态度"。

03 应用场景

场景 1 · ESM 升级踩坑：包没在 exports 里声明 dist 子路径

背景：把项目从 CJS 切到 ESM（package.json 加 "type": "module"）。突然报错：Package subpath './dist/internal' is not defined by "exports"。

诊断：在 CJS 时代 require('some-pkg/dist/internal') 能跑（CJS resolver 不强制 exports）；切 ESM 后 exports 字段是强制 gateway，包作者没在 exports 里声明 ./dist/internal 路径，import 直接被拒。

修复：

  • 正确做法：用包公开 API 替代深层路径（import { internal } from 'some-pkg'）
  • 临时方案：给包提 PR 加 exports 子路径声明
  • 最后退路：用 node --no-experimental-require-module + 手动改 import 为 createRequire + require（不推荐）

背景知识：这是 ESM 严格化的代价。早期"看到 dist 就 import"的习惯在 ESM 下不可持续——包作者第一次有真正的 API 边界控制。

场景 2 · Vite + pnpm monorepo 内部包不热更新

背景：pnpm monorepo，apps/web 引用 packages/ui 的组件。改 packages/ui 的源文件，apps/web 的 Vite dev server 不响应——必须刷页才看到。

诊断：Vite 默认 server.watch.ignored 包含 **/node_modules/**，避免监听数千个依赖包。但 pnpm 把 packages/ui symlink 到 node_modules/.pnpm/...，修改的物理文件路径也在 node_modules 内，被 Vite 忽略。

修复：在 vite.config.js 显式不忽略 workspace 包：

javascript 
// vite.config.js
export default {
  server: {
    watch: {
      ignored: ['!**/node_modules/.pnpm/@my-org+ui*/**']
    }
  },
  // 或者直接禁用 server.fs 限制
  resolve: {
    preserveSymlinks: false  // 默认就是 false；让 Vite 跟随 symlink 监听真实路径
  }
}

详见 Vite P2.1。这是 pnpm symlink layout 与 Vite watcher 的著名交互陷阱。

场景 3 · 切 PnP 后 webpack-loader 报错

背景：从 Yarn classic 升级 Yarn 4（PnP 模式）。webpack 自定义 loader 报 Cannot find module 'foo'，明明 foo 在 yarn.lock 里。

诊断：自定义 loader 用 require.resolve('foo', { paths: ['/some/dir/node_modules'] }) 显式指定查找路径。PnP 下没有 node_modules，这种"假设布局"的代码直接失败。

修复：

  • 升级到 PnP 兼容版的 webpack（webpack 5+ 内置支持）+ pnp-webpack-plugin（旧版 webpack 4）
  • 修改自定义 loader：用 require.resolve('foo')（不带 paths）让 PnP loader 接管
  • fallback：yarnRcYml 配 nodeLinker: node-modules 退回扁平 hoisting

教训：PnP 的"严格优雅"换来的是所有"假设 node_modules 存在"的代码都要适配。这是 PnP 5 年来未能主流化的根本原因。

PnP loader 工作原理（简化） 
// .pnp.cjs（简化伪代码）
const manifest = {
  packageRegistryData: [
    ['react', [['npm:18.3.1', {
      packageLocation: './.yarn/cache/react-npm-18.3.1-abc.zip/node_modules/react/',
      packageDependencies: [['loose-envify', 'npm:1.4.0']]
    }]]],
    // ...
  ]
};

// 劫持 Node 的 resolution
const Module = require('module');
const originalResolve = Module._resolveFilename;
Module._resolveFilename = function(request, parent, ...rest) {
  // 如果是 PnP 管理的包，从 manifest 查找
  const pnpResolved = pnpResolve(request, parent);
  if (pnpResolved) return pnpResolved;
  // 否则走原始 resolution
  return originalResolve.call(this, request, parent, ...rest);
};

// 加上 zip-fs 让 Node 能读 zip 内的文件
require('@yarnpkg/fslib').xfs.mountZip();

05 常见陷阱

陷阱 1 · ESM 下省略 .js 扩展名

从 CJS 切 ESM 时最常见的报错。import './foo' 不会自动加 .js。修法：要么所有 import 显式带扩展名，要么打开 TypeScript 的 "moduleResolution": "Bundler" 让 bundler 解决（但这意味着 tsc 编译产物不能直接被 Node 跑，需要 bundler 介入）。

陷阱 2 · 把 exports 字段当文档

"exports 是给文档用的吧？" —— 不，它是强制约束。ESM 下没声明的子路径直接 throw。这意味着发包时设计 exports 等于设计 API surface——一旦发了某个子路径，就不能轻易撤销，否则 break 用户。

陷阱 3 · 在 monorepo 把内部包当外部包"硬编码路径"

有人写 require('../../packages/ui/src/Button') 直接走相对路径。这绕过了 workspace 的 symlink + package.json exports，等于故意制造"硬编码内部细节"。正确做法是用 workspace:* 声明依赖 + 走包名 import（import { Button } from '@app/ui'），让 resolution 算法 + workspace 管理。

陷阱 4 · 假设 require.cache 跨 layout 一致

require 的 module singleton 依赖同一物理路径只 require 一次。pnpm symlink 下，如果某个包被 require 两次但走不同 symlink（罕见但 monorepo 中可能），可能命中两个 cache 项。pnpm 通过 realpath 把 symlink 指向的物理路径作为 cache key 解决（默认行为），但配置不当（如 preserveSymlinks: true）会出现"同一个包加载两次"的诡异 bug。

陷阱 5 · 在 PnP 下用 fs.readdirSync(node_modules)

很多老工具假设 node_modules 目录存在，直接 fs.readdir 列出依赖。PnP 下没有这个目录，readdir 抛 ENOENT。正确做法：用 Node 标准的 require.resolve / import.meta.resolve，让 resolver（包括 PnP）接管。这也是 PnP 兼容性差的最大单一原因。
