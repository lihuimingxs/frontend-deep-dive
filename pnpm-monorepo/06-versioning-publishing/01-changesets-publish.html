Changesets + provenance + workspace publish

monorepo 写出来要发到 npm。但12 个包同时 publish、版本怎么联动、内部 workspace:* 怎么改写、CI 怎么对抗供应链攻击——这些是 P6 的主题。本章讲清楚 2026 年开源 monorepo 的事实标准方案：Changesets 工作流 + npm provenance 签名 + workspace 协议自动改写。完整 GitHub Actions 配置 + 真实开源案例。

阅读时长 ~25 分钟 前置 P4（workspace 协议）+ P5（任务编排） 主要来源 changesets/changesets repo / npm provenance 文档 / typescript-eslint / Astro / Vite 项目实践

01 速览

  1. Changesets 是 2026 monorepo 发版事实标准 —— 由 Atlassian 出品（typescript-eslint / Astro / Vite / Vitest / Remix / shadcn/ui / Cloudflare Workers 等都用）。核心思路：每次 PR 写一个 markdown 描述变更，CI 聚合成 changelog + 自动发版 PR。
  2. 4 步工作流 —— (1) 开发 + 写 changeset（pnpm changeset 交互生成 markdown）；(2) PR 合并到 main，CI 跑 changeset version 把 pending changesets 聚合成 changelog + bump 版本号；(3) CI 创建"Version Packages" PR；(4) 人工 review 合并 → CI 触发 changeset publish 自动发到 npm。
  3. 独立版本（默认） vs 固定版本 —— 独立：每个包独立 SemVer，依赖关系自动联动（packages/ui 改 → 依赖 ui 的 apps/web 自动 bump）；固定：fixed: ['@app/*'] 让一组包始终相同版本（紧耦合 SDK 用）。
  4. prerelease 模式 —— changeset pre enter alpha 进入预发布阶段，所有 publish 自动加 alpha.0 / alpha.1 后缀。changeset pre exit 退出。用于 major 版本发布前的多轮 RC。
  5. snapshot 模式 —— changeset publish --snapshot pr-123：CI 上每个 PR 发一个临时版本（如 1.0.0-pr-123-abc），让外部消费者可以验证 PR。Vite / Vitest 等大型 OSS 用得多。
  6. npm provenance（2023+） —— GitHub Actions OIDC 自动签名包来源。消费者能验证"这个 npm 包确实是从这个 git commit 在这个 GitHub Actions workflow 上构建的"。对抗供应链攻击的现代标准。不需付费，开源 + 私有项目都能用。
  7. workspace:* 在 publish 时自动改写 —— pnpm publish / Changesets publish 都把 "@app/utils": "workspace:^" 改写为 "@app/utils": "^1.5.0"（packages/utils 当前版本）。源码 git 仓库保持 workspace: 协议，发到 npm 的 tarball 是改写版。
  8. 私有 registry 选项 —— Verdaccio（开源自建）/ npm Enterprise / GitHub Packages / JFrog Artifactory / GitLab Package Registry 等。.npmrc 中按 scope 路由。

02 底层逻辑

Changesets 完整工作流

典型 monorepo 用 Changesets 的工作流：

Step 1 · 开发阶段：每次 PR 写 changeset

bash 
# 工程师写代码 + 准备发版（在 PR 分支上）
pnpm changeset

# 交互式问答：
# ? Which packages would you like to include? · @app/ui, @app/utils
# ? Which packages should have a major bump? · (空，无 breaking change)
# ? Which packages should have a minor bump? · @app/ui
# ? Which packages should have a patch bump? · @app/utils
# ? Please enter a summary for this change · "Add new Button variants"

# 生成 .changeset/some-random-name.md：
.changeset/some-random-name.md 
---
"@app/ui": minor
"@app/utils": patch
---

Add new Button variants and fix utils edge case

这个 markdown 文件随 PR commit 进 git。多个 PR 可以累积多个 changeset 文件，等到要发版时一起处理。

Step 2 · 合并到 main 后：CI 跑 changeset version

bash 
# CI 自动跑（不是手动）
pnpm changeset version

# Changesets 自动：
# - 读所有 .changeset/*.md 文件
# - 计算每个包的最终版本（多个 minor 取最高）
# - 生成 / 更新每个包的 CHANGELOG.md
# - 修改每个包的 package.json version 字段
# - 删除已处理的 .changeset/*.md（保留 README.md / config.json）

# 此外还会自动联动：
# - 如果 @app/ui patch 升级，依赖 @app/ui 的所有内部包也 patch 升级（关联 bump）

Step 3 · CI 创建"Version Packages" PR

实际由 changesets/action（GitHub Actions 官方）自动完成：

  • Action 在 main 分支跑 changeset version
  • 把所有改动（package.json bumps + CHANGELOGs + 删除的 changeset 文件）打包成一个 PR
  • PR 标题如 Version Packages
  • 这个 PR 由人工 review 后合并——给个最后审视机会

Step 4 · "Version Packages" PR 合并 → CI 触发 publish

bash 
# CI 检测到 main 分支没有 pending changeset（刚被 version 处理掉）
# + package.json 版本号变化 = 该 publish 了

pnpm changeset publish

# Changesets 自动：
# - 检测每个包的当前版本是否已发到 npm
# - 没发的：跑 npm publish（实际是 pnpm publish 或 bun publish）
# - 改写 workspace:* 为真实版本号
# - 给每个版本打 git tag（如 @app/ui@1.5.0）
# - 输出每个包的 publish 结果

整体效果：每个 PR 加 changeset 文件 → 几个 PR 后批量发版（避免每 PR 一发），changelog 自动维护，版本号语义化。

独立 vs 固定版本

Changesets 默认每个包独立 SemVer。某 patch 不要求所有包都 bump。

.changeset/config.json 
{
  "$schema": "https://unpkg.com/@changesets/config@3/schema.json",
  "changelog": "@changesets/changelog-github",
  "commit": false,
  "fixed": [],                           // 默认空，全独立
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch"
}

fixed 模式：

json 
{
  "fixed": [
    ["@my-sdk/*"]                        // 这组包始终保持相同版本
  ]
}

fixed 适合紧耦合的 SDK——一个变化所有包都升。例如 React 的 react / react-dom / react-server-dom-webpack 都同版本（虽然 React 不用 Changesets）。

linked 模式（弱版的 fixed）：

  • 多个包互相联动：A 升 minor，B 也升 minor
  • 但版本号不强制相同（可能 A 1.5.0 / B 2.5.0，下次都升 minor 变 1.6.0 / 2.6.0）
  • 用得少，主要是中间过渡选择

updateInternalDependencies：

  • 当 monorepo 内部 A 是 B 的依赖时，A 升级，B 的依赖范围怎么处理
  • "patch"（默认）：B 的 package.json 中 A 的依赖范围自动 patch（如 ^1.5.0 → ^1.5.1）；如果 A 升 minor，B 也跟着 minor 升级（确保下游知道 A 变了）
  • "minor"：仅当 A 升 minor 以上才更新 B 的范围

prerelease 模式 · 多轮 RC 发布

major 版本发布前通常要多轮 RC 让用户测试。Changesets 支持：

bash 
# 进入预发布阶段（标识符 alpha）
pnpm changeset pre enter alpha

# 此时所有 changeset version + publish 自动加 alpha 后缀
pnpm changeset             # 写新 changeset
pnpm changeset version     # 1.5.0 → 2.0.0-alpha.0
pnpm changeset publish     # 发到 npm 的 alpha tag

# 继续迭代
pnpm changeset
pnpm changeset version     # 2.0.0-alpha.0 → 2.0.0-alpha.1
pnpm changeset publish

# 准备正式发布
pnpm changeset pre exit
pnpm changeset version     # 2.0.0-alpha.1 → 2.0.0
pnpm changeset publish     # 正式 2.0.0 发到 latest tag

npm dist-tag：

  • Changesets prerelease 默认发到 npm 的 alpha / beta / next tag（不是 latest）
  • 用户安装 npm install @app/ui@alpha 拿预发布版
  • 正式发布时切换到 latest tag

snapshot 模式 · 每个 PR 一个临时版本

大型 OSS 项目（Vite / Vitest）常见痛点：用户报 bug，PR 修复了想给用户验证。但 Vite 不能为每个 PR 都发正式版。

snapshot 解：

bash 
# 在 PR CI 上跑（不在 main 上）
pnpm changeset version --snapshot pr-123
pnpm changeset publish --tag pr-123 --no-git-tag

# 自动：
# - 版本号变成 0.0.0-pr-123-<commit-hash>
# - 不加 git tag（snapshot 不留 tag）
# - 发到 npm 的 pr-123 dist-tag（不污染 latest / next）

# 用户验证
npm install @app/vite@pr-123

这是 Vite / Vitest 等项目让"贡献者验证 PR"成为标准流程的方式。详见 Vite 主题。

npm provenance · 供应链签名

2022 年 npm 大范围供应链攻击事件后（如恶意 maintainer 推恶意更新），npm 在 2023 年引入 provenance：

原理：

  1. 包发布时，npm 签发 OIDC token（GitHub Actions / GitLab CI 自动有）
  2. token 包含：repo URL + commit SHA + workflow ID + actor
  3. npm 把 token + 包 manifest 一起签名 + 存到 transparency log（公开可审计）
  4. 用户安装时可以验证 "这个包是从我看到的 GitHub repo 的某 commit 在某 CI 跑出来的"

启用 provenance：

yaml 
# .github/workflows/release.yml
- name: Publish
  run: pnpm changeset publish
  env:
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}

# package.json 加 publishConfig
{
  "publishConfig": {
    "provenance": true              # 关键开关
  }
}

# GitHub Actions 必须有 id-token: write 权限
permissions:
  contents: write
  id-token: write          # ← provenance 必需
  pull-requests: write

用户验证：

bash 
# 装包后验证
npm install @app/ui
npm view @app/ui dist.signatures
npm audit signatures @app/ui

# npm 网页上每个有 provenance 的包都有"Provenance"标签 + 链接到 commit

2026 现状：开源生态广泛采用 provenance（typescript-eslint / Vite / Astro / shadcn 等都开了）。用户开始期待"没有 provenance 的包不可信"。新发布的包应该默认开 provenance。

workspace:* 在 publish 时的处理

monorepo 中 packages/ui 的 package.json 写 "@app/utils": "workspace:^"。如果直接发到 npm，下游用户装时找不到（npm 不知道 workspace: 协议）。

publish 自动改写：

改写前后 
// 源码（git）
{
  "name": "@app/ui",
  "dependencies": {
    "@app/utils": "workspace:^",
    "@app/types": "workspace:*"
  }
}

// pnpm publish / changeset publish 改写后（发到 npm 的 tarball）
{
  "name": "@app/ui",
  "dependencies": {
    "@app/utils": "^1.5.0",            // packages/utils 当前是 1.5.0
    "@app/types": "1.0.0"              // packages/types 当前是 1.0.0（* → 精确）
  }
}

规则：

  • workspace:* → 精确版本（1.5.0）
  • workspace:^ → caret 范围（^1.5.0）
  • workspace:~ → tilde 范围（~1.5.0）

这让"monorepo 内开发用 workspace 协议方便 + publish 时还原成标准范围"成为完整工作流。

publishConfig 字段 · publish 时覆盖配置

有些字段开发时和 publish 时希望不同（详见 P4.2 src-as-main）：

json 
{
  "name": "@app/ui",
  "version": "1.5.0",
  "type": "module",
  "main": "./src/index.ts",
  "types": "./src/index.ts",
  "exports": {
    ".": "./src/index.ts"
  },

  "publishConfig": {
    "main": "./dist/index.js",
    "module": "./dist/index.mjs",
    "types": "./dist/index.d.ts",
    "exports": {
      ".": {
        "types": "./dist/index.d.ts",
        "import": "./dist/index.mjs",
        "require": "./dist/index.js"
      }
    },
    "access": "public",
    "provenance": true
  }
}

publish 时 publishConfig 中的字段覆盖顶层。这让一份 package.json 兼顾 dev（src）和 prod（dist）两个 view。

私有 registry · scope 路由

企业项目常用私有 registry 发内部包。配置原理：.npmrc 按 scope 路由。

.npmrc 
# 默认走 npmjs.org
registry=https://registry.npmjs.org

# @my-org scope 走自家 registry
@my-org:registry=https://npm.my-company.com

# 私有 registry 鉴权
//npm.my-company.com/:_authToken=${INTERNAL_NPM_TOKEN}

# scope 路由（.npmrc 里）
# pnpm publish @my-org/* 自动用 https://npm.my-company.com

主流私有 registry 选项：

  • Verdaccio—— 开源 + 自建（最低成本，单机运行 + npm 协议代理）
  • npm Enterprise—— GitHub 子产品，付费
  • GitHub Packages—— GitHub 自带，对 OSS / GitHub 用户最低门槛
  • JFrog Artifactory—— 大企业全栈 artifact（Docker / Maven / npm 一统）
  • GitLab Package Registry—— GitLab 自带
  • AWS CodeArtifact—— AWS 子服务

选型建议：

  • OSS 项目 + 自动化：不需要私有 registry，发 npmjs.org
  • 小团队 + 内部库：Verdaccio 自建（一台 VM 即可）
  • 已用 GitHub Enterprise：GitHub Packages 自然
  • 大企业 + 已用 JFrog/Artifactory：复用现有基础设施

真实开源案例 · typescript-eslint / Astro / Vite

typescript-eslint（github）：

  • 13 个包的 monorepo，pnpm workspace
  • Changesets + Yarn Berry → 后切 pnpm + Changesets
  • 每周自动发版（cron 跑 changeset version + publish）
  • provenance 已启用

Astro（github）：

  • 30+ 包，pnpm + Turborepo + Changesets
  • snapshot 模式给 PR 临时版本（用户验证 + bug fix 反馈快）
  • core / integration plugin 各自独立 SemVer
  • provenance 已启用（2024+）

Vite / Vitest（github）：

  • Vite + plugin + Vitest 多 monorepo（不在同一 repo）
  • Changesets + pnpm + GitHub Actions
  • 用 changeset bot（自动检测 PR 是否有 changeset 文件）
  • 定期 release week（积累几周再发，避免频繁打扰用户）

这些项目把 Changesets + provenance 工作流做成新项目可直接抄的模板。

💡 关键洞见：发版自动化是 monorepo 成熟度的标志

有些团队 monorepo 维护得很好，但发版仍是"手动 npm publish 每个包"——这种状态会卡在小规模无法扩张。Changesets 把"每个 PR 描述变更 → 批量发版 → changelog 维护 → 版本联动"全套自动化，一旦设好后维护成本几乎为零。2026 年新 monorepo 默认起步就用 Changesets，已是事实标准。

03 应用场景

场景 1 · 新 monorepo 接入 Changesets（从零）

步骤：

bash 
# 1. 装 Changesets
pnpm add -D -w @changesets/cli @changesets/changelog-github

# 2. 初始化（创建 .changeset/ 目录 + config.json）
pnpm changeset init

# 3. 编辑 .changeset/config.json
# 关键字段：
# - changelog: ["@changesets/changelog-github", { "repo": "my-org/my-repo" }]
# - access: "public"
# - baseBranch: "main"

# 4. 写第一个 changeset（让流程跑起来）
pnpm changeset

# 5. 配 GitHub Actions（见下面）

# 6. push + 让 CI 跑一遍验证流程

场景 2 · 给 PR 用 snapshot 让用户验证

背景：开源 React 库收到 bug report，PR 修复了。想给用户测试不发正式版。

方案：

yaml 
# .github/workflows/snapshot.yml（手动触发）
name: Snapshot Release

on:
  workflow_dispatch:
    inputs:
      pr_number:
        required: true

jobs:
  snapshot:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4
        with:
          ref: refs/pull/${{ inputs.pr_number }}/head
      - uses: pnpm/action-setup@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - run: pnpm changeset version --snapshot pr-${{ inputs.pr_number }}
      - run: pnpm changeset publish --tag pr-${{ inputs.pr_number }} --no-git-tag
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}

用户：npm install @app/ui@pr-123 拿到 PR 对应版本。

场景 3 · 把已有 monorepo 切到 provenance

背景：项目用 Changesets 发了几年，没启用 provenance。决定切。

步骤：

  1. 每个要发包的 package.json 加 "publishConfig": { "provenance": true }
  2. GitHub Actions workflow 加 permissions: { id-token: write }
  3. 确认 npm token 是"Automation"类型（不是 publish/grant）—— provenance 要求 Automation token
  4. 下次 release 时 npm 上每个包自动有 Provenance 标记

整个迁移 30 分钟，向后兼容（旧版本不受影响，新版本带 provenance）。

04 典型代码

完整 .changeset/config.json 
{
  "$schema": "https://unpkg.com/@changesets/config@3/schema.json",
  "changelog": [
    "@changesets/changelog-github",
    { "repo": "my-org/my-monorepo" }
  ],
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": [
    "@app/web",
    "@app/web-e2e"
  ]
}

ignore：列出不发布的包（apps 通常不发；只有 lib 包发）。但注意：apps 也要有 "private": true 才更稳。

完整 GitHub Actions release.yml 
name: Release

on:
  push:
    branches: [main]

concurrency: ${{ github.workflow }}-${{ github.ref }}

jobs:
  release:
    name: Release
    runs-on: ubuntu-latest
    permissions:
      contents: write          # 写 git tag + commit
      pull-requests: write     # 创建 Version Packages PR
      id-token: write          # provenance 必需

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0       # Changesets 需要完整 git 历史

      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'

      - run: pnpm install --frozen-lockfile

      - run: pnpm build         # 先 build 所有 lib

      - name: Create Release Pull Request or Publish to npm
        id: changesets
        uses: changesets/action@v1
        with:
          # 当 main 分支没有 pending changesets 时跑这条
          publish: pnpm changeset publish
          # 否则创建 Version Packages PR
          version: pnpm changeset version
          commit: 'chore: version packages'
          title: 'Version Packages'
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}

05 常见陷阱

陷阱 1 · 忘记给 apps 加 private: true

Changesets 不区分 lib 和 app（除非配 ignore）。如果 apps/web/package.json 没写 "private": true，changeset publish 可能尝试发它。修复：所有 app 类项目必须 "private": true；只有真要发的 lib 包不加。

陷阱 2 · provenance 失败因为 token 类型错

npm provenance 要求 Automation token（在 npm 的 Access Tokens 页面创建时选 "Automation"），不是 publish 类型 token。token 错时 provenance 静默失败但 publish 仍 OK。修复：检查 token 类型；可以用 npm whoami --registry=https://registry.npmjs.org/ 排查。

陷阱 3 · 漏写 fetch-depth: 0

GitHub Actions 默认浅 clone（fetch-depth: 1）。Changesets 需要完整 git 历史比对——浅 clone 时 changeset version 可能错判 baseBranch。修复：checkout step 加 fetch-depth: 0。

陷阱 4 · 把所有变化都写 changeset

每次 PR 都写 changeset 显得很勤奋，但非用户可见的变化（重构 / 内部 refactor / 测试）不应写——它们不该 bump 版本。修复：只对"影响用户的"变化写 changeset；refactor / docs / test 不写。Changesets bot 可以提醒"这个 PR 没 changeset，是不是无 user-facing change？"。

陷阱 5 · publishConfig 写错路径

publishConfig.main 指向不存在的 dist 路径，publish 时不报错（npm 不验证），但下游用户安装后立即报 ERR_MODULE_NOT_FOUND。修复：CI 在 publish 前跑 publint（package.json lint）+ arethetypeswrong（双产物 + types 验证）。
