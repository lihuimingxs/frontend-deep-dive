编写约定

每章遵循统一的 6 段结构：速览 → 底层逻辑 → 应用场景 → 典型代码 → 常见陷阱 → 延伸阅读。本主题相比其他主题有以下额外约定：

  • 「短重述 + 链回」 —— Bun install 内部、Vite dev server workspace 利用、Node module resolution spec 都在 Bun / Vite / Node 主题；本主题用 1-3 句话点到本质，紧跟一句"详细的 X 请参考 Y"
  • API / 配置定义优先官方文档 —— pnpm.io / yarnpkg.com / docs.npmjs.com / turbo.build/repo/docs 是表面规则的源头
  • 设计决策以 RFC + 讨论为准 —— 严格 peer / Yarn PnP / catalog 协议都有公开 RFC；写"为什么这么设计"必须引 RFC，不能臆测
  • 历史以原始公告为准 —— Yarn classic 2016 公告、npm v3 扁平化 RFC、pnpm 1.0 announcement、Lerna 项目交接给 Nx 的公告——这些是历史叙事的源头
  • 性能数字必标条件 —— 包管理器宣传的"X 比 Y 快 N 倍"需要复现条件（项目规模、cache 状态、网络条件）；引用时必带 source + 复现说明，否则不写

与既有主题的关系

本主题不重复 Bun / Vite / Node / ECMA 等主题已经讲透的底层机制，但会在每个交叉点提供"短重述 + 链接"。常见对应：

  • P1.2 同代人 + P3.1 CAS 对比 + P4.1 catalog → Bun P4.1 Bun install（二进制 lockfile + Bun workspace 详细）
  • P2.1 Node module resolution → Node 主题（CJS / ESM resolution 算法源头）
  • P2.1 ESM resolution + import attributes → ECMA P7.4 模块系统（spec 层语义）
  • P2.2 双 package hazard / interop → Vite P3.2 CJS Interop
  • P4.2 内部包 hot reload → Vite P2.1 Dev mode（dev server 监听 workspace 变化）
  • P4.2 TypeScript path mapping → TypeScript（path mapping vs workspace symlink 边界）
