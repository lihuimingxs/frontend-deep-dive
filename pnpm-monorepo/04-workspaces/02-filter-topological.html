--filter / topological / 内部联调

上章讲了 workspace 协议（怎么声明）。本章讲怎么使用——pnpm --filter 选择器全谱、topological run 自动按依赖顺序执行、内部包 hot reload 联调、TypeScript path mapping 与 workspace symlink 的协作边界。这些是 monorepo 日常 dev 的核心操作。

阅读时长 ~25 分钟 前置 P4.1（workspace 协议） 主要来源 pnpm.io filter docs / Vite docs / TypeScript handbook

01 速览

  1. pnpm --filter 是 monorepo 操作的核心 —— 选哪些 workspace 包参与某次操作。支持 4 类选择器：(1) 按包名（@app/ui / '@app/*'）；(2) 按路径（'./apps/**'）；(3) 按依赖图前后向（...{packages/ui}^...）；(4) 按 git 变更（'[main]' 表示自 main 以来变动的包）。
  2. topological run 自动按依赖顺序执行 —— pnpm -r run build 让 packages/ui（被依赖）先 build，然后 apps/web（依赖 ui）。--parallel 取消依赖排序（仅在确定无依赖时用）；--workspace-concurrency=N 限制并行度。
  3. 内部包 hot reload 取决于 dev 工具 —— Vite / Webpack-dev-server 默认不监听 node_modules。pnpm symlink 让 packages/ui 出现在 node_modules/.pnpm 下，直接被忽略。需要配置 watcher 例外，或用 tsc --watch + Vite 监听 dist 二段流程。
  4. TypeScript path mapping vs workspace symlink —— path mapping（"paths": { "@app/ui": ["packages/ui/src"] }）是编辑期的别名（让 IDE / tsc 跳到源码）；workspace symlink 是运行期的物理引用。两者不冲突，但容易混淆。2026 推荐：用 workspace + package.json exports，少用 path mapping。
  5. --filter 选择器是组合表达式 —— ...{...}^... 中三个 dot 各代表一种依赖图遍历，可叠加。学会几个组合后能精确表达"我想跑哪些包"。
  6. monorepo 命令的两个层级 —— 根级（pnpm install、pnpm -r run lint 跑所有 workspace）；子级（pnpm --filter @app/web add lodash 在某个 workspace 里加依赖）。混淆这两层是新手常见错误。
  7. packageManager 字段防止团队工具不一致 —— package.json 顶层 "packageManager": "pnpm@10.12.0" 让 corepack 自动用对应版本；防止"我装的包跟同事的不一样"。

02 底层逻辑

--filter 选择器全谱

pnpm 的 filter 是一套小型 DSL，4 种基本形式 + 组合操作符。

(1) 按包名：

bash 
# 单个包
pnpm --filter @app/web build

# glob 匹配
pnpm --filter '@app/*' build              # 所有 @app/ 命名空间下
pnpm --filter '*-utils' test              # 所有以 -utils 结尾

# 排除
pnpm --filter '!@app/legacy' lint         # 除 @app/legacy 外

(2) 按路径：

bash 
# 显式路径
pnpm --filter './packages/ui' build

# glob 路径
pnpm --filter './apps/**' build           # apps 下所有
pnpm --filter './packages/**' lint        # packages 下所有

(3) 按依赖图遍历（pnpm 独家强大）：

bash 
# 包本身 + 它的依赖（递归）
pnpm --filter '@app/web...' build
# 选中 @app/web 和它直接/间接依赖的所有 workspace 包
# (依赖 packages/ui, packages/utils → 都被选中)

# 包本身 + 它的依赖者（递归）
pnpm --filter '...@app/ui' build
# 选中 @app/ui 和所有依赖它的 workspace 包
# (apps/web 依赖 ui → web 被选中)

# 包本身 + 直接依赖（不递归）
pnpm --filter '@app/web^...' build
pnpm --filter '...^@app/ui' build

# 仅依赖（不含本身）
pnpm --filter '@app/web^...' --no-workspace build

这套依赖图遍历的语法记忆要点：

  • ... 在前 → 上游（dependencies）
  • ... 在后 → 下游（dependents）
  • ^ 限制为直接关系（不递归）

(4) 按 git 变更（CI 极有用）：

bash 
# 自 main 分支以来变动过的 workspace 包
pnpm --filter '[main]' build

# 自某个 commit 以来变动
pnpm --filter '[abc123]' build

# 包含被影响的下游
pnpm --filter '...[main]' build
# 选中 main 以来变化的包 + 所有依赖它们的下游包
# (CI 的 affected 模式典型实现)

组合：交集 / 并集：

bash 
# 多个 filter 是"交集"
pnpm --filter '@app/*' --filter '[main]' build
# = 在 @app/ 命名空间下、且自 main 以来变动的包

# 表示并集要分别跑或用 glob
pnpm --filter '{@app/web,@app/admin}' build

topological run · 自动按依赖顺序

monorepo 中跑 build：packages/ui 必须先 build（产 dist），apps/web 才能用。手动顺序写脚本既繁琐又易错。pnpm 提供自动 topological 排序：

bash 
# 跑所有 workspace 的 build
pnpm -r run build
# pnpm 自动分析依赖图：
#   1. 没有 workspace 内依赖的（packages/utils, packages/config）先并行 build
#   2. 等它们完成，packages/ui（依赖 utils）再 build
#   3. 等它完成，apps/web（依赖 ui）再 build
# 全程自动 + 各层最大并行

关键参数：

--parallel：

  • 取消依赖排序，所有包并行跑
  • 仅当任务不依赖其他包的输出时使用（如 lint / test）
  • 对 build 用 --parallel 通常错（依赖关系会被违反）

--workspace-concurrency=N：

  • 限制同时跑的 workspace 数量
  • 默认是 CPU 核数
  • 大 monorepo 下可能要降低（每个进程吃内存）

--aggregate-output：

  • 把所有 workspace 的 stdout 收集起来分批输出（按包分组）
  • 避免日志交错混乱

--bail / --no-bail：

  • --bail（默认）：第一个失败就停
  • --no-bail：继续跑后面的（用于 CI 想看所有失败）

实测对比（中型 monorepo, 30 包）：

命令 耗时 说明
原生顺序 pnpm -r run build ~45s 自动 topological + 各层并行
错用 --parallel ~25s + 失败 下游包看不到上游 dist
手写 sequential（每包一条命令） ~120s 没有任何并行

内部包 hot reload 联调

monorepo 中典型场景：apps/web 用 import { Button } from '@app/ui'。改 packages/ui 的源码，期望 apps/web 的 Vite dev server 自动 reload。

问题 1：Vite 默认不监听 node_modules

Vite 启动时把所有 dependencies 视为"已构建产物，不变化"，做 dependency pre-bundling 缓存。watcher 默认 ignored: ['**/node_modules/**']。

但 pnpm 把 packages/ui symlink 到 node_modules/.pnpm/@app/ui@.../node_modules/@app/ui——修改的物理文件路径在 node_modules 内，被 Vite 忽略。详见 Vite P2.1。

解法 1 · Vite 配置不忽略 workspace 包：

vite.config.ts 
import { defineConfig } from 'vite';

export default defineConfig({
  // 让 Vite 跟随 symlink 到真实文件，监听真实路径
  resolve: {
    preserveSymlinks: false   // 默认就是 false
  },
  server: {
    watch: {
      // 不忽略 .pnpm 中的 workspace 包
      ignored: ['!**/node_modules/.pnpm/@app+*/**']
    }
  },
  // 让 pre-bundle 不缓存 workspace 包（永远走 dev transform）
  optimizeDeps: {
    exclude: ['@app/ui', '@app/utils']
  }
});

解法 2 · 双进程：tsc --watch + Vite 监听 dist（兜底）：

bash 
# Terminal 1: packages/ui 持续编译到 dist/
cd packages/ui && pnpm tsc --watch

# Terminal 2: apps/web 跑 Vite dev
cd apps/web && pnpm dev

# packages/ui/package.json 的 main 指向 dist/index.js
# Vite 看到 dist 文件变化（虽然在 .pnpm 中）触发 HMR
# 这是更"传统"的方案，对 Vite 配置不熟时易上手

解法 3 · turbo / nx 的 dev mode（详见 P5）：

  • Turborepo 的 turbo dev 自动并发跑所有相关 dev 任务 + 监听 cross-package 变化
  • Nx 类似
  • 把"workspace 联调"做成一等公民

TypeScript path mapping vs workspace symlink

这是 monorepo 中最让人困惑的两个概念。

workspace symlink：

  • 运行期机制：node_modules/@app/ui 是 symlink，Node resolution / bundler / runtime 都看真实文件
  • 由包管理器（pnpm/npm/Yarn/Bun）管理
  • 用 workspace:* 协议触发

TypeScript path mapping：

  • 编辑期 + tsc 编译期机制：tsconfig.json 的 paths 让 import { foo } from '@app/ui' 解析到指定路径
  • 由 TypeScript 管理（IDE 跳转、类型检查）
  • 不影响运行期（除非用 tsc-paths 或 bundler 配套）
tsconfig.json（path mapping 例子） 
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@app/ui": ["./packages/ui/src"],
      "@app/ui/*": ["./packages/ui/src/*"],
      "@app/utils": ["./packages/utils/src"]
    }
  }
}

这个配置让 IDE / tsc 把 @app/ui 解析为源码目录（packages/ui/src），不经过 dist build。

问题：path mapping 与 workspace symlink同时存在时容易出错：

  • 编辑期 IDE 用 path mapping（看到 src 源码 → 类型最准）
  • 运行期用 symlink → packages/ui/dist/index.js
  • 两边解析到不同路径——可能 IDE 报无错但 build 失败（src 有但 dist 还没 build）

2026 推荐路线：

  1. 不用 path mapping，让 workspace symlink 一统
  2. 子包 package.json 配 exports 字段把 src 暴露作为 dev "main"：
    json 
    // packages/ui/package.json
{
  "name": "@app/ui",
  "type": "module",
  "main": "./src/index.ts",        // ← 直接指向 src
  "exports": {
    ".": {
      "types": "./src/index.ts",
      "default": "./src/index.ts"
    }
  }
}
  3. 用 Vite / Bun 这种"原生 TS 直跑"的工具——它们能直接读 .ts，不需要 dist build
  4. publish 时再生成真正的 dist + 改 exports

这是Vite + pnpm + workspace 模式的现代写法——单一来源（src），dev / publish 都靠 exports 切换。

monorepo 命令的两个层级

新手最容易混淆的：根级命令 vs 子级命令。

根级（在 monorepo 根目录跑）：

bash 
# 装所有 workspace 的依赖（一次性）
pnpm install

# 在所有 workspace 跑某个 script
pnpm -r run build
pnpm -r run test

# 加根级 dev 依赖（不属于某个具体 workspace）
pnpm add -D -w typescript          # -w 表示 workspace root

子级（在某个 workspace 子目录跑 / 用 --filter）：

bash 
# 方式 1：cd 进子目录
cd packages/ui
pnpm add lodash                   # lodash 加到 packages/ui/package.json

# 方式 2：根目录 + --filter
pnpm --filter @app/ui add lodash

# 在 @app/web 跑 dev
pnpm --filter @app/web dev

关键陷阱：

  • 在 monorepo 根直接跑 pnpm add lodash（不带 -w） → pnpm 报错或装到根（看 workspace 配置）
  • 错误地把项目 dep 装到根，子包看不到
  • 建议团队内规约："根级只放 dev 工具（typescript / eslint / prettier），项目代码 dep 一律装到子 workspace"

packageManager 字段 · 强制工具一致性

monorepo 团队最痛的问题：A 用 pnpm 9.5 装的 lockfile，B 用 pnpm 10 装出现 diff，merge 冲突。解：在根 package.json 写：

json 
{
  "name": "my-monorepo",
  "packageManager": "pnpm@10.12.0",
  "engines": {
    "node": ">=22.0.0"
  }
}

配合 corepack（Node 16+ 自带）：

bash 
# 启用 corepack（一次性，全局）
corepack enable

# 之后在 monorepo 目录跑 pnpm 命令
pnpm install
# corepack 自动检查 packageManager 字段
# 如果当前 pnpm 版本不匹配，下载对应版本并使用
# 团队所有人 + CI 确保用同一版本

npm 9+ / Yarn 4 都支持 packageManager 字段。pnpm 10+ 在 install 时会主动检查并提示。这是 2024-2026 monorepo 的新标准实践。

💡 关键洞见：filter 表达式是 monorepo 的"jq"

初学时 ...{...}^... 看着奇怪，但学会后你会发现 monorepo 操作能精确表达。"我想跑被这次 PR 影响到的所有包，包括下游" 用 pnpm --filter '...[main]' run test 一行写完。这是 pnpm 在 monorepo 操作灵活性上的核心优势——npm/Yarn 都没这套表达力。

03 应用场景

场景 1 · CI 上只跑被改影响的包

背景：monorepo 50 个包，每次 PR 跑全部测试 + lint 要 15 分钟。想缩短到只跑被改影响的部分。

方案：

yaml 
# .github/workflows/ci.yml
- name: Run tests on affected
  run: pnpm --filter '...[origin/main]' run test
  # ... 表示包括下游（如果某个包变化，所有依赖它的下游也要测）

- name: Run lint on changed
  run: pnpm --filter '[origin/main]' run lint
  # 不带 ... 表示只 lint 直接变化的包

典型 PR 只改了 2-3 个包及其下游（5-10 个），CI 时间从 15 分钟降到 3-5 分钟。

场景 2 · workspace 内部包 hot reload 不触发

背景：apps/web 用 Vite，import packages/ui 的 Button。改 packages/ui/src/Button.tsx，Vite dev server 没反应。

诊断 + 修复：见上文"内部包 hot reload 联调"。3 个方案任选：

  • vite.config 改 server.watch + optimizeDeps.exclude
  • 双进程：tsc --watch + Vite
  • 用 Turborepo 的 turbo dev 一统

更进一步：让 packages/ui 的 package.json main 直接指 src/.ts（详见 path mapping 章节）。这是 Vite 最自然的方式——根本不需要 build 步骤，dev 时直接读源码。

场景 3 · 给 monorepo 引入 packageManager 字段

背景：团队反映"我装出来跟你的不一样"。决定强制 pnpm 版本。

方案：

  1. 团队达成共识：用 pnpm 10.x 当前最新版
  2. 根 package.json 加 "packageManager": "pnpm@10.12.0"
  3. 团队所有人 corepack enable
  4. 清空各自机器的 node_modules + pnpm-lock.yaml
  5. 统一一个人重新 pnpm install 生成新 lockfile，commit
  6. 其他人 pull 后 pnpm install 即可（corepack 自动用对应版本）
  7. CI workflow 用 actions/setup-node 后 corepack enable

之后 lockfile 不再因为 pnpm 版本差异而 diff。

  "private": true,

  "engines": {
    "node": ">=22.0.0"
  },
  "scripts": {
    "dev": "pnpm --filter @app/web dev",
    "build": "pnpm -r run build",
    "test": "pnpm -r run test",
    "lint": "pnpm -r run lint",
    "typecheck": "pnpm -r run typecheck",

Vite + pnpm workspace hot reload 完整配置 
// apps/web/vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  resolve: {
    preserveSymlinks: false       // 跟随 symlink 到真实路径

    watch: {

});

  "type": "module",

  "module": "./src/index.ts",
  "types": "./src/index.ts",

      "types": "./src/index.ts",
      "default": "./src/index.ts"

    "build": "tsc -p tsconfig.build.json",
    "test": "vitest"

    "react": "catalog:react18"

05 常见陷阱

陷阱 1 · 在根目录直接 pnpm add 而不带 -w

新手常在 monorepo 根 pnpm add lodash，pnpm 9+ 默认报错（必须显式指定 -w 或子 workspace）。修复：

  • pnpm add -w lodash（装到根 package.json）
  • pnpm --filter @app/web add lodash（装到子 workspace）
陷阱 2 · 用 --parallel 跑 build

看到 --parallel 觉得"加快速度"，对 build 用了。结果某些包 fail 因为依赖的下游还没 build 完。修复：build 用默认 topological 排序；只对真正独立的任务（lint / test / typecheck 单包内）用 --parallel。

陷阱 3 · TypeScript path mapping 与 workspace 双声明

同时配 tsconfig paths + workspace:*。编辑器 IDE 用 paths 解析（看 src），构建用 workspace（看 dist）。结果：dev 时 IDE 显示一切正常但 prod build 失败（dist 没 build）。修复：选一种——推荐去掉 path mapping，让 workspace + exports 一统。

陷阱 4 · 忘了 src-as-main 模式 publish 时改回 dist

用 src-as-main 在 dev 极方便，但publish 时不能直接发 src（用户拿不到编译后的 .js）。修复：用 Changesets / pnpm 自定义 publish hook 在 publish 时改 main / exports 指向 dist；或用publishConfig 字段（npm 标准）覆盖 publish 时的字段：

json 
{
  "main": "./src/index.ts",
  "publishConfig": {
    "main": "./dist/index.js",
    "module": "./dist/index.mjs",
    "types": "./dist/index.d.ts"
  }
}
陷阱 5 · 用 pnpm exec 而非 pnpm run

pnpm -r exec eslint . vs pnpm -r run lint —— 前者跑外部命令（每个包跑 eslint），后者跑 package.json 的 scripts。混淆容易导致命令找不到。建议：日常用 run + 在 scripts 里定义命令，统一管理；只在临时一次性命令用 exec。
