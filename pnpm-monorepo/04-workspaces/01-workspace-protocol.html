4 家 workspace 对比 + catalog

monorepo 不是把代码塞一个目录就成立——它需要一套协议描述"内部多个包如何互相引用 + 如何统一升级 + 如何选择性 publish"。本章对比 4 家（npm / Yarn / pnpm / Bun）的 workspace 配置入口、解析协议（workspace:* / link: / portal:），重点讲 2024-2025 的 catalog 协议——第一次让 workspace 在多家工具间互通的共识。

阅读时长 ~25 分钟 前置 P3 全部（pnpm 内部） 主要来源 pnpm.io / yarnpkg.com / docs.npmjs.com / bun.com/docs 4 家 workspace 文档

01 速览

  1. 4 家 workspace 配置入口 —— npm / Yarn classic / Yarn Berry：在根 package.json 加 "workspaces" 字段；pnpm：单独 pnpm-workspace.yaml 文件；Bun：根 package.json 的 "workspaces"（兼容 npm 写法）。配置语法都用 glob，但语义略有差异。
  2. workspace 内部依赖用 workspace:* 协议 —— 写 "@app/ui": "workspace:*" 表示"这是 monorepo 内部包，按当前版本链接"。workspace:^ / workspace:~ / workspace:./packages/ui 是变体。publish 时自动改写为真实版本号。
  3. link: vs portal: vs workspace: —— link: 是开发期纯 symlink，不在 lockfile（旧时代 monorepo 用法）；portal: 是 Yarn 特有，类似 link 但解决一致性问题；workspace: 是2026 推荐——记入 lockfile + publish 时改写。
  4. catalog 协议（2024-08 pnpm 9.5 引入） —— monorepo 顶层定义版本字典：catalog: { react: ^18.3.1 }；子包用 "react": "catalog:" 引用。升级 React 改一处，所有包同步。Bun 1.1.42 / Yarn 4 RFC 已跟进，事实上的"workspace 协议跨工具兼容化"。
  5. 命名 catalog —— 同一 monorepo 多版本并存（catalog:react18 和 catalog:react19 各管各的）。比"全部统一"灵活。
  6. workspace 内部 publish 时的处理 —— pnpm/Bun/Changesets 都会在 pnpm publish / bun publish / changeset publish 时自动把 workspace:* 改写为真实版本号，发到 npm 的 tarball 不带 workspace: 协议。这让"monorepo 内开发 + 单独 publish"工作流自然。
  7. 4 家在严格性 / 灵活性 / 生态成熟度上各有侧重 —— pnpm 最严格 + 最成熟的 workspace 实现；Yarn Berry 最丰富的协议（portal / patch / git 都有）；npm workspace 最基础但兼容性最高；Bun 是 npm 兼容 + 速度优势。

02 底层逻辑

workspace 配置入口对比

npm（v7+）/ Yarn classic / Yarn Berry / Bun：

根 package.json 
{
  "name": "my-monorepo",
  "private": true,
  "workspaces": [
    "apps/*",
    "packages/*"
  ]
}

简单形式（数组 of glob）。npm/Yarn/Bun 都用这个写法。

Yarn Berry 扩展形式（支持 nohoist）：

json 
{
  "workspaces": {
    "packages": ["apps/*", "packages/*"],
    "nohoist": ["**/react-native"]
  }
}

pnpm（独立文件）：

pnpm-workspace.yaml 
packages:
  - 'apps/*'
  - 'packages/*'
  - '!**/test/**'         # 排除（pnpm 特性）

# pnpm 9+ 还能在这里定义 catalog（详见下面）
catalog:
  react: ^18.3.1
  react-dom: ^18.3.1

为什么 pnpm 用独立文件：

  • pnpm 的 monorepo 配置（packages / catalog / pkg-overrides 等）逐渐变多，塞进 package.json 会乱
  • 独立文件更清晰，与 npm/Yarn 的"在 package.json 配置"路线分道扬镳
  • YAML 比 JSON 注释友好

结果是 pnpm 的 monorepo 配置一目了然，npm/Yarn 把所有东西塞 package.json 后期会膨胀。

workspace:* 协议详解

monorepo 中 apps/web 依赖 packages/ui。怎么写？

三种写法：

(1) 古老方式：相对路径或 file:

json 
// apps/web/package.json
{
  "dependencies": {
    "@app/ui": "file:../../packages/ui"
  }
}
  • 各家都支持但不在 monorepo 中常用——它是外部包的本地化引用，不是 workspace 概念
  • 缺点：publish 时不会被改写，发到 npm 的包仍带 file: 路径，下游用户装不上

(2) 旧 monorepo 方式：直接版本号

json 
// apps/web/package.json
{
  "dependencies": {
    "@app/ui": "1.0.0"
  }
}
  • 包管理器看到 @app/ui 是 workspace 内部包，自动 symlink 而不是从 npm 拉
  • 但版本号要手动跟 packages/ui/package.json 同步——容易漂移
  • 2018 年前的 Lerna monorepo 常见

(3) 现代方式：workspace:* 协议（Yarn Berry 引入，pnpm/Bun 跟进）

json 
// apps/web/package.json
{
  "dependencies": {
    "@app/ui": "workspace:*"
  }
}
  • workspace:*：跟随 packages/ui 当前版本（写啥都行，反正 workspace 内）
  • workspace:^：跟随当前主版本（packages/ui 是 1.5.0 → ^1.5.0）
  • workspace:~：跟随当前次版本（packages/ui 是 1.5.3 → ~1.5.3）
  • workspace:./packages/ui：显式路径（罕见，覆盖默认按 name 解析）
  • publish 时自动改写（下面详解）

支持矩阵：

协议 npm Yarn classic Yarn Berry pnpm Bun
workspace:* v9+ ❌ ✅（首发） ✅ ✅
workspace:^ v9+ ❌ ✅ ✅ ✅
workspace:~ v9+ ❌ ✅ ✅ ✅
link: ✅ ✅ ⚠️ 弱化 ✅ ✅
portal: ❌ ❌ ✅（独有） ❌ ❌

publish 时自动改写：从 workspace: 到真实版本号

monorepo 中 apps/web 的 package.json 写 "@app/ui": "workspace:*"。如果直接 publish，下游用户装时找不到 @app/ui（npm registry 没有 workspace: 协议的概念）。

所以包管理器在 publish 时自动改写：

改写前后对比 
// 源码（git 中）
{
  "dependencies": {
    "@app/ui": "workspace:*",
    "@app/utils": "workspace:^"
  }
}

// publish 时自动改写（packages/ui 当前是 2.3.1，packages/utils 当前是 1.5.0）
{
  "dependencies": {
    "@app/ui": "2.3.1",                    // workspace:* → 当前确切版本
    "@app/utils": "^1.5.0"                 // workspace:^ → ^当前主版本
  }
}

// 这个改写后的 package.json 才是发到 npm 的版本
// 但 git 仓库中的 package.json 仍保持 workspace: 协议

谁来改写：

  • pnpm publish：自动
  • bun publish：自动
  • Changesets publish：自动（详见 P6.1）
  • npm publish：v9+ 支持 workspace: 但改写行为略有不同

关键洞见：workspace: 协议是 monorepo 的"开发期占位符"——dev 时让 symlink 即时生效，publish 时还原成真实版本。这分离了"开发期方便"和"发布期严格"两个需求。

catalog 协议 · 2025 共识

monorepo 长期痛点：50 个包都用 React 18.3.1，每次升级要改 50 处 package.json。社区 lint 工具（syncpack）治标——版本写错了能检测，但不能从源头统一。

2024 年 8 月 pnpm 9.5 引入 catalog 协议。在 monorepo 顶层定义版本字典：

pnpm-workspace.yaml（pnpm） 
packages:
  - 'apps/*'
  - 'packages/*'

catalog:
  react: ^18.3.1
  react-dom: ^18.3.1
  typescript: ^5.6.0
  '@types/node': ^22.0.0

# 命名 catalog（高级用法）
catalogs:
  react18:
    react: ^18.3.1
    react-dom: ^18.3.1
  react19:
    react: ^19.0.0
    react-dom: ^19.0.0

子包用 catalog: 引用：

packages/ui/package.json 
{
  "name": "@app/ui",
  "dependencies": {
    "react": "catalog:",                 // 默认 catalog
    "react-dom": "catalog:",
    "typescript": "catalog:"
  }
}

// 用命名 catalog（跑 React 19 的 app）
// packages/ui-experimental/package.json
{
  "dependencies": {
    "react": "catalog:react19",
    "react-dom": "catalog:react19"
  }
}

升级 React 18.3.1 → 18.3.2 只改一处（pnpm-workspace.yaml 顶层），所有用 catalog: 的子包自动跟随。

publish 时改写（与 workspace: 类似）：

json 
// 源码
{ "dependencies": { "react": "catalog:" } }

// publish 改写
{ "dependencies": { "react": "^18.3.1" } }     // 用顶层 catalog 中的范围

Bun 1.1.42（2024-12）跟进：

package.json（根，Bun 写法） 
{
  "workspaces": ["apps/*", "packages/*"],
  "catalog": {
    "react": "^18.3.1"
  },
  "catalogs": {
    "react19": { "react": "^19.0.0" }
  }
}

Bun 选择把 catalog 放 package.json（与 npm 工作流一致），pnpm 放 pnpm-workspace.yaml（与 pnpm 风格一致）。但子包的引用语法两家都用 "react": "catalog:"——这就是跨工具兼容的关键。

Yarn 4 RFC 阶段（2024-2025）：

  • RFC #570 讨论 catalog 协议
  • API 设计可能与 pnpm/Bun 兼容
  • 2026 年正式支持时间未定

npm 11：截至 2026-05 仍未跟进 catalog。这是 monorepo 选 pnpm/Bun 的具体理由之一。

命名 catalog 的用途

monorepo 内不同子包用不同主版本的同一包是常见需求：

  • 主 app 用 React 18，实验 app 用 React 19
  • 新组件库用 TypeScript 5.6，老 lib 兼容 5.0
  • 不同 testing 框架共存（Vitest 1.x 和 Vitest 2.x 渐进迁移）

命名 catalog 让多版本各管各：

pnpm-workspace.yaml 
catalogs:
  react18:
    react: ^18.3.1
    react-dom: ^18.3.1
  react19:
    react: ^19.0.0
    react-dom: ^19.0.0

子包按需引用：

json 
// apps/main-app/package.json
{ "dependencies": { "react": "catalog:react18" } }

// apps/experimental/package.json
{ "dependencies": { "react": "catalog:react19" } }

升级实验 app 时只改 react19 catalog，不影响主 app。这比"全部统一一个版本"灵活得多。

link: 协议 · 与 workspace: 的微妙差异

link: 是纯 symlink，不进 lockfile。它的设计意图是"我有一个本地目录想直接接入项目"。

json 
{
  "dependencies": {
    "some-pkg": "link:../some-other-project"
  }
}

vs workspace:*：

  • workspace:* 在同一 monorepo（受 packages glob 包含）；link: 可以指任何路径
  • workspace:* 进 lockfile + publish 改写；link: 不进 lockfile + publish 时报错（"包含 link: 不能 publish"）
  • workspace:* 是 monorepo 用法；link: 是"开发期临时接入外部项目"用法

典型 link: 场景：

  • 调试一个外部 npm 包：clone 它的源码到 ../some-pkg-source，在自己项目用 link:../some-pkg-source
  • 本地开发跨 monorepo 联调（"我的 app monorepo + 我的 lib monorepo 互相调用"）
  • Yarn Berry 的 portal: 是 link: 的"严格版"，校验依赖一致

4 家 workspace 的"个性"

npm workspace：

  • v7+ 支持，v9+ 支持 workspace:* 协议
  • 没有 catalog（落后一代）
  • filter 简单（--workspace=<name>）
  • 主要场景：从 npm 单 repo 渐进迁移到 monorepo，最低门槛

Yarn classic workspace：

  • v1.x 的 workspace 设计简单，不支持 workspace: 协议（要 v2+）
  • 主流 Yarn classic monorepo 在 2026 年已迁移到 pnpm 或 Yarn Berry

Yarn Berry workspace：

  • workspace: 协议的发明者
  • 额外协议：portal: / patch: / git: / patch://...
  • nohoist 字段（PnP 模式不太用）
  • catalog 在 RFC，2026 实验

pnpm workspace：

  • pnpm-workspace.yaml 独立文件，配置量大时清晰
  • workspace: 协议完整支持
  • catalog 协议主导者（2024-08 引入）
  • filter 强大（--filter '...{packages/ui}^...'）
  • 主流大型 monorepo 选择

Bun workspace：

  • npm 兼容写法（package.json 中 workspaces 数组）
  • workspace: 协议支持（Bun 1.0+）
  • catalog 跟进（Bun 1.1.42+）
  • 速度优势（详见 Bun P4.1）
  • 新项目 + 已用 Bun runtime 的场景首选
💡 关键洞见：catalog 是 2026 monorepo 的分水岭

选 monorepo 工具时，支不支持 catalog 是日常生产力的关键。它的影响比"装包速度差几秒"大得多——升级 React/TypeScript/Vitest 等核心依赖的痛苦差距是改一处 vs 改 50 处。这是 npm 11 / Yarn 4 默认在 2026 年仍落后一代的具体表现。

03 应用场景

场景 1 · 把老 monorepo 迁移到 catalog

背景：5 年的 pnpm monorepo，30 个包，每个包的 package.json 单独写 react 版本。已经被 react@18 升级时改 30 处折腾过几次。决定迁移 catalog。

步骤：

  1. 确认 pnpm 版本 ≥ 9.5（pnpm -v）
  2. 编辑 pnpm-workspace.yaml 顶层加 catalog 字段
  3. 对每个核心依赖（react / react-dom / typescript / @types/node 等）：在 catalog 里写当前版本范围
  4. 子包 package.json 把 "react": "^18.3.1" 改成 "react": "catalog:"
  5. 跑 pnpm install，确认 lockfile 没变化（应该等价）
  6. 跑全套测试 + build 确认没问题
  7. commit + 教育团队"以后升级核心依赖只改 catalog"

30 个包的迁移通常 1-2 小时（grep + sed 自动化）。一次投入，长期收益。

场景 2 · monorepo 里同时支持 React 18 和 React 19

背景：主线 app 跑生产 React 18，新实验 app 想用 React 19 的新特性。怎么共存？

方案：用命名 catalog

pnpm-workspace.yaml 
catalogs:
  react18:
    react: ^18.3.1
    react-dom: ^18.3.1
  react19:
    react: ^19.0.0
    react-dom: ^19.0.0
json 
// apps/main/package.json
{ "dependencies": { "react": "catalog:react18", "react-dom": "catalog:react18" } }

// apps/experimental/package.json
{ "dependencies": { "react": "catalog:react19", "react-dom": "catalog:react19" } }

// packages/ui/package.json （内部库，跑双版本）
{
  "peerDependencies": {
    "react": "^18 || ^19",
    "react-dom": "^18 || ^19"
  }
}

pnpm 会在 .pnpm 中分别 sandbox react@18 和 react@19，互不串味。各 app 跑各自的 React 版本。

场景 3 · 切包管理器：从 npm workspace 到 pnpm

背景：npm workspace monorepo（10 个包），决定切 pnpm 用 catalog + 严格性。

步骤：

  1. 装 pnpm（npm install -g pnpm 或 corepack enable pnpm）
  2. 把根 package.json 的 "workspaces": [...] 复制到新建的 pnpm-workspace.yaml（YAML 格式）
  3. 删除根 package.json 的 "workspaces" 字段
  4. 子包 package.json 中的 "@app/ui": "1.0.0" 改成 "@app/ui": "workspace:*"
  5. 删 node_modules + package-lock.json
  6. pnpm install
  7. 跑全套测试——可能暴露幻影依赖（详见 P2.2 + 修复指引）
  8. 修完幻影依赖后 commit pnpm-workspace.yaml + pnpm-lock.yaml + 各包改动
  9. 更新 CI workflow（npm install → pnpm install --frozen-lockfile）

典型耗时 1-3 天（取决于幻影依赖数量）。

05 常见陷阱

陷阱 1 · 忘记把 monorepo 内部包标 private

monorepo 中 apps/* 通常不该 publish 到 npm（它们是 app，不是 lib）。如果 apps/web/package.json 没写 "private": true，pnpm publish 时可能误发。修复：所有 app 类包加 "private": true；只有真正要发的 lib 包不加。

陷阱 2 · 把 link: 当 workspace: 用

monorepo 内部依赖错写成 "@app/ui": "link:../ui"。表面能跑（dev 模式 symlink 工作），但publish 时崩溃（link: 不能 publish）+ lockfile 不记录依赖图。修复：monorepo 内部用 workspace:*，跨 monorepo（外部本地目录）才用 link:。

陷阱 3 · catalog 没声明的依赖也写 catalog:

子包写 "react-router": "catalog:" 但顶层 catalog 中没 react-router。pnpm install 报错 No version specified in catalog for react-router。修复：要用 catalog 必须先在顶层声明，或回到显式版本号。

陷阱 4 · 期望 npm 11 / Yarn 4 默认理解 catalog

2026-05 npm 11 不支持 catalog；Yarn 4 在 RFC 阶段。如果团队混用 npm/Yarn 装某个 pnpm monorepo，"catalog:" 写法直接装不上。修复：monorepo 团队统一用 pnpm 或 Bun；强制 packageManager 字段（详见下面）。

陷阱 5 · workspace:^ 在 pre-1.0 版本不直观

packages/ui 当前是 0.5.3，workspace:^ 解析为 ^0.5.3。但 SemVer 规则下 ^0.5.3 等价于 0.5.x（0.x 时主版本未稳定，^ 等价 ~）。建议：1.0 之前用 workspace:*（精确）或 workspace:~（明确次版本）；1.0+ 用 workspace:^。
