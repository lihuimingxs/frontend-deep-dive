测量 + ALPN/HTTPS DNS/ECH + CDN/Edge + 选型决策树 + 陷阱清单

整个主题的收束章。从"该不该开 HTTP/3"到"fetchpriority 怎么用"到"Early Hints 在我的 CDN 上行不行"，给一条端到端的决策链。同时把测量工具三件套、协议升级辅助、CDN/Edge 部署影响、12 个陷阱配修复合在一章。最后给 2027 展望。

阅读时长 ~22 分钟 前置 读过 P1-P6（本章是收束） 主要来源 RFC 7301（ALPN）+ RFC 9460（HTTPS DNS RR, 2023）+ ECH draft + Chrome DevTools 文档 + curl --http3 build 文档 + nghttp 文档 + Web Almanac 2025 + Cloudflare radar

01 速览

  1. 测量工具三件套必会：(a) curl --http3 -v 单请求看协议层；(b) nghttp -nv 看 HTTP/2 帧；(c) Wireshark + SSLKEYLOGFILE 看 QUIC / HTTP/2 完整 wire；外加 Chrome DevTools Network。
  2. 协议升级辅助：(a) ALPN（RFC 7301）在 TLS 握手协商应用层协议；(b) Alt-Svc 头让 server 通过 HTTP/2 响应告诉 client "下次也可以用 h3"；(c) HTTPS DNS RR（RFC 9460, 2023）让 DNS 直接返回 ALPN + IP + ECH 配置 → 第一次连就能直接 HTTP/3；(d) ECH（Encrypted Client Hello, draft）TLS 1.3 SNI 加密。
  3. CDN / Edge 部署的协议层影响：浏览器→CDN HTTP/3 默认开（Cloudflare / Fastly / Akamai / Vercel）；CDN→origin 仍以 HTTP/2 over TCP 为主；Edge runtime 可在 edge 直接发 Early Hints / Server Action。
  4. 3 层选型决策树：(a) 协议层升级（用 CDN？升 HTTP/3 监控 fallback；自建？评估 LB / NAT / firewall 是否准备好 UDP）；(b) 资源加载（LCP 关键资源 fetchpriority="high"；backend 慢加 103 Early Hints；不再用 Server Push）；(c) 流式响应（RSC / AI 走 fetch+ReadableStream；监控走 SSE；双向用 WebSocket；P2P 用 WebRTC）。
  5. 12 个陷阱清单：每条配修复 —— Server Push 配置、preload 滥用、SSE buffering、0-RTT POST、HTTP/3 fallback 不监控、自建 LB 不识别 connection ID、cookie 过大、HTTPS DNS RR 没配、WebSocket 心跳缺失、WebTransport 没 fallback、Early Hints 头不一致、引旧 RFC。
  6. 2027 展望：HTTP/3 普及率到 80%+；WebTransport 在 Firefox / Safari 落地；ECH 在更多浏览器默认开；Web Vitals 与协议层关联工具进一步成熟；QUIC 服务端实现进一步标准化（msquic / picoquic / aioquic / quiche / nghttp3 仍多元）；HTTP/2 over TCP 仍是 CDN ↔ origin 主力。
  7. 2026 强观点收尾：(a) Server Push 是失败实验，不要再配；(b) 用 CDN 的项目升级 HTTP/3 是默认动作；(c) LCP 优化的单点最大杠杆是给 hero image 加 fetchpriority="high"；(d) 流式响应优先 fetch + ReadableStream，不优先 SSE。

02 底层逻辑

测量工具三件套

1. curl --http3 -v

需要 quiche / ngtcp2 build 的 curl（curl --version 看是否含 "HTTP3" feature）。看一个请求走 HTTP/3：

bash 
curl --http3 -v https://example.com/

# 关键查看：
* ALPN: offers h3
* QUIC connection established  ← QUIC 握手完成
* h3 negotiated                ← HTTP/3 选定
> GET / HTTP/3
...

如果没装 HTTP/3 build：brew install curl --HEAD 或 docker pull archlinux:base（Arch Linux 默认 curl 含 HTTP/3）。

2. nghttp -nv

HTTP/2 调试瑞士军刀。可以看完整的帧序、HPACK 解码、stream 状态：

bash 
nghttp -nv https://example.com/

# 关键查看：
[  0.234] h2 negotiated via ALPN
[  0.235] send SETTINGS frame
[  0.238] send HEADERS frame <flags=0x05, stream_id=1>
          ; END_STREAM | END_HEADERS
          :method: GET
          :scheme: https
          :path: /
          :authority: example.com
[  0.345] recv HEADERS frame ...

3. Wireshark + SSLKEYLOGFILE

看真实 wire（特别是 QUIC，必须解密）：

bash 
# 1. 让 Chrome 把 TLS keys 写到日志
export SSLKEYLOGFILE=/tmp/sslkeylog.log
google-chrome --enable-quic https://example.com/

# 2. 抓包
tshark -i wlan0 -w /tmp/quic.pcap -f "udp port 443"

# 3. Wireshark 打开 quic.pcap
# Edit → Preferences → Protocols → TLS → Pre-Master-Secret log filename
# 选 /tmp/sslkeylog.log → Wireshark 自动解密

4. Chrome DevTools 内置

  • Network 面板：每个请求显示 Protocol 列（h1 / h2 / h3）；筛选 / 排序方便
  • chrome://net-internals/#http2：HTTP/2 连接状态 + frame 历史
  • chrome://net-internals/#quic：QUIC 会话 + 0-RTT 状态 + connection ID
  • chrome://flags：调实验功能（开/关 HTTP/3 / Early Hints / WebTransport）

协议升级辅助

ALPN（RFC 7301）

TLS 握手时协商应用层协议。Client 在 ClientHello 的 ALPN extension 里列出支持的协议；server 在 ServerHello 选定一个。常见协议名：

  • "h2" — HTTP/2
  • "http/1.1" — HTTP/1.1
  • "h3" — HTTP/3
  • "webrtc" — WebRTC

这是 web 协议升级的根机制。HTTPS over QUIC 也用 ALPN（在 QUIC 握手内部）。

Alt-Svc 头

http 
HTTP/2 200 OK
Content-Type: text/html
Alt-Svc: h3=":443"; ma=86400

# 解读: "下次连接也可以走 h3 端口 443，可信任此信息 86400 秒"
# Browser 缓存这个信息，下次访问 example.com 直接尝试 HTTP/3

Alt-Svc 是"老协议告诉 client 新协议在哪"的机制。HTTP/3 的初始普及大量靠 Alt-Svc：第一次访问走 HTTP/2，server 回 Alt-Svc 头通告 h3，第二次访问浏览器直接尝试 h3。

HTTPS DNS RR（RFC 9460, 2023）

新的 DNS 资源记录类型，把 ALPN + IP + ECH 配置直接放在 DNS 里。这样第一次访问就能直接走 HTTP/3，不用先经过 Alt-Svc。

text 
example.com.   3600   IN   HTTPS   1 . alpn="h3,h2" port=443 ipv4hint=...

# 浏览器查 DNS 时除了拿 A 记录，也拿 HTTPS RR
# 看到 alpn 含 h3 → 第一次连接就尝试 HTTP/3

2026 现状：Cloudflare / Fastly 已支持 HTTPS DNS RR；Chrome / Firefox / Safari 都已实现客户端解析。这是 HTTP/3 first-connection 优化的关键。

ECH（Encrypted Client Hello, draft）

TLS 1.3 ClientHello 里的 SNI 字段（标识访问哪个 server）目前是明文 → 中间人能看到你访问哪个域名。ECH 用 server 的公钥加密 SNI → 中间人看不见。

2026 现状：Cloudflare 已支持；Chrome / Firefox 实验开启；Safari 没做。ECH 配置通过 HTTPS DNS RR 下发（这是 RFC 9460 同时设计 ECH 配置的原因）。

CDN / Edge 部署的协议层影响

浏览器 → CDN（HTTP/3 默认开）

Cloudflare / Fastly / Akamai / AWS CloudFront / Vercel 都已默认对支持 QUIC 的 client 协商 HTTP/3。应用层零配置 —— 域名指向 CDN + HTTPS 证书有了，自动开。

但必须监控 fallback 率。CDN dashboard 都有 H3 connection / fallback rate 仪表。fallback 主要原因：

  • UDP 被 firewall / NAT drop
  • UDP 被 corp proxy 限速
  • 移动运营商对 UDP 限速
  • 客户端不支持 QUIC

CDN → origin（HTTP/2 主力，1.1 仍多）

Web Almanac 2025 数据：HTTP/2 占 75%，HTTP/1.1 占 23%，HTTP/3 仅 ~2%。理由：

  • 固定路由的稳定链路 → QUIC 优势体现不出来
  • origin LB / 防火墙改造成本高
  • HTTP/2 over TCP 实现成熟 + 调试工具好
  • 很多老 backend 仍只支持 HTTP/1.1

Edge runtime 的协议层能力

Cloudflare Workers / Vercel Edge Functions / Fastly Compute@Edge 等"在 edge 跑代码"的 runtime：

  • 可在 edge 直接发 103 Early Hints（在回源前 / 同时）
  • 可在 edge 直接处理 Server Action / API 路由
  • 可在 edge 操纵 priority 头转发
  • 大部分 Edge runtime 默认就在 HTTP/3 上跑（CDN 自动协商）

3 层选型决策树

第一层：协议层升级

text 
用主流 CDN（Cloudflare / Vercel / Fastly）？
  ├─ 是 → HTTP/3 默认开 ✅
  │       └─ 监控 fallback 率（< 5% 健康，> 15% 调查）
  │
  └─ 自建 / 老 CDN
      └─ 评估 LB / NAT / firewall 是否准备好 UDP
          ├─ 是 → 配置 HTTP/3 + 监控
          └─ 否 → 继续 HTTP/2 over TCP，先升级基础设施

数据中心内部？
  └─ 用 HTTP/2 over TLS（QUIC 优势小）

CDN ↔ origin 回源？
  └─ HTTP/2 over TCP 默认（QUIC 现阶段不必）

第二层：资源加载

text 
LCP 关键资源 (hero image / above-the-fold)
  └─ <img fetchpriority="high">
      + <link rel="preload" as="image" fetchpriority="high">

第三方 origin (字体 / 分析 / payment SDK)
  └─ <link rel="preconnect" crossorigin>

backend 慢 (TTFB > 300ms)
  └─ 103 Early Hints (CDN / Vercel / Cloudflare 默认开)
      或显式 res.writeEarlyHints()

路由切换预热 (用户 hover 链接)
  └─ Speculation Rules API (eagerness="moderate")

绝不再做的事
  └─ Server Push (RFC 9113 不再推荐, 2026 已退场)

第三层：流式响应

text 
RSC / AI / 实时 LLM 流式
  └─ fetch + ReadableStream + 自定义协议
      (NDJSON / Vercel AI SDK Data Stream Protocol)

监控 / 通知 / dashboard 推送
  └─ SSE (自动重连 + Last-Event-ID)

通用双向 (chat / collaboration)
  └─ WebSocket (RFC 6455 over HTTP/1.1 Upgrade)

低延迟多 stream + 知道用户在 Chromium
  └─ WebTransport over HTTP/3

P2P / 视频会议 / 实时协作
  └─ WebRTC DataChannel (+ TURN 后备)

绝不再做的事
  └─ HTTP/2 Server Push 推流式数据
      WebSocket 跑 RFC 8441/9220 over HTTP/2/3 (无人实现)

03 应用场景：决策实战

场景 A · 新 SaaS 项目（2026 首选栈）

(1) 域名指向 Cloudflare / Vercel → HTTP/3 自动开；(2) backend SSR 慢 → 103 Early Hints（CDN 默认开）；(3) LCP hero image 加 fetchpriority="high"；(4) 路由切换 Speculation Rules prefetch；(5) 实时通知 SSE；(6) chat 用 WebSocket；(7) 不写 Server Push 配置；(8) 监控 H3 fallback 率。

场景 B · 改造老应用（HTTP/2-only → HTTP/3）

步骤：(1) 用 Cloudflare / Fastly 前置 → 立刻有 HTTP/3 浏览器侧；(2) origin 仍 HTTP/2 over TCP 不动；(3) 移除老 Server Push 配置（nginx http2_push / Apache H2PushResource）；(4) 在 Link 头加 nopush 标记防止意外 push；(5) 添加 fetchpriority 到 LCP 资源；(6) 监控 fallback 率 + LCP 改善。预期：LCP 改善 10-25%，无业务代码改动。

场景 C · 全自建（电信 / 金融内网）

不能用公网 CDN（合规 / 数据本地化）。决策：(1) HTTPS over HTTP/2（仍是首选 —— 实现成熟、工具成熟）；(2) 内部 LB 用 HAProxy / Nginx；(3) 不上 HTTP/3（除非内网网络条件特殊，比如卫星链路）；(4) Server Push 仍不要用；(5) Early Hints 在自建场景需要应用代码改造，先评估 backend 慢的程度。

场景 D · 移动 SDK（iOS / Android）

移动客户端到云服务：(1) 用 CDN（含 HTTP/3）能直接显著改善体验；(2) iOS 17+ / Android 14+ 对 HTTP/3 / QUIC 已成熟；(3) 老移动版本 fallback HTTP/2 自动；(4) 长连接（IM 推送）用 WebSocket（成熟）或 MQTT-over-WebSocket。

05 常见陷阱（12 条收束清单）

陷阱 1 · 仍配 Server Push

nginx http2_push / Apache H2PushResource / 应用代码不加 nopush。修复：删配置 + Link 头加 nopush + 改 103 Early Hints + preload + fetchpriority。

陷阱 2 · preload 滥用

什么都 preload + 标 high。修复：每页 preload 最多 3-5 个资源；只标真正的 LCP 关键资源 high；其他用 auto / low。

陷阱 3 · preload 漏写 as / 字体没 crossorigin

导致 preload 失效（fetch 两次）。修复：preload 必须有 as="image|style|script|font|fetch"；字体加 crossorigin；type 显式（type="font/woff2"）。

陷阱 4 · SSE / RSC 被 nginx buffer

nginx 默认 proxy_buffering on。修复：proxy_buffering off; proxy_cache off; proxy_read_timeout 24h; + 应用响应头 X-Accel-Buffering: no。

陷阱 5 · HTTP/3 部署但不监控 fallback

大量用户实际仍 HTTP/2 但你以为 100% H3。修复：监控 navigation timing API 的 nextHopProtocol + 上报 RUM；CDN dashboard 看 H3 connection rate。

陷阱 6 · 自建 LB 不识别 QUIC connection ID

QUIC 连接迁移后 4-tuple 变 → LB 把它转到不同后端。修复：用 QUIC-aware LB（HAProxy 2.6+ / 云 LB）；或暂不开 HTTP/3（用 CDN 前置）。

陷阱 7 · 0-RTT 用在 POST

0-RTT 数据可被回放。POST 重放导致状态变更两次。修复：浏览器 / CDN 默认对 POST 关 0-RTT；自建 client 必须自己保证 0-RTT 仅幂等。

陷阱 8 · cookie 过大 + HTTP/1.1

每请求重发 4KB cookie，移动网络浪费带宽。修复：(a) 升级 HTTP/2 / HTTP/3 享受 HPACK / QPACK 自动压缩；(b) 关键认证用 Authorization header（精简）；(c) 大数据放 localStorage / IndexedDB。

陷阱 9 · WebSocket 没心跳 → 中间盒断

NAT / LB 通常对 idle TCP 5-30 分钟超时。修复：30-60 秒发心跳（ping/pong frame 或应用层 'ping' 字符串）；服务端识别心跳不当真实消息。

陷阱 10 · WebTransport 没 fallback → 非 Chromium 全挂

Safari / Firefox 不支持。修复：探测 typeof WebTransport !== 'undefined' → 不可用 fallback WebSocket；或直接用 WebSocket（多数项目这样）。

陷阱 11 · Early Hints 头和最终响应不一致

103 里 preload v1.css，最终 HTML 引用 v2.css。修复：必须严格同步。Next.js / 框架自动；自建必须谨慎。审计：抓包对比 103 Link 头 vs 最终 HTML 引用。

陷阱 12 · 引旧 RFC（7230 / 7540 / 7541）

2022 RFC 9110-9114 + 9204 已替代旧 RFC。文档 / 代码 review / 学习材料引旧 RFC 错过 erratum 修复。修复：2026 引用规范全用 9xxx；删旧 RFC 7230 / 7540 / 7541 引用。

06 延伸阅读 + 2027 展望

2027 展望

  • HTTP/3 普及率：2027 浏览器→CDN 段 80%+，CDN→origin 段 ~15-25%（仍以 HTTP/2 为主）
  • WebTransport：Firefox 稳定（预计 2026 末），Safari 仍未实现（无 roadmap）；2027 仍是"Chromium-only 实验"
  • ECH：Cloudflare / Mozilla 推动；2027 主流浏览器默认开
  • Web Vitals 与协议层关联工具：Chrome DevTools 显示 H3 0-RTT 命中 / fallback 原因；Lighthouse 加 HTTP/3 检查项
  • QUIC 服务端：msquic / picoquic / aioquic / quiche / nghttp3 / nginx-quic 仍多元；预计 2027-2028 出现"事实标准"实现（很可能是 Cloudflare quiche 或 nghttp3）
  • QUIC v2（RFC 草案）：在 QUIC 框架内演化的第一个案例 —— 不重起新协议名，仅版本号变化 + 新 transport 参数；2027 可能 RFC 化
  • "AI streaming over HTTP/3"：LLM 应用进一步普及，AI SDK Data Stream Protocol 等会成为流式响应的事实标准格式

本主题 2026 强观点收尾

  • "Server Push 已死" —— 2022 RFC 9113 不再推荐；2026 几乎所有 server 已停用；遇到老配置直接删
  • "用 CDN 的项目升 HTTP/3 是默认动作" —— Cloudflare / Vercel / Fastly 都默认开，应用层零配置
  • "LCP 优化的单点最大杠杆是 fetchpriority='high'" —— 给 hero image 加这一个属性，LCP 改善 中位数 9%、p75 改善 18%
  • "流式响应优先 fetch + ReadableStream" —— 比 SSE 更灵活；RSC / TanStack streamedQuery / Vercel AI SDK 全部走这条
  • "WebSocket 在 2026 仍是双向通用方案" —— WebTransport 等浏览器全支持后再说
  • "QUIC 不是 UDP，是在 UDP 上重建的完整传输层" —— 这个心智不立起来，HTTP/3 的所有讨论都偏
