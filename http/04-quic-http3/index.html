P4 · QUIC + HTTP/3 协议层 ⭐

本主题最难两章。QUIC 不是"换 TCP 为 UDP"，是在 UDP 上重建一个完整的、用户态的、加密内嵌的传输层；HTTP/3 不是"HTTP/2 升级版"，是把 HTTP/2 在 TCP 上打的所有补丁都扔掉之后的减法版。这一阶段把 RFC 9000-9002 + 9114 + 9204 这一套规范的核心机制讲清。

章节数 2 主要来源 RFC 9000 (QUIC core) + RFC 9001 (TLS in QUIC) + RFC 9002 (loss + congestion) + RFC 9114 (HTTP/3) + RFC 9204 (QPACK) + Cloudflare/Fastly/Google QUIC 工程博客 + Daniel Stenberg "HTTP/3 Explained" + BBR 论文（Cardwell et al., Google 2016） 状态 ✅ 全部完成

本阶段覆盖

  • QUIC 5 个核心创新 · 每条 stream 独立丢包恢复 / TLS 1.3 内嵌 / 0-RTT / 连接迁移 / 用户态可演化
  • QUIC 帧 / 包结构 · long header vs short header · 几十种 frame 类型 · packet number space
  • 0-RTT 安全权衡 · 没有前向保密 · 可被回放 · 仅用于幂等请求
  • 连接迁移工程现实 · NAT / LB connection ID-aware 路由 · 切 wifi/4G 不断连
  • HTTP/3 帧大幅简化 · 不再有 STREAM / WINDOW_UPDATE / PING / RST_STREAM（全交给 QUIC）
  • QPACK · 静态表 99 项 · encoder stream / decoder stream 双 stream · 解决乱序引用
  • RFC 9218 简单优先级 · urgency 0-7 + incremental · 替代 HTTP/2 prioritization tree 失败
  • 拥塞控制：BBR vs Cubic · QUIC 用户态可换算法 · BBRv1 → BBRv2 → BBRv3 公平性问题 · 2026 部署现状
  • HTTP/3 部署真实路径 · 浏览器 → CDN：/3 over QUIC（2026 普及率 ~70%）；CDN → origin：通常仍 /2 over TCP

⚠ 本阶段是难章

P4 是整个主题最难的两章。建议先读 P1.2（设计哲学）建立"三代抽象升级"心智，再读 P3.2（HTTP/2 的 TCP HOL 限制）建立"为什么 QUIC 必要"动机，最后读 P4。本阶段大量引用 RFC 章节号（9000 §17 long header / §19 frames / §22 transport parameters；9001 §4 packet protection；9114 §4 frames；9204 §3 dynamic table），建议手边开 RFC 一起读。
