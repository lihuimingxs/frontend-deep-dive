HTTP/3 over QUIC + QPACK + 拥塞控制（BBR / Cubic）

HTTP/3 是减法。RFC 9114 把 HTTP/2 的 stream / 流控 / RST_STREAM / PING 全扔回 QUIC，HTTP/3 frame 只剩 DATA / HEADERS 等少数几种。但 HPACK 在 QUIC 乱序传输上不工作，必须重做 QPACK（RFC 9204）—— 用独立的 encoder/decoder stream 同步动态表。同时 QUIC 用户态让拥塞控制能从 Cubic 快速切到 BBRv3 而不依赖 OS 内核，这一章把这条工程意义讲清。

阅读时长 ~25 分钟 前置 读过 P4.1（QUIC 在 UDP 上重建传输层） 主要来源 RFC 9114（HTTP/3）+ RFC 9204（QPACK）+ RFC 9218（priority signaling）+ Google BBR 系列论文（Cardwell et al., 2016 / 2019 / 2023）+ Cloudflare 关于 BBR 部署的工程博客

01 速览

  1. HTTP/3（RFC 9114, 2022-06）= HTTP-over-QUIC。它不是 HTTP/2 升级版，是 HTTP/2 减法版。把 stream 抽象 / 流控 / RST_STREAM / WINDOW_UPDATE / PING 全交给底下的 QUIC，HTTP/3 自身的 frame 类型大幅简化。
  2. HTTP/3 frame 类型：DATA / HEADERS / CANCEL_PUSH / SETTINGS / PUSH_PROMISE（保留 wire 但不用）/ GOAWAY / MAX_PUSH_ID。不再有 STREAM / WINDOW_UPDATE / PING / RST_STREAM。
  3. HTTP/3 stream 映射：每个 HTTP 请求 = 一条 QUIC bidirectional stream（client 发起，奇数 ID）。HEADERS frame 在 stream 起始；DATA frame 跟上。
  4. QPACK（RFC 9204）：解决 HPACK 在 QUIC 乱序传输上不工作的问题。静态表扩到 99 项，动态表更新走独立的 encoder stream（unidirectional, server→client / client→server 各一对），解码走独立的 decoder stream。
  5. QPACK 关键能力：解码器看到未知动态表索引时可以"等" encoder stream 上对应的更新到达，不用立刻报错。这是 HPACK 不能做的（HPACK 假设 header block 按发送顺序到达）。
  6. HTTP/3 优先级：HTTP/2 prioritization tree 失败被废弃，HTTP/3 直接采用 RFC 9218 简单方案：Priority: u=3, i 头（urgency 0-7 + incremental flag）。
  7. 拥塞控制：BBR vs Cubic：QUIC 在用户态，可以快速换算法。Cubic（Linux 默认 2006）基于丢包；BBR（Google 2016）基于带宽 + RTT。BBRv2（2019）/ BBRv3（2023）解决 BBRv1 公平性问题。2026 Google / Cloudflare 大量部署 BBRv3，自建多用 Cubic 默认。
  8. HTTP/3 部署真实路径：浏览器 ↔ CDN 这一段 HTTP/3 普及率约 70%（Web Almanac 2025）；CDN ↔ origin 这一段仍以 HTTP/2 over TCP 为主（约 75%）。HTTP/3 主要解决"最后一公里"，不是端到端。

02 底层逻辑

HTTP/3 是减法

HTTP/2 在 TCP 上打了大量补丁来模拟"多 stream + 流控 + 重置"。这些功能 QUIC 都已经原生提供了。所以 HTTP/3 的设计哲学是：把 QUIC 已经做好的事全交还给 QUIC，HTTP/3 只管 HTTP 自己的事。

HTTP/2 vs HTTP/3 frame 对照

HTTP/2 frame HTTP/3 是否保留 为什么
HEADERS ✅ 保留（用 QPACK 编码） HTTP 语义层需要 headers
DATA ✅ 保留 HTTP 语义层需要 body
SETTINGS ✅ 保留（连接级） QPACK 配置 / 优先级配置
GOAWAY ✅ 保留 HTTP 层有序关闭语义
STREAM 抽象 ❌ 移到 QUIC QUIC 已有 stream
WINDOW_UPDATE ❌ 移到 QUIC QUIC 流控代替（MAX_DATA / MAX_STREAM_DATA）
RST_STREAM ❌ 移到 QUIC QUIC 的 RESET_STREAM frame
PING ❌ 移到 QUIC QUIC PING frame
PRIORITY ❌ 移除 RFC 9218 用 Priority 头
PUSH_PROMISE ⚠ 保留 wire 但不用 Server Push 已废弃
CONTINUATION ❌ 不需要 QUIC stream 自己处理大消息分段

HTTP/3 的实际 wire 主要是 HEADERS 和 DATA 两种 frame。比 HTTP/2 简单很多。

HTTP/3 stream 映射

RFC 9114 §6 定义了几种 QUIC stream 用途：

  • Bidirectional client-initiated stream（QUIC stream id 4n）：每个 HTTP 请求/响应一个
  • Unidirectional client-initiated stream（4n+2）：control stream（连接级 SETTINGS + GOAWAY）/ QPACK encoder stream / QPACK decoder stream
  • Unidirectional server-initiated stream（4n+3）：server 的 control stream / QPACK encoder stream / QPACK decoder stream / push stream

所以一个 HTTP/3 连接典型 stream 布局：

text 
QUIC connection (client ↔ server)

Unidirectional control stream (client → server)  ← 一条
  (载 SETTINGS / GOAWAY)

Unidirectional QPACK encoder stream (client → server)  ← 一条
  (载动态表更新 from client)

Unidirectional QPACK decoder stream (client → server)  ← 一条
  (载 ack from client decoder)

Unidirectional control stream (server → client)  ← 一条
  (载 SETTINGS / GOAWAY)

Unidirectional QPACK encoder stream (server → client)  ← 一条
  (载动态表更新 from server)

Unidirectional QPACK decoder stream (server → client)  ← 一条
  (载 ack from server decoder)

Bidirectional request streams                    ← N 条 (每个 HTTP 请求一条)
  Stream 0: GET /index.html  →  HTML response
  Stream 4: GET /style.css   →  CSS response
  Stream 8: GET /script.js   →  JS response
  ...

这是 HTTP/3 的关键设计：QPACK 用独立的 unidirectional stream 解决动态表同步问题。

QPACK：HPACK 在 QUIC 上的重做

HPACK 在 QUIC 上为什么不工作

已在 P3.2 解释。简短：HPACK 假设动态表更新按 header block 发送顺序解码。HTTP/2 over TCP 是字节流有序，没问题；QUIC 上 streams 乱序到达，HPACK 解码器看到未知索引时不知道该等还是该报错。

QPACK 的解决思路

QPACK 把动态表的更新和引用分开：

  • 动态表更新走独立的 encoder stream（unidirectional），按发送顺序到达
  • request stream 上的 header block 引用动态表索引
  • 如果 request stream 比 encoder stream 上对应更新先到达 → request stream 解码暂存，等 encoder stream 上对应更新到达再继续
  • decoder 通过独立的 decoder stream回 ack，让 encoder 知道哪些动态表索引已被对方知晓

QPACK 静态表（99 项）

QPACK 静态表比 HPACK 大（99 项 vs 61 项），覆盖更多现代 web 常见 header：

  • HTTP/2 时代不存在的 header（strict-transport-security / content-security-policy / access-control-allow-origin）
  • 更多 method / status code 组合（更多 :status 值）
  • 更多 cookie 相关字段

QPACK encoder stream 工作流

text 
场景：client 第一次发请求 /api/users，header 含
  authorization: Bearer xxx
  user-agent: Mozilla/5.0 ...

Encoder 决策（per header）：
  - 静态表有：用静态表索引（比如 :method GET 用静态表索引 17）
  - 静态表没但值未变：加进动态表 + 用动态表索引引用
  - 静态表没且值唯一：直接 literal 编码（不进动态表）

QPACK encoder stream 上发送（unidirectional, in-order):
  → "Insert with name reference (static idx 23): Bearer xxx"   (动态表新增 idx 62)
  → "Insert with name reference (static idx 58): Mozilla/5.0..."(动态表新增 idx 63)

Request stream 0 上发送 HEADERS frame:
  → ":method" → 静态表索引 17
  → ":path"   → literal "/api/users"
  → "authorization" → 动态表索引 62  ← 引用
  → "user-agent"    → 动态表索引 63  ← 引用

Decoder 端收到 request stream 0 的 HEADERS frame，但 encoder stream 上的更新还没到：
  → 暂存这个 HEADERS frame，等 encoder stream 解析到 idx 62 / 63 的定义
  → 一旦 idx 62 / 63 定义到达，恢复解码 HEADERS frame

这个机制让 QPACK 能在 QUIC 乱序传输上工作。代价是实现复杂度比 HPACK 高一截，特别是 stream 缓冲管理。

QPACK 的"draining" / "blocked stream" 概念

  • blocked stream：等 encoder stream 上对应更新的 request stream（暂时解不出 header）
  • SETTINGS_QPACK_BLOCKED_STREAMS：双方协商最多允许多少个 blocked stream（避免无限暂存）
  • encoder 必须避免引用让 blocked stream 数超过协商值的动态表索引

这是 QPACK 实现里最微妙的部分。生产实现（Chromium / Firefox / nghttp3 / Cloudflare quiche）都花了大量精力调这个。

HTTP/3 优先级（RFC 9218）

HTTP/2 prioritization tree 失败已在 P3.1 + P3.2 讲。HTTP/3 直接采用 RFC 9218 的简单方案。

Priority 头格式

http 
# Priority 头放在 request headers 里:
GET /critical.css HTTP/3
priority: u=2, i

# u=N 是 urgency, 0-7 (0 最高)
# i 表示 incremental（progressive image / streaming JSON 等）
# 默认: u=3, 不带 i

典型 urgency 分配

资源类型 建议 urgency incremental
HTML u=2 i
关键 CSS u=2 —
JS（async / defer） u=3 —
图片（above the fold） u=3 或 u=4 i
图片（below the fold） u=4 或 u=5 i
分析 / 跟踪 u=6 或 u=7 —

浏览器（Chrome / Firefox / Safari）2026 都已实现自动设置 Priority 头 + fetchpriority API 让开发者覆盖（详见 P5.1）。CDN（Cloudflare / Fastly）已支持转发 + 按 Priority 调度 stream。

拥塞控制：Cubic vs BBR vs BBRv3

RFC 9002 给出 QUIC 的推荐拥塞控制（默认 NewReno）+ 拥塞控制接口规范。允许（且鼓励）实现替换为其他算法（Cubic / BBR / 自家算法）。

Cubic（Linux 默认 2006-至今）

  • 核心思想：拥塞窗口（cwnd）按三次曲线增长；丢包时按比例缩减。
  • 假设：丢包 = 网络拥塞信号。
  • 问题：在缓冲区大的网络（bufferbloat）上，Cubic 一直增长 cwnd 直到塞满缓冲区 → 高延迟。
  • 2026 仍是事实主力：Linux 内核默认；大部分 Web 服务器不调；自建 QUIC 实现也大多默认 Cubic。

BBR（Google 2016）

  • 核心思想：测量带宽（BtlBw）+ 测量 RTT（RTprop），cwnd ≈ BtlBw × RTprop。不用丢包作拥塞信号。
  • 优势：在 bufferbloat 网络上保持低延迟 + 高带宽利用。Google 自己的服务（YouTube / google.com）大幅受益。
  • 问题：BBRv1 在共享链路上对 Cubic 流不公平 —— BBR 流抢带宽，Cubic 流被挤。

BBRv2（2019）/ BBRv3（2023）

  • BBRv2：引入 ECN（Explicit Congestion Notification）+ inflight cap，改善公平性。
  • BBRv3：进一步调整带宽探测策略，对 Cubic 公平性提升 + 多链路场景更稳。
  • 2026 部署：Google / Cloudflare 后端大量 BBRv3；自建服务多用 Cubic 默认或 BBRv2。

QUIC 用户态可换的工程意义

  • TCP cubic → BBR：Linux 内核 4.9（2016-12 进主线）。服务器实际部署到 2018-2019 才普及。3 年 lag。
  • QUIC cubic → BBRv1 → BBRv2 → BBRv3：Cloudflare 内部 release 周期，几周到几个月。每次升级不需要 OS 内核改动。

这是 QUIC "用户态可演化"在生产价值的最直接体现。

HTTP/3 部署真实路径

Web Almanac 2025 数据告诉我们 HTTP/3 部署的真实分布：

text 
浏览器 / 客户端
       ↓ HTTPS 请求
[这一段：HTTP/3 普及率 ~70%]
       ↓
   CDN POP（边缘节点）
       ↓
[这一段：HTTP/2 ~75%, HTTP/1.1 ~23%, HTTP/3 ~5%]
       ↓
  Origin Server

为什么浏览器→CDN 是 HTTP/3 优势位

  • 用户网络条件多变（4G / wifi / 跨海光缆）
  • 用户切换网络（手机移动）
  • RTT 高（跨国）
  • 丢包率不稳定（1-5%）
  • QUIC 5 个核心创新全派上用场

为什么 CDN→origin 仍是 HTTP/2 主力

  • 固定路由的稳定链路
  • 低延迟（同区域 < 10ms）
  • 低丢包（< 0.1%）
  • QUIC 优势体现不出来
  • origin LB / 防火墙改造成本高
  • HTTP/2 over TCP 实现成熟稳定

03 应用场景

场景 A · CDN 默认开 HTTP/3：应用层零配置

Cloudflare / Fastly / Akamai / AWS CloudFront / Vercel 都默认对支持 QUIC 的 client 协商 HTTP/3。你的应用域名指向 CDN + HTTPS 证书有效，自动开。不需要应用层做任何事。但要看 CDN dashboard 监控 H3 fallback 率。

场景 B · 跨海 / 移动应用：QUIC 显著加速

跨海光缆 200-300ms RTT + 1-3% 丢包 + 移动场景。QUIC 0-RTT 重连免一个 RTT；连接迁移让切网络不断；每 stream 独立丢包恢复让大文件 / 视频流稳定。实测首屏 LCP 改善 30-50%。这是 HTTP/3 对 web 性能最大的贡献。

场景 C · 自建 origin：HTTP/2 仍主力，逐步 HTTP/3

自建 origin server 通常用 nginx / Apache / Node.js / Go / Rust。HTTP/2 over TCP 实现都已成熟；HTTP/3 实现仍较新（nginx-quic / Caddy / LiteSpeed / Cloudflare's quiche）。2026 自建 origin 仍以 HTTP/2 为主，逐步引入 HTTP/3。

场景 D · 拥塞控制：用 BBRv3 还是 Cubic

Google / Cloudflare 大量部署 BBRv3，对自家服务流量公平性 + 带宽利用都有优势。但自建 server 默认 Cubic 是合理的：BBR 在共享链路上对 Cubic 不公平，如果你的流量仅占链路一小部分，用 BBR 可能"抢"了别人的带宽。除非你跑大规模 CDN，默认 Cubic 是稳妥选择。

Cloudflare quiche 配置 BBRv3

rust 
// Rust + quiche 创建 QUIC server config
use quiche;

let mut config = quiche::Config::new(quiche::PROTOCOL_VERSION).unwrap();

// 拥塞控制选 BBR (quiche 内部已实现)
config.set_cc_algorithm_name("bbr").unwrap();
// 备选: "cubic" / "reno" / "bbr2"

// QPACK 配置
config.set_initial_max_streams_bidi(1000);
config.set_initial_max_streams_uni(100);
config.set_max_idle_timeout(30000);

// HTTP/3 ALPN
config.set_application_protos(&[b"h3"]).unwrap();

nginx-quic 启用 HTTP/3 + BBR

nginx 
http {
    server {
        listen 443 ssl;
        listen 443 quic reuseport;       # 启用 HTTP/3 over QUIC
        ssl_protocols TLSv1.3;
        http2 on;
        http3 on;

        # Alt-Svc 告诉旧 client "你下次也可以用 h3"

        # QUIC 拥塞控制由 nginx 内部实现选定，目前仍是 cubic
    }
}

测 HTTP/3 vs HTTP/2 在弱网下的差异

bash 
# Linux netem 模拟 200ms RTT + 3% 丢包
sudo tc qdisc add dev eth0 root netem delay 200ms loss 3%

# 测 5 个并发拉
URL="https://example.com/asset"
echo "HTTP/2:"
time curl --http2 -s -o /dev/null \
  $URL/{1,2,3,4,5}.jpg --parallel

  $URL/{1,2,3,4,5}.jpg --parallel

# 典型结果：
#   HTTP/2 over TCP:  ~5.0s (TCP HOL 影响明显)
#   HTTP/3 over QUIC: ~2.8s (每 stream 独立 + 1-RTT 握手)

05 常见陷阱

陷阱 1 · 把 HTTP/3 当成"HTTP/2 升级版"

HTTP/3 不是 HTTP/2 的升级版。它是HTTP 协议自身的简化 + 委托给 QUIC 做"多 stream + 流控"。性能改善主要来自底下的 QUIC，不是 HTTP/3 自身。在内网低丢包环境，HTTP/3 可能比 HTTP/2 还慢（QUIC 用户态有 CPU 开销）。

陷阱 2 · 想用 HPACK 压缩 HTTP/3

HPACK 在 QUIC 乱序传输上不工作。HTTP/3 必须用 QPACK。不要从 HTTP/2 实现配置直接套到 HTTP/3。HPACK 配置（headerTableSize 等）都要换成 QPACK 等价物（QPACK_MAX_TABLE_CAPACITY / QPACK_BLOCKED_STREAMS）。

陷阱 3 · QPACK encoder stream 阻塞 → 全连接 stall

如果 encoder stream 被阻塞（流控满 / encoder 实现 bug），所有引用动态表的 request stream 都会等。生产实测：QPACK_BLOCKED_STREAMS 配太小（比如 10）会让大量 streams 解码 stall。建议配 100 以上。

陷阱 4 · 用 RFC 9218 Priority 头但 server 不支持

2026 浏览器都已实现 fetchpriority → Priority 头映射，但很多自建 server（特别是不在主流 CDN 后面的）仍不识别 Priority 头，按"FIFO"调度 stream。在这种 server 上 fetchpriority 无效。检查方法：抓包看响应顺序是否符合 priority 期望。

陷阱 5 · 配 BBR 但没监控其他流量

BBRv1 / BBRv2 / BBRv3 在共享链路上对 Cubic 流不公平。如果你的服务跑在共享数据中心 / VPS（与其他业务共用上行带宽），开 BBR 可能"挤"了别人。除非你跑大规模 CDN（带宽完全自己掌控），默认 Cubic 是更礼貌的选择。

陷阱 6 · 期望 HTTP/3 解决 backend 慢的问题

HTTP/3 优化的是网络传输层。如果你的 backend 处理一个请求要 2 秒（数据库慢 / 业务逻辑慢），HTTP/3 一个 RTT 都救不了。这种情况看 103 Early Hints + 资源提示（详见 P5.2）—— 让 client 在 backend 处理时已经在 preload 资源。

陷阱 7 · 升级 HTTP/3 但不监控 QUIC 0-RTT 命中率

0-RTT 只对回访用户生效（有 PSK）。新用户 / 清 cookie 用户仍是 1-RTT。实测命中率 30-50%。如果你 dashboard 只看"总响应时间"不分 0-RTT vs 1-RTT，无法区分协议层优化效果。分桶看：0-RTT 命中流量的首响应时间 vs 1-RTT 流量的首响应时间。
