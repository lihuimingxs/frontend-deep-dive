QUIC 在 UDP 上重建传输层 · streams / 0-RTT / 连接迁移 / TLS 1.3 集成

整个主题最难一章。QUIC（RFC 9000-9002）不是 UDP 的快速包装，是在 UDP 上重新实现的、用户态的、加密内嵌的、多路独立流的完整传输层。这一章按 RFC 9000（core）+ 9001（TLS 集成）+ 9002（loss & congestion）拆解 QUIC 的 5 个核心创新：每条 stream 独立丢包恢复、TLS 1.3 内嵌、0-RTT 重连、连接迁移、用户态可演化；同时把 0-RTT 的安全权衡和连接迁移的工程现实讲清。

阅读时长 ~30 分钟（最长一章） 前置 读过 P1.2（设计哲学）+ P3.2（HTTP/2 的 TCP HOL 限制） 主要来源 RFC 9000 / 9001 / 9002 + Daniel Stenberg "HTTP/3 Explained" + Cloudflare blog "The Road to QUIC" + Robin Marx 的 QUIC 论文 + Google QUIC team 的 0-RTT 安全分析

01 速览

  1. QUIC 是跑在 UDP 上的完整传输层。它不是 UDP 包装，是在 UDP 之上重新实现"可靠 + 有序 + 加密 + 多路独立"。一个生产级 QUIC 实现至少 5-10 万行代码（RFC 9000 三份 RFC 加起来 280 页）。
  2. 5 个核心创新：(1) 每条 stream 独立丢包恢复；(2) TLS 1.3 内嵌握手；(3) 0-RTT 重连；(4) 连接 ID 解耦于 IP+port 的连接迁移；(5) 用户态可演化。
  3. 包结构：long header（握手期）+ short header（建连后）。长头有版本号 + 源/目的 connection ID；短头只有目的 connection ID + packet number。
  4. frame：QUIC 包内含一个或多个 frame（CRYPTO / STREAM / ACK / PADDING / NEW_CONNECTION_ID 等几十种）。
  5. 每条 stream 独立丢包恢复：QUIC streams 在 UDP 上各自有序，互不影响。stream A 丢包 stall stream A，stream B / C 继续。这是消灭 TCP HOL blocking 的核心。
  6. TLS 1.3 内嵌（RFC 9001）：握手 = TLS 1.3 + QUIC transport 参数 + 密钥派生合并成 1 RTT。HTTP/2 over TLS 是 TCP 1 RTT + TLS 1 RTT。QUIC 省掉一个 RTT。
  7. 0-RTT：第二次连同一 server，client 用上次的 PSK 直接发 application data 在第一个 RTT 内。代价：没有前向保密 + 可被回放 → 只能用于幂等请求（GET）。
  8. 连接迁移：连接由独立的 connection ID 标识，不是 4-tuple（IP+port）。手机切 wifi → 4G IP 变了，QUIC 连接 ID 不变 → 连接照旧。但工程现实是NAT / LB 必须 connection-ID-aware，2026 大型 CDN 都做了，自建 LB 大多没做。
  9. 用户态可演化：拥塞控制 / 流控 / 丢包恢复全在用户态（浏览器 / CDN 进程内），换算法不需要 OS 内核升级 → 演化速度从"OS 升级周期"提到"软件 release 周期"。

02 底层逻辑

QUIC 包结构

QUIC packet 跑在 UDP datagram 里。一个 UDP datagram 可以塞一个或多个 QUIC packet（coalesced packets）。

Long header（握手期）

text 
RFC 9000 §17.2 · Long Header Packet:

+-+-+-+-+-+-+-+-+
|1|1|T T|X X X X|       Header Form (1) + Fixed Bit (1) + Type (2) + Type-specific (4)
+-+-+-+-+-+-+-+-+
|        Version (32)               |
+-+-+-+-+-+-+-+-+
|DCID Len (8)   |
+---------------+
|  Destination Connection ID (0..160)   |
+---------------+
|SCID Len (8)   |
+---------------+
|  Source Connection ID (0..160)        |
+---------------+
|  Type-specific payload (...)          |
+---------------+

Type 4 种:
  Initial    (00) — 握手起始
  0-RTT      (01) — 0-RTT 数据
  Handshake  (10) — 握手中间
  Retry      (11) — server 要求 client 重发握手 (DDoS 防御)

Short header（建连后）

text 
RFC 9000 §17.3 · Short Header (1-RTT) Packet:

+-+-+-+-+-+-+-+-+
|0|1|S|R R|K|P P|       Header Form (0) + Fixed Bit (1) + Spin Bit (1)
+-+-+-+-+-+-+-+-+         + Reserved (2) + Key Phase (1) + Packet Number Length (2)
|  Destination Connection ID (0..160)   |
+---------------+
|  Packet Number (8/16/24/32)           |
+---------------+
|  Encrypted Payload (...)              |
+---------------+

短头：去掉 version / source CID，因为建连后双方都知道
       packet number 加密保护

注意几点：

  • 除了 first byte + connection ID 部分，整个 packet payload 加密。中间盒看不见 packet 内容。
  • connection ID 是核心：它解耦了"连接"和"4-tuple（IP+port）"。即使 client IP 变了，server 仍按 connection ID 找到对应的连接状态。
  • packet number 单调递增，独立于 stream offset，用于 ACK 和重传。

QUIC frame 类型（几十种）

QUIC packet 内承载 frame。frame 类型按功能分类：

类别 frame 类型 用途
密钥/握手 CRYPTO 承载 TLS 1.3 握手消息
数据 STREAM stream 上的应用数据（HTTP/3 frame 等）
ACK ACK / ACK_ECN 确认收到 packet（按 packet number range）
流控 MAX_DATA / MAX_STREAM_DATA / DATA_BLOCKED / STREAM_DATA_BLOCKED 连接级 + stream 级窗口
stream 控制 RESET_STREAM / STOP_SENDING 取消 stream
连接管理 NEW_CONNECTION_ID / RETIRE_CONNECTION_ID / NEW_TOKEN 下发新 conn ID（连接迁移用）
路径验证 PATH_CHALLENGE / PATH_RESPONSE 验证迁移后新路径可用
探活/连接 PING / CONNECTION_CLOSE / HANDSHAKE_DONE 保活、关闭
填充 PADDING 放大/抗放大

一个 packet 可以塞多个 frame，甚至跨 stream（一个 packet 里包含 stream 1 的 STREAM frame + stream 3 的 STREAM frame + 一个 ACK frame）。

每条 stream 独立丢包恢复

这是 QUIC 消灭 HOL blocking 的核心机制。

关键观察

TCP 的可靠 + 有序保证作用于整条字节流：任何一个 byte 丢了，TCP 等到所有前面的字节都到齐才能往应用层送。HTTP/2 多 stream 在 TCP 字节流上 → 一个 packet 丢影响所有 streams。

QUIC 把"可靠 + 有序"做在每条 stream 内部：

  • 每条 stream 有自己的逻辑 byte sequence（stream offset）
  • 每条 stream 有自己的 ACK 状态（"我收到了 stream 1 的 0-1499 字节"，"我没收到 stream 1 的 1500-2999 字节"）
  • stream A 缺字节只让 stream A stall，stream B / C 不受影响

实现层面：packet number 和 stream offset 解耦

这是 QUIC 设计的精巧处。RFC 9000 §13.1：

  • packet number：单调递增，整个连接共享一个序列。用于 ACK 和拥塞控制。
  • stream offset：每条 stream 自己的字节序号。用于 stream 内部的有序保证。

同一个 packet 里可以包含 stream 1 offset 1000-1499 + stream 3 offset 0-499。如果这个 packet 丢了：

  • QUIC 知道 packet number X 丢了 → 重传 packet X 内的 frames
  • 但 stream 1 已收到 0-999 + 1500-1999？没关系，stream 1 等 1000-1499 重传到位再交付应用层
  • stream 3 已收到 500-999？stream 3 等 0-499 重传到位再交付
  • stream 5 完全没受影响（它的字节根本不在丢的 packet 里）→ 继续推进

这是 HTTP/2 不能做到的（HTTP/2 没有"每条 stream 自己的 ACK 状态"，所有 stream 共用 TCP 的 byte stream ACK）。

TLS 1.3 内嵌（RFC 9001）

QUIC 不是"先建 UDP 连接 + 然后跑 TLS"，是把 TLS 1.3 内嵌进 QUIC 握手。RFC 9001 详细规定怎么嵌。

关键事实

  • TLS 1.3 record layer不在 QUIC 里。TLS 只贡献握手（key exchange / certificate / finished）+ 密钥派生。
  • 数据加密由 QUIC 的 packet protection（AEAD）做，密钥来自 TLS。
  • 必须用 TLS 1.3，不能用 TLS 1.2 —— QUIC 对加密 API 的要求只有 1.3 提供。
  • TLS 1.3 握手消息（ClientHello / ServerHello / EncryptedExtensions / Certificate / Finished）通过 CRYPTO frame 承载。

握手时序对比

text 
HTTP/2 over TLS 1.3 握手时序：

  TCP SYN              →
                       ← TCP SYN+ACK         (RTT 1: TCP 握手)
  TCP ACK              →
  TLS ClientHello      →
                       ← TLS ServerHello
                         ServerCertificate
                         ServerFinished       (RTT 2: TLS 1.3)
  TLS ClientFinished   →
  HTTP request         →
                       ← HTTP response        (RTT 3: 第一个请求)

  Total: 3 RTTs

QUIC 握手时序：

  Initial (CRYPTO: ClientHello)        →
                                       ← Initial (CRYPTO: ServerHello)
                                         Handshake (CRYPTO: certificate +
                                                    Finished)             (RTT 1: 握手 + TLS 合并)
  Handshake (CRYPTO: ClientFinished)
  + 1-RTT (HTTP request)               →
                                       ← 1-RTT (HTTP response)            (RTT 2: 第一个响应)

  Total: 2 RTTs (省一个)

跨海 200ms RTT 链路上，省一个 RTT 是 200ms 实打实的差异。这是 HTTP/3 在跨国场景显著加速的根本理由。

0-RTT 重连

QUIC 在第二次连同一 server 时可以省更多。

原理

  1. 第一次连接成功后，server 给 client 发一个 NEW_TOKEN frame（含 PSK / session ticket）。
  2. 第二次连接时，client 用 PSK 直接派生 0-RTT 密钥，在第一个 packet 里就能发加密的 application data。
  3. Server 验 PSK 通过 → 直接处理 application data 不等 1-RTT 握手完成。

0-RTT 时序

text 
QUIC 0-RTT 时序 (第二次连接):

  Initial (CRYPTO: ClientHello + PSK)
  + 0-RTT (HTTP GET request, 用 0-RTT 密钥加密)  →
                                                  ← Initial (CRYPTO: ServerHello)
                                                    Handshake (CRYPTO: Finished)
                                                    1-RTT (HTTP response)         (RTT 1)

  Total: 1 RTT (含数据)
  (实际上 application data 在 0 RTT 就发出去了)

0-RTT 让"第二次访问已经认识的 server"几乎免延迟。CDN 大量场景受益（用户回访、跨页面导航）。

0-RTT 安全权衡

0-RTT 不是免费午餐：

  • 没有前向保密（forward secrecy）：0-RTT 数据用 PSK 加密；如果 PSK 泄露，过去的 0-RTT 数据可被解密。1-RTT 数据用握手生成的 ephemeral 密钥，PSK 泄露不影响过去数据。
  • 可被回放（replay attack）：攻击者抓到 0-RTT packet → 在另一个时间重放 → server 看到合法 PSK + 合法应用数据，处理它。如果是 GET 不影响（幂等），如果是 POST → 多次执行同一动作（CSRF-like）。

工程姿势：

  • 浏览器（Chrome / Firefox）默认只对 GET 启用 0-RTT，POST / PUT / DELETE 走 1-RTT 握手
  • CDN 默认对幂等请求开 0-RTT，非幂等关
  • RFC 9001 §9.2 给出"0-RTT 数据应该幂等"的明确规范

连接迁移：connection ID 解耦于 4-tuple

TCP 连接由 4-tuple（src IP + src port + dst IP + dst port）唯一标识。手机切 wifi → 4G，src IP 变 → TCP 4-tuple 变 → TCP 看到这是另一个连接 → 原连接断开 → 重连。

QUIC 的解法：连接由独立的 connection ID 标识。手机切网络后：

  1. Client 看到自己 IP 变了
  2. Client 在新 IP 上发一个带原 connection ID 的 QUIC packet（含 PATH_CHALLENGE frame）
  3. Server 看到旧 connection ID 来自新 IP → 触发 path validation（PATH_RESPONSE）
  4. Path validation 通过 → server 把新 IP 关联到此连接 → 连接照旧

HTTP/3 over QUIC 的视频流不卡断 / 下载不中断就是这个机制。

工程现实：NAT / LB 必须 connection-ID-aware

但实际生产中连接迁移没那么简单：

  • NAT 路由器：很多 NAT 把 UDP 流按 4-tuple 分流。客户端 IP 变了 → NAT 把它当新连接，可能 drop。
  • Load balancer：默认按 4-tuple 哈希分配后端 → 4-tuple 变了 → 转到不同后端 → 后端看到旧 connection ID 不知道是谁。
  • Anti-DDoS / 防火墙：可能丢掉"突然变 IP 的连接"作为攻击防御。

解决：LB 必须按 connection ID 路由（QUIC-aware LB）。Cloudflare / Google / Fastly 后端的 LB 都做了；自建 LB（HAProxy / Nginx LB）大多没做或刚做。

RFC 9000 §5.1 + §9 详细规定了 connection ID 的下发机制：server 通过 NEW_CONNECTION_ID frame 持续给 client 备用 conn ID；client 在迁移时换用备用 ID 让中间盒看起来更像"新连接"，避免被路径上某些设备识别为"同一连接换 IP"而被 drop。

用户态可演化

QUIC 的所有逻辑（拥塞控制 / 流控 / loss recovery / packet protection）都在用户态。

对比 TCP

  • TCP cubic → BBR：Linux 内核 4.9（2016-12 发布）才进 BBR 主线；服务器实际部署到 2018-2019 才普及。
  • QUIC 拥塞控制：Cloudflare 把 BBRv1 → BBRv2 → BBRv3 一路升级，从决定换到全量上线只需要几周。

实际收益

  • 新拥塞算法（BBRv3, COPA, Brisk）能更快进生产
  • flow control 调优能针对应用场景定制（视频流和大文件下载用不同窗口策略）
  • 实验性功能（multipath QUIC 等）可以快速试错

03 应用场景

场景 A · 移动应用 / 视频流：QUIC 优势位

用户在地铁里 wifi → 4G 切换 / 高铁里基站切换 / 跨国漫游。这些场景丢包率 1-5%、网络条件多变、连接被切断的概率高。QUIC 的连接迁移 + 0-RTT + 每 stream 独立丢包恢复全派上用场。视频流 / 直播 / 大文件下载在 QUIC 上稳定性比 HTTP/2 over TCP 好 30-50%。

场景 B · 跨海 / 高 RTT 链路：QUIC 握手优化收益最大

跨海光缆 200-300ms RTT。HTTP/2 over TLS 首请求要 3 RTT = 600-900ms。QUIC 首请求 2 RTT = 400-600ms（节省 200-300ms），0-RTT 重连 1 RTT = 200-300ms（节省 400-600ms）。跨国 SaaS 服务在 QUIC 上的首屏 LCP 改善能到 30-50%。

场景 C · CDN 部署：默认开

Cloudflare / Fastly / Akamai / AWS CloudFront / Vercel 都默认对支持 QUIC 的 client 协商 HTTP/3 over QUIC。应用层不需要做什么 —— 域名指向 CDN，HTTPS 证书有了，自动开。但要监控 fallback 率（CDN dashboard 都能看）。

场景 D · 数据中心内部：QUIC 优势体现不出来

内网链路低延迟、低丢包、固定路由。QUIC 的连接迁移 / 0-RTT 没意义；用户态拥塞控制 vs Linux 内核 cubic 在数据中心内差异 < 5%。仍用 HTTP/2 over TCP（mTLS）是合理选择。Kubernetes / Istio 内部全部 HTTP/2。

用 Node.js（实验）跑一个 HTTP/3 server

javascript 
// Node.js 22+ 实验性 HTTP/3 支持，需要 --experimental-quic 标志
// 生产 server 通常用 Cloudflare quiche / google quiche / nginx-quic / Caddy / LiteSpeed

import { createQuicSocket } from 'node:net';  // 实验 API

const socket = createQuicSocket({
  endpoint: { port: 443 },
  server: {
    key: fs.readFileSync('key.pem'),
    cert: fs.readFileSync('cert.pem'),
    alpn: 'h3',
  },
});

socket.on('session', (session) => {
  console.log(`new QUIC session, conn id = ${session.connectionID}`);
  session.on('stream', (stream) => {
    // stream 是 QUIC stream（不是 HTTP/3）
    // 实际生产用 nghttp3 实现 HTTP/3 帧解析
    stream.on('data', (chunk) => {
      console.log('stream data:', chunk);
    });
  });
});

socket.listen();

Node.js 22 / 23 的 QUIC 支持仍是实验性的（--experimental-quic 标志）。生产 HTTP/3 server 用 Cloudflare quiche、Google quiche、nginx-quic、Caddy、LiteSpeed —— 都已成熟。

05 常见陷阱

陷阱 1 · 把 QUIC 当成"快速 UDP"

QUIC 不是 UDP 的简单包装。它在 UDP 上重新实现了一个完整的传输层（可靠 + 有序 + 多路 + 加密 + 流控 + 拥塞）。生产实现至少 5-10 万行代码。"用 UDP 替代 TCP"是完全错误的概括。

陷阱 2 · 0-RTT 用在 POST / 状态变更请求

0-RTT 数据可被回放（replay attack）。POST / PUT / DELETE 等非幂等请求重放会导致状态变更两次（CSRF-like）。RFC 9001 §9.2 明确要求 0-RTT 数据应该幂等。浏览器 / CDN 默认对非幂等请求关 0-RTT，自建 client 必须自己保证。

陷阱 3 · 自建 LB 不识别 QUIC connection ID

HAProxy / Nginx LB 默认按 4-tuple 哈希分配后端。QUIC 连接迁移后 4-tuple 变 → LB 把它转到不同后端 → 后端看到陌生 connection ID 直接断。必须用 QUIC-aware LB（HAProxy 2.6+ / Nginx Plus / Cloudflare 内部 LB）。

陷阱 4 · 部署 HTTP/3 但不监控 fallback 率

UDP 在某些 corp firewall / 老 NAT 被 drop / 限速 → QUIC 握手失败 → 浏览器 fallback 到 HTTP/2 over TCP。如果你不监控 fallback 率，可能你以为 100% 用户在 HTTP/3，实际上只有 50%。CDN dashboard 都有 H3 connection / fallback rate 仪表，必须看。

陷阱 5 · UDP 被 firewall / NAT 限速 / drop

很多企业 firewall 默认对 UDP 限速（怕 DDoS / 流媒体占带宽）。HTTPS over UDP（QUIC）有时被当成"奇怪的 UDP 流量"丢。QUIC 的 anti-amplification 机制（RFC 9000 §8）有部分缓解，但 UDP 路径存在永远是 H3 部署的潜在风险。

陷阱 6 · 把 QUIC 默认 wire 当成"明文 UDP"调试

QUIC 包除了几个握手字段全部加密。tcpdump 抓到的 QUIC packet 你看不到内容，连 stream id / packet number 都加密保护。调试 QUIC 必须 SSLKEYLOGFILE + Wireshark。直接 tcpdump -X 看到的全是密文。

陷阱 7 · 误以为 0-RTT 普遍生效

0-RTT 需要客户端有上次连接的 PSK / session ticket。新用户首次访问、清 cookie 后访问、切设备访问 —— 这些场景都没 PSK，仍是 1-RTT。0-RTT 只对回访用户生效。实测大部分 web 流量 0-RTT 命中率 30-50%，不是 100%。

陷阱 8 · TLS 1.2 想跑 QUIC

QUIC 强制 TLS 1.3（RFC 9001 §4）。TLS 1.2 不提供 QUIC 需要的 API。如果 server 只支持 TLS 1.2 → QUIC 协商失败，必须升级到 TLS 1.3。2026 几乎所有 production server 都已 TLS 1.3，但内网老 server 偶尔仍有 1.2-only。
