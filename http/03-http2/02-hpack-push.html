HPACK 头压缩 + Server Push 兴衰 + HTTP/2 的 TCP HOL 限制

这一章讲 HTTP/2 的两个独立创新（HPACK 真成功 / Server Push 真失败）和一个根本性局限（TCP HOL）。HPACK 把 header 压到 10-20%，但它的"动态表必须按顺序更新"这一假设在 QUIC 上不成立 → 必须重做 QPACK。Server Push 是 HTTP/2 最大的失败实验。TCP HOL 是 HTTP/2 over TCP 不可逾越的根本局限，也是 QUIC + HTTP/3 必要性的根。

阅读时长 ~22 分钟 前置 读过 P3.1（HTTP/2 帧 + 多路复用） 主要来源 RFC 7541（HPACK，被 RFC 9113 引用）+ Chrome 团队 2017-2020 关于 Server Push 的实测博文 + Robin Marx 关于 TCP HOL 在 HTTP/2 上的 2020 年研究 + Cloudflare 退役 Server Push 的工程博客

01 速览

  1. HPACK（RFC 7541，被 9113 引用）= 静态表 61 项 + 动态表 + Huffman。把典型 1-4 KB headers 压到 100-300 字节。
  2. 静态表：常见 header 名 + 值的固定 61 项（:method GET / content-type: text/html / cache-control: max-age=0 等）。一字节索引引用。
  3. 动态表：连接生命周期内 client 和 server 各维护一份；新 header 可以加进去，后续引用时就一字节索引。
  4. Huffman 编码：动态表里的字符串 header 用预定义 Huffman 表压缩；典型字符压到 4-6 bit。
  5. HPACK 的核心缺陷：动态表更新必须按 header block 发送顺序解码 —— TCP 上有序传输没问题，但 QUIC 上 streams 乱序到达 → HPACK 解不出来 → HTTP/3 必须重做 QPACK（详见 P4.2）。
  6. Server Push（PUSH_PROMISE 帧）：HTTP/2 的"杀手锏"，server 主动推 client 还没请求的资源。设计意图：HTML 解析未到 <link rel="stylesheet"> 时 server 已经把 CSS 推过去。
  7. Server Push 实测失败：Chrome 2017-2020 的 production 数据显示绝大部分场景 push 伤性能（cache 协调失败 + 重复推送 + 浏览器实现 bug）。2018 Chrome 限制 / 2022 RFC 9113 不再推荐 / 2026 几乎所有 server 已停用。Web Almanac 2025 启用率 < 1%。
  8. TCP HOL blocking 限制：HTTP/2 多路复用解了应用层 HOL，但 wire 上仍是一条 TCP 字节流。任何 IP packet 丢失整条 TCP 流 stall → 所有 streams 看不到新字节。Robin Marx 2020 研究：3% 丢包率下 HTTP/2 over 1 个连接比 HTTP/1.1 over 6 个连接还慢。这是 HTTP/3 出现的根本原因，不可在 TCP 上修复。

02 底层逻辑

HPACK：静态表 + 动态表 + Huffman

HPACK 的核心思想：HTTP headers 高度重复。每个请求都有 :method GET / :scheme https / user-agent: ... / cookie: ...。如果能用极短的索引引用这些重复内容，压缩比能到 80-90%。

静态表（RFC 7541 Appendix A）

61 项预定义 header，常见的几个：

Index Header Name Header Value
1:authority
2:methodGET
3:methodPOST
4:path/
5:path/index.html
6:schemehttp
7:schemehttps
8:status200
.........
17accept-charset
.........
61www-authenticate

Encoder 看到 :method: GET → 输出一字节 0x82（高位 1 表示"索引引用"，6 位 = 2 = 静态表第 2 项）。Decoder 看到 0x82 → 查表 → :method: GET。整个 header 被压成 1 字节。

动态表

静态表只覆盖 61 项常见 header。具体值（比如 cookie: sessionid=abc... / user-agent: Mozilla/...）需要动态表：

  • 动态表是 连接生命周期内 client 和 server 各维护一份的 FIFO
  • 大小受 SETTINGS_HEADER_TABLE_SIZE 控制（默认 4096 字节）
  • 新 header 加到动态表，索引从静态表大小（61）+ 1 = 62 起递增
  • 引用动态表的 header 用 1-2 字节索引
  • 动态表满了按 FIFO 淘汰

所以一个连接里 cookie / user-agent 这种长 header 第一次发送时编码完整字符串（占字节多），后续引用时只占 1-2 字节索引。这是 HPACK 真正的省字节地方。

Huffman 编码

动态表里仍是字符串的 header（第一次发送或未进表的）用预定义 Huffman 表（RFC 7541 Appendix B）：

  • 字母 / 数字 / 常见符号用 5-6 bit
  • 不常见符号用 8-30 bit
  • 典型 ASCII header 字符串压到 60-70%

HPACK 编码流程示例

同一连接发两个请求：

text 
请求 1:
  :method: GET                    → 索引 2 (静态表)         = 1 字节
  :scheme: https                  → 索引 7 (静态表)         = 1 字节
  :path: /api/users               → name 索引 4 + value 字符串 (Huffman) = 8 字节
                                    + 加进动态表 (index 62)
  :authority: example.com          → name 索引 1 + value 字符串 = 9 字节
                                    + 加进动态表 (index 63)
  user-agent: Mozilla/5.0 ...     → name 索引 58 + value 字符串 = 35 字节
                                    + 加进动态表 (index 64)
  ────────────────────────────
  请求 1 头总计 ≈ 54 字节

请求 2 (同连接):
  :method: GET                    → 1 字节
  :scheme: https                  → 1 字节
  :path: /api/users/1             → name 索引 4 + value 字符串 = 9 字节
                                    + 加进动态表
  :authority: example.com          → 索引 63 (动态表) = 1 字节  ← 复用
  user-agent: Mozilla/5.0 ...     → 索引 64 (动态表) = 1 字节  ← 复用
  ────────────────────────────
  请求 2 头总计 ≈ 13 字节  (节省 76%)

这是 HPACK 在长连接上的核心优势。第一个请求带全量编码，之后每个请求只发"变化部分"。实测对 cookie 大的网站节省 80% header 流量。

HPACK 的核心缺陷：动态表必须按顺序更新

HPACK 的"省"基于一个假设：client 和 server 的动态表完全同步。这个同步靠"按发送顺序解码"实现：

  1. encoder 把 stream 1 的 headers 编进 wire；动态表加 cookie 索引 62
  2. encoder 把 stream 3 的 headers 编进 wire；引用索引 62（cookie）
  3. HTTP/2 over TCP 下，stream 1 的 HEADERS 帧总是先到，stream 3 后到
  4. decoder 先解 stream 1 → 动态表加 cookie 索引 62 → 再解 stream 3 → 用索引 62 找到 cookie ✅

这在 TCP 上工作良好，因为 TCP 保证字节流有序。

但在 QUIC 上 streams独立有序、互相之间无序。如果 stream 3 的 HEADERS 先到达：

  1. decoder 收到 stream 3 的 headers，引用索引 62
  2. 动态表里没索引 62（stream 1 的 headers 还没到）
  3. 解码失败 ❌

这就是 HTTP/3 必须重做 QPACK 的根：QPACK 用独立的 encoder stream / decoder stream（单向）发送动态表更新，让 decoder 看到未知索引时可以等 encoder stream 上对应的更新到达。详见 P4.2。

Server Push：从杀手锏到失败实验

设计意图（2009-2015）

HTML 解析串行：浏览器先收 HTML → 解析出 <link rel="stylesheet" href="style.css"> → 发请求拿 style.css。这一段串行"等"是个浪费。

SPDY / HTTP/2 的解决方案：server 知道 index.html 必然依赖 style.css，发完 index.html 顺手 push style.css。客户端拿到 HTML 时 CSS 已经在 cache 里。

Wire 上长这样：

text 
Client → Server:
  HEADERS (stream=1, GET /index.html)

Server → Client:
  PUSH_PROMISE (stream=1, promised stream=2, headers for /style.css)
  HEADERS (stream=1, :status 200, content-type text/html)
  DATA (stream=1, <html>...)
  HEADERS (stream=2, :status 200, content-type text/css)  ← 主动推
  DATA (stream=2, body { ... })

看起来很美好。客户端不用等 HTML 解析就能拿到 CSS。

工程现实：失败的几个根本理由

理由 1 · 浏览器 cache 协调失败：

浏览器可能已经在自己的 cache 里有 style.css（上次访问留下的）。但 server 不知道 cache 状态。怎么办？

  • Server 总是 push → 浪费带宽（重复推送已有资源）
  • Server 不 push → 失去 push 的意义
  • 引入 cache digest（client 告诉 server "我有什么"）→ 计算贵 + 实现复杂 + 隐私问题

RFC 7540 留了 RST_STREAM 给 client "不要这个 push"，但客户端要在 push 已经在 wire 上之后才能拒绝 → 已经浪费带宽。

理由 2 · 浏览器实现 bug 多：

Chrome / Firefox / Safari 实现 push cache 各不相同。Chrome 2017-2020 多次发现 push 资源没正确进 cache、或者进了 cache 但下次请求没用上。这种 bug 让"push 真的有用"在生产中不稳定。

理由 3 · 实测数据显示伤性能：

Chrome 团队 2018 公布 production 数据：

  • 使用 push 的站点 vs 不使用 push 的站点：大部分指标（FCP / LCP / TTI）差异 < 0.5%
  • 但流量浪费明显（重复 push 已 cache 资源）
  • 有些场景 push 还更慢（push 占了 server 上行带宽，挤压主请求）

2018 年 Chrome 限制 push（默认禁用某些场景）。2020 Chrome 完全考虑移除。

正式退场（2022-2026）

  • 2022-06 RFC 9113 §8.4 明确"不再推荐"使用 PUSH_PROMISE
  • 2022 Chrome 默认禁用 push（仍可手动启用）
  • 2023 Cloudflare / Fastly / Akamai 移除 push 功能
  • 2024-2026 几乎所有 server 已停用；Web Almanac 2025 启用率 < 1%
  • 替代方案：103 Early Hints + preload（详见 P5.2）—— 让 client 自己决定要不要下、cache-aware

HTTP/2 的 TCP HOL blocking 限制

这是 HTTP/2 不可在 TCP 上修复的根本局限。

TCP 字节流模型

TCP 给应用层的抽象是"一条有序、可靠的字节流"。它假设：

  • Sender 发 N 字节
  • Receiver 必须按发送顺序收到这 N 字节，无丢失
  • 任何字节丢失，TCP 必须重传到全部到齐才能往应用层送

这套模型在 1980 年代为"应用是文件传输 / telnet / 邮件"设计是合理的。但 web 多路复用场景下不再合理。

HTTP/2 在 TCP 字节流上的现实

假设 HTTP/2 一个连接里 3 个 streams 在传：

text 
HTTP/2 wire (在 TCP 字节流上):

  [DATA frame stream=1] [DATA frame stream=3] [DATA frame stream=5] [DATA frame stream=1] ...

  ↓ TCP 把这条字节流拆成 IP packets:

  TCP packet 1: bytes 0-1499      包含 stream 1 的部分 DATA
  TCP packet 2: bytes 1500-2999   包含 stream 3 的部分 DATA  ← LOST!
  TCP packet 3: bytes 3000-4499   包含 stream 5 的部分 DATA  ← 到达
  TCP packet 4: bytes 4500-5999   包含 stream 1 的部分 DATA  ← 到达
  ...

  TCP 看到 packet 2 丢了 → 等重传
  TCP packet 3、4 已到达，但 TCP 不能把它们往应用层送，因为缺 packet 2
  → 整条 TCP 连接 stall

应用层 HTTP/2 看到的：
  stream 1 数据 ……stall……
  stream 3 数据 ……stall……
  stream 5 数据 ……stall……

  全部 streams 都看不到新字节，等 packet 2 重传完成。

这就是 TCP HOL blocking：一条字节流上多个独立逻辑流，任何 packet 丢失影响所有逻辑流。

Robin Marx 2020 实测

Robin Marx（Hasselt University 博士）2020 年发表 "A Quic look at HTTP/2 vs QUIC HOL blocking"，测了多种丢包率下的真实表现：

丢包率 HTTP/1.1 (6 connections) HTTP/2 (1 connection) QUIC + HTTP/3
0% (理想) 基线 -15% 时间（更快） -15% 时间
1% +2% -5%（仍快） -15%
3% +5% +10%（HTTP/2 更慢！） -10%
5% +8% +25%（更明显） -5%

关键洞察：

  • 0% 丢包：HTTP/2 多路复用 + HPACK 让它比 HTTP/1.1 快 ~15%
  • 3-5% 丢包：HTTP/2 反而比 HTTP/1.1 慢，因为 1.1 的 6 个连接里只有一个被 HOL block 影响，其他 5 个仍能跑；HTTP/2 一个连接被 block 等于全部 stream 被 block
  • QUIC 在所有丢包率下都最稳：每条 stream 独立丢包恢复

这是 HTTP/3 必要性的实测数据基础。在弱网环境（3G / 4G 切换 / 跨国光缆），3-5% 丢包是常态。HTTP/2 在这些场景下不是"加速"是"减速"。

03 应用场景

场景 A · HPACK 在 cookie 大网站的真实价值

大量 cookie / Authorization header 的 SaaS 后台（每请求 3-4 KB headers）：HPACK 把第一个请求之后的 headers 压到 100-200 字节。对移动用户带宽和电池的节省非常实在。这是 HTTP/2 vs 1.1 最稳定的优势。

场景 B · 仍配 Server Push 的老站点：删掉

2018-2022 推 Server Push 的工程师设计的配置（nginx http2_push / link header rel=preload + nopush 反向标记）2026 已经是反模式。遇到老配置直接删，改 103 Early Hints + preload（详见 P5.2）。

场景 C · 弱网应用首选 HTTP/3

移动应用 / 跨国服务 / 视频流 / 旅行场景应用 —— 用户网络条件多变 + 丢包率 1-5%。HTTP/2 over TCP 在这些场景表现差。这是 HTTP/3 优势位。Cloudflare / Fastly 默认对支持 QUIC 的客户端协商 HTTP/3，应用不需要做什么。

场景 D · 数据中心内部 HTTP/2 仍最优

低延迟 + 低丢包 + 固定路由 + 双向都 mTLS。HTTP/2 + HPACK 在这种场景比 HTTP/3 还稳（QUIC 用户态有额外 CPU 开销，数据中心场景不划算）。Kubernetes / Istio / Linkerd 内部仍 HTTP/2。

看 SETTINGS_HEADER_TABLE_SIZE 的影响

javascript 
// 服务端调小动态表
import http2 from 'node:http2';

const server = http2.createSecureServer({
  /* tls config */
  settings: {
    headerTableSize: 0,  // 完全禁用动态表（每个 header 都全量编码）
  },
});

// 实测：cookie 大的请求 headers 体积膨胀 5-10x
// 一般不要这样做；调到 0 只在调试 / 安全审计时

HPACK 也有过侧信道攻击风险（CRIME-like：通过观察压缩后大小推测 cookie 内容）。HTTP/2 RFC 提到，但实际在 TLS 内部 + 动态表分隔下风险很低。完全禁用动态表代价大，几乎无人这样做。

05 常见陷阱

陷阱 1 · 仍在配置 Server Push

nginx 的 http2_push / 应用代码里发 Link: ...; rel=preload 但没标 nopush（仍触发 push）—— 都是反模式。2026 全部删掉，改 103 Early Hints + preload（详见 P5.2）。

陷阱 2 · 误以为 HTTP/2 处处都比 HTTP/1.1 快

3-5% 丢包率下 HTTP/2 over 1 个连接比 HTTP/1.1 over 6 个连接还慢（实测）。所以"我开了 HTTP/2 性能就一定提升"是错的判断。HTTP/2 的优势在低丢包 + 大量并发请求 + headers 重复多的场景。

陷阱 3 · 期待 HPACK 在 HTTP/3 上工作

HPACK 的"动态表必须按顺序更新"假设在 QUIC 乱序传输上不成立。HTTP/3 必须用 QPACK。不要把 HTTP/2 的 HPACK 实现 / 配置直接套用到 HTTP/3。

陷阱 4 · 动态表大小没设上限

SETTINGS_HEADER_TABLE_SIZE 默认 4096。如果 server 配大（比如 64KB）但又有很多 cookie 大的 client，动态表占用内存 × 连接数可能撑爆。生产 server 应该限制在 4-16 KB。

陷阱 5 · 用 HTTP/2 over 跨海链路低估 RTT 成本

HTTP/2 解了应用层 HOL，但在跨海 200ms RTT + 1-3% 丢包链路上，TCP HOL 仍然让大文件下载明显卡。这种场景必须升 HTTP/3（QUIC 每 stream 独立丢包恢复）。

陷阱 6 · 把 HPACK 看作"安全机制"

HPACK 是压缩，不是加密。它降低了 CRIME-like 攻击的可行性（动态表分隔 + Huffman 表固定），但不是"安全特性"。HTTPS 的安全靠 TLS 1.3，HPACK 只是省字节。
