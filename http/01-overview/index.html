P1 · 是谁 / 为什么

不讲帧格式细节，只解决两个问题：从 1991 年 HTTP/0.9 到 2022 年 RFC 9000-9114 这 35 年里，每隔 7-10 年就要换一代协议是必然还是偶然；以及"消息 → 帧+流 → 用户态传输层"这条三代抽象升级路径背后的设计哲学是什么。给后面所有"为什么这样设计"提供历史与思想坐标。

章节数 2 主要来源 IETF HTTP/QUIC WG charter + Google SPDY whitepaper + Jim Roskind QUIC 早期演讲 + Daniel Stenberg "HTTP/3 Explained" + Robin Marx 的 HOL blocking 研究 状态 ✅ 全部完成

本阶段覆盖

  • 1991 · HTTP/0.9（Tim Berners-Lee）一行 GET，无 status code / 无 headers
  • 1996 · HTTP/1.0（RFC 1945）引入 headers / methods / status codes
  • 1997-1999 · HTTP/1.1（RFC 2068 → 2616）keep-alive / Host / chunked，主导 web 20 年
  • 2009 · SPDY（Google）实验多路复用 / 头压缩 / Server Push
  • 2014 · RFC 7230-7235 把 HTTP/1.1 重切为 6 个 RFC
  • 2015 · HTTP/2（RFC 7540）IETF 标准化，基于 SPDY
  • 2012-2013 · Google QUIC（gQUIC）在 UDP 上重建传输层
  • 2016 · IETF QUIC WG 成立，从 gQUIC 抽象出标准
  • 2021-05 · RFC 9000-9002（QUIC v1）正式标准化
  • 2022-06 · RFC 9110-9114 大重构 + HTTP/3 标准化 + Server Push 正式不再推荐
  • 2024-2026 · HTTP/3 在 Cloudflare/Fastly/Akamai 全面默认；Server Push 全面退场
  • 三代抽象升级 · 消息（HTTP/1.x）→ 帧+流（HTTP/2）→ 用户态传输层（QUIC + HTTP/3）
  • 5 个核心设计决策 · 加密默认 / 用户态可演化 / 每 stream 独立 / 中间盒防御 / TLS 1.3 内嵌
