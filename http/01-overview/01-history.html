HTTP 35 年项目史 + 同代人横评

从 1991 年 Tim Berners-Lee 一行 GET / 到 2022 年 RFC 9000-9114 一次性把 HTTP/1.1/2/3 + QUIC + 头压缩 + 优先级信号全部修订完毕，HTTP 这条线 35 年没有"改良"，一共发生了三次抽象层重做。这一章把演化串起来，让"为什么 HTTP/2 / HTTP/3 各自必要"以及"WebSocket / WebTransport / gRPC 在 2026 年各占什么位置"有历史根因。

阅读时长 ~22 分钟 前置 用过 fetch / curl，知道 HTTP request / response 结构 主要来源 IETF HTTP/QUIC WG charter 与各 RFC + Google SPDY whitepaper（2009）+ Jim Roskind QUIC 早期演讲（2013）+ Daniel Stenberg "HTTP/3 Explained" + Web Almanac 2024-2025

01 速览

  1. HTTP 不是一个协议，是三代协议共用一套语义层。RFC 9110（HTTP semantics, 2022）把 method / status / header / cache 抽到独立文档；HTTP/1.1（RFC 9112）/ HTTP/2（RFC 9113）/ HTTP/3（RFC 9114）只决定"这套语义在 wire 上怎么编码"。
  2. 1991-1996 · HTTP/0.9 → 1.0：从一行 GET 到引入 method / status / header / Content-Type。HTTP/1.0（RFC 1945）让 HTTP 第一次"长得像一个真正的协议"。
  3. 1997-1999 · HTTP/1.1（RFC 2068 → 2616）：keep-alive 默认开 / Host header 让一台 server 跑多站 / chunked transfer / pipelining（理论上）/ Range 请求。主导 web 20 年，2026 仍占 25-30% 流量。
  4. 2009 · SPDY（Google 实验）：Mike Belshe + Roberto Peon 把"二进制帧 + 多路复用 + 头压缩 + Server Push"打包成一个实验协议。HTTP/2 后来基本是 SPDY 的标准化。
  5. 2014-05 · RFC 7230-7235：把 1999 年的 HTTP/1.1（RFC 2616）的混乱定义重切成 6 个独立 RFC，第一次给 HTTP 一个"模块化"的规范结构。
  6. 2015-05 · HTTP/2 RFC 7540 + 7541：基于 SPDY，IETF 标准化。二进制 / 多路复用 / HPACK / Server Push 一次性进规范。
  7. 2012-2013 · Google QUIC（gQUIC）：Jim Roskind 团队在 UDP 上重建传输层；Google.com / YouTube 实验。动机：TCP 的 HOL blocking 在弱网下让 HTTP/2 也救不了。
  8. 2016 · IETF QUIC WG 成立：从 gQUIC 抽象出标准化版本。
  9. 2021-05 · RFC 9000-9002（QUIC v1）正式标准化。core / TLS / loss & congestion 三份 RFC。
  10. 2022-06 · HTTP 大重构：RFC 9110（semantics）/ 9111（caching）/ 9112（1.1 wire）/ 9113（2.0）/ 9114（3.0）/ 9204（QPACK）一次性发布；HTTP/2 RFC 7540 / 7541 被替代；Server Push 被正式不再推荐；prioritization tree 被废弃改用 RFC 9218。
  11. 2024-2026 · HTTP/3 在 Cloudflare / Fastly / Akamai 全面默认；浏览器到 CDN 这一段普及率约 70%；CDN 到 origin 仍以 HTTP/2 为主；Server Push 全面退场，103 Early Hints 接班。
  12. 2026 年同代人分工：HTTP/1.1 仍是 fallback 与回源主力；HTTP/2 是 CDN ↔ origin 默认；HTTP/3 是浏览器 ↔ CDN 默认；WebSocket 仍是双向通用方案；WebTransport 是 Chromium-only 实验；gRPC over HTTP/2 是服务端到服务端的事实主力。没有"一个协议统一所有场景"。

02 底层逻辑

1991-1996 · HTTP/0.9 → 1.0：从"一行 GET"到"长得像协议"

HTTP/0.9 不是规范，是 Tim Berners-Lee 1991 年在 CERN 实现的一个最小协议。整个协议只有一行：

http 
GET /index.html
<-- 服务器返回 HTML 内容，连接关闭

没有 status code、没有 headers、没有 method（除了 GET），响应只能是 HTML，连接每次重建。这套设计在 1991 年是合理的：那时 web 上只有几台服务器，每个文档就是 HTML。

1996 年 HTTP/1.0（RFC 1945） 让 HTTP 长得像一个真正的协议：

  • 引入 method（GET / HEAD / POST）
  • 引入 status code（200 / 301 / 404 / 500 等）
  • 引入 headers（请求与响应都有）
  • 引入 Content-Type，HTTP 不再是"只能传 HTML"的协议

但 HTTP/1.0 默认每个请求一个 TCP 连接 + 用完关闭。当 web 页面开始有几十张图片之后，每张图片都做一次 TCP 三次握手 + TLS 握手，延迟无法接受。

1997-1999 · HTTP/1.1：定型 web 20 年

HTTP/1.1（RFC 2068, 1997 → RFC 2616, 1999） 引入了一组延续到今天的核心机制：

  • keep-alive 默认开：一个 TCP 连接发完一个请求不关，下一个请求复用 → 省 TCP/TLS 握手
  • Host header：让一台服务器能 vhost 多个站点 → web hosting 商业模式的协议基础
  • Transfer-Encoding: chunked：动态长度响应；这是 SSE / 流式 RSC 的 wire 基础（详见 P6.1）
  • Range / 206 Partial Content：断点续传、视频拖动 → 静态 CDN 必备
  • pipelining：理论上可以连发多个请求不等响应 → 实际上中间盒兼容性问题严重，所有现代浏览器都禁用
  • Cache-Control / ETag / Vary：HTTP 缓存语义系统化（详见 RFC 9111，本主题不重讲）

HTTP/1.1 主导 web 20 年。但它的设计在 web 流量爆炸之后显出三个根本性天花板（详见 P2.1）：应用层 HOL blocking、6 connection limit、headers 不压缩重复发送。

1999 年的 RFC 2616 因为模块切分混乱、erratum 累积，到 2014 年被 RFC 7230-7235 重切成 6 个独立 RFC（消息语法 / 语义 / 条件请求 / range / cache / 认证）。这次重切是后来 RFC 9110 把"语义层"独立的前奏。

2009 · SPDY 实验：HTTP/2 的真正起点

2009 年 Google 的 Mike Belshe 和 Roberto Peon 公开 SPDY（读"speedy"）—— 一个 HTTP 上层的实验协议，目标是在不改 HTTP 语义的前提下，把 1.1 的天花板砸掉。

SPDY 引入的四个机制后来全部进入 HTTP/2 标准：

  • 二进制帧：替代 ASCII 文本协议，解析更快、状态机更明确
  • 多路复用 streams：一个 TCP 连接里跑多个虚拟 stream，解除应用层 HOL
  • 头压缩（SPDY 用 zlib，HTTP/2 改成更安全的 HPACK）
  • Server Push：server 主动推 client 还没请求的资源

SPDY 在 Google 自己的服务上跑了 2009-2015，然后让位给基于它的 HTTP/2 标准。SPDY 的代码并入 Chromium 后逐步移除（最终在 Chrome 51, 2016 年彻底删掉）。

关键事实：HTTP/2 不是"凭空想出来的标准"，是把 SPDY 这个跑了 6 年的实验协议规范化。它的所有设计选择都有 production 数据背书。

2015 · HTTP/2 RFC 7540：标准化 SPDY

HTTP/2（RFC 7540, 2015-05） + HPACK（RFC 7541） 由 Belshe 和 Peon（SPDY 同班人马）主导起草，IETF httpbis WG 历时 3 年完成。HTTP/2 保持HTTP 语义不变（method / status / headers 都不动），只重做 wire format。

HTTP/2 在 2015-2018 完成普及。Web Almanac 2018 数据：约 35-45% web 流量过 HTTP/2。但同时，工程现实暴露了 HTTP/2 的两个根本性局限：

  1. TCP HOL blocking 仍在：HTTP/2 多路复用解了应用层 HOL，但 TCP 字节流仍是有序的；任何一个 IP packet 丢失，整条 TCP 流 stall，所有 stream 都要等。弱网下 HTTP/2 可能比 HTTP/1.1（拆 6 个 TCP 连接）还慢。Robin Marx 的研究（2020）给了详细的实测数据。
  2. Server Push 是失败实验：浏览器 cache 和 server push 协调机制在工程上不成立 —— server 不知道 client 已有什么 cache，push 经常重复，反而吃带宽。Chrome 团队 2017-2020 实测表明大部分场景 Server Push 伤性能。2018 起 Chrome 开始限制 push；2022 RFC 9113 正式不再推荐；2026 几乎所有 server 已停用。

2012-2016 · gQUIC：在 UDP 上重做传输层

2012 年 Google 网络团队意识到：HTTP/2 在 TCP 上的所有补丁都救不了 TCP 自身的 HOL。同年 Jim Roskind 启动 gQUIC（Google QUIC）项目 —— 把传输层从 TCP 整体替换为 UDP + 用户态实现的新协议。

gQUIC 的关键决策（后来全部进入标准 QUIC）：

  • UDP 而不是新协议：因为新的 IP 协议号会被中间盒（NAT / firewall）丢弃；UDP 是中间盒认得的最少阻力路径
  • 每条 stream 独立的丢包恢复：streams 不再共享一个有序字节流 → 真正消灭 HOL blocking
  • 加密强制：内嵌 TLS-like 加密；中间盒看不见内部，不能像对 TCP 那样"理解+破坏"
  • 用户态实现：浏览器和 CDN 拥有完整控制权，不依赖 OS 内核升级
  • 0-RTT 重连：第二次连接同一 server 时直接发 application data，省 1 个 RTT
  • 连接 ID 解耦于 IP+port：手机切 wifi → 4G 时连接不断

gQUIC 在 Google 服务上跑了 2013-2018，2017 年 Google 报告 7% 的 internet 流量已经在 QUIC 上（绝大部分是 google.com / YouTube）。这是 IETF 启动 QUIC 标准化的实际数据基础。

2016-2021 · IETF QUIC：从实验到标准

2016 年 IETF QUIC WG 成立。WG 决定：标准 QUIC 不是 gQUIC v1.5，而是一次语义清理 + TLS 1.3 集成 + 抽象通用化的重设计 —— gQUIC 把 HTTP/2 的语义缠在 transport 里，标准 QUIC 把 transport 和 application 完全分层。

WG 历时 5 年，中间反复推翻早期决策（最大的一次是 2018 把 TLS 1.3 从"贴在 QUIC 旁边"改成"内嵌到 QUIC 包结构"）。最终成果：

  • RFC 9000（QUIC core, 2021-05）—— 包结构 / streams / 流控 / 连接迁移
  • RFC 9001（TLS in QUIC）—— TLS 1.3 与 QUIC packet protection 的集成
  • RFC 9002（loss & congestion）—— 包丢失检测 + 推荐拥塞控制（NewReno/CUBIC/BBR 都允许）

QUIC v1 锁定后，HTTP-over-QUIC 即 HTTP/3（RFC 9114, 2022-06） 才能定稿。

2022-06 · HTTP 大重构：6 份 RFC 一次性发布

2022 年 6 月 IETF 一次性发布了一套相互引用的 HTTP RFC，是 HTTP 自 RFC 2616（1999）以来最大的规范重构：

  • RFC 9110 · HTTP semantics —— 把 method / status / header / range / 认证 / cache 头等"和 wire 无关的语义"独立成单一文档；HTTP/1.1/2/3 都引用它
  • RFC 9111 · HTTP caching —— 替代 RFC 7234；含 RFC 5861 的 stale-while-revalidate 归并
  • RFC 9112 · HTTP/1.1 wire format —— 替代 RFC 7230；只剩 1.1 的字节怎么编码
  • RFC 9113 · HTTP/2 —— 替代 RFC 7540 + 7541 errata；明确移除 prioritization tree、Server Push 不再推荐
  • RFC 9114 · HTTP/3 —— HTTP-over-QUIC，frame 大幅简化
  • RFC 9204 · QPACK —— HTTP/3 的头压缩，HPACK 不能用在 QUIC 乱序传输上的替代
  • RFC 9218（紧接着 7 月发布）· priority signaling —— 替代 HTTP/2 prioritization tree 失败的尝试

这次重构的工程意义在于：HTTP 不再是"一坨规范，1.1 是基线、2/3 是补丁"的形态，而是语义层（9110）+ 缓存层（9111）+ 三种 wire 编码（9112/9113/9114）。三种 wire 完全平等，没有"主版本"。

2024-2026 · HTTP/3 全面默认 + Server Push 退场

2024 起 HTTP/3 在头部 CDN 全面默认开启：Cloudflare、Fastly、Akamai、AWS CloudFront、Vercel 都已默认对支持 QUIC 的客户端协商 HTTP/3。Web Almanac 2025 数据：

  • 浏览器 ↔ CDN 这一段：HTTP/3 普及率约 70%；HTTP/2 约 25%；HTTP/1.1 约 5%（多为老 client / 内网工具）
  • CDN ↔ origin 这一段：HTTP/2 约 75%；HTTP/1.1 约 23%；HTTP/3 仍少（因为回源 LB / 防火墙改造少）
  • Server Push 启用率：< 1%，等同于退场
  • 103 Early Hints：约 8% 网站启用，2022 年的 1% 翻了 8 倍（详见 P5.2）

03 应用场景：2026 年同代人分工

每代协议在 2026 年都没有完全退场，而是在自己擅长的位置占住。这一节把"在哪一段路上用什么"讲清。

场景 A · 浏览器 ↔ CDN：HTTP/3 默认

这一段路是 HTTP/3 普及最快的位置。理由：用户网络条件多变（4G / wifi / 国际跨海光缆），是 QUIC 的优势位（连接迁移 + 0-RTT + 每 stream 独立丢包恢复）。CDN 通过 Alt-Svc 头或 HTTPS DNS RR 通告 HTTP/3 端点；浏览器自动升级。

同时保留 HTTP/2 fallback：UDP 在某些企业 firewall 被丢，QUIC 握手失败 → 浏览器 fallback 到 HTTP/2 over TCP。CDN 仪表板都能看到 fallback 率。

场景 B · CDN ↔ origin：HTTP/2 仍主力

这一段路 HTTP/3 普及缓慢。理由：CDN 到 origin 通常是固定路由的高质量链路，UDP/TCP 性能差异小；origin LB / 防火墙 / 内部基础设施改造成本高；HTTP/2 over TCP 已经够用。Cloudflare 2024 给出的数据：回源用 HTTP/2 占 75%，HTTP/1.1 占 23%（仍有大量老 backend），HTTP/3 占比仍是个位数。

场景 C · 服务端 ↔ 服务端 RPC：gRPC over HTTP/2

gRPC（Google 出品的 RPC 框架）在 2015 年就锁定 HTTP/2 作为底层 wire；它依赖 HTTP/2 的多路复用 + 二进制帧 + trailers。2026 年仍是服务端到服务端 RPC 的事实主力（特别是 Kubernetes / Istio 生态）。gRPC over HTTP/3 是实验状态，未广泛部署。

场景 D · 浏览器双向：WebSocket 仍默认

WebSocket（RFC 6455, 2011）通过 HTTP/1.1 Upgrade 握手，切换到自己的二进制/文本帧协议。2026 仍是浏览器双向通信的默认选择 —— 库丰富、跨浏览器兼容、proxy 兼容性最好。RFC 8441（WebSocket over HTTP/2）和 RFC 9220（over HTTP/3）几乎无人实现。详见 P6.2。

场景 E · 内网工具 / 调试 / curl 一行：HTTP/1.1

HTTP/1.1 的 ASCII 文本格式让 telnet / curl -v / 写 shell 脚本能一行调通。HTTP/2 / HTTP/3 是二进制，需要 nghttp / curl --http3 build / Wireshark 才能看 wire。所以"工具链 + 内网调试"这一段 HTTP/1.1 不会消失。

场景 F · 低延迟实验 / 游戏：WebTransport over HTTP/3

WebTransport（W3C / IETF draft）把 QUIC 多 stream + datagram 暴露给浏览器 JS，用于"我不需要 HTTP 语义，只需要 QUIC 的低延迟双向流"场景。2026 仅 Chromium 系实现完整；Firefox 实验，Safari 完全没做。所以是"知道用户在 Chrome 才用"的实验路线。

04 典型代码：用 curl 看四代协议

历史最直观的体验方式是用 curl 一个个跑过去。下面用同一个 URL（假设 https://example.com）展示四代协议的 wire 长什么样。

HTTP/1.1：ASCII 文本

bash 
curl --http1.1 -v https://example.com/

# 关键 wire（去掉 TLS 部分）：
> GET / HTTP/1.1
> Host: example.com
> User-Agent: curl/8.5.0
> Accept: */*
>
< HTTP/1.1 200 OK
< Content-Type: text/html; charset=UTF-8
< Content-Length: 1256
< ...
< (body...)

注意：你能直接读 wire。这是 HTTP/1.1 至今活着的核心理由。

HTTP/2：二进制帧（用 nghttp 看）

bash 
nghttp -nv https://example.com/

# 关键 wire（nghttp 翻译过的人类可读视图）：
[  0.123] Connected
[  0.234] h2 negotiated via ALPN
[  0.235] send SETTINGS frame <length=18, flags=0x00, stream_id=0>
          (niv=3)
          [SETTINGS_MAX_CONCURRENT_STREAMS(0x03):100]
          [SETTINGS_INITIAL_WINDOW_SIZE(0x04):65535]
[  0.236] send HEADERS frame <length=43, flags=0x05, stream_id=1>
          ; END_STREAM | END_HEADERS
          :method: GET
          :scheme: https
          :path: /
          :authority: example.com
          accept: */*
          user-agent: nghttp2/1.55.1
[  0.345] recv HEADERS frame <length=125, flags=0x04, stream_id=1>
          ; END_HEADERS
          :status: 200
          content-type: text/html; charset=UTF-8
          content-length: 1256
[  0.346] recv DATA frame <length=1256, flags=0x01, stream_id=1>
          ; END_STREAM

注意：协议变成了 frame 序列，每个 frame 有 type / flags / stream id。HEADERS 和 DATA 走在同一个 stream（id=1）上，和别的请求复用同一 TCP 连接。

HTTP/3：QUIC 上的 frame（curl 需要 quiche/ngtcp2 build）

bash 
curl --http3 -v https://example.com/

# 关键 wire：
* ALPN: offers h3
* Connected to example.com (...)
* QUIC connection established
* h3 negotiated
> GET / HTTP/3
> Host: example.com
> User-Agent: curl/8.5.0 (+nghttp3)
> Accept: */*
>
< HTTP/3 200
< content-type: text/html; charset=UTF-8
< content-length: 1256
< (body...)

表面看 HTTP/3 的 curl 输出很像 HTTP/2 的"语义视图"，但底下走的是完全不同的传输层（UDP + QUIC streams）。要看真实 wire 必须 Wireshark + SSLKEYLOGFILE 解密 QUIC 包（详见 P7.1）。

WebSocket：HTTP/1.1 Upgrade 握手

http 
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

<-- 之后切换到 WebSocket 帧协议（不是 HTTP）-->

WebSocket 是"借 HTTP/1.1 握手然后切走"。握手完成后 wire 不再是 HTTP，是 WebSocket 自己的 frame 格式（RFC 6455）。这就是为什么 WebSocket 不能简单升级到 HTTP/2 / HTTP/3 —— 它的 wire 设计假设了 1.1 风格的"独占连接"。

05 常见陷阱

陷阱 1 · 把 HTTP/2 / HTTP/3 当成"主版本号"看待

HTTP/3 不是 HTTP/2 的升级版，HTTP/2 也不是 HTTP/1.1 的升级版。RFC 9110 把语义独立后，三个 wire 完全平等：你可以用 HTTP/3 走老 1.1 的 cookie 认证，也可以用 HTTP/1.1 走 RFC 9218 优先级头。"协议升级"是 wire 升级，不是语义升级。

陷阱 2 · 仍在配置 Server Push

2022 起 RFC 9113 已经明确不再推荐；2026 几乎所有 server 已停用；浏览器 cache 协调机制让 Server Push 实际上伤性能。遇到老配置直接删掉，改用 103 Early Hints + preload（详见 P5.2）。

陷阱 3 · 误以为 HTTP/3 处处比 HTTP/2 快

HTTP/3 的优势位是用户网络条件多变 + 高 RTT（4G / 跨国 / wifi 切换）。在数据中心内部低延迟低丢包的网络上，HTTP/2 over TCP 可能仍然更快（因为 TCP 拥塞控制经过 30 年优化、QUIC 仍有用户态开销）。所以 CDN ↔ origin 大部分仍是 HTTP/2。

陷阱 4 · WebSocket 想升级到 HTTP/2 / HTTP/3

RFC 8441（over /2）和 RFC 9220（over /3）几乎无人实现。如果你的应用想用 QUIC 的多 stream，应该考虑 WebTransport 而不是 WebSocket-over-HTTP/3。但 WebTransport 2026 仅 Chromium 系实现 —— 选型时必须考虑用户浏览器分布。

陷阱 5 · 把 SPDY 当成"被淘汰的失败品"

SPDY 不是失败，是实验阶段的 HTTP/2。它跑在 Google production 上 6 年，给 HTTP/2 提供了 production 数据。它在 2016 年被 Chrome 移除是因为 HTTP/2 标准化完成，不是因为设计错。HTTP/2 的几乎所有创新都来自 SPDY。

陷阱 6 · 引用旧 RFC（7230 / 7540 / 7541）

2022 RFC 9110-9114 + 9204 已经替代了 RFC 7230-7235（HTTP/1.1 模块化）+ RFC 7540 + 7541（HTTP/2 + HPACK）。文档 / 代码 review / 学习材料里如果引用旧 RFC，会错过 2022 重构里很多重要细节（比如 HTTP/2 的 prioritization tree 已废弃 / Server Push 不再推荐）。2026 写 HTTP 相关文档，引一手规范应该全是 9xxx。

陷阱 7 · 把 gRPC 当成"HTTP/2 的应用"

gRPC 锁定 HTTP/2 是因为它需要 trailers + 流式双向。它不能简单切到 HTTP/3 是因为部分 gRPC 框架的中间件假设了 HTTP/2 的 stream 模型。gRPC-over-HTTP/3 是 IETF draft，2026 部署率仍很低。如果你的服务正在升级 HTTP/3，gRPC 仍走 HTTP/2 是合理选择。
