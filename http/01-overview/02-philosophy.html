从消息 → 帧 → 流 → 用户态传输层 · 设计哲学

HTTP/1.x 把信息看作消息，HTTP/2 把信息看作帧 + 流，QUIC + HTTP/3 把信息看作独立用户态传输层上的多条流。三代抽象升级背后是四件事被工程现实逼出来的：TCP 不能在内核之外改、中间盒会破坏没加密的协议、加密握手太贵、网络是多链路的。这一章把"为什么是这条路"讲清。

阅读时长 ~22 分钟 前置 读过 P1.1（35 年史） 主要来源 Jim Roskind 2013 QUIC 设计文档 + IETF QUIC WG charter + Mark Nottingham（前 IETF httpbis WG chair）的多次演讲与博文 + 《HTTP/3 Explained》第 2-3 章

01 速览

  1. HTTP 35 年的核心动力不是"让协议更快"，是让协议能继续演化。每一代被推出来都是因为前一代撞到了"演化天花板"。
  2. 第一代抽象 · 消息（HTTP/1.x）：一个 TCP 连接里串行发请求/响应消息。keep-alive 让连接复用，但同时只能跑一个消息。慢请求堵后面所有请求 —— 应用层 HOL blocking。
  3. 第二代抽象 · 帧 + 流（HTTP/2）：一个 TCP 连接里跑几百几千个虚拟 stream，每个 stream 是消息序列；wire 上是 frame 序列，按 stream id 归组。解了应用层 HOL，但 TCP 字节流仍是有序的 → 传输层 HOL blocking 仍在。
  4. 第三代抽象 · 用户态传输层（QUIC + HTTP/3）：放弃 TCP，在 UDP 上自己实现"可靠+有序+加密+多路独立"传输层。每条 stream 有独立丢包恢复 —— 真正消灭 HOL blocking。
  5. 第四代是减法（HTTP/3）：HTTP/2 在 TCP 上打的几乎所有补丁（streams 抽象 / 流控 / RST_STREAM / WINDOW_UPDATE）扔回到 QUIC；HTTP/3 frame 只剩 DATA / HEADERS 等少数几种。HTTP/3 是 HTTP 协议自身的简化。
  6. 5 个核心设计决策：加密默认（中间盒看不见就不能破坏）/ 用户态可演化（不依赖 OS 内核升级）/ 每 stream 独立（放弃"一个有序字节流"幻觉）/ 中间盒防御（用 UDP 不用新协议号 + greasing）/ TLS 1.3 内嵌（握手压成 1 RTT，0-RTT 重连免 RTT）。
  7. "为什么不改 TCP"有三个根本理由：TCP 在OS 内核态（升级需要换 OS），中间盒已经"理解"TCP 字节序号（改 TCP 字节序就让一大批中间盒挂掉），TCP 拥塞控制30 年没大改（cubic / BBR / BBRv3 想推都太慢）。QUIC 在用户态 + UDP 直接绕开三个问题。
  8. 2026 协议设计新原则："加密默认 + 用户态 + 可演化"。这条原则不只用于 HTTP/3，未来的传输层创新都会沿着它来。HTTP/3 是它的第一个完整应用，不会是最后一个。

02 底层逻辑

第一代 · 消息抽象（HTTP/1.x）

HTTP/1.x 的核心抽象是 消息（message）：一个请求消息 + 一个响应消息构成一个交互单位。一个 TCP 连接里可以串行跑多个消息（keep-alive），但同一时刻只能有一个消息在传。

为什么这样设计？1991 年的 web 一个页面只有 1-3 个资源，串行就够了。1996 年加 keep-alive 已经是工程妥协 —— 那时一个页面可能 10+ 张图片，每张都重建 TCP 显然不行。

但当 web 进入 2000 年代，一个页面 50+ 资源、CDN 引入、第三方 script / 字体 / 图片爆炸增长，"同一连接同时只能跑一个消息"成为根本性瓶颈。这是应用层 HOL blocking —— 一个慢响应堵住后面所有响应。

当时浏览器的解决办法：开 6 个并发 TCP 连接。每个 TCP 连接独立跑 HTTP/1.1 keep-alive。再不够就domain sharding（拆 img1.example.com / img2.example.com / ... 因为 6 connection limit 是按 origin 计的）。

这种 workaround 在 wire 层是浪费的（每个连接独立握手 / 拥塞控制窗口起步小 / TLS 重协商），但是协议本身没办法。需要换抽象层。

第二代 · 帧 + 流抽象（HTTP/2 / SPDY）

SPDY → HTTP/2 把抽象层换成 frame + stream：

  • Frame：wire 上的最小单位。每帧有 type / flags / stream id / payload。一个 TCP 连接上 frame 无序穿插，按 stream id 归组。
  • Stream：虚拟逻辑流。一个 HTTP 请求 + 响应 = 一个 stream（HEADERS frame → 多个 DATA frame → 可选 trailers）。

这一层抽象的工程意义：多路复用。一个 TCP 连接能并发承载几百到几千 stream。浏览器不再需要开 6 个连接 / domain sharding 这套 workaround；CDN 也不用拆子域名。

但 HTTP/2 仍跑在 TCP 上。TCP 是字节流（byte stream），它假设它传的所有数据是一条有序的字节流。任何一个 IP packet 丢了，TCP 必须等到重传完成、按序交付，才能往应用层送字节。

所以即使你的 HTTP/2 流量是 100 个独立 streams，wire 上是一条 TCP 字节流。一个 packet 丢 → 整条 TCP 流 stall → 100 个 streams 全都看不到新字节。这就是 TCP HOL blocking：

⚠ 关键洞察

HTTP/2 多路复用是语义层的多路复用，wire 层仍是单路。这不是 HTTP/2 实现的 bug，是 TCP 这个底层不愿意理解"我有多个独立流"。Robin Marx 2020 年的实测显示：3% 丢包率下，HTTP/2（一个连接）比 HTTP/1.1（6 个连接）慢 —— 因为 1.1 的 6 个连接里只有一个被 HOL block 影响，另 5 个仍能跑。

这是 HTTP/2 的根本性局限，不能在 TCP 上修复。需要换传输层。

第三代 · 用户态传输层抽象（QUIC + HTTP/3）

QUIC 的根本性创新是放弃"在一个有序字节流上多路复用"的幻觉，承认网络本来就是丢包 + 乱序的：

  • QUIC 不再是"一条字节流"，是多条独立的 stream。每条 stream 有自己的有序保证。
  • 每条 stream 有独立的丢包恢复。stream A 丢包不影响 stream B。
  • QUIC 跑在 UDP 上。UDP 不假设有序、不假设可靠 —— 留给 QUIC 自己去做。
  • 所有 QUIC 实现在用户态（浏览器 / CDN 进程内），不依赖 OS 内核。

这层抽象升级后，"多路复用"在传输层就是真的多路 —— 不再有 HOL blocking。

但 UDP 自己什么保证都没给。QUIC 必须自己在 UDP 上重新实现：

  • 包丢失检测（packet number space）
  • 重传与拥塞控制（loss recovery + congestion control，RFC 9002）
  • 流控（stream + connection 两层 flow control）
  • 加密（TLS 1.3 内嵌握手 + AEAD packet protection）
  • 连接生命周期（握手 / 关闭 / 迁移）

QUIC 是"一个完整的传输层"，不是"UDP 的一层包装"。把 QUIC 当成"快速 UDP"是 2026 仍很普遍的误解。

第四代 · HTTP/3 是减法

QUIC 把"多路独立 stream + 流控 + 丢包恢复"全做了。HTTP/3（RFC 9114）就把 HTTP/2 在 TCP 上打的补丁全扔掉：

HTTP/2 帧 HTTP/3 是否保留 不保留的原因
HEADERS / DATA ✅ 保留 HTTP 语义层需要
SETTINGS ✅ 保留 连接级参数协商
STREAM 概念 ❌ 移到 QUIC QUIC 已经有 stream 抽象
WINDOW_UPDATE ❌ 移到 QUIC QUIC 流控代替
RST_STREAM ❌ 移到 QUIC QUIC 的 RESET_STREAM
PING ❌ 移到 QUIC QUIC PING frame
PUSH_PROMISE ⚠ 保留 wire 但不用 Server Push 已废弃
PRIORITY ❌ 移除 RFC 9218 改用 Priority 头

HTTP/3 的 wire 比 HTTP/2 简单很多。这就是为什么"HTTP/3 是减法 + QUIC 是加法"这句话准确：HTTP/3 自身的协议复杂度其实下降了。

同时 HPACK 不能再用 —— HPACK 假设 header block 按 stream 顺序到达，因为它的动态表更新是顺序的。在 QUIC 上 streams 乱序到达，HPACK 解码器看到一个未定义的索引时不知道该等还是该报错。

QPACK（RFC 9204）的解决方法：用独立的 encoder stream（单向）发送动态表更新，独立的 decoder stream（单向）回 ack。每个 header block 引用动态表时，可以"等" encoder stream 上的对应更新到达。详见 P4.2。

5 个核心设计决策（QUIC + HTTP/3）

1. 加密默认 —— 中间盒看不见就不能破坏

TCP 的字节序号、拥塞控制窗口、option 字段等等，被 30 年沉淀的中间盒（NAT / 透明代理 / "智能"防火墙 / 运营商 LB / 公司 firewall）"理解"了。任何对 TCP 的修改都可能让某些中间盒挂掉。

这种现象叫 protocol ossification（协议固化）。TCP 已经被中间盒固化到无法演化的程度。

QUIC 的解法：除了几个握手字段，QUIC 包内容全部加密。中间盒看不到 packet 内部 → 不能"理解"任何字段 → 没办法基于内容做决策 → 协议可以自由演化。

这是 QUIC 没有明文模式的真实原因。不是"安全更好"那么浅，是用加密对抗中间盒固化。

2. 用户态可演化 —— 不依赖 OS 升级

TCP 是 OS 内核态实现。改 TCP 拥塞控制（比如从 cubic 换 BBR）需要 Linux 内核 4.9+；改 TCP option 需要内核更深的修改；新协议号通常需要内核支持。

结果：BBR 2016 年 Google 出，到 Linux 主线 4.9（2016-12 发布）才进来。再到大部分服务器升级到能用 BBR 的内核，又是 2-3 年。TCP 的演化速度被 OS 升级速度卡住。

QUIC 在用户态：拥塞控制算法、流控参数、loss detection、握手都在浏览器 / CDN 的进程里。换算法只需要 release 新版本：Cloudflare 把 BBRv1 → BBRv2 → BBRv3 的升级都做完了，不用等 Linux 内核。

3. 每 stream 独立 —— 放弃"有序字节流"幻觉

TCP 的核心抽象是"有序、可靠的字节流"。这是 1980 年代基于"应用程序需要简单 API"的合理设计。

但 web 应用不需要"一条字节流"。web 应用需要"多条独立的消息流"。强行把多条独立流塞进一条字节流，就有了 HOL blocking 这个无解问题。

QUIC 的抽象是"多条独立有序的 streams"。每条 stream 自己有序、自己丢包恢复。stream 之间互不影响。这才匹配 web 应用的真实需求。

4. 中间盒防御 —— 用 UDP 不用新协议号 + greasing

新建一个 IP 协议号（比如 IPPROTO_QUIC = 17 类似的常数）需要让所有路由器 / 防火墙 / NAT 设备认识它。这条路在 2020 年代不现实：上百万的中间盒会丢掉它们不认识的协议号包。

QUIC 选择走 UDP（IPPROTO_UDP = 17，所有中间盒都认得）。这样 IP 层中间盒看不出 QUIC 和普通 UDP 的差别。

另外 QUIC 用 greasing（RFC 8701 等借鉴的思想）：在握手里塞一些"故意不被识别"的字段，强制中间盒/服务端实现"忽略不认识的字段"而不是"假设这些字段不会出现"。这是从 TLS 学的教训 —— 防止未来的 QUIC 演化被中间盒固化。

5. TLS 1.3 内嵌 —— 握手压成 1 RTT，0-RTT 重连免 RTT

HTTP/2 over TLS 的连接建立要：

  1. TCP 三次握手：1 RTT
  2. TLS 1.3 握手：1 RTT（TLS 1.2 是 2 RTT）
  3. HTTP/2 SETTINGS 交换：通常和 HTTP 请求 piggyback，0 额外 RTT

所以一个新 HTTPS 连接发出第一个请求大约要 2 RTT。用户在跨海光缆延迟 200ms 的链路上，发请求前要等 400ms。

QUIC 把 TLS 1.3 握手内嵌进 QUIC 握手（RFC 9001）：transport 参数 + TLS handshake 在同一组 packet 里完成。一个新 QUIC 连接 1 RTT 就能发第一个请求。

更进一步，0-RTT：第二次连同一 server 时，client 用上次的 PSK（pre-shared key）直接发 application data。0 个 RTT 启动。代价是 0-RTT 数据没有前向保密、可被回放（详见 P4.1），所以浏览器只对 GET 启用。

在跨海链路上，1 RTT vs 2 RTT 是 200ms 的差别。0-RTT 更彻底。QUIC 的握手优化主要是在跨国 / 移动网络环境下生效。

为什么不直接改 TCP

这是每次讲 QUIC 都要回答的问题。三个根本理由：

  1. TCP 在 OS 内核态。升级 TCP 协议要：内核更新 → 服务器升级 → CDN 升级 → 终端 OS 升级（Windows / macOS / iOS / Android 各自的节奏）。这条链上任何一个环节卡住，新功能就用不上。Web 不能等。
  2. TCP 字节序号已被中间盒固化。改 TCP 字节序、改 option 字段、加新 flag 位 —— 任何一个都可能让一批中间盒挂掉。BBR / TCP Fast Open 这些"小改"已经踩过这个坑（TCP Fast Open 因为中间盒兼容性几年都没普及）。
  3. TCP 加密无处可放。即使你想给 TCP 加默认加密（防中间盒），TCP header 本身就在中间盒可见。要改成默认加密 → 改字节序 → 一批中间盒挂掉。死循环。

所以唯一出路是跳过 TCP，重起一个新协议。QUIC 就是这条路。

03 应用场景：三代抽象在 2026 的真实分工

场景 A · 消息抽象（HTTP/1.x）：调试 + 老系统 + curl 一行

消息抽象的核心优点是人类可读。telnet 一行调通、curl -v 直接看 wire、shell 脚本 grep 一下。这是 HTTP/1.1 在工具链 / 调试 / 内网保留 25-30% 流量的根因。当你只需要 1-2 个请求 + 不在乎并发 + 想要可读 wire，HTTP/1.1 仍是最好选择。

场景 B · 帧+流抽象（HTTP/2）：CDN ↔ origin + gRPC + 数据中心内部

HTTP/2 的多路复用 + HPACK 在低延迟 + 低丢包的网络上是最优的。CDN ↔ origin、Kubernetes 内部服务通信、gRPC RPC 框架都默认 HTTP/2。这些场景里 TCP HOL blocking 几乎不出现，QUIC 的优势体现不出来；同时 HTTP/2 over TCP 的实现更稳定、调试工具更成熟。

场景 C · 用户态传输层抽象（QUIC + HTTP/3）：浏览器 ↔ CDN + 移动网络

QUIC 的优势位是多变网络条件 + 高 RTT + 移动场景。浏览器到 CDN 这一段，用户可能在 4G / wifi / 跨国光缆上，QUIC 的连接迁移 / 0-RTT / 每 stream 独立丢包恢复全派上用场。所以 2026 年浏览器 ↔ CDN 这一段 HTTP/3 普及率 70%，是 QUIC 优势体现最明显的位置。

场景 D · 极致低延迟 / 自定义协议：直接用 QUIC（不走 HTTP）

QUIC 的 transport 层是通用的，不绑定 HTTP。WebTransport（详见 P6.2）让浏览器 JS 直接用 QUIC stream + datagram；服务端也可以基于 quiche / msquic / aioquic 跑自定义应用层协议。"在 QUIC 上跑非 HTTP 应用"是 2026 起的新工程方向。

第三代：QUIC + HTTP/3（用户态传输层）

text 
一条 UDP datagram 流，多条独立 stream：

UDP packets (each independently lost/recovered):
  [pkt 100: stream 1 frame] → arrives
  [pkt 101: stream 3 frame] → LOST! (stream 3 重传)
  [pkt 102: stream 5 frame] → arrives  ← 不等 pkt 101！
  [pkt 103: stream 1 frame] → arrives  ← 不等 pkt 101！

  每条 stream 有独立的丢包恢复和有序保证：
    stream 1: 持续推进 (没受 pkt 101 丢失影响)
    stream 5: 持续推进 (没受 pkt 101 丢失影响)
    stream 3: 等 pkt 101 重传，自己 stall (但只 stall 自己)

  ✅ 真正的多路独立 → 没有 HOL blocking
  ✅ 加密内嵌 → 中间盒看不见就不能破坏
  ✅ 用户态 → 升级算法不用等 OS 内核

  HTTP request:     1 RTT (200ms)

  首请求:           2 RTT = 400ms
  0-RTT 重连:       1 RTT = 200ms (含数据)

05 常见陷阱

陷阱 1 · 把 QUIC 当成"快速 UDP"

QUIC 不是 UDP 的简单包装，是在 UDP 上重新实现的完整传输层。它有自己的拥塞控制、丢包恢复、流控、加密握手。把 QUIC 等同 UDP 会让你低估它的实现复杂度（一个生产级 QUIC 实现至少 5-10 万行代码）。

陷阱 2 · 误以为 HTTP/3 是 HTTP/2 的"快速版"

HTTP/3 的"快"主要来自底下的 QUIC，不是 HTTP/3 自身。HTTP/3 frame 比 HTTP/2 还简单（减法）。在低延迟低丢包数据中心里，HTTP/3 可能没有比 HTTP/2 快。优势位是高 RTT 多变网络。

陷阱 3 · 想给 TCP 加默认加密

历史上有过几次尝试（tcpcrypt 等），都没成。理由就是设计哲学这一节讲的：TCP header 本身在中间盒可见 → 改 TCP 加密会让一批中间盒挂掉。"不能改 TCP"是 QUIC 存在的根本理由，不是"觉得 TCP 不够好"。

陷阱 4 · 误以为加密"主要是为了安全"

QUIC 的强制加密在工程哲学层面主要是为了对抗中间盒固化，安全是副产品。这不是说安全不重要，是说"加密 = 防中间盒"这个动机一旦理解，QUIC 的很多设计选择（比如所有包内容加密、传输层参数都加密、版本协商专门处理）就都说得通了。

陷阱 5 · 期待"以后还会有新协议每隔 7 年来一次"

HTTP/0.9 → 1.0 → 1.1 → 2 → 3 看似每隔 7-10 年换一代，但QUIC 是协议演化机制本身的革命。QUIC 在用户态、加密内嵌、可演化，未来的传输层创新可以直接在 QUIC 框架内做版本升级，不需要再发明 "QUIC v2 / HTTP/4"这样的全新协议。当前 IETF 在 QUIC v2（draft）讨论的就是这个思路 —— 在 QUIC 框架内演化，不重起新协议。

陷阱 6 · 把 0-RTT 当成"普遍加速"

0-RTT 数据没有前向保密、可被回放。只能用于幂等请求（GET）。浏览器和 CDN 都默认对 POST 关闭 0-RTT。如果你部署 QUIC 时把 0-RTT 全开，可能引入 replay attack 风险（CSRF-like）。详见 P4.1。
