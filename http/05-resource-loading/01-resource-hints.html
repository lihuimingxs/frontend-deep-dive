资源提示家族 + HTTP/2 优先级 + fetchpriority + RFC 9218

每天用 preload / preconnect / prefetch，但说不清谁负责 DNS、谁建连、谁下载、谁影响优先级 —— 这一章把整个资源提示家族按"激进度阶梯"串起来：dns-prefetch → preconnect → prefetch → preload → modulepreload → Speculation Rules。同时讲 fetchpriority API（2023 起新加）和 RFC 9218 Priority 头的关系，给 LCP 优化一条工程闭环。

阅读时长 ~22 分钟 前置 用过 fetch / 写过 link 标签 主要来源 W3C Resource Hints + W3C Speculation Rules + HTML 标准 fetchpriority + RFC 9218 + Chrome 团队博客 + web.dev 性能指南

01 速览

  1. 资源提示激进度阶梯：dns-prefetch（只 DNS）→ preconnect（DNS + TCP + TLS）→ prefetch（低优先级下载下次用）→ preload（高优先级当前用）→ modulepreload（preload 的 JS 变种带依赖图）→ Speculation Rules API（prerender / 跨页面 prefetch）。
  2. dns-prefetch：<link rel="dns-prefetch" href="//cdn.example.com"> 提前解析 DNS。最便宜，几乎免费。仅对未来要用的第三方域名有意义。
  3. preconnect：<link rel="preconnect" href="https://cdn.example.com"> DNS + TCP + TLS 全做（不发请求）。预热第三方 origin。
  4. prefetch：低优先级下载到 cache 给下次导航用；浏览器实现差异大（Chrome 实现质量最好）。
  5. preload：高优先级强制下载当前页面用。必须指定 as="..."，否则浏览器 fetch 两次（一次 preload + 一次正常请求 → 拿不到 cache 命中）。
  6. modulepreload：preload 的 JS module 专用，浏览器会顺带解析 import 依赖图（import 'foo' 引用的子模块也预热）。
  7. Speculation Rules API（2024 稳定）：JSON-based 配置 prerender / prefetch；替代过时的 <link rel="prerender">。Chrome 121+ 默认开。
  8. fetchpriority API（2023 起）：<img fetchpriority="high"> / fetch(url, { priority: 'high' })。浏览器自动转换为 RFC 9218 Priority 头发给 server。LCP 优化最直接的工具。
  9. RFC 9218 priority signaling：HTTP header Priority: u=3, i（urgency 0-7 + incremental）。CDN（Cloudflare / Fastly）支持转发并按它调度 stream。HTTP/2 / HTTP/3 都用同一套。
  10. 2026 实现现状：Chromium 系完整支持所有提示 + fetchpriority + Speculation Rules；Firefox / Safari 部分支持（Speculation Rules / fetchpriority Safari 17.5+）。

02 底层逻辑

资源提示激进度阶梯

text 
激进度由低到高：

dns-prefetch
  ↓ 只做 DNS 解析
  ↓ 几乎免费 (1-3 KB DNS query)
  ↓ 适用：未来 5-30s 内可能要的第三方域名

preconnect
  ↓ DNS + TCP 三次握手 + TLS 1.3 握手 (不发请求)
  ↓ 成本：1-2 RTT 延迟 + 一个空闲连接
  ↓ 适用：5-15s 内一定要发请求的第三方 origin

prefetch
  ↓ 低优先级下载到 HTTP cache，等下次导航用
  ↓ 成本：占带宽 + 占 cache
  ↓ 适用：用户大概率要去的下一页 (产品详情页 hover)

preload
  ↓ 高优先级下载，当前页要立即用
  ↓ 必须指定 as="image" / "style" / "script" / "font" / ...
  ↓ 成本：抢占主资源带宽
  ↓ 适用：LCP 关键资源 / 首屏字体 / 关键 CSS

modulepreload
  ↓ preload 的 JS module 变种 + 自动解析 import 图
  ↓ 适用：ES module 入口 (现代 SPA / Vite / Rollup 输出)

Speculation Rules API
  ↓ prefetch / prerender 整个目标页面
  ↓ 浏览器自己决定置信度阈值
  ↓ 适用：明确知道用户下一步去哪 (导航主入口)

dns-prefetch · 几乎免费的 DNS 预热

html 
<!-- 在 <head> 里 -->
<link rel="dns-prefetch" href="//fonts.googleapis.com" />
<link rel="dns-prefetch" href="//www.google-analytics.com" />
<link rel="dns-prefetch" href="//cdn.example.com" />

浏览器收到 dns-prefetch 后会异步发 DNS query。下次需要这些域名的资源时，DNS 已经在 cache 里。

注意：preconnect 已经包含 DNS 解析，所以 preconnect 同一域名时 dns-prefetch 是冗余。但实践中两个一起写作 fallback —— 老浏览器 / 不支持 preconnect 时退化到 dns-prefetch。

preconnect · 提前打开第三方连接

html 
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
<link rel="preconnect" href="https://api.example.com" />

关键点：

  • 不发 HTTP 请求，只建立 TCP + TLS 连接 + ALPN 协商
  • 有 idle timeout：浏览器只保持几秒空闲连接，超时关闭。所以 preconnect 必须紧接着用 —— 提前 30 秒 preconnect 没意义
  • crossorigin 属性：preconnect 字体 / fetch CORS 资源时必须加，否则连接没用 anonymous credentials mode 的资源还是要重建连接

preconnect 的 1-2 RTT "省"在第三方资源加载首次发请求时体现：DNS + TCP + TLS 都已完成，真正的 GET 是 1 RTT 直接发出。

prefetch · 下次导航用

html 
<!-- 用户大概率会点 /pricing 这个链接，提前下 -->
<link rel="prefetch" href="/pricing" as="document" />
<link rel="prefetch" href="/api/pricing-data" as="fetch" crossorigin />
<link rel="prefetch" href="/pricing.chunk.js" as="script" />

注意：

  • 低优先级：浏览器只在主页面 idle 时才下
  • 下完进 HTTP cache：下次导航时被命中（必须 cache headers 允许）
  • 浏览器实现差异大：Chrome 实现质量最高；Firefox / Safari 实现节奏不同；某些场景跨 origin prefetch 被忽略（CORS / cookie 策略）
  • 2026 渐被 Speculation Rules 替代 —— 后者更精细控制

preload · 当前页高优先级

html 
<!-- LCP 关键图：不等 HTML 解析就下载 -->
<link rel="preload" as="image" href="/hero.jpg" fetchpriority="high" />

<!-- 关键 CSS -->
<link rel="preload" as="style" href="/critical.css" />
<link rel="stylesheet" href="/critical.css" />

<!-- 字体（必须 crossorigin） -->
<link rel="preload" as="font" type="font/woff2" href="/fonts/regular.woff2" crossorigin />

关键点：

  • as 属性必须指定。原因：浏览器不知道资源类型时会按"通用"下载，无法应用类型相关的优先级 / cache key 规则。漏掉 as → 浏览器 fetch 两次（一次 preload + 一次主请求），preload 失效
  • preload 抢占主请求带宽：preload 同时太多反而让主资源慢
  • preload 的资源进 memory cache + HTTP cache，主请求用同一资源时直接命中
  • 2026 起 LCP 关键资源加 fetchpriority="high" 比单纯 preload 更精确

modulepreload · ES module 专用

html 
<link rel="modulepreload" href="/app.mjs" />

<!-- 浏览器解析 /app.mjs 找到它 import 的子模块 -->
<!-- 自动 modulepreload 这些子模块（深度解析） -->
<!-- 等用户真正 import('./app.mjs') 时整个图已就绪 -->

这是 ES module 时代的 preload 优化。Vite / Rollup 等 bundler 会自动注入 modulepreload 链接给 lazy chunk。

Speculation Rules API · 2024 起替代 prerender

html 
<script type="speculationrules">
{
  "prerender": [
    {
      "source": "list",
      "urls": ["/pricing", "/about"]
    },
    {
      "where": { "selector_matches": "a.likely-next" },
      "eagerness": "moderate"
    }
  ],
  "prefetch": [
    {
      "where": { "href_matches": "/products/*" },
      "eagerness": "conservative"
    }
  ]
}
</script>

关键点：

  • prerender：浏览器在后台完整渲染目标页面（执行 JS、应用 CSS、生成 DOM）。用户点击时瞬间切换，几乎 0ms LCP
  • prefetch：只下载资源，不渲染（类似旧 <link rel="prefetch">，但更精细的控制）
  • eagerness：immediate / eager / moderate / conservative，控制浏览器何时触发
  • Chrome 121+ 默认开启；Firefox / Safari 仍在实现中
  • 替代旧 <link rel="prerender">：旧 API 仅 prefetch，2024 已 deprecated

fetchpriority API · LCP 优化的杀器

2023 年加入 HTML 标准，2024 在 Chromium / Safari / Firefox 全部稳定。

三种用法

html 
<!-- 1. img / link 标签上 -->
<img src="/hero.jpg" fetchpriority="high" />
<img src="/below-fold.jpg" fetchpriority="low" />
<link rel="stylesheet" href="/non-critical.css" fetchpriority="low" />

<!-- 2. JS fetch() -->
fetch('/api/critical', { priority: 'high' });
fetch('/api/analytics', { priority: 'low' });

<!-- 3. preload 配合 -->
<link rel="preload" as="image" href="/hero.jpg" fetchpriority="high" />

三个值的语义

  • high：覆盖浏览器默认，提到 high。用于 LCP 关键资源
  • low：降级到 low。用于 below-the-fold 图、analytics、非关键 CSS
  • auto（默认）：浏览器按资源类型默认

映射到 RFC 9218 Priority 头

浏览器根据 fetchpriority + 资源类型自动设置 HTTP 请求的 Priority 头：

资源类型 / fetchpriority auto (默认) high low
HTML u=2, i u=1, i u=3, i
CSS（render-blocking） u=2 u=1 u=3
JS（async / defer） u=3 u=2 u=4
img（above the fold） u=3, i u=2, i u=4, i
img（below the fold） u=4, i u=2, i u=5, i
fetch (default) u=3 u=1 u=4

RFC 9218 Priority 头

RFC 9218 定义了 HTTP header Priority: u=3, i：

  • u=N：urgency，0-7（0 最紧急，默认 3）
  • i：incremental，表示资源可以增量利用（progressive image / streaming JSON）

HTTP/2 + HTTP/3 都用这套，替代 HTTP/2 失败的 prioritization tree。CDN（Cloudflare / Fastly）2026 已支持转发并按 priority 调度 stream 顺序。

fetchpriority + LCP 实测

web.dev 2024 实测数据：

  • 给 LCP 图加 fetchpriority="high"：LCP 改善 中位数 9%、p75 改善 18%
  • 这是 2024 起单点优化对 LCP 影响最大的一招（不需要改业务代码、不需要后端配合）

03 应用场景

场景 A · LCP 优化：fetchpriority + preload 组合

典型 e-commerce 产品详情页：hero image 是 LCP 元素。组合姿势：(1) <link rel="preload" as="image" href="/hero.jpg" fetchpriority="high"> 在 head 提前下；(2) <img src="/hero.jpg" fetchpriority="high"> 渲染时复用 preload 的 cache；(3) below-fold 图加 fetchpriority="low" loading="lazy"。这套组合 LCP 改善 15-30%。

场景 B · 第三方 origin 预热

页面要用 Google Fonts + Stripe checkout SDK + Sentry SDK：head 里 preconnect 这三个 origin。每个省 1-2 RTT。注意 fonts 等需要 CORS 凭据的加 crossorigin。

场景 C · 路由切换预热

用户在产品列表页 hover 某个商品 → 在 hover 事件里 prefetch 商品详情页 + API 数据。用户点击时大概率 cache 命中，瞬间切换。Next.js / TanStack Router 都内置这套。详见 TanStack Query P4.2。

场景 D · Speculation Rules prerender 大幅 LCP

新闻类站点：用户点首页大头条 → prerender 大头条页面。用户真正点击时已渲染完成，LCP 几乎 0ms。eagerness=moderate（hover 时触发）/ conservative（更保守，避免浪费）。

场景 E · ES module 应用预加载

Vite 输出的 SPA：每个 route 一个 chunk，bundler 自动注入 modulepreload 给当前路由可能跳的下一路由 chunk。开发者不用手动写。但生产配置可以加 prefetch / Speculation Rules 进一步优化。

04 典型代码：完整 LCP 优化案例

电商首页 head 完整组合

html 
<!doctype html>
<html>
<head>
  <meta charset="utf-8" />

  <!-- 第三方 origin 预热 -->
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
  <link rel="preconnect" href="https://api.example.com" />
  <link rel="dns-prefetch" href="//www.google-analytics.com" />

  <!-- 关键 CSS preload + stylesheet 配对 -->
  <link rel="preload" as="style" href="/critical.css" />
  <link rel="stylesheet" href="/critical.css" />

  <!-- 字体（CORS 必须） -->
  <link rel="preload" as="font" type="font/woff2"
        href="/fonts/inter-v12-latin-regular.woff2" crossorigin />

  <!-- LCP hero 图：preload + fetchpriority high -->
  <link rel="preload" as="image" href="/hero.webp"
        type="image/webp" fetchpriority="high" />

  <!-- modulepreload JS module -->
  <link rel="modulepreload" href="/app.mjs" />

  <!-- Speculation Rules: hover 商品时 prefetch 详情 -->
  <script type="speculationrules">
  {
    "prefetch": [{
      "where": { "href_matches": "/product/*" },
      "eagerness": "moderate"
    }]
  }
  </script>
</head>
<body>
  <!-- LCP 元素：fetchpriority high + 同 src 配 preload -->
  <img src="/hero.webp" alt="..." fetchpriority="high"
       width="1200" height="600" />

  <!-- below the fold：fetchpriority low + lazy -->
  <img src="/below.webp" alt="..." fetchpriority="low" loading="lazy" />

  <!-- 主 JS：modulepreload 已经预热，import 时 cache 命中 -->
  <script type="module" src="/app.mjs"></script>
</body>
</html>

05 常见陷阱

陷阱 1 · preload 滥用：什么都标 high

把所有资源都 preload + fetchpriority="high"，浏览器无法合理排序。LCP 反而更慢（关键资源被次要资源挤）。原则：每页 preload 最多 3-5 个资源，只标真正的 LCP 关键资源。

陷阱 2 · preload 漏写 as 属性

<link rel="preload" href="/hero.jpg"> 没 as="image"。结果：浏览器不知道类型 → 不知道哪个 cache key 用 → 主请求来时不命中 → 下了两次。preload 必须有 as。

陷阱 3 · 字体 preload 没 crossorigin

字体在 CSS 里 fetch 时浏览器自动加 CORS（anonymous），但 preload 默认不加。结果：preload 进了"普通 cache"，CSS fetch 走"CORS cache"，不命中。字体 preload 必须加 crossorigin。

陷阱 4 · prefetch 跨 origin 没 cookie 处理

跨 origin prefetch 时，浏览器策略各异。某些场景 prefetch 不带 cookie → 实际导航时仍要重新拉。Chrome 2024 起改进，但 Firefox / Safari 仍旧。跨 origin prefetch 不要假设一定生效；用 Speculation Rules 更可控。

陷阱 5 · preconnect 太早 / 用不上就 idle 关

preconnect 后浏览器空闲连接 5-10 秒就关。如果你 preconnect 30 秒前用不到的 origin，连接已被关闭，preconnect 浪费。preconnect 要紧接着用，否则不如不写。

陷阱 6 · fetchpriority 用了但 server 不识别 Priority 头

fetchpriority 在浏览器侧已经影响调度（请求顺序、并发数），即使 server 不识别 Priority 头也有效果。但如果你的 server / CDN 也支持 Priority 头（Cloudflare / Fastly 已支持），server 端调度也会按 priority。如果你自建 origin 不识别 Priority 头，效果只在浏览器侧。

陷阱 7 · Speculation Rules prerender 触发太多

prerender 整个页面意味着下完整 HTML + 执行所有 JS + 应用 CSS + 跑分析脚本。如果你 eagerness=immediate 同时 prerender 5 个页面，等于在用户后台跑 5 个完整 web app。CPU / 带宽 / 内存全爆。用 conservative / moderate，让浏览器自己决定置信度。

陷阱 8 · 仍在用 <link rel="prerender">

旧 prerender 已 deprecated（2024）。Chrome 121+ 走 Speculation Rules。移除 <link rel="prerender">，改 <script type="speculationrules">。
