P5 · 资源加载工程

前端工程师每天都在用 preload / preconnect / prefetch 这堆"资源提示"，但说不清谁负责 DNS、谁建连、谁下载、激进度阶梯长什么样。这一阶段把整个家族 + RFC 9218 优先级信号 + 2024 起新加的 fetchpriority API 串起来；同时把 Server Push 之死和 103 Early Hints 怎么"接班"用 Cloudflare/Vercel 实测数据讲清。

章节数 2 主要来源 RFC 9218 (priority signaling) + RFC 8297 (103 Early Hints) + W3C Resource Hints + W3C Speculation Rules + HTML 标准 fetchpriority + Cloudflare/Vercel/Fastly 实测博客 + Chrome 团队博客 状态 ✅ 全部完成

本阶段覆盖

  • 资源提示激进度阶梯 · dns-prefetch → preconnect → prefetch → preload → modulepreload → Speculation Rules
  • fetchpriority API（2023 起） · LCP 关键资源标 high · 浏览器映射到 RFC 9218 Priority 头
  • RFC 9218 优先级信号 · urgency 0-7 + incremental · CDN 转发支持现状
  • preload 滥用诊断 · 没 as / 跨 origin / 全标 high 等典型反模式
  • Server Push 之死 · 2017-2020 Chrome 实测数据 · 2022 全面退场
  • 103 Early Hints · server 处理前发 preload 信号 · 浏览器主动决定要不要下 · cache-aware
  • Cloudflare 实测 · backend 慢场景 LCP 减少 20-30%
  • 现代替代 Server Push 的完整方案 · preload + fetchpriority + Early Hints + Speculation Rules 组合姿势
