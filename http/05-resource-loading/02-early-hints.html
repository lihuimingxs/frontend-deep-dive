Server Push 之死 + 103 Early Hints + 现代替代方案

Server Push 是 HTTP/2 时代最大的失败实验。2015 标准化、2017-2020 实测伤性能、2022 RFC 9113 不再推荐、2026 全面退场。接班的是 103 Early Hints（RFC 8297）—— server 在最终响应前先发 preload 信号，浏览器自己决定要不要下。Cloudflare / Vercel 实测：在 backend 慢的场景下 LCP 减少 20-30%。

阅读时长 ~18 分钟 前置 读过 P3.2（Server Push 兴衰）+ P5.1（资源提示家族） 主要来源 RFC 8297（103 Early Hints）+ RFC 9113 §8.4（Server Push 不再推荐）+ Cloudflare 2022 Early Hints 实测博客 + Vercel Next.js 16 文档 + Chrome team 退役 Server Push 的官方说明

01 速览

  1. Server Push 已经死了。2022 RFC 9113 §8.4 明确不再推荐；2022 Chrome 默认禁用；2023 Cloudflare / Fastly / Akamai 移除；2026 几乎所有 server 已停用；Web Almanac 2025 启用率 < 1%。遇到老配置直接删。
  2. Server Push 失败 4 个根本理由：(a) 浏览器 cache 协调失败（server 不知道 client 已 cache 什么）；(b) 浏览器实现 bug 多；(c) push 占 server 上行带宽挤压主请求；(d) Chrome 2017-2020 production 实测显示伤性能。
  3. 103 Early Hints（RFC 8297, 2017）：HTTP 1xx 系列状态码，server 在最终 200 响应前先发一个 103 + Link: ...; rel=preload 头。浏览器立刻开始 preload 这些资源。
  4. 关键差异 vs Server Push：浏览器主动决定要不要下；如果 cache 已有就不下 → 不会浪费带宽。
  5. 关键差异 vs 普通 preload：可以在 server 还在处理 backend / 数据库时就发 preload 信号 → 节省 server 处理时间这一段。
  6. Cloudflare 2022 实测：在 backend 慢的场景（首字节时间 500ms+）下，103 Early Hints LCP 减少 20-30%。
  7. Vercel + Next.js 16 集成：Server Component 渲染时可以发 Early Hints（response.writeEarlyHints() 或框架封装）。Next.js 16 起部分场景自动发。
  8. 现代替代 Server Push 的完整方案：(1) 首屏关键资源 → preload + fetchpriority="high" 内联在 head；(2) backend 慢 → 103 Early Hints 在 backend 处理前发 preload 信号；(3) 路由切换预热 → Speculation Rules API。
  9. 2026 启用率：103 Early Hints 约 8% 站点启用（2022 还只是 1%），Cloudflare / Vercel 默认开 → 接下来 2-3 年会快速普及。

02 底层逻辑

Server Push 复盘：失败的工程现实

这一节是 P3.2 的延伸：Server Push 死的具体过程。

设计意图（2009-2015）

HTML 解析串行：浏览器收 HTML → 解析到 <link rel="stylesheet"> → 才发 CSS 请求。这一段"等"是浪费。

Server Push 解决：server 知道 index.html 必然依赖 style.css，发完 HTML 顺手 push CSS。客户端拿到 HTML 时 CSS 已经在 cache。

失败理由 1 · cache 协调不可解

浏览器 cache 状态对 server 不可见：

  • Server push 总是推 → 浪费带宽（重复 push 已 cache 资源）
  • Server 不推 → 失去 push 意义
  • 引入 cache digest（client 告诉 server "我有什么"）→ 计算贵 + 实现复杂 + 隐私问题
  • RFC 7540 留了 RST_STREAM 给 client "拒绝这个 push"，但 push 已经在 wire 上才能拒绝 → 已浪费

这是本质的工程矛盾，不是实现问题。

失败理由 2 · 浏览器实现 bug 多

Chrome / Firefox / Safari 的 push cache 实现各不相同：

  • Chrome 2017-2020 多次发现 push 资源没正确进 cache
  • 有些场景进了 cache 但下次请求没用上
  • 跨 connection 的 push 资源协调更复杂

这种"理论上有效但实现 bug 多"让生产部署不稳定。

失败理由 3 · 实测数据

Chrome 团队 2018 公布数据：

  • 使用 push vs 不使用：FCP / LCP / TTI 差异 < 0.5%（统计噪音内）
  • 但流量浪费明显（重复推送已 cache 资源）
  • 有些场景 push 还更慢（push 占 server 上行带宽挤压主请求）

退场时间线

  • 2018 Chrome 限制 push（默认禁用某些场景）
  • 2020 Chrome 团队公开考虑移除
  • 2022-06 RFC 9113 §8.4 明确"不再推荐"
  • 2022 Chrome 默认禁用 push（仍可手动启用）
  • 2023 Cloudflare / Fastly / Akamai 移除 push 功能
  • 2024-2026 几乎所有 server 停用

103 Early Hints（RFC 8297）

HTTP 1xx 系列简介

HTTP 状态码 1xx 是 informational，不是最终响应。server 可以发一个或多个 1xx 后再发最终 2xx/3xx/4xx/5xx。

1xx 的著名例子：

  • 100 Continue：client 大 body 时先发 headers + Expect: 100-continue → server 检查 headers OK → 发 100 → client 才上传 body
  • 101 Switching Protocols：HTTP/1.1 → WebSocket Upgrade 用
  • 103 Early Hints：本章主角

103 Early Hints wire 长这样

http 
Client → Server:
  GET /index.html HTTP/1.1
  Host: example.com

Server → Client (立即发，不等 backend):
  HTTP/1.1 103 Early Hints
  Link: </style.css>; rel=preload; as=style
  Link: </critical.js>; rel=preload; as=script
  Link: <https://fonts.gstatic.com>; rel=preconnect; crossorigin

Server → Client (backend 处理完后):
  HTTP/1.1 200 OK
  Content-Type: text/html
  Content-Length: ...

  <!doctype html>
  <html>
  ...

关键时序：

  1. Client 发 GET
  2. Server 立即发 103（在 backend 还在处理时）
  3. Client 看到 103 + Link 头 → 立刻开始 preload 这些资源
  4. Backend 处理完成（500ms-2s）
  5. Server 发 200 + body
  6. Client 收到 HTML 时，preload 资源可能已经下完一部分

"立即发 103"是关键：在 backend 还在查数据库 / 渲染 SSR 时，client 已经在 preload。这一段时间不再浪费。

为什么不是 Server Push

Early Hints 和 Server Push 关键差异：

维度 Server Push 103 Early Hints
谁决定要不要下 Server Browser
cache 协调 Server 盲发 → 浪费 Browser 检查 cache，已有就不下
带宽风险 有（重复 push） 无（browser cache 检查）
实现复杂度 高（需要 cache digest） 低（标准 preload 机制）
HTTP 协议要求 仅 HTTP/2 + 部分 /3 HTTP/1.1 / 2 / 3 都支持
2026 状态 已退场 普及中（8%→ 持续增长）

HTTP/2 / HTTP/3 上的 103 Early Hints

HTTP/2 / HTTP/3 也支持 1xx 系列响应。每个 1xx 是一个独立的 HEADERS frame（带 :status 1xx）。最终 2xx 是另一个 HEADERS frame。

实际上 HTTP/3 上的 Early Hints 比 HTTP/1.1 上更可靠：HTTP/1.1 上某些老代理 / proxy 不识别 1xx 直接吞掉（兼容性问题）；HTTP/2 / HTTP/3 上代理对 1xx 的处理更标准。

Cloudflare 2022 实测

Cloudflare 2022 在大量站点 A/B 测试 103 Early Hints：

  • FCP（First Contentful Paint）减少 中位数 5-10%
  • LCP（Largest Contentful Paint）减少 中位数 10%、p75 减少 20%、backend 慢的站点（500ms+ TTFB）减少 30%
  • TTFB（Time to First Byte）不变（103 不影响 TTFB 计算）

关键洞察：backend 越慢，103 Early Hints 收益越大。原因：backend 处理时间这一段以前是"浏览器干等"，现在变成"浏览器在 preload"。

Vercel + Next.js 16 集成

Vercel Edge runtime 2022 起支持 103 Early Hints。Next.js 16 的具体集成：

  • 静态资源自动发：Next.js 16 在 SSR 阶段，把页面引用的关键 CSS / JS chunks 自动转换成 Link header 发 103
  • Server Component 提示：可以在 Server Component 里调用 API 提示 preload（API 仍在演进，2026 主线方案）
  • Vercel Edge Functions：res.writeEarlyHints() 直接写 103

详见 Next.js 主题（链回）。

现代替代 Server Push 的完整方案

2026 替代 Server Push 不是单一工具，是一套组合：

资源类型 替代方案
首屏关键 CSS / 字体 <link rel="preload"> 内联在 head
LCP 图 fetchpriority="high" + preload
backend 慢的关键资源 103 Early Hints（在 backend 处理前发）
第三方 origin 预热 <link rel="preconnect">
路由切换预热 Speculation Rules API（prerender / prefetch）
同 origin 下次访问 HTTP cache（Cache-Control / ETag）

这套组合覆盖了 Server Push 想解决的所有场景，而且每个工具都更精确、更可控、更 cache-aware。

03 应用场景

场景 A · backend 500ms+ 的 SSR 应用

典型 Next.js / Rails / Django SSR 应用，backend 渲染 + 数据库查询要 500-1500ms。103 Early Hints 收益最大的场景。在 backend 处理前发 preload CSS / JS / 字体 → 用户感知首屏减少 200-500ms。Cloudflare / Vercel 部署时默认开。

场景 B · 静态站点：103 Early Hints 收益小

静态 HTML 直接从 CDN edge cache 出来，TTFB 已经 50-100ms。在这个时间窗口里 103 没多少时间帮忙 preload。对静态站点 LCP 影响 1-3%。仍可以开（Cloudflare 默认开），但不是关键优化。

场景 C · CDN 缓存 103 头给后端 cache miss 时用

典型工程姿势：CDN 在 origin 响应里抓 Link 头，把它们当 103 缓存。下次同一 URL cache miss 时，CDN 立刻发 103（带 Link 头）+ 同时回源拿正文。把 Early Hints 的"提前提示"扩展到 CDN 层。Cloudflare Smart Hints 2022 起自动做这件事。

场景 D · 仍配 Server Push 的老站点：迁移工作清单

(1) 删 nginx http2_push 指令；(2) 删 Apache H2PushResource；(3) 应用代码里 Link: ...; rel=preload 的资源标 ; nopush（避免被 nginx-quic 当 push 触发）；(4) 改用 103 Early Hints 或纯 preload；(5) 验证：抓包确认没 PUSH_PROMISE 帧。

04 典型代码：发 103 Early Hints

Node.js Express / Fastify 发 103

javascript 
// Node.js 18+ ServerResponse 内置 writeEarlyHints
import express from 'express';

const app = express();

app.get('/page', async (req, res) => {
  // 立即发 103，不等 backend
  res.writeEarlyHints({
    link: [
      '</critical.css>; rel=preload; as=style',
      '</app.js>; rel=preload; as=script',
      '<https://fonts.gstatic.com>; rel=preconnect; crossorigin',
    ],
  });

  // 慢 backend 处理（数据库 / SSR 渲染）
  const data = await fetchSlowData();    // 500ms+
  const html = renderSSR(data);

  // 最终 200
  res.status(200).send(html);
});

app.listen(3000);

Cloudflare Workers 发 103

javascript 
// Cloudflare Workers (2022+ 支持 Early Hints)
export default {
  async fetch(request, env, ctx) {
    // 用 ctx.waitUntil 不阻塞响应
    // 但 Early Hints 是同步发出的（在 fetch 处理过程中）

    // Cloudflare 自动从 origin 响应 Link 头转 103
    // 也可以显式控制：
    const earlyHintsHeaders = new Headers();
    earlyHintsHeaders.set('Link', '</style.css>; rel=preload; as=style');

    // 传给 fetch 的 cf 配置启用 Early Hints
    const originResponse = await fetch(request, {
      cf: {
        earlyHints: true,
      },
    });

    return originResponse;
  },
};

Next.js 16 自动发 103

javascript 
// next.config.js
module.exports = {
  experimental: {
    earlyHints: true,  // 启用 (Next.js 16 起默认 true 在 Vercel 上)

  return <div>...</div>;
}

nginx 删 push 配置

nginx 
# 旧配置（删掉）：
http {
  server {
    location = / {
      # http2_push /style.css;        ← 删
      # http2_push /app.js;           ← 删
    }
  }
}

# 新配置：用 Link header + nopush 标记 + 让 Cloudflare/Vercel/CDN 转 103
http {
  server {
    location = / {
      # 加 Link 头让上游 CDN 转 103 Early Hints
      # nopush 防止 nginx 自己触发 HTTP/2 push
      add_header Link "</style.css>; rel=preload; as=style; nopush" always;
      add_header Link "</app.js>; rel=preload; as=script; nopush" always;
    }
  }
}

05 常见陷阱

陷阱 1 · 仍在配 Server Push

nginx http2_push / Apache H2PushResource / 应用代码发 Link 头但没 nopush —— 都是反模式。2026 全部删，改 103 Early Hints 或纯 preload。

陷阱 2 · 老 proxy 吞 103

HTTP/1.1 上某些老代理（特别是企业内网代理）不识别 1xx，直接吞掉 → 客户端从来收不到 103。检查方法：用 curl 直连 origin 看 wire；用 curl 经过 proxy 看 wire 是否还有 103。HTTP/2 / HTTP/3 上代理处理 1xx 标准化更好，问题更少。

陷阱 3 · 103 头发了但和最终响应不一致

103 里 preload 了 /style.v1.css，最终 HTML 引用的是 /style.v2.css。结果：浏览器下了 v1.css 又下 v2.css，浪费带宽。103 必须和最终响应严格一致。Next.js 等框架自动同步；自建 server 必须谨慎。

陷阱 4 · 把 103 当"魔法加速"

Backend 已经很快（TTFB < 100ms）的场景，103 增益极小（< 3%）。甚至有些场景下，多发一个 103 响应增加少许网络开销 → 可能为负。103 是 backend 慢场景的优化，不是处处加速。

陷阱 5 · 103 链接太多，反而拖慢主请求

103 里 preload 50 个资源 → 浏览器全部高优先级下 → 主 HTML 请求被挤。原则：103 只 preload 5-10 个真正关键的资源。同 P5.1 的 preload 滥用一样的道理。

陷阱 6 · 浏览器不支持 103 → 仍发不报错

不支持 103 的浏览器（极少见，2026 主流全支持）会忽略 103 直接等 200。103 是渐进增强，发了不会报错；不发也只是失去优化。无需 feature detection。

陷阱 7 · 框架文档说自动发 103，实际没发

Next.js / Hono / Bun 等框架声称支持 103，但实际触发条件可能很苛刻（必须 Vercel Edge / 必须特定配置）。用 curl -v 或 Chrome DevTools 验证。如果没发，看框架配置 / 升级版本。
