单向流：fetch streams / chunked / SSE / 与 RSC + streamedQuery + AI SDK 协作

RSC 流式渲染、TanStack streamedQuery、Vercel AI SDK 流式响应底下都是协议层的 1-2 条特定路径。这一章把单向流（server → client）的三个层次讲清：HTTP/1.1 chunked（基础）/ fetch + ReadableStream（浏览器统一抽象）/ SSE（chunked + 格式约定 + 自动重连）。然后给三个应用层场景（RSC payload / TanStack streamedQuery / AI SDK Data Stream）的协议层根因。

阅读时长 ~25 分钟 前置 用过 fetch；写过 SSE 或 React 19 use API 至少一种 主要来源 WHATWG Streams 标准 + WHATWG EventSource 标准 + RFC 9112 §7（chunked）+ Vercel AI SDK Data Stream Protocol 文档 + React Flight serialization format（公开但未文档化）+ Cloudflare Workers / Vercel Edge 关于流式响应的工程博客

01 速览

  1. 单向流（server → client，client 不发回写）的协议层三个层次：(a) HTTP/1.1 chunked transfer encoding（wire 基础）；(b) fetch + ReadableStream（浏览器侧统一抽象）；(c) SSE（chunked + text/event-stream 格式 + 自动重连）。
  2. chunked transfer encoding（RFC 9112 §7）：HTTP/1.1 流式 wire 基础。每 chunk = <size hex>\r\n<data>\r\n，0 size 结束。HTTP/2 / HTTP/3 上不再用 chunked，用各自的 DATA frame 替代，但应用层流式语义一样。
  3. fetch + ReadableStream：WHATWG Streams 标准的浏览器实现。fetch(url).then(r => r.body) 返回 ReadableStream，可以 .getReader() 边收边解析。对 chunked / HTTP/2 DATA / HTTP/3 DATA 三种 wire 统一抽象。
  4. SSE（Server-Sent Events，EventSource API）：HTTP body 是 text/event-stream，每事件 = data: ...\n\n。浏览器内置自动重连（断网后重连 + Last-Event-ID 续传）。
  5. SSE vs fetch + ReadableStream 选择：SSE 自动重连 + 跨 origin EventSource 比较开箱；fetch + ReadableStream 更灵活（自定义协议 / 二进制 / POST 请求 / 取消控制）。2024+ 新项目流式响应几乎都走 fetch + ReadableStream。
  6. RSC payload 协议层：HTTP/1.1 chunked 或 HTTP/2/3 DATA frame 承载 React Flight serialization（0:[...]\n1:I[...]\n2:HL[...]\n 每行一个 JSON 块）。详见 Next.js 主题。
  7. TanStack streamedQuery 协议层：fetch + ReadableStream，边收边累加进 query cache + 触发组件重渲。详见 TanStack Query P4.2（双向链）。
  8. Vercel AI SDK Data Stream Protocol：fetch + ReadableStream（不用 SSE）+ NDJSON-like 自定义格式（每行 <type>:<value>\n，支持 text-delta / tool-call / annotation 等）。
  9. HTTP/2 / 3 上的流式 vs HTTP/1.1：HTTP/2 流式数据走 DATA 帧 multiplexed，不阻塞别的 stream；HTTP/3 每条 stream 独立丢包恢复 → 流式响应在丢包下表现最好。HTTP/1.1 流式独占连接（不能并发 + 6 connection limit 是真实约束）。

02 底层逻辑

三层抽象的关系

text 
应用层抽象        | RSC payload | TanStack streamedQuery | Vercel AI SDK | SSE | 普通 NDJSON
─────────────────|─────────────|──────────────────────|─────────────|─────|────────────
浏览器 API        | fetch+RS    | fetch+RS              | fetch+RS    | EventSource | fetch+RS
─────────────────|─────────────|──────────────────────|─────────────|─────|────────────
协议层 wire        | chunked / DATA frames                                  | chunked
─────────────────|────────────────────────────────────────────────────────|────────────
传输层            | TCP / QUIC                                              | TCP

RS = ReadableStream

三层抽象的关键观察：

  • SSE 选了独立的浏览器 API（EventSource），不复用 fetch；这给了它"自动重连"的内置能力，但牺牲了灵活性
  • RSC / streamedQuery / AI SDK 都用 fetch + ReadableStream，自己定义应用层格式 → 完全可控
  • 底下都是 chunked / DATA frame，wire 层完全相同

HTTP/1.1 chunked transfer encoding（wire 基础）

已在 P2.1 简述。这里展开。

http 
HTTP/1.1 200 OK
Content-Type: application/x-ndjson
Transfer-Encoding: chunked

7\r\n
{"a":1}\r\n     ← chunk 1: 大小 0x7=7 字节 + data 7 字节 + \r\n

8\r\n
{"b":22}\r\n    ← chunk 2: 大小 0x8=8 字节 + data 8 字节 + \r\n

0\r\n            ← 终结 chunk: 0 大小
\r\n             ← 空行（可放 trailer）

关键点：

  • chunked 是wire 层格式，应用层看到的是连续 body
  • 每 chunk 大小用16 进制 ASCII 字符，不是二进制（这是 HTTP/1.1 文本协议的痕迹）
  • 0 大小表示流结束
  • HTTP/2 / HTTP/3 不用 chunked —— 它们的 DATA frame 自带长度。但response 的 wire 长度可以是渐进的（一个 frame 一个 frame 发），应用层语义一样

fetch + ReadableStream（浏览器统一抽象）

WHATWG Streams 标准定义了三种流：ReadableStream / WritableStream / TransformStream。fetch 的 response.body 是 ReadableStream。

核心 API

javascript 
const res = await fetch('/stream-endpoint');

// 1. 直接当 stream 用（适合传给 Response 或转换）
return new Response(res.body);

// 2. getReader 手动消费
const reader = res.body.getReader();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  // value is Uint8Array
  console.log('chunk:', value);
}

// 3. for await...of (现代姿势)
for await (const chunk of res.body) {
  // chunk is Uint8Array
  console.log('chunk:', chunk);
}

// 4. pipe to TextDecoderStream + line splitter
const lines = res.body
  .pipeThrough(new TextDecoderStream())
  .pipeThrough(new LineSplitter());  // 自定义 transform
for await (const line of lines) {
  console.log('line:', line);
}

关键能力

  • 边收边消费：不等整个响应到齐
  • 背压（backpressure）：消费慢时通知 producer 暂停
  • 取消（cancel）：reader.cancel() 中断 + 关闭底层 TCP 连接
  • 转换（pipe through）：链式 transform，每一段独立处理
  • 统一抽象：HTTP/1.1 chunked / HTTP/2 DATA / HTTP/3 DATA 都给同一个 ReadableStream

SSE（Server-Sent Events）

SSE 是 chunked + 一层文本格式约定 + 浏览器内置 EventSource API。

Wire 格式

http 
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Transfer-Encoding: chunked

data: hello\n\n           ← 一个事件，两个 \n 结束
data: world\n\n           ← 另一个事件

event: update\n           ← 自定义事件类型
data: {"id":1,"x":2}\n\n

id: 42\n                  ← Last-Event-ID（重连续传用）
data: with id\n\n

retry: 5000\n             ← 重连间隔（ms）
data: with retry\n\n

: this is a comment\n\n   ← 注释（: 开头）+ 兼容性 keep-alive

EventSource API

javascript 
const es = new EventSource('/api/events');

// 默认事件
es.onmessage = (e) => {
  console.log('data:', e.data);
  console.log('lastEventId:', e.lastEventId);
};

// 自定义事件
es.addEventListener('update', (e) => {
  const data = JSON.parse(e.data);
  console.log('update:', data);
});

// 错误（自动重连）
es.onerror = (e) => {
  console.log('error or connection lost; auto reconnecting...');
};

// 手动关闭
es.close();

SSE 内置能力

  • 自动重连：连接断开后自动重连（间隔由 server 通过 retry: ms 控制，默认 3 秒）
  • Last-Event-ID 续传：重连时浏览器自动带 Last-Event-ID: 42 header；server 可以续从 ID 42 之后的事件
  • 跨 origin：标准支持 EventSource(url, { withCredentials: true })；server 配 CORS 即可
  • 事件分发：浏览器自动按 event: 字段分发到不同 listener

SSE 局限

  • 仅文本（UTF-8）：不能传二进制（要传二进制只能 base64 编码）
  • 仅 GET 请求：不能 POST（这是 EventSource API 的硬限制）
  • HTTP/1.1 上独占连接：6 connection limit 影响多 SSE 同 origin
  • 没法"取消请求"+ 立刻发新请求：必须 close + new EventSource

这些局限是 2024+ 新项目改用 fetch + ReadableStream 的理由。

三个应用层场景的协议层根因

RSC payload（React Server Components）

  • 协议层：HTTP/1.1 chunked 或 HTTP/2/3 DATA frame
  • 应用层格式：React Flight serialization。每行一个 JSON 块：0:[...]\n / 1:I[...]\n / 2:HL[...]\n。其中第一个数字是 ID，后面是元素描述
  • 工程意义：浏览器在收完整页面前已开始渲染部分组件 → 改善 TTFB / FCP
  • 关键设计：每个 chunk 是独立可解析的 JSON 行（NDJSON-like），即使流中断也能渲染已收到的部分
  • 详见 Next.js 主题 RSC streaming 和 React 主题 P6.1

TanStack streamedQuery

  • 协议层：fetch + ReadableStream
  • 应用层格式：用户自定义（NDJSON / SSE format / 自定义协议）
  • 工程意义：边收边累加进 query cache + 触发订阅组件重渲
  • 关键 API：v5 的 streamedQuery 在 queryFn 里返回 async iterator，TanStack Query 自动处理 cache 增量更新
  • 详见 TanStack Query P4.2（双向链）

Vercel AI SDK Data Stream Protocol

  • 协议层：fetch + ReadableStream（不用 SSE）
  • 应用层格式：NDJSON-like，每行 <type>:<json>\n。0 = text delta；2 = data；3 = error；9 = tool call；a = tool result；e = finish reason；等
  • 为什么不用 SSE：(1) 需要 POST（用户消息在 body 里）；(2) 需要二进制能力（attachment / image）；(3) 需要 fine-grained 取消控制；(4) 自定义 frame 类型（tool calls 等不是简单"事件"）
  • 2026 现状：v5 起标准协议；v6（latest）/ v7（canary）继续演进
  • 详见待写 AI SDK 主题（双向链）

HTTP/2 / HTTP/3 上流式 vs HTTP/1.1

HTTP/2 上流式

  • 流式数据走 DATA 帧
  • 同一 TCP 连接里多个 stream 并发流式（与 6 connection limit 无关）
  • 但仍受 TCP HOL blocking：丢包时整连接 stall

HTTP/3 上流式

  • 流式数据走 QUIC stream 上的 STREAM frame
  • 每条 stream 独立丢包恢复 → 一条流式响应丢包不影响其他流
  • 流式响应在丢包下表现最好

HTTP/1.1 上流式

  • 流式独占一个 TCP 连接（连接里不能并发别的请求）
  • 6 connection limit 是真实约束 —— 一个页面同时跑 4-5 个 SSE 已经吃掉一半连接配额
  • 这是不应该在生产用 HTTP/1.1 跑多 SSE 的根本理由

03 应用场景

场景 A · LLM 流式聊天（GPT / Claude）

用户提问 → server 边生成边返回 token → 客户端边收边显示。fetch + ReadableStream + 自定义协议（Vercel AI SDK Data Stream Protocol）是 2026 主流。SSE 不行因为：(1) 需要 POST 提交用户消息；(2) 需要二进制（图片）；(3) 需要细粒度取消（用户 stop 生成）。

场景 B · 实时通知 / 监控 dashboard

server push 状态变化（订单状态 / 监控指标 / 股票价格）。SSE 是经典选择 —— 自动重连 + Last-Event-ID 续传 + 跨 origin 简单。HTTP/1.1 / 2 / 3 都支持。

场景 C · RSC 流式渲染

Next.js / React Server Components：server 渲染时边渲边推 RSC payload chunks。fetch + ReadableStream + React Flight format。浏览器在收完整页面前已渲染部分组件 → FCP 改善 30-50%。详见 Next.js 主题。

场景 D · TanStack Query streamedQuery 大量数据

查询返回大量数据（搜索结果 / 列表分页）：server 边查边返回。fetch + ReadableStream + NDJSON。TanStack v5 的 streamedQuery API 自动处理 cache 增量。详见 TanStack Query P4.2。

场景 E · 单向数据流监控/日志

server 持续推日志 / 指标到 client。SSE 仍然合适，特别是 client 不关心顺序、希望 reconnect 时拿到漏掉的事件（Last-Event-ID）。

04 典型代码：完整 NDJSON / SSE / RSC 实现

Express NDJSON 流式 server

javascript 
import express from 'express';

const app = express();

app.get('/api/stream-data', async (req, res) => {
  res.setHeader('Content-Type', 'application/x-ndjson');
  res.setHeader('Transfer-Encoding', 'chunked');
  res.setHeader('X-Accel-Buffering', 'no');  // 防止 nginx buffer

  // 边查数据库边推
  for (let i = 0; i < 100; i++) {
    const data = await fetchSomething(i);
    res.write(JSON.stringify(data) + '\n');
    // res.write 立即刷到 wire（chunked）
  }

  res.end();
});

app.listen(3000);

fetch + ReadableStream 解析 NDJSON

javascript 
async function* streamNDJSON(url) {
  const res = await fetch(url);
  if (!res.ok) throw new Error('fetch failed');

  const decoder = new TextDecoder();
  let buf = '';

  for await (const chunk of res.body) {
    buf += decoder.decode(chunk, { stream: true });
    const lines = buf.split('\n');
    buf = lines.pop();  // 留最后一行（可能不完整）
    for (const line of lines) {
      if (line.trim()) yield JSON.parse(line);
    }
  }
  // 处理最后一行
  if (buf.trim()) yield JSON.parse(buf);
}

// 使用
for await (const item of streamNDJSON('/api/stream-data')) {
  console.log('item:', item);
}

SSE 完整 server + client

javascript 
// server
app.get('/api/events', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('X-Accel-Buffering', 'no');

  const lastEventId = req.headers['last-event-id'];
  let id = lastEventId ? parseInt(lastEventId) + 1 : 1;

  const interval = setInterval(() => {
    res.write(`id: ${id}\n`);
    res.write(`event: update\n`);
    res.write(`data: ${JSON.stringify({ id, time: Date.now() })}\n\n`);
    id++;
  }, 1000);

  // 客户端断开
  req.on('close', () => clearInterval(interval));
});

// client
const es = new EventSource('/api/events');
es.addEventListener('update', (e) => {
  const data = JSON.parse(e.data);
  console.log(`event ${e.lastEventId}:`, data);
});

Vercel AI SDK 风格的流式协议（手写）

javascript 
// server: 模拟 LLM 流式
app.post('/api/chat', async (req, res) => {
  res.setHeader('Content-Type', 'text/plain; charset=utf-8');
  res.setHeader('Transfer-Encoding', 'chunked');
  res.setHeader('X-Accel-Buffering', 'no');

  const { messages } = req.body;
  const tokens = generateTokens(messages);  // 假设逐 token 生成

  for await (const token of tokens) {
    // 0: text delta
    res.write(`0:${JSON.stringify(token)}\n`);
  }

  // e: finish reason
  res.write(`e:${JSON.stringify({ finishReason: 'stop' })}\n`);
  res.end();
});

// client: 解析协议
async function streamChat(messages) {
  const res = await fetch('/api/chat', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ messages }),
  });

  const decoder = new TextDecoder();
  let buf = '';

  for await (const chunk of res.body) {
    buf += decoder.decode(chunk, { stream: true });
    const lines = buf.split('\n');
    buf = lines.pop();

    for (const line of lines) {
      if (!line) continue;
      const colonIdx = line.indexOf(':');
      const type = line.slice(0, colonIdx);
      const value = JSON.parse(line.slice(colonIdx + 1));
      yield { type, value };
    }
  }
}

// 使用
for await (const part of streamChat([{ role: 'user', content: 'hi' }])) {
  if (part.type === '0') {
    appendToUI(part.value);
  } else if (part.type === 'e') {
    console.log('done:', part.value.finishReason);
  }
}

nginx 配置：让流式响应不被 buffer

nginx 
location /api/stream {
    # 禁用 nginx 对响应的 buffering
    proxy_buffering off;
    proxy_cache off;
    proxy_set_header X-Accel-Buffering no;

    # 长流式连接需要长 timeout
    proxy_read_timeout 24h;
    proxy_send_timeout 24h;

    # HTTP/1.1（与 upstream 通信用 1.1 + keep-alive）
    proxy_http_version 1.1;
    proxy_set_header Connection "";

    proxy_pass http://upstream;
}

05 常见陷阱

陷阱 1 · 流式响应被 nginx buffer 吞掉

nginx 默认对响应做 buffering，把 chunked 流"攒一阵再发" → 客户端体感不流式。必须配 proxy_buffering off; + 应用响应头加 X-Accel-Buffering: no。这是 SSE / RSC / AI SDK 在生产部署最常见的坑。

陷阱 2 · SSE 跨 origin 没正确处理 CORS preflight

EventSource 默认不带 cookie；要带要 new EventSource(url, { withCredentials: true }) + server 设 Access-Control-Allow-Credentials: true + Access-Control-Allow-Origin: <具体 origin>（不能是 *）。漏掉任何一个 → SSE 失败。

陷阱 3 · HTTP/1.1 上多个 SSE 同 origin → 6 connection limit 撑爆

典型场景：dashboard 同时开 5 个 SSE → 占 5 个连接 → 用户其他请求只剩 1 个连接配额 → 整个页面慢。用 HTTP/2 / HTTP/3 解决（不再有 6 connection limit），或合并多个 SSE 到一个端点（用 event 字段区分）。

陷阱 4 · SSE 想传二进制

SSE 标准明确要求 UTF-8 文本。要传二进制 → base64 编码 → 体积膨胀 33%。需要二进制就别用 SSE，改 fetch + ReadableStream（流式 binary 直接传）或 WebSocket。

陷阱 5 · ReadableStream 取消但服务端不知道

客户端 reader.cancel() 后浏览器关 TCP 连接，但 Node.js / Express 的 res 不一定收到 close 事件（特别是经过 proxy 时）。server 监听 req.on('close') + req.on('aborted') 双重保险。否则 server 一直在生成，资源浪费。

陷阱 6 · Service Worker 拦截流式响应被 buffer

Service Worker 注册了 fetch handler，默认行为可能 buffer 整个响应再返回。SW 内必须显式传 stream：event.respondWith(fetch(event.request)) 而不是先 read body 再返。

陷阱 7 · NDJSON 解析没处理"半行 chunk"

fetch chunk 可能切在 JSON 中间（一行 JSON 跨两个 chunk）。简单 chunk.split('\n').forEach(JSON.parse) 会失败。必须 buffer 跨 chunk + 只解析完整行（最后一行留到下次 chunk 一起处理）。

陷阱 8 · 误以为 SSE 自动重连无脑可靠

SSE 重连时 server 必须支持 Last-Event-ID 续传（自己管 event ID + 重连时从 ID+1 续）。如果 server 不实现 → 重连后从头开始，client 收重复事件。SSE 自动重连只是连接层面，应用层续传逻辑要自己写。
