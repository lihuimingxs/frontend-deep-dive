Style → Layout（Reflow）：Cascade + Layout 算法 + Containment

渲染管线第二段。DOM + CSSOM 怎么变成「每个元素的位置 + 大小」（Layout tree）。关键洞察：Reflow 是性能大头——改一个 style 可能导致整树 reflow，几十 ms 量级；批量 DOM 操作 + read/write batching + CSS Containment 是优化核心。这一章讲透 Style cascade（CSS 优先级 + inheritance）+ Computed style + Layout tree（不含 display:none）+ BFC/IFC/Flexbox/Grid 算法 + Box model + Reflow 触发条件 + CSS Containment / content-visibility 现代性能特性。

阅读时长 ~30 分钟 前置 P3.1 主要来源 developer.mozilla.org/CSS · web.dev/animations · WHATWG / W3C CSS specs

01 速览

  1. Reflow 是性能大头—— 改一个元素的几何（width / height / position / margin 等）几乎一定让其他元素也 reflow（流式布局的特性）；几十 ms 量级。批量操作 + read/write batching 是优化核心。
  2. CSS Containment 是 2020+ 浏览器最重要的性能 API—— contain: layout 明确告诉浏览器「这块独立」，让 reflow 不外扩。content-visibility: auto 让浏览器跳过屏幕外元素的 layout/paint。
  3. Layout tree 不等于 DOM tree—— display: none 的元素不在 layout tree（不占位），但仍在 DOM tree（仍在内存）。理解这点能解释很多 layout 反直觉行为。

02 底层逻辑

Style Cascade：从 CSS 规则到每个元素的样式

CSS Cascade 是把 CSS rules 应用到 DOM elements的算法。每个元素的computed style来自所有匹配 CSS rule 的累积，按以下顺序决定：

(1) Origin + Importance：用户代理样式 (user agent) → 用户样式 → 作者样式 (author)；!important 反转优先级。

(2) Specificity (优先级)：选择器优先级——id (100) > class / attr / pseudo-class (10) > element / pseudo-element (1)。inline style 比 selector 高。

(3) Source order：specificity 相同时，后定义的胜。

(4) Inheritance：某些属性（color / font-family / line-height）从父元素继承；某些（margin / border）不继承。

(5) Initial / inherit / unset / revert：显式控制。initial 用规范 default 值；inherit 用父元素值；unset = inherit (可继承属性) 或 initial (不可继承)；revert 回到上一层 origin。

这套算法对每个元素跑一次 → 得到 Computed style—— 每个元素的最终样式值（resolved %、计算继承、解析 var()）。Computed style 是 layout 的输入。

性能：CSS 选择器匹配是 hot path。Chrome / Firefox / Safari 都用从右到左匹配—— div.foo > span.bar 先找 span.bar，再看父是不是 div.foo。这是为什么"避免深嵌套选择器"是 CSS 性能建议（深嵌套从右到左走更多步）。

Layout tree：不等于 DOM tree

Layout tree（也叫 Render tree）是「需要 layout 的元素」的子集。和 DOM tree 关系：

情况 DOM tree Layout tree
普通可见元素 ✅ 在 ✅ 在
display: none ✅ 在 ❌ 不在
visibility: hidden ✅ 在 ✅ 在（占位置但不可见）
opacity: 0 ✅ 在 ✅ 在（占位置 + 不可见）
pseudo-element ::before ❌ 不在 ✅ 在
<head> / <script> / <style> ✅ 在 ❌ 不在

关键差异让一些行为合理：

(1) display: none 不参与 layout——所以切换它时整个 layout 重算。

(2) visibility: hidden 参与 layout 但不绘制——切换时不触发 reflow（只 paint）。

(3) Pseudo-element 没 DOM 节点但有 layout box——CSS 创建的 layout 单元，DevTools Elements panel 看不到（要看 Computed）。

Layout 算法：从 box 到位置

Layout 算法计算每个元素的"位置 + 大小"。CSS 提供多种 layout mode：

(1) BFC (Block Formatting Context)—— 块级布局。default 行为。块级元素从上到下排列，每行占满父宽。display: block / display: flow-root / overflow: hidden（创建独立 BFC） / float / position 等触发新 BFC。

(2) IFC (Inline Formatting Context)—— 行内布局。inline 元素从左到右排，自动换行。文字 / span 是 IFC。Line box 高度由最高的 inline 元素决定。

(3) Flexbox—— display: flex。一维布局（行或列）。子元素 flex-grow / flex-shrink / flex-basis 算尺寸。

(4) Grid—— display: grid。二维布局（行列）。grid-template-rows / columns 定义网格，子元素自动 / 显式定位到 cell。

(5) Multi-column—— column-count / column-width。流式分多列。

(6) Subgrid (CSS Grid Level 2)—— Grid 嵌套时子 grid 复用父 grid 的 track。

每种 mode 有自己的算法。Flexbox / Grid 是现代 layout 主力，IFC / BFC 仍用于文本流。

性能：layout 复杂度通常 O(n) 元素数（n 是 layout tree size），但实际更复杂——某些 layout（深嵌套 flex / grid 含 percentage） 可能 O(n²) 或更糟。Chrome LayoutNG (2017+) 改进了这点——新一代 layout 引擎按"独立 layout subtree"切分，子树不影响父树。

Box Model：CSS 的几何基础

每个元素有 4 层 box：

Content box → Padding box → Border box → Margin box。

关键属性：

width / height：默认指 content box 宽高（standard model）。

box-sizing: border-box：让 width / height 指 border box（包含 padding + border）。这是现代 CSS reset 的标配—— * { box-sizing: border-box }—— 让宽度计算更直觉。

Margin collapsing：垂直 margin 重叠（取大值）。这是 BFC 的特性，flex / grid 子元素不重叠。新手常踩坑：两个 div 上下排，每个 margin: 20px → 间距是 20px（不是 40px）。

Negative margin：margin 可以是负数，让元素重叠或超出父 box。常用 hack 但难维护，现代 CSS 用 grid / flex 替代。

Reflow 触发条件

Reflow（也叫 Layout）是 layout 算法重新跑——性能大头。常见触发：

(1) 几何变化—— width / height / margin / padding / border / position / top / left / float 改变。

(2) 内容变化—— text 改变 / 添加删除 DOM / 图片加载完。

(3) 视口变化—— 窗口 resize / orientation 改变。

(4) 强制 reflow（forced layout）—— JS 读 layout 属性（offsetWidth / offsetHeight / getBoundingClientRect / getComputedStyle 涉及 layout 的属性）。这种"读"会强制浏览器立即跑 layout（如果有 pending invalidation），是性能反模式。

关键反模式："layout thrashing"—— 读 / 写 layout 属性交替，每次读都强制跑 layout：

// ❌ Layout thrashing：每次循环都强制 reflow
for (let i = 0; i < items.length; i++) {
  const w = items[i].offsetWidth;  // 强制 reflow（之前 write 让 layout 失效）
  items[i].style.width = w + 10 + 'px';  // write 失效 layout
}

// ✅ Read / Write batching：先全读，后全写
const widths = items.map(item => item.offsetWidth);  // 一次 reflow
for (let i = 0; i < items.length; i++) {
  items[i].style.width = widths[i] + 10 + 'px';
}

这种 "read-then-write" 模式是 DOM 优化基础。React / Vue 等框架内部都做这种 batching。

CSS Containment：明确隔离 reflow 影响

CSS Containment（contain 属性）让你明确告诉浏览器"这个元素的内部变化不影响外部"——浏览器优化 reflow 范围。

Containment 类型：

contain: layout—— 子元素的 layout 不影响外部。比如这个元素内的子元素改 width，外部 layout 不重算。

contain: paint—— 子元素的 paint 不绘制到外面（实际相当于 overflow: hidden + 创建 stacking context）。

contain: size—— 元素 size 不依赖子元素（你必须显式声明 width/height）。让子元素 layout 不影响这个元素。

contain: content—— layout + paint + style 三个 contain。常用速记。

contain: strict—— size + layout + paint 都 contain。最严格。

实操：

/* 长列表的每一项都 contain，让局部更新不影响整列表 */
.list-item {
  contain: content;
  height: 80px;  /* 显式 height 配合 size containment */
}

/* dashboard widget 用 contain 隔离 */
.widget {
  contain: layout style;
}

Containment 是现代大型应用的关键性能优化—— 没有 contain，DOM 一处变化可能让全树 reflow。Contain 后变化局限在这个 box 内。Chrome / Safari / Firefox 全支持（Container Queries 也基于 contain）。

content-visibility：跳过屏幕外元素

content-visibility: auto（Chrome 85+）让浏览器跳过屏幕外元素的 layout / paint—— 直到滚动接近时才 layout。

.below-fold-section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px;  /* 占位高度 */
}

用法：长列表 / dashboard / 文档每个 section 都设 content-visibility: auto—— 浏览器只 layout 屏幕内的，屏幕外完全跳过。性能提升巨大——长页面 layout 时间能从 1000ms 降到 100ms。

限制：(1) contain-intrinsic-size 必须设（否则浏览器不知道占多少位置）；(2) 屏幕外元素的内容 SEO 仍可见（搜索引擎能爬到，不像 display:none）；(3) 不影响屏幕内元素行为。

这是2020+ 浏览器最重要的性能 API 之一—— 把"虚拟列表"这种复杂前端 hack 变成 CSS 一行。Chrome / Edge 完整支持，Safari 18+ / Firefox 2024 跟进。

Flexbox 关键概念

Flexbox（display: flex）是一维布局，6 个核心属性：

容器属性：flex-direction（row / column）、justify-content（主轴对齐）、align-items（交叉轴对齐）、flex-wrap（换行 / 不换）、gap（子元素间距）。

子元素属性：flex: grow shrink basis。flex: 1 是最常用的——三值 1 1 0%，让子元素均分剩余空间。

Flexbox 的性能：现代浏览器优化得好。但大量嵌套 flex 仍可能慢—— flex 算法对子元素有 multiple pass（先算 basis、再算 grow/shrink、再 align），嵌套深度高时复杂度叠加。

Grid 关键概念

Grid（display: grid）是二维布局，最适合复杂布局结构（dashboard / 卡片网格 / 复杂表单）。核心：

grid-template-rows / columns—— 定义 grid 网格。值可以是 1fr 2fr（按比例分）/ repeat(3, 1fr) / auto / min-content / max-content。

grid-template-areas—— 用名字定义网格区域，子元素 grid-area 引用。可读性极高。

fr unit—— Grid 独有，"fraction unit"，按可用空间比例分配。

Subgrid (Level 2)—— 子 grid 复用父 grid 的 track。Chrome 117+ / Safari 16+ / Firefox 71+。

Grid 比 Flexbox 表达力强但学习曲线略陡。2020+ 现代 CSS 布局基本：Grid 做整体框架 + Flexbox 做局部组件。

Min-content / Max-content / fr / 各种 size

CSS 现代 size 关键字：

min-content—— 元素的最小内容尺寸（不会让内容溢出）。文字的 min-content 是最长单词的宽度。

max-content—— 元素的最大内容尺寸（内容不换行）。

fit-content(N)—— 在 min-content 和 max-content 之间，最大不超过 N。

fr (Grid)—— 按 fraction 分配剩余空间。

auto—— 复杂语义（context-dependent）。在 grid 是 max-content；在 flex 是 flex-basis。

这些关键字让 layout 表达力远超 px / %。现代 CSS 很多布局可以"只用 Grid + min/max-content + fr 表达"，不需要 JS 算尺寸。

Aspect-ratio：避免 layout shift

aspect-ratio（CSS 2021+）让你声明元素的宽高比，浏览器自动按比例计算。

.video {
  width: 100%;
  aspect-ratio: 16 / 9;  /* 自动算 height */
}

.avatar {
  width: 60px;
  aspect-ratio: 1;  /* 圆形头像 */
}

这是避免 CLS（Cumulative Layout Shift）的关键 API——之前要用 padding-bottom hack 才能保留比例占位，现在一行搞定。详见 P4.3 CLS。

03 应用场景

场景 A · 用 content-visibility 优化长页面 first paint

项目是长内容页面（博客文章 / 文档 / 列表）—— first paint 慢因为浏览器 layout 整页。优化：每个 section 加 content-visibility: auto + contain-intrinsic-size: auto 500px—— 浏览器只 layout 屏幕内的 section，屏幕外完全跳过。具体做法：(1) 把页面切成逻辑 section（每个 article / 每个 list 块）；(2) 每个 section 加 content-visibility；(3) 用 DevTools Performance panel 测 layout 时间——通常能从几百 ms 降到几十 ms。注意：(1) contain-intrinsic-size 一定要设（否则没占位）；(2) 屏幕外内容仍可被 Ctrl+F 搜到、SEO 可见；(3) IE / 老 Safari 不支持，但作为渐进增强 OK。

场景 B · 用 contain 隔离 dashboard widget

Dashboard 有 20+ widget，每个 widget 内部数据频繁更新——担心一个 widget 更新让整个 dashboard reflow。优化：每个 widget 加 contain: content—— 浏览器知道这个 widget 内部变化不影响外部。现实效果：单 widget 数据更新只 reflow 这个 widget 的内部子树，dashboard 整体不动。具体做法：(1) widget 设 contain: layout style（不必 paint，paint 是另外问题）；(2) widget 设固定 / aspect-ratio，配合 contain: size；(3) 用 DevTools Performance Recording 验证 reflow 范围。这种隔离让大型 dashboard 在频繁更新场景仍 60fps。

场景 C · 排查「Layout thrashing」性能问题

页面交互卡——DevTools Performance Recording 看到大量 "Layout" 事件。常见原因：layout thrashing。识别方法：(1) 看 Recording 中 "Recalculate Style" + "Layout" 频繁交替；(2) 用 Chrome DevTools Performance Insights → "Forced reflow"。定位代码：搜索代码里读 + 写交替的 DOM 操作（offsetWidth / scrollHeight + style.X）。修复：(1) batching——先全读再全写；(2) 用 requestAnimationFrame 让浏览器自然 batch；(3) framework 里不要在 render 函数里读 layout 属性。这是大部分前端性能问题的根因之一。

场景 D · 用 Grid + Subgrid 做复杂表单 layout

项目要做多列表单（label / input 对齐多行）——传统方法用 table（语义不对）或 flex（label 对齐难）。Grid + Subgrid 优雅解决：父 grid 定义"label 列 + input 列"的整体网格；每个 row 用 subgrid 复用父 grid track，确保 label / input 跨多 row 对齐。具体做法：父 display: grid; grid-template-columns: max-content 1fr；每个 row 用 subgrid。对照传统：传统 flex 做这种"多行整体对齐"要 JS 算最长 label 宽度；Grid + Subgrid CSS 一句话表达。这是 CSS 现代 layout 的力量——把"曾经 JS 解决"的问题变成 CSS 表达。

// Read / Write batching：避免 layout thrashing

// ❌ 反模式
function badResize(items) {
  for (const item of items) {
    item.style.width = item.offsetWidth + 10 + 'px';
    // 每次 offsetWidth 强制 reflow
  }
}

// ✅ 正确：先全读
function goodResize(items) {
  // 1. 全部 read
  const widths = Array.from(items, item => item.offsetWidth);

  // 2. 全部 write
  items.forEach((item, i) => {
    item.style.width = widths[i] + 10 + 'px';
  });
}

05 常见陷阱

读 offsetWidth / getBoundingClientRect 不知道触发 forced reflow

JS 读 element.offsetWidth / getBoundingClientRect() / getComputedStyle(el).width 等 layout 属性时，如果之前有 pending DOM 改动，浏览器必须立即跑完 layout 才能给你准确值——这是 forced reflow / synchronous layout。一次还行，循环里反复就是 layout thrashing。规避：(1) DevTools Performance 找 "Forced reflow"；(2) read / write batching；(3) 用 ResizeObserver / IntersectionObserver 异步获取 layout 信息（避免 forced reflow）。

在 React render 里读 layout 属性

React 组件在 render 里读 ref.current.offsetWidth—— 每次 render 都触发 forced reflow。React 18 + Concurrent Mode 让这种问题更严重（render 可能被打断 + 重跑）。正确做法：(1) 用 useLayoutEffect 读 layout（在 paint 前同步跑）；(2) 用 ResizeObserver 异步监听变化；(3) 不要在 render 里读 layout。

用 contain 但没设 contain-intrinsic-size

contain: size 让浏览器认为元素 size 不依赖子元素——但你必须显式声明 width/height，否则浏览器不知道占多少位置。常见错误：写 contain: content（包含 size），但元素 height 是 auto——结果元素塌成 0 高度。规避：用 contain: layout（不含 size）；或显式设 contain-intrinsic-size: 200px；或用 aspect-ratio 配合。

滥用深嵌套选择器影响 CSS 选择器性能

.foo .bar .baz .qux .item { ... } 这种深嵌套选择器——浏览器从右到左匹配，每个 .item 元素都要往上看 4 层 ancestor。Hot path 上累积影响。规避：(1) 单层 class 选择器（用 BEM / Tailwind utility class）；(2) 不超过 3 层嵌套；(3) 避免 universal selector + 复杂组合（如 * > .x）。现代 CSS 最佳实践是单 class 风格—— 性能 + 可读性都好。
