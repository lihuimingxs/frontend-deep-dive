HTML → DOM → CSSOM 解析：从字节流到树

渲染管线第一段。HTML / CSS 怎么变成 DOM 树 + CSSOM 树。讲透 HTML parser（流式构造 DOM + 边下载边 parse）+ Preload scanner（HTML parser 之外的预扫描器，让浏览器并行下载资源）+ CSSOM 阻塞渲染（必须 parse 完才能 layout）+ JS 三种加载顺序（default / async / defer / module）+ 现代 preload / modulepreload / fetchpriority。关键洞察：preload scanner 是大部分人不知道的"隐藏功能"，让 HTML parser 看似串行实则并行下载。

阅读时长 ~30 分钟 前置 P2 主要来源 web.dev/critical-rendering-path · whatwg.org/HTML · Chrome team blog

01 速览

  1. Preload scanner 是浏览器隐藏的并行下载机制—— 大部分人以为 HTML parser 是串行（遇到 script 阻塞下载、再 parse 下面），实际浏览器有第二个扫描器并行扫描整个 HTML 找 link/script/img 提前下载。这是现代浏览器载入快的核心机制。
  2. CSS 阻塞渲染但不阻塞 HTML parsing——CSS 没下载完前浏览器不会 first paint（要等 CSSOM 完整）；但 HTML parsing 仍在继续。CSS 阻塞 JS 执行——JS 可能查询 style，所以 JS 执行前 CSS 必须 parse 完。
  3. JS 三种加载顺序：default / async / defer，加上 type=module 默认 defer——理解这四种语义是优化首屏加载的基础。<script type="module"> 默认 defer 是现代写法。

02 底层逻辑

HTML parser：流式 DOM 构造

HTML parser 是把 HTML 字节流变成 DOM 树的组件。两个阶段：

(1) Tokenizer：把字节流变成 token 序列（开标签 / 关标签 / 属性 / 文本）。规则在 WHATWG HTML spec 定义，包含大量边界 case（HTML5 容错性强——错误的标签嵌套也能 parse）。

(2) Tree builder：把 token 变成 DOM 节点 + 维护元素栈构造 tree 结构。处理嵌套规则（<p> 不能嵌套 <div>，浏览器自动闭合）。

关键设计：流式 parse——HTML parser 不等整个 HTML 下载完，是边收到字节边 parse。这意味着：

(1) HTML 顶部的内容更快可见——浏览器可以先 parse + 渲染顶部，再继续 parse 下面。

(2) CDN / 服务器流式响应有意义—— Server 用 chunked transfer encoding 持续输出 HTML，浏览器持续 parse，比"等整个 HTML 生成完再发"快很多。

(3) HTML 顺序影响渲染速度—— 把首屏关键内容放 HTML 前面，让 first paint 早。

Preload scanner：隐藏的并行下载

HTML parser 看起来是串行的——遇到 <script> 默认阻塞 parsing 等下载 + 执行；遇到 <link> 引用 CSS 也阻塞。但实际浏览器有第二个 parser 叫 Preload scanner——它是 HTML parser 之外的预扫描器，不阻塞主 parsing 流程。

Preload scanner 的工作：

(1) 提前扫描整个 HTML—— 跳过当前 parser 在的位置，往后看；

(2) 找 <link rel="stylesheet"> / <script src="..."> / <img src="..."> 等需要下载的资源；

(3) 提前发起请求—— 让浏览器并行下载这些资源；

(4) 不影响主 HTML parser 状态—— 主 parser 还按串行行为，但资源已经在后台下。

这套机制让"看起来串行的 HTML parser"实际有大量并行下载。如果没有 preload scanner：HTML parser 串行扫到第 100 行的 <script> → 阻塞 → 下载 → 执行 → 继续；扫到第 200 行的 <img> → 才开始下载这个 img。有 preload scanner：HTML 一开始下载浏览器就整体扫描，所有资源（script / img / css）都并行下载。

Preload scanner 的限制：

(1) 动态创建的资源看不到—— JS document.createElement('script') 后插入的资源 preload scanner 不知道。这是为什么 dynamic import / framework lazy load 的资源加载较慢。

(2) 只看 src 属性—— 如果你用 data-src + 自家 lazy loading，preload scanner 看不到。

这是为什么 <link rel="preload"> + fetchpriority 等显式 hint API 重要——它们告诉浏览器（包括 preload scanner）哪些资源高优先级。

CSS 解析：CSSOM 与渲染阻塞

CSS 也要 parse 成CSSOM (CSS Object Model)——和 DOM 平行的树。每个元素的 style 来自所有匹配的 CSS rule 的 cascade。

CSS 的关键性质：

(1) CSS 是渲染阻塞资源 (render-blocking)——CSSOM 必须完整 parse 才能 first paint。原因：浏览器不能"先用旧 CSS 渲染，新 CSS 来了再覆盖"——会出现 FOUC (Flash of Unstyled Content)。所以等 CSS。

(2) CSS 不阻塞 HTML parsing—— HTML parser 继续 parse，构造 DOM 树。但不会进入 layout/paint 阶段。

(3) CSS 阻塞 JS 执行—— 这是反直觉的。原因：JS 可能 getComputedStyle() 查询 style，所以浏览器在 JS 之前必须确保 CSSOM 完整。如果 CSS 还在下载，<script> 会等 CSS 完成才执行——即使 script 在 CSS 之前。

所以经典优化：CSS 放 head，script 放 body 底部。CSS 早 parse 不阻塞渲染过晚；script 在 body 底部不阻塞 HTML parsing 看到所有 DOM。

现代写法：用 <script defer> 或 <script type="module">（默认 defer），脚本放 head 也不阻塞 HTML parsing。

JS 加载的四种语义

<script> 标签有四种加载语义：

语法 下载阻塞 HTML parse 执行阻塞 HTML parse 执行顺序
<script src> (default) 不阻塞（preload scanner 已并行下载） 阻塞（下载完立即同步执行） 文档顺序
<script async> 不阻塞 不阻塞（下载完异步执行） 下载完成顺序（不可预测）
<script defer> 不阻塞 不阻塞（DOMContentLoaded 前执行） 文档顺序
<script type="module"> 不阻塞 不阻塞（默认 defer） 文档顺序（依赖图解析）

具体语义：

default（不带 async / defer）：旧默认行为。脚本下载 + 执行阻塞 HTML parser—— 这是为什么经典优化"script 放 body 底部"。但因为 preload scanner，下载是并行的，只有执行阻塞。

async：下载完立即执行（即使 HTML 还没 parse 完）。执行顺序按下载完成顺序，不保证文档顺序。适合独立脚本（analytics / ads），不依赖其他 script。

defer：下载完等 HTML parse 完再执行（DOMContentLoaded 前）。多个 defer 按文档顺序执行。适合依赖 DOM 但不阻塞 first paint 的脚本。

type="module"：默认 defer 行为 + 模块系统（import / export）。现代标准。type="module" + async 也行，混合两种语义。

实战推荐：现代 JS 几乎全用 <script type="module" src="app.js">—— 默认 defer + 模块化 + 不阻塞。老代码用 <script defer> 也好。避免 default 阻塞写法（除非你真的需要 inline 同步脚本）。

DOMContentLoaded vs load：两个生命周期 event

两个常用 event：

DOMContentLoaded：HTML 完全 parse 完 + 所有 defer scripts 执行完触发。不等 async scripts、不等 image / iframe / 其他资源。这时 DOM tree 完整，可以操作 DOM。

load：所有资源（图片 / iframe / async script / etc.）都加载完才触发。比 DOMContentLoaded 晚很多——大型页面可能差几秒。

实战用法：

(1) 操作 DOM 用 DOMContentLoaded——大部分 init 代码不需要图片加载完。

(2) 计算页面整体高度 / 处理图片用 load——需要所有资源就位。

(3) 现代 JS 框架（React / Vue）通常用 DOMContentLoaded 或更早——hydrate / mount 不必等图片。

现代 hint API：preload / modulepreload / fetchpriority

<link rel="preload">：显式声明高优先级资源。浏览器会立即开始下载（不等 preload scanner / CSS / JS 决定）。

<link rel="preload" href="hero.jpg" as="image" fetchpriority="high">
<link rel="preload" href="critical.css" as="style">
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>

<link rel="modulepreload">：预加载 ES Module。比 preload 更知道是 module，能预解析依赖图。

<link rel="modulepreload" href="/app.js">
<link rel="modulepreload" href="/utils.js">
<script type="module" src="/app.js"></script>

fetchpriority 属性 (Chrome 102+)：调整资源优先级。fetchpriority="high"（首屏关键资源）/ "low"（below-the-fold）/ "auto"（默认）。

<img src="hero.jpg" fetchpriority="high">
<img src="below-fold.jpg" fetchpriority="low" loading="lazy">

<link rel="preconnect"> / "dns-prefetch"：预连接到第三方 domain。preconnect 建立 TLS 连接（最大优化），dns-prefetch 只 DNS 解析。

<link rel="preconnect" href="https://api.example.com">
<link rel="dns-prefetch" href="https://cdn.example.com">

这些 hint 是优化关键资源加载顺序的核心工具。LCP 优化（详见 P4.1）大量依赖这些 hint。

资源优先级模型

浏览器内部对资源有优先级队列—— 不是"看到 A 资源就立即下载 A 完成再下 B"，而是按优先级调度多个并发下载。优先级（Chrome 视角）：

Highest：HTML 主文档、render-blocking CSS、async/defer script（前 ~5 个）、preload + fetchpriority="high" 资源。

High：non-async / non-defer 同步 script（HTML 内联的）、首屏 image (Largest Contentful Paint candidate) 自动提升。

Medium：async/defer script（后续）、其他 image。

Low：lazy load image、prefetch 资源、低优先级图片。

Lowest：preconnect、dns-prefetch、prefetch。

实操：用 DevTools Network panel 看每个资源的 Priority 列。如果你的 LCP image 不是 Highest 优先级，加 fetchpriority="high" 或 preload。

HTTP 协议层影响

HTTP 协议版本影响并行下载能力：

HTTP/1.1：每个连接串行处理请求。浏览器对每个 origin 最多 6 个并行连接 → 同时下载 ≤ 6 个资源。这是为什么早年要做 domain sharding（把资源分到多个 subdomain 绕过 6 连接限制）。

HTTP/2：单个连接多路复用—— 一个连接能并发处理几十几百个请求。Domain sharding 不再必要（甚至有害）。HEADERS 用 HPACK 压缩。

HTTP/3 (QUIC)：基于 UDP，进一步降低握手 + 队头阻塞。Mobile 网络下显著好于 HTTP/2。详见待写的 HTTP/2/3 主题。

现代 web 开发必须 HTTPS + HTTP/2 或 HTTP/3—— 这是性能基线。Cloudflare / AWS CloudFront / Vercel / Netlify 等 CDN 都默认 HTTP/2 或 HTTP/3。

03 应用场景

场景 A · 用 preload 优化首屏 LCP image

项目 LCP 是 hero image（首屏顶部大图）—— 浏览器要 parse HTML → 找到 <img> → 开始下载 → 渲染。延迟在"找到 img 才开始下载"这步。优化：在 <head> 加 <link rel="preload" href="hero.jpg" as="image" fetchpriority="high">—— 浏览器解析 head 时就开始下载 hero image，不用等到 body 的 img tag。LCP 通常能提升 200-500ms（取决于网络）。这是最有效的 LCP 优化之一。注意：不要 preload 一切—— preload 让资源占用首屏带宽，过度 preload 反而拖慢真正关键的资源。只 preload 你确定首屏需要的 1-3 个关键资源。

场景 B · 排查「为什么我的 page 渲染慢」

页面 first paint 慢——用 Chrome DevTools Performance panel 录制，看渲染管线时间线。常见原因：(1) render-blocking CSS 太大—— 看 Network panel CSS 文件大小 + Performance 上 CSS parse 时间。优化：critical CSS inline 到 HTML head；其他 CSS 异步加载（用 media="print" onload hack 或 modern preload）。(2) JS 阻塞—— head 里有大同步 script。改 defer 或 async。(3) 第三方资源阻塞—— Google Fonts / GTM / 广告 SDK 加载在前 head。改 async load 或 lazy load。(4) FCP 已快但 LCP 慢—— 真正的 LCP image 没 preload。加 fetchpriority。这套排查流程让你 1-2 小时定位 first paint 瓶颈。

场景 C · 用 preconnect 优化第三方 API 延迟

项目首屏要调 api.example.com 拿数据——TLS 握手 + DNS resolve 占 200-500ms。优化：<link rel="preconnect" href="https://api.example.com"> 在 head——浏览器解析 head 时就开始建 TLS 连接 + DNS。当真正调 fetch 时，连接已就绪——节省 200-500ms。注意：(1) preconnect 不是 prefetch——只建连接不下载内容；(2) 浏览器会限制 preconnect 数（避免滥用），最多 ~6-10 个 origin；(3) 跨 origin 的 font 必须加 crossorigin attribute。具体：CDN / API 主 origin 用 preconnect；不重要的 third party 用 dns-prefetch（更廉价）。

场景 D · 给做 SSR 框架的工程师：理解流式响应

做 SSR 框架（Next.js / Remix / SvelteKit）—— 流式响应（HTTP chunked transfer encoding）是关键性能特性。机制：server 不等所有数据生成完再发 HTML，而是边生成边 chunk 发。浏览器边收到 chunk 边 parse → 顶部内容更早可见。实操要点：(1) HTML head 先 flush—— 包含 critical CSS / preload / 关键 meta，浏览器立即 preload scanner 启动；(2) main content 流式输出—— React 18 Suspense / Next.js streaming SSR 自动支持；(3) 慢数据用 Suspense fallback—— 先发 fallback skeleton，数据来后流出更新。这种"流式 SSR"让 LCP 比传统 "render full HTML 再 send" 快几百 ms。

<!doctype html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">

  <!-- 1. preconnect 第三方 origin -->
  <link rel="preconnect" href="https://api.example.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

  <!-- 2. preload 关键资源 -->
  <link rel="preload" href="/hero.jpg" as="image" fetchpriority="high">
  <link rel="preload" href="/font-bold.woff2" as="font" type="font/woff2" crossorigin>

  <!-- 3. critical CSS inline -->
  <style>
    /* 首屏可见区域的 CSS */
    body { margin: 0; font-family: sans-serif; }
    .hero { height: 100vh; ... }
  </style>

  <!-- 4. 非关键 CSS 异步加载 -->
  <link rel="stylesheet" href="/non-critical.css" media="print" onload="this.media='all'">

  <!-- 5. JS 用 module / defer -->
  <script type="module" src="/app.js"></script>

  <!-- 6. 第三方独立脚本用 async -->
  <script async src="https://www.googletagmanager.com/gtag/js?id=G-XXX"></script>
</head>
<body>
  <img src="/hero.jpg" alt="hero" fetchpriority="high">
  <!-- below-the-fold 图片用 loading="lazy" -->
  <img src="/below.jpg" loading="lazy" alt="...">
</body>
</html>

// 用 PerformanceObserver 测 HTML / CSS / JS 加载时间
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.entryType === 'resource') {
      console.log(`${entry.name}: ${entry.duration}ms (priority: ${entry.fetchPriority})`);
    }
  }
}).observe({ entryTypes: ['resource'] });

// 监听 DOMContentLoaded 和 load
document.addEventListener('DOMContentLoaded', () => {
  console.log('DOM ready:', performance.now());
  // 可以操作 DOM
});

window.addEventListener('load', () => {
  console.log('all resources loaded:', performance.now());
  // 所有图片 / iframe / async script 完成
});
javascript 
// 动态创建 script —— preload scanner 看不到，可能加载慢
// 优化：手动 preload 让浏览器知道
async function loadModuleAndUse() {
  // 提示浏览器 preload
  const link = document.createElement('link');
  link.rel = 'modulepreload';
  link.href = '/heavy-module.js';
  document.head.appendChild(link);

  // 真正加载
  const mod = await import('/heavy-module.js');
  return mod.doStuff();
}

05 常见陷阱

过度 preload 让首屏更慢

有人看到 preload 优化首屏 LCP，给所有图片 / 脚本都加 preload——反而拖慢。原因：preload 让所有 preload 资源争夺首屏带宽，真正关键的 LCP image 反而被挤占。正确做法：只 preload 1-3 个最关键资源（LCP image、关键 CSS、关键 font）；其他让浏览器自然处理。规避：用 DevTools Network panel 看每个资源 priority + 时序，确认 preload 真的让关键资源更早完成（不是被其他 preload 挤占）。

用 default script 阻塞渲染

有人写 <script src="app.js"></script> 在 head——这是 default 语义，下载完立即执行 + 阻塞 HTML parsing。即使 preload scanner 提前下载了，执行仍阻塞。这让 first paint 推迟到 script 执行完。规避：永远用 <script defer> 或 <script type="module">（默认 defer）。除非你真需要 inline 同步脚本（如 polyfill / framework loader），否则不要 default 行为。

认为「JS 放 body 底部」=「不阻塞」

"script 放 body 底部"是经典优化，但不完美。问题：(1) HTML 大时仍要 parse 整个 HTML 才到 script—— 不能利用 preload scanner；(2) script 仍阻塞 DOMContentLoaded——比 defer 慢。更好的写法：<script defer src="app.js"></script> 在 head——下载并行（preload scanner）+ 不阻塞 parsing + DOMContentLoaded 前执行。这是 2026 现代 best practice。

忘记 CSS 阻塞 JS 执行

有人发现 script 在 CSS 之前但仍要等 CSS——以为是 bug。实际是浏览器规范：JS 可能查 style，所以 JS 执行前 CSS 必须 parse 完。即使 script 在 CSS link 之前。规避：(1) critical CSS inline + 非关键 CSS 异步加载，让 CSS 不阻塞 JS；(2) 用 defer script（DOMContentLoaded 前执行，CSS 已经 parse 完）；(3) 别在 JS 里用 getComputedStyle 太早——可能强制 layout。
