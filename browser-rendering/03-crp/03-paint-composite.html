Paint → Composite：GPU 合成 + Layer tree + Tile-based raster

渲染管线第三段。Layout tree 怎么变成屏幕像素。关键洞察：Reflow vs Repaint vs Composite 三态是浏览器性能的核心模型——geometry 改 → reflow + repaint + composite；颜色改 → repaint + composite；transform/opacity 改 → composite only。后者是 60fps 动画的黄金路径。这一章讲透 Paint phase（生成 paint records 不是真绘制）+ Layer tree + Tile-based rasterization + GPU 合成 + will-change 滥用 + GPU raster + Vulkan/Metal 后端 + OOP-D（Out-of-Process Display Compositor）。

阅读时长 ~30 分钟 前置 P3.2 主要来源 developer.chrome.com / Skia docs / Chrome team blog

01 速览

  1. Reflow vs Repaint vs Composite 三态是浏览器性能心智模型——你做的每个动画 / 视觉变化都对应这三态之一。composite-only 是黄金路径，不需要 layout / paint，只走 GPU 合成。
  2. transform / opacity 走 composite-only——这是为什么这两个 CSS 属性是动画首选。width / height / left / top / margin 等触发 reflow—— 即使在 transition 里也会让动画卡。
  3. Compositor thread 独立运行—— 主线程被 100ms 长任务卡时，transform 动画仍 60fps 顺滑（因为 Compositor 在另一线程跑）。这是浏览器架构给前端最大的性能礼物。

02 底层逻辑

Paint Phase：生成 paint records（不是真绘制）

Paint 是把 layout tree 变成「绘制指令列表」—— 每个元素生成一组 paint records（绘制指令）。这一步不真的画到屏幕—— 真绘制（rasterization）在后续步骤。

Paint 阶段的工作：

(1) 遍历 layout tree，按 z-index / stacking context 顺序生成 paint records；

(2) 每个元素生成多个绘制指令—— background / border / text / decoration 各一段指令；

(3) Paint records 是 immutable + 可序列化—— 让多线程 / 跨进程合成成为可能。

这种"生成指令而非直接绘制"的设计让浏览器有巨大的优化空间：

(1) Layer 分组—— 把 paint records 按 layer 分组，每个 layer 独立 raster；

(2) 异步 raster—— 主线程生成 paint records 后扔给 Raster threads（worker pool）真绘制；

(3) 缓存 layer—— 没变的 layer 复用之前的 raster 结果，不重画；

(4) GPU 合成—— GPU 把所有 layer 合成最终图像。

Layer tree：分组 + 独立 raster

Layer tree把 layout tree 中的元素分组—— 每组（layer）独立 raster + 合成。

哪些情况会创建独立 layer？

(1) 3D / transform / 透视——transform: translate3d / perspective / transform-style: preserve-3d。

(2) will-change 提示——will-change: transform / opacity 强制创建 layer。

(3) backface-visibility hidden。

(4) position: fixed——通常独立 layer。

(5) filter / mix-blend-mode / 某些 effect。

(6) Video / Canvas / iframe—— 内容独立来源，独立 layer。

(7) Stacking context 在某些 z-index 复杂情况。

Layer 多的好处：每个 layer 独立 raster + 合成，变化只影响那一个 layer，性能高（特别是动画场景）。

Layer 多的坏处：每个 layer 占 GPU 内存（一张 bitmap）。过度 layer 会让 GPU 内存爆—— DevTools 看到 "Layer count: 100+"，GPU 内存可能几百 MB。这是为什么 will-change 滥用是反模式。

Chrome DevTools "Layers" panel 能可视化 layer tree—— 看每个 layer 的尺寸、被 promote 原因、内存占用。性能优化时第一站。

Tile-based Rasterization：异步分块绘制

Rasterization是真把 paint records 画成像素。Chromium 用tile-based 策略：

(1) 每个 layer 切成 256×256 tiles—— 一个 1920×1080 layer 切成约 32 个 tile。

(2) 只 raster 屏幕内 tile—— 屏幕外的 tile 不画（懒加载）。滚动时按需画新 tile。

(3) Raster 在独立线程—— Raster threads（worker pool）并行画多个 tile。主线程不被阻塞。

(4) GPU raster (Chrome 默认 2020+)—— 把 raster 移到 GPU。比 CPU raster 快 5-10x。

这套设计的好处：(1) 滚动顺滑（只画新 tile，不重画整页）；(2) 内存效率（屏幕外不占像素内存）；(3) 多核 + GPU 并行。

Chrome 还在演进：

SkiaRenderer（Chrome 87+）—— 用 Skia 画整个浏览器 UI，统一渲染管线。

Vulkan / Metal 后端—— Skia 可以用 Vulkan（Linux/Android）/ Metal（macOS/iOS）/ DirectX（Windows）作为底层 GPU API。比传统 GL 快 + 现代。

GPU Compositing：合成最终图像

Compositing 是把所有 raster 完的 tile 合成成最终屏幕图像。在 GPU process 跑（详见 P2.1），独立于 Renderer 进程。

合成步骤：

(1) 收 layer tree + tile bitmap—— 从 Renderer 主线程来。

(2) 计算每个 layer 的最终 transform—— 应用 CSS transform / opacity / filter。这一步对 transform/opacity 几乎免费—— GPU 用矩阵乘法，硬件加速。

(3) 按 z-order 把 layer 叠加—— 透明的 layer 用 alpha blending。

(4) 输出到 framebuffer—— 屏幕显示。

这套合成不需要 Renderer 主线程——主线程被 long task 卡时，Compositor 仍能 60fps 跑（基于上次 raster 的 tile）。这是 transform/opacity 动画顺滑的根本原因。

Reflow vs Repaint vs Composite：性能心智模型

每个 CSS 属性变化对应渲染管线的不同阶段。三态模型：

变化类型 触发管线 代价 典型 CSS 属性
Reflow Style → Layout → Paint → Composite 最高（几十 ms） width / height / margin / padding / position / top / left / float
Repaint Paint → Composite 中（几 ms 到 10ms） color / background-color / box-shadow / border-color
Composite Composite only 最低（< 1ms，GPU 几乎免费） transform / opacity / filter (部分)

这个心智模型是Web 动画 + 性能优化的核心——你看每个 CSS 属性会判断它走哪条路径，决定动画方案。

实操规则：

(1) 动画用 transform / opacity—— 走 composite-only，60fps 顺滑。

(2) 避免在动画中改 width / height / margin / padding / left / top / position—— 触发 reflow，掉帧。

(3) 颜色 / 阴影 / 边框颜色变化—— 触发 repaint，能用但不如 composite-only。

(4) 用 transform: translate 替代 left/top 动画—— 视觉效果一样，性能完全不同。

transform / opacity 黄金路径详解

为什么 transform / opacity 走 composite-only？

CSS Transform（transform: translate / rotate / scale / matrix）—— 只改变元素的视觉位置，不改变 layout box 的位置。这意味着：

(1) Layout 不变（其他元素位置不动）；

(2) Paint 不变（每个 layer 内容不变）；

(3) Composite 时 GPU 用矩阵乘法应用 transform——硬件加速近免费。

CSS Opacity—— 改变元素透明度。原理：

(1) Layout 不变；

(2) 如果是独立 layer，paint 不变（layer 已经画好）；

(3) Composite 时 GPU 应用 alpha blending—— 硬件加速近免费。

对 opacity 重要的是"独立 layer"—— 如果 opacity 元素不在独立 layer，opacity 变化会触发 repaint。规避：用 will-change: opacity 强制独立 layer。

这两个属性是GPU 合成时代浏览器给前端的最大礼物—— 让 60fps 动画从"奢侈"变成"默认"。

will-change：明示 layer 提升

will-change是 CSS 属性，提示浏览器"这个元素将来要变化某个属性，提前优化"。最常见用法：

.animated-element {
  will-change: transform;
  /* 或：will-change: opacity, transform */
}

浏览器看到 will-change 后提前创建独立 layer—— 让后续 transform 变化只走 composite-only。

但 will-change 极易滥用—— 给所有元素加 will-change 让浏览器创建过多 layer，GPU 内存爆 + 整体性能下降。正确使用：

(1) 只给真要动画的元素加—— 不动的元素不需要 layer。

(2) 动画结束后移除—— el.style.willChange = 'auto'，让浏览器回收 layer。

(3) 避免给容器加—— 给容器加让所有子元素都升 layer，过度。

(4) 用其他方式触发 layer—— transform: translateZ(0) 或 backface-visibility: hidden 也能创建 layer，更隐式。

Compositor 优化的边界

Compositor 不能优化所有动画。具体限制：

(1) 必须是独立 layer—— 元素如果不在独立 layer，transform 变化也会触发 repaint。规避：will-change 或 translate3d。

(2) filter 部分走 composite—— 简单 filter（blur / brightness）GPU 加速；复杂 filter（drop-shadow / convolution）可能仍要 paint。

(3) clip-path 部分走 composite—— 简单 clip-path GPU 加速；复杂 polygon 可能要 paint。

(4) border-radius 大值 + transform—— 某些组合会让 GPU 不能完全 cache layer。

(5) Sub-pixel 渲染—— transform 非整数像素位置可能导致 anti-aliasing 不一致。

实操：用 DevTools Performance + "Show frames" 看每帧的 paint vs composite 时间。如果你预期 composite-only 但发现 paint 时间高，调整 layer 提升策略。

OOP-D：Out-of-Process Display Compositor

OOP-D是 Chromium 的进程模型演进——把 display compositor（最终把所有 Renderer 合成结果输出到屏幕的组件）独立到专用进程。之前 display compositor 在 Browser 或 GPU 进程，OOP-D 让它独立。

好处：

(1) Renderer 崩溃时屏幕不闪烁—— display compositor 仍能 paint 上次帧。

(2) 多 Renderer 合成更高效—— 一个 display compositor 服务所有 Renderer。

(3) 屏幕输出延迟降低—— compositor 能更早 swap framebuffer。

这种 OOP-D 是 Chromium 内部架构演进，对开发者透明—— 但理解它能让你看懂 Chromium 源码的最新组织。

查看 Layer 状态：DevTools 工具

实操工具：

(1) DevTools "Layers" panel（Chrome 内：More tools → Layers）—— 可视化所有 layer，看每个 layer 的尺寸、内存占用、被 promote 原因。

(2) DevTools "Rendering" panel—— 开 "Layer borders" 看每个 layer 边界（橘色框）；开 "Paint flashing" 看 repaint 区域（绿色高亮）。性能调试 essential。

(3) DevTools Performance Recording—— 录制一段时间，看 timeline 上每帧的 Style / Layout / Paint / Composite 时间。

(4) chrome://gpu—— GPU 进程状态、driver 版本、硬件加速是否启用。如果硬件加速没启用，所有 composite 走 CPU——性能巨差。

这些工具是诊断浏览器渲染性能的核心。每个 Web 工程师应该熟悉。

03 应用场景

场景 A · 把 left/top 动画改成 transform：60fps 提升

项目有 sidebar 滑入 / 弹窗弹出动画——用 left 或 top 动画卡顿。改成 transform: translateX / translateY 立即顺滑。原因：left/top 触发 reflow（可能影响其他元素位置） + repaint + composite，几十 ms / 帧；transform 走 composite-only，< 1ms / 帧。具体做法：(1) 用 transform 替代 left/top；(2) 加 will-change: transform 提前创建 layer；(3) 动画结束移除 will-change；(4) 测试用 DevTools Performance Record + Frames panel。常见踩坑：(1) 如果 sidebar 撑开 layout（影响 main content），transform 不能替代——需要重新设计 layout 让 sidebar 浮动；(2) 老 IE 11 不支持 transform 3D 语义—— 现代项目不必担心。

场景 B · 排查「主线程卡 100ms 但动画仍顺滑」

团队成员问"主线程被 JS 卡了 100ms，为什么动画还顺滑"——多进程 + Compositor thread 给出答案。核心机制：(1) Renderer 主线程跑 JS / Layout / Paint；Compositor thread 跑 GPU 合成；(2) 主线程被 JS 卡时，Compositor 仍以 60fps 用上次 raster 的 tile合成；(3) 如果动画走 composite-only（transform/opacity），新帧 = 旧 tile + 新 transform 矩阵——完全不需要主线程。这就是为什么 transform 动画能在主线程卡死时仍流畅——是浏览器架构给前端的礼物。验证：故意写个 setTimeout 阻塞主线程 500ms，看 transform 动画是否仍顺滑（应该是）。

场景 C · 用 DevTools Layers panel 排查 GPU 内存

用户报"页面 GPU 内存占用高 + 滚动卡"——用 Chrome DevTools Layers panel 排查。步骤：(1) 打开 More tools → Layers；(2) 看 layer 总数 + 每个 layer 内存（"Memory estimate"）；(3) 点 layer 看"Compositing reasons"—— 找出哪些 layer 是被错误提升的（如全局 will-change 滥用）。常见原因：(1) 给所有 list-item 加 will-change: transform → 100+ layer；(2) 给容器加 will-change → 子元素全升 layer；(3) 第三方 SDK 滥用 transform-3d。修复：移除不必要的 will-change；只在动画时加，结束移除；用 transform: translateZ(0) 代替 will-change（更精细控制）。

场景 D · 给做复杂滚动动画的工程师：用 IntersectionObserver + transform

做"滚动到这个元素时元素 fade in"动画——传统方法用 scroll 监听 + 改 opacity。问题：scroll 事件高频触发 + opacity 改如果元素不在独立 layer 触发 repaint。现代做法：(1) IntersectionObserver 监听元素 visibility—— 不需要 scroll 监听，自动 batch；(2) 触发时改 transform / opacity—— 走 composite-only；(3) 预先 will-change: opacity, transform—— 让元素在独立 layer。具体代码：new IntersectionObserver((entries) => { if (entries[0].isIntersecting) el.classList.add('visible') })；CSS 用 transition: opacity 0.3s, transform 0.3s。这套组合让滚动动画零卡顿。

04 典型代码

css 
/* ✅ 黄金动画路径：transform / opacity */
.modal {
  transform: translateY(-20px);
  opacity: 0;
  transition: transform 0.3s, opacity 0.3s;
  will-change: transform, opacity;  /* 提前创建 layer */
}

.modal.visible {
  transform: translateY(0);
  opacity: 1;
}

/* ❌ 反模式：top / margin / height 动画 */
.modal-bad {
  margin-top: -20px;
  height: 0;
  transition: margin-top 0.3s, height 0.3s;
  /* 触发 reflow + repaint + composite，每帧几十 ms */
}

/* ✅ 移除 will-change（动画结束后） */
.modal:not(.animating) {
  will-change: auto;
}
javascript 
// IntersectionObserver + transform 实现滚动 fade in
const observer = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    if (entry.isIntersecting) {
      entry.target.classList.add('visible');
      observer.unobserve(entry.target);  // 一次性
    }
  }
}, {
  threshold: 0.1  // 10% 进入视口就触发
});

document.querySelectorAll('.fade-in').forEach(el => observer.observe(el));

// CSS:
// .fade-in { opacity: 0; transform: translateY(20px); transition: 0.3s; will-change: opacity, transform }
// .fade-in.visible { opacity: 1; transform: translateY(0) }
javascript 
// 用 PerformanceObserver 监听 Long Tasks
// long task 期间 composite-only 动画仍能顺滑

new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log('Long Task:', entry.duration + 'ms');
  }
}).observe({ entryTypes: ['longtask'] });

// 模拟主线程长任务
function blockMainThread(ms) {
  const start = performance.now();
  while (performance.now() - start < ms) {}
}

// 启动 transform 动画
const box = document.querySelector('.box');
box.style.transition = 'transform 2s';
box.style.transform = 'translateX(500px)';

// 1 秒后阻塞主线程 500ms
setTimeout(() => blockMainThread(500), 1000);

// 观察：transform 动画在主线程阻塞期间仍然顺滑
// PerformanceObserver 会报 long task = 500ms
// 但用户看到的动画 60fps

05 常见陷阱

滥用 will-change 让 GPU 内存爆

给所有元素加 will-change: transform—— 浏览器创建一堆 layer，GPU 内存爆，整体性能反而差。规避：(1) 只给真要动画的元素加；(2) 动画结束移除（el.style.willChange = 'auto'）；(3) 不给容器加（让所有子元素升 layer）；(4) 用 DevTools Layers panel 监控 layer 数。一般原则：layer 数 < 50 是合理，> 100 过多。

用 transform: scale 做尺寸动画

看起来 transform: scale 比改 width / height 顺滑——但scale 改变视觉大小，不改 layout box。所以：(1) 元素本身视觉变小，但仍占原 box 的 layout 空间；(2) 内部文字 / 图片按 scale 缩放（可能模糊）；(3) 周围元素位置不变。对开发者影响：scale 适合"提示性动画"（按钮 hover 微缩 / 图片 zoom），不适合"真正改变布局"的尺寸变化。后者要么 reflow（接受成本），要么重新设计 layout。

opacity 动画不在独立 layer 触发 repaint

opacity 走 composite-only 的前提是元素在独立 layer。如果元素不在独立 layer，opacity 变化触发 paint（重画整个元素）—— 仍比 reflow 好但不是 composite-only。规避：opacity 动画的元素加 will-change: opacity 强制独立 layer；或加 transform: translateZ(0)（隐式 layer 提升）；用 DevTools Layers panel 验证元素在独立 layer。

硬件加速没启用时 composite 走 CPU

极少情况下硬件加速没启用（GPU driver 问题 / 用户 disabled / Chrome 黑名单某些 GPU）—— 整个 composite pipeline 走 CPU，性能巨差。检查：chrome://gpu 看 "Graphics Feature Status"——如果 "Hardware accelerated" 是 disabled / software only，就是这种情况。规避：(1) 用户层面：升级 GPU driver / Chrome；(2) 开发层面：测试 fallback 行为，不要假设 composite 总是 GPU；(3) 监控用户中硬件加速比例（用 WEBGL 检测代理）。
