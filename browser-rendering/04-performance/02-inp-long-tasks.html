交互侧 metrics：INP / Long Tasks

从用户输入到响应完成的性能。INP（Interaction to Next Paint）是 2024 起替代 FID 的核心 Web Vital——目标 ≤ 200ms。讲透 INP vs FID 差异 + Long Tasks（> 50ms 阻塞主线程）+ PerformanceObserver API 监听 + scheduler.yield()（Chrome 129+ 让出主线程）+ scheduler.postTask（精细调度 user-blocking/visible/background）+ Event 处理链 + isInputPending + requestIdleCallback。关键洞察：INP 是大部分应用真正卡顿的根源——FID 只看首次，INP 看所有交互。

阅读时长 ~30 分钟 前置 P4.1 + ECMA P7.3 Event Loop 主要来源 web.dev/inp · web.dev/optimize-long-tasks · scheduler API spec

01 速览

  1. INP 在 2024 替代 FID 成为核心 Web Vital——FID 只看首次交互，INP 看所有交互的最差延迟。很多页面 FID 好但 INP 差——开始很快但后续越来越卡。
  2. Long Tasks（> 50ms）是主线程阻塞的标志——单个 long task 让所有交互延迟。常见来源：大 React reconcile / 大 JSON.parse / 同步 layout flushing。
  3. scheduler.yield() 是 2024+ 的关键 API——比 setTimeout(0) 更优的"让出主线程 + 立即继续"。让长任务能 cooperative yield，提升 INP。

02 底层逻辑

FID → INP：为什么 2024 替换

FID (First Input Delay)是 2020-2023 核心 Web Vital。定义：第一次用户交互（click / tap / key）到事件 handler 开始执行的时间。

FID 的问题：

(1) 只看首次交互—— 之后所有交互不算。很多 SPA 首次加载好但用户后续操作越来越卡，FID 完全察觉不到。

(2) 只看 input delay 不看 processing—— FID 只是"事件触发到 handler 开始"，不包括 handler 执行时间。但用户感知是 input → 看到响应的全部时间。

(3) 大部分页面 FID 都好—— FID < 100ms 是 "good"，2023 数据 90%+ 页面达标。这让 FID 成为不能区分的 metric。

2024 年 Google 把核心 Web Vital 从 FID 替换为 INP (Interaction to Next Paint)。

INP 定义：从输入到下一次 paint

INP是用户交互（click / tap / keydown）到下一次浏览器 paint的全部时间。包含：

(1) Input delay—— 事件触发到 handler 开始（旧 FID）；

(2) Processing time—— event handler 执行 + 任何 JS 工作；

(3) Presentation delay—— handler 完成到下一次 paint（包括 React 重 render / Layout / Paint）。

INP 报告整个会话最差的交互延迟（98th percentile，长会话用 P98 防 outlier）。

阈值：≤ 200ms 是 "good"，500ms+ 是 "poor"。

典型场景：

(1) 用户点按钮——按钮高亮 + 弹窗显示。INP = 点击到弹窗可见。

(2) 用户输入文字——每个 keydown 触发 INP；输入框响应慢直接体现。

(3) 用户滚动——scroll 不算 INP（性能模型不同），但 scroll 触发的 onClick / hover 算。

Long Tasks：主线程被阻塞的标志

Long Task是主线程上跑 > 50ms 的连续任务。50ms 是浏览器规范定义的阈值——主线程被卡 50ms+ 就感知到延迟。

Long Task 期间：

(1) 用户输入排队—— click / keydown 触发但 handler 不执行（等当前 task 完成）；

(2) 渲染暂停—— 下一帧不绘制（默认 60fps = 16.67ms / 帧，long task 直接掉帧）；

(3) 动画卡—— 主线程动画卡（CSS animation 部分受影响）；只有 composite-only 动画（transform/opacity）不卡（因为 Compositor 独立线程）。

常见 Long Task 来源：

(1) JS 长任务—— 大循环 / 复杂正则 / 同步 array 操作 / 复杂 React reconcile。

(2) JSON.parse 大数据—— 1MB+ JSON parse 容易 50ms+。

(3) 同步 layout flushing—— 在 JS 中读 layout 属性触发 forced reflow。

(4) GC pause—— V8 GC 可能 stop-the-world 几十 ms。详见 V8 P4.2 Orinoco GC。Orinoco 大幅降低暂停时间，但仍可能有 long task。

(5) 大 DOM 操作—— 一次性插入几千个 DOM 节点。

用 PerformanceObserver 监听 Long Tasks

用 PerformanceObserver API 监听 Long Tasks：

new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log('Long Task:', entry.duration + 'ms', entry.attribution);
    // 发到 RUM 系统
  }
}).observe({ entryTypes: ['longtask'] });

每个 long task entry 有 attribution—— 哪个 frame / iframe 触发，但不告诉你具体代码位置。要定位代码用 DevTools Performance Recording。

Long Tasks API 的升级版：Long Animation Frames API (LoAF)（Chrome 123+）—— 不只看 long task，看整个"animation frame"（包括 layout / paint）。给出更完整的 INP 归因数据。

scheduler.yield()：让出主线程

scheduler.yield()（Chrome 129+ 稳定）让任务"让出主线程一下，立即继续"。

async function processItems(items) {
  for (const item of items) {
    process(item);  // 假设每个 process 5ms

    // 每处理 10 个 item 让出一次主线程
    if (items.indexOf(item) % 10 === 0) {
      await scheduler.yield();
      // 让浏览器处理用户输入 / 渲染下一帧
      // 然后 yield resolve，继续循环
    }
  }
}

对比传统方式：

传统：setTimeout(fn, 0)—— 让任务排到下一个 task。但下个 task 可能被其他高优先级任务挤后，恢复时间不可控。

scheduler.yield()—— 高优先级让出，浏览器处理完用户输入 + 渲染后立即恢复。延迟最低。

这是2024+ 优化 INP 的关键 API。Long task 改成 cooperative：处理一段 → yield → 继续。即使总耗时不变，INP 显著改善（用户输入不被阻塞）。

注意：scheduler.yield() 只 Chrome 129+ 支持。Safari / Firefox 未支持。Polyfill：用 setTimeout(0) 或 scheduler.postTask({ priority: 'user-blocking' }) 作 fallback。

scheduler.postTask()：精细调度

scheduler.postTask()提供3 个优先级调度任务：

user-blocking—— 阻塞用户感知的任务（如响应点击）。最高优先级。

user-visible（默认）—— 用户可见但不立即（如更新 dashboard 数据）。中等优先级。

background—— 后台任务（如 analytics / 缓存预热）。最低优先级。

// 高优先级：处理点击
scheduler.postTask(() => updateUI(), { priority: 'user-blocking' });

// 中优先级：渲染列表
scheduler.postTask(() => renderList(), { priority: 'user-visible' });

// 低优先级：发 analytics
scheduler.postTask(() => sendAnalytics(), { priority: 'background' });

浏览器自动按优先级调度——高优先级任务先跑。这套 API 替代了之前用 setTimeout 0 / requestIdleCallback / Promise.then 等不同 idiom 来表达不同优先级的混乱状态。

支持：Chrome 94+ 稳定。Safari / Firefox 未支持。Polyfill 用 setTimeout 模拟。

Event 处理链：浏览器内部时序

当用户点击按钮，浏览器内部的处理时序（影响 INP）：

(1) Input event 投递到主线程（input delay 阶段）。如果主线程在跑 long task，事件排队等。

(2) Event handler 执行（processing 阶段）。你的 onClick handler。

(3) Microtasks drain—— Promise.then 等。在 handler 之后立即跑。

(4) Rendering steps—— Style / Layout / Paint。

(5) Composite + paint to screen—— Compositor thread 完成最终输出。

INP = 步骤 (1)-(5) 的全部时间。

优化点：

(1) 减少 input delay—— 让主线程不被 long task 阻塞（scheduler.yield 让 long work cooperative）；

(2) 减少 processing—— 复杂 handler 拆分（重 work async 处理）；

(3) 减少 presentation delay—— React 重 render 用 useTransition 标记 non-urgent；layout 改动用 contain。

requestIdleCallback：低优先级任务

requestIdleCallback(fn)（旧 API，Chrome 47+）让任务在浏览器空闲时跑—— 没有 long task / 没有 pending render 的间隙。

requestIdleCallback((deadline) => {
  // deadline.timeRemaining() 给你 ms 数表示剩余空闲时间
  while (deadline.timeRemaining() > 0 && tasks.length) {
    processTask(tasks.shift());
  }
});

用例：(1) prefetch 下一页资源；(2) 预渲染 / 预算计算；(3) analytics 上报。

scheduler.postTask({ priority: 'background' }) 是更现代的替代，但 requestIdleCallback 仍广泛使用。

isInputPending：让长任务能"中途让出"

navigator.scheduling.isInputPending()（Chrome 87+）让你检查是否有 pending 用户输入。如果有，立即让出。

function processItems(items) {
  for (const item of items) {
    process(item);

    if (navigator.scheduling.isInputPending()) {
      // 有用户输入等着 → 让出主线程
      await new Promise(r => setTimeout(r, 0));
    }
  }
}

对比 scheduler.yield()：isInputPending 是条件性让出（仅在有 input 时让），yield 是无条件让出。前者更精细但需要更多代码。

React 18 Concurrent Mode：framework 级别 INP 优化

React 18 Concurrent Mode 大幅改善 INP：

(1) useTransition—— 标记 state update 为 non-urgent。React 把这种 update 放低优先级，不阻塞用户输入：

const [isPending, startTransition] = useTransition();

function handleSearch(query) {
  // 紧急：输入框立即更新
  setQuery(query);

  // 非紧急：搜索结果可以稍后渲染
  startTransition(() => {
    setSearchResults(filterItems(query));
  });
}

(2) Time slicing—— React 把大 reconcile 切成小片，每片间检查是否有 user input，有就让出。

(3) Suspense + Streaming SSR—— 数据没到时显示 fallback，不阻塞。

这套机制让 React 应用大幅改善 INP—— 大 list 渲染 / 复杂表单 / heavy 状态更新都受益。

测 INP 的工具

类似 LCP，工具链：

(1) PageSpeed Insights—— 公开页面 lab + field data。

(2) Chrome DevTools Performance Insights—— 录制后看 INP 元素 + 优化建议。

(3) web-vitals 库—— onINP((m) => ...) 收集生产真实 INP。

(4) Long Animation Frames API—— 详细归因。

实战：先 PageSpeed Insights 看 INP 数字，再用 DevTools Performance 录制慢交互，定位代码。

03 应用场景

场景 A · 用 scheduler.yield() 改善搜索框 INP

项目搜索框：用户输入 → 立即过滤 10000 个 item 显示——每次输入都触发 long task → INP 差。优化：用 scheduler.yield() 让 filter 过程 cooperative：for (const item of items) { process(item); if (i % 100 === 0) await scheduler.yield() }—— 每处理 100 个 item 让出一次。效果：用户输入不被阻塞，前几百 item 立即显示，剩下的逐步显示。INP 从 500ms+ 降到 100ms 内。注意：(1) yield 只 Chrome 支持，Safari / Firefox fallback setTimeout(0)；(2) yield 增加总时间但 INP 大幅改善——是 trade-off；(3) 配合 React 18 useTransition 效果更好。

场景 B · 排查「用户报某个按钮点击慢」

用户报"点这个按钮要等 1 秒才反应"——分层排查：(1) DevTools Performance 录制这次点击——看 INP 几个组成（input delay / processing / presentation）哪个大；(2) Input delay 大—— 主线程在跑 long task。看 long task 来源（哪个函数）。(3) Processing 大—— Event handler 太重。拆分 + scheduler.yield。(4) Presentation 大—— React 重 render 慢 / Layout 复杂。用 useTransition / contain。最常见根因是处理函数里读 layout 属性触发 forced reflow，或同步 fetch（不该有）。这套排查流程让你定位 INP 瓶颈。

场景 C · 给 React 18 项目用 useTransition 改善大 list

React 项目有大 list（10000+ items）—— 改 filter 触发整 list 重 render，主线程卡 200-500ms。改用 useTransition：filter input 的 setState 是紧急（输入框立即更新），filtered list 的 setState 是 transition（标记 non-urgent）。React 把 transition update 放低优先级，主线程不被阻塞，用户输入立即响应。实测效果：INP 从 300ms 降到 100ms 内。配合 virtualization（react-window / react-virtual）只渲染屏幕内 item，性能进一步改善。这种"useTransition + virtualization"是 React 大 list 的标配优化。

场景 D · 用 Web Worker 把重 JS 移出主线程

项目有CPU 密集任务（图像处理 / 加密 / 大量计算）—— 在主线程跑直接卡死。用 Web Worker：把任务移到 worker 跑，主线程只接收结果。具体：(1) const worker = new Worker('worker.js')；(2) worker.postMessage({ task, data })；(3) worker.onmessage = (e) => updateUI(e.data)。但 Worker 不能操作 DOM—— 计算结果传回主线程后再更新 UI。典型场景：(1) Excel-like spreadsheet 计算；(2) image filter / 处理；(3) 复杂数据 transform。Worker 让 INP 不受这些工作影响。限制：(1) postMessage 有 structured clone 开销；(2) 大数据用 SharedArrayBuffer 共享（详见 P2.2 cross-origin isolated）；(3) Worker 启动开销 ~10ms，简单任务不值得。

04 典型代码

// 用 scheduler.yield() 让长任务 cooperative
async function processBigList(items) {
  const results = [];

  for (let i = 0; i < items.length; i++) {
    results.push(processItem(items[i]));

    // 每 50 个 item 让出主线程
    if (i % 50 === 0 && i > 0) {
      // 现代写法：scheduler.yield (Chrome 129+)
      if ('scheduler' in self && 'yield' in scheduler) {
        await scheduler.yield();
      } else {
        // Fallback for Safari / Firefox
        await new Promise(r => setTimeout(r, 0));
      }
    }
  }

  return results;
}

// 调用
const results = await processBigList(items);
console.log('done, results:', results.length);

05 常见陷阱

把 INP 当成 FID 优化

有人把"INP 优化"理解为"减少 input delay"——只看 input delay 不看 processing / presentation。错误—— INP 是全链路（input + processing + presentation）。Processing（你的 handler 跑）和 presentation（重 render + layout）都计入。规避：(1) 用 web-vitals 库看 INP attribution，知道时间花在哪段；(2) 短 handler + 用 useTransition + 让 long work async 是综合优化；(3) 不要只优化 input delay 假装"INP 改善了"。

用 setTimeout(0) 替代 scheduler.yield 没有等价效果

setTimeout(0) 让任务排到下一个 task—— 但下个 task 可能被其他高优先级任务挤后，恢复时间可能 100ms+。scheduler.yield() 是"让出 + 立即恢复"，恢复延迟通常 < 10ms。在 INP 敏感场景差异显著。规避：现代 Chrome 用 scheduler.yield；fallback Safari / Firefox 用 setTimeout(0) 但接受延迟更高；测试 INP 比较两种实现差异。

用 requestAnimationFrame 替代 scheduler.yield

requestAnimationFrame(fn)是"下一帧渲染前调用"——和 yield 不同。rAF 适合动画 / 视觉更新（每帧调用），不适合"长任务 cooperative yield"。如果你长任务用 rAF 让出，每帧只跑一段，总时间可能延长很多。规避：(1) 长任务用 scheduler.yield 或 setTimeout(0)；(2) 视觉更新用 rAF；(3) 不要混用。

认为 React 18 自动解决所有 INP 问题

React 18 Concurrent Mode 改善 INP——但不自动。要正确使用 useTransition / Suspense / startTransition 才能拿到收益。如果你所有 setState 都是同步更新，React 仍会同步处理，INP 不变。规避：(1) 标记 non-urgent state update 为 transition；(2) 大 list 用 virtualization；(3) 重计算用 useMemo / useDeferredValue；(4) 用 React DevTools Profiler 看 reconcile 时间。
