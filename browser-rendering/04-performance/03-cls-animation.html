CLS + 动画性能（fps + jank）

视觉稳定性 + 流畅性。CLS（Cumulative Layout Shift）测量视口内元素位置移动量——目标 ≤ 0.1。讲透常见 CLS 来源（未声明尺寸图片 / 异步广告 / Web font 替换 / 异步注入内容）+ width/height + aspect-ratio + font-display 策略 + 60fps 硬约束（16.67ms / frame）+ Jank + DevTools Performance panel + Animation worklet / OffscreenCanvas 高级方案 + transform/opacity 黄金动画路径回顾。关键洞察：CLS 优化的核心是「占位」—— 浏览器需要知道元素的最终尺寸才能不 shift。

阅读时长 ~30 分钟 前置 P4.2 + V8 P4.2 GC（GC pause 作 jank 来源） 主要来源 web.dev/cls · web.dev/animations-guide · DevTools docs

01 速览

  1. CLS ≤ 0.1 是 "good"—— 测量页面生命周期内元素意外位移的累积量。常见来源是未声明尺寸的图片 / 异步加载的广告 / Web font 替换让文字 reflow。
  2. CLS 优化核心是「占位」—— width/height 属性 + CSS aspect-ratio + Web font 用 size-adjust 让 fallback font 与 web font 字号尺寸一致。这些预留空间让异步加载的资源不"挤"已渲染内容。
  3. 60fps 是硬约束（16.67ms/frame）—— 主线程任务超过这个就掉帧。transform/opacity 动画走 Compositor 不受主线程影响（详见 P3.3）。Jank 来源主要是 long task / GC pause / 走 layout 路径的动画。

02 底层逻辑

CLS 定义：「累积」位移量

CLS是页面整个生命周期内元素意外位移的累积量—— 不是单次 shift，是所有 shift 的累积值。

计算公式：

CLS = sum(impact fraction × distance fraction)

Impact fraction：shift 影响的视口面积占总视口的比例（0-1）。

Distance fraction：shift 距离占视口最大尺寸的比例（0-1）。

关键：用户输入触发的 shift（点击 / 滚动）不计入—— 用户预期的变化不算 unstable。500ms 内的连续 shift 累积成一次"session window"。

阈值：≤ 0.1 是 "good"，0.25+ 是 "poor"。

实测：用 web-vitals 库 onCLS((m) => ...) 收集生产数据。

CLS 5 大常见来源

(1) 未声明尺寸的图片—— <img src="..."> 不写 width/height。图片加载完后浏览器才知道实际尺寸 → 后续内容被推下去。

规避：always 写 width/height（HTML 属性，不是 CSS）：<img src="..." width="800" height="600">。浏览器根据比例预留空间。

(2) 异步加载的广告 / iframe—— 广告 SDK 异步插入 iframe，没有占位。一来就把内容推下去。

规避：广告位用固定高度 placeholder。即使广告没加载或没卖掉，placeholder 仍占位。

(3) Web font 替换—— 网页用 web font（Google Fonts），加载前用 fallback font。Web font 加载完后字符尺寸变化 → 文本 reflow → CLS。

规避：(a) font-display: optional（如果 web font 100ms 内没加载，永远不用 web font，避免替换）；(b) 用 CSS Font Loading API 的 size-adjust + ascent-override 调整 fallback font 让它与 web font 字号一致。

(4) 异步注入的 banner / cookie consent—— 顶部 cookie consent 弹出把内容推下去。

规避：(a) 顶部 banner 用 position: fixed 不影响 layout；(b) 如果必须 push 内容，固定高度 placeholder—— 加载前后高度一致；(c) 一旦显示就别让用户关闭后再次出现引发新 shift。

(5) 异步加载的 dynamic content—— React / Vue 等 SPA 异步 fetch 数据后渲染长 list，把页脚推下去。

规避：(a) skeleton screen 占位与最终内容尺寸一致；(b) 用固定高度 + scroll；(c) content-visibility: auto + contain-intrinsic-size 预留空间。

aspect-ratio：现代 CLS 优化

CSS aspect-ratio 属性（CSS 2021+）让你声明元素宽高比，浏览器自动按比例算高度——避免图片 / video / iframe 加载前后尺寸跳变：

.video {
  width: 100%;
  aspect-ratio: 16 / 9;  /* 自动算 height */
}

.avatar {
  width: 60px;
  aspect-ratio: 1;  /* 圆形头像 */
}

.responsive-image {
  width: 100%;
  aspect-ratio: 4 / 3;
  object-fit: cover;
}

这是替代 padding-bottom hack的现代写法。Padding-bottom hack 是几年前流行的占位方案：.aspect-16-9 { position: relative; padding-bottom: 56.25% }—— 写起来累且语义不清。aspect-ratio 一行解决。

支持：Chrome 88+ / Safari 15+ / Firefox 89+。2026 年全引擎稳定。

font-display 策略

@font-face 的 font-display 属性控制 web font 加载期间的行为：

值 行为 CLS 影响
auto 浏览器默认（多数 = block） 有 CLS（替换时 shift）
block 3s 内不显示（show invisible），3s 后用 fallback；web font 来后立即替换 有 CLS（替换 shift）
swap 立即用 fallback；web font 来后替换 有 CLS（替换 shift）+ FOUT（Flash of Unstyled Text）
fallback 100ms 内不显示，之后 fallback；3s 内 web font 来替换，否则永远 fallback 少 CLS（多数情况不替换）
optional 100ms 内不显示，之后 fallback；web font 100ms 内没来就永远不用 无 CLS（不替换）

实战推荐：

(1) font-display: optional—— 如果 web font 是装饰用（不影响品牌识别），用这个。100ms 内没加载就用 fallback，无 CLS。

(2) font-display: swap + size-adjust—— 如果 web font 是品牌识别核心（如自家 logo font），用 swap + 调整 fallback 字号让 swap 时不 shift。

@font-face {
  font-family: 'MyFont';
  src: url('/myfont.woff2');
  font-display: swap;
}

/* fallback font 调整尺寸匹配 MyFont */
@font-face {
  font-family: 'MyFont-fallback';
  src: local('Arial');
  size-adjust: 105%;  /* Arial 比 MyFont 略小，调到 105% */
  ascent-override: 90%;
  descent-override: 22%;
  line-gap-override: 0%;
}

body {
  font-family: 'MyFont', 'MyFont-fallback', sans-serif;
}

这套配置让 fallback font 视觉与 web font 接近，swap 时几乎无 shift。

60fps 硬约束

显示器主流刷新率是 60Hz（部分游戏 / mobile 设备 90Hz / 120Hz / 144Hz）。每帧的预算：

60Hz: 16.67ms / frame

90Hz: 11.11ms / frame

120Hz: 8.33ms / frame

浏览器要在每帧预算内完成所有工作（Style / Layout / Paint / Composite + JS）才能不掉帧。

其中浏览器自家工作约占 4-6ms（depending on complexity）。JS 的预算只有 ~10ms / frame。如果你的 frame 内 JS 超 10ms，下一帧延迟。连续多帧 jank（不顺滑）。

实战意义：

(1) scroll handler 必须 < 10ms—— 否则 scroll 卡。

(2) 动画每帧的工作必须 < 10ms—— 否则动画掉帧。

(3) 大计算用 Worker 或 yield—— 不能在主线程 hot path 跑。

这是现代 web 性能的硬约束—— 不能突破物理（屏幕刷新率）。

Jank：可感知的卡顿

Jank是用户感知到的卡顿—— 通常是连续多帧 > 16.67ms。常见来源：

(1) Long Task 主线程—— 详见 P4.2。

(2) GC pause—— V8 GC 可能 stop-the-world 几十 ms。详见 V8 P4.2。Orinoco 大幅降低暂停，但 major GC 仍可能 pause。

(3) 走 layout 路径的动画—— width / height / margin 改动 → 触发 reflow + paint + composite，每帧 50ms+。

(4) Forced reflow—— JS 在循环里读 layout 属性，每次强制 layout（详见 P3.2）。

(5) GPU driver 慢—— 极少情况但确实有，硬件加速没启用导致 composite 走 CPU。

诊断 jank：DevTools Performance Recording → 看 timeline 上每帧时间 → 红色 / 橙色帧是 jank。

DevTools Performance panel：jank 排查工具

Chrome DevTools Performance panel 是jank 排查的核心工具。常用功能：

(1) Recording—— 录制一段时间的性能数据。包括 JS / Layout / Paint / Composite / Network。

(2) Frame chart—— 每帧的耗时可视化。绿帧 = 顺；橙帧 = 慢；红帧 = jank。

(3) Main thread flame chart—— 主线程的函数调用栈，看每个函数耗时。

(4) Network—— 资源加载时序。

(5) Bottom-up / Call tree—— 聚合调用，找最耗时的函数。

(6) Performance Insights（Chrome 102+ 新工具）—— 自动分析录制 + 给出优化建议（"Forced reflow" / "Long task" / "INP candidate" 等）。

实战流程：

(1) 重现卡顿场景；

(2) DevTools 开 Performance panel + 点 Record；

(3) 操作页面（点击 / 滚动 / 输入）；

(4) 停 Record，看时间线；

(5) 找 jank 帧 → 看是什么原因（long task / forced reflow / 大 paint）；

(6) 修复 + 再 Record 验证。

CSS Animations vs JS Animations

CSS animations 和 JS-driven animations 性能不同：

CSS Animations / Transitions：浏览器原生支持。如果是 transform/opacity 动画 → 走 Compositor thread，主线程不影响。如果是 layout 属性 → 走主线程，受 jank 影响。

JS animations（requestAnimationFrame）：JS 控制每帧。即使做 transform 改动，仍要在主线程跑 JS → 主线程卡时动画卡。但 JS 灵活——能做物理引擎 / 复杂插值。

Web Animations API (WAAPI)：JS API 但底层走 Compositor（如果属性合适）。比 requestAnimationFrame 更优。

// Web Animations API：JS 控制 + Compositor 加速
element.animate(
  [
    { transform: 'translateX(0)' },
    { transform: 'translateX(500px)' }
  ],
  { duration: 1000, iterations: Infinity }
);
// 这种走 Compositor，主线程不影响

实战推荐：

(1) 简单动画用 CSS transition / animation——transform/opacity 走 Compositor。

(2) 复杂插值用 WAAPI——比 requestAnimationFrame 性能好。

(3) 物理 / 游戏用 requestAnimationFrame + canvas——主线程不可避免。

Animation Worklet + OffscreenCanvas：高级方案

对于极复杂动画（科学可视化 / 大型 dashboard / 游戏）—— 主线程根本承担不了。两个高级 API：

Animation Worklet（Chrome 实验，未稳定）—— JS 在独立线程跑动画逻辑。比 Worker 轻量，专为动画设计。在 Compositor thread 跑。

OffscreenCanvas（Chrome 69+ / Firefox 105+ / Safari 16.4+）—— 把 canvas 移到 Worker。Worker 用 OffscreenCanvas 的 2D / WebGL context 渲染 → 不阻塞主线程。

// 主线程：把 canvas 移交给 Worker
const canvas = document.querySelector('canvas');
const offscreen = canvas.transferControlToOffscreen();
const worker = new Worker('canvas-worker.js');
worker.postMessage({ canvas: offscreen }, [offscreen]);

// canvas-worker.js
self.onmessage = (e) => {
  const ctx = e.data.canvas.getContext('2d');
  function frame() {
    // 在 worker 跑渲染，主线程不被阻塞
    ctx.clearRect(0, 0, e.data.canvas.width, e.data.canvas.height);
    ctx.fillRect(...);
    self.requestAnimationFrame(frame);
  }
  frame();
};

用例：复杂数据可视化、游戏、AR/VR、实时图像处理。OffscreenCanvas 让 60fps 动画即使主线程满载也能保持。

CrUX 真实数据 + 业务关联

CLS 优化的业务效果：

电商场景的研究：CLS 从 0.25 降到 0.05 通常带来：

(1) bounce rate 降低 5-10%；

(2) session duration 增加 10-15%；

(3) conversion rate 提升 2-5%（电商场景）。

原因：CLS 让用户烦——点击按钮但内容跳走点错；阅读到一半内容跳让人懊恼。这种"小烦恼"累积起来影响用户留存。

这是"性能即业务"的好例子——CLS 不是技术 metric 是用户体验 metric，直接影响业务指标。

03 应用场景

场景 A · 给电商网站做 CLS 全面优化

电商页面 CLS 0.3 (poor)——分层修复：(1) 所有图片加 width/height + aspect-ratio—— 商品图 / 用户头像 / banner 全部声明尺寸。(2) 广告位预留固定高度—— 即使没卖出广告也保留 placeholder。(3) Web font 用 font-display: optional 或 swap + size-adjust—— 减少替换 shift。(4) 动态注入的 banner（折扣弹窗）用 position: fixed—— 不影响 layout。(5) Skeleton screen 占位—— 异步加载的商品列表用 skeleton 占位，与最终高度一致。具体 ROI：CLS 从 0.3 降到 0.05，conversion 提升 3-7%（电商场景实测）。

场景 B · 排查「我们的 LCP 好但 CLS 差」

LCP 1.5s (good) 但 CLS 0.2 (needs improvement)——典型场景：内容加载快但布局不稳。排查：(1) 用 web-vitals 库 + LayoutShift attribution—— 看每次 shift 的源元素 / 时机；(2) 定位 shift sources—— 通常是图片 / 广告 / web font / cookie banner；(3) 逐个修复—— 每修一个 source，重测 CLS。常见结论：80% CLS 来自 1-3 个特定源（如某个第三方广告 SDK 或某个未声明尺寸的图片）。修复优先级：先修最大来源（用 attribution data 排序），再修小的。

场景 C · 给做实时 dashboard 的 60fps 动画

项目是实时数据 dashboard（金融 / 监控）—— 数据频繁更新 + 动画展示变化。要保持 60fps。关键策略：(1) 数据计算用 Web Worker—— 主线程只接收结果更新 UI；(2) 大量 DOM 更新用 virtualization—— 只渲染屏幕内的 row；(3) 动画用 transform / opacity—— Compositor 加速；(4) Long task 用 scheduler.yield—— 让 critical update 不被卡；(5) 用 DevTools Performance Recording 持续监控帧时间，确保 60fps。如果数据极复杂，OffscreenCanvas + Worker 是终极方案——主线程几乎只做 input 处理。

场景 D · 给团队建立「性能预算」机制

团队规模化后需要持续监控性能 — 引入"性能预算"机制：(1) 设定 baseline—— LCP < 2.5s, INP < 200ms, CLS < 0.1（核心 Web Vitals "good"）。(2) CI 集成 Lighthouse CI—— 每个 PR 跑 Lighthouse，超阈值阻塞 merge。(3) 生产 RUM 持续监控—— web-vitals 库 + 自家 RUM，每天看 P75 数据。(4) 设性能 OKR—— Q1 目标 INP P75 < 150ms 等。(5) 性能回归报警—— 某个版本上线后 P75 LCP 涨 200ms+ → 报警，找回归原因。这套机制让"性能"从"偶尔注意"变成"持续保证"—— 是规模化团队 web 性能保持的关键。

04 典型代码

javascript 
// 收集 CLS + attribution
import { onCLS } from 'web-vitals';

onCLS((metric) => {
  console.log('CLS:', metric.value, metric.rating);

  // attribution 给每次 shift 的详情
  for (const entry of metric.attribution.entries || []) {
    console.log('shift source:', entry.sources);
    console.log('shift value:', entry.value);
    console.log('time:', entry.startTime);
  }

  // 发到 RUM
  navigator.sendBeacon('/api/rum', JSON.stringify({
    name: 'CLS',
    value: metric.value,
    rating: metric.rating,
    attribution: metric.attribution,
  }));
});

// 监听 LayoutShift events 直接（更精细）
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (!entry.hadRecentInput) {
      console.log('LayoutShift:', entry.value, entry.sources);
    }
  }
}).observe({ entryTypes: ['layout-shift'] });

05 常见陷阱

用 CSS width/height 替代 HTML 属性

有人写 <img src="..."> 不写 width/height attribute，但 CSS 设了 img { width: 100%; height: auto }—— 仍然有 CLS。原因：浏览器需要在HTML parse 阶段就知道宽高比预留空间，CSS 要 parse 完才生效。规避：always 在 HTML img tag 上写 width="800" height="600"—— 浏览器 parse 时立即用比例预留空间。CSS 仍可以 override 让它 responsive。

font-display: swap 不配 size-adjust

用 font-display: swap 但不调 fallback 尺寸——swap 时仍有明显 CLS。正确：swap 必须配 size-adjust + ascent-override 让 fallback 与 web font 字号匹配。Google Fonts 现在自动生成这种 fallback declaration（叫 "Adjusted size declarations"），可以参考 / 借用。

把 Animation 在主线程跑造成 jank

用 requestAnimationFrame 写动画，逻辑在主线程跑——主线程被其他工作卡时动画卡。规避：(1) 简单动画用 CSS transition / animation（transform/opacity 走 Compositor）；(2) 复杂插值用 Web Animations API（也走 Compositor 如果属性合适）；(3) 极复杂动画用 OffscreenCanvas + Worker。requestAnimationFrame 仅适合"主线程不可避免的动画"（如 canvas 游戏）。

用 lab data 优化但不验证 field data

在 DevTools 本地测 CLS / 60fps 动画都好——但用户报卡。原因：你的本地是高端机器 + 快网络，用户可能低端机器 + 慢网络。规避：(1) DevTools throttling（CPU 4x slowdown / 3G 网络模拟）测低端体验；(2) 真机测试低端 mobile 设备；(3) 生产 web-vitals 库 + RUM 看 field data；(4) P75 / P95 是关键指标——P50 好不代表 P75 也好。
