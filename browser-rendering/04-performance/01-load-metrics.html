载入侧 metrics：LCP / FCP / TTFB

从 URL 输入到主要内容渲染完成的性能。讲透 LCP（最大内容绘制 ≤ 2.5s 是 "good"）+ FCP（首次内容绘制 ≤ 1.8s）+ TTFB（首字节 ≤ 800ms）+ LCP 优化路径（服务器响应 + critical CSS inline + 关键资源 preload + 图片格式 / 大小优化）+ fetchpriority + preload/preconnect/dns-prefetch 三层级 + HTTP/3 在 mobile 网络下的影响。关键洞察：LCP 提升 1s 通常带来 conversion 提升 8-20%，电商场景特别敏感——LCP 是 SEO + 业务效果的核心 metric。

阅读时长 ~30 分钟 前置 P3 主要来源 web.dev/lcp · web.dev/fcp · web.dev/ttfb · CrUX 数据

01 速览

  1. LCP 是 Web Vitals 中最直接影响业务的 metric——LCP 减 1s 通常带来 conversion 提升 8-20%（电商场景）；Google 搜索排名将 LCP 作为 ranking signal 之一。
  2. LCP 的元素通常是首屏 hero image / 主标题 / 视频海报——可预测性强。优化思路：(1) 服务器响应快（TTFB）；(2) Critical CSS inline；(3) 关键资源 preload + fetchpriority="high"；(4) 图片格式 / 大小优化。
  3. HTTP/2 vs HTTP/3 在 mobile 网络下差距明显—— QUIC + 0-RTT 让 mobile 用户的 TTFB 显著好。CDN + HTTP/3 是现代 web 性能基线。

02 底层逻辑

Web Vitals：把用户体验量化

2020 年 Google 推出 Web Vitals——三个核心 metric 量化用户体验。核心 Web Vitals（Core Web Vitals）随时间演进：

2020 版：LCP（载入）+ FID（交互）+ CLS（稳定性）。

2024 版：LCP（载入）+ INP（交互，替代 FID）+ CLS（稳定性）。INP 比 FID 更全面（FID 只看首次交互，INP 看所有交互）。

Web Vitals 的意义：

(1) SEO 影响—— Google 搜索排名将 Web Vitals 作为 ranking signal（影响小但确实有）。

(2) 业务影响—— 大量研究表明 LCP / INP 改善带来 conversion 提升。

(3) 行业标准化—— 让"网页快慢"从感性变成数字，团队能客观讨论性能。

每个 metric 有三档阈值："good"（绿色）/ "needs improvement"（橙）/ "poor"（红）。Google 建议75% 的页面浏览达到 "good"。

TTFB（Time To First Byte）：服务器响应快慢

TTFB是从浏览器发请求到收到第一个字节的时间。包括：

(1) DNS 解析—— 域名 → IP；

(2) TCP 握手—— 3-way handshake；

(3) TLS 握手（HTTPS）—— 证书验证 + key exchange；

(4) HTTP 请求 → 服务器处理 → 第一字节响应。

阈值：≤ 800ms 是 "good"，1800ms+ 是 "poor"。

TTFB 影响因素：

(1) 网络延迟—— 用户到 CDN edge 的物理距离 + 网络质量。这是 mobile 网络下 TTFB 大头。

(2) 服务器处理时间—— 业务逻辑 + DB 查询 + 模板渲染。SSR 慢的最常见原因。

(3) 协议开销—— TCP / TLS 握手。HTTP/2 是单连接，握手只一次；HTTP/3 进一步优化（0-RTT 重连）。

优化路径：

(1) 用 CDN—— 让用户连接到最近的 edge；

(2) 静态生成 / 边缘缓存—— 服务器不动态生成，TTFB ~10ms；

(3) HTTP/3—— mobile 网络下 RTT 减少；

(4) SSR 性能优化—— DB index / 缓存 / 减少嵌套 RPC；

(5) Streaming SSR—— 不等整个 HTML 生成完发，边生成边发。Next.js / Remix / SvelteKit 都支持。

FCP（First Contentful Paint）：第一个像素

FCP是浏览器第一次绘制任何内容（文字 / 图像 / SVG / canvas）的时间。比 LCP 早。

阈值：≤ 1.8s 是 "good"，3s+ 是 "poor"。

FCP 取决于：

(1) TTFB—— FCP 不可能比 TTFB 早；

(2) 关键 CSS 下载 + parse—— FCP 必须等关键 CSS（render-blocking）；

(3) JS 阻塞—— 同步 script 阻塞 first paint。

优化：

(1) Critical CSS inline—— 把首屏关键 CSS 放在 HTML head 的 <style> 标签里，不需要额外 round-trip 下载；

(2) 非关键 CSS 异步加载—— <link rel="stylesheet" media="print" onload="this.media='all'"> hack 或 modern preload 技巧；

(3) JS 用 defer / module—— 不阻塞 HTML parsing；

(4) SSR + 流式响应—— 让 HTML 顶部立即渲染。

LCP（Largest Contentful Paint）：最大内容

LCP是视口内最大内容元素绘制完成的时间。最大内容元素的判定：

(1) <img> / <svg> image / <video poster> / 背景图（background-image: url）。

(2) <p> / <h1> / 等文本块（含至少一个 child node）。

(3) 元素必须在视口内。

(4) "最大"按渲染面积算（width × height）。

阈值：≤ 2.5s 是 "good"，4s+ 是 "poor"。

典型 LCP 元素场景：

(1) 电商 / 媒体网站—— LCP 通常是 hero image / 商品大图。

(2) 博客 / 文档—— LCP 是主标题 <h1> 或第一段 <p>。

(3) Dashboard—— LCP 是首屏最大的 widget。

(4) 视频站—— LCP 是视频 poster image。

测 LCP 元素：用 web-vitals 库—— onLCP((metric) => console.log(metric.element, metric.value)) 直接报告 LCP element + 时间。

LCP 优化路径

LCP 优化的关键四步：

(1) 减少 TTFB—— 见上文。CDN + 缓存 + HTTP/3。

(2) Critical CSS inline + 非关键 CSS 异步—— FCP / LCP 都受 render-blocking CSS 影响。

(3) 关键资源 preload + fetchpriority—— 提前下载 LCP image / 关键 font。

(4) 图片优化—— 格式（AVIF / WebP）+ 大小（responsive image）+ 懒加载（below-fold）。

具体每步：

LCP image preload

LCP image 通常是 hero image。优化：

<head>
  <!-- 提前 preload，浏览器立即下载 -->
  <link rel="preload" href="/hero.jpg" as="image" fetchpriority="high">
</head>
<body>
  <img src="/hero.jpg" alt="hero" fetchpriority="high">
  <!-- 注：img 上加 fetchpriority="high" 也行 -->
</body>

效果：浏览器解析 head 时立即下载 hero.jpg，不用等到 body 的 img。LCP 通常提升 200-500ms。

图片格式优化

现代图片格式：

AVIF (2020+)—— 当前最佳压缩比。比 JPEG 小 50%+。Chrome 85+ / Firefox 93+ / Safari 16.4+ 支持。

WebP (2010)—— 比 JPEG 小 25-35%。全引擎支持（Safari 14+）。

JPEG / PNG—— 老格式 fallback。

用 <picture> 提供多格式：

<picture>
  <source srcset="/hero.avif" type="image/avif">
  <source srcset="/hero.webp" type="image/webp">
  <img src="/hero.jpg" alt="hero" width="1200" height="600">
</picture>

浏览器选第一个支持的格式。Chrome → AVIF, Safari 14 → WebP, IE → JPEG。

Responsive image：不下载用户用不到的尺寸

不要给所有用户下载同一张大图——给 mobile 小图，desktop 大图：

<img
  srcset="
    /hero-400.jpg 400w,
    /hero-800.jpg 800w,
    /hero-1600.jpg 1600w,
  "
  sizes="(max-width: 600px) 400px,
         (max-width: 1200px) 800px,
         1600px"
  src="/hero-800.jpg"
  alt="hero"
  fetchpriority="high"
>

浏览器根据 viewport 选合适尺寸。Mobile 用户拿 400px 图，desktop 拿 1600px 图。Mobile 节省的字节量巨大。

Lazy loading：below-the-fold 图片不立即下

<img src="/below-fold.jpg" loading="lazy" alt="...">
<iframe src="..." loading="lazy"></iframe>

loading="lazy" 让浏览器在元素接近视口时才下载。可以省下大量首屏带宽给真正关键的资源（LCP image）。

但首屏图片不要 lazy—— LCP image 需要 loading="eager" + fetchpriority="high"。

fetchpriority 完整用法

fetchpriority 属性（Chrome 102+）让你显式调整资源优先级。值：

high—— 高于浏览器默认。给 LCP image / 关键 font / above-fold 资源用。

low—— 低于浏览器默认。给 below-fold image / 第三方分析 SDK 用。

auto（默认）—— 浏览器自决。

具体场景：

<!-- LCP image：高优先级 -->
<img src="/hero.jpg" fetchpriority="high" alt="...">

<!-- below-fold image：低优先级 -->
<img src="/footer.jpg" fetchpriority="low" loading="lazy" alt="...">

<!-- 关键 font preload -->
<link rel="preload" href="/font-bold.woff2" as="font"
      type="font/woff2" crossorigin fetchpriority="high">

<!-- 第三方分析（异步 + 低优先级）-->
<script async fetchpriority="low" src="https://gtm.example.com/..."></script>

preconnect / dns-prefetch / preload：三层 hint

不同层级的资源 hint：

Hint 含义 成本 用例
dns-prefetch 提前 DNS resolve 极低 不重要的第三方 origin
preconnect 提前 TCP + TLS 握手 低（一次握手） 关键第三方 API / CDN
preload 提前下载具体资源 中（占带宽） LCP image / 关键 CSS / font
prefetch 下次导航的资源 低（idle 时下） 下一页可能用到的资源

用法层级：dns-prefetch < preconnect < preload，从轻到重。一般首屏关键资源用 preload，第三方 origin 用 preconnect，不重要的第三方用 dns-prefetch。

注意 preconnect 不要滥用——浏览器限制最多 ~6-10 个 preconnect origin，超过被忽略。

HTTP/2 vs HTTP/3 协议层影响

HTTP/2 (2015)：

(1) 单连接多路复用——一个 TCP 连接处理多个请求；

(2) HEADERS 压缩 (HPACK)；

(3) Server Push（已废弃）。

对 LCP 影响：连接复用让多资源下载快，但 mobile 网络高延迟下单 TCP 仍受 head-of-line blocking 影响（一个丢包卡所有 streams）。

HTTP/3 (2022)：

(1) 基于 QUIC（UDP 上）—— 不是 TCP；

(2) 没有 head-of-line blocking——每个 stream 独立；

(3) 0-RTT 重连——熟悉的 server 重连不需要新握手；

(4) 更好的 connection migration——切换 WiFi / 4G 不断连接。

对 LCP 影响：mobile 网络下显著改善（10-30%）。WiFi / 良好网络下差距小（< 5%）。

实操：用 CDN 提供的 HTTP/3 支持（Cloudflare / Fastly / CloudFront 都默认开）。Server 自家配置 HTTP/3 仍较复杂。

测 LCP 的工具链

测 LCP 的工具：

(1) Field data（实地数据）：CrUX—— Chrome User Experience Report，Google 收集真实用户的 Web Vitals。是 SEO ranking signal 的源数据。

(2) PageSpeed Insights—— Google 综合 lab + field data 的工具。给出 LCP / FCP / TTFB 等数字 + 优化建议。

(3) Lighthouse—— Chrome DevTools 内置 + 独立 CLI。Lab 数据（受 lab 环境限制，不一定反映用户实际体验）。

(4) web-vitals 库—— Google 出品的 JS 库，在生产收集真实用户 LCP/FCP/etc.，发到自家 RUM (Real User Monitoring) 系统。

(5) Chrome DevTools Performance Insights—— Performance panel 录制后看 "LCP element" + 优化建议。

实战流程：(1) PageSpeed Insights 看公开页面 lab + field data；(2) DevTools Performance 录制看 timeline 找 LCP 元素；(3) web-vitals 库收集生产真实数据。

03 应用场景

场景 A · 给电商首页做 LCP 优化（最经典场景）

电商首页 LCP 是 hero banner image / 商品大图——这是 LCP 优化的最典型场景。具体步骤：(1) 识别 LCP element—— 用 PageSpeed Insights 或 web-vitals 库；(2) preload + fetchpriority="high"—— 让浏览器立即下载；(3) 用 AVIF / WebP 格式—— responsive image (srcset/sizes)；(4) 用 CDN + HTTP/3—— 全球用户都快；(5) 测试 + 监控—— 生产部署后用 web-vitals 库监控真实 LCP，确认改善生效。典型效果：LCP 从 4s 降到 1.5s，conversion 提升 8-15%。这是 ROI 最高的性能优化之一。

场景 B · 排查「为什么我们的 LCP 仍然慢」

已经做了 preload + 图片优化，LCP 仍 > 3s——分层排查：(1) TTFB 慢？— 看 PageSpeed Insights TTFB 数字。> 800ms 是源头问题，要先解决（CDN / SSR 优化）。(2) FCP 慢？—— 看是不是 render-blocking CSS / 同步 JS。Critical CSS inline + JS defer。(3) FCP 快但 LCP 慢？—— LCP image 没 preload；或 preload 了但被其他资源挤占带宽。检查 fetchpriority。(4) image 下载完但绘制慢？—— 图片格式不对（PNG 用了 1MB）；或 image 在独立 layer 触发额外 paint。换 AVIF + 检查 layer。每层都用 DevTools Network + Performance panel 验证。

场景 C · 给 SSR 框架项目用 streaming SSR

项目用 Next.js / Remix / SvelteKit 做 SSR——传统模式 server 等所有数据生成完整 HTML 再发，TTFB / LCP 慢。Streaming SSR解决：(1) Server 立即发 HTML head + above-fold shell；(2) 慢数据用 Suspense fallback 占位；(3) 数据来后流式发到浏览器。实操：Next.js App Router 默认 streaming（用 Suspense boundary）；Remix 用 defer；SvelteKit 用 streaming load function。效果：TTFB 不变（仍要等服务器 setup），但 FCP / LCP 显著快——shell 立即可见，数据后续填充。这是现代 SSR 框架的标配优化。

场景 D · 用 web-vitals 库收集生产 LCP 数据

项目要持续监控生产 Web Vitals——用 Google web-vitals 库 + 自家 RUM 系统：onLCP((m) => sendToAnalytics(m)) 收集真实用户 LCP，发到 Datadog / NewRelic / 自家系统。关键设计：(1) 采样率—— 每个 metric 不一定都发（高流量 site 可能 1%-10% 采样）；(2) 用户分群—— 按 device / network / country 分群，看不同维度的 LCP 分布；(3) P75 / P95 是关键—— Web Vitals 阈值是 P75（75% 用户达到 "good"）；P95 反映长尾用户体验；(4) 关联业务 metric—— LCP 改善对 conversion / bounce rate 的影响。这套监控让你"性能改善"和"业务效果"建立量化关系。

04 典型代码

javascript 
// 用 web-vitals 库收集 Web Vitals
import { onLCP, onFCP, onCLS, onINP, onTTFB } from 'web-vitals';

function sendToAnalytics(metric) {
  // 发到你的 RUM 系统
  navigator.sendBeacon('/api/rum', JSON.stringify({
    name: metric.name,
    value: metric.value,
    id: metric.id,
    delta: metric.delta,
    rating: metric.rating,  // 'good' / 'needs-improvement' / 'poor'
    navigationType: metric.navigationType,
    url: location.href,
  }));
}

onLCP(sendToAnalytics);
onFCP(sendToAnalytics);
onCLS(sendToAnalytics);
onINP(sendToAnalytics);
onTTFB(sendToAnalytics);

// LCP 还能拿到具体元素
onLCP((metric) => {
  console.log('LCP element:', metric.element);
  console.log('LCP value:', metric.value, 'ms');
  console.log('rating:', metric.rating);
});

05 常见陷阱

忘记 preload LCP image

项目首屏有 hero image 但没 preload——LCP 显著慢。原因：浏览器要 parse HTML → 找到 img tag → 才开始下载。如果 HTML 大或 head 有 render-blocking CSS，img 下载推迟几百 ms。规避：识别 LCP element + preload。Web Vitals 库 / PageSpeed Insights 都告诉你 LCP element 是哪个，照着 preload。

所有图片用 lazy loading 包括 LCP

有人看到 lazy loading 是优化，给所有图片加 loading="lazy"—— 包括 LCP image。反效果：浏览器要计算 viewport 才知道图片是否需要下载，比立即下载慢。规避：LCP image / above-fold image 不要 lazy，用 loading="eager"（默认）+ fetchpriority="high"。Below-fold image 才 lazy。

background-image LCP 不能用 preload 优化

LCP element 是 CSS background-image: url('/hero.jpg') 的——浏览器要先 parse CSS 才知道这张图要下载。preload 仍然能用但效果有限，因为 preload 后浏览器不知道这是 LCP candidate。规避：(1) 把 hero 改成 <img>—— LCP candidate 算法对 img 友好；(2) 仍要 preload + fetchpriority；(3) 如果必须用 background-image，CSS 也要 inline 进 critical CSS（不用额外下载）。

用 lab data 而不看 field data

开发者用 Lighthouse 测 LCP 都是 "good"——但field data（真实用户）显示 LCP 差。原因：Lighthouse 是 lab 环境（固定 CPU / 网络），不反映真实用户。真实用户用各种设备 + 网络条件，LCP 分布跨度大。规避：(1) 优化时看 PageSpeed Insights field data；(2) 生产部署后用 web-vitals 库 + RUM 系统持续监控；(3) Lab data 用于"是否引入回归"（CI 测试 baseline），field data 用于"用户实际体验"。
