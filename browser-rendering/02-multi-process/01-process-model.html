Chromium 进程模型：Browser / Renderer / GPU / Network / Utility

Chrome 2008 年立项的核心创新：多进程架构——把网页 / 浏览器 UI / GPU / 网络各放独立进程。这一章讲透 5 类进程的职责、Mojo IPC 通信机制、主线程 vs Compositor thread 分工。关键洞察：Renderer 内部「主线程一切」是 INP / Long Tasks 的来源——JS / Layout / Paint 都在主线程跑，long task 卡所有渲染。这条认识是后面 P4 性能章节的基础。

阅读时长 ~25 分钟 前置 P1.2 + Node P5.1 多核（多进程概念对照） 主要来源 chromium.org/developers/design-documents/multi-process-architecture

01 速览

  1. 多进程架构是 Chrome 击败 Firefox / IE 的核心创新——2008 年其他浏览器都是单进程，Chrome 一个 tab 崩不影响其他、恶意网页不能访问系统资源、多核可利用。
  2. 5 类进程：Browser / Renderer / GPU / Network / Utility，各自职责清晰。Browser 是 UI + 协调，Renderer 跑网页，GPU 合成屏幕，Network 处理所有请求，Utility 跑临时任务。
  3. Renderer 内部「主线程一切」——JS、Layout、Paint 都在 Renderer 主线程跑；Compositor thread 单独跑 GPU 合成。这种分工决定了为什么 transform/opacity 动画顺滑（走 Compositor）而 width/height 动画卡（走主线程）。

02 底层逻辑

为什么多进程：Chrome 2008 的设计动机

2008 年其他浏览器（Firefox / Safari / IE）都是单进程——一个浏览器 = 一个 OS process，所有 tab + UI + plugin 共享同一进程地址空间。这有三个根本问题：

(1) 安全：恶意网页能通过浏览器漏洞访问系统资源（fs / network / 进程间通信），整个 OS 都暴露。IE 6 时代 ActiveX 让任何网页跑 native 代码，间谍软件泛滥。

(2) 稳定：一个 tab 的 JS bug / 内存崩溃 → 整个浏览器挂。常见情况：用户开 20 个 tab 工作，某一个网页用 Flash 卡死 → 全部 tab 关闭重启 → 工作丢失。

(3) 性能：单进程不能利用多核。2008 年消费级 CPU 已经普遍 2-4 核，但浏览器只跑在一个核上。一个 CPU 密集任务卡所有 tab。

Chrome 2008 推出多进程架构解决这三个问题：每个 tab 独立 Renderer 进程，跑在沙箱内（不能访问系统资源），主进程协调 UI。如果一个 Renderer 崩溃 → 显示「Aw, snap!」错误页面但其他 tab 不受影响。多个 Renderer 并行跑利用多核。

这套架构 2008 年是革命性的——其他浏览器陆续跟进：Firefox 2017 「Quantum」、Safari macOS 2014、IE 退出后 Edge 直接 Chromium。多进程已经是现代浏览器的事实标准。

Browser 进程：UI + 协调中枢

Browser process（也叫 main process）是浏览器的唯一主进程——一个浏览器实例只有一个 Browser process。职责：

(1) UI 控制：菜单 / 地址栏 / 标签栏 / 设置面板 / 下载管理 / 书签 / 历史。整个浏览器外壳都是 Browser process 跑的。

(2) 进程协调：决定何时创建 / 销毁 Renderer 进程，把每个 tab 分配给哪个 Renderer，路由 IPC 消息。所有跨进程通信都经过 Browser process 中转（或建立 direct channel 后绕过）。

(3) 系统级操作：访问 OS API（文件对话框 / 系统通知 / 安装 PWA / 注册 protocol handler）。Renderer 进程在沙箱内不能直接访问系统，要请求 Browser process 代为执行。

(4) 安全策略：决定哪些请求允许、哪些 origin 隔离、CSP / CORS / sandbox 等安全策略的执行点。

Browser process 是不能崩溃的关键进程——如果 Browser 崩了，整个浏览器关闭。所以 Chrome 团队对 Browser 代码质量要求极高，新功能默认放 Renderer 或 Utility 进程，不放 Browser。

Renderer 进程：跑网页内容

Renderer process是跑网页的进程——HTML/CSS/JS 在这里执行，Layout / Paint 在这里发生。每个 tab 通常 1 个 Renderer（Site Isolation 后每个 origin 独立）。

Renderer 内部结构：

主线程（Main thread）：跑大部分工作——HTML parser、CSS parser、Style 计算、Layout、Paint、JS 执行、DOM 操作、event 处理。这是「Renderer 主线程一切」的来源。

Compositor thread：独立线程跑 GPU 合成。接受主线程的 paint records → 合并 layer → 上传 GPU → 显示。不会被主线程阻塞——所以 transform/opacity 动画即使主线程卡也能保持 60fps（详见 P3.3）。

Worker threads：Web Workers / Service Worker 跑在独立线程，不阻塞主线程。但 DOM 操作仍只能在主线程。

Raster threads：把 paint records 真正栅格化为像素。多个 raster thread 并行处理不同 tile（详见 P3.3 tile-based rasterization）。

Renderer 跑在沙箱（sandbox）内——不能访问 fs / 不能起子进程 / 不能直接调系统 API。要做这些操作必须通过 IPC 请求 Browser process 代为执行。这是 Chrome 安全模型的核心。

Renderer 内部跑1 个 V8 Isolate—— V8 的隔离单元（详见 V8 P5.1 嵌入 V8）。一个 Isolate 包含完整的 V8 状态（heap / context / 内置对象）。Worker / iframe 等可能有独立 Isolate。

GPU 进程：合成 + 硬件加速

GPU process是专门负责 GPU 操作的进程——只 1 个，所有 Renderer 共享。职责：

(1) 合成（Compositing）：把所有 Renderer 提交的 layer tree 合成成最终屏幕图像。详见 P3.3。

(2) 硬件加速绘制：把 paint records 用 GPU 栅格化（GPU raster，2020+ 默认开启）。比 CPU raster 快 5-10x。

(3) WebGL / WebGPU：浏览器的 3D 图形 API 都在 GPU process 实现。

(4) 视频解码：硬件视频解码器在 GPU process。

GPU 进程独立的好处：(1) GPU driver 崩溃（很常见）不会影响 Browser 或 Renderer——整个浏览器仍然可用，只是动画 / 视频暂时降级到 CPU；(2) 可以共享给所有 Renderer，避免每个 Renderer 单独管理 GPU 资源。

Chromium 还在演进 OOP-D（Out-of-Process Display Compositor）——把 display compositor（最终把所有 Renderer 的合成结果输出到屏幕的组件）独立到 GPU process 或新进程。这让 Renderer 崩溃时屏幕不闪烁。

Network 进程：所有网络请求集中

Network process处理所有网络请求—— HTTP / HTTPS / WebSocket / WebRTC（部分） / DNS。Chrome 2017 之后独立出来（之前 Network 是 Browser process 的一部分）。

独立的好处：

(1) 安全：网络代码（TLS / HTTP parser / cookies）历史上 bug 多，独立沙箱让漏洞不能直接访问其他进程数据。

(2) 资源管理：所有请求集中管理 → 连接池复用、HTTP/2 多路复用、QUIC 连接共享、缓存共享。

(3) 多 Renderer 共享：所有 Renderer 的请求都走 Network process，连接 / 缓存 / cookies 共享。

Network process 用 Mojo IPC 与其他进程通信——Renderer 调 fetch() 时实际是发 Mojo 消息给 Network，Network 处理 + 响应。这套机制对 fetch / XHR API 透明。

Utility 进程：临时任务的隔离沙箱

Utility processes是每个临时任务一个进程——音频解码 / 图像解码 / PDF 渲染 / Spell check / Print preview / Storage / IndexedDB 等。任务完成后进程销毁。

这套设计的动机：

(1) 隔离：每个解码 / 渲染任务有独立沙箱，恶意文件（畸形 PNG / 恶意 PDF）不能访问主进程数据。

(2) 资源：任务完成立即销毁进程释放内存——比让 Renderer 长期持有解码状态更高效。

(3) 解耦：每个 Utility 类型可以独立优化 / 升级 / 重启，不影响其他。

Utility 进程数量随用户行为波动——一个时刻可能有几十个 Utility 进程跑（每个 PDF / 视频 / 图片解码一个），用户关掉相关 tab 后逐渐释放。

Mojo IPC：进程间通信框架

这么多进程怎么通信？答案是 Mojo—— Chromium 自家 IPC 框架（2014+ 替代老的 Chrome IPC）。

Mojo 核心特性：

(1) message passing：进程间发送结构化消息（类似 Protocol Buffers），跨语言 / 跨平台。

(2) shared memory：大数据（image bitmap / video frame）通过共享内存零拷贝传递，不走消息队列。

(3) 接口绑定：每个进程暴露的接口用 .mojom 文件声明，编译期生成 binding 代码。

(4) 通用 vs 特定通道：进程之间可以建立 generic broker（走 Browser 转发）或 direct broker（绕过 Browser 直连）。direct broker 性能高但建立有开销。

Mojo 是 Chromium 多进程架构的基石——所有跨进程调用都用 Mojo。Web 开发者通常感知不到 Mojo（fetch / postMessage / requestAnimationFrame 等 API 在浏览器内部转化为 Mojo 调用），但理解 Mojo 能让你看懂 Chromium 源码（如果你想贡献 Chromium）。

多进程的代价

多进程架构有成本：

(1) 内存翻倍：每个 Renderer 独立 V8 + 自家堆 + 标准库。Site Isolation 后每个 origin 独立 Renderer，内存可能 100-200MB / origin。开 20 个 tab 用 4GB+ 内存是常态。

(2) 启动慢：创建新进程比创建新线程慢得多（OS 层 fork + setup 沙箱 + 加载 V8 等）。所以 Chrome 用进程预热—— 后台维护 1-2 个空 Renderer，新 tab 直接用预热的进程。

(3) IPC 开销：跨进程调用比函数调用慢几个数量级。Mojo 优化得好但仍有开销。这就是为什么 Chrome 在低端机器（< 4GB 内存）上有时不如单进程浏览器顺畅。

Chrome 团队的应对：动态调整进程数（系统资源紧张时把多个 site 合并到同一 Renderer）、Renderer 进程复用（关 tab 不立即销毁进程）、内存压缩（把不活跃 tab 的内存压缩或 swap）。

主线程的重要性 + INP / Long Tasks 的来源

Renderer 主线程是所有 Web 性能问题的核心。它跑：

(1) JS 执行—— 你的应用代码 + framework（React / Vue / etc.）

(2) Style 计算—— 给每个元素算 computed style

(3) Layout—— 算每个元素的位置和尺寸

(4) Paint—— 生成 paint records（不是真绘制，真绘制在 raster thread）

(5) DOM 操作—— 用户代码改 DOM 触发 invalidation

(6) Event 处理—— 鼠标 / 键盘 / 触摸事件分发

这些任务串行跑——一个任务长（> 50ms）就是 Long Task，期间无法处理用户输入 / 渲染下一帧。这是 INP（Interaction to Next Paint）metric 的根本来源（详见 P4.2 INP / Long Tasks）。

对比 Compositor thread——它独立运行，不被主线程阻塞。所以：

✅ transform / opacity 动画走 Compositor → 即使主线程卡 100ms，动画仍 60fps 顺滑。

❌ width / height / left / top 动画走主线程 → 主线程卡 → 动画掉帧。

这条主线程 vs Compositor thread 分工是 Web 性能优化的核心模式—— 把动画放 Compositor 路径，把重 JS 放 Worker 或 scheduler。详见 P4.

查看进程情况：Chrome 内置工具

实操工具：

(1) chrome://process-internals—— 看当前所有进程的列表 + 类型 + 关联 URL。能看到 site isolation 是否生效。

(2) chrome://memory-internals—— 详细内存分布。

(3) Task Manager（Shift + Esc）—— 浏览器内置任务管理器，看每个进程的 CPU / 内存。

(4) chrome://gpu—— GPU 进程状态、driver 版本、硬件加速是否启用。

这些工具是诊断浏览器性能 / 内存 / 安全问题的入口。每个 Web 开发者应该知道这些路径。

03 应用场景

场景 A · 解释「为什么 transform 动画顺滑而 width 动画卡」

团队成员问"我用 width 做尺寸动画卡，改成 transform: scale 就顺滑，为什么"——多进程架构给出明确解释。Width 动画触发 Layout（需要重新计算所有元素位置）→ Layout 在主线程跑 → 如果你的页面同时有 JS / DOM 操作，主线程被阻塞 → 动画掉帧。Transform 动画不触发 Layout（transform 只改变 visual，不改变 layout box）→ 走 Compositor thread → 即使主线程被卡，Compositor 仍 60fps 跑动画 → 顺滑。这条认识是 Web 动画优化的基础。最佳实践：动画用 transform / opacity（详见 P4.3），避免 width / height / margin / padding 动画。

场景 B · 排查「Chrome 内存占用过高」

用户报"Chrome 用了 8GB 内存"——理解多进程能给出诊断框架。步骤：(1) 开 Task Manager（Shift + Esc）看每个进程的内存；(2) 看类型分布—— 大部分是 Renderer 进程？还是 GPU 进程？还是某个 Utility 进程异常？(3) Renderer 内存高—— 通常是某个 site 的网页内存泄漏（DOM 引用 / 闭包持有）。用 DevTools Heap Snapshot 排查。(4) GPU 进程内存高—— 通常是 GPU resource 泄漏（WebGL / video）。检查 layer 数 + GPU memory。(5) 很多小 Renderer—— Site Isolation 让每个 origin 独立进程，多 origin 自然多进程。这是设计而非 bug。开很多 tab 内存自然多。这套排查框架是 Chrome 内存问题的入口。

场景 C · 决策「我们的 SaaS 要不要用多 origin 架构」

团队设计 SaaS 架构——多 subdomain（app.foo.com / api.foo.com / cdn.foo.com）vs 单 domain。多进程架构对决策的影响：(1) Site Isolation 让每个 origin 独立 Renderer 进程 → 多 origin = 用户多内存；(2) 跨 origin 的 cookie / SharedArrayBuffer / Worker 受限——如果你的应用需要这些，单 origin 更顺；(3) 每个 Renderer 独立沙箱—— 多 origin 在安全上是好事（一个 origin 漏洞不影响其他）。综合：(1) 主应用（用户长时间使用）单 origin 减内存；(2) 第三方嵌入（embed / iframe）多 origin 增隔离；(3) CDN / static asset 用专门 domain（与主应用分离）。这种 origin 设计是 Web 架构的隐性决策点。

场景 D · 给做 Web 安全的工程师：理解多进程的安全边界

做 Web 安全相关项目——理解多进程是基础。Chrome 沙箱模型：(1) Renderer 沙箱—— Renderer 进程不能直接访问 fs / network / 其他进程内存，只能通过 Mojo IPC 请求 Browser；(2) Browser 是「安全协调中枢」—— 决定每个 IPC 请求是否允许，CSP / CORS 等策略在这里执行；(3) Site Isolation—— 每个 origin 独立 Renderer，恶意 site 不能直接读其他 origin 的内存；(4) GPU 沙箱—— GPU 进程也在沙箱内（虽比 Renderer 弱）；(5) Utility 沙箱—— 临时任务也在沙箱里。这套设计让浏览器是当前最复杂的「最小信任系统」—— 每个组件只能信任自己最少的范围。理解这套模型让你设计安全 Web 应用时知道哪些边界可以依赖。

05 常见陷阱

假设「主线程 = JS 单线程 = 单核」

有人推论"既然 Renderer 主线程是单线程跑 JS，那 Chrome 不能利用多核"——错。Chrome 多进程让不同 tab 跑在不同核上；Renderer 内部的 Compositor thread / Worker thread / Raster threads 也利用多核；GPU process 独立利用 GPU。Chrome 整体能很好利用多核，只是同一个 Renderer 主线程是单线程。这是设计选择（DOM 操作并发太复杂），不是限制。规避：理解多核利用的机制；不要把"主线程单线程"误读为"浏览器单核"。

Site Isolation 让多 subdomain 内存翻倍

项目用 a.foo.com / b.foo.com / c.foo.com 等多 subdomain——Site Isolation 默认让每个 origin 独立 Renderer 进程。如果你的用户同时打开多个 subdomain，内存可能翻 3-5x。注意：subdomain 是不同 origin（即使共享 root domain）。如果你的设计假设 subdomain 共享内存 / 共享 worker，Site Isolation 后不再成立。规避：(1) 评估是否真需要多 subdomain；(2) 用 single domain + path-based routing；(3) 接受多 origin 的内存代价（安全收益更重要）。

用 SharedArrayBuffer 不知道需要 cross-origin isolated

SharedArrayBuffer（SAB）允许多线程共享内存——但 Spectre 后默认禁用，只在 cross-origin isolated 网页可用（详见 P2.2）。要让你的网页 cross-origin isolated，要发 Cross-Origin-Opener-Policy: same-origin + Cross-Origin-Embedder-Policy: require-corp 两个 header。常见踩坑：项目想用 WASM 多线程（依赖 SAB）但页面没设这两个 header → SAB 报 ReferenceError → WASM 失败。规避：先发 COOP/COEP header + 验证 self.crossOriginIsolated === true，再用 SAB。

在 Renderer 进程沙箱里期待访问系统资源

Renderer 进程在沙箱内——不能直接访问 fs / 子进程 / 系统设置。Web API 看起来访问"系统"（fetch / Notification / Geolocation）实际是通过 IPC 请求 Browser process 代为执行。这层隐性请求有性能开销 + 受 Browser 安全策略限制（CORS / CSP / permission prompt）。对你的影响：(1) 不要假设 Web API 「免费」—— 跨进程 IPC 有 ~ms 开销；(2) 受限是设计而非 bug——某些"native 能做的事"Web 永远做不到（比如启动子进程、读任意文件路径）。
