浏览器项目史（1993-2026）：Mosaic → Netscape → IE → WebKit → Blink → Chromium 主导

把浏览器 33 年的故事讲成一条主线——不是按版本号，而是按每次范式转移及其根因。从 1993 NCSA Mosaic、第一次浏览器战争（IE vs Netscape）、IE 6 停滞、WebKit 与 Gecko 平行演进、Chrome 2008 + V8 + Blink 横空出世、2018 Edge 切 Chromium、2020+ Chromium 90%+ 主导、2024 EU DMA 解禁 iOS 引擎。把"为什么是这样"的历史坐标提供给后面所有章节。

阅读时长 ~25 分钟 前置 无 主要来源 chromium.org / webkit.org / hacks.mozilla.org / Web 历史文献

01 速览

  1. 浏览器历史是「平台之争」史——第一次浏览器战争（IE 1995-2003 击败 Netscape）+ 第二次（Chrome 2008+ 击败 IE）+ 当下（Chromium 90%+ 主导）。每次转折都对应一个新平台 / 新引擎设计的崛起。
  2. IE 6 停滞 5 年是 Web 标准化的转折点——2001-2008 微软不更新 IE 让 Web 几乎僵死；这段窒息期催生了 W3C / WHATWG 标准化运动 + Firefox / Safari / Chrome 反弹。
  3. 2026 年 Chromium 占 90%+ 浏览器份额是好是坏——标准化加速但生态多样性损失；Apple WebKit / Mozilla Gecko 是关键的「反向制衡」，决定了 Web 不会变成 Chrome-only。

02 底层逻辑

1993-1995 · Mosaic 与 Netscape：图形 Web 的起点

1993 年 4 月 NCSA（美国国家超级计算应用中心）发布 Mosaic 1.0——第一款让普通人能用的图形 Web 浏览器。在 Mosaic 之前，Web 只有命令行 + 纯文本（Tim Berners-Lee 的 WorldWideWeb 浏览器仅 NeXT 平台），图片、表单、点击式导航都不存在。Mosaic 让 Web 进入大众视野——1993 年 12 月已有上百万下载。

核心创业者 Marc Andreessen 是 Mosaic 主要开发者。他 1994 年离开 NCSA 创立 Mosaic Communications（后改名 Netscape Communications），1994 年 12 月发布 Netscape Navigator 1.0。Netscape 1.0 是第一个商业浏览器——免费给个人，付费给企业。一年内拿到 Web 浏览器市场 80%+ 份额，成为 1990 年代中期最有影响力的科技公司。

Netscape 期间还诞生了一个改变 Web 的产品：JavaScript。1995 年 5 月 Brendan Eich 在 10 天内写出 Mocha（后改名 LiveScript，再改 JavaScript）。设计目标是「让 HTML 网页能跑脚本」——本来 Sun Microsystems 想推 Java applet 做这事，但 Java 嵌入太重，Eich 写了个轻量级解释型语言。1995 年 12 月集成进 Netscape 2.0 发布。详见 JavaScript 主题。

1995-2003 · 第一次浏览器战争：IE vs Netscape

1995 年 8 月 Microsoft 推出 Internet Explorer 1.0，与 Windows 95 Plus! Pack 一起卖。微软当时还不重视 Web——Bill Gates 那年才发了著名的「Internet Tidal Wave」内部备忘录承认错过了 Web 趋势。但 1996 年 IE 3.0、1997 年 IE 4.0 开始免费捆绑 Windows——这成为 Netscape 的致命打击。

到 1998 年 IE 占浏览器市场 50%+，2002 年达到 95%+ 顶峰。Netscape 这边节节败退：1998 年 11 月被 AOL 收购；同年 Netscape 团队启动 Mozilla 项目——把 Netscape 代码开源。但代码太烂（5 年 spaghetti C++），1998 年开源后很快决定从零重写，新引擎叫 Gecko。重写花了 4 年，2002 年才发布 Mozilla 1.0；2004 年 Firefox 1.0 才上线。

这段时间 Microsoft 因 IE 捆绑被美国司法部反垄断起诉（United States v. Microsoft, 2001）——但裁决相对宽松，Microsoft 没被拆分。Netscape 实际上已经死了——商业层面 1999 年被 AOL 收购后逐渐边缘化，2008 年 AOL 正式停产 Netscape 浏览器。

第一次浏览器战争的教训是后来 30 年 Web 标准化运动的根源：单一公司控制平台 → 标准停滞 → Web 创新窒息。这给 W3C / WHATWG 等标准化组织提供了存在意义。

2001-2008 · IE 6 停滞 + Web 标准化运动

2001 年 8 月 IE 6.0 发布，与 Windows XP 一起捆绑。然后 Microsoft 几乎不再更新 IE——他们觉得「已经赢了」。2001-2006 年 IE 6 是默认浏览器，没出新主版本。这 5 年是 Web 历史最黑暗的时期：

(1) 标准实现停滞——IE 6 不实现 CSS2 完整规范，box model 错误（IE 把 padding 算进 width），CSS3 / HTML5 基本没支持。Web 开发者要为 IE 6 写 hack（* html、_property、conditional comments）。

(2) 安全漏洞累积—— ActiveX 让网页能跑任意 native 代码，间谍软件 / 勒索软件泛滥。2004 年 Microsoft 内部承认 IE 安全模型彻底失败。

(3) JS 性能糟糕—— IE JScript 引擎是早期解释器，现代意义上的优化（JIT / hidden class）都没有。复杂 web app 在 IE 上慢 10x+。

这段窒息期催生了Web 标准化运动：(1) WaSP（Web Standards Project，1998）给浏览器厂商施压；(2) WHATWG（2004）由 Apple/Mozilla/Opera 创立，绕开 W3C 推 HTML5；(3) Acid Test（2005-2008）用一张图测试浏览器规范实现度，让「标准合规」成为公众讨论。

同时反弹崛起：Firefox 1.0 (2004)——开源、快、安全；2008 年 Firefox 占 30%+ 份额，把 IE 拉下王座。Apple Safari 1.0 (2003)——基于 KHTML/KJS（KDE 项目）fork 出来，2007 年 iPhone Safari 让 WebKit 在 mobile 直接主宰。

2008-2013 · Chrome + V8 + Blink：第二次浏览器战争

2008 年 9 月 Google 推出 Chrome 1.0。这是浏览器历史最重要的产品发布——它带来了三个革命性创新：

(1) V8 JavaScript 引擎——比当时最快的 SpiderMonkey 快 10x。详见 V8 P1.1 项目史。V8 让 JS 性能从「勉强能用」变成「接近 native」，让 Gmail / Google Maps 这种重 JS 应用真正可行。

(2) 多进程架构——每个 tab 独立进程，安全 + 稳定 + 性能（详见 P2.1）。Firefox / Safari 当时都是单进程，一个 tab 崩整个浏览器挂。

(3) 沙箱安全模型——Renderer 进程跑在受限沙箱内，恶意网页不能访问系统资源。

Chrome 当时基于 WebKit（Apple 2001-2003 fork KHTML 出来的引擎）。但 Google / Apple 的产品节奏不一样：Apple 优先 Safari + macOS/iOS 平台，节奏慢；Google 想要更激进的多进程架构 + 更快迭代。2013 年 4 月 Google 宣布从 WebKit fork 出 Blink，独立演进。Apple WebKit 继续给 Safari，Google Blink 给 Chrome / Edge / Brave / Opera 等其他 Chromium 系。

这一波让浏览器市场从「IE 一家独大」变成「Chrome / Firefox / Safari / IE 四分天下」——也就是第二次浏览器战争。这次战争 Chrome 几乎完胜：到 2018 年 Chrome 占 60%+，IE 跌到 10% 以内。

2015-2020 · IE 退出 + Edge 切 Chromium

2015 年 7 月 Microsoft 发布 Microsoft Edge，与 Windows 10 一起。Edge 「Legacy」 用自家新引擎 EdgeHTML（从 IE Trident fork 改造）+ 自家 JS 引擎 Chakra。但市场反应冷淡——Edge Legacy 兼容性 / 性能 / 生态都干不过 Chrome。2018 年 Edge Legacy 份额只有 ~5%。

2018 年 12 月 Microsoft 宣布 Edge 切 Chromium——放弃 EdgeHTML / Chakra，改用 Blink + V8。这是浏览器历史另一个重大事件：意味着 Microsoft 承认 Chromium 是事实标准，自己也要用。2020 年新 Edge（Chromium 版）取代 Edge Legacy 发布。

同年 2020 年 6 月 Microsoft 宣布 IE 11 在 2022 年退役。2022 年 6 月 IE 在 Windows 10 上彻底关闭——Web 浏览器历史上一个时代结束。在 IE 27 年生命周期里，它从「统治者」（2002 95% 份额）→ 「守旧者」（2008-2018 拖累 Web 创新）→ 「退役者」（2022 完全退场）。

2020-2026 · Chromium 主导 + 三引擎制衡

2020 年代浏览器市场变成「Chromium 90%+ + WebKit 20%+ + Gecko 3%」的格局（数字相加超 100% 因为 mobile/desktop 重叠）。具体看：

Chromium 系（占 desktop 70%+ + mobile 65%+）：Chrome / Edge / Brave / Opera / Vivaldi / Arc。这些浏览器共享 Blink 引擎 + V8 JS 引擎，差异在 UI / 隐私功能 / 同步服务。其中 Chrome 是绝对老大（占 Chromium 系 80%+）。

WebKit 系（占 desktop 10% + mobile 30%+）：Safari（Apple）+ iOS/iPadOS 上所有第三方浏览器（强制用 WebKit，2024 EU DMA 前）。Apple 主要在 mobile 占优势。

Gecko 系（占 ~3%）：Firefox（Mozilla）几乎是唯一的 Gecko 浏览器。Mozilla 财务困境（依赖 Google 搜索默认协议费）让 Gecko 投入逐年减少，2024 Mozilla 关闭了多个项目（Pocket / Hubs / Voice）。

2024 年 3 月 EU DMA（Digital Markets Act）生效——Apple 必须允许 iOS 上第三方浏览器引擎（不强制 WebKit）。理论上 Chromium / Gecko 能上 iOS。但实际市场反应慢——Google / Mozilla 都还没大力推 iOS 版第三方引擎，Safari + WebKit 仍是 iOS 上事实主流。这个解禁的长期影响仍待观察——可能 iOS 上 Chromium 份额慢慢涨，让 WebKit 在 mobile 失去强制壁垒。

底层洞察 1：单一引擎主导是 Web 平台的固有矛盾

Chromium 90%+ 份额是好是坏？这是 Web 社区一直争论的问题。两面看：

好处：(1) 标准化加速——开发者只需要在一个主要引擎上测试，跨浏览器兼容性问题大幅减少；(2) 新特性推动快——Chrome 团队推 Origin Trial，让开发者提前试新 API，反向推 Web 标准化（如 View Transitions / Speculation Rules）；(3) 性能 baseline 高——V8 + Blink 的优化投入巨大，Web 性能 baseline 持续提升。

坏处：(1) 「Chrome-only」 的隐性垄断——开发者优先在 Chrome 测试，Safari / Firefox 上 corner case 不被发现；(2) Google 商业利益绑定——Chrome 是 Google 广告业务的入口，某些 API 设计可能偏向 Google 利益（如 FLoC / Topics API 的隐私争议）；(3) 生态多样性损失——如果 Chromium 设计错误，整个 Web 都要错（IE 6 时代的反复）。

三引擎制衡是 Web 平台健康的必要条件，不是奢侈品：Apple WebKit 拒绝实现某些 API（Web Bluetooth / WebUSB / Notifications API）让 Chrome 不能彻底主导；Mozilla Gecko 标准合规态度严格，让 Web 不至于变 Chrome-only。这种「反向制衡」需要 Apple 和 Mozilla 持续有话语权——这取决于 Apple 商业地位（Safari 强势）和 Mozilla 财务健康（持续受质疑）。

底层洞察 2：浏览器引擎 fork 的成本

从浏览器项目史看到一个反复的模式：fork 已有引擎比从零写新引擎更可行——但 fork 成本仍然巨大。

WebKit fork KHTML (2001-2003)：Apple fork 后维护成本不小——前 5 年 Apple 大量重构 KHTML 代码，2005 年 Apple 把 WebKit 作为独立开源项目发布（之前是 Safari 内部）。

Blink fork WebKit (2013)：Google 从 fork 后立即大规模重构——把 WebKit 的某些遗留代码（如不需要的 KDE 兼容代码）剔除，多进程架构改造，组件解耦。fork 第一年就有几十万行代码差异。

Gecko 重写 Netscape (1998-2002)：4 年时间从零重写，期间 Netscape 持续掉份额——这种「全部重写」的代价是极端的。Mozilla 从此学到不能再做大重构。

Servo (Mozilla 2012-2020)：Mozilla 想用 Rust 写一个全新引擎，目标是替代 Gecko。最后部分组件被吸收回 Gecko（CSS engine: Stylo, 2017），但完整 Servo 没有取代 Gecko；2020 年 Mozilla 大裁员后 Servo 被分拆出去。

这些故事说明：浏览器引擎是 Web 平台中最复杂的软件之一，新引擎的成本极高。这也是为什么 2020 年代没有新的浏览器引擎出现——三个现有引擎（Blink / WebKit / Gecko）全是 90 年代/00 年代立项的，新进入者几乎不可能。

底层洞察 3：每次范式转移的 driver

把 33 年浏览器史的关键转折点抽象出 driver：

1993 Mosaic → driver: 让 Web 大众化（图形界面替代命令行）。

1995 IE 捆绑 → driver: Microsoft 借平台优势打 Netscape。

2001 IE 6 停滞 → driver: 单一公司无竞争压力，技术不更新。

2004 Firefox 反弹 → driver: 用户对 IE 安全 + 性能不满，开源 + 标准合规给替代方案。

2008 Chrome + V8 → driver: Web app（Gmail / Google Maps）需要 native 级 JS 性能。

2013 Blink fork → driver: Google / Apple 产品节奏不匹配，Google 要更激进的多进程改造。

2018 Edge 切 Chromium → driver: Microsoft 承认无法在引擎层与 Chromium 抗衡，转向「借力」。

2024 EU DMA 解禁 iOS → driver: 监管力量打破 Apple 在 iOS 上的强制 WebKit 锁定。

这些 driver 都不是技术驱动，是商业 / 监管 / 社会压力驱动。技术只是 enabler——决定哪种平台 / 哪种引擎能赢的，往往是商业模式和市场份额。这是浏览器（和大多数互联网平台）的共同模式。

03 应用场景

场景 A · 给前端工程师做技术选型时的「浏览器兼容矩阵」基线

评估「我们项目要支持哪些浏览器」——理解浏览器历史能让你的兼容矩阵基线合理。2026 年合理的基线：(1) 必须支持：Chrome 100+ / Edge 100+ / Firefox 100+ / Safari 16+；(2) 不需要支持：IE 11（已退役）+ Edge Legacy + Chrome < 90 + 老 Safari；(3) 看用户群：政府 / 银行 / 大企业内部系统可能仍在用老 Edge / IE，要单独评估。具体策略：用 caniuse.com 查每个特性的兼容性，browserslist 配置自动生成 babel/PostCSS target；@supports CSS 渐进增强；现代化场景全用 Web Platform Baseline 2024（Chrome 100+ / Edge 100+ / Firefox 100+ / Safari 16+ 共同支持的特性）。这套基线让你不被 IE 6 时代的 hack 遗留拖累，又能覆盖 95%+ 用户。

场景 B · 评估「我们要不要支持 iOS Safari WebKit-only 限制」

项目要部署到 mobile，问「iOS 上 WebKit 强制对我们影响多大」——历史脉络给出答案。(1) 2024 EU DMA 之前：iOS 上所有浏览器必须用 WebKit（包括 Chrome iOS），所以 iOS 上 Web 体验完全由 Apple 决定。Apple 拒绝实现 PWA 完整入口、Web Bluetooth、Background Sync 等 API → 这些功能 iOS 用户用不到。(2) 2024 EU DMA 之后：理论上 Chromium / Gecko 能上 iOS，但实际推进慢——Google 仍未发布 iOS 版完整 Chromium，Mozilla 也未行动。iOS 上 WebKit 仍事实主流（98%+）。(3) 对你的项目影响：iOS 上要测试 WebKit corner case；某些「Chrome-only」 API 在 iOS 上不能用要 fallback。具体做法：用 @supports 检测 + Polyfill；关键功能在 iOS Safari 真机测试；接受 iOS 用户拿不到某些「Chrome-only」特性（如 PWA Install Prompt）。

场景 C · 给团队解释「为什么我们要写跨浏览器兼容代码」

团队成员（特别是从 React Native / 桌面 app 转过来的）问「既然 Chromium 占 90%+，为什么还要写跨浏览器代码」——历史给出明确答案。(1) Apple WebKit 拒绝实现某些 API——Web Bluetooth / WebUSB / Background Fetch / Web Share Target 等，iOS 用户用不到；(2) Mozilla Gecko 进度落后——某些 CSS 新特性（如 Container Queries 早期）Firefox 跟进慢；(3) 三引擎制衡是 Web 平台健康的保证——如果你只在 Chromium 测试，Web 就变 Chrome-only，长期对你的产品也不利（如果未来 Apple 推自家平台 Web 不能跑就完蛋）。综合：写跨浏览器代码不是「技术债」，是对 Web 平台健康的投资。具体做法：CI 矩阵跑 Chrome + Firefox + Safari；用 caniuse + browserslist 自动控制 polyfill 范围；优先用 Web 标准 API 而非「Chrome-only」 实验性 API。

场景 D · 给做技术决策的架构师：评估「Web 平台 vs Native」长期路线

架构师问「Web 平台未来 5-10 年还能跟得上 native app 吗」——历史给的判断。(1) Web 平台优势：Chromium 主导让 Web 标准化加速、JS 性能持续提升（V8 改进 + WebAssembly）、新 API 不断扩展（File System API / Web Bluetooth / WebGPU）；(2) Web 平台劣势：Apple 仍在 iOS 上限制 Web（虽然 2024 EU DMA 解禁但实际推进慢）；某些 native 能力（深度 OS 集成、AR/VR）Web 仍滞后；(3) 趋势：Web 在 desktop / Android 上几乎能完全替代 native；iOS 上 Web 仍受限。结论：Web 平台是长期方向，但 iOS 用户体验仍要单独评估；某些深度 OS 集成场景仍需 native + WebView 混合。这种判断让你的技术路线在 5-10 年视角下站得住。

04 典型代码

历史章不是 API 章，没有典型 API 用法可示。这里展示「项目史相关的几段实证代码」——能直接验证文中的关键节点。

javascript 
// 检测当前浏览器引擎（验证「三引擎」概念）
function detectEngine() {
  const ua = navigator.userAgent;

  // Blink / Chromium 系（V8）
  if (/Chrome/.test(ua) && !/Edg/.test(ua) && !/OPR/.test(ua)) {
    return { engine: 'Blink', js: 'V8', browser: 'Chrome' };
  }
  if (/Edg/.test(ua)) {
    return { engine: 'Blink', js: 'V8', browser: 'Edge (Chromium)' };
  }

  // WebKit 系（JavaScriptCore）
  if (/Safari/.test(ua) && !/Chrome/.test(ua)) {
    return { engine: 'WebKit', js: 'JavaScriptCore', browser: 'Safari' };
  }

  // Gecko 系（SpiderMonkey）
  if (/Firefox/.test(ua)) {
    return { engine: 'Gecko', js: 'SpiderMonkey', browser: 'Firefox' };
  }

  return { engine: 'unknown' };
}

  navigator.clipboard.writeText(location.href);

05 常见陷阱

把「Chromium 90%」误读为「只在 Chrome 测试就够了」

看到 Chromium 占 90%+ 份额，有人推论「那我们只在 Chrome 测试就够了」——这错误且危险。原因：(1) iOS 上 WebKit 占 95%+——你的 mobile 用户大半是 WebKit；(2) Safari 是新 Web 标准的滞后引擎——某些 API Chrome 早就有，Safari 半年到一年才跟进，不在 Safari 测会遇到大量 corner case；(3) 三引擎制衡是 Web 平台健康的保证——长期看，让 Web 变 Chrome-only 对所有人都不利。规避：CI 矩阵必须跑 Chrome + Firefox + Safari 三引擎，关键流程在 iOS Safari 真机测试。

假设「IE 已死」=「不需要考虑老浏览器」

IE 11 在 2022 年退役，但不意味着所有用户都升级了。某些场景仍可能遇到老浏览器：(1) 政府 / 银行 / 大企业内部系统——锁定 Windows 7 + IE 11 是常态，2026 仍在用；(2) 非 EU/北美市场——一些发展中国家盗版 Windows + 老浏览器仍在；(3) 嵌入式 WebView——一些老硬件嵌的 Chromium 可能是几年前版本。判断你的用户基线：用 Google Analytics / 自家 telemetry 看 user agent 分布，决定要支持的最低版本。如果用户群有 5%+ 老浏览器，可能需要 polyfill / 降级 UI。

用「浏览器战争」叙事简化复杂事实

「第一次浏览器战争 IE 击败 Netscape」 / 「第二次 Chrome 击败 IE」——这种叙事简单但掩盖了关键细节。Netscape 不是被 IE 技术击败，是被 IE 捆绑 + Microsoft 商业策略击败；Chrome 不是单纯技术胜出，是 Google 平台优势 + Web app 兴起 + IE 6 停滞共同作用。正确认识：浏览器市场份额变化通常是技术 + 商业 + 监管 + 社会多因素叠加，不是单一原因。这种认识让你不会过度乐观（「做出一个比 Chrome 更好的浏览器就能赢」）也不会过度悲观（「Chrome 永远不会失去市场」）。

期待 EU DMA 立即改变 iOS 浏览器格局

2024 年 3 月 EU DMA 生效解禁 iOS 第三方浏览器引擎——但「理论上能上」和「实际上会上」是两回事。截至 2026 年实际状态：(1) Google 还没发布 iOS 版完整 Chromium（仅 EU 版 limited beta）；(2) Mozilla 也没大力推 iOS Gecko；(3) Apple 仍在用各种「合规但不友好」的方式增加第三方引擎落地难度（每年合规重审、限制特定 API）。短期内 iOS 上 WebKit 仍 95%+ 份额。对你的影响：不要假设 iOS 上能用 Chromium-only API；按 WebKit 兼容矩阵做基线；3-5 年后再重新评估 iOS 上 Chromium 份额是否上来。
