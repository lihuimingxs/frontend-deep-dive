三大引擎对照：Blink V8 / WebKit JSC / Gecko SpiderMonkey

横向对照章。讲清三个浏览器引擎各自做了什么不同选择 + 现状份额（Blink 70% / WebKit 20% / Gecko 3%）+ 互相制约。关键洞察：Apple 拒绝实现某些 API 让 Chrome 不能彻底主导；Mozilla 标准合规态度让 Web 不至于变 Chrome-only。三引擎制衡是 Web 平台健康的关键，不是奢侈品——这一章把这种制衡的具体机制讲透。

阅读时长 ~25 分钟 前置 P1.1 主要来源 chromium.org / webkit.org / mozilla.org · caniuse.com

01 速览

  1. 三引擎不是「谁更好」的关系，是「分工 + 制衡」——Blink 推 Web 平台前进、WebKit 守 mobile + 隐私边界、Gecko 守标准合规底线。这种分工让 Web 平台不会过度倾斜。
  2. 引擎差异主要在 JS 引擎 + 多进程 + 新特性速度——三引擎在 HTML/CSS 渲染基础上分歧不大，差异集中在 V8/JSC/SpiderMonkey 的 JIT 性能、Renderer 进程模型、新 API 实现节奏。
  3. Apple 在 iOS 上的 WebKit 强制（2024 EU DMA 前）让 WebKit 在 mobile 不能被 Chromium 完全替代——这是 Web 多引擎生态最关键的"反向制衡力"。

02 底层逻辑

Blink (Chromium 系)：推平台前进的引擎

Blink 是 Google 2013 年从 WebKit fork 出来的浏览器引擎。当前 Chrome / Edge / Brave / Opera / Vivaldi / Arc 等 Chromium 系浏览器都用它。市场份额最大（desktop 70%+ / mobile 65%+，合并约 90%+ 全球浏览器份额）。

JS 引擎：V8。详见 V8 主题。简短回顾：4 层编译流水线（Ignition → Sparkplug → Maglev → TurboFan），Hidden Class + Inline Cache 是核心优化，Orinoco GC 处理内存。V8 是当前公认最快 / 最成熟的 JS 引擎。

渲染引擎：Blink。从 WebKit fork 后大规模重构——剥离不需要的 KDE 兼容代码、组件解耦、多进程架构改造。现在与原始 WebKit 已经差异巨大。

多进程架构：最激进。每个 site 独立 Renderer 进程（Site Isolation，2018 后）；GPU / Network / Utility 都独立进程；甚至 iframe 跨域也独立进程（OOPIF）。详见 P2.1。

新特性速度：最快。Chrome 团队主动推 Origin Trial（让开发者在 Chrome 上提前试新 API），反向推 Web 标准化。View Transitions / Speculation Rules / Shared Element Transitions / Anchor Positioning 等都是 Chrome 先实现，再推标准。这种「优先实现 + 推标准」模式有时被 WebKit / Gecko 反对（觉得 Chrome 强推未成熟标准）。

商业绑定：Chrome 是 Google 广告业务入口。某些 API 设计可能偏向 Google 利益—— Privacy Sandbox（FLoC / Topics / FLEDGE）就是为了在抛弃 cookie 的同时保留广告生态。这种"商业绑定"是 Chromium 主导的潜在隐忧。

WebKit (Apple/Safari)：守 mobile + 隐私边界的引擎

WebKit 是 Apple 2001-2003 年从 KHTML/KJS（KDE 项目的浏览器引擎）fork 出来的引擎。当前用于 Safari (macOS/iOS) + iOS 上所有第三方浏览器（2024 EU DMA 前强制；DMA 后理论上可换但实际仍主流）。市场份额：desktop ~10% / mobile 30%+（iOS 上 95%+）。

JS 引擎：JavaScriptCore (JSC)。详见 Bun P2.1 JavaScriptCore。简短回顾：4 层流水线（LLInt → Baseline → DFG → FTL/B3）；启动快是核心优势（LLInt 用 LLVM IR 写）；hot 性能略逊 V8 TurboFan。

渲染引擎：WebKit。原始 WebKit 仍由 Apple 维护；2013 Blink fork 后两边独立演进，10 年下来代码差异巨大但规范层面对齐——基础 HTML/CSS 行为基本一致。

多进程架构：较激进。每个 tab 独立 Renderer（macOS 上自 2014 Safari 起；iOS 上较晚）。但 Site Isolation 不如 Chromium 完整。

新特性速度：最慢。Apple 通常等 W3C / WHATWG 规范成熟后才实现，不轻易跟 Chrome Origin Trial。但 Apple 在 mobile 优化上领先——iOS Safari 的内存效率 / 启动速度 / 触摸响应都比 Chrome iOS 好（这部分 Chrome iOS 也只能用 WebKit，所以是 Apple 的功劳）。

关键拒绝：Apple 不实现某些 API——Web Bluetooth / WebUSB / Web NFC / Notifications API（部分） / Web Share Target / PWA 完整入口。Apple 公开理由是「这些 API 隐私 / 安全风险」（fingerprinting / 设备访问），实际隐含商业理由：这些 API 让 Web app 太接近 native app，威胁 App Store 商业模式。这种拒绝是 Apple 在 Web 平台上的"反向制衡力"——它阻止了 Chrome 把 Web 拉得太接近 native。

iOS 强制 WebKit（2024 前）：iOS 上所有第三方浏览器（包括 Chrome iOS）必须用 WebKit。这意味着 Chrome iOS 不是"用 V8 跑的 Chrome"——它是"在 WebKit 引擎上套 Chrome UI 的浏览器"。性能 / 特性都受 Apple 限制。2024 年 3 月 EU DMA 生效解禁——但实际上 Google / Mozilla 仍未推完整第三方引擎到 iOS（受 Apple 各种合规限制阻挡），WebKit 仍 iOS 95%+ 主流。

Gecko (Mozilla/Firefox)：守标准合规底线的引擎

Gecko 是 Mozilla 1998-2002 年从 Netscape 代码全部重写的引擎。当前几乎只用于 Firefox + 一些 Mozilla-related 项目（Thunderbird / Tor Browser）。市场份额：~3%（持续下滑）。

JS 引擎：SpiderMonkey。SpiderMonkey 是世界上第一个 JS 引擎——Brendan Eich 1995 在 Netscape 写的。后续大量重构（IonMonkey / WarpMonkey），现在的 SpiderMonkey 是 4 层 JIT（Interpreter → Baseline Interpreter → Baseline JIT → IonMonkey）。性能略逊 V8，但与 V8 + JSC 同等数量级。

渲染引擎：Gecko。Mozilla 2012-2020 年用 Rust 写新引擎 Servo，目标是替代 Gecko。最后 Servo 部分组件被吸收回 Gecko（CSS 引擎 Stylo 在 Firefox 57 / 2017 上线）；2020 Mozilla 大裁员后 Servo 被分拆出去。完整 Servo 没有取代 Gecko —— Gecko 仍是 Firefox 主引擎。

多进程架构：中等。Firefox 自 2017 「Quantum」 起逐步多进程（Project Electrolysis），目前 4-8 个 Content process（每 site 不独立，按数量分组）。Site Isolation 投入不如 Chromium。

标准合规态度：最严格。Mozilla 长期是 W3C / WHATWG / TC39 的核心贡献者，对规范文本质量要求极高。一些 Chrome Origin Trial 的特性 Mozilla 拒绝实现（直到规范成熟）—— 这种「保守但严谨」的态度让 Web 不会被 Chrome 单方面推着走。

财务困境：Mozilla 主要收入来自 Google 给的「搜索默认协议费」（每年约 4 亿美元）。这个收入受 Google / 美国反垄断诉讼影响，长期不稳定。Mozilla 2024 关闭多个产品（Pocket / Hubs / Voice），裁员 30%+。Gecko 投入逐年减少，新特性跟进越来越慢。

三引擎制衡的具体机制

「三引擎制衡」不是抽象口号，是有具体机制的：

(1) Apple 拒绝实现 = "硬墙"。Web Bluetooth / WebUSB / Web NFC / PWA 完整入口等 API，Chrome 实现了但 Apple 拒绝。这意味着开发者不能假设这些 API 在所有平台可用——必须 fallback。这阻止了 Web 平台被 Chrome 单方面拉向 native。

(2) Mozilla 标准合规 = "底线"。Mozilla 在规范讨论里的立场严格——某些 Chrome 想推的 API 设计如果 Mozilla 觉得不合理，Mozilla 拒绝实现 + 提反对意见 + 提替代方案。Chrome 团队为了让 API 进入 W3C/WHATWG 标准，必须妥协设计 → 让 API 更好。

(3) caniuse 是「制衡的可见化」。caniuse.com 把每个 API 在三引擎的实现状态 visualize 出来。开发者一看就知道「这个 API 全引擎兼容吗」——制衡的存在直接转化为开发实践。

(4) Web Platform Tests (WPT)。三引擎共同维护的测试套件，确保规范实现一致。测试不通过 → 引擎修复。这套机制让三引擎在基础规范层面行为对齐，差异主要在新特性。

这种制衡的结果是：Web 平台 API 设计经过三引擎博弈后，比单一公司单独决定更合理（虽然进度慢）。这是 Web 30+ 年长期健康的核心机制。

JS 引擎横评

三个 JS 引擎细节差异：

V8 vs JSC vs SpiderMonkey 性能：

  • 启动时间：JSC ~5-10ms < V8 ~30ms < SpiderMonkey ~30ms。JSC 优势来自 LLInt 用 LLVM IR 写。
  • 冷启内存：JSC ~25MB < V8/SpiderMonkey ~50MB。
  • 稳态 hot 性能：V8 TurboFan ≈ SpiderMonkey IonMonkey > JSC FTL/B3（V8/SpiderMonkey 略胜 5-15%）。
  • 大对象处理：V8 / SpiderMonkey 略好（更成熟的 GC）。

实际差异：在常规 web app 上三引擎差距不到 20%——用户体验差异主要来自渲染引擎 + 多进程架构 + 网络层，不是 JS 引擎。

渲染引擎差异

三引擎在 HTML/CSS 渲染上规范层面对齐，差异在实现细节：

(1) Layout 算法实现——Blink LayoutNG（2017+，新一代 layout 引擎）；WebKit 用更传统的 layout 实现；Gecko Stylo（CSS 引擎用 Rust 写，从 Servo 吸收）。LayoutNG / Stylo 都比传统实现快很多 + 更准确。

(2) Paint 实现——Blink 用 Skia（2D 图形库）；WebKit 用 Core Graphics（macOS/iOS native）+ Skia（其他平台）；Gecko 用 WebRender（Rust，从 Servo 吸收）。WebRender 是 Gecko 性能跃升的关键（2019 上线）。

(3) GPU 合成——Blink / WebKit / Gecko 都用 tile-based + GPU 合成，机制相似但实现不同。详见 P3.3。

(4) 渲染开销实测——同一个网页在三引擎上的 LCP / INP / CLS 差异通常 < 20%。如果差异大，往往是某个特性 corner case。

03 应用场景

场景 A · 用 Web Platform Baseline 2024 作为开发基线

团队问"我们用什么 baseline 开发"——Web Platform Baseline（web.dev 推出的"全引擎共同支持"列表）是 2026 年最合理的答案。Baseline 2024 意味着 Chrome / Edge / Firefox / Safari 在 2024 年共同支持的特性集合。包括：CSS Grid / Flexbox / Custom Properties / :has() / Container Queries / Popover API（部分）等。开发新项目时所有用到的特性都在 Baseline 内就能保证三引擎都跑——不需要 polyfill / 渐进增强。具体做法：(1) caniuse.com 查每个特性的 Baseline 状态；(2) browserslist 配置 baseline: 2024（PostCSS / babel 自动控制）；(3) 用 @supports 处理 Baseline 之外的"实验性"特性。这套基线让你不必为兼容性焦虑。

场景 B · 评估「我们的项目能不能依赖 View Transitions API」

团队想用 View Transitions 做页面间动画——但这是 Chrome-only 特性（Safari / Firefox 未支持）。判据：(1) 用户分布—— 你的项目 Chrome 占多少？如果 80%+ 是 Chrome（典型 desktop SaaS），用 View Transitions + fallback 给 Safari/Firefox 是合理的渐进增强。(2) Fallback 设计—— Safari/Firefox 上没有过渡，但页面切换仍然 work（不卡死）；用 @supports 检测。(3) 等待 Safari/Firefox 跟进—— Chrome 的实验性特性通常 1-2 年被其他引擎跟进；如果项目长期，等是合理选择。综合：Web 平台特性可以用，但要做 fallback 设计，不能假设全引擎都跑。

场景 C · 给 PWA 项目评估「iOS 上 Apple 限制」

团队想做 PWA（Progressive Web App）覆盖 mobile——但 iOS 上 Apple 限制 PWA 完整体验。具体限制：(1) Add to Home Screen 必须用户手动操作——iOS 不会弹"安装" prompt；(2) Push Notifications 16.4 才支持——之前完全不支持 web push；(3) Service Worker 限制——Background Sync / Periodic Background Sync 不支持；(4) App Store 不能上架 PWA——用户只能从浏览器装。综合：iOS 上 PWA 体验显著弱于 Android。规避：(1) 接受 iOS PWA 是"次等体验"；(2) 关键功能仍走 native app + WebView 混合；(3) 等 EU DMA 生效后看是否有改善（短期内不明显）。这种限制是 Apple 对 PWA 商业威胁的故意防御——不会快速消失。

场景 D · 给做技术决策的架构师：评估「Mozilla 财务困境对 Firefox 影响」

架构师关心"Firefox 在 2030 年还存在吗"——这是合理担心，影响"是否在 Firefox 上测试"的决策。评估：(1) Mozilla 主要收入——Google 搜索默认协议费（每年 ~4 亿美元）。如果 Google 因反垄断被迫停付，Mozilla 财务立即崩溃。(2) 美国反垄断诉讼——Google v. DOJ 2024 判决 Google 是搜索垄断者，但 Search Engine Default Payment 是否被禁止仍未定。(3) Firefox 投入趋势——Mozilla 团队 2024 年裁员 30%+，Gecko 投入持续减少；新特性跟进越来越慢。(4) 3-5 年预判—— Firefox 不会立即消失（Mozilla 仍有现金 + 多年衰退迹象但未崩盘），但会持续衰退。对你的影响：Firefox 仍要支持（合规 + 用户群），但优先级降低；Chrome / Safari 是优先 + Firefox 可接受 corner case。

04 典型代码

javascript 
// 检测特性 + 三引擎兼容性 + 渐进增强
// 用例：用 View Transitions API 做页面切换动画

async function navigateWith动画(url) {
  // 检测浏览器是否支持 View Transitions API
  if (!document.startViewTransition) {
    // Safari / Firefox 未支持 → 直接导航
    location.href = url;
    return;
  }

  // Chrome / Edge 支持 → 用 View Transitions
  const transition = document.startViewTransition(async () => {
    // 改变 DOM（异步）
    await loadPage(url);
  });

  await transition.finished;
}

// 这种"特性检测 + fallback"是 Web 平台开发的标准模式
// 三引擎的差异通过这种 pattern 抹平
javascript 
// 检测三引擎的具体行为差异（开发时排查 corner case 用）
const tests = {
  'View Transitions': 'startViewTransition' in document,
  'Container Queries': CSS.supports('container-type: inline-size'),
  ':has() 选择器': CSS.supports('selector(:has(*))'),
  'Popover API': HTMLElement.prototype.hasOwnProperty('showPopover'),
  'scheduler.yield': 'scheduler' in window && 'yield' in scheduler,
  'Web Bluetooth': 'bluetooth' in navigator,
  'Web Share': 'share' in navigator,
};

console.table(tests);
// 在三引擎跑这段代码会看到不同的特性支持表
// Chrome: 大部分 true
// Safari: View Transitions / scheduler.yield / Web Bluetooth false
// Firefox: View Transitions / scheduler.yield / Anchor Positioning false

05 常见陷阱

把「全引擎都支持」作为唯一标准

有人坚持「除非全引擎都支持，否则不能用」——这过于保守。Web 平台演进的现实是：Chrome 先实现 → 1-2 年其他引擎跟进。如果你坚持等全引擎，你比 Chrome 用户晚拿到所有新特性 1-2 年。合理的策略：(1) Baseline 内的特性放心用（已全引擎支持）；(2) Baseline 之外的实验性特性用 + fallback——Chrome 用户拿到优秀体验，Safari/Firefox 用户拿到 baseline 体验；(3) 测试覆盖关键 fallback 路径。这种渐进增强是 Web 平台开发的成熟模式。

假设 iOS 上的 Chrome 是「真 Chrome」

用户在 iOS 上装的 Chrome 实际是「WebKit + Chrome UI」——JS 引擎是 JSC（不是 V8），渲染引擎是 WebKit（不是 Blink）。所以 iOS Chrome 的特性支持 = Safari 特性支持 + Chrome UI（同步 / 历史 / etc.）。对你的影响：iOS Chrome 用户拿不到 Chrome desktop 上的 Chrome-only 特性（View Transitions / Speculation Rules）。如果你在 Chrome desktop 上测试通过，iOS Chrome 上仍可能不工作。规避：iOS 测试当成 Safari 测试看待。

期待 Mozilla 「永远」 跟进新特性

Mozilla 财务困境 + 团队裁员让 Firefox 跟进新特性越来越慢。某些 Chrome 已实现 1-2 年的特性，Firefox 仍未排上日程。规避策略：(1) 不期待 Firefox 跟 Chrome 节奏一致；(2) 用 Web Platform Baseline 而非 Chrome 最新特性作为基线；(3) 关键特性的 Firefox 进度可以查 Bugzilla（Mozilla bug tracker）。如果 Mozilla 5 年内继续衰退，Web 多引擎制衡会从 3 个变 2 个（Blink + WebKit），格局会有质变。

用「Origin Trial」=「正式 API」

Chrome 推 Origin Trial 让开发者提前用某个新 API（注册一个 token，在 HTTP header 声明）。这不等于该 API 是正式标准——只是 Chrome 的"实验"。Origin Trial 期间 API 可能变签名 / 行为 / 甚至取消。规避：production 不依赖 Origin Trial 特性；要用先评估"如果这个 API 变了我们要改多少代码"。Origin Trial 适合 prototype / 有限场景，不适合作为产品基础。
