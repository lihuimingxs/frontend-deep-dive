现代 Web Platform：View Transitions / Container Queries / Speculation Rules / Popover

把 6-7 个 2024+ 关键特性集中讲，每个独立但都体现「Web Platform 现代化」方向。View Transitions（替代 Framer Motion 部分场景）+ Container Queries（基于父容器 responsive）+ Speculation Rules（真预渲染下一页）+ Popover API（原生 modal）+ CSS Anchor Positioning（替代 floating-ui）+ CSS Nesting（不需要 SCSS）+ Origin Private File System。关键洞察：每个特性都对应一个长期 JS hack——浏览器把开发者反复写的「应该是平台特性」东西标准化 + 实现。

阅读时长 ~30 分钟 前置 P3 + P4 主要来源 web.dev/blog · developer.chrome.com/blog · caniuse.com

01 速览

  1. 2024-2026 是 Web Platform 现代化的关键期—— 每年浏览器添加几个"应该早就有"的能力。开发者过去用 JS / 第三方库 hack 实现的，现在变成 CSS 一行 / 浏览器原生 API。
  2. 三引擎实现速度差异显著—— Chrome 通常先 1-2 年实现，Safari / Firefox 跟进。开发用Web Platform Baseline作基线（详见 P1.2），再用 @supports 渐进增强 Chrome 独有特性。
  3. 从 "JS hack" 到 "平台原语" 的趋势是 Web 平台健康的标志—— View Transitions 替代 Framer Motion 部分；Popover 替代 Headless UI modal；Anchor Positioning 替代 floating-ui。这种"内置化"让生态依赖减少 + 性能提升。

02 底层逻辑

View Transitions API：浏览器内置页面过渡

View Transitions API（Chrome 111+，2023-03）让你用一个 API 实现页面间动画过渡—— 之前要用 Framer Motion / GSAP / 自家手写。

核心 API：document.startViewTransition(updateCallback)。

机制：

(1) 浏览器抓取当前页面 snapshot—— "old" 状态；

(2) 调用 updateCallback—— 你改 DOM 到新状态；

(3) 浏览器抓 "new" snapshot；

(4) Compositor 在 old → new 之间渐变—— 默认是 cross-fade（透明度过渡），可用 CSS 自定义。

用 CSS 控制过渡：

/* 默认所有元素 cross-fade */
::view-transition-old(root),
::view-transition-new(root) {
  animation-duration: 300ms;
}

/* 给特定元素 named view transition */
.hero {
  view-transition-name: hero;
}

::view-transition-old(hero),
::view-transition-new(hero) {
  animation: morph 500ms;
}

用例：

(1) SPA 路由切换动画—— React Router / Vue Router 切换时调 startViewTransition。

(2) 列表项 expand—— 卡片展开成详情页，元素位置 morph。

(3) Tab 切换—— Tab 内容平滑过渡。

支持：Chrome 111+ 稳定。Safari 18+ 跟进（部分支持）。Firefox 暂未稳定支持。

用 @supports + fallback：

async function navigate(url) {
  if (!document.startViewTransition) {
    location.href = url;
    return;
  }
  const transition = document.startViewTransition(async () => {
    await loadPage(url);
  });
  await transition.finished;
}

Container Queries：基于父容器的 responsive

传统 CSS responsive 用 media query—— 基于视口大小。但很多场景应该基于父容器大小（如 sidebar 有多种尺寸，组件应该按容器宽度调整）。

Container Queries（Chrome 105 / Safari 16 / Firefox 110）让组件根据父容器宽度变化：

/* 父容器开 container query */
.sidebar {
  container-type: inline-size;
  /* 或 container-type: size （宽 + 高）*/
}

/* 子元素根据父容器宽度变化 */
@container (min-width: 400px) {
  .card {
    display: grid;
    grid-template-columns: 100px 1fr;
  }
}

@container (min-width: 600px) {
  .card {
    grid-template-columns: 200px 1fr 200px;
  }
}

这个特性解决组件可移植性问题——.card 组件不再需要知道整个页面的 viewport 大小，只需要知道自己父容器的大小。把组件用到 sidebar 还是 main content 自动适配。

命名容器（多容器场景）：

.sidebar { container: sidebar / inline-size; }
.main { container: main / inline-size; }

@container sidebar (min-width: 400px) {
  /* 仅响应 .sidebar 容器 */
}

Container Queries 是近 10 年最重要的 CSS 特性之一—— 解决了 responsive 设计的根本问题。

Speculation Rules API：真预渲染

Speculation Rules API（Chrome 121+，2024-01）让浏览器真预渲染下一页—— 不只是 prefetch HTML / JS / CSS，而是整个页面在隐藏的 Renderer 进程渲染好，用户点击时瞬间显示（< 100ms）。

用法：

<script type="speculationrules">
{
  "prerender": [
    {
      "where": { "href_matches": "/articles/*" },
      "eagerness": "moderate"
    }
  ]
}
</script>

浏览器看到这个规则会：(1) 等用户 hover / 接近 link 时；(2) 在隐藏 Renderer 渲染目标页面；(3) 用户点击时瞬间 swap。

eagerness 三档：

(1) immediate—— 立即预渲染（最激进，浪费带宽）；

(2) moderate—— hover 时预渲染（平衡）；

(3) conservative—— mousedown 时预渲染（最保守）。

这个特性的优势：(1) 真"瞬间导航"（< 100ms 切换）；(2) 浏览器自动管理（不用 JS 控制）；(3) 配合 View Transitions 让 SPA-feel 的 MPA 体验。

限制：(1) 只支持 same-origin（safety）；(2) 浏览器有资源限制（不会预渲染所有 link）；(3) 仅 Chrome / Edge 实现，Safari / Firefox 未跟进。

这种"MPA 用 Speculation Rules + View Transitions 实现 SPA-feel"的模式是 Chrome 团队推的"MPA renaissance"—— 让传统多页应用有 SPA 般的体验，不需要 React Router / Next.js 等 SPA 框架。

Popover API：原生 modal

之前做 modal / popover / tooltip 都要 JS 库（Headless UI / Radix UI / floating-ui）—— 处理 z-index 管理、focus trap、ESC 关闭、点外部关闭、accessibility。

Popover API（Chrome 114+ / Safari 17 / Firefox 125+，全引擎支持）让这些变成HTML 原生：

<!-- 触发按钮 -->
<button popovertarget="my-popover">打开</button>

<!-- popover 元素 -->
<div id="my-popover" popover>
  <p>Popover 内容</p>
  <button popovertarget="my-popover" popovertargetaction="hide">关闭</button>
</div>

浏览器自动处理：

(1) Top layer rendering—— popover 渲染在最顶层（z-index 自动），不被其他元素 overlap。

(2) ESC 关闭—— 内置。

(3) Light dismiss—— 点外部自动关闭（popover="auto" 默认）。

(4) Focus management—— 内置 focus trap。

(5) Accessibility—— 自动 ARIA 属性。

类型：

popover="auto"—— light dismiss + 同时只有一个 auto popover（弹出新的关闭旧的）。

popover="manual"—— 必须手动关闭（不 light dismiss）。用于持续显示的 toast / banner。

JS API：el.showPopover() / el.hidePopover() / el.togglePopover()。

这个特性替代了 Headless UI / Radix UI / Tippy.js 等库的 80% 用例—— 你的 modal / dropdown / tooltip 大部分可以用原生 popover 实现。

CSS Anchor Positioning：替代 floating-ui

定位浮动元素（tooltip / dropdown）相对触发元素—— 之前要用 floating-ui / Popper.js 库计算位置。

CSS Anchor Positioning（Chrome 125+，2024）让CSS 一行声明：

/* 锚点元素 */
.button {
  anchor-name: --my-button;
}

/* 浮动元素相对锚点 */
.tooltip {
  position: absolute;
  position-anchor: --my-button;
  /* 锚点底部 + 居中 */
  top: anchor(bottom);
  left: anchor(50%);
  translate: -50% 0;
}

/* 智能 fallback：如果 anchor 在视口边缘，自动调位置 */
.tooltip {
  position-try-fallbacks: top, right, left;
}

浏览器自动：

(1) 计算位置；

(2) 监听锚点 resize / scroll；

(3) Fallback 在视口边缘的情况。

这是替代 floating-ui 库的现代写法—— 之前要装 floating-ui + 用 JS 监听更新；现在 CSS 表达。

限制：仅 Chromium 系实现（Chrome 125+），Safari / Firefox 暂未支持。需要 fallback 到 floating-ui 给非 Chromium 用户。

CSS Nesting：原生支持嵌套

之前只能用 SCSS / Less / PostCSS 写嵌套：

/* SCSS */
.card {
  padding: 1rem;
  &:hover {
    background: gray;
  }
  .title {
    font-size: 1.5rem;
  }
}

CSS Nesting（Chrome 112+ / Safari 17.2+ / Firefox 117+，2023-2024 全引擎稳定）原生支持：

/* 原生 CSS */
.card {
  padding: 1rem;

  &:hover {
    background: gray;
  }

  .title {
    font-size: 1.5rem;
  }
}

这消除了 SCSS / PostCSS-nested 的最大用例—— 嵌套是 SCSS 用户用得最多的特性。原生 CSS 支持后，很多项目可以放弃 SCSS 改用纯 CSS + variables + nesting + container queries。

注意：原生 CSS Nesting 语法和 SCSS不完全兼容—— SCSS 允许 .parent { .child {} }（无 &），CSS Nesting 要 & .child {}。但差异小，迁移容易。

Origin Private File System (OPFS)：浏览器内文件系统

OPFS（Chrome 102+ / Safari 17+ / Firefox 111+）给网页一个私有文件系统—— 用户看不到，仅这个 origin 能访问。比 IndexedDB 性能好得多（特别是大文件）。

const root = await navigator.storage.getDirectory();
const file = await root.getFileHandle('data.bin', { create: true });
const writable = await file.createWritable();
await writable.write(largeBlob);
await writable.close();

// 读
const f = await file.getFile();
const arrayBuffer = await f.arrayBuffer();

用例：

(1) PWA 大文件存储—— 视频 / 音频 / 大数据 cache，比 IndexedDB 快 10x。

(2) WASM 应用—— FFmpeg.wasm / SQLite WASM 把数据库 / 文件存 OPFS。

(3) 离线游戏 / 应用—— 本地大量数据。

限制：(1) 仅 origin 隔离访问（用户不能"打开文件"看）；(2) 用户清理浏览器缓存可能清掉；(3) 大小受 quota 限制。

:has() 选择器：CSS 父选择器

CSS 历史上一直没有"父选择器"——你不能写"选中含 .foo 子元素的 .bar"。:has()（Chrome 105 / Safari 15.4 / Firefox 121，2022-2024 全引擎稳定）解决：

/* 选中含 .new 标签的卡片 */
.card:has(.new) {
  border: 2px solid red;
}

/* 选中第一个含 img 的 article */
article:has(img) {
  /* ... */
}

/* 复杂条件 */
form:has(input:invalid) button[type="submit"] {
  background: gray;
  cursor: not-allowed;
}

这是 CSS 表达力的巨大提升—— 之前只能 JS 实现的逻辑，现在 CSS 一行。性能也好（浏览器优化得当）。

Scheduler API：详见 P4.2

scheduler.yield() + scheduler.postTask() 是 2024+ 的关键 INP 优化 API。详见 P4.2 交互侧 metrics。

从「JS hack」到「平台原语」的趋势

把这些特性归纳一下，看趋势：

需求 之前的 JS hack 现在的平台原语
页面间动画 Framer Motion / GSAP View Transitions API
组件 responsive useResizeObserver hook Container Queries
预加载下一页 JS prefetch + 自家路由 Speculation Rules
Modal / Dropdown Headless UI / Radix Popover API
定位浮动元素 floating-ui / Popper.js CSS Anchor Positioning
嵌套 CSS SCSS / PostCSS-nested CSS Nesting
父选择器 JS 选择 + 加 class :has() 选择器
大文件存储 IndexedDB OPFS
调度优先级 setTimeout 0 / requestIdleCallback scheduler.postTask / yield
fallback font 字号 JS 算尺寸 + 改 CSS CSS size-adjust / ascent-override

这个趋势的意义：

(1) 性能提升—— 平台原生比 JS 库通常快得多。

(2) 包体积减小—— 不用装第三方库 → bundle 小 → 加载快。

(3) 标准化—— 所有应用用一致行为，accessibility / 表现一致。

(4) 减少生态依赖—— 不用追随 npm 包升级 / 维护。

(5) 浏览器投资被 leveraged—— 你的 stylesheet 直接享受浏览器引擎的优化。

这是"Web Platform 现代化"的核心模式——浏览器持续把开发者反复写的"应该是平台特性"东西变成原语。

未来方向：2026+ 的 Web Platform

从趋势看 2026+ 的 Web Platform 演进方向：

(1) 更多 framework 级原语—— Web Components 持续完善 / Signals API 标准化（Solid / Vue 启发）/ 更多 React-style 特性内置。

(2) Edge runtime 标准化—— Cloudflare Workers / Vercel Edge 的 API 形成标准（Web-interop），所有 Edge runtime 共享。

(3) AI / ML 集成—— WebGPU 成熟、浏览器内置 AI API（Chrome 实验中）。

(4) Privacy 持续演进—— Cookie 退出 / Tracking Protection / Privacy Sandbox API 落地。

(5) Performance 进一步原生化—— Long Animation Frames / Element Timing 等更细粒度 metric。

关注 web.dev/blog + developer.chrome.com/blog + WHATWG / W3C 工作组邮件列表能跟上 Web Platform 演进。

03 应用场景

场景 A · 用 View Transitions 替代 Framer Motion 部分场景

项目用 Framer Motion 做页面间过渡（10kb gzipped + 复杂 API）—— View Transitions API 可以替代部分场景。具体做法：(1) 同 SPA 内的页面切换—— React Router 的 useNavigate wrap 调 startViewTransition；(2) 列表项展开成详情—— 用 view-transition-name 给元素命名，浏览器自动 morph；(3) Tab 切换—— 同样命名 + 默认 cross-fade。限制：(1) Safari / Firefox 不完全支持，需要 fallback；(2) 复杂 spring 动画 / 物理插值 View Transitions 做不到，仍需要 Framer Motion；(3) 跨页面导航（MPA）用 Speculation Rules + View Transitions 组合。综合：简单过渡用 View Transitions，复杂动效仍 Framer Motion—— 减少 50% 库使用。

场景 B · 用 Container Queries 重构组件库

团队组件库长期痛点：组件在 sidebar / main / modal 不同尺寸下 layout 不对。之前用 ResizeObserver hook + JS 算 className——Container Queries 一行 CSS 替代。具体做法：(1) 组件根加 container-type: inline-size；(2) 用 @container (min-width: 400px) 写不同尺寸下的 layout；(3) 移除 ResizeObserver hook + 相关 JS。效果：bundle 小（少几个 hook 库）+ 性能好（CSS 比 JS 监听快）+ 可维护（CSS 写在组件 stylesheet 里，不分散）。这是组件库 2024-2026 重构的标配模式。

场景 C · 用 Speculation Rules 让 MPA 有 SPA 体验

项目是传统 MPA（Rails / Django / WordPress 等）—— 想要 SPA 般的"瞬间导航"但不想重写为 React。Speculation Rules + View Transitions 是答案。具体做法：(1) HTML 加 <script type="speculationrules"> 声明 prerender 规则；(2) 加 View Transitions handler 给跨页面导航过渡；(3) Server 优化 TTFB（CDN + cache）让 prerender 不浪费。效果：用户 hover link 时浏览器后台预渲染下一页，点击时 < 100ms 切换 + 平滑过渡——SPA-feel 体验。但限制：仅 Chrome / Edge 实现，Safari / Firefox 用户仍是传统导航；渐进增强策略——增强用户体验不强制依赖。

场景 D · 给团队制定「现代特性采用」策略

团队问"什么时候采用 View Transitions / Container Queries / Popover 等新特性"——分层策略。(1) Web Platform Baseline 内的特性放心用—— Container Queries / :has / Popover / CSS Nesting 已 baseline 2024，无 fallback 也能跑（仅极老浏览器降级）。(2) Chrome-only 特性用 + fallback—— View Transitions / Speculation Rules / scheduler.yield 用 @supports / 特性检测，提供 fallback。(3) 实验性 / 未稳定特性谨慎用—— Anchor Positioning / Animation Worklet 等仅 Chrome，等其他引擎跟进再用。(4) 持续监控 caniuse—— 每 6 个月 review 你用的特性的兼容性进度，是否能去掉 polyfill。这套策略让团队"早采用红利"而不被兼容性拖累。

05 常见陷阱

用 View Transitions 不做 fallback

document.startViewTransition 在 Safari / Firefox 是 undefined—— 直接调用报错。规避：always 检测特性 + fallback：if (!document.startViewTransition) { directUpdate(); return }。或用 framework 抽象层（如 React Router 7+ 内置 View Transitions support 自动 fallback）。

Container Queries 忘记设 container-type

写了 @container (min-width: 400px) 但父元素没设 container-type——查询不生效。规避：(1) 父元素必须设 container-type: inline-size 或 size；(2) 用命名容器（container-name）让规则关联到具体容器；(3) 用 DevTools 的 Container Queries 调试工具看哪些容器是 active container。

Speculation Rules 浪费带宽

设 eagerness: "immediate" 让浏览器立即预渲染所有匹配 link——浪费带宽 + 占用 CPU / 内存。规避：(1) 用 moderate（hover 时）或 conservative（mousedown 时）；(2) 限制 where 条件，不预渲染所有 link；(3) 监控真实数据看预渲染命中率—— 命中率低就调保守一些。

Popover API 当成 dialog 用

Popover 适合 tooltip / dropdown / menu—— 但不是 dialog 替代品。Dialog 用 <dialog> 元素 + showModal()—— 它有 modal 行为（focus trap 强制 + 隐藏背景）。Popover 是透明 dismiss 的覆盖层，与 dialog 语义不同。规避：modal 用 dialog；tooltip / dropdown 用 popover；具体场景用合适的 API。
