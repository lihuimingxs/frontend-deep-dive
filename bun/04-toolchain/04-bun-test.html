bun test：替代 Vitest / Jest / node:test

测试 runner 章。bun test 是 Jest 兼容 + 启动快的方案——和 Vitest / node:test 三方混战。关键洞察：bun test 定位是"已用 Jest 想加速"；Vitest 护城河是 DOM 测试 + UI dashboard；node:test 护城河是零依赖 + 与 Node 同生命周期——三者各有不可替代的场景。这一章讲透 Jest 兼容范围、snapshot testing、mock 系统、内置 coverage、启动快的真正原因、与 Vitest / Jest / node:test 的横评。

阅读时长 ~30 分钟 前置 Node P6.1 node:test 主要来源 bun.com/docs/test · Vitest docs · Jest docs

01 速览

  1. bun test 的核心定位是「已用 Jest 想加速」——API 和 Jest 大量兼容（describe / test / expect / mock / spyOn），迁移成本低；启动快 5-10x。
  2. Vitest / node:test 各有不可替代的场景—— Vitest 的 DOM 测试 + UI dashboard 是前端组件测试事实标准；node:test 是 Node 22+ 内置零依赖；bun test 是 Jest 兼容加速。三方混战，没有"赢家通吃"。
  3. 启动快的真正原因是工具链一体化—— bun test 启动 ~50ms vs Vitest ~500ms vs Jest ~1-2s。差距来自共享 parser + 没有"启动 Node + 加载 vitest 包"的额外开销。

02 底层逻辑

2026 年的 test runner 格局

JS test runner 在 2026 年是三方混战——和 bundler 赛道相似但定位不同：

Jest (2014+)：Facebook 出品。早期事实标准。但启动慢（~1-2s）+ ESM 支持差 + 配置复杂。新项目越来越少选 Jest，但遗留 Jest 代码量极大（几百万项目）。

Vitest (2021+)：Vite 团队出品。基于 Vite 的 transform pipeline，dev 体验极好（HMR-style watch、UI dashboard、coverage report 都美观）。前端组件测试（DOM + JSX + jsdom）是事实标准。

node:test (Node 18+ 实验，20 稳定)：Node 内置。零依赖、与 Node 同生命周期、CI 配置极简。详见 Node P6.1。但 API 与 Jest 不兼容（更接近 Deno test），迁移成本高。

bun test (2022+)：Bun 内置。Jest API 兼容 + 启动快 + 工具链一体化。是 Jest 用户的最直接迁移路径。

这四个 test runner 的正确认识：不是"谁取代谁"，是各自抢不同的细分市场。Vitest 抢前端；node:test 抢"零依赖小项目"；bun test 抢"Jest 用户想要更快"；Jest 守住"已经在用 Jest 的项目"。未来 5 年这四个会长期共存，不会一家独大。

Jest 兼容：迁移成本最低的卖点

bun test 的核心定位是"Jest API 兼容"。具体兼容到什么程度：

✅ 完全兼容：

  • 核心 API：describe / test / it / beforeEach / afterEach / beforeAll / afterAll
  • expect() 主要 matcher：.toBe / .toEqual / .toContain / .toHaveBeenCalled / .toHaveBeenCalledWith / .toThrow / .toMatch / .toBeNull / .toBeUndefined / .toBeInstanceOf / 等 90%+ 常用 matcher
  • mock() / spyOn()：基础 mock 功能
  • useFakeTimers() / useRealTimers()：定时器 mock
  • Snapshot testing：.toMatchSnapshot()
  • Test filtering：.skip / .only / .each

⚠️ 部分兼容（API 在但行为略有差异）：

  • Module mocking（jest.mock('./module')）—— Bun 实现是 mock.module()，行为对齐但 API 略有差异
  • 一些高级 matcher（.toMatchInlineSnapshot / .toMatchSnapshot({ printer })）部分支持
  • Custom matcher（expect.extend()）—— Bun 1.1+ 支持，但 corner case 可能差异

❌ 不兼容：

  • Jest 完整 plugin 系统（jest.config.js 的 transformer / reporters / projects）
  • Jest 的 globals 配置（一些 Jest plugin 给 globals 加东西）
  • jest.config.js 配置文件（Bun 用 bunfig.toml 配置）

导入路径差异：Jest 通常假设 describe / test 是 globals（"globals": ["describe", "test", "expect"] 在 ESLint 配）。Bun 推荐显式 import：

javascript 
// Jest 风格（globals）
describe('add', () => {
  test('positive', () => {
    expect(add(1, 2)).toBe(3);
  });
});

// bun test 风格（推荐显式 import）
import { describe, test, expect } from 'bun:test';

describe('add', () => {
  test('positive', () => {
    expect(add(1, 2)).toBe(3);
  });
});

// 但 Bun 也接受 globals（Jest 兼容）
// 不 import 直接用 describe/test/expect 也能跑

实测大部分 Jest 测试代码可以直接跑 bun test——基础 expect / describe / test / mock 都兼容。深度用 Jest 高级特性（custom transformer / 复杂 globals / 自定义 reporter）的项目迁移成本高。具体迁移工作量：(1) 试跑 bun test 看哪些测试失败；(2) 失败的查兼容性 issue；(3) 大部分 patch 几行代码就好。

Mock 系统：与 Jest 接近但 API 略不同

Bun mock 系统覆盖 Jest 的常用功能：

javascript 
import { test, expect, mock, spyOn } from 'bun:test';

// 1. 创建 mock function
const fn = mock((x) => x * 2);
fn(3);                          // 6
expect(fn).toHaveBeenCalledWith(3);
expect(fn.mock.calls.length).toBe(1);

// 2. spy on existing method
const obj = { greet: (n) => `hello ${n}` };
const spy = spyOn(obj, 'greet');
obj.greet('world');
expect(spy).toHaveBeenCalledWith('world');

// 3. mock module（替代 jest.mock）
import { mock } from 'bun:test';
mock.module('./db', () => ({
  query: () => Promise.resolve([{ id: 1 }])
}));
// 之后 import './db' 拿到 mock 版

// 4. fake timers
import { setSystemTime } from 'bun:test';
setSystemTime(new Date('2026-01-01'));
expect(new Date().getFullYear()).toBe(2026);

// 5. fetch mock（内置！）
mock.module('global:fetch', () => async (url) => new Response('mocked'));
// 任何 fetch 调用都返回 mocked

这套 mock API 覆盖率 80%+ 的 Jest 用法。复杂场景（如 jest.requireActual、模块工厂的 partial mock）可能差异。规避：(1) 简单场景直接用；(2) 复杂场景查 Bun 文档或提 issue。

Snapshot Testing

Bun 支持 Jest 风格的 snapshot：

javascript 
import { test, expect } from 'bun:test';

test('user shape', () => {
  const user = createUser('Alice');
  expect(user).toMatchSnapshot();
  // 第一次跑：snapshot 写入 __snapshots__/file.test.ts.snap
  // 后续跑：与之前 snapshot 比较，不一致则失败
});

// inline snapshot（snapshot 直接写在 test 文件里）
test('greeting', () => {
  const msg = greet('Bob');
  expect(msg).toMatchInlineSnapshot(`"hello Bob"`);
});

// 更新 snapshot：bun test --update-snapshots

Snapshot 文件格式与 Jest 一致——这意味着从 Jest 迁移时已有的 snapshot 文件能直接复用。这是 Jest 兼容设计的另一个细节红利。

内置 Coverage（不需要 c8 / nyc）

测试 coverage 在 Node 生态传统上要装 c8（V8 native coverage）或 nyc（istanbul-based）。Bun 内置 coverage：

bash 
# 跑测试 + 输出 coverage
bun test --coverage

# 输出格式
# Lines:    87.5%
# Funcs:    91.2%
# Branches: 78.4%
# Statements: 86.9%

# LCOV 格式（CI 报告 / Codecov 上传）
bun test --coverage --coverage-reporter=lcov

# JSON 格式（自定义后处理）
bun test --coverage --coverage-reporter=json

# 强制 coverage 阈值（CI 用）
bun test --coverage --coverage-threshold=80

底层是 JSC native coverage（类似 V8 native coverage），开销小 + 准确度高。比 nyc 的 istanbul instrumentation（在源码插入计数代码）快很多 + 不影响代码执行。

这套内置 coverage 让 Bun test 在 CI 配置上极简：bun test --coverage --coverage-reporter=lcov + 上传 LCOV 给 Codecov / Coveralls 就完事，不需要装 c8 或配置 nyc。

启动快的真正原因

bun test 比 Vitest / Jest 启动快 5-10x。原因和 P4.3 讲的"共享 parser"是同一套机制：

Jest 启动流程：(1) Node 启动 ~30ms；(2) require Jest 包（~700+ 间接依赖）~500ms；(3) 加载 jest.config.js + transformer（如 ts-jest）~300ms；(4) 找测试文件 + transform ~200ms。合计 ~1-1.5s。

Vitest 启动流程：(1) Node 启动 ~30ms；(2) require Vitest（~300 间接依赖，比 Jest 少）~200ms；(3) Vite transform pipeline 启动 ~200ms；(4) 找测试 + transform ~100ms。合计 ~500-700ms。

bun test 启动流程：(1) Bun 启动 ~5ms；(2) bun:test 加载（嵌在 Bun 二进制）~0ms；(3) 用 Bun 自家 parser 找测试 + transpile ~30ms。合计 ~50-100ms。

这个差距在"频繁跑测试"场景累积巨大——每次写一段代码 → 跑 test → 等结果 → 改代码。Vitest 每次 ~500ms 等待，Bun ~50ms 等待，差 10x。一天写代码跑测试 100 次，Bun 比 Vitest 省 50 分钟纯等待。

DOM 测试：Vitest 的护城河

前端组件测试（React / Vue / Svelte）需要 DOM 环境——通常用 jsdom 或 happy-dom 包模拟。Vitest 在这块的优势：

(1) 内置 jsdom / happy-dom 集成—— Vitest 配置 environment: 'jsdom' 一行启用。bun test 不内置 DOM，要装 happy-dom 包 + 在测试 setup 里手动 inject。

(2) Testing Library 集成—— @testing-library/react 等库与 Vitest 配合极好；bun test 也能用但配置更繁琐。

(3) UI dashboard—— Vitest UI 显示测试结果 + 覆盖率热图 + 时间轴，浏览器里点点就能看。bun test 没有等价物（CI 命令行输出）。

(4) Watch 模式—— Vitest watch 是 HMR-style（改一个测试只重跑那个，状态保留）；bun test --watch 行为略简单。

这些差异让前端组件测试场景 Vitest 仍是事实标准。bun test 在这块短期内不能替代。

03 应用场景

场景 A · 从 Jest 迁移到 bun test 加速 CI

团队后端项目用 Jest 跑测试 5 年了——CI 单次 test 要 3-5 分钟（启动 + transform + 跑），改一行代码 push 后等很久。迁移到 bun test：(1) bun install（已有 Bun）；(2) 改 package.json scripts.test 从 jest 改 bun test；(3) 跑 bun test 看失败；(4) 大部分测试直接绿；(5) patch 少数 corner case。典型工作量 1-2 天（中型项目）。CI 时间从 3-5 分钟降到 30s-1min——一天提交 50 次 PR 的团队节省几小时 CI 时间。这是 bun test 的最直接 ROI 场景——已有 Jest 代码 + 想要更快。前端组件测试如果在用 Jest + jsdom，迁移要谨慎：jsdom 兼容性 corner case 多，可能要保留 Jest 或切 Vitest。

场景 B · 决策「前端项目 Vitest 还是 bun test」

新 React / Vue 项目选 test runner——Vitest 还是 bun test？判据：(1) 组件测试需求—— 测 React 组件（jsx + DOM + Testing Library + user-event）→ Vitest 体验远胜（jsdom 集成 + DOM matcher 完整）。(2) UI dashboard 需求—— 想要测试结果 + coverage 的浏览器 UI → Vitest 独有。(3) 已用 Vite—— 同一套 Vite 配置 / plugin / transformer 在 Vitest 复用 → 选 Vitest。(4) 纯逻辑测试（utils / stores / hooks 不带 DOM）—— 两者都行，bun test 更快但 Vitest 体验也好。综合：有 DOM 测试需求选 Vitest；纯逻辑 / 后端 / CLI 选 bun test。前端项目通常 Vitest 主导，bun test 用于跑 utils / API / 后端 server 测试。

场景 C · 用 bun test 跑 monorepo 中所有 package 测试

Monorepo 有 10 个 package，每个有自己的测试。bun test 在根目录跑——自动找所有 package 的 *.test.ts / *.spec.ts 文件，并行跑。比 Jest --projects 配置简单（不需要 jest.config.js），比 Vitest --workspace 配置简单（不需要 vitest.workspace.ts）。具体做法：根 package.json 的 scripts.test = bun test，CI 跑这个。每个 package 不需要单独配置。注意：(1) 测试文件命名—— Bun 默认查 *.test.{js,ts,jsx,tsx} 和 *.spec.{js,ts,jsx,tsx}；(2) 路径过滤—— bun test packages/foo 只跑 foo 包；(3) 并行—— Bun 默认每个测试文件并行（不同 process）。这套 monorepo test 体验是 bun test 的另一个杀手场景。

场景 D · 给 OSS 库维护者：在 CI 矩阵里加 bun test

你维护 npm 库，CI 跑 jest 测试 Node 兼容性——很多 Bun 用户报你的库在 Bun 上挂。解法：CI 矩阵加一行 bun test。具体配置：GitHub Actions 用 matrix: { runtime: ['node-20', 'node-22', 'bun-latest'] }，每个 cell 用对应 runtime 跑测试。不需要给 bun 单独写测试——你的 Jest 测试 90%+ 能在 bun test 直接跑。需要做的：(1) 测试文件用显式 import (import { test } from 'bun:test') 兼容性差，建议保留 Jest 风格 globals（Bun 也接受）；(2) 看哪些测试在 bun 上失败 + 提 issue 给 Bun 团队（他们响应快）；(3) 不能修的 corner case 在 README 标注。这给你的库带来"Bun 兼容"的可信度，几乎零额外维护成本。

04 典型代码

bash 
# 跑所有测试
bun test

# 跑特定文件
bun test ./src/utils.test.ts

# 跑匹配名字的测试（正则）
bun test --test-name-pattern '^add'

# Watch 模式
bun test --watch

# Coverage
bun test --coverage
bun test --coverage --coverage-reporter=lcov > coverage.lcov
bun test --coverage --coverage-threshold=80   # 阈值不达标 fail

# 并行 / 串行
bun test --bail              # 第一个失败就停
bun test --concurrency=4     # 限制并发数

# Update snapshots
bun test --update-snapshots

# Reporter 选择
bun test --reporter=spec     # 详细输出
bun test --reporter=junit > report.xml  # CI 格式
bun test --reporter=tap      # TAP 格式
javascript 
// 完整的测试文件示例
import {
  describe, test, expect,
  beforeEach, afterEach,
  mock, spyOn
} from 'bun:test';

import { createUser, validateEmail } from './user';

describe('User module', () => {
  let db;

  beforeEach(() => {
    db = mock.module('./db', () => ({
      query: mock(async () => [{ id: 1 }])
    }));
  });

  afterEach(() => {
    mock.restore();  // 清除所有 mock
  });

  describe('createUser', () => {
    test('valid email creates user', async () => {
      const user = await createUser('alice@example.com');
      expect(user).toEqual({ id: 1, email: 'alice@example.com' });
    });

    test('invalid email throws', async () => {
      await expect(createUser('not-an-email')).rejects.toThrow('invalid email');
    });

    test.each([
      ['alice@example.com', true],
      ['bob+filter@example.co.uk', true],
      ['no-at-sign', false],
      ['@no-name.com', false]
    ])('validateEmail(%s) === %s', (input, expected) => {
      expect(validateEmail(input)).toBe(expected);
    });
  });

  describe('snapshot', () => {
    test('user shape stable', () => {
      const user = { id: 1, email: 'a@b.c', createdAt: new Date('2026-01-01') };
      expect(user).toMatchSnapshot();
    });
  });

  describe('mock + spy', () => {
    test('spies on existing method', () => {
      const obj = { greet: (n) => `hi ${n}` };
      const spy = spyOn(obj, 'greet');
      obj.greet('world');
      expect(spy).toHaveBeenCalledWith('world');
      expect(spy.mock.calls.length).toBe(1);
    });
  });
});

    runs-on: ubuntu-latest

          - { name: node, version: 20 }

        with: { bun-version: '${{ matrix.runtime.version }}' }

      # 同一份测试在 Node 和 Bun 都跑
      - run: bun install
      - run: bun test --coverage --coverage-reporter=lcov

      - uses: codecov/codecov-action@v4
        with: { files: ./coverage.lcov }

05 常见陷阱

期待 bun test 100% 兼容 Jest 高级特性

bun test 兼容 Jest 的核心 API（90%+ 用法），但高级特性（custom transformer / 复杂 reporter / Jest plugin / 一些 globals 配置）不完全。深度用 Jest 的项目迁移可能要改测试代码或 fallback。常见踩坑：(1) 用了 jest-extended 或类似 matcher 扩展包；(2) 用了 jest.config.js 的 transform / preset；(3) 用了 jest-environment-* 自定义环境。规避：迁移前先 spike——跑 bun test 看哪些测试失败，估算修复成本。如果失败的多 / 难修，保留 Jest 或切 Vitest（Vitest 也兼容 Jest 但生态更深）。

bun test 没有 DOM 内置

从 Vitest 迁移想测 React 组件——发现 document is not defined。Bun test 不内置 DOM；要装 happy-dom（推荐）或 jsdom 包，再在测试 setup 里手动 inject。具体做法：bun add -d happy-dom，然后测试文件顶部 import { GlobalRegistrator } from "@happy-dom/global-registrator"; GlobalRegistrator.register();。或在 bunfig.toml 里 [test] preload = ["./test-setup.ts"] 集中配置。比 Vitest 一行 environment: 'jsdom' 麻烦。前端组件测试场景 Vitest 仍是更顺畅的选择。

把 bun test 启动快误读为 「测试本身也快」

bun test 启动快（~50ms vs Vitest ~500ms），但测试本身的执行时间不一定显著快——一个跑 10s 的测试在哪个 runner 上都是 10s（看你的代码逻辑）。Bun test 的优势集中在"频繁启动"场景：dev 时改一行跑一次 test、CI 启动开销摊薄、watch mode reactivity。如果你的瓶颈是测试本身慢（数据库 setup / 长 hot loop），换 runner 收益小——优化测试代码本身。

期望 bun test 能直接跑 Vitest 测试

Vitest 测试代码用 import { test } from 'vitest'——bun test 不认识这个 import 路径。从 Vitest 迁移要把 'vitest' 改 'bun:test'（或删除 import 用 globals）。Vitest 一些独有 API（如 vi.mock / vi.useFakeTimers）有 Bun 等价物（mock / useFakeTimers）但 API 不完全一致。规避：迁移时用 sed / codemod 批量替换；或保留 Vitest（DOM 测试场景仍 Vitest 主导）。
