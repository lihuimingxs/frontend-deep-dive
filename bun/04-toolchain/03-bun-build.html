bun build：bundler + --compile SEA 替代

bundler 章。bun build 设计目标：在 Vite / Rolldown / esbuild / Turbopack 这个高度内卷的赛道里，靠"和 runtime / install 共享 parser"差异化。关键洞察：bun build 单看 bundler 速度差距不大，差异化在于"一体化"——和 install / run / test 共享 parser，每个工具的启动都摊薄了 parser 加载成本。这一章讲透 Zig parser、tree shaking、code splitting、source map、--compile（SEA 替代）、bytecode caching、CSS bundler、与 Vite / Rolldown / Turbopack 横评。

阅读时长 ~30 分钟 前置 P4.2 + Node P6.1 SEA 主要来源 bun.com/docs/bundler · Bun 1.2 bytecode caching blog

01 速览

  1. bun build 不是要颠覆 Vite / Rolldown——它在已经内卷的 bundler 赛道里靠"和其他 Bun 工具共享 parser"差异化，单看速度和 esbuild / Rolldown 接近。
  2. --compile 是 Node SEA 的「完成版」——交叉编译、ESM 默认、产物 ~50MB、bytecode caching。Node SEA 还在追赶。
  3. 共享 parser 红利在大型项目和工具链一体化场景才显著——install + run + build + test 都用 Bun 时，每次启动的 parser 加载被摊薄；只用 build 不用其他场景，差距小。

02 底层逻辑

2026 年的 bundler 竞争格局

bundler 是 JS 工具链最内卷的赛道。2026 年的格局：

Vite (2020+)：尤雨溪发起，目前事实标准 dev server。dev 用 esbuild + native ESM，build 用 Rollup。Vite 5+ 转向 Rolldown 替代 Rollup（性能更好）。生态最强——Vue / React / Svelte 等框架都默认 Vite。

Rolldown (2024+)：Rust 写的 Rollup 替代品（同样作者：尤雨溪 + Vite 团队）。目标是"用 Rust 重写 Rollup 拿到 esbuild 的性能 + Rollup 的功能"。Vite 6 默认 build 工具。

esbuild (2020+)：Evan Wallace 写的 Go bundler。最早的"native bundler"，速度极快。但 plugin 生态弱、tree shake 不完美。多数被用作"底层 bundler"（被 Vite / 其他工具调用），不是独立产品。

Turbopack (2022+)：Vercel 出品 Rust bundler，Next.js 14+ 默认。但目前只在 Next.js 用，不通用。

SWC (2019+)：Donny Wong 写的 Rust JS/TS compiler。多数被用作"底层 transformer"（如被 Next.js / Parcel 调用）。和 esbuild 是互补关系。

Webpack (2014+)：仍然是企业级遗留项目的主力。新项目极少选。

bun build (2022+)：Bun 自家 Zig bundler。性能与 esbuild 持平 + 功能与 Rolldown 持平 + 工具链一体化。

这个赛道性能差异已经被压到很小——esbuild / Rolldown / SWC / Turbopack / bun build 在常规项目上 build 时间差距不到 50%。继续优化纯速度边际效益低。差异化转向：(1) 工具链一体化（Bun）；(2) Plugin 生态（Vite / Rollup）；(3) 框架深度集成（Turbopack 之于 Next.js）。bun build 的核心定位是 (1)。

共享 parser 红利

"共享 parser"是 Bun 工具链的核心架构红利。具体看：Bun 的 install / run / build / test 都用同一个 Zig 写的 JS/TS parser——它在 Bun 二进制里只加载一次，每个子命令调用直接用，不需要单独启动 / 加载 / 初始化。

对比 Node 工具链：

Node 工具链：npm install → 启动 Node + 加载 npm；vite dev → 启动 Node + 加载 Vite（含 esbuild）；vitest → 启动 Node + 加载 Vitest（再含 esbuild）；rollup build → 启动 Node + 加载 Rollup（含 acorn parser）。每次都重新启动 Node + 重新加载 parser——加载次数 = 启动次数。

Bun 工具链：bun install / bun run / bun build / bun test 都是同一个 Bun 二进制的子命令。Bun 启动一次（~5ms），parser 加载一次（嵌在二进制里几乎瞬间），后续命令直接用。

具体的累积差距看一个典型 dev 流程：

步骤 Node 工具链 Bun 工具链 差距
install（冷启） Node + npm 启动 ~50ms Bun 启动 ~5ms ~45ms
run dev Node + Vite ~300ms Bun ~10ms ~290ms
每次改代码 transpile esbuild loaded → transpile 文件 Bun parser 直接 transpile microseconds
跑 test Node + Vitest 启动 ~500ms Bun test ~50ms ~450ms
build Node + Vite + Rolldown ~500ms 启动 + build Bun build ~10ms 启动 + build ~490ms 启动开销

每个步骤的差距单看不大，但 dev 一天里 install 1 次 / run 5 次 / test 50 次 / build 10 次——累积下来 Bun 比 Node 工具链节省几十秒到几分钟每天。一年下来一个工程师几十小时。一个 50 人团队几千小时——这是 Bun 工具链一体化的真正 ROI，不是单个工具的 5x / 10x。

bun build 核心特性

把 bun build 主要功能列一遍：

(1) 输入支持——.js / .ts / .jsx / .tsx / .mjs / .cjs / .json / .css（1.2+） / .html（1.2+） / 静态资源（.png / .svg 等）。

(2) 输出格式——ESM / CJS / IIFE。--target=bun（针对 Bun 优化）/ --target=node（针对 Node）/ --target=browser（针对浏览器）。每个 target 启用不同的内置代码（如 polyfill）。

(3) Tree shaking——基于 ES Module 静态分析，未用到的 export 会被剪掉。和 esbuild / Rolldown 等价。

(4) Code splitting——动态 import() 自动分块，每块单独输出文件。Bundler 默认行为。

(5) Source map——--sourcemap=linked（默认）/ --sourcemap=inline（嵌入产物）/ --sourcemap=external（独立 .map 文件）。

(6) Minify——--minify 启用。压缩 + mangle 名 + 删 dead code。比 terser / swc minify 速度接近。

(7) Plugin API——和 esbuild 类似的 plugin 接口（onLoad / onResolve）。许多 esbuild plugin 能直接用。

(8) Asset 处理——.png / .svg 等静态资源 hash + 复制到产物，import 返回 URL 字符串。

(9) CSS bundler (1.2+)—— @import 解析、url() 处理、CSS modules 实验。1.2 之前 CSS 是 stub，1.2 完整支持。

(10) Bytecode caching (1.2+)—— Bun 把 JSC bytecode 缓存到 .bun-cache，第二次跑时跳过 parse。启动时间额外节省 30%。

--compile：Node SEA 的「完成版」

bun build --compile 是 Bun 的"编译成单文件可执行程序"特性。这是 Node SEA（详见 Node P6.1 现代 Node 工程）的对应物：

bash 
# 把 server.ts + Bun runtime 打成一个二进制
bun build --compile src/server.ts --outfile=my-server

# 跑：
./my-server
# 启动 ~5ms，零依赖（不需要装 Bun）

# 交叉编译（在 macOS 编译给 Linux 跑）
bun build --compile --target=bun-linux-x64 src/server.ts --outfile=my-server-linux
bun build --compile --target=bun-linux-arm64 src/server.ts --outfile=my-server-arm64
bun build --compile --target=bun-windows-x64 src/server.ts --outfile=my-server.exe

对比 Node SEA：

维度 Node SEA Bun --compile
稳定性 Node 22+ 实验 Bun 1.0+ 稳定
入口模块系统 仅 CJS（22+ 开始 ESM 实验） ESM 原生
多文件支持 实验中 完整支持（默认 bundle 整个 import 树）
产物体积 ~80MB（Node 二进制） ~50MB（Bun 二进制）
交叉编译 不支持 支持（Linux / macOS / Windows + x64 / arm64）
Bytecode caching 有（V8 startup snapshot） 有（JSC bytecode）
启动时间 ~30-50ms ~5-10ms

bun --compile 在每个维度上都领先 Node SEA——这不奇怪，因为 Bun 团队把 SEA 当作核心特性优先做，Node 跟进得晚。

典型用例：

(1) CLI 工具分发——团队内部 deploy / migrate / lint 工具，给运维 / 非前端同事用。bun build --compile 一次，scp 上去就能跑，不需要装 Bun / Node。

(2) 容器极简化——Dockerfile 只 COPY 这一个二进制，不需要 base image 装 runtime + node_modules。镜像体积从 ~200MB（Node alpine + deps）降到 ~50MB（单二进制）。

(3) Lambda 部署——AWS Lambda custom runtime / Cloudflare Workers（实际上 Workers 不能跑 Bun，但其他类似平台能）。一个二进制就是整个 deployment。

(4) 桌面工具—— 写一个跨平台的桌面 CLI / GUI（不带 Electron），--compile 三个 target 编译三份。

Bytecode caching（Bun 1.2+）

Bun 1.2 引入了JSC bytecode 缓存。机制：第一次跑代码时 JSC parse 出 bytecode → Bun 把 bytecode 缓存到磁盘 → 第二次跑直接 mmap 缓存的 bytecode，跳过 parse。

这套机制借鉴自Node 的 V8 startup snapshot（详见 V8 P5.1 嵌入 V8）和 Chrome 的 V8 cache。但 Bun 实现得更激进——所有 JS 文件 + npm 包都自动缓存，不需要用户配置。

实际效果：

冷启（无缓存）：和不开缓存一样，~30ms（中型项目）。

热启（有缓存）：~10-20ms，节省 30-60%。

这个优化对"启动很多次"的场景显著——CLI 工具、Lambda warm 调用、CI 任务。对长跑服务（启动一次跑几天）影响小，因为启动时间不重要。

限制：

(1) 缓存失效——代码改动后缓存自动失效（Bun 检查文件 hash），不需要手动清。

(2) 占磁盘——一个中型项目的 cache 几十 MB。~/.bun/install/cache 路径，可以手动清。

(3) 不影响 production 部署——production 通常用 bun build --compile，cache 内嵌在二进制里，不依赖磁盘 cache 目录。

与 Vite / Rolldown / esbuild / Turbopack 横评

把 5 个主流 bundler 在不同维度对比（2026 年状态）：

维度 bun build Vite (Rolldown) esbuild Turbopack SWC
语言 Zig Rust Go Rust Rust
定位 一体化 bundler + runtime dev server + bundler 底层 bundler Next.js bundler 底层 transformer
速度（中型项目） ~快 ~快 极快 ~快 极快
Tree shake ✅ ✅ ⚠️ 不完美 ✅ N/A
Code split ✅ ✅ ⚠️ 简单 ✅ N/A
Plugin 生态 esbuild 兼容 极强（Vite + Rollup） 中等 仅 Next.js 内部 少
HMR --hot（runtime 端） 极强（dev server 端） 无 极强（Next.js 内） 无
Compile 单二进制 ✅ ❌ ❌ ❌ ❌
CSS 处理 1.2+ 完整 极强 有 极强 无
HTML imports 1.2+ 实验 无 无 有 无

结论：

选 bun build 的场景：(1) 已用 Bun runtime / install / test，工具链一体化最大化；(2) 需要 --compile 单二进制；(3) 后端项目（Node-style，不需要复杂前端 dev server）。

选 Vite (Rolldown) 的场景：(1) 前端项目（Vue / React / Svelte），需要强 dev server + HMR + plugin 生态；(2) 复杂的 CSS 处理需求；(3) 已用 Vite 的项目继续用。

选 esbuild / SWC 的场景：作为底层组件被其他工具调用——大部分情况不直接用。

选 Turbopack 的场景：用 Next.js 14+ 的项目（不需要选，Next.js 默认）。

这种横评里 bun build 不是赢家也不是输家——它在自己的定位上表现优秀（一体化 + --compile），其他 bundler 在自己的定位上更强。Selecting 取决于你的场景。

03 应用场景

场景 A · 用 bun build --compile 分发 CLI 工具

团队内部做了个 deploy CLI（含 30 个 npm 依赖），给运维同事用。传统方案：让运维装 nvm + Node + npm install -g。痛点：每次更新都要让运维重新装；运维机器可能没 Node 环境。bun build --compile 解决：bun build --compile src/cli.ts --outfile=deploy-cli，scp 给运维 → 直接 ./deploy-cli。一个二进制 50MB，零依赖。如果要支持多平台：交叉编译 --target=bun-linux-x64 + bun-darwin-arm64 + bun-windows-x64，一次 build 三个二进制。这种场景下 Bun --compile 的体验远胜 Node SEA（Node 22 SEA 仍是实验，且不支持交叉编译）。

场景 B · 用 bun build 替代 esbuild 做 server 后端 build

典型 Node 后端项目：用 esbuild bundle TS server 代码到 dist/。这个流程在 Bun 上 native 支持：bun build src/server.ts --target=bun --outfile=dist/server.js。差异：(1) --target=bun 输出针对 Bun 优化（保留 Bun-specific API）；(2) 启动从 npx esbuild 的 ~500ms 降到 bun build 的 ~10ms（Bun 工具一体化红利）；(3) 与 install + run + test 共享 parser 加速 dev 流程。如果你的 server 仍然 deploy 到 Node，用 --target=node。如果 deploy 到 Bun，--target=bun。Production deploy 时 build 产物 + 启动 bun dist/server.js 而不是 bun src/server.ts——避免每次启动 transpile 开销。

场景 C · 用 bun build 做 Docker 极简化

典型 Node 后端 Dockerfile：FROM node:20-alpine + RUN npm install + COPY src + CMD node ... 。镜像 ~200MB（Node base + deps）。改用 Bun --compile：FROM scratch + COPY my-server /my-server + CMD ./my-server。镜像 ~50MB，只有一个二进制。优势：(1) 体积小—— 4x 减少，加速 push / pull；(2) 安全—— scratch base 没有 shell / OS 工具，attack surface 极小；(3) 启动快—— 容器冷启动从几秒到 ~50ms。劣势：(1) 调试不方便（容器里没 sh）；(2) 不能 exec 进容器跑命令；(3) 部分系统库依赖（如 OpenSSL）需要静态链接（Bun 默认包含）。这种 deployment 适合无状态微服务 + Lambda + Cloudflare Container等场景。

场景 D · 决策「前端项目用 Vite 还是 bun build」

团队前端项目（React / Vue / Svelte）选 bundler——Vite 还是 bun build？判据：(1) HMR 需求——前端开发依赖 HMR（改 component 不丢 state）。Vite HMR 极强（生态深 + 框架集成 React Refresh / Vue HMR）；bun build 1.2 时仍在追，复杂场景体验不如 Vite。(2) Plugin 生态——Vite/Rollup plugin 上千个（PWA / mock / sentry / 各种 CSS 预处理）；bun build plugin 兼容 esbuild 但生态较小。(3) CSS 处理——Vite 极强（PostCSS / Sass / Less / CSS Modules / Tailwind 集成）；bun build 1.2 完整支持但生态薄。(4) build 速度—— Vite 5+ 用 Rolldown 已经接近 bun build。综合后：前端项目优先 Vite（成熟生态 + HMR 体验）；后端 / CLI / 工具项目优先 bun build（一体化 + --compile）。前端 + 后端都用 Bun 的 fullstack 项目可以试 Bun.serve + HTML imports，但生态尚浅，先做 prototype 验证。

04 典型代码

bash 
# 基础 build 命令
bun build src/index.ts --outfile=dist/bundle.js
bun build src/index.ts --outdir=dist/ --target=node       # ESM 默认
bun build src/index.ts --outdir=dist/ --format=cjs        # CJS 输出

# Tree shake + minify + sourcemap
bun build src/index.ts \
  --outfile=dist/bundle.js \
  --minify \
  --sourcemap=linked \
  --target=browser

# Compile 成单二进制
bun build --compile src/cli.ts --outfile=my-cli

# 交叉编译
bun build --compile --target=bun-linux-x64 src/cli.ts --outfile=my-cli-linux-x64
bun build --compile --target=bun-darwin-arm64 src/cli.ts --outfile=my-cli-darwin-arm64
bun build --compile --target=bun-windows-x64 src/cli.ts --outfile=my-cli.exe

# Watch 模式（边改边 build）
bun build src/index.ts --outdir=dist/ --watch

# External（不打进 bundle，运行时 require）
bun build src/index.ts \
  --external=react \
  --external=react-dom \
  --target=browser

# 多入口
bun build src/index.ts src/worker.ts --outdir=dist/ --target=browser
javascript 
// 编程式 API（在脚本里调 build）
import { build } from 'bun';

const result = await build({
  entrypoints: ['./src/index.ts'],
  outdir: './dist',
  target: 'bun',
  minify: true,
  sourcemap: 'linked',
  external: ['react'],
  define: {
    'process.env.NODE_ENV': JSON.stringify('production'),
  },
  plugins: [
    {
      name: 'my-loader',
      setup(builder) {
        builder.onLoad({ filter: /\.svg$/ }, async ({ path }) => {
          const text = await Bun.file(path).text();
          return {
            contents: `export default ${JSON.stringify(text)}`,
            loader: 'js'
          };
        });
      }
    }
  ]
});

if (!result.success) {
  for (const log of result.logs) {
    console.error(log);
  }
}
json 
// package.json 集成 build script
{
  "scripts": {
    "build": "bun build src/server.ts --outdir=dist --target=bun --minify",
    "build:cli": "bun build --compile src/cli.ts --outfile=bin/my-cli",
    "build:cross": "bun run build:cli && bun build --compile --target=bun-linux-x64 src/cli.ts --outfile=bin/my-cli-linux-x64",
    "dev": "bun --hot src/server.ts",
    "start": "bun dist/server.js"
  }
}

05 常见陷阱

用 --target=node 打包但跑 Bun runtime

--target=node 输出针对 Node 优化的 bundle（含 Node API 兼容代码 / 不内联 Bun-specific 优化）。如果产物部署到 Bun runtime 跑，会少拿到 Bun 优化。规避：deploy 到 Bun 用 --target=bun；deploy 到 Node 用 --target=node；deploy 到浏览器用 --target=browser。target 选错不会崩，只是没拿到对应优化。

--compile 产物跨 macOS Intel / ARM 不兼容

Apple Silicon（M1/M2/M3）是 ARM64 架构，Intel Mac 是 x64 架构——两个二进制不兼容。如果你在 M1 上 bun build --compile 编了一个二进制，scp 给 Intel Mac 同事会"cannot execute binary file"。规避：知道自己的目标架构 + 用 --target=bun-darwin-x64 或 --target=bun-darwin-arm64 显式指定；distribute 时给两个二进制让用户选。

bun build 不自动处理 .css 在 1.2 之前

Bun 1.2 之前 import styles from './app.css' 是 stub——产物里 CSS 不被处理 / 不被 inline / 不被 hash。Bun 1.2 完整支持。如果你用 Bun 1.0 / 1.1 做前端 build，CSS 部分要 fallback 到 Vite / esbuild 单独跑。规避：升 Bun 1.2+；或前端构建仍用 Vite 而 server build 用 bun build。

期望 esbuild plugin 100% 兼容 bun build

Bun 实现了 esbuild 风格的 plugin API，大部分 esbuild plugin 能直接用。但少数 plugin 用了 esbuild 特定的内部 API 或行为，在 Bun 上可能有微妙差异。常见问题：(1) plugin 用 build.initialOptions（esbuild 内部）—— Bun 实现可能不一致；(2) plugin 依赖 esbuild 的特定文件解析顺序——Bun 可能不同；(3) plugin 用 esbuild 的 metafile 格式——Bun 输出格式略有差异。规避：(1) 看 plugin GitHub 是否声明 Bun 兼容；(2) build 失败时跑 minimal repro 提 issue；(3) 不能用的 plugin 找 Bun 等价物或自己写。
