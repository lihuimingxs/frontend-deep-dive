FFI / Macros / 内置 SQLite + Redis

高级特性章。这些不是入门必须，但 Bun 能在某些"超 Node 范畴"的场景胜出靠这些。关键洞察：FFI vs N-API 不是替代关系——FFI 灵活、不需要编译，适合"调系统库"；N-API 兼容性好（Node 也能跑），适合"复用 npm 生态"。Bun 同时支持两者。这一章讲透 bun:ffi（dlopen 风格调系统共享库）、Macros（编译期代码生成）、bun:sqlite（内置 SQLite）、Bun.sql（Postgres）、Bun.redis、HTML imports。

阅读时长 ~25 分钟 前置 P3 + Node P5.2 N-API 主要来源 bun.com/docs/api/ffi · bun.com/docs/api/sqlite · bun.com/docs/api/sql

01 速览

  1. bun:ffi 是「不编译就能调 native」的快速通道——比 N-API 写起来简单（不用编 C++ + node-gyp），但每次调用有 marshal 开销，比 N-API 慢。适合调系统库 / Rust/Zig 库 / 快速原型。
  2. Macros 是 JS 版的编译期代码生成——build 时跑某些函数把结果嵌进产物（如 SVG → 组件、版本号注入）。比 Zig comptime 弱（JS 表达力限制），但比 babel-plugin / TS transformer 简单。
  3. 内置 DB 客户端的争议——bun:sqlite + Bun.sql + Bun.redis 给低门槛体验，但锁定 API 风格；如果未来需求超出（要 ORM / 高级特性），仍要 fallback npm 包。

02 底层逻辑

bun:ffi：调 native 库的快速通道

bun:ffi 是 Bun 提供的外部函数接口——能在 JS 里直接调用共享库（.so / .dylib / .dll）的函数，不需要写 C++ 包装。

javascript 
import { dlopen, FFIType } from 'bun:ffi';

// 加载 SQLite 共享库（系统已装）
const lib = dlopen('libsqlite3.dylib', {
  sqlite3_libversion: {
    args: [],
    returns: FFIType.cstring  // 返回 C string
  },
  sqlite3_open: {
    args: [FFIType.cstring, FFIType.ptr],
    returns: FFIType.i32
  }
});

// 直接调
console.log(lib.symbols.sqlite3_libversion());
// 输出：3.45.0 或类似版本

对照 Node 的 native 集成：

Node N-API 路径（详见 Node P5.2 N-API）：(1) 写 C++ 包装代码用 N-API 暴露 JS 接口；(2) 写 binding.gyp 描述编译；(3) node-gyp rebuild 编译成 .node 文件；(4) require 这个 .node。每个 native module 都是一个独立 npm 包，要发布维护。

Bun FFI 路径：(1) dlopen 已存在的共享库；(2) 声明函数签名；(3) 直接调。不需要编译、不需要发布、不需要维护额外包。

这种差异让"调系统库"和"快速试 native"场景下 Bun FFI 远胜 N-API。比如：

(1) 调系统库—— OpenSSL / SQLite / libcurl / libgit2 / libpq 这些系统已装的共享库。Bun FFI 一行 dlopen 就能用，不用 npm 包装。

(2) 调 Rust / Zig 写的库—— 把 Rust crate 编译成 cdylib，Bun FFI 直接调。Rust 不需要写 N-API binding 代码（napi-rs 那套），就是普通 extern "C"。

(3) 快速原型—— 试一下"调 X 库能不能跑"，FFI 5 分钟搞定；N-API 要写 C++ + 编译，半天起步。

FFI 的劣势：

(1) 调用开销比 N-API 高—— FFI 每次调用要 marshal 参数（JS 类型 → C 类型），返回值也要 marshal。N-API 编译时已经把 marshal 内联了。Hot path（每秒百万次调用）上 FFI 慢 2-5x。

(2) 类型系统受限—— FFI 只支持基础 C 类型（i8/i16/i32/i64/f32/f64/ptr/cstring）。复杂结构（C struct / union）需要手动 marshal。N-API 可以用 C++ 模板表达任意类型。

(3) 错误处理薄—— FFI 函数 panic / segfault 直接挂掉整个进程；N-API 有 napi_status 错误传播机制。

判断 FFI vs N-API：

场景 FFI 适合 N-API 适合
调系统库 ✅ 开箱即用 ⚠️ 要写 C++ 包装
调 Rust crate ✅ cdylib 直接调 ⚠️ 要 napi-rs
性能 hot path ⚠️ marshal 开销大 ✅ inline marshal
复杂 C++ 类型 ⚠️ 手动 marshal ✅ C++ 模板表达
跨 Node + Bun ❌ FFI 是 Bun-only ✅ N-API 两边都能跑
分发到用户 ⚠️ 用户要装 lib ✅ npm 包带 prebuild

结论：本地开发 / 工具 / 内部服务用 FFI（开发体验好）；分发 npm 包 / 跨 runtime / hot path用 N-API（兼容性好 + 性能好）。

Macros：JS 版编译期代码生成

Bun Macros 让你在build 时跑某些 JS 函数 + 把结果嵌入产物。语法用 import attribute 标记：

javascript 
// macros.ts —— 在 build 时跑的函数
export async function readSvgIcon(name: string) {
  const path = `./icons/${name}.svg`;
  return await Bun.file(path).text();
}

export function buildTime() {
  return Date.now();
}

// app.ts —— 用 macro
import { readSvgIcon, buildTime } from './macros.ts' with { type: 'macro' };

// build 时 readSvgIcon('home') 在编译期被调用
// 产物里直接是 SVG 字符串字面量，不会运行时读文件
const homeIcon = readSvgIcon('home');
const compiledAt = buildTime();

这种"编译期跑函数 + 把结果嵌入产物"在 Rust 里叫宏，Zig 里叫 comptime（详见 P2.2 Zig 系统层），TS 里叫 transformer。Bun Macros 是 JS 层的简化版。

典型用例：

(1) 静态资源 inline—— SVG / 图片 base64 / 字体在 build 时读进产物，运行时不读文件。

(2) 编译期常量—— version / commit hash / build time 嵌入产物，不需要运行时查 git。

(3) 数据预处理—— JSON / YAML 在 build 时 parse + 验证 + 嵌入产物，运行时直接用对象不用 parse。

(4) 类型生成—— 从 schema 生成 TS 类型（虽然这块 zod / valibot 已经做得很好）。

Macros 的限制：

(1) JS 表达力受限—— Macros 跑的是普通 JS，不能像 Zig comptime 那样做"编译期反射"。复杂场景仍然要 babel plugin / TS transformer。

(2) 只能在编译期跑—— 不能"运行时也能调"。如果你想"build 时优化但 runtime 也能用"要写两份。

(3) Bun-only—— Node + Vite 等其他工具没有这个 import attribute 语法。Macros 的代码不能跨 runtime。

Macros 是 Bun 社区不太常用的特性——大部分项目不需要编译期生成代码，需要的有 Vite plugin / babel plugin 解决。Macros 的定位是"轻量级编译期助手"，而不是"通用代码生成系统"。

bun:sqlite：内置 SQLite 客户端

bun:sqlite 是 Bun 内置的 SQLite 客户端——底层是 SQLite C 库 + Zig glue。

javascript 
import { Database } from 'bun:sqlite';

// 打开（或创建）数据库
const db = new Database('./mydb.sqlite');

// 建表
db.run(`CREATE TABLE IF NOT EXISTS users (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT UNIQUE
)`);

// Prepared statement + 参数
const insert = db.prepare('INSERT INTO users (name, email) VALUES (?, ?)');
insert.run('Alice', 'alice@example.com');
insert.run('Bob', 'bob@example.com');

// 查询
const user = db.query('SELECT * FROM users WHERE name = ?').get('Alice');
const allUsers = db.query('SELECT * FROM users').all();

// 事务
const insertMany = db.transaction((users) => {
  for (const u of users) insert.run(u.name, u.email);
});
insertMany([{ name: 'C', email: 'c@x.com' }, { name: 'D', email: 'd@x.com' }]);

API 设计学习自 better-sqlite3（npm 上的事实标准 SQLite 客户端），但不需要 native 编译（不需要 node-gyp / Python / g++）。

性能与 better-sqlite3 持平：

better-sqlite3 是 N-API 包，内部直接调 SQLite C 库 + 高度优化。在 Node 上跑得很快。

bun:sqlite 是 Bun 内置，Zig 包装 SQLite C 库。性能与 better-sqlite3 几乎相同（都是 native call to SQLite），实测 micro benchmark 差距 < 5%。

Bun 的优势是"不需要编译"—— Node 用 better-sqlite3 在 Docker 里 install 经常遇到 node-gyp 错（缺 Python / g++ / 内核 header）；Bun 用 bun:sqlite 一切内置。

Bun 1.2+ 还在 bun:sqlite 上加了：

(1) Stream cursor—— 大结果集 streaming，不爆内存。

(2) WAL mode 默认推荐—— Write-Ahead Logging，并发性能更好。

(3) Type cast—— 自动把 SQLite 类型映射到 JS（INTEGER → number / TEXT → string / BLOB → Buffer）。

Bun.sql：Postgres 客户端

Bun.sql（Bun 1.2+）是内置 Postgres 客户端。详细介绍在 P3.1 Bun.serve / Bun.file / Bun.sql，本节简要展开：

javascript 
import { sql } from 'bun:sql';

// 默认从 DATABASE_URL 读配置
const db = sql({ url: 'postgres://user:pwd@localhost/mydb' });

// Tagged template 自动 prepared statement + 防 injection
const users = await db`SELECT * FROM users WHERE active = ${true}`;

// 事务
await db.begin(async (tx) => {
  await tx`INSERT INTO orders (user_id) VALUES (${userId})`;
  await tx`UPDATE inventory SET count = count - 1 WHERE id = ${itemId}`;
});

// 流式 cursor
for await (const row of db`SELECT * FROM big_table`.cursor(1000)) {
  process(row);
}

API 学习自 postgres.js（社区 ESM-friendly Postgres 客户端），底层是 Bun 自家 Zig 实现 + Postgres 二进制协议直通。性能比 npm pg 包快约 50-100%。

Bun.redis：Redis 客户端（实验）

Bun.redis（Bun 1.2+，实验）是内置 Redis 客户端：

javascript 
import { redis } from 'bun';

// 默认从 REDIS_URL 读配置
await redis.set('user:1', JSON.stringify({ name: 'Alice' }));
const data = await redis.get('user:1');

// Pipeline
const pipeline = redis.pipeline();
pipeline.set('a', 1);
pipeline.set('b', 2);
pipeline.set('c', 3);
await pipeline.exec();

// Pub/Sub
const sub = redis.subscribe('channel');
for await (const msg of sub) {
  console.log('received:', msg);
}

底层是 Zig 实现的 Redis 二进制协议。性能与 ioredis 持平 + 不需要装 npm 包。

仍然实验性——少数高级特性（Lua scripting / Cluster mode / Sentinel 自动 failover）支持还不完整。生产用 Redis 重度场景仍推荐 ioredis。

HTML imports：fullstack 实验路径

HTML imports（Bun 1.2+）让 Bun 能直接服务 HTML + 自动 bundle 引用的 TSX / CSS / 资源。详见 P3.1。这里点出几个细节：

定位：Bun 想让用户用一个 runtime + 一个二进制做完整 fullstack 项目，不需要 Vite / Webpack / Next.js。

当前限制：(1) 不支持 RSC (React Server Components)；(2) HMR 比 Vite 弱；(3) SSR 需要手动写。

适合场景：(1) 简单内部工具 / dashboard；(2) 学习 / prototype；(3) 技术 demo。

不适合：(1) 复杂前端项目（仍 Next.js / Vite 主流）；(2) 大型团队协作（Bun fullstack 生态薄）。

"内置 DB 客户端"的争议

Bun 把 SQLite / Postgres / Redis 客户端做成内置——社区有不同看法：

支持方：(1) 零依赖体验好；(2) 不需要 npm 包 + node-gyp 编译；(3) 性能与社区主流持平或更好；(4) Web 标准 API 风格（Promise / 模板字面量）；(5) 文档统一在 bun.com 上。

反对方：(1) API 锁定—— 用 Bun.sql 写的代码 vendor lock-in 到 Bun，切回 Node 要改；(2) 功能不全—— 高级特性（如 Postgres 的 listen/notify / replication）实现可能不完整；(3) ORM 仍然要装包—— Bun.sql 是底层客户端，Prisma / Drizzle 这种 ORM 还是 npm 包；(4) 覆盖范围有限—— 没有 MySQL / MongoDB / Cassandra 客户端，那些场景仍要 npm 包。

这种争议没有标准答案——看你的项目场景：(1) 简单 SQLite / Postgres / Redis 的项目，Bun 内置更舒服；(2) 复杂场景或要切回 Node 的灵活性，仍 npm 包。Bun 的策略是"提供选项"——内置 + npm 包都支持，让用户选。

03 应用场景

场景 A · 用 bun:ffi 调系统 OpenSSL（不写 native module）

你有个加密 / 哈希 / TLS 相关需求，想直接调系统 OpenSSL（不依赖 npm node-forge 这种纯 JS 实现，也不想写 N-API native module）。Bun FFI 让你 5 分钟搞定：dlopen('/usr/lib/libssl.dylib', { ... }) 声明你需要的函数（如 SSL_library_init / SHA256 等），直接调。比 N-API 路径快十倍——不用写 C++ + binding.gyp + 编译。但要注意：(1) 系统库版本差异（OpenSSL 1.x vs 3.x ABI 不同）—— 部署到不同 OS 要分别测试；(2) 错误处理薄 —— 调错参数 / 返回值 segfault 整个进程；(3) 二进制依赖 —— 用户要装 OpenSSL（一般都装了，但 musl Linux 容器要确认）。这种场景下 FFI 适合内部工具 / 临时脚本 / 快速原型；分发给外部用户的库还是 N-API 稳。

场景 B · 用 Macros 把 SVG 图标 inline 进 React 组件

前端项目有 100+ SVG 图标，每个图标都要 inline 进 React 组件（<svg>...</svg> 字面量）。传统方法：用 babel-plugin-inline-svg 或 vite-plugin-svgr 在 build 时转换。Bun Macros 让你不用 plugin：写一个 macro readSvg(name) 在 build 时读文件返回字符串，import with macro attribute → build 时替换为字面量。具体做法：(1) 写 icons/macro.ts 函数 readSvg；(2) 组件文件 import { readSvg } from './icons/macro.ts' with { type: 'macro' }；(3) const home = readSvg('home')；(4) bun build 时 readSvg 在编译期跑，产物里直接是 SVG 字面量。这种用法的好处：build 简单（不需要 plugin）+ 产物小（没有运行时读文件代码）。但限制：只能 Bun build 用，Vite / Next.js 不认。

场景 C · 评估「我们项目能不能用 bun:sqlite 替代 better-sqlite3」

项目用 better-sqlite3 跑 SQLite 5 年了——稳定但 Docker 部署一直踩 node-gyp 编译坑。考虑切 bun:sqlite。判据：(1) API 兼容—— bun:sqlite API 80%+ 学习自 better-sqlite3，迁移工作量小；(2) 性能—— 实测 benchmark 持平（都是直接调 SQLite C 库）；(3) SQLite extension—— 用了 sqlite3-extensions（如 spatialite / ICU）的项目，bun:sqlite 是否支持 extension 加载（最新版支持）；(4) BLOB / 复杂类型—— 用了 Custom function / aggregate function 的项目，bun:sqlite 实现可能不完整；(5) Vendor lock-in—— 切了 bun:sqlite 后切回 Node 不简单（API 不完全一致）。综合后大部分简单 CRUD SQLite 项目可以切；高级 SQLite 用法（custom function / extension）暂不切。但仅这一个改动就能消除 Docker 部署的 node-gyp 痛点，ROI 巨大。

场景 D · 给做技术决策的架构师：「内置 DB 客户端」是 vendor lock-in 吗

架构师担心：用了 Bun.sql / bun:sqlite 后切回 Node 要改代码——这是 vendor lock-in 吗？技术上是有锁定但可控。规避策略：(1) 抽象层—— 把 Bun.sql 调用包在 db/index.ts 文件里，业务代码只依赖你的抽象接口；切换时只改 db/index.ts，业务不动。(2) 用 Drizzle / Prisma 等 ORM—— 这些 ORM 用 driver adapter pattern，能从 Bun.sql 切到 pg.js 改一行 driver。(3) 评估实际切换概率—— 如果你 5 年内不会切回 Node，lock-in 不是真痛点；如果可能切，写抽象层。综合：用 Bun.sql 拿性能 + 简单部署的红利；写 thin abstraction 层避免业务代码硬绑；保持优化路径开放。这是对待"内置一等公民"哲学的成熟态度——用红利但不裸奔。

04 典型代码

javascript 
// FFI 完整示例：调 Rust crate 编译的 cdylib
import { dlopen, FFIType, suffix } from 'bun:ffi';

// suffix 是 'so' / 'dylib' / 'dll' 自动选
const path = `./libmycrate.${suffix}`;

const lib = dlopen(path, {
  add: {
    args: [FFIType.i32, FFIType.i32],
    returns: FFIType.i32
  },
  hash_string: {
    args: [FFIType.cstring, FFIType.u64],
    returns: FFIType.cstring
  },
  free_string: {
    args: [FFIType.ptr],
    returns: FFIType.void
  }
});

// 调用
console.log(lib.symbols.add(1, 2));  // 3

// 字符串调用 + 手动释放（Rust 用 Box::into_raw 返回的）
const ptr = lib.symbols.hash_string('hello', 5);
const result = ptr.toString();  // CString → JS string
lib.symbols.free_string(ptr);   // 必须释放

05 常见陷阱

期待 FFI 性能持平 N-API

FFI 每次调用要 marshal 参数（JS 类型 → C 类型），这部分开销 N-API 编译时已经 inline 了。Hot path 上（每秒百万次调用）FFI 比 N-API 慢 2-5x。如果你的需求是"调用频率不高"（每秒几千次以下）—— FFI 完全够。如果是 hot path（如游戏 / 实时数据流） —— 用 N-API。判断标准：profile 你的代码看 FFI 调用占多少 CPU。占 > 10% 考虑切 N-API；占 < 1% 不用关心。

Macros 写「依赖运行时环境」的代码

Macros 在 build 时跑——不能依赖运行时环境（process.env / 网络请求 / DB）。常见错误：(1) macro 函数 read process.env.API_KEY —— 拿到的是 build 机器的 env，不是 runtime；(2) macro 函数 fetch some-url —— build 时跑可能没网或拿到旧数据；(3) macro 函数 now() —— 拿到的是 build 时间，不是 runtime 时间。规避：明确分清"build time"和"runtime"——build time 适合做静态资源 / 编译期常量；runtime 数据要用普通 import。

用 bun:sqlite 但不开 WAL mode

SQLite 默认是 rollback journal mode，并发写性能差。生产场景一定开 WAL mode：db.exec('PRAGMA journal_mode = WAL')。WAL 让多读单写并发性能好 5-10x。Bun docs 推荐 WAL，但默认不开（向后兼容）—— 你要在初始化时显式开启。

假设 Bun.redis 完全替代 ioredis

Bun.redis 在 1.2 仍是实验性，覆盖基础命令（GET / SET / DEL / EXPIRE / Pipeline）但Cluster mode / Sentinel 自动 failover / Streams API（XADD / XREADGROUP）/ Lua scripting支持还不完整。生产 Redis 重度用户（特别是用 Cluster / Streams）暂时仍 ioredis。简单 cache 场景可以试 Bun.redis；复杂场景等 Bun.redis 完善（预计 Bun 1.3+）。
