P5 · 高级与生产决策

FFI / Macros / 内置 SQLite + Postgres + Redis（非入门必须，但能在某些「超 Node 范畴」场景胜出）+ 何时切 Bun 何时仍用 Node 决策树（5 维度 + 7 典型场景 verdict + 风险评估）。把前 4 阶段所有内容收束到 production 决策。

章节数 2 主要来源 Bun FFI 文档 状态 ⏳ 待撰写

本阶段覆盖

  • bun:ffi —— dlopen 风格调 .so/.dylib/.dll；声明类型自动 marshal；写法比 N-API 简单，调用比 N-API 慢
  • 使用场景 —— 调系统库（OpenSSL/sqlite3/libcurl）/ 调 Rust/Zig/C 写的高性能函数 / 快速试验 native 集成
  • Macros —— 编译期代码执行；with { type: "macro" } import attribute；JS 表达力有限——主要 build 时嵌入数据
  • bun:sqlite —— 内置 SQLite；与 better-sqlite3 同等速度但不需 native 编译
  • Bun.sql / Bun.redis —— Postgres / Redis 一等公民客户端；连接池内置
  • HTML imports —— Bun 1.2+ 把 HTML 当模块；集成 Bun.serve fullstack 路径
  • FFI vs N-API 取舍 —— FFI 灵活不需编译 / N-API 兼容 Node 生态
  • 决策维度 1 · 启动速度敏感性 —— Lambda/Serverless/CLI ✅ Bun 决定性 / 长跑服务 ⚠️ 差异不重要
  • 决策维度 2 · npm 生态依赖 —— native module 多 ⚠️ 验证 N-API 兼容；纯 JS 依赖 ✅ 风险低
  • 决策维度 3 · 工具链一体化收益 —— 单一服务 / 库 ⚠️ 收益小；大 monorepo ✅ 收益最大
  • 决策维度 4 · 团队工程化能力 —— Bun 兼容性边界需要排查；保守团队选 Node
  • 决策维度 5 · 部署目标 —— Cloudflare Workers / Vercel Edge ❌ 仍 V8 Isolate；Lambda / Docker ✅
  • 典型场景 verdict —— CI 加速 ✅ / monorepo dev ✅ / Web API（Docker）✅ / Lambda ✅ / Edge ❌ / 遗留 Node 项目 ⚠️ 渐进切换 / npm 库 ⚠️ Bun dev / tsup 发布
  • 风险评估 —— bus factor / bug 率 / Oven 商业化进展决定可持续性
