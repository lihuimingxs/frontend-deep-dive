Bun 深度学习 · JavaScriptCore + Zig 一体化 JS 平台

基于 bun.com/docs + 官方 release blog + 源码 / Issues 整理。把 Bun 当作同一产品的 4 个面讲：JSC + Zig 怎么取代 V8 + libuv（启动快 + 系统调用直通）、内置一等公民 API 怎么取代 Node 标准库 + npm 包（Bun.serve / Bun.file / Bun.sql）、install / run / build / test 一体化工具链（共享 parser 的红利）、何时切 Bun / 何时仍用 Node 的生产决策树。是 Node.js Runtime 主题之上的"同代人对照"延伸。

受众 已学完 Node 的工程师 来源 bun.com / WebKit JSC / oven-sh GitHub 章节 12 章 / 5 阶段 状态 ✅ 12/12 章完成

分阶段目录

共 5 个阶段、12 章，按"产品决策轴"展开：Bun 是谁 → JSC + Zig 内核 → "内置一等公民"哲学 → 一体化工具链 → 高级特性与生产决策。建议顺序读；已用 Bun 的工程师可直接跳 P3.2 + P5.2 看兼容性边界 + 生产决策；只想换 pnpm 的看 P4.1 + P5.2。

P3 · "内置一等公民" 哲学

⏳ 2 章规划中

P4 · 一体化工具链

⏳ 4 章规划中

编写约定

每章遵循统一的 6 段结构：速览 → 底层逻辑 → 应用场景 → 典型代码 → 常见陷阱 → 延伸阅读。Bun 主题相比 Node 有以下额外约定：

  • 「短重述 + 链回」 —— 当一个机制在 V8 / Node / ECMA 主题已详述时（如 V8 编译流水线 / libuv / Streams / Module Records），本主题用 1-3 句话点到本质，紧跟一句"详细的 X 请参考 Y"
  • 版本号必标 —— Bun 迭代极快，每个特性必标"Bun X.Y 引入 / Z 稳定"。0.x 时代 / 1.0+ 时代 / 1.1 Windows 一级支持 / 1.2+ 是关键分界
  • bun.com/docs 一手优先，但兼容性以 GitHub Issues 为准 —— Bun 文档美观但简略，compat:nodejs label 下的 issue 列表是 Node API 兼容进度的最权威记录
  • 性能数字要可复现 —— 写作时不引用未注明硬件 / 负载的 benchmark；引用时必带 source + 复现说明
  • 「production realities」框架 —— 用户已用 Bun 替代 pnpm，写作角度是"生产工程师视角的取舍判据"，不是"Bun 教程"

与既有主题的关系

Bun 主题大量调用 V8 / Node / ECMA 主题的概念，必须严格"链回"，否则会重复早期主题。常见对应：

  • P2.1 JSC 流水线 → V8 P2 编译流水线（V8 视角对照）+ P1.2 引擎横评
  • P2.1 JSC Structure → V8 P3.1 Hidden Class（同等机制）
  • P2.2 Zig 系统层 → Node P2.1 libuv 设计（被 uSockets 替代）
  • P3.1 Bun.serve / Bun.file → Node P4.2 HTTP server + P4.3 fs/Buffer
  • P3.2 Node 兼容层 → Node P3.1 CJS/ESM + P5.1 Worker + P5.2 N-API
  • P4.3 bun build --compile → Node P6.1 现代 Node 工程 · SEA
  • P4.4 bun test → Node P6.1 node:test
  • P5.1 bun:ffi → Node P5.2 N-API（不同的 native 集成路径）
