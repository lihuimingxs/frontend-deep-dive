Bun.serve / Bun.file / Bun.sql：内置一等公民

把 Bun 的"内置 API 套"系统讲一遍——它们的设计共性是什么、为什么比 Node + npm 等价物快。关键洞察：内置不是为了"少装包"，是为了"零拷贝路径"——Bun.serve(req).write(Bun.file()) 走 sendfile 系统调用，绕过用户态 buffer，这种优化第三方 npm 包永远做不到。这一章讲透 Bun.serve / Bun.file / Bun.write / Bun.password / Bun.glob / Bun.spawn / Bun.sql / HTTP routes 一整套内置 API 的设计哲学 + 性能根源。

阅读时长 ~30 分钟 前置 Node P4.2 HTTP + Node P4.3 fs/Buffer 主要来源 bun.com/docs/api · Bun 1.2 release notes

01 速览

  1. "内置一等公民"不是为了少装包，是为了零拷贝路径——Bun.serve + Bun.file 组合走 sendfile / splice / mmap 系统调用，文件直接从内核页缓存到 socket，绕过用户态 buffer。这种优化第三方 npm 包做不到，因为它们必须先把数据 copy 到 V8 ArrayBuffer。
  2. API 设计哲学：与 Web 标准对齐——Bun.serve 接受 fetch-style handler（{ fetch(req) { return Response } }），Bun.file 可作为 fetch body，Bun.password 用 Web Crypto 风格 Promise——这让 Bun 代码可以部分跑 Cloudflare Workers / Vercel Edge，反之亦然。
  3. 性能差异在「热路径」最明显——简单 hello world server: Bun.serve 比 http.createServer 快 2-3x；文件分发: Bun.write 比 Node Streams pipe 快 5-10x；密码哈希: Bun.password 比 npm bcrypt 快 1.5-2x。差距来自 Zig 实现 + 系统调用直通。

02 底层逻辑

"一等公民" vs "npm 包" 的设计哲学差异

Node 的设计是"runtime 提供最小标准库 + 一切能装包的都装包"——HTTP / fs / crypto / stream / net 是 runtime 自带，但任何"业务级"功能（密码哈希 / glob / dotenv / postgres / redis）都靠 npm 包。这是 2009 年 Ryan Dahl 立项时的设计哲学：runtime 应该 minimal，生态由社区扩展。

Bun 的设计完全反向——"任何高频需求都内置成一等公民 API"。Bun.password.hash() 替代 bcrypt 包；Bun.glob() 替代 glob/fast-glob 包；Bun.env 替代 dotenv 包；Bun.sql 替代 pg/postgres.js 包；Bun.spawn 替代 execa 包。Jarred 在 Bun 1.0 launch 博客里把这套定位叫"batteries included, but better"——直接对标 Node 的 minimal core 哲学。

这两种哲学各自的代价：

Node minimal core：runtime 体积小（Node 二进制 ~80MB）、社区生态扩张快（任何人能发包）、但每个 npm 包都要走 V8 binding 层 + 不能做系统调用直通优化，性能受限；用户要装 + 维护一堆小依赖（典型 Node 项目 1000+ 间接依赖）。

Bun batteries included：runtime 体积大（Bun 二进制 ~70MB，未来会涨）、内置 API 性能极致（Zig 直接热路径 + 系统调用直通）、用户依赖简单（一个 Bun 二进制带很多功能）；但不够灵活——如果 Bun 内置 API 的某个行为不符合你的需求，要 fallback 到 npm 包，体验断层。

Jarred 的判断是"性能优势 > 灵活性损失"——大部分用户用的是 80% 通用场景，不需要极端定制。这套哲学在 2026 年看是对的——Bun 1.2 内置的 API 集合已经覆盖 90%+ 业务场景的高频需求。

Bun.serve()：fetch-style handler + uSockets 后端

Bun.serve() 是 Bun 的 HTTP server 一等公民 API。它和 Node 的 http.createServer() 形态完全不同：

javascript 
// Node: http.createServer 接受 (req, res) 回调
import http from 'node:http';
http.createServer((req, res) => {
  res.writeHead(200, { 'content-type': 'text/plain' });
  res.end('hello');
}).listen(3000);

// Bun: Bun.serve 接受 fetch-style handler
Bun.serve({
  port: 3000,
  fetch(req) {
    return new Response('hello', {
      headers: { 'content-type': 'text/plain' }
    });
  }
});

差异看起来只是 API 形态——(req, res) => mutate res 改成 (req) => new Response()——但底层影响深远：

(1) 与 Web 标准对齐——Request / Response / Headers / Blob 都是 Web 标准类型，浏览器 fetch API + Service Worker 用一样。这意味着同一段 handler 函数可以在 Bun + Cloudflare Workers + Vercel Edge + Deno 上跑——只换 server 启动代码就行。这是 Bun 不锁定生态的关键。

(2) 函数式而非命令式——返回 Response 而非 mutate res。这让中间件 / 路由库设计更简单（compose Response 比 compose mutation 容易）；也让单元测试更直接（直接 call handler 拿 Response 验证，不用模拟 res 对象）。

(3) 异步原生——fetch 可以是 async，返回 Promise<Response>。Bun 内部自动 await，不需要用户自己处理异步流。Node 的 (req, res) 回调对异步处理不友好（async 函数里没法 return，要 explicit res.end）。

(4) 后端是 uSockets + Zig——uSockets 直接跑 io_uring（Linux）/ kqueue（macOS）/ IOCP（Windows），不走 libuv 抽象层。Zig 写的 HTTP parser 用 comptime 生成专用 header 比较代码（详见 P2.2 Zig 系统层）。这些组合让 Bun.serve 在 hello-world benchmark 上比 http.createServer 快 2-3x。

Bun 1.2+ 还在 Bun.serve() 上加了 HTTP routes（路由内置）：

javascript 
// Bun 1.2+ 内置路由
Bun.serve({
  port: 3000,
  routes: {
    '/api/users/:id': (req) => {
      const { id } = req.params;
      return Response.json({ id });
    },
    '/api/users': {
      GET: () => Response.json({ users: [] }),
      POST: async (req) => {
        const body = await req.json();
        return Response.json({ created: body }, { status: 201 });
      }
    }
  },
  fetch(req) {
    return new Response('Not Found', { status: 404 });
  }
});

这套路由 API 学习自 Hono / itty-router，但不需要装包——内置的好处是路由匹配走 Bun 内部优化的 trie + 编译期 specialize，性能比社区路由库更好。简单业务场景可以彻底不用 Express / Fastify / Hono——Bun.serve 一行搞定。

Bun.file / Bun.write：零拷贝路径的核心

Bun.file(path) 不是立即读文件——它返回一个 BunFile 对象，lazy 表示一个文件资源。文件内容在你显式访问（.text() / .json() / .arrayBuffer()）或传给消费者（fetch body / Bun.write）时才读。

javascript 
// Bun.file 创建 lazy 引用，不读文件
const file = Bun.file('./data.json');

// 显式读：拿到内容
const text = await file.text();
const json = await file.json();
const buffer = await file.arrayBuffer();

// 传给消费者：可能走零拷贝路径
const res = new Response(file);  // serve 时 sendfile
await Bun.write('./copy.json', file);  // copy 时 splice
await fetch('https://api.example.com/upload', {
  method: 'POST',
  body: file  // upload 时 sendfile 到 socket
});

"零拷贝路径"指的是"文件数据从内核页缓存直接送到 socket / 目标文件，不经过用户态 buffer"。Linux 上对应的系统调用是 sendfile()（文件 → socket）和 splice()（文件 → 文件 / pipe）；macOS 是 sendfile()；Windows 是 TransmitFile()。

对照 Node 的处理流程：你 fs.createReadStream(file).pipe(res) 时——(1) Node 用 libuv 的 worker pool 读文件到 ArrayBuffer（一次 user-space copy）→ (2) ArrayBuffer 传给 Stream → (3) Stream 写到 socket（再一次 user-space copy）。每个 chunk 都要走两次 user-space copy + Stream 抽象层的开销。

对照 Bun 的处理流程：你 new Response(Bun.file(path)) + Bun.serve 把 Response 写到 socket 时——Bun 看到 Response.body 是 BunFile，直接调 sendfile，内核态完成"文件 → socket"的传输。完全没有 user-space copy。

性能差距在"大文件分发"场景巨大——传 1MB 文件到 socket：

  • Node fs.createReadStream + pipe：~5-10ms（取决于硬件），CPU 主要花在 user-space copy
  • Bun.serve + Bun.file：~0.5-1ms，CPU 几乎为 0（内核态完成）

差距是 5-10x。对静态文件分发 / CDN 边缘 / 视频流服务，Bun 的优势可能让单台服务器吞吐量翻几倍。

Bun.write(dest, src) 是 Bun.file() 的写入对应物——dest 可以是 path / BunFile，src 可以是 string / ArrayBuffer / Blob / Response / BunFile。Bun 自动选最优路径：BunFile → BunFile 走 splice（文件 → 文件零拷贝），Response → BunFile 走 sendfile（如果 Response 来自 fetch），其他场景走标准 write。

Bun.password：内置密码哈希

密码哈希在 Node 生态是个老问题——常用包是 bcrypt（C++ native 编译）或 argon2（同 native）。这两个都需要 node-gyp 编译，部署到 Docker 经常踩坑（缺 Python / 缺 g++）；版本与 Node ABI 绑定，Node 升级要重新编译。

Bun 把密码哈希做成一等公民：

javascript 
// Bun.password 默认用 argon2id（推荐算法）
const hash = await Bun.password.hash('user-password');
// $argon2id$v=19$m=65536,t=2,p=1$...

const ok = await Bun.password.verify('user-password', hash);
// true

// 也支持 bcrypt（兼容遗留 hash）
const bcryptHash = await Bun.password.hash('pwd', { algorithm: 'bcrypt', cost: 12 });

底层是 Bun 内嵌的 argon2 / bcrypt C 库，Zig glue 直接调，不需要 node-gyp 编译 + 不依赖系统。性能比 npm bcrypt 快 1.5-2x（Zig glue 比 N-API 调用开销小）；部署简单（Bun 二进制带）。

这是"内置一等公民"哲学的最纯粹案例——一个普遍需求，npm 解决方案有切实痛点（编译 + 部署），Bun 直接做成内置。其他类似的内置 API：Bun.glob()（替代 glob/fast-glob 包）、Bun.env（替代 dotenv 包）、Bun.spawn()（更人体工学的 child_process）。每一个都对应一个 npm 老痛点。

Bun.sql（Bun 1.2+）：Postgres 客户端一等公民

Bun.sql 是 Bun 1.2 引入的 Postgres 客户端，是"内置一等公民"哲学的最强表达——把数据库客户端这种"传统上必须装包"的东西做成 runtime 自带。

javascript 
// 1.2+ 内置 Postgres 客户端
import { sql } from 'bun:sql';

// 配置（默认从环境变量 DATABASE_URL 读）
const db = sql({ url: 'postgres://user:pwd@localhost/mydb' });

// 简单查询
const users = await db`SELECT * FROM users WHERE active = ${true}`;
// 自动 prepared statement + 自动 SQL injection 防护

// 事务
await db.begin(async (tx) => {
  await tx`INSERT INTO orders (user_id) VALUES (${userId})`;
  await tx`UPDATE inventory SET count = count - 1 WHERE id = ${itemId}`;
});

// 流式 cursor（大结果集不爆内存）
for await (const row of db`SELECT * FROM big_table`.cursor(1000)) {
  process(row);
}

API 形态学习自 postgres.js（社区 ESM-friendly Postgres 客户端），但底层是 Bun 自家 Zig 实现：

(1) 二进制协议直通——Postgres 协议是文本 / 二进制混合协议，Bun 用 Zig 直接 parse，不走 npm pg 的 JS 解析层。

(2) Connection pooling 内置——不需要手动 setup pool；首次查询自动建连接 + 复用。

(3) Prepared statements 自动——模板字面量自动转 prepared statement，参数走 protocol 而非 SQL 文本（防 injection）。

(4) Row → JS 对象映射用 comptime——SQL 列类型到 JS 类型的映射在 Zig 编译期 specialize，运行时无反射开销。

性能比 npm pg 快约 50%-100%（具体看负载）；连接管理更简单。但有前提：(1) 只支持 Postgres（不是 MySQL / MongoDB）；(2) ORM 高级功能（Prisma / Drizzle 那种）仍要装包——Bun.sql 是底层客户端不是 ORM；(3) 一些 Postgres 高级特性（如 listen/notify、复制流）支持还不完整，可能需要 fallback 到 postgres.js。

Bun 1.2 还引入了实验性的 Bun.redis（Redis 客户端）和 Bun.S3（S3 风格对象存储客户端）——同样的"一等公民"哲学。覆盖范围还在扩展中。

HTTP imports（Bun 1.2+）：fullstack 路径

Bun 1.2 还引入了一个有争议但很有意思的特性：HTML imports。

javascript 
// Bun 1.2+ 把 HTML 当成可 import 的模块
import index from './index.html';
import dashboard from './dashboard.html';

Bun.serve({
  routes: {
    '/': index,
    '/dashboard': dashboard,
    '/api/data': () => Response.json({ data: 42 })
  }
});
// HTML 内 <script src="./client.tsx"> 自动被 Bun build bundle
// 资源（CSS / 图片）自动被 Bun 处理

这是 Bun 把"前端 build"和"后端 serve"合并为一个 runtime 任务的尝试——Bun.serve 能直接服务 HTML + 自动 bundle 引用的 TSX / CSS / 资源，不需要 Vite / Next.js / Webpack。

这个特性的定位：(1) 简单 fullstack 项目（个人博客 / dashboard / 内部工具）可以零配置起；(2) Bun 1.2 时仍是实验性，复杂场景（HMR / SSR / RSC）不如 Next.js / Remix 成熟；(3) 与 Bun.serve + bun build 共享底层，性能比传统 Vite + Express 组合好。

从产品定位看，这是 Bun 想抢 Next.js / Remix 的市场份额——但目前社区生态还很小，主要适合 prototype + 内部工具。生产 fullstack 仍然 Next.js 主流。

性能差异的真正来源

把 Bun 内置 API 比 npm 等价物快 2-10x 的原因归纳成 4 类：

(1) 不走 V8/JSC binding 层——npm native 包通过 N-API 调 C++，每次调用有 marshal overhead；Bun 内置 API 直接 Zig 实现，无 binding 层。这是基本机制。

(2) 系统调用直通（零拷贝路径）——Bun.serve + Bun.file 走 sendfile / splice / mmap，绕过用户态 buffer。npm 包做不到，因为它们必须把数据先放进 V8 ArrayBuffer。

(3) Zig comptime specialization——hot path 上的 parser / 字段映射在编译期生成专用代码（详见 P2.2），运行时无反射 / 无通用解析开销。

(4) 共享 Bun.file / Response 等抽象——Bun.serve 接受 Bun.file 作为 body 时知道走 sendfile；Bun.write 接受 Response 时知道走 splice；这些"内部 API 之间互相理解"让组合优化成为可能。npm 包之间互相不认识，做不到这种组合优化。

这 4 类合在一起，大部分内置 API 比 npm 等价物快 1.5-3x，少数（文件 + socket 组合）快 5-10x。这是 Bun 性能的核心来源。

03 应用场景

场景 A · 用 Bun.serve + Bun.file 做静态文件服务器

团队需要做一个静态文件服务（如 CDN 边缘 / 内部资源服务器 / 文件分享平台）—— Bun.serve + Bun.file 的零拷贝路径让这种场景拿到极致性能。10 行代码搞定：Bun.serve({ fetch: (req) => new Response(Bun.file('./public' + new URL(req.url).pathname)) })。性能比 nginx 略低（nginx 是 C 写的专用文件服务器），但比 Node + serve-static 快 5-10x，写起来更灵活（可以加 auth / log / 重写规则）。这种场景下 Bun 真正发挥"内置 API 组合"的优势——你写 npm 包等价物（serve-static + connect 中间件 + body-parser）累积下来 1000+ 行依赖，Bun 一行 lambda 解决。生产建议：CDN 边缘场景仍用 nginx / Cloudflare（更专业）；中型应用 Bun.serve 是好选择。

场景 B · 用 Bun.password 替代 bcrypt 包

项目用了 bcrypt 做密码哈希——典型痛点：Docker 部署 base image 要装 Python + g++（bcrypt native 编译依赖）；CI 的 npm install 很慢（编译 bcrypt）；Node 升级要 rebuild。切到 Bun.password 之后这些都消失：const hash = await Bun.password.hash(pwd)，不需要 node-gyp，不需要 system 依赖。Docker base image 可以用最小的 Bun official image，CI 速度提升明显。bcrypt 哈希格式向后兼容（Bun.password 默认 argon2id 但能 verify bcrypt 老 hash），迁移路径清晰：(1) 新密码用 Bun.password.hash 默认 argon2id；(2) 用户登录时如果 verify 老 bcrypt hash 成功，re-hash 成 argon2id。这种场景下"内置 API"的部署 + 维护体验差距远大于性能差距——这是把"用户痛点"作为产品决策的好例子。

场景 C · 评估「我们的 Postgres 项目能切 Bun.sql 吗」

团队用 npm pg 或 postgres.js 包做 Postgres 客户端，考虑切 Bun.sql。判据：(1) 查询复杂度—— 简单 SELECT / INSERT / UPDATE / 事务 100% 兼容；(2) 是否用 ORM（Prisma / Drizzle）—— ORM 不能直接切 Bun.sql（ORM 自己管底层连接），但 Prisma / Drizzle 有 driver adapter 可以用 Bun.sql 做底层；(3) 是否用 listen/notify—— Bun 1.2 仍在完善这块；密集用 LISTEN/NOTIFY 做 pub/sub 的项目暂时不切；(4) 是否用 streaming replication—— Bun.sql 还不支持，要 fallback；(5) 性能 sensitive—— 用 Bun.sql benchmark 你的 hot 查询，通常快 50-100%。综合后大部分纯 CRUD 项目可以切 Bun.sql；ORM 项目用 driver adapter；高级 Postgres 特性继续用 postgres.js。

场景 D · 给写库的人：尽量用 Web 标准 API 兼容多 runtime

维护一个被广泛使用的 npm 库（如 SDK、middleware、helper），设计 API 时优先用 Web 标准 / Bun 兼容形态能让你的库自动支持 Bun + Node + Deno + Edge runtime。具体做法：(1) HTTP handler 用 fetch-style（接受 Request 返 Response），不用 Node-specific (req, res)；(2) 文件读用 fetch(file://) 或 fs.promises.readFile，避免 Node-specific 的 readFileSync；(3) 密码 / 加密用 Web Crypto API（globalThis.crypto.subtle），避免 Node 的 crypto 模块；(4) 流式数据用 ReadableStream 而非 Node Stream。这种"以 Bun 为基准设计"的 API 在 Node 上也能跑（Node 22+ 全部支持），还能跑 Cloudflare Workers / Vercel Edge。从你库用户的角度看，他切任何 runtime 都不会被卡——这是显著的竞争优势。

04 典型代码

展示 Bun 内置 API 的"组合用法"——单个 API 简单，组合起来才看出威力。

javascript 
// 完整的小型 web app — 一个文件，零依赖
import { sql } from 'bun:sql';

const db = sql();  // 从 DATABASE_URL 读配置

Bun.serve({
  port: 3000,
  routes: {
    // 静态文件 — 走 sendfile 零拷贝
    '/public/:file': (req) => {
      return new Response(Bun.file('./public/' + req.params.file));
    },
    // API — 用 Bun.sql 做数据库
    '/api/users': {
      GET: async () => {
        const users = await db`SELECT id, name FROM users`;
        return Response.json(users);
      },
      POST: async (req) => {
        const { name, password } = await req.json();
        const hash = await Bun.password.hash(password);  // 内置哈希
        const [user] = await db`
          INSERT INTO users (name, password_hash)
          VALUES (${name}, ${hash})
          RETURNING id, name
        `;
        return Response.json(user, { status: 201 });
      }
    },
    // 文件上传 — 接收的 Request body 直接 write 到文件
    '/api/upload': {
      POST: async (req) => {
        const filename = `./uploads/${crypto.randomUUID()}`;
        await Bun.write(filename, req);  // body 流直接写文件
        return Response.json({ ok: true, file: filename });
      }
    }
  },
  fetch(req) {
    return new Response('Not Found', { status: 404 });
  }
});

// 这一段代码包含：HTTP server / 路由 / 静态文件 / 数据库 / 密码哈希 / 文件上传
// Node 等价物需要：express + serve-static + pg + bcrypt + multer
// 6 个 npm 包 + 配置 + 调试。Bun 一份代码搞定，性能还更好。
javascript 
// 零拷贝路径的实证：传文件到 socket
const file = Bun.file('./big-video.mp4');  // 100MB 文件

// ✅ 零拷贝（Bun 内部走 sendfile）
return new Response(file);

// ❌ 显式 buffer（用户态 copy 一次）
return new Response(await file.arrayBuffer());

// 实测差距：100MB 文件分发
// 零拷贝路径：~10ms，CPU ~0%
// 显式 buffer：~80ms，CPU 占用明显
// 大文件 / 高并发场景下差距可能让一台机器吞吐翻 5-10x

05 常见陷阱

Bun.serve 用 Node 风格 (req, res) 写法

从 Node 切 Bun 的人有时把习惯带进来，写 Bun.serve({ fetch(req, res) { res.end(...) } })——这不工作。Bun.serve 的 handler 只接受 (req) 一个参数，必须 return new Response(...)。res 风格是 Node http 模块的，Bun 没有对应物。错误信息可能是 "res is undefined"——但根因是 API 形态不同。规避：从一开始就按 fetch-style 写，不要 mix 两种风格。如果一定要兼容 Node 风格代码，用 Bun.serve 包装层（或用 node:http 兼容层，Bun 提供）。

Bun.file().text() / .json() 不是同步方法

Bun.file(path) 看起来"立即返回"，所以有人写 const text = Bun.file(p).text() 期望同步读——错的。.text() / .json() 都是异步的，返回 Promise。Bun 这块设计是"lazy + Promise"，没有同步读 API（避免 sync I/O 阻塞 event loop）。如果你需要同步（如启动时读配置），用 fs.readFileSync（Node 兼容层支持）。常见错误：const data = Bun.file(p).text()，data 实际是 Promise 不是 string；后续 data.includes(...) 报错。规避：always await。

期待 Bun.sql 兼容所有 Postgres 高级特性

Bun.sql 1.2 覆盖大部分常用查询场景，但不是 postgres.js / node-postgres 的完美替代。已知不完全支持的：(1) LISTEN/NOTIFY pub/sub（部分支持但有 corner case）；(2) Streaming replication（不支持）；(3) PostGIS / 自定义类型（type cast 可能不完整）；(4) prepared statement 预编译 / pipeline mode（实现细节差异）。如果你重度用这些，先在 dev 环境验证再切。规避：写抽象层包装 Bun.sql + postgres.js，根据 Bun.sql 是否支持 fallback。但大部分团队不需要——简单 CRUD + 事务用 Bun.sql 完全够。

假设「内置 API 性能一定比 npm 包快」

"内置 = 快"是 Bun 的市场口号，但不是所有内置 API 都比 npm 包快——有些场景 npm 包优化得更深。比如：(1) Bun.glob() 在简单 pattern 下比 fast-glob 快，但复杂 pattern（多 extglob / 大目录）可能慢；(2) Bun.spawn 比 child_process 略快，但 execa 包提供更人体工学的 API（promise / streaming output / killSignal handling）；(3) Bun.write 比 fs.writeFile 快，但小文件场景差距可忽略。判断 Bun 内置 vs npm 包：如果该 API 是 hot path（每次请求调），内置通常更值得；如果是 setup 期一次性调，性能差距不重要，选用得舒服的就行。不要为了"用 Bun 内置而 Bun 内置"。
