Node 兼容层：边界与 fallback

Bun 的"Node 兼容"是它最大卖点，但兼容到哪一步是 production 决策的关键。这一章讲清 2026 年的兼容现状 + 何时仍需 fallback。关键洞察：兼容层不是"copy Node 实现"——Bun 用 Zig 重写了 fs/http/crypto，表面 API 一样、行为对齐、性能更好；但角落 case 仍可能差异，必须查 GitHub Issues 上的 compat:nodejs label 列表。

阅读时长 ~30 分钟 前置 P3.1 + Node 主题 P3 / P4 / P5 主要来源 github.com/oven-sh/bun/issues · bun.com/docs/runtime/nodejs-apis

01 速览

  1. Bun 不是「copy Node 实现」——是用 Zig 重写 fs / http / crypto / stream / events 等核心模块；表面 API 与 Node 一致，行为对齐到 Node 22 LTS。性能更好，但角落 case 可能微妙差异。
  2. 2026 年兼容矩阵：完全兼容 80% / 部分兼容 15% / 不兼容 5%——核心标准库（fs/http/crypto 等）完全兼容；高级特性（async_hooks / vm / inspector）部分兼容；v8 模块和 V8 内部 API 不兼容（Bun 用 JSC）。
  3. 判断「能不能切」的工具是 GitHub Issues 上的 compat:nodejs label——比官方文档更新更快，是兼容性的最权威进度表。production 决策前先 grep 你用的 npm 包 / Node API。

02 底层逻辑

「100% Node 兼容」是什么意思

Bun 在 marketing 上反复说"100% Node 兼容"——这个说法不严谨但抓住核心。准确的定位是"前 1000 名 npm 包能直接装直接跑"+ "Node 标准库 API 表面完全兼容、行为对齐到 Node 22 LTS"。但角落 case 仍可能差异。

这个定位的实现路径分三层：

(1) Node 标准库 API 表面——fs.readFile / http.createServer / crypto.createHash / stream.Readable 这些 API 在 Bun 里都存在，签名一致，import 路径也一致（node:fs / node:http）。这部分覆盖率 95%+。

(2) Node 标准库行为对齐——比如 fs.watch 在不同 OS 上的行为差异、stream.Readable 的 backpressure 时机、http 的 keep-alive 默认行为——Bun 都努力对齐 Node 实际行为（不是规范）。这部分覆盖率 85-90%——一些角落 case 仍有差异。

(3) npm 生态可装可跑——bun install express + import express from 'express' + express() 直接工作。这要求兼容 CJS、ESM、conditional exports、require.resolve 等模块系统机制，加上各种 npm 包能用到的 Node API。Top 1000 npm 包覆盖率 95%+。

这三层加起来构成"100% Node 兼容"。Bun 团队的工作量大头在 (2) 和 (3)——"行为对齐"和"npm 生态"才是真正的工程负担。

2026 年兼容矩阵：完全 / 部分 / 不兼容

把 Node 标准库分成三档：

兼容度 模块 备注
✅ 完全兼容 fs · fs/promises · path · url · crypto · events · buffer · util · os · process · querystring · string_decoder · http · https · http2 · net · tls · dns · zlib · stream · readline · child_process · cluster 核心 80% Node 用法
⚠️ 部分兼容 worker_threads（Bun 自家 Worker 实现）· vm（部分 API）· async_hooks（AsyncLocalStorage 稳定，hooks API 部分）· inspector（基础 debugger 支持）· perf_hooks · diagnostics_channel · module（一些内部 API） 常用功能 OK，高级 corner case 需验证
❌ 不兼容 / 不实现 v8（JSC 不是 V8）· repl（Bun 用自家 REPL）· 一些 unstable Node 内部 API · 实验性 Node 22 引入但 Bun 还没跟的 API 必须 fallback 或避开

这张表的几个值得放大的细节：

worker_threads 的"部分兼容"是API 表面完全兼容，底层实现不同。Bun 用自家 Worker 实现（基于 JSC 的 isolate-like 概念但不完全是 Isolate）。简单 Worker 用法（postMessage、structured clone、SharedArrayBuffer）100% 工作；复杂用法（如 worker 之间的 native handle 共享、特殊的 transferable types）可能有 corner case。详见 Node P5.1 的 Worker Threads 章节。

vm 模块的"部分兼容"——vm.runInNewContext / vm.runInThisContext 等基础 API 工作，但 vm.Script.cachedData（V8 specific bytecode caching）这种 V8 内部细节不能。需要用 vm 做 sandboxing 的项目要自己验证。

v8 模块完全不兼容——v8.serialize / v8.deserialize / v8.getHeapStatistics 等 V8 特有 API 在 Bun 上抛 ReferenceError。一些 npm 包（如旧版 structured-clone）会用 v8.serialize，这种包在 Bun 上不能用。规避：用 structuredClone()（Web 标准）替代。

N-API 兼容：让大部分 native module 能跑

npm 生态有相当一部分包用了 native module（C++ 代码 + N-API binding，详见 Node P5.2）——better-sqlite3 / bcrypt / sharp / node-canvas / node-gyp 编译的各种包都属于。Bun 兼容 Node 必须兼容 N-API，否则一大批 npm 包不能用。

Bun 的 N-API 兼容性是逐步建设的——0.x 时代基本不能跑 native module；Bun 1.0 (2023-09) 覆盖到大部分 N-API v8 函数；Bun 1.1+ 继续完善。2026 年状态：

(1) Top 50 native module 几乎全部能跑——better-sqlite3 / bcrypt / sharp / node-canvas / @prisma/client / @neondatabase/serverless 等主流包都验证过。

(2) 中长尾 native module 95%+ 能跑——一些用 N-API 老版本（v3-v6）的包可能有 corner case，但大部分能直接 work。

(3) 少数极端 case 不能跑——直接用 V8 C++ API（不通过 N-API）的包（少见但存在）；用 Node 内部未公开 API 的包；用某些操作系统特定 syscall 的包。这些占整个生态 < 5%。

判断你的 native module 能不能跑：(1) 在 GitHub Issues 上 grep 包名（compat:nodejs label 下）；(2) 直接 bun install + bun run 试，能跑就 OK；(3) 看包的 GitHub README 是否声明 Bun 兼容（越来越多包开始测 Bun）。

CommonJS / ESM 双系统：解析算法对齐

Node 模块系统有两套：CommonJS（require）和 ES Modules（import）。Node 16+ 完整支持双系统，但解析行为有大量 corner case（package.json type 字段、conditional exports、import paths 是否带扩展名等，详见 Node P3.1 CJS/ESM）。Bun 必须对齐这些行为，否则 npm 包加载会出问题。

Bun 的实现：

(1) CJS 解析算法 100% 对齐 Node——require() 走 Node 算法（向上查 node_modules、自动加 .js / .json 扩展名、index.js fallback）。这部分稳定，corner case 少。

(2) ESM 解析对齐 Node 行为——import 走 ECMA Module Records 规范，但 Node 在规范上叠了一些行为（如 conditional exports 的 key 优先级、imports 字段的内部解析）。Bun 跟着 Node 走。

(3) Conditional Exports 字段精确——"exports": { ".": { "import": "./esm.js", "require": "./cjs.js", "types": "./types.d.ts" } } 这种结构 Bun 完全理解。还能识别 "bun" 作为 condition key（如果你的库想给 Bun 用户提供专门入口）。

(4) Bun-only 扩展——Bun 允许 ESM 文件里直接 require()（Node 严格分离两者）。这是 Bun 的"扩展兼容"而非"严格兼容"——遗留 npm 包混用 ESM/CJS 的能在 Bun 上跑，Node 上可能不行。写库时不要依赖这种扩展，否则你的库在 Node 上不工作。

这套对齐让 Bun 上的"bun install + import express"完全无感工作——你不需要知道 express 是 CJS 还是 ESM、不需要管 conditional exports，Bun 自动处理。这是 Bun "100% 兼容" 的核心红利。

process.nextTick / setImmediate / Promise 微妙差异

Node 的事件循环 + microtask 调度有大量行为细节（详见 Node P2.2）：process.nextTick 的优先级、setImmediate vs setTimeout(0)、Promise.then microtask 的 drain 时机——这些在 Node 22 是稳定行为。

Bun 1.0 早期在这些行为上和 Node 微妙差异——process.nextTick 在嵌套场景下顺序错；setImmediate 与 setTimeout(0) 在 I/O 回调内的相对顺序不对。这些导致一些复杂异步代码在 Bun 上行为不一致。

Bun 1.1+ 这块基本对齐——大部分用户不会感知。但非常依赖事件循环精确顺序的代码（如自家写的 task scheduler、event-driven framework 内部、一些精密的 async middleware）可能踩 corner case。

规避：(1) 写代码不依赖具体 nextTick / Promise / setImmediate 的相对顺序；用 explicit await 或 Promise.all 表达依赖；(2) 复杂异步逻辑在 Bun + Node 都跑测试，验证行为一致；(3) 报 corner case 给 Bun 团队（Issue tracker），他们响应很快。

「Bun 行为更好但与 Node 不一致」的 trade-off

Bun 在某些地方有意行为与 Node 不同——因为 Node 的行为是"历史包袱"，Bun 团队认为更合理的实现可以做得更好。这些情况下 Bun 倾向用更合理的实现，但保留 Node 兼容选项。例子：

(1) process.env 自动加载 .env——Bun 启动时自动读 .env / .env.local。Node 不会（要装 dotenv）。这意味着 Bun 上的 process.env.FOO 行为与 Node 不一样——前者带自动加载。如果你想 disable，--no-env-file。

(2) ESM 里允许 require()——Bun 允许，Node 严格不允许。这让一些遗留代码在 Bun 上能跑，Node 上不能。

(3) TypeScript / JSX 直跑——Bun 直接 transpile 跑 .ts / .tsx；Node 22 strip-types 部分支持，但 JSX 不能。这是 ergonomic 差异。

(4) 默认 fetch timeout 行为——Bun fetch 默认无 timeout（与 Web fetch 一致）；Node 18+ 内置 fetch 也是。但 Bun 在某些 corner case 下处理 timeout 比 Node 更激进。

这些 trade-off 都不影响"100% 兼容"声明——Bun 仍能装跑 npm 包，但 Bun-only 行为给 Bun 用户额外便利。写库时要确认你没用 Bun-only 行为（否则你的库在 Node 上挂）；写应用时可以放心用 Bun 扩展，部署仍然 Bun。

判断「能不能切 Bun」的实操流程

实际项目评估能不能切 Bun，分这几步：

(1) bun install 试装——cd your-project && bun install。失败说明 lockfile / 包结构有 Bun 不能处理的 case；通常能装。

(2) 跑测试套件——bun run test（或你的测试命令）。看 native module 能不能加载、API 行为是否一致。失败的 case 列出来。

(3) 跑应用 smoke test——bun run start（或 dev / build），让应用跑起来访问主要功能。看是否报错或行为异常。

(4) 查 GitHub Issues——失败的 case 在 oven-sh/bun GitHub 搜，看是否 known issue / 有 workaround / 何时会修。compat:nodejs label 下能找到大部分 Node 兼容相关 Issue。

(5) 决策——失败的 case 全是已知小 issue 且不阻塞 → 可以切；失败影响关键功能 → 暂缓 + 等 Bun 修；某个 native module 不能跑 → 找替代或暂缓。

这套流程通常 1-2 天可以走完。大部分项目的结论是"可以切，少数 corner case 用 patch 或 workaround 解决"——这是 Bun production-ready 的实际状态。

03 应用场景

场景 A · 评估「公司 Express + Postgres + Redis 项目能切 Bun 吗」

典型 Node 后端项目栈：Express + node-postgres / Prisma + ioredis + bcrypt + jsonwebtoken + 大量自家业务代码。评估流程：(1) Express—— 100% 兼容，可以保留；或者考虑切 Bun.serve（性能 +50-200%）。(2) node-postgres / Prisma—— pg 包 100% 兼容；Prisma N-API 兼容，能跑（也可以用 Bun.sql 做底层加速）。(3) ioredis—— 兼容（也可以试 Bun.redis 实验性 API）。(4) bcrypt—— N-API 兼容，能跑（也可切 Bun.password）。(5) jsonwebtoken—— 纯 JS 包，100% 兼容。综合后这种栈切 Bun 几乎无 blocker，主要 ROI 来自启动加速 + bun install 提速 + bun test 提速。建议路径：先切 dev 环境（bun install + bun run dev + bun test），运行 1-2 周稳定后切 production。

场景 B · 排查「特定 npm 包在 Bun 上崩溃」

偶尔遇到某个包在 Bun 上 import 崩溃 / 行为异常。排查流程：(1) 找包的 GitHub repo——是否有 issue 提到 Bun？是否有 PR 修复？多数情况下是 known issue。(2) 找 oven-sh/bun——是否有 issue 跟踪这个包？search "包名 site:github.com/oven-sh/bun"。(3) 跑 minimal repro——把崩溃简化成 5-10 行代码 reproduce，方便提 issue。(4) 提 issue 给 Bun——附 minimal repro + 包版本 + Bun 版本。(5) workaround——暂时 fallback 到 Node（仅这一个服务）或找替代包。Bun 团队对兼容 issue 响应很快——典型 turnaround 几天到 2 周。这种场景下"提 issue + workaround" 是 Bun 用户的标准流程。

场景 C · 给写库的人：测试 Node + Bun 兼容性矩阵

你维护的 npm 库被广泛使用，要在 README 上声明"Node + Bun 兼容"——CI 上配置矩阵测试。具体配置：GitHub Actions 用 matrix: { runtime: ['node-18', 'node-20', 'node-22', 'bun-latest', 'bun-1.1'] }，每个 cell 跑测试套件。关键的事：(1) 同一份测试代码——别为每个 runtime 写不同测试，那是反模式；用 Web 标准 / Node 兼容 API 写测试。(2) 关注 corner case—— stream backpressure / event loop 顺序 / native module loading / TS imports 是常见差异点。(3) Bun 上失败 ≠ 你的库挂——可能是 Bun 兼容 bug，提 issue 让他们修。(4) 不要为 Bun 单独 release 二进制——你的库是 JS / TS，Bun 用户跟 Node 用户用同一个 npm 包就行。

场景 D · 决策「应用某个模块切 Bun 还是保留 Node」

团队大部分服务考虑切 Bun，但少数特殊服务（如复杂 worker_threads / Cloudflare Workers / 重度 async_hooks）犹豫。判据：(1) worker_threads 重度使用——简单 worker（CPU 密集任务、SharedArrayBuffer）100% 兼容，切；复杂 worker（多 worker 间转 native handle、自定义 transferable）先验证 spike，可能要保留。(2) Cloudflare Workers / Vercel Edge 部署——Bun 不能跑 Edge（详见 P1.2），保留 Node 或用 wrangler / vercel CLI。(3) async_hooks instrumentation 工具—— OpenTelemetry / Datadog APM 用 async_hooks 做 trace，Bun AsyncLocalStorage 稳定，简单场景能跑；深度 instrumentation 可能踩 corner case，spike 验证。(4) vm sandbox—— 不强烈推荐（V8 vm 沙箱有逃逸风险），但如果项目用了，验证 Bun vm 实现满足你的需求。综合后通常的结论：保留 Node 的服务占 10-20%，剩余切 Bun——渐进式切换的力量。

// Node 风格代码在 Bun 上直接跑（"100% 兼容"的实证）
import fs from 'node:fs/promises';
import http from 'node:http';
import crypto from 'node:crypto';
import { Worker } from 'node:worker_threads';

// 全部正常工作 —— Bun 用 Zig 重写，但 API 完全对齐 Node
const data = await fs.readFile('./config.json', 'utf-8');

const server = http.createServer((req, res) => {
  res.writeHead(200);
  res.end('ok');
});
server.listen(3000);

const hash = crypto.createHash('sha256').update('input').digest('hex');

const worker = new Worker('./worker.js');
worker.postMessage({ task: 'compute' });
javascript 
// 在同一个项目里既能用 Node 标准 API 又能用 Bun 内置 API
// （但写跨 runtime 库时仅用 Node 标准 API）

// 写库（要在 Node + Bun + Deno 都能用）
export async function readConfig(path) {
  const fs = await import('node:fs/promises');  // ✅ 跨 runtime 标准
  return JSON.parse(await fs.readFile(path, 'utf-8'));
}

// 写应用（确定只跑 Bun，可以用 Bun-only API）
export async function readConfigBun(path) {
  return await Bun.file(path).json();  // ✅ Bun 上更快
  // 但这段代码在 Node 上 typeof Bun === 'undefined' 报错
}

// 自适应：检测 runtime
export async function readConfigSmart(path) {
  if (typeof Bun !== 'undefined') {
    return await Bun.file(path).json();  // 快速路径
  }
  const fs = await import('node:fs/promises');
  return JSON.parse(await fs.readFile(path, 'utf-8'));
}
javascript 
// V8-only API 检测和 fallback
// 一些 npm 包用 v8.serialize 做对象序列化 — Bun 不支持

let serialize, deserialize;
try {
  // Node：用 v8.serialize（支持循环引用 / 复杂类型）
  const v8 = await import('node:v8');
  serialize = v8.serialize;
  deserialize = v8.deserialize;
} catch {
  // Bun / 其他 runtime：用 structuredClone（Web 标准）
  serialize = (o) => structuredClone(o);
  deserialize = (o) => o;  // structuredClone 是同步的
}

// 用法
const cloned = deserialize(serialize(complexObject));

05 常见陷阱

假设「100% 兼容」=「行为完全一致」

Bun 的 marketing 说"100% Node 兼容"——这是表面 API 兼容，不是行为完全一致。在 stream backpressure 时机、process.nextTick 调度、TLS 协议默认配置、HTTP keep-alive 行为等细节上，Bun 与 Node 偶尔会有微妙差异。这些差异在 90% 应用代码中不会被感知，但在非常依赖事件循环精确顺序的代码里会出 corner case。规避：production 切 Bun 前 spike 测试关键路径；遇到行为差异先查 GitHub Issues 确认是否 known issue；不依赖具体调度顺序写代码。

用 V8 内部 API 假设 Bun 也支持

v8.serialize / v8.deserialize / v8.getHeapStatistics / v8.setFlagsFromString 等 V8 模块是 V8 specific 的，Bun 不支持。一些 npm 包（旧版 structured-clone / 一些 cluster 库）会用 v8.serialize，这些在 Bun 上直接 ReferenceError。规避：写库时只用 Web 标准 API（structuredClone() / postMessage）；用 v8 模块的库要在 Bun 上特殊 fallback。

依赖 Bun-only 行为写库

Bun 允许 ESM 文件里 require()、自动加载 .env、TS / JSX 直跑——这些是 Bun 的扩展兼容，Node 严格不允许。写应用时用这些扩展可以；写库时千万不要——你的库被 Node 用户用时会报错。具体：库的源码用纯 ESM 或纯 CJS，不混用；不依赖 process.env 自动加载（让用户自己装 dotenv）；TS 库发布前用 tsc 编译成 .js（不要让用户 import .ts）。

认为「Bun 不支持的 npm 包要 fork」

遇到某个 npm 包在 Bun 上崩溃时——先报 issue给 Bun，再考虑 workaround。Bun 团队对兼容性 issue 响应很快（典型 1-2 周修），fork 包是最后手段。fork 的代价：(1) 你要持续 sync upstream changes；(2) 你的项目其他人 onboard 时多一层认知负担；(3) 安全 patch 跟不上。规避路径：(1) 报 issue + 提 minimal repro 给 Bun；(2) 临时 workaround（这一个服务暂用 Node）；(3) 如果包维护者不积极 + Bun 团队修不了，再考虑 fork。
