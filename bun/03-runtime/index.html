P3 · "内置一等公民" 哲学

Bun 与 Node 设计哲学最大分歧：Bun.serve / Bun.file / Bun.sql / Bun.password 是一等公民（runtime 自带 + Zig 实现热路径）；Node 的 http / fs / pg 是 npm 包或标准库（走 V8 binding 层）。这一阶段讲清这套哲学差异 + Node 兼容层 2026 年的现状边界。

章节数 2 主要来源 Bun API 文档 状态 ⏳ 待撰写

本阶段覆盖

  • Bun.serve() —— HTTP/HTTPS/HTTP2 server 一等公民；fetch-style handler；显著快于 http.createServer (~2-3x)
  • Bun.file(path) + Bun.write() —— lazy 文件对象；底层 mmap + zero-copy；可作 fetch body；自动选 sendfile/splice/mmap
  • Bun.password / Bun.glob / Bun.spawn / Bun.env —— 一等公民版的常用任务；不需要 bcrypt / glob / dotenv 包
  • Bun.sql (Bun 1.2+) —— Postgres 一等公民客户端；连接池 + prepared statements 内置
  • HTTP routes (Bun 1.2+) —— Bun.serve({ routes: { ... } }) 内置路由
  • 一等公民 vs npm 包性能差 —— 不走 require 解析、不走 V8 binding 层、Zig 直接热路径（2-10x）
  • 零拷贝设计 —— Bun.serve(req).write(Bun.file()) 走 sendfile 系统调用，绕用户态 buffer
  • Web 标准对齐 —— Request / Response / Blob / ReadableStream，让代码部分跑 Cloudflare Workers / Edge runtime
  • Node 兼容矩阵 (2026) —— 完全兼容 / 部分兼容 / 不兼容 / 不打算实现
  • N-API 兼容 —— better-sqlite3 / sharp / bcrypt 大部分能跑，corner case 看 GitHub Issues
  • compat:nodejs label —— GitHub 上兼容性进度的最权威记录
  • 何时仍 fallback 第三方包 —— Bun 内置等价物边界 case / 未实现的 Node 内部 API
