P2 · JavaScriptCore + Zig

Bun 两个最关键技术决策。JavaScriptCore（不是 V8）：4 层编译流水线（LLInt → Baseline → DFG → FTL/B3）/ JSC Structure / Riptide GC / 启动快的真正原因 / Apple 政治限制下沉淀的 mobile 优化恰好契合服务端。Zig（不是 Rust）：comptime 编译期 / 显式内存管理 / uSockets 替代 libuv / 为什么不强制安全反而是 Bun 的优势。

章节数 2 主要来源 JSC blog 状态 ⏳ 待撰写

本阶段覆盖

  • JSC 起源 —— KJS (KDE) → Apple 接管 (2002) → Safari/WebKit → iOS (2007) → macOS Big Sur 系统组件
  • JSC 4 层编译流水线 —— LLInt（LLVM 解释器）→ Baseline（线性 JIT）→ DFG（数据流图优化）→ FTL/B3（top-tier JIT）
  • JSC Structure vs V8 Hidden Class —— 同等机制，Polymorphic Inline Cache 差异点
  • Riptide GC —— 增量 + 并发；Bun 大对象用自家分配器
  • JSC 启动快的真正原因 —— LLInt 用 LLVM 写比 Ignition (V8) 启动开销低；mmap bytecode 跳过 parse
  • JSC 的尴尬 —— Bun 之外没有大型 embedder；社区比 V8 薄；嵌入文档少
  • Zig 是什么 —— 2016 年 Andrew Kelley 启动；目标 better C；comptime 编译期执行；显式内存（无 GC、无所有权）；2026 仍是 0.x
  • Zig 对 Bun 的核心价值 —— comptime 让 hot path 内联 / 显式内存避免 Rust 借用检查 ergonomic 开销 / C ABI 兼容好
  • uSockets —— Alex Hultman 写的 C++ 异步 I/O；Bun 用 Zig glue 替代 libuv；Linux 上直接 io_uring
  • Zig vs Rust 取舍 —— Rust 强制安全（编译慢但 runtime 安全）/ Zig 让程序员管内存（Bun 选 Zig 是性能压榨 > 编译期安全）
  • Zig 风险 —— pre-1.0 / 社区比 Rust 小一两个数量级 / 实质是「赌 Zig 能跟上」
