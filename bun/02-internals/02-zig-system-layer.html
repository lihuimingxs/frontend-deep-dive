Zig 系统层：为什么不是 Rust

Zig 视角。讲清 Zig 在 Bun 里扮演什么角色、为什么不是 Rust / C++、Zig + uSockets 怎么替代 libuv。关键洞察：Zig 不强制安全反而是 Bun 的优势——comptime 让"硬编码"不再是反模式（专用代码免费），uSockets 在 Linux 直接 io_uring 跳过 epoll 包装。但 Zig 仍是 0.x（pre-1.0），社区比 Rust 小一两个数量级，是 Bun 实质的"赌一把 Zig 能跟上"决策。

阅读时长 ~30 分钟 前置 P2.1 + Node P2.1 libuv 主要来源 ziglang.org · uNetworking/uSockets · oven-sh/bun 源码

01 速览

  1. Zig 是 better C，不是 Rust 替代品——目标是给 C 程序员一个更安全的工具，不是给 Rust 程序员一个更快的工具。Zig 没有借用检查器、没有所有权语义、不强制安全；Bun 选 Zig 是因为性能压榨 > 编译期安全的工程决策。
  2. uSockets 替代 libuv 让 Bun 在 Linux 上直接走 io_uring——libuv 的跨平台抽象层有 overhead；uSockets 跳过这一层直接调底层 syscall。单连接吞吐能高 20-30%（场景相关）。
  3. Zig 的 comptime 是 Bun 性能差异化的隐性武器——编译期代码执行 + 反射，让 hot path 上的 JSON parser / HTTP parser / SQL 字段映射等代码不需要写得通用，每种类型生成专用代码。这种优化在传统语言里叫"过度工程"，Zig 里 comptime 让它免费。

02 底层逻辑

Zig 是什么：better C，不是 Rust 替代品

Zig 是 2016 年 Andrew Kelley 启动的系统语言。Andrew 的设计目标在 Zig 官网首页就写得清楚——"a better way to maintain robust software"。这个表述刻意避开了"Rust"或"C++ 替代品"——Zig 的目标用户是C 程序员，不是 Rust 用户。具体差异：

(1) 没有所有权语义 / 借用检查器——Rust 的核心特性，Zig 完全没有。Zig 程序员需要自己管内存（用 allocator + defer），不能依赖编译器证明安全。这听起来像 C，但 Zig 加了几个改进：每个分配必须显式传 allocator（不能用全局默认 allocator）、defer 语义自动释放、optional types 防 NULL pointer。

(2) 没有运行时——Zig 编译产物没有 GC / 没有异常 / 没有类型擦除。和 C 一样直接编译成 native binary，可以静态链接、跨平台编译、嵌入到任何位置。Rust 也没有 GC 但有 panic / runtime allocator —— Zig 的 runtime 比 Rust 还小。

(3) comptime 是核心特性——任何 Zig 函数都可以在编译期跑（不只是 const expression）。这让 Zig 同时是"运行时语言 + 编译期元编程语言"——后面单独展开。

(4) 显式控制流——没有运算符重载、没有隐式类型转换、没有 RAII、没有 destructor。所有行为都看代码就能预测，没有"幕后魔法"。Rust 有相对显式的语义但仍然有 Drop trait / Deref coercion / Index trait 等隐式机制；Zig 把这些全去掉。

这套设计在Bun 这种"对每一字节内存 + 每一纳秒延迟都极致优化"的项目里非常合适——Jarred 不需要 Rust 的安全保证（Bun 是单一作者主导，bug 自己排），但需要 C 风格的低级控制 + 比 C 好的工程性。Zig 同时给了这两点。

为什么不是 Rust：Jarred 公开的几个理由

Bun 选 Zig 不选 Rust 在社区里被反复讨论。Jarred 在 Twitter / Discord / Hacker News 上多次解释，归纳起来是：

(1) Rust 借用检查器在 Bun 这种"低级数据结构频繁互引用"的代码里太繁琐。Bun 内部有大量"buffer 引用 buffer 引用 buffer"的结构（HTTP parser → header buffer → body chunk → response object），这种循环引用在 Rust 里要用 Rc / RefCell / Arc 等 wrapper 表达，每个 wrapper 都有运行时开销 + 写起来啰嗦。Zig 不强制安全所以可以用裸指针，写起来直白且零开销。

(2) Rust 编译时间太长。Bun codebase ~50万行，Rust 重写编译会显著比 Zig 慢——Zig 编译速度优势是 5-10x（取决于版本和场景）。Bun 团队迭代节奏很快，编译速度是开发体验的瓶颈。

(3) Rust 标准库 / 第三方库更"高层"，对低级控制需求不友好。比如 String / Vec 都自动处理 grow / reallocation，但 Bun 需要的是"我自己控制 buffer 的每一次 grow"——Rust 标准库写起来要绕一层。Zig 标准库刻意保持低级，更接近 Bun 的需求。

(4) C ABI 兼容性 Zig 比 Rust 好。Bun 大量调 JSC 的 C++ 接口、调系统库（OpenSSL / SQLite / libcurl 等）的 C 接口——这些都是 C ABI。Zig 的 extern 直接写 C 函数原型，调用零包装；Rust 要写 #[repr(C)] + extern "C"，再做 unsafe wrapper，工作量大。

这些理由不意味着 Rust 不好——Rust 在"团队协作 + 长期维护 + 安全敏感"场景仍然是最强系统语言。但 Bun 的具体场景（单作者主导 + 极致性能 + 大量 C ABI 调用 + 快速迭代）让 Zig 的取舍更合适。这是一个工程决策，不是路线之争。

comptime：Zig 的核心武器

comptime 是 Zig 最独特的特性。简单说：任何函数都可以在编译期执行，而不只是 const expression。这个能力把"运行时代码"和"编译期代码"统一成同一套语法，给 hot path 优化打开了新空间。

具体看几个 Bun 内部的应用：

(1) HTTP parser 编译期生成专用代码。HTTP 协议有几十个 header（Content-Type / Content-Length / Transfer-Encoding 等），每个 header 都需要 parser 识别。传统写法：parser 用 hash map 或 binary search 找 header；comptime 写法：编译期为每个常见 header 生成专用比较代码，不用 hash map，直接 inline 比较 string 字面量。这种代码在 Rust 用 macros 也能写（如 match!），但 Zig 的 comptime 让它更自然——不用学新语法，普通函数加 comptime 关键字就行。

(2) 编译期生成 SQL row 解析器。Bun.sql 接受 query 时通过 comptime 在编译期分析 query 结构 → 生成专用的 row → JS 对象映射代码。运行时直接 jump 到这段专用代码，不需要"解析 query 结构 + 反射映射字段"——把通用反射的代价从运行时移到编译期。

(3) 编译期 dispatch table。Bun 内部很多 enum-style dispatch（如根据文件扩展名选 transformer）用 comptime 生成 jump table——每个 case 都是 inline，没有运行时 switch。

这种"编译期生成专用代码"在传统系统语言里叫过度工程——你为每个特殊情况写专用代码，代码膨胀 + 维护噩梦。但 Zig comptime 让同一段代码在编译期 specialize 出多个版本，源代码是一份，产物是多份——叫"零成本元编程"。这是 Bun 性能优势的隐性来源。

uSockets：替代 libuv 的异步 I/O 层

Bun 不直接用 libuv（Node 的异步 I/O 库，详见 Node P2.1 libuv 设计）——它用 uSockets。uSockets 是 Alex Hultman 写的 C++ 异步网络库，原本用于 uWebSockets.js（一个号称"比 Node 内置 ws 快 10x"的 WebSocket 库）。Bun 把它和 Zig glue 起来作为 event loop 基础。

uSockets 与 libuv 对比：

维度 libuv (Node 用) uSockets (Bun 用)
语言 C C++ (Bun 用 Zig glue)
跨平台抽象 厚（统一 epoll/kqueue/IOCP/event ports） 薄（直接 syscall）
Linux 优化 epoll（伪 io_uring 也支持但默认不开） io_uring 直接（Linux 5.1+）
handle 类型 多（TCP / UDP / FS / Pipe / Timer / ...） 少（聚焦网络 socket / timer）
Worker pool 有（4 线程默认，处理 fs / DNS / 用户 work） 无（Bun 自家管理 worker）
ABI 稳定性 高（libuv 1.x 长期） 低（按 Bun 需求改）

libuv 是"通用跨平台异步 I/O"定位——给所有想用异步 I/O 的程序用，所以抽象层厚、handle 类型多、ABI 稳定（保证 Node 17 年向后兼容）。这个通用性的代价是：每次 socket 操作要走 libuv 的抽象层（epoll → libuv handle → 用户回调），有几层间接。

uSockets 是"专为 web server 优化"定位——只关心网络 socket + timer，不管 fs / DNS / pipe。这让它可以在每个平台直接调最优的 syscall：

Linux：直接 io_uring（5.1+ 内核）——比 epoll 在高并发下吞吐量高 20-30%，因为 io_uring 是"批量提交 + 批量获取"模型，每次 syscall 处理多个事件，syscall 切换开销摊薄。libuv 1.45+ 也开始实验性支持 io_uring 但默认仍 epoll，因为兼容性顾虑（旧内核不支持）。Bun 直接用 io_uring 是因为不需要兼容老内核。

macOS：kqueue——和 libuv 一样，但调用层更薄。

Windows：IOCP——这是 Bun 1.1 (2024-04) 才完整支持 Windows 的核心。Bun 1.0 之前 Windows 缺位 fundamentally 是因为 uSockets 的 IOCP 适配没做完。

这层差异让 Bun 在"高并发 web server"场景比 Node 快——单机能扛 100k+ 并发连接 / RPS 高 30-50%（具体数字看 benchmark / 负载形态）。但在"很多 fs 操作"场景 Bun 优势不明显——因为 fs 操作 Bun 自己管理 worker pool，和 libuv 类似。

Zig + uSockets 协作：Bun event loop

Bun 的 event loop 是 Zig 写的协调层 + uSockets 的异步 I/O 层。简化看：

(1) main loop 是 Zig 写的循环——每个 tick 调 uSockets 的 us_loop_run 处理一批 I/O 事件 → 调 JSC 的 JSC::Heap::collectAsync 触发 GC → 调 Bun 自己的 worker pool 取结果 → 下一 tick。

(2) JS 回调通过 JSC 的 JSValue protect/unprotect 保留引用——和 Node 调 V8 类似，但 JSC API 不一样。Zig 用 extern 直接调 JSC C++ 接口，没有 wrapper 层。

(3) microtask 处理在每个 phase 之间 drain——和 Node 行为对齐（详见 Node P2.2 Event Loop），不然 Promise / async/await 行为会出 corner case。

这套设计的关键差异：Node 的 event loop 是 libuv → V8 → Node JS callback三层；Bun 是 uSockets → JSC → Bun Zig glue → JS callback。看起来 Bun 多一层（Zig glue），但 Zig glue 实际是"无开销 wrapper"——Zig comptime + 直接 extern C 让 glue 在编译后几乎不留痕迹。所以 Bun 实际开销不比 Node 多。

Zig 的风险：pre-1.0 + 社区小

Zig 在 2026 年仍然是 0.13 / 0.14（具体版本随时间推进）——未到 1.0。这意味着：

(1) 语法 / 标准库还在变——每个 0.X 版本都可能有 breaking change。Bun 团队需要持续跟进 Zig 升级，每次都要适配代码。这是 Oven 团队额外的工程负担。Zig 1.0 何时发布没有官方时间表（Andrew 多次说"等 LLVM 升级 + 标准库稳定 + ...才会 1.0"），目前看 2026-2028 之间是可能的。

(2) 社区比 Rust 小一两个数量级——Rust 大概 100 万+ 活跃开发者；Zig 估计 1-5 万。这意味着遇到问题搜不到答案、第三方库少、社区贡献的 review 少。Bun 团队大部分情况下要自己解决问题或直接提 PR 给 Zig upstream。

(3) Zig 编译器自身仍有 bug——pre-1.0 软件，corner case 编译错误不少见。Bun 团队偶尔要绕过 Zig bug 写 workaround code，等 upstream 修复后再清理。

(4) 标准库可能"被替换"——Zig 0.x 时代 Andrew 多次大规模重构标准库（如 IO 抽象、async/await 系统），每次都让现有代码失效。Bun 用了部分 Zig 标准库，每次都要适配。

这些风险加起来意味着 Bun 实质是"赌 Zig 能跟上"——如果 Zig 发展顺利、1.0 按预期发布、社区扩展，Bun 的 Zig 选择会越来越好；如果 Zig 出现重大问题（Andrew 离开 / 项目内斗 / 关键设计错误），Bun 会受拖累。

从风险管理角度看，Bun 团队的对冲策略是"Bun 自己 vendor 大部分 Zig 代码"——核心依赖不靠 Zig 标准库，而是自己实现 / fork。这降低了 Zig 标准库变化的影响，但增加了 Bun codebase 体积。

性能：Zig + uSockets 的实际收益

把 Zig + uSockets 与 C/C++ + libuv 公平比较：

低级 hot path 性能：Zig + comptime 比 C / C++ 略快（5-15%），主要来自 comptime specialization + 内联激进。

HTTP server 单机 RPS：Bun.serve（Zig + uSockets）比 Node http.createServer（C + libuv）快 ~2-3x 在简单场景（"hello world" 路由）；复杂场景（数据库 + 业务逻辑）差距压缩到 20-50%（因为瓶颈不在 I/O）。

WebSocket 连接密度：单机 Bun 能扛 100k+ WebSocket 连接，Node + ws 包通常 50k 以内崩溃。这是 uSockets 设计的核心场景。

fs 操作：Bun 略快于 Node（10-30%），因为 Zig 直接 syscall + 没有 libuv handle 抽象层。但差距比网络 I/O 小，因为 Bun 仍走 worker pool。

编译速度：Bun（Zig）每次 incremental rebuild ~1-3s；同等规模的 Rust 项目 ~10-20s；C++（Node）~5-10s。这影响 Bun 团队迭代速度。

这些数字加起来表明 Zig + uSockets 选择在性能维度上是对的——Bun 性能优势真实可测。剩下的问题是这些优势能否在 Zig pre-1.0 风险消化后持续——这是未来 5 年要回答的。

03 应用场景

场景 A · 给团队解释「Bun 性能优势是真实的还是 marketing」

团队成员怀疑 Bun benchmark 有水分——给出技术细节而不是空泛回答。底层来源：(1) JSC LLInt 让启动开销降一个数量级（详见 P2.1）；(2) uSockets 在 Linux 直接 io_uring 跳过 libuv 抽象层，单连接吞吐 +20-30%；(3) Zig comptime 让 hot path 代码（HTTP parser、JSON parse、SQL 字段映射）编译期 specialize 出专用代码；(4) Bun 大对象用 Zig 直接分配不走 JSC GC，减少 GC 暂停。这四件事都不可争议——可以分别用 micro benchmark 验证。Bun marketing 的 5x / 10x 数字往往把这些优势都叠加到极端场景上，看起来夸张但底层机制是真的。生产实际收益视场景而定，I/O 密集服务能拿到 50-200% 提升，CPU 密集服务可能只有 10-30%。

场景 B · 评估「我们能不能用 Zig 写自己的 native module」

遇到团队想跟随 Bun 的潮流，用 Zig 写 native 扩展（如自定义图像处理模块）。判据：(1) 团队能力——Zig 学习曲线比 Go 陡，比 Rust 平；要团队成员愿意投资 1-2 周学习。(2) 项目需求——Zig 适合"低级 + 性能 + C ABI 集成"；如果你的需求是"业务逻辑写 native"（如 ORM 加速）那 Rust + N-API 仍然更主流。(3) 生态成熟度——Zig 第三方库少 + 没有 cargo 这种成熟包管理；2026 年仍然要自己 vendor 依赖。(4) 对接 Bun 还是 Node——Zig 更容易写 Bun FFI（详见 P5.1），N-API 集成 Rust 更主流。综合后：已用 Bun 且团队愿意投资 Zig的场景适合，其他大部分场景仍 Rust。

场景 C · 排查「io_uring 在我们的服务器上没生效」

你部署 Bun 服务到 Linux 服务器，发现性能没达到 Bun 文档承诺的水平——首先检查 io_uring 是否生效。具体步骤：(1) uname -r 确认内核 ≥ 5.1（一般是的）；(2) cat /proc/kallsyms | grep io_uring 确认内核启用了 io_uring（一般是）；(3) Bun 是否 fallback 到 epoll —— 检查 Bun 启动日志或 strace 验证。(4) 容器环境（Docker / K8s）—— 某些 seccomp profile 禁用 io_uring syscall（如 Docker 默认 seccomp 在 5.10 之前内核上 block io_uring），需要调整 profile 或升内核。这种场景下 Bun 性能会回落到接近 Node 水平。最佳实践：production Bun 部署确保内核 ≥ 6.0 + Docker 用 unconfined seccomp 或更新 base image。

场景 D · 给做长期技术决策的 CTO：评估 Zig 风险

CTO 关注"5 年后 Zig 会不会还是 0.x"——这是合理担心。评估角度：(1) Andrew Kelley 个人因素——Andrew 是 Zig 几乎单点的核心维护者（情况像早期 Node 之于 Ryan Dahl）；他离开 / 健康问题会显著拖慢项目；2026 Zig 已开始有几位资深贡献者，bus factor 在缓解。(2) Zig Foundation 财务可持续——Zig 是非营利基金会，靠捐赠（GitHub Sponsors / 企业赞助）。如果 Bun 商业化成功，Oven 可能成为重要赞助方，反向稳定 Zig。(3) 1.0 时间表——Andrew 公开表态 1.0 不会是终点而是稳定承诺；多数估计 2027-2028 之间。(4) 对 Bun 影响——即使 Zig 最坏情况（项目停滞），Bun 已有的 Zig 代码可以继续维护（Bun 团队懂 Zig），只是不能享受 Zig 后续优化。综合后：Bun 的 Zig 风险不是致命，是"长期天花板"——Bun 不会因为 Zig 死，但可能因为 Zig 进展慢而错失某些优化机会。

04 典型代码

这一章不展开 Zig 语法（这是个 JS 主题），但展示几段代码理解 Zig + uSockets 在 Bun 内部的位置。

zig 
// Bun 内部的简化版 HTTP parser（伪代码示意 comptime 思路）
// 真实 Bun 源码更复杂；这里展示 comptime 特殊化的核心想法

const std = @import("std");

// 编译期生成的 header 名 → enum
const HeaderName = enum {
    content_type,
    content_length,
    transfer_encoding,
    // ...更多
};

// 编译期生成专用比较代码
fn matchHeader(comptime name: []const u8, input: []const u8) bool {
    // comptime 生成 inline 比较 —— 不走 hash map / loop
    if (input.len != name.len) return false;
    inline for (name, 0..) |c, i| {
        if (input[i] != c) return false;
    }
    return true;
}

// 用法
fn parseHeaderName(input: []const u8) ?HeaderName {
    if (matchHeader("content-type", input)) return .content_type;
    if (matchHeader("content-length", input)) return .content_length;
    // comptime 让每一个 if 都被 inline 成专用代码
    return null;
}

  },
  fetch(req, server) {
    if (server.upgrade(req)) return;
    return new Response('expect ws');
  }
});
// Bun 单机能扛 100k+ 并发 WebSocket connections
// Node + ws 通常 50k 内 OOM

05 常见陷阱

假设「Zig 是 Rust 替代品」

看到 Bun 用 Zig，有人推论"Zig = better Rust"。这是错的定位——Zig 是 better C，目标用户是 C 程序员，不是 Rust 用户。Zig 没有借用检查器，写代码安全要靠程序员自己；Rust 强制安全。两个语言的取舍点不同，不是替代关系。判断你的项目用 Zig 还是 Rust：(1) 团队已有 Rust 经验 → 继续 Rust；(2) 团队有 C 经验需要更安全工具 → Zig 合适；(3) 极致性能 + 单作者 / 小团队主导 → Zig 合适；(4) 团队协作 + 长期维护 + 安全敏感 → Rust 仍然主流。

期待 Zig 1.0「快到了」

Zig 是 2016 年立项，2026 年仍是 0.x。社区里反复有"1.0 还有几年"的讨论，但 Andrew 没有公开承诺时间表，多数估计 2027-2028 之间。如果你做长期技术决策（"5 年后还能用 Zig 吗"），不要假设 1.0 在短期内到来——按"持续 0.x"规划风险更稳。Bun 的对冲是 vendor 大部分依赖 + 团队 Zig 专家持续投入；其他想用 Zig 的项目要类似考虑。

把 Bun 性能优势归因于「Zig 比 C 快」

Bun 性能优势不是因为 Zig 比 C 快——Zig 和 C 在低级层面性能差不多。Bun 优势的真正来源是组合：(1) JSC LLInt 启动快；(2) uSockets 直接 syscall 比 libuv 抽象层薄；(3) Zig comptime 让 hot path 编译期 specialize；(4) 大对象绕过 JSC GC。其中 (1)(2)(4) 不需要 Zig，C 也能做；只有 (3) 是 Zig 独有。换句话说"Bun 用 C 写也能拿到 70% 性能优势"——Zig 加分，但不是全部。这种 attribution 错位会让你低估"换个语言重做 Bun"的难度——以为只是语言选择，其实是整套架构选择。

Linux 容器里 io_uring 默认被 seccomp block

Docker 默认 seccomp profile 在内核 5.10 之前 block 大部分 io_uring syscall。这导致 Bun 在某些容器环境下 fallback 到 epoll，性能优势打折扣。常见症状：本地 macOS 跑 Bun 性能很好，部署到 Docker 容器变得和 Node 差不多。规避：(1) 升内核到 6.0+；(2) Docker 用 --security-opt seccomp=unconfined（生产不建议）；(3) 自定义 seccomp profile 显式 allow io_uring syscall。生产 Bun 部署务必先验证 io_uring 是否生效，不要假设容器和裸机表现一致。
