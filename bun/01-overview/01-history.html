Bun 项目史（2021-2026）：Jarred Sumner / Oven / 1.0 → 1.2

Bun 5 年的故事讲成一条主线——不是按版本号叙述，而是按每次设计决策及其根因。从 2021-07 Jarred Sumner 在 Twitter 演示「Node 启动 50ms vs Bun 5ms」引爆讨论，到 Oven 公司化拿 YC + a16z 投资，到 Bun 1.0 (2023-09)、1.1 Windows 一级支持 (2024-04)、1.2 Bun.sql + binary lockfile (2025+)。把"为什么 2021 挑战 Node 是合理的"、"Jarred 单人项目 → 公司化的张力"、"Bun 与 Deno 的路线分歧"三个底层问题讲透。

阅读时长 ~25 分钟 前置 Node P1.1 项目史 主要来源 bun.com/blog · oven-sh/bun GitHub · Jarred Sumner Twitter

01 速览

  1. 5 年从 demo 到生产 ready，速度本身是关键洞察——2021-07 Twitter demo → 2023-09 Bun 1.0 production-ready 仅 26 个月，是 Node 用 6 年（2009-2015）才走完的同等成熟度。
  2. Jarred Sumner 至今仍是核心开发者——从单人项目到 Oven 公司化（2022），但 GitHub 上很多关键 PR 仍是 Jarred 自己写。这是优势（决策快、技术品味统一），也是隐忧（bus factor）。
  3. Bun 选「兼容 + 加速」路线，不是「颠覆 Node」——这是与 Deno 最大的路线分歧。Deno 2018 选「另起炉灶 + URL 导入 + 默认无 npm」结果生态拉不起来；Bun 选「100% Node 兼容，然后在 install/run/build/test 各做更快」，绕开了生态护城河。

02 底层逻辑

2021-07 · Twitter demo 引爆

Jarred Sumner 2021 年 7 月在 Twitter 发了一个 30 秒的视频：终端里 node --version 启动用 50ms，bun --version 启动用 5ms——10 倍差距。视频迅速 viral，几小时内转发上万。这是 Bun 第一次被前端 / 服务端社区注意到。

这个数字本身不是 Bun 设计的全部，但它为什么会成为 viral hook 揭示了一个被低估的需求：过去十年 Node 社区一直关注 RPS / 内存 / GC 这些"长期跑"的性能指标，没人特别在意启动时间。但 2020 年代 Lambda / Serverless 普及之后，启动时间从"无关紧要"变成了"决定成本"——AWS Lambda 冷启动每 100ms 直接转化为请求延迟和按时长计费。Jarred 这个 demo 戳的就是这个被忽视的痛点。

Twitter 帖子下面的讨论也很有意思：第一批留言主要是怀疑——"是不是 cherry-pick 数字"、"包大就慢了"、"实际跑业务呢"。但接下来几周 Jarred 陆续放出 HTTP server benchmark（Bun.serve vs http.createServer，~2-3x 提升）、bun install 安装 React 项目的对比（vs npm 5x、vs pnpm 2x），数字一路撑住——Bun 不是单点优化，是整套技术栈的重新选型在每一处都拿到了红利。

2021-2022 · 私下开发：JSC + Zig + uSockets 选型

从 2021 年中到 2022 年中的一年里，Jarred 几乎没公开任何东西，处于"全职做产品"状态。期间确立的三个核心技术决策——JavaScriptCore（不是 V8）/ Zig（不是 Rust 或 C++）/ uSockets（不是 libuv）——是 Bun 后来一切性能特性的基础。这三个选择会在 P2 阶段展开。这里只点出一个共性：每一项都和 Node 反向选择，且每一项都是少数派路线。

少数派路线的代价是没有现成参考——Node 用 V8 + libuv 已经 12 年，文档 / 教程 / 排错经验全在；Bun 选 JSC + Zig + uSockets，每个组件 Jarred 都得自己读源码 + 试错。这一年没有 release 不是 Jarred 在偷懒，是真的在写底层。Bun 0.1 出来的时候 codebase 已经是 ~50 万行 Zig，绝不是「拿现成轮子粘起来」的程度。

2022 年期间 Jarred 拿到了 Y Combinator 加速器的 W22 batch（2022 年冬季批次），随后从 a16z 等机构拿到了种子轮——具体金额 Bun 没有完全公开，但社区共识是足以让 Jarred 招起一支小团队（~5 人）持续开发。Oven Inc. 成立，Bun 从"个人副业"变成"商业产品"——这个结构性转变会贯穿后面所有故事。

2022-07 · Bun 0.1 公开 Beta

2022 年 7 月 Bun 0.1 公开 Beta 发布——这是第一次让外部用户能 curl -fsSL https://bun.sh/install | bash 装上 Bun。但更重要的是 0.1 的形态：发布日就同时具备 bun run / bun install / bun build / bun test 四件套。这不是后续慢慢扩展的——是从 day-1 就以"工具链一体化"为产品定位。

这个定位在 2022 年还很激进。当时 Node 生态的标准做法是"每个工具一个仓库一个团队"——Node 自己只管 runtime，npm 管包管理，esbuild/Rollup 管 bundle，Jest/Vitest 管测试。Bun 出来直接说"这四件事都是同一个 Zig parser 的不同前端"——把行业默认假设打破了。社区一开始的反应混合：一部分人觉得"重复造轮子"，另一部分人立刻意识到"如果共享 parser 真能让四个工具都更快，这是一个新的价值主张"。后来事实证明后者是对的。

0.x 时代的 Bun 仍然问题很多——很多 npm 包跑不起来、Windows 完全不支持、native module 兼容性破破烂烂、文档相比 Node 单薄。0.5、0.6、0.7 一路往前 ship 的过程中，Jarred 在 GitHub Issues 上的活跃度可以说反常地高——单日处理几十个 issue 是常态，半夜回 PR review 也常见。Bun 早期"问题多但响应快"的口碑就是那一年攒起来的。

2023-09 · Bun 1.0：「production-ready」声明

2023 年 9 月 8 日，Bun 1.0 发布。这是 Bun 第一次正式声明「production-ready」——意思是 Jarred + Oven 团队认为 Bun 已经达到可以在生产环境替代 Node 的成熟度。Linux + macOS 一级支持，Windows 仍然是 WIP（很多生产用户因此暂时还不能切）。

1.0 的关键内容不是新功能（runtime/install/build/test 都已经在 0.x 里有了），而是稳定性 + Node 兼容覆盖率的一次集中收尾。0.x 后期 Jarred 一直说"我们快到 1.0 了"——但 1.0 真正发布前的几个月，团队把焦点完全压到 Node 兼容上：测一遍前 1000 名 npm 包能不能在 Bun 上跑通；测 Express / Fastify / Next.js / Prisma / TypeORM 这些主流框架的全功能。1.0 发布博客里有一张图：Top 100 npm packages，96 个能跑、4 个有 known issue——这个数字成为 1.0 production-ready 的最有力背书。

1.0 发布同时启动了 Bun.sh 域名 + 官方文档站的全面重写。文档变得真的能用——之前 0.x 时代很多 API 只在源码里能查到。这是 Bun 从"hacker 项目"向"开发者产品"过渡的标志性时刻。

2024-04 · Bun 1.1：Windows 一级支持

2024 年 4 月，Bun 1.1 发布，Windows 一级支持。这是过去一年的大事件——之前 Windows 用户要么用 WSL（Linux 子系统），要么完全不能用 Bun；1.1 之后能直接装原生 Bun。这个特性 Jarred 早在 1.0 时就承诺，但实现比预计晚——Windows 异步 I/O（IOCP 模型）和 Linux/macOS（io_uring/kqueue）差异巨大，uSockets 适配 Windows 花了团队接近一年。

Windows 支持的意义比想象中大。前端开发者中 Windows 用户占比不低（特别是 .NET / 游戏 / 桌面背景的工程师），CI 系统也有大量 Windows runners。在 1.1 之前，"我们公司有 Windows 开发者"是一个无脑切 Bun 的 blocker；1.1 之后这个 blocker 消失了，社区里能看到大量"我们终于可以全员切 Bun"的讨论。

1.1 还附带了一波Node 兼容深化：worker_threads 实测兼容、tls 模块更接近 Node 行为、process.binding（不应该用但很多老库还在用）的兼容补丁、N-API corner case 修复——这些不性感但是"能不能切"的硬指标。

2024-2025 · 1.1.x → 1.2.x：从「能用」到「值得用」

1.1 之后到 2025 年间的 1.1.x → 1.2.x 系列，Bun 的重心从"Node 替代品"转向"提供 Node 没有的能力"。几个关键里程碑：

HTTP/2 server（Bun 1.1.x 系列）：Bun.serve() 支持 HTTP/2，包括 server push、stream multiplexing。Node 的 http2 模块虽然 2017 就稳定，但 API 复杂；Bun 把它统一到 Bun.serve() 里，http: 2 一行配置。

Bun.sql（Bun 1.2+）：内置 Postgres 客户端，零依赖、连接池 + prepared statements 自动管理。Node 用户做 Postgres 一般要装 pg 或 postgres.js；Bun 把 SQL 客户端做成一等公民 API，与 Bun.serve / Bun.file 等齐平。这是"内置一等公民"哲学的最强表达——见 P3.1。

Binary lockfile 改文本格式（Bun 1.2 系列）：早期 Bun 1.0 用纯二进制 lockfile（bun.lockb）——install 速度快但用户在 PR review 中看不见 diff，是个被诟病的设计选择。1.2 改成文本格式 bun.lock + 内部二进制 cache：表面是 JSON、人类可读、PR diff 友好；底层仍然有 binary cache 让 install 命令本身依然快。这种"听用户反馈调整设计"的能力是 Bun 团队的核心优势——很多 OSS 项目做不到。

Catalogs（Bun 1.2+）：monorepo 中统一版本管理。声明 [catalog.react]，所有 workspace package 引用相同版本。这是从 pnpm 借鉴过来的特性（pnpm 2023 引入），Bun 跟进迅速。

HTML imports + fullstack 路径（Bun 1.2+）：import index from "./index.html"——Bun 把 HTML 当成模块，集成到 Bun.serve 里。这是把 Bun 推向"全栈框架级别基础设施"的尝试——不只是后端 runtime，前端打包也能管。

2025-2026 · bun.sh → bun.com：商业化探索

2025 年某个时间点，Oven 把官方域名从 bun.sh 切到 bun.com——表面是品牌升级，背后是商业化重心的转移。.sh 后缀是开发者社区文化的暗号，.com 是企业市场的硬通货——这个域名变化反映了 Oven 从"开发者工具品牌"向"企业级 JS 平台"扩张的意图。

2025-2026 期间 Oven 公开探索的商业化方向有两个：Bun Cloud（托管的 Bun runtime，类似 Vercel + Bun 一体化）和Bun for Enterprises（企业级支持合同 + 私有镜像 + LTS 承诺）。这两条路谁会成功还不确定——Vercel 已经占据了"前端开发者云"的位置；企业级支持合同传统上是 Red Hat / Confluent 这种公司的强项，开发工具公司很少能做大。但 Oven 必须解决商业化才能持续投入开发——OSS 项目"靠社区驱动"是浪漫的故事，"靠融资续命"是现实。

截至本章撰写日（2026-04），Bun 1.2.x 仍在密集迭代。每两到三周一个 1.2.X release，每个 release 都有 5-10 个 Node 兼容修复 + 1-2 个新特性。这个迭代节奏即使在 OSS 项目里也是异常快的——可以参考 Node 22 LTS 现在每月一个 patch release 的节奏对比。

底层洞察 1：2021 挑战 Node 为什么合理

2021 年挑战 Node 不是疯狂行为——有三个客观条件聚到一起：(1) Node CJS/ESM 二元状态拖了 5 年没解决。Node 12 (2019) 实验 ESM、Node 14 稳定，但到 2021 年很多库还没 ESM 化、conditional exports 用法混乱、双系统的库经常踩 dual package hazard（详见 Node P3.2）。这给"重新做一个 ESM-friendly 的 runtime"留出了空间。

(2) Node 工具链已经分裂到反生产力。一个标准前端项目要装 Node + npm + Vite + Vitest + Rollup + tsc + 各种 babel plugin / postcss plugin，"setup 一个新项目"已经变成一个需要专门讲的话题。每加一层工具就多一层配置 + 多一个版本协调点。这给"工具链一体化"的卖点留出了空间。

(3) JavaScriptCore 在 Safari 上证明了"启动快"是产品体验差异化卖点。Apple 几年来一直在 mobile 上压榨启动时间——iOS Safari 的页面加载时间在 mobile 比 Chrome 短，部分原因是 JSC 启动比 V8 快。Jarred 把这个 mobile 上验证过的优势挪到服务端，恰好契合 Lambda / 边缘场景的需求。

这三个条件单独看都不够——只有 ESM 痛点没有 JSC 选项就只能优化 Node；只有工具链碎片没有差异化引擎就只是另一个 esbuild。三者叠加才让 Bun 这个产品定位站得住。

底层洞察 2：Jarred 单人 → 公司化的张力

Bun 最大的优势之一是 Jarred 的"个人产品品味"——他对 API 形态、错误信息、CLI ergonomics 的把控极强。Bun.serve() 接受 fetch-style handler、Bun.file() 直接传给 fetch body、bun --hot 保留状态——这些 API 决策都有"一个人对整套产品有完整想象"的痕迹。

同样的优势是隐忧。2026 年的今天，Bun 的核心架构决策仍然由 Jarred 主导——团队扩到 ~10-15 人，但很多关键 PR 仍然 Jarred 亲自写或亲自 review。这意味着如果 Jarred 因为某些原因离开 Oven（健康 / 个人选择 / 公司收购），Bun 的架构演进会有显著真空期。社区里讨论"切 Bun 的风险"时，bus factor 永远是排第一的——比 bug 率、比商业化进展更靠前。

Bun 1.0 之后这个张力有所缓解。core team 的几位资深开发者（Linus Groh / Ciro Spaciari 等）开始独立负责子系统，PR 历史不再是 Jarred 一人打半数。但完全去除 bus factor 风险，估计还需要 2-3 年的团队建设。

底层洞察 3：Bun vs Deno 路线分歧

Bun 和 Deno（2018+）常被并列为"Node 替代品"，但路线截然不同，对照能看清两个项目的取舍：

Deno 路线：「另起炉灶 + 修正 Node 错误」。TypeScript 内置；URL 导入（import { x } from "https://deno.land/std/..."）；权限白名单默认；不兼容 Node API。Ryan Dahl 创立 Deno 时公开了一份"Node 的 10 个我后悔的设计"，Deno 几乎每个决策都是对那份清单的修正。

Bun 路线：「100% Node 兼容 + 在每件事上做更快」。npm 兼容、CommonJS 兼容、Node API 兼容、N-API 兼容。Jarred 几乎没有"修正 Node 错误"的雄心——他承认 Node 设计有遗憾但选择"绕开"而不是"重做"。Bun 的 ESM/CJS 双系统支持、process.binding 兼容、N-API 兼容都是务实的"我承认这是历史包袱但我接住"。

结果分野：Deno 在 2018-2023 期间生态拉不起来——npm 包不能直接用、TS 编译比 Node 慢、URL 导入对企业 git monorepo 不友好。2024 Deno 2.0 发布，核心改动是「默认兼容 npm」+「兼容 Node API 子集」——本质上承认了 Bun 路线对、自己路线错。Deno 2.0 之后 Deno 和 Bun 在产品形态上越来越像。

这个分歧的更深一层：开源生态有惯性，颠覆比改良难一个数量级。Ryan Dahl 是 Node 的发明人，对 Node 痛点的理解最深，但他用 Deno 修正错误的尝试反而比 Jarred 用 Bun 兜底承接更慢——因为 Jarred 选择尊重既有生态，让用户切换成本接近 0。这是 Bun 项目史给整个开源社区的一个重要案例。

03 应用场景

场景 A · 从历史给现在的 Bun 选型决策提供锚点

2026 年评估"我们项目要不要切 Bun"时——理解 Bun 5 年的演进节奏比看当前文档更能告诉你 Bun 是不是稳定。1.0（2023-09）到 1.2（2025+）期间没有 breaking change（不像 0.x 时代每个 0.X 都改 API），稳定性进入了 production curve；Node 兼容率从 1.0 的 96% Top 100 npm 涨到 1.2 的 99%+；Windows 在 1.1 一级支持后 CI 系统全平台可用——这些演进数据合在一起，比任何一篇 benchmark 博客都能告诉你 Bun 已经过了"试试"的阶段，进入了"评估边界"的阶段。判断标准是：你的项目里如果用了 Top 1000 之外的小众 npm 包，或者依赖 Node 内部 API（少见但有），需要单独验证；其他场景，1.2+ 的 Bun 在功能维度上是 production-ready 的。

场景 B · 给 Node 18+ 项目做「切 Bun 成熟度」评估

不少团队在 Node 18 → Node 22 升级路径上犹豫：要不要顺便切 Bun？历史给的判断锚点是——Bun 1.1（2024-04）之前不要切，1.1+ 可以做小服务试点，1.2+ 可以做主力服务。Bun 1.0 production-ready 了，但 Windows 一级支持要等到 1.1；1.1 之后到 1.2 期间 worker_threads / tls / N-API 一系列 corner case 修完，到 1.2 才进入"切了不会被冷门 issue 卡住"的状态。具体评估清单：(1) bun install 能不能跑通（90%+ 项目能）；(2) bun run app.js 能不能跑（95%+ 能）；(3) bun test 能不能跑你的 Jest 测试（70-90% 兼容，复杂 Jest plugin 不兼容）；(4) production 流量跑 24h 看 memory / RPS（应该比 Node 好）。这套评估流程也适用于以后类似的"新 runtime 该不该切"决策。

场景 C · 给 npm 库维护者：发布策略要点

维护一个被广泛使用的 npm 库，2026 年的发布策略要把 Bun 用户列为目标受众，但不需要为 Bun 单独做 build。原因从历史看清楚：(1) Bun N-API 兼容、CJS/ESM 兼容、conditional exports 兼容——你按 Node 标准发，Bun 用户能装能用；(2) 但要测试一下 Bun runtime 跑测试（bun test 或 bunx vitest）发现的 corner case 通常是你的库依赖了 Node-specific 的隐式行为——这种行为往往迟早会成为 bug，提前发现是好事。具体做法：CI 矩阵加一行 bun-version: latest，跑你现有的测试套件——大概率全绿，遇红的查一下 GitHub Issues 上有没有 known issue。这给你的库带来"在两个 runtime 上都跑过"的可信度，几乎零额外维护成本。

场景 D · 给团队 lead：评估 Bun 的可持续性

"切 Bun 5 年后还能活吗"是技术负责人最关心的问题。从 2021-2026 这 5 年的轨迹评估：Oven 拿到 YC + a16z 投资意味着至少 2-3 年的现金流跑道——一线 VC 给的种子轮 + Series A 一般够 18-36 个月运营；2025-2026 探索的 Bun Cloud / Enterprise 商业化方向如果跑通，Oven 进入自我造血状态，Bun 项目可持续；如果跑不通，Oven 要么二次融资要么收缩团队。最坏情况：Oven 倒闭/被收购，Bun 进入纯社区维护模式——但即使如此，已发布的 Bun 二进制不会消失、生态可以 fork。具体决策：核心服务可以切 Bun（你的代码可移植，runtime 倒了你能切回 Node）；强依赖 Bun 独有 API（如 Bun.sql 的特定行为）的代码要写抽象层方便未来回退；商业关系（如选 Bun Cloud 做主力部署）要看 SLA 合同 + 退出条款。

05 常见陷阱

把 Bun 误读为「颠覆 Node」

媒体经常说"Bun 要颠覆 Node"——这是错的，且会导致错误的技术决策。Bun 的产品定位是"Node 加速器 + 工具链一体化"——它声明 100% Node 兼容，目标不是让 Node 消失，而是给现有 Node 用户一条更快的路径。这个定位让"切 Bun 的回退成本"接近 0：你的代码仍然是 JS / TS、仍然装 npm 包、仍然走 package.json——切回 Node 只是改 CI 配置一行。Deno 才是真正"颠覆 Node"的项目（URL 导入、TS 内置、不兼容 npm），但 Deno 的路线已经被市场否决了，2024 Deno 2.0 也回到了 npm 兼容。理解这个区别能帮你正确评估"切 Bun"的风险——它不是赌一个新生态，而是在熟悉的生态里换一个 runtime。

「Jarred 一人维护」风险评估错位

很多人评估 Bun 风险时把"Jarred 一人维护"等同于"Bun 是个人项目，不能用于生产"——这个评估错位。Bun 已经是 Oven Inc. 商业产品，有 ~10-15 人团队，YC + a16z 投资，全职在维护。"Jarred 主导核心架构"和"Bun 是个人项目"是两件事——Linux 也是 Linus 主导核心架构，但你不会说"Linux 是个人项目"。正确的风险评估是：(1) Oven 的商业化进展（决定 5 年存活率）；(2) Bun 1.0+ 之后 breaking change 历史（实际很少，比 Node 早期还稳）；(3) bus factor（如果 Jarred 离开会发生什么——这是真实风险，但和"个人项目"不是一回事）。

把 Deno 衰退误读为「Node 替代品都做不起来」

看到 Deno 2018 → 2024 没拉起生态、最后回到 npm 兼容，有人推论"Node 替代品都做不起来，Bun 也会失败"——这是错的因果归因。Deno 失败的具体原因是路线选择：「不兼容 npm」直接堵死了生态护城河。Bun 路线相反——「100% 兼容 npm」从根本上避开了 Deno 踩的坑。从生态学的角度看，Bun 不是"重做一个 Node"，而是"在 Node 生态里做一个更快的入口"——用户切 Bun 不需要放弃任何东西。这两个项目的成败应该独立评估。

忽视 Bun 0.x → 1.0 的稳定性断层

有些资料 / 博客是 Bun 0.x 时代写的——那个时候 Bun bug 很多、API 在变、Node 兼容率约 70%。如果你 2024-2026 看到这些资料还原封不动信，会得到过度悲观的判断。Bun 1.0（2023-09）是稳定性的重大断层——之后 API 不再 breaking、bug 数量大幅下降、Node 兼容率涨到 96%+。评估 Bun 时只看 1.0 之后的数据，0.x 时代的吐槽和现状基本无关。
