WebAssembly 深度学习 · wasm32 / WASI / Component Model

基于 webassembly.org + W3C 规范 + Bytecode Alliance 整理。回答三个问题：wasm 怎么走到今天（NaCl 2009 → asm.js 2013 → 2015 提案 → 2017 MVP → 2025 W3C 2.0 → 2026 Component Model 走向生产）、它和 JS 的关系（栈机器 ISA + 沙箱 + 多语言编译目标，跨边界调用与"序列化税"）、它如何走出浏览器（WASI + wasmtime/wasmer/wazero + Component Model + 与 Docker 的"快两个数量级"冷启动差距）。是 V8 + 浏览器渲染 主题之上的"跨语言性能通道"延伸。

受众 已学完 V8 / 浏览器渲染 / JS 的工程师 来源 webassembly.org / W3C / Bytecode Alliance / V8 团队 章节 13 章 / 7 阶段 状态 ✅ 13/13 章完成

分阶段目录

共 7 个阶段、13 章，按"WASM 平台演化"展开：是谁 → 机器模型 → 与宿主互操作 → 跨语言编译路径 → 高级提案 → WASI 与服务端 → Component Model。建议顺序阅读；浏览器前端工程师看 P1-P3 + P5；服务端 / 平台工程师看 P1-P3 + P6 + P7；想用 Rust / C++ 写 wasm 看 P1-P3 + P4.1。

P1 · WASM 是谁

✅ 2 章完成

wasm 17 年项目史（含前史 NaCl / asm.js）+ 同代人对照（NaCl / PNaCl / asm.js / WebGL2 compute / WebGPU）。回答"为什么 NaCl 死了、asm.js 死了，wasm 不仅活了还走出浏览器"——四大引擎联合 + 二进制格式 + 多语言 + 跨平台是设计决策的链条。

NaCl / PNaCl asm.js 2015 四大引擎联合 2025 WASM 2.0 WebGPU 互补

P2 · 机器模型与格式

✅ 2 章完成

wasm32 栈机器 ISA（指令在隐式操作数栈 push/pop） + 二进制 module 结构（magic / sections / 流式 parse） + load-time validation（不允许"运行时发现类型错"） + wat 文本格式（S-expression，与二进制 1:1） + 类型系统（数值 / 向量 / 引用类型 + 结构化控制流）。

栈机器 ISA 二进制 sections load-time validation wat 文本 引用类型 结构化控制流

P3 · 与宿主互操作

✅ 2 章完成

wasm 是嵌入式语言，必须和宿主交换数据。Linear Memory（连续 byte 数组 + 64KB 页 + memory.grow + Memory64）+ Tables（间接调用 / 函数指针替身）+ Imports/Exports（JS-WASM 两条调用路径性能不对称）+ 跨边界数据交换"序列化税"（wasm-bindgen 模式）。

Linear Memory Memory64 Tables / call_indirect JS-WASM 跨边界 wasm-bindgen externref

P4 · 跨语言编译路径

✅ 2 章完成

wasm 最大差异化是多语言。系统语言（Rust + wasm-pack/wasm-bindgen 黄金组合 / C++ via Emscripten / Zig 原生支持）+ 托管语言 + 其他（Go 官方 vs TinyGo / AssemblyScript / Java/Kotlin/Dart/.NET 借 WasmGC 翻身 / Pyodide 把 CPython 编译到 wasm）。

Rust + wasm-bindgen Emscripten TinyGo WasmGC 翻身 Kotlin/Wasm Pyodide

P5 · 高级特性提案

✅ 2 章完成

wasm 2.0+ 关键演化。Threads + SIMD（基于 SAB + 必须 cross-origin isolated；v128 跨 CPU 抽象）+ WasmGC + Tail call + Exception handling + JSPI（让托管语言一等公民、函数式语言可生产部署、C++ 异常原样编译、同步 wasm 调异步 JS）。

Threads + SAB SIMD v128 WasmGC Tail call Exception handling JSPI

P6 · WASI 与服务端运行时

✅ 2 章完成

wasm 走出浏览器是它最大的"超出预期"。WASI（Preview 1 模仿 POSIX 失败 → Preview 2 转向 capability-based + WIT）+ 服务端 runtime 家族（wasmtime 事实标准 / wasmer 多后端 / wazero 纯 Go / WasmEdge AI 优化）+ wasm vs Docker（冷启动快两个数量级 + capability 细粒度安全）。

WASI Preview 1 vs 2 capability-based wasmtime wazero 纯 Go vs Docker Cloudflare Workers

P7 · Component Model 与典型应用

✅ 1 章完成

wasm 2026 终极形态。Component Model + WIT（让 wasm 成为"跨语言的二进制库格式" —— Rust 库被 Go/Java/Python 当作 wasm component 引用，不需 FFI/CGO/JNI）+ 典型应用（Figma/Photoshop Web/Google Earth/FFmpeg/SQLite/DuckDB/Pyodide/Shopify Functions/Envoy 插件/区块链合约）。

Component Model WIT canonical ABI Figma / Photoshop SQLite / DuckDB Shopify Functions

编写约定

每章遵循统一的 6 段结构：速览 → 底层逻辑 → 应用场景 → 典型代码 → 常见陷阱 → 延伸阅读。本主题相比 V8 / 浏览器渲染有以下额外约定：

  • 「短重述 + 链回」 —— V8 的 wasm 实现细节（Liftoff / TurboFan / WasmGC 引擎实现 / JSPI / Memory64）都在 V8 P6 讲透；本主题用 1-3 句话点到本质，紧跟一句"详细的 X 请参考 Y"
  • 每个特性都要标"提案 phase + 引擎实现状态" —— wasm 提案分 Phase 0-5；Phase 4 才能浏览器默认开启，Phase 5 才进入正式标准。每章涉及提案必标 phase + V8 / SpiderMonkey / JSC / wasmtime / wasmer 状态
  • webassembly.org / W3C 一手优先 —— 每个机制的"权威定义"必须来自 wasm 工作组规范文档；社区文章只用来佐证或补充
  • 关注「机器模型 vs 实际执行」差距 —— spec 定义抽象栈机器，但实际引擎用 register-based JIT 跑；这种差距对性能心智很重要

与既有主题的关系

本主题不重复 V8 / 浏览器渲染 / JS / ECMA 主题已经讲透的底层机制，但会在每个交叉点提供"短重述 + 链接"。常见对应：

  • P2.1 wasm 编译流水线 → V8 P6.1 V8 Wasm 编译流水线（Liftoff / TurboFan / Tier-up）
  • P3.1 Linear Memory → JS JavaScript 主题（ArrayBuffer / TypedArray）
  • P5.1 Threads + SAB → 浏览器渲染 P2.2 Site Isolation + COOP/COEP + ECMA P7.5 §24 内存模型
  • P5.2 WasmGC + JSPI → V8 P6.1 V8 Wasm 实现（引擎细节）+ V8 P4.2 Orinoco GC（被 WasmGC 复用）
  • P6 服务端 runtime → V8 P5.1 Embedder API（V8 isolate 也能跑 wasm，与 wasmtime 是两种部署模型）+ Bun（Zig 实现栈对照）
  • P7 典型应用 → React（前端框架与 wasm 的协作）
