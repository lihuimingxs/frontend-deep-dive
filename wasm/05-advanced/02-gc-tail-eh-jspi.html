WasmGC + Tail call + Exception handling + JSPI

让 wasm 适配更多语言场景的 4 个关键提案。WasmGC（phase 5, 2023-12 正式）引入 struct / array / i31ref + subtyping，让 Java/Kotlin/Dart/.NET/OCaml 等托管语言不必带 GC runtime。Tail call（phase 5, 2024 正式）让函数式语言（OCaml/Scala/Scheme）的尾递归可正确编译，不需要 trampoline 模拟。Exception handling（phase 4，浏览器 default 开启 2023+）让 C++ / Java / JS 异常在 wasm 内可表达。JSPI（phase 4, Chrome 137+）让同步 wasm 调用异步 JS，底层用 stack switching 实现——把同步 IO 假设的代码（如 SQLite C 代码）原样跑在浏览器。

阅读时长 ~30 分钟 前置 P3（host interop）+ P4.2（托管语言） 主要来源 WebAssembly proposals · v8.dev/blog/wasm-gc · v8.dev/blog/jspi · spidermonkey blog

01 速览

  1. WasmGC 是 wasm 历史"用户群扩展最大"的单个特性—— 2023 之前 Java/Kotlin/Dart 编译 wasm 都是工程奇迹（带 JVM/runtime 太重）；2023 后这些语言的 wasm 启动从"秒级"变"毫秒级"。Kotlin/Wasm + Compose Multiplatform / Flutter Web wasm / .NET Blazor NativeAOT 都基于 WasmGC。
  2. WasmGC 不是"通用 GC API"—— spec 不暴露 mark/sweep/compact 控制；引擎自由实现（V8 用现有 Orinoco / SpiderMonkey 用自己的 / JSC 用自己的）。开发者只能用 struct.new / array.new / ref.cast 等高层操作。WasmGC 对象与 linear memory 完全隔离。
  3. Tail call 让函数式语言进入 wasm—— return_call / return_call_indirect。OCaml / Scala / Scheme / 等之前要用 trampoline 模拟尾调（性能差），有 tail call 后达到 native 性能。Phase 5 完成 2024。
  4. Exception handling 的 "two-phase" 设计—— try / catch 的查找阶段 + unwind 阶段分离；和 C++ EH 模型一致；让现有 C++ EH 代码可直接编译。EH 之前的"模拟"方案（Emscripten 用 setjmp/longjmp + global trampoline）性能差，已被 EH 替代。
  5. JSPI 让 SQLite C 代码原样跑浏览器—— 之前 SQLite 的同步 read/write 假设要 Asyncify hack 转换；JSPI 后 SQLite C 代码看起来同步，底层 await Promise。Chrome 137+ 默认开启。底层用 stack switching：wasm 栈被保存，主线程让出，promise 完成后栈恢复。
  6. 这 4 个提案共同主题：让"现实世界语言"无损映射到 wasm—— Java 要 GC、OCaml 要 tail call、C++ 要异常、SQLite 要同步 IO 假设。每个提案都消除一个特定阻塞，让现有代码"原样跑 wasm"成为可能。

02 底层逻辑

WasmGC：从"自带 runtime"到"宿主管理"

没有 WasmGC 时托管语言两种方案：

方案 A · 自带 GC runtime（如 Go 官方 / 早期 Java TeaVM）—— 在 linear memory 上自己实现 mark / sweep；编译产物大，慢

方案 B · 把 JVM 编译进 wasm（如 CheerpJ）—— 极度笨重，启动 5+ 秒

WasmGC（phase 5, 2023-12 正式）方案 C：spec 直接支持 GC managed 类型，引擎 GC 接管：

(type $Person (struct
  (field $name (ref $String))
  (field $age i32)
  (field $email (ref null $String))))   ;; nullable

(type $List (array (mut (ref $Person))))

(func $create (param $n (ref $String)) (param $a i32) (result (ref $Person))
  (struct.new $Person
    (local.get $n)
    (local.get $a)
    (ref.null $String)))

关键属性：

(1) GC 类型与 linear memory 完全分离—— struct / array 实例不在 linear memory，由引擎 GC 堆管理

(2) 引擎自由实现—— V8 用 Orinoco（详见 V8 P4.2）；SpiderMonkey 用自己的；JSC 用自己的

(3) 跨语言 GC 共享—— Java / Kotlin / Dart 三个 wasm module 共享同一引擎 GC，不会重复扫描

(4) 类型安全—— 编译时检查 struct / array 字段访问；越界 / 错类型 trap

WasmGC 类型层级

WasmGC 引入新引用类型层级：

anyref     ← 顶部，能持有任何 GC managed 引用
  ├── eqref       ← 可 == 比较
  │     ├── i31ref     ← 31-bit 整数（小整数 unboxed 优化）
  │     ├── structref  ← struct 实例
  │     └── arrayref   ← array 实例
  ├── funcref     ← 函数引用（也在 anyref 下）
  └── externref   ← 外部宿主对象（JS 对象）

i31ref 是性能优化关键—— 让小整数（< 2^31）能 unboxed（不分配 heap object），与 V8 SMI 类似（详见 V8 P4.1）。Java 的 Integer / Kotlin 的 Int? 不会触发 GC alloc。

WasmGC 关键指令

struct.new $T arg1 arg2 ...           ;; 创建 struct 实例
struct.get $T $field obj               ;; 读字段
struct.set $T $field obj val           ;; 写字段（mutable 字段）

array.new $T elem n                    ;; 创建 array 长度 n，每个元素 elem
array.new_default $T n                 ;; 默认值数组
array.new_fixed $T n e1 e2 ...         ;; 固定长度数组
array.get $T arr i                     ;; 读元素
array.set $T arr i val                 ;; 写元素
array.len arr                          ;; 长度

ref.null $T                            ;; null 引用
ref.is_null obj                        ;; 测 null
ref.cast $T obj                        ;; 类型 down-cast；失败 trap
ref.test $T obj                        ;; 类型测试，返回 i32

i31.new x                              ;; 创建 i31ref
i31.get_s / i31.get_u                  ;; 读 i31 值（signed / unsigned）

WasmGC subtyping

WasmGC 支持 nominal subtyping（类似 Java extends）：

(type $Animal (sub (struct (field $name (ref $String)))))
(type $Dog (sub $Animal (struct
  (field $name (ref $String))
  (field $breed (ref $String)))))

;; Dog 是 Animal 的子类
;; (ref $Dog) <: (ref $Animal)

ref.cast / ref.test 在运行时检查实际类型 → Java/Kotlin 的 instanceof / cast 直接编译到这。

WasmGC 性能

典型场景对比（Java Hello World）：

方案体积启动稳态性能
原生 JVM50MB+200ms100%
TeaVM 旧（带自家 GC）5-10MB1-2s~30%
WasmGC（V8）~500KB-1MB~100ms~70%

引擎 GC 处理 wasm 对象的开销 ~2x 自带 GC（因为已优化 + 没重复扫描）。

Tail call 提案

函数式语言（OCaml / Scala / Scheme / Haskell）大量用尾递归表达循环。原始 wasm 没 tail call 优化—— 每次递归占用栈帧，栈溢出风险。

没 tail call 时的 trampoline 模拟（慢）：

;; OCaml fact 编译时：
;; fact n acc = if n = 0 then acc else fact (n-1) (acc*n)

;; trampoline 包装：函数返回 "continue" 或 "done"
(func $fact_step (param $n i32) (param $acc i32) (result i32 i32)   ;; (continue?, value)
  ...)

(func $fact (param $n i32) (param $acc i32) (result i32)
  (loop ...))   ;; loop 反复调 fact_step 直到 done

Tail call 提案（phase 5, 2024 正式）：

return_call $fact (i32.sub (local.get $n) (i32.const 1))
                  (i32.mul (local.get $acc) (local.get $n)))
;; 等价于：先 pop 当前栈帧，再调 $fact

关键：return_call 在调用 callee 之前先 pop caller 栈帧，避免栈深度增长。

另有 return_call_indirect（间接尾调用）。

引擎实现：编译器把 return_call 转 jmp（不是 call）。简单且快。

受益语言：OCaml / Scala（Scala.js Wasm target）/ Scheme / SML / 等。Rust / C++ 也偶尔受益（手写 tail recursion）。

Exception Handling 提案

原始 wasm 没 exception。C++ / Java / JS 异常要在 wasm 里实现，早期方案：

Emscripten setjmp/longjmp 模拟—— 用全局 jmp_buf + setjmp 标记 try block；longjmp 跳回。慢且占内存

EH 提案（phase 4，2023+ 浏览器默认开启）：

(tag $MyException (param i32))     ;; 定义异常类型

(func $caller
  (try (result i32)
    (do
      (call $maybe_throw)
      (i32.const 0))     ;; 正常返回
    (catch $MyException
      ;; 栈上是异常参数 i32
      (i32.const -1))))  ;; 异常处理结果

(func $maybe_throw
  (throw $MyException (i32.const 42)))

关键指令：

  • tag $T—— 声明异常 tag（类似 Java exception class）
  • throw $T arg—— 抛异常
  • try ... catch $T ... end—— 捕获
  • rethrow—— 重抛
  • delegate N—— 委托给外层 try

Two-phase EH—— spec 定义两阶段：

(1) Search phase—— 异常抛出，引擎遍历调用栈找匹配 catch；不实际 unwind

(2) Unwind phase—— 找到 handler 后，实际清理栈帧（运行 destructor 等）

这与 C++ Itanium EH ABI 一致，让 LLVM 后端容易输出。

性能：

  • 正常路径（不抛异常）：零开销（"zero-cost EH"）
  • 抛异常路径：栈搜索 + 清理 ~µs 级

异常应该是异常情况，不是控制流；正常路径快是设计目标。

JSPI（JS Promise Integration）

典型问题：你把 SQLite C 代码编译成 wasm。SQLite 内部用同步 IO（read / write 同步）。但浏览器的 IndexedDB / fetch / OPFS 是异步。怎么办？

原始方案 · Asyncify（Emscripten）—— 编译时把所有可能 await 的函数转成 state machine + 全局栈保存恢复。性能损失 ~10-30%；体积 +50%。

JSPI 方案（phase 4, Chrome 137+ 默认）：

// 把异步 JS 函数包成"看起来同步"的 wasm import
const fetchSync = WebAssembly.promising(async (urlPtr, urlLen) => {
  const url = decodeUrlFromWasm(urlPtr, urlLen);
  const resp = await fetch(url);
  const text = await resp.text();
  return encodeStringToWasm(text);
});

const wasmInstance = await WebAssembly.instantiate(wasmBytes, {
  env: { fetch_sync: fetchSync }
});

// wasm 内部调 fetch_sync 是"同步"语义，底层 await
wasmInstance.exports.process();  // 这个调用会暂停 wasm 栈，等 Promise，恢复

底层用 stack switching：

(1) wasm 调 fetch_sync

(2) 引擎检测它是 promising-wrapped，保存当前 wasm 栈到 Promise.then 回调

(3) 当前 task 让出，主线程继续

(4) Promise 完成，引擎在新 task 中恢复 wasm 栈

(5) wasm 看起来 fetch_sync 直接返回了

对 wasm 代码完全透明 —— SQLite C 代码以为它在做同步 IO。

JSPI 的实现复杂性

"保存 / 恢复 wasm 栈"听起来简单，实际需要引擎层支持：

  • V8 实现：把 wasm 栈保存到 separate stack（不是主 V8 栈）；详见 V8 P6.1
  • SpiderMonkey 实现：类似 stack switching 机制
  • JSC 实现：计划中

限制：

  • 只对包装过的 import 生效（必须用 WebAssembly.promising）
  • 主线程频繁 stack switching 会让 UI 卡顿（因为每次 switch 让出 main task）；适合"重型同步代码"，不适合每秒上千次 small await
  • 调试栈跟踪复杂

JSPI vs Asyncify 性能

方案编译成本运行时性能体积
无 async（手写）低100%baseline
Asyncify+编译复杂~70-90%+50%
JSPI低~95-100%baseline

JSPI 是明显优胜方案，但需要 Chrome 137+ / 等其他引擎跟进；Asyncify 仍是"兼容 fallback"。

提案解决问题主要受益语言Phase

03 应用场景

场景 A · Kotlin 写 wasm SPA（WasmGC + EH）

Kotlin/Wasm 全面用 WasmGC 管理对象 + EH 处理异常：

@JsExport
class TodoList {
    private val items = mutableListOf<String>()

    fun add(item: String) {
        items.add(item)
    }

    fun remove(index: Int) {
        if (index !in items.indices) throw IndexOutOfBoundsException("Invalid: $index")
        items.removeAt(index)
    }

    fun toJsArray(): Array<String> = items.toTypedArray()
}

编译产物用 WasmGC 表示 List 内部存储 + 用 EH 处理 IndexOutOfBoundsException。JS 端：

const list = new TodoList();
list.add("buy milk");
list.add("clean house");
try {
  list.remove(99);
} catch (e) {
  console.error("error:", e);  // Kotlin exception 自动转 JS Error
}

场景 B · OCaml 编译到 wasm（Tail call）

OCaml 5.2+ 支持 wasm target，借 tail call 让递归代码 native 性能：

(* lib.ml *)
let rec sum acc n =
  if n = 0 then acc
  else sum (acc + n) (n - 1)
;;

let total = sum 0 1_000_000  (* 不会栈溢出 *)

编译用 wasm32 target；产物用 return_call 实现 sum 的尾递归。

场景 C · SQLite C 代码 + JSPI 跑在浏览器

SQLite WASM 的 OPFS 异步版本用 JSPI（替代旧的 Asyncify）：

// JSPI 包装 OPFS 异步 IO 为同步导入
const opfs = await navigator.storage.getDirectory();
const fileHandle = await opfs.getFileHandle('db.sqlite');
const accessHandle = await fileHandle.createSyncAccessHandle();

const sqliteImports = {
  env: {
    fs_read: WebAssembly.promising(async (fd, buf_ptr, len) => {
      const buf = new Uint8Array(memory.buffer, buf_ptr, len);
      return accessHandle.read(buf, { at: ... });
    }),
    fs_write: WebAssembly.promising(async (fd, buf_ptr, len) => {
      const buf = new Uint8Array(memory.buffer, buf_ptr, len);
      return accessHandle.write(buf, { at: ... });
    })
  }
};

const sqlite3Wasm = await WebAssembly.instantiateStreaming(
  fetch('sqlite3-jspi.wasm'),
  sqliteImports
);

// SQLite C 代码假设同步 IO
const db = new sqlite3Wasm.exports.Db('mydb.db');
db.exec('CREATE TABLE users (id INTEGER, name TEXT);');
db.exec("INSERT INTO users VALUES (1, 'alice');");

SQLite 内部 read/write 看起来同步 → JSPI 让它们底层 await OPFS。SQLite 源码不改一行。

场景 D · C++ 异常处理在 wasm 里

#include <stdexcept>

extern "C" int divide(int a, int b) {
    if (b == 0) throw std::runtime_error("division by zero");
    return a / b;
}

用 EH 编译：

$ /opt/wasi-sdk/bin/clang++ -O3 -fwasm-exceptions divide.cpp -o divide.wasm

JS 端：

try {
  instance.exports.divide(10, 0);
} catch (e) {
  // C++ exception 转成 WebAssembly.Exception
  console.error("got exception:", e);
}

EH 让 C++ 异常和 JS try/catch 互通。

场景 E · Dart Flutter Web（WasmGC + EH）

Flutter Web 2024 起的 wasm 部署同时用 WasmGC（Dart 对象）+ EH（Dart 异常）+ Skia/CanvasKit：

$ flutter build web --wasm

产物 ~2-5MB（含 Flutter framework + Skia）；启动 ~500ms-1s。性能与 Flutter desktop 接近。

04 典型代码：手写 wat 用提案特性

例 1：WasmGC struct 操作（手写 wat）

(module
  (type $Point (struct (field $x f64) (field $y f64)))

  (func (export "create_point") (param $x f64) (param $y f64) (result (ref $Point))
    (struct.new $Point (local.get $x) (local.get $y)))

  (func (export "distance") (param $p1 (ref $Point)) (param $p2 (ref $Point)) (result f64)
    (local $dx f64)
    (local $dy f64)
    (local.set $dx
      (f64.sub
        (struct.get $Point $x (local.get $p1))
        (struct.get $Point $x (local.get $p2))))
    (local.set $dy
      (f64.sub
        (struct.get $Point $y (local.get $p1))
        (struct.get $Point $y (local.get $p2))))
    (f64.sqrt
      (f64.add
        (f64.mul (local.get $dx) (local.get $dx))
        (f64.mul (local.get $dy) (local.get $dy))))))

例 2：tail call factorial

(module
  (func $fact_helper (param $n i32) (param $acc i32) (result i32)
    (if (result i32) (i32.eq (local.get $n) (i32.const 0))
      (then (local.get $acc))
      (else
        (return_call $fact_helper
          (i32.sub (local.get $n) (i32.const 1))
          (i32.mul (local.get $acc) (local.get $n))))))

  (func (export "factorial") (param $n i32) (result i32)
    (call $fact_helper (local.get $n) (i32.const 1))))

用 return_call 让递归不增长栈深；factorial(1_000_000) 不溢出（虽然 i32 溢出 :)）。

例 3：Exception handling

(module
  (tag $DivByZero (param i32))

  (func $div (param $a i32) (param $b i32) (result i32)
    (if (i32.eqz (local.get $b))
      (then
        (throw $DivByZero (local.get $a))))
    (i32.div_s (local.get $a) (local.get $b)))

  (func (export "safe_div") (param $a i32) (param $b i32) (result i32)
    (try (result i32)
      (do
        (call $div (local.get $a) (local.get $b)))
      (catch $DivByZero
        (drop)   ;; 弹掉异常参数
        (i32.const -1)))))   ;; 返回 -1 表示错误

例 4：JSPI 包装异步 fetch

// 把 fetch 包成"同步" wasm import
const fetchSync = WebAssembly.promising(async (urlPtr, urlLen) => {
  const url = readStringFromWasm(urlPtr, urlLen);
  const resp = await fetch(url);
  const text = await resp.text();
  return writeStringToWasm(text);
});

const { instance } = await WebAssembly.instantiate(wasmBytes, {
  env: { fetch_sync: fetchSync }
});

// wasm 内部 fetch_sync 看起来同步
const result = instance.exports.fetch_and_process("/api/data");
// 实际：wasm 暂停 → fetch await → wasm 恢复 → 返回

例 5：Kotlin/Wasm 利用 WasmGC + EH 完整示例

import kotlin.js.JsExport

@JsExport
class Stack<T> {
    private val items = mutableListOf<T>()

    fun push(item: T) { items.add(item) }

    fun pop(): T {
        if (items.isEmpty()) throw IllegalStateException("Stack is empty")
        return items.removeAt(items.size - 1)
    }

    fun peek(): T? = items.lastOrNull()

    val size: Int get() = items.size
}

构建产物用 WasmGC 表示 ArrayList 内部存储 + EH 处理 IllegalStateException。

JS 端：

const stack = new Stack();
stack.push("a");
stack.push("b");
console.log(stack.pop());  // "b"
console.log(stack.size);   // 1
try {
  stack.pop();
  stack.pop();   // throws IllegalStateException
} catch (e) {
  console.error("Error:", e.message);
}

05 常见陷阱

陷阱 1：以为 WasmGC 是"通用 GC API"

WasmGC不暴露 mark/sweep/compact 控制；不能强制 GC（没有 WebAssembly.gc() 这种 API）；不能查看堆使用。引擎实现自由。

开发者只能用 struct.new / array.new / ref.cast 这些类型操作。GC 行为黑盒。

陷阱 2：混用 linear memory 和 WasmGC

linear memory 和 WasmGC 堆完全分离。一个 wasm module 可以同时用两者（C++ 部分 + Java 部分），但互相不能 deref：

  • 不能把 struct 的 field 写到 linear memory（要先序列化）
  • 不能把 linear memory 的指针 cast 成 struct 引用

这是设计上的隔离—— GC 不扫描 linear memory，避免性能问题。

陷阱 3：tail call 不是所有递归都能用

return_call 只在函数尾部有效（最后一条指令）。中间的 call（不在 tail position）不能用 return_call 替换。

;; ❌ 错：不在尾部
(call $foo)
(call $bar)   ;; 这条 call 在 foo 之后，不能 return_call

;; ✅ 正：在尾部
(call $foo)
(return_call $bar)

陷阱 4：Exception handling 部署门槛

EH phase 4，浏览器默认开启 2023+；但：

  • 旧 Safari（< 16.4）不支持
  • wasmtime 0.40+ 支持，但旧版本不行
  • 编译器需要显式启用 -fwasm-exceptions（Emscripten / clang）

fallback 仍要 setjmp/longjmp 或 Asyncify 模拟。

陷阱 5：JSPI 主线程频繁 await 性能差

每次 JSPI 调用都让出 main task；频繁 await 让 UI 任务排队，UI 看起来卡。

规则：

  • JSPI 适合"重型同步代码 + 偶发 IO"（SQLite 类）
  • 不适合"循环每次小 await"（这种用普通 async/await 更好）

陷阱 6：WasmGC 跨 module 共享

WasmGC 对象引用可以跨 wasm instance 共享（如果用同一 GC 堆，即同一 V8 isolate / 同一进程）。但不能跨 worker / 跨进程—— GC 堆不同。

跨 worker 传 GC 对象要序列化（structuredClone）或不直接传，用 SAB + 手工 marshalling。

陷阱 7：i31ref 范围有限

i31ref 只能存 31-bit signed 整数（-2^30 ~ 2^30 - 1）。超过范围要用 (ref $Box) 包装。Java Integer.MAX_VALUE（2^31 - 1）不在 i31ref 范围内，会 alloc box。

陷阱 8：忘记 ref.cast 失败 trap

;; ref.cast 失败 trap（运行时错误）
(ref.cast $Dog (local.get $animal))   ;; 如果 animal 不是 Dog 子类型 → trap

;; 安全方案：先 ref.test
(if (ref.test $Dog (local.get $animal))
  (then
    ;; 安全 cast
    (ref.cast $Dog (local.get $animal))
    ...))
