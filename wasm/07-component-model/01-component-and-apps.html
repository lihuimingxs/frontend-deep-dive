Component Model + WIT + 典型应用

WebAssembly 2026 的终极形态在 Component Model + WIT —— 让 wasm 真正成为"跨语言的二进制库格式"。Rust 写组件 A → 编译 wasm component → 通过 WIT 接口被 Go 写的组件 B 调用，不需要 C ABI / FFI / CGO / JNI。WIT（WebAssembly Interface Types）是 IDL，类似 protobuf / thrift 但 wasm 原生；world 概念把一组 imports + exports 打包成"环境契约"。canonical ABI 定义跨组件传 string / list / record 的标准。典型应用：浏览器内（Figma / Photoshop Web / Google Earth / FFmpeg / SQLite / DuckDB / Pyodide）+ 服务端（Shopify Functions / Envoy plugins）+ 区块链（Polkadot / Internet Computer / NEAR）三大市场覆盖了 wasm 真正的用户群。

阅读时长 ~30 分钟 前置 P6（WASI + runtime） 主要来源 github.com/WebAssembly/component-model · bytecodealliance.org · 各应用官方博客

01 速览

  1. Component Model 是 wasm core 之上的封装层—— 加上"接口、类型、链接"概念。core module 只有 i32/i64/f32/f64/v128/funcref/externref；component 有 string / list / record / variant / option / result 等高级类型。
  2. WIT 是 IDL（接口定义语言）—— 类似 protobuf / thrift，但wasm 原生。WIT 文件定义函数签名 + 类型；wit-bindgen 工具生成各语言 binding（Rust / Go / JS / Python / Java / 等）。
  3. world = imports + exports = wasm 想要的环境契约—— 一个 component 可声明"我需要 wasi:filesystem + wasi:http"，宿主提供这些 worlds 的实现。WIT 让"WASI 兼容性" = "WIT 接口兼容性"。
  4. 跨语言互操作的黄金路径—— Rust 写组件 A（暴露 WIT 接口）→ 编译 wasm component → 通过 WIT 接口被 Go 写的组件 B 调用。不需要 C ABI / FFI / CGO / JNI。这是 Component Model 的野心。
  5. wasm 杀手级应用 3 类—— (1) 浏览器内既有大代码库（Figma / Photoshop / SQLite / FFmpeg）；(2) 服务端 / Edge 多租户（Cloudflare / Fastly / Shopify）；(3) 区块链智能合约（Polkadot / NEAR / Internet Computer）。三类共同特点：需要安全沙箱 + 高性能 + 跨语言。
  6. wasm 不擅长的—— (1) 频繁调 DOM（跨边界税）；(2) 短小工具（启动 + glue 体积）；(3) 现有 JS 库已 mature 的场景（不要为了 wasm 而 wasm）。2026 趋势：wasm 不取代 JS，是性能敏感模块的 5%。

02 底层逻辑

为什么需要 Component Model

core wasm（MVP + 2.0）的限制：

(1) 类型贫乏—— 只有 i32/i64/f32/f64/v128 + ref types。要传 string / list / struct，必须手工序列化到 linear memory（详见 P3.2 序列化税）。

(2) 跨语言互操作困难—— Rust 写的 wasm 与 C 写的 wasm 互调，必须达成 ABI 约定（如 wasm-bindgen 的 Rust 特定 marshalling，C 没有）。

(3) 没有"组件"概念—— core wasm 是 module；多个 module 互调要靠手工 linker（wasm-merge 等工具）。

(4) 接口契约不正式—— WASI Preview 1 是一组 imports 名称；不是 IDL，不能 typecheck。

Component Model 解决：

  • 更丰富类型（string / list / record / variant / option / result）
  • 跨语言 canonical ABI（任何语言遵循 = 互通）
  • 组件链接（多个 component 自动 link）
  • WIT IDL（typecheck + 生成各语言 binding）

Component vs Module 区别

Module（core wasm）：

(module
  (import "env" "log" (func (param i32)))
  (func (export "main") ...))

imports / exports 类型受限于 wasm 数值类型 + memory + table。

Component：

(component
  (import "wasi:logging/logger" (func (param "level" string) (param "message" string)))
  (export "main" (func)))

imports / exports 用 WIT 类型（string 等高级类型）；底层多个 core module 组成，外层是 component wrapper。

Component 二进制结构：\0asm magic + version （和 core 不同，有 layer 标志） + nested core modules + component-level type definitions。

WIT 类型系统

WIT 类型分几层：

原始类型

bool, s8, s16, s32, s64, u8, u16, u32, u64, f32, f64, char, string

容器类型

list<T>              // 动态数组
option<T>            // 可选值（None / Some(T)）
result<T, E>         // 错误处理（Ok(T) / Err(E)）
tuple<T1, T2, ...>   // 元组

用户定义类型

// record（类似 struct）
record point {
  x: f64,
  y: f64,
}

// variant（类似 enum / sum type）
variant http-status {
  ok,
  not-found,
  error(string),
}

// enum（无 payload variant）
enum direction {
  north, south, east, west,
}

// flags（位标志集合）
flags permissions {
  read, write, execute,
}

资源（resource，类似面向对象）

resource counter {
  constructor(initial: u32);
  increment: func() -> u32;
  get: func() -> u32;
}

WIT world 概念

world 把一组 imports + exports 打包成环境契约：

// hello.wit
package example:hello@0.1.0;

world hello-world {
  // imports：宿主必须提供
  import wasi:cli/environment@0.2.0;
  import wasi:filesystem/preopens@0.2.0;
  import wasi:io/streams@0.2.0;

  // exports：本组件提供
  export run: func() -> result<_, string>;
  export greet: func(name: string) -> string;
}

组件实现这个 world → 任何宿主提供这些 imports → 跑得起来。这是"WASI 标准化"的真正机制。

Canonical ABI

Component Model 定义 canonical ABI—— 跨语言 component 互通的标准 ABI。

问题：不同语言对 string / list / record 的内存表示不一样：

  • Rust String：长度 + capacity + ptr 三字段
  • C++ std::string：通常 SSO 优化（小字符串内联）
  • Java String：UTF-16
  • JS String：UTF-16

canonical ABI 规定：

(1) string 是 UTF-8 字节序列 + 长度—— 跨组件传 string 一律按这个布局

(2) list<T> 是 ptr + len—— linear memory 中的连续元素

(3) record 字段顺序 + 对齐—— 与 C struct 类似但更严格

(4) variant tag + payload—— tag 在前，payload 紧随

每个语言 wit-bindgen 生成的 marshalling 代码都遵循这个 ABI，让跨语言互调透明。

跨语言互操作的黄金路径

设想你想用 Rust 写一个"图像滤镜"库，让 Go / Java / Python 都能调用。

没 Component Model 时：

  • Rust → C ABI → Go 调（用 CGO，部署痛）
  • Rust → JNI → Java 调（手写 native 代码）
  • Rust → CFFI → Python 调（手写 ctypes）
  • 每个语言都要手写不同的 binding

Component Model 时代：

  • Rust 写组件，定义 WIT 接口
  • cargo component build → image-filter.wasm（一个 wasm component）
  • Go：用 wit-bindgen-go 生成 Go binding，import 这个 component
  • Java：用 wit-bindgen-java 生成 Java binding
  • Python：用 wit-bindgen-python 生成 Python binding
  • 同一个 wasm 二进制服务所有语言

这是 Component Model 的野心：让 wasm 成为"跨语言的二进制库格式"，类似 jar 之于 JVM 语言、so/dll 之于系统语言。

shared-nothing linking

Component Model 默认 shared-nothing linking—— 组件之间不共享 memory。每个组件有自己的 linear memory 和 GC 堆。

跨组件通信通过 ABI marshalling：传 string / list 时，从源组件的 memory 拷贝到目标组件的 memory。

好处：

  • 强隔离—— 一个组件 crash / OOM 不影响其他
  • 独立内存上限—— 每组件单独限制
  • 语言混合无障碍—— Rust 用 manual memory, Java 用 GC，互不影响

代价：

  • 跨组件传大数据有拷贝开销（但 wasm 内存拷贝快）
  • 不能共享对象引用（必须传值）

典型应用 3 类

类别 1：浏览器内既有大代码库

共同特点：原本是 desktop / native app（C/C++ 写，几百万行代码），通过 wasm 移植到浏览器。

  • Figma（2017 上线 wasm 版）—— C++ 布局引擎；性能接近 native；wasm 处理 CRDT / 矢量计算 / 文本布局；JS 处理 React UI / 网络 / 协作。wasm 设计的"教科书案例"。
  • Photoshop Web（Adobe 2023 上线）—— 完整 Photoshop C++ 代码库（数百万行）+ Emscripten 编译；用 wasm 跑滤镜引擎、用 WebGPU 加速渲染。
  • Google Earth Web（C++ 3D 引擎）—— 把 desktop Google Earth 移植到浏览器。
  • FFmpeg.wasm—— 视频转码 / 格式转换；浏览器内本地处理避免上传。多线程版需要 cross-origin isolated。
  • SQLite WASM（官方支持）—— 浏览器内有真正的 SQL 数据库；OPFS 持久化；jSPI 让 SQLite C 代码原样跑。
  • DuckDB-WASM—— 浏览器内 OLAP 数据库；SQL aggregation / join 在客户端跑；适合数据分析仪表板。
  • Pyodide—— CPython + numpy/scipy/pandas 编译 wasm；JupyterLite 浏览器内 Jupyter。
  • AutoDesk Tinkercad（CAD Web）—— C++ 几何引擎 → wasm。
  • Roblox Studio Web（部分功能）—— Lua VM + 客户端 wasm。

类别 2：服务端 / Edge 多租户

共同特点：需要安全沙箱 + 快冷启动 + 跨语言。Docker 太重，wasm 刚好。

  • Cloudflare Workers（V8 isolate 跑 wasm）—— 每天调用百亿级；JS 主导，部分用户 Rust+wasm。
  • Fastly Compute@Edge（wasmtime）—— Rust / Go (TinyGo) / AssemblyScript；冷启动 < 1ms。
  • Vercel Edge Functions（V8 isolate）—— 类似 Cloudflare 模型。
  • Shopify Functions（Component Model 早期生产）—— 商家用 Rust/JS 写 wasm 插件影响商品定价 / 折扣 / 订单逻辑。
  • Envoy Wasm Plugins（wazero 嵌入）—— 服务网格的扩展点；用户写 wasm filter，Envoy 在每请求跑。
  • Tetrate Service Bridge（多租户 service mesh）。
  • Spin (Fermyon)—— wasm 应用框架；本地开发 + Fermyon Cloud 部署。
  • Database Functions—— 一些数据库（如 SingleStore / Aurora）开始支持 wasm UDF（用户定义函数）。

类别 3：区块链智能合约

共同特点：需要确定性执行 + 跨语言 + 沙箱。早期区块链用自家 VM（EVM / Solana BPF），新一代普遍 wasm 化。

  • Polkadot / Substrate—— wasm 是 runtime 升级机制（链可以改逻辑而不分叉）；智能合约用 ink!（Rust 框架）。
  • NEAR Protocol—— 智能合约 wasm，编译自 Rust / AssemblyScript；NEAR 早期推 AS 给前端工程师。
  • Internet Computer (DFINITY)—— canister = wasm 实例；每个 canister 独立 actor；可持久化。
  • CosmWasm（Cosmos 生态）—— Cosmos SDK 的智能合约扩展。
  • Fluence / Wasmer Edge—— 去中心化计算平台。
  • Ethereum ewasm（实验性，未正式上线）—— EVM 的 wasm 替代品，进展慢。

wasm 不擅长的

三类场景不要用 wasm：

(1) 频繁调 DOM—— wasm 调 DOM 必须经 JS 桥（跨边界税 ~20ns/call）。频繁 DOM 操作（rAF 内大量 setText）反而比纯 JS 慢。Rust SPA 框架（Yew/Leptos）通过 wasm 内 VDOM 计算 + 批量 DOM 更新缓解，但仍不如 React 顺。

(2) 短小工具—— 启动开销（compile + instantiate ~50-100ms）+ glue 体积（150KB+）让小工具用 wasm 不划算。10 行计算 / 小数据处理用 JS 直接做。

(3) 现有 JS 库已 mature—— 不要为了 wasm 而 wasm。lodash / date-fns / immutable.js 等通用库 JS 实现已极致；用 wasm 重写体积大、维护贵、性能不一定快。

规则：wasm 是性能敏感模块的 5%，不是替代 JS 的"主流"。

wasm 未来：3 个方向

方向 1：Component Model 走向生产—— 2024 Preview 2 / 2026 Preview 3（async）/ 2027+ 跨语言互操作真正普及。Bytecode Alliance 在用 Cloudflare / Fastly / Shopify 等大客户驱动。

方向 2：浏览器内 wasm 与 WebGPU 融合—— 大型应用（Figma/Photoshop）已是 wasm + WebGPU 双引擎；onnxruntime-web 类似。未来 wasm 是 CPU 性能通道，WebGPU 是并行加速通道。

方向 3：服务端 wasm 与容器互补—— wasm 不取代 Docker，但占据 "lambda / edge function / 多租户插件" 市场。K8s + runwasi 让 wasm workload 像普通 container 调度。2030 预期：5-15% 服务端 workload 是 wasm。

03 应用场景

场景 A · 用 Rust 写 Component 让 Go / JS 调用

1. 写 WIT 定义：

// hash.wit
package example:hash@0.1.0;

world hash {
  export sha256: func(input: list<u8>) -> list<u8>;
}

2. Rust 实现（cargo-component）：

// src/lib.rs
mod bindings;
use bindings::Guest;
use sha2::{Sha256, Digest};

struct Component;

impl Guest for Component {
    fn sha256(input: Vec<u8>) -> Vec<u8> {
        let mut hasher = Sha256::new();
        hasher.update(&input);
        hasher.finalize().to_vec()
    }
}

bindings::export!(Component with_types_in bindings);

3. 构建：

$ cargo component build --release
# 产 hash.component.wasm

4. Go 调用（wasmtime-go SDK + wit-bindgen-go）：

// Go 端通过 wit-bindgen-go 生成的 binding 调用
hash := example.NewHash(componentBytes)
result := hash.Sha256([]byte("hello"))
fmt.Printf("%x", result)

5. JS 调用（jco transpile）：

$ jco transpile hash.component.wasm -o hash-js
# 产生 hash-js/ 目录，含 JS module + glue
import { sha256 } from './hash-js/hash.js';
const hash = sha256(new TextEncoder().encode("hello"));
console.log(hash);

场景 C · Shopify Functions 写商品折扣插件

use shopify_function::prelude::*;

#[shopify_function]
fn function(input: input::ResponseData) -> Result<output::FunctionResult> {
    let cart = input.cart;
    let discounts = if cart.cost.subtotal_amount.amount > "100".parse()? {
        vec![output::Discount {
            value: output::Value::Percentage(output::Percentage { value: dec!(10) }),
            targets: vec![output::Target::OrderSubtotal(output::OrderSubtotalTarget {
                excluded_variant_ids: vec![],
            })],
            message: Some("10% off orders over $100".to_string()),
        }]
    } else { vec![] };

    Ok(output::FunctionResult {
        discounts,
        discount_application_strategy: output::DiscountApplicationStrategy::First,
    })
}

商家在 Shopify Admin 上传 wasm，每次结账时 Shopify 调用 wasm 决定折扣。Capability：function 只看到自己被传入的 cart，不能访问其他商家数据。

场景 E · 用 Polkadot ink! 写智能合约

#![cfg_attr(not(feature = "std"), no_std, no_main)]

#[ink::contract]
mod counter {
    #[ink(storage)]
    pub struct Counter { value: u32 }

    impl Counter {
        #[ink(constructor)]
        pub fn new() -> Self { Self { value: 0 } }

        #[ink(message)]
        pub fn increment(&mut self) { self.value += 1; }

        #[ink(message)]

    }
}
$ cargo contract build

$ cargo contract upload --suri //Alice
$ cargo contract instantiate --suri //Alice
$ cargo contract call --message increment --suri //Alice

04 典型代码：完整 Component 项目

项目：image filter component（Rust 写，Go + JS 都能用）

1. WIT 定义：

// wit/world.wit
package example:image-filter@0.1.0;

interface filter {
  variant filter-type {
    grayscale,
    invert,
    blur(u32),
    threshold(u8),
  }

  record image-data {
    width: u32,
    height: u32,
    pixels: list<u8>,
  }

  apply: func(input: image-data, kind: filter-type) -> image-data;
}

world image-filter {
  export filter;
}

2. Rust 实现：

// src/lib.rs
mod bindings;

use bindings::exports::example::image_filter::filter::{
    Guest, ImageData, FilterType
};

struct Component;

impl Guest for Component {
    fn apply(input: ImageData, kind: FilterType) -> ImageData {
        let mut pixels = input.pixels.clone();
        match kind {
            FilterType::Grayscale => {
                for chunk in pixels.chunks_exact_mut(4) {
                    let r = chunk[0] as u32;
                    let g = chunk[1] as u32;
                    let b = chunk[2] as u32;
                    let gray = ((r * 299 + g * 587 + b * 114) / 1000) as u8;
                    chunk[0] = gray; chunk[1] = gray; chunk[2] = gray;
                }
            }
            FilterType::Invert => {
                for chunk in pixels.chunks_exact_mut(4) {
                    chunk[0] = 255 - chunk[0];
                    chunk[1] = 255 - chunk[1];
                    chunk[2] = 255 - chunk[2];
                }
            }
            FilterType::Threshold(t) => {
                for chunk in pixels.chunks_exact_mut(4) {
                    let avg = (chunk[0] as u32 + chunk[1] as u32 + chunk[2] as u32) / 3;
                    let v = if avg > t as u32 { 255 } else { 0 };
                    chunk[0] = v; chunk[1] = v; chunk[2] = v;
                }
            }
            FilterType::Blur(_radius) => { /* ... 简化省略 ... */ }
        }
        ImageData {
            width: input.width,
            height: input.height,
            pixels,
        }
    }
}

bindings::export!(Component with_types_in bindings);

3. 构建：

$ cargo component build --release
$ ls target/wasm32-wasip2/release/*.wasm
target/wasm32-wasip2/release/image_filter.wasm

4. JS 端用 jco：

$ jco transpile target/wasm32-wasip2/release/image_filter.wasm -o image-filter-js
$ ls image-filter-js
filter.js
image_filter.core.wasm
image_filter.d.ts
package.json
import { filter } from './image-filter-js/image_filter.js';

const imageData = ctx.getImageData(0, 0, w, h);
const result = filter.apply(
  { width: w, height: h, pixels: imageData.data },
  { tag: 'grayscale' }
);

const newImageData = new ImageData(
  new Uint8ClampedArray(result.pixels),
  result.width,
  result.height
);
ctx.putImageData(newImageData, 0, 0);

5. Go 端用 wit-bindgen-go：

$ wit-bindgen-go generate --out-dir gen wit/world.wit
// 用生成的 binding
import "example.com/myapp/gen"

result := imagefilter.Apply(input, imagefilter.FilterTypeGrayscale)

同一个 wasm component 服务 JS / Go / 任何能用 wit-bindgen 的语言。这是 Component Model 的目标。

05 常见陷阱

陷阱 1：以为 Component = 编译好的 wasm

core wasm module 和 component 是不同二进制格式。普通 cargo build --target wasm32-wasip1 产 module；cargo component build --target wasm32-wasip2 产 component。不能混用。

检查产物：

$ wasm-tools print my.wasm | head -1
(module ...)        # core module
# 或
(component ...)     # component

陷阱 2：Component Model 工具链不稳

2026-05 cargo-component / wit-bindgen / jco / wasm-tools 仍在快速演进。API 可能 break，每隔几个月需要更新工具版本。

生产用 Component Model 要：

  • 锁定工具版本（Cargo.toml + .tool-versions）
  • 跟踪 spec 进度（github.com/WebAssembly/component-model）
  • 有 Preview 1 fallback 计划

陷阱 3：把 Component 直接给浏览器

浏览器原生还不支持 Component（只支持 core module）。要在浏览器跑，需 jco transpile：

$ jco transpile my.component.wasm -o my-js
# 产 JS module + core wasm，浏览器能加载

2026 浏览器原生 Component 支持还在路上。

陷阱 4：以为 wasm 取代所有 JS

反复强调：wasm 不是 JS 替代。2026 浏览器代码量 95%+ 仍 JS / TS。wasm 是性能敏感模块的 5%。把 wasm 看成"高性能补丁"，不是"下一代 JS"。

陷阱 5：跨组件传大数据用值传递

Component Model 默认 shared-nothing，跨组件传 list<u8> / string 都拷贝。大数据（> MB）拷贝开销不小。

方案：

  • 设计 API 让组件自己读 / 写共享存储（filesystem / OPFS / S3 等）
  • 用 wasi:io/streams 流式传输（避免一次拷贝整块）
  • 等待 spec 引入 shared memory linking（实验中）

陷阱 6：误用资源（resource）

WIT 的 resource 类似面向对象 class，但跨组件边界传递时是 handle（不是值）。源组件持有真实数据，目标组件持有 handle，调用时 RPC。频繁跨组件方法调用不便宜。

陷阱 7：高估 Component Model 普及

2026 现实：

  • Cloudflare Workers 仍主要 Preview 1（迁移中）
  • Fastly Compute@Edge 早期支持
  • Shopify Functions / Spin 已用
  • 大部分生产 wasm 仍 Preview 1

Component Model潜力大但实际部署慢。新项目可以押注；现有项目按需迁移。

陷阱 8：不评估 wasm 适用性就引入

常见错误：看到 wasm 流行就引入。但 wasm 工具链复杂、调试难、体积大、跨边界税。评估清单：

(1) 真有性能瓶颈吗？JS Profile 看清

(2) 有现成 wasm 库吗？（FFmpeg.wasm / SQLite WASM 等）→ 直接用

(3) 是计算密集（不是 IO 密集）吗？wasm 才有意义

(4) 团队有 Rust / C++ 能力吗？工具链投入大

(5) 体积 / 启动时间预算够吗？

5 个都"是"才考虑 wasm。否则 JS 实现更划算。
