二进制格式与栈机器执行模型

wasm32 是栈机器（不是寄存器机器）：所有指令在隐式操作数栈上 push / pop。为什么这样设计—— (1) 二进制紧凑（不编寄存器号）、(2) 验证简单（typing 推导）、(3) 引擎自由（实际 V8 / wasmtime 都把它再编译成寄存器机器）。二进制 module = magic + version + 12 类 sections（每个 LEB128 + length prefix），让streaming compilation成为可能。load-time validation 是 wasm 安全的基石——module 整个验证通过才执行，要么直接拒绝。

阅读时长 ~30 分钟 前置 P1（项目史 + 同代人） 主要来源 WebAssembly Core Spec · WABT · v8.dev/blog

01 速览

  1. wasm32 是栈机器—— 指令隐式从操作数栈取参数、压结果。i32.const 5; i32.const 3; i32.add 把 5 / 3 push 后 add 弹出两值相加压回 8。栈机器的紧凑性 + 易验证是 wasm 设计核心。
  2. 二进制 module 自描述、流式可读—— magic \0asm + version + 12 类 sections，每个 section 有 length prefix 和 LEB128 编码。引擎能边下载边 parse 边验证，不必等整个 module 到位。
  3. load-time validation 是 wasm 安全模型核心—— wasm 不允许"先 load 后逐渐发现类型错误"。每条指令的栈类型变化是确定的，逐条推导即可验证整个函数。失败的 module 直接拒绝，没有"运行到一半发现 bug"。
  4. 结构化控制流—— 没有任意 goto；只有 block / loop / if / br / br_if / return。这让验证 + 编译器优化都简单。NaCl 的 SFI 验证器要处理任意 jump，wasm 不需要。
  5. 抽象栈机器 vs 实际寄存器执行—— spec 定义抽象栈，但实际引擎（V8 / wasmtime）把它重编译成 register-based JIT 跑。性能心智不能基于"栈操作多慢"。
  6. 双层管道（V8 模型）—— Liftoff（baseline，1:1 翻译，~5 µs/KB）+ TurboFan（optimizing，~50 µs/KB）+ Tier-up；wasmtime 用 Cranelift 一次编译到位。详见 V8 P6.1。

02 底层逻辑

栈机器 vs 寄存器机器：为什么 wasm 选栈

计算机执行模型有两种：

寄存器机器（x86 / ARM / RISC-V / JVM 后端）—— 指令显式指定寄存器编号（add r0, r1, r2）；CPU 自然映射；操作快但二进制要编寄存器号

栈机器（JVM 字节码 / .NET CIL / wasm）—— 指令操作隐式栈；push 5; push 3; add；二进制紧凑但 CPU 不直接映射

wasm 选栈机器的 4 个理由：

(1) 二进制紧凑—— 寄存器机器每条指令至少要编 1-3 个寄存器号；栈机器只要操作码。wasm 的 i32.add 是 1 字节（0x6a）。同样的"两数相加"，wasm 比 x86 紧凑 2-3x

(2) 类型推导简单—— 栈机器每条指令的栈变化是确定的：i32.add 弹两个 i32、压一个 i32。逐条推导即可知道函数任意点的栈类型；寄存器机器要 dataflow 分析才能推断类型

(3) 引擎自由—— 抽象栈机器不约束执行模型。V8 / SpiderMonkey / wasmtime 都把栈机器再编译成 register-based JIT，根据 CPU 寄存器分配；不同引擎可以采用不同策略

(4) 历史经验—— JVM / .NET 都用栈机器，证明可行。wasm 借鉴这两个的设计经验

栈机器的代价：

抽象栈机器在 CPU 上不能直接跑（CPU 是寄存器机器）。所以引擎必须做一次编译（栈 → 寄存器），不能像 x86 ELF 那样直接执行。这是 wasm 启动需要"compile"步骤的根因。

wasm 指令集结构

wasm 指令分类（MVP + 2.0）：

(1) 数值常量—— i32.const N / i64.const N / f32.const N / f64.const N；把常数压栈

(2) 局部变量访问—— local.get $i（读到栈）/ local.set $i（从栈写）/ local.tee $i（写但不弹）

(3) 全局变量访问—— global.get $i / global.set $i

(4) 算术 / 比较 / 位操作—— i32.add / i32.sub / i32.mul / i32.div_s / i32.eq / i32.lt_s / i32.and / i32.or / i32.shl / 等

(5) 类型转换—— i32.wrap_i64 / i64.extend_i32_s / f32.convert_i32_s / 等

(6) 内存访问—— i32.load / i32.store（带对齐 + 偏移参数）；从 linear memory 读 / 写

(7) 控制流—— block / loop / if / else / end / br / br_if / br_table / return / call / call_indirect

(8) 引用类型操作（2.0 正式）—— ref.null / ref.is_null / ref.func

(9) SIMD（2.0 正式，v128）—— v128.load / v128.store / i32x4.add / f32x4.mul / 等 200+ 指令

(10) 表 / 内存批量操作（2.0）—— memory.copy / memory.fill / table.get / table.set

总指令数：MVP 约 170 条；2.0 含 SIMD 约 400+；提案中含 GC / EH 约 500+。比 x86（数千条）小一个数量级。

二进制 module 结构

wasm 二进制 module 由 magic header + version + sections 组成：

magic:    [0x00 0x61 0x73 0x6D]   ← "\0asm"
version:  [0x01 0x00 0x00 0x00]   ← 当前是 1（包括 wasm 1.0 + 2.0）

然后是 N 个 sections，每个：
  section_id:  1 byte（0-12）
  section_size: LEB128 编码的字节数
  section_body: section_size 字节

12 类 sections（按通常顺序）：

ID名称内容
0Custom自定义元数据（如 name section / debug info）；引擎忽略未识别的
1Type函数签名列表（(i32, i32) -> i32 等）
2Import导入的函数 / memory / table / global 声明
3Function函数到 type 的索引映射（声明每个 function 用哪个 type）
4Table表声明（funcref / externref 数组）
5Memory内存声明（初始页 + max 页 + shared 标志）
6Global全局变量声明 + 初始值
7Export导出列表（function / memory / table / global 名 → 索引）
8Startmodule 加载后自动调用的函数（可选）
9Element初始化 table 内容（funcref 列表）
10Code函数体（local 声明 + 字节码）—— 占 90%+ 体积
11Data初始化 memory 内容（byte segments）
12DataCountdata segments 数量（用于 bulk memory ops 验证）

section 顺序约定：除 Custom 外，其他 sections 必须按 ID 升序出现。这让流式 parse 有确定顺序。

LEB128 编码（变长整数）

wasm 大量使用 LEB128（Little-Endian Base-128）编码整数。LEB128 是变长编码：小数用 1-2 字节，大数用更多字节。

规则：每字节 7 bit 数据 + 1 bit "继续标志"（高位）。最后一字节高位为 0，前面字节高位为 1。

例子（unsigned）：

     0 → 0x00                  （1 字节）
   127 → 0x7F                   （1 字节）
   128 → 0x80 0x01              （2 字节）
 16383 → 0xFF 0x7F              （2 字节）
 16384 → 0x80 0x80 0x01         （3 字节）
624485 → 0xE5 0x8E 0x26         （3 字节）

带符号 LEB128用类似规则但最后一字节最高数据位是符号位。i32.const -1 编码为 0x7F（1 字节）。

为什么用 LEB128：

  • 大多数偏移 / 索引 / 函数 ID 是小数（< 128），1 字节；体积优化
  • 不需要预先声明字段大小（自描述）
  • parse 简单

代价：编码 / 解码慢于定长（要逐字节判断），但 wasm parse 不是性能瓶颈。

典型 module 二进制 dump

最简 wasm module（导出一个 add(i32, i32) -> i32 函数）：

wat 源：

(module
  (func $add (param $x i32) (param $y i32) (result i32)
    local.get $x
    local.get $y
    i32.add)
  (export "add" (func $add)))

对应二进制（41 字节）：

00 61 73 6D                 ; magic "\0asm"
01 00 00 00                 ; version 1

01 07                       ; Type section, 7 bytes
01                          ; 1 type
60 02 7F 7F 01 7F           ; func type: (i32, i32) -> (i32)
                            ;   60 = func；02 = 2 params；7F = i32；
                            ;   01 = 1 result；7F = i32

03 02                       ; Function section, 2 bytes
01 00                       ; 1 function；type index 0

07 07                       ; Export section, 7 bytes
01                          ; 1 export
03 61 64 64                 ; "add"（length=3 + chars）
00 00                       ; export kind 0 (func)；func index 0

0A 09                       ; Code section, 9 bytes
01                          ; 1 function
07                          ; func body length 7
00                          ; 0 local declarations
20 00                       ; local.get 0
20 01                       ; local.get 1
6A                          ; i32.add
0B                          ; end

整个 module 41 字节。同样功能 asm.js ~150 字节、x86 ELF 几 KB。

load-time validation：wasm 安全模型核心

问题：wasm 模块来自不可信源（用户上传 / 第三方），如何保证它不会破坏宿主？

NaCl 方案：SFI 在每条指令插入运行时检查；性能损失大、验证器复杂

JVM 方案：load 时部分验证 + lazy verification（首次执行某 method 才完整验证）；造成"运行时发现 bug"

wasm 方案：load 时完整验证，验证通过才能执行；失败的 module 直接拒绝

wasm validation 包括：

(1) Module structure—— sections 顺序 / 索引边界 / 类型一致

(2) 每个函数的类型推导—— 逐条指令推导栈类型；任何点栈类型必须确定

(3) 内存 / table 边界—— 偏移 + 大小不能越界（编译时 + 运行时双重检查）

(4) 控制流嵌套—— block / loop / if 配对；br 索引必须在合法范围

关键属性：验证一次，永远安全。运行时不会发现"原来类型错了"或"原来栈不平衡"。

为什么 wasm validation 简单：

  • 栈机器：栈类型变化确定，逐条推导即可
  • 结构化控制流：没有任意 goto，只有嵌套 block / loop / if
  • 强类型：每条指令期望/产生确定类型
  • 没有"动态加载代码"：module 内容固定，不允许 self-modifying

Streaming compilation：边下载边编译

因为 sections 有 length prefix，引擎能不等整个 module 下载完就开始 parse + 编译。

典型流程：

(1) 浏览器开始 fetch wasm

(2) 收到 magic + Type section + Function section → 引擎开始 baseline 编译第一批函数

(3) 边下载边编译 → 当 module 下载完时，大部分函数已经编译好

(4) instantiate → 链接 imports，几乎瞬时

API：WebAssembly.compileStreaming(fetch(url)) / WebAssembly.instantiateStreaming(fetch(url), imports)。

对比："非流式" WebAssembly.instantiate(arrayBuffer) 必须等整个 module 下载完才能开始编译。

双层管道（V8 模型）

V8 的 wasm 实现采用双层管道（详见 V8 P6.1，这里短重述）：

(1) Liftoff（baseline）—— 一次扫描，每条 wasm 指令直接生成 native 指令；不做寄存器分配 / inlining；编译速度极快（~5 µs/KB），可以与下载并行。

(2) TurboFan（optimizing）—— 复杂 IR + 寄存器分配 + inlining + 死代码消除；编译慢（~50 µs/KB）但代码质量高。

(3) Tier-up—— 函数热度高时从 Liftoff 升级到 TurboFan；类似 JS 的 Ignition → TurboFan。

SpiderMonkey 类似（WasmBaseline + Ion）；JSC 也类似（BBQ + OMG）。wasmtime 不分双层，直接用 Cranelift 一次编译到位（适合服务端，因为可 AOT 缓存）。

抽象栈 vs 实际寄存器执行

spec 定义抽象栈机器，但 V8 / wasmtime 实际把它编译成 register-based。

例子：

wasm:
  local.get 0
  local.get 1
  i32.add
  local.set 2

抽象栈机器执行：
  栈：[]
  push x0  → 栈: [x0]
  push x1  → 栈: [x0, x1]
  add      → 栈: [x0+x1]
  pop → x2 → 栈: []

V8 编译后（x86_64 简化）：
  mov  rax, [rsp+0]    ; load local 0
  mov  rbx, [rsp+8]    ; load local 1
  add  eax, ebx        ; eax = eax + ebx
  mov  [rsp+16], rax   ; store local 2

或更优（如果 locals 已在寄存器）：
  add  eax, ebx        ; 直接寄存器运算

所以"栈操作"在抽象层；真实执行不会真的 push / pop。

性能心智的影响：

  • 不要担心"local.get / local.set 多用是否慢" —— 编译器会消除冗余
  • 不要担心"用栈顶值 vs 用 local 是否快" —— 都会变成寄存器操作
  • 真正影响性能的：函数大小（影响 inlining）/ 控制流复杂度（影响优化）/ 内存访问模式（影响 cache）/ SIMD 使用 / 类型选择（i32 vs i64）

03 应用场景

场景 A · 调试 wasm：从二进制还原 wat

开发中遇到 .wasm 文件想看里面是什么，用 WABT (WebAssembly Binary Toolkit)：

$ npm install -g wabt
$ wasm2wat my.wasm -o my.wat
$ cat my.wat
(module
  (func $add (param i32 i32) (result i32)
    local.get 0
    local.get 1
    i32.add)
  (export "add" (func $add)))

反向（写 wat 学习）：

$ wat2wasm my.wat -o my.wasm

生产中：wasm-tools print my.wasm（bytecodealliance 工具，更现代）。

场景 B · 优化 wasm 体积

Rust + wasm-pack 默认输出 hello world ~150KB。关键优化：

(1) panic = "abort" —— Cargo.toml 中设置；省略 panic unwinding 代码（~30KB）

(2) lto = "fat" —— Link Time Optimization；跨 crate 内联（~20% 体积）

(3) codegen-units = 1 —— 单线程编译让 LTO 更彻底

(4) opt-level = "z" 或 "s" —— 优化体积（z 极致 / s 平衡）

(5) 跑 wasm-opt -O3（binaryen）—— Binaryen 后处理，再压 30-40%

(6) wasm-strip —— 去 debug symbols + custom sections

典型 Cargo.toml：

[profile.release]
opt-level = "z"
lto = "fat"
codegen-units = 1
panic = "abort"
strip = true

压缩工具链 + 简单 hello world 后能压到 ~10KB。

场景 C · 在 Worker 里 streaming compile + 不阻塞主线程

// 主线程
const worker = new Worker('wasm-worker.js');
worker.postMessage({ url: '/big-module.wasm' });
worker.onmessage = (e) => {
  console.log('result:', e.data.result);
};

// wasm-worker.js
self.onmessage = async ({ data: { url } }) => {
  // streamingInstantiate 不阻塞主线程
  const { instance } = await WebAssembly.instantiateStreaming(fetch(url));
  const result = instance.exports.compute();
  self.postMessage({ result });
};

为什么用 streamingInstantiate 而不是先 fetch 再 instantiate：

  • streamingInstantiate 边下载边编译，wallclock 时间短
  • 非 streaming 必须等下载完才能开始编译，浪费 download 与 compile 的并行
  • 大 module（5MB+）能省 30-50% 启动时间

场景 D · 用 wasm-tools 检查 module 合规性

# 验证 module 符合 spec
$ wasm-tools validate my.wasm

# 查看 module 的 sections
$ wasm-tools dump my.wasm
   0x00 | 00 61 73 6d | version 1 (Module)
   0x04 | 01 00 00 00 |
   0x08 | 01 07       | type section
   ...

# 反汇编（更详细）
$ wasm-tools print --no-strip my.wasm

04 典型代码：手写最小 wasm

纯字节数组手写一个 add module

不用任何工具链，直接写出二进制：

// 41 字节的 wasm，导出 add(i32, i32) -> i32
const wasmBytes = new Uint8Array([
  // magic + version
  0x00, 0x61, 0x73, 0x6D, 0x01, 0x00, 0x00, 0x00,

  // Type section: id=1, size=7
  0x01, 0x07,
  0x01,                        // 1 type
  0x60, 0x02, 0x7F, 0x7F, 0x01, 0x7F,  // func: (i32, i32) -> (i32)

  // Function section: id=3, size=2
  0x03, 0x02,
  0x01, 0x00,                  // 1 function；type index 0

  // Export section: id=7, size=7
  0x07, 0x07,
  0x01,                        // 1 export
  0x03, 0x61, 0x64, 0x64,      // name "add"
  0x00, 0x00,                  // export kind=func；func index=0

  // Code section: id=10, size=9
  0x0A, 0x09,
  0x01,                        // 1 function
  0x07,                        // func body length 7
  0x00,                        // 0 local declarations
  0x20, 0x00,                  // local.get 0
  0x20, 0x01,                  // local.get 1
  0x6A,                        // i32.add
  0x0B                         // end
]);

const { instance } = await WebAssembly.instantiate(wasmBytes);
console.log(instance.exports.add(2, 3));  // 5

这是能跑的最小 wasm。理解每个字节的含义后，wasm 二进制不再神秘。

用 streaming API 加速大 module 加载

// 良好实践
async function loadLargeWasm(url, importObject) {
  // streamingInstantiate 比 instantiate 快 30-50%（大 module）
  return WebAssembly.instantiateStreaming(fetch(url), importObject);
}

// 反例（非 streaming）
async function slowLoad(url, importObject) {
  const response = await fetch(url);
  const buffer = await response.arrayBuffer();
  return WebAssembly.instantiate(buffer, importObject);  // 必须等下载完才编译
}

注意：fetch 必须返回 Content-Type: application/wasm，否则 streaming 不工作（浏览器会回退到非流式）。Nginx 配置：

types {
  application/wasm wasm;
}

module caching：编译一次、多处复用

// 编译 module（耗时一次）
const module = await WebAssembly.compileStreaming(fetch('/heavy.wasm'));

// 多次 instantiate（瞬时；imports 可不同）
const instance1 = await WebAssembly.instantiate(module, imports1);
const instance2 = await WebAssembly.instantiate(module, imports2);
const instance3 = await WebAssembly.instantiate(module, imports3);

// 跨 worker 共享 module（postMessage 传 module）
worker.postMessage({ module });  // module 是 transferable

关键：编译只做一次（昂贵，~10-100ms 中等 module），instantiate 多次（便宜，~1ms）。在多 worker / 多 instance 场景下能省大量启动时间。

05 常见陷阱

陷阱 1：以为"栈操作多 = 性能差"

抽象层是栈，真实执行是寄存器。local.get / local.set 编译后基本免费。

正确性能心智：函数大小（< 50 行更易 inline）+ 控制流（避免深嵌套）+ 内存访问局部性（避免随机访问 GB 级 memory）+ SIMD 使用。

陷阱 2：忘记设置 application/wasm MIME type

常见症状：WebAssembly.instantiateStreaming(fetch(url)) 报警 "incorrect Content-Type"，回退到非 streaming。后端没配置 .wasm 的 MIME type 是常见原因。

检查：curl -I https://your-site/foo.wasm 看 Content-Type 是否 application/wasm。

陷阱 3：手写 wasm 不验证就上线

wasm 二进制看似简单（41 字节小 module），但section 大小 / LEB128 / type index 任何错都让 module 拒绝。手写 wasm 必须经 wat2wasm 或 wasm-tools 验证：

$ wasm-tools validate my.wasm  # 不输出即通过；有错会指出位置

陷阱 4：以为 validation 失败 = bug 报告

wasm validation 失败是 spec 行为。如果 module 类型错 / 控制流不平衡 / 索引越界，引擎必须拒绝。这不是 V8 bug 是 module bug。

调试：先用 wasm-tools validate 离线验证；通过后再担心引擎差异。

陷阱 5：忽略 module 大小对启动的影响

"我的 wasm 才 5MB" —— 5MB 在 4G 网络上要 5+ 秒下载，移动设备是严重启动瓶颈。体积优化是 wasm 工程的第一关：

  • wasm-opt -Oz
  • strip debug symbols
  • code splitting（按需 instantiate 子模块）
  • Brotli 压缩（wasm 二进制 Brotli 通常压 70-80%）

陷阱 6：不用 streaming + 不缓存 module

典型反模式：每次访问页面都从 0 编译 wasm。HTTP cache + module caching 让二次访问几乎瞬时。

HTTP cache：设 Cache-Control: public, max-age=31536000, immutable（hash 文件名 + 永久缓存）。

引擎缓存：浏览器自动缓存编译后的 wasm（V8 Isolate cache）；二次 instantiate 跳过编译。
