wat 文本格式与类型系统

wat（WebAssembly Text format）是 wasm 的"汇编"——S-expression 语法 + 与二进制 1:1 对应。给人看 / 调试 / 学习 / 手写小模块用。类型系统分三层：(1) 数值类型 i32/i64/f32/f64（MVP）+ 向量类型 v128（2.0）+ 引用类型 funcref/externref（2.0）。结构化控制流（block/loop/if/br/br_if）让 wasm 区别于 NaCl 的任意 jump，是 load-time validation 简单的根基。multi-value（2.0）让多返回值不必通过 linear memory 中转。

阅读时长 ~30 分钟 前置 P2.1（栈机器 + 二进制格式） 主要来源 WebAssembly Core Spec · WABT · MDN wasm 文档

01 速览

  1. wat 是 wasm 的"汇编 + S-expression"—— 给人看的格式，与二进制 1:1 对应。生产部署的是 .wasm，wat 用于学习 / 调试 / 反汇编。开发者很少手写 wat（用 Rust / C++ 编译），但读懂 wat 是 debug 必备技能。
  2. 类型系统三层—— 数值（i32/i64/f32/f64）+ 向量（v128，SIMD）+ 引用（funcref/externref）。MVP 只有数值；2.0 引入向量和引用；GC 提案再加 struct/array/i31ref（详见 P5.2）。
  3. 结构化控制流—— 没有任意 goto，只有 block / loop / if-else / br / br_if / br_table。br N 跳到第 N 层 block 末尾，类似 break to label，但用相对深度。这种限制让 load-time validation 简单 + 编译器优化更激进。
  4. 函数签名 = 参数列表 → 返回列表—— MVP 时单返回值；2.0 multi-value 让多返回值一等公民。(func (param i32 i32) (result i32 i32) ...) 同时返回和差。
  5. 引用类型让 wasm 能持有 JS 对象—— MVP 时 wasm 只能存数值，要存 JS 对象要绕道 memory + 自己管 ID；externref（2.0）让 JS 对象作 opaque 引用直接传给 wasm，与 GC barrier 无缝集成。
  6. 类型检查是 load-time + 一次完成的—— 不允许"运行到某条指令才发现类型错"；module load 即验证完成。这种设计让 wasm sandbox 成本极低。

02 底层逻辑

wat 语法基础：S-expression

wat 用 S-expression（Lisp 风格）：(operator operand1 operand2 ...)。每个 expression 用括号包裹。

最简 module：

(module)

带一个函数的 module：

(module
  (func $hello))

函数加参数 / 返回值 / 局部变量：

(module
  (func $add (param $x i32) (param $y i32) (result i32)
    local.get $x
    local.get $y
    i32.add))

命名 vs 索引：$add / $x 是名字（仅 wat 用，二进制只有索引）。可以用索引代替：

(module
  (func (param i32) (param i32) (result i32)
    local.get 0  ;; 第一个参数
    local.get 1  ;; 第二个参数
    i32.add))

注释：;; 行注释、(; ... ;) 块注释。

两种语法风格：

线性风格（stack-based）：

local.get $x
local.get $y
i32.add

嵌套风格（folded / S-expression）：

(i32.add (local.get $x) (local.get $y))

两者等价；嵌套风格更接近"表达式树"，可读性高；线性风格更接近实际栈操作，调试时直观。混用也允许。

数值类型（MVP，4 种）

wasm MVP 只有 4 种数值类型：

类型含义大小对应 JS对应 Rust对应 C
i3232 位整数4 字节Number（< 2^32）i32 / u32int32_t / uint32_t
i6464 位整数8 字节BigInti64 / u64int64_t / uint64_t
f3232 位浮点（IEEE 754）4 字节Number（精度损失）f32float
f6464 位浮点（IEEE 754）8 字节Numberf64double

关键观察：

(1) 没有 sign / unsigned 类型区分—— wasm 只有 i32 / i64，签名性由指令决定。i32.div_s（signed divide）vs i32.div_u（unsigned divide）。让指令多但类型少。

(2) 没有 i8 / i16 / u8 / u16—— 所有小整数操作 zero / sign extend 到 i32。但内存读写有专门指令：i32.load8_s（读 1 字节符号扩展）/ i32.load8_u（无符号）/ i32.load16_s 等。

(3) i64 在 JS API 用 BigInt—— 因为 JS Number 只能精确表示 2^53；wasm i64 与 JS 互操作必须 BigInt（2.0 后默认）。

(4) bool 不是独立类型—— 用 i32（0=false / 非 0=true）。比较指令 i32.eq 等返回 i32 0/1。

向量类型 v128（2.0 正式）

v128 = 128 位向量，可解释为：

  • i8x16—— 16 个 8 位整数
  • i16x8—— 8 个 16 位整数
  • i32x4—— 4 个 32 位整数
  • i64x2—— 2 个 64 位整数
  • f32x4—— 4 个 32 位浮点
  • f64x2—— 2 个 64 位浮点

SIMD 指令操作 v128（详见 P5.1）。例如：

;; 加载 4 个 f32 → 同时相加
(v128.load (local.get $aPtr))    ;; 4 个 f32 from a
(v128.load (local.get $bPtr))    ;; 4 个 f32 from b
f32x4.add                        ;; 同时 4 路加法
(v128.store (local.get $cPtr))   ;; 同时存 4 个

v128 在 x86 映射到 SSE / AVX 寄存器；ARM 映射到 NEON 寄存器；都是免费性能。

引用类型 funcref / externref（2.0 正式）

MVP 时 wasm 只能存数值；引用类型让 wasm 能opaque 持有函数 / JS 对象。

funcref—— 函数引用；用于 table（间接调用）+ ref.func 指令

externref—— 外部引用（通常是 JS 对象）；wasm 不能 deref 它，但可以传递 / 存储 / 作为参数返回值

例子：

(module
  (func (export "store_obj") (param $obj externref) (result externref)
    local.get $obj   ;; 直接返回，wasm 不知道 obj 内部结构但能持有
  ))

JS 端：

const obj = { name: "alice" };
const result = instance.exports.store_obj(obj);
// result === obj （引用相等，wasm 透明传递）

这解决了 MVP 的"wasm 怎么持有 JS 对象"问题。之前要绕道：JS 端维护 ID → object Map，wasm 端只存 ID，用 ID 调 JS 函数操作对象。externref 让这一切自动化。

函数签名 + multi-value

函数签名 = 参数类型列表 → 返回类型列表。MVP 时只能 0 或 1 个返回值；multi-value（2.0）允许多返回值。

例子（multi-value）：

(module
  (func $divmod (param $a i32) (param $b i32) (result i32 i32)
    local.get $a
    local.get $b
    i32.div_s   ;; 商
    local.get $a
    local.get $b
    i32.rem_s)  ;; 余数
  (export "divmod" (func $divmod)))

JS 端调用：

const [quotient, remainder] = instance.exports.divmod(17, 5);
// quotient = 3, remainder = 2

multi-value 也用于 block—— block 不只能 push 一个结果，还能 push 多个：

(block (result i32 i32)
  i32.const 1
  i32.const 2)
;; 栈状态：[1, 2]

这让"返回 struct"可以不通过 linear memory 中转 —— 直接栈传递两个 / 多个值。

局部变量 + 参数

wasm 函数有局部变量（locals），包括参数（参数也算 locals，按声明顺序索引）+ 显式声明的 local。

(func $foo (param $a i32) (param $b i32) (local $tmp i32) (result i32)
  ;; locals: 0=$a, 1=$b, 2=$tmp
  local.get $a
  local.get $b
  i32.add
  local.set $tmp     ;; 存到 local 2

  local.get $tmp
  local.get $tmp
  i32.mul)           ;; tmp * tmp

local declarations 优化：wat 允许合并相同类型 locals：

(func (local i32 i32 i32 f64 f64)
  ;; 等价二进制（更紧凑）：
  ;; local count groups: [(3, i32), (2, f64)]
  )

编译器实际把 locals 映射到 native 寄存器；frequent locals 不会真的"进栈"。

结构化控制流

wasm 没有 goto。控制流由 5 类构造组成：

(1) block—— 引入一个嵌套层，可被 br 跳出（跳到 block 末尾）

(block $outer
  (block $inner
    br $inner   ;; 跳到 $inner 末尾
  )
  ;; $inner 块外
)

(2) loop—— 引入一个嵌套层，可被 br 回到 loop 开头（不是末尾！这是关键）

(loop $L
  ;; 计算
  br_if $L (some condition)  ;; 条件回到 loop 开头
)

(3) if-else—— 条件分支

(if (i32.lt_s (local.get $x) (i32.const 0))
  (then
    i32.const -1)
  (else
    i32.const 1))

(4) br / br_if / br_table—— 分支：

  • br N—— 无条件跳到第 N 层（外层）的 block 末尾或 loop 开头
  • br_if N—— 条件跳（栈顶非 0 时跳）
  • br_table—— switch 风格分发到多个目标

关键点：br N 是相对深度，不是绝对 label。br 0 跳最内层，br 1 跳外一层。这让验证简单。

(5) return / call / call_indirect—— 函数级控制流

没有 goto / 任意 jump 的好处：

  • load-time validation 可一次推导整个函数的栈类型
  • 编译器知道每个 block 的入口 / 出口，优化（DCE / inlining / SSA 转换）容易
  • 不会跳到指令中间（不像 x86 / ELF 可能 jump 到任意地址）

类型检查规则（验证算法）

wasm validation 是线性时间的（不像 JVM 的复杂 dataflow）。算法概要：

初始：函数入口栈类型 = 参数类型反序（最后一个参数在栈顶）

每条指令推导：

  • i32.const N—— 栈顶 push i32
  • i32.add—— 栈顶必须 [i32, i32]，弹出两个，push i32
  • local.get $i—— push local 类型
  • local.set $i—— 弹出，必须匹配 local 类型
  • call $f—— 弹出 $f 参数类型，push 返回类型
  • br N—— 检查目标 block 的"出口栈类型"匹配；标记当前点 unreachable（之后指令栈类型可任意，因为永远不会执行到）

函数末尾：栈类型必须 == 返回类型

失败例子：

(func (result i32)
  i32.const 1
  i64.const 2   ;; 栈：[i32, i64]
  i32.add)      ;; ❌ i32.add 期望 [i32, i32]，但栈是 [i32, i64]

这种 module 在 load 时就被拒绝，永远不会运行到这条指令。

导入 / 导出（imports / exports）

wat 中：

(module
  ;; 从 "env" 模块导入函数 "log"，签名 (i32) -> ()
  (import "env" "log" (func $log (param i32)))

  ;; 从 "env" 导入 memory（最少 1 页）
  (import "env" "mem" (memory 1))

  (func $main
    (call $log (i32.const 42)))

  (export "main" (func $main))
  (export "memory" (memory 0)))   ;; 也可以重新导出

JS 端 instantiate 时提供 imports：

const importObject = {
  env: {
    log: (n) => console.log("wasm logs:", n),
    mem: new WebAssembly.Memory({ initial: 1 })
  }
};
const { instance } = await WebAssembly.instantiate(wasmBytes, importObject);
instance.exports.main();  // 输出 "wasm logs: 42"

详见 P3.2。

tables + 间接调用

table 是函数引用数组，用 call_indirect 索引调用（动态分发的基础）：

(module
  (type $sig (func (param i32) (result i32)))
  (table 2 funcref)
  (elem (i32.const 0) $double $square)

  (func $double (param i32) (result i32)
    (i32.mul (local.get 0) (i32.const 2)))

  (func $square (param i32) (result i32)
    (i32.mul (local.get 0) (local.get 0)))

  (func (export "dispatch") (param $i i32) (param $x i32) (result i32)
    (call_indirect (type $sig)
      (local.get $x)
      (local.get $i))))

JS 端：

instance.exports.dispatch(0, 5);  // double(5) = 10
instance.exports.dispatch(1, 5);  // square(5) = 25

C/C++ 的 vtable / 函数指针 / 动态分发都编译到 table + call_indirect。详见 P3.1。

03 应用场景

场景 A · 调试 wasm 性能问题：读 wat

性能怀疑 Rust 编译的 wasm 没用 SIMD，或某个热函数没 inline。读 wat 是必备技能：

$ wasm-tools print my.wasm | grep -A 20 "func \$hot_function"

看是否有 v128 指令、call 是否被 inline、控制流是否合理。

典型 issue：

  • 编译没启用 SIMD：wat 里全是 i32.add / f32.add，没 i32x4 / f32x4
  • 没 inline：本应内联的小函数仍是 call
  • 多余 bounds check：每个 i32.load 前后插了运行时检查

场景 B · 手写小工具（gas counter / instrument）

有时需要给 wasm module 注入额外指令（如 fuel metering / coverage）。直接在二进制操作复杂；wat 操作直观：

$ wasm2wat my.wasm -o my.wat
# 编辑 my.wat，每个函数入口插入 (call $consume_gas (i32.const 1))
$ wat2wasm my.wat -o my-instrumented.wasm

场景 C · 教学 / 学习 wasm 设计

读 wat 是学 wasm 最快路径。比看 Rust 源码或二进制都直观：每条指令对栈做什么、控制流如何流转一目了然。

场景 D · 跨工具链 / 跨语言调试

你 Rust + wasm-bindgen 编了 wasm，但运行时报错"function signature mismatch"。读 wat 看 Rust 编译出的实际签名，对比 JS 端期望，定位差异。

04 典型代码：手写 wat 进阶

例 1：递归 fib

(module
  (func $fib (param $n i32) (result i32)
    (if (result i32) (i32.lt_s (local.get $n) (i32.const 2))
      (then
        local.get $n)
      (else
        (i32.add
          (call $fib (i32.sub (local.get $n) (i32.const 1)))
          (call $fib (i32.sub (local.get $n) (i32.const 2)))))))
  (export "fib" (func $fib)))

例 2：循环求和（loop + br_if）

(module
  (func $sum_to_n (param $n i32) (result i32)
    (local $i i32)
    (local $acc i32)
    (local.set $i (i32.const 1))
    (local.set $acc (i32.const 0))
    (loop $L
      ;; acc += i
      (local.set $acc (i32.add (local.get $acc) (local.get $i)))
      ;; i++
      (local.set $i (i32.add (local.get $i) (i32.const 1)))
      ;; 如果 i <= n 继续
      (br_if $L (i32.le_s (local.get $i) (local.get $n))))
    local.get $acc)
  (export "sum_to_n" (func $sum_to_n)))

例 3：multi-value 返回 quotient + remainder

(module
  (func $divmod (param $a i32) (param $b i32) (result i32 i32)
    ;; 同时返回商和余数
    local.get $a
    local.get $b
    i32.div_s
    local.get $a
    local.get $b
    i32.rem_s)
  (export "divmod" (func $divmod)))

JS 端：

const [q, r] = instance.exports.divmod(17, 5);
// q = 3, r = 2

例 4：用 externref 持有 JS 对象

(module
  (table $objs 10 externref)
  (func $store (param $i i32) (param $obj externref)
    (table.set $objs (local.get $i) (local.get $obj)))
  (func $load (param $i i32) (result externref)
    (table.get $objs (local.get $i)))
  (export "store" (func $store))
  (export "load" (func $load)))

JS 端用 wasm 作 typed object store：

const alice = { name: "alice" };
const bob = { name: "bob" };
instance.exports.store(0, alice);
instance.exports.store(1, bob);

const obj = instance.exports.load(0);
console.log(obj === alice);  // true

例 5：用 SIMD 加速向量加法

(module
  (memory (export "mem") 1)
  (func $add_vec_simd (param $aPtr i32) (param $bPtr i32) (param $cPtr i32) (param $n i32)
    (local $i i32)
    (local.set $i (i32.const 0))
    (loop $L
      ;; load 4 个 f32 from a
      ;; load 4 个 f32 from b
      ;; f32x4.add
      ;; store 4 个 f32 to c
      (v128.store
        (i32.add (local.get $cPtr) (local.get $i))
        (f32x4.add
          (v128.load (i32.add (local.get $aPtr) (local.get $i)))
          (v128.load (i32.add (local.get $bPtr) (local.get $i)))))
      ;; i += 16 (4 个 f32)
      (local.set $i (i32.add (local.get $i) (i32.const 16)))
      (br_if $L (i32.lt_s (local.get $i) (i32.shl (local.get $n) (i32.const 2)))))
  )
  (export "add_vec_simd" (func $add_vec_simd)))

05 常见陷阱

陷阱 1：以为 br 是跳到 label

wasm 的 br N 是相对深度，不是 label 名。

(block $A
  (block $B
    br 0   ;; 跳到 $B 末尾（最内层）
    br 1   ;; 跳到 $A 末尾（外一层）
  )
)

wat 允许用名字 br $A / br $B，编译时转成索引。但二进制只有索引。

陷阱 2：loop 的 br 跳的不是末尾

关键差别：

  • br 在 block 内 → 跳到 block 末尾（forward）
  • br 在 loop 内 → 跳到 loop 开头（backward，构成循环）

这是 wasm 表达循环的方式：用 loop + br_if 形成"条件跳回"。loop 自己不会自动循环，必须显式 br 回去。

陷阱 3：忘记栈平衡

每个 block / function 末尾，栈类型必须匹配 result 类型。多压一个值不弹会失败：

(func (result i32)
  i32.const 1
  i32.const 2)   ;; ❌ 栈：[i32, i32]，期望 [i32]

用 drop 弹值不消费：

(func (result i32)
  i32.const 1
  i32.const 2
  drop)          ;; ✅ 栈：[i32]，弹掉栈顶 2，剩 1

陷阱 4：i64 与 JS 互操作忘记 BigInt

wasm 函数返回 i64 时，JS 端必须用 BigInt 接收：

// wasm: (func (result i64) (i64.const 999999999999))
const result = instance.exports.big_num();
console.log(result);          // 999999999999n（BigInt）
console.log(typeof result);   // "bigint"
console.log(Number(result));  // 转 Number 可能精度损失

如果 wasm 函数 export 接受 i64 参数，JS 也必须传 BigInt：instance.exports.foo(123n)。传 Number 会 throw。

陷阱 5：手写 wat 忘了 export

新手常见 bug：函数定义了但没 export，JS 端 instance.exports.foo 是 undefined。

(module
  (func $hidden (param i32) (result i32)
    local.get 0))   ;; ❌ 没 export，外部访问不到

修复：

(module
  (func $visible (export "visible") (param i32) (result i32)
    local.get 0))   ;; ✅ inline export 语法

或：

(module
  (func $foo (param i32) (result i32) local.get 0)
  (export "foo" (func $foo)))

陷阱 6：以为 unsigned 类型是单独类型

wasm 没有 u32 / u64 类型，只有 i32 / i64。签名性由指令决定：

  • i32.div_s—— signed divide（-6/2 = -3）
  • i32.div_u—— unsigned divide（0xFFFFFFFE / 2 = 0x7FFFFFFF）
  • i32.lt_s vs i32.lt_u—— 同样区别
  • i32.shr_s（算术右移）vs i32.shr_u（逻辑右移）

编译器（Rust / C++）选择正确的指令；手写 wat 时要自己判断。

陷阱 7：滥用 multi-value

multi-value 看似方便（返回多值不通过 memory），但跨 JS-WASM 边界仍是开销：每个返回值要生成 JS array entry。返回 5+ 值时不如让 wasm 写到 linear memory，JS 端 read 一次。
