WASI Preview 1 / Preview 2 + Capability-based 安全

WASI = WebAssembly System Interface，让 wasm 在浏览器外（服务端 / Edge / 嵌入式）有"系统接口"——文件 / 网络 / 时钟 / 环境变量。Preview 1（2020-2022）POSIX 风格 imports（fd_read / fd_write / clock_time_get），广泛实现但有设计缺陷：POSIX 假设单语言可信进程，wasm 是多语言不可信沙箱。Preview 2（2024-01 正式）整体重构，转向 Component Model + WIT + capability-based 安全模型。Capability-based 让宿主显式授予能力（"只能读这目录"），比 Docker 的"Linux capabilities + namespace"细粒度。

阅读时长 ~30 分钟 前置 P3（host interop）+ P4（编译路径） 主要来源 wasi.dev · github.com/WebAssembly/WASI · bytecodealliance.org · hacks.mozilla.org WASI 系列

01 速览

  1. WASI 是 wasm 的"操作系统接口"—— 浏览器 wasm 通过 JS API 访问环境（fetch / DOM）；服务端 wasm 通过 WASI 访问 fs / net / clock / random / env。WASI 让 wasm 摆脱"必须在浏览器"的限制，是 wasm 走出浏览器的关键。
  2. Bytecode Alliance 是 WASI 主导组织—— Mozilla / Fastly / Intel / Red Hat 2019 成立。2026 主推 Preview 2 + Component Model；wasmtime 是参考实现。
  3. Preview 1 (2020)—— 第一版 WASI；POSIX 风格 imports；广泛实现但有根本缺陷（POSIX 不适合 wasm）。Rust wasm32-wasi target、Go wasi 编译、Zig 等都基于 Preview 1。
  4. Preview 1 设计缺陷—— (1) 复制 POSIX 但 wasm 没进程概念，fork/exec/signal 没意义；(2) 网络模型不完整（仅 socket fd 抽象，没 DNS/TLS/HTTP）；(3) 不支持 async；(4) 没有"接口定义"标准（不同 runtime 实现可能差异）。
  5. Preview 2 (2024-01) 整体重构—— 转向 Component Model + WIT；不再列 POSIX 风格 imports，用 WIT（WebAssembly Interface Types）定义接口。新模型：wasi:io / wasi:filesystem / wasi:clocks / wasi:random / wasi:http / wasi:sockets 等 worlds。
  6. Capability-based 安全模型—— 宿主显式授予能力："这个 wasm 只能读 /workdir，不能读其他目录"；"只能连 example.com:443，不能连其他"。与 Docker 的"Linux capabilities + namespace"（粗粒度）不同——更细。这让 wasm 适合"运行不可信第三方代码"（plugin / edge functions / 智能合约）。

02 底层逻辑

为什么 wasm 需要 WASI

2017-2020 wasm 已在浏览器成熟。但有两个问题让它不能直接走向服务端：

问题 1：浏览器外没有"宿主 imports"—— 浏览器给 wasm 提供 console.log / fetch / DOM API 等。服务端没有浏览器，wasm 调"打印一行"都没有 import。

问题 2：每个 runtime 自己定一套 imports → 生态分裂—— Cloudflare 自家 runtime 一套 API、Fastly 自家一套、wasmtime 一套；同一个 wasm 不能跨 runtime。

WASI 解决：定义一个标准化"系统接口"。wasm 模块声明 (import "wasi_snapshot_preview1" "fd_write" ...)，所有兼容 runtime 都提供这个 import。一次编译，跨 runtime 跑。

这是 wasm 的"POSIX 时刻"——POSIX 让 C 代码能跨 Unix 跑；WASI 让 wasm 能跨 runtime 跑。

Preview 1：POSIX 致敬

2020-04 WASI Preview 1 第一版正式发布。设计哲学：致敬 POSIX。

Preview 1 imports（部分）：

wasi_snapshot_preview1::args_get          // 命令行参数
wasi_snapshot_preview1::args_sizes_get
wasi_snapshot_preview1::environ_get        // 环境变量
wasi_snapshot_preview1::clock_time_get     // 读时钟
wasi_snapshot_preview1::fd_read            // 从 fd 读
wasi_snapshot_preview1::fd_write           // 写到 fd
wasi_snapshot_preview1::fd_close           // 关 fd
wasi_snapshot_preview1::fd_seek
wasi_snapshot_preview1::fd_fdstat_get      // 查 fd 属性
wasi_snapshot_preview1::path_open          // 打开文件
wasi_snapshot_preview1::path_create_directory
wasi_snapshot_preview1::path_remove_directory
wasi_snapshot_preview1::path_rename
wasi_snapshot_preview1::random_get         // 读随机字节
wasi_snapshot_preview1::poll_oneoff        // 多路 IO
wasi_snapshot_preview1::proc_exit          // 退出
wasi_snapshot_preview1::sock_accept        // socket 接收
wasi_snapshot_preview1::sock_recv
wasi_snapshot_preview1::sock_send

看起来就是 POSIX read/write/open/close/socket。

Preview 1 立刻被广泛采纳：

  • wasmtime / wasmer / WasmEdge / wazero 都实现 Preview 1
  • Rust 的 wasm32-wasi target（编译成 Preview 1）
  • Go 编译 wasm 时（用 wasi 后端，2021+）
  • Zig 内置支持
  • WASI SDK（C/C++ → wasm + WASI）

Preview 1 设计缺陷

但 Preview 1 在 2021-2022 暴露根本问题：

缺陷 1：POSIX 假设与 wasm 不符

POSIX 假设：单一可信进程。fork / exec / signal / shared memory（mmap MAP_SHARED）都基于这个假设。

wasm 现实：多语言不可信沙箱。wasm 没有进程概念（多个 instance 是逻辑概念，不是 OS 进程）；fork 没意义（wasm 没"复制进程"操作）；signal 没接受方（wasm 是单调用栈）。

Preview 1 包含 proc_exit（OK）但试图加 fork 时发现没法实现。

缺陷 2：网络模型不完整

Preview 1 只有 socket fd 抽象（sock_accept / sock_recv / sock_send）。但服务端 wasm 大量场景需要：

  • DNS 解析（wasi 没标准化）
  • TLS 握手（wasi 没暴露）
  • HTTP client / server（wasi 完全不涉及）

结果：每个 runtime 自己实现网络扩展（wasmtime 的 wasmtime-wasi-net、Cloudflare 的 fetch API、Fastly 的 hyper-light），不兼容。

缺陷 3：不支持 async

Preview 1 imports 全是同步（fd_read 阻塞直到读完）。但浏览器的 fetch / IndexedDB 都是异步；服务端的 epoll / io_uring 也是异步。同步语义需要包装异步底层（spinlock / OS 阻塞），效率差。

Asyncify 提案试图修复（编译时把同步函数转 state machine），但是 hack 不是干净方案。

缺陷 4：没有"接口定义"标准

Preview 1 是一组 imports 名称，没有正式 IDL。不同 runtime 实现细节可能略有差异（如 fd 0/1/2 是否预绑定 stdin/stdout/stderr）。文档化不足。

这让"WASI 兼容性"难定义——同一 wasm 在 wasmtime 跑得好不一定在 wasmer 跑得好。

Preview 2：Component Model + WIT 重启

2022-2023 Bytecode Alliance 决定整体重启 WASI。Preview 2（2024-01-25 minor 0.2.0 发布，2026 完善中）：

(1) 不再 POSIX 风格—— 接受"wasm 不是 Unix 进程"事实，重新设计

(2) 转向 Component Model + WIT—— 详见 P7.1。WIT 是 IDL（类似 protobuf / thrift），定义接口签名 + 类型

(3) Capability-based 安全—— 宿主授权细粒度

(4) 标准化 worlds—— 一组 imports + exports = 一个 world，是 wasm 想要的环境契约

Preview 2 的核心 worlds（部分）：

wasi:io/poll              // 异步 IO 多路复用
wasi:io/streams           // 输入 / 输出流
wasi:filesystem/types     // 文件系统类型 + 操作
wasi:filesystem/preopens  // 预开目录
wasi:clocks/wall-clock    // 墙钟
wasi:clocks/monotonic-clock
wasi:random/random        // 随机
wasi:cli/environment      // 环境变量
wasi:cli/exit             // 退出
wasi:cli/stdin / stdout / stderr
wasi:http/types           // HTTP 类型
wasi:http/incoming-handler
wasi:http/outgoing-handler
wasi:sockets/network      // 网络 sockets
wasi:sockets/tcp / udp
wasi:keyvalue/store       // 键值存储（草案）

每个 world 定义在 WIT 文件里。例如 wasi:filesystem（部分）：

interface types {
  enum descriptor-type {
    unknown,
    block-device,
    character-device,
    directory,
    fifo,
    symbolic-link,
    regular-file,
    socket,
  }

  resource descriptor {
    read-via-stream: func(offset: filesize) -> result<input-stream, error-code>;
    write-via-stream: func(offset: filesize) -> result<output-stream, error-code>;
    sync: func() -> result<_, error-code>;
    set-size: func(size: filesize) -> result<_, error-code>;
    ...
  }
}

Capability-based 安全模型

"Capability"（能力）= 一个 token / handle 代表"能做某事"。Capability-based 安全：能力必须从外部显式授予，wasm 不能凭空创建。

对比传统 Unix 安全模型（ambient authority）：

  • Unix 进程从启动起就有 uid / gid → 能 access 该 uid/gid 允许的所有文件
  • Linux capabilities（CAP_NET_BIND / CAP_SYS_ADMIN 等）粒度仍粗

Capability-based：

  • wasm 启动时没有任何能力—— 不能读任何文件，不能连任何网络
  • 宿主显式给"directory descriptor for /workdir"作 capability —— wasm 现在能 access /workdir 下文件，但不是其他目录
  • 给 "tcp socket factory bound to example.com:443" —— wasm 能连 example.com:443，但不是其他主机

这种模型对多租户场景（Shopify Functions / Cloudflare Workers / 智能合约）极适合：

  • 租户 A 的 wasm 只看到租户 A 的存储 / 网络
  • 租户 B 的 wasm 完全隔离
  • 恶意 wasm 偷不到其他租户数据，因为根本没有能力

WASI 与浏览器的关系

浏览器 wasm 不直接用 WASI（用 JS API）。但有趋同迹象：

  • WASIX（wasmer 推动）—— 把 WASI Preview 1 + 浏览器 fetch / DOM 等"统一"
  • wasi:http vs 浏览器 fetch—— 试图标准化但 Cloudflare/Fastly 已自家实现，进度慢
  • wasi-browser（实验）—— 让 WASI 兼容 wasm 在浏览器跑

2026 现实：浏览器仍主要用 JS API；服务端 / Edge 用 WASI。两个生态并行。

WIT 文件示例

WIT 是 Preview 2 的核心。一个简单 WIT 文件：

// counter.wit
package my:counter@0.1.0;

world counter {
  export increment: func() -> u32;
  export get: func() -> u32;
  export reset: func();
}

Rust 端用 cargo-component 生成 binding：

// src/lib.rs
mod bindings;
use bindings::Guest;
use std::sync::atomic::{AtomicU32, Ordering};

static COUNT: AtomicU32 = AtomicU32::new(0);

struct Component;

impl Guest for Component {
    fn increment() -> u32 {
        COUNT.fetch_add(1, Ordering::SeqCst) + 1
    }
    fn get() -> u32 { COUNT.load(Ordering::SeqCst) }
    fn reset() { COUNT.store(0, Ordering::SeqCst); }
}

bindings::export!(Component with_types_in bindings);

构建：

$ cargo component build --release
# 产 counter.component.wasm（不是普通 wasm，是 Component）

Preview 1 vs Preview 2 兼容性

Preview 1 不会马上消失。Rust wasm32-wasip1 target 仍主流；很多生产代码基于 Preview 1。

2026 状态：

  • 新项目推荐 Preview 2（component-based）
  • 已有项目可继续 Preview 1，逐步迁移
  • wasmtime / wasmer 同时支持两者
  • Cloudflare Workers / Fastly 仍主要 Preview 1（迁移中）

WASI 0.3 / 异步

Preview 2 仍主要同步语义（async 在 component 层模拟）。WASI 0.3 提案（进行中）将原生 async：

  • imports 函数能返回 future
  • component 间能传 stream
  • 解决 Preview 2 仍存在的 spin-loop 问题

2026 进度：design 阶段；2026-2027 预期实现。

03 应用场景

场景 A · 用 Rust 写 WASI 命令行工具

Preview 1 兼容性好，仍是 CLI 工具的首选：

// src/main.rs
use std::io::{self, Read, Write};

fn main() -> io::Result<()> {
    let mut input = String::new();
    io::stdin().read_to_string(&mut input)?;
    let upper = input.to_uppercase();
    io::stdout().write_all(upper.as_bytes())?;
    Ok(())
}

构建：

$ cargo build --target wasm32-wasip1 --release
$ wasmtime target/wasm32-wasip1/release/upper.wasm < input.txt

场景 B · Cloudflare Workers (Preview 1 + 自家扩展)

Cloudflare Workers 不严格用 WASI。它们提供自家 fetch / KV / R2 / D1 等 API，wasm 通过 imports 访问：

use worker::*;

#[event(fetch)]
async fn main(req: Request, env: Env, _ctx: Context) -> Result<Response> {
    let kv = env.kv("MY_KV")?;
    let value = kv.get("counter").text().await?.unwrap_or("0".to_string());
    let next = value.parse::<u32>().unwrap() + 1;
    kv.put("counter", next.to_string())?.execute().await?;
    Response::ok(format!("count: {next}"))
}

这里的 worker crate 包装了 Cloudflare 自家 imports（不是标准 WASI）。2026 Cloudflare 在跟进 wasi:http 标准化。

场景 C · Fastly Compute@Edge（Preview 1 + Hyper-light）

Fastly 用 wasmtime + 自家 hyper-light（HTTP 处理）：

use fastly::{Request, Response, Error};

#[fastly::main]
fn main(req: Request) -> Result<Response, Error> {
    Ok(Response::from_body("Hello from Fastly Compute@Edge"))
}

编译产 wasm，Fastly 平台部署。冷启动 < 1ms（wasmtime AOT cache）。

场景 D · Shopify Functions（Component Model 实战）

Shopify Functions 是Preview 2 + Component Model的早期生产用户。商家用 Rust/JS 写 wasm 插件，影响 Shopify 商品定价 / 折扣 / 订单逻辑：

// 简化的 Shopify Function
use shopify_function::prelude::*;

#[shopify_function]
fn function(input: input::ResponseData) -> Result<output::FunctionResult> {
    let discount = if input.cart.cost.subtotal_amount.amount > 100.0 {
        Some(output::Discount {
            value: output::Value::Percentage(output::Percentage { value: 10.0 }),
            ...
        })
    } else { None };
    Ok(output::FunctionResult { discounts: vec![discount].into_iter().flatten().collect(), ... })
}

Shopify 用 WIT 定义 input/output schema（capability-based：function 只能看到自己被传入的 cart 数据，不能访问其他商家数据）。

场景 E · Spin（Fermyon）—— Preview 2 全面用

Spin 是基于 Preview 2 + Component Model 的 wasm 应用框架：

# spin.toml
spin_manifest_version = 2

[application]
name = "hello"
version = "0.1.0"

[[trigger.http]]
route = "/..."
component = "hello"

[component.hello]
source = "target/wasm32-wasip2/release/hello.wasm"
allowed_outbound_hosts = ["https://api.example.com"]   # ← capability 授权

Rust 代码：

use spin_sdk::http::{IntoResponse, Request, Response};
use spin_sdk::http_component;

#[http_component]
fn handle(_req: Request) -> anyhow::Result<impl IntoResponse> {
    Ok(Response::builder()
        .status(200)
        .header("content-type", "text/plain")
        .body("Hello from Spin")
        .build())
}

Spin 通过 spin.toml 显式声明 capability（哪些主机能连），wasm 不能突破。

04 典型代码：完整 WASI 工具

例 1：手写 wat 调 WASI Preview 1（Hello World）

(module
  ;; 导入 fd_write
  (import "wasi_snapshot_preview1" "fd_write"
    (func $fd_write (param i32 i32 i32 i32) (result i32)))

  (memory (export "memory") 1)

  ;; "Hello, WASI!\n" 在 memory 偏移 8 处
  (data (i32.const 8) "Hello, WASI!\n")

  ;; iovec: ptr=8, len=13 在 memory 偏移 0 处
  (data (i32.const 0)
    "\08\00\00\00"   ;; ptr = 8
    "\0d\00\00\00")  ;; len = 13

  (func $_start (export "_start")
    (call $fd_write
      (i32.const 1)   ;; fd 1 = stdout
      (i32.const 0)   ;; iovec ptr
      (i32.const 1)   ;; iovec count
      (i32.const 24)) ;; bytes_written 写到这（不用，drop）
    drop))

编译运行：

$ wat2wasm hello.wat
$ wasmtime hello.wasm
Hello, WASI!

例 2：Rust + WASI Preview 1 完整 CLI

// src/main.rs
use std::env;
use std::fs;
use std::io::{self, Write};

fn main() -> io::Result<()> {
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("Usage: {} <file>", args[0]);
        std::process::exit(1);
    }
    let path = &args[1];
    let contents = fs::read_to_string(path)?;
    let lines = contents.lines().count();
    let words = contents.split_whitespace().count();
    let chars = contents.chars().count();
    println!("{} {} {} {}", lines, words, chars, path);
    Ok(())
}
$ cargo build --release --target wasm32-wasip1
$ wasmtime --dir=. target/wasm32-wasip1/release/wc.wasm input.txt
# 必须 --dir=. 显式授权当前目录

例 3：Capability-based 限制（wasmtime 命令行）

# 不授权任何 dir → wasm 不能读任何文件
$ wasmtime my.wasm
Error: failed to find a pre-opened file descriptor

# 显式授权 ./data 目录
$ wasmtime --dir=./data my.wasm

# 映射宿主路径到 wasm 内路径
$ wasmtime --dir=./data::/app/data my.wasm
# wasm 内代码访问 /app/data，宿主对应 ./data

# 授权环境变量（默认不传任何 env）
$ wasmtime --env=API_KEY=secret my.wasm

# 限制内存（fuel 计费）
$ wasmtime --invoke main --fuel 1_000_000 my.wasm

例 4：Component Model + WIT（Preview 2）

WIT 文件（add.wit）：

package example:adder@0.1.0;

world adder {
  export add: func(a: u32, b: u32) -> u32;
}

Rust 实现（用 cargo-component）：

// src/lib.rs
#[allow(warnings)]
mod bindings;
use bindings::Guest;

struct Component;

impl Guest for Component {
    fn add(a: u32, b: u32) -> u32 {
        a + b
    }
}

bindings::export!(Component with_types_in bindings);

构建：

$ cargo install cargo-component
$ cargo component new adder
# 编辑 wit/world.wit + src/lib.rs
$ cargo component build --release
# 产 adder.wasm（是 Component 不是 plain module）

多语言互操作：用 wit-bindgen 生成 Go / JS / Python binding 调用此 component。

例 5：用 wasmtime API 嵌入 WASI

// 在 Rust 程序里嵌入 wasmtime 跑 wasm
use wasmtime::{Engine, Module, Store, Linker};
use wasmtime_wasi::{WasiCtxBuilder, WasiCtx};

fn main() -> anyhow::Result<()> {
    let engine = Engine::default();
    let mut linker = Linker::new(&engine);
    wasmtime_wasi::preview1::add_to_linker_sync(&mut linker, |s| s)?;

    // capability：只授权当前目录 + 标准输入输出
    let wasi = WasiCtxBuilder::new()
        .inherit_stdio()
        .preopened_dir(".", "/").unwrap()
        .build_p1();

    let mut store = Store::new(&engine, wasi);
    let module = Module::from_file(&engine, "guest.wasm")?;
    let instance = linker.instantiate(&mut store, &module)?;
    let start = instance.get_typed_func::<(), ()>(&mut store, "_start")?;
    start.call(&mut store, ())?;
    Ok(())
}

05 常见陷阱

陷阱 1：用 Preview 1 期望 Preview 2 特性

Preview 1 不支持：

  • HTTP（要用第三方 wasi-net 或 runtime 自家扩展）
  • Async（同步语义）
  • Component 间通信（要 Preview 2）

需要这些特性 → 用 Preview 2 / Component Model。

陷阱 2：忘记 --dir 授权

wasmtime 默认不授权任何文件系统访问。直接 wasmtime my.wasm 让 wasm 看不到任何文件。必须 --dir 授权：

$ wasmtime --dir=. my.wasm   # 授权当前目录

这是 capability-based 安全的体现—— 默认 deny，必须显式 allow。

陷阱 3：以为 WASI 模拟完整 POSIX

Preview 1 是 POSIX 子集，很多 POSIX 调用没有：

  • fork / exec —— 没有进程概念
  • signal / sigaction —— 没接受方
  • mmap / munmap —— linear memory 自己管
  • setuid / setgid —— 没用户概念
  • ioctl —— 大部分

用这些 API 的 C 代码编译会 link error 或 runtime trap。

陷阱 4：同步 IO 阻塞所有 worker

WASI Preview 1 的 fd_read 是同步的。在多线程 wasm runtime 里，一个线程阻塞 fd_read 不影响其他线程；但在浏览器主线程或 Cloudflare Workers 单线程模型，阻塞让整个 worker 卡住。

方案：(1) 用 JSPI（浏览器）；(2) 用 Preview 2 async；(3) 永远在 worker thread 内做同步 IO。

陷阱 5：Preview 1 vs 2 binary 不兼容

Preview 1 wasm（普通 module）和 Preview 2 component 不是同一种二进制：

  • Preview 1: (module ...)，跑在 wasmtime 经典模式
  • Preview 2: (component ...)，跑在 wasmtime serve 或 wasmtime run -W component-model

不能混用。cargo-component vs cargo build 产物不同。

陷阱 6：自定义 imports 与 WASI 冲突

如果 wasm 同时用 WASI imports 和自家 imports，要注意命名空间：

(import "wasi_snapshot_preview1" "fd_write" ...)   ;; WASI
(import "env" "my_func" ...)                       ;; 自家

自家 imports 用 env 或自家 namespace，不要用 wasi_*。

陷阱 7：Component Model 工具链不成熟

2026-05 cargo-component / wit-bindgen / wasm-tools 仍在快速演进。API 不稳定，rebuild 时可能 break。

生产用 Preview 2 + Component 要：

  • 锁定工具版本
  • 有 fallback 计划（能切回 Preview 1）
  • 跟踪 spec 进度（github.com/WebAssembly/component-model）

陷阱 8：WASI HTTP 互操作差

wasi:http（Preview 2）尝试标准化 HTTP，但 Cloudflare / Fastly 已自家实现。同一个 wasm 不一定跨平台：

  • Cloudflare Workers 用自家 fetch API（不严格遵循 wasi:http）
  • Fastly 用 hyper-light（部分 wasi:http 兼容）
  • Spin / wasmtime serve 严格用 wasi:http

跨平台部署需要 abstraction layer（如 Spin SDK）。
