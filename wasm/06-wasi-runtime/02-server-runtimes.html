服务端 Runtime 家族 + wasm vs Docker

浏览器外跑 wasm 的运行时家族。wasmtime（Bytecode Alliance, Rust + Cranelift, 事实标准）/ wasmer（多后端 Cranelift/LLVM/Singlepass，商业公司）/ wazero（纯 Go 零 C 依赖，Go 项目嵌入首选）/ WasmEdge（CNCF, C++ 实现，AI 推理优化）/ V8 isolate（Cloudflare Workers / Bun / Node 用 JS 引擎跑 wasm）。wasm vs Docker：冷启动快两个数量级（< 1ms vs 100ms-1s），内存基线小 50x（< 1MB vs 50MB+），capability-based 比 Linux capabilities 细粒度。不是替代而是补足"边缘 / 插件 / 多租户"场景。

阅读时长 ~30 分钟 前置 P6.1（WASI） 主要来源 wasmtime.dev · wasmer.io · wazero.io · wasmedge.org · 各 Edge 平台技术博客

01 速览

  1. wasmtime 是事实标准—— Bytecode Alliance 维护，Rust 实现，Cranelift 后端。Fastly Compute@Edge / Spin / 大量服务端项目用它。spec 实现最完整 + 安全审计严。
  2. wasmer 是商业玩家 + 多后端—— 同时提供 Cranelift / LLVM / Singlepass 三个后端：Cranelift 平衡 / LLVM 性能极致 / Singlepass 启动最快。商业公司维护，跨语言 SDK 最丰富（Python / JS / Go / Java / C / Ruby / 等）。
  3. wazero 是 Go 项目嵌入首选—— Tetrate 开发，纯 Go 实现 + 零 C 依赖。让 Go 应用嵌入 wasm 不需 CGO（CGO 是 Go 部署痛点）。Envoy / Hashicorp 工具链 / Tetrate Service Bridge 用它。
  4. WasmEdge 是 CNCF + AI 优化—— C++ 实现，针对 cloud-native + AI 推理优化（内置 LLM 加速 / TensorFlow / OpenVINO 集成）。Kubernetes 生态 + 边缘 AI 场景。
  5. V8 isolate 是另一种部署模型—— Cloudflare Workers / Bun / Node 用 JS 引擎跑 wasm，与 JS 共享 runtime；不用 wasmtime。优势：已有 V8 平台加 wasm 几乎零成本；劣势：wasm-specific 优化少。Cloudflare Workers 是这种模型最大用户。
  6. wasm 不取代 Docker—— wasm 适合"短生命周期 / 按请求实例化 / 不可信代码"；Docker 适合"长生命周期 / 整机系统镜像"。两者并存。2026 趋势：Edge / Functions / 插件用 wasm；微服务 / 数据库 / 复杂 stateful 应用仍 Docker。

02 底层逻辑

wasmtime：事实标准

wasmtime（Bytecode Alliance）是 wasm 服务端事实标准。架构：

(1) Cranelift 编译器后端—— Rust 实现的编译器框架（同时是 Rust nightly 备选 LLVM 替代）；wasm → native 一次编译到位

(2) WASI 实现—— Preview 1 + Preview 2 都支持

(3) Component Model 支持—— wasmtime 是 Component Model 参考实现

(4) Embedder API—— 让其他程序嵌入 wasmtime（Rust API 最丰富，Python/Go/C 等也有 binding）

关键特性：

  • AOT 编译 + 缓存—— wasmtime 把 wasm 提前编译成 native shared object 缓存（.cwasm 文件）；下次启动直接 mmap，毫秒级
  • Module instantiation pooling—— 每请求 new instance（隔离）但复用编译产物
  • Fuel metering—— 每条 wasm 指令计费，让 multi-tenant 平台限制单 module 占用
  • Epoch interruption—— 让 host 中断长跑 wasm（防 DoS）

wasmtime 命令行：

$ wasmtime my.wasm
$ wasmtime --dir=. --invoke add my.wasm 2 3
$ wasmtime compile my.wasm -o my.cwasm    # AOT 编译
$ wasmtime --allow-precompiled my.cwasm   # 跑预编译
$ wasmtime serve --addr 0.0.0.0:8080 component.wasm   # 跑 wasi:http component

wasmer：商业 + 多后端

wasmer（商业公司，Rust）的差异：

  • 3 个后端：
  • - Cranelift（编译速度中、性能中，默认）
  • - LLVM（编译慢、性能极致 ~100% native）
  • - Singlepass（编译极快 ~1µs/byte，性能 60-70% native）

多后端让 wasmer 适应不同场景：

  • 启动敏感（短生命周期 lambda）→ Singlepass
  • 稳态性能敏感（长跑服务）→ LLVM
  • 平衡 → Cranelift（与 wasmtime 等价）

wasmer 跨语言 SDK—— wasmer-python / wasmer-js / wasmer-go / wasmer-java / wasmer-c-api / wasmer-ruby / 等都是官方维护，比 wasmtime 多。

WAPM (wasmer package manager)—— wasm 包管理器，类似 npm；2026 仍小众，没成事实标准。

wazero：纯 Go 零依赖

wazero（Tetrate, 2021-）的最大差异：纯 Go 实现，零 C 依赖。

为什么这是杀手锏：

(1) Go 项目用 CGO 是部署痛点 —— CGO 让交叉编译困难（要 C 工具链）、动态链接 libc、性能开销

(2) wasmtime / wasmer 都是 Rust 实现，C ABI 暴露给 Go → 必须 CGO

(3) wazero 全 Go 实现 → Go 项目嵌入 wasm 不需要 CGO，cross-compile / static link 都正常

wazero 用户：

  • Envoy proxy 的 wasm filter（用 Go 写代理插件，wazero 跑用户 wasm）
  • Hashicorp Boundary / Vault 等 Go 项目嵌入 wasm 插件
  • Tetrate Service Bridge（服务网格扩展点）
  • Buildless / Mosn 等 Go 服务端项目

性能：wazero 解释器比 wasmtime 慢 5-10x（纯 Go 无 JIT）；2024+ wazero 加了 optimizing compiler（也是 Go 实现），追平到 ~50% wasmtime。

WasmEdge：CNCF + AI

WasmEdge（CNCF Sandbox 2021，毕业 2024）C++ 实现。差异：

  • 针对 cloud-native 优化—— Kubernetes / containerd / Docker 集成最好（runwasi shim）
  • AI 推理—— 内置 wasi-nn 实现（TensorFlow / PyTorch / OpenVINO 集成）；让 wasm 调用 ML 推理 API
  • 边缘场景—— ARM / RISC-V 嵌入式平台支持好
  • JS / Python interpreter 集成—— 在 wasm 里跑 JS / Python（用 QuickJS / Pyodide）

典型 WasmEdge 用户：边缘 AI 推理、Kubernetes wasm workload、IoT 设备。

V8 isolate 跑 wasm（Cloudflare Workers / Bun / Node）

另一种部署模型：不用专门 wasm runtime，用 JS 引擎。

原因：

  • 已有 V8 / JSC 平台（运行 JS）
  • V8 / JSC 都内置 wasm 实现（详见 V8 P6.1）
  • 加 wasm 几乎零成本（不需要新 runtime）
  • JS / wasm 在同 isolate，跨边界开销小

典型用户：

  • Cloudflare Workers—— 用 V8 isolate；每请求一个 isolate，wasm 在 isolate 内
  • Bun—— 用 JSC（详见 Bun 主题）；wasm import 像 JS module
  • Node—— 用 V8；WebAssembly.instantiate 与浏览器一致
  • Deno—— 用 V8；同 Node
  • Vercel Edge Functions—— V8 isolate（Cloudflare 提供基础设施）

JS 引擎跑 wasm 的限制：

  • WASI 支持有限（Cloudflare 自家扩展，不严格 WASI 标准）
  • Component Model 支持滞后（V8 / JSC 都是 wasm core 优先）
  • wasm 优化方向偏 JS 协作，不是单独 wasm 性能

关键洞察：

(1) wasm 比 Docker 快 ~100x 启动 + 50x 内存—— 这是"按请求实例化"场景的杀手特性。Docker 1000 个 instance 占 50GB 内存，wasm 占 1GB。

(2) Docker 安全靠 Linux 内核，wasm 安全靠 sandbox + spec—— Docker 漏洞通常源于 Linux 内核 bug；wasm 漏洞通常源于 spec 实现 bug。两者攻击面不同。

(3) wasm 跨平台是 Docker 没有的—— 一个 wasm 跑 amd64 / arm64 / RISC-V，Docker 必须 build per arch。

(4) Docker 系统能力完整—— 你能跑 Postgres / Redis / Kafka 在 Docker；wasm 还跑不了这些（系统调用太多没标准化）。

wasm + K8s：runwasi shim

2024+ Kubernetes 通过 runwasi（containerd shim）支持 wasm workload：

# Pod spec
apiVersion: v1
kind: Pod
spec:
  runtimeClassName: wasmtime    # ← 用 wasmtime 而不是 runc
  containers:
  - name: my-wasm-app
    image: my-registry/my-wasm:v1
    # image 实际是 OCI 包装的 wasm

原理：containerd 看到 RuntimeClass = wasmtime，调 runwasi shim 而不是 runc；shim 用 wasmtime 跑 wasm。让 wasm workload 像普通 container 一样调度。

2026 仍属早期，但 Kubernetes / wasm 集成在快速演进。

03 应用场景

场景 A · 选 runtime 决策树

问题 1：嵌入语言是什么？

  • Rust → wasmtime（同语言，API 顺）
  • Go → wazero（不需要 CGO）
  • C/C++ → wasmtime / wasmer（C-API）
  • Python → wasmer-python（最完整）
  • JavaScript / Node → 直接用 V8 内置 WebAssembly API

问题 2：场景需求？

  • Edge / Functions（短跑）→ Cloudflare Workers / Fastly Compute / Spin / wasmtime
  • Plugin / 多租户 → wasmer / wasmtime（fuel metering）
  • K8s workload → WasmEdge / wasmtime（runwasi）
  • 边缘 AI 推理 → WasmEdge（wasi-nn）
  • 区块链智能合约 → wasmer / wasmtime（多用 fork 版本）

问题 3：性能优先级？

  • 启动最快 → wasmer Singlepass
  • 稳态最快 → wasmer LLVM
  • 平衡 → wasmtime / wasmer Cranelift

场景 B · Spin 部署完整 wasm 应用

Spin（Fermyon）是基于 wasmtime + Component Model 的应用框架：

# spin.toml
spin_manifest_version = 2

[application]
name = "my-app"
version = "0.1.0"

[[trigger.http]]
route = "/api/..."
component = "api"

[component.api]
source = "target/wasm32-wasip2/release/api.wasm"
allowed_outbound_hosts = ["https://example.com"]   # capability

[component.api.build]
command = "cargo component build --release"
$ spin build
$ spin up                    # 本地跑
$ spin deploy                # 部署到 Fermyon Cloud

场景 C · Envoy wasm filter（用 wazero 嵌入）

Envoy 代理通过 wasm filter 让用户写自定义代理逻辑：

// Rust filter
use proxy_wasm::traits::*;
use proxy_wasm::types::*;

#[no_mangle]
pub fn _start() {
    proxy_wasm::set_log_level(LogLevel::Trace);
    proxy_wasm::set_http_context(|_, _| -> Box<dyn HttpContext> {
        Box::new(MyHttpFilter)
    });
}

struct MyHttpFilter;
impl HttpContext for MyHttpFilter {
    fn on_http_request_headers(&mut self, _: usize, _: bool) -> Action {
        if let Some(value) = self.get_http_request_header("x-special") {
            self.send_http_response(200, vec![], Some(b"intercepted"));
            return Action::Pause;
        }
        Action::Continue
    }
}

编译产 wasm，Envoy 配置加载 → wasm filter 跑在每个请求。Envoy 内部用 wazero（Go 代码）嵌入。

场景 D · K8s 跑 wasm workload（runwasi）

用 spin-runtime-class（Fermyon 维护）：

apiVersion: node.k8s.io/v1

Pod 启动时 containerd 调 spin shim，用 wasmtime 跑 wasm。冷启动 < 100ms（vs Docker 1-3s）。

场景 E · 把 wasmtime 嵌入 Rust 程序（plugin 系统）

use wasmtime::*;
use wasmtime_wasi::WasiCtxBuilder;

fn run_plugin(wasm_path: &str, input: &str) -> anyhow::Result<String> {
    let engine = Engine::default();
    let mut linker = Linker::new(&engine);
    wasmtime_wasi::preview1::add_to_linker_sync(&mut linker, |s: &mut WasiCtx| s)?;

    // capability：只允许读 /tmp，不允许其他 dir
    let wasi = WasiCtxBuilder::new()
        .preopened_dir("/tmp", "/tmp").unwrap()
        .build_p1();

    let mut store = Store::new(&engine, wasi);
    let module = Module::from_file(&engine, wasm_path)?;
    let instance = linker.instantiate(&mut store, &module)?;

    // 调 plugin 的 process 函数
    let process = instance.get_typed_func::<(i32, i32), i32>(&mut store, "process")?;

    // 写 input 到 wasm memory，调 process，读返回
    // ... (省略 marshalling 细节) ...
    Ok(result)
}

这种 pattern 让任何 Rust 应用支持 wasm plugin（用户提交 wasm，沙箱内跑）。

场景 F · Bun 嵌入 wasm

// Bun import wasm 像普通 module
import { add } from './my.wasm';
console.log(add(2, 3));  // 5

// 或用 Bun 的 WebAssembly API
const wasm = Bun.file('./my.wasm');
const { instance } = await WebAssembly.instantiate(await wasm.arrayBuffer());
console.log(instance.exports.add(2, 3));

Bun 用 JSC 的 wasm 实现（详见 Bun 主题），与浏览器 Safari 一致行为。

04 典型代码：完整集成

例 1：Go 项目用 wazero 嵌入 wasm

package main

import (
    "context"
    "fmt"
    "os"

    "github.com/tetratelabs/wazero"
    "github.com/tetratelabs/wazero/imports/wasi_snapshot_preview1"
)

func main() {
    ctx := context.Background()

    // Runtime 配置
    r := wazero.NewRuntime(ctx)
    defer r.Close(ctx)

    // 注册 WASI imports
    wasi_snapshot_preview1.MustInstantiate(ctx, r)

    // 加载 wasm
    wasmBytes, err := os.ReadFile("my.wasm")
    if err != nil { panic(err) }

    // 配置 capability：stdout / stderr inherit
    config := wazero.NewModuleConfig().
        WithStdout(os.Stdout).
        WithStderr(os.Stderr).
        WithFSConfig(wazero.NewFSConfig().WithDirMount(".", "/data"))

    // 实例化并跑 _start
    _, err = r.InstantiateWithConfig(ctx, wasmBytes, config)
    if err != nil { panic(err) }
}

例 2：wasmtime CLI + AOT 缓存

# AOT 编译（一次）
$ wasmtime compile big-module.wasm -o big-module.cwasm

# 跑预编译版本（启动 ~1ms vs ~50ms 原 wasm）
$ wasmtime --allow-precompiled big-module.cwasm

# 部署：把 .cwasm 上传，运行时跳过 compile

例 3：fuel metering 限制 wasm 资源

// 嵌入 wasmtime 时启用 fuel
let mut config = Config::new();
config.consume_fuel(true);
let engine = Engine::new(&config)?;

let mut store = Store::new(&engine, ());
store.set_fuel(1_000_000)?;   // 给 1M units fuel

let module = Module::from_file(&engine, "untrusted.wasm")?;
let instance = Linker::new(&engine).instantiate(&mut store, &module)?;

// 跑 wasm；如果消耗 > 1M units，trap fuel exhausted
match instance.get_typed_func::<(), ()>(&mut store, "main")?.call(&mut store, ()) {
    Ok(()) => println!("OK; fuel left: {}", store.get_fuel().unwrap()),
    Err(e) => println!("trapped: {}", e),
}

用于 multi-tenant：每用户限制 fuel，防止单 wasm 跑死整机。

例 4：epoch interruption 中断长跑 wasm

use std::sync::Arc;
use std::time::Duration;

let mut config = Config::new();
config.epoch_interruption(true);
let engine = Arc::new(Engine::new(&config)?);

let mut store = Store::new(&engine, ());
store.set_epoch_deadline(1);   // 1 epoch 后 trap

// 启动后台线程定时增加 epoch
let engine_clone = engine.clone();
std::thread::spawn(move || {
    loop {
        std::thread::sleep(Duration::from_millis(100));
        engine_clone.increment_epoch();
    }
});

// wasm 运行 100ms 后会 trap "interrupted"

用于 server-side wasm 防 DoS：单请求最多跑 100ms，超时中断。

例 5：wasmer 选不同 backend

use wasmer::{Store, Module, Instance, imports};
use wasmer_compiler_cranelift::Cranelift;
use wasmer_compiler_singlepass::Singlepass;
use wasmer_compiler_llvm::LLVM;

// 启动最快（短生命周期）
let mut store = Store::new(Singlepass::new());

// 性能极致（长跑）
let mut store = Store::new(LLVM::new());

// 平衡（默认）
let mut store = Store::new(Cranelift::new());

let module = Module::from_file(&store, "my.wasm")?;
let instance = Instance::new(&mut store, &module, &imports! {})?;

05 常见陷阱

陷阱 1：以为 wasm 取代 Docker

2026 年 Docker 仍是主流容器运行时。wasm 补足"按请求实例化 / 多租户 / 边缘"场景，但跑 Postgres / Redis / Kafka 等复杂应用仍用 Docker。不要为了 wasm 而 wasm。

陷阱 2：选错 runtime

常见错误：

  • Go 项目用 wasmer（要 CGO 部署痛）→ 应该用 wazero
  • 性能极致用 wasmtime（Cranelift 中等）→ 应该用 wasmer LLVM
  • 启动敏感 lambda 用 wasmtime 默认 → 应该用 wasmer Singlepass

陷阱 3：以为冷启动一定 < 1ms

需要前提：(1) AOT 缓存命中；(2) 模块小 / instantiate 复用编译产物；(3) imports 简单。第一次跑（无 AOT cache）启动可能 100ms+。生产部署必须 AOT 编译 + 缓存。

陷阱 4：Cloudflare Workers 期望严格 WASI

Cloudflare 用 V8 isolate + 自家扩展，不严格遵循 WASI。在 wasmtime 跑得好的 wasm 不一定在 Cloudflare 跑得好。

方案：用 Cloudflare 推荐的 worker SDK（有 WASI 抽象层）；或用 Spin SDK（多平台）。

陷阱 5：忘记 capability 授权

wasmtime / wasmer / WasmEdge 默认不授权任何文件 / 网络。直接跑 wasm 看不到任何文件。必须显式 --dir / --env / etc.。

陷阱 6：fuel metering 误用

fuel 计费每条 wasm 指令 1 unit；不同操作消耗一致。不能精确限制时间（指令执行速度受 CPU / cache 影响）。fuel 适合"防止无限循环"，不适合"严格限制 100ms"。

严格时间限制用 epoch interruption。

陷阱 7：嵌入式 runtime 内存不释放

Rust 嵌入 wasmtime / wasmer 时，每次 instantiate 会分配 wasm linear memory。store 不 drop 内存不释放。multi-tenant 场景必须管理 store 生命周期：

// 每请求 new store + drop
{
    let mut store = Store::new(&engine, wasi);
    let instance = linker.instantiate(&mut store, &module)?;
    // 处理请求 ...
}   // store drop, memory 释放

陷阱 8：用 Singlepass 跑长跑服务

Singlepass 启动快（~1µs/byte 编译），但稳态性能 60-70% native。长跑服务（大流量持续运行）不要用 Singlepass，用 Cranelift / LLVM。Singlepass 适合启动密集 + 短生命周期（lambda 类）。
