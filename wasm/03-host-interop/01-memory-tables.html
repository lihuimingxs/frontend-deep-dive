Linear Memory + Tables

wasm 的"内存模型"由两部分组成。Linear Memory：连续 byte 数组、64KB 页粒度、初始/max 页声明、memory.grow 动态扩容；JS 端 WebAssembly.Memory.buffer 就是 ArrayBuffer，可被 TypedArray view。Tables：另一种"内存"，元素是函数引用（funcref）或 JS 对象引用（externref），通过 call_indirect 实现间接调用——C/C++ 函数指针 / vtable / 动态分发的编译目标。Memory64（phase 4）让 SQLite-WASM / DuckDB-WASM 突破 4GB 限制。Bulk memory ops（2.0 正式）让大块拷贝 / 填充走优化路径。

阅读时长 ~30 分钟 前置 P2（栈机器 + 类型系统） 主要来源 WebAssembly Core Spec · MDN WebAssembly.Memory · v8.dev/blog/memory64

01 速览

  1. Linear Memory 是连续 byte 数组—— wasm module 默认 1 块 memory（multi-memory 提案 phase 4 让多块），64KB 页粒度，初始 N 页 + max 页声明。指令 i32.load / i32.store 带对齐 + 偏移参数。bounds check 现代引擎用 mmap + guard pages 几乎零成本。
  2. WebAssembly.Memory.buffer 就是 ArrayBuffer—— JS 端能直接 view：new Uint8Array(memory.buffer)。这是 wasm-JS 数据交换的核心机制。但 wasm 端 memory.grow 后旧 buffer 可能 detach，需要重新 view。
  3. Tables 是函数指针的替身—— wasm 不暴露函数地址（安全），动态调用必须通过 table 索引 + call_indirect。C/C++ vtable / 函数指针都编译到 table。table 支持 funcref / externref（2.0+）。
  4. Memory64 突破 4GB 限制—— 默认 i32 索引 → 4GB 上限；Memory64（V8 11+，phase 4）i64 索引 → 16EB 理论上限。SQLite-WASM 处理大数据库 / DuckDB-WASM 跑 100GB OLAP 都需要这个。代价：64-bit pointer 让代码体积增加。
  5. Bulk memory ops（2.0 正式）—— memory.copy / memory.fill / memory.init / memory.drop。替代 MVP 时 hand-rolled loop；引擎用 SIMD + cache-aware 实现，比纯 wasm 循环快 10x+。
  6. wasm 内存与 JS 堆完全隔离—— wasm linear memory 是独立 ArrayBuffer，不在 V8 GC 堆里。这让 wasm GC 不会触发 V8 GC，也让 JS 不能直接访问 wasm 对象（必须通过 memory view）。详见 V8 P4.1。

02 底层逻辑

Linear Memory：模型

wasm linear memory = 连续 byte 数组。和 C 的 malloc / heap 一样的物理模型，但有几个关键约束：

(1) 单块（MVP）—— 每模块默认 1 个 memory；multi-memory 提案让多块，但 2026 还在 phase 4，主流 runtime 实现不齐

(2) 页粒度 64KB—— 1 page = 65536 bytes。所有 grow / 初始 / max 都以 page 为单位

(3) 初始 + 最大声明：

(memory $mem 1 100)   ;; 初始 1 页 (64KB)，最大 100 页 (6.4MB)
(memory $mem 1)       ;; 初始 1 页，无 max 限制（最大 4GB）
(memory $mem 1 1 shared)  ;; threads 用，shared memory

(4) i32 索引（默认）—— 偏移用 i32，理论上限 4GB（2^32 bytes）

(5) Memory64 提案—— i64 索引，上限 16EB（理论；实际 OS 限制）

memory.grow + memory.size

动态扩容：

(memory.grow (i32.const 1))   ;; 增加 1 页；返回旧 size 或 -1（失败）
(memory.size)                 ;; 返回当前 size（页数）

JS 端：

const memory = new WebAssembly.Memory({ initial: 1, maximum: 100 });
console.log(memory.buffer.byteLength);  // 65536 (1 page)
memory.grow(2);                          // 增 2 页
console.log(memory.buffer.byteLength);  // 196608 (3 pages)

关键陷阱：grow 后旧 buffer detach—— grow memory 时浏览器可能要 realloc（实际不一定，看实现），JS 端持有的旧 ArrayBuffer变成 detached state，访问会 throw。必须重新 memory.buffer：

// 反例
const view = new Uint8Array(memory.buffer);
memory.grow(1);
view[0] = 42;  // ❌ 可能 throw "ArrayBuffer is detached"

// 正确
let view = new Uint8Array(memory.buffer);
memory.grow(1);
view = new Uint8Array(memory.buffer);  // 重新创建 view
view[0] = 42;

i32.load / i32.store + 对齐 + 偏移

从 memory 读 / 写指令：

i32.load offset=0 align=2
;; 从栈顶（i32 地址）读 4 字节，按 4 字节对齐

load 指令家族：

  • i32.load / i64.load / f32.load / f64.load—— 全宽度读
  • i32.load8_s / i32.load8_u—— 读 1 byte 符号 / 无符号扩展到 i32
  • i32.load16_s / i32.load16_u—— 读 2 bytes 扩展到 i32
  • i64.load8_s / i64.load8_u / i64.load16_s / 等—— 类似但 i64
  • v128.load（SIMD）—— 读 16 bytes 到 v128

store 类似但少 sign 区分（store 不需要扩展）：i32.store / i32.store8 / i32.store16 / i64.store / 等。

对齐参数：align=N 是提示，不是要求。引擎可用更高效路径如果对齐；不对齐时仍可工作（但慢）。align=2 表示 2^2 = 4 字节对齐。

偏移参数：offset=N 在栈上地址外加常量偏移。便利常量偏移（不需多一条 i32.add）。

Bounds check：现代引擎几乎零成本

每次 memory 读 / 写需检查不越界。MVP 时这是真的成本（每次 load 加 cmp + jmp）。现代引擎用 mmap + guard pages 让它免费：

方案 A · 软件 bounds check（旧）：

if (addr + 4 > memory_size) trap;
load i32 at addr

方案 B · mmap + guard pages（V8 / wasmtime 现代）：

  • 分配 8GB 虚拟地址（i32 索引最大 4GB + 4GB guard）
  • 实际只 commit 需要的页；其余是 reserved 不可访问
  • load 不做软件 check；如果地址越界，CPU page fault → 引擎拦截 → trap

结果：正常情况零成本（不查界），仅越界时 page fault。这是 wasm 性能接近 native 的关键。

注意：mmap + guard pages 只在 64-bit OS 上 work。32-bit OS（极罕见 2026）回退到软件 check。Memory64 因为 i64 上限太大无法 mmap 全部，回退软件 check（性能损失）。

Memory64 提案（phase 4，V8 11+）

MVP 限制：i32 索引 → 4GB 内存上限。对于 SQLite / DuckDB / 大型 ML 模型不够用。

Memory64 提案（2024 phase 4）：

  • memory 声明加 i64 标志：(memory i64 1 1024)（页数 i64）
  • load / store 指令偏移变 i64：i32.load offset=0（地址 i64）
  • JS API：new WebAssembly.Memory({ index: "i64", initial: 1n, maximum: 1024n })

实现状态：

  • V8 11+（2023）—— 实现完整
  • SpiderMonkey / JSC—— 实现中或部分
  • wasmtime / wasmer—— 完整支持

典型用户：

  • SQLite-WASM 处理 > 4GB 数据库
  • DuckDB-WASM 跑大规模 OLAP
  • Pyodide 加载大型科学计算数据
  • wasm-shop 跑 LLM 推理（模型权重几 GB）

代价：

  • 64-bit pointer 让代码体积增加 ~10-30%
  • 不能用 mmap + guard pages 优化（虚拟地址不够），bounds check 软件实现，性能损失 ~5-10%
  • 在 32-bit 平台不可用（其实 2026 也没人在 32-bit 平台跑 wasm）

Bulk memory operations（2.0 正式）

MVP 时大块内存操作要写 wasm 循环：

;; 复制 N 字节，慢
(loop $L
  (i32.store8
    (i32.add (local.get $dst) (local.get $i))
    (i32.load8_u (i32.add (local.get $src) (local.get $i))))
  (local.set $i (i32.add (local.get $i) (i32.const 1)))
  (br_if $L (i32.lt_s (local.get $i) (local.get $n))))

Bulk memory（2.0）提供原生指令：

;; memory.copy: dst, src, size
(memory.copy
  (local.get $dst)
  (local.get $src)
  (local.get $n))

;; memory.fill: dst, value, size
(memory.fill
  (local.get $dst)
  (i32.const 0)
  (local.get $n))

;; memory.init: dst, src_offset_in_data_segment, size
(memory.init $dataSeg
  (local.get $dst)
  (i32.const 0)
  (local.get $n))

;; data.drop: 释放 data segment（不再需要）
(data.drop $dataSeg)

引擎实现用 SIMD + cache-aware copy，比纯 wasm loop 快 10x+。所有现代 wasm 编译器（Rust / Emscripten）的 memcpy / memset 都编译到这些指令。

Tables：函数指针的替身

wasm 不暴露函数地址（安全考虑：让任意指针调用很难做 SFI）。动态调用必须通过 table：

(table $t 10 funcref)   ;; 声明 table，10 个 funcref 槽
(elem (i32.const 0) $f1 $f2 $f3)   ;; 初始化 0/1/2 槽

(call_indirect (type $sig) (...args...) (local.get $idx))
;; 用 idx 索引 table；类型必须匹配 $sig；否则运行时 trap

关键属性：

(1) 类型签名匹配检查—— call_indirect 必须显式声明期望签名 (type $sig)；运行时检查 table[idx] 函数签名是否一致；不一致 trap

(2) 索引边界检查—— idx 越界 trap

(3) null 检查—— table[idx] 是 null（未初始化）trap

所以 call_indirect 仍是安全的：错误的索引或签名不会跑到任意代码。

典型应用：

  • C++ vtable —— 编译器为每个虚函数表分配 table 槽，方法调用变 call_indirect
  • 函数指针 —— C 的 void (*fp)(int) 编译成 i32 索引；调用 fp 变 call_indirect
  • switch / dispatch table
  • plugin / hot reload —— 通过 table.set 动态替换函数引用

externref + table

2.0 后 table 也可以装 externref（JS 对象引用）：

(table $objs 10 externref)
(func $store (param $i i32) (param $obj externref)
  (table.set $objs (local.get $i) (local.get $obj)))
(func $load (param $i i32) (result externref)
  (table.get $objs (local.get $i)))

这让 wasm 能管理一个"JS 对象池"，用 i32 索引访问。比 wasm-bindgen 早期的 "ID → object Map" hack 干净。

多个 tables（reference types 提案带来）

2.0 后 module 可有多个 tables：

(table $funcs 10 funcref)
(table $objs 10 externref)
(call_indirect $funcs (type $sig) ...)   ;; 显式指定哪个 table

典型用：funcs table 给 C++ vtable，objs table 给 JS 对象池。

JS 端 view memory：TypedArray + DataView

JS 端把 wasm memory 当 ArrayBuffer，用 TypedArray view 访问：

const memory = instance.exports.memory;

// 不同 typed view（共享底层 buffer）
const u8 = new Uint8Array(memory.buffer);
const u16 = new Uint16Array(memory.buffer);
const u32 = new Uint32Array(memory.buffer);
const f32 = new Float32Array(memory.buffer);
const f64 = new Float64Array(memory.buffer);

// 写入
u32[0] = 0xDEADBEEF;
console.log(u8[0]);  // 0xEF (little-endian)
console.log(u8[3]);  // 0xDE

// 大端 / 小端 / 显式控制：用 DataView
const view = new DataView(memory.buffer);
view.setUint32(0, 0xDEADBEEF, false);  // big-endian
view.setUint32(0, 0xDEADBEEF, true);   // little-endian (wasm 默认)

wasm 是 little-endian—— 跨平台一致；不像 C 取决于宿主 CPU。i32.load 写入的字节顺序确定。

linear memory 与 V8 堆的关系

wasm linear memory 是独立 ArrayBuffer，不在 V8 GC 堆里（详见 V8 P4.1）。

  • wasm allocate 不触发 V8 GC
  • V8 GC 不扫描 wasm linear memory
  • JS 持有 wasm Memory 对象的引用 → wasm memory 不被回收
  • wasm 不能直接访问 V8 堆里的 JS 对象（必须通过 externref 间接）

这是 wasm 性能可预测的一个原因：wasm 内的内存管理与 JS 堆解耦，wasm hot path 不会被 JS GC 中断。

WasmGC 对象（短重述链回 P5.2 / V8 P6）

WasmGC 提案（2023 phase 5）引入 struct / array / i31ref 等 GC managed 类型。这些不在 linear memory 里，由引擎 GC 管理（V8 用 Orinoco）。

三层内存模型：

  • Linear Memory—— wasm 自己管的 ArrayBuffer；C/C++/Rust 等系统语言用
  • WasmGC heap—— 引擎 GC 管理；Java/Kotlin/Dart 等托管语言用
  • JS heap—— V8 GC；JS 对象 + externref 持有的 JS 对象

详见 P5.2。

03 应用场景

场景 A · 让 Rust wasm 与 JS 交换大量数据

方案：JS 端写入 wasm memory，wasm 端处理，JS 端读结果。

// Rust wasm 导出 process(ptr, len) 处理一段 memory
// JS 端：
const memory = instance.exports.memory;
const ptr = instance.exports.alloc(1024);  // wasm 端分配
const buffer = new Uint8Array(memory.buffer, ptr, 1024);

// 写入数据
const text = new TextEncoder().encode("hello");
buffer.set(text);

// 调用 wasm 处理
const result_ptr = instance.exports.process(ptr, text.length);

// 读结果
const result = new Uint8Array(memory.buffer, result_ptr, /* len */).slice();
const decoded = new TextDecoder().decode(result);

// 释放
instance.exports.free(ptr, 1024);
instance.exports.free(result_ptr, /* len */);

这是wasm-bindgen 自动做的事。手写时容易出错（对齐 / 释放 / endian），用 wasm-bindgen 让宏自动生成 marshalling 代码。详见 P3.2。

场景 B · 用 SharedArrayBuffer 做 wasm threads

详见 P5.1。简短：

// 主线程
const memory = new WebAssembly.Memory({
  initial: 10,
  maximum: 100,
  shared: true       // ← 关键
});
// 必须 cross-origin isolated 才能用

// 多个 worker 共享同一 memory
const worker1 = new Worker('w.js');
worker1.postMessage({ memory });

const worker2 = new Worker('w.js');
worker2.postMessage({ memory });

场景 C · C++ vtable 编译到 wasm table

C++ 源：

class Animal {
public:
  virtual void speak() = 0;
};
class Dog : public Animal {
public:
  void speak() override { puts("woof"); }
};
class Cat : public Animal {
public:
  void speak() override { puts("meow"); }
};

void doSpeak(Animal* a) { a->speak(); }

Emscripten 编译后 wasm（简化）：

(table $vtable 4 funcref)
(elem (i32.const 0) $Dog_speak $Cat_speak ...)

(func $Dog_speak ...)
(func $Cat_speak ...)

(func $doSpeak (param $a i32)   ;; a 是 Animal* (i32 ptr)
  ;; 读对象的 vtable_index 字段（假设偏移 0）
  (local $vtable_idx i32)
  (local.set $vtable_idx (i32.load (local.get $a)))
  ;; 间接调用
  (call_indirect (type $speak_sig) (local.get $a) (local.get $vtable_idx)))

场景 D · 用 Memory64 跑 SQLite 大数据库

import sqlite3InitModule from '@sqlite.org/sqlite-wasm';

场景 E · 用 bulk memory 加速大块清零

// Rust + wasm-bindgen
#[wasm_bindgen]
pub fn clear_buffer(ptr: *mut u8, len: usize) {
    unsafe {
        std::ptr::write_bytes(ptr, 0, len);

04 典型代码：内存 + table 完整示例

例 1：wat 完整 module（memory + alloc + free）

(module
  (memory $mem (export "memory") 1)

  ;; 简单 bump allocator
  (global $heap_top (mut i32) (i32.const 0))

  (func $alloc (export "alloc") (param $size i32) (result i32)
    (local $ptr i32)
    (local.set $ptr (global.get $heap_top))
    (global.set $heap_top
      (i32.add (global.get $heap_top) (local.get $size)))
    ;; 检查越界，必要时 grow
    (if (i32.ge_s
          (global.get $heap_top)
          (i32.mul (memory.size) (i32.const 65536)))
      (then
        (drop (memory.grow (i32.const 1)))))
    local.get $ptr)

  ;; bump allocator 不真正 free，仅占位
  (func $free (export "free") (param $ptr i32) (param $size i32)
    nop))

例 2：在 wasm memory 里实现一个 ring buffer

(module
  (memory $mem 1)
  (global $head (mut i32) (i32.const 0))
  (global $tail (mut i32) (i32.const 0))
  (global $cap i32 (i32.const 1024))

  (func $push (param $val i32) (result i32)
    (local $next i32)
    (local.set $next
      (i32.rem_s
        (i32.add (global.get $head) (i32.const 1))
        (global.get $cap)))
    (if (i32.eq (local.get $next) (global.get $tail))
      (then
        ;; full
        (return (i32.const -1))))
    (i32.store
      (i32.mul (global.get $head) (i32.const 4))
      (local.get $val))
    (global.set $head (local.get $next))
    (i32.const 0))   ;; success

  (func $pop (result i32 i32)   ;; multi-value (status, value)
    (if (i32.eq (global.get $head) (global.get $tail))
      (then
        ;; empty
        (return (i32.const -1) (i32.const 0))))
    (i32.const 0)   ;; status
    (i32.load (i32.mul (global.get $tail) (i32.const 4)))   ;; value
    (global.set $tail
      (i32.rem_s
        (i32.add (global.get $tail) (i32.const 1))
        (global.get $cap))))

  (export "push" (func $push))
  (export "pop" (func $pop)))

例 3：Rust 通过 wasm-bindgen 暴露 memory + alloc

use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn alloc(size: usize) -> *mut u8 {
    let mut buf = Vec::with_capacity(size);
    let ptr = buf.as_mut_ptr();
    std::mem::forget(buf);
    ptr
}

#[wasm_bindgen]
pub fn dealloc(ptr: *mut u8, size: usize) {
    unsafe { Vec::from_raw_parts(ptr, 0, size); }
}

#[wasm_bindgen]
pub fn process(ptr: *mut u8, len: usize) -> usize {
    let slice = unsafe { std::slice::from_raw_parts_mut(ptr, len) };
    // 处理 slice
    for byte in slice.iter_mut() {
        *byte = byte.wrapping_add(1);
    }
    len
}

例 4：JS 端 high-level wrapper

class WasmHelper {
  constructor(instance) {
    this.instance = instance;
    this.memory = instance.exports.memory;
    this.alloc = instance.exports.alloc;
    this.dealloc = instance.exports.dealloc;
  }

  // 把 JS string 写入 wasm memory，返回 (ptr, len)
  writeString(str) {
    const bytes = new TextEncoder().encode(str);
    const ptr = this.alloc(bytes.length);
    new Uint8Array(this.memory.buffer, ptr, bytes.length).set(bytes);
    return { ptr, len: bytes.length };
  }

  // 从 wasm memory 读 string
  readString(ptr, len) {
    const bytes = new Uint8Array(this.memory.buffer, ptr, len);
    return new TextDecoder().decode(bytes);
  }

  // 释放
  free(ptr, len) {
    this.dealloc(ptr, len);
  }
}

05 常见陷阱

陷阱 1：grow 后忘记重新 view ArrayBuffer

最常见 bug。memory.grow() 后旧 TypedArray 可能 detach，访问 throw "ArrayBuffer is detached"。

解决：每次需要 view 时重新创建，不要缓存：

function getView() {
  return new Uint8Array(memory.buffer);
}

// 每次用 getView() 不缓存
const view = getView();
view[0] = 42;

陷阱 2：Rust / C++ alloc 出来的内存忘记 free

wasm 内存泄漏不会让浏览器 OOM 立刻（linear memory 上限 4GB），但长期跑会触发 grow + 内存膨胀。

解决：成对管理 alloc / dealloc；用 wasm-bindgen 的自动管理；或用 try-finally：

const { ptr, len } = helper.writeString("hello");
try {
  return helper.process(ptr, len);
} finally {
  helper.free(ptr, len);
}

陷阱 3：跨边界传字符串忘记编码

wasm memory 是 byte 数组；JS string 是 UTF-16。必须显式编码：

// ❌ 直接拷不行
const view = new Uint8Array(memory.buffer);
const str = "hello";
for (let i = 0; i < str.length; i++) {
  view[i] = str.charCodeAt(i);  // 只对 ASCII work，多字节字符错
}

// ✅ 用 TextEncoder
const bytes = new TextEncoder().encode(str);
new Uint8Array(memory.buffer, ptr, bytes.length).set(bytes);

陷阱 4：以为 align 参数是要求

i32.load align=2 不是说"必须 4 字节对齐"，是提示。不对齐时仍能 load，但慢。

但声明的 align 不能超过指令的自然对齐：i32.load align=4（> 自然对齐 align=2）会被引擎拒绝。

陷阱 5：不释放 data segment

data segment 用于初始化 memory。初始化后不再需要就用 data.drop $seg 释放。否则编译产物里存了一份冗余 data。

Rust / Emscripten 编译器自动处理。手写 wasm 要注意。

陷阱 6：以为 memory 是隔离每个 instance 的

多次 instantiate 同一 module 得到独立 memory（默认）。但如果 import memory 共享：

const sharedMem = new WebAssembly.Memory({ initial: 1 });

const i1 = await WebAssembly.instantiate(module, { env: { memory: sharedMem }});
const i2 = await WebAssembly.instantiate(module, { env: { memory: sharedMem }});
// i1 和 i2 共享 sharedMem，互相能看见对方写入

这是 wasm 多 instance 共享数据的方式（不需要 SAB），但要小心数据竞态。

陷阱 7：Memory64 性能损失

开 Memory64 让 bounds check 走软件路径（不能 mmap guard pages），每次 load 性能损失 ~5-10%。只在真需要 > 4GB 时启用。SQLite-WASM / DuckDB-WASM 等大数据应用值得；普通应用不要 default 开。
