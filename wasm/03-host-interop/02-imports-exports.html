Imports / Exports + 跨边界调用

wasm 模块通过 Imports（声明依赖：函数 / memory / table / global）+ Exports（声明暴露），与宿主交换"代码 + 数据"。跨边界调用两边性能不对称：JS-to-WASM 几乎免费（直接 native call），WASM-to-JS 需要寄存器搬运 + GC barrier 处理，开销大。复杂数据交换的"序列化税"是 wasm 性能心智核心 —— 字符串 / 对象 / 数组必须穿越 linear memory ↔ JS 堆，typical 方案 wasm-bindgen / Embind 自动生成 glue。externref 直传（2.0）让 JS 对象作 opaque 引用直接传给 wasm，不必序列化。

阅读时长 ~30 分钟 前置 P3.1（Linear Memory + Tables） 主要来源 WebAssembly JS API · wasm-bindgen 文档 · Emscripten Embind 文档 · v8.dev/blog

01 速览

  1. Imports / Exports 是 wasm 模块的契约—— 模块声明 imports（外部要提供什么），通过 instantiate 时的 importObject 注入；声明 exports（暴露给外部）。失败的 imports 让 instantiate throw。
  2. WebAssembly.instantiate / instantiateStreaming 是核心 API—— 二者签名 (buffer | module | response, importObject) => Promise<{module, instance}>。Streaming 版边下载边编译，大 module 节省 30-50% 启动。
  3. 跨边界调用两边性能不对称—— JS-to-WASM 现代引擎接近零成本（直接 native call）；WASM-to-JS 需要寄存器搬运 + JS calling convention + 处理 GC barrier，每次调用 ~10-50ns；高频反向调用是性能反模式。
  4. "序列化税"是 wasm 工程化第一难题—— 字符串 / 对象 / 数组在 linear memory 与 JS 堆之间穿越，必须 marshalling。手写易错，wasm-bindgen（Rust）/ Embind（C++）自动生成 glue 是行业标配。
  5. externref（2.0）解决"wasm 持有 JS 对象"问题—— 不必再用 ID + Map hack；JS 对象作 opaque 引用直接传，与 GC barrier 集成。复杂场景（DOM 操作 / 事件系统）依赖此特性。
  6. WebAssembly.compile vs instantiate 解耦—— compile 只 parse + 验证 + 编译（可缓存）；instantiate 给定 imports 创建 instance（瞬时）。"编译一次、多处实例化"在 Web Worker 多 worker / 服务端 multi-tenant 都是关键。

02 底层逻辑

Imports：声明依赖

wat 中 imports 写法：

(module
  ;; 从 "env" 模块导入函数 "log"，签名 (i32) -> ()
  (import "env" "log" (func $log (param i32)))

  ;; 从 "env" 导入 memory（最少 1 页）
  (import "env" "mem" (memory 1))

  ;; 从 "env" 导入 table（10 个 funcref 槽）
  (import "env" "tbl" (table 10 funcref))

  ;; 从 "env" 导入 global
  (import "env" "g" (global i32))

  (func $main
    (call $log (i32.const 42)))
  (start $main))

JS 端 importObject：

const importObject = {
  env: {
    log: (n) => console.log("wasm:", n),
    mem: new WebAssembly.Memory({ initial: 1 }),
    tbl: new WebAssembly.Table({ initial: 10, element: "anyfunc" }),
    g: new WebAssembly.Global({ value: "i32", mutable: false }, 0)
  }
};

const { instance } = await WebAssembly.instantiate(wasmBytes, importObject);

关键约束：

(1) 名称必须匹配—— wasm 声明 (import "env" "log" ...)，JS 端必须提供 importObject.env.log；缺失 throw LinkError

(2) 类型必须匹配—— wasm 声明 (func (param i32) -> ())，JS 端 log 必须是 function（不能 number）；类型不符 throw

(3) instantiate 是 fail-fast—— 任何 import 失败让整个 module 不能用

Exports：声明暴露

wat 中 exports：

(module
  (memory $mem 1)
  (table $tbl 10 funcref)
  (global $g (mut i32) (i32.const 0))

  (func $add (param i32 i32) (result i32)
    (i32.add (local.get 0) (local.get 1)))

  (export "add" (func $add))
  (export "memory" (memory $mem))
  (export "table" (table $tbl))
  (export "globalCounter" (global $g)))

JS 端访问：

const { instance } = await WebAssembly.instantiate(wasmBytes);

instance.exports.add(2, 3);                     // 5
instance.exports.memory;                        // WebAssembly.Memory
instance.exports.table;                         // WebAssembly.Table
instance.exports.globalCounter.value;           // 0（mutable global）
instance.exports.globalCounter.value = 10;      // 设置

四种导出类型 + JS 包装：

  • function—— JS 函数（自动 marshalling 数值类型）
  • memory—— WebAssembly.Memory 对象（.buffer 是 ArrayBuffer）
  • table—— WebAssembly.Table 对象（.get / .set / .grow / .length）
  • global—— WebAssembly.Global 对象（.value getter / setter）

WebAssembly.instantiate 完整 API

// 形式 1：从 buffer / module
WebAssembly.instantiate(
  buffer | module: BufferSource | WebAssembly.Module,
  importObject?: object
): Promise<{ module: WebAssembly.Module, instance: WebAssembly.Instance }>

// 形式 2：streaming（从 Response，浏览器 / Deno / Bun 有）
WebAssembly.instantiateStreaming(
  source: Response | Promise<Response>,
  importObject?: object
): Promise<{ module, instance }>

// 形式 3：仅编译，不实例化
WebAssembly.compile(buffer): Promise<WebAssembly.Module>
WebAssembly.compileStreaming(source): Promise<WebAssembly.Module>

三种使用模式：

(1) 一次性 instantiate—— 最常见，编译 + 实例化合一

const { instance } = await WebAssembly.instantiateStreaming(
  fetch('module.wasm'),
  importObject
);

(2) compile 一次 + instantiate 多次—— Web Worker 多 worker / 服务端 multi-tenant

// 一次编译
const module = await WebAssembly.compileStreaming(fetch('module.wasm'));

// 多次实例化（不同 imports）
const i1 = await WebAssembly.instantiate(module, imports1);
const i2 = await WebAssembly.instantiate(module, imports2);
const i3 = await WebAssembly.instantiate(module, imports3);

// 跨 worker 传 module（transferable）
worker.postMessage({ module });

(3) 同步 API（极少用，性能差）—— new WebAssembly.Module(buffer) + new WebAssembly.Instance(module, imports)。主线程禁用大 module 的同步 API（> 4KB），浏览器会 throw。Worker 内可用。

跨边界调用：两边性能不对称

wasm 与 JS 互调的两个方向性能不一样：

JS-to-WASM（JS 调 wasm 导出函数）

开销：

  • 参数数值类型（i32 / i64 / f32 / f64）—— 直接放寄存器
  • 没有 GC barrier（wasm 内存不在 V8 堆）
  • 没有 type check（wasm 函数签名固定）

结果：现代引擎几乎零成本，~5ns 每次调用。V8 Tier-up 后甚至能 inline wasm 函数到 turbofan IR，开销 0。

WASM-to-JS（wasm 调导入的 JS 函数）

开销：

  • 把 wasm 寄存器值搬到 JS 调用约定（每参数 number 转 V8 SMI / HeapNumber）
  • GC barrier（如果传 externref 给 JS）
  • JS 函数本身的执行开销（解释器 / Ignition / TurboFan）
  • 返回值反向 marshalling

结果：每次调用 ~10-50ns。如果 JS 函数本身复杂，更慢。

性能对照

调用开销/次1M 次开销
纯 JS 函数调用~1-2ns~2ms
JS-to-WASM~5ns~5ms
WASM-to-WASM (内部)~1ns~1ms
WASM-to-JS~20ns~20ms
WASM-to-JS-to-WASM (perform DOM op)~50-100ns~50-100ms

性能心智核心规则：

"在 wasm 内做大块 work，少跨边界"。典型反模式：JS 端循环每次调 wasm 单步 → 跨边界开销吃掉所有性能收益。

序列化税：复杂数据怎么穿越边界

数值类型（i32/f64/...）跨边界几乎免费。复杂数据是难题：

问题 1：字符串—— wasm memory 是 byte array；JS string 是 UTF-16；必须编码 / 解码

问题 2：对象—— wasm 没有"对象"概念；要在 linear memory 里手工序列化字段

问题 3：数组—— 类似对象，更多元素

问题 4：所有权 / 释放—— wasm 端 alloc 的内存什么时候由谁 free？

三种方案：

方案 A · 手工 marshalling

JS 端编码 + 写 memory + 调 wasm + 解码 + free。出错率高，重复劳动：

function callWasmWithString(str) {
  const bytes = new TextEncoder().encode(str);
  const ptr = instance.exports.alloc(bytes.length);
  new Uint8Array(memory.buffer, ptr, bytes.length).set(bytes);
  const result_ptr = instance.exports.process(ptr, bytes.length);
  // ... 读 result ...
  instance.exports.free(ptr, bytes.length);
  return result;
}

方案 B · wasm-bindgen（Rust 主流）

Rust 端用 #[wasm_bindgen] 宏：

use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn greet(name: &str) -> String {
    format!("Hello, {}!", name)
}

JS 端自动得到 marshalling 后的接口：

import { greet } from './pkg/my_wasm.js';
const result = greet("alice");  // wasm-bindgen 自动 alloc / encode / call / decode / free
console.log(result);  // "Hello, alice!"

wasm-bindgen 生成的 glue 处理：

  • 字符串 UTF-8 编码 / 解码
  • memory alloc / free（用 Rust 的 std::alloc）
  • JS object <-> Rust struct 通过 ID + Map（旧方案）或 externref（新方案）
  • 异常传播（Rust panic → JS Error）
  • Promise / Future（async wasm 函数）

方案 C · Embind（C++ 主流）

C++ 用 Embind 宏：

#include <emscripten/bind.h>
using namespace emscripten;

std::string greet(std::string name) {
    return "Hello, " + name + "!";
}

EMSCRIPTEN_BINDINGS(my_module) {
    function("greet", &greet);
}

JS 端：

await Module.then(m => {
  console.log(m.greet("alice"));  // "Hello, alice!"
});

Embind 比 wasm-bindgen 重（依赖 Emscripten runtime），但功能丰富（C++ class / inheritance / smart pointers）。

externref 直传：从 ID hack 到一等公民

MVP 时 wasm 持有 JS 对象用 ID + Map：

// 旧方案
const objMap = new Map();
let nextId = 0;

function jsToWasm(obj) {
  const id = nextId++;
  objMap.set(id, obj);
  return id;
}

function wasmToJs(id) {
  return objMap.get(id);
}

// wasm 调 JS 函数处理 obj
const id = jsToWasm({ name: "alice" });
instance.exports.processObj(id);  // wasm 拿到 i32 id
// wasm 端调 import "deref"(id) 用 wasmToJs 拿到 obj

问题：

  • obj 不会被 GC（被 objMap 引用）—— 内存泄漏
  • 需要手动管理生命周期（释放 id）
  • 跨 module 共享 id 不可能

2.0 后 externref：

(module
  (import "env" "process" (func $process (param externref)))

  (func (export "main") (param $obj externref)
    (call $process (local.get $obj))))

JS 端：

const obj = { name: "alice" };
instance.exports.main(obj);  // 直接传，不用 ID

引擎自动处理：

  • obj 仍在 V8 堆，由 V8 GC 管理
  • wasm 持有 externref（V8 内是带类型的 tagged pointer）
  • wasm 不能 deref externref（不能读字段）；只能传给其他 import 函数
  • obj 被 wasm 引用时不会被 V8 GC（V8 知道 wasm 持有它）
  • wasm 不再持有时 obj 可被 V8 GC

这让"wasm 操作 DOM / 事件 / 任意 JS 对象"自然，不需要 ID hack。

JSPI：让同步 wasm 调异步 JS（短重述链回 V8）

典型问题：你把 SQLite C 代码编译成 wasm，SQLite 内部用同步 IO 假设（read / write 同步）。但浏览器的 IndexedDB / fetch 是异步。怎么办？

MVP 方案：把所有 SQLite 代码改成异步（要用 Asyncify hack 转换栈）。

JSPI 方案（phase 4，Chrome 137+）：

const sqliteWasm = await WebAssembly.instantiateStreaming(fetch('sqlite.wasm'), {
  env: {
    // 这个函数返回 Promise，但 wasm 端调它是同步语义
    fs_read: WebAssembly.promising(async (fd, buf, len) => {
      const data = await idb.read(fd, len);
      // ... 写 buf ...
      return data.length;
    })
  }
});

// wasm 端调 fs_read，看起来同步但底层 await
sqliteWasm.exports.exec_query("SELECT * FROM users");

底层用 stack switching：wasm 栈被保存，JS 主线程继续；Promise 完成后栈恢复。详见 V8 P6.1。

module caching：编译产物复用

WebAssembly.compile 产出 WebAssembly.Module 是 transferable（postMessage 可传）+ structuredClone-able + 可序列化（implicit 浏览器缓存）。

浏览器自动缓存编译产物—— 第二次 fetch 同 URL wasm，引擎直接复用上次编译结果。Chrome / Firefox / Safari 都实现 IDB-backed wasm cache。

跨 worker 共享 module—— 主线程编译一次，postMessage 给所有 worker 共用：

// 主线程
const module = await WebAssembly.compileStreaming(fetch('worker.wasm'));
for (let i = 0; i < 8; i++) {
  const w = new Worker('w.js');
  w.postMessage({ module });
}

// worker
self.onmessage = async ({ data: { module } }) => {
  const instance = await WebAssembly.instantiate(module, imports);
  // 编译已完成，instantiate 几乎瞬时
};

03 应用场景

场景 A · Rust 库通过 wasm-bindgen 暴露给 JS

典型项目结构（Cargo.toml）：

[package]
name = "my-wasm"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
wasm-bindgen = "0.2"
js-sys = "0.3"
web-sys = { version = "0.3", features = ["console", "Document", "Window"] }

[profile.release]
opt-level = "z"
lto = "fat"
codegen-units = 1
panic = "abort"

Rust 代码：

use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn process_array(input: &[f64]) -> Vec<f64> {
    input.iter().map(|x| x * 2.0).collect()
}

#[wasm_bindgen]
pub struct Counter {
    value: i32,
}

#[wasm_bindgen]
impl Counter {
    #[wasm_bindgen(constructor)]
    pub fn new() -> Self { Self { value: 0 } }
    pub fn increment(&mut self) { self.value += 1; }
    pub fn get(&self) -> i32 { self.value }
}

构建：

$ wasm-pack build --target web --release

JS 使用：

import init, { process_array, Counter } from './pkg/my_wasm.js';
await init();

const result = process_array([1.0, 2.0, 3.0]);  // [2, 4, 6]

const counter = new Counter();
counter.increment();
counter.increment();
console.log(counter.get());  // 2

场景 B · 不用 wasm-bindgen 直接手工 marshalling（极简场景）

简单纯数值函数不需要 wasm-bindgen（额外体积 + 工具链复杂度）：

// no_std lib，无 wasm-bindgen
#[no_mangle]
pub extern "C" fn add(x: i32, y: i32) -> i32 {
    x + y
}

编译：

$ rustc --target wasm32-unknown-unknown -O --crate-type cdylib lib.rs

JS 直接调（< 1KB wasm）：

const { instance } = await WebAssembly.instantiateStreaming(fetch('lib.wasm'));
console.log(instance.exports.add(2, 3));  // 5

场景 C · 多 worker 共享编译产物

// 主线程：编译一次
const heavyModule = await WebAssembly.compileStreaming(fetch('heavy.wasm'));

// 4 个 worker 共享
const workers = Array.from({ length: 4 }, () => new Worker('worker.js'));
for (const w of workers) {
  w.postMessage({ module: heavyModule });
}

// worker.js
self.onmessage = async ({ data: { module } }) => {
  // 不需重编，instantiate 瞬时
  const instance = await WebAssembly.instantiate(module, imports);
  // ... 处理任务 ...
};

对比"每个 worker 自己 compile"：节省 ~80% wasm 启动时间（4 worker × 200ms compile = 800ms → 200ms 主线程一次编译）。

场景 D · 用 externref 操作 DOM

(module
  (import "dom" "querySelector" (func $qs (param externref) (result externref)))
  (import "dom" "setTextContent" (func $setText (param externref) (param externref)))

  (func (export "updateTitle") (param $doc externref) (param $newText externref)
    (call $setText
      (call $qs (local.get $doc) (... selector externref ...))
      (local.get $newText))))

JS 端：

const importObject = {
  dom: {
    querySelector: (parent, selector) => parent.querySelector(selector),
    setTextContent: (el, text) => { el.textContent = text; }
  }
};

实际上：直接 wasm 操作 DOM 跨边界税重，开销不低于纯 JS 操作。典型应用仍把 DOM 操作放 JS，wasm 只算"DOM 该是什么状态"。Yew / Leptos 等 Rust SPA 框架就是这种架构。

场景 E · JSPI 让同步 SQLite 调异步 OPFS

const opfsRoot = await navigator.storage.getDirectory();
const fileHandle = await opfsRoot.getFileHandle('db.sqlite', { create: true });
const accessHandle = await fileHandle.createSyncAccessHandle();

const sqliteWasm = await WebAssembly.instantiateStreaming(fetch('sqlite.wasm'), {
  env: {
    // 同步语义包装异步 IO
    fs_read: WebAssembly.promising(async (fd, buf_ptr, len) => {
      const buf = new Uint8Array(memory.buffer, buf_ptr, len);
      return accessHandle.read(buf, { at: ... });
    }),
    fs_write: WebAssembly.promising(async (fd, buf_ptr, len) => {
      const buf = new Uint8Array(memory.buffer, buf_ptr, len);
      return accessHandle.write(buf, { at: ... });
    })
  }
});

// SQLite C 代码看起来在做同步 IO，实际底层 await
sqliteWasm.exports.execute_query("SELECT * FROM users");

04 典型代码：完整项目

例 1：image processing wasm（Rust + wasm-bindgen）

use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn grayscale(input: &[u8], width: u32, height: u32) -> Vec<u8> {
    let mut output = Vec::with_capacity(input.len());
    for chunk in input.chunks(4) {
        // RGBA → grayscale
        let r = chunk[0] as u32;
        let g = chunk[1] as u32;
        let b = chunk[2] as u32;
        let gray = ((r * 299 + g * 587 + b * 114) / 1000) as u8;
        output.extend_from_slice(&[gray, gray, gray, chunk[3]]);
    }
    output
}

JS 调用：

import init, { grayscale } from './pkg/img.js';
await init();

// 从 canvas 读 ImageData
const imageData = ctx.getImageData(0, 0, w, h);
// 调 wasm 处理（自动 marshalling Uint8Array ↔ Vec<u8>）
const grayBytes = grayscale(imageData.data, w, h);
// 写回 canvas
const newImageData = new ImageData(new Uint8ClampedArray(grayBytes), w, h);
ctx.putImageData(newImageData, 0, 0);

例 2：Counter class（带 state）

use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub struct Counter {
    count: i32,
    history: Vec<i32>,
}

#[wasm_bindgen]
impl Counter {
    #[wasm_bindgen(constructor)]
    pub fn new() -> Self { Self { count: 0, history: Vec::new() } }

    pub fn increment(&mut self) {
        self.count += 1;
        self.history.push(self.count);
    }

    pub fn decrement(&mut self) {
        self.count -= 1;
        self.history.push(self.count);
    }

    pub fn get(&self) -> i32 { self.count }
    pub fn history_len(&self) -> usize { self.history.len() }
}

JS：

const c = new Counter();
c.increment();
c.increment();
c.decrement();
console.log(c.get());           // 1
console.log(c.history_len());   // 3

例 3：双向回调（wasm 持有 JS 函数）

use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn run_with_callback(items: Vec<i32>, cb: &js_sys::Function) {
    for item in items {
        let this = JsValue::null();
        let arg = JsValue::from(item);
        let _ = cb.call1(&this, &arg);
    }
}

JS：

run_with_callback([1, 2, 3, 4, 5], (item) => {
  console.log(`got: ${item}`);
});

性能注意：每次 cb.call1 是 wasm-to-JS 跨边界 ~20ns；5 次 ok，但1M 次会很慢。在 wasm 内做大块 work，最后只回调 1 次更优。

例 4：手写 import / export（不用 bindgen）

(module
  (import "console" "log" (func $log (param i32)))
  (import "math" "random" (func $rand (result f64)))

  (memory $mem (export "memory") 1)

  (func (export "test")
    ;; 调 import
    (call $log (i32.const 42))
    ;; 调 import 拿 random
    (drop (call $rand))))

JS：

const { instance } = await WebAssembly.instantiate(wasmBytes, {
  console: {
    log: (n) => console.log("wasm logs:", n)
  },
  math: {
    random: () => Math.random()
  }
});

instance.exports.test();
// 输出 "wasm logs: 42"

05 常见陷阱

陷阱 1：高频跨边界调用

反模式：

// JS 端循环每次调 wasm
for (let i = 0; i < 1_000_000; i++) {
  result += instance.exports.compute_one(input[i]);
}
// 1M × ~5ns = 5ms，仅跨边界开销

正确：

// 把循环移进 wasm
const result = instance.exports.compute_all(inputArray);
// 一次跨边界，wasm 内做 1M iter

典型 ratio：每 100 行 wasm 工作配 1 次跨边界调用比较健康。

陷阱 2：用 wasm-bindgen 但不优化体积

wasm-bindgen 默认生成的 wasm + JS glue 可能 200KB+。必须配置 release 优化：

[profile.release]
opt-level = "z"
lto = "fat"
codegen-units = 1
panic = "abort"
strip = true

+ wasm-pack build --release + wasm-opt -Oz。能压到 ~10-50KB。

陷阱 3：忘记 importObject 字段

wasm 声明 imports，JS 端 importObject 缺字段 → instantiate throw LinkError。错误信息通常清晰：

LinkError: WebAssembly.instantiate(): Import #0 module="env" function="log" error: function import requires a callable

陷阱 4：以为 streamingInstantiate 能处理任意 fetch

streamingInstantiate 要求 Content-Type: application/wasm。后端没配置 → 浏览器警告 + 回退到非流式（性能损失）：

[warning] WebAssembly.instantiateStreaming(): Response has unsupported MIME type 'text/plain' expected 'application/wasm'

修复：Nginx types { application/wasm wasm; }；Express app.use(express.static('public', { setHeaders: (res, path) => { if (path.endsWith('.wasm')) res.set('Content-Type', 'application/wasm'); }}))。

陷阱 5：以为 wasm 函数自动 throw JS Error

wasm 内部 trap（除 0 / 越界 / unreachable / call_indirect 类型不匹配）会让导出函数 throw WebAssembly.RuntimeError。但 Rust panic 不直接 throw；要配 panic_hook：

#[wasm_bindgen(start)]
pub fn main() {
    console_error_panic_hook::set_once();
    // 之后 Rust panic 会 console.error + throw JS Error
}

陷阱 6：i64 参数 / 返回值忘记 BigInt

// wasm: (func $foo (param i64) (result i64))
instance.exports.foo(42);    // ❌ TypeError: Cannot convert 42 to a BigInt
instance.exports.foo(42n);   // ✅
const result = instance.exports.foo(42n);  // 返回 BigInt

陷阱 7：不释放 wasm-bindgen 创建的 Rust struct

const counter = new Counter();
// ... 用一段时间 ...
// 不调 counter.free()，Rust 端的 struct 不会释放
// 直到 JS 端 counter 变量被 GC（最终会 free，但延迟不可控）

// 良好实践
counter.free();  // 显式释放

新版 wasm-bindgen 支持 FinalizationRegistry 自动释放，但延迟不可控；性能敏感场景显式 free。
