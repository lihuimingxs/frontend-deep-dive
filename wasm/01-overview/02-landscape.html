深度学习/ WebAssembly/ P1 · WASM 是谁/ 02 · 同代人对照

同代人对照：NaCl / asm.js / WebGL2 compute / WebGPU

横向对照"想在浏览器跑性能代码"的所有方案。NaCl / PNaCl（2009-2017，Chrome 锁定，死）+ asm.js（2013-2018，"标注的 JS"，被 wasm 替代）+ WebGL2 / WebGPU compute（GPU 优先，并行强但限制多）+ WebAssembly（活到今天的唯一通用 CPU 性能通道）。核心洞察：跨引擎是底线、二进制格式不可选、CPU 和 GPU 是互补不是替代、服务端化是 NaCl 没做到的"质变"。

阅读时长 ~25 分钟 前置 P1.1（项目史） 主要来源 webassembly.org · webgpu.org · MDN · Chromium / Mozilla 博客

01 速览

  1. "想跑性能代码"在 Web 上有 4 条历史路线—— NaCl（native 二进制 + SFI 沙箱）/ asm.js（JS 严格子集 + AOT）/ GPU compute（WebGL2 → WebGPU）/ wasm（独立 ISA + 二进制）。前两个死了，后两个活着且互补。
  2. 跨引擎是底线—— NaCl 失败的核心原因不是技术烂，是 Apple/Mozilla 不接受 Chrome 单家方案。任何"性能通道"想活，必须四大引擎联合实现。Flash / Silverlight / NaCl 都死在这条上。
  3. 二进制格式是"非可选"—— asm.js 跨引擎但下载体积大（hello world ~10MB+）、parse 慢；wasm 紧凑二进制 + streaming compile 是"性能心智"基础。
  4. CPU vs GPU 是互补—— wasm 跑通用 CPU 代码（控制流复杂、串行也可），WebGPU 跑大规模并行（图像 / AI 推理）。Figma 用 wasm 跑布局引擎 + WebGPU 跑加速渲染——典型互补。
  5. wasm 走出浏览器是 NaCl 没做到的"质变"—— NaCl 只活在浏览器；wasm 不仅活了，还把战场扩到 Edge / 嵌入式 / 区块链。这背后是独立 ISA + 多语言 + capability-based 安全三个设计的累积效应。

02 底层逻辑

设计选择对比表（2026-05 现状）

维度 NaCl/PNaCl asm.js WebGL2/WebGPU compute WebAssembly

方案 1：NaCl / PNaCl（2009-2017，已死）

定位：在浏览器里跑 native 代码 + SFI 沙箱 + Pepper API（PPAPI）暴露浏览器 API

技术核心：

(1) SFI（Software Fault Isolation）—— 编译时插入运行时检查（每次跳转 / 内存访问验证）；让恶意 native 代码不能跳出 sandbox 区。原理与 Linux seccomp 类似但软件实现

(2) NaCl module 是 ELF—— Linux 风格的 ELF 文件，与系统二进制几乎一致；ISA 锁定 x86 / ARM

(3) PNaCl 用 LLVM bitcode—— 客户端编译；解决 ISA 锁定但启动慢、体积大

(4) Pepper API—— 浏览器暴露给 NaCl 的接口；类似 NPAPI 但更安全；只有 Chrome 实现

失败原因（按重要性排序）：

(1) 跨引擎失败—— Mozilla / Apple 公开拒绝；理由是"NaCl 把 Web 锁定在 Chrome + native 二进制";Mozilla 同期推 asm.js 作为对抗方案

(2) ISA 锁定 / 体积问题—— NaCl 锁 x86/ARM；PNaCl 用 LLVM bitcode 体积大（hello world ~5MB）+ 启动慢（要本地编译）

(3) 安全审查复杂—— SFI 验证器要理解 x86 / ARM 全部指令；新指令 / 新 CPU 都要审；2018 Spectre 让 SFI 假设崩溃

(4) 工具链锁定 Chrome—— Pepper API 让生态绑死；开发者要为 NaCl 学一套独立 API

2015 wasm 提案后，NaCl 就被 Chrome 内部"逐步弃用"。2017-05 Chromium 博客 Goodbye PNaCl, Hello WebAssembly 正式宣告 NaCl 死。

NaCl 留给 wasm 的教训：

  • 必须跨引擎（不能 Chrome only）
  • 必须独立 ISA（不能锁 x86 / ARM）
  • 验证器必须简单（wasm 的 load-time validation 比 SFI 简单一个数量级）
  • Pepper API 的"浏览器特定"教训 → wasm 不直接暴露浏览器 API，通过 imports + JS glue 间接访问

方案 2：asm.js（2013-2018，被替代）

定位：把 C/C++ 编译到 JS 严格子集，用 JS 引擎已有的 AOT 路径跑

技术核心：

(1) "use asm" 指令—— 函数顶部 "use asm";，引擎检测后切 AOT 编译路径

(2) 类型注解 hack—— 用 JS 已有运算符做类型标注：x | 0（i32）、+x（f64）、x >>> 0（u32）、fround(x)（f32）

(3) Heap = ArrayBuffer—— 所有"内存"是一个 ArrayBuffer，TypedArray view 访问；模拟 C 的 linear memory

(4) Emscripten 工具链—— Mozilla 主导；clang → LLVM bitcode → asm.js；自动生成"模拟 POSIX"的 JS glue

典型 asm.js 代码：

function asmModule(stdlib, ffi, heap) {
  "use asm";
  var HEAP32 = new stdlib.Int32Array(heap);
  var imul = stdlib.Math.imul;
  function fib(n) {
    n = n | 0;          // n: i32
    if ((n | 0) < 2) return n | 0;
    return (fib((n - 1) | 0) + fib((n - 2) | 0)) | 0;
  }
  return { fib: fib };
}

成绩：

  • 2014 Unreal Engine 4 跑在 Firefox（Citadel demo）
  • 2015 Unity 5 默认导出 asm.js
  • 2017 wasm 出来前，asm.js 是 Web 性能代码事实标准

失败原因：

(1) 仍是 JS 文本—— 下载 / parse 都比二进制慢；hello world ~10MB+

(2) 标注 hack—— x | 0 是利用 JS 现有运算符做类型注解，本质 hack；不是干净设计

(3) 单语言（C/C++）—— Emscripten 是唯一成熟工具链；其他语言（Rust / Go / Java）支持差

(4) 性能上限 ~80% native—— 因为引擎仍要 parse JS、做一些 JS 语义保证；纯二进制能再 push 10-15%

2017 wasm MVP 启用后，Emscripten 默认输出从 asm.js 切换到 wasm。2018 年起 asm.js 实际上退役（仍被引擎兼容，但新项目都用 wasm）。

asm.js 留给 wasm 的教训：

  • 跨引擎确实有可能（不像 NaCl 锁 Chrome）
  • 类型化的"JS 子集"性能能接近 native（论证 wasm 性能可行）
  • C/C++ 进浏览器有市场（Unity / Unreal 推动）
  • 但下一代必须二进制（asm.js 自身证明文本格式有上限）
  • 必须多语言（不能只服务 C/C++）

方案 3：WebGL2 / WebGPU compute（活，GPU 优先）

定位：用 GPU shader 跑通用计算（不只是图形）

技术核心：

(1) WebGL2（2017）+ Compute shader 提案—— 通过 GLSL ES 3.10 写 compute shader；浏览器实现差强人意，Compute shader 在 WebGL2 上没真正通用

(2) WebGPU（2023 W3C, Chrome 113+ / Safari 18 / Firefox 141）—— 现代 GPU API，对应 Vulkan / Metal / DX12；正式支持 compute shader

(3) WGSL（WebGPU Shading Language）—— 替代 GLSL；Rust 风格语法；类型严格

(4) Buffer / Texture binding—— GPU 内存通过 buffer 与 CPU（JS / wasm）交换

典型 WGSL compute shader（向量加法）：

@group(0) @binding(0) var<storage, read> a: array<f32>;
@group(0) @binding(1) var<storage, read> b: array<f32>;
@group(0) @binding(2) var<storage, read_write> c: array<f32>;

@compute @workgroup_size(64)
fn main(@builtin(global_invocation_id) gid: vec3<u32>) {
  let i = gid.x;
  c[i] = a[i] + b[i];
}

定位差异（vs wasm）：

(1) GPU 适合大规模并行—— 千万级数据相同操作；图像 / 视频 / AI 推理是黄金场景

(2) GPU 不适合控制流复杂代码—— branch divergence 严重伤性能；不能 malloc / 任意内存访问

(3) GPU 不能链接 C / Rust 库—— shader 是独立小语言，没法把 SQLite / FFmpeg 移植上来

(4) GPU 数据传输是瓶颈—— CPU ↔ GPU 内存拷贝开销大；不适合"小批数据频繁来回"

WebGPU 与 wasm 的互补关系：

  • Figma：wasm 跑布局引擎（控制流复杂）+ WebGPU 加速渲染（大规模 raster）
  • Photoshop Web：wasm 跑滤镜算法（移植 C++）+ WebGPU 跑实时预览
  • AI 推理（onnxruntime-web / TensorFlow.js）：wasm 跑 op 调度 + WebGPU 跑矩阵运算

WebGPU 的限制：

  • 启动慢（device init + shader compile 几十 ms）
  • 不能跑通用 CPU 代码（不能跑游戏逻辑 / 业务代码）
  • 移动设备 GPU 差异大（性能 / 兼容性）
  • 2026 仍在快速迭代（subgroups / mesh shaders / ray tracing 提案中）

方案 4：WebAssembly（活，唯一通用 CPU 性能通道）

已在 P1.1 详述。这里只对照其他方案的关键差异：

(1) vs NaCl：

  • NaCl: ELF + SFI + Chrome only → 死
  • wasm: 独立 ISA + load-time validation + 4 引擎联合 → 活
  • 关键差：跨引擎 + 独立 ISA

(2) vs asm.js：

  • asm.js: JS 文本子集 + AOT hint + 单 C/C++ 工具链 → 退役
  • wasm: 二进制 + 独立 spec + 多语言 → 活
  • 关键差：二进制格式 + 多语言

(3) vs WebGPU：

  • WebGPU: GPU 优先，大规模并行；不能跑通用 CPU 代码
  • wasm: CPU 优先，控制流任意；不擅长大规模并行（需要 SIMD + threads，仍有限）
  • 关键差：CPU vs GPU 是互补不是替代

三引擎对 wasm 的实现差异（2026-05）

wasm spec 是统一的，但实际实现细节有差异：

引擎 编译流水线 WasmGC JSPI 新提案速度
V8 (Chrome/Edge/Brave) Liftoff (baseline) + TurboFan (optimizing) + Tier-up（详见 V8 P6.1） ✅（共享 Orinoco GC） ✅ Chrome 137+ 最快
SpiderMonkey (Firefox) WasmBaseline (baseline) + Ion (optimizing) ✅（自实现 GC） 实验中 跟随，标准最严
JavaScriptCore (Safari) BBQ (baseline) + OMG (optimizing) ✅ Safari 18.2+ 计划中 较慢，更保守

实际工程影响：新提案要等 3 个引擎都 ship 才能放心用。Chrome 抢先 1-2 年实现是常态。

服务端 wasm runtime（不在浏览器里）

详见 P6.2，这里只对照定位：

  • wasmtime（Bytecode Alliance, Rust + Cranelift）—— 事实标准
  • wasmer（商业公司，Rust，多后端 Cranelift/LLVM/Singlepass）—— 跨语言绑定丰富
  • wazero（Tetrate, 纯 Go）—— Go 项目嵌入 wasm 不需 CGO
  • WasmEdge（CNCF, C++）—— 针对 cloud-native + AI 推理
  • V8 isolate / JSC—— Cloudflare Workers / Bun 用 JS 引擎跑 wasm，与 JS 共享 runtime

03 应用场景：选择哪条路

场景 A · 浏览器内跑既有 C++/Rust 代码库

选 wasm。NaCl 死了，asm.js 退役了，WebGPU 不能跑通用 CPU 代码。wasm 是唯一选择。

典型：Figma（C++ 布局引擎）/ Photoshop Web / FFmpeg.wasm / SQLite WASM / Pyodide / DuckDB-WASM。

场景 B · 浏览器内跑大规模并行计算（图像 / AI 推理 / 物理）

选 WebGPU，可能配合 wasm 做调度。

典型：onnxruntime-web（WASM + WebGPU 双 backend）/ TensorFlow.js / WebLLM / 浏览器内 Stable Diffusion / 图像滤镜。

选择规则：(1) 数据量 < 1MB 或控制流复杂 → wasm；(2) 数据量 > 10MB 且高度并行 → WebGPU；(3) 中间地带 → 实测两者。

场景 C · 浏览器内业务逻辑（CRUD / 表单 / 简单交互）

选 JS / TS，不要用 wasm 也不要用 WebGPU。

原因：(1) 现代 V8 跑 JS 已经接近 native 80%+；(2) 业务代码工具链 / 招人 / 调试 JS 完胜；(3) wasm 启动开销 + glue 体积 + 跨边界税在小工具上得不偿失。

场景 D · 服务端 / Edge 多租户沙箱

选 wasm + WASI。NaCl / asm.js / WebGPU 在服务端都没有意义。

典型：Cloudflare Workers（V8 isolate 跑 wasm）/ Fastly Compute@Edge（wasmtime）/ Shopify Functions / Envoy plugins。

场景 E · 区块链智能合约

选 wasm。早期区块链用自家 VM（EVM / Solana BPF），新一代普遍 wasm 化（Polkadot / NEAR / Internet Computer / CosmWasm）。原因：跨语言 + 沙箱 + 性能 + 生态。

04 典型代码：相同任务四种方案

任务：计算 1M 个 float32 的两两相加

方案 A · 纯 JS

function addJS(a, b) {
  const c = new Float32Array(a.length);
  for (let i = 0; i < a.length; i++) {
    c[i] = a[i] + b[i];
  }
  return c;
}

性能基准（V8, 1M 元素）：~5ms。简单、无依赖。

方案 B · asm.js（已退役）

function asmAdd(stdlib, ffi, heap) {
  "use asm";
  var f32 = new stdlib.Float32Array(heap);
  function add(aPtr, bPtr, cPtr, n) {
    aPtr = aPtr | 0; bPtr = bPtr | 0;
    cPtr = cPtr | 0; n = n | 0;
    var i = 0;
    for (; (i | 0) < (n | 0); i = (i + 1) | 0) {
      f32[(cPtr + (i << 2)) >> 2] =
        +f32[(aPtr + (i << 2)) >> 2] + +f32[(bPtr + (i << 2)) >> 2];
    }
  }
  return { add: add };
}

性能（Firefox 旧版）：~3ms。2026 没有引擎再为 asm.js 做特殊优化，仍按普通 JS 跑，性能与方案 A 相当。

方案 C · WASM（Rust + wasm-bindgen）

// lib.rs
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn add(a: &[f32], b: &[f32]) -> Vec<f32> {
    a.iter().zip(b).map(|(x, y)| x + y).collect()
}

JS 端：

import { add } from './pkg/my_wasm.js';
const c = add(a, b);  // wasm-bindgen 自动 marshalling

性能：~2.5ms（无 SIMD）/ ~0.8ms（带 SIMD wasm32-unknown-unknown +simd128）。SIMD 是 wasm 的杀手锏。

方案 D · WebGPU compute shader

// add.wgsl
@group(0) @binding(0) var<storage, read> a: array<f32>;
@group(0) @binding(1) var<storage, read> b: array<f32>;
@group(0) @binding(2) var<storage, read_write> c: array<f32>;

@compute @workgroup_size(64)
fn main(@builtin(global_invocation_id) gid: vec3<u32>) {
  c[gid.x] = a[gid.x] + b[gid.x];
}

JS 端：

// 简化代码：实际要 device.createBuffer / writeBuffer / setBindGroup / dispatch / submit
const c = await runComputeShader(shaderCode, [a, b]);

性能：~10ms 计算 + ~5ms 内存传输 = 15ms 总。1M 元素太小，GPU 启动开销吃掉收益。10M 元素后 GPU 才领先；100M 元素 GPU 是 wasm 的 5-10x。

性能对照表

数据量 JS WASM (无 SIMD) WASM + SIMD WebGPU 推荐
1K ~0.05ms ~0.03ms ~0.01ms ~5ms (启动) JS
1M ~5ms ~2.5ms ~0.8ms ~15ms WASM + SIMD
100M ~500ms ~250ms ~80ms ~30ms WebGPU

规律：小数据 JS 最优（启动开销小）；中等数据 WASM SIMD 最优（GPU 启动太贵）；大数据 GPU 最优。

05 常见陷阱

陷阱 1：以为 WebGPU 会取代 wasm

错误认知："GPU 比 CPU 快，所以 WebGPU 是未来，wasm 是过渡"。

实际：GPU 只在大规模高度并行场景快。控制流复杂 / 数据量小 / 串行依赖 / 不规则内存访问 —— GPU 都是输的。CPU 通用代码永远需要 wasm，GPU 永远是补充。

陷阱 2：以为 asm.js 已经"完全没用"

2026 仍有少量场景 asm.js 有意义：(1) 极老旧浏览器（IE 11 等仍偶尔遇到）的 fallback；(2) 某些 polyfill 需要 fallback 路径。但新项目不要再写 asm.js。

陷阱 3：把 NaCl 教训过度推广到 wasm

有人因为"NaCl 死了"对 wasm 也悲观。错误。NaCl 失败的核心是Chrome 单家方案，不是技术路线。wasm 跨引擎从一开始就解决了这个根本问题。

陷阱 4：选错"性能通道"

常见错误：

  • 简单 CRUD 用 wasm（启动开销 + glue 体积反而拖慢）
  • 大规模图像处理用 wasm（应该用 WebGPU）
  • ML 推理只用一种（onnxruntime-web 同时用 wasm + WebGPU 是经典案例）

正确判据：(1) 数据量；(2) 计算特点（并行 vs 串行 / 控制流复杂度）；(3) 已有代码资产；(4) 启动时间预算。综合判断。

陷阱 5：以为"wasm 主导浏览器"

2026 年浏览器代码量 95%+ 仍是 JS / TS。wasm 是性能敏感模块的 5%，不是替代 JS 的"主流"。把 wasm 看成"高性能补丁"，不是"下一代 JS"。
