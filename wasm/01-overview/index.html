P1 · WASM 是谁

不讲技术细节，只解决两个问题：wasm 17 年（含前史 NaCl / asm.js）怎么走到今天、同代人现在的格局是什么。给后面所有"为什么这样设计"提供历史坐标。

章节数 2 主要来源 webassembly.org · hacks.mozilla.org · v8.dev/blog 状态 ✅ 全部完成

本阶段覆盖

  • NaCl / PNaCl 失败（2009-2017）—— Chrome 单家方案 + 平台锁定 → 死
  • asm.js 部分成功（2013-2018）—— JS 严格子集 + AOT 编译 + 全引擎 → 让 Unreal Engine 跑在浏览器，但仍是"标注的 JS"，体积大
  • 2015 wasm 提案—— Mozilla / Google / Microsoft / Apple 四大引擎团队罕见联合；目标 = asm.js 的二进制版 + 多语言 + 跨引擎
  • 2017-03 MVP—— 四大浏览器同时启用；功能极小（i32/i64/f32/f64 + 线性内存 + 基本控制流）
  • 2019-12 W3C 推荐 1.0—— 与 HTML / CSS / DOM 同等地位
  • 2020 WASI Preview 1—— wasm 走出浏览器；Bytecode Alliance 成立
  • 2023 WasmGC 落地—— Java / Kotlin / Dart 等托管语言一等公民
  • 2025-03 WASM 2.0—— SIMD / bulk memory / reference types / multi-value 全部正式化
  • 2026 Component Model + WASI Preview 2—— wasm 成为"跨语言二进制库格式"
  • 同代人对照—— NaCl / PNaCl / asm.js / WebGL2 compute / WebGPU 各自定位与失败/留存原因
