系统语言：Rust / C / C++ / Zig

手动内存管理 + LLVM 后端语言怎么编译到 wasm。Rust 是 wasm 一等公民（70%+ 生产 wasm），rustc --target wasm32-unknown-unknown 直接产出 + wasm-pack/wasm-bindgen 工具链生成 npm 包；C/C++ via Emscripten 让现有代码不改源就能跑（带 libc 模拟，产物大），现代倾向 WASI SDK + clang 直编（更轻、与 WASI 自然集成）；Zig 原生支持 wasm32 / wasm64 target，编译器内置不需要 emscripten，"显式分配"哲学契合 wasm 内存模型。代码体积管理是工程化第一关：wasm-opt + strip + panic abort + lto fat + opt-level z 是组合拳。

阅读时长 ~30 分钟 前置 P3（与宿主互操作） 主要来源 rustwasm.github.io · emscripten.org · ziglang.org · wasi-sdk

01 速览

  1. Rust 是 wasm 黄金组合—— 70%+ 生产 wasm 用 Rust。原因：(1) LLVM 后端、(2) 无 GC 不带 runtime、(3) wasm-bindgen 自动 marshalling、(4) Cargo 生态、(5) Mozilla 同时推 Rust 和 wasm。Cloudflare Workers / Shopify Functions / Figma 等大型 wasm 项目都 Rust 主导。
  2. C/C++ via Emscripten 是历史路径—— Emscripten = clang + 模拟 POSIX runtime + JS glue。让 Unity / Unreal / FFmpeg / SQLite 等现有 C/C++ 代码不改源就能跑。但 产物大（hello world > 100KB）、启动慢；现代趋势是 WASI SDK + clang 直编（轻、与 WASI 集成）。
  3. Zig 原生 wasm 支持—— Zig 编译器内置 wasm32 / wasm64 target，不需要 Emscripten。Zig 的 "显式分配 + comptime" 哲学契合 wasm（内存可控 / 编译期生成）。Bun 用 Zig + JavaScriptCore 也部分支持嵌入 Zig wasm。
  4. 编译目标的两条路：wasm32-unknown-unknown（裸 wasm，只能用 imports 自己提供）vs wasm32-wasi（导入 WASI 函数，能用 fs / env / clock）。前者用于浏览器 / 嵌入；后者用于服务端。
  5. 代码体积优化是组合拳—— Rust：panic = "abort"（省 unwinding）+ lto = "fat"（跨 crate inline）+ codegen-units = 1（让 LTO 彻底）+ opt-level = "z"（极致体积）+ strip = true + 跑 wasm-opt -Oz。组合后 hello world 从 ~150KB 压到 ~10KB。
  6. C/C++ 内存安全在 wasm 里被部分修复—— C/C++ buffer overflow 写不出 wasm sandbox 外（受 linear memory bounds check 限制）；这是 wasm 给老旧 C/C++ 代码的"额外保险"。

02 底层逻辑

Rust → wasm：黄金组合的根因

2026 年 Rust 是 wasm 事实标准（npm 上 wasm-bindgen 包占大多数）。背后是5 个设计契合：

(1) LLVM 后端—— rustc 用 LLVM；LLVM 已有 wasm32 backend；rustc --target wasm32-unknown-unknown 直接产出 wasm，无需中间 IR / 自定义编译器

(2) 无 GC—— Rust 用所有权 + 借用编译期检查内存安全；不需要运行时 GC。这意味着 Rust wasm 不带 GC runtime，体积小 / 启动快 / 性能可预测

(3) wasm-bindgen 自动 marshalling—— 宏 #[wasm_bindgen] 标注函数 / struct，自动生成 JS glue（详见 P3.2）。开发者不必手写跨边界 marshalling

(4) Cargo 生态—— Cargo / crates.io 让"添加依赖"一行；wasm-pack 让"build + publish to npm"一行。比 Emscripten 的"配置半天"友好

(5) Mozilla 同时推—— Mozilla 同时推 Rust（Servo / Firefox CSS engine）+ wasm（Bytecode Alliance）；社区融合自然

2026 Rust wasm 工具链：

  • rustc—— Rust 编译器；产 wasm 二进制
  • cargo—— 项目管理 + 依赖
  • wasm-bindgen—— 跨边界 marshalling 宏 + glue 生成器（70%+ 项目用）
  • wasm-pack—— 一站式 build + 测 + 发 npm 包工具
  • web-sys / js-sys—— Rust binding for browser API（document / fetch / WebGL 等）
  • wasm-opt—— Binaryen 后处理优化器
  • cargo-component—— Component Model 工具（详见 P7）

Rust 编译目标对比

Rust 有 4 个 wasm target：

Target 用途 imports 典型场景
wasm32-unknown-unknown 裸 wasm 需自己提供 浏览器（配 wasm-bindgen 注入 JS imports）
wasm32-wasi (preview1) WASI Preview 1 WASI imports（fs / clock / env） 服务端 wasmtime / wasmer 早期
wasm32-wasip2 WASI Preview 2 + Component WIT 接口 2024+ 服务端 wasm + Cloudflare Workers
wasm32-unknown-emscripten Emscripten 兼容（罕用） Emscripten POSIX 模拟 历史 / 兼容旧项目

wasm-bindgen 工作原理

当你写 #[wasm_bindgen] 标注函数：

#[wasm_bindgen]
pub fn greet(name: &str) -> String {
    format!("Hello, {}!", name)
}

实际编译产物有4 个东西：

(1) wasm 二进制—— 包含 greet 实际实现，但参数 / 返回值类型变成 wasm 原生：(func $greet (param i32 i32 i32) (result i32))

i32 i32 = string ptr + len（指向 wasm linear memory）；返回 i32 = "ABI 区"指针，包含返回 string 的 ptr + len

(2) __wbindgen_* imports—— wasm 调 JS 端 helper：__wbindgen_string_new（new JS String from ptr/len）/ __wbindgen_alloc / 等

(3) JS glue 文件（my_wasm.js）—— 包装 wasm exports：

// 简化的 wasm-bindgen 生成的 glue
import * as wasm from './my_wasm_bg.wasm';

export function greet(name) {
  // 编码 string 到 wasm memory
  const ptr = passStringToWasm0(name, wasm.__wbindgen_alloc, ...);
  const len = WASM_VECTOR_LEN;

  // 调 wasm
  const ret = wasm.greet(ptr, len);

  // 读返回 string
  const r0 = getInt32Memory0()[ret / 4 + 0];
  const r1 = getInt32Memory0()[ret / 4 + 1];
  const result = getStringFromWasm0(r0, r1);

  // free wasm 端的返回 buffer
  wasm.__wbindgen_free(r0, r1, 1);

  return result;
}

(4) TypeScript 类型声明—— my_wasm.d.ts 让 TS 用户得到类型提示

这套机制让开发者写 Rust 就像普通 Rust，glue 完全透明。

C/C++ via Emscripten：模拟 POSIX 哲学

Emscripten 的核心思想：让现有 C/C++ 代码不改源就能跑在浏览器。要做到这一点，必须把 C 标准库（fopen / printf / malloc / pthread / signal 等）模拟到浏览器。

Emscripten 模拟层（部分）：

  • libc 模拟—— 实现 musl 子集（malloc / free / printf / fopen / mmap / 等）；运行时调用 JS imports
  • filesystem (MEMFS / IDBFS / WORKERFS)—— 内存文件系统 / IndexedDB-backed / Worker 共享；让 fopen / fread 工作
  • pthread 模拟—— 用 Web Workers + SharedArrayBuffer 实现 pthread_create / mutex / 等
  • signal 模拟—— 大部分 signal 转成 noop 或 throw
  • OpenGL 桥接—— OpenGL ES 2 / 3 调用映射到 WebGL
  • SDL / Bullet / 其他库—— 集成 port，预编译版本

典型 Emscripten 命令：

$ emcc hello.c -o hello.html
# 产生 hello.wasm + hello.js（loader） + hello.html（demo）

$ emcc -O3 -s WASM=1 -s EXPORTED_FUNCTIONS='["_my_func"]' \
       -s ALLOW_MEMORY_GROWTH=1 hello.c -o hello.js

Emscripten 产物结构：

  • hello.wasm（C/C++ 编译结果 + libc 实现）
  • hello.js（loader，初始化 runtime + module 加载）
  • hello.html（可选 demo 页）

Emscripten 的代价：

(1) 产物大—— hello world ~100KB+（带 libc）；优化后能压到 ~30KB 但仍比裸 wasm 重

(2) 启动慢—— runtime init（FS 挂载 / 内存分配 / 主函数包装）开销几十 ms

(3) 与 wasm 哲学冲突—— wasm 本意"无环境假设"；Emscripten 偷偷塞 libc，让产物背离 wasm 简单沙箱原则

(4) 工具链复杂—— emsdk / emcc / 各种 -s 参数；学习曲线陡

WASI SDK + clang：现代 C/C++ → wasm

2024+ 趋势：用 WASI SDK（bytecodealliance 维护的 clang + WASI libc）替代 Emscripten。

WASI SDK 的不同：

  • 不模拟 POSIX—— 直接用 WASI imports（fd_read / fd_write / clock_time_get）；更轻、与 WASI 自然集成
  • libc 是 wasi-libc（musl fork）—— 把系统调用映射到 WASI imports
  • 没有 JS glue—— 产物纯 wasm，由 wasmtime / wasmer / Cloudflare Workers / Node 等执行
  • 没有 OpenGL / SDL 桥接—— 服务端场景不需要

典型命令：

# 用 WASI SDK 的 clang
$ /opt/wasi-sdk/bin/clang -O3 hello.c -o hello.wasm

# 跑
$ wasmtime hello.wasm

产物明显更小：

  • Emscripten hello world ~50KB（优化后）
  • WASI SDK hello world ~10KB

什么时候用哪个：

  • 浏览器 + 现有 C++ 代码（带 OpenGL / SDL / 复杂 IO）→ Emscripten
  • 服务端 / Edge / WASI runtime → WASI SDK
  • 新项目纯 C/C++ + 不依赖 POSIX → WASI SDK 或纯 clang wasm32

Zig → wasm：原生支持

Zig 编译器（自家实现，不依赖 LLVM 在某些场景）原生支持 wasm32 / wasm64 target。Zig 哲学契合 wasm：

(1) 显式分配—— Zig 没有 hidden allocation；malloc 必须显式调 allocator.alloc。这与 wasm 内存模型（手工管理 linear memory）契合

(2) comptime—— 编译期执行代码 / 生成代码；让"零成本抽象"在 wasm 也能用（比 Rust trait 更激进）

(3) C 互操作—— Zig 能直接 import C 头文件，编译 C 代码；适合"渐进迁移"现有 C 项目

Zig wasm 命令：

# wasi 目标
$ zig build-exe hello.zig -target wasm32-wasi

# freestanding（裸 wasm）
$ zig build-lib hello.zig -target wasm32-freestanding -dynamic --export=add

Zig 用 wasm 的典型项目：

  • Bun（Bun 自己用 Zig，但也接受用户 wasm 模块）
  • TigerBeetle（金融数据库，用 Zig 写）
  • Roc 语言（Roc compiler 用 Zig 写，支持产 wasm）

Zig wasm 生态相对小（vs Rust），但未来潜力大—— Zig 编译器自身轻量、产物小、与 wasm 哲学契合。

代码体积优化：工程化第一关

Rust + wasm-pack 默认 hello world ~150KB。不优化的 wasm 在生产几乎不可接受：

  • 4G 网络下载 150KB ~1.5s
  • 解压 + parse + 编译 ~50-100ms
  • 启动延迟严重影响 LCP（详见 浏览器渲染 P4.1）

组合拳优化：

步骤 1：Cargo profile 优化

[profile.release]
opt-level = "z"          # 极致体积优化（z > s > 3 > 2 > 1 > 0）
lto = "fat"              # 全 crate Link Time Optimization
codegen-units = 1        # 单线程编译让 LTO 彻底
panic = "abort"          # 省 panic unwinding 代码（~30KB）
strip = "symbols"        # 去 debug symbols
overflow-checks = false  # release 通常不需要

这一步：~150KB → ~80KB

步骤 2：wasm-opt 后处理

$ wasm-opt -Oz pkg/my_wasm_bg.wasm -o pkg/my_wasm_bg.wasm

Binaryen wasm-opt 做wasm 特定优化（DCE / inline / 寄存器复用 / 指令选择），LLVM 不擅长。

这一步：~80KB → ~30KB

步骤 3：移除不用的标准库

Rust 默认依赖 std；某些只需要 core / alloc：

#![no_std]                              // 不要 std
extern crate alloc;                     // 但要 alloc

use alloc::vec::Vec;
use alloc::string::String;

#[wasm_bindgen]
pub fn process(input: Vec<u8>) -> Vec<u8> {
    input.into_iter().map(|x| x.wrapping_add(1)).collect()
}

这一步：~30KB → ~15KB

步骤 4：Brotli 传输压缩

设置 HTTP 服务 Brotli 压缩 .wasm 文件。wasm 二进制 Brotli 压缩比通常 70-80%：

15KB wasm × 0.25 = ~4KB 传输

所以最终用户感受 ~4KB 下载。

Rust panic 与 wasm 错误处理

Rust panic 默认在 wasm 里静默触发 trap（unreachable instruction）：

#[wasm_bindgen]
pub fn divide(x: i32, y: i32) -> i32 {
    x / y  // y=0 时 panic
}

JS 端调 divide(1, 0) 报：

RuntimeError: unreachable executed

看不到 panic message。开发时设 panic_hook：

#[wasm_bindgen(start)]
pub fn main() {
    console_error_panic_hook::set_once();
}

之后 panic 会 console.error + throw JS Error，能看到 message + stack。

release 时设 panic = "abort"，panic 直接 trap（不走 unwinding，省体积）。

C/C++ 内存安全：wasm 帮的忙

C/C++ 经典痛：buffer overflow / use-after-free / null deref。在 wasm 里部分被修复：

(1) Buffer overflow 不能逃出 sandbox—— C 的 strcpy(buf, attacker_input) 写超了 buf；写入仍在 wasm linear memory 内（不能写到 V8 堆 / OS 内存）；最坏情况是写坏 wasm 自己的内存，引起 wasm crash 但不会感染宿主

(2) 仍能 corrupt wasm 自己内部状态—— 如果攻击者能精确控制溢出位置，可能改 wasm 内的函数指针 → 间接调用恶意函数。但call_indirect 检查类型签名，不能跳到任意位置；攻击难度大幅上升

(3) 不能 use-after-free 出 wasm—— free 后的内存仍是 wasm 管理；可能 corrupt wasm 状态但不污染宿主

这是 wasm 给老旧 C/C++ 代码（比如 SQLite / FFmpeg / Photoshop 内核）的"免费保险"——把不可信 C/C++ 装到 wasm 里跑，比直接跑 native 更安全。

03 应用场景

场景 A · 用 Rust 写 SPA 框架（Yew / Leptos / Dioxus）

Rust SPA 框架（类似 React）通过 wasm 跑在浏览器：

  • Yew—— 老牌 Rust SPA，VDOM + JSX-like 宏
  • Leptos—— signals 反应式（类似 Solid）；性能优于 Yew
  • Dioxus—— React 风格，跨 desktop / mobile / web

典型项目：

use leptos::*;

#[component]
fn Counter() -> impl IntoView {
    let (count, set_count) = create_signal(0);
    view! {
        <button on:click=move |_| set_count.update(|n| *n += 1)>
            "Clicked: " {count}
        </button>
    }
}

fn main() {
    leptos::mount_to_body(Counter);
}

性能 vs JS 框架：Rust SPA 框架的 wasm 体积通常 ~150-300KB（vs React+ReactDOM ~150KB），但启动后响应快（无 GC 暂停）。适合性能敏感场景；不适合"小项目+追求最小依赖"。

场景 B · 把 SQLite 编译到 wasm（C → wasm 经典案例）

SQLite 有官方 wasm 版本（sqlite.org/wasm），用 Emscripten 编译：

$ emcc -O3 -s WASM=1 \
       -s EXPORTED_FUNCTIONS='["_sqlite3_open", "_sqlite3_exec", ...]' \
       -s EXPORTED_RUNTIME_METHODS='["ccall", "cwrap"]' \
       -s ALLOW_MEMORY_GROWTH=1 \
       sqlite3.c -o sqlite3.js

JS 端：

import sqlite3InitModule from '@sqlite.org/sqlite-wasm';

const sqlite3 = await sqlite3InitModule({
  print: console.log,
  printErr: console.error
});

const db = new sqlite3.oo1.OpfsDb('mydb.db');
db.exec('CREATE TABLE users (id INTEGER, name TEXT);');
db.exec('INSERT INTO users VALUES (1, "Alice");');
const rows = db.exec({ sql: 'SELECT * FROM users', returnValue: 'resultRows' });

SQLite WASM 包含：~1MB wasm + JS glue。OPFS（Origin Private File System）做持久化。

场景 C · FFmpeg.wasm 视频转码

Emscripten 编译 FFmpeg：

import { FFmpeg } from '@ffmpeg/ffmpeg';
import { fetchFile, toBlobURL } from '@ffmpeg/util';

const ffmpeg = new FFmpeg();
await ffmpeg.load({
  coreURL: await toBlobURL('/ffmpeg-core.js', 'text/javascript'),
  wasmURL: await toBlobURL('/ffmpeg-core.wasm', 'application/wasm'),
});

await ffmpeg.writeFile('input.mp4', await fetchFile(file));
await ffmpeg.exec(['-i', 'input.mp4', '-c:v', 'libx264', 'output.mp4']);
const data = await ffmpeg.readFile('output.mp4');

FFmpeg 完整 wasm ~30MB（多线程版需要 cross-origin isolated）。这种场景SIMD + threads是性能关键（详见 P5.1）。

场景 D · Cloudflare Workers 用 Rust + WASI

# Cargo.toml
[package]
name = "my-worker"
version = "0.1.0"

[lib]
crate-type = ["cdylib"]

[dependencies]
worker = "0.1"
serde_json = "1"
use worker::*;

#[event(fetch)]
async fn main(req: Request, env: Env, ctx: Context) -> Result<Response> {
    Response::ok("Hello from Rust + WASM at the edge!")
}

构建：

$ wrangler deploy

Cloudflare 把 wasm 在 V8 isolate 里跑（不是 wasmtime），冷启动 < 5ms。

场景 E · Zig 写小工具 wasm

// hello.zig
const std = @import("std");

export fn add(a: i32, b: i32) i32 {
    return a + b;
}

export fn fib(n: i32) i32 {
    if (n < 2) return n;
    return fib(n - 1) + fib(n - 2);
}

编译：

$ zig build-lib hello.zig \
       -target wasm32-freestanding \
       -dynamic --export=add --export=fib \
       -O ReleaseSmall

产物 hello.wasm 只有 ~200 字节（无 std）。比 Rust hello world 小一个数量级（因为 Zig 没有"默认带 std"）。

04 典型代码：完整 Rust 项目

项目结构

my-wasm/
├── Cargo.toml
├── src/
│   └── lib.rs
└── pkg/                  # wasm-pack build 产出
    ├── my_wasm_bg.wasm
    ├── my_wasm.js
    ├── my_wasm.d.ts
    └── package.json

Cargo.toml

[package]
name = "my-wasm"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib", "rlib"]

[dependencies]
wasm-bindgen = "0.2"
js-sys = "0.3"
web-sys = { version = "0.3", features = [
    "console", "Document", "Window", "Element", "HtmlElement"
]}
console_error_panic_hook = { version = "0.1", optional = true }

[features]
default = ["console_error_panic_hook"]

[profile.release]
opt-level = "z"
lto = "fat"
codegen-units = 1
panic = "abort"
strip = true

src/lib.rs

use wasm_bindgen::prelude::*;

#[wasm_bindgen(start)]
pub fn init() {
    #[cfg(feature = "console_error_panic_hook")]
    console_error_panic_hook::set_once();
}

// 简单数值函数
#[wasm_bindgen]
pub fn add(a: i32, b: i32) -> i32 { a + b }

// 字符串处理
#[wasm_bindgen]
pub fn greet(name: &str) -> String {
    format!("Hello, {}!", name)
}

// 数组处理（zero-copy view，更快）
#[wasm_bindgen]
pub fn sum_array(data: &[f64]) -> f64 {
    data.iter().sum()
}

// 带状态的 struct
#[wasm_bindgen]
pub struct Calculator {
    history: Vec<f64>,
}

#[wasm_bindgen]
impl Calculator {
    #[wasm_bindgen(constructor)]
    pub fn new() -> Self { Self { history: Vec::new() } }

    pub fn compute(&mut self, expr: &str) -> Result<f64, JsValue> {
        // 简化：只支持 a + b
        let parts: Vec<&str> = expr.split('+').collect();
        if parts.len() != 2 {
            return Err(JsValue::from_str("invalid expression"));
        }
        let a: f64 = parts[0].trim().parse()
            .map_err(|e: std::num::ParseFloatError| JsValue::from_str(&e.to_string()))?;
        let b: f64 = parts[1].trim().parse()
            .map_err(|e: std::num::ParseFloatError| JsValue::from_str(&e.to_string()))?;
        let result = a + b;
        self.history.push(result);
        Ok(result)
    }

    pub fn history_len(&self) -> usize { self.history.len() }
    pub fn last(&self) -> Option<f64> { self.history.last().copied() }
}

// 调 JS console
#[wasm_bindgen]
extern "C" {
    #[wasm_bindgen(js_namespace = console)]
    fn log(s: &str);
}

#[wasm_bindgen]
pub fn debug(message: &str) {
    log(&format!("[Rust] {}", message));
}

// 异步函数（自动生成 JS Promise）
#[wasm_bindgen]
pub async fn fetch_data(url: &str) -> Result<JsValue, JsValue> {
    let window = web_sys::window().unwrap();
    let resp_promise = window.fetch_with_str(url);
    let resp_value = wasm_bindgen_futures::JsFuture::from(resp_promise).await?;
    let resp: web_sys::Response = resp_value.dyn_into()?;
    let text_promise = resp.text()?;
    let text_value = wasm_bindgen_futures::JsFuture::from(text_promise).await?;
    Ok(text_value)
}

05 常见陷阱

陷阱 1：用 Cargo dev profile 上线

dev profile 不优化体积（hello world ~500KB），生产必须 release。

$ wasm-pack build --release   # 不是 --dev

陷阱 2：忘记跑 wasm-opt

wasm-opt 是 Binaryen 提供的wasm 特定后处理。LLVM 不会做 wasm 特定优化（如 register reuse / instruction selection for stack machine）。必须跑 wasm-opt -Oz。

陷阱 3：Rust panic 在 release 看不到 message

release + panic = "abort"，panic 直接 trap，看不到 message（错误是 "unreachable executed"）。

调试方案：

(1) dev build 加 panic_hook—— console.error 显示完整 panic info

(2) production 用 Result + ? 显式处理错误—— 把可能 panic 的代码（unwrap / expect）改成 Result

陷阱 4：Emscripten + 多线程忘记 cross-origin isolated

Emscripten 多线程（pthread）依赖 SharedArrayBuffer，必须 COOP/COEP（详见 浏览器渲染 P2.2）。

检查：JS 端 self.crossOriginIsolated === true。否则 SAB 不可用，Emscripten 回退单线程。

陷阱 5：用 Emscripten 编译 server-side wasm

Emscripten 假设浏览器环境（带 JS glue）。服务端不要用 Emscripten，用 WASI SDK：

# ❌ 服务端不要这样
$ emcc hello.c -o hello.js  # 产 JS glue，wasmtime 不能直跑

# ✅ 服务端用 WASI SDK
$ /opt/wasi-sdk/bin/clang -O3 hello.c -o hello.wasm

陷阱 6：Rust struct 跨 wasm-bindgen 边界忘记 free

function process() {
  const calc = new Calculator();
  calc.compute("1 + 2");
  return calc.last();
  // ❌ calc 没 free，Rust struct 一直在 wasm memory
}

// ✅ 显式 free
function process() {
  const calc = new Calculator();
  try {
    calc.compute("1 + 2");
    return calc.last();
  } finally {
    calc.free();
  }
}

新版 wasm-bindgen 用 FinalizationRegistry 自动 free，但延迟不可控；性能敏感场景手动 free。

陷阱 7：Zig wasm 没有标准库 panic

Zig wasm32-freestanding 默认没 panic handler；越界访问 / 整数溢出会导致 wasm trap。需要自己定义：

pub fn panic(msg: []const u8, _: ?*std.builtin.StackTrace, _: ?usize) noreturn {
    @setCold(true);
    // 调 JS imports 报告 panic
    js_panic(msg.ptr, msg.len);
    @trap();
}
