托管语言 + 其他：Go / AssemblyScript / Java / Kotlin / Dart / .NET / Python

带 runtime / GC 的语言编译到 wasm 的核心难题是 GC：要么把 GC runtime 一起编译进去（产物大、慢），要么用宿主 GC（依赖 WasmGC 提案）。2023 WasmGC 落地（V8 + SpiderMonkey 同时启用）让 Java/Kotlin/Dart/.NET/OCaml 等托管语言一等公民——这是 wasm 历史"用户群扩展最大"的单个特性。Go 是例外（runtime 强耦合自己的 GC，不接受 WasmGC，仍走 GOOS=js 或 TinyGo 两条路）。Pyodide 不是 Python 编译，是 CPython 解释器编译 wasm + numpy/scipy/pandas 预编译。

阅读时长 ~30 分钟 前置 P4.1（系统语言） + P5.2（WasmGC，简短了解即可） 主要来源 golang.org/wasm · tinygo.org · kotlinlang.org/wasm · pyodide.org · v8.dev/blog/wasm-gc-porting

01 速览

  1. WasmGC 是托管语言转折点—— 2023 之前 Java/Kotlin/Dart 编译 wasm 都是工程奇迹（带 JVM/runtime 太重）；2023+ 这些语言变成"一等公民"。Kotlin/Wasm（JetBrains 官方）/ Dart Flutter Web（替代 dart2js）/ .NET Blazor（NativeAOT + WasmGC 替代 mono）。
  2. Go 不接受 WasmGC，走两条路—— 官方 GOOS=js GOARCH=wasm 带完整 GC runtime（~2MB hello world，启动慢）；TinyGo 子集编译器（~50KB hello world，缺一些标准库）。Go runtime 强耦合自己的 GC（写 barrier / GMP 调度），WasmGC 不能直接接管。
  3. AssemblyScript 是 TypeScript 子集—— 语法亲切（前端工程师无障碍），但有自家 GC（产物 50-200KB），性能不及 Rust。2026 用户群在缩小，Rust + wasm-bindgen 在浏览器端已成熟到吸纳大部分场景；AssemblyScript 仍在嵌入式 / 智能合约（NEAR）有用户。
  4. Kotlin/Wasm 是 WasmGC 的旗手—— JetBrains 官方 target；让 Compose Multiplatform 跑在 Web；Kotlin 1.9（2023）alpha，1.10+（2024）稳定。Compose Web Wasm 启动 ~500ms（vs 老 Kotlin/JS 的 1-2s）。
  5. Pyodide 不是"Python → wasm"—— 是 CPython 解释器（C 代码）编译成 wasm，然后跑 Python 字节码。性能比原生 CPython 慢 2-5x；适合数值计算 + 教育（Jupyter / Streamlit），不适合 hot path。numpy/scipy/pandas 是预编译的二进制 wheels。
  6. 选语言的判据—— 浏览器纯计算 → Rust（70%+ 项目）；既有 Java/Kotlin/Dart 大代码库 → 它们的 WasmGC target；学习成本最低 → AssemblyScript；Python 数据科学 → Pyodide；Go 项目（一定要 Go）→ TinyGo（小）或官方（兼容性高）。

02 底层逻辑

"带 GC 语言"编译 wasm 的根本难题

系统语言（Rust/C/C++/Zig）没有 runtime——内存管理在编译期 / 显式调用。它们编译 wasm 简单：直接产数值类型 + linear memory 操作。

托管语言（Java/Kotlin/Dart/Go/Python/.NET）有 runtime——GC + 调度器 + 类元数据 + 字符串池等。要在 wasm 里跑，必须解决：

问题 1：GC 怎么实现—— wasm linear memory 没有"对象"概念，怎么 mark / sweep？

问题 2：runtime 怎么部署—— Go runtime ~1.5MB，Java JVM ~50MB+；编译进 wasm 让产物巨大

问题 3：调度器—— Go goroutine / Java Thread / Dart isolate；wasm MVP 是单线程

历史上有3 种应对策略：

策略 A · 编译完整 runtime 进 wasm—— Go 官方 / Java 早期 TeaVM；产物大、启动慢，但兼容性好

策略 B · 子集编译器—— TinyGo / Subset Java；放弃部分语言特性换体积，但生态分裂

策略 C · WasmGC 提案（2023+）—— wasm spec 直接支持 GC managed 类型（struct/array/i31ref），引擎 GC 接管；产物轻、性能好；主流方案

WasmGC：2023 转折点

WasmGC 提案（2017 起，2023-12 phase 5）引入 GC managed 类型（详见 P5.2）：

(type $Person (struct
  (field $name (ref $String))
  (field $age i32)))

(func $create_person (param $name (ref $String)) (param $age i32) (result (ref $Person))
  (struct.new $Person (local.get $name) (local.get $age)))

关键属性：

  • 引擎实现 GC—— V8 用 Orinoco（详见 V8 P4.2）；SpiderMonkey 用自己的；JSC 用自己的
  • WasmGC 对象不在 linear memory—— 在引擎 GC 堆，独立内存空间
  • 自动管理—— 编译器不用生成 mark/sweep 代码
  • 跨语言 GC 共享—— Java + Kotlin + Dart 三个 wasm module 共享同一引擎 GC

WasmGC 受益最大的语言：Java / Kotlin / Dart / .NET / OCaml / Scala。它们都不再需要自带 GC runtime，wasm 产物从 5-50MB 缩到 200KB-2MB。

Go → wasm：两条独立路线

Go 是例外：不接受 WasmGC（Go runtime 与自家 GC 紧耦合，写 barrier / GMP 调度都要自己实现），仍走两条独立路线。

路线 A · 官方 GOOS=js GOARCH=wasm

Go 1.11+（2018）支持。完整 Go runtime + GC + goroutine 编译进 wasm：

$ GOOS=js GOARCH=wasm go build -o main.wasm main.go
# 产物 main.wasm 通常 1.5-3MB

需要 wasm_exec.js（Go 提供，把 GMP 调度桥到 JS Promise / setTimeout）：

<script src="wasm_exec.js"></script>
<script>
const go = new Go();
WebAssembly.instantiateStreaming(fetch("main.wasm"), go.importObject)
  .then((result) => { go.run(result.instance); });
</script>

优点：

  • Go 标准库完整（fmt / encoding/json / net/http client / etc.）
  • goroutine 工作（用 setTimeout 调度）
  • 兼容性好，Go 已写代码改动极小

缺点：

  • 体积大（~2MB hello world）
  • 启动慢（runtime init ~200ms+）
  • 性能：Go runtime 在 wasm 里比 native 慢 2-5x（GMP 调度通过 setTimeout 模拟）

路线 B · TinyGo

TinyGo 是 Go 的子集编译器，2018 起社区项目，2026 仍是 Go wasm 的"轻量化方案"：

$ tinygo build -o main.wasm -target wasm main.go
# 产物 main.wasm 通常 30-100KB

TinyGo 的子集策略：

  • 放弃完整 reflect（部分支持）
  • 放弃 net/http server（client OK）
  • 简化 GC（mark-and-sweep，不带分代）
  • 简化 goroutine（cooperative scheduling）
  • 放弃 cgo（嵌入式平台不可用）

TinyGo 适合：

  • Edge / Cloudflare Workers / Fastly Compute@Edge（体积关键）
  • 嵌入式（Arduino / 树莓派）
  • 智能合约（NEAR 等）
  • 简单 wasm 工具

TinyGo 不适合：

  • 需要完整 Go 标准库的项目
  • 大量用 reflect 的代码（如 GORM / encoding/gob）
  • cgo 依赖项目

典型选择：你的 Go 代码改 5 行能跑 → TinyGo（轻）；改 100 行才能跑 → 官方（兼容）。

AssemblyScript：TypeScript 写 wasm

AssemblyScript（2018-）是 TypeScript 严格子集 → wasm。让前端工程师"不用学新语言"就能写 wasm。

语法看起来就是 TypeScript：

// add.ts
export function add(a: i32, b: i32): i32 {
  return a + b;
}

export function fib(n: i32): i32 {
  if (n < 2) return n;
  return fib(n - 1) + fib(n - 2);
}

编译：

$ npm install -g assemblyscript
$ asc add.ts -o add.wasm -O

关键差异（vs TypeScript）：

  • 类型严格——必须 i32 / i64 / f32 / f64 / 等 wasm 类型；不能 number / any
  • 没有 union / intersection types
  • 没有 destructuring（部分支持）
  • 带自家 GC runtime（mark-and-sweep；产物 ~50-200KB）
  • 有限标准库（不是完整 TS / Node lib）

AssemblyScript 2026 现状：

  • 用户群在缩小—— Rust + wasm-bindgen 已吸纳大部分浏览器 wasm 场景
  • 仍有用户：智能合约（NEAR Protocol 主推）/ 简单算法库 / 教学
  • 性能不及 Rust（GC 开销 + 不及 LLVM 优化）
  • 体积比 Rust 释放后大（带 AS 自家 GC）

什么时候用：(1) 学习 wasm 入门，TypeScript 背景；(2) 智能合约（NEAR 等）；(3) 不想引入 Rust 工具链的小项目。否则2026 优先 Rust。

Java / Kotlin / Dart / .NET：WasmGC 翻身

Kotlin/Wasm（JetBrains 官方）

Kotlin 1.9（2023）开始 alpha，1.10+（2024）稳定。基于 WasmGC：

// main.kt
fun add(a: Int, b: Int): Int = a + b

fun greet(name: String): String = "Hello, $name!"

fun main() {
    println(greet("Kotlin/Wasm"))
}

构建（Gradle）：

// build.gradle.kts
kotlin {
    wasmJs {
        browser()
        binaries.executable()
    }
}

Compose Multiplatform Web（2024+）让 Kotlin UI 跑在 Web（基于 Kotlin/Wasm + WasmGC + Skia/CanvasKit 渲染）。Jetbrains 自家 Fleet IDE 部分用此架构。

Dart / Flutter Web

Flutter Web 2024 起转 wasm 部署：

$ flutter build web --wasm

替代旧的 dart2js（编译到 JS）；启动 / 渲染都改善。基于 WasmGC + Skia。

Java / Scala / Clojure

历史上：

  • TeaVM（早，2014-）—— Java 字节码 → JS 或 wasm；wasm 部分用 WasmGC（2024+）
  • CheerpJ（商业）—— 完整 JVM 跑在浏览器（带 wasm runtime）

2026 trend：直接Java → WasmGC（OpenJDK 实验中，Project Leyden / Loom 关联）。Scala / Clojure 等 JVM 语言通过 Java bytecode → WasmGC 跟进。

.NET / Blazor

Blazor 是 Microsoft 的 .NET → wasm 方案。两条路：

  • Blazor Server—— 服务端渲染 + SignalR；不真正用 wasm
  • Blazor WebAssembly—— 客户端 wasm；早期带 mono runtime（产物 ~5MB），2024+ NativeAOT + WasmGC（产物 ~1MB）

Pyodide：把 CPython 编译到 wasm

Pyodide（Mozilla, 2018-）是 wasm 在数据科学领域的旗舰项目。关键：不是"Python → wasm"，是"CPython 解释器（C 代码）→ wasm，然后 wasm 跑 Python"：

  Python 源码 (.py)
       ↓
  CPython 解释器（写在 C 里）
       ↓ 用 Emscripten 编译
  cpython.wasm（含解释器）
       ↓ 加载 .py 文件
  Python 程序在 wasm 里跑

Pyodide 包含：

  • CPython 解释器（~10MB wasm）
  • 预编译的 numpy / scipy / pandas / matplotlib / 等（每个 ~MB 级 wheels）
  • JS-Python 双向互操作 helper

性能：纯 Python 代码比原生 CPython 慢 ~2-5x（额外 wasm 间接开销）；numpy / scipy 数值代码因为底层 C 已编译 wasm，性能与原生差距小。

典型用户：

  • JupyterLite（浏览器内 Jupyter，不用 server）
  • Streamlit Cloud（部分应用 Pyodide 做客户端 demo）
  • 教育平台（CodeCademy / DataCamp 部分场景）
  • 数据可视化博客（直接在博客内跑 numpy / pandas）

启动慢—— 第一次加载 ~10MB Pyodide 核心 + 各包；缓存后好转。不适合首屏关键路径。

Ruby / PHP / 其他动态语言

Ruby（mruby-wasm / Wasmer Ruby）和 PHP（PHP-WASM, WordPress Playground）都通过类似 Pyodide 的方式（解释器 → wasm）。

WordPress Playground 让 WordPress（PHP + MySQL）整个跑在浏览器，是有趣的工程奇迹：

  • PHP 解释器编译 wasm
  • SQLite 替代 MySQL（也是 wasm）
  • 整个 stack 跑在浏览器

详见 wordpress.github.io/wordpress-playground。

03 应用场景

场景 A · 选语言：跨决策树

问题 1：你已有大代码库吗？

  • 已有 Rust → Rust + wasm-bindgen
  • 已有 C++ → Emscripten 或 WASI SDK
  • 已有 Java/Kotlin/Scala → 等待官方 WasmGC target（2026 仍 alpha-beta）
  • 已有 Dart Flutter → flutter build web --wasm
  • 已有 .NET → Blazor
  • 已有 Python → Pyodide（数据科学）或重写

问题 2：浏览器 vs 服务端？

  • 浏览器 → Rust（最成熟 ecosystem）；AssemblyScript（前端友好）
  • 服务端 → Rust + WASI；Go (TinyGo) + WASI

问题 3：体积 vs 兼容性？

  • 体积优先 → Rust / Zig / TinyGo
  • 兼容性优先 → Go 官方 / Pyodide / Blazor（带 runtime）

场景 B · TinyGo 写 Cloudflare Workers

// main.go
package main

import (
    "github.com/syumai/workers"
)

func main() {
    workers.Serve(workers.HandlerFunc(handle))
}

func handle(w workers.ResponseWriter, r *workers.Request) {
    w.Write([]byte("Hello from TinyGo Worker!"))
}
$ tinygo build -o build/app.wasm -target wasm main.go
$ wrangler deploy

典型 TinyGo Worker 产物 ~150-300KB（含 worker SDK）；冷启动 < 5ms。

场景 C · Kotlin/Wasm + Compose UI

// App.kt
import androidx.compose.foundation.layout.*
import androidx.compose.material3.*
import androidx.compose.runtime.*

@Composable
fun App() {
    var count by remember { mutableStateOf(0) }
    Column(modifier = Modifier.padding(16.dp)) {
        Text("Count: $count")
        Button(onClick = { count++ }) { Text("Increment") }
    }
}

fun main() {
    onWasmReady {
        ComposeViewport(viewportContainerId = "ComposeTarget") {
            App()
        }
    }
}

构建：

$ ./gradlew wasmJsBrowserProductionWebpack

场景 D · Pyodide 跑 numpy 在浏览器

<script src="https://cdn.jsdelivr.net/pyodide/v0.25.0/full/pyodide.js"></script>
<script>
async function main() {
  let pyodide = await loadPyodide();
  await pyodide.loadPackage(['numpy', 'pandas']);

  const result = pyodide.runPython(`
import numpy as np
arr = np.array([1, 2, 3, 4, 5])
arr.mean()
  `);

  console.log(result);  // 3.0
}
main();
</script>

第一次加载 Pyodide 核心 ~10MB（含 CPython）；缓存后好转。numpy ~3MB 也缓存。

场景 E · NEAR Protocol 智能合约（AssemblyScript）

// contract.ts
import { context, storage, logging } from "near-sdk-as";

@nearBindgen
export class Counter {
  count: i32 = 0;

  increment(): void {
    this.count += 1;
    logging.log("Counter incremented to " + this.count.toString());
  }

  get(): i32 {
    return this.count;
  }
}

NEAR 生态历史上推 AssemblyScript（前端工程师友好）+ Rust（性能 + 安全）。两条路并行。

04 典型代码：跨语言对比

同样的"fib(20)"实现：5 种语言

Rust

use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn fib(n: i32) -> i32 {
    if n < 2 { n } else { fib(n - 1) + fib(n - 2) }
}

体积：~300 字节 wasm（无 wasm-bindgen 时）

AssemblyScript

export function fib(n: i32): i32 {
  if (n < 2) return n;
  return fib(n - 1) + fib(n - 2);
}

体积：~200 字节 wasm（无 GC 引用时）

TinyGo

package main

//export fib
func fib(n int32) int32 {
    if n < 2 {
        return n
    }
    return fib(n-1) + fib(n-2)
}

func main() {}

体积：~50KB（带 TinyGo runtime）

Kotlin/Wasm

@JsExport
fun fib(n: Int): Int = if (n < 2) n else fib(n - 1) + fib(n - 2)

体积：~500KB（带 Kotlin/Wasm 标准库 + WasmGC）

Zig

export fn fib(n: i32) i32 {
    if (n < 2) return n;
    return fib(n - 1) + fib(n - 2);
}

体积：~150 字节 wasm（极致）

同样的字符串处理：4 种方式

Rust + wasm-bindgen

#[wasm_bindgen]
pub fn process(input: &str) -> String {
    input.to_uppercase()
}

Go (TinyGo)

// 复杂：Go wasm 字符串处理需手工 marshalling
//export process
func process(strPtr *uint8, strLen int) *uint8 {
    str := unsafe.String(strPtr, strLen)
    upper := strings.ToUpper(str)
    return unsafe.StringData(upper)
}

AssemblyScript

export function process(input: string): string {
  return input.toUpperCase();
}
// AssemblyScript 自家 GC 自动处理 string

Pyodide（不编译，运行时解释）

def process(s: str) -> str:
    return s.upper()
pyodide.runPython(`
def process(s):
    return s.upper()
`);
const upper = pyodide.globals.get('process')("hello");
console.log(upper);  // "HELLO"

05 常见陷阱

陷阱 1：用 Go 官方 GOOS=js 做生产 web app

~2MB hello world + 启动 ~200ms+ 让用户体验差。Web 生产用 TinyGo（容忍子集）或换 Rust。Go 官方 wasm 适合：(1) 内部工具 / dashboard（用户少要求低）；(2) 验证 / demo。

陷阱 2：以为 AssemblyScript "等于 TypeScript"

AssemblyScript是 TypeScript 严格子集，不能用：

  • any、union types、mapped types
  • 大部分 array methods 没实现或行为不同
  • Object.fromEntries / Promise 等高级 API 缺失
  • 动态属性访问 / 反射

把 TS 代码扔进 AS 编译器通常报一堆错。必须按 AS 文档重写。

陷阱 3：Pyodide 用在性能关键路径

纯 Python wasm 慢 ~2-5x；不要把 hot path（每秒上千调用）用 Pyodide 实现。Pyodide 适合：

  • 数据科学交互式探索（Jupyter / Streamlit demo）
  • 教育（学 Python 不用装环境）
  • 偶发性 numpy / pandas 计算

陷阱 4：以为 WasmGC = 垃圾收集 API 暴露给 wasm

WasmGC 不是"通用 GC API"。spec 不暴露 mark/sweep/compact 控制。引擎自由实现。开发者只能用 struct.new / array.new / ref.cast / 等高层操作。详见 P5.2。

陷阱 5：Kotlin/Wasm 误用 Kotlin/JS 文档

Kotlin 有3 个 Web target：Kotlin/JS（编译到 JS）、Kotlin/Wasm（编译到 WasmGC）、Kotlin/Native（编译到 wasm32-unknown，无 GC）。互不兼容。文档要看 wasm 那一份。

陷阱 6：Flutter Web wasm 误以为加载快

Flutter Web wasm 产物含 Skia/CanvasKit + Flutter framework，整体 ~2-5MB。启动 ~500ms-2s。适合后续交互密集 app；不适合"闪开网页就关"场景。

陷阱 7：cgo / JNI 项目不能直接编译 wasm

Go 项目用 cgo（call C）→ TinyGo 不支持，官方 GOOS=js 不支持 cgo。必须先去掉 cgo（用纯 Go 替代）才能编译 wasm。

Java JNI 类似 —— 调 native 库的 Java 项目不能直接 → wasm。
