P2 · TanStack Query 心智

TanStack Query 的心智门槛比 API 数量大得多。这一阶段把核心三件套（queryKey / queryFn / queryClient）和最反直觉的状态机（fresh / stale / inactive / gc 四态 + 5 个 refetch 触发条件）讲透，让"打开页面立刻看到旧数据 + 后台请求"这种行为不再是黑魔法。

章节数 2 主要来源 tanstack.com/query/latest 官方文档 + Tanner Linsley 关于 staleTime 的多篇博文 状态 ✅ 全部完成

本阶段覆盖

  • queryKey 是数组而不是字符串 · 第一段命名空间 + 后面是依赖；invalidate 模糊匹配的根
  • queryFn 签名 + signal 自动 cancel · v5 默认带 AbortSignal；卸载 / 切换 query 自动 abort
  • queryClient 跨组件共享 · 同 queryKey 的 query 在多个组件订阅时共享一份 cache
  • stale-while-revalidate 来自 RFC 5861 · TanStack Query 把它从 HTTP cache 提到 React hook 层
  • 4 态状态机 · fresh / stale / inactive / gc；每态间转换条件清楚
  • staleTime 不是过期 · 是"超过这个值后激活订阅就 refetch"的阈值；fresh 永远不 refetch
  • gcTime（v4 叫 cacheTime） · 默认 5 分钟；inactive 超过 gcTime 才从内存清掉
  • 5 个 refetch 触发条件 · mount + 已 stale / window focus / network reconnect / 显式 refetch / refetchInterval
  • queryKey factory 模式 · 把 queryKey 集中管理避免散落引用陷阱
