queryKey + queryFn + queryClient 三件套

TanStack Query 的 API 表面看就是 useQuery({ queryKey, queryFn }) 两个参数，但每个参数都有反直觉的设计决定 —— queryKey 为什么是数组、queryFn 的 signal 怎么用、queryClient 谁在管。这一章把三件套讲透，让"突然请求多发一次"或"invalidate 没生效"的问题不再黑魔法。

阅读时长 ~25 分钟 前置 P1 历史与哲学 + 写过任意 fetch 库 主要来源 tanstack.com/query/latest/docs/framework/react/guides/query-keys + query-functions + tkdodo.eu/blog

01 速览

  1. queryKey 是数组：第一段命名空间，后面是依赖 —— ['users'] / ['users', userId] / ['users', { filter, page }]。模糊匹配是数组的核心能力。
  2. queryKey 必须可序列化：TanStack Query 内部用稳定 hash（自动按 key 排序 object）当 cache map 索引；不可序列化的引用（class instance / 函数 / Symbol）不能放进 queryKey。
  3. queryFn 接受 context：(context: QueryFunctionContext) => Promise<TData>，context 含 { queryKey, signal, meta, pageParam }。signal 是 v5 自带的 AbortSignal。
  4. signal 自动 cancel：把 signal 透传给 fetch，组件卸载 / 切换 query / window blur 都会自动 abort 老请求。无需手写 AbortController。
  5. queryFn 必须返回 Promise：v5 严格了；返回 undefined 会报错。如果数据可能为空，返回 null。
  6. queryClient 整个 app 一个：通过 <QueryClientProvider client={queryClient}> 注入；每个请求 / Server Component 应该独立 new（详见 P5.2）。
  7. queryClient 提供 imperative API：invalidateQueries / setQueryData / getQueryData / prefetchQuery / fetchQuery / ensureQueryData。这些不依赖 React 渲染，可以从 event handler / WebSocket callback / Service Worker 调。
  8. 跨组件共享 cache：同一个 queryKey 的 query，N 个组件订阅时共享一份 cache + 一次请求。这是 server state 库的标志性能力。
  9. queryKey factory 模式：把 queryKey 集中管理（userKeys.detail(id)），避免散落引用陷阱。是大型项目必备实践。
  10. DevTools 必装：<ReactQueryDevtools /> 显示所有 query / cache 状态 / 发出请求 / refetch 触发。本地开发必开。

02 底层逻辑

queryKey · 为什么是数组

SWR 用字符串 key（useSWR('/api/users/1')），TanStack Query 用数组（useQuery({ queryKey: ['users', 1] })）。这个差异不是风格之争，是能力不同。

看一段典型场景。Page A 显示 user list，Page B 显示 user 1 detail，Page C 显示 user 1 的 posts。用户在 Page C 修改了 user 1 的 name，期望：

  • Page B（user 1 detail）刷新
  • Page A（user list）刷新
  • Page C（user 1 posts）不需要刷新（不依赖 name）

用 SWR 字符串 key：

javascript 
// 必须分别 mutate 每个相关 URL
mutate('/api/users');
mutate('/api/users/1');
// 加 filter / pagination 时变得 fragile

用 TanStack Query 数组 key：

tsx 
// 模糊匹配 ['users'] 命中所有 ['users', ...]
queryClient.invalidateQueries({ queryKey: ['users'] });
// 自动失效：
//   ['users']
//   ['users', 'list', { filter: 'active' }]
//   ['users', 'detail', 1]
//   ['users', 'detail', 2]
// 不失效：
//   ['posts', 'by-user', 1]  ← Page C 的 posts 不受影响

核心机制：TanStack Query 的 invalidate 是前缀匹配。['users'] 是任何 ['users', ...] 的前缀。这是 server state 实战中最常用的 cache 操作 —— "我改了 user，所有跟 user 有关的 cache 都该刷"。

queryKey 内部是怎么变成 cache 索引的

TanStack Query 内部用一个 queryKeyHashFn（默认实现：稳定排序 + JSON.stringify），把 queryKey 转成字符串当 map 索引：

javascript 
// 默认 hash 函数（简化版）
function hashKey(queryKey) {
  return JSON.stringify(queryKey, (_, val) => {
    if (val && typeof val === 'object' && !Array.isArray(val)) {
      // 关键：object 的 key 自动按字典序排序
      return Object.keys(val).sort().reduce((acc, k) => ({ ...acc, [k]: val[k] }), {});
    }
    return val;
  });
}

// 这两个 queryKey 内部 hash 相同：
['users', { page: 1, filter: 'active' }]
['users', { filter: 'active', page: 1 }]
// → 都映射到同一 cache entry

这个细节解决了一个常见坑：用户传 object 时 key 顺序不一样不会被当成不同 query。但反过来，object 引用稳定性仍然重要（详见陷阱章）。

queryKey 必须可序列化

JSON.stringify 能处理的：string / number / boolean / null / array / plain object（含嵌套）。不能处理的：

  • undefined 在数组里会变 null，在 object 里直接消失 → 推荐避免
  • Date 会被 toISOString → 通常 OK 但要意识到
  • 函数 / Symbol → 直接消失 → 不要放
  • class instance（new Date() 之外的）→ 只 stringify 自身可枚举属性，prototype 链丢失
  • 循环引用 → 直接报错

实战经验：queryKey 里只放原始值 + plain object + array。复杂对象先序列化成 ID。

queryFn · signal 自动 cancel

TanStack Query v5 默认给 queryFn 注入 AbortSignal。把它透传给 fetch，组件卸载 / 切换 query 自动 abort：

tsx 
useQuery({
  queryKey: ['user', id],
  queryFn: async ({ signal }) => {
    const r = await fetch(`/api/users/${id}`, { signal });
    if (!r.ok) throw new Error('Failed');
    return r.json();
  },
});

// 当下面任一发生时，signal.aborted 变 true：
// - 组件卸载
// - id 变化（旧 query 被 cancel，新 query 启动）
// - 显式 queryClient.cancelQueries({ queryKey: ['user', id] })

对比手写 useEffect+fetch 的繁琐：

tsx 
// useEffect 版本
useEffect(() => {
  const ctrl = new AbortController();
  fetch(`/api/users/${id}`, { signal: ctrl.signal })
    .then(r => r.json())
    .then(data => setData(data))
    .catch(e => { if (e.name !== 'AbortError') setError(e); });
  return () => ctrl.abort();
}, [id]);

// TanStack Query 版本（一行 signal 解决）
queryFn: ({ signal }) => fetch(`/api/users/${id}`, { signal }).then(r => r.json())

底层：TanStack Query 在每个 query lifecycle 创建 AbortController；query 离开 active 状态时调 controller.abort()，触发 fetch 内部的 reject + cleanup。

queryFn · 错误必须 throw

TanStack Query 区分 success / error 是通过 promise resolved / rejected。fetch 拿到 4xx / 5xx 不会自动 reject（这是 fetch API 的特性，HTTP 错误码仍然是 resolved promise）。所以必须手动 throw：

tsx 
// ❌ 错：404 / 500 不会触发 useQuery 的 error 状态
queryFn: () => fetch('/api/users').then(r => r.json())

// ✅ 正：检查 r.ok 后 throw
queryFn: async () => {
  const r = await fetch('/api/users');
  if (!r.ok) throw new Error(`HTTP ${r.status}`);
  return r.json();
}

// ✅ 用 axios（自动 throw 4xx/5xx）
queryFn: () => axios.get('/api/users').then(r => r.data)

这个坑非常常见。如果 query 一直显示 success 但 data 是错误体，多半是 queryFn 没 throw。

queryClient · 整个 app 一个 + provider 注入

tsx 
// app/Providers.tsx
'use client';
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';
import { useState } from 'react';

export function Providers({ children }: { children: React.ReactNode }) {
  // ⚠️ useState 包装确保每次 render 不重新 new
  const [queryClient] = useState(() => new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 60 * 1000,           // 默认 1 分钟
        retry: 1,                        // 失败重试 1 次
        refetchOnWindowFocus: false,    // 关闭 focus 自动 refetch
      },
    },
  }));
  return (
    <QueryClientProvider client={queryClient}>
      {children}
      <ReactQueryDevtools initialIsOpen={false} />
    </QueryClientProvider>
  );
}

关键点：

  • 用 useState(() => new QueryClient())，不要直接 const client = new QueryClient()。后者每次 render 都创建新 client，会丢 cache
  • defaultOptions 在 client 层设全局默认；每个 query 可以 override
  • 在 Next.js App Router 下需要标 'use client'；Server-side prefetch 用法见 P5.2

queryClient · imperative API 全谱

queryClient 不只是被 useQuery 隐式用。它本身是一等公民，提供 imperative API：

API 用途 典型场景
invalidateQueries({ queryKey }) 标记为 stale，订阅者激活时 refetch mutation 成功后
refetchQueries({ queryKey }) 立即 refetch（无视 stale 状态） 用户点 refresh 按钮
setQueryData(queryKey, updater) 直接改 cache（同步） mutation 成功后乐观写入 / WebSocket 推送
getQueryData(queryKey) 读 cache（同步） 检查数据存在 / 派生计算
prefetchQuery({ queryKey, queryFn }) 预热 cache，不订阅 路由 hover prefetch
fetchQuery({ queryKey, queryFn }) 拿 promise（不进 cache 失败 throw） router loader 集成
ensureQueryData({ queryKey, queryFn }) cache 有就返回，没有就 fetch + cache 组件 mount 前确保数据在
cancelQueries({ queryKey }) 取消 in-flight 请求 乐观更新前避免 race
removeQueries({ queryKey }) 从 cache 移除 logout 清空所有 user 数据

跨组件共享 cache · dedup 机制

5 个组件同时挂载，全部用 queryKey: ['user', 1] —— 只发一次请求。原理：

  1. 每个 useQuery 在 mount 时订阅对应 cache entry
  2. 第一个订阅者发现 cache 不存在或 stale → 触发 queryFn
  3. queryFn 还在 in-flight 时第 2-5 个订阅者接入 → 看到 isPending=true，等待同一 promise
  4. promise resolve → 5 个订阅者同时拿到 data

这是 server state 库的标志性能力。useState + useEffect 写法做不到这点（除非把数据上提到 context，但又有新坑）。

queryKey factory 模式

大型项目里 queryKey 散落各处会出问题：

tsx 
// 反模式：queryKey 散落，typo 难找
function UserList() {
  useQuery({ queryKey: ['users'], queryFn: api.getUsers });
}
function UserDetail({ id }) {
  useQuery({ queryKey: ['user', id], queryFn: () => api.getUser(id) });
  // 注意：上面 'users' 复数，下面 'user' 单数 → invalidate ['users'] 不会失效 detail
}

// 用 factory 模式集中管理
// queries/users.ts
export const userKeys = {
  all: ['users'] as const,
  lists: () => [...userKeys.all, 'list'] as const,
  list: (filters: UserFilters) => [...userKeys.lists(), filters] as const,
  details: () => [...userKeys.all, 'detail'] as const,
  detail: (id: string) => [...userKeys.details(), id] as const,
};

// 使用
useQuery({ queryKey: userKeys.list({ active: true }), queryFn: ... });
useQuery({ queryKey: userKeys.detail(id), queryFn: () => api.getUser(id) });

// invalidate 全部 user 相关
queryClient.invalidateQueries({ queryKey: userKeys.all });

// 只 invalidate user list
queryClient.invalidateQueries({ queryKey: userKeys.lists() });

// 只 invalidate 某个 detail
queryClient.invalidateQueries({ queryKey: userKeys.detail(id) });

好处：

  • queryKey 形状集中（命名空间一致）
  • TS 类型推导（as const 让 invalidate 时类型更精准）
  • 分层 invalidate（all / lists / detail 三档）
  • refactor 改 key 形状只改 factory，调用方不动

这个模式的最佳参考是 TkDodo 的 "Effective React Query Keys" 博客。社区有个开源库 @lukemorales/query-key-factory 包装了这个模式。

DevTools · 必装

TanStack Query DevTools 是调试 server state 的关键工具。本地必开：

  • 看所有活跃 query 的 queryKey + 状态
  • 看 cache entry 的 data + dataUpdatedAt
  • 看 query 是 fresh / stale / inactive / fetching
  • 手动触发 refetch / invalidate / setQueryData
  • 看每次 refetch 的触发原因（mount / focus / interval / manual）

很多疑问（"为什么我看到旧数据"、"为什么 invalidate 没效果"、"为什么发了两次请求"）打开 DevTools 5 秒就有答案。

03 应用场景

场景 1 · 设计一个完整的 todos queryKey 体系

tsx 
// queries/todos.ts
export const todoKeys = {
  all: ['todos'] as const,
  lists: () => [...todoKeys.all, 'list'] as const,
  list: (filter: 'all' | 'active' | 'done') => [...todoKeys.lists(), filter] as const,
  details: () => [...todoKeys.all, 'detail'] as const,
  detail: (id: string) => [...todoKeys.details(), id] as const,
  // 派生数据 query：todo 数量统计
  stats: () => [...todoKeys.all, 'stats'] as const,
};

// 使用
function TodoList({ filter }: { filter: 'all' | 'active' | 'done' }) {
  const { data } = useQuery({
    queryKey: todoKeys.list(filter),
    queryFn: ({ signal }) => fetch(`/api/todos?filter=${filter}`, { signal }).then(r => r.json()),
  });
  // ...
}

function TodoDetail({ id }: { id: string }) {
  const { data } = useQuery({
    queryKey: todoKeys.detail(id),
    queryFn: ({ signal }) => fetch(`/api/todos/${id}`, { signal }).then(r => r.json()),
  });
}

// 添加 todo 后失效 list（不影响 detail）
const { mutate } = useMutation({
  mutationFn: api.addTodo,
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: todoKeys.lists() });
    queryClient.invalidateQueries({ queryKey: todoKeys.stats() });
  },
});

// 编辑 todo 后失效该 detail + lists
const { mutate: edit } = useMutation({
  mutationFn: api.editTodo,
  onSuccess: (_, variables) => {
    queryClient.invalidateQueries({ queryKey: todoKeys.detail(variables.id) });
    queryClient.invalidateQueries({ queryKey: todoKeys.lists() });
  },
});

// logout 时清空所有 todos
function logout() {
  queryClient.removeQueries({ queryKey: todoKeys.all });
}

场景 2 · WebSocket 推送数据进 cache

WebSocket / SSE 推送的数据进 query cache，让所有订阅者同时更新：

tsx 
function useNotificationsSocket() {
  const queryClient = useQueryClient();

  useEffect(() => {
    const ws = new WebSocket('wss://api.example.com/notifications');
    ws.onmessage = (event) => {
      const notification = JSON.parse(event.data);
      // 直接写进 cache，所有用 notificationKeys.list() 的组件自动 re-render
      queryClient.setQueryData(notificationKeys.list(), (old: Notification[] = []) => [
        notification,
        ...old,
      ]);
    };
    return () => ws.close();
  }, [queryClient]);
}

场景 3 · 路由 hover prefetch

用户 hover 链接时预热数据：

tsx 
function UserListItem({ user }: { user: User }) {
  const queryClient = useQueryClient();

  function prefetchDetail() {
    queryClient.prefetchQuery({
      queryKey: userKeys.detail(user.id),
      queryFn: ({ signal }) => api.getUser(user.id, { signal }),
      staleTime: 30 * 1000,  // prefetch 的数据 30 秒内 fresh
    });
  }

  return (
    <Link href={`/users/${user.id}`} onMouseEnter={prefetchDetail} onFocus={prefetchDetail}>
      {user.name}
    </Link>
  );
}

用户点进 detail 页面时数据已经在 cache，瞬间显示。这是 TanStack Query 比 SWR 更好用的场景之一（SWR 没有结构化的 prefetch API）。

04 典型代码

代码 1 · queryClient 完整配置

tsx 
// app/Providers.tsx
'use client';
import { QueryClient, QueryClientProvider, MutationCache, QueryCache } from '@tanstack/react-query';
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';
import { useState } from 'react';
import { toast } from 'sonner';

export function Providers({ children }: { children: React.ReactNode }) {
  const [queryClient] = useState(() => new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 60 * 1000,
        gcTime: 5 * 60 * 1000,
        retry: (failureCount, error: any) => {
          // 4xx 不重试
          if (error?.status >= 400 && error?.status < 500) return false;
          return failureCount < 3;
        },
        refetchOnWindowFocus: false,
        refetchOnReconnect: true,
      },
      mutations: {
        retry: 1,
      },
    },
    queryCache: new QueryCache({
      onError: (error: any, query) => {
        // 全局 error 处理
        if (query.meta?.errorMessage) toast.error(query.meta.errorMessage as string);
        else toast.error(`Query failed: ${error.message}`);
      },
    }),
    mutationCache: new MutationCache({
      onError: (error: any) => toast.error(`Mutation failed: ${error.message}`),
    }),
  }));
  return (
    <QueryClientProvider client={queryClient}>
      {children}
      <ReactQueryDevtools initialIsOpen={false} buttonPosition="bottom-right" />
    </QueryClientProvider>
  );
}

代码 2 · 自定义 hook 包装 query

tsx 
// hooks/useUser.ts
import { useQuery } from '@tanstack/react-query';
import { userKeys } from '@/queries/users';

export function useUser(id: string) {
  return useQuery({
    queryKey: userKeys.detail(id),
    queryFn: ({ signal }) => fetch(`/api/users/${id}`, { signal }).then(r => {
      if (!r.ok) throw new Error(`HTTP ${r.status}`);
      return r.json() as Promise<User>;
    }),
    staleTime: 60 * 1000,
    meta: { errorMessage: 'Failed to load user' },  // 给全局 error handler 用
  });
}

// 调用方
function UserAvatar({ id }: { id: string }) {
  const { data: user, isPending } = useUser(id);
  if (isPending) return <Skeleton />;
  return <img src={user?.avatar} alt={user?.name} />;
}

包装的好处：

  • 调用方不必关心 queryKey 形状
  • 统一处理 staleTime / retry / error
  • 类型推导集中
  • 测试时 mock 更容易（mock api.getUser 即可）

代码 3 · meta 字段携带元信息

tsx 
// query 上挂 meta
useQuery({
  queryKey: userKeys.detail(id),
  queryFn: ...,
  meta: {
    errorMessage: 'Failed to load user',
    requiresAuth: true,
    auditCategory: 'user-data',
  },
});

// 在全局 onError 里读 meta
queryCache: new QueryCache({
  onError: (error, query) => {
    if (query.meta?.requiresAuth && error.status === 401) {
      router.push('/login');
    }
  },
});

代码 4 · setQueryData 类型安全

tsx 
// setQueryData 的 updater 函数会拿到正确类型（如果 factory 用 as const）
queryClient.setQueryData(userKeys.detail(id), (old: User | undefined) => {
  if (!old) return old;
  return { ...old, name: 'New Name' };
});

// 推荐 helper：immer-style
import { produce } from 'immer';

queryClient.setQueryData(userKeys.detail(id), (old: User | undefined) =>
  old && produce(old, draft => { draft.name = 'New Name'; })
);

05 常见陷阱

  • queryKey 用了不稳定引用 —— 最常见死循环根源。useQuery({ queryKey: ['users', { filter: { active: true } }] }) 内部 object 每次 render 新引用 → query 被识别为新的 → 持续 refetch。修复：把 filter 提到 useState / 用 useMemo 稳定
  • queryKey 缺少依赖 —— queryKey: ['user'], queryFn: () => api.getUser(id) id 在闭包里。id 变化时 queryKey 不变 → 仍命中旧 cache。修复：queryKey: ['user', id]
  • queryFn 闭包过期变量 —— queryFn 在 query 第一次创建时被记住，闭包里的 props / state 是当时的值。queryKey 变化才会重新创建 queryFn。把变量放进 queryKey 才能让闭包正确更新
  • queryFn 不 throw 4xx/5xx —— fetch 不自动 reject HTTP 错误码。手动检查 r.ok 后 throw，或用 axios。否则 useQuery 永远 success 但 data 是错误体
  • queryFn 返回 undefined —— v5 严格了，return undefined 报错。如果可能空，返回 null
  • 每次 render new QueryClient —— const client = new QueryClient() 在组件内每次 render 都创建新 client → cache 全丢。用 useState(() => new QueryClient())
  • queryKey 大小写不一致 —— ['users'] 和 ['Users'] 是不同 cache。用 factory 模式集中管理避免
  • queryKey 单复数混用 —— ['user'] 和 ['users'] 是不同 cache。invalidate ['users'] 不会命中 ['user', id]
  • 不装 DevTools —— 调试 server state 没有 DevTools 等于盲调。本地必开
  • queryKey 里放函数 / class instance —— 不可序列化，会被 JSON.stringify 吃掉。queryKey 只放原始值 + plain object
  • queryClient 跨请求复用 —— 在 Server Component 里全局 new 一个 queryClient 会跨请求污染。每请求独立 new。详见 P5.2
  • setQueryData 直接 mutate cache —— TanStack Query 内部用 structural sharing 比对前后引用决定是否触发 re-render。直接 mutate（oldData.name = 'X'）会破坏引用比对。要返回新对象
  • 把 invalidateQueries 当 setQueryData 用 —— invalidate 只标 stale 等下次激活 refetch，不会立即更新。要立即写新值用 setQueryData 或 refetchQueries
