深度学习/ TanStack Query/ P2 · 心智/ 02 · stale-while-revalidate + staleTime

stale-while-revalidate + staleTime / gcTime + fetcher 行为模型

TanStack Query 最反直觉的部分。"为什么打开页面立刻看到旧数据 + 后台又请求一次"、"我没改代码为什么页面切回来又请求一次"、"我把 staleTime 设成 1 小时为什么还在 refetch" —— 这些问题全部由 4 态状态机 + 5 个 refetch 触发条件回答。

阅读时长 ~25 分钟 前置 P2.1 三件套 + 写过至少一个 useQuery 主要来源 tanstack.com/query/latest/docs/framework/react/guides/caching + Tanner 关于 staleTime 的多篇博文 + RFC 5861

01 速览

  1. stale-while-revalidate（来自 HTTP RFC 5861）：cache 在 stale 之后短时间内还能用，同时后台 revalidate。TanStack Query 把它从 HTTP cache 提到 React hook 层。
  2. 4 态状态机：fresh（刚拿到，比 staleTime 短，永远不 refetch）→ stale（超过 staleTime，订阅者激活时触发 refetch）→ inactive（没人订阅，进入 gc 倒计时）→ gc（超过 gcTime，从内存清掉）。
  3. staleTime 默认 0：每次激活都 refetch。这是 TanStack Query 最有争议的默认值，多数项目调成 60 秒以上。
  4. gcTime 默认 5 分钟（v4 之前叫 cacheTime）。inactive 后 5 分钟才从内存清掉。影响"切走又切回"是否还有 cache。
  5. 5 个 refetch 触发条件（默认全开，可单独关）：(1) mount + 已 stale / (2) window focus（切回 tab） / (3) network reconnect / (4) 显式 refetch / invalidate / (5) refetchInterval（轮询）。
  6. staleTime 不是过期，是 refetch 阈值。fresh 永远不 refetch，无论 mount 多少次；stale 也不会自动 refetch，只在订阅者激活时触发。这是最常见的误解。
  7. "打开页面立刻看到旧数据 + 后台请求"是设计不是 bug：组件 mount 时 cache 命中先 render（旧数据），同时检查 stale 触发后台 refetch；新数据回来时 re-render。
  8. 选择 staleTime 不是技术问题是产品问题：数据多新算"够新"。Marketing 文案 1 天，dashboard 5 分钟，库存 30 秒，金融实时数据 0。
  9. gcTime 应该 ≥ staleTime。否则 cache 还没 stale 就被 gc 了，stale-while-revalidate 失去意义。
  10. refetchOnWindowFocus 默认开。后台运行型应用（dashboard / monitoring）应该关掉，避免切回来一波请求风暴。

02 底层逻辑

stale-while-revalidate 的根 · HTTP RFC 5861

2010 年 IETF 发布 RFC 5861，定义了 Cache-Control: stale-while-revalidate=N 头：

  • cache 进入 stale 后，N 秒内仍然可以返回给客户端（不阻塞）
  • 同时后台 revalidate，下次请求时拿到新数据
  • 对终端用户：感觉永远不 loading

这个心智在 CDN（Fastly / Cloudflare）已经用了 10+ 年。SWR（2019）和 React Query（2020）把它从 HTTP cache 抬到 React hook 层，让客户端 cache 也享受同样的体验：

  • 用户进页面 → cache 有就立刻显示（即使 stale）
  • 同时后台 refetch
  • 新数据来了 → 悄悄更新 UI（不闪 loading）

这是过去 5 年客户端数据获取最重要的体验提升。比"loading 1 秒"或"用旧数据"都好。

4 态状态机

TanStack Query 内部每个 query 在 4 态之间转换：

状态 条件 行为 触发下一态
fresh 有数据 + 距离 dataUpdatedAt 不超过 staleTime 组件读到数据立即返回；永远不 refetch 超过 staleTime → stale
stale 有数据 + 超过 staleTime 组件读到数据立即返回；订阅者激活时触发 refetch refetch 完成 → 回到 fresh / 没人订阅 → inactive
inactive 没有组件订阅这个 queryKey cache 仍在内存；不再 refetch 有组件订阅 → fresh/stale / 超过 gcTime → gc
gc inactive 超过 gcTime 从内存清掉 下次订阅 → 重新 fetch（无 cache）

关键观察：

  • fresh 是绝对的 —— 在 fresh 期内即使你切 tab、reconnect、mount 100 次也不 refetch
  • stale 不自动 refetch —— 必须有订阅者激活（mount / focus / reconnect / 显式调用）才触发
  • inactive 不等于 gc —— 没人订阅但 cache 还在内存，gcTime 内可以复用
  • "过期"这个词在 TanStack Query 里没意义 —— stale 不是过期，gc 才是

5 个 refetch 触发条件

当 query 在 stale 状态时，5 个条件中任一发生都会触发 refetch（默认全开）：

条件 1 · mount + 已 stale（refetchOnMount）

组件第一次挂载或重新挂载时，如果 cache 已 stale，触发 refetch。这是最常见触发条件。

条件 2 · window focus（refetchOnWindowFocus）

用户切回 tab 时触发。基于 visibilitychange + focus 事件。

条件 3 · network reconnect（refetchOnReconnect）

网络断开后重新连接触发。基于 online 事件。

条件 4 · 显式 refetch / invalidate

调用 queryClient.refetchQueries / refetch() / invalidateQueries。

条件 5 · refetchInterval（轮询）

设置后每隔 N 毫秒触发一次（无视 stale 状态）。

每个条件可以单独关：

tsx 
useQuery({
  queryKey: ['users'],
  queryFn: api.getUsers,
  refetchOnMount: false,
  refetchOnWindowFocus: false,
  refetchOnReconnect: 'always',  // 即使 fresh 也强制
  refetchInterval: 30 * 1000,
  refetchIntervalInBackground: false,  // tab 在后台时不轮询
});

staleTime · 默认 0 的争议

v5 默认 staleTime: 0。后果：

  • 每次 mount + focus + reconnect 都 refetch
  • 新手开发会觉得"为什么我不停发请求"
  • 但保证数据最新（保守的默认）

Tanner 在多次博文中解释为什么默认是 0："不知道你的数据更新频率，默认最保守"。但实战 95% 项目把 default 调到至少 60 秒：

tsx 
// 在 QueryClient 全局调
new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60 * 1000,  // 1 分钟
    },
  },
});

个别 query 可以 override：

tsx 
// 实时性高的 query 单独设短
useQuery({
  queryKey: ['stock-price', symbol],
  queryFn: ...,
  staleTime: 0,         // 强制每次 refetch
  refetchInterval: 5000, // 5 秒轮询
});

// 几乎不变的 query 设很长
useQuery({
  queryKey: ['site-config'],
  queryFn: ...,
  staleTime: 24 * 60 * 60 * 1000,  // 24 小时
});

选择 staleTime 的产品判断

不是技术问题。三档参考：

数据类型 建议 staleTime 说明
marketing 文案 / 静态配置 1 小时 - 24 小时 几乎不变；浪费请求是错的
用户 profile / 设置 5 分钟 偶尔变；focus 时 refetch 够用
dashboard / 列表 30 秒 - 1 分钟 有人在改；适度 refetch
库存 / 价格 5 - 30 秒 实时性重要
金融实时数据 0 + refetchInterval 每次都要新；强制轮询
个人聊天 / 通知 0 + WebSocket setQueryData 不轮询，靠推送

判断方法："如果用户看到 N 秒前的旧数据，会出问题吗"。N 就是 staleTime。

gcTime · 默认 5 分钟

gcTime（v4 之前叫 cacheTime）控制 inactive query 多久从内存清掉。默认 5 分钟。

影响"切走又切回"体验：

  • 用户在 page A 看完 → 切到 page B（page A 的 query 进 inactive）
  • 3 分钟内切回 page A → cache 还在，立刻显示（stale-while-revalidate）
  • 10 分钟后切回 → cache 已 gc，重新 fetch（loading）

设置建议：

  • 5 分钟（默认）适合大部分场景
  • 长会话型应用（用户在站内浏览 30 分钟+）调到 30 分钟
  • SPA + 用户经常 tab 切换 + 想保持流畅 → 调到 30 分钟到 1 小时
  • 不要小于 staleTime（cache 还没 stale 就被 gc 了，没意义）

"打开页面立刻看到旧数据 + 后台请求"为什么是设计

第一次访问 page A → fetch users → cache。

切走 → users 进 stale，再 inactive。

3 分钟后切回 → 组件 mount → useQuery 的逻辑：

  1. 检查 cache：有 users 数据 → 立刻 render（旧数据）
  2. 检查 staleTime：已 stale → 触发 refetch（后台）
  3. refetch 完成 → re-render（新数据）

用户的体验：页面瞬间出现 + 几百毫秒后悄悄刷新。比"loading 1 秒 + 显示新数据"好得多。这是 stale-while-revalidate 的核心价值。

refetchOnWindowFocus 的两面性

默认开。好处：用户切回 tab 看到最新数据（如果 stale）。坏处：

  • dashboard 类应用（多 tab 来回切）会发请求风暴
  • 移动端切回 app 也算 focus
  • 开 DevTools 也算 blur/focus，调试时频繁 refetch

常见做法：

  • dashboard / monitoring 应用 → 关，靠 refetchInterval 控制
  • 普通 web app → 开（默认）
  • 极重 query（生成报告 / 大列表）→ 单独关

fetcher 行为模型 · placeholderData / keepPreviousData

切换 queryKey 时（比如切页 ['users', page] 的 page 变化），默认行为是：

  • 新 query 进 pending（无数据）
  • 组件看到 isPending = true → 显示 loading
  • 新数据来 → 显示

导致切页闪烁。placeholderData + keepPreviousData 解决：

tsx 
import { keepPreviousData, useQuery } from '@tanstack/react-query';

function UserList({ page }: { page: number }) {
  const { data, isPlaceholderData } = useQuery({
    queryKey: ['users', { page }],
    queryFn: () => fetchUsers({ page }),
    placeholderData: keepPreviousData,  // v5 的写法
  });
  // 切页时立即显示上一页数据 + isPlaceholderData=true → 后台拉新页
  // 新页来了自动替换
  return (
    <div style={{ opacity: isPlaceholderData ? 0.5 : 1 }}>
      {data?.map(u => <li key={u.id}>{u.name}</li>)}
    </div>
  );
}

这是分页 / 过滤器场景的标配。详细见 P3.1。

03 应用场景

场景 1 · 选 staleTime 的具体场景

一个产品 dashboard：

  • 导航栏 user 头像 → user 信息几乎不变 → staleTime: 5 * 60 * 1000
  • 主面板订单列表 → 30 秒一更新 → staleTime: 30 * 1000
  • 实时库存数字 → 5 秒一刷 → staleTime: 0, refetchInterval: 5000
  • 历史报告（已生成） → 永远不变 → staleTime: Infinity

实战不要全部用同一 staleTime。按 query 调。

场景 2 · 后台运行型应用关 refetchOnWindowFocus

一个监控大屏，用户经常切 tab 看其他参考资料：

tsx 
// 关掉 focus refetch，靠 interval 控制
new QueryClient({
  defaultOptions: {
    queries: {
      refetchOnWindowFocus: false,
      staleTime: 30 * 1000,
      refetchInterval: 60 * 1000,  // 1 分钟轮询
      refetchIntervalInBackground: false,  // tab 后台时不轮询省流量
    },
  },
});

场景 3 · 长会话 SPA 调大 gcTime

用户在站内浏览 1 小时，频繁切多个页面：

tsx 
new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60 * 1000,
      gcTime: 60 * 60 * 1000,  // 1 小时
    },
  },
});
// 用户切走 30 分钟再切回来，cache 还在
// 立刻显示旧数据 + 后台 refetch

场景 4 · DevTools 看状态机

本地开发打开 DevTools，能看到每个 query 当前状态：

  • fresh · 绿色，刚 fetch 完
  • fetching · 橙色，正在请求
  • stale · 蓝色，过 staleTime 等订阅
  • inactive · 灰色，没人订阅

切 tab + window focus + 手动 refetch 都能在 DevTools 实时看到状态变化。比看代码理解状态机更直观。

04 典型代码

代码 1 · 状态机的完整渲染矩阵

tsx 
function UserList() {
  const {
    data,
    status,        // 'pending' | 'success' | 'error'
    fetchStatus,   // 'idle' | 'fetching' | 'paused'
    isPending,     // status === 'pending'
    isFetching,    // fetchStatus === 'fetching'
    isStale,       // 是否处于 stale 状态
    isSuccess,
    isError,
    error,
    dataUpdatedAt,
    refetch,
  } = useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
    staleTime: 60 * 1000,
  });

  // 4 种主要状态组合
  if (isPending) return <Loading />;                                  // 第一次 fetch
  if (isError) return <Error onRetry={refetch} />;                    // 失败
  return (
    <div>
      {/* 永远显示 data，无论 fresh / stale / fetching */}
      <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>
      {/* stale + fetching 时显示后台刷新 indicator */}
      {isFetching && <Spinner size="small" />}
      {/* 显示数据多新 */}
      <p>Updated {Math.floor((Date.now() - dataUpdatedAt) / 1000)}s ago</p>
    </div>
  );
}

代码 2 · staleTime 三档分组

tsx 
// queries/staleTimes.ts
export const STALE_TIMES = {
  RARELY_CHANGES: 60 * 60 * 1000,   // 1 小时（marketing / 配置）
  SOMETIMES_CHANGES: 5 * 60 * 1000,  // 5 分钟（user profile / settings）
  OFTEN_CHANGES: 30 * 1000,          // 30 秒（dashboards / lists）
  REAL_TIME: 0,                       // 0（实时数据）
} as const;

// hooks/useUser.ts
export function useUser(id: string) {
  return useQuery({
    queryKey: userKeys.detail(id),
    queryFn: () => api.getUser(id),
    staleTime: STALE_TIMES.SOMETIMES_CHANGES,
  });
}

// hooks/useStockPrice.ts
export function useStockPrice(symbol: string) {
  return useQuery({
    queryKey: ['stock-price', symbol],
    queryFn: () => api.getStockPrice(symbol),
    staleTime: STALE_TIMES.REAL_TIME,
    refetchInterval: 5000,
  });
}

代码 3 · 暂停轮询（query 不可见时）

tsx 
// refetchInterval 可以是函数，返回 false 暂停
function StockChart({ symbol }: { symbol: string }) {
  const { data } = useQuery({
    queryKey: ['stock-price', symbol],
    queryFn: () => api.getStockPrice(symbol),
    refetchInterval: (query) => {
      // 市场关闭时停止轮询
      if (isMarketClosed()) return false;
      // 价格波动大时加快轮询
      if (query.state.data?.volatility > 0.05) return 1000;
      return 5000;
    },
  });
  return <Chart data={data} />;
}

代码 4 · 手动控制 stale

tsx 
// 想强制让 cache 进 stale 但不立刻 refetch
queryClient.invalidateQueries({
  queryKey: ['users'],
  refetchType: 'none',  // 标 stale 但不 refetch；下次激活订阅时才 refetch
});

// 强制立刻 refetch（无视 staleTime）
queryClient.refetchQueries({ queryKey: ['users'] });

// 检查某个 query 当前状态
const state = queryClient.getQueryState(['users']);
console.log(state?.dataUpdatedAt, state?.isInvalidated);

05 常见陷阱

  • "我设了 staleTime 怎么还在 refetch" —— staleTime 只控制是否触发 refetch；如果你显式 invalidate / refetch / refetchInterval，仍会 refetch。看 DevTools 的 trigger 字段
  • staleTime 设很大但数据其实变了 —— 用户看到旧数据。修复：mutation 后 invalidate；或调小 staleTime
  • staleTime 设 0 + 频繁切 tab —— 网络请求爆炸。每个 tab 切回都 refetch。修复：调大 staleTime 或关 refetchOnWindowFocus
  • gcTime < staleTime —— cache 还没 stale 就被 gc 了，stale-while-revalidate 失去意义。永远 gcTime ≥ staleTime
  • 把 staleTime 当过期理解 —— stale 不是过期。stale 期 cache 仍可读，只是激活时会 refetch。"过期"在 TanStack Query 里对应的概念是 gcTime
  • refetchInterval 不停 + 组件卸载 —— 默认 query 没有订阅者会暂停 interval；但如果有其他订阅者保持活跃，会继续
  • refetchOnWindowFocus 在调试时被触发 —— 打开 DevTools 算 blur/focus → 调试时反复 refetch。建议本地默认关 focus refetch
  • 把 refetchInterval 当 staleTime 用 —— refetchInterval 是定时强制 refetch，不管 staleTime；staleTime 是触发条件阈值。语义不同
  • "我没改代码为什么页面切回来又请求一次" —— 默认 refetchOnMount + refetchOnWindowFocus + staleTime=0 全开。改 staleTime 或关 focus refetch
  • 不监控 isFetching 显示后台刷新 indicator —— stale + fetching 时用户看不到正在刷新；可以加个小 spinner 提示
  • refetchInterval 设很短 + queryFn 很慢 —— 上一个还没回来下一个又起。TanStack Query 内部会跳过（同 key 同时只一个 in-flight），但仍要避免
  • 关 refetchOnWindowFocus 后期望 mutation 也不刷 —— mutation 触发的 invalidate 仍会 refetch（条件 4），与 focus 设置无关
