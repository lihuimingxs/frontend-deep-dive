server state 设计哲学 · Tanner 的 manifesto + Vercel 的 SWR 视角

这一章是整个主题的灵魂。读完应该能从本质上回答 —— 我为什么不能用 useState + useEffect 自己写、Tanner Linsley 提的 server state 二分法到底在分什么、Vercel 的 SWR 极简哲学和 TanStack Query 走的是不是同一条路。

阅读时长 ~25 分钟 前置 P1.1 历史回顾 + 写过任意一种 fetch 库 主要来源 Tanner Linsley 关于 server state 的多次公开演讲与博文 + Shu Ding 关于 SWR 的设计文章 + 各项目官方文档

01 速览

  1. Tanner Linsley 在 React Query v1（2020）首次提出：server state 和 client state 是两类东西，不能用同一套心智管。这是整个主题的根。
  2. Client state：UI 状态（modal 是否打开 / 表单输入 / 滚动位置）。同步、即时、组件销毁就没了。用 useState / Zustand / Jotai 管。
  3. Server state：远程数据。具备 5 个独特性质：远程拥有（你不是 source of truth）/ 异步 API（要发请求才能拿）/ 共享（可能多人同时改）/ 可能 stale（拿到的可能不是最新）/ 需要 caching + invalidation（不可能每次从头拿）。
  4. Redux 把所有状态当一类管，导致 server state 写法臃肿（30 行 thunk + reducer 才完成一个简单请求）。这不是 Redux 错，是把错的东西塞进了对的容器。
  5. TanStack Query 4 个核心设计赌注：(1) queryKey 是缓存身份证（数组形式，模糊匹配 invalidate）(2) stale-while-revalidate 是默认（先返回 cache + 后台刷新）(3) 不抢 store 的活（只管 server state，client state 自己用 useState）(4) vendor-neutral（不绑定 GraphQL / React / fetch）。
  6. SWR（Vercel + Shu Ding）走另一条路：极简 API（只有 useSWR + mutate）+ 没有 queryClient + 默认 refetch 更激进。哲学不同：把"好用"当首要目标，能力可以少。
  7. 2026 客户端 fetch 库市场份额（粗略）：TanStack Query 70% / SWR 15% / RTK Query 10% / Apollo + Relay 5%。但 RSC 时代客户端 fetch 库总需求量在下降。
  8. 选 TanStack Query 还是 SWR 不是技术对决，是项目复杂度 + 团队规模对决。详细决策表见 P6.1。
  9. "我推荐 TanStack Query 给大部分项目"是强观点，原因：(1) 功能最完整 (2) 社区最活跃 (3) DevTools 是调试 server state 的利器 (4) 多框架（Vue/Solid/Svelte）让跨技术栈团队共用心智 (5) 与 React 19 / Next.js / RSC 的官方集成最完善。
  10. "不分 server state / client state 的项目，三年内必然踩坑"是更强观点。这不是新技术新潮流，是 React 生态 5 年来沉淀下来的事实。

02 底层逻辑

Tanner 的二分法 · 为什么 server state 必须独立

看一段 Tanner 在 2020 年 React Query v1 README 里写下的原话：

💡 Tanner 原文

「Server state, in its essence, is very different from client state. Server state:
- Is persisted remotely in a location you do not control or own
- Requires asynchronous APIs for fetching and updating
- Implies shared ownership and can be changed by other people without your knowledge
- Can potentially become "out of date" in your applications if you're not careful」
—— React Query v1 docs

这段话定义了 5 个 server state 特性（v1 是 4 条，后续演讲补了第 5 条 caching/invalidation）：

性质 1 · 远程拥有 —— 你不是数据的 source of truth。Database 才是。这个性质让 client 端的 state 永远只能是 cache，不能是 truth。当 truth（server）变了，cache 就 stale。

性质 2 · 异步 API —— 拿数据要发请求；改数据也要发请求。这个 latency 不能消除，只能用 cache + optimistic 掩盖。

性质 3 · 共享 —— 多个用户、多个 tab、多个设备可能同时改一份数据。client 拿到的是某个时刻的快照，下一秒可能就过时。

性质 4 · 可能 stale —— 性质 1 + 性质 3 的合成。client 端的 cache 必须接受"可能 stale"为常态。这逼着库设计 stale-while-revalidate / 后台 refetch / cache invalidation 这些机制。

性质 5 · 需要 caching + invalidation —— 性质 2 让你不能每次都从头拿（latency 难受），所以要 cache。性质 4 让 cache 不能永远（会 stale），所以要 invalidation。这两个加起来 = 一个完整的 server state lifecycle。

Client state 一条都不沾边：

  • Modal 是否打开 → 你拥有，不是远程
  • 表单输入值 → 同步，不需要 API
  • 滚动位置 → 不共享给其他用户
  • Theme 切换 → 不会 stale（用户不切就不变）
  • Hover 状态 → 不需要 cache（短命）

用 useState / Zustand / Jotai 管 client state 是正确的工具用对了地方。问题是过去 5 年大量项目把 server state 也塞进 Redux store，相当于用螺丝刀拧钉子 —— 能拧，但费劲。

4 个核心设计赌注

赌注 1 · queryKey 是缓存身份证

每个 query 通过 queryKey 唯一标识。这个 key 决定了：

  • cache 怎么存（哈希 key 当 map 索引）
  • 多组件订阅怎么共享（同 key = 同 cache entry）
  • invalidate 怎么模糊匹配（前缀匹配数组）
  • refetch 怎么去重（同 key 同时只发一个请求）

关键决定：queryKey 是数组而不是字符串。因为：

  • 第一段是命名空间（['users'] / ['posts']）
  • 后面是依赖（['users', userId, { filter }]）
  • invalidate ['users'] 自动模糊匹配 ['users', ...] 全部
  • 对比 SWR 的字符串 key（/api/users/1），TanStack Query 的数组让 invalidate 语义更精准

详细 queryKey 设计见 P2.1。

赌注 2 · stale-while-revalidate 是默认

TanStack Query 和 SWR 共享这一赌注：用户进页面立刻看到上次缓存的内容，同时后台 refetch 拿最新。这个心智来自 HTTP RFC 5861，是过去 30 年 web 缓存的成熟方案。

对比方案：

  • 无 cache（裸 useEffect+fetch）：每次进页面 loading 1 秒，体验差
  • cache-first（永不 refetch）：永远看到旧数据，stale 严重
  • cache-then-network（每次都过期）：cache 没意义
  • stale-while-revalidate：立刻显示 cache + 后台 refetch + 拿到新数据再悄悄更新

这是当前最优解。代价是用户偶尔看到 1 秒前的旧数据 —— 但通常这无所谓，金融 / 实时数据可以单独把 staleTime 调成 0。

赌注 3 · 不抢 store 的活

TanStack Query 不是 Redux 替代品。它只管 server state；client state 你自己用 useState / Zustand / Jotai。

这个边界很重要。早期不少人把 TanStack Query 当 Redux 替代试图把 modal 状态、表单状态都塞进 query cache —— 不工作。query cache 是异步、远程、共享的；用来管同步 client state 是滥用。

正确分工：

  • Server state（用户数据、列表、详情、配置）→ TanStack Query
  • UI state（modal 开关、tab 选中、sidebar 折叠）→ useState
  • 跨组件 client state（购物车草稿、theme、locale）→ Zustand / Jotai
  • Form 状态 → React Hook Form / Formik / 原生

赌注 4 · vendor-neutral

不绑定 GraphQL（对比 Apollo / Relay），不绑定 React（2022 起支持 Vue / Solid / Svelte / Angular），不绑定 fetch 实现（你传任意 queryFn，axios / fetch / GraphQL client / SDK 都行）。

这个赌注的回报：

  • 跨技术栈团队共用 server state 心智
  • 前后端协议变化（REST → GraphQL → tRPC → RSC fetch）不影响 query 写法
  • 不被 vendor lock-in（对比 Apollo / Relay 的 GraphQL 绑定）

SWR 的简化哲学 · Vercel 视角

SWR 比 React Query 早一年，但路线截然不同。Shu Ding 在多次演讲里强调：

💡 SWR 的设计原则（官方文档）

"Lightweight, focused on the right primitives. Provide the bare minimum that solves data fetching well, leave everything else to user."

具体表现：

  • 极简 API：只有 useSWR(key, fetcher, options) + mutate 两个核心。没有 queryClient 概念
  • 没有 staleTime：默认每次 mount + focus + reconnect 都拉。"激进刷新"哲学
  • mutator 局部：useSWR 返回的 mutate 只作用于当前 key；全局 mutate 走 useSWRConfig().mutate
  • 无 mutation 系统：v2 加了 useSWRMutation 但比 useMutation 弱很多
  • 没有 DevTools：到 2026 仍然没有官方调试工具

SWR 的目标不是覆盖所有场景，是把 80% 简单场景做到极简。这个哲学在 Pages Router 时代的 Next.js 里很合身（页面级 fetch + 偶尔变更）。

但 App Router + RSC 时代两个力量同时挤压 SWR：

  • 简单首屏 fetch → RSC 接管，不需要 SWR
  • 复杂客户端 query（infinite / 多组件订阅 / 乐观更新）→ SWR 体系不足，TanStack Query 更合适

结果是 SWR 的"中间地带"被两头蚕食。Vercel 自己的工程重心也明显从 SWR 转向 RSC + AI SDK，SWR 维护节奏 2025 起放缓。

TanStack Query vs SWR · 不分胜负，分场景

2026 客户端 fetch 库市场份额

根据 npm trends + State of JS 2025 + 各项目 GitHub stars 估算（粗略）：

  • TanStack Query 约 70%（新项目首选，2026 事实标准）
  • SWR 约 15%（Pages Router 历史项目 + 极简偏好用户）
  • RTK Query 约 10%（Redux 全家桶项目）
  • Apollo + Relay 约 5%（GraphQL 重度项目，绝大多数是历史存量）
  • Jotai-Query / Zustand+自管 / 其他 < 1%

关键趋势：

  • RSC 让客户端 fetch 库总需求量下降（Next.js 项目里 30-50% 的 fetch 移到 server-side）
  • 但 TanStack Query 在剩余客户端场景里份额仍在提升（吃 SWR 留下的份额）
  • GraphQL 客户端总体收缩（Apollo / Relay 加起来 5%，5 年前是 25%）

Tanner 的回应 · TanStack Query 在 RSC 时代的位置

2024 年 React 19 + RSC 落地后，"TanStack Query 还有用吗"是社区高频问题。Tanner 在多次 Twitter / 演讲中明确表态：

💡 Tanner 的回应

「TanStack Query is not going away. RSC and Server Actions handle a different layer — they don't manage long-lived client-side cache, optimistic updates, infinite scroll, or WebSocket-pushed data with multi-component subscribers. We're complementary, not competing.」

具体复用场景见 P5.1。简单总结：TanStack Query 仍然是客户端独有 server state 的标准工具，只是不再覆盖首屏 fetch。

03 应用场景

场景 1 · 团队从 Redux 迁 TanStack Query 怎么分阶段

典型 React 项目里 Redux store 的 70% 内容是 server state（user / posts / config / dashboards）。迁移分 3 步：

  1. 第一步：识别 store 里哪些是 server state。判断：是不是来自 API？是不是会因后端变化而 stale？答"是"就是 server state
  2. 第二步：用 TanStack Query 重写这些 slice，逐个替换。同时 Redux store 收窄到只剩真正的 client state（auth token、theme、cart draft）
  3. 第三步：评估剩余 client state 是否还需要 Redux。少量 client state 可以用 Zustand / Jotai 替代；多了或团队熟悉就保留 Redux

常见错误：一次性迁完所有 slice。应该新代码不再加 Redux thunk，旧代码痛了再迁那一处。半年后绝大部分自然替换。

场景 2 · 多组件订阅同一份数据 · 共享 vs 重复请求

5 个组件各自渲染 user 信息（avatar / name / email / role / lastLogin），传统 useEffect+fetch 写法发 5 次请求。TanStack Query 自动去重：

tsx 
// 5 个不同组件，都用同一个 queryKey
function Avatar({ id }) {
  const { data } = useQuery({ queryKey: ['user', id], queryFn: () => fetchUser(id) });
  return <img src={data?.avatar} />;
}
function Name({ id }) {
  const { data } = useQuery({ queryKey: ['user', id], queryFn: () => fetchUser(id) });
  return <span>{data?.name}</span>;
}
// ... 其他 3 个组件相同

// 一个 page 同时挂载 5 个 → 只发 1 次 fetch
// 这是 server state 库的标志性能力

原理：queryKey 一致 → 命中同一 cache entry → 触发 dedup → 只发一次请求 + 5 个组件订阅同一份数据。useState + useEffect 写法做不到这点（除非把数据提到 context，但又引发新坑）。

场景 3 · 小项目要不要装 TanStack Query

"我项目就 3 个页面，是不是杀鸡用牛刀？"

判断标准：

  • 3 个页面但每个都有 fetch → 装。TanStack Query 减少的是每处 fetch 的样板代码，不是项目大小
  • 整个项目只有 1 处 fetch（极少见） → 不装。1 处 useEffect+fetch 可控
  • 用 Next.js + RSC，所有数据来自 RSC fetch → 不装

关键认识：项目会增长。今天 3 页明天 10 页，最初不装将来重构成本陡增。

04 典型代码

代码 1 · 同一份数据用 Redux thunk 和 TanStack Query 写

对比一个简单的"用户列表 + 添加用户"功能：

javascript · Redux + thunk 
// store/users.js
const FETCH_REQUEST = 'users/FETCH_REQUEST';
const FETCH_SUCCESS = 'users/FETCH_SUCCESS';
const FETCH_FAILURE = 'users/FETCH_FAILURE';
const ADD_REQUEST = 'users/ADD_REQUEST';
const ADD_SUCCESS = 'users/ADD_SUCCESS';
const ADD_FAILURE = 'users/ADD_FAILURE';

export function fetchUsers() {
  return async (dispatch) => {
    dispatch({ type: FETCH_REQUEST });
    try {
      const users = await api.getUsers();
      dispatch({ type: FETCH_SUCCESS, payload: users });
    } catch (e) { dispatch({ type: FETCH_FAILURE, payload: e }); }
  };
}

export function addUser(user) {
  return async (dispatch) => {
    dispatch({ type: ADD_REQUEST });
    try {
      const created = await api.addUser(user);
      dispatch({ type: ADD_SUCCESS, payload: created });
      // 想刷列表？再 dispatch fetchUsers()
      dispatch(fetchUsers());
    } catch (e) { dispatch({ type: ADD_FAILURE, payload: e }); }
  };
}

const initialState = { loading: false, data: [], error: null, adding: false };
export default function reducer(state = initialState, action) {
  switch (action.type) {
    case FETCH_REQUEST: return { ...state, loading: true };
    case FETCH_SUCCESS: return { loading: false, data: action.payload, error: null };
    case FETCH_FAILURE: return { ...state, loading: false, error: action.payload };
    case ADD_REQUEST: return { ...state, adding: true };
    case ADD_SUCCESS: return { ...state, adding: false };
    case ADD_FAILURE: return { ...state, adding: false, error: action.payload };
    default: return state;
  }
}

// component.jsx
function UserList() {
  const { loading, data, error, adding } = useSelector(s => s.users);
  const dispatch = useDispatch();
  useEffect(() => { dispatch(fetchUsers()); }, [dispatch]);

  return (
    <>
      {loading ? <Loading /> : data.map(u => <li key={u.id}>{u.name}</li>)}
      <button disabled={adding} onClick={() => dispatch(addUser({ name: 'New' }))}>
        Add
      </button>
    </>
  );
}
// 总行数：~50 行
tsx · TanStack Query v5 
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';

function UserList() {
  const queryClient = useQueryClient();
  const { data, isPending } = useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
  });
  const { mutate: addUser, isPending: isAdding } = useMutation({
    mutationFn: api.addUser,
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ['users'] }),
  });

  if (isPending) return <Loading />;
  return (
    <>
      {data.map(u => <li key={u.id}>{u.name}</li>)}
      <button disabled={isAdding} onClick={() => addUser({ name: 'New' })}>Add</button>
    </>
  );
}
// 总行数：~15 行

对比：

  • 50 行 → 15 行（70% 代码减少）
  • 样板代码（action types / reducers / dispatch）全部消失
  • 错误处理 / loading / 多组件共享 / cache 失效全部内置
  • 类型推导更好（v5 的 TS 类型是顶级）

代码 2 · client state 仍然用 useState 的边界

tsx 
function UserListPage() {
  // ✅ Server state · TanStack Query
  const { data: users } = useQuery({ queryKey: ['users'], queryFn: api.getUsers });

  // ✅ Client state · useState
  const [filter, setFilter] = useState('');
  const [selectedIds, setSelectedIds] = useState<string[]>([]);
  const [isAddModalOpen, setIsAddModalOpen] = useState(false);

  // ❌ 反模式：把 filter 当 server state 写
  // const { data: filter } = useQuery({ queryKey: ['filter'], queryFn: () => ... })
  // 错的原因：filter 是 UI 状态，不是远程数据

  const filtered = users?.filter(u => u.name.includes(filter));
  return (
    <>
      <input value={filter} onChange={e => setFilter(e.target.value)} />
      {filtered?.map(u => <li key={u.id}>{u.name}</li>)}
    </>
  );
}

05 常见陷阱

  • 把 TanStack Query 当 Redux 替代 —— 它只管 server state。client state 仍然用 useState / Zustand / Jotai。把 modal 状态塞进 query cache 是滥用
  • 把 server state 塞进 useState —— 反方向：拿了 API 数据 setState(data) 是反模式。失去 cache / dedup / 后台 refetch 等所有能力
  • "我们用 GraphQL 所以一定要 Apollo" —— 不一定。TanStack Query 可以接 GraphQL（自己写 queryFn 调 graphql-request）；缺 normalized cache。GraphQL 重度团队选 Apollo / Relay；GraphQL 浅尝团队 TanStack Query 也行
  • "我们已经有 Redux 所以一定要 RTK Query" —— 不一定。RTK Query 只在整个项目都用 Redux且不想引入第三方时才有优势。新模块完全可以单独装 TanStack Query 与 Redux 共存
  • SWR 在 App Router 项目里硬塞 —— 2026 起 App Router 默认 RSC + TanStack Query。SWR 可以用但不是首选
  • "server state 二分法只是营销话术" —— 不是。看一遍 P1.1 + P1.2 应该信服。这是 5 年沉淀的认识，不是新潮流
  • 不装 DevTools —— TanStack Query DevTools 是调试 server state 的关键工具，看 cache 状态 / 发出请求 / refetch 触发条件一目了然。本地必装
  • "TanStack Query 太重了我们项目小用不上" —— 13KB gzip。任何现代 React 项目都不在意这个量级。SWR 4KB 的优势在 bundle 极致优化场景才有意义
  • 用 query 代替 form state —— 表单输入用 React Hook Form / 原生；提交后用 useMutation 写。不要把表单临时输入做成 query
  • RSC 项目里继续用 useEffect+fetch —— RSC 项目首选 server-side fetch + 'use cache'；客户端独有场景用 TanStack Query。不要倒退到 useEffect+fetch。详见 P5.1
