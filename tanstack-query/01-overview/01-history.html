客户端数据获取 15 年史 + 同代人横评

从 2006 年 jQuery $.ajax 到 2024 React 19 use API，客户端数据获取这条线 15 年踩过 7 类坑、出现过 8 个有影响力的方案。这一章把演化串起来，让"为什么 TanStack Query 现在这样长"有历史根因。

阅读时长 ~25 分钟 前置 写过 React + 至少用过一种 fetch 库 主要来源 各项目 GitHub 历史 + Tanner Linsley / Shu Ding 公开演讲与博文 + React 团队 RFC

01 速览

  1. 客户端数据获取的核心矛盾 15 年没变 —— 异步、共享、可能 stale 的远程数据不能用同步 client state 心智去管。每个时代都在重新发现这件事。
  2. 2006-2014 jQuery $.ajax + 回调地狱 → ES6 Promise → React componentDidMount 取数 / componentDidUpdate 同步。"组件挂载就发请求"的直觉从这时定型，并一直拖累后面所有方案。
  3. 2015-2018 三条路线分头试：Redux + thunk / saga（把异步当 client state 写）/ Apollo Client（GraphQL only，normalized cache）/ Relay（Meta 内部生态）。各自解决一部分，没有共识。
  4. 2018-2019 Hooks（16.8）+ useEffect + fetch 反模式 —— 表面简洁，实际暗藏竞态、cleanup、loading 状态散落、双重请求 4 个坑。React 文档 2024 才正式承认这是反模式。
  5. 2019 SWR（Vercel + Shu Ding）—— 把 HTTP RFC 5861 的 stale-while-revalidate 提到 React hook 层；极简 API + 全局缓存。开始把客户端数据获取当独立问题对待。
  6. 2020 React Query v1（Tanner Linsley）—— 关键转折：明确 server state ≠ client state，提出独立的 query lifecycle 抽象。这是这条线的根本性认识。
  7. 2021 RTK Query（Redux Toolkit 内置）—— Redux 阵营对 React Query 的回应；endpoint-based + 自动生成 hooks；让全家桶 Redux 用户不必引入第三方 fetch 库。
  8. 2022 React Query → TanStack Query 改名 —— 从"React 专用"扩展到 Vue / Solid / Svelte / Angular 多框架；vendor-neutral 路线明确。
  9. 2023 TanStack Query v5 —— 单一 object signature（useQuery({ queryKey, queryFn })）/ gcTime 取代 cacheTime 命名 / useSuspenseQuery 三件套进一等公民 / pending 取代 loading。
  10. 2024-2026 React 19 use(promise) + Server Components + Server Actions —— 部分 server state 直接 server-side 解决，不再过 client cache。客户端 fetch 库的位置变了，但没消失。详见 P5.1。

02 底层逻辑

2006-2014 · 回调时代 + componentDidMount 直觉的形成

这段时间没有"客户端数据获取库"的概念，只有怎么发 HTTP 请求这个问题。jQuery 的 $.ajax 是事实标准：

javascript 
// 2008 年的真实写法
$.ajax({
  url: '/api/users',
  success: function(data) {
    $('#user-list').html(renderUsers(data));
  },
  error: function() {
    alert('Failed');
  }
});

问题清单：回调嵌套（callback hell）、无超时控制、无 cancel、无 cache、错误处理散落。ES6 Promise（2014-2015 浏览器普及）解决了回调嵌套，但其他问题仍在。

React 出现后（2013），componentDidMount 成为发请求的"自然"位置：

jsx 
// React 0.14 / 16 时代
class UserList extends Component {
  state = { users: [], loading: true, error: null };

  componentDidMount() {
    fetch('/api/users')
      .then(r => r.json())
      .then(users => this.setState({ users, loading: false }))
      .catch(error => this.setState({ error, loading: false }));
  }

  render() {
    if (this.state.loading) return <Loading />;
    if (this.state.error) return <Error />;
    return <ul>{this.state.users.map(...)}</ul>;
  }
}

这段代码定型了一种心智："组件挂载 = 发请求；卸载 = 不管"。它的隐患直到 Hooks 时代才被看清：

  • 同时挂载 5 个 UserList → 5 个一样的请求（没有 cache 概念）
  • 组件卸载请求不 abort（虽然 setState 会报"on unmounted component" warning）
  • props 变了要不要重新拿数据？要写 componentDidUpdate 自己同步
  • 错误重试 / 缓存 / 后台刷新 / 多组件共享等高级需求全部要自己实现

2015-2018 · 三条路线分头试

路线 A · Redux + thunk / saga · 把异步当 client state 写

javascript 
// 典型 redux-thunk 写法
const FETCH_USERS_REQUEST = 'FETCH_USERS_REQUEST';
const FETCH_USERS_SUCCESS = 'FETCH_USERS_SUCCESS';
const FETCH_USERS_FAILURE = 'FETCH_USERS_FAILURE';

function fetchUsers() {
  return async (dispatch) => {
    dispatch({ type: FETCH_USERS_REQUEST });
    try {
      const users = await api.getUsers();
      dispatch({ type: FETCH_USERS_SUCCESS, payload: users });
    } catch (error) {
      dispatch({ type: FETCH_USERS_FAILURE, payload: error });
    }
  };
}

// reducer 里写一遍 loading/error/data 三个 case
// 每个 query 都要写 3 个 action type + 1 个 thunk + 1 个 reducer 分支

问题：样板代码爆炸。10 个 query 就是 30 个 action + 10 个 thunk + 30 个 reducer 分支。redux-saga 用 generator 改善了可测试性，但代码量更大。

更深层问题：Redux 把 server state 当成普通 client state 存 store，忽略了 server state 的5 个独特性质（远程拥有、异步 API、共享、可能 stale、需要 caching/invalidation —— 详见 P1.2）。结果是 Redux store 越来越胖，invalidation / refetch / 后台刷新都得手写。

路线 B · Apollo Client · GraphQL only 的 normalized cache

jsx 
// Apollo 经典写法
import { useQuery, gql } from '@apollo/client';

const GET_USER = gql`
  query GetUser($id: ID!) {
    user(id: $id) { id name email }
  }
`;

function User({ id }) {
  const { loading, error, data } = useQuery(GET_USER, { variables: { id } });
  if (loading) return <Loading />;
  if (error) return <Error />;
  return <div>{data.user.name}</div>;
}

Apollo 把客户端缓存做成 normalized store（每个 entity 用 __typename:id 索引）—— 改一个 user，所有引用它的组件自动更新。这是当时最先进的设计，但代价：必须用 GraphQL，schema-first 工作流绑定，cache 调试复杂（一个 mutation 改了不该改的东西很难追）。

路线 C · Relay · Meta 内部生态

Relay 把 GraphQL + 编译时优化做到极致：fragment colocation、自动生成类型、按需 fetch。但学习曲线陡到非 Meta 公司很少能用起来；2020 年的 Relay Modern → Relay Hooks 重构进一步抬高门槛。

这三条路在 2018 年前没有共识 —— 用 GraphQL 就 Apollo / Relay；不用 GraphQL 就 Redux + thunk + 自己写。

2018-2019 · Hooks 时代的 useEffect+fetch 反模式

React 16.8 引入 Hooks，"组件挂载发请求"心智从 class 平移到函数：

jsx 
// 2019-2024 网上"React 怎么 fetch 数据"教程的标准写法
function UserList() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('/api/users')
      .then(r => r.json())
      .then(setUsers)
      .catch(setError)
      .finally(() => setLoading(false));
  }, []);

  // ...
}

这段代码看起来简洁，4 个坑同时存在：

坑 1 · 竞态（race condition）：如果 props 变化驱动重新 fetch，请求 B 可能比请求 A 早返回（B 是新的，A 是旧的），最终 setState(A) 把新值覆盖：

jsx 
// 用户从 user/1 切到 user/2
function User({ id }) {
  const [user, setUser] = useState(null);
  useEffect(() => {
    fetch(`/api/users/${id}`).then(r => r.json()).then(setUser);
    // 切换 id 时旧 fetch 还在跑，可能晚于新 fetch 返回
    // setUser(旧 user) 会覆盖 setUser(新 user)
  }, [id]);
}

修复需要 cleanup + 取消标志：

jsx 
useEffect(() => {
  let cancelled = false;
  fetch(`/api/users/${id}`).then(r => r.json()).then(data => {
    if (!cancelled) setUser(data);
  });
  return () => { cancelled = true; };
}, [id]);
// 或用 AbortController

但 99% 的网上教程没有这段。竞态在 dev 环境很难发现（网络快），prod 偶尔出现 bug 难追。

坑 2 · React StrictMode 双 effect：React 18 起 StrictMode 在 dev 故意 mount 组件两次以暴露 effect 不幂等问题。fetch + setState 会执行两次。修复方法和坑 1 一样（cleanup + 取消标志），但很多人只是把 StrictMode 关掉。

坑 3 · loading/error/data 三态散落：每个 query 三个 useState；多个 query 复合就是 6、9、12 个 useState。

坑 4 · 多组件共享同一份数据各拉一遍：UserList 和 UserAvatar 都拿 user，没有共享 cache，所以发两次请求。

2024 年 React 团队的官方文档明确写了这段是反模式：

⚠️ React 官方表态（2024）

「If you're using a framework, use its built-in data fetching mechanism. ... If you're not using a framework, consider using a client-side cache library like React Query or SWR.」
—— react.dev/learn/you-might-not-need-an-effect

这段话等于承认：原生 useEffect + fetch 是错误起点；要么用 framework（Next.js RSC fetch / Remix loader），要么用客户端 cache 库。

2019 SWR · stale-while-revalidate 提到 hook 层

SWR（Stale-While-Revalidate）由 Vercel + Shu Ding 在 2019 年开源，名字直接来自 HTTP RFC 5861 的 Cache-Control: stale-while-revalidate 头：cache 在 stale 之后短时间内还能用，同时后台 revalidate。

SWR 把这个心智从 HTTP 缓存抬到 React hook 层：

jsx 
import useSWR from 'swr';

function User({ id }) {
  const { data, error, isLoading } = useSWR(`/api/users/${id}`, fetcher);
  if (isLoading) return <Loading />;
  if (error) return <Error />;
  return <div>{data.name}</div>;
}

SWR 一上来就解决了 useEffect+fetch 的全部 4 个坑：

  • 切换 id 自动 cancel + cleanup（无竞态）
  • StrictMode 双 mount 自动去重
  • loading/error/data 三态自动管
  • 多组件用同 key 自动共享一份请求 + cache

SWR 极简 API（只有 useSWR + mutate）+ 和 Next.js 强绑定，让它在 2019-2022 Pages Router 时代成为默认选择。

2020 React Query v1 · server state ≠ client state 的关键认识

Tanner Linsley 在 2020 年发布 React Query v1，比 SWR 晚一年但走得更远。Tanner 在 最早的 README 和后来的多次演讲里提出一个根本性观察：

💡 Tanner 原话

「Server state is fundamentally different from client state. Server state is persisted remotely, requires asynchronous APIs, implies shared ownership, and can become out-of-date in your applications if you're not careful.」
—— React Query v1 docs

这一句话定位了整个 server state 库的设计：不是又一个 useState 替代，而是一个全新概念。详细 5 个性质 + 4 个设计赌注见 P1.2。

React Query v1 的 API 比 SWR 复杂但更结构化：

jsx 
// React Query v3 时代写法
import { useQuery } from 'react-query';

function User({ id }) {
  const { data, isLoading, isError } = useQuery(
    ['user', id],         // queryKey 是数组
    () => fetcher(`/api/users/${id}`)
  );
  // ...
}

// invalidate 通过 queryClient
queryClient.invalidateQueries(['user']);  // 模糊匹配所有 ['user', ...]

关键差异（vs SWR）：

  • queryKey 是数组而不是字符串 —— 第一段命名空间，后面是依赖；invalidate 时模糊匹配。详见 P2.1。
  • queryClient 是一等公民 —— 提供 imperative API（prefetch / setQueryData / invalidate / refetch），不只是 hook
  • 更多 lifecycle 钩子 —— onSuccess / onError / onSettled / select / placeholderData / refetchInterval
  • useMutation 是独立 hook —— 写操作有专门生命周期 + onMutate 乐观更新支持

2021 RTK Query · Redux 阵营的回应

2021 年 Redux Toolkit 推出 RTK Query，对 React Query 的回应：endpoint-based + 自动生成 hooks，与 Redux store 共存：

javascript 
// RTK Query 把 query/mutation 写成 endpoint
const userApi = createApi({
  reducerPath: 'userApi',
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  endpoints: (builder) => ({
    getUser: builder.query({ query: (id) => `users/${id}` }),
    updateUser: builder.mutation({
      query: ({ id, ...patch }) => ({ url: `users/${id}`, method: 'PATCH', body: patch }),
      invalidatesTags: ['User'],
    }),
  }),
});

// 自动生成 useGetUserQuery, useUpdateUserMutation hooks
import { useGetUserQuery } from './userApi';
function User({ id }) {
  const { data, isLoading } = useGetUserQuery(id);
  // ...
}

RTK Query 的定位很清晰：已经全家桶 Redux 的项目不需要再引入第三方 fetch 库。但新项目从零开始，RTK Query 比 React Query / TanStack Query 多了 Redux store 依赖（即使你不需要 client state），代价不划算。详细横评见 P6.1。

2022 React Query → TanStack Query · vendor-neutral 多框架

2022 年 Tanner 把 React Query 改名成 TanStack Query，扩展到 Vue / Solid / Svelte / Angular。同时把整个组织改名 TanStack（包括 TanStack Table / TanStack Router / TanStack Form）。

意义：vendor-neutral。一个跨框架团队（React + Vue 都有）能用统一 server state 心智；非 React 项目也能享受同一套 staleTime / queryKey 设计。这一步让 TanStack Query 跳出"React 库"的格局。

2023 TanStack Query v5 · API 重整

v5（2023 年 10 月）做了几个 breaking changes，把 v4 多年积累的 API 杂质清掉：

  • 单一 object signature：useQuery({ queryKey, queryFn, ...options })；v3/v4 时代多种重载（位置参数 / 多参数）全砍掉
  • cacheTime → gcTime：原名容易和 staleTime 混淆，gcTime 更直白（"超过这个时间从内存清掉"）
  • loading → pending：状态名与 React 19 Promise 状态对齐
  • useSuspenseQuery 三件套进一等公民：useSuspenseQuery / useSuspenseInfiniteQuery / useSuspenseQueries
  • 移除 onSuccess / onError / onSettled 在 useQuery 上：这三个 callback 在 v5 只保留在 useMutation；query 层用 useEffect 监听 status 改

v5 是写本主题的目标版本。v3/v4 老 API 不在本主题正文展开。

2024-2026 React 19 + RSC · 部分 server state 离开 client

2024 年起 React 19 + Server Components 落地，客户端 fetch 库的位置发生根本性变化：

  • 首屏数据 → Server Component 直接 async function + fetch + Next.js 'use cache'；不再需要 useQuery + 客户端 hydrate
  • 简单 mutation → Server Action + revalidatePath / revalidateTag；不再需要 useMutation + invalidate 闭环
  • use(promise) API → 让 Server Component 创建的 promise 在 Client Component 里被流式消费；和 useSuspenseQuery 部分功能重叠

但 TanStack Query 没有消失：

  • 客户端长生命周期 cache（infinite scroll / 跨页面 cache 复用）
  • WebSocket / SSE 推送数据多组件订阅
  • 第三方客户端 API（CORS / API key 在 client）
  • 乐观更新（roundtrip 太慢）

这是 P5 章的主战场，详见 P5.1 RSC 时代还该用吗。

03 应用场景

场景 1 · 老项目要不要从 useEffect+fetch 迁到 TanStack Query

判断标准：

  • 项目里 useEffect + fetch 调用 > 5 处 → 必迁。每多一处 useEffect+fetch 是一份 4 大坑的复制代码
  • 已经感受到具体痛（竞态出过 bug、loading 状态散落、多组件重复请求）→ 必迁
  • 迁移路径：先包一层 wrapper hook（useUser / useUsers），逐个替换内部实现，不动调用方

场景 2 · Apollo / Relay 老项目要不要切 TanStack Query

判断标准：

  • 仍在 GraphQL → 不切。Apollo / Relay 的 normalized cache 是 GraphQL 专属优势
  • 从 GraphQL 迁回 REST → 切。GraphQL 客户端在 REST 项目里浪费
  • 团队对 GraphQL schema 不再投入（schema 长期不更新）→ 考虑切

场景 3 · Next.js App Router 项目从 SWR 切 TanStack Query

App Router 时代两个力量同时影响：

  • RSC 接管首屏 fetch → 客户端 fetch 库需求量下降；如果项目 80% 数据走 RSC，留 SWR 也够用
  • 剩余客户端 query 复杂度上升（infinite scroll / 多组件订阅 / 乐观更新） → SWR 体系化不足；TanStack Query 更合适

最优策略通常是RSC + TanStack Query 组合。SWR 在 App Router + RSC 时代的位置进一步收窄。详见 P6.1 + P5。

04 典型代码

代码 1 · 同一个需求的 5 种历史写法对比

需求："拿用户列表，渲染，loading / error 状态显示"。看 5 个时代怎么写：

javascript · jQuery 2008 
$.ajax({ url: '/api/users', success: function(users) {
  $('#list').html(users.map(u => '<li>' + u.name + '</li>').join(''));
}, error: function() { $('#list').html('Error'); } });
jsx · Redux+thunk 2017 
// actions.js
const fetchUsers = () => async (dispatch) => {
  dispatch({ type: 'USERS_REQUEST' });
  try {
    const users = await api.getUsers();
    dispatch({ type: 'USERS_SUCCESS', payload: users });
  } catch (e) {
    dispatch({ type: 'USERS_FAILURE', payload: e });
  }
};

// reducer.js
function users(state = { loading: false, data: [], error: null }, action) {
  switch (action.type) {
    case 'USERS_REQUEST': return { ...state, loading: true };
    case 'USERS_SUCCESS': return { loading: false, data: action.payload, error: null };
    case 'USERS_FAILURE': return { loading: false, data: [], error: action.payload };
    default: return state;
  }
}

// component.jsx
function UserList() {
  const { loading, data, error } = useSelector(s => s.users);
  const dispatch = useDispatch();
  useEffect(() => { dispatch(fetchUsers()); }, [dispatch]);
  if (loading) return <Loading />;
  if (error) return <Error />;
  return <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
jsx · useEffect+fetch 2019 
function UserList() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false;
    fetch('/api/users')
      .then(r => r.json())
      .then(data => { if (!cancelled) setUsers(data); })
      .catch(e => { if (!cancelled) setError(e); })
      .finally(() => { if (!cancelled) setLoading(false); });
    return () => { cancelled = true; };
  }, []);

  if (loading) return <Loading />;
  if (error) return <Error />;
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
jsx · SWR 2020 
import useSWR from 'swr';

function UserList() {
  const { data, error, isLoading } = useSWR('/api/users', (u) => fetch(u).then(r => r.json()));
  if (isLoading) return <Loading />;
  if (error) return <Error />;
  return <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
tsx · TanStack Query v5 2026 
import { useQuery } from '@tanstack/react-query';

function UserList() {
  const { data, error, isPending } = useQuery({
    queryKey: ['users'],
    queryFn: () => fetch('/api/users').then(r => r.json()),
  });
  if (isPending) return <Loading />;
  if (error) return <Error />;
  return <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}

关键观察：

  • 从 Redux+thunk 到 SWR 是样板代码量级下降（30 行 → 1 行）
  • 从 useEffect+fetch 到 SWR 是正确性提升（4 大坑全免）
  • 从 SWR 到 TanStack Query 是能力扩展（cache invalidation / mutation / DevTools）

// 反模式版（4 个坑全在）
function User({ id }) {
  const [user, setUser] = useState(null);
  useEffect(() => {
    fetch(`/api/users/${id}`).then(r => r.json()).then(setUser);
    // 坑 1 竞态：切换 id 时旧 fetch 可能晚于新 fetch 返回
    // 坑 2 StrictMode 双 mount：dev 下双重请求
    // 坑 3 三态散落：另要 setLoading / setError
    // 坑 4 多组件不共享：5 个组件渲染 = 5 个一样的请求
  }, [id]);
}

// 正确版（手写）
function User({ id }) {
  const [state, setState] = useState({ loading: true, data: null, error: null });
  useEffect(() => {
    const ctrl = new AbortController();
    setState(s => ({ ...s, loading: true }));
    fetch(`/api/users/${id}`, { signal: ctrl.signal })
      .then(r => r.json())
      .then(data => setState({ loading: false, data, error: null }))
      .catch(error => {
        if (error.name === 'AbortError') return;
        setState({ loading: false, data: null, error });
      });
    return () => ctrl.abort();
  }, [id]);
  // 修了 1 + 2，没修 3 + 4
}

// TanStack Query 版（4 坑全免 + 自动多组件共享）
function User({ id }: { id: string }) {
  const { data, isPending, error } = useQuery({
    queryKey: ['user', id],
    queryFn: ({ signal }) => fetch(`/api/users/${id}`, { signal }).then(r => r.json()),
  });
  // 切换 id 自动 cancel；StrictMode 自动去重；三态自动管；多组件共享 cache
}

05 常见陷阱

  • 把"useEffect + fetch 简单"当真 —— 网上 90% 的"React 怎么 fetch"教程是反模式版本。简洁是表面，4 大坑全在
  • 项目里只有 1-2 处 fetch 就觉得不需要 TanStack Query —— 项目会增长。先 1 处 useEffect+fetch，半年后变 10 处，重构成本陡增
  • 从 Redux+thunk 全量迁 TanStack Query 一次性改完 —— 应该渐进迁：先把新代码用 TanStack Query 写，旧的留着；旧代码痛了再迁那一处
  • Apollo 项目盲目切 TanStack Query —— 还在写 GraphQL 的项目继续用 Apollo / Relay 更合适。TanStack Query 是 REST 友好但 GraphQL 上没有 normalized cache
  • 新 Next.js App Router 项目装 SWR —— 2026 起 App Router 项目客户端 fetch 库默认 TanStack Query。SWR 维护已放缓，App Router 又被 RSC 抢戏
  • RTK Query 用在非 Redux 项目 —— 为了 RTK Query 引入 Redux store 不划算。新项目直接 TanStack Query
  • "我用 useEffect+fetch 没出过 bug" —— 通常是项目还小或网络稳定的幸存者偏差。竞态在 dev 看不到、prod 偶发，难追
  • 把 TanStack Query 当 Redux 替代品 —— 它只管 server state；client state（modal 是否打开 / 表单输入）仍用 useState / Zustand。详见 P1.2
  • RSC 项目里把 useQuery 套在 Server Component 外层 —— Server Component 不能用 hooks；首屏数据应该走 RSC fetch + 'use cache'，TanStack Query 留给客户端独有场景。详见 P5.1
  • "我们项目老，已经写了 useEffect+fetch，不想改" —— 不必一次改完，但新代码不再用 useEffect+fetch。两年后老代码会被自然替换
