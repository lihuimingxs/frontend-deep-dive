useQuery 全谱：状态机 + 选项 + 派生数据

useQuery 返回的 status / fetchStatus / isPending / isFetching / isError 一堆，到底有什么差别？什么时候用哪个？这一章把状态机讲清楚 + 把 select / placeholderData / enabled 这些容易被忽略但杀手级的选项讲透。

阅读时长 ~25 分钟 前置 P2 心智 + 写过至少 5 个 useQuery 主要来源 tanstack.com/query/latest/docs/framework/react/reference/useQuery + TkDodo 关于 status 状态的多篇博文

01 速览

  1. 两套状态正交：status（'pending' | 'success' | 'error' —— 数据视角，"我有没有数据"）+ fetchStatus（'idle' | 'fetching' | 'paused' —— 网络视角，"现在正在请求吗"）。组合产生 stale-while-revalidate UI。
  2. 4 种关键状态组合：(1) pending + fetching 第一次拿数据 / (2) success + idle fresh 状态稳定 / (3) success + fetching stale-while-revalidate 后台刷 / (4) error + idle 失败稳定。
  3. v5 改名：loading → pending（与 React 19 promise 状态对齐）；isLoading 仍可用（语义：第一次 fetch 中），但 isPending 是新规范。
  4. select 是被低估的性能武器：从 data 派生子集 + memoize；只有 select 结果变化才 re-render。多个组件用同一 query 但只关心不同字段时差异巨大。
  5. placeholderData + keepPreviousData：切 queryKey 时（分页 / 过滤）显示上一份数据 + 后台拉新；切页不闪烁的标配。
  6. enabled 控制 dependent query：enabled: !!previousData 串行触发；不要用 if 包 useQuery（hooks 不能条件调用）。
  7. refetchInterval：轮询；可以是函数（按状态决定停 / 继续）；refetchIntervalInBackground 控制 tab 后台时是否继续。
  8. retry 默认 3 次指数退避：4xx 自动 throw 不重试是常见诉求 → 把 retry 写成函数检查 status code。
  9. structuralSharing 默认开：v5 内部深比对前后引用，只有真正变化的字段触发 re-render。这是为什么 setQueryData 不能直接 mutate。
  10. v5 移除了 useQuery 的 onSuccess / onError / onSettled：query 层不再有这些 callback，只有 useMutation 保留。query 层用 useEffect 监听 status 改。

02 底层逻辑

双状态正交 · status vs fetchStatus

这是 useQuery 最容易混淆的部分。两套状态描述不同维度：

  • status 是数据视角 · "我现在有没有数据"
    • 'pending' · 没数据，第一次拿（之前没 cache）
    • 'success' · 有数据（无论 fresh / stale / fetching）
    • 'error' · 拿数据失败（且没有更老的成功数据）
  • fetchStatus 是网络视角 · "现在请求在不在跑"
    • 'idle' · 没在请求
    • 'fetching' · 正在请求
    • 'paused' · 暂停（如 networkMode: 'offlineFirst' 离线时）

两个组合 4×3=12 种理论状态，但实际有意义的就 4 种：

status fetchStatus 含义 UI 该怎么显示
pending fetching 第一次拿数据 Loading skeleton
success idle fresh 数据稳定 正常显示
success fetching stale-while-revalidate 后台刷新 显示数据 + 小 spinner
error idle 失败稳定（无 cache 数据） 错误 + retry 按钮

渲染惯用模式：

tsx 
function UserList() {
  const { data, isPending, isFetching, isError, error, refetch } = useQuery(...);

  if (isPending) return <Loading />;          // 没数据 + 在拿
  if (isError) return <Error error={error} onRetry={refetch} />;
  // 这里 status === 'success'，data 一定有
  return (
    <>
      <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>
      {isFetching && <Spinner size="small" />}  {/* 后台刷新提示 */}
    </>
  );
}

这个写法的好处：用户进页面 → 第一次 loading → 拿到数据 → 之后切回都直接显示（无 loading），后台刷新静默进行。

v5 改名细节

  • isLoading 在 v5 仍可用，但语义被收窄：等同于 isPending && isFetching（"第一次 fetch 进行中"）。原因：在没有 cache 但 query 已 disabled 的情况下，pending 但 fetchStatus 是 idle，这时 loading 不直观
  • status: 'loading' → status: 'pending'：与 React 19 Promise 状态对齐
  • cacheTime → gcTime：原名容易和 staleTime 混淆
  • useQuery(['key'], fn, options) 多参签名 → useQuery({ queryKey, queryFn, ...options }) 单一 object 签名

v3/v4 老代码迁 v5 主要工作量在这些改名 + 移除 useQuery 上的 onSuccess/onError/onSettled。

select · 派生数据 + memo

select 是从 data 派生的转换函数。多个组件用同一 query 但只关心不同字段时杀手级有用：

tsx 
// 组件 A：只关心 user count
function UserCount() {
  const { data: count } = useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
    select: (users) => users.length,  // 派生 number
  });
  return <span>{count} users</span>;
  // 即使列表里某个 user 改名，count 不变 → 这个组件不 re-render
}

// 组件 B：只关心 active users
function ActiveUserList() {
  const { data: activeUsers } = useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
    select: (users) => users.filter(u => u.active),
  });
  return <ul>{activeUsers?.map(...)}</ul>;
}

// 组件 C：拿全量
function FullUserList() {
  const { data: users } = useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
    // 不传 select，拿原始 data
  });
}

// 三个组件共享同一 cache + 同一 fetch
// 但 re-render 范围按 select 结果隔离

关键点：

  • select 结果跟 prev 用 structuralSharing 比较（v5 默认开）
  • 引用稳定才能避免 re-render；select 返回新引用每次都触发
  • 性能场景：大列表（1000+ 条）+ 多组件订阅 + 每个组件只看一部分 → select 几乎必备

select 引用稳定的细节

tsx 
// ❌ 每次 select 创建新对象 → re-render 频繁（v5 structuralSharing 部分救场，但仍非最优）
useQuery({
  ...,
  select: (users) => ({ count: users.length, max: Math.max(...users.map(u => u.age)) }),
});

// ✅ 用 useCallback 稳定 select 函数（v5 不强制，但更可控）
const select = useCallback(
  (users: User[]) => ({ count: users.length, max: Math.max(...users.map(u => u.age)) }),
  []
);
useQuery({ ..., select });

// ✅ 或派生原始值（number / string / boolean）
useQuery({ ..., select: (users) => users.length });

placeholderData + keepPreviousData

切 queryKey 时（分页 / 过滤 / 排序）默认行为：新 query 进 pending → 显示 loading → 新数据来 → 显示。每次切都闪一下。

v5 提供 placeholderData: keepPreviousData helper：

tsx 
import { keepPreviousData, useQuery } from '@tanstack/react-query';

function UserList({ page, filter }: { page: number; filter: string }) {
  const { data, isPlaceholderData, isFetching } = useQuery({
    queryKey: ['users', { page, filter }],
    queryFn: () => api.getUsers({ page, filter }),
    placeholderData: keepPreviousData,
  });

  // 切页时：
  // - data 立即变成上一次成功的数据（旧页数据）
  // - isPlaceholderData = true（指示这是占位）
  // - isFetching = true（后台拉新页）
  // 新页来了：
  // - data 自动替换
  // - isPlaceholderData = false

  return (
    <div style={{ opacity: isPlaceholderData ? 0.5 : 1 }}>
      <ul>{data?.map(...)}</ul>
      {isFetching && <Spinner />}
    </div>
  );
}

不依赖前一份数据时，placeholderData 也可以是静态值：

tsx 
// 显示骨架数据
useQuery({
  queryKey: ['user', id],
  queryFn: ...,
  placeholderData: { id, name: 'Loading...', avatar: null } as User,
  // 第一次 mount 也立刻有数据，避免 loading
});

// initialData vs placeholderData 区别：
// - placeholderData：不进 cache；新数据来覆盖；适合骨架
// - initialData：进 cache（视为 fresh，无 dataUpdatedAt 不会触发 refetch）；适合 SSR / hydration

enabled · dependent query

有时候 query B 依赖 query A 的结果。不要用 if 包 useQuery（hooks 不能条件调用）：

tsx 
// ❌ 错：违反 hooks 规则
function UserPosts() {
  const { data: user } = useQuery({ queryKey: ['user'], queryFn: api.getUser });
  if (!user) return <Loading />;
  // 下面这行违反规则（条件调用 hook）
  const { data: posts } = useQuery({ queryKey: ['posts', user.id], queryFn: () => api.getPosts(user.id) });
}

// ✅ 正：用 enabled
function UserPosts() {
  const { data: user } = useQuery({ queryKey: ['user'], queryFn: api.getUser });
  const { data: posts } = useQuery({
    queryKey: ['posts', user?.id],
    queryFn: () => api.getPosts(user!.id),
    enabled: !!user,  // user 拿到才触发
  });
  // user 还没来时 posts query status = 'pending' but fetchStatus = 'idle'
}

enabled 也可以用于"用户操作后才拉":

tsx 
function SearchBox() {
  const [query, setQuery] = useState('');
  const debouncedQuery = useDebouncedValue(query, 300);
  const { data } = useQuery({
    queryKey: ['search', debouncedQuery],
    queryFn: () => api.search(debouncedQuery),
    enabled: debouncedQuery.length >= 2,  // 至少 2 字才搜
  });
  // ...
}

refetchInterval · 轮询

简单形态：

tsx 
useQuery({
  queryKey: ['stock-price', symbol],
  queryFn: ...,
  refetchInterval: 5000,  // 每 5 秒
  refetchIntervalInBackground: false,  // tab 后台时停（默认 false）
});

函数形态（按状态决定）：

tsx 
useQuery({
  queryKey: ['job-status', jobId],
  queryFn: () => api.getJobStatus(jobId),
  refetchInterval: (query) => {
    const status = query.state.data?.status;
    if (status === 'completed' || status === 'failed') return false;  // 终态停止
    if (status === 'processing') return 1000;  // 处理中加快
    return 5000;
  },
});

这个模式（"任务进度查询"）非常常用：terminal state 停轮询，processing 时加快。

retry · 失败重试

默认：失败重试 3 次，指数退避（1s / 2s / 4s）。常见调整：

tsx 
// 4xx 不重试（业务错误，重试也是 4xx）
useQuery({
  queryKey: ['user', id],
  queryFn: async () => {
    const r = await fetch(`/api/users/${id}`);
    if (!r.ok) {
      const err = new Error(`HTTP ${r.status}`);
      (err as any).status = r.status;
      throw err;
    }
    return r.json();
  },
  retry: (failureCount, error: any) => {
    if (error.status >= 400 && error.status < 500) return false;
    return failureCount < 3;
  },
});

// 关键操作只重试一次
useQuery({
  ...,
  retry: 1,
});

// 完全不重试
useQuery({
  ...,
  retry: false,
});

// 自定义退避
useQuery({
  ...,
  retryDelay: (attemptIndex) => Math.min(1000 * 2 ** attemptIndex, 30_000),
});

structuralSharing · re-render 减负

v5 默认开。原理：fetch 完成后，TanStack Query 内部用 replaceEqualDeep 比对前后 data；只在真正变化的地方替换引用，其他保持原引用：

javascript 
// 简化原理
const oldData = { user: { id: 1, name: 'A' }, count: 5 };
const newData = { user: { id: 1, name: 'A' }, count: 5 };
// 比对后：完全相同 → 返回 oldData（同一引用）
// useQuery 不触发 re-render

const oldData2 = { user: { id: 1, name: 'A' }, count: 5 };
const newData2 = { user: { id: 1, name: 'A' }, count: 6 };
// 比对：user 没变 → 复用 user 引用；count 变了 → 新对象
// 用 select: data => data.user 的组件不 re-render
// 用 select: data => data.count 的组件 re-render

这是为什么 setQueryData 不能直接 mutate（破坏引用比对）。也是为什么 select 引用稳定能减少 re-render。

v5 移除 useQuery 上的 onSuccess / onError / onSettled

v4 时代 useQuery 支持这三个 callback：

tsx 
// v4 写法（v5 已移除）
useQuery(['users'], api.getUsers, {
  onSuccess: (data) => toast.success('Loaded'),
  onError: (err) => toast.error(err.message),
});

v5 移除原因（Tanner / TkDodo 多次解释）：

  • 这些 callback 触发时机有歧义（每次 success 都跑？还是仅第一次？多组件订阅时跑几次？）
  • 导致重复 toast / side effect
  • 正确处理"成功后做 X"应该用 useEffect 监听 isSuccess

v5 推荐写法：

tsx 
// v5 推荐：useEffect 监听
const { data, isSuccess, isError, error } = useQuery({ ... });

useEffect(() => {
  if (isSuccess) toast.success('Loaded');
}, [isSuccess]);

// 或全局 callback（QueryCache）
new QueryClient({
  queryCache: new QueryCache({
    onError: (error, query) => toast.error(`${query.queryKey} failed: ${error.message}`),
    onSuccess: (data, query) => { /* ... */ },
  }),
});

useMutation 仍然保留 onSuccess / onError / onSettled，因为 mutation 是命令式触发，时机明确。详见 P3.2。

03 应用场景

场景 1 · 大列表多组件订阅 · 用 select 优化 re-render

1000 条 user 列表，10 个组件分别显示统计、过滤后子集、特定 user 详情：

tsx 
// 1 个 query，10 个组件订阅
function useUsers() {
  return useQuery({ queryKey: ['users'], queryFn: api.getUsers });
}

// 各组件用 select 派生
function UserCount() {
  const { data } = useQuery({ ...useUsers, select: (users) => users.length });
  return <span>{data}</span>;
}

function ActiveCount() {
  const { data } = useQuery({ ...useUsers, select: (users) => users.filter(u => u.active).length });
  return <span>{data}</span>;
}

function UserDetail({ id }: { id: string }) {
  const { data } = useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
    select: (users) => users.find(u => u.id === id),
  });
  return <div>{data?.name}</div>;
}

// 改一个 user 的 name → 列表整体引用变 → users 引用变
// 但 UserCount 的 select 结果（length）没变 → 不 re-render
// ActiveCount 的 select 结果（active count）可能变 → 视情况 re-render
// UserDetail（id 是改的那个）的 select 结果变 → re-render

场景 2 · 异步任务进度查询 · refetchInterval 函数形态

tsx 
type JobStatus = 'queued' | 'processing' | 'completed' | 'failed';

function JobProgress({ jobId }: { jobId: string }) {
  const { data, isError } = useQuery({
    queryKey: ['job', jobId],
    queryFn: () => api.getJobStatus(jobId),
    refetchInterval: (query) => {
      const status = query.state.data?.status;
      // 终态：停止
      if (status === 'completed' || status === 'failed') return false;
      // queued：慢轮询
      if (status === 'queued') return 5000;
      // processing：快轮询
      return 1000;
    },
    refetchIntervalInBackground: false,
  });

  if (isError) return <Error />;
  return (
    <>
      <p>Status: {data?.status}</p>
      {data?.status === 'processing' && <ProgressBar value={data.progress} />}
      {data?.status === 'completed' && <DownloadButton url={data.resultUrl} />}
    </>
  );
}

场景 3 · 搜索 debounce + enabled

tsx 
function useDebouncedValue<T>(value: T, delay: number): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const t = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(t);
  }, [value, delay]);
  return debounced;
}

function SearchBox() {
  const [query, setQuery] = useState('');
  const debounced = useDebouncedValue(query, 300);

  const { data, isFetching } = useQuery({
    queryKey: ['search', debounced],
    queryFn: () => api.search(debounced),
    enabled: debounced.length >= 2,
    placeholderData: keepPreviousData,
  });

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {isFetching && <Spinner />}
      {data?.map(r => <li key={r.id}>{r.title}</li>)}
    </>
  );
}

04 典型代码

代码 1 · status 状态完整渲染矩阵

tsx 
function UserList() {
  const query = useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
    staleTime: 60 * 1000,
  });

  // 模式 A：早返回（推荐）
  if (query.isPending) return <Loading />;
  if (query.isError) return <Error error={query.error} onRetry={() => query.refetch()} />;
  return <UserListContent users={query.data} isFetching={query.isFetching} />;

  // 模式 B：switch（适合多状态）
  switch (query.status) {
    case 'pending': return <Loading />;
    case 'error': return <Error />;
    case 'success': return <Content />;
  }
}

代码 2 · 自定义 hook 配 staleTime / select / placeholderData

tsx 
// hooks/useUsers.ts
import { keepPreviousData, useQuery } from '@tanstack/react-query';

type UsersFilter = { active?: boolean; role?: string };

export function useUsers(filter: UsersFilter = {}) {
  return useQuery({
    queryKey: ['users', filter],
    queryFn: ({ signal }) => fetch('/api/users?' + new URLSearchParams(filter as any), { signal }).then(r => r.json()),
    staleTime: 30 * 1000,
    placeholderData: keepPreviousData,
  });
}

export function useUserCount(filter: UsersFilter = {}) {
  return useQuery({
    queryKey: ['users', filter],
    queryFn: ({ signal }) => fetch('/api/users?' + new URLSearchParams(filter as any), { signal }).then(r => r.json()),
    select: (users: User[]) => users.length,
    staleTime: 30 * 1000,
  });
}

export function useActiveUsers() {
  return useQuery({
    queryKey: ['users'],
    queryFn: api.getUsers,
    select: (users: User[]) => users.filter(u => u.active),
  });
}

代码 3 · networkMode（离线友好）

tsx 
// v5 networkMode 选项
useQuery({
  queryKey: ['users'],
  queryFn: api.getUsers,
  networkMode: 'online',         // 默认：离线 paused，连上后继续
  // networkMode: 'always',      // 永远尝试（适合 SW + offline cache）
  // networkMode: 'offlineFirst', // 离线时仍 cache 优先；上线后 revalidate
});

代码 4 · meta 字段携带元信息

tsx 
// 把 query 的"含义"挂在 meta 上
useQuery({
  queryKey: ['users'],
  queryFn: api.getUsers,
  meta: {
    persist: true,           // 自定义：是否进 localStorage 持久化
    errorMessage: '加载用户失败',
    auditCategory: 'admin',
  },
});

// 全局 onError 读 meta
queryCache: new QueryCache({
  onError: (error, query) => {
    const msg = query.meta?.errorMessage as string | undefined;
    toast.error(msg ?? error.message);
  },
}),

05 常见陷阱

  • 把 isPending 当 isFetching —— isPending 是"没数据"，isFetching 是"在请求"。stale 时 fetching 但 isPending=false。后台刷新用 isFetching 显示 indicator
  • v5 仍用 v4 的 onSuccess/onError —— useQuery 上已移除（mutation 仍有）。改用 useEffect 监听 isSuccess / isError 或全局 onError
  • 把 onSuccess 用来 invalidate 其他 query —— 即使 v4 也是反模式（多组件订阅时跑多次）。invalidate 应该在 useMutation 的 onSuccess 里
  • select 返回新对象 + 没 useCallback —— select 函数引用每次 render 变 → v5 仍工作（structuralSharing 比对结果）但不最优。返回原始值或 useCallback 包
  • 用 useState 缓存 useQuery 的 data —— 反模式。const [users, setUsers] = useState([]); useEffect(() => setUsers(query.data), [query.data]); 直接用 query.data
  • 条件调用 useQuery（用 if 包） —— 违反 hooks 规则。改用 enabled
  • enabled 用了不稳定值 —— enabled: query.length > 0 中 query 是个对象 → 引用每次变。enabled 应该是 boolean 或稳定 boolean 表达式
  • refetchInterval 不停 + 关 tab —— 默认 refetchIntervalInBackground=false，tab 后台不轮询。但有些项目错误调成 true 导致 tab 后台仍发请求
  • retry 默认 3 次让 4xx 也重试 —— 业务错误（401 / 404 / 422）重试也是 4xx，浪费。把 retry 写成函数检查 status
  • placeholderData 进 cache 期望持久化 —— placeholderData 不进 cache；想进 cache 用 initialData
  • 把 select 当 reducer 用（在 select 里调 setState） —— select 是纯函数。副作用应该在 useEffect
  • setQueryData 直接 mutate —— setQueryData(key, (old) => { old.name = 'X'; return old; }) 破坏 structuralSharing。要返回新对象
  • refetchInterval 设短 + queryFn 长 —— 上一个还没回下一个又起。TanStack Query 同 key 同时只一个 in-flight，但实战仍建议 interval > 平均请求时间
