useMutation + invalidation + 乐观更新

useMutation 是 TanStack Query 的"写"操作。最容易踩坑的部分是乐观更新的 4 步缺一不可（cancelQueries → 保存 prev → setQueryData 写新值 → onError 回滚 / onSettled invalidate）。漏一步出 bug。这一章把 4 步讲透 + 介绍 v5 的 useMutationState 新写法 + 厘清与 React 19 useOptimistic 的关系。

阅读时长 ~30 分钟 前置 P3.1 useQuery 全谱 + 写过至少一个 useMutation 主要来源 tanstack.com/query/latest/docs/framework/react/guides/optimistic-updates + Tanner 关于乐观更新的多篇博文

01 速览

  1. useMutation 不自动触发：与 useQuery 不同，调用 mutate(variables) 才执行。无 queryKey（默认；v5 可选 mutationKey）。
  2. 状态字段：isPending（在跑） / isSuccess / isError / data（成功值） / error / variables（最新提交参数）。
  3. 4 个回调：onMutate（提交前，乐观写入）/ onSuccess（成功）/ onError（失败回滚）/ onSettled（无论成败，最后一步 invalidate）。
  4. 乐观更新 4 步缺一不可：(1) cancelQueries 取消进行中的 fetch（避免 race） → (2) 保存 prev cache（onError 回滚用） → (3) setQueryData 写乐观值 → (4) onError 回滚 + onSettled invalidate。
  5. v5 useMutationState 派生乐观 UI：不动 cache 的乐观更新；适合"乐观显示但不影响其他组件"的场景。
  6. 与 React 19 useOptimistic 的关系：不冲突，分工不同。useOptimistic 管 form transition 临时状态；TanStack Query 管 cache。复杂场景两者协作。
  7. useMutation 仍保留 onSuccess / onError / onSettled（v5 没像 useQuery 那样移除），因为 mutation 是命令式触发，时机明确。
  8. mutationFn 调 Server Action：在 Next.js App Router 项目里 mutationFn 直接调 'use server' 函数；与 React 19 form action 协作。详见 P5.2。
  9. onMutate 返回的对象会作为 context 传给 onError / onSettled：这是 4 步乐观更新的实现机制 —— 在 onMutate return prev，onError 拿到 context.prev 回滚。
  10. "乐观更新出错没回滚"是踩坑率最高的 bug：症状是 UI 永远显示乐观值，server 已失败但用户不知。修复：onError 必须把 prev 写回 cache。

02 底层逻辑

useMutation 基础

tsx 
import { useMutation, useQueryClient } from '@tanstack/react-query';

function AddTodoForm() {
  const queryClient = useQueryClient();
  const { mutate, isPending, isError, error } = useMutation({
    mutationFn: (newTodo: { text: string }) => api.addTodo(newTodo),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['todos'] });
    },
    onError: (err) => {
      toast.error(`Failed: ${err.message}`);
    },
  });

  return (
    <form onSubmit={(e) => {
      e.preventDefault();
      const text = (new FormData(e.currentTarget).get('text') as string);
      mutate({ text });
    }}>
      <input name="text" required />
      <button disabled={isPending}>{isPending ? 'Adding...' : 'Add'}</button>
      {isError && <p>{error.message}</p>}
    </form>
  );
}

关键点：

  • mutationFn 接受 variables，返回 promise
  • 调用 mutate({...}) 触发；mutateAsync({...}) 返回 promise
  • 4 个回调按顺序：onMutate → onSuccess / onError → onSettled
  • 不需要 queryKey（mutation 不进 cache）；需要 mutationKey 才能用 useMutationState 查询

onMutate / onSuccess / onError / onSettled · 完整 lifecycle

tsx 
useMutation({
  mutationFn: api.addTodo,
  onMutate: async (newTodo) => {
    // 1. 提交前。乐观写入 + 返回 context（传给 onError）
    return { rollback: () => { /* ... */ } };
  },
  onSuccess: (data, variables, context) => {
    // 2a. 成功。data 是 mutationFn 返回值
    console.log('Created:', data);
  },
  onError: (error, variables, context) => {
    // 2b. 失败。context 是 onMutate 返回的对象
    context?.rollback?.();
  },
  onSettled: (data, error, variables, context) => {
    // 3. 无论成败。通常用来 invalidate
    queryClient.invalidateQueries({ queryKey: ['todos'] });
  },
});

乐观更新 4 步 · 缺一不可

乐观更新（optimistic update）的目标：用户提交后立即显示"似乎成功"，不等 server roundtrip。如果 server 实际失败，回滚 UI 到提交前。

正确实现需要 4 步，缺任一步都会出 bug：

tsx 
useMutation({
  mutationFn: api.addTodo,
  onMutate: async (newTodo) => {
    // 步骤 1：cancelQueries 防止 race
    // 如果有正在跑的 todos fetch，等它取消，避免回来覆盖我们的乐观值
    await queryClient.cancelQueries({ queryKey: ['todos'] });

    // 步骤 2：保存 prev（onError 回滚用）
    const prev = queryClient.getQueryData<Todo[]>(['todos']);

    // 步骤 3：setQueryData 写乐观值
    queryClient.setQueryData<Todo[]>(['todos'], (old = []) => [
      ...old,
      { id: `temp-${Date.now()}`, text: newTodo.text, done: false },
    ]);

    // 把 prev 通过 context 传给 onError
    return { prev };
  },
  onError: (_err, _newTodo, context) => {
    // 步骤 4a：失败回滚
    if (context?.prev) {
      queryClient.setQueryData(['todos'], context.prev);
    }
  },
  onSettled: () => {
    // 步骤 4b：无论成败 invalidate（最终一致）
    queryClient.invalidateQueries({ queryKey: ['todos'] });
  },
});

每一步为什么必须有：

  • 步骤 1 cancelQueries：如果用户提交时有别的 in-flight fetch（refetchOnWindowFocus 或别的组件触发），那个 fetch 回来会用 server 旧数据覆盖你的乐观值。cancel 掉避免这个 race
  • 步骤 2 保存 prev：失败时需要回到提交前状态。getQueryData 拿到的是当前 cache 引用，传给 context
  • 步骤 3 setQueryData 写乐观值：用户立刻看到新 todo 出现在列表里
  • 步骤 4a onError 回滚：失败时把 prev 写回 cache。如果不回滚，UI 永远显示乐观值，但 server 已失败，用户不知道
  • 步骤 4b onSettled invalidate：成功时让 cache 与 server 对账（拿 server 真实数据替换乐观值，特别是 server 生成的 id）；失败时也 invalidate（让 cache 拿到 server 当前真实状态）

4 步合一，给用户的体验是：提交瞬间 UI 更新，要么 server 确认（数据保留），要么悄悄回滚。

v5 useMutationState · 不动 cache 的乐观更新

有时候不想动 cache（怕影响其他组件），只想本地组件显示乐观状态。v5 提供 useMutationState：

tsx 
// hooks/useTodoMutations.ts
export function useAddTodo() {
  return useMutation({
    mutationKey: ['add-todo'],  // 必须有 mutationKey 才能查
    mutationFn: api.addTodo,
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ['todos'] }),
  });
}

// 组件里同时拿真实列表 + 进行中的乐观项
import { useQuery, useMutationState } from '@tanstack/react-query';

function TodoList() {
  const { data: todos = [] } = useQuery({ queryKey: ['todos'], queryFn: api.getTodos });

  // 拿所有 mutationKey=['add-todo'] 且 status=pending 的 mutation 的 variables
  const pendingTodos = useMutationState({
    filters: { mutationKey: ['add-todo'], status: 'pending' },
    select: (mutation) => mutation.state.variables as { text: string },
  });

  // 合并真实数据 + 乐观项
  return (
    <ul>
      {todos.map(t => <li key={t.id}>{t.text}</li>)}
      {pendingTodos.map((t, i) => (
        <li key={`pending-${i}`} style={{ opacity: 0.5 }}>{t.text} (saving...)</li>
      ))}
    </ul>
  );
}

对比 onMutate setQueryData 路线：

  • setQueryData 路线：动 cache，所有订阅者立即更新；要管回滚；适合"成功率高 + 不想用户感到延迟"
  • useMutationState 路线：不动 cache，只本地组件显示乐观项；自动随 mutation 状态消失；适合"想强调'还在保存'"

与 React 19 useOptimistic 的关系

React 19 引入了 useOptimistic（详见 React P6.2）。它的设计目标是：form transition 期间显示乐观状态，transition 结束自动 revert。

看一下 useOptimistic 的典型用法：

tsx 
'use client';
import { useOptimistic, startTransition } from 'react';

function LikeButton({ likes, addLikeAction }) {
  const [optimisticLikes, addOptimisticLike] = useOptimistic(
    likes,
    (current, _add: number) => current + 1
  );

  async function handleClick() {
    startTransition(async () => {
      addOptimisticLike(1);
      await addLikeAction();  // Server Action
    });
  }

  return <button onClick={handleClick}>❤️ {optimisticLikes}</button>;
}

看起来和 TanStack Query 乐观更新很像，但分工不同：

能力 TanStack Query (onMutate) React 19 useOptimistic
作用范围 cache 全局（所有订阅者立即更新） 组件本地（form transition 期）
持续时间 到 invalidate / refetch 替换 到 transition 结束自动 revert
失败处理 手动 onError 回滚 自动 revert（无需手写）
跨组件共享 是 否
典型场景 添加 / 删除 / 编辑列表项 like 计数 / 喜欢按钮

实战分工：

  • RSC + Server Actions 项目里简单乐观（按钮 like / star） → useOptimistic 更顺手
  • 跨组件 cache 共享（添加 todo 后多个组件要看到）→ TanStack Query onMutate
  • 复杂场景两者协作：useOptimistic 管按钮 transition，TanStack Query 管列表 cache

详细配合见 P5.2。

mutationFn 调 Server Action

Next.js App Router 项目里 mutationFn 可以直接调 'use server' 函数。这是 RSC 时代 TanStack Query 的常见用法之一：

tsx 
// app/_actions/todos.ts
'use server';
export async function addTodoAction(text: string) {
  await db.todos.create({ data: { text } });
  // 不在这里 revalidatePath（让 client 决定是否跨 cache 同步）
  return { success: true };
}

// app/_components/AddTodo.tsx
'use client';
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { addTodoAction } from '@/app/_actions/todos';

function AddTodo() {
  const queryClient = useQueryClient();
  const { mutate, isPending } = useMutation({
    mutationFn: addTodoAction,  // 直接调 Server Action
    onMutate: async (text) => {
      // 乐观更新 client cache
      await queryClient.cancelQueries({ queryKey: ['todos'] });
      const prev = queryClient.getQueryData(['todos']);
      queryClient.setQueryData(['todos'], (old: any[] = []) => [
        ...old,
        { id: `temp-${Date.now()}`, text, done: false },
      ]);
      return { prev };
    },
    onError: (_e, _v, ctx) => ctx && queryClient.setQueryData(['todos'], ctx.prev),
    onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos'] }),
  });
  // ...
}

详细的 server cache + client cache 双向同步问题见 P5.2。

mutationKey · 给 mutation 命名

不传 mutationKey 也能用，但传了就能用 useMutationState 查询、按 key 隔离 retry：

tsx 
useMutation({
  mutationKey: ['todo', 'add'],
  mutationFn: api.addTodo,
});

// 查所有 ['todo', 'add'] 的 pending mutation
const pending = useMutationState({
  filters: { mutationKey: ['todo', 'add'], status: 'pending' },
});

// 全局 mutationCache 配置 retry / 错误处理
new QueryClient({
  defaultOptions: {
    mutations: {
      retry: 1,
      onError: (err, vars, ctx, mutation) => {
        if (mutation.options.mutationKey?.[0] === 'todo') {
          toast.error(`Todo operation failed: ${err.message}`);
        }
      },
    },
  },
});

并发 mutation 的处理

用户连续点击 3 次"Add Todo"按钮，3 个 mutation 同时跑：

  • 每个 mutation 独立有自己的 onMutate / onError / onSettled
  • 每个 onMutate 都 cancelQueries + setQueryData → 第 2 个 mutate 的 prev 是"第 1 个 onMutate 后的 cache"，第 3 个的 prev 是"第 2 个 onMutate 后的 cache"
  • 3 个都成功：onSettled 触发 3 次 invalidate（其实 1 次也够）
  • 第 2 个失败：回滚到"第 1 个 onMutate 后状态"（不是初始状态）

实战如果担心并发问题，用 mutate 的 onSuccess 触发下一个，或用 mutateAsync 串行 await。

03 应用场景

场景 1 · todo 添加 / 删除 / 编辑完整闭环

tsx 
// hooks/useTodoMutations.ts
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { todoKeys } from '@/queries/todos';

export function useAddTodo() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: api.addTodo,
    onMutate: async (newTodo: { text: string }) => {
      await qc.cancelQueries({ queryKey: todoKeys.lists() });
      const prev = qc.getQueryData<Todo[]>(todoKeys.list('all'));
      qc.setQueryData<Todo[]>(todoKeys.list('all'), (old = []) => [
        ...old,
        { id: `temp-${Date.now()}`, text: newTodo.text, done: false },
      ]);
      return { prev };
    },
    onError: (_e, _v, ctx) => ctx && qc.setQueryData(todoKeys.list('all'), ctx.prev),
    onSettled: () => qc.invalidateQueries({ queryKey: todoKeys.lists() }),
  });
}

export function useToggleTodo() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: ({ id, done }: { id: string; done: boolean }) => api.updateTodo(id, { done }),
    onMutate: async ({ id, done }) => {
      await qc.cancelQueries({ queryKey: todoKeys.lists() });
      const prev = qc.getQueryData<Todo[]>(todoKeys.list('all'));
      qc.setQueryData<Todo[]>(todoKeys.list('all'), (old = []) =>
        old.map(t => t.id === id ? { ...t, done } : t)
      );
      return { prev };
    },
    onError: (_e, _v, ctx) => ctx && qc.setQueryData(todoKeys.list('all'), ctx.prev),
    onSettled: () => qc.invalidateQueries({ queryKey: todoKeys.lists() }),
  });
}

export function useDeleteTodo() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (id: string) => api.deleteTodo(id),
    onMutate: async (id) => {
      await qc.cancelQueries({ queryKey: todoKeys.lists() });
      const prev = qc.getQueryData<Todo[]>(todoKeys.list('all'));
      qc.setQueryData<Todo[]>(todoKeys.list('all'), (old = []) => old.filter(t => t.id !== id));
      return { prev };
    },
    onError: (_e, _v, ctx) => ctx && qc.setQueryData(todoKeys.list('all'), ctx.prev),
    onSettled: () => qc.invalidateQueries({ queryKey: todoKeys.lists() }),
  });
}

// 调用方
function TodoList() {
  const { data: todos } = useQuery({ queryKey: todoKeys.list('all'), queryFn: api.getTodos });
  const { mutate: add } = useAddTodo();
  const { mutate: toggle } = useToggleTodo();
  const { mutate: del } = useDeleteTodo();

  return (
    <>
      <form onSubmit={(e) => { e.preventDefault(); const text = (new FormData(e.currentTarget).get('text') as string); add({ text }); e.currentTarget.reset(); }}>
        <input name="text" required />
        <button>Add</button>
      </form>
      <ul>
        {todos?.map(t => (
          <li key={t.id}>
            <input type="checkbox" checked={t.done} onChange={() => toggle({ id: t.id, done: !t.done })} />
            <span>{t.text}</span>
            <button onClick={() => del(t.id)}>✕</button>
          </li>
        ))}
      </ul>
    </>
  );
}

注意：3 个 mutation 全都重复 4 步骤。可以抽 helper 减少样板：

tsx 
function useOptimisticListMutation<T extends { id: string }, V>(
  queryKey: any[],
  mutationFn: (vars: V) => Promise<unknown>,
  optimisticUpdater: (old: T[], vars: V) => T[],
) {
  const qc = useQueryClient();
  return useMutation({
    mutationFn,
    onMutate: async (vars: V) => {
      await qc.cancelQueries({ queryKey });
      const prev = qc.getQueryData<T[]>(queryKey);
      qc.setQueryData<T[]>(queryKey, (old = []) => optimisticUpdater(old, vars));
      return { prev };
    },
    onError: (_e, _v, ctx: any) => ctx?.prev && qc.setQueryData(queryKey, ctx.prev),
    onSettled: () => qc.invalidateQueries({ queryKey }),
  });
}

// 调用
const useAddTodo = () => useOptimisticListMutation<Todo, { text: string }>(
  todoKeys.list('all'),
  api.addTodo,
  (old, { text }) => [...old, { id: `temp-${Date.now()}`, text, done: false }],
);

场景 2 · 用 useMutationState 显示 pending 项

添加 todo 不动 cache，只本地组件叠加 pending 项显示：

tsx 
function TodoList() {
  const { data: todos = [] } = useQuery({ queryKey: ['todos'], queryFn: api.getTodos });

  const pendingAdds = useMutationState({
    filters: { mutationKey: ['add-todo'], status: 'pending' },
    select: (m) => m.state.variables as { text: string },
  });

  const pendingDeletes = useMutationState({
    filters: { mutationKey: ['delete-todo'], status: 'pending' },
    select: (m) => m.state.variables as string,  // id
  });

  // 视觉合并
  const visible = todos
    .filter(t => !pendingDeletes.includes(t.id))  // 删除中的 todo 提前隐藏
    .concat(pendingAdds.map((v, i) => ({ id: `pending-${i}`, text: v.text, done: false })));

  return <ul>{visible.map(t => <li key={t.id}>{t.text}</li>)}</ul>;
}

场景 3 · 与 React 19 useOptimistic 协作

RSC 项目里：列表 cache 用 TanStack Query 管，单个按钮 transition 用 useOptimistic：

tsx 
'use client';
import { useOptimistic } from 'react';
import { useMutation } from '@tanstack/react-query';

function PostCard({ post }: { post: Post }) {
  const [optimisticPost, updatePost] = useOptimistic(post, (curr, action: 'like') => ({
    ...curr,
    likes: curr.likes + 1,
    likedByMe: true,
  }));

  const { mutate } = useMutation({
    mutationFn: () => api.likePost(post.id),
    // 不动 cache，让 RSC revalidate 更新
  });

  return (
    <button onClick={async () => {
      updatePost('like');  // 立刻更新本地 transition 状态
      mutate();             // 异步发请求
    }}>
      ❤️ {optimisticPost.likes}
    </button>
  );
}

04 典型代码

代码 1 · mutate vs mutateAsync

tsx 
const { mutate, mutateAsync } = useMutation({ mutationFn: api.addTodo });

// mutate：fire-and-forget，错误进 onError，组件里用 isError / error 显示
function handleClick() {
  mutate({ text: 'X' });
  // 不能 await mutate，它返回 undefined
}

// mutateAsync：返回 promise，可以 await + try/catch
async function handleClick2() {
  try {
    const created = await mutateAsync({ text: 'X' });
    console.log('Got id:', created.id);
    // 然后 navigate / 弹 modal / 触发别的事
  } catch (e) {
    // 也可以这里 catch
  }
}

// 注意：mutateAsync 不抑制 onError；onError 仍会跑 + promise 也 reject
// 选择：需要 await 后续动作 → mutateAsync；只是触发 → mutate

代码 2 · 全局 mutation 错误处理

tsx 
new QueryClient({
  mutationCache: new MutationCache({
    onError: (error, variables, context, mutation) => {
      // 全部 mutation 错误的兜底
      console.error('Mutation failed:', mutation.options.mutationKey, error);
      toast.error(`操作失败: ${error.message}`);
    },
    onSuccess: (data, variables, context, mutation) => {
      // 全部 mutation 成功的兜底（可选）
    },
  }),
});

代码 3 · 用 zod 校验 mutation 输入

tsx 
import { z } from 'zod';

const addTodoSchema = z.object({ text: z.string().min(1).max(200) });

useMutation({
  mutationFn: async (input: unknown) => {
    const parsed = addTodoSchema.parse(input);  // throw if invalid
    return api.addTodo(parsed);
  },
  onError: (err) => {
    if (err instanceof z.ZodError) {
      toast.error('输入不合法: ' + err.issues[0].message);
    } else {
      toast.error('请求失败');
    }
  },
});

// structuralSharing 也会复用其他 9999 项的引用

05 常见陷阱

  • 乐观更新没 cancelQueries —— 后台 fetch 回来覆盖你的乐观值。这是踩坑率最高的 bug。乐观更新第 1 步必须 cancel
  • 乐观更新没保存 prev —— 出错没法回滚。第 2 步必须 getQueryData 存 context
  • onError 没回滚 —— 错误吞掉，UI 永远显示乐观值。第 4a 步必须 setQueryData(prev)
  • onSettled 没 invalidate —— 成功后 cache 是乐观值（可能和 server 真实值有差异，比如 server 生成的 id）；失败也没让 cache 与 server 对账。第 4b 步必须 invalidate
  • 用 invalidateQueries 代替 setQueryData 做乐观 —— invalidate 只让 cache stale 等下次激活才 refetch，不立即写新值。乐观更新必须 setQueryData
  • setQueryData 直接 mutate —— setQueryData(key, (old) => { old.x = 1; return old; }) 破坏 structuralSharing。要返回新对象 / 用 immer
  • onMutate 是 async 但忘了 await cancelQueries —— cancelQueries 返回 promise，没 await 就接着 setQueryData 时 race 仍可能发生
  • mutationFn 扔了 4xx 没 throw —— 同 useQuery，fetch 不自动 reject HTTP 错误。手动 throw 才能进 onError
  • 用 onMutate 触发 toast —— onMutate 是乐观更新阶段，不该有 user-facing side effect。toast 放 onSuccess / onError
  • mutate 后立刻 navigate —— mutate 是 fire-and-forget，立刻 navigate 走 onSettled 没跑。用 mutateAsync await + 然后 navigate；或在 onSuccess 里 navigate
  • onSuccess 里 setQueryData(server 返回值) —— 通常多此一举（onSettled 的 invalidate 会 refetch）。但如果想避免一次额外 fetch，是合法优化（直接 setQueryData server 返回的真实数据替换乐观值）
  • 多个 mutation 同时跑乐观 —— prev 串成链，回滚不一定到初始状态。如果担心，用 mutationKey + 串行（mutateAsync await）
  • useMutationState 不传 mutationKey —— 没 key 没法 filter 查询。需要 useMutationState 派生时必须给 mutation 加 mutationKey
  • "用 useEffect 监听 isSuccess 来 navigate" —— 不必。useMutation 的 onSuccess 是同步触发的，直接在 onSuccess 里 navigate 更合适
  • 把 useOptimistic 当 onMutate 替代 —— 不是同一类工具。useOptimistic 是 form transition 状态；onMutate 是 cache 操作。看场景选
