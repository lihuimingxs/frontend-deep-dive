P5 · RSC 共存 ⭐

整个主题最难、最容易误判的一节。Next.js 16 + React 19 之后，server-side fetch + Server Actions 已经能做大部分 server state 工作 —— 客户端 fetch 库还有位置吗？答案是有，但位置变了。这一阶段给完整决策矩阵 + 5 个 TanStack Query 仍不可替代场景 + 3 个被 RSC 接管场景，再讲透 hydration 桥（dehydrate / HydrationBoundary）+ use(promise) + Server Actions 协作的具体姿势。

章节数 2 主要来源 tanstack.com/query/latest/docs/framework/react/guides/advanced-ssr + Tanner Linsley 关于 RSC 的回应博文 + React 团队（Sebastian / Dan）公开讨论 + Next.js 官方 TanStack Query 集成文档 状态 ✅ 全部完成

本阶段覆盖

  • RSC 时代 server state 路径变化 · 首屏 → RSC fetch + 'use cache'；变更 → Server Action；客户端独有 → 仍 TanStack Query
  • 5 个 TanStack Query 不可替代场景 · 长生命周期 cache / infinite scroll / 乐观更新 / WebSocket 多组件订阅 / 第三方客户端 API
  • 3 个 RSC 接管场景 · 首屏渲染数据 / 简单数据变更 / SEO 关键页面
  • Tanner / React 团队 / Vercel 三方对照 · 官方说法对账，避免"我猜他们想这样"
  • dehydrate + HydrationBoundary · server-side prefetch → RSC payload 序列化 → client cache 反序列化
  • 每请求 new QueryClient() · 跨请求隔离避免 cache 污染
  • use(promise) 与 useSuspenseQuery · 关系是底层与上层；不要把 use(promise) 当 cache library
  • Server Actions + useMutation 协作 · mutationFn 直接调用 server action + 双向 invalidate（revalidatePath + queryClient.invalidateQueries）
  • 两层 cache 同步问题 · server cache + client cache 不同步是 RSC 时代新挑战

⚠️ 重要提示

这一阶段是 Next.js 项目最容易踩坑的部分。不读 P5 直接在 RSC 项目里装 TanStack Query 通常出现两类错误：（1）该用 RSC fetch 的地方仍然装客户端 fetch（包体膨胀 + 双重请求）；（2）跨请求 new QueryClient() 写错（cache 污染）。两章必须连读。
