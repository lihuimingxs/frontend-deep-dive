RSC 时代 TanStack Query 还该用吗 · 决策矩阵

整个主题最难、最容易误判的一节。Next.js 16 + React 19 之后，客户端 fetch 库的位置发生根本性变化 —— 不是消失，是收窄。这一章给清楚的决策矩阵：5 个 TanStack Query 仍不可替代的场景 + 3 个被 RSC 接管的场景 + Tanner / React 团队 / Vercel 三方官方说法对账。

阅读时长 ~30 分钟 前置 P1-P4 全部 + 读过 Next.js P3 RSC + Next.js P4 Server Actions 主要来源 Tanner Linsley 关于 RSC 的多次回应 + React 团队（Sebastian / Dan）公开讨论 + tanstack.com 官方 RSC 集成文档

01 速览

  1. 这一章没有"标准答案"。RSC 时代 TanStack Query 的位置取决于具体场景，不能一刀切"用"或"不用"。
  2. RSC 时代 server state 路径分三层：(1) 首屏数据 → Server Component fetch + 'use cache'；(2) 简单变更 → Server Action + revalidate；(3) 客户端独有 server state → TanStack Query。
  3. 5 个 TanStack Query 仍不可替代的场景：(1) 长生命周期 cache（infinite scroll / 跨导航 cache 复用） / (2) 实时刷新 / 轮询（refetchInterval / WebSocket setQueryData） / (3) 乐观更新（roundtrip 太慢） / (4) WebSocket / SSE 多组件订阅（推送数据进 cache 让多个组件共享） / (5) 第三方客户端 API（CORS / API key 在 client）。
  4. 3 个被 RSC 接管的场景：(1) 首屏渲染数据（async Server Component 比 useQuery 简单 + 零客户端 bundle） / (2) 简单数据变更（Server Action + revalidatePath 比 useMutation + invalidate 直接） / (3) SEO 关键页面（必须 server-render）。
  5. Tanner Linsley 的官方表态（2024 多次重申）："TanStack Query is not going away. We're complementary, not competing."
  6. React 团队（Sebastian / Dan）的官方表态："use API 是底层；上层 cache library 仍然有位置；不要把 useQuery 强行替换成 use(promise)。"
  7. Vercel（Next.js 团队）的官方表态：推 RSC + Server Actions，同时官方文档明确支持 hydration 桥的 TanStack Query 用法。
  8. RSC 时代客户端 fetch 库总需求量下降（30-50% 的 fetch 移到 server-side），但 TanStack Query 在剩余客户端场景份额仍在提升（吃 SWR 留下的份额）。
  9. 新项目决策原则：(1) Next.js + 重 SEO + 数据基本静态 → 不装客户端 fetch 库；(2) Next.js + 客户端交互重 → RSC 管首屏 + TanStack Query 管客户端；(3) 纯 SPA → 装 TanStack Query。
  10. "2026 起，新项目第一选择是 RSC + Server Actions，TanStack Query 用在剩下的客户端独有场景"是强观点。RSC 不是 TanStack Query 的替代品，是不同层的工具。

02 底层逻辑

RSC 之前 · 客户端 fetch 库覆盖全部 server state

2020-2023 年的 React 项目里，server state 全部走客户端 fetch 库：

  • 首屏数据 → useQuery（client mount 后 fetch + render）
  • 列表 / 详情 / 表单 → useQuery
  • 变更 → useMutation + onSuccess invalidate
  • 实时数据 → useQuery + refetchInterval / WebSocket setQueryData
  • 客户端 cache → 全部统一进 queryClient

这套模型简单一致。代价：

  • 首屏 SEO 差（client render → 搜索引擎拿不到内容）
  • 包体积大（client bundle 含所有 fetch 逻辑）
  • "loading 1 秒"是常态（client mount 后才发请求）

RSC 出现 · server state 路径分层

2024 年 React 19 + Next.js 16 RSC 落地后，server state 不再是"客户端 cache 一种心智"。变成三层结构：

层 谁管 典型场景 工具
1. 首屏 server-side RSC + 'use cache' blog 文章 / dashboard 静态部分 / 列表页首屏 async Server Component + fetch
2. 数据变更 Server Actions + revalidate 表单提交 / 简单 CRUD 'use server' + revalidatePath/Tag
3. 客户端独有 server state TanStack Query infinite / 实时 / WebSocket / 第三方 API useQuery / useMutation

每一层做自己最擅长的事：

  • 第 1 层：server-render 直接出 HTML，SEO 好，包体小，零 client loading
  • 第 2 层：表单提交不必走客户端 cache，server 直接处理 + revalidate
  • 第 3 层：客户端独有的复杂 server state（cache 跨导航 / 乐观 / 实时）仍然需要专门工具

5 个 TanStack Query 仍不可替代的场景 · 详细

场景 1 · 长生命周期客户端 cache

典型例子：用户在 dashboard 看 user list（page A） → 点进 detail（page B） → 切回 page A。

  • RSC 路径：每次进 page A 都重新 server-render（除非用 router cache，但行为有限）
  • TanStack Query 路径：第一次进 fetch + cache → 之后 stale-while-revalidate（立刻显示 + 后台刷）

结论：跨导航复用客户端 cache，TanStack Query 体验更好。Next.js Router Cache 部分覆盖这个场景但更受限（v15 默认 stale time 缩短）。

场景 2 · 实时刷新 / 轮询

典型例子：股票价格每 5 秒刷新；任务进度每秒查询；订单状态轮询。

  • RSC 路径：不擅长。RSC 是 server-render，轮询需要 client 重复请求 → 不如直接 client-side
  • TanStack Query 路径：refetchInterval 一行解决；函数形态可以根据状态调整

结论：客户端轮询场景 TanStack Query 是首选。

场景 3 · 乐观更新

典型例子：like 按钮 + 1；删除 todo；编辑评论。用户期望提交瞬间 UI 更新。

  • RSC + Server Action 路径：用户点击 → Server Action → revalidatePath → 重新渲染。roundtrip 不可避免，至少 200ms-500ms 延迟。可以配 React 19 useOptimistic 部分缓解
  • TanStack Query 路径：onMutate 4 步乐观更新 + 失败回滚 + onSettled 对账。体验最好

结论：复杂乐观更新（list 增删改）TanStack Query 仍是最完整的方案。简单 like 按钮可以用 useOptimistic + Server Action。

场景 4 · WebSocket / SSE 多组件订阅

典型例子：聊天消息推送；通知推送；多用户协作（Figma / Notion 风格）。

  • RSC 路径：不能用。RSC 是 server-render 静态产物，不能管 WebSocket lifecycle
  • TanStack Query 路径：WebSocket 收到消息 → setQueryData 写进 cache → 所有用 useQuery 订阅的组件同时更新

结论：实时 push 数据 TanStack Query 不可替代。

场景 5 · 第三方客户端 API

典型例子：

  • 用户用自己的 API key 调第三方 API（OpenAI / GitHub）
  • CORS 限制必须从 client 调（不能 proxy 到 server）
  • 客户端 SDK（Stripe / Firebase / Supabase JS SDK）

结论：这些场景 server-side 接不到，TanStack Query 是天然选择。

3 个被 RSC 接管的场景 · 详细

场景 1 · 首屏渲染数据

tsx 
// RSC 路径：极简、零客户端 bundle、SEO 友好
// app/posts/page.tsx (Server Component)
export default async function PostsPage() {
  const posts = await fetch('/api/posts', { next: { revalidate: 60 } }).then(r => r.json());
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}

// 等价的 TanStack Query 写法（client-side）
'use client';
export default function PostsPage() {
  const { data: posts, isPending } = useQuery({
    queryKey: ['posts'],
    queryFn: () => fetch('/api/posts').then(r => r.json()),
  });
  if (isPending) return <Loading />;  // 1 秒 loading
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}
// 包体积大（含 TanStack Query）；SEO 差；首屏慢

结论：首屏数据 RSC 完胜。

场景 2 · 简单数据变更

tsx 
// RSC + Server Action 路径
// app/_actions/posts.ts
'use server';
export async function deletePost(id: string) {
  await db.posts.delete({ where: { id } });
  revalidatePath('/posts');
}

// 调用（任意 Server / Client Component）
<form action={async () => deletePost(id)}>
  <button>Delete</button>
</form>

// 等价的 TanStack Query 路径
const { mutate } = useMutation({
  mutationFn: api.deletePost,
  onSuccess: () => queryClient.invalidateQueries({ queryKey: ['posts'] }),
});
<button onClick={() => mutate(id)}>Delete</button>

结论：

  • 简单变更（无乐观更新需求）→ Server Action 更简洁，progressive enhancement 也好（关 JS 也能 submit）
  • 复杂变更（乐观更新 / 多组件 cache 同步）→ TanStack Query 更体系

场景 3 · SEO 关键页面

marketing 页 / blog / 文档站 / 商品详情：必须 server-render（搜索引擎要看到内容）。client-side fetch 不可选。RSC 是默认。

三方官方说法对账

RSC 时代 TanStack Query 的位置不是猜测，三方都有公开表态：

Tanner Linsley（TanStack 创始人）

💡 Tanner 的多次回应（2024-2026）

「TanStack Query is not going away. RSC and Server Actions handle a different layer — they don't manage long-lived client-side cache, optimistic updates, infinite scroll, or WebSocket-pushed data with multi-component subscribers. We're complementary, not competing.」

「If your data is read-once-on-load and never changes from the client side, you don't need TanStack Query. If you have any of: client-side cache invalidation, optimistic updates, real-time data, infinite queries — you still want it.」

React 团队（Sebastian Markbåge / Dan Abramov）

💡 React 团队的表态

「The use API is a low-level primitive. Higher-level cache libraries like React Query are still needed for client-side caching, mutations, and lifecycle management.」

「Don't replace useQuery with use(promise). They serve different purposes — use is for consuming a promise within a render; useQuery is for a full query lifecycle with caching.」

Vercel（Next.js 团队）

Next.js 官方文档明确支持 TanStack Query 集成（dehydrate + HydrationBoundary 模式）。Vercel 工程 blog 也多次推荐"RSC 管首屏 + TanStack Query 管客户端独有场景"的组合。

三方说法一致：不是替代关系，是互补关系。

决策矩阵

拿到一个新场景，按 4 步判断要不要用 TanStack Query：

  1. 问 1：是首屏渲染数据吗？
    • 是 → RSC fetch + 'use cache'（不用 TanStack Query）
    • 不是 → 进下一问
  2. 问 2：只是表单提交一次性变更吗？
    • 是（无乐观更新需求） → Server Action + revalidatePath
    • 是（要乐观更新） → 进下一问
    • 不是 → 进下一问
  3. 问 3：有以下任一场景吗？
    • 跨导航 cache 复用
    • infinite scroll / 加载更多
    • 实时刷新 / 轮询
    • WebSocket / SSE 多组件订阅
    • 乐观更新
    • 第三方客户端 API
    • 是 → 用 TanStack Query
    • 不是 → 进下一问
  4. 问 4：还有什么是 RSC 解决不了的？
    • 没有 → 不用客户端 fetch 库
    • 有 → 用 TanStack Query 处理那部分

新项目典型决策

项目 A · marketing 站 / 文档站 / blog

  • 核心：内容静态，重 SEO，少交互
  • 结论：100% RSC，不装 TanStack Query

项目 B · 用户 dashboard（B2B SaaS）

  • 核心：用户登录后看复杂 dashboard，多列表 / 表单 / 实时数据
  • 结论：RSC 管 layout / 静态部分；TanStack Query 管动态列表 / 实时数据 / 乐观更新

项目 C · 电商商品详情

  • 核心：商品信息 SEO 重要；购物车 / 推荐 / 库存实时
  • 结论：商品信息 RSC；购物车 + 库存 TanStack Query

项目 D · 聊天 / 协作（Notion / Figma 风格）

  • 核心：实时 / WebSocket / 乐观更新 / 多用户协作
  • 结论：以 TanStack Query 为主，RSC 只做登录页 / 设置页

项目 E · 纯 SPA（不用 Next.js）

  • 核心：CSR，没有 RSC 选项
  • 结论：TanStack Query 是默认

从 SWR 切 TanStack Query 的判断

App Router 项目里之前装了 SWR，要不要切？

  • RSC 已经接管了 80%+ fetch → 留 SWR 也行（剩余客户端 query 简单）
  • 剩余客户端 query 复杂（infinite / 多组件 / 乐观）→ 切 TanStack Query 体验明显更好
  • 团队对 SWR 熟悉 + 没有具体痛点 → 不必切

详细对照见 P6.1。

03 应用场景

场景 1 · Next.js dashboard 完整路径

典型 dashboard：

  • 顶部 nav + user info → RSC（async Server Component 读 cookies + fetch user）
  • 主面板列表（重 SEO） → RSC + 'use cache'（静态时 ISR）
  • 实时数据卡片（订单数 / 在线人数） → TanStack Query + refetchInterval
  • infinite log 列表 → TanStack Query useInfiniteQuery
  • 编辑表单（乐观更新需求弱） → Server Action + form action
  • 编辑表单（乐观更新强：发评论） → TanStack Query useMutation + onMutate
  • 实时聊天 → TanStack Query + WebSocket setQueryData

每个区块用最适合的工具，整页协作。详见 P5.2 的具体代码。

场景 2 · 商品详情页混合

tsx 
// app/products/[id]/page.tsx (RSC)
export default async function ProductPage({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  // 商品基本信息走 RSC（SEO 关键）
  const product = await fetch(`/api/products/${id}`, { next: { revalidate: 300 } }).then(r => r.json());

  return (
    <Layout>
      <ProductInfo product={product} />  {/* server-rendered */}
      <Suspense fallback={<Skeleton />}>
        <ClientStockAndCart productId={id} initialPrice={product.price} />
      </Suspense>
    </Layout>
  );
}

// app/_components/ClientStockAndCart.tsx
'use client';
export default function ClientStockAndCart({ productId, initialPrice }) {
  // 库存实时刷新 → TanStack Query
  const { data: stock } = useQuery({
    queryKey: ['stock', productId],
    queryFn: () => api.getStock(productId),
    refetchInterval: 10_000,
  });

  // 加入购物车 → 乐观更新
  const { mutate: addToCart } = useMutation({...});

  return (
    <>
      <p>{stock?.count} in stock</p>
      <button onClick={() => addToCart()}>Add</button>
    </>
  );
}

商品信息 SEO + 静态 → RSC；库存实时 + 购物车乐观 → TanStack Query。两层共存。

场景 3 · 聊天应用 100% TanStack Query

tsx 
'use client';
function ChatRoom({ roomId }: { roomId: string }) {
  const queryClient = useQueryClient();

  // 历史消息：infinite query（双向）
  const messages = useInfiniteQuery({
    queryKey: ['messages', roomId],
    queryFn: ({ pageParam }) => api.getMessages(roomId, pageParam),
    initialPageParam: null,
    getNextPageParam: (last) => last.nextCursor,
    getPreviousPageParam: (first) => first.prevCursor,
    maxPages: 20,
  });

  // 发消息：useMutation + 乐观
  const { mutate: send } = useMutation({
    mutationFn: api.sendMessage,
    onMutate: async (text) => {
      // 4 步乐观更新到 infinite cache 末页
    },
  });

  // WebSocket 接收：setQueryData 推进 cache
  useEffect(() => {
    const ws = new WebSocket(`/ws/rooms/${roomId}`);
    ws.onmessage = (event) => {
      const msg = JSON.parse(event.data);
      queryClient.setQueryData<typeof messages.data>(['messages', roomId], (old) => {
        if (!old) return old;
        // 把 msg 追加到末页
        const lastPage = old.pages[old.pages.length - 1];
        return {
          ...old,
          pages: [...old.pages.slice(0, -1), { ...lastPage, items: [...lastPage.items, msg] }],
        };
      });
    };
    return () => ws.close();
  }, [roomId, queryClient]);

  // ... 渲染 messages.data.pages
}

聊天应用 RSC 几乎用不上 —— 全部是实时 + 乐观 + WebSocket。100% TanStack Query。

04 典型代码

代码 1 · RSC vs TanStack Query 同一需求对比

tsx 
// 需求：显示用户列表

// RSC 版（Next.js App Router）
// app/users/page.tsx
export default async function UsersPage() {
  const users = await fetch('/api/users', { next: { revalidate: 60 } }).then(r => r.json());
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
// 优势：零客户端 bundle、SEO 友好、首屏快
// 限制：用户切走再切回需要重新 server-render（除非 router cache）

// TanStack Query 版
'use client';
export default function UsersPage() {
  const { data: users, isPending } = useQuery({
    queryKey: ['users'],
    queryFn: () => fetch('/api/users').then(r => r.json()),
  });
  if (isPending) return <Loading />;
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
// 优势：跨导航 cache、stale-while-revalidate
// 限制：包体大、SEO 差、首屏慢

// 混合版（首屏 + 跨导航）
// app/users/page.tsx (RSC)
export default async function UsersPage() {
  const queryClient = new QueryClient();
  await queryClient.prefetchQuery({
    queryKey: ['users'],
    queryFn: () => fetch('/api/users').then(r => r.json()),
  });
  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <UserList />  {/* Client Component 用 useQuery 命中 cache */}
    </HydrationBoundary>
  );
}
// 详细见 P5.2

05 常见陷阱

  • "RSC 出来了我把所有 useQuery 都换掉" —— 错。RSC 接管首屏，但客户端独有场景仍需要 TanStack Query。强行换会失去 cache / 乐观 / 实时能力
  • "用 use(promise) 替代 useQuery" —— 不是同一类东西。use 是消费 promise 的底层 API，没有 cache / refetch / invalidate / mutation 体系。详见 P5.2
  • 客户端组件里 useQuery 拿首屏数据 —— 反模式。应该 RSC fetch 后通过 dehydrate 桥过来。客户端再 useQuery 是双重请求
  • 用 TanStack Query 拿 SEO 关键页面数据 —— 搜索引擎抓不到。SEO 重的页面必须 server-render（RSC 或传统 SSR）
  • RSC fetch + 'use cache' 用来管实时数据 —— RSC cache 是 server-side，client 拿到就静态了。实时数据走 client TanStack Query
  • 把 Server Action 当 useMutation 替代 —— Server Action 没有客户端 cache 同步能力。复杂场景（多 cache 失效 / 乐观）仍 useMutation
  • "装了 TanStack Query 就不用 RSC" —— 反方向错。两个工具不互斥。新项目都要
  • 跨请求 new QueryClient() —— Server-side prefetch 时每请求必须独立 new。复用会跨用户污染 cache。详见 P5.2
  • "Tanner 说 TanStack Query 不会消失，所以全部场景都用" —— 误读。Tanner 强调的是complementary —— 互补不是覆盖
  • 新项目不评估直接装 TanStack Query —— 不一定需要。marketing 站 / 文档站可能整个项目都不需要客户端 fetch 库
  • SSR 项目（非 RSC，用旧 SSR）也以为不需要 TanStack Query —— 旧 SSR 没有 RSC 的 cache 机制；客户端 fetch 仍然走 useQuery
  • 用 useEffect+fetch 替代 RSC 心智 —— 两次倒退。客户端不需要复杂 cache 时直接 RSC，需要时直接 TanStack Query。useEffect+fetch 没有意义
