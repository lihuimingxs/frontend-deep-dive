hydration 桥 + use(promise) + Server Actions 协作

P5.1 讲了 TanStack Query 与 RSC 的边界，这一章讲具体姿势。dehydrate + HydrationBoundary 是首屏数据从 server 桥到 client cache 的标准模式；use(promise) 与 useSuspenseQuery 关系厘清；mutation 调 Server Action 时 server cache + client cache 怎么双向同步。

阅读时长 ~30 分钟 前置 P5.1 决策矩阵 + 读过 Next.js P4.1 Server Actions + Next.js P5 缓存 主要来源 tanstack.com/query/latest/docs/framework/react/guides/advanced-ssr + nextjs.org/docs 关于 third-party libraries with hydration 的官方说明

01 速览

  1. hydration 桥的本质：server-side prefetch 数据 → 序列化进 RSC payload → client 反序列化进 queryClient.cache。首屏不发请求 + 客户端导航后仍能复用 cache。
  2. 4 个组件协作：(1) Provider 包 QueryClient（client component） / (2) 每请求 new QueryClient（server-side） / (3) prefetchQuery 进 cache（server-side） / (4) HydrationBoundary 接收 dehydrate state（包客户端组件）。
  3. 每请求独立 new QueryClient 是关键：跨请求复用会让用户 A 的 cache 泄漏给用户 B。Next.js 推荐写法是用 React 的 cache() helper 包 getQueryClient。
  4. dehydrate：把 queryClient 当前 cache 序列化成可 JSON 化的对象（含 queryKey + data + dataUpdatedAt + error）。
  5. HydrationBoundary：包在 Client Component 外层；在 client 端把 dehydrate state 反序列化进 queryClient.cache。
  6. use(promise) 与 useSuspenseQuery 的关系：use 是 React 19 底层 API（消费 promise 时 throw promise 给 Suspense）；useSuspenseQuery 在底层用类似机制实现 + 加了 cache lifecycle。两者不冲突但分工不同。
  7. 不要直接用 use(promise) 当 cache library：每次 render 创建新 promise → 每次都 fetch。只有 RSC server-side 创建的 promise 才能被 client use 安全消费（基于 Promise referential equality）。
  8. useMutation 调 Server Action：mutationFn 直接调 'use server' 函数；onSuccess 双向 invalidate（revalidatePath 让 server cache 失效 + queryClient.invalidateQueries 让 client cache 失效）。
  9. 两层 cache 同步是 RSC 时代新挑战：server-side cache（'use cache' / Data Cache）+ client-side cache（queryClient）独立存在；某个数据变了要确保两层都失效。
  10. 典型陷阱：忘了每请求 new QueryClient；prefetch 但没 dehydrate；HydrationBoundary 没正确包；client 拿到 RSC 创建的 promise 后用 use 但 RSC 又重 render → promise 引用变。

02 底层逻辑

桥接的目标

RSC + TanStack Query 共存想达成的体验：

  1. 用户进 page A → server-side fetch users（直接 server-render HTML，零 client loading）
  2. HTML 里同时嵌入 dehydrate state（users 数据序列化）
  3. client hydrate 时 → 把 dehydrate state 写入 queryClient.cache
  4. page A 上的 Client Component 用 useQuery(['users']) → 直接命中 cache，不再发请求
  5. 用户切到 page B → users cache 在 inactive 状态保留
  6. 用户切回 page A → cache 命中，立刻显示（stale-while-revalidate 后台刷）

对比"纯 RSC"路径：每次进 page A 都 server-render 一次，无客户端 cache。

对比"纯 TanStack Query"路径：第一次进有 client loading；server-side render 内容缺失 SEO 差。

4 个组件协作

组件 1 · QueryClientProvider（Client Component）

tsx 
// app/Providers.tsx
'use client';
import { QueryClient, QueryClientProvider, isServer } from '@tanstack/react-query';
import { useState } from 'react';

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        // 关键：staleTime > 0 防止 hydrate 后立刻 refetch
        staleTime: 60 * 1000,
      },
    },
  });
}

let browserQueryClient: QueryClient | undefined;

function getQueryClient() {
  if (isServer) {
    // server: 永远 new（每次 SSR / RSC 都是新的）
    return makeQueryClient();
  }
  // browser: 单例
  if (!browserQueryClient) browserQueryClient = makeQueryClient();
  return browserQueryClient;
}

export function Providers({ children }: { children: React.ReactNode }) {
  // useState 包确保每次 render 不重新调 getQueryClient
  const queryClient = getQueryClient();
  return <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>;
}

关键细节：

  • browser 是单例（一个 tab 一个 QueryClient）
  • server 每次 new（避免跨请求污染）
  • 用 isServer 区分（来自 @tanstack/react-query）
  • browserQueryClient 用 module-level 变量，不放 useState 里（不需要响应组件 lifecycle）

组件 2 · server-side prefetch + dehydrate

tsx 
// app/users/page.tsx (Server Component)
import { dehydrate, HydrationBoundary, QueryClient } from '@tanstack/react-query';
import UserList from './UserList';  // Client Component

export default async function UsersPage() {
  // 每请求独立 new
  const queryClient = new QueryClient();

  // server-side prefetch
  await queryClient.prefetchQuery({
    queryKey: ['users'],
    queryFn: () => fetch('http://api.example.com/users').then(r => r.json()),
  });

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <UserList />
    </HydrationBoundary>
  );
}

组件 3 · Client Component 读 cache

tsx 
// app/users/UserList.tsx
'use client';
import { useQuery } from '@tanstack/react-query';

export default function UserList() {
  const { data: users } = useQuery({
    queryKey: ['users'],
    queryFn: () => fetch('/api/users').then(r => r.json()),
    // queryFn 必须传，因为 cache 失效后 client 仍要能 refetch
    // 但首次 render 命中 server prefetch 的 cache，不会调 queryFn
  });
  return <ul>{users?.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}

整个流程：

  1. server: new QueryClient + prefetch users + dehydrate → 序列化进 React payload
  2. HydrationBoundary 在 server 端 render 一段含 dehydrate state 的标记
  3. client: hydrate 时读到 dehydrate state → 写入 browser 的 queryClient
  4. UserList 的 useQuery → cache 命中 → 直接 return data
  5. 切走再切回 → cache 仍在（stale-while-revalidate）

每请求 new QueryClient · 跨请求隔离

错误写法：

tsx 
// ❌ 全局 new → 跨请求复用 → 用户 A 的 cache 给用户 B 看到
const queryClient = new QueryClient();

export default async function UsersPage() {
  await queryClient.prefetchQuery({...});
  return ...;
}

这是严重 bug。Server Component 在 server 长期运行（同一 process 处理多请求），全局变量会被并发请求共享。结果：

  • 请求 1（用户 A）prefetch user-a 数据进 cache
  • 请求 2（用户 B）来到同 page → cache 命中 → 看到用户 A 的数据
  • 泄漏！

正确写法：每次 new。性能担心？new 一个 QueryClient 是廉价操作（只是几个 Map 初始化），不必担心。

更好的写法（用 React 的 cache helper 在同请求内复用）：

tsx 
import { cache } from 'react';
import { QueryClient } from '@tanstack/react-query';

// React cache 是 per-request 缓存（同请求内多次调用复用，跨请求新建）
export const getQueryClient = cache(() => new QueryClient());

// 同一请求多个 Server Component 调 getQueryClient 拿到同一个
// 不同请求各自独立

dehydrate 序列化什么

dehydrate(queryClient) 返回一个对象：

javascript 
{
  mutations: [],  // 通常 server 不 prefetch mutation
  queries: [
    {
      queryKey: ['users'],
      queryHash: '["users"]',
      state: {
        data: [{ id: 1, name: 'A' }, ...],
        dataUpdatedAt: 1731972000000,
        error: null,
        status: 'success',
        // ...
      },
    },
    // 所有 prefetched query
  ],
}

这个对象会被 React 序列化进 RSC payload（JSON.stringify），然后 client 端反序列化。所以：

  • data 必须可 JSON 化（无函数 / Symbol / class 实例）
  • error 也要可序列化（Error 对象的 message / name 会保留，stack trace 通常丢）
  • 大 cache 会让 RSC payload 膨胀，影响首屏 TTFB

HydrationBoundary 的工作

tsx 
// HydrationBoundary 简化实现
'use client';
function HydrationBoundary({ state, children }: { state: DehydratedState; children: React.ReactNode }) {
  const queryClient = useQueryClient();
  // hydrate 把 state 写进 client queryClient
  hydrate(queryClient, state);
  return <>{children}</>;
}

之后子组件的 useQuery 命中 cache。

HydrationBoundary 可以嵌套：父级 prefetch 一些通用数据，子级再 prefetch 自己的，client 端各自 hydrate。

use(promise) 与 useSuspenseQuery 的关系

React 19 引入 use(promise) API：

tsx 
import { use } from 'react';

function User({ promise }: { promise: Promise<User> }) {
  // pending 时 throw promise（Suspense 接住）
  // resolved 后返回 value
  const user = use(promise);
  return <div>{user.name}</div>;
}

// 父级
<Suspense fallback={<Loading />}>
  <User promise={fetchUserPromise} />
</Suspense>

看起来类似 useSuspenseQuery，但关键差异：

  • use 没有 cache：每次 render 用同一 promise 引用 → 复用结果（基于 Promise referential equality）；引用变了 → 重新 fetch
  • useSuspenseQuery 有完整 query lifecycle：cache + invalidate + refetch + lifecycle hook
  • use 可以在条件 / 循环里用（不像 hook）
  • useSuspenseQuery 限制更多（不能 enabled: false / placeholderData）

RSC 创建 promise + Client use 消费的标准模式

这是 React 19 + RSC 的核心模式：

tsx 
// app/posts/page.tsx (Server Component)
import { Suspense } from 'react';
import ClientPosts from './ClientPosts';

export default function PostsPage() {
  // 启动 fetch，promise 通过 props 传给 Client Component
  const postsPromise = fetch('/api/posts').then(r => r.json());
  return (
    <Suspense fallback={<Loading />}>
      <ClientPosts postsPromise={postsPromise} />
    </Suspense>
  );
}

// app/posts/ClientPosts.tsx
'use client';
import { use } from 'react';

export default function ClientPosts({ postsPromise }: { postsPromise: Promise<Post[]> }) {
  const posts = use(postsPromise);
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}

这个模式适合简单场景：server 启动 fetch + client 流式消费 + 不需要 cache / refetch / invalidate。

不适合：需要 cache / refetch / mutation 的场景。那些用 useSuspenseQuery 配 hydration 桥。

不要直接用 use(promise) 当 cache library

常见错误：

tsx 
// ❌ 反模式：每次 render 创建新 promise
'use client';
function User({ id }: { id: string }) {
  const promise = fetch(`/api/users/${id}`).then(r => r.json());  // 每次 render 新 promise
  const user = use(promise);  // 每次都重新 fetch
  return <div>{user.name}</div>;
}

修复方案：

  • 方案 A：promise 由 Server Component 创建（如上面"标准模式"）
  • 方案 B：用 useSuspenseQuery（自带 cache）
  • 方案 C：把 promise 缓存到 useState 或 useRef（罕见）

useMutation + Server Action 协作

典型场景：客户端组件想用乐观更新，但 server 数据变更要走 Server Action（比 Route Handler 更顺）：

tsx 
// app/_actions/todos.ts
'use server';
import { revalidatePath, revalidateTag } from 'next/cache';

export async function addTodoAction(text: string) {
  await db.todos.create({ data: { text } });
  revalidateTag('todos');  // 让 server cache（'use cache' / next: { tags }）失效
  return { success: true };
}

// app/_components/AddTodo.tsx
'use client';
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { addTodoAction } from '@/app/_actions/todos';

export function AddTodo() {
  const queryClient = useQueryClient();
  const { mutate, isPending } = useMutation({
    mutationFn: addTodoAction,  // 直接调 Server Action
    onMutate: async (text) => {
      // 4 步乐观更新 client cache
      await queryClient.cancelQueries({ queryKey: ['todos'] });
      const prev = queryClient.getQueryData(['todos']);
      queryClient.setQueryData(['todos'], (old: any[] = []) => [...old, { id: 'temp', text, done: false }]);
      return { prev };
    },
    onError: (_e, _v, ctx) => ctx && queryClient.setQueryData(['todos'], ctx.prev),
    onSettled: () => {
      // 双向 invalidate：server cache 由 Server Action 内的 revalidateTag 处理
      // client cache 由这里 invalidate
      queryClient.invalidateQueries({ queryKey: ['todos'] });
    },
  });
  // ...
}

双向同步：

  • server cache 失效：Server Action 内 revalidatePath / revalidateTag / updateTag
  • client cache 失效：useMutation 的 onSettled 里 queryClient.invalidateQueries

两者都要做。少一个会导致：

  • 只 server invalidate：client cache 仍是旧；用户切走再回来才看到新
  • 只 client invalidate：client refetch 拿到的还是 server 旧 cache

两层 cache 同步的细节

实战中两层 cache 同步是 RSC 时代新挑战。常见复杂场景：

场景 A · 第三方 push 通知：WebSocket 推送数据 → client setQueryData。但 server cache 还是旧的，下次 SSR 仍渲染旧数据。

  • 解决：用 router.refresh() 让 RSC 重新跑（间接刷新 server cache）

场景 B · client mutation 后 navigation：mutation 改了 todos → invalidate client cache → 用户跳到新页面 → 这页的 server-render 用了旧 server cache。

  • 解决：mutation 内 revalidatePath 或 revalidateTag；client onSuccess 后 router.refresh()

详细 server-side cache 机制见 Next.js P5 缓存系统。

场景 4 · 完整 mutation + Server Action 双向同步

tsx 
// app/_actions/todos.ts
'use server';
import { revalidateTag } from 'next/cache';
import { auth } from '@/lib/auth';

export async function addTodo(text: string) {
  const session = await auth();
  if (!session) throw new Error('Unauthorized');
  await db.todos.create({ data: { text, userId: session.userId } });
  revalidateTag(`todos-${session.userId}`);  // server cache 失效
}

export async function deleteTodo(id: string) {
  const session = await auth();
  if (!session) throw new Error('Unauthorized');
  await db.todos.delete({ where: { id, userId: session.userId } });
  revalidateTag(`todos-${session.userId}`);
}

// app/todos/page.tsx (RSC)
export default async function TodosPage() {
  const session = await auth();
  const queryClient = getQueryClient();
  await queryClient.prefetchQuery({
    queryKey: ['todos', session.userId],
    queryFn: () => fetch(`/api/todos?userId=${session.userId}`, {
      next: { tags: [`todos-${session.userId}`] }  // 配合 Server Action 的 revalidateTag
    }).then(r => r.json()),
  });
  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <TodoListClient userId={session.userId} />
    </HydrationBoundary>
  );
}

// app/todos/TodoListClient.tsx
'use client';
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import { addTodo, deleteTodo } from '@/app/_actions/todos';

export default function TodoListClient({ userId }: { userId: string }) {
  const queryClient = useQueryClient();
  const { data: todos = [] } = useQuery({
    queryKey: ['todos', userId],
    queryFn: () => fetch(`/api/todos?userId=${userId}`).then(r => r.json()),
  });

  const addMutation = useMutation({
    mutationFn: addTodo,
    onMutate: async (text) => {
      await queryClient.cancelQueries({ queryKey: ['todos', userId] });
      const prev = queryClient.getQueryData(['todos', userId]);
      queryClient.setQueryData(['todos', userId], (old: any[] = []) => [
        ...old, { id: `temp-${Date.now()}`, text, done: false }
      ]);
      return { prev };
    },
    onError: (_e, _v, ctx) => ctx && queryClient.setQueryData(['todos', userId], ctx.prev),
    onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos', userId] }),
    // 注意：Server Action 内已 revalidateTag → server cache 失效
    // 这里 invalidateQueries → client cache 失效
    // 双向同步完成
  });

  const deleteMutation = useMutation({
    mutationFn: deleteTodo,
    onMutate: async (id) => {
      await queryClient.cancelQueries({ queryKey: ['todos', userId] });
      const prev = queryClient.getQueryData(['todos', userId]);
      queryClient.setQueryData(['todos', userId], (old: any[] = []) => old.filter(t => t.id !== id));
      return { prev };
    },
    onError: (_e, _v, ctx) => ctx && queryClient.setQueryData(['todos', userId], ctx.prev),
    onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos', userId] }),
  });

  // ...
}

04 典型代码

代码 1 · Providers 完整模板

tsx 
// app/Providers.tsx
'use client';
import { QueryClient, QueryClientProvider, isServer } from '@tanstack/react-query';
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 60 * 1000,
        retry: (failureCount, error: any) => {
          if (error?.status >= 400 && error?.status < 500) return false;
          return failureCount < 3;
        },
      },
    },
  });
}

let browserQueryClient: QueryClient | undefined;

function getQueryClient() {
  if (isServer) return makeQueryClient();
  if (!browserQueryClient) browserQueryClient = makeQueryClient();
  return browserQueryClient;
}

export function Providers({ children }: { children: React.ReactNode }) {
  const queryClient = getQueryClient();
  return (
    <QueryClientProvider client={queryClient}>
      {children}
      {!isServer && <ReactQueryDevtools initialIsOpen={false} />}
    </QueryClientProvider>
  );
}

// app/queryClient.ts (server-side helper)
import 'server-only';
import { cache } from 'react';
import { QueryClient } from '@tanstack/react-query';

export const getServerQueryClient = cache(() => new QueryClient({
  defaultOptions: { queries: { staleTime: 60 * 1000 } },
}));

  const slowDataPromise = fetch('/api/slow').then(r => r.json());

  );
}

}

代码 3 · LLM 流式 + TanStack Query

tsx 
// app/_actions/chat.ts (Server Action)
'use server';
export async function* streamChat(prompt: string) {
  // server-side stream from OpenAI / Claude
  const res = await fetch('https://api.example.com/chat', {
    method: 'POST',
    body: JSON.stringify({ prompt }),
  });
  const reader = res.body!.getReader();
  const decoder = new TextDecoder();
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    yield decoder.decode(value);
  }
}

// app/_components/Chat.tsx
'use client';
import { useQuery, streamedQuery } from '@tanstack/react-query';
import { streamChat } from '@/app/_actions/chat';

export function Chat({ prompt }: { prompt: string }) {
  const { data: chunks = [], isFetching } = useQuery({
    queryKey: ['chat', prompt],
    queryFn: streamedQuery({ streamFn: () => streamChat(prompt) }),
    enabled: !!prompt,
  });
  return <div>{chunks.join('')}{isFetching && '▍'}</div>;
}

05 常见陷阱

  • 跨请求复用 QueryClient —— 全局 new 一个 → 用户 A 的 cache 给用户 B 看到。严重 bug + 隐私问题。每请求 new + 用 React cache helper
  • Server Component 里直接用 useQuery —— Server Component 不支持 hooks。要么用 server-side prefetch + dehydrate，要么改 Client Component
  • prefetch 但没 dehydrate / HydrationBoundary —— prefetch 进了 server 端 queryClient cache，但没桥到 client → client 仍要重发请求
  • HydrationBoundary 包错位置 —— 必须包在用 useQuery 的 Client Component 外。包错（如包整 layout）反而行为异常
  • server prefetch 的 query 没传 queryFn —— prefetchQuery 必须传 queryFn 才能拿到数据。client 端的 useQuery 也必须传（否则 cache 失效后 client 不能 refetch）
  • staleTime 设 0 让 hydrate 后立刻 refetch —— 默认 staleTime 0 → cache hydrate 进来立刻 stale → client mount 立刻 refetch。预期行为但可能不希望。设 staleTime > 0
  • 每次 render 创建 promise 给 use 消费 —— 每次 render 新 promise → 每次都 fetch。promise 应该由 RSC 创建（Server Component 内）
  • use 在 Server Component 里用 —— use 是 Client API（也能在 RSC 用某些场景但限制多）。一般用法是 Client Component 内 use server 创建的 promise
  • Server Action 内忘了 revalidate —— server cache 不失效；下次 SSR 仍渲染旧数据；client cache 失效但 SSR 又给旧的
  • useMutation 调 Server Action 后只 client invalidate —— 如果 Server Action 内没 revalidate，server cache 未失效。两边都要做
  • WebSocket setQueryData 后没 router.refresh —— client cache 更新但 server cache 仍旧；用户切 page 再回（走 server-render）会拿旧数据
  • dehydrate 数据太大 —— 大 cache 进 RSC payload 影响 TTFB。只 prefetch 必要的数据；不要 prefetch 整个 dashboard 所有 query
  • Server Action 抛错没处理 —— useMutation 拿到的 error 是 Server Action 抛的；注意业务错误用 return（不 throw）让 useMutation 走 onSuccess
  • ensureQueryData 而不是 prefetchQuery —— ensureQueryData 在 server 也能用，命中 cache 不发 → 适合预热但请求间不复用（每请求 new client cache）
  • 把 useSuspenseQuery 换成 use(promise) —— 失去 cache + lifecycle。详见 P5.1
