深度学习/ TanStack Query/ P4 · 高级模式/ 02 · Suspense + prefetch + streaming

Suspense mode + queryClient direct API + prefetch + streaming

v5 把 useSuspenseQuery 三件套提到一等公民。这一章讲清 useQuery 与 useSuspenseQuery 的选择、queryClient direct API（不依赖 hook 调用 query）、hover prefetch / router loader 集成、streamedQuery（LLM 流式）。这些是 P5 RSC 共存章的前置。

阅读时长 ~25 分钟 前置 P3 useQuery 全谱 + 了解 React 18+ Suspense / ErrorBoundary 主要来源 tanstack.com/query/latest/docs/framework/react/guides/suspense + prefetching + advanced-ssr

01 速览

  1. v5 useSuspenseQuery 三件套（一等公民）：useSuspenseQuery / useSuspenseInfiniteQuery / useSuspenseQueries。永远 throw promise（pending）/ throw error；data 永远不是 undefined。
  2. useQuery vs useSuspenseQuery：前者组件自己处理 loading/error；后者把控制权交给上层 Suspense + ErrorBoundary。Suspense 模式更声明式但限制更多。
  3. useSuspenseQuery 不能用 enabled: false：Suspense 没办法 throw 一个不存在的 promise。需要条件触发的 query 仍用 useQuery。
  4. queryClient 不一定要被 hook 调用：prefetchQuery / fetchQuery / ensureQueryData 是 imperative API，可以从 router loader / event handler / WebSocket 调。
  5. 3 个 imperative fetch API 区别：prefetchQuery（不返回 data，进 cache）/ fetchQuery（返回 promise，进 cache 失败 throw）/ ensureQueryData（cache 有就返回，没有就 fetch）。
  6. hover prefetch：用户 hover 链接时调 prefetchQuery 预热下一页 → 点进去瞬间显示。TanStack Query 比 SWR 强项之一。
  7. router loader 集成（React Router v7 / TanStack Router）：在 loader 里 ensureQueryData，组件用 useSuspenseQuery 直接读 cache。是"路由数据驱动"模式。
  8. streamedQuery（v5 新）：支持 SSE / chunked 流式响应。边收边累加进 cache。和 Vercel AI SDK useChat 哲学接近。
  9. RSC 时代 prefetch / dehydrate 是桥接关键：server-side prefetch → 序列化进 RSC payload → client hydrate。详见 P5.2。
  10. Suspense 边界要包合理：太粗（整页一个 Suspense）失去并发渲染好处；太细（每个 query 一个）UX 闪烁。一般按"内容区块"分段。

02 底层逻辑

Suspense 模式 · 把 loading/error 上提

传统 useQuery 让组件自己处理三态：

tsx 
function User({ id }: { id: string }) {
  const { data, isPending, isError } = useQuery({...});
  if (isPending) return <Loading />;
  if (isError) return <Error />;
  return <div>{data.name}</div>;
}

useSuspenseQuery 把控制权交给上层：

tsx 
function User({ id }: { id: string }) {
  const { data } = useSuspenseQuery({
    queryKey: ['user', id],
    queryFn: () => api.getUser(id),
  });
  // 这里 data 类型是 User（不是 User | undefined）
  // pending 时会 throw promise → 上层 Suspense 接住
  // error 时会 throw error → 上层 ErrorBoundary 接住
  return <div>{data.name}</div>;
}

function App() {
  return (
    <ErrorBoundary fallback={<Error />}>
      <Suspense fallback={<Loading />}>
        <User id="1" />
      </Suspense>
    </ErrorBoundary>
  );
}

对比：

维度 useQuery useSuspenseQuery
data 类型 T | undefined T（已 narrow）
loading 处理 组件内 isPending 父级 Suspense fallback
error 处理 组件内 isError 父级 ErrorBoundary
enabled: false 支持 不支持
placeholderData 支持 不支持
条件触发 query 方便（enabled） 不方便
声明式程度 低 高

选择：

  • 组件需要细粒度控制 loading（骨架样式 / 部分加载）→ useQuery
  • 整页或区块统一 loading + error → useSuspenseQuery
  • 条件 query / 用户操作触发 → useQuery（用 enabled）
  • 路由级别 query（页面打开就要） → useSuspenseQuery（配 router loader 更顺）

Suspense 模式的限制

限制 1 · 不能用 enabled: false

tsx 
// ❌ 这不工作
useSuspenseQuery({
  queryKey: ['user', userId],
  queryFn: ...,
  enabled: !!userId,  // 不允许
});

// ✅ 父级判断
function UserContainer() {
  if (!userId) return <Empty />;
  return (
    <Suspense fallback={<Loading />}>
      <User id={userId} />
    </Suspense>
  );
}

限制 2 · 不能用 placeholderData

Suspense 模式下没有 isPlaceholderData 概念。要"切页不闪烁"用 useQuery + keepPreviousData。

限制 3 · waterfall 风险

多个 useSuspenseQuery 在同组件可能串行（一个 throw promise 后下一个还没开始）。修复：用 useSuspenseQueries 并发；或用 prefetch 先并发触发。

tsx 
// ❌ 可能 waterfall
function Dashboard() {
  const { data: user } = useSuspenseQuery({...});
  const { data: posts } = useSuspenseQuery({...});  // user 没拿到这里不会跑
  return ...;
}

// ✅ 并发
function Dashboard() {
  const [user, posts] = useSuspenseQueries({
    queries: [
      { queryKey: ['user'], queryFn: ... },
      { queryKey: ['posts'], queryFn: ... },
    ],
  });
  return ...;
}

queryClient direct API · 3 个 imperative fetch

不是所有场景都要从 hook 触发 query。queryClient 提供 imperative API：

API 返回 cache 行为 失败行为 典型场景
prefetchQuery Promise<void> 进 cache 静默吞错 预热（hover / 路由 idle）
fetchQuery Promise<TData> 进 cache throw error router loader 拿数据
ensureQueryData Promise<TData> 有 cache 直接返回，否则 fetch + 进 cache throw error 组件 mount 前确保数据在

区别：

  • prefetchQuery：fire-and-forget，不需要 await 结果。失败也不影响主流程
  • fetchQuery：要拿到结果用，比如 router loader
  • ensureQueryData：cache 友好版本的 fetchQuery，避免重复 fetch

hover prefetch · 标杆模式

tsx 
function UserListItem({ user }: { user: User }) {
  const queryClient = useQueryClient();

  function prefetch() {
    queryClient.prefetchQuery({
      queryKey: ['user', user.id],
      queryFn: () => api.getUser(user.id),
      staleTime: 30 * 1000,  // prefetch 的数据 30 秒内 fresh
    });
  }

  return (
    <Link
      href={`/users/${user.id}`}
      onMouseEnter={prefetch}
      onFocus={prefetch}
    >
      {user.name}
    </Link>
  );
}

用户 hover 100ms+ 才点击 → 点击时数据已 fetch 完 → 详情页瞬间显示。这是 TanStack Query 比 SWR 杀手锏的功能之一（SWR 没结构化 prefetch API）。

router loader 集成 · TanStack Router / React Router v7

TanStack Router 自带 loader 概念，配合 ensureQueryData：

tsx 
// router 路由定义
const userRoute = createRoute({
  path: '/users/$id',
  loader: ({ params }) => queryClient.ensureQueryData({
    queryKey: ['user', params.id],
    queryFn: () => api.getUser(params.id),
  }),
  component: UserPage,
});

// 组件直接 useSuspenseQuery 读 cache（已经在了）
function UserPage() {
  const { id } = userRoute.useParams();
  const { data } = useSuspenseQuery({
    queryKey: ['user', id],
    queryFn: () => api.getUser(id),
  });
  return <div>{data.name}</div>;
}

好处：

  • 路由切换前数据已经在 fetch（不等组件 mount）
  • 组件用 useSuspenseQuery 时 cache 已经命中（无 loading）
  • 错误在 loader 阶段就抛出（不进 component），路由层处理

React Router v7 也支持类似模式（loaders）。详见各路由库文档。

streamedQuery · v5 流式响应支持

v5 新增 streamedQuery 实验性 helper，支持 SSE / chunked 流：

tsx 
import { streamedQuery, useQuery } from '@tanstack/react-query';

// streamFn 返回一个 AsyncIterable
async function* fetchChatStream(prompt: string) {
  const res = await fetch('/api/chat', {
    method: 'POST',
    body: JSON.stringify({ prompt }),
  });
  const reader = res.body!.getReader();
  const decoder = new TextDecoder();
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    yield decoder.decode(value);
  }
}

function ChatBox() {
  const [prompt, setPrompt] = useState('');
  const { data, isFetching } = useQuery({
    queryKey: ['chat', prompt],
    queryFn: streamedQuery({
      streamFn: ({ pageParam }) => fetchChatStream(prompt),
    }),
    enabled: !!prompt,
  });
  // data 是数组 —— 每个 chunk 是一个元素
  const fullText = data?.join('') ?? '';
  return <div>{fullText}{isFetching && '▍'}</div>;
}

原理：streamFn 是 async generator，每 yield 一次 TanStack Query 把 chunk 累加进 cache。组件订阅 cache → 每次 chunk 来都 re-render。

这个模式和 Vercel AI SDK 的 useChat 哲学接近 —— LLM 流式响应在 client cache 里增量更新。详见待写 AI SDK 主题（双向链）。

Suspense 边界放在哪

Suspense 边界粒度影响 UX：

  • 整页一个 Suspense：等所有 query 完成才显示。简单但慢
  • 每 query 一个 Suspense：闪烁严重
  • 按内容区块分段（推荐）：navigation / main / sidebar 各一个 Suspense；快的先显示，慢的等
tsx 
function Dashboard() {
  return (
    <Layout>
      <Suspense fallback={<NavSkeleton />}><Nav /></Suspense>
      <Suspense fallback={<MainSkeleton />}><MainContent /></Suspense>
      <Suspense fallback={<SidebarSkeleton />}><Sidebar /></Suspense>
    </Layout>
  );
}
// Nav / MainContent / Sidebar 各自 useSuspenseQuery
// 三段独立流式显示，快的先出

这与 Next.js App Router 的 loading.tsx + 嵌套 Suspense 心智一致。详见 Next.js P3 RSC 与流式。

03 应用场景

场景 1 · 完整 useSuspenseQuery dashboard

tsx 
function App() {
  return (
    <ErrorBoundary FallbackComponent={ErrorPage}>
      <Suspense fallback={<FullPageLoading />}>
        <Dashboard />
      </Suspense>
    </ErrorBoundary>
  );
}

function Dashboard() {
  // 整体并发拉
  const [user, stats, notifications] = useSuspenseQueries({
    queries: [
      { queryKey: ['current-user'], queryFn: api.getCurrentUser },
      { queryKey: ['stats'], queryFn: api.getStats },
      { queryKey: ['notifications'], queryFn: api.getNotifications },
    ],
  });

  return (
    <Layout user={user.data}>
      <StatsCard stats={stats.data} />
      <NotificationList items={notifications.data} />
    </Layout>
  );
}

场景 2 · hover prefetch 链接

tsx 
// 自定义 hook 封装
function usePrefetchUser() {
  const queryClient = useQueryClient();
  return useCallback((id: string) => {
    queryClient.prefetchQuery({
      queryKey: ['user', id],
      queryFn: () => api.getUser(id),
      staleTime: 30 * 1000,
    });
  }, [queryClient]);
}

// 使用
function UserListItem({ user }: { user: User }) {
  const prefetch = usePrefetchUser();
  return (
    <Link
      href={`/users/${user.id}`}
      onMouseEnter={() => prefetch(user.id)}
      onFocus={() => prefetch(user.id)}
    >
      {user.name}
    </Link>
  );
}

场景 3 · LLM 流式响应（streamedQuery）

tsx 
// 使用 OpenAI 风格的 SSE
async function* streamChat(messages: Message[]): AsyncIterable<string> {
  const res = await fetch('/api/chat', {
    method: 'POST',
    body: JSON.stringify({ messages }),
    headers: { 'Content-Type': 'application/json' },
  });
  if (!res.ok || !res.body) throw new Error('stream failed');
  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  let buffer = '';
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    // 简化：按 SSE 'data: ...\n\n' 分块
    const lines = buffer.split('\n\n');
    buffer = lines.pop() ?? '';
    for (const line of lines) {
      if (line.startsWith('data: ')) {
        const json = line.slice(6);
        if (json === '[DONE]') return;
        const parsed = JSON.parse(json);
        const content = parsed.choices?.[0]?.delta?.content;
        if (content) yield content;
      }
    }
  }
}

function Chat({ messages }: { messages: Message[] }) {
  const { data: chunks = [], isFetching } = useQuery({
    queryKey: ['chat', messages],
    queryFn: streamedQuery({
      streamFn: () => streamChat(messages),
    }),
    enabled: messages.length > 0,
  });
  const text = chunks.join('');
  return (
    <div>
      <ReactMarkdown>{text}</ReactMarkdown>
      {isFetching && <span className="cursor">▍</span>}
    </div>
  );
}

场景 4 · TanStack Router loader 集成

tsx 
import { createRoute } from '@tanstack/react-router';

// 路由定义带 loader
export const userRoute = createRoute({
  getParentRoute: () => rootRoute,
  path: '/users/$id',
  loader: async ({ params, context }) => {
    // context.queryClient 是 router 上下文里的 queryClient
    return context.queryClient.ensureQueryData({
      queryKey: ['user', params.id],
      queryFn: () => api.getUser(params.id),
    });
  },
  component: UserPage,
  errorComponent: UserError,
  pendingComponent: UserLoading,
});

function UserPage() {
  const { id } = userRoute.useParams();
  // 数据保证已经在 cache（loader 已 ensure）
  const { data } = useSuspenseQuery({
    queryKey: ['user', id],
    queryFn: () => api.getUser(id),
  });
  return <div>{data.name}</div>;
}

04 典型代码

代码 1 · 全局 ErrorBoundary + Suspense 组合

tsx 
import { ErrorBoundary } from 'react-error-boundary';
import { Suspense } from 'react';

function QueryErrorBoundary({ children, fallback }: { children: React.ReactNode; fallback: React.ReactNode }) {
  // 包装 ErrorBoundary 让 query 错误能 reset 重试
  const queryClient = useQueryClient();
  return (
    <ErrorBoundary
      fallbackRender={({ error, resetErrorBoundary }) => (
        <div>
          <p>Error: {error.message}</p>
          <button onClick={() => {
            queryClient.resetQueries();  // 重置错误的 query
            resetErrorBoundary();
          }}>Retry</button>
        </div>
      )}
    >
      <Suspense fallback={fallback}>{children}</Suspense>
    </ErrorBoundary>
  );
}

// 使用
<QueryErrorBoundary fallback={<Loading />}>
  <UserDashboard />
</QueryErrorBoundary>

代码 2 · 用 throwOnError 控制 useQuery 也走 Suspense 风格

tsx 
// useQuery 也可以让错误冒泡到 ErrorBoundary（不用 useSuspenseQuery）
const { data, isPending } = useQuery({
  queryKey: ['user', id],
  queryFn: ...,
  throwOnError: true,  // error 时 throw 让 ErrorBoundary 接住
  // throwOnError: (error) => error.status >= 500,  // 只 throw 5xx
});

代码 3 · 主动 reset query（错误后重试）

tsx 
function ErrorFallback({ error, resetErrorBoundary }: any) {
  const queryClient = useQueryClient();
  return (
    <>
      <p>Failed: {error.message}</p>
      <button onClick={() => {
        // 重置所有失败 query 的状态
        queryClient.resetQueries();
        resetErrorBoundary();
      }}>Retry All</button>
      <button onClick={() => {
        // 只重置某个 key
        queryClient.resetQueries({ queryKey: ['users'] });
        resetErrorBoundary();
      }}>Retry Users</button>
    </>
  );
}

代码 4 · prefetch 进 staleTime 控制

tsx 
// 默认 prefetch 进 cache 但 staleTime 是 0 → 立刻 stale → 用户点进去时再 refetch
queryClient.prefetchQuery({ queryKey: ['user', id], queryFn: ... });

// 推荐：prefetch 时给个 staleTime 让 cache 一段时间内 fresh
queryClient.prefetchQuery({
  queryKey: ['user', id],
  queryFn: ...,
  staleTime: 30 * 1000,  // prefetch 数据 30 秒内 fresh，期间不 refetch
});

// 或全局调 staleTime 默认（推荐做法）
new QueryClient({
  defaultOptions: { queries: { staleTime: 60 * 1000 } },
});

05 常见陷阱

  • useSuspenseQuery 没包 Suspense —— pending 时 throw promise 没人接住，整个 React tree 崩溃。任何 useSuspenseQuery 都必须有父级 Suspense
  • useSuspenseQuery 用 enabled: false —— 不允许。条件 query 用 useQuery + enabled
  • 多个 useSuspenseQuery 串联触发 waterfall —— 第一个 throw promise 后第二个还没开始。改用 useSuspenseQueries 或 prefetch
  • prefetchQuery 没传 staleTime —— prefetch 数据立刻 stale，用户点进去仍然 refetch，prefetch 失去价值
  • 把 prefetchQuery 当 fetchQuery 用 await 它的 data —— prefetchQuery 返回 Promise<void>，没有 data。要用 fetchQuery / ensureQueryData
  • Suspense 边界太粗（整页一个） —— 等所有 query 完才显示，慢且影响 LCP。按内容块分段
  • Suspense 边界太细（每个 query 一个） —— 闪烁严重；并发渲染好处用不上
  • ErrorBoundary 没 reset 机制 —— 错误后用户卡住没法重试。结合 queryClient.resetQueries + resetErrorBoundary
  • throwOnError 全开 —— 业务错误（404 not found）也走 ErrorBoundary，UX 差。throwOnError 写成函数只 throw 致命错误
  • streamedQuery 没处理 abort —— 用户离开页面 stream 还在跑。streamFn 接 signal 透传给 fetch
  • router loader 用 fetchQuery 而不是 ensureQueryData —— fetchQuery 每次都发请求；ensureQueryData 命中 cache 不发。常驻路由用 ensureQueryData 更高效
  • 把 Suspense 当 loading 替代 —— Suspense 是声明式，不能动态控制（"再等 1 秒"）。要细粒度控制仍用 useQuery isPending
  • Suspense 嵌套层级过深 —— 6 层嵌套 Suspense + ErrorBoundary 很难 debug。保持 2-3 层够用
  • useSuspenseQueries 的 combine 触发 throw 时机 —— combine 接收的 results 仍可能含 isLoading；要确保所有都 success 才用 data
