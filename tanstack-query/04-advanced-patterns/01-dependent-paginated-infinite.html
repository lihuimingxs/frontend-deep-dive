dependent / parallel / paginated / infinite queries

单个 useQuery 解决不了真实业务的所有形状。这一章把 4 种结构化查询写法讲清：dependent（一个 query 等另一个）、parallel（多 query 并发）、paginated（页码型，keepPreviousData 不闪烁）、infinite（"加载更多"型，useInfiniteQuery + IntersectionObserver）。

阅读时长 ~28 分钟 前置 P3 useQuery 全谱 + useMutation 4 步乐观 主要来源 tanstack.com/query/latest/docs/framework/react/guides/dependent-queries + parallel-queries + paginated-queries + infinite-queries

01 速览

  1. dependent query：query B 等 query A 的结果。用 enabled: !!previousData 控制；不要用 if 包 useQuery（违反 hooks 规则）。
  2. parallel query 静态数量：同组件多次 useQuery 自动并发。React 自然行为，不用特殊 API。
  3. parallel query 动态数量：用 useQueries({ queries: [...] })；返回数组，可 map 渲染。典型场景"批量加载头像"。
  4. paginated query 页码型：cursor 在 queryKey（['posts', { page }]）+ placeholderData: keepPreviousData 切页时显示上一页 + 后台拉新页。
  5. infinite query "加载更多"型：用专门的 useInfiniteQuery hook（不是 useQuery）；data 结构是 { pages: [], pageParams: [] }；getNextPageParam 决定下一页 param。
  6. infinite + IntersectionObserver：滚到底自动 fetchNextPage 是标配实现。可以用 react-intersection-observer 库简化。
  7. infinite 单项更新陷阱：invalidate 整个 infinite query 会重置所有 pages（用户滚的位置丢）。需要 setQueryData 局部改单页。
  8. useQueries combinations：v5 新增 combine option，把多个 query 结果聚合派生（"全部 isSuccess 才显示" / "totalCount = 各页累加"）。
  9. parallel 数量很大慎用：100 个 useQuery 同时跑会发 100 个请求。看后端能否扛；考虑批量 API（一个请求拿 100 条）。
  10. dependent + parallel 混合：典型 dashboard 场景。先拿 user，再并行拿 user 的 posts / settings / notifications。

02 底层逻辑

dependent query · enabled 串行

真实业务里经常遇到："拿到 user 才能拿 user 的 posts"。不能用 if 跳过 useQuery（违反 hooks 规则），用 enabled：

tsx 
function UserPosts() {
  // Query A: 拿 user
  const { data: user } = useQuery({
    queryKey: ['user'],
    queryFn: api.getCurrentUser,
  });

  // Query B: 等 user 拿到才发
  const { data: posts, isPending } = useQuery({
    queryKey: ['posts', user?.id],
    queryFn: () => api.getPosts(user!.id),
    enabled: !!user,  // user 没拿到时 enabled=false → query 不发
  });

  // user 还没来时：
  // - posts query: status='pending', fetchStatus='idle'
  // - 不会触发 queryFn
  // user 来了：
  // - enabled 变 true → 自动发请求
  // - status='pending' → 'fetching' → 'success'
}

关键细节：

  • enabled=false 时 query 处于 idle 状态（pending + idle）
  • enabled=false 时 queryFn 不会被调用，但 query entry 仍占 cache 槽
  • 切换 enabled 从 false 到 true 等同于 mount → 会触发 fetch（如果 stale）

多步 dependent · 链式

tsx 
// 拿 user → 拿 organization → 拿 organization 的 members
const { data: user } = useQuery({
  queryKey: ['user'],
  queryFn: api.getCurrentUser,
});

const { data: org } = useQuery({
  queryKey: ['org', user?.orgId],
  queryFn: () => api.getOrg(user!.orgId),
  enabled: !!user?.orgId,
});

const { data: members } = useQuery({
  queryKey: ['org', org?.id, 'members'],
  queryFn: () => api.getMembers(org!.id),
  enabled: !!org,
});

串行链长 = 串行延迟累积。3 步串行 = 3 次 RTT。如果后端能聚合（一个 API 返回 user + org + members），考虑后端聚合而不是 client 多次 fetch。

parallel query 静态数量 · 自动并发

同组件里多次 useQuery 自动并发，无需特殊 API：

tsx 
function Dashboard() {
  // 这 3 个 query 同时发出（并发）
  const userQuery = useQuery({ queryKey: ['user'], queryFn: api.getUser });
  const settingsQuery = useQuery({ queryKey: ['settings'], queryFn: api.getSettings });
  const notificationsQuery = useQuery({ queryKey: ['notifications'], queryFn: api.getNotifications });

  if (userQuery.isPending || settingsQuery.isPending || notificationsQuery.isPending) {
    return <Loading />;
  }
  // ...
}

3 个 useQuery 并发 = 3 个请求同时出发。React 顺序调用 hook 但 query 是异步起的，不会等前一个完成。

parallel query 动态数量 · useQueries

数量在编译时未知（比如根据数组渲染）：不能用循环调 useQuery（违反 hooks 规则），用 useQueries：

tsx 
import { useQueries } from '@tanstack/react-query';

function UserAvatars({ ids }: { ids: string[] }) {
  const queries = useQueries({
    queries: ids.map(id => ({
      queryKey: ['user', id],
      queryFn: () => api.getUser(id),
    })),
  });

  // queries 是数组，每个元素是一个 useQuery 的返回
  return (
    <>
      {queries.map((q, i) => (
        <img key={ids[i]} src={q.data?.avatar} alt={q.data?.name} />
      ))}
    </>
  );
}

// 100 个 ids 会发 100 个请求 —— 注意后端能否扛

useQueries combine 选项 · v5 新

v5 给 useQueries 加了 combine option，把多个 query 结果聚合：

tsx 
const { data, isPending, isError } = useQueries({
  queries: ids.map(id => ({
    queryKey: ['user', id],
    queryFn: () => api.getUser(id),
  })),
  combine: (results) => ({
    data: results.map(r => r.data),
    isPending: results.some(r => r.isPending),  // 任何一个 pending = 整体 pending
    isError: results.some(r => r.isError),
  }),
});

// 一行拿到聚合状态，不用每个组件都自己写

paginated query · keepPreviousData

页码型分页（page=1, 2, 3...）。queryKey 包含 page，切页时切 query：

tsx 
import { keepPreviousData, useQuery } from '@tanstack/react-query';

function PostList() {
  const [page, setPage] = useState(1);

  const { data, isPlaceholderData, isFetching } = useQuery({
    queryKey: ['posts', { page }],
    queryFn: () => api.getPosts({ page }),
    placeholderData: keepPreviousData,
  });

  return (
    <>
      <ul style={{ opacity: isPlaceholderData ? 0.5 : 1 }}>
        {data?.items.map(p => <li key={p.id}>{p.title}</li>)}
      </ul>
      <button disabled={page === 1 || isFetching} onClick={() => setPage(p => p - 1)}>Prev</button>
      <span>Page {page} of {data?.totalPages}</span>
      <button disabled={page === data?.totalPages || isFetching} onClick={() => setPage(p => p + 1)}>Next</button>
    </>
  );
}

体验：

  • 用户点 Next → page 变 → queryKey 变 → 新 query
  • placeholderData 让 data 立即变成上一页（旧数据），isPlaceholderData=true
  • 新页拉到 → data 替换 → isPlaceholderData=false
  • 用户感知：内容平滑过渡，无 loading 闪烁

用 prefetch 进一步优化：

tsx 
// 预拉下一页（hover 或 idle 时）
const queryClient = useQueryClient();
useEffect(() => {
  if (data?.hasNext) {
    queryClient.prefetchQuery({
      queryKey: ['posts', { page: page + 1 }],
      queryFn: () => api.getPosts({ page: page + 1 }),
    });
  }
}, [page, data?.hasNext, queryClient]);

infinite query · "加载更多" / 无限滚动

用专门的 useInfiniteQuery hook（不是 useQuery）。关键差异：

  • data 结构是 { pages: TPage[], pageParams: TPageParam[] } —— 不是单页数据
  • 必须提供 initialPageParam 和 getNextPageParam
  • 调 fetchNextPage() 加载下一页（追加进 pages）
  • 组件层 data.pages.flatMap(...) 渲染所有项
tsx 
import { useInfiniteQuery } from '@tanstack/react-query';

type Page = { items: Post[]; nextCursor: string | null };

function PostFeed() {
  const {
    data,
    fetchNextPage,
    hasNextPage,
    isFetchingNextPage,
    isPending,
  } = useInfiniteQuery({
    queryKey: ['posts-feed'],
    queryFn: ({ pageParam }) => api.getFeed({ cursor: pageParam }),
    initialPageParam: null as string | null,
    getNextPageParam: (lastPage: Page) => lastPage.nextCursor,
    // 返回 undefined / null 表示没下一页（hasNextPage = false）
  });

  if (isPending) return <Loading />;

  // pages 是数组，每页是 Page；扁平化成单一 items 数组
  const allItems = data.pages.flatMap(p => p.items);

  return (
    <>
      <ul>{allItems.map(p => <li key={p.id}>{p.title}</li>)}</ul>
      {hasNextPage && (
        <button onClick={() => fetchNextPage()} disabled={isFetchingNextPage}>
          {isFetchingNextPage ? 'Loading...' : 'Load more'}
        </button>
      )}
    </>
  );
}

infinite + IntersectionObserver · 滚到底自动加载

把 fetchNextPage 绑到一个底部 sentinel 元素的 IntersectionObserver：

tsx 
import { useEffect, useRef } from 'react';

function PostFeed() {
  const { data, fetchNextPage, hasNextPage, isFetchingNextPage } = useInfiniteQuery({...});
  const sentinelRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    if (!sentinelRef.current || !hasNextPage) return;
    const observer = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting && !isFetchingNextPage) {
        fetchNextPage();
      }
    }, { rootMargin: '200px' });  // 200px 提前触发
    observer.observe(sentinelRef.current);
    return () => observer.disconnect();
  }, [hasNextPage, isFetchingNextPage, fetchNextPage]);

  // ...
  return (
    <>
      <ul>...</ul>
      <div ref={sentinelRef}>
        {isFetchingNextPage && <Spinner />}
        {!hasNextPage && <p>End</p>}
      </div>
    </>
  );
}

用 react-intersection-observer 库简化：

tsx 
import { useInView } from 'react-intersection-observer';

function PostFeed() {
  const { data, fetchNextPage, hasNextPage, isFetchingNextPage } = useInfiniteQuery({...});
  const { ref, inView } = useInView({ rootMargin: '200px' });

  useEffect(() => {
    if (inView && hasNextPage && !isFetchingNextPage) {
      fetchNextPage();
    }
  }, [inView, hasNextPage, isFetchingNextPage, fetchNextPage]);

  return (
    <>
      <ul>...</ul>
      <div ref={ref}>{isFetchingNextPage && <Spinner />}</div>
    </>
  );
}

infinite query 单项更新陷阱

用户滚到第 5 页 → 有 50 个 todo → 编辑第 3 个 todo → 想刷新这一项。如果直接 invalidate：

tsx 
queryClient.invalidateQueries({ queryKey: ['posts-feed'] });
// 后果：infinite query 整体 refetch
// 默认行为是只重新拉第 1 页 + 砍掉 2-5 页（refetchPage 设置）
// 用户的滚动位置丢了

v5 default behavior 是只重新 fetch 已加载的所有 pages（每页一个请求），保留所有页。但仍然有 N 次请求。

更精细的做法：用 setQueryData 局部改：

tsx 
// mutation 成功后，找到对应 page 局部改
const { mutate: editPost } = useMutation({
  mutationFn: api.editPost,
  onSuccess: (updated) => {
    queryClient.setQueryData<{ pages: Page[]; pageParams: any[] }>(
      ['posts-feed'],
      (old) => {
        if (!old) return old;
        return {
          ...old,
          pages: old.pages.map(page => ({
            ...page,
            items: page.items.map(item => item.id === updated.id ? updated : item),
          })),
        };
      }
    );
  },
});

maxPages · 限制 infinite 内存

v5 新增 maxPages 选项防止 infinite query 占用过多内存：

tsx 
useInfiniteQuery({
  queryKey: ['posts-feed'],
  queryFn: ...,
  initialPageParam: null,
  getNextPageParam: ...,
  getPreviousPageParam: (firstPage) => firstPage.prevCursor,  // 双向 infinite
  maxPages: 5,  // 最多保留 5 页（前后滚动时丢弃远端）
});

典型场景：聊天消息列表（双向 infinite）。用户滚回顶部加载历史，滚到底部加载新消息；中间太远的消息可以 gc 掉省内存。

03 应用场景

场景 1 · dashboard 混合 dependent + parallel

tsx 
function Dashboard() {
  // 第一步：拿 user（必须先有）
  const { data: user, isPending: userPending } = useQuery({
    queryKey: ['current-user'],
    queryFn: api.getCurrentUser,
  });

  // 第二步：并行拿 user 的多个相关数据
  const [postsQuery, statsQuery, notificationsQuery] = useQueries({
    queries: [
      {
        queryKey: ['posts', user?.id],
        queryFn: () => api.getPosts(user!.id),
        enabled: !!user,
      },
      {
        queryKey: ['stats', user?.id],
        queryFn: () => api.getStats(user!.id),
        enabled: !!user,
      },
      {
        queryKey: ['notifications', user?.id],
        queryFn: () => api.getNotifications(user!.id),
        enabled: !!user,
      },
    ],
  });

  if (userPending) return <Loading />;
  // 这里 user 一定有；下面 3 个 query 可能仍在 pending 但可以分别显示
  return (
    <Layout user={user}>
      <Section title="Posts">{postsQuery.isPending ? <Skeleton /> : <PostList posts={postsQuery.data} />}</Section>
      <Section title="Stats">{statsQuery.isPending ? <Skeleton /> : <StatsCard stats={statsQuery.data} />}</Section>
      <Section title="Notifications">{notificationsQuery.isPending ? <Skeleton /> : <NotificationList items={notificationsQuery.data} />}</Section>
    </Layout>
  );
}

场景 2 · 分页表格（排序 / 过滤 / page）

tsx 
type SortBy = 'name' | 'createdAt';

function UsersTable() {
  const [page, setPage] = useState(1);
  const [sortBy, setSortBy] = useState<SortBy>('createdAt');
  const [filter, setFilter] = useState({ active: true });

  const { data, isPlaceholderData, isFetching } = useQuery({
    queryKey: ['users', { page, sortBy, filter }],
    queryFn: () => api.getUsers({ page, sortBy, ...filter }),
    placeholderData: keepPreviousData,
    staleTime: 30 * 1000,
  });

  // 切排序 / 过滤时重置 page（不然可能 page=999 但新 sort 只有 5 页）
  function changeSort(newSort: SortBy) {
    setSortBy(newSort);
    setPage(1);  // 重置
  }

  return (
    <>
      <Filters value={filter} onChange={(f) => { setFilter(f); setPage(1); }} />
      <table style={{ opacity: isPlaceholderData ? 0.5 : 1 }}>
        <thead>
          <tr>
            <th onClick={() => changeSort('name')}>Name {sortBy === 'name' && '↓'}</th>
            <th onClick={() => changeSort('createdAt')}>Created {sortBy === 'createdAt' && '↓'}</th>
          </tr>
        </thead>
        <tbody>
          {data?.items.map(u => (
            <tr key={u.id}><td>{u.name}</td><td>{u.createdAt}</td></tr>
          ))}
        </tbody>
      </table>
      <Pagination page={page} totalPages={data?.totalPages} onChange={setPage} disabled={isFetching} />
    </>
  );
}

场景 3 · 双向 infinite 聊天消息

tsx 
function ChatMessages({ chatId }: { chatId: string }) {
  const {
    data,
    fetchNextPage,
    fetchPreviousPage,
    hasNextPage,

  // 底部 sentinel：滚到底 → 加载新消息
  // ...
}

场景 4 · 批量 query · 动态用户列表头像

tsx 
function MentionList({ mentionedUserIds }: { mentionedUserIds: string[] }) {
  const queries = useQueries({
    queries: mentionedUserIds.map(id => ({
      queryKey: ['user', id],
      queryFn: () => api.getUser(id),
      staleTime: 5 * 60 * 1000,
    })),
    combine: (results) => ({
      // 派生：拿到所有成功的 user 数据
      users: results.filter(r => r.isSuccess).map(r => r.data!),
      isLoading: results.some(r => r.isPending),
    }),
  });

  return (
    <>
      {queries.users.map(u => (
        <img key={u.id} src={u.avatar} alt={u.name} />
      ))}
      {queries.isLoading && <Spinner />}
    </>
  );
}

注意：100 个 mention 会发 100 个请求。如果后端有批量 API（POST /users/batch?ids=1,2,3），考虑用一个 query 拿全部，效率更高。

04 典型代码

代码 1 · infinite query 完整 + IntersectionObserver

tsx 
import { useInfiniteQuery } from '@tanstack/react-query';
import { useInView } from 'react-intersection-observer';
import { useEffect } from 'react';

type Feed = { items: Post[]; nextCursor: string | null };

function PostFeed() {
  const {
    data,
    fetchNextPage,
    hasNextPage,
    isFetchingNextPage,
    isPending,
    isError,
  } = useInfiniteQuery({
    queryKey: ['posts-feed'],
    queryFn: ({ pageParam, signal }) =>
      fetch(`/api/feed?cursor=${pageParam ?? ''}`, { signal }).then(r => r.json()) as Promise<Feed>,
    initialPageParam: null as string | null,
    getNextPageParam: (lastPage) => lastPage.nextCursor,
    staleTime: 60 * 1000,
  });

  const { ref, inView } = useInView({ rootMargin: '200px' });

  useEffect(() => {
    if (inView && hasNextPage && !isFetchingNextPage) fetchNextPage();
  }, [inView, hasNextPage, isFetchingNextPage, fetchNextPage]);

  if (isPending) return <Loading />;
  if (isError) return <Error />;

  return (
    <>
      <ul>
        {data.pages.flatMap(p => p.items).map(post => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
      <div ref={ref} className="sentinel">
        {isFetchingNextPage ? <Spinner /> : !hasNextPage ? <p>End</p> : null}
      </div>
    </>
  );
}

代码 2 · infinite query 局部更新（不破坏滚动）

tsx 
// 编辑 post：mutation 成功后局部改 cache
const { mutate: edit } = useMutation({
  mutationFn: api.editPost,
  onSuccess: (updated) => {
    queryClient.setQueryData<{ pages: Feed[]; pageParams: any[] }>(
      ['posts-feed'],
      (old) => old && ({
        ...old,
        pages: old.pages.map(page => ({
          ...page,
          items: page.items.map(item => item.id === updated.id ? updated : item),
        })),
      })
    );
    // 不 invalidate，避免重新拉所有页
  },
});

// 删除 post：移除某项
const { mutate: del } = useMutation({
  mutationFn: api.deletePost,
  onMutate: async (id: string) => {
    await queryClient.cancelQueries({ queryKey: ['posts-feed'] });
    const prev = queryClient.getQueryData(['posts-feed']);
    queryClient.setQueryData<{ pages: Feed[]; pageParams: any[] }>(
      ['posts-feed'],
      (old) => old && ({
        ...old,
        pages: old.pages.map(page => ({
          ...page,
          items: page.items.filter(item => item.id !== id),
        })),
      })
    );
    return { prev };
  },
  onError: (_e, _v, ctx) => ctx && queryClient.setQueryData(['posts-feed'], ctx.prev),
});

代码 3 · 用 useQueries 实现"先批量再批量"

tsx 
// 拿 5 个 org，每个 org 拿成员列表
function OrgsWithMembers({ orgIds }: { orgIds: string[] }) {
  // 第一批：并行拿所有 org
  const orgs = useQueries({
    queries: orgIds.map(id => ({
      queryKey: ['org', id],
      queryFn: () => api.getOrg(id),
    })),
    combine: (results) => results.filter(r => r.isSuccess).map(r => r.data!),
  });

  // 第二批：等所有 org 拿到后并行拿成员
  const members = useQueries({
    queries: orgs.map(o => ({
      queryKey: ['org', o.id, 'members'],
      queryFn: () => api.getMembers(o.id),
      enabled: !!o.id,  // org 拿到才发
    })),
  });

  return (
    <>
      {orgs.map((org, i) => (
        <div key={org.id}>
          <h3>{org.name}</h3>
          {members[i]?.isPending ? <Skeleton /> :
            <ul>{members[i]?.data?.map(m => <li key={m.id}>{m.name}</li>)}</ul>}
        </div>
      ))}
    </>
  );
}

代码 4 · paginated query 配 prefetch 下一页

tsx 
function PostsPaginated() {
  const queryClient = useQueryClient();
  const [page, setPage] = useState(1);

  const { data, isPlaceholderData } = useQuery({
    queryKey: ['posts', { page }],
    queryFn: () => api.getPosts({ page }),
    placeholderData: keepPreviousData,

  // 当前页拿到后立刻预拉下一页

    if (data?.hasNext) {
      queryClient.prefetchQuery({

        queryFn: () => api.getPosts({ page: page + 1 }),
      });
    }
  }, [page, data?.hasNext, queryClient]);

  // 用户点 Next 时下一页 cache 已经在了，瞬间显示
  return (
    <>
      {/* ... */}
      <button onClick={() => setPage(p => p + 1)}>Next</button>
    </>
  );
}

05 常见陷阱

  • 用 if 包 useQuery 实现 dependent —— 违反 hooks 规则。改用 enabled
  • dependent 链很长（5+ 步） —— 串行延迟累积。考虑后端聚合 API
  • useQueries 数量爆炸（100+ 并发请求） —— 后端可能扛不住。考虑批量 API（一个请求拿 100 条）
  • paginated 切排序 / 过滤忘了重置 page —— 用户在第 5 页，切排序仍 page=5，但新结果可能只有 3 页 → 显示空
  • paginated 没用 keepPreviousData —— 切页 loading 闪烁，UX 差
  • infinite query 用 useQuery 而不是 useInfiniteQuery —— 用错 hook，没 fetchNextPage / hasNextPage
  • infinite query 忘了 initialPageParam —— v5 强制要求；v4 自动 undefined
  • infinite query getNextPageParam 返回 0 / 空字符串 —— 0 是 falsy，TanStack Query 视为没下一页。要么返回明确的 cursor，要么用 nullish
  • infinite invalidate 整体 refetch 滚动位置丢 —— v5 默认重新拉所有 pages（保留位置但 N 次请求）。精细更新用 setQueryData 局部改
  • infinite 单页失败重试整体 —— 默认 retry 在每页层面。某页失败不影响其他页
  • useQueries 的 combine 引用每次新 —— combine 函数返回新对象 → 触发 re-render。返回原始值或 useCallback 包
  • maxPages 设太小 —— 用户滚回顶部时丢失中间页 → 重新拉。maxPages 应至少能容纳一屏
  • 把 infinite 当 paginated 用（每次 fetchNextPage 替换上一页） —— infinite 是追加，不是替换。用 paginated + page state 才是替换
  • 双向 infinite 没 getPreviousPageParam —— 只有 next 没 previous，hasPreviousPage 永远 false
  • parallel 用 Promise.all + 单个 useQuery —— queryFn: () => Promise.all([fetchA(), fetchB()]) 不利于 cache 复用（合并的 cache key 是 ['ab']，A 和 B 不能单独 invalidate）。分开多个 useQuery 更好
