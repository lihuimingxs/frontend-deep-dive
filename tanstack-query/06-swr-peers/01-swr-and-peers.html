SWR 设计 + Apollo / Relay / RTK Query / Jotai-Query 横评

一章把 TanStack Query 之外的客户端 fetch 库讲完。SWR 极简哲学（Vercel 视角）、Apollo / Relay 在 GraphQL 阵营的现状、RTK Query 是不是 Redux 项目的"自然延续"、Jotai-Query 等小众路线。最后给 2026 客户端 fetch 库市场份额和决策表。

阅读时长 ~30 分钟 前置 P1 设计哲学 + 至少了解 SWR / Apollo / RTK Query 中的一个 主要来源 swr.vercel.app + apollographql.com/docs/react + relay.dev + redux-toolkit.js.org/rtk-query + 各项目 GitHub 维护节奏

01 速览

  1. SWR 极简哲学：只有 useSWR + mutate 两个核心 API；没有 queryClient；默认 refetch 更激进（每次 mount + focus + reconnect 都拉）；无 staleTime 概念。
  2. SWR 适合：极简偏好 / Pages Router 历史项目 / Vercel 部署生态。不适合：复杂 mutation 体系 / DevTools 需求 / 大型应用 cache invalidation 网络。
  3. Apollo Client：GraphQL only；normalized cache（用 __typename:id 索引）；适合 GraphQL 重度团队。运维负担重，2026 总体收缩。
  4. Relay：Meta 内部生态；强类型 + 编译时优化；学习曲线陡；非 Meta 系大公司很少用。
  5. RTK Query：Redux Toolkit 内置；endpoint-based + 自动生成 hooks；只在已经全家桶 Redux 项目里有优势；新项目不必引入 Redux 来用 RTK Query。
  6. Jotai-Query / Zustand+自管 fetch：状态库阵营对 server state 的回应。市场份额小（<1%）；不推荐新项目用。
  7. 2026 市场份额（粗略）：TanStack Query 70% / SWR 15% / RTK Query 10% / Apollo + Relay 5% / 其他 <1%。
  8. 客户端 GraphQL 收缩：5 年前 Apollo + Relay 共占 25%，现在 5%。原因：REST + tRPC + RSC fetch 抢回份额。
  9. 选型决策不是技术对决：是项目复杂度 + 团队规模 + 现有技术栈。
  10. "我推荐 TanStack Query 给大部分新项目"是强观点。"GraphQL 项目继续 Apollo / Relay"是弱观点（看团队和规模）。"全家桶 Redux 老项目用 RTK Query"是不观点（替换不替换都行）。

02 底层逻辑

SWR · 极简哲学

SWR 由 Vercel 在 2019 年开源，作者 Shu Ding。设计原则（官方文档原话）：

💡 SWR 设计哲学

"Lightweight, focused on the right primitives. Provide the bare minimum that solves data fetching well, leave everything else to user."

典型用法：

tsx 
import useSWR, { mutate, useSWRConfig } from 'swr';

const fetcher = (url: string) => fetch(url).then(r => r.json());

function User({ id }: { id: string }) {
  const { data, error, isLoading, mutate: localMutate } = useSWR(
    `/api/users/${id}`,
    fetcher,
    { revalidateOnFocus: false }
  );

  if (isLoading) return <Loading />;
  if (error) return <Error />;
  return <div>{data.name}</div>;
}

// 全局 mutate（任意 key）
function useUpdateUser() {
  const { mutate: globalMutate } = useSWRConfig();
  return (id: string, newName: string) => {
    return fetch(`/api/users/${id}`, { method: 'PATCH', body: JSON.stringify({ name: newName }) })
      .then(() => globalMutate(`/api/users/${id}`));  // 通过 key 触发 revalidate
  };
}

SWR 的特点：

  • API 极简：useSWR + mutate。没有 queryClient / queryKey factory / mutation hook 体系
  • key 是字符串或数组：默认字符串，复杂场景用数组（类似 TanStack Query）
  • 没有 staleTime：每次 mount + focus + reconnect 都 revalidate（除非显式关）。激进刷新哲学
  • mutator 局部：useSWR 返回的 mutate 只对当前 key；全局 mutate 走 useSWRConfig
  • 没有 mutation 体系：v2 加了 useSWRMutation，但 vs useMutation 体系功能弱很多
  • 没有 DevTools：到 2026 仍然没有官方调试工具

SWR 默认行为更激进的原因

SWR 名字直接来自 stale-while-revalidate（HTTP RFC 5861），但它的"激进"是指更频繁触发 revalidate：

  • 无 staleTime：默认 0（mount 就 stale）
  • focus 重新激活：默认开（切回 tab 就 revalidate）
  • 无静默期（dedupingInterval 默认 2 秒）

结果：用户感觉 SWR "总是最新"。代价：网络请求更多。

对比 TanStack Query 的"保守 + 可调"：

  • 默认 staleTime 0（保守）
  • 但鼓励调到 60 秒（实战首选）
  • 每个 query 可以 override

SWR vs TanStack Query 全维度对比

维度 SWR TanStack Query
API 复杂度 极简（2 个核心） 中（多 hook 体系）
key 形态 字符串 / 数组 必须数组
命名空间 + 模糊匹配 有限（手动管理） 原生支持（前缀匹配）
queryClient 无（隐式全局） 一等公民（imperative API）
Mutation useSWRMutation（弱） useMutation 完整（4 步乐观）
Infinite query useSWRInfinite useInfiniteQuery（更结构化）
Suspense suspense option（实验） useSuspenseQuery 三件套（v5 一等）
staleTime 无（dedupingInterval 替代） 有（细粒度控制）
多框架 React only React / Vue / Solid / Svelte / Angular
DevTools 无 官方一等公民
Bundle size (gzip) ~4KB ~13KB
SSR / RSC 支持但弱（fallback prop） dehydrate + HydrationBoundary 完整
维护节奏 放缓（Vercel 重心转移） 非常活跃
2026 市场份额 ~15% ~70%

选择：

  • 项目小 + 极简偏好 + Vercel 部署 → SWR
  • 大型应用 / 复杂 mutation / 多组件订阅 / 需要 DevTools → TanStack Query
  • 跨框架团队 / 重 RSC 集成 → TanStack Query

SWR 在 App Router 时代的位置变化

SWR 起家于 Next.js Pages Router 时代（2019-2022）：

  • Pages Router 没有 server-side fetch 默认能力
  • SWR 的 fallback prop（getStaticProps + 客户端 hydrate）是当时主流模式
  • Vercel 强力推广 → 大量 Next.js Pages Router 项目装了 SWR

App Router 时代（2024+）：

  • RSC 接管首屏 fetch → SWR 的 fallback prop 不再必要
  • Server Actions 接管简单 mutation → useSWRMutation 用得少
  • SWR 的"中间地带"被两头蚕食
  • Vercel 自己工程重心明显转 RSC + AI SDK
  • SWR 维护节奏 2025+ 放缓（基本只 bug fix）

对比 TanStack Query：官方文档完整支持 RSC 集成（dehydrate + HydrationBoundary），社区方案成熟，吃了 SWR 让出的份额。

Apollo Client · GraphQL 阵营老大哥

Apollo Client 由 Apollo（公司）2016 年发布，是 GraphQL 客户端的事实标准之一。

tsx 
import { useQuery, useMutation, gql } from '@apollo/client';

const GET_USER = gql`
  query GetUser($id: ID!) {
    user(id: $id) {
      id name email
      posts { id title }
    }
  }
`;

const UPDATE_NAME = gql`
  mutation UpdateName($id: ID!, $name: String!) {
    updateUser(id: $id, name: $name) {
      id name
    }
  }
`;

function User({ id }) {
  const { data, loading, error } = useQuery(GET_USER, { variables: { id } });
  const [updateName] = useMutation(UPDATE_NAME);

  // ...
}

Apollo 的核心特性：

  • normalized cache：每个 entity 用 __typename:id 索引；改一个 user 所有引用它的组件自动更新
  • schema-first 工作流：GraphQL schema + codegen 生成类型
  • fragment colocation：组件声明自己需要的字段
  • InMemoryCache 强大配置：typePolicies / fieldPolicies / cache.modify
  • Apollo Studio（付费）：调试 + monitoring + schema 管理

Apollo 的代价：

  • 必须用 GraphQL：REST / tRPC 项目不能用
  • schema 维护负担：schema 长期演化、向后兼容是工程问题
  • cache 调试复杂：normalized cache 有时不直观（"为什么这个 mutation 没让那个 query 失效"）
  • 包体积大：Apollo Client 30+KB gzip

2026 现状：

  • 大型 B2B SaaS / 重 GraphQL 项目仍用
  • 新项目极少选 GraphQL（REST + tRPC 抢回份额）
  • 市场份额从 5 年前的 ~15% 跌到 ~3%

Relay · Meta 内部生态外溢

Relay 由 Meta 2015 年开源，是 Facebook / Instagram 内部使用的 GraphQL 客户端。

核心特点：

  • 编译时优化：Relay Compiler 解析 GraphQL → 生成类型 + 优化 query
  • fragment colocation 极致：组件声明 fragment + 数据形状强类型
  • data-driven dependencies：组件按需 fetch fragment 字段
  • store + cache 自动管理：normalized + reactive

Relay 的代价：

  • 学习曲线陡：fragment / connection / refetch container 等概念多
  • 编译步骤：Relay Compiler 必须配置；增加 build 复杂度
  • Meta 生态：很多设计决定优化 Meta 内部场景，外部不一定合适

2026 现状：

  • Meta 系（Facebook / Instagram / Threads）内部主力
  • 外部少数大型 GraphQL 团队（Shopify 部分项目）
  • 市场份额 <2%

RTK Query · Redux 阵营内置

RTK Query 在 2021 年加入 Redux Toolkit，是 Redux 阵营对 React Query 的回应。

tsx 
// store/userApi.ts
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

export const userApi = createApi({
  reducerPath: 'userApi',
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  tagTypes: ['User'],
  endpoints: (builder) => ({
    getUser: builder.query<User, string>({
      query: (id) => `users/${id}`,
      providesTags: (_, __, id) => [{ type: 'User', id }],
    }),
    updateUser: builder.mutation<User, { id: string; name: string }>({
      query: ({ id, ...patch }) => ({ url: `users/${id}`, method: 'PATCH', body: patch }),
      invalidatesTags: (_, __, { id }) => [{ type: 'User', id }],
    }),
  }),
});

export const { useGetUserQuery, useUpdateUserMutation } = userApi;

// 组件
function User({ id }: { id: string }) {
  const { data, isLoading } = useGetUserQuery(id);
  const [updateUser] = useUpdateUserMutation();
  // ...
}

RTK Query 的特点：

  • endpoint-based：API 集中定义在 createApi；自动生成 hooks
  • tag-based 失效：providesTags + invalidatesTags 类似 TanStack Query 的 queryKey 模糊匹配
  • 与 Redux store 共存：cache 进 Redux store 的 reducer
  • codegen 支持：从 OpenAPI / GraphQL schema 生成 endpoints

RTK Query 的定位：

  • 已经全家桶 Redux 的项目（client state 走 Redux）→ 新加 server state 时直接 RTK Query
  • 新项目想"一栈到底" → RTK Query 也行（但通常 TanStack Query 更轻）
  • 不要为了 RTK Query 引入 Redux（仅为 server state 不划算）

RTK Query vs TanStack Query 对比

维度 RTK Query TanStack Query
API 心智 endpoint-based（集中定义） hook-based（分散定义）
失效机制 tag-based queryKey prefix-based
Cache 存储 Redux store 独立 queryClient
客户端 state 已经在 Redux 分开管（useState / Zustand）
codegen 有（OpenAPI / GraphQL） 需要第三方
学习曲线 需要 Redux 心智 独立学习
Bundle size 需要 RTK 全套 单独 ~13KB
多框架 React only React / Vue / Solid / Svelte / Angular

Jotai-Query / Zustand + 自管 fetch

状态库阵营对 server state 的回应：

Jotai-Query（jotai-tanstack-query）：把 Jotai atom 系统延伸到 server state。优势：与 Jotai atomic state 心智一致。劣势：学习成本高 + 不如直接 TanStack Query 简洁。

Zustand + 自管 fetch：Zustand 是 client state 库，本身不管 server state。一些团队用 Zustand store 写 fetch 逻辑（实质上是手动复刻 TanStack Query），不推荐 —— 重复造轮子且功能弱。

市场份额：合计 <1%。新项目不推荐用，除非已经 Jotai 重度依赖。

2026 客户端 fetch 库市场份额

根据 npm trends + State of JS 2025 + 各项目 GitHub stars 估算：

库 2026 占比 5 年前占比 趋势
TanStack Query ~70% ~15% ↗ 持续上升
SWR ~15% ~30% ↘ 下降
RTK Query ~10% ~10% → 稳定（与 Redux 总占比联动）
Apollo Client ~3% ~15% ↘ 严重下降（与 GraphQL 整体收缩同步）
Relay ~2% ~5% ↘ 下降（基本是 Meta 生态）
useEffect+fetch（仍在用） ~未统计 — 残留但下降
其他 <1% — —

关键趋势：

  • RSC 让客户端 fetch 库总需求量下降
  • 但 TanStack Query 在剩余客户端场景份额仍在提升（吃 SWR 留下的份额）
  • GraphQL 客户端收缩（Apollo + Relay 5%，5 年前是 25%）
  • RTK Query 与 Redux 整体绑定，没有独立增长动力

03 应用场景

场景 1 · 新项目客户端 fetch 库决策

"我新起一个 React 项目，选什么客户端 fetch 库？"

  1. 项目用 GraphQL？
    • 是 + 重度（schema 主导工程）→ Apollo / Relay
    • 是 + 浅尝（少数 query）→ TanStack Query 也行（自己写 GraphQL queryFn）
    • 否 → 进下一问
  2. 项目已经全家桶 Redux？
    • 是 → RTK Query 自然延续
    • 否 → 进下一问
  3. 项目偏好极简 + Vercel 部署 + 简单 fetch 多？
    • 是 → SWR
    • 否 → TanStack Query（默认）

大部分新项目走到第 3 问选 TanStack Query。

场景 2 · SWR 老项目要不要切 TanStack Query

典型场景：Pages Router 项目用 SWR 5+ 年，最近迁 App Router。

判断：

  • RSC 接管了 80%+ 数据 fetch → 留 SWR 也行（剩余客户端 query 简单）
  • 剩余客户端 query 复杂（infinite / 多组件订阅 / 乐观更新）→ 切 TanStack Query 体验明显更好
  • 团队对 SWR 熟悉 + 没具体痛点 → 不必切

切的话：渐进迁移。新代码用 TanStack Query，旧代码留 SWR；半年到一年自然替换大部分。

场景 3 · GraphQL 项目 Apollo vs TanStack Query

"我们是 GraphQL 项目，必须用 Apollo / Relay 吗？"

不一定。判断：

  • 需要 normalized cache（一个 entity 改了所有引用自动更新）→ Apollo / Relay
  • 不需要 normalized cache（mutation 后手动 invalidate）→ TanStack Query 也行
  • 团队已经用 Apollo + 没具体痛点 → 留
  • Apollo 用得很重但项目里 GraphQL 比例下降（部分迁 REST）→ 考虑切 TanStack Query 统一

用 TanStack Query 调 GraphQL：

tsx 
import request, { gql } from 'graphql-request';

const getUserDocument = gql`query GetUser($id: ID!) { user(id: $id) { id name } }`;

useQuery({
  queryKey: ['user', id],
  queryFn: () => request('/graphql', getUserDocument, { id }),
});

能用，缺 normalized cache（手动 setQueryData / invalidate）。

场景 4 · Redux 老项目演化路径

Redux + thunk 老项目（2017-2020 左右），现在想现代化：

  1. 识别 store 里的 server state（70% 是）
  2. 方案 A：迁 RTK Query（仍在 Redux 里）—— 渐进、心智一致
  3. 方案 B：迁 TanStack Query（Redux 收窄到只剩 client state）—— 长期更现代但迁移工作量大
  4. 方案 C：保留现状（如果痛点不大）

选 A 还是 B 看团队规模 + 现代化预算。小团队选 A 更稳；大团队 + 长期投入选 B 更彻底。

代码 2 · Apollo Client 完整 setup

tsx 
// apollo.ts
import { ApolloClient, InMemoryCache, ApolloProvider, gql, useQuery, useMutation } from '@apollo/client';

const client = new ApolloClient({
  uri: '/graphql',
  cache: new InMemoryCache({
    typePolicies: {
      User: {
        fields: {
          posts: { merge: false },  // 替换不合并
        },
      },
    },
  }),
});

// _app.tsx
function App({ children }) {
  return <ApolloProvider client={client}>{children}</ApolloProvider>;
}

// 组件
const GET_USER = gql`query Get($id: ID!) { user(id: $id) { id name posts { id title } } }`;
const UPDATE_USER = gql`mutation Upd($id: ID!, $name: String!) { updateUser(id: $id, name: $name) { id name } }`;

function User({ id }) {
  const { data, loading, error } = useQuery(GET_USER, { variables: { id } });
  const [update] = useMutation(UPDATE_USER, {
    optimisticResponse: ({ id, name }) => ({
      __typename: 'Mutation',
      updateUser: { __typename: 'User', id, name },
    }),
  });
  // ...
}

代码 3 · RTK Query 完整 setup

tsx 
// store/userApi.ts
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

export const userApi = createApi({
  reducerPath: 'userApi',
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  tagTypes: ['User'],
  endpoints: (builder) => ({
    getUser: builder.query<User, string>({
      query: (id) => `users/${id}`,
      providesTags: (_, __, id) => [{ type: 'User', id }],
    }),
    listUsers: builder.query<User[], void>({
      query: () => 'users',
      providesTags: (result) => result
        ? [...result.map(u => ({ type: 'User' as const, id: u.id })), { type: 'User', id: 'LIST' }]
        : [{ type: 'User', id: 'LIST' }],
    }),
    updateUser: builder.mutation<User, Partial<User> & { id: string }>({
      query: ({ id, ...patch }) => ({ url: `users/${id}`, method: 'PATCH', body: patch }),
      invalidatesTags: (_, __, { id }) => [{ type: 'User', id }],
    }),
  }),
});

export const { useGetUserQuery, useListUsersQuery, useUpdateUserMutation } = userApi;

// store/index.ts
import { configureStore } from '@reduxjs/toolkit';
export const store = configureStore({
  reducer: { [userApi.reducerPath]: userApi.reducer },
  middleware: (getDefault) => getDefault().concat(userApi.middleware),
});

代码 4 · TanStack Query 调 GraphQL

tsx 
import { GraphQLClient, gql } from 'graphql-request';
import { useQuery } from '@tanstack/react-query';

const client = new GraphQLClient('/graphql');

const GET_USER = gql`query Get($id: ID!) { user(id: $id) { id name } }`;

function User({ id }: { id: string }) {
  const { data } = useQuery({
    queryKey: ['user', id],
    queryFn: () => client.request(GET_USER, { id }),
  });
  // 没有 normalized cache，但 TanStack Query 的其他能力都在
}

05 常见陷阱

  • SWR 老项目盲目切 TanStack Query —— 代价不一定值得。先评估剩余客户端 query 复杂度
  • 新项目装 SWR + 后悔切 TanStack Query —— 2026 起 App Router 项目首选 TanStack Query。除非你确认极简偏好 + 不会复杂化
  • "用 GraphQL 必须 Apollo" —— 不一定。浅尝 GraphQL 项目 TanStack Query + graphql-request 也行
  • "用 TanStack Query 必须放弃 GraphQL" —— 反方向错。TanStack Query 不在乎 queryFn 内部用什么协议
  • "为了 RTK Query 引入 Redux" —— 单纯为 server state 不划算。新项目不需要 Redux 的话直接 TanStack Query
  • Apollo 项目里也装 TanStack Query —— 通常没必要。Apollo 一个就够。除非 GraphQL 比例下降想渐进迁出
  • Apollo cache 调试问题以为是 bug —— 多半是 typePolicies 配错或 cache.modify 没写对。Apollo cache 调试是经验问题
  • Relay 在小团队用 —— 学习曲线陡，团队规模 + 长期投入不够时不要选
  • 多个客户端 fetch 库共存 —— Apollo + TanStack Query + SWR 同时装在一个项目，cache 各自独立 → 同一份数据三套 cache 互相不知道。统一选一个
  • "SWR 简单所以一定好" —— SWR 简单是优势，但功能少也是事实。看项目复杂度
  • "TanStack Query 复杂所以不学" —— 学习曲线投资 1-2 周，受益 5+ 年。是最值得学的客户端库之一
  • 市场份额数字当成"必须选 70% 那个" —— 选型看场景不是看市场份额。70% 是参考，不是命令
