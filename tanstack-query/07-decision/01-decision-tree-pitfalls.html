02 底层逻辑

3 层决策树详细

第一层 · 要不要装客户端 fetch 库

项目类型 客户端 fetch 库需求 建议
marketing 站 / 文档站 / blog 无 纯 RSC，不装
SPA + 后端 REST + 客户端交互重 有 装 TanStack Query
Next.js + 客户端交互重 + 实时数据 有（部分） RSC 管首屏 + TanStack Query 管客户端
电商商品详情 有（部分） RSC 管商品信息 + TanStack Query 管库存购物车
聊天 / 协作 有（重） TanStack Query 为主
纯 dashboard SPA 有（重） TanStack Query 是默认

第二层 · 选哪个

看决策矩阵：

条件 推荐
大部分场景 TanStack Query（默认）
极简偏好 + Vercel 部署 + 简单 fetch 多 SWR
已经全家桶 Redux + 不想引入第三方 cache RTK Query
重度 GraphQL + 需要 normalized cache Apollo Client
Meta 系 / 大型 GraphQL Relay
已重度依赖 Jotai Jotai-Query（仅参考）

详细对比见 P6.1。

第三层 · 怎么配

选了 TanStack Query 之后，全局 defaultOptions 的几个关键值：

选项 默认 推荐 说明
staleTime 0 60 * 1000（1 分钟） 大部分项目需要调；保持默认会请求过多
gcTime 5 * 60 * 1000（5 分钟） 默认即可；长会话应用调到 30 分钟 不要小于 staleTime
refetchOnWindowFocus true 普通应用 true；后台运行型 false dashboard / 监控应用关掉
refetchOnReconnect true true（保持） 断网重连应该 refetch
retry 3 次指数退避 函数（4xx 不重试） 4xx 重试也是 4xx，浪费请求
networkMode 'online' 默认 需要 PWA 离线时考虑 'offlineFirst'

个别 query 可以 override 全局配置（实时数据调短 staleTime / 关键操作调小 retry / 长会话数据调大 gcTime）。

staleTime 选择的产品判断

数据类型 staleTime 判断标准
marketing / 静态配置 1-24 小时 几乎不变；浪费请求是错
用户 profile / settings 5 分钟 偶尔变；focus 时 refetch 够用
dashboard / 列表 30 秒 - 1 分钟 有人在改；适度 refetch
库存 / 价格 5 - 30 秒 实时性重要
金融实时 0 + refetchInterval 每次都要新；强制轮询
WebSocket / SSE 0 + setQueryData 不轮询，靠推送

判断方法："如果用户看到 N 秒前的旧数据，会出问题吗"。N 就是 staleTime。

测试 query 的正确姿势

姿势 1 · 用 MSW mock fetch 层而不是 mock useQuery

为什么不要 mock useQuery：

  • mock useQuery 等于跳过 TanStack Query 的所有行为（cache / refetch / staleTime / invalidate）
  • 测的是"组件用了某个值会渲染什么"，不是测"组件在 server state 变化时怎么响应"
  • 真实 bug（竞态 / cache 问题 / refetch 时机）测不出来

正确：mock 网络层（fetch / axios），让 TanStack Query 真实跑：

tsx 
// test/setup.ts
import { setupServer } from 'msw/node';
import { http, HttpResponse } from 'msw';

export const server = setupServer(
  http.get('/api/users', () => HttpResponse.json([
    { id: '1', name: 'Alice' },
    { id: '2', name: 'Bob' },
  ])),
  http.post('/api/users', async ({ request }) => {
    const body = await request.json();
    return HttpResponse.json({ id: '3', ...body });
  }),
);

// 测试前 / 后 / 各 test 之间
beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

详细 MSW 见 测试主题 P6 MSW。

姿势 2 · 每个 test 独立 queryClient

tsx 
// test/utils.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { render, RenderOptions } from '@testing-library/react';

export function renderWithQuery(ui: React.ReactElement, options?: RenderOptions) {
  // 每个 test 独立 client，避免 cache 污染
  const queryClient = new QueryClient({
    defaultOptions: {
      queries: { retry: false },  // 测试时不重试加快失败
      mutations: { retry: false },
    },
  });
  return render(ui, {
    wrapper: ({ children }) => (
      <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
    ),
    ...options,
  });
}

// 用法
test('User loads', async () => {
  renderWithQuery(<User id="1" />);
  expect(screen.getByText('Loading')).toBeInTheDocument();
  await waitFor(() => expect(screen.getByText('Alice')).toBeInTheDocument());
});

姿势 3 · 用 waitFor 等 query 完成

tsx 
test('User shows data after fetch', async () => {
  renderWithQuery(<User id="1" />);
  // 不要 await fetch / await server response —— TanStack Query 异步管
  // 用 waitFor 等 UI 出现
  await waitFor(() => {
    expect(screen.getByText('Alice')).toBeInTheDocument();
  });
});

test('Error UI shows when fetch fails', async () => {
  // override server handler 让请求失败
  server.use(http.get('/api/users/1', () => HttpResponse.error()));
  renderWithQuery(<User id="1" />);
  await waitFor(() => {
    expect(screen.getByText(/error/i)).toBeInTheDocument();
  });
});

姿势 4 · 验证 cache 状态

tsx 
test('mutation invalidates cache', async () => {
  const queryClient = new QueryClient({ defaultOptions: { queries: { retry: false } } });
  render(
    <QueryClientProvider client={queryClient}>
      <TodoListWithAdd />
    </QueryClientProvider>
  );

  // 等列表加载
  await waitFor(() => expect(screen.getByText('Existing Todo')).toBeInTheDocument());

  // 触发 add
  await userEvent.type(screen.getByRole('textbox'), 'New Todo');
  await userEvent.click(screen.getByText('Add'));

  // 验证 cache 已 invalidate
  await waitFor(() => {
    expect(queryClient.getQueryState(['todos'])?.isInvalidated).toBe(true);
  });

  // 验证新 todo 出现
  await waitFor(() => expect(screen.getByText('New Todo')).toBeInTheDocument());
});

2027 展望

趋势 1 · RSC 进一步蚕食客户端 fetch 场景

2024-2026 RSC 已经接管首屏 + 简单变更。2027 预期：

  • Cache Components（Next.js 16 的 'use cache'）心智稳定，更多团队用
  • Server Actions 体验进一步提升（错误处理 / progressive enhancement / 文件上传）
  • 客户端 fetch 库总需求量继续下降（30-50% → 20-30%）

但 TanStack Query 在客户端独有场景仍不可替代（infinite / 实时 / 乐观 / WebSocket）。

趋势 2 · streamedQuery + AI SDK 集成

LLM 流式（OpenAI / Claude / Gemini）成为新基础设施。streamedQuery 让 TanStack Query 在 LLM 流式 + 客户端 cache 场景占位。

预期：

  • Vercel AI SDK 的 useChat 与 TanStack Query 进一步集成（或互补）
  • 客户端 LLM cache（多轮对话历史 + 实时流式）成为新需求
  • TanStack Query 可能引入 LLM 专用 hook

趋势 3 · 市场份额变化

2027 预测：

  • TanStack Query: 75-80%（继续上升）
  • SWR: 10-12%（继续下降）
  • RTK Query: 8-10%（与 Redux 共减）
  • Apollo + Relay: 3-4%（GraphQL 持续收缩）

趋势 4 · 跨框架 server state 心智统一

TanStack Query 多框架（React / Vue / Solid / Svelte / Angular）让"server state 二分法"成为跨技术栈共识。

趋势 5 · Server Components 协议被其他框架采用

RSC 协议会被 React Router v7 / TanStack Start 等其他元框架采用。"server-side fetch + client cache 桥"成为通用模式。TanStack Query 作为 client cache 一等公民地位稳固。

03 应用场景

场景 1 · 拿到新项目第一周做什么

  1. Day 1：评估项目类型 → 走第一层决策树。如果 marketing 站，关掉这个文档去用 RSC
  2. Day 2：选库（第二层决策）→ 大部分项目 TanStack Query
  3. Day 3：装 + Provider setup（详见 P5.2）
  4. Day 4：定 staleTime / gcTime / retry 全局值（第三层决策）
  5. Day 5：写 queryKey factory + 第一个 useQuery hook
  6. Week 2+：写业务 query / mutation；定期回看 DevTools 查 cache 状态

场景 2 · 现有项目症状速查

"我项目出现了 X 症状，怎么修"对照表：

症状 可能原因 修复
持续 refetch 死循环 queryKey 用了不稳定引用 useMemo 稳定 / queryKey factory
切换 id 拿到旧值 queryKey 缺依赖 / queryFn 闭包 id 放 queryKey + 从 context 读
请求爆炸（一直发请求） staleTime 0 + refetchOnWindowFocus 调大 staleTime / 关 focus refetch
切 tab 切回又请求 refetchOnWindowFocus 默认开 关 focus 或调大 staleTime
乐观更新一直显示乐观值 onError 没回滚 onError setQueryData(prev)
乐观更新被 server 旧数据覆盖 onMutate 没 cancelQueries 第一步 await cancelQueries
invalidate 没生效 queryKey 不匹配 检查命名空间 / DevTools 看 active queries
Server Component cache 跨用户泄漏 全局 new QueryClient 每请求 new + React cache helper
useSuspenseQuery app 崩溃 没包 Suspense 父级加 Suspense + ErrorBoundary
4xx 也重试 retry 默认 3 次 retry 写成函数检查 status
infinite invalidate 整体重拉 用了 invalidateQueries 而不是 setQueryData 局部改 setQueryData 局部改
select 优化没效果 select 返回新引用 / 没 useCallback 返回原始值或 useCallback 包

场景 3 · 团队培训该讲什么

代码 2 · 测试 useMutation + 乐观更新

tsx 
test('add todo with optimistic update', async () => {
  const { queryClient } = renderWithQuery(<TodoApp />);

  // 等列表加载

  // 输入新 todo

代码 3 · 用 Sentry 监控 query 错误

tsx 
// app/Providers.tsx
import * as Sentry from '@sentry/react';
import { QueryClient, QueryCache, MutationCache } from '@tanstack/react-query';

const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError: (error, query) => {
      Sentry.captureException(error, {
        tags: { queryKey: JSON.stringify(query.queryKey).slice(0, 100) },
        contexts: { query: { meta: query.meta } },
      });
    },
  }),
  mutationCache: new MutationCache({
    onError: (error, variables, _ctx, mutation) => {
      Sentry.captureException(error, {
        tags: { mutationKey: JSON.stringify(mutation.options.mutationKey).slice(0, 100) },
        contexts: { mutation: { variables } },
      });
    },
  }),
});

// 监控慢 query
new QueryClient({

05 常见陷阱 · 12 条权威清单

下面是每个 TanStack Query 用户都会踩的 12 个坑，按危害排序，每条配修复方案：

  1. 陷阱 1 · queryKey 用了不稳定引用 → 死循环

    症状：query 持续 refetch，组件不停 re-render。

    根因：queryKey 里的 object 每次 render 是新引用 → query 被识别为新的 → 重新 fetch → re-render → 又是新引用。

    修复：

    tsx 
    // ❌ 死循环
useQuery({ queryKey: ['users', { filter: { active: true } }], ... });

// ✅ 用 useMemo 稳定
const filter = useMemo(() => ({ active: true }), []);
useQuery({ queryKey: ['users', filter], ... });

// ✅ 或用 queryKey factory
const queryKey = userKeys.list({ active: true });
useQuery({ queryKey, ... });
  2. 陷阱 2 · queryFn 闭包过期变量 → 永远拿到旧值

    症状：切 id 但拿到旧 user。

    根因：queryFn 在第一次创建时记住闭包；之后 id 变化但 queryKey 没变 → 复用旧 queryFn → 闭包里仍是旧 id。

    修复：把变量放进 queryKey，从 context 读：

    tsx 
    // ❌ 错
useQuery({
  queryKey: ['user'],
  queryFn: () => api.getUser(id),  // id 闭包，永远是初始值
});

// ✅ 对
useQuery({
  queryKey: ['user', id],
  queryFn: ({ queryKey: [, userId] }) => api.getUser(userId as string),
});
  3. 陷阱 3 · staleTime 太小 + refetchOnWindowFocus 开 → 网络请求爆炸

    症状：切 tab 切回每次都发请求；DevTools 看一直 fetching。

    根因：默认 staleTime: 0；focus 总触发 refetch。

    修复：调大 staleTime（60+ 秒）或关 focus refetch。

  4. 陷阱 4 · 乐观更新 4 步漏一步

    症状：失败没回滚 / 乐观值被覆盖 / 拿不到 server 真实 id。

    修复：4 步缺一不可（cancelQueries → 保存 prev → setQueryData → onError 回滚 + onSettled invalidate）。详见 P3.2。

  5. 陷阱 5 · invalidate 后立刻 getQueryData → 拿到旧值

    症状：mutation 成功 → invalidate → 立刻 getQueryData 期望新值，但拿到旧的。

    根因：invalidate 只标 stale 等下次激活才 refetch。不是同步替换。

    修复：等 refetch 完成 → 用 await refetchQueries 或 invalidateQueries 配 await。

  6. 陷阱 6 · Server Component 全局 new QueryClient → 跨用户 cache 泄漏

    症状：用户 A 看到用户 B 的数据（生产严重 bug）。

    根因：Server Component 在 server 长跑，全局变量被并发请求共享。

    修复：每请求 new。用 React cache helper：

    tsx 
    import { cache } from 'react';
export const getQueryClient = cache(() => new QueryClient());
  7. 陷阱 7 · dehydrate 数据膨胀 → RSC payload 过大 → TTFB 慢

    症状：首屏 HTML 几 MB。

    根因：prefetch 太多 query；或 query 数据本身大。

    修复：只 prefetch 必要的；数据大的 query 不 prefetch（让 client 拿）。

  8. 陷阱 8 · useSuspenseQuery 没包 Suspense → app 崩溃

    症状：组件挂掉，整个 React tree 报错。

    修复：父级必须有 Suspense + ErrorBoundary。

  9. 陷阱 9 · select 返回新引用 → 优化失效

    症状：用了 select 仍然每次 re-render。

    根因：select 返回新对象 / 数组 → structuralSharing 部分救场但不最优。

    修复：返回原始值（number / string）；或 useCallback 稳定 select。

  10. 陷阱 10 · mutation onSuccess 立刻 navigate → onSettled 没跑 cache 没刷

    症状：navigate 到新页面后看到旧 cache 数据。

    修复：用 mutateAsync await 后 navigate；或在 onSettled 里 navigate。

  11. 陷阱 11 · RSC 项目里继续用 useEffect+fetch → 倒退

    症状：技术栈不一致，混乱。

    修复：RSC 项目首屏走 RSC fetch + 'use cache'；客户端独有走 TanStack Query。不要用 useEffect+fetch。

  12. 陷阱 12 · 多个客户端 fetch 库共存（Apollo + TanStack Query + SWR）

    症状：同一份数据三套 cache 互相不知道；改一个不刷另两个。

    修复：选一个，统一。
