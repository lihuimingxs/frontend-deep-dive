Tailwind 项目史（2017-2026）

Tailwind 9 年的演化主线，按"三次范式转移及其根因"讲：2017 utility-first 论战（挑战 BEM 主流）→ 2021 JIT 引擎革命（按需生成 + arbitrary values）→ 2025 Oxide Rust 重写 + CSS-first 配置（断代式重写）。从 Adam Wathan 一篇争议 blog 到 2026 年 React/Vue/Svelte/Astro 全生态事实标准，每次范式转移都是回应前一版的具体痛点。把这条线拉直，后面所有"为什么这样设计"都不再是天书。

阅读时长 ~30 分钟 前置 写过 CSS（任何方式） 主要来源 tailwindcss.com/blog 各版本 release notes + Adam Wathan blog + "Refactoring UI" 书 + Tailwind Labs YouTube

01 速览

  1. Tailwind 9 年由 3 次范式转移组成 —— 2017 utility-first 论战（Adam Wathan blog 挑战 BEM）→ 2021 JIT 革命（按需生成 + arbitrary values 第一次成为可能）→ 2025 Oxide 重写 + CSS-first（Rust 引擎 + @theme 替代 tailwind.config.js）。每次都解决前一版的具体生产痛点。
  2. 2017 年 8 月 Adam Wathan 发表 "CSS Utility Classes and 'Separation of Concerns'" blog，挑战当时 BEM / SMACSS / ITCSS 主流的 semantic naming 哲学。当时被骂"反工程实践"，但吸引到一批认同的工程师。
  3. 2017 年 11 月 v0.1 alpha 发布。Adam Wathan + Jonathan Reinink + Steve Schoger 三人小组。最初定位是"配套 Refactoring UI 书的工具",没料到独立成功。
  4. 2019 年 5 月 v1.0 GA，2020 年 11 月 v2.0 + 暗色模式 + 全新调色板，生态开始扎根。这两年 Tailwind 还不是主流，但在 React 生态早期采用者中口碑积累。
  5. 2021 年 3 月 v2.2 引入 JIT engine 作为实验特性 —— 解决 v1-v2 时代"完整生成所有 utility（几 MB CSS）后再 PurgeCSS"的慢。JIT 让"扫源码、按需生成"成为可能。
  6. 2021 年 12 月 v3.0 把 JIT 作为默认引擎，启用 arbitrary values（w-[123px]），完整 build 1-3 秒、增量 50-200ms。Tailwind 在 React 生态的市场份额这一年起飞。
  7. 2022-2024 年 v3.1-3.4 大量增量改进：container queries / @container / :has() 变体 / aria-* / data-* 变体 / arbitrary properties 等。这段时期 Tailwind 在 Vue / Svelte / Astro 生态扩张，shadcn/ui 出现并把 Tailwind 推到 React 生态事实标准。
  8. 2025 年 1 月 v4.0 是 9 年来最大的一次断代式重写 —— Oxide 引擎用 Rust 完整重写（与 Turbopack / SWC / Rolldown 同时代的 Rust 重写浪潮）+ CSS-first 配置（@theme / @utility / @custom-variant 在 CSS 里、tailwind.config.js 不再需要）+ 默认调色板用 OKLCH 重做。
  9. 2025 年 4 月 v4.1 增量（text-shadow / mask 等）；2026 年 2 月 v4.2 加 Webpack plugin（v4 之前主推 Vite plugin） + 4 个新色板（mauve/olive/mist/taupe）+ logical properties + 编译再快 3.8×。
  10. 纵观 9 年，Tailwind 重构频率不算高（3 次范式转移），但每次都是大的。每次重写都伴随社区争议（2017 utility-first 被骂、2021 JIT 早期 bug、2025 v4 配置反转），但每次也都被市场验证。2026 年新项目大约 70-80% 用 Tailwind，是 React 生态绝对主流样式方案。

02 底层逻辑

前史 · 2010-2017 · BEM / SMACSS / ITCSS 时代

2010-2017 年是 CSS 方法论的黄金期。主流共识是 "CSS 应该有结构、有命名规范、可维护"，几个流派：

  • BEM（Block Element Modifier，2010，Yandex）：.button / .button__icon / .button--primary。强调 block / element / modifier 三层结构
  • SMACSS（Scalable and Modular Architecture for CSS，2011，Jonathan Snook）：分类为 base / layout / module / state / theme
  • ITCSS（Inverted Triangle CSS，2014，Harry Roberts）：按特异性递增的金字塔结构
  • OOCSS（Object-Oriented CSS，2008，Nicole Sullivan）：分离结构与皮肤、容器与内容

这些方法论的共同假设：CSS class 应该 semantic（描述组件含义而非视觉效果）。代表性写法：

html 
<article class="author-bio author-bio--featured">
  <img class="author-bio__avatar" src="..." />
  <h2 class="author-bio__name">Alice</h2>
  <p class="author-bio__description">...</p>
</article>

对应 CSS：

css 
.author-bio { display: flex; padding: 16px; background: #fff; border-radius: 8px; }
.author-bio__avatar { width: 48px; height: 48px; border-radius: 50%; }
.author-bio__name { font-size: 18px; font-weight: 600; }
.author-bio__description { color: #6b7280; }
.author-bio--featured { border: 2px solid #3b82f6; }

这套写法在 2010-2017 年是 best practice。但有几个长期痛点：

  • 每个组件都要起 semantic class 名（author-bio / user-card / article-summary ...），但样式可能完全一样
  • 改设计要在 HTML 和 CSS 文件之间跳来跳去
  • 组件名重命名 = CSS 全改一遍
  • 跨组件复用样式不容易（要么继承要么 mixin）

2017 年 Adam Wathan 公开质疑这套主流共识，引发了 utility-first 论战。

2017-08 · Adam Wathan blog · utility-first 论战

Adam Wathan 是 Laravel 生态的工程师，2016 年开始与 Steve Schoger 合作设计 UI 组件库 + 写 "Refactoring UI" 书。这个过程中他逐步形成 utility-first 哲学。

2017 年 8 月 7 日，他在自己 blog 发表 "CSS Utility Classes and 'Separation of Concerns'"。核心论点：

  • "Separation of concerns" 在 CSS 里是个误解：HTML 和 CSS 不是真正"独立"的关注点。改组件外观必然要改 HTML（class 名）和 CSS（样式）。所谓"分离"只是文件物理分离，不是逻辑分离
  • Semantic class 是个糟糕的抽象：你给 div 起名 .author-bio，假设它的"含义"稳定。但实际产品迭代中，今天是 author bio、明天是 user card、后天是 testimonial。class 名跟不上含义变化
  • 不同组件常常样式相同：.author-bio 和 .product-card 可能样式 95% 一样，但 BEM 强迫你写两份
  • Utility class 反而是更好的抽象：.flex .p-4 .bg-white .rounded-lg 直接描述视觉效果。这些 utility 在所有组件里复用，不需要 semantic 中间层

blog 发表后引发激烈争论。反对派（很多 CSS 大佬）认为这是反工程实践、HTML 会变臃肿、违反 separation of concerns、不可维护。支持派（早期 React 工程师为主）认为这正是 React 时代需要的——组件即视觉单元，class 与 component co-location 自然契合。

这场论战 9 年后回看胜负已分。2026 年新项目 70-80% 用 Tailwind 系，semantic naming 主流地位被颠覆。但当年 Adam 顶着争议坚持立场，是 Tailwind 哲学根基。

2017-11 · Tailwind v0.1 alpha · 三人小组的副产品

2017 年 11 月 1 日，Tailwind v0.1 alpha 发布。Adam Wathan 后来在公开访谈中说，最初这只是 "Refactoring UI" 书的配套工具——他和 Steve Schoger 在写书过程中实现了一些 utility class，发现自己用着舒服，决定开源。

初版 v0.1 的 utility 集合相对小（color / spacing / typography / display 等），但已经定下了核心约定：

  • 4px-based spacing scale（p-1 = 4px，p-2 = 8px，p-4 = 16px...）
  • color palette 分成 50/100/200/300/.../900 多 shade
  • responsive prefix（md:p-4）
  • state variants（hover:bg-blue-500）

这套约定 9 年没怎么变。2026 v4 的默认 spacing scale / color palette 思路与 2017 v0.1 几乎一致。

v0.1 时代 Tailwind 没什么名气。Laravel 生态的小众工具，主要在 PHP / Rails 圈里传播。在 React / Vue 主流生态几乎没声音。

2019-05 · v1.0 + 2020-11 · v2.0 · 早期生态扎根

v1.0（2019 年 5 月）GA 标志 Tailwind 正式进入"production ready"阶段。这一年 Tailwind 开始在 React 生态获得早期采用者，特别是 indie hacker / 小团队 SaaS。

v2.0（2020 年 11 月）是第一次大版本升级。关键特性：

  • 暗色模式一等支持：dark:bg-gray-900 这种变体（class 策略 + media 策略）
  • extended color palette：从 9 个 shade（50-900）扩到 10 个（加 950）
  • 更现代的默认值：font / spacing 微调

v2 时期 Tailwind 仍不是主流。Vercel、Netlify 这些公司还在用 styled-components / emotion。但 Tailwind 在 docs / 工具站点 / SaaS 生态扎根，社区开始壮大。

v1-v2 时代有一个长期痛点：编译慢 + bundle 大。Tailwind 启动时生成"所有可能的 utility"（几 MB CSS），然后用 PurgeCSS 删掉项目里没用的。这个流程：

  • 启动慢（5-10 秒）
  • 不能 arbitrary values（w-[123px] 在 v2 不可用）
  • PurgeCSS 配置复杂，扫描漏 class 是常见 bug

这是 v2.2 JIT engine 出现的根因。

2021-03 · v2.2 JIT engine 实验 · 第一次范式转移

2021 年 3 月，Tailwind v2.2 引入 JIT engine 作为实验性特性（要在 config 启用）。JIT 反转了 v1-v2 的编译模型：

  • v1-v2：先生成所有 utility（穷举），再删未用
  • JIT：扫源码，按需生成

这个反转看似简单，但带来巨大变化：

  1. 启动从几秒到几百毫秒：不需要先生成几 MB CSS
  2. arbitrary values 第一次成为可能：w-[123px] / bg-[#ff5733] / grid-cols-[repeat(3,minmax(0,1fr))] —— 因为 JIT 扫到这个 class 字符串就生成对应 CSS
  3. variants 任意组合：group-hover:focus:ring-2 这种深度组合在 v1-v2 不可能（要预生成所有组合 = 爆炸性多）
  4. 开发体验大幅提升：HMR 几乎瞬时

JIT 是 Tailwind 历史上最大的工程突破之一。它让 utility-first 的"约束"从 hard constraint（只能用 spacing scale 里有的值）变成 soft constraint（默认提供 scale，但需要时可以 escape）—— 这反而强化了 Tailwind 的市场地位。

2021-12 · v3.0 · JIT 默认 + 黄金期开始

2021 年 12 月 v3.0 把 JIT 作为默认引擎（不再需要显式启用），并大量优化：

  • 完整 build 1-3 秒（与 v2 PurgeCSS 流程相比快 3-10 倍）
  • 增量 build 50-200ms（HMR 体验飞升）
  • arbitrary values + variants + media queries 任意组合
  • 新增大量 utility（aspect-ratio、scroll-snap、accent-color 等）

v3 是 Tailwind 起飞的版本。这一年 React 生态发生几件配合事件：

  • Vercel / Next.js 全面拥抱 Tailwind：next.js create-next-app 把 Tailwind 作为默认选项之一
  • Vite 生态扩张：Vue / Svelte / Astro 等都内置 Tailwind 集成
  • 2022 shadcn/ui 出现：copy-paste UI 库基于 Tailwind + Radix，把 Tailwind 推到 React 事实标准
  • CSS-in-JS 退潮：React 18 Concurrent Mode + RSC 让 runtime CSS-in-JS 出现 hydration 问题，Vercel 公开建议远离 styled-components / emotion runtime 模式

2022-2024 年 v3.1-3.4 大量增量更新：

  • v3.2（2022-10）：container queries（@container）+ data-* / aria-* 变体
  • v3.3（2023-03）：line-clamp / dynamic viewport units / logical properties
  • v3.4（2023-12）：has-* 变体（CSS :has() 在 Tailwind 的封装）+ size-* utility

这段时期 Tailwind 没有大重构，但工具链不断成熟。Tailwind v3.4 在 2024-2025 年仍是大量项目的存量主线。

2025-01 · v4.0 · Oxide + CSS-first 断代式重写

2025 年 1 月 22 日，Tailwind v4.0 发布。这是 9 年来最大的一次断代式重写，核心变化两件：

1. Oxide 引擎：Adam Wathan 团队招了 Andrey Sitnik（PostCSS 作者）等多位 Rust 工具链专家，把 Tailwind 引擎从 PostCSS plugin（JS）完整重写为 Rust。性能数字（基于官方 benchmark）：

  • 完整 build：v3 ~378ms → v4 ~100ms（3.5-5×）
  • 增量 rebuild：v3 ~44ms → v4 ~5ms（8-10×）
  • 新增 CSS 重编译：v3 几百 ms → v4 5ms（100×）

Oxide 是 web 工具链 2024-2026 大潮的一部分（Turbopack / SWC / Rolldown / Biome 等都是 Rust 重写）。Tailwind 不能例外。

2. CSS-first 配置：tailwind.config.js（JS 配置）被废弃，所有配置移到 CSS：

css 
/* v4 写法 */
@import "tailwindcss";

@theme {
  --color-brand: oklch(0.7 0.15 240);
  --spacing-em: 1em;
  --font-display: "Inter Display", sans-serif;
}

@utility tab-* {
  tab-size: --value(--tab-size-*);
}

对比 v3：

javascript 
// v3 写法（仍兼容但不推荐）
// tailwind.config.js
module.exports = {
  content: ['./src/**/*.{js,ts,jsx,tsx}'],
  theme: {
    extend: {
      colors: { brand: '#3b82f6' },
      spacing: { em: '1em' },
      fontFamily: { display: ['Inter Display', 'sans-serif'] },
    },
  },
};

为什么从 JS 转 CSS？Adam Wathan 的解释：

  • JS 配置启动慢（每次都要 require Node module）
  • JS 配置类型推导差（IDE 不能跳转 token）
  • theme 配置最终产物是 CSS 变量，中间转换层让人困惑
  • CSS 配置直接对应输出，所见即所得

v4 同时把默认调色板用 OKLCH 重做（perceptually uniform、wide gamut 支持），这与现代 CSS 的 OKLCH 主流化同步。

v4 升级时社区争议：

  • 很多团队对 "tailwind.config.js 没了" 不适应
  • OKLCH 调色板视觉略有不同（亮色 / 饱和度微调），需要 visual regression
  • 部分 plugin 生态需要时间迁移到 v4

但官方提供 tailwindcss/upgrade codemod 自动迁移大部分代码。tailwind.config.js 兼容模式仍可用，给老项目留余地。

2025-04 · v4.1 · text-shadow + mask

2025 年 4 月 v4.1 发布。增量更新：

  • text-shadow utilities（text-shadow-sm/md/lg）
  • mask utilities（mask-* 系列，配合 mask-image / mask-position）
  • 额外 color utilities（dynamic / oklch arbitrary）
  • 性能再优化（具体场景 build 时间继续下降）

这是典型的"社区反馈驱动 + 增量改进"版本，没有破坏式变更。

2026-02 · v4.2 · Webpack plugin + 新色板 + logical properties

2026 年 2 月 18 日 v4.2 发布。三件大事：

  1. Webpack plugin（@tailwindcss/webpack）：v4 之前主推 Vite plugin / PostCSS plugin。但 Webpack 仍是大量企业项目的事实，用户呼声让团队把 Webpack plugin 补全
  2. 4 个新色板：mauve / olive / mist / taupe —— 中性色补充。让 design 选择更丰富
  3. Logical properties utilities：inset-s / inset-e / inset-bs / inset-be、inline-size / block-size 等。支持 RTL（阿拉伯语 / 希伯来语）和双向布局
  4. 编译再快 3.8×（特定场景）

v4.2 是 v4 路线下的稳定增量，没有破坏式变更。2026 年 5 月（本文写作时）是 Tailwind 最新主线。

纵观 9 年 · Tailwind 三次范式转移的根因

Tailwind 3 次范式转移的共同模式：每次都解决前一版的具体痛点，且痛点已经被生产场景验证过。

  • 2017 utility-first 论战：解决 BEM 时代"semantic class 跟不上产品迭代 + 不同组件样式重复"的痛点。痛点早就存在，Adam 只是公开命名它
  • 2021 JIT 革命：解决 v1-v2 时代"启动慢、不能 arbitrary values、PurgeCSS 配置复杂"的痛点。痛点是 Tailwind 自己制造的，但社区呼声推动重写
  • 2025 Oxide + CSS-first：解决 v3 时代"JS 配置启动慢、类型推导差、CSS 变量中间层困惑、PostCSS plugin 性能瓶颈"的痛点。痛点也是自己制造，但同时享受 Rust 工具链 + 现代 CSS 的时代红利

这种"每隔几年回应自己造的痛点"的演化模式，让 Tailwind 保持活力但也带来代价 —— 每次重写都要团队迁移、社区学习、生态适配。但相比 Bootstrap 9 年几乎停滞、CSS-in-JS 因 RSC 受重创，Tailwind 这条路走得相对稳。

03 应用场景

场景 1 · 从 v3 项目升 v4

2025-2026 年最常见的迁移场景。理解历史的实战价值：

  • v3 项目仍可用（v4 兼容模式支持 tailwind.config.js）；不必着急升
  • 升级路径：npx @tailwindcss/upgrade@latest codemod 自动改大部分
  • codemod 处理：tailwind.config.js → @theme、@tailwind base/components/utilities → @import "tailwindcss"
  • 需要手动审视：默认调色板视觉差异（v4 OKLCH 重做、亮度饱和度微调）
  • plugin 生态：检查所有 tailwind plugin 是否有 v4 兼容版本
  • 建议先在 staging 跑 visual regression，对比 v3 / v4 输出差异

场景 2 · 选 v3 还是 v4 起新项目

2026 年起新项目几乎都应该 v4：

  • v4 配置更直观（CSS-first）
  • 性能显著优于 v3
  • OKLCH 调色板更现代
  • shadcn/ui 等主要 UI 库已全面支持 v4

例外：

  • 团队成员对 v4 CSS-first 极度抗拒（少见但存在）
  • 依赖某个仅 v3 兼容的 plugin（罕见）

场景 3 · 评估"我们要不要从 styled-components 迁 Tailwind"

2024-2026 年大量 React 团队的选型决策。理解历史给的判断：

  • styled-components 等 runtime CSS-in-JS 与 React 18+ Concurrent Mode + RSC 不兼容（hydration 问题）；Vercel 已公开建议远离
  • 替代方案：Tailwind / vanilla-extract（build-time CSS-in-TS）/ CSS Modules / 纯 CSS
  • 选 Tailwind 的优势：生态最成熟（shadcn/ui 等）+ 与 RSC 完全兼容 + 团队接受度高
  • 选 vanilla-extract 的优势：TypeScript 优先 + 零运行时 + 无 utility 心智
  • 2026 年事实标准：80%+ React 项目从 CSS-in-JS 迁向 Tailwind

04 典型代码

代码 1 · 同一组件在四个时代的写法对比

以 "author bio card" 为例，看 CSS 9 年范式演变：

html 
<!-- 时代 1 · 2010-2017 BEM 时代 -->
<article class="author-bio author-bio--featured">
  <img class="author-bio__avatar" src="..." />
  <h2 class="author-bio__name">Alice</h2>
  <p class="author-bio__description">CSS engineer at Vercel</p>
</article>
<style>
  .author-bio { display: flex; gap: 16px; padding: 16px; background: #fff; border-radius: 8px; }
  .author-bio--featured { border: 2px solid #3b82f6; }
  .author-bio__avatar { width: 48px; height: 48px; border-radius: 50%; }
  .author-bio__name { font-size: 18px; font-weight: 600; }
  .author-bio__description { color: #6b7280; font-size: 14px; }
</style>
tsx 
// 时代 2 · 2018-2023 styled-components / CSS-in-JS
import styled from 'styled-components';

const Card = styled.article<{ featured?: boolean }>`
  display: flex;
  gap: 16px;
  padding: 16px;
  background: #fff;
  border-radius: 8px;
  ${p => p.featured && `border: 2px solid #3b82f6;`}
`;
const Avatar = styled.img`width: 48px; height: 48px; border-radius: 50%;`;
const Name = styled.h2`font-size: 18px; font-weight: 600;`;

function AuthorBio({ featured }) {
  return (
    <Card featured={featured}>
      <Avatar src="..." />
      <div><Name>Alice</Name>...</div>
    </Card>
  );
}
tsx 
// 时代 3 · 2021-2025 Tailwind v3

05 常见陷阱

  • v3 项目以为 v4 升级很复杂不敢升 —— codemod 自动迁移 80%+ 代码。Visual regression 测试一下，大多数项目几小时升完
  • 看 2018-2020 年的 Tailwind 教程 —— 大量已过时（PurgeCSS 流程、@apply 推荐、tailwind.config.js）。看 tailwindcss.com 当前版本 docs
  • "Tailwind class 让 HTML 太丑" —— 装 prettier-plugin-tailwindcss 自动排序、用 CVA + cn 抽组件，丑感大幅减少（详见 P6）
  • "我用 @apply 把 utility 合并成 .btn 不就好了" —— 反模式（详见 P2.1）。你又回到 BEM，失去 Tailwind 价值。正确做法是抽 React component
  • 动态拼 class 字符串 —— className={`bg-${color}-500`} JIT 扫不到这种动态 class（看到的是模板字符串字面量，不知道 color 取什么值）。要么用 safelist，要么完整写出所有可能 class（color === 'red' ? 'bg-red-500' : 'bg-blue-500'）
  • v4 项目还在用 tailwind.config.js —— v4 兼容模式支持，但这是老 v3 心智。新项目用 @theme
  • OKLCH 调色板与设计师 Figma sRGB 颜色对不上 —— v4 默认调色板 OKLCH 重做，与 v3 / Figma sRGB 略有差异。需要：1) 让设计师切到 OKLCH；2) 或在 @theme 里覆盖默认色
  • 升 v4 后 PostCSS plugin 报错 —— v4 不再依赖 PostCSS pipeline。如果你有自定义 PostCSS plugin，迁到 v4 的 @utility / @custom-variant
  • v4 项目装 v3 plugin 报错 —— 检查 plugin 是否有 v4 兼容版本（大部分主流 plugin 已有）
  • "Tailwind 9 年没大改还是 utility 那套" —— 表面对，但 JIT（2021）和 Oxide（2025）两次引擎重写改变了使用体验。心智确实没变（这是优势），但工具链每隔几年大改
  • 等 v5 不升 v4 —— v5 路线图未公开，没有公开时间表。v4 在 2026-2028 都将是主线
