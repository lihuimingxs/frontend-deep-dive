同代人横评 + 设计哲学

2026 年的 CSS 方案格局比 2018 年丰富得多 —— Tailwind 主流 / CSS-in-JS 退潮 / UnoCSS 增长 / vanilla-extract 在 TS 团队稳态 / CSS Modules 仍是 scoped 默认 / Sass 几乎停滞。这一章把六个核心维度（心智模型 / 配置方式 / 编译时机 / 生态 / design system 友好度 / 性能）摊开做对比，再讲清 Tailwind 的 4 个核心设计赌注。读完能回答"我们项目应该选哪个 CSS 方案"。

阅读时长 ~30 分钟 前置 P1.1 项目史 主要来源 各 CSS 方案官方文档 + 2025-2026 主要框架团队公开文章 + State of CSS 调研

01 速览

  1. 2026 年 CSS 方案不再是"Tailwind vs CSS-in-JS"二选一。新格局是 Tailwind 主流 / CSS-in-JS 退潮 / UnoCSS 增长 / vanilla-extract 稳态 / CSS Modules 仍默认 / Sass 几乎停滞。
  2. Tailwind 在新 React 项目里占 70-80% 份额；剩 15% 用 CSS Modules / 纯 CSS；不到 10% 用 vanilla-extract / UnoCSS / 其他。CSS-in-JS（runtime 模式）几乎被新项目放弃。
  3. CSS-in-JS 退潮的根因：React 18+ Concurrent Mode + RSC（Server Components）让 runtime CSS-in-JS 出现 hydration 问题。2024 年 Vercel 公开建议远离 styled-components / emotion runtime 模式。这个生态级判断让大量团队迁向 Tailwind / vanilla-extract。
  4. UnoCSS 是 Tailwind 兼容预设 + 零运行时 + 与 Vue / Vite / Astro 生态原生集成的原子化 CSS 引擎。性能更激进（比 Tailwind v3 快几倍），但生态比 Tailwind 小很多。Vue / Astro 生态偏多。
  5. vanilla-extract 是 build-time CSS-in-TS，类型安全。重 TypeScript 团队优先。与 Tailwind 不冲突可共存（一些团队 utility 用 Tailwind、token / theme 用 vanilla-extract）。
  6. CSS Modules 仍是 React / Next.js 内置 scoped CSS 方案。"组件级局部样式"场景适合，不解决 design system。与 Tailwind 共存的项目不少（少量组件用 CSS Modules、大部分用 Tailwind）。
  7. Bootstrap 5 在 2026 年仍存量大但增长几乎停滞。9 年前是绝对主流，9 年后被 Tailwind 推到边缘。新项目几乎不选。
  8. 纯 CSS / Sass 仍有大量项目。Sass 增长几乎停滞 —— CSS native nesting + custom properties + container queries 已能覆盖 90% 用例。新项目不必引入 Sass。
  9. Tailwind 的 4 个核心设计赌注：utility-first（class 描述视觉）/ constraints over creativity（默认有限 scale）/ no semantic class abstraction（不用 .btn-primary）/ co-location with markup（与 React/JSX 配合）。这 4 条决定了它和其他方案的所有结构性差异。
  10. 横评的关键不是"哪个最好"，是"在你的场景里哪个最合适"。React + 团队多人协作 → Tailwind；TypeScript 重度团队 → vanilla-extract；Vue / Astro → UnoCSS / Tailwind 都行；存量 Bootstrap 项目 → 继续。

02 底层逻辑

同代人 1 · Bootstrap 5（老牌生态）

Bootstrap 由 Twitter 团队 2011 年开源，2016-2020 年是 CSS 框架绝对主流。v5（2021）去除 jQuery 依赖，加入 utility class，但已经晚了。

设计立场：

  • 组件优先：.btn / .card / .modal 等 semantic class，每个组件预设样式
  • 双重模式（v5+）：组件 + utility 并存（学 Tailwind）
  • Sass 配置：theme 通过 Sass 变量覆盖
  • HTML 即 UI：复制粘贴官网示例就能跑

Bootstrap 在 2026 年的现状：

  • 存量大（大量企业 web 项目、CMS 主题、内部工具）
  • 增长几乎停滞（2024-2026 年新项目份额持续下滑）
  • v6 路线图模糊
  • 不与 React / Vue 现代生态深度集成

何时仍选 Bootstrap：

  • 已有大量 Bootstrap 项目（迁移成本高）
  • 团队完全不接受 utility-first（Bootstrap 组件优先 + 完整文档对小团队友好）
  • 纯 HTML / 后端模板项目（Rails / Django / Laravel 老式 SSR）

同代人 2 · CSS-in-JS（styled-components / emotion）· 退潮

CSS-in-JS 在 2016-2022 年是 React 生态主流：

  • styled-components：模板字符串语法，组件即样式
  • emotion：API 更灵活，小一些
  • @emotion/css：runtime 注入

设计哲学：CSS 与组件 co-location（与 Tailwind 类似），但实现方式是 runtime 注入 CSS 字符串到 <style> 标签。

2024 年 CSS-in-JS 突然退潮，根因：

  • React 18 Concurrent Mode：Suspense / streaming 与 runtime CSS 注入不兼容（hydration mismatch）
  • React Server Components：CSS-in-JS runtime 在 server 不能跑（动态 className 不能在 RSC 序列化）
  • Vercel 公开建议（2023-12）：远离 styled-components / emotion runtime 模式
  • 性能：runtime 注入有开销（每次 render 计算 hash + 注入 style）

2026 年 CSS-in-JS 现状：

  • 新项目几乎不选
  • 存量项目逐步迁移（最常见迁向 Tailwind）
  • vanilla-extract 等 "build-time CSS-in-TS" 方案接住部分 TS 用户需求

这是 9 年前不可想象的转折。CSS-in-JS 当时被认为是"React 时代的样式终极方案"，但 RSC 改变了游戏规则。

同代人 3 · UnoCSS（增长，Vue / Astro 偏多）

UnoCSS 由 Anthony Fu（Vite / Vitest / Slidev 等核心维护者）2022 年发布。设计立场：

  • 原子化 CSS 引擎，不是固定 utility 集合
  • preset 系统：可加载 Tailwind preset / Wind preset / 等
  • 零运行时，build-time 完成所有计算
  • 性能极致：Anthony Fu 公开 benchmark 比 Tailwind v3 快几倍
  • 与 Vite 生态深度集成：Vue / Svelte / Astro / Nuxt 一等支持

UnoCSS 与 Tailwind 的关键差异：

  • Tailwind 是"框架"，UnoCSS 是"引擎"
  • Tailwind 默认 utility 集合稳定，UnoCSS 通过 preset 灵活组合
  • Tailwind 生态远大（plugin / UI 库 / 教程），UnoCSS 生态在 Vue / Vite 圈集中

2026 年 UnoCSS 现状：

  • Vue / Astro / Vite 生态有 15-20% 份额
  • React 生态份额很低（Tailwind 太强）
  • 性能优势在大型项目可见，小项目不明显

同代人 4 · vanilla-extract（TS 优先）

vanilla-extract 由 SEEK（澳洲招聘平台）团队开源。设计立场：

  • build-time CSS-in-TS：写 TS / JS 文件描述样式，build 时编译为 CSS + class hash
  • 类型安全：theme / sprinkles / variants 全 TypeScript 推导
  • 零运行时：与 styled-components 不同，输出真 CSS 文件
  • 与 Tailwind 不冲突：可共存

典型代码：

typescript 
// theme.css.ts
import { createTheme } from '@vanilla-extract/css';

export const [themeClass, vars] = createTheme({
  color: { brand: 'oklch(0.7 0.15 240)', text: '#000' },
  space: { sm: '8px', md: '16px' },
});

// Button.css.ts
import { style } from '@vanilla-extract/css';
import { vars } from './theme.css.ts';

export const button = style({
  background: vars.color.brand,
  padding: vars.space.md,
});

vanilla-extract 的优势：

  • 类型安全程度比 Tailwind 强（theme token 自动补全 + 类型 catch）
  • 无 utility 心智（接受不了 utility-first 的团队选这个）
  • 零运行时，与 RSC / SSR 完全兼容
  • 生态稳定（有 Sprinkles 提供 utility 风格、Recipe 提供 variant）

vanilla-extract 的劣势：

  • 生态比 Tailwind 小（没有 shadcn/ui 这种事实标准 UI 库）
  • 学习曲线（需要懂 TS + CSS variable + 编译模型）
  • 2026 年市场份额小（< 5%）

典型采用场景：重 TypeScript 团队 + 不接受 utility-first 文化 + 需要类型安全 design system。

同代人 5 · CSS Modules（仍默认 scoped 方案）

CSS Modules 是 webpack 生态早期工具，让 CSS class 自动 scoped（hash class 名）。Next.js / Vite 内置原生支持。

css 
/* Button.module.css */
.button { padding: 16px; background: blue; }
.button:hover { background: darkblue; }
tsx 
import styles from './Button.module.css';

export function Button() {
  return <button className={styles.button}>Click</button>;
  // 渲染：<button class="Button_button__a3f9d">...
}

CSS Modules 的位置：

  • 解决"组件级局部样式"问题（class 不会全局冲突）
  • 不解决 design system / token / utility 问题
  • 与 Tailwind 共存的项目很常见（绝大多数用 Tailwind、少数复杂组件用 CSS Modules）
  • Next.js / Vite 零配置支持，迁移成本几乎为 0

同代人 6 · 纯 CSS / Sass（稳定但增长停滞）

2024-2026 年纯 CSS 能力大幅增强 —— native nesting / cascade layers / container queries / @scope / :has() / OKLCH / @property。这些特性让"不用 Sass 也能写好 CSS"成为现实。

Sass 增长几乎停滞的根因：

  • nesting：CSS 原生支持（2023-2024 浏览器主流）
  • variables：CSS custom properties 已能覆盖
  • mixins：在 utility-first 时代价值降低
  • functions：CSS 原生 calc / clamp / color-mix 已够

纯 CSS / Sass 何时仍合理：

  • 极小项目（5 页营销站、不引入工具链）
  • 邮件模板（HTML 不能改、必须 inline / 简单 CSS）
  • WordPress 主题 / Drupal 等 CMS 输出
  • 团队反 utility 文化

Tailwind 的 4 个核心设计赌注

把 Tailwind 与同代人对比，能看清它的 4 个根本立场。

1. Utility-first 不 semantic-first —— class name 描述视觉效果（bg-blue-500 p-4 rounded-lg），不描述语义（.author-bio）。这是 Tailwind 最大的文化赌注，也是初学者最不接受的。优势：class 与组件解耦、跨组件复用样式自动、改组件名无成本。代价：HTML 看起来"丑"（工具能解决，详见 P6）。

2. Constraints over creativity —— 默认提供有限的 spacing scale（4px 基数，0/1/2/4/8/...）/ color palette（24 个色家族 × 11 个 shade）/ breakpoints（5 个）。不让你随便写 padding: 13px。约束让团队风格一致、视觉系统连贯。优势：design system 即工具自身、Figma 与代码自然对齐。代价：偶尔需要 escape（arbitrary values 解决）。

3. No semantic class abstraction —— 不像 Bootstrap 提供 .btn-primary，让你每次都写完整 utility 串。看似冗长，但消除"这个 class 在改什么"的认知负担。优势：你看 className 即知道视觉、不用查 CSS 文件。代价：重复 utility 串（CVA + cn 解决，详见 P6）。

4. Co-location with markup —— CSS 与 HTML / JSX 并排写，不分离。与 React / JSX "组件 = 视觉 + 行为"哲学高度契合。优势：改样式不需要在两个文件之间跳。这是 Tailwind 在 React 生态爆火的根因。代价：HTML 行长（折行 / prettier-plugin 解决）。

2026 实际市场分布

基于 State of CSS 2024-2025 + npm 下载量 + State of JS / Stack Overflow 调研：

  • Tailwind：70-80% 新 React / Vue / Svelte / Astro 项目（含 Next.js / Vite 默认）
  • CSS Modules：约 10%（多与 Tailwind 共存，处理"复杂局部样式"）
  • UnoCSS：5-8%，Vue / Astro 圈偏多
  • vanilla-extract：3-5%，重 TS 团队
  • 纯 CSS / Sass：~5%，小项目 / CMS / 老团队
  • CSS-in-JS（runtime）：< 5%，且持续下滑（存量项目维护中）
  • Bootstrap：在新 React 项目几乎不选；老项目 / 后端模板仍大量存量

03 应用场景

场景 1 · 起新 React 项目（dashboard / SaaS）

选 Tailwind v4 + shadcn/ui。理由：

  • React 生态事实标准
  • shadcn/ui 提供高质量组件代码（Radix + Tailwind）
  • 与 Next.js / Vite / Bun 都零配置集成
  • 团队接受度高（招人友好）
  • 性能 + 维护性 + 生态规模都最优

场景 2 · Vue / Astro / Nuxt 项目

选 Tailwind（首选）或 UnoCSS（次选）。理由：

  • Tailwind 生态规模仍胜出
  • UnoCSS 与 Vite 生态深度集成有性能优势
  • 两者心智几乎兼容（UnoCSS 有 Tailwind preset）
  • 具体看团队偏好：Anthony Fu 生态团队优先 UnoCSS、跨框架团队优先 Tailwind

场景 3 · 重 TypeScript 团队 + 不接受 utility-first

评估 vanilla-extract。理由：

  • 类型安全比 Tailwind 强
  • 无 utility 心智
  • build-time 编译，与 RSC / SSR 完全兼容
  • Sprinkles 可补足 "需要 utility 风格" 的需求

但要接受：生态小、招人难（vanilla-extract 工程师比 Tailwind 少很多）。

场景 4 · 已有 Bootstrap 项目，要不要迁

看场景：

  • 项目仍在持续投入 + 团队接受 utility-first → 迁 Tailwind（迁移成本中等）
  • 项目存量维护 → 继续 Bootstrap（不要为迁移而迁移）
  • 纯 HTML / 后端模板（Rails / Django） → 继续 Bootstrap（Tailwind 在没有 build pipeline 的场景成本高）

场景 5 · 已有 styled-components 项目（React）

建议迁向 Tailwind（首选）或 vanilla-extract（TS 重）。理由：

  • styled-components runtime 模式与 RSC 不兼容
  • 2024 年 Vercel 公开建议远离
  • 长期维护成本高（hydration 问题、性能、bundle）
  • 迁移成本：30-40%（每个 styled component 改成 Tailwind className）；codemod 工具能帮一部分

04 典型代码

代码 1 · Button 在 5 个 CSS 方案的写法

tsx 
// 1. Tailwind v4
<button className="px-4 py-2 bg-blue-500 hover:bg-blue-600 text-white rounded-md">
  Click
</button>
tsx 
// 2. UnoCSS（与 Tailwind 几乎一样）
<button className="px-4 py-2 bg-blue-500 hover:bg-blue-600 text-white rounded-md">
  Click
</button>
typescript 
// 3. vanilla-extract
// Button.css.ts
import { style } from '@vanilla-extract/css';
export const button = style({
  paddingInline: '16px',
  paddingBlock: '8px',
  background: '#3b82f6',
  color: '#fff',
  borderRadius: '6px',
  ':hover': { background: '#2563eb' },
});

// Button.tsx
import * as styles from './Button.css.ts';
<button className={styles.button}>Click</button>
tsx 
// 4. CSS Modules
// Button.module.css
.button {
  padding: 8px 16px;
  background: #3b82f6;
  color: #fff;
  border-radius: 6px;
}
.button:hover { background: #2563eb; }

// Button.tsx
import styles from './Button.module.css';
<button className={styles.button}>Click</button>
tsx 
// 5. styled-components（退潮但仍存在）
import styled from 'styled-components';

const Button = styled.button`
  padding: 8px 16px;
  background: #3b82f6;
  color: white;
  border-radius: 6px;
  &:hover { background: #2563eb; }
`;
<Button>Click</Button>

代码 2 · 同样支持暗色模式的写法对比

tsx 
// Tailwind（dark: 变体内置）
<div className="bg-white text-gray-900 dark:bg-gray-900 dark:text-white p-4">
  Hello
</div>

// CSS Modules（自己处理暗色）
// Card.module.css
.card { background: white; color: #111; padding: 16px; }
@media (prefers-color-scheme: dark) {
  .card { background: #1f2937; color: white; }
}
// 或用 .dark 父类策略
:global(.dark) .card { background: #1f2937; color: white; }

05 常见陷阱

  • "Tailwind 一定比 CSS Modules 好" —— 不一定。CSS Modules 在"复杂局部样式 + 多个 :nth-child / 复杂选择器"场景比 Tailwind 简单。两者可以共存，按场景选
  • "Bootstrap 就是过时" —— 在纯 HTML / 后端模板 / 内部工具场景，Bootstrap 仍是合理选择。Tailwind 需要 build pipeline，不是所有场景适合
  • 看 2018-2022 年的 CSS-in-JS vs Tailwind 对比文章 —— 那时候 RSC 还没主流、Tailwind v3 还没出、UnoCSS 还没起。结论几乎全过时
  • 用 styled-components 起新 React 项目（2026 年） —— 不建议。RSC 兼容性问题 + Vercel 公开建议远离 + 团队招人难。除非有强非迁不可的理由
  • "vanilla-extract 类型安全所以一定好" —— 类型安全是优势，但生态小是真实劣势。小团队招人难、缺组件库、社区资源少
  • 纠结半年选不出来 —— 选型不可逆程度低（Tailwind / UnoCSS 之间几乎兼容）。先选一个写代码 2-3 周；遇到问题再调整
  • "我们大公司必须用 vanilla-extract 因为类型安全" —— 大公司 Tailwind 反而更普遍（Vercel / Shopify / GitHub / Linear / Notion 等都用 Tailwind）。类型安全不是选型唯一指标
  • 同时用 Tailwind + styled-components —— 不推荐。两套样式系统冲突、bundle 重复、心智混乱。选一个为主
  • UnoCSS 在 React 项目用 Tailwind preset 然后纠结要不要迁 Tailwind —— 如果你已经用 Tailwind preset 写代码，UnoCSS 几乎等价于 Tailwind 引擎。生态考虑可以迁 Tailwind，性能考虑可以留 UnoCSS
  • 被 "Tailwind 好丑" 吓跑 —— 这是表面观感问题，不是工程问题。装 prettier-plugin-tailwindcss 排序、用 cn() 抽组件、几周就习惯了。生态 / 工程 / 招人多角度看 Tailwind 优势远大于"美观"代价
  • "Sass 是必须的" —— 2026 年纯 CSS 已能覆盖 Sass 90% 用例。新项目不必引入 Sass
