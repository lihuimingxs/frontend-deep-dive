CSS-first 配置（v4 革命）

v4 把所有配置从 tailwind.config.js（JS）移到 CSS（@theme / @utility / @custom-variant），是 9 年来最大的一次断代式重写。这一章把 v4 五大指令（@import / @theme / @utility / @custom-variant / @source）全部讲透 + 给出 v3 → v4 完整迁移路径 + 解释"为什么从 JS 转 CSS"的根本动机。

阅读时长 ~30 分钟 前置 P3.1 引擎演化 + P2.2 设计 token 主要来源 tailwindcss.com/blog v4 + tailwindcss.com/docs Theme/Adding-Custom-Styles + Adam Wathan 公开访谈

01 速览

  1. v4 用 5 个核心 CSS 指令替代 v3 tailwind.config.js：@import "tailwindcss"（替代 @tailwind base/components/utilities）/ @theme（设计 token + 自动生成 utility）/ @utility（自定义 utility）/ @custom-variant（自定义变体）/ @source（显式扫描路径）/ @plugin（加载 v3 兼容 plugin）。
  2. "为什么从 JS 转 CSS"的 4 个理由：JS 配置启动慢（要 require Node module）/ 类型推导差（IDE 不能跳转 token）/ theme 最终产物是 CSS 变量、中间转换层让人困惑 / CSS 配置直接对应输出，所见即所得。
  3. @theme 是 v4 配置的核心。在 CSS 里定义 --color-brand: oklch(...)，自动生成 bg-brand / text-brand / border-brand 一套 utility。namespace 命名规则替代 v3 的 theme 配置对象。
  4. @utility 让自定义 utility 一行 CSS 搞定，且支持参数化（tab-* { tab-size: --value(--tab-size-*); } 自动生成 tab-2 / tab-4 / tab-8）。这在 v3 要写 plugin 函数。
  5. @custom-variant 让自定义变体一行 CSS 搞定（@custom-variant theme-dark (&:where([data-theme=dark] *)) 自动生成 theme-dark: 前缀）。这在 v3 也要写 plugin。
  6. @source 替代 v3 content 配置 —— 显式指定扫描路径，含 inline mode（@source inline("bg-red-500 ...") 等价 v3 safelist）和 not mode（排除路径）。
  7. 从 v3 升 v4 用 npx @tailwindcss/upgrade 自动迁移 80%+。codemod 处理 tailwind.config.js → @theme + @tailwind 指令 → @import + PostCSS plugin 配置变化。
  8. v4 兼容模式：tailwind.config.js 仍可用（@plugin "tailwindcss-v3-config" 加载），但官方建议新项目用 @theme。这给老项目留余地。
  9. CSS-first 的隐藏优势：theme 与 CSS variables 一致。@theme 里定义的 token 直接是 CSS variables（--color-brand），可以在 inline style 用、可以被 .dark 覆盖、可以被 JS 读取（getComputedStyle）。这是 v3 时代要绕弯实现的。
  10. CSS-first 心智上的转变：把 Tailwind 当 CSS 库（不是 JS-build-tool），让 CSS 的能力（cascade / variables / scoping）成为你的工具。一旦适应这个心智，v4 比 v3 直观得多。

02 底层逻辑

v3 tailwind.config.js 的痛点

看一份典型的 v3 配置：

javascript 
// tailwind.config.js (v3)
const colors = require('tailwindcss/colors');

module.exports = {
  content: [
    './src/**/*.{html,js,jsx,ts,tsx}',
    './packages/ui/**/*.{ts,tsx}',
  ],
  darkMode: 'class',
  theme: {
    extend: {
      colors: {
        brand: {
          50: '#eff6ff',
          500: '#3b82f6',
          900: '#1e3a8a',
        },
      },
      fontFamily: {
        display: ['Inter Display', 'sans-serif'],
      },
      spacing: {
        em: '1em',
        128: '32rem',
      },
      screens: {
        '3xl': '1920px',
      },
    },
  },
  plugins: [
    require('@tailwindcss/forms'),
    require('@tailwindcss/typography'),
    function ({ addUtilities }) {
      addUtilities({
        '.tab-2': { tabSize: 2 },
        '.tab-4': { tabSize: 4 },
        '.tab-8': { tabSize: 8 },
      });
    },
  ],
};

痛点：

  • 启动慢：每次 build 要 require() tailwind.config.js，跑 Node module 解析、加载 plugins
  • 类型推导差：IDE 不能从 className 跳转到 brand 定义；只能看 IntelliSense 提示
  • token 与 CSS 脱节：你定义的是 JS 对象（colors.brand[500] = '#3b82f6'），但 Tailwind 输出的是 CSS（.bg-brand-500 { background: #3b82f6 }）。中间有转换层
  • plugin 写起来繁琐：自定义 utility 要写 JS 函数（function ({ addUtilities }) {...}），不直观
  • 暗色模式配置分散：darkMode: 'class' 在 config，但 dark 变体怎么用要查文档
  • theme.extend 与 theme 区别让人困惑：extend 合并、无 extend 覆盖默认

这些痛点让 Tailwind v3 时代很多团队对 config 文件又爱又恨。

v4 CSS-first 配置 · 5 大指令

v4 用 5 个 CSS 指令替代上面整套 JS config：

1. @import "tailwindcss"

css 
/* v4 */
@import "tailwindcss";

/* 等价 v3 的：
   @tailwind base;
   @tailwind components;
   @tailwind utilities;
*/

一行 import 替代 v3 的三行 @tailwind 指令。Tailwind 知道你想要默认的 base + components + utilities 全套。

如果只想要 utilities（不要 base）：

css 
@import "tailwindcss/utilities";
/* 不含 preflight (base reset) */

2. @theme（最重要）

css 
@theme {
  --color-brand-50: oklch(0.97 0.013 240);
  --color-brand-500: oklch(0.62 0.190 240);
  --color-brand-900: oklch(0.27 0.110 240);

  --font-display: "Inter Display", sans-serif;

  --spacing-em: 1em;
  --spacing-128: 32rem;

  --breakpoint-3xl: 1920px;
}

命名规则 --{namespace}-{key} 自动映射到 utility。namespace 完整列表：

Namespace生成的 utility
--color-*bg-* / text-* / border-* / ring-* / from-* / to-* / fill-* / stroke-*
--spacing-*p-* / m-* / w-* / h-* / size-* / gap-* / inset-* / top-* ...
--font-*font-*
--text-*text-*（含 size + line-height 配对）
--breakpoint-*sm: / md: / lg: 等响应式前缀
--container-*@sm: / @md: 容器查询前缀
--radius-*rounded-*
--shadow-*shadow-*
--ease-*ease-*
--animate-*animate-*

看到这个表就明白了 —— @theme 不只是定义 CSS variables，同时是 utility 的 schema 声明。Tailwind 看到 --color-brand-500 就知道要生成 bg-brand-500 / text-brand-500 / border-brand-500 一整套 utility。

这是 v3 → v4 的核心心智差异。v3 你写 JS 对象（colors.brand[500] = '...'），v4 你直接写 CSS variable（--color-brand-500）。前者要转换、后者直接是输出。

3. @utility

自定义 utility，支持参数化：

css 
/* 简单 utility */
@utility scrollbar-thin {
  scrollbar-width: thin;
}

/* 参数化 utility（用 --value 引用 @theme 里的 token） */
@theme {
  --tab-size-2: 2;
  --tab-size-4: 4;
  --tab-size-8: 8;
}

@utility tab-* {
  tab-size: --value(--tab-size-*);
}

/* 自动生成 .tab-2 / .tab-4 / .tab-8 */

v3 同样功能要写 JS plugin（10 行函数）。v4 几行 CSS 搞定。

4. @custom-variant

css 
/* 自定义变体：当父级有 [data-theme=dark] 时应用 */
@custom-variant theme-dark (&:where([data-theme=dark] *));

/* 自动生成 theme-dark: 前缀 */
/* <div class="bg-white theme-dark:bg-black">...</div> */

v3 同样功能要写 plugin function。v4 一行 CSS。

常用自定义变体：

css 
/* 暗色（class 策略） */
@custom-variant dark (&:where(.dark, .dark *));

/* 移动设备 */
@custom-variant mobile (&:where([data-mobile=true]));

/* 受控状态 */
@custom-variant data-active (&:where([data-state=active]));

5. @source（替代 v3 content 配置）

css 
/* 显式指定扫描路径（默认扫 src/**\/* + 当前 CSS 同级文件） */
@source "./src/**/*.{html,js,jsx,ts,tsx}";
@source "./packages/ui/**/*.{ts,tsx}";

/* 排除某路径 */
@source not "./node_modules/**";

/* inline safelist */
@source inline("bg-red-500 bg-blue-500 bg-green-500");

/* 模式语法 */
@source inline("{hover:,focus:}bg-{red,blue,green}-500");

命名空间映射的 namespace 规则

@theme 里的 CSS variable 名字遵循 --{namespace}-{key} 模式。

例如：

  • --color-brand-500 → namespace = "color"，key = "brand-500" → 生成 bg-brand-500
  • --spacing-em → namespace = "spacing"，key = "em" → 生成 p-em / m-em / gap-em
  • --font-display → namespace = "font"，key = "display" → 生成 font-display
  • --breakpoint-3xl → namespace = "breakpoint"，key = "3xl" → 生成 3xl: 响应式前缀

不在标准 namespace 里的 variable 不生成 utility（仅作为普通 CSS variable）：

css 
@theme {
  --my-custom: 42px;  /* 不生成 utility，仅作为 CSS variable 可用 */
}

/* 用法 */
.my-component { padding: var(--my-custom); }

从 v3 → v4 的完整迁移路径

大部分项目跑 codemod：

bash 
npx @tailwindcss/upgrade@latest

codemod 做 5 件事：

  1. tailwind.config.js → @theme：把 theme.extend 转成 @theme 里的 CSS variables
  2. @tailwind base/components/utilities → @import "tailwindcss"
  3. PostCSS plugin 配置改：tailwindcss → @tailwindcss/postcss
  4. 个别 utility 重命名（极少）
  5. 更新 package.json 依赖：tailwindcss 升 v4，加 @tailwindcss/postcss 等

codemod 不做 的事，需手动审视：

  • 默认色调整（OKLCH 重做）—— visual regression 测试
  • plugin 兼容性 —— 检查每个 plugin 是否有 v4 兼容版本
  • 自定义 plugin 函数（addUtilities / addVariant）—— 迁到 @utility / @custom-variant
  • @apply 滥用代码 —— 这次重构是好时机一并清理（详见 P2.1）

v4 兼容模式 · tailwind.config.js 仍可用

v4 提供兼容模式给老项目：

css 
@import "tailwindcss";
@config "./tailwind.config.js";  /* 加载老 v3 config */

这让"暂时不迁 config 也能升级到 v4 引擎"成为可能。但官方建议：升级时同时迁 config，避免长期保留兼容包袱。

多文件组织 @theme

大型项目 @theme 内容多，可以拆多个文件：

css 
/* app.css */
@import "tailwindcss";
@import "./theme/colors.css";
@import "./theme/typography.css";
@import "./theme/spacing.css";

/* theme/colors.css */
@theme {
  --color-brand-50: oklch(0.97 0.013 240);
  --color-brand-500: oklch(0.62 0.19 240);
  /* ... */
}

/* theme/typography.css */
@theme {
  --font-display: "Inter Display", sans-serif;
  --font-mono: "JetBrains Mono", monospace;
}

多个 @theme 块会合并。Tailwind 处理时把所有 @theme 内容当作一个统一 token 集合。

CSS-first 的隐藏优势

v4 把 token 用 CSS variables 表达，带来 v3 时代要绕弯实现的能力：

1. token 直接在 inline style 用：

tsx 
// 动态尺寸（utility 不支持的情况）
<div style={{ width: 'calc(var(--spacing-em) * 10)' }}>...</div>

2. token 可被 .dark / .theme-x 覆盖：

css 
@theme {
  --color-bg: oklch(1 0 0);  /* 亮色默认 */
}

.dark {
  --color-bg: oklch(0.15 0 0);  /* 暗色覆盖 */
}

/* <div class="bg-bg"> 自动跟随主题 */

3. token 可被 JS 读取：

javascript 
// 在 JS 里读 Tailwind theme 值
const brandColor = getComputedStyle(document.documentElement)
  .getPropertyValue('--color-brand-500')
  .trim();
// 结果：'oklch(0.62 0.19 240)'

这些能力 v3 都要绕弯（要么把 theme export 成 JS、要么用 css-vars plugin）。v4 原生支持。

03 应用场景

场景 1 · 起新项目的最简 v4 配置

css 
/* app.css 全部内容 */
@import "tailwindcss";

@theme {
  --color-brand: oklch(0.62 0.19 240);
  --font-sans: "Inter", system-ui, sans-serif;
}

这就够了。Tailwind 默认配置（spacing scale / color palette / typography）已包含在 @import "tailwindcss" 里。

场景 2 · 从 v3 大型项目迁移

步骤：

  1. 跑 codemod：npx @tailwindcss/upgrade
  2. 检查生成的 app.css 内容（@import + @theme + @plugin）
  3. 检查 plugin 兼容性（@tailwindcss/forms / typography 在 v4 已兼容）
  4. 跑 build、跑测试
  5. 跑 visual regression（Playwright snapshot）—— 默认色微调可能影响视觉
  6. 修复颜色差异（要么团队适应、要么显式覆盖）
  7. 清理 @apply 滥用（这次重构机会）

场景 3 · 多 CSS 文件组织

大型项目把 @theme 拆分：

text 
src/
├── app.css                # 入口（@import "tailwindcss" + @import 子文件）
├── theme/
│   ├── colors.css         # @theme 颜色
│   ├── typography.css     # @theme 字体 + size
│   ├── spacing.css        # @theme 间距
│   └── components.css     # @theme 组件级 token
├── utilities/
│   ├── scrollbar.css      # @utility 自定义
│   └── tab-size.css
└── variants/
    └── theme-modes.css    # @custom-variant

04 典型代码

代码 1 · 完整 v4 配置（生产级）

css 
/* app.css */
@import "tailwindcss";

@source "./src/**/*.{html,js,jsx,ts,tsx,vue}";
@source "../../packages/ui/**/*.tsx";

@theme {
  /* 品牌色 - 11 个 shade */
  --color-brand-50:  oklch(0.97 0.013 240);
  --color-brand-100: oklch(0.94 0.026 240);
  --color-brand-200: oklch(0.87 0.060 240);
  --color-brand-300: oklch(0.78 0.105 240);
  --color-brand-400: oklch(0.70 0.150 240);
  --color-brand-500: oklch(0.62 0.190 240);
  --color-brand-600: oklch(0.54 0.190 240);
  --color-brand-700: oklch(0.46 0.170 240);
  --color-brand-800: oklch(0.38 0.140 240);
  --color-brand-900: oklch(0.30 0.110 240);
  --color-brand-950: oklch(0.20 0.080 240);

  /* 主题语义色（亮色默认） */
  --color-bg: oklch(1 0 0);
  --color-fg: oklch(0.15 0 0);
  --color-card: oklch(0.98 0 0);
  --color-border: oklch(0.90 0 0);

  /* 字体 */
  --font-sans: "Inter Variable", system-ui, sans-serif;
  --font-display: "Inter Display", "Inter Variable", system-ui, sans-serif;
  --font-mono: "JetBrains Mono", ui-monospace, monospace;

  /* 自定义 spacing */
  --spacing-em: 1em;

  /* 自定义 breakpoint */
  --breakpoint-3xl: 1920px;
}

/* 暗色覆盖（class 策略） */
.dark {
  --color-bg: oklch(0.15 0 0);
  --color-fg: oklch(0.95 0 0);
  --color-card: oklch(0.20 0 0);
  --color-border: oklch(0.30 0 0);
}

/* 自定义 variant */
@custom-variant theme-dark (&:where([data-theme=dark] *));

/* 自定义 utility */
@utility scrollbar-thin {
  scrollbar-width: thin;
  scrollbar-color: var(--color-brand-500) transparent;
}

/* 加载兼容 v3 plugin */
@plugin "@tailwindcss/forms";
@plugin "@tailwindcss/typography";

javascript 
// v3 · tailwind.config.js
module.exports = {
  content: ['./src/**/*.{html,js,ts,jsx,tsx}'],
  darkMode: 'class',
  theme: {
    extend: {
      colors: {
        brand: {
          500: '#3b82f6',
        },
      },
      spacing: {
        em: '1em',
      },
    },
  },
  plugins: [require('@tailwindcss/forms')],
};
css 
/* v4 · app.css */
@import "tailwindcss";
@source "./src/**/*.{html,js,ts,jsx,tsx}";
@plugin "@tailwindcss/forms";

@theme {
  --color-brand-500: oklch(0.62 0.19 240);  /* OKLCH 等价 #3b82f6 */
  --spacing-em: 1em;
}

/* dark mode 'class' 策略：v4 默认就是 (.dark) */
/* 不需要 darkMode: 'class' 显式配置 */

代码 3 · 自定义参数化 utility

css 
/* 自定义带参数的 tab-size utility */
@theme {
  --tab-size-2: 2;
  --tab-size-4: 4;
  --tab-size-8: 8;
}

@utility tab-* {
  tab-size: --value(--tab-size-*);
}

/* 用法
   <pre class="tab-2">...</pre>  → tab-size: 2
   <pre class="tab-4">...</pre>  → tab-size: 4
   <pre class="tab-8">...</pre>  → tab-size: 8
*/

代码 4 · 自定义变体（连锁）

css 
/* 自定义 "桌面 hover" 变体（仅在桌面设备 hover 生效） */
@custom-variant desktop-hover (&:hover:where(@media (hover: hover) and (pointer: fine) *));

/* 用法 */
/* <button class="desktop-hover:bg-brand-600">...</button> */
/* 移动端 tap 时不触发，桌面端 hover 才触发 */

/* 自定义 "data-state=open" 变体 */
@custom-variant data-open (&:where([data-state=open]));

/* 与 Radix UI 配合：
   <div data-state="open" class="data-open:translate-y-0"> */

05 常见陷阱

  • v4 项目仍写 tailwind.config.js —— 兼容模式可用，但偏离 v4 新心智。建议迁 @theme
  • @theme 写在组件级 CSS —— @theme 只能在 root（@import 的入口）。组件级 @theme 不会生效
  • "为什么 bg-brand 不工作" —— 检查 @theme 里是否定义了 --color-brand（namespace 必须是 color）。--brand 不会生成 utility
  • @theme 里写了 --color-brand 但 IntelliSense 不提示 —— 装 Tailwind VS Code extension 最新版本（v4 兼容）
  • 跨多个 CSS 文件 @theme 但 token 没合并 —— 检查 @import 顺序（@theme 必须在 @import "tailwindcss" 之后）
  • v3 plugin 在 v4 不工作 —— 检查 plugin 版本（@tailwindcss/forms / typography 等主流 plugin 已全部 v4 兼容）。第三方 plugin 看其 GitHub
  • codemod 跑了但配置乱 —— codemod 处理 80%。剩 20% 手动审视，特别是自定义 plugin 函数（要迁 @utility / @custom-variant）
  • "v4 默认色和 v3 不一样" —— OKLCH 重做的视觉差异。要么团队适应、要么 @theme 显式覆盖回 sRGB 值
  • @source 路径错 —— 用相对于 css 文件的路径。./src/**/* 是相对路径、/src/**/* 是绝对路径（不推荐）
  • @source inline 滥用 —— inline 是 safelist 替代品，给"动态 class 不能静态分析"的场景用。能完整字符串写就完整写
  • "darkMode: 'class' 我没看到 v4 文档说怎么配" —— v4 默认就是 .dark class 策略，不需要显式配置。要 media 策略用 @custom-variant 自定义
  • 团队对 CSS-first 抗拒 —— 给 1-2 周适应期。@theme 的可读性 + theme-与-CSS-一致 优势体验过就回不去
