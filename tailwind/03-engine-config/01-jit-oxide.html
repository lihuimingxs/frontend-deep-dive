JIT → Oxide 引擎演化

Tailwind 引擎 9 年的演化主线：v1-v2 时代"完整生成所有 utility（几 MB CSS） + PurgeCSS 删未用"的笨重 → v2.2-v3 JIT 引擎（扫源码、按需生成、arbitrary values 第一次成为可能）→ v4 Oxide 引擎（Rust 重写，与 SWC / Turbopack 同时代）。这一章把每代引擎的扫描算法、性能数字、动态 class 处理讲透。

阅读时长 ~30 分钟 前置 P1.1 项目史 + P2 心智 主要来源 tailwindcss.com/blog v3 / v4 release notes + Tailwind Labs YouTube + GitHub source

01 速览

  1. Tailwind 引擎 3 代演化：v1-v2.2 完整生成（穷举所有可能 utility + PurgeCSS 删）→ v2.2-v3 JIT（按需生成）→ v4 Oxide（Rust 重写 + 增量缓存）。
  2. v1-v2 时代："先生成几 MB CSS，再 PurgeCSS 删未用"。启动 5-10 秒、不能 arbitrary values、PurgeCSS 配置复杂。这是 v2.2 JIT 出现的根因。
  3. v2.2-v3 JIT 引擎反转编译模型：扫源码 → 提取 class 字符串 → 按需生成对应 CSS。完整 build 1-3 秒、增量 50-200ms、arbitrary values（w-[123px]）第一次成为可能。
  4. v4 Oxide 引擎用 Rust 重写（与 SWC / Turbopack / Rolldown 同时代的 Rust 重写浪潮）。完整 build 100-200ms（v3 的 1/5）、增量 5ms（v3 的 1/40）、新 CSS 重编译快 100×。
  5. 扫描算法的核心：不解析整个 AST（太慢）、不依赖 build pipeline（独立运行）。Tailwind 用正则 + 简单规则识别可能是 class 的字符串。误判可接受（生成 unused CSS 不严重）；漏判不可接受（缺 class）。
  6. 动态 class 是常见 bug 来源：className={`bg-${color}-500`} 这种动态拼接 JIT 扫不到（看到的是模板字符串字面量、不知道 color 取什么值）。修复方式 3 种：safelist（显式白名单）/ 完整字符串写法（color === 'red' ? 'bg-red-500' : 'bg-blue-500'）/ 变体表 lookup（colors[color] 配合完整字符串）。
  7. v4 Oxide 与 build tool 通过专用 plugin 集成：@tailwindcss/vite（推荐）/ @tailwindcss/postcss（兼容传统 PostCSS）/ @tailwindcss/webpack（v4.2 新增）/ Next.js 自带集成。
  8. Oxide 的 Rust 重写不只是性能。它让 Tailwind 与 SWC / Biome 等 Rust 工具链更深度集成（共享 AST 缓存、共享文件 watch）成为可能。这是长期价值。
  9. 从 v3 升 v4 的引擎层迁移几乎透明（用户不需要改代码）。需要改的是 build tool 配置（PostCSS plugin → Vite plugin / Webpack plugin）。
  10. 这一章是"为什么 Tailwind 编译这么快"的技术解释。理解扫描算法 + 动态 class 处理是日常 debug 的关键能力。

02 底层逻辑

v1-v2 时代 · 完整生成 + PurgeCSS

2017-2021 年的 Tailwind v1-v2 时代，编译模型是"穷举 + 删"：

  1. 启动时：根据 tailwind.config.js 生成所有可能 utility（每个 color × 每个 shade × 每个 utility 类型 × 每个 variant ...）
  2. 输出：几 MB 的 CSS 文件（含几万个 class）
  3. PurgeCSS 阶段：扫源码（HTML / JSX），找出实际用到的 class，从 CSS 删未用
  4. 最终 bundle：只剩用到的几十 KB

看一下当时 tailwind.config.js + PurgeCSS 配置：

javascript 
// v2 时代 tailwind.config.js
module.exports = {
  // PurgeCSS 配置（v2 内嵌）
  purge: [
    './src/**/*.{html,js,jsx,ts,tsx}',
    './public/index.html',
  ],
  theme: { /* ... */ },
  variants: {
    extend: {
      // 必须显式列出每个 utility 要支持的 variants
      backgroundColor: ['hover', 'focus', 'active', 'group-hover'],
      textColor: ['hover', 'focus', 'group-hover'],
      // ... 上百行
    },
  },
  plugins: [],
};

这套模型的痛点：

  • 启动慢：生成几 MB CSS 要几秒
  • 不能 arbitrary values：w-[123px] 不可能（不在穷举集合里）
  • variants 配置爆炸：每个 utility 要显式列变体，配置文件几百行
  • variants 组合受限：group-hover:focus: 这种深度组合在 v2 几乎不可能（穷举太多）
  • PurgeCSS 配置复杂：扫描漏 class、误删常用 class 是常见 bug

2021 年 v2.2 引入 JIT engine 解决这一切。

v2.2-v3 JIT 引擎 · 按需生成的反转

JIT（Just-In-Time）反转 v1-v2 编译模型：

  1. 扫源码阶段：扫所有匹配的文件（HTML / JSX / TSX / Vue / Svelte），提取所有可能是 Tailwind class 的字符串
  2. 解析阶段：对每个字符串判断"这是不是 Tailwind class"，是的话生成对应 CSS rule
  3. 输出：只含项目实际用到的 utility，几十 KB

核心是不再穷举 + 不再 PurgeCSS。生成的 CSS 一开始就只含用到的。

JIT 带来的能力：

  • arbitrary values：w-[123px] / bg-[#ff5733] / grid-cols-[repeat(3,minmax(0,1fr))] —— 因为 JIT 扫到这个 class 字符串就生成对应 CSS rule
  • variants 任意组合：group-hover:focus:dark:ring-2 —— JIT 看到就生成、不需要预先穷举
  • variants 配置极简：v3 不再需要 variants 配置（默认所有 utility 支持所有 variants）
  • 启动快：1-3 秒（v2 的 5-10 秒）
  • HMR 极快：增量 50-200ms（v2 几秒）

v2.2 JIT 是实验特性（要在 config 启用）；v3.0（2021-12）作为默认引擎。从此 Tailwind 编译模型彻底改变。

JIT 扫描算法 · 不解析 AST 的核心权衡

JIT 怎么"扫源码提取 class"？关键决策：不解析整个 JSX/HTML AST。

原因：

  • 解析 AST 慢（每个文件都要 parse）
  • 支持的语言多（HTML / JSX / Vue / Svelte / Pug / Nunjucks ...）每种 parser 都不同
  • 误判 class 不严重（生成 unused CSS，被 dead code elimination 删掉）

替代方案：正则 + 简单规则。Tailwind 的扫描器（v3 时代叫 oxide-prep，v4 时代用 Rust 重写）大致：

  1. 读文件内容
  2. 用正则匹配 "可能是 class 的字符串"（字母、数字、-、:、[、]、/ 等字符的连续序列）
  3. 对每个候选字符串，用规则判断"这是不是 Tailwind class"（看前缀、变体、值）
  4. 是的话生成对应 CSS rule

这种方式的优势：

  • 不依赖任何特定 parser、能扫所有文本文件
  • 极快（正则比 AST 快几十倍）
  • 误判生成 unused CSS 不严重（CSS 去重 + tree-shake 处理）

劣势：

  • 动态拼接 class 扫不到（核心限制，看下文）
  • 个别 false positive（生成不必要 utility，但量小）

动态 class 的根本问题

JIT 看的是"源码字符串"，不是"运行时值"。所以这种代码扫不到：

tsx 
// ❌ 扫不到
function Button({ color }: { color: 'red' | 'blue' | 'green' }) {
  return <button className={`bg-${color}-500 text-white`}>Click</button>;
}
// JIT 扫到的字符串是 "bg-${color}-500"（带模板占位符）
// 不知道 color 实际取什么值，无法生成对应 CSS
// 运行时 className 是 "bg-red-500"，但 CSS 里没有 bg-red-500 → 样式不应用

修复方式 1 · safelist（显式白名单）：

css 
/* v4 用 @source 内联 */
@source inline("bg-red-500 bg-blue-500 bg-green-500");

/* v3 用 tailwind.config.js
   safelist: ['bg-red-500', 'bg-blue-500', 'bg-green-500'],
*/

修复方式 2 · 完整字符串写法（最推荐）：

tsx 
// ✅ JIT 能扫到完整 class 字符串
function Button({ color }: { color: 'red' | 'blue' | 'green' }) {
  const colorClasses = {
    red: 'bg-red-500',
    blue: 'bg-blue-500',
    green: 'bg-green-500',
  };
  return <button className={`${colorClasses[color]} text-white`}>Click</button>;
}
// JIT 扫到 "bg-red-500" / "bg-blue-500" / "bg-green-500" 三个完整字符串
// 都生成对应 CSS

修复方式 3 · 条件三元：

tsx 
// ✅ 也行
<button className={color === 'red' ? 'bg-red-500' : color === 'blue' ? 'bg-blue-500' : 'bg-green-500'}>

团队规则：禁止动态拼 class 字符串。要么完整写出所有可能值，要么用 safelist。这是 Tailwind 项目最常见的 bug 来源之一。

v4 Oxide 引擎 · Rust 重写 + 增量缓存

v4 Oxide 引擎用 Rust 完整重写。核心动机：

  • JIT v3 是 PostCSS plugin，性能受 PostCSS 限制
  • Rust 工具链（SWC / Turbopack / Rolldown）成熟，重写时机到了
  • 增量缓存能力（task-based incremental computation）让大型项目的二次 build 极快

性能数字（基于官方 benchmark）：

场景v3 时间v4 时间提升
完整 build（中型项目）378ms100ms3.5×
增量 rebuild44ms5ms8-10×
新 CSS 添加重编译几百 ms5ms100×
大型 monorepo build5s+500ms10×+

性能数字依赖项目规模 + 配置。小项目差异不明显；中大型项目（1000+ 模块、monorepo）优势大。

Oxide 内部架构：

  • Rust 解析器：扫源码、提取 class（替代 v3 JS 实现）
  • Lightning CSS：CSS minify / vendor prefix（替代 PostCSS）
  • 增量缓存：每个文件的 class 列表缓存，未变文件不重扫
  • Vite plugin / Webpack plugin / PostCSS plugin：与不同 build tool 集成

与 build tool 集成路径

v4 提供 4 种集成方式：

1. Vite plugin（推荐）：

typescript 
// vite.config.ts
import { defineConfig } from 'vite';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  plugins: [tailwindcss()],
});

2. PostCSS plugin（兼容传统 pipeline）：

javascript 
// postcss.config.js
export default {
  plugins: {
    '@tailwindcss/postcss': {},
  },
};

3. Webpack plugin（v4.2+ 新）：

javascript 
// webpack.config.js
import { TailwindcssPlugin } from '@tailwindcss/webpack';
export default {
  plugins: [new TailwindcssPlugin()],
};

4. Next.js（零配置）：

bash 
# Next.js 16 起内置集成，npm i tailwindcss + 在 globals.css 加 @import 即可
npm install tailwindcss @tailwindcss/postcss postcss

选哪个看 build tool。Vite 项目首选 Vite plugin（最快）；老 Webpack 项目用 Webpack plugin；多 build tool 项目用 PostCSS plugin（兼容性最好）。

从 v3 升 v4 引擎层的变化

引擎层迁移用户层面几乎透明：

  • v3 项目代码 className="..." 不需要改
  • v3 的 utility 名字 v4 大多保留（个别小调整，codemod 处理）
  • v3 的 PostCSS plugin 配置在 v4 改为 @tailwindcss/postcss（codemod 处理）
  • v3 的 tailwind.config.js 在 v4 仍可用（兼容模式），但官方建议迁到 @theme

实际迁移：跑 npx @tailwindcss/upgrade。codemod 处理 80%+。剩 20% 手动审视（特别是默认色微调 + plugin 兼容）。

03 应用场景

场景 1 · 调试 "为什么 className 没生效"

排查路径：

  1. 检查 Tailwind 是否正常 build（CSS 文件是否包含期望 class）
  2. 检查动态 class（template literal 拼接？）—— 最常见原因
  3. 检查 @source 路径（是否扫描了正确的文件）
  4. 检查 class 拼写（容易拼错 bg-blue-500 写成 bg-blue500）
  5. 检查 variants（hover:bg-blue-500 在 hover 时才生效）
  6. 检查 specificity（其他 CSS 优先级覆盖）

场景 2 · 给团队定 "禁止动态 class" 规则

ESLint plugin 配合：

json 
// .eslintrc.json
{
  "plugins": ["readable-tailwind"],
  "rules": {
    "readable-tailwind/no-unnecessary-whitespace": "error",
    "readable-tailwind/multiline": "error"
  }
}

review 时 catch 动态拼接（template literal 含 ${ 在 className 里）。

场景 3 · 大型 monorepo 升 v4 看效果

v3 项目升 v4 在大型 monorepo 优势最明显：

  • Turborepo / Nx 配合 cache，Tailwind build 不必每次跑
  • 未改 package 的 build 直接命中 cache
  • Vercel remote cache 跨实例共享 build 产物
  • 整体 CI 时间下降 30-50%

04 典型代码

代码 1 · 动态 class 反例 + 修复

tsx 
// ❌ 反例 1：模板字符串拼接
function Status({ status }: { status: 'success' | 'warning' | 'error' }) {
  return <div className={`bg-${status === 'success' ? 'green' : status === 'warning' ? 'yellow' : 'red'}-100`}>
    {status}
  </div>;
}

// ✅ 修复：完整字符串
function Status({ status }: { status: 'success' | 'warning' | 'error' }) {
  const bgClasses = {
    success: 'bg-green-100',
    warning: 'bg-yellow-100',
    error: 'bg-red-100',
  };
  return <div className={bgClasses[status]}>{status}</div>;
}
tsx 
// ❌ 反例 2：拼接尺寸
function Avatar({ size }: { size: number }) {
  return <img className={`w-${size} h-${size}`} />;
}

// ✅ 修复 1：用 arbitrary value
function Avatar({ sizePx }: { sizePx: number }) {
  return <img className={`w-[${sizePx}px] h-[${sizePx}px]`} />;
  // 注意：arbitrary value 必须是字面量不能是 JS 变量
  // 实际上这个写法 JIT 也扫不到（因为 ${sizePx}px 是变量）
}

// ✅ 修复 2：用 inline style（合理使用）
function Avatar({ sizePx }: { sizePx: number }) {
  return <img className="rounded-full" style={{ width: sizePx, height: sizePx }} />;
}

// ✅ 修复 3：固定几个尺寸 → enum
function Avatar({ size = 'md' }: { size?: 'sm' | 'md' | 'lg' }) {
  const sizeClasses = { sm: 'w-8 h-8', md: 'w-12 h-12', lg: 'w-16 h-16' };
  return <img className={`${sizeClasses[size]} rounded-full`} />;
}

代码 2 · @source 显式扫描路径

css 
/* app.css */
@import "tailwindcss";

/* 默认扫 src/**\/*.{html,js,jsx,ts,tsx,vue,svelte}
   但项目可能有特殊路径 */

@source "./src/**/*.{html,js,jsx,ts,tsx}";
@source "./packages/ui/**/*.{ts,tsx}";

/* safelist：显式白名单 class */
@source inline("bg-red-500 bg-blue-500 bg-green-500");
@source inline("{hover:,focus:}bg-{red,blue,green}-500");  /* 模式语法 */

05 常见陷阱

  • 动态拼 class 字符串（最常见） —— className={`bg-${color}-500`} 不工作。用完整字符串 lookup 表
  • arbitrary value 用变量 —— w-[${size}px] 不工作（变量值不是编译时常量）。用 inline style 或固定 size enum
  • 跨文件扫描漏 —— 项目结构特殊（monorepo packages / vendored modules）需要 @source 显式指定路径
  • @source 路径错 —— 用相对于 css 文件的路径，不是项目根。注意 **/* 与 **/**/* 差异
  • 升 v4 后 PostCSS plugin 不工作 —— 改 plugin 名为 @tailwindcss/postcss（codemod 处理），不是 tailwindcss
  • Vite 项目用了 PostCSS plugin 但没装 Vite plugin 优势 —— 改用 @tailwindcss/vite，速度更快
  • 性能 benchmark 在小项目对比 —— 小项目差距不明显。Oxide 的优势在中大型项目（1000+ 模块、monorepo）
  • 升 v4 后默认色与 Figma 不一致 —— OKLCH 重做的视觉差异。要么团队适应、要么在 @theme 显式覆盖
  • safelist 滥用 —— safelist 是逃生口、不是常态。能用完整字符串就用，safelist 只给"真无法静态分析"的场景
  • JIT 慢比 v2 还慢 —— 检查是否 @source 配错（扫描了 node_modules 等大量无关文件）；检查 plugin 兼容性
  • Tailwind class 在 markdown 文件不工作 —— @source 默认不扫 .md。要扩展 @source 包含 md 文件
  • "我以为 Oxide 是 build tool" —— Oxide 是 Tailwind v4 的引擎名（与 Turbopack 等 build tool 不同概念）。Tailwind 仍通过 plugin 与 Vite/Webpack 集成
