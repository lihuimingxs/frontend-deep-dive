Utility-first 哲学 + co-location

Tailwind 反直觉的根基：class name 描述视觉效果不描述语义。这与 BEM / SMACSS 主流哲学完全相反。这一章讲清"为什么 semantic CSS 是错误抽象"，以及 co-location with markup 为什么与 React / JSX 时代天然契合。最重要的是讲清 @apply 反模式 —— 把 utility 合并成 .btn 就回到 BEM 世界、失去 Tailwind 全部价值。建立 Tailwind 心智的根基章。

阅读时长 ~25 分钟 前置 P1.1 项目史 + 写过任何 CSS 方法论（BEM / 纯 CSS / CSS-in-JS） 主要来源 Adam Wathan 2017 blog + tailwindcss.com/docs Utility-First Fundamentals + "Refactoring UI" 书

01 速览

  1. Utility-first 反直觉的核心论点：class name 不应该描述组件含义，而应该描述视觉效果。.author-bio 是 semantic class（描述含义）；.flex .p-4 .bg-white .rounded-lg 是 utility class（描述视觉）。
  2. Semantic CSS 的悖论：不同组件常常样式相同。.author-bio 和 .product-card 可能视觉 95% 一样，但 BEM 强迫你写两份 CSS。class name 与样式没有 1:1 关系。
  3. 抽象层级错位：semantic naming 假设"组件含义稳定"，但产品迭代中含义总在变。今天的 author-bio 明天可能变 user-card 后天变 testimonial。class 名跟不上含义变化。
  4. "分离关注点"是个误解：HTML 和 CSS 不是真正独立的关注点。改组件外观必然要同时改 HTML（class 名）和 CSS（样式）。所谓"分离"只是文件物理分离、不是逻辑分离。Adam Wathan 2017 blog 的核心论点。
  5. Co-location with markup：CSS 与 HTML / JSX 写在一起（不在另一个文件）。与 React / JSX "组件 = 视觉 + 行为"哲学契合。改样式不需要在两个文件之间跳，所有上下文集中在一个组件。
  6. 跨组件复用 = 组件复用，不是 class 复用。React 时代要复用样式，最自然的方式是抽 React component（<Card>），不是抽 CSS class（.card）。
  7. @apply 是 Tailwind 提供的"把 utility 合并成 semantic class"的逃生口（.btn { @apply px-4 py-2 ... }）。这是反模式 —— 你回到 BEM 世界，失去 Tailwind 全部价值。
  8. 初学者常问"HTML 不会变得很臃肿吗" —— 短期答案：是会有点。长期答案：1) prettier-plugin-tailwindcss 自动排序、2) 抽 React component、3) 看习惯了反而比 BEM 信息密度高。
  9. "那 utility 串重复怎么办"：抽 React component（最常用）/ CVA + cn 抽变体（详见 P6）。不要用 @apply。
  10. 建立 utility-first 心智需要 1-2 周。一旦建立后，看回去用 BEM 写代码反而觉得困难。这一章不读，后面所有 utility 知识都是表面。

02 底层逻辑

Semantic CSS 的悖论

看一段 BEM 风格代码：

html 
<article class="author-bio">
  <img class="author-bio__avatar" src="..." />
  <div class="author-bio__content">
    <h2 class="author-bio__name">Alice</h2>
    <p class="author-bio__description">CSS engineer</p>
  </div>
</article>

<article class="user-card">
  <img class="user-card__avatar" src="..." />
  <div class="user-card__content">
    <h2 class="user-card__name">Bob</h2>
    <p class="user-card__title">PM</p>
  </div>
</article>

这两个组件 class 名完全不同（author-bio vs user-card）。但视觉上几乎一样。CSS 写两份：

css 
.author-bio {
  display: flex; gap: 16px; padding: 16px; background: #fff; border-radius: 8px;
}
.author-bio__avatar { width: 48px; height: 48px; border-radius: 50%; }
.author-bio__name { font-size: 18px; font-weight: 600; }
.author-bio__description { color: #6b7280; font-size: 14px; }

.user-card {
  display: flex; gap: 16px; padding: 16px; background: #fff; border-radius: 8px;
}
.user-card__avatar { width: 48px; height: 48px; border-radius: 50%; }
.user-card__name { font-size: 18px; font-weight: 600; }
.user-card__title { color: #6b7280; font-size: 14px; }

问题：

  • 样式重复（DRY 失败）
  • 未来要改样式（比如圆角从 8px 改 10px），要改两处
  • 第三个类似组件（.testimonial）出现，又要重复一遍

BEM 解决方案是"抽 mixin / 抽 base class"：

scss 
// Sass mixin 风格
@mixin card-base {
  display: flex; gap: 16px; padding: 16px; background: #fff; border-radius: 8px;
}

.author-bio { @include card-base; }
.user-card { @include card-base; }
.testimonial { @include card-base; }

看似解决了，但暴露问题：你需要给 mixin 起名（card-base）。这个名字也是 semantic（描述含义），同样面临"组件含义变化时怎么办"的问题。

Tailwind 的回答：不要纠结 class 命名。直接 utility：

html 
<article class="flex gap-4 p-4 bg-white rounded-lg">
  <img class="w-12 h-12 rounded-full" src="..." />
  <div>
    <h2 class="text-lg font-semibold">Alice</h2>
    <p class="text-gray-500 text-sm">CSS engineer</p>
  </div>
</article>

你看到的 class 串就是这个组件的视觉效果。不需要查 CSS 文件验证。改样式就改 class。

"那如果有 10 个相似组件？" → 抽 React component（看下文）。

抽象层级错位

semantic naming 的另一个问题：组件名跟不上产品迭代。

看一个真实场景。你早期项目里有：

html 
<div class="author-bio">...</div>

3 个月后，PM 要把这个组件改造为"通用用户卡片"，可以用于：

  • 作者简介（原来用例）
  • 评论者头像 + 名字
  • 团队成员列表
  • 提及（@mention）popup

这时候 .author-bio 类名变得反讽 —— 你在用一个名为"作者简介"的 class 显示团队成员。但改 class 名意味着：

  • HTML 改
  • CSS 改
  • 所有引用此组件的地方改
  • git history 变得难追

实际项目中大家常常"将就"，留着错误命名。半年后接手项目的人对 .author-bio 出现在团队列表里完全困惑。

Tailwind 的回答：class 描述视觉、不描述含义。.flex.gap-4.p-4 在所有场景里都正确。组件名变化（React component 名 <AuthorBio> 改 <UserCard>）只影响 React，不影响 CSS。

"分离关注点"是个误解

反对 utility-first 的常见论点："HTML 是结构、CSS 是样式，应该分离"。

Adam Wathan 2017 blog 反驳：HTML 和 CSS 不是真正独立的关注点。改组件外观（关注点 = "视觉"）必然要同时：

  • 改 HTML（添加 class、改结构）
  • 改 CSS（定义样式）

所谓"separation of concerns"只是文件物理分离（HTML 在 .html、CSS 在 .css），不是逻辑分离。逻辑层面"组件视觉"是一个完整的关注点。

同理 React 时代："JSX = HTML + JS 混合"被早期反对者批评违反 SoC。但实际证明：组件即"视觉 + 行为"的完整单元，文件物理分离反而妨碍开发。

Tailwind 是把这个思想推到 CSS：组件即"视觉 + 行为 + 样式"的完整单元，class 直接写在 markup 旁边。

Co-location with markup · React/JSX 时代的天然契合

Tailwind 在 2021-2024 年爆火的根本原因：与 React / JSX 哲学高度契合。

React 组件的本质：

  • HTML 结构（JSX）
  • JS 行为（hooks / event handlers）
  • 样式（className）

这三件事在一个组件文件里完整描述。Tailwind 让"样式"也用 className 表达，不需要单独的 .css 文件。改样式 = 改 className 字符串。改组件 = 改这一个文件。

看实际差异。BEM + 单独 CSS 文件方式：

tsx 
// AuthorBio.tsx
import './AuthorBio.css';

export function AuthorBio({ name, description }) {
  return (
    <article className="author-bio">
      <img className="author-bio__avatar" src="..." />
      <div className="author-bio__content">
        <h2 className="author-bio__name">{name}</h2>
        <p className="author-bio__description">{description}</p>
      </div>
    </article>
  );
}

// AuthorBio.css
.author-bio { /* ... */ }
.author-bio__avatar { /* ... */ }
.author-bio__name { /* ... */ }
.author-bio__description { /* ... */ }

改头像尺寸要：

  1. 打开 AuthorBio.tsx 看 class 名
  2. 切到 AuthorBio.css 找 .author-bio__avatar
  3. 改 width / height
  4. 切回 AuthorBio.tsx 看效果

Tailwind co-location 方式：

tsx 
// AuthorBio.tsx
export function AuthorBio({ name, description }) {
  return (
    <article className="flex gap-4 p-4 bg-white rounded-lg">
      <img className="w-12 h-12 rounded-full" src="..." />
      <div>
        <h2 className="text-lg font-semibold">{name}</h2>
        <p className="text-gray-500 text-sm">{description}</p>
      </div>
    </article>
  );
}

改头像尺寸：把 w-12 h-12 改成 w-16 h-16。一处。

这个生产力差异在大型项目（每天改十几次样式）累积成显著优势。

跨组件复用 = 组件复用，不是 class 复用

Utility-first 反对者最常问："那如果 5 个组件样式相同，每个都写一遍 utility 串吗？"

答案：不写 5 遍，但不是抽 class，是抽 React component。

tsx 
// 抽组件方式（推荐）
function Card({ children, className }: { children: React.ReactNode; className?: string }) {
  return (
    <article className={`flex gap-4 p-4 bg-white rounded-lg ${className || ''}`}>
      {children}
    </article>
  );
}

// 5 个使用方
function AuthorBio({ name }) {
  return (
    <Card>
      <img className="w-12 h-12 rounded-full" src="..." />
      <h2 className="text-lg font-semibold">{name}</h2>
    </Card>
  );
}

function UserCard({ user }) {
  return (
    <Card>
      <Avatar user={user} />
      <UserName user={user} />
    </Card>
  );
}
// ... 其他 3 个

关键：组件即样式复用单元。改 Card 样式只改一处（Card 组件内的 className），影响所有 5 个使用方。

对比 BEM 方式 "抽 .card class"：

  • BEM：要起名 .card、定义 CSS、记得在 5 个 HTML 加 class
  • Tailwind：抽 <Card> 组件，5 个使用方自然 import

组件复用比 class 复用更稳健 —— 组件可以传 props（变体）、可以包含 children、可以 useEffect。class 只能控制视觉。

@apply 反模式（最重要的反模式）

Tailwind 提供 @apply 指令，可以把 utility 合并成 semantic class：

css 
/* 反模式：用 @apply 把 utility 合并成 .btn */
.btn {
  @apply px-4 py-2 bg-blue-500 hover:bg-blue-600 text-white rounded-md;
}
.btn-secondary {
  @apply px-4 py-2 bg-gray-200 hover:bg-gray-300 text-gray-900 rounded-md;
}

表面看起来更"干净"，但这是反模式。理由：

  1. 你回到 BEM 世界：又要给每个组件起 semantic 名（.btn / .btn-secondary / .btn-icon ...）。utility-first 全部价值被否定
  2. JSX 与 CSS 重新分离：改 .btn 样式要回 .css 文件、不再 co-location
  3. 失去 utility 的视觉直观性：看 className="btn" 不知道这个 button 长啥样
  4. 跨组件复用反而麻烦：你用 .btn 来 style Button 组件，但其他组件想要相同样式必须用 className="btn" —— class 名传染
  5. 浪费 Tailwind 引擎：Tailwind 设计上是"扫源码、按需生成 utility"。@apply 让你又回到"自己写 CSS class"

正确做法是抽 React component：

tsx 
// 正确：抽组件
function Button({ children, variant = 'primary', ...props }: { ... }) {
  const variantClasses = {
    primary: 'bg-blue-500 hover:bg-blue-600 text-white',
    secondary: 'bg-gray-200 hover:bg-gray-300 text-gray-900',
  };
  return (
    <button
      className={`px-4 py-2 rounded-md ${variantClasses[variant]}`}
      {...props}
    >
      {children}
    </button>
  );
}

// 使用方
<Button variant="primary">Save</Button>
<Button variant="secondary">Cancel</Button>

更进阶用 CVA + cn（详见 P6）。

Adam Wathan 在多个公开演讲中明确说：@apply 是给"无法改 HTML"的场景用的逃生口（如 markdown 渲染输出 / WordPress 主题），日常项目不要用。但很多团队把它当成"让 HTML 看起来干净"的工具，就是反模式。

初学者的"HTML 看起来很丑"问题

这是最常见的初学者反对意见。承认：

tsx 
<article className="flex flex-col md:flex-row gap-4 md:gap-6 p-4 md:p-6 bg-white dark:bg-gray-900 rounded-lg shadow-sm hover:shadow-md transition-shadow border border-gray-200 dark:border-gray-700">
  ...
</article>

这一行确实很长。但解决方法不是 @apply，是工具：

  1. prettier-plugin-tailwindcss：自动按推荐顺序排序 class（layout → spacing → color → state），可读性提升一档
  2. 抽 React component：把这堆 class 抽成 <Card>，使用方只看 <Card>...</Card>
  3. CVA + cn（详见 P6）：变体级别的抽取
  4. 编辑器配合：VS Code 的 word wrap + Tailwind IntelliSense 让长行可阅读
  5. 团队习惯：用 1-2 周适应；之后看 BEM 的多文件跳转反而觉得繁琐

"丑"是表面观感，工程上 Tailwind 的优势远大于代价。

03 应用场景

场景 1 · 从 BEM 项目迁 Tailwind 的心智转变

常见心理障碍 + 应对：

  • "我要给每个组件起 class 名" → 不需要，用 utility 串
  • "重复样式怎么办" → 抽 React component，不抽 class
  • "HTML 太长" → prettier-plugin 排序、抽组件、习惯
  • "找设计师 review CSS 怎么办" → 设计师可以读 utility（p-4 = 16px padding，bg-blue-500 = 蓝色）；其实比 BEM 更直观
  • "我团队反对" → 让团队读 Adam Wathan 2017 blog；用 1 个小项目验证；让结果说话

场景 2 · 团队评审看到 @apply 滥用

Code review 时看到：

css 
.btn { @apply px-4 py-2 bg-blue-500 ...; }
.card { @apply flex gap-4 p-4 ...; }
.input { @apply px-3 py-2 border ...; }

沟通：

  • 解释 @apply 是反模式（参考本节）
  • 给出抽 React component 的例子
  • 引用官方文档（Tailwind 团队公开建议）
  • 渐进重构：新代码不用 @apply、老 @apply 逐步迁组件

场景 3 · "为什么我们不用 styled-components 写 BEM 风格"

有人提出 styled-components 也能 co-location，且支持 props 变体。回答：

  • styled-components runtime 模式与 RSC 不兼容（详见 P1.2）
  • styled-components 不解决"约束"问题（详见 P2.2）
  • styled-components 仍要给 styled component 起名（StyledCard / StyledButton），又是 semantic naming
  • Tailwind co-location + utility 是更彻底的方案

04 典型代码

代码 1 · 同一个 card 用 BEM、CSS Modules、Tailwind 三种写法

html 
<!-- 1. BEM -->
<article class="card card--featured">
  <img class="card__avatar" />
  <div class="card__content">
    <h2 class="card__name">Alice</h2>
    <p class="card__description">CSS engineer</p>
  </div>
</article>
<style>
  .card { display: flex; gap: 16px; padding: 16px; ... }
  .card--featured { border: 2px solid #3b82f6; }
  .card__avatar { width: 48px; height: 48px; ... }
  .card__name { font-size: 18px; ... }
  .card__description { color: #6b7280; ... }
</style>
tsx 
// 2. CSS Modules
import styles from './Card.module.css';

function Card({ featured, name, description }) {
  return (
    <article className={`${styles.card} ${featured ? styles['card--featured'] : ''}`}>
      <img className={styles.avatar} />
      <div>
        <h2 className={styles.name}>{name}</h2>
        <p className={styles.description}>{description}</p>
      </div>
    </article>
  );
}
tsx 
// 3. Tailwind（推荐）
import { cn } from '@/lib/utils';

function Card({ featured, name, description }) {
  return (
    <article className={cn(
      'flex gap-4 p-4 bg-white rounded-lg',
      featured && 'border-2 border-blue-500',
    )}>
      <img className="w-12 h-12 rounded-full" />
      <div>
        <h2 className="text-lg font-semibold">{name}</h2>
        <p className="text-gray-500 text-sm">{description}</p>
      </div>
    </article>
  );
}

代码 2 · @apply 反模式 → 修复（抽组件）

css 
/* ❌ 反模式 */
.btn-primary { @apply px-4 py-2 bg-blue-500 hover:bg-blue-600 text-white rounded-md; }
.btn-secondary { @apply px-4 py-2 bg-gray-200 hover:bg-gray-300 text-gray-900 rounded-md; }
.btn-ghost { @apply px-4 py-2 hover:bg-gray-100 text-gray-900 rounded-md; }
tsx 
// ✅ 修复：抽 React component
function Button({
  variant = 'primary',
  children,
  ...props
}: React.ButtonHTMLAttributes<HTMLButtonElement> & { variant?: 'primary' | 'secondary' | 'ghost' }) {
  const variantClasses = {
    primary: 'bg-blue-500 hover:bg-blue-600 text-white',
    secondary: 'bg-gray-200 hover:bg-gray-300 text-gray-900',
    ghost: 'hover:bg-gray-100 text-gray-900',
  };
  return (
    <button
      className={`px-4 py-2 rounded-md ${variantClasses[variant]}`}
      {...props}
    >
      {children}
    </button>
  );
}

// 使用
<Button variant="primary">Save</Button>
<Button variant="ghost">Cancel</Button>

代码 3 · 跨组件复用（组件复用而非 class 复用）

tsx 
// 抽 Card 组件
function Card({ children, className }: { children: React.ReactNode; className?: string }) {
  return (
    <article className={cn(
      'rounded-lg border border-gray-200 bg-white p-4 shadow-sm dark:bg-gray-900 dark:border-gray-700',
      className,
    )}>
      {children}
    </article>
  );
}

// 多个使用场景
function AuthorBio({ author }) {
  return <Card>{/* author specific content */}</Card>;
}
function ProductCard({ product }) {
  return <Card>{/* product specific content */}</Card>;
}
function NotificationCard({ notification }) {
  return <Card className="border-yellow-300">{/* notification content */}</Card>;
}

05 常见陷阱

  • 用 @apply 让 HTML"干净" —— 反模式（详见本章）。把 utility 串抽 React component，不抽 class
  • 给 React component 起 class 名再用 @apply —— 双重错误。React 组件已经是抽象层，不需要再加 class 抽象
  • "我们用 .btn 是因为设计师要 .btn 名字" —— 设计师不需要看 class 名。Figma 里组件名是 Button、React 里组件名是 Button、CSS 里不需要再有 .btn。
  • 纠结 Tailwind class 排序 —— prettier-plugin-tailwindcss 自动排序，团队不需要手动维护
  • "Tailwind class 与设计 token 脱节" —— Tailwind 的 utility scale 就是设计 token（详见 P2.2）。@theme 让 Figma token 同步到 Tailwind token
  • "BEM 项目我不能直接迁 Tailwind 太混乱" —— 渐进迁移：新组件用 Tailwind、老组件保留 BEM 直到自然重写。两套可共存
  • "我团队多年 BEM，不会接受" —— 用一个小新项目验证、让团队 1-2 周内尝试。多数人会在 2-3 周内转变看法
  • 认为 utility-first 等于"反对所有抽象" —— Tailwind 的抽象层在组件（React component）和 token（spacing scale / color palette），不在 class 名
  • "co-location 让 Git diff 大" —— 改样式 diff 在 component 文件；不改样式 diff 不动。BEM 改样式 diff 在 .css 文件 + .html 文件，本质工作量没差
  • 看到一个 Tailwind class 串 100 字符就以为 Tailwind 都这样 —— 那是没用 prettier-plugin / 没抽组件的代码。正确写法 className 一般 30-80 字符
  • 用 Tailwind 但仍以 .css 文件为主、JSX 为辅 —— 心智没跟上。Tailwind 项目的 .css 文件应该极短（只有 @theme、global reset、特殊 utility），不是组件样式
