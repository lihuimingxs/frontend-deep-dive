设计 token 与约束系统

Tailwind 不是"反正你想要什么 utility 就生成什么"。它的精髓在约束 —— 默认有限的 spacing scale（4px 基数）、OKLCH 调色板（24 色家族）、typography scale、breakpoints。约束让团队不纠结"是 14px 还是 15px"、视觉跨页面一致、design system 即工具自身。这一章把每套 token 讲透，再讲 @theme 扩展、与 Figma 同步、多品牌 CSS variables 的实战。

阅读时长 ~25 分钟 前置 P2.1 utility-first 心智 主要来源 tailwindcss.com/docs Theme + "Refactoring UI" 书 + OKLCH spec

01 速览

  1. Tailwind 默认 token 系统包含 6 类：spacing scale（间距）/ color palette（颜色）/ typography（字号 + 行高）/ breakpoints（响应断点）/ z-index（层级）/ border-radius（圆角）。每套都基于工程 + 设计经验沉淀的最佳实践。
  2. Spacing scale 基于 4px 基数：p-0 = 0 / p-0.5 = 2px / p-1 = 4px / p-2 = 8px / p-4 = 16px / p-6 = 24px / p-8 = 32px / p-12 = 48px / p-16 = 64px ... 不让你写 padding: 13px。
  3. v4 默认调色板用 OKLCH 色彩空间 重做，24 个色家族 × 11 个 shade（50/100/.../900/950）。OKLCH 的优势：perceptually uniform（亮度跨色相一致）、wide gamut（P3 / Rec.2020 支持）。这是 v4 的重大变化。
  4. Typography scale：text-xs/sm/base/lg/xl/2xl/3xl/4xl/5xl/6xl/7xl/8xl/9xl，每个 size 配对 line-height（base = 1rem font-size + 1.5rem line-height 等）。不需要单独设 line-height，scale 自动配对。
  5. Breakpoints（v4）：sm 640px / md 768px / lg 1024px / xl 1280px / 2xl 1536px。Mobile-first（默认是最小，prefix 应用更大）。容器 query（@sm:）用相同断点。
  6. Z-index 默认 6 个层级：z-0 / z-10 / z-20 / z-30 / z-40 / z-50 + z-auto。10 的步长留扩展空间。
  7. Border-radius 7 个尺度：none/sm/md/lg/xl/2xl/3xl/full。rounded-lg = 8px、rounded-2xl = 16px、rounded-full = 9999px（圆形）。
  8. 约束反而是优势：设计师不纠结 13px / 14px / 15px、团队全用同一套 spacing、改主题只改 token。"约束让创造力聚焦于真正重要的"是 Tailwind 的核心理念之一。
  9. v4 用 @theme 扩展 token：--color-brand: oklch(...) 自动生成 bg-brand / text-brand / border-brand 一套 utility。多品牌通过 CSS variables 覆盖（在 root / 不同 .theme-x class 设不同值）。
  10. 与 Figma 对接：Tokens Studio plugin（最流行）/ Figma Variables → JSON → @theme（自建脚本）/ 反向同步（少见）。设计师在 Figma 改 token，自动同步到代码。这是 Tailwind 时代 design system 的标准工作流。

02 底层逻辑

Spacing Scale · 4px 基数的根因

Tailwind 的 spacing scale 基于 4px 基数（rem 表示是 0.25rem）：

Class 值 常见用法
p-00无 padding
p-0.52px极小间隙
p-14pxicon padding
p-1.56px小间隙
p-28px常用 button 内边距
p-312px表单 input padding
p-416px卡片标准 padding
p-520px稍大间隙
p-624px大卡片 padding
p-832pxsection 间距
p-1248px大 section
p-1664px极大区块
p-2496px页面级

为什么是 4px？这是 design system 工程实践的经验沉淀：

  • 视觉对齐：4px 是大多数屏幕的 1 字符高度（12px = 3 × 4）的因子，对齐文字 baseline 自然
  • 团队一致性：4px 基数让设计师和工程师不需要纠结"是 13px 还是 14px"，强制选 12 / 16
  • 性能：rem-based scale 让全局缩放（root font-size 改）按比例工作
  • 响应式：4px 在不同设备 DPI 下都能渲染清晰

Material Design 用 8px 基数（步长更大）；Apple HIG 用 4px 基数（与 Tailwind 一致）。Tailwind 选 4px 是与 Apple 设计语言对齐 + 设计师社区主流共识。

Spacing scale 不是只用于 padding，所有间距 utility 都用同一 scale：

  • m-4（margin）/ p-4（padding）/ gap-4（flex gap）/ space-x-4（兄弟间距）
  • w-4（width）/ h-4（height）/ size-4（width + height，v4 新）
  • top-4 / left-4 / inset-4（位置）

这种"一套 scale 处处用"是约束的核心。一个组件 padding-4 + gap-4 + width-12 自动视觉对齐。

Color Palette · OKLCH 重做（v4）

Tailwind v4 默认调色板用 OKLCH 色彩空间重做。核心特性：

1. 24 个色家族 × 11 个 shade：

  • 中性：slate / gray / zinc / neutral / stone（5 个灰色家族）
  • 暖色：red / orange / amber / yellow（4 个）
  • 冷色：lime / green / emerald / teal / cyan / sky / blue / indigo / violet / purple / fuchsia / pink / rose（13 个）
  • v4.2 新加：mauve / olive / mist / taupe（4 个中性补充）

每个家族 11 个 shade：50（最浅）/ 100 / 200 / 300 / 400 / 500（基准）/ 600 / 700 / 800 / 900 / 950（最深）。

2. OKLCH 色彩空间：

css 
/* v3 用 sRGB hex */
--color-blue-500: #3b82f6;

/* v4 用 OKLCH */
--color-blue-500: oklch(0.62 0.19 240);
/* 0.62 = 亮度 (Lightness, 0-1)
   0.19 = 饱和度 (Chroma, 0-0.4 左右)
   240  = 色相 (Hue, 0-360)
*/

OKLCH 的工程优势：

  • Perceptually uniform：blue-500 和 red-500 视觉亮度一致（sRGB 下不同色相的同 lightness 看起来不一样）
  • Wide gamut：支持 P3 / Rec.2020（更宽广的色域，新 Mac / iPhone 屏幕能显示更鲜艳的颜色）
  • 更好的 mix / lighten / darken：OKLCH 数学计算比 HSL / sRGB 更接近人眼感知
  • 更平滑的渐变：在两个 OKLCH 颜色间渐变不会出现"灰色中间带"（HSL 的常见问题）

v3 → v4 升级时，默认色微调（同样 blue-500 视觉略有差异）。需要 visual regression 测试。如果团队不接受默认色调整，可以在 @theme 显式覆盖回 v3 sRGB 值。

3. opacity 修饰符：

html 
<!-- bg-blue-500 with 50% opacity -->
<div class="bg-blue-500/50">...</div>
<!-- 任意 opacity -->
<div class="bg-blue-500/[0.37]">...</div>

底层用 CSS color-mix() 函数（详见 P4）。bg-blue-500/50 编译为 color-mix(in oklab, var(--color-blue-500) 50%, transparent)。

Breakpoints · Mobile-first 哲学

Prefix Min-width 设备
（无）0mobile（默认）
sm:640px大屏手机 / 小平板
md:768pxiPad portrait
lg:1024pxiPad landscape / 小笔记本
xl:1280px桌面
2xl:1536px大显示器

Mobile-first 心智：默认 class 是最小尺寸样式，prefix 应用更大尺寸：

html 
<div class="text-base md:text-lg lg:text-xl">
  <!-- mobile: 16px / md+: 18px / lg+: 20px -->
</div>

这与传统 desktop-first 媒体查询相反（desktop-first 是默认大屏样式、media query 应用更小屏）。Mobile-first 让"渐进增强"成为默认：基础体验所有设备都有，更大屏幕加更多。详见 P5.1。

Z-index · 6 层 + auto

  • z-0 / z-10 / z-20 / z-30 / z-40 / z-50
  • z-auto（默认 inherit）

步长 10 留扩展空间（项目可以扩 z-15 / z-25 等 arbitrary）。

常见用法：

  • z-0：默认
  • z-10：sticky header / floating button
  • z-20：dropdown / tooltip
  • z-30：modal / dialog
  • z-40：popover / 通知
  • z-50：toast / 全屏 modal

不要写 z-9999。约束让团队不打 z-index 战争。

v4 @theme · 扩展 token 的 CSS-first 方式

v4 的 @theme 指令让你在 CSS 里定义 token，自动生成对应 utility：

css 
@import "tailwindcss";

@theme {
  /* 自定义品牌色 */
  --color-brand-50: oklch(0.97 0.013 240);
  --color-brand-500: oklch(0.62 0.19 240);
  --color-brand-900: oklch(0.27 0.10 240);

  /* 自定义字体 */
  --font-display: "Inter Display", sans-serif;
  --font-mono: "JetBrains Mono", ui-monospace, monospace;

  /* 自定义 spacing */
  --spacing-em: 1em;
  --spacing-128: 32rem; /* 512px */

  /* 自定义 breakpoint */
  --breakpoint-3xl: 1920px;
}

/* 自动生成的 utility：
   .bg-brand-50 / .text-brand-500 / .border-brand-900
   .font-display / .font-mono
   .p-em / .p-128 / .gap-em
   .3xl:flex
*/

命名规则：--{namespace}-{key} 自动映射到 utility。namespace 包括：

  • --color-* → bg-* / text-* / border-* / ring-*
  • --spacing-* → p-* / m-* / w-* / h-* / gap-* / inset-*
  • --font-* → font-*
  • --text-* → text-*（含 size + line-height 配对）
  • --breakpoint-* → sm: / md: ...
  • --radius-* → rounded-*

多品牌 · CSS variables 覆盖

大型项目可能需要多品牌（同一应用支持多个客户的 white-label）。通过 CSS variables 在不同 root 类设不同值：

css 
@theme {
  --color-brand: oklch(0.62 0.19 240);  /* 默认蓝 */
}

/* 红色品牌 */
.theme-red {
  --color-brand: oklch(0.62 0.21 30);
}

/* 绿色品牌 */
.theme-green {
  --color-brand: oklch(0.62 0.18 145);
}
html 
<html class="theme-red">
  <body>
    <button class="bg-brand text-white">Buy now</button>
    <!-- 自动用红色，因为 .theme-red 覆盖了 --color-brand -->
  </body>
</html>

Figma 与 Tailwind token 同步

2026 年的标准工作流：

  1. Figma 里建 Variables（颜色 / spacing / typography token）
  2. Tokens Studio plugin 把 Figma Variables 导出 JSON
  3. 自建脚本 把 JSON 转 @theme CSS
  4. 提交到 Git，CI / Vercel 自动 build

这让设计师 Figma 的 token 改动几小时内反映到生产。Tailwind 时代 design system 工作流的标杆。

反向同步（代码 → Figma）也存在但少见，因为通常设计师驱动 token 变化。

03 应用场景

场景 1 · 起新项目时定义品牌色

步骤：

  1. 设计师在 Figma 给品牌色（11 个 shade）
  2. 用 OKLCH 表达（让设计师习惯 OKLCH，比 HSL/sRGB 更可控）
  3. 在 @theme 加 --color-brand-50 到 --color-brand-950
  4. 验证 utility 生成（bg-brand-500 等可用）
  5. 建 Button / Card 组件用品牌色

场景 2 · 团队不熟悉 OKLCH 怎么办

给团队 1 小时讲座：

  • OKLCH = Lightness + Chroma + Hue（与 HSL 类似）
  • 提供 oklch.com 在线工具（粘贴 hex 看 OKLCH 值）
  • 设计师工具：Figma 已支持 OKLCH 输入
  • 提供 sRGB → OKLCH 转换函数（codemod / 在线工具）

2-3 周团队就习惯了。

场景 3 · 多品牌应用

SaaS 平台支持 5 个客户 white-label，每个客户自己的色：

tsx 
// app/layout.tsx
async function RootLayout({ children, params }) {

  const themeClass = `theme-${tenant.id}`;
  return (
    <html className={themeClass}>

代码 2 · 用 Tokens Studio 同步 Figma

json 
// tokens-studio export 出的 JSON（部分）
{
  "color": {
    "brand": {
      "50":  { "value": "oklch(0.97 0.013 240)" },
      "500": { "value": "oklch(0.62 0.19 240)" },
      "950": { "value": "oklch(0.20 0.08 240)" }
    }
  },
  "spacing": {
    "card": { "value": "16px" }
  }
}
javascript 
// scripts/tokens-to-theme.js
import fs from 'fs';
const tokens = JSON.parse(fs.readFileSync('design-tokens.json', 'utf-8'));

let css = '@theme {\n';
for (const [shade, { value }] of Object.entries(tokens.color.brand)) {
  css += `  --color-brand-${shade}: ${value};\n`;
}
for (const [name, { value }] of Object.entries(tokens.spacing)) {
  css += `  --spacing-${name}: ${value};\n`;
}
css += '}\n';

fs.writeFileSync('src/theme.css', css);
console.log('✅ Generated src/theme.css from design-tokens.json');

CI 配置：每次 design-tokens.json 改动 → 跑 node scripts/tokens-to-theme.js → commit theme.css → 部署。

代码 3 · 暗色模式 token 切换

css 
@theme {
  /* 默认（亮色）token */
  --color-bg: oklch(1 0 0);          /* white */
  --color-text: oklch(0.15 0 0);     /* near black */
  --color-card: oklch(0.98 0 0);     /* off-white */
  --color-border: oklch(0.90 0 0);   /* light gray */
}

/* 暗色覆盖 */
.dark {
  --color-bg: oklch(0.15 0 0);       /* near black */
  --color-text: oklch(0.95 0 0);     /* off-white */
  --color-card: oklch(0.20 0 0);
  --color-border: oklch(0.30 0 0);
}
html 
<!-- 用 utility，token 自动跟随 -->
<div class="bg-bg text-text">
  <div class="bg-card border border-border p-4">...</div>
</div>

05 常见陷阱

  • 不熟悉 4px 基数，常用 arbitrary value 写 13px / 15px —— 大多数情况应该接受 12 或 16，强制设计对齐 4px scale。少数特殊情况用 arbitrary，不要滥用
  • v4 默认色比 v3 鲜艳了，设计师抱怨 —— OKLCH 调色板视觉差异。要么团队适应（推荐），要么在 @theme 显式覆盖回 v3 sRGB 值
  • 不知道 OKLCH 怎么读 —— L 是亮度（0-1）/ C 是饱和度（0-0.4）/ H 是色相（0-360）。oklch(0.62 0.19 240) ≈ 中等亮度 + 高饱和的蓝
  • 给 Figma 设计师传 hex 让 ta 转 OKLCH —— 给设计师 OKLCH 工具（oklch.com / Figma 内置 OKLCH 输入），让 ta 自己用 OKLCH 思维设计
  • 多品牌用 utility 而不是 CSS variables —— 错。多品牌应该用 CSS variables 让 token 切换；不应该 5 个品牌 build 5 套 utility
  • Z-index 写 9999 / 99999 —— Tailwind 默认 z-50 已经是最高。不要打 z-index 战争，用 portal / z-stack 重新设计
  • 把 @theme 写在每个组件 css 里 —— @theme 应该在 root（app.css），不是组件级。组件级 @theme 不会生效
  • v3 项目 tailwind.config.js 的 theme 不知道怎么迁 v4 —— 跑 codemod（npx @tailwindcss/upgrade）。手动迁也简单：theme.extend.colors → @theme --color-* 等
  • typography scale 改 line-height 单独写 —— 大多数情况不需要，scale 自带 line-height 配对。除非真的需要 leading-snug 这种特殊调整
  • 用 arbitrary 改字号 text-[15px] —— scale 里没 15px 通常意味着设计需要重新审视。强制选 14（text-sm）或 16（text-base）
  • Figma → @theme 同步出错没人发现 —— CI 里跑 visual regression（Playwright snapshot），token 变化导致视觉变化第一时间能 catch
  • "我们不需要 design token，直接 hex" —— 短期省事，长期亏。design token 是项目超过 10 个组件后必备
