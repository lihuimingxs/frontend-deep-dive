P4 · DOM 测试

单元测试 + 集成测试中最大的一块：组件 / DOM 测试。本阶段讲 Testing Library 的"按用户视角查找"哲学落地，以及 jsdom / happy-dom / Vitest browser mode 三种 DOM 实现的权衡。

章节数 2 主要来源 testing-library.com / Kent Dodds blog / jsdom & happy-dom 源码 状态 ⏳ 待写作

本阶段覆盖

  • 跨框架统一 API· @testing-library/react / vue / svelte 共享 @testing-library/dom 内核
  • query 优先级· getByRole > getByLabelText > getByText > getByTestId（accessibility 优先）
  • userEvent vs fireEvent· 真实用户操作链 vs 底层 DOM event
  • async query· findBy / waitFor / 何时 await 何时不
  • 常见反模式· setTimeout / className 查找 / 测组件 props 传递
  • jest-dom matcher· toBeInTheDocument / toHaveTextContent / toBeDisabled
  • jsdom（2010+）· 完整 DOM Level 4 + HTML5 spec；纯 JS；慢但兼容好
  • happy-dom（2020+）· 比 jsdom 快 3-5x；spec 覆盖不全；Vitest 默认推荐
  • Vitest browser mode（2024 GA）· 真浏览器 via Playwright；CSS / Canvas 准确
  • 选型矩阵· happy-dom 60% / jsdom 25% / browser mode 15%
