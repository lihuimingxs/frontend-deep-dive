jsdom vs happy-dom vs browser mode

DOM 测试需要某种 DOM 实现。三种选择各有权衡：jsdom（2010+ 元老，spec 严格但慢）/ happy-dom（2020+ 后起之秀，快 3-5x 但 spec 不全）/ Vitest browser mode（真浏览器，2024 GA）。本章讲清三者的实现差异、什么时候用哪个、以及 2026 年实测分布。

阅读时长 ~20 分钟 前置 P4.1（Testing Library） 主要来源 jsdom 与 happy-dom 源码 / Vitest browser mode 文档

01 速览

  1. jsdom——2010+ 元老。完整模拟 DOM Level 4 + HTML5 spec。慢但 spec 兼容性最好。Jest 默认。不计算 layout（offsetWidth 等始终 0）。
  2. happy-dom——2020+ 后起之秀。性能比 jsdom 快 3-5x。spec 覆盖不全（部分边界 case 行为不一致）。Vitest 默认推荐。同样不计算 layout。
  3. Vitest browser mode——2024 GA。真浏览器（Chromium / Firefox / WebKit via Playwright）。真 layout / 真 CSS / 真事件。慢于 jsdom 但快于完整 e2e。
  4. 选型——业务组件 → happy-dom（默认）；spec 严格 case → jsdom；测真实 layout / Canvas / CSS → browser mode。2026 年实测分布：happy-dom 60% / jsdom 25% / browser mode 15%。
  5. 切换零成本——Vitest 配置 environment: 'jsdom' / 'happy-dom' / 'browser' 即可。测试代码不感知（Testing Library 自动适配）。
  6. jsdom 不支持的——CSS layout（offsetWidth/getBoundingClientRect）/ Canvas 渲染 / WebGL / animation 实际触发 / 真 CSS 计算（getComputedStyle 仅看 inline + 链接的 stylesheet）。
  7. happy-dom 与 jsdom 不一致点——少数 spec 边界 case；如 querySelector 某些复杂 selector / Range API / iframe 行为。日常 99% 测试两者无差。

02 底层逻辑

jsdom · 完整 spec 实现

jsdom 由 Elijah Insua 等 2010 年起步，是 Node 上最早的 DOM 实现。设计目标：严格按 W3C / WHATWG spec 实现整套 DOM API。

实现特点：

  • 纯 JS 实现，跑在 Node 进程内
  • 覆盖 DOM Level 1-4 + HTML5 + 部分 CSS OM
  • spec 兼容性最好：Document / Element / Node / Event / Range / Selection / FormData / URL 等等
  • 支持 fetch（通过 undici）、XMLHttpRequest、cookies、history、localStorage
  • 不支持：CSS layout 计算、Canvas 渲染、WebGL、动画、媒体元素的实际播放

性能：跑 500 个 React 组件测试约 8-12 秒（cold）。慢的原因是纯 JS 的 spec 严格实现——每个 DOM 操作都要跑大量 spec 验证（normalizeWhitespace / 各种 InvalidStateError 等）。

2026 年地位：Jest 默认 environment；spec 严格 case 唯一选择。

happy-dom · 性能优先

happy-dom 由 David Ortner 2019 年起步。设计目标：性能优先，覆盖 95% 常见 case。

实现特点：

  • 纯 JS 实现，但内部数据结构和算法都为速度优化
  • 简化 spec 实现（一些不常用的边界 case 故意不实现或行为不同）
  • API 与 jsdom 95% 兼容（getElementById / querySelector / addEventListener 等）
  • 同样不支持：layout / Canvas / WebGL

性能：跑同样 500 个 React 组件测试约 2-4 秒（cold）。比 jsdom 快 3-5x。

不一致 case 例子（罕见）：

  • 某些复杂 CSS selector（:has() / :is() 嵌套）行为略不同
  • Range API 部分方法未实现
  • iframe 内部独立 window / document 的隔离不如 jsdom 严格
  • NodeIterator / TreeWalker 部分行为简化

日常 React 组件测试 99% 不会触发这些不一致。Vitest 自 2.0+ 默认推荐 happy-dom。

Vitest browser mode · 真浏览器

Vitest 2.0 GA 的 browser mode（详见 P3.1）：

实现：

  • 用 Playwright（默认）/ WebdriverIO 启动真 Chromium / Firefox / WebKit
  • Vitest 把测试代码注入浏览器（通过 service worker / iframe）
  • 测试代码在真浏览器 V8 / SpiderMonkey / JSC 上跑
  • 支持完整 DOM + CSS layout + 真 fetch + 真 WebGL + Canvas

能力：

  • 真实 layout（offsetWidth / getBoundingClientRect 准确）
  • 真 CSS（包括 :hover / animation / transition）
  • 真事件（包括 wheel / touch / pointer）
  • 真 Canvas / WebGL 渲染
  • 跨浏览器（一份代码跑 Chrome + Firefox + Safari）

性能：单个浏览器约 5-10 秒启动；跑测试比 happy-dom 慢 3-5x；多浏览器并行成本叠加。

2026 年地位：组件库跨浏览器验证 / layout 测试 / Canvas 测试的事实标准；与 Playwright Component Testing 在能力上重叠（两者并存）。

三种 DOM 性能对比

同样 500 个组件测试，相同硬件：

DOM cold start watch 重跑 full re-run spec 覆盖
jsdom ~10-12s ~3-5s ~8-10s ★★★★★ 最严格
happy-dom ~3-5s ~0.8-1.5s ~2-4s ★★★★ 95% case OK
browser mode（单 Chromium） ~15-20s ~2-3s ~12-15s ★★★★★ 真浏览器
browser mode（3 浏览器） ~25-35s ~5-8s ~20-30s ★★★★★ + 跨浏览器验证

不支持 layout 的影响

jsdom / happy-dom 都不计算 CSS layout——所有 offsetWidth / offsetHeight / getBoundingClientRect / clientWidth 等始终返回 0 或简化值。

影响：

  • 测响应式（@media query 触发的样式）—— 测不出
  • 测元素位置（"按钮在第二行第三列"）—— 测不出
  • 测 CSS animation / transition —— 测不出
  • 测 IntersectionObserver / ResizeObserver —— 不会真触发（要 mock）

解决方案：

  • 这些场景用 browser mode（真浏览器有 layout）
  • 或用 Playwright e2e 测（真浏览器 + 真页面）
  • 不要在 jsdom / happy-dom 强测 layout —— 测不准 + 容易 flaky

选型决策矩阵

场景 推荐 理由
普通 React 业务组件测试 happy-dom 速度 3-5x，95% case 兼容
spec 严格 case（罕见复杂 DOM） jsdom spec 兼容性最好
测 CSS layout / 响应式 browser mode 真 layout 计算
测 Canvas / WebGL browser mode jsdom / happy-dom 不支持
组件库跨浏览器验证 browser mode 多浏览器 覆盖 Chrome / Firefox / Safari
Jest 项目（无 Vitest） jsdom（默认） Jest 与 happy-dom 集成不深
已用 Bun test happy-dom（Bun 推荐） Bun runtime 与 happy-dom 集成好

2026 实测分布

根据真实 React 项目调查：

  • happy-dom：60%（Vitest 默认 + 业务组件够用）
  • jsdom：25%（Jest 项目存量主流）
  • browser mode：15%（组件库 / layout 测试 / 跨浏览器验证）

趋势：jsdom 份额逐渐降到 happy-dom；browser mode 在组件库领域增长。2026-2030 预测 happy-dom 50%+ / jsdom 20%- / browser mode 25%+。

💡 关键洞见：DOM 实现选择是性能 vs 兼容性的权衡

三种 DOM 没有"最好"，只有"匹配场景"。happy-dom 是 2026 年的合理默认（速度 + 兼容性平衡）；jsdom 是 spec 兜底（罕见但需要时不可替代）；browser mode 是真实性兜底（layout / Canvas / 跨浏览器）。多数项目主用 happy-dom，需要时切到另一个，不是非此即彼。

03 应用场景

场景 1 · 新 React 项目选 DOM environment

背景：刚搭好 Vitest，要选 environment。

建议：happy-dom。理由：

  • 速度 3-5x of jsdom，对 watch 模式体验影响大
  • 业务组件 99% 不触发 spec 边界 case
  • Vitest 默认推荐
  • 遇到罕见兼容问题再切 jsdom（少见）

场景 2 · 测一个画布交互组件

背景：写一个 React Canvas 涂鸦板组件。要测：用户拖动鼠标 → 画线 → 保存图像。

建议：browser mode。理由：

  • jsdom / happy-dom 不实现 Canvas API（只有空 stub）
  • 测画线效果需要真实 Canvas 2D context
  • 可以用 Vitest browser mode + Chromium
  • 每个测试约 5-10x 慢于 happy-dom，但能覆盖否则测不到的场景

场景 3 · 项目中 happy-dom 偶尔出兼容性问题

背景：项目主用 happy-dom，但 1-2 个测试在 happy-dom 下行为异常（jsdom 下正常）。

方案：单文件覆盖 environment。

javascript 
// 测试文件顶部
// @vitest-environment jsdom

import { describe, test, expect } from 'vitest';

describe('特殊 spec case', () => {
  // 这个文件用 jsdom，其他仍用 happy-dom
});

不需要为少数 case 切换整个项目的 environment。

04 典型代码

vitest.config.ts · happy-dom（默认推荐） 
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    environment: 'happy-dom'
  }
});
vitest.config.ts · jsdom 
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    environment: 'jsdom',
    environmentOptions: {
      jsdom: {
        url: 'http://localhost:3000'
      }
    }
  }
});
vitest.config.ts · browser mode（真浏览器） 
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    browser: {
      enabled: true,
      provider: 'playwright',
      headless: true,
      instances: [
        { browser: 'chromium' },
        { browser: 'firefox' },
        { browser: 'webkit' }
      ]
    }
  }
});
单文件覆盖 environment 
// some-test.test.ts 顶部
// @vitest-environment jsdom

// 这个文件用 jsdom，覆盖项目默认
workspace 中按 project 配 environment 
// vitest.workspace.ts
import { defineWorkspace } from 'vitest/config';

export default defineWorkspace([
  {
    test: {
      name: 'utils',
      root: './packages/utils',
      environment: 'node'    // 纯函数 lib 用 node
    }
  },
  {
    test: {
      name: 'ui',
      root: './packages/ui',
      environment: 'happy-dom'   // 组件用 happy-dom
    }
  },
  {
    test: {
      name: 'canvas-lib',
      root: './packages/canvas-lib',
      browser: { enabled: true, provider: 'playwright', instances: [{ browser: 'chromium' }] }
    }
  }
]);
happy-dom 偶尔需要的 polyfill 
// vitest.setup.ts
// happy-dom 默认不实现 IntersectionObserver / ResizeObserver
// 大多数测试不需要触发它们；如果需要可以 mock

import { vi } from 'vitest';

class MockIntersectionObserver {
  observe = vi.fn();
  unobserve = vi.fn();
  disconnect = vi.fn();
}

class MockResizeObserver {
  observe = vi.fn();
  unobserve = vi.fn();
  disconnect = vi.fn();
}

global.IntersectionObserver = MockIntersectionObserver as any;
global.ResizeObserver = MockResizeObserver as any;

05 常见陷阱

陷阱 1 · 在 jsdom / happy-dom 测 layout

"为什么 offsetWidth 始终是 0？" —— jsdom / happy-dom 不计算 layout。不要在它们里测 layout 相关（响应式 / 元素位置 / 尺寸）。要测 layout 用 browser mode 或 Playwright e2e。

陷阱 2 · 期望 happy-dom 100% spec 兼容

happy-dom 故意简化部分 spec 边界 case 换性能。99% React 组件测试 OK，但偶尔有"jsdom 过 happy-dom 不过"的测试。不要试图 hack happy-dom 让它兼容 spec——直接切单文件用 jsdom 即可。

陷阱 3 · 给所有项目用 browser mode

"browser mode 是真浏览器，最准！" —— 但慢 3-5x。99% 业务组件测试在 happy-dom 已经准。browser mode 留给真正需要真浏览器的场景（layout / Canvas / 跨浏览器）。

陷阱 4 · 在 happy-dom 测 fetch 然后 mock 错

happy-dom 自带 fetch（基于 undici-like 实现），但行为不完全等同 Node fetch / 浏览器 fetch。如果测试涉及 fetch，用 MSW 拦截（详见 P5）而不是依赖 happy-dom 的 fetch 实现。

陷阱 5 · browser mode 多浏览器并行 CI 时间爆炸

3 浏览器 × 1000 测试 = 3000 测试运行。CI 时间从 2 分钟变成 10+ 分钟。仅对组件库 / 关键 lib 跑多浏览器；业务 app 通常单 Chromium 就够（用户主流浏览器）。
