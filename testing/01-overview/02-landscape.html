深度学习/ 测试工具/ P1 · 测试工具是谁/ 02 · 同代人横评

同代人横评：8 家全维度对比

横向对照 2026-05 主流测试工具：Vitest 3 / Jest 30 / Mocha / Bun test / node:test / Playwright / Cypress / Testing Library。每家都有"世界观"——选错工具不是性能问题，是不匹配场景。本章一次讲清各自定位 + 8 维度对比 + 选型决策矩阵。

阅读时长 ~25 分钟 前置 P1.1（项目史） 主要来源 各家官方文档 + 实测 benchmark + State of JS 2024 调查

01 速览

  1. 没有"通用最佳"测试工具——每家都有 sweet spot：新项目单元测试 → Vitest；存量 React 项目 → Jest；已用 Bun runtime → Bun test；微型 lib 不引依赖 → node:test；e2e → Playwright；DOM 测试 helper → Testing Library（与任意 runner 配合）。选错 ≠ 性能问题，而是不匹配场景。
  2. 2026 主流分布（粗估，State of JS 2024 + 真实项目调查）—— 单元测试：Vitest ~45% / Jest ~35% / Bun test ~10% / Mocha ~5% / 其他 ~5%；E2E：Playwright ~55% / Cypress ~30% / 其他 ~15%。Vitest 在 2024-2026 完成对 Jest 的反超；Cypress 增长明显放缓。
  3. 选型最重要的 3 个维度——(1) ESM 原生支持（Vite 项目 / Bun 项目必须）；(2) 兼容 API 形状（迁移成本）；(3) 生态位置（与 Storybook / Next.js / React 生态的契合度）。性能差距通常不是最重要的——除非 CI 时间紧到秒。
  4. "快"的真相——Bun test cold install 最快，但单元测试运行时差距收窄到 1.2-2x（Vitest / Bun test / Jest 30 都"足够快"）。性能数字常被营销夸大；多数项目选型应优先看生态契合，不是 benchmark。
  5. Jest 30 的现状是"存量维护期"——不再是新项目首选，但 React Native / Next.js / 大量企业项目存量主力。Facebook 退出维护后社区接手；ESM 改善但仍非首选 ESM 项目。
  6. Cypress 的现状是"失去新项目份额"——团队 2024 年商业化压力 + 内部动荡；Playwright 5 维度（跨浏览器 / 多 tab / iframes / 网络 / 并行）全面更优。但已有 Cypress 的项目继续维护合理，迁移成本高于收益。
  7. Testing Library 是配合品而非独立 runner——它是 query API + jest-dom matcher 集合，必须配 Vitest / Jest 等 runner 用。2026 是 DOM 测试事实标准，跨框架（React / Vue / Svelte / Preact）统一 API。

性能横评（典型中型项目，500 个测试）

需要先标注：测试工具性能数字依硬件 / 项目规模 / ESM/CJS / workspace 维度变化巨大。下面是一个"中型 React 项目（500 测试，TypeScript + happy-dom）"在 macOS / 现代 SSD 的近似数字。复现条件 + 真实 benchmark 见各家官方公告。

工具 cold start（首次跑） watch 重跑（改 1 文件） full re-run（无 cache）
Jest 30 ~15-20s ~3-5s ~12-15s
Vitest 3 ~5-8s ~0.5-1.5s ~4-6s
Bun test ~3-5s ~0.3-1s ~2-4s
node:test ~4-6s —（无 watch） ~3-5s

关键观察：

  • Bun test 是性能标杆，但与 Vitest 差距在 1.5-2x，多数日常开发场景感知不显著
  • Vitest 比 Jest 快 2-3x（在 ESM/TypeScript 项目尤其明显，因为 Jest 要 Babel）
  • watch 重跑差距最大——Jest 的 worker 启动比 Vitest 慢得多
  • node:test 性能不错但缺 watch，开发体验受限
  • 差距随项目规模放大：5000+ 测试时 Vitest vs Jest 差距能到 5-10x
⚠️ 性能数字读法

不要把这些数字当作"绝对真理"。所有工具都在快速演化（Jest 30 比 Jest 28 快 30%；Vitest 3 比 Vitest 1 快 50%）。Benchmark 必须复现：相同项目 / 相同硬件 / 相同 happy-dom 还是 jsdom / 相同 ESM 还是 CJS。营销文章里的"快 30x"通常是 cherry-picked 场景。

Vitest 3 · "Vite 生态原生 runner"

世界观：复用 Vite dev server 的所有能力（编译 / plugin / HMR / ESM resolution），让 Vite 项目"零配置"测试。Anthony Fu 主导，2021 年起步，2024-2026 年完成对 Jest 的反超。

设计立场：

  • Jest 兼容 API——describe / test / expect / vi.mock / vi.fn 几乎可直接迁移
  • 原生 ESM——不需要 Babel；TypeScript / JSX 通过 esbuild 转换
  • workspace 模式——vitest.workspace.ts 配多 project（详见 P3.1）
  • browser mode（2024 GA）——真浏览器 via Playwright；适合 layout / Canvas / 组件库
  • 类型测试——expectTypeOf 静态验证类型
  • 覆盖率 v8 provider 默认（详见 P7）

Sweet spot：

  • 新项目（默认选择）
  • Vite / Vue / Svelte / Solid 生态
  • 从 Jest 迁移（API 兼容让迁移成本低）
  • 大型 monorepo（workspace 模式 + Turborepo cache 详见 pnpm P5.1）

避坑：

  • 非 Vite 项目可用但收益降低（复用价值在 Vite 用户身上最大）
  • Next.js 项目兼容存在边界 case（Next.js 官方仍推荐 Jest）
  • browser mode 在 2024 刚 GA，复杂场景可能踩坑

Jest 30 · "存量维护期"

世界观：2014 年 Facebook 出品的"零配置全家桶"。2014-2020 年是 React 生态绝对主流，2025 年后转 OpenJS Foundation 治理，进入存量维护期。

2026 现状：

  • 仍有 35-40% 单元测试份额（存量大）
  • 新项目几乎不选 Jest
  • Babel 依赖让性能始终不及 Vitest
  • ESM 仍需 --experimental-vm-modules（性能折损）
  • Snapshot / mock / coverage 内置度仍是优势

Sweet spot：

  • 存量 React / React Native 项目（迁移成本高）
  • Next.js 项目（官方推荐）
  • 大型企业项目（"我们用了多年了"路径依赖）

避坑：

  • 新项目无理由选 Jest（除非有特殊 React Native 兼容要求）
  • 不要把"全公司用 Jest"当作不评估 Vitest 的理由（API 兼容让协作成本低）

Mocha 11 · "灵活组合工具的代表"

世界观：runner only，组合 chai + sinon + nyc 自配工具栈。2011-2016 年是 Node 生态主流，被 Jest "零配置全家桶"取代。

2026 现状：

  • 主要剩 Node lib / 工具类项目（无 React / DOM 需求）
  • 新项目几乎不选 Mocha（配置成本太高）
  • BDD 描述语法仍是经典
  • 插件生态丰富但不再活跃增长

Sweet spot：

  • Node 后端 lib 维护期
  • 团队偏好"组合工具不锁定"哲学

避坑：新项目选 Mocha 几乎总是错误——Vitest / Bun test / node:test 都更省事。

Bun test · "一体化工具链中的 runner"

世界观：install / run / build / test 是同一产品的 4 个面，共享 parser + 共享缓存。详见 Bun P4.4。

设计选择：

  • Jest 兼容 API（迁移成本低）
  • Bun 速度（5-10x of Jest，1.5-2x of Vitest）
  • watch 模式（Bun 1.1+ 优化）
  • 无需安装依赖（Bun 自带）

2026 现状：

  • 已用 Bun runtime 的项目首选
  • 但功能仍在补——snapshot / coverage / mock 完整度落后 Vitest
  • 无 browser mode（Bun 不能跑真浏览器）
  • 大型 React 生态项目仍倾向 Vitest（生态成熟度）

Sweet spot：

  • 已用 Bun runtime 的项目（一体化最大化）
  • 个人 / 副业项目（"装一个 Bun 替代 Node + npm + Vitest + tsx"）
  • 极致 CI 速度需求

避坑：大型 React 生态切 Bun test 之前要做兼容性试跑；少量复杂依赖可能需要 fallback。

node:test · "内置零依赖 runner"

世界观：Node 22+ 内置（详见 Node P6.1）；轻量但功能不全；适合微型 lib 不引外部依赖。

2026 现状：

  • API 与 Jest / Vitest 兼容（describe / test / 但 expect 接口略不同）
  • 无 mock 内置（需要外部 lib）
  • 无 snapshot 内置
  • 无 watch（依赖外部 watcher）
  • 覆盖率：通过 c8（V8 coverage）外接

Sweet spot：

  • 微型 lib（"我不引 vitest 这种重依赖"）
  • Node 内部工具 / 脚本
  • 追求"不引依赖"哲学的项目

避坑：项目稍微复杂（有 mock / snapshot 需求）就放弃 node:test，直接 Vitest。

Playwright · "2026 e2e 事实标准"

世界观：Andrey Lushnikov 等前 Puppeteer 团队从 Google 跳 Microsoft 的二代品。三协议统一抽象（CDP + WebDriver BiDi + WebKit Inspector）+ auto-wait + trace viewer + 真并行。详见 P6.1。

2026 现状：

  • 新项目 70-80% 选 Playwright
  • 跨浏览器（特别 Safari / WebKit）能力让它在 SaaS / 公开网站项目无可替代
  • Component Testing（Playwright 1.30+）模糊单元/e2e 边界
  • Vercel / Astro / Vue / shadcn 等旗舰项目都用 Playwright

Sweet spot：

  • 新项目 e2e（默认选择）
  • 跨浏览器需求
  • 多 tab / 跨域 / 复杂场景
  • 需要 trace viewer 调试 flaky

避坑：Component Testing 与 Vitest browser mode 重叠；按团队偏好二选一即可，不要同时用。

Cypress 14 · "失去新项目份额"

世界观：测试代码跑在浏览器内部（不外控）；GUI runner + time-travel debugging 是它的护城河。2017-2020 年红极一时，2022-2024 团队动荡 + 商业化压力，2026 进入"失去增量但仍维护存量"阶段。

2026 现状：

  • 仍占 30% e2e 份额（存量大）
  • 新项目几乎不选 Cypress
  • 跨浏览器（特别 Safari）受限是硬伤
  • Promise chain DSL 不能用真 async/await，调试有摩擦
  • Cypress Cloud 付费（并行 / 报告）vs Playwright 内置免费

Sweet spot：

  • 已有大量 Cypress 测试 + 团队熟悉
  • 单一 Chrome 场景 + 强调 GUI 调试
  • 新项目几乎没有合理理由选 Cypress

避坑：不要把 Cypress 的"GUI runner 漂亮"当成不评估 Playwright 的理由——Playwright UI mode（2024+）已经追平这个体验。

Testing Library · "DOM 测试事实标准"

世界观：Kent C. Dodds 主导。"测行为不测实现"哲学的落地——不查 className / 不调 instance / 不读 state，所有 query 通过 accessibility role / label 文本 / 页面文字。详见 P4.1。

设计立场：

  • 跨框架统一 API（@testing-library/react / vue / svelte / preact 共享内核）
  • 不是 runner——必须配合 Vitest / Jest / Bun test
  • userEvent 模拟真实用户操作链（vs fireEvent 底层 DOM event）
  • jest-dom matcher（toBeInTheDocument / toHaveTextContent 等）
  • 2026 是 React / Vue / Svelte 测试默认

Sweet spot：所有 DOM / 组件测试。2026 年没有合理理由不用 Testing Library ——它已经是哲学共识。

选型决策矩阵（2026-05）

把上面所有信息收敛成一张决策矩阵：

场景 单元 runner DOM 网络 mock E2E
新建 React + Vite 项目 Vitest 3 Testing Library + happy-dom MSW 2.0 Playwright
新建 Vue + Vite 项目 Vitest 3 Testing Library + happy-dom MSW 2.0 Playwright

💡 关键洞见：选型不是选"最快的"，是选"匹配场景的"

测试工具领域的"快"差距在多数项目中感知不显著（Vitest vs Bun test 差 1.5-2x，日常开发 2-3 秒差距可忽略）。真正决定生产力的是生态契合 + API 兼容 + 团队熟悉度。Vitest 在 Vite 生态里"零配置"是它的护城河；Bun test 在 Bun 项目里"一体化"是它的护城河；Jest 在 React Native / Next.js 是路径依赖。选型时优先看场景契合，不是 benchmark。

03 应用场景

场景 1 · 新建 SaaS 项目栈选型

背景：5 人创业团队，做 B2B SaaS。React + Next.js 14 + Hono on Bun 后端 + pnpm monorepo。要从零搭测试栈，未来 1-2 年扩到 30+ 包。

建议：

  • 单元测试：Vitest 3（不是 Bun test，因为 Next.js 兼容性 + 生态成熟）
  • 组件测试：Vitest + Testing Library + happy-dom
  • 集成测试：Vitest + Testing Library + MSW（mock Hono API）
  • E2E：Playwright（覆盖 5-10 个关键路径）
  • workspace 配 vitest.workspace.ts；CI 配 Turborepo cache

理由：Next.js 14 在 Vitest 上比 Jest 快得多（ESM 友好）；Bun test 虽然快但与 Next.js 兼容仍有边界 case；Playwright 覆盖未来扩到 Safari 用户的需求。

场景 2 · 老 React + Webpack + Jest 项目升级评估

背景：5 年的项目，500+ Jest 测试。CI 跑测试 8 分钟，团队抱怨慢。考虑迁 Vitest。

分析：

  • Vitest 兼容 Jest API，绝大部分测试代码可不改
  • Webpack 项目不需要切 Vite——Vitest 自带 Vite 编译，与 Webpack 共存
  • 预期 CI 时间 8 → 2-3 分钟（5x 改善）
  • 风险：jest.config 中复杂的 transformIgnorePatterns / moduleNameMapper 需要在 vitest.config 重写

建议路径：先在一个独立 workspace 试跑 Vitest（保留 Jest 主线），验证 1-2 周后逐步切。不要全部一次性切。

场景 3 · 评估"Cypress 切 Playwright"

背景：8 人团队，3 年的 Cypress（200+ 用例）。最近发现 Safari bug 没被测出，且 Cypress 团队动荡让人担心长期支持。

建议：渐进迁移，不要全切。

  • 新增的 e2e 用例用 Playwright 写
  • 关键 Safari / 跨域场景用 Playwright 补
  • 老 Cypress 用例继续维护（直到自然失效或显著价值时再迁）
  • 2-3 年后 Cypress 用例自然减少，再决定是否完全切

理由：200 个 Cypress 用例切 Playwright 要 1-2 个月人力；如果项目没 Safari 必需性，渐进策略 ROI 更高。

04 典型代码

同一个测试在 4 家单元 runner 下的写法对比：

Vitest 3 
import { describe, test, expect, vi } from 'vitest';
import { add } from './math';

describe('add', () => {
  test('adds two numbers', () => {
    expect(add(1, 2)).toBe(3);
  });

  test('mocks dependency', () => {
    const fn = vi.fn().mockReturnValue(42);
    expect(fn()).toBe(42);
  });
});
Jest 30 
// 与 Vitest API 几乎相同！只是 import 来源不同
const { add } = require('./math');

describe('add', () => {
  test('adds two numbers', () => {
    expect(add(1, 2)).toBe(3);
  });

  test('mocks dependency', () => {
    const fn = jest.fn().mockReturnValue(42);
    expect(fn()).toBe(42);
  });
});
Bun test 
import { describe, test, expect, mock } from 'bun:test';
import { add } from './math';

describe('add', () => {
  test('adds two numbers', () => {
    expect(add(1, 2)).toBe(3);
  });

  test('mocks dependency', () => {
    const fn = mock(() => 42);
    expect(fn()).toBe(42);
  });
});
node:test（Node 22+） 
import { describe, test, mock } from 'node:test';
import assert from 'node:assert';   // expect 不内置
import { add } from './math.js';

describe('add', () => {
  test('adds two numbers', () => {
    assert.strictEqual(add(1, 2), 3);   // 不是 expect().toBe()
  });

  test('mocks dependency', () => {
    const fn = mock.fn(() => 42);
    assert.strictEqual(fn(), 42);
  });
});
Playwright（e2e） 
import { test, expect } from '@playwright/test';

test('user can log in', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('Username').fill('alice');
  await page.getByLabel('Password').fill('secret');
  await page.getByRole('button', { name: 'Sign in' }).click();

  // auto-wait + 真 async/await
  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByText('Welcome, Alice')).toBeVisible();
});
Cypress（e2e，promise chain DSL） 
describe('Login', () => {
  it('user can log in', () => {
    cy.visit('/login');
    cy.get('input[name="username"]').type('alice');
    cy.get('input[name="password"]').type('secret');
    cy.contains('button', 'Sign in').click();

    // 不是真 Promise；不能 await
    cy.url().should('include', '/dashboard');
    cy.contains('Welcome, Alice').should('be.visible');
  });
});

05 常见陷阱

陷阱 1 · 用"快"作为唯一选型依据

测试工具营销战激烈，"我比 X 快 N 倍"是常见话术。但实际开发中，单元测试速度差距通常不是决定性——你日常 watch 重跑差几百毫秒不那么戏剧。选型应优先看生态契合 + API 兼容 + 团队熟悉度，性能是次要维度。

陷阱 2 · 把 Jest 30 当"零配置"忽视它的 Babel 依赖

Jest 的"零配置"是 ES5 / Babel 时代的零配置。在 ESM + TypeScript 项目里它需要 babel-jest + ts-jest + 复杂的 transformIgnorePatterns 配置。这恰好是 Vitest 解决的痛点——但很多人没意识到自己已经"配置了一堆 Jest workaround"。

陷阱 3 · 在 Next.js 项目里强切 Vitest

Next.js 官方在 2026 年仍推荐 Jest，原因是 Next.js 内部某些 server-only 模块在 Vitest 下兼容性边界 case 较多。可以切，但要做好踩坑准备。除非有具体 ESM / 性能痛点，Next.js 项目用 Jest 仍合理。

陷阱 4 · 用 Cypress 测跨域 / 多 tab 场景

Cypress 跨域受同源策略限制；多 tab / 多 window 完全不支持。如果你的项目有 OAuth 重定向 / 第三方登录 / 多 tab 协作场景，Cypress 会逼你用 hack（cy.origin）或干脆不测。Playwright 这些场景原生支持。

陷阱 5 · 把 node:test 当 Jest 替代

看到 Node 22 自带 test runner 兴奋切，结果发现 mock / snapshot / coverage 都要外接，配置成本不比装 Vitest 低。node:test 的 sweet spot 是"微型 lib 不想引依赖"，不是"通用替代 Jest"。
