测试工具项目史（2011-2026）

把 JS 测试工具 15 年史讲成一条主线 —— 不是按工具版本，按"每次范式转移及其根因"。核心问题：测试工具怎么从 2011 年的 Mocha 一统天下演化到 2026 年的 Vitest + Testing Library + MSW + Playwright 这套现代默认？答案藏在四次范式转移：(1) BDD 描述语法（Mocha/Jasmine 2011-2014）；(2) 零配置全家桶（Jest 2014）；(3) 测行为不测实现（Testing Library 2018，Enzyme 退场）；(4) ESM 原生 + 多浏览器 e2e（Vitest + Playwright 2020-2024）。

阅读时长 ~30 分钟 前置 写过单元测试 主要来源 Kent C. Dodds / Anthony Fu / Andrey Lushnikov 公开文章 + 各家官方公告

01 速览

  1. 测试工具 15 年由 4 次范式转移组成 —— BDD 语法（Mocha 2011 / Jasmine）→ 零配置 Jest（2014，自带 mock + snapshot + 并行）→ 测行为不测实现（Testing Library 2018，淘汰 Enzyme）→ 现代默认（Vitest + Playwright + MSW，2020-2024）。每次转移都有具体痛点驱动。
  2. 2011-2014 是 Mocha / Jasmine 主导期 —— BDD 语法（describe / it / expect）成为后续所有 runner 的 API 模板。但需要自己组合 chai + sinon + nyc 才有完整能力。
  3. 2014 Jest 用"零配置全家桶"打破组合工具范式 —— Facebook 出品；自带 expect / mock / snapshot / coverage / 并行 / watch；React 默认；后来 React Native / Next.js / Babel 全跟进。Jest 在 2016-2020 年是绝对主流。
  4. 2018 Kent C. Dodds 推出 Testing Library —— 标志着"测行为不测实现"哲学的落地。Enzyme 的 shallow rendering 路线被边缘化，最终 2022 年正式停止维护。
  5. 2017 Cypress 开启真浏览器 e2e时代 —— 第一个有 GUI runner + time-travel debugging 的 e2e 工具。但单浏览器、跨域受限，2020 年 Playwright 接班。
  6. 2020 Playwright 由前 Puppeteer 团队推出 —— 三协议统一抽象（CDP + WebDriver BiDi + WebKit Inspector）+ auto-wait + trace viewer + 真并行。2026 年是新项目 e2e 默认。
  7. 2021 Vitest 出生 —— Anthony Fu 起步；Vite 生态需要原生 ESM runner；Jest 兼容 API + Vite 复用让它在 2024-2026 年成为新项目默认。
  8. 2023-09 Bun test 跟进 —— Jest 兼容 + Bun 速度。但生态仍以 Vitest 为主。详见 Bun P4.4。
  9. 2024 三个里程碑 —— node:test 进入 Node 22 LTS（轻量内置）；MSW 2.0 重写（Service Worker + Node + WebSocket 三模一统）；Vitest browser mode GA（真浏览器组件测试）。
  10. 2025 Jest 30 转 OpenJS 治理 —— Facebook 退出维护；社区接管；ESM 改善但仍非 ESM 项目首选。Jest 进入"存量维护"阶段。

02 底层逻辑

前史 · 2009-2010 · jQuery 时代的"测试是奢侈品"

2009 年 Node 诞生前，前端测试基本是奢侈品。jQuery 时代主流做法：

  • QUnit（2008，jQuery 团队出品）—— 跑在浏览器里测 jQuery 插件
  • YUI Test（Yahoo）—— 跟随 YUI 框架
  • 多数项目根本不写测试，靠手动点

这些工具有共同问题：跑在浏览器里、没有 Node 支持、配置零碎。Node 2009 年出来后，"在 Node 里跑 JS 测试"成了新机会，开启了下一代工具的可能性。

2010-2014 · Jasmine + Mocha · BDD 语法成型

这两个工具几乎同时定义了"现代 JS 测试"的写法：

Jasmine（2010 起，Pivotal Labs 出品）—— BDD（Behavior-Driven Development）描述语法的早期主流：

javascript 
describe('Calculator', function() {
  it('should add two numbers', function() {
    expect(add(1, 2)).toEqual(3);
  });
});

这套 describe / it / expect 三件套被后续所有 runner 沿用——Mocha / Jest / Vitest / Bun test / node:test 都用这个 API 形状。

Mocha（2011, TJ Holowaychuk）—— 第一个流行的 Node-first runner。设计哲学是"灵活组合"：

  • Mocha 只是 runner（跑 describe / it 调用，输出结果）
  • 断言库自选：chai（最流行）/ should / power-assert
  • mock 库自选：sinon（最流行）
  • 覆盖率自选：nyc（基于 Istanbul）/ Istanbul 直接
  • 异步：支持 callback / Promise / async-await

Mocha 2012-2016 年是 Node 生态主流。优点：极度灵活，团队按需组合工具栈。缺点也明显：每个项目都要装 5-6 个包 + 维护配置；不同项目栈不一致，跨项目协作要重学。

2014-04 · Jest · "零配置全家桶"打破组合范式

2014 年 Christoph Pojer（后来去 Yarn 团队）在 Facebook 启动 Jest 项目。设计立场与 Mocha 完全相反：把 runner / 断言 / mock / snapshot / 覆盖率 / 并行全部内置，零配置开箱跑：

javascript 
// jest.config.js 不需要写就能跑
test('add two numbers', () => {
  expect(1 + 2).toBe(3);
});

// mock 内置
jest.mock('./api');
const fetchUser = jest.fn().mockResolvedValue({ name: 'Alice' });

// snapshot 内置
expect(component).toMatchSnapshot();

// 覆盖率内置
// 跑 jest --coverage 即输出 coverage/

Jest 几个关键创新：

  • parallel by default——每个测试文件独立 worker 进程，并行跑
  • snapshot 测试——把渲染输出存成 .snap 文件，下次比对；React UI 测试因此普及
  • jest.mock ——对模块自动打桩，不用手动改源码
  • watch 模式——只重跑变化的相关测试
  • 友好 CLI 输出——彩色 + 失败行高亮 + 进度条

Jest 2016-2020 年是 React 生态绝对主流。create-react-app（CRA）默认 Jest；React Native 默认 Jest；Babel / Webpack / Next.js 全部以 Jest 为标准。它的成功说明了"开箱即用 + 零配置"是真实开发者诉求，而不是 Mocha 主张的"灵活组合"。

但 Jest 也埋下了几个长期痛点：

  • Babel 依赖深——Jest 用 babel-jest 转换源码，build 慢；ESM 时代成为最大瓶颈
  • jsdom 依赖——默认跑 DOM 测试，但 jsdom 性能慢且 spec 不完全
  • mock 模式独特——jest.mock('./api') 用 hoisting 魔法实现，不容易调试
  • ESM 支持长期实验性——2026 年仍需 --experimental-vm-modules

2017 · Enzyme · React 测试一度的标准

Enzyme 由 Airbnb 2015 年发布，2017-2020 年是 React 组件测试主流。它的核心特性是 shallow rendering ——只渲染当前组件，不渲染子组件：

javascript 
// Enzyme 风格（已退场）
import { shallow } from 'enzyme';

const wrapper = shallow(<Button label="Save" />);
expect(wrapper.find('.btn-primary').length).toBe(1);
expect(wrapper.state('count')).toBe(0);
wrapper.instance().handleClick();
expect(wrapper.state('count')).toBe(1);

能查 className、能直接调实例方法、能读 state——这是 implementation detail trap 的源头（详见 P2.2）。组件内部稍微重构，测试就破。

Enzyme 在 2018-2020 年开始走下坡路，原因：

  • React 16 的 Fiber 重构改变了内部结构，Enzyme 适配跟不上
  • Hooks 出现后（React 16.8, 2019）class 组件减少；Enzyme 的 instance().method() 不适合 hooks
  • Kent C. Dodds 2018 年起推 Testing Library，测行为不测实现哲学逐步主流化（详见 P2.2）
  • React 18（2022）严格模式 + 并发渲染让 Enzyme 完全不工作

2022 年 Airbnb 正式声明 Enzyme 进入维护停滞。整个 React 生态在 2020-2022 年完成迁移到 Testing Library。这是一次重大测试哲学的胜利——"测用户能看到的东西"取代了"测组件内部结构"。

2017 · Cypress · 真浏览器 e2e 时代开启

Selenium 主导 e2e 自动化十多年，但有几个明显痛点：

  • 需要 WebDriver 协议 + 浏览器 driver 二进制
  • 测试代码跑在外部进程，每个 action 通过协议发命令，慢
  • 调试困难——出错很难知道发生了什么
  • flaky 严重——网络 / 时序 / 竞态难处理

Cypress（2014 启动，2017 GA）抛出一个激进想法：测试代码跑在浏览器内部，不通过 WebDriver 协议。这带来：

  • 极佳调试——time-travel debugging（每个 action 的 DOM snapshot）
  • 速度快——同进程通信，不走协议序列化
  • GUI runner——开发者打开窗口看测试跑，鼠标悬停看每步状态
  • 自动 wait——内置等元素出现 + 网络请求完成

Cypress 2018-2020 年迅速普及。但这套设计也有结构性限制：

  • 只支持 Chromium-based 浏览器（早期）+ Firefox（2020+）；不支持 Safari / WebKit
  • 跨域测试受限（同源策略）—— 测试需要 origin 切换时复杂
  • 不支持多 tab / 多 window
  • Promise chain DSL（cy.visit().get().click()）—— 看起来流畅但不能用真异步 await，影响调试

Cypress 是大胆实验的成功案例，但它的限制也为下一代留下了空间。

2017-2020 · Puppeteer → Playwright · 协议驱动 + 多浏览器

Google 团队 2017 年发布 Puppeteer ——直接用 Chrome DevTools Protocol（CDP）控制 Chrome。比 Selenium WebDriver 更直接、更快、更稳定。

但 Puppeteer 仅 Chrome。Andrey Lushnikov 等核心团队 2019 年从 Google 跳到 Microsoft，2020 年发布 Playwright，目标是跨浏览器：

  • Chromium / Edge —— 通过 CDP
  • Firefox —— 通过 WebDriver BiDi（W3C 标准）
  • WebKit / Safari —— 通过 WebKit Inspector Protocol

Playwright 在三种协议上做统一抽象层，对外暴露同一套 API。这是 Cypress 单浏览器路线无法做到的。

Playwright 几个关键创新（详见 P6.1）：

  • auto-wait——每个 action 自动等元素稳定（visible / enabled / not animating）
  • fixtures——测试隔离 + 资源共享（每个 worker 独立 browser context）
  • trace viewer——失败时录制 DOM + console + network 完整回放
  • 真并行——进程级；retries 区分 flaky vs 真失败

2022-2024 年 Playwright 迅速取代 Cypress 成为新项目 e2e 首选。2026 年的态势：

  • 新项目 70-80% 选 Playwright
  • 存量 Cypress 项目继续维护
  • Cypress 团队 2024 年商业化压力 + 团队动荡，增长停滞

2018 · Testing Library · 测行为不测实现的哲学胜利

Kent C. Dodds 2016 年开始写一系列 blog 反思 Enzyme 的问题。核心论点："测试应该反映用户体验，不是反映内部实现"。2018 年他发布 react-testing-library，定义了一套全新的 query API：

javascript 
// Testing Library 风格（2026 年主流）
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

render(<LoginForm />);

// 按 accessibility role 找元素（用户视角）
const usernameInput = screen.getByLabelText('Username');
const submitBtn = screen.getByRole('button', { name: 'Sign in' });

// 模拟真实用户操作链
await userEvent.type(usernameInput, 'alice');
await userEvent.click(submitBtn);

// 断言用户能看到的结果
expect(screen.getByText('Welcome, Alice')).toBeInTheDocument();

关键差异：不查 className / 不调 instance / 不读 state。所有 query 都是"用户能感知到的方式"——通过 accessibility role、label 文本、页面文字等。

Testing Library 的影响远超 React。它的核心 API 被移植到所有主流框架：

  • @testing-library/vue（Vue 2/3）
  • @testing-library/svelte
  • @testing-library/preact
  • @testing-library/dom（框架无关）
  • Playwright 的 getByRole / getByLabel API（直接抄 Testing Library）
  • Vitest 的 expect 用 jest-dom matcher（同样 Testing Library 生态）

到 2024 年，Testing Library 已是事实标准。新写的组件测试几乎不再用 Enzyme 风格。详见 P4.1。

2021-10 · Vitest · Vite 生态需要原生 ESM runner

2020-2021 年 Vite 在 Vue / Svelte / Solid 等生态迅速普及（详见 Vite 主题）。但 Vite 项目用 Jest 测试有几个明显问题：

  • Jest 用 Babel 转换 ESM → CJS，慢且配置复杂
  • Vite plugin 系统在 Jest 完全不可用
  • Vite 的 ESM resolution 在 Jest 失效
  • 需要双份配置（vite.config 和 jest.config）

Anthony Fu（Vite / unocss / Slidev 等多个项目核心维护者）2021 年 10 月发布 Vitest 0.0.1，核心立场：复用 Vite dev server 的所有能力，让 Vite 项目"零配置"测试。

Vitest 关键设计：

  • Jest 兼容 API ——describe / test / expect / vi.mock / vi.fn 几乎可直接迁移
  • Vite 复用——dev server 编译 + plugin + HMR + ESM resolution 全部直接用
  • 原生 ESM ——不需要 Babel；TypeScript / JSX 通过 esbuild 转换
  • Tinypool 进程池——基于 piscina，比 Jest worker 启动快
  • watch 模式——HMR 启发的"改一个文件只重跑相关"

Vitest 在 2022-2024 年迅速主流化。关键里程碑：

  • 2022 年 Vue 生态全切 Vitest（Vue 自家 / Vue Router / Pinia 等）
  • 2023 年 Astro / Nuxt / SvelteKit / Solid 跟进
  • 2024 年 Vitest 2.0 + browser mode GA（真浏览器组件测试）
  • 2024-2025 年大量 React 项目从 Jest 迁 Vitest（性能 3-5x）
  • 2026 年 Vitest 3 是 Vite 6 配套主线

详见 P3.1。

2023-09 · Bun test · 一体化工具链中的 runner

Bun 1.0（2023-09）GA 时已包含 bun test——Jest 兼容 API + Bun 速度。详见 Bun P4.4。

Bun test 的设计立场：install / run / build / test 是同一个产品的 4 个面，共享 parser + 共享缓存。这与 Jest / Vitest 都不同——它们是独立 runner。

2026 年 Bun test 现状：

  • 已用 Bun runtime 的项目首选 Bun test
  • 速度比 Vitest 略快（启动 + 装包）
  • 但功能仍在补——snapshot / coverage / mock 完整度落后 Vitest
  • 无 browser mode（Bun 不能跑真浏览器）
  • 大型 React 生态项目仍倾向 Vitest

2024 · 三个里程碑 · node:test / MSW 2.0 / Vitest browser mode

node:test 进 Node 22 LTS——Node 22+ 内置 test runner（详见 Node P6.1）。

  • API 与 Jest / Vitest 兼容（describe / test / expect）
  • 不需要装依赖；最低门槛
  • 但功能轻量：无 mock / 无 snapshot / 无 coverage（要配合 c8）
  • 适合：小型 lib（不引依赖测试）/ Node 内部工具

MSW 2.0 重写（2024-01）—— 把网络 mock 从单一浏览器/Node 模式变成三模一统：

  • 浏览器 —— Service Worker 拦截 fetch/XHR
  • Node —— undici interceptor（不再 polyfill）
  • WebSocket —— 全新 interceptor（MSW 2.0 重大新增）

详见 P5.1。MSW 2.0 是"同一份 handlers 服务 dev / test / Storybook 三场景"的事实标准。

Vitest browser mode GA（2024-Q3）—— 在真浏览器（Chromium / Firefox / WebKit via Playwright）跑 Vitest，与 jsdom / happy-dom 形成第三选项。详见 P4.2。这模糊了"单元测试 vs e2e"的边界——组件测试可以跑在真浏览器。

2025 · Jest 30 · 转 OpenJS 治理，进入存量维护期

Facebook 2022 年宣布 Jest 转 OpenJS Foundation。2025 年 Jest 30 发布，由社区维护团队（含 Christoph Pojer 兼职）继续推进。

Jest 30 改进：

  • ESM 改善（仍非首选 ESM 项目）
  • 性能优化（启动时间缩短）
  • 支持 TypeScript native（experimental）

但 Jest 已经进入存量维护期：

  • 新项目几乎不选 Jest（Vitest / Bun test 是默认）
  • 大量存量 React 项目仍用 Jest（迁移成本）
  • Next.js 仍推荐 Jest（出于历史 + 团队偏好，不是技术）
  • Babel 依赖让性能始终不及 Vitest

这与 Yarn classic / Cypress 类似——它们曾是事实标准，但被新工具的范式转移超越后，进入"遗留维护"阶段。

💡 历史规律：每代都是回应上代某个具体痛点

看 15 年史可以总结一条规律：新一代不是"更好的 X"，而是"针对 X 某个具体痛点的解"。

  • Jest → 解 Mocha 组合痛苦（零配置全家桶）
  • Cypress → 解 Selenium 调试困难（GUI runner + time-travel）
  • Playwright → 解 Cypress 单浏览器 + 跨域受限（多协议抽象）
  • Testing Library → 解 Enzyme 测实现细节（测行为）
  • Vitest → 解 Jest ESM 痛苦 + Vite 项目双配置（Vite 复用）
  • Bun test → 解 npm install + 跑测试两次启动（一体化）
  • MSW → 解"测试代码 mock fetch + 生产代码不一样"（网络层拦截）
  • node:test → 解"小项目装 Vitest 太重"（内置零依赖）

选型时不该问"最新的最好吗"，应问"它解的是不是我的痛点"。

03 应用场景

场景 1 · 老项目从 Jest 迁 Vitest 评估

背景：5 年的 React + Webpack 项目，500+ Jest 测试，用 enzyme 写的居多。CI 跑测试要 8 分钟，团队抱怨"改一行代码 watch 重跑也要 30 秒"。

分析：

  • Vitest 兼容 Jest API，迁移成本主要在 enzyme 测试改写
  • enzyme 在 React 18 已不工作，迟早要改
  • Webpack 项目可保留——Vitest 自带 Vite 编译，不需要改 Webpack 配置
  • 预期收益：CI 时间 8 分钟 → 2-3 分钟；watch 30s → 2-3s

建议路径：

  • 第 1 周：把 Jest 测试从 enzyme 迁到 Testing Library（这是最大块工作）
  • 第 2 周：把测试 runner 从 Jest 切到 Vitest（兼容 API 让大部分文件无需改）
  • 第 3 周：CI 集成 + 性能验证

典型 ROI：3 周投入换长期生产力提升 + 测试代码可维护性大幅改善。

场景 2 · 新创业项目的测试栈选型

背景：3 人小团队，做 SaaS。React + Next.js 14 + Bun runtime。要从零搭测试栈。

建议：

  • 单元测试：Vitest（不是 Bun test，因为生态完整度更好）
  • 组件测试：Vitest + Testing Library + happy-dom
  • 集成测试：Vitest + Testing Library + MSW（mock 后端）
  • E2E：Playwright（覆盖关键用户路径 5-10 个）
  • CI：GitHub Actions + Turborepo cache（详见 pnpm P5.1）

不建议什么：

  • 不用 Jest（新项目无理由选 Jest）
  • 不用 enzyme（已退场）
  • 不用 Cypress（Playwright 全面更好）
  • 不用 nock（MSW 全面更好）
  • 不用 jsdom 默认（happy-dom 快 3-5x）

这是 2026 年现代默认。

场景 3 · "我们用 Cypress 多年要切 Playwright 吗？"

背景：8 人团队，3 年的 Cypress e2e 测试（200+ 用例），团队都很熟 Cypress GUI runner。但最近发现 Safari 兼容性 bug 没被 Cypress 测出。

建议：

  • 不要全切。Cypress 的 200+ 测试切到 Playwright 要 1-2 个月
  • 渐进策略：新增的用例用 Playwright 写；老用例保留 Cypress 直到自然失效
  • Safari / WebKit 测试单独用 Playwright 补（Cypress 仍跑 Chrome）
  • 2-3 年后 Cypress 用例自然减少，再决定是否完全切

这种"新增走新路径"策略避免大爆炸式迁移，符合工程理性。

04 典型代码

同一个测试用例在不同代工具下的写法对比：

2012 Mocha + chai + sinon（组合工具范式） 
// 需要装 5 个包：mocha + chai + sinon + nyc + nock
const chai = require('chai');
const sinon = require('sinon');
const nock = require('nock');
const expect = chai.expect;

describe('UserService', function() {
  beforeEach(function() {
    this.api = sinon.stub().resolves({ name: 'Alice' });
    nock('https://api.example.com').get('/users/1').reply(200, { name: 'Alice' });
  });

  it('fetches user', async function() {
    const result = await fetchUser(1);
    expect(result.name).to.equal('Alice');
  });
});
2016 Jest + enzyme（零配置 + shallow rendering） 
// 装 jest + enzyme 即可
import { shallow } from 'enzyme';
import UserCard from './UserCard';

jest.mock('./api');
import { fetchUser } from './api';

describe('UserCard', () => {
  it('renders user name', () => {
    fetchUser.mockResolvedValue({ name: 'Alice' });
    const wrapper = shallow(<UserCard userId={1} />);
    expect(wrapper.find('.user-name').text()).toBe('Alice');  // ❌ 测实现
    expect(wrapper.state('loading')).toBe(false);              // ❌ 测内部状态
  });
});
2026 Vitest + Testing Library + MSW（现代默认） 
// 装 vitest + @testing-library/react + msw
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { http, HttpResponse } from 'msw';
import { setupServer } from 'msw/node';
import UserCard from './UserCard';

const server = setupServer(
  http.get('https://api.example.com/users/1', () =>
    HttpResponse.json({ name: 'Alice' })
  )
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

test('renders user name', async () => {
  render(<UserCard userId={1} />);

  // ✅ 等用户能看到的内容
  expect(await screen.findByText('Alice')).toBeInTheDocument();
});
2017 Cypress（Promise chain DSL） 
describe('Login flow', () => {
  it('logs in', () => {
    cy.visit('/login');
    cy.get('input[name="username"]').type('alice');
    cy.get('input[name="password"]').type('secret');
    cy.contains('button', 'Sign in').click();
    cy.url().should('include', '/dashboard');
    cy.contains('Welcome, Alice').should('be.visible');
  });
});
2024 Playwright（async/await + locator + auto-wait） 
import { test, expect } from '@playwright/test';

test('logs in', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('Username').fill('alice');
  await page.getByLabel('Password').fill('secret');
  await page.getByRole('button', { name: 'Sign in' }).click();

  // auto-wait：等元素出现 + URL 改变
  await expect(page).toHaveURL(/\/dashboard/);
  await expect(page.getByText('Welcome, Alice')).toBeVisible();
});

05 常见陷阱

陷阱 1 · "新一代肯定比上一代好"

看到 Vitest 火就全员切 Vitest，看到 Playwright 火就全员切 Playwright——这种思路常翻车。每代工具都解某个具体痛点，选型应基于"它解的是不是我的痛点"。例如老 Webpack 项目里 200 个 Jest 测试用 enzyme 写的，盲目迁 Vitest 不会自动让 enzyme 测试变好——核心工作是改写 enzyme 到 Testing Library。

陷阱 2 · 把 Jest 当"零配置"忽视它的 Babel 依赖

Jest 的"零配置"是 ES5/Babel 时代的零配置。在 ESM + TypeScript 项目里它需要 babel-jest + ts-jest + 复杂的 transformIgnorePatterns 配置。这恰好是 Vitest 解决的痛点——但很多人没意识到自己已经"配置了一堆 Jest workaround"。

陷阱 3 · 把 Cypress 的 Promise chain 当"现代异步"

Cypress 的 cy.visit().get().click() 看起来像 Promise chain，但它不是真的 Promise ——是 Cypress 自定义的 chainable command queue。你不能 await，不能 try/catch，不能用 async/await 与 cy 命令交错。这是 Cypress 长期被批评的设计点，也是 Playwright 用真 async/await 的根本动机。

陷阱 4 · 把 Enzyme 的退场看作 React 的问题

Enzyme 退场不是因为 React 18 的"问题"，而是因为 Enzyme 设计本身已经过时——shallow rendering 让"测内部"变得太容易。即使 Enzyme 适配了 React 18，"测行为不测实现"的哲学也会让它失去主流地位。Testing Library 取代 Enzyme 是哲学胜利不是技术胜利。

陷阱 5 · 把"我们公司用 Jest"当作不切的理由

"全公司都用 Jest，切了影响协作" —— 实际 Vitest 兼容 Jest API，跨项目协作几乎无影响。真正的协作壁垒是测试哲学不是工具选择。如果两个团队一个用 Enzyme 风格一个用 Testing Library 风格，工具一致也协作不了；反过来，工具不同但都遵循"测行为不测实现"，跨团队读测试代码毫无障碍。
