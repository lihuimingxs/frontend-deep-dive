Playwright 内部

Playwright 是 Microsoft 团队（前 Google Puppeteer 团队）2020 年起步的二代 e2e 工具。它的核心创新在三协议统一抽象（CDP + WebDriver BiDi + WebKit Inspector）+ auto-wait + 真异步。本章把这套架构讲透：为什么它能跨浏览器、为什么比 Cypress 不那么 flaky、fixtures / trace viewer / parallel 怎么协作。

阅读时长 ~30 分钟 前置 P2（测试心智） 主要来源 playwright.dev / Andrey Lushnikov 演讲 / Microsoft Playwright Engineering Blog

01 速览

  1. Playwright 起源——Andrey Lushnikov 等核心 Puppeteer 团队 2019 年从 Google 跳 Microsoft，2020 年发布 Playwright。设计目标：跨浏览器 + 不依赖单一协议。
  2. 三协议抽象——CDP（Chromium / Edge）+ WebDriver BiDi（Firefox + 标准化路径）+ WebKit Inspector Protocol（Safari）。Playwright 在三协议上做统一 API 层。
  3. auto-wait 是核心创新——每个 action 自动等元素稳定（visible / enabled / not animating / not detached）。这是 Playwright 比 Cypress 不那么 flaky 的根因。
  4. fixtures 系统——测试隔离 + 资源共享。每个 worker 独立 browser context；每个测试独立 page；fixtures 自动清理。
  5. trace viewer——失败时录制 DOM snapshot + console + network + screenshot 完整回放。trace.zip 文件可拖到 trace.playwright.dev 在线查看。
  6. 真并行——进程级。默认 worker 数 = CPU 核数。retries 区分 flaky vs 真失败（CI 默认开 2 retries）。
  7. locator 而不是 selector——page.locator() 是懒求值的引用，使用时才 query；自带 auto-wait + 自动 retry。比 Cypress 的 cy.get() 更准确。

02 底层逻辑

Playwright vs Puppeteer · 二代品的差异

Andrey Lushnikov 团队 2017 年在 Google 做 Puppeteer——Chrome DevTools Protocol（CDP）直驱 Chrome。比 Selenium WebDriver 更快、更直接。但有限制：

  • CDP 仅 Chrome 支持（Firefox / Safari 没有 CDP）
  • 跨浏览器要走 WebDriver（性能差、能力弱）
  • 多 tab / 多 window / 跨域不优雅

2019 年团队跳 Microsoft，2020 发布 Playwright。设计立场：每个浏览器用其原生协议，做统一抽象层。

  • Chromium / Edge —— 用 CDP（最成熟）
  • Firefox —— 用 WebDriver BiDi（W3C 标准化路径）
  • WebKit / Safari —— 用 WebKit Inspector Protocol

这套抽象层让 page.click(selector) 在三个浏览器上行为一致（虽然底层协议完全不同）。这是 Cypress 单浏览器路线无法做到的。

架构层次

text 
┌─────────────────────────────────────────┐
│  你的测试代码（Node 进程）                │
│  page.click('button'); 等                │
└─────────────────────────────────────────┘
                  ↕
┌─────────────────────────────────────────┐
│  Playwright API 层（@playwright/test）    │
│  统一 API：locator / page / context     │
│  auto-wait / 重试 / fixtures            │
└─────────────────────────────────────────┘
                  ↕
┌─────────────────────────────────────────┐
│  protocol bridge 层                      │
│  对应 3 种协议的具体实现                  │
└─────────────────────────────────────────┘
        ↓             ↓              ↓
   ┌────────┐   ┌──────────┐   ┌──────────┐
   │  CDP   │   │ WD BiDi  │   │ WebKit   │
   │  WSS   │   │  WSS     │   │ Inspect  │
   └────────┘   └──────────┘   └──────────┘
        ↓             ↓              ↓
   ┌────────────────────────────────────┐
   │  浏览器进程（Chromium/Firefox/WebKit）│
   └────────────────────────────────────┘

关键：测试代码跑在 Node 进程，不是浏览器内（这是与 Cypress 最大架构差异）。Playwright 通过协议命令"远程操作"浏览器。

这套架构带来：

  • 真 async/await（Node 标准）—— 不是 Cypress 的 promise chain DSL
  • 跨进程隔离 —— 测试 crash 不影响浏览器
  • 无跨域限制 —— 测试代码不在浏览器中，不受同源策略约束
  • 多 tab / 多 window 自然支持 —— 都是协议层的对象

auto-wait · 解决 flaky 的关键

e2e 测试 flaky 的最大原因：时序问题。元素还没渲染完就点击；动画进行中点击；元素被覆盖而点击没响应。

传统解：手动 sleep / 自定义 wait。慢、不可靠。

Playwright 解：每个 action 自动等到元素满足"可操作"条件：

Action 等待条件
click / dblclick visible + stable + not detached + receives events
fill / type visible + enabled + editable
hover visible + stable
check / uncheck visible + enabled + 是 checkbox
selectOption visible + enabled + 是 select

每个条件的具体含义：

  • visible—— 占据 layout（offsetWidth/Height > 0）+ display 不是 none + opacity 不是 0
  • stable—— 连续 2 帧位置 / 大小不变（动画结束）
  • not detached—— 仍在 DOM 树中
  • receives events—— pointer 落点在元素上（不被其他元素覆盖）
  • enabled—— 没 disabled 属性
  • editable—— 不 readonly

默认每个 action 等 30s。如果 30s 后仍不满足，throw 详细错误（说明哪个条件未满足）。

这套机制让用户几乎不需要写 wait 代码——auto-wait 把"时序问题"从开发者负担转移到 Playwright 内核。

locator API · 懒求值 + auto-retry

Playwright 鼓励 locator 而不是 selector：

javascript 
// ❌ 传统 selector：立即 query
const button = await page.$('button');   // 现在 query
await button.click();                     // 后续操作可能元素已变

// ✅ Playwright locator：懒求值
const button = page.locator('button');   // 只是引用，没 query
await button.click();                     // click 时才 query + auto-wait

locator 的特性：

  • 懒求值——构造时不 query，使用时才 query
  • 自动 retry——每次使用都 re-query（DOM 可能变化了）
  • chainable——page.locator('table').locator('tr').first()
  • filter——locator.filter({ hasText: 'Alice' })

Playwright 还提供与 Testing Library 一致的 query：getByRole / getByLabel / getByText 等（详见 P4.1）。这样 e2e 测试与单元测试代码风格一致——都按"用户视角"找元素。

fixtures · 测试隔离 + 资源共享

fixtures 是 Playwright 的依赖注入系统，让测试隔离和资源共享自动化：

javascript 
import { test as base, expect } from '@playwright/test';

// 自定义 fixture
type MyFixtures = {
  authenticatedPage: Page;
};

const test = base.extend<MyFixtures>({
  authenticatedPage: async ({ page }, use) => {
    // setup：登录
    await page.goto('/login');
    await page.fill('[name="username"]', 'alice');
    await page.fill('[name="password"]', 'secret');
    await page.click('button[type="submit"]');
    await page.waitForURL('/dashboard');

    // 把 page 提供给测试
    await use(page);

    // teardown（可选）：清理
  }
});

// 用 fixture
test('view profile', async ({ authenticatedPage }) => {
  await authenticatedPage.goto('/profile');
  // 已登录状态
});

内置 fixtures：page / context / browser / browserName / request 等。

Playwright 的隔离层级：

  • worker 级——每个 worker 独立 browser 实例（多 worker 并行）
  • test 级——每个测试独立 browser context（cookies / storage 隔离）
  • page 级——每个测试独立 page（默认）

这套隔离让测试间互不影响，测试 A 留下的 cookie 不污染测试 B。

trace viewer · 失败时的回放

Playwright 录制完整测试执行：

  • 每个 action 的 DOM snapshot
  • console 日志
  • network 请求
  • 每帧的 screenshot
  • Source code 关联（点 action 跳代码）

配置：

playwright.config.ts 
export default {
  use: {
    trace: 'on-first-retry'   // 第一次重试时录制（CI 推荐）
    // 或 'on'（每次都录）/ 'retain-on-failure'（仅失败保留）
  }
};

失败后查看：

bash 
# 命令行
pnpm playwright show-trace trace.zip

# 或拖到网页 https://trace.playwright.dev/

trace viewer 是 Playwright 的调试杀手锏——CI 上失败的测试，开发者下载 trace 文件就能完整回放。Cypress 的 time-travel 类似但仅在 dev 模式有效。

parallel + retries

Playwright 默认进程级并行：

playwright.config.ts 
export default {
  workers: process.env.CI ? 4 : undefined,   // CI 用 4 worker；本地用 CPU 核数
  fullyParallel: true,                        // 同文件内的测试也并行
  retries: process.env.CI ? 2 : 0             // CI 重试 2 次
};

retries 的工作流：

  1. 测试失败
  2. Playwright 自动重试（默认 0 次本地 / 2 次 CI）
  3. 重试通过 → 标记 "flaky"（不算失败但有警告）
  4. 重试仍失败 → 真失败

这套机制让"flaky 测试可见但不阻塞 CI"——开发者能看到 flaky 列表，定期修，但不影响发布。

Component Testing · 模糊单元 / e2e 边界

Playwright 1.30+ 提供 Component Testing——在真浏览器跑组件单元测试：

javascript 
// Button.spec.tsx
import { test, expect } from '@playwright/experimental-ct-react';
import Button from './Button';

test('renders and clicks', async ({ mount }) => {
  let clicked = false;
  const component = await mount(<Button onClick={() => { clicked = true; }}>Click</Button>);

  await expect(component).toContainText('Click');
  await component.click();
  expect(clicked).toBe(true);
});

这模糊了"单元测试 vs e2e"边界——组件级测试可以跑在真浏览器（layout / CSS / 真事件）。与 Vitest browser mode 重叠。

选哪个：

  • 已用 Playwright e2e —— Component Testing 共享配置 + trace
  • 已用 Vitest unit —— browser mode 与 Vitest 配置一致
  • 2026 两者并存，按团队偏好选
💡 关键洞见：Playwright 的护城河是 auto-wait + 真异步

Playwright 比 Cypress 性能差距不大，但 flaky 率显著低。原因是 auto-wait 把"时序"从用户负担变成内核责任。加上真 async/await（vs Cypress 的 promise chain DSL），调试体验质变。如果你的团队 e2e 测试经常 flaky，多数情况是工具问题不是测试问题——切到 Playwright 通常立竿见影。

03 应用场景

场景 1 · 新项目搭 Playwright

步骤：

bash 
pnpm create playwright@latest

# 选择：
# - TypeScript / JavaScript
# - tests 目录
# - 是否加 GitHub Actions
# - 是否装浏览器（推荐是）

# 跑测试
pnpm playwright test

# UI 模式（推荐开发用）
pnpm playwright test --ui

# 看 report
pnpm playwright show-report

typical 配置在 playwright.config.ts —— 设 baseURL / projects（多浏览器）/ trace / retries。

场景 2 · 跨浏览器矩阵测试

背景：组件库要保证 Chrome / Firefox / Safari 三浏览器兼容。

方案：

playwright.config.ts 
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit', use: { ...devices['Desktop Safari'] } },
    // 移动端
    { name: 'mobile-chrome', use: { ...devices['Pixel 5'] } },
    { name: 'mobile-safari', use: { ...devices['iPhone 13'] } }
  ]
});

每个测试在 5 个 project 中跑（5x 时间，但覆盖完整）。CI 中可以并行多个 project。

场景 3 · CI 上 flaky 测试治理

背景：CI 偶尔某些测试失败，重跑就过。怀疑 flaky。

方案：

  • 开 retries: 2 让 CI 自动重试，flaky 不阻塞合并
  • 开 trace: 'on-first-retry' 录制重试时的 trace
  • 每周 review HTML report 中的 flaky 列表
  • 下载 trace.zip 拖到 trace.playwright.dev 看回放
  • 找出根因（动画 / 网络时序 / 元素重渲染）→ 修复

详见 P7 "flaky 治理"章节。

04 典型代码

基本测试（用 Testing Library 风格 query） 
import { test, expect } from '@playwright/test';

test('user can sign in', async ({ page }) => {
  await page.goto('/login');

  // 用 accessibility role / label（与 Testing Library 一致）
  await page.getByLabel('Username').fill('alice');
  await page.getByLabel('Password').fill('secret');
  await page.getByRole('button', { name: 'Sign in' }).click();

  // auto-wait + assertion
  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByText('Welcome, Alice')).toBeVisible();
});
完整 playwright.config.ts 
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 4 : undefined,
  reporter: [['html'], ['list']],
  use: {
    baseURL: process.env.PREVIEW_URL || 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure'
  },
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit', use: { ...devices['Desktop Safari'] } }
  ],
  // 起 dev server（可选）
  webServer: {
    command: 'pnpm dev',
    url: 'http://localhost:3000',
    reuseExistingServer: !process.env.CI
  }
});
网络拦截 + mock 
test('error state', async ({ page }) => {
  // 拦截 API 返回 500
  await page.route('/api/users/*', route => {
    route.fulfill({ status: 500, body: 'Server error' });
  });

  await page.goto('/users');
  await expect(page.getByRole('alert')).toContainText(/error/i);
});
认证状态复用（避免每个测试重登录） 
// global-setup.ts
import { chromium } from '@playwright/test';

export default async () => {
  const browser = await chromium.launch();
  const page = await browser.newPage();
  await page.goto('http://localhost:3000/login');
  await page.fill('[name=username]', 'alice');
  await page.fill('[name=password]', 'secret');
  await page.click('button[type=submit]');
  await page.context().storageState({ path: 'auth.json' });
  await browser.close();
};

// playwright.config.ts
export default {
  globalSetup: './global-setup.ts',
  use: {
    storageState: 'auth.json'   // 所有测试自动用此认证状态
  }
};
截屏 + 视觉对比 
test('homepage looks correct', async ({ page }) => {
  await page.goto('/');
  // 与基线图对比；不一致则失败
  await expect(page).toHaveScreenshot('homepage.png');
});

// 第一次跑：生成基线 homepage.png
// 后续跑：对比
// 故意改后跑：pnpm playwright test --update-snapshots

05 常见陷阱

陷阱 1 · 用 page.$() / page.$$() 而不是 locator

page.$() 是 ElementHandle（立即 query），不会 auto-retry。元素重渲染后旧的 ElementHandle 失效。始终用 page.locator()——懒求值 + 自动重试。

陷阱 2 · 写 page.waitForTimeout(2000)

这是硬等。Playwright 的 auto-wait 已经处理大多数时序问题。如果你写 waitForTimeout，说明你不理解 auto-wait——找出真正想等的事件（元素出现 / 网络请求完成 / URL 变化）用 page.waitForXxx。waitForTimeout 仅在调试时用。

陷阱 3 · 不开 retries / trace 直接上 CI

CI 不开 retries —— 偶发 flaky 直接 fail PR；不开 trace —— 失败时无法回放调试。CI 推荐配置：retries: 2 + trace: 'on-first-retry' + screenshot/video on-failure。

陷阱 4 · Component Testing 与 Vitest browser mode 同时用

两者能力重叠。同时用导致组件测试在两个工具间分散，团队 confused。选一个：偏 e2e 流程选 Playwright Component Testing；偏 Vitest unit 工作流选 Vitest browser mode。

陷阱 5 · 期望 Playwright 取代单元测试

"Playwright 真浏览器最准，全用它写！" —— 单元测试比 e2e 快 100x。把单元测试写成 Playwright e2e 是测试梯形反模式（详见 P2.1）。Playwright 应该专注关键用户路径（5-15% 比例），不是覆盖一切。
