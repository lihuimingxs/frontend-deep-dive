Playwright vs Cypress 横评

2026 年 e2e 选型答案明确：新项目选 Playwright，存量 Cypress 项目继续维护。本章给详细判据：5 个核心维度对比 + Cypress 团队动荡背景 + 真实迁移案例 + 何时不切。

阅读时长 ~25 分钟 前置 P6.1（Playwright 内部） 主要来源 Playwright / Cypress 官方文档 + 真实开源迁移

01 速览

  1. 新项目选 Playwright——2026 年 e2e 默认选择。新项目 70-80% 选 Playwright。除非有强 Cypress 团队偏好，无理由选 Cypress。
  2. 5 维度差距——跨浏览器（Playwright 支持 Safari，Cypress 不）/ 多 tab / iframes（Playwright 原生，Cypress 受限）/ 网络拦截（两者都强但 Playwright 更灵活）/ 并行（Playwright 内置免费，Cypress 需 Cypress Cloud 付费）。
  3. 真异步 vs Promise chain DSL —— Playwright 用标准 async/await；Cypress 用 cy.get().click() 链式 API。后者看似流畅但不能用真 await，调试难。
  4. Cypress 团队动荡——2024 年商业化压力 + 关键工程师离职。Cypress 仍在维护但增速放缓。新项目选 Cypress 是路径依赖+品牌惯性，不是技术理性。
  5. 已有 Cypress 项目继续维护合理——不要为切而切。200 个 Cypress 用例切 Playwright 要 1-2 个月人力，除非有具体痛点（如必须 Safari 测试），ROI 通常不划算。
  6. 渐进迁移策略——新增 e2e 用 Playwright；老 Cypress 维护到自然失效。两栈共存 1-3 年是合理过渡。
  7. 真实迁移案例——Vercel / Netflix / Adobe 等大公司在 2023-2024 年从 Cypress 迁 Playwright；理由集中在跨浏览器 + 性能 + 内置并行。

02 底层逻辑

5 维度全面对比

维度 Playwright Cypress
架构 外部进程（Node）通过协议控制浏览器 测试代码跑在浏览器内
跨浏览器 Chromium / Firefox / WebKit Chromium-based + Firefox（无 Safari）
多 tab / window 原生支持 不支持（强制单 tab）
跨域 无限制 cy.origin() 切换（受限）
iframes frameLocator 自然 cy.iframe() 复杂
异步语法 真 async/await Promise chain DSL（不能 await）
auto-wait 每个 action 自动 5 维度检查 有 auto-wait 但不如 Playwright 全
网络拦截 page.route() + 完整 fulfill 控制 cy.intercept() 简单 + 强大
并行 内置免费（进程级） 需 Cypress Cloud 付费
GUI runner UI mode（2024+，追平） Cypress App（原生强项）
trace / 调试 trace viewer（CI 也录） time-travel（仅 dev）
Component Testing 原生支持（experimental） 原生支持（GA）
2026 主流度 新项目首选 存量维护

架构差异 · 根因决定能力

两者最深差异在测试代码跑在哪：

Cypress：测试代码跑在浏览器内部（注入 iframe）。

  • + 同进程通信，速度快
  • + 能直接操作 window / document / spy 应用代码
  • − 受同源策略约束（跨域困难）
  • − 不能 await（浏览器内不能跨 frame 等待）
  • − 不能开多 tab（浏览器内 iframe 限制）

Playwright：测试代码跑在外部 Node 进程，通过协议控制浏览器。

  • + 真 async/await（Node 标准）
  • + 跨域无限制（Node 不受同源策略）
  • + 多 tab / window 自然
  • + 协议层抽象让跨浏览器统一
  • − 略慢（跨进程通信）

这套差异是根本性的，不是 Cypress 升级能修复的。

真异步 vs Promise chain · 调试体验差距

Cypress 的链式 API 看似流畅：

javascript 
cy.visit('/login')
  .get('input[name=username]').type('alice')
  .get('input[name=password]').type('secret')
  .contains('button', 'Sign in').click()
  .url().should('include', '/dashboard');

但它不是真的 Promise——是 Cypress 自定义的 chainable command queue。陷阱：

  • 不能 await cy.something()
  • 不能 try/catch
  • 不能与 async/await 标准代码混用
  • console.log 时序与执行时序不一致（log 立即执行，cy 命令排队后执行）

Playwright 用真 async/await：

javascript 
await page.goto('/login');
await page.fill('input[name=username]', 'alice');
await page.fill('input[name=password]', 'secret');
await page.click('button:has-text("Sign in")');
await expect(page).toHaveURL(/\/dashboard/);

每行立即执行；可以 await 任意操作；可以 try/catch；可以与标准 Node 代码混用。调试体验质变。

跨浏览器：Cypress 的硬伤

Cypress 仅支持：

  • Chromium-based（Chrome / Edge / Brave）
  • Firefox（实验性）

不支持：

  • Safari / WebKit
  • 移动端（仅模拟，无真机）

对于公开网站 / SaaS 产品，Safari 必须测（Mac / iOS 用户主流浏览器）。Cypress 用户只能用其他工具补 Safari 测试，分散维护成本。

Playwright 三大引擎齐全（Chromium + Gecko + WebKit），单工具搞定。

并行：Cypress 商业化痛点

Cypress 并行需要 Cypress Cloud 付费服务：

  • 免费层：3 用户 + 500 测试结果/月
  • 商业层：按团队规模 + 测试数计费

没有 Cypress Cloud 时，Cypress 在单 CI runner 上串行跑，慢且无法 scale。

Playwright 内置并行（进程级）+ 免费：

javascript 
// playwright.config.ts
export default {
  workers: 4,           // 4 worker 并行
  fullyParallel: true   // 同文件内测试也并行
};

这是 Cypress 商业化压力的来源——Cypress 团队需要靠 Cloud 付费维持收入，但 Playwright 免费提供同等能力。

Cypress 团队动荡（2023-2024）

2023-2024 年公开信息：

  • 多名核心工程师离开
  • 商业化压力加大（Cypress Cloud 是主营收入）
  • Component Testing GA 但市场反响一般
  • 主线创新放缓（与 Playwright 加速发布对比鲜明）

Cypress 仍在维护，但"未来 5 年押注 Cypress 是否合理"成为团队选型时的关键问题。Playwright 由 Microsoft 资助，背后稳定性更高。

选型决策矩阵

场景 推荐 理由
新项目 e2e Playwright 跨浏览器 + 真异步 + 内置并行
需要 Safari 测试 Playwright Cypress 不支持
需要多 tab / 跨域 Playwright 架构原因 Cypress 受限
已有 Cypress 团队 + 200+ 用例 Cypress（继续）+ Playwright 新增 渐进迁移；ROI 不立即划算
已有 Cypress + Safari bug 漏过 Playwright（重要测试迁移） Safari 兼容是切换的明确理由
极简单项目（1-5 个 e2e） 都行 差距感知不大
团队仅有 Cypress 经验 Playwright（学习曲线低） API 风格类似；1-2 周上手

真实迁移案例

2023-2024 年公开的迁移：

  • Vercel—— 2023 年从 Cypress 迁 Playwright。理由：Next.js 项目需要跨浏览器 + 性能。
  • Netflix—— 主用 Playwright（部分内部工具）。理由：跨浏览器矩阵 + 真并行。
  • Adobe—— Adobe Experience Cloud 切 Playwright。理由：trace viewer + 调试体验。
  • shadcn/ui —— 默认 Playwright Component Testing。
  • Astro —— Playwright e2e。

典型迁移成本：100-200 个 Cypress 用例切 Playwright 约 4-8 周（含逐个 review + 重写 + 验证）。

Cypress 仍有大量存量用户：

  • 已写大量 Cypress 用例的项目
  • 团队 Cypress 经验深
  • 仅 Chrome 用户（如内部工具）

这些项目维护 Cypress 是合理选择，不需要为切而切。

💡 关键洞见：选型 = 评估"长期 5 年下注"

e2e 测试代码维护周期长（5+ 年），选型不是看"现在哪个好"，是看"5 年后哪个仍稳定可用"。Playwright 由 Microsoft 资助 + 团队活跃 + 开源中立 + 用户增长，5 年后大概率仍是主流。Cypress 商业化压力 + 团队动荡 + 增长停滞，5 年后地位不确定。新项目选 Playwright 是降低长期风险的合理决定。

03 应用场景

场景 1 · 新 SaaS 项目 e2e 选型

背景：3 人团队，新 React SaaS。前期需要写 5-10 个 e2e 覆盖关键路径。

建议：Playwright。零纠结。

  • 跨浏览器（未来需要 Safari 测试时已经有）
  • 真 async/await 调试体验好
  • 免费内置并行
  • UI mode 开发体验追平 Cypress

选 Cypress 在 2026 年只在"团队全部 Cypress 经验 + 0 Playwright 经验"时合理——但这种情况现在罕见。

场景 2 · 已有 Cypress 200 用例要切吗

背景：3 年的 React 项目，200 个 Cypress 用例，团队熟悉。

分析：

  • 切完整成本：4-8 周人力
  • 立即收益：性能略好 / 真异步调试舒服 / 跨浏览器（如果需要）
  • 长期收益：5 年后 Playwright 仍主流的概率高
  • 风险：迁移期间稳定性

建议：不立即全切。

  • 新增 e2e 用 Playwright 写
  • 关键 Safari 路径用 Playwright 补充
  • 老 Cypress 维护到自然失效
  • 2-3 年后老用例减少 50%+，再决定全切还是渐进

场景 3 · 紧急切：Cypress 漏 Safari bug

背景：项目用 Cypress 一年，最近发现 Safari 兼容性 bug 在生产暴雷（Cypress 没 Safari 测不到）。

方案：选择性切 Playwright。

  • 不全切——保留 Chrome 路径的 Cypress 测试
  • 关键 Safari 兼容性测试用 Playwright 单独写（5-10 个）
  • 新增 e2e 一律 Playwright
  • 1 年后看 Cypress 用例数量趋势再做下一步决定

这套"最小动作回应明确痛点"的策略避免大迁移代价同时立即获得 Safari 覆盖。

04 典型代码

同测试 · Cypress 写法 
describe('Login', () => {
  it('successful login', () => {
    cy.visit('/login');
    cy.get('input[name=username]').type('alice');
    cy.get('input[name=password]').type('secret');
    cy.contains('button', 'Sign in').click();
    cy.url().should('include', '/dashboard');
    cy.contains('Welcome, Alice').should('be.visible');
  });

  it('error on invalid', () => {
    cy.visit('/login');
    cy.intercept('POST', '/api/login', { statusCode: 401, body: { error: 'Invalid' } });
    cy.get('input[name=username]').type('bad');
    cy.get('input[name=password]').type('bad');
    cy.contains('button', 'Sign in').click();
    cy.contains('Invalid').should('be.visible');
  });
});
同测试 · Playwright 写法 
import { test, expect } from '@playwright/test';

test.describe('Login', () => {
  test('successful login', async ({ page }) => {
    await page.goto('/login');
    await page.getByLabel('Username').fill('alice');
    await page.getByLabel('Password').fill('secret');
    await page.getByRole('button', { name: 'Sign in' }).click();
    await expect(page).toHaveURL(/\/dashboard/);
    await expect(page.getByText('Welcome, Alice')).toBeVisible();
  });

  test('error on invalid', async ({ page }) => {
    await page.route('/api/login', (route) => {
      route.fulfill({ status: 401, body: JSON.stringify({ error: 'Invalid' }) });
    });
    await page.goto('/login');
    await page.getByLabel('Username').fill('bad');
    await page.getByLabel('Password').fill('bad');
    await page.getByRole('button', { name: 'Sign in' }).click();
    await expect(page.getByText('Invalid')).toBeVisible();
  });
});
Playwright 多 tab 测试（Cypress 不支持） 
test('opens link in new tab', async ({ page, context }) => {
  await page.goto('/');

  // 等新 tab 打开
  const [newPage] = await Promise.all([
    context.waitForEvent('page'),   // 监听新 tab 事件
    page.click('a[target="_blank"]')
  ]);

  await newPage.waitForLoadState();
  await expect(newPage).toHaveURL(/external\.com/);
});
Playwright 跨域 OAuth 流程（Cypress 受限） 
test('oauth flow', async ({ page }) => {
  await page.goto('/');
  await page.click('text=Sign in with Google');

  // Playwright 自然跨域跳转
  await page.waitForURL(/accounts\.google\.com/);
  await page.fill('input[type=email]', 'test@example.com');
  // ... continue to OAuth provider
  await page.click('button[type=submit]');

  // 跳回我们站点
  await page.waitForURL(/our-app\.com\/dashboard/);
  await expect(page.getByText('Welcome')).toBeVisible();
});

05 常见陷阱

陷阱 1 · 因为"我们用 Cypress"就不评估 Playwright

路径依赖是常见决策反模式。Cypress 在某些场景仍合理（已有大量用例 / 仅 Chrome 项目），但新项目应该评估 Playwright，不只是默认沿用。

陷阱 2 · 全切 Cypress 而不评估 ROI

看到 Playwright 火就全员切。200 个 Cypress 用例切 Playwright 4-8 周人力，节省的"长期维护"是否抵 break-even？通常需要 2-3 年。渐进迁移更现实——新增写 Playwright，老的维护到自然失效。

陷阱 3 · 把 Cypress 的限制当 e2e 通病

"e2e 都跨域受限 / 都不能多 tab" —— 错。Cypress 的限制是其架构（测试代码跑浏览器内）造成的，不是 e2e 通病。Playwright 这些场景都原生支持。不要把工具限制泛化为类别限制。

陷阱 4 · 用 Playwright 写大量 e2e 替代单元测试

"Playwright 真浏览器最准！全用它写！" —— 测试梯形反模式（详见 P2.1）。Playwright 适合 5-15% 关键用户路径，不是覆盖一切。

陷阱 5 · 不开 retries 让 CI 阻塞 PR

e2e 测试本质有一定 flaky 率（网络 / 时序 / 第三方依赖）。CI 不开 retries 让偶发 flaky 阻塞 PR 合并。CI 推荐 retries: 2——区分 flaky vs 真失败 + 不阻塞合并。
