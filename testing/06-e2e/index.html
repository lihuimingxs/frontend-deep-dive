P6 · E2E 测试

单元 + 集成测试覆盖大部分代码，但真用户路径需要真浏览器跑。本阶段讲 Playwright（2026 e2e 事实标准）的内部机制 + 与 Cypress 的横评。理解这两章后，你能为项目选对 e2e 工具 + 写出不 flaky 的测试。

章节数 2 主要来源 playwright.dev / Andrey Lushnikov 演讲 / Microsoft Playwright Engineering Blog 状态 ✅ 全部完成

本阶段覆盖

  • Playwright 起源· Andrey Lushnikov 等 Puppeteer 团队从 Google 跳 Microsoft 的"二代品"
  • 三协议统一抽象· CDP（Chrome）+ WebDriver BiDi（Firefox）+ WebKit Inspector（Safari）
  • auto-wait 机制· 自动等元素 visible / enabled / not animating 才执行 action
  • fixtures 系统· 测试隔离 + 资源共享；每 worker 独立 browser context
  • trace viewer· DOM snapshot + console + network 完整回放
  • parallel· 进程级并行（默认 = CPU 核数）；retries 区分 flaky vs 真失败
  • locator API· page.locator() 懒求值 + auto-retry；getByRole / getByLabel 与 Testing Library 一致
  • Component Testing· Playwright 1.30+ 单测组件在真浏览器
  • Cypress 设计· 跑在浏览器内 / Promise chain / GUI runner / 跨域受限
  • 5 维度横评· 跨浏览器 / 多 tab / iframes / 网络拦截 / 并行
  • 2026 现状· 新项目 70-80% 选 Playwright；Cypress 团队动荡 + 增长停滞
  • 真实迁移案例· Vercel / Astro 从 Cypress 迁 Playwright
