Vitest vs Jest vs Bun test vs node:test 横评

4 家单元测试 runner 全维度对比 + 选型决策矩阵 + 真实迁移案例。2026 年新项目默认 Vitest；存量 Jest 维护合理；Bun runtime 项目用 Bun test；微型 lib 用 node:test。本章给详细判据，让你能为项目精准选型。

阅读时长 ~25 分钟 前置 P3.1（Vitest 内部） 主要来源 4 家官方文档 + 真实迁移案例（Astro / Vue / typescript-eslint）

01 速览

  1. 新项目默认 Vitest——除非有强 Bun runtime / Next.js / 微型 lib 理由，2026 单元测试默认选 Vitest 3。
  2. Jest 30 进入存量维护期——OpenJS 治理后维护活跃但创新放缓。Babel 依赖让 ESM 性能始终落后 Vitest。新项目无理由选 Jest。
  3. Bun test 适合 Bun runtime 项目——一体化优势 + 速度优势；与 Bun runtime 项目一起用最大化收益。
  4. node:test 适合微型 lib——内置零依赖；但缺 mock / snapshot / coverage 内置；适合不想引依赖的小项目。
  5. API 兼容性——4 家共享 BDD 风格（describe / test）；expect 接口 Vitest / Jest / Bun 几乎相同；node:test 用 assert 不是 expect。
  6. 性能差距——Bun test 最快 / Vitest 次之 / Jest 慢 3-5x（ESM 项目）/ node:test 中等。但多数项目感知不显著。
  7. 生态成熟度——Jest 最大（10 年积累）/ Vitest 快速增长（2026 已是 React/Vue 生态主流）/ Bun test 跟随 Bun / node:test 最少。

性能对比（中型 React 项目，500 测试）

测试场景：500 个测试 / TypeScript / happy-dom / macOS M2 Pro / 现代 SSD。数字仅作参考，复现条件 + 真实 benchmark 见各家官方公告。

场景 Vitest 3 Jest 30 Bun test node:test
cold start（首次跑） ~5-8s ~15-20s ~3-5s ~4-6s
watch 重跑（改 1 文件） ~0.5-1.5s ~3-5s ~0.3-1s —（无 watch）
full re-run（无 cache） ~4-6s ~12-15s ~2-4s ~3-5s
CI（5000 测试，4 worker） ~25-35s ~80-120s ~15-25s ~20-30s

关键观察：

  • Bun test 是性能标杆（Zig 实现 + 系统调用直通）
  • Vitest 与 Bun test 差距 1.5-2x，与 Jest 差距 3-5x
  • watch 重跑差距最大——Vitest module graph 让 incremental 极快
  • 差距随项目规模放大：5000+ 测试 Vitest vs Jest 差距能到 5-10x

Vitest 3 详解

核心定位：Vite 生态原生 runner。详见 P3.1。

2026 优势：

  • 新项目默认（React / Vue / Svelte / Solid 全生态首选）
  • workspace 模式优秀（monorepo 友好）
  • browser mode 是独家能力
  • 类型测试 expectTypeOf 内置
  • UI 模式 @vitest/ui 提供交互式 dashboard

2026 弱项：

  • 非 Vite 项目收益降低
  • Next.js 在 2026 仍推荐 Jest（边界 case）
  • Babel 插件不直接兼容（要找 Vite plugin 等价）

真实大型用户：Vue / Vite / Vitest 自身 / Astro / Nuxt / Solid / SvelteKit / shadcn/ui / typescript-eslint / Cloudflare Workers / Slidev / Tauri。

Jest 30 详解

核心定位：10 年积累的"零配置全家桶"，2026 年存量主力。

2026 优势：

  • 10 年生态最成熟（Stack Overflow / GitHub 答案最多）
  • React Native 官方默认（仍最佳选择）
  • Next.js 官方推荐（兼容性边界 case 比 Vitest 少）
  • 大量企业项目存量主力
  • Jest 30 由 OpenJS 治理，维护稳定

2026 弱项：

  • ESM 仍需 --experimental-vm-modules（性能折损）
  • Babel 依赖让 cold start 慢 3-5x（vs Vitest）
  • jest.config 在现代项目通常 50-100 行（transformIgnorePatterns / moduleNameMapper 等）
  • 无 browser mode
  • 无类型测试内置

真实大型用户：React / React Native / Next.js / Babel / Webpack / Lerna / Yarn / 大量企业 React 项目。

Bun test 详解

核心定位：Bun 一体化工具链中的 runner。详见 Bun P4.4。

2026 优势：

  • 性能标杆（Zig 实现，比 Jest 快 5-10x，比 Vitest 快 1.5-2x）
  • 无需安装依赖（Bun runtime 自带）
  • Jest 兼容 API（迁移成本低）
  • watch 模式优秀（Bun 1.1+）
  • 与 Bun runtime / Bun build 共享 parser 和缓存

2026 弱项：

  • 仍在补 Vitest 已有功能（snapshot / mock / coverage 完整度落后）
  • 无 browser mode（Bun 不能跑真浏览器）
  • 大型 React 生态项目仍倾向 Vitest（生态成熟度）
  • Windows 一级支持到 Bun 1.1+ 才完整

真实大型用户：Bun runtime 自家 / Hono / Elysia / 大量 SaaS 工具型项目。

node:test 详解

核心定位：Node 22+ 内置的轻量 runner。详见 Node P6.1。

2026 优势：

  • 不需要装依赖（Node 自带）
  • 启动快（无 Babel / 无 esbuild 转换）
  • API 与 Jest / Vitest 部分兼容（describe / test）
  • 跑 Node lib 时与生产环境完全一致

2026 弱项：

  • 无 watch 模式（依赖外部 nodemon / tsx --watch）
  • 无 mock / snapshot / coverage 内置
  • 无 workspace
  • 用 assert 不用 expect（API 与主流不一致）
  • 跑 TypeScript 需要 tsx / ts-node
  • 生态非常少

真实大型用户：Node 内部测试 / 一些极简 lib（如 hono 的部分模块） / 不想引依赖的工具脚本。

真实迁移案例

以下是 2024-2025 年公开的迁移数据（含真实开源项目）：

  • Astro —— 从 Mocha 迁 Vitest（2022）。理由：与 Vite 生态一致；CI 速度提升 2x；workspace 友好。
  • Vue 全家桶 —— 从 Jest 迁 Vitest（2022）。理由：Vue 自家 ESM 优先；Jest babel 配置过于复杂。
  • Storybook —— 从 Jest 部分迁 Vitest（2024）。理由：与 Vite Storybook builder 集成。
  • typescript-eslint —— 从 Jest 迁 Vitest（2024）。理由：monorepo workspace 模式 + 性能。
  • Cloudflare Workers —— Vitest 默认（2023+）。理由：原生 ESM + 与 Wrangler 工具链契合。
  • shadcn/ui —— Vitest 默认（2024+）。React + Vite 生态。

反例：Next.js / React Native 仍用 Jest。Next.js 推荐 Jest 是因为某些 server-only 模块在 Vitest 下兼容性边界 case；React Native 是因为生态深度绑定。

典型迁移成本：500-1000 测试规模约 1-2 周（含修兼容性 case）。性能收益：CI 时间通常 2-5x 改善。

💡 关键洞见：选型基于场景，不是技术评分

看 4 家技术对比，Vitest / Bun test 都比 Jest 强。但 Jest 在 Next.js / React Native 仍是最佳选择——不是因为它技术更好，而是因为它生态契合度更高。选型不是给工具打分，是看工具与你项目的契合度。新 Vite 项目选 Vitest 是简单决定；存量 Jest 项目要不要切看具体痛点（CI 时间 / ESM 性能 / monorepo 需求），不是看 Vitest 评分高就盲切。

03 应用场景

场景 1 · 老 Webpack + Jest 项目想迁 Vitest

背景：5 年的 React + Webpack 5 + Jest 27 项目。500 个测试，CI 跑 8 分钟。考虑迁 Vitest 提性能。

分析：

  • Webpack 项目能跑 Vitest（不需要切 Vite）—— Vitest 自带 Vite 编译
  • jest.config 中的 transformIgnorePatterns / moduleNameMapper 要在 vitest.config 重写
  • vi.mock 与 jest.mock 行为略有差异（hoisting 机制）—— 少数测试需要修
  • 预期 CI 8 分钟 → 2-3 分钟

策略：

  • 第 1 周：在一个独立 workspace 试跑 Vitest（不影响 Jest 主线）
  • 第 2 周：解决兼容性问题；运行性能 benchmark
  • 第 3 周：全量切 + 删除 Jest 依赖；CI 改
  • 典型问题：jest.fn().mockResolvedValueOnce 在 Vitest 等价；某些 jest hoist 边界 case

场景 2 · 已用 Bun runtime 选 Bun test 还是 Vitest

背景：5 人团队，Hono on Bun 后端 + React 前端。问该用什么测试 runner。

分析：

  • 后端 Hono on Bun → Bun test（一体化最大化，速度优势）
  • 前端 React + Vite → Vitest（生态成熟度 + Vite 契合）

不要试图用一个 runner 测两边。前后端项目本质不同（DOM vs Node），用各自最适合的工具是合理的。

场景 3 · 微型 npm lib 要不要装 Vitest

背景：写一个 utility lib（5-10 个函数），发布 npm。0 运行时依赖。

建议：node:test。理由：

  • 不需要 mock / snapshot / DOM
  • 不引依赖让 dev install 极快
  • 下游用户 install 时不会附带 Vitest 巨大体积
  • API 与 Vitest / Jest 兼容（learning cost 低）

例外：如果要求 watch 模式 + 高 DX，仍可选 Vitest（开发期依赖，下游用户不受影响）。

04 典型代码

同一个测试在 4 家的写法对比：

Vitest 
import { describe, test, expect, vi, beforeEach } from 'vitest';
import { fetchUser } from './api';

vi.mock('./api');

describe('UserService', () => {
  beforeEach(() => {
    vi.mocked(fetchUser).mockResolvedValue({ name: 'Alice' });
  });

  test('returns user name', async () => {
    const user = await fetchUser('1');
    expect(user.name).toBe('Alice');
  });
});
Jest 
// 几乎与 Vitest 完全相同！
import { fetchUser } from './api';

jest.mock('./api');

describe('UserService', () => {
  beforeEach(() => {
    (fetchUser as jest.Mock).mockResolvedValue({ name: 'Alice' });
  });

  test('returns user name', async () => {
    const user = await fetchUser('1');
    expect(user.name).toBe('Alice');
  });
});
Bun test 
import { describe, test, expect, mock, beforeEach } from 'bun:test';
import { fetchUser } from './api';

mock.module('./api', () => ({
  fetchUser: mock(() => Promise.resolve({ name: 'Alice' }))
}));

describe('UserService', () => {
  test('returns user name', async () => {
    const user = await fetchUser('1');
    expect(user.name).toBe('Alice');
  });
});
node:test 
import { describe, test, mock, beforeEach } from 'node:test';
import assert from 'node:assert';
import { fetchUser } from './api.js';

describe('UserService', () => {
  test('returns user name', async () => {
    const mockedFetch = mock.fn(async () => ({ name: 'Alice' }));
    // 注意：API 风格与 Jest/Vitest/Bun 略不同
    const user = await mockedFetch('1');
    assert.strictEqual(user.name, 'Alice');   // 不是 expect.toBe()
  });
});

05 常见陷阱

陷阱 1 · 用 benchmark 数字说服团队切

"Vitest 比 Jest 快 5x！我们必须切！" —— 性能差距通常不是首要决策因素。更重要的是：迁移成本 vs 长期收益。500 个测试切 Vitest 要 1-2 周人力，节省的 CI 时间多久能 break-even？看具体项目。

陷阱 2 · 把 Bun test 当通用 Vitest 替代

看到 Bun test 比 Vitest 快 1.5-2x，全员切 Bun test。但 Bun test 在大型 React 生态项目仍有功能空白（snapshot 完整度 / 复杂 mock / 类型测试 / browser mode）。除非项目用 Bun runtime，多数 React 项目仍以 Vitest 为佳。

陷阱 3 · Next.js 项目盲目切 Vitest

Next.js 14 在 Vitest 下边界 case 较多（server actions / RSC 模拟 / 内部模块兼容）。Next.js 官方在 2026 年仍推荐 Jest。除非有具体性能痛点，Next.js 项目用 Jest 仍合理——别盲目跟风。

陷阱 4 · 在 Lib 里依赖 Vitest 而下游 install 慢

开源 lib 把 vitest 写在 dependencies 而不是 devDependencies。下游用户 install 时附带巨大测试 runner 依赖。修复：测试相关依赖永远在 devDependencies。

陷阱 5 · 期望 node:test 完整替代 Jest

看到 Node 22 自带 test runner 兴奋切，结果 mock / snapshot / coverage 都要外接，最后配置成本不比装 Vitest 低。node:test 的 sweet spot 是"微型 lib 不想引依赖"，不是通用替代。
