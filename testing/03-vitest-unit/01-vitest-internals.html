Vitest 内部

Vitest 不是"另一个测试 runner"——它是对 Vite 生态的零成本扩展。它复用 Vite dev server 的所有能力（编译、plugin、HMR、ESM resolution），用 Tinypool 跑测试 worker。本章讲清这套架构、workspace 模式、browser mode、类型测试，以及为什么它比 Jest 快 3-5x。

阅读时长 ~30 分钟 前置 P2（测试心智）+ Vite 主题 P2.1（Vite dev server） 主要来源 vitest.dev / Anthony Fu blog / Tinypool 文档 / Vitest 源码

01 速览

  1. Vitest 三层架构 —— 主进程跑 Vite dev server（编译 + plugin），Worker 进程跑测试用例（隔离），通信走 Tinypool（基于 piscina 的进程池）。详见 Vite P2.1。
  2. API 兼容 Jest —— describe / test / expect / vi.mock / vi.fn 几乎可直接迁移。vi 替代 jest 全局；其他 99% 兼容。
  3. 原生 ESM —— 不需要 Babel；TypeScript / JSX / JSON / CSS 通过 esbuild + Vite plugin 转换。这是性能优势的根因。
  4. workspace 模式 —— vitest.workspace.ts 配多 project（每个 workspace 子包独立 environment / 配置）。配合 pnpm filter（详见 pnpm P4.2）。
  5. browser mode（Vitest 2.0+ GA）—— 真浏览器（Chromium / Firefox / WebKit via Playwright）跑测试，与 jsdom / happy-dom 形成第三选项（详见 P4.2）。
  6. 类型测试 —— expectTypeOf / assertType 在测试中混用类型断言。配合 tsd / type-coverage 工具补全。
  7. 覆盖率 —— v8 provider（默认，基于 V8 内置 coverage）vs istanbul provider（基于 babel-istanbul，更精确但慢）。详见 P7。
  8. watch 模式启发自 HMR —— 改一个文件只重跑被它影响的测试。Vite 的 module graph 让"哪些测试被影响"准确推断。

02 底层逻辑

三层架构

Vitest 启动时建立 3 层架构：

text 
┌──────────────────────────────────────────┐
│  主进程（Main Process）                     │
│  ─────────────────────────                │
│  • Vite dev server 实例                   │
│    - 编译 TS/JSX/CSS（esbuild）            │
│    - plugin 链（@vitejs/plugin-react 等）  │
│    - module graph（追踪文件依赖）           │
│  • Tinypool 进程池管理器                   │
│  • 测试结果聚合 + reporter                  │
└──────────────────────────────────────────┘
               ↕ IPC（消息 + 编译产物）
┌──────────────────────────────────────────┐
│  Worker 进程（默认 = CPU 核数）              │
│  ─────────────────────────                │
│  • 跑测试用例（独立 V8 isolate）             │
│  • happy-dom / jsdom（如果配 environment） │
│  • 测试结果上报主进程                       │
└──────────────────────────────────────────┘

关键点：

  • 主进程只编译一次——多个 worker 共享同一份编译结果（Vite module graph 缓存）
  • Worker 进程是独立 V8 isolate，测试间完全隔离（mock / 全局变量 / DOM 状态不串味）
  • Tinypool 是 Vitest 团队基于 piscina 的优化版，启动 worker 比 Jest 的 jest-worker 快 30-50%

这套架构让 Vitest 同时拿到两个优势：

  1. Vite 的编译速度（esbuild + 增量）
  2. Jest 的进程隔离（避免 mock 串味）

Vite 复用的具体内容

Vitest 直接用 Vite 的能力，不重新发明：

Vite 能力 Vitest 怎么用
esbuild 转换 TypeScript / JSX 测试代码直接走 Vite 编译，不需要 babel-jest
plugin 链 @vitejs/plugin-react / @vitejs/plugin-vue / 等都自动生效
resolve 配置 vite.config 中的 alias / dedupe / preserveSymlinks 直接复用
module graph watch 模式 "改 A 影响哪些测试" 用 Vite 的依赖图推断
SSR transform Vitest 在 worker 里用 Vite 的 SSR fetchModule 加载模块
CSS 处理 组件引入 .css 直接 import 不报错（jsdom 下空字符串）

这就是为什么 Vite 项目用 Vitest "零配置"——所有 vite.config 中的设置自动作用于测试。

对比 Jest：要让 Jest 跑 ESM + TypeScript + JSX 项目，需要：

  • babel-jest（转 ES5）+ ts-jest（转 TS）
  • jest.config 的 moduleNameMapper 重写 alias
  • jest.config 的 transformIgnorePatterns 处理 ESM-only 依赖
  • setupFiles / setupFilesAfterEach 配 jest-dom / 全局 mock
  • 大约 50-100 行 jest.config 才能跑现代项目

Vitest 用 Vite 项目时，vitest.config.ts 通常 5-15 行就够。

workspace 模式

monorepo 中每个 workspace 子包可能需要不同测试配置：

  • UI 组件库（packages/ui）—— 用 happy-dom + Testing Library
  • 纯函数 lib（packages/utils）—— 用 node environment
  • app（apps/web）—— 用 happy-dom + 各种 plugin
  • browser-only 包 —— 用 browser mode

Vitest workspace 让每个 project 独立配置：

vitest.workspace.ts（项目根） 
import { defineWorkspace } from 'vitest/config';

export default defineWorkspace([
  // packages/ui · 组件测试
  {
    extends: './vitest.config.ts',
    test: {
      name: 'ui',
      root: './packages/ui',
      environment: 'happy-dom',
      setupFiles: './packages/ui/vitest.setup.ts'
    }
  },
  // packages/utils · 纯函数
  {
    extends: './vitest.config.ts',
    test: {
      name: 'utils',
      root: './packages/utils',
      environment: 'node'
    }
  },
  // apps/web · app 测试
  {
    extends: './vitest.config.ts',
    test: {
      name: 'web',
      root: './apps/web',
      environment: 'happy-dom'
    }
  }
]);

跑测试时：

bash 
# 跑全部 workspace 的测试
pnpm vitest

# 仅跑 ui workspace
pnpm vitest --project ui

# 跑变更影响的（结合 turbo）
pnpm turbo test --filter=...[main]
# Turborepo 按 git 变更选 workspace + 跑各自 vitest

这套配合 pnpm filter + Turborepo cache 是 monorepo 测试的现代标准（详见 pnpm P5.1）。

browser mode · 真浏览器组件测试

Vitest 2.0（2024）GA 了 browser mode——在真浏览器（不是 jsdom 模拟）跑测试：

vitest.config.ts 
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    browser: {
      enabled: true,
      provider: 'playwright',  // 或 'webdriverio'
      instances: [
        { browser: 'chromium' },
        { browser: 'firefox' },
        { browser: 'webkit' }
      ]
    }
  }
});

跑测试时 Vitest 起一个真 Chromium / Firefox / WebKit 实例，把测试代码注入浏览器执行。

什么时候用 browser mode：

  • 测真实 layout（happy-dom 不计算 layout）
  • 测 Canvas / WebGL（jsdom / happy-dom 不支持）
  • 测真实 CSS（包括 :hover / animation / transition）
  • 测复杂交互（拖拽 / 多点触摸 / 视口滚动）
  • 组件库 lib 想跨浏览器验证

什么时候不用：

  • 简单业务组件（happy-dom 够用，速度快 5-10x）
  • 纯逻辑函数
  • CI 资源紧张（每个浏览器实例占内存）

2026 年实测分布：80% 测试在 happy-dom；15% 用 browser mode；5% 用 jsdom（spec 严格 case）。详见 P4.2。

类型测试

TypeScript 用户的痛点：单元测试只能测运行时，类型错误要 tsc 才发现。Vitest 提供类型断言 API：

typescript 
import { describe, test, expectTypeOf, assertType } from 'vitest';

function pickKey<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

describe('pickKey types', () => {
  test('returns the correct type', () => {
    const result = pickKey({ name: 'Alice', age: 30 }, 'name');

    // 运行时断言
    expect(result).toBe('Alice');

    // 类型断言（编译期检查 + Vitest 验证）
    expectTypeOf(result).toEqualTypeOf<string>();
    expectTypeOf(pickKey).parameter(1).toEqualTypeOf<'name' | 'age'>();
  });
});

跑 vitest typecheck 时，Vitest 会用 tsc 验证所有 expectTypeOf 断言。

这套替代了过去的tsd（独立类型测试工具）。开源 lib 维护者尤其受益——能保证 API 类型签名向后兼容。

关键配置项

vitest.config.ts 常用字段：

  • test.environment—— jsdom / happy-dom / node / browser
  • test.globals—— 是否注入 describe / test / expect 全局（默认 false，要 import）
  • test.setupFiles—— 跑测试前执行的 setup（如 import jest-dom matcher）
  • test.coverage.provider—— v8（默认）/ istanbul
  • test.poolOptions—— Tinypool 配置（threads / forks / vmThreads）
  • test.alias—— 测试中的额外 alias（覆盖 vite alias）
  • test.reporters—— default / verbose / dot / junit / json / html
  • test.snapshotFormat—— snapshot 序列化选项

注意：test 字段嵌套在 vite.config 顶层，不是单独的 vitest.config（虽然支持）。这让 vite 项目的 dev server + 测试共享一份配置。

watch 模式 · HMR 启发

Vitest 的 watch 不是简单"文件改了就重跑全部"。它用 Vite module graph 推断：

  1. 文件 A 改动
  2. 查 module graph，找出依赖 A 的所有模块（直接 + 间接）
  3. 找到所有"依赖图中包含 A"的测试文件
  4. 只重跑这些测试

结果：典型项目 watch 重跑 0.3-1.5s（vs Jest 的 3-5s）。

这套机制借鉴了 Vite 的 HMR 思路（详见 Vite P5.1），但目标是"重跑测试"而不是"热更新组件"。

💡 关键洞见：Vitest 的护城河是 Vite 生态契合

Vitest 比 Jest 快、API 更简洁、配置更少——但这些都不是 Vitest 独有。它真正的护城河是与 Vite 生态深度契合：vite.config 一份配置同时管 dev / build / test；Vite plugin 自动作用于测试；module graph 让 watch 极快。如果你的项目不在 Vite 生态（如 Webpack 老项目），Vitest 仍能用，但收益减半。

03 应用场景

场景 1 · Vite + React 新项目接入 Vitest

背景：刚 create-vite 出来的 React 项目，要加测试。

步骤：

bash 
pnpm add -D vitest @testing-library/react @testing-library/jest-dom \
  @testing-library/user-event happy-dom @vitest/ui

修改 vite.config.ts：

typescript 
/// <reference types="vitest" />
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'happy-dom',
    globals: true,            // 让 describe / test / expect 全局可用
    setupFiles: './src/test/setup.ts'
  }
});

添加 src/test/setup.ts：

typescript 
import '@testing-library/jest-dom/vitest';

package.json scripts："test": "vitest" + "test:ui": "vitest --ui"。完成。

场景 2 · monorepo 配 Vitest workspace

背景：pnpm monorepo（5 包），想用 Vitest workspace 让每个包独立 environment。

方案：根目录建 vitest.workspace.ts 列出 5 个 project，每个 project 设置自己的 environment / setup。子包不需要单独的 vitest.config（继承根目录）。详见上文"workspace 模式"代码示例。

跑测试：pnpm vitest（全部）/ pnpm vitest --project ui（仅 ui）。

场景 3 · 给组件库加 browser mode

背景：开源 React 组件库，要在 Chromium / Firefox / WebKit 三个浏览器都跑测试，确保跨浏览器兼容。

方案：

bash 
pnpm add -D @vitest/browser playwright
pnpm playwright install
vitest.config.ts 
export default defineConfig({
  test: {
    browser: {
      enabled: true,
      provider: 'playwright',
      headless: true,    // CI 用 headless
      instances: [
        { browser: 'chromium' },
        { browser: 'firefox' },
        { browser: 'webkit' }
      ]
    }
  }
});

每个测试在 3 个浏览器都跑一遍。CI 时间增加 2-3x，但对组件库的跨浏览器信心值得。

04 典型代码

最简单的 Vitest 配置 
// vite.config.ts
/// <reference types="vitest" />
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'happy-dom',
    globals: true,
    coverage: { provider: 'v8' }
  }
});
setup 文件（jest-dom matcher） 
// src/test/setup.ts
import '@testing-library/jest-dom/vitest';
import { afterEach } from 'vitest';
import { cleanup } from '@testing-library/react';

afterEach(() => {
  cleanup();    // 每个测试后清理 DOM
});
vi.mock · 模块级 mock 
import { describe, test, expect, vi } from 'vitest';
import { fetchUser } from './api';

vi.mock('./api', () => ({
  fetchUser: vi.fn().mockResolvedValue({ name: 'Alice' })
}));

describe('user fetch', () => {
  test('returns mocked user', async () => {
    const user = await fetchUser('1');
    expect(user.name).toBe('Alice');
    expect(fetchUser).toHaveBeenCalledWith('1');
  });
});
vi.spyOn · 局部 spy 
import { describe, test, expect, vi, beforeEach } from 'vitest';
import * as api from './api';

describe('login flow', () => {
  beforeEach(() => {
    vi.spyOn(api, 'login').mockResolvedValue({ token: 'abc' });
  });

  test('stores token', async () => {
    const result = await api.login('alice', 'pass');
    expect(result.token).toBe('abc');
  });
});
vi.useFakeTimers · 时间控制 
import { describe, test, expect, vi } from 'vitest';

describe('debounce', () => {
  test('triggers after delay', () => {
    vi.useFakeTimers();
    const fn = vi.fn();
    const debounced = debounce(fn, 1000);

    debounced();
    expect(fn).not.toHaveBeenCalled();

    vi.advanceTimersByTime(1000);
    expect(fn).toHaveBeenCalledOnce();

    vi.useRealTimers();
  });
});
snapshot 测试 
import { test, expect } from 'vitest';
import { render } from '@testing-library/react';
import Card from './Card';

test('Card renders correctly', () => {
  const { container } = render(<Card title="Hello" />);
  expect(container).toMatchSnapshot();
});

// 第一次跑生成 __snapshots__/Card.test.tsx.snap
// 后续跑比对；不一致时 fail
// 故意改动后跑 vitest -u 更新 snapshot
类型测试 
import { test, expectTypeOf } from 'vitest';

type Result<T> = { ok: true; data: T } | { ok: false; error: string };

function unwrap<T>(result: Result<T>): T {
  if (!result.ok) throw new Error(result.error);
  return result.data;
}

test('unwrap return type', () => {
  const r1: Result<string> = { ok: true, data: 'hello' };
  expectTypeOf(unwrap(r1)).toEqualTypeOf<string>();
});

05 常见陷阱

陷阱 1 · 把 vitest.config 与 vite.config 分开维护

"我想测试配置独立" —— 不必要。把 test 字段嵌在 vite.config.ts 中是 Vitest 推荐方式（也是默认）。两份配置容易漂移（alias / plugin 改了 vite 没改 vitest）。除非有强烈理由，保持单一 vite.config.ts。

陷阱 2 · 设 globals: false 但每个测试都 import describe/test

globals: true 让 describe / test / expect 全局可用，写 import 太啰嗦。Jest 用户从 Jest 迁来时常忘记设这个，导致每个测试文件都要 import { describe, test, expect } from 'vitest'。建议默认设 globals: true。

陷阱 3 · 期望 vi.mock 像 jest.mock 一样 hoisted

Jest 的 jest.mock 用 babel-jest hoisting，调用顺序灵活。Vitest 的 vi.mock 也 hoisted（通过 vite plugin），但限制更严：必须在顶层调用，不能在 describe / beforeEach 里。否则报 "vi.mock must be called at top level"。

陷阱 4 · Vite plugin 不在测试中生效

装了某个 Vite plugin（如 vite-plugin-svgr），dev 时正常但测试报错。原因：plugin 可能不支持 SSR transform（Vitest 用 SSR 模式加载模块）。解：要么找 SSR 兼容的版本，要么在 vitest.config 中用 test.deps.inline 把该 plugin 处理的依赖标记内联。

陷阱 5 · 设 pool: 'threads' 后 vi.mock 串味

Vitest 默认 pool: 'forks'（每测试文件一个进程）。改成 'threads'（worker_threads）会更快，但thread 之间共享 V8 isolate，vi.mock 可能串味导致测试相互影响。除非你确定测试代码不依赖全局状态，否则保持默认 forks。
