MSW · Service Worker + Node 双模

网络 mock 不是"模拟 fetch 函数"，是在网络层拦截请求——让测试代码与生产代码完全一致。MSW（Mock Service Worker）2.0 把这套理念做成事实标准：浏览器用 Service Worker 拦截 / Node 用 undici interceptor / WebSocket 也支持。一份 handlers 同时服务 dev / test / Storybook 三个场景。

阅读时长 ~25 分钟 前置 P4（DOM 测试） 主要来源 mswjs.io / Aleksandra Sikora 演讲 / undici interceptor 源码

01 速览

  1. MSW 设计原则——在网络层拦截而不是在应用代码层替换 fetch。优点：测试代码与生产代码完全一致；切换 dev / test / prod 只换 server.start() / handlers。
  2. 三模实现——浏览器用 Service Worker（拦截 fetch / XHR）；Node 用 undici interceptor（拦截 global fetch）；WebSocket 用 MSW 2.0+ 新增的 ws interceptor。
  3. handlers 写法——http.get('/api/users', handler)；支持 path params / query / body 解析；支持 dynamic response（按 input 返回不同 mock）。
  4. vs nock——nock 仅 Node + 拦截 http module（旧接口）。MSW 拦截 fetch（新接口）+ 跨浏览器/Node。MSW 2024 接管 nock 的市场份额。
  5. vs vi.mock('axios')——应用层 mock 让测试代码与生产代码不一致；多 endpoint 时累。MSW 在网络层拦截更干净。
  6. handlers 三场景复用——一份 handlers.ts 同时服务：dev 模式（worker.start() 在浏览器入口注入）、单元/集成测试（server.listen() in setup）、Storybook（msw-storybook-addon）。
  7. 常见陷阱——Service Worker 的 cache 干扰；handlers 顺序问题（first match wins）；Node 18+ 不需要 node-fetch polyfill。

02 底层逻辑

"网络层 mock"vs"应用层 mock"

测试中需要 mock API 请求。两种思路：

应用层 mock（旧路线）：

javascript 
// 在测试中 mock 应用代码导入的 fetch / axios
vi.mock('axios', () => ({
  default: { get: vi.fn().mockResolvedValue({ data: { name: 'Alice' } }) }
}));

// 或更糙：替换 global fetch
globalThis.fetch = vi.fn().mockResolvedValue({
  ok: true,
  json: () => Promise.resolve({ name: 'Alice' })
});

问题：

  • 测试代码与生产代码不一致（生产用 fetch / axios，测试 mock 它们）
  • 多 endpoint 时累（每个 endpoint 写一段 mock）
  • 如果生产改了请求方式（fetch → axios），测试 mock 全废

网络层 mock（MSW 路线）：

javascript 
// handlers 描述"网络层"的契约
const handlers = [
  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json({ id: params.id, name: 'Alice' });
  })
];

// 测试代码完全不知道 mock 的存在
const user = await fetchUser('1');     // 真实代码，不变
expect(user.name).toBe('Alice');

优势：

  • 应用代码完全不变（无论用 fetch / axios / 自定义 client）
  • handlers 描述"API 契约"——前后端协议级别的真相
  • 同一份 handlers 服务 dev / test / Storybook 三场景
  • switching MSW 只是设置变化，应用代码 0 改

这就是 MSW 在 2024 年取代 nock / fetch-mock 等"应用层 mock"工具的原因。

三模实现详解

MSW 2.0 的三模架构：

环境 拦截机制 启动 API
浏览器 Service Worker worker.start()
Node undici interceptor（拦截 global fetch） server.listen()
WebSocket（MSW 2.0+） ws interceptor（拦截 WebSocket 构造） 同上 worker / server

浏览器 · Service Worker 拦截：

  • worker.start() 注册一个 Service Worker（mockServiceWorker.js）
  • SW 拦截页面所有 fetch / XHR 请求
  • 匹配 handlers → 返回 mock 响应；不匹配 → 透传到真实网络
  • 开发者工具的 Network 面板能看到 SW 拦截的请求（标记 "from ServiceWorker"）

Node · undici interceptor：

  • server.listen() 在 Node 中安装一个 undici interceptor
  • 拦截 global fetch（Node 18+ 内置）
  • 不是 polyfill global fetch；而是 hook undici dispatcher
  • 对应用代码透明（fetch 用法不变）

WebSocket interceptor（MSW 2.0+）：

  • 支持 mock WebSocket connection + message + close
  • handlers 用 ws.link('wss://...')
  • 2024 年新增；填补"实时通信测试"空白

handlers 写法

handlers 是一个数组，每个 handler 描述一个 endpoint：

handlers.ts（共享在 dev / test / Storybook） 
import { http, HttpResponse, delay } from 'msw';

export const handlers = [
  // 简单 GET
  http.get('/api/health', () => {
    return HttpResponse.json({ status: 'ok' });
  }),

  // path params
  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json({ id: params.id, name: `User ${params.id}` });
  }),

  // query params
  http.get('/api/search', ({ request }) => {
    const url = new URL(request.url);
    const q = url.searchParams.get('q');
    return HttpResponse.json({ query: q, results: [] });
  }),

  // POST + body 解析
  http.post('/api/login', async ({ request }) => {
    const body = await request.json() as { username: string; password: string };
    if (body.username === 'alice') {
      return HttpResponse.json({ token: 'abc' });
    }
    return HttpResponse.json({ error: 'Invalid' }, { status: 401 });
  }),

  // 模拟延迟
  http.get('/api/slow', async () => {
    await delay(2000);
    return HttpResponse.json({ data: 'finally' });
  }),

  // 错误响应
  http.get('/api/error', () => {
    return new HttpResponse(null, { status: 500 });
  }),

  // 网络错误（连接失败）
  http.get('/api/unreachable', () => {
    return HttpResponse.error();
  })
];

三场景集成

同一份 handlers.ts 用在三处：

1. 测试 setup（Vitest / Jest）：

test/setup.ts 
import { setupServer } from 'msw/node';
import { handlers } from '../mocks/handlers';
import { beforeAll, afterEach, afterAll } from 'vitest';

export const server = setupServer(...handlers);

beforeAll(() => server.listen({ onUnhandledRequest: 'error' }));
afterEach(() => server.resetHandlers());   // 每测试后重置 handlers（清掉 server.use）
afterAll(() => server.close());

2. dev 模式（让前端在没真后端时跑）：

src/main.tsx（开发入口） 
async function enableMocking() {
  if (process.env.NODE_ENV !== 'development') return;
  const { worker } = await import('./mocks/browser');
  return worker.start({
    onUnhandledRequest: 'bypass'   // 未 handler 的请求透传到真后端
  });
}

enableMocking().then(() => {
  ReactDOM.createRoot(document.getElementById('root')!).render(<App />);
});
src/mocks/browser.ts 
import { setupWorker } from 'msw/browser';
import { handlers } from './handlers';

export const worker = setupWorker(...handlers);

3. Storybook：

.storybook/preview.ts 
import { initialize, mswLoader } from 'msw-storybook-addon';
import { handlers } from '../src/mocks/handlers';

initialize();

export const loaders = [mswLoader];
export const parameters = {
  msw: { handlers }
};

三处共享一份 handlers.ts。修改 mock 一处生效全部。这是同源 mock 的工程价值。

动态 handlers · 测试中临时改

测试中常需要"这一个测试用不同的 mock"。MSW 提供 server.use：

javascript 
test('shows error on 500', async () => {
  // 临时覆盖 /api/users/:id 返回 500
  server.use(
    http.get('/api/users/:id', () => {
      return new HttpResponse(null, { status: 500 });
    })
  );

  render(<UserCard userId="1" />);

  expect(await screen.findByRole('alert')).toHaveTextContent(/error/i);
  // afterEach 中 server.resetHandlers() 会清掉这个临时 handler
});

关键：afterEach(() => server.resetHandlers()) 必加。否则测试间 handlers 漂移。

onUnhandledRequest 策略

测试中遇到未 handlers 的请求，怎么办？三种策略：

  • 'error'（推荐测试）—— 抛 error 让测试 fail。强制覆盖所有请求
  • 'warn'（默认）—— console.warn 但通过
  • 'bypass'（推荐 dev）—— 透传到真实网络

测试场景建议设 'error'——任何未预料的请求立即报错，避免静默 bug。

vs nock vs fetch-mock 对比

维度 MSW nock fetch-mock vi.mock('axios')
跨浏览器/Node ✅ 双模 ❌ 仅 Node 🟡 仅 fetch 仅应用层
拦截层 网络层 http module fetch global 应用层
dev 模式可用 ✅ ❌ ❌ ❌
Storybook 集成 ✅ msw-storybook-addon — — —
WebSocket ✅ MSW 2.0+ — — —
2026 主流度 事实标准 存量维护 少用 不推荐
💡 关键洞见：handlers 是 API 契约的代码化

MSW 把 handlers 提升到"API 契约的真相来源"层级。它不只是测试 mock，是开发协作的契约——前端写 handlers 时已经定义了"我期望后端怎么响应"。后端按这个契约实现，自然对齐。这套理念让 MSW 从测试工具进化成前端驱动开发（FDD）的工具。

03 应用场景

场景 1 · 新项目搭 MSW

步骤：

bash 
pnpm add -D msw

# 生成 Service Worker 文件
pnpm dlx msw init public/ --save

建 src/mocks/handlers.ts（共享）+ src/mocks/browser.ts（dev）+ test/setup.ts（test）。

vitest.config.ts 的 setupFiles 加上 test/setup.ts。完成。

场景 2 · 后端还没好，前端先开发

背景：前端 + 后端并行开发。后端 API 还没好。前端要怎么不被 block？

方案：用 MSW dev 模式。

  • 前后端讨论 API 契约（OpenAPI / GraphQL schema / 简单文档）
  • 前端把契约写成 handlers.ts
  • 开发模式跑 worker.start()，前端在浏览器看到"假后端"返回的数据
  • 后端按契约实现真接口
  • 后端好了，前端关 worker（dev 配置切到 bypass 或不启动）

这套流程让前后端真正并行。

场景 3 · Storybook 与 MSW 配合

背景：组件库要在 Storybook 里展示组件的"loading / error / success"三种状态。

方案：用 msw-storybook-addon。

UserCard.stories.tsx 
import { http, HttpResponse, delay } from 'msw';
import UserCard from './UserCard';

export default { component: UserCard };

export const Loading = {
  parameters: {
    msw: {
      handlers: [
        http.get('/api/users/:id', async () => {
          await delay('infinite');   // 永久 loading
          return HttpResponse.json({ name: 'Alice' });
        })
      ]
    }
  }
};

export const Error = {
  parameters: {
    msw: {
      handlers: [
        http.get('/api/users/:id', () => HttpResponse.json({ error: 'Not found' }, { status: 404 }))
      ]
    }
  }
};

export const Success = {
  parameters: {
    msw: {
      handlers: [
        http.get('/api/users/:id', () => HttpResponse.json({ name: 'Alice' }))
      ]
    }
  }
};

三个 story 各自定义 handlers；预览组件自然展示 3 种状态。

04 典型代码

完整 handlers + GraphQL 支持 
import { http, graphql, HttpResponse, delay } from 'msw';

export const handlers = [
  // REST
  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json({ id: params.id, name: 'Alice' });
  }),

  http.post('/api/users', async ({ request }) => {
    const body = await request.json() as any;
    return HttpResponse.json({ id: '123', ...body }, { status: 201 });
  }),

  // GraphQL
  graphql.query('GetUser', ({ variables }) => {
    return HttpResponse.json({
      data: { user: { id: variables.id, name: 'Alice' } }
    });
  }),

  graphql.mutation('CreateUser', async ({ variables }) => {
    return HttpResponse.json({
      data: { createUser: { id: '123', ...variables.input } }
    });
  })
];
条件 mock（按用户 ID 返回不同数据） 
http.get('/api/users/:id', ({ params }) => {
  const id = params.id as string;
  if (id === '1') return HttpResponse.json({ name: 'Alice', role: 'admin' });
  if (id === '2') return HttpResponse.json({ name: 'Bob', role: 'user' });
  return HttpResponse.json({ error: 'Not found' }, { status: 404 });
});
访问真实网络的 passthrough 
import { http, passthrough } from 'msw';

http.get('/api/external/*', () => passthrough())
// 让该请求透传到真实网络（即使配了 onUnhandledRequest: 'error'）
WebSocket mock（MSW 2.0+） 
import { ws } from 'msw';

const chat = ws.link('wss://chat.example.com');

export const handlers = [
  chat.addEventListener('connection', ({ client }) => {
    client.addEventListener('message', (event) => {
      // 客户端发消息后，模拟服务端回复
      client.send(JSON.stringify({
        type: 'echo',
        data: event.data
      }));
    });

    // 主动推消息给客户端
    setTimeout(() => {
      client.send(JSON.stringify({ type: 'welcome' }));
    }, 100);
  })
];
test/setup.ts 完整模板 
import { setupServer } from 'msw/node';
import { handlers } from '../mocks/handlers';
import { beforeAll, afterEach, afterAll } from 'vitest';

export const server = setupServer(...handlers);

beforeAll(() => {
  server.listen({
    onUnhandledRequest: 'error'   // 测试中未 mock 的请求立即 fail
  });
});

afterEach(() => {
  server.resetHandlers();   // 清掉 server.use 加的临时 handlers
});

afterAll(() => {
  server.close();
});

05 常见陷阱

陷阱 1 · 不设 onUnhandledRequest: 'error'

测试默认 'warn'，未 mock 的请求只 console warn 但通过。结果：测试代码里偷偷调了真网络（可能命中 production API），导致测试 flaky 或污染真实数据。测试场景永远设 'error'——强制每个请求被显式 mock。

陷阱 2 · 忘记 server.resetHandlers()

测试 A 用 server.use 加临时 handler，测试 B 没意识到这个 handler 还在生效。afterEach(() => server.resetHandlers()) 必加。这是 MSW 文档明确写出的标准 pattern。

陷阱 3 · Service Worker cache 干扰 dev 模式

开发时改 handlers，浏览器仍跑旧 SW 缓存的版本。修复：DevTools → Application → Service Workers → "Update on reload" 勾上；或开发时用 hard reload（Cmd+Shift+R）。

陷阱 4 · handlers 顺序错（first match wins）

有 http.get('/api/users/:id', ...) 和 http.get('/api/users/me', ...)。第一个会匹配 /api/users/me（":id" 是 'me'）。修复：把更具体的 path 放前面（http.get('/api/users/me', ...) 在前）。

陷阱 5 · Node 18+ 装 node-fetch 干扰 MSW

项目继续装 node-fetch（Node 18+ 已内置 fetch）。MSW Node interceptor 拦截 undici fetch；node-fetch 走自己的实现，绕过 MSW。修复：删除 node-fetch 依赖，用 Node 内置 fetch。
