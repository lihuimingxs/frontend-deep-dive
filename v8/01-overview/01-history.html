V8 项目史 2008-2025

V8 17 年技术演化讲成一条主线 —— 不是按版本号流水账，而是按"每次架构突变及其根因"来组织。三个阶段：2008-2014 单层 JIT 时代（Crankshaft 救场） → 2015-2017 重构为现代管道（Ignition+TurboFan 大重构、Crankshaft 退役） → 2018-2025 多层管道精细化（Sparkplug 填空档、Maglev 提速度、Pointer Compression 省内存、V8 Sandbox 限影响）。读完能回答：为什么 Node 升级 V8 时性能曲线会突变、Crankshaft 这个名字为什么 2018+ 的资料里仍然出现。

阅读时长 ~30 分钟 前置 任意 JavaScript 经验 主要来源 v8.dev/blog/10-years · Wikipedia

01 速览

三句话定义本章：

  1. V8 是 Google 2008 年开源的 JavaScript / WebAssembly 引擎，最初由 Lars Bak 主导；命名取自肌肉车引擎，目标"让 web app 像桌面程序一样快"。
  2. V8 历史上最关键的转折点是 2017 年 V8 5.9 —— Ignition + TurboFan 成为默认管道，Crankshaft 与 Full-codegen 被完全删除。这是 V8 史上最大的一次架构换骨。
  3. 今天的 V8 是 4 层 JIT 管道（Ignition → Sparkplug → Maglev → TurboFan）+ Orinoco GC + Pointer Compression + V8 Sandbox 的工程综合体；每一层都对应一段历史压力。

02 底层逻辑

史前史（2006-2008）：丹麦农场里写出来的引擎

第一阶段 · 2008-2014 · 单层 JIT 时代

正文待写。关键节点：

  • 2008-09：V8 + Chrome 同步开源，初版仅 ia32 + ARM；直接 JIT 无解释器（Full-codegen）
  • 2009：x64 支持；Irregexp 正则引擎；Node.js 诞生并嵌入 V8 —— V8 第一次走出浏览器
  • 2010-12：Crankshaft 优化编译器问世（机器码翻倍 / 体积 -30%）—— V8 正式有了"baseline + optimizer"双层
  • 2011：增量 GC、Isolates 概念引入（多线程隔离的基础）
  • 2013：Octane 2.0 基准；Handle API 重写（"Handlepocalypse"，破坏性 API 变更）
  • 2014：后台并发编译减少卡顿；TurboFan 初版落地（替代 Crankshaft 的下一代优化器）

第二阶段 · 2015-2017 · 重构为现代管道

正文待写。关键节点：

  • 2015：代码缓存、脚本流式加载；Ignition 解释器启动开发（目标：减小 Android 内存）；strong mode（失败的实验）
  • 2016：ES2015/2016 全特性落地；Orinoco GC 启动（并行 + 并发 + 增量）；获 ACM SIGPLAN 软件奖
  • 2017 年 V8 5.9 · 大重构：Ignition + TurboFan 成为默认管道，移除 Crankshaft + Full-codegen；WebAssembly 正式启用 —— 这是 V8 史上最大的一次架构换骨

第三阶段 · 2018-2025 · 多层管道精细化

正文待写。关键节点：

  • 2018：Spectre 防护补丁；Liftoff —— Wasm baseline 编译器；BigInt 原始类型；async/await zero-cost
  • 2020：Pointer Compression（64-bit 系统堆 -43%）—— 从此 V8 堆上限受 4GB 区域限制
  • 2021：Sparkplug —— non-optimizing JIT，bytecode → 机器码 1:1，无 IR；填补 Ignition 与 TurboFan 之间空档（+5-15%）
  • 2023：Maglev（M117 全平台）—— 中层优化器，编译速度比 TurboFan 快 10x，性能 80%；Turboshaft（TurboFan 后端重写，编译速度 2x）；WasmGC 正式发布
  • 2024：V8 Sandbox（Chrome 123 默认开启）—— pointer cage + heap sandbox 阻断从 V8 漏洞升级到进程级 RCE
  • 2025：管道层级稳定为 4 层 Ignition → Sparkplug → Maglev → TurboFan

三个底层逻辑要点

正文待写。三条主线（每个写成一段）：

  • 每次架构突变都有具体压力源：Crankshaft 退役不是因为"过时"，是因为"跟不上 ES2015+ 新语法的优化需求"；Sparkplug 的存在不是"还想再快一点"，是因为 Speedometer 衡量的是页面加载时段
  • "删代码"和"加代码"一样重要：2017 删 Crankshaft + Full-codegen 让团队腾出 30% 工程精力投入 TurboFan；2024 V8 Sandbox 是把"修每一个内存安全 bug"换成"假设 bug 一定有，但限制其影响半径"
  • 嵌入式生态拉动 V8 设计：Isolates 由 Workers 驱动；Snapshot 由 Chrome 启动驱动；V8 Sandbox 由 Cloudflare Workers 驱动

V8 史与 ECMAScript 标准史是平行线

2008 V8 推出 → 2009 ES5 救场（V8 抢先实现）；2015 ES2015 → V8 同年完成 90% 落地；今天 TC39 提案进 Stage 3 后通常 V8 是第一个引擎实现。两条线互为驱动。完整 ECMAScript 标准史详见 ECMA P1.1 · 标准发展史。

05 常见陷阱

"V8 引擎"和"Chrome JS 引擎"不完全等同

Chrome 还有 Blink、Skia 等；V8 只是 JS / Wasm 部分。用 V8 ≠ 用 Chrome。

Crankshaft / Full-codegen 在 2017 已删，2018+ 资料里出现这两个名字基本是过时博客

看 V8 性能调优博客时务必查发布日期 —— 在 2017 后讨论 Crankshaft 优化技巧的资料完全过时。

"V8 没有解释器"是 2015 之前的事

现在 Ignition 是默认入口；所有代码先变 bytecode 再说。
