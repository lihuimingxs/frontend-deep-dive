V8 的 Wasm 编译流水线

这一章是与独立的 WebAssembly 主题之间的"接口章" —— 只讲 V8 怎么实现 Wasm，不讲 wasm32 ISA / WASI / Component Model 等通用 Wasm 知识。把 V8 的 4 层 JS 管道和 2 层 Wasm 管道（Liftoff + TurboFan）放在一起对比，再讲 WasmGC（让 Java/Kotlin/Dart 编译到 wasm 性能 2x）、JSPI（同步 wasm 调异步 JS）、Memory64（让 SQLite-WASM 可用）。

阅读时长 ~25 分钟 前置 P2.3（优化编译器） 主要来源 v8.dev/docs/wasm-compilation-pipeline · holiday-season-2023

01 速览

  1. wasm 没解释器是因为它本身就是 bytecode：JS 的 Ignition 把源码翻译成 bytecode；wasm 直接是 bytecode 形式 → V8 跳过翻译，直接 Liftoff 编译。
  2. WasmGC 的真正意义不是"快"，是"语言生态"：在它之前 Kotlin/wasm 必须自带 GC（包体大、性能差）；现在直接复用 V8 GC。
  3. wasm 在 V8 内部"和 JS 共享一切"：堆、GC、TurboFan、debugger、profiler —— 都是同一套，只是入口不同。

02 底层逻辑

Wasm 双层管道

V8 跑 Wasm 用2 层 JIT 管道——比 JS 的 4 层（Ignition + Sparkplug + Maglev + TurboFan）少一半。原因：wasm 本身就是 bytecode，不需要 parser + bytecode generator；而且 wasm 设计上类型已知（不像 JS 动态类型），不需要为收集类型 feedback 而做多层 IC。

Liftoff（2018 引入）：baseline 编译器

Liftoff 是 V8 的 wasm baseline JIT——把 wasm bytecode 1:1 翻译成机器码，无 IR、无 SSA、无寄存器分配。设计哲学和 JS 的 Sparkplug 完全相同（事实上 Sparkplug 借鉴了 Liftoff 的成功），但 Liftoff 比 Sparkplug 早 3 年出现。这是因为 wasm 没有"先有解释器"的历史包袱——上来就是给 baseline JIT 设计的格式。

一个关键差异：V8 不为 wasm 提供解释器——所有 wasm 函数启动时同步编译为 Liftoff 机器码。这意味着"启动" = "编译完所有函数"。这个设计选择对启动时间影响巨大——见后文"启动时间评估"应用场景。

TurboFan（共享 JS 路径）

热点 wasm 函数会被tier-up 到 TurboFan——和 JS 用同一份 sea of nodes IR、同一套优化遍。这是 V8 节省工程力量的设计——wasm 不需要重写一套优化器。tier-up 触发条件和 JS 类似（call counter 越过阈值），编译同样在 helper thread 上后台跑。

为什么没有 Maglev？因为 wasm 类型已知，Liftoff 与 TurboFan 之间的"中频代码"空档不像 JS 那么明显——TurboFan 编译开销也比 JS 小（不需要做 type feedback 推断）。Liftoff + TurboFan 两层基本够用。

共享 IR + 共享优化遍 = V8 工程效率

一个值得注意的设计：wasm 在 V8 内部"和 JS 共享一切"——同一套堆、同一套 GC（Orinoco / WasmGC 复用）、同一套 TurboFan 优化遍、同一套 debugger / profiler。这就是为什么 V8 团队增加 wasm 支持时不需要写另一个引擎，只需要加 Liftoff 和 wasm 字节码 → sea of nodes 的转换层。JS 引擎的所有优化技术 free upgrade 给 wasm 用——这是 V8 选 wasm 这条路时的隐性收益。

WasmGC（2023 正式）：让托管语言真正能编译到 wasm

原始 wasm 设计假设源语言是 C/C++/Rust——手动管理内存。但很多语言（Java、Kotlin、Dart、Scala、C#、Go）依赖 GC——它们的运行时模型与 wasm 不匹配。

WasmGC 之前：自带 GC + 包体爆炸

2023 之前，托管语言编译到 wasm 必须把整个 GC 实现也编译进 wasm bundle——bundle 通常 5-20MB（光 GC 就几 MB），启动慢、性能差。Kotlin/wasm 早期 bundle 12MB+ 是常态，对比同源代码编译到 JS 仅 200KB——根本无法用于生产。

替代方案是编译到 JS（Kotlin/JS、Scala.js、TeaVM）——但失去 wasm 性能优势，不上不下。这条路 10 年都没走通。

WasmGC：让 wasm 模块复用 V8 GC

WasmGC 提案（Stage 4，2023 年 V8 / Chrome 117 全平台）让 wasm 模块直接复用宿主引擎的 GC——在 V8 里就是复用 Orinoco。wasm 字节码新增了"GC 对象类型"（struct / array of references），创建这些对象时 V8 把它们放进 V8 堆，参与 Orinoco 标准回收流程。

这意味着 Kotlin/wasm bundle 不再需要带 GC——直接说"创建 struct"就行，回收交给 V8。包体下降 80%+，性能提升 2×（V8 团队官方数字）。

生态影响：Kotlin Multiplatform、Dart Web 都在 2023-2025 期间加 wasm target。Scala.js 团队也在跟进。Java 是 Oracle 项目（OpenJDK 的 Project Leyden）单独探索。这是 wasm 真正"不只是 C/C++/Rust 后端"的关键里程碑。

技术细节：WasmGC 对象在 V8 堆内，受 Pointer Compression / V8 Sandbox 保护（详见 P5.2）。这是 wasm 加入 V8 安全模型的入口——相比之前 wasm 线性内存独立于 V8 堆、不享受这些保护。

V8 Wasm 与 JS 的互操作

Wasm 不是孤立的——它通过 Imports / Exports 与 JS 双向交互。Wasm 模块声明它需要从 JS 端拿什么（imports：JS 函数、JS 内存、JS 全局），暴露给 JS 什么（exports：wasm 函数、wasm 内存、wasm 全局）。这是 wasm 嵌入 web 的基础——单纯一段 wasm 没法 fetch、没法操作 DOM、连 console.log 都没有，必须靠 JS 提供这些 API。

Imports / Exports：双向边界

典型用法：

javascript 
// JS 端实例化 wasm 模块
const wasmModule = await WebAssembly.instantiateStreaming(
  fetch('image-processor.wasm'),
  {
    env: {
      // 注入 JS 函数给 wasm 调用
      log: (msg) => console.log(msg),
      // 注入 JS 内存给 wasm 共享（少见）
      memory: new WebAssembly.Memory({ initial: 256 })
    }
  }
);

// JS 端调用 wasm 导出的函数
const { processImage, getResult } = wasmModule.instance.exports;
processImage(imageData);
const result = getResult();

跨边界调用开销：从 100ns 降到 10ns

每次 JS 调 wasm 或 wasm 调 JS 都有边界税——参数转换（JS Number ↔ wasm i32/i64/f32/f64）+ 栈切换 + 调用约定转换。V8 早期（2017-2018）每次跨边界调用 ~100ns，对热路径来说不可接受——大多数早期 wasm 应用避免频繁调用 JS（一次调用处理一大批数据）。

V8 持续优化这个开销：今天典型跨边界调用 ~10ns，特殊情况（如 monomorphic 站点 + 简单参数类型）能降到 ~3ns，逼近 JS 内部函数调用 ~1ns。但仍有"边界税"——在每秒百万级跨边界调用的场景仍可观。

一个隐藏不对称：wasm-to-JS 调用比 JS-to-wasm 慢——V8 优化方向不对称（用户更常 JS 调 wasm 用 wasm 性能，反向少见）。如果你设计 wasm-heavy 应用，让 wasm 主导调度比"JS 频繁回调 wasm"更高效。

Reference Types（externref / funcref）：让 wasm 持有 JS 对象

原始 wasm 类型只有 i32 / i64 / f32 / f64——无法直接持有 JS 对象。Reference Types 提案（2021）加了两个新类型：

  • externref：不透明的 JS 对象引用——wasm 不能解引用，但可以传给 JS 函数
  • funcref：函数引用——可以是 wasm 函数也可以是 JS 函数

这是 WasmGC 之前的"半步" GC 支持——能持有 JS 对象但 GC 责任在 JS。WasmGC 之后这条路相对让位。

JS Promise Integration (JSPI)：同步 wasm 调异步 JS

一个真实痛点：很多 web API 是异步的（fetch、IndexedDB），但wasm 函数都是同步——Rust / C++ 编译到 wasm 后没法直接 await。早期解决方案是把整个 wasm 程序"切成多段"——每次需要异步 IO 都返回控制权给 JS，JS 等结果后再调下一段——极其难写。

JSPI 提案（Stage 4，2025 V8 实现）让 wasm 函数能"暂停"等 Promise resolve 再继续——本质是给 wasm 加了 stackful coroutine 支持。从 wasm 程序员视角看就是"调一个 fetch 直接拿结果"，从 V8 视角是把 wasm 栈帧整个保存挂起、Promise resolve 后恢复。这让 SQLite-WASM、PostgreSQL-WASM 等需要异步 IO 的应用真正可用。

Wasm 内存模型与 V8 堆的边界

Wasm 内存模型与 V8 堆是两套独立的内存系统，需要分清边界：

  • Wasm 线性内存是个大块连续 byte buffer——背后是 V8 通过 ArrayBuffer Allocator 分配的 ArrayBuffer，在 V8 堆外（不进 Pointer Compression cage）。Wasm 的 i32/i64/f32/f64 全部存这里。
  • WasmGC 对象（用 struct.new / array.new 创建）在 V8 堆内，受 Pointer Compression + V8 Sandbox 保护。

这种分离是务实选择：线性内存适合密集数值计算（图像、音频、物理），让 wasm 像 C 一样直接访问 byte；WasmGC 对象适合托管语言用例（Java / Kotlin），让 GC 自动管理。一个 wasm 程序可以两种都用。

Memory64：让 wasm 线性内存超过 4GB

原始 wasm 线性内存用 32-bit index——理论上限 4GB（实际由于 V8 内部限制更低）。这对图像处理、视频编辑、大型数据库 / 分析引擎 / 机器学习模型加载是真实瓶颈——一个 4K 视频帧就要几十 MB，几百帧就触上限。

Memory64 提案（Stage 4，V8 11+ 实现）扩展到 64-bit index——理论上限 16 EB，实际取决于宿主可分配地址（典型 256GB-1TB 数量级）。代价：每次 memory access 多一条指令做 64-bit bound check，V8 fast path 上做了优化但仍有 ~5-15% 性能损失。

意义：让 SQLite-WASM（数据库引擎跑在浏览器，绕开服务端）、duckdb-wasm（OLAP 分析查询直接在浏览器跑）、ffmpeg.wasm（浏览器视频转码）这些"重量级 wasm 应用"真正能用于生产负载。这是把 wasm 从"轻量计算加速"推向"完整应用平台"的关键提案。

Spectre 防护对 Wasm 的影响

2018 年 Spectre 漏洞发布后，wasm 也是攻击面之一——wasm 线性内存可以构造任意指针访问，而 wasm 与 JS 共享进程，理论上能利用 CPU 推测执行旁路读其他 origin 数据。

V8 应对：(1) Site Isolation——不同 origin 的 wasm 跑在不同进程，物理隔离 cache；(2) Guard pages——在 wasm 线性内存边界放不可访问页，OOB 访问 fault 而非泄漏；(3) SharedArrayBuffer 限制——wasm threads 用 SAB 共享内存的能力需要 COOP/COEP（Cross-Origin-Opener-Policy / Cross-Origin-Embedder-Policy）头部验证 cross-origin 安全；详见 ECMA P7.5 · §24 Memory Model。

03 应用场景

场景 A · 评估 wasm 应用启动时间

wasm 启动 ≠ JS 启动。wasm 是"启动 = Liftoff 编译完所有函数"——一个 100MB wasm 模块要花数秒编译，即使你只用其中 1 个函数。对比 JS 的 lazy parsing（详见 P2.2）只完整解析被调用的函数（约 50% 实际调用）——同样代码 JS 启动远快于 wasm。

优化方向：(1) Streaming compilation——WebAssembly.instantiateStreaming(fetch(...)) 让 V8 边下载边编译，下载完编译也基本完成；(2) 用更小的 wasm 模块——按 feature 切分多个小 wasm，按需 load；(3) 用 Code Cache——V8 缓存编译过的 wasm 机器码，二次访问跳过编译。

场景 B · wasm vs JS 选型判据

常见判据"wasm 比 JS 快"过于简化。实际选型：

  • 极算术密集（图像 / 音频 / 物理 / 加密 / ML 推理）：wasm 完胜——线性内存 + 静态类型让 SIMD / 紧密循环跑出近 native 性能
  • 调用密集 / DOM 交互：JS 更优——跨边界税在每秒百万级调用累计可观，wasm 还没法直接操作 DOM
  • 逻辑密集 / 业务代码：JS 通常够——TurboFan 优化后 JS 性能足以满足绝大多数业务逻辑，wasm 增加复杂度不值得
  • 已有 C/C++/Rust 库：wasm 是理性选择——Emscripten / wasm-bindgen 让现有库直接拿来用

场景 C · WasmGC 落地：Kotlin / Dart / Scala

托管语言的 wasm target 在 2023-2025 期间陆续可用：Kotlin Multiplatform 加 wasm target、Dart Web 用 WasmGC 替代 dart2js、Scala.js 团队探索 Scala-wasm。这些项目共同点：包体下降 80%+，性能比对应"编译到 JS"版本快 2-3×。如果你团队用这些语言开发 web，2025 年起值得评估迁移到 wasm target——生态成熟度还不如 wasm-Rust 但快速追赶中。

场景 D · 重量级 wasm 应用：SQLite / DuckDB / ffmpeg

Memory64 + JSPI + WasmGC 让"重量级应用"能完整跑在浏览器里：SQLite-WASM（嵌入式数据库，绕开服务端 SQL）、duckdb-wasm（OLAP 分析查询，处理 GB 级 parquet 直接在浏览器）、ffmpeg.wasm（视频转码不需要服务器）、onnxruntime-web + WASM EP（ML 推理在客户端）。这是 wasm 真正成为"浏览器内的应用平台"的方向——很多原本必须服务端做的事开始客户端化，给隐私 / 离线 / 成本带来革命性影响。

04 典型代码

javascript 
// 跨边界调用开销示例
const wasmModule = await WebAssembly.instantiateStreaming(fetch('mod.wasm'));
const { add } = wasmModule.instance.exports;

// JS → Wasm：~10ns 边界税
console.time('wasm');
for (let i = 0; i < 1e7; i++) add(i, 1);
console.timeEnd('wasm');

// 纯 JS 对照
function jsAdd(a, b) { return a + b; }
console.time('js');
for (let i = 0; i < 1e7; i++) jsAdd(i, 1);
console.timeEnd('js');
// 通常 wasm 慢一些 —— 因为简单算术 V8 内联得极好

05 常见陷阱

wasm 启动 ≠ JS 启动

一个 100MB wasm 模块要花数秒编译（即使 Liftoff），即使你只调用 1 个函数。

wasm 不是"沙盒里的 JS"

是独立 ISA、独立内存模型、独立 spec。

wasm-to-JS 调用比 JS-to-wasm 调用慢

V8 优化方向不对称。
