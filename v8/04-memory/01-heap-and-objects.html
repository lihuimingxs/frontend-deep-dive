堆布局与对象表示

V8 在堆上怎么放对象、怎么压缩指针、怎么把 SMI 塞进指针位。这一章讲清"V8 堆是什么样子"：分代结构（Young / Old / Code / Map / Trusted Space）、Tagged Pointer 编码、SMI 与 HeapNumber 的边界、Pointer Compression（4GB cage 让堆 -43%）、多种 String 表示（ConsString / SlicedString / ExternalString）、以及 2023 加入的 Conservative Stack Scanning。

阅读时长 ~30 分钟 前置 ECMA P4（数据类型） 主要来源 v8.dev/blog/pointer-compression · mutable-heap-number
📝 章节状态

骨架版本。对照大纲 v8/outline.md § P4.1。

01 速览

  1. "V8 中的 number"是 SMI ⊎ HeapNumber 的 union：算术热路径上 V8 做 SMI fast path（无堆分配），溢出回 HeapNumber。
  2. Pointer Compression（V8 8.0+）：64-bit 系统中限制堆到 4GB cage，pointer 用 32-bit offset，堆 -43%。
  3. Pointer Compression 为 V8 Sandbox 铺路：当所有指针都是 32-bit offset 时，corruption 影响范围天然限制在 cage 内。

02 底层逻辑

Heap 分代结构

正文待写。

  • Young Generation (NewSpace)：小、新对象、Scavenger 频繁回收（默认 16MB-64MB）；又分 From-space 和 To-space（Cheney 半空间复制）
  • Old Generation：大、长寿对象、Mark-Compact 偶尔回收；又分 Old Pointer Space / Old Data Space
  • Code Space：JIT 编译产物（机器码）
  • Map Space：Hidden Class（Map 对象）单独区域
  • Large Object Space：> 大于 spaces 的对象（典型 1MB+）
  • Trusted Space / Trusted Code Space（V8 Sandbox 引入）

Tagged Pointer（V8 表示一切的方式）

正文待写。

  • 64-bit 系统：每个 V8 值是 64-bit 字
  • 最低位 = 0 → SMI (Small Integer)，剩 31 位整数（compressed pointer 模式下）
  • 最低位 = 1 → HeapObject 指针（实际指向堆，地址按 4 字节对齐所以低位为 0；运行时 -1 还原）

SMI 的代价

正文待写。

  • 31-bit signed range：-2^30 到 2^30 - 1
  • 超出 → 升级成 HeapNumber（堆上 8 字节 IEEE 754 double）
  • 数组下标、循环计数器都希望落在 SMI 范围

Pointer Compression（V8 8.0+，2020）

正文待写。

  • 问题：64-bit 系统的指针字段占 8 字节，但 V8 堆 < 4GB → 高 32 位都一样
  • 方案：在虚拟地址空间预留 4GB 区域（isolate cage），所有堆对象住在里面；对象内部指针字段用 32-bit offset 表示
  • 效果：堆 -43%、指令 cache 命中率提升、SMI 改成 31-bit（与压缩指针一致）
  • 代价：堆上限 4GB（之前可以无限）；进入 isolate 时要计算 base

HeapObject 通用 layout

layout 
[ Map pointer (4B) ]    ← Hidden Class（即 [[Class]]）
[ Properties (4B)  ]    ← 指向 properties backing store 数组
[ Elements (4B)    ]    ← 指向 elements backing store（数组元素）
[ in-object slot 0 ]    ← 前 N 个属性直接存这里
[ in-object slot 1 ]
...

String 表示（多种）

正文待写。

  • SeqOneByteString：纯 ASCII 单字节
  • SeqTwoByteString：UTF-16
  • ConsString：a + b 字符串拼接的"延迟链表"（不立即合并）
  • SlicedString：另一字符串的子串（共享 backing）
  • ExternalString：指向 V8 外部内存（嵌入器分配）
  • InternalizedString：去重的字符串池（用于属性名）

Conservative Stack Scanning（V8 11.2+，2023）

正文待写。

  • 背景：V8 GC 一直是精确式 —— 每个栈位置都要精确知道是否是指针。要求编译器维护精确的 stack maps，复杂且慢
  • 做法：GC 扫描栈时，把所有"看起来像指针的字"都当作潜在 root —— 可能多保留一些对象（保守），但允许编译器丢掉 stack maps
  • 为什么现在能这么做：Pointer Compression 让指针只能落在 4GB cage 内，"看起来像指针"的判断准确率极高；误判保留的对象很少
  • 收益：编译器代码简化、Maglev 编译速度可观提升；为 V8 与 Cppgc / Oilpan 互操作铺路

04 典型代码

javascript 
// SMI vs HeapNumber 边界
const small = 100;            // SMI（fast path）
const big = 2 ** 31;          // HeapNumber（堆分配）
const float = 1.5;            // HeapNumber（堆分配）

// d8 用 %DebugPrint 看类型
%DebugPrint(small);  // type: Smi
%DebugPrint(big);    // type: HeapNumber
%DebugPrint(float);  // type: HeapNumber

05 常见陷阱

Pointer Compression 的 4GB 上限是 per-isolate 不是进程

多 isolate（Workers）互不影响。

--max-old-space-size 不是总堆上限

设的是 Old Gen 上限；总堆 ≈ Old + 1.5×Young + Code + ...

ConsString 内存延迟释放

切片大文本可能让源字符串永不能 GC（用 SlicedString 也有类似问题，需要 flatten）。
