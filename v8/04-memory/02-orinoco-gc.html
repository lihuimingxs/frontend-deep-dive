Orinoco GC：Parallel + Concurrent + Incremental

Orinoco 是 V8 的 GC 项目代号（2016 启动，至今仍在演进）。把"stop-the-world 单线程 GC"重构成"主线程几乎不停"。三件套：Parallel Scavenger（year gen，主线程暂停 1-3ms）+ Concurrent Marking（old gen，heavy WebGL 暂停 -50%）+ Mark-Compact。再讲与 Cppgc / Oilpan 的双堆协作 —— V8 工程上最复杂的 GC 集成之一。

阅读时长 ~35 分钟 前置 P4.1 · 堆布局 主要来源 v8.dev/blog/trash-talk · concurrent-marking · orinoco-parallel-scavenger

01 速览

  1. "主线程几乎不停"是 Orinoco 的核心叙事：传统 mark-sweep 在长串引用上能 stop world 几百毫秒；并发 + 并行 + 增量三件套把这个降到 ~10ms 内。
  2. 分代假说驱动设计：大多数对象死得早 → 把堆分代，年轻代频繁回收，老年代偶尔回收。
  3. 性能优化基本只关注年轻代频率：年轻代 GC 极频繁（每秒数十次）但每次 1-3ms；老年代 GC 罕见但每次更长。

02 底层逻辑

分代假说：90% 对象死得早

GC 设计的根本问题是规模——一个长跑应用堆里可能有几百万对象，每次 GC 都全扫一遍代价太大。Orinoco（和绝大多数现代 GC）用分代假说避免全扫：

统计观察：JS 程序里 90%+ 的对象在第一次 GC 时已经不可达。临时变量、循环内的中间结果、API 请求的临时 payload——大部分对象从生到死跨度极短。

设计推论：把堆分成"年轻代"和"老年代"，频繁回收年轻代（因为大多数死得早，回收 ROI 高），偶尔回收老年代（因为活下来的多半还会继续活）。这就是 Orinoco 的"双频道"设计：Minor GC 每秒几十次但每次极快、Major GC 几秒到几分钟一次但每次更深。

Minor GC（Young Gen）—— Parallel Scavenger

年轻代的回收算法叫 Cheney 半空间复制（1970 年 Cheney 发表的经典算法），是 V8 NewSpace 的工作原理。

Cheney 半空间复制：From → To

NewSpace 内部分两半（详见 P4.1）：From-space 和 To-space。新对象总是分配在 From-space。Minor GC 触发时：

  1. 从根集合（栈、全局、handle）出发遍历指针图，找出 From 中所有活的对象
  2. 把每个活对象拷贝到 To-space，原 From 位置留下"forwarding pointer"指向新位置
  3. 更新所有指向这些对象的指针为新位置
  4. From-space 整体丢弃（不需要逐个回收死对象——就当 From 整个不存在）
  5. From / To 角色互换，下次 GC 反向

这套设计精妙在不需要 sweep——死对象自动丢弃。代价是每次都要拷贝活对象，但因为分代假说活对象少（10% 以下），拷贝量小。分配也极快——只是 From-space 的指针递增，不需要 free list 管理。

Parallel Scavenger（V8 6.2+，2017）：多线程 work stealing

原始 Cheney 是单线程。V8 6.2 起改成 Parallel Scavenger——多个 helper thread 并行扫描和拷贝，用 work-stealing queue 平衡负载（一个 thread 队列空了就从别的 thread 队列偷活干）。

效果：典型 Minor GC 主线程暂停 1-3ms。这是 60FPS 流畅渲染（每帧 16.7ms 预算）时不会被察觉的级别。主线程仍要暂停——Cheney 复制要更新所有指针，并发 marking 不适用（详见后文 Major GC）；但 helper threads 大幅缩短了 stop-the-world 窗口。

Promotion：晋升到 Old Gen

活过 N 次 Minor GC（默认 N=2）的对象会被晋升到 Old Gen——V8 判断它"已经活过初始阶段，大概率是长寿对象"。晋升机制实现简单：拷贝到 Old Gen 而非 To-space。

晋升不可逆——一旦进 Old Gen 就永远不会回 NewSpace。这给一条避坑建议：不要在长循环中持续创建大对象——前几次循环创建的对象活过 2 次 Minor GC 后晋升 Old Gen，给 Major GC 增加压力。如果对象本来就是临时用的，应该控制循环节奏让它们在 Minor GC 阶段就被回收。

💡 NewSpace 满了不一定立即 GC

实际上 V8 在 NewSpace 满时会触发 Minor GC——但 NewSpace 容量是动态调整的（受 --max-semi-space-size 上限）。短跑应用 NewSpace 较小、GC 频繁但快；长跑服务端应用 NewSpace 自适应变大，每次 GC 间隔更长但压力分散。这套自适应 V8 自己调节，几乎不需要手动干预。

Major GC（Old Gen）—— Concurrent Mark-Compact

老年代不能用 Cheney——老年代占堆的 80%+，复制半空间需要预留同样大的 To-space，浪费太多。Old Gen 用经典 Mark-Compact（标记 + 整理）算法，分 3 阶段：

阶段 1 · Marking：找出所有活对象

从根集合出发遍历指针图，把所有可达的对象标记为"活"，没标的就是死的。这是3 阶段中最耗时的——遍历整个对象图，每个对象至少访问一次。

原始 mark-sweep GC 在这阶段必须 stop-the-world——主线程暂停 100-500ms 是常态。这种暂停在 60FPS 应用里是致命卡顿。Orinoco 的核心创新就是把 Marking 阶段并发化：

  • Concurrent Marking（V8 6.4+，2018）：marking 在 helper thread 上做，主线程继续跑 JS。主线程偶尔需要短暂暂停做"safepoint"——确保 helper 看到一致的对象图状态。
  • Incremental Marking 兜底：如果某些场景 concurrent 不可用（如 isolate 没启用 helper threads），主线程做小步增量——每次 1-5ms，多次累积完成 marking。

典型 heavy WebGL 游戏暂停时间从 100ms 级降到 10ms 级，暂停 -50%。

阶段 2 · Sweeping：回收死对象空间

Marking 完成后，遍历堆把所有"未标记"的对象空间登记到 free list，可供后续分配复用。

Sweeping 也能并发——Concurrent Sweeping：helper thread 在主线程跑 JS 时同步整理 free list。不需要 safepoint，因为 sweeping 不修改活对象图，只动死对象空间——主线程根本不可能访问到这部分。这是 Orinoco 设计的精彩点：分清"哪些操作改活图（需要 safepoint）"和"哪些只改死区（可完全并发）"。

阶段 3 · Compacting（可选）：消除碎片

Sweeping 后空间被回收但内存碎片化——free list 是一堆零散的小空段。Compacting 把活对象搬到一起，让自由空间合并成大块。Parallel Compaction：多 helper thread 同时移动对象 + 更新所有指向它们的指针。

Compacting 不是每次都做——开销大（要更新所有指针），只在碎片严重影响分配时触发。这是 V8 自动判断的——典型几次 Major GC 才做一次 compact。

Tri-color Abstraction：并发 marking 怎么不漏？

并发 marking 有个核心难题：marking 在 helper thread 跑，主线程同时在改对象图——主线程新建对象、改属性、删引用——helper 怎么保证不漏标活对象？答案是经典的 Tri-color Abstraction（三色标记）+ Write Barrier。

每个对象在 marking 阶段属于三种颜色之一：

  • White（白）：还没被访问过——marking 结束时仍是 white 的就是死对象，被回收
  • Grey（灰）：自己已访问，但子节点还没全访问——marker 工作队列里待处理
  • Black（黑）：自己和所有子节点都已访问——已经"安全确认活"

marking 流程：根从 black 起步 → 把直接引用的对象标 grey 进队列 → 队列里取出 grey 对象，把它引用的对象标 grey 进队列、自己变 black → 队列空时 marking 完成。

问题：主线程并发跑时如果做 black_obj.foo = white_obj（黑对象指向白对象），白对象再也不会被 marker 访问到——会被错误回收。这种情况叫"漏标"，是并发 marking 的致命 bug。

Write Barrier：写指针的安全网

解决漏标的机制是 Write Barrier——每次写指针都做一段额外检查代码，发现可能漏标时纠正。V8 用一种叫 Snapshot-At-The-Beginning (SATB) 的策略：

逻辑伪代码：

write-barrier-pseudo 
// 每次 obj.field = value 翻译成：
function write_field(obj, field, value):
  // SATB 写屏障
  if marking_in_progress and value is white:
    mark_grey(value)   // 把 value 加进 marker 队列
  obj[field] = value

这样无论主线程怎么改图，helper marker 都不会漏看任何曾经被引用过的对象。代价是每次指针写多几条机器指令——这是分代 GC 的"固定税"。在 hot path 上累计可观（典型 +5-10% 写指针开销），但相比 stop-the-world 暂停，绝对值得。

Remembered Set：跨代引用的索引

Write Barrier 还解决另一个问题——跨代引用。Minor GC 只看 NewSpace，但 Old Gen 可能持有指向 NewSpace 的指针（如 cache 引用新对象）——如果不知道这些跨代引用，会错误回收 NewSpace 中其实活着的对象。

V8 让 Write Barrier 多做一件事：检测到 old → young 的写指针，记录到 Remembered Set。Minor GC 时把 Remembered Set 当作"额外 root"——既扫栈和全局，也扫这些跨代引用源。

Remembered Set 的实现是 card table（粗粒度）或精确 slot 集——card table 把堆划成 512 字节的 card，记录每个 card 是否含跨代指针。Minor GC 时只要扫"标了的 card"——不需要扫整个 Old Gen，效率极高。

GC 调优旋钮：什么时候手动调，什么时候不要碰

V8 自适应 GC 行为的能力很强——大多数应用完全不需要手动调。但有几个场景值得知道这些 flag 存在：

常用 Node 命令行 flag

  • --max-old-space-size=N（MB）—— old gen 上限。Node 默认 1.5-4GB（视版本）。服务端长跑应用有时需要调大（如 10GB+ 内存数据库）。客户端 / Workers 默认就够。
  • --max-semi-space-size=N（MB）—— NewSpace 半空间大小。默认 16MB-64MB 自适应。极少手动调。
  • --gc-interval=N —— 每 N 次分配强制触发 GC。仅 debug 用——压力测试 GC 行为时用。
  • --trace-gc / --trace-gc-verbose / --trace-gc-nvp —— 打印每次 GC 详情：触发原因、暂停时间、young/old 比例、堆大小变化。性能调优必备。
  • --expose-gc + global.gc() —— 主动触发 Major GC。仅测试用，生产严禁（详见后文陷阱）。

什么时候真的需要调？

典型有效场景：

  • 服务端 Node 应用 OOM：长跑服务堆涨到 1.5GB（Node 默认上限）就 OOM——确认确实需要这么多内存（不是泄漏）后，调 --max-old-space-size=4096 或更高。
  • 大数据处理脚本：批处理 10GB+ JSON / CSV 时调大 max-old-space-size 是必须的，否则 V8 会强制 OOM 而非 swap。

不该调的场景：

  • "应用感觉慢"——这通常是 deopt loop / megamorphic IC（详见 P3.2），跟 GC 无关
  • "GC 暂停太长"——先用 --trace-gc 看暂停频率和时长。Orinoco 默认就很优秀，多半暂停是 1-3ms 级别，"觉得长"通常是错觉
  • "想让 GC 更激进"——调小 NewSpace 反而让 promotion 更频繁，给 Old Gen 增加压力，整体性能下降

Cppgc / Oilpan 与 V8 的"双堆协作"

一个浏览器场景的复杂问题：DOM 节点是 C++ 对象（Blink 创建），JS 包装对象（如 document.body）是 V8 堆对象——两边互相引用。光靠 V8 GC 看不到 C++ 端的对象图（不知道 DOM 节点之间的父子关系）；光靠 C++ GC 看不到 JS 端（不知道 JS 闭包持有了哪个 DOM 节点）。谁都不能独立判断对象是否可回收。

从 Oilpan 到 Cppgc：DOM GC 的演化

Blink 团队为 DOM 写了一个 C++ GC 叫 Oilpan（2014 年起）——支持 mark-sweep + concurrent，类似 Orinoco 但工作在 C++ 对象上。2020 年起 Oilpan 从 Blink 抽离，独立成 V8 的 Cppgc 库——这样不只 Blink、其他嵌入器（如 Cobalt 之类）也能用同一套 C++ GC。

Unified Heap：跨堆 marking 协作

V8 与 Cppgc 协作的设计叫 Unified Heap——两个独立的 GC 在 marking 阶段互相回调对方:

  1. V8 marker 走到一个"指向 C++ 对象的引用" → 回调 Cppgc 让它继续标记 C++ 对象图
  2. Cppgc marker 走到一个"指向 V8 对象的引用" → 回调 V8 让它继续标记 V8 对象图
  3. 两边并发运行，互相喂活对象
  4. 双方都达成"全图标记完成"才算 marking 完成

这设计精妙在不需要把两个堆合并——V8 和 Cppgc 各自独立的 GC 算法、生命周期、暂停时机都保留，只在 marking 阶段做"调用同步"。这就是 Unified Heap 这个名字的由来——逻辑上一个统一的可达性图，物理上两个独立堆。

TracedReference：Embedder 持有 V8 对象的安全 handle

Embedder 端 C++ 代码持有 V8 对象时不能直接持 raw pointer（V8 GC 会移动对象）。需要用 TracedReference<T>——这种 handle 类型参与 Unified Heap 的 marking——V8 GC 移动对象时它跟着更新指针。详见 P5.1 Embedder API。

P4.1 介绍的 Conservative Stack Scanning 是 Unified Heap 的前置——C++ 端栈本来就只能保守扫描（C++ 编译器不维护精确 stack maps），V8 端切换到保守扫描后两边一致，跨堆 GC 才好做。这是又一例"老优化解锁新优化"——Pointer Compression → Conservative Stack Scanning → Unified Heap，三步链式演化。

📊 Chrome 内存表现一半看 V8 一半看 Oilpan

DOM 操作密集的场景（频繁创建/删除节点的 SPA），Oilpan 的回收效率直接影响内存占用。Chrome 改进 V8 GC 的同时也持续改进 Cppgc——两个 GC 协同优化才能让 Chrome 在高强度 web 应用下保持内存稳定。这也是为什么 Node 服务端 GC 调优经验（只调 V8）不能直接套用到 Chrome 扩展 / Electron 应用——后者的内存压力一半在 Cppgc 那边。

03 应用场景

场景 A · Chrome DevTools Memory Profiler 抓 retainer 链

内存泄漏诊断的标准流程：DevTools → Memory → Take heap snapshot → 等待几秒 → 重复操作（点击、路由切换等）→ 再 snapshot → Comparison 视图比对两次差异。重点看"Allocated but not freed"对象，沿 Retainer 树往上看谁持有它们——通常根因是某个全局缓存 / event listener / closure 没释放。注意 DevTools 抓的是逻辑 retain 图，不是 V8 实际堆布局——一些 V8 内部结构（Map / Code 等）DevTools 不显示。

场景 B · 用 --trace-gc 看频率与暂停时间

Node 服务端调优起点：node --trace-gc app.js。输出每行对应一次 GC，包含触发原因（Scavenger / Mark-Sweep / Mark-Compact）、暂停时间（实际 stop-the-world 时长）、堆变化（before/after 大小）。健康指标：Minor GC 1-3ms 暂停 + 频率 < 50/秒、Major GC < 50ms 暂停 + 频率 < 1/分钟。如果某项严重超标，看是否对象寿命异常（不该长存的对象在 Old Gen）或某些路径分配过多临时对象。

场景 C · 服务端 Node 长跑应用堆调优

Node 默认 --max-old-space-size 是 1.5GB（< v18）或 4GB（v18+）——服务端应用涨过这个值就 OOM。调整步骤：(1) 用 --trace-gc + Memory Profiler 确认确实需要更大堆（不是泄漏）；(2) 调到接近物理内存的合理值（如 8GB / 16GB）；(3) 监控生产环境实际峰值堆占用决定是否进一步调整。不要随意调到极大值——大堆会让 Major GC 暂停时间显著增加（Old Gen 大 → marking 时间长）。

场景 D · Electron / Chrome 扩展的 DOM 内存压力

DOM 操作密集场景（如可视化编辑器、画布应用、富文本编辑器），内存压力一半在 Cppgc 那边——只调 V8 flag 没用，要用 DevTools 看 DOM 节点数量 / Detached DOM Nodes（"已从 DOM 移除但 JS 还引用"的节点是常见泄漏源）。Electron 应用还要注意 Main Process（Node）和 Renderer Process（Chrome）各自有独立 V8 堆——内存峰值要分别看。

# Node 启动时调优
node --max-old-space-size=4096 \

05 常见陷阱

"强制 GC"的 global.gc() 只在 --expose-gc 标志下可用

生产不要用 —— 你不比 V8 知道更多。

大数组 promote 到 old gen 后不会被 promote 回 young

意味着短暂的大数组也走重 GC 路径。

FinalizationRegistry / WeakRef 不保证立即调用 finalizer

V8 把它放在 idle task 里，不要用作资源释放（详见 ECMA P6.2 · Reflect & WeakRef）。

DevTools Memory Profiler 抓的是"逻辑 retain 图"

和 V8 实际堆布局不完全对应。
