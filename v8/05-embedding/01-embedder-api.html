Embedder API：Isolate / Context / Handle

从 C++ 视角看 V8。Embedder API 是 Node / Deno / Workers 通用的接入面 —— 理解它就理解了"为什么 Worker Threads = 多个 Isolate、为什么 Cloudflare Workers 冷启动能 < 5ms"。讲清核心对象层级（Isolate ≈ Agent / Context ≈ Realm / HandleScope / Persistent）、Snapshot 启动加速、Promise / Microtask Hooks，以及 C++ ↔ JS 互调（FunctionTemplate / N-API）。

阅读时长 ~35 分钟 前置 ECMA P7.1（执行模型） 主要来源 v8.dev/docs/embed · Node embedding API · Cloudflare Workers 设计文档

01 速览

  1. Isolate 是隔离的最强保证：JS 代码绝无可能跨 Isolate 看到对方的对象 —— 这是 Workers / Cloudflare Workers / Lambda@Edge 的安全基础。
  2. Snapshot 是工程奇迹：把 built-ins 构造结果序列化到磁盘，启动时直接 mmap 进堆，毫秒级冷启动。
  3. V8 不拥有 Event Loop：microtask drain 时机由宿主决定（Node 在每个 task 结束时 drain；浏览器在 task + 渲染前 drain）。

02 底层逻辑

核心对象层级

layout 
v8::Isolate            ← 一个独立的 V8 实例（自己的堆、GC、optimizer）
   │
   ├── v8::Context     ← JS 的全局执行环境（≈ Realm）
   │   ├── globalThis
   │   ├── built-ins (Object, Array, Promise, ...)
   │   └── 用户代码
   │
   ├── v8::HandleScope ← Local handles 的生命周期容器（栈分配）
   │
   └── v8::Persistent  ← 跨函数/跨 scope 持有的引用

Isolate ≈ ECMAScript Agent：隔离的最强保证

Isolate 是 V8 嵌入器最重要的抽象——可以理解为"一个完整独立的 V8 实例"：自己的堆、GC、optimizer、bytecode cache。多个 Isolate 之间绝对没有共享状态——Isolate A 里的 JS 对象在 Isolate B 里根本不可见。这种强隔离是 V8 在多租户、多 worker、安全沙箱场景下的基础。

对应到 ECMAScript 规范，Isolate 大致对应一个 Agent（详见 ECMA P7.1）——规范定义的"独立执行环境，自己的 task queue、microtask queue、agent state"。V8 用 Isolate 这个名字主要是历史原因（Isolate 概念 2011 年引入时 Agent 还没正式定义）。

线程约束：一次一个，要并发用 Locker

一个关键约束：一个 thread 在任意时刻只能进入一个 Isolate。进入用 Isolate::Scope（栈上 RAII 对象）：

cpp 
v8::Isolate* isolate = v8::Isolate::New(create_params);
{
  v8::Isolate::Scope isolate_scope(isolate);  // 进入
  v8::HandleScope handle_scope(isolate);
  // ... JS 工作 ...
}  // 自动退出
isolate->Dispose();

多 thread 想轮流用同一 Isolate？必须用 v8::Locker / v8::Unlocker——但这种用法很少（大多数 embedder 走"一 thread 一 Isolate"模型）。Worker Threads / Cloudflare Workers 模型就是直接每 worker 一个 Isolate，互相完全隔离——通信只能通过结构化 clone（postMessage）跨 isolate 复制数据，没有共享内存（除非用 SharedArrayBuffer）。

Snapshot：Isolate 启动加速的核心

一个 Isolate 启动需要做什么？构造所有 ECMAScript 内置对象——Object.prototype 上几十个方法、Array.prototype 上几十个、Promise / Map / Set / 正则 / Date / Intl... 加起来几百个 built-in。这些 built-in 是用 Torque/CSA 写的（详见 P2.2），但它们的"对象图"——prototype 链、property descriptor、闭包等——必须运行时构造起来。

没有任何加速的话，每次 Isolate 启动要花 50-100ms 跑这堆初始化代码。Chrome 启动 1000 个 Isolate（每个 tab 一个）就要 50 秒——明显不可接受。这就是 Snapshot 机制要解决的问题。

Startup Snapshot：把堆状态烤成二进制

V8 在编译期就跑一遍所有 built-in 初始化，然后把结果"对象图整体序列化成一段二进制 blob"——叫 Startup Snapshot，存在 V8 binary 里。Isolate 启动时直接 mmap 这段 blob 到内存，然后做必要的指针修补（fix-up，因为虚拟地址可能不同）——不需要执行任何初始化代码。冷启动从 50-100ms 降到 ~3-5ms。

这是为什么 V8 binary 比想象的大——里面包含了完整的预初始化对象图。但用户拿到的收益是巨大的：每个新 Isolate 启动几乎瞬时。

Custom Snapshot：嵌入器自己烤业务初始化

V8 还允许嵌入器构造自己的 Snapshot。典型用法：先创建一个 Isolate、跑业务的"初始化阶段"（如加载 React framework、注册全局 helper、初始化 i18n 字典），然后调 v8::SnapshotCreator::CreateBlob 把当前堆状态序列化。下次启动直接用这个 custom snapshot——得到的是已经初始化好业务的 Isolate，连 React 都已经加载完成。

Cloudflare Workers 的核心 trick 就在这里——他们的 Worker 启动模板是个 custom snapshot，包含 Workers Runtime API（fetch、addEventListener、KV、R2 等）已经全部 ready。每个 Worker 实例从这个 snapshot 启动，毫秒级冷启动 + ~5MB 内存就能跑起来——这是单进程跑几千 Workers 的基础。

Snapshot vs Code Cache：两个正交机制

很多人混淆这俩。区别：

  • Snapshot 缓存 V8 built-ins + 业务初始化的堆状态——一次性烤好，所有 Isolate 共享
  • Code Cache（详见 P2.2）缓存 用户脚本的 bytecode——按 source hash 缓存，多次访问加速

两者正交：Chrome 同时用 Snapshot（Isolate 启动）+ Code Cache（页面 JS 解析）；Workers 用 Snapshot 大量但 Code Cache 较少（短脚本意义不大）。Node 也都用，N-API 模块还能挂自己的 Snapshot。

跨版本不兼容

Snapshot 格式与 V8 commit hash 强绑定——任何 V8 升级都要重新生成。这就是为什么用 Snapshot 的嵌入器（Workers / Deno / 自定义嵌入）不能直接用 prebuilt V8——必须自己 build V8 + 在同一 build 里生成 snapshot。这是嵌入 V8 的工程门槛之一。

Context ≈ ECMAScript Realm：JS 全局执行环境

一个 Isolate 内可以有多个 Context——每个 Context 是一个完整的 JS 全局执行环境，对应 ECMAScript 规范的 Realm 概念（详见 ECMA P7.1）。每个 Context 有独立的 globalThis 和独立的 built-in 实例（自己的 Object、Array、Promise 构造器）。

典型多 Context 场景：Chrome iframe（每个 iframe 一个 Context，但同源 iframe 共享 Isolate）、Node 的 vm 模块（vm.createContext() 创建独立的 Context 跑沙箱代码）、Electron 的 contextIsolation（preload 脚本和 Renderer 在不同 Context 中跑，防止相互访问）。

Context 与 Isolate 的关系：底层共享但表面隔离

一个有趣的细节：多个 Context 共享 Isolate 的底层 Map / Code / 内置 stub——这是因为 hidden class 系统全局共享、JIT 编译产物全局复用。这意味着同一 Isolate 内的两个 Context "看似隔离实则部分共享"——攻击者通过 prototype pollution 跨 Context 是真实的安全威胁，所以 Chrome 才有 Site Isolation（不同 origin 强制不同进程）作为额外防线。

进入 Context

跑 JS 之前必须显式进入某个 Context（用 v8::Context::Scope 或显式 Enter/Exit）。代码示例：

cpp 
v8::Local<v8::Context> context = v8::Context::New(isolate);
{
  v8::Context::Scope context_scope(context);  // 进入 Context
  // 现在可以编译/执行 JS
  v8::Local<v8::String> source = v8::String::NewFromUtf8Literal(isolate, "1 + 1");
  v8::Local<v8::Script> script = v8::Script::Compile(context, source).ToLocalChecked();
  v8::Local<v8::Value> result = script->Run(context).ToLocalChecked();
}  // 退出

Handle 系统：GC 安全的指针

一个核心问题：C++ 代码持有 V8 对象指针时，V8 GC 可能移动对象（Cheney 复制 / Compacting）——直接持 raw pointer 会指向无效内存。所以 V8 不允许 C++ 直接持指针，必须用 Handle——一种"GC 安全"的间接引用，GC 移动对象时 V8 自动更新所有 handle 指向新地址。

V8 有 4 种主要 handle，按生命周期和用途选用。

Local<T>：栈上临时引用

最常用的 handle 类型。所有 Local 必须归属一个 HandleScope——HandleScope 是栈上 RAII 对象，scope 销毁时所有 Local 一次性失效（V8 内部释放对应槽位）。这套设计让 Local 几乎零运行时开销——分配只是 HandleScope 内部数组 push 一项，释放是 scope 析构时整体丢弃。

cpp 
void DoWork(v8::Isolate* isolate) {
  v8::HandleScope scope(isolate);  // 创建 scope
  v8::Local<v8::String> str = v8::String::NewFromUtf8Literal(isolate, "hello");
  v8::Local<v8::Number> num = v8::Number::New(isolate, 42);
  // ... 用 str 和 num ...
}  // scope 销毁，str 和 num 自动失效

常见错误：忘记 HandleScope。Local 永不被释放——内存泄漏。V8 在 debug build 会对未在 HandleScope 中创建的 Local crash。

Persistent<T>：堆上长期引用

需要跨函数 / 跨 scope 持有 V8 对象时用 Persistent——存在堆上，由 embedder 显式管理生命周期。必须显式 .Reset() 释放，否则 V8 GC 永不能回收对象（即使该对象已经"逻辑上"不再被需要）。

cpp 
class MyEmbedder {
  v8::Persistent<v8::Function> callback_;  // 字段持有

public:
  void SetCallback(v8::Local<v8::Function> cb) {
    callback_.Reset(isolate_, cb);  // 从 Local 转换为 Persistent
  }
  ~MyEmbedder() {
    callback_.Reset();  // 必须显式释放
  }
};

Eternal<T>：永不释放

一些值（如 hidden global 符号、内置原型）整个进程生命周期都不释放——用 Eternal 比 Persistent 更轻量（V8 内部跳过释放检查）。常用于嵌入器初始化时构造的"永不变"对象。

TracedReference<T>：Cppgc/Oilpan 集成

P4.2 介绍过：当 C++ 持有的 V8 对象需要参与 Unified Heap 跨堆 GC 时（典型 Chrome / Electron 场景），用 TracedReference<T>。它会在 Cppgc marker 标到这个引用时回调进 V8 让 V8 marker 继续。详见 P4.2 · Cppgc 协作。

⚠️ Handle 类型选择决策树

(1) 函数内临时用 → Local + HandleScope，最常见
(2) 跨函数长期持有 → Persistent，记得 Reset
(3) 整个进程永不释放 → Eternal，省一点开销
(4) 需要参与 Cppgc 跨堆 GC → TracedReference，Chrome / Electron 场景

Promise / Microtask Hooks：V8 不拥有 Event Loop

一个常被忽略的事实：V8 自己不拥有 Event Loop——它只暴露 microtask queue 的原语，由 embedder 决定怎么调度。这是 V8 与宿主分工的关键边界——浏览器有 task source / requestAnimationFrame / 渲染管线等复杂调度，Node 有 libuv 的 phases，Workers 有自己的极简调度——它们都从 V8 读取微任务但用各自的策略 drain。

Microtask Checkpoint：drain 时机由宿主决定

V8 暴露 Isolate::PerformMicrotaskCheckpoint()——调用一次会执行所有累积的 microtask（即所有 pending Promise.then / await 后续）。什么时候调由 embedder 决定：

  • Chrome / 浏览器：每个 task 结束时 + 渲染前调一次，符合 HTML 规范的 microtask 时机
  • Node 默认：每次 V8 API 调用返回时调一次（"scope_" 模式），保证 native 代码每次给回 JS 控制权前 microtask 都已 drain
  • Workers：每个 fetch event handler 结束时 drain，没有跨事件持续 microtask

这就是为什么同一段 await/then 代码在不同宿主里时序可能略有差异——V8 只决定 microtask 在 macrotask 之前跑（规范要求），但具体哪一刻 drain 是宿主选的。详见 JS P3 异步 + ECMA P7.3 Event Loop 的规范论述。

Promise Hook：观察 Promise 生命周期

Embedder 通过 Isolate::SetPromiseHook(callback) 注册 hook，V8 在 Promise 生命周期的 4 个时点回调：

  • Init：新 Promise 被创建
  • Resolve：Promise resolve 或 reject
  • Before：Promise 关联的 microtask 即将执行
  • After：Promise 关联的 microtask 已执行完

Node 的 async_hooks 模块就是包这层 V8 接口给 JS 用的——可以追踪 async context 在异步链中传递，是 OpenTelemetry / 分布式追踪的基础设施。代价是 hook 开销不小（每个 Promise 都额外回调），生产环境如果未启用 async tracking 这层是关掉的。

C++ ↔ JS 互调：FunctionTemplate / ObjectTemplate / N-API

嵌入 V8 的核心需求之一：把 C++ 函数 / 对象暴露给 JS 调用。Node 的 fs.readFile、Workers 的 fetch、Electron 的 app.getVersion——背后都是 C++ 代码通过 V8 嵌入 API 暴露给 JS。

FunctionTemplate：定义 C++ 函数

创建一个能从 JS 调用的 C++ 函数：

cpp 
// C++ 实现
void GetVersion(const v8::FunctionCallbackInfo<v8::Value>& args) {
  v8::Isolate* isolate = args.GetIsolate();
  args.GetReturnValue().Set(
    v8::String::NewFromUtf8Literal(isolate, "1.0.0")
  );
}

// 注册到 JS 全局
v8::Local<v8::ObjectTemplate> global = v8::ObjectTemplate::New(isolate);
global->Set(
  v8::String::NewFromUtf8Literal(isolate, "getVersion"),
  v8::FunctionTemplate::New(isolate, GetVersion)
);
v8::Local<v8::Context> context = v8::Context::New(isolate, nullptr, global);

// JS 端使用：
// getVersion()  →  返回 "1.0.0"

ObjectTemplate：定义 C++ "类"

ObjectTemplate 是 FunctionTemplate 的"对象版"——定义一个 JS 对象的 shape，包含哪些方法、属性。Node 的 fs module、Workers 的 caches object 都是这样构造出来的。可以挂 internal field 让 C++ 端往 JS 对象里塞 native 数据（如把 C 文件描述符 fd 存进 JS 的 FileHandle 对象）。

N-API：Node 给 native module 的稳定 ABI

直接用 V8 嵌入 API 的最大问题：V8 升级会破坏二进制兼容性。Node 14 编译的 native module（如 node-sqlite3）拿到 Node 22 跑可能 crash 或行为异常——V8 12 的 API 与 V8 9 已有不兼容变更。

N-API（也叫 Node-API）是 Node 在 V8 之上封的一层ABI 稳定接口——保证用 N-API v8（或 v9）写的 native module，从此后 Node 版本编译的二进制都能跑，不需要重新编译。代价是 N-API 更冗长（每个 V8 操作都要做错误码检查 + 转换），但工程价值巨大。

三层选择：raw V8 / N-API / node-addon-api

  • raw V8：直接用 V8 C++ API，最灵活、性能最好、版本绑定最严。适合 V8 自家工具、Chrome 内部、Workers / Deno / 自定义嵌入
  • N-API（C 接口）：跨 Node 版本二进制兼容，写法 verbose（C 风格）。适合"必须支持多 Node 版本"的库（npm 上大多数 native module）
  • node-addon-api（C++ 包装）：N-API 上的 C++ 类包装，写法接近 raw V8 但保持 N-API 兼容性。最常用，是 95% 现代 native module 的选择

决策：除非你在写 V8 自家组件或自己嵌入 V8，否则用 node-addon-api——既好写又跨版本兼容。原始 N-API 留给极致性能场景。

03 应用场景

场景 A · 写 native module 给 Node

典型库如 node-sqlite3（SQLite 绑定）/ bcrypt（密码哈希原生实现）/ sharp（图像处理）。选 node-addon-api——C++ 类包装，跨 Node 版本兼容，写法接近 raw V8 体验。模板项目用 node-gyp + binding.gyp 配置编译。要绑定的 C++ 库通常是 libsqlite3 / libbcrypt / libvips——native 性能优势直接传给 JS 用户。

场景 B · 自己嵌入 V8

非 Node 场景：游戏引擎用 V8 跑脚本逻辑（Cocos / Unity 早期版本）、数据库用 V8 跑 stored procedures（MongoDB 的 Atlas Functions）、自定义运行时（Workers / Deno / Bun-like 项目）。直接用 raw V8 API + libplatform——参考 V8 自带的 samples/hello-world.cc 和 samples/process.cc 做起步。注意 build 系统：必须用同一份 V8 build 生成 Snapshot，所以不能用 prebuilt V8——通常用 depot_tools 拉源码自己 build。

场景 C · 理解 Worker Threads 的隔离限制

Node Worker Threads 不是 OS 线程也不是协程——是独立的 V8 Isolate。理解这点解释了所有"为什么 Worker 不能..."的问题：(1) postMessage 必须 structured clone 或 transfer——不能直接传引用，因为对方 Isolate 的对象图不可见；(2) 不共享 V8 全局——Worker 里改 Array.prototype 不影响主线程；(3) SharedArrayBuffer 是唯一共享内存——它在 V8 之外（ArrayBuffer Allocator 那层），多 Isolate 都能挂。Cloudflare Workers / Deno Workers 同样模型。

场景 D · Electron 应用 contextIsolation

Electron 默认 Renderer 进程能直接看到 Node 全局——这是历史包袱也是巨大安全洞。contextIsolation: true（v12+ 默认）让 Renderer 和 preload 脚本运行在不同 Context——Renderer 看不到 Node、preload 通过 contextBridge.exposeInMainWorld 显式暴露白名单 API。这就是 V8 多 Context 在 Electron 里的实际应用——把"看似隔离实则部分共享"做成了"显式控制隔离边界"。

04 典型代码

cpp 
// 嵌入 V8 的最小代码骨架
V8::InitializeExternalStartupData(...);
Isolate::CreateParams create_params;
create_params.array_buffer_allocator = ArrayBuffer::Allocator::NewDefaultAllocator();
Isolate* isolate = Isolate::New(create_params);
{
  Isolate::Scope isolate_scope(isolate);
  HandleScope handle_scope(isolate);
  Local<Context> context = Context::New(isolate);
  Context::Scope context_scope(context);

  Local<String> source = String::NewFromUtf8Literal(isolate, "1 + 1");
  Local<Script> script = Script::Compile(context, source).ToLocalChecked();
  Local<Value> result = script->Run(context).ToLocalChecked();
  // ...
}
isolate->Dispose();

05 常见陷阱

忘记 HandleScope → 内存泄漏

Local<T> 永不释放。

跨 thread 直接传 Isolate → 崩溃

必须 Locker。

Persistent 忘记 Reset → 永久 retain

对象永不被 GC。

Snapshot 在不同 V8 版本间不兼容

每个 V8 版本要重新生成。
