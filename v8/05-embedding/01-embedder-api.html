Embedder API：Isolate / Context / Handle

从 C++ 视角看 V8。Embedder API 是 Node / Deno / Workers 通用的接入面 —— 理解它就理解了"为什么 Worker Threads = 多个 Isolate、为什么 Cloudflare Workers 冷启动能 < 5ms"。讲清核心对象层级（Isolate ≈ Agent / Context ≈ Realm / HandleScope / Persistent）、Snapshot 启动加速、Promise / Microtask Hooks，以及 C++ ↔ JS 互调（FunctionTemplate / N-API）。

阅读时长 ~35 分钟 前置 ECMA P7.1（执行模型） 主要来源 v8.dev/docs/embed · Node embedding API · Cloudflare Workers 设计文档
📝 章节状态

骨架版本。对照大纲 v8/outline.md § P5.1。

01 速览

  1. Isolate 是隔离的最强保证：JS 代码绝无可能跨 Isolate 看到对方的对象 —— 这是 Workers / Cloudflare Workers / Lambda@Edge 的安全基础。
  2. Snapshot 是工程奇迹：把 built-ins 构造结果序列化到磁盘，启动时直接 mmap 进堆，毫秒级冷启动。
  3. V8 不拥有 Event Loop：microtask drain 时机由宿主决定（Node 在每个 task 结束时 drain；浏览器在 task + 渲染前 drain）。

02 底层逻辑

核心对象层级

layout 
v8::Isolate            ← 一个独立的 V8 实例（自己的堆、GC、optimizer）
   │
   ├── v8::Context     ← JS 的全局执行环境（≈ Realm）
   │   ├── globalThis
   │   ├── built-ins (Object, Array, Promise, ...)
   │   └── 用户代码
   │
   ├── v8::HandleScope ← Local handles 的生命周期容器（栈分配）
   │
   └── v8::Persistent  ← 跨函数/跨 scope 持有的引用

Isolate（≈ ECMAScript Agent）

正文待写。

  • 一个 thread 在任意时刻只能进入一个 Isolate（用 Isolate::Scope）
  • 多个 thread 想进同一 Isolate 必须用 Locker / Unlocker
  • Workers 模型：每个 Worker 一个 Isolate，相互完全隔离

Snapshot · Isolate 启动加速的核心

正文待写。

  • 没有 Snapshot 时：每次 Isolate 启动要从零执行所有 built-ins 的 JS / Torque 代码 —— 构造 Object.prototype、Array.prototype、Promise.prototype 上几百个方法等等。冷启动 ~50-100ms
  • Startup Snapshot：V8 build 时把上述构造完的堆状态序列化成一段二进制；启动时直接 mmap 进新 Isolate，跳过所有 built-ins 初始化。冷启动降到 ~3-5ms
  • Custom Snapshot / Embedder Snapshot：嵌入器还能把"业务级别的初始化"（如 React framework 的初次模块加载）也烤进 snapshot
  • Cloudflare Workers 的核心 trick：每个 isolate 启动 < 5ms 就靠 Snapshot + Pointer Compression（cage 共享底层布局）
  • Code Cache vs Snapshot：Code Cache 缓存的是用户脚本 bytecode；Snapshot 缓存的是 V8 built-ins 堆状态。两者正交，Chrome 同时用两者
  • 跨版本不兼容：Snapshot 格式与 V8 commit hash 强绑定

Context（≈ ECMAScript Realm）

正文待写。

  • 一个 Isolate 内可以有多个 Context（如 iframe、Node vm 模块、Workers）
  • 每个 Context 有独立的 globalThis 和 built-ins，但共享底层 Map / 函数
  • 跨 Context 调用：Embedder 必须显式 Context::Enter() / Context::Exit()
  • 详细规范请参考 ECMA P7.1 · 执行模型

Handle 系统（GC 安全的指针）

正文待写。

  • 直接持 raw pointer 不安全 —— GC 移动对象后 pointer 失效
  • Local<T>：栈上临时引用，归 HandleScope 管理
  • Persistent<T>：堆上长期引用，必须显式 .Reset()
  • Eternal<T>：永不释放（如 hidden global）
  • TracedReference<T>：Cppgc / Oilpan 集成的 traced 引用（详见 P4.2 · Cppgc 协作）

Promise / Microtask Hooks

正文待写。

  • Embedder 通过 Isolate::SetPromiseHook 注册 Promise 生命周期回调（init / resolve / before / after）—— Node 的 async_hooks 就是包这层
  • Isolate::PerformMicrotaskCheckpoint 由 Embedder 主动调用 —— 因为 V8 不拥有 Event Loop，microtask drain 时机由宿主决定
  • 详见 JS P3 · 异步 与 ECMA P7.3 · Event Loop

C++ ↔ JS 互调

正文待写。

  • FunctionTemplate / ObjectTemplate：定义 C++ 暴露给 JS 的对象 / 函数
  • N-API（Node.js 自家抽象）：在 V8 之上再封一层 ABI 稳定接口（V8 升级不破坏 native module）
  • NAPI vs node-addon-api vs raw V8：选择因素是"是否要绑定特定 Node 版本"

04 典型代码

cpp 
// 嵌入 V8 的最小代码骨架
V8::InitializeExternalStartupData(...);
Isolate::CreateParams create_params;
create_params.array_buffer_allocator = ArrayBuffer::Allocator::NewDefaultAllocator();
Isolate* isolate = Isolate::New(create_params);
{
  Isolate::Scope isolate_scope(isolate);
  HandleScope handle_scope(isolate);
  Local<Context> context = Context::New(isolate);
  Context::Scope context_scope(context);

  Local<String> source = String::NewFromUtf8Literal(isolate, "1 + 1");
  Local<Script> script = Script::Compile(context, source).ToLocalChecked();
  Local<Value> result = script->Run(context).ToLocalChecked();
  // ...
}
isolate->Dispose();

05 常见陷阱

忘记 HandleScope → 内存泄漏

Local<T> 永不释放。

跨 thread 直接传 Isolate → 崩溃

必须 Locker。

Persistent 忘记 Reset → 永久 retain

对象永不被 GC。

Snapshot 在不同 V8 版本间不兼容

每个 V8 版本要重新生成。
