Inline Cache：Mono → Poly → Mega

IC 是连接 Hidden Class（P3.1）与编译器（P2.3）的桥。每个属性访问 / 函数调用站点都有一个 IC，在运行时积累"我见过哪些形状"，再喂给优化编译器做 speculation。状态机 4 档：Uninitialized → Monomorphic → Polymorphic → Megamorphic。一旦 megamorphic V8 就放弃缓存，性能与未优化无异。

阅读时长 ~30 分钟 前置 P3.1 · Hidden Class 主要来源 mrale.ph · v8-perf · V8 source

01 速览

  1. IC 是 V8 的"神经系统"：所有优化决策都依赖 IC 收集的运行时形状信息。
  2. 4 个状态：Uninitialized → Monomorphic（最快）→ Polymorphic（2-4 形状，仍快）→ Megamorphic（≥5 形状，V8 放弃 IC）。
  3. megamorphic 不是末日：仍比纯解释执行快 —— 只是没有了 Hidden Class 带来的 1-2 条指令神话。

02 底层逻辑

从"我们刚刚见过同样的形状"开始

P3.1 讲清了 Hidden Class 怎么让形状相同的对象访问起来跟 C 一样快。但有个隐含前提：V8 怎么知道运行时下一次访问看到的形状跟上次一样？答案是 Inline Cache——每个属性访问 / 函数调用站点都"就地缓存上次见过的形状信息"，下次访问先快速比对，命中就走快路径。

名字"Inline"不是空话——这套缓存就埋在调用站点的机器码里。每个站点对应 Feedback Vector 的一个 slot（详见 P2.2），slot 内容随运行时演化。Sparkplug / Maglev / TurboFan 编译时读这些 slot 决定怎么生成机器码——IC 是连接 Hidden Class（数据结构）和编译器（速度）的桥。

IC 的 4 个状态：Uninitialized → Mono → Poly → Mega

每个 IC 站点都按一个状态机演化，根据"见过几种不同形状"逐级劣化。

Uninitialized · 从未执行过

函数刚被调用、IC 站点还没积累任何信息。Feedback Vector slot 是空的。下次执行时按"通用慢路径"走（Runtime::Get 这种 V8 内部函数），同时记录"这次见到的 Map 是 X"——下次进入就升级到 Premonomorphic / Monomorphic。

Monomorphic · 黄金状态

只见过1 种 hidden class（Map_A）。这是性能最优的状态——V8 可以为这一种形状生成最特化的机器码：

pseudo-asm 
; obj.foo (mono on Map_A，foo 在 in-object slot 1)
mov rax, [rdi]            ; 读 obj 的 Map 指针
cmp rax, [Map_A_addr]     ; 比对是否 Map_A
jne deopt_or_miss_handler ; 不等？走慢路径或 deopt
mov rbx, [rdi + 16]       ; 直接 load 属性值（offset 16 是 slot 1）

这就是 V8 让 JS 接近静态语言性能的真正机器码层证据——3 条指令完成属性访问，其中 2 条是 Map check 安全开销。

Polymorphic · 链式 Map check

见过 2-4 种 hidden class。V8 仍能生成快路径，但要做链式比对——挨个检查每个 Map：

pseudo-asm 
; obj.foo (poly on Map_A, Map_B, Map_C)
mov rax, [rdi]
cmp rax, [Map_A_addr]
je load_for_A           ; 是 A？跳到 A 的 load 代码
cmp rax, [Map_B_addr]
je load_for_B
cmp rax, [Map_C_addr]
je load_for_C
jmp slow_path            ; 都不是？走慢路径
load_for_A: mov rbx, [rdi + 16]; jmp continue
load_for_B: mov rbx, [rdi + 8]; jmp continue
...

典型情况下 IC 命中第一个 Map（最常见的形状），实际只多一两个 cmp 指令。但 cache 利用率下降（多个 Map 的 metadata 都要在 L1 cache 里），整体性能比 mono 慢 1.5-2×。仍比走全局 hash table 查询快得多。

Megamorphic · V8 放弃 IC

见过 5+ 种 hidden class。V8 认为"形状不稳定"，放弃缓存——把这个 IC 站点降级成走全局共享的 hash table 查询。访问开销跳到 ~10ns，跟 dictionary mode 类似。

这是 V8 性能调优的重要分水岭。Mono / Poly 都还有救，megamorphic 几乎没救——因为编译器看到 mega 时就放弃了 speculation。但要注意：megamorphic 不是末日。megamorphic IC 走 stub 查询仍比纯解释执行快——只是没有了 Hidden Class 带来的 1-2 条指令神话。在 hot path 之外的代码里，megamorphic 完全没问题；真正的问题只在每秒调用上百万次的核心路径。

典型 IC 站点

V8 内部有十几种 IC 类型，按操作不同分类。最重要的几种：

  • LoadIC：属性读 obj.foo。最常用、性能影响最大。
  • StoreIC：属性写 obj.foo = x。比 LoadIC 复杂——可能触发 Hidden Class transition（详见 P3.1）。
  • CallIC：函数调用 f() / obj.method()。除了 receiver 形状还要追踪函数本身的指针——同一站点调过哪几个函数。这就是"arr.map(callback) 用同一 callback 比每次新建快"的根因。
  • StoreOwnIC：对象字面量定义属性（{x: 1} 内部）。
  • BinaryOp / CompareIC：算术（a + b）/ 比较（a == b）。Feedback 记录"操作数类型组合"——SMI+SMI / String+String / Mixed。
  • InstanceOfIC：x instanceof Y。追踪 Y 的 prototype 形状。
  • InIC：'foo' in obj。
  • TypeOfIC：typeof x。

每种 IC 都有自己的状态机但都遵循"见 1 种 → mono → 见 N 种 → poly → 见太多 → mega"的演化模式。性能调优的目标是让 hot path 上的关键 IC 站点保持 mono——这就是 V8 性能优化的"主旋律"。

从 Mono 升到 Poly 再到 Mega

javascript 
function getX(obj) { return obj.x; }

getX({x:1});       // IC: Mono (Map_A)
getX({x:2});       // IC: Mono (Map_A) ← 同形状
getX({x:1, y:2});  // IC: Poly (Map_A, Map_B) ← 加了 y，新 Map
getX({x:1, z:3});  // IC: Poly (Map_A, Map_B, Map_C)
getX({x:1, w:4});  // IC: Poly (4 maps)
getX({x:1, q:5});  // IC: MEGA ← 5+ maps，放弃缓存

各 tier 的 IC 形态：从 stub 调用到完全内联

IC 在 V8 4 层 JIT 里的形态各不相同——越上层优化越激进、IC 检查越被"消化"进周围代码。

Ignition：动态查 Feedback Vector

解释器对每条 IC 字节码都要运行时查 Feedback Vector——读出当前状态（mono/poly/mega）+ 缓存的 Map 列表，再调对应 IC stub。这套机制开销比 mono fast path 大几纳秒，但灵活：状态变化随时反映。

Sparkplug：内联 IC stub 调用

Sparkplug 还是要走 IC stub（前文 P2.2 提过——它"不重新发明语义"），但不再做 dispatch + operand decoding，而是直接 emit 一条机器码 call 到对应 stub。所以 Sparkplug 加速主要来自消除解释器开销而非优化 IC 本身——这就是 Sparkplug 只 +5-15% 的根因。

Maglev：根据 Feedback 内联 fast path

Maglev 是第一层真正"读 Feedback Vector 做 speculation"的层。编译时它看 IC 当前状态：

  • Mono on Map_A：直接内联 Map_A 的 fast path（Map check + offset load 两条机器指令），跳过 stub 调用
  • Poly：生成switch 结构链式 Map check，每个 case 是对应 Map 的 fast path
  • Mega：仍走 stub，因为 Maglev 不值得为 mega 做特化（大概率本来就不在热路径上）

如果 Maglev 编译版本运行时 Map check 失败（即出现编译时没见过的 Map）→ 触发 deopt 跳回 Ignition，让 Feedback 继续积累。下次 Maglev 编译可能升级到 poly 版本。

TurboFan：完全内联到 sea of nodes，跨函数 inline

TurboFan 走得更远——它把 IC 信息完全融进 sea of nodes IR，让其他优化遍能利用这些类型信息。例：mono on SMI 的 a + b 编译时直接变成"SMI add"节点，escape analysis / range analysis 都能用这个类型做进一步推导。

更厉害的是 跨函数 inline——TurboFan 看到 arr.map(callback)、callback 站点 mono on 某函数 X → 直接把 X 的 sea of nodes 图嵌进当前函数的图，连同 X 内部的 IC 也一起利用。这是为什么 TurboFan 能跑出近 native 性能：所有 speculation 链条全部展开成单一 sea of nodes 图。

代价是：任何一处 speculation 错都会让整个内联函数 deopt。这就是为什么 TurboFan 需要精细的 deopt info——能从内联深处跳回原始 bytecode 的对应位置（详见 P2.3 反优化）。

Feedback Vector 是闭环的中心

可以把 V8 的 speculation 体系画成一个闭环：

feedback-loop 
┌──────────────────────────────────────────────┐
│  Ignition / Sparkplug 执行 → 更新 Feedback Vector  │
│         ↓                                    │
│  (call counter 过线 + IC mature)             │
│         ↓                                    │
│  Maglev / TurboFan 编译 → 读 Feedback Vector │
│         ↓                                    │
│  Speculation 内联 fast path                   │
│         ↓                                    │
│  运行时 Map check 失败 → Deopt              │
│         ↓                                    │
│  跳回 Ignition → 继续积累 Feedback ─────────┘

这条闭环回答了几乎所有 V8 性能现象：为什么循环 5000 次还没被优化（IC 不够 mature）、为什么 deopt loop（Feedback 反复变 + 反复编译）、为什么"同样的代码不同输入跑得慢"（IC 站点被喂了不稳定形状）。

理解这个闭环 + P3.1 的 Hidden Class，就掌握了 V8 性能的所有底层逻辑。后续 P4 内存 / P5 嵌入都是相对独立的话题，性能层面已经讲完。

megamorphic 怎么救：三种救法 + 一种放弃

诊断出某 IC 站点 megamorphic 后（用 --trace-ic 看，详见后文场景），如果它确实在 hot path 上影响可观，有几种修法。

救法 1 · 手动分离：写多个特化函数

常见反模式是"通用处理函数被喂多形状"：

javascript 
// ❌ 反模式：getX 接受 Circle / Square / Triangle 三种形状 → mega
function getX(shape) {
  return shape.x;
}
shapes.forEach(s => getX(s));   // shapes 里有三种形状
// IC 在 getX 内部一个站点，看到三种 Map → poly 升级到 mega

// ✅ 修正：分离成多个特化函数
function getXFromCircle(circle)   { return circle.x; }
function getXFromSquare(square)   { return square.x; }
function getXFromTriangle(triangle) { return triangle.x; }
// 调用前先按类型分组
circles.forEach(c => getXFromCircle(c));     // mono
squares.forEach(s => getXFromSquare(s));     // mono
triangles.forEach(t => getXFromTriangle(t)); // mono

代价：代码重复。收益：每个 IC 站点都 mono，hot path 上 3-5× 加速。视场景判断是否值得。

救法 2 · 统一 shape：base class 包含所有属性

如果几个对象逻辑上是同一族（如 UI 元素），可以让它们用同一个 Map——base class 里初始化所有可能的属性，子类只改值不加结构：

javascript 
// ❌ 反模式：每个子类自己加属性 → 不同 Map
class Circle   { constructor(r) { this.x = 0; this.r = r; } }
class Square   { constructor(s) { this.x = 0; this.s = s; } }

// ✅ 修正：base class 初始化所有属性
class Shape {
  constructor() {
    this.x = 0; this.y = 0;
    this.r = 0;       // radius (for circles)
    this.s = 0;       // side (for squares)
    this.kind = '';
  }
}
class Circle extends Shape { constructor(r) { super(); this.r = r; this.kind = 'circle'; } }
class Square extends Shape { constructor(s) { super(); this.s = s; this.kind = 'square'; } }
// 所有实例形状相同 → 同一个 Map → mono

代价：每个对象稍多内存（多了未用的属性）。收益：所有 IC 站点 mono。适合"少数几种相关类型"的场景，不适合"无关对象族"。

救法 3 · 用 Map 数据结构而非动态属性

如果对象本质是"动态 key-value 集合"（如 i18n 翻译表、配置对象、运行时数据），不要用普通 object——直接用 ES6 Map。Map 内部是优化好的 hash table，根本不走 IC + Hidden Class 系统，性能稳定可预测：

javascript 
// ❌ 反模式：把 object 当 dict 用 → dictionary mode 降级
const cache = {};
for (const key of dynamicKeys) cache[key] = compute(key);

// ✅ 修正：用 Map
const cache = new Map();
for (const key of dynamicKeys) cache.set(key, compute(key));

放弃：通用 deserializer / 反射 / 库代码

有些场景本质上就是 megamorphic，没法救：

  • JSON.parse 后的反序列化处理——不同 schema 的 JSON 走同一段代码
  • ORM / 数据库映射库——映射任意 entity 形状到 SQL
  • 测试 / 序列化 / 比较库（lodash、deep-equal 等）——按定义就要支持任意形状

这些场景接受 megamorphic 现实——库作者不要为这种通用性焦虑，用户应用层的 mono 才是性能主战场。

03 应用场景

场景 A · 何时该担心 megamorphic IC

判据：调用频次 × 单次开销 > 1ms 累计影响才值得修。例：每秒调用 100 万次的 hot path 上 mega IC，每次比 mono 多花 ~10ns → 累计 10ms/秒，明显影响。但每秒调用 100 次的代码 mega 完全没事。不要为冷代码做 IC 优化——可读性 / 维护性优先于这种纳秒级提升。

场景 B · 用 --trace-ic 找 megamorphic 站点

node --trace-ic app.js 2>&1 | grep "MEGAMORPHIC" | sort | uniq -c | sort -rn | head——这条命令列出 mega 转换最多的站点。输出包含文件:行号:列号，能直接定位到代码。如果某个站点在 hot path 上反复 mega，就是优化目标。注意 --trace-ic 输出极大（每个 IC 状态变化都打），grep 必须配合 head/tail 限量。

场景 C · JSON 解析后的对象 IC 行为

同源 JSON 走同 transition path → mono：API 返回相同 schema 的对象，JSON.parse 出来的对象都是同一形状，访问代码 IC 自然 mono。混合 schema → poly/mega：例如某接口返回 {type: 'user', name: ...} 和 {type: 'post', title: ...} 两种形状，统一处理代码就 poly/mega。修法：按 type 分支后用 P3.2 的"手动分离"或"base class 统一 shape"。

场景 D · 闭包工厂的 IC 站点独立

arr.map(item => item.x) 在循环里被调用上千次——每次创建新闭包，每个闭包的 item.x IC 站点都独立从 Uninitialized 起步，永远积累不出 mature feedback。这就是为什么"循环内 inline 闭包"总比"提到外面的命名函数"慢——根本不是创建闭包的开销，是 IC 站点不能复用 Feedback。修正：把 callback 提到循环外定义共享，或直接 arr.map(getX)（如果签名匹配）。

05 常见陷阱

不要为了 mono 把所有对象设计成同 shape

大多数代码热度不够，过度优化 = 复杂度。

Object.create(null) 的对象走专门路径

不走 prototype 链 IC，性能特征不同（但通常更快）。

Symbol 属性走专门的 IC 路径

不和字符串属性共用 slot。

Proxy 对象完全跳过 IC

任何 Proxy 操作都是慢路径（Proxy 详细机制见 ECMA P6.1）。
