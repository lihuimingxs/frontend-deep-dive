Hidden Class（Map / Transition Tree）

V8 实现 ECMAScript Object Internal Methods（[[Get]] / [[Set]] / [[OwnPropertyKeys]] 等）的关键数据结构。讲清"为什么 JS 对象访问能跟 C 结构体一样快"——V8 把"动态语言"约束到运行时常见模式，大多数对象的属性集合在创建后不变，让相同形状的对象共享一个 Map（hidden class），让属性访问变成"Map check + offset load"两条指令。

阅读时长 ~35 分钟 前置 ECMA P6（Internal Methods） 主要来源 v8.dev/docs/hidden-classes · slack-tracking 博客
📝 章节状态

骨架版本。对照大纲 v8/outline.md § P3.1。

01 速览

  1. Hidden Class 是 V8 把"动态语言"伪造成"静态"的核心。所有 JS 对象的第一字段是一个 Map 指针，相同形状对象共享同一个 Map。
  2. 属性存储 3 档：in-object（前 3 个 slot）/ fast properties（backing store）/ dictionary mode（hash table，慢路径）。
  3. 属性添加顺序很重要：相同顺序 → 相同 Map；不同顺序 → 走不同 transition path → IC 站点会 polymorphic。

02 底层逻辑

三件套数据结构

正文待写。

  • Map（即 hidden class）：每个 object 第一字段，标识"形状"
  • DescriptorArray：Map 共享，记录每个属性的名字 + storage location（in-object slot 或 backing store index）+ attributes
  • TransitionArray：Map 之间的边，记录"如果加属性 X 则转到 Map Y"

Transition Tree 示意

map-graph 
{} (Map_0)
 │ +x
 ▼
{x} (Map_1)
 │ +y
 ▼
{x, y} (Map_2)
 │ +z
 ▼
{x, y, z} (Map_3)

所有 {x, y, z} 形状的 object 共享 Map_3
内存占用极低且 Map 比较 = 指针比较

属性存储 3 档

正文待写。

  • In-object properties（前 3 个）：直接存在 object 字段里，访问 = 一次 load
  • Fast properties（4+）：存在 backing store 数组，descriptor 给出 index
  • Dictionary mode：当属性数量超过阈值 / 频繁 delete / 名字奇怪时降级成 hash table

从 ECMA Internal Method 到 V8 实现

正文待写。规范层与实现层的对应（详细规范定义见 ECMA P6.1）：

  • [[Get]] (P, Receiver) 规范步骤一堆 → V8 编译成"Map check + offset load"两条指令
  • [[Set]] (P, V, Receiver) → "Map check + transition lookup + store"
  • [[OwnPropertyKeys]] () → 遍历 DescriptorArray

'const' 标注与编译时内联

正文待写。

  • 如果一个属性从未被改（V8 跟踪），DescriptorArray 把它标 const
  • TurboFan 看到 const 直接把值内联进机器码（如 obj.foo 编译成 mov eax, 42）
  • 后来如果属性变了 → Map deprecated → 触发 deopt

属性添加顺序的代价

javascript 
// 慢：两个 object 走不同 transition path
const a = {}; a.x = 1; a.y = 2;
const b = {}; b.y = 2; b.x = 1;
// a 的 Map 和 b 的 Map 不同 —— IC 站点会 polymorphic

// 快：相同顺序 → 相同 Map
const a = {x: 1, y: 2};
const b = {x: 1, y: 2};

数组也有 Map（"Elements Kind"）

正文待写。

  • PACKED_SMI → PACKED_DOUBLE → PACKED_ELEMENTS（一旦混入 string / object 就回不去）
  • HOLEY_* vs PACKED_*：稀疏数组（如 [,1,,3]）走 holey 慢路径

Slack Tracking

正文待写。V8 处理"不知道最终多少属性"的工程式答案：

  • 构造对象时 V8 不知道用户最终会加多少属性 —— 给少了后期要走 backing store 慢路径，给多了浪费内存
  • 初始 Map 上带一个 construction counter（默认 7），新对象多预留 8 个 in-object slot；每构造一次计数器递减
  • 计数归零后 V8 认为构造模式稳定，填 filler 对象占住没用的 slot，下次 GC 时这些 filler 被回收，Map 收缩到实际用量
  • 典型例子：Peak 对象初始 52 字节（10 slot），slack tracking 完成后缩到 20 字节（2 slot）

04 典型代码

javascript 
// 用 %HaveSameMap 验证两 object 是否同 hidden class（仅 d8 / --allow-natives-syntax）
const a = {x: 1, y: 2};
const b = {x: 3, y: 4};
const c = {y: 4, x: 3};

console.log(%HaveSameMap(a, b));  // true
console.log(%HaveSameMap(a, c));  // false（顺序不同）

// 用 %DebugPrint 看 V8 内部表示
%DebugPrint(a);
// 输出包含 Map 地址、in-object slot 内容、descriptor array

05 常见陷阱

看到"V8 优化对象"博客时注意年代

2017 之前可能讲 Crankshaft 的 hidden class 优化（与今天的 Map 系统差异不大但实现细节不同）。

class 不是魔法

class Foo { ... } 和 function Foo() { ... } + prototype 在 V8 内部是同一套 Map 机制。

Object.assign({}, src) 走 normalized 路径

性能不如直接 {...src}（V8 11+ 已优化但仍有差异）。

delete 是 V8 性能反模式

transition tree 是单向不可逆的；一旦走到一个 Map，删属性也不会回到旧 Map，反而触发 dictionary mode。
