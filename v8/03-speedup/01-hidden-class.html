Hidden Class（Map / Transition Tree）

V8 实现 ECMAScript Object Internal Methods（[[Get]] / [[Set]] / [[OwnPropertyKeys]] 等）的关键数据结构。讲清"为什么 JS 对象访问能跟 C 结构体一样快"——V8 把"动态语言"约束到运行时常见模式，大多数对象的属性集合在创建后不变，让相同形状的对象共享一个 Map（hidden class），让属性访问变成"Map check + offset load"两条指令。

阅读时长 ~35 分钟 前置 ECMA P6（Internal Methods） 主要来源 v8.dev/docs/hidden-classes · slack-tracking 博客

01 速览

  1. Hidden Class 是 V8 把"动态语言"伪造成"静态"的核心。所有 JS 对象的第一字段是一个 Map 指针，相同形状对象共享同一个 Map。
  2. 属性存储 3 档：in-object（前 3 个 slot）/ fast properties（backing store）/ dictionary mode（hash table，慢路径）。
  3. 属性添加顺序很重要：相同顺序 → 相同 Map；不同顺序 → 走不同 transition path → IC 站点会 polymorphic。

02 底层逻辑

从"哈希表很慢"说起

ECMAScript 规范把对象定义成"属性名 → 值的映射"——naive 实现就是 hash table。但 hash table 访问每次都要：算 hash、找 bucket、字符串比较、可能还要走链表。每次属性访问 ~50ns，一段 obj.x.y.z 要 150ns——对纯计算密集场景这个开销大得吓人。

而静态语言（C++、Java）的对象访问只需要"加一个偏移量"——一条 mov rax, [rdi+16] 指令几纳秒。差距是10 倍以上。

V8 的核心问题是：能不能让 JS 对象访问大多数情况下跟静态语言一样快？观察发现一个关键事实——真实代码里，对象的属性集合一旦构造完毕几乎不变。{x: 1, y: 2} 这样的 point 对象，构造时确定形状之后，运行时只是改 x 和 y 的值而不是结构。如果能识别出"这一群对象形状相同"，就能让它们共享同一份"形状描述"——属性访问退化为偏移量加载。

这就是 Hidden Class（V8 内部叫 Map）的本质：把"动态语言的形状信息"外置到一个共享对象里，让形状相同的对象访问起来跟静态语言一样快。这个 trick Lars Bak 在 Self / Strongtalk / HotSpot 已经做过三次（详见 P1.1），V8 是它在 JS 引擎里第四次出场。

三件套数据结构：Map + DescriptorArray + TransitionArray

一个 V8 object 在堆上的物理布局：第一个 4-byte slot（启用 Pointer Compression 时）是 Map 指针。Map 是 hidden class——所有形状相同的对象指向同一个 Map。

Map 又指向两个数据结构：

  • DescriptorArray：属性目录。每个属性占一条 entry，记录属性名（字符串或 Symbol）+ 存储位置（in-object slot 索引 / backing store 索引）+ attributes（writable / enumerable / configurable / const-ness）。所有形状相同的对象共享同一份 DescriptorArray——内存极省。
  • TransitionArray：形状演化的"出边"。记录"如果给当前形状的对象加属性 X，转到哪个 Map Y"。例：{x:1} 形状的 Map_1 的 TransitionArray 里有条 +y → Map_2，意思是"在 {x} 上加 y 得到 {x,y} 形状"。

可以这样理解：Map = 节点、TransitionArray = 出边——所有 Map 形成一个有向图（实际上是树，每个 Map 只有一个父亲），叫做 Transition Tree。从空对象 {} 的 root Map 出发，每加一个属性走一条边到达一个新 Map。

💡 一个可视化技巧

想象 Transition Tree 像 git commit 树——每条边是"在父形状基础上加一个属性"，每个 Map 是"形状的快照"。两个对象走相同序列的边到达同一个 Map——形状相同就共享。这就是 V8 内存效率高的根源：哪怕 100 万个 {x, y, z} 形状的对象，全都共享一个 Map_3 + 一个 DescriptorArray。

Transition Tree 示意

map-graph 
{} (Map_0)
 │ +x
 ▼
{x} (Map_1)
 │ +y
 ▼
{x, y} (Map_2)

属性存储 3 档：In-object → Fast → Dictionary

知道了 Map / DescriptorArray，下一个问题是属性的值实际存在哪？V8 用三档存储策略，按对象使用模式自动选择。

第一档 · In-object properties（前 3 个属性）

最快的存储位置：直接存在 object 自身的内存槽位里。V8 默认给每个 object 预留 3 个 in-object slots（实际数量受 Slack Tracking 调整，详见后文），新 object 的前 3 个属性按顺序填进去。访问时 DescriptorArray 给出 slot 索引（0/1/2），机器码就是一条 load 指令：mov rax, [rdi + 16]（rdi 是对象指针，16 是 slot 1 偏移）。

这一档跟 C++ struct field 访问开销几乎相同——这就是 V8 让 JS 跑出近静态语言性能的关键。绝大多数 JS 对象（Point、Vector、Color、ConfigObject）都属于"3-5 个属性"范围，能完全或大部分进 in-object 档。

第二档 · Fast properties（4+ 属性，仍是固定形状）

属性多于 in-object 容量后，V8 给对象额外分配一个数组叫 properties backing store，多余属性存这里。访问需要两次 load：先从 object 加载 properties 指针，再从 backing store 加偏移加载值。比 in-object 慢一倍，但仍是 O(1)——比 hash table 快得多。

DescriptorArray 仍记录每个属性"在 in-object 还是 backing store"以及具体索引——所以 V8 在编译时就能确定走单 load 还是双 load。这是 fast properties 与 dictionary mode 最大的区别。

第三档 · Dictionary mode（hash table 慢路径）

以下情况会让 V8 把对象降级到 dictionary mode：

  • 属性数量超过阈值（典型 1000+ 属性，少见但发生过）
  • 频繁 delete ——transition tree 是单向的，删属性后 V8 走慢路径
  • 属性名奇怪——含空格、特殊字符、纯数字超大整数等
  • 原型链 dictionary 化—— Object.prototype 被改也会让访问走 dictionary

dictionary mode 下 V8 把对象的属性表换成 SwissNameDictionary（V8 11+，之前是 NameDictionary）——一个开放寻址 hash table。访问开销跳到 ~10ns，比 fast 慢 5-10×。这就是为什么 delete obj.foo 是 V8 性能反模式——不只是单次删除慢，是把整个对象拖进 dictionary mode，之后所有属性访问都变慢。修正方式：obj.foo = undefined 替代 delete。

从 ECMA Internal Method 到 V8 实现

ECMAScript 规范用 Internal Methods（如 [[Get]] / [[Set]] / [[OwnPropertyKeys]]）抽象描述对象操作——每个方法都是一段抽象算法（详见 ECMA P6.1）。规范不规定怎么实现，只规定语义。V8 用 Hidden Class 把这些抽象方法变成几条机器指令：

  • [[Get]] (P, Receiver)：规范有 10+ 步骤（找属性、检查 getter、走 prototype 链等）。V8 在 monomorphic IC 下编译成两条机器指令：Map check + offset load。所有规范步骤都被 speculation 推到了编译时——只要 Map 没变，规范步骤的结果就是预先算好的常量。
  • [[Set]] (P, V, Receiver)：编译成 Map check + transition lookup（如果是新属性）+ store。新增属性会触发 transition——V8 沿着 TransitionArray 走或创建新边。
  • [[OwnPropertyKeys]] ()：直接遍历 DescriptorArray——属性顺序就是 DescriptorArray 顺序，这就是 ES2015 之后属性顺序稳定的实现根因。
  • [[GetPrototypeOf]] ()：从 Map 上读 prototype 字段——一次 load。

这种"规范定义抽象语义、引擎用具体数据结构加速"的对应关系，是 V8 主题与 ECMA 主题的核心边界。规范层学语义、引擎层学实现——对照着读两边能让"为什么这样写性能好"的根因彻底打通。

'const' 标注与编译时内联

DescriptorArray 的每个属性条目都有一个 const-ness 标记——记录"自从对象构造以来这个属性是否被改过"。V8 默认假设新属性是 const，写过一次后变 mutable。

这个标记看似不起眼，但对优化层意义巨大。TurboFan 看到某个属性是 const，直接把值内联进机器码：

javascript 
const config = { TIMEOUT: 5000, DEBUG: false };

function check(req) {
  if (Date.now() - req.start > config.TIMEOUT) {
    throw new Error('timeout');
  }
}
// hot loop 里调用 check(...)
// TurboFan 编译时：config.TIMEOUT 标记 const，直接内联成机器码立即数 5000
// 优化后机器码大致：
//   mov rax, [Date_now]    ; ...
//   sub rax, [rdi + 16]    ; req.start
//   cmp rax, 5000          ; 直接立即数，不再读 config 对象
//   ja throw_path

后来如果你写 config.TIMEOUT = 10000，会发生什么？V8 标记 Map 为 deprecated，触发所有依赖这个 const 标记的优化代码 deopt。代码本次执行完成后 lazy deopt 跳回 Ignition，重新积累 feedback——之后 Maglev/TurboFan 编译版本不再内联这个值。

这给一条 actionable 优化原则：真正不变的配置就用 const + 不再赋值，让 V8 直接把值烤进机器码。这比"语义上是 readonly 但运行时仍走属性读"快几纳秒——hot path 上累计可观。

属性添加顺序的代价

javascript 
// 慢：两个 object 走不同 transition path
const a = {}; a.x = 1; a.y = 2;
const b = {}; b.y = 2; b.x = 1;
// a 的 Map 和 b 的 Map 不同 —— IC 站点会 polymorphic

// 快：相同顺序 → 相同 Map
const a = {x: 1, y: 2};
const b = {x: 1, y: 2};

数组也有 Map：Elements Kind

数组在 V8 里也是普通对象——所以也有 Map。但 V8 给数组做了特别优化：除了通用 Map 还有 Elements Kind 标记，记录"数组元素的类型集合"。这是 V8 让数值数组跑出近 C 性能的关键。

Elements Kind 是一条单向通用化链：

elements-kind-chain 
PACKED_SMI_ELEMENTS         ← 全是 SMI（整数）
       │ 加入 double（如 NaN / 1.5 / 2**31）
       ▼
PACKED_DOUBLE_ELEMENTS      ← SMI + double
       │ 加入 string / object / undefined
       ▼
PACKED_ELEMENTS             ← 任意 JS 值
       │ 出现空洞（如 arr[5] = x 跳过 0-4）
       ▼
HOLEY_ELEMENTS              ← 稀疏数组慢路径
       │
       ▼
DICTIONARY_ELEMENTS         ← 极稀疏 hash table

关键点：转换是单向的——一旦 PACKED_SMI 数组里塞进了 double，整个数组从此是 PACKED_DOUBLE，就算后来把 double 改回 SMI 也不会回退。V8 不会"走回头路"——这就是 PACKED_SMI 数组性能可贵的原因，一旦丢了就回不去。

PACKED 系列对应"密集数组"——所有 index 0 到 length-1 都有值；HOLEY 系列对应"稀疏数组"——有空洞。HOLEY 数组每次访问都要检查"这个 slot 是真值还是空洞"，比 PACKED 慢 30-50%。常见制造 holey 的反模式：const arr = new Array(1000)（直接创建 1000 长度的稀疏数组）、arr[1000] = x（在远超当前 length 的位置赋值跳过中间）。优化方式：用 const arr = []; for (...) arr.push(...) 让 V8 看到每个位置都被赋值，保持 PACKED。

还有一条避坑线：不要在 SMI 数组里塞 NaN——NaN 是 double，会触发 PACKED_SMI → PACKED_DOUBLE 转换。如果你的数组本来是整数下标 / 数据，意外赋个 0/0 之类就把整个数组类型变了。这是为什么很多数学库用 -1 而非 NaN 作"无效值"标记——能保 PACKED_SMI。

Slack Tracking：V8 处理"不知道最终多少属性"的工程式答案

上面讲 in-object slots 时说"前 3 个属性进 in-object"——这个 3 不是写死的常数。V8 实际有个动态机制叫 Slack Tracking，专门处理"构造时不知道最终多少属性"这个 JS 特有难题。

想想 class Point { constructor(x, y) { this.x = x; this.y = y; } }——构造函数明确是 2 个属性。但 V8 不能直接信任构造函数体——可能后面还有 this.z = ...，可能子类还要加属性，可能用户在外部 p.color = ...。给少了浪费 in-object 优势，给多了浪费内存。怎么办？

V8 的方案是初期乐观给多、运行时观察实际、最后回收多余：

  1. 初始 Map 上带一个 construction counter（默认 7）；新对象额外多预留 8 个 in-object slot（实际容量比构造函数明显使用的属性数大不少）
  2. 每次用这个构造函数创建对象，counter 递减一
  3. counter 归零（即创建过 7 个对象后），V8 认为构造模式已稳定——观察到的 N 个属性就是真实需求
  4. V8 用 filler 对象把没用的 slots 占住（防 GC 误读）；下次 GC 时这些 filler 被识别并回收，Map 收缩到实际用量

v8.dev 给的真实例子：Peak 对象（实际 2 个属性）初始 52 字节，slack tracking 完成后缩到 20 字节——内存 -60%。在大量同类对象的程序里（如 React 内部的 Fiber 节点、graph database 的 entity），这种自动收缩是巨大胜利。

💡 Slack Tracking 给了一条 actionable 建议

所有属性都在 constructor 里赋值——这样 slack tracking 收敛后 in-object slots 数量就是真实属性数，所有访问都是单 load。如果有的属性"用到了再加"，slack tracking 收敛后这些属性会被推到 backing store（fast properties），多一次 load。这是为什么 V8 性能博客一直推"初始化所有字段"——不是规范要求，是引擎层最优实践。

03 应用场景

场景 A · 工厂模式：让所有实例共享同一 Map

如果你创建 100 万个 {x, y} 对象，希望它们全部共享 Map_2。做法很简单——用同一构造路径：要么都用 {x: a, y: b} 字面量（V8 会识别成同一字面量站点）、要么都用 class Point { constructor(x, y) { this.x = x; this.y = y; } }。不要有的对象 {x, y} 有的对象 {y, x}——属性顺序不同走不同 transition path，Map 不共享，IC 站点会 polymorphic。

场景 B · 避免 delete 让对象 dictionary 化

delete obj.foo 是 V8 性能反模式——transition tree 是单向的，删属性不会回到旧 Map，反而触发 dictionary mode 降级。修正方式：obj.foo = undefined 或 obj.foo = null——保持 Map 不变，只是 slot 值变成空标记。这种区别在 hot path 上累积影响 5-10×（dictionary access vs in-object access）。

场景 C · 数组优化：保 PACKED_SMI / 避免空洞

数值密集场景（图像处理、物理计算、ML）保 PACKED_SMI：用 for (let i = 0; i < n; i++) arr.push(value)（让 V8 看到每个 slot 被赋值）、不用 new Array(n)（创建稀疏数组）、避免无效值用 NaN（用 -1 或 sentinel int）。极致性能需求用 Int32Array / Float64Array 等 TypedArray——根本绕开 Elements Kind 转换问题。

场景 D · 真不变的配置用 const + 不再赋值

利用 V8 的 const 标注：const config = { TIMEOUT: 5000, ... } + 永不修改 → TurboFan 把值烤进机器码。常见反模式：let config = {...defaults, ...options}（运行时合并 → 非 const）、或事后 config.foo = newVal 改属性（触发 deopt）。修正：把动态参数和静态配置分开，静态部分独立成 frozen const。

05 常见陷阱

看到"V8 优化对象"博客时注意年代

2017 之前可能讲 Crankshaft 的 hidden class 优化（与今天的 Map 系统差异不大但实现细节不同）。

class 不是魔法

class Foo { ... } 和 function Foo() { ... } + prototype 在 V8 内部是同一套 Map 机制。

Object.assign({}, src) 走 normalized 路径

性能不如直接 {...src}（V8 11+ 已优化但仍有差异）。

delete 是 V8 性能反模式

transition tree 是单向不可逆的；一旦走到一个 Map，删属性也不会回到旧 Map，反而触发 dictionary mode。
