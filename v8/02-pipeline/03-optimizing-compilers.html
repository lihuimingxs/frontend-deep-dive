优化编译器三件套：Sparkplug / Maglev / TurboFan

V8 现在有 3 层 JIT，每层都有不同的 IR、不同的优化目标、不同的编译速度 / 性能权衡。这一章讲清三者的边界：Sparkplug 零 IR 1:1 翻译、Maglev SSA + CFG 中层优化、TurboFan Sea of Nodes（2024 后端换成 Turboshaft）。再讲反优化（eager / lazy）机制 —— JIT 编译器假设运行时类型固定，错了就退回。

阅读时长 ~40 分钟 前置 P2.1 + P2.2 主要来源 v8.dev/blog/sparkplug · maglev · turboshaft 系列博客
📝 章节状态

骨架版本。对照大纲 v8/outline.md § P2.3。本章是 P2 最长一章。

01 速览

  1. 3 层不是"备份"是"分工"：Sparkplug 给所有代码用、Maglev 给中频代码用、TurboFan 给热点代码用。
  2. Sea of Nodes 强大但难调试：节点之间无显式顺序，scheduling 在最后做。Turboshaft 改回 CFG-based 后端就是为了"更易理解、更快编译"。
  3. deopt 不是 bug 是设计：JIT 编译器假设运行时类型固定，错了就退回 Ignition。

02 底层逻辑

Sparkplug（2021 引入）

正文待写。关键点：

  • 零 IR 设计：核心结构是 for (each bytecode) switch (opcode) emit_machine_code()，没有 IR、没有 SSA、没有寄存器分配
  • 栈帧与 Ignition 兼容：profiler / debugger / OSR 不需要任何特殊处理就能穿越
  • 不做优化但也不慢：因为消除了 bytecode dispatch + register decoding 开销，5-15% 性能提升
  • Why 没有更早做：Crankshaft 时代没有 bytecode 中间层；只有等 Ignition 稳定后才有"忠实复刻"的对象

Maglev（2023，Chrome M117 全平台）

正文待写。关键点：

  • SSA + CFG（不是 sea of nodes）：所有变量唯一定义、控制流图明确
  • 单一 IR、最少 pass：从 bytecode 一步构建 SSA，做轻量优化（escape analysis 简化版、constant folding、dead code）
  • 编译速度比 TurboFan 快 10x，性能 80%；填补 Sparkplug 与 TurboFan 之间空档
  • Why SSA 而非 sea of nodes：sea of nodes 优化能力强但调度复杂；Maglev 故意选简单 IR 换更快编译

TurboFan（2014 引入，2024 后端换 Turboshaft）

正文待写。关键点：

  • Sea of Nodes IR：节点表示值 + 控制流，依赖边显式 —— Cliff Click 1995 论文那个
  • 优化遍：inlining → escape analysis → loop peeling → load elimination → instruction selection → register allocation → code emission
  • type feedback 驱动 speculation：从 Feedback Vector 读"这里 99% 是 SMI" → 内联 SMI fast path → 错了 deopt
  • Turboshaft 后端（2024+）：替换 sea of nodes 的后端阶段（指令选择 / 调度 / 寄存器分配），编译速度 2x，前端仍是 sea of nodes

反优化（Deoptimization）机制

正文待写。

  • Eager deopt：检查失败立即跳回 Ignition；用 deopt entry 表记录 bytecode offset
  • Lazy deopt：标记 code 失效，下次进入时跳回；多用于"假设别处的 code 不会改但实际改了"
  • OSR (On-Stack Replacement)：把当前 stack frame 从优化版本换成 Ignition 版本，逐 register 拷贝；反向也成立（Ignition → optimized）

04 典型代码

javascript 
// 强制让 TurboFan 编译某函数（仅 d8 / --allow-natives-syntax）
function add(a, b) { return a + b; }

%PrepareFunctionForOptimization(add);
add(1, 2);
add(3, 4);
%OptimizeFunctionOnNextCall(add);
add(5, 6);  // 这次会用 TurboFan 编译版

// 配合 --trace-opt --trace-deopt 看效果

05 常见陷阱

Maglev / TurboFan 都不能保证某段代码"被编译"

V8 自己根据 budget 决定。强制：%PrepareFunctionForOptimization + %OptimizeFunctionOnNextCall（仅 d8 / --allow-natives-syntax）。

"deopt 多就一定慢"——错

少量 lazy deopt 是正常的"类型适应"；只有 deopt loop 才是问题。

不要混淆"编译"和"执行"

TurboFan 编译花 50ms，但执行可能是几纳秒一次。性能调优要看的是"执行总时间"而非"编译时间"。
