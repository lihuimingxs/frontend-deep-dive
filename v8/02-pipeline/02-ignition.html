Ignition 解释器与字节码

Ignition 是 V8 的"嘴巴"——所有代码先变 bytecode 再说。这一章讲它的设计哲学与字节码格式：register-based VM（vs JSC LLInt 的 stack-based）、bytecode handler 用 CodeStubAssembler 写、Feedback Vector 是运行时画像簿。再补充三个常被忽略的话题：Code Cache & Script Streaming（启动期优化两条腿）、Generator / Async function 状态机化（含 V8 7.0 zero-cost desugar）、以及 Builtins 不走这条路——内置函数用 Torque DSL 写、预编译嵌进 Snapshot。

阅读时长 ~35 分钟 前置 P2.1 + ECMA P4（数据类型） 主要来源 v8.dev/docs/ignition · v8.dev/docs/torque · slack-tracking 博客
📝 章节状态

骨架版本。对照大纲 v8/outline.md § P2.2。

01 速览

  1. Ignition 是 register-based 解释器（不是 stack-based），bytecode 操作显式引用寄存器。
  2. bytecode 是 V8 内部的"通用语言" —— Sparkplug / Maglev / TurboFan 都从这里起步。
  3. Builtins 不走这条路 —— Array.prototype.map / Object.assign 等用 Torque 写、预编译嵌进 Snapshot。

02 底层逻辑

Register-based VM

正文待写。每个函数有 N 个虚拟寄存器（accumulator + locals + parameters）。bytecode 操作显式引用寄存器：Add r0, r1, [feedback_slot_2]。优势：减少 dispatch 次数；劣势：bytecode 体积稍大。

bytecode handler 用 CodeStubAssembler (CSA) 写

正文待写。CSA 是 V8 内部的 portable assembler DSL（C++）；每个 opcode 对应一段 CSA 代码 → 编译时生成所有平台的机器码 handler → 运行时 dispatch 到对应 handler。

典型 bytecode 示例

bytecode 
function sum(a, b) { return a + b; }
sum(1, 2);

// d8 --print-bytecode 输出（简化）：
LdaSmi.Wide [1]            // Load SMI 1 to accumulator
Star r0                    // Store to register r0
LdaSmi.Wide [2]
Star r1
LdaUndefined
CallUndefinedReceiver2 r2, r0, r1, [4]   // [4] is feedback slot
Return

关键 opcode 类别

正文待写。

  • Lda* / Sta*：Load / Store accumulator
  • Add / Sub / Mul / Div：算术（每个都带 feedback slot）
  • LdaNamedProperty / StaNamedProperty：属性访问（带 feedback slot）
  • Call*（多种特化形式）：根据 receiver / arg count 不同走不同路径
  • Jump / JumpIfTrue / Throw：控制流

Feedback Vector：每个函数的"运行时画像簿"

正文待写。每个 IC 站点对应一个 slot；slot 内容：上次见到的 Map（hidden class）+ 状态机（Uninitialized / Mono / Poly / Mega）；Sparkplug / Maglev / TurboFan 都从这里读取类型反馈。

替代了什么 —— Full-codegen 退役

正文待写。2008-2015 是 Full-codegen（直接 source → 机器码，无 bytecode）；2017 完全切到 Ignition。好处：内存 -50%（bytecode 比机器码小 4-8x）、统一了所有 tier 的输入。

Code Cache & Script Streaming

正文待写。启动期优化的两条腿：

  • Code Cache（V8 4.2+）：首次执行某段脚本时把 bytecode + 编译元数据序列化到磁盘；下次同源脚本直接 deserialize，跳过 parse + bytecode generation
  • Script Streaming（V8 4.6+）：网络下载 JS 时边下载边解析——不等整个脚本到达就开始 parsing
  • 两者配合：典型大型 SPA 启动期 V8 工作量 -30%
  • Cache 失效条件：V8 版本变更、源码 hash 变更、flag 变更；命中失效时静默回退

Generator / Async function 怎么编（状态机化）

正文待写。关键点：

  • function* foo() { yield a; yield b; } 是 suspend/resume 语义；V8 编译成带状态字段的状态机
  • bytecode 里出现 SuspendGenerator / ResumeGenerator 指令
  • Async function：在规范层是"自动调用 await 的 generator"；V8 7.0+ 直接给 async 一组专门的字节码避免中间 Promise 分配
  • Async/Await Zero-cost（V8 7.0，2018）：async 调用比同义 promise.then() 快约 11×、内存少 -2.5×
  • 用户视角的 async/await 语义请参考 JS P3 · 异步

Strict Mode / TDZ 在 bytecode 层

正文待写。

  • 'use strict' 是 SourceTextModule 级 flag，影响 bytecode 生成时哪些 helper 被调用；不是单独 opcode
  • TDZ（let / const）：bytecode 用 ThrowReferenceErrorIfHole 做未初始化检查；这就是 let/const 比 var 慢一点点的根因

内置函数（Builtins）不走这条路

正文待写。关键点：

  • Array.prototype.map / Object.assign / String.prototype.split / Promise.then 这些不是 JS 写的；是 V8 自己的代码，预编译成机器码嵌进 Snapshot
  • 写法分两代：早期用 CodeStubAssembler (CSA)——C++ 内嵌的汇编 DSL；现在用 Torque——TypeScript 风格的高级 DSL，编译器生成 CSA 调用
  • Why Torque：CSA 容易写错且无类型；Torque 强类型 + 结构化控制流 + 直接对应 ECMAScript 规范步骤号（如 // 6.4.1.1 Step 3 写法）
  • Snapshot Embedded：Builtins 在 V8 编译期就生成机器码，启动时通过 mmap 进 Isolate；调用 builtins ≈ 调用 native function，没有 JIT 开销

05 常见陷阱

bytecode 在不同 V8 版本间不兼容

不要保存 bytecode 缓存到磁盘期望跨版本可用（V8 自己的 code cache 会校验版本）。

Wide opcode（.Wide / .ExtraWide）是操作数扩展，不是不同语义

看 bytecode dump 时容易误读。

bytecode dump 中 register 编号是相对函数开始的偏移

不是绝对位置。
