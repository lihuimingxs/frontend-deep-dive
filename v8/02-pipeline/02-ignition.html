Ignition 解释器与字节码

Ignition 是 V8 的"嘴巴"——所有代码先变 bytecode 再说。这一章讲它的设计哲学与字节码格式：register-based VM（vs JSC LLInt 的 stack-based）、bytecode handler 用 CodeStubAssembler 写、Feedback Vector 是运行时画像簿。再补充三个常被忽略的话题：Code Cache & Script Streaming（启动期优化两条腿）、Generator / Async function 状态机化（含 V8 7.0 zero-cost desugar）、以及 Builtins 不走这条路——内置函数用 Torque DSL 写、预编译嵌进 Snapshot。

阅读时长 ~35 分钟 前置 P2.1 + ECMA P4（数据类型） 主要来源 v8.dev/docs/ignition · v8.dev/docs/torque · slack-tracking 博客

01 速览

  1. Ignition 是 register-based 解释器（不是 stack-based），bytecode 操作显式引用寄存器。
  2. bytecode 是 V8 内部的"通用语言" —— Sparkplug / Maglev / TurboFan 都从这里起步。
  3. Builtins 不走这条路 —— Array.prototype.map / Object.assign 等用 Torque 写、预编译嵌进 Snapshot。

02 底层逻辑

Register-based VM（vs Stack-based）

解释器设计有两种基本架构：stack-based（操作数压栈、指令从栈顶取）和 register-based（指令显式引用虚拟寄存器）。JVM、CPython、JSC LLInt 都是 stack-based；V8 Ignition 是 register-based。这个选择不是随意的——register-based VM 每条指令做更多事，dispatch 次数减少 30-50%。

具体差异：算个 a + b，stack-based 要 3 条指令（push a / push b / add）；register-based 一条 Add r1, r2 搞定。在 stack-based VM 里，dispatch（决定下一条指令是什么）的开销和 cache miss 相对计算占比很大——register-based VM 减少了 2/3 的 dispatch，对解释器整体性能提升明显。

代价是 bytecode 体积稍大（每条指令要带寄存器编号操作数，stack-based 不需要）和 bytecode 生成实现复杂度更高（要做寄存器分配，stack-based 不需要）——但 V8 团队认为生产环境性能换这点工程成本值得。每个 Ignition 函数的虚拟寄存器集是 accumulator（特殊寄存器，许多算术指令默认用它）+ locals（用户局部变量）+ parameters（函数参数），编号都是相对函数起点的偏移。

Bytecode handler 用 CodeStubAssembler (CSA) 写

每条 bytecode 都对应一段叫 handler 的代码——它知道怎么执行这条指令并跳到下一条。这些 handler 怎么写？

一个朴素方案是用 C++ 写每个 handler。但 C++ 编译出来的 dispatch 代码不够紧凑——每条 handler 入口要做很多寄存器保存 / 调用约定处理，dispatch 一次的开销轻松到几十纳秒。另一个朴素方案是直接写汇编。但要给 x64 / ARM64 / ARM32 / RISC-V 都各自维护一份汇编，工程成本高得离谱。

V8 的方案是 CodeStubAssembler (CSA)——一个内嵌在 C++ 里的 portable assembler DSL。开发者用 C++ 接口写 handler 逻辑（如 auto value = LoadObjectField(obj, kValueOffset)），CSA 在 V8 编译期把它转换成各平台的机器码——开发者一次写完，多个 ISA 同时受益。这套机制其实是从 TurboFan 的指令选择阶段抽出来的：CSA 复用 TurboFan 的 backend 生成机器码，所以 handler 的代码质量和 TurboFan 自己生成的机器码相当。

后来 V8 发现 CSA 还是太底层、容易写错（比如忘记检查 hidden class、忘记 GC 安全），所以又在 CSA 之上做了一层叫 Torque 的 DSL（详见本章末尾"Builtins"段），现在新的 handler / built-in 优先用 Torque 写、旧的逐步迁移。

典型 bytecode 示例

bytecode 
function sum(a, b) { return a + b; }
sum(1, 2);

// d8 --print-bytecode 输出（简化）：
LdaSmi.Wide [1]            // Load SMI 1 to accumulator
Star r0                    // Store to register r0
LdaSmi.Wide [2]
Star r1
LdaUndefined
CallUndefinedReceiver2 r2, r0, r1, [4]   // [4] is feedback slot
Return

关键 opcode 类别

Ignition 字节码集大约 200 条 opcode（V8 12.x），按功能分几大类：

  • Load / Store（Lda* / Sta*）—— Lda = "Load to Accumulator"，Sta = "Store from Accumulator"。许多算术指令都隐式使用 accumulator，避免显式指定 destination 寄存器节省 bytecode 体积。
  • 算术族（Add / Sub / Mul / Div / BitwiseOr / ...）—— 每条都带 feedback slot 操作数，运行时记录"本次操作数类型是 SMI / HeapNumber / String"，编译器用这些信息做 speculation。
  • 属性访问族（LdaNamedProperty / StaNamedProperty / LdaKeyedProperty / ...）—— 属性访问是 V8 性能最关键的操作，每条 opcode 都有 feedback slot，运行时记录见过的 hidden class（详见 P3.2 IC）。
  • 调用族（CallProperty0 / CallUndefinedReceiver1 / CallNoFeedback / ...）—— V8 根据 receiver 类型 + 参数个数有多个特化版本。CallProperty2 表示"用 receiver 调用，2 个参数"——比通用 Call 少几次类型判断。这是用 bytecode 体积换运行时速度的典型例子。
  • 控制流（Jump / JumpIfTrue / JumpIfNullOrUndefined / Throw / SuspendGenerator）—— 包括 generator 专用的 suspend / resume 指令（详见后文 Generator 段）。
  • 对象创建（CreateObjectLiteral / CreateArrayLiteral / CreateClosure）—— 复杂对象构造单独 opcode 处理，编译器能识别这些是"已知形状"。

每条 opcode 还有 .Wide（操作数 16-bit）/ .ExtraWide（操作数 32-bit）变体——不是不同语义，仅当寄存器编号 / feedback slot 编号大于 256 时使用。看 bytecode dump 时容易把 Wide 误读成"特殊指令"。

Feedback Vector：每个函数的"运行时画像簿"

Feedback Vector 是 V8 性能体系的神经系统。每个被编译的 JS 函数都关联一个 Feedback Vector——本质是一个数组，存"运行时观察到的类型信息"。每个 IC 站点（属性访问 / 函数调用 / 算术 / 类型判断）在 Feedback Vector 里有一个 slot，slot 内容随运行时演化。

典型 slot 内容：状态机标记（Uninitialized / Premonomorphic / Monomorphic / Polymorphic / Megamorphic）+ 观察到的 Map 列表（指向 hidden class）+ 命中计数。Ignition 在执行每条 IC opcode 时同时更新 Feedback Vector 和返回结果——更新成本几纳秒，但收集的信息让后续优化层得以做激进决策。

这就是为什么 V8 性能优化的所有出发点都是 Feedback Vector：Sparkplug 在生成机器码时从这里读取"是否已 monomorphic"决定要不要内联快路径；Maglev 从这里读"用过的 Map 集合"决定 inline cache 链长度；TurboFan 从这里读"99% 是 SMI"假设并做 speculation。一旦运行时违反 Feedback Vector 的假设——deopt 跳回 Ignition，重新积累。这是个完整闭环：解释器边跑边记录 → 优化器读记录做假设 → 假设错了退回到解释器。

Feedback Vector 与函数独立——每个闭包实例都有自己的 Feedback Vector。这就是为什么"高频闭包工厂"（如 arr.map(item => item.x) 在循环里被调用上千次但每次都是新闭包）会让 V8 性能受损：每个闭包都要从头积累 feedback。这是 P3.2 详细讨论的话题，本章只点到。

替代了什么 —— Full-codegen 退役

Ignition 不是 V8 第一天就有的——它在 2017 年才取代 Full-codegen 成为默认入口。Full-codegen 是 2008-2015 期间 V8 的"基线编译器"——直接把源代码编译成机器码，无 bytecode 中间层。这在当时是激进选择（同代 SpiderMonkey 仍是解释器为主），但留下了几个不可调和的问题。

第一是内存。机器码比 bytecode 大 4-8 倍——一段简单的 function add(a, b) { return a + b; } 编译成 Full-codegen 机器码可能 200+ 字节，编成 bytecode 仅 20-30 字节。低端 Android 设备（2015 年 1GB 内存常见）跑 web 应用时 Full-codegen 占 RAM 100MB+ 是浏览器卡顿的主因。

第二是各 tier 输入不一致。Full-codegen 时代 Crankshaft 优化器从 AST 起步重新生成 IR，TurboFan 从 AST 起步生成 sea of nodes——两个优化器各自重做一遍 AST 走查。换成 Ignition 之后所有优化器都从 bytecode 起步，省了大量重复工作，且 bytecode 是稳定的中间格式，不会因为 parser 修改而连锁影响优化器。

第三是Code Cache难做。机器码与 CPU ISA 强绑定，缓存到磁盘后跨设备不能共享；bytecode 是平台无关的，可以在用户首次访问页面时缓存到磁盘，下次访问跳过 parse 直接 deserialize。Code Cache 是 Chrome 启动期的关键优化，Ignition 是它落地的前置条件。

这三个理由叠加，让 V8 团队在 2015-2017 完成了 Ignition 项目并在 V8 5.9 一次性删除 Full-codegen——也就是 P1.1 讲的"2017 大重构"那次。今天看 V8 内部的 bytecode，本质是一份"为支持低端设备 + 多 tier 优化器 + 跨设备 cache"三方需求设计的中间表示。

Code Cache & Script Streaming：启动期优化的两条腿

首次执行 Ignition 仍要做完整工作（parse 源码 → 生成 bytecode → 解释执行）；但一段脚本的第二次访问不应该重做这些。这就是 Code Cache 的目的。

Code Cache（V8 4.2+，2015 起）：第一次执行某脚本时，V8 把 bytecode + 编译元数据序列化到磁盘缓存；下次同源脚本（同 URL 或同 source hash）直接 deserialize，跳过 parse + bytecode generation。Chrome 把它存在 HTTP 缓存旁边——访问页面时和 HTML / JS / CSS 一起从磁盘读。Node 也暴露了这套机制：vm.Script 的 cachedData 选项让嵌入器手动管理 cache。

Script Streaming（V8 4.6+）：补 Code Cache 第一次访问的 case。网络下载 JS 时 V8 不等整个脚本到达，边下载边 parse——一段 1MB 的 JS 在 100KB 时就开始 parsing，下载完时 parsing 也基本完成。这把 V8 的工作"藏在"网络等待里，对感知性能影响巨大。

两者配合：典型大型 SPA 启动期 V8 工作量 -30%。失效条件：V8 版本变更（cache 格式改了）、源码 hash 变更、相关 flag 变更——命中失效时 V8 静默回退到完整 parse，没有性能崩溃只是没有加速。

Generator / Async function 怎么编（状态机化）

看一段 generator：

javascript 
function* counter() {
  let i = 0;
  while (true) {
    yield i++;
    yield i * 2;
  }
}

每次外部调 .next() 函数要从上次 yield 处恢复——这不是普通函数调用语义，是 suspend / resume。V8 怎么实现？

编译时 V8 把 generator 转成带状态字段的状态机——bytecode 里多出 SuspendGenerator 和 ResumeGenerator 两条专用指令。SuspendGenerator 把当前所有 register 状态（locals、accumulator、当前 bytecode offset）保存到 generator object 上，然后 return 给调用方；下次 ResumeGenerator 时把这些状态还原回 register，从保存的 offset 继续执行。本质是显式做了协程的 stack switch。

Async function 在 ECMAScript 规范层是"自动调用 await 的 generator + Promise 包装"。早期 V8（5.x-6.x）就是这样实现的——把 async desugar 成 generator + Promise hook，每次 await 创建一个中间 Promise。这个实现简单但有性能问题：每个 await 都分配 Promise，造成大量短命对象给 GC 压力。

2018 年 V8 7.0 落地的 "Faster async functions and promises"（俗称 zero-cost async）改写了这套：V8 给 async 一组专门的字节码——SuspendGenerator 在 async 上下文中变体直接持有 await 的值不再创建中间 Promise，仅在最外层 return 时构造一个 Promise。结果：async 调用比同义 promise.then() 快约 11×、内存少 2.5×。这是 V8 历史上单一优化收益最大的一次之一。

用户视角的 async/await 语义请参考 JS P3 · 异步；规范层定义参见 ECMA P7.3 · Event Loop & 异步演进。

Strict Mode / TDZ 在 bytecode 层

Strict Mode（'use strict'）不是单独的 opcode 集，而是SourceTextModule 级别的 flag——影响 bytecode 生成时调用哪些 helper。例：strict 下 this 不再自动盒装（function f() { return this; } 直接调用时返回 undefined 而非 global object）；这种行为差异不是运行时检查，而是 bytecode 生成时直接选了不同 opcode。所以 strict mode 没有运行时性能开销，反而因为 V8 能做更多假设有时更快。

Temporal Dead Zone（let / const 的"声明前不能用"）就需要运行时检查了。bytecode 里 V8 用 ThrowReferenceErrorIfHole 在每次访问 let / const 变量前做检查——如果当前 slot 还是 "the hole"（未初始化标记值），抛 ReferenceError。这是 let/const 比 var 慢一点点的根因——多了一次 hole check。在 hot loop 里 V8 优化层会把这个 check 提到循环外（一次 check 验证所有迭代），实际开销可忽略，但 cold path 上仍可观测。

内置函数（Builtins）不走 bytecode 路径

Array.prototype.map / Object.assign / String.prototype.split / Promise.then ——这些 ECMAScript 规范定义的"标准库函数"在 V8 内部不是用 JS 写的。它们是 V8 团队精心写的、被几十亿台设备每天执行数万亿次的核心代码。

为什么不用 JS 写？因为 built-in 写成 JS 会走 Ignition / Sparkplug / Maglev / TurboFan 全流水——每个 isolate 启动时都要重新编译，开销大且不稳定。所以 V8 把 built-in 预编译成机器码嵌进 Snapshot——启动时直接 mmap 进 Isolate，调用 built-in ≈ 调用 native function，没有 JIT 开销也不需要重复编译。

写 built-in 的方式分两代：早期用 CSA（前文介绍过的 CodeStubAssembler）——C++ 嵌入式 DSL，但太底层、容易出错（忘记 hidden class 检查、忘记 GC 安全、寄存器分配错）。V8 团队 2018 起推出 Torque——TypeScript 风格的高级 DSL，强类型 + 结构化控制流 + GC 安全自动检查。Torque 编译器生成 CSA 调用，CSA 再生成机器码——三层架构看起来繁琐但每层有明确职责。

torque 
// V8 内部 Array.isArray 的 Torque 实现（简化）
@export
transitioning javascript builtin ArrayIsArray(
    js-implicit context: NativeContext)(arg: JSAny): Boolean {
  if (TaggedIsSmi(arg)) return False;
  const argMap: Map = LoadMap(UnsafeCast<HeapObject>(arg));
  return IsJSArrayMap(argMap) ? True : False;
}

可以看到 Torque 长得很像 TypeScript，但每个类型都映射到 V8 内部表示（Smi / HeapObject / Map），transitioning 关键字标记函数可能触发 GC 等。Torque 还能直接对应 ECMA-262 规范步骤——很多 V8 built-in 源码里直接用 // 6.4.1.1 Step 3 这样的注释标规范位置，让代码与规范保持可追溯一一对应。

这意味着：JS 程序里看着像"标准库函数调用"的东西，背后是 Torque/CSA 的 native 代码。理解这一点解释了很多性能现象——例 Array.from(iterable) 比手写 for-of loop 快，是因为 Torque 实现避开了 IC 站点；而 Object.keys(obj).map(...) 链有时不比手写循环快，因为 map 的 callback 是用户 JS 函数，仍走 IC + tier-up 流程。

03 应用场景

场景 A · 用 --print-bytecode 验证 V8 实际看到了什么

最直接的调试方式：node --print-bytecode-filter=funcName --print-bytecode script.js。看到的 bytecode 是 V8 优化决策的实际输入——如果你以为某段代码"被内联了"，看 bytecode 就能直接验证。常见用法：观察某个函数在不同写法下生成的 bytecode 数量（更长 = 更慢解释 + 更慢 tier-up），或确认某个 ES 特性（如 destructuring）是否被编译器特别优化（destructuring 通常比手动 unpack 多几条指令但被后续优化层折叠）。

场景 B · 闭包 vs 共享函数：Feedback Vector 视角

常见反模式：arr.map(item => process(item)) 在循环里被调用上千次。每次循环都创建新闭包——每个新闭包都有独立的 Feedback Vector，要从 Uninitialized 重新积累 IC 状态，且 Maglev/TF 不可能编译这个一次性闭包。优化方式：把 item => process(item) 提到循环外定义一次（共享闭包共享 Feedback Vector），或直接用 arr.map(process)（如果 process 签名匹配）。这条原则也适用 React 渲染——每次 render 都新建 callback 是同样问题。

场景 C · 函数大小与 Lazy Parsing

大函数（> 600 字符）V8 默认不做 lazy parsing——成本太高、不如直接完整 parse。这就是为什么"webpack bundle 一个超大模块" V8 启动慢的原因之一：所有函数被强制 eagerly parse。优化方向：让模块代码模块化、避免单文件几千行、用动态 import 切代码——既减小启动期 V8 工作量，又让被切出去的模块在真正用到时才 parse。这也是为什么 React Server Components 启动相对快——大量"只在服务端跑"的代码不会到客户端 V8 那里。

05 常见陷阱

bytecode 在不同 V8 版本间不兼容

不要保存 bytecode 缓存到磁盘期望跨版本可用（V8 自己的 code cache 会校验版本）。

Wide opcode（.Wide / .ExtraWide）是操作数扩展，不是不同语义

看 bytecode dump 时容易误读。

bytecode dump 中 register 编号是相对函数开始的偏移

不是绝对位置。
