3 层选型决策树 + 12 个陷阱 + v7 展望

整个主题的收束章。从"选 Vercel AI SDK 还是 Claude Agent SDK"到"v5 → v6 升级时机"到"具体 hook / API 怎么调"一条决策链。同时把 12 个陷阱配修复合在一章，最后给 v7 canary 展望（WorkflowAgent / durable execution）+ 2026 强观点收尾。

阅读时长 ~22 分钟 前置 读过 P1-P6（本章是收束） 主要来源 整合 P1-P6 + ai-sdk.dev/v7/docs（v7 canary）+ vercel.com/blog/a-new-programming-model-for-durable-execution + GitHub issue #14011（v7 跟踪）

01 速览

  1. 3 层选型决策树：(a) 第一层：Vercel AI SDK vs Claude Agent SDK（基于场景）；(b) 第二层：Vercel 内部 API（generateText / streamText / useChat / useObject / streamUI / Agent）；(c) 第三层：v5 → v6 升级时机 + v7 canary 等等。
  2. 第一层主要规则：前端 web app + 主要 chat / generative UI → Vercel；CLI 工具 / 终端 agent → Claude SDK；后端复杂 agent orchestration（多 subagent / 长任务） → Claude SDK；前后端混合 → Vercel 跑前端 + Claude SDK 跑后端。
  3. 第二层主要规则：简单单次调用 → generateText / generateObject；UI 流式 → streamText / streamObject；chat 应用 → useChat；流式结构化 UI → useObject；LLM 决定渲染什么 React 组件 → streamUI；复杂 agent + 多 tool + 复用 → Agent / ToolLoopAgent。
  4. 第三层主要规则：新项目直接 v6（current latest stable）；已在 v5 升 v6 较平滑（spec 改进）；仍在 v4 必须先升 v5（协议层 + state 模型双重重做）；v7 canary 不要在生产用。
  5. 12 个陷阱清单：每条配修复（v5 SSE 协议适配 / UIMessage 塞 metadata / tool 错误未处理 / streamUI 在 client / Edge timeout / schema z.any() / v4 v5 混用 / Gateway budget / 没用 prompt caching / 缺 requiresApproval / 初始 messages 缺 / 漏 convertToModelMessages）。
  6. v7 canary 展望：(a) WorkflowAgent（durable execution，agent task 可持久化、跨 process 重启、跨小时几天运行）；(b) Loop control（stopWhen / prepareStep 让 agent loop 可显式控制）。2026 不要在生产用 v7。
  7. 2027 推测：MCP 在 Vercel 一等公民（不再 experimental）；durable execution 进 stable；多 agent orchestration 标准化；模型分级 cost 优化成行业最佳实践。
  8. 2026 强观点收尾：(a) "前端 chat / generative UI 选 Vercel AI SDK 没争议"；(b) "复杂后端 agent + 与 Claude Code 共生态选 Claude Agent SDK"；(c) "v6 是 2026 主线，新项目直接 v6"；(d) "MCP 是工具接口的事实标准，不要再自定义"；(e) "AI Gateway 不是必须，但简化多 provider 切换"；(f) "模型分级是最大 cost 杠杆（可省 50-80%）"。

v7 canary 展望

v7 canary 在 2026 年陆续 release，关键创新：

WorkflowAgent（durable execution）

问题背景：v6 ToolLoopAgent 假设 "agent 跑在一个 process 里直到结束"。但生产现实里：

  • 深度研究 agent 跑几小时几天
  • 批处理 agent 跑几千个 task
  • 用户中途切走，agent 继续后台跑
  • process 重启 / 容器迁移 / 服务器宕机 → state 丢失

v7 WorkflowAgent + Vercel durable execution programming model 解法：

  • 状态持久化：每步执行后 state 写入 durable storage
  • 跨 process 重启：process 死了，恢复后从断点继续
  • 跨小时几天执行：长任务不依赖单一 process 生命周期

Loop control（stopWhen / prepareStep）

v6 ToolLoopAgent 已有简单的 stopWhen / onStep。v7 进一步：

  • stopWhen 更精细：基于 tool calls / token usage / 自定义业务条件
  • prepareStep：每步执行前的 prepare hook（修改 messages / 调整 tool 子集 / 注入 context）
  • 更细粒度控制 agent loop 行为

2026 vs 2027 推测

  • 2026：v6 主线稳定使用；v7 canary 仅实验 / 评估方向；MCP 已 de facto 标准但 Vercel 集成 experimental
  • 2027（推测）：v7 stable release；durable execution 进生产；MCP 在 Vercel 一等公民；多 agent orchestration 出现行业标准模式；模型分级 + AI Gateway 成默认实践

2026 强观点收尾

本主题的核心论断（按确定性强弱排序）：

  • "前端 chat / generative UI 选 Vercel AI SDK 没争议"。useChat / useObject / streamUI 没有竞争对手。Claude SDK 没 UI 层。
  • "复杂后端 agent + 与 Claude Code 共生态选 Claude Agent SDK"。Subagents 一等公民 / 完整 hooks / sessions / built-in tools 让复杂 agent 比 Vercel 更直接。
  • "v6 是 2026 新项目主线"。Agent abstraction + Human-in-the-loop + AI Gateway 都是 v6 起。v5 不要新启。
  • "MCP 是 2026 工具接口事实标准"。自家工具发布为 MCP server（in-process 或 standalone），不要自定义协议。
  • "AI Gateway 不是必须但通常划算"。多 provider + observability + cost 控制 + caching 一站。除非合规不允许。
  • "模型分级是最大 cost 杠杆"。Haiku 跑 routing / Sonnet 跑核心 / o1 仅复杂推理 → 可省 50-80% cost。
  • "prompt caching 是被低估的 cost 优化"。长 system prompt 项目开 caching → token 账单减少 80%。
  • "v7 canary 不要在生产用"。WorkflowAgent / durable execution 仍 unstable。等 2027。
  • "前后端混合架构 = Vercel 跑前端 + Claude SDK 跑后端 agent"是 2026 复杂应用的标准模式。
  • "OTel 监控 LLM 调用是 day-0 设置"，不是"以后再加"。从开始就接 collector 看 token / latency / 错误率。

03 应用场景：决策实战

场景 A · 新 SaaS 项目（2026 首选栈）

(1) Next.js + Vercel AI SDK v6（前端 useChat + 后端 streamText）；(2) AI Gateway（多 provider + observability + budget）；(3) Anthropic Prompt Caching（长 system prompt）；(4) 模型分级（Haiku 路由 / Sonnet 核心）；(5) 敏感操作 requiresApproval；(6) OTel 接 Honeycomb / Langfuse；(7) MCP server 接外部工具（GitHub / Slack）。

场景 B · 升级老项目（v4 → v6）

步骤：(1) 升 v5（自定义后端必须适配 SSE 协议；Next.js Route Handler 几乎无感）；(2) UIMessage / ModelMessage 二分（前端 messages 加 metadata 不污染 LLM）；(3) tool-invocations 改 type-specific parts（前端渲染按 type 分支）；(4) 升 v6（Agent abstraction 复用 + Human-in-the-loop）；(5) 接 AI Gateway（如果多 provider）。

场景 C · 内部 dev tool（Claude Agent SDK）

(1) Claude Agent SDK Python / TypeScript；(2) Subagents（codeReviewer / testRunner / securityScanner）；(3) Hooks 阻止危险命令 + 脱敏输出；(4) In-process MCP server（自家业务工具）+ 接 GitHub MCP（外部）；(5) Sessions 持久化对话；(6) OTel 监控。

场景 D · 前后端混合架构（chat + 复杂 agent）

(1) 前端 Next.js + Vercel useChat；(2) 后端 Route Handler 转给 Claude Agent SDK（Node runtime / 或单独 Python service）；(3) 后端 agent 多 subagent 协作；(4) 后端流式输出转换成 Vercel SSE 协议给前端；(5) 用户审批走 Vercel useChat 的 Human-in-the-loop（或后端 hooks 双重保险）。

场景 E · 小创业 MVP（最简栈）

仅 useChat + streamText + 一个 model（Anthropic Sonnet 4.6）+ 几个 tools。不上 Agent abstraction / Gateway / OTel / Prompt Caching / 模型分级。等用户量上来再优化。"过度工程化是敌人"。

场景 F · 数据 ETL agent（批处理）

每天跑 1000 个数据源 → ETL → 入库。不要在 HTTP request 跑。进 queue（Inngest / Trigger.dev / 自建 BullMQ）→ Claude Agent SDK Python worker 跑 → 失败重试 + 进度持久化。前端只看 dashboard 不直接跑 agent。

核心代码骨架

typescript 
// agents/support.ts
import { Agent, tool } from 'ai';
import { gateway } from '@ai-sdk/gateway';
import { z } from 'zod';
import * as supportTools from '@/tools/support';

export const supportAgent = new Agent({
  model: gateway('anthropic/claude-sonnet-4-6'),  // 通过 Gateway
  system: COMPANY_SYSTEM_PROMPT,  // 长 prompt → 自动 prompt caching by Gateway
  tools: {
    lookupOrder: supportTools.lookupOrder,
    lookupCustomer: supportTools.lookupCustomer,
    refundOrder: supportTools.refundOrder,  // requiresApproval: true
    sendEmail: supportTools.sendEmail,      // requiresApproval: true
  },
});

// app/api/chat/route.ts
import { supportAgent } from '@/agents/support';
import { convertToModelMessages } from 'ai';

export const runtime = 'edge';
export const maxDuration = 60;

export async function POST(req: Request) {
  const { messages } = await req.json();

  const result = supportAgent.stream({
    messages: convertToModelMessages(messages),
    experimental_telemetry: {
      isEnabled: true,
      functionId: 'support-chat',
      metadata: { sessionId: messages[0].id },
    },
  });

  return result.toUIMessageStreamResponse();
}

// app/chat/page.tsx
'use client';
import { useChat } from 'ai/react';
import { ApprovalPrompt } from '@/components/ApprovalPrompt';

export default function ChatPage() {
  const { messages, sendMessage, respondToApproval } = useChat({
    api: '/api/chat',
  });

  return (
    <div>
      {messages.map((m) => (
        <div key={m.id}>
          <strong>{m.role}:</strong>
          {m.parts.map((part, i) => {
            if (part.type === 'text') return <span key={i}>{part.text}</span>;
            if (part.type === 'tool-refundOrder' && part.state === 'pending-approval') {
              return <ApprovalPrompt key={i} part={part} respond={respondToApproval} />;
            }
            if (part.type === 'tool-lookupOrder') {
              return <OrderCard key={i} {...part} />;
            }
            return null;
          })}
        </div>
      ))}
      <ChatInput onSend={sendMessage} />
    </div>
  );
}

cost 优化清单（按 ROI 排序）

text 
1. 模型分级 (省 50-80% cost)
   ├─ Haiku 4.5 跑路由 / 分类 / 简单总结
   ├─ Sonnet 4.6 跑核心对话 / agent
   └─ Opus 4.7 / o1 仅用于复杂推理

2. Prompt Caching (省 60-80% on repeat)
   ├─ Anthropic cacheControl: { type: 'ephemeral' }
   └─ AI Gateway 自动启用

3. 限制 maxSteps (避免 agent 失控)
   └─ ToolLoopAgent maxSteps: 10

4. 缓存常见 query (Redis / KV)
   └─ FAQ / 静态分类 → 完全 cache 化响应

5. AI Gateway budget controls
   └─ 月度上限 + per-user rate limit

6. 监控 token usage 异常
   └─ OTel + alert: 单次调用 > 50k tokens / 用户单日 > $5

7. 精简 system prompt
   ├─ 移除冗余示例
   └─ 用 prompt caching 代替"重发 examples"

8. 选 model 时考虑 context window
   └─ 不要为了 "更稳" 都用 200k context model

05 12 个陷阱清单（收束）

陷阱 1 · v5 自定义后端没适配 SSE 协议

FastAPI / Python / 自建后端用 v5 useChat 收不到流。修复：用官方 Python adapter 或手写 SSE 响应（详见 P2.2）。

陷阱 2 · UIMessage 塞 metadata 给 LLM

把要发给 LLM 的信息放 metadata（不会被发）。修复：明确"前端展示用 metadata；发给 LLM 用 parts"。

陷阱 3 · Tool execute 抛错没处理

整个 chat 死。修复：在 execute 内 try/catch（return error 对象）；或让它抛（SDK 自动作为 tool result 喂回 LLM）。

陷阱 4 · streamUI 在 Client Component 调

报错。修复：streamUI 必须在 Server Action / Route Handler 调，client 通过 server action 触发。

陷阱 5 · Edge runtime 跑长 agent

25s timeout 不够。修复：用 Node runtime（maxDuration 15min）或拆任务进 queue。

陷阱 6 · Tool schema 用 z.any()

LLM 不知传什么 → 输出可能任何东西。修复：精确 schema + 每个字段 .describe('...')。

陷阱 7 · 混用 v4 v5 API

类型不兼容 / 协议错。修复：全升 v5+；按 v5 升级文档改。

陷阱 8 · AI Gateway 没设 budget

月底惊喜账单。修复：dashboard 设月度 budget + alert。

陷阱 9 · 长 system prompt 没用 prompt caching

token 账单膨胀 5-10x。修复：开 Anthropic / OpenAI 各自 prompt caching；或用 AI Gateway 自动启用。

陷阱 10 · 敏感 tool 没 requiresApproval

LLM silent 删除 / 发邮件 / 付款。修复：所有"修改 / 发送 / 删除 / 付款"类 tool 加 requiresApproval: true。

陷阱 11 · useChat 初始 messages 没设

每次刷新丢历史。修复：通过 initialMessages 注入（从 server / localStorage / DB 读）。

陷阱 12 · 漏 convertToModelMessages

v5+ 必须把 UIMessage[] 转 ModelMessage[]。修复：messages: convertToModelMessages(messages)。
