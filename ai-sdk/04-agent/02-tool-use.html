Tool use 全谱：定义 / 流式 input / Human-in-the-loop / 错误处理

Tool use 是 LLM 应用最关键的能力。这一章把 Vercel AI SDK 的 tool 定义、v5+ 流式 input（partial-call → call → result 状态机）、v6 stable 的 Human-in-the-loop 审批（敏感操作人审）、多 provider 协议差异（OpenAI / Anthropic / Google）、错误处理（execute 抛错喂回 LLM / retry 策略）讲清。

阅读时长 ~22 分钟 前置 读过 P4.1（Agent + ToolLoopAgent） 主要来源 ai-sdk.dev/docs/ai-sdk-core/tools-and-tool-calling + Vercel AI SDK 6 Human-in-the-loop 文档 + OpenAI Function Calling docs + Anthropic Tool Use docs + Zod 文档

01 速览

  1. Tool 三件套：description（LLM 看的 docstring）+ parameters（Zod schema）+ execute（实际执行的异步函数）。也可以用 tool() 工厂函数包装。
  2. 流式 input（v5+ 默认）：LLM 边生成 tool input 边发送（不等整个 input 生成完）。前端通过 part state 看到 state: 'partial-call'（部分输入）→ 'call'（输入完整）→ 'result'（已执行）。
  3. UI 边收边显示：partial-call 时显示 spinner / "正在准备调用 X"；call 阶段显示 "正在调用 X with args"；result 阶段显示结果卡片。
  4. Human-in-the-loop tool approval（v6 stable）：给 tool 加 requiresApproval。useChat 收到 pending approval 的 tool call 时不立即执行；UI 渲染审批 prompt（"是否允许调用 X？"）；用户批准后通过 SDK API 继续。
  5. 错误处理：tool execute 抛错 → SDK 捕获 → 把错误信息作为 tool result 喂回 LLM → LLM 决定 retry / 用其他 tool / 告诉用户。可配置 onError 回调 + 自定义 toModelOutput。
  6. 多 provider 协议差异：OpenAI Function Calling（JSON schema 驱动 + parameters 字段）/ Anthropic Tool Use（input_schema 字段 + 流式协议略不同）/ Google（Gemini function calling）。Vercel AI SDK 抽象掉差异 —— 一份 tool 定义所有 provider 都能用。
  7. tool 类别：(a) Execute tools（典型）—— execute 函数被调用；(b) Provider tools（特定 provider 内置）—— 比如 OpenAI web_search / file_search / Anthropic computer_use，由 provider 直接执行；(c) UI tools（streamUI 风格）—— execute 返回 React 组件（详见 P3.2）。
  8. 2026 实际项目分布：~80% 项目用 execute tools（业务自定义）；~10% 用 provider tools（搜索 / 文件等）；~10% 混合。Human-in-the-loop 在涉及"删除"/"发送"/"付款"的项目里 ~30% 启用率。

02 底层逻辑

Tool 定义全谱

typescript 
import { tool } from 'ai';
import { z } from 'zod';

// 基础 tool 定义
const getWeather = tool({
  description: 'Get the current weather for a city',  // LLM 决定何时调用基于这段描述
  parameters: z.object({
    city: z.string().describe('The city name, e.g. "San Francisco"'),
    unit: z.enum(['celsius', 'fahrenheit']).optional().default('celsius'),
  }),
  execute: async ({ city, unit }) => {
    const data = await fetchWeatherAPI(city);
    return {
      temperature: unit === 'celsius' ? data.tempC : data.tempF,
      condition: data.condition,
      unit,
    };
  },
});

// 用法
await streamText({
  model: openai('gpt-4o'),
  tools: { getWeather },  // tool 名 = key
  prompt: 'What is the weather in Tokyo?',
});

关键设计：

  • description 决定 LLM 何时调用：精确描述 tool 用途；包含示例输入更好
  • parameters 用 Zod schema：SDK 自动转成 LLM 能理解的 JSON schema
  • 每个 schema 字段加 .describe('...')：让 LLM 理解每个字段的语义和格式期望
  • execute 是异步函数：返回值直接作为 tool result 喂回 LLM

流式 input 状态机（v5+ 默认）

v5 起 tool input 默认流式生成。前端能看到三种状态：

text 
LLM 决定调用 getWeather(city: 'San Francisco', unit: 'celsius')

时间线：
  t=0    LLM 开始生成 tool call (输入未完整)
        ↓
        前端收到 part: {
          type: 'tool-getWeather',
          toolCallId: 'call_1',
          state: 'partial-call',
          input: { city: 'Sa' }  // 部分输入
        }
        ↓
  t=1   LLM 继续生成
        ↓
        前端收到 part: {
          type: 'tool-getWeather',
          state: 'partial-call',
          input: { city: 'San Fran' }
        }
        ↓
  t=2   LLM 完成 tool call 输入
        ↓
        前端收到 part: {
          type: 'tool-getWeather',
          state: 'call',
          input: { city: 'San Francisco', unit: 'celsius' }
        }
        ↓
  t=3   SDK 自动调用 execute
        ↓
  t=4   execute 返回结果
        ↓
        前端收到 part: {
          type: 'tool-getWeather',
          state: 'result',
          input: { city: 'San Francisco', unit: 'celsius' },
          output: { temperature: 18, condition: 'Cloudy' }
        }

UI 渲染策略：

  • partial-call：显示 spinner + "正在准备调用 getWeather"（不显示 input，因为不完整）
  • call：显示 "正在调用 getWeather with args: {input}"
  • result：显示 WeatherCard with output
  • error：显示错误 message

Human-in-the-loop tool approval（v6 stable）

给敏感 tool 加 requiresApproval：

typescript 
const deleteFile = tool({
  description: 'Delete a file',
  parameters: z.object({ path: z.string() }),
  requiresApproval: true,  // 关键
  execute: async ({ path }) => {
    await fs.unlink(path);
    return { deleted: true };
  },
});

const sendEmail = tool({
  description: 'Send email to user',
  parameters: z.object({ to: z.string(), subject: z.string(), body: z.string() }),
  requiresApproval: ({ to }) => !to.endsWith('@example.com'),  // 函数式：仅外部邮件需审批
  execute: async ({ to, subject, body }) => await mailer.send({ to, subject, body }),
});

工作流程：

  1. LLM 决定调 deleteFile → SDK 看到 requiresApproval=true → 不立即执行 execute
  2. SDK 把 tool call 标记为 state: 'pending-approval'（v6 新状态）+ 流到前端
  3. 前端 useChat 看到 pending approval → 渲染审批 prompt（"是否允许删除文件 /tmp/foo？"）
  4. 用户点 "批准" → 调 SDK API respondToApproval(toolCallId, { approved: true })
  5. SDK 执行 execute → 流式返回 result
  6. 用户点 "拒绝" → SDK 把"拒绝"作为 tool result 喂回 LLM → LLM 决定下一步

错误处理

typescript 
const fetchUser = tool({
  description: 'Fetch user by ID',
  parameters: z.object({ id: z.string() }),
  execute: async ({ id }) => {
    const user = await db.user.findUnique({ where: { id } });
    if (!user) {
      // 抛错 - SDK 会捕获并喂回 LLM
      throw new Error(`User ${id} not found`);
    }
    return user;
  },
});

// 默认行为：error 会被 SDK 捕获，作为 tool result 喂回 LLM
// LLM 看到 result: { error: 'User xxx not found' } 会决定下一步
// (告诉用户找不到 / 调其他 tool / etc.)

// 自定义 error 序列化
const fetchUser = tool({
  // ...
  execute: async ({ id }) => { ... },
  toModelOutput: ({ result, error }) => {
    if (error) {
      return { error: error.message };
    }
    // 也可以转换成功结果（比如精简 / 脱敏）
    return { id: result.id, name: result.name };  // 不发 email / phone 给 LLM
  },
});

onError 回调（agent 级别）：

typescript 
await streamText({
  model: openai('gpt-4o'),
  tools: { fetchUser, fetchOrder },
  onError: ({ error }) => {
    // tool execute 失败时调
    console.error('Tool error:', error);
    metrics.increment('tool.error');
  },
  // ...
});

多 provider 协议差异

不同 LLM provider 的 tool calling 协议略不同。SDK 抽象掉差异：

text 
OpenAI Function Calling 原生协议:
{
  "tools": [{
    "type": "function",
    "function": {
      "name": "getWeather",
      "description": "...",
      "parameters": { "type": "object", "properties": {...} }  // ← parameters 字段
    }
  }]
}

Anthropic Tool Use 原生协议:
{
  "tools": [{
    "name": "getWeather",
    "description": "...",
    "input_schema": { "type": "object", "properties": {...} }  // ← input_schema 字段
  }]
}

Google Gemini Function Calling 原生协议:
{
  "tools": [{
    "functionDeclarations": [{
      "name": "getWeather",
      "description": "...",
      "parameters": {...}
    }]
  }]
}

Vercel AI SDK 统一抽象:
tools: { getWeather: tool({ description, parameters: z.object({...}), execute }) }
- SDK 在内部转成各 provider 的协议

流式协议差异：

  • OpenAI：tool call 流式发 delta（incremental JSON）
  • Anthropic：tool call 流式发 input_json_delta（partial JSON 片段）
  • SDK 统一暴露给前端的是 state: 'partial-call' 状态 + 累积的 input object

Provider tools（内置）

某些 provider 提供内置 tool，由 provider 直接执行（不需要你写 execute）：

typescript 
// OpenAI built-in tools
import { openai } from '@ai-sdk/openai';

await streamText({
  model: openai.responses('gpt-4o'),  // 注意是 .responses
  tools: {
    web_search_preview: openai.tools.webSearchPreview({}),
    file_search: openai.tools.fileSearch({ vectorStoreIds: ['vs_xxx'] }),
  },
  prompt: 'Find latest news about AI safety',
});

// Anthropic built-in tools
import { anthropic } from '@ai-sdk/anthropic';

await streamText({
  model: anthropic('claude-sonnet-4-6'),
  tools: {
    computer_use: anthropic.tools.computerUse({
      displayWidthPx: 1920,
      displayHeightPx: 1080,
    }),
  },
  prompt: 'Open Chrome and search for cats',
});

2026 主流 provider built-in tools：

  • OpenAI：web_search / file_search / code_interpreter
  • Anthropic：computer_use（屏幕截图 + 鼠标键盘）/ text_editor / bash
  • Google：grounding_with_google_search

03 应用场景

场景 A · 客服 chat 调内部 API

客户问"我的订单状态" → LLM 调 lookupOrder tool → 返回订单数据 → LLM 生成回复。execute tool 最常见场景。

场景 B · 流式 input UX 优化

tool 输入 schema 大（多个字段）时，partial-call 状态让用户看到"正在准备调用"。避免"按 send 之后好几秒才有反馈"的差体验。前端按 part state 渲染对应 UI。

场景 C · 敏感操作 → Human-in-the-loop

"删除文件 / 发邮件 / 付款 / 取消订单" 等操作必须用户确认。requiresApproval 让 LLM 不能 silent 执行敏感操作。生产 agent 强烈推荐对所有"修改 / 发送 / 删除"操作启用。

场景 D · 网络搜索 → OpenAI web_search built-in

不要自己实现 search tool（搜索引擎 API + 抓取 + 解析），用 OpenAI 内置 web_search_preview。OpenAI 后端自动搜 + 综合结果给 LLM。2026 起 provider built-in tools 越来越多，能用就用。

场景 E · 屏幕自动化 → Anthropic computer_use

需要 LLM 操作浏览器 / 桌面（自动化测试 / 数据提取 / GUI agent）→ Anthropic computer_use built-in tool。LLM 接受截图 + 输出鼠标键盘动作。这是 Anthropic 独家 + 2026 主流 GUI agent 路径。

场景 F · 多 provider 切换

项目支持用户选 model（GPT / Claude / Gemini）。同一份 tool 定义跨 provider 工作 —— SDK 抽象掉协议差异。但要注意 provider built-in tools 是provider-specific，不能跨 provider 用。

04 典型代码：完整 Tool 实战

多 tool 客服 agent + 流式 input UI

typescript 
// agents/support.ts
import { Agent, tool } from 'ai';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

export const supportAgent = new Agent({
  model: openai('gpt-4o'),
  system: 'You are a customer support agent.',
  tools: {
    lookupOrder: tool({
      description: 'Look up order details by ID',
      parameters: z.object({
        orderId: z.string().describe('The order ID, e.g. "ORD-12345"'),
      }),
      execute: async ({ orderId }) => {
        const order = await db.order.findUnique({ where: { id: orderId } });
        return order ?? { error: 'Order not found' };
      },
    }),
    lookupCustomer: tool({
      description: 'Look up customer by email',
      parameters: z.object({ email: z.string().email() }),
      execute: async ({ email }) =>
        await db.customer.findUnique({ where: { email } }),
    }),
    createRefund: tool({
      description: 'Process a refund (REQUIRES APPROVAL)',
      parameters: z.object({
        orderId: z.string(),
        amount: z.number(),
        reason: z.string(),
      }),
      requiresApproval: true,  // 关键
      execute: async ({ orderId, amount, reason }) => {
        return await processRefund(orderId, amount, reason);
      },
    }),
  },
});

// app/api/chat/route.ts
import { supportAgent } from '@/agents/support';
import { convertToModelMessages } from 'ai';

export async function POST(req: Request) {
  const { messages } = await req.json();
  const result = supportAgent.stream({
    messages: convertToModelMessages(messages),
  });
  return result.toUIMessageStreamResponse();
}

前端：流式 input 状态 + 审批 UI

tsx 
'use client';
import { useChat } from 'ai/react';

function SupportChat() {
  const { messages, sendMessage, respondToApproval } = useChat({
    api: '/api/chat',
  });

  return (
    <div>
      {messages.map((m) => (
        <div key={m.id}>
          <strong>{m.role}:</strong>
          {m.parts.map((part, i) => {
            // Text part
            if (part.type === 'text') {
              return <span key={i}>{part.text}</span>;
            }

            // Tool calls (流式 input 状态机)
            if (part.type === 'tool-lookupOrder') {
              switch (part.state) {
                case 'partial-call':
                  return <Spinner key={i} text="Preparing to look up order..." />;
                case 'call':
                  return <Spinner key={i} text={`Looking up order ${part.input.orderId}...`} />;
                case 'result':
                  return part.output.error
                    ? <ErrorCard key={i} message={part.output.error} />
                    : <OrderCard key={i} order={part.output} />;
                case 'error':
                  return <ErrorCard key={i} message={part.errorText} />;
              }
            }

            // Refund tool with approval
            if (part.type === 'tool-createRefund') {
              if (part.state === 'pending-approval') {
                return (
                  <ApprovalPrompt
                    key={i}
                    title="Approve Refund?"
                    details={`Refund ${part.input.amount} for order ${part.input.orderId}`}
                    reason={part.input.reason}
                    onApprove={() => respondToApproval(part.toolCallId, { approved: true })}
                    onReject={() => respondToApproval(part.toolCallId, {
                      approved: false,
                      message: 'User declined the refund',
                    })}
                  />
                );
              }
              if (part.state === 'result') {
                return <RefundConfirmation key={i} {...part.output} />;
              }
            }

            return null;
          })}
        </div>
      ))}
    </div>
  );
}

错误处理 + retry 策略

typescript 
const fetchExternalAPI = tool({
  description: 'Fetch data from external API',
  parameters: z.object({ url: z.string().url() }),
  execute: async ({ url }) => {
    let lastError: Error | null = null;

    // 自己实现 retry（execute 内部）
    for (let attempt = 0; attempt < 3; attempt++) {
      try {
        const res = await fetch(url, { signal: AbortSignal.timeout(10000) });
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return await res.json();
      } catch (err) {
        lastError = err;
        await new Promise((r) => setTimeout(r, 1000 * (attempt + 1)));
      }
    }

    // 都失败了，throw 让 SDK 喂回 LLM
    throw lastError;
  },
  toModelOutput: ({ result, error }) => {
    if (error) {
      // 简化错误信息给 LLM
      return { error: 'Service unavailable, please try again later' };
    }
    return result;
  },
});

05 常见陷阱

陷阱 1 · tool description 太简陋

"description: 'Get weather'" → LLM 不知道何时该调（"Get weather of where? for what?"）。修复：精确描述用途 + 输入示例 + 何时该调用。"Get current weather for a specific city. Use when user asks about temperature, conditions, or weather forecasts."

陷阱 2 · parameters 用 z.any() / z.unknown()

schema 不具体 → LLM 不知传什么 → 输出可能任何东西。修复：精确 schema + 每个字段加 .describe('...') 让 LLM 理解语义。详见 P2.1。

陷阱 3 · tool execute 抛错没处理

整个 chat 死。修复：要么在 execute 内部 try/catch（return error 对象），要么让它抛出（SDK 自动捕获作为 tool result 喂回 LLM）。不要用 process.exit 这种破坏性错误处理。

陷阱 4 · 流式 input 期间用 input 完整渲染

partial-call 状态的 input 是不完整的 JSON（可能 { city: 'Sa' }）。直接渲染 {part.input.city} 会显示半截字符串。修复：partial-call 状态显示 spinner / "正在准备"，不显示 input；call / result 才显示完整 input。

陷阱 5 · 敏感 tool 没标 requiresApproval

LLM 可能 silent 执行删除 / 发邮件 / 付款。这是生产 agent 最大风险。修复：所有"修改 / 发送 / 删除 / 付款"类 tool 必须 requiresApproval: true（或函数式判断更精细）。

陷阱 6 · approval UI 没设 timeout

用户离开页面没批准 → tool call 永远 pending。修复：(a) UI 层显示 timeout 倒计时；(b) server 端记录 pending 时间，超 X 分钟自动 reject + 通知用户；(c) 重要 agent task 进 queue（不依赖 UI 实时）。

陷阱 7 · 期待跨 provider 用 built-in tools

OpenAI 的 web_search / Anthropic 的 computer_use / Google 的 grounding 都是 provider-specific。修复：built-in tools 只在该 provider model 上工作；切换 provider 时要把 built-in 换成自己实现的 execute tool。

陷阱 8 · execute 暴露敏感数据给 LLM

execute 返回 user 对象含 password hash / phone / SSN → 都喂给 LLM context。修复：用 toModelOutput 精简 / 脱敏返回值。把"完整数据"留给 server 端逻辑，把"LLM 需要的最少信息"喂给 LLM。
