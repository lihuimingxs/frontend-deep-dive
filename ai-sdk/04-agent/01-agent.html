Agent + ToolLoopAgent + structured output with tool calling

Vercel AI SDK v6 主线最重要的章节之一。Agent abstraction 把"一次定义 model + system + tools，多处复用"做成一等公民；ToolLoopAgent 内置 agent loop（自动循环调用直到 LLM 没新 tool call）；structured output with tool calling 让"调几个 tool 后返回结构化 object"在 v6 stable。这一章把这三个 v6 新增能力的设计动机、API、与 v5 streamText + tools 的关系讲清。

阅读时长 ~22 分钟 前置 读过 P2（架构 + Stream Protocol）+ 用过 streamText + tools 至少一次 主要来源 ai-sdk.dev/docs/agents + vercel.com/blog/ai-sdk-6 + ai-sdk.dev/v7/docs/introduction（v7 canary 对照）

01 速览

  1. 裸 streamText + tools 的局限：你需要自己写 agent loop（"调 LLM → 看 tool calls → 调 tool → 喂回 LLM → 直到没 tool call"）；多步骤推理（搜索→总结→再搜）需要手写；多个组件复用同一组 tools 不优雅。
  2. v6 Agent abstraction：new Agent({ model, system, tools }) 一次定义，多处复用（Server Action / Route Handler / Client useChat 都能调）。
  3. ToolLoopAgent（Agent 的子类）：内置 tool loop + maxSteps（防 infinite loop）+ stopWhen（自定义停止条件）+ onStep（每步回调用于 observability / debug）+ 自动 tool 调用 / 喂回。
  4. structured output with tool calling（v6 stable）：之前 generateObject 和 tool calling 不能同时用；v6 起两者协作 —— LLM 先调几个 tool 收集信息，再返回符合 schema 的 object。
  5. Agent 不是必须用：简单单次调用仍用 generateText / streamText；只有"复用 + 多 tool + 多步推理"时才升 Agent。
  6. 类型安全：tools 的输入输出类型自动推导；agent.generate / agent.stream 返回的 result 包含完整类型信息。
  7. 与 UI 集成：agent.stream 返回兼容 UIMessageStreamResponse 的 result，能直接给 useChat 消费。
  8. v7 canary 进一步强化：WorkflowAgent（durable execution）+ loop control（stopWhen / prepareStep 更精细）。本章正文锁定 v6 stable，v7 在 P7 展望提及。
  9. 2026 实际项目分布：~40% 项目仍用 streamText + tools（简单场景）；~50% 项目升级到 v6 Agent（复用 + 多 tool）；~10% 用 ToolLoopAgent + 复杂 loop control。

02 底层逻辑

裸 streamText + tools 的工程现实

v5 时代用 streamText + tools 的典型代码：

typescript 
// v5: 后端 Route Handler
export async function POST(req: Request) {
  const { messages } = await req.json();

  const result = streamText({
    model: openai('gpt-4o'),
    system: 'You are a customer support agent.',
    messages: convertToModelMessages(messages),
    tools: {
      lookupOrder: { description: '...', parameters: ..., execute: ... },
      sendEmail: { description: '...', parameters: ..., execute: ... },
    },
    // streamText 默认 maxSteps=1（只调一次 LLM）
    maxSteps: 5,  // 允许多步循环
  });

  return result.toUIMessageStreamResponse();
}

问题暴露：

  • 同样的 model + system + tools 配置每个 endpoint 都要写一遍。如果 customer support agent 在 10 个 endpoint / Server Action 里用，配置散落十处
  • maxSteps 只是简单数字，没法表达"如果 LLM 调了 sendEmail 就停"这种业务逻辑
  • 没有 onStep 钩子（v5 只有 onFinish）—— observability 不够
  • 测试时很难复用 agent 配置（每个 test 重写一遍 streamText 调用）

v6 Agent abstraction 解法

typescript 
// v6: 一次定义
import { Agent } from 'ai';
import { openai } from '@ai-sdk/openai';

export const supportAgent = new Agent({
  model: openai('gpt-4o'),
  system: 'You are a customer support agent.',
  tools: {
    lookupOrder: { description: '...', parameters: ..., execute: ... },
    sendEmail: { description: '...', parameters: ..., execute: ... },
  },
});

// 多处复用
// Route Handler
export async function POST(req: Request) {
  const { messages } = await req.json();
  const result = supportAgent.stream({ messages: convertToModelMessages(messages) });
  return result.toUIMessageStreamResponse();
}

// Server Action
'use server';
export async function askSupport(question: string) {
  const result = await supportAgent.generate({ prompt: question });
  return result.text;
}

// 后台批处理
async function processTickets(tickets: Ticket[]) {
  return Promise.all(tickets.map(async (t) => {
    const result = await supportAgent.generate({ prompt: t.body });
    return { id: t.id, response: result.text };
  }));
}

// 测试
test('支持订单查询', async () => {
  const result = await supportAgent.generate({ prompt: 'where is order 42?' });
  expect(result.toolCalls.some((t) => t.toolName === 'lookupOrder')).toBe(true);
});

关键设计：

  • 定义一次，多处复用：supportAgent 在所有调用点共享配置
  • 类型安全：tools 类型自动推导，agent.generate / agent.stream 返回完整类型
  • 同时支持 generate（非流式）+ stream（流式）：根据场景选

ToolLoopAgent 内置 agent loop

ToolLoopAgent 是 Agent 的子类，专门处理"agent loop until done"：

typescript 
import { ToolLoopAgent } from 'ai';

const researchAgent = new ToolLoopAgent({
  model: openai('gpt-4o'),
  system: 'You are a research assistant. Use tools to find information, then summarize.',
  tools: {
    search: { ... },
    fetchUrl: { ... },
    summarize: { ... },
  },

  maxSteps: 10,  // 最多调用 LLM 10 次（防 infinite loop）

  // 自定义停止条件
  stopWhen: ({ steps }) => {
    const lastStep = steps[steps.length - 1];
    return lastStep.toolCalls.some((t) => t.toolName === 'summarize');
    // 一旦调了 summarize 就停（认为研究完毕）
  },

  // 每步回调（observability / debug）
  onStep: ({ step, totalUsage }) => {
    console.log(`Step ${step.stepNumber}: tools=${step.toolCalls.map((t) => t.toolName).join(',')}`);
    console.log(`Total tokens so far: ${totalUsage.totalTokens}`);
  },
});

const result = await researchAgent.generate({
  prompt: 'Research the impact of AI on healthcare in 2024',
});
console.log(result.text);  // 最终总结
console.log(result.steps);  // 完整步骤历史

agent loop 内部机制

text 
ToolLoopAgent.generate({ prompt }) 内部流程：

1. 初始 messages = [{ role: 'user', content: prompt }]
2. step = 0
3. while step < maxSteps:
     a. 调 LLM (model + system + messages + tools)
     b. 如果 LLM 没 toolCalls (只输出 text):
          → return { text, steps }
     c. 否则按 stopWhen 检查:
          if (stopWhen({ steps })) return { ... }
     d. 并发调每个 tool.execute
     e. 调用 onStep({ step, totalUsage })
     f. 把 LLM response + tool results 加到 messages
     g. step++
4. 达到 maxSteps 强制结束 (返回最后状态)

structured output with tool calling（v6 stable）

v5 之前 generateObject 和 tools 不能同时用：

  • generateObject 让 LLM 输出符合 schema 的 JSON（用 OpenAI structured outputs / Anthropic tool calling 间接实现）
  • 同时给 tools → 协议冲突

v6 起两者协作：

typescript 
import { generateObject, tool } from 'ai';
import { z } from 'zod';

const result = await generateObject({
  model: openai('gpt-4o'),
  tools: {
    getWeather: tool({
      description: 'Get weather for a city',
      parameters: z.object({ city: z.string() }),
      execute: async ({ city }) => await fetchWeather(city),
    }),
    getForecast: tool({
      description: 'Get 7-day forecast',
      parameters: z.object({ city: z.string() }),
      execute: async ({ city }) => await fetchForecast(city),
    }),
  },
  schema: z.object({
    city: z.string(),
    currentTemp: z.number(),
    nextWeekForecast: z.string(),
    recommendation: z.string(),
  }),
  prompt: 'Tell me about NYC weather and what to expect next week',
  maxSteps: 5,
});

// LLM 自动:
// 1. 调 getWeather('NYC') → temp 72°F
// 2. 调 getForecast('NYC') → "rainy weekend"
// 3. 综合信息返回符合 schema 的 object
console.log(result.object);
// {
//   city: 'NYC',
//   currentTemp: 72,
//   nextWeekForecast: 'Rainy weekend, mild weekdays',
//   recommendation: 'Bring an umbrella for the weekend'
// }

这是 v6 的关键能力：data extraction + tool research 合并。以前要分两步（先 streamText + tools 收集信息 → 再 generateObject 提取结构化），v6 一步到位。

Agent vs streamText + tools 选择

场景 推荐 API
简单单次 LLM 调用（无 tools） generateText / streamText
单次调用 + 1-2 个 tool + 1-2 步 streamText + tools + maxSteps:3
同一 agent 在 3+ 处复用 Agent
多步推理（research / 复杂 orchestration） ToolLoopAgent
tool 调用后返回结构化 object generateObject + tools（v6 stable）
需要 onStep / stopWhen 控制 ToolLoopAgent
durable execution（跨小时几天） v7 canary WorkflowAgent（生产暂不推荐）

Agent 不是 LangChain agent

有人会问：v6 Agent 是不是 LangChain agent / chains 重做？不是。区别：

  • 抽象层级：Vercel Agent 是"配置打包"（model + system + tools 一起复用），不是 LangChain 那种"高层 orchestration"。Agent 没有 chain / memory / retriever 概念
  • 仍是低抽象：你仍能直接 streamText / generateText。Agent 只是把"重复配置"打包，没有引入 LangChain 风格的"过度抽象"
  • 与 v5 streamText 兼容：底层就是 streamText，Agent 只是它的便捷 wrapper

03 应用场景

场景 A · 客服 agent 在多个 endpoint 复用 → Agent

客服 agent 在 chat endpoint / 邮件处理 worker / 后台批量分类 / 测试中都要用同一组 model + system + tools。用 Agent 一次定义 + 多处复用，避免配置散落。

场景 B · 研究 agent 多步推理 → ToolLoopAgent

"研究 X 主题"→ LLM 调 search 找资料 → 调 fetchUrl 读全文 → 多次循环 → 调 summarize 总结。ToolLoopAgent + maxSteps + stopWhen("调了 summarize 就停") + onStep（监控 token 消耗）。

场景 C · 邮件分类 + 提取结构化 → generateObject + tools

客服收到邮件 → LLM 先调 lookupCustomer 拿历史 → 调 classifyIssue 用 ML 模型分类 → 综合信息返回符合 schema 的 object（含 customer / category / priority / suggested actions）。v6 generateObject + tools 一步到位。

场景 D · 简单翻译 → 仍用 streamText 不需要 Agent

纯翻译 / 摘要 / 文本改写 —— 没 tools、没多步、不复用。streamText 即可，不要为了"现代"硬上 Agent abstraction。

场景 E · agent 在 useChat 里跑 → Agent.stream + toUIMessageStreamResponse

前端 chat 调 server agent。Agent 内部 maxSteps + tools 完整 loop；前端 useChat 看到完整 messages（含 tool 调用 / 结果）。2026 主流模式。

场景 F · 多 agent 协作 → 嵌套 agent

主 agent 协调多个专项 agent（researchAgent / writerAgent / reviewerAgent）。每个子 agent 是独立 Agent 实例，主 agent 通过 tool 调子 agent。v6 没有 Claude Agent SDK 那种"subagent"专属概念，但能用嵌套 Agent 实现。详见 P5.1 对比。

04 典型代码：完整 Agent 实战

客服 Agent 完整定义 + 多处使用

typescript 
// agents/support.ts
import { Agent, tool } from 'ai';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

export const supportAgent = new Agent({
  model: openai('gpt-4o'),
  system: `You are a customer support agent for ACME Corp.
  Use tools to look up information before responding.
  Be polite, helpful, and concise.`,
  tools: {
    lookupOrder: tool({
      description: 'Look up an order by ID',
      parameters: z.object({ orderId: z.string() }),
      execute: async ({ orderId }) => {
        const order = await db.order.findUnique({ where: { id: orderId } });
        return order ?? { error: 'Order not found' };
      },
    }),
    lookupCustomer: tool({
      description: 'Look up customer by email',
      parameters: z.object({ email: z.string().email() }),
      execute: async ({ email }) =>
        await db.customer.findUnique({ where: { email } }),
    }),
    sendEmail: tool({
      description: 'Send notification email to customer',
      parameters: z.object({
        to: z.string().email(),
        subject: z.string(),
        body: z.string(),
      }),
      execute: async ({ to, subject, body }) =>
        await mailer.send({ to, subject, body }),
    }),
  },
});

// app/api/chat/route.ts (Route Handler)
import { supportAgent } from '@/agents/support';
import { convertToModelMessages } from 'ai';

export async function POST(req: Request) {
  const { messages } = await req.json();
  const result = supportAgent.stream({
    messages: convertToModelMessages(messages),
  });
  return result.toUIMessageStreamResponse();
}

// app/actions.ts (Server Action)
'use server';
import { supportAgent } from '@/agents/support';

export async function processTicket(ticketBody: string) {
  const result = await supportAgent.generate({ prompt: ticketBody });
  return result.text;
}

// workers/batch.ts (后台批处理)
import { supportAgent } from '@/agents/support';

async function processNewTickets() {
  const tickets = await db.ticket.findMany({ where: { status: 'new' } });
  for (const ticket of tickets) {
    const result = await supportAgent.generate({ prompt: ticket.body });
    await db.ticket.update({
      where: { id: ticket.id },
      data: { autoResponse: result.text, status: 'auto-responded' },
    });
  }
}

// __tests__/support.test.ts (测试)
import { supportAgent } from '@/agents/support';

test('查询订单调 lookupOrder tool', async () => {
  const result = await supportAgent.generate({
    prompt: 'Where is my order 12345?',
  });
  const usedLookupOrder = result.toolCalls.some(
    (t) => t.toolName === 'lookupOrder' && t.input.orderId === '12345'
  );
  expect(usedLookupOrder).toBe(true);
});

研究 Agent + 多步推理

typescript 
import { ToolLoopAgent, tool } from 'ai';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

const researchAgent = new ToolLoopAgent({
  model: openai('gpt-4o'),
  system: `You are a research assistant.
  Use search to find sources, fetchUrl to read full content.
  When you have enough info, call summarize to give the final answer.`,

  tools: {
    search: tool({
      description: 'Search the web',
      parameters: z.object({ query: z.string() }),
      execute: async ({ query }) => await searchAPI(query),
    }),
    fetchUrl: tool({
      description: 'Fetch full content of a URL',
      parameters: z.object({ url: z.string().url() }),
      execute: async ({ url }) => await fetch(url).then((r) => r.text()),
    }),
    summarize: tool({
      description: 'Final summary - call when research is complete',
      parameters: z.object({
        summary: z.string(),
        keyPoints: z.array(z.string()),
        sources: z.array(z.string().url()),
      }),
      execute: async (args) => args,  // 直接返回 (这是"信号" tool)
    }),
  },

  maxSteps: 15,

  stopWhen: ({ steps }) => {
    const lastStep = steps[steps.length - 1];
    return lastStep.toolCalls.some((t) => t.toolName === 'summarize');
  },

  onStep: ({ step, totalUsage }) => {
    console.log(`Step ${step.stepNumber}:`);
    console.log(`  tools called: ${step.toolCalls.map((t) => t.toolName).join(', ')}`);
    console.log(`  total tokens: ${totalUsage.totalTokens}`);
  },
});

// 使用
const result = await researchAgent.generate({
  prompt: 'Research recent breakthroughs in quantum computing in 2024',
});

// 拿最终 summarize tool 的 input（包含 summary / keyPoints / sources）
const summarizeCall = result.toolResults.find(
  (t) => t.toolName === 'summarize'
);
console.log(summarizeCall?.input.summary);
console.log(summarizeCall?.input.keyPoints);

console.log(result.object.issue.priority);
console.log(result.object.suggestedActions);
console.log(result.toolCalls);  // 实际调用的 tools

05 常见陷阱

陷阱 1 · 简单场景过度用 Agent

翻译 / 摘要 / 单次问答没 tools → Agent 是过度抽象。修复：streamText / generateText 即可。Agent 只用于"复用 + 多 tool"。

陷阱 2 · ToolLoopAgent 没设 maxSteps

默认 maxSteps 较低（一般 5）。复杂多步 agent 可能没完成就停。修复：评估实际需要的步数，设 maxSteps（通常 10-20）。同时必须设避免 infinite loop。

陷阱 3 · stopWhen 写错导致永远停不下来

条件没匹配上 → agent 跑到 maxSteps 才停 → 浪费 token。修复：测试 stopWhen 条件；可以打日志（onStep 里）观察实际 tool 调用情况。

陷阱 4 · 期待 generateObject + tools 在 v5 工作

v5 不支持 generateObject + tools 协作。修复：升级 v6（current latest stable）；或在 v5 分两步（streamText + tools 收集 → 再 generateObject 提取）。

陷阱 5 · Agent 跨进程共享

Agent 实例是 module 级单例（一个文件 export 一次实例）。但每个 process 是独立的—— Vercel Edge Function 跨 invocation 不一定保留 module state。修复：Agent 配置不要依赖 process state（避免在 Agent 定义里塞动态数据）；用工厂函数 createAgent() 在每个请求里造新 Agent，如果需要 per-request 配置。

陷阱 6 · onStep 阻塞流

onStep 里做 async 重操作（写 DB / 发外部请求）会阻塞 agent loop。修复：onStep 只做 lightweight observability（log / metrics）；重操作 fire-and-forget（不 await）或 enqueue 到 queue。

陷阱 7 · 嵌套 Agent 的 token 爆炸

主 Agent 调子 Agent 作为 tool → 子 Agent 内部又有完整 messages。如果嵌套层级深 + 每层 messages 都不精简 → token 用量爆炸。修复：子 Agent 的 system + messages 要精简（只传必要 context）；监控 onStep 里 totalUsage 异常增长。

陷阱 8 · v7 WorkflowAgent 生产用

v7 仍 canary，WorkflowAgent / durable execution 不稳定。修复：生产用 v6 stable Agent / ToolLoopAgent；durable execution 需求暂时用 Inngest / Trigger.dev / Temporal 等专业库。
