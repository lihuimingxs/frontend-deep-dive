MCP 协议在两边的位置 + Vercel 里用 MCP 的姿势

MCP（Model Context Protocol）是 Anthropic 主导的开放标准，2026 已成事实工具接口标准。Cursor / Claude Code / VS Code / Cloudflare AI 等都已实现。这一章把 MCP 协议核心、生态现状（上千个公开 MCP server）、在 Claude Agent SDK（中心地位）和 Vercel AI SDK（experimental）的不同位置讲清。

阅读时长 ~22 分钟 前置 读过 P5.1（Claude Agent SDK） 主要来源 modelcontextprotocol.io（官方规范）+ Anthropic engineering blog + Cursor / VS Code MCP 实现 + Vercel AI SDK MCP 文档（experimental）

01 速览

  1. MCP（Model Context Protocol）：Anthropic 主导的开放标准（2024-11 公开），定义"LLM ↔ 外部工具 / 数据源"的接口协议。不是 Anthropic 专属，是开放标准（类似 LSP / Language Server Protocol 之于代码编辑器）。
  2. MCP 三个核心概念：(a) Tools（agent 能调用的函数）；(b) Resources（数据源 / 文件 / DB 等只读数据）；(c) Prompts（预定义 prompt 模板）。
  3. MCP 架构：MCP Server（暴露 tools / resources / prompts）+ MCP Client（消费）。通信走 JSON-RPC over stdio / HTTP / WebSocket。
  4. 2026 生态：上千个公开 MCP server（GitHub / Slack / Linear / Notion / 各种数据库 / 浏览器自动化 / 文件系统 / etc.）。已是 LLM 应用接外部工具的事实标准。
  5. 跨实现互通：同一个 GitHub MCP server 可被 Cursor / Claude Code / VS Code Continue / Cloudflare AI / Vercel AI SDK / 自建 LLM 应用消费 —— 写一次，到处用。
  6. Claude Agent SDK 里的 MCP：中心地位。SDK 内置 MCP client；声明 MCP server 即可用；in-process MCP server（2026-05 起，自定义工具不需要单独进程）。
  7. Vercel AI SDK 里的 MCP：experimental_createMCPClient（仍 experimental，2026 状态）。把 MCP tools 转成 Vercel AI SDK tool 定义，喂给 streamText / Agent。不是中心地位但能用。
  8. 2026 vs 2027 展望：2026 MCP 已 de facto 标准但 Vercel 集成仍 experimental；2027（推测）Vercel 把 MCP 提升到一等公民。MCP 不会消失，反而扩大。
  9. 2026 强观点：自建 LLM 应用的工具接口优先选 MCP（不要再自定义 tool calling 协议）；选自家工具实现时用 in-process MCP server（轻量 + 部署简单）。

02 底层逻辑

MCP 协议核心

text 
┌─────────────────────────────────────────────────┐
│  LLM Application (Claude Code / Cursor / etc.)   │
│                                                   │
│  ┌────────────────────────────────────────────┐ │
│  │  MCP Client (内置)                          │ │
│  │  - 发现 tools / resources / prompts          │ │
│  │  - 调用 tool / 读 resource / 用 prompt        │ │
│  └────────────────────┬─────────────────────────┘ │
│                       │ JSON-RPC                  │
└───────────────────────┼─────────────────────────┘
                        │
                        │ stdio / HTTP / WebSocket
                        │
┌───────────────────────▼─────────────────────────┐
│  MCP Server (third-party / 自建)                  │
│                                                   │
│  暴露:                                              │
│  - Tools: list_repos, create_pr, comment_pr, ...  │
│  - Resources: file://docs.md, db://users/*        │
│  - Prompts: refactor-prompt, review-prompt, ...   │
└─────────────────────────────────────────────────┘

三个核心概念

Tools（agent 能调用的函数）

类似 OpenAI Function Calling 的 tools，但定义和调用走 MCP 协议。MCP server 启动时通告"我有这些 tools"；client（LLM 应用）拿到列表后让 LLM 选用。

json 
// MCP server 通告的 tool（JSON-RPC tools/list 响应）
{
  "tools": [
    {
      "name": "create_pr",
      "description": "Create a pull request",
      "inputSchema": {
        "type": "object",
        "properties": {
          "owner": { "type": "string" },
          "repo": { "type": "string" },
          "title": { "type": "string" },
          "body": { "type": "string" },
          "head": { "type": "string" },
          "base": { "type": "string" }
        },
        "required": ["owner", "repo", "title", "head", "base"]
      }
    },
    // ...更多 tools
  ]
}

Resources（数据源）

Resources 是只读数据，可被 LLM 读取但不可调用。常见 resources：文件 / DB 表 / API endpoint / 文档。

json 
{
  "resources": [
    {
      "uri": "file:///workspace/README.md",
      "name": "README",
      "mimeType": "text/markdown"
    },
    {
      "uri": "db://users/active",
      "name": "Active users",
      "mimeType": "application/json"
    }
  ]
}

Prompts（预定义 prompt 模板）

Prompts 是可被复用的 prompt 模板。MCP server 提供 "review-pr" / "refactor" / "explain-code" 等模板，client 应用直接用。

json 
{
  "prompts": [
    {
      "name": "review-pr",
      "description": "Review a pull request",
      "arguments": [
        { "name": "pr_url", "type": "string", "required": true }
      ]
    }
  ]
}

JSON-RPC 通信

MCP 用 JSON-RPC 2.0 作为通信协议。三种 transport：

  • stdio：进程间通信。MCP server 是子进程，与 client 通过 stdin/stdout 交换 JSON-RPC 消息。本地最常用。
  • HTTP（含 SSE）：远程 MCP server。client 通过 HTTP POST 调 RPC，response 走 SSE 流式返回
  • WebSocket：双向远程

2026 生态：上千个公开 MCP server

2024-11 MCP 公开后生态快速扩张。2026 年公开的 MCP server 包含但不限于：

  • 开发工具：GitHub MCP（issues / PRs / file system）/ GitLab / Bitbucket / Linear / Jira
  • 通信：Slack / Discord / Teams
  • 文档：Notion / Confluence / Obsidian
  • 数据库：PostgreSQL / MySQL / Redis / SQLite / MongoDB
  • 云服务：AWS / GCP / Cloudflare / Vercel
  • 浏览器自动化：Playwright / Puppeteer 包装
  • 文件系统：本地文件 / S3 / R2
  • 专门工具：Sentry / DataDog / Stripe / Resend

跨实现互通：写一次到处用

关键好处：同一个 MCP server 可被多个 LLM 应用消费。例如 GitHub MCP server：

  • Cursor：在 Cursor 配置里加 GitHub MCP → 用户 chat 时 LLM 可调 GitHub tools
  • Claude Code：claude-code 配置加 → 在 CLI 里用 GitHub tools
  • VS Code Continue / Cline：装 MCP 扩展加 → 编辑器内 LLM agent 用 GitHub tools
  • Cloudflare AI：在 Cloudflare AI Gateway 加 → workers 调 LLM 时支持 GitHub tools
  • Vercel AI SDK：experimental_createMCPClient 加 → 自建 web 应用用 GitHub tools

这是 MCP 的核心价值—— 跨实现互通让生态成网络效应。

Claude Agent SDK 里的 MCP（中心地位）

typescript 
import { ClaudeAgent } from '@anthropic-ai/claude-agent-sdk';

const agent = new ClaudeAgent({
  systemPrompt: 'You are a coding assistant.',

  // External MCP servers（subprocess 跑）
  mcpServers: [
    {
      command: 'github-mcp-server',  // 全局安装的 MCP server CLI
      args: ['--token', process.env.GITHUB_TOKEN!],
    },
    {
      command: 'slack-mcp-server',
      args: ['--workspace', 'mycompany'],
    },
    {
      // HTTP MCP server
      url: 'https://mcp.example.com',
      headers: { Authorization: `Bearer ${apiKey}` },
    },
  ],

  // SDK 自动:
  // 1. 启动 stdio MCP servers (子进程)
  // 2. 通过 JSON-RPC 拿 tools / resources / prompts
  // 3. 把 tools 注册到 agent，LLM 能用
  // 4. tool 调用时自动转 JSON-RPC 调到 MCP server
});

In-process MCP server（2026-05 改进）

以前自定义工具要单独进程跑 MCP server。2026-05 起 Claude SDK 支持in-process MCP server：

typescript 
import { ClaudeAgent, createInProcessMCPServer } from '@anthropic-ai/claude-agent-sdk';

const customServer = createInProcessMCPServer({
  name: 'my-custom-tools',

  tools: [
    {
      name: 'lookupOrder',
      description: 'Look up order by ID',
      inputSchema: {
        type: 'object',
        properties: { orderId: { type: 'string' } },
        required: ['orderId'],
      },
      handler: async ({ orderId }) => await db.order.findUnique({ where: { id: orderId } }),
    },
  ],

  resources: [
    {
      uri: 'config://app/feature-flags',
      handler: async () => await db.featureFlag.findMany(),
    },
  ],

  prompts: [
    {
      name: 'customer-greeting',
      arguments: [{ name: 'name', required: true }],
      handler: ({ name }) => `Hello ${name}, how can I help you today?`,
    },
  ],
});

const agent = new ClaudeAgent({
  systemPrompt: '...',
  mcpServers: [customServer],  // 直接传 in-process server
});

好处：

  • 性能：进程内调用，没 IPC 开销
  • 部署：单一进程 / 容器，不用管理多个 subprocess
  • 开发体验：直接调试，不用 attach 到 subprocess
  • 仍是 MCP 协议：内部实现差异，对外接口仍是 MCP（如果以后想改成外部 server，移走代码即可）

Vercel AI SDK 里的 MCP（experimental）

typescript 
import { experimental_createMCPClient } from 'ai';
import { streamText } from 'ai';

// 启动 MCP client，连到 MCP server
const mcpClient = await experimental_createMCPClient({
  transport: {
    type: 'stdio',
    command: 'github-mcp-server',
    args: ['--token', process.env.GITHUB_TOKEN!],
  },
});

// 拿 MCP server 的 tools，转成 Vercel AI SDK tool 定义
const tools = await mcpClient.tools();

// 用在 streamText / Agent 里
const result = await streamText({
  model: openai('gpt-4o'),
  tools,
  prompt: 'List my GitHub PRs',
});

// 用完关掉
await mcpClient.close();

2026 状态：

  • 仍是 experimental_* 命名（API 可能改）
  • 支持 stdio / HTTP transports
  • 不支持 resources / prompts（仅 tools） —— 可能在后续版本加
  • 没有 in-process MCP server 概念（与 Claude SDK 区别）

2027 推测：Vercel 把 MCP 提升到一等公民（不再 experimental），可能也支持 in-process 模式。

选型：什么时候用 MCP vs 自定义 tool

场景 选择
第三方服务（GitHub / Slack / Linear / 等） 用现成 MCP server（节省开发）
自家业务工具（lookupOrder / sendEmail / 等） 用 in-process MCP server（Claude SDK）或直接 tool 定义（Vercel）
跨多个 LLM 应用复用同一组 tools 必须用 MCP（standalone MCP server）
仅一个 LLM 应用用 + Vercel 生态 直接 tool 定义即可，不必 MCP
仅一个 LLM 应用用 + Claude SDK 生态 in-process MCP server（与未来扩展兼容）

03 应用场景

场景 A · Claude Code 配 GitHub MCP

用户在 Claude Code CLI 里 chat："列出我所有 open PR"→ Claude Code 调 GitHub MCP server 的 list_prs tool → 返回 PR 列表。用户零代码，仅需要在配置里加 GitHub MCP。

场景 B · Cursor 配 Linear / Slack MCP

开发者在 Cursor 里："把这个 bug 创建到 Linear" → Cursor 调 Linear MCP create_issue tool。"通知 #engineering channel" → Slack MCP send_message。编辑器变成多服务 hub。

场景 C · 自建 web app 接 GitHub MCP

用 Vercel AI SDK + experimental_createMCPClient + GitHub MCP server。前端 chat 提供"GitHub 助手"，用户聊天调 GitHub tools。不需要自己实现 GitHub API 包装。

场景 D · 自家业务工具发布为 MCP server 共享

公司内部"订单查询 / 客户管理"工具 → 写成 standalone MCP server → 部署到内网 → 多个 AI 应用（客服 chat / 分析 dashboard / Slack bot）共用。"内部 MCP server library"成为 2026-2027 公司架构新模式。

场景 E · Cloudflare AI Gateway + MCP

Cloudflare AI Gateway 集成 MCP（2026），Workers 调 LLM 时直接通过 Gateway 用 MCP servers。无需 Worker 内部实现 MCP client。边缘 LLM 应用接 MCP 工具的简化路径。

场景 F · 自建 in-process MCP server 替代散落 tool 定义

项目里散落 20+ tool 定义在不同文件。refactor：合并成一个 in-process MCP server（按业务领域分组），agent 里只声明 mcpServers: [orderServer, userServer, billingServer]。代码组织更清晰 + 未来若想发布为 standalone MCP server 也容易。

04 典型代码：MCP 实战

Claude SDK 接现成 GitHub MCP

typescript 
import { ClaudeAgent, query } from '@anthropic-ai/claude-agent-sdk';

const agent = new ClaudeAgent({
  systemPrompt: 'You are a GitHub assistant.',
  mcpServers: [
    {
      command: 'github-mcp-server',
      args: ['--token', process.env.GITHUB_TOKEN!],
    },
  ],
});

for await (const message of query({
  agent,
  prompt: 'List my open PRs in vercel/ai repo',
})) {
  console.log(message);
}

Vercel AI SDK 接同一个 GitHub MCP

typescript 
import { experimental_createMCPClient as createMCPClient, streamText } from 'ai';
import { openai } from '@ai-sdk/openai';

const mcpClient = await createMCPClient({
  transport: {
    type: 'stdio',
    command: 'github-mcp-server',
    args: ['--token', process.env.GITHUB_TOKEN!],
  },
});

const tools = await mcpClient.tools();

const result = await streamText({
  model: openai('gpt-4o'),
  system: 'You are a GitHub assistant.',
  tools,
  prompt: 'List my open PRs in vercel/ai repo',
});

for await (const chunk of result.textStream) {
  process.stdout.write(chunk);
}

await mcpClient.close();

自建 in-process MCP server（Claude SDK）

typescript 
import { ClaudeAgent, createInProcessMCPServer } from '@anthropic-ai/claude-agent-sdk';

// orderServer.ts
export const orderServer = createInProcessMCPServer({
  name: 'order-management',
  tools: [
    {
      name: 'lookupOrder',
      description: 'Look up an order',
      inputSchema: { type: 'object', properties: { orderId: { type: 'string' } } },
      handler: async ({ orderId }) => await db.order.findUnique({ where: { id: orderId } }),
    },
    {
      name: 'cancelOrder',
      description: 'Cancel an order',
      inputSchema: { /* ... */ },
      handler: async ({ orderId }) => await cancelOrder(orderId),
      requiresApproval: true,  // 敏感操作
    },
  ],
  resources: [
    {
      uri: 'orders://recent',
      handler: async () => await db.order.findMany({ take: 100, orderBy: { createdAt: 'desc' } }),
    },
  ],
});

// agent.ts
const agent = new ClaudeAgent({
  systemPrompt: '...',
  mcpServers: [orderServer],
});

自建 standalone MCP server（其他 LLM 应用也能用）

typescript 
// my-mcp-server.ts (standalone Node 进程)
import { Server } from '@modelcontextprotocol/sdk/server/index.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
import { ListToolsRequestSchema, CallToolRequestSchema } from '@modelcontextprotocol/sdk/types.js';

const server = new Server(
  { name: 'order-management', version: '1.0.0' },
  { capabilities: { tools: {} } }
);

server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: 'lookupOrder',
      description: 'Look up an order',
      inputSchema: {
        type: 'object',
        properties: { orderId: { type: 'string' } },
        required: ['orderId'],
      },
    },
  ],
}));

server.setRequestHandler(CallToolRequestSchema, async (req) => {
  if (req.params.name === 'lookupOrder') {
    const order = await db.order.findUnique({
      where: { id: req.params.arguments.orderId as string },
    });
    return { content: [{ type: 'text', text: JSON.stringify(order) }] };
  }
  throw new Error('Unknown tool');
});

const transport = new StdioServerTransport();
await server.connect(transport);

// 现在这个 server 能被任意 MCP client 使用 (Claude Code / Cursor / VS Code / Vercel AI SDK / 等)

05 常见陷阱

陷阱 1 · 自建 tool 接口而不用 MCP

2026 仍自定义"我的 LLM 调 tool 协议"，导致工具不能被 Cursor / Claude Code 等其他 LLM 应用复用。修复：自家工具发布为 MCP server。即使内部用，未来想给其他 AI 应用接也容易。

陷阱 2 · 期待 Vercel 的 MCP 支持完全等同 Claude SDK

Vercel 的 experimental_createMCPClient 仍 experimental + 仅支持 tools（不支持 resources / prompts）+ 没 in-process MCP server。修复：(a) Vercel 项目接成熟 MCP server（GitHub / Slack 等）OK；(b) 自建 MCP 中心地位的项目 → 选 Claude Agent SDK；(c) 等 2027 Vercel MCP 提升到一等公民。

陷阱 3 · stdio MCP server 在 serverless 跑

stdio MCP server 是 subprocess。Vercel Edge Function / Cloudflare Workers 不允许启动 subprocess。修复：(a) 用 HTTP transport 的 MCP server（远程跑）；(b) 用 in-process MCP server（仅 Claude SDK，但仍要在能跑 subprocess 的环境）；(c) Edge / serverless 项目暂时仅用 HTTP MCP server。

陷阱 4 · 期待 MCP 解决所有 tool calling 问题

MCP 是"工具接口标准"，不是"agent framework"。它不解决 agent loop / state management / observability 等问题。修复：MCP 配合 Claude Agent SDK / Vercel AI SDK 用 —— SDK 管 agent loop，MCP 管 tool 接口。

陷阱 5 · MCP server 接口设计太"低层"

暴露 "execute_sql_query" 这种低层 tool 给 LLM → LLM 容易写错 SQL / 暴露安全问题。修复：MCP tools 应该是业务级抽象（"lookupCustomer" / "createInvoice"），不是数据库底层 SQL / API endpoint 直通。

陷阱 6 · 没监控 MCP server 性能

每次 tool 调用走 JSON-RPC（stdio / HTTP）。如果 MCP server 慢 → 整个 agent 慢。修复：监控 MCP tool 调用延迟（agent 的 onStep / hooks 里）；发现慢的 server 优化或换 in-process。

陷阱 7 · 相信任意 MCP server

第三方 MCP server 可能是恶意的（exfiltrate 你的数据 / 调危险 API）。修复：只用可信源的 MCP server（官方 / 大公司 / 经过 review 的开源）；不要给 MCP server 过多权限（API key 最小化 scope）。

陷阱 8 · 不限制 MCP tool 调用频率

MCP server 调用第三方 API（GitHub / Slack / Linear）会消耗 rate limit。LLM 可能 silent 调几十次。修复：(a) MCP server 内部限流；(b) agent hooks（PreToolUse）限制调用频率；(c) 监控 + alert。
