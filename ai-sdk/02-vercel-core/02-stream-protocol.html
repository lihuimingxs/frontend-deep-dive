Stream Protocol 演化 · v4 NDJSON → v5 SSE + UIMessage / ModelMessage

v5 是 Vercel AI SDK 历史上最大的 breaking change。流式协议从自家 Data Stream Protocol（NDJSON）切到 W3C SSE 标准 + UIMessage / ModelMessage 二分法 + tool-invocations type-specific parts。这一章把三代协议演化、v5 升级最大迁移点（自定义后端必须重写）、UIMessage / ModelMessage 二分的工程价值讲清。

阅读时长 ~22 分钟 前置 读过 P2.1（三层架构 + 四件套）+ 看过 useChat 至少一次 主要来源 vercel.com/blog/ai-sdk-5（v5 设计文章）+ ai-sdk.dev/docs/ai-sdk-ui/stream-protocol + GitHub Issues v5 升级讨论 + HTTP 主题 P6.1（流式协议层根因）

01 速览

  1. Vercel AI SDK 流式协议三代：v4 自家 Data Stream Protocol（NDJSON）→ v5 SSE 标准 → v6 维持 v5 协议小调整。
  2. v4 协议：自家 NDJSON-like 格式，每行 <type>:<json>\n。类型符 0: = text delta / 2: = data / 3: = error / 9: = tool call / a: = tool result / e: = finish reason 等。优点：自定义灵活；缺点：非标准，与 OpenAI / Anthropic 各自的流式协议不互通，浏览器 EventSource API 不能直接用。
  3. v5 协议（2025-08）：切到 W3C SSE 标准（data: ...\n\n）。但SSE 数据 payload 仍是 Vercel 自家的 JSON 结构（含 type / id / 等字段）—— 不是 OpenAI / Anthropic 原生 streaming。优点：标准协议；中间盒兼容性好；与生态其他 SSE 工具互通。
  4. v6 维持 v5 协议，少量字段调整（spec 改进）。从 v5 升 v6 协议层兼容。
  5. UIMessage / ModelMessage 二分法（v5 起最大概念变更）：UIMessage 是前端 state source of truth（含完整消息历史 + metadata + tool 结果 + 各种 attachment part）；ModelMessage 是发给 LLM 的精简表示（OpenAI / Anthropic 各自能理解的格式）。
  6. 关键洞察：以前一份 message 同时承担"前端展示"和"LLM context"两个角色 → metadata（toast / debug / 时间戳）也喂给 LLM 浪费 token + 污染 context。v5 把它们解耦。
  7. tool-invocations type-specific parts（v5 起）：v4 是 { type: 'tool-invocation', toolName: 'getWeather', ... }；v5+ 是 { type: 'tool-getWeather', ... }。好处：TypeScript 能精确推断每个 tool 的输入输出类型 + 渲染时按 tool name 分支更直观。
  8. v5 升级最大迁移点：自定义后端（FastAPI / Python / 自建）必须适配新协议。GitHub Issues 中大量"我的 FastAPI 后端用 v5 useChat 出问题"是这类。Next.js Route Handler / Vercel Edge Function 用户感知小。
  9. 底层 wire 走的是 HTTP 主题 P6.1 讲的 chunked / HTTP/2 DATA / HTTP/3 DATA frame；应用层是 SSE format（v5+）；浏览器侧用 fetch + ReadableStream（不是 EventSource API，因为需要 POST + 自定义 headers + cancel 控制）。详见 HTTP 主题 P6.1。

02 底层逻辑

v4 自家 Data Stream Protocol（NDJSON）

2024 年 Vercel AI SDK v3-v4 时代用自家协议：

text 
# v4 wire 长这样（NDJSON-like）：
0:"Hello"\n
0:" world"\n
2:[{"customMetric": 42}]\n
9:{"toolCallId":"call_1","toolName":"getWeather","args":{"city":"NYC"}}\n
a:{"toolCallId":"call_1","result":{"temp":72}}\n
e:{"finishReason":"stop","usage":{"promptTokens":10,"completionTokens":20}}\n

# 类型符约定：
0: text delta（文字片段）
2: data（应用自定义 data part）
3: error
9: tool call
a: tool result
e: finish reason + usage
1: 等其他

这套协议在 v4 时代简单且高效：

  • 每行一个 JSON 块 → 易解析
  • NDJSON 格式 → 边收边解（与 fetch + ReadableStream 配合好）
  • 类型符 + JSON 组合表达力够（cover text / data / tool / error / finish 等所有事件）

但 v4 协议有几个长期问题：

  • 非标准：浏览器原生 EventSource API 不能直接用
  • 与生态其他 SSE 工具不互通：proxy / monitoring / debugging 工具按 SSE 处理，遇到 NDJSON 不识别
  • 中间盒兼容性：某些老 corp proxy 对自定义 content-type 处理奇怪

v5 切到 SSE 标准

2025-08 v5 把流式协议切到 W3C SSE 标准（详见 HTTP 主题 P6.1）：

text 
# v5+ wire 长这样（SSE）：
data: {"type":"text-delta","textDelta":"Hello"}\n
\n
data: {"type":"text-delta","textDelta":" world"}\n
\n
data: {"type":"tool-call","toolCallId":"call_1","toolName":"getWeather","args":{"city":"NYC"}}\n
\n
data: {"type":"tool-result","toolCallId":"call_1","result":{"temp":72}}\n
\n
data: {"type":"finish","finishReason":"stop","usage":{"promptTokens":10,"completionTokens":20}}\n
\n

# Content-Type: text/event-stream
# 每条事件以 "data: " 开头 + \n\n 结尾
# 数据 payload 仍是 Vercel 自家 JSON 结构（含 type 字段）

关键点：

  • 外层是标准 SSE：浏览器原生工具能识别（DevTools / proxy / debugging 工具）
  • 内层 JSON payload 仍是 Vercel 自家结构：这不是 OpenAI / Anthropic 原生 streaming 格式
  • 仍需要 SDK 解析：不能简单 new EventSource(url) 收完事 —— 需要 useChat 等 hook 解析 Vercel JSON payload

为什么不用纯 EventSource API？

  • EventSource 仅 GET，但 chat 需要 POST（用户消息在 body）
  • EventSource 不能传 custom headers / 自定义 fetch 选项
  • EventSource 自动重连机制对 chat 不合适（重连后 LLM context 错乱）
  • cancel 控制差

所以 SDK 用 fetch + ReadableStream + 自己解析 SSE 格式。详见 HTTP 主题 P6.1。

UIMessage / ModelMessage 二分法

v5 最大的概念变更。先看以前怎么做：

typescript 
// v4 时代
const messages = [
  { role: 'user', content: 'Hello' },
  { role: 'assistant', content: 'Hi! How can I help?' },
  { role: 'user', content: 'Help with order #42' },
  { role: 'assistant', content: '...', toolInvocations: [...] },
];

// 这份 messages 既是前端 state，也直接发给 LLM

问题：前端经常想在 message 上挂额外信息（debug / metadata / 时间戳 / 用户 reaction），但所有这些信息都会被发给 LLM，浪费 token + 污染 context。

v5 解法：

typescript 
// v5+: UIMessage（前端 state）
type UIMessage = {
  id: string;
  role: 'user' | 'assistant' | 'system';
  parts: Array<
    | { type: 'text'; text: string }
    | { type: 'tool-getWeather'; toolCallId: string; state: 'partial-call' | 'call' | 'result'; input?: ...; output?: ... }
    | { type: 'tool-sendEmail'; ... }
    | { type: 'data-customMetric'; data: ... }    // 应用自定义 data part
    | { type: 'source'; source: ... }              // RAG citation
    | { type: 'step-start' | 'step-end' }          // multi-step
    | ...
  >;
  metadata?: {                                      // 完全不发给 LLM
    debug?: string;
    timestamp?: number;
    customField?: any;
  };
};

// v5+: ModelMessage（发给 LLM）
type ModelMessage = {
  role: 'user' | 'assistant' | 'system' | 'tool';
  content: string | Array<{ type: 'text'; text: string } | { type: 'tool-call'; ... }>;
};

// 转换边界
import { convertToModelMessages } from 'ai';

const modelMessages = convertToModelMessages(uiMessages);
// SDK 自动：
// - 提取 parts 里的 text → ModelMessage content
// - 提取 tool-{name} parts → tool calls + tool results
// - 丢弃 metadata（不发给 LLM）
// - 丢弃 data-* parts（应用自定义，不发给 LLM）

const result = await streamText({ model, messages: modelMessages });

这套设计的工程价值：

  • 前端可以塞任意 metadata，不污染 LLM context
  • LLM 只看到精简的 ModelMessage，token 用得高效
  • 数据流向清晰：UI 端是 UIMessage 完整状态；server 端调 LLM 时转 ModelMessage
  • tool 输入输出有强类型（tool-{name} type-specific parts）

tool-invocations type-specific parts

v4 → v5 另一个重要变化：

typescript 
// v4 - 通用 tool-invocation part
{
  type: 'tool-invocation',
  toolInvocation: {
    state: 'result',
    toolCallId: 'call_1',
    toolName: 'getWeather',
    args: { city: 'NYC' },          // 任意类型
    result: { temp: 72 },            // 任意类型
  }
}

// v5+ - type-specific part
{
  type: 'tool-getWeather',           // type 里编码 tool name
  toolCallId: 'call_1',
  state: 'result',
  input: { city: 'NYC' },             // 类型自动推导（基于 tool 定义）
  output: { temp: 72 },
}

好处：

  • TypeScript 类型推导精确：part.input 自动是 getWeather 的 input 类型
  • 渲染时按 tool name 分支更直观：case 'tool-getWeather': return <WeatherCard />
  • 不同 tool 的 part 不会混淆

v5 升级最大迁移点：自定义后端

Next.js Route Handler / Vercel Edge Function / Hono / Express + ai SDK 的项目升 v5 几乎无感（SDK 内部已切协议）。

但自定义后端（特别是非 Node 后端）需要适配新协议：

  • FastAPI / Python：要写 SSE 响应（text/event-stream + data: ...\n\n 格式）+ 按 Vercel JSON 结构发 events
  • Go / Rust / Java 后端：同样
  • 需要按新协议发 type: 'text-delta' / type: 'tool-call' 等事件

GitHub Issues 中大量讨论：FastAPI 用户用 v5 useChat 一开始无法收到流，需要按新协议重写后端。

03 应用场景

场景 A · Next.js 项目升级 v4 → v5

Next.js Route Handler 用 streamText + 返回 result.toUIMessageStreamResponse() → SDK 自动新协议。前端 useChat 升级到 v5 → 自动使用新协议。感知极小。但要注意 messages 里如果用了 v4 的 toolInvocations 字段，要改成 v5 的 parts 数组。

场景 B · FastAPI / Python 后端 + 前端 useChat 升级

这是 v5 升级最大的坑。Python 后端必须按新 SSE 协议发响应。Vercel 提供 Python adapter（vercel-ai-sdk-python-rsc / 类似包），但 2026 年仍在演进。建议：用 Python adapter；或自己手写 SSE 响应（按 Vercel JSON 格式）；或考虑迁移到 Node 后端。

场景 C · 前端要塞 metadata 不发给 LLM

典型需求：在 message 上挂"用户 reaction"（赞 / 踩）/ "调试信息" / "前端时间戳" 等。v5 做法：放 UIMessage.metadata。前端 state 完整保留，convertToModelMessages 自动丢弃。

场景 D · 多 tool 的 chat UI

v5 type-specific parts 让多 tool 渲染清晰：case 'tool-getWeather': <WeatherCard input={part.input} output={part.output} /> / case 'tool-sendEmail': <EmailStatus ... />。每个 tool 一个组件，类型安全。

场景 E · 自定义 data part 推 progress / 调试信息

后端在流式响应里推自定义 data：writer.write({ type: 'data-progress', data: { step: 2, total: 5 } })。前端收到 part type 是 'data-progress'，渲染进度条。data part 不发给 LLM。

04 典型代码：v5 协议实战

Next.js Route Handler v5 标准做法

typescript 
// app/api/chat/route.ts
import { streamText, convertToModelMessages, type UIMessage } from 'ai';
import { openai } from '@ai-sdk/openai';

export async function POST(req: Request) {
  const { messages }: { messages: UIMessage[] } = await req.json();

  const result = streamText({
    model: openai('gpt-4o'),
    // 关键：UIMessage[] → ModelMessage[]
    messages: convertToModelMessages(messages),
    system: 'You are helpful.',
  });

  // 关键：toUIMessageStreamResponse() 自动用 v5 SSE 协议
  return result.toUIMessageStreamResponse();
}

FastAPI 后端适配 v5 SSE 协议（手写）

python 
# Python 后端手写 v5 SSE 协议
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import json
import openai

app = FastAPI()

@app.post('/api/chat')
async def chat(req: dict):
    messages = req['messages']
    # 把 UIMessage[] 转成 OpenAI 能理解的 messages
    model_messages = [
        {'role': m['role'], 'content': extract_text_from_parts(m['parts'])}
        for m in messages
    ]

    async def generate():
        stream = openai.ChatCompletion.create(
            model='gpt-4o',
            messages=model_messages,
            stream=True,
        )
        for chunk in stream:
            delta = chunk.choices[0].delta.content
            if delta:
                # 按 v5 SSE 协议发：data: {JSON}\n\n
                event = json.dumps({'type': 'text-delta', 'textDelta': delta})
                yield f'data: {event}\n\n'

        # 结束事件
        finish = json.dumps({'type': 'finish', 'finishReason': 'stop'})
        yield f'data: {finish}\n\n'

    return StreamingResponse(generate(), media_type='text/event-stream')


def extract_text_from_parts(parts):
    return ''.join(p['text'] for p in parts if p['type'] == 'text')

前端处理 type-specific tool parts

tsx 
import { useChat } from 'ai/react';

function Chat() {
  const { messages } = useChat({ api: '/api/chat' });

  return (
    <div>
      {messages.map((message) => (
        <div key={message.id}>
          <strong>{message.role}</strong>:
          {message.parts.map((part, i) => {
            switch (part.type) {
              case 'text':
                return <span key={i}>{part.text}</span>;

              case 'tool-getWeather':
                // TypeScript 自动推导 part.input / part.output 类型
                return (
                  <WeatherCard
                    key={i}
                    state={part.state}
                    city={part.input?.city}
                    weather={part.output}
                  />
                );

              case 'tool-sendEmail':
                return <EmailStatus key={i} state={part.state} {...part} />;

              case 'data-progress':
                return <ProgressBar key={i} step={part.data.step} total={part.data.total} />;

              case 'source':
                return <Citation key={i} {...part.source} />;

              default:
                return null;
            }
          })}
        </div>
      ))}
    </div>
  );
}

用 fetch + ReadableStream 手写解析 v5 SSE wire

typescript 
async function parseV5SSEStream(url: string, body: any) {
  const res = await fetch(url, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body),
  });

  if (!res.body) throw new Error('no body');
  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  let buf = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buf += decoder.decode(value, { stream: true });

    // 按 SSE 格式解析: data: {JSON}\n\n
    const events = buf.split('\n\n');
    buf = events.pop() ?? '';  // 最后一段可能不完整

    for (const event of events) {
      if (!event.startsWith('data: ')) continue;
      const json = event.slice(6);
      const data = JSON.parse(json);
      // data.type: 'text-delta' | 'tool-call' | 'tool-result' | 'finish' | ...
      console.log(data);
    }
  }
}

05 常见陷阱

陷阱 1 · 没用 convertToModelMessages

v5+ useChat messages 是 UIMessage[]（含 metadata / parts 等）。直接传给 streamText 的 messages 参数会出错。修复：messages: convertToModelMessages(messages)。

陷阱 2 · FastAPI / Python 后端不适配 v5 SSE 协议

升级 v5 后前端 useChat 收不到流。修复：(a) 用官方 Python adapter；(b) 自己手写 SSE 响应（参考本章 04 节代码）；(c) 暂时降级到 v4（不推荐 —— 错过 v5+ 所有改进）。

陷阱 3 · 仍用 v4 useChat 的 toolInvocations 字段

v5 改成 parts 数组里的 type-specific tool 部分。修复：把 message.toolInvocations 改成 message.parts.filter(p => p.type.startsWith('tool-'))；按 type-specific part 渲染。

陷阱 4 · 在 metadata 里塞要发给 LLM 的信息

metadata 字段不发给 LLM。如果你想让 LLM 看到时间戳 / 用户上下文，应该塞进 message text 或新 part type，不是 metadata。修复：明确"发给 LLM 的去 parts；前端展示用的去 metadata"。

陷阱 5 · 期待 v5 SSE 能直接用 EventSource

v5 SSE 仍需要 SDK 解析（payload 是 Vercel JSON），不是浏览器 EventSource API 能直接吃的。同时 EventSource 仅 GET，chat 需要 POST。修复：用 useChat hook（自动处理）；或手写 fetch + ReadableStream 解析。

陷阱 6 · 流式响应被 nginx / proxy buffer

nginx 默认对响应做 buffering → 客户端体感不流式。修复：(a) proxy_buffering off;；(b) 应用响应头 X-Accel-Buffering: no；(c) 详见 HTTP 主题 P6.1 完整方案。

陷阱 7 · 新项目锁定 v4

2026 年新项目应该直接 v5 / v6（current latest stable），不要新启 v4。v4 不会有新 feature；v5+ 协议更标准、UIMessage / ModelMessage 二分让代码更清晰。修复：新项目直接 v6。

陷阱 8 · v5 的 partial tool call 没处理

v5+ tool call inputs 默认流式。tool part 状态机：state: 'partial-call' → 'call' → 'result'。如果前端只处理 'result' 状态，partial-call 阶段什么都不显示。修复：partial-call 状态显示 spinner / "正在准备调用 X"，提升体验。详见 P4.2。
