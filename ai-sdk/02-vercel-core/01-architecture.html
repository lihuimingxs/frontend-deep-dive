三层架构（Core / UI / RSC）+ 四件套 + AI Gateway

Vercel AI SDK 不是单一包，是三层 SDK 家族：Core（与 LLM 交互）/ UI（前端 hooks）/ RSC（服务端 React 渲染）。Core 四件套（generateText / streamText / generateObject / streamObject）是与 LLM 交互的最核心 API。这一章把三层架构、四件套适用场景、provider 抽象、AI Gateway（v6 起）讲清。

阅读时长 ~22 分钟 前置 读过 P1（5 年史 + 设计哲学） 主要来源 ai-sdk.dev/docs/ai-sdk-core + ai-sdk.dev/docs/ai-sdk-ui + vercel.com/blog/ai-sdk-6（AI Gateway 设计文章）+ Zod 文档

01 速览

  1. Vercel AI SDK 是三层架构：(a) Core（ai 包）—— 与 LLM 交互的核心 API + Agent abstraction；(b) UI（ai/react / ai/vue / ai/svelte / ai/angular）—— 框架专用 hooks；(c) RSC（ai/rsc）—— React Server Components 专用，含 streamUI 等。三层独立可用，按需选。
  2. Core 四件套：generateText / streamText / generateObject / streamObject。这是与 LLM 交互的最核心 API，覆盖"单次 vs 流式"和"文本 vs 结构化"两个维度。
  3. generateText：单次文本生成。返回 { text, usage, finishReason, toolCalls? }。适合后端非流式调用、批量任务、不需要 UI 流式的场景。
  4. streamText：流式文本。返回 { textStream, fullStream, ... }。适合 UI 流式响应。底层走 SSE（v5+）。
  5. generateObject：单次结构化输出（按 Zod schema）。LLM 输出符合 schema 的 JSON。适合"提取结构化数据"场景。
  6. streamObject：流式结构化输出。partialObjectStream 边收边给逐渐完整的 object。适合"流式生成 form / 表格 UI"。
  7. Provider 抽象：@ai-sdk/openai / @ai-sdk/anthropic / @ai-sdk/google / @ai-sdk/mistral / @ai-sdk/amazon-bedrock 等。同一份代码切换 model 即可换 provider。
  8. AI Gateway（v6 起）：@ai-sdk/gateway 包，统一访问 100+ provider model；zero markup pricing（用户只付 provider token 价格）；$5/月免费额度（Vercel 团队赠）；包含 prompt caching / observability / budget controls。极大降低多 provider 切换成本。
  9. 2026 实际项目分布：~70% 项目用 Core + UI（不含 RSC）；~20% 加 RSC（streamUI）；~10% 仅用 Core（后端批处理 / agent）。AI Gateway 在新项目里普及率快速上升（v6 起接近 30%）。

三层独立可用

关键设计：三层不是必须一起用：

  • 仅用 Core：后端批处理、agent orchestration、CLI 工具。不需要前端
  • Core + UI：典型 chat 应用。前端 useChat + 后端 streamText 配对
  • Core + UI + RSC：复杂 generative UI 场景（详见 P3.2）
  • Core + RSC（不含 UI）：纯 RSC 应用，所有交互通过 Server Action

这种设计让 Vercel AI SDK 在不同项目类型都能用 —— 不强迫"全栈用我"。

Core 四件套深入

generateText：单次文本

typescript 
import { generateText } from 'ai';
import { openai } from '@ai-sdk/openai';

const { text, usage, finishReason, toolCalls } = await generateText({
  model: openai('gpt-4o'),
  prompt: 'Translate "hello" to French',
});

console.log(text);
// usage: { promptTokens, completionTokens, totalTokens }
// finishReason: 'stop' | 'length' | 'tool-calls' | 'content-filter' | 'error'

关键：不流式。等 LLM 完整生成完才返回。适合：

  • 后端批处理（一次跑 1000 条数据，每条调一次 LLM）
  • 不展示给用户的内部决策（路由 / 分类 / 摘要供下游用）
  • 测试 / 调试时简单调用

streamText：流式文本

typescript 
import { streamText } from 'ai';
import { openai } from '@ai-sdk/openai';

const result = streamText({
  model: openai('gpt-4o'),
  prompt: 'Tell me a story',
});

// 多种消费方式：

// 1. textStream（仅 text delta）
for await (const chunk of result.textStream) {
  process.stdout.write(chunk);
}

// 2. fullStream（含 tool calls / finish reason / errors）
for await (const event of result.fullStream) {
  switch (event.type) {
    case 'text-delta': process.stdout.write(event.textDelta); break;
    case 'tool-call': console.log('tool:', event.toolName); break;
    case 'finish': console.log('done:', event.finishReason); break;
  }
}

// 3. 转 Response 给 useChat（v5+ SSE 协议）
return result.toUIMessageStreamResponse();

// 4. 等所有完成后拿最终结果
const text = await result.text;
const finishReason = await result.finishReason;

关键：边生成边返回。适合 UI 流式响应（用户看到文字逐字出现）。

generateObject：单次结构化输出

typescript 
import { generateObject } from 'ai';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

const { object, usage } = await generateObject({
  model: openai('gpt-4o'),
  schema: z.object({
    name: z.string(),
    email: z.string().email(),
    intent: z.enum(['support', 'sales', 'feedback']),
    summary: z.string(),
  }),
  prompt: `Extract structured info from this email:\n${emailBody}`,
});

// object 100% 符合 schema (TypeScript 类型推导精确)
console.log(object.name, object.intent);

关键：schema 驱动。SDK 把 Zod schema 转成 LLM 能理解的格式（OpenAI structured outputs / Anthropic tool calling），保证输出严格符合。

2026 状态：

  • OpenAI response_format: { type: 'json_schema' }（structured outputs）
  • Anthropic 通过 tool calling 间接实现
  • Vercel AI SDK 抽象掉差异，统一 API

streamObject：流式结构化

typescript 
const result = streamObject({
  model: openai('gpt-4o'),
  schema: z.object({
    title: z.string(),
    items: z.array(z.object({ name: z.string(), price: z.number() })),
  }),
  prompt: 'Generate an order with 3 items',
});

// 边收边拿"逐渐完整的 object"
for await (const partial of result.partialObjectStream) {
  // 第一次: { title: 'Or...' }
  // 第二次: { title: 'Order #42', items: [{ name: 'App...' }] }
  // 第三次: { title: 'Order #42', items: [{ name: 'Apple', price: 1 }, { name: 'B...' }] }
  // ...
  updateUI(partial);
}

const finalObject = await result.object;

关键：partial 不一定符合 schema（中间状态字符串可能截断、数组可能不完整）。但最终 object 一定符合。

Provider 抽象

Vercel AI SDK 的核心价值之一：同一份代码切换 model：

typescript 
import { streamText } from 'ai';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';
import { google } from '@ai-sdk/google';
import { mistral } from '@ai-sdk/mistral';
import { bedrock } from '@ai-sdk/amazon-bedrock';

// 同一段代码，换 model：
await streamText({ model: openai('gpt-4o'), prompt: '...' });
await streamText({ model: anthropic('claude-sonnet-4-6'), prompt: '...' });
await streamText({ model: google('gemini-2.5-flash'), prompt: '...' });
await streamText({ model: mistral('mistral-large'), prompt: '...' });
await streamText({ model: bedrock('claude-3-haiku'), prompt: '...' });

SDK 在内部抽象掉各 provider 的协议差异：

  • OpenAI 的 messages 格式 vs Anthropic 的 messages 格式
  • OpenAI Function Calling vs Anthropic Tool Use（input_schema vs parameters / tool result 字段名 / 流式协议各不同）
  • provider-specific 选项（temperature / top_p / max_tokens 命名差异）

2026 主流 provider 的 latest model 列表：OpenAI gpt-4o / gpt-5 / o1 / Anthropic Claude Sonnet 4.6 / Opus 4.7 / Haiku 4.5 / Google Gemini 2.5 / Mistral Large / Bedrock 各 model / Azure OpenAI / Cohere / Fireworks / Groq / 等等。

AI Gateway（v6 起）

v6 引入的统一访问层：

typescript 
import { gateway } from '@ai-sdk/gateway';
import { streamText } from 'ai';

// 通过 Gateway 访问任意 provider model（无需各自 API key）
await streamText({
  model: gateway('openai/gpt-4o'),
  prompt: '...',
});

await streamText({
  model: gateway('anthropic/claude-sonnet-4-6'),
  prompt: '...',
});

Gateway 价值：

  • 单一 API key：不用为每个 provider 单独申请
  • Zero markup pricing：付 provider token 原价（不加价）
  • $5/月免费额度：Vercel 团队赠送
  • 统一 observability：所有 LLM 调用走 Gateway，一站监控
  • Budget controls：设月度上限避免账单意外
  • Caching：Gateway 层 cache（相同输入 cache 化）
  • Fallback：一个 provider 挂了自动切另一个
  • Prompt caching：自动判断长 system prompt 时启用 Anthropic prompt caching 等

2026 状态：

  • 不强制 —— 仍可不用 Gateway 直接打 provider
  • 新项目 + 多 provider 切换需求大 → 用 Gateway 划算
  • 已有大量直连各 provider 的项目 → 渐进迁移即可

03 应用场景：四件套 + 三层选型

场景 A · 简单 chat 应用 → Core + UI（不含 RSC）

典型 ChatGPT-like UI：useChat hook（前端） + streamText（后端 Route Handler 或 Server Action）。不需要 RSC streamUI。不需要 Agent abstraction。这是 80% 项目的栈。

场景 B · 后端批处理 / 数据提取 → 仅 Core

批量处理 1000 封邮件提取结构化信息：generateObject + Zod schema。不需要 UI hooks。可以在 cron job / queue worker 里跑。

场景 C · 流式生成结构化 UI → Core + UI

用户输入需求 → LLM 流式生成订单确认 form / 比较表格。后端 streamObject + Zod schema；前端 useObject hook 边收边渲染。比 streamText + 自己 parse JSON 更可靠。

场景 D · LLM 决定渲染什么 React 组件 → Core + RSC

"用户问'我下周怎么穿' → LLM 调 getWeather + 决定渲染 OutfitCard 组件 → server 直接渲染"。streamUI（ai/rsc）+ Server Action。详见 P3.2。

场景 E · 多 provider 切换 → AI Gateway

项目要支持用户选 model（"用 GPT-5 还是 Claude Sonnet 还是 Gemini"）+ 监控所有 LLM 调用 + 控制成本。AI Gateway 是答案。一个 API key + 统一 dashboard + budget。

场景 F · 非 React 框架（Vue / Svelte / Angular）→ Core + UI（对应包）

v5+ Vue / Svelte / Angular feature parity。同一份后端 streamText 代码 + 前端用 ai/vue / ai/svelte / ai/angular 各自的 useChat。这是 v5 的关键改进。

04 典型代码：四件套实战

generateText：批处理邮件分类

typescript 
import { generateText } from 'ai';
import { openai } from '@ai-sdk/openai';

async function classifyEmails(emails: Email[]) {
  const results = await Promise.all(
    emails.map(async (email) => {
      const { text } = await generateText({
        model: openai('gpt-4o-mini'),  // 用便宜 model 跑分类
        prompt: `Classify this email as: support/sales/spam.\n\nEmail: ${email.body}\n\nCategory:`,
      });
      return { id: email.id, category: text.trim() };
    })
  );
  return results;
}

streamText：Next.js Route Handler

typescript 
// app/api/chat/route.ts
import { streamText, convertToModelMessages } from 'ai';
import { openai } from '@ai-sdk/openai';

export async function POST(req: Request) {
  const { messages } = await req.json();

  const result = streamText({
    model: openai('gpt-4o'),
    messages: convertToModelMessages(messages),  // UIMessage → ModelMessage
    system: 'You are a helpful assistant.',
  });

  return result.toUIMessageStreamResponse();
}

generateObject：结构化数据提取

typescript 
import { generateObject } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';
import { z } from 'zod';

const { object } = await generateObject({
  model: anthropic('claude-sonnet-4-6'),
  schema: z.object({
    customer: z.object({
      name: z.string(),
      email: z.string().email(),
      phone: z.string().optional(),
    }),
    intent: z.enum(['quote', 'support', 'complaint']),
    urgency: z.enum(['low', 'medium', 'high']),
    summary: z.string().max(200),
    suggestedActions: z.array(z.string()).max(3),
  }),
  prompt: `Analyze this customer email:\n\n${emailBody}`,
});

// object 完整类型推导：
// object.customer.name: string
// object.intent: 'quote' | 'support' | 'complaint'
// object.suggestedActions: string[]

streamObject：流式生成 form UI

typescript 
// 后端
import { streamObject } from 'ai';
import { z } from 'zod';

export async function POST(req: Request) {
  const { description } = await req.json();
  const result = streamObject({
    model: openai('gpt-4o'),
    schema: z.object({
      title: z.string(),
      fields: z.array(z.object({
        name: z.string(),
        type: z.enum(['text', 'number', 'email', 'select']),
        label: z.string(),
        required: z.boolean(),
      })),
    }),
    prompt: `Generate a form for: ${description}`,
  });
  return result.toTextStreamResponse();
}

import { useObject } from 'ai/react';

function FormGenerator() {

多 provider 切换：用户可选 model

typescript 
import { streamText } from 'ai';
import { gateway } from '@ai-sdk/gateway';

const MODELS = {
  'gpt-4o': 'openai/gpt-4o',
  'claude-sonnet-4-6': 'anthropic/claude-sonnet-4-6',
  'gemini-2.5-flash': 'google/gemini-2.5-flash',
};

export async function POST(req: Request) {
  const { messages, modelKey } = await req.json();
  const modelId = MODELS[modelKey] ?? MODELS['gpt-4o'];

  const result = streamText({
    model: gateway(modelId),
    messages: convertToModelMessages(messages),
  });
  return result.toUIMessageStreamResponse();
}

05 常见陷阱

陷阱 1 · 在 UI 流式场景用 generateText

用户等几秒后一次性看到全部输出 → 体验差。修复：UI 流式场景用 streamText / streamObject + useChat / useObject。generateText 仅用于后端非交互场景。

陷阱 2 · streamText 在 Route Handler 里 await result.text

这等于把流式变成非流式（等所有完成后才返回）。修复：直接 return result.toUIMessageStreamResponse()，让 SDK 把 stream 转 SSE 响应。

陷阱 3 · generateObject 用 z.any() / z.unknown()

schema 没具体定义 → LLM 不知道传什么 → 输出可能任何东西。修复：精确 schema + 每个字段加 .describe('...') 让 LLM 理解语义。

陷阱 4 · streamObject 在 partial 阶段 schema 验证

partial object 在中间状态不一定符合 schema（字符串可能截断 / 数组可能不完整）。如果你 schema.parse(partial) 会抛错。修复：用 .partial() 校验或仅在最终 object 校验。

陷阱 5 · 没用 convertToModelMessages

useChat 端 messages 是 UIMessage[]（含 metadata / tool result parts 等）。直接传给 streamText 的 messages 参数会出错（streamText 期望 ModelMessage[]）。修复：v5+ 必须 convertToModelMessages(messages)。详见 P2.2。

陷阱 6 · 想用 RSC streamUI 但放在 Client Component

streamUI 必须在 Server Action 或 Route Handler 调用 —— 它依赖 RSC server-side 渲染能力。在 client 调会报错。修复：把 streamUI 调用移到 server 文件 + client 组件通过 server action 触发。详见 P3.2。

陷阱 7 · 多个 provider 各自 SDK 而不用 Gateway

项目同时支持 4 个 provider → 4 个 API key + 4 套限流 + 4 个监控 dashboard → 运维负担。修复：用 AI Gateway 统一。1 个 API key + 1 个 dashboard + 统一 budget + caching + fallback。除非你有 Gateway 不支持的 provider 或合规原因不能走 Gateway。

陷阱 8 · 期待四件套之间能切换

generateText / streamText / generateObject / streamObject 不能简单互换 —— 它们的返回类型完全不同。一个项目里这四个 API 通常都会用，对应不同场景。修复：理解四件套是互补，不是"一个能替代另一个"。
