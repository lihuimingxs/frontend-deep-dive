设计哲学：从"包 LLM API"到"agent abstraction"的范式演化

Vercel AI SDK 和 Claude Agent SDK 都做"LLM 集成 SDK"，但底层心智完全不同。Vercel 把它当"前端 / 全栈 web 应用 SDK"；Anthropic 把它当"agent 官方库"。这一章把三阶段范式演化（包 LLM API → agent abstraction → durable execution）+ 两种视角的共同点和分歧讲清，给 P5 跨范式对比提供基础。

阅读时长 ~22 分钟 前置 读过 P1.1（5 年史） 主要来源 vercel.com/blog/ai-sdk-{5,6,a-new-programming-model-for-durable-execution} + Anthropic engineering blog（"How we built Claude Code" / "Building agents with the Claude API"）+ MCP 设计文档

01 速览

  1. LLM 集成 SDK 5 年的核心动力不是"让 SDK 更轻"，是"让用 LLM 这件事别每次都从零开始"。每一代被推出来都是因为前一代撞到了"重复代码天花板"。
  2. 第一阶段 · "包 LLM API"（2023-2024）：抽象掉各 provider 协议差异；提供 streamText / generateText 等单次调用 API；UI 层提供 useChat 等 hooks。关键洞察：开发者要的是"我有 prompt + tools，给我个流式响应"，不是 LLM 的 raw protocol。Vercel AI SDK v3-v5 是这一代代表。LangChain JS 也在这阶段，但抽象层级更高（chains / agents / memory），被认为过度抽象。
  3. 第二阶段 · "agent loop 一等公民"（2025-2026）：把"调 LLM → 看 tool call → 调 tool → 把结果再喂回 LLM → 直到 stop"这个循环抽象成 Agent 类。关键洞察：业务里 99% 的 agent 长得一样（system prompt + tools + loop until done），不应该让每个项目重写一遍。Vercel v6 的 Agent / ToolLoopAgent + Claude Agent SDK 都是这一代。
  4. 第三阶段 · "durable execution"（2026 起，v7 canary）：agent 跑几小时几天怎么办？Vercel WorkflowAgent + Vercel durable execution programming model 在尝试。关键洞察：现实里很多 agent task 是"持续多天"的（深度研究、批处理、多人协作），不能让 process 死了 agent 就丢。本主题正文不锁定 v7，仅 P7 展望。
  5. Vercel 视角："我们是 React/Next.js 生态的 SDK"。优先解决前端 / 全栈 web app 集成 LLM的痛点；useChat / useObject / streamUI 等 UI hooks 一等公民；多 provider 抽象 + AI Gateway；与 RSC / Server Actions 深度集成。
  6. Anthropic 视角："我们是 agent 的官方库"。优先解决构建生产 AI agents的痛点（含 CLI 工具如 Claude Code）；agent loop / subagents / tool permissions 一等公民；与 MCP 深度耦合；hooks 让 agent 行为可观察可拦截可改写；同一个 SDK 驱动 Claude Code 自身（dogfooding）。
  7. 共同点（2026 起的收敛）：都支持 MCP（外部工具接口标准）；都支持 streaming（Vercel SSE / Claude SDK 内部 stream）；都支持 multi-provider（程度不同）；都有 hooks / interception 机制。
  8. 不同点（核心分歧）：UI 层（Vercel 有 / Claude 无）；状态管理（Vercel UIMessage / Claude session）；子任务分解（Claude subagents 一等公民 / Vercel 嵌套 agent 实现）；Provider（Vercel 完全 neutral / Claude 默认 Anthropic）。

02 底层逻辑

三阶段范式演化

第一阶段：包 LLM API（2023-2024）

2023 年 LangChain JS 兴起，2024 年 Vercel AI SDK v3-v4 接班。两者都属于这一阶段，但抽象层级不同：

  • LangChain JS：抽象层级高 —— chain / agent / memory / retriever / output parser 等概念。每个概念都是"对 LLM 应用模式的抽象"。问题：现实模式比抽象多 + 抽象漏洞频繁 → 简单场景写一堆 boilerplate
  • Vercel AI SDK v3-v4：抽象层级低 —— 只有 generateText / streamText / generateObject + provider 抽象 + UI hooks。关键设计选择：不抽象"orchestration"，让用户自己写 loop。这是 LangChain "过度抽象"批评的反面

Vercel AI SDK v3-v4 在这阶段成功的原因：抽象只到"避免重写 boilerplate"那一层，不再多。开发者保留所有控制权（自己写 message 数组 / 自己处理 tool call loop / 自己管 state）。

第二阶段：agent loop 一等公民（2025-2026）

到 2025 年，工程社区注意到一个现象：大量项目都在重写同一个 agent loop：

javascript 
// 大家都在写的 agent loop
async function runAgent(prompt, tools) {
  let messages = [{ role: 'user', content: prompt }];

  while (true) {
    const result = await callLLM({ messages, tools });

    if (result.toolCalls.length === 0) {
      return result.text;  // 没 tool call 了，结束
    }

    // 调每个 tool
    const toolResults = await Promise.all(
      result.toolCalls.map(async (tc) => {
        const tool = tools[tc.name];
        const output = await tool.execute(tc.input);
        return { toolCallId: tc.id, output };
      })
    );

    messages = [
      ...messages,
      { role: 'assistant', toolCalls: result.toolCalls },
      ...toolResults.map((tr) => ({ role: 'tool', ...tr })),
    ];
  }
}

这段代码每个项目都写一遍。2025 年 Anthropic 公开 Claude Agent SDK，把这个 loop 抽成 SDK 内置。2026 年 Vercel AI SDK v6 Agent abstraction 也做了同一件事。

关键洞察："loop until done" 是 agent 的本质特征，应该是 SDK 内置而不是用户自己写。这不是"过度抽象" —— 这是真正可复用的工程抽象。

第三阶段：durable execution（2026 起）

第二阶段的 agent loop 仍假设"agent 跑在一个 process 里直到结束"。但生产现实里：

  • 深度研究 agent 跑几小时几天
  • 批处理 agent 跑几千个 task
  • 用户中途切走，agent 继续后台跑
  • process 重启 / 容器迁移 / 服务器宕机

这些场景需要持久化 agent state + 可恢复执行 + 跨 process 调度。这就是 durable execution。Vercel 2026 年同时发布了 durable execution programming model 和 v7 canary 的 WorkflowAgent。

本主题正文不锁定 v7，但方向是清晰的：未来 1-2 年 agent SDK 的核心创新会在这层。

Vercel 视角：React/Next.js 生态的 SDK

Vercel AI SDK 的设计哲学贯穿整个 SDK：

  • UI hooks 一等公民：useChat / useCompletion / useObject 不是"附加品"，是核心价值。其他 SDK 没有同等地位的 hooks
  • 多 provider neutral：完全不绑定任一 provider。AI Gateway 进一步降低切换成本
  • 与 RSC / Server Actions 深度集成：streamUI（ai/rsc）让 LLM 直接驱动 server-side React 渲染
  • "轻抽象"原则：v6 加 Agent abstraction 但仍保留 streamText 等低层 API。开发者可以选抽象级别
  • vercel.com 自己 dogfooding：v0 / Vercel 平台内部使用

Anthropic 视角：agent 的官方库

Claude Agent SDK 的设计哲学贯穿其全部：

  • 同一个 agent loop 驱动 Claude Code：这是工程级的 dogfooding —— Anthropic 用它建生产 CLI 产品。理论上的 SDK 设计被现实需求反复推敲
  • Agent loop 是核心抽象：subagents（独立 context）/ MCP（外部工具接口）/ hooks（生命周期）/ sessions 全是一等公民
  • 没有 UI 层：假设用户自己做 UI 或不需要 UI（CLI 场景）。这不是缺失，是选择性专注
  • Provider 默认 Anthropic：可经 Bedrock / Vertex 切，但行为优化围绕 Claude API。这是"做深而不做广"
  • MCP 中心地位：外部工具接入完全靠 MCP，没有专属"tool"概念（tool 就是 MCP server 的一种形式）

共同点（2026 起的收敛）

2026 年两边的 SDK 在几个维度收敛：

维度 2024 年 2026 年
Tool 接口 各家自定义 都支持 MCP（标准协议）
流式协议 各家自定义 Vercel 走 SSE / Claude SDK 内部 stream（应用层都走类似 NDJSON-like 格式）
Agent 抽象 无（自己写 loop） 都有（Vercel Agent / Claude SDK Agent loop）
多 provider 各家不同 Vercel 完全 neutral / Claude 经 Bedrock+Vertex 也算多 provider
Hooks / interception 无 都有（Vercel onStep / Claude PreToolUse / PostToolUse 等完整）

这种收敛说明：2026 年大家对"LLM 应用 SDK 长什么样"达成了一定共识。两家走不同路径，但目的地有重叠。

不同点（核心分歧）

但仍有几个根本分歧：

维度 Vercel AI SDK v6 Claude Agent SDK
主战场 React/Next.js web 应用 CLI / 后端 / 任何 agent 场景
UI 层 useChat / useObject / streamUI 一等公民 没有内置 UI 层
Agent 抽象层级 Agent / ToolLoopAgent Agent loop + Subagents（更细粒度）
多 provider 完全 vendor-neutral（含 AI Gateway） 默认 Anthropic（可经 Bedrock / Vertex）
MCP 支持但非中心（experimental_createMCPClient） 中心地位
Hooks 简单（onStep / onFinish 等） 完整 hook 生命周期（PreToolUse / PostToolUse / UserPromptSubmit / Stop / 等）
状态管理 UIMessage（前端 state） Session（SDK 内部）
Dogfooding Vercel.com / v0 内部 Claude Code 自己驱动

03 应用场景：选哪个 SDK

场景 A · 前端 chat / generative UI → Vercel AI SDK

React/Next.js web app + chat UI + 流式响应 + 与 RSC / Server Actions 集成。Vercel AI SDK 没有竞争对手 —— Claude Agent SDK 没 UI 层；LangChain JS 已收缩。2026 这是 Vercel AI SDK 的本垒。

场景 B · CLI agent / 终端工具 → Claude Agent SDK

建 CLI 工具（dev tools / 自动化脚本 / 终端 chat）+ 与 Claude Code 共生态。Vercel AI SDK 在 CLI 场景能用但不够 idiomatic（hooks 浪费）。这是 Claude Agent SDK 的本垒。

场景 C · 后端复杂 agent orchestration → Claude Agent SDK

多 subagent / 长任务 / 严格 tool 权限 / MCP 深度集成。例如代码 review agent 编排 frontend QA + security check + test runner 三个 subagents。Claude Agent SDK 的 subagents 一等公民地位让这种场景天然。

场景 D · 简单后端 agent → 都行

单一 agent + 几个 tools + 中等长度任务。两家都能做。决策：(a) TS + 已用 Vercel 生态 → Vercel AI SDK；(b) Python 或要与 Claude Code 共生态 → Claude Agent SDK；(c) 其他 → 看团队偏好。

场景 E · 前后端混合架构 → 两者一起用

前端 useChat 跑用户 chat → 后端 Server Action / API → 后端 Claude Agent SDK 跑复杂 agent。Vercel 跑前端，Claude 跑后端 agent。两者通信走标准 HTTP / streaming。2026 起 production 架构常见模式。

04 典型代码：两种范式对比

同一个需求，两种实现：客户支持 agent

需求：用户问问题 → agent 调 lookup_order tool 查订单 → 调 send_email tool 通知用户 → 返回总结。

Vercel AI SDK v6 实现

typescript 
import { Agent, tool } from 'ai';
import { openai } from '@ai-sdk/openai';
import { z } from 'zod';

const supportAgent = new Agent({
  model: openai('gpt-4o'),
  system: 'You are a customer support agent.',
  tools: {
    lookupOrder: tool({
      description: 'Look up an order by ID',
      parameters: z.object({ orderId: z.string() }),
      execute: async ({ orderId }) => await db.orders.findUnique({ where: { id: orderId } }),
    }),
    sendEmail: tool({
      description: 'Send notification email to user',
      parameters: z.object({ to: z.string(), subject: z.string(), body: z.string() }),
      execute: async ({ to, subject, body }) => await mailer.send({ to, subject, body }),
    }),
  },
});

// 在 Server Action / Route Handler / Client 都能调
const result = await supportAgent.generate({ prompt: 'Help me with order #42' });
return result.text;

Claude Agent SDK 实现（TypeScript）

typescript 
import { ClaudeAgent } from '@anthropic-ai/claude-agent-sdk';

const agent = new ClaudeAgent({
  systemPrompt: 'You are a customer support agent.',
  // 自定义工具作为 in-process MCP server
  tools: [
    {
      name: 'lookupOrder',
      description: 'Look up an order by ID',
      inputSchema: { type: 'object', properties: { orderId: { type: 'string' } } },
      handler: async ({ orderId }) => await db.orders.findUnique({ where: { id: orderId } }),
    },
    {
      name: 'sendEmail',
      description: 'Send notification email',
      inputSchema: { /* ... */ },
      handler: async ({ to, subject, body }) => await mailer.send({ to, subject, body }),
    },
  ],
  hooks: {
    PreToolUse: async ({ tool, input }) => {
      console.log(`Calling ${tool.name} with`, input);
    },
  },
});

const result = await agent.run({ prompt: 'Help me with order #42' });
return result.summary;

差别：

  • Vercel: Zod schema / tool() helper / Agent class with system + tools
  • Claude: JSON schema / inline tool definitions / hooks 一等公民
  • 都能完成同一需求，但 SDK API 风格不同

都用 MCP（跨范式收敛）

typescript 
// Vercel 里用 MCP（experimental）
import { experimental_createMCPClient } from 'ai';
const mcpClient = await experimental_createMCPClient({
  transport: { type: 'stdio', command: 'github-mcp-server' },
});
const tools = await mcpClient.tools();
await streamText({ model, tools, prompt });

// Claude Agent SDK 里用 MCP（中心地位）
const agent = new ClaudeAgent({
  mcpServers: [{ command: 'github-mcp-server' }],
  // tools 自动从 MCP 来
});

05 常见陷阱

陷阱 1 · 把 Vercel AI SDK 和 Claude Agent SDK 比较"哪个更好"

它们解决不同问题，不在同一维度比较。Vercel 解决"前端 / web app 集成 LLM"；Claude SDK 解决"构建生产 AI agents"。修复：基于场景选，不基于"通用排名"。

陷阱 2 · 误以为 v6 Agent abstraction 是"新框架"

Agent abstraction 是 Vercel AI SDK新增的可选抽象，不是替代 streamText / useChat。简单单次调用仍用 streamText；只有"复用 agent + 多 tool + 多步推理"时才升 Agent。修复：按需选抽象级别。

陷阱 3 · 期待 Claude Agent SDK 提供 useChat-like UI hooks

Claude Agent SDK 没有 UI 层。这是选择性专注，不是缺失。如果你需要前端 chat UI，请用 Vercel AI SDK；或自己手写 React hooks（Claude SDK 不会帮你）。修复：理解 Claude SDK 的定位是"agent 库"不是"前端 SDK"。

陷阱 4 · 过度抽象 / "我要建 agent framework"

2024-2025 LangChain JS 衰落给的教训：过度抽象的成本极高。如果你在思考"我要建一个 agent framework 给团队所有项目用"，先停下来问：(a) 真的有 3+ 项目共用这套抽象的需求吗？(b) Vercel Agent / Claude SDK 已经有的抽象不够用吗？大部分情况下答案是"够用"。

陷阱 5 · 在简单场景上 v6 Agent abstraction

简单一次性 LLM 调用（"翻译这段文本" / "总结这篇文章"）仍用 generateText / streamText 即可。不必所有 LLM 调用都包成 Agent。Agent 是为"多 tool + 复用 + 多步循环"设计的。修复：按需选；简单用低层，复杂用 Agent。

陷阱 6 · 在生产用 v7 canary 的 WorkflowAgent

v7 仍 canary（pre-release），WorkflowAgent / durable execution / loop control 等概念仍在演进，API 可能 breaking changes。修复：生产用 v6 stable；v7 仅用于实验 / 评估方向；durable execution 需求暂时自己实现（Inngest / Trigger.dev 等可帮）。

陷阱 7 · 期待"agent SDK 解决所有 LLM 应用问题"

2026 年 agent SDK 解决了 90% 通用模式，但仍有 10% 自定义场景需要你自己写。不要期待 SDK "全包"。比如：复杂多 agent 协作 + 长记忆 / 多模态融合 / 特定领域 prompt 工程 / RAG 优化 —— 都仍需要自定义。
