JavaScript 项目史 1995-2025

JavaScript 30 年的故事讲成一条主线 —— 不是按版本号流水账，而是按"每次关键转折及其根因"来组织。从 1995 年 Brendan Eich 在 Netscape 用 10 天写出 Mocha，到 1996 浏览器战争分裂、2003-2008 ES4 大爆炸式失败、2008 V8 + Node 爆发改变服务端、2015 ES2015 现代化重启、2020+ Deno / Bun / Edge Runtime 多元运行时——理解这条线之后，许多"JS 为什么这么古怪"的问题就有了根因。本章是语言层视角（"我能用什么"），与 ECMA P1.1 的规范层视角（"哪些进了 ECMA-262 标准"）互补。

阅读时长 ~30 分钟 前置 任意 JavaScript 经验 主要来源 Brendan Eich 演讲 + blog · ECMA-262 §4 · TC39 历史档案

01 速览

三句话定义本章：

  1. JavaScript 是 1995 年 Brendan Eich 在 Netscape 用 10 天写出来的脚本语言。"用 10 天"不是夸张——是 Marc Andreessen 想抢在浏览器战争里推出"嵌入网页的脚本"，给 Eich 的截止日期就是十几天。这个仓促的诞生过程让 JS 留下大量"如果当时多想一周也许不会这样"的设计。
  2. JavaScript 不是 Java。名字里的"Java"是 Netscape 蹭 Sun 营销的产物（Sun 当时正大力推 Java），语言本身和 Java 没有继承关系——它的真正灵感来自 Self（原型）+ Scheme（一等函数）+ Java（语法外观） 三家。这条命名史是 30 年来的混乱之源。
  3. JavaScript 30 年走过四个阶段：1995-2004 标准化与僵局（ES1-3 + ES4 失败）→ 2004-2015 重生（V8 / Node / HTML5 / ES5 救场）→ 2015-2020 年度小步快跑（ES2015 现代化 + 全栈 JS 时代）→ 2020+ 多元运行时（Deno / Bun / Edge / WASM）。每一段都对应一次范式转移。

02 底层逻辑

史前史（1993-1995）：Andreessen 的"业余脚本"想法

1993 年 Marc Andreessen 在 NCSA 写出 Mosaic 浏览器，1994 年带头创立 Netscape Communications，目标是"把互联网带给每一个人"。当时的网页是纯静态 HTML——没有动画、没有交互、连一个能算 1+1 的方式都没有。Andreessen 想要的不是"在浏览器里跑 Java"（那是给专业开发者用的），而是让业余写网页的人——市场营销、设计师、HTML 编辑——能给网页加点小动作的胶水语言。

1995 年初 Netscape 招进了 Brendan Eich，本来想让他做 Scheme 嵌入浏览器（Eich 的本意）。但管理层临时改变想法，要求语言"看起来像 Java"——因为 Sun 1995 年 5 月 23 日 Java 公开发布，业界所有人都在谈 Java，Netscape 想蹭这波热度。Eich 后来回忆："I was hired to do Scheme in the browser. The diktat from upper engineering management was that the language must 'look like Java'."

1995-05：10 天写出 Mocha

1995 年 5 月 6 日到 5 月 15 日左右，Brendan Eich 用 10 天写出第一个原型，代号 Mocha。这是个有原型继承（来自 Self）+ 一等函数（来自 Scheme）+ 类 C/Java 语法（管理层要求）的奇怪混血。10 天能做出来是因为 Eich 复用了已有 Scheme 实现的大量想法——但 10 天也意味着他没时间精心打磨设计：null vs undefined 的双值、global scope 默认污染、type coercion 的意外行为（"5" - 1 === 4 但 "5" + 1 === "51"）等"JS 三大设计错误"都从这里来。

1995 年 9 月 Mocha 改名 LiveScript（Netscape 内部 marketing 决策），随 Netscape Navigator 2.0 beta 发布。1995 年 12 月 4 日，Netscape 与 Sun 签了协议——Sun 同意让 Netscape 用 "JavaScript" 名字。LiveScript 改名 JavaScript。这是个纯营销决策——和 Java 在技术上没继承关系。从这天起，JS 工程师就要一辈子解释"JavaScript 不是 Java，正如猫不是猫头鹰"。

💡 一个被低估的事实

JavaScript 名字里的 "Java" 是 30 年技术沟通混乱的源头。直到今天，仍有人误以为 "JavaScript 是 Java 的脚本版"。Brendan Eich 多次公开为这个名字道歉——他本人想叫 "LiveScript" 或 "ScriptEase"，但管理层决定蹭 Java 营销。这个决策至少误导了几代程序员的语言入门——很多人在面试里会问"JavaScript 和 Java 的关系"才发现没有关系。

第一阶段 · 1995-2004 · 标准化与僵局

这 9 年的故事核心是：浏览器分裂 → 标准化 → ES4 大爆炸式失败 → 标准化僵局 10 年。

1996-08：Microsoft 在 IE 3.0 推出自己的 JS 实现，叫 JScript（避开 Sun 的"JavaScript"商标）。这就是浏览器战争的开始——Netscape 的 JS 和 IE 的 JScript 在很多细节上不一致，开发者要写 if (document.all) 这种 IE 检测代码维护两套兼容。直到今天，document.all 这个 falsy-但-typeof-是-undefined 的怪诞东西仍是 JS 规范里"web 兼容性遗留"的活化石。

1996-11：Netscape 把 JS 提交给 ECMA 国际组织标准化。TC39 工作组成立，由 Microsoft、Netscape、Sun 等公司组成。1997-06：ES1（92 页，"基线版"）正式发布。1998-08 ES2（仅文字校对）。1999-12 ES3（加正则、try/catch、do-while）—— 这之后是整整 10 年的标准化僵局。

2003-2008 · ES4 大爆炸式失败：TC39 起草 ES4，想一次性加入 class、module、namespace、强类型、operator overloading 等。Mozilla 主导设计。但 Microsoft 和 Yahoo 反对——他们认为 ES4 改动太大，会破坏现有 web。2008 年正式放弃 ES4。"不存在 ES4"成为 JS 历史上最痛的注脚。这次失败成为 TC39 流程改革的"创始创伤"——之后的特性必须有 ≥2 个引擎独立实现 + 充分测试覆盖才能进 Stage 4。详细规范层视角见 ECMA P1.1。

第二阶段 · 2004-2015 · JS 重生

2004-09：Firefox 1.0 发布——Mozilla 用 Phoenix → Firebird → Firefox 重新挑战 IE。同年 Gmail 发布，证明"用 JS 真的能做大型 web 应用"。Google Maps 2005 年发布，"AJAX"作为术语被 Jesse James Garrett 命名。2005 年是 web 复兴元年——服务端渲染 + 静态页面的传统 web 开始让位给"富客户端"。

2008-09 是 JavaScript 历史上最关键的一年：

  • Chrome + V8 发布（2008-09-02）—— Lars Bak 主导的 V8 引擎引入隐藏类、内联缓存、JIT，让 JS 性能瞬间 10-100×。在此之前，JS 是"小动画 + 表单验证"的玩具语言；在此之后，JS 能跑大型 web 应用、游戏、甚至服务器。详见 V8 P1.1。
  • Node.js 发布（2009-05，Ryan Dahl JSConf EU）—— 把 V8 嵌入服务端，让 JS 第一次走出浏览器。这开启了"全栈 JS"时代。详见 Node P1.1。

2009-12 ES5 救场之作：在 ES4 失败后，TC39 用一年多时间完成 ES5——加 strict mode、JSON、Object.create、属性描述符。ES5 不激进但务实——它是修补 ES3而非革命。同时为 ES2015 的现代化铺路。2010-2014：jQuery / Backbone / Angular / Ember / React 各家百花齐放，"前端工程化"概念出现，npm 成为模块生态中心。

第三阶段 · 2015-2020 · 现代化与年度小步快跑

2015-06 ES2015（曾叫 ES6）—— JavaScript 历史上最大的一次跃升。一次性加入：class / let / const / arrow functions / destructuring / spread / Promise / module (import/export) / Symbol / Iterator / Generator / Proxy / Reflect / Map / Set / WeakMap / WeakSet / template literals。"现代 JS"从这里开始。

同时 TC39 把发布节奏从"大爆炸式"切换为年度小步快跑 + Stage 流程：每年 6 月发布一个版本，包含已经达到 Stage 4（≥2 引擎实现 + 测试通过）的特性。ES2016 → ES2025 共 10 个年度版本。每年都有新特性但都不再"大重构"——**（2016）/ async/await（2017）/ spread/rest in object（2018）/ optional chaining ?.（2020）/ nullish coalescing ??（2020）/ top-level await（2022）等等。

2017+：TypeScript 上升（详见 TS Part 1.1）/ React Hooks（2019）/ Webpack 主导构建。"全栈 TypeScript"开始替代"全栈 JavaScript"——大型项目几乎都用 TS 写。

第四阶段 · 2020+ · 多元运行时与 Edge

2018-11 Deno 发布：Ryan Dahl 自己批评 Node 后，做了 Deno（V8 + Rust + Web 标准 API + 权限模型）。Deno 没成主流但反向影响了 Node——Node 22+ 加内置 fetch / test runner / Permission Model 都是回应。

2022 Bun 发布：JSC + Zig 实现的 Node 兼容层 + 工具一体化。比 Node 启动快、tools 链更整。Bun 给 Node 带来直接竞争压力——这是"runtime 战国时代"。

2017-2025 Edge Runtime 浮现：Cloudflare Workers（2017）/ Vercel Edge（2020）/ Deno Deploy / AWS Lambda@Edge——把 V8 Isolate 作为单进程多租户的隔离单元，毫秒级冷启动。"JS 运行在哪里"从二选一（浏览器 / Node）变成多选（浏览器 / Node / Deno / Bun / Workers / Lambda / Tauri / Electron / RN / Hermes / Edge）。

2025 当下：JS 仍是 web 第一语言、最大 npm 生态。同时面对 WebAssembly（更高性能选择）和 Rust（高性能 tooling 偏好）的竞争。但 JS 短期内不会被取代——它的护城河是浏览器原生支持 + 全栈生态。

📊 JS 30 年节奏对照

1995-2004 仓促诞生 + 浏览器战争 + ES4 失败 → "JS 长这么古怪是历史遗留"
2004-2015 V8 + Node + HTML5 + ES5 救场 → "JS 真能做大事"
2015-2020 ES2015 现代化 + 年度小步 + 全栈 JS → "JS 长大了"
2020-2025 多元运行时 + Edge + Bun/Deno → "JS 跑在任何地方"
每个 5-10 年一个主旋律。下一个 5 年：可能是 Type Annotations（TS 类型成 JS 标准）+ AI 集成（react-server-context-like 模式）+ WASM 互操作深化。

三条底层逻辑要点

每次重大转折都有具体压力源

ES4 失败不是"标准想做太多"——根因是 Microsoft 已经主导 IE 浏览器市场，没动力实现激进新特性。ES2015 成功不是"提案变好"——是2008 后 Chrome / V8 性能让 JS 能承载更多语言特性，且 Mozilla / Google 在引擎实现上重新具备话语权。Node.js 不是凭空发明——是 V8 性能突破让"用 JS 写服务端"从笑话变成现实。每个看似"语言演化"的事件背后都是引擎能力 / 浏览器市场 / 生态压力三件事的合力。

JS 设计的"原罪"和"运气"并存

原罪：仓促诞生 → null vs undefined / type coercion / global scope / this 四规则 / hoisting / 等。这些今天的开发者天天踩坑——但都改不掉，因为 web 兼容性约束所有 JS 实现必须永远向下兼容。运气：一等函数 + 闭包让 JS 在 ES2015 之前就能模拟 module / class / async；原型链让 V8 能用隐藏类 + IC 优化（详见 V8 P3.1）；JSON 用 JS 字面量语法让数据交换标准化。"原罪"和"运气"是同一些设计决定的两面——10 天写出来不可能完美，但凑合能用就足够触发滚雪球效应。

JS 的护城河是生态而非语言

每隔几年就有人喊"JS 要被取代了"——CoffeeScript / Dart / TypeScript / WebAssembly / Rust...。结果都没真的取代 JS，反倒大多数都编译到 JS或与 JS 互操作。原因不在语言本身——JS 作为语言确实有缺陷——而在生态：npm 几百万个包 + 浏览器原生支持 + 几亿 JS 工程师。任何替代品要么兼容 JS（TypeScript），要么放弃 80% 现成生态（Dart / Elm）。前者活下来，后者萎缩。

JS 史与 ECMAScript 标准史的关系

本章是 语言层视角（"我能用什么 / 我用 JS 写代码遇到了什么"）；ECMA P1.1 是 规范层视角（"哪些进了 ECMA-262 标准 / TC39 怎么决策"）。两者讲的是同一段历史的不同侧面：

  • 语言层：1995 Eich 写出 Mocha → 1996 浏览器战争 → 1999 ES3 → 2008 V8 + Node → 2015 ES2015 现代化 → 2020+ 多元运行时
  • 规范层：1997 ES1 / 1998 ES2 / 1999 ES3 → 2003-2008 ES4 失败 → 2009 ES5 / 2015 ES2015 / 年度发布 → Stage 流程 / TC39 流程改革

建议读完本章后立刻对照 ECMA P1.1 读——同一时间线上规范怎么编辑，能让你完全把握 JS 30 年的演化脉络。

03 应用场景

场景 A · 给团队讲"为什么 JS 这么古怪"

新人入职、转岗、面试反馈"JS 太怪"时，不要直接讲怪点列表（typeof null === 'object' / NaN !== NaN / 0.1 + 0.2 !== 0.3 等）。先给他们 30 分钟讲 1995-2004 这段历史——"10 天写出来 + 浏览器战争 + 不能改 web 兼容性"——之后再看怪点列表，每个怪点都有具体的历史根因。这种背景比直接背规则记得久。

场景 B · 老项目升级 ES 版本

一个老 Express 服务从 ES5 升到 ES2020：理解 ES5 → ES2015 → ES2017 → ES2020 各自加的特性能让你分阶段升级而不是一次性切。例：先把 callback 改成 Promise（ES2015）→ 再改 async/await（ES2017）→ 再加可选链 ?.（ES2020）。每步都有 polyfill / babel-preset-env 兜底。

场景 C · 评估"换语言"诱惑

团队讨论"要不要把 JS 项目重写成 Rust / Go / TS"。理解 JS 30 年史的人会问几个根本问题：(1) 你真正的痛点是语言还是工具链？大多是后者，换 Bun / Vite 解决；(2) 失去 npm 生态的成本能承担吗？(3) 团队学习曲线值得吗？大多数情况下答案是"不"——重写到 TS 是务实选择，重写到 Rust / Go 通常不值得。

场景 D · 解释"现代 JS"边界

"现代 JS"通常指 ES2015+ 的 JS。当面试或博客里提到时，明确这意味着：class / let-const / arrow / Promise / module / async-await（ES2017）等都默认可用。"老 JS"或"ES5 JS"则需要 var / function / IIFE / callback 这套写法。这条分界线很重要——影响打包器目标版本、polyfill 选择、面试期望。

05 常见陷阱

"JavaScript 是 Java 的脚本版"

命名混乱的根源。两者无继承关系——JS 灵感来自 Self（原型）/ Scheme（一等函数）/ Java（语法外观）三家。

"ES4 是失败的版本号"——错

不存在 ES4。版本编辑号空缺；2003-2008 的 ES4 提案被官方废弃。直接从 ES3（1999）跳到 ES5（2009）。

"ES2015 ≠ ES6"——其实是同一个东西

2015 年 TC39 改革后从"序号命名"切换为"年份命名"。第 6 版 = 2015 年版 = ES2015 = ES6。

把 "JavaScript" 和 "ECMAScript" 当同义词

JavaScript 是 Netscape 商标 + 浏览器实现 + DOM/BOM 等宿主 API；ECMAScript 是核心语言规范。同一时间线两个视角。

看老博客里讲的"JS 引擎"——多半是过时资料

2008 之前的 JS 引擎（SpiderMonkey 1.x / JScript / KJS）和今天的 V8 / SpiderMonkey / JSC / Hermes 完全不同。性能优化技巧大多过时。
