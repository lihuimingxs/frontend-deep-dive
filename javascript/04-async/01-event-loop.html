Event Loop 模型

"JS 单线程但不阻塞"的根基是 Event Loop。它不是 ECMAScript 规范内容 —— ECMAScript 只定义了 Jobs 队列（微任务）；真正的宿主环境循环（task queue / rendering / raf）由 HTML 规范（浏览器）和 libuv（Node.js）定义。本章聚焦浏览器的 HTML Event Loop，Node.js 的差异留到 P3-08。

阅读时长 ~20 分钟 前置 P1-13 迭代器 · P2-02 闭包 对应 javascript.info Event loop: microtasks and macrotasks

01 速览

最小心智模型：主线程 = 一个无限循环 { 执行栈 → 清空 microtask → 渲染（如果有） → 取下一个宏任务 }。

javascript 
// 简化的浏览器 event loop 伪代码
while (true) {
  // 1. 取一个宏任务（oldest task）
  const task = taskQueues.mostUrgent().shift();
  execute(task);                         // 跑到同步代码结束

  // 2. 清空所有 microtask（包括 microtask 过程中新增的）
  while (microtaskQueue.length) {
    execute(microtaskQueue.shift());
  }

  // 3. 渲染（如果需要）
  if (shouldRender()) {
    runAnimationFrameCallbacks();        // rAF 回调
    runResizeObservers();
    runIntersectionObservers();
    updateRenderingPipeline();           // style/layout/paint/composite
  }

  // 4. 如果 task queue 空，等待（browser 挂起）
}

三种队列同时存在

队列 来源 何时清空
Task queue（宏任务） setTimeout / setInterval / I/O / DOM event / postMessage 每轮循环取一个执行
Microtask queue Promise then/catch/finally、queueMicrotask、MutationObserver、ES2022 Finalization 宏任务结束后完全清空（包括过程中新增的）
Animation frame callbacks requestAnimationFrame 渲染阶段调用

02 底层逻辑

执行栈（Call Stack）

JS 引擎用栈来跟踪函数调用。每调用一个函数，压入 "execution context"；函数返回就弹出。栈空 = 一次"运行"结束。

异步函数（setTimeout、fetch）不会让 JS 等待 —— 它们把回调注册到浏览器内部，立即返回。等条件满足（定时器到期、网络返回）时，浏览器把回调入队，等主线程栈空再执行。

javascript 
console.log("A");
setTimeout(() => console.log("B"), 0);   // 浏览器注册，0ms 后入宏任务队列
Promise.resolve().then(() => console.log("C"));  // 入微任务队列
console.log("D");

// 输出顺序：A, D, C, B
// 1. 同步完成 "A" "D"
// 2. 栈空 → 清空 microtask → "C"
// 3. 下一轮宏任务 → "B"

HTML 规范的 Task Source

规范给每种来源分配一个 "task source"，引擎可以给不同 source 不同优先级：

  • DOM manipulation task source —— DOM 操作触发
  • User interaction task source —— 鼠标/键盘/触摸事件（浏览器可优先）
  • Networking task source —— XHR/Fetch 完成
  • History traversal task source —— popstate 等
  • Timer task source —— setTimeout/setInterval
  • Postmessage task source —— postMessage
  • File reading task source —— FileReader
  • Idle task source —— requestIdleCallback

每轮循环，浏览器选一个 task source（一般按 FIFO，但用户交互常被提前），从中取最早的 task 执行。所以 setTimeout 和 click 即使都在队列里，click 可能先跑。

为什么要有 microtask 队列

Promise 的 .then 回调必须异步（规范强制），否则 Promise.resolve().then(fn) 和 fn() 会同步执行 —— 破坏"Promise 总是异步"的保证（Zalgo 问题）。但如果 .then 走普通 setTimeout(fn, 0)，回调会排在其他宏任务后面（包括 UI 事件），可能产生视觉"跳帧"。

Microtask 队列介于"同步"和"下一轮 task"之间：当前同步代码结束后立即清空，不参与 task source 调度。Promise、queueMicrotask、MutationObserver 用它保证"尽早异步但不跨越一轮 event loop"。

Microtask 清空的关键规则

javascript 
// 规则：microtask 清空过程中新增的 microtask，也在本轮清完
Promise.resolve().then(() => {
  console.log(1);
  Promise.resolve().then(() => console.log(2));
  Promise.resolve().then(() => console.log(3));
});
// 输出：1, 2, 3（全部在同一轮 event loop）

// ⚠ 陷阱：无限增加 microtask 会死锁渲染
function spin() {
  Promise.resolve().then(spin);
}
spin();    // 页面永远卡住 —— microtask 队列永远非空，渲染永远轮不到

渲染周期与 requestAnimationFrame

浏览器渲染不是每轮都发生 —— 目标帧率 60fps ≈ 16.67ms/frame。规范建议每帧渲染一次；宏任务执行过快（如 10 个 setTimeout 连续到期），浏览器可以跳过中间轮的渲染。

javascript 
// 渲染时序：
// [task] → [microtasks] → [rAF callbacks] → [style/layout/paint] → [下一轮]
//                              ↑ 这一步每帧最多一次

requestAnimationFrame(t => {
  // 在下次渲染前调用，t 是 DOMHighResTimeStamp（sub-ms 精度）
  animate(t);
});

// rAF 在 microtask 之后、渲染之前 —— 读取 DOM 布局时稳定
// setTimeout(fn, 16) 近似但不精确，会抖动

长任务阻塞主线程

主线程一次 task 执行不能被中断。一个 while 循环跑 500ms，浏览器在这 500ms 内不能处理任何 UI 事件、渲染、网络回调。Chrome 定义 "long task" = 超 50ms 的 task，PerformanceObserver 能观测：

javascript 
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.warn("long task:", entry.duration, "ms", entry.attribution);
  }
}).observe({ entryTypes: ["longtask"] });

// 避免长任务的 3 种手段：
// 1. 切片：chunked execution + setTimeout(next, 0)
// 2. Web Worker：CPU 密集型挪到后台线程
// 3. scheduler.yield() / scheduler.postTask()（新 API，Chrome 129+）

"单线程但不阻塞 IO"的真实含义

JS 主线程只有一条执行栈。但浏览器/Node 内部用多线程处理 IO：

  • 浏览器：网络、定时器、IO、渲染线程独立。JS 主线程是 UI 线程。
  • Node.js：libuv 线程池（默认 4 线程）做 DNS、文件 IO、CPU 密集型。网络 IO 多用 epoll/kqueue 非阻塞 IO，不占线程。

所以"fetch 10 个 URL"的并发真实发生在浏览器内部；JS 只是"注册 10 个回调，等回调被派发回主线程"。

03 应用场景

避免长任务：切片执行

javascript 
// 处理 100k 条数据，不卡 UI
async function processLarge(items, handler, chunkSize = 1000) {
  for (let i = 0; i < items.length; i += chunkSize) {
    const chunk = items.slice(i, i + chunkSize);
    for (const item of chunk) handler(item);
    // 让渲染有机会：等下一轮 task
    await new Promise(r => setTimeout(r, 0));
  }
}

// ES2024+ 的 scheduler API（更智能）
async function processLarge2(items, handler) {
  for (const item of items) {
    handler(item);
    if (performance.now() % 50 < 1) await scheduler.yield();
  }
}

rAF 做 60fps 动画

javascript 
let startTime;
function animate(t) {
  if (!startTime) startTime = t;
  const elapsed = t - startTime;
  const progress = Math.min(elapsed / 1000, 1);
  el.style.transform = `translateX(${progress * 300}px)`;
  if (progress < 1) requestAnimationFrame(animate);
}
requestAnimationFrame(animate);

// 比 setInterval(fn, 16) 好的三点：
// 1. 和显示器刷新同步，无撕裂
// 2. 标签页不可见时暂停（省电）
// 3. 在渲染前，DOM 读写在同一帧稳定

queueMicrotask 延迟到栈顶

javascript 
class EventEmitter {
  emit(event, ...args) {
    // 确保订阅者异步调用，防止抛错影响 emit 调用方
    queueMicrotask(() => {
      this.listeners[event]?.forEach(fn => fn(...args));
    });
  }
}
// 优势：
// - 比 setTimeout(fn, 0) 快得多（同一轮 event loop）
// - 比同步调用安全（订阅者错误不影响 emit）
// ⚠ 缺点：不能在里面放无限递归（死循环）

MutationObserver 等效的异步钩子

MutationObserver 的回调走 microtask queue —— 一批 DOM 变更后，回调在 microtask 阶段统一触发。Vue 3 的 nextTick 就是基于这个机制：

javascript 
// Vue 3 nextTick 的简化版
let resolved = Promise.resolve();
function nextTick(fn) {
  return resolved.then(fn);          // 微任务，在 DOM 更新之后触发
}

state.count++;
await nextTick();
// DOM 已更新
read(domValue);

04 典型代码

时序面试题

javascript 
console.log("script start");

setTimeout(() => console.log("timeout"), 0);

Promise.resolve()
  .then(() => console.log("p1"))
  .then(() => console.log("p2"));

queueMicrotask(() => console.log("microtask"));

console.log("script end");

// 输出：
// script start
// script end
// p1
// microtask       ← queueMicrotask 排在 p1 的第二个 then 之前
// p2
// timeout

嵌套 microtask

javascript 
Promise.resolve().then(() => {
  console.log(1);
  Promise.resolve().then(() => console.log(2));
  // ↑ 新加的 microtask 会在本轮清空前执行
  setTimeout(() => console.log(3), 0);
  // ↑ 下一轮 macrotask
});
Promise.resolve().then(() => console.log(4));

// 输出：1, 4, 2, 3

测量长任务

javascript 
// 开发环境插入 long task 监控
if (typeof PerformanceObserver !== "undefined") {
  try {
    new PerformanceObserver((list) => {
      for (const e of list.getEntries()) {
        if (e.duration > 100) {
          console.warn(`Blocked ${e.duration.toFixed(0)}ms:`, e);
          // 上报到 RUM / Sentry
        }
      }
    }).observe({ entryTypes: ["longtask"] });
  } catch { /* 不支持 */ }
}

rAF 批量写 DOM

javascript 
// ❌ 每次写都可能触发 reflow
for (const item of data) {
  item.el.style.height = `${compute(item)}px`;
}

// ✓ 把写操作集中到下一帧 rAF
requestAnimationFrame(() => {
  for (const item of data) {
    item.el.style.height = `${compute(item)}px`;
  }
});

// 读写分离模式（fastdom 思路）：
// - rAF 前读（measure）
// - rAF 里写（mutate）

05 常见陷阱

setTimeout(fn, 0) 最少也有 4ms

HTML 规范规定嵌套 5 层以上的 setTimeout，浏览器强制 4ms 最小延迟（防止 CPU 占满）。Chrome 还会对 background tab 做 1s 节流。想"尽快异步"用 queueMicrotask 或 MessageChannel（绕过 4ms 限制）。

无限 microtask 卡死页面

Promise.resolve().then(loop) 递归会让 microtask 队列永远非空 → 渲染永远不发生 → 页面冻结。排查时看 DevTools Performance 火焰图，看不到渲染帧就是这个问题。用 setTimeout 插入 macrotask "让行"。

长同步代码冻结 UI

JSON.parse 一个 10MB JSON、sort 10 万条数据、大 regex 匹配都可能几百 ms 阻塞。Chrome Lighthouse 的 TBT（Total Blocking Time）指标就是统计这个。切片或 Worker 处理。

rAF 在后台标签页暂停

后台标签的 requestAnimationFrame 回调会被大幅降频（Chrome 1 次/秒）或完全不跑。做时间驱动的逻辑（游戏 tick、倒计时）要用 performance.now() 差值 + setTimeout 兜底。

规范讲 "task" 不是 "macrotask"

HTML 规范里只有 task（= 很多文章说的"宏任务"）和 microtask。"macrotask" 是社区术语。规范里还有 "job"（ECMAScript 层对 microtask 的叫法）。阅读规范时要理解这些术语映射。

event.stopPropagation() 不阻止 microtask

事件处理器里 event.stopPropagation() 阻止事件冒泡到外层，但已经入队的 microtask（比如之前 Promise.then 注册的）不会被取消。事件处理只管自己的监听器链。

浏览器和 Node Event Loop 不同

Node 的 process.nextTick 优先级比 Promise microtask 更高；libuv 把宏任务分 6 个 phase；setImmediate 和 setTimeout(0) 顺序不确定。细节见 P3-08。
