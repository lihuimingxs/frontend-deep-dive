Node Event Loop + 错误收口

浏览器 Event Loop 是 HTML 规范定义的；Node.js 的底层是 libuv，有 6 个明确的 phase，process.nextTick / setImmediate 等 API 的时序行为和浏览器不同。本章讲 Node 特有的机制 + 把 P1-14 的错误处理在 Node 语境下收口：unhandledRejection 政策、graceful shutdown、AsyncLocalStorage。

阅读时长 ~22 分钟 前置 01 Event Loop · P1-14 错误处理 对应 Node 文档 Event Loop, Timers, and process.nextTick

01 速览

Node libuv 的 6 个 phase

每轮 Node event loop 按固定顺序走 6 个 phase，每个 phase 有自己的回调队列：

# Phase 处理什么
1 timers setTimeout / setInterval 到期
2 pending callbacks 上轮延迟的系统调用回调（TCP errors 等）
3 idle, prepare Node 内部
4 poll 接收新 I/O 事件（大部分 fs/net 回调在这）；阻塞等待
5 check setImmediate 回调
6 close callbacks socket.on("close") 等

每个 phase 结束后，清空 microtask（含 nextTick + Promise）再进入下一 phase。

浏览器 vs Node 的时序差异

场景 浏览器 Node
setTimeout(0) 最少 4ms（嵌套 5+） 最少 1ms
setImmediate 非标准（Node 特有） check phase 回调
process.nextTick 不存在 microtask 的"更高优先级"队列
microtask 清空时机 每个 task 结束后 每个 phase 结束后（更粗粒度）
Promise 和 nextTick 优先级 Promise 唯一 nextTick > Promise

02 底层逻辑

process.nextTick 的特殊性

javascript 
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
Promise.resolve().then(() => console.log("promise"));
process.nextTick(() => console.log("nextTick"));

// 大致输出（timer 和 immediate 相对顺序不稳定）：
// nextTick       ← 最高优先级 microtask
// promise        ← 普通 microtask
// timeout        ← timer phase（可能，取决于启动时机）
// immediate      ← check phase

// nextTick 在当前操作完成后、下一个 phase 前立即执行
// 规则："在 Node 的每个阶段切换点清空 nextTick 队列，再清空其他 microtask"

setTimeout(0) vs setImmediate 的顺序

规范说 setTimeout(0) 会被 clamp 到 1ms —— 但精度不准。在 IO 回调之外调用两者的顺序不确定：

javascript 
// 脚本顶层 —— 顺序不确定
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
// 可能先 timeout 也可能先 immediate

// 在 IO 回调里 —— setImmediate 一定先
fs.readFile("x", () => {
  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));
  // 输出：immediate, timeout
  // 因为此时在 poll phase，下一个就是 check，再下一轮 timers
});

Microtask 在每 phase 间清空

浏览器是"每 task 结束清空 microtask"。Node 更粗：每个 phase 结束后清空一次。所以 Node 里 microtask 之间穿插的是 phase，而不是单个回调。

javascript 
// Node 的典型执行单位
// [一个 phase 跑完所有到期 callback] → [清空 nextTick 队列] → [清空 Promise microtask 队列] → [下一 phase]

// 所以同一 phase 里的多个 timer 回调之间，microtask 会被插入：
setTimeout(() => {
  console.log("timer 1");
  Promise.resolve().then(() => console.log("promise 1"));
}, 0);
setTimeout(() => {
  console.log("timer 2");
  Promise.resolve().then(() => console.log("promise 2"));
}, 0);

// Node 11+ 输出：
// timer 1
// promise 1
// timer 2
// promise 2
// （Node 10 及更早是 timer 1, timer 2, promise 1, promise 2）

unhandledRejection 政策演变

Node 版本 unhandled rejection 默认行为
< 15 只警告，不崩溃
15+ 抛出且崩溃进程（推荐）
任意版本 --unhandled-rejections=warn/throw/none CLI flag 控制

生产代码应当让它崩溃 —— unhandled rejection 几乎一定是代码 bug。靠 PM2 / systemd 重启恢复服务。

AsyncLocalStorage 传递请求上下文

Node 独有的 API（基于 async_hooks），让你跨 async 边界传递上下文（如 requestId、traceId）而不用显式参数传递：

javascript 
import { AsyncLocalStorage } from "node:async_hooks";

const context = new AsyncLocalStorage();

// HTTP server 入口：每个请求 run 一次，建立独立 store
app.use((req, res, next) => {
  context.run({ reqId: req.headers["x-request-id"] ?? randomUUID() }, () => next());
});

// 深层业务代码：拿 reqId 不需要传参
function log(msg) {
  const store = context.getStore();
  console.log(`[${store?.reqId}] ${msg}`);
}

// 跨 await / setTimeout / Promise.then 仍可见
async function handler() {
  await fetch(...);
  log("done");            // 能拿到 reqId
}

Node 特有异步 API 速览

  • fs.promises.* / fs/promises —— Promise 风格 fs API
  • node:timers/promises —— setTimeout(1000, value, { signal }) 返回 Promise，支持取消
  • node:stream / stream/promises —— pipeline、finished 的 Promise 版本
  • worker_threads —— 真正的多线程 CPU 任务
  • child_process —— 进程级并发

03 应用场景

全局错误上报 + graceful shutdown

javascript 
// 启动时一次性注册
process.on("uncaughtException", (err, origin) => {
  logger.fatal({ err, origin }, "uncaught exception");
  // ⚠ 状态已不可信 —— 尽快退出，让 PM 重启
  shutdown(1);
});

process.on("unhandledRejection", (reason) => {
  logger.fatal({ reason }, "unhandled rejection");
  shutdown(1);
});

async function shutdown(code = 0) {
  try {
    await Promise.race([
      Promise.all([
        server.close(),
        db.disconnect(),
        redis.quit(),
      ]),
      new Promise((_, rej) => setTimeout(() => rej(new Error("shutdown timeout")), 10_000)),
    ]);
  } catch (err) {
    logger.error({ err }, "shutdown failed");
  } finally {
    process.exit(code);
  }
}

// SIGTERM：容器 / PM 发送的"优雅退出"信号
process.on("SIGTERM", () => shutdown(0));
process.on("SIGINT", () => shutdown(0));   // Ctrl+C

请求级上下文追踪

javascript 
// 配合 pino logger
import pino from "pino";
const baseLogger = pino();

const als = new AsyncLocalStorage();

function wrappedLogger() {
  const ctx = als.getStore();
  return ctx ? baseLogger.child({ reqId: ctx.reqId, userId: ctx.userId }) : baseLogger;
}

// middleware
app.use(async (req, res, next) => {
  const store = {
    reqId: req.headers["x-request-id"] ?? randomUUID(),
    userId: req.user?.id,
  };
  als.run(store, async () => next());
});

// 业务里任何位置
wrappedLogger().info({ event: "login" });   // 自动带 reqId/userId

Node 原生的 timers/promises + AbortSignal

javascript 
import { setTimeout, setImmediate } from "node:timers/promises";

// 可取消的 sleep
async function poll(fn, { intervalMs = 1000, signal } = {}) {
  while (true) {
    if (await fn()) return;
    await setTimeout(intervalMs, undefined, { signal });
  }
}

// 让一次行
await setImmediate();  // 相当于 await new Promise(setImmediate)

CPU 密集任务转 Worker

javascript 
// main.mjs
import { Worker } from "node:worker_threads";
function runHeavy(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker("./heavy.mjs", { workerData: data });
    worker.on("message", resolve);
    worker.on("error", reject);
    worker.on("exit", (code) => { if (code !== 0) reject(new Error(`Worker exit ${code}`)); });
  });
}

// heavy.mjs
import { parentPort, workerData } from "node:worker_threads";
const result = cpuIntensiveCompute(workerData);
parentPort.postMessage(result);

04 典型代码

错误分类与恢复策略

javascript 
class AppError extends Error {
  constructor(message, { cause, statusCode = 500, code, recoverable = false } = {}) {
    super(message, { cause });
    this.name = this.constructor.name;
    this.statusCode = statusCode;
    this.code = code;
    this.recoverable = recoverable;
  }
}

class ValidationError extends AppError {
  constructor(msg, fields) { super(msg, { statusCode: 400, recoverable: true }); this.fields = fields; }
}
class NotFoundError extends AppError {
  constructor(entity, id) { super(`${entity} ${id} not found`, { statusCode: 404, recoverable: true }); }
}
class AuthError extends AppError {
  constructor(msg) { super(msg, { statusCode: 401, recoverable: true }); }
}

// Express / fastify error handler
app.use((err, req, res, next) => {
  const appErr = err instanceof AppError ? err : new AppError("internal error", { cause: err });
  if (!appErr.recoverable) logger.error({ err: appErr }, "non-recoverable");
  res.status(appErr.statusCode).json({
    error: appErr.message,
    code: appErr.code,
    ...(appErr instanceof ValidationError && { fields: appErr.fields }),
  });
});

生产级 graceful shutdown

javascript 
class Shutdown {
  #handlers = [];
  #shuttingDown = false;
  register(name, fn) { this.#handlers.push({ name, fn }); return this; }
  async run(timeout = 30_000) {
    if (this.#shuttingDown) return;
    this.#shuttingDown = true;
    logger.info("shutdown started");
    for (const { name, fn } of this.#handlers.reverse()) {
      try {
        await Promise.race([
          fn(),
          new Promise((_, rej) => setTimeout(() => rej(new Error("timeout")), timeout / this.#handlers.length)),
        ]);
        logger.info(`✓ ${name}`);
      } catch (err) {
        logger.error({ err, name }, "handler failed");
      }
    }
    logger.info("shutdown complete");
  }
}

const shutdown = new Shutdown()
  .register("http", () => server.close())
  .register("db", () => db.pool.end())
  .register("redis", () => redis.quit())
  .register("queue", () => queue.drain());

for (const sig of ["SIGTERM", "SIGINT"]) {
  process.on(sig, () => shutdown.run().then(() => process.exit(0)));
}

Node 时序观察

javascript 
// 在 Node 里运行以下代码：
console.log("sync 1");
process.nextTick(() => console.log("nextTick 1"));
Promise.resolve().then(() => console.log("promise 1"));
setImmediate(() => console.log("immediate 1"));
setTimeout(() => console.log("timeout 1"), 0);
process.nextTick(() => console.log("nextTick 2"));
Promise.resolve().then(() => console.log("promise 2"));
console.log("sync 2");

/* 典型输出：
sync 1
sync 2
nextTick 1         ← 所有 nextTick 先清
nextTick 2
promise 1          ← 然后所有 Promise microtask
promise 2
timeout 1          ← timers phase（或 immediate 先，取决于启动）
immediate 1        ← check phase
*/

AsyncLocalStorage 高阶用法

javascript 
// 嵌套 scope
als.run({ a: 1 }, () => {
  als.getStore();          // { a: 1 }
  als.run({ b: 2 }, () => {
    als.getStore();        // { b: 2 } —— 不自动合并
  });
});

// 添加字段到当前 store（需要 mutate）
function addToContext(key, value) {
  const store = als.getStore();
  if (store) store[key] = value;
  else throw new Error("not in async context");
}

// ⚠ AsyncLocalStorage 在 worker_threads 间不共享
// 每个 thread 有自己的 ALS 实例

05 常见陷阱

process.nextTick 饿死 IO

nextTick 队列必须清空后才能进下一 phase。在 nextTick 回调里再加 nextTick 会无限循环 → timers/IO 永远不跑。Node 文档明确警告。用 queueMicrotask 或 setImmediate 代替。

uncaughtException 后不应继续运行

Node 文档警告：进入此处理器时程序状态已不确定（部分对象已损坏）。应该记日志后 process.exit，让 PM 重启。保留运行只适合"写崩溃日志到磁盘"这种最小化善后。

Node 15+ 的 unhandledRejection = crash

从 Node 15 开始默认 --unhandled-rejections=throw。升级时遇到"突然进程退出"的情况很可能是隐藏的 floating Promise 暴露。用 ESLint 的 no-floating-promises + 全局 handler 记录 + 在生产做监控告警。

AsyncLocalStorage 有性能开销

为了追踪异步上下文 Node 要拦截所有 async 原语。Node 17+ 做了大量优化，但仍比直接参数传递慢。性能关键路径可以用回调+显式 ctx 参数替代。日志/追踪场景 ALS 开销可以接受。

Domain API 已弃用

domain 模块自 Node 4 弃用但仍在（某些老框架如 Hapi 某些版本用）。新代码用 AsyncLocalStorage。迁移时注意错误隔离边界不同。

ESM 顶层 await 阻塞启动

Node 22+ 原生 ESM 顶层 await 可以用，但 import 链上任一模块 await 慢就会拖慢整体启动。部署容器健康检查时如果启动 > 30s 会被 K8s 认为失败 → 无限重启。

混用 require 和 import

CJS 和 ESM 的加载模型不同 —— CJS 同步、ESM 异步。require("./esm.mjs") 抛错，import("./cjs.js") 返回 { default: module.exports }。Next.js / tRPC 等工具链默认处理了，手写时要注意。

worker_threads 不共享 memory 也不共享 AsyncLocalStorage

每个 worker 是独立 V8 实例。传递大对象要 transferList（Transferable 对象）或 SharedArrayBuffer。请求级上下文需要在主线程通过 message 手动发过去。
