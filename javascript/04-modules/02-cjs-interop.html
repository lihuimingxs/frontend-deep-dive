CJS 与 ESM 互操作

CommonJS 先于 ESM 统治了 Node 生态十多年。今天的生态是"双模共存"—— npm 上 ESM 包、CJS 包、dual-package 都有。本章讲 CJS 的语义、ESM import CJS 的 default 规则、循环 require 的半成品陷阱、Node 22+ 的 require(esm) 实验支持，以及打包器是怎么做兼容的。

阅读时长 ~20 分钟 前置 01 ESM 对应 Node 文档 CommonJS modules

01 速览

CJS 基本语法（Node）

javascript 
// 导出
module.exports = { foo: 1 };             // 替换整个对象
exports.bar = 2;                          // 在对象上挂属性（等价 module.exports.bar）
module.exports.baz = 3;                   // 同上

// ⚠ exports = { x: 1 } 无效（只改变本地变量，不影响 module.exports）

// 导入
const fs = require("fs");
const { readFile } = require("fs");      // 对象解构
const lodash = require("lodash");

// 环境变量
__dirname;    // 当前文件目录绝对路径
__filename;   // 当前文件绝对路径
require.cache;  // 模块缓存
require.main === module;  // 是否作为主入口运行

ESM vs CJS 核心差异

维度 CJS ESM
加载 运行时同步 静态 parse + async load
绑定 值拷贝 live binding
位置 任意（require 是函数） 仅顶层（或动态 import）
this 顶层 module.exports undefined
全局变量 __dirname, __filename import.meta.*
严格模式 可选 默认严格
循环依赖 返回当下的 exports 对象（可能不完整） live binding（后续会填充）
Top-level await 不支持 支持

02 底层逻辑

CJS 的 "function wrapper"

Node 加载 CJS 文件时，把整个文件包进一个函数里：

javascript 
// 你的 foo.js 源码
module.exports.hi = "hi";

// Node 实际执行的（简化）
(function (exports, require, module, __filename, __dirname) {
  module.exports.hi = "hi";
})(module.exports, require, module, __filename, __dirname);

// 所以：
// - 每个文件是独立作用域（var 不污染全局）
// - 5 个"全局变量"其实是函数参数
// - require 是闭包捕获的函数
// - exports 只是 module.exports 的初始引用

exports 和 module.exports 的陷阱

javascript 
// ✓ 能工作
exports.x = 1;
module.exports.x = 1;
module.exports = { x: 1 };

// ❌ 无效（切断了 exports 和 module.exports 的连接）
exports = { x: 1 };
// 现在 exports 指向新对象，但 module.exports 仍是原对象
// require 拿到的是 module.exports —— 空对象

// 记忆：永远用 module.exports 替换整体，exports 只挂属性

循环 require 的半成品

CJS 同步执行，循环时返回"当前时刻的 exports 对象"—— 可能是空的或不完整的：

javascript 
// a.js
console.log("a start");
exports.x = "ax";
const b = require("./b");
console.log("a got b.y =", b.y);
exports.x2 = "ax2";
console.log("a end");

// b.js
console.log("b start");
exports.y = "by";
const a = require("./a");
console.log("b got a.x =", a.x);      // "ax"（a 已 exports x）
console.log("b got a.x2 =", a.x2);    // undefined（x2 还没 exports）
exports.y2 = "by2";
console.log("b end");

// 执行 require("./a")：
// a start
// b start
// b got a.x = ax
// b got a.x2 = undefined    ← ⚠ 半成品
// b end
// a got b.y = by
// a end

CJS 循环依赖难调试的根源。ESM 的 live binding 避免了这个问题 —— 即使先拿到 undefined，后续读会看到最新值。

Node 双模系统

Node 如何决定一个 .js 文件是 ESM 还是 CJS？

  1. 文件扩展名：.mjs → ESM；.cjs → CJS
  2. 最近的 package.json 的 "type" 字段："module" → ESM；"commonjs" → CJS；缺省 → CJS
  3. 命令行 --input-type=module（用于 --eval / stdin）
javascript 
// package.json
{
  "name": "my-pkg",
  "type": "module",                  // 所有 .js 按 ESM
  "main": "./index.js",              // CJS 入口（传统）
  "exports": {
    ".": {
      "import": "./index.mjs",       // ESM 消费者
      "require": "./index.cjs",      // CJS 消费者
      "default": "./index.mjs"
    }
  }
}

ESM import CJS 的默认化

ESM 里 import pkg from "cjs-pkg"：CJS 没有"default"概念，Node 把 整个 module.exports 视为 default export。命名 import 有限制：

javascript 
// CJS: cjs-pkg
module.exports = { x: 1, y: 2 };

// ESM 消费
import pkg from "cjs-pkg";           // pkg = { x: 1, y: 2 }
import { x } from "cjs-pkg";         // ✓ Node 做静态分析尝试解析命名导出
// ⚠ 但 Node 的"CJS 命名 export 检测"是启发式的
// 打包器（Vite/Next）通常更可靠

// 最保险：整体 default import 再解构
import pkg from "cjs-pkg";
const { x, y } = pkg;

CJS require ESM：Node 22+ 才能

javascript 
// Node 22 之前：require("./esm-module.mjs") 抛 ERR_REQUIRE_ESM
// Node 22+ 有限支持同步 require ESM（仍有 flag / 限制）

// 绕路 1：动态 import（CJS 里也能用）
async function load() {
  const mod = await import("./esm-module.mjs");
  return mod.default;
}

// 绕路 2：打包成 CJS 分发（构建时 rollup/tsup 转换）
// 库作者的常见模式：
// package.json
// {
//   "exports": {
//     ".": {
//       "import": "./dist/index.mjs",
//       "require": "./dist/index.cjs"
//     }
//   }
// }

打包器的 __esModule marker

TypeScript / Babel / Webpack 编译 ESM 到 CJS 时，给 exports 加 __esModule: true 标记。运行时的 _interopRequireDefault 函数根据此标记决定"是否需要包一层 { default: ... }"：

javascript 
// TS 编译结果
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.foo = 1;

// TS 编译 import foo from "mod" 时
const _mod = require("./mod");
const _default = _mod.__esModule ? _mod.default : _mod;
//                                  ↑ "这不是 TS 编译的 ESM，当 default 用整个"

03 应用场景

dual package：同时支持 ESM 和 CJS

javascript 
// 库作者常用方案（tsup / unbuild / rollup）
// 构建产物：
//   dist/index.mjs   ← ESM
//   dist/index.cjs   ← CJS
//   dist/index.d.ts  ← 类型

// package.json
{
  "type": "module",
  "main": "./dist/index.cjs",       // 老旧 tool 回退
  "module": "./dist/index.mjs",     // 打包器识别
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}

渐进迁移：ESM 主项目用 CJS 依赖

javascript 
// package.json: { "type": "module" }

// index.mjs
import lodash from "lodash";                 // ✓ lodash 是 CJS，default 化
const { debounce } = lodash;                 // 解构取命名

// 或用命名 import（Node 会尝试静态分析）
import { debounce } from "lodash";           // 可能成功，取决于模块写法

// 最兼容的写法：动态 import
const lodash = (await import("lodash")).default;

ESM 替代 __dirname

javascript 
// CJS
const path = require("path");
const config = require(path.join(__dirname, "config.json"));

// ESM (Node 20.11+)
import { join } from "node:path";
import { readFileSync } from "node:fs";
const config = JSON.parse(readFileSync(join(import.meta.dirname, "config.json"), "utf8"));

// ESM 且支持老 Node
import { fileURLToPath } from "node:url";
import { dirname, join } from "node:path";
const __dirname = dirname(fileURLToPath(import.meta.url));

条件加载 CJS 插件（老代码兼容）

javascript 
// ESM 主应用想加载 CJS 插件（不提前知道文件名）
async function loadPlugin(path) {
  const mod = await import(path);
  // 统一解包 default（同时兼容 ESM 和 CJS 插件）
  return mod.default ?? mod;
}

const plugin = await loadPlugin(process.env.PLUGIN_PATH);

04 典型代码

检测当前模块类型

javascript 
// 在库的入口文件里检测环境
const isESM = typeof require === "undefined";
const isCJS = typeof module !== "undefined" && module.exports;
// 但这种检测脆弱 —— 现代做法是导出两份构建产物

// 新代码直接用 import.meta
const isMainEntry = import.meta.main === true;  // Deno / Bun / 新 Node
// Node 22+ 检测 main 有些另外方式

完整 dual package.json

javascript 
{
  "name": "my-pkg",
  "version": "1.0.0",
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.mjs",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    },
    "./utils": {
      "import": "./dist/utils.mjs",
      "require": "./dist/utils.cjs"
    },
    "./package.json": "./package.json"
  },
  "files": ["dist"],
  "scripts": {
    "build": "tsup src/index.ts src/utils.ts --format=esm,cjs --dts"
  }
}

写一个同时适配两种消费的模块

javascript 
// mylib.mjs （用 ESM 写）
export default function myLib() { return "v1"; }
export const version = "1.0";

// 如果要给 CJS 用，构建成 mylib.cjs：
// (rollup / tsup 等构建工具自动转换)
/*
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.default = function myLib() { return "v1"; };
exports.version = "1.0";
*/

// 不要手写双版本！让工具统一输出

CJS 特性：require.cache

javascript 
// CJS 独有：清缓存做热重载（ESM 不支持）
const path = require("path");
function reload(modulePath) {
  const abs = require.resolve(modulePath);
  delete require.cache[abs];
  return require(modulePath);
}

// ESM 对等方案：import.meta.resolve + 动态 import
// 但 ESM 的模块一旦 link 无法清，每次 "hot reload" 要启动新 Worker / 进程
// Vite / Next / Nodemon 这类工具的 HMR 大都基于此差异做封装

05 常见陷阱

exports = { ... } 无效

在 CJS 文件里只写 exports = { foo: 1 } → require 得到空对象。必须 module.exports = { foo: 1 } 或 exports.foo = 1。新手经典坑。

require() 不能用在 ESM（Node 22 前）

Uncaught ReferenceError: require is not defined in ES module scope。用 import 或 createRequire(import.meta.url)（Node util）替代。

ESM 命名 import CJS 包的兼容性

import { readFile } from "fs" —— 老 Node / 早期 Node 22 可能报错。最兼容：import { readFile } from "node:fs"（node: 前缀明确内置模块）或 import fs from "fs"; const { readFile } = fs;。

dual package hazard

同一个 class 从 ESM 入口和 CJS 入口分别加载会得到两份独立的 class，instanceof 失败。解决：让 CJS 版本只是 ESM 版本的 wrapper，或用 exports 条件映射到单一产物。

TS 的 esModuleInterop 改变 import 语义

TS 配置 "esModuleInterop": true 让 import fs from "fs" 编译成 const fs = require("fs").default ?? require("fs")。关掉后要写 import * as fs from "fs"。不同 TS 项目的 import 风格看 tsconfig。

循环 require 的半成品赋值

const { x } = require("./circular") 如果 x 还没定义，解构得到 undefined。改用函数内 require（延迟加载）或重构成无循环依赖。

ESM 内的 require 用 createRequire
javascript 
import { createRequire } from "node:module";
const require = createRequire(import.meta.url);
const pkg = require("./pkg-that-only-has-cjs");
// 偶尔需要强制 CJS 加载时用，一般避免
Jest 与 Vitest 的 ESM 支持差异

Jest 对原生 ESM 支持仍有限（需要 --experimental-vm-modules）。Vitest 默认 ESM 友好。选测试工具时注意。
