Web Workers 与 Module Workers

Worker 是 JS 的多线程能力 —— 把 CPU 密集型计算挪出主线程，保持 UI 60fps 响应。本章讲三种 Worker（Dedicated / Shared / Service）的角色定位、Module Worker 的 ESM 支持、postMessage + Transferable 的零拷贝传输、SharedArrayBuffer + Atomics 的共享内存模型、以及 Comlink 带来的 RPC 编程体验。

阅读时长 ~22 分钟 前置 P3 异步 · 01 ESM MDN Web Workers API

01 速览

三种 Worker 对比

Dedicated Worker Shared Worker Service Worker
用途 单页面 CPU 并行 多 tab 共享一个线程 网络代理、推送、离线
生命周期 页面关闭即终止 所有 tab 关闭才结束 独立于页面，浏览器管理
通信方式 postMessage（1 对 1） MessagePort（1 对多） fetch/message 事件
访问 DOM ✗ ✗ ✗
浏览器支持 几乎所有 Chrome / Firefox，Safari 17+ 全主流
典型场景 图像处理、大数据、加密 多 tab 共享连接 PWA、离线缓存

基本使用（Dedicated Module Worker）

javascript 
// main.mjs
const worker = new Worker(
  new URL("./worker.mjs", import.meta.url),
  { type: "module" }                   // ← 启用 ESM
);

worker.postMessage({ cmd: "compute", data: largeArray });
worker.addEventListener("message", (e) => console.log("got", e.data));
worker.addEventListener("error", (e) => console.error(e));

// 发送后终止
worker.terminate();

// worker.mjs
import { heavyCompute } from "./lib.mjs";   // ESM 里能用 import
self.addEventListener("message", (e) => {
  const result = heavyCompute(e.data);
  self.postMessage(result);
});

02 底层逻辑

Worker 的隔离模型

每个 Worker 是一个独立的 realm：独立的 globalThis、独立的 V8 isolate、独立的 event loop。

  • 无 window / document（有 self, location）
  • 无 DOM API（但能用 fetch, setTimeout, IndexedDB）
  • 有自己的内存堆 —— 和主线程完全不共享（除非用 SharedArrayBuffer）
  • 出错不影响主线程，但主线程能监听 error 事件

postMessage 的结构化克隆

worker.postMessage(data) 不是直接传引用，而是走结构化克隆算法（见 P1-05 / P1-12）：

  • 支持：primitive、Object、Array、Date、RegExp、Map、Set、TypedArray、ArrayBuffer、Blob、File、循环引用
  • 不支持：Function、Symbol、Error 的某些字段、DOM 节点、class 原型
  • 对于大对象是深拷贝 —— 有性能成本（几 MB 数据要几 ms）

Transferable 对象（零拷贝）

javascript 
// 大 ArrayBuffer 如果都要拷贝一份就浪费
const buf = new ArrayBuffer(100 * 1024 * 1024);   // 100MB
worker.postMessage({ buf }, [buf]);
//                          ↑ Transferable 数组
// 效果：buf 的所有权转移到 worker，主线程的 buf.byteLength 变 0

// 可转移类型：ArrayBuffer、MessagePort、ImageBitmap、OffscreenCanvas、ReadableStream、WritableStream、TransformStream、AudioData、VideoFrame

// 常用模式：TypedArray 的底层 buffer 转移
const data = new Uint8Array(1024 * 1024);
worker.postMessage(data, [data.buffer]);
// 转移后 data 本身不可用

SharedArrayBuffer + Atomics

真正的共享内存：主线程和 Worker 可以同时读写同一块内存：

javascript 
// 主
const sab = new SharedArrayBuffer(4);
const view = new Int32Array(sab);
worker.postMessage(sab);

// 主线程
Atomics.store(view, 0, 42);      // 原子写
Atomics.notify(view, 0, 1);      // 唤醒等待者

// worker
self.onmessage = (e) => {
  const view = new Int32Array(e.data);
  Atomics.wait(view, 0, 0);      // 阻塞直到 view[0] !== 0
  console.log(Atomics.load(view, 0));  // 42
};

// ⚠ 启用限制：
// 要求 cross-origin-isolated 环境（Response headers）：
// Cross-Origin-Opener-Policy: same-origin
// Cross-Origin-Embedder-Policy: require-corp
// 否则 SharedArrayBuffer 不可用（Spectre 缓解）

Service Worker 的特殊性

Service Worker 是网络代理，不是通用 Worker：

  • 拦截 fetch 事件 → 自定义响应（缓存/离线/网络拦截）
  • 响应 push 通知
  • 后台同步（background sync）
  • 必须 HTTPS（或 localhost）
  • 浏览器决定何时启动/休眠，不受页面控制
javascript 
// 注册（主线程）
if ("serviceWorker" in navigator) {
  await navigator.serviceWorker.register("/sw.js", { scope: "/" });
}

// sw.js
self.addEventListener("install", (e) => {
  e.waitUntil(caches.open("v1").then(c => c.addAll(["/app.js", "/style.css"])));
});
self.addEventListener("fetch", (e) => {
  e.respondWith(caches.match(e.request).then(r => r ?? fetch(e.request)));
});

Comlink：RPC 风格的 Worker 通信

javascript 
// 手动 postMessage 绑 message id、等回复 … 非常烦
// Comlink 用 Proxy 把它封装成"像调本地函数一样"

// worker.mjs
import * as Comlink from "comlink";
class Api {
  async compute(data) { /* ... */ return result; }
  counter = 0;
  inc() { return ++this.counter; }
}
Comlink.expose(new Api());

// main.mjs
import * as Comlink from "comlink";
const worker = new Worker(new URL("./worker.mjs", import.meta.url), { type: "module" });
const api = Comlink.wrap(worker);

await api.compute(data);         // 实际发 postMessage，等 reply
await api.inc();                 // 1
await api.counter;               // 1  ← 读属性也是异步 proxy

// 底层：每次调用生成 id，message 里带 id，reply 里带 id 匹配对应 Promise

03 应用场景

图像处理

javascript 
// main.mjs
const canvas = document.getElementById("preview");
const offscreen = canvas.transferControlToOffscreen();
const worker = new Worker(new URL("./image-worker.mjs", import.meta.url), { type: "module" });
worker.postMessage({ canvas: offscreen }, [offscreen]);
// 主线程完全不再碰 canvas，所有绘制在 worker

// image-worker.mjs
self.onmessage = ({ data }) => {
  const ctx = data.canvas.getContext("2d");
  // 在这里做滤镜 / 合成 / 动画 —— 不阻塞主线程
};

大数据排序 / 聚合

javascript 
// 100 万行 Excel 排序 —— 主线程 300ms+ 卡顿
// 挪到 Worker：
const worker = new Worker(new URL("./sort-worker.mjs", import.meta.url), { type: "module" });
worker.postMessage({ data, column: "price" }, [data.buffer]);  // 转移 buffer
worker.onmessage = (e) => renderTable(e.data);

// 用户在排序期间还能滚动、点其他按钮
// 适合场景：bundle 里的大数据、CSV parse、ProtoBuf 编解码

加密 / 压缩

javascript 
// WebCrypto 在主线程也能用，但大文件加密会阻塞 UI
// Worker + 流式处理
const worker = new Worker(new URL("./crypto-worker.mjs", import.meta.url), { type: "module" });
const encrypted = await new Promise((resolve, reject) => {
  worker.onmessage = (e) => resolve(e.data);
  worker.onerror = reject;
  worker.postMessage({ op: "encrypt", file }, []);
});

// pdf.js、ffmpeg.wasm 等 CPU 密集型库都运行在 Worker

Monorepo 里 Worker 的 bundling

javascript 
// Vite / Webpack 5 都支持 new Worker(new URL(...)) 静态分析
const worker = new Worker(
  new URL("./workers/image.mjs", import.meta.url),

Service Worker 离线优先

javascript 
// sw.js（workbox 简化示例）
self.addEventListener("fetch", (e) => {
  if (e.request.url.endsWith(".json")) {
    // API：网络优先，离线回退缓存
    e.respondWith(
      fetch(e.request).then((r) => {
        caches.open("v1").then((c) => c.put(e.request, r.clone()));
        return r;
      }).catch(() => caches.match(e.request))
    );
  }
});

04 典型代码

简易 Worker 池

javascript 
class WorkerPool {
  #workers = [];
  #queue = [];
  constructor(url, size = navigator.hardwareConcurrency ?? 4) {
    for (let i = 0; i < size; i++) {
      this.#workers.push({ w: new Worker(url, { type: "module" }), busy: false });
    }
  }
  exec(msg, transfer = []) {
    return new Promise((resolve, reject) => {
      const task = { msg, transfer, resolve, reject };
      const free = this.#workers.find((x) => !x.busy);
      if (free) this.#run(free, task);
      else this.#queue.push(task);
    });
  }
  #run(slot, { msg, transfer, resolve, reject }) {
    slot.busy = true;
    slot.w.onmessage = (e) => {
      slot.busy = false;
      resolve(e.data);
      if (this.#queue.length) this.#run(slot, this.#queue.shift());
    };
    slot.w.onerror = reject;
    slot.w.postMessage(msg, transfer);
  }
}

const pool = new WorkerPool(new URL("./compute.mjs", import.meta.url));
const results = await Promise.all(tasks.map((t) => pool.exec(t)));

类型安全的消息协议

javascript 
// shared.ts
export type MainToWorker =
  | { type: "compute"; payload: { data: Float64Array } }
  | { type: "cancel"; id: string };

export type WorkerToMain =
  | { type: "result"; payload: { data: Float64Array } }
  | { type: "progress"; payload: number }
  | { type: "error"; message: string };

// main.ts
const worker = new Worker(url, { type: "module" });
function send(msg: MainToWorker, transfer: Transferable[] = []) {
  worker.postMessage(msg, transfer);
}
worker.onmessage = (e: MessageEvent<WorkerToMain>) => {
  switch (e.data.type) {
    case "result": handleResult(e.data.payload); break;
    case "progress": updateBar(e.data.payload); break;
    case "error": showError(e.data.message); break;
  }
};

SharedArrayBuffer 计数器

javascript 
// 多 Worker 并行递增同一计数器
const sab = new SharedArrayBuffer(4);
const counter = new Int32Array(sab);

for (let i = 0; i < 4; i++) {
  const w = new Worker(new URL("./inc.mjs", import.meta.url), { type: "module" });
  w.postMessage(sab);
}

// inc.mjs
self.onmessage = ({ data }) => {
  const counter = new Int32Array(data);
  for (let i = 0; i < 1_000_000; i++) {
    Atomics.add(counter, 0, 1);    // 原子递增
  }
};

// 4 个 worker 跑完后 counter[0] === 4_000_000（无竞态）
// 用普通 += 就丢数据了

OffscreenCanvas 完整示例

javascript 
// main.mjs
const canvas = document.getElementById("game");
const offscreen = canvas.transferControlToOffscreen();
const worker = new Worker(new URL("./renderer.mjs", import.meta.url), { type: "module" });
worker.postMessage({ canvas: offscreen, width: canvas.width, height: canvas.height },
                   [offscreen]);

// renderer.mjs
let ctx, width, height;
self.onmessage = (e) => {
  ({ canvas: { getContext }, width, height } = e.data);
  ctx = e.data.canvas.getContext("2d");
  animate();
};

function animate() {
  ctx.clearRect(0, 0, width, height);
  // ... 绘制
  requestAnimationFrame(animate);   // Worker 里也有 rAF（Chrome 支持）
}

05 常见陷阱

postMessage 大对象阻塞

结构化克隆 100MB 对象可能 ~几十 ms。大数据用 Transferable 或 SAB。频繁传小数据不如批量传大消息。

Worker 启动有成本

创建 Worker + 下载 + parse 大概 ~100ms。频繁创建/销毁不值。用 Worker 池复用。

type: "module" 不是所有浏览器支持

Chrome 80+ / Firefox 114+ / Safari 15+。老版本 Worker 只能经典脚本（不能 import）。降级策略：构建器把 worker 源码 inline 或用 importScripts。

importScripts 是经典脚本专属

Module Worker 里不能用 importScripts（抛 Error）—— 要用原生 import。Service Worker 现在也支持 module 形式，但库生态还在迁移。

Worker 文件路径相对于脚本 URL

new Worker("worker.js") 的 "worker.js" 相对于页面 URL。用 new URL("./worker.js", import.meta.url) 相对于当前模块。构建工具也会识别这种写法并处理路径替换。

SharedArrayBuffer 需要 cross-origin isolation

现代浏览器要求 COOP + COEP 响应头才能用 SAB。否则 typeof SharedArrayBuffer === "undefined"。本地开发可以关，生产必须配。

Atomics.wait 不能在主线程调

主线程会抛 TypeError（防死锁）。只能在 Worker 里用。Atomics.waitAsync（ES2024）返回 Promise，主线程可用但仍不建议。

Service Worker 作用域陷阱

register("/a/sw.js") 默认 scope 是 /a/，不能拦截 / 的请求。要 scope 到 / 要服务响应头加 Service-Worker-Allowed: /。生产常见困惑。
