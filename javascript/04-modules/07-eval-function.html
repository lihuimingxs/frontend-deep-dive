eval 与 new Function

eval 是 JS 的"后门"—— 能在运行时把字符串变成代码。这既是它的能力，也是它的罪 —— 破坏静态分析、被用于 XSS、影响 V8 优化。本章讲直接 eval vs 间接 eval 的作用域差异、new Function 为什么"比 eval 安全"、CSP 如何封堵、以及生产环境的替代方案。

阅读时长 ~15 分钟 前置 P2 作用域 · P4 模块 MDN eval

01 速览

javascript 
// eval：当场执行字符串
eval("2 + 2");                        // 4
eval("const x = 1; x");               // 1（但 x 不泄漏到外部）

// new Function：编译字符串为函数，在全局作用域执行
const fn = new Function("a", "b", "return a + b");
fn(2, 3);                              // 5

// Function 构造器等价 new Function
const fn2 = Function("a", "return a * 2");
fn2(5);                                // 10

// 间接 eval（指针/别名）—— 在全局作用域执行
const indirectEval = eval;
indirectEval("const x = 1;");          // x 变全局变量

三种执行字符串代码的方式对比

直接 eval 间接 eval new Function
作用域 当前作用域 全局 全局
能访问外层局部变量 ✓ ✗ ✗
严格模式行为 创建独立作用域 同上 独立函数
V8 优化受损 严重（整个作用域 de-opt） 轻微 轻微
CSP 拦截 unsafe-eval 禁止 同上 同上

02 底层逻辑

直接 vs 间接 eval

规范对"eval(...) 这个调用形式"有特殊判定：

  • 直接 eval：字面量 eval(code)（被调用名字是 eval 且能解析到全局 eval 函数）。代码在调用者作用域执行，能读写外层的 let/const。
  • 间接 eval：除此以外的所有调用形式（(0, eval)(code)、const e = eval; e(code)、globalThis.eval(code)）。代码在全局作用域执行。
javascript 
function test() {
  const x = 42;

  eval("console.log(x)");              // 42  ← 直接 eval，能看到 x

  (0, eval)("try { console.log(x) } catch (e) { console.log('x not found') }");
  // "x not found"  ← 间接 eval，x 不可见

  eval("var y = 1");
  console.log(y);                      // 1  ← 非严格下泄漏到函数作用域

  "use strict";
  eval("var z = 2");
  console.log(z);                      // ReferenceError  ← 严格模式 eval 有独立作用域
}

V8 对 eval 的悲观

编译器优化依赖"静态分析外层作用域的变量"。一旦函数里出现直接 eval，编译器就不知道 eval 会不会创建/修改外层变量 → 放弃优化整个作用域：

javascript 
// V8 能内联 / 寄存器分配 / escape analysis
function hot() {
  let counter = 0;
  for (let i = 0; i < 1e7; i++) counter++;
  return counter;
}

// 有直接 eval → V8 保守，counter 必须在堆上分配（防 eval 读写）
function hotWithEval(code) {
  let counter = 0;
  eval(code);                          // ← 即使不调用，也让整个函数变慢
  for (let i = 0; i < 1e7; i++) counter++;
  return counter;
}
// hotWithEval 比 hot 慢 3-10 倍

Function 构造器的隔离

javascript 
function test() {
  const x = 42;
  const fn = new Function("return x");
  fn();                                // ReferenceError（x 不在全局）
  // 因为 new Function 在全局作用域编译函数体
}

// 好处：比直接 eval 安全 —— 不会意外读写调用方变量
// 坏处：仍然是"字符串变代码"，仍被 CSP 的 unsafe-eval 拦

// Function 构造器的形参列表可以多个字符串
const sum = new Function("a", "b, c", "return a + b + c");
sum(1, 2, 3);                         // 6

CSP 的 unsafe-eval 与 strict-dynamic

http 
// Response header
Content-Security-Policy: script-src 'self';
// eval / new Function → EvalError: Refused to evaluate a string as JavaScript because 'unsafe-eval' is not an allowed source of script

// 允许 eval（不推荐）
Content-Security-Policy: script-src 'self' 'unsafe-eval';

// 但有例外：WebAssembly.instantiate 默认被 unsafe-eval 拦（有独立的 wasm-eval）
// 某些框架（Vue 3、Svelte 5）有模板编译依赖 new Function，生产应该预编译

Trusted Types（新 API）

Chrome 83+ / Edge 83+ 支持。强制"把字符串转成代码"必须通过 policy，从根源防 XSS：

javascript 
// CSP: trusted-types default require-trusted-types-for 'script';

// 现在传字符串给 innerHTML / eval 都被拒
el.innerHTML = "user input";         // TypeError

// 必须走 policy
const policy = trustedTypes.createPolicy("my-policy", {
  createHTML: (s) => DOMPurify.sanitize(s),
  createScript: (s) => /* 白名单检查 */ s,
});
el.innerHTML = policy.createHTML(userInput);

// Google 内部防 DOM XSS 的核心手段，推荐生产启用

03 应用场景

用 JSON.parse 代替 eval JSON

javascript 
// ❌ 远古写法（ES5 前）
const data = eval("(" + jsonText + ")");

// ✓ JSON.parse 从 ES5 起可用，且不会执行代码
const data = JSON.parse(jsonText);

// JSON.parse 严格子集：
// - 字符串必须双引号
// - 不能有注释
// - 不支持 undefined / 函数 / NaN / Infinity

// 想要稍松的 JSON5：解析器库（无 eval）

模板引擎的沙箱化

javascript 
// Vue 模板 / Handlebars 等运行时编译模板用 new Function
// 生产环境应 precompile 到 render 函数，避免 runtime eval

// Vue 3
// - vue.runtime.esm-bundler.js  ← 推荐，模板在构建时编译
// - vue.esm-bundler.js           ← 包含 compiler，运行时编译（需 unsafe-eval）

// 如果必须 runtime：
// 1. 限制模板来源（不让用户输入模板）
// 2. 模板解析完转 AST 而非字符串拼接
// 3. 考虑 lit-html / preact 的 tagged template（不用 new Function）

动态表达式求值的替代方案

javascript 
// 场景：用户提供表达式字符串（spreadsheet、条件规则）
// ❌ 直接 eval 用户输入 —— 严重 RCE 漏洞
const result = eval(userExpression);

// ✓ 用表达式解析器（AST-based，可白名单）
// 例：@jsep/core、mathjs.evaluate
import { evaluate } from "mathjs";
const result = evaluate("a + b * 2", { a: 3, b: 4 });
// mathjs 只允许数学运算 + 白名单函数，不能调用任意 JS

// 或用更严的 JSONata / expr-eval
// 根本方案：限定 DSL + 自己写解析器

在 Worker 里隔离执行

javascript 
// 用户代码在 Worker 里跑 —— 和主线程完全隔离（无 DOM 访问、超时可杀）
const code = `
  onmessage = (e) => {
    const result = (${userFunction})(e.data);
    postMessage(result);
  };
`;
const blob = new Blob([code], { type: "application/javascript" });
const worker = new Worker(URL.createObjectURL(blob));

// 设置超时
const timer = setTimeout(() => worker.terminate(), 5000);
worker.onmessage = (e) => { clearTimeout(timer); /* ... */ };

// 更安全：用 iframe + sandbox="allow-scripts"（无 allow-same-origin）
// 用户代码在独立 realm 里，无法读取 parent document

配置驱动 vs 代码驱动

javascript 
// ❌ 代码驱动（让配置变成 JS 执行）
const rule = eval(config.rule);
if (rule(user)) doAction();

// ✓ 配置驱动（JSON 描述规则）
const rule = { age: { ">=": 18 }, country: "CN" };
if (matchRule(user, rule)) doAction();
// matchRule 按规则 DSL 求值，不 eval

04 典型代码

检测 eval 是否被 CSP 禁用

javascript 
function canUseEval() {
  try {
    // eslint-disable-next-line no-new-func
    new Function("return 1")();
    return true;
  } catch (err) {
    if (err instanceof EvalError) return false;
    throw err;
  }
}

if (!canUseEval()) {
  // 加载不依赖 eval 的 fallback
  await import("./vue-runtime-precompiled.mjs");
}

白名单的"安全 eval"（示意）

javascript 
// 只允许简单的数学表达式
function safeCalc(expr) {
  if (!/^[\d+\-*/(). ]+$/.test(expr)) throw new Error("invalid");
  // 经过正则白名单，可以 Function
  return new Function("return " + expr)();
}

safeCalc("1 + 2 * 3");              // 7
safeCalc("alert(1)");                // throws

// ⚠ 仍有注入风险（如 "1/0" 合法但结果 Infinity）
// 生产用 AST-based 解析器更安全

用 dynamic import 替代 eval 加载模块

javascript 
// ❌ 老式动态加载代码
const code = await fetch(url).then(r => r.text());
eval(code);

// ✓ 动态 import（ESM）
const mod = await import(url);       // 走浏览器的 script 加载机制
// CSP 友好，走 script-src 而不是 unsafe-eval

// ⚠ 注意跨源和 CORS

用 Function.toString 克隆函数

javascript 
// 少数合理用 Function 的场景：跨 worker 传函数（Worker 不能直接传函数）
function cloneFunction(fn) {
  return new Function("return " + fn.toString())();
}

// ⚠ 丢失：闭包变量、原型链、name
// 更好：传数据 + 工厂 id，worker 里用 id 查预定义函数

05 常见陷阱

eval 永远是 last resort

99% 的 eval 用法有更好替代：JSON.parse、动态 import、AST 解析器、Function 构造器 + 白名单。工程中出现 eval 就应该被 code review 质疑。

直接 eval 破坏所在函数的 V8 优化

即使 eval 路径不触发，V8 仍保守。性能敏感代码绝不该有直接 eval。用 (0, eval)(...) 间接版本好一点，但仍不理想。

CSP 一旦严格，很多框架需要配置

Vue / Svelte / AngularJS 运行时模板编译需要 unsafe-eval。生产应该预编译。测 CSP 严格策略时注意框架行为。

new Function 也被 CSP 拦

"new Function 不是 eval 吧？" ——CSP 视角它和 eval 一样都是 unsafe-eval。被同一条指令拦截。WebAssembly.instantiate 也被 wasm-eval 控制。

eval 的返回值

eval 返回最后一个表达式的值。多行时只有最后一个：eval("1; 2; 3") 是 3。语句（var x = 1）返回 undefined。

eval 在严格模式下的作用域变化

严格模式下 eval("var x = 1") 的 x 不泄漏。这是规范有意修正的非严格 bug 行为。ESM 默认严格，所以 ESM 里用 eval 更可控（但仍不推荐）。

Function toString 在某些环境不返回源码

V8 正常返回 source。但某些 minifier 或 Function.prototype.toString 被改写的环境返回 "[native code]" 或占位。不要依赖 toString 做序列化。

setTimeout / setInterval 接受字符串

setTimeout("alert(1)", 100) 历史遗留，相当于 eval 字符串。绝不使用，传函数 setTimeout(() => alert(1), 100)。CSP 也会拦截字符串形式。
