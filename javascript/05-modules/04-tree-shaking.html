Tree-shaking 原理

Tree-shaking 不是魔法 —— 它是"静态分析 + 死代码消除"。有效性取决于三个前提：模块系统静态可分析（ESM）、没有顶层副作用、使用方没破坏静态结构。本章讲原理、sideEffects 声明、/*#__PURE__*/ 注释，以及"为什么我的包没被 shake"诊断法。

阅读时长 ~20 分钟 前置 01 ESM 打包器文档 Webpack Tree Shaking

01 速览

javascript 
// lib.mjs —— 导出 3 个函数
export function used() { /* 200 行 */ }
export function unused1() { /* 5000 行 */ }
export function unused2() { /* 5000 行 */ }

// app.mjs —— 只用 1 个
import { used } from "./lib.mjs";
used();

// 打包后（理想情况）只保留 used
// unused1 / unused2 被"摇掉"

关键前提

  1. ESM 语法：import/export 是静态的，打包器可以静态追踪"哪个 export 被哪个 import 引用"。
  2. 无副作用：移除未用的 export 不能破坏程序语义。如 polyfill、CSS import、副作用注册等必须保留。
  3. 静态访问：obj.foo 可追踪；obj[runtimeKey] 不行。

02 底层逻辑

为什么 ESM 才能 tree-shake

javascript 
// ESM：静态结构
import { a } from "./mod";           // 只导入 a
// 打包器扫 AST → 知道 "mod.b" 没被引用 → 可以删

// CJS：运行时结构
const mod = require("./mod");
const a = mod.a;                     // ← 这里赋值后打包器看不到
// 或者更糟：
const { [key]: v } = mod;            // 动态键，完全无法静态分析

// 所以 CJS 包想被 tree-shake 几乎不可能
// 打包器只能保守保留整个模块

什么是"副作用"

模块顶层执行时修改模块外部状态的代码都是副作用：

  • 给全局对象挂属性（window.Foo = ...）
  • Polyfill（Array.prototype.at = ...）
  • CSS import（import "./style.css"）
  • 注册全局监听器、事件订阅
  • 写入 DOM、启动定时器
  • console.log（理论上也算，但大多数打包器忽略）

打包器默认假设每个模块都有副作用 —— 即使某个 export 未被使用，如果模块 A 只是被 import "./a"（无命名 import），打包器必须保留它。

package.json 的 sideEffects 字段

json 
// 完全无副作用 —— 打包器可大胆裁剪
{
  "name": "my-lib",
  "sideEffects": false
}

// 只有这几个文件有副作用
{
  "sideEffects": [
    "**/*.css",
    "./src/polyfills.js"
  ]
}

// Webpack 5、Rollup、Vite 都遵守
// 启用后体积可能减少 30%+

Pure Annotations

形如 /*#__PURE__*/ 或 /*@__PURE__*/ 的注释告诉打包器"这个函数调用无副作用，如果结果未使用可以删"：

javascript 
// 没有注释：打包器保守认为 createInstance 可能有副作用
export const x = createInstance();    // 保留

// 加注释：明确标记"如果 x 未使用则调用也可删"
export const x = /*#__PURE__*/ createInstance();

// 常见场景：库内部工厂函数
export const debug = /*#__PURE__*/ makeDebugger("app:main");
// 打包时如果 debug 没被 import，makeDebugger 调用也删了

// Terser / Rollup / esbuild 都支持这种注释

打包器的裁剪流程

  1. Parse：构建 AST，收集 import/export。
  2. Link：建立 module graph，标记每个 export 的引用位置。
  3. Mark：从 entry 开始，递归标记"被使用"的 export。
  4. Sweep：生成代码时跳过未被 mark 的 export 和它们的依赖。
  5. Minify：Terser/esbuild minifier 再做 DCE（死代码消除）—— 把 if(false) 分支删了等。

Re-export 桶文件的问题

javascript 
// 桶文件 index.mjs
export * from "./big-a.mjs";
export * from "./big-b.mjs";

// 使用方
import { onlyA } from "./index.mjs";

// 静态分析时：onlyA 来自 big-a 还是 big-b？
// Webpack/Rollup 通常能判断（跟着链走）
// 但如果 big-a 有副作用 → 即使 onlyA 来自 big-b，big-a 也被引入

// 解决：
// 1. 在 big-a 顶上写 sideEffects: false
// 2. 直接从源模块 import：
import { onlyA } from "./big-a.mjs";

ESM vs CJS 打包体积对比

场景 CJS 包 ESM 包
按需 import 3 个函数（模块共 30 个） 包含全部 30 个 只有 3 个
深层依赖传递 全量保留 递归裁剪
多入口共享 不稳定，常出现重复代码 chunk 分裂更合理

03 应用场景

为自己的库开启 tree-shaking

javascript 
// package.json
{
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.mjs",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  },
  "sideEffects": false       // ← 重点
}

// 如果某些文件有副作用，列出来
"sideEffects": ["./dist/polyfill.mjs", "**/*.css"]

发现"意料外的保留"

javascript 
// Webpack: stats analyze
// - webpack-bundle-analyzer
// - source-map-explorer

npm install -D webpack-bundle-analyzer
// webpack.config.js
plugins: [ new BundleAnalyzerPlugin() ]

// Rollup/Vite: rollup-plugin-visualizer
npm install -D rollup-plugin-visualizer
// vite.config.js
import { visualizer } from "rollup-plugin-visualizer";
plugins: [ visualizer({ open: true }) ]

// 产物里出现"不该有的大模块" → 常见 3 原因：
// 1. 整包 import（import _ from "lodash" vs lodash-es）
// 2. 副作用未标注
// 3. CJS 依赖无法 shake

用 pure annotation 优化库代码

javascript 
// ❌ createXX 看起来是运行时调用
export const defaultFormatter = createFormatter("default");

// 消费者只用 customFormatter 时，createFormatter("default") 也被打包

// ✓ 加 pure 注释
export const defaultFormatter = /*#__PURE__*/ createFormatter("default");

// ✓ 或者 lazy
let _default;
export function getDefaultFormatter() {
  return _default ??= createFormatter("default");
}
// 只有 getDefaultFormatter 被调用时才真运行

Lodash / date-fns 的按需导入

javascript 
// ❌ 全量导入（CJS 包，70KB+）
import _ from "lodash";
_.debounce(...);
// 打包后包含整个 lodash

// ✓ ESM 版本（sideEffects: false）
import { debounce } from "lodash-es";
// 只打包 debounce 和它依赖的几个内部函数（~5KB）

// ✓ 按模块 import（Webpack 老项目）
import debounce from "lodash/debounce";
// 手动指定路径，避开桶文件

// date-fns 同理：
// ❌ import dateFns from "date-fns"
// ✓ import { format, parseISO } from "date-fns"

React / Preact 的 tree-shakable API

javascript 
// ✓ 命名 import 让打包器能 shake 未用的
import { useState, useEffect } from "react";

// React 17+ 的 "automatic runtime"：
// 不需要 import React from "react" 就能写 JSX
// 也让打包器能 shake 未用的 React API

04 典型代码

诊断未被 shake 的原因

javascript 
// 1. 用 bundle analyzer 看产物
// 2. 怀疑特定包没 shake → 在测试项目里只 import 一个函数构建
// 3. 看产物里是否只有那一个函数

// 4. 如果保留了整个包，检查：
//    a. package.json sideEffects 字段
//    b. 该包的 main/module/exports 是否 ESM 版本
//    c. 打包器 mode: 是 production 吗（dev 模式不 shake）
//    d. babel/preset-env 是否把 ESM 转成 CJS 了（配 modules: false）

避免 Babel 破坏 tree-shaking

javascript 
// babel.config.js
module.exports = {
  presets: [
    ["@babel/preset-env", {
      modules: false,          // ← 重要！默认是 "commonjs"
      // 传给 Webpack 让它自己处理 ESM
    }]
  ]
};

// 如果漏了 modules: false，Babel 把 import/export 转成 CJS
// 之后 Webpack 就无法 tree-shake 了

CSS side effects

javascript 
// components/Button.mjs
import "./Button.css";                 // 副作用（插入样式）
export function Button() { /* ... */ }

// 如果 Button 未被 import，Button.css 不应被 shake
// package.json 里标注
{
  "sideEffects": ["**/*.css"]
}

// 或在 webpack 配置
module: {
  rules: [{
    test: /\.css$/,
    sideEffects: true,
  }]
}

Polyfill 的 sideEffects 标注

javascript 
// core-js 的 polyfill 是副作用（改全局对象）
// 不能被 shake，否则浏览器缺对应 API

// package.json
{
  "sideEffects": [
    "./dist/polyfills/*",
    "./dist/core-js-shim.mjs"
  ]
}

// core-js 官方 sideEffects 配置就是这样

05 常见陷阱

忘了 sideEffects: false

库作者不声明，消费者再怎么按需 import 也 shake 不干净。新库创建时 package.json 默认就加上（除非确有顶层副作用）。

Babel 把 ESM 转 CJS

@babel/preset-env 默认 modules: "auto" / "commonjs"，把 ESM 变 CJS → 破坏静态结构。配置 modules: false（让 Webpack/Rollup 自己处理）。

Dynamic re-export 的静态不可达

export * from "./" + dynamicName 不存在（语法错），但类似的"根据 key 动态取属性"让打包器放弃。namespace[key] 也一样。

CJS 包无法 tree-shake

即使你写 import { x } from "cjs-pkg"，打包器拿到的还是 CJS → 全量保留。选库时优先看是否有 ESM 版（"module" 字段或 "exports.import"）。

顶层副作用函数调用

registerPlugin(foo) 写在模块顶层 → 打包器不敢删。如果它是注册全局，加 sideEffects；如果是"工厂函数结果"，加 pure annotation 或改成 lazy。

Class 的 static 字段和方法可能被保留

class Foo { static unused() { ... } } —— 即使 Foo.unused 没调用过，class 结构必须保留。打包器对 class 的 DCE 较保守。想极致体积可以拆成独立函数。

dev 模式下 tree-shake 不启用

Webpack production 才 shake（默认）；dev mode 为了 HMR 速度不 shake。测体积一定要用 npm run build 的产物。

桶文件的副作用传染

index.mjs: export * from "./a"; export * from "./b"; —— 如果 a 有副作用而 sideEffects 没列清楚，import 任何 b 的东西也会拖入 a。解决：列出具体副作用文件，或让消费者直接 import 子模块。
