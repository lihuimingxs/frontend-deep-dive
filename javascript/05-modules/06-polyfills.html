Polyfill 策略

Polyfill 是"用 JS 代码在老浏览器上实现新 API"。问题不在于 polyfill 本身，而在选择策略：全量 polyfill 浪费 modern 浏览器；按需 polyfill 需要编译期分析；polyfill.io 省事但有信任问题。本章讲每种策略的取舍 + modern baseline + differential serving。

阅读时长 ~18 分钟 前置 01 ESM · Babel/构建基础 core-js 文档 zloirock/core-js

01 速览

Polyfill vs Transpile 的区别

Polyfill Transpile
解决的问题 新 API（Array.prototype.at, Promise.any） 新 语法（?., async/await, class fields）
实现 运行时 JS 代码 编译时重写源码
代价 包体积 + 运行时开销 仅包体积
工具 core-js / polyfill.io Babel / SWC / tsc

主流方案

  • core-js：最全的 polyfill 库。配 @babel/preset-env 自动按需注入。
  • polyfill.io（已关停） / polyfill-library：按 UA 返回差异化 polyfill bundle。
  • unbundled + modulepreload：浏览器不支持某特性时异步加载 polyfill chunk。
  • baseline + differential serving：给 modern 浏览器发 modern bundle、给老的发带 polyfill 的 legacy bundle。

02 底层逻辑

core-js + Babel 的 3 种模式

javascript 
// babel.config.js

// 模式 1：entry —— 在入口文件 import "core-js" 一次
presets: [
  ["@babel/preset-env", {
    useBuiltIns: "entry",
    corejs: "3.39",
    targets: { browsers: [">0.5%", "not dead"] }
  }]
]
// 消费者代码：
// import "core-js/stable";
// import "regenerator-runtime/runtime";
// Babel 按 targets 把这两行展开成必要的 polyfill import
// 优点：可控、可预测
// 缺点：不管代码用不用，target 需要的都注入

// 模式 2：usage —— Babel 扫描源码用了什么，只注入相关 polyfill
presets: [
  ["@babel/preset-env", {
    useBuiltIns: "usage",
    corejs: "3.39",
    targets: { ... }
  }]
]
// 优点：只注入实际用到的
// 缺点：第三方包的 polyfill 需求可能漏掉（Babel 默认不扫 node_modules）

// 模式 3：false —— 不自动 polyfill（自己管理）
useBuiltIns: false

@babel/plugin-transform-runtime 的意义

Babel 转某些语法（async/await、generator、spread）需要"helper 函数"。默认把这些 helper 内联到每个文件 → 体积膨胀。@babel/plugin-transform-runtime 把 helper 抽到独立包 @babel/runtime，各文件共用：

javascript 
// plugin-transform-runtime 还做另外两件事：
// 1. "沙箱化" polyfill（corejs: 3）—— 把 Array.from 等内置改成从 _corejs 引用
//    避免污染全局原型
// 2. regenerator-runtime 挪到单独模块

// 库作者应该用 transform-runtime（不污染全局）
// 应用作者用 preset-env 的 useBuiltIns: "usage"（更小 bundle）

Modern + Legacy 双产物

Vite 内置 @vitejs/plugin-legacy、Webpack 有 @module/nomodule 模式。思路：

html 
<!-- Modern 浏览器（2017+ 都支持 type="module"）-->
<script type="module" src="/dist/index-modern.js"></script>
<!-- Modern bundle: 不带 polyfill, 体积小, ES2022+ 语法 -->

<!-- Legacy 浏览器（IE 等老的）-->
<script nomodule src="/dist/index-legacy.js"></script>
<!-- Legacy bundle: 带 polyfill, ES5 语法 -->

<!-- 现代浏览器忽略 nomodule 脚本；老浏览器忽略 type="module" 脚本
     结果：各取所需，无浪费 -->

Web Platform Baseline

Web.dev 推动的 "baseline" 概念：某个 feature 在 Chrome / Safari / Firefox / Edge 都支持达 30 个月后，视为 Baseline: widely available。以 baseline 为"不 polyfill"的下限：

  • Baseline 2023：支持 Array.prototype.at、Object.hasOwn、结构化克隆、AbortSignal.timeout 等。
  • Baseline 2024：Array.prototype.toSorted / toReversed、groupBy、Set 集合运算等。

生产决策：支持 baseline + 1-2 年前的浏览器，polyfill 需求几乎为零。

polyfill.io 的历史教训

2024 年 6 月，cdn.polyfill.io 被转手给不明实体，开始给部分用户返回恶意脚本。Cloudflare 自建替代、Google Fonts 下架引用。教训：

  • 任何 <script src="cdn-不受自己控制"> 都是供应链风险。
  • 用 SRI（integrity 属性）可以部分缓解。
  • 最佳实践：polyfill 打进自己的 bundle，或用受信 CDN（cloudflare.com 自建的 polyfill.io、jspm.io）。

异步加载 polyfill 的模式

javascript 
// 特性检测 + 动态加载
async function ensurePolyfills() {
  const tasks = [];
  if (!("at" in Array.prototype)) {
    tasks.push(import("core-js/actual/array/at"));
  }
  if (typeof structuredClone === "undefined") {
    tasks.push(import("core-js/actual/structured-clone"));
  }
  await Promise.all(tasks);
}

ensurePolyfills().then(() => import("./app.mjs"));

// 优点：新浏览器零加载
// 缺点：额外的特性检测代码 + 一次异步加载延迟

03 应用场景

应用型：Vite + 默认 legacy plugin

javascript 
// vite.config.ts
import legacy from "@vitejs/plugin-legacy";
export default {
  plugins: [
    legacy({
      targets: ["defaults", "not IE 11"],   // browserslist 语法
      additionalLegacyPolyfills: ["regenerator-runtime/runtime"],
    })
  ]
};

// 自动生成 modern + legacy 双产物
// index.html 自动插入 module / nomodule
// 产物体积 modern vs legacy 可能差 2-3 倍

库型：只声明依赖，不 bundle polyfill

json 
// library package.json
{
  "peerDependencies": {
    "core-js": "^3.39"
  },
  "engines": {
    "node": ">=18"
  }
}

// 在 README 里写明：
// "如果要支持 ES2022 之前的环境，请手动 import core-js"

// 库不应自己 bundle polyfill —— 会和应用重复

Modern Only：完全不 polyfill

javascript 
// package.json
{
  "browserslist": [
    "last 2 Chrome versions",
    "last 2 Firefox versions",
    "last 2 Safari versions",
    "not dead"
  ]
}

// 适合：企业内部应用、对 browser 有硬要求的产品
// 完全不引 core-js，bundle 体积小 40-60%
// Babel preset-env 会按 browserslist 决定要 transpile 什么

特性检测 + lazy polyfill

javascript 
// app.mjs
async function bootstrap() {
  // 检测 + 按需 polyfill
  if (!window.structuredClone) {
    await import("core-js/actual/structured-clone");
  }
  if (!Array.prototype.toSorted) {
    await import("core-js/actual/array/to-sorted");
  }
  const { main } = await import("./main.mjs");
  main();
}
bootstrap();

// 适合：想"不 bundle polyfill 但兼容部分老浏览器"的 SSG / 静态站

Node 版本差异 polyfill

javascript 
// Node 17 没有 structuredClone，18+ 有
import { structuredClone as polyfill } from "structured-clone";
const clone = globalThis.structuredClone ?? polyfill;

// 更严格：
const structuredClone = globalThis.structuredClone
  ?? (await import("structured-clone")).default;

// package.json 锁定最低 Node 版本
{ "engines": { "node": ">=18" } }

现代 Babel 配置模板

// babel.config.js（应用，非库）
module.exports = {
  presets: [
    ["@babel/preset-env", {
      modules: false,                    // 保留 ESM 让 bundler tree-shake
      useBuiltIns: "usage",              // 按需注入 core-js

      bugfixes: true,                    // 启用现代 bugfix 逻辑
    }]
  ]

// .browserslistrc 或 package.json "browserslist"
// > 0.25%
// not dead
// not IE 11

// 库 —— 用 transform-runtime 不污染全局
module.exports = {
  presets: [
    ["@babel/preset-env", { modules: false, targets: { esmodules: true } }]
  ],
  plugins: [
    ["@babel/plugin-transform-runtime", {
      corejs: { version: 3, proposals: false },
      helpers: true,
      regenerator: true,
    }]
  ]
};

特性检测模板

javascript 
// 通用模式：globalThis 先查、原型先查
function supports(obj, prop) {
  return obj != null && (prop in obj);
}

const hasTopLevelAwait = (async () => { try { await 0; return true; } catch { return false; } })();
const hasStructuredClone = typeof structuredClone === "function";
const hasAt = supports(Array.prototype, "at");
const hasGroupBy = supports(Map, "groupBy");

// 封装成 feature detect 模块供全局查
export const features = {
  structuredClone: hasStructuredClone,
  arrayAt: hasAt,
  // ...
};

Node 内置 polyfill shim

javascript 
// shim.mjs
if (typeof globalThis.fetch === "undefined") {
  const { default: fetch, Headers, Request, Response } = await import("node-fetch");
  globalThis.fetch = fetch;
  globalThis.Headers = Headers;
  globalThis.Request = Request;
  globalThis.Response = Response;
}

if (typeof globalThis.AbortSignal?.timeout !== "function") {
  const origTimeout = globalThis.AbortSignal?.timeout;
  globalThis.AbortSignal = globalThis.AbortSignal ?? {};
  globalThis.AbortSignal.timeout = function (ms) {
    const c = new AbortController();
    setTimeout(() => c.abort(new DOMException("timeout", "TimeoutError")), ms);
    return c.signal;
  };
}

// 入口文件首行 import "./shim.mjs"

05 常见陷阱

polyfill 污染全局

useBuiltIns: "entry"/"usage" + core-js/stable 会给 Array.prototype 加真实方法。库作者这样做会和应用冲突 → 用 @babel/plugin-transform-runtime 的沙箱化 corejs。

browserslist 同时出现在多处

package.json browserslist、.browserslistrc、babel targets 都可以配。优先级混乱导致 polyfill 不按预期。选一处，其他工具都读它。

useBuiltIns: "usage" 不扫 node_modules

Babel 默认只 transpile 你的源码，第三方 CJS / ESM 包不扫。如果某包用了新 API 但没 polyfill → 老浏览器报错。解决：在 exclude 里去掉该包，或对 node_modules 开启 transpile。

多 polyfill 源冲突

项目里同时 import "core-js" 和 import "airbnb-browser-shims" → 两个包都给 Array.prototype 打方法，版本不同行为可能不同。选一家。

polyfill 和 tree-shaking

import "core-js/stable" 是副作用 import —— 不能 shake。必须列在 sideEffects 里，或用 preset-env 的 auto-inject 模式。

Regenerator runtime 意外巨大

某些编译组合把 regenerator-runtime 每个文件都嵌入 → bundle 里出现多份。用 @babel/plugin-transform-runtime 统一到一份。

IE 11 的 polyfill 重担

支持 IE11 意味着：ES5 transpile + 大量 polyfill（Promise / fetch / URL / Array methods / ...）。bundle 体积膨胀 2-3 倍。2025+ 项目基本都已 drop IE 11（MS 也 EOL 了）。

不要用 polyfill.io 这类 UA-based CDN

2024 供应链事件后这类服务信任度大降。要用 CDN-based polyfill，选 cdnjs.cloudflare.com 等有声誉、可审计、可固定版本号的。加 integrity="sha384-..." 子资源完整性校验。
