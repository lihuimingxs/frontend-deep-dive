函数基础

函数是 JS 最重要的抽象单元：一等对象（可赋值、可传递、可作为返回值）+ 闭包能力（捕获定义处的词法环境）。本章只讲"形状"——三种声明方式、参数语法、箭头与普通函数的差异；this 与闭包内存模型等"抽象机制"留到 P2。

阅读时长 ~20 分钟 前置 01 - 03 对应 javascript.info Functions

01 速览

javascript 
// 1. 函数声明（function declaration）
function add(a, b) { return a + b; }

// 2. 函数表达式（function expression）
const add = function (a, b) { return a + b; };

// 3. 命名函数表达式（named function expression）
const add = function add(a, b) { return a + b; };

// 4. 箭头函数（arrow function）
const add = (a, b) => a + b;

// 5. 方法简写（在对象/类里）
const obj = {
  add(a, b) { return a + b; },
};

这 5 种形式在语义上有微妙差异，关键决策维度在下表：

维度 声明 表达式 箭头 方法
提升 ✓（整体提升） ✗（仅绑定提升） ✗ ✗
自己的 this ✓（动态） ✓（动态） ✗（词法） ✓（动态）
arguments ✓ ✓ ✗（用 rest 代替） ✓
可 new ✓ ✓ ✗ ✗（方法简写）
有 prototype ✓ ✓ ✗ ✗
可 yield ✓（需要 function*） ✓ ✗ ✓（*name()）

02 底层逻辑

函数是对象

每个函数都是 Function 的实例，可以被赋值、传参、返回，可以给它挂属性（fn.custom = 1）。规范给函数对象附带几个"内部槽"（[[Call]]、[[Construct]]、[[Environment]]）+ 可见属性：

  • fn.name —— 函数名（匿名表达式赋给变量时推断为变量名）
  • fn.length —— 首个带默认值或 rest 参数之前的形参个数
  • fn.prototype —— 作为构造函数时被 new 的对象的原型（箭头函数无此属性）
  • Function.prototype 上有 call / apply / bind（见 P2）

声明提升 vs 表达式提升

函数声明在所在作用域的顶部被整体提升 —— 定义前就能调用。函数表达式只把变量绑定提升（到 TDZ 或 undefined），函数体在运行到赋值时才存在。

javascript 
greet();                // ✓ "hi"   声明被整体提升
function greet() { console.log("hi"); }

greet2();               // ✗ TypeError: greet2 is not a function
var greet2 = function () { console.log("hi"); };

greet3();               // ✗ ReferenceError  TDZ
const greet3 = () => console.log("hi");

工程建议：顶层模块统一用 const fn = () => ... 或 function fn() { ... }，不要混用。让读者不用关心提升。

箭头函数的 7 个差异

  1. 无自己的 this —— 从定义处的词法作用域继承，bind/call/apply 改不了。
  2. 无 arguments —— 需要用 rest 参数 ...args。
  3. 不能 new —— 没有 [[Construct]] 内部槽。
  4. 无 prototype —— 无法作为构造函数。
  5. 无 super —— 在 class 方法里用箭头函数时 super 指向外层的 super。
  6. 无 new.target。
  7. 不能作为 generator（yield 必须在 function* 内）。

背后原因是箭头函数的设计初衷：解决"回调里 this 丢失"问题的语法糖，不是全功能函数。因此它故意砍掉了这些特性。

参数：默认值、rest、解构

默认值在调用时按需求值（lazy），每次调用新建，避免共享引用陷阱：

javascript 
// 默认值每次调用重新求值，不是首次求值后缓存
function addItem(item, list = []) {
  list.push(item);
  return list;
}
addItem(1);    // [1]
addItem(2);    // [2]  ← 不是 [1, 2]！每次都是新 []
// Python 的默认可变参数坑在 JS 里不存在

// 后面的参数可以引用前面的
function range(start, end = start + 10, step = 1) { ... }

// 默认值仅在 undefined 时触发（null 不触发）
function f(x = 10) { return x; }
f(undefined);   // 10
f(null);        // null

Rest 参数收集剩余实参为真实数组（不是 arguments 那种 Array-like）：

javascript 
function log(level, ...args) {
  console.log(`[${level}]`, ...args);  // spread 展开
}
log("info", "user", 123);               // [info] user 123

// rest 必须是最后一个参数
function wrong(...args, last) { }       // SyntaxError

参数解构让 API 命名参数化，避免顺序依赖：

javascript 
// 传统位置参数（调用端看不出语义）
function createUser(name, age, isAdmin, email) { ... }
createUser("Alice", 30, false, "a@b");

// 选项对象 + 解构 + 默认值
function createUser({ name, age, isAdmin = false, email }) { ... }
createUser({ name: "Alice", age: 30, email: "a@b" });
// 调用端清晰、参数顺序无关、可选字段有默认值

// 不传对象时给空对象默认值，避免解构 undefined 抛错
function createUser({ name, age } = {}) { ... }
createUser();        // 不抛错

arguments 对象

非箭头函数里自动可用的 Array-like 对象，包含所有实参。现代代码应该用 rest 参数替代，原因：

  • rest 是真数组，直接有 .map/.filter；arguments 需要 Array.from(arguments)。
  • 严格模式下 arguments 和形参不再双向绑定（arguments[0] = x 不改第一个形参）。
  • 引擎优化 hint：函数内用到 arguments 会阻止一些内联优化。

03 应用场景

箭头函数 + 数组方法：主力模式

React/Vue 组件的 event handler、arr.map(x => ...)、promises.then(data => ...) 几乎全用箭头。优势：简短 + this 透传。缺点：无 name（Chrome DevTools 堆栈有时难读）。

命名函数表达式：递归 + 调试堆栈

javascript 
// name 在函数内可见，外部不可见 —— 便于递归
const factorial = function fact(n) {
  return n <= 1 ? 1 : n * fact(n - 1);
};
console.log(factorial.name);   // "fact"
// fact(5);  // ReferenceError: fact is not defined (外部看不到)

// 对比匿名表达式
const factorial2 = function (n) {
  return n <= 1 ? 1 : n * factorial2(n - 1);  // 依赖外部变量
};
// 若 factorial2 被重新赋值，递归就崩了

IIFE：ESM 之前的模块化

立即执行函数 (function(){ ... })() 在模块系统出现之前用于：创建私有作用域、避免污染全局、启用严格模式。现代代码已经被 ESM 替代，但你仍会在库的 UMD 打包产物、老代码、<script> 内联代码里见到。

默认值作为"必填校验"

javascript 
const required = () => { throw new Error("param required"); };

function sendEmail({ to = required(), subject = "", body = "" }) { ... }

sendEmail({});           // Error: param required
sendEmail({ to: "a@b" }); // ok

TypeScript 有了更好的方式，但在纯 JS 项目里这个模式仍然有用。

高阶函数 + 柯里化：构造特化函数

javascript 
// 通用的延迟执行
const debounce = (fn, ms = 300) => {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), ms);
  };
};
const debouncedSearch = debounce(doSearch, 500);

// 柯里化：把 2 参数函数变成单参数链
const add = a => b => a + b;
const inc = add(1);
inc(5);   // 6

04 典型代码

参数 = 解构 + 默认 + rest 组合

javascript 
function request(
  url,
  {
    method = "GET",
    headers = {},
    body,
    timeout = 10_000,
    ...rest
  } = {},
) {
  return fetch(url, { method, headers, body, signal: AbortSignal.timeout(timeout), ...rest });
}

request("/api/users");
request("/api/users", { method: "POST", body });
request("/api/users", { cache: "no-store" });   // rest 收走

箭头函数的表达式体 vs 块体

javascript 
// 表达式体：隐式返回
const double = x => x * 2;
const pair = (a, b) => [a, b];
const makeObj = x => ({ x });           // ⚠ 返回对象字面量要加括号

// 块体：需要显式 return
const process = x => {
  const y = x * 2;
  return y + 1;
};

// 无 return 的块体相当于返回 undefined
const sideEffect = x => { console.log(x); };
sideEffect(1);    // 打印 1，返回 undefined

name 与 length

javascript 
function a(x, y) {}           a.name;    // "a"
const b = function () {};     b.name;    // "b"   ← 推断
const c = () => {};           c.name;    // "c"   ← 推断
const d = { fn() {} }.fn;     d.name;    // "fn"

// length 只数到首个默认值/rest 之前
function f(a, b, c) {}        f.length;  // 3
function g(a, b = 1, c) {}    g.length;  // 1    ← 从 b 开始不数
function h(a, ...rest) {}     h.length;  // 1

// 工程用途：柯里化库检测"已接收多少参数"

老项目里还常见的 IIFE 模块化

javascript 
const Counter = (function () {
  let count = 0;                    // 私有，外部无法访问
  return {
    increment() { return ++count; },
    get() { return count; },
  };
})();

Counter.increment();  // 1
Counter.get();        // 1
Counter.count;        // undefined  ← 无法外部访问

05 常见陷阱

箭头函数作为对象方法时 this 丢失

const obj = { x: 1, get: () => this.x }; —— this 不是 obj，是外层（模块里是 undefined，浏览器全局里是 window）。方法要用方法简写 get() { return this.x; }。

回调里 this 丢失用 bind 或箭头修复

elem.addEventListener("click", this.handle) 里的 this 在调用时是 DOM 元素。解决：this.handle.bind(this)，或定义时用 class field handle = () => { ... }（箭头继承词法 this）。详细见 P2 this 章。

默认值与 arguments 的交互

非严格模式下，函数形参和 arguments 有"神奇绑定"。一旦用了默认值、rest 或解构，这个绑定失效，行为和严格模式一致。工程实践：现代代码都是严格模式（ESM 自动启用），不用纠结非严格行为。

同名函数声明互相覆盖

同一作用域里 function f() { ... } 声明两次，后者覆盖前者（因为提升时都被处理了）。严格模式下的块级作用域中行为更严格一些。用 const 代替可以让重复声明编译期报错。

用 function 在 if 块里声明的古怪行为

if (cond) { function f() {} } 在严格模式下属块级作用域；非严格模式下有条件提升（历史坑）。规避：条件定义时用 const f = cond ? () => a : () => b;。

参数变量 = 默认值 vs 解构默认值的执行顺序

function f({ a = 1 } = {}) { } 的求值顺序：(1) 接收实参，(2) 若为 undefined 用外层默认 {}，(3) 解构 a，(4) 若 a 为 undefined 用内层默认 1。多层默认值调试困难，保持"一层默认值"更易读。

箭头函数没有 prototype，不能作构造

const Foo = () => {}; new Foo(); → TypeError。这不是 bug，是规范。需要构造函数用 class 或 function。
