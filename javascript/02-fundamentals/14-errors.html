错误处理与严格模式

错误处理的本质是控制"未知失败"的蔓延范围。JS 的 try/catch/finally 语法简单，但围绕它的典型问题都在别处：async 错误何时算 unhandled、throw 原始值丢失堆栈、如何透传根因（cause）、跨 realm 的 instanceof Error 陷阱。本章最后顺带收口"严格模式"—— P1 收尾。

阅读时长 ~22 分钟 前置 04 函数 · 13 iterator 对应 javascript.info Error handling; The modern mode, "use strict"

01 速览

javascript 
try {
  riskyOperation();
} catch (err) {              // catch 参数可省略（ES2019+ "optional catch binding"）
  console.error(err);
  throw err;                  // 抛回去 / 包装后抛
} finally {
  cleanup();                  // 总执行（即使 try 里有 return / throw）
}

// 抛出
throw new Error("message");
throw new TypeError("x must be number");
throw new RangeError("out of bounds");
// ⚠ 反模式：throw "string" / throw { code: 1 }  —— 丢失 stack

Error 的内置类型层次

类型 触发场景
Error 基类。自定义错误一般继承自这里
TypeError 值的类型不匹配（null.x、1n + 1）
RangeError 值超出有效范围（new Array(-1)）
SyntaxError 语法错（JSON.parse("{")、eval("1+")）
ReferenceError 访问未声明变量、TDZ 访问
URIError decodeURIComponent("%")、malformed URI
EvalError 历史遗留（现代引擎不再主动抛，仅保留构造函数）
AggregateError Promise.any 全部 reject 时抛出，聚合多个错误
DOMException 浏览器 API 错（AbortError、NetworkError、...）

02 底层逻辑

Error 对象的标准字段

字段 含义
name 构造函数名（"Error" / "TypeError" / ...）
message 构造时传入的字符串
stack 非标准但引擎普遍实现；调用栈文本
cause ES2022；底层根因（见下）

cause：根因链

ES2022 起 Error 构造的第二参支持 { cause }：

javascript 
try {
  await db.query("...");
} catch (err) {
  throw new Error("Failed to load user", { cause: err });
  //           ↑ 业务层语义                   ↑ 底层原因
}

// 消费端可以追溯
try { /* ... */ }
catch (err) {
  console.error(err.message);        // "Failed to load user"
  console.error(err.cause?.message); // 原始 DB 错误
  // cause 可以嵌套，形成链
}

// Node 在 console 里自动打印 cause chain
// Caused by: Error: connection timeout
//     at ...

async / Promise 错误传播

Async 函数的 throw 不是同步抛出，而是返回一个 rejected Promise。try/catch 配合 await 才能捕获：

javascript 
async function get() { throw new Error("boom"); }

// ✓ 能捕获
try { await get(); } catch (e) { console.error(e); }

// ✗ 捕获不到（没 await，try 已经"过去"了）
try { get(); } catch (e) { console.error(e); }
// → 这是一个 unhandled rejection

// ✗ 在 setTimeout 回调里抛 —— try/catch 外层捕获不到
try {
  setTimeout(() => { throw new Error("timer boom"); }, 0);
} catch (e) { /* 永远不会到 */ }
// → 会成为全局 uncaught error

finally 的优先级最高

finally 中的 return 会覆盖 try / catch 的 return。同样 finally 中的 throw 会淹没原本要传播的异常。

javascript 
function f() {
  try {
    return "A";
  } finally {
    return "B";              // ⚠ 覆盖了 try 的 return
  }
}
f();                         // "B"

function g() {
  try {
    throw new Error("original");
  } finally {
    throw new Error("replaced");   // ⚠ 原 error 被吞
  }
}
// 尽量让 finally 只做清理（close/release），避免 return / throw

Stack trace 的实现

err.stack 不是规范字段，但所有现代引擎都有。V8 的格式：

  • 第 0 行：{name}: {message}
  • 第 1+ 行： at {funcName} ({file}:{line}:{col})

V8 还提供 Error.captureStackTrace(targetObj, constructorOpt?) —— 把"当前调用栈"挂到对象的 .stack 上，并从栈中移除 constructorOpt 之前的帧（去掉自定义 Error 构造函数本身）。Node/Chrome 专属。

javascript 
class AppError extends Error {
  constructor(message, options) {
    super(message, options);
    this.name = "AppError";
    if (Error.captureStackTrace) {
      Error.captureStackTrace(this, AppError);
    }
  }
}

全局错误捕获

场景 浏览器 Node.js
未捕获同步错误 window.onerror / addEventListener("error", ...) process.on("uncaughtException", ...)
未捕获 Promise rejection window.onunhandledrejection / addEventListener("unhandledrejection", ...) process.on("unhandledRejection", ...)
rejected Promise 后来被捕获 addEventListener("rejectionhandled", ...) process.on("rejectionHandled", ...)
Web 平台资源加载失败（图片/脚本） window.addEventListener("error", e => ..., true) 捕获阶段 —

03 应用场景

自定义 Error class 做领域错误分类

javascript 
// 基类
class AppError extends Error {
  constructor(message, options) {
    super(message, options);
    this.name = this.constructor.name;   // 自动用类名
  }
}

// 各种业务错误
class NotFoundError extends AppError {
  constructor(entity, id) { super(`${entity} ${id} not found`); this.entity = entity; this.id = id; }
}
class PermissionError extends AppError {
  constructor(action) { super(`No permission for ${action}`); this.action = action; }
}
class ValidationError extends AppError {
  constructor(fields) { super("Validation failed"); this.fields = fields; }
}

// 调用层用 instanceof 分派
try {
  await svc.update(id, data);
} catch (err) {
  if (err instanceof ValidationError) return res.status(400).json({ fields: err.fields });
  if (err instanceof NotFoundError)   return res.status(404).json({ entity: err.entity });
  if (err instanceof PermissionError) return res.status(403).json({ action: err.action });
  throw err;   // 未知错误透传给上层
}

Result / Either 模式（反 try/catch）

Rust / Go 风格的显式错误 —— 把错误作为返回值的一部分，强制调用者处理。适合"错误是预期内分支"的场景（表单校验、HTTP 状态码）。

javascript 
async function fetchUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) return { ok: false, error: `HTTP ${res.status}` };
    return { ok: true, value: await res.json() };
  } catch (err) {
    return { ok: false, error: err.message };
  }
}

const r = await fetchUser(1);
if (!r.ok) { handleError(r.error); return; }
use(r.value);

// TypeScript 下可以定义判别联合类型确保类型安全：
// type Result<T, E = string> = { ok: true; value: T } | { ok: false; error: E };

全局拦截 + 上报

javascript 
// 浏览器
window.addEventListener("error", (e) => {
  report({ type: "error", message: e.message, stack: e.error?.stack });
});
window.addEventListener("unhandledrejection", (e) => {
  report({ type: "rejection", reason: e.reason?.message ?? String(e.reason) });
  e.preventDefault();    // 阻止控制台红字（可选）
});

// Node
process.on("uncaughtException", (err) => {
  logger.fatal(err);
  process.exit(1);       // ⚠ 按规范建议：异常后退出，让进程管理器重启
});
process.on("unhandledRejection", (reason) => {
  logger.error(reason);
});

Promise.any + AggregateError

javascript 
try {
  const first = await Promise.any([
    fetch(primary),
    fetch(mirror1),
    fetch(mirror2),
  ]);
  // 任一成功就返回
} catch (err) {
  // err instanceof AggregateError
  // err.errors 是数组，每个对应失败的 Promise
  console.error("All sources failed:", err.errors);
}

assert 与 invariant 式防御

javascript 
function invariant(cond, msg) {
  if (!cond) throw new Error(`Invariant failed: ${msg}`);
}

function processOrder(order) {
  invariant(order, "order required");
  invariant(order.items.length > 0, "order has no items");
  invariant(Number.isFinite(order.total), "total must be a number");
  // ...业务逻辑
}

// TypeScript 的 asserts 能让类型窄化：
// function invariant(cond: unknown, msg: string): asserts cond { ... }

04 典型代码

高阶 try 工具

javascript 
// 转成 [err, value] 元组（Go 风）
async function to(promise) {
  try { return [null, await promise]; }
  catch (err) { return [err]; }
}

const [err, user] = await to(fetchUser(id));
if (err) { /* ... */ return; }

// 默认值 fallback
function tryOr(fn, fallback) {
  try { return fn(); } catch { return fallback; }
}
const cfg = tryOr(() => JSON.parse(raw), {});

重试 + 指数退避

javascript 
async function retry(fn, { maxAttempts = 3, baseMs = 200, signal } = {}) {
  let attempt = 0, lastErr;
  while (attempt < maxAttempts) {
    if (signal?.aborted) throw new DOMException("aborted", "AbortError");
    try {
      return await fn();
    } catch (err) {
      lastErr = err;
      if (err.name === "AbortError" || err.retryable === false) throw err;
      attempt++;
      if (attempt >= maxAttempts) break;
      const delay = baseMs * 2 ** (attempt - 1) + Math.random() * 100;
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error(`Failed after ${maxAttempts} attempts`, { cause: lastErr });
}

把同步异常转为 rejection

javascript 
// setTimeout 回调里的同步抛 —— 逃逸 try/catch
setTimeout(() => { throw new Error("boom"); }, 0);    // ❌ 上层无法捕获

// 改造：把回调包进 async，rejection 进入全局 unhandledrejection
setTimeout(async () => {
  try { await task(); }
  catch (err) { handle(err); }
}, 0);

Optional catch binding

javascript 
// ES2019+：不用声明参数
try { JSON.parse(text); }
catch { return fallback; }    // 省略 (err)

// 用处：不关心错误内容只要 fallback 的场景
// 线上生产代码建议仍写 (err)，便于后续扩展

05 常见陷阱

throw 非 Error 值丢失堆栈

throw "error" / throw { code: 1 } 合法，但 Sentry 等错误追踪工具收集不到 stack。永远 throw Error 实例。自定义数据挂在 Error 实例的属性上。

instanceof Error 跨 realm 不可靠

iframe / Worker 中 throw 的 Error，主文档 catch 到时 err instanceof Error 是 false（构造函数不同）。替代：err && typeof err.message === "string" && typeof err.stack === "string"，或用 err?.name === "TypeError"。

finally 中 return / throw 覆盖

见上文。好习惯：finally 只做 conn.close() / fd.release() 类清理，不做业务返回。

async 函数调用没 await，rejection 逃逸

fetch(url).then(...) 忘写 .catch，或 await 外面没 try/catch。现代工具链（Promise.prototype.finally + ESLint no-floating-promises）能查。生产必须接全局 unhandledrejection 监控。

异步错误在定时器 / 事件处理器里逃逸

setTimeout(fn, 0)、element.addEventListener("click", fn) 里抛的错不在外层 try 里。两种办法：回调里自己 try；或绑全局 window.onerror。

子类 Error 默认 name 还是 "Error"

直接 class MyErr extends Error {}，new MyErr().name 是 "Error" 而不是 "MyErr"。需要在构造函数里 this.name = "MyErr" 或 this.constructor.name。

catch 捕获了就"吞掉"了

catch { /* empty */ } 是经典反模式。至少要日志；最好 throw 包装后的错。"Silent catch" 是生产事故根源 TOP 3。ESLint no-empty 能拦。

Error.cause 不被所有工具链显示

Node 20+ 的 console 原生展示 cause chain，旧版本只打印表层。Sentry / Bugsnag 等工具正在支持中。安全起见，在自定义 Error 里还把 cause 的 message 拼到自己的 message：super(\`${msg}: ${cause.message}\`, { cause })。

附 严格模式

启用方式

方式 作用域
文件首行 "use strict" 整个文件
函数首行 "use strict" 该函数及嵌套
ES Module（<script type="module">、.mjs、import/export） 整个模块（隐式严格）
class 内部 class body（隐式严格）

严格模式的 8 条主要变化

  1. 未声明变量赋值抛 ReferenceError（非严格模式会静默创建全局变量）
  2. 函数调用时 this 是 undefined（非严格是全局对象）
  3. 删除不可删除属性抛 TypeError（非严格静默失败）
  4. 重复的函数参数名报 SyntaxError（function(a, a) {}）
  5. 八进制字面量 0123 禁（用 0o123）
  6. with 语句禁
  7. eval / arguments 不能作为变量名
  8. eval 内部的变量声明不污染外层

工程建议：现代代码必须严格模式。用 ESM（"type": "module"、.mjs、Vite/Next 等）自动严格；打包器输出默认严格；TypeScript 的 "alwaysStrict" 也给所有文件加 "use strict"。不需要手写 "use strict"。
