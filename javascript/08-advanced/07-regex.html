正则表达式引擎

JavaScript 正则由 V8 的 Irregexp（其他引擎类似）实现，属于带回溯的 NFA 模型。不理解回溯 → 写出 ^(a+)+$ 在恶意输入下指数爆炸（ReDoS），线上服务挂掉。本章讲清回溯机制、u / v flag 的 Unicode 语义、如何预防 ReDoS、以及 2024 / 2025 的新特性（duplicate named groups、原子组、\p{...} property escape）。

阅读时长 ~25 分钟 前置 P1 字符串 规范 ECMA-262 §22.2

01 速览

  • JS 正则是 NFA + 回溯：匹配失败时回退到上个分支继续试；最坏情况时间复杂度指数。
  • 贪婪（默认） *+?{n,m} 先抢最多再回吐；惰性加 ?（*?、+?）先抢最少再增加。
  • u flag 启用 Unicode：. 匹配 surrogate pair 里的完整 emoji、\p{...} 可用、字符类按码点解析。
  • v flag（ES2024）扩展 u：支持 set notation（集合交/差/并）、字符串字面量、部分错误升级为 SyntaxError。
  • d flag（ES2022）让 match 结果带 indices（每个捕获组的 [start, end]）。
  • sticky y：从 lastIndex 处精确匹配，不跳过；配合 g 实现 tokenizer。
  • ES2025：重复命名组 (?<year>\d{4})|(?<year>\d{2})。
  • ReDoS 防御：避免嵌套量词 (a+)+、用锚点 + 非贪婪、用 WASM 的 RE2 或超时限制。

02 底层逻辑

NFA 与回溯

正则 /a(b|c)d/ 对字符串 "acd"：

状态机：
   a             b
  ──→ q1 ──→ q2 ──→ q3
                │    ↑
                │ c  │
                └────┘

匹配过程：
1. q1 读 "a" → 成功（index 0→1）
2. q2 试 b 分支：读 "c" ≠ b → 失败，回溯
3. q2 试 c 分支：读 "c" ✓ → q3（index 2）
4. q3 读 "d" ✓ → 接受

贪婪 vs 惰性

const s = "<a><b>";

"<a><b>".match(/<.*>/)[0];   // "<a><b>" 贪婪：抢最多
"<a><b>".match(/<.*?>/)[0];  // "<a>" 惰性：先抢最少

// 贪婪的回溯代价：
//   /.*d/ 对 "aaaa...aaa" 最后没 d：
//   1. .* 抢完整个字符串
//   2. 匹配 d 失败
//   3. 回退 1 字符，再试 d → 失败
//   4. 继续回退 ... n 次回溯才报 no match

// 惰性 /.*?d/：
//   1. .* 先抢 0 个
//   2. 试 d → 失败
//   3. .* 抢 1 个，试 d → 失败
//   4. ... 同样 n 次
// 不比贪婪快；只是匹配到的子串不同

灾难性回溯（ReDoS）

// 嵌套量词 → 指数爆炸
/^(a+)+$/.test("aaaaaaaaaaaaaaaaaaaaaaaaaX");
// 每个 a 可以被内层 + 捕获 1,2,3...次，外层 + 再组合
// n 个 a 需要 2^n 次回溯 → 几十个 a 就卡死几秒

// 重叠交替
/^(a|a)+$/.test("aaaaaaaaaaaX");  // 指数

// 即使单层量词，匹配条件重叠也有问题
/\w+\s\w+\s\w+$/.test(" ".repeat(100) + "x"); // 多个 \w+ 可吃空格导致回溯

// 安全写法：锚定 + 非重叠
/^a+$/.test(...);           // 线性
/^[a-z]+@[a-z]+\.[a-z]+$/.test(...); // 线性（字符类互不重叠）

u 与 v flag 的 Unicode 语义

// 无 u：正则按 UTF-16 code unit
"🎉".length;            // 2（surrogate pair）
/./.test("🎉");         // true（匹配高代理）
"🎉".match(/./)[0];     // "\uD83C"（只匹配一半！）

// u flag：按 Unicode code point
/./u.test("🎉");        // true
"🎉".match(/./u)[0];    // "🎉"

// Unicode property escape（u 必需）
/\p{L}/u.test("中");      // 是 Letter
/\p{Script=Han}/u.test("中"); // 汉字
/\p{Emoji_Presentation}/u.test("🎉"); // emoji
/\p{ASCII}/u.test("a");   // ASCII 字符
/\P{L}/u.test("1");       // 非 Letter（大写 P）

// v flag（ES2024）扩展：set notation
/[\p{Decimal_Number}&&\p{ASCII}]/v.test("3"); // 交集
/[\p{L}--[a-z]]/v.test("A");                   // 差集
/[\q{foo|bar|baz}]/v.test("foo");              // 字符串字面量（v 独有）

捕获组、非捕获组、命名组、反向引用

const date = "2026-04-24";
const m = date.match(/(\d{4})-(\d{2})-(\d{2})/);
m[0]; // "2026-04-24"
m[1]; // "2026"

// 命名捕获
const m2 = date.match(/(?<year>\d{4})-(?<mon>\d{2})-(?<day>\d{2})/);
m2.groups; // { year: "2026", mon: "04", day: "24" }

// 非捕获 (?:...)：性能更好（不分配捕获槽）
/(?:foo|bar)/.test("foo");

// 反向引用
/(\w)\1/.test("aa");   // true：\1 引用第 1 组
/(?<c>\w)\k<c>/.test("aa"); // 命名组反向引用

// ES2025：重复命名组（不同分支）
"2026-04".match(/(?<y>\d{4})-\d+|(?<y>\d{2})年/); // 两分支都能用 y

Lookahead / Lookbehind

// 零宽断言 —— 只检查不消费

/foo(?=bar)/.exec("foobar")[0];  // "foo"（后面是 bar 但不吃 bar）
/foo(?!bar)/.exec("fooqux")[0];  // "foo"（后面不是 bar）

// lookbehind（ES2018）
/(?<=\$)\d+/.exec("$100")[0];   // "100"（前面是 $ 但不包括）
/(?<!\$)\d+/.exec("¥100")[0];   // "100"（前面不是 $）

// 实用：替换不带指定前缀的单词
"price $10, cost 20".replace(/(?<!\$)\d+/g, "N"); // "price $10, cost N"

// lookbehind 可变长度（ES2018 后大部分引擎支持）
/(?<=\w+=)\d+/.exec("key=123")[0]; // "123"

Sticky & Multiline

// y (sticky)：从 lastIndex 处必须立即匹配，不跳过
const re = /\w+/y;
re.lastIndex = 5;
"abc def ghi".match(re); // null（index 5 是 "def"，但 y 要求从 5 开始 fit）
// 等等，实际 "abc def ghi" 的 index 5 是 'd'，能匹配，返回 "def"
// 关键差异：如果 lastIndex=4（空格）→ null（不会跳过空格）

// m (multiline)：^ $ 匹配每行开始/结尾
"line1\nline2".match(/^\w+$/gm); // ["line1", "line2"]

// s (dotAll)：. 也匹配换行符
/foo.bar/.test("foo\nbar");   // false
/foo.bar/s.test("foo\nbar");  // true

d flag - 捕获组位置索引（ES2022）

const m = "hello world".match(/(\w+) (\w+)/d);
m.indices;       // [[0,11], [0,5], [6,11]]
m.indices.groups; // { }  如果有命名组：{ name: [start, end] }

03 应用场景

模板解析 / 简易 lexer（sticky）

const rules: [RegExp, string][] = [
  [/\s+/y, "whitespace"],
  [/\d+(?:\.\d+)?/y, "number"],
  [/"([^"\\]|\\.)*"/y, "string"],
  [/[a-z_][a-z0-9_]*/iy, "ident"],
  [/[+\-*/()]/y, "punct"],
];

function* tokenize(src: string) {
  let pos = 0;
  while (pos < src.length) {
    for (const [re, type] of rules) {
      re.lastIndex = pos;
      const m = re.exec(src);
      if (m && m.index === pos) {
        yield { type, value: m[0], pos };
        pos = re.lastIndex;
        break;
      }
    }
  }
}

URL 拆分

// 生产用 URL 构造器，但正则的教学版值得看
const URL_RE = /^(?<scheme>https?):\/\/(?<host>[^/:]+)(?::(?<port>\d+))?(?<path>\/[^?#]*)?(?:\?(?<query>[^#]*))?(?:#(?<hash>.*))?$/;
const m = "https://example.com:443/a/b?q=1#top".match(URL_RE);
m.groups; // { scheme, host, port, path, query, hash }

// 生产：优先 new URL()
new URL("https://example.com:443/a/b?q=1#top");

HTML 简单解析（仅教学；生产用 DOMParser）

// 这个例子展示非贪婪在嵌套标签的局限
const html = "<b>X</b><b>Y</b>";
html.match(/<b>(.*)<\/b>/)[1];   // "X</b><b>Y" 贪婪
html.match(/<b>(.*?)<\/b>/)[1];  // "X" 惰性

// 正则无法处理平衡嵌套：<div><div></div></div>
// 无限嵌套的 HTML 需要 DOMParser

替换 + 函数计算

// 所有 %xx 转 HTML entity
"café".replace(/[^\x00-\x7F]/g, (ch) => `&#${ch.codePointAt(0)};`);

// 反向排序单词
"hello world".replace(/\b(\w+)\b/g, (_, w) => [...w].reverse().join(""));

// 用命名组构造
"2026-04-24".replace(/(?<y>\d{4})-(?<m>\d{2})-(?<d>\d{2})/, "$<d>/$<m>/$<y>");
// "24/04/2026"

Unicode 感知的字符处理

// 分割 emoji（无 u 会把 surrogate 切断）
"a🎉b".split("");        // ["a", "\uD83C", "\uDF89", "b"]
[..."a🎉b"];            // ["a", "🎉", "b"] 可迭代协议按 code point

// 按字素簇（grapheme cluster）才能正确处理
// emoji + ZWJ + emoji（👨‍👩‍👧‍👦 family）是多个 code point 组成的单个"字"
// 必须用 Intl.Segmenter（P7-08）

// 按 Unicode script 筛选
const CJK = /\p{Script=Han}/u;
"中国 123".replace(/\p{P}/gu, ""); // 去标点

04 典型代码

ReDoS 安全的 email 校验

// 绝对不要用网上流传的复杂 email regex
// HTML5 input[type=email] 规范版本（简单 + 不会回溯爆炸）
const EMAIL = /^[a-zA-Z0-9.!#$%&'*+/=?^_`{|}~-]+@[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$/;

// 分析：每段字符类互不重叠，无嵌套量词 → 线性
// 真要严格校验 email，发确认邮件最靠谱

日志解析（命名组 + d flag）

const LOG_RE = /^\[(?<ts>[\d-T:.Z]+)\]\s+\[(?<lvl>\w+)\]\s+(?<msg>.*)$/d;

function parseLog(line: string) {
  const m = line.match(LOG_RE);
  if (!m?.groups) return null;
  return {
    ts: new Date(m.groups.ts),
    level: m.groups.lvl,
    message: m.groups.msg,
    // 用 d flag 得到每个字段的位置，便于高亮显示
    positions: m.indices!.groups,
  };
}

Token 化的简单 markdown inline parser

// **bold**、*italic*、`code`、[text](url)
const rules = [
  [/\*\*([^*]+)\*\*/g, (_, t) => `<strong>${t}</strong>`],
  [/\*([^*]+)\*/g, (_, t) => `<em>${t}</em>`],
  [/`([^`]+)`/g, (_, t) => `<code>${escape(t)}</code>`],
  [/\[([^\]]+)\]\(([^)]+)\)/g, (_, t, u) => `<a href="${u}">${t}</a>`],
];
function md(src) {
  let r = src;
  for (const [re, fn] of rules) r = r.replace(re, fn);
  return r;
}

防 ReDoS：timeout 正则

// JS 原生没有正则超时。用 Worker + timeout 实现
function matchWithTimeout(src: string, pattern: string, flags: string, timeoutMs: number): Promise<RegExpMatchArray | null> {
  return new Promise((resolve, reject) => {
    const worker = new Worker(
      URL.createObjectURL(new Blob([`
        self.onmessage = ({ data: { src, pattern, flags } }) => {
          try { self.postMessage({ m: src.match(new RegExp(pattern, flags)) }); }
          catch (e) { self.postMessage({ err: e.message }); }
        };
      `], { type: "application/javascript" })),
    );
    const timer = setTimeout(() => {
      worker.terminate();
      reject(new Error("regex timeout"));
    }, timeoutMs);
    worker.onmessage = ({ data }) => {
      clearTimeout(timer); worker.terminate();
      if (data.err) reject(new Error(data.err));
      else resolve(data.m);
    };
    worker.postMessage({ src, pattern, flags });
  });
}

05 常见陷阱

陷阱 1：ReDoS 让服务挂掉

用户输入走到正则 —— 恶意构造可触发指数回溯。审计所有用户输入的正则：用 devina.io / safe-regex 工具扫描；改用 RE2 或 WASM 引擎。

陷阱 2：g flag 和 lastIndex 的副作用

re.test(str) 在 g flag 下第二次可能从 lastIndex 开始返回 false。重复使用同一 regex 实例前 re.lastIndex = 0，或每次 new RegExp。

陷阱 3：字符串里转义反斜杠要写两次

new RegExp("\\d")（字符串里 \d 要写成 \\d）；字面量 /\d/ 不需要。构造动态正则时用 String.raw 缓解：new RegExp(String.raw`\d+`)。

陷阱 4：用户输入拼正则未转义

new RegExp(userInput) → 用户输入 ( 直接 SyntaxError，输入 .+.+.+.+.+.+.+.+ 可触发 ReDoS。先用 userInput.replace(/[.*+?^${}()|[\]\\]/g, "\\$&") 转义。

陷阱 5：无 u flag 处理 emoji / CJK 大量 code point 时奇怪

/./ 匹配 surrogate 的一半；[a-z] 在无 u 时只按 UTF-16 code unit，非 BMP 字符行为怪异。统一加 u（或 v）。

陷阱 6：. 不匹配换行

/^.+$/ 对 "a\nb" 不整体匹配（. 不含 \n）。加 s flag 让 . 吃换行：/^.+$/s。

陷阱 7：\b 的定义是 \w↔\W 边界

\b 定义是 ASCII 字母/数字/下划线边界，对中文不生效：/\b中文\b/ 匹配 "a中文b" 但不匹配 " 中文 "。中文边界需自己构造 lookaround。

陷阱 8：replace 回调中的位置参数

"abc".replace(/b/, (match, offset, string) => ...)。有捕获组时参数变：(match, p1, p2, ..., offset, string, groups)。记错顺序返回乱码。

陷阱 9：$1 vs \1

替换字符串里用 $1（不是 \1）："ab".replace(/(a)/, "$1$1") === "aab"。模板字符串里 \\1 被当成 \1，无效。

陷阱 10：v flag 报错但 u 不报

v 把一些无意义的语法升级为 SyntaxError（如 [a-z] 以外的奇怪字符类）。升级到 v 要跑测试确认没吃到错误。
