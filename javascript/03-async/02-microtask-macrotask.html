微任务 vs 宏任务

"把异步 API 按队列分类"是面试高频题，但真正的工程价值是"选对异步 API"—— 在正确的时序执行回调。本章给出 10+ 常用 API 的完整归类表，讲清楚"尽快异步"、"下帧执行"、"浏览器空闲时跑" 三种需求对应什么 API。

阅读时长 ~18 分钟 前置 01 Event Loop 对应 javascript.info Event loop

01 速览

API 类型 最小延迟 典型用途
queueMicrotask(fn) microtask 本轮末尾 尽快异步 + 不跨轮 event loop
Promise.then/catch/finally microtask 本轮末尾 Promise 链
MutationObserver microtask DOM 变更后本轮 监听 DOM 变化
process.nextTick（Node） microtask（优先级高于 Promise） 本轮末尾 Node 特有
setTimeout(fn, 0) macrotask 至少 4ms（嵌套 5+） 延后到下一轮
setInterval macrotask 至少 4ms 重复 timer（少用，用 rAF）
setImmediate（Node） macrotask（check phase） 本轮 poll 后 Node 特有
MessageChannel + postMessage macrotask 0ms（不受 4ms 限制） "尽快下一轮"
requestAnimationFrame(fn) rendering callback 下帧前 动画、DOM 批量写
requestIdleCallback(fn) idle 阶段 浏览器空闲 非紧急后台任务
scheduler.postTask(fn, {priority}) macrotask（带优先级） 0ms 新 API，显式优先级
scheduler.yield() macrotask 0ms 让行给更高优先级

02 底层逻辑

怎么选："需要多快"

  1. 尽快但不跨事件循环 → queueMicrotask / Promise microtask。适合"合并 batch"、"确保回调异步"。
  2. 下一轮 task 但想绕过 4ms 最小延迟 → MessageChannel。React/Scheduler 早期用它做调度。
  3. 下一帧渲染前 → requestAnimationFrame。动画、批量写 DOM。
  4. 有空闲再跑 → requestIdleCallback。非关键后台（埋点上报、缓存清理）。
  5. 按优先级调度 → scheduler.postTask({ priority: "user-blocking" | "user-visible" | "background" })。

Microtask 的设计哲学

Promise 规范要求 .then 回调必须异步（避免 Zalgo）。但如果走和 setTimeout 同样的队列，会产生两个问题：

  • 被其他 task（如用户点击）抢先 → Promise 链中断感觉不连贯。
  • 多个 .then 之间穿插 UI 渲染 → "半成品"状态被用户看见。

Microtask 队列解决这两个问题：当前同步代码结束 → 清空 microtask → 渲染/下一轮 task。一个 Promise 链里的所有 .then 都在同一轮 event loop 里跑完，用户看不到中间态。

Node process.nextTick 的特殊性

process.nextTick 在 Node 里是 "micro-microtask" —— 优先级高于 Promise microtask：

javascript 
// Node.js
Promise.resolve().then(() => console.log("promise"));
process.nextTick(() => console.log("nextTick"));

// 输出：nextTick, promise
// nextTick 队列先被清空

历史原因：process.nextTick 在 Promise 规范前出现。现在 Node 文档推荐用 queueMicrotask 替代，除非真的需要"最高优先级"（极少）。

rAF vs rIC 的时序窗口

浏览器渲染周期里的关键时间点：

javascript 
// 每帧（~16.67ms @ 60Hz）：
//
// [前一帧结束] → 浏览器检查是否渲染
//                  ├─ task（一个）
//                  ├─ 清空 microtask
//                  ├─ rAF 回调          ← requestAnimationFrame
//                  ├─ resize / scroll observer
//                  ├─ IntersectionObserver
//                  ├─ style → layout → paint → composite
//                  └─ 渲染完毕
//               ↓
// [空闲时间]    → requestIdleCallback 执行（最多到帧末）

requestAnimationFrame(() => {
  // 在样式计算前：可以读取布局值不会触发 reflow
  // 可以安全写 style（会在本帧渲染）
});

requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0 && queue.length) {
    process(queue.shift());
  }
  if (queue.length) requestIdleCallback(...);
}, { timeout: 1000 });   // 1s 后必调度

MessageChannel 绕过 4ms 最小延迟

setTimeout(fn, 0) 在嵌套 5 层以上会被夹紧到 4ms。想要"纯 0ms 的下一轮 task"用 MessageChannel：

javascript 
const channel = new MessageChannel();
const port1 = channel.port1;
const port2 = channel.port2;
const queue = [];
port1.onmessage = () => {
  const fn = queue.shift();
  fn?.();
};
function yieldToMain(fn) {
  queue.push(fn);
  port2.postMessage(null);    // 触发 onmessage（下一轮 macrotask）
}

// React 18 的 scheduler、FastDOM 早期版本都用这个技巧
// Chrome 129+ 有原生 scheduler.yield() 替代

新的 Scheduler API

scheduler.postTask 和 scheduler.yield 是 Chrome 129+ 支持的新 API（Firefox 尚未）：

javascript 
// 3 种优先级
await scheduler.postTask(() => { /* ... */ }, { priority: "user-blocking" });
// user-blocking > user-visible > background

// yield 让行给更高优先级任务
async function processAll(items) {
  for (const item of items) {
    process(item);
    if (performance.now() % 50 < 1) await scheduler.yield();  // 让浏览器响应用户
  }
}

// 替代 setTimeout(fn, 0) 切片 —— 更智能、不受 4ms 限制

03 应用场景

批量订阅者通知（microtask 聚合）

javascript 
class Store {
  #state = {};
  #subs = new Set();
  #pending = false;
  set(k, v) {
    this.#state[k] = v;
    if (!this.#pending) {
      this.#pending = true;
      queueMicrotask(() => {
        this.#pending = false;
        this.#subs.forEach(fn => fn(this.#state));
      });
    }
  }
}
// 效果：同步连续 set 10 次 → 订阅者只被调用 1 次（本轮末尾）
// Vue 3、MobX 的批量更新机制核心是这个 pattern

动画 + DOM 批量写

javascript 
// 读写分离（FastDOM / fastdom 库思路）
let reads = [], writes = [];
let scheduled = false;
function schedule() {
  if (scheduled) return;
  scheduled = true;
  requestAnimationFrame(() => {
    reads.forEach(fn => fn());   // 先读布局值
    writes.forEach(fn => fn());  // 再写 DOM
    reads = []; writes = []; scheduled = false;
  });
}
const measure = (fn) => { reads.push(fn); schedule(); };
const mutate  = (fn) => { writes.push(fn); schedule(); };

// 使用
measure(() => { const h = el.offsetHeight; mutate(() => el.style.height = h + 20 + "px"); });

后台非关键任务（埋点上报）

javascript 
const analyticsQueue = [];
function trackEvent(evt) {
  analyticsQueue.push(evt);
  if (typeof requestIdleCallback !== "undefined") {
    requestIdleCallback(flush, { timeout: 5000 });
  } else {
    setTimeout(flush, 500);       // fallback（Safari）
  }
}
function flush(deadline) {
  while (analyticsQueue.length && (!deadline || deadline.timeRemaining() > 0)) {
    send(analyticsQueue.shift());
  }
  if (analyticsQueue.length) requestIdleCallback(flush);
}

React Scheduler 的时间切片

React 18 的 concurrent mode 用自研 Scheduler（之前靠 MessageChannel，现在兼容新 scheduler API）。每 5ms 让行一次，保持主线程响应性。理解这一章能帮你读懂 React 源码里的 shouldYield 逻辑。

替代 setTimeout(fn, 0) 做切片

javascript 
// ❌ 用 setTimeout 做切片：嵌套 5 层后变 4ms，累计时间过长
async function process1(items) {
  for (const item of items) {
    work(item);
    await new Promise(r => setTimeout(r, 0));
  }
}

// ✓ 用 scheduler.yield 或 MessageChannel
async function process2(items) {
  for (const item of items) {
    work(item);
    if (globalThis.scheduler) await scheduler.yield();
    else await new Promise(r => (new MessageChannel()).port1.onmessage = () => r(),
                                 (ch = new MessageChannel(), ch.port2.postMessage(null)));
  }
}

04 典型代码

时序判断题集

javascript 
// 题 1
async function f() {
  console.log("1");
  await null;                 // await 等价 Promise.resolve(null).then
  console.log("2");
}
f();
console.log("3");
// 输出：1, 3, 2
// 解释：await 把后面的代码移到 microtask；console.log(3) 是同步

// 题 2
Promise.resolve().then(() => console.log(1));
setTimeout(() => console.log(2));
requestAnimationFrame(() => console.log(3));
console.log(4);
// 输出：4, 1, （下帧前：3）, 2
// 解释：rAF 在下帧前、setTimeout 在下一轮 macrotask
// 注：具体顺序可能因浏览器 rendering 时机而异

// 题 3
const p = Promise.resolve();
p.then(() => console.log("A")).then(() => console.log("B"));
p.then(() => console.log("C")).then(() => console.log("D"));
// 输出：A, C, B, D
// 解释：两个 .then 交替入队（规范的链式处理）

queueMicrotask 实现

javascript 
// 老浏览器 polyfill
const queueMicrotaskPoly = typeof queueMicrotask === "function"
  ? queueMicrotask
  : (fn) => Promise.resolve().then(fn).catch(e => setTimeout(() => { throw e; }));

// ⚠ 错误处理差异：queueMicrotask 里抛错会变成 unhandled rejection
// polyfill 的 .catch 在老浏览器可能差异，上面用 setTimeout 重抛同步异常

requestIdleCallback polyfill

javascript 
// Safari 无原生 rIC（2026 仍如此）
const ric = globalThis.requestIdleCallback ?? ((fn) => setTimeout(() => {
  const start = Date.now();
  fn({
    didTimeout: false,
    timeRemaining: () => Math.max(0, 50 - (Date.now() - start)),
  });
}, 1));

智能 yield

javascript 
// 连续任务中每 N ms 让一次行
async function processWithYield(items, fn) {
  let lastYield = performance.now();
  for (const item of items) {
    fn(item);
    if (performance.now() - lastYield > 50) {
      if (globalThis.scheduler?.yield) await scheduler.yield();
      else await new Promise(r => setTimeout(r));
      lastYield = performance.now();
    }
  }
}

05 常见陷阱

setTimeout(fn, 0) 的最小 4ms

HTML 规范：嵌套 ≥5 层 setTimeout / setInterval 的最小延迟被 clamp 到 4ms。某些浏览器甚至对后台 tab 用 1s。性能敏感场景用 queueMicrotask（本轮末尾）或 MessageChannel（下一轮但不限 4ms）。

queueMicrotask 中抛错不会被 try/catch 捕获

microtask 回调的错误成为全局 uncaught exception（浏览器会派发 error 事件，Node 里触发 uncaughtException）。自己 wrap try/catch 或用 Promise 链处理。

rAF 回调在后台 tab 完全不跑

Chrome/Safari 后台 tab 的 rAF 停止。如果做"时间驱动"逻辑（倒计时、动画），要用 performance.now() 差值补帧，不要依赖 rAF 调用次数。document.visibilityState 检测可见性。

requestIdleCallback 的 deadline 不准

deadline.timeRemaining() 是估计值，实际可能被低估 → 回调跑太早结束；或高估 → 回调超出空闲时间导致下一帧掉帧。总是设置 timeout 选项作为最迟保障。

Promise 链中混入 setTimeout 改变时序

p.then(() => setTimeout(fn)).then(next) 中 next 不等 fn！setTimeout 返回 undefined，.then 立即继续。要等 timeout 结束：p.then(() => new Promise(r => setTimeout(r, 100))).then(next)。

Node process.nextTick "饿死" IO

process.nextTick 队列会在当前 phase 结束前完全清空 —— 如果回调里又加 nextTick，无限循环下 IO/定时器永远不跑。Node 文档警告：优先用 setImmediate 或 queueMicrotask。

浏览器的 throttling（省电 / 后台）

Chrome 对后台 tab / 省电模式做多种节流：setTimeout min 1s、setInterval 合并、rAF 停止、fetch 延迟。线上 RUM 数据可能受此影响；看 Performance panel 要过滤后台 tab 数据。

setTimeout(fn, n) 的 n 是"最早"而非"准确"

setTimeout 承诺"至少 n ms 后执行"，不是"恰好 n ms"。主线程繁忙时会延迟。动画/时间驱动逻辑一律用 performance.now() 差值计算，不要累加 n。
