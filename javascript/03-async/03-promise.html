Promise 状态机

Promise 是一个状态机：三个状态、单向转换、不可逆。理解它的规范算法（尤其是 resolve procedure 的 thenable 吸收）能让 Promise 链的一切行为都可预测 —— 包括 return Promise、.catch 位置、unhandled rejection 等高频坑点。

阅读时长 ~22 分钟 前置 01 Event Loop · P1-14 错误处理 对应 javascript.info Promises; Promise chaining

01 速览

javascript 
const p = new Promise((resolve, reject) => {
  // 执行器函数：同步执行
  if (ok) resolve(value);             // → fulfilled
  else reject(new Error("fail"));     // → rejected
});

p
  .then(v => v.upper)                 // 转换值
  .then(v => fetchMore(v))            // 返回 Promise，链自动等待
  .catch(err => fallback(err))        // 接住上游任意位置的 reject
  .finally(() => cleanup());          // 无论成败都执行；不改变值

// 快捷构造
Promise.resolve(value);               // 已 fulfilled 的 Promise
Promise.reject(error);                // 已 rejected 的 Promise（⚠ 立即形成 unhandled rejection）
Promise.withResolvers();              // ES2024：返回 { promise, resolve, reject }

三种状态

状态 含义 可转换到
pending 初始，未决 fulfilled / rejected
fulfilled 成功，有值 无（终态）
rejected 失败，有原因 无（终态）

状态一旦转换，不可逆转 —— 再调 resolve/reject 被静默忽略。这是 Promise "一次性"语义的由来。

02 底层逻辑

规范定义的内部槽

slot 存什么
[[PromiseState]] "pending" / "fulfilled" / "rejected"
[[PromiseResult]] fulfilled 的 value / rejected 的 reason
[[PromiseFulfillReactions]] fulfilled 时要触发的 .then 回调列表
[[PromiseRejectReactions]] rejected 时要触发的回调列表
[[PromiseIsHandled]] 是否已被 catch（决定 unhandled rejection）

Resolve Procedure：thenable 吸收

resolve(x) 不是简单地设 [[PromiseResult]] = x。规范流程：

  1. 如果 x 是 Promise 本身（自引用）→ 抛 TypeError。
  2. 如果 x 是一个 thenable（有 .then 方法的对象）→ 递归"跟随"它：等 x 决议后再决议当前 Promise。
  3. 否则 → 当作普通值，设 state = fulfilled, result = x。
javascript 
const inner = Promise.resolve(42);
const outer = new Promise(r => r(inner));    // resolve 了一个 Promise

outer === inner;                // false（是新对象）
outer.then(v => v);             // 42   ← 自动 unwrap

// thenable（不一定是 Promise）也被吸收
const fakeThenable = {
  then(onFul) { onFul("hello"); }
};
Promise.resolve(fakeThenable).then(v => v);   // "hello"

// 递归深度无限（层层剥到非 thenable）
Promise.resolve(Promise.resolve(Promise.resolve(5)))
  .then(v => v);                // 5  ← 自动扒三层

工程意义：.then 回调里 return anyValue 或 return somePromise 都会被上游正确处理，不需要手动区分。这是 Promise 链"自动扁平化"的根基。

.then 的规范语义

promise.then(onFulfilled, onRejected) 返回一个新 Promise，它的结果由回调决定：

  • 如果 promise fulfilled 且 onFulfilled 是函数 → 以 onFulfilled(value) 的返回值 resolve 新 Promise。
  • 如果 promise rejected 且 onRejected 是函数 → 以 onRejected(reason) 的返回值 resolve 新 Promise（注意不是 reject！）。
  • 如果回调抛错 → 新 Promise rejected。
  • 如果对应的回调不是函数 → 状态和值"透传"给新 Promise。
javascript 
// 透传
Promise.reject("e")
  .then(v => v.upper)       // onFulfilled 不调用（当前是 rejected）
  .then(v => v.upper)       // 仍 rejected
  .catch(e => "caught");    // → fulfilled("caught")

// onRejected 返回值 → fulfilled（而不是继续 rejected）
Promise.reject("e")
  .catch(e => "recovered")
  .then(v => v);            // "recovered"  ← 成功链继续

// onRejected 里再抛 → 继续 rejected
Promise.reject("e")
  .catch(e => { throw new Error("wrap"); })
  .then(v => "ok", err => err.message);  // "wrap"

.catch 和 .finally 是语法糖

写法 等价
p.catch(fn) p.then(undefined, fn)
p.finally(fn) p.then(v => (fn(), v), e => (fn(), Promise.reject(e))) 的近似（规范更严谨）

.finally 的关键特性：不改变值和状态。抛错时才会影响下游（以抛出的错误 reject）。

回调调度到 microtask

规范强制：.then 的回调永远异步执行，进入 microtask 队列。这是 Zalgo 防御：

javascript 
console.log("A");
Promise.resolve().then(() => console.log("B"));
console.log("C");
// 输出：A, C, B

// 即使 Promise 已经 resolved，.then 回调也是异步的

错误沿链传播

javascript 
fetch(url)
  .then(r => r.json())
  .then(data => process(data))
  .then(result => render(result))
  .catch(err => showError(err));   // ← 上面任一 .then 出错都到这里

// 这是 Promise 相比 callback 最大的优势：错误不会被"漏掉"
// 但要小心：.catch 之后的 .then 仍会执行（catch 恢复了成功态）

Unhandled Rejection 的触发

一个 rejected Promise 如果在本轮 microtask 清空时还没有 .catch（或等价的 .then 第二参），就会被标记为 unhandled：

  • 浏览器：派发 window.onunhandledrejection 事件。
  • Node.js：触发 process.on("unhandledRejection")，Node 15+ 默认崩溃进程。
  • 如果后来 .catch 了 → 触发 rejectionhandled / "rejectionHandled"。
javascript 
Promise.reject("boom");           // 本轮末尾 → unhandledrejection
// vs
Promise.reject("boom").catch(() => {});   // 本轮就 handle 了，不触发

// "延迟挂 catch" 仍然会先触发 unhandledrejection
const p = Promise.reject("x");
setTimeout(() => p.catch(() => {}), 100);
// 浏览器：触发 unhandledrejection → 100ms 后触发 rejectionhandled

03 应用场景

包装回调 API

javascript 
// 老式回调 API
function readFile(path, cb) { /* cb(err, data) */ }

// promisify
function readFileP(path) {
  return new Promise((resolve, reject) => {
    readFile(path, (err, data) => err ? reject(err) : resolve(data));
  });
}
// Node 原生 util.promisify(readFile)
// ES2024 的 Promise.withResolvers 更干净：
function readFileP2(path) {
  const { promise, resolve, reject } = Promise.withResolvers();
  readFile(path, (err, data) => err ? reject(err) : resolve(data));
  return promise;
}

链式扁平化（避免回调地狱）

javascript 
// ❌ 嵌套
fetchUser(id).then(user => {
  fetchOrders(user.id).then(orders => {
    fetchItems(orders[0].id).then(items => {
      render(user, orders, items);
    });
  });
});

// ✓ 扁平：return Promise → 链自动等待
fetchUser(id)
  .then(user => Promise.all([user, fetchOrders(user.id)]))
  .then(([user, orders]) => Promise.all([user, orders, fetchItems(orders[0].id)]))
  .then(([user, orders, items]) => render(user, orders, items));

// ✓ 最佳：async/await 可读性最好（下一章讲）

延迟器（deferred）模式

javascript 
// ES2024 Promise.withResolvers —— 外部控制 resolve/reject
function deferred() {
  return Promise.withResolvers();
  // 手写等价：
  // let resolve, reject;
  // const promise = new Promise((res, rej) => { resolve = res; reject = rej; });
  // return { promise, resolve, reject };
}

// 典型用法：事件触发时 resolve
const { promise, resolve } = Promise.withResolvers();
el.addEventListener("click", () => resolve("clicked"), { once: true });
await promise;     // 等待点击

finally 做资源清理

javascript 
showLoader();
fetch(url)
  .then(r => r.json())
  .then(renderData)
  .catch(showError)
  .finally(hideLoader);    // 成功/失败都 hide
// ⚠ finally 里抛错会把整个链变成 rejected

缓存 Promise（请求去重）

javascript 
const cache = new Map();
function fetchCached(url) {
  if (!cache.has(url)) cache.set(url, fetch(url).then(r => r.json()));
  return cache.get(url);
  // 同时触发多次 fetchCached(url)，后来的都等同一个 Promise
  // SWR/React Query 这类库的核心机制之一
}

// ⚠ 缓存失败 Promise 要清掉：
function fetchCached2(url) {
  if (!cache.has(url)) {
    const p = fetch(url).then(r => r.json())
      .catch(err => { cache.delete(url); throw err; });
    cache.set(url, p);
  }
  return cache.get(url);
}

04 典型代码

最小 Promise 实现（A+ 规范子集）

javascript 
class MyPromise {
  #state = "pending";
  #value;
  #reactions = [];

  constructor(executor) {
    const resolve = (v) => this.#settle("fulfilled", v);
    const reject = (r) => this.#settle("rejected", r);
    try { executor(resolve, reject); }
    catch (e) { reject(e); }
  }

  #settle(state, value) {
    if (this.#state !== "pending") return;   // 一次性
    // thenable 吸收
    if (state === "fulfilled" && value && typeof value.then === "function") {
      value.then((v) => this.#settle("fulfilled", v),
                 (r) => this.#settle("rejected", r));
      return;
    }
    this.#state = state;
    this.#value = value;
    queueMicrotask(() => this.#reactions.forEach((r) => r()));
  }

  then(onFul, onRej) {
    return new MyPromise((resolve, reject) => {
      const run = () => {
        try {
          if (this.#state === "fulfilled") {
            resolve(typeof onFul === "function" ? onFul(this.#value) : this.#value);
          } else {
            if (typeof onRej === "function") resolve(onRej(this.#value));
            else reject(this.#value);
          }
        } catch (e) { reject(e); }
      };
      if (this.#state === "pending") this.#reactions.push(run);
      else queueMicrotask(run);
    });
  }

  catch(fn) { return this.then(undefined, fn); }
}

链式返回的几种情况

javascript 
Promise.resolve(1)
  .then(v => v + 1)                          // 返回值 → 新 Promise fulfilled(2)
  .then(v => Promise.resolve(v + 1))         // 返回 Promise → 等待它 fulfilled(3)
  .then(v => { throw new Error(v); })        // 抛错 → 新 Promise rejected(Error(3))
  .then(v => "not called",
        e => e.message)                       // onRejected 处理 → fulfilled("3")
  .then(v => console.log(v));                 // "3"

串行执行数组

javascript 
// 顺序执行 Promise-returning 函数
function sequence(tasks) {
  return tasks.reduce(
    (p, task) => p.then(results => task().then(r => [...results, r])),
    Promise.resolve([]),
  );
}

sequence([() => fetch(a), () => fetch(b), () => fetch(c)])
  .then(results => /* ... */);

// 或 for-of + await（下一章）—— 可读性更好

给 Promise 加超时

javascript 
function withTimeout(promise, ms, err = new Error("timeout")) {
  return Promise.race([
    promise,
    new Promise((_, reject) => setTimeout(() => reject(err), ms)),
  ]);
}

await withTimeout(fetch(url), 5000);    // 5s 未响应就 reject
// ⚠ 这只是让 Promise 链 reject，原 fetch 仍在后台继续 —— 要真取消用 AbortSignal

05 常见陷阱

Promise 执行器同步运行

new Promise(fn) 的 fn 同步执行。new Promise(() => console.log(1)) 立即打印。常见误解："Promise 里的代码都是异步的" —— 不是，只有 .then 回调是。

.catch 后的 .then 仍会执行

.catch(fn) 如果不再抛错，返回 fulfilled Promise → 下游 .then 照常跑。想真正"终止链"就在 catch 里再 throw 或 return new Promise(() => {})（永远 pending）。

忘记 return 导致链失去追踪

.then(v => { doAsync(v); }) —— 没 return 导致 doAsync 返回的 Promise 脱离链。下游 .then 不等它，错误也不会进 .catch。规则：.then 回调里所有异步操作都要 return。ESLint 有 promise/always-return 规则。

Unhandled rejection（"floating" Promise）

忘写 .catch 的 rejected Promise → 本轮末尾 unhandledrejection。Node 15+ 默认崩溃进程。工程约定：要么 await，要么 .catch，要么 .then(null, errHandler)。ESLint no-floating-promises 能拦。

Promise 不是可取消的

race([promise, timeout]) 让链 reject，但原 Promise 仍在运行（网络请求不中断、定时器不 clear）。真正取消要用 AbortController（下一章详讲）。

async 函数里 throw 变 rejection

async function f() { throw new Error(); } 返回 rejected Promise，不是同步抛错。调用方用 try/await 或 .catch。

构造器里的 resolve/reject 多次调用被忽略

new Promise((res) => { res(1); res(2); }) —— 只记 1。规范规定。有时候看到代码里 if (ok) res(x); else rej(e); 不是 else 的作者加了防御性分支，其实"先 resolve 再 reject"是静默忽略。

循环 Promise 自引用

const p = new Promise((r) => r(p)); 抛 TypeError。规范明确禁止。同样 .then 里 return p 本身（当前的 thenable）也是 TypeError。

"Thenable"不是 Promise

任何有 .then 方法的对象都是 thenable，会被 resolve procedure 吸收。危险：某些库的对象不经意有 .then 方法会被当 Promise 处理，丢失原值。Promise.resolve(obj) === obj 的判断也只对"真" Promise 成立。
