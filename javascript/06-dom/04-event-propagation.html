事件传播三阶段

DOM 事件走 capture → target → bubble 三阶段。这三阶段决定了谁先收到事件、stopPropagation 有什么效果、currentTarget 和 target 差在哪。工程里几乎所有 "为什么我的 listener 没触发 / 触发了两次" 问题都可以追溯到这里。

阅读时长 ~20 分钟 前置 01 DOM 树 对应 javascript.info Bubbling and capturing

01 速览

事件三阶段

text 
HTML    <div id="outer">
            <div id="inner">
                <button>Click</button>  ← 用户点这里
            </div>
        </div>

Event 从 Window 往下传到目标再往上冒泡：

Capture: Window → document → html → body → #outer → #inner → (button 为目标)
Target:  button
Bubble:  button → #inner → #outer → body → html → document → Window

addEventListener 的完整签名

javascript 
el.addEventListener("click", handler, options);

// options：boolean（老版本）或对象
{
  capture: false,     // true 则在 capture 阶段触发，否则 bubble
  once: false,        // true 则触发一次后自动 remove
  passive: false,     // true 则 preventDefault 无效（但浏览器可更优化）
  signal: abortSig,   // 传入 AbortSignal 可用 controller.abort() 移除
}

事件对象核心属性

属性 含义
event.type "click" / "input" / ...
event.target 触发事件的原始元素（一直是实际点击的）
event.currentTarget 当前 listener 所在的元素
event.eventPhase 1=capture, 2=target, 3=bubble
event.bubbles 事件是否冒泡
event.cancelable 能否 preventDefault
event.defaultPrevented 是否已调用 preventDefault
event.composedPath() 返回完整传播路径数组（含 Shadow DOM）
event.timeStamp 事件发生时间戳（高精度）

02 底层逻辑

三阶段的真实执行顺序

javascript 
outer.addEventListener("click", () => console.log("outer capture"), true);
outer.addEventListener("click", () => console.log("outer bubble"));
inner.addEventListener("click", () => console.log("inner capture"), true);
inner.addEventListener("click", () => console.log("inner bubble"));
btn.addEventListener("click", () => console.log("btn target"));

// 用户点 button，输出：
// outer capture
// inner capture
// btn target             ← 目标阶段（capture 和 bubble 都触发）
// inner bubble
// outer bubble

// 规则：capture 先、从上往下；bubble 后、从下往上
// 同一元素注册的同类 listener 按注册顺序执行

target vs currentTarget

javascript 
outer.addEventListener("click", (e) => {
  e.target;          // 实际点击的元素（button）—— 不变
  e.currentTarget;   // outer（listener 所在）—— 在每个 listener 里不同
  // this === e.currentTarget（非箭头函数）
});

// 事件委托（05 章）的核心：靠 target 判断"被点击的是谁"
list.addEventListener("click", (e) => {
  const btn = e.target.closest("button");
  if (btn) handle(btn);
});

stopPropagation / stopImmediatePropagation

javascript 
// stopPropagation：停止传播到后续元素（本元素上的其他 listener 仍执行）
btn.addEventListener("click", (e) => {
  e.stopPropagation();
  // 同元素的下一个 listener 仍会跑
  // 但 outer.click 不再触发
});
btn.addEventListener("click", () => console.log("仍会跑"));
outer.addEventListener("click", () => console.log("不会跑"));

// stopImmediatePropagation：连本元素的后续 listener 也停
btn.addEventListener("click", (e) => e.stopImmediatePropagation());
btn.addEventListener("click", () => console.log("也不跑"));

// ⚠ stopPropagation 不阻止默认行为（链接仍跳转，表单仍提交）
// 要阻止默认：preventDefault

preventDefault 的范围

javascript 
// 阻止默认行为：表单提交、链接跳转、右键菜单、滚轮滚动、复选框切换
form.addEventListener("submit", (e) => {
  e.preventDefault();          // 不提交
  customSubmit();
});

link.addEventListener("click", (e) => {
  e.preventDefault();          // 不跳转
  showPreview(link.href);
});

// ⚠ passive listener 里 preventDefault 无效（下面讲）
el.addEventListener("touchstart", (e) => {
  e.preventDefault();          // 如果 passive: true，这里静默无效
}, { passive: false });        // 显式 passive: false 才允许

Passive listener：滚动性能救星

滚动事件（scroll、wheel、touchstart、touchmove）的 handler 可能调 preventDefault → 浏览器必须等 handler 跑完才能滚 → 卡顿。Passive 告诉浏览器"不会 preventDefault"，允许浏览器立即滚动并异步调用 handler：

javascript 
// 性能优化：告诉浏览器这个 handler 不阻止默认
el.addEventListener("wheel", onWheel, { passive: true });

// Chrome 的 touchstart/touchmove 默认 passive: true
// 想 preventDefault 必须显式
el.addEventListener("touchmove", onMove, { passive: false });

// 优先级
// - passive 未指定：浏览器根据元素类型决定（一般是 false）
// - passive: true：最好，浏览器优化
// - passive: false：明确要 preventDefault 时

once 与 signal

javascript 
// once：触发一次后自动 remove
el.addEventListener("click", handler, { once: true });

// signal：用 AbortController 一次性清空一组 listener
const ctrl = new AbortController();
el.addEventListener("click", h1, { signal: ctrl.signal });
el.addEventListener("input", h2, { signal: ctrl.signal });
document.addEventListener("keydown", h3, { signal: ctrl.signal });

ctrl.abort();       // 全部 remove —— 比手动 removeEventListener 三次简洁

// React useEffect cleanup 的现代模式：
useEffect(() => {
  const ctrl = new AbortController();
  document.addEventListener("keydown", onKey, { signal: ctrl.signal });
  element.addEventListener("scroll", onScroll, { signal: ctrl.signal });
  return () => ctrl.abort();
}, []);

composedPath 与 Shadow DOM

javascript 
// 获取事件传播的完整路径（从 target 到 Window）
el.addEventListener("click", (e) => {
  const path = e.composedPath();
  // [target, ...父元素一直到 Window]
});

// Shadow DOM 场景
// Shadow 内部的点击事件冒泡时，默认被"retarget"到 host 元素
// e.target 在外部看是 host，不是 shadow 内真实点击的
// e.composedPath() 可以看到完整路径（包含 shadow 内部节点）

// composed: true 的事件才跨越 shadow 边界
// click / focus 等是 composed；custom event 默认不是（下一章讲）

03 应用场景

模态框"点击外部关闭"

javascript 
modal.addEventListener("click", (e) => {
  if (e.target === modal) closeModal();     // 点到背景
  // 点到内容不触发（不是 target 本身）
});

// 或者用 closest 判断
modal.addEventListener("click", (e) => {
  if (!e.target.closest(".modal-content")) closeModal();
});

// 更现代：用 <dialog> 的原生 light-dismiss

全局快捷键 + ESC 关闭

javascript 
const ctrl = new AbortController();

document.addEventListener("keydown", (e) => {
  if (e.key === "Escape" && isModalOpen) closeModal();
  if ((e.ctrlKey || e.metaKey) && e.key === "k") {
    e.preventDefault();
    openCommandPalette();
  }
}, { signal: ctrl.signal });

// 组件卸载时
ctrl.abort();

捕获阶段做全局拦截

javascript 
// 用户点击 anchor 前拦截（路由框架）
document.addEventListener("click", (e) => {
  const a = e.target.closest("a[href]");
  if (!a || a.origin !== location.origin) return;
  if (e.metaKey || e.ctrlKey || e.button !== 0) return;  // 新标签页放行
  e.preventDefault();
  router.navigate(a.href);
}, true);   // 第三参 true = capture 阶段

// capture 让路由优先于页面自己的 click handler
// React Router / Next Link 就用这个模式

Passive listener 优化滚动监听

javascript 
// 监听滚动位置（如 scrollspy / lazy load）
let ticking = false;
window.addEventListener("scroll", () => {
  if (!ticking) {
    requestAnimationFrame(() => {
      updateScrollState(window.scrollY);
      ticking = false;
    });
    ticking = true;
  }
}, { passive: true });

// Lighthouse 会警告"非 passive 的 scroll listener"
// 浏览器开发者工具的 Performance 也能看到 handler 阻塞

同一元素多个 listener 的顺序

javascript 
el.addEventListener("click", h1);
el.addEventListener("click", h2);
el.addEventListener("click", h3);

// 按注册顺序调用：h1 → h2 → h3
// 如果 h1 里 stopImmediatePropagation，h2/h3 不执行
// 如果 h1 里 stopPropagation，h2/h3 仍执行（只阻止冒泡到 parent）

// 规范保证这个顺序，可以安全依赖

04 典型代码

带取消的 handler 注册

javascript 
function createEventManager() {
  const ctrl = new AbortController();
  return {
    on(target, event, handler, options = {}) {
      target.addEventListener(event, handler, { ...options, signal: ctrl.signal });
    },
    offAll() { ctrl.abort(); },
  };
}

// 使用
const events = createEventManager();
events.on(window, "resize", onResize);
events.on(document, "keydown", onKey);
events.on(button, "click", onClick);
// cleanup
events.offAll();

点击外部检测

javascript 
function onClickOutside(el, handler) {
  const listener = (e) => {
    if (!el.contains(e.target)) handler(e);
  };
  // 用 capture 确保先于内部 handler 触发
  document.addEventListener("click", listener, true);
  return () => document.removeEventListener("click", listener, true);
}

// 注意：el.contains 对 Shadow DOM 不穿透
// 如果 el 是 shadow host：
const listener = (e) => {
  const path = e.composedPath();
  if (!path.includes(el)) handler(e);
};

Event debounce

javascript 
function debounce(fn, ms) {
  let timer;
  return function (e) {
    clearTimeout(timer);
    // ⚠ event 对象有时被复用（老浏览器），同步访问属性
    const eventData = { target: e.target, type: e.type, timeStamp: e.timeStamp };
    timer = setTimeout(() => fn(eventData), ms);
  };
}

input.addEventListener("input", debounce(onType, 300));

dispatchEvent 触发自定义事件

javascript 
// 原生事件（click / input 等）
el.click();                              // 等同于触发 click（用户交互的模拟）

// 手动 dispatch
const event = new Event("click", { bubbles: true, cancelable: true });
el.dispatchEvent(event);

// 带数据的 custom event
const event = new CustomEvent("mycomponent:change", {
  detail: { value: 42 },
  bubbles: true,
});
el.dispatchEvent(event);
// 其他地方监听
document.addEventListener("mycomponent:change", (e) => console.log(e.detail.value));

05 常见陷阱

stopPropagation 破坏其他组件

第三方库在 document 级监听事件做路由 / 追踪 / 快捷键，你的 stopPropagation 可能让它失灵。滥用 stopPropagation 是团队事故源。优先用 e.target.closest 判断再处理。

preventDefault 不阻止 JS 副作用

preventDefault 只阻止浏览器默认行为。自定义 handler 里的代码仍会运行。想真正"取消后续处理"用 stopPropagation + 函数层面 return。

Passive 下 preventDefault 静默失败

addEventListener("touchmove", fn, { passive: true }) 里调 preventDefault 不报错但没效果。控制台会有警告。想阻止默认必须 passive: false。

React SyntheticEvent 与原生事件的 phase 不同

React 17 前：所有 SyntheticEvent 在 document 级一个 listener 统一分发（"顶层代理"）。17+ 改为 root 级。你直接在 DOM 上 addEventListener 的事件和 React 的 onClick 顺序复杂。避免混用：在 React 组件里用 React 事件系统。

移除 listener 必须函数引用相同

removeEventListener("click", () => ...) 不生效 —— 第二次写的箭头函数是新引用。先 const fn = () => ...; el.addEventListener(..., fn)，移除时也用 fn。signal 方式更省心。

Capture 和 bubble 要分别注册/移除

addEventListener(type, fn, true) 和 addEventListener(type, fn, false) 是两个独立的 listener。removeEventListener 的第三参必须匹配注册时的值。

事件在 target 阶段的 eventPhase

目标元素上既会触发 capture 又会触发 bubble 的 listener（eventPhase === 2），按注册顺序执行（不是 capture 先）。容易被"capture 总是先"的直觉误导。

异步 handler 里的 event 对象

现代浏览器里 event 对象可以安全地在 await 后继续访问属性（规范化后不再 reuse）。但如果需要 preventDefault，必须在同步代码里调 —— await 之后再 preventDefault 无效（浏览器已经处理完）。
