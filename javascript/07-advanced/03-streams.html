Streams API

Streams API 让 JS 以惰性、背压感知的方式处理任意大的数据流（文件、网络响应、媒体、子进程输出）。三种核心类型：ReadableStream（读）、WritableStream（写）、TransformStream（读写拼接，做映射/过滤）。本章讲清 reader/writer 的 lock 语义、背压如何自动协调生产速率、以及何时该用 stream 而非 ArrayBuffer。

阅读时长 ~25 分钟 前置 01 Binary · P6 Fetch 规范 WHATWG Streams

01 速览

  • 三类 stream：ReadableStream（可读源，如 fetch body）、WritableStream（可写目的，如下载保存）、TransformStream（双向，中间处理）。
  • 消费 readable：getReader() → read() 循环 / for await (const chunk of stream)（ES2024 原生 async iterable）/ pipeTo(writable)。
  • 背压（backpressure）：内部 queue 满时 controller.desiredSize ≤ 0，pull 暂停；writer ready Promise 让生产者等待下游消费完。
  • pipeThrough(transform) 返回 transform.readable，支持链式：fetch → decodeStream → gunzipStream → textStream。
  • 一个 ReadableStream 只能被一个 reader消费；需要多消费者用 stream.tee() 分成两条。
  • BYOB（bring your own buffer）ReadableStream 让消费者传入 Uint8Array，减少内存拷贝（仅 byte streams 支持）。

02 底层逻辑

ReadableStream 的生命周期

const stream = new ReadableStream({
  start(controller) {
    // 初始化（可选）
    controller.enqueue("first");
  },
  pull(controller) {
    // 消费者需要更多数据时调用；可异步
    // 当 controller.desiredSize <= 0 时不再调用 pull
    controller.enqueue("next");
    if (done) controller.close();
  },
  cancel(reason) {
    // 消费者放弃时调用（reader.cancel / pipeTo 中止）
    console.log("cleaned up:", reason);
  },
}, new CountQueuingStrategy({ highWaterMark: 3 }));
// ↑ 队列最多 3 个 chunk，满了暂停 pull

// 消费方式 A：getReader() 手动循环
const reader = stream.getReader();
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  console.log(value);
}
reader.releaseLock();

// 消费方式 B：async iteration（ES2024）
for await (const chunk of stream) console.log(chunk);

// 消费方式 C：pipe
await stream.pipeTo(new WritableStream({
  write(chunk) { console.log(chunk); },
}));

背压是如何自动协调的

Producer（pull 被调用）                     Consumer（read 被调用）
       │                                            │
       ↓ enqueue(chunk)                            │
    ┌──────────────────┐                           │
    │ internal queue    │ ←─ chunk 进入队列       │
    │ [c1][c2][c3]      │                           │
    └──────────────────┘                           │
       ↑                                            │
       │ desiredSize = highWaterMark - queueSize   │
       │ 若 <= 0 → 停止调用 pull                    │
       │ 若 >  0 → 继续调用 pull                    │
                                                    ↓
                                               read() 取走一个 chunk
                                               队列减少 → desiredSize > 0
                                               → 唤醒 pull

生产者和消费者都不需要显式协调。只要 pull 是 async 函数或返回 Promise，它完成前不会再次被调用；write 的 Promise 未 resolve 时上游自然等待。

TransformStream

// TransformStream 有两侧：writable（上游写入）+ readable（下游读取）
const upperCase = new TransformStream({
  transform(chunk, controller) {
    controller.enqueue(chunk.toUpperCase());
  },
  flush(controller) {
    // 所有输入结束后调用一次（可输出尾部 chunk）
  },
});

// 链式 pipe
await fetch("/text").then(r =>
  r.body
    .pipeThrough(new TextDecoderStream())  // Uint8Array → string
    .pipeThrough(upperCase)                // string → string
    .pipeTo(new WritableStream({
      write(chunk) { document.body.append(chunk); },
    })),
);

内置 Transform

Stream 作用
TextDecoderStream Uint8Array → string（按 UTF-8 等解码，跨 chunk 安全）
TextEncoderStream string → Uint8Array
CompressionStream("gzip" / "deflate" / "deflate-raw") 压缩（ES2023）
DecompressionStream("gzip" / "deflate") 解压 
// gzip 浏览器端压缩
const compressed = new Response(
  new Blob([largeText]).stream().pipeThrough(new CompressionStream("gzip"))
);
const gzBlob = await compressed.blob();
// 上传 gzBlob 减少 70% 体积

tee() 分流

const stream = await fetch("/data").then(r => r.body);
const [s1, s2] = stream.tee();

// s1 和 s2 是独立的 ReadableStream，共享同一个上游
// 每个 chunk 被内部复制成两份（会占内存，小心大数据）
await Promise.all([
  s1.pipeTo(saveStream),
  s2.pipeTo(parseStream),
]);

Response 与 ReadableStream 的互操作

// fetch body 是 ReadableStream
const response = await fetch("/file");
const reader = response.body.getReader();

// 自己构造 Response with stream（Service Worker 里常用）
return new Response(
  new ReadableStream({
    async pull(controller) {
      const data = await computeChunk();
      if (data) controller.enqueue(data);
      else controller.close();
    },
  }),
  { headers: { "Content-Type": "text/plain" } },
);

// 转换为其他格式
await response.blob();
await response.arrayBuffer();
await response.text();
await response.json();
// 以上任何一个方法会 lock body stream，不能再用 response.body

BYOB（Bring Your Own Buffer）

// 普通 ReadableStream 返回 Uint8Array，每个 chunk 是新分配的
// BYOB 让消费者提供 buffer，避免频繁分配

const stream = new ReadableStream({
  type: "bytes", // ← 关键
  pull(controller) {
    // controller 是 ReadableByteStreamController
    const view = controller.byobRequest?.view;
    if (view) {
      // 直接写入消费者提供的 buffer
      view.set([0x01, 0x02, 0x03]);
      controller.byobRequest.respond(3);
    } else {
      controller.enqueue(new Uint8Array([0x01, 0x02]));
    }
  },
});

const reader = stream.getReader({ mode: "byob" });
const buffer = new Uint8Array(1024);
const { value, done } = await reader.read(buffer);
// value 指向 buffer 的前 n 字节

03 应用场景

SSE / AI 流式响应

// 消费 SSE（参见 P6-04），但用 fetch + stream（可以传 header）
async function* sseLines(response: Response) {
  const reader = response.body!
    .pipeThrough(new TextDecoderStream())
    .getReader();
  let buffer = "";
  while (true) {
    const { value, done } = await reader.read();
    if (done) return;
    buffer += value;
    const chunks = buffer.split("\n\n");
    buffer = chunks.pop() ?? "";
    for (const raw of chunks) {
      const data = raw.split("\n").find(l => l.startsWith("data: "))?.slice(6);
      if (data && data !== "[DONE]") yield JSON.parse(data);
    }
  }
}

const response = await fetch("/api/chat", {
  method: "POST",
  headers: { Authorization: `Bearer ${token}` },
  body: JSON.stringify({ messages }),
});
for await (const event of sseLines(response)) {
  appendToken(event.delta);
}

带进度的下载

async function downloadWithProgress(url: string, onProgress: (pct: number) => void) {
  const response = await fetch(url);
  const total = Number(response.headers.get("Content-Length")) || 0;
  let received = 0;

  const stream = response.body!.pipeThrough(
    new TransformStream({
      transform(chunk, controller) {
        received += chunk.length;
        if (total) onProgress(received / total);
        controller.enqueue(chunk);
      },
    }),
  );

  return new Response(stream).blob();
}

大文件分行处理

// 100GB 日志文件，内存不爆
async function* readLines(stream: ReadableStream<Uint8Array>) {
  const reader = stream.pipeThrough(new TextDecoderStream()).getReader();
  let buf = "";
  while (true) {
    const { value, done } = await reader.read();
    if (done) { if (buf) yield buf; return; }
    buf += value;
    const lines = buf.split("\n");
    buf = lines.pop() ?? "";
    yield* lines;
  }
}

const fileStream = file.stream();
for await (const line of readLines(fileStream)) {
  if (line.includes("ERROR")) console.log(line);
}

流式 JSON 解析（NDJSON）

// API 返回 ndjson：每行一个 JSON，流式边收边解析
async function* parseNdjson<T>(response: Response): AsyncGenerator<T> {
  for await (const line of readLines(response.body!)) {
    const trimmed = line.trim();
    if (trimmed) yield JSON.parse(trimmed) as T;
  }
}

const res = await fetch("/api/search?stream=ndjson");
for await (const hit of parseNdjson<{ id: string; score: number }>(res)) {
  renderHit(hit);
}

SW 中合成响应

self.addEventListener("fetch", (event) => {
  if (!event.request.url.endsWith("/stream-demo")) return;

04 典型代码

从 AsyncIterable 构造 ReadableStream

// ES2024：ReadableStream.from() 静态方法
const numbers = ReadableStream.from(async function* () {
  for (let i = 0; i < 10; i++) { yield i; await delay(100); }
}());

// 兼容老环境的手写版本
function fromIterable<T>(iterable: AsyncIterable<T>): ReadableStream<T> {
  const iterator = iterable[Symbol.asyncIterator]();
  return new ReadableStream({
    async pull(controller) {
      const { value, done } = await iterator.next();
      if (done) controller.close();
      else controller.enqueue(value);
    },
    cancel(reason) { iterator.return?.(reason); },
  });
}

限速流（节流下载）

function throttleStream(bytesPerSecond: number) {
  let lastTime = Date.now();
  let bytesThisSecond = 0;

  return new TransformStream({
    async transform(chunk, controller) {
      bytesThisSecond += chunk.length;
      const now = Date.now();
      const elapsed = now - lastTime;
      const expected = bytesThisSecond / bytesPerSecond * 1000;
      if (elapsed < expected) await delay(expected - elapsed);
      if (elapsed >= 1000) {
        lastTime = now;
        bytesThisSecond = 0;
      }
      controller.enqueue(chunk);
    },
  });
}

await fetch(url).then(r =>
  r.body.pipeThrough(throttleStream(100_000)).pipeTo(writableStream),
);

Hash 流（SHA-256 流式计算）

// 浏览器原生 SubtleCrypto 不支持增量 hash，用 WASM
import { createSHA256 } from "hash-wasm";

async function hashStream(stream: ReadableStream<Uint8Array>) {
  const hasher = await createSHA256();
  const reader = stream.getReader();
  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    hasher.update(value);
  }
  return hasher.digest();
}

const hash = await hashStream(file.stream());

错误传播

const upper = new TransformStream({
  transform(chunk, controller) {
    if (typeof chunk !== "string") {
      controller.error(new TypeError("expected string")); // 传播错误
      return;
    }
    controller.enqueue(chunk.toUpperCase());
  },
});

try {
  await someReadable.pipeThrough(upper).pipeTo(someWritable);
} catch (err) {
  // 任何一环抛错都会触发 pipe 链中止，错误从 pipeTo 返回的 Promise reject
}

// pipeTo 选项
await source.pipeTo(dest, {
  preventClose: true,  // 源关闭不关闭目的
  preventAbort: true,  // 目的出错不 abort 源
  preventCancel: true, // 源 cancel 不 cancel 目的
  signal: ac.signal,    // AbortController 中止 pipe
});

05 常见陷阱

陷阱 1：getReader 后忘记 releaseLock

stream 被 reader 锁住后，不能再 pipeTo / tee。获取 reader 做完事必须 reader.releaseLock()，否则后续链式调用抛 TypeError。

陷阱 2：response.body 已被 locked

调用 response.json() 等任一 body 消费方法后，response.body 就被 locked，不能再 getReader。要流式处理先 clone：response.clone().body。

陷阱 3：tee 会在内存中缓冲

两个分支消费速率差很多时，快的那支等慢的那支，中间 chunk 留在内部 queue 里。慢消费者长期滞后会导致内存堆积。要保证两支消费速率相近，或用 BroadcastChannel / clones 替代。

陷阱 4：不理解背压导致生产端 OOM

自定义 ReadableStream 在 pull 中 for (i=0; i<1e9; i++) controller.enqueue(...) 忽略 desiredSize，内部 queue 无限增长。永远检查 controller.desiredSize 或用 async pull（天然等待）。

陷阱 5：Firefox 上 body async iteration 支持较晚

for await (const chunk of response.body) 在 Chrome/Safari 较早支持，Firefox 2024 年才稳定。兼容老环境用 getReader()。

陷阱 6：cancel 不等于 abort pipe

reader.cancel() 只告诉上游放弃，但 pipeTo 的整条管道中止要用 AbortController.signal。

陷阱 7：TransformStream 的 flush 容易被遗忘

实现行分割 transform 时，收到 transform 的最后一块可能没有 newline，剩余 buffer 要在 flush(controller) 里 enqueue，否则最后一行丢失。

陷阱 8：错把 Node.js 的 stream 当成 Web Streams

Node.js 有自己的 stream.Readable，API 完全不同（push/pipe/on）。Node 18+ 提供 Readable.fromWeb() / Readable.toWeb() 互转。SSR / API Route 代码两者都会遇到。

陷阱 9：pipeTo 默认会 close writable

source → dest pipe 完后默认调用 dest.close()。如果 dest 还要接收其他源的数据，传 { preventClose: true }。

陷阱 10：TextDecoderStream 的解码有状态

它会跨 chunk 缓冲未完整的 UTF-8 字节。不能把 TextDecoderStream 实例复用在两个独立流上（状态会污染）。每个 stream 都 new TextDecoderStream()。
