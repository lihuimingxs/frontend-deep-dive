this 四规则

this 不由"函数在哪定义"决定，而由"函数怎么被调用"决定。四条规则 + 一个例外（箭头函数）能解释 99% 的 this 行为。本章给出规则的优先级、判定算法、典型"this 丢失"的原因，以及 class field 箭头 vs 方法简写的内存差异。

阅读时长 ~20 分钟 前置 01 作用域 · 02 闭包 对应 javascript.info Object methods, "this"

01 速览

判定优先级（从高到低）

# 规则 形式 this 指向
0 箭头函数 () => ... 词法 this（外层作用域）
1 new 绑定 new Foo() 新建的对象
2 显式绑定 fn.call(obj) / .apply / .bind 传入的第一个参数
3 隐式绑定 obj.fn() 点号左边的对象
4 默认绑定 fn() 严格模式 undefined / 非严格 globalThis

箭头函数是例外 —— 它不参与这四条规则。它的 this 是定义时词法作用域里的 this，call/apply/bind 也改不了。

02 底层逻辑

规范视角：[[ThisMode]] 内部槽

每个函数对象在创建时被标记 [[ThisMode]]：

  • "lexical" —— 箭头函数。不会设置自己的 this，沿作用域链查。
  • "strict" —— 严格模式下的普通函数。调用时若无显式 this，设为 undefined。
  • "global" —— 非严格模式。无显式 this 时设为 globalThis。

call/apply/bind、new、obj.fn() 都是设置显式 this 的方式。没这三者之一时才走默认。

new 绑定的 4 步

new Foo(args) 实际发生：

  1. 创建一个新空对象 obj = Object.create(Foo.prototype)。
  2. 以 obj 作为 this 调用 Foo.call(obj, args)。
  3. 如果 Foo 返回对象，就用那个对象；否则用 obj。
  4. 结果作为 new Foo() 表达式的值。
javascript 
function User(name) {
  this.name = name;
  // 没显式 return
}
new User("Alice");             // { name: "Alice" }

function Weird() {
  this.x = 1;
  return { x: 2 };             // 显式返回对象 → 覆盖 this
}
new Weird();                   // { x: 2 }

function Normal() {
  this.x = 1;
  return 42;                   // 返回原始值 → 忽略
}
new Normal();                  // { x: 1 }

显式绑定：call / apply / bind

详细在下一章（P2-04）。这里记住一件事：箭头函数的 this 不能被这三个方法改写。

javascript 
const arrow = () => this;
arrow.call({ x: 1 });          // 不是 { x: 1 }，是外层 this

隐式绑定：点号左边即 this

javascript 
const user = {
  name: "Alice",
  greet() { return `Hi, ${this.name}`; },
};
user.greet();                  // "Hi, Alice"       ← this = user

// 链式时：最后一层决定
const a = { x: 1, b: { x: 2, get: function () { return this.x; } } };
a.b.get();                     // 2      ← this = a.b

方法"丢失 this"的本质

obj.fn() 的 this 是 obj。但提取 fn 再单独调用时，就是普通调用 → 走默认绑定：

javascript 
const user = { name: "Alice", greet() { return this.name; } };

const f = user.greet;
f();                           // undefined（严格）/ window.name（非严格）
                               // ← 这里没有点号左边的对象

setTimeout(user.greet, 100);   // 同样丢 this
[].forEach(user.greet);        // 同样

// 修复 1：包一层保留调用形式
setTimeout(() => user.greet(), 100);

// 修复 2：bind
setTimeout(user.greet.bind(user), 100);

// 修复 3：初始化时用箭头挂到实例（class field 模式）
class User {
  name = "Alice";
  greet = () => this.name;    // 箭头 + class field
}

默认绑定的严格 vs 非严格

javascript 
// 严格模式（ESM、class body、显式 "use strict"）
function f() { return this; }
f();                           // undefined

// 非严格
function g() { return this; }
g();                           // globalThis（浏览器 window）

// ⚠ 即使函数定义在严格模式，调用处的模式不重要
// this 由函数定义时的 [[ThisMode]] 决定

class field 箭头 vs 方法简写

这是一个内存/this 绑定的权衡：

javascript 
class A {
  greet = () => this.name;     // class field 箭头
  // 效果：每个实例自己的方法（this 永不丢）
  // 代价：每个实例都占一份内存
}

class B {
  greet() { return this.name; } // 方法简写
  // 效果：方法在 prototype 上，所有实例共享
  // 代价：this 需调用方维护（如 React 回调要 bind）
}

// React class 组件时代：回调稳定性优先 → field 箭头
// 性能敏感场景（高频实例化）：prototype 方法 + bind

03 应用场景

事件回调里的 this

javascript 
button.addEventListener("click", function (e) {
  this === button;             // true   ← DOM 规范：this 是触发元素
});

button.addEventListener("click", (e) => {
  this === button;             // false  ← 箭头：this 是外层（模块/window）
  e.currentTarget === button;  // true   ← 用 e.currentTarget 替代
});

数组方法的 thisArg

javascript 
// map/filter/forEach/find/some/every 都有第二参 thisArg
const ctx = { multiplier: 10 };
[1, 2, 3].map(function (n) { return n * this.multiplier; }, ctx);
// [10, 20, 30]

// ⚠ 箭头函数会忽略 thisArg
[1, 2, 3].map((n) => n * this.multiplier, ctx);   // this 仍是外层，thisArg 失效

// 现代写法：闭包捕获，不用 thisArg
const multiplier = 10;
[1, 2, 3].map(n => n * multiplier);

构造函数工厂

javascript 
// 保护防止忘记 new：new.target 检测
function User(name) {
  if (!new.target) return new User(name);  // 自动转成 new 调用
  this.name = name;
}
User("Alice");                 // 也能正确创建实例
new User("Alice");             // 也可以

// 或者用 class，class 强制 new
class User2 {
  constructor(name) { this.name = name; }
}
User2("Alice");                // TypeError: Class constructor cannot be invoked without 'new'

借用方法（call/apply）

javascript 
// 把 Array.prototype.slice 借给 Array-like
function toArray(arrayLike) {
  return Array.prototype.slice.call(arrayLike);
}
// 等价，但更推荐
Array.from(arrayLike);

// Object.prototype.toString 借来做精确类型检查
function typeOf(v) {
  return Object.prototype.toString.call(v).slice(8, -1);
}
typeOf([]);                    // "Array"
typeOf(new Date());            // "Date"

代理模式保留 this

javascript 
// 装饰器 / AOP：包装方法的同时保留 this
function withLog(fn) {
  return function (...args) {
    console.log("call", fn.name, args);
    return fn.apply(this, args);     // ← 用 apply 把 this 转发
  };
}

class Calc {
  add(a, b) { return a + b; }
}
Calc.prototype.add = withLog(Calc.prototype.add);
new Calc().add(1, 2);          // 正确 this（Calc 实例）

04 典型代码

四规则决策实例

javascript 
function fn() { return this; }

// 默认绑定
fn();                               // 严格 undefined / 非严格 globalThis

// 隐式绑定
const o = { fn };
o.fn();                             // o

// 显式绑定
fn.call({ x: 1 });                  // { x: 1 }

// new 绑定
new fn();                           // 新实例

// 显式 + new 冲突？new 优先级更高
const f = fn.bind({ x: 1 });
new f();                            // 新实例（bind 的 this 被忽略）

// 箭头无视所有规则
const arrow = () => this;
arrow.call({ x: 1 });               // 外层 this（不是 {x:1}）

多次 bind 只有第一次生效

javascript 
function fn() { return this; }
const f1 = fn.bind({ x: 1 });
const f2 = f1.bind({ x: 2 });       // 第二次 bind 被忽略
f2();                               // { x: 1 }

// 规范层：bind 返回的函数不能再被改变 this
// 实际工程：不要重复 bind

class 里 this 的稳定性

javascript 
class Counter {
  count = 0;

  // 方法简写：this 需调用方维护
  increment() { this.count++; }

  // class field 箭头：this 永远是实例
  decrement = () => { this.count--; };
}

const c = new Counter();
const inc = c.increment;
inc();                              // TypeError: this is undefined（严格）
const dec = c.decrement;
dec();                              // ✓ 正常工作

// React 16 class 组件时代的 onClick
class Btn extends Component {
  state = { count: 0 };

  // ❌ 方法简写：onClick={this.handle} 时 this 丢
  handle() { this.setState({ count: this.state.count + 1 }); }

  // ✓ field 箭头：稳定 this
  handle = () => this.setState({ count: this.state.count + 1 });
}

手动绑定自定义方法

javascript 
// constructor 里 bind —— React 15 时代常见
class App {
  constructor() {
    this.onClick = this.onClick.bind(this);   // 方法简写 + 显式 bind
    this.onChange = this.onChange.bind(this);
  }
  onClick() { /* this 安全 */ }
  onChange() { /* this 安全 */ }
}
// 缺点：构造器膨胀
// 优点：方法仍在 prototype 上，不占实例内存

05 常见陷阱

对象字面量里的箭头方法

const o = { name: "A", get: () => this.name }; —— this 不是 o，是外层（模块/window）。对象方法应用方法简写 get() { ... } 或普通 function 表达式。

对象方法的 this 是"最后一层点号"左边

a.b.c() 中 this = a.b，不是 a。嵌套对象方法引用外层的 state 需要保留根对象引用或用箭头 + 闭包。

setTimeout / setInterval 的 this

setTimeout(obj.method, 100) —— obj.method 被当成普通函数调用，this 丢失。修复：setTimeout(() => obj.method(), 100) 或 setTimeout(obj.method.bind(obj), 100)。

React 旧 class 组件的 onClick={this.handle}

React 把 handle 从实例提取，调用时 this 丢。ClassFields + 箭头是事实标准。函数组件 + hooks 已经不需要考虑这个问题 —— 是 class 组件转函数组件的强力理由之一。

this 在模块顶层是什么

<script>（传统）：this === window。
<script type="module"> / ESM：this === undefined。
CommonJS (Node.js): this === module.exports（不是 globalThis！）。

new 构造函数但忘记写 new

User("Alice")（忘 new）：严格模式下 this 是 undefined，this.name = ... 抛 TypeError。非严格模式下 this = window，在 window 上挂了 name！用 class（强制 new）或 new.target 检查防此坑。

数组方法的 thisArg 对箭头无效

arr.forEach((v) => this.x, ctx) —— 箭头不接收 thisArg，外层 this 无变化。要么用 function 表达式，要么用闭包变量代替。

class field 箭头方法的性能陷阱

每创建一个实例就复制一份箭头方法到实例属性上。上千实例 × 多方法 = 可观内存。且方法不在 prototype 上 → 子类 super.method() 访问不到。大规模实例化场景用 prototype 方法 + 显式 bind。
