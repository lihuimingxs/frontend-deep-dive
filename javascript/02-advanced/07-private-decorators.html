私有字段与装饰器

ES2022 加入的 #field 是真正的语言级私有（不是靠约定），实现上借助编译期的 PrivateName。Stage 3 装饰器（ES2025 候选、TypeScript 5.0+ 支持）重新设计了一套 context-based 装饰器模型，比早期 TypeScript experimentalDecorators 更干净。

阅读时长 ~22 分钟 前置 06 class 对应 javascript.info Private and protected properties

01 速览

私有字段 —— # 前缀

javascript 
class Counter {
  #count = 0;                           // 实例私有字段
  #notify;                              // 私有方法
  static #total = 0;                    // 静态私有字段

  constructor(notify) { this.#notify = notify; }

  #notifyChange() { this.#notify?.(this.#count); }     // 私有方法

  inc() {
    this.#count++;
    Counter.#total++;
    this.#notifyChange();
  }

  // 静态工厂里用 #private 访问另一实例的私有字段（同 class 内可见）
  static fromPair(a, b) {
    const c = new Counter();
    c.#count = a + b;
    return c;
  }
}

const c = new Counter();
c.inc();
c.#count;                // SyntaxError: 只在 class body 内合法
c["#count"];             // undefined（不是普通属性名）

Stage 3 装饰器 —— 4 种形式

javascript 
@logged                          // class 装饰器
class Service {
  @readonly VERSION = "1.0";     // field 装饰器
  @memoize                       // method 装饰器
  compute(x) { /* ... */ }
  @trace
  accessor value = 0;            // accessor 装饰器（配合 accessor 关键字）
}

02 底层逻辑

#field 的规范语义

#name 不是"字符串键"，而是私有名称（Private Name）—— 编译期为每个 #name 生成一个唯一 identity，运行时通过 PrivateFieldGet/PrivateFieldSet 操作访问，跳过所有普通属性机制。

  • 不存在于 this 作为普通属性：Object.keys、in、Reflect.ownKeys 都看不见它。
  • 不能动态访问：obj["#x"] 是读取名为 "#x" 的字符串键，返回 undefined。
  • 跨实例可见（同 class 内）：在 class body 里拿到另一个同类实例也能读它的 #field。
  • 不继承：子类不能访问父类的 #field（封装的核心意义）。

Brand check: #x in obj

想知道一个对象是不是"确实 这个 class 的实例"，最可靠的方法不是 instanceof（可绕过），而是检查它有没有这个 class 的某个私有字段：

javascript 
class Safe {
  #brand;
  static isSafe(obj) {
    return #brand in obj;               // ES2022：类内部可用的 brand check
  }
}

const real = new Safe();
const fake = Object.create(Safe.prototype);
Safe.isSafe(real);         // true
Safe.isSafe(fake);         // false   ← instanceof 会返回 true，但 brand 检测失败

和 Symbol "伪私有"的区别

Symbol key #private field
访问方式 obj[symbol] obj.#name（仅 class body 内）
外部能否读 能（拿到 Symbol 引用） 绝对不能
Reflect.ownKeys 可见 是 否
能挂非 class 对象 能 不能（只在 class 字段声明）
性能 普通属性访问 引擎有专门 slot 优化，常常更快

装饰器的签名

Stage 3 装饰器是函数：decorator(target, context)。context 是个对象，字段取决于装饰目标：

装饰目标 target context.kind 返回值
class class 本身 "class" 可返回新 class 替换
method 方法函数 "method" 可返回新函数替换
accessor { get, set } "accessor" 可返回新 { get, set, init? }
field undefined（无值） "field" 可返回 init 函数，对初始值做变换
getter/setter 对应函数 "getter"/"setter" 替换函数

context 还有 name、static（是否静态）、private（是否 #private）、access.get/set（取/赋值的"万能钥匙"）、addInitializer(fn)（注册 class 初始化完后运行的钩子）等。

装饰器应用顺序

  1. 成员装饰器按从里到外、从下到上应用：
    @a @b foo() {} 等价 a(b(foo))。
  2. 所有成员处理完后，才应用 class 装饰器。
  3. 最后依次运行 addInitializer 注册的钩子。

Legacy（TS < 5）vs Stage 3 装饰器

旧版 TypeScript 的装饰器是基于 Property Descriptor + reflect-metadata 的模型，签名完全不同：

  • Legacy：method(target, propKey, descriptor)；返回 descriptor。
  • Stage 3：method(fn, context)；返回函数或 undefined。

TypeScript 5.0+ 默认使用 Stage 3；显式 "experimentalDecorators": true 切回 legacy。NestJS、Angular、TypeORM 等重度依赖 legacy + reflect-metadata，暂不能升级；新项目推荐 Stage 3。

03 应用场景

封装内部状态

javascript 
class EventEmitter {
  #listeners = new Map();

  on(event, fn) {
    if (!this.#listeners.has(event)) this.#listeners.set(event, new Set());
    this.#listeners.get(event).add(fn);
    return () => this.#listeners.get(event)?.delete(fn);
  }

  emit(event, ...args) {
    this.#listeners.get(event)?.forEach(fn => fn(...args));
  }
}

// #listeners 对外完全不可见
// JSON.stringify(emitter) 是 "{}"，不泄漏

brand check 做类型守卫

javascript 
class ApiResponse {
  #brand;
  constructor(data) { this.data = data; }
  static is(v) { return typeof v === "object" && v !== null && #brand in v; }
}

if (ApiResponse.is(maybeRes)) {
  // TypeScript 里还可以 assert is ApiResponse
  maybeRes.data;
}

// 比 instanceof 更可靠（跨 realm 也可以，因为 #brand 的 identity 仍是同一个）

@memoize 装饰器（Stage 3）

javascript 
function memoize(fn, context) {
  if (context.kind !== "method") throw new Error("only for methods");
  const cache = new WeakMap();
  return function (...args) {
    const key = JSON.stringify(args);
    let byThis = cache.get(this);
    if (!byThis) { byThis = new Map(); cache.set(this, byThis); }
    if (byThis.has(key)) return byThis.get(key);
    const result = fn.apply(this, args);
    byThis.set(key, result);
    return result;
  };
}

class Math {
  @memoize
  factorial(n) { return n < 2 ? n : n * this.factorial(n - 1); }
}

@autobind 装饰器

javascript 
function autobind(fn, context) {
  if (context.kind !== "method") throw new Error("only methods");
  context.addInitializer(function () {
    this[context.name] = fn.bind(this);    // 在 class 初始化完后执行
  });
}

class Component {
  #state = { count: 0 };

  @autobind
  handleClick() { this.#state.count++; }
}

const c = new Component();
const h = c.handleClick;
h();                                       // this 已绑定，不丢

@deprecated 装饰器

javascript 
function deprecated(hint) {
  return function (fn, context) {
    if (context.kind !== "method") throw new Error("only methods");
    return function (...args) {
      console.warn(`[deprecated] ${context.name}: ${hint}`);
      return fn.apply(this, args);
    };
  };
}

class API {
  @deprecated("use fetchUsers")
  getUsers() { /* ... */ }
}

04 典型代码

私有方法与静态私有方法

javascript 
class Parser {
  #buffer = [];

  // 私有方法
  #flush() { const s = this.#buffer.join(""); this.#buffer.length = 0; return s; }

  // 静态私有方法
  static #cleanup(s) { return s.trim().toLowerCase(); }

  push(chunk) { this.#buffer.push(Parser.#cleanup(chunk)); }
  parse() { return this.#flush(); }
}

class 装饰器

javascript 
function singleton(Klass, context) {
  if (context.kind !== "class") throw new Error("only class");
  let instance;
  return class extends Klass {
    constructor(...args) {
      if (instance) return instance;
      super(...args);
      instance = this;
    }
  };
}

@singleton
class Config {
  value = 0;
}

new Config() === new Config();     // true

accessor 关键字 + 装饰器

javascript 
// accessor x = 0  等价于
// #x = 0; get x() { return this.#x; } set x(v) { this.#x = v; }

function logged(target, context) {
  if (context.kind !== "accessor") return;
  return {
    get() {
      console.log(`get ${context.name}`);
      return target.get.call(this);
    },
    set(v) {
      console.log(`set ${context.name} = ${v}`);
      target.set.call(this, v);
    },
    init(initialValue) {           // 初始化时的 hook
      return initialValue;
    },
  };
}

class Box {
  @logged accessor value = 0;
}
const b = new Box();
b.value;                            // get value
b.value = 10;                       // set value = 10

field 装饰器

javascript 
function capitalize(_, context) {
  if (context.kind !== "field") return;
  return function (initialValue) {      // 返回的 initializer 会替换字段的初始值
    return typeof initialValue === "string"
      ? initialValue.toUpperCase()
      : initialValue;
  };
}

class User {
  @capitalize name = "alice";
}
new User().name;                    // "ALICE"

05 常见陷阱

obj["#x"] 不等于 obj.#x

前者是字符串键 "#x"，undefined。#x 是语法结构（PrivateName），只能在 class body 内用点号语法访问。没有动态访问私有字段的方法 —— 这是设计目标。

Proxy 透传私有字段问题

ES2022 前，new Proxy(new Foo(), {}) 然后调用 foo 方法时访问 this.#field 会报 "cannot read private member"。ES2022 规范调整后，大多数情况已可正常工作。但若 Proxy 的 target 变了（不是原 instance），仍可能失败。详细见 P2-08。

static #field 在子类 this 访问失败

class A { static #x = 1; static get() { return this.#x; } }。class B extends A {}。B.get() 抛 TypeError —— 因为 this 是 B，不是 A，而 #x 只属于 A。规避：在静态方法里显式用 A.#x。

装饰器的顺序陷阱

@a @b method() {} 按"从下到上"应用 → 先 b(method) 然后 a(b(method))。和 Python / TypeScript 装饰器一致。但别按阅读顺序臆想。

addInitializer 的 this 是实例

context.addInitializer(function () { ... }) —— 回调里 this 是新实例，在所有字段初始化之后运行。是实现 @autobind 的正确位置。用箭头函数会丢 this。

Legacy TS 装饰器 ≠ Stage 3

从 legacy 升 Stage 3 不是无痛。NestJS/Angular/TypeORM 等框架目前只支持 legacy；自己写的装饰器也要按新签名重写。TypeScript 5.0+ 默认是 Stage 3，选 legacy 要 "experimentalDecorators": true。

装饰器 + 继承的微妙

父类方法上的装饰器产生的"包装函数"在子类的 super.method() 调用里仍生效（因为方法在 prototype 上）。但如果子类覆盖方法，父类的装饰器就绕过了。

static block 的变量不泄漏

class X { static { const tmp = ...; X.data = tmp; } } 里的 tmp 在块外不可见。static block 就是一个 class-scoped 的初始化块。
