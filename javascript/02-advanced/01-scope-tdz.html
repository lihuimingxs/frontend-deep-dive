作用域与 TDZ

作用域（scope）决定了"一个变量名能在哪里被访问"。JS 的三个声明关键字 var、let、const 对应三种不同的作用域策略，ES6 引入的 TDZ（Temporal Dead Zone）是一个很多人知其然不知其所以然的机制 —— 本章把它放回规范术语里讲清楚。

阅读时长 ~18 分钟 前置 P1 全部 对应 javascript.info The old "var"; Variable scope

01 速览

维度 var let const

💡 默认约定

默认 const；只在需要重新赋值时用 let；永远不要用 var（除非维护古老代码）。Lint 规则 prefer-const + no-var 能自动强制。

02 底层逻辑

Lexical Environment 规范结构

ECMAScript 规范把"作用域"建模为 Lexical Environment（词法环境），一个链式结构：

  • Environment Record —— 存变量绑定的哈希表（key 是变量名、value 是 binding）
  • outer —— 指向外层 Lexical Environment 的引用（形成作用域链）

变量查找（ResolveBinding(name)）在当前环境找不到就沿 outer 往上走，走到 global 还没有就抛 ReferenceError。

Environment Record 有两种：

  • Declarative Environment Record —— 函数/块/try/模块作用域。let/const/function/class 的绑定放这里。
  • Object Environment Record —— 全局作用域、with 语句。绑定挂在一个对象上（全局是 globalThis），所以 var x 在全局等同于 window.x = ...。let/const 在全局是 Declarative，不挂 window。

提升（Hoisting）的真实机制

"提升"不是"代码物理上被搬到顶部"，而是：进入一个作用域时，引擎先扫描该作用域里的所有声明，创建 binding；然后才执行代码。差异在"binding 的初始状态"：

声明形式 binding 初始状态 访问时
var x 已初始化为 undefined 得到 undefined
function f() {} 已初始化为函数 可调用
let x / const x 未初始化 抛 ReferenceError（TDZ）
class X {} 未初始化 TDZ
javascript 
console.log(a);      // undefined   ← var 提升且已初始化
console.log(b);      // ReferenceError ← let TDZ
console.log(c);      // ReferenceError
f();                 // "hi"         ← function 整体提升

var a = 1;
let b = 2;
const c = 3;
function f() { console.log("hi"); }

TDZ 的精确定义

TDZ = 从"进入作用域"到"执行到声明语句"之间的区域。在这段时间内，let/const 的 binding 存在但未初始化，任何读写都抛 ReferenceError。

javascript 
function f() {
  // 从这里开始 x 的 TDZ
  console.log(x);       // ReferenceError
  typeof x;             // ReferenceError  ← 注意！typeof 也不安全
  let x = 1;            // TDZ 结束
  console.log(x);       // 1
}

设计意图：让"变量声明前不能使用"成为硬规则，避免 var 时代的意外使用。是特性不是 bug。

块作用域的 4 种触发点

{ }、if/else/for/while 的 body（含 for 的 init 部分）、try/catch 的 catch 参数、switch 的 case block（需手动用 { }）。

javascript 
if (true) {
  let x = 1;
}
console.log(x);        // ReferenceError  ← 块外看不到

for (let i = 0; i < 3; i++) { }
console.log(i);        // ReferenceError

// ⚠ var 会逃逸
if (true) { var y = 1; }
console.log(y);        // 1  ← var 不管块作用域

// switch case 默认共享作用域
switch (x) {
  case "A": let a = 1; break;
  case "B": let a = 2; break;   // ❌ SyntaxError: already declared
}
// 修法：显式 { } 包裹
switch (x) {
  case "A": { let a = 1; break; }
  case "B": { let a = 2; break; }
}

for(let) 每次迭代的新绑定

P1-03 已讲过。规范上每次迭代创建一个全新的 Lexical Environment，旧值被复制进来。这就是为什么：

javascript 
const fs = [];
for (let i = 0; i < 3; i++) fs.push(() => i);
fs.map(f => f());            // [0, 1, 2]   ← 每次迭代 i 是独立 binding

const gs = [];
for (var j = 0; j < 3; j++) gs.push(() => j);
gs.map(g => g());            // [3, 3, 3]   ← var 全程共享一个 j

全局 var 的特殊行为

在全局作用域：var x = 1 同时在 globalThis（浏览器是 window）上挂了 x 属性。let/const 不会。这是历史遗留 + ES6 的兼容设计。

javascript 
// <script> 里
var a = 1;
let b = 2;
window.a;              // 1    ← var 挂了
window.b;              // undefined

// ESM 里（type="module" 或 .mjs）
// 顶层的 var/let/const 都不挂 window —— ESM 有独立模块作用域

03 应用场景

默认 const + 工具强制

约定：先用 const 写出来，lint 报"never reassigned"就留 const，lint 不报就改 let。ESLint prefer-const 自动检测。好处：

  • 阅读者不用怀疑变量是否被修改
  • 重构时减少意外 mutation
  • 引擎可做额外优化（V8 对 const 有专门优化）

块作用域做"局部工作"

javascript 
// 无需 IIFE，直接块作用域隔离
function setup() {
  // ... 主逻辑

  {
    const temp = computeExpensive();
    cache.set("key", temp);
    // temp 在块结束后可被 GC，不占外层作用域
  }

  // ...
}

const 对复合值只冻结绑定

javascript 
const arr = [1, 2, 3];
arr.push(4);         // ✓ 修改内容
arr[0] = 99;         // ✓ 修改元素
// arr = [5, 6];     // ✗ TypeError: 不能重新赋值

const user = { name: "Alice" };
user.age = 30;       // ✓
// user = {};        // ✗

// 想要内容也不可变：Object.freeze（浅）或 deepFreeze（见 P1-05）

循环里创建回调的 let 写法

javascript 
// ❌ var：所有 button 共享一个 i，打印的都是 buttons.length
for (var i = 0; i < buttons.length; i++) {
  buttons[i].onclick = () => console.log(i);
}

// ✓ let：每次迭代 i 是新的
for (let i = 0; i < buttons.length; i++) {
  buttons[i].onclick = () => console.log(i);
}

// ES6 前的常见 workaround：IIFE 捕获
for (var i = 0; i < buttons.length; i++) {
  ((i) => buttons[i].onclick = () => console.log(i))(i);
}
// 现在已不需要

04 典型代码

TDZ 的典型表现

javascript 
// 1. 自引用初始化
const x = x;                       // ReferenceError
// 原因：右侧 x 读取时还在 TDZ

// 2. 默认参数里的 TDZ
function f(a = b, b = 1) { }
f();                               // ReferenceError （b 在 a 初始化时还在 TDZ）
function g(b = 1, a = b) { }
g();                               // ✓

// 3. 参数作用域 vs 函数体作用域
let x = "outer";
function h(y = x) {                // x 这里指外层
  let x = "inner";
  return y;
}
h();                               // "outer"

函数声明 vs 表达式的提升

javascript 
foo();   // "declaration"
bar();   // TypeError: bar is not a function
baz();   // ReferenceError: Cannot access 'baz' before initialization

function foo() { console.log("declaration"); }
var bar = function () { console.log("var expression"); };
const baz = function () { console.log("const expression"); };

// 结论：
// - 函数声明：整体提升（可前调用）
// - var 表达式：只提升绑定，函数还未赋值（TypeError）
// - const/let 表达式：TDZ（ReferenceError）

作用域链的查找顺序

javascript 
const a = "global";
function outer() {
  const a = "outer";
  function inner() {
    const a = "inner";
    console.log(a);                // "inner"
  }
  inner();
  console.log(a);                  // "outer"
}
outer();
console.log(a);                    // "global"

// 查找：inner 作用域 → outer 作用域 → global 作用域
// 找到第一个匹配就停

const 的"引用不可变"

javascript 
const a = 1;
// a++;        // TypeError
// a = 2;      // TypeError

const arr = [];
arr.push(1);   // ✓
// arr = [2];  // TypeError

const fn = () => 1;
// fn = () => 2;   // TypeError
// 但可以挂属性（函数是对象）
fn.version = "1";  // ✓

05 常见陷阱

typeof 在 TDZ 中也抛错

typeof undeclaredVar 返回 "undefined"（P1-02 讲过）。但 typeof letVarInTDZ 抛 ReferenceError —— 因为 binding 存在但未初始化，不是"未声明"。细节陷阱。

命名冲突：let 同名参数

function f(x) { let x = 1; } 抛 SyntaxError —— 块作用域里 x 已经是形参，let x 视为重复声明。用不同名字或改赋值 x = 1。

全局 var 污染

脚本里 var x = 1 等于 window.x = 1，容易和浏览器 API 撞名（var name = "Alice" 会改 window.name 这个全局 string）。ESM 模块作用域避免此问题 —— 这也是"用 ESM"的核心理由之一。

严格模式下未声明赋值抛错

非严格模式：x = 1（没有 var/let）悄悄创建全局变量。严格模式（ESM 默认）：ReferenceError。这是严格模式最有价值的改变之一 —— 拦住"拼错变量名"的低级 bug。

var 在 try/catch 中的逃逸

try { throw 1; } catch (e) { var x = e; } console.log(x); —— x 在外面可见（var 不受块作用域限制），但 e 不可见（catch 参数是 lexical scope）。令人困惑的组合。用 let 避免。

class 和 let 一样有 TDZ

new Foo(); class Foo {} 抛 ReferenceError，不是"声明提升"。这防止了 class-before-declaration 的引用。function 是唯一"整体提升"的声明形式。

浏览器 <script> 和 ESM 的作用域差别

传统 <script>：共享一个全局作用域，var 跨文件可见。<script type="module">：每个模块独立作用域，顶层声明不互通。混用（传统脚本 + ESM）是调试噩梦的源头。

循环中的 const 语义

for (const x of arr) { } 合法 —— 每次迭代是新绑定，所以不算"重新赋值"。但 for (const i = 0; i < 3; i++) 抛 TypeError —— 因为 i++ 确实是重新赋值。C 风格 for 用 let，for-of/for-in 用 const。
