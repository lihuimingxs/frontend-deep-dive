闭包内存模型

闭包是 JS 最强大也最容易被误解的机制。一个实用的定义："函数 + 它创建时的词法环境"。本章用规范术语（[[Environment]] slot）拆解闭包的内存行为，讲 V8 的 context 分配优化，给出 debounce/throttle/memoize 的手写实现以及三种典型泄漏。

阅读时长 ~22 分钟 前置 01 作用域与 TDZ 对应 javascript.info Variable scope, closure

01 速览

javascript 
function makeCounter() {
  let count = 0;                       // 外部局部变量
  return {
    inc:  () => ++count,
    get:  () => count,
    reset: () => { count = 0; },
  };
}

const c = makeCounter();
c.inc(); c.inc();
c.get();           // 2
// count 在 makeCounter 返回后仍"活着" —— 三个内部函数都闭住了它
// 外部无法直接访问 count，只能通过返回的方法

这一个例子展示了闭包的 3 个核心特性：

  1. 函数可以"记住"定义时的作用域，即使外层函数已经返回。
  2. 捕获的是变量不是值：内部每次调用看到的都是 count 最新的状态。
  3. 天然的信息隐藏：外部拿不到 count，除非通过暴露的接口。

02 底层逻辑

[[Environment]] 内部槽

规范规定：每个函数对象都有一个内部槽 [[Environment]]，指向它被创建时所在的 Lexical Environment。函数调用时，新建的执行上下文的 outer 指针就是这个 [[Environment]] —— 作用域链由此延伸。

"闭包"不是一个特殊特性，而是"函数在被调用时能沿 [[Environment]] 访问外部作用域"的必然结果。

javascript 
function outer() {
  const x = 10;
  function inner() { return x; }    // inner.[[Environment]] 指向 outer 的 Lexical Environment
  return inner;
}

const f = outer();                   // outer 返回，调用栈弹出
// 但 outer 的 Lexical Environment 仍存活 —— 因为 f.[[Environment]] 指向它
f();                                 // 10

捕获的是"引用"不是"值"

关键结论：闭包保存的是对变量 binding 的引用。所以内部函数每次读取时看到的是变量的当前值。

javascript 
function make() {
  let x = 1;
  const read = () => x;
  const write = (v) => { x = v; };
  return { read, write };
}

const { read, write } = make();
read();                // 1
write(42);
read();                // 42  ← 看到了新值

垃圾回收的"能否释放"

闭包会"延长"外层作用域里被引用变量的生命周期。GC 的判断规则：

  • 只要闭包函数自身还被引用（变量持有、事件监听、Promise 链等），它的 [[Environment]] 就活着。
  • [[Environment]] 活着 → 外层作用域的 Environment Record 活着 → 里面的所有变量活着。
  • 一旦闭包函数不再被引用（null 赋值、事件 removeEventListener、Promise 终结），整条链才能被 GC。

这就是为什么事件监听器忘记 remove、setInterval 忘记 clear 会是经典内存泄漏模式。

V8 的 Context 分配优化

朴素实现会把整个外层作用域都保留。V8 做了优化：

  1. 只分配被捕获的变量 —— 编译期分析函数体引用了哪些外层变量，只这些进 "Context"（堆上的 Environment Record）。
  2. 未被捕获的变量进"栈"（函数返回就回收）。
  3. 同一外层产生多个闭包共享一个 Context（但每个闭包 [[Environment]] 指向它）。

工程启示：在闭包里访问大对象会让那个大对象被保留。即使你只用了其中一个小字段：

javascript 
function big() {
  const hugeData = await fetch("/api/dump").then(r => r.json()); // 100MB
  const userName = hugeData.user.name;

  return () => userName;    // ⚠ 看起来只用了 userName
  // 实际 V8 分析后只捕获 userName，hugeData 可以释放
  // 但某些复杂引用（eval、with、函数内部引用整个对象）会阻止此优化
}

// 保险做法：显式提取 + nullify
function big2() {
  let hugeData = fetchData();
  const userName = hugeData.user.name;
  hugeData = null;          // 显式释放引用，防止悲观捕获
  return () => userName;
}

闭包 ≈ 对象（另一种视角）

函数式编程名言："闭包是穷人的对象，对象是穷人的闭包"。两者在表达力上等价：

javascript 
// 闭包风格
function makeCounter() {
  let c = 0;
  return { inc: () => ++c, get: () => c };
}

// 类风格
class Counter {
  #c = 0;
  inc() { return ++this.#c; }
  get() { return this.#c; }
}

// 两者在大多数场景等价。闭包更轻量（无 prototype），
// class 更适合"多实例共享方法"（方法只一份，在 prototype 上）。

03 应用场景

Debounce（防抖）

连续触发时只在"停下来"后执行一次。输入搜索、窗口 resize、form 校验都用。

javascript 
function debounce(fn, ms = 300) {
  let timer = null;                       // ← 闭包捕获
  return function (...args) {
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => {
      fn.apply(this, args);
      timer = null;
    }, ms);
  };
}

const onSearch = debounce((q) => fetch(`/api/search?q=${q}`), 500);
input.addEventListener("input", (e) => onSearch(e.target.value));

Throttle（节流）

固定时间间隔最多执行一次。scroll 监听、mousemove、实时预览。

javascript 
function throttle(fn, ms = 100) {
  let last = 0;
  let timer = null;
  return function (...args) {
    const now = Date.now();
    const remaining = ms - (now - last);
    if (remaining <= 0) {
      if (timer) { clearTimeout(timer); timer = null; }
      last = now;
      fn.apply(this, args);
    } else if (!timer) {
      timer = setTimeout(() => {
        last = Date.now();
        timer = null;
        fn.apply(this, args);
      }, remaining);
    }
  };
}

const onScroll = throttle(() => updateHeader(), 100);
window.addEventListener("scroll", onScroll);

Memoize（记忆化）

javascript 
function memoize(fn, getKey = (...args) => JSON.stringify(args)) {
  const cache = new Map();
  return function (...args) {
    const key = getKey(...args);
    if (cache.has(key)) return cache.get(key);
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

const fib = memoize((n) => n < 2 ? n : fib(n - 1) + fib(n - 2));
fib(50);                                // 瞬时返回

// 注意：Map 永远增长 → 长期运行的进程要有 LRU 策略

模块私有状态（ESM 之前的经典模式）

javascript 
const Store = (function () {
  const state = new Map();         // 私有，外部无法访问
  const listeners = new Set();
  return {
    get(key) { return state.get(key); },
    set(key, v) { state.set(key, v); listeners.forEach(l => l(key, v)); },
    subscribe(fn) { listeners.add(fn); return () => listeners.delete(fn); },
  };
})();

Store.set("user", { name: "Alice" });
// Store.state;   // undefined   ← 拿不到内部

ESM 有"模块作用域"后，这个模式被简化为普通的模块变量 + 命名导出。但闭包模式在单文件脚本、UMD 包里仍常见。

工厂函数 / Currying

javascript 
// 柯里化：把多参函数变成链
const add = (a) => (b) => (c) => a + b + c;
add(1)(2)(3);        // 6

// 特化版本
const addTen = add(10);
addTen(5)(3);        // 18

// 工厂：返回预配置的函数
const makeLogger = (tag) => (msg) => console.log(`[${tag}] ${msg}`);
const logInfo = makeLogger("INFO");
logInfo("hello");    // [INFO] hello

04 典型代码

只执行一次

javascript 
function once(fn) {
  let called = false;
  let result;
  return function (...args) {
    if (!called) {
      called = true;
      result = fn.apply(this, args);
    }
    return result;
  };
}

const init = once(() => expensiveInit());
init(); init(); init();    // expensiveInit 只跑一次

有状态的生成器

javascript 
function idGenerator(prefix = "id") {
  let seq = 0;
  return () => `${prefix}-${++seq}`;
}

const nextId = idGenerator("user");
nextId();     // "user-1"
nextId();     // "user-2"

// 也适合做"限量消费"等状态容器
function takeN(arr, n) {
  let i = 0;
  return () => i < n ? arr[i++] : undefined;
}

简易 observer

javascript 
function createSignal(initial) {
  let value = initial;
  const subs = new Set();
  return {
    get: () => value,
    set: (v) => { value = v; subs.forEach(fn => fn(v)); },
    subscribe: (fn) => { subs.add(fn); return () => subs.delete(fn); },
  };
}

const count = createSignal(0);
const unsub = count.subscribe(v => console.log("changed:", v));
count.set(1);       // changed: 1
count.set(2);       // changed: 2
unsub();
count.set(3);       // 不再触发

Partial application

javascript 
function partial(fn, ...presetArgs) {
  return function (...laterArgs) {
    return fn.apply(this, [...presetArgs, ...laterArgs]);
  };
}

const greet = (greeting, name) => `${greeting}, ${name}!`;
const hello = partial(greet, "Hello");
hello("Alice");                // "Hello, Alice!"

// 原生等价：fn.bind(null, ...presetArgs)（见下一章 04）

05 常见陷阱

循环里用 var + 闭包共享变量

经典面试题 for (var i; ...) setTimeout(() => i) 全部打印同值。规避：用 let（每迭代新绑定）。详见 P1-03 和 P2-01。

DOM 节点的闭包泄漏

const el = document.getElementById("x"); el.onclick = () => el.foo; —— 回调通过闭包持有 el。即使 el 从 DOM 移除，只要事件回调还在挂着，el 就不会被 GC。经典 React 组件卸载时忘 removeEventListener 的根因。

setInterval 忘记 clear

const timer = setInterval(() => { /* 用 state */ }, 1000); —— 即使组件卸载 / 页面切换，定时器仍持有 state 引用。React/Vue 的 useEffect 的 cleanup / onUnmounted 必须 clear。Node.js 里 server 重启间的大对象保留同理。

console.log 的"延迟求值"

console.log(obj); obj.x = 1; 在某些 DevTools 里展开时显示的是展开时的值（因为 log 保留引用）。排查 async 状态要 console.log({...obj}) 或 structuredClone(obj) 固定快照。

过多闭包造成性能压力

每个闭包都有 [[Environment]] 分配 + 可能的 context 堆对象。在高频循环（如 60 fps 的 animation loop）里创建大量短生命周期的闭包会增加 GC 压力。解决：提到循环外创建一次复用，或用普通函数 + 参数传递。

eval 或 with 会阻止闭包优化

V8 只能在"函数体里能静态分析出用到哪些变量"的前提下优化。eval("x") 或 with (obj) { ... } 让静态分析失效，引擎只能悲观保留整个作用域。永远不用 eval / with。

忘记解绑订阅

store.subscribe(listener) 里的 listener 通过闭包持有组件 state。不调用 unsubscribe 就会泄漏。统一用 const unsub = store.subscribe(...); return unsub（React useEffect cleanup、Vue watch effect 清理）。

共享可变 state 的竞态

function make() { let s = []; return () => s; } —— 多次调用返回同一个数组！多个消费者互相影响。需要"独立实例"就每次在工厂函数里 return () => []，每次调用返回新的。
