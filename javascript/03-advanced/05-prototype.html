原型链

JS 是基于原型的面向对象语言。class 是 ES6 加的语法糖 —— 底下仍然是原型链。理解这一章以后，class 的行为（继承、方法查找、instanceof、super）就全是常识。

阅读时长 ~22 分钟 前置 P1 对象、函数 对应 javascript.info Prototypal inheritance

01 速览

两个名字很像、实质完全不同的东西：

[[Prototype]] 内部槽 prototype 属性
谁有？ 所有对象都有 只有 函数（构造器）才有
指向什么 对象的"父原型" 一个被 new fn() 时设为实例 [[Prototype]] 的对象
访问方式 Object.getPrototypeOf(o) / o.__proto__（弃用但可用） fn.prototype（直接属性）
修改 Object.setPrototypeOf（性能差）/ Object.create（构造时定） 直接赋值 fn.prototype = ...
javascript 
function Animal() {}
Animal.prototype.speak = function () { return "sound"; };

const a = new Animal();

a.[[Prototype]]       === Animal.prototype;  // true  （内部槽关系）
Object.getPrototypeOf(a) === Animal.prototype;  // true
Animal.prototype.[[Prototype]]  === Object.prototype;
Object.prototype.[[Prototype]]  === null;    // 链尽头

// 属性查找：a.speak 先查 a 自有 → 再查 Animal.prototype → 再查 Object.prototype → null
a.speak();            // "sound"

02 底层逻辑

原型链查找算法

obj.prop 读取时：

  1. 如果 obj 自有 prop → 直接返回。
  2. 否则查 obj.[[Prototype]]，递归步骤 1。
  3. 直到某一层是 null → 返回 undefined。

写入 obj.prop = v 行为不同：总是在 obj 自身上设置（除非原型链上有 setter 或 prop 不可写）。这叫 "property shadowing"（属性遮蔽）。

构造对象的 3 种方式 → 原型来源

方式 实例的 [[Prototype]]
new Foo() Foo.prototype
Object.create(proto) proto（可为任意对象或 null）
{ ... } 字面量 Object.prototype
[] 字面量 Array.prototype
new class extends Base { } 实例 子类 prototype → Base.prototype → ...

instanceof 的规范语义

obj instanceof Klass 的规范伪代码（忽略 Symbol.hasInstance 分支）：

javascript 
function instanceOf(obj, Klass) {
  const target = Klass.prototype;
  let proto = Object.getPrototypeOf(obj);
  while (proto !== null) {
    if (proto === target) return true;
    proto = Object.getPrototypeOf(proto);
  }
  return false;
}

换句话说，instanceof 沿 obj 的原型链找 Klass.prototype。这有两个启示：

  1. 跨 realm 时，不同 iframe 的 Array 构造器 .prototype 不是同一个对象 → arr instanceof Array 可能 false。用 Array.isArray(arr) 规避。
  2. 可以自定义：给 class 加 static [Symbol.hasInstance] 方法，替换内置逻辑（见 P1-11 Symbol）。

constructor 属性的约定

每个 fn.prototype 上默认有个 constructor 属性指回 fn。这是"约定"，不是"规则"：你能覆盖或删掉（但最好别）。

javascript 
function F() {}
F.prototype.constructor === F;       // true

// 直接赋值 prototype 会丢
F.prototype = { sayHi() { return "hi"; } };   // ❌ constructor 链断了
new F().constructor === F;            // false（变 Object）

// 正确做法：扩展而不是替换
Object.assign(F.prototype, { sayHi() { return "hi"; } });

__proto__ 是弃用 getter/setter

obj.__proto__ 在 ES6 前是实现细节，ES6 在附录 B（Web 兼容）中写进规范保留。不要直接用：

  • 使用读：Object.getPrototypeOf(obj)
  • 使用写：Object.setPrototypeOf(obj, proto)（但性能差：V8 会去优化整个对象）
  • 创建时指定：Object.create(proto)（推荐，无性能惩罚）

原型污染（prototype pollution）

如果攻击者能控制任意对象的 __proto__ 键赋值 → 污染 Object.prototype → 影响所有对象：

javascript 
// 恶意 JSON
const dirty = JSON.parse('{"__proto__": {"admin": true}}');
// JSON.parse 本身不会污染（__proto__ 作为字符串 key 是 own property）
// 但如果你再深拷贝 / merge：
const target = {};
Object.assign(target, dirty);          // 这里不会污染 Object.prototype（assign 只拷自有属性）

// 真正的坑：自己实现的"递归 merge"
function merge(a, b) {
  for (const k in b) {
    if (typeof b[k] === "object") merge(a[k] ??= {}, b[k]);
    else a[k] = b[k];
  }
}
merge({}, dirty);
({}).admin;                             // true  ← 所有 {} 都"继承"了 admin！

防御：

  • merge 时跳过 __proto__ / constructor / prototype 键
  • 用 Object.create(null) 建"无原型对象"作字典 key → __proto__ 是普通字符串 key，不污染
  • 现代库（lodash.merge、structuredClone）已加防护

03 应用场景

方法共享节省内存

实例数多时，把方法挂 prototype 让所有实例共享（而不是每个实例一份）。class 的"方法简写"默认是这个行为。

javascript 
// ❌ 每个实例一份 method
function User(name) {
  this.name = name;
  this.greet = function () { return `Hi, ${this.name}`; };    // 每个实例独立
}

// ✓ 共享一份
function User(name) { this.name = name; }
User.prototype.greet = function () { return `Hi, ${this.name}`; };

// 1000 个实例：前者分配 1000 个 greet 函数，后者只 1 个
// class 等价物：class User { greet() {...} }

Record / Dictionary: Object.create(null)

javascript 
const dict = Object.create(null);
dict["toString"] = 1;             // 普通赋值，不和 Object.prototype.toString 冲突
"hasOwnProperty" in dict;         // false
// 但：dict.hasOwnProperty(k) 抛 TypeError（没继承 Object.prototype）
Object.hasOwn(dict, k);            // ✓ 用这个

Mixin（多重继承模拟）

javascript 
const Serializable = {
  serialize() { return JSON.stringify(this); },
};
const Comparable = {
  equals(other) { return JSON.stringify(this) === JSON.stringify(other); },
};

class User { /* ... */ }
Object.assign(User.prototype, Serializable, Comparable);

new User().serialize();          // ✓

// 更现代：用 class factory
const WithSerialize = (Base) => class extends Base {
  serialize() { return JSON.stringify(this); }
};
class User extends WithSerialize(Object) { }

Object.create 做"继承"

javascript 
// ES5 时代的原型继承
function Animal() {}
Animal.prototype.eat = function () { };

function Dog() { Animal.call(this); }
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;       // 修回 constructor
Dog.prototype.bark = function () { };

new Dog().eat();                       // ✓
new Dog() instanceof Animal;           // true

// 现代直接用 class extends，行为等价

Null-prototype 避免原型污染

处理用户输入 JSON 后若需要作为"字段名 key"的对象，先转成 null-prototype：

javascript 
function safeObject(src) {
  const out = Object.create(null);
  for (const k of Object.keys(src)) {
    if (k === "__proto__" || k === "constructor") continue;
    out[k] = src[k];
  }
  return out;
}

04 典型代码

原型链可视化

javascript 
function printChain(obj) {
  let current = obj;
  let depth = 0;
  while (current !== null) {
    console.log(`${" ".repeat(depth * 2)}→`, current.constructor?.name ?? "(no constructor)", current);
    current = Object.getPrototypeOf(current);
    depth++;
  }
}

class A {}
class B extends A {}
printChain(new B());
// → B { }
//   → A { constructor: A }
//     → Object { constructor: Object, ... }
//       → null

hasOwn 的正确方式

javascript 
// ❌ 在 null-prototype 对象上崩
const o = Object.create(null);
o.x = 1;
o.hasOwnProperty("x");                     // TypeError

// ❌ 被覆盖时失效
const evil = { hasOwnProperty: () => "LOL" };
evil.hasOwnProperty("x");                  // "LOL"

// ✓ ES2022 Object.hasOwn
Object.hasOwn(o, "x");                     // true
Object.hasOwn(evil, "x");                  // false（正确）

// ✓ 老项目中的安全写法
Object.prototype.hasOwnProperty.call(o, "x");

Object.create vs setPrototypeOf

javascript 
// ✓ 创建时指定 —— 无性能代价
const a = Object.create(proto);

// ✓ 作为字面量 + null 原型
const dict = Object.create(null);

// ⚠ 事后修改 —— V8 会去优化（de-optimize）所有依赖该对象形状的代码
const o = {};
Object.setPrototypeOf(o, proto);     // 尽量避免

// 另一个弃用写法，性能同 setPrototypeOf：o.__proto__ = proto

自定义 instanceof

javascript 
// 让数字"属于"Even 类（鸭子类型）
class Even {
  static [Symbol.hasInstance](v) {
    return typeof v === "number" && v % 2 === 0;
  }
}
2 instanceof Even;                   // true
3 instanceof Even;                   // false
"2" instanceof Even;                 // false

// 跨 realm 的兼容性 check
class SerializedError {
  static [Symbol.hasInstance](v) {
    return v && typeof v.name === "string" && typeof v.message === "string";
  }
}

05 常见陷阱

赋值 fn.prototype = { ... } 丢 constructor

整块替换会让 prototype.constructor 指回 Object。用 Object.assign(fn.prototype, { ... }) 或 class 语法。

修改内置 Array.prototype 等

Array.prototype.myMethod = ... 会污染所有数组的 for-in、in、第三方代码。永远不扩展内置原型。要加通用工具就写独立函数或用 class wrapper。

for-in 爬原型链

for (const k in obj) 包含继承的可枚举属性。看起来干净的对象可能有意外键。要只遍历自有属性：for (const k of Object.keys(obj))。详见 P1-03。

跨 iframe / realm 的 instanceof 失败

父文档的 Array 和 iframe 的 Array 是不同构造器 —— 它们的 prototype 不相等 → iframeArray instanceof Array 是 false。Array.isArray 用特殊的内部 brand check，跨 realm 可靠。

原型链上的 setter 屏蔽赋值

写 obj.x = 1 时，如果 obj 自身没 x 但原型链上有 setter，调的是原型链上的 setter，不会在 obj 上创建属性。如果原型链上有 writable: false 的数据属性，严格模式下赋值抛错。

箭头函数没有 prototype 属性

(() => {}).prototype 是 undefined。不能 new。如果打算让函数作为构造器，用 function 或 class。

Object.setPrototypeOf 的性能

V8 对"稳定 shape"对象有 hidden class 优化；改 prototype 会让所有缓存失效。热点代码路径上一次 setPrototypeOf 能让整个函数 de-opt。建议只在对象创建时通过 Object.create 定。

子类 static 方法的继承

class B extends A {} 时，B.someStaticMethod 的查找也走原型链（B.__proto__ === A）。大多数人以为 static 方法不继承 —— 实际是会继承的，通过 class 本身的原型链。
