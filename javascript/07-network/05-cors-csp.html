CORS 与 CSP

浏览器两大安全防线：CORS（Cross-Origin Resource Sharing）控制能不能读跨源响应；CSP（Content Security Policy）控制能不能加载/执行各种资源。这两个 header 配置错一个字母，生产就挂。本章彻底讲清规则。

阅读时长 ~25 分钟 前置 01 Fetch MDN CORS / CSP

01 速览

同源策略基础

"同源" = 协议 + 域名 + 端口全部相同。

页面 目标 同源？
https://a.com/page https://a.com/api ✓
https://a.com http://a.com ✗（协议不同）
https://a.com https://b.com ✗（域名不同）
https://a.com https://a.com:8080 ✗（端口不同）
https://sub.a.com https://a.com ✗（子域名不同）

CORS 的 4 类响应 header

Header 含义
Access-Control-Allow-Origin 允许的源（* 或具体域名）
Access-Control-Allow-Credentials true = 允许带 cookie
Access-Control-Allow-Methods 预检时允许的方法
Access-Control-Allow-Headers 预检时允许的自定义 header
Access-Control-Expose-Headers JS 能读的响应 header（默认只 6 个安全的）
Access-Control-Max-Age 预检结果缓存时间（秒）

02 底层逻辑

简单请求 vs 预检请求

简单请求（不触发预检）必须满足：

  • 方法：GET / HEAD / POST
  • Content-Type：application/x-www-form-urlencoded / multipart/form-data / text/plain
  • 无自定义 header（除规范允许的几个）
  • 没有 ReadableStream body

不符合就是预检请求：浏览器先发 OPTIONS 请求询问服务器是否允许：

http 
预检请求（浏览器自动发）：
OPTIONS /api/data HTTP/1.1
Origin: https://mysite.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, X-Token

预检响应（服务器必须）：
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://mysite.com
Access-Control-Allow-Methods: POST, GET, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, X-Token
Access-Control-Max-Age: 86400

预检通过后，浏览器发真正请求：
POST /api/data HTTP/1.1
Origin: https://mysite.com
Content-Type: application/json
X-Token: abc

实际响应（也要带）：
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://mysite.com

credentials: include 的额外要求

带 cookie 的跨源请求对服务器配置更严格：

  • Access-Control-Allow-Origin 不能是 *，必须具体域名
  • 必须设 Access-Control-Allow-Credentials: true
  • 客户端 fetch 必须 credentials: "include"
javascript 
// 前端
fetch("https://api.other.com/me", { credentials: "include" });

// 后端（Node Express）
app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (allowedOrigins.includes(origin)) {   // ← 白名单
    res.setHeader("Access-Control-Allow-Origin", origin);    // 具体域名
    res.setHeader("Access-Control-Allow-Credentials", "true");
    res.setHeader("Vary", "Origin");                          // 让缓存区分不同源
  }
  next();
});

CORS 阻止什么不阻止什么

  • 阻止：JS 读取跨源响应 body / header（请求还是发出去了）
  • 不阻止：<img>、<script>、<link> 加载（这些是"副作用使用"不是读取）
  • 不阻止：简单的跨源 POST（fetch 以 no-cors 模式）—— 但响应是 opaque，不可读
⚠ CSRF 和 CORS 是两回事

CORS 阻止跨源读取，但不阻止跨源发送。CSRF 攻击是"第三方网站让用户的浏览器带着 cookie 发请求" —— CORS 不防 CSRF。防 CSRF 靠 SameSite cookie + CSRF token + 检查 Origin / Referer header。

Expose-Headers：JS 默认只能读 6 个

跨源响应的 JS res.headers 默认只能访问：Cache-Control, Content-Language, Content-Length, Content-Type, Expires, Last-Modified, Pragma。自定义 header（如 X-Total-Count、X-Rate-Limit-Remaining）要显式 expose：

http 
Access-Control-Expose-Headers: X-Total-Count, X-Rate-Limit-Remaining

Sec-Fetch-* 客户端 hints

现代浏览器自动发的 request metadata headers，服务端可据此识别请求意图：

Header 典型值
Sec-Fetch-Site same-origin / same-site / cross-site / none
Sec-Fetch-Mode cors / navigate / no-cors / same-origin
Sec-Fetch-Dest document / script / image / empty（fetch）/ ...
Sec-Fetch-User ?1（用户手动触发）

工程用途：防 CSRF（拒绝 cross-site + 非 navigate）、分析流量来源、绕过 Referer 的不可靠。

CSP 基础

CSP 是响应 header（Content-Security-Policy），告诉浏览器"本页允许加载/执行什么"。违反策略的资源被拒绝，控制台报错，可选上报 endpoint。

http 
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-abc123' https://cdn.example.com;
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.example.com wss://realtime.example.com;
  font-src 'self' https://fonts.gstatic.com;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
  upgrade-insecure-requests;
  report-uri /csp-report;

CSP 主要指令

指令 控制
default-src 所有类型的默认值
script-src JS（含 inline、eval）
style-src CSS
img-src 图片
font-src 字体
connect-src fetch / XHR / WebSocket / EventSource
frame-src iframe 的 src
frame-ancestors 谁能把本页嵌进 iframe（替代 X-Frame-Options）
base-uri <base> 标签的 href
form-action form submit 的目标
report-uri / report-to 违规上报端点

CSP 源列表值

值 含义
'self' 同源
'none' 禁止所有
'unsafe-inline' 允许 inline <script> / style=""（不推荐）
'unsafe-eval' 允许 eval / new Function
'strict-dynamic' 信任 nonce/hash 允许的 script 加载的子 script
'nonce-xxx' 只允许带此 nonce 的 inline script
'sha256-xxx' 只允许 hash 匹配的 inline script
https: / data: / blob: 协议白名单
具体域名 https://cdn.example.com

strict-dynamic 推荐模式

http 
Content-Security-Policy:
  script-src 'nonce-{RANDOM}' 'strict-dynamic';
  object-src 'none';
  base-uri 'none';

工作流程：

  1. 服务器每次响应生成随机 nonce，插入到 script 标签和 CSP header
  2. 浏览器只允许带此 nonce 的 script 执行
  3. strict-dynamic 让这些"信任 script"加载的子 script / chunk 自动被信任（无需各自加 nonce）
  4. 不信任其他所有 script（即使域名在白名单）

Google 推荐的生产 CSP 模式。比"列 CDN 域名白名单"更安全（域名可能被劫持）。

Report-Only 模式调试

http 
Content-Security-Policy-Report-Only:
  script-src 'self' 'nonce-xxx' 'strict-dynamic';
  report-uri /csp-report;

只报告违规、不阻止。先用这个收集一周数据再切成强制策略。避免"一上来全站白屏"。

03 应用场景

Express / Fastify 开启 CORS

javascript 
// Express + cors 中间件
import cors from "cors";
app.use(cors({
  origin: (origin, cb) => {
    const allowed = ["https://mysite.com", "https://admin.mysite.com"];
    if (!origin || allowed.includes(origin)) cb(null, true);
    else cb(new Error("Not allowed by CORS"));
  },
  credentials: true,
  maxAge: 3600,
  exposedHeaders: ["X-Total-Count"],
}));

// Fastify + @fastify/cors 同理

Next.js CSP with nonce

javascript 
// middleware.ts
import { NextResponse } from "next/server";
import crypto from "crypto";

export function middleware(req) {
  const nonce = crypto.randomBytes(16).toString("base64");
  const csp = `
    default-src 'self';
    script-src 'self' 'nonce-${nonce}' 'strict-dynamic';
    style-src 'self' 'unsafe-inline';
    img-src 'self' blob: data:;
    connect-src 'self' https://api.mysite.com;
    frame-ancestors 'none';
    upgrade-insecure-requests;
  `.replace(/\s+/g, " ").trim();

  const res = NextResponse.next();
  res.headers.set("Content-Security-Policy", csp);
  res.headers.set("x-nonce", nonce);   // 用 header 传给页面
  return res;
}

// 页面里
// import { headers } from "next/headers";
// const nonce = headers().get("x-nonce");
// <Script nonce={nonce} ... />

CORS 预检缓存

javascript 
// 高频 API：每次预检都浪费往返
// 设 max-age 让浏览器缓存

app.options("/api/*", (req, res) => {
  res.setHeader("Access-Control-Allow-Origin", req.headers.origin);
  res.setHeader("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE");
  res.setHeader("Access-Control-Allow-Headers", "Content-Type, Authorization");
  res.setHeader("Access-Control-Max-Age", "86400");   // 24h
  res.status(204).end();
});

// Chrome 预检缓存上限实际是 2 小时（规范允许 24h）
// Firefox 24h
// Safari 更短

CSP 违规上报

javascript 
// 前端 CSP 头
Content-Security-Policy:
  script-src 'self' 'nonce-xxx';
  report-to csp-endpoint;

// 新的 Reporting-Endpoints header（替代旧 report-uri）
Reporting-Endpoints: csp-endpoint="/api/csp-report"

// 后端接收
app.post("/api/csp-report", express.json({ type: "application/reports+json" }), (req, res) => {
  logger.warn({ csp: req.body }, "CSP violation");
  res.status(204).end();
});

javascript 
// ❌ 触发预检
fetch("/api/data", {
  method: "POST",
  headers: { "Content-Type": "application/json", "X-Token": "abc" },
  body: JSON.stringify(data),

// ✓ 如果 token 能放 cookie / 能用 Bearer in URL / 能放 form body

CSP + inline event handler 替代

html 
<!-- ❌ CSP 严格下不允许 -->
<button onclick="submit()">Go</button>
<a href="javascript:submit()">Go</a>

<!-- ✓ 用 addEventListener -->
<button id="go">Go</button>
<script nonce="abc">
document.getElementById("go").addEventListener("click", submit);
</script>

<!-- 或更好：委托 + data-action 模式（见 P5-05） -->
<button data-action="submit">Go</button>

Origin 白名单检查

javascript 
// 更复杂的白名单（支持通配子域）
const ALLOWED_PATTERNS = [
  /^https:\/\/([a-z0-9-]+\.)?mysite\.com$/,
  /^https:\/\/(staging|preview)--[a-z0-9-]+\.netlify\.app$/,
];

function checkOrigin(origin) {
  return ALLOWED_PATTERNS.some((re) => re.test(origin));
}

app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (origin && checkOrigin(origin)) {
    res.setHeader("Access-Control-Allow-Origin", origin);
    res.setHeader("Access-Control-Allow-Credentials", "true");
    res.setHeader("Vary", "Origin");
  }
  next();
});

05 常见陷阱

Allow-Origin: * + credentials: include

这个组合被浏览器拒绝。带 cookie 的请求必须 Allow-Origin 是具体域名。服务端要维护白名单，根据 request Origin header 回显。

忘设 Vary: Origin

服务端根据 Origin 决定 Allow-Origin 时，如果没加 Vary: Origin，CDN / 浏览器缓存会把第一个请求的响应缓存给所有其他源 → 跨源请求可能错误地命中缓存 → 403 或 CORS 错误。

预检失败但无明确错误

Chrome 开发者工具的 Network 里预检请求失败时 status 可能是 200（因为服务器响应了 OPTIONS），但 CORS check 失败 → console "blocked by CORS policy"。要看 Network 面板里的 CORS 标签，不只是 status。

X-Frame-Options 和 CSP frame-ancestors 并存

两者功能重叠（防 clickjacking）。CSP frame-ancestors 是标准，优先级更高。设了两个时 frame-ancestors 生效。现代推荐只用 frame-ancestors。

CSP 阻止所有 inline style

style-src 'self' 不允许 style="" 属性。Emotion / styled-components / TailwindCSS 的 arbitrary 样式都可能受影响。解决：'unsafe-inline'（不推荐）、'nonce-xxx'、或 hash 所有 inline 样式。

strict-dynamic 对 <link> / <img> 无效

strict-dynamic 只作用于 script-src。其他指令（img-src、style-src）仍按白名单。别期待一个 nonce 搞定全部。

CSP 的 eval 限制（回顾 P4-07）

'unsafe-eval' 不加，eval / new Function / setTimeout(string) 都被拒。Vue 运行时 compiler / 模板引擎需要，生产应 precompile。

CSP meta 标签的 限制

<meta http-equiv="Content-Security-Policy"> 不支持 frame-ancestors / sandbox / report-uri。必须用 header。SPA 静态托管如果想用 meta，注意这些指令要走 CDN/服务端 header。
