IndexedDB

浏览器内置的事务型 NoSQL 数据库，支持索引、游标、version 迁移、structured clone 原生存储（Date/Blob/File/Map/Set/ArrayBuffer 全部保真）。原生 API 极其丑陋（回调套回调），生产一律用 idb 封装。本章讲清事务边界、版本迁移、游标的实际用法和常见陷阱。

阅读时长 ~25 分钟 前置 06 Storage 规范 W3C IndexedDB 3.0

01 速览

  • 数据库层级：Database（有版本号）→ ObjectStore（≈表，有 keyPath + 索引）→ Record（key + value，value 为任意 structured clone 可复制对象）。
  • 所有操作在 Transaction 内：readonly（并发）/ readwrite（互斥）/ versionchange（升级，独占）。
  • 事务自动提交：当所有 pending request 完成且事件循环空转一轮后提交；期间不能有任何 await 非 IDB 操作（会丢失事务）。
  • 容量：共享配额（通常磁盘 60%），navigator.storage.estimate() 查询，persist() 申请持久化避免自动清理。
  • 键空间有序，支持 IDBKeyRange 范围查询；索引可设 unique / multiEntry（数组值展开）。
  • 生产一律用 idb（Promise 包装）或 idb-keyval（key-value 糖）；Dexie 提供更高级 ORM。

02 底层逻辑

原生 API 的事件模型

const request = indexedDB.open("myDB", 2); // 版本号必须是正整数

request.onupgradeneeded = (e) => {
  // 仅当版本号变化时触发，独占 versionchange 事务
  const db = request.result;
  const oldVersion = e.oldVersion;
  if (oldVersion < 1) {
    const store = db.createObjectStore("tasks", { keyPath: "id", autoIncrement: true });
    store.createIndex("by_status", "status");
    store.createIndex("by_tags", "tags", { multiEntry: true });
  }
  if (oldVersion < 2) {
    // 逐版本迁移
    const tx = request.transaction; // versionchange 事务
    const store = tx.objectStore("tasks");
    store.createIndex("by_createdAt", "createdAt");
  }
};

request.onsuccess = () => {
  const db = request.result;
  // 用 db 创建事务...
};

request.onerror = () => console.error(request.error);
request.onblocked = () => {
  // 其他标签打开了旧版本，升级被阻塞
  // 需要提示用户关闭其他标签或主动 close 其他连接
};

事务自动提交陷阱

// ❌ 事务在 await fetch 后已提交，store 不可用
const tx = db.transaction("tasks", "readwrite");
const store = tx.objectStore("tasks");
await store.put({ id: 1, name: "A" });
await fetch("/api/log");                  // ← 事件循环让出
await store.put({ id: 2, name: "B" });    // TransactionInactiveError

// ✅ 事务内只做 IDB 操作；外部副作用放事务外
const tx = db.transaction("tasks", "readwrite");
const store = tx.objectStore("tasks");
store.put({ id: 1, name: "A" });
store.put({ id: 2, name: "B" });
await tx.done; // 等待事务完成（idb 库）或监听 tx.oncomplete
await fetch("/api/log");

为什么会自动提交：规范要求事务在所有 request 完成且 microtask queue 空时提交，防止长事务阻塞其他连接。一个非 IDB 的 await 让出控制权，触发提交检查。

版本迁移：只能递增、只能在 upgrade 事件内

操作 允许时机
createObjectStore / deleteObjectStore 仅 onupgradeneeded（versionchange 事务内）
createIndex / deleteIndex 仅 onupgradeneeded
修改 keyPath / autoIncrement 不允许，只能删库重建
put / delete / add / get 任何事务

onupgradeneeded 必须幂等：用户可能跳过中间版本（从 v1 直接到 v5），代码要检查 oldVersion 按顺序迁移。

键与索引

// 主键来源：3 种
db.createObjectStore("a", { keyPath: "id" });              // 从 value 取 id 字段
db.createObjectStore("b", { keyPath: "id", autoIncrement: true }); // 取 id，无则自动生成
db.createObjectStore("c", { autoIncrement: true });        // 外部指定或自动生成
db.createObjectStore("d");                                 // 必须 .put(value, key) 手动传 key

// 键合法类型（有序）
// number | string | Date | binary | Array
store.put(value, [2024, "a", "b"]);  // 复合键

// 范围查询
const range = IDBKeyRange.bound("a", "z", false, true); // [a, z)
const index = store.index("by_name");
const request = index.getAll(range, 100);

// multiEntry：数组值展开成多条索引
store.createIndex("by_tags", "tags", { multiEntry: true });
// 存 { id: 1, tags: ["rust", "wasm"] }
// 索引产生两条：("rust", 1) 和 ("wasm", 1)
// 支持 index.getAll("rust") 直接查

游标遍历

// 原生 API：每次 continue() 触发一次 onsuccess
const tx = db.transaction("tasks", "readonly");
const store = tx.objectStore("tasks");
const request = store.openCursor();
request.onsuccess = () => {
  const cursor = request.result;
  if (!cursor) return; // 遍历结束
  console.log(cursor.key, cursor.value);
  cursor.continue();
};

// idb 封装：用 for await
import { openDB } from "idb";
const db = await openDB("myDB", 2);
const tx = db.transaction("tasks", "readonly");
for await (const cursor of tx.store) {
  console.log(cursor.key, cursor.value);
  // cursor.update(newValue) / cursor.delete() 可用
  if (cursor.key > 1000) break; // 提前退出
}

为什么用游标而不是 getAll：百万行数据 getAll 一次性载入内存会 OOM；游标逐条拉取 + 提前退出可做分页、流式渲染。

structured clone：IndexedDB 的隐藏超能力

LocalStorage 只能存字符串；IndexedDB 用 structured clone 算法序列化，支持：

  • Date、RegExp、Blob、File、FileList、ImageData
  • Map、Set、ArrayBuffer、TypedArray、DataView
  • 循环引用（const o = { self: null }; o.self = o 可存可取）

不支持：函数、Symbol、Error（部分浏览器支持）、DOM 节点。存图片可直接塞 Blob，比 base64 省 33%。

配额与持久化

// 查询配额
const { quota, usage } = await navigator.storage.estimate();
console.log(`已用 ${usage} / ${quota} 字节`);

// 申请持久化（防止被自动清理）
if (navigator.storage.persist) {
  const persisted = await navigator.storage.persist();
  console.log("Persistent storage:", persisted);
  // 依赖站点评分：已安装 PWA / 高频访问 / 用户交互后会授予
}

// 检查是否已持久化
const isPersisted = await navigator.storage.persisted();

Chrome 默认给单个 origin 最多 可用磁盘的 60%；所有 origin 总和不超过磁盘的 60%。超限时 put 抛 QuotaExceededError。

IndexedDB vs OPFS vs Cache Storage

场景 推荐 理由
结构化数据 + 索引查询 IndexedDB 事务 + 索引 + 游标原生支持
大文件读写（>100MB） OPFS（navigator.storage.getDirectory()） 文件系统 API，同步访问在 Worker 中极快
HTTP 响应缓存 Cache Storage（Service Worker） Request/Response 对象原生存储，与 fetch 匹配
跨 Worker 共享 OPFS + AccessHandle（同步 IO） Atomics + SAB 做无锁同步

SQLite WASM（sqlite-wasm）基于 OPFS 提供真 SQL 引擎；复杂查询场景比 IndexedDB 易用得多。

03 应用场景

离线队列（Background Sync 前的兜底）

// 用户离线时把操作排队
import { openDB } from "idb";

const dbPromise = openDB("pending-ops", 1, {
  upgrade(db) {
    db.createObjectStore("queue", { keyPath: "id", autoIncrement: true });
  },
});

async function enqueueOp(op: { url: string; method: string; body?: unknown }) {
  const db = await dbPromise;
  await db.add("queue", { ...op, createdAt: Date.now() });
  // 尝试触发 Background Sync（详见 P6-08）
  const reg = await navigator.serviceWorker.ready;
  await reg.sync?.register("flush-queue");
}

async function flushQueue() {
  const db = await dbPromise;
  const tx = db.transaction("queue", "readwrite");
  for await (const cursor of tx.store) {
    const op = cursor.value;
    try {
      await fetch(op.url, { method: op.method, body: JSON.stringify(op.body) });
      await cursor.delete();
    } catch {
      // 网络仍不通，保留到下次
      break;
    }
  }
}

大列表缓存（替代 LocalStorage 的 5MB 限制）

import { openDB } from "idb";

interface CachedItem<T> { key: string; value: T; expiresAt: number; }

const db = await openDB("app-cache", 1, {
  upgrade(db) {
    const store = db.createObjectStore("items", { keyPath: "key" });
    store.createIndex("by_expires", "expiresAt");
  },
});

async function setCache<T>(key: string, value: T, ttlMs: number) {
  await db.put("items", { key, value, expiresAt: Date.now() + ttlMs });
}

async function getCache<T>(key: string): Promise<T | null> {
  const item = await db.get("items", key) as CachedItem<T> | undefined;
  if (!item) return null;
  if (item.expiresAt < Date.now()) {
    await db.delete("items", key);
    return null;
  }
  return item.value;
}

// 定期清理过期项
async function gc() {
  const range = IDBKeyRange.upperBound(Date.now());
  const tx = db.transaction("items", "readwrite");
  for await (const cursor of tx.store.index("by_expires").iterate(range)) {
    await cursor.delete();
  }
}

idb-keyval：最简单的 key-value 接口

import { get, set, del, keys } from "idb-keyval";

await set("user", { id: 1, name: "Alice", createdAt: new Date() });
const user = await get("user"); // Date 保持 Date 类型
await del("user");
const all = await keys();

// 自定义 store
import { createStore } from "idb-keyval";
const settingsStore = createStore("app", "settings");
await set("theme", "dark", settingsStore);

Dexie：更像 ORM 的封装

import Dexie, { type EntityTable } from "dexie";

interface Task { id?: number; title: string; status: "todo" | "done"; tags: string[]; }

const db = new Dexie("TaskDB") as Dexie & {
  tasks: EntityTable<Task, "id">;
};
db.version(1).stores({
  tasks: "++id, status, *tags", // ++id 自增主键，*tags 多值索引
});

// 查询像链式调用
const done = await db.tasks.where("status").equals("done").reverse().limit(20).toArray();
const rustTasks = await db.tasks.where("tags").equals("rust").toArray();

// 事务显式
await db.transaction("rw", db.tasks, async () => {
  const task = await db.tasks.get(1);
  if (task) await db.tasks.update(1, { status: "done" });
});

全文搜索

IndexedDB 原生不支持全文检索。常见方案：（a）建倒排索引 store，自己维护 token → id 映射；（b）用 lunr 或 FlexSearch 在内存建索引，IndexedDB 持久化；（c）SQLite WASM + FTS5。百万级以下数据 FlexSearch 足矣。

04 典型代码

带版本迁移的 idb 封装

import { openDB, type IDBPDatabase } from "idb";

interface Schema {
  tasks: { key: number; value: { id: number; title: string; status: string; createdAt: Date; tags: string[] }; indexes: { by_status: string; by_tags: string; by_createdAt: Date } };
  projects: { key: string; value: { slug: string; name: string } };
}

let dbPromise: Promise<IDBPDatabase<Schema>> | null = null;

export function getDb() {
  if (dbPromise) return dbPromise;
  dbPromise = openDB<Schema>("app", 3, {
    upgrade(db, oldVersion, newVersion, tx) {
      if (oldVersion < 1) {
        const tasks = db.createObjectStore("tasks", { keyPath: "id", autoIncrement: true });
        tasks.createIndex("by_status", "status");
      }
      if (oldVersion < 2) {
        const tasks = tx.objectStore("tasks");
        tasks.createIndex("by_tags", "tags", { multiEntry: true });
      }
      if (oldVersion < 3) {
        db.createObjectStore("projects", { keyPath: "slug" });
        const tasks = tx.objectStore("tasks");
        tasks.createIndex("by_createdAt", "createdAt");
      }
    },
    blocked(currentVersion, blockedVersion) {
      console.warn(`其他标签在用 v${currentVersion}，升级到 v${blockedVersion} 被阻塞`);
    },
    blocking(currentVersion, blockedVersion, event) {
      // 其他标签要升级，提示用户刷新
      alert("有新版本可用，请刷新页面");
    },
    terminated() {
      // 连接被浏览器意外关闭（磁盘满、隐私模式等）
      dbPromise = null;
    },
  });
  return dbPromise;
}

并发安全的 "upsert"

// 原子递增计数器（事务保证无竞态）
async function incrementCounter(key: string) {
  const db = await getDb();
  const tx = db.transaction("counters", "readwrite");
  const store = tx.objectStore("counters");
  const current = (await store.get(key))?.count ?? 0;
  await store.put({ key, count: current + 1 });
  await tx.done; // 等事务提交
}

监测配额并在接近上限时清理

async function ensureQuota(minFree: number) {
  const { quota = 0, usage = 0 } = await navigator.storage.estimate();
  const free = quota - usage;
  if (free >= minFree) return;

  // 清理最旧的缓存直到达到阈值
  const db = await getDb();
  const tx = db.transaction("items", "readwrite");
  for await (const cursor of tx.store.index("by_createdAt")) {
    await cursor.delete();
    const { usage: u2 = 0 } = await navigator.storage.estimate();
    if (quota - u2 >= minFree) break;
  }
}

05 常见陷阱

陷阱 1：事务内 await fetch 必丢事务

IDB 事务在 microtask 空转后自动提交。事务内只做 IDB 操作，网络/计算放事务外。需要原子性时先读完所有数据→计算→新开事务写。

陷阱 2：版本升级时其他标签未关闭 → 永久阻塞

用户开了 5 个标签，其中 1 个升级了代码，其他仍连着旧 DB。监听 blocking 事件主动 db.close() 并提示刷新，否则新标签永远卡在 onblocked。

陷阱 3：onupgradeneeded 不幂等

用户跳过多版本（旧标签从 v1 直接升到 v5），代码写 if (oldVersion === 1) 只处理一档，中间迁移丢失。必须用 if (oldVersion < N) 级联处理所有版本。

陷阱 4：keyPath 修改只能删库

原用 { keyPath: "_id" }，想改成 "id"？不行。只能 deleteObjectStore 后重建，数据丢失。设计 schema 时 keyPath 要稳定。

陷阱 5：Safari 隐私模式 IndexedDB 可用但写失败

Safari Private Browsing 允许打开 DB，但写操作抛错。用户可能看到半 bug 的界面。所有写操作 catch，fallback 到内存 Map。

陷阱 6：Firefox 无痕模式 DB 被强制隔离

Firefox Private Window 的 IDB 关闭窗口即清空；Chrome 2020 前清理时机不一致。不要把核心数据只存在 IDB —— 云端要能恢复。

陷阱 7：multiEntry 索引对非数组值静默跳过

tags 字段偶尔是字符串而非数组时，该条不进索引（既不报错也不建 key）。数据层保证类型一致或迁移时清理。

陷阱 8：getAll 一次性吃光内存

百万条记录 store.getAll() 会把所有 value 克隆到主线程堆，几 GB 内存占用。改用 openCursor + 分页。

陷阱 9：structured clone 不复制 function / Symbol / Proxy

类实例（带方法）存进去，取出来方法全丢失，只剩属性。存纯数据，业务对象在应用层重建。

陷阱 10：versionchange 事件导致打开的连接被关闭

标签 A 打开 DB，标签 B 升级，A 的连接收到 onversionchange 事件 —— 若不处理，B 的 onblocked 永远 pending。A 必须监听并 db.close()。idb 库的 blocking 回调已帮你抽象。
