Service Worker + 跨标签通信

Service Worker 是浏览器与网络之间的代理线程，拦截所有同源 fetch 请求，用 Cache API 决定直连 / 回源 / 响应合成。它也是 PWA、Push、Background Sync 的基座。本章同时讲清跨上下文通信：BroadcastChannel（最常用）、SharedWorker（共享状态）、postMessage（点对点）。

阅读时长 ~28 分钟 前置 07 IndexedDB · P4 Workers 规范 W3C Service Workers

01 速览

  • SW 是独立线程，没有 DOM、没有 window；只有 self、caches、clients、fetch、indexedDB。
  • 生命周期：register → install → activate → fetch/message；新版本默认等所有旧 SW 控制的页面关闭才激活，skipWaiting + clients.claim 可强制立即生效。
  • SW 只能注册在 HTTPS（localhost 豁免），作用域受 SW 脚本路径限制。
  • Cache API 存的是 Request/Response 键值对；和 IndexedDB 共享配额。
  • 四种缓存策略：cache-first（静态资源）、network-first（API）、stale-while-revalidate（推荐默认）、cache-only / network-only（边界）。
  • 跨标签通信：BroadcastChannel（同源广播）、SharedWorker（共享状态 + 复杂协议）、postMessage（点对点 + Transferable）。

02 底层逻辑

SW 生命周期

// 注册（主线程）
if ("serviceWorker" in navigator) {
  window.addEventListener("load", async () => {
    const reg = await navigator.serviceWorker.register("/sw.js", {
      scope: "/",                    // 作用域默认是脚本所在目录
      updateViaCache: "none",        // sw.js 自身不缓存（HTTP 缓存层面）
      type: "classic",               // 或 "module" 支持 import
    });
    console.log("SW registered:", reg.scope);
  });
}

// sw.js
self.addEventListener("install", (event) => {
  // 预缓存静态资源
  event.waitUntil(
    caches.open("static-v1").then((cache) => cache.addAll([
      "/", "/app.js", "/app.css",
    ])),
  );
  // self.skipWaiting(); // 跳过 waiting 阶段
});

self.addEventListener("activate", (event) => {
  // 清理旧版本缓存
  event.waitUntil((async () => {
    const names = await caches.keys();
    await Promise.all(names.filter((n) => n !== "static-v1").map((n) => caches.delete(n)));
    // await self.clients.claim(); // 立即接管已存在的页面
  })());
});

self.addEventListener("fetch", (event) => {
  event.respondWith(handleFetch(event.request));
});

install 与 activate 的微妙区别

阶段 触发时机 适合做
install SW 文件变化后首次解析 预缓存静态资源，不要在此清理旧缓存（旧 SW 还在用）
waiting install 完成，等旧 SW 退役 显示更新提示，用户确认后 skipWaiting
activate 所有旧 SW 控制页面关闭 清理旧缓存、升级 IDB schema、clients.claim
idle 40 秒无事件自动 sleep —

skipWaiting + clients.claim 的风险：旧页面加载了 v1 资源（JS/CSS/chunks），立即换成 v2 的 SW，如果 v2 缓存里没有 v1 的 chunk，用户触发懒加载会 404。生产一般用"通知 + 用户主动刷新"模式，不默认 skipWaiting。

scope 与 clients

// scope 是注册时的 URL 前缀
navigator.serviceWorker.register("/sw.js", { scope: "/" }); // 控制整个站
navigator.serviceWorker.register("/app/sw.js", { scope: "/app/" }); // 只控制 /app/*
// 想让根目录 /sw.js 控制 / 但允许放在 /app/ 需服务端返回 Service-Worker-Allowed: /

// clients：SW 管理下所有同源页面的引用
self.addEventListener("message", async (event) => {
  const allClients = await self.clients.matchAll({ type: "window" });
  for (const client of allClients) {
    client.postMessage({ type: "UPDATE" }); // 向所有页面广播
  }
});

// 唯一客户端模式：新窗口聚焦已打开的
self.addEventListener("notificationclick", (event) => {
  event.waitUntil((async () => {
    const clients = await self.clients.matchAll({ type: "window" });
    for (const c of clients) {
      if (c.url.endsWith("/app") && "focus" in c) return c.focus();
    }
    return self.clients.openWindow("/app");
  })());
});

Cache API

// Cache 对象：Request → Response 映射
const cache = await caches.open("runtime-v1");

await cache.put(request, response.clone()); // response 是流，put 会消费，需 clone
const cached = await cache.match(request);   // 支持字符串或 Request
await cache.delete(request);
const keys = await cache.keys();              // 所有 Request

// 按 URL 路径匹配（忽略 search）
const resp = await cache.match(request, { ignoreSearch: true });
// 跨 cache 匹配
const resp2 = await caches.match(request);  // 搜索所有 cache

// 批量添加（失败则全部回滚）
await cache.addAll(["/a", "/b", "/c"]);

response.clone() 的必要性：Response.body 是 ReadableStream，只能读一次。put(req, resp) 消费流后，resp.body 已 locked，不能再用 resp.text()。

四种缓存策略

策略 适合场景 优点 缺点
cache-first 版本化静态资源（JS/CSS 带 hash） 最快，离线可用 资源更新依赖版本化 URL
network-first HTML / 动态 API 始终最新 离线时才用缓存，网络抖动时慢
stale-while-revalidate 默认推荐 / 头像 / 配置 立即返回旧缓存 + 后台更新 用户首次看到可能是旧数据
cache-only 纯离线资源 — 无法更新
network-only 鉴权 API / 支付 保证实时 SW 完全不缓存 
// stale-while-revalidate（推荐默认）
async function staleWhileRevalidate(request) {
  const cache = await caches.open("swr-v1");
  const cached = await cache.match(request);

  const fetchPromise = fetch(request).then((resp) => {
    if (resp.ok) cache.put(request, resp.clone());
    return resp;
  }).catch(() => null);

  return cached ?? (await fetchPromise) ?? new Response("offline", { status: 503 });
}

Workbox 简介

Google 出的 SW 编排库，声明式路由 + 缓存策略。Next.js PWA 插件（next-pwa、@ducanh2912/next-pwa）底层就是它。 

import { registerRoute } from "workbox-routing";
import { CacheFirst, NetworkFirst, StaleWhileRevalidate } from "workbox-strategies";
import { ExpirationPlugin } from "workbox-expiration";

registerRoute(
  ({ request }) => request.destination === "image",
  new CacheFirst({
    cacheName: "images",
    plugins: [new ExpirationPlugin({ maxEntries: 50, maxAgeSeconds: 30 * 24 * 3600 })],
  }),
);

registerRoute(
  ({ url }) => url.pathname.startsWith("/api/"),
  new NetworkFirst({ cacheName: "api", networkTimeoutSeconds: 3 }),
);

Background Sync

// 主线程：注册同步任务
const reg = await navigator.serviceWorker.ready;
await reg.sync.register("send-pending-messages");

// sw.js
self.addEventListener("sync", (event) => {
  if (event.tag === "send-pending-messages") {
    event.waitUntil(flushMessageQueue());
  }
});

浏览器在网络恢复后（可能跨浏览器重启）自动触发 sync 事件。仅 Chromium 支持，Safari / Firefox 仍在跟进。

跨上下文通信对比

机制 通信对象 API 特点 典型用途
window.postMessage iframe / 弹出窗口 / opener 点对点，需目标引用 跨源嵌入通信（SSO、支付）
worker.postMessage 主线程 ↔ Worker 结构化克隆 + Transferable CPU 密集任务
BroadcastChannel 所有同源上下文（tab/SW/Worker） 发布订阅，自己不收 登出广播、数据同步
SharedWorker 所有同源标签共享一个 Worker 状态集中，port 通信 WebSocket 多路复用
Service Worker clients SW ↔ 所有控制页面 SW 主动广播 推送、更新通知
storage 事件 同源标签（非当前） 依赖 LocalStorage 写入 简易同步（见 P6-06）
IndexedDB + polling 任何上下文 慢、需定时 持久化协作状态

BroadcastChannel 语义

// 所有同源上下文共享同名 channel
const bc = new BroadcastChannel("auth");
bc.postMessage({ type: "LOGOUT" });

bc.addEventListener("message", (event) => {
  // 发送者自己不会收到；其他所有同名 channel 的上下文会收
  if (event.data.type === "LOGOUT") location.href = "/login";
});

bc.close(); // 释放，防止内存泄漏

// 支持 structured clone：Date、Map、Blob 都能传

SharedWorker 语义

// 主线程：连接到共享 Worker
const worker = new SharedWorker("/shared.js");
worker.port.start();
worker.port.postMessage({ type: "PING" });
worker.port.addEventListener("message", (e) => console.log(e.data));

// shared.js：被所有同源标签共享的单例
const connections = new Set();

self.addEventListener("connect", (event) => {
  const port = event.ports[0];
  connections.add(port);
  port.addEventListener("message", (e) => {
    // 广播到所有连接
    for (const p of connections) p.postMessage(e.data);
  });
  port.start();
});

SharedWorker 不在 Chrome Android / Safari iOS 上可用，移动端生产不可依赖。桌面端可以用它做 WebSocket 多路复用（10 个标签共享 1 个 WS 连接）。

03 应用场景

PWA 离线骨架

// sw.js
const APP_SHELL = "shell-v3";
const SHELL_URLS = ["/", "/offline.html", "/manifest.webmanifest", "/icon.svg"];

self.addEventListener("install", (e) => {
  e.waitUntil(caches.open(APP_SHELL).then((c) => c.addAll(SHELL_URLS)));
});

self.addEventListener("activate", (e) => {
  e.waitUntil((async () => {
    const names = await caches.keys();
    await Promise.all(names.filter((n) => n !== APP_SHELL).map(caches.delete.bind(caches)));
  })());
});

self.addEventListener("fetch", (e) => {
  const req = e.request;
  if (req.mode === "navigate") {
    // 导航请求：网络优先，失败回 offline 页
    e.respondWith((async () => {
      try { return await fetch(req); }
      catch { return (await caches.match("/offline.html")) ?? Response.error(); }
    })());
    return;
  }
  // 其他：cache-first
  e.respondWith(caches.match(req).then((r) => r ?? fetch(req)));
});

SW 更新提示

const reg = await navigator.serviceWorker.register("/sw.js");
reg.addEventListener("updatefound", () => {
  const newSW = reg.installing;
  newSW?.addEventListener("statechange", () => {
    if (newSW.state === "installed" && navigator.serviceWorker.controller) {
      // 有新版本且当前已被旧 SW 控制
      showUpdateToast(() => {
        newSW.postMessage({ type: "SKIP_WAITING" });
      });
    }
  });
});

// sw.js
self.addEventListener("message", (e) => {
  if (e.data?.type === "SKIP_WAITING") self.skipWaiting();
});

navigator.serviceWorker.addEventListener("controllerchange", () => {
  // 新 SW 接管，刷新页面
  location.reload();
});

跨标签广播登出

const authChannel = new BroadcastChannel("auth");

export async function logout() {
  await fetch("/api/logout", { method: "POST" });
  authChannel.postMessage({ type: "LOGOUT" });
  window.location.href = "/login"; // 自己跳转
}

// 在 App 根组件监听
authChannel.addEventListener("message", (e) => {
  if (e.data.type === "LOGOUT") window.location.href = "/login";

WebSocket 连接共享（SharedWorker 场景）

// ws-shared.js（SharedWorker 脚本）
let ws = null;
const ports = new Set();

function ensureWs() {
  if (ws?.readyState <= WebSocket.OPEN) return ws;
  ws = new WebSocket("wss://api.example.com/stream");
  ws.onmessage = (e) => ports.forEach((p) => p.postMessage({ type: "ws", data: e.data }));
  return ws;
}

self.addEventListener("connect", (event) => {
  const port = event.ports[0];
  ports.add(port);
  port.addEventListener("message", (e) => {
    if (e.data.type === "send") ensureWs().send(e.data.payload);
  });
  port.addEventListener("close", () => ports.delete(port));
  port.start();
  ensureWs();
});

优点：10 个标签共享一个 WS 连接，节省服务端连接数 + 带宽。缺点：SharedWorker 在移动浏览器支持差，需有 BroadcastChannel fallback。

04 典型代码

类型化的 SW 通信

// types.ts 共享
type SWRequest =
  | { type: "PREFETCH"; urls: string[] }
  | { type: "CLEAR_CACHE"; name: string }
  | { type: "SKIP_WAITING" };

type SWResponse =
  | { type: "PREFETCH_DONE"; failed: string[] }
  | { type: "CACHE_CLEARED" };

// 主线程
async function ask<T extends SWResponse>(msg: SWRequest): Promise<T> {
  const reg = await navigator.serviceWorker.ready;
  return new Promise((resolve) => {
    const channel = new MessageChannel();
    channel.port1.onmessage = (e) => resolve(e.data);
    reg.active?.postMessage(msg, [channel.port2]);
  });
}

// sw.js
self.addEventListener("message", (event) => {
  const msg = event.data;
  const [port] = event.ports; // MessagePort 或 undefined
  if (msg.type === "PREFETCH") {
    event.waitUntil((async () => {
      const failed: string[] = [];
      const cache = await caches.open("prefetch");
      for (const url of msg.urls) {
        try { await cache.add(url); } catch { failed.push(url); }
      }
      port?.postMessage({ type: "PREFETCH_DONE", failed });
    })());
  }
});

缓存版本化 + 清理

const CACHE_VERSION = "v5";
const CACHES = {
  static: `static-${CACHE_VERSION}`,
  images: `images-${CACHE_VERSION}`,
  api: `api-${CACHE_VERSION}`,
};

self.addEventListener("activate", (e) => {
  e.waitUntil((async () => {
    const names = await caches.keys();
    const expected = new Set(Object.values(CACHES));
    await Promise.all(names.filter((n) => !expected.has(n)).map((n) => caches.delete(n)));
    await self.clients.claim();
  })());
});

带 TTL 的缓存响应

async function cacheWithTtl(request, ttlMs) {
  const cache = await caches.open("api-v1");
  const resp = await fetch(request);
  if (!resp.ok) return resp;

  // 在 Response header 注入过期时间
  const body = await resp.clone().blob();
  const headers = new Headers(resp.headers);
  headers.set("x-cached-at", String(Date.now()));
  headers.set("x-cached-ttl", String(ttlMs));
  const stored = new Response(body, { status: resp.status, statusText: resp.statusText, headers });
  await cache.put(request, stored);
  return resp;
}

async function matchWithTtl(request) {
  const cache = await caches.open("api-v1");
  const resp = await cache.match(request);
  if (!resp) return null;
  const cachedAt = Number(resp.headers.get("x-cached-at"));
  const ttl = Number(resp.headers.get("x-cached-ttl"));
  if (Date.now() - cachedAt > ttl) {
    await cache.delete(request);
    return null;
  }
  return resp;
}

05 常见陷阱

陷阱 1：sw.js 自身被 HTTP 缓存

如果 CDN 把 /sw.js 缓存 1 小时，用户 1 小时内拿不到新版本 SW。必须 服务端对 sw.js 返回 Cache-Control: no-cache 或 max-age=0，且注册时用 updateViaCache: "none"。

陷阱 2：skipWaiting 导致 chunk 404

SPA 懒加载的 chunk 文件名带 hash，SW v2 激活后旧页面懒加载 v1 的 chunk，缓存里没有→ 404。要么保留最近 N 个版本的 chunk 缓存，要么更新时用"提示 + 刷新"替代 skipWaiting。

陷阱 3：event.waitUntil 之外的异步操作会被 kill

SW 进入 idle 会被浏览器终止。install/activate/fetch/sync 中的异步工作必须包在 event.waitUntil(promise) 内，否则 SW 睡眠后任务中断。

陷阱 4：fetch 拦截不跨 origin

SW 只能拦截同 scope 下的 fetch。跨源（CDN 图片、第三方 API）你仍然可以用 fetch(crossOriginUrl) 取回并缓存，但响应必须是 cors 模式且对方允许；否则拿到 opaque response（状态码 0，body 不可读）。

陷阱 5：Response.body 读一次即锁

cache.put(req, resp) 消费了 resp.body；同一响应想同时返给客户端必须 resp.clone()。忘记 clone 后 event.respondWith(resp) 会抛 "Response body is already used"。

陷阱 6：Cache API 不尊重 HTTP Cache-Control

Cache API 是显式存储：你 put 了就永久在，直到你删除或用户清理。Cache-Control: no-store 不会阻止 SW 写入。过期策略全靠自己实现（上面的 TTL 示例）。

陷阱 7：调试时忘记关 "Update on reload"

DevTools Application → Service Workers 里勾了 "Update on reload"，每次刷新都强制更新，线上行为不一致。定位线上 bug 时记得关掉。

陷阱 8：BroadcastChannel 发送者自己不收

发了 bc.postMessage(...) 想让自己和其他标签一起处理，但自己标签不触发。要么在 postMessage 后手动调函数，要么在 post 前先执行本地副作用。

陷阱 9：SharedWorker 移动端不可用

Chrome Android / Safari iOS 不支持 SharedWorker。用它做核心通信 → 移动端直接挂。BroadcastChannel + 单标签做主的选举模式是更安全的替代。

陷阱 10：postMessage Transferable 传完原对象不可用

worker.postMessage(buffer, [buffer]) 转移 ArrayBuffer 所有权后，主线程 buffer.byteLength === 0。想共享不转移，去掉第二个参数让 structured clone 复制（开销大但原对象仍可用）。真正共享用 SharedArrayBuffer + Atomics。
