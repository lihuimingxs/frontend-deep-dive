操作符与强制转换

JS 的操作符底下藏着一套规范算法：ToPrimitive、ToNumber、ToString、Abstract Equality Comparison。学这一章的正确姿势不是背 "== 转换表"，而是记住这几个算法在什么时候被触发 —— 之后一切行为都能推导出来。

阅读时长 ~25 分钟 前置 01 · 数据类型系统 对应 javascript.info Basic operators

01 速览

按规范分类的操作符：

类别 操作符 关键点
算术 + - * / % ** -(一元) +(一元) + 有双语义（加法/拼接）
赋值 = += -= *= /= %= **= &&= ||= ??= 赋值是表达式，返回右值
比较 == != === !== < > <= >= === 无强转；== 走 Abstract Equality
逻辑 && || ! ?? 短路返回操作数而非 boolean
可选链 ?. ?.() ?.[] 左侧是 null/undefined 时整体 undefined
位运算 & | ^ ~ << >> >>> 先 ToInt32，仅 32-bit 整数安全
一元 typeof delete void ! typeof 对未声明变量不抛错
其他 , in instanceof ? : , 返回最右；in 查原型链
展开/剩余 ... 同一符号在调用/解构处语义相反

02 底层逻辑

ToPrimitive 算法：万物归原

任何把对象用到需要原始值的位置（加法、比较、模板字符串、String(obj) 等），引擎都会调用 ToPrimitive(input, hint)：

  1. 如果 input 是原始值，直接返回。
  2. 查 input 上的 Symbol.toPrimitive 方法 —— 若有，调用并以其返回值作为结果（必须是原始值，否则抛 TypeError）。
  3. 否则根据 hint 选择 fallback 顺序：
    • "string"（模板字符串、String()）：先 toString，失败才 valueOf
    • "number"（算术、比较）：先 valueOf，失败才 toString
    • "default"（+、==）：同 "number"，但可被 Symbol.toPrimitive 单独区分
  4. 若 valueOf/toString 返回对象，继续下一个；都失败则抛 TypeError。
javascript 
const money = {
  amount: 100,
  currency: "CNY",
  [Symbol.toPrimitive](hint) {
    if (hint === "number") return this.amount;
    if (hint === "string") return `${this.amount} ${this.currency}`;
    return `${this.amount} ${this.currency}`;  // default
  },
};

+money;             // 100         ← hint "number"
`${money}`;         // "100 CNY"   ← hint "string"
money + "";         // "100 CNY"   ← hint "default" + 字符串拼接
money == 100;       // true        ← hint "default" → 100, 再数字比较

+ 的双重语义

二元 + 的决策树：

  1. 两边都 ToPrimitive(hint: "default")。
  2. 如果任一转换结果是字符串 → 两边 ToString，拼接。
  3. 否则两边 ToNumber，相加（ToNumber 会将 BigInt 混算抛 TypeError）。

这就是为什么 1 + "2" === "12"，而 "2" - 1 === 1 —— 减号没有字符串语义，直接走 ToNumber。

抽象相等比较（==）的 7 条规则

x == y 的规范流程（节选，忽略了 BigInt 分支）：

  1. 类型相同 → 走 === 的判断。
  2. null == undefined → true（特判）。
  3. Number == String → 把 string ToNumber 再比。
  4. Boolean == 任意 → 把 boolean ToNumber（true→1, false→0）后再比。
  5. Number/String == Object → 对象 ToPrimitive("default") 后再比。
  6. 其他一律 false。
⚠ 只记三条就够了

（1） null == undefined，但它们都不 == 其他任何值（包括 0, "", false）。 （2） 有对象参与的 == 会调用 ToPrimitive("default")。 （3） NaN 跟谁都不等（包括自己）。其余情况用 ===。

短路求值返回操作数

这是 JS 和 C/Java 最大的行为差异之一。&& / || 在 JS 中不返回 boolean，而是返回决定结果的那个操作数：

  • a && b：若 a falsy 返回 a，否则返回 b。
  • a || b：若 a truthy 返回 a，否则返回 b。
  • a ?? b：若 a 是 null 或 undefined 返回 b，否则返回 a。

! 是唯一强制返回 boolean 的逻辑操作符，因此 !!x 是把任意值转 boolean 的地道写法（等价于 Boolean(x)）。

位运算的 32-bit 约束

& | ^ ~ << >> 都先对操作数 ToInt32：把 Number 截断成 32-bit 有符号整数。这意味着：

  • x | 0 曾是 Math.trunc 的快速替代（现代引擎两者性能相近，可读性更重要）。
  • 数字 > 2³¹-1 或是小数会被"咔嚓"截断，结果可能意外。
  • >>> 是"无符号右移"，把 32-bit 视为无符号整数。唯一能让 -1 >>> 0 返回 4294967295 的方式。

运算符优先级要点

完整表见 MDN，记住几个高频坑点：

  • 一元操作符（+ - ! typeof）比二元更紧。typeof x + "" 先算 typeof x 再拼接。
  • ** 是右结合：2 ** 3 ** 2 === 2 ** 9 === 512。
  • 赋值是右结合：a = b = c 等价 a = (b = c)。
  • && 优先级高于 ||，|| 高于 ??（但 ?? 和 &&/|| 不能混用，必须加括号，语法层强制）。
  • , 是优先级最低的操作符。let x = 1, 2; 不合法（声明里 , 是分隔符），但 let x = (1, 2); 合法，x 为 2。

03 应用场景

默认值：?? vs ||

|| 把所有 falsy 值当成"没提供"，?? 只把 null/undefined 当成"没提供"。
经典 bug：const volume = config.volume || 50; 会把用户设置的 0（静音）替换成 50。正确写法：const volume = config.volume ?? 50;。

可选链：链式取值防崩

user?.profile?.avatar?.url 在链路任一节点是 null/undefined 时返回 undefined，不抛错。比 user && user.profile && ... 短得多。注意：?. 只能防 null/undefined，不能防 obj.method is not a function。

逻辑赋值：状态机初始化

cache[key] ??= computeValue(); 等价 cache[key] = cache[key] ?? computeValue(); —— 但只在 cache[key] 是 null/undefined 时才调用 computeValue()，避免不必要的开销。&&= 反之（只在已有值时更新）。

位运算：权限位与颜色通道

Canvas pixel data、权限系统（RWX）、网络协议位标志、TypedArray 索引 —— 这些场景下位运算比字符串/数组操作快 10-100 倍且空间占用极小。现代前端工程师最常用的场景其实是位掩码做 feature flag。

展开语法：不可变更新

Redux、Zustand 等状态库要求 immutable 更新：{ ...state, count: state.count + 1 }、[...list, newItem]。性能不如原地修改，但换来时间旅行、浅比较、结构共享。React 的 useState 只在引用不等时才重渲染，所以直接 mutate 原对象不会触发更新。

04 典型代码

== 决策表（背不如推）

javascript 
null == undefined;       // true   (规则 2)
null == 0;               // false  (null 只等 undefined)
null >= 0;               // true   (>= 不走 null 特判，走 ToNumber)
null == false;           // false  (false→0, null 不转 0)
"" == 0;                 // true   ("" → 0, 数字比较)
"0" == false;            // true   ("0" → 0, false → 0)
[] == false;             // true   ([] → "" → 0, false → 0)
[0] == false;            // true   ([0] → "0" → 0)
NaN == NaN;              // false  (NaN 跟谁都不等)

+ 的陷阱演示

javascript 
1 + 2 + "3";            // "33"   (左结合：(1+2) + "3")
"1" + 2 + 3;            // "123"  (第一步就拼接了)
[] + [];                // ""     ([] 转空串，空串拼接)
[] + {};                // "[object Object]"
{} + [];                // 0      ⚠ 在语句位置 {} 是空块，+[] 是 ToNumber("")=0
({}) + [];              // "[object Object]"  (加括号变表达式)
+"";                    // 0      (一元 + 是 ToNumber)
+"abc";                 // NaN
+null;                  // 0
+undefined;             // NaN
+"  12  ";              // 12     (ToNumber 会 trim)
+"";                    // 0
+[];                    // 0      ([] → "" → 0)
+[1];                   // 1      ([1] → "1" → 1)
+[1,2];                 // NaN    ([1,2] → "1,2" → NaN)

优先级陷阱

javascript 
// 1. typeof 是一元，绑得更紧
typeof 1 + "";           // "number"     → typeof 1 === "number", 再拼 ""
                         // 不是 typeof (1 + "") === "string"

// 2. ** 右结合
2 ** 3 ** 2;             // 512  (= 2 ** 9, 不是 64)

// 3. ?? 不能与 && / || 混用，必须括号
// a ?? b && c;          // SyntaxError
a ?? (b && c);           // ok

// 4. 赋值表达式
const a = b = 0;         // b 成了全局（非严格）/ ReferenceError（严格）
                         // a 声明了，b 没有

// 5. , 操作符
const arr = [1, (2, 3), 4];  // [1, 3, 4]  中间的 (2,3) 返回 3

可选链 + 空值合并的常用组合

javascript 
// 链式调用带默认
const count = data?.items?.length ?? 0;

// 调用可能不存在的方法
element?.scrollIntoView?.();

// 动态键 + 链式
const value = obj?.[key]?.toString?.() ?? "N/A";

// ?? 的特性：只区分 null/undefined
const page = params.page ?? 1;    // 允许 page=0
const page2 = params.page || 1;   // ⚠ page=0 被替换成 1

// ⚠ 短路仍会求值左侧表达式（包括 .get 的 throw）
// map.get?.(key) 不能防止 get 本身抛错，只防 map 为 null/undefined

05 常见陷阱

[] == ![] 居然是 true

右边先算：![] → false（因为 [] 是 truthy）。变成 [] == false。规则 4：false → 0，得 [] == 0。规则 5：[] → ToPrimitive → ""。规则 3："" → 0，得 0 == 0 → true。全程没有任何 bug，就是规则层层套娃。用 === 免疫。

{} + [] 的含义取决于位置

在语句起始位置 {} 被解析为空块，+[] 是一元加，结果 0。在表达式位置 ({}) + [] 是对象 + 数组 = "[object Object]"。REPL 和源文件行为可能不同 —— 这不是 bug，而是 ASI（自动分号插入）的副作用。

typeof 对已声明但未赋值变量返回 "undefined"

let x; typeof x 是 "undefined"。typeof undeclaredVar 也是 "undefined" 而不抛错 —— 这是语言层面唯一允许引用未声明变量的地方。历史原因（早期用于检测全局对象），现代代码基本用不到。

null >= 0 是 true，但 null > 0 是 false

关系操作符走 ToNumber，null → 0，所以 null >= 0 === 0 >= 0 === true。而 null == 0 是 false（因为 == 有 null 特判）。>= 不等于 > || ==。

链式 ?. 的短路会跳过后续调用

a?.b().c：如果 a 是 null/undefined，整个表达式短路为 undefined，.b() 也不会被调用。如果 a.b() 返回 null，.c 会抛错。链中每个可能为空的节点都需要 ?.。

delete 不会删除变量，只删对象属性

const x = 1; delete x; 返回 true 但 x 仍存在（严格模式下直接语法错误）。delete 只对对象属性生效，且对非 configurable 属性也静默失败。清空状态用赋值 obj.key = undefined 或删键 delete obj.key（后者影响 for-in/hasOwn）。

void 0 是获取 undefined 的"安全"方式

在 ES5 严格模式/现代代码里 undefined 已经是只读全局，写 undefined 就够了。void 0 是 ES3 时代兼容写法（当时 undefined 可被覆盖）。Minifier 仍会用 void 0 替换 undefined 省字节 —— 这是工具做的，你不需要手写。

展开语法的浅拷贝陷阱

{ ...obj } 和 [...arr] 只拷贝一层。嵌套对象/数组仍共享引用。改 copy.nested.x 会改原对象。深拷贝用 structuredClone(obj)（原生、支持循环引用/Map/Set/Date/TypedArray，不支持函数和 Symbol key）。
