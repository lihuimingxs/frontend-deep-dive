Date 与 Intl

Date 是 JS 里被吐槽最多的内置类型：月份从 0 开始、getDay 返回星期而不是几号、ISO 短格式按 UTC 解析、没有原生时区支持…… 本章把这些"常识"的规范依据、Intl.* 的正确用法、未来替代（Temporal）梳清楚。

阅读时长 ~22 分钟 前置 08 数字 对应 javascript.info Date and time

01 速览

javascript 
// 构造：5 种形式
new Date();                                 // 当前时间
new Date(1_700_000_000_000);                // epoch 毫秒
new Date("2024-03-15T10:00:00Z");           // ISO 字符串（推荐）
new Date(2024, 2, 15, 10, 0, 0, 0);         // ⚠ month 从 0 开始
Date.now();                                 // 当前 epoch 毫秒（不建对象，更快）

// 读
d.getFullYear();  d.getMonth();  d.getDate();
d.getDay();       // 星期几（0 = Sun）
d.getHours();     d.getMinutes();  d.getSeconds();  d.getMilliseconds();
d.getTime();      // epoch 毫秒
d.getTimezoneOffset();  // 本地与 UTC 的分钟偏移 —— ⚠ 符号反直觉，UTC+8 返回 -480

// UTC 版（getter / setter 都有）
d.getUTCHours();  d.setUTCHours(0);

// 写
d.setFullYear(2025);  d.setDate(15);  d.setHours(12);
// setters 全部 mutating，返回 epoch 毫秒

// 格式化
d.toISOString();                // "2024-03-15T10:00:00.000Z"  ← 永远 UTC
d.toJSON();                     // 同 toISOString
d.toLocaleString("zh-CN");      // 本地化（底下是 Intl.DateTimeFormat）

Intl 家族成员

API 用途
Intl.DateTimeFormat 本地化日期时间格式
Intl.RelativeTimeFormat "3 天前" / "in 2 hours"
Intl.NumberFormat 数字 / 货币 / 百分比（见 08 章）
Intl.Collator 本地化排序（见 07 章）
Intl.Segmenter 字素簇 / 词 / 句切分（见 07 章）
Intl.ListFormat 列表格式（"A, B, and C"）
Intl.PluralRules 单复数规则（英文/俄文差异）
Intl.Locale locale 本身的结构化对象
Intl.DurationFormat 持续时间（ES2025）

02 底层逻辑

Date 本质 = epoch 毫秒

Date 对象内部就存一个数字：自 1970-01-01T00:00:00Z 起的毫秒数（Unix 时间戳 × 1000）。所有 getter / setter / format 方法都是基于这个数字 + 一个隐含的"以本地时区解释"操作。

javascript 
const d = new Date("2024-03-15T10:00:00Z");
+d;                      // 1710496800000  （epoch 毫秒）
d.getTime();             // 同上
d.valueOf();             // 同上

// 日期减法：毫秒差
const start = new Date();
// ... 工作 ...
const ms = Date.now() - start;

字符串解析的"短格式按 UTC"规则

规范对日期字符串解析极其严格。只有下列格式能保证跨引擎一致解析（其他格式由引擎自行决定）：

  • "YYYY"
  • "YYYY-MM"
  • "YYYY-MM-DD" — 按 UTC（注意！）
  • "YYYY-MM-DDTHH:mm"
  • "YYYY-MM-DDTHH:mm:ss"
  • "YYYY-MM-DDTHH:mm:ss.sss"
  • "YYYY-MM-DDTHH:mm:ssZ"（带 Z，显式 UTC）
  • "YYYY-MM-DDTHH:mm:ss+08:00"（显式偏移）
javascript 
// 在 UTC+8 的机器上：
new Date("2024-03-15");          // 2024-03-15T00:00:00Z  → 本地时间 2024-03-15 08:00
new Date("2024-03-15T00:00");    // 2024-03-14T16:00Z     → 本地时间 2024-03-15 00:00

// ⚠ 短格式和带时间的格式，时区解释不一样！
// 短格式 "2024-03-15" 按 UTC 午夜，带时间的按本地午夜

// 只有显式带 Z 或偏移的格式语义最清楚
new Date("2024-03-15T00:00:00Z");       // 明确 UTC
new Date("2024-03-15T00:00:00+08:00");  // 明确 +8 时区
⚠ 记住三条

（1） 跨时区系统一律用 ISO 字符串传递，带 Z 或明确偏移。 （2） new Date("2024-03-15") 不等于 new Date(2024, 2, 15)（前者 UTC，后者本地）。 （3） 不用 Date.parse(nonStandardString) —— 引擎差异大（IE / Safari / Firefox 行为不同）。

月份从 0 开始的历史

继承自 Java，Java 继承自 C 的 struct tm。历史没有合理原因。getMonth() / setMonth() 返回/接受 0-11（0 = 一月），getDate() 返回 1-31（日）。日期 / 月份分别用不同 getter 也是这个历史。

Setter 的溢出"好心"

setDate、setMonth 等接受超范围的值，自动进位/退位。这是 JS Date 最有用的特性之一：

javascript 
// 获取下个月同一天
const d = new Date(2024, 0, 31);    // 1月31
d.setMonth(d.getMonth() + 1);       // 2月 → 自动溢出到 3月2/3（因为没有 2/31）

// 获取本月最后一天：设为下月 0 号 = 本月最后一天
const lastDay = new Date(year, month + 1, 0);

// 获取 N 天后
const nDaysLater = new Date();
nDaysLater.setDate(nDaysLater.getDate() + n);   // setDate 支持负数 / 超 31

// 获取昨天
const yesterday = new Date();
yesterday.setDate(yesterday.getDate() - 1);

时区与 DST 是全局陷阱

Date 对象不存储时区。显示时用"运行环境的本地时区"解释。这意味着：

  • 同一 Date 对象在纽约服务器显示 "2024-03-14 20:00"，在上海浏览器显示 "2024-03-15 08:00"。
  • 涉及服务端渲染（SSR）时，服务器和浏览器的 locale 必须一致，否则 hydration mismatch。
  • DST（夏令时）会让某些日期"不存在"或"存在两次"。setHours 不会处理 DST 切换。
  • 没有原生方式把一个 Date "转到另一个时区"，只能借助库（date-fns-tz、luxon）或 Intl.DateTimeFormat 的 timeZone 选项仅用于格式化。

Intl.DateTimeFormat 的配置

javascript 
const fmt = new Intl.DateTimeFormat("zh-CN", {
  year: "numeric", month: "2-digit", day: "2-digit",
  hour: "2-digit", minute: "2-digit",
  hour12: false,
  timeZone: "Asia/Shanghai",     // 指定时区 ← 这个参数！
});
fmt.format(new Date());           // "2024/03/15 10:30"

// 预设风格
new Intl.DateTimeFormat("zh", { dateStyle: "full", timeStyle: "short" })
  .format(new Date());             // "2024年3月15日星期五 10:30"

// 所有 locale 支持的时区
Intl.supportedValuesOf("timeZone"); // ["Africa/Abidjan", ...]

// formatToParts 拆分字段（用于自定义拼接）
const parts = fmt.formatToParts(new Date());
// [{type: "year", value: "2024"}, {type: "literal", value: "/"}, ...]

Temporal 提案：未来的时间 API

TC39 Stage 3 提案 Temporal 是 Date 的彻底替代：

  • 区分 PlainDate（无时区）、ZonedDateTime（带时区）、Instant（时间戳）。
  • 所有操作不可变（返回新对象）。
  • 一等公民的时区、日历系统、持续时间。
  • 当前（2026 年）Firefox Nightly、Safari TP 已实现，可用 polyfill @js-temporal/polyfill 试用。

生产代码：继续用 date-fns、dayjs、luxon 之一。避免 moment.js（已进入"维护模式"，且包大）。

03 应用场景

相对时间：Intl.RelativeTimeFormat

javascript 
const rtf = new Intl.RelativeTimeFormat("zh", { numeric: "auto" });
rtf.format(-1, "day");           // "昨天"
rtf.format(0, "day");            // "今天"
rtf.format(3, "hour");           // "3 小时后"

// 工具函数：距离现在多久
function timeAgo(date) {
  const diff = (Date.now() - date.getTime()) / 1000;   // 秒
  const units = [
    [60, "second"], [60, "minute"], [24, "hour"], [30, "day"], [12, "month"], [Infinity, "year"],
  ];
  let n = diff, u = "second";
  for (const [step, unit] of units) {
    if (Math.abs(n) < step) { u = unit; break; }
    n /= step;
  }
  return rtf.format(-Math.round(n), u);
}
timeAgo(new Date(Date.now() - 3600_000));    // "1 小时前"

日期加减：避免时区坑

javascript 
// ❌ 直接加毫秒 —— 穿越 DST 时差会错 1 小时
const oneDay = 24 * 60 * 60 * 1000;
const tomorrow1 = new Date(Date.now() + oneDay);

// ✓ 用 setDate —— 引擎处理 DST
const tomorrow2 = new Date();
tomorrow2.setDate(tomorrow2.getDate() + 1);

// 减法可以直接减（结果是毫秒差）
const ms = endDate.getTime() - startDate.getTime();
const days = Math.floor(ms / oneDay);    // 粗略天数，跨 DST 可能误差 1h

仅日期比较（忽略时分秒）

javascript 
function sameDay(a, b) {
  return a.getFullYear() === b.getFullYear()
    && a.getMonth() === b.getMonth()
    && a.getDate() === b.getDate();
}

function startOfDay(d) {
  const c = new Date(d);
  c.setHours(0, 0, 0, 0);    // 本地时区零点
  return c;
}

持续时间格式化

javascript 
// Intl.DurationFormat（ES2025，Chrome 129+）
const df = new Intl.DurationFormat("zh", { style: "long" });
df.format({ hours: 1, minutes: 30 });   // "1 小时 30 分钟"

// polyfill 时段：手写
function formatDuration(ms) {
  const s = Math.floor(ms / 1000);
  const h = Math.floor(s / 3600);
  const m = Math.floor((s % 3600) / 60);
  const ss = s % 60;
  return [h, m, ss].map(n => String(n).padStart(2, "0")).join(":");
}
formatDuration(3_723_000);             // "01:02:03"

跨时区展示

javascript 
// 把同一个瞬间在不同时区展示
const d = new Date();
["Asia/Shanghai", "America/New_York", "Europe/London"].forEach(tz => {
  const fmt = new Intl.DateTimeFormat("en-US", {
    dateStyle: "short", timeStyle: "short", timeZone: tz,
  });
  console.log(tz, fmt.format(d));
});
// Asia/Shanghai   3/15/24, 10:30 AM
// America/New_York 3/14/24, 10:30 PM
// Europe/London    3/15/24, 2:30 AM

04 典型代码

formatToParts 自定义拼接

javascript 
// 需要 "03月15日 星期五" 这种非默认组合
const fmt = new Intl.DateTimeFormat("zh", {
  month: "2-digit", day: "2-digit", weekday: "long",
});
const parts = fmt.formatToParts(new Date());
// 拼成自己想要的格式
const map = Object.fromEntries(parts.map(p => [p.type, p.value]));
`${map.month}月${map.day}日 ${map.weekday}`;   // "03月15日 星期五"

安全解析 ISO 字符串

javascript 
function parseISO(s) {
  if (typeof s !== "string") return null;
  const d = new Date(s);
  return Number.isNaN(d.getTime()) ? null : d;
}

parseISO("2024-03-15T10:00Z");     // Date
parseISO("bad");                    // null
parseISO("");                       // null  ← ⚠ new Date("") 是 Invalid Date 但 null 被 Number(null)=0 坑到
// 所以必须显式 isNaN 校验

日期区间生成器

javascript 
function* dateRange(start, end) {
  const d = new Date(start);
  while (d <= end) {
    yield new Date(d);
    d.setDate(d.getDate() + 1);
  }
}

for (const d of dateRange(new Date("2024-03-01"), new Date("2024-03-05"))) {
  console.log(d.toISOString().slice(0, 10));
}
// 2024-03-01 ... 2024-03-05

周一开始的本周 / 本月开始结束

javascript 
// 本周一（周一 = 1，周日 = 0 → 周日要减 6 而非 0）
function mondayOfWeek(d = new Date()) {
  const copy = new Date(d);
  const day = copy.getDay() || 7;     // 周日 0 → 7
  copy.setDate(copy.getDate() - day + 1);
  copy.setHours(0, 0, 0, 0);
  return copy;
}

// 本月第一天 / 最后一天
function startOfMonth(d) { return new Date(d.getFullYear(), d.getMonth(), 1); }
function endOfMonth(d)   { return new Date(d.getFullYear(), d.getMonth() + 1, 0, 23, 59, 59, 999); }

05 常见陷阱

月份从 0 开始

new Date(2024, 3, 15) 是 4月15日（月份 0-11）。但new Date("2024-03-15") 是 3月。构造函数和字符串两套系统共存。统一用 ISO 字符串最不容易错。

getDay 返回星期，getDate 返回几号

命名反直觉但无法修改（向后兼容）。记忆：date = 日期数字 (1-31)，day = 星期 (0-6)。

"2024-03-15" vs "2024-03-15T00:00"

前者 UTC 午夜，后者本地午夜。UTC+8 地区差 8 小时。跨系统传递日期永远用带 Z 的完整 ISO。

getTimezoneOffset 的符号反直觉

UTC+8 返回 -480（不是 +480）。历史原因：这是"本地 → UTC"的偏移，符号和常见"UTC → 本地"相反。计算当地毫秒戳用 date.getTime() - date.getTimezoneOffset() * 60_000。

new Date("invalid") 是 Invalid Date，不是抛错

new Date("abc").getTime() 是 NaN；d.toString() 是 "Invalid Date"。校验用 Number.isNaN(d.getTime()) 或 d instanceof Date && !isNaN(d)。

Date.parse(非 ISO) 结果由引擎决定

Date.parse("15/03/2024") 在 Chrome 可能解析成功，Safari 返回 NaN。不要解析非 ISO 格式，要么强制前端格式，要么用 date-fns 的 parse。

JSON.stringify(date) 自动转 ISO 字符串

通过 Date.prototype.toJSON 实现（等价 toISOString()）。好处：传输简单。代价：JSON.parse 后是字符串不是 Date，需要用 reviver 手动还原。详见 12 章 JSON。

DST 跨越时刻的 setHours 可能意外

在夏令时切换日，date.setHours(2) 可能跳过 2:00（春令切换时 2:00 不存在）或有两个 2:00（秋令切换时）。精确日历操作用 Temporal 或 date-fns-tz。

循环里反复 new Intl.DateTimeFormat

创建 Intl 对象有成本（初始化 ICU 数据）。提到模块顶层一次创建、多次 .format，性能差异能有 10-100 倍。
