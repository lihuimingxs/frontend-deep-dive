Cookie / Storage

浏览器提供四种主流客户端存储：Cookie（自动随请求发送、用于身份）、LocalStorage（同步 key-value、持久）、SessionStorage（同步 key-value、标签生命周期）、IndexedDB（异步事务、大容量，下一章）。本章彻底讲清前三种的语义、陷阱与选型。

阅读时长 ~22 分钟 前置 05 CORS 规范 RFC 6265bis · Web Storage

01 速览

  • Cookie 三个维度：作用域（Domain + Path）、生命周期（Expires/Max-Age）、安全属性（Secure + HttpOnly + SameSite + Partitioned）。
  • 2020 年 Chrome 把 SameSite 默认值从 None 改为 Lax；2024 年起第三方 Cookie 进入 CHIPS（分区）时代。
  • LocalStorage / SessionStorage 是 同步阻塞 API，主线程访问会阻塞渲染；只存字符串（对象需 JSON.stringify）。
  • storage 事件只在其他同源标签触发，不触发当前标签（用 BroadcastChannel 或自己派发）。
  • SessionStorage 以 标签页（browsing context）为范围，新标签页开同一 URL 也是独立实例；但 window.open 打开的子页会继承父标签的 SessionStorage。
  • 选型：身份凭证 → HttpOnly Cookie；UI 偏好 → LocalStorage；一次性流程数据 → SessionStorage；大数据/查询 → IndexedDB。

02 底层逻辑

Cookie 属性完整语义

Set-Cookie: sid=abc123;
  Domain=.example.com;      # 作用域：example.com 及所有子域
  Path=/;                   # 作用路径：/ 表示站点内所有路径
  Max-Age=2592000;          # 30 天（秒）；与 Expires 二选一，Max-Age 优先
  Expires=Wed, 23 May 2026 10:00:00 GMT;
  Secure;                   # 只在 HTTPS 发送
  HttpOnly;                 # JS document.cookie 不可见（防 XSS 窃取）
  SameSite=Lax;             # 跨站请求规则
  Partitioned;              # CHIPS：按顶层站点分区（第三方 Cookie 必备）
  Priority=High;            # Chrome 扩展：驱逐时优先保留

Domain 属性的容易误用点

Set-Cookie 来自 Domain 属性 生效范围
api.example.com 未设 仅 api.example.com（host-only）
api.example.com Domain=example.com example.com 及所有子域
example.com Domain=api.example.com 被拒绝（不能设到子域）
example.com Domain=other.com 被拒绝（跨站）

Domain 不允许设 公共后缀列表（.com / .co.uk / vercel.app）。PSL 定义见 publicsuffix.org，浏览器内置副本。

SameSite 三值对比

值 跨站 GET（<a>、top-level nav） 跨站 POST / XHR / iframe 要求
Strict ❌ 不发送 ❌ 不发送 —
Lax（默认） ✅ 发送（仅 GET） ❌ 不发送 —
None ✅ ✅ 必须 Secure；2024 起需 Partitioned

Lax 的豁免窗口：Chrome 曾对"2 分钟内创建、通过 POST top-level 跳转"的 Cookie 发送（"Lax+POST"），2022 年已移除。现在 Lax + POST 严格拒绝。OAuth 回调必须用 GET 或显式 SameSite=None; Secure。

CHIPS（Partitioned Cookie）

Chrome 2024 起逐步废弃未分区的第三方 Cookie。原来一个 ads.example.com 在所有嵌入它的网站共享一份 Cookie；CHIPS 改为按顶层站点分区： 

Set-Cookie: id=123;
  SameSite=None; Secure;
  Partitioned

用户访问 news.com（其中嵌入 ads.example.com iframe）时 Cookie 存在 (news.com, ads.example.com) 这对 key 下；访问 blog.com 时用另一份。跨站追踪被阻断，但嵌入式三方服务（聊天 widget、SSO iframe）仍能工作。

Cookie 容量与优先级

  • 每个 Cookie ≤ 4096 字节（name + value + attributes）。
  • 每个 domain ≤ 180 个（Chrome），所有 domain 合计 ≤ 3000。
  • 超限时浏览器按 Priority（Low/Medium/High）+ LRU 驱逐。
  • Session Cookie（无 Max-Age/Expires）关闭浏览器即失效，但 Chrome "继续运行后台应用" 会保留。

document.cookie vs Cookie Store API

// 传统 API：同步、字符串拼接、丑陋
document.cookie; // "a=1; b=2; sid=abc"  ← 读所有 Cookie 但不含 HttpOnly
document.cookie = "theme=dark; Max-Age=31536000; Path=/; SameSite=Lax";
// ↑ 只能一次设一个 Cookie，且看起来像覆盖整个 cookie 但实际是追加

// 现代 API：异步、结构化（Chrome 87+，Safari 未支持）
await cookieStore.set({
  name: "theme",
  value: "dark",
  expires: Date.now() + 31536000000,
  sameSite: "lax",
  partitioned: true,
});
const cookie = await cookieStore.get("theme");
cookieStore.addEventListener("change", (e) => {
  console.log(e.changed, e.deleted);
});

Cookie Store API 在 Service Worker 中很有用（SW 内 document 不可用）。生产环境需判断 "cookieStore" in window。

LocalStorage / SessionStorage 的本质

存储是同源隔离的 key-value map，但 API 设计早于 IndexedDB（2009 HTML5），规范要求同步执行： 

localStorage.setItem("key", value);    // 同步：阻塞主线程直到写入磁盘
const v = localStorage.getItem("key");  // 同步：阻塞读取
localStorage.length;                    // 同步
localStorage.key(0);                    // 同步
localStorage.removeItem("key");
localStorage.clear();

// 也可用属性语法（不推荐）
localStorage.key = "value";
localStorage["key"] = "value";

为什么同步是问题：Chrome 把 LocalStorage 存在 LevelDB / SQLite 中，进程隔离后写入跨进程。在低端 Android 上一次 setItem 可能 10-50ms，触发 long task，影响 INP。Firefox 曾把它改成内存缓存 + 异步落盘，但规范要求同步读写视图，仍会阻塞。

容量与计费

  • LocalStorage / SessionStorage：每个 origin 约 5MB（UTF-16 编码，实际字符数减半）。
  • 超限抛 QuotaExceededError（DOMException name），不是普通 Error。
  • IndexedDB、Cache Storage 共享更大配额（navigator.storage.estimate()），详见 P6-07。

storage 事件的怪异之处

// 标签 A
localStorage.setItem("theme", "dark");

// 标签 B（同源、已打开）收到事件
window.addEventListener("storage", (e) => {
  console.log(e.key, e.oldValue, e.newValue, e.url, e.storageArea);
});

// 标签 A 自己不会收到 storage 事件！
// 跨标签通信可用，同标签内监听无效。

同一标签内监听 LocalStorage 变化需自己发自定义事件或用 BroadcastChannel（详见 P6-08）。事件只在主线程触发，Worker 内的 storage 事件不支持。

SessionStorage 的继承规则

操作 新页面的 SessionStorage
地址栏输入 URL 全新、空
复制标签（Chrome "Duplicate tab"） 完全复制父标签的 SessionStorage
window.open("/b") 继承父标签的 SessionStorage
<a href="/b" target="_blank" rel="noopener"> 全新、空（noopener 切断继承）
<a target="_blank">（无 noopener） 继承（但存在 tab-nabbing 风险，默认不推荐）
刷新同一标签（F5） 保留
关闭后恢复（Ctrl+Shift+T） 保留

iframe 的 SessionStorage 与父页独立（按源分区）；如果 iframe 同源，两者独立但通过 parent.sessionStorage 可互访。

03 应用场景与选型

决策表

数据 推荐 理由
Session ID / Auth Token HttpOnly Cookie 防 XSS 窃取；自动发送；服务端可撤销
CSRF Token Cookie + 响应 body / header Double Submit 模式（见 P6-05）
CDN / 灰度标识 Cookie（Secure + Lax） CDN 在 Edge 读 Cookie 分流
UI 偏好（主题、语言） LocalStorage 客户端独享、不必上传服务端
表单草稿 LocalStorage + debounce 防止刷新丢失；定期清理
多步流程中间态 SessionStorage 关闭标签自动清理，避免泄漏
OAuth state 参数 SessionStorage 仅此标签需要，回调后立即清理
离线数据 / 查询 IndexedDB 异步、事务、索引（见 P6-07）
大文件 / 图片 Cache API Response 对象原生支持（见 P6-08）

典型的跨站请求 Cookie 故障

前端 app.com 调用后端 api.com：

# 后端设置 Cookie
Set-Cookie: sid=abc; SameSite=Lax; HttpOnly

# 前端 fetch("https://api.com/me", { credentials: "include" })
# → Cookie 不发送！
# 原因：SameSite=Lax 在跨站 XHR/fetch 中不发送
# 修复：SameSite=None; Secure; Partitioned
Set-Cookie: sid=abc; SameSite=None; Secure; HttpOnly; Partitioned

同时后端 CORS 必须 Access-Control-Allow-Credentials: true 且 Allow-Origin 不为 *（见 P6-05）。

防止 LocalStorage 阻塞主线程

// ❌ 启动时批量读取，阻塞 FCP
const settings = JSON.parse(localStorage.getItem("settings") ?? "{}");
const cache = JSON.parse(localStorage.getItem("cache") ?? "{}");
const history = JSON.parse(localStorage.getItem("history") ?? "[]");
// → 三次同步 IO + 三次 JSON.parse，可能 20-100ms

// ✅ 只在需要时读，且用 requestIdleCallback 预热
requestIdleCallback(() => {
  const settings = JSON.parse(localStorage.getItem("settings") ?? "{}");
  window.__settings = settings;
});

// ✅ 大数据用 IndexedDB（异步）
import { get } from "idb-keyval";
const cache = await get("cache"); // 不阻塞主线程

Tab 间状态同步

// 方案 1：storage 事件（简单，但自标签不触发）
function broadcastSetItem(key, value) {
  localStorage.setItem(key, value);
  // 手动派发，让自标签也感知变化

Next.js 场景

// middleware.ts：Edge 读 Cookie 做认证
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(req: NextRequest) {
  const sid = req.cookies.get("sid")?.value;
  if (!sid && req.nextUrl.pathname.startsWith("/app")) {
    return NextResponse.redirect(new URL("/login", req.url));
  }
  const res = NextResponse.next();
  // 设置/更新 Cookie
  res.cookies.set("trace", crypto.randomUUID(), {
    httpOnly: true,
    secure: true,
    sameSite: "lax",
    maxAge: 3600,
  });
  return res;
}

// Server Action：读 Cookie
import { cookies } from "next/headers";
const sid = (await cookies()).get("sid")?.value;

04 典型代码

封装 SafeStorage（处理 QuotaExceededError、隐私模式）

interface SafeStorage {
  get<T>(key: string, fallback: T): T;
  set(key: string, value: unknown): boolean;
  remove(key: string): void;
}

function createSafeStorage(area: "local" | "session"): SafeStorage {
  const storage = area === "local" ? localStorage : sessionStorage;

  return {
    get(key, fallback) {
      try {
        const raw = storage.getItem(key);
        return raw === null ? fallback : JSON.parse(raw);
      } catch {
        // JSON 损坏 → 清理并返回 fallback
        storage.removeItem(key);
        return fallback;
      }
    },
    set(key, value) {
      try {
        storage.setItem(key, JSON.stringify(value));
        return true;
      } catch (e) {
        // QuotaExceededError 或隐私模式（Safari 早期）
        if (e instanceof DOMException && e.name === "QuotaExceededError") {
          // 策略 1：清理旧缓存再试
          evictOldEntries(storage);
          try { storage.setItem(key, JSON.stringify(value)); return true; }
          catch { return false; }
        }
        return false;
      }
    },
    remove(key) { storage.removeItem(key); },
  };
}

function evictOldEntries(storage: Storage) {
  // 清理带过期时间的缓存
  for (let i = storage.length - 1; i >= 0; i--) {
    const key = storage.key(i);
    if (!key || !key.startsWith("cache:")) continue;
    try {
      const { expiresAt } = JSON.parse(storage.getItem(key) ?? "{}");
      if (expiresAt < Date.now()) storage.removeItem(key);
    } catch { storage.removeItem(key); }
  }
}

Cookie 辅助函数（处理 encodeURIComponent + HttpOnly 不可见）

function getCookie(name: string): string | undefined {
  const target = encodeURIComponent(name);
  // 注意：HttpOnly Cookie 读不到；只能读非 HttpOnly
  const match = document.cookie
    .split("; ")
    .find((row) => row.startsWith(target + "="));
  return match ? decodeURIComponent(match.slice(target.length + 1)) : undefined;
}

function setCookie(
  name: string,
  value: string,
  opts: { maxAge?: number; path?: string; sameSite?: "lax" | "strict" | "none"; secure?: boolean } = {},
) {
  let cookie = `${encodeURIComponent(name)}=${encodeURIComponent(value)}`;
  if (opts.maxAge != null) cookie += `; Max-Age=${opts.maxAge}`;
  cookie += `; Path=${opts.path ?? "/"}`;
  cookie += `; SameSite=${opts.sameSite ?? "lax"}`;
  if (opts.secure ?? location.protocol === "https:") cookie += "; Secure";
  document.cookie = cookie;
}

function deleteCookie(name: string, path = "/") {
  document.cookie = `${encodeURIComponent(name)}=; Max-Age=0; Path=${path}`;
}

React Hook：useLocalStorage（含 storage 事件同步）

import { useState, useEffect, useCallback } from "react";

export function useLocalStorage<T>(key: string, initialValue: T) {
  const [value, setValue] = useState<T>(() => {
    if (typeof window === "undefined") return initialValue; // SSR
    try {
      const raw = localStorage.getItem(key);
      return raw === null ? initialValue : JSON.parse(raw);
    } catch {
      return initialValue;
    }
  });

  // 监听其他标签的修改
  useEffect(() => {
    const onStorage = (e: StorageEvent) => {
      if (e.key !== key || e.storageArea !== localStorage) return;
      try {
        setValue(e.newValue === null ? initialValue : JSON.parse(e.newValue));
      } catch { /* ignore */ }
    };
    window.addEventListener("storage", onStorage);
    return () => window.removeEventListener("storage", onStorage);
  }, [key, initialValue]);

  const set = useCallback(
    (v: T | ((prev: T) => T)) => {
      setValue((prev) => {
        const next = typeof v === "function" ? (v as (p: T) => T)(prev) : v;
        try { localStorage.setItem(key, JSON.stringify(next)); } catch {}
        return next;
      });
    },
    [key],
  );

  return [value, set] as const;
}

05 常见陷阱

陷阱 1：把 JWT 存 LocalStorage 就完了

LocalStorage 对任何同源 JS 可见 —— 只要有一处 XSS（CDN 脚本被攻破、第三方库被投毒），token 全泄。改用 HttpOnly; Secure; SameSite=Lax Cookie，JS 根本读不到。

陷阱 2：SameSite=Strict 破坏第一次访问

用户在 Google 搜索点击你的站，Strict 的登录 Cookie 不发送，用户看到未登录页。用 Lax 是标准做法；Strict 只适合银行级敏感操作。

陷阱 3：Secure 属性在 HTTP 本地开发时被静默忽略（但部署时生效）

本地 http://localhost 浏览器特殊放行 Secure Cookie；部署到 https://staging.com 也正常。但如果某环境走 HTTP（某些企业内网），带 Secure 的 Cookie 不发送，登录循环。

陷阱 4：document.cookie 赋值像覆盖，实际是追加

document.cookie = "a=1" 再 document.cookie = "b=2"，两个 Cookie 都存在。删除要用 Max-Age=0 或过期 Expires，且 Path 和 Domain 必须完全匹配原 Cookie 否则删不掉。

陷阱 5：LocalStorage 跨子域不共享

app.example.com 和 www.example.com 的 LocalStorage 是独立的。想共享必须用 Cookie（Domain=.example.com）或 iframe postMessage。

陷阱 6：Safari ITP（Intelligent Tracking Prevention）

Safari 对第三方 Cookie 默认阻止；对通过 document.cookie 设置的第一方 Cookie 也限制 7 天过期（即使你设 Max-Age=1 年）。通过 Set-Cookie header 设置的 Cookie 不受影响 —— 登录凭证必须走服务端 Set-Cookie。

陷阱 7：JSON.stringify 丢失 Date / Map / Set / undefined

localStorage.setItem("k", JSON.stringify(obj)) 中 Date 变字符串、undefined 字段被删、函数丢失、Map/Set 变空对象。需要保留这些类型用 superjson 或 IndexedDB（后者原生支持 structured clone）。

陷阱 8：隐私模式下 setItem 直接抛

Safari 早期版本在 Private Browsing 下 LocalStorage 可见但写入即抛 QuotaExceededError（quota=0）。任何写入必须 try/catch，否则整个页面 JS 崩溃。

陷阱 9：SessionStorage 以 tab 为单位但 Duplicate Tab 会完全复制

依赖 SessionStorage 存"当前 OAuth state"时，用户复制标签，两个标签同时处理回调，state 冲突。解决：绑定 performance.navigation.type 或使用 OAuth nonce。

陷阱 10：storage 事件 newValue 为 null 代表删除，不是空字符串

监听 storage 事件时 e.newValue === null 表示 removeItem/clear；e.newValue === "" 表示被设为空字符串。两者语义不同。
