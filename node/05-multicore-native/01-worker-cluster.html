Worker Threads / Cluster / Child Process

Node "单线程"是 JS 层面的——通过三种多进程 / 多线程模型可以充分利用多核 CPU。三者各有适用场景：Worker Threads（CPU 密集 JS）/ Cluster（I/O 密集 web server 多实例）/ Child Process（调外部程序）。讲清三者选型判据 + Worker Threads 与 V8 Isolate 的关系。

阅读时长 ~30 分钟 前置 V8 P5.1 Isolate 主要来源 nodejs.org/api/worker_threads · cluster · child_process

01 速览

  1. 三者选型判据：CPU 密集 → Worker Threads；多 web server 实例 → Cluster；调外部程序 → Child Process。
  2. Worker Threads = 多 V8 Isolate（详见 V8 P5.1）——互相完全隔离，postMessage 必须 structured clone。
  3. SharedArrayBuffer 是 Workers 真正共享内存的唯一方式——其他都是拷贝。

02 底层逻辑

worker_threads（Node 10.5+）：进程内多 Isolate

worker_threads 是 Node 10.5（2018 实验）/ Node 12（稳定）加入的多线程模型。"多线程"在这里有特殊含义：

  • 进程内多 V8 Isolate—— 每个 Worker 是独立的 V8 Isolate（详见 V8 P5.1），自己的堆 / GC / EventLoop。不是 POSIX 线程
  • 主线程 ↔ Worker 通信走 postMessage—— 用 structured clone 算法（与浏览器 Web Worker 一致）。不能直接传函数 / Promise—— 序列化不支持
  • SharedArrayBuffer 是真正共享内存的唯一方式—— 其他对象都是拷贝
  • 适合 CPU 密集 JS：图像处理 / 加密 / AI 推理 / 数据解析。把这些任务从主线程搬到 Worker，主线程 EventLoop 不被阻塞，仍能服务其他请求
  • 不适合 I/O 密集—— Node 主线程 EventLoop 本身就足够 I/O 并发，开 Worker 反而增加 postMessage 开销

cluster 模块：fork 多进程跑同一代码

cluster 是更早就有的多核模型（Node 0.6+）—— fork 多个 Node 进程跑同一段代码，OS 内核做请求负载均衡：

  • master 进程 listen 端口，fork N 个 worker 进程；worker 进程通过共享 socket file descriptor 处理请求
  • OS 内核做负载均衡—— Linux 3.9+ 的 SO_REUSEPORT（多个 socket 监听同一端口，内核轮转分配新连接）
  • 每个 worker 是独立进程—— 崩溃互不影响。一个 worker OOM 不会拖垮全部
  • 适合 web server 多核扩展—— Node 单线程不能用多核 CPU，cluster 起 N 个 worker（N = CPU 核数）
  • 不适合 worker 之间需要共享状态的场景—— 进程隔离意味着没有共享内存

2025 年实践：cluster 仍能用，但大多数生产部署不直接用 cluster—— 而是把"多进程"交给容器编排（k8s 起多 pod）或 process manager（PM2 提供更好的监控 / 重启策略）。cluster 直接用主要在简单单机部署。

child_process：调外部程序或子 Node

child_process 是 Node 与外部程序交互的接口。和前两个不同—— child_process 子进程不必是 Node，可以是任意可执行程序（ffmpeg / git / python / shell 脚本）。

  • spawn：低层 API，返回 streaming I/O（stdin / stdout / stderr 都是 stream）。大输出场景必用——避免 buffer 爆内存
  • fork：spawn 的 Node 特化版—— 子进程也是 Node，自带 IPC channel（process.send() / 'message' 事件）。Node 多进程间通信用这个
  • exec：buffer 整个输出。仅适合短命令（< 200KB 输出）—— 大输出会爆内存或 hang
  • execFile：exec 但直接 spawn 程序不走 shell（更安全 / 更快）—— 注入风险低

安全提醒：如果命令字符串包含用户输入，绝对不要用 exec—— 用 execFile 或 spawn + 参数数组形式（避免 shell injection）。

SharedArrayBuffer + Atomics：真正共享内存

Worker Threads 之间默认所有数据都拷贝—— postMessage 走 structured clone。这对小数据 OK 但大数据（几十 MB 图像 / 视频帧）拷贝开销巨大。

SharedArrayBuffer（SAB）是 ES2017 标准 + Node Worker Threads 内置——多个 Worker 真正共享同一块内存，零拷贝。代价是必须用 Atomics 原子操作避免 race condition：

  • Workers 之间真正共享内存的唯一方式
  • 必须用 Atomics.add / Atomics.compareExchange / Atomics.wait 等避免数据竞争
  • 详细规范见 ECMA P7.5 · §24 Memory Model

典型用例：(1) 多 Worker 共享配置 / 状态（避免重复 deserialize）；(2) 大 image buffer 多 Worker 协同处理（每个 Worker 处理图像一部分）；(3) 实现 lock-free queue / ring buffer 在 Worker 间传消息。

MessageChannel / MessagePort：双向通信原语

除了 postMessage 主线程 ↔ Worker 通信，Node 还支持 MessageChannel 让任意两个 Worker 直接对话—— 不需要主线程中转：

  • const { port1, port2 } = new MessageChannel() 创建一对端口
  • 把 port1 给 Worker A，port2 给 Worker B（用 transferList 转移所有权）
  • 之后 Worker A.postMessage(...) → Worker B 直接收
  • 主线程不参与，零额外开销

典型用例：worker pool 之间任务分发 / 多 Worker 协同（如 web server worker 之间共享 cache invalidation 消息）/ Web Worker 风格的 actor 模型。

03 应用场景

场景 A · CPU 密集任务用 Worker Threads

web server 接到一个图像处理请求——不能直接在主线程跑 sharp（会卡 EventLoop，所有其他请求等几百 ms）。修法：维护 worker pool（用 piscina npm 包封装），把图像处理任务发到 pool。主线程 EventLoop 不被阻塞，仍能服务其他请求。worker 数量 = CPU 核数。

场景 B · web server 用 Cluster 还是 PM2 还是容器

判据：(1) 本地 dev / 简单部署：cluster 直接用；(2) VM / 单机生产：PM2（自带监控 / 自动重启 / 日志聚合）；(3) 容器化部署（k8s / Docker Swarm）：不用 cluster 也不用 PM2—— 让容器编排起多 pod / replica，每个容器跑单进程 Node。这是 2025 年大多数生产部署的现状。

场景 C · 调 ffmpeg 用 spawn 不用 exec

ffmpeg 转码大文件输出到 stdout——必须 spawn，把 stdout 当 Readable stream 读 / pipe 到目标。如果用 exec，整个 ffmpeg 输出会被 buffer 在 Node 进程里——视频文件可能几 GB 直接爆内存。规则：任何大输出 / 流式命令用 spawn；短输出（如 git rev-parse HEAD）才用 exec。

场景 D · 三件套选型快速判据

"什么场景用什么": (1) CPU 密集 JS（图像处理、加密、AI 推理）→ Worker Threads；(2) 多核 web server（生产部署）→ 容器编排起多实例，cluster 仅 dev 用；(3) 调用外部程序（ffmpeg / git / python / 编译器）→ child_process（首选 spawn）；(4) I/O 密集（数据库 / 网络）→ 不需要任何多核扩展—— Node 单线程 EventLoop 已经能扛几万并发。

04 典型代码

javascript 
// Worker Threads 跑 CPU 密集任务
import { Worker } from 'node:worker_threads';

function runCPUTask(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./cpu-task.js', { workerData: data });
    worker.on('message', resolve);
    worker.on('error', reject);
  });
}

// Cluster web server
import cluster from 'node:cluster';
import { availableParallelism } from 'node:os';
import http from 'node:http';

if (cluster.isPrimary) {
  for (let i = 0; i < availableParallelism(); i++) cluster.fork();
} else {
  http.createServer((req, res) => res.end('OK')).listen(3000);
}

05 常见陷阱

postMessage 不能传函数 / 不能传 Promise

structured clone 不支持。要跨 Worker 传函数，用 MessageChannel + 自定义协议。

Cluster 不是真正的"线程"

每个 worker 是独立进程；不能直接共享 V8 堆 / 内存。

SharedArrayBuffer 必须配 Atomics

无 Atomics 直接读写会触发 V8 内部内存模型 UB；必然 race。
