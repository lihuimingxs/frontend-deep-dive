N-API 与 Native Module

用 C++ 写性能关键代码 + 调外部 C 库。N-API 是 Node 给 native module 的稳定 ABI 接口——保证用 N-API v8 写的二进制能在所有 Node 18+ 跑，不需重新编译。讲清三种写法（raw V8 / N-API / node-addon-api）+ node-gyp 工程实践 + WASM 替代方案。

阅读时长 ~25 分钟 前置 V8 P5.1 Embedder API 主要来源 nodejs.org/api/n-api · node-addon-api 文档
📝 章节状态

骨架版本。对照大纲 node/outline.md § P5.2。

01 速览

  1. 三种 native module 写法：raw V8（最灵活）/ N-API（C 接口稳定 ABI）/ node-addon-api（C++ 包装，95% 现代库选择）。
  2. N-API ABI 稳定性：用 N-API v8 写的二进制能在所有 Node 18+ 跑——不需重新编译。
  3. WASM 是越来越多 native 需求的替代：esbuild / parcel-css 用 wasm 部署；跨平台、无 node-gyp 痛苦。

02 底层逻辑

三种 native module 写法

正文待写。

  • raw V8：直接调 V8 C++ API；最灵活、最快、最严绑定 V8 版本
  • N-API（C 接口）：跨 Node 版本二进制兼容；C 风格写法 verbose
  • node-addon-api（C++ 包装）：N-API 之上的 C++ 类包装；写法接近 raw V8 但保持 N-API 兼容性；95% 现代 native module 的选择

N-API ABI 稳定性承诺

正文待写。

  • N-API v8 / v9 是版本化的 stable ABI
  • 承诺：用 N-API v8 写的 native module 二进制，从此后 Node 版本编译的都能跑，不需重新编译
  • 对比直接用 V8：V8 升级会破坏二进制兼容性
  • 这套机制让 Node 生态的 native module 不再需要"针对每个 Node 版本编译一份"

典型 native module

正文待写。

  • better-sqlite3：同步 SQLite 绑定；性能比 node-sqlite3 快几倍
  • bcrypt：密码哈希原生实现
  • sharp：图像处理，绑定 libvips
  • node-gyp / node-pre-gyp / better-call：FFI 工具

node-gyp 构建系统

正文待写。

  • 基于 GYP（Google）的 fork
  • 用 binding.gyp 文件定义编译选项（include 目录、链接库、平台条件等）
  • 跨平台编译（Linux gcc / macOS clang / Windows MSVC）
  • 用户安装 npm 包时本地编译——这是 native module 安装慢 / 失败的主要原因

prebuild-install

正文待写。

  • 避免用户安装时编译——下载预编译二进制
  • 开发者 publish 时用 prebuild 工具上传所有平台 / 架构 / Node 版本组合的二进制
  • 用户 npm install 时尝试匹配 prebuild 二进制；匹配失败才回退到 node-gyp 编译
  • 实际生产 native module 几乎都用这套——没有预编译的库口碑会很差（安装慢 / 失败率高）

WASM 替代方案

正文待写。

  • 越来越多 native 需求改用 wasm（如 esbuild 用 Go + wasm，parcel-css 用 Rust + wasm）
  • 优势：部署简单（一份 .wasm 文件全平台跑）/ 跨平台 / 无需 node-gyp 痛苦
  • 劣势：性能略弱于 native（跨边界税）/ 不能调 OS 级 API
  • 选型：纯计算用 wasm；要调系统库（如 libvips / libsqlite）仍用 native module

04 典型代码

cpp 
// 用 node-addon-api 写一个简单的 native 函数
#include <napi.h>

Napi::Number Add(const Napi::CallbackInfo& info) {
  Napi::Env env = info.Env();
  double a = info[0].As<Napi::Number>().DoubleValue();
  double b = info[1].As<Napi::Number>().DoubleValue();
  return Napi::Number::New(env, a + b);
}

Napi::Object Init(Napi::Env env, Napi::Object exports) {
  exports.Set("add", Napi::Function::New(env, Add));
  return exports;
}

NODE_API_MODULE(addon, Init)

05 常见陷阱

用 raw V8 直接绑定 Node 版本

除非写 V8 自家组件，否则永远用 N-API / node-addon-api。

没配 prebuild-install 用户体验差

每次 npm install 都要本地编译，慢且容易失败。生产 native 库必配 prebuild。

node-gyp 在 Node 版本切换时易坏

nvm / fnm 切版本后可能 binding 失效，需要 npm rebuild。
