编写约定

每章遵循统一的 6 段结构：速览 → 底层逻辑 → 应用场景 → 典型代码 → 常见陷阱 → 延伸阅读。Node 主题相比 V8 / ECMA 有以下额外约定：

  • 「短重述 + 链回」 —— 当一个机制在 V8 / ECMA / JavaScript 主题已详述时（如 V8 Isolate / Microtask Queue 原语 / Module Records 规范 / Promise 用户视角），本主题用 1-3 句话点到本质，紧跟一句"详细的 X 请参考 Y"
  • 版本号必标 —— Node 改动也不慢，没有版本号的论述不可信。每个特性必标"Node X 引入 / Node Y 稳定"
  • nodejs.org 一手优先 —— 每个 API 的"权威定义"必须是 nodejs.org/api。社区博客只用来佐证或补充实测数据
  • 重视 RFC 与 Issue —— 很多 Node 设计决策记录在 nodejs/node GitHub PR / Issue 里——这是讲"为什么"最权威的资料
  • ⚠️ 标注 —— io.js 永远标"2014-2015 abandoned"；早期 callback API 不再推荐永远标"用 fs.promises"

与既有主题的关系

Node 主题不重复 V8 / ECMA / JS 主题已经讲透的底层机制，但会在每个交叉点提供"短重述 + 链接"，重点讲 Node 在应用层做的包装。常见对应：

  • P2 Event Loop → V8 P5.1 Embedder API（V8 不拥有 EL，只暴露 microtask 原语）+ ECMA P7.3 Event Loop 规范
  • P3 模块系统 → ECMA P7.4 Module Records + TLA（规范定义）
  • P4.3 Buffer → V8 P4.1 堆布局（Buffer 不在 V8 堆，由 ArrayBuffer Allocator 分配）
  • P5.1 Worker Threads → V8 P5.1 Embedder API（每 Worker 一个 Isolate）+ ECMA P7.5 §24 Memory Model（SAB + Atomics）
  • P5.2 N-API → V8 P5.1 Embedder API（V8 C++ API 之上的 ABI 稳定层）
  • P6 Inspector Protocol → V8 P5.2 诊断工具（V8 同一套协议）
  • P1.2 同代人 Bun → 独立的 Bun 主题（待写，讲 JSC + Zig + 工具一体化）
  • P1.2 同代人 Edge Runtime → V8 P1.2 V8 分身史（Cloudflare Workers Isolate-per-tenant）
