CJS 与 ESM 双系统：解析算法与配置

把"为什么我的 import 跑不起来"这种典型问题的根因讲清楚。重点是解析算法 + package.json 字段。CJS 用 Node 自家算法，ESM 用 ECMA 规范算法——同一个路径可能解析成不同文件，这是双系统并存最大的工程现实。

阅读时长 ~30 分钟 前置 ECMA P7.4 Module Records 主要来源 nodejs.org/api/esm · nodejs.org/api/packages

01 速览

  1. CJS 是"运行时模块"：被 require 时执行；同步；可以条件 require。
  2. ESM 是"静态模块"：解析时 V8 已知所有依赖；异步；不能条件 import（除非用 import()）。
  3. package.json 的 exports 字段是双系统兼容的核心——通过 conditional exports 给不同 import 方式不同入口。

02 底层逻辑

CommonJS 解析算法（Node 2009-）

CJS 是 Node 自创的模块系统（2009），不是 ES 标准—— ECMAScript 直到 2015 才有 ESM。CJS 设计选择影响深远：

  • 同步加载：require('foo') 阻塞当前执行直到加载完。这是服务端 OK 的设计（启动时加载完所有依赖再开始服务），但浏览器不能用——浏览器加载用 HTTP 是异步的。这是 CJS 不能直接搬到浏览器的根本原因，也是 ESM 必须异步的设计动机。
  • Node 自家解析算法（不是 ES 规范）：先看 core module（fs / http 等内置）→ 再看 file 路径 → 再看 node_modules 从当前目录向上查找。这套算法被 npm 生态采用了 16 年，是事实标准但不是规范标准
  • 支持省略扩展名：require('./foo') 自动尝试 foo.js / foo.json / foo.node（native module）/ foo/index.js
  • 支持条件 require：可以 if (process.env.NODE_ENV === 'production') require('./prod-config')—— 因为 require 是同步的运行时函数调用
  • module.exports 是动态对象：可以用 module.exports = function() {} 整体替换，也可以分别 module.exports.foo = ... 加属性

ESM 解析（Node 12+）：完全不同的世界

ESM（ECMAScript Modules）是 ES2015 规范定义的模块系统。设计完全不同：

  • 异步加载：import 是规范级 async——浏览器和 Node 都能用
  • 按 ECMA Module Records 规范解析（详见 ECMA P7.4）—— 严格的"parse → link → evaluate"三阶段
  • 支持 HTTP / file URL：import x from "https://cdn.com/foo.js" 浏览器能跑，Node 默认禁用网络 import（安全 + 性能）
  • 必须带扩展名（.js / .mjs）—— 没有"自动尝试"。这是 ESM 与 CJS 最大解析差异，也是新人最常踩的坑
  • 不能条件 import：import 必须在文件顶层、不能在 if 里。动态 import() 才能条件加载（返回 Promise）
  • 导出是静态绑定：export const foo 之后 foo 不能改赋值——Module Record 锁死

ESM 的"静态"特征让 bundler / tree-shaking 等工具能编译期分析依赖图——这是 CJS 做不到的（CJS 的条件 require 让静态分析必然不完整）。

package.json 字段全谱：Node 模块系统的"控制面板"

package.json 决定 Node 怎么解析这个包。关键字段：

  • type: "module"：让本包内 .js 文件按 ESM 处理。默认（不写）是 CJS。选这个 flag 是单向决定——以后这个包永远是 ESM
  • main: "./dist/index.cjs"：CJS 入口（旧字段，2009 起）
  • module: "./dist/index.js"：ESM 入口（bundler 约定，不是 Node 标准，但所有打包器都看）
  • exports：现代标准（Node 12.7+），覆盖以上 main/module 字段。复杂场景的唯一正确选择
  • imports：包内导入别名。如 "imports": {"#config": "./src/config.js"} 让本包文件用 import x from '#config'—— 避免相对路径地狱（../../../config）
  • types / typings：TypeScript 类型入口（指向 .d.ts）

Conditional Exports：key 顺序敏感

exports 字段最重要的能力是 conditional exports——给不同 import 方式映射不同入口。key 顺序敏感：earlier 优先级更高，Node 找到第一个匹配就停。

json 
{
  "name": "my-lib",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./utils": {
      "import": "./dist/utils.js",
      "require": "./dist/utils.cjs"
    }
  }
}

key 顺序敏感：earlier entries 优先级更高。一般规则"specific → general"——types 在前，import 在后。

导入时的扩展名问题：TS + ESM 最大踩坑点

扩展名规则在 CJS / ESM / TS 三端不同：

  • CJS：require('./foo') 自动加 .js / .json / .node 后缀。可以省扩展名
  • ESM：import x from './foo.js' 必须带 .js 后缀。不能省
  • TypeScript + ESM 的奇怪交互：源文件叫 foo.ts，但 import 时写 import x from './foo.js'（带 .js 不是 .ts）。tsc 编译后输出 foo.js，import 路径正好匹配

为什么 TS 这么诡异？因为 tsc 默认不改 import 路径—— TS 5 NodeNext 模式要求"路径写运行时实际路径"。所以你在 .ts 源码里写的 .js 后缀，编译时不被改写，Node 跑编译产物时刚好能解析。

这是 TS + ESM 项目最常见的踩坑——新人想"我在写 .ts，为什么要写 .js"。理解后就清楚——TS 不是"更好的 JS"，是编译期 + 类型擦除，运行时跑的是 .js。

--loader Hooks：自定义模块加载器

Node 提供 module loader hook 接口，让外部工具能介入模块解析过程。典型用例：ts-node（运行 .ts 不预编译）/ Jest ESM / Vitest / swc-node。这些工具注册 loader hook，把 import './foo.ts' 拦截后即时编译成 JS 再返回。

API 演化：

  • Node 12-19：--experimental-loader（实验性）+ --experimental-vm-modules（Jest 等用）
  • Node 20+：改成 --import + 内部 module.register() API（更清晰、稳定）

实际项目通常不直接写 loader——用 ts-node / tsx / Vitest 等已封装好的工具就够。但理解这层机制有助于排障"为什么我的 import 行为奇怪"。

03 应用场景

场景 A · 给项目从 CJS 迁到 ESM

老 Node 项目从 CJS 迁 ESM 的关键步骤：(1) 加 "type": "module"；(2) 所有 require → import；(3) 加 .js 后缀；(4) __dirname / __filename 改用 import.meta.url + fileURLToPath；(5) 检查所有 条件 require 改成动态 import()；(6) 检查依赖里是否有纯 CJS 包—— 仍可被 ESM import，但默认导出可能行为不同。

场景 B · 调试 import 解析失败

"我的 import 跑不起来"几乎都是 5 类问题之一：(1) 忘加 .js 后缀；(2) "type": "module" 没设；(3) 包的 exports 字段没列你想要的子路径；(4) 依赖是 CJS 但你想 named import（CJS 默认只有 default export）；(5) ESM 文件用了 __dirname。诊断工具：node -e "console.log(require.resolve('foo'))" 看 CJS 实际解析；node --trace-warnings 看 ESM 警告。

场景 C · TS + ESM 项目正确写法

tsconfig 关键配置："module": "NodeNext" + "moduleResolution": "NodeNext"。源文件用 import x from './foo.js'（.js 不是 .ts）。package.json 加 "type": "module"。如果用 ESM-only 工具（vitest / tsx）配合得好；如果生产用 tsc 编译再用 node 跑，按 NodeNext 规范来不会有问题。

场景 D · 库作者：发什么格式

写库的人面对的选择：(1) 只发 CJS（兼容老 Node，新项目能 import）—— 安全选；(2) 只发 ESM（现代，Node 22+ 起 require ESM 实验支持）—— 激进选；(3) 发双格式（CJS + ESM + .d.ts）—— 大多数库的现代选择，用 tshy / tsup / unbuild 工具自动产出。详见 P3.2 Dual Package Hazard。

04 典型代码

json 
// 现代库 package.json 双格式发布
{
  "name": "my-lib",
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs",
      "default": "./dist/index.js"
    }
  }
}

05 常见陷阱

ESM 里没有 __dirname / __filename

用 import.meta.url + fileURLToPath 替代。

conditional exports 顺序不对会解析错文件

types 必须放最前；import / require 顺序也敏感。

TS 里 import 必须带 .js 不是 .ts

这是 TS5 NodeNext 模式的"路径写运行时实际路径"约定。
