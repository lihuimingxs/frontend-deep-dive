Dual Package Hazard 与现代库设计

实战章。当一个库要同时支持 CJS 和 ESM 用户时会撞上"双实例"问题——require('pkg') 和 import 'pkg' 加载的是两个独立实例，单例失效、instanceof 失败、状态不一致。讲清楚怎么规避 + 现代库设计模板。

阅读时长 ~25 分钟 前置 P3.1 解析算法 主要来源 nodejs.org/api/packages · tshy/tsup/unbuild 文档
📝 章节状态

骨架版本。对照大纲 node/outline.md § P3.2。

01 速览

  1. Dual Package Hazard 是双系统兼容的最大风险——同一个库的两份独立实例破坏单例、instanceof、共享状态。
  2. 3 种规避策略：纯无状态库（最推荐）/ CJS 设为 ESM wrapper / 只发 ESM 让 CJS 用 import()。
  3. 现代库构建工具：tshy / tsup / unbuild —— 自动生成双格式 + types 的发布产物。

02 底层逻辑

Dual Package Hazard 是什么

正文待写。

当一个 package 同时提供 CJS 和 ESM 入口（通过 conditional exports），不同导入方式会加载不同文件——module loader 不知道两者是"同一个东西"，独立创建两份实例：

javascript 
// CJS 用户
const { Singleton } = require('my-lib');
Singleton.set('foo');

// ESM 用户（同一进程，同一 my-lib）
import { Singleton } from 'my-lib';
console.log(Singleton.get('foo'));  // ← undefined! 不同实例

// 更隐蔽的：instanceof 失败
const a = require('my-lib').create();
import { Klass } from 'my-lib';
console.log(a instanceof Klass);  // ← false! 不同 class

3 种规避策略

正文待写。

策略 1 · 纯无状态（推荐）

库只导出纯函数 / 常量 / 无状态 class——两份独立实例不冲突。这是最简单也最现代的做法，适合 utility 库（lodash、zod、date-fns 等）。

策略 2 · CJS 设为 ESM wrapper

CJS 入口文件实际只是一个 wrapper，运行时动态 require + re-export ESM 的内部实现。这样保证只有 ESM 一份实例。代价：CJS 入口变异步（要用 import()），少数同步库不能这样做。

策略 3 · 只发 ESM

直接放弃 CJS 用户—— Node 22+ 起 ESM 已经足够稳定，新库可以这样选。CJS 用户用 const x = await import('pkg') 也能用。代价：失去一些老项目的兼容性。

04 典型代码

json 
// tsup 配置：双格式 + types
// tsup.config.ts
{
  "entry": ["src/index.ts"],
  "format": ["cjs", "esm"],
  "dts": true,
  "clean": true,
  "sourcemap": true
}

// 配套 package.json
{
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  }
}

05 常见陷阱

单例 / 全局状态会被双格式破坏

避免在库里维护可变模块级状态——或用 globalThis 共享。

instanceof 跨格式不可靠

用 duck typing 或 Symbol 标记替代——不要在库 API 里要求 instanceof。

Top-Level Await 可能引入死锁

循环依赖 + TLA 时模块永远 evaluate 不完。详见 ECMA P7.4。
