Dual Package Hazard 与现代库设计

实战章。当一个库要同时支持 CJS 和 ESM 用户时会撞上"双实例"问题——require('pkg') 和 import 'pkg' 加载的是两个独立实例，单例失效、instanceof 失败、状态不一致。讲清楚怎么规避 + 现代库设计模板。

阅读时长 ~25 分钟 前置 P3.1 解析算法 主要来源 nodejs.org/api/packages · tshy/tsup/unbuild 文档

01 速览

  1. Dual Package Hazard 是双系统兼容的最大风险——同一个库的两份独立实例破坏单例、instanceof、共享状态。
  2. 3 种规避策略：纯无状态库（最推荐）/ CJS 设为 ESM wrapper / 只发 ESM 让 CJS 用 import()。
  3. 现代库构建工具：tshy / tsup / unbuild —— 自动生成双格式 + types 的发布产物。

02 底层逻辑

Dual Package Hazard 是什么

P3.1 讲到一个库通过 exports 字段的 conditional exports，可以同时提供 CJS 和 ESM 入口。这听起来很美——用户不管用 require 还是 import，都能用同一个包。但实际生产中这种"双格式"会带来一类微妙 bug，叫 Dual Package Hazard。

根本原因：当一个 package 同时提供 CJS 和 ESM 入口（通过 conditional exports），不同导入方式会加载不同文件——module loader（Node 内部）不知道这两个文件是"同一个东西"，独立创建两份实例。如果库内部有共享状态（单例、缓存、计数器等），两份实例的状态互不可见：

javascript 
// CJS 用户
const { Singleton } = require('my-lib');
Singleton.set('foo');

// ESM 用户（同一进程，同一 my-lib）
import { Singleton } from 'my-lib';
console.log(Singleton.get('foo'));  // ← undefined! 不同实例

// 更隐蔽的：instanceof 失败
const a = require('my-lib').create();
import { Klass } from 'my-lib';
console.log(a instanceof Klass);  // ← false! 不同 class

3 种规避策略

库作者面对 Dual Package Hazard 有三种处理路径——按"从最简单到最激进"顺序：

策略 1 · 纯无状态（最推荐）

库只导出纯函数 / 常量 / 无状态 class——两份独立实例完全不冲突，因为没有共享状态可以"不一致"。这是最简单也最现代的做法，适合大多数 utility 库（lodash / zod / date-fns / clsx 等）。这条策略不需要任何特殊配置——直接发双格式，靠"无状态设计"绕过 hazard。

实操原则：不在模块顶层维护可变状态。如果一定要维护状态（如全局缓存），用 globalThis.__myLibCache 这种显式跨实例共享，让两份独立实例都看到同一份。

策略 2 · CJS 设为 ESM wrapper

CJS 入口文件不是"另一份完整实现"——而是一个thin wrapper，运行时动态 import ESM 然后 re-export。本质上保证 ESM 是唯一真实源，CJS 只是包装。

cjs-wrapper 
// dist/index.cjs（CJS wrapper）
module.exports = (async () => {
  const esm = await import('./index.js');
  return { ...esm };
})();
// 用户必须 await：
// const lib = await require('my-lib');

代价：CJS 入口变异步（用户必须 await）。同步库（如 const lib = require('my-lib'); lib.foo()）不能用这条策略。但对纯异步 API 库（HTTP client / 数据库 driver）可行。

策略 3 · 只发 ESM（激进）

直接放弃 CJS 用户。Node 22+ 起 ESM 已经足够稳定 + Node 23 起 require ESM 实验支持，新库这条路越来越可行。代表项目：chalk 5（2022 起只发 ESM，引发 npm 生态争议但坚持）/ node-fetch 3。

代价：CJS 用户必须用动态 const lib = await import('my-lib')—— 不能直接 require。这对老项目转型阻力大，但对纯新项目几乎无感。

⚠️ "多发包名解决"是反模式

有人会想"那我发两个包：my-lib 和 my-lib-cjs"绕过 hazard。这是反模式——npm 生态会更乱，依赖树双倍。正确做法是上面 3 种之一，不要发多个包。

现代库设计构建工具

双格式发布需要工具链支持——单独写一份代码，编译出 CJS + ESM + .d.ts 三套产物。主流工具（2025 年现状）：

  • tshy（Isaac Schlueter，npm 作者）：TypeScript-native，自动生成双格式 + types。配置极简——只需 "main" + "exports" 几行。2025 推荐起手项目
  • tsup（egoist）：基于 esbuild，配置简单，bundler 风格（一个文件一个产物）。常用于现代库
  • unbuild（UnJS / Vite 生态）：多入口友好，rollup-based。Nuxt / Nitro 生态默认
  • pkgroll：极简包装 rollup，适合喜欢 rollup 配置的人

import.meta.resolve / require.resolve：路径查找 API

做插件 loader / 动态加载 / 路径解析时需要在运行时知道某个模块的实际文件路径。两个 API 各自工作：

  • CJS：require.resolve('foo') 返回完整文件路径（同步）
  • ESM：import.meta.resolve('foo') 返回 file:// URL（Node 20.6+ 稳定）

典型用例：插件系统加载用户指定的扩展（require.resolve(userPluginName)）/ monorepo 工具链定位包（如 nx / lerna）/ 自定义 loader（ts-node / vitest）。

Top-Level Await 限制：仅 ESM 支持

ES2022 引入 Top-Level Await（TLA）——可以在 ESM 模块顶层直接 await，不需要包在 async function 里。这让"模块加载时等异步操作完成"成为可能：

javascript 
// config.mjs - ESM 模块顶层 await
const config = await fetch('/api/config').then(r => r.json());
export default config;

// 其他模块 import 时，会等 config.mjs 加载完
import config from './config.mjs';
console.log(config);  // 已经是 resolved value
  • 仅 ESM 支持——CJS 不行（require 是同步，不能等 Promise）
  • 让 ESM 模块"等待"才完成 evaluation——所有 import 这模块的也会等
  • 常见用法：顶层 fetch config / 异步初始化数据库连接 / 加载 native dependency
  • 陷阱：循环依赖 + TLA 可能导致死锁—— 详见 ECMA P7.4 · Module Records + TLA

03 应用场景

场景 A · 写一个新的 npm 库（推荐起步）

2025 年新库推荐起步：(1) 用 tshy（最新 npm 作者出品，配置极简）；(2) 优先纯无状态设计（function / 常量 / 无状态 class）；(3) 同时发 CJS + ESM + .d.ts—— exports 字段配置完整。这套组合让用户无论用什么 import 方式都能用，没有 hazard。如果业务必然需要状态（如客户端实例），改用策略 2（CJS wrapper）或 3（只发 ESM）。

场景 B · 诊断 instanceof 失败 / 单例失效 bug

用户报"instanceof MyClass 在我项目里 false"或"singleton 状态丢了"——大概率是 dual package hazard。诊断步骤：(1) require.resolve('the-lib') 看 CJS 解析路径；(2) 临时加 console.log(import.meta.url) 看 ESM 解析路径；(3) 对比两条路径——如果指向不同文件，hazard 确认。修法：让用户全部用 ESM 或全部 CJS，或库换无状态设计。

场景 C · 老库迁现代发布格式

老 CJS-only 库要支持 ESM 用户：最小改动是 用 tsup 加双格式构建 + 加 exports 字段 + types。改动通常 30 行配置。如果库有状态（singleton / cache），同时检查是否需要走策略 2 / 3。2025 年用户期待是双格式—— 只发 CJS 会失去新项目用户。

场景 D · 评估"该不该用 Top-Level Await"

模块顶层异步加载（fetch config / DB 连接）时考虑 TLA。陷阱：(1) TLA 让模块 import 时所有 importer 都等—— 启动慢；(2) 循环依赖 + TLA 可能死锁；(3) CJS 用户不能 require 这种 ESM 模块。建议：TLA 适用 entry point（应用入口）或明确异步初始化的 service 模块，不要在共享 utility 里用。

// tsup 配置：双格式 + types
// tsup.config.ts
{
  "entry": ["src/index.ts"],
  "format": ["cjs", "esm"],
  "dts": true,
  "clean": true,
  "sourcemap": true
}

// 配套 package.json
{
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  }
}

05 常见陷阱

单例 / 全局状态会被双格式破坏

避免在库里维护可变模块级状态——或用 globalThis 共享。

instanceof 跨格式不可靠

用 duck typing 或 Symbol 标记替代——不要在库 API 里要求 instanceof。

Top-Level Await 可能引入死锁

循环依赖 + TLA 时模块永远 evaluate 不完。详见 ECMA P7.4。
