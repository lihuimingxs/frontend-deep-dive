Test Runner / Permission / SEA / async_hooks

Node 22+ 引入的"内置工具"——回应 Bun / Deno 的工具一体化趋势。一章覆盖：内置 node:test runner、Permission Model（仿 Deno）、Single Executable Application (SEA)、async_hooks / AsyncLocalStorage（OpenTelemetry 基础）、Watch mode、Inspector Protocol。

阅读时长 ~30 分钟 前置 Node 22 LTS 基础 主要来源 nodejs.org/api/test · permissions · async_hooks

01 速览

  1. Node 22+ 加大量内置工具——是对 Bun / Deno 工具一体化趋势的回应。
  2. node:test 已经够用——简单 / 中型项目可替代 vitest / Jest，启动快、零依赖。
  3. AsyncLocalStorage 是 OpenTelemetry 的 V8 端基础——跨 async boundary 传递 context。

02 底层逻辑

node:test（Node 18+ 实验，20 稳定）

node:test 是 Node 内置的测试 runner，2022 年 Node 18 以实验形式引入，2023 年 Node 20 标记稳定。它不是要"取代" Jest / Vitest，而是给 CLI 工具、库、小服务一个零依赖的起点——以前一个 50 行的 utility 库要装 Jest + 配置 babel + 写 jest.config.js 才能跑测试，现在 node --test 就够。这是 Node 22+ 时代"内置工具一体化"的标志性产物。

核心 API 和主流框架基本一致：test / describe / before / after / beforeEach / afterEach，但导入路径是 node:test。断言用 node:assert/strict 而不是 expect——这一点就让大量 Jest 测试无法直接复用。运行模式是每个测试文件起一个独立子进程，文件之间天然并行隔离；文件内的 test() 默认串行，加 { concurrency: true } 则并行。

输出格式是 TAP（Test Anything Protocol）——一个 1990 年代 Perl 社区流出来的纯文本协议，CI 工具普遍支持。--test-reporter=spec 切到人类友好的 spec 格式，--test-reporter=tap / junit / dot 各有用途。Node 20.11+ 还内置了 --experimental-test-coverage——基于 V8 的 coverage（参见 V8 P5.2 · 诊断工具），不需要 c8 / nyc。

Mock API 是个亮点：t.mock.fn() / t.mock.method() / t.mock.timers.enable()，自动在测试结束时 restore。和 Jest 的全局 jest.fn() 不同，node:test 的 mock 挂在 TestContext 上——更显式、更不容易污染。

不能用的场景：DOM 测试（要 jsdom + 配套），snapshot 测试（实验，UX 一般），watch mode UI（Vitest 的 dashboard / coverage UI 是杀手锏）。前端项目仍然 Vitest 的天下；Node 后端 / lib / CLI 是 node:test 的主战场。

node:assert

node:assert 是 Node 最古老的内置模块之一（v0.1 就有），但十几年来一直被 Jest 的 expect 边缘化。node:test 兴起后它重新成为主角，因为 node:test 默认建议用 assert 而不是 expect——避免引入 chai / expect 等额外依赖。

关键的 API 分两套：legacy（assert.equal，==）和 strict（assert.strictEqual，===）。生产代码必须用 strict——legacy 在 {} == "" 这种边界返回 true，遗留几代的坑。推荐写法：import assert from 'node:assert/strict'，这样所有 API 自动走 strict 路径，连 assert.equal 都等价于 strictEqual。

常用 API：strictEqual（值相等）、deepStrictEqual（结构深相等，对齐 ECMA SameValue 算法）、throws（同步抛错）、rejects（async 拒绝）、match（正则）、partialDeepStrictEqual（Node 22+，子集匹配）。和 Jest 比少了 toBeTruthy / toContain / toMatchObject 等糖衣，但够用——node:assert 的设计哲学就是"够用且零依赖"。

失败时的诊断信息也升级过：deep 比较失败会输出彩色 diff（类似 Jest），不是早年那种"undefined !== {a:1}"的尴尬。错误对象是 AssertionError，带 actual / expected / operator 字段，CI 工具能精确定位。

Permission Model（Node 20+ 实验，22+ 稳定化）

Permission Model 是 Node 对 Deno 安全模型的回应。Deno 2018 年立项时把"默认无权限、需要显式 allow"作为核心卖点，戳中了 Node 生态最大的痛——一个 npm install 拉进来的 transitive dep 可以读 SSH key、发请求到任意域名、起子进程。Ryan Dahl 创立 Deno 就是因为 Node "不能不让脚本访问任意系统资源"是他公开承认的设计错误之一。

Node 20 引入 --experimental-permission，22 LTS 改名为 --permission 并标记稳定化。基本用法：node --permission --allow-fs-read=./data --allow-net=api.example.com server.js。颗粒度覆盖 fs（read / write 分开）、net（按域名 / IP / 端口）、child_process、worker、addons。没显式 allow 的 API 调用会抛 ERR_ACCESS_DENIED——和 Deno 的 PermissionDenied 异曲同工。

但 Node 的实现比 Deno 起步晚——Deno 是从 V8 isolate 的 host 层就嵌入了权限检查；Node 是在 JS 标准库各 API 入口加 if 判断。结果是原生模块（C++ addon）和 native binding 不在 Permission Model 管控范围——你可以禁掉 fs，但 require('better-sqlite3') 的 C++ 代码照样能 open() 文件。这是 Node 生态历史包袱的体现。

核心使用场景是受信代码 + 第三方代码混跑：评测系统（学生提交的代码）、Serverless 平台（用户函数）、CI 沙箱（开源 PR 跑测试）。比起容器（Docker / firecracker）启动快、内存开销小，但不能完全替代——见后面"常见陷阱"。日常 CLI / 业务后端反而不需要开，开了反而限制工程能力。

Single Executable Application (SEA)

SEA 把 Node 二进制 + 用户脚本打包成单文件可执行程序——用户拿到一个 my-cli 就能直接跑，不需要装 Node、不需要 npm install。这是对 Bun 的 bun build --compile 和 Deno 的 deno compile 的回应。在 Bun 出现前，Node 社区靠 pkg / nexe 这些第三方工具凑活，体验一直不好。

底层机制：把 JS 代码序列化成一个 blob，用 postject 工具注入到 Node 二进制的特定 section（macOS Mach-O / Linux ELF / Windows PE 都有专门约定）。运行时 Node 启动时检测到 SEA blob 就直接加载执行，跳过 require resolution。Node 20+ 还支持嵌入 V8 startup snapshot，让冷启动时间从几百 ms 压到 50 ms 内（参见 V8 P5.1 · 嵌入 V8）。

当前限制：SEA 目前只支持单一入口的 CommonJS 文件（Node 22 才开始支持 ESM 入口和 require 多文件，且仍标记实验）；产物体积大（包含整个 Node 二进制，~80MB）；不能跨平台编译——Linux 上只能产出 Linux 二进制。Bun 的 --compile 这块明显更成熟：交叉编译、产物 ~50MB、ESM 默认。

典型场景是分发独立 CLI 工具：内部团队做 deploy / migrate / lint 工具，给运维同事用——不需要他们先装 nvm。但如果用户本来就有 Node 环境，npx my-tool 比下载 100MB 二进制划算。这是 SEA 的核心定位：runtime-less 分发的最后一公里。

async_hooks + AsyncLocalStorage

async_hooks 是 Node 8+ 引入的异步资源追踪机制：每个异步操作（Promise / setTimeout / IO）创建时都有一个 asyncId，async_hooks 让你在它的 init / before / after / destroy 四个生命周期点注入回调。这是 Node 对"跨 async boundary 传递上下文"这个核心难题的内核级答案——其他语言用线程本地存储（TLS）解决，Node 因为单线程 + 异步只能从 Promise 链路里追上下文。

AsyncLocalStorage（ALS）是 async_hooks 之上的高层封装，Node 12+ 稳定。语义上等价于"async 版的 ThreadLocal"：als.run(value, fn) 之后，fn 内部所有同步 + 异步代码（无论嵌套多深、跨多少 await / setTimeout / 回调）都能通过 als.getStore() 拿到 value。Node 内部用 AsyncContextFrame 在每个异步 hop 时复制 context 引用——开销仅一个指针。

这是整个 OpenTelemetry / 分布式追踪 / APM 生态的 V8 端基础。HTTP 请求一进来 middleware 就 als.run({ traceId, userId, requestId }, ...)，后面的 logger / DB query / 下游 HTTP 调用不需要把 traceId 一路手动传——所有需要它的代码就 als.getStore()。Datadog / New Relic / Sentry 的 Node SDK 都是这么实现的。从 React 视角看，这相当于 React Context 的 async 版——同样是"避免 prop drilling"，但传递的是请求级运行时状态。

性能成本：早期 async_hooks（Node 12 之前）开销显著——每个 Promise 都触发 4 个 hook 回调，热路径上能让 RPS 降 20%。Node 16+ 之后大幅优化，特别是 Node 20 引入的 AsyncContextFrame（不再依赖 PromiseHook，直接走 V8 的 ContinuationPreservedEmbedderData），ALS 单独使用的 overhead 降到 1-3%。但 启用 async_hooks 全套监听（init/before/after/destroy）的开销仍然不容忽视，5-10% 量级——生产只在需要诊断时打开。

Inspector Protocol

Inspector Protocol 是 V8 提供的统一调试 / 性能分析协议——本质上 Chrome DevTools Protocol（CDP）的 V8 子集。Node 6+ 弃用了老的 V8 Debugger Protocol（v8-debug），转向和 Chrome 共享 CDP，意义在于同一套协议能调 Chrome 也能调 Node，工具生态不需要分裂。详细实现见 V8 P5.2 · 诊断与沙箱，本节聚焦 Node 侧用法。

启动方式：node --inspect app.js（不阻塞启动）或 node --inspect-brk app.js（在第一行断住，等调试器接入）。Node 在默认 127.0.0.1:9229 起一个 WebSocket server，DevTools / VSCode / WebStorm 通过 CDP 接入——发送 Debugger.setBreakpoint、Profiler.start、HeapProfiler.takeHeapSnapshot 等 JSON 命令。

实战价值远超"打断点":CPU profile（--cpu-prof 一行启动，跑完产出 .cpuprofile 文件，DevTools 直接打开看火焰图）、heap snapshot（--heap-prof 或运行时 v8.writeHeapSnapshot()，定位内存泄漏）、heap timeline（连续抓多个 snapshot 看 retained 增长）。生产排查 OOM 的标准流程就是 node --heap-prof app.js 跑一段时间，再用 DevTools 分析。

安全提醒：--inspect=0.0.0.0 暴露到公网等于给攻击者远程任意代码执行——Inspector Protocol 没有 auth。生产开调试只在 localhost 上开，再用 SSH 端口转发到本地。这是 Node 上经常翻车的小坑。

Watch mode（Node 18+）

Watch mode 用一个 flag 替代了 nodemon 这个用了十多年的 dev 工具。node --watch app.js 启动后会监视文件变更，改到就自动重启进程。--watch-path=./src 指定 watch 路径，--watch --test 还能让 node:test 在文件变化时自动跑测试。

底层基于 fs.watch（P4.3 · fs / Buffer / process 提到过），不同平台借助 inotify / FSEvents / ReadDirectoryChangesW。和 nodemon 比，Watch mode 不依赖 chokidar，启动更快、依赖更少；但缺少 nodemon 的 ignore 规则、debounce 时间、--exec 灵活配置——简单 dev server / lib 开发够用，复杂前端 dev server（Next.js / Remix）仍然要用框架自带的 hot reload，因为 Watch mode 只能"重启"，不能模块级 HMR。

Bun 的 bun --watch 和 Deno 的 deno run --watch 都做了同样的事——这是 runtime 级标准化的体现：基础工具不再是社区包，而是 runtime 内置。Node 跟进得不算晚。

03 应用场景

场景 A · 小项目用 node:test 替代 vitest / Jest

一个 50 行的 utility 库、一个内部 CLI 工具、一个胶水脚本——这些场景以前为了写测试要先 npm install jest @types/jest ts-jest，再写 jest.config.js + tsconfig.test.json。Node 22+ 直接 node --test --experimental-strip-types test/*.test.ts 就能跑 TS 测试。零依赖、启动 < 100ms、CI 配置一行 node --test --test-reporter=spec。但前端项目（React / Vue 组件测试）还是 Vitest——它的 jsdom 集成、UI dashboard、HMR-style watch 模式短期内 node:test 追不上。判断标准：有没有 DOM？有就 Vitest，没有就 node:test。

场景 B · 用 AsyncLocalStorage 实现请求级 trace ID

HTTP server 入口的 middleware 中 als.run({ traceId: req.headers['x-trace-id'] ?? crypto.randomUUID() }, () => next())。后续 logger（pino / winston）的 hook 函数里 const { traceId } = als.getStore() 自动 attach 到日志；下游 HTTP 客户端的拦截器也从 store 取 traceId 写入 x-trace-id header 透传给微服务；DB 慢查询日志带上 traceId 直接关联到具体请求。这是每个生产 Node 后端都该有的能力。OpenTelemetry 的 Node SDK 已经把这套自动化——但理解底层 ALS 的人才知道为什么 instrumentation 偶尔会"丢 context"（嵌套 worker / 自定义 thenable / native callback 都是常见 corner case）。

场景 C · CLI 工具用 SEA 打包分发

团队内部做了个 deploy CLI，依赖 30 个 npm 包。给运维 / 非前端同事用时——每次说"先装 nvm，再 nvm install 20，再 npm install -g @yourcorp/deploy-cli"——成本太高，更别说生产环境的离线服务器。SEA 把这一切打包成一个 deploy-cli 二进制，scp 上去就能跑。但实战中 Bun 的 bun build --compile 体验明显更好——产物小、支持 ESM、可交叉编译，Node SEA 还在追赶。真要分发独立 CLI，目前优先选 Bun 编译；只有强依赖 Node 特定 API（特定版本的 quirks、native module 兼容）时才用 SEA。

场景 D · 评测系统 / 沙箱用 Permission Model

在线编程评测系统跑学生提交的 JS 代码——传统方案是 Docker 容器 + seccomp，启动 200-500ms、内存基线 ~30MB。改用 node --permission --allow-fs-read=/sandbox/in --allow-fs-write=/sandbox/out user.js 跑，启动 < 50ms、内存基线 ~12MB——批量并发评测时省的资源很可观。但不能完全替代容器：native addon、V8 自身漏洞（CVE）、prototype pollution 引发的逻辑越权都不在 Permission Model 管控内。生产敏感场景的标准做法是双层沙箱——容器外层做强隔离，容器内 Node 开 Permission Model 做内层防护。

04 典型代码

javascript 
// node:test 简单用法
import { test, describe } from 'node:test';
import assert from 'node:assert/strict';

describe('add', () => {
  test('positive', () => {
    assert.equal(add(1, 2), 3);
  });
  test('negative', () => {
    assert.equal(add(-1, -2), -3);
  });
});

// AsyncLocalStorage 实现请求级 context
import { AsyncLocalStorage } from 'node:async_hooks';
const als = new AsyncLocalStorage();

http.createServer((req, res) => {
  const traceId = req.headers['x-trace-id'] ?? crypto.randomUUID();
  als.run({ traceId }, () => handleRequest(req, res));
}).listen(3000);

// 任何 async 深度都能拿到
function deepAsync() {
  const { traceId } = als.getStore();
  console.log({ traceId });
}

05 常见陷阱

node:test 不能直接复用 vitest / Jest 测试

API 不兼容；迁移需要改写测试代码。

async_hooks 默认开销不小

生产 hot path 启用 hooks 影响性能 5-10%；仅在需要 trace 时启用。

Permission Model 不能完全替代进程沙箱

仅 Node 自身 API 受控；native module / V8 自身漏洞仍可能逃逸。生产敏感场景配进程沙箱（容器）做双层。
