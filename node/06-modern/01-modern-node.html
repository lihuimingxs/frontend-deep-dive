Test Runner / Permission / SEA / async_hooks

Node 22+ 引入的"内置工具"——回应 Bun / Deno 的工具一体化趋势。一章覆盖：内置 node:test runner、Permission Model（仿 Deno）、Single Executable Application (SEA)、async_hooks / AsyncLocalStorage（OpenTelemetry 基础）、Watch mode、Inspector Protocol。

阅读时长 ~30 分钟 前置 Node 22 LTS 基础 主要来源 nodejs.org/api/test · permissions · async_hooks
📝 章节状态

骨架版本。对照大纲 node/outline.md § P6.1。

01 速览

  1. Node 22+ 加大量内置工具——是对 Bun / Deno 工具一体化趋势的回应。
  2. node:test 已经够用——简单 / 中型项目可替代 vitest / Jest，启动快、零依赖。
  3. AsyncLocalStorage 是 OpenTelemetry 的 V8 端基础——跨 async boundary 传递 context。

正文待写。

async_hooks + AsyncLocalStorage

正文待写。

  • 追踪 async 资源的生命周期（init / before / after / destroy）

04 典型代码

javascript 
// node:test 简单用法
import { test, describe } from 'node:test';
import assert from 'node:assert/strict';

describe('add', () => {
  test('positive', () => {
    assert.equal(add(1, 2), 3);
  });
  test('negative', () => {
    assert.equal(add(-1, -2), -3);
  });
});

// AsyncLocalStorage 实现请求级 context
import { AsyncLocalStorage } from 'node:async_hooks';
const als = new AsyncLocalStorage();

http.createServer((req, res) => {
  const traceId = req.headers['x-trace-id'] ?? crypto.randomUUID();
  als.run({ traceId }, () => handleRequest(req, res));
}).listen(3000);

// 任何 async 深度都能拿到
function deepAsync() {
  const { traceId } = als.getStore();
  console.log({ traceId });
}

05 常见陷阱

node:test 不能直接复用 vitest / Jest 测试

API 不兼容；迁移需要改写测试代码。

async_hooks 默认开销不小

生产 hot path 启用 hooks 影响性能 5-10%；仅在需要 trace 时启用。

Permission Model 不能完全替代进程沙箱

仅 Node 自身 API 受控；native module / V8 自身漏洞仍可能逃逸。生产敏感场景配进程沙箱（容器）做双层。
