HTTP / undici / fetch

Node 服务端的核心场景——HTTP server / client。讲清传统 http 模块（基于 Streams）、新一代 undici（Node 团队自己写的高性能 client，用作内置 fetch 后端）、HTTP/2 支持、TLS。读完能选对 client，知道为什么 Node 18+ 的 fetch 是 undici。

阅读时长 ~30 分钟 前置 P4.1 Streams 主要来源 nodejs.org/api/http · undici 文档

01 速览

  1. Node http server 整个 API 是 Streams 的应用——req 是 Readable、res 是 Writable。
  2. undici 是 Node 18+ 内置 fetch 的后端——比旧 http.request 快 2-3×。
  3. HTTP/2 是单独的 http2 模块——不是 http 模块的扩展，使用方式略有不同。

02 底层逻辑

http 模块（2009-）：Streams 抽象的最佳应用

http 模块是 Node 最老的内置模块之一——和 Node 同年（2009）发布。设计哲学完全建立在 Streams 之上：

  • createServer((req, res) => ...)：每个 HTTP 请求来时，回调拿到 req（IncomingMessage = Readable，请求 body 流）+ res（ServerResponse = Writable，响应 body 流）
  • 整个 API 是 Streams 应用—— 读 request body 用 stream API（async iterator / pipeline）；写 response 用 res.write() + res.end() 或 readable.pipe(res)
  • Keep-Alive 自动管理：HTTP/1.1 默认 keep-alive，连接复用降低握手开销
  • HTTP/1.1 only（HTTP/2 在独立的 http2 模块）

这套"HTTP = Streams"的抽象让大请求体 / 大响应体不爆内存—— 用 pipeline 把 request body 直接 pipe 到 fs / S3 / 下游服务，全程流式无需缓冲。

https 模块：TLS 终止

https 是 http 的 TLS 加密版。API 几乎一样（除了多个 key / cert / ca 等 TLS 配置）：

  • 同 http 但带 TLS 终止—— Node 端处理证书 + 加密 / 解密
  • 证书管理：key（私钥）+ cert（公钥证书）+ 可选 ca（中间证书链）
  • ALPN 协议协商：TLS 握手时客户端告知支持哪些协议（http/1.1, h2），server 选择。这是 HTTP/2 升级的标准方式

实践注意：大多数生产环境不让 Node 直接处理 TLS—— 前面放 nginx / Caddy / Cloudflare 做 TLS 终止 + 反向代理到内部 Node（HTTP/1.1）。这样 Node 不需要管证书续签 / TLS 性能 / OCSP 等复杂事。例外：内部 service 间通信（mTLS）/ 不能放反向代理的边缘场景才直接用 https。

http2 模块（Node 8.4+）：与 http 不兼容

HTTP/2 是 2015 标准，主要改进是流多路复用（一个 TCP 连接上跑多个并发 request）+ 头压缩（HPACK）+ 服务端推送（Server Push，已弃用）。Node 8.4 起加了 node:http2 模块。

关键事实：http2 API 与 http 不兼容—— 新概念 Http2Session / Http2Stream / 不同的 push API。Express / Koa 等框架不直接支持 http2 Server，要换专门的 framework（如 Fastify 早期支持）。

2025 实践：http2 模块直接用得不多——大多数应用让前面的反向代理处理 HTTP/2，Node 内部仍用 HTTP/1.1。直接用 http2 仅在内部 microservice 间通信（gRPC 走 HTTP/2）/ 极致性能场景使用。

undici（Node 18+ 内置）：高性能 HTTP/1.1 client

undici 是 Node 团队（Matteo Collina 主导）2018+ 写的下一代 HTTP/1.1 client—— 设计目标"比旧 http.request 快几倍"。结果：吞吐量 2-3× 旧 client，特别是高并发 + Keep-Alive 场景。

undici 关键优化点：

  • 连接池 + Keep-Alive 自动管理——内置 Pool 类，配置最大连接数 / pipelining 数
  • Pipelining 支持——HTTP/1.1 同一连接上不等响应连续发请求（Apache / Nginx 都支持但 Node 旧 client 不用）
  • 不分配多余 Buffer—— 对 Node 内部 zero-copy 优化
  • 单一接口—— request() / fetch() / stream() / upgrade() 同源

Node 18+ 内置 undici 作为 fetch 的实现后端。也作为独立 npm 包 undici 发布——旧 Node 项目能直接装来用。

fetch 内置（Node 18+）：与浏览器同名同义

Node 18（2022-04）起内置 fetch—— 是 undici 的"友好门面"，API 完全照搬浏览器 fetch 标准。这意味着同一段 fetch 代码能跑 Node / Deno / Bun / 浏览器 / Edge Runtime。

  • API 与浏览器一致：fetch(url, options) + Response + Headers + FormData + AbortSignal
  • 实现基于 undici—— 性能与直接用 undici 相当
  • 不再需要 node-fetch / cross-fetch / isomorphic-fetch 等 polyfill
  • Node 18-20 标"实验性"，21+ 稳定。生产应用 Node 22+ 用 fetch 没风险

实践推荐：新代码用 fetch，跨 runtime 兼容。需要极致性能 / 高级配置（pipelining / 连接池调优）才直接用 undici 底层 API。

TLS / SSL：底层抽象

tls 模块是 Node 与 OpenSSL 的桥梁——https / http2 内部都用它。直接用 tls 模块的场景较少：

  • 实现非 HTTP 的 TLS 协议（如 自定义协议 over TLS）
  • 调试 TLS 握手细节
  • 实现 mTLS（双向认证）

常见生产配置：HTTP/2 + TLS 1.3 + ALPN + OCSP stapling。但实际上大多数 Node 应用不需要管这些—— 让 nginx / Cloudflare 处理。Node 自己的 tls.DEFAULT_MIN_VERSION 在 17+ 已经是 TLS 1.2，正确的安全默认。

03 应用场景

场景 A · 旧代码 http.request 迁 undici / fetch

老 Node 服务很多用 http.request + callback 风格调外部 API。迁现代 fetch 收益：(1) 性能 2-3×（连接池 + pipelining）；(2) API 更简洁（async/await 替代 callback）；(3) 跨 runtime 兼容（同一段代码能跑 Edge / Deno）。改动点：把 http.request(options, cb) 换成 await fetch(url, options)—— options 字段名稍有不同（method 同 / headers 同 / body 名字变 body）。

场景 B · HTTP/2 server 部署判据

大多数情况让反向代理处理 HTTP/2—— nginx / Caddy / Cloudflare 在前端做 HTTP/2 终止 + TLS，反向代理到内部 Node 用 HTTP/1.1。这样 Node 不需要管证书 / HTTP/2 流多路复用复杂性，仍能享受客户端 HTTP/2 的体验改善。例外：内部 microservice 间用 gRPC（必须 HTTP/2）/ 边缘部署没有反向代理 / 极致性能场景测过 HTTP/2 真有改善——这些情况才直接用 Node http2 模块。

场景 C · 大文件下载用 stream 不爆内存

下载 1GB 文件——不能 const buf = await response.arrayBuffer()（爆内存），要用 stream：

javascript 
import { pipeline } from 'node:stream/promises';
import { Readable } from 'node:stream';
import { createWriteStream } from 'node:fs';

const res = await fetch(url);
// fetch 返回 web stream → 转 Node stream → pipe 到文件
await pipeline(
  Readable.fromWeb(res.body),
  createWriteStream('big-file.bin')
);

内存稳定 16KB 左右，不管文件多大。

场景 D · undici 高级用法（连接池调优）

高并发场景（如大量调外部 API 的中间层服务），直接用 undici 的 Pool 配置连接池上限：

javascript 
import { Pool } from 'undici';
const pool = new Pool('https://api.example.com', {
  connections: 100,        // 最多 100 个并发连接
  pipelining: 10,          // 每连接最多 10 个请求 pipelining
});
const { body } = await pool.request({ path: '/users/1' });

比 fetch 默认配置吞吐高几倍——但要管连接池生命周期。

04 典型代码

javascript 
// 传统 http.createServer + Streams 风格
import { createServer } from 'node:http';
import { pipeline } from 'node:stream/promises';

const server = createServer(async (req, res) => {
  if (req.method === 'POST' && req.url === '/upload') {
    // req 是 Readable，写到磁盘
    await pipeline(req, createWriteStream('upload.bin'));
    res.statusCode = 200;
    res.end('OK');
  }
});
server.listen(3000);

// 现代客户端用 fetch（基于 undici）
const res = await fetch('https://api.example.com/data', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ x: 1 }),
});
const data = await res.json();

05 常见陷阱

fetch 在 Node 仍是"实验性"标记

Node 21+ 起 stable，但 21 之前 console 会有 warning。

http2 API 与 http 不互通

不能直接换；通常用反向代理（nginx）做 HTTP/2 终止给 Node 看 HTTP/1.1。

Keep-Alive 默认行为陷阱

http.Agent 默认 Keep-Alive 关闭（Node 19+ 改为开启）—— 老代码可能因这变更性能曲线突变。
