Streams 全谱：4 类 + backpressure

Streams 是 Node 数据处理的统一抽象——http req/res 是 Stream、fs.createReadStream 是 Stream、Worker 通信也是 Stream。理解 Streams 是理解 Node 的关键。讲 4 类 stream + flowing vs paused 模式 + highWaterMark + backpressure + pipeline + async iterator + WHATWG Streams 跨标准融合。

阅读时长 ~35 分钟 前置 P2 完整 主要来源 nodejs.org/api/stream

01 速览

  1. Streams 是 Node 数据处理的统一抽象——http req/res / fs / Worker 通信都是 Stream。
  2. Backpressure 让 Node 不爆内存—— write() 返回 false → 暂停 source → 'drain' 恢复。
  3. 现代写法用 pipeline + async iterator——pipe 已不推荐（错误传播 / 资源释放有坑）。

02 底层逻辑

4 类 stream：Node 数据处理的统一抽象

Node 把所有数据流都抽象成 4 类 stream，让上层代码用同一套 API 处理文件、网络、进程通信、加密等。这种统一是 Node 早期最聪明的设计——让用户不需要为每种数据源学不同 API：

  • Readable（数据源）：发出数据的 stream。例：fs.createReadStream（文件读）/ HTTP 请求的 req（请求 body）/ process.stdin（标准输入）/ net.Socket（TCP 收数据时）
  • Writable（数据汇）：接收数据的 stream。例：fs.createWriteStream（文件写）/ HTTP 响应的 res（响应 body）/ process.stdout（标准输出）
  • Duplex（双向）：既是 Readable 又是 Writable。例：net.Socket（TCP socket，能读能写）/ Web Worker 通信
  • Transform（变换器，Duplex 的特化）：写入一个数据，处理后从读端出来。例：zlib.createGzip()（写明文出 gzip）/ crypto.createCipher()（写明文出密文）

这套抽象让"读文件 → gzip 压缩 → 写网络" 这种链式处理可以用纯 stream pipeline 表达 —— 中间不需要把整个文件加载到内存。

两种模式：flowing vs paused（最难懂的部分）

Readable stream 有两种消费模式，新手最常踩坑的就是混用两者：

  • Flowing（push）模式：数据自动流向消费者。监听 'data' 事件触发 → 默认进入 flowing。如果监听器处理速度跟不上，数据会被丢失（事件不缓存）
  • Paused（pull）模式：消费者主动调 .read() 拉数据。stream 内部缓存数据，不丢但需要主动循环

模式切换：监听 'data' / .pipe() / .resume() → flowing；.pause() → paused。新代码不需要知道这个区分—— 用 async iterator（for await）或 pipeline 自动管理。但读老代码会遇到 'data' 事件 + .pause() 这套手动管理。

highWaterMark：内部 buffer 阈值

每个 stream 有内部 buffer。highWaterMark 是这个 buffer 的满载阈值—— 决定何时触发 backpressure：

  • 默认值：字节流 16 KB（16384 字节）；object mode 16 个对象
  • Duplex / Transform 有独立的 readableHighWaterMark / writableHighWaterMark（Node 8+）—— 读端和写端可以不同阈值
  • 调大（如 64KB / 1MB）：减少 read 次数 → 吞吐增加，但内存占用也增加
  • 调小：减少内存占用，但增加 syscall 次数

实践：默认 16KB 几乎适合所有场景。除非你做特别的吞吐 / 内存优化，不要改这个值——改大可能让 backpressure 反应慢，改小可能让 EventLoop 太忙处理 read。

Backpressure：让 Node 不爆内存的核心机制

Backpressure 是 Streams 最重要的设计——让 Node 处理 10GB 文件时内存只占 16KB，而不是全加载。

机制：writable.write(chunk) 返回 boolean——

  • true：buffer 还没满，可以继续 write
  • false：buffer 已超过 highWaterMark，消费者跟不上—— 必须暂停 source 等 'drain' 事件触发后再继续

这就是为什么正确处理 backpressure 的代码看起来很罗嗦——必须监听返回值 + 'drain' 事件。修法：用 pipeline 或 async iterator，它们自动处理 backpressure，不需要手动管理。

pipeline vs .pipe()：现代 vs 传统

连接多个 stream 有两种 API：

  • .pipe()（2009 起）：传统写法。readable.pipe(writable) 把数据自动从 source 流到 destination。致命缺陷：错误传播不全—— 一个 stream 出错时，链上其他 stream 不会被自动清理（资源泄漏）
  • stream.pipeline()（Node 10+，现代推荐）：pipeline(readable, transform, writable, callback)。自动处理错误传播 + 资源释放—— 任何一环出错，所有 stream 都被清理
  • stream/promises.pipeline()（Promise 版本）：await pipeline(...)，最现代写法

实践规则：新代码永远用 pipeline，不用 .pipe()。老代码逐步迁移。"不就是一个数据流向"看似差不多，但生产环境 .pipe() 的资源泄漏 bug 会让你某天突然 OOM。

Async iterator：现代消费方式

Node 10+ 起，所有 Readable stream 都实现了 Symbol.asyncIterator—— 可以用 for await 消费：

javascript 
async function processLogs(filePath) {
  const stream = createReadStream(filePath, 'utf-8');
  for await (const chunk of stream) {
    // 每个 chunk 自动处理 backpressure
    await sendToService(chunk);
  }
}

这种写法比 'data' 事件 + 手动 backpressure 简洁得多—— async iterator 内部处理了 pause / resume。错误处理：直接 try/catch 包住即可。对新写的 stream 消费代码，async iterator 是默认选择。

WHATWG Streams（stream/web）：与 Web 标准融合

Node 16+ 起加了一套独立的 web 标准 stream API——ReadableStream / WritableStream / TransformStream，住在 node:stream/web 模块。这套是 WHATWG（Web Hypertext Application Technology Working Group）规范，浏览器与 Node 同名同义。

关键事实：这是新一套 API，不是包装传统 Node Streams—— 两者不直接兼容。fetch() 返回的 response.body 是 web stream，不是 Node Readable。互转：Readable.fromWeb(webStream) / Readable.toWeb(nodeStream)。

使用建议：新代码优先 web streams—— 跨 Node / 浏览器 / Edge / Deno / Bun 兼容好，是未来标准。但 Node 生态大多数库（fs / http / zlib）仍是传统 Node Streams—— 短期内两套并存。

03 应用场景

场景 A · 大文件处理用 pipeline 不爆内存

10GB 文件 + gzip 压缩 + 写到目标：用 pipeline 链式处理，内存稳定 16KB。关键点：(1) 用 pipeline 而非 .pipe()；(2) 用 stream/promises 版本配 await；(3) try/catch 处理任意一环错误。和传统"readFile + 处理 + writeFile"对比——后者会爆 10GB 内存。

场景 B · backpressure 调试

生产服务突然 OOM——可能是source 太快、destination 太慢、没正确处理 backpressure。诊断方法：(1) 看 writable.writableLength 是否长期接近 / 超 highWaterMark；(2) 监听 'drain' 事件触发频率——从不触发 = source 太慢；超频触发 = highWaterMark 太小。修法：换 pipeline 写法（自动管理）；或显式监听返回值 + 'drain'。

场景 C · web 与 node streams 互转

fetch() 给 web stream，但你想传给 Node 的 fs / zlib（传统 Node Streams）—— 用 Readable.fromWeb(response.body) 转换。反向：把 Node stream 给 fetch 上传 body，用 Readable.toWeb(nodeStream)。Node 17+ 起稳定。

场景 D · 写自定义 Transform stream

自定义数据变换处理（如 CSV 解析、protobuf 解码、自定义压缩）：继承 Transform 类实现 _transform(chunk, encoding, callback)。关键点：(1) 处理完一个 chunk 调 callback(err, output)；(2) 不要在 _transform 里做 CPU 密集计算（会阻塞）；(3) 用 this.push(data) 输出多份数据。

04 典型代码

javascript 
import { pipeline } from 'node:stream/promises';
import { createReadStream, createWriteStream } from 'node:fs';
import { createGzip } from 'node:zlib';

// 现代写法：pipeline + Promise
await pipeline(
  createReadStream('input.txt'),
  createGzip(),
  createWriteStream('input.txt.gz')
);

// 用 async iterator 消费
async function processLargeFile(path) {
  const readable = createReadStream(path);
  for await (const chunk of readable) {
    // 每个 chunk 自动 backpressure
    await processChunk(chunk);
  }
}

05 常见陷阱

不要混用 .pipe() 和 await pipeline()

同一 stream 链选一种风格，避免错误传播混乱。

监听 'data' 后无法回 paused 模式

除非显式 .pause()。新代码用 async iterator 避免这个状态机问题。

node streams 与 web streams 不直接兼容

fetch 给的是 web ReadableStream，要用 Readable.fromWeb() 转成 node stream。
