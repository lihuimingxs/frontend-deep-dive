Streams 全谱：4 类 + backpressure

Streams 是 Node 数据处理的统一抽象——http req/res 是 Stream、fs.createReadStream 是 Stream、Worker 通信也是 Stream。理解 Streams 是理解 Node 的关键。讲 4 类 stream + flowing vs paused 模式 + highWaterMark + backpressure + pipeline + async iterator + WHATWG Streams 跨标准融合。

阅读时长 ~35 分钟 前置 P2 完整 主要来源 nodejs.org/api/stream
📝 章节状态

骨架版本。对照大纲 node/outline.md § P4.1。

01 速览

  1. Streams 是 Node 数据处理的统一抽象——http req/res / fs / Worker 通信都是 Stream。
  2. Backpressure 让 Node 不爆内存—— write() 返回 false → 暂停 source → 'drain' 恢复。
  3. 现代写法用 pipeline + async iterator——pipe 已不推荐（错误传播 / 资源释放有坑）。

02 底层逻辑

4 类 stream

正文待写。

  • Readable：数据源（fs.createReadStream / http req / process.stdin）
  • Writable：数据汇（fs.createWriteStream / http res / process.stdout）
  • Duplex：双向，如 socket（net.Socket）
  • Transform：过滤器（zlib.createGzip / crypto.createCipher）

两种模式：flowing vs paused

正文待写。

  • Flowing (push)：数据自动流，监听 'data' event 处理；数据来不及处理会丢失
  • Paused (pull)：调用 .read() 主动拉数据；不丢失但需要主动循环
  • 切换：监听 'data' / .pipe() / .resume() → flowing；.pause() → paused
  • 理解这俩是 stream 心智模型的关键

highWaterMark：内部 buffer 阈值

正文待写。

  • 默认 16KB（字节流）或 16（object mode）
  • Duplex / Transform 有独立的 readableHighWaterMark / writableHighWaterMark（Node 8+）
  • 调大可减少 read 次数（吞吐增加）；调小可减少内存占用

Backpressure：让 Node 不爆内存的核心机制

正文待写。

  • writable.write(chunk) 返回 true：buffer 还没满，继续写
  • writable.write(chunk) 返回 false：buffer 满了，暂停 source 等 'drain' event
  • 'drain' 触发后恢复
  • 这套机制让 Node 即使处理 10GB 文件也不会爆 16KB 内存

pipeline vs .pipe()

正文待写。

  • .pipe()：传统写法；错误传播不全（一个 stream 错另一个不会清理）
  • stream.pipeline()：现代写法；自动处理错误传播 + 资源释放
  • stream.promises.pipeline()：Promise 版本

Async iterator

正文待写。for await (const chunk of readable) { ... } ——现代 stream 消费方式（Node 10+）。比 'data' 事件更易写、易读、错误处理自然。

WHATWG Streams（stream/web）

正文待写。Node 16+ 加的web 标准 stream API（ReadableStream / WritableStream / TransformStream）——与传统 Node Streams不兼容但正在融合。fetch 返回的 body 就是 web stream。

04 典型代码

javascript 
import { pipeline } from 'node:stream/promises';
import { createReadStream, createWriteStream } from 'node:fs';
import { createGzip } from 'node:zlib';

// 现代写法：pipeline + Promise
await pipeline(
  createReadStream('input.txt'),
  createGzip(),
  createWriteStream('input.txt.gz')
);

// 用 async iterator 消费
async function processLargeFile(path) {
  const readable = createReadStream(path);
  for await (const chunk of readable) {
    // 每个 chunk 自动 backpressure
    await processChunk(chunk);
  }
}

05 常见陷阱

不要混用 .pipe() 和 await pipeline()

同一 stream 链选一种风格，避免错误传播混乱。

监听 'data' 后无法回 paused 模式

除非显式 .pause()。新代码用 async iterator 避免这个状态机问题。

node streams 与 web streams 不直接兼容

fetch 给的是 web ReadableStream，要用 Readable.fromWeb() 转成 node stream。
