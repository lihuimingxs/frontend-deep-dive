fs / Buffer / process

Node 标准库的另外三个核心 API 群。文件 I/O 走 Worker Pool（与网络不同）；Buffer 是 Node 二进制基础，不在 V8 堆里；process 是运行时环境的 entry。

阅读时长 ~25 分钟 前置 P2.1 libuv + V8 P4.1 堆布局 主要来源 nodejs.org/api/fs · nodejs.org/api/buffer · nodejs.org/api/process
📝 章节状态

骨架版本。对照大纲 node/outline.md § P4.3。

01 速览

  1. fs Promise API（fs.promises）是现代写法——避免 callback hell。
  2. Buffer 不在 V8 堆里——由 ArrayBuffer Allocator 分配（详见 V8 P4.1）。
  3. process.on('uncaughtException') 不要用作错误恢复——进程已不可信。

02 底层逻辑

fs Promise API（Node 10+）

正文待写。

  • import { readFile } from 'node:fs/promises'——现代标准
  • 避免 callback hell，配合 async/await 干净
  • 底层仍是 libuv Worker Pool（4 线程默认）

fs.readFile vs fs.createReadStream

正文待写。

  • readFile：一次性加载整个文件到内存——小文件 < 100MB 用
  • createReadStream：streaming 加载——大文件用，配合 backpressure 不爆内存
  • 判据：能不能预估文件大小？预估 > 100MB 必须 stream

fs.watch 跨平台坑

正文待写。

  • Linux：inotify —— 限制 inotify watcher 数量，监视大目录会 EMFILE
  • macOS：FSEvents —— 延迟较大（默认数百 ms 聚合）
  • Windows：ReadDirectoryChangesW —— 行为略不同
  • 跨平台一致需求建议用 chokidar 包装

Buffer：Node 的二进制基础

正文待写。

  • 固定长度的字节数组
  • 不在 V8 堆内——用 ArrayBuffer Allocator 分配（详见 V8 P4.1）
  • 自 Node 4 起本质是 Uint8Array——但 API 比 Uint8Array 更丰富（toString / write / slice 等）
  • 分配方式：Buffer.alloc(n)（zero-fill）/ Buffer.allocUnsafe(n)（不 fill，更快但有未定义内容）
  • SharedArrayBuffer：跨 Worker Threads 共享内存的唯一方式（详见 P5.1）

process 运行时基础

正文待写。

  • process.env：环境变量；POSIX 习惯命名（UPPER_CASE）
  • process.argv：命令行参数；argv[0] 是 node 路径，argv[1] 是脚本路径
  • process.exit(code)：立即退出；不会 drain pending I/O
  • process.exitCode：设置退出码但让 EventLoop 自然结束 —— 推荐
  • process.on('uncaughtException')：crashing 前最后机会做清理；不要用作错误恢复
  • process.cwd() / process.platform / process.version 等运行时信息

03 应用场景

场景 A · 文件 I/O 选 readFile vs createReadStream

正文待写。判据：文件大小预估 + 是否需要边读边处理。

场景 B · 安全用 Buffer.alloc 而非 allocUnsafe

正文待写。除非性能极敏感且确认会立即 fill；allocUnsafe 内容是上次内存里的随机字节（信息泄漏风险）。

场景 C · 优雅退出 process

正文待写。用 process.exitCode = 1 + 让 EventLoop 自然结束；不要硬调 process.exit()。

04 典型代码

javascript 
// 现代 fs 用法
import { readFile, writeFile } from 'node:fs/promises';
import { createReadStream } from 'node:fs';

// 小文件
const config = JSON.parse(await readFile('config.json', 'utf-8'));

// 大文件流式
const readable = createReadStream('huge.log');
for await (const chunk of readable) {
  process(chunk);
}

// 优雅退出
process.on('SIGTERM', async () => {
  console.log('shutting down');
  await server.close();   // 等 server 关
  process.exitCode = 0;   // 让 EventLoop 自然结束
});

05 常见陷阱

Buffer.allocUnsafe 内容是未定义的

可能含上次该内存里的敏感数据 —— 安全敏感场景必须 alloc 不用 allocUnsafe。

process.exit() 强行终止丢数据

正在写的文件 / 正在发的请求会被截断。优先用 process.exitCode + 自然退出。

uncaughtException 后进程已不可信

不要在 handler 里继续业务；只做日志 + 优雅退出。
