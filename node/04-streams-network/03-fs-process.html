fs / Buffer / process

Node 标准库的另外三个核心 API 群。文件 I/O 走 Worker Pool（与网络不同）；Buffer 是 Node 二进制基础，不在 V8 堆里；process 是运行时环境的 entry。

阅读时长 ~25 分钟 前置 P2.1 libuv + V8 P4.1 堆布局 主要来源 nodejs.org/api/fs · nodejs.org/api/buffer · nodejs.org/api/process

01 速览

  1. fs Promise API（fs.promises）是现代写法——避免 callback hell。
  2. Buffer 不在 V8 堆里——由 ArrayBuffer Allocator 分配（详见 V8 P4.1）。
  3. process.on('uncaughtException') 不要用作错误恢复——进程已不可信。

02 底层逻辑

fs Promise API（Node 10+）：现代写法

fs 模块原本只有 callback API（Node 0.x 设计选择，那时 Promise 还没进 ES 标准）。Node 10 起加 fs.promises—— 同名函数的 Promise 版本：

javascript 
// 旧 callback 风格
import fs from 'node:fs';
fs.readFile('file.txt', 'utf-8', (err, data) => {
  if (err) return console.error(err);
  console.log(data);
});

// 现代 Promise 风格（Node 10+）
import { readFile } from 'node:fs/promises';
const data = await readFile('file.txt', 'utf-8');
console.log(data);

底层仍是 libuv Worker Pool（4 线程默认，详见 P2.1）—— Promise 只是 API 包装，并发限制不变。新代码默认用 node:fs/promises，老代码逐步迁移。

fs.readFile vs fs.createReadStream：什么时候用哪个

两个 API 解决同一问题（读文件）但语义完全不同：

  • readFile(path)：一次性加载整个文件到内存 → 返回 Buffer / String。简单小文件用（< 100 MB）。配置 / JSON / 模板等场景默认选择
  • createReadStream(path)：streaming 加载 → 返回 Readable stream。大文件 / 流式处理用（> 100 MB 或不预知大小），配合 pipeline + backpressure 不爆内存

决策判据：能不能预估文件大小？预估 > 100 MB 必须 stream；不能预估（用户上传 / 外部数据）也用 stream 保险。10 MB 的 JSON 用 readFile 没问题—— 不要过早 over-engineering。

fs.watch 跨平台坑

fs.watch（监听文件变化）在三个平台行为差异大—— Node 用各 OS 原生 API 但行为不一致：

  • Linux · inotify：每个监视占用一个 inotify watcher。系统默认限制几千个—— 监视大目录递归会 EMFILE 错误。修法：echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf
  • macOS · FSEvents：默认聚合延迟数百毫秒—— 写文件后看到事件慢半拍。这对 dev server 体验影响大
  • Windows · ReadDirectoryChangesW：递归 watch 行为略不同

实践：跨平台一致需求用 chokidar npm 包—— 它在 fs.watch 之上做平台差异统一 + 事件去抖。Vite / Vitest / nodemon 都用 chokidar。

Buffer：Node 的二进制基础

Buffer 是 Node 处理二进制数据的核心 —— HTTP 请求体 / 文件内容 / TCP socket 数据 / 加密 / 压缩，所有"非字符串"数据都是 Buffer。设计要点：

  • 固定长度的字节数组——分配后大小不可变（要变就 slice 出新 Buffer）
  • 不在 V8 堆内——用 ArrayBuffer Allocator 分配（详见 V8 P4.1）。意味着 Buffer 不受 V8 堆大小限制（--max-old-space-size 不影响），也不被 V8 GC 直接管理
  • 自 Node 4 起本质是 Uint8Array——但 API 比 Uint8Array 更丰富（toString('utf-8') / write / slice / compare 等）

分配方式：alloc vs allocUnsafe

有两种分配 API：

  • Buffer.alloc(n)：分配 + zero-fill（清零内存）。安全选择，初始内容确定为 0
  • Buffer.allocUnsafe(n)：分配但不 fill—— 内容是上次该内存里的随机字节。更快（省 zero-fill 一步）但有信息泄漏风险—— 上一次的内存可能含敏感数据（密码 / token / 用户信息）

实践规则：默认用 Buffer.alloc。除非性能极敏感且你立刻完整 fill 这块内存（如 fs.read 立即填满），否则不要用 allocUnsafe。

相关：SharedArrayBuffer—— 跨 Worker Threads 共享内存的唯一方式，配合 Atomics 用（详见 P5.1 与 ECMA P7.5 §24 Memory Model）。

process：运行时环境的入口

process 是 Node 全局对象，提供运行时环境的所有信息和控制：

  • process.env：环境变量。POSIX 习惯命名（UPPER_CASE）。所有值都是字符串，process.env.PORT === "3000" 不是 number，要 parseInt
  • process.argv：命令行参数数组。argv[0] 是 node 路径，argv[1] 是脚本路径，argv[2+] 是用户参数。CLI 工具用 commander / yargs / Node 22+ 内置 util.parseArgs 解析
  • process.exit(code)：立即退出 —— 不会 drain pending I/O，正在写的文件 / 正在发的请求会被截断。除非紧急情况否则不要用
  • process.exitCode：设置退出码但让 EventLoop 自然结束 —— 推荐用法
  • process.on('uncaughtException')：crashing 前最后机会做清理（写日志 / 发告警）。不要用作错误恢复—— 进程已不可信，应该尽快退出让 process manager 重启
  • process.on('SIGTERM') / SIGINT：优雅关闭信号处理（容器编排 / Ctrl+C）
  • process.cwd() / process.platform / process.version：运行时信息
⚠️ process.exit() 是危险操作

process.exit(0) 会立即终止—— 任何 pending I/O（文件写、网络发）都被截断。这在生产环境会导致数据损坏（半写文件、半发请求）。正确做法：设 process.exitCode = 0—— EventLoop 自然结束后退出。或者显式关闭资源（await server.close()）后再退出。

03 应用场景

场景 A · 文件 I/O 选 readFile vs createReadStream

判据：(1) 能预估文件 < 100MB → readFile（简单）；(2) > 100MB 或不可预知 → createReadStream + pipeline；(3) 需要边读边处理（流式解析 / 流式发往下游）→ stream 路径，不要先 readFile 再切。常见反模式：先 readFile 再 split('\n') 处理大日志—— 100GB 日志会爆 100GB 内存。

场景 B · 默认用 Buffer.alloc，不用 allocUnsafe

除非性能极敏感且确认立刻完整 fill，否则永远用 Buffer.alloc。allocUnsafe 内容是上次内存里的随机字节—— 上一次可能含密码 / token / 用户信息，留在 buffer 里发给客户端 = 信息泄漏 CVE。性能差异 alloc vs allocUnsafe 大概 2-3×—— 但只在分配热路径明显，绝大多数场景看不出。

场景 C · 优雅退出 process（生产必须）

生产服务需要优雅关闭—— 收到 SIGTERM（容器编排发的）后停止接受新请求 + 完成 in-flight 请求 + 关闭 DB 连接。模板：

javascript 
async function shutdown(signal) {
  console.log(`Received ${signal}, shutting down`);
  server.close(async () => {           // 停止接受新请求
    await db.close();                   // 关 DB 连接
    process.exitCode = 0;               // 让 EventLoop 自然结束
  });
  setTimeout(() => process.exit(1), 30_000);  // 30s 兜底强退
}
process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT',  () => shutdown('SIGINT'));

场景 D · process.env 类型陷阱

常见 bug：const port = process.env.PORT——port 是字符串不是 number。后面 port + 1 变成 "30001" 不是 3001。修法：用 zod / valibot 等 schema 库做 env 解析 + 类型校验。或最简：const port = Number(process.env.PORT) || 3000。生产应用启动期就验证所有环境变量—— 缺失或类型错应该 fail-fast 而不是运行时炸。

04 典型代码

javascript 
// 现代 fs 用法
import { readFile, writeFile } from 'node:fs/promises';
import { createReadStream } from 'node:fs';

// 小文件
const config = JSON.parse(await readFile('config.json', 'utf-8'));

// 大文件流式
const readable = createReadStream('huge.log');
for await (const chunk of readable) {
  process(chunk);
}

// 优雅退出
process.on('SIGTERM', async () => {
  console.log('shutting down');
  await server.close();   // 等 server 关
  process.exitCode = 0;   // 让 EventLoop 自然结束
});

05 常见陷阱

Buffer.allocUnsafe 内容是未定义的

可能含上次该内存里的敏感数据 —— 安全敏感场景必须 alloc 不用 allocUnsafe。

process.exit() 强行终止丢数据

正在写的文件 / 正在发的请求会被截断。优先用 process.exitCode + 自然退出。

uncaughtException 后进程已不可信

不要在 handler 里继续业务；只做日志 + 优雅退出。
