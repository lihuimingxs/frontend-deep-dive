Node 视角的 Event Loop

libuv 提供机制（P2.1），Node 在上面叠了 V8 microtask、process.nextTick、unhandled rejection 等运行时语义。这一章讲完整的 Node 异步模型——读完能精确说出"await fetch() 的回调究竟在哪个 phase 跑"。

阅读时长 ~30 分钟 前置 P2.1 + V8 P5.1 Microtask Hooks 主要来源 nodejs.org/learn/event-loop · V8 Microtask Hook 文档

01 速览

  1. V8 不拥有 Event Loop（详见 V8 P5.1）——V8 提供 microtask 原语，Node 决定每个 phase 之间调用 PerformMicrotaskCheckpoint。
  2. nextTick 是 Node 特有不是规范——在 Deno / Bun 行为可能略有差异。
  3. Microtask 在每个 phase 切换前 drain——所以 setTimeout 回调里 await 的 then，会在下一个 phase 之前执行完。

02 底层逻辑

Node 的 6 phase 与 V8 microtask 协作

P2.1 讲了 libuv 的 6 phase 是EventLoop 的骨架。但 Node 在这骨架之上又叠了几层运行时语义——nextTick queue + V8 microtask queue + unhandled rejection 监测等。理解这三者怎么协作是搞清楚 Node 异步行为的关键。

核心规则：每次 phase 切换前 + 每次 macrotask 完成后，Node 都要 drain（清空）两个队列：先 drain nextTick queue，再 drain microtask queue。也就是说所有 process.nextTick(fn) + Promise.then(fn) + queueMicrotask(fn) 都会"插入"到下一个 phase 之前跑完。

flow 
┌──── timers phase ────┐
│  setTimeout 回调       │
└──────────┬───────────┘
           │ ← drain nextTick queue + microtask queue
┌──── pending callbacks ─┐
│                        │
└──────────┬─────────────┘
           │ ← drain
┌──── poll phase ────┐
│  I/O 回调             │
└──────────┬─────────┘
           │ ← drain
┌──── check phase ────┐
│  setImmediate 回调      │
└──────────┬──────────┘
           │ ← drain
┌──── close phase ────┐
└─────────────────────┘

process.nextTick vs setImmediate vs Promise.then：精确时序

这三个看似类似但语义完全不同。精确顺序：

  • process.nextTick(fn)：最高优先级—— 当前操作完成后立即执行，在任何 phase 切换前。Node 特有，不是 ES 标准。比 microtask 还早 drain。
  • Promise.then(fn) / queueMicrotask(fn)：V8 microtask queue，在 nextTick queue 之后 drain。仍在 phase 切换前。
  • setImmediate(fn)：到下一轮 EventLoop 的 check phase 才跑——慢一个完整 phase。Node 特有，不是浏览器标准。

验证顺序的实验：

javascript 
setImmediate(() => console.log(4));     // check phase 跑
Promise.resolve().then(() => console.log(3));  // microtask 后跑
process.nextTick(() => console.log(2));         // nextTick 先跑
console.log(1);                                  // 同步代码立即跑

// 输出顺序：1 → 2 → 3 → 4

process.nextTick 反模式：递归饿死后续 phase

因为 microtask + nextTick 在 phase 切换前 drain 时不让出控制权，递归 nextTick 会让 EventLoop 永远卡在"drain nextTick"阶段——后续 phase（包括 timers / I/O / setImmediate）永远跑不到。这是 Node 最隐蔽的性能 bug 之一：

javascript 
// ❌ 灾难：会永远不到 setTimeout 回调
function recurse() {
  process.nextTick(recurse);
}
recurse();
setTimeout(() => console.log('starved'), 0);  // 永远不执行

setTimeout vs setImmediate 的微妙差异

这是 Node 异步模型最常被搞错的点。两者看起来都是"立即异步执行"，但精确时序差很多：

  • I/O 回调内部（如在 fs.readFile(...).then(...) 内调用）：setImmediate 一定先于 setTimeout(fn, 0)。原因：I/O 回调在 poll phase 跑，下一个 phase 是 check（setImmediate），再下一轮才到 timers（setTimeout）。
  • 脚本顶层 / 非 I/O 上下文：顺序不定。取决于"EventLoop 启动到第一次 timers phase 用了多少时间"——如果不到 1ms，setTimeout(fn, 0) 会被本轮 timers 处理；超过 1ms，会延后。

实践规则：需要"下一轮异步执行"用 setImmediate（语义更明确）；需要"延迟 N 毫秒"用 setTimeout（这才是它的本意）。不要用 setTimeout(fn, 0) 当"下个 tick"——那是浏览器历史用法，Node 应该用 setImmediate 或 queueMicrotask。

AbortController / AbortSignal：现代取消机制

Node 14+ 全面支持 AbortController（Web 标准 API，浏览器同名）。这是 Node 异步取消的新标准——比传统的 Promise.race + setTimeout 干净得多。

支持 AbortSignal 的 Node API（不完全列表）：fetch / fs.readFile / writeFile / fs.promises.* / net.connect / http.request / setTimeout(fn, ms, { signal }) / events.on(..., { signal })。新写的异步代码默认应该接受 signal 参数—— 这是 Node 异步 API 的现代规范。

--unhandled-rejections=strict：默认 crash

Node 15+ 起，未捕获的 Promise rejection 默认让进程 crash（之前是 warning + 继续运行）。这条改动是 Node 团队的安全决策——未捕获的 rejection 通常意味着代码有 bug，继续跑可能造成数据损坏。

生产应用应该在顶层装 handler：

javascript 
// 优雅处理未捕获 rejection（生产推荐）
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled rejection:', reason);
  // 记日志、发告警、执行清理
  // 然后让进程优雅退出（让 process manager 重启）
  process.exitCode = 1;
});

注意 不要在 handler 里继续业务逻辑——进程已不可信，应该尽快退出让 systemd / pm2 / k8s 重启。

03 应用场景

场景 A · 长循环让出 CPU 防 EventLoop 卡顿

处理大数组 / 长计算时，主线程被占住会让 web server 拒绝新请求。修法：每 N 次循环让出一下：

javascript 
async function processHugeArray(arr) {
  for (let i = 0; i < arr.length; i++) {
    process(arr[i]);
    if (i % 1000 === 0) await new Promise(r => setImmediate(r));
  }
}

setImmediate 让 EventLoop 进入 check phase——意味着 poll phase 的 I/O 回调能跑（处理 web 请求）。比 setTimeout(fn, 0) 更明确。

场景 B · 用 AbortController 实现超时取消

现代 Node 的取消机制，比 Promise.race 干净——取消传到底层（fetch 真断开连接，不只是丢弃响应）：

javascript 
async function fetchWithTimeout(url, ms) {
  const ctrl = new AbortController();
  const timer = setTimeout(() => ctrl.abort(), ms);
  try {
    const res = await fetch(url, { signal: ctrl.signal });
    return await res.json();
  } finally {
    clearTimeout(timer);
  }
}

场景 C · 排查 EventLoop 卡顿

生产环境监控 EventLoop 阻塞——任何主线程长时间不让出都会让所有 web 请求 timeout。用 perf_hooks.monitorEventLoopDelay()：

javascript 
import { monitorEventLoopDelay } from 'node:perf_hooks';
const histogram = monitorEventLoopDelay({ resolution: 20 });
histogram.enable();
setInterval(() => {
  if (histogram.max > 100_000_000) {  // 100ms in ns
    logger.warn('EventLoop blocked', { p99: histogram.percentile(99) });
  }
  histogram.reset();
}, 5000);

EventLoop 延迟 p99 > 100ms 是问题信号——某处有同步 CPU 密集 / 大同步 fs 读 / 大同步 JSON.parse。

场景 D · 不要用 process.nextTick 做"异步初始化"

常见误用：用 process.nextTick 把"同步代码"挪到下一 tick 让初始化排序。这是 Node 反模式——nextTick 容易引入饿死 bug。用 queueMicrotask 或 Promise.resolve().then——它们走 V8 microtask 队列，行为更可预测。

04 典型代码

javascript 
// AbortController 超时取消
const ctrl = new AbortController();
const timer = setTimeout(() => ctrl.abort(), 5000);
try {
  const res = await fetch('https://api.example.com', { signal: ctrl.signal });
  clearTimeout(timer);
  return await res.json();
} catch (e) {
  if (e.name === 'AbortError') console.error('超时');
  throw e;
}

// 长循环让出 CPU
async function processHugeArray(arr) {
  for (let i = 0; i < arr.length; i++) {
    process(arr[i]);
    if (i % 1000 === 0) await new Promise(r => setImmediate(r));
  }
}

05 常见陷阱

不要用 process.nextTick 做"异步初始化"

用 queueMicrotask 或 Promise.resolve().then —— nextTick 容易引入饿死 bug。

setTimeout(fn, 0) 不是立即

libuv 向上取整 1ms；想"下个 tick"用 queueMicrotask。

unhandled rejection 默认 crash（Node 15+）

生产用 process.on('unhandledRejection') 兜底监控+优雅退出。
