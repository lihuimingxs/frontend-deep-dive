libuv 设计：事件循环的"底盘"

libuv 是 Node 用 C 写的跨平台异步 I/O 库——Node 异步行为 100% 由它定义。这一章独立于 Node 讲 libuv 自己：核心抽象（Handle vs Request）、跨平台 I/O 多路复用（epoll / kqueue / IOCP）、Worker Pool（默认 4 线程）、6 个 Event Loop phase。

阅读时长 ~25 分钟 前置 P1 完整 主要来源 docs.libuv.org

01 速览

  1. libuv 是 Node 异步行为的来源——所有 fs / net / http 的"异步"都是 libuv 在底下做。
  2. libuv 给 Node 一个统一抽象——掩盖了 Linux epoll / macOS kqueue / Windows IOCP 的平台差异。
  3. 网络 I/O 与文件 I/O 走不同路——网络用 epoll/kqueue 非阻塞 socket（OS 原生异步），文件 I/O 没有跨平台异步原语，靠 Worker Pool 模拟。

02 底层逻辑

libuv 起源与定位

2011 年 Ryan Dahl 决定把 Node 内部的 libev（Linux 异步 I/O）+ libeio（文件 I/O 模拟）抽离出来，重写成一个独立的跨平台异步 I/O 库——libuv。动机有两层：(1) Node 当时只支持 Linux / macOS（依赖 libev），Windows 支持差，需要原生 IOCP 实现；(2) 团队意识到这套异步 I/O 抽象本身有独立价值——其他语言 / 项目也能用。

libuv 2013 年正式独立成项目（之前住在 Node 仓库内）。后来 Microsoft（赞助 Windows IOCP 部分）/ Apple / Indutny 等多家贡献者参与。今天 libuv 是跨平台异步 I/O 的事实标准——除了 Node，Julia 编程语言、Luvit、neovim 等都用 libuv 做底层 I/O。"用 C 写一份跨平台异步 I/O，让 N 个项目共享"这条路径被 libuv 走通了。

核心抽象 1 · Handle（长寿对象）

libuv 把所有 I/O 资源分两类。Handle 是"长寿对象"——能反复触发回调，存活到显式 uv_close：

  • uv_timer_t：定时器。Node 的 setInterval / setTimeout 底层用这个
  • uv_tcp_t：TCP socket。Node 的 net.Server / net.Socket / HTTP server 都建在这之上
  • uv_fs_event_t：文件系统 watcher。Node 的 fs.watch
  • uv_signal_t：信号处理。Node 的 process.on('SIGTERM')
  • uv_async_t：用于线程间唤醒主循环（Worker Threads 给主线程发消息用这个）

Handle 在 libuv 内部用双向链表挂在 EventLoop 上——loop 每次迭代检查所有 active handle，看哪个有事件触发。

核心抽象 2 · Request（短期一次性操作）

Request 是"一次性操作"——发起 → 完成 → 自动清理：

  • uv_write_t：写入操作（写完触发 callback 然后销毁）
  • uv_getaddrinfo_t：DNS 查询
  • uv_fs_t：文件读写。虽然名字是 fs 但实际是 request——每次 fs.readFile 创建一个 request，完成后销毁
  • uv_work_t：用户自定义异步任务（在 Worker Pool 跑）

Handle vs Request 的区别本质是生命周期：Handle 是"资源"（如 socket、timer），Request 是"动作"（如写一次、查询一次）。同一个 socket（Handle）可以发起多次写（多个 Request）。

跨平台 I/O 多路复用：3 个底层 OS 机制

libuv 最大价值是掩盖 OS 平台差异。三大主流操作系统的异步 I/O 机制完全不同：

  • Linux · epoll（Linux 2.5.44+，2002）—— readiness model：内核告知"文件描述符已就绪可读 / 可写"，应用主动调 read() / write()。是 Linux 高并发服务器的标准（Nginx / Redis 都用）
  • macOS / BSD · kqueue（FreeBSD 4.1，2000）—— readiness model 类似 epoll，但能监听更多事件类型（文件变化、信号、子进程退出等）。Apple 平台默认
  • Windows · IOCP（I/O Completion Port，Win NT）—— completion model：应用直接发 WriteFile()，I/O 完成时内核通知。语义和前两个相反——前两个告知"可以做"，IOCP 告知"已经做完"

libuv 把这两种相反的语义统一成completion 风格的 API——所有操作"发起 → 等待回调通知完成"。在 Linux / macOS 上 libuv 内部把 readiness 转换成 completion（监听 readiness → 立刻执行 read/write → 触发 callback），让上层 API 一致。这就是为什么 Node 异步 API 在三个平台行为相同。

Worker Pool（默认 4 线程）：处理无原生异步的操作

有些操作OS 没有原生异步接口。最典型的是文件 I/O——POSIX 的 read() / write() 是阻塞的，没有 epoll-like 异步版本（2018 年 Linux 5.1 加的 io_uring 是新选择，但 libuv 直到 2024 才开始支持）。

libuv 解决方案：Worker Pool——一个线程池在后台跑这些操作。当应用调 fs.readFile，libuv：(1) 把 read 请求扔进 Worker Pool 队列；(2) 主线程继续跑别的；(3) 某个 worker 线程执行阻塞的 read()，完成后；(4) 通知主线程通过 callback 处理结果。

Worker Pool 默认 4 线程。通过 UV_THREADPOOL_SIZE 环境变量调整（最大 1024）。除了文件 I/O，还有几类操作走 Worker Pool：

  • DNS 查询（dns.lookup 用阻塞的 getaddrinfo；dns.resolve* 用真异步 UDP）
  • crypto 重计算（pbkdf2 / scrypt / randomBytes(big)）
  • zlib 压缩（gzip / deflate）
  • 用户的 uv_work_t（native module 提交的异步工作）
⚠️ Worker Pool 隐藏限制：4 线程默认值

4 线程意味着同时只能有 4 个文件 I/O / DNS / crypto 并发——第 5 个开始排队等。生产环境如果发现"大量并发文件操作时延迟陡升"，几乎就是这个原因。修法：UV_THREADPOOL_SIZE=64 node app.js 调大。注意调到太大反而 thrashing——经验值 16-128。

6 个 Event Loop phase

libuv 的核心是 EventLoop——一个 while 循环，每次迭代按固定顺序经过 6 个 phase 处理对应类型的回调。每个 phase 有自己的 FIFO 队列：

  1. timers：到期的 setTimeout / setInterval 回调
  2. pending callbacks：上一轮延迟到本轮执行的某些 I/O 回调（如 TCP error 延迟）
  3. idle, prepare：内部使用，Node 用户基本看不到
  4. poll：核心阶段——检索新 I/O 事件（epoll_wait / kqueue / GetQueuedCompletionStatus）+ 执行 I/O 回调。如果其他 phase 都空闲，Node 在这里阻塞等 I/O 来
  5. check：setImmediate 回调（与 setTimeout 主要差异在这——setImmediate 一定在 I/O 回调之后跑，setTimeout 在下一轮 timers phase 跑）
  6. close：close 回调（socket.on('close') 等）

关键点：每次 phase 切换前，Node（不是 libuv 自己）会插入 microtask checkpoint—— drain V8 microtask queue（详见 V8 P5.1 + 下一章 P2.2）。这就是为什么 setTimeout(fn1) + Promise.resolve().then(fn2) 的 fn2 比 fn1 早跑——microtask 在 phase 之间 drain。

Node 20 (libuv 1.45+) 的变化

libuv 1.45（2023，对应 Node 20+）做了一个用户可见改动：timer 现在只在 poll 后跑，而非"poll 前后各一次"。这修复了 Node 18 之前的一个微妙时序——某些 timer + I/O 组合行为不可预测。Node 20 之后行为更符合"每个 EventLoop 迭代一轮 timer"的直觉。

03 应用场景

场景 A · 文件 I/O 密集场景调大 Worker Pool

批处理大量文件（日志分析、图像处理、数据迁移）时遇到延迟瓶颈，先用 UV_THREADPOOL_SIZE=64 node app.js 调大。判断是不是 Worker Pool 满：用 process.nextTick 测量 EventLoop 阻塞时间——如果文件操作期间 EventLoop 没卡但文件操作自身延迟陡增，就是 Worker Pool 满了。

场景 B · 理解"网络 1 万并发不卡，10 文件读卡"

常见困惑：Node 服务能扛 1 万 WebSocket 连接，但 10 个并发 fs.readFile 就延迟陡增。原因：网络走 epoll/kqueue 真异步（OS 内核帮你管，可以挂几万 socket）；文件走 Worker Pool 4 线程（用线程池模拟异步，10 个并发就排队）。这是 Node 性能调优最常踩的坑——网络瓶颈和文件 I/O 瓶颈本质完全不同。

场景 C · 跨平台 fs.watch 行为差异

fs.watch 在三个平台行为有差：Linux inotify（限制 watcher 数量，监视大目录会 EMFILE 错误）/ macOS FSEvents（事件聚合延迟数百 ms，看到变化"慢半拍"）/ Windows ReadDirectoryChangesW（递归 watch 行为不同）。跨平台一致需求建议用 chokidar npm 包包一层。

场景 D · setTimeout vs setImmediate 时序

微妙差异：在I/O 回调内部 setImmediate 一定先于 setTimeout(fn, 0)（前者本轮 check phase 跑，后者下轮 timers phase 跑）。其他场景（如脚本顶层）顺序不定（取决于 EventLoop 启动到 timers phase 的时间）。需要"下个 tick 执行"用 setImmediate 或 queueMicrotask，不要用 setTimeout(fn, 0)。

04 典型代码

bash 
# 调大 Worker Pool（默认 4，最多 1024）
UV_THREADPOOL_SIZE=64 node app.js

# 看 libuv 内部活动（debug build 才有）
UV_DEBUG=1 node app.js

05 常见陷阱

UV_THREADPOOL_SIZE 在 Worker 启动后改无效

必须启动前设置环境变量。

crypto 重计算也走 Worker Pool

大量 bcrypt / pbkdf2 操作会和文件 I/O 抢线程。

setTimeout(fn, 0) 实际不是立即执行

libuv 内部会向上取整到 1ms。需要"下个 tick 执行"用 setImmediate / queueMicrotask。
