libuv 设计：事件循环的"底盘"

libuv 是 Node 用 C 写的跨平台异步 I/O 库——Node 异步行为 100% 由它定义。这一章独立于 Node 讲 libuv 自己：核心抽象（Handle vs Request）、跨平台 I/O 多路复用（epoll / kqueue / IOCP）、Worker Pool（默认 4 线程）、6 个 Event Loop phase。

阅读时长 ~25 分钟 前置 P1 完整 主要来源 docs.libuv.org
📝 章节状态

骨架版本。对照大纲 node/outline.md § P2.1。

01 速览

  1. libuv 是 Node 异步行为的来源——所有 fs / net / http 的"异步"都是 libuv 在底下做。
  2. libuv 给 Node 一个统一抽象——掩盖了 Linux epoll / macOS kqueue / Windows IOCP 的平台差异。
  3. 网络 I/O 与文件 I/O 走不同路——网络用 epoll/kqueue 非阻塞 socket（OS 原生异步），文件 I/O 没有跨平台异步原语，靠 Worker Pool 模拟。

核心抽象 1 · Handle

核心抽象 2 · Request

正文待写。短期一次性操作，完成后自动清理：

  • uv_write_t：写入操作
  • uv_getaddrinfo_t：DNS 查询
  • uv_fs_t：文件读写（虽然名字是 fs 但实际是 request）

跨平台 I/O 多路复用

正文待写。

  • Linux：epoll（Linux 2.5.44+）—— readiness model，文件描述符可读 / 可写时通知
  • macOS / BSD：kqueue —— 类似 epoll 但能听更多事件类型
  • Windows：IOCP（I/O Completion Port）—— completion model，I/O 完成时通知
  • libuv 把三种不同模型抽象成统一 API

Worker Pool（默认 4 线程）

正文待写。

  • 处理"无原生异步"的操作：文件 I/O（POSIX 没有真异步文件 API）/ DNS / 用户 work（如 crypto 重计算）
  • 默认 4 线程；通过 UV_THREADPOOL_SIZE 环境变量调整（最多 1024）
  • 隐藏限制：4 线程意味着 4 个并发文件读会"卡住"其他操作
  • 大文件密集服务必须调大

6 个 Event Loop phase

正文待写。顺序：

  1. timers：执行 setTimeout / setInterval 到期回调
  2. pending callbacks：执行某些 I/O 回调（如 TCP error 延迟回调）
  3. idle, prepare：内部使用
  4. poll：检索新 I/O 事件 + 执行 I/O 回调；Node 在这里阻塞等 I/O（如果其他 phase 都空闲）
  5. check：执行 setImmediate 回调
  6. close：执行 close 回调（如 socket.on('close')）

05 常见陷阱

UV_THREADPOOL_SIZE 在 Worker 启动后改无效

必须启动前设置环境变量。

crypto 重计算也走 Worker Pool

大量 bcrypt / pbkdf2 操作会和文件 I/O 抢线程。

setTimeout(fn, 0) 实际不是立即执行

libuv 内部会向上取整到 1ms。需要"下个 tick 执行"用 setImmediate / queueMicrotask。
