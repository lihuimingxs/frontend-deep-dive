Node 与同代人

横向对照。Node 不是孤立的——同时代有 Deno（Ryan Dahl 自己做的"反 Node"）、Bun（用 JSC + Zig 走兼容加速）、Edge Runtime（Cloudflare Workers / Vercel Edge 的受限子集）各自不同的设计选择。读完能回答：Node 的护城河是什么、Bun 的威胁有多大、Edge Runtime 为什么不是"小号 Node"。

阅读时长 ~25 分钟 前置 P1.1 项目史 主要来源 Deno / Bun / Cloudflare Workers 官方文档

01 速览

  1. Node 的护城河是 npm 2.5M+ 包生态——任何替代品要么兼容 Node API，要么放弃 80% 现成生态。
  2. Bun 选择"兼容 + 加速"路径——比 Deno 的"另起炉灶"务实，威胁更直接。
  3. Edge Runtime 是受限子集——只支持 Node API 子集（无 fs / 长连接），写代码要做"边缘可移植性"判断。

02 底层逻辑

同代人 · Deno（Ryan Dahl 2018+）

Deno 是Ryan Dahl 2018 年发布的 Node 替代品——把他 2018 演讲里批评 Node 的 10 条遗憾全部"做成产品"。Deno 不是 fork Node，是另起炉灶——但仍用 V8（这点和 Node 一样）。

Deno 的核心设计选择：

  • 引擎：V8（与 Node 同）+ Tokio（Rust 异步运行时，替代 libuv）。Tokio 是 Rust 生态最成熟的异步运行时，让 Deno 把 I/O 层做得比 Node 更"现代"
  • Web 标准 API 优先：fetch / Response / URL / WebSocket 与浏览器同名同义——同一段代码能跑浏览器和服务器
  • 权限模型：--allow-read=./data --allow-net=api.example.com 显式授权——这是对 Node "默认全权"的最直接反思
  • TypeScript 内置：通过 SWC 即时转译，无需 tsc 编译步骤
  • URL 导入：import { x } from "https://deno.land/std/path/mod.ts"——直接从 URL 加载，绕开 npm 包管理。这是 Deno 最激进的实验

结果：Deno 没成主流。"放弃 npm 生态"代价太大——大多数业务逻辑都依赖 npm 包，Deno 用户要么手动重写依赖，要么转回 Node。2023 年 Deno 终于妥协加了 npm 兼容（npm: 前缀）—— 等于承认 URL 导入路线失败。但 Deno 的设计理念赢了—— Node 22+ 内置 fetch / Permission Model / TS 实验支持都是"偷师" Deno。

同代人 · Bun（Jarred Sumner 2022+）

Bun 是 2022 年 Jarred Sumner 发布的 Node 兼容运行时。Sumner 之前是 Stripe 工程师，专门做服务端 JS 性能优化——发现 Node 启动慢、工具链碎片化、内存占用高。Bun 的设计哲学和 Deno 相反——不试图取代 Node，而是兼容 Node 然后做得更快。

Bun 的核心设计选择：

  • 引擎：JSC（Apple 的 JavaScriptCore，不是 V8）+ Zig 写的运行时（Zig 是新型系统语言，目标取代 C）。详见 V8 P1.2 同代人段
  • 定位 Node 兼容：宣传"Drop-in replacement for Node.js"，大多数 npm 包直接能跑
  • 工具一体化：bun run（runtime）+ bun build（bundler）+ bun install（package manager）+ bun test（test runner）全在一个 binary。装一个 Bun 替代 node + npm + webpack + vitest 整套
  • 启动速度优势：JSC 启动比 V8 快（Apple 优化用于 iOS / Safari 启动场景）+ Zig 实现的内存效率 + 内置 bundler 减少 require 链 → 实测启动比 Node 快 4-5×

对 Node 的威胁：Bun 直接抢走 Node 用户而不要求生态迁移。这种"兼容 + 加速"路径比 Deno "另起炉灶"务实得多。Bun 1.0（2022-07）发布后增长极快——开发体验好的项目（启动 + 测试 + 构建快几倍）尝试 Bun 几乎没成本。

但 Bun 的边界：(1) Node 兼容率约 90%（仍有少量包不能跑）；(2) JSC 在稳态长跑性能不如 V8（V8 多层 JIT 更适合服务端长跑）；(3) Bun 自己作为 runtime 还不到 4 年，生产稳定性比 Node 差。大公司大多数还是 Node + 实验项目用 Bun。

同代人 · Edge Runtime（Cloudflare Workers / Vercel Edge）

Edge Runtime 不是单一项目，是一类"在 CDN 边缘节点跑 JS"的运行时。代表：Cloudflare Workers（2017 起，市场领先）/ Vercel Edge Functions / AWS Lambda@Edge / Deno Deploy。它们和 Node 关系是"不同物种"——不是 Node 的兼容版，是受限子集。

Edge Runtime 核心特征：

  • 引擎：Cloudflare Workers / Vercel 用 V8 Isolate（不是完整 V8 进程）；详见 V8 P1.2 V8 分身史段 和 V8 P5.1 Embedder API
  • 支持的 API 严重受限：只 Web 标准 API（fetch / Response / URL / Headers / WebSocket）+ 部分 Node API（如 node:crypto）。无 fs（CDN 节点没本地文件系统）/ 无长连接（每个请求 30 秒上限）/ 无 Worker Threads（Isolate 已经是隔离单元）
  • 冷启动毫秒级：Workers 用 V8 Isolate-per-tenant + Snapshot，冷启动 < 5ms。Lambda 用 Container per-tenant，冷启动 100ms-1s
  • 定价模型不同：Workers 按 CPU 时间计费（不按内存），鼓励轻量短任务

"边缘可移植性"判断：写代码时考虑"这段代码能不能在 Edge 跑"——避免 fs / 长连接 / setTimeout > 30s / Worker Threads。Next.js / Remix 等 meta-framework 提供"edge-compatible" 模式，限制 API 让代码可移植。

💡 Edge ≠ Lambda

两者都是 serverless，但架构差一个数量级：Cloudflare Workers 是 V8 Isolate-per-tenant（毫秒冷启动 + ~5MB 内存）/ AWS Lambda 是 Container-per-tenant（100ms-1s 冷启动 + 几十 MB 内存）。Workers 适合高并发短任务（API 网关、Auth 中间件、简单 SSR）；Lambda 适合重计算 / 长任务 / 完整 Node API 需求。不要把它们当一类。

三条底层逻辑

Node 的护城河是生态而非技术

Bun 启动比 Node 快、Deno 设计比 Node 现代、Edge Runtime 比 Node 边缘部署快——但 Node 仍是服务端 JS 主流。原因不在 Node 自己技术多强，而在 npm 生态规模——3M+ 包、几十亿 weekly downloads、几乎所有 web 工程的"默认依赖来源"。Node 跑不动一个流行 npm 包是大事件，但 Bun / Deno 跑不动可以"用户自行解决"。这种不对称让任何替代品都要先解决兼容性，然后才能谈加速。

Bun "兼容 + 加速"比 Deno "另起炉灶"更务实

Deno 2018-2022 投入了类似 Bun 量级的资源，结果用户量远小于 Bun。原因：Deno 要求用户改代码（URL 导入、权限模型、不同的标准库），Bun 不要求（直接当 Node 用）。开发者切换工具的痛点是"改代码"而不是"学新语法"。Bun 团队意识到这点——声明 Node-compatible 然后在内部做更快，让用户"无感受地"切换。这条"务实击败学术"的胜利和 TypeScript vs Flow 是同一逻辑——参考 TS Part 1.2。

Edge Runtime 不是"小号 Node"是不同物种

很多人误以为 Cloudflare Workers / Vercel Edge 就是"轻量版 Node"——错。它们的架构 / 定价 / 适用场景都不同：Workers 是 Isolate-per-tenant（同一进程几千 isolate，详见 V8 P5.1），Node 是进程 per 应用；Workers 按 CPU 时间计费鼓励短任务，Node Lambda 按时长计费允许长任务；Workers 不能 fs / 长连接，Node 完整 API。Edge 不是 Node 的子集——是"web 标准 + V8 Isolate"组合的全新平台。设计可移植代码要从一开始就考虑这点。

03 应用场景

场景 A · 选型：Node vs Bun（生产）

判据 4 条：(1) 项目稳定性优先选 Node（Bun 4 年生产历史 vs Node 16 年）；(2) 启动速度敏感（CLI 工具 / 函数计算）选 Bun；(3) 已有 Node 项目大规模迁移成本不如换 LTS 版本来得直接；(4) 新项目尝鲜且团队接受新工具——Bun 适合。2025-04 当前大公司大多 Node + 实验项目 Bun。

场景 B · 选型：什么场景用 Edge Runtime

适合 Edge：API 网关 / Auth 中间件 / 简单 SSR / 静态资源动态变体 / A-B 测试 routing / 边缘缓存逻辑。不适合：长 WebSocket（30 秒上限）/ 文件上传（无 fs）/ 重计算（CPU 时间贵）/ 完整 Node API 需求（如 sharp 图像处理）。把"不需要 fs / 长连接 / 重计算的轻量逻辑"挪到 Edge，能极大降延迟（边缘节点离用户近）+ 降成本（按 CPU 时间计费）。

场景 C · 多 runtime 兼容写法

一段代码想同时跑 Node / Deno / Bun / Edge：(1) 优先用 Web 标准 API（fetch / Response / URL / WebSocket / Crypto）—— 4 个 runtime 都支持；(2) 避免 require 用 import（ESM 是公共标准）；(3) 不依赖 fs / child_process 等 Node 特有 API；(4) 用 process.env 替代会失败——改用 globalThis..env 或 import.meta.env（Vite/Bun 标准）。Next.js / Remix / Hono 等 meta-framework 提供这套抽象。

场景 D · 评估"该不该换 Bun"

团队提出"换 Bun 取代 Node"。务实评估：(1) Bun 兼容率 ~90%——你的关键 npm 包都能跑吗？（用 bun install + bun run 跑测试套件）；(2) 你的瓶颈真的是引擎吗？大多数 web 服务瓶颈在数据库 / 网络 / 算法，换引擎只能省 5-10%；(3) Bun 4 年生产历史——你能容忍少数边缘 bug 吗？大多数情况答案是"等 Bun 5.0 + 1 年"再考虑全面切换。但 CI / CLI / 工具链场景可以现在就用。

05 常见陷阱

"Bun 比 Node 快"过于简化

启动 / 短脚本 Bun 优势明显；长跑稳态因引擎差异（V8 vs JSC）部分场景 Node 反而快。

Edge Runtime ≠ Node

没有 fs / 长连接 / setTimeout > 30s / Worker Threads。直接搬 Node 代码几乎一定坏。

Deno 的 URL 导入并未成为主流

2024 起 Deno 也在加 npm 兼容（npm: 前缀）——市场用脚投票了。
