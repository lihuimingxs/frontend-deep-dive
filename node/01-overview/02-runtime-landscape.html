Node 与同代人

横向对照。Node 不是孤立的——同时代有 Deno（Ryan Dahl 自己做的"反 Node"）、Bun（用 JSC + Zig 走兼容加速）、Edge Runtime（Cloudflare Workers / Vercel Edge 的受限子集）各自不同的设计选择。读完能回答：Node 的护城河是什么、Bun 的威胁有多大、Edge Runtime 为什么不是"小号 Node"。

阅读时长 ~25 分钟 前置 P1.1 项目史 主要来源 Deno / Bun / Cloudflare Workers 官方文档
📝 章节状态

骨架版本。对照大纲 node/outline.md § P1.2。

01 速览

  1. Node 的护城河是 npm 2.5M+ 包生态——任何替代品要么兼容 Node API，要么放弃 80% 现成生态。
  2. Bun 选择"兼容 + 加速"路径——比 Deno 的"另起炉灶"务实，威胁更直接。
  3. Edge Runtime 是受限子集——只支持 Node API 子集（无 fs / 长连接），写代码要做"边缘可移植性"判断。

02 底层逻辑

同代人 · Deno（Ryan Dahl 2018+）

正文待写。关键点：

  • 引擎：V8（与 Node 同）+ Tokio（Rust 异步运行时，替代 libuv）
  • Web 标准 API：fetch / Response / URL 与浏览器同名同义
  • 权限模型：--allow-read / --allow-net 显式授权——是对 Node "默认全权"的反思
  • TypeScript 内置：通过 swc 即时转译
  • URL 导入：import { x } from "https://deno.land/std/path/mod.ts"——绕开 npm 包管理
  • 试图解决 Ryan 自己批评的 Node 问题

同代人 · Bun（Jarred Sumner 2022+）

正文待写。关键点：

  • 引擎：JSC（不是 V8）+ Zig 写的运行时
  • 定位：Node 兼容层 + 加速——大多数 Node 包能直接跑
  • 工具一体化：runtime + bundler + package manager + test runner 全在一个 binary
  • 启动速度优势：JSC 本身比 V8 启动快 + Zig 实现内存效率高 + 内置 bundler 减少 require 链
  • 对 Node 的威胁：直接抢走 Node 用户而不要求生态迁移

同代人 · Edge Runtime（Cloudflare Workers / Vercel Edge）

正文待写。关键点：

  • 引擎：V8（Workers）或 V8 cold-start 优化（Lambda@Edge）
  • 支持子集：只 Web 标准 API + 部分 Node API；通常无 fs / 长连接 / Worker Threads
  • 冷启动：Workers Isolate-per-tenant 毫秒级；Lambda Container 秒级
  • 详见 V8 P1.2「V8 分身史」段
  • "边缘可移植性"判断：写代码时考虑哪些 API 能在 Edge 跑

05 常见陷阱

"Bun 比 Node 快"过于简化

启动 / 短脚本 Bun 优势明显；长跑稳态因引擎差异（V8 vs JSC）部分场景 Node 反而快。

Edge Runtime ≠ Node

没有 fs / 长连接 / setTimeout > 30s / Worker Threads。直接搬 Node 代码几乎一定坏。

Deno 的 URL 导入并未成为主流

2024 起 Deno 也在加 npm 兼容（npm: 前缀）——市场用脚投票了。
