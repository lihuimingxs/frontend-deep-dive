P3.3 · useReducer / useContext

useReducer 是复杂状态管理的工具——多个字段联动 / 多种 action 类型的场景下比 useState 更清晰。useContext 是跨组件共享的工具——避免 prop drilling。两个 Hook 经常组合使用（如全局主题 / 用户登录态 / Toast 系统）。本章讲清两者的正确用法、性能陷阱、TypeScript discriminated union 模式，以及 React 19 use(Context) 与 useContext 的取舍。

📐 本章为什么把两个 Hook 合一章

useReducer 和 useContext 经常组合使用—— 全局状态管理的"轻量 Redux 替代"模式：useReducer 管 state，useContext 跨组件共享。P7.1 生态章已讲了 Zustand / Redux Toolkit 等专业方案，本章讲不引第三方库时怎么用 React 内置 Hooks 实现。

阅读时长 ~30 分钟 前置 P3.1 useState 深度 + P3.2 useEffect 对应 react.dev useReducer · useContext

01 速览

三句话定义本章：

  1. useReducer 是 useState 的超集（P3.1 已讲）。4 个场景必用：① 多字段联动（一个 action 改多个 state） ② 多种 action 类型（dispatch 不同 action） ③ 复杂转换逻辑（reducer 集中管理） ④ 测试 / 重放需求（pure function 易测）。简单场景仍用 useState。
  2. useContext 解决 prop drilling—— 但有性能陷阱：Context value 引用变化时所有消费者都重渲染，不管它们关心的字段变没变。所以"大对象 Context" 是反模式。
  3. 组合用法：useReducer + useContext = "轻量 Redux"。state + dispatch 通过 Context 传给所有后代。但要拆 2 个 Context（StateContext + DispatchContext），让只用 dispatch 的组件不被 state 变化触发重渲染。

看一段最小代码识别两个 Hook：

javascript · useReducer + useContext 组合 
// 1. 定义 reducer + initial state
const initialState = { count: 0, user: null };

function appReducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + 1 };
    case 'set_user':
      return { ...state, user: action.payload };
    default:
      return state;
  }
}

// 2. 创建两个 Context（state + dispatch 分离）
const StateContext = createContext(initialState);
const DispatchContext = createContext(() => {});

// 3. Provider 用 useReducer
function AppProvider({ children }) {
  const [state, dispatch] = useReducer(appReducer, initialState);
  return (
    <StateContext.Provider value={state}>
      <DispatchContext.Provider value={dispatch}>
        {children}
      </DispatchContext.Provider>
    </StateContext.Provider>
  );
}

// 4. 子组件按需消费
function CountDisplay() {
  const { count } = useContext(StateContext);   // 仅订阅 state
  return <span>{count}</span>;
}

function IncrementButton() {
  const dispatch = useContext(DispatchContext);  // 仅订阅 dispatch（永远稳定）
  return <button onClick={() => dispatch({ type: 'increment' })}>+</button>;
  // 不会因为 state 变化而重渲染（DispatchContext 的 value 永远是同一 dispatch）
}

02 底层逻辑

useReducer 完整签名与心智

typescript · useReducer 类型定义 
// 重载 1：基础
function useReducer<R extends Reducer<any, any>>(
  reducer: R,
  initialState: ReducerState<R>,
): [ReducerState<R>, Dispatch<ReducerAction<R>>];

// 重载 2：含 lazy initializer（init 函数延迟求值，类似 useState lazy initial）
function useReducer<R extends Reducer<any, any>, I>(
  reducer: R,
  initialArg: I,
  init: (initial: I) => ReducerState<R>,
): [ReducerState<R>, Dispatch<ReducerAction<R>>];

type Reducer<S, A> = (state: S, action: A) => S;
type Dispatch<A> = (action: A) => void;

// 用法：
const [state, dispatch] = useReducer(reducer, initialState);
const [state, dispatch] = useReducer(reducer, initArg, (arg) => computeInitial(arg));

// 关键事实：
// - reducer 必须是纯函数（不能 mutate state、不能调副作用）
// - dispatch 引用永远稳定（同 useState 的 setState）
// - state 是不可变的（reducer 必须返回新引用）

何时用 useReducer（4 场景）

场景 useState 的痛点 useReducer 优势
多字段联动 多个 setState 难以协调（"用户登录"要同时改 user、token、loading、error） 一个 action 集中改多个字段
多种 action 类型 handler 散布各处（add/remove/toggle/edit todo） reducer 集中所有转换逻辑
复杂转换逻辑 setState 内调函数嵌套 reducer 是 pure function，易测试
状态机 / 向导流 useState 表达不清晰（每步 transition 散落） reducer 自然适配状态机模式

反向：以下场景仍用 useState：

  • 简单 state（boolean / 数字 / 字符串）
  • 3 个以下独立字段（form 几个 input）
  • state 变化逻辑简单（直接 setState 新值）

action 类型设计（discriminated union）⭐

TypeScript 中的 discriminated union（详见 TS · Narrowing）是 reducer action 类型的事实标准：

typescript · action discriminated union 模式 
type State = {
  todos: Todo[];
  filter: 'all' | 'active' | 'completed';
};

// ⭐ Action 是 discriminated union —— type 字段是 discriminator
type Action =
  | { type: 'add'; payload: { text: string } }
  | { type: 'toggle'; payload: { id: string } }
  | { type: 'remove'; payload: { id: string } }
  | { type: 'edit'; payload: { id: string; text: string } }
  | { type: 'set_filter'; payload: { filter: State['filter'] } }
  | { type: 'clear_completed' };  // 没有 payload

function todoReducer(state: State, action: Action): State {
  switch (action.type) {
    case 'add':
      return {
        ...state,
        todos: [...state.todos, { id: crypto.randomUUID(), text: action.payload.text, done: false }],
        //                                                       ↑ TS narrowing：知道 payload 是 { text: string }
      };
    case 'toggle':
      return {
        ...state,
        todos: state.todos.map(t => t.id === action.payload.id ? { ...t, done: !t.done } : t),
      };
    case 'remove':
      return {
        ...state,
        todos: state.todos.filter(t => t.id !== action.payload.id),
      };
    case 'edit':
      return {
        ...state,
        todos: state.todos.map(t => t.id === action.payload.id ? { ...t, text: action.payload.text } : t),
      };
    case 'set_filter':
      return { ...state, filter: action.payload.filter };
    case 'clear_completed':
      return { ...state, todos: state.todos.filter(t => !t.done) };
    default: {
      // exhaustive check：TS 知道 default 分支不可达
      const _exhaustive: never = action;
      return state;
    }
  }
}

// 使用：
dispatch({ type: 'add', payload: { text: 'Hello' } });
//        ↑ TS 自动检查 type + payload 类型组合

dispatch({ type: 'toggle', payload: { text: 'wrong' } });
// ❌ TS 错：toggle 的 payload 是 { id }，不是 { text }

dispatch({ type: 'invalid' });
// ❌ TS 错：'invalid' 不在 Action 类型里

// 加新 action 类型？
// 只要在 Action 中加一个变体，TS 自动报错所有未处理 case 的 reducer
// → "exhaustive checking" 神器：忘了处理就不能编译

// 详细 discriminated union 见 TS · Narrowing

useContext 完整签名

typescript · useContext + Context 类型 
// 1. 创建 Context（含默认值）
const ThemeContext = createContext<Theme>({ mode: 'light', primary: '#000' });
const UserContext = createContext<User | null>(null);  // 允许 null 作为"未登录"

// 2. Provider 组件
function App() {
  const [theme, setTheme] = useState({ mode: 'light', primary: '#000' });
  return (
    <ThemeContext.Provider value={theme}>
      <Layout />
    </ThemeContext.Provider>
  );
}

// 3. 消费者用 useContext
function Button() {
  const theme = useContext(ThemeContext);
  return <button style={{ background: theme.primary }}>Click</button>;
}

// 4. 没有 Provider 时用 createContext 的默认值
//    所以 createContext 的默认值要设计成"合理的 fallback"
//    或者强制必须有 Provider（默认值用 null + 自定义 hook 检查）

// 5. 自定义 Hook 模式（推荐 —— 避免到处写 useContext + null 检查）
function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme must be used within ThemeProvider');
  return ctx;
}

Context 的性能陷阱 ⭐

这是 useContext 最常见的 bug 来源。

🎯 Context value 引用变化 → 所有消费者都重渲染

React 用 Object.is 比较 Context value。引用变化时所有 useContext 消费者都重新渲染，不管它们关心的字段是否变化—— 这是性能陷阱的根源。React.memo 包子组件也救不了，因为是 useContext hook 触发的重渲染（React.memo 仅比较 props）。

javascript · Context 性能陷阱演示 
// ❌ 性能问题：每次 App render 都创建新对象 → 所有消费者重渲染
function App() {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');

  // 每次 App render 都新建 value 对象 → 所有 useContext 消费者重渲染
  const value = { user, theme, setUser, setTheme };

  return (
    <AppContext.Provider value={value}>
      <Header />     {/* 用了 user，每次都重渲染 */}
      <Sidebar />    {/* 用了 theme，theme 没变也重渲染 */}
      <Footer />     {/* 没用 context，但也...等等 Footer 不消费 context 不受影响 */}
    </AppContext.Provider>
  );
}

// 解 1：用 useMemo 稳定 value
function App() {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');

  const value = useMemo(
    () => ({ user, theme, setUser, setTheme }),
    [user, theme]   // 仅 user / theme 变化时才新建 value
  );

  return <AppContext.Provider value={value}>...</AppContext.Provider>;
}
// 但仍有问题：user 变 → theme 消费者也重渲染（value 引用变了）

// 解 2：拆 Context（按更新频率分组）⭐ 推荐
const UserContext = createContext(null);
const ThemeContext = createContext(null);

function App() {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');

  return (
    <UserContext.Provider value={user}>
      <ThemeContext.Provider value={theme}>
        ...
      </ThemeContext.Provider>
    </UserContext.Provider>
  );
}
// 每个 Context 单独维护 → 仅相关 Context 变化时消费者重渲染

// 解 3：state + dispatch 分两个 Context（前面已讲）
<StateContext.Provider value={state}>
  <DispatchContext.Provider value={dispatch}>
    {children}
  </DispatchContext.Provider>
</StateContext.Provider>
// dispatch 永远稳定，仅订阅 dispatch 的组件永不重渲染

Context Selector 模式（user-land 实现）

React 内置 useContext 没有"仅订阅一个字段" 的能力（这是 Redux selector / Zustand 的核心）。社区有几个 user-land 实现：

javascript · 自定义 Context Selector 
// 简化版（基于 useSyncExternalStore，React 18+）
function createContextSelector(initialState) {
  const subscribers = new Set();
  let state = initialState;

  return {
    setState(updater) {
      state = typeof updater === 'function' ? updater(state) : updater;
      subscribers.forEach(fn => fn());
    },
    useSelector(selector) {
      return useSyncExternalStore(
        (cb) => { subscribers.add(cb); return () => subscribers.delete(cb); },
        () => selector(state),
      );
    },
  };
}

const userStore = createContextSelector({ name: '', age: 0 });

function Component() {
  const name = userStore.useSelector(s => s.name);  // 仅 name 变化时重渲染
  return <div>{name}</div>;
}

// 实际项目用第三方库：
// - useContextSelector (use-context-selector npm 包)
// - Zustand 内置 selector
// - Jotai 用 atom 天然解决这个问题

// 详见 P7.1 生态章节的状态管理对比

React 19 use(Context) vs useContext

React 19 引入 use API（详见 P5.1 / P6.3），其中一个用法是读 Context。它和 useContext 的关键差异：

维度 useContext use(Context)
调用位置 必须组件顶层（Rules of Hooks） 可在条件 / 循环 / 函数嵌套调用
是 Hook 吗 是（链表节点） 不是 —— 新机制（不依赖 hooks linked list）
性能 同样订阅 Context 同样订阅 Context
取舍 大多数场景用 仅在条件分支调用 Context 时用
javascript · use(Context) 在条件分支调用 
// useContext 必须顶层（Rules of Hooks）
function Component({ shouldUseTheme }) {
  const theme = useContext(ThemeContext);  // 即使 shouldUseTheme = false 也调用

  if (shouldUseTheme) {
    return <div className={theme.mode}>...</div>;
  }
  return <div>...</div>;
}

// React 19 use(Context) 可在条件分支
function Component({ shouldUseTheme }) {
  if (shouldUseTheme) {
    const theme = use(ThemeContext);  // ✅ 仅条件成立时才 subscribe
    return <div className={theme.mode}>...</div>;
  }
  return <div>...</div>;
}

// 实际场景：根据 props 决定是否需要某个 Context
// 大多数场景仍用 useContext（更清晰的依赖关系）
// use(Context) 仅作为"解锁条件订阅" 的工具

03 应用场景

场景 A · Toast 系统（轻量"全局状态" 不引库）

团队不想引 Zustand 等库，用 useReducer + useContext 实现 Toast：

javascript · 轻量 Toast 系统 
// types
type Toast = { id: string; message: string; type: 'info' | 'error' | 'success' };
type State = { toasts: Toast[] };
type Action =
  | { type: 'show'; payload: Omit<Toast, 'id'> }
  | { type: 'dismiss'; payload: { id: string } };

// reducer
function toastReducer(state: State, action: Action): State {
  switch (action.type) {
    case 'show':
      return {
        toasts: [...state.toasts, { ...action.payload, id: crypto.randomUUID() }],
      };
    case 'dismiss':
      return { toasts: state.toasts.filter(t => t.id !== action.payload.id) };
  }
}

// Context（拆 state + dispatch）
const StateContext = createContext<State>({ toasts: [] });
const DispatchContext = createContext<Dispatch<Action>>(() => {});

// Provider
function ToastProvider({ children }: { children: ReactNode }) {
  const [state, dispatch] = useReducer(toastReducer, { toasts: [] });
  return (
    <StateContext.Provider value={state}>
      <DispatchContext.Provider value={dispatch}>
        {children}
        <ToastList />
      </DispatchContext.Provider>
    </StateContext.Provider>
  );
}

// Custom hooks（封装 useContext + 错误兜底）
function useToasts() {
  return useContext(StateContext).toasts;
}
function useToastDispatch() {
  return useContext(DispatchContext);
}

// 用法
function MyButton() {
  const dispatch = useToastDispatch();
  return (
    <button onClick={() => dispatch({
      type: 'show',
      payload: { message: 'Saved!', type: 'success' },
    })}>
      Save
    </button>
  );
}

// 这里 MyButton 仅订阅 dispatch（永远稳定）—— 任何 toast 增删都不重渲染它

// 如果用 Zustand 替代：~ 5 行代码 + 全局 hook，无 Provider 嵌套
// 取舍：useReducer + useContext 不引库；Zustand 更简洁但有依赖

场景 B · 多步表单状态机（XState 之外的轻量方案）

注册流程 5 步，每步可能跳到不同下一步，可前进 / 后退 / 重置。用 useReducer 实现状态机：

javascript · 状态机风格 reducer 
type Step = 'basic' | 'profile' | 'verify' | 'plan' | 'done';
type State = {
  step: Step;
  data: { basic?: BasicInfo; profile?: Profile; plan?: Plan };
};
type Action =
  | { type: 'next'; payload: { from: Step; data: any } }
  | { type: 'back' }
  | { type: 'reset' }
  | { type: 'jump'; payload: { to: Step } };

const transitions: Record<Step, { next: Step | null; back: Step | null }> = {
  basic: { next: 'profile', back: null },
  profile: { next: 'verify', back: 'basic' },
  verify: { next: 'plan', back: 'profile' },
  plan: { next: 'done', back: 'verify' },
  done: { next: null, back: null },
};

function wizardReducer(state: State, action: Action): State {
  switch (action.type) {
    case 'next': {
      const next = transitions[state.step].next;
      if (!next) return state;
      return {
        step: next,
        data: { ...state.data, [action.payload.from]: action.payload.data },
      };
    }
    case 'back': {
      const back = transitions[state.step].back;
      return back ? { ...state, step: back } : state;
    }
    case 'reset':
      return { step: 'basic', data: {} };
    case 'jump':
      return { ...state, step: action.payload.to };
  }
}

// 用法
function Wizard() {
  const [state, dispatch] = useReducer(wizardReducer, { step: 'basic', data: {} });
  // 根据 state.step 渲染对应表单页面
  // 用户点 Next → dispatch({ type: 'next', payload: {...} })
}

// 复杂状态机（> 10 状态 / 嵌套状态 / Side Effects）→ 用 XState（P7.1 推荐）
// 简单状态机（< 10 状态）→ useReducer 足够

场景 C · 主题 Context 性能优化

主题包含很多字段（colors / fonts / spacing / radii）。某些字段经常变（用户切换暗色），某些字段不变（spacing 系统）。怎么优化：

javascript · 主题 Context 拆分优化 
// ❌ 一个大 Context —— colors 变化让所有消费者重渲染
const ThemeContext = createContext({
  colors: {...},
  fonts: {...},
  spacing: {...},
  radii: {...},
});

// ✅ 按更新频率拆 2 个 Context
const ColorThemeContext = createContext(initialColors);     // 经常变（暗色切换）
const StaticThemeContext = createContext({                  // 不变（spacing / fonts / radii）
  fonts: {...},
  spacing: {...},
  radii: {...},
});

function App() {
  const [colors, setColors] = useState(initialColors);
  // staticTheme 是模块级常量 —— 完全不变 → 引用永远稳定

  return (
    <StaticThemeContext.Provider value={staticTheme}>
      <ColorThemeContext.Provider value={colors}>
        ...
      </ColorThemeContext.Provider>
    </StaticThemeContext.Provider>
  );
}

// 仅用 spacing / fonts 的组件 —— 永不重渲染（StaticThemeContext 引用永远稳定）
// 仅用 colors 的组件 —— 仅 colors 变化时重渲染
// 性能与"按字段订阅" 接近

// === useState 风格 ===
function TodoListWithUseState() {
  const [todos, setTodos] = useState([]);

  const add = (text) => setTodos(prev => [...prev, { id: Date.now(), text, done: false }]);
  const toggle = (id) => setTodos(prev => prev.map(t => t.id === id ? { ...t, done: !t.done } : t));

  return /* JSX */;

function todoReducer(state, action) {
  switch (action.type) {
    case 'add':
      return [...state, { id: Date.now(), text: action.text, done: false }];
    case 'toggle':

    case 'remove':
      return state.filter(t => t.id !== action.id);

用 useReducer 实现 undo / redo

javascript · undo/redo 状态机 
type State<T> = { past: T[]; present: T; future: T[] };
type Action<T> =
  | { type: 'set'; payload: T }
  | { type: 'undo' }
  | { type: 'redo' }
  | { type: 'reset'; payload: T };

function undoReducer<T>(state: State<T>, action: Action<T>): State<T> {
  switch (action.type) {
    case 'set':
      return {
        past: [...state.past, state.present],
        present: action.payload,
        future: [],   // 新操作清除 redo 历史
      };
    case 'undo':
      if (state.past.length === 0) return state;
      return {
        past: state.past.slice(0, -1),
        present: state.past[state.past.length - 1],
        future: [state.present, ...state.future],
      };
    case 'redo':
      if (state.future.length === 0) return state;
      return {
        past: [...state.past, state.present],
        present: state.future[0],
        future: state.future.slice(1),
      };
    case 'reset':
      return { past: [], present: action.payload, future: [] };
  }
}

// 自定义 Hook
function useUndoableState<T>(initial: T) {
  const [state, dispatch] = useReducer<Reducer<State<T>, Action<T>>>(
    undoReducer,
    { past: [], present: initial, future: [] }
  );

  return {
    state: state.present,
    set: (value: T) => dispatch({ type: 'set', payload: value }),
    undo: () => dispatch({ type: 'undo' }),
    redo: () => dispatch({ type: 'redo' }),
    canUndo: state.past.length > 0,
    canRedo: state.future.length > 0,
  };
}

// 用法
function TextEditor() {
  const editor = useUndoableState('');
  return (
    <>
      <textarea
        value={editor.state}
        onChange={(e) => editor.set(e.target.value)}
      />
      <button disabled={!editor.canUndo} onClick={editor.undo}>Undo</button>
      <button disabled={!editor.canRedo} onClick={editor.redo}>Redo</button>
    </>
  );
}

// 这就是 Excalidraw / Tldraw 等画板软件的 undo/redo 基础

05 常见陷阱

⚠️ 陷阱 1：reducer 中 mutate state（非纯函数）

case 'add': state.todos.push(newTodo); return state—— mutate 后返回同引用 → React Object.is 等 → bailout，UI 不更新。

规避：reducer 必须返回新引用。{...state, todos: [...state.todos, newTodo]}。Immer 缓解嵌套：produce(state, draft => { draft.todos.push(newTodo) })。

⚠️ 陷阱 2：reducer 中调副作用（非纯函数）

case 'submit': fetch('/api', {...}); return state—— reducer 必须是纯函数，不能调 fetch / 改 DOM / log（log 一般容忍但不推荐）。Concurrent rendering 可能让 reducer 跑多次。

规避：副作用放 useEffect 或 event handler。reducer 仅负责"给定 state + action 算新 state"。

⚠️ 陷阱 3：Context value 没 useMemo 让所有消费者重渲染

<Context.Provider value={{ user, setUser }}>—— 每次 App render 创建新对象 → 所有 useContext 消费者重渲染。

规避：① 用 useMemo 稳定 value ② 拆 Context（state + dispatch 分离 / 按更新频率分组）③ 用 useReducer 让 dispatch 引用永远稳定。

⚠️ 陷阱 4：在大 Context 上消费小字段

const { user } = useContext(BigContext)—— BigContext 任何字段变都重渲染。

规避：① 拆细 Context ② 用 use-context-selector 库 ③ 用 Zustand（专门解决这个）。详见 P7.1 生态。

⚠️ 陷阱 5：忘记给 Context 加 default value 或自定义 hook 包装

createContext()（无默认值）+ 组件忘记包 Provider → useContext 返回 undefined → 后续访问 .field 报 "cannot read of undefined"。

规避：① createContext 给合理默认值或 null + 自定义 hook 检查 ② 用 TypeScript 类型 createContext<User | null>(null)。
