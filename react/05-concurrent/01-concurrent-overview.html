P5.1 · Concurrent 与异步 ——「单线程怎么并发？」

P4.1 讲了 Fiber 架构是"可中断渲染"的物质基础。本章把这个能力翻译成用户可见的特性—— React 18 的 useTransition / useDeferredValue / 自动 batching、React 18+ 的 Suspense for data fetching、React 19 的 Server Components / Actions / use API。这些特性合在一起回答了一个问题：单线程的 JS 怎么处理并发—— 答案是 React 用 Fiber 把"渲染"切成时间片 + 把"异步等待"用 throw promise 模式集成进同步组件树。

📐 与 ECMA P5.1 异步演进的边界

ECMAScript · P5.1 异步演进 已讲清 JS 层的 5 代异步模型（callback → Promise → async/await → SharedArrayBuffer → using）。本章不重复这些，只讲 React 在 JS 异步基础上叠加的"UI 并发能力"。读完 ECMA P5.1 + 本章，就有了"JS 异步" + "React 并发渲染" 完整心智。

阅读时长 ~40 分钟 前置 P4.1 Fiber 架构（必读）+ ECMA P5.1 异步演进（可选但推荐） 对应 react.dev React 18 release · React 19 release

01 速览

三句话定义本章：

  1. React 的"单线程并发" 不是真多线程，是"把渲染拆成时间片"—— 每片 5ms，让出后浏览器响应交互、再继续。这是 P4.1 Fiber 架构能力的用户层表达。Web Worker 才是真多线程，但 React 不强制使用——所有 UI 渲染默认在主线程。
  2. Suspense 用"throw promise" 把异步等待集成进同步组件树。组件渲染时 throw 一个未 resolve 的 Promise → 最近的 Suspense boundary 捕获 → 显示 fallback → Promise resolve 后重试渲染。这是规范级 hack—— JS 没有"暂停渲染" 原语，React 用 throw / catch + Fiber 的 unwind 机制模拟。
  3. React 19 把"异步" 推到极致：use API 让组件 render 中直接读 Promise / Context（取代 useEffect 模式）；Actions 把"表单异步操作 + UI 状态" 打包；Server Components 让组件可以 async + await 直接 fetch（在服务器跑）。这是第二次范式扩展（第一次是 Hooks）。

5 代 React 异步能力对比：

代 React 版本 代表特性 核心抽象
第 1 代 0.x-15.x（2013-2017） componentDidMount + setState "类生命周期里 fetch + 手动 loading state"
第 2 代 16.6（2018） React.lazy + Suspense "throw promise 暂停渲染"——但仅代码分割
第 3 代 16.8（2019） useEffect + 自定义 Hooks "同步外部系统" 心智 + 第三方库（SWR / TanStack Query）
第 4 代 18.0（2022） useTransition / useDeferredValue / 自动 batching "把更新标为 transition" + 时间切片
第 5 代 19.0（2024） use / Actions / RSC "render 中读 Promise" + Server-first 架构

02 底层逻辑

Concurrent rendering 的核心机制

P4.1 已详述 Fiber 的 work loop + scheduler 包。本节从用户视角看"时间切片" 实际怎么发生：

javascript · 时间切片实测时序 
// 假设用户在搜索框输入 'react'，触发过滤 10000 项列表

// === 没有 Concurrent rendering（React 17 及以前） ===
// time 0:    用户输入 'r'
// time 0-1000ms:  React 同步渲染 10000 项 ── 主线程被占满
// time 1000ms:    用户输入 'e'（实际等了 1 秒）
// time 1000-2000ms: 又渲染 10000 项 ── 再卡 1 秒
// 结果：输入框完全卡顿

// === Concurrent rendering（React 18+） ===
// time 0:    用户输入 'r'
// time 0-5ms:  React 渲染部分 Fiber → shouldYield → 让出主线程
// time 5-16ms: 浏览器渲染屏幕（让用户看到输入框的 'r'）
// time 16-21ms: React 继续渲染下一段 Fiber → shouldYield
// time 21-32ms: 浏览器渲染（用户输入 'e'）
//              ⚡ 关键：用户输入 'e' 触发 SyncLane 高优先级
//              ⚡ React 丢弃当前 wip tree，从 'e' 重新开始
// time 32+ms: React 渲染基于 're' 的过滤结果
// 结果：输入框丝滑（每 5ms 让出一次），列表在用户停止输入后~100ms 出现

// 关键 API：useTransition 把"列表过滤"标为低优先级
const [isPending, startTransition] = useTransition();
const handleChange = (e) => {
  setQuery(e.target.value);  // SyncLane（输入框立即更新）
  startTransition(() => {
    setItems(filterLargeList(e.target.value));  // TransitionLane（可被打断）
  });
};

"时间切片" 的工程意义：

  • 不是"渲染更快"—— 总渲染时间不变（甚至略增加，因为有调度开销）。是"让出主线程" 让浏览器响应交互
  • 对小渲染无意义—— 渲染量小（< 5ms）就用同步模式，时间切片帮不上忙
  • 对超大渲染（10000+ 项）才显著—— 这是 useTransition / useDeferredValue 真正的用武之地

Suspense 与 Promise 集成 —— "Throw a Promise" 模式

Suspense 是 React 16.6 引入的语法，但语义在 React 18 才完整。它解决一个经典问题："怎么在同步组件树里表达异步等待"。

javascript · Suspense 用户视角 
function App() {
  return (
    <Suspense fallback={<Spinner />}>
      <UserProfile userId={42} />
    </Suspense>
  );
}

function UserProfile({ userId }) {
  const user = use(fetchUser(userId));  // ← 异步！返回 Promise，但 use 让它"看起来同步"
  return <div>{user.name}</div>;
}

// 用户视角：
// - fetchUser 还没 resolve → 显示 <Spinner />
// - resolve 后 → 显示 <div>{user.name}</div>
// - 期间 UserProfile 函数体看起来是同步代码

// 内部机制：
// 1. UserProfile 第一次 render 时 use(promise) 检查 promise 状态
// 2. promise 还 pending → use 直接 throw promise
// 3. React 沿 Fiber tree unwind 到最近的 Suspense boundary
// 4. Suspense 接住 throw，渲染 fallback
// 5. Suspense 注册 promise.then 回调，resolve 后重新渲染 children
// 6. 重新渲染 UserProfile → use(promise) 看到 fulfilled → 返回值

"throw promise" 模式的妙处：

  • 组件函数体看起来同步—— 不需要 if (loading) return ... 样板代码
  • 异步边界由调用方决定—— 不是组件内部决定显示什么 fallback，是 Suspense boundary（更灵活）
  • 多个组件共享 fallback—— 一个 Suspense 包住多个异步组件，所有都 ready 才一起显示
  • 规范级 hack—— JS 没有"挂起函数执行" 原语（除 generator），React 用 throw / catch + Fiber unwind 模拟

触发 Suspense fallback 的边界：从 throw promise 的 Fiber 沿 return 指针上溯，找到最近的 Suspense boundary。配合 Fiber tree 嵌套：

javascript · Suspense 边界冒泡 
<Suspense fallback={<PageSpinner />}>
  <Header />
  <Suspense fallback={<ListSpinner />}>
    <UserList />          {/* throw promise 时只触发 ListSpinner */}
  </Suspense>
  <Footer />
</Suspense>

// UserList 异步加载时：
// - throw promise → 沿 Fiber tree 找最近 Suspense
// - 找到内层 <Suspense fallback={<ListSpinner />}> → 显示 ListSpinner
// - Header 和 Footer 已经渲染，不受影响
//
// 如果 Header 也 throw（如它内部也有 use）：
// - 沿 Fiber tree 找最近 Suspense → 找到外层 Suspense
// - 显示 PageSpinner（整页 loading）
//
// 这就是"Suspense 嵌套粒度" 的工程意义：
// 把 fallback 放在合适层级，控制 loading 显示范围

Transition / Deferred Value / 自动 batching

useTransition

useTransition 把包在 startTransition 内的更新标为 TransitionLane（低优先级、可被打断）。它的 API:

javascript · useTransition 完整用法 
const [isPending, startTransition] = useTransition();

// isPending：transition 是否正在进行中（用于显示进度指示器）
// startTransition：把 setState 包起来标为低优先级

const handleChange = (e) => {
  setInputValue(e.target.value);   // SyncLane —— 输入框立即响应

  startTransition(() => {
    // 这里的 setState 都是 TransitionLane
    setSearchQuery(e.target.value);
    setFilteredList(filter(largeList, e.target.value));
  });
};

// 渲染时显示 isPending：
return (
  <>
    <input value={inputValue} onChange={handleChange} />
    {isPending ? <Spinner /> : null}
    <FilteredList items={filteredList} />
  </>
);

// React 19 增强：startTransition 接受 async 函数
startTransition(async () => {
  await fetch(...);    // 整个 async 链都在 transition 中
  setData(result);
});

useDeferredValue

和 useTransition 表达同样意图但 API 不同——"把这个值的更新推迟到 transition"：

javascript · useDeferredValue 
function SearchableList({ query }) {
  // query 直接用：每次输入都立即响应
  // deferredQuery：会延迟到主线程空闲时才更新（标为 DeferredLane）
  const deferredQuery = useDeferredValue(query);

  // 用 deferredQuery 而非 query 做昂贵计算
  const filtered = useMemo(
    () => filterLargeList(deferredQuery),
    [deferredQuery]
  );

  // 当 query !== deferredQuery 时显示"正在更新" 提示
  const isStale = query !== deferredQuery;

  return (
    <ul style={{ opacity: isStale ? 0.5 : 1 }}>
      {filtered.map(item => <li key={item.id}>{item.name}</li>)}
    </ul>
  );
}

// useDeferredValue vs useTransition：
// - useTransition：你主动把 setState 标为 transition（你能控制）
// - useDeferredValue：你接收一个值，框架决定何时把它推迟
// - 用法：父组件传 query 给你（你不控制 setQuery），用 useDeferredValue 推迟

自动 batching

React 18 的另一个改进：自动合并多个 setState 到一次渲染。React 17 仅在 React 事件回调里 batching，Promise / setTimeout / 原生事件回调里的多个 setState 各自触发独立渲染。React 18 改成所有上下文都 batch：

javascript · 自动 batching 行为变化 
// 一个回调内的 3 个 setState
function handleClick() {
  setCount(c => c + 1);
  setName('Alice');
  setItems([1, 2, 3]);
}
// React 17 / 18 都 batch —— 1 次渲染（行为一致）

// Promise.then 内的 3 个 setState
fetch('/api').then(() => {
  setCount(c => c + 1);
  setName('Alice');
  setItems([1, 2, 3]);
});
// React 17：不 batch —— 3 次渲染（性能差）
// React 18：自动 batch —— 1 次渲染 ✅

// setTimeout / 原生事件回调同理
setTimeout(() => {
  setCount(c => c + 1);
  setName('Alice');
}, 100);
// React 17：不 batch
// React 18：自动 batch ✅

// 强制不 batch（极少用）：
import { flushSync } from 'react-dom';
flushSync(() => setCount(c => c + 1));   // 强制立即同步渲染

Server Components 异步边界

Server Components（RSC）是 React 19 的架构层转变—— 组件分两种：永远在服务器跑（默认）、永远在客户端跑（标记 'use client'）。详细心智模型见 P6.1 Server Components（编写中），本节聚焦"异步边界" 这一面：

javascript · RSC 异步组件 
// app/page.tsx —— 默认是 Server Component
async function Page() {
  // ✅ 直接 await —— 在服务器跑，完成后才返回 HTML
  const user = await db.users.findById(42);
  const posts = await db.posts.findByUser(user.id);

  return (
    <>
      <h1>{user.name}</h1>
      <Suspense fallback={<PostsSkeleton />}>
        <Posts data={posts} />       {/* 这里也可以是 async Server Component */}
      </Suspense>
      <ClientWidget />                {/* 客户端组件 */}
    </>
  );
}

// 服务器流式渲染（streaming SSR）：
// - 服务器先渲染同步部分（h1）+ 发送给浏览器
// - 异步部分（Posts）渲染完成 → 通过流补发到浏览器
// - 浏览器持续接收 chunks，逐步替换 fallback
// - 用户看到：h1 → PostsSkeleton → 真实 Posts，分阶段加载

RSC 的异步特性：

  • Server Components 可以 async —— 不像 Client Components（'use client'）只能用 useEffect / use API
  • props 必须可序列化 —— 跨 'use client' 边界传递的 props 走 RSC 协议（类似 JSON.stringify 但支持更多类型如 Date、Map），不能传函数 / Symbol / class instance
  • Server Action：'use server' 标记的函数可以从 Client Component 调用——RPC 风格的服务器调用
  • RSC + Suspense 配合流式渲染—— 服务器把已 ready 的部分先发，未 ready 的用 fallback 占位，resolve 后追加

"能 / 不能做什么" 速查：

能力 Server Component（默认） Client Component（'use client'）
useState / useEffect ❌ 不能 ✅
事件处理（onClick 等） ❌ ✅
直接 await + fetch / 数据库 ✅ ❌（要用 useEffect / use API）
读 process.env / 读文件 ✅ ❌（不暴露给浏览器）
使用浏览器 API（window / document） ❌ ✅
嵌入子组件 可以嵌 Client Component（通过 children 传） 不能直接嵌 Server Component（要 props 传 ReactNode）

Actions —— 把"异步操作 + UI 状态"打包

React 19 引入 Actions 解决一个 Hooks 时代痛点：每个表单都要手写 useState（loading / error / success）+ try/catch + dispatch。Actions 把这些样板代码打包进 3 个 hooks：

javascript · Actions vs 传统 Hooks 写法 
// === 传统 Hooks 写法（React 18） ===
function CreatePostForm() {
  const [title, setTitle] = useState('');
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState(null);

  const handleSubmit = async (e) => {
    e.preventDefault();
    setIsLoading(true);
    setError(null);
    try {
      await fetch('/api/posts', {
        method: 'POST',
        body: JSON.stringify({ title }),
      });
      setTitle('');
    } catch (err) {
      setError(err.message);
    } finally {
      setIsLoading(false);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={title} onChange={e => setTitle(e.target.value)} />
      <button type="submit" disabled={isLoading}>
        {isLoading ? 'Posting...' : 'Post'}
      </button>
      {error && <p>{error}</p>}
    </form>
  );
}

// === React 19 Actions 写法 ===
function CreatePostForm() {
  const [error, submitAction, isPending] = useActionState(
    async (prevError, formData) => {
      try {
        await fetch('/api/posts', {
          method: 'POST',
          body: JSON.stringify({ title: formData.get('title') }),
        });
        return null;  // 成功
      } catch (err) {
        return err.message;  // 错误返回值
      }
    },
    null,
  );

  return (
    <form action={submitAction}>
      <input name="title" />
      <SubmitButton />       {/* 子组件用 useFormStatus 感知 isPending */}
      {error && <p>{error}</p>}
    </form>
  );
}

function SubmitButton() {
  const { pending } = useFormStatus();  // 自动从最近的 form 取 pending 状态
  return (
    <button type="submit" disabled={pending}>
      {pending ? 'Posting...' : 'Post'}
    </button>
  );
}

// 减少了：useState（loading / error）、try/catch 模板、disabled 透传
// 新增了：自动管理 pending 状态、formData 集成、action 可在 RSC + Client 通用

3 个核心 Actions hooks：

  • useActionState(action, initialState)：替代 useState + try/catch + isLoading 三件套
  • useFormStatus()：子组件自动感知最近 form 的 pending / data / method 状态（不需要 prop drilling）
  • useOptimistic(state, updateFn)：乐观更新——立即显示新状态，async 操作失败时自动回滚

use API（React 19）

use 是 React 19 引入的"读异步资源" 统一 API。可以读 Promise（搭配 Suspense）/ 读 Context（替代 useContext 的灵活版）：

javascript · use API 双重用法 
// 用法 1：读 Promise（替代 useEffect + useState 模式）
function UserProfile({ userPromise }) {
  const user = use(userPromise);  // 等价于 await，但在 render 中
  return <div>{user.name}</div>;
}

<Suspense fallback={<Spinner />}>
  <UserProfile userPromise={fetchUser(42)} />
</Suspense>

// 用法 2：读 Context（替代 useContext 的"必须顶层调用"约束）
const ThemeContext = createContext('light');

function Component({ shouldRenderThemed }) {
  if (shouldRenderThemed) {
    const theme = use(ThemeContext);  // ⚡ 在条件分支里调用！useContext 不能这样
    return <div className={theme}>...</div>;
  }
  return null;
}

// 关键差异：
// - useContext：必须在组件顶层调用（Rules of Hooks）
// - use：可以在条件 / 循环 / 函数嵌套里调用（"不是 Hook"）

// 为什么 use 不受 Rules of Hooks 限制？
// 因为它不依赖 hooks linked list 的位置对齐 —— 它是"新机制"
// 详见 P6.3 use API（编写中）

5 个高密度洞察

① "单线程并发" 不是真多线程

React 的所有 Concurrent 特性都在主线程的同一个 Agent 里跑（P4.1 / ECMA · P4.1 已讲）。"并发" 指的是"把渲染拆分让出主线程"，不是"多个 CPU 核心同时算"。要真多线程要用 Web Worker（React 不强制使用）。

② Suspense 的 "throw promise" 是规范级 hack

JS 没有"挂起函数" 原语（除 generator）。React 用 throw 异常 + Fiber unwind 模拟"组件渲染挂起"。这是"用现有语言机制做新事" 的极佳案例—— 不需要 JS 改规范。代价：throw promise 不能被普通 try/catch 捕获（Suspense 必须特殊处理），异常调试 stack trace 不直观。

③ Server Components 是 React 19 最大的认知挑战

UI 不再是"客户端状态 → 视图"，而是"服务器状态 + 客户端状态 → 视图" 两层模型。'use client' 边界不直观：把组件标 client 太多 → 失去 RSC 优势；标太少 → 客户端交互失效。需要重新建立心智，详见 P6.1。

④ Actions 把异步状态机内化到框架

React 18 之前每个表单都自己写一套"loading / error / pending / submitting" useState。React 19 Actions 把这套提升为框架原语—— 减少样板代码 + 与 Server Actions 无缝集成。这是 React 团队"看到全行业都在重复造轮子" 后的统一答案。

⑤ use 不是 Hook

虽然名字像，但 use 不受 Rules of Hooks 约束。可以在条件 / 循环 / 函数嵌套里调用。技术上它是新机制—— 直接对接 React 19 的 Suspense + Context 系统，不依赖 P4.1 hooks linked list 的位置对齐。这点很容易踩坑—— 见陷阱部分。

03 应用场景

场景 A · 大列表渲染丝滑

搜索框过滤 50000 项地址簿。不优化的话每次输入都卡 800ms。用 useDeferredValue（适合"接受 prop 推迟"场景）或 useTransition（适合"主动 setState 推迟"场景）：

javascript · 大列表 + useDeferredValue 
function ContactList({ allContacts }) {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);

  const filtered = useMemo(
    () => allContacts.filter(c => c.name.includes(deferredQuery)),
    [allContacts, deferredQuery],
  );

  const isStale = query !== deferredQuery;

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <ul style={{ opacity: isStale ? 0.5 : 1 }}>
        {filtered.map(c => <li key={c.id}>{c.name}</li>)}
      </ul>
    </>
  );
}

// 效果：输入框立即响应（query 立即更新），列表平滑过滤（deferredQuery 推迟）

场景 B · 数据 fetching 取代 useEffect 模式

React 19 之前，数据加载用 useEffect + useState + 第三方库（SWR / TanStack Query）。React 19 + RSC 让这变得简单：

javascript · 18 vs 19 数据加载对比 
// === React 18 + TanStack Query ===
'use client';
function UserProfile({ userId }) {
  const { data, isLoading, error } = useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetch(`/api/users/${userId}`).then(r => r.json()),
  });
  if (isLoading) return <Spinner />;
  if (error) return <ErrorMsg error={error} />;
  return <div>{data.name}</div>;
}

// === React 19 + RSC（Next.js App Router）===
async function UserProfile({ userId }) {
  const user = await fetch(`/api/users/${userId}`).then(r => r.json());
  return <div>{user.name}</div>;
}
<Suspense fallback={<Spinner />}>
  <UserProfile userId={42} />
</Suspense>

// 客户端组件 + use API：
'use client';
function UserProfile({ userPromise }) {
  const user = use(userPromise);
  return <div>{user.name}</div>;
}

// 优势：减少了 isLoading 状态、错误用 Error Boundary 统一处理、
//       与 Suspense 自然组合
// 代价：要用 Next.js / Waku 等支持 RSC 的元框架

场景 C · 表单异步提交 + 乐观更新

点赞按钮：点击后立即显示"已点赞"，async API 失败时自动回滚。用 useOptimistic：

javascript · useOptimistic 
function LikeButton({ post, currentUserId }) {
  const [optimisticLikes, setOptimisticLikes] = useOptimistic(
    post.likes,
    (currentLikes, action) => {
      if (action === 'like') return [...currentLikes, currentUserId];
      if (action === 'unlike') return currentLikes.filter(id => id !== currentUserId);
    },
  );

  const isLiked = optimisticLikes.includes(currentUserId);

  const handleLike = async () => {
    setOptimisticLikes(isLiked ? 'unlike' : 'like');  // 立即更新 UI
    try {
      await fetch(`/api/posts/${post.id}/${isLiked ? 'unlike' : 'like'}`, {
        method: 'POST',
      });
      // 成功 —— 乐观更新已经显示，不需要再做什么
    } catch (err) {
      // 失败 —— React 自动回滚 optimisticLikes 到原值（post.likes）
      alert('Failed: ' + err.message);
    }
  };

  return (
    <button onClick={handleLike}>
      {isLiked ? '❤️' : '🤍'} {optimisticLikes.length}
    </button>
  );
}

// useOptimistic 自动管理：① 立即应用乐观更新 ② async 完成后回归 actual state ③ 失败时回滚

04 典型代码

实测时间切片：用 useTransition 优化大列表

javascript · 完整 useTransition 案例 
import { useState, useTransition, useMemo } from 'react';

const allItems = Array.from({ length: 50000 }, (_, i) => ({
  id: i,
  text: `Item ${i}`,
}));

function SearchableList() {
  const [input, setInput] = useState('');         // 输入框值（高优先级）
  const [query, setQuery] = useState('');          // 实际过滤的 query（低优先级）
  const [isPending, startTransition] = useTransition();

  const filtered = useMemo(
    () => allItems.filter(item => item.text.includes(query)),
    [query],
  );

  const handleChange = (e) => {
    const next = e.target.value;
    setInput(next);   // SyncLane —— 立即响应
    startTransition(() => {
      setQuery(next);  // TransitionLane —— 可被打断
    });
  };

  return (
    <>
      <input value={input} onChange={handleChange} />
      {isPending && <span> (updating...)</span>}
      <ul>
        {filtered.slice(0, 100).map(item => (   // 只渲染前 100 项可见的
          <li key={item.id}>{item.text}</li>
        ))}
      </ul>
    </>
  );
}

// 测试方法：在 input 里快速输入 'item123'
// 没有 useTransition：每个字符都卡顿明显
// 有 useTransition：输入丝滑，列表稍后更新

javascript · 嵌套 Suspense 设计 
// ❌ 错的：粒度太粗 —— 整页 spinner
function App() {

  // 用户看到整页 loading 几秒

}

05 常见陷阱

⚠️ 陷阱 1：useTransition 不是"异步函数"

startTransition(() => setState(...)) 仍是同步执行，只是把 setState 标为 transition lane。它不会让你的代码异步执行。

javascript · 常见误解 
// ❌ 误解 —— startTransition 不延迟代码
startTransition(() => {
  console.log('1');     // 同步执行
  setState(...);
  console.log('2');     // 同步执行
});
console.log('3');        // 同步执行
// 输出：1 → 2 → 3（顺序固定）

// React 18：startTransition 不支持 async 函数 —— async 函数里 setState 不会被标记
// React 19：startTransition 支持 async 函数 —— 整个 async 链都是 transition

规避：理解 startTransition 是"给 setState 标 lane"，不是"把代码异步化"。要异步执行用 setTimeout / Promise / async function。React 19 的 async startTransition 可用但 React 18 不支持。

⚠️ 陷阱 2：Suspense fallback 频繁闪烁

Suspense 边界放太低（如每个 list item 一个 Suspense）—— 每个 promise 都触发 fallback 闪烁，用户体感"跳来跳去"。

规避：① 按"逻辑组件块" 分边界（一个 card / 一个 sidebar），不是每个数据点 ② 用 React 19 的 Activity（之前叫 Offscreen）保留隐藏内容的状态 ③ 用 useDeferredValue 让数据"慢慢更新" 而不是闪 spinner

⚠️ 陷阱 3：在 Server Component 里用 useState / useEffect

Server Components 不能用任何 Hook（除部分 Server-only 的如 cache）。parse 时报错，不是运行时。

javascript · 错误示例 
// app/page.tsx —— 默认 Server Component
async function Page() {
  const [count, setCount] = useState(0);  // ❌ 编译错：useState in Server Component
  useEffect(() => { ... }, []);            // ❌ 同上
  return <div>...</div>;
}

// 解决：标 'use client' 让组件变 Client Component
'use client';
function Page() {
  const [count, setCount] = useState(0);   // ✅ 现在 OK 了
  return <div>...</div>;
}

// 或者：保持 Server Component，把交互部分提取到 Client 子组件
// app/page.tsx —— Server Component
async function Page() {
  const data = await fetchData();
  return (
    <div>
      <h1>{data.title}</h1>
      <ClientCounter initial={data.initial} />
    </div>
  );
}
// app/client-counter.tsx
'use client';
function ClientCounter({ initial }) {
  const [count, setCount] = useState(initial);  // ✅
  return <button onClick={...}>{count}</button>;
}

规避：理解 Server / Client 边界。不要"整个组件树都标 'use client'"——失去 RSC 价值。从根开始用 Server Components，只在需要交互的叶子加 'use client'。

⚠️ 陷阱 4：use API 在条件分支里读 Promise 引发死循环

use(promise) 可以在条件分支里调用，但每次 render 必须传同一个 promise 引用—— 不然 React 看到不同 promise 就重新等待，死循环。

javascript · use 死循环 
// ❌ 每次 render 都创建新 promise —— 死循环
function UserProfile({ userId }) {
  const user = use(fetch(`/api/users/${userId}`).then(r => r.json()));
  // 每次 render 调用 fetch（创建新 Promise）→ use 等待 → resolve → 重新 render
  // → 又创建新 Promise → 又等待 → 永不停止
  return <div>{user.name}</div>;
}

// ✅ Promise 在 parent 创建（引用稳定）
function App() {
  const userPromise = useMemo(() => fetchUser(42), []);  // 只创建一次
  return (
    <Suspense fallback={<Spinner />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  );
}
function UserProfile({ userPromise }) {
  const user = use(userPromise);   // 同一引用 → 无限循环避免
  return <div>{user.name}</div>;
}

// RSC 中是另一回事：Server Component 是"跑一次"，没有重复 render 问题

规避：在 Client Component 用 use(promise) 时，promise 必须在更高层（parent / RSC）创建，引用稳定地传下来。不要在组件函数体里 fetch(...)。

⚠️ 陷阱 5：跨 'use client' 边界传不可序列化的 props

Server Component 给 Client Component 传 props 时，props 必须可序列化—— 走 RSC 协议（类似 JSON 但支持更多类型）。函数 / class instance / Symbol 不能传。

javascript · 不可序列化 props 
// app/page.tsx (Server Component)
async function Page() {
  const data = await fetchData();
  const handleClick = () => { ... };  // ❌ 函数不能传过边界

  return <ClientButton data={data} onClick={handleClick} />;  // ⚠️ 编译错
}

// 解决 1：children 形式传 Server Component（不是 props）
function Page() {
  return (
    <ClientWrapper>
      <ServerData />     {/* 这是合法的 —— children 是 ReactNode */}
    </ClientWrapper>
  );
}

// 解决 2：Server Action（标 'use server' 的函数可以传过边界）
async function Page() {
  async function handleAction() { 'use server'; ... }
  return <ClientButton onAction={handleAction} />;  // ✅
}

// 解决 3：interaction 完全在 Client 内部
'use client';
function ClientButton({ data }) {
  const handleClick = () => { ... };   // 函数在客户端创建
  return <button onClick={handleClick}>{data.title}</button>;
}

规避：跨边界 props 限制为 JSON-like + Date / Map / Set / ReactNode。函数用 Server Actions（'use server'）或在 Client 内部创建。
