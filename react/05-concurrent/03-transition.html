P5.3 · useTransition / useDeferredValue

React 18 引入 useTransition 和 useDeferredValue 两个 Hook—— 把更新标为"低优先级 transition"。这不是 debounce / throttle（不是延迟），不是异步（代码同步执行）—— 是给 React 调度系统提示"这部分更新可以被打断 / 在用户输入这种高优更新前让位"。本章讲清两个 Hook 的精确语义、与 debounce 的差异、与 Suspense 配合，以及 React 19 引入的 async transition 改进。

⚠️ "transition 是异步函数"是最常见误解

startTransition(() => setState(...)) 仍是同步执行—— 只是把 setState 标为 TransitionLane（低优先级、可被打断）。它不会让你的代码异步执行。debounce / throttle 才是延迟。transition 是"调度优先级"，不是"延迟"。

阅读时长 ~25 分钟 前置 P5.1 Concurrent + P4.1 Lane 模型 对应 react.dev useTransition · useDeferredValue

01 速览

三句话定义本章：

  1. useTransition 返回 [isPending, startTransition]。startTransition(callback) 把 callback 内的 setState 标记为 TransitionLane（P4.1 Lane 模型）—— 低优先级、可被高优更新打断。isPending 表示 transition 是否在进行中（用于显示进度提示）。
  2. useDeferredValue(value) 返回 value 的"延后版本"。React 在主线程空闲时才更新 deferred 值。适合"父组件传值，子组件想推迟用" 的场景—— 与 useTransition 是"主动 vs 被动" 关系。
  3. 3 个核心区别要记牢：① 不是异步（代码同步跑） ② 不是 debounce / throttle（不是延迟，是优先级） ③ 不是性能优化银弹（仍要昂贵渲染才有意义；小渲染加 transition 是负优化）。

看一段最小代码识别两个 Hook 用法差异：

javascript · useTransition vs useDeferredValue 
// useTransition：主动把 setState 标低优先级（你拥有触发权）
function SearchActive() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    setQuery(e.target.value);   // SyncLane（输入框立即响应）
    startTransition(() => {
      setResults(filterLargeList(e.target.value));   // TransitionLane（可被打断）
    });
  };

  return (
    <>
      <input value={query} onChange={handleChange} />
      {isPending && <span> (filtering...)</span>}
      <ResultsList items={results} />
    </>
  );
}

// useDeferredValue：被动接收 value 推迟（你不拥有触发权）
function SearchPassive() {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);  // 推迟版本

  const filtered = useMemo(
    () => filterLargeList(deferredQuery),
    [deferredQuery],
  );

  const isStale = query !== deferredQuery;
  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      {isStale && <span> (filtering...)</span>}
      <ResultsList items={filtered} />
    </>
  );
}

// 关键差异：
// - useTransition：你主动把 setState 标为 transition（你能控制何时触发）
// - useDeferredValue：你接收一个值，框架决定何时推迟（你不控制 setQuery）
// - 用法决定了选哪个：
//   - 自己拥有 setState → useTransition
//   - 接收 props 想推迟 → useDeferredValue

02 底层逻辑

useTransition 完整签名

typescript · useTransition 类型 
function useTransition(): [
  isPending: boolean,
  startTransition: (scope: () => void) => void,
];

// React 19+ 增强：startTransition 接受 async 函数
function useTransition(): [
  isPending: boolean,
  startTransition: (scope: () => void | Promise<void>) => void,
];

// 关键事实：
// 1. startTransition 引用永远稳定（同 setState dispatch，P3.1）
// 2. isPending 是 boolean —— transition 是否在进行
// 3. scope 内的 setState 全部标为 TransitionLane
// 4. scope 必须是同步函数（18）或 async 函数（19+）
// 5. 多次 startTransition 调用会合并到同一 transition

useDeferredValue 完整签名

typescript · useDeferredValue 类型 
function useDeferredValue<T>(value: T, initialValue?: T): T;

// 用法：
const deferredQuery = useDeferredValue(query);

// 行为：
// 1. 第一次 render：返回 initialValue（如果传了）或 value
// 2. value 变化时：先返回旧 value（保持），再在低优先级 render 中切到新 value
// 3. 之后稳定后：返回最新 value
//
// 这就是 React 18 SSR + useDeferredValue 的初始值优化：
//    服务器端用 initialValue 渲染（避免水合不匹配）
//    客户端 hydrate 后切到真实 value

// 何时该用：
// - 接收 prop / context 想推迟使用
// - 不想暴露 startTransition 给调用方

transition 不是异步（核心澄清）⭐

javascript · transition 是同步代码 + 调度提示 
// ❌ 误解：startTransition 让代码异步
startTransition(() => {
  console.log('1');     // 同步执行
  setState(...);        // 同步加 update 到 queue（标 TransitionLane）
  console.log('2');     // 同步执行
});
console.log('3');        // 同步执行
// 输出：1 → 2 → 3（顺序固定，全部同步）

// ✅ 正确认知：startTransition 不延迟代码，只是调度提示
// 它让 setState 进 TransitionLane → 渲染时低优先级 → 可被打断

// React 18：scope 必须是同步函数
startTransition(() => {
  setCount(c => c + 1);   // ✅ OK
  setName('Alice');        // ✅ OK
});

// ❌ React 18 不支持 async scope
startTransition(async () => {
  await fetch('/api');      // ⚠️ await 之后的 setState 不在 transition 中
  setData(result);          // 这是普通 SyncLane 更新
});

// ✅ React 19：支持 async scope
startTransition(async () => {
  const result = await fetch('/api');
  setData(result);          // ✅ 19+ 在 transition 中
  setProgress(100);         // ✅ 同上
});

// 19+ 的 async transition 让"异步操作 + transition" 一体化
// 是 Actions（P6.2）的底层基础

Lane 调度（短重述链回 P4.1）

短重述：P4.1 详述 Fiber 的 Lane 模型——31 位 bitmask 表达优先级。useTransition 把 setState 标为 TransitionLane1-16（低优先级），useDeferredValue 内部用 DeferredLane。本节回顾 transition 在 Lane 模型中的位置：

javascript · Lane 优先级层级 
// React 18+ Lane 优先级（高 → 低）：
// SyncLane (1)            ← 用户输入 / 紧急更新
// InputContinuousLane     ← 鼠标拖拽 / 输入连续
// DefaultLane             ← 默认（Promise.then / setTimeout 触发的 setState）
// TransitionLane1-16      ← startTransition 内的 setState
// RetryLanes              ← Suspense 重试
// IdleLane                ← 空闲时
// OffscreenLane           ← Offscreen 内容（19 Activity）

// 关键调度规则：
// 1. 高优先级 Lane 打断低优先级 → React 丢弃低优 wip tree 重新算
// 2. 同 Lane 多个更新 batch（合并到一次 render）
// 3. transition 在 5ms 时间片内跑，超时让出主线程
// 4. 高优更新（如用户输入）触发 → React 暂停当前 transition、先处理高优、再恢复 transition

// 实战例子：搜索框过滤 50000 项
// - 用户输入 'r' → setQuery 在 SyncLane → 立即更新 input 显示
// - startTransition 内 setItems → TransitionLane → 开始渲染过滤结果
// - 用户立刻输入 'e' → setQuery SyncLane → 打断当前 TransitionLane 渲染
// - 丢弃 'r' 的过滤 wip tree，重新基于 're' 渲染
// - 用户停手后 transition 完成 → 显示 're' 的结果

// 这就是"输入丝滑 + 列表稍后过滤" 的原理

与 debounce / throttle 对比

维度 debounce throttle useTransition
本质 延迟（用户停止 N ms 后才执行） 限频（每 N ms 最多一次） 调度优先级提示
会让代码"不立即执行" ✅ 是 ✅ 是 ❌ 否（同步）
用户立即输入响应 需配合非 debounce 路径 同上 ✅ 自动（SyncLane 优先）
渲染中可被打断 ❌ ❌ ✅
异步 fetch 场景 常用（避免每键打 API） 偶用 不直接解决（fetch 本身仍要 debounce）

关键认知：

  • "避免每次输入都打 API" 用 debounce（限制 fetch 频率）
  • "避免大列表渲染卡顿" 用 useTransition（让 React 调度让位用户输入）
  • "同时两者都需要" 配合用：debounce 控 fetch + transition 控渲染
javascript · debounce + transition 配合 
function SearchWithBoth() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const debouncedQuery = useDebounce(query, 300);
  const [isPending, startTransition] = useTransition();

  // 1. debouncedQuery 变化（用户停手 300ms 后）→ fetch
  useEffect(() => {
    if (!debouncedQuery) return;
    const ctrl = new AbortController();
    fetch(`/api/search?q=${debouncedQuery}`, { signal: ctrl.signal })
      .then(r => r.json())
      .then(data => {
        // 2. 用 transition 渲染结果（避免大列表卡顿）
        startTransition(() => setResults(data));
      });
    return () => ctrl.abort();
  }, [debouncedQuery]);

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {isPending && <Spinner />}
      <ResultsList items={results} />
    </>
  );
}

// 协作分工：
// - debounce：避免每键打 API（节省后端资源）
// - transition：避免大结果渲染卡输入（提升 UX）
// 两者解决不同问题，可同时用

与 Suspense 配合（P5.2 提到）

javascript · transition 防 Suspense fallback 闪烁 
// 痛点：tab 切换时已显示内容被 fallback 替换 → 闪烁

// ❌ 不用 transition
function App() {
  const [tab, setTab] = useState('home');
  return (
    <>
      <Tabs activeTab={tab} onChange={setTab} />
      <Suspense fallback={<Spinner />}>
        {tab === 'home' && <Home />}
        {tab === 'profile' && <Profile />}
      </Suspense>
    </>
  );
}
// 切 home → profile：Profile throw promise → Suspense 显示 Spinner
// 用户看到"home → spinner → profile" 三阶段闪烁

// ✅ 用 transition
function App() {
  const [tab, setTab] = useState('home');
  const [isPending, startTransition] = useTransition();

  const handleTabChange = (next) => {
    startTransition(() => setTab(next));
  };

  return (
    <>
      <Tabs activeTab={tab} onChange={handleTabChange} />
      {isPending && <ProgressBar />}
      <Suspense fallback={<Spinner />}>
        {tab === 'home' && <Home />}
        {tab === 'profile' && <Profile />}
      </Suspense>
    </>
  );
}
// 切 home → profile：tab 在 transition lane（低优先级）
// React 在 Profile ready 之前保留 Home 显示
// 用户看到"home（仍显示）+ ProgressBar → profile（替换）"
// fallback 不闪 —— 因为有旧内容可保留

React 19 改进：async transition + 顶层 startTransition

javascript · React 19 改进 
// 1. async transition（P5.3 已演示）
// React 18：scope 必须同步
// React 19：scope 可以是 async 函数 → 整个 async 链都是 transition
startTransition(async () => {
  await fetch('/api');
  setData(result);          // 19+ 仍在 transition
  setLoading(false);
});

// 2. 顶层 React.startTransition（不需要 useTransition）
import { startTransition } from 'react';

function handleSubmit() {
  // 不需要在组件内 —— 任何位置都能调
  startTransition(() => {
    setData(...);
  });
}

// useTransition vs startTransition 顶层 API：
// - useTransition：组件内用 + 提供 isPending 状态
// - startTransition：任何位置用 + 不提供 isPending（你自己管理 loading state）

// React 19 还引入 useActionState（P6.2）—— 提供 isPending + 错误处理
// 这是 transition 在表单场景的"专用版"，更易用

useTransition 不是性能优化银弹

⚠️ transition 仅对昂贵渲染有意义

transition 让 React 在 5ms 时间片内跑、可被打断。对小渲染（< 5ms）没意义—— 没有可让位的时间。对大渲染（> 100ms）效果明显—— 用户能感知"输入不卡了"。

判断方法：用 React DevTools Profiler 测渲染时间。< 16ms 不需要 transition；> 100ms 必须用 transition。

03 应用场景

场景 A · 50000 项地址簿实时过滤

通讯录有 5 万联系人，搜索框实时过滤。普通 setState 让输入卡顿（每键 ~500ms 渲染）。useTransition 让输入丝滑：

javascript · 大列表过滤完整实现 
function ContactSearch({ allContacts }) {
  const [query, setQuery] = useState('');
  const [filtered, setFiltered] = useState(allContacts);
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    const next = e.target.value;
    setQuery(next);   // SyncLane —— 输入框立即更新

    startTransition(() => {
      const filteredNext = next
        ? allContacts.filter(c => c.name.toLowerCase().includes(next.toLowerCase()))
        : allContacts;
      setFiltered(filteredNext);
    });
  };

  return (
    <>
      <input value={query} onChange={handleChange} placeholder="Search..." />
      {isPending && <span style={{ opacity: 0.6 }}> filtering...</span>}
      <ContactList contacts={filtered} />
    </>
  );
}

// 实测效果（DevTools Profiler）：
// 不用 transition：每个键导致 ~500ms 渲染卡顿，输入"react" 用户感觉卡 2.5s
// 用 transition：输入立即响应（< 5ms），列表 100ms 后更新

场景 B · Tab 切换不闪 fallback

管理后台有 5 个 tab，每个 tab 是 lazy chunk。普通切换闪 spinner。useTransition 保留旧 tab 显示直到新 tab ready：

javascript · Tab 不闪 fallback 完整实现 
const Dashboard = lazy(() => import('./Dashboard'));
const Reports = lazy(() => import('./Reports'));
const Settings = lazy(() => import('./Settings'));

const tabs = {
  dashboard: Dashboard,
  reports: Reports,
  settings: Settings,
};

function AdminPanel() {
  const [tab, setTab] = useState('dashboard');
  const [isPending, startTransition] = useTransition();
  const ActiveTab = tabs[tab];

  const handleTabClick = (next) => {
    startTransition(() => setTab(next));
  };

  return (
    <>
      <TabList>
        {Object.keys(tabs).map(key => (
          <TabButton
            key={key}
            active={key === tab}
            disabled={isPending}
            onClick={() => handleTabClick(key)}
          >
            {key}
          </TabButton>
        ))}
      </TabList>
      {isPending && <ProgressBar />}
      <Suspense fallback={<Spinner />}>
        <ActiveTab />
      </Suspense>
    </>
  );
}

// 切换体验：
// - 旧 tab 内容保留显示
// - 顶部 ProgressBar 提示加载
// - 新 tab ready 后立即替换
// - 没有"整页 spinner 闪一下" 的体验

场景 C · 配合 useDeferredValue 避免父强制刷新

自家组件库 SearchableList 接收 query prop。库的用户不用 useTransition（也无法），库内部用 useDeferredValue 推迟昂贵渲染：

javascript · 库内部 useDeferredValue 
// 库代码
export function SearchableList({ query, items, renderItem }) {
  const deferredQuery = useDeferredValue(query);

  const filtered = useMemo(
    () => items.filter(i => i.name.toLowerCase().includes(deferredQuery.toLowerCase())),
    [items, deferredQuery],
  );

  const isStale = query !== deferredQuery;

  return (
    <ul style={{ opacity: isStale ? 0.5 : 1, transition: 'opacity 0.2s' }}>
      {filtered.map(renderItem)}
    </ul>
  );
}

// 用户代码 —— 不需要 startTransition
function App() {
  const [query, setQuery] = useState('');
  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <SearchableList
        query={query}
        items={largeItems}
        renderItem={item => <li key={item.id}>{item.name}</li>}
      />
    </>
  );
}

// 库设计原则：
// - useTransition：用户主动控制（库不能强制用户用）
// - useDeferredValue：库内部推迟（用户不需要知道）
// - 库通过 useDeferredValue 默认提供"丝滑体验"，用户零成本

05 常见陷阱

⚠️ 陷阱 1：以为 startTransition 让代码异步

startTransition(() => { ... }) 内代码仍同步执行。需要异步用 setTimeout / Promise / async 函数。

规避：理解 transition 是"调度优先级提示"，不是延迟工具。要延迟用 debounce / throttle。

⚠️ 陷阱 2：React 18 在 startTransition async 函数里 await

startTransition(async () => { await fetch(); setData(); })—— React 18 中 await 之后的 setState 不在 transition 中（async 函数返回 Promise，scope 外的 setState 不被标记）。

规避：① React 18：在 await 后再调一次 startTransition 包 setData ② React 19：直接传 async 函数 OK ③ 不确定版本时用 React 19 + 升级。

⚠️ 陷阱 3：在小渲染上加 transition（负优化）

transition 本身有 Lane 调度开销。简单 setState（< 5ms 渲染）加 transition 比直接更新还慢。

规避：用 Profiler 测。仅当渲染 > 50ms 才用 transition。简单更新直接 setState。

⚠️ 陷阱 4：把 useTransition 当 debounce 用

const [, startTransition] = useTransition() + fetch 频繁触发 —— transition 不阻止 fetch 频率，只让渲染让位。fetch 仍每键打。

规避：理解两者解决不同问题。API 频率用 debounce，UI 渲染用 transition。配合用（场景 A 已演示）。

⚠️ 陷阱 5：startTransition 内 setState 后立即读 state（同 setState 异步陷阱）

startTransition(() => { setX(5); console.log(x); })—— x 仍是当前 render 的值（P2.4 setState 异步性）。transition 不改变这点。

规避：理解 setState 异步性（P2.4）。要看新值用 useEffect 监听。
