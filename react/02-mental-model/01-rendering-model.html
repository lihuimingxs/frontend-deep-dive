P2.1 · 渲染模型与 React 元素

本阶段从最底层概念开始：React 元素是什么？什么样的对象能被 React 渲染？组件的 7 种类型各自做什么？render 在什么时候触发？为什么 render 必须是纯函数？读完本章，你能精确回答"render() 到底返回了什么" 和"什么操作会让 React 重新渲染我的组件"—— 这是后续所有 本主题章节的基础。

📐 本章是 P2 的入口

如果你跳过了 P1 直接读后续阶段，本章是必读。建议先读 P1.2 设计哲学（5 分钟）建立心智地基——本章的"为什么" 都能在 P1.2 找到答案。

阅读时长 ~25 分钟 前置 JSX 基础语法 + ES2015 class（可选） 对应 react.dev Your First Component · createElement

01 速览

三句话定义本章：

  1. React 元素是普通 JS 对象：{ $$typeof, type, props, key, ref }。不是 DOM 节点 / 不是 class 实例 / 不是组件—— 它只是"UI 应该是什么样" 的描述。<div /> 在编译后就是 { type: 'div', props: {} }。
  2. 组件是接受 props 返回 React 元素的函数。"函数组件 / 类组件" 是写法不同，本质都是"props → element tree" 的映射。React 19 时代函数组件是默认，class 组件仍支持但仅用于 Error Boundary（极少数场景）。
  3. render 在 5 种情况触发：① state 变化 ② props 变化（含 parent 重渲染让你 props 变）③ context 变化 ④ 父组件重渲染（即使 props 引用相等也会触发，除非 React.memo）⑤ React 自身的 Strict Mode 双重渲染。render 必须是纯函数—— 因为 React 18+ Concurrent rendering 可能多次调用组件函数。

看一段最小代码，识别其中的"元素 / 组件 / render" 三个概念：

javascript · 三个概念在一段代码里 
// 1. Greeting 是「组件」—— 一个返回 React 元素的函数
function Greeting({ name }) {
  return <h1>Hello, {name}</h1>;
  //     ↑ 这里的 <h1> 编译后是「React 元素」对象：
  //       { type: 'h1', props: { children: ['Hello, ', name] } }
}

// 2. 调用 Greeting() 得到一个「React 元素」（不是 HTML，不是 class 实例）：
const element = <Greeting name="Alice" />;
//                ↑ 编译后：{ type: Greeting, props: { name: 'Alice' } }
console.log(element);  // 一个普通 JS 对象

// 3. 「render」是 React 把这个元素树映射到 DOM 的过程：
import { createRoot } from 'react-dom/client';
createRoot(document.getElementById('root')).render(element);
//                                          ↑ 这里的 render 是 root.render
//                                            React 内部会调用 Greeting 函数
//                                            "Greeting 函数被调用" = "组件 render"
//                                            （这两个 render 不是同一个东西，注意区分）

02 底层逻辑

React 元素的精确结构

React 元素是普通 JS 对象，React.createElement 创建（JSX 编译后调用的就是它）。结构如下：

javascript · React 元素对象结构 
// JSX
<div className="card" onClick={handleClick}>
  Hello {name}
</div>

// 编译后（旧 JSX runtime，React.createElement 风格）：
React.createElement(
  'div',                               // type
  { className: 'card', onClick: handleClick },  // props（不含 children）
  'Hello ',                            // children（可变参数）
  name,
);

// 实际产生的元素对象：
{
  $$typeof: Symbol(react.element),     // ← 用 Symbol 防止 XSS（详见下文）
  type: 'div',                         // ← 字符串（host 元素）/ 函数（组件）/ Symbol（Fragment 等）
  props: {
    className: 'card',
    onClick: handleClick,
    children: ['Hello ', name],        // ← children 也是 props 的一部分
  },
  key: null,                           // ← list 中的身份证（详见 P2.5）
  ref: null,                           // ← DOM 引用（详见 P3.5）
}

// 新 JSX runtime（React 17+，更高效）：
import { jsx as _jsx } from 'react/jsx-runtime';
_jsx('div', { className: 'card', onClick: handleClick, children: ['Hello ', name] });
// 详见 P2.2 JSX 编译与运行时

关键事实：

  • React 元素是 immutable—— 一旦创建不能改。要改 UI 要重新创建新元素树（这就是 P1.2 讲的"不可变 + 重新渲染"哲学）
  • 元素描述 UI，不是 UI 本身—— 元素是蓝图，DOM 节点是 React 根据蓝图实际操作浏览器创建的产物。Fiber 节点（P4.1）是 React 内部连接两者的桥梁
  • $$typeof 是 Symbol—— 不是字符串。这是 React 的 XSS 防御：服务器返回的 JSON 不可能含真正的 Symbol（JSON 不支持 Symbol 序列化），所以恶意服务器无法构造合法的 React 元素冒充。详见 React 安全文档

JSX 编译细节：详见下一章 P2.2 · JSX 编译与运行时（新 runtime / classic runtime / jsxImportSource 等）。本章只需理解 <X /> 编译后是一个对象。

组件的 7 种类型

React 19 中"什么能作为元素的 type"有 7 种：

组件类型 type 字段是什么 用途
Host 元素（'div' / 'span' / ...） 小写字符串 原生 DOM 节点（react-dom）/ 原生 view（react-native）
Function Component（默认） 函数引用 大部分组件。可用 Hooks
Class Component ⚠️ class 引用（含 prototype.render） 仅 Error Boundary（P5.4）+ 老代码遗产。新代码避免
memo 包装 { $$typeof: Symbol(react.memo), type: Comp } 跳过 props 浅相等的渲染（详见 P4.2）
lazy 包装 { $$typeof: Symbol(react.lazy), _payload, _init } 代码分割，配合 Suspense（详见 P5.2）
forwardRef 包装 ⚠️ { $$typeof: Symbol(react.forward_ref), render } 转发 ref。React 19 退役—— 用 ref as prop 替代
特殊组件（Fragment / Suspense / Profiler / StrictMode / Provider / Consumer） Symbol（如 Symbol(react.fragment)） React 内置语义组件

"组件首字母大小写" 的根因：JSX 编译时小写识别为 host 元素（type 是字符串）；大写识别为组件（type 是变量引用）。这就是 <div /> 和 <Div /> 完全不同——前者是字符串 'div'，后者要求作用域内有 Div 变量。

javascript · 大小写约定 
function MyButton() { return <button>Click</button>; }

<mybutton />     // ❌ 编译后 type: 'mybutton'（找不到此 host 元素）
<MyButton />     // ✅ 编译后 type: MyButton（函数引用）
<myButton />     // ❌ 同 'mybutton'
<my-button />    // ✅ Web Component 风格（含连字符 → host 元素）

render 触发的 5 种条件

这是React 性能优化的基础知识—— 准确知道"什么时候组件会被调用" 才能合理用 memo / useMemo / useCallback。

  1. state 变化：调用 setState / setReducer 更新本组件 state（用 Object.is 比较，相等则跳过）
  2. props 变化：父组件传给本组件的 props 引用变化（注意：父重渲染时即使 props 内容相同，没用 memo 也会重渲染）
  3. 父组件重渲染：父组件 render 时，所有子组件默认重渲染—— 不管 props 变没变。这是常被误解的点。React.memo 才能跳过这种"无意义" 重渲染
  4. context 变化：本组件订阅的 Context value 变化（用 Object.is 比较 value）
  5. StrictMode 双重渲染（仅开发环境）：React 主动多调用一次组件函数，帮你检测渲染期副作用
javascript · 父重渲染会让所有子重渲染（默认） 
function Parent() {
  const [count, setCount] = useState(0);
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <Child />     {/* 没传任何 props，但每次 Parent 重渲染都会被调用 */}
      <ExpensiveChild data={largeData} />  {/* 同样，largeData 引用没变也会被调用 */}
    </>
  );
}

function Child() {
  console.log('Child rendered');  // 每次点击 button 都打印
  return <div>child</div>;
}

// React 不会"智能跳过没变的子组件"——你必须用 memo 显式声明
const MemoedChild = React.memo(Child);
function ExpensiveChild({ data }) {
  return <div>{data.length}</div>;
}
const MemoedExpensiveChild = React.memo(ExpensiveChild);

// 用 MemoedChild 替代 Child 后：
// - MemoedChild 不接收 props → 永远跳过
// - MemoedExpensiveChild 浅比较 data props → 引用没变就跳过

"父重渲染默认让子重渲染" 是 React 的核心行为——它在 Fiber 阶段（P4.1）调 beginWork 处理每个子 Fiber，不管 props 是否变。memo 的作用就是"在 beginWork 之前加一道浅比较"，跳过这次调用。

render 必须是纯函数（这是为什么 useEffect 存在）

React 18+ 的 Concurrent rendering（P5.1）让组件函数可能被多次调用—— 一次渲染中途被高优先级更新打断 → 丢弃 wip tree → 重新渲染。所以组件函数必须是纯函数：

  • 相同的 props + state 必须返回相同的元素树
  • 不能修改 React 之外的状态（DOM、全局变量、外部 store）
  • 不能调有副作用的 API（fetch、setTimeout、console.log ⚠️——技术上副作用但 React 容忍）
  • 不能调 setState 导致死循环（除非有终止条件）

所有副作用必须放在 useEffect（commit 后异步）/ useLayoutEffect（commit 后同步）/ useState 的 lazy initializer（首次渲染） 中。详见 P3.2 useEffect 完整生命周期（编写中）。

javascript · 渲染期副作用违规清单 
function Component() {
  const [count, setCount] = useState(0);

  // ❌ 1. 渲染期间修改外部变量
  externalCounter++;

  // ❌ 2. 渲染期间调 fetch
  fetch('/api/log', { method: 'POST', body: JSON.stringify({ count }) });

  // ❌ 3. 渲染期间修改 state（无限循环）
  if (count < 10) setCount(count + 1);
  // 注意：useState 的 lazy initializer 不算 —— useState(() => expensive())
  //       那只在首次渲染调用一次

  // ❌ 4. 渲染期间修改 DOM
  document.title = 'Loading';

  // ❌ 5. 渲染期间修改 ref（在 react-strict-effects 时代视为违规）
  ref.current.value = 'something';

  return <div>{count}</div>;
}

// ✅ 全部副作用放 useEffect
useEffect(() => {
  externalCounter++;
  fetch('/api/log', { method: 'POST', body: JSON.stringify({ count }) });
  document.title = 'Loaded';
}, [count]);

StrictMode 双重渲染 —— React 帮你检测纯度

开发环境下，把组件树包在 <StrictMode> 里，每个组件函数会被调用两次（仅开发环境，生产构建无影响）。这是 React 主动检测"渲染期是否有副作用"的工具：

javascript · StrictMode 暴露副作用 
// main.tsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';

createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>
);

// 在 StrictMode 下：
// - 组件函数被调用 2 次（每次状态变化）
// - useState / useReducer 的 reducer 被调用 2 次
// - useEffect 在 mount 时跑 2 次（cleanup → mount → cleanup → mount）
// - useMemo / useCallback 函数被调用 2 次

// 如果你的代码在 2 次调用下行为不同 → 有渲染期副作用 → bug
// 例：
let id = 0;
function Component() {
  const myId = id++;  // ❌ 每次调用 id 都变 —— StrictMode 让你立即发现
  return <div>{myId}</div>;
  // StrictMode 下 myId 一会是 0 一会是 1，UI 闪烁
}

// 修复：把 myId 放进 useState 或 useId
const [myId] = useState(() => nextId++);

StrictMode 还检测：① componentWillMount 等过时生命周期 ② legacy context API ③ findDOMNode（已废弃）④ Concurrent 不安全的代码模式。新项目应该总是开 StrictMode—— 它是 React 给你的免费"纯度检测器"。

03 应用场景

场景 A · 调试"组件被调用太多次"

发现某个组件渲染开销大但被频繁调用——用 React DevTools Profiler 看，发现 parent 每次重渲染都让它跑。5 种 render 触发条件排查：

  1. 本组件 state 变了？看 useState / useReducer 调用 → 没改就排除
  2. props 变了？DevTools 的"Why did this render?"列会显示哪个 prop 变化
  3. 父组件重渲染？是的 → 加 React.memo 跳过（如 props 引用稳定）
  4. Context value 变？看本组件用了哪些 useContext —— Context value 用 useMemo 缓存
  5. StrictMode 双重？开发环境正常，生产 1 次 —— 不用担心

诊断后通常 React.memo + useMemo（稳定 props 引用）+ useCallback（稳定函数引用）组合解决。详见 P3.4（编写中）。

场景 B · 给同事讲清"父重渲染会让子重渲染"

新人常误以为 "子组件 props 没变就不会重渲染"。这是 Vue 心智—— Vue 用 Proxy 自动追踪依赖，确实能精确决定。React 的默认行为是相反的：父重渲染就调用所有子组件函数，不管 props。

解释步骤：① 引导他看 React DevTools Profiler，亲眼看到子组件被反复调用 ② 解释根因：P4.1 Fiber 的 beginWork 默认对每个子 Fiber 都跑 ③ 引出 React.memo 是显式的"跳过" 声明 ④ 提醒：memo 的浅比较会让 inline object / function props 失效（要用 useMemo / useCallback 配合）

场景 C · 评估"第三方组件库是否兼容 Concurrent"

升级到 React 18 后某第三方组件出现奇怪行为—— 怀疑它有渲染期副作用。验证：

  1. 本地开 StrictMode → 重现 bug？说明它在双重渲染下行为不同 → 有副作用
  2. 看库源码：组件函数体里有 useState 之外的修改 / 非 useEffect 的 fetch / 外部变量赋值？
  3. 提 issue / 提 PR / 寻找替代库

React 18 之前很多老库依赖"组件函数只跑一次" 的隐含假设。React 18 + StrictMode 把这些 bug 暴露出来——这是 18 升级时常见的痛点。

function App() {
  const element = (
    <div className="container">
      <h1>Title</h1>
      <Greeting name="Alice" />
    </div>
  );

  console.log(element);
  /* 控制台输出：
  {
    $$typeof: Symbol(react.element),
    type: 'div',
    key: null,
    ref: null,
    props: {
      className: 'container',
      children: [
        {
          $$typeof: Symbol(react.element),

javascript · 5 种触发条件可视化 
// 用 console.log 让每个 render 可见
function Parent() {
  const [count, setCount] = useState(0);
  const [theme] = useContext(ThemeContext);  // 假设有
  console.log('Parent render', { count, theme });

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>
        Count: {count}
      </button>
      <Child propValue="static" />
      <ChildMemo propValue="static" />
    </>
  );
}

function Child({ propValue }) {
  console.log('Child render', { propValue });
  return <div>static child</div>;
}

const ChildMemo = React.memo(function ChildMemo({ propValue }) {
  console.log('ChildMemo render', { propValue });
  return <div>memoed child</div>;
});

//   （ChildMemo 不打印）                       ← memo 跳过：propValue 字符串没变
//
// 测试 StrictMode（开发环境）：
//   Parent render { count: 1 }              ← 双重
//   Parent render { count: 1 }

05 常见陷阱

⚠️ 陷阱 1：把"render"等同于"DOM 渲染"

"render" 在 React 至少有 3 种不同含义：① 调用组件函数（"component render"）② React Fiber render 阶段（P4.1）③ 浏览器 paint。混着用必然误解。

规避：精确用语：

  • "组件 render" / "调用组件函数" → 函数被执行
  • "commit" / "提交" → React 真的修改了 DOM
  • "页面 paint" / "页面渲染" → 浏览器实际绘制
⚠️ 陷阱 2：以为子组件 props 没变就不重渲染

这是 Vue 心智。React 默认每次父重渲染都调用所有子组件函数，不管 props 变没变。要"props 没变就跳过" 必须显式 React.memo 包裹。

规避：理解 React 的默认行为是"积极调用"。性能问题用 memo + DevTools Profiler 定位，不要凭直觉假设。

⚠️ 陷阱 3：组件首字母小写

function myButton() { ... } + <myButton /> → JSX 编译为 type: 'mybutton'（host 元素）。React 会尝试把它当原生 HTML 元素，渲染失败或不生效。

规避：组件名永远首字母大写。ESLint react/jsx-pascal-case 自动检测。或者用 const MyButton = (props) => ... 函数表达式赋给大写常量。

⚠️ 陷阱 4：在 render 中产生副作用导致 StrictMode 暴露 bug

升级到 React 18 + StrictMode 后，看到组件 mount 时跑了 2 遍 fetch、计数器跳了 2 步、订阅注册了 2 次—— 不是 React 18 引入的 bug，是你之前就有渲染期副作用，StrictMode 把它暴露出来。

规避：所有副作用搬进 useEffect。useEffect 在 StrictMode 下也会跑两次（cleanup + mount + cleanup + mount），所以 cleanup 必须正确处理（详见 P3.2）。

⚠️ 陷阱 5：以为 console.log 在 render 中是"无副作用" 的

技术上 console.log 是副作用（它修改控制台），但 React 容忍——开发调试用没问题。但在 StrictMode 下 console.log 也会打两次，不要据此判断"组件被调用 N 次" 的实际次数（生产是 N/2 次）。

规避：调试时知道这点。Profiler 给的是真实数据，不依赖 console.log 计数。
