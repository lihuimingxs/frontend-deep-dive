A3 · Fiber 架构 ——「机器怎么跑？」

A1 讲了 2017 年 React 16 Fiber 重写是 React 的"创世事件"。A2 讲了"UI = f(state)" 等 4 个设计决策。本章把这些翻译成具体的运行时机器结构—— Fiber 节点是什么、Work loop 怎么循环、双缓冲怎么实现"原子提交"、Lane 模型怎么调度优先级、Hooks linked list 为什么决定了 Rules of Hooks。这是 React 主题的"引擎章"，对应 ECMAScript 主题的 A3 执行模型。

⚠️ 这是 React 主题最难的一章

Fiber 节点 / 双缓冲 / Work loop / Lane 模型 / Hooks linked list 是 React 内部的"5 件套"，互相依赖。绝大多数 React 工程师从没碰到过它们——但它们是后续"Concurrent 怎么做时间切片" / "Suspense 怎么 throw promise" / "useEffect 顺序为什么必须固定" 等问题的根因。建议至少读两遍，第二遍带着具体场景（如"调用 setState 时引擎做了什么"）回头对照。

📐 本章是唯一引用 React 源码的章节

按 outline 约定，仅本章读 React 源码（react-reconciler + scheduler 包），其他章节以 react.dev + RFCs 为限，避免实现细节稀释心智。本章的代码片段是简化伪代码，结构准确、字段名与源码一致，但去除了 dev / debug / 性能优化等细节。

阅读时长 ~50 分钟 前置 A2 设计哲学（必读）+ JS 链表心智 + 可选 ECMA · A3 执行模型 权威来源 Andrew Clark · React Fiber Architecture · facebook/react 源码

01 速览

三句话定义 React Fiber 架构：

  1. Fiber 节点 = 工作单元。每个 React 元素对应一个 Fiber 节点，含 child / sibling / return（parent）指针。Fiber tree 实际是链表而非树—— 这就是可中断遍历的关键。同时含 memoizedState（hooks 链表）/ flags（要做什么 DOM 操作）/ lanes（优先级）等字段。
  2. 渲染分两阶段：① Render 阶段（可中断）—— 沿 Fiber tree 遍历，beginWork 走下、completeUnitOfWork 走上，构建 workInProgress tree；② Commit 阶段（不可中断）—— 应用 DOM 变更（3 个子阶段：BeforeMutation / Mutation / Layout）+ 异步 Passive effects（useEffect）。双缓冲 让 commit 能"原子切换"两棵树。
  3. Lane 模型 = 31 位 bitmask 表达优先级。每个 lane 是一个 bit（SyncLane / DefaultLane / TransitionLane / IdleLane 等）。多个更新合并到同 lane → batching；高优先级 lane 打断低优先级 → 时间切片。Hooks linked list 存在每个 Fiber 节点的 memoizedState 字段—— 这就是"useState/useEffect 必须按相同顺序调用" 的引擎根因。

02 底层逻辑

Fiber 节点结构 —— 链表式工作单元

每个 React 元素（无论 <div> / 函数组件 / class 组件 / Fragment / Suspense）都对应一个 Fiber 节点。Fiber 节点的关键字段（简化版）：

字段 用途
type 组件 / 元素类型（'div' / 函数 / class / Symbol）
stateNode 对应的实例：DOM 节点（host）/ class 实例 / FiberRoot
child 第一个子 Fiber（链表头）
sibling 下一个兄弟 Fiber
return 父 Fiber（命名为 return 是因为 completeUnitOfWork 完成后"返回" 给它）
memoizedState 函数组件：Hooks linked list 的头。class 组件：上次的 state
memoizedProps 上次提交后的 props
pendingProps 本次渲染待应用的 props
flags 位标记，表示这个 Fiber 需要做什么操作：Placement（插入新节点）/ Update（更新）/ Deletion（删除）/ Passive（有 useEffect）/ Layout（有 useLayoutEffect）等
subtreeFlags 子树是否有任何 flag —— 用于"跳过没有变化的子树" 优化
lanes 这个节点的待处理优先级（lane bitmask）
childLanes 整个子树合并的 lanes —— 用于跳过低优先级子树
alternate 指向另一棵树（current ↔ workInProgress）的对应 Fiber
updateQueue 待处理的 state 更新队列（class component / useReducer）

关键洞察：Fiber tree 看起来是树（有 parent / child 关系），但实际遍历用的是链表 + 指针。这就是"可中断" 的物质基础——遍历到任意 Fiber，把指针记下，下次从这个指针继续。如果是 React 15 的递归实现，"暂停" 意味着把整个 JS 调用栈保存下来，根本做不到。

双缓冲（Double Buffering）—— 原子提交的关键

React 始终维护两棵 Fiber tree：

  • current tree：当前屏幕上看到的（已 commit 的）
  • workInProgress tree（简称 wip）：正在构建中的下一版本

每个 Fiber 节点的 alternate 字段指向另一棵树的对应节点—— current 节点的 alternate 是 wip 节点，反之亦然。这让 React 能复用 Fiber 对象（避免 GC 压力）：每次新一轮渲染，wip 树克隆 current 树（用 alternate 指针快速找到对应节点），然后修改 wip 节点的 props / state。

javascript · 双缓冲伪代码 
// 简化版 createWorkInProgress
function createWorkInProgress(current, pendingProps) {
  let workInProgress = current.alternate;

  if (workInProgress === null) {
    // 第一次渲染：创建新 Fiber，建立 alternate 关联
    workInProgress = createFiber(current.tag, pendingProps, current.key);
    workInProgress.stateNode = current.stateNode;
    workInProgress.alternate = current;
    current.alternate = workInProgress;
  } else {
    // 复用：从 alternate 拿，重置必要字段
    workInProgress.pendingProps = pendingProps;
    workInProgress.flags = NoFlags;
    workInProgress.subtreeFlags = NoFlags;
  }

  // 复制 current 的状态（state / hooks 链表 / 子节点引用）
  workInProgress.child = current.child;
  workInProgress.memoizedProps = current.memoizedProps;
  workInProgress.memoizedState = current.memoizedState;
  workInProgress.updateQueue = current.updateQueue;
  return workInProgress;
}

// Commit 阶段最后一步：原子切换两棵树
function commitRoot(root) {
  // ... 各种 effects 处理 ...
  root.current = finishedWork;  // ⚡ 一行代码切换 —— 这就是"原子提交"
}

双缓冲的工程意义：commit 阶段必须把 wip 的所有 DOM 变更一次性应用，中间不能有"渲染了一半" 的中间态。如果只用一棵树，render 时直接修改它，render 中途让出主线程，浏览器看到的就是"半成品"（一些 div 已更新、一些没更新）。双缓冲让 render 在独立的 wip 树上工作，commit 时切换 root.current 指针——浏览器永远只看到"已完成" 的 current tree。

这个机制和图形学的 double buffer（屏幕缓冲与后台缓冲）是同一思想，也是所有"支持时间切片" 框架（Solid 用相同模式，Vue 3 不需要因为它的 reactivity 是细颗粒度）的必备技巧。

Work Loop —— 可中断的遍历

React 把"渲染整棵树" 拆成"遍历每个 Fiber 节点" 的循环，每个节点叫一个 work unit：

javascript · Work Loop 主循环（简化） 
// 同步模式（不可中断）—— 旧版本 / 强制同步更新
function workLoopSync() {
  while (workInProgress !== null) {
    performUnitOfWork(workInProgress);
  }
}

// Concurrent 模式（可中断）—— React 18+
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress);
  }
  // shouldYield() 用 scheduler 包检查是否超过 5ms 时间片
  // 超过 → return → 让出主线程 → 浏览器响应输入 / 渲染 / 动画
}

// 单个工作单元的处理
function performUnitOfWork(unitOfWork) {
  const current = unitOfWork.alternate;

  // ① beginWork：处理当前节点，决定是否继续往下走
  let next = beginWork(current, unitOfWork, renderLanes);

  unitOfWork.memoizedProps = unitOfWork.pendingProps;

  if (next === null) {
    // 没有子节点 → 进入 complete 阶段，往上"unwind"
    completeUnitOfWork(unitOfWork);
  } else {
    // 有子节点 → 继续往下深度优先遍历
    workInProgress = next;
  }
}

// shouldYield 简化版（实际在 scheduler 包）
const FRAME_INTERVAL = 5;  // ms
let deadline = performance.now() + FRAME_INTERVAL;
function shouldYield() {
  return performance.now() >= deadline;
}

beginWork 阶段（render 阶段的"下行"）：

  • 对函数组件：调用组件函数（"render"），处理 hooks，返回 children JSX
  • 对 class 组件：调用 render 方法
  • 对 host 组件（DOM）：处理 props，返回 children
  • 对 Fragment / Provider / Context：转发 children
  • 返回第一个子 Fiber，遍历继续往下；或返回 null 进入 complete 阶段

completeUnitOfWork 阶段（render 阶段的"上行"）：

  • 对 host 组件：创建 / 更新 DOM 节点（但还不 append 到屏幕——只是准备好），收集 effects
  • 合并子树的 effects 到当前节点的 subtreeFlags
  • 检查 sibling，有就走 sibling 的 beginWork；没有就 return（往上回到 parent 的 complete）

整个遍历是深度优先 + 后序：先 beginWork 走到最深的叶子，然后 completeUnitOfWork 一路 unwind 上来。这是经典的"iterative tree traversal" 模式，但用linked list 实现而非递归，所以可以中断。

Scheduler 包 —— 怎么"让出主线程"

shouldYield() 是关键——它怎么实现"让浏览器先响应用户输入再继续"？React 18+ 用独立的 scheduler 包，机制是：

  • 不用 requestIdleCallback—— 部分浏览器实现太慢（如 Safari），fps 不稳定
  • 用 MessageChannel 投递宏任务—— 比 setTimeout(0) 更快，浏览器能在每个宏任务后做 UI 渲染。详见 JS · 宏任务/微任务
  • 5ms 时间片—— 经验值：人眼对 16.67ms（60fps）以下卡顿不敏感，留 11.67ms 给浏览器渲染
  • 动态调整：如果上一帧渲染时间长，下一帧时间片缩短（让出更多）

Lane 模型 —— 31 位 bitmask 表达优先级

React 18 引入 Lane 模型—— 用 31 位 bitmask（不用 32 位是为了避免 JavaScript 把 32 位整数转成负数）表达优先级。每个 lane 是一个 bit：

javascript · Lane 定义（简化） 
const TotalLanes = 31;

// 单 lane（每个是一个 bit）
const NoLane                  = 0b0000000000000000000000000000000;
const SyncHydrationLane       = 0b0000000000000000000000000000001;
const SyncLane                = 0b0000000000000000000000000000010;
const InputContinuousLane     = 0b0000000000000000000000000001000;
const DefaultLane             = 0b0000000000000000000000000100000;
const TransitionLane1         = 0b0000000000000000000000010000000;
const TransitionLane2         = 0b0000000000000000000000100000000;
// ... 共 16 个 TransitionLanes ...
const RetryLane1              = 0b0000000000010000000000000000000;
// ... 4 个 RetryLanes ...
const SelectiveHydrationLane  = 0b0000010000000000000000000000000;
const IdleHydrationLane       = 0b0000100000000000000000000000000;
const IdleLane                = 0b0001000000000000000000000000000;
const OffscreenLane           = 0b0010000000000000000000000000000;
const DeferredLane            = 0b0100000000000000000000000000000;

// 多 lane 合并（bitwise OR）
const lanes = SyncLane | DefaultLane | TransitionLane1;

// 检查 lane 包含
const includesSyncLane = (lanes & SyncLane) !== NoLane;

// 取最高优先级 lane（最低 bit）
function getHighestPriorityLane(lanes) {
  return lanes & -lanes;  // 经典 bitwise 技巧
}

Lane 模型的设计精髓：

  • 多个更新合并：同一 event handler 的多个 setState 合并到同一 lane bitmask（用 OR），一次渲染处理
  • 优先级抢占：高优先级 lane 在低优先级 lane 渲染中途到达时，丢弃低优先级 wip tree，重新从高优先级开始
  • 位运算极快：所有 lane 操作（OR / AND / 找最高位）都是单 CPU 指令
  • 同时表达多种状态：一个 Fiber 可以同时有 SyncLane（urgent 更新）和 TransitionLane1（transition 更新），各自独立处理

典型 lane 分配：

  • setState 在 click handler 内 → SyncLane（同步、最高优先级）
  • setState 在 useEffect / Promise.then → DefaultLane（默认）
  • startTransition(() => setState(...)) → TransitionLane1-16（低优先级，可被打断）
  • useDeferredValue 内部 → DeferredLane
  • render 错误 retry → RetryLane1-4

Render 阶段 vs Commit 阶段

Render 阶段（可中断）

上面 Work Loop 描述的全部过程都是 Render 阶段——构建 wip tree、决定有哪些 effects。render 阶段不修改 DOM，只在内存里准备数据结构。所以可以中断，中间让出主线程没有问题。

因此 React 18+ 加了一些约定：render 阶段执行的代码（组件函数体）应该是纯函数，因为它可能被多次调用（被中断后从头重试）。这就是 StrictMode 双重渲染的目的——开发环境主动多调用一次，帮你检测渲染期副作用。

Commit 阶段（不可中断）⭐

render 完成后进入 commit—— 同步、不可中断、必须一次性完成。分 3 个子阶段：

  1. BeforeMutation（变更前）：调 getSnapshotBeforeUpdate（class component），保存即将变更前的 DOM 状态。极少使用。
  2. Mutation（变更）：实际操作 DOM —— 创建 / 更新 / 删除节点 + 处理 ref（先 detach 老 ref，再 attach 新 ref）。这是用户最终看到的"页面变化" 时刻。
  3. Layout（布局后）：调 useLayoutEffect 回调 + class 的 componentDidMount / componentDidUpdate。同步执行，阻塞浏览器渲染—— 所以 useLayoutEffect 内的 DOM 测量能"看到准确尺寸但用户还没看到" 的中间态，可在这里调整布局而不闪烁。

之后异步：Passive Effects（useEffect 回调）—— 在 commit 完成后下一个 microtask 或 macrotask 跑（不阻塞屏幕渲染）。这是 useEffect vs useLayoutEffect 的核心区别。

javascript · commit 阶段时序 
// commit 同步部分（不可中断）：
// 1. BeforeMutation 子阶段
//    → 遍历 effect list，调 getSnapshotBeforeUpdate
// 2. Mutation 子阶段
//    → DOM 节点 placement / update / deletion
//    → ref detach（旧 ref 设为 null）
// 3. Layout 子阶段
//    → ref attach（新 ref 赋值）
//    → useLayoutEffect 回调（同步！）
//    → componentDidMount / componentDidUpdate（同步！）
//
// commit 同步部分结束 ──── 浏览器渲染屏幕 ────
//
// 之后异步：
// 4. Passive Effects（useEffect）
//    → flushPassiveEffects 在下一个 macrotask 执行
//    → 不阻塞屏幕渲染，对性能更友好

Hooks Linked List ⭐ —— Rules of Hooks 的引擎根因

这是本章最重要的部分—— 解释为什么"useState/useEffect 必须按相同顺序调用"是规范级要求，不是约定。

每个函数组件 Fiber 的 memoizedState 字段是一个单链表。每次组件函数执行时，每个 useXxx 调用按调用顺序对应链表的一个节点：

javascript · Hooks linked list 结构 
function Counter() {
  const [count, setCount] = useState(0);     // ← Hook 节点 #1
  const [name, setName] = useState('Alice'); // ← Hook 节点 #2
  useEffect(() => {                          // ← Hook 节点 #3
    document.title = name;
  }, [name]);
  const ref = useRef(null);                  // ← Hook 节点 #4
  return /* JSX */;
}

// fiber.memoizedState 链表：
// {                          // Hook #1: useState
//   memoizedState: 0,        //   当前值
//   queue: { ... },          //   pending updates
//   next: ─────────────►     {                   // Hook #2: useState
//                              memoizedState: 'Alice',
//                              queue: { ... },
//                              next: ─────────────►   {                  // Hook #3: useEffect
//                                                       memoizedState: { tag, create, destroy, deps },
//                                                       next: ─────────► {              // Hook #4: useRef
//                                                                          memoizedState: { current: null },
//                                                                          next: null
//                                                                        }
//                                                     }
//                            }
// }

// 第一次 render：依次调用 useState/useEffect/useRef，
//                每个 hook 创建新节点 append 到链表尾部
// 第二次 render：组件函数重新执行，
//                每个 hook 按"调用次序"取对应节点
//                useState() 第一次调 → 取 #1，第二次 → 取 #2
//                完全靠"位置"对应，不靠"名字"！

这就是 Rules of Hooks 的根因：

  • 规则 1：Hooks 必须在顶层调用—— 因为 React 完全靠"调用顺序" 找对应链表节点。如果你把 useState 放在 if 里，下次渲染条件不同，第 N 个 hook 的链表节点对应的可能是另一个 hook，类型 / 数据完全错位。
  • 规则 2：只在 React 函数内调用—— 因为 hooks 实现依赖 fiber.memoizedState 字段，普通函数不在 React 调度上下文里，没有 fiber 可写。
javascript · 违反 Rules of Hooks 的危害 
function Counter({ shouldTrack }) {
  const [count, setCount] = useState(0);

  // ❌ Conditional Hooks
  if (shouldTrack) {
    const [name, setName] = useState('Alice');  // 有时跑、有时不跑
  }

  useEffect(() => {  // 这个 hook 的"调用次序号"会变！
    console.log(count);
  }, [count]);

  // 第一次 render（shouldTrack=true）：3 个 hooks 调用
  //   #1 useState(0) → memoizedState = 0
  //   #2 useState('Alice') → memoizedState = 'Alice'
  //   #3 useEffect → memoizedState = effect 描述符
  //
  // 第二次 render（shouldTrack=false）：2 个 hooks 调用
  //   #1 useState(0) → 取链表第 1 个 → OK
  //   #2 useEffect → 取链表第 2 个 → 但第 2 个是 'Alice' 的 useState 节点！
  //   类型错位 → React 报错或行为完全不可预期 ⚠️
}

// ESLint plugin react-hooks/rules-of-hooks 在 lint 时拦截
// React 运行时也会检测到 hook 数量变化并 warn

对比 Vue 3 Composition API：Vue 的 setup() 函数每个组件实例只跑一次，ref() / reactive() 调用次数和位置都不影响（因为没有"重新渲染时按顺序对齐"的需求—— Vue 用 Proxy 自动追踪）。React 选了"每次重新跑函数体 + 按顺序对齐" 这个简单模型，代价就是 Rules of Hooks。

Reconciliation Diff 算法 —— 怎么决定复用 / 销毁 / 创建

beginWork 处理一个 Fiber 时，对比 current children 和新 children（JSX 返回值），决定每个 child 是 reuse / unmount / mount。算法核心：

  • 同 type + 同 key → 复用：保留 current Fiber，更新 props，可能继续 reuse 子树（如果 props 没变 + memo）
  • 不同 type 或不同 key → 销毁 + 创建：unmount 老 Fiber（含其子树），mount 新 Fiber
  • 列表（多 child）：用 key 配对——key 相同视为同一个，按顺序变化时仅移动而非重建
javascript · 列表 diff 关键场景 
// 旧 children: [<li key="a">A</li>, <li key="b">B</li>, <li key="c">C</li>]
// 新 children: [<li key="b">B</li>, <li key="a">A</li>, <li key="c">C</li>]
// → React 识别到 key=a 仍存在，仅"移动"—— 不销毁重建（保留 DOM 节点状态）

// 不用 key（用 index）：
// 旧 children: [<li>A</li>, <li>B</li>, <li>C</li>]
// 新 children: [<li>B</li>, <li>A</li>, <li>C</li>]
// → React 按 index 配对：position 0 的 'A' 改成 'B'（实际是更新文本），
//   position 1 的 'B' 改成 'A' —— 看起来正确但实际是"覆写"，
//   原 li 的内部状态（如 input 焦点）会跑到错误位置 ⚠️

// 这就是为什么 list 必须用稳定 key：
// 1. key 必须唯一（同一 list 内）
// 2. key 必须稳定（重新渲染时同一逻辑项 key 不变）
// 3. 不要用 array index 作 key（除非 list 永不重排序）

// 详见 B5 / B12 —— 用户视角的 reconciliation

03 应用场景

场景 A · 调试"useEffect 顺序异常"

生产环境收到 React Error: "Rendered more hooks than during the previous render" —— 知道是 hook 数量变化引发的。从 Fiber 视角看：

  • 第一次 render 函数组件调用了 3 个 hook → 链表 3 个节点
  • 第二次 render 调用了 4 个 hook → React 检测到 hook 数量变化 → 报错

排查：① 搜索组件代码所有 use* 调用 ② 检查每个 hook 是否在顶层无条件调用（不在 if / for / try-catch / 函数嵌套里）③ ESLint react-hooks/rules-of-hooks 应该 lint 阶段就发现，配 CI 拦截

场景 B · 理解 React DevTools Profiler 输出

DevTools Profiler 录一段交互后，看到 commit phase 的 flame graph。每个矩形条对应一个 Fiber 节点的 render 时间。从 Fiber 视角解读：

  • "Why did this render?" 列：显示触发本组件 render 的原因——state changed / props changed / parent re-rendered / context changed
  • commit 时间：所有 mutation + Layout phase 同步执行的总时间
  • passive effects 时间：useEffect 回调异步执行时间
  • flag 颜色：橘色 / 黄色 / 绿色对应不同 lane 优先级

理解 Fiber 内部后，Profiler 不再是"看不懂的图"，而是 "每个 Fiber 节点 work loop 中的开销可视化"。

场景 C · 写自定义 host（react-reconciler）

react-reconciler 是 React 的"核心引擎" 包，给 react-dom / react-native / Ink（终端 UI）/ react-three-fiber（3D）/ Remotion（视频生成）等使用。如果要给自己的渲染目标写 host（如把 React tree 渲染到 PDF / canvas / 自家原生平台）：

javascript · react-reconciler host 接口（节选） 
import Reconciler from 'react-reconciler';

const reconciler = Reconciler({
  // 必须实现的 host config
  createInstance(type, props, rootContainer, hostContext, internalHandle) {
    // 创建一个原生 instance（如 PDF 节点 / canvas 形状）
  },
  appendInitialChild(parentInstance, child) { /* ... */ },
  appendChild(parentInstance, child) { /* ... */ },
  removeChild(parentInstance, child) { /* ... */ },
  insertBefore(parentInstance, child, beforeChild) { /* ... */ },

  // commit 阶段的 mutation
  commitUpdate(instance, updatePayload, type, oldProps, newProps) { /* ... */ },
  commitTextUpdate(textInstance, oldText, newText) { /* ... */ },

  // 配置选项
  supportsMutation: true,
  supportsPersistence: false,
  isPrimaryRenderer: true,
});

// 使用：
const root = reconciler.createContainer(rootInstance, ...);
reconciler.updateContainer(reactElement, root);

// 整个 React Fiber + Hooks + Concurrent 系统都能跑在你的自定义 host 上！

写过 react-reconciler host 的工程师对 Fiber 的理解会上一个台阶——你必须实现 commit 阶段的所有 mutation 操作，这就是 host 与 React 内核的接口。

追踪一次 setState 在 Fiber 内部的完整路径

javascript · setState 引发的内部流程 
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

// 用户点击 button 后内部发生的事：

// ① 浏览器派发 click 事件 → React 合成事件系统接管
//    React 把 setCount(count + 1) 包在 batchedUpdates 中

// ② setCount 实质：dispatchSetState(fiber, queue, value)
//    - 创建一个 update 对象 { lane: SyncLane, value: 1, next: null }
//    - 把 update 加入 queue.pending（环形链表）
//    - 在 fiber.lanes 上 OR 这个 lane

// ③ 调度：scheduler 检查 root 的 pendingLanes
//    - 选最高优先级 lane（这里 SyncLane）
//    - 创建 wip tree（用 alternate 复制 current）

// ④ Render 阶段（work loop）
//    - performUnitOfWork(rootFiber)
//    - beginWork：调用 Counter 函数
//      - useState 内部：从 hook linked list 取节点 #1
//      - 处理 update queue：count = 0 + 1 = 1
//      - 返回 JSX <button onClick={...}>1</button>
//    - 创建 button 子 Fiber（diff 老 Fiber）
//    - completeUnitOfWork(buttonFiber)：
//      - 准备 DOM 操作 flag = Update（textContent）
//    - completeUnitOfWork(rootFiber)

// ⑤ Commit 阶段
//    - BeforeMutation：无操作
//    - Mutation：button.firstChild.textContent = '1'
//    - Layout：无 useLayoutEffect

// ⑥ Commit 完成 → root.current = wip tree
//    - 浏览器渲染屏幕 → 用户看到 1

// ⑦ Passive Effects（无）
//    - 如果有 useEffect 依赖 count，会在这里触发

// 整个过程同步完成（SyncLane），用户感知"立即响应"。
// 如果是 startTransition 包裹，会走 TransitionLane，可被打断。

对比 Fiber 时间切片 vs 不切片

javascript · 大列表渲染的时间切片实测 
// 假设有一个 10000 项的列表，渲染每个项要 0.1ms

function HugeList({ items }) {
  return (
    <ul>
      {items.map(item => <Item key={item.id} {...item} />)}
    </ul>
  );
}

// 场景：用户在搜索框输入文字过滤列表
const [query, setQuery] = useState('');
const [items, setItems] = useState(allItems);

const handleChange = (e) => {
  setQuery(e.target.value);              // SyncLane（用户输入）
  setItems(filter(allItems, e.target.value));  // 默认 DefaultLane
};

// 没有 useTransition：
// - 每次输入触发完整 work loop 渲染 10000 个 Item
// - work loop 占用主线程 ~1000ms
// - 期间用户输入完全卡顿

// 有 useTransition：
const [isPending, startTransition] = useTransition();
const handleChange = (e) => {
  setQuery(e.target.value);              // SyncLane（输入框立即响应）
  startTransition(() => {
    setItems(filter(allItems, e.target.value));  // TransitionLane（低优先级）
  });
};

// React 内部：
// 1. 输入框 setQuery → SyncLane → 完整渲染输入框（5ms 内完成）
// 2. setItems 走 TransitionLane → work loop 跑 5ms → shouldYield → 让出
// 3. 浏览器响应下一次输入 → 再次进入 SyncLane → ...
// 4. TransitionLane 一直被打断到用户停止输入 → 最后渲染完整列表
//
// 用户感知：输入框丝滑，列表在最终输入完成后~100ms 出现

05 常见陷阱

⚠️ 陷阱 1：把"render"等同于"DOM 渲染"

React 的 "render" 是Render 阶段—— 调用组件函数 + 构建 wip Fiber tree，不修改 DOM。Commit 阶段的 Mutation 子阶段才动 DOM。

规避：精确用语：①"组件 render"指调用组件函数 ②"commit"指 DOM 实际更新 ③"页面渲染"指浏览器 paint。术语混淆 = 沟通成本上升。

⚠️ 陷阱 2：把 useLayoutEffect 当 useEffect 用

useLayoutEffect 在 Layout 子阶段同步执行，阻塞浏览器渲染；useEffect 在 commit 之后异步执行，不阻塞。把昂贵操作放 useLayoutEffect 会让用户看到"延迟一帧" 才显示。

规避：useLayoutEffect 仅用于必须在浏览器 paint 前完成的 DOM 测量 / 同步调整（如调整工具提示位置避免闪烁）。其他全部用 useEffect。详见 B7 useEffect（编写中）。

⚠️ 陷阱 3：Conditional Hooks（最严重）

if (cond) useState() / for (...) useState() / useState() 在 try-catch 里 都是违反 Rules of Hooks。引擎根因如本章 Hooks linked list 部分讲过——按调用顺序对应链表节点，条件调用导致错位。

规避：① ESLint react-hooks/rules-of-hooks CI 拦截 ② 改写：把条件移到 hook 内部（useEffect(() => { if (cond) ... }, [cond])）③ 用早返回前确保所有 hooks 已调用

⚠️ 陷阱 4：用 index 当 list key

如果 list 排序 / 过滤 / 增删，index 当 key 会让 React 错误"复用" 错位的 Fiber——内部 input 的焦点 / 滚动状态 / 自定义 component state 全部跑到错误的 li 上。

规避：① 用业务 ID 作为 key（todo.id、user.email 等稳定唯一）② 实在没有 ID，用 crypto.randomUUID() 在数据生成时分配 ③ 仅"list 永不重排序" 的特殊场景才能用 index

⚠️ 陷阱 5：在 render 期间产生副作用破坏并发

React 18 的 Concurrent rendering 让 render 阶段可能被中断后从头重试。如果你在组件函数体中修改了外部变量 / 全局状态 / DOM，重试时副作用会跑两次。

javascript · 并发不安全代码 
let externalCounter = 0;

function Component() {
  externalCounter++;  // ❌ render 期间修改外部 —— Concurrent 下会跑多次
  return <div>{externalCounter}</div>;
}

// StrictMode 双重渲染会让 externalCounter 永远是 2、4、6（双倍）
// 真正的 Concurrent 中断重试会让计数完全不可预测

规避：组件函数体只做"给定 props/state 计算 JSX"，所有副作用进 useEffect。Strict Mode 帮你检测——开发环境双重渲染暴露这种 bug。
