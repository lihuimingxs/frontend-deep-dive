A2 · 设计哲学 ——「它是什么？」

A1 讲了 React 的历史时间线。本章讲 React 的设计基因—— 4 个核心决策，每个都对应一个具体痛点（Facebook 2010-2013 年遇到的 jQuery 时代问题）。理解这 4 个决策能解释 React 的所有"古怪"行为：为什么 JSX 不是模板？为什么不能直接 mutate state？为什么 useEffect 看起来像生命周期但实际不是？为什么需要 key？答案都在 4 个决策里。

📐 本章的"心智地基"角色

本章是 React 主题所有 API 章节（B 轨）的"为什么"。读完后再看 useState / useEffect / Reconciliation / Server Components，所有"API 为什么是这个样子"的问题都有了一致的答案。如果只读 React 主题一章，应该是这一章。

阅读时长 ~30 分钟 前置 A1 起源与演化 + 任意 JS 框架经验（jQuery / Vue / Angular 中至少一个） 权威来源 react.dev · Thinking in React · Dan Abramov · overreacted.io

01 速览

三句话定义 React 的设计本质：

  1. React 的 4 个核心决策构成"声明式 UI 系统"：① UI = f(state)（声明式 vs 命令式）② 单向数据流（vs 双向绑定）③ Composition over inheritance（组合 vs 继承）④ 不可变 + 重新渲染（vs Proxy 反应式）。4 个决策互相支撑，去掉任何一个 React 都不再是 React。
  2. 每个决策都是为了解决具体痛点。不是哲学洁癖，是工程答案：
    • UI = f(state) → 解决"状态散布在 DOM 各处"（jQuery 时代痛点）
    • 单向数据流 → 解决"双向绑定让数据传播路径不可预测"（Angular 1.x 痛点）
    • Composition → 解决"类继承的菱形问题"
    • 不可变 + 重渲染 → 解决"怎么知道要重渲染" 的工程问题（O(1) 引用比较 vs O(n) 结构比较）
  3. 每个决策都有代价，必须接受才能用 React。"React 怪" = 不接受这些代价。比如：JSX 不是 HTML（class → className）、key 必须稳定唯一、不可变更新繁琐（Immer 缓解）、函数组件每次重新创建函数（useMemo/useCallback 缓解）。

用一张表把 4 个决策和"它替代了什么" 对齐：

决策 解决的痛点 替代了什么
UI = f(state) 状态散布 DOM 各处 jQuery 命令式 DOM 操作 / Backbone view.render
单向数据流 状态传播路径不可预测 Angular 1.x 双向绑定 / Knockout observable
Composition 类继承的菱形 / mixin 冲突 Backbone extend / Angular service 注入
不可变 + 重渲染 怎么知道要重渲染（O(1) vs O(n)） Vue Proxy 反应式 / MobX observable

02 底层逻辑

① UI = f(state) —— 声明式编程的具体落地

这是 React 的第一公理。你不告诉 React "把这个 div 的 textContent 改成 X"，你告诉它"状态是这样，UI 应该长这样"——由 React 决定怎么变 DOM。

javascript · 命令式 vs 声明式 
// 命令式（jQuery 时代）—— "怎么做"
$('#count-display').text(count);
$('#increment-btn').on('click', function () {
  count++;
  $('#count-display').text(count);  // 必须手动同步 DOM
});

// 声明式（React）—— "是什么"
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <span>{count}</span>                          {/* 状态变 → UI 自动变 */}
      <button onClick={() => setCount(count + 1)}>+</button>
    </div>
  );
}

这个决策的痛点驱动是 A1 讲过的"chat bug"——当多个事件处理器各自更新 DOM，没人有"整体状态" 视角，bug 必然出现。UI = f(state) 把"整体状态" 强制集中到 React state，UI 永远是状态的纯函数派生，bug 来源被根除。

横向对比：

框架声明方式实现机制
React JSX（JS 表达式） 每次状态变化重新调用组件函数 → Fiber 比对差异 → 最小 DOM 更新
Vue 3 SFC 模板（HTML 风格） Proxy 拦截属性访问，自动追踪依赖，依赖变才重新渲染
Svelte SFC 模板 编译时优化为 imperative DOM 操作（"消失的框架"）
Solid JSX（看起来像 React） signals 细颗粒度订阅，组件函数只跑一次，状态变化只更新订阅它的 DOM

关键洞察：4 个框架都是声明式 UI，但实现机制完全不同。React 的"每次状态变化重新调用整个组件函数"是最简单 / 最可预测的模型，但性能上限受限——这就是 Fiber + Concurrent rendering 存在的原因（A3 详述）。

UI = f(state) 的进一步推论：

  • 组件函数必须是纯函数：相同 props + state 必须返回相同 JSX。渲染期间不能修改 state、不能调 API、不能改 DOM—— 副作用必须放 useEffect
  • 同一段代码可能跑多次：React 18 StrictMode 下开发环境会双重渲染，正是利用"纯函数应该幂等" 这个性质来检测副作用
  • 状态变化触发整个子树重渲染—— 不是"差量更新"。你看到的 "性能很好" 是 Fiber 比对后只更新差异 DOM 的结果，不是 React 跳过组件函数

② 单向数据流 —— 数据传播路径可预测

数据从 parent 通过 props 传到 child。child 不能修改 props——只能通过调用 parent 传下来的回调函数请求 parent 修改它自己的 state。

javascript · 单向数据流模式 
function Parent() {
  const [count, setCount] = useState(0);
  return <Child count={count} onIncrement={() => setCount(c => c + 1)} />;
}

function Child({ count, onIncrement }) {
  // child 只能"请求"修改，不能直接改 props
  return (
    <div>
      Count: {count}
      <button onClick={onIncrement}>+</button>
    </div>
  );
}

// 数据流：Parent state → Child props（向下）
// 事件流：Child onClick → 调用 onIncrement → Parent setCount（向上请求）
// 永远不会：Child 直接修改 count（这是 props，不可变）

这个决策的痛点驱动是 Angular 1.x 时代的"双向绑定地狱"——ng-model 让 input 和 controller 双向同步，看起来很方便但多个 input 互相影响时，状态传播路径完全无法追踪。Angular 团队在 2.0+ 也放弃了双向绑定（默认改单向）。

横向对比：

框架父子通信
React props 向下 + 回调向上请求（默认且唯一）
Vue 3 props 向下 + emit 向上 + v-model（双向语法糖，本质仍是 props + emit）
Angular（2+） @Input + @Output（默认单向）+ [(ngModel)]（双向语法糖）
Svelte 5 $bindable rune（双向但显式标注）

关键洞察：现代框架默认都是单向。双向绑定退化为"语法糖" + "显式标注"。React 是最严格的——没有双向语法糖，永远是 "props 向下 + 回调向上" 模式。

单向数据流的代价：

  • 表单写法繁琐：每个 <input> 都要 value + onChange 双绑（受控组件）。Vue 一个 v-model 解决
  • 跨多层组件传 state 痛苦：要么逐层 props 传（"prop drilling"），要么用 Context / 状态管理库
  • "提升 state"的额外步骤：两个兄弟组件需要共享 state 时，必须把 state 提升到共同 parent

这些代价催生了大量缓解方案：React Hook Form（表单库）、Context API（跨层）、Zustand / Redux（全局状态）。理解 React 的单向决策是这些方案存在的根本原因。

③ Composition over inheritance —— 组合而非继承

React 组件从不通过继承复用—— 不存在 class Card extends BaseCard 这种模式。复用通过 3 种 composition 模式：

  • Children prop（最常见）：<Card><CardBody /></Card>
  • Render Props：<DataLoader>{(data) => <ui />}</DataLoader>
  • 自定义 Hooks（19 时代主流）：const { data, loading } = useFetch(url)
javascript · 3 种 composition 模式 
// ① Children prop —— "把内容传给我的 slot"
function Card({ children, title }) {
  return (
    <div className="card">
      <h2>{title}</h2>
      {children}
    </div>
  );
}

<Card title="Welcome">
  <p>This is the body</p>
</Card>

// ② Render Props —— "我提供数据，你决定怎么渲染"
function DataLoader({ url, children }) {
  const [data, setData] = useState(null);
  useEffect(() => { fetch(url).then(r => r.json()).then(setData); }, [url]);
  return children(data);
}

<DataLoader url="/api/user">
  {(data) => (data ? <UserCard {...data} /> : <Spinner />)}
</DataLoader>

// ③ 自定义 Hooks —— "我提供逻辑，你自己渲染"（19 时代首选）
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    setLoading(true);
    fetch(url).then(r => r.json()).then(d => { setData(d); setLoading(false); });
  }, [url]);
  return { data, loading };
}

function UserPanel() {
  const { data, loading } = useFetch('/api/user');
  if (loading) return <Spinner />;
  return <UserCard {...data} />;
}

这个决策的痛点驱动是类继承的"菱形问题"——多继承时，两条父类链都包含同一基类的方法，子类继承哪个？Java / C# 通过 interface 解决；Mixin（Ruby / Python）通过 MRO 解决；JS 之前用 mixin（如 Vue 2 的 mixin），但仍有命名冲突 / 调试困难等问题。Composition 通过"显式参数传递"避开所有继承问题。

横向对比：

  • Vue 3：<slot>（类似 children）+ provide/inject（类似 Context）+ Composition API（类似自定义 Hooks）。3 种机制几乎一对一
  • Angular：内容投影（ng-content）+ DI service（依赖注入复用逻辑）—— 依赖注入仍是 OOP 心智
  • Svelte 5：<slot> + Snippets（render props 的等价物）+ Runes（响应式逻辑可在普通 JS 文件复用）

React 在 Composition 这点上是极致主义者——连 mixin 都不支持（Vue 2 / Vue 3 / Angular 都有过 mixin 设计）。所有复用必须通过显式 composition。

④ 不可变 + 重新渲染 —— "怎么知道要重渲染" 的工程答案

React 的 state 必须不可变。修改 state 的唯一方式是调用 setState 传入新引用：

javascript · 必须新引用 
// ❌ Mutate —— React 不知道 state 变了
const [user, setUser] = useState({ name: 'Alice', age: 30 });
user.age = 31;             // 直接 mutate
setUser(user);             // 同一引用！React 跳过更新

// ✅ 创建新引用
setUser({ ...user, age: 31 });          // 新对象
setUser(prev => ({ ...prev, age: 31 })); // updater function（推荐）

// ❌ 数组 mutate
const [items, setItems] = useState([1, 2, 3]);
items.push(4);
setItems(items);           // 同一引用，React 跳过

// ✅ 数组新引用
setItems([...items, 4]);
setItems(prev => [...prev, 4]);

"为什么不可变"的本质：React 在每次状态变化时需要回答"哪些组件需要重渲染"。两个候选答案：

  • 引用比较（O(1)）：用 Object.is(prevState, nextState) 比较两个引用。新对象 → 重渲染；同一引用 → 跳过。极快但要求开发者保证 mutate state 时创建新引用
  • 结构比较（O(n)）：递归比较两个对象的所有属性。能检测到 mutate，但性能差且对深嵌套数据爆炸

React 选了引用比较—— 这就是不可变要求的根因。不是哲学洁癖，是工程权衡的必然结果。

详细引用比较机制请参考 JS · 数据类型系统（Object.is / SameValueZero / 引用相等）和 ECMA · B3 抽象操作（IsStrictlyEqual）。

横向对比：

框架"知道要重渲染"机制开发者代价
React 引用比较（不可变要求） 必须创建新引用
Vue 3 Proxy 拦截属性访问，自动追踪依赖 可以直接 mutate（state.x = 1）
MobX 同 Vue 3（observable + 自动追踪） 同 Vue 3
Solid signals 显式订阅 用 setCount(c => c + 1) 模式
Svelte 5 Runes（编译时分析 + 运行时 signals） $state 直接赋值

关键洞察：React 是5 个框架里"开发者代价最高"的—— Vue/MobX 直接 mutate、Solid/Svelte 直接赋值，只有 React 必须 {...state, x: 1}。但 React 的引用比较心智也是最简单的——只要"新引用 = 重渲染"，没有"这个属性被 Proxy 追踪了吗" 这种隐式问题。代价换简单。

这就是为什么 Immer（社区方案）这么流行—— 它让你"看起来 mutate"，底层用 Proxy 自动产生新引用：

javascript · Immer 缓解不可变更新繁琐 
import { produce } from 'immer';

// 不用 Immer：嵌套更新地狱
setUser(prev => ({
  ...prev,
  profile: {
    ...prev.profile,
    address: {
      ...prev.profile.address,
      city: 'Beijing',
    },
  },
}));

// 用 Immer：看起来 mutate，实际产生新引用
setUser(produce(prev => {
  prev.profile.address.city = 'Beijing';
}));

// Zustand 内置 Immer 支持
useStore.setState(produce(state => { state.x = 1 }));

设计决策的代价 —— 必须接受才能用 React

4 个决策合在一起带来一组"代价"，不接受这些代价 = 觉得 React 怪：

  • JSX 不是 HTML：class → className（class 是 JS 关键字）/ for → htmlFor（同上）/ style 是对象 / 注释用 {/* */}。这是 JS 限制，不是 React 选择
  • 必须 key 才能高效 list reconciliation：UI = f(state) + 重渲染机制要求识别"哪些元素是同一个"。key 是身份证。详见 B5（编写中）
  • 不可变更新繁琐：嵌套数据更新写起来痛苦。Immer / Zustand-with-immer / Redux Toolkit 缓解
  • 函数组件每次渲染都重新创建函数：每个 onClick={() => ...} 都是新函数。useCallback 缓解但加心智成本。React 19 实验性的 React Compiler 自动加 memoization 是这个问题的"系统级答案"
  • useEffect 心智复杂：依赖数组、cleanup、闭包陷阱。详见 B7 useEffect（编写中）
  • 性能优化不直观：什么时候该用 memo / useMemo / useCallback 没有绝对规则—— 只能基于实际 measure

这些代价让 React不是"最易学的" 框架—— Vue 和 Svelte 入门曲线低得多。但 React 的设计决策让它在大型项目 / 长期维护 上有结构优势：单向数据流可预测、不可变让 time-travel debugging 可能、Composition 让代码组织灵活。选 React 是选"初期代价高但长期可维护" 的工程权衡。

03 应用场景

场景 A · 给 Vue / Angular / Svelte 工程师讲 React

最有效的切入点是"4 个决策 + 它解决了什么"，而不是从语法 / API 开始。具体讲法：

  1. 从痛点开始：先讲 jQuery 时代的 chat bug 和 Angular 1.x 的双向绑定地狱。让对方理解"为什么需要新框架"
  2. 4 个决策按顺序：UI=f(state) → 单向 → Composition → 不可变。每个决策都给"没有它会怎样"的反例
  3. 用对方框架的 API 做对照：v-if ↔ {cond && ...}、@click ↔ onClick、props（Vue）↔ props（React）
  4. 承认代价：JSX 怪、key 必要、不可变繁琐——但这些是 4 个决策的必然代价

最常见的转换困难：① Vue 工程师习惯直接 state.x = 1 mutate，React 必须 setState({...})；② Angular 工程师习惯 DI service，React 用自定义 Hooks 替代但语义不同。提前讲清这两点能减少 80% 的"React 怪" 抱怨。

场景 B · 设计自己的状态管理库

团队需要轻量状态管理，不想引第三方库——自己写一个"类 Zustand"。设计要点：

  1. 必须遵守"不可变 + 引用比较"—— 用 Object.is 浅比较 selected slice
  2. API 设计仿 Hooks：const count = useStore(s => s.count)（selector 模式）
  3. 更新 API 强制不可变：setState((prev) => ({...})) 或集成 Immer
  4. 跨组件共享靠 module scope + subscribers：不依赖 Context（Context 性能陷阱见 B8）

读 Zustand / Jotai / Valtio 的源码会发现它们都遵守这些设计。理解了 React 的 4 个决策，你能预测一个状态管理库的设计。

场景 C · 评估第三方组件是否符合 React 哲学

选 UI 库 / 业务组件时，可以用 4 个决策做"嗅觉测试"：

  • API 形态遵循 UI = f(state) 吗？<Modal open={isOpen} onClose={...} /> ✅ vs modalRef.current.show() ❌
  • 遵守单向数据流吗？onChange 形式 vs 内部 mutate parent 传入的对象
  • Composition 友好吗？支持 children / render props / asChild 等组合模式
  • 不可变安全吗？传入数据后内部不会 mutate

不符合 4 个决策的组件 = 与 React 哲学冲突，长期维护成本高。Material UI v5 / shadcn/ui / Radix UI 都符合这 4 点，是"哲学一致" 的库。某些 jQuery 时代移植到 React 的库（如老版 antd）有"命令式 API" 残留——避免使用。

04 典型代码

对比：命令式 jQuery vs 声明式 React 的同一 todo list

jQuery · 命令式 
$(document).ready(function () {
  var todos = [];

  $('#add-btn').on('click', function () {
    var text = $('#input').val();
    todos.push({ id: Date.now(), text, done: false });
    renderList();
    $('#input').val('');
  });

  function renderList() {
    var html = '';
    todos.forEach(function (todo) {
      html += '<li data-id="' + todo.id + '" class="' + (todo.done ? 'done' : '') + '">' +
              '<input type="checkbox" ' + (todo.done ? 'checked' : '') + '>' +
              todo.text +
              '<button class="delete">X</button></li>';
    });
    $('#list').html(html);

    // 重新绑定事件 —— 因为 innerHTML 替换了所有 DOM
    $('#list .delete').on('click', function () {
      var id = $(this).parent().data('id');
      todos = todos.filter(t => t.id !== id);
      renderList();
    });
    $('#list input[type="checkbox"]').on('change', function () {
      var id = $(this).parent().data('id');
      todos = todos.map(t => t.id === id ? {...t, done: this.checked} : t);
      renderList();
    });
  }
});
// 问题：每次 renderList 都要重新绑定事件 / DOM 全部 innerHTML 重建
React · 声明式 
function TodoList() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState('');

  const add = () => {
    setTodos(prev => [...prev, { id: Date.now(), text: input, done: false }]);
    setInput('');
  };
  const remove = (id) => setTodos(prev => prev.filter(t => t.id !== id));
  const toggle = (id) => setTodos(prev =>
    prev.map(t => t.id === id ? { ...t, done: !t.done } : t)
  );

  return (
    <>
      <input value={input} onChange={e => setInput(e.target.value)} />
      <button onClick={add}>Add</button>
      <ul>
        {todos.map(todo => (
          <li key={todo.id} className={todo.done ? 'done' : ''}>
            <input type="checkbox" checked={todo.done} onChange={() => toggle(todo.id)} />
            {todo.text}
            <button onClick={() => remove(todo.id)}>X</button>
          </li>
        ))}
      </ul>
    </>
  );
}
// 优势：UI 永远和 state 一致；不需要手动同步 DOM；事件绑定在元素上不会丢失

关键观察：jQuery 版本约 30 行，React 版本约 25 行——代码量差不多。但 jQuery 版本的所有 bug 风险都集中在"renderList 重建 DOM 后必须重绑事件" 这一行。React 版本根本没有这个问题——UI = f(state) + 单向数据流让"状态同步 UI" 成为框架的责任，开发者只关心状态本身。

引用比较 vs 结构比较的实测

javascript · React 的引用比较心智 
function Component() {
  const [obj, setObj] = useState({ count: 0 });

  const handleClick = () => {
    // ❌ Mutate —— React 跳过更新
    obj.count++;
    setObj(obj);  // setObj(obj) 时 Object.is(obj, obj) === true，跳过

    // ✅ 新引用
    setObj({ ...obj, count: obj.count + 1 });

    // 验证：
    const a = { count: 0 };
    const b = { count: 0 };
    Object.is(a, b);             // false —— 不同引用
    Object.is(a, a);             // true  —— 同一引用
    Object.is(a, {...a});        // false —— 展开产生新引用 ✅

    // React 内部就是用 Object.is 决定 state 是否变了
    // 所以 mutate state 不会触发渲染
  };

  return <button onClick={handleClick}>{obj.count}</button>;
}

// React.memo 的浅比较是逐个 props 用 Object.is
// 所以传 inline 对象 / 函数会让 memo 失效：
<Child config={{ x: 1 }} />        // 每次渲染都是新对象 —— memo 失效
<Child onClick={() => foo()} />    // 每次渲染都是新函数 —— memo 失效

// 解决：useMemo / useCallback 保证引用稳定
const config = useMemo(() => ({ x: 1 }), []);
const onClick = useCallback(() => foo(), []);

05 常见陷阱

⚠️ 陷阱 1：把 React 当"view 层"

"React 只是渲染层，状态管理用 Redux"——这是 2015 年的旧观念。今天 React 19 已是完整的"UI = f(state)" 运行时，包含状态机制（useState / useReducer / Context）+ 副作用机制（useEffect）+ 异步机制（Suspense / Actions）。

规避：理解 React 自带"整套状态机"。Redux / MobX 是补充工具（应对全局 / 跨组件 / 复杂状态机场景），不是必需品。简单应用 useState + useReducer 就够。

⚠️ 陷阱 2：用 Vue 的反应式心智写 React

Vue 工程师转 React 最常见错误：

javascript · Vue 心智的 React 代码 
// ❌ Vue 风格在 React 不工作
const [user, setUser] = useState({ name: 'Alice', items: [] });

// 直接 mutate（Vue 3 Proxy 能感知，React 不能）
user.name = 'Bob';
user.items.push('item1');

// React 完全不知道 state 变了，UI 不更新

// ✅ React 风格
setUser(prev => ({
  ...prev,
  name: 'Bob',
  items: [...prev.items, 'item1'],
}));

规避：写 React 时把 state 当"不可变快照"，永远 setState(prev => ({...prev, ...}))。如果嵌套太深，引入 Immer。

⚠️ 陷阱 3：滥用 useMemo / useCallback

"缓存就是优化" 是常见误区。useMemo / useCallback 本身有 cache 维护开销——对于简单计算 / 简单函数，cache 比直接重新计算还慢。

规避：useMemo / useCallback 只在以下场景有意义：① 传给 React.memo 子组件 ② 作为 useEffect / useMemo 依赖 ③ 真的昂贵的计算（>1ms）。其他场景直接重新计算。React Compiler（19 实验性）是终极答案——自动加 memoization，未来减少手动管理。

⚠️ 陷阱 4：在渲染期间产生副作用

UI = f(state) 要求组件函数是纯函数。常见违规：

javascript · 渲染期副作用 
function Component() {
  const [count, setCount] = useState(0);

  // ❌ 渲染期间修改 state（无限循环！）
  if (count < 10) setCount(count + 1);

  // ❌ 渲染期间调 API
  fetch('/api/log', { method: 'POST', body: JSON.stringify({ count }) });

  // ❌ 渲染期间修改外部变量
  externalCounter++;

  return <div>{count}</div>;
}

// ✅ 副作用必须放 useEffect
useEffect(() => {
  fetch('/api/log', { method: 'POST', body: JSON.stringify({ count }) });
}, [count]);

规避：写组件时问自己"这一行代码每次渲染跑都没问题吗"。Strict Mode 双重渲染就是用来检测这种违规的。详见 B7 useEffect（编写中）。
