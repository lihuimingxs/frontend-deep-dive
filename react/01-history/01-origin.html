A1 · 起源与演化 ——「为什么存在？」

React 不是 2013 年突然出现的——它是 Facebook 内部从 2010 年开始的一系列工程实践 + 一个被忽视已久的"chat 通知 bug"驱动的重写。从 2013 年 5 月公开发布到今天，React 经历了 4 个关键阶段：2013-2015 公开发布与 JSX 争议 → 2016-2018 Fiber 架构重写（创世事件） → 2019-2021 Hooks 革命 → 2022-至今 Concurrent / Server Components / Actions。本章是 React 主题的叙事地基——读完后所有"为什么 React 这样设计"的问题都有时间线上的答案。

阅读时长 ~30 分钟 前置 JS / 任意框架经验 权威来源 React Conf history · React RFCs · Andrew Clark / Dan Abramov 的 blog 与 Twitter

01 速览

三句话定义本章：

  1. React 出生于 Facebook 内部痛点：2011 年 Jordan Walke 用 PHP 的 XHP 思想原型出"声明式 UI"，最初代号 FaxJS。2013 年 5 月在 JSConf 公开发布——当时 JSX 引发"HTML 写在 JS 里？这不是反模式吗" 的巨大争议，但 Facebook 的实战已证明可行。
  2. Fiber 重写（2016 React 16.0）是 React 的"创世事件"。2016 之前的 React 是不可中断的递归渲染，性能瓶颈无法突破。Fiber 把 reconciliation 改成可中断的链表遍历——所有后续特性（Concurrent / Suspense / Server Components / Actions）都依赖这次重写，没有它就没有今天的 React。
  3. Hooks（2019 React 16.8）和 Server Components（2024 React 19）是两次"编写范式" 转变。Hooks 让函数组件能拥有状态，把 React 从"类的世界"翻到"纯函数 + 副作用容器"；RSC 进一步把 React 从"客户端渲染"扩到"服务器 + 客户端混合"——这是 13 年里最大的两次重构。

02 底层逻辑

史前史（2010-2013）—— 一个 chat bug 引发的革命

2010 年 Facebook 的 News Feed 团队遇到一个反复出现的 bug：通知栏的未读消息数和聊天面板里的实际状态经常不一致——用户看到"3 条新消息"，点开发现只有 2 条；或者反过来。原因是 jQuery 风格的"多个事件处理器各自更新 DOM"模式——每个处理器只知道自己的那一小块，没人有"整体状态" 的视角。状态散布在 DOM 各处，永远会不一致。

Facebook 的 Tom Occhino / Jordan Walke 受 PHP 的 XHP（Facebook 自家发明的 PHP 扩展，让 PHP 内能写 XML 字面量）启发，开始思考"能不能用同样思路在 JS 里声明 UI"。Walke 在 2011 年写出 FaxJS 原型——这就是后来的 React。核心思想：每次状态变化时重新生成完整的 UI 描述，让框架决定怎么更新 DOM——开发者再也不用手动同步 DOM 和数据。

这个想法当时极其反直觉。"每次状态变了就重渲染一切"——大部分人第一反应是"这不慢死"。但 Facebook 内部用 React 重写 News Feed 后，发现：① 维护成本断崖式下降；② 因为 React 用 virtual DOM 比对差异，实际只更新需要变的 DOM——性能和手写差不多。这两点验证了 React 的核心赌注。

第一阶段 · 2013-2015 · JSX 争议与 React Native

2013-05 JSConf 公开发布（version 0.3.0）。Jordan Walke + Pete Hunt 在 JSConf US 演讲——观众反应"很迷惑"。当时 Backbone.js / Ember.js / AngularJS（1.x）是主流，它们都用 HTML 模板 + 双向数据绑定。React 完全反过来：JSX 把 HTML 写进 JS + 单向数据流。社区第一反应：

  • "HTML 写在 JS 里？这违反 separation of concerns"（最常见的批评）
  • "每次渲染整个 UI？这一定很慢"
  • "没有双向数据绑定？写表单要写多少代码"

但 Facebook 的实战说服力很强——Pete Hunt 在 2013 年的著名演讲 "Rethinking Best Practices" 中论证："JSX 不是把 HTML 塞进 JS，是把渲染逻辑统一到一个地方"。把 UI 逻辑（数据 → 节点结构）和数据逻辑视为一体——他们都是 view layer 的 concern。这个论证缓慢但持续地说服了社区。

2014 年 React Native 出现——证明 React 的核心抽象不只能渲染 DOM。同年 Facebook 发布 Flux 模式（Action / Dispatcher / Store / View）—— 首个明确的 React 状态管理范式。2015 年 Dan Abramov 写了 Redux（Flux 的简化 + 函数式版本），迅速成为社区事实标准。

2015-03 React 0.13 引入 ES2015 class component——开始可以写 class App extends React.Component。之前用 React.createClass({ render() {...} })。这次改动让 React 进入"面向 class 的时代"，持续到 2019。

第二阶段 · 2016-2018 · Fiber 重写（创世事件）

2015 年起 Andrew Clark + Sebastian Markbåge + 团队开始秘密重写 React 内核。原因：React 15 之前的 reconciliation 算法是不可中断的递归——一旦开始渲染一棵树就必须跑完。这意味着：

  • 大列表渲染会阻塞主线程几百毫秒，期间 UI 完全无响应（输入卡顿、动画卡顿）
  • 无法实现"优先渲染重要的部分，次要部分稍后"
  • 无法做"异步等数据加载"（Suspense 的前置）

解决方案是Fiber 架构——把递归改成可中断的链表遍历。每个 React 元素对应一个 Fiber 节点（含 child / sibling / return 指针），整棵 Fiber 树本质是个链表。React 可以在任意 Fiber 暂停，让浏览器响应输入 / 渲染 / 动画，然后从断点继续。详细机制见 A3 Fiber 架构（编写中）。

2017-09 React 16.0 发布——Fiber 上线。这是 React 历史上最重要的版本，因为：

  • 所有后续重大特性都依赖 Fiber：Concurrent rendering（18）、Suspense for data fetching（18）、Server Components（19）、Actions（19）—— 没有 Fiber 这些都做不出
  • 同时不破坏向后兼容：用户代码 99% 不需要改
  • Bundle size 缩小 ~30%

React 16 还引入：Error Boundaries（捕获组件树错误）、Fragments（<> 不需要包裹 div）、Portals（渲染到任意 DOM 位置）、自定义 DOM 属性透传。

2018-10 React 16.6 发布 Suspense + lazy + memo。Suspense 当时仅支持代码分割（React.lazy），但 API 形态已确定——data fetching 集成被推迟到 React 18+。React.memo 让函数组件也能浅比较 props 跳过渲染（之前只有 class 的 shouldComponentUpdate / PureComponent）。

第三阶段 · 2019-2021 · Hooks 革命

2018-10 React Conf，Sophie Alpert + Dan Abramov 联合发布 Hooks proposal——观众反应是"这是 React 历史上最大的 API 变化"。Hooks 解决的是 class component 的几个长期痛点：

  • 状态逻辑难复用：HOC（Higher-Order Component）和 render props 让组件树嵌套过深（"wrapper hell"）
  • 生命周期方法逻辑分散：componentDidMount / componentDidUpdate / componentWillUnmount 各自处理一部分订阅 / 取消订阅，逻辑被切散
  • this 难懂：class 方法的 this 经常丢失（详见 JS this 四规则）
  • class 和 minifier 不友好：class 和函数体积差距大、tree-shaking 不彻底

Hooks 的解：函数组件 + Hooks API（useState / useEffect / useContext / 自定义 Hooks）。函数组件之前已经存在但只能"纯展示"——没有状态、没有生命周期。Hooks 让函数组件等价表达 class component 能做的所有事，而且：

  • 状态逻辑通过自定义 Hook 复用：useFetch / useDebounce 等模式取代 HOC
  • 相关逻辑组织在一起：订阅 + 取消订阅写在同一个 useEffect 里，而非分散到 3 个生命周期
  • 无 this：函数组件没有 this，所有变量都是普通 JS 闭包

2019-02 React 16.8 正式发布 Hooks——立即改变 React 写法。从此点起，新 React 代码 95% 是函数组件 + Hooks。class component 仍支持但成"历史代码"。

2020-10 React 17 发布—— 这是 React 历史上罕见的"无新特性版本"。核心目的是"渐进升级"基础设施：

  • 让"同一页面运行多个 React 版本"成为可能（事件系统改用根节点委托而非 document）
  • 新 JSX runtime—— 不再要 import React from 'react'，jsx-runtime 自动 import
  • 移除 componentWillMount / componentWillReceiveProps 等不安全的生命周期

17 看似"什么都没做"，但为 18 的 Concurrent rendering 铺平道路—— 没有 17 的渐进升级机制，18 的 breaking change 会让大型项目无法升级。

第四阶段 · 2022-至今 · Concurrent / Server Components / Actions

2022-03 React 18.0 发布—— Fiber 架构在 16 上线后，5 年的"等待时机"终于产出真正的 Concurrent rendering：

  • useTransition / useDeferredValue—— 把更新标为 transition（低优先级、可被打断）
  • 自动 batching—— 一个 event handler / Promise / setTimeout 中的多个 setState 自动合并（之前仅事件回调里合并）
  • 新 SSR API：renderToPipeableStream / renderToReadableStream 支持流式 SSR + Suspense
  • Strict Mode 双重渲染—— 帮助检测副作用（开发环境，生产无影响）
  • Suspense 数据 fetching 集成（experimental，要等 19 RSC 才稳定）

React 18 是真正释放 Fiber 潜力的版本。但同时它也是 React 历史上"最难升级" 的版本之一—— Concurrent 的语义影响 useEffect 双重触发、StrictMode 行为变化等，许多老代码升级后行为微妙差异。

2024-04 React 19 RC 发布 → 2024-12 正式发布—— 19 是第二次大型范式扩展（第一次是 Hooks）。核心特性：

  • Server Components 稳定—— 'use client' / 'use server' 指令、跨边界 props 序列化
  • Actions—— useActionState / useFormStatus / useOptimistic / form action prop
  • use API—— 在 render 中读 Promise（搭配 Suspense）/ 读 Context（替代 useContext 在某些场景）
  • ref as prop—— function Comp({ ref }) 直接接收 ref，不再需要 forwardRef
  • document metadata—— 在组件里直接写 <title> / <meta>，自动 hoist 到 head
  • Asset preloading API—— preload / preinit / preinitModule

三个高密度洞察

① Fiber 重写是 React 的"创世事件"

所有后续重大特性都依赖 Fiber：Concurrent rendering（时间切片需要可中断渲染）、Suspense（throw promise 需要 Fiber 的 unwind 机制）、Hooks（hooks linked list 存在 Fiber 节点上）、Server Components（流式 reconciliation 需要 Fiber）、Actions（异步 state 更新需要 lane 模型）。没有 2017 年的 Fiber 重写，今天的 React 不存在。这是 React 工程史上最有"杠杆效应"的一次重构。

② React 的特性发布节奏远慢于 Vue / Svelte

Vue 3（2020-09）→ Vue 3.4（2024）大版本周期 ~4 年；Svelte 4（2023-06）→ Svelte 5（2024-10）~16 个月。React 16 → 17 用了 3 年，17 → 18 用了 1.5 年，18 → 19 用了 2.5 年。原因是 Facebook 内部使用规模太大（Instagram + WhatsApp + 工作平台 + 数十亿用户产品），任何 breaking change 必须有渐进升级路径——不能一次性 break 所有用户代码。这就是为什么 React 17 是"无新特性版本"，纯铺路。

③ 每次"编写范式"转变都伴随长期社区分歧

React 历史上有 3 次"怎么写 React"的范式转变，每次都引发持续 1-2 年的社区争议：

  • 2015 ES2015 class：React.createClass 用户抱怨"this 太多 / 装饰器不友好"
  • 2019 Hooks：class 用户抱怨"函数组件每次重新创建函数性能差 / 闭包陷阱难调试"
  • 2024 Server Components：客户端 React 用户抱怨"'use client' 边界不直观 / RSC 心智完全不同"

每次最终都被社区接受，但过渡期都有 12-24 个月的混乱。学 React 19 的 RSC 现在大致处于这种"过渡期混乱"——但根据历史经验，2026-2027 应该会成为新主流。

03 应用场景

场景 A · 老 React 项目升级（16 → 19）路径

一个 5 年前的 React 16.x 项目，class components 主导 + Redux + react-router v5。要升级到 19。分阶段升级：

  1. 16 → 17（最简单）：依赖更新 + 移除 unsafe lifecycle + 切新 JSX runtime（不再 import React）。一周内完成
  2. 17 → 18（重点）：StrictMode 双重渲染暴露副作用问题、useEffect 行为微调、SSR API 改变。可能需要修复几十个 effect 中的副作用 bug
  3. class → 函数 + Hooks 改写（最大工程量）：增量改写。新代码全函数，老 class 保留
  4. 18 → 19：移除 forwardRef / defaultProps / 字符串 ref 等已退役 API。19 整体兼容 18 代码

关键：不要"原地大改"，逐 component 增量。React 17 的渐进升级基础设施允许同一页面跑多版本——大型项目几乎必经此路。

场景 B · 给 Vue / Angular 工程师讲 React

最有效的切入点是"JSX 设计动机 + UI = f(state) 哲学"，而不是从 API 开始：

  • "为什么 JSX 不是模板"→ 因为 JSX 是 JS 表达式，所以能在任意位置组合（map / 三元 / 函数返回值），这是 Composability 的根
  • "为什么不是双向绑定"→ 因为单向数据流让状态传播路径可预测（detail 见 A2 设计哲学）
  • "为什么用 className 不用 class"→ 因为 JSX 转换成 JS 对象时 class 是关键字（这是 JS 限制，不是 React 设计选择）

把"React 怪"重新框定为"不同的设计哲学"，团队接受度高很多。详细对比见 A2 章。

场景 C · 评估 Vue / Svelte / Solid 时的横向对比锚点

团队在 React 19 / Vue 3.4 / Svelte 5 / Solid 1.x 之间选型——用 React 的 4 个维度做对比锚点：

维度React 19Vue 3.4Svelte 5Solid 1.x
响应式机制不可变 + 重渲染（Fiber 比对）Proxy 反应式编译时反应式（Runes）signals（细颗粒度）
JSX / 模板JSX（JS 表达式）SFC 模板（HTML 风格）SFC 模板JSX
组件运行机制每次状态变化重新调用函数（Fiber 比对）setup() 调用一次，依赖追踪编译时优化为 imperativesetup 调用一次，signal 订阅
SSR / RSCRSC（服务端组件）SSR + PiniaSvelteKit SSRSolidStart SSR
生态规模最大大（中国主流）中小但精致

选型策略：① 大团队 / 长期项目 / 招聘考虑 → React；② 中国市场 / 渐进式迁移老项目 → Vue；③ 小团队 / 性能优先 → Svelte 或 Solid。

04 典型代码

看一个组件穿越 11 年

同样一个"计数器 + 异步加载初始值 + cleanup"组件，在 React 各阶段的写法演化：

React 0.13 · 2015 · createClass + ES5 
// React.createClass + ES5 mixin
var Counter = React.createClass({
  getInitialState: function () { return { count: 0, loaded: false }; },

  componentDidMount: function () {
    var self = this;
    fetch('/api/initial-count')
      .then(function (r) { return r.json(); })
      .then(function (data) {
        self.setState({ count: data.count, loaded: true });
      });
  },

  componentWillUnmount: function () {
    // 没有 AbortController，只能希望组件还活着时 fetch 已完成
  },

  render: function () {
    if (!this.state.loaded) return React.createElement('div', null, 'Loading...');
    return React.createElement('button', {
      onClick: function () { /* 这里 this 丢了，要 bind */ }
    }, 'Count: ' + this.state.count);
  },
});
React 16.0 · 2017 · ES2015 class + Fiber 时代 
class Counter extends React.Component {
  state = { count: 0, loaded: false };  // class fields
  abortController = new AbortController();

  componentDidMount() {
    fetch('/api/initial-count', { signal: this.abortController.signal })
      .then(r => r.json())
      .then(data => this.setState({ count: data.count, loaded: true }))
      .catch(err => { if (err.name !== 'AbortError') throw err; });
  }

  componentWillUnmount() {
    this.abortController.abort();
  }

  // 用箭头方法绑定 this
  handleClick = () => this.setState(s => ({ count: s.count + 1 }));

  render() {
    if (!this.state.loaded) return <div>Loading...</div>;
    return <button onClick={this.handleClick}>Count: {this.state.count}</button>;
  }
}
React 16.8 · 2019 · Hooks 时代 
function Counter() {
  const [count, setCount] = useState(0);
  const [loaded, setLoaded] = useState(false);

  useEffect(() => {
    const controller = new AbortController();
    fetch('/api/initial-count', { signal: controller.signal })
      .then(r => r.json())
      .then(data => { setCount(data.count); setLoaded(true); })
      .catch(err => { if (err.name !== 'AbortError') throw err; });

    return () => controller.abort();   // cleanup 在同一个 effect 里
  }, []);

  if (!loaded) return <div>Loading...</div>;
  return <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>;
}
React 19 · 2024 · Server Component + use API 
// counter.tsx —— Server Component（默认）
async function Counter() {
  // 直接在服务器读初始值，不需要 useEffect / useState 处理 loading
  const data = await fetch('/api/initial-count').then(r => r.json());
  return <CounterClient initialCount={data.count} />;
}

// counter-client.tsx
'use client';
function CounterClient({ initialCount }) {
  const [count, setCount] = useState(initialCount);
  // 不再有 loading 状态、不再有 cleanup —— 服务器已处理
  return <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>;
}

关键观察：① 代码量随版本下降；② 副作用从"分散到生命周期"集中到"同一个 useEffect"；③ React 19 把"初始数据加载" 完全移出客户端——这是 13 年演化的"简化" 主线。

05 常见陷阱

⚠️ 陷阱 1：把 React 16 / 17 / 18 / 19 当语言版本

React 主版本 ~2 年一个，但每次都有 breaking change。不是 ES 那种"每年小幅添加"模式，而是"2 年一次大重构"。

规避：升级要分阶段，先看 changelog 中的 breaking changes。React 团队会提供 codemod 工具（自动迁移代码），务必用。npm: npx react-codemod。

⚠️ 陷阱 2：Class component 仍未"deprecated"

React 团队没有官方宣布弃用 class——大量老代码、第三方库（如 react-error-boundary）仍依赖。新代码用 Hooks，但读 / 维护 class 代码仍是日常。

规避：① 不主动写 class component（除 Error Boundary）；② 但要能读 class 代码（理解 lifecycle、this binding、HOC 模式）。

⚠️ 陷阱 3：把 "Hooks" 和 "函数组件" 混为一谈

函数组件早就有（React 0.x 时代），但只能"纯展示"——没有状态、没有生命周期。Hooks 是 16.8 加的给函数组件加状态的运行时机制，依赖 Fiber 的 hooks linked list。两者出现时间相隔 5 年。

规避：理解 Hooks 不是"新写法"，是"函数组件能拥有状态" 的运行时机制。详细见 A3 Fiber 架构 的 hooks linked list 部分（编写中）。

⚠️ 陷阱 4：以为 React 19 的 Server Components 是 SSR 的升级

RSC 不是 SSR。SSR 是"把 React 在服务器跑一遍生成 HTML 字符串"，结果仍是客户端 React app。RSC 是"组件分两种：永远在服务器、永远在客户端"，是架构层面的转变。

规避：学 RSC 时放弃 SSR 心智，从"服务器组件 vs 客户端组件 各自能 / 不能做什么"重新建立。详细见 B16 Server Components（编写中）。
