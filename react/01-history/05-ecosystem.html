A5 · 生态与工程化

A1-A4 讲了 React 的"历史"和"引擎"。本章是 Track A 收官——把 React 19 时代的主流生态系统化梳理一遍：元框架 / 状态管理 / 数据 fetching / 路由 / 表单 / 测试 / UI 库 共 6 大类。本章不教你怎么用每个库（那是各库官网的事），是给一份"何时选什么" 的决策框架。读完后你能在新项目启动时 30 分钟内拍板技术栈，且每个选择有可解释的理由。

📐 本章的"速查 + 决策树" 角色

前 4 章是"深度"（理解 React 内部）。本章是"广度"（看清 2026 年的生态全图）。两者互补——理解了 A2 设计哲学 + A3 Fiber 之后，你判断每个生态库"是否符合 React 哲学 + 是否压榨了 React 能力" 会快得多。

阅读时长 ~25 分钟 前置 A1 起源 推荐先看（理解 React 19 + RSC 的位置） 权威来源 react.dev · Start a New Project + 各库官网

01 速览

三句话定义本章：

  1. "选什么生态" 决策不是技术问题，是产品 + 团队问题。SEO 重要 → 元框架；SPA 已有后端 → Vite + React；toC 大众产品 → React Router v7；内部工具 → 任何选择都行。没有放之四海皆准的"最佳栈"，只有匹配场景的栈。
  2. 2025-2026 的大势：① 客户端复杂度下降（更多搬到服务器，靠 RSC + Server Actions）② 端到端类型安全（tRPC / Server Actions / Drizzle 让 TS 类型从 DB schema 到 React Component 全打通）③ shadcn/ui "复制源码" 模式取代传统组件库 ④ shadcn 模式延伸到其他库（TanStack Form 等也开始用）。
  3. Stage 4 提案进规范 ≠ 应该用（详见 ECMA A5 现代生态）—— 同理，"新出的 React 库" ≠ 应该用。生态选型保守优于激进——选 2+ 年稳定 + 大用户量 + 持续维护的库。

6 大类生态速查：

类别 事实标准 主流替代
元框架 Next.js App Router React Router v7 / TanStack Start / Vite + React
状态管理 Zustand（最简） Redux Toolkit / Jotai / MobX / XState
数据 fetching TanStack Query SWR / Apollo Client / tRPC / RSC（19+）

02 底层逻辑

元框架（Meta-frameworks）

"元框架" 不是"React 的附加功能"，是"定义 React 的运行环境"—— 它决定何时 RSC、何时 'use client'、何时 Static / Dynamic、用什么路由 / 数据 fetching / 部署模式。选元框架就是选 React 19 的运行模型。

元框架 阶段 定位与适用
Next.js App Router 事实标准 RSC + Actions 一等公民、文件路由、ISR、Edge runtime、Vercel 深度集成。推荐场景：SEO 关键 / 全栈应用 / 服务端数据加载主导 / 团队接受 Vercel 锁定
React Router v7 主流 Remix 与 React Router 合并后专注 SPA + SSR / "fullstack 框架"。推荐场景：toC 大众产品 / 不要 Vercel 锁定 / 团队有 Cloudflare / 自家服务器
TanStack Start 新兴 类型友好（路由 / loader / action 自动类型推导）。推荐场景：类型安全极致 / 内部工具 / 团队接受新选手 + 持续跟进风险
Vite + React SPA 默认 不要元框架的"纯 React"。推荐场景：SPA / 已有后端（如 Spring Boot / Django）/ 不需要 SSR / RSC / 内部 dashboard

关键决策点：

  • 需要 SEO？ 是 → Next.js / React Router v7（含 SSR）；否 → Vite + React 足矣
  • 需要 RSC？ 是 → Next.js（成熟）/ Waku（实验）；否 → 任何选择
  • 怕被 Vercel 锁定？ 是 → React Router v7 / TanStack Start；否 → Next.js
  • 纯客户端 SPA？ Vite + React + React Router v7（用作 SPA 路由）

状态管理

状态管理选择本质是"状态生命周期" 问题。不是"哪个库最好"，是"这个状态属于哪个层级"——选择应匹配状态范围：

状态范围 推荐方案 理由
组件级 useState / useReducer React 自带，无依赖，覆盖 80% 场景
跨少数组件 提升 state + props（"状态提升"） 不引库的简单方案，< 5 层组件树
跨多层组件 / 全局 Context API + useReducer 或 Zustand Context 适合"少更新" 状态（theme / locale），Zustand 适合"频繁更新"（避免 Context 全部重渲染）
大型应用 + 复杂业务 Redux Toolkit / Jotai Redux Toolkit 适合"事件溯源" + DevTools time-travel；Jotai 适合"原子组合" 风格
状态机 / 向导流程 XState 结账 / 表单向导 / 游戏状态——任何"状态 + 转换" 模型
服务器状态（缓存） TanStack Query / RSC 看下一节"数据 fetching"

5 个主要客户端状态库快速对比：

  • Zustand（推荐入门）：~1KB，无 provider，store 直接 import 用，API 极简（create((set) => ({...}))）。Vue 工程师过来最容易上手
  • Jotai：原子化—— 每个 state 是一个 atom，组件订阅 atom。读时才订阅，避免 Zustand "整个 store 订阅" 的全订阅模式。适合"state 互不相关 + 想精细控制重渲染"
  • Redux Toolkit（RTK）：传统选择 + 现代化。RTK Query 集成数据 fetching。适合"需要 time-travel 调试" / "团队已有 Redux 经验" / "事件溯源风格"
  • MobX：响应式（Vue 风格）—— 直接 mutate state 自动触发更新。违反 React 哲学（A2 讲的不可变 + 重渲染），少数项目用
  • Valtio：和 MobX 类似但更轻量（基于 Proxy）。同样违反 React 哲学
  • XState：状态机库——不是通用状态管理，是"有限状态 + 转换规则" 的专门工具。结账流程 / 多步表单 / 游戏 / WebRTC 协商等用

数据 fetching

2025 年最大的转变：React 19 + RSC 让服务端组件直接 async + await（A4 已讲），传统的"useEffect + useState 加载数据" 模式在很多场景被取代。但客户端数据加载仍需要库——哪个？

方案 定位 推荐场景
TanStack Query 事实标准 客户端复杂状态：缓存 / 后台刷新 / 乐观更新 / SSR 集成 / DevTools。不用 RSC 时首选
SWR（Vercel） 轻量替代 简单缓存 + revalidate-on-focus，API 比 TanStack Query 简单 50%。适合中小项目
Apollo Client GraphQL 专属 用 GraphQL + 想要 normalized cache + optimistic mutations
tRPC 类型安全 RPC 端到端 TypeScript（Node 后端 + React 前端共享 schema），不用 GraphQL，但要全栈 TS
RSC（React 19） 新模式 服务端组件直接 fetch，无客户端状态。用 Next.js / Waku 时首选，能减少客户端 JS 体积

决策核心：用 RSC 还是客户端 fetch？

  • 用 RSC（推荐）：用 Next.js / Waku 时，初始数据加载 走 RSC（async function Page() { const data = await fetch(...) }）—— 减少客户端 JS、SEO 友好、首屏快
  • 用 TanStack Query：客户端动作触发的 fetch（如点赞 / 添加评论 / 搜索过滤）—— 需要本地缓存 + 乐观更新 + 后台刷新等能力。RSC 不擅长这些
  • 组合用：RSC 加载初始数据 + TanStack Query 处理后续交互—— 是 Next.js + RSC 的主流模式

路由 / 表单 / 测试 / UI 库（速查）

路由

  • React Router v7（事实标准）：嵌套路由、loaders、actions、SSR。Remix 合并后的版本
  • TanStack Router：路由参数自动类型推导（路径 /user/$id 中 $id 类型自动 string）。类型安全极致派
  • 元框架内置（Next.js / Remix）：用文件路由替代独立路由库。约定优于配置

表单

  • React Hook Form（事实标准）：最少重渲染（用 ref + uncontrolled）+ 简单 API。和 zod / valibot 验证库完美集成
  • TanStack Form（新兴）：类型友好—— form values 字段类型自动推导。和 React Hook Form 竞争中
  • Formik（传统选择）：仍维护但社区在迁移到 RHF
  • React 19 Actions（A4 讲过）：取代部分简单表单场景。复杂表单仍用 RHF

测试

  • Vitest（推荐）+ React Testing Library：单元 / 集成测试事实标准。Vitest 替代 Jest（启动快 + 原生 ESM + Vite 集成）
  • Playwright（推荐 E2E）：跨浏览器、API 直观。取代 Cypress—— Cypress 仍可用但 2024 后明显 momentum 转向 Playwright
  • Storybook：组件 isolation 开发 + 视觉测试。Chromatic 集成自动 visual regression

UI 库 / 样式（重点：shadcn/ui 模式）

  • shadcn/ui（推荐）："复制源码" 模式—— 不是 npm 包，用 CLI 把组件源码复制到你的项目里，你完全 own 这些代码。基于 Radix UI primitives + Tailwind CSS。2024-2025 占据 React UI 库主导地位，因为它解决了"组件库被作者绑死 / 升级困难 / 改样式难" 几个长期痛点
  • Radix UI：headless primitives—— 提供逻辑（如 dropdown / dialog / tooltip 的可访问性 + 焦点管理 + 键盘交互），样式留给你。shadcn/ui 内部就是基于 Radix
  • Material UI / Mantine / Chakra：传统组件库。仍可用但相对 shadcn 已显得"沉重"。如果团队已有依赖、不想换栈，继续用是合理的
  • Tailwind CSS：原子 CSS 事实标准。深度学习见 独立 Tailwind 主题（规划中）。和 shadcn/ui 是天然搭配
  • 纯 CSS Modules / styled-components：仍是合法选择，只是 momentum 大幅下降。新项目除非有特殊需求，建议 Tailwind

决策框架（"30 分钟新项目选型"）

按以下顺序回答 5 个问题，能快速锁定技术栈：

  1. SEO 重要吗？
    • 是 → Next.js App Router 或 React Router v7（必须 SSR）
    • 否 → Vite + React（SPA 模式）
  2. 用 React 19 RSC 吗？
    • 是 → Next.js（事实标准）或 Waku（实验）
    • 否 → 任何选择，但失去 RSC 减少客户端 JS 的好处
  3. 状态复杂度？
    • 简单（< 10 个全局状态）→ Context + useReducer 或 Zustand
    • 中等（频繁更新 + 跨多层组件）→ Zustand 或 Jotai
    • 复杂（事件溯源 / time-travel）→ Redux Toolkit
    • 状态机 / 向导 → XState
  4. 数据 fetching？
    • 用 RSC → 初始 RSC + 后续 TanStack Query
    • 不用 RSC → TanStack Query（推荐）或 SWR（轻量替代）
    • GraphQL → Apollo Client
    • 全栈 TS → tRPC
  5. UI 库？
    • 新项目 + 自由度 → shadcn/ui + Tailwind CSS
    • 团队已有 antd / MUI 经验 + 不想换 → 继续用
    • 极简 / 自建 → Radix UI primitives + 自家 CSS

2025-2026 年的大势

3 个明显的方向变化：

① 客户端复杂度下降

越来越多状态搬到服务器（RSC + Server Actions + Drizzle / Prisma）。客户端只处理"纯交互状态"（input value / dropdown open / animation）。结果：

  • 客户端 JS bundle 缩小 30-50%
  • SEO 自动友好（HTML 直接发，不用 hydrate 全部）
  • 状态管理库使用减少（很多过去用 Redux 的状态现在直接 server-side）

② 端到端类型安全

tRPC / Server Actions / Drizzle 让 TS 类型从 DB schema 到 React Component 全程打通：

  • DB schema（Drizzle）→ TS 类型自动生成
  • Server Action 函数 → 类型自动跨边界传到 Client
  • React Component props → 类型来自 Server Action 返回值
  • 结果：改 DB column 名 → 整条链路自动报错

③ shadcn 模式延伸

"复制源码 over npm 安装" 模式从 UI 库扩展到其他类别：TanStack Form 也开始用 shadcn 风格的 CLI 复制；Origin UI、Park UI 等新 UI 库都是 shadcn 模式；甚至 utility 库（如 zod 验证 schema 模板）也开始用复制方式分发。这是 React 生态对"npm 升级地狱" 的回应。

03 应用场景

场景 A · 新项目（toC SaaS 产品）选型

团队 5 人，目标"产品管理 SaaS"，多用户、SEO 重要、复杂表单 + dashboard。决策树走起：

  • 元框架：SEO 重要 + 想用 RSC → Next.js App Router ✅
  • 状态管理：复杂 dashboard 状态 + 频繁更新 → Zustand（够用，避免 Redux 复杂度）✅
  • 数据 fetching：初始数据 RSC + 后续交互 TanStack Query ✅
  • 路由：Next.js App Router 内置 ✅
  • 表单：复杂表单 → React Hook Form + zod 验证 ✅
  • UI 库：自由度 + 速度 → shadcn/ui + Tailwind CSS ✅
  • 测试：Vitest + React Testing Library + Playwright（E2E）✅
  • 部署：Vercel（Next.js 原生支持）或 Cloudflare Workers

选型时间 ~30 分钟，每个选择有可解释理由。这就是决策框架的价值——不浪费时间在"X vs Y vs Z 哪个最好" 的无尽辩论。

场景 B · 老项目（Next.js Pages Router）→ App Router 升级

3 年前的 Next.js 12 项目，Pages Router + React Hook Form + Redux + SWR。要升级到 Next.js 15 + App Router + RSC。分阶段：

  1. 升级 Next.js 版本（1 周）：12 → 13 → 14 → 15。每步看 codemod。暂不切 App Router—— Pages Router 在 15 仍可用
  2. 添加 App Router 共存（2 周）：新页面用 App Router（app/ 目录），老页面留在 pages/。两个 router 可同时存在
  3. 逐页面迁移（数月）：从最简单的页面开始，每周迁移 2-3 个。每次迁移要：① 拆 'use client' 边界 ② SWR fetch 改 RSC async ③ 评估状态是否能搬到服务器
  4. 评估 Redux 是否还需要：很多状态搬到服务器后，Redux 用例减少。剩余的可考虑换 Zustand（更轻量）
  5. 移除 Pages Router：所有页面迁完后

关键：不要原地大改—— 共存 + 渐进迁移。Next.js 15 仍支持 Pages Router 是这种迁移的明确允许。详见 Next.js 官方迁移指南。

场景 C · 评估"是否换状态管理库"

项目用 Redux 3 年，最近团队抱怨"boilerplate 多 / 学习曲线陡"。要换吗？

  • 不换的理由：团队熟悉 / DevTools time-travel 救过命 / 已有 100+ slice 不想重写
  • 换 Zustand 的理由：boilerplate 减少 90% / 不需要 Provider / 学习曲线低
  • 换 Jotai 的理由：原子化 + 精细订阅 → 性能优势 / 适合"状态互不相关" 场景
  • 换 RTK 的理由：保持 Redux 心智但 boilerplate 少很多 / RTK Query 替代 SWR

建议：不要全量换。新模块用新库 + 老模块保留—— 共存几个月，团队体验后决定。如果新库体验好，再渐进迁移老模块（每个 slice 单独迁移）。不要为换而换—— Redux 仍是合法选择。

04 典型代码

对比：3 个状态管理库的同一计数器

javascript · 同一计数器在 3 个库中的写法 
// === Zustand ===
import { create } from 'zustand';
const useCounterStore = create((set) => ({
  count: 0,
  increment: () => set((s) => ({ count: s.count + 1 })),
}));
function Counter() {
  const { count, increment } = useCounterStore();
  return <button onClick={increment}>{count}</button>;
}
// 优势：~10 行，直接 import 用，无 provider

// === Jotai ===
import { atom, useAtom } from 'jotai';
const countAtom = atom(0);
function Counter() {
  const [count, setCount] = useAtom(countAtom);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
// 优势：原子化，每个 atom 独立订阅；适合"多个互不相关 state"

// === Redux Toolkit ===
import { createSlice, configureStore } from '@reduxjs/toolkit';
import { Provider, useDispatch, useSelector } from 'react-redux';

const counterSlice = createSlice({
  name: 'counter',
  initialState: { count: 0 },
  reducers: { increment: (s) => { s.count++ } },  // 用 Immer 自动产生新引用
});
const store = configureStore({ reducer: { counter: counterSlice.reducer } });
function Counter() {
  const count = useSelector(s => s.counter.count);
  const dispatch = useDispatch();
  return <button onClick={() => dispatch(counterSlice.actions.increment())}>{count}</button>;
}
// + 必须包 <Provider store={store}> 在 root
// 优势：DevTools time-travel + middleware 强大；劣势：boilerplate 多

// 选择：业务逻辑简单 → Zustand；状态分散 → Jotai；要 DevTools → RTK

bash · shadcn/ui 工作流

05 常见陷阱

⚠️ 陷阱 1：过度引入库

"新项目要用 Redux + TanStack Query + React Hook Form + ..."——很多场景 useState + useReducer 已经能解决。过早引库 = 复杂度税。

规避：从 useState 开始，遇到痛点再引库——状态跨 5+ 层组件用 Zustand；服务端缓存复杂用 TanStack Query；表单复杂用 React Hook Form。"痛点驱动" 优于"预先架构"。

⚠️ 陷阱 2：把 RSC 当"SSR 的升级版"

RSC 不是 SSR。SSR 是"把 React 在服务器跑一遍生成 HTML"，结果仍是客户端 React app（hydrate）。RSC 是"组件分两种：永远在服务器、永远在客户端"，是架构层面转变（A1 / A4 已讲）。

规避：学 RSC 时放弃 SSR 心智，从"Server / Client 各自能 / 不能做什么" 重新建立（A4 已给完整对照表）。详见 B16 Server Components（编写中）。

⚠️ 陷阱 3：shadcn 复制源码后忘了"自己 own 它"

shadcn 的卖点是"你 own 这些代码"——但很多团队复制后把它当 npm 包看待，遇到 bug 等 shadcn 修复，遇到样式不合适抱怨"怎么改"。

规避：① 团队 onboarding 时明确说 components/ui/ 是你的代码 ② bug / 样式问题直接改这些文件 ③ shadcn 升级时看 diff 决定（不是无脑覆盖）④ 项目自己加的组件也放这个目录，统一风格

⚠️ 陷阱 4：用 Redux 处理服务端状态

老项目把"API 数据缓存" 也放 Redux（state.users.list、state.posts.byId）—— 自己手写 cache invalidation / loading / error 状态。这是 TanStack Query 存在的原因。

规避：明确区分客户端状态（UI 状态、用户操作中的临时状态）vs 服务端状态（API 数据缓存）。客户端状态用 Zustand / Redux；服务端状态用 TanStack Query / RSC。这两类完全不同的需求，混着处理代码会膨胀 5x。

⚠️ 陷阱 5：跟风用最新最潮的库

"X 库刚出，类型安全更好，我们换吧"—— 然后 6 个月后 X 库不维护了，团队要再换。新库往往2 年内死亡率 50%+。

规避：选 2+ 年稳定 + 大用户量 + 持续维护的库。看 GitHub stars / 周下载量 / 最近 6 个月 commit 频率 / 主要维护者是否还在。事实标准 ≠ 最新—— Zustand 5 年了仍是事实标准就是因为稳定。
