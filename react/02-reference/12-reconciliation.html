B12 · Reconciliation 用户视角

A3 详述了 Fiber 内部 reconciliation diff 算法。本章从用户视角讲——你能在工程中感知到的 reconciliation 行为：什么时候组件被复用、什么时候被销毁重建、key 起到什么作用、React.memo 真正能跳过什么、什么情况让 memo 失效。读完后你能精确解释"为什么我的 input 焦点丢了"、"为什么 React.memo 没用"。

📐 Part 3 起点：从用户视角看引擎

Part 3 的 4 章（B12-B15）都是"把 A3 / A4 讲过的引擎机制翻译为用户能感知的工程行为"。Reconciliation（B12）/ Suspense（B13）/ Concurrent（B14）/ Error Boundary（B15）—— 用户视角的实战指南。

阅读时长 ~25 分钟 前置 B5 列表 + key + A3 Fiber（推荐） 对应 react.dev Preserving and Resetting State · memo

01 速览

三句话定义本章：

  1. Reconciliation 算法用户感知：相同 type + 相同 key → 复用 Fiber（保留 state / 内部 ref）；不同 type 或不同 key → 销毁旧、创建新（state 重置、effect cleanup + remount）。这是 React 性能优化的基本规律。
  2. key 有 4 种工程用法：① 列表项稳定 ID（最常见，B5 已讲）② 用 key 强制重置组件 state（如 form key={userId} 切换用户重置表单）③ Fragment 多元素列表 ④ Animation 进出场标识。不要用 array index 当 key（除非永不重排序）。
  3. React.memo 用浅比较 props 决定是否跳过 render（同 useState 浅比较，B6）。但4 种情况让 memo 失效：① 每次新对象 / 数组 / 函数作 props ② children 是 JSX（每次新元素对象）③ 内部消费 Context 且 Context 变 ④ 内部 useState / useReducer 状态变。memo 必须配 useMemo / useCallback 用（B9 详述）。

看一段最小代码识别 reconciliation 行为：

javascript · reconciliation 三种行为 
// 行为 1：同 type → 复用（state 保留）
{showA ? <Counter /> : <Counter />}     // 都是 Counter
// 切换 showA 时 Counter 不被销毁 —— 内部 useState 的 count 保持

// 行为 2：不同 type → 销毁重建
{showCounter ? <Counter /> : <Display />}
// 切换时 Counter 被销毁、Display 被创建（或反之）—— state 重置

// 行为 3：同 type + 不同 key → 销毁重建
{users.map(user => <UserCard key={user.id} user={user} />)}
// user.id 变化的项被销毁重建（key 不同）
// user.id 不变的项被复用（key 相同）

// 行为 4：用 key 强制重置 state
<Form key={editingUserId} userId={editingUserId} />
// 切换 editingUserId 时 Form 被销毁重建 → 内部 state 全部重置
// 不需要手动写"reset" 逻辑

02 底层逻辑

Reconciliation 用户视角的精确规则

A3 已详述 React 在 beginWork 阶段的 diff 算法。从用户视角看，3 条精确规则：

情况 React 行为 用户感知
同 type + 同 key（默认 null） 复用 Fiber，更新 props state 保留 / DOM 复用 / effect 不 cleanup
不同 type 销毁旧 Fiber 子树 + 创建新 Fiber state 重置 / DOM 重建 / effect cleanup + remount
同 type + 不同 key 同上（销毁重建） 同上

关键点：React 看的是同位置（同 parent）+ 同 type + 同 key。位置变化（Fiber 在 children 数组的索引）+ key 配对，决定哪些 Fiber 是"同一个"。

javascript · 同 type 复用的微妙陷阱 
function App() {
  const [showA, setShowA] = useState(true);

  return (
    <div>
      {showA ? (
        <Counter />
      ) : (
        <Counter />
      )}
    </div>
  );
}

function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

// 切换 showA 时：
// - Counter 在两个分支都存在
// - 但 React 看的是"同位置 + 同 type"
// - 三元两边都是 Counter（同 type） + 都在 div 的第 1 个 child（同位置）
// - 所以 React 复用！
// - 内部 useState 的 count 保持

// 这种"看似切换实则复用" 是 React 设计选择 —— 让条件渲染高效
// 想强制重置？用 key：
{showA ? <Counter key="a" /> : <Counter key="b" />}  // 不同 key → 重建

key 的 4 种工程用法

用法 1：列表项稳定 ID（最常见）

B5 已详述。核心：业务 ID（todo.id / user.email）作 key，永远不用 index（除非 list 永不重排序）。

用法 2：用 key 强制重置组件 state ⭐

javascript · key 强制重置 state 
// 场景：编辑用户表单。切换不同用户时要"重置表单 state"

// ❌ 不用 key —— Form 内部 state 不重置
function App({ editingUserId }) {
  return <Form userId={editingUserId} />;
}
function Form({ userId }) {
  const [name, setName] = useState('');  // 切换 userId 时 name 不变
  // 用户输错了 'Alice' 给 user 1，切到 user 2 后 input 仍显示 'Alice'
  return <input value={name} onChange={e => setName(e.target.value)} />;
}

// ✅ 用 key={userId} —— 切换时强制重建 Form
function App({ editingUserId }) {
  return <Form key={editingUserId} userId={editingUserId} />;
  //              ↑ key 改变 → React 销毁旧 Form 创建新 Form
  //              ↑ 内部 useState 重置 / useEffect cleanup + 重新跑
}

// 同款模式：
// - Tab 切换重置 form
// - 编辑不同 todo 重置 modal 状态
// - 切换路由参数重置组件
// - 把 key 当作"组件实例 ID"

// 替代方案：手写 reset
// 用 useEffect 监听 userId 变化 + 调用 setName('') 重置
// 但要重置 N 个 state 就要写 N 次 —— key 一行解决

// 注意陷阱：key 不是 prop，组件内部访问不到
// 如果想"重置 + 知道当前 key"，要传 prop 同时给 key

用法 3：Fragment 多元素列表

javascript · Fragment 列表 
// list 中每项渲染多个相邻元素 —— 用完整 Fragment 形式
import { Fragment } from 'react';

{items.map(item => (
  <Fragment key={item.id}>
    <dt>{item.title}</dt>
    <dd>{item.body}</dd>
  </Fragment>
))}

// 短语法 <>...</> 不支持 key（B2 / B5 已讲）—— 必须用完整 Fragment

用法 4：Animation 进出场标识

javascript · 进出场动画依赖 key 
import { AnimatePresence, motion } from 'framer-motion';

function Carousel({ slides, currentIndex }) {
  return (
    <AnimatePresence mode="wait">
      <motion.div
        key={currentIndex}                      {/* key 变化触发动画 */}
        initial={{ opacity: 0, x: 100 }}
        animate={{ opacity: 1, x: 0 }}
        exit={{ opacity: 0, x: -100 }}
      >
        {slides[currentIndex]}
      </motion.div>
    </AnimatePresence>
  );
}

// AnimatePresence 通过 key 变化识别"这是同一组件的新实例"
// → 触发 exit 动画 + initial 动画
// 没 key 或 key 不变 → 没动画（因为 React 把它们当同一 Fiber 复用）

React.memo 完整签名

typescript · React.memo 类型 
function memo<P>(
  Component: React.ComponentType<P>,
  arePropsEqual?: (prevProps: P, nextProps: P) => boolean,
): React.MemoExoticComponent<React.ComponentType<P>>;

// 用法 1：默认浅比较
const MyButton = memo(function MyButton(props) {
  return <button>...</button>;
});

// 用法 2：自定义比较函数
const MyChart = memo(
  function MyChart({ data }) { return <svg>...</svg>; },
  (prev, next) => prev.data.length === next.data.length &&
                  prev.data.every((d, i) => d.id === next.data[i].id),
);
// arePropsEqual 返回 true → 跳过渲染（注意：返回 true 表示"相等" 跳过！）
// 与 shouldComponentUpdate 反过来 —— 那个返回 true 表示"应该更新" 不跳过

// 默认浅比较实现（简化）：
function shallowEqual(a: object, b: object): boolean {
  const keysA = Object.keys(a);
  const keysB = Object.keys(b);
  if (keysA.length !== keysB.length) return false;
  return keysA.every(key => Object.is(a[key], b[key]));
}
// 同 useState / useEffect / useMemo 的浅比较

memo 失效 4 种情况 ⭐

新人最常见的"memo 没用" 原因—— 不是 memo 本身的问题，是props 引用每次都变：

javascript · memo 失效 4 种典型情况 
const Child = memo(function Child({ data, onClick, items, children }) {
  console.log('Child render');
  return ...;
});

function Parent() {
  const [count, setCount] = useState(0);

  // ❌ 失效 1：每次新对象
  <Child data={{ x: 1 }} />
  // 每次 Parent render 创建新对象 → 浅比较 false → memo 失效

  // ❌ 失效 2：每次新函数
  <Child onClick={() => doSomething()} />
  // 每次 Parent render 创建新箭头函数 → memo 失效

  // ❌ 失效 3：每次新数组
  <Child items={[1, 2, 3]} />
  // 同上

  // ❌ 失效 4：children 是 JSX（每次新元素对象）
  <Child>
    <span>Hello</span>
  </Child>
  // children prop 是 React.createElement 返回的新对象 → 浅比较 false

  // ✅ 修复 1：useMemo 稳定对象 / 数组
  const data = useMemo(() => ({ x: 1 }), []);
  const items = useMemo(() => [1, 2, 3], []);

  // ✅ 修复 2：useCallback 稳定函数
  const onClick = useCallback(() => doSomething(), []);

  // ✅ 修复 3：children 是 JSX 时... 没法直接修复
  // 选项 A：把 children 提到外面（compound components 模式）
  // 选项 B：用 React.memo 加自定义比较函数对比 children 元素

  return <Child data={data} onClick={onClick} items={items} />;
}

// 实际工程：
// 用 React DevTools Profiler 看哪些 props 变化导致 memo 失效
// "Why did this render?" 列会精确指出原因
// 然后用 useMemo / useCallback 稳定对应 props

memo 失效的 2 种"不可见" 情况

除上面 4 种 props 引用变化，还有 2 种 memo 不能阻止的重渲染：

javascript · memo 也阻止不了的重渲染 
const Child = memo(function Child({ data }) {
  // ① 内部消费 Context —— Context 变化触发重渲染（memo 不管）
  const theme = useContext(ThemeContext);
  // 即使 props 没变，theme 变 → Child 重渲染

  // ② 内部 useState / useReducer —— 自身 state 变化触发重渲染
  const [count, setCount] = useState(0);
  // 调 setCount 后 Child 重渲染（自然行为）

  // ③ 内部用 useSyncExternalStore —— 外部 store 变化（B7 提过）
  const data = useSyncExternalStore(...);

  return ...;
});

// memo 仅跳过"因 props 不变 + 父重渲染" 触发的 render
// 不能跳过：① 内部 state/reducer 变 ② 订阅的 Context 变 ③ 订阅的外部 store 变

// 这是合理设计 —— Hook 内部状态变化必须触发重渲染才能更新 UI

何时 memo 真正起作用

memo 真正有用的场景—— 满足全部 3 个条件：

  1. 父组件频繁渲染（如父有 useState 频繁变 + memo 子树渲染相对昂贵）
  2. 子组件 props 大多不变（少数几个变化）
  3. 子组件 render 真昂贵（> 1ms，DevTools 测过）

反之三件事都不满足，memo 是负优化（cache 开销 > 节省的 render 时间，B9 详述）。

React Compiler 19 实验性（B9 已讲）会自动加 memoization—— 启用后大部分手动 memo 不必要。这是性能优化的未来方向。

memo + useMemo + useCallback 三件套（短重述链回 B9）

短重述：B9 已详述"memo 必须配 useMemo / useCallback 才有意义"。复习要点：

  • 给 memo 子组件传对象 / 数组 / 函数 prop 时，用 useMemo / useCallback 稳定引用
  • 三件套必须同时用 —— 单用任何一个都白费
  • 没有 memo 包子组件，useMemo / useCallback 单用没意义（B9）
  • React Compiler 启用后这套自动化 —— 新代码不必预先这么写

03 应用场景

场景 A · 编辑表单切换实体重置 state

用户列表 + 详情编辑 panel。点击不同用户进入编辑—— 表单 state 必须重置（不能让 user 1 的输入跑到 user 2）。用 key 一行解决：

javascript · key 强制重置实战 
function UserManager() {
  const [users, setUsers] = useState<User[]>([...]);
  const [editingId, setEditingId] = useState<string | null>(null);
  const editingUser = users.find(u => u.id === editingId);

  return (
    <div>
      <UserList users={users} onSelect={setEditingId} />
      {editingUser && (
        <UserEditForm
          key={editingUser.id}                  {/* ← 关键 */}
          user={editingUser}
          onSave={(updated) => {
            setUsers(prev => prev.map(u => u.id === updated.id ? updated : u));
            setEditingId(null);
          }}
        />
      )}
    </div>
  );
}

function UserEditForm({ user, onSave }) {
  // 初始值从 user 来，但用户编辑时 state 是独立的
  const [name, setName] = useState(user.name);
  const [email, setEmail] = useState(user.email);

  // key 变化 → UserEditForm 销毁重建 → useState 重新初始化为新 user
  // 不需要写 useEffect 同步 user 到 state（那是反模式 —— B7 讲过）

  return (
    <form onSubmit={() => onSave({ ...user, name, email })}>
      <input value={name} onChange={e => setName(e.target.value)} />
      <input value={email} onChange={e => setEmail(e.target.value)} />
      <button type="submit">Save</button>
    </form>
  );
}

// 优势：
// 1. 不写 useEffect 同步 props → state（避免反模式）
// 2. 不写 reset 逻辑（key 自动处理）
// 3. 切换用户时 effect cleanup 也自动跑（关闭 WebSocket / 取消请求等）

场景 B · 用 React DevTools Profiler 找 memo 失效

团队反馈"列表组件渲染慢"。用 Profiler 排查：

  1. 启动 Profiler—— Chrome DevTools → React 面板 → Profiler tab → 点击录制
  2. 触发慢动作—— 输入搜索 / 点击按钮等
  3. 停止录制 → 看 flame graph—— 慢组件高亮
  4. 点慢组件—— 右侧"Why did this render?"列显示原因：
    • "The parent component rendered"—— 父重渲染让子重渲染（默认行为，加 memo 解）
    • "Props changed (data, onClick)"—— 哪些 props 变了，可能需要 useMemo / useCallback 稳定
    • "Hooks changed"—— 内部 useState / useReducer / useContext 变化（memo 不能解，是合理重渲染）
  5. 对症下药：① 父重渲染让子重渲染 → 加 memo ② props 引用不稳 → useMemo / useCallback ③ Context 变 → 拆 Context（B8）

生产推荐工具栈：React DevTools Profiler + React Scan（B9 提过，自动检测重渲染）+ Chrome Performance（看时间线）。

场景 C · Compound Components 模式 + 隐式重渲染

shadcn/ui 风格的 Tabs：

javascript · Compound Components 重渲染分析 
function Tabs({ defaultValue, children }) {
  const [active, setActive] = useState(defaultValue);

  // ⚠️ 注意：Tabs.Provider value 每次 render 新对象 —— 所有 useContext 消费者重渲染
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      {children}
    </TabsContext.Provider>
  );
}

// 优化：useMemo 稳定 value（B8 详述）
function Tabs({ defaultValue, children }) {
  const [active, setActive] = useState(defaultValue);

  const value = useMemo(
    () => ({ active, setActive }),
    [active]   // setActive 引用稳定，仅 active 变化时新建
  );

  return (
    <TabsContext.Provider value={value}>
      {children}
    </TabsContext.Provider>
  );
}

// 用法
<Tabs defaultValue="home">
  <Tabs.List>
    <Tabs.Trigger value="home">Home</Tabs.Trigger>
    <Tabs.Trigger value="profile">Profile</Tabs.Trigger>
  </Tabs.List>
  <Tabs.Content value="home">Home content</Tabs.Content>
  <Tabs.Content value="profile">Profile content</Tabs.Content>
</Tabs>

// 进一步优化：拆 state + dispatch 两个 Context（B8 模式）
// 仅用 setActive 的 Trigger 不会因 active 变化重渲染

04 典型代码

memo 自定义比较函数 —— 何时该用

javascript · 自定义 arePropsEqual 
// 场景：图表组件，data 是大数组（10000+ 项）
// 大多数变化是新增 / 删除尾部 —— 仅 length 变化
// 浅比较每次 false（数组引用变）但实际数据没变（仅 length 加一）

const HeavyChart = memo(
  function HeavyChart({ data, color }) {
    return /* 用 D3 渲染 10000+ 数据点 */;
  },
  // 自定义比较：仅当 data 长度或 color 变才重渲染
  (prev, next) => {
    if (prev.color !== next.color) return false;  // false = 不相等 = 重渲染
    if (prev.data.length !== next.data.length) return false;
    // 数据没变（length 相同 + color 相同）→ 跳过 render
    return true;
  },
);

// ⚠️ 自定义比较函数的陷阱：
// 1. 容易写错（应该比较什么 / 怎么比较）
// 2. 比较本身有开销（深比较大数组比 render 还慢）
// 3. 维护成本高（数据结构变了要改比较函数）
//
// 大多数场景不需要自定义 —— 用 useMemo / useCallback 让浅比较通过更简单
// 仅当：① 数据结构稳定（不会大改） ② 浅比较确实不能满足
//      ③ render 真昂贵（> 50ms） 才考虑

// React 19 + React Compiler 启用后这种手动优化更不必要

识别"本应稳定但每次都变" 的 props

javascript · 稳定 prop 的常见错误 
function Parent() {
  const [count, setCount] = useState(0);

  // ❌ 看起来稳定，实际每次新对象
  return <Child config={{ theme: 'dark', size: 'md' }} />;

  // ✅ 移到组件外（永远稳定）
}
const STABLE_CONFIG = { theme: 'dark', size: 'md' };
function Parent() {
  const [count, setCount] = useState(0);
  return <Child config={STABLE_CONFIG} />;  // 永远同一引用
}

// ❌ 看起来稳定，实际每次新数组
function Parent() {
  return <Child options={['a', 'b', 'c']} />;
}

// ✅ 模块级常量
const OPTIONS = ['a', 'b', 'c'] as const;
function Parent() {
  return <Child options={OPTIONS} />;
}

// ❌ inline 字符串模板（看似稳定）—— 这个其实 OK
<Child label={`Hello ${name}`} />
// 字符串是原始值 —— Object.is 比较值不比引用 → 同样 'Hello Alice' 比较相等 ✅
// 仅引用类型（对象 / 数组 / 函数）才有引用稳定性问题

// 决策：
// - 永远不变 → 模块级常量
// - 依赖 props/state 变化 → useMemo / useCallback
// - 简单字面量原始值 → 直接写 inline OK

05 常见陷阱

⚠️ 陷阱 1：用 array index 当 key 在 list 重排序时（B5 已讲）

list 排序 / 过滤 / 删除时 index 让 React 错配 Fiber—— input 焦点 / 滚动状态 / component state 跑到错误项。

规避：业务 ID 作 key。仅 list 永不重排序（如 static headers）才用 index。

⚠️ 陷阱 2：用 key 强制重置导致动画 / 焦点丢失

<Form key={Date.now()} /> 这种每次都用新 key—— Form 永远 mount unmount，根本没办法用。

规避：key 仅用业务相关稳定值（userId / postId / tabId）。"每次重置" 不是 key 的合理用法。

⚠️ 陷阱 3：memo 子组件不带 useMemo / useCallback 稳定 props

const Child = memo(...) + <Child onClick={() => ...} />—— onClick 每次新函数 → memo 失效。

规避：B9 三件套（memo + useMemo + useCallback）必须配套。React Compiler 启用后这套自动化。

⚠️ 陷阱 4：用 memo 期望阻止 Context / state 触发的重渲染

memo 仅跳过"因 props 不变 + 父重渲染" 触发的 render，不阻止 Context / 内部 state 变化。

规避：理解 memo 的范围。Context 性能问题用拆 Context（B8）/ Selector 模式（B8）/ Zustand 等。

⚠️ 陷阱 5：memo 自定义比较函数返回反了

(prev, next) => prev.data !== next.data—— 这是返回"不相等"，但 React 期望返回"相等"（true = 相等 = 跳过）。结果反向。

规避：记住 true = props 相等 = 跳过 render。这与 class 的 shouldComponentUpdate 反过来（那个 true = should update = 不跳过）。
