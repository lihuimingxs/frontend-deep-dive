B13 · Suspense 用户视角

A4 讲了 Suspense 的"throw promise" 引擎机制。本章讲用户怎么用—— 4 种典型用法（代码分割 / use(promise) / RSC 流式 / 嵌套）+ 边界粒度设计 + 与 useTransition 配合。读完后你能解释"什么时候 fallback 闪烁" 和"怎么用 Suspense 取代 useEffect 加载"。

📐 Suspense 不是"异步组件"，是"异步等待边界"

Suspense 不让组件本身变异步—— 它建立边界，捕获 throw 的 promise / lazy / RSC payload，显示 fallback 直到 ready。组件本身代码看起来同步—— 这是 Suspense 设计的精妙之处（A4 详述）。

阅读时长 ~25 分钟 前置 A4 Concurrent（Suspense 引擎机制） 对应 react.dev Suspense · lazy · use

01 速览

三句话定义本章：

  1. <Suspense fallback={ui}>...</Suspense> 是"异步等待边界"。子树中任何组件 throw promise（含 React.lazy / use(promise) / RSC）都被这个边界捕获 → 显示 fallback → promise resolve 后渲染真实内容。
  2. 4 种典型用法：① React.lazy 代码分割（最常见）② use(promise) 数据 fetching（19+ 取代 useEffect）③ RSC 流式渲染（Next.js 服务端组件）④ 嵌套 Suspense（不同粒度的 loading 状态）。
  3. 边界粒度是核心设计决策：粗边界（外层一个）→ 整页 spinner（用户体感差）；细边界（每个组件）→ fallback 闪烁（"跳来跳去"）；合理粒度按"逻辑组件块" 分（list / sidebar / card）。配合 useTransition 可避免"已显示内容被替换为 fallback" 的回退闪烁。

看一段最小代码识别 Suspense 三种用法：

javascript · Suspense 三种典型用法 
// 用法 1：React.lazy 代码分割
const Dashboard = lazy(() => import('./Dashboard'));

function App() {
  return (
    <Suspense fallback={<PageSpinner />}>
      <Dashboard />       {/* 加载 chunk 时显示 PageSpinner */}
    </Suspense>
  );
}

// 用法 2：use(promise) 数据加载（19+）
function UserProfile({ userPromise }) {
  const user = use(userPromise);   // promise 未 resolve 时 throw → Suspense 捕获
  return <div>{user.name}</div>;
}

<Suspense fallback={<Spinner />}>
  <UserProfile userPromise={fetchUser(42)} />
</Suspense>

// 用法 3：RSC（Next.js App Router）
async function Page() {
  const data = await db.fetch();   // Server Component 直接 await
  return (
    <>
      <h1>{data.title}</h1>
      <Suspense fallback={<Skeleton />}>
        <SlowChart dataPromise={db.fetchSlowStats()} />
      </Suspense>
    </>
  );
}

02 底层逻辑

Suspense 完整签名与心智

typescript · Suspense props 
type SuspenseProps = {
  children?: ReactNode;
  fallback?: ReactNode;       // 加载时显示的内容（可以是任何 JSX）
  // 19 加：name 用于 React DevTools 显示（仅调试用）
  name?: string;
};

// 心智模型：
// Suspense 是 React 树中的一个特殊节点
// 子树渲染时如果有 throw promise → 沿 Fiber 向上找到最近 Suspense
// → 渲染 fallback、注册 promise.then 回调
// → promise resolve 后，重新尝试渲染子树
// → 如果还有 throw → 继续 fallback；如果 resolve 了 → 渲染真实内容

用法 1：React.lazy 代码分割（最常见）

javascript · 代码分割完整模式 
// 1. lazy 包装动态 import
import { lazy, Suspense } from 'react';

const Dashboard = lazy(() => import('./Dashboard'));
const Settings = lazy(() => import('./Settings'));
const Reports = lazy(() => import('./Reports'));

// 2. 用 Suspense 包裹
function App() {
  return (
    <Routes>
      <Route path="/dashboard" element={
        <Suspense fallback={<PageSpinner />}>
          <Dashboard />
        </Suspense>
      } />
      <Route path="/settings" element={
        <Suspense fallback={<PageSpinner />}>
          <Settings />
        </Suspense>
      } />
    </Routes>
  );
}

// 3. 配 bundler 自动 chunk 拆分
// Vite / Webpack / Rollup 都自动处理 dynamic import 拆 chunk
// 用户访问 /dashboard 才下载 dashboard.chunk.js

// 4. 给 lazy 加 named export 支持
// dynamic import 默认拿 default export，要用 named export：
const Dashboard = lazy(() =>
  import('./Dashboard').then(m => ({ default: m.Dashboard }))
);

// 5. 预加载（用户即将访问时提前下载）
function NavLink({ to, children }) {
  const handleHover = () => {
    if (to === '/dashboard') import('./Dashboard');  // 预热
  };
  return <a href={to} onMouseEnter={handleHover}>{children}</a>;
}

// 工程意义：
// - 减小初始 bundle 体积（仅当前页面 chunk 加载）
// - 路由级懒加载（每路由独立 chunk）
// - 大第三方库懒加载（如 chart.js / monaco-editor）

用法 2：use(promise) 数据 fetching（React 19）

javascript · use API 数据加载完整模式 
import { use, Suspense } from 'react';

function UserProfile({ userPromise }) {
  const user = use(userPromise);   // promise 未 resolve 时 throw
  return <div>{user.name}</div>;
}

// 用法
function App() {
  // ⚠️ 关键：promise 必须在更高层创建（引用稳定）
  const userPromise = useMemo(() => fetchUser(42), []);

  return (
    <Suspense fallback={<Spinner />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  );
}

// ❌ 致命错误：每次 render 创建新 promise
function App() {
  return (
    <Suspense fallback={<Spinner />}>
      <UserProfile userPromise={fetchUser(42)} />
                              {/* ↑ 每次 render 新 promise！*/}
    </Suspense>
  );
}
// 后果：① 每次都新建 fetch 请求 ② Suspense 无限触发 → 死循环

// 替代当前 useEffect 模式：
// === 18 风格（useEffect + useState）===
function UserProfile({ id }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    fetch(`/api/users/${id}`).then(r => r.json()).then(d => {
      setUser(d); setLoading(false);
    });
  }, [id]);
  if (loading) return <Spinner />;
  return <div>{user.name}</div>;
}

// === 19 风格（use + Suspense）===
async function fetchUser(id) {
  return fetch(`/api/users/${id}`).then(r => r.json());
}
function UserProfile({ userPromise }) {
  const user = use(userPromise);
  return <div>{user.name}</div>;
}
// 父：
const userPromise = useMemo(() => fetchUser(id), [id]);
return (
  <Suspense fallback={<Spinner />}>
    <UserProfile userPromise={userPromise} />
  </Suspense>
);

// 优势：
// - 组件代码看起来同步（无 loading state）
// - Suspense fallback 由调用方定（更灵活）
// - 错误用 ErrorBoundary 统一处理（B15）

// 但注意：
// - use 可在条件分支调用（A4 / B8 提过）
// - 但 promise 引用必须稳定 —— 在 RSC 中天然稳定（每次请求服务端创建）
// - 在 Client Component 中要 useMemo / useRef / 父级管理

用法 3：RSC 流式渲染（Next.js App Router）

javascript · RSC + Suspense 流式渲染 
// app/page.tsx —— Server Component（默认）
async function Page() {
  // 同步部分（快）
  const user = await db.users.findById(currentUserId);

  return (
    <>
      <h1>Welcome, {user.name}</h1>       {/* 立即渲染 */}

      {/* 异步部分流式 */}
      <Suspense fallback={<PostsSkeleton />}>
        <UserPosts userId={user.id} />     {/* SlowChart 是 async Server Component */}
      </Suspense>

      <Suspense fallback={<StatsSkeleton />}>
        <UserStats userId={user.id} />
      </Suspense>
    </>
  );
}

async function UserPosts({ userId }) {
  const posts = await db.posts.findByUser(userId);  // 假设要 800ms
  return <PostList posts={posts} />;
}

async function UserStats({ userId }) {
  const stats = await db.stats.compute(userId);     // 假设要 1500ms
  return <StatsChart stats={stats} />;
}

// 服务器流式渲染时序：
// 1. 服务器立即发送同步部分 HTML：
//    <h1>Welcome, Alice</h1>
//    <div>PostsSkeleton</div>     {/* fallback 占位 */}
//    <div>StatsSkeleton</div>
// 2. 浏览器立即显示骨架（用户感知"页面已加载"）
// 3. UserPosts 异步完成（800ms 后）→ 服务器流式补发：
//    <script>React.replace('id-1', ...);</script>
//    替换 PostsSkeleton 为真实内容
// 4. UserStats 完成（1500ms）→ 同上替换 StatsSkeleton

// 用户体感：
// - TTFB（首字节时间）：很快（同步部分立即返回）
// - 主要内容：分阶段填充
// - 没有"整页空白等几秒" 的体验

用法 4：嵌套 Suspense + showFallback 行为

javascript · 嵌套 Suspense fallback 触发规则 
function App() {
  return (
    <Suspense fallback={<PageSpinner />}>        {/* 外层 */}
      <Header />
      <Suspense fallback={<ListSpinner />}>     {/* 内层 1 */}
        <UserList />
      </Suspense>
      <Suspense fallback={<ChartSpinner />}>    {/* 内层 2 */}
        <Chart />
      </Suspense>
      <Footer />
    </Suspense>
  );
}

// 触发规则：throw promise 沿 Fiber 树向上找最近的 Suspense
//
// UserList throw promise → 沿 Fiber 上溯
//   → 找到 <Suspense fallback={<ListSpinner />}>（内层 1）
//   → 显示 ListSpinner
//   → Header / Chart / Footer 不受影响（已渲染或独立 Suspense）
//
// Chart throw promise → 同样找内层 2 → 显示 ChartSpinner
//
// 如果 Header throw promise（很少见，比如 Header 也有 use）：
//   → 沿 Fiber 上溯找不到内层 Suspense（Header 不在内层 Suspense 内）
//   → 找到外层 Suspense → 显示 PageSpinner（整页 loading）

// 设计原则：按"逻辑组件块" 分边界
// - 一个 list / 一个 sidebar / 一个 card 各自一个 Suspense
// - 不要"每个数据点一个 Suspense"（fallback 闪烁）
// - 也不要"整页只有一个 Suspense"（用户体感差）

SuspenseList（实验性，仅 unstable）

SuspenseList 是实验性 API，协调多个 Suspense 的显示顺序：

javascript · SuspenseList 实验性 
// ⚠️ 不稳定：仅 react@experimental 可用
import { unstable_SuspenseList as SuspenseList } from 'react';

<SuspenseList revealOrder="forwards" tail="collapsed">
  <Suspense fallback={<Skeleton />}><Card1 /></Suspense>
  <Suspense fallback={<Skeleton />}><Card2 /></Suspense>
  <Suspense fallback={<Skeleton />}><Card3 /></Suspense>
</SuspenseList>

// revealOrder：
// - "forwards"：按顺序显示（前面 ready 才显示前面，后面等）
// - "backwards"：反顺序显示
// - "together"：等所有 ready 一起显示

// tail：
// - "collapsed"：仅显示下一个 fallback（其余隐藏）
// - "hidden"：所有未 ready 的都隐藏（不显示 fallback）

// 用例：避免"列表项乱序闪现"

// 当前状态（写作时点 2026-04）：
// - 仍是 unstable_ 前缀
// - React 团队还在评估稳定 API 形态
// - 生产代码不要用

与 useTransition 配合（避免 fallback 回退闪烁）

javascript · transition 防止 fallback 闪烁 
// 痛点：用户切换 tab → 已显示的内容被 fallback 替换 → 闪烁

// ❌ 不用 transition
function App() {
  const [tab, setTab] = useState('home');
  return (
    <>
      <Tabs activeTab={tab} onChange={setTab} />
      <Suspense fallback={<Spinner />}>
        {tab === 'home' && <HomeContent />}
        {tab === 'profile' && <ProfileContent />}
      </Suspense>
    </>
  );
}
// 切换时：HomeContent 立即被卸载 → ProfileContent throw promise → 显示 Spinner
// 用户看到"已显示的 home → spinner → profile" 的 3 阶段闪烁

// ✅ 用 useTransition
function App() {
  const [tab, setTab] = useState('home');
  const [isPending, startTransition] = useTransition();

  const handleTabChange = (next) => {
    startTransition(() => setTab(next));
  };

  return (
    <>
      <Tabs activeTab={tab} onChange={handleTabChange} />
      {isPending && <ProgressBar />}
      <Suspense fallback={<Spinner />}>
        {tab === 'home' && <HomeContent />}
        {tab === 'profile' && <ProfileContent />}
      </Suspense>
    </>
  );
}
// 切换时：tab 在 transition lane（低优先级）
// React 在新内容 ready 之前保留旧内容
// 用户看到"home（仍显示）+ ProgressBar → profile（替换 home）"
// fallback 不会触发（因为有旧内容可保留）

// 详细 useTransition 见 B14（编写中）

03 应用场景

场景 A · 路由级代码分割

大型应用按路由拆分 chunk—— 每个路由独立 Suspense + lazy：

javascript · 路由级 lazy + Suspense 
import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

// 每个路由的页面组件懒加载
const HomePage = lazy(() => import('./pages/Home'));
const DashboardPage = lazy(() => import('./pages/Dashboard'));
const SettingsPage = lazy(() => import('./pages/Settings'));

function App() {
  return (
    <Routes>
      <Route path="/" element={
        <Suspense fallback={<PageSpinner />}>
          <HomePage />
        </Suspense>
      } />
      <Route path="/dashboard" element={
        <Suspense fallback={<PageSpinner />}>
          <DashboardPage />
        </Suspense>
      } />
      <Route path="/settings" element={
        <Suspense fallback={<PageSpinner />}>
          <SettingsPage />
        </Suspense>
      } />
    </Routes>
  );
}

// Vite/Webpack 自动拆 chunk：
// - app.[hash].js   主 bundle
// - Home.[hash].js
// - Dashboard.[hash].js
// - Settings.[hash].js

// 优化建议：
// 1. 用 Suspense 套一层共享路由布局，避免每个 Route 重复包装
// 2. 配合 useTransition 让路由切换不闪 fallback
// 3. 鼠标 hover 链接时预加载下一页（import('./pages/Dashboard')）

场景 B · 详情页 + 推荐区块独立 Suspense

商品详情页有 3 个独立数据：商品信息（快）/ 用户评价（中）/ 推荐商品（慢）。每个独立 Suspense 让用户尽快看到主信息：

javascript · 多独立 Suspense 边界 
// Next.js RSC 风格
async function ProductPage({ productId }) {
  // 主信息 await（关键路径，必须有）
  const product = await db.products.findById(productId);

  return (
    <>
      <ProductHeader product={product} />
      <ProductGallery images={product.images} />

      {/* 评价独立加载 */}
      <Suspense fallback={<ReviewsSkeleton />}>
        <ProductReviews productId={productId} />
      </Suspense>

      {/* 推荐独立加载（慢，放最后）*/}
      <Suspense fallback={<RecommendationsSkeleton />}>
        <Recommendations productId={productId} />
      </Suspense>
    </>
  );
}

async function ProductReviews({ productId }) {
  const reviews = await db.reviews.findByProduct(productId);
  return <ReviewList reviews={reviews} />;
}

async function Recommendations({ productId }) {
  // 推荐用 ML 模型计算，可能 2-3s
  const items = await ml.recommend(productId);
  return <ItemGrid items={items} />;
}

// 用户体感：
// - 0ms：进入路由开始 SSR
// - 200ms：商品信息显示（关键路径完成）
// - 500ms：评价骨架被真实评价替换
// - 2000ms：推荐骨架被真实推荐替换
// - 全程"渐进显示"，不阻塞主信息

场景 C · 用 use API 重写老 useEffect 数据加载

老项目用 useEffect + useState 加载数据。React 19 + use API 重写更简洁：

javascript · 18 vs 19 重写 
// === React 18 风格 ===
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [error, setError] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);
    fetchUser(userId)
      .then(d => { if (!cancelled) { setUser(d); setLoading(false); } })
      .catch(e => { if (!cancelled) { setError(e); setLoading(false); } });
    return () => { cancelled = true; };
  }, [userId]);

  if (loading) return <Spinner />;
  if (error) return <ErrorMsg error={error} />;
  return <UserCard user={user} />;
}

// === React 19 + use API ===
function UserProfile({ userPromise }) {
  const user = use(userPromise);   // 同步代码看起来
  return <UserCard user={user} />;
}

function UserPanel({ userId }) {
  const userPromise = useMemo(() => fetchUser(userId), [userId]);
  return (
    <ErrorBoundary fallback={<ErrorMsg />}>
      <Suspense fallback={<Spinner />}>
        <UserProfile userPromise={userPromise} />
      </Suspense>
    </ErrorBoundary>
  );
}

// 优势：
// 1. UserProfile 代码量减半
// 2. loading / error 状态由 Suspense / ErrorBoundary 集中管理
// 3. 配合 RSC 时 promise 可在 server 端创建（更优）

// 实际工程仍推荐 TanStack Query —— 它内置缓存 / 后台刷新 / 乐观更新
// use API 适合简单场景 / 配合 RSC

04 典型代码

共享路由布局 + Suspense 边界

javascript · 共享布局减少重复 Suspense 
// ❌ 每个 Route 重复包装 Suspense（B12 场景 A 那样）
<Routes>
  <Route path="/home" element={<Suspense fallback={<PageSpinner />}><Home /></Suspense>} />
  <Route path="/dash" element={<Suspense fallback={<PageSpinner />}><Dash /></Suspense>} />

// ✅ Layout 层共享 Suspense
function AppLayout() {
  return (
    <>
      <Header />

      </Suspense>
      <Footer />
    </>

}

<Routes>
  <Route element={<AppLayout />}>
    <Route path="/home" element={<Home />} />
    <Route path="/dash" element={<Dash />} />

// 优势：

Suspense + ErrorBoundary 双重边界

javascript · 异步加载 + 错误处理 
import { ErrorBoundary } from 'react-error-boundary';   // 社区库
import { Suspense, lazy } from 'react';

const Dashboard = lazy(() => import('./Dashboard'));

function App() {
  return (
    <ErrorBoundary
      fallback={({ error, resetErrorBoundary }) => (
        <div>
          <h1>Error: {error.message}</h1>
          <button onClick={resetErrorBoundary}>Try again</button>
        </div>
      )}
    >
      <Suspense fallback={<PageSpinner />}>
        <Dashboard />
      </Suspense>
    </ErrorBoundary>
  );
}

// 双重边界协作：
// - Suspense 处理 loading state
// - ErrorBoundary 处理错误
// - 都不需要在 Dashboard 内部写状态管理代码

// 推荐组合：ErrorBoundary 在外、Suspense 在内
// - lazy chunk 加载失败 → ErrorBoundary 捕获
// - 数据加载中 → Suspense 显示 fallback
// - 数据加载失败 → use 抛错 → ErrorBoundary 捕获

// 详见 B15 Error Boundary 章节（编写中）

05 常见陷阱

⚠️ 陷阱 1：promise 引用不稳定致 use 死循环

<UserProfile userPromise={fetchUser(42)} />—— 父每次 render 都新建 promise → use 等待 → resolve 后 render → 又新 promise → 死循环。A4 / B6 已警告。

规避：promise 必须在更高层创建。Client Component 用 useMemo(() => fetchUser(id), [id])。RSC 中天然稳定（每次请求服务端创建）。

⚠️ 陷阱 2：Suspense 边界粒度太细 fallback 闪烁

每个 list item 一个 Suspense → 项数据 ready 时间略不同 → 看到 spinner 一个个跳出来。

规避：按"逻辑组件块" 分（一个 list / 一个 sidebar 共享 Suspense）。或者用 SuspenseList（实验性）协调显示顺序。

⚠️ 陷阱 3：Suspense fallback 也使用 use → 死循环

<Suspense fallback={<LoadingFromAPI />}><Content /></Suspense>—— LoadingFromAPI 内部也用 use(promise)—— Content throw → 显示 LoadingFromAPI → LoadingFromAPI throw → 没法捕获（fallback 已经在 fallback 位置）→ 死循环或崩溃。

规避：fallback 是同步组件—— 不要在 fallback 中用 lazy / use / 任何 throw promise 的东西。fallback 应该是简单 spinner / skeleton。

⚠️ 陷阱 4：tab 切换 Suspense 闪烁（不用 useTransition）

切换 tab 时 React 立即卸载旧内容 → 新内容 throw → 显示 fallback。用户体感"已加载内容被 spinner 替换"。

规避：用 useTransition 包 setState（B14）—— React 在新内容 ready 前保留旧内容显示。

⚠️ 陷阱 5：lazy chunk 加载失败没 ErrorBoundary 兜底

网络问题导致 chunk 下载失败—— 没 ErrorBoundary 时整个 app 崩溃白屏。

规避：每个 lazy 都用 ErrorBoundary 包裹（B15）。Suspense 处理 loading，ErrorBoundary 处理错误，是 19 时代标准组合。
