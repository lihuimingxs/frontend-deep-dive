B3 · 组件与 Props

组件是 React 的复用单元，Props 是组件的输入。这两个概念决定了 React 应用的组合形态—— 你的应用就是"组件树"，每层组件接受 props 渲染 UI 并把数据传给子组件。本章讲清组件签名、Props 不可变规则、解构最佳实践、children prop 与组合、默认值演化（defaultProps 19 退役）、TypeScript Props 类型、以及 4 种 Composition 模式（children / Render Props / HOC / 自定义 Hooks）的 19 时代选型。

📐 本章是 A2 设计哲学的"怎么用" 篇

A2 · 设计哲学 讲了"Composition over inheritance" 是 React 的核心决策。本章讲具体怎么 compose—— 4 种模式各自的代码形态、何时选哪个。

阅读时长 ~25 分钟 前置 B1 渲染模型 + B2 JSX 对应 react.dev Passing Props · Passing Data Deeply

01 速览

三句话定义本章：

  1. 函数组件 = (props) => JSX。props 是只读对象—— 组件不能修改 props。这就是 React 单向数据流（A2）的具体实现：parent 决定 child 的 props，child 不能反向修改。
  2. 4 种 Composition 模式对应不同复用场景：① children prop（最常用，"把内容传给我的 slot"）② Render Props（"我提供数据，你决定怎么渲染"）③ 自定义 Hooks（"我提供逻辑，你自己渲染"，19 时代主流）④ HOC（"我包装组件加能力"，已不推荐，自定义 Hooks 替代）。
  3. React 19 退役 defaultProps（仅函数组件，class 仍支持）。新代码用解构默认值：function Btn({ size = 'md' }) {...}。同时也退役 forwardRef—— 19+ 直接 function Comp({ ref, ... }) 接收 ref 作为 prop（详见 B19）。

看一段最小组件代码，识别其中的关键概念：

javascript · 组件 + Props 完整骨架 
// 1. 函数组件 = 接受 props 返回 JSX
function Greeting({ name, age = 18, children }) {
  //              ↑ 解构 + 默认值（19 推荐）
  //              ↑ children 是特殊 prop
  return (
    <div>
      <h1>Hello, {name}</h1>
      <p>Age: {age}</p>
      {children}
    </div>
  );
}

// 2. 使用：标签里属性都是 props
<Greeting name="Alice" age={30}>
  <p>Welcome back!</p>       {/* ← 这部分变成 children */}
</Greeting>

// 3. 等价于
React.createElement(Greeting, {
  name: 'Alice',
  age: 30,
  children: React.createElement('p', null, 'Welcome back!'),
});

// 关键事实：
// - props 是普通 JS 对象
// - children 在 props 里
// - props 不可变 —— Greeting 函数体不能 props.name = 'Bob'

02 底层逻辑

组件签名与命名约定

函数组件有 4 种合法写法，本质相同：

javascript · 4 种函数组件写法 
// 1. function 声明（最常见）
function MyButton(props) {
  return <button>{props.label}</button>;
}

// 2. 函数表达式赋值给 const（适合包装高阶组件）
const MyButton = function(props) {
  return <button>{props.label}</button>;
};

// 3. 箭头函数（最简洁）
const MyButton = (props) => {
  return <button>{props.label}</button>;
};

// 4. 箭头函数 + 隐式 return（一行函数）
const MyButton = ({ label }) => <button>{label}</button>;

// 4 种行为完全相同。
// 团队风格选一种：
// - 推荐 1 或 3，可读性最佳
// - 写大于 5 行用 1（function 声明），有 hoisting 优势 + DevTools 显示函数名
// - 写一行小组件用 4（箭头隐式 return）
// - 2 几乎没人用

命名约定（B1 已讲）：

  • 组件名首字母大写：MyButton、UserCard（PascalCase）
  • 文件名常用 kebab-case 或 PascalCase 都可：my-button.tsx / MyButton.tsx
  • 导出：export default function Component(props) { ... } 或具名 export function Component(props) { ... }

Props 不可变规则

这是 React 单向数据流的硬规则—— 组件函数体内不能修改 props：

javascript · Props 不可变 
// ❌ 直接 mutate
function UserCard({ user }) {
  user.name = 'Bob';        // ❌ 修改 props！
  return <div>{user.name}</div>;
  // 后果：① 父组件持有的 user 对象被你改了 ② 跨组件污染 ③ React 不知道
  //       state 变了，可能看到不一致 UI
}

// ❌ 修改 props 上的数组 / 嵌套对象
function TodoList({ items }) {
  items.push({ id: 1, text: 'New' });  // ❌ 同样禁止
  return ...;
}

// ✅ 想"修改" props？让 parent 改，child 通过回调请求
function TodoList({ items, onAdd }) {
  return (
    <>
      {items.map(item => <li key={item.id}>{item.text}</li>)}
      <button onClick={() => onAdd({ id: Date.now(), text: 'New' })}>Add</button>
    </>
  );
}

function Parent() {
  const [items, setItems] = useState([]);
  return <TodoList
    items={items}
    onAdd={item => setItems(prev => [...prev, item])}  // 创建新数组（A2 不可变）
  />;
}

// ✅ 想"派生" 新值？用 useMemo / 直接计算
function UserCard({ user }) {
  const fullName = `${user.firstName} ${user.lastName}`;  // ✅ 派生值不算 mutate
  return <div>{fullName}</div>;
}

"为什么不可变"：A2 设计哲学 + A3 Fiber 已讲过引用比较的根因。简言之：① 单向数据流可预测 ② React.memo 浅比较依赖 props 引用稳定 ③ Concurrent rendering 期间组件可能多次调用，mutate 会让多次调用结果不一致。

解构 Props 的最佳实践

javascript · 解构 vs 完整 props 对比 
// 风格 1：保留完整 props（不解构）
function Card(props) {
  return (
    <div>
      <h2>{props.title}</h2>
      <p>{props.body}</p>
      {props.footer}
    </div>
  );
}
// 优势：rest 参数透传方便（<Card {...props} />）
// 劣势：每个使用都要 props.X，啰嗦

// 风格 2：参数解构（推荐）
function Card({ title, body, footer, ...rest }) {
  return (
    <div {...rest}>
      <h2>{title}</h2>
      <p>{body}</p>
      {footer}
    </div>
  );
}
// 优势：清晰知道用了哪些 props；rest 仍可透传
// 这是 React 19 的事实标准

// 风格 3：解构 + 默认值
function Card({ title, body, footer = null, variant = 'default' }) {
  // ...
}
// 默认值在解构时给 —— 19 推荐写法（取代 defaultProps）

// 风格 4：解构 + 重命名（避免冲突）
function Modal({ open: isOpen, ...props }) {
  // 内部用 isOpen，外部 prop 名是 open
}

// 风格 5：嵌套解构（深度访问）
function UserPanel({ user: { name, age, address: { city } } }) {
  return <div>{name} ({age}) - {city}</div>;
}
// ⚠️ 不推荐 —— user 可能 undefined 时会抛错
// 改用：const { user = {} } = props; const { name = '?' } = user;

children prop 与组合

children 是特殊 prop—— JSX 标签之间的内容自动变成 props.children。这是 React Composition 的最基本工具。

javascript · children 全用法 
// 1. 基础：children 是 ReactNode
function Container({ children }) {
  return <div className="container">{children}</div>;
}
<Container>
  <p>Anything here</p>
</Container>

// 2. children 类型：可以是 string / 元素 / 数组 / 函数 / null
<Container>Hello</Container>                    // string
<Container><p /><p /></Container>                // 元素数组
<Container>{condition ? <A /> : null}</Container> // 表达式

// 3. 显式作为 prop 传（当不在 JSX 标签之间）
<Container children={<p>Hello</p>} />
// 等价于上面的标签嵌套写法

// 4. 多个 "slot"（命名 children）
function Card({ header, body, footer }) {
  return (
    <div>
      <div className="header">{header}</div>
      <div className="body">{body}</div>
      <div className="footer">{footer}</div>
    </div>
  );
}
<Card
  header={<h1>Title</h1>}
  body={<p>Body</p>}
  footer={<button>OK</button>}
/>
// 类似 Vue 的 named slots

// 5. children 作为函数（Render Props 模式，下文详述）
<Tooltip>
  {(isOpen) => isOpen ? <div>Visible</div> : null}
</Tooltip>

// 6. 包装层级（compound components 模式）
<Tabs>
  <Tabs.List>
    <Tabs.Tab name="a">A</Tabs.Tab>
    <Tabs.Tab name="b">B</Tabs.Tab>
  </Tabs.List>
  <Tabs.Panel name="a">Content A</Tabs.Panel>
  <Tabs.Panel name="b">Content B</Tabs.Panel>
</Tabs>
// shadcn/ui / Radix 等都用 compound components 模式

默认值 ——「defaultProps 退役」

React 19 退役 defaultProps（仅函数组件，class 仍支持）。这是个长期演化：

javascript · 默认值 3 个时代 
// === 时代 1：React.createClass / class 时代 ===
class Button extends React.Component {
  static defaultProps = {
    size: 'md',
    variant: 'primary',
  };
  render() {
    return <button className={`btn btn-${this.props.size}`}>...</button>;
  }
}

// === 时代 2：函数组件 + defaultProps（17-18 时代仍支持）===
function Button({ size, variant }) {
  return <button className={`btn btn-${size}`}>...</button>;
}
Button.defaultProps = {
  size: 'md',
  variant: 'primary',
};
// React 19 警告：⚠️ defaultProps will be deprecated for function components

// === 时代 3：解构默认值（19 推荐）===
function Button({ size = 'md', variant = 'primary' }) {
  return <button className={`btn btn-${size}`}>...</button>;
}
// 优势：① TypeScript 类型推导更精确 ② 性能更好（不需运行时合并）
//       ③ 视觉上参数和默认值在一起，可读性好

升级路径：codemod npx react-codemod replace-defaultprops-with-default-args 自动迁移。

TypeScript Props 类型

短重述：用 type 或 interface 定义 Props 类型。组件用 type（一致性 + 联合类型友好），对外 API / 库类型用 interface（declaration merging 优势）。详见 TS · Everyday Types 的 type vs interface 决策树。

typescript · React Props 类型常见模式 
import { ReactNode, MouseEvent } from 'react';

// 1. 基础 Props 类型
type ButtonProps = {
  label: string;
  size?: 'sm' | 'md' | 'lg';   // 联合类型 + 可选
  variant?: 'primary' | 'secondary' | 'danger';
  onClick?: (e: MouseEvent<HTMLButtonElement>) => void;
  disabled?: boolean;
  children?: ReactNode;        // 推荐显式声明 children
};

function Button({ label, size = 'md', variant = 'primary', onClick, disabled, children }: ButtonProps) {
  return <button onClick={onClick} disabled={disabled}>{label}{children}</button>;
}

// 2. 透传原生属性（继承 HTML element props）
type CardProps = React.ComponentPropsWithoutRef<'div'> & {
  title: string;
};
function Card({ title, children, ...rest }: CardProps) {
  return <div {...rest}><h2>{title}</h2>{children}</div>;
}
// 现在 Card 接受所有 div 原生属性（onClick / className / id / style / role / aria-*）

// 3. polymorphic 组件（as prop）
type AsProp<T extends React.ElementType> = { as?: T };
type PolymorphicProps<T extends React.ElementType, P> =
  P & AsProp<T> & Omit<React.ComponentPropsWithoutRef<T>, keyof P | 'as'>;

function Box<T extends React.ElementType = 'div'>({
  as, children, ...rest
}: PolymorphicProps<T, { children?: ReactNode }>) {
  const Component = as || 'div';
  return <Component {...rest}>{children}</Component>;
}

<Box>Default div</Box>
<Box as="section">Now section</Box>
<Box as="a" href="/">Now link</Box>       // href 类型来自 'a'

// 4. ⚠️ React.FC（不推荐）
const Button: React.FC<ButtonProps> = (props) => {...};
// React 19 之前：React.FC 自动加 children type，导致即使 Props 没声明 children 也能传
// React 19+：FC 仍可用但建议不用 —— 显式写 children: ReactNode 更清晰
// 详见 react.dev 官方建议

Composition 4 模式 + 19 时代选什么

4 种 Composition 模式 + 推荐度（React 19）：

模式 形态 19 推荐度
children prop <Wrapper>{x}</Wrapper> ⭐⭐⭐ 最常用
命名 prop（slots） <Card header={x} body={y} /> ⭐⭐⭐ 多 slot 时
Render Props <DataLoader>{(data) => ...}</DataLoader> ⭐⭐ 仍合法但被自定义 Hooks 替代
自定义 Hooks const { data } = useFetch(url) ⭐⭐⭐ 19 时代主流
HOC（Higher-Order Component） withAuth(Comp) ⚠️ 已不推荐，自定义 Hooks 替代
Compound Components <Tabs><Tabs.List>...</Tabs> ⭐⭐⭐ 复杂组件库（shadcn / Radix）
javascript · 同一需求的 4 种实现 
// 需求：复用"fetch 数据 + 显示" 逻辑

// === Render Props（老） ===
function DataLoader({ url, children }) {
  const [data, setData] = useState(null);
  useEffect(() => { fetch(url).then(r => r.json()).then(setData); }, [url]);
  return children(data);
}
<DataLoader url="/api/user">
  {(data) => data ? <UserCard {...data} /> : <Spinner />}
</DataLoader>
// 缺点：嵌套深 + 类型推导差

// === HOC（更老，已不推荐） ===
function withFetch(Component, url) {
  return function Wrapped(props) {
    const [data, setData] = useState(null);
    useEffect(() => { fetch(url).then(r => r.json()).then(setData); }, []);
    return data ? <Component {...props} data={data} /> : <Spinner />;
  };
}
const UserCardWithFetch = withFetch(UserCard, '/api/user');
<UserCardWithFetch />
// 缺点：① 名字混乱（withFetchedData / withAuth / withLogger 多个 wrap 嵌套）
//       ② displayName 丢失 / TypeScript 类型推导极复杂
//       ③ ref 丢失（要 forwardRef，19 不再有）

// === 自定义 Hooks（19 推荐） ===
function useFetch(url) {
  const [data, setData] = useState(null);
  useEffect(() => { fetch(url).then(r => r.json()).then(setData); }, [url]);
  return data;
}
function UserPanel() {
  const data = useFetch('/api/user');
  return data ? <UserCard {...data} /> : <Spinner />;
}
// 优点：① 扁平 ② 类型推导清晰 ③ 多个 Hook 可组合 ④ 无 wrapper 层级污染

// === RSC（19 + Next.js）===
async function UserPanel() {
  const data = await fetch('/api/user').then(r => r.json());
  return <UserCard {...data} />;
}
// 服务器渲染，无客户端 JS / 无 loading state（详见 A4 / B16）

19 时代决策：

  • UI 复用 → children / 命名 prop（最常见）
  • 逻辑复用 → 自定义 Hooks（替代 HOC + Render Props）
  • 复杂组件库 → Compound Components（Tabs / Accordion / Menu）
  • HOC 已不推荐—— 老代码可保留，新代码不写
  • Render Props 仍合法但少用—— 自定义 Hooks 更扁平

03 应用场景

场景 A · 设计灵活的 Card 组件

做一个 Card 组件，要支持"多种用法"—— 简单时只传 title + body，复杂时要自定义 header / footer / actions。

javascript · 灵活 Card 设计 
// 简单场景：title + body
<Card title="Hello" body="World" />

// 复杂场景：自定义 slot
<Card>
  <Card.Header>
    <Avatar src={user.avatar} />
    <h2>{user.name}</h2>
  </Card.Header>
  <Card.Body>
    <Markdown content={post.content} />
  </Card.Body>
  <Card.Footer>
    <Button>Like</Button>
    <Button>Comment</Button>
  </Card.Footer>
</Card>

// 实现：compound components + 默认值
function Card({ title, body, children }) {
  // 既支持简单（title + body）也支持复杂（children）
  if (children) return <div className="card">{children}</div>;
  return (
    <div className="card">
      <Card.Header>{title}</Card.Header>
      <Card.Body>{body}</Card.Body>
    </div>
  );
}
Card.Header = ({ children }) => <div className="card-header">{children}</div>;
Card.Body = ({ children }) => <div className="card-body">{children}</div>;
Card.Footer = ({ children }) => <div className="card-footer">{children}</div>;

场景 B · 把 HOC 重构成自定义 Hook

老项目用 withAuth(Component) HOC 检查用户登录态。重构成自定义 Hook：

javascript · HOC → Hook 重构 
// === 老 HOC ===
function withAuth(Component) {
  return function ProtectedComponent(props) {
    const user = useContext(UserContext);
    if (!user) return <Login />;
    return <Component {...props} user={user} />;
  };
}
const ProtectedDashboard = withAuth(Dashboard);

// === 重构成 Hook ===
function useAuth() {
  const user = useContext(UserContext);
  return user;
}

function ProtectedDashboard(props) {
  const user = useAuth();
  if (!user) return <Login />;
  return <Dashboard {...props} user={user} />;
}

// 优势：
// 1. 不丢 ref（HOC 不 forwardRef 时 ref 丢）
// 2. 多个保护逻辑可组合：const user = useAuth(); const role = useRole(user);
// 3. TypeScript 类型清晰
// 4. 调试容易（DevTools 显示 ProtectedDashboard 而非 withAuth(Dashboard)）

// 迁移策略：
// 1. 保留老 HOC 不动（兼容）
// 2. 新代码用 Hook
// 3. 修 bug 时顺便重构，不要专门组织 sprint 改

场景 C · 设计 polymorphic 组件（as prop）

shadcn/ui 的 Button 支持 asChild—— 让 Button 渲染成不同 HTML 元素（a / button / 自定义 Link 组件等），保留 Button 样式但语义改变。Radix UI 同样模式。

javascript · 简化版 polymorphic 
// 用法：
<Button>Click</Button>            // 默认渲染为 button
<Button as="a" href="/about">About</Button>  // 渲染为 a
<Button as={Link} to="/about">About</Button> // 渲染为 React Router Link

// 实现（TS 简化版，详细实现见 Radix UI / Slot 源码）
function Button({ as: Component = 'button', children, ...props }) {
  return <Component className="btn" {...props}>{children}</Component>;
}

// 优势：保留 Button 样式 + 灵活语义
// 缺点：TS 类型复杂（需要 polymorphic 类型工具，见上面 TypeScript 部分）

// shadcn 的 asChild 模式更灵活：
<Button asChild>
  <Link to="/about">About</Link>
</Button>
// 内部用 Slot 把 Button 的 className / onClick 等"合并到" Link 上
// 详见 Radix Slot 源码

04 典型代码

Compound Components 完整模式（Tabs）

javascript · Compound Components + Context 
import { createContext, useContext, useState } from 'react';

const TabsContext = createContext(null);

function Tabs({ defaultValue, children }) {
  const [active, setActive] = useState(defaultValue);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  );
}

function TabsList({ children }) {
  return <div className="tabs-list" role="tablist">{children}</div>;
}

function Tab({ value, children }) {
  const { active, setActive } = useContext(TabsContext);
  const isActive = active === value;
  return (
    <button
      role="tab"
      aria-selected={isActive}
      className={isActive ? 'tab tab-active' : 'tab'}
      onClick={() => setActive(value)}
    >
      {children}
    </button>
  );
}

function TabsPanel({ value, children }) {
  const { active } = useContext(TabsContext);
  if (active !== value) return null;
  return <div role="tabpanel">{children}</div>;
}

// Compound exports
Tabs.List = TabsList;
Tabs.Tab = Tab;
Tabs.Panel = TabsPanel;

// 用法：
<Tabs defaultValue="profile">
  <Tabs.List>
    <Tabs.Tab value="profile">Profile</Tabs.Tab>
    <Tabs.Tab value="settings">Settings</Tabs.Tab>
  </Tabs.List>
  <Tabs.Panel value="profile"><ProfileForm /></Tabs.Panel>
  <Tabs.Panel value="settings"><SettingsForm /></Tabs.Panel>
</Tabs>

// 设计要点：
// 1. Context 共享 active state —— 子组件不需要逐层传 props
// 2. Tabs.X 是命名空间组合，DX 友好
// 3. 灵活：用户可随意编排 List / Panel 顺序、加自定义内容

TypeScript Props 完整范例

typescript · 工业级 Button 类型 
import { ReactNode, ComponentPropsWithoutRef, ButtonHTMLAttributes } from 'react';

// 基础变体定义（字符串 union 限制）
type Size = 'sm' | 'md' | 'lg';
type Variant = 'primary' | 'secondary' | 'danger' | 'ghost';

// 继承原生 button 的所有属性 —— 用户可传 onClick / disabled / type / ...
interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  size?: Size;
  variant?: Variant;
  loading?: boolean;
  leftIcon?: ReactNode;
  rightIcon?: ReactNode;
}

function Button({
  size = 'md',
  variant = 'primary',
  loading = false,
  leftIcon,
  rightIcon,
  children,
  className,
  disabled,
  ...rest
}: ButtonProps) {
  const cls = `btn btn-${size} btn-${variant} ${className ?? ''}`.trim();
  return (
    <button {...rest} className={cls} disabled={disabled || loading}>
      {loading ? <Spinner /> : leftIcon}
      {children}
      {!loading && rightIcon}
    </button>
  );
}

// 用户能用：
<Button size="lg" variant="danger" onClick={handleClick}>Delete</Button>
<Button leftIcon={<TrashIcon />} loading={isPending}>Submit</Button>
// 全部属性都有 IDE 自动补全 + 类型检查

05 常见陷阱

⚠️ 陷阱 1：直接 mutate props

props.user.name = 'X' 或 props.items.push(...)—— 看起来"能跑" 但污染 parent 持有的对象，导致跨组件状态混乱。

规避：① 把 props 当 readonly 看 ② TypeScript 配 readonly props 类型 ③ ESLint 配 react/no-direct-mutation-state（虽然主要针对 state，但思路通用）④ 想"修改" 通过回调请求 parent 改

⚠️ 陷阱 2：用 React.FC 让 children 自动可传

React.FC<Props> 历史版本会自动给 Props 加 children?: ReactNode—— 即使 Props 没声明 children 也能传。这违反类型契约。React 18.3 已修复（不再自动加），但仍不推荐 React.FC。

规避：① 不用 React.FC ② 直接函数注解：function Comp(props: Props): JSX.Element ③ 显式声明 children: ReactNode 在 Props 里

⚠️ 陷阱 3：仍写 defaultProps（React 19 警告）

React 19 给函数组件用 defaultProps 加了警告。生产构建仍工作，但开发环境 console 会刷"defaultProps will be deprecated"。

规避：用解构默认值替代。codemod：npx react-codemod replace-defaultprops-with-default-args

⚠️ 陷阱 4：用 HOC 写新代码

HOC 仍合法但不该用于新代码。问题：① 名字混乱（多层 wrap） ② displayName 容易丢 ③ ref 不自动转发（要 forwardRef，19 已退役）④ TypeScript 类型推导极复杂。

规避：所有 HOC 用例都能用自定义 Hooks 替代。"注入数据" 用 Hook + Context；"权限保护" 用 Hook + 早返回 fallback；"logger / analytics" 用 Hook + useEffect。

⚠️ 陷阱 5：spread props 透传时盖住自己的 prop

<div {...props} className="my-class" /> vs <div className="my-class" {...props} />—— 顺序不同结果不同。后写的覆盖前面的。

javascript · spread 顺序陷阱 
// 想保证自己的 className 不被覆盖：
function Card({ children, ...rest }) {
  return <div {...rest} className="card">{children}</div>;
  //         ↑ rest 里如果有 className 会被后面的 "card" 覆盖
  //         结果：用户 <Card className="my" /> 时 my 被吃掉
}

// ✅ 想合并 className 而非覆盖
import clsx from 'clsx';  // 或 classnames 库
function Card({ children, className, ...rest }) {
  return <div {...rest} className={clsx('card', className)}>{children}</div>;
  //         先解构出 className，再用 clsx 合并
}

规避：① 重要 props 单独解构，再 spread rest ② className / style 用工具库合并（clsx / cn / tailwind-merge）③ TypeScript 配严格类型让 IDE 提示冲突
