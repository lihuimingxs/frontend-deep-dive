B9 · useMemo / useCallback

useMemo 缓存计算结果，useCallback 缓存函数引用——这两个 Hook 是"性能优化" 名头下被滥用最严重的工具。事实是：大多数场景不该用。本章给清晰的"该用 vs 不该用" 决策树，讲明白浅比较机制、与 React.memo 配合的工程模式，以及 React 19 实验性的 React Compiler—— 它有可能让手动 useMemo/useCallback 退役 80%。

⚠️ "缓存就是优化" 是 React 中最大的误区之一

useMemo / useCallback 本身有 cache 维护开销（保存 deps + 比较 deps + 存结果）。简单计算 / 简单函数用它们比直接重新计算还慢—— React 团队反复在 react.dev 强调这点。react-compiler-runtime 的目标就是让人类不再纠结这个问题。

阅读时长 ~25 分钟 前置 B6 useState 深度 + B7 useEffect 对应 react.dev useMemo · useCallback

01 速览

三句话定义本章：

  1. useMemo 缓存计算结果，useCallback 缓存函数引用。useCallback(fn, deps) 等价于 useMemo(() => fn, deps)—— 是糖。两者都用浅比较 deps 数组（同 useEffect / useState）。
  2. 3 个该用场景：① 传给 React.memo 子组件（保证 props 引用稳定）② 作为其他 Hook 的依赖（避免下游 effect 反复触发）③ 真的昂贵的计算（> 1ms 或耗费内存）。其他场景都不该用—— cache 开销 > 计算开销。
  3. React Compiler（19 实验） 是革命性变化—— 编译器自动分析组件和 Hooks，自动加 memoization。开启后手动 useMemo / useCallback 在 80% 场景变得多余。当前还实验阶段（部分大公司试用），但2026-2027 大概率成为新标准。

看一段最小代码识别两个 Hook：

javascript · useMemo / useCallback 完整骨架 
function Component({ items, query }) {
  // ① useMemo：缓存计算结果
  const filtered = useMemo(
    () => items.filter(i => i.name.includes(query)),  // 计算函数（仅 deps 变时执行）
    [items, query],                                     // 依赖数组（浅比较）
  );

  // ② useCallback：缓存函数引用
  const handleClick = useCallback(
    (id) => { console.log(id); },     // 函数（仅 deps 变时新建）
    [],                                // 依赖数组（空数组 = 永远稳定）
  );

  // ③ 等价转换
  const handleClick2 = useMemo(
    () => (id) => { console.log(id); },  // useMemo 返回函数
    [],
  );
  // useCallback(fn, deps) === useMemo(() => fn, deps)

  return ...;
}

02 底层逻辑

完整签名与等价关系

typescript · useMemo / useCallback 类型 
function useMemo<T>(factory: () => T, deps: ReadonlyArray<unknown>): T;

function useCallback<T extends (...args: any[]) => any>(
  callback: T,
  deps: ReadonlyArray<unknown>,
): T;

// 行为：
// 第一次 render：调用 factory / 用 callback，存结果到 hook 节点 memoizedState
// 第二次 render：浅比较 deps（Object.is）
//   - 相等 → 返回上次缓存
//   - 不等 → 重新调用 factory / 用 new callback，更新缓存

// 关键：useMemo / useCallback 在 hooks linked list 中各占一个节点（A3 / B6）
// 所以遵循 Rules of Hooks：必须顶层调用、调用次数固定

3 个该用场景

场景 1：传给 React.memo 子组件 ⭐ 最常见

javascript · 配合 React.memo 
const ExpensiveChild = React.memo(function ExpensiveChild({ data, onAction }) {
  console.log('Child render');
  return /* 复杂渲染 */;
});

function Parent() {
  const [count, setCount] = useState(0);
  const [items, setItems] = useState([1, 2, 3]);

  // ❌ 不用 useMemo / useCallback —— Child memo 失效
  // 每次 Parent render 都新建对象 / 新建函数 → Child memo 浅比较 false → 重渲染
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <ExpensiveChild
        data={{ items }}                                // 每次新对象
        onAction={(id) => doSomething(id)}              // 每次新函数
      />
    </>
  );

  // ✅ useMemo / useCallback 让引用稳定
  const data = useMemo(() => ({ items }), [items]);
  const onAction = useCallback((id) => doSomething(id), []);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <ExpensiveChild data={data} onAction={onAction} />
      {/* count 变化时，data 和 onAction 引用不变 → memo 跳过 ExpensiveChild */}
    </>
  );
}

// 关键认知：
// - 没用 React.memo 的子组件，给它传 useMemo 后的 props 也没用 —— 子组件仍然每次重渲染
// - useMemo / useCallback 必须配合 React.memo 才有意义
// - React Compiler 19 实验性自动加这套 —— 见下文

场景 2：作为其他 Hook 的依赖

javascript · useMemo 稳定 useEffect 依赖 
function Component({ id }) {
  // ❌ 每次 render 新建 options 对象 → useEffect 反复触发
  const options = { id, includeProfile: true };
  useEffect(() => {
    fetchUser(options);
  }, [options]);   // options 引用每次都新 → effect 每次都跑

  // ✅ useMemo 稳定 options
  const options = useMemo(
    () => ({ id, includeProfile: true }),
    [id],
  );
  useEffect(() => {
    fetchUser(options);
  }, [options]);  // 仅 id 变化时重跑

  // ✅ 更简洁：直接列原始值依赖
  useEffect(() => {
    fetchUser({ id, includeProfile: true });
  }, [id]);   // 不需要 options 中间变量
  // 这通常是更好的写法 —— 避免引用稳定性问题
}

// 决策：
// - 简单依赖（原始值）→ 直接列依赖（不需要 useMemo）
// - 复杂依赖对象 / 必须传整个 object → useMemo 稳定

场景 3：真的昂贵的计算

javascript · 真昂贵的计算 
function Component({ items, query }) {
  // ✅ 真昂贵：处理 10000 项 + 复杂过滤 / 排序 / 聚合
  const stats = useMemo(() => {
    return computeExpensiveStats(items);  // 假设耗时 50ms
  }, [items]);

  // ✅ 真昂贵：解析大 JSON / 编译正则 / 加密哈希
  const parsed = useMemo(() => parseLargeJSON(rawData), [rawData]);
  const regex = useMemo(() => new RegExp(query, 'g'), [query]);

  // ❌ 不昂贵：基本类型计算 / 字符串拼接 / 简单数组操作
  const fullName = useMemo(() => `${first} ${last}`, [first, last]);
  // 字符串拼接 0.001ms，cache 维护 0.005ms —— 反而慢

  // ❌ 小数组操作
  const doubled = useMemo(() => [a, b, c].map(x => x * 2), [a, b, c]);
  // 3 项 map：0.001ms，cache 维护：0.005ms

  return ...;
}

// 怎么判断"昂贵"？
// 1. 用 React DevTools Profiler 测组件 render 时间
// 2. 用 console.time / performance.now() 测函数执行时间
// 3. > 1ms 才考虑 useMemo
// 4. 不要凭直觉 —— 大多数计算是不需要 memoize 的

何时不该用

场景 原因
基本类型计算（数字加减 / 比较） 计算开销 < cache 开销
字符串拼接 / 模板字面量 同上
小数组 map / filter（< 100 项） 同上
条件分支判断 JS 引擎分支预测远比 cache 快
没有传给 React.memo / 没作为依赖 引用稳定性没意义
所有 prop 都是原始值 原始值天然稳定
组件渲染时间已 < 16ms 没必要优化

核心原则：useMemo / useCallback 本身有固定开销—— 创建 hook 节点 + 存 deps + 浅比较 + 存结果。除非节省的计算 > 这个开销，否则就是负优化。

浅比较机制（同 useEffect / useState）

deps 数组用逐项 Object.is 比较—— 与 B7 useEffect / B6 useState 完全一致：

  • 原始值—— 比较值（数字 / 字符串 / boolean）
  • 对象 / 数组 / 函数—— 比较引用（每次新引用 = 不等 = 重算）
  • NaN vs NaN—— Object.is 等
  • +0 vs -0—— Object.is 不等
javascript · 浅比较陷阱 
function Component({ user }) {
  // ❌ user 是对象 prop —— parent 每次 render 新引用 → useMemo 每次都重算
  const greeting = useMemo(
    () => `Hello, ${user.name}`,
    [user],
  );

  // ✅ 取需要的原始字段
  const greeting = useMemo(
    () => `Hello, ${user.name}`,
    [user.name],   // 仅 name 变化时重算
  );

  // 注意：user.name 是 string（原始值）—— 引用稳定地比较

  return ...;
}

useMemo + 引用稳定性的"层叠效应"

javascript · 多层引用稳定的层叠 
function App() {
  const [count, setCount] = useState(0);

  // 第一层稳定
  const config = useMemo(
    () => ({ theme: 'dark', api: '/api' }),
    [],   // 永远稳定
  );

  // 第二层稳定 —— 依赖 config
  const heavyData = useMemo(
    () => processData(config),
    [config],  // config 永远稳定 → heavyData 永远稳定
  );

  // 第三层稳定 —— 依赖 heavyData
  const handler = useCallback(
    () => doStuff(heavyData),
    [heavyData],  // 同上 → handler 永远稳定
  );

  // 传给 memoed 子组件 —— 永不重渲染
  return <ExpensiveChild data={heavyData} onAction={handler} />;
}

// 但每多一层 useMemo / useCallback：
// - 多一个 hook 节点（运行时开销）
// - 多 deps 浅比较（CPU 开销）
// - 代码可读性下降

// 所以：仅在需要的地方用，不要"所有 useMemo 化"

与 React.memo 配合

React.memo 用浅比较 props 决定是否跳过渲染（详见 B12 Reconciliation 用户视角）：

javascript · React.memo + useMemo + useCallback 三件套 
const ExpensiveChart = React.memo(function ExpensiveChart({ data, onClick, theme }) {
  // 复杂渲染（假设 50ms）
  return /* SVG / Canvas / Three.js */;
});

function Dashboard() {
  const [filter, setFilter] = useState('all');
  const [count, setCount] = useState(0);

  const data = useMemo(
    () => transformData(rawData, filter),
    [rawData, filter],
  );

  const handleClick = useCallback((id) => {
    console.log('clicked', id);
  }, []);  // 永远稳定

  const theme = 'dark';   // 字符串字面量 —— 引用稳定

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <FilterSelector filter={filter} onChange={setFilter} />
      <ExpensiveChart
        data={data}            {/* useMemo */}
        onClick={handleClick}  {/* useCallback */}
        theme={theme}          {/* 字面量 */}
      />
    </>
  );
}

// count 变化时：
// - Dashboard render
// - data / handleClick / theme 引用不变（filter / rawData 没变）
// - ExpensiveChart memo 浅比较 → 全部 props 不变 → 跳过 render
// - 节省 50ms

// filter 变化时：
// - data 重新计算（useMemo deps 变了）
// - data 引用变了 → ExpensiveChart 浅比较 false → 重渲染（这是必要的）

// 这是性能优化的"3 件套"模式：useMemo + useCallback + React.memo

React Compiler 19 实验性 ⭐

React Compiler 是 React 19 引入的编译时优化器—— 通过分析组件源码自动加 memoization。开启后手动 useMemo / useCallback 大部分变多余：

javascript · React Compiler 自动 memoization 
// 你写：
function Component({ items, query }) {
  const filtered = items.filter(i => i.name.includes(query));
  const handleClick = (id) => console.log(id);
  return <ExpensiveChild data={filtered} onClick={handleClick} />;
}

// React Compiler 编译后（简化伪代码）：
function Component({ items, query }) {
  const $ = useMemoCache();   // 编译器内置缓存
  let filtered;
  if ($[0] !== items || $[1] !== query) {
    filtered = items.filter(i => i.name.includes(query));
    $[0] = items;
    $[1] = query;
    $[2] = filtered;
  } else {
    filtered = $[2];
  }
  let handleClick;
  if ($[3] === undefined) {
    handleClick = (id) => console.log(id);
    $[3] = handleClick;
  } else {
    handleClick = $[3];
  }
  return <ExpensiveChild data={filtered} onClick={handleClick} />;
}

// 优势：
// 1. 你不用写 useMemo / useCallback —— 代码更干净
// 2. 编译器分析比人聪明 —— 知道哪些值依赖哪些
// 3. 避免漏依赖 / 多依赖等手动错误
// 4. React.memo 也变得不必要（编译器自动决定哪些 prop 需要"稳定"）

// 实验状态（2026-04）：
// - Meta / Vercel 等大公司已部分使用
// - eslint-plugin-react-compiler 帮你检查代码是否"compiler-friendly"
// - 还有些边缘 case（refs / mutating closures）不适用
// - 2026-2027 预计稳定，可能成为 React 默认行为

// 怎么试用：
// npm install babel-plugin-react-compiler
// 在 babel.config.js 加 plugin
// 或 Next.js 15+ 配 experimental.reactCompiler: true

对当前工作的指导：

  • 新代码不要"预先 useMemo 化"—— 等 Compiler 稳定后会自动处理
  • 已有 useMemo 不必删—— Compiler 不会冲突，多一份保险
  • 遵守 Rules of Hooks + 不 mutate—— 这是 Compiler 友好的代码风格
  • 用 eslint-plugin-react-compiler—— 提前发现不兼容代码

03 应用场景

场景 A · 性能优化决策树

发现某组件渲染慢，怎么决定用什么优化工具？

  1. 用 React DevTools Profiler 测—— 找到真正慢的组件，不要凭直觉
  2. 慢组件的渲染时间是多少？
    • < 5ms：不需要优化
    • 5-50ms：可以优化（useMemo / React.memo 配合）
    • > 50ms：必须优化 + 考虑虚拟滚动 / 代码分割
  3. 慢的原因是什么？
    • render 函数本身昂贵（计算 / DOM 节点多）→ useMemo 缓存计算 + 虚拟滚动
    • render 不慢但被频繁调用 → React.memo 跳过不必要重渲染（配 useMemo / useCallback 稳定 props）
    • 子树都重渲染但只有局部变化 → 拆 Context / 用状态管理库 / 提升 state
  4. 用 React Compiler？—— 试用阶段，新项目可启用，老项目谨慎
  5. 不行就 profiling 工具栈—— Chrome DevTools / Why Did You Render / React Scan

场景 B · 自定义 Hook 返回值的引用稳定性

写自定义 Hook 时，返回值要保证引用稳定—— 避免下游 useEffect 反复触发：

javascript · 自定义 Hook 引用稳定性 
// ❌ 每次返回新对象 → 调用者 useEffect 反复触发
function useUser(id) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  // ...
  return { user, loading, refetch: () => refetchUser(id) };
  //                              ↑ 每次新建函数
}

// 调用方：
function Profile({ id }) {
  const { user, refetch } = useUser(id);
  useEffect(() => {
    refetch();   // 想 mount 时触发一次
  }, [refetch]);  // ⚠️ refetch 每次都新引用 → 死循环
}

// ✅ 用 useCallback 稳定 refetch
function useUser(id) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  const refetch = useCallback(() => refetchUser(id), [id]);

  return useMemo(
    () => ({ user, loading, refetch }),
    [user, loading, refetch]
  );
  // 仅 user / loading / refetch 变化时返回新对象
}

// 设计原则：
// 1. 自定义 Hook 返回值的引用稳定性是公共契约
// 2. setState 引用永远稳定（B6 讲过）—— 直接返回不需 wrap
// 3. 函数返回值 → useCallback
// 4. 对象返回值 → useMemo（如果用户可能解构后用 useEffect）
// 5. React Compiler 启用后这些 wrap 自动加

场景 C · 评估"是否启用 React Compiler"

团队问：新项目要启用 React Compiler 吗？ 决策因素：

  • 启用 (推荐)：
    • 新项目 + 团队接受实验性技术
    • 愿意配 eslint-plugin-react-compiler 修代码
    • 用 Next.js 15+ 或 Vite + Babel 等支持 Compiler 的栈
    • 愿意持续跟 Compiler 进展（仍在演进）
  • 不启用：
    • 老项目大量 ref / mutation 代码（Compiler 不兼容）
    • 团队不习惯实验功能 + 升级风险厌恶
    • 已经手动 useMemo / useCallback 调优过

2026 推荐：新项目试用 + 持续观察。Compiler 应该 2027 年成为新事实标准。

05 常见陷阱

⚠️ 陷阱 1："所有计算都 useMemo 化" 反而变慢

const fullName = useMemo(() => first + last, [first, last])—— 字符串拼接 0.001ms，cache 维护 0.005ms，5 倍慢。代码还更难读。

规避：用 React DevTools Profiler 测，**仅当计算 > 1ms 才用 useMemo**。直接计算优于过度缓存。

⚠️ 陷阱 2：useMemo 不传依赖（每次都算）

useMemo(() => expensive())—— 不传依赖数组等于"每次 render 都算"，cache 完全无效。

规避：必须传 deps 数组（即使是 []）。ESLint react-hooks/exhaustive-deps 自动检查。

⚠️ 陷阱 3：useMemo 的 deps 包含每次新建的对象

useMemo(() => ..., [{ x: 1 }])—— deps 数组里的对象每次新引用 → 永不相等 → 每次都重算 → cache 完全无效。

规避：deps 里只放原始值或已 useMemo 稳定的引用。复杂对象用 JSON.stringify 或解构出原始字段。

⚠️ 陷阱 4：useCallback 配合非 memo 子组件没意义

useCallback(handler, []) + 子组件没 React.memo—— 子组件每次仍然重渲染（父重渲染让子重渲染，B1 讲过）。useCallback 完全白费。

规避：useCallback / useMemo 必须配 React.memo 才有意义。要么 3 个一起用，要么都不用。

⚠️ 陷阱 5：useMemo 缓存的对象导致 mutation 风险

const obj = useMemo(() => ({ x: 1 }), [])—— 然后某处 mutate obj.x = 2。下次 render 仍返回同一对象（mutate 后的）—— 行为难调试。

规避：useMemo 缓存的值视为不可变。要修改？重新 setState 触发 deps 变化让 useMemo 重算。或加 Object.freeze 在 dev 环境。
