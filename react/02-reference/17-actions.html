B17 · Actions + useFormStatus + useOptimistic

React 19 引入的 Actions 是过去 12 年里 React 第一次把"异步操作" 设计成一等公民—— 之前所有"提交表单 → 显示 loading → 处理错误 → 乐观更新" 模式都得自己用 useState 拼。React 19 给出三件套：useActionState（替代 state + try/catch + loading 状态机）、useFormStatus（子组件感知父 form pending，不需要 prop 传递）、useOptimistic（乐观更新 + 失败自动 rollback）。本章讲清这三个 Hook 各自的边界、与 B16 Server Actions 的关系、和 React Hook Form 这种"老世界" 库的取舍。

⚠️ Action ≠ Server Action

Action（广义）：任何传给 form action prop 或 startTransition 的异步函数—— React 自动把它包成 transition（B14），自动管理 pending 状态。Server Action（狭义）：在 Server Component 或 'use server' 模块里声明的、客户端可调用的服务器端函数（B16）。Action 是机制，Server Action 是它在 RSC 场景下的特化。非 RSC 项目（CSR / 普通 SSR）也能用 Actions—— 把异步函数传给 form action 即可。

阅读时长 ~30 分钟 前置 B14 useTransition（Action 内部就是 transition） + B16 Server Components（Server Actions 来源） 对应 react.dev useActionState · useFormStatus · useOptimistic RFC #229 First-class support for promises and async/await

01 速览

四句话定义本章：

  1. Action 是任何 async 函数，传给 <form action={fn}> 或 startTransition(async () => ...)。React 自动把它包成 transition（B14）—— 即更新被标记为 TransitionLane（低优先级、可被打断、不阻塞 UI）。React 19 之前 form 提交要写 onSubmit={e => { e.preventDefault(); ... }}，19 之后直接 action={fn}，React 自动 preventDefault + 调用函数 + 处理 pending。
  2. useActionState(action, initialState) = 老世界 useState + try/catch + loading + setError 的"四合一"。返回 [state, formAction, isPending]。state 是 action 上次返回值，formAction 传给 <form action>，isPending 自动管理。错误也是 state—— action 内 throw 会冒泡到最近 ErrorBoundary（B15），但通常用 return 错误对象更友好。
  3. useFormStatus()（注意：从 react-dom 导入，不是 react）让子组件不通过 prop drilling 就能感知父 form 的 pending 状态。返回 { pending, data, method, action }。典型用法：在 <SubmitButton /> 内 const { pending } = useFormStatus()，无需父组件传 disabled。
  4. useOptimistic(state, updateFn) 是乐观更新原语：在 transition 期间显示预测的乐观值，transition 结束后自动回到真实 state。失败自动 rollback（不需要手动 catch + reset），因为乐观值只在 transition 内有效。常用于 like / vote / 评论等"用户预期会成功" 的场景。

02 底层逻辑

1. 老世界的"四件事" 与 Action 的"一件事"

React 18 之前提交一个表单要管理 4 个独立状态： 

// React 18 时代典型的"四状态机"
function CreatePost() {
  const [title, setTitle] = useState('');     // ① 表单值
  const [loading, setLoading] = useState(false); // ② loading
  const [error, setError] = useState(null);      // ③ error
  const [data, setData] = useState(null);        // ④ result

  async function handleSubmit(e) {
    e.preventDefault();
    setLoading(true);
    setError(null);
    try {
      const res = await fetch('/api/posts', {
        method: 'POST',
        body: JSON.stringify({ title }),
      });
      if (!res.ok) throw new Error(await res.text());
      setData(await res.json());
      setTitle(''); // reset
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input value={title} onChange={e => setTitle(e.target.value)} />
      <button disabled={loading}>{loading ? 'Saving…' : 'Save'}</button>
      {error && <p style={{color:'red'}}>{error}</p>}
    </form>
  );
}
// ❌ 4 个 useState / 必须 e.preventDefault / try/catch/finally / 手动管 loading

React 19 重写后：

// React 19 — useActionState 折叠四件事
async function createPost(prevState, formData) {
  const title = formData.get('title') as string;
  if (!title?.trim()) return { error: 'Title required' };
  const res = await fetch('/api/posts', {
    method: 'POST',
    body: JSON.stringify({ title }),
  });
  if (!res.ok) return { error: await res.text() };
  return { data: await res.json(), error: null };
}

function CreatePost() {
  const [state, formAction, isPending] = useActionState(createPost, { error: null });
  return (
    <form action={formAction}>  {/* 自动 preventDefault */}
      <input name="title" />     {/* 不需要受控（FormData 自动取） */}
      <button disabled={isPending}>{isPending ? 'Saving…' : 'Save'}</button>
      {state?.error && <p style={{color:'red'}}>{state.error}</p>}
    </form>
  );
}
// ✅ 1 个 Hook 替代 4 个 / 不需要 onSubmit / 不需要受控 / pending 自动

2. form action prop：React 19 的"新 onSubmit"

原生 HTML <form action="/api/x" method="POST"> 提交时浏览器跳页 + 全页刷新。React 19 接管 action prop：

  • action 是字符串 → 走原生 HTML form 行为（仍跳页）
  • action 是函数 → React 接管：① 自动 preventDefault ② 收集 FormData 传入 ③ 用 startTransition 包裹调用 ④ 提交期间该 form 的 pending 状态可被 useFormStatus 读到 ⑤ 提交完成后自动 reset 表单（uncontrolled 字段）
  • action 是 Server Reference（B16 'use server' 转出来的）→ 上面所有 + RPC 序列化 + 调用 + 反序列化结果

3. useActionState 内部机制（关键不变量）

const [state, formAction, isPending] = useActionState(action, initialState, permalink?)

  • action 签名固定为 (prevState, formData) => nextState（如果配 form action）或 (prevState, ...args) => nextState（手动调用 formAction）。第一个参数是上次的返回值，不是表单数据—— 这是最容易写错的地方。
  • state 在每次 action 完成后更新为返回值—— 即"action 是 state reducer"。这让"多次提交累积错误" / "记住上次提交内容" 这类需求很自然。
  • action 抛错不被 useActionState 捕获，会冒泡到最近 ErrorBoundary（B15）。**生产中推荐 return 错误对象**而非 throw —— 错误是数据流的一部分，不是异常。
  • permalink 第三个参数（可选，String）—— RSC + 渐进增强场景下，JS 还没加载时表单提交跳转到的 URL。
  • isPending 自动管理—— action 调用期间为 true，结果到达后变 false。同一个 useActionState 同时只允许一个 action 执行（自动排队，避免竞态）。

4. useFormStatus：感知最近父 form 的状态

const { pending, data, method, action } = useFormStatus()

  • 必须在 form 内部的子组件里调用—— 它读取的是祖先链上最近的 <form> 元素的提交状态。直接在 form 所在组件用是读不到的（只能读祖先 form，不能读自己渲染的 form）。
  • 从 react-dom 导入，不是 react。原因：它依赖 DOM 概念（form 元素），react 包要保持 platform-agnostic。
  • 替代 prop drilling—— 老世界要把 loading prop 一路传给 <SubmitButton> 才能 disable。新世界子组件自查询。
  • data 是 FormData 实例（pending 期间）—— 可读取正在提交的字段值，做"提交中" 预览。

5. useOptimistic：乐观更新 + transition 边界

const [optimisticValue, addOptimistic] = useOptimistic(realValue, updateFn?)

  • optimisticValue 在 transition 内 = 乐观值，transition 结束 = realValue。所以失败 rollback 是"免费" 的—— 没有真正存数据，只是 transition 期间显示预测。
  • updateFn 签名 (currentState, optimisticArg) => nextOptimisticState—— 类似 reducer，描述"给定当前状态和乐观参数，返回乐观状态"。
  • 必须在 transition 内调用 addOptimistic—— 通常配合 startTransition 或 <form action> 使用。在 transition 外调用 addOptimistic 会被忽略 / 报警告。
  • 不替代 useState—— useOptimistic 持有的是"视图层叠加" 状态。真实状态仍在原 state（来自 props / context / Server Component）。
  • 多个并发 optimistic 更新可叠加—— 用户连点 5 次 like，5 个乐观值合并显示。

6. 三个 Hook 的协同关系

// 完整组合：useActionState + useFormStatus + useOptimistic
'use client';
import { useActionState, useOptimistic } from 'react';
import { useFormStatus } from 'react-dom';
import { addComment } from './actions';

// SubmitButton 在 form 子树中读 form 的 pending
function SubmitButton() {
  const { pending } = useFormStatus();
  return <button disabled={pending}>{pending ? 'Posting…' : 'Post'}</button>;
}

export default function Comments({ initialComments }) {
  // 真实评论列表（来自 Server Component props）
  const [comments, formAction] = useActionState(
    async (prev, formData) => {
      const text = formData.get('text');
      const newComment = await addComment(text);
      return [...prev, newComment];
    },
    initialComments
  );

  // 乐观叠加层
  const [optimisticComments, addOptimistic] = useOptimistic(
    comments,
    (state, text) => [...state, { id: 'pending', text, pending: true }]
  );

  return (
    <>
      <ul>
        {optimisticComments.map(c =>
          <li key={c.id} style={c.pending ? {opacity:0.5} : {}}>{c.text}</li>
        )}
      </ul>
      <form
        action={async (formData) => {
          addOptimistic(formData.get('text'));  // 乐观叠加
          await formAction(formData);            // 触发真正 action
        }}
      >
        <input name="text" />
        <SubmitButton />  {/* 自动读 pending */}
      </form>
    </>
  );
}
// ✅ 三件套清晰分工：useActionState 管真实 state + isPending + 错误
//                   useFormStatus 让按钮自查 pending
//                   useOptimistic 叠加乐观显示

03 应用场景

场景 1：登录表单（最经典 useActionState）

'use client';
import { useActionState } from 'react';

async function loginAction(prevState, formData) {
  const email = formData.get('email');
  const password = formData.get('password');
  const res = await fetch('/api/login', {
    method: 'POST',
    body: JSON.stringify({ email, password }),
  });
  if (!res.ok) {
    return {
      error: '账号或密码错误',
      lastEmail: email, // 保留输入，下次显示
    };
  }
  window.location.href = '/dashboard';
  return { error: null };
}

export default function LoginForm() {
  const [state, formAction, isPending] = useActionState(loginAction, {
    error: null,
    lastEmail: '',
  });
  return (
    <form action={formAction}>
      <input name="email" defaultValue={state?.lastEmail} required />
      <input name="password" type="password" required />
      <button disabled={isPending}>{isPending ? '登录中…' : '登录'}</button>
      {state?.error && <p className="error">{state.error}</p>}
    </form>
  );
}
// 收益：① 失败时邮箱保留 ② 提交期间按钮自动 disabled ③ 不写 useState

场景 2：评论列表（useOptimistic 立即响应）

提交评论瞬间显示在列表，不等服务器返回。失败自动消失（rollback）。 

'use client';
import { useOptimistic } from 'react';
import { addComment as addCommentAction } from './actions';

export default function Comments({ comments }) {
  const [optimistic, addOptimistic] = useOptimistic(
    comments,
    (state, newText) => [
      ...state,
      { id: `temp-${Date.now()}`, text: newText, pending: true },
    ]
  );

  async function submit(formData) {
    const text = formData.get('text');
    addOptimistic(text);  // 立即叠加
    await addCommentAction(text);  // 真正提交
    // transition 结束 → optimistic 自动回到 comments（已经被 revalidate 更新）
  }

  return (
    <>
      {optimistic.map(c => (
        <li key={c.id} className={c.pending ? 'opacity-50' : ''}>{c.text}</li>
      ))}
      <form action={submit}>
        <input name="text" />
        <button>Post</button>
      </form>
    </>
  );
}

04 典型代码

例 1：startTransition 直接调用 action（不用 form）

// 不是所有 action 都通过 form 触发——按钮点击也行
'use client';
import { startTransition, useState } from 'react';
import { deletePost } from './actions';

function DeleteButton({ postId }) {
  const [isPending, setPending] = useState(false);
  return (
    <button
      onClick={() => {
        startTransition(async () => {
          setPending(true);
          await deletePost(postId);
          setPending(false);
        });
      }}
      disabled={isPending}
    >
      {isPending ? 'Deleting…' : 'Delete'}
    </button>
  );
}

// React 19 改进：startTransition 内可以 await（async transition）
// 整个 await 期间被标为 pending transition

例 2：复杂的 form state（错误对象 / 字段级错误）

// state 是任意结构——可以包含字段级错误
type FormState = {
  errors?: { email?: string; password?: string; general?: string };
  values?: { email?: string };  // 保留输入
};

async function signupAction(prev: FormState, formData: FormData): Promise<FormState> {
  const email = formData.get('email') as string;
  const password = formData.get('password') as string;

  const errors: FormState['errors'] = {};
  if (!email.includes('@')) errors.email = '请输入有效邮箱';
  if (password.length < 8) errors.password = '密码至少 8 位';
  if (Object.keys(errors).length) {
    return { errors, values: { email } };
  }

  try {
    await api.signup(email, password);
    return { errors: {} };
  } catch (e) {
    return {
      errors: { general: e.message },
      values: { email },
    };
  }
}

function Signup() {
  const [state, formAction, isPending] = useActionState(signupAction, {});
  return (

      {state.errors?.password && <Error>{state.errors.password}</Error>}

例 3：useOptimistic 多状态叠加（投票 like）

'use client';
import { useOptimistic } from 'react';
import { vote } from './actions';

export default function VoteButton({ postId, initialCount, initialVoted }) {
  // 同时管"是否已投票" 和"票数" 两个值
  const [optimistic, addOptimistic] = useOptimistic(
    { count: initialCount, voted: initialVoted },
    (state, action: 'up' | 'down') => ({
      count: state.count + (action === 'up' ? 1 : -1),
      voted: action === 'up',
    })
  );

  return (
    <form
      action={async () => {
        const action = optimistic.voted ? 'down' : 'up';
        addOptimistic(action);
        await vote(postId, action);
      }}
    >
      <button>
        {optimistic.voted ? '👍' : '🤍'} {optimistic.count}
      </button>
    </form>
  );
}
// 用户连点 5 次：5 个乐观更新依次叠加，每次点击都立即看到反馈
// 服务器返回后真实 count 与 voted 替换，自然过渡

05 常见陷阱

陷阱 1：以为 useActionState action 第一参数是 formData

现象：action(formData) 拿到的是上次的 state 对象 / null，而不是 FormData，console.log 的全是错的。

根因：useActionState 把 action 包装成 reducer 风格——第一个参数永远是 prevState，第二个才是 formData / 你传的参数。这是为了让"action 累积返回值" 模式自然。

规避：永远写 async function action(prevState, formData) { ... }。如果不需要 prevState，也要把第一参数留出来（用 _ 占位）。

陷阱 2：useFormStatus 在 form 自身组件里调用

现象：useFormStatus() 永远返回 { pending: false }，不管 form 在不在提交。

根因：useFormStatus 读的是祖先链最近的 form。如果在 form 的同一组件里调用，自身 form 还没在 React tree 里出现 / 不在它的祖先链上。 

// ❌ 错：在渲染 form 的同一组件读 useFormStatus
function MyForm() {
  const { pending } = useFormStatus(); // 永远 false
  return (
    <form action={fn}>
      <button disabled={pending}>Save</button>
    </form>
  );
}

// ✅ 对：把按钮抽成 form 的子组件
function SubmitButton() {
  const { pending } = useFormStatus();  // OK，最近祖先 form 提交时为 true
  return <button disabled={pending}>Save</button>;
}
function MyForm() {
  return (
    <form action={fn}>
      <SubmitButton />
    </form>
  );
}

陷阱 3：useOptimistic 在 transition 外调用 addOptimistic

现象：addOptimistic 调用后视图没变化，控制台报警告 An optimistic state update occurred outside a transition or action.

根因：useOptimistic 的乐观值只在当前 transition 内有效—— 因为 transition 结束后 React 要"放弃乐观值"，不在 transition 内就找不到回退点。

规避：addOptimistic 必须在 <form action> 提交、startTransition 内、或 useActionState 的 action 函数内调用。最常见的写法： 

// ❌ onClick 直接调
<button onClick={() => { addOptimistic(...); doApiCall(); }}>
  Like
</button>

// ✅ 包在 startTransition 里
<button onClick={() => {
  startTransition(async () => {
    addOptimistic(...);
    await doApiCall();
  });
}}>Like</button>

// ✅ 或者用 form action（最干净）
<form action={async () => {
  addOptimistic(...);
  await doApiCall();
}}>
  <button>Like</button>
</form>

陷阱 4：把 controlled input 和 form action 混用

现象：用 value + onChange 受控的 input，提交后想 reset 但 input 不清空。

根因：<form action> 提交后仅自动 reset uncontrolled input（依赖原生 form.reset()）。受控 input 由 React state 控制，state 不变 → input 不变。

规避：① 优先非受控（用 defaultValue + name，FormData 取值）—— 这是 19 时代推荐写法 ② 必须受控时，在 action 返回成功后手动 setValue('') ③ 给 form 加 key 强制重建（比 state 重置成本低）

陷阱 5：Action 抛错后 UI 没反应（被 ErrorBoundary 吞了）

现象：用户操作失败但页面没显示错误，或整个区块被 ErrorBoundary 替换为 fallback。

根因：useActionState 不捕获 action 抛出的错误—— 错误冒泡到最近 ErrorBoundary（B15）。如果你只想显示一个友好提示而不是 unmount 整个组件，不应该 throw，应该 return 一个错误对象。

规避：分层处理 —— ① 业务校验 / 网络错误 / 用户输入错 → return { error: msg }，state 显示 ② 真正不可恢复错误（数据损坏 / 程序 bug）→ throw，让 ErrorBoundary 兜底

陷阱 6：和 React Hook Form 重复造轮子

现象：项目里同时用了 useActionState 和 React Hook Form，两套逻辑混乱。

取舍建议：

  • 简单表单（提交 + loading + 错误展示）：useActionState 完全够，不引入 RHF
  • 复杂表单（多字段联动 / 动态字段 / 实时校验 / array fields）：仍用 React Hook Form（提供 register / watch / fieldArray 等强大原语），但提交时调用 Action（handleSubmit(action)）
  • RSC 项目：useActionState + zod 校验是新最佳实践，比 RHF 更适合 Server Actions（type-safe + 渐进增强 + 零 client bundle）
  • Next.js App Router 大型项目：可以全 useActionState + Conform 库（conform.guide）—— 服务器优先 + zod 校验 + 渐进增强
