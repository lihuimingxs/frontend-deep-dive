B2 · JSX 编译与运行时

JSX 是 React 最有标志性、也是最被误解的特性。它不是模板语言（不像 Vue SFC / Svelte SFC），而是JavaScript 表达式的语法糖—— <div>Hello</div> 在编译后就是一个普通函数调用。本章讲清 JSX 的编译机制（新 / 旧 runtime）、插值规则、命名约定，以及它在 TypeScript / Emotion / 自定义 runtime 中的集成方式。读完后你能解释"为什么 JSX 不是 HTML" 和"为什么 React 17 之后不再需要 import React"。

📐 JSX 设计动机请回看 A1 / A2

"JSX 为什么存在"已在 A1 起源与演化 讲过——2013 JSConf 引发的争议 + Pete Hunt "Rethinking Best Practices" 论证。"JSX 是 JS 表达式 vs 模板" 的设计哲学差异在 A2 设计哲学。本章不重复，只讲编译细节。

阅读时长 ~25 分钟 前置 B1 渲染模型 + ES2015 语法 对应 react.dev Writing Markup with JSX · New JSX Transform (2020)

01 速览

三句话定义本章：

  1. JSX 是 JS 的表达式语法糖，不是模板。<div /> 编译后是函数调用，结果是 React 元素对象。JSX 出现在任何 JS 表达式可以出现的位置—— 函数返回值 / 三元 / map 回调 / 变量赋值 / 函数参数。这是 React Composability 的语言层根基。
  2. 有两种编译 runtime：classic runtime（React 16 及之前的默认）编译为 React.createElement(...)，必须 import React；new JSX runtime（React 17+ 推荐）编译为 jsx(...) / jsxs(...)，不需要 import React，bundle 更小，性能更好。新项目都用新 runtime。
  3. JSX 的"奇怪"都是 JS 限制 + React 选择的结果：① className 而非 class（class 是 JS 关键字）② htmlFor 而非 for（for 是 JS 关键字）③ style 是对象不是字符串（避免拼接 + 类型安全）④ 大小写区分（小写 → host 元素，大写 → 组件）⑤ 注释用 {/* */}（不能用 HTML 注释）。

看一段 JSX 编译前后对比：

javascript · JSX vs 编译产物 
// 编写：
function Greeting({ name }) {
  return (
    <div className="card">
      <h1 onClick={() => alert('hi')}>Hello {name}</h1>
      {name === 'admin' && <Badge />}
    </div>
  );
}

// 新 JSX runtime 编译后（React 17+）：
import { jsx as _jsx, jsxs as _jsxs } from 'react/jsx-runtime';
function Greeting({ name }) {
  return _jsxs('div', {
    className: 'card',
    children: [
      _jsx('h1', { onClick: () => alert('hi'), children: ['Hello ', name] }),
      name === 'admin' && _jsx(Badge, {}),
    ],
  });
}

// 关键观察：
// 1. JSX 完全消失，剩下的是普通函数调用
// 2. children 在 props 对象里
// 3. 组件名（Badge）变成函数引用
// 4. 表达式（{...}) 变成普通 JS 值传入

02 底层逻辑

JSX 编译两种 runtime

Classic Runtime（React 16 及之前默认）

每个 JSX 元素编译为 React.createElement 调用。需要在每个文件顶部 import React—— 即使你不直接用 React 变量。

javascript · classic runtime 编译 
// 编写
import React from 'react';   // ← 必须 import React，否则编译报错

function App() {
  return (
    <div className="app">
      <h1>Title</h1>
      <p>Body</p>
    </div>
  );
}

// classic runtime 编译后：
const React = require('react');
function App() {
  return React.createElement(
    'div',
    { className: 'app' },
    React.createElement('h1', null, 'Title'),
    React.createElement('p', null, 'Body'),
  );
}

// createElement 签名：
// React.createElement(type, props, ...children)
//   - children 作为可变参数，运行时拼成数组
//   - 每次调用都创建一个 props 对象副本
//   - 性能：每次 render 都跑这个 createElement 链

Classic runtime 的问题：① 每个文件必须 import React—— 即使只用了 JSX 没直接用 React 变量。这是常见的"已 import 但未使用" 警告 ② tree-shaking 不彻底—— 整个 React 都被引用 ③ 性能差—— children 用可变参数拼成数组每次都新建。

New JSX Runtime（React 17+ 推荐）

React 17 引入新 JSX runtime。编译器自动 import jsx / jsxs / Fragment，不再需要手动 import React：

javascript · new JSX runtime 编译 
// 编写（注意：不需要 import React）
function App() {
  return (
    <div className="app">
      <h1>Title</h1>
      <p>Body</p>
    </div>
  );
}

// new JSX runtime 编译后（React 17+）：
import { jsx as _jsx, jsxs as _jsxs } from 'react/jsx-runtime';

function App() {
  return _jsxs('div', {
    className: 'app',
    children: [
      _jsx('h1', { children: 'Title' }),
      _jsx('p', { children: 'Body' }),
    ],
  });
}

// 区别：
// - jsx：单个 child（或 0/1 child）
// - jsxs：多个 children（s = static array）
// - children 直接放进 props 对象，不再可变参数
// - 编译器自动决定用 jsx 还是 jsxs（基于 children 数量）
// - import 是编译器加的，用户代码不需要写

新 runtime 的优势：

  • 不需要 import React—— 文件更干净，新人不会问"为什么必须 import React"
  • 更小 bundle—— jsx 函数比 createElement 简单（不处理可变参数 / 不做 ref / key 警告）
  • tree-shaking 友好—— 只 import 用到的函数，不引整个 React
  • 类型推导更精确—— 配合 TypeScript jsx: "react-jsx" 选项
  • 错误信息更好—— jsx-dev-runtime 在开发时给出更精确的位置信息

怎么启用新 runtime：

配置 · 启用 new JSX runtime 
// tsconfig.json（TypeScript 项目）
{
  "compilerOptions": {
    "jsx": "react-jsx",      // ← 新 runtime（"react" 是 classic）
    "jsxImportSource": "react"  // ← 默认是 react，下文讲怎么自定义
  }
}

// .babelrc（Babel 项目）
{
  "presets": [
    ["@babel/preset-react", { "runtime": "automatic" }]  // ← automatic 是新 runtime
  ]
}

// vite.config.ts（Vite 默认就是新 runtime，无需配置）

// next.config.js（Next.js 默认就是新 runtime，无需配置）

// 老项目升级：
// 1. tsconfig.jsx 从 "react" 改 "react-jsx"
// 2. 移除所有不必要的 import React from 'react'
// 3. 用 codemod：npx react-codemod update-react-imports

jsxImportSource —— 自定义 runtime 来源

新 JSX runtime 默认从 react/jsx-runtime import jsx 函数。但这个来源可以替换—— 让 JSX 编译为其他库的 jsx 函数。这就是 jsxImportSource 配置。

典型用例：

javascript · jsxImportSource 用法 
// 1. Emotion CSS-in-JS
// tsconfig.json
{ "jsx": "react-jsx", "jsxImportSource": "@emotion/react" }
//                       ↑ 编译后 import 来自 @emotion/react/jsx-runtime
// 这个 jsx 函数支持额外的 css prop:
<div css={{ color: 'red' }}>Hello</div>
// 编译后调用 emotion 的 jsx，处理 css prop 并生成 styled component

// 2. Theme UI（已废弃但仍可见于老代码）
{ "jsx": "react-jsx", "jsxImportSource": "theme-ui" }

// 3. Preact（用 React API 但跑 Preact runtime）
{ "jsx": "react-jsx", "jsxImportSource": "preact" }
// import { jsx } from 'preact/jsx-runtime'
// 让 React 代码直接跑 Preact（比 React 小 80%，几乎完全兼容 API）

// 4. 文件级覆盖（pragma 注释）
/** @jsxImportSource @emotion/react */
function MyEmotionComponent() {
  return <div css={{ color: 'red' }}>Hello</div>;
}
// 这一文件用 emotion，其他文件用默认 react

jsxImportSource 的工程意义：让"JSX 语法" 和"JSX 运行时" 解耦。同一份 JSX 源码可以跑 React、Preact、Emotion 增强版、自家自定义渲染器—— 不需要改业务代码。

Fragment 的三种写法

组件必须返回单个根元素—— 但实际经常需要"多个相邻元素无包装 div"。Fragment 解决这个：

javascript · Fragment 三种写法 
// 写法 1：<React.Fragment>（最完整）
import { Fragment } from 'react';
function App() {
  return (
    <Fragment>
      <h1>Title</h1>
      <p>Body</p>
    </Fragment>
  );
}
// 唯一支持 key prop 的写法（用在 list 中）
<Fragment key={item.id}>...</Fragment>

// 写法 2：<>...</>（短语法）
function App() {
  return (
    <>
      <h1>Title</h1>
      <p>Body</p>
    </>
  );
}
// 不支持 key —— 在 list 中需要 key 时退回到 <Fragment key={...}>

// 写法 3：返回数组（不推荐）
function App() {
  return [
    <h1 key="title">Title</h1>,
    <p key="body">Body</p>,
  ];
}
// 必须每个元素有 key
// 可读性差，不推荐

// 编译后（new runtime）：
// <Fragment>X</Fragment> → _jsx(Fragment, { children: X })
// 其中 Fragment 是 Symbol(react.fragment)，特殊处理（不渲染包装节点）

JSX 表达式插值规则

{} 内可以放任何 JS 表达式—— 包括函数调用、三元、map / filter / reduce、对象 spread 等：

javascript · JSX 插值能放什么 
function Demo({ items, user, isAdmin }) {
  const greeting = `Hello, ${user.name}`;
  return (
    <div>
      {/* 字符串 / 数字 / 表达式 */}
      <h1>{greeting}</h1>
      <p>{user.age * 2}</p>

      {/* 三元 */}
      {isAdmin ? <AdminPanel /> : <UserPanel />}

      {/* 短路 &&（注意陷阱：见下文）*/}
      {isAdmin && <AdminBadge />}

      {/* 数组（要 key） */}
      <ul>
        {items.map(item => <li key={item.id}>{item.text}</li>)}
      </ul>

      {/* IIFE（少用，不推荐 —— 倾向于提取子组件）*/}
      {(() => {
        if (cond1) return <A />;
        if (cond2) return <B />;
        return null;
      })()}

      {/* spread 属性 */}
      <Modal {...modalProps} />

      {/* 多个表达式拼一起（要用空格 / template literal）*/}
      <div>{user.firstName} {user.lastName}</div>
      <div>{`${user.firstName} ${user.lastName}`}</div>

      {/* 不能放：}
      {/* if 语句（不是表达式）*/}
      {/* for 循环（不是表达式）*/}
      {/* 对象字面量（要双花括号）*/}
      <div style={{ color: 'red' }}>...</div>
      {/*       ↑↑ 外层 {} 是 JSX 插值 */}
      {/*          ↑ 内层 {} 是对象字面量 */}
    </div>
  );
}

// 不会渲染的值（自动跳过）：
// - false / null / undefined
// - true（也会跳过！）
// - 空数组 / 空字符串 ''
// 例：{isLoading && ...} 当 isLoading 是 false 时跳过，不渲染 'false' 文本

// ⚠️ 会渲染但常出 bug 的值：
// - 0（会渲染为 '0' 文本！）—— 这是 && 短路陷阱的根因
{items.length && <List items={items} />}
// 当 items 是空数组（length===0），渲染 '0' 而非空 ⚠️
// 修复：{items.length > 0 && ...} 或 {items.length ? ... : null}

JSX 在 TypeScript 中的集成

短重述：TS 通过 tsconfig.jsx + JSX.IntrinsicElements 类型集成 JSX。jsx: "react-jsx"（新 runtime）/ "react-jsxdev"（开发版，含 source 信息）/ "preserve"（不编译，留给后续 Babel）/ "react"（classic）/ "react-native"（RN 专用）。

详细机制请参考 TS · 16 Iterators / JSX—— 那里详述了 JSX.Element / JSX.IntrinsicElements / React.FC 类型 / 事件类型推导。本章只点本质：

typescript · JSX 类型集成最小例子 
// React 自动扩展全局 JSX namespace
declare global {
  namespace JSX {
    interface IntrinsicElements {
      div: React.HTMLAttributes<HTMLDivElement>;
      button: React.ButtonHTMLAttributes<HTMLButtonElement>;
      // ... 所有 host 元素都在这里
    }
  }
}

// 这就是为什么 <div onClick={fn} /> 中 fn 类型是 (e: React.MouseEvent) => void
// 由 IntrinsicElements 的 div 类型推导

// 用户可以扩展（自定义 host 元素，如 Web Components）：
declare module 'react' {
  namespace JSX {
    interface IntrinsicElements {
      'my-custom-element': { customAttr?: string; children?: ReactNode };
    }
  }
}
// 之后就可以 <my-custom-element customAttr="x" /> 不报错

JSX 命名规则与限制

完整命名规则：

  • 小写开头（<div />）→ host 元素，type 是字符串
  • 大写开头（<MyButton />）→ 组件，type 是变量引用，必须在作用域内
  • 带连字符（<my-element />）→ 视为 host 元素（Web Components）
  • 带点（<Form.Input />）→ 属性访问，组件，等价 Form.Input
  • 泛型（<Select<string> />）→ TS 才支持，传类型参数

JSX 不能做的事：

  • 语句—— 不能直接放 if / for / while。要用三元 / map / 提取子组件
  • 注释—— 不能用 <!-- -->（HTML 注释）。用 {/* ... */}
  • 关键字 prop—— class / for / catch 是 JS 关键字，不能直接用作 prop。改名 className / htmlFor / catchHandler 等
  • style 字符串—— style 必须是对象 {{ color: 'red' }}，不能是字符串 "color: red"

03 应用场景

场景 A · 老项目升级到 New JSX Runtime

React 16 项目，每个组件文件顶部都 import React from 'react'（即使只用了 JSX）。升级到 17+ 后想清理这些 import：

bash · 自动化迁移 
// 1. 升级 React 到 17.0+ / 18.x / 19.x
npm install react@latest react-dom@latest

// 2. 改 tsconfig.json
{ "compilerOptions": { "jsx": "react-jsx" } }

// 3. 移除不必要的 import（codemod 自动处理）
npx react-codemod update-react-imports

// 处理结果：
// - 仅有 <JSX> 没用 React 变量的文件 → 删除 import React
// - 用了 React.useState 等 → 改为 import { useState } from 'react'
// - 还在用 React.createElement 等 → 保留 import

// 4. ESLint 规则更新
{
  "rules": {
    "react/react-in-jsx-scope": "off",        // 不再要求 import React
    "react/jsx-uses-react": "off"
  }
}

场景 B · 用 Emotion CSS-in-JS（jsxImportSource）

团队选 Emotion 做 CSS-in-JS，想用它的 css prop（<div css={{...}} />）：

javascript · Emotion + jsxImportSource 
// 1. 安装
npm install @emotion/react

// 2. tsconfig.json（项目级）
{
  "compilerOptions": {
    "jsx": "react-jsx",
    "jsxImportSource": "@emotion/react"
  }
}

// 3. 用 css prop
import { css } from '@emotion/react';

const style = css`
  color: red;
  font-size: 18px;
`;

function MyComponent() {
  return (
    <div css={style}>        {/* css prop 现在合法 */}
      Hello
    </div>
  );
}

// 或文件级（不想全局改 tsconfig）：
/** @jsxImportSource @emotion/react */
import { css } from '@emotion/react';
// 仅这个文件用 emotion runtime

场景 C · 用 Preact 替代 React（90% 兼容 + 80% 更小）

性能极致追求 / bundle 体积敏感的场景（如 SDK / 嵌入式 widget），用 Preact 替代 React：

javascript · Preact 替代 React 
// 1. 安装
npm install preact
npm uninstall react react-dom

// 2. tsconfig.json
{
  "compilerOptions": {
    "jsx": "react-jsx",
    "jsxImportSource": "preact"
  }
}

// 3. vite.config.ts（用 alias 让 import 'react' 指向 preact）
import { defineConfig } from 'vite';
export default defineConfig({
  resolve: {
    alias: {
      'react': 'preact/compat',
      'react-dom': 'preact/compat',
    },
  },
});

// 4. 业务代码完全不变 —— 仍写 React 风格代码
// 但 bundle 大小：React + react-dom ~120KB → Preact ~10KB

// ⚠️ 局限：
// - 部分高级 API（Concurrent / RSC）Preact 支持有限
// - 第三方库的"react-only" hook 可能不工作
// - 适合 SDK / 小 widget；大型应用建议用真 React

04 典型代码

看 JSX 的"编译过程"

javascript · 复杂 JSX 编译实例 
// 编写
function Form({ fields, onSubmit }) {
  return (
    <form onSubmit={onSubmit}>
      {fields.map(field => (
        <label key={field.id}>
          {field.label}
          <input
            type={field.type}
            name={field.name}
            required={field.required}
            {...field.extraProps}
          />
        </label>
      ))}
      <button type="submit">Submit</button>
    </form>
  );
}

// 新 JSX runtime 编译后：
import { jsx as _jsx, jsxs as _jsxs } from 'react/jsx-runtime';

function Form({ fields, onSubmit }) {
  return _jsxs('form', {
    onSubmit,
    children: [
      fields.map(field => _jsxs('label', {
        children: [
          field.label,
          _jsx('input', {
            type: field.type,
            name: field.name,
            required: field.required,
            ...field.extraProps,    // 编译为 spread
          }),
        ],
      }, field.id)),                 // ← key 作为第三个参数（特殊位置）
      _jsx('button', { type: 'submit', children: 'Submit' }),
    ],
  });
}

// 关键点：
// 1. key 是 jsx 函数的第三个参数（特殊处理，不放进 props）
//    这是为了让 React 在创建元素前就能拿到 key 做 reconciliation
// 2. spread props（{...field.extraProps}）编译为 ES2018 spread
// 3. children 是数组（jsxs）/ 单值（jsx）—— 编译器自动决定

// 想看自己代码的编译结果：
// https://babeljs.io/repl 或 https://swc.rs/playground
// 选 React preset，把代码贴进去看输出

JSX 是表达式的威力 —— Composability 的语言根基

javascript · JSX 在表达式位置的灵活组合 
// JSX 是表达式，可以放在任何 JS 表达式位置：

// 1. 函数返回值（最常见）
function Greeting() { return <h1>Hi</h1>; }

// 2. 变量赋值
const heading = <h1>Hi</h1>;
const buttons = [<button key="a">A</button>, <button key="b">B</button>];

// 3. 函数参数 / 返回值
const wrap = (el) => <div className="wrapper">{el}</div>;
wrap(<Greeting />);

// 4. 数组 map
items.map(item => <li key={item.id}>{item.text}</li>);

// 5. 条件运算
const view = isAdmin ? <AdminView /> : <UserView />;

// 6. 默认参数
function Modal({ children, title = <h2>Default Title</h2> }) {
  return <div>{title}{children}</div>;
}

// 7. 解构默认值
const { content = <p>Empty</p> } = props;

// 8. 对象属性
const config = {
  header: <Header />,
  footer: <Footer />,
};

// 9. switch 模拟（用对象映射）
const components = {
  text: <TextField />,
  number: <NumberField />,
  date: <DateField />,
};
return components[type] ?? <UnknownField />;

// 这种"处处可用 JSX" 是 React Composability 的语言层根基。
// Vue / Svelte 的 SFC 模板不能这样组合（templates 是字符串/编译时结构，不是 JS 值）

05 常见陷阱

⚠️ 陷阱 1：&& 短路渲染 0 文本

{items.length && <List items={items} />} 当 items.length 是 0（空数组）时，&& 短路返回 0—— 而 0 是会被渲染为 '0' 文本的（不像 false / null / undefined 自动跳过）。结果页面上出现一个孤独的 '0'。

javascript · 修复方式 
// ❌ 出 bug
{items.length && <List items={items} />}

// ✅ 修复 1：明确比较
{items.length > 0 && <List items={items} />}

// ✅ 修复 2：用三元
{items.length ? <List items={items} /> : null}

// ✅ 修复 3：Boolean 强转
{Boolean(items.length) && <List items={items} />}

// 同样陷阱：字符串 ''、NaN
{user.name && ...}        // ❌ 当 name === '' 时
{user.name?.length > 0 && ...}  // ✅

规避：JSX 中的 && 短路只在左侧明确是 boolean 时安全。其他时候用三元或显式比较。

⚠️ 陷阱 2：style 用字符串

HTML 里 style="color: red" 是字符串。JSX 里 style 必须是对象—— style={{ color: 'red' }}。CSS 属性名要 camelCase（backgroundColor 而非 background-color）。

规避：① 现代项目用 Tailwind CSS / CSS Modules / Emotion 替代 inline style ② 必须 inline 时用对象语法 ③ Stylelint 配 react/style-prop-object 自动检测

⚠️ 陷阱 3：忘记升级到 New JSX Runtime

React 17+ 项目仍用 "jsx": "react"（classic runtime）—— 每个文件都要 import React，bundle 多 ~5KB，性能略差。常见原因是从 16 升级时只升 React 包没改 tsconfig。

规避：升级 React 17+ 时务必同时改 tsconfig.jsx: "react-jsx" + 跑 codemod。

⚠️ 陷阱 4：Fragment 短语法不能传 key

<>...</> 简洁但不支持 key 属性。在 list 里需要 key 时必须用完整形式 <Fragment key={item.id}>...</Fragment>。

javascript · Fragment 在 list 中 
// ❌ 编译错：<> 不支持属性
{items.map(item => (
  <> key={item.id}        // ⚠️ 语法错
    <dt>{item.title}</dt>
    <dd>{item.body}</dd>
  </>
))}

// ✅ 用完整 Fragment
import { Fragment } from 'react';
{items.map(item => (
  <Fragment key={item.id}>
    <dt>{item.title}</dt>
    <dd>{item.body}</dd>
  </Fragment>
))}
⚠️ 陷阱 5：组件首字母小写让 JSX 当 host 元素

function myButton() { ... } + <myButton /> → 编译为 type: 'mybutton'（host 元素），React 当 HTML 元素处理。

规避：组件名永远首字母大写。ESLint react/jsx-pascal-case。或者用 const MyButton = (props) => ... 函数表达式赋给大写常量。
