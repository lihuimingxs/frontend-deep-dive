B7 · useEffect 完整生命周期 ⭐

useEffect 是 React 中最难掌握的 Hook—— 闭包陷阱、依赖数组、cleanup、StrictMode 双重触发、useLayoutEffect 区分、Server Components 时代角色变化。这章用大篇幅把所有 useEffect 心智讲透。核心论断：useEffect 不是"componentDidMount 替代品"，是"同步 React state 到外部系统" 的工具。理解这点之前所有 useEffect 用法都是似懂非懂；理解后所有"什么时候用 useEffect"问题都有清晰答案。

⚠️ 这是 Track B 最难的一章

useEffect 陷阱密度全 React 最高—— 无限循环、stale closure、跑两次、cleanup 时机、依赖错配。建议读两遍，第二遍带着具体场景（如"怎么订阅 WebSocket" / "怎么 fetch 用户数据"）回头对照。

阅读时长 ~40 分钟 前置 B4 State 基础 + B6 useState 深度 + JS · 闭包（强烈推荐） 对应 react.dev useEffect API · Synchronizing with Effects

01 速览

三句话定义本章：

  1. useEffect 的心智模型是"同步外部系统"，不是"componentDidMount 等生命周期替代"。每个 useEffect 描述"当 X 变化时，让外部系统也跟着变"—— 外部系统包括 DOM API（document.title）/ 浏览器订阅（事件 listener）/ 网络订阅（WebSocket / SSE）/ 第三方库（Map SDK / Chart.js）/ 本地存储（localStorage）等。
  2. 完整生命周期 5 阶段：① mount 后 effect 跑 ② state/props 变化触发 cleanup（如果有）→ effect 重跑 ③ 重复 ② ④ unmount 时跑最后一次 cleanup。cleanup 在 React 18 StrictMode 开发环境会跑两次—— 强迫你写正确的 cleanup。
  3. 3 条铁律：① 把 effect 内用到的所有响应式值都列入依赖数组（state / props / context）—— 否则 stale closure ② 每个订阅必须有 cleanup —— 否则内存泄漏 + 重复订阅 ③ 不要把"事件响应"放 useEffect—— 那是 event handler 的工作。

看一段最小代码识别 useEffect 三要素：

javascript · useEffect 完整骨架 
function ChatRoom({ roomId }) {
  const [messages, setMessages] = useState([]);

  useEffect(() => {
    // ① 副作用代码（连接外部系统）
    const ws = new WebSocket(`wss://api.com/chat/${roomId}`);
    ws.onmessage = (e) => setMessages(prev => [...prev, JSON.parse(e.data)]);

    // ② cleanup（断开外部系统）—— 在 effect 重跑或组件 unmount 时调
    return () => ws.close();

  }, [roomId]);   // ③ 依赖数组：roomId 变化时 cleanup + 重跑 effect

  return <ul>{messages.map(m => <li key={m.id}>{m.text}</li>)}</ul>;
}

// 心智解读：
// "当 roomId 变化时，让 WebSocket 连接也跟着切换"
// = "同步 React state（roomId）到外部系统（WebSocket 连接）"
// 不是 "组件 mount 时连接 WebSocket" —— 这只是其中一个阶段

02 底层逻辑

心智模型：「同步外部系统」

这是 useEffect 最重要的概念—— 也是大部分 useEffect bug 的根源。

📐 useEffect 不是"生命周期替代品"

React 16.x class 时代，componentDidMount / componentDidUpdate / componentWillUnmount 是 3 个独立 hook，开发者把订阅 / 取消订阅 / fetch 等逻辑分散在三个方法里。
React 16.8 Hooks 时代，useEffect 把"订阅 + 取消订阅"统一到一个 effect 里——不再按"组件何时" 分，而是按"同步什么" 分。
新人最常见的错误 是用 class 心智写 useEffect：把所有"组件 mount 时的初始化" 都堆在一个 useEffect。结果遇到状态变化要重跑时陷入"怎么改" 的迷茫。

正确心智：每个 useEffect 描述一个独立的"同步关系"。多个独立关系应该拆成多个 useEffect。

javascript · 不要用 class 心智写 useEffect 
// ❌ class 心智：所有 init 在一个 useEffect
function Component({ userId, theme }) {
  useEffect(() => {
    // 同步用户数据
    fetchUser(userId).then(setUser);
    // 应用主题
    document.body.className = theme;
    // 注册全局事件
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
  }, [userId, theme]);
  // 问题：theme 变了也会重新 fetchUser！userId 变了也会重新设 className
}

// ✅ 同步系统心智：每个独立同步关系一个 useEffect
function Component({ userId, theme }) {
  // 同步关系 1：userId → user data
  useEffect(() => {
    fetchUser(userId).then(setUser);
  }, [userId]);

  // 同步关系 2：theme → body className
  useEffect(() => {
    document.body.className = theme;
  }, [theme]);

  // 同步关系 3：window resize → state（无依赖，挂载时建立 + 卸载时清理）
  useEffect(() => {
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
  }, []);
}

// 拆分后：每个 effect 只管一件事，依赖最小，不互相干扰

完整生命周期 5 阶段

javascript · useEffect 完整生命周期 
function Component({ id }) {
  useEffect(() => {
    console.log('Effect ran with id:', id);
    return () => console.log('Cleanup with id:', id);
  }, [id]);
  return null;
}

// 完整时序（生产环境）：
//
// Step 1: mount —— 组件首次挂载
//   - render 函数体跑
//   - commit 阶段：DOM 更新
//   - 浏览器 paint
//   - effect 跑（passive effect 异步，不阻塞 paint）
//   - console: "Effect ran with id: 1"
//
// Step 2: 父更新让 id 变成 2
//   - render 函数体跑（用新 id）
//   - commit 阶段：DOM 更新
//   - 浏览器 paint
//   - cleanup 跑（用上次 effect 的闭包）
//   - console: "Cleanup with id: 1"
//   - effect 重跑（用新 id）
//   - console: "Effect ran with id: 2"
//
// Step 3: id 不变（仅 state 变化导致重渲染）
//   - render + commit
//   - 因为依赖 [id] 没变 → cleanup 不跑、effect 不跑
//   - console: 无输出
//
// Step 4: unmount —— 组件销毁
//   - cleanup 跑
//   - console: "Cleanup with id: 2"

// 开发环境（StrictMode）多 1 步：
//   mount 时跑两遍：mount → effect → cleanup → effect
//   帮你检测 cleanup 是否正确

关键观察：

  • cleanup 用上次 effect 的闭包—— 这是 cleanup 能"断开正确的 WebSocket / 移除正确的 listener"的根因
  • effect 在 commit 之后异步跑—— 不阻塞浏览器 paint（A3 详述）
  • 依赖不变跳过 cleanup + effect—— 这是 React.memo / useMemo / useCallback 配合 effect 优化的基础

依赖数组规则 ⭐

依赖数组是 useEffect 最容易出错的部分：

写法 含义
不写 useEffect(() => {}) 每次 render 都跑（极少用）
空数组 useEffect(() => {}, []) 仅 mount 时跑一次（cleanup 仅 unmount 时跑）
有依赖 useEffect(() => {}, [a, b]) mount + 任一依赖 Object.is 不等时重跑

依赖比较用 Object.is 浅比较—— 与 useState 同款（B6）。注意：

  • 原始值（数字 / 字符串 / boolean）—— Object.is 比较值
  • 对象 / 数组 / 函数—— 比较引用！每次 render 创建新引用 → effect 每次都重跑
  • NaN vs NaN—— Object.is 等（不会重跑）
  • +0 vs -0—— Object.is 不等（会重跑）

「把 effect 用到的所有响应式值都列入依赖」是规则—— ESLint react-hooks/exhaustive-deps 强制：

javascript · 依赖数组完整规则 
function Chat({ roomId, theme }) {
  const [messages, setMessages] = useState([]);

  // ❌ 漏依赖 —— stale closure
  useEffect(() => {
    const ws = new WebSocket(`wss://api.com/${roomId}`);
    ws.onmessage = (e) => {
      console.log('theme:', theme);    // theme 是 effect 创建时的快照！
    };
    return () => ws.close();
  }, [roomId]);   // 漏 theme

  // ✅ 完整依赖 —— theme 变化也重连 ws（语义上对的）
  useEffect(() => {
    const ws = new WebSocket(`wss://api.com/${roomId}`);
    ws.onmessage = (e) => {
      console.log('theme:', theme);
    };
    return () => ws.close();
  }, [roomId, theme]);  // ✅ ESLint 通过

  // 但 theme 变化重连 ws 没必要 —— 见 React 19 useEffectEvent 解决方案
}

// ESLint exhaustive-deps 自动检测：
// - state / props / context / 自定义 Hook 返回值都要列入
// - setState（dispatch）/ useRef / 模块级常量 不需要列（引用稳定 / 非响应式）

// 永远不要 disable lint 跳过这个规则！
// 99% 的"我必须 disable" 案例都是 effect 写错了，正确的写法是：
// 1. 重构让函数 / 对象不必每次 render 创建新引用（useMemo / useCallback）
// 2. 用 useEffectEvent（19 提案）拿"非响应式" 的最新值
// 3. 拆分成多个 effect（每个 effect 关心更少依赖）
// 4. 把 state 提升到组件外（useReducer / Zustand）

cleanup 模式

每个 effect 建立外部连接就必须有 cleanup 断开。常见模式：

javascript · cleanup 经典 5 种模式 
// 1. 事件 listener
useEffect(() => {
  const handler = (e) => doSomething(e);
  window.addEventListener('keydown', handler);
  return () => window.removeEventListener('keydown', handler);
}, []);

// 2. WebSocket / EventSource
useEffect(() => {
  const ws = new WebSocket(url);
  ws.onmessage = handleMessage;
  return () => ws.close();
}, [url]);

// 3. 定时器
useEffect(() => {
  const id = setInterval(() => doSomething(), 1000);
  return () => clearInterval(id);
}, []);

// 4. AbortController（fetch 取消）
useEffect(() => {
  const controller = new AbortController();
  fetch(`/api/users/${userId}`, { signal: controller.signal })
    .then(r => r.json())
    .then(setUser)
    .catch(err => { if (err.name !== 'AbortError') throw err; });
  return () => controller.abort();
}, [userId]);
// 当 userId 快速变化时，旧请求自动取消 —— 避免"响应顺序竞态" bug

// 5. 第三方 SDK / 库
useEffect(() => {
  const map = new MapSDK(containerRef.current, { center });
  return () => map.destroy();
}, [center]);

// 关键原则：每个 setUp 必须有 tearDown
// "如果我创建了它，我必须负责销毁它"

闭包陷阱（短重述）

短重述：组件每次 render 创建新闭包，effect 函数捕获的 state / props 是那次 render 的快照。详细机制见 JS · 闭包 + B4 stale state 章节。

javascript · useEffect 中的闭包陷阱 
// ❌ 经典 stale closure
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log('count:', count);   // 永远是 effect 创建时的 count
    }, 1000);
    return () => clearInterval(id);
  }, []);                              // 空依赖 —— effect 仅初次创建

  // 用户点击 button 多次，count 实际更新了，但日志永远是初始值

  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

// 解 1：把 count 加进依赖（每次 count 变都 cleanup + 重新 setInterval）
// 缺点：interval 时间会"重置"

// 解 2：用 ref 拿最新 count（B10 详述）
const countRef = useRef(count);
useEffect(() => { countRef.current = count; });
useEffect(() => {
  const id = setInterval(() => {
    console.log('count:', countRef.current);  // 永远最新
  }, 1000);
  return () => clearInterval(id);
}, []);

// 解 3（19+ 推荐）：useEffectEvent
// 见下文 React 19 useEffectEvent 节

useLayoutEffect vs useEffect

维度 useEffect useLayoutEffect
触发时机 commit 后异步（passive effect） commit 中同步（layout phase）
是否阻塞浏览器 paint 不阻塞 阻塞（用户看不到中间状态）
性能影响 对帧率友好 慢的 layout effect 让帧率掉
SSR 行为 SSR 不跑（仅客户端） SSR 不跑 + 警告（服务器无 layout）
典型场景 数据 fetch / 订阅 / 同步外部 同步 DOM 测量后修改避免闪烁（如 tooltip 位置）
javascript · useLayoutEffect 必要场景 
// ✅ useLayoutEffect 必要：测量 + 调整 避免 paint 闪烁
function Tooltip({ text, target }) {
  const tooltipRef = useRef(null);
  const [position, setPosition] = useState({ top: 0, left: 0 });

  // ⚠️ 用 useEffect 会闪烁：
  //    paint 1 frame（tooltip 在 (0,0)）→ effect 算位置 setState → paint 2 frame
  // ✅ 用 useLayoutEffect：
  //    layout phase 算位置 setState → 重新 layout → paint（用户看到正确位置）
  useLayoutEffect(() => {
    const targetRect = target.getBoundingClientRect();
    const tooltipRect = tooltipRef.current.getBoundingClientRect();
    setPosition({
      top: targetRect.bottom + 5,
      left: targetRect.left + (targetRect.width - tooltipRect.width) / 2,
    });
  }, [target]);

  return (
    <div ref={tooltipRef} style={{ position: 'absolute', ...position }}>
      {text}
    </div>
  );
}

// 99% 场景用 useEffect。useLayoutEffect 仅用于：
// 1. 测量 DOM 后立即调整（避免闪烁）
// 2. 同步修改 DOM 不让用户看到中间态
// 3. 集成命令式动画库（GSAP / Three.js）需要在 paint 前同步

StrictMode 双重触发 + cleanup 检测

B1 已讲过 StrictMode 让组件函数双重调用。对 useEffect 的具体影响：

javascript · StrictMode 双重 effect 
// 在 StrictMode 下，组件 mount 时：
useEffect(() => {
  console.log('mount');
  return () => console.log('cleanup');
}, []);

// 控制台输出（仅开发环境）：
// mount
// cleanup
// mount
// 而非生产环境的：mount

// 为什么 React 这样做？
// 检测你的 cleanup 是否正确写了：
// - 如果 cleanup 不正确，会暴露 bug（如订阅了两次但只取消一次）
// - 强迫你用"每个 setup 都有 tearDown" 的模式

// 常见 bug 模式（StrictMode 暴露）：
useEffect(() => {
  // ❌ 没 cleanup —— StrictMode 下事件 listener 注册两次
  window.addEventListener('keydown', handleKey);
  // 生产环境单次：表面看起来正常
  // 开发环境双次：addEventListener 被调用两次 —— 但 React 不知道，无法 cleanup
  // → 实际有内存泄漏 + 双重响应
}, []);

useEffect(() => {
  // ✅ 有 cleanup —— StrictMode 双调时 cleanup 也跑两次，最终状态正确
  window.addEventListener('keydown', handleKey);
  return () => window.removeEventListener('keydown', handleKey);
}, []);

// 关键原则：所有 effect 都应该假设可能跑多次 —— cleanup 必须正确

React 19 useEffectEvent 提案

长期痛点：useEffect 内有时需要"非响应式" 值——它会变但变了不该重跑 effect。React 19 引入实验性 useEffectEvent（仍在 RFC 阶段，未稳定）：

javascript · useEffectEvent 解决"非响应式" 值 
// 痛点：roomId 变要重连 ws，但 onMessage 变化不该重连
function Chat({ roomId, onMessage }) {
  // ❌ 列入依赖：onMessage 每次 render 都新引用 → ws 反复重连
  useEffect(() => {
    const ws = new WebSocket(`wss://api.com/${roomId}`);
    ws.onmessage = (e) => onMessage(JSON.parse(e.data));
    return () => ws.close();
  }, [roomId, onMessage]);

  // ❌ 不列入：stale closure —— onMessage 变化但 effect 仍用旧的

  // ✅ React 19 useEffectEvent（实验性）
  const handleMessage = useEffectEvent((e) => {
    onMessage(JSON.parse(e.data));   // 总是用最新 onMessage
  });

  useEffect(() => {
    const ws = new WebSocket(`wss://api.com/${roomId}`);
    ws.onmessage = handleMessage;     // 引用稳定 + 自动拿最新
    return () => ws.close();
  }, [roomId]);   // ✅ 不需要列 onMessage / handleMessage
}

// 注意：
// 1. useEffectEvent 仍在实验阶段（写作时点 2026-04 未稳定）
// 2. 设计意图：声明"这个 callback 是事件响应，不是同步关系"
// 3. 不能在 render 中调用 useEffectEvent 返回的函数 —— 仅 effect 内 / event handler 内

何时不用 useEffect ⭐

这是 React 团队在 react.dev 反复强调的一节—— useEffect 被滥用太多。"You Might Not Need an Effect" 是官方推荐阅读。常见误用：

javascript · useEffect 反模式与正确替代 
// === 反模式 1：派生 state ===
// ❌ useEffect 同步派生值
const [items, setItems] = useState([]);
const [filteredItems, setFilteredItems] = useState([]);
useEffect(() => {
  setFilteredItems(items.filter(i => i.active));
}, [items]);
// 问题：① 多一次渲染 ② 同步逻辑可能出 bug ③ 性能差

// ✅ 直接计算
const filteredItems = items.filter(i => i.active);
// 昂贵时用 useMemo

// === 反模式 2：响应事件 ===
// ❌ useEffect 响应 button click
const [submitted, setSubmitted] = useState(false);
useEffect(() => {
  if (submitted) {
    showToast('Success!');
    setSubmitted(false);
  }
}, [submitted]);

// ✅ 直接在 event handler 里
const handleSubmit = () => {
  showToast('Success!');
};

// === 反模式 3：通知 parent state 变化 ===
// ❌ useEffect 通知 parent
function Toggle({ onChange }) {
  const [isOn, setIsOn] = useState(false);
  useEffect(() => {
    onChange(isOn);   // 通知 parent
  }, [isOn, onChange]);
  return ...;
}

// ✅ 在 setState 同时通知 parent（更新 + 通知是同一逻辑）
function Toggle({ onChange }) {
  const [isOn, setIsOn] = useState(false);
  const handleToggle = () => {
    const next = !isOn;
    setIsOn(next);
    onChange(next);   // 同步通知
  };
  return ...;
}

// === 反模式 4：初始化全局单例 ===
// ❌ 每次组件 mount 都跑（即使已初始化）
useEffect(() => {
  initAnalytics();
}, []);

// ✅ 模块级初始化（仅一次）
if (typeof window !== 'undefined' && !window.__analyticsInitialized) {
  initAnalytics();
  window.__analyticsInitialized = true;
}
function App() { return ...; }

// === 反模式 5：用 useEffect 加载数据（19+ 用 use API / RSC）===
// ❌ 经典 useEffect fetch
useEffect(() => {
  fetch(`/api/users/${id}`).then(r => r.json()).then(setUser);
}, [id]);

// ✅ React 19 + RSC：服务端组件直接 await
async function UserProfile({ id }) {
  const user = await fetch(`/api/users/${id}`).then(r => r.json());
  return <UserCard {...user} />;
}

// ✅ 客户端组件 + use API
'use client';
function UserProfile({ userPromise }) {
  const user = use(userPromise);
  return <UserCard {...user} />;
}

// ✅ 或用 TanStack Query / SWR（生产推荐）
const { data, isLoading } = useQuery(['user', id], () =>
  fetch(`/api/users/${id}`).then(r => r.json()),
);

// 2025 年的 useEffect 真正用途越来越少 —— 主要是"同步外部系统"
// （订阅 / DOM 操作 / 第三方库）。数据加载有更好的工具

03 应用场景

场景 A · 订阅外部 store（Redux / Zustand 之外的）

集成自家的 EventBus / 第三方状态库（不是 React 集成的）—— useEffect 订阅是经典用例：

javascript · 订阅外部 store 
function useExternalStore(store) {
  const [value, setValue] = useState(store.getValue());

  useEffect(() => {
    const unsubscribe = store.subscribe(() => setValue(store.getValue()));
    return unsubscribe;
  }, [store]);

  return value;
}

// 用法
function Component() {
  const userData = useExternalStore(globalUserStore);
  return <div>{userData.name}</div>;
}

// React 18+ 实际推荐用 useSyncExternalStore（专为外部 store 设计）：
const userData = useSyncExternalStore(
  globalUserStore.subscribe,
  () => globalUserStore.getValue(),
  () => globalUserStore.getServerValue(),  // SSR 快照
);
// 内部仍是 useEffect 订阅，但解决了 tearing（Concurrent rendering 期间撕裂）问题

场景 B · 集成命令式 DOM 库（Chart.js / Three.js / D3）

useEffect 是 React 与命令式 DOM 库的桥梁——React 管声明式渲染，effect 管命令式初始化和销毁：

javascript · 集成 Chart.js 
function Chart({ data, type = 'bar' }) {
  const canvasRef = useRef(null);
  const chartRef = useRef(null);

  // 初始化（仅 type 变化时重建）
  useEffect(() => {
    chartRef.current = new ChartJS(canvasRef.current, {
      type,
      data,
      options: { responsive: true },
    });
    return () => chartRef.current?.destroy();
  }, [type]);

  // data 变化时调用 update（不重建）
  useEffect(() => {
    if (!chartRef.current) return;
    chartRef.current.data = data;
    chartRef.current.update();
  }, [data]);

  return <canvas ref={canvasRef} />;
}

// 关键设计：
// 1. 拆 2 个 useEffect —— 一个负责"组件生命周期"，一个负责"data 同步"
// 2. type 变重建（chart 类型变了）；data 变 update（数据变了不重建）
// 3. cleanup 必须 destroy 老 chart，否则内存泄漏

场景 C · React 18 升级时修 effect 双跑 bug

升级 React 17 → 18 + StrictMode 后，发现 useEffect 跑两次—— 不是 React 18 引入的 bug，是暴露了原有的 cleanup 缺失。

  1. 检查每个 useEffect 是否有 cleanup—— 订阅 / listener / 定时器 / fetch 都要
  2. fetch 加 AbortController—— 双调时第一个会被取消，避免重复请求 + 响应顺序竞态
  3. WebSocket / EventSource—— cleanup 必须 close，否则双调建立两个连接
  4. analytics / 计数器—— 用 ref 标记是否已初始化，避免双调重复
  5. StrictMode 让你"必须写正确"—— 是好事，提前暴露 bug

04 典型代码

fetch 数据 + AbortController + 竞态保护

javascript · 完整 fetch 模式（教学用，生产用 TanStack Query） 
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [error, setError] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    const controller = new AbortController();
    setLoading(true);
    setError(null);

    fetch(`/api/users/${userId}`, { signal: controller.signal })
      .then(r => {
        if (!r.ok) throw new Error(`HTTP ${r.status}`);
        return r.json();
      })
      .then(data => {
        setUser(data);
        setLoading(false);
      })
      .catch(err => {
        if (err.name === 'AbortError') return;  // 被取消，忽略
        setError(err);
        setLoading(false);
      });

    return () => controller.abort();
  }, [userId]);

  if (loading) return <Spinner />;
  if (error) return <ErrorMsg error={error} />;
  if (!user) return null;
  return <UserCard {...user} />;
}

// 关键点：
// 1. AbortController —— userId 快变时旧请求自动取消，避免响应顺序竞态
// 2. 每个 effect 跑都重置 loading / error
// 3. AbortError 必须忽略（不是真错）
// 4. 生产推荐 TanStack Query / SWR 而非手写

05 常见陷阱

⚠️ 陷阱 1：依赖数组漏依赖（stale closure）

最常见的 useEffect bug。effect 内用了 state / props 但没列入依赖 → 闭包捕获旧值。

规避：① 永远不要 disable react-hooks/exhaustive-deps ② 配 ESLint 在 CI 强制 ③ 真不想列依赖时用 useEffectEvent（19+）/ ref 模式（B10）/ updater function。

⚠️ 陷阱 2：依赖列了但是对象 / 函数 / 数组 → 永远 unequal

useEffect(() => {...}, [{ x: 1 }]) 或 [arrayProp]—— 每次 render 新引用 → effect 每次都重跑 → 可能死循环。

javascript · 引用稳定化 
// ❌ 死循环
function Component({ config }) {
  const [data, setData] = useState(null);
  useEffect(() => {
    fetch('/api', { body: JSON.stringify(config) }).then(setData);
  }, [config]);   // config 每次 parent render 都新对象 → 死循环
}

// ✅ 1. 只取需要的原始字段
useEffect(() => {...}, [config.id, config.name]);  // 假设是字符串

// ✅ 2. parent 用 useMemo 稳定化 config
const config = useMemo(() => ({ id, name }), [id, name]);

// ✅ 3. 用 ref 模式（B10）

// ✅ 4. 用 JSON.stringify 比较（仅简单数据）
const configKey = JSON.stringify(config);
useEffect(() => {...}, [configKey]);

规避：依赖里只放原始值 是最安全。复杂对象 / 函数依赖要用 useMemo / useCallback 稳定化。

⚠️ 陷阱 3：用 useEffect 做"事件响应"

useEffect(() => { if (formData.submitted) doSubmit() }, [formData])—— 这是事件响应，应该在 onSubmit handler 里直接调 doSubmit，不需要 useEffect 中转。

规避：识别"这是事件响应吗"—— 是的话放 event handler，不要绕道 useEffect。详见 react.dev "You Might Not Need an Effect"。

⚠️ 陷阱 4：用 useEffect 派生 state

useEffect(() => { setFiltered(items.filter(...)) }, [items])—— 多渲染一次 + 同步逻辑出 bug 风险。

规避：派生值直接计算。昂贵时用 useMemo。state 只存独立的源数据。

⚠️ 陷阱 5：忘记 cleanup 导致 StrictMode 暴露 bug

React 18 升级后看到 effect 跑两次—— 不是 bug 是 feature。StrictMode 帮你检测 cleanup。

规避：每个 setUp 都有 tearDown。双跑暴露的 bug 在生产是真实存在的—— StrictMode 只是让你提前发现。

⚠️ 陷阱 6：用 useLayoutEffect 跑昂贵操作让帧率掉

useLayoutEffect(() => { processLargeData() }, [data])—— 阻塞 paint，UI 卡顿。

规避：useLayoutEffect 仅用于必须在 paint 前完成的 DOM 测量 / 同步调整。其他全部用 useEffect。
