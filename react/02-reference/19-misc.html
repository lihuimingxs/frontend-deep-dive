B19 · ref as prop + 19+ 杂项

React 主题最后一章，收口 React 19 的"非 Hook" 杂项。两个关键变更：① ref as prop—— 函数组件可以直接接收 ref，forwardRef 软退役 ② defaultProps 退役（函数组件）—— 改用 ES 默认参数。再加上几个少用但重要的 API：React.startTransition 顶层 API（vs useTransition Hook）、renderToPipeableStream / renderToReadableStream（SSR 出口）、React.cache（仅 RSC 的 dedupe 原语）、以及 <title> / <meta> / <link> 在组件树中的原生提升。读完本章，Track B · React 官网知识精读 19 章完结。

01 速览

本章覆盖 7 个独立但相关的"19+" API / 行为变更：

  1. ref as prop—— 函数组件可以直接接收 ref prop（function MyInput({ ref })），forwardRef 不再需要。forwardRef 软退役但仍可用（19 不报警告，未来版本会）。
  2. defaultProps 退役（仅函数组件）—— MyComp.defaultProps = {...} 不再生效，会被 React 19 警告。改用 ES 默认参数（function MyComp({ size = 'md' })）。class 组件仍支持 defaultProps（向后兼容）。
  3. String refs 彻底移除—— <input ref="myInput" /> 写法 19 完全不能用（之前是警告）。改用 useRef + 对象 ref / callback ref。
  4. startTransition 顶层 API—— import { startTransition } from 'react'，可以在组件外 / 普通函数 / event handler 中调用，区别于 useTransition Hook（必须在组件内）。没有 isPending。
  5. SSR 双流 API—— Node 用 renderToPipeableStream（pipe 到 stream），Edge / Browser / Deno 用 renderToReadableStream（返回 Web Stream）。同步的 renderToString 只用于极简 SSR—— 不支持 Suspense / 流式 / RSC。
  6. React.cache(fn)—— 仅 Server Components / Server Actions 可用。同 framework 请求内（per request）相同参数返回同一 Promise，自动 dedupe 数据获取。不能在 Client Component 里用。
  7. 文档元数据原生支持—— <title> / <meta> / <link> 写在组件树任何位置都会自动提升到 <head>。取代了 react-helmet 等老库。

02 底层逻辑

1. ref as prop —— forwardRef 软退役的根因

B10 讲过 forwardRef 的存在原因：函数组件没有"实例"，ref 是 React 内部保留的特殊 prop，不会通过 props 透传。R&D 体感的麻烦是：每个想要被 ref 的组件都要套一层 forwardRef(function(...))，TS 类型也变得复杂。

React 19 重构 reconciler，函数组件可以接收 ref 作为普通 prop。语义和 forwardRef 完全等价，但代码干净得多： 

// React 18 — forwardRef
import { forwardRef } from 'react';
const MyInput = forwardRef<HTMLInputElement, { label: string }>(
  ({ label }, ref) => (
    <label>
      {label}: <input ref={ref} />
    </label>
  )
);
MyInput.displayName = 'MyInput';  // forwardRef 必须手动加 displayName

// React 19 — ref as prop
function MyInput({ label, ref }: {
  label: string;
  ref?: React.Ref<HTMLInputElement>;
}) {
  return (
    <label>
      {label}: <input ref={ref} />
    </label>
  );
}
// ✅ 普通函数 / 普通 props 类型 / 不需要 displayName / 同样支持 callback ref

为什么 18 不能这么做？因为老 reconciler 把 ref 提到 type / key / ref 这一层（特殊处理），不通过 props 流。19 修改了内部表示，ref 既能当 prop 解析，也能维持原本的特殊语义（仍然只在 commit 阶段赋值，不参与 reconciliation diff）。

2. defaultProps 退役（仅函数组件）的根因

MyComp.defaultProps 在 React 18 还能用，但从 React 19 开始：函数组件的 defaultProps 不再生效，会触发警告。原因：

  • ES 默认参数（function ({ size = 'md' })）已经是事实标准 —— 类型推导更好 + 不需要运行时查表
  • defaultProps 阻止某些 React 优化—— React 必须在每次 render 前合并 defaultProps，多一步开销
  • 语义不一致—— defaultProps 只在 prop 是 undefined 时生效（不是 null），ES 默认参数语义相同但更显式

class 组件保留 defaultProps—— class 用 ES 默认参数会绕一圈（constructor(props) 内手动合并）。React 19 不强行迁移 class，避免大批量代码 break。

3. startTransition 顶层 API vs useTransition Hook

维度useTransition()startTransition(fn)
导入import { useTransition } from 'react'import { startTransition } from 'react'
是否 Hook是否
调用位置组件 / Hook 顶层任何地方（含 event handler / 普通函数 / 库代码）
返回 isPending✅ 提供❌ 不提供
典型场景组件内有"是否 transition 中" UI 反馈需求状态库（Zustand / Jotai）/ event handler 内的快速调用
 
// useTransition：组件内
function Tabs() {
  const [tab, setTab] = useState('home');
  const [isPending, startTransition] = useTransition();
  return (
    <>
      <button onClick={() => startTransition(() => setTab('reports'))}>
        Reports {isPending && '...'}
      </button>
    </>
  );
}

// startTransition 顶层：在 Zustand store action 里
import { startTransition } from 'react';
import { create } from 'zustand';

const useStore = create((set) => ({
  filter: '',
  setFilterDeferred: (value) => {
    startTransition(() => {  // 在 Hook 之外调用
      set({ filter: value });  // 这次更新被标为 transition
    });
  },
}));

4. SSR API：renderToPipeableStream vs renderToReadableStream vs renderToString

API环境支持 Suspense支持 RSC典型用户
renderToString所有❌ 不流式（fallback 直接渲染）❌极简 SSR / 邮件 HTML
renderToPipeableStreamNode.js✅ 流式✅Express / Fastify / Next.js Pages
renderToReadableStreamEdge / Browser / Deno✅ 流式✅Cloudflare Workers / Vercel Edge
renderToStaticMarkup所有❌❌纯静态 HTML（无 hydration）
 
// Node.js 示例
import { renderToPipeableStream } from 'react-dom/server';
app.get('/', (req, res) => {
  const { pipe } = renderToPipeableStream(<App />, {
    bootstrapScripts: ['/main.js'],
    onShellReady() {
      res.setHeader('content-type', 'text/html');
      pipe(res);  // 流式传输（Suspense 边界 ready 时分块发送）
    },
  });
});

// Edge / Cloudflare Workers
import { renderToReadableStream } from 'react-dom/server';
export default {
  async fetch(request) {
    const stream = await renderToReadableStream(<App />, {
      bootstrapScripts: ['/main.js'],
    });
    return new Response(stream, {
      headers: { 'content-type': 'text/html' },
    });
  },
};

5. React.cache(fn)—— RSC 专用的 per-request 缓存

B16 讲过 RSC 的死循环陷阱：每次 render 创建新 Promise → 死循环。cache 是 React 团队提供的官方解决方案—— 包装函数后，同 request、同参数返回同一 Promise（identity 稳定）。 

// 仅在 Server Component / Server Action 里使用
import { cache } from 'react';
import { db } from '@/lib/db';

const getUser = cache(async (id: string) => {
  return db.users.findUnique({ where: { id } });
});

// 在多个 Server Component 调用同一个 getUser(42) → 仅查询数据库一次
async function Header() {
  const user = await getUser('42');
  return <span>{user.name}</span>;
}
async function Sidebar() {
  const user = await getUser('42');  // 同 ID → 复用 Header 的 Promise
  return <img src={user.avatar} />;
}
// ✅ 一次 request 内 dedupe，不同 request 之间不缓存（正确的隔离）

// ❌ 不能在 Client Component 用
'use client';
const wrong = cache((id) => ...);  // 编译报错或运行时无效

关键不变量：cache 的隔离粒度是 request—— 不同用户、不同请求互不污染。这是它和 SWR / TanStack Query 客户端缓存的根本区别（后者是 cross-request 的）。

6. 文档元数据原生支持

React 18 之前，写在组件里的 <title> / <meta> / <link> 必须用 react-helmet-async 这种库手动提升到 <head>。React 19 原生支持—— 写在组件树任何位置都会自动提升： 

// 19 之前要装库
import { Helmet } from 'react-helmet-async';
function Page() {
  return (
    <>
      <Helmet>
        <title>My Page</title>
        <meta name="description" content="..." />
      </Helmet>
      <main>...</main>
    </>
  );
}

// 19 直接写
function Page() {
  return (
    <>
      <title>My Page</title>       {/* 自动提升到 head */}
      <meta name="description" content="..." />
      <link rel="canonical" href="https://example.com/page" />
      <main>...</main>
    </>
  );
}
// 注意：动态切换 title 时仍要用 useEffect / RSC 重 render

这是给非 framework 用户的礼物—— Next.js / Remix 已经有自己的 metadata API，原生支持让纯 React + Vite 的项目也能优雅做 SEO。

7. 还有几个细节

  • Asset Loading—— React 19 内置 preload(href, options) / preinit() API，主动声明依赖让浏览器尽早加载。配合 RSC 流式渲染效果显著。
  • useId—— 18 引入但 19 加强，用于 SSR 一致的稳定 ID（替代手动 Math.random()），常用于 ARIA 属性 / 表单 label-input 关联。
  • Error 改进—— 19 默认行为：未捕获错误 console.error 显示完整组件栈（之前是简略）；ErrorBoundary 捕获后不再 console.error 重复（避免噪音）。详见 B15。
  • Hydration 错误信息—— diff 显示更精确（哪个属性 / 哪个文本节点不匹配）。

03 应用场景

场景 1：组件库迁移（forwardRef → ref as prop）

shadcn/ui / Radix UI 这类组件库 19 版已批量迁移到 ref as prop。自家组件库可以渐进迁移： 

// 旧版（仍支持，无警告）
import { forwardRef } from 'react';
export const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ children, ...props }, ref) => (
    <button ref={ref} {...props}>{children}</button>
  )
);

// 新版
type ButtonProps = {
  children: React.ReactNode;
  ref?: React.Ref<HTMLButtonElement>;
} & React.ButtonHTMLAttributes<HTMLButtonElement>;

export function Button({ children, ref, ...props }: ButtonProps) {
  return <button ref={ref} {...props}>{children}</button>;
}

// 调用方完全不变
const btnRef = useRef<HTMLButtonElement>(null);
<Button ref={btnRef}>Click</Button>

场景 2：状态库集成 startTransition 顶层 API

Zustand / Jotai 这类库的 setter 可以包 startTransition，让"慢更新" 自动走低优先级： 

import { startTransition } from 'react';
import { create } from 'zustand';

const useFilters = create((set) => ({
  search: '',
  setSearch: (value) => set({ search: value }),  // 同步：高优先级
  setSearchDeferred: (value) => {
    startTransition(() => {
      set({ search: value });  // transition：低优先级，可被打断
    });
  },
}));

// 组件中
function SearchBox() {
  const setSearchDeferred = useFilters((s) => s.setSearchDeferred);
  const [local, setLocal] = useState('');
  return (
    <input
      value={local}
      onChange={(e) => {
        setLocal(e.target.value);                   // 输入框立即响应
        setSearchDeferred(e.target.value);          // 列表更新走 transition
      }}
    />
  );
}

场景 3：自家 SSR（不用 Next.js）

自己搭 SSR 的少数项目： 

// server.ts (Node + Express)
import express from 'express';
import { renderToPipeableStream } from 'react-dom/server';
import App from './App';

const app = express();
app.use(express.static('dist'));

app.get('*', (req, res) => {
  const { pipe, abort } = renderToPipeableStream(<App url={req.url} />, {
    bootstrapScripts: ['/dist/client.js'],
    onShellReady() {
      res.statusCode = 200;
      res.setHeader('content-type', 'text/html');
      res.write('<!DOCTYPE html><html><body><div id="root">');
      pipe(res);
    },
    onShellError(error) {
      res.statusCode = 500;
      res.send('<h1>Error</h1>');
    },
    onAllReady() {
      // 全部 Suspense 边界 ready 后追加 closing tags
    },
  });
  setTimeout(() => abort(), 10000);  // 10s 超时
});

场景 4：RSC 数据 fetching dedupe（cache）

// lib/data.ts
import 'server-only';
import { cache } from 'react';
import { db } from './db';

export const getCurrentUser = cache(async () => {
  const session = await getSession();
  if (!session) return null;
  return db.users.findUnique({ where: { id: session.userId } });
});

// app/layout.tsx (Server)
import { getCurrentUser } from '@/lib/data';
export default async function Layout({ children }) {
  const user = await getCurrentUser();  // ① 第一次调用，查 DB
  return (
    <html>
      <body>
        <Header user={user} />
        {children}
      </body>
    </html>
  );
}

// app/page.tsx (Server)
import { getCurrentUser } from '@/lib/data';
export default async function Page() {
  const user = await getCurrentUser();  // ② 同请求复用 ①，不查 DB
  return <Welcome user={user} />;
}

04 典型代码

例 1：完整迁移：forwardRef + defaultProps + 老 ref → 19 写法

// React 18 — 老组件
import React, { forwardRef } from 'react';

export const Avatar = forwardRef(({ size, alt, src }, ref) => (
  <img ref={ref} src={src} alt={alt} className={`avatar avatar-${size}`} />
));
Avatar.defaultProps = { size: 'md', alt: 'User avatar' };
Avatar.displayName = 'Avatar';

// React 19 — 新写法
type AvatarProps = {
  size?: 'sm' | 'md' | 'lg';
  alt?: string;
  src: string;
  ref?: React.Ref<HTMLImageElement>;
};

export function Avatar({ size = 'md', alt = 'User avatar', src, ref }: AvatarProps) {
  return <img ref={ref} src={src} alt={alt} className={`avatar avatar-${size}`} />;
}
// 全部干净了：① 普通函数 ② TS 类型清晰 ③ ES 默认参数代替 defaultProps

例 2：动态 title / meta（SEO）

// 基于路由 / 数据动态变化
function ProductPage({ product }) {
  return (
    <>
      <title>{product.name} — Acme Store</title>
      <meta name="description" content={product.description} />
      <meta property="og:image" content={product.image} />
      <link rel="canonical" href={`https://store.com/p/${product.slug}`} />

      <article>
        <h1>{product.name}</h1>
        <p>{product.description}</p>
      </article>
    </>
  );
}
// React 自动：① 提升到 <head> ② 多个 <title> 取最深的（嵌套覆盖）
//             ③ <meta name=> 同 name 去重 ④ SSR / RSC 都正确工作

例 3：Asset preload / preinit（性能优化）

import { preload, preinit } from 'react-dom';

function VideoPlayer({ src, posterUrl }) {
  // 在 render 时声明依赖，浏览器尽早加载
  preload(posterUrl, { as: 'image' });
  preload(src, { as: 'video' });
  preinit('/styles/player.css', { as: 'style' });

  return <video src={src} poster={posterUrl} controls />;
}
// 比手动写 <link rel="preload"> 更优雅 + 自动 dedupe

05 常见陷阱

陷阱 1：升级 19 后所有 defaultProps 警告

现象：项目升级 React 19 后控制台大量警告 Support for defaultProps will be removed from function components in a future major release.

规避：批量迁移到 ES 默认参数。如果用 React Codemod，跑 npx codemod react/19/replace-default-props 自动改写。class 组件的 defaultProps 不会警告，可保留。

陷阱 2：以为 ref as prop 就不需要 useImperativeHandle

现象：把 function MyComp({ ref }) 直接 spread 到内部多个元素 <div ref={ref}>...<input ref={ref}>。第一个 ref 被覆盖。

根因：ref 是单 slot，不能像 className 那样合并。要暴露多个 ref，仍需 useImperativeHandle（B10）。

陷阱 3：在 Client Component 用 React.cache

现象：'use client' 文件里 import cache，运行时 cache 不工作 / 编译报错。

根因：cache 的"per request 隔离" 概念在客户端没有意义（客户端没有"request"）。客户端要 dedupe 用 SWR / TanStack Query / Map 缓存。

陷阱 4：startTransition 顶层 API 期望 isPending

现象：写 const isPending = startTransition(...)，isPending 永远是 undefined。

根因：顶层 startTransition 不返回 isPending—— 因为它能在组件外调用，没有"当前组件" 的上下文存储 pending 状态。需要 isPending 必须用 useTransition Hook。

陷阱 5：renderToString 用于 Suspense 应用

现象：用 renderToString 渲染含 Suspense 的应用，输出 HTML 全是 fallback 内容（loading…）。

根因：renderToString 是同步 API，遇到 Suspense throw promise 时直接渲染 fallback，不等 promise resolve。要流式 + Suspense，必须用 renderToPipeableStream / renderToReadableStream。

陷阱 6：动态 title 没生效

现象：路由切换 / state 变化时，title 不变。

根因：React 19 的元数据提升发生在 render 期间。如果组件没重 render（例如全局 state 没驱动这个组件），title 不会变。

规避：① 确保 title 所在组件在路由 / 数据变化时重 render ② 用 SSR / RSC 时，title 在 Server 端渲染输出 ③ 极端动态场景（imperatively 改 title）仍可用 document.title = '...' in useEffect。
