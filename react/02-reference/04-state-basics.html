B4 · State 基础

useState 是 React 最常用的 Hook，但陷阱密度也是 Hooks 之最—— stale state、批量更新顺序、闭包捕获、初始值惰性求值、setState 异步性。本章把 useState 的完整心智讲清，包括：何时用 updater function、State 为什么必须不可变、setState 多次调用如何合并、闭包捕获的旧 state 如何成为 bug 源。这章是所有后续 Hooks 章的基石—— useReducer / useEffect / 自定义 Hooks 都建立在 setState 心智之上。

📐 本章是 Hooks 基础

后续 B6 useState 深度（hooks linked list 视角）/ B7 useEffect / B8 useReducer 都基于本章建立的心智。如果你已经熟练用 useState，可以快速浏览跳到陷阱部分；如果你刚开始学，本章每一节都要细读。

阅读时长 ~25 分钟 前置 B1 渲染模型 + B3 Props + JS · 闭包（推荐） 对应 react.dev useState API · State as a Snapshot

01 速览

三句话定义本章：

  1. useState 返回 [currentState, setState] 二元组。setState 是 React 内部维护的"请求更新此 state" 函数—— 调用它 React 把更新加入队列、安排重新渲染。函数引用永远稳定（不会随 render 变化），可放心传给子组件 / useEffect 依赖。
  2. State 是"当前 render 的快照"。在一次 render 中，count 永远是当时的值；调用 setCount(count + 1) 不会立刻改变 count—— 它在下次 render 时 是新值。这就是 "state 像快照" 心智 —— 也是 stale state 陷阱的根源。
  3. 2 个核心规则 + 1 个性能技巧：① State 不可变（不能直接 state.x = 1，必须 setState({...state, x: 1})）② 多次调用基于上一次结果时用 updater function（setCount(c => c + 1) 而非 setCount(count + 1)）③ 初始值昂贵时用 lazy initializer（useState(() => expensive())）。

看一段最小代码识别 useState 三个核心概念：

javascript · useState 完整签名 + 3 个核心概念 
function Counter() {
  // ① 普通初始值：每次 render 都求值（但只首次使用）
  const [count, setCount] = useState(0);

  // ② Lazy initializer：函数仅首次 render 调用
  const [items, setItems] = useState(() => loadFromLocalStorage());

  // ③ Updater function：基于上一次 state 计算新值
  const handleClick = () => {
    setCount(c => c + 1);    // ← 推荐
    setCount(c => c + 1);    // ← 多次调用合并：c 拿到最新值
    setCount(c => c + 1);    // ← 最终 count + 3
    // 如果用 setCount(count + 1) 三次：count 是当前 render 的快照值，
    // 三次都基于同一旧值，最终只 +1
  };

  return <button onClick={handleClick}>{count}</button>;
}

02 底层逻辑

useState 完整签名

typescript · useState 类型定义 
// 重载 1：传初始值
function useState<S>(initialState: S | (() => S)): [S, Dispatch<SetStateAction<S>>];

// 重载 2：仅传类型，没有初始值（state 类型 includes undefined）
function useState<S = undefined>(): [S | undefined, Dispatch<SetStateAction<S | undefined>>];

type Dispatch<A> = (value: A) => void;
type SetStateAction<S> = S | ((prevState: S) => S);

// 用法对比：
const [count, setCount] = useState(0);              // S 推导为 number
const [name, setName] = useState<string | null>(null);  // 显式指定 S
const [user, setUser] = useState<User>();           // 没传值，user 是 User | undefined

// setState 接受两种参数：
setCount(5);                  // 直接值
setCount((c) => c + 1);       // updater function

关键事实：

  • setState 引用永远稳定—— React 保证。可以放心传给 useEffect 依赖（不必在 deps 数组里加）/ 子组件 prop（不会触发不必要 memo 失效）
  • setState 是异步的—— 调用后 state 不立刻变。下一次 render 时变。这就是 setState 后立刻 console.log(state) 还是旧值的原因
  • 同一函数内多次 setState—— 见下面 "updater function" 节

Lazy Initializer —— 昂贵初始值优化

初始值是普通表达式时，每次 render 都求值（虽然只用首次结果）：

javascript · 普通初始值 vs Lazy initializer 
// ❌ 每次 render 都跑 expensive()，只是结果被丢弃
function Component() {
  const [data, setData] = useState(expensiveCompute());  // 跑一次取结果，下次再跑一次取结果丢弃
  return <div>{data}</div>;
}

// 同理：每次 render 都从 localStorage 读
const [theme, setTheme] = useState(localStorage.getItem('theme'));
// localStorage 读不算昂贵，但仍是浪费

// ✅ Lazy initializer：函数仅首次 render 调用
function Component() {
  const [data, setData] = useState(() => expensiveCompute());
  return <div>{data}</div>;
}

const [theme, setTheme] = useState(() => localStorage.getItem('theme') ?? 'light');

// 何时用 lazy：
// 1. 计算昂贵（解析大 JSON / 算 hash / 处理大数组）
// 2. 副作用（读 localStorage / sessionStorage）
// 3. 想避免每次 render 创建新对象（但仅初始值用对象的场景）

// 何时不用：
// 简单字面量（0 / '' / null / [] / {}）—— 创建开销 << 函数调用开销

Updater Function vs 直接传值 ⭐

这是 useState 最重要的概念—— 不理解这个会写出大量"看起来对但行为错" 的代码：

javascript · 直接传值 vs Updater function 
// === 场景：同一 handler 多次 setState ===
function Counter() {
  const [count, setCount] = useState(0);

  // ❌ 直接传值：count 是当前 render 的快照
  const handleClick1 = () => {
    setCount(count + 1);    // 假设当前 count = 0 → 这里传 1
    setCount(count + 1);    // count 还是 0（同一 render 的快照）→ 还是传 1
    setCount(count + 1);    // 同上 → 还是 1
    // 结果：count 变成 1（不是 3！）
  };

  // ✅ Updater function：每次基于上一次更新结果
  const handleClick2 = () => {
    setCount(c => c + 1);   // c = 0 → 1
    setCount(c => c + 1);   // c = 1（拿到上一次结果）→ 2
    setCount(c => c + 1);   // c = 2 → 3
    // 结果：count 变成 3 ✅
  };

  return (
    <>
      <button onClick={handleClick1}>Buggy +3</button>
      <button onClick={handleClick2}>Correct +3</button>
    </>
  );
}

// === 异步场景：setTimeout / Promise ===
const handleAsync = () => {
  setTimeout(() => {
    setCount(count + 1);    // ⚠️ count 是 timeout 创建时的快照（旧值）
    // 如果 timeout 期间用户多点了几次按钮，count 已经更新但这里仍用旧值
  }, 1000);
};

const handleAsyncCorrect = () => {
  setTimeout(() => {
    setCount(c => c + 1);    // ✅ 拿最新 state，不依赖闭包捕获
  }, 1000);
};

什么时候用 updater function：

  • 新值依赖旧值（计数器 / 切换 boolean / 给数组加元素 / 给对象加字段）—— 用 updater function
  • 同一 handler 多次 setState —— 用 updater function（保证基于上一次结果）
  • 异步代码里 setState（setTimeout / Promise / 事件 listener） —— 用 updater function（避免 stale closure）
  • 新值不依赖旧值（直接设新值）—— 直接传值即可

团队规则建议：默认全部用 updater function—— 简单 + 永远正确。少打几个字符不值得 stale state bug。

State 不可变规则

短重述：A2 设计哲学已详述"不可变 + 重渲染"决策。state 必须创建新引用，不能 mutate：

javascript · 不可变更新模式 
// ❌ Mutate
const [user, setUser] = useState({ name: 'Alice', age: 30 });
user.age = 31;
setUser(user);  // 同一引用，React Object.is 比较跳过更新 → UI 不更新

// ✅ 创建新对象
setUser({ ...user, age: 31 });
setUser(prev => ({ ...prev, age: 31 }));      // 推荐 updater

// 数组操作 —— 全部创建新数组
const [todos, setTodos] = useState([]);
setTodos(prev => [...prev, newTodo]);                            // append
setTodos(prev => [newTodo, ...prev]);                            // prepend
setTodos(prev => prev.filter(t => t.id !== id));                 // remove
setTodos(prev => prev.map(t => t.id === id ? { ...t, done: true } : t));  // update item
setTodos(prev => {                                                // insert at index
  const next = [...prev];
  next.splice(index, 0, newItem);
  return next;
});

// 嵌套对象更新地狱 —— 用 Immer 缓解
import { produce } from 'immer';
setUser(produce(prev => {
  prev.profile.address.city = 'Beijing';   // 看起来像 mutate，实际产生新引用
}));

// 详细 mutation pattern 见 react.dev "Updating Objects/Arrays in State"
// 详细引用比较根因见 A2 设计哲学

自动 batching —— 多个 setState 合并

短重述：A4 Concurrent 已详述。React 18+ 在所有上下文（事件回调 / Promise / setTimeout / 原生事件）自动合并多个 setState 到一次渲染。

javascript · 自动 batching 行为 
// 一个事件回调里 3 个 setState
function handleClick() {
  setCount(c => c + 1);
  setName('Alice');
  setItems(prev => [...prev, newItem]);
}
// React 17 / 18：都 batch 成 1 次渲染

// Promise.then 里 3 个 setState
fetch('/api').then(() => {
  setCount(c => c + 1);
  setName('Alice');
  setItems(prev => [...prev, newItem]);
});
// React 17：3 次渲染（性能差）
// React 18：自动 batch 成 1 次渲染 ✅

// 强制不 batch（极少需要）：
import { flushSync } from 'react-dom';
flushSync(() => setCount(c => c + 1));  // 强制立即同步渲染再继续

State 与闭包陷阱（stale state）⭐

这是 useState 最经典的陷阱。组件函数每次 render 都创建新闭包—— 每个闭包捕获的 state 是那次 render 的快照。详细闭包机制见 JS · 闭包。

javascript · setInterval 中的 stale state 
// ❌ 经典陷阱：setInterval 永远显示 0 → 1 → 2 → 3 ...
// 但用户预期看到的是当前 count 值
function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);   // ⚠️ count 是 useEffect 创建时的快照（首次 render 时是 0）
      // 闭包捕获了"那一刻的 count"，永远是 0 + 1 = 1
      // 第二次回调还是 0 + 1 = 1，第三次还是 0 + 1 = 1 ...
      // 结果 count 永远停在 1
    }, 1000);
    return () => clearInterval(id);
  }, []);   // 空依赖 —— effect 仅初次创建

  return <div>{count}</div>;
}

// 错误的"修复"：把 count 加进 useEffect 依赖
useEffect(() => {
  const id = setInterval(() => {
    setCount(count + 1);
  }, 1000);
  return () => clearInterval(id);
}, [count]);  // ⚠️ 每次 count 变就 cleanup + 重新 setInterval
// 结果：每次 count 变都重启 timer —— interval 时间不准（每次重置 1000ms）

// ✅ 正确：用 updater function（不依赖闭包捕获的 count）
useEffect(() => {
  const id = setInterval(() => {
    setCount(c => c + 1);   // 拿最新 state，不靠闭包
  }, 1000);
  return () => clearInterval(id);
}, []);

// useEffect 依赖闭包陷阱在 B7 详述
// 还有 useRef + useCallback 各种解决方案，见 B10 / B9

多个 useState vs 一个 useReducer

多个相关 state 应该合一个 useReducer 还是多个 useState？

场景 推荐
独立 state（互不影响） 多个 useState
2-3 个相关 state（如 form 字段） 多个 useState 或一个 useState 对象（取决于偏好）
4+ state 互相联动（一个 state 变要更新另几个） useReducer（单点更新，避免遗漏）
"添加 todo" / "切换 todo 状态" 等多种 action useReducer（action 类型清晰）
需要 dispatch 模式 / time-travel 调试 useReducer（自然适配）

详细 useReducer 用法见 B8 · useReducer / useContext（编写中）。

StrictMode 双重渲染对 setState 的影响

B1 已讲过 StrictMode 让组件函数双重调用。对 setState 的具体影响：

  • useState 的 initial value 函数双调—— Lazy initializer 跑两次。如果 initial 函数有副作用（如 localStorage.getItem 没问题，但 localStorage.setItem 会出问题），就会 bug
  • updater function 双调—— React 主动跑两次帮你检测 updater function 不是纯函数
  • setState 调用双倍—— 但实际只触发一次渲染（自动 batching）
javascript · StrictMode 暴露 updater 副作用 
// ❌ updater function 有副作用 —— StrictMode 暴露
let logId = 0;
function Component() {
  const [items, setItems] = useState([]);

  const handleAdd = () => {
    setItems(prev => {
      logId++;                           // ❌ updater 有副作用
      console.log('Adding item', logId);  // StrictMode 下打印两次
      return [...prev, { id: logId }];   // logId 双倍跳跃
    });
  };
  return <button onClick={handleAdd}>Add</button>;
}

// ✅ updater 纯函数：只用 prev 计算新值
const handleAdd = () => {
  setItems(prev => [...prev, { id: prev.length }]);
  // 副作用（log / 生成 id）放别处
};

场景 B · 复杂初始值优化

组件 mount 时要从 localStorage 读 + 解析大 JSON + 转换格式—— 不能让这个昂贵计算在每次 render 都跑。用 lazy initializer：

javascript · lazy initializer 实战 
function TodoApp() {
  // ❌ 每次 render 都读 + parse + 转换
  const [todos, setTodos] = useState(
    JSON.parse(localStorage.getItem('todos') ?? '[]').map(parseAndNormalize)
  );

  // ✅ 仅首次 render 调用
  const [todos, setTodos] = useState(() => {
    const raw = localStorage.getItem('todos');
    if (!raw) return [];
    try {
      return JSON.parse(raw).map(parseAndNormalize);
    } catch (e) {
      return [];   // 容错
    }
  });

  return <TodoList items={todos} onChange={setTodos} />;
}

// 何时检测到该用 lazy initializer？
// - 用 React DevTools Profiler 看初始 render 时间
// - 注意：lazy initializer 不修复"每次 render 都跑" 的问题（普通 init 表达式）
//          它修复的是"仅首次需要的昂贵计算"的浪费

场景 C · WebSocket / 长轮询连接里的 setState

建立 WebSocket 连接接收消息，每收到一条 push 进 messages 数组。WebSocket 的 onmessage 回调是异步的、跨多次 render 都活着 —— 必须用 updater function：

javascript · WebSocket + setState 
function ChatRoom({ roomId }) {
  const [messages, setMessages] = useState([]);

  useEffect(() => {
    const ws = new WebSocket(`wss://api.example.com/chat/${roomId}`);

    ws.onmessage = (event) => {
      const msg = JSON.parse(event.data);
      // ❌ setMessages([...messages, msg])：messages 是 effect 创建时的旧值（[]）
      //    永远只显示最后一条消息
      // ✅ updater function 拿最新 state
      setMessages(prev => [...prev, msg]);
    };

    return () => ws.close();
  }, [roomId]);

  return <ul>{messages.map(m => <li key={m.id}>{m.text}</li>)}</ul>;
}

// 关键：任何"跨多次 render 活着的回调"都要用 updater function
// - WebSocket onmessage / onerror / onclose
// - addEventListener 的 listener
// - setInterval / setTimeout 回调
// - Promise 链中的回调（如果跨多次 render）

04 典型代码

3 种"派生 state" 的处理方式

javascript · 派生 state 不存 state，直接计算 
// 需求：根据 firstName + lastName 显示 fullName

// ❌ 风格 1：把派生值存 state（最常见错误）
function UserCard() {
  const [firstName, setFirstName] = useState('');
  const [lastName, setLastName] = useState('');
  const [fullName, setFullName] = useState('');  // ❌ 派生值不该是 state

  // 然后必须每次 firstName / lastName 变都同步：
  useEffect(() => {
    setFullName(`${firstName} ${lastName}`);
  }, [firstName, lastName]);
  // 问题：① 同步逻辑出 bug 风险 ② 多一次渲染 ③ 类型不一致风险

  return <div>{fullName}</div>;
}

// ✅ 风格 2：直接计算（推荐）
function UserCard() {
  const [firstName, setFirstName] = useState('');
  const [lastName, setLastName] = useState('');
  const fullName = `${firstName} ${lastName}`;  // ✅ 每次 render 直接算

  return <div>{fullName}</div>;
}

// ✅ 风格 3：useMemo（仅当计算昂贵时）
function UserCard() {
  const [firstName, setFirstName] = useState('');
  const [lastName, setLastName] = useState('');

  const fullName = useMemo(
    () => expensiveFormatName(firstName, lastName),
    [firstName, lastName]
  );

  return <div>{fullName}</div>;
}

// 决策：
// - 派生 + 计算便宜 → 直接算（风格 2）
// - 派生 + 计算昂贵（> 1ms）→ useMemo（风格 3）
// - 不要存 state（风格 1）

当 setState 不被识别 —— 跨 React 边界

javascript · 跨 React 边界传 setState 
// 需求：把 setState 传给非 React 代码（如 Web SDK / 全局事件）

// 场景：监听全局事件 + 更新 React state
function MapComponent() {
  const [marker, setMarker] = useState(null);

  useEffect(() => {
    const handler = (e) => setMarker({ lat: e.lat, lng: e.lng });

    // 第三方 SDK 注册全局回调
    window.MapSDK.on('click', handler);

    return () => window.MapSDK.off('click', handler);
  }, []);   // setMarker 引用稳定，不需要加依赖

  return marker ? <Marker {...marker} /> : null;
}

// 关键事实：
// 1. setMarker 引用永远稳定 —— useEffect deps 数组可不写
// 2. handler 闭包捕获的 setMarker 是初始的 setMarker —— 仍然有效
// 3. 与 stale state 陷阱不同：state 值会过期，但 setState 函数引用永远稳定
// 4. 如果 handler 用 marker（state 值），就要用 updater function
//    setMarker(prev => ...)

05 常见陷阱

⚠️ 陷阱 1：setState 后立刻 console.log(state) 还是旧值

这不是 bug，是 React 设计。setState 是异步的—— 它把更新加入队列、安排重新渲染，但当前 render 的 state 不变。

javascript · setState 异步性 
function Counter() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1);
    console.log(count);  // 仍是当前 render 的 count（不是 +1 后的值）
    // 想看新值？在下一次 render 里，或在 useEffect 里
  };

  // ✅ 想看变化后的值
  useEffect(() => {
    console.log('Count is now', count);
  }, [count]);

  return <button onClick={handleClick}>{count}</button>;
}

规避：理解"state 是当前 render 的快照"心智。要看"setState 之后"用 useEffect。

⚠️ 陷阱 2：在 setState 中 mutate state

setItems([...items, ...]); items.sort();—— 同步修改了刚 setState 的数组，下次 render 取的还是 mutate 后的数组（因为 setState 异步）。

规避：永远 setItems(prev => [...prev].sort())—— 创建新引用。或用 toSorted（ES2023，不可变排序）。

⚠️ 陷阱 3：useState 接收对象但忘记 mutate 不更新

setUser(user);（同一引用）—— React Object.is 比较跳过更新。新人最常踩。

规避：永远创建新引用：setUser({...user}) 或 setUser(prev => ({...prev}))。Immer 缓解嵌套。

⚠️ 陷阱 4：把派生值存 state

useState(() => firstName + lastName)—— 派生值存 state 后必须 useEffect 同步，多一次渲染 + 同步逻辑出错。

规避：直接计算 const fullName = firstName + lastName。昂贵时 useMemo。state 只存"独立的、用户输入或 API 来的源数据"，派生值都直接算。

⚠️ 陷阱 5：异步代码（setTimeout / Promise）里直接传值 setState

最常见 bug 形态。异步回调跨多次 render 活着，捕获的 state 是闭包创建时的旧值。

规避：异步代码里始终用 updater function。setX(prev => ...)—— 拿最新 state，不依赖闭包。这是团队代码评审的固定项。
