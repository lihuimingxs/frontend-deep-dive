B15 · Error Boundary

Error Boundary 是 React 16 引入的错误捕获机制—— 也是 React 19 时代唯一仍必须用 class component 的场景。本章讲清 Error Boundary 的完整模式（componentDidCatch + getDerivedStateFromError）、不捕获的 4 种情况（最容易踩坑）、边界粒度选择、重试机制，以及社区事实标准 react-error-boundary 库 和 React 19 新增的 onUncaughtError / onCaughtError root options。读完本章 React 主题 Part 3 渲染与并发 4 章 ✅ 收官。

⚠️ Error Boundary 不是万能

Error Boundary 仅捕获渲染期错误（render / lifecycle / constructor）。4 种情况完全不捕获：① event handler 内的错（用 try/catch）② 异步代码的错（fetch / setTimeout / Promise）③ SSR 服务端 HTML 生成阶段的错 ④ Error Boundary 自身的错。这 4 个限制是工程实战中最容易踩的坑。

阅读时长 ~25 分钟 前置 B1 渲染模型 + B13 Suspense（双重边界配合） 对应 react.dev Error Boundary

01 速览

三句话定义本章：

  1. Error Boundary 必须是 class component—— React 19 时代唯一的 class 用例。它通过实现 2 个特殊静态 / 实例方法（static getDerivedStateFromError + componentDidCatch）来声明自己是 Error Boundary。函数组件没有等价的 Hook（React 至今未提供 useErrorBoundary）。
  2. 仅捕获渲染期错误：组件 render 函数 / 生命周期 / constructor 抛出的错。不捕获：event handler / async 代码（fetch / setTimeout / Promise）/ SSR / Boundary 自身错。
  3. 实战推荐用 react-error-boundary 库（社区事实标准）—— 提供函数式 API + reset 机制 + onError 回调 + Suspense 集成。React 19 新增 root options（onUncaughtError / onCaughtError / onRecoverableError）作为全局兜底，但不替代 Error Boundary 的局部处理。

看一段最小代码：

javascript · Error Boundary 完整 class 模式 
// 必须 class component
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  // 1. 静态方法：从 error 派生新 state（render 用）
  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  // 2. 生命周期：捕获后副作用（log / report）
  componentDidCatch(error, errorInfo) {
    console.error('Caught:', error, errorInfo.componentStack);
    // 上报到 Sentry / Datadog / 自家监控
    Sentry.captureException(error, { extra: errorInfo });
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? <div>Something went wrong.</div>;
    }
    return this.props.children;
  }
}

// 用法
function App() {
  return (
    <ErrorBoundary fallback={<ErrorPage />}>
      <Dashboard />
    </ErrorBoundary>
  );
}
// Dashboard 内部任何渲染错误 → Error Boundary 显示 ErrorPage

02 底层逻辑

2 个特殊方法的角色

方法 调用时机 用途
static getDerivedStateFromError(error) render 阶段（错误发生后） 更新 state 以渲染 fallback —— 必须纯函数 + 同步
componentDidCatch(error, errorInfo) commit 阶段（fallback 已渲染） 副作用：log / report / 通知用户 / 触发外部处理

为什么要分两个方法：

  • getDerivedStateFromError 是静态 + 纯函数—— React 在 render 阶段调用，要求纯函数（A3 详述纯函数规则）。不能调 console.log / fetch / setState
  • componentDidCatch 是实例 + 副作用—— commit 阶段调用，可以做任何副作用
  • 这是"render 纯函数 + commit 副作用" 的强制分离
javascript · 两个方法的精确分工 
class ErrorBoundary extends React.Component {
  state = { hasError: false };

  static getDerivedStateFromError(error) {
    // ✅ 仅返回新 state
    return { hasError: true };

    // ❌ 不能在这里调 console.log / fetch / setState
    // React 18+ Concurrent rendering 可能多次调用 → 副作用会跑多次
  }

  componentDidCatch(error, errorInfo) {
    // ✅ 副作用全放这里
    console.error('Boundary caught:', error);
    logToService(error, errorInfo);
    notifyUser('Something broke');

    // 可以 setState 进一步细化（一般不必要）
    this.setState({ errorMessage: error.message });
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback />;
    }
    return this.props.children;
  }
}

不捕获的 4 种情况 ⭐

🎯 这 4 种错误必须用其他方式处理

Error Boundary 只是"渲染期错误捕获器"。下面 4 种错误静默丢失，需要你主动处理。

① Event handler 内的错

javascript · event handler 错 
function Button() {
  return (
    <button onClick={() => {
      throw new Error('Boom');   // ❌ Error Boundary 不捕获
    }}>Click</button>
  );
}

// 原因：
// onClick 不在 render 阶段执行（用户点击时才跑）
// React 不知道这个错来自组件树
// 错抛到 window.onerror（未处理则浏览器控制台打印）

// ✅ 修复：用 try/catch
<button onClick={() => {
  try {
    doRiskyOperation();
  } catch (e) {
    console.error(e);
    setError(e);   // 用 state 显示错误
  }
}}>

② 异步代码的错

javascript · 异步错 
function Component() {
  useEffect(() => {
    setTimeout(() => {
      throw new Error('Async boom');   // ❌ Error Boundary 不捕获
    }, 1000);
  }, []);
}

// 原因：
// setTimeout 回调不在 render 调用栈内
// React 完全不知道这个错（错跑到 window.onerror）

// ✅ 修复：用 try/catch + setState 触发 Error Boundary
function Component() {
  const [error, setError] = useState(null);

  useEffect(() => {
    setTimeout(() => {
      try {
        doAsyncWork();
      } catch (e) {
        setError(e);   // 触发 state 变化
      }
    }, 1000);
  }, []);

  if (error) throw error;   // 在 render 阶段 throw → Error Boundary 捕获

  return ...;
}

// 同样陷阱：
// - Promise.then 内的错（要 .catch）
// - fetch 内的错（要 .catch）
// - WebSocket onmessage 内的错

③ SSR 服务端 HTML 生成阶段的错

javascript · SSR 错 
// 服务端 renderToString / renderToPipeableStream 阶段
// 组件 render 抛错 —— Error Boundary 不捕获（仅客户端 hydrate 后才捕获）

// React 18 改进：renderToPipeableStream onError option
import { renderToPipeableStream } from 'react-dom/server';

const { pipe, abort } = renderToPipeableStream(<App />, {
  onError(error) {
    console.error('SSR error:', error);
    logToService(error);
    // 决定是否 abort 或继续 fallback
  },

// React 19 新 API（Next.js App Router 也用类似机制）：
// 服务端组件抛错 → 沿组件树找最近的 error.tsx 边界
// app/dashboard/error.tsx 是 Next.js 约定 Error Boundary
'use client';   // 必须客户端组件
export default function Error({ error, reset }) {
  return (

      <h2>Something went wrong</h2>

}

④ Error Boundary 自身的错

javascript · Boundary 自身错 
class ErrorBoundary extends React.Component {
  state = { hasError: false };

  static getDerivedStateFromError(error) {
    return { hasError: true };
  }

  render() {
    if (this.state.hasError) {
      return <FallbackThatThrows />;   // ❌ fallback 自己抛错
      // → Error Boundary 不能捕获自己的错
      // → 沿 Fiber 上溯找外层 Error Boundary
      // → 没有外层 → 整个 React tree unmount
    }
    return this.props.children;
  }
}

// 教训：
// 1. fallback UI 必须简单 + 不依赖可能失败的资源
// 2. 在最外层加一个"最终兜底" Error Boundary（永远不可能错的简单 div）
// 3. React 19 onUncaughtError root option 作为最终兜底（见下文）

// ✅ 正确：fallback 极简
render() {
  if (this.state.hasError) {
    return <div>App crashed. <a href="/">Go home</a></div>;
    // 不依赖任何状态 / 数据 / 第三方库
  }
}

边界粒度选择

javascript · 边界粒度策略 
// 策略 1：单一全局边界（粗粒度）
function App() {
  return (
    <ErrorBoundary fallback={<ErrorPage />}>
      <Layout>
        <Header />
        <Sidebar />
        <Main />
      </Layout>

// 优点：简单
// 缺点：任何组件错误整页崩 → 用户被强制看 ErrorPage

// 策略 2：按区域分边界（中粒度）
function App() {
  return (
    <ErrorBoundary fallback={<AppCrashed />}>
      <Header />
      <ErrorBoundary fallback={<SidebarFallback />}>
        <Sidebar />
      </ErrorBoundary>
      <ErrorBoundary fallback={<MainFallback />}>
        <Main />
      </ErrorBoundary>
    </ErrorBoundary>
  );
}
// Sidebar 错只让 Sidebar 显示 SidebarFallback，Main 仍正常
// 推荐：每个独立功能区一个 Boundary

// 策略 3：每个 widget 独立边界（细粒度）—— dashboard 模式
function Dashboard() {
  return (
    <Grid>
      {widgets.map(w => (
        <ErrorBoundary
          key={w.id}
          fallback={<WidgetError name={w.name} />}
        >
          <Widget {...w} />
        </ErrorBoundary>
      ))}
    </Grid>
  );
}
// 任何 widget 崩只让该格显示错误，其他正常
// 适合：dashboard / 多 panel / 多组件并列

// 决策：
// 1. 顶层必须有兜底 Error Boundary（避免白屏）
// 2. 每个独立功能区加一个（避免连锁崩溃）
// 3. 关键 widget（chart / map / 第三方组件）单独包
// 4. 不要每个组件都包（过细维护成本高）

重试机制（reset）

javascript · Error Boundary 重试机制 
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  // 暴露 reset 方法
  reset = () => {
    this.setState({ hasError: false, error: null });
  };

  render() {
    if (this.state.hasError) {
      return (
        <div>
          <h2>Error: {this.state.error?.message}</h2>
          <button onClick={this.reset}>Try again</button>
        </div>
      );
    }
    return this.props.children;
  }
}

// 重试模式：
// 1. 用户点 "Try again" → reset state → React 重新渲染 children
// 2. 如果错误根因还在（如 props 有问题）→ 立即再 throw
// 3. 解决根因（用户操作 / 重新加载）后 → 成功渲染

// 进阶：用 key 强制重新挂载子树
class ErrorBoundary extends React.Component {
  state = { hasError: false, resetKey: 0 };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  reset = () => {
    this.setState({ hasError: false, resetKey: prev => prev + 1 });
  };

  render() {
    if (this.state.hasError) {
      return <ErrorFallback onReset={this.reset} />;
    }
    // 用 resetKey 强制重新挂载（B12 key 强制重置）
    return <React.Fragment key={this.state.resetKey}>{this.props.children}</React.Fragment>;
  }
}
// 重试时整个 children 子树重新 mount —— effect cleanup + 重跑

javascript · react-error-boundary 完整用法 
import { ErrorBoundary } from 'react-error-boundary';

// 基础用法
<ErrorBoundary
  fallback={<div>Something went wrong</div>}
>
  <App />
</ErrorBoundary>

// 完整用法 —— 函数式 fallback + reset + onError
<ErrorBoundary
  fallbackRender={({ error, resetErrorBoundary }) => (
    <div role="alert">
      <p>Error: {error.message}</p>
      <button onClick={resetErrorBoundary}>Try again</button>
    </div>
  )}
  onError={(error, info) => {
    Sentry.captureException(error, { extra: info });
  }}
  onReset={() => {
    // reset 时执行（清空缓存 / 重新拉数据等）
    queryClient.invalidateQueries();
  }}
  resetKeys={[userId]}   // 这些值变化时自动 reset
>
  <UserProfile userId={userId} />
</ErrorBoundary>

// 配 useErrorBoundary Hook（仅在 ErrorBoundary 内子组件用）
import { useErrorBoundary } from 'react-error-boundary';

function MyComponent() {
  const { showBoundary } = useErrorBoundary();

  const handleClick = async () => {
    try {
      await riskyOperation();
    } catch (e) {
      showBoundary(e);   // 手动触发 Error Boundary
    }
  };
}

// 这是 "事件 / 异步代码触发 Error Boundary" 的标准方式
// 解决 Error Boundary 不捕获 4 种情况的 ① 和 ②

// 这个库是社区事实标准 —— 几乎所有 React 项目都用

React 19 root options

javascript · React 19 全局错误处理 
import { createRoot } from 'react-dom/client';

createRoot(document.getElementById('root'), {
  // 1. 没被任何 Error Boundary 捕获的错（最终兜底）
  onUncaughtError: (error, errorInfo) => {
    console.error('Uncaught:', error, errorInfo);
    Sentry.captureException(error, { tags: { uncaught: true } });
    // 这是 React 19 之前在 window.onerror 才能捕获的错
  },

  // 2. 被 Error Boundary 捕获的错（用于全局 log）
  onCaughtError: (error, errorInfo) => {
    console.warn('Caught by boundary:', error);
    Sentry.captureException(error, { tags: { caught: true } });
    // 即使有 Boundary 处理，全局也能 log
  },

  // 3. React 自动恢复的错（hydration 错等）
  onRecoverableError: (error, errorInfo) => {
    console.warn('Recovered:', error);
    // React 已经处理 —— 仅供监控
  },
}).render(<App />);

// 关键作用：
// 1. onUncaughtError 是最终兜底—— 任何 Error Boundary 都没捕获的错都来这
// 2. 不替代局部 Error Boundary —— 仍要用 Boundary 提供 UI fallback
// 3. 全局 logging 集中（Sentry / Datadog 上报）
// 4. React 19 之前需要 window.onerror + window.onunhandledrejection 凑

// 推荐组合：
// - 顶层 + 关键区域：Error Boundary（提供 UI fallback）
// - createRoot options：全局 logging + 最终兜底

03 应用场景

场景 A · Dashboard 多 widget 独立 Error Boundary

管理后台 dashboard 8 个 widget 并列。某 widget 的图表组件偶尔崩—— 不能让整个 dashboard 崩。每个 widget 包独立 Boundary：

javascript · widget 独立 Boundary 
import { ErrorBoundary } from 'react-error-boundary';

const widgets = [
  { id: 'sales', component: SalesChart, name: 'Sales Chart' },
  { id: 'users', component: UserList, name: 'Users' },
  { id: 'analytics', component: Analytics, name: 'Analytics' },
  // ...
];

function Dashboard() {
  return (
    <Grid>
      {widgets.map(({ id, component: Widget, name }) => (
        <GridItem key={id}>
          <ErrorBoundary
            fallbackRender={({ error, resetErrorBoundary }) => (
              <WidgetError
                name={name}
                error={error}
                onRetry={resetErrorBoundary}
              />
            )}
            onError={(error, info) => {
              Sentry.captureException(error, {
                tags: { widget: id, name },
                extra: info,
              });
            }}
          >
            <Widget />
          </ErrorBoundary>
        </GridItem>
      ))}
    </Grid>
  );
}

// 效果：
// - 任何 widget 崩 → 仅该格显示 WidgetError + Retry 按钮
// - 其他 widget 仍正常工作
// - Sentry 收到 widget 名 + error stack 可定位
// - 用户可点 Retry 重试（或刷新页面）

场景 B · Suspense + Error Boundary 双重边界

B13 已铺垫，本节展开。19 时代典型 data fetching 模式：

javascript · 双重边界完整模式 
import { ErrorBoundary } from 'react-error-boundary';
import { Suspense, use } from 'react';

function UserProfile({ userPromise }) {
  const user = use(userPromise);
  return <UserCard {...user} />;
}

function App() {
  const userId = useStore(s => s.currentUserId);
  const userPromise = useMemo(() => fetchUser(userId), [userId]);

  return (
    <ErrorBoundary
      fallbackRender={({ error, resetErrorBoundary }) => (
        <div>
          <h2>Failed to load user</h2>
          <p>{error.message}</p>
          <button onClick={resetErrorBoundary}>Retry</button>
        </div>
      )}
      resetKeys={[userId]}    {/* userId 变化自动 reset */}
    >
      <Suspense fallback={<Spinner />}>
        <UserProfile userPromise={userPromise} />
      </Suspense>
    </ErrorBoundary>
  );
}

// 边界协作：
// - Suspense 处理 loading（promise pending）
// - ErrorBoundary 处理错误（promise rejected → use throw）
// - resetKeys={[userId]} 让用户切换时自动 reset Boundary
// - 组件代码 UserProfile 看起来同步，无 loading / error state

// 推荐顺序：ErrorBoundary 在外、Suspense 在内
// 反过来也工作但语义不清

场景 C · 异步代码触发 Error Boundary（useErrorBoundary）

异步 fetch 失败时想让 Error Boundary 接管—— 用 react-error-boundary 的 useErrorBoundary Hook：

javascript · 异步触发 Boundary 
import { useErrorBoundary } from 'react-error-boundary';

function Form() {
  const { showBoundary } = useErrorBoundary();
  const [submitting, setSubmitting] = useState(false);

  const handleSubmit = async (data) => {
    setSubmitting(true);
    try {
      await api.submit(data);
    } catch (error) {
      showBoundary(error);   // 手动触发外层 Error Boundary
    } finally {
      setSubmitting(false);
    }
  };

  return <form onSubmit={handleSubmit}>...</form>;
}

// 调用方
<ErrorBoundary fallbackRender={({ error, resetErrorBoundary }) => (
  <ErrorView error={error} onRetry={resetErrorBoundary} />
)}>
  <Form />
</ErrorBoundary>

// 工作原理：
// showBoundary 内部 setState 一个 error，下次 render 时 throw
// → Error Boundary 捕获 → 显示 fallback
// 这是"异步错误转 Boundary" 的官方模式

04 典型代码

生产级 Error Boundary 完整模板

typescript · 生产级模板 
// 用 react-error-boundary 库 + 自家定制
import { ErrorBoundary, FallbackProps } from 'react-error-boundary';
import { ReactNode, useEffect } from 'react';

interface AppErrorBoundaryProps {
  children: ReactNode;
  context?: string;     // 用于 Sentry tag
  fallback?: ReactNode | ((props: FallbackProps) => ReactNode);
  resetKeys?: unknown[];
}

function DefaultErrorFallback({ error, resetErrorBoundary }: FallbackProps) {
  return (
    <div role="alert" className="error-fallback">
      <h2>Something went wrong</h2>
      <details>
        <summary>{error.message}</summary>
        <pre>{error.stack}</pre>
      </details>
      <button onClick={resetErrorBoundary}>Try again</button>
    </div>
  );
}

export function AppErrorBoundary({
  children,
  context = 'unknown',
  fallback = DefaultErrorFallback,
  resetKeys,
}: AppErrorBoundaryProps) {
  return (
    <ErrorBoundary
      fallbackRender={typeof fallback === 'function' ? fallback : () => fallback}
      onError={(error, info) => {
        // 上报
        Sentry.captureException(error, {
          tags: { context },
          extra: { componentStack: info.componentStack },
        });
        // 控制台 log（仅 dev）
        if (import.meta.env.DEV) {
          console.error('[ErrorBoundary]', context, error);
        }
      }}
      onReset={() => {
        // 重置时清缓存
        queryClient.invalidateQueries({ queryKey: [context] });
      }}
      resetKeys={resetKeys}
    >
      {children}
    </ErrorBoundary>
  );
}

// 用法
<AppErrorBoundary context="dashboard" resetKeys={[userId]}>
  <Dashboard />
</AppErrorBoundary>

<AppErrorBoundary context="settings" fallback={<SettingsError />}>
  <Settings />
</AppErrorBoundary>

Next.js App Router error.tsx 约定

javascript · Next.js Error Boundary 
// app/dashboard/error.tsx
'use client';   // ⚠️ 必须客户端组件

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    // 上报
    Sentry.captureException(error);
  }, [error]);

  return (
    <div>
      <h2>Something went wrong!</h2>
      <p>{error.message}</p>
      <button onClick={reset}>Try again</button>
    </div>
  );
}

// Next.js 会自动把 app/dashboard/* 包裹在这个 Error Boundary 内
// dashboard 内任何渲染错都被这个 error.tsx 捕获
// 可以嵌套：app/error.tsx 是顶层、app/dashboard/error.tsx 是局部

05 常见陷阱

⚠️ 陷阱 1：以为 Error Boundary 捕获 event handler 错

<button onClick={() => { throw ... }}>—— Error Boundary 完全不捕获。错跑到 window.onerror。

规避：event handler 内用 try/catch + setState 显示错误。或用 react-error-boundary 的 useErrorBoundary Hook 触发 Boundary。

⚠️ 陷阱 2：以为 Error Boundary 捕获异步错

setTimeout / Promise / fetch 内的错—— Error Boundary 不捕获。

规避：Promise 链 .catch / async 函数 try/catch / fetch 后判断 r.ok。失败后用 setState + render 时 throw 触发 Boundary，或用 useErrorBoundary。

⚠️ 陷阱 3：fallback UI 自身报错

ErrorFallback 用了某个可能失败的 prop / 第三方组件 → fallback 渲染时再次报错 → Boundary 不能自捕 → 沿 Fiber 上溯找外层 Boundary（如果有）→ 整个 app 崩。

规避：fallback UI 极简（纯 div + 文字 + button），不依赖任何可能失败的东西。最外层加"最终兜底" Boundary 或 React 19 onUncaughtError。

⚠️ 陷阱 4：在 getDerivedStateFromError 里调副作用

static getDerivedStateFromError(error) { console.log(error); return { hasError: true }; }—— React 18+ Concurrent rendering 可能多次调用，console.log 跑多次。

规避：getDerivedStateFromError 仅返回 state（纯函数）。所有副作用放 componentDidCatch。

⚠️ 陷阱 5：忘记顶层兜底 Error Boundary 致白屏

老代码没用 Error Boundary—— 任何小组件的渲染错让整个 React tree unmount → 白屏。

规避：① 顶层包一个 Error Boundary（最低成本兜底）② React 19 用 createRoot onUncaughtError 上报 + 显示降级 UI ③ Next.js 用 global-error.tsx 兜底。
