B6 · useState 深度

B4 讲了 useState 的用法。本章讲它怎么工作—— Hook linked list 节点结构、dispatch 函数为什么引用永远稳定、Object.is bailout 机制、函数 state 的特殊处理、useState 与 useReducer 的等价关系。读完后你能解释"useState 在引擎层做了什么"，调试 setState 相关 bug 时能直接定位根因。

📐 这是 Part 2 Hooks 全谱的起点

B4 是"用户视角"（怎么用 useState）；本章是"引擎视角"（怎么工作）。看 B6 时如果遇到不熟悉的术语（hooks linked list / Fiber / lane），随时回看 A3 Fiber 架构。两章互补能形成完整心智。

阅读时长 ~25 分钟 前置 B4 State 基础 + A3 Fiber 架构（强烈推荐） 对应 react.dev useState API

01 速览

三句话定义本章：

  1. 每个 useState 调用对应 Fiber.memoizedState 链表的一个节点。节点含 memoizedState（当前值）/ queue（pending 更新链表）/ next（指向下一个 hook 节点）。React 完全靠"调用顺序" 配对节点—— 这就是 Rules of Hooks 的引擎根因（A3 详述）。
  2. setState 函数引用永远稳定—— 不会随 render 变化。React 在初次 render 创建 dispatch 函数时把它存到 hook 节点的 queue.dispatch，每次 render 复用同一引用。这就是为什么可以放心传给 useEffect 依赖 / 子组件 prop / window.addEventListener。
  3. Object.is 比较 + bailout：setState(newValue) 时 React 用 Object.is(currentValue, newValue) 比较，相等则跳过整个组件 render（bailout）。这是 React 的"免费性能优化"—— 但它只跳过这次 setState 引发的 render，不跳过 parent 重渲染传下来的 render。

看一段代码，看 React 内部对应的数据结构变化：

javascript · 用户代码 vs 引擎数据结构 
function Counter() {
  const [count, setCount] = useState(0);    // hook 节点 #1
  const [name, setName] = useState('Alice'); // hook 节点 #2
  return /* JSX */;
}

// 第一次 render 后，Counter 的 Fiber.memoizedState：
//
// {                                  // hook 节点 #1
//   memoizedState: 0,                //   当前值
//   baseState: 0,
//   queue: {
//     pending: null,                 //   等待处理的更新链表
//     dispatch: setCount,            //   永远稳定的 dispatch 函数
//     lastRenderedReducer: basicStateReducer,
//     lastRenderedState: 0,
//   },
//   next: ─────────────►  {          // hook 节点 #2
//                           memoizedState: 'Alice',
//                           baseState: 'Alice',
//                           queue: {
//                             dispatch: setName,
//                             ...
//                           },
//                           next: null
//                         }
// }
//
// setCount(5) 触发：
// 1. 创建 update 对象 { lane: SyncLane, action: 5, next: ... }
// 2. 加入 queue.pending 环形链表
// 3. 调度本 Fiber 重渲染
//
// 第二次 render：
// - useState() 仍然第一次调用 → 取链表 #1 节点
// - 处理 queue.pending 中的 update（应用 action）→ 新 state = 5
// - 写回 memoizedState
// - 返回 [5, setCount]（dispatch 同一引用！）

02 底层逻辑

Hook linked list 节点结构（短重述）

短重述：A3 Fiber 架构详述了 hooks linked list 的完整机制。本节回顾useState 视角的核心字段：

javascript · Hook 节点字段（精确） 
// 简化版 Hook 节点（react-reconciler/src/ReactFiberHooks.js）
type Hook = {
  memoizedState: any,           // 当前 state（render 后固定）
  baseState: any,               // 上次 commit 后的"基线" state（用于 lane 调度）
  baseQueue: Update | null,     // 跨多次 render 累积的待处理 updates
  queue: UpdateQueue | null,    // 本次 render 待处理的 updates
  next: Hook | null,            // 链表指针 —— 指向下个 hook 节点
};

type UpdateQueue = {
  pending: Update | null,       // 等待处理的 update 链表（环形）
  dispatch: (a: A) => mixed,    // setState 函数 —— 引用永远稳定
  lastRenderedReducer: ((s, a) => s) | null,
  lastRenderedState: S | null,
};

type Update = {
  lane: Lane,                   // 优先级（A3 Lane 模型）
  action: A,                    // 用户传入的值或 updater function
  hasEagerState: boolean,       // 是否预算了 state（性能优化）
  eagerState: any,
  next: Update | null,          // 环形链表
};

// 关键：链表完全靠"调用顺序" 对齐
// 每次 useState() 调用 → 取下一个 Hook 节点
// 这就是 Rules of Hooks 的引擎根因

dispatch 函数引用稳定性（保证）

setState / dispatch 函数引用永远稳定—— 这是 React 的明确保证，写在 react.dev：

javascript · dispatch 引用稳定性的工程意义 
function Component() {
  const [count, setCount] = useState(0);
  const [name, setName] = useState('');

  // ✅ 1. 不需要在 useEffect 依赖里加 setState
  useEffect(() => {
    const id = setInterval(() => setCount(c => c + 1), 1000);
    return () => clearInterval(id);
  }, []);                                // setCount 不在依赖 → 仍然永远是同一引用

  // ✅ 2. 传给 React.memo 子组件不会破坏 memo
  return <Child onIncrement={setCount} />;  // 父重渲染时 setCount 引用不变 → memo 跳过

  // ✅ 3. 传给 useCallback 依赖不需要加
  const handler = useCallback(() => {
    setCount(c => c + 1);
    setName('');
  }, []);                                  // setCount / setName 不需在依赖里

  // ⚠️ 但传给 useMemo 不要加 setState 依赖（不会变 + 计算昂贵也不该依赖它）
}

// 引擎实现（简化）：
// 第一次 render：dispatch = bindFunction(currentlyRenderingFiber, queue);
// 之后每次 render：dispatch = hook.queue.dispatch（复用之前的）
// 同一函数引用 —— Object.is 永远 true

关键结论：setState 是免依赖的特殊值。useEffect / useCallback / useMemo 的依赖数组都不需要列它。但 ESLint react-hooks/exhaustive-deps 规则知道这点—— 不会要求你加。

Object.is 比较 + bailout 机制

setState(newValue) 时 React 内部用 Object.is 比较新旧值（详见 ECMA · B3 抽象操作 SameValue）。相等则 bailout—— 跳过整个组件本次 render：

javascript · Object.is bailout 行为实测 
function Component() {
  const [user, setUser] = useState({ name: 'Alice' });
  console.log('Render');

  return (
    <>
      {/* 同一引用 —— bailout */}
      <button onClick={() => setUser(user)}>
        Same reference (no rerender)
      </button>

      {/* 新对象同样内容 —— Object.is 不等 → 重渲染 */}
      <button onClick={() => setUser({ ...user })}>
        New reference (rerender)
      </button>

      {/* 同一原始值 —— bailout */}
      <button onClick={() => setCount(5)}>Set 5</button>  {/* count 已是 5 时不重渲染 */}

      {/* +0 vs -0 —— Object.is 不等！会重渲染 */}
      <button onClick={() => setCount(-0)}>Set -0</button>  {/* 当前 +0 时也重渲染 */}

      {/* NaN vs NaN —— Object.is 等！bailout */}
      <button onClick={() => setNum(NaN)}>Set NaN</button>  {/* 当前 NaN 时不重渲染 */}
    </>
  );
}

// Object.is 行为（同 ECMA SameValue）：
// - 严格相等的引用 / 原始值 → true（bailout）
// - 不同引用 → false（重渲染）
// - +0 vs -0 → false（区分）
// - NaN vs NaN → true（特殊处理）

// 关键限制：bailout 仅跳过本组件 render，不影响父组件
// 父重渲染传下 props 时，本组件仍会被调用（B1 讲过）

"预求值优化"（eager state）：React 18+ 引入。如果 setState 的新值能立即计算且与旧值 Object.is 相等，React 在 dispatch 时直接 bailout—— 不调度重渲染：

javascript · 预求值优化 
// 直接传值：可以预求值
setCount(5);
// React: 如果 currentState === 5 → 直接 bailout，不调度

// updater function：要先求值才能比较
setCount(c => c + 1);
// React: 必须先调用 updater 才知道新值 → 安排重渲染
// 但 render 时如果发现新值相等也会 bailout

// 性能小技巧：相同值场景用直接传值更高效
// 如：每次输入都 setQuery(e.target.value)，输入相同字符时 bailout 立刻发生

函数 state 的特殊处理 ⭐

useState 在用法 1（initial value）和用法 2（updater function）都接受函数—— React 必须区分这两种意图：

javascript · 函数作为 state 的特殊处理 
// 1. 函数作为初始值 → 调用作为 lazy initializer
const [data, setData] = useState(() => loadFromLocalStorage());
// React 调用这个函数取返回值作为初始 state
// 函数本身不存到 state

// 2. 函数作为 updater → 调用拿新值
setCount(c => c + 1);
// React 调用 c => c + 1 拿新 count

// 3. ⚠️ 想把函数本身作为 state 怎么办？
// 直接传 setState(myFunction) 会被当 updater 调用！
// 解决：用函数包装
setHandler(() => myFunction);  // ✅ updater 返回 myFunction
// 等价：
setHandler((prev) => myFunction);

// 4. 同样陷阱：lazy initializer
const [handler, setHandler] = useState(myFunction);
// ❌ React 把 myFunction 当 lazy initializer，调用它取返回值
// 如果 myFunction() 返回 undefined → handler = undefined！

// ✅ 正确写法
const [handler, setHandler] = useState(() => myFunction);
// React 调用 () => myFunction 取返回值 myFunction → handler = myFunction

// === 实战例子：state 存回调函数 ===
function Component({ onSomething }) {
  const [callback, setCallback] = useState(() => onSomething);   // ✅
  // 注意：现在 callback === onSomething（首次 render 的）
  // 后续 onSomething 变化时不会自动同步 —— 要手动 setCallback

  return <button onClick={() => callback()}>Run</button>;
}

// 实际工程：state 存回调函数极少见 —— 通常用 useRef 或直接传 props
// 但理解这个特殊处理对调试时帮助大

useReducer 是 useState 的超集

React 内部 useState 实现就是 useReducer 的特化—— 用一个固定的 basicStateReducer：

javascript · useState 是 useReducer 的特例 
// React 内部（简化）
function basicStateReducer(state, action) {
  return typeof action === 'function' ? action(state) : action;
  //         ↑ 检测 updater function
  //         ↑ 否则当作直接值
}

function useState(initial) {
  return useReducer(basicStateReducer, initial);
}

// 这就是为什么 setState 既能传值又能传 updater function
// useReducer 的 dispatch 永远是单参（action），但 useState 的 setState 看起来支持两种用法

// 用 useReducer 模拟 useState：
function MyCounter() {
  const [count, dispatch] = useReducer(basicStateReducer, 0);
  return (
    <>
      <button onClick={() => dispatch(c => c + 1)}>+</button>
      <button onClick={() => dispatch(0)}>Reset</button>
      <span>{count}</span>
    </>
  );
}
// 行为完全等同 useState

// 实际工程区分：
// - 简单 state（boolean / 数字 / 字符串 / 浅对象）→ useState
// - 复杂 state（多字段联动 / 多种 action 类型 / 状态机）→ useReducer
// 详见 B8 useReducer / useContext

useState 与 Concurrent rendering 的交互

React 18+ 的 Concurrent rendering 让 useState 增加了几个微妙行为（A4 详述）：

  • updater function 必须是纯函数—— 因为 render 中途打断后会重试，updater 可能跑两次。如果有副作用就 bug。StrictMode 双调暴露这种 bug
  • Lane-based 调度—— event handler 内的 setState → SyncLane（高优先级）；startTransition 内 → TransitionLane（可被打断）
  • 多次 setState 自动 batch—— React 18 之后所有上下文（包括 Promise / setTimeout）都自动 batching
  • 跨 lane 的 state 一致性—— 高优先级渲染拿"最新" state，低优先级 transition 拿"之前快照"。这就是 useDeferredValue 的实现机制

03 应用场景

场景 A · 调试"setState 似乎不起作用"

报"setState 调了但 UI 没更新"。从引擎层视角排查：

  1. 新值与旧值 Object.is 相等？ bailout 跳过。常见情况：mutate 后传同一引用 / 传相同原始值（不会 bug 只是没更新）
  2. updater function 返回了同一引用？ setUser(prev => { prev.x = 1; return prev; }) —— 返回同一引用 → bailout
  3. updater function 抛错？ render 阶段抛错让整个 render 失败，看 console
  4. Hook 调用顺序变了？ Conditional Hook → React 警告 + 数据错位

诊断工具：① console.log 当前 state ② React DevTools 看 state 变化 ③ 用 Object.is 手动比较 ④ 加一个 useEffect 监听 state 变化看是否真触发

场景 B · 写自定义 Hook 利用 setState 引用稳定性

写 useToggle，要保证返回的 toggle 函数引用稳定（用户传 useEffect 依赖不报 lint 警告）：

javascript · useToggle 引用稳定实现 
// ❌ 直接返回新箭头函数 —— 每次 render 引用不同
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = () => setValue(v => !v);  // 每次 render 创建新函数
  return [value, toggle];
}

// ✅ 用 useCallback 缓存
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = useCallback(() => setValue(v => !v), []);
  // setValue 引用稳定 → 依赖 [] 即可，永远不会变
  return [value, toggle];
}

// ✅ 更优：直接返回 setValue（dispatch 引用永远稳定）
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  // 用户用 setValue(v => !v) 来 toggle —— setValue 引用稳定，无 wrap 开销
  return [value, setValue];
}

// 设计理念：利用 React 已经保证的"setState 引用稳定"，
//          少包一层 wrapper，性能 + 简洁双赢

场景 C · 利用 Object.is bailout 实现"幂等更新"

后端 push 一份 list，可能有重复推送（同样内容）。如果每次推送都 setState 触发 render 浪费。利用 bailout：

javascript · 幂等更新 
// 错的方式 —— 每次 push 都新建 state
function ChatRoom() {
  const [messages, setMessages] = useState([]);
  useEffect(() => {
    const ws = new WebSocket(WS_URL);
    ws.onmessage = (e) => {
      const allMessages = JSON.parse(e.data);
      setMessages(allMessages);  // 即使 allMessages 内容和上次一样也重渲染
    };
    return () => ws.close();
  }, []);
  // ...
}

// 改进 —— 用 ref 比较或 selector
function ChatRoom() {
  const [messages, setMessages] = useState([]);
  const lastDataRef = useRef('');

  useEffect(() => {
    const ws = new WebSocket(WS_URL);
    ws.onmessage = (e) => {
      // 用字符串相等手动检查
      if (e.data === lastDataRef.current) return;  // 内容相同跳过
      lastDataRef.current = e.data;
      setMessages(JSON.parse(e.data));
    };
    return () => ws.close();
  }, []);
}

// 或者依赖业务逻辑：每条消息有唯一 timestamp
function ChatRoom() {
  const [state, setState] = useState({ lastTs: 0, messages: [] });
  useEffect(() => {
    const ws = new WebSocket(WS_URL);
    ws.onmessage = (e) => {
      const data = JSON.parse(e.data);
      setState(prev => {
        if (data.ts === prev.lastTs) return prev;  // ✅ 返回同引用 → bailout
        return { lastTs: data.ts, messages: data.items };
      });
    };
    return () => ws.close();
  }, []);
}
// 利用 React 的 bailout 自动跳过重复更新

setState 实战中的引用稳定性技巧

javascript · 引用稳定性的 4 个工程应用 
// === 应用 1：传给子组件不破 memo ===
const Child = React.memo(function Child({ onClick }) {
  console.log('Child render');
  return <button onClick={onClick}>Click</button>;
});

function Parent() {
  const [count, setCount] = useState(0);

  // ✅ setCount 引用稳定 —— Child memo 不会失效
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>Inc</button>
      <Child onClick={setCount} />   {/* 传 setState 本身 */}
      {/* 不要传 () => setCount(...)，那是新箭头函数每次都变 */}

}

// === 应用 2：useEffect 依赖最小化 ===
function Component({ id }) {

    // setData 不在依赖里 —— ESLint exhaustive-deps 也不要求加

  return data ? <DataView data={data} /> : null;
}

// === 应用 3：传给非 React 代码（地图 SDK / 全局事件）===
function Component() {

    const handler = (e) => setMarker(e);
    MapSDK.on('click', handler);
    return () => MapSDK.off('click', handler);
  }, []);
  // setMarker 在 handler 里被引用 —— 但因为引用稳定，闭包永远拿到正确版本

  return ...;
}

// === 应用 4：把 setState 暴露给父组件（imperative API）===
const Modal = forwardRef(function Modal(props, ref) {

  useImperativeHandle(ref, () => ({

// 19+ 用 ref as prop 替代 forwardRef，但同款逻辑：
function Modal({ ref }) {
  const [open, setOpen] = useState(false);

函数 state 的 3 个真实场景

javascript · state 存函数的实战 
// === 场景 1：state 存回调（动态绑定 handler）===
function Form() {
  const [submitHandler, setSubmitHandler] = useState(() => defaultHandler);
  //                                                ↑ 注意双层括号

  // 切换 handler
  const useHandlerA = () => setSubmitHandler(() => handlerA);
  const useHandlerB = () => setSubmitHandler(() => handlerB);

  return (
    <form onSubmit={submitHandler}>
      ...
    </form>
  );
}

// === 场景 2：lazy initializer 返回函数（极少见但合法）===
const [factory, setFactory] = useState(() => {
  // 这个函数本身是 lazy initializer
  // 它要返回一个函数作为 state
  return () => computeSomething();
});
// state 现在是一个函数，调用 factory() 拿值

// === 场景 3：通常的反例 —— 不要这样 ===
const [callback, setCallback] = useState(myCallback);
// ❌ React 把 myCallback 当 lazy initializer 调用！
// 如果 myCallback() 抛错或返回 undefined 就 bug

// 实际工程：90% 时间不需要 state 存函数
// 用 useRef 或者从 props 直接取通常更好

05 常见陷阱

⚠️ 陷阱 1：setState(myFunction) 当作 lazy initializer 被调用

想 state 存函数，setState(myFunction)—— React 把它当 updater 调用。结果 state 变成 myFunction(prevState) 的返回值，不是 myFunction 本身。

规避：用包装函数 setState(() => myFunction)。同理 lazy initializer 用法 useState(() => myFunction)。

⚠️ 陷阱 2：updater 内部 mutate 导致 bailout

setUser(prev => { prev.x = 1; return prev; })—— 返回同一引用，Object.is 等 → bailout，UI 不更新。

规避：updater 永远返回新引用 setUser(prev => ({...prev, x: 1}))。Immer 缓解。

⚠️ 陷阱 3：在 useEffect 依赖里加 setState 导致 lint 误报

老版 react-hooks/exhaustive-deps 偶尔报"setState 应在依赖里"—— 这是误报。setState 引用稳定，不需要加。

规避：升级 ESLint plugin 到最新版。如果仍报，// eslint-disable-next-line react-hooks/exhaustive-deps 加在 useEffect 行。

⚠️ 陷阱 4：依赖 bailout 跳过组件 render（错的依赖）

认为"setState 同值不重渲染" 所以可以放心写 setX(x)—— 但这只跳过本组件 render，不影响：① parent 重渲染让子重渲染（默认行为）② 副作用 useEffect（依赖 x 仍触发）。

规避：理解 bailout 的精确范围—— 仅跳过 setState 引发的本组件 render。要全局优化用 React.memo / useMemo / useCallback 组合。

⚠️ 陷阱 5：updater function 有副作用 StrictMode 暴露

setItems(prev => { logId++; return [...prev, item] })—— StrictMode 双调让 logId 跳两步。Concurrent 重试也让逻辑错乱。

规避：updater 必须是纯函数。"给定 prev 永远返回相同新值"。任何 ID 生成 / 时间戳 / log 等副作用都搬到 useEffect。
