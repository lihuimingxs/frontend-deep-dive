B16 · Server Components

React Server Components（以下简称 RSC）是 React 19 最大的心智转变—— 也是过去 11 年里 React 最有争议的特性。本章不讲 Next.js 怎么配，讲 RSC 的双图模型（Server graph + Client graph）、'use client' / 'use server' 边界、为什么 RSC 不能用 useState / useEffect、RSC payload 是什么（不是 HTML 也不是 JSON）、props 序列化要求。读完后你能解释 React 团队为什么花了 5 年才把它推到稳定版，以及它和"SSR" 的根本区别。

⚠️ RSC ≠ SSR ≠ SSG（必须区分）

SSR（Next.js Pages Router 时代）：服务器把同一份组件 render 成 HTML 字符串，客户端再 hydrate 一次（同一份组件跑两遍）。RSC（App Router 默认）：组件分两类——Server Components 只在服务器跑、永不到客户端；Client Components 仍是 SSR + hydrate。RSC 的革命点是"有些组件根本不发到客户端"—— bundle 减小 + 直连数据库变可能。

阅读时长 ~35 分钟 前置 B13 Suspense + A4 Concurrent（RSC 引擎机制） 对应 react.dev Server Components · 'use client' · 'use server' RFC #188 React Server Components

01 速览

四句话定义本章：

  1. RSC 把组件分两类：Server Components（默认，仅服务器执行，永不发到客户端 bundle）和 Client Components（顶部 'use client' 指令标记，正常的客户端组件）。Server Components 可以是 async 函数，能 await 直连数据库 / 读 env / 读文件，但不能用 useState / useEffect / 浏览器 API / event handler。
  2. 边界用指令声明：模块顶部写 'use client' 表示"本模块以下导出的组件是 Client Component"，跨边界的 props 必须可序列化（数字 / 字符串 / 数组 / 对象 / Promise / Server Reference / 不可传 class 实例 / 函数 / Symbol / Map / Set 等）。'use server' 是反向：声明一个客户端可调用的服务器函数（Server Action）。
  3. RSC 输出的不是 HTML、也不是 JSON、是 RSC payload—— 一种序列化的 React 元素树（含 Client Component 的引用 + 序列化后的 props）。客户端拿到 payload 后，渲染 Server 部分（直接读取已序列化的元素）+ 加载 Client 模块并执行（正常的 hydration）。SSR 是它的可选附加层（先把 payload + Client Component 渲染成 HTML，再水合）。
  4. 必须配套 RSC framework 才能用—— React 自身只提供原语（指令解析 / payload 协议 / Suspense 集成），不提供完整运行时。生产可用的 framework：Next.js App Router（事实标准）、Waku（轻量学习用）、Parcel / Vite 实验性插件。纯 CRA / Vite SPA 用不了 RSC。
📐 双图模型（最重要的心智）

脑子里要建立两张图：Server graph（在服务器执行的组件树，根是 page.tsx）+ Client graph（被 'use client' 切出来的子树）。Client Component 在 Server graph 中以"引用 + 序列化的 props"形式存在—— 服务器不会执行它的 render 函数，只把"用什么 Client 组件 + 什么 props" 序列化进 payload。客户端拿到后才真正 render 这个 Client Component。

02 底层逻辑

1. RSC 解决的真实问题

React 14-18 时代，工程师为了"组件 A 需要服务器数据"踩了 11 年坑：

  • 路径 1（CSR + useEffect fetch）：组件 mount → useEffect → fetch → setState → render。问题：① 瀑布式（父组件渲染完才知道要 fetch 什么）② 每次都要"转 spinner → 显示" ③ SEO 差 ④ 数据库查询要走完整 HTTP API 层
  • 路径 2（getServerSideProps / getStaticProps）：Next.js Pages Router 的方案。问题：① 只能页面级（不能给单个组件 fetch）② props drilling（数据从顶层一路往下传）③ 数据 fetch 和组件代码在不同位置
  • 路径 3（GraphQL fragment colocation）：Relay 方案，能 colocate。问题：① 需要 GraphQL（强约束）② 学习曲线陡 ③ 仍是客户端 fetch（bundle 不减）

RSC 的回答是第 4 条路径：组件本身就在服务器上 render —— 数据 fetch 和组件代码 colocate（写在一个文件里），结果是序列化好的元素树，零客户端 JS 成本。 

// 14-18 时代的 useEffect 模式（4 个问题）
function ProductDetail({ productId }) {
  const [product, setProduct] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch(`/api/products/${productId}`)
      .then(r => r.json())
      .then(setProduct)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [productId]);

  if (loading) return <Spinner />;
  if (error) return <ErrorMessage error={error} />;
  return <ProductView product={product} />;
  // ❌ 瀑布式 / SEO 差 / 多一层 HTTP API / bundle 含 fetch + 状态机代码
}

// 19 时代的 Server Component（同一段逻辑）
async function ProductDetail({ productId }) {
  const product = await db.products.findUnique({ where: { id: productId } });
  // ✅ 直连数据库 / 一次往返 / 零客户端 JS
  // ✅ loading 由父级 <Suspense> 处理（B13）
  // ✅ error 由父级 <ErrorBoundary> 处理（B15）
  return <ProductView product={product} />;
}

2. 'use client' 是"边界声明"，不是"开关"

很多人误解 'use client' = "把这个组件标记为客户端组件"。更准确的理解：它是从 Server graph 切到 Client graph 的边界。具体规则：

  1. 模块顶部写 'use client' → 该模块及其 import 的所有依赖都进入 Client graph
  2. 一个 Server Component 可以 import Client Component 并使用 → 边界是有向的（Server → Client 单向，反向用 children prop）
  3. Client Component 不能 import Server Component 用于渲染—— Server Component 不能跑在客户端
  4. 但 Client Component 可以接收 Server Component 作为 children prop—— 因为 children 此时已是序列化好的元素，不是要执行的代码 
// ❌ 错：Client Component 直接 import Server Component
'use client';
import ServerComp from './ServerComp'; // ServerComp 是 async 函数 / 用了 await db
function ClientComp() {
  return <ServerComp />; // 编译报错或运行时崩溃
}

// ✅ 对：通过 children 传递（"组合而非嵌套"模式）
// page.tsx（Server Component，default）
import ClientShell from './ClientShell';
import ServerData from './ServerData';
export default function Page() {
  return (
    <ClientShell>
      <ServerData />  {/* 在 Server graph render 完，作为序列化元素传入 */}
    </ClientShell>
  );
}

// ClientShell.tsx
'use client';
export default function ClientShell({ children }) {
  const [open, setOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setOpen(o => !o)}>Toggle</button>
      {open && children}  {/* children 已是序列化好的 React 元素 */}
    </div>
  );
}

3. props 序列化要求（最容易踩的坑）

跨 'use client' 边界传 props 时，props 必须是RSC payload 可序列化的。规则：

类型能否序列化说明
原始类型（数字 / 字符串 / 布尔 / null / undefined）✅原样发送
BigInt / Symbol（只 Symbol.for）✅BigInt 自带支持，Symbol 限注册过的
普通对象 / 数组✅递归序列化字段
Date / Map / Set✅RSC payload 协议支持，比 JSON 强
Promise✅边界两侧自动配对（B18 use API 详述）
React 元素（JSX）✅序列化为 RSC payload 节点
Server Reference（'use server' 函数）✅序列化为引用 ID，客户端调用时走 RPC
普通函数❌报错—— 函数无法跨进程
class 实例❌方法 / 原型链不可序列化
循环引用对象❌无法序列化

这条规则的根因：RSC payload 要从服务器经网络发到客户端，它和 JSON 一样是字节流—— 服务端的函数闭包、类原型、DOM 引用都没法跨进程。React 团队设计了比 JSON 强的协议（支持 Date / Map / Set / Promise），但函数和 class 实例的限制没法绕。

4. RSC payload 长什么样

RSC payload 是一种类似 JSON 但更强的格式（用编号 + 类型前缀）。一个简化的例子： 

// 假设页面是这样：
// page.tsx (Server)
async function Page() {
  const user = await db.users.findUnique({ where: { id: 1 } });
  return (
    <div>
      <h1>Welcome, {user.name}</h1>
      <ClientCounter initial={user.points} />
    </div>
  );
}

// 简化的 RSC payload（实际格式更紧凑）
0:["$","div",null,{"children":[
  ["$","h1",null,{"children":["Welcome, ","Alice"]}],
  ["$","$L1",null,{"initial":42}]   // $L1 是 ClientCounter 的引用 ID
]}]
1:I["./ClientCounter.js",["chunk-A.js"],"default"]
// L1 引用 = 客户端要从 chunk-A.js 加载 ./ClientCounter.js 的 default 导出

这个 payload 流到客户端后，React 客户端运行时：

  1. 解析 payload 中的 div 和 h1 → 直接用作 React 元素（已是结果）
  2. 遇到 $L1 引用 → 加载 ClientCounter.js → 用解析出来的 props 渲染
  3. 如果有 SSR，第 1-2 步先在服务器执行一次产生 HTML（流式 + Suspense 分块）
  4. 客户端水合时，水合的只有 Client Component 部分（Server 部分永远是结果）

5. 为什么 RSC 不能用 useState / useEffect

这是最常见的疑问。根因：

  • RSC 在服务器执行一次就结束—— 没有"重新渲染" 的概念。useState 的本质是"跨多次渲染保持值"，单次执行场景下毫无意义
  • 没有 commit 阶段—— RSC 不挂 DOM，useEffect 是"挂载后副作用"，根本没"挂载" 这件事
  • 没有事件循环—— RSC 输出 payload 后就结束了，event handler 没地方注册
  • 没有浏览器 API—— window / document / localStorage 在服务器不存在

需要状态 / 副作用 / 事件 / 浏览器 API 时，把那部分组件标 'use client' 即可。RSC 不是"更强的 Client Component"，是"不同范畴的东西"—— 一个像数据库查询，一个像 UI 状态机。

6. 'use server'：反向边界 + Server Action

'use server' 不是"这个文件是 Server Component"（默认就是 Server Component，不需要标）。它是另一个机制：声明一个客户端可以调用的服务器函数。两种用法： 

// 用法 1：模块顶部 'use server' → 整个文件导出的所有函数都是 Server Action
'use server';
import { db } from './db';

export async function createPost(formData: FormData) {
  const title = formData.get('title');
  await db.posts.create({ data: { title } });
  revalidatePath('/posts');
}

// Client Component 直接 import 调用
'use client';
import { createPost } from './actions';
export function PostForm() {
  return (
    <form action={createPost}>  {/* form action 自动调用 */}
      <input name="title" />
      <button>Create</button>
    </form>
  );
}

// 用法 2：函数体内 'use server' → 单个 inline action
async function Page() {
  async function deletePost(id: string) {
    'use server';
    await db.posts.delete({ where: { id } });
  }
  return <DeleteButton onDelete={deletePost} />;  // onDelete 是 Server Reference
}

底层机制：编译器把 'use server' 函数转成一个 Server Reference（带 ID）。Client 调用时实际是 POST 到一个 framework 提供的 endpoint，参数序列化、服务器查表执行、结果序列化回来。对开发者来说像"直接 await 服务器函数"，对运行时来说是 RPC。B17 Actions 章节会详述。

03 应用场景

场景 1：数据展示页（直连数据库 / API / 文件系统）

最经典的 RSC 用法。原本要走 BFF / API Route 的数据获取，现在 colocate 到组件里： 

// app/posts/[id]/page.tsx (Next.js App Router)
import { db } from '@/lib/db';
import { notFound } from 'next/navigation';
import Comments from './Comments'; // 'use client' 组件

export default async function PostPage({ params }: { params: { id: string } }) {
  const post = await db.posts.findUnique({
    where: { id: params.id },
    include: { author: true, tags: true }
  });
  if (!post) notFound();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>by {post.author.name}</p>
      <div dangerouslySetInnerHTML={{ __html: post.contentHtml }} />
      <Comments postId={post.id} />  {/* Client Component 处理评论交互 */}
    </article>
  );
}

// 收益：
// ✅ 零客户端 fetch 代码 / 零 loading 状态机 / 零 SEO 问题
// ✅ db 连接池只在服务器（不会泄漏到 bundle）
// ✅ 评论的交互代码隔离在 <Comments> 这个 Client Component

场景 2：流式 + Suspense 分块加载（用户感知最快首屏）

RSC 配合 Suspense 实现"有结果就先发，慢的部分稍后流过来"： 

// 慢接口（推荐 / 销量榜）不阻塞主商品信息
export default async function ProductPage({ params }) {
  const product = await db.products.findUnique({ where: { id: params.id } });
  // ↑ 主信息快，~10ms

  return (
    <div>
      <ProductInfo product={product} />  {/* 立即 render */}

      <Suspense fallback={<Skeleton />}>
        <Recommendations productId={product.id} />
        {/* ↑ async server component，慢接口（300ms+）在这里独立等待 */}
      </Suspense>

      <Suspense fallback={<Skeleton />}>
        <SalesRanking categoryId={product.categoryId} />
        {/* ↑ 另一个独立 Suspense 边界，并行加载 */}
      </Suspense>
    </div>
  );
}
// 用户体验：~50ms 看到主信息 → ~300ms 推荐进来 → ~500ms 销量榜进来
// 比传统 SSR（必须等所有数据） + 比 CSR（什么都没有就转圈）都好

场景 3：表单 mutation（Server Actions 替代 API Route）

原本要写 Express / API Route + fetch + state 机的事，现在一个 action prop 搞定： 

// app/todos/actions.ts
'use server';
import { db } from '@/lib/db';
import { revalidatePath } from 'next/cache';

export async function addTodo(formData: FormData) {
  const title = formData.get('title') as string;
  if (!title.trim()) return { error: 'Title required' };
  await db.todos.create({ data: { title } });
  revalidatePath('/todos');
}

// app/todos/page.tsx (Server Component)
import { addTodo } from './actions';
import { db } from '@/lib/db';

export default async function TodosPage() {
  const todos = await db.todos.findMany();
  return (
    <div>
      <form action={addTodo}>
        <input name="title" />
        <button>Add</button>
      </form>
      <ul>
        {todos.map(t => <li key={t.id}>{t.title}</li>)}
      </ul>
    </div>
  );
}
// 注意：纯 Server Component 也能用 form action，无需 'use client'
// 进阶（带 pending 状态 / 错误展示 / 乐观更新）→ B17 Actions

场景 4：dashboard 混合应用（多种边界粒度）

真实业务里的典型组合：外层 Server fetch 数据 + 内层 Client 处理交互 + Suspense 分块： 

// Server / Client 混合：每个块按"是否需要交互" 决定边界
async function Dashboard() {
  return (
    <Layout>
      <Header user={await getUser()} />       {/* Server，零交互 */}

      <Suspense fallback={<ChartSkeleton />}>
        <ChartsSection />                       {/* Server，慢，独立 Suspense */}
      </Suspense>

      <FilterBar>                              {/* Client，需要 useState */}
        <Suspense fallback={<TableSkeleton />}>
          <DataTable filters={DEFAULT_FILTERS} />  {/* Server，初始数据 */}
        </Suspense>
      </FilterBar>
    </Layout>
  );
}

// 'use client' 边界尽量靠近交互点：
// - FilterBar 自己是 Client，但 children <DataTable> 仍是 Server（通过 children prop 通道）

04 典型代码

例 1：完整的 RSC 项目结构（Next.js App Router）

// app/products/page.tsx — Server Component（默认）
import { db } from '@/lib/db';
import ProductList from './ProductList';
import SearchInput from './SearchInput';

export default async function ProductsPage({
  searchParams,
}: {
  searchParams: { q?: string }
}) {
  const products = await db.products.findMany({
    where: searchParams.q
      ? { name: { contains: searchParams.q } }
      : undefined,
  });

  return (
    <main>
      <h1>Products</h1>
      <SearchInput defaultValue={searchParams.q} />
      <ProductList products={products} />
    </main>
  );
}

// app/products/SearchInput.tsx
'use client';
import { useRouter } from 'next/navigation';
import { useState, useTransition } from 'react';

export default function SearchInput({ defaultValue }: { defaultValue?: string }) {
  const router = useRouter();
  const [value, setValue] = useState(defaultValue ?? '');
  const [isPending, startTransition] = useTransition();

  return (
    <input
      value={value}
      onChange={(e) => {
        setValue(e.target.value);
        startTransition(() => {
          router.push(`?q=${encodeURIComponent(e.target.value)}`);
        });
      }}
      placeholder={isPending ? 'Searching…' : 'Search'}
    />
  );
}

// app/products/ProductList.tsx — 注意：这里没有 'use client'
// 是 Server Component，但接收 Client Component 不能传给它的 props
import ProductCard from './ProductCard';
export default function ProductList({ products }) {
  return (
    <ul>
      {products.map(p => <ProductCard key={p.id} product={p} />)}
    </ul>
  );
}

// app/some-client.tsx
'use client';
import ServerData from './ServerData';  // ❌ ServerData 是 async Server Component
export default function SomeClient() {

例 3：Server Action + 乐观更新（与 B17 衔接）

// app/likes/actions.ts
'use server';
export async function toggleLike(postId: string, currentlyLiked: boolean) {
  if (currentlyLiked) {
    await db.likes.delete({ where: { postId } });
    return { liked: false };
  }
  await db.likes.create({ data: { postId } });
  return { liked: true };
}

// app/likes/LikeButton.tsx
'use client';
import { useOptimistic, useTransition } from 'react';
import { toggleLike } from './actions';

export default function LikeButton({
  postId,
  initialLiked,
}: {
  postId: string;
  initialLiked: boolean;
}) {
  const [optimisticLiked, setOptimisticLiked] = useOptimistic(initialLiked);
  const [, startTransition] = useTransition();

  return (
    <button
      onClick={() => {
        startTransition(async () => {
          setOptimisticLiked(!optimisticLiked); // 立即更新 UI
          await toggleLike(postId, optimisticLiked);
          // 服务器返回后会通过 revalidate 刷新真实状态
        });
      }}
    >
      {optimisticLiked ? '❤️' : '🤍'}
    </button>
  );
}
// useOptimistic / useActionState 详细机制 → B17

05 常见陷阱

陷阱 1：在 Server Component 里写 useState / useEffect

现象：编译报错 You're importing a component that needs useState. It only works in a Client Component but none of its parents are marked with "use client", so they're Server Components by default.

根因：Server Component 单次执行后输出 payload，没有"渲染周期" / "挂载" / "事件循环"。所有 Hook（除了 use 等少数 RSC-safe 的）都是 Client-only。

规避：把需要 state / effect 的部分提取成单独的 'use client' 组件，通过 props 接收数据。边界尽量靠近交互点—— 不要把整个页面标 'use client'，那等于退化成 SPA。

陷阱 2：跨边界传函数 prop（最常见）

现象：Functions cannot be passed directly to Client Components unless you explicitly expose it by marking it with "use server".

根因：Client Component 在浏览器执行，从 RSC payload 反序列化 props。函数闭包 / 引用无法跨进程。要么传 Server Action（编译器转 Reference ID），要么不传。 

// ❌ 错：Server Component 给 Client Component 传普通函数
async function Page() {
  const handleClick = () => console.log('clicked');  // 普通函数
  return <ClientButton onClick={handleClick} />;     // 报错
}

// ✅ 对方案 A：把回调标 'use server' 变成 Server Action
async function Page() {
  async function handleClick() {
    'use server';
    // 在服务器上执行
    await db.logs.create({ data: { event: 'click' } });
  }
  return <ClientButton onClick={handleClick} />;  // OK
}

// ✅ 对方案 B：把交互逻辑搬进 Client Component 自身
'use client';
function ClientButton() {
  return <button onClick={() => console.log('clicked')}>Click</button>;
}

陷阱 3：Client Component 里 import 含 server-only 代码的模块

现象：bundle 包含数据库连接代码，或敏感 env 泄漏到客户端，或运行时崩溃。

根因：'use client' 标的模块及其所有 import 链都进 Client bundle。如果不小心 import 了带 db / process.env.SECRET_KEY 的模块，整个 server-only 代码会被打包到客户端 JS。

规避：① 用 server-only 包做"护栏"——在 server-only 模块顶部 import 'server-only'，被 Client 链 import 时构建直接报错 ② 用 client-only 反向保护 ③ 把 server-only 工具函数和 Client utility 物理分离到不同目录（/lib/server vs /lib/client）

陷阱 4：把整个页面标 'use client' 退化为 SPA

现象：app/layout.tsx 或顶层页面顶上写 'use client' 想"简单点"，结果整个 app 进 Client graph，RSC 收益归零。

根因：'use client' 的传播是从这个模块向下—— 所有被它直接引用的组件树都被卷入 Client graph。

规避：边界往内缩，不往外扩。默认所有组件是 Server，只有真正需要 useState / 浏览器 API 的"叶子"组件才标 'use client'。判断标准：① 有用户输入交互？② 有动画 / 滚动监听？③ 用浏览器 API（localStorage / window）？三者都没有就让它留在 Server graph。

陷阱 5：误以为 RSC = SSR，HTML 已经发了所以"不需要 Suspense fallback"

现象：开发者认为 Server 部分已经渲染完，不需要包 Suspense。结果慢的查询阻塞整个页面 HTML 输出—— TTFB 很差。

根因：没用 Suspense 时，Next.js 会等所有 async server component 完成才发第一个字节。包了 Suspense 才能流式输出 —— 主体先发，慢的部分用 chunked transfer 后续注入。

规避：对慢接口必须包 Suspense—— 即使在 RSC 里也是。把"慢部分" 用 Suspense 边界切开是 RSC 性能优化的核心手段。规则：单个组件 await 超过 100ms → 应该被 Suspense 包裹。

陷阱 6：Context Provider 必须在 Client 边界内

现象：尝试在 Server Component 里 <ThemeContext.Provider value={...}>，报错或 value 不生效。

根因：Context 是跨渲染共享值的机制（B8）—— 它依赖 React 客户端 Fiber tree。Server Component 单次执行后就结束，没有 Fiber tree 概念。

规避：把 Provider 包成 Client Component，再在 Server tree 里使用： 

// theme-provider.tsx
'use client';
import { createContext, useState } from 'react';
export const ThemeContext = createContext('light');
export default function ThemeProvider({ children }) {
  const [theme] = useState('light');
  return <ThemeContext.Provider value={theme}>{children}</ThemeContext.Provider>;
}

// app/layout.tsx (Server)
import ThemeProvider from './theme-provider';
export default function Layout({ children }) {
  return (
    <html>
      <body>
        <ThemeProvider>{children}</ThemeProvider>
      </body>
    </html>
  );
}
