B11 · 自定义 Hooks + Rules of Hooks

自定义 Hooks 是 React Hooks 设计的精髓—— 把行为从组件抽离到可复用单元，比 HOC / Render Props 更扁平、类型友好（B3 已对比）。本章讲清命名约定（use* 前缀）、Rules of Hooks 的 2 条铁律（含 A3 Hook linked list 引擎根因短重述）、6 个经典模式（useFetch / useDebounce / useEventListener / useLocalStorage / usePrevious / useIsMounted），以及"何时该抽 / 不该抽" 决策框架。本章是 Part 2 Hooks 全谱的收官章—— 读完 Part 2 你具备完整 Hooks 心智 + 复用能力。

📐 Part 2 收官章

读完 B6-B11 共 6 章 Hooks 全谱后，你掌握了 React 19 的所有核心 Hooks + 复用模式。下一步进入 Part 3（B12-B15）渲染与并发，从用户视角讲 Reconciliation / Suspense / Transition / Error Boundary。

阅读时长 ~30 分钟 前置 Part 2 全部 + A3 Fiber（理解 Rules of Hooks 引擎根因） 对应 react.dev Reusing Logic with Custom Hooks · Rules of Hooks

01 速览

三句话定义本章：

  1. 自定义 Hook 是普通函数 + 内部用其他 Hooks + 名字以 use 开头。命名约定不是装饰—— ESLint react-hooks/rules-of-hooks 用前缀识别什么是 Hook 调用，从而强制 Rules of Hooks。
  2. Rules of Hooks 2 条铁律：① 只在顶层调用（不在 if / for / try / nested function 里）② 只在 React 函数 / 自定义 Hook 内调用（不在普通函数 / class / event handler 里）。引擎根因：A3 讲过 React 完全靠"调用顺序" 配对 Hook linked list 节点—— 顺序变 = 节点错位 = bug。
  3. 判断"该抽自定义 Hook 吗"：① 多组件复用相同逻辑？✅ 抽 ② 单组件但逻辑复杂（> 30 行 effect / useReducer）？✅ 抽（提高可读性）③ 单组件 + 逻辑简单？❌ 别抽（增加心智成本）④ 想抽 HOC 替代？✅ 用自定义 Hook（19 时代主流，B3）。

看一段最小代码识别自定义 Hook 三要素：

javascript · 自定义 Hook 完整骨架 
// ① 名字 use 开头（ESLint 识别）
// ② 普通函数（参数任意 / 返回值任意）
// ③ 内部可用 useState / useEffect 等其他 Hooks
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);

  // 自定义返回值（数组 / 对象 / 单值都可以）
  return [value, setValue] as const;
  //     ↑ TypeScript: as const 让返回类型推导为元组
}

// 用法：调用方完全像用内置 Hook
function Component() {
  const [isOpen, setIsOpen] = useToggle(false);

  return <button onClick={() => setIsOpen((prev) => !prev)}>
    {isOpen ? 'Open' : 'Closed'}
  </button>;
}

// 关键：自定义 Hook 不是"组件" —— 它不渲染 UI，只复用逻辑
// 每个使用 useToggle 的组件，都有独立的 isOpen state
// （不像 Context 那样共享状态）

02 底层逻辑

自定义 Hook 是什么

自定义 Hook 不是新概念—— 它就是普通 JS 函数，唯一特殊之处是：

  • 名字以 use 开头（不是技术限制，是 ESLint 识别 + 团队约定）
  • 内部可调用其他 Hooks（useState / useEffect / 其他自定义 Hook）
  • 遵守 Rules of Hooks（顶层调用、React 函数内）

关键认知：每次 Component 调用 useToggle—— 每个调用都独立工作，像内嵌 useState 一样。两个组件用 useToggle 不共享 state（不像 Context）。自定义 Hook 是"逻辑模板"，不是"共享状态"。

命名约定：use* 前缀

javascript · 命名规则 
// ✅ 命名规则
function useUserData() {}        // 推荐
function useFetch() {}            // 推荐
function useDebounce() {}         // 推荐
function useIsMounted() {}        // 推荐（即使不是 use + Verb 也合法）

// ❌ 命名错误
function fetchData() {}           // 没有 use 前缀 —— ESLint 不识别为 Hook
function getUserData() {}         // 同上
function useFn() {}               // 太抽象，可读性差
function UseFetch() {}            // 大写 = 看起来像组件
function use_fetch() {}           // snake_case 不是 React 风格

// 命名规则深度：
// 1. ESLint react-hooks/rules-of-hooks 用 use 前缀 + 小写第二字母作为识别
//    use + V (大写) → Hook
//    UseV → 组件 / 类
//    use V (小写) → 普通函数（如 useragent —— 巧合，但 ESLint 看 useV 模式判断）
// 2. 团队代码评审时 use* 一眼看出"这调用了 Hook"
// 3. 编辑器 IntelliSense 按 use 前缀分组显示

// 自定义 Hook 的"嵌套"也合法：
function useUserData(id) {
  const cache = useRef(new Map());           // 内嵌 useRef
  const [data, setData] = useState(null);    // 内嵌 useState
  const debouncedId = useDebounce(id, 500);  // 调用其他自定义 Hook

  useEffect(() => { ... }, [debouncedId]);   // 内嵌 useEffect

  return { data, refetch: () => ... };
}
// 多层嵌套调用都遵守 Rules of Hooks（每个 Hook 都在顶层调）

Rules of Hooks 2 条铁律 ⭐

🎯 Rules of Hooks 是规范级要求，不是约定

违反 Rules of Hooks 不是风格问题—— 是 React 引擎层必然出错。Hook 在不同 render 间靠"调用顺序" 配对 Fiber.memoizedState 链表节点（A3 详述）—— 顺序变化 = 节点错位 = state 数据混乱、可能崩溃、可能静默 bug。

规则 1：只在顶层调用

javascript · 规则 1 反例与修复 
// ❌ 在 if 里调
function Component({ shouldTrack }) {
  if (shouldTrack) {
    const [count, setCount] = useState(0);  // 违规
  }
  return ...;
}

// ❌ 在 for 里调
function Component({ items }) {
  for (const item of items) {
    const [name, setName] = useState(item.name);  // 违规
  }
}

// ❌ 在 try-catch 里调
function Component() {
  try {
    const data = useFetch('/api');  // 违规
  } catch (e) { ... }
}

// ❌ 在嵌套函数里调
function Component() {
  const handler = () => {
    const [count, setCount] = useState(0);  // 违规：在 event handler 里
  };
}

// ❌ 早返回后调
function Component({ user }) {
  if (!user) return <Login />;
  const data = useFetch(`/api/users/${user.id}`);  // 违规：早返回让 useFetch 调用次数不固定
  return ...;
}

// ✅ 修复：所有 hook 顶层调，把条件移到 hook 内部
function Component({ shouldTrack }) {
  // 始终调用 useState，但用 shouldTrack 决定是否同步
  const [count, setCount] = useState(0);
  // 在 effect 内做条件判断
  useEffect(() => {
    if (shouldTrack) trackEvent(count);
  }, [count, shouldTrack]);
}

function Component({ user }) {
  // 给 useFetch 传 null 让它跳过
  const data = useFetch(user ? `/api/users/${user.id}` : null);
  if (!user) return <Login />;
  return ...;
}

// 19 引入 use API（B18）—— 唯一可以条件调用的"hook 风格"调用
// const theme = use(ThemeContext);  // 可在条件分支调
// 但 use 不是 hook（不依赖 hooks linked list）

规则 2：只在 React 函数内调用

javascript · 规则 2 合法位置 
// ✅ 合法位置：
// 1. 函数组件
function Component() {
  const [count, setCount] = useState(0);  // ✅
}

// 2. 自定义 Hook
function useCount() {
  return useState(0);  // ✅
}

// ❌ 非法位置：
// 1. 普通 JS 函数
function helper() {
  const [count, setCount] = useState(0);  // ❌
}

// 2. class component
class MyComponent extends React.Component {
  render() {
    const [count, setCount] = useState(0);  // ❌
  }
}

// 3. event handler / callback
function Component() {
  const handler = () => {
    const [count, setCount] = useState(0);  // ❌
  };
}

// 4. 模块顶层
const [count, setCount] = useState(0);  // ❌（没在组件内）

// 5. 异步函数
async function loadData() {
  const data = await fetch('/api');
  const [state, setState] = useState(data);  // ❌（async 函数不是组件）
}

// 引擎根因：Hook 通过 dispatcher 全局变量找到当前 Fiber
// React 在调用组件函数前 set dispatcher，调用完后 reset
// 在非组件函数里调 useState → dispatcher 是 null/wrong → throw error

引擎根因：Hook linked list 配对（短重述链回 A3）

短重述：A3 Fiber 架构详述了 Hook linked list 的完整数据结构。本节用一段代码具象化"顺序错位" 的危害：

javascript · 顺序错位的危害实测 
function BadCounter({ shouldShow }) {
  // ❌ 条件 hook 调用
  let count, setCount;
  if (shouldShow) {
    [count, setCount] = useState(0);   // 仅 shouldShow=true 时调用
  }
  const [name, setName] = useState('Alice');

  // 第一次 render（shouldShow = true）：
  //   useState(0)      → Hook 节点 #1: { memoizedState: 0 }
  //   useState('Alice') → Hook 节点 #2: { memoizedState: 'Alice' }
  // 链表：#1 → #2
  //
  // 第二次 render（shouldShow = false）：
  //   useState('Alice') → 取链表第 1 个！但第 1 个是 number 0
  //   → 类型错位
  //   → 可能崩溃 / 静默 bug / 行为乱
  //
  // React 19 检测到 hook 数量变化会 throw：
  // "Rendered fewer hooks than expected. This may be caused by an
  //  accidental early return statement."
  //
  // 但这只是幸运—— 同样调用次数但顺序变了 检测不到
  return shouldShow ? <div>{count} {name}</div> : <div>{name}</div>;
}

// ✅ 正确写法
function GoodCounter({ shouldShow }) {
  const [count, setCount] = useState(0);     // 始终调
  const [name, setName] = useState('Alice');  // 始终调

  return shouldShow ? <div>{count} {name}</div> : <div>{name}</div>;
}

检测工具：

  • ESLint react-hooks/rules-of-hooks—— 静态分析 + lint 阶段拦截
  • React DevTools—— 运行时检测 Hook 数量变化并警告
  • React 19 错误信息—— "Rendered fewer / more hooks than expected"

务必在 CI 强制 ESLint 规则—— 不要把 Rules of Hooks 当"风格建议"，它是规范要求。

经典自定义 Hooks 6 模式

javascript · 1. useDebounce 
function useDebounce<T>(value: T, delay = 500): T {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const id = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(id);
  }, [value, delay]);

  return debouncedValue;
}

// 用法
function Search() {
  const [query, setQuery] = useState('');
  const debouncedQuery = useDebounce(query, 500);

  useEffect(() => {
    if (debouncedQuery) fetchResults(debouncedQuery);
  }, [debouncedQuery]);

  return <input value={query} onChange={e => setQuery(e.target.value)} />;
}
javascript · 2. useEventListener 
function useEventListener<K extends keyof WindowEventMap>(
  eventName: K,
  handler: (e: WindowEventMap[K]) => void,
  element: Window | Document = window,
) {
  // 用 ref 保存最新 handler，避免依赖 changes 让 effect 重跑
  const savedHandler = useRef(handler);

  useEffect(() => {
    savedHandler.current = handler;
  }, [handler]);

  useEffect(() => {
    const listener: EventListener = (e) => savedHandler.current(e as any);
    element.addEventListener(eventName, listener);
    return () => element.removeEventListener(eventName, listener);
  }, [eventName, element]);
}

// 用法
function Component() {
  const [scrollY, setScrollY] = useState(0);
  useEventListener('scroll', () => setScrollY(window.scrollY));
  return <div>Scroll Y: {scrollY}</div>;
}
javascript · 3. useLocalStorage 
function useLocalStorage<T>(key: string, initial: T) {
  // Lazy initial：仅首次 render 读 localStorage
  const [value, setValue] = useState<T>(() => {
    try {
      const stored = localStorage.getItem(key);
      return stored ? JSON.parse(stored) : initial;
    } catch {
      return initial;
    }
  });

  // 同步到 localStorage
  useEffect(() => {
    try {
      localStorage.setItem(key, JSON.stringify(value));
    } catch (e) {
      console.error('localStorage write failed', e);
    }
  }, [key, value]);

  return [value, setValue] as const;
}

// 用法
function Component() {
  const [theme, setTheme] = useLocalStorage<'light' | 'dark'>('theme', 'light');
  return (
    <button onClick={() => setTheme(t => t === 'light' ? 'dark' : 'light')}>
      Toggle theme: {theme}
    </button>
  );
}
javascript · 4. usePrevious 
function usePrevious<T>(value: T): T | undefined {
  const ref = useRef<T>();
  useEffect(() => {
    ref.current = value;
  });
  return ref.current;
}

// 用法（B10 已演示）
function Component({ count }) {
  const prevCount = usePrevious(count);
  return <div>Now: {count}, was: {prevCount ?? 'N/A'}</div>;
}
javascript · 5. useIsMounted 
function useIsMounted() {
  const ref = useRef(true);

  useEffect(() => {
    return () => { ref.current = false; };
  }, []);

  return useCallback(() => ref.current, []);
}

// 用法（避免组件卸载后 setState）
function Component() {
  const isMounted = useIsMounted();
  const [data, setData] = useState(null);

  const handleClick = async () => {
    const result = await fetch('/api').then(r => r.json());
    if (!isMounted()) return;  // 已卸载，跳过
    setData(result);
  };

  return ...;
}

// 注意：现代代码用 AbortController 更优（B7 详述）
javascript · 6. useFetch（生产推荐 TanStack Query） 
// 教学版 useFetch（生产推荐用 TanStack Query / SWR）
function useFetch<T>(url: string | null, options?: RequestInit) {
  const [data, setData] = useState<T | null>(null);
  const [error, setError] = useState<Error | null>(null);
  const [loading, setLoading] = useState(false);

  useEffect(() => {
    if (!url) return;
    const controller = new AbortController();
    setLoading(true);
    setError(null);

    fetch(url, { ...options, signal: controller.signal })
      .then(r => {
        if (!r.ok) throw new Error(`HTTP ${r.status}`);
        return r.json() as Promise<T>;
      })
      .then(d => { setData(d); setLoading(false); })
      .catch(e => {
        if (e.name !== 'AbortError') {
          setError(e);
          setLoading(false);
        }
      });

    return () => controller.abort();
  }, [url, options]);

  return { data, error, loading };
}

// 用法
function UserPanel({ userId }) {
  const { data: user, loading, error } = useFetch<User>(
    userId ? `/api/users/${userId}` : null
  );
  if (loading) return <Spinner />;
  if (error) return <ErrorMsg error={error} />;
  if (!user) return null;
  return <UserCard {...user} />;
}

// 生产推荐 TanStack Query —— 缓存 / 后台刷新 / 乐观更新等都内置
import { useQuery } from '@tanstack/react-query';
const { data, error, isLoading } = useQuery({
  queryKey: ['user', userId],
  queryFn: () => fetch(`/api/users/${userId}`).then(r => r.json()),
});

命名 vs 实现解耦

自定义 Hook 名字 和 实现 完全解耦—— 用户只关心"这个 Hook 给我什么"，不关心"它内部用什么 Hook"。

javascript · 命名 vs 实现解耦 
// 用户只关心 useUserData 的输入和输出
function UserPanel({ id }) {
  const { user, loading, refetch } = useUserData(id);
  ...
}

// 实现可以是任何方式 —— 用户不关心
// 版本 1：fetch + useState
function useUserData(id) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  useEffect(() => {
    fetch(`/api/users/${id}`).then(r => r.json()).then(setUser);
  }, [id]);
  return { user, loading, refetch: () => ... };
}

// 版本 2：用 TanStack Query 内部实现（用户代码不变）
function useUserData(id) {
  const query = useQuery({
    queryKey: ['user', id],
    queryFn: () => fetch(`/api/users/${id}`).then(r => r.json()),
  });
  return {
    user: query.data,
    loading: query.isLoading,
    refetch: query.refetch,
  };
}

// 版本 3：用 RSC（React 19）—— 但这要求 useUserData 不存在（直接在 Server Component await）
// 这种"API 抽象 + 实现可换" 是自定义 Hook 的核心价值
// 它让重构 / 性能优化 / 切换库 都不影响业务代码

何时该抽 / 不该抽自定义 Hook

情况 该抽吗
多组件复用相同逻辑（>= 2 处） ✅ 抽
单组件但逻辑复杂（> 30 行 effect / useReducer） ✅ 抽（提高可读性）
把 HOC 重构（B3 已讲） ✅ 抽（19 主流模式）
给团队 / 库分发可复用逻辑 ✅ 抽
单组件 + 逻辑简单（< 10 行） ❌ 别抽（增加心智成本）
仅是为"看起来更整洁" ❌ 别抽（无实质价值）
逻辑紧紧耦合于业务上下文 ❌ 别抽（抽出去也不能复用）

判断要点：自定义 Hook 是抽象—— 抽象有认知成本（学新名字 + 看实现 + 理解返回值含义）。仅当抽象带来的复用价值 / 可读性提升 > 认知成本 时才值得抽。

03 应用场景

场景 A · 抽业务自定义 Hook（团队共享）

团队多个组件都用"当前用户 + 权限检查" 模式。抽 useCurrentUser + usePermission：

javascript · 业务自定义 Hooks 
// useCurrentUser.ts —— 包装 Context + 错误兜底
function useCurrentUser() {
  const ctx = useContext(UserContext);
  if (!ctx) throw new Error('useCurrentUser must be used within UserProvider');
  return ctx;
}

// usePermission.ts —— 检查当前用户是否有某权限
function usePermission(permission: string) {
  const user = useCurrentUser();
  return user.permissions.includes(permission);
}

// 业务组件
function AdminPanel() {
  const canEdit = usePermission('edit_admin');
  if (!canEdit) return <NoPermission />;
  return <EditableForm />;
}

// 优势：
// 1. 业务代码自描述（usePermission('edit_admin') 比 user.permissions.includes 清晰）
// 2. 实现集中（权限逻辑改了只改一个文件）
// 3. 测试方便（mock useCurrentUser 即可）
// 4. 类型安全（permission 字符串可用 union 限制）

场景 B · 给开源库分发可复用逻辑

写 React 工具库（如 react-use / usehooks-ts），核心是把通用逻辑抽成自定义 Hook 集合：

javascript · 库式自定义 Hooks 
// 库设计要点：
// 1. 单一职责（每个 Hook 做一件事）
// 2. 类型友好（TS 泛型）
// 3. cleanup 完整（所有订阅都正确取消）
// 4. SSR 安全（不直接访问 window / document）
// 5. 文档清晰（参数 / 返回值 / 副作用）

// 例：useMediaQuery
export function useMediaQuery(query: string): boolean {
  // SSR 兜底
  const getMatch = () => {
    if (typeof window === 'undefined') return false;
    return window.matchMedia(query).matches;
  };

  const [matches, setMatches] = useState(getMatch);

  useEffect(() => {
    const mq = window.matchMedia(query);
    const handler = (e: MediaQueryListEvent) => setMatches(e.matches);

    setMatches(mq.matches);  // 初始同步
    mq.addEventListener('change', handler);
    return () => mq.removeEventListener('change', handler);
  }, [query]);

  return matches;
}

// 用户用：
const isMobile = useMediaQuery('(max-width: 640px)');
const isDarkMode = useMediaQuery('(prefers-color-scheme: dark)');

// 推荐参考：
// - https://github.com/uidotdev/usehooks
// - https://github.com/streamich/react-use
// 学他们怎么写库式 Hook 是个好教材

场景 C · 把已有 HOC 重构为自定义 Hook

老项目 withAuth(Component) HOC—— B3 已讲过怎么改 useAuth Hook。本节展示渐进迁移策略：

javascript · HOC → Hook 渐进迁移 
// 老代码（保留）
const ProtectedDashboard = withAuth(Dashboard);
const ProtectedSettings = withAuth(Settings);
// 不动 —— 仍然能用

// 新代码用 Hook
function NewFeature() {
  const user = useAuth();
  if (!user) return <Login />;
  return <Feature user={user} />;
}

// 内部实现：useAuth + withAuth 共享同一逻辑
function useAuth() {
  const user = useContext(UserContext);
  return user;
}

function withAuth(Component) {
  return function ProtectedComponent(props) {
    const user = useAuth();   // 内部用 useAuth
    if (!user) return <Login />;
    return <Component {...props} user={user} />;
  };
}

// 这样：
// 1. 老 HOC 仍 work（兼容老代码）
// 2. 新代码用 Hook（写起来扁平）
// 3. 共享一份实现（修 bug 改一个地方）
// 4. 慢慢把老 HOC 用法改成 Hook（不需要"大爆炸式" 重构）

组合多个自定义 Hook

javascript · 自定义 Hook 组合 —— 完整搜索功能 
// 组合 useDebounce + useFetch + useLocalStorage 实现完整搜索
function useSearch<T>(endpoint: string) {
  const [query, setQuery] = useLocalStorage('search:lastQuery', '');
  const debouncedQuery = useDebounce(query, 500);

  const { data, loading, error } = useFetch<T[]>(
    debouncedQuery ? `${endpoint}?q=${encodeURIComponent(debouncedQuery)}` : null
  );

  return {
    query,
    setQuery,
    results: data ?? [],
    loading,
    error,
    hasResults: !!data && data.length > 0,
  };
}

// 用法 —— 组件代码极简
function SearchPanel() {
  const { query, setQuery, results, loading } = useSearch<User>('/api/users/search');

  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      {loading && <Spinner />}
      <ul>
        {results.map(user => <li key={user.id}>{user.name}</li>)}
      </ul>
    </>
  );
}

// 关键观察：
// 1. useSearch 内部组合 3 个自定义 Hook —— 各自单一职责
// 2. 用户代码极简 —— 一行 useSearch 拿到所有需要的
// 3. 切换实现：改 useSearch 内部不影响用户
// 4. 类型安全：TS 泛型把 T 透传到 results 类型

测试自定义 Hook（@testing-library/react）

javascript · 自定义 Hook 单元测试 
import { renderHook, act } from '@testing-library/react';
import { useToggle } from './useToggle';

describe('useToggle', () => {
  it('initial value defaults to false', () => {
    const { result } = renderHook(() => useToggle());
    expect(result.current[0]).toBe(false);
  });

  it('initial value can be set', () => {
    const { result } = renderHook(() => useToggle(true));
    expect(result.current[0]).toBe(true);
  });

  it('toggles between true and false', () => {
    const { result } = renderHook(() => useToggle());
    const [, setValue] = result.current;

    act(() => setValue(prev => !prev));
    expect(result.current[0]).toBe(true);

    act(() => setValue(prev => !prev));
    expect(result.current[0]).toBe(false);
  });
});

// renderHook 是 @testing-library/react 提供
// 它创建一个测试组件 + 调用你的 Hook + 让你访问返回值
// act() 包裹任何会触发 setState 的操作 —— 确保 React 完成所有 effect
// 推荐配合 vitest 一起用（A5 生态）

05 常见陷阱

⚠️ 陷阱 1：自定义 Hook 名字不以 use 开头

function getUserData()—— ESLint 不识别为 Hook，违反 Rules of Hooks 时不报错。某天突然遇到莫名 bug 才发现。

规避：CI 强制 ESLint react-hooks/rules-of-hooks + react-hooks/exhaustive-deps。代码评审拒绝非 use 命名。

⚠️ 陷阱 2：在条件 / 循环 / 嵌套函数里调用 Hook

违反 Rules of Hooks 规则 1。React 19 通常会 throw "Rendered fewer hooks than expected"，但调用次数同但顺序变（极少见但可能）会静默 bug。

规避：始终顶层调。把条件移到 Hook 内部（传 null 等参数让 Hook 跳过）。

⚠️ 陷阱 3：自定义 Hook 返回值引用不稳定

return { data, refetch: () => ... }—— 每次返回新对象 / 新函数，调用方传 useEffect 依赖会反复触发。

规避：返回函数用 useCallback 稳定（B6 / B9）。返回对象用 useMemo（如果用户可能解构后用 useEffect 依赖）。setState dispatch 引用永远稳定不需 wrap（B6）。React Compiler 19 启用后这些 wrap 自动加。

⚠️ 陷阱 4：滥抽自定义 Hook

把const [open, setOpen] = useState(false); const toggle = () => setOpen(o => !o); 抽成 useToggle—— 用 1 次没必要，增加心智成本。

规避：Rule of three —— 至少 3 处用相同逻辑才抽。过早抽象比重复更糟。

⚠️ 陷阱 5：自定义 Hook 内调用 Hooks 时漏 deps

useEffect(() => doSomething(propValue))（漏 propValue 依赖）—— 嵌在自定义 Hook 里更难发现。

规避：自定义 Hook 同样受 ESLint react-hooks/exhaustive-deps 约束。CI 强制 + 代码评审。
