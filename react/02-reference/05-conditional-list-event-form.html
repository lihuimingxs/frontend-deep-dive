B5 · 条件 / 列表 / 事件 / Form

B1-B4 建立的心智模型（元素、JSX、组件 + Props、State）落到日常代码，最常见的 4 类模式是：条件渲染、列表渲染、事件处理、表单输入。本章把这 4 类模式讲清，包括每类模式的常见写法 + 微妙陷阱 + React 19 改动。读完后你能写中等复杂度业务代码—— 这也是 Track B Part 1 心智模型的收官章，之后进入 Part 2 Hooks 全谱。

📐 Part 1 收官

读完本章你具备 React 19 写中等业务代码的能力。Part 2（B6-B11）深入 Hooks 系统（useState 深度 / useEffect 完整生命周期 / useReducer / useMemo / useRef / 自定义 Hooks），应对更复杂场景。

阅读时长 ~25 分钟 前置 B1-B4 全部 对应 react.dev Conditional Rendering · Rendering Lists · Responding to Events

01 速览

三句话定义本章：

  1. 条件渲染有 4 种模式：&& 短路 / 三元 / 早返回 null / 提取子组件。避免 && 短路渲染数字（B2 讲过陷阱），不嵌套深条件（提取子组件）。
  2. 列表渲染必须 key：稳定 + 唯一。不要用 array index 当 key（除非 list 永不重排序），否则 reconciliation 错配会让 input 焦点 / 滚动状态跑到错误项。详细机制 A3 / B12 讲过。
  3. 合成事件 + 顶层委托：React 17+ 把所有事件委托到 root container（不是 document）—— 让多版本 React 共存可能。受控 vs 非受控表单都合法，看场景选；React 19 引入 Form Actions 取代部分手写 useState 模板。

看一段最小代码识别 4 类模式：

javascript · 4 类模式在一段代码里 
function TodoApp({ todos, currentUser }) {
  const [filter, setFilter] = useState('all');
  const filtered = filter === 'all'
    ? todos
    : todos.filter(t => filter === 'done' ? t.done : !t.done);

  // 早返回（条件渲染模式）
  if (!currentUser) return <Login />;

  return (
    <>
      {/* 三元 + 短路（条件渲染）*/}
      <h1>{currentUser.isAdmin ? 'Admin Panel' : 'Welcome'}</h1>
      {filtered.length === 0 && <p>No todos</p>}

      {/* 列表 + key（列表渲染）*/}
      <ul>
        {filtered.map(todo => (
          <TodoItem key={todo.id} todo={todo} />   {/* ← 必须 key */}
        ))}
      </ul>

      {/* 事件处理 */}
      <button onClick={() => setFilter('done')}>Done only</button>

      {/* 表单（19+ Action 风格）*/}
      <form action={createTodo}>
        <input name="title" />
        <button type="submit">Add</button>
      </form>
    </>
  );
}

条件渲染 4 种模式

javascript · 条件渲染 4 种模式各自适用 
// === 模式 1：&& 短路（最常用）===
{isLoading && <Spinner />}
{user.isAdmin && <AdminBadge />}
// 适用：单分支 + 左值是 boolean
// ⚠️ 陷阱：左值是 0 / '' / NaN 会渲染（B2 详述）

// === 模式 2：三元（双分支）===

{count > 10 ? <Warning /> : null}
// 适用：双分支 / 简单替代

// === 模式 3：早返回（推荐 - 复杂分支）===
function UserPanel({ user }) {
  if (!user) return <Login />;
  if (user.banned) return <Banned />;
  if (user.expired) return <Renew />;

// 适用：3+ 分支或前置校验
// 优势：扁平、易读、TS narrowing 友好
// ⚠️ 早返回前不能 useState 后又条件 return（破坏 Rules of Hooks）

// === 模式 4：提取子组件（最干净）===
// ❌ 嵌套深条件可读性差
function Page({ status }) {

      {status === 'error' && <ErrorBox />}
      {status === 'idle' && (
        <div>
          {/* 复杂内容 */}
        </div>
      )}
      {status === 'success' && (
        <div>
          {/* 更多复杂内容 */}
        </div>
      )}
    </div>
  );
}

// ✅ 提取子组件 + 表驱动
const STATUS_VIEWS = {
  loading: Spinner,
  error: ErrorBox,
  idle: IdleView,
  success: SuccessView,
};
function Page({ status }) {
  const View = STATUS_VIEWS[status];
  return <div><View /></div>;
}

// ✅ 或 switch 提取
function StatusView({ status }) {
  switch (status) {
    case 'loading': return <Spinner />;
    case 'error': return <ErrorBox />;
    case 'idle': return <IdleView />;
    case 'success': return <SuccessView />;
    default: return null;
  }
}

列表 + key 完整规则

列表渲染最常见的工程错误是 key 用错。A3 / B12 已详述底层 reconciliation 算法——本节聚焦"怎么用对 key"。

3 条 key 规则：

  • 必须唯一—— 同一 list 里不能重复
  • 必须稳定—— 同一逻辑项每次渲染 key 不变
  • 必须放最外层数组的元素上—— map 回调 return 的元素，不是嵌套元素
javascript · key 4 种典型用法 
// ✅ 用业务 ID（最优）
{todos.map(todo => <li key={todo.id}>{todo.text}</li>)}

// ✅ 唯一字段（如果保证不变）
{users.map(user => <UserCard key={user.email} user={user} />)}

// ✅ 数据生成时分配 UUID
const newItem = { id: crypto.randomUUID(), ...data };

// ⚠️ index（仅 list 永不重排序时）
{staticHeaders.map((h, i) => <th key={i}>{h}</th>)}
// ⚠️ 任何排序 / 过滤 / 删除 / 插入操作都让 index 失效

// ❌ Math.random()（每次渲染新 key → 重建组件 → 永远 mount unmount）
{items.map(item => <Item key={Math.random()} {...item} />)}

// 用 Fragment 包裹时也要 key（短语法不支持 key，用完整 Fragment）
{items.map(item => (
  <Fragment key={item.id}>
    <dt>{item.title}</dt>
    <dd>{item.body}</dd>
  </Fragment>
))}

// 用 key 强制重建（高级用法）
{/* 切换 userId 时强制重建 UserDetails，重置内部 state */}
<UserDetails key={userId} userId={userId} />

"不要用 index"的根因：

javascript · index key bug 演示 
// 假设 list：[{ name: 'A' }, { name: 'B' }, { name: 'C' }]
// 每个 li 内部有 input，用户在 'A' 的 input 输入 'foo'

{items.map((item, i) => (
  <li key={i}>
    <span>{item.name}</span>
    <input type="text" defaultValue="" />
  </li>
))}

// 现在用户点删除 'A' → list 变成 [{ name: 'B' }, { name: 'C' }]
// React 看 keys：原来 [0, 1, 2] 现在 [0, 1]
// 配对：
//   key 0 还在 → 复用第一个 li，但内容现在是 'B' （!）
//   key 1 还在 → 复用第二个 li，但内容现在是 'C'
//   key 2 不在 → 销毁
// 结果：① 'B' / 'C' 显示对了 ② 但 input 内容跑到 'B' 上（key 0 复用）
//
// 因为 React 用 key 配对，不是按内容判断同一项

// ✅ 用 todo.id 作 key 后：
//   key 'a' 不见了 → 销毁第一个 li（含 input）
//   key 'b' 还在 → 不动
//   key 'c' 还在 → 不动
// 结果：input 内容随 'A' 一起销毁，不串到 'B' 上

合成事件系统

React 用合成事件（SyntheticEvent）封装原生事件——所有 onClick / onChange / onKeyDown 等都是 React 自己的事件系统，不是直接绑定 DOM。

特性 合成事件 vs 原生事件
命名 camelCase（onClick / onMouseEnter），原生是小写（onclick）
事件对象 SyntheticEvent 实例，包装原生事件。访问 e.nativeEvent 可拿原生
跨浏览器一致 合成事件抹平浏览器差异（如 IE 旧 quirk）—— 现代项目其实已经不太需要
17 之前委托 所有事件委托到 document—— 多版本 React 共存时事件错乱
17+ 委托 所有事件委托到 root container（你 createRoot 的 DOM 节点）—— 多版本共存可能
preventDefault e.preventDefault()—— 同原生
stopPropagation 合成事件层的 stop—— 不影响原生 listener。如果加了 native listener，仍会跑
javascript · 事件处理常见模式 
// 基础
<button onClick={handleClick}>Click</button>
<button onClick={(e) => handleClick(e, 'extra')}>Click</button>

// e 是 SyntheticEvent
function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
  console.log(e.target.value);
  console.log(e.nativeEvent);  // 原生 InputEvent
}

// preventDefault / stopPropagation
<form onSubmit={(e) => { e.preventDefault(); doSomething(); }}>

// 阻止冒泡（仅 React 层）
<div onClick={parent}>
  <button onClick={(e) => {
    e.stopPropagation();   // parent 不会被调用
    doButton();
  }}>X</button>
</div>

// 拿原生事件做高级操作（罕见）
function handleKeyDown(e) {
  if (e.nativeEvent.isComposing) {
    // 中文输入法 composing 状态
    return;
  }
  if (e.key === 'Enter') {
    submit();
  }
}

// React 17+ 之前：委托 document，所以 document.addEventListener 监听
//                能在 React 处理之前拦截
// React 17+：委托 root，document listener 仍能拦截但顺序不同
//            这是 17 升级常见的"事件顺序变了"bug 来源

受控 vs 非受控表单

类型 受控 非受控
表单值来源 React state DOM 自身
取值 value={state} 用 ref 读 inputRef.current.value
更新 onChange => setState 不需要（用户输入直接改 DOM）
初始值 value={state} + state 初始 defaultValue
实时验证 / 派生 容易（state 同步） 困难（要 onChange + ref）
性能 每次输入都 setState → 重渲染 无重渲染（DOM 自管理）
典型场景 实时搜索 / 联动校验 / 多字段同步 简单表单 / 大量字段 / React Hook Form
javascript · 受控 vs 非受控对比 
// === 受控 ===
function ControlledForm() {
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');

  return (
    <form onSubmit={(e) => { e.preventDefault(); submit({ name, email }); }}>
      <input value={name} onChange={(e) => setName(e.target.value)} />
      <input value={email} onChange={(e) => setEmail(e.target.value)} />
      <button type="submit">Submit</button>
    </form>
  );
}

// === 非受控 ===
function UncontrolledForm() {
  const nameRef = useRef(null);
  const emailRef = useRef(null);

  const handleSubmit = (e) => {
    e.preventDefault();
    submit({
      name: nameRef.current.value,
      email: emailRef.current.value,
    });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input defaultValue="" ref={nameRef} />
      <input defaultValue="" ref={emailRef} />
      <button type="submit">Submit</button>
    </form>
  );
}

// === React Hook Form 模式（混合）===
function HookFormForm() {
  const { register, handleSubmit } = useForm();
  return (
    <form onSubmit={handleSubmit(submit)}>
      <input {...register('name')} />       {/* 内部用 ref 非受控 */}
      <input {...register('email')} />
      <button type="submit">Submit</button>
    </form>
  );
}
// React Hook Form 内部用 ref 实现非受控 → 性能极佳（不重渲染）
// 但 API 看起来像受控 → 学习曲线低
// 详见 A5 生态

React 19 Form Actions（短重述）

短重述：React 19 引入 <form action={fn}> 直接绑函数 + useActionState + useFormStatus + useOptimistic—— 减少手写 useState + try/catch + isLoading 模板。详细机制见 A4 Concurrent 和 B17 Actions（编写中）。

javascript · React 19 Form Actions 极简版

  await fetch('/api/todos', {
    method: 'POST',
    body: JSON.stringify({ title: formData.get('title') }),
  });

function TodoForm() {

      <input name="title" />

// 详见 B17 Actions

事件委托与 React 17+ 改动

React 17 之前所有事件委托到 document—— 同一页面跑多版本 React 时事件会混乱（多个 React 都监听 document，处理顺序不可控）。

React 17+ 改成委托到 root container（你 createRoot(rootEl) 的那个 DOM）—— 让多版本共存成为可能。但带来一些升级期 bug：

  • document.addEventListener 拦截顺序变化：17 之前 document listener 在 React 之前跑，17 之后可能在之后
  • ReactDOM.unmountComponentAtNode 行为微调：root 上的 listener 和组件树解耦
  • Portal 行为不变：仍委托到组件树所属的 root，不是 portal 目标

17 升级时若发现"事件顺序错乱"，先查是否有 document.addEventListener 直接监听—— 改成在 root 元素上监听通常解决。

03 应用场景

场景 A · 实时搜索 + 防抖

搜索输入实时过滤大列表。受控 input + 防抖 + transition：

javascript · 实时搜索完整实现 
function SearchableList({ items }) {
  const [query, setQuery] = useState('');
  const [isPending, startTransition] = useTransition();
  const deferredQuery = useDeferredValue(query);

  const filtered = useMemo(
    () => items.filter(item => item.name.includes(deferredQuery)),
    [items, deferredQuery]
  );

  return (
    <>
      {/* 受控 input —— 实时显示输入 */}
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="Search..."
      />

      {/* 提示推迟更新中 */}
      {query !== deferredQuery && <Spinner />}

      {/* 列表 + key（业务 ID）*/}
      <ul>
        {filtered.map(item => (
          <li key={item.id}>{item.name}</li>
        ))}
      </ul>

      {/* 条件渲染：空结果 */}
      {filtered.length === 0 && <p>No results</p>}
    </>
  );
}

// 涉及 4 类模式：
// - 受控 input（onChange + setState）
// - 列表 + key
// - 条件渲染（空状态）
// - 性能优化（useTransition / useDeferredValue / useMemo）

场景 B · 多步表单（Wizard）

注册流程 3 步：基本信息 → 详细信息 → 确认。每步是不同 form。用条件渲染 + state 管理当前步：

javascript · 多步表单 
function SignupWizard() {
  const [step, setStep] = useState(1);
  const [data, setData] = useState({
    basic: {},
    details: {},
  });

  // 早返回每步组件（条件渲染模式 3）
  if (step === 1) return <BasicStep onNext={(d) => {
    setData(prev => ({ ...prev, basic: d }));
    setStep(2);
  }} />;

  if (step === 2) return <DetailsStep
    initial={data.details}
    onBack={() => setStep(1)}
    onNext={(d) => {
      setData(prev => ({ ...prev, details: d }));
      setStep(3);
    }}
  />;

  if (step === 3) return <ConfirmStep
    data={data}
    onBack={() => setStep(2)}
    onSubmit={() => submitSignup(data)}
  />;

  return null;
}

// 复杂场景（5+ 步、可跳转、带状态机）→ 用 XState（A5 提过）
// 简单场景（< 4 步）→ useState 足够

场景 C · 用 key 重置组件状态

用户切换 tab，每个 tab 有独立的 SearchableList 但 list 类型不同。要让切换 tab 时重置 SearchableList 的内部 state（query 等）—— 用 key 强制重建：

javascript · key 强制重建 
function App() {
  const [activeTab, setActiveTab] = useState('users');

  return (
    <>
      <Tabs activeTab={activeTab} onChange={setActiveTab} />

      {/* 关键：key={activeTab} → tab 切换时 key 变 → React 销毁旧组件创建新组件 */}
      {/* 内部 useState 的 query / filter 等全部重置 */}
      <SearchableList key={activeTab} type={activeTab} />
    </>
  );
}

// 等效写法（更显式但同效果）：
{activeTab === 'users' && <SearchableList type="users" />}
{activeTab === 'posts' && <SearchableList type="posts" />}
// 这种写法用条件渲染让组件 unmount + remount —— 同样重置 state

// 但 key 写法的优势：
// 1. 简洁：一行代码搞定
// 2. 不依赖布尔条件（多个 tab 时不用写多个分支）
// 3. 重命名 tab 不影响代码结构

05 常见陷阱

⚠️ 陷阱 1：&& 短路渲染数字

{items.length && ...}—— 当 length 为 0 时渲染 '0' 文本。B2 已详述。

规避：左值 boolean 化（items.length > 0 / Boolean(items.length)）或用三元（items.length ? ... : null）。

⚠️ 陷阱 2：用 array index 当 key 在重排序场景

list 排序 / 过滤 / 删除 / 插入时，index 当 key 让 React 错配 Fiber—— input 焦点 / 滚动状态 / component state 跑到错误项。

规避：用业务 ID 作 key。仅"list 永不重排序" 时（如 static headers）才能用 index。生成数据时分配 crypto.randomUUID()。

⚠️ 陷阱 3：早返回前调 hook 又条件 return（破坏 Rules of Hooks）

if (!user) return <Login /> —— OK。但 if (!user) { useEffect(...) ; return ...} —— 破坏 Rules of Hooks（A3 详述）。

javascript · Rules of Hooks 与早返回 
// ❌ 早返回后还有 hook
function UserPanel({ userId }) {
  if (!userId) return <Login />;       // ← 早返回
  const data = useFetch(`/api/users/${userId}`);  // ❌ 条件调用 hook
  return <UserCard data={data} />;
}

// ✅ 所有 hook 在顶层调用
function UserPanel({ userId }) {
  const data = useFetch(userId ? `/api/users/${userId}` : null);
  if (!userId) return <Login />;       // ← 早返回放在 hooks 之后
  return <UserCard data={data} />;
}
⚠️ 陷阱 4：用 stopPropagation 阻不了原生 listener

合成事件的 e.stopPropagation() 只阻止合成事件层冒泡，不影响原生 document.addEventListener 注册的。混合用容易出 bug。

规避：避免混用合成事件 + 原生 listener。如必须用，e.nativeEvent.stopImmediatePropagation() 才能阻原生（17+）。

⚠️ 陷阱 5：受控 input 切到非受控（或反之）

一个 input 时而 value={x} 时而 defaultValue=""—— React 会 warn "changing an uncontrolled input to controlled"。原因是组件定义"这个 input 是哪种" 摇摆。

规避：组件初始决定受控 / 非受控，不要切换。受控 input 即使初始值是空字符串也用 value={state ?? ''} 而非 value={state}（避免传 undefined 让它退化成非受控）。
