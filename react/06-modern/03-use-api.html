P6.3 · use API + Promise 集成

use 是 React 19 引入的读异步资源 / 上下文的统一原语—— use(promise) 把"等 Promise resolve" 变成同步代码（配合 Suspense），use(Context) 是 useContext 的"更灵活版本"（可以在条件 / 循环 / 早 return 之后调用）。本章讲清两个核心用法、为什么 use 是"不是 Hook 的 Hook"（Rules of Hooks 不适用）、Promise 必须可缓存的原因，以及它和 SWR / TanStack Query / RSC 的取舍。

⚠️ use(promise) 不是替代 useEffect 的"魔法 fetcher"

use 不会帮你管理 Promise 的创建—— 它只负责读取。每次 render 必须给它同一个 Promise 引用（或可缓存的）。如果在组件内部 const promise = fetch(...)，每次 render 都创建新 Promise → 永远 pending → 死循环。Promise 必须来自父级 / 缓存层 / Server Component prop。

阅读时长 ~25 分钟 前置 P5.2 Suspense（throw promise 机制） + P3.3 useContext（对比） 对应 react.dev use RFC #229 First-class support for promises

01 速览

四句话定义本章：

  1. use(promise)：在组件 render 中读 Promise 的值。Promise pending → React 抛出 promise（P5.2 Suspense 机制）→ 最近 Suspense 边界显示 fallback。Promise resolved → 返回值。Promise rejected → 抛 error → 最近 ErrorBoundary 兜底（P5.4）。
  2. use(Context)：等价于 useContext(Context)，但可以在条件分支 / 循环 / 早 return 之后调用。这是 useContext 不能做到的（Rules of Hooks 禁止条件调用 Hook）。
  3. use 不是 Hook—— 不受 Rules of Hooks 约束。可以在 if / for / 早 return 之后调用。但仍然只能在组件函数 / Hook 函数内调用（不能在 event handler / 普通函数里）。这是 React 19 第一个"组件级原语" 的扩展，区别于 Hook 必须按调用顺序匹配 hook list（P3.1）。
  4. Promise 必须有稳定 identity—— 来自父组件 props / Server Component / 缓存层（cache() / SWR / TanStack Query）。组件内部直接 fetch() 创建 Promise 会死循环。RSC 是 use(promise) 的"原生家"—— Server Component 把 unawaited Promise 作为 prop 传给 Client Component，Client 用 use 读。

02 底层逻辑

1. use 是"挂靠 Suspense 的同步读取"

P5.2 讲过 Suspense 的引擎机制：组件 throw promise → React 捕获 → 显示 fallback → promise resolve 后重新 render。use(promise) 是把这个机制包成了一个函数调用： 

// 概念实现（简化）
function use(thenable) {
  if (thenable.status === 'fulfilled') return thenable.value;
  if (thenable.status === 'rejected') throw thenable.reason;
  if (thenable.status === 'pending') {
    throw thenable;  // ← 触发 Suspense fallback
  }
  // 首次见到 thenable —— 附加状态追踪
  thenable.status = 'pending';
  thenable.then(
    (v) => { thenable.status = 'fulfilled'; thenable.value = v; },
    (e) => { thenable.status = 'rejected'; thenable.reason = e; },
  );
  throw thenable;
}

// 用户层代码就变得"看似同步"
function Profile({ userPromise }) {
  const user = use(userPromise);  // 等待 → 拿到 user
  return <div>{user.name}</div>;
}
// React 会自动在父级最近的 Suspense 边界显示 fallback

关键洞察："等待" 这件事不在组件里发生—— 组件只做"throw 然后被 React 重 render"。这就是为什么 use(promise) 不需要 useState 持有数据，也不引入"loading 标志"。loading 状态是 Suspense 在更高层管理的。

2. 为什么 use 不是 Hook？Rules of Hooks 不适用？

P3.6 讲过 Rules of Hooks 的根因：Hook 通过 hook linked list 按调用顺序匹配（P3.1），条件调用会让顺序错乱。但 use 不依赖 hook linked list—— 它依赖：

  • thenable 自身的状态字段（status / value / reason）—— 状态在 Promise 上，不在 Fiber 上
  • Suspense 边界的重 render 机制—— React 重启组件时，use 第二次见到同一个 thenable（identity 相同），直接读 status === 'fulfilled' 返回 value

这意味着调用顺序对 use 不重要—— 第 1 次 render 在 if 分支里调用，第 2 次 render 不在了，也没问题。所以 use 可以放在条件 / 循环 / 早 return 后。 

// ✅ 合法：use 在条件里
function User({ userPromise, isVip }) {
  if (!isVip) return <BasicProfile />;  // 早 return，不调用 use

  const user = use(userPromise);  // 仅 VIP 路径调用
  return <DetailedProfile user={user} />;
}

// ❌ 同样写法，useContext 不行
function User({ isVip }) {
  if (!isVip) return <BasicProfile />;
  const ctx = useContext(SomeContext);  // 报错：Hooks 调用顺序不一致
  return ...;
}

// ✅ use(Context) 替代 useContext，合法
function User({ isVip }) {
  if (!isVip) return <BasicProfile />;
  const ctx = use(SomeContext);  // OK
  return ...;
}

3. Promise identity 稳定性是死循环的根因

use 通过Promise 引用（identity）判断"是不是同一个等待"： 

// ❌ 死循环陷阱
function User({ userId }) {
  const userPromise = fetch(`/api/users/${userId}`).then(r => r.json());
  // ↑ 每次 render 都创建新 Promise / 新 identity
  const user = use(userPromise);
  // 1. 首次 render: 新 Promise pending → throw → Suspense fallback
  // 2. Promise resolve → React 重 render
  // 3. 重 render 又创建新 Promise → pending → throw → 永远 fallback
  return <div>{user.name}</div>;
}

// ✅ 解决方案 1：Promise 来自 props（父级稳定持有）
function User({ userPromise }) {
  const user = use(userPromise);
  return <div>{user.name}</div>;
}
function Parent({ userId }) {
  // 父级用 useMemo / useState 持有
  const userPromise = useMemo(
    () => fetch(`/api/users/${userId}`).then(r => r.json()),
    [userId]
  );
  return <User userPromise={userPromise} />;
}

// ✅ 解决方案 2：用 React.cache（仅 RSC）
import { cache } from 'react';
const fetchUser = cache((id) => fetch(`/api/users/${id}`).then(r => r.json()));
async function ServerUser({ userId }) {
  const user = await fetchUser(userId); // 同 userId 自动复用
  return <div>{user.name}</div>;
}

// ✅ 解决方案 3：用 SWR / TanStack Query（最常见）
import useSWR from 'swr';
function User({ userId }) {
  const { data } = useSWR(`/api/users/${userId}`, fetcher);
  // SWR 内部缓存 + 复用 promise
  return <div>{data?.name}</div>;
}

4. RSC + use(promise) 是黄金组合

P6.1 讲过 Server Component 是 async 函数，可以 await。但有时不想 await —— 想让 Server 更快返回 + 让 Client 在 Suspense 边界显示 fallback： 

// page.tsx (Server Component)
import Comments from './Comments';

export default function PostPage({ id }) {
  // 不 await：把 unawaited Promise 作为 prop 传下去
  const commentsPromise = fetch(`/api/posts/${id}/comments`).then(r => r.json());

  return (
    <article>
      <PostBody postId={id} />  {/* 主体可以立即渲染 */}
      <Suspense fallback={<CommentsSkeleton />}>
        <Comments commentsPromise={commentsPromise} />
      </Suspense>
    </article>
  );
}

// Comments.tsx
'use client';
import { use } from 'react';
export default function Comments({ commentsPromise }) {
  const comments = use(commentsPromise);  // 等待
  return <ul>{comments.map(c => <li key={c.id}>{c.text}</li>)}</ul>;
}

// 收益：
// ✅ Server 立即返回 PostBody，commentsPromise 序列化随 RSC payload 流式发出
// ✅ Client 端 Comments 等 promise resolve 时显示 skeleton
// ✅ promise identity 来自 Server prop，天然稳定（P6.1 序列化）

5. use(Context) vs useContext：可读性差异 vs 灵活性

两者读取的值完全相同—— Context Provider 提供的值。差异：

维度useContextuse(Context)
是否 Hook是（受 Rules of Hooks 约束）否
条件 / 循环调用❌ 禁止✅ 允许
早 return 后调用❌ 禁止✅ 允许
调用位置组件 / Hook 顶层组件 / Hook 内任何位置（仍不能 event handler）
导入import { useContext } from 'react'import { use } from 'react'
性能相同相同

什么时候选 use(Context)：① 早 return 之后还需要 context ② 循环里读 context（罕见）③ 已经在用 use(promise)，统一风格。否则保持 useContext—— 它在生态中更熟悉。

6. use 不能在 event handler / 普通函数中

虽然 use 不受 Rules of Hooks 完整约束，但仍要在组件 render / Hook 内调用。原因：

  • throw promise 必须发生在 render 期间—— React 只在 render 阶段捕获 thrown promise 用作 Suspense
  • event handler 不能 throw promise—— React 调度系统对事件不做 Suspense 处理

要在事件中读 promise → 用 await + state 更新（startTransition 内），或用 SWR / TanStack 这类有 imperative API 的库。

03 应用场景

场景 1：RSC 流式 + Client 端 use(promise)（最经典）

Server 立即返回 + Client 渐进显示。这是 React 19 取代"useEffect 数据获取" 的官方推荐路径。 

// app/dashboard/page.tsx (Server)
import { Suspense } from 'react';
import RecommendationsClient from './RecommendationsClient';
import StatsClient from './StatsClient';

export default function Dashboard() {
  // 都不 await —— 让两个慢请求并行 + 各自 Suspense 边界
  const recsPromise = fetch('/api/recommendations').then(r => r.json());
  const statsPromise = fetch('/api/stats').then(r => r.json());

  return (
    <>
      <h1>Dashboard</h1>
      <Suspense fallback={<Skeleton h={200} />}>
        <RecommendationsClient promise={recsPromise} />
      </Suspense>
      <Suspense fallback={<Skeleton h={300} />}>
        <StatsClient promise={statsPromise} />
      </Suspense>
    </>
  );
}

// RecommendationsClient.tsx
'use client';
import { use } from 'react';
export default function RecommendationsClient({ promise }) {
  const recs = use(promise);
  return <ul>{recs.map(r => <li key={r.id}>{r.title}</li>)}</ul>;
}

场景 2：与 Context 配合，灵活早 return

'use client';
import { use } from 'react';
import { ThemeContext } from './theme';
import { UserContext } from './user';

function Header({ minimal }) {
  // 早 return：未登录用户用极简 header
  if (!minimal) {
    return <MinimalHeader />;
  }
  // 早 return 之后还可以读 context（useContext 不允许）
  const theme = use(ThemeContext);
  const user = use(UserContext);
  return (
    <header data-theme={theme}>
      <Avatar user={user} />
    </header>
  );
}

场景 3：仍用 SWR / TanStack Query 的客户端 fetching（最现实）

大多数中后台 / dashboard 应用没用 RSC（仍是 SPA / Vite + React Router）。在这种场景：

  • 简单数据 fetching：仍用 SWR / TanStack Query —— 它们内置缓存 / revalidate / dedupe / mutate
  • 但可以用 use(promise) + 库的 Promise：TanStack Query v5 的 queryClient.ensureQueryData() 返回 Promise，可传给 use 
// 与 TanStack Query 配合
import { useQueryClient } from '@tanstack/react-query';
import { use, Suspense } from 'react';

function User({ userId }) {
  const queryClient = useQueryClient();
  // queryClient 内部缓存：同 userId 返回同一 promise（identity 稳定）
  const userPromise = queryClient.ensureQueryData({
    queryKey: ['user', userId],
    queryFn: () => fetch(`/api/users/${userId}`).then(r => r.json()),
  });
  const user = use(userPromise);
  return <div>{user.name}</div>;
}

// 使用时
<Suspense fallback={<Skeleton />}>
  <User userId={42} />
</Suspense>

场景 4：渐进式接入：useEffect → use 的迁移

// 老世界
function Profile({ userId }) {
  const [user, setUser] = useState(null);
  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(setUser);
  }, [userId]);
  if (!user) return <Spinner />;
  return <div>{user.name}</div>;
}

// 新世界（CSR / 没 RSC 的场景，用缓存层包装）
const userCache = new Map();
function getUser(id) {
  if (!userCache.has(id)) {
    userCache.set(id, fetch(`/api/users/${id}`).then(r => r.json()));
  }
  return userCache.get(id);
}

function Profile({ userId }) {
  const user = use(getUser(userId));  // 缓存保证 identity 稳定
  return <div>{user.name}</div>;
}

// 使用
<Suspense fallback={<Spinner />}>
  <ErrorBoundary fallback={<Error />}>
    <Profile userId={42} />
  </ErrorBoundary>
</Suspense>
// 自己手写缓存可行但容易踩坑，生产推荐 SWR / TanStack Query / RSC

04 典型代码

例 1：use(promise) + ErrorBoundary + Suspense 三件套

'use client';
import { use, Suspense } from 'react';
import { ErrorBoundary } from 'react-error-boundary';

function PostBody({ postPromise }) {
  const post = use(postPromise);  // pending → throw → Suspense
                                   // rejected → throw error → ErrorBoundary
  return <article>{post.content}</article>;
}

export default function Post({ postPromise }) {
  return (
    <ErrorBoundary fallback={<ErrorPanel />}>
      <Suspense fallback={<ArticleSkeleton />}>
        <PostBody postPromise={postPromise} />
      </Suspense>
    </ErrorBoundary>
  );
}

// 三个状态机用三层组件管理：
// - 数据未到 → Suspense fallback
// - 数据失败 → ErrorBoundary fallback
// - 数据成功 → 真实 UI

例 2：use 与传统 Hook 的对比

// 传统 Hook 受 Rules of Hooks 约束
function Old({ items, query }) {
  const [filtered, setFiltered] = useState(items);
  // ❌ 不能在 if / 循环里调用 useState / useEffect
  if (!query) return <Empty />;
  // ↑ 早 return 之前必须把所有 Hook 调用完
  useEffect(() => { ... }, [query]);
  return ...;
}

// use 没有这个约束
function New({ items, query, themePromise }) {
  if (!query) return <Empty />;  // 早 return ok

  const theme = use(ThemeContext);  // ✅ 早 return 后调用
  const themeData = use(themePromise);  // ✅ 顺序无关

  // 但 useState / useEffect 仍需在顶层！use 是个例
  return ...;
}

例 3：use(Context) 在 utility 函数里（"共享 logic"）

// 自定义 Hook 中混用 use 和 useState
import { use, useState } from 'react';
import { LocaleContext } from './locale';

function useFormatted(value, options) {
  const locale = use(LocaleContext);
  // 如果 options 没传，提前 return
  if (!options) return String(value);
  // ↑ 早 return 后还可以继续用 useState / useEffect 吗？
  //    答：不可以。useState 仍受 Rules of Hooks 约束！
  //    use 是唯一例外。
  const [cached] = useState(() => new Intl.NumberFormat(locale, options));
  // ❌ 这样写在早 return 之后调用 useState，违反 Rules of Hooks
  return cached.format(value);
}

// 正确写法：useState 永远在顶层，use 可以在条件里
function useFormatted(value, options) {
  const [cache] = useState(() => new Map());  // ✅ 顶层
  const locale = use(LocaleContext);  // ✅ 顶层（也可在条件里）
  if (!options) return String(value);
  let formatter = cache.get(`${locale}|${JSON.stringify(options)}`);
  if (!formatter) {
    formatter = new Intl.NumberFormat(locale, options);
    cache.set(`${locale}|${JSON.stringify(options)}`, formatter);
  }
  return formatter.format(value);
}

05 常见陷阱

陷阱 1：组件内创建 Promise → 死循环

现象：页面永远显示 Suspense fallback，DevTools 显示组件不停 mount / unmount。

根因：每次 render 创建新 Promise → 新 identity → use 永远见到 pending 状态 → throw → React 重 render → 又创建新 Promise → …

规避：① Promise 来自 props（父级稳定持有）② 用 cache()（仅 RSC）③ 用 SWR / TanStack Query / 自己实现 Map 缓存 ④ 用 useMemo 缓存 promise（少见，因为 useMemo 不保证 stable，仅性能优化）

陷阱 2：把 use 当成 useEffect 的"异步版"

现象：在组件里写 const data = use(fetch(...)) 想"更优雅地获取数据"，结果踩陷阱 1 + 没有 cleanup / cancel 能力。

根因：use(promise) 只读，不管理 Promise 生命周期。useEffect 还有 cleanup 函数，可以 abort fetch / 清理订阅。use 没有等价物。

规避：要管理生命周期 / abort / 防抖等，仍用 useEffect 或专业库。use 只适合"有外部已经持有的 Promise，我只想读它" 的场景。

陷阱 3：在 event handler 里调用 use

现象：onClick 里写 const data = use(promise) 报错 use cannot be called outside of a Component or Hook。

根因：use 依赖 Suspense 的"throw → 捕获 → 重 render" 机制，该机制只在 render 阶段工作。事件处理是 React 在 commit 阶段后调度的，没有 Suspense 边界处理。

规避：事件中读 Promise → 用 await： 

// ❌ 错
<button onClick={() => {
  const result = use(somePromise);  // 报错
}} />

// ✅ 对：用 await + 把结果存 state
const [result, setResult] = useState();
<button onClick={async () => {
  const r = await somePromise;
  setResult(r);
}} />

陷阱 4：不用 ErrorBoundary，用户看到白屏

现象：Promise reject 后整个组件树崩溃，整页白屏。

根因：use 在 promise rejected 时会 throw error。如果没 ErrorBoundary 兜底，错误冒泡到根组件，React 卸载整个 tree（P5.4）。

规避：每个 use(promise) 都应该被 ErrorBoundary + Suspense 双重包裹。Next.js App Router 的 error.tsx 文件约定可以自动做这件事。

陷阱 5：误以为 use(Context) 是性能优化

现象：把 useContext 全替换为 use(Context) 想"提性能"。

根因：两者性能完全相同—— 都依赖同一套 Context 订阅机制。Context value 变 → 所有读它的组件重 render。use(Context) 仅在位置灵活性上比 useContext 强，不是性能优化。

规避：性能问题用 selector 模式（P3.3）/ Zustand / Jotai 解决。不要为了"新" 而替换 useContext。

陷阱 6：与 SWR / TanStack 重复 fetch

现象：把 use(fetch(...)) 和 useSWR(...) 混用，同一份数据请求两次。

根因：use 没有缓存层，SWR / TanStack 有自己的缓存。两套同时跑会双倍请求。

规避：项目要么统一用 use(promise) + cache 层（RSC 项目），要么统一用 SWR / TanStack Query 的 hook（CSR 项目）。混用时通过 queryClient.ensureQueryData() 桥接（场景 3）。
