Registry 系统 + 私有 registry

Registry 是 shadcn 模式真正的护城河。这一章把 registry-item.json schema、1300+ blocks 系统、自建私有 registry（团队 design system 工具链）、monorepo 共享 design system 的两种模式（包模式 vs 复制模式）讲透。读完能搭团队级 design system 工具链。

阅读时长 ~30 分钟 前置 P5.1 CLI 用法 主要来源 ui.shadcn.com/docs/registry + 2026-02 changelog blocks + 真实生产案例

01 速览

  1. Registry = "组件源码 + 元数据"的远程仓库。默认 registry：ui.shadcn.com（含 50+ 组件 + 1300+ blocks）。私有 registry：自建仓库 / GitHub raw / Vercel / CF Pages / S3 等任何静态文件 server。
  2. Registry 数据格式：registry-item.json。核心字段：name（组件名）/ type（registry:ui / registry:block / registry:hook 等）/ files（文件列表，含 path + content + target）/ dependencies（npm 包）/ devDependencies（dev 包）/ registryDependencies（其他 shadcn 组件依赖）/ tailwind（额外 Tailwind 配置）/ cssVars（额外 CSS variables）/ docs（说明）。
  3. Blocks（v4 + 2026-02 大扩张）：1300+ blocks，比组件大、比 page 小。例：dashboard sidebar / login form / multi-step form / pricing table。Radix 和 Base UI 双变体；CLI 根据 components.json engine 拉对应。
  4. 自建私有 registry：shadcn build 把本地 components/registry/ 编译为 JSON。部署到任何静态文件 server（Vercel / CF Pages / S3）。用户在 components.json 设 registry URL。这是团队私有组件库分发新模式，替代 npm pack。
  5. monorepo 共享 design system 两种模式：包模式（design-system 包，import @myorg/design-system/button，一处升级）/ 复制模式（私有 registry，每个 app shadcn add 自己复制源码，独立演化）。两种模式各有取舍。
  6. Theme 分发：完整 design token（CSS variables）作为 registry item。用户 shadcn add @theme/dark 一次安装整套主题。
  7. Registry 与 npm package 的对比：npm 是"install + 黑盒"，registry 是"copy 进项目 + 完全可见"。两者各有适用场景，不是替代关系。
  8. 2026 Registry 标准化：1300+ blocks + 双引擎让 registry 成为 React 组件库分发新事实标准。第三方 registry（v0.dev / Aceternity UI 等）也都基于 shadcn registry schema。
  9. monorepo 包模式 vs 复制模式判断：包模式适合"团队规模大 + 严格设计系统 + 一处改全局生效"；复制模式适合"app 间需要独立演化 + 不想被一个 app 升级影响其他"。
  10. 本章是"团队级 design system 工具链"的核心。如果只用 ui.shadcn.com 默认 registry 加几个组件，没用到 registry 一半的价值。

02 底层逻辑

Registry 概念

Registry = "组件源码 + 元数据"的远程仓库。CLI 通过 HTTP 拉取 JSON 文件、按文件描述写组件源码到用户项目。

默认 registry：https://ui.shadcn.com/r/{style}/{name}.json。例如：

  • /r/styles/default/button.json
  • /r/styles/new-york/dialog.json
  • /r/styles/default/blocks/dashboard-01.json

自定义 registry URL：components.json 配置 registries 字段（详见 P5.1）。

registry-item.json schema 完整

json 
{
  "$schema": "https://ui.shadcn.com/schema/registry-item.json",
  "name": "button",
  "type": "registry:ui",  // registry:ui | registry:block | registry:hook | registry:lib | registry:theme

  "title": "Button",
  "description": "Displays a button or a component that looks like a button.",

  // npm 依赖
  "dependencies": [
    "@radix-ui/react-slot",
    "class-variance-authority",
    "clsx",
    "tailwind-merge"
  ],
  "devDependencies": [],

  // shadcn 组件依赖（CLI 自动 add 它们）
  "registryDependencies": [],

  // 文件列表
  "files": [
    {
      "path": "components/ui/button.tsx",
      "content": "import * as React from 'react';\nimport { Slot } from '@radix-ui/react-slot';\n...",
      "type": "registry:ui",
      "target": "components/ui/button.tsx"  // 实际写入路径（可与 path 不同）
    }
  ],

  // 额外 Tailwind 配置（合并到用户 tailwind.config）
  "tailwind": {
    "config": {
      "theme": {
        "extend": {
          "keyframes": {
            "accordion-down": { /* ... */ }
          }
        }
      }
    }
  },

  // 额外 CSS variables（合并到用户 globals.css）
  "cssVars": {
    "light": {
      "primary": "240 5.9% 10%"
    },
    "dark": {
      "primary": "0 0% 98%"
    }
  },

  // 说明文档（可选）
  "docs": "https://ui.shadcn.com/docs/components/button"
}

关键字段说明：

type（决定 CLI 处理方式）：

  • registry:ui：单个 UI 组件（Button / Card 等）
  • registry:block：组合 block（dashboard / login form 等）
  • registry:hook：React hook（useDebounce 等）
  • registry:lib：utility 函数（cn / format-date 等）
  • registry:theme：完整主题（多个 cssVars）

files（文件列表）：

  • path：文件相对路径（registry 内）
  • content：文件内容（字符串）
  • target：实际写入用户项目的路径（CLI 用 aliases 替换）
  • type：文件类型（决定写入哪个目录）

registryDependencies：CLI 自动递归 add 这些。例如 dialog 依赖 button → add dialog 时自动加 button。

tailwind / cssVars：CLI 自动合并到用户 tailwind.config / globals.css。例如 accordion 需要 keyframes，合并后用户的 tailwind.config 就有了。

1300+ Blocks（v4 + 2026-02 扩张）

Blocks 是比组件大、比 page 小的组合。例子：

  • dashboard-01：完整 dashboard 模板（侧栏 + main + 数据卡片 + 图表）
  • login-01：登录页（含表单 + OAuth 按钮 + 右侧广告位）
  • signup-01：注册页
  • sidebar-01 ~ 16：16 种侧栏布局变体
  • multi-step-form-01：多步表单
  • pricing-table-01：定价表
  • 更多...

2026-02 起每个 block 同时有 Radix 和 Base UI 两个变体。CLI 根据 components.json engine 拉对应。

Blocks vs 单组件的区别：

维度单组件Block
粒度小（Button / Card）大（dashboard / login form）
用途building block页面级模板
add 命令shadcn add buttonshadcn add @blocks/dashboard-01
文件数1-25-20+
定制方式整个组件改挑部分文件改

自建私有 registry · 团队 design system 工具链

团队私有 design system 想分发给多个 app。传统方式：

  • npm package（@myorg/ui）：黑盒、升级破坏 / 不能改
  • git submodule：复杂、易出 bug
  • monorepo workspace：可以但不灵活（详见下文）

shadcn registry 方式：

  1. 建一个"design system 项目"（含 components/registry/ 目录）
  2. 跑 shadcn build 编译为 JSON（输出到 public/r/）
  3. 部署到 Vercel / CF Pages / GitHub Pages 等
  4. 用户项目 components.json 配自定义 registry URL
  5. 用户 shadcn add @myorg/button 拉私有组件源码

优势：

  • 组件源码进用户项目（完全可见 / 可改）
  • 不是 npm 包不会被 npm update 破坏
  • 支持 auth（registry URL header 加 Authorization）
  • 与 ui.shadcn.com 默认 registry 同等模式（团队学习成本低）

自建私有 registry · 完整步骤

1. 建 registry 项目：

bash 
# 初始化（基于 Next.js 模板）
npx create-next-app@latest my-registry
cd my-registry
npx shadcn@latest init  # 也起 shadcn

# 建 registry 目录
mkdir -p components/registry/components
mkdir -p public/r

2. 写 registry source：

text 
my-registry/
├── components/registry/
│   ├── components/
│   │   ├── payment-form.tsx
│   │   ├── pricing-card.tsx
│   │   └── header.tsx
│   ├── hooks/
│   │   └── use-debounce.ts
│   └── lib/
│       └── format-currency.ts
├── registry.json  # 描述所有 items
└── ...

3. 写 registry.json（root 配置）：

json 
{
  "$schema": "https://ui.shadcn.com/schema/registry.json",
  "name": "@myorg",
  "homepage": "https://shadcn.myorg.com",
  "items": [
    {
      "name": "payment-form",
      "type": "registry:block",
      "title": "Payment Form",
      "description": "Stripe-integrated payment form",
      "registryDependencies": ["button", "input", "label", "form"],
      "files": [
        {
          "path": "components/registry/components/payment-form.tsx",
          "type": "registry:block"
        }
      ],
      "dependencies": ["@stripe/stripe-js"]
    },
    {
      "name": "use-debounce",
      "type": "registry:hook",
      "files": [
        {
          "path": "components/registry/hooks/use-debounce.ts",
          "type": "registry:hook"
        }
      ]
    }
  ]
}

4. Build registry：

bash 
npx shadcn@latest build

# 编译 registry.json + source 文件 → public/r/{name}.json
# 输出：
# public/r/payment-form.json
# public/r/use-debounce.json
# public/r/index.json（registry 索引）

5. 部署：

bash 
# 部署到 Vercel
vercel deploy
# 你的 registry URL：https://my-registry.vercel.app/r/{name}.json

6. 用户项目配置：

json 
// components.json
{
  "registries": {
    "@myorg": {
      "url": "https://my-registry.vercel.app/r/{name}.json"
    }
  }
}

7. 用户 add 私有组件：

bash 
npx shadcn@latest add @myorg/payment-form

# CLI 从 https://my-registry.vercel.app/r/payment-form.json 拉
# 自动 add 依赖（registryDependencies）
# 自动装 npm 依赖（@stripe/stripe-js）
# 写文件到 components/ 目录

带 auth 的私有 registry

商业项目不希望 registry 公开。加 auth：

json 
// 用户 components.json
{
  "registries": {
    "@myorg": {
      "url": "https://my-registry.vercel.app/r/{name}.json",
      "headers": {
        "Authorization": "Bearer ${MYORG_REGISTRY_TOKEN}"
      }
    }
  }
}

用户 .env：

bash 
MYORG_REGISTRY_TOKEN=secret_token_here

registry server 验证 token（用 Vercel Edge Function 或类似）。

monorepo 共享 design system · 两种模式

团队 monorepo 多 app 共享 design system，两种模式：

模式 A · 包模式（@myorg/design-system）

text 
monorepo/
├── apps/
│   ├── web/        # Next.js
│   ├── admin/      # Vite + React
│   └── docs/       # Astro
├── packages/
│   └── design-system/
│       ├── src/
│       │   ├── components/
│       │   │   ├── button.tsx
│       │   │   ├── card.tsx
│       │   │   └── dialog.tsx
│       │   ├── lib/utils.ts
│       │   └── index.ts
│       ├── package.json
│       └── tsconfig.json
├── package.json
└── pnpm-workspace.yaml

每个 app import：

tsx 
// apps/web/app/page.tsx
import { Button } from '@myorg/design-system/button';
import { Card } from '@myorg/design-system/card';

优势：一处升级所有 app 自动跟随。劣势：app 不能独立改样式。

模式 B · 复制模式（私有 registry）

text 
monorepo/
├── apps/
│   ├── web/        # 自己 components/ui/button.tsx
│   ├── admin/      # 自己 components/ui/button.tsx
│   └── docs/       # 自己 components/ui/button.tsx
├── packages/
│   └── design-system-registry/
│       ├── components/registry/
│       └── public/r/  # build 后输出

每个 app shadcn add（独立）：

bash 
# apps/web 配 components.json registry URL
# 跑：
cd apps/web
npx shadcn@latest add @myorg/button
# 组件源码进 apps/web/components/ui/button.tsx

# apps/admin 同样
cd ../admin
npx shadcn@latest add @myorg/button

优势：app 间独立演化（改 web 的 button 不影响 admin）。劣势：升级 design system 需要每个 app 重新 add。

选哪个：

  • 团队规模 5-20 人 + 严格设计统一 → 包模式
  • 团队 50+ 人 / 多产品线 / 不同 app 设计需求差异大 → 复制模式
  • 极小团队（1-3 人）→ 直接 ui.shadcn.com 默认 registry，不必自建

Theme 作为 registry item

完整 design token（颜色 / spacing / radius / fonts）可作为 registry item 分发：

json 
{
  "name": "theme-acme",
  "type": "registry:theme",
  "cssVars": {
    "light": {
      "background": "0 0% 100%",
      "foreground": "240 10% 3.9%",
      "primary": "262 83% 58%",
      "primary-foreground": "0 0% 100%",
      ...
    },
    "dark": {
      "background": "240 10% 3.9%",
      "foreground": "0 0% 98%",
      "primary": "263 70% 60%",
      ...
    }
  }
}

用户：

bash 
npx shadcn@latest add @myorg/theme-acme
# CLI 自动合并 cssVars 到 globals.css

这让"切换主题"成为一个 CLI 命令的事。多品牌 white-label 用这种方式分发主题。

03 应用场景

场景 1 · 5-20 人团队建私有 registry

场景：公司有 design system 团队，几个 app 共享 button / card / form 等。

推荐流程：

  1. 建 my-registry 仓库（next.js + shadcn 项目）
  2. 把内部组件 + theme 写到 components/registry/
  3. shadcn build → 部署到 Vercel
  4. 各 app 配 components.json 加 registry URL
  5. 各 app shadcn add @myorg/button 拉源码
  6. 升级时：改 my-registry 仓库 → rebuild → 各 app 主动 shadcn add 同步（或 shadcn diff 看改动）

场景 2 · 大型企业 monorepo（50+ 人）

建议复制模式。理由：

  • 不同产品线需求差异大
  • 不同 app 独立演化（不希望一处改影响所有）
  • monorepo workspace 管 registry source；apps 复制源码

场景 3 · 用 v0.dev 输出加到自定义 registry

v0.dev 是 Vercel AI 生成 shadcn 代码的工具。生成代码可以加到自己 registry：

  1. v0.dev 生成 React 组件（基于 shadcn）
  2. copy 代码到 my-registry/components/registry/
  3. 更新 registry.json
  4. shadcn build + 部署
  5. 团队 add @myorg/{name} 用

场景 4 · 多品牌 theme 分发

SaaS 平台支持多客户 white-label，每客户自己的 theme：

  1. 建 themes registry（每个客户一个 theme item）
  2. 用户 (动态) shadcn add @themes/customer-acme
  3. theme 的 CSS variables 进 globals.css
  4. 自动应用

代码 4 · 自建带 auth 的 registry server

tsx 
// 用 Next.js Route Handler 提供 registry 端点
// app/r/[name]/route.ts
import { readFile } from 'node:fs/promises';
import { NextRequest, NextResponse } from 'next/server';

export async function GET(req: NextRequest, { params }: { params: { name: string } }) {
  // 验证 token
  const authHeader = req.headers.get('authorization');
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return new NextResponse('Unauthorized', { status: 401 });
  }
  const token = authHeader.slice(7);
  if (!await verifyToken(token)) {
    return new NextResponse('Forbidden', { status: 403 });
  }

  // 返回 registry item
  const file = `public/r/${params.name}.json`;
  const content = await readFile(file, 'utf-8');
  return NextResponse.json(JSON.parse(content));
}

05 常见陷阱

  • 没用 shadcn build 直接 commit JSON —— 手写 JSON 易错。用 shadcn build 从 source 生成
  • registryDependencies 写错 —— add 时 CLI 找不到依赖组件。仔细列依赖关系
  • CSS variables 命名冲突 —— 多个 theme item 都定义 --primary，最后 add 的覆盖。分 namespace（--theme-acme-primary）
  • 私有 registry 没 auth —— 公开访问、商业组件被偷。Vercel Edge Function / 其他 server 加 token 校验
  • monorepo 包模式但 app 想 fork 一个组件 —— 复制源码到 app 本地、改 import 路径。但要记得这个组件不再跟随 design-system 升级
  • monorepo 复制模式但 app 间样式飘移 —— 升级时各 app 不主动 sync，时间长了各自演化。建 CI 检查
  • 1300+ blocks 全部 add 到项目 —— 不需要。按需 add 实际用的（每个 block 几百行代码）
  • 用 v0.dev 生成代码不放进自己 registry —— 浪费"团队复用"机会。每次 v0 输出加到 my-registry 让团队都能用
  • Theme registry item 与组件 cssVars 冲突 —— 组件 cssVars 是组件需要的（如 accordion keyframes），theme cssVars 是主题色。两者应该不冲突；冲突说明 theme 在覆盖组件特殊变量
  • 用户 components.json registries URL 写错 —— CLI 拉失败报 404。检查 URL 模板（{name} 占位）
  • shadcn build 在 Windows 路径错 —— path 用 forward slash（registry 内）；CLI 在 Windows 自动转
  • 把私有 registry 当 npm 替代品 —— 不一样。registry 是"copy 进项目"，npm 是"install + 链接"。复杂业务逻辑（含可执行代码 + 副作用）仍 npm 包更合适
