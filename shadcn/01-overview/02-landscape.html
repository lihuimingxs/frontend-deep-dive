同代人横评 + 设计哲学

2026 年的 React UI 库格局可以分三个阵营：Headless（Radix / Headless UI / Ark UI / Park UI）/ Styled（Mantine / MUI / NextUI / Chakra）/ Hybrid（shadcn / Catalyst）。这一章把每个阵营代表横评，再讲 shadcn + Radix 的 4 个核心设计赌注。读完能回答"我们项目应该选哪个 React UI 方案"。

阅读时长 ~30 分钟 前置 P1.1 项目史 + 用过任何 React UI 库 主要来源 各 UI 库官方 + 2025-2026 npm 下载量 / GitHub stars + 公开调研

01 速览

  1. 2026 React UI 库分三阵营：Headless（提供行为 + a11y、不提供样式）/ Styled（自带样式 + 自有 design system）/ Hybrid（headless + 推荐样式工具链）。
  2. Headless 阵营：Radix Primitives（事实标准、shadcn 底层）/ Headless UI（Tailwind Labs、被 Radix 反超）/ Ark UI（Chakra 团队、基于 Zag.js 状态机、多框架）/ Park UI（Ark UI + Panda CSS styled wrapper）。
  3. Styled 阵营：Mantine（独立 styling 系统、组件丰富）/ MUI（Material Design、企业大量存量）/ NextUI（基于 Tailwind v3 但 v4 兼容滞后）/ Chakra UI（v3 改造为 styled-system）。
  4. Hybrid 阵营：shadcn/ui（事实标准 60%+）/ Catalyst（Tailwind UI 商业 React kit）/ Tailwind UI（HTML 模板，不是组件库）。
  5. shadcn + Radix 的 4 个核心设计赌注：1) Headless + 标准 a11y（Radix 实现 WAI-ARIA APG，不发明）/ 2) Copy-paste 不 npm install（组件源码进项目）/ 3) CVA + cn 标准模式（变体管理 + 类名冲突解决）/ 4) 双引擎可选（2026 起 Radix 或 Base UI）。
  6. 2026 实战分布：约 60% 新 React 项目用 shadcn/ui；15% 用 MUI / Mantine / Chakra（自有 design system）；10% 用 Ark UI / NextUI 等其他 headless；15% 自建组件库（基于 Radix 或自己写）。
  7. "何时不用 shadcn/ui"：已有 Mantine / MUI 项目（迁移成本高）/ 不接受 copy-paste 模式（要 npm 升级路径）/ 不写 Tailwind / 严格反 utility-first。
  8. Headless UI vs Radix（同阵营直接竞争）：Radix 在 React 生态完全反超。Headless UI 仍是 Vue 生态主流，shadcn-vue 也基于它。
  9. Ark UI 是 Radix 的竞争者—— 基于 Zag.js 状态机；React + Vue + Solid 多框架；2026 仍属增长期但份额远不及 Radix。Park UI 是 Ark + Panda CSS 的 styled 版本。
  10. 选哪个不是"哪个最好"，是"哪个匹配你团队心智 + 已有技术栈"。React + Tailwind + 接受 utility-first → shadcn；React + 不要 Tailwind → Mantine；多框架（React + Vue + Solid 共存）→ Ark UI；企业 Material Design → MUI。

02 底层逻辑

三阵营划分

2026 年 React UI 库可清晰分三阵营：

阵营 提供什么 不提供什么 代表
Headless 行为 + 状态 + a11y 样式（用户自己写） Radix / Headless UI / Ark UI
Styled 组件 + 样式 + theme 自由定制（要 override） Mantine / MUI / NextUI / Chakra
Hybrid headless 行为 + 推荐样式工具链 npm 包（用 copy-paste） shadcn/ui / Catalyst

三阵营各有适合人群。不存在普世最优。

Headless 阵营 · Radix Primitives（事实标准）

已在 P1.1 详细讲过。核心特点：

  • 提供 Dialog / Popover / Combobox 等组件的行为 + a11y + 状态
  • 不提供样式（用户用 Tailwind / CSS 写）
  • WAI-ARIA APG 严格遵循
  • shadcn 默认底层
  • 2026-02 出统一包 radix-ui

2026 React UI 生态的"a11y 安全网"——用 Radix 几乎自动获得 a11y 一等公民。

Headless 阵营 · Headless UI（被 Radix 反超）

Tailwind Labs 2020 年出品。设计立场：与 Tailwind 配套的 unstyled 组件。

提供的 primitives：

  • Dialog / Popover / Listbox / Combobox / Switch / Tab / Disclosure / Menu
  • 支持 React 和 Vue 两个版本
  • a11y 完整但不如 Radix 严格

与 Radix 对比：

维度Headless UIRadix
组件数量少（~10）多（30+）
a11y 完整度好更好（更严格 APG）
API 风格render prop / function childrenCompound Components
多框架React + VueReact-only
shadcn 是否用否（shadcn 选 Radix）是（默认）
2026 React 份额~5%事实标准（含 shadcn 60%+）
2026 Vue 份额主流（shadcn-vue 用它）无（Radix React-only）

Radix 在 React 生态完全反超的根因：

  • 组件更全（30+ vs 10）
  • shadcn 把 Radix 推到事实标准
  • WAI-ARIA APG 投入更深
  • WorkOS 商业支撑保证长期维护

Headless UI 在 Vue 生态仍是主流（shadcn-vue 也基于它）。

Headless 阵营 · Ark UI（Chakra 团队 / Zag.js）

Chakra UI v3 改造时，Chakra 团队（Segun Adebayo 等）拆分出 Zag.js（headless 状态机库）+ Ark UI（基于 Zag.js 的多框架 React/Vue/Solid 组件库）。

核心特点：

  • Zag.js 是状态机库（基于 XState 思想）—— 用 statechart 描述组件状态转换
  • Ark UI 是 Zag.js 的 React/Vue/Solid 组件包装
  • 多框架共享同一套状态逻辑（React 和 Vue 用同一个 Zag.js）
  • a11y 完整（Chakra 团队投入）

与 Radix 对比：

  • Ark UI 优势：多框架支持、状态机心智更明确
  • Radix 优势：生态成熟、shadcn 推广、a11y 投入更深

2026 年 Ark UI 仍属增长期，但份额远不及 Radix（React 生态被 Radix + shadcn 锁定）。Chakra v3 自己用 Ark UI 重写。

Headless 阵营 · Park UI（Ark + Panda CSS）

Park UI 是 Ark UI 的 styled 版本，用 Panda CSS（Chakra 团队出的 zero-runtime CSS-in-JS 库）添加样式。

定位：给"想要类似 shadcn 但用 Panda 不 Tailwind"的人。Ark UI 多框架支持让 Park UI 也是多框架。

2026 份额很小（不到 1%）。但是 Chakra v3 + Ark + Park 这条路线（Chakra 团队主推）保持活跃。

Styled 阵营 · Mantine（独立 styling）

Vitaly Rtishchev 2021 年发布 Mantine。设计立场：自有 styling 系统 + 组件丰富。

核心特点：

  • 自有 styling 系统（Emotion-based + sx prop）
  • 组件丰富（150+，含 DatePicker / RichTextEditor / Notifications 等）
  • 不基于 Tailwind
  • 与 Tailwind 不冲突可共存（少数项目）
  • a11y 比 Mantine 1.x 改善大（v8 起严格遵循 ARIA）

2026 年 Mantine 现状：

  • 稳定增长（独立生态）
  • 受众：不接受 utility-first / 想要"组件就组件"的团队
  • v8 改进了 a11y
  • 仍是 React UI 第二梯队（仅次于 shadcn）

何时选 Mantine：

  • 不接受 utility-first（不想写 Tailwind class）
  • 需要 DatePicker / RichTextEditor 等开箱即用复杂组件
  • 团队历史项目 Mantine 已经在用

Styled 阵营 · MUI（Material Design）

2014 年起的 React UI 库元老。Material Design 在 React 实现。

核心特点：

  • Material Design v3 视觉语言
  • 组件极丰富（180+）
  • 企业级（B2B SaaS / admin / dashboard 大量存量）
  • 自有 sx prop + theme 系统
  • v6（2024）+ React 19 兼容

2026 年 MUI 现状：

  • 新项目份额持续下滑（被 shadcn 推到边缘）
  • 但企业存量极大（5+ 年老项目大量在用）
  • MUI 团队推 Base UI（unstyled）作为新一代尝试
  • shadcn 双引擎选 Base UI 让 MUI 团队进 shadcn 生态

何时选 MUI：

  • 已有 MUI 项目（迁移成本高）
  • 需要 Material Design 视觉
  • 组件需求极复杂（DataGrid / DatePicker / Charts 等开箱即用）

Styled 阵营 · NextUI / Chakra（小众）

NextUI（HeroUI）：基于 Tailwind v3 但 v4 兼容滞后；2025-2026 年关注度下降。

Chakra UI：v2 时代曾与 Mantine 并列；v3（2024+）改造为 styled-system / Panda CSS 路线，与 Tailwind 心智冲突。Chakra 团队的精力转到 Ark UI / Park UI / Panda CSS。

这两个 2026 都属于"老牌但失势"。

Hybrid 阵营 · shadcn/ui（事实标准）

已在 P1.1 详细讲。核心特点：

  • copy-paste 模式：组件源码进项目
  • 基于 Radix + Tailwind + CVA（默认）
  • 双引擎（2026-02 起，可选 Base UI）
  • 1300+ blocks（dashboard / login 等组合）
  • 免费 + MIT 开源
  • 2026 React 60%+ 份额

Hybrid 阵营 · Catalyst（Tailwind Labs 商业）

Tailwind Labs 2024 年出品的 React UI kit。商业（含 Tailwind UI 订阅，299$/lifetime）。

与 shadcn 对比：

  • Catalyst：付费、Tailwind Labs 出品、组件代码进项目（与 shadcn 类似）
  • shadcn：免费、社区驱动、生态更大
  • 设计风格：Catalyst 更精致（Steve Schoger 设计）；shadcn 更克制（默认 zinc/slate 中性色）

2026 年定位：商业项目预算充裕 + 喜欢 Tailwind Labs 设计 → Catalyst；其他场景 → shadcn。Catalyst 实际市场份额很小（< 5%）。

shadcn + Radix 的 4 个核心设计赌注

把 shadcn + Radix 与同代人对比，能看清它的 4 个根本立场。

1. Headless + 标准 a11y（Radix）—— Radix 实现 WAI-ARIA APG 标准。不发明 a11y 行为，把 W3C 标准做到位。这让任何用 Radix 的项目自动获得 a11y 一等公民。优势：a11y 不需要团队懂 / 跨浏览器一致。代价：要懂 Compound Components 心智。

2. Copy-paste 不 npm install（shadcn）—— 组件源码进项目而不是黑盒 npm 依赖。优势：可任意修改、不被升级破坏、git diff 透明。代价：要手动 sync upstream（用 shadcn diff）。这是 React 组件库分发模式的根本变革。

3. CVA + cn 标准模式（shadcn）—— 用 class-variance-authority 管理变体、用 cn 解决 className 冲突。这套模式被广泛复制到非 shadcn 项目，成为 Tailwind 时代组件变体管理事实标准（详见 Tailwind P6.1）。

4. 双引擎可选（shadcn 2026）—— 2026-02 起，shadcn 不再绑死 Radix；用户可选 Base UI（MUI 出品的 unstyled）。这反映"shadcn 是模板而不是 Radix wrapper"的本质。优势：用户根据需求选 primitive 库；shadcn 自身不被特定 primitive 库绑定。代价：2026 仍处于"两个引擎并存"过渡期。

2026 实战分布

基于 npm 下载量 + GitHub stars + 公开调研：

  • shadcn/ui：60%+ 新 React 项目（含 Vercel / Linear / Resend 等明星公司）
  • MUI：~10%（企业大量存量、Material Design 项目）
  • Mantine：~5%（不接受 utility-first 团队）
  • Radix Primitives 独立用：~5%（建自己的 design system）
  • Headless UI / Ark UI / NextUI / Chakra v3：合计 ~10%
  • 自建组件库：~10%（大型企业基于 Radix 自建）

03 应用场景

场景 1 · 起新 React SaaS 项目

选 shadcn/ui + Radix（默认引擎）。理由：

  • 2026 React 事实标准
  • 免费 + MIT
  • 组件代码可控
  • 与 Tailwind v4 兼容
  • a11y 完整（Radix）
  • 团队招人友好

场景 2 · 不接受 utility-first 文化的团队

选 Mantine。理由：

  • 不需要 Tailwind
  • 组件极丰富（DatePicker / RichTextEditor 等开箱即用）
  • v8 起 a11y 改善
  • 不需要 copy-paste 模式（团队不接受）

场景 3 · 大型企业 Material Design 项目

继续 MUI。理由：

  • Material Design 视觉
  • 已有大量 MUI 代码
  • 组件极丰富（DataGrid 商业版含分页 / 排序 / 编辑）
  • 不要为"跟随主流"迁 shadcn

场景 4 · 多框架（React + Vue + Solid 共存）

选 Ark UI。理由：

  • 多框架共享 Zag.js 状态机
  • 团队多 stack 心智一致
  • 但要接受 React 生态资源比 Radix 少

场景 5 · 自建团队 design system

选 Radix Primitives 独立用（不通过 shadcn）。理由：

  • 团队有自己设计语言（不接受 shadcn 默认风格）
  • 大型企业 5-50+ 工程师投入设计系统
  • 用 Radix 提供 a11y 基础，自己写样式
  • shadcn 模板不需要（团队会自己组织代码）

04 典型代码

代码 1 · Dialog 在 5 个 UI 库的对比

tsx 
// 1. shadcn/ui（推荐）
import { Dialog, DialogContent, DialogTrigger } from '@/components/ui/dialog';

<Dialog>
  <DialogTrigger>Open</DialogTrigger>
  <DialogContent>
    <h2>Confirm</h2>
    <p>Are you sure?</p>
  </DialogContent>
</Dialog>
tsx 
// 2. Radix 直接用（不通过 shadcn）
import { Dialog } from 'radix-ui';

<Dialog.Root>
  <Dialog.Trigger>Open</Dialog.Trigger>
  <Dialog.Portal>
    <Dialog.Overlay className="..." />
    <Dialog.Content className="...">
      <Dialog.Title>Confirm</Dialog.Title>
      <Dialog.Description>Are you sure?</Dialog.Description>
    </Dialog.Content>
  </Dialog.Portal>
</Dialog.Root>
tsx 
// 3. Mantine
import { Modal, Button } from '@mantine/core';
const [opened, { open, close }] = useDisclosure(false);

<Button onClick={open}>Open</Button>
<Modal opened={opened} onClose={close} title="Confirm">
  <p>Are you sure?</p>
</Modal>
tsx 
// 4. MUI
import { Dialog, DialogTitle, DialogContent, Button } from '@mui/material';

<Button onClick={() => setOpen(true)}>Open</Button>
<Dialog open={open} onClose={() => setOpen(false)}>
  <DialogTitle>Confirm</DialogTitle>
  <DialogContent>Are you sure?</DialogContent>
</Dialog>
tsx 
// 5. Headless UI（与 shadcn 类似但不那么完整）
import { Dialog } from '@headlessui/react';

<Dialog open={isOpen} onClose={setIsOpen} className="...">
  <Dialog.Panel>
    <Dialog.Title>Confirm</Dialog.Title>
    <Dialog.Description>Are you sure?</Dialog.Description>
  </Dialog.Panel>
</Dialog>

05 常见陷阱

  • "shadcn 一定比 Mantine 好" —— 不一定。不接受 utility-first 的团队 Mantine 更合适。判断要看场景
  • 用 MUI 但纠结要不要切 shadcn —— 已用 MUI 的项目继续 MUI（迁移成本高）。不要为"跟随主流"切
  • 用 Headless UI 在 React 项目 —— React 生态选 Radix（shadcn 也用 Radix）；Headless UI 留给 Vue 项目
  • 新项目用 NextUI —— 关注度下降 + v4 兼容滞后。新项目选 shadcn 更稳
  • 认为 Ark UI 因为多框架就该选 —— 多框架是优势，但 React 生态 Radix + shadcn 远成熟。除非真有多框架需求否则选 shadcn
  • 看 2022-2023 年的 React UI 库对比文章 —— 那时候 shadcn 还没火、双引擎不存在、Radix 多包未统一。结论几乎全过时
  • "vendor-neutral 一定比 vendor-locked 好" —— shadcn 是 MIT 开源 + 不绑特定平台，已是 vendor-neutral。批评 shadcn 因为它"被 Vercel 推"是误解
  • 同时用 shadcn + MUI —— 不推荐。两套样式 + theme 系统冲突，bundle 重复
  • "Radix Themes = Radix Primitives" —— 是两个产品。shadcn 用 Primitives 不是 Themes
  • 用 Chakra v3 然后纠结 v2 → v3 迁移 —— v3 改造为 Panda CSS 心智差异大。如果要迁移成本高考虑直接切 shadcn
  • "shadcn 太流行所以反着选" —— 流行不等于错。shadcn 流行是因为它真好用
