决策树 + 2027 展望

shadcn/ui + Radix 主题收束章。把 P1-P7.1 学到的串起来，给一份能直接用到生产的决策框架：什么时候用 shadcn / 什么时候应该选别的、Base UI 还是 Radix、shadcn vs Catalyst vs 自建组件库、2027 展望。

阅读时长 ~25 分钟 前置 P1-P7.1 全部 主要来源 ui.shadcn.com docs + 各替代方案官方 + 2025-2026 公开调研

01 速览

  1. 选型决策第一问：项目用 React + Tailwind 吗？是 → shadcn/ui 强烈推荐；否（Vue / Svelte → shadcn-vue / shadcn-svelte 第三方移植；不要 Tailwind → Mantine / MUI）。
  2. 第二问：接受 copy-paste 模式吗？接受 → shadcn；不接受 → Radix Primitives 直接用 + 自己包装 / Mantine（npm package 模式）。
  3. 第三问（仅 shadcn 用户）：选 Radix 引擎还是 Base UI 引擎？默认 Radix（生态成熟）；性能极致 / 与 MUI 共存 → Base UI。
  4. 何时不用 shadcn/ui：1) 已有 Mantine / MUI / Chakra 项目（迁移成本高）/ 2) 不写 React（shadcn React-only）/ 3) 不写 Tailwind / 4) 严格反 utility-first 文化 / 5) 极小项目（5-10 页面）overkill。
  5. shadcn vs Catalyst：shadcn 免费 + 社区驱动 + 生态大；Catalyst 付费（299$ lifetime）+ Tailwind Labs 出品 + 设计精致。商业项目能买可补 Catalyst 作设计参考，主代码仍 shadcn。
  6. shadcn vs 自建组件库：5-20 人团队 → shadcn（不重新发明）/ 50+ 人企业 → 自建（基于 Radix）以适应特殊设计系统 / 极小团队（1-3 人）→ 直接 Mantine（少一层心智）。
  7. 2027 展望 1 · 多框架完整：shadcn-vue / shadcn-svelte / shadcn-solid 等成熟，"copy-paste 模式"成为跨框架事实。
  8. 2027 展望 2 · Visual Builder 进化：从 init 扩展到组件级编辑（拖拽改样式 → 生成代码）。
  9. 2027 展望 3 · Registry 标准化：成为 React 组件库分发新事实标准；第三方 registry（v0.dev / Aceternity / Magic UI）共享 schema。
  10. 2027 展望 4 · AI 辅助 component 生成：从 Figma → shadcn 代码（已有 v0.dev、Subframe 等）；Cursor / Claude / GitHub Copilot 流畅写 shadcn。

何时用 shadcn/ui

4 个核心场景：

1. 起新 React + Tailwind 项目：

  • 2026 React UI 事实标准
  • 免费 + 开源
  • 组件代码可控（不会被 npm 升级破坏）
  • 团队招人友好（多数 React 工程师懂）
  • 1300+ blocks 起步快

2. 已用 Tailwind 想要 UI 库：

  • shadcn 与 Tailwind 哲学契合（CVA + cn 标准模式）
  • 已有 Tailwind 配置可复用
  • 设计 token 与 Tailwind 集成自然

3. 需要完整可定制（不能被 npm 升级破坏）：

  • shadcn copy-paste 模式让组件源码进项目
  • 对应 Mantine / MUI 的 npm package 模式
  • shadcn diff sync upstream 改进（手动选择）

4. 团队接受 copy-paste 模式：

  • 不接受 → 用 Radix 直接 + 自己包装（仍保持 npm package 模式）
  • 接受 → shadcn 工作流

何时不用 shadcn/ui

5 个核心场景：

1. 已有 Mantine / MUI / Chakra 项目：

  • 迁移成本高（每个组件改 import + 重新调样式）
  • 不要为"跟随主流"而迁
  • 除非有强需求（性能 / a11y / RSC 兼容性）否则保持

2. 不写 React：

  • shadcn 是 React-only
  • shadcn-vue / shadcn-svelte 是社区移植，质量与官方有差距
  • Vue 用 Headless UI Vue / Mantine 类
  • Svelte 用 SvelteKit 内置 + 自己写

3. 不写 Tailwind：

  • shadcn 默认 Tailwind 集成（CVA + cn 用 Tailwind class）
  • 不用 Tailwind 选 Mantine（独立 styling 系统）
  • vanilla-extract（CSS-in-TS）+ Radix 自己组合也可以

4. 严格反 utility-first 文化：

  • Tailwind utility-first 的接受度决定 shadcn 接受度
  • 团队就是不接受 utility 串 → Mantine（semantic class）
  • 不要强推

5. 极小项目：

  • 5-10 页 简单营销站 → 直接写 Tailwind class，不需要 shadcn 组件
  • shadcn 的"工程化工具链"价值在中型项目+ 才显

Base UI vs Radix UI 选哪个

2026-02 起 shadcn 支持双引擎。详细对比：

维度Radix PrimitivesBase UI
出品WorkOS（前 Modulz）MUI 团队
组件数30+20+（仍在扩张）
bundle 大小较大（feature-rich）较小（minimal）
a11y 完整度事实标准好（MUI 团队投入）
API 风格Compound ComponentsCompound + render props 混合
社区资源大（shadcn 生态）小（新加入）
2026 React 份额事实标准（含 shadcn）新加入（< 5% 选这个）
shadcn 默认是（默认）opt-in

选 Radix 的场景（多数）：

  • 新项目 / 团队不熟双引擎
  • 需要丰富组件（30+）
  • 看重社区资源 + 文章 + 教程
  • 要 a11y 最完整

选 Base UI 的场景（少数）：

  • 极致性能（bundle 每 KB 重要）
  • 已用 MUI 想生态共存
  • 团队偏好 MUI 风格 API
  • 愿意承担"新加入选项"风险（生态资源少）

2026 实战建议：默认 Radix。Base UI 等 2027-2028 生态成熟再考虑。

shadcn vs Catalyst vs 自建组件库

shadcn vs Catalyst：

维度shadcn/uiCatalyst
价格免费 + MIT299$ lifetime（含 Tailwind UI）
社区大（事实标准）小（商业产品）
设计风格克制（zinc 默认）精致（Tailwind Labs 设计）
组件源码进项目是（copy-paste）是（npm-installable 但你也得到源码）
升级shadcn diff 手动npm update
2026 份额60%+~3%

选哪个：

  • 免费 + 社区 → shadcn（多数）
  • 商业项目能买 + 喜欢 Tailwind Labs 设计 → 买 Catalyst 作设计参考 + 用 shadcn 实现

shadcn vs 自建组件库：

团队规模推荐原因
极小（1-3 人）直接 Mantineshadcn 工程链 overkill；Mantine 少一层心智
小-中（5-20 人）shadcn不重新发明 + 起步快 + 团队学习成本低
中-大（20-50 人）shadcn + 私有 registry团队 design system 工具链（详见 P5.2）
大型企业（50+）自建（基于 Radix）有特殊设计系统 + 投入资源足以维护

"自建组件库"≠ "不用 Radix"。多数大型企业用 Radix Primitives 作 a11y 基础，自己写包装层（包括样式 / 主题 / API 风格）。这是 LinkedIn / Microsoft / GitHub 的模式。

2027 展望 · 4 个方向

1. 多框架完整：

  • shadcn-vue / shadcn-svelte / shadcn-solid 等社区移植成熟
  • "copy-paste 模式"成为跨框架事实
  • NativeWind 让 React Native 也走 shadcn 类似模式
  • shadcn 官方可能正式收编一些移植版（仍在讨论中）

2. Visual Builder 进化：

  • 2026 Visual Builder 仅做项目 init
  • 2027 期望扩展到组件级编辑（拖拽改样式 → 生成代码）
  • v0.dev 在这方向已经走前面（AI 生成 + 编辑）
  • shadcn 官方 Visual Builder 与 v0.dev 整合可能

3. Registry 标准化：

  • shadcn registry schema 成为 React 组件库分发新事实标准
  • 第三方 registry（v0.dev / Aceternity UI / Magic UI / Tremor 等）都基于 shadcn schema
  • private registry + auth 在企业普及
  • monorepo design system 工作流标准化

4. AI 辅助 component 生成：

  • Cursor / Claude / GitHub Copilot 已能流畅写 shadcn
  • v0.dev 从 Figma → shadcn 代码
  • 2027 期望：从设计稿 → shadcn registry item（直接进 registry，团队复用）
  • AI 写组件 a11y 仍是挑战（要 review 是否符合 WAI-ARIA）

其他方向：

  • Radix v2 路线图：含 Calendar / Toolbar / 更多 primitives
  • Base UI 成熟：组件覆盖度提升，性能优势让一部分团队切换
  • Cache Components 兼容：shadcn 组件与 Next.js 16 Cache Components 集成
  • Server Components 深度：shadcn 组件越来越多原生 RSC 兼容

03 应用场景

场景 1 · 起新 React SaaS 完整 stack

2026 推荐栈：

  • Framework：Next.js 16 + App Router
  • 样式：Tailwind v4 + CSS-first 配置
  • UI 组件：shadcn/ui v4 + Radix 引擎
  • 变体管理：CVA + cn (shadcn 内置)
  • 表单：react-hook-form + Zod + shadcn Form
  • 暗色：next-themes（class 策略）
  • Toast：Sonner
  • 表格：TanStack Table + shadcn DataTable
  • 命令面板：cmdk（shadcn Command）
  • 设计 token：shadcn theme builder + globals.css
  • 测试：Vitest + Playwright

场景 2 · 不接受 utility-first 的团队

选 Mantine。理由：

  • 不需要 Tailwind
  • 组件极丰富（DatePicker / RichTextEditor 等开箱即用）
  • 自有 styling 系统
  • v8 起 a11y 改善

场景 3 · 大型企业自建组件库

50+ 人企业，有专门 design system 团队：

  • 基于 Radix Primitives 自己写包装组件
  • 用自己的设计 token / 命名规范
  • 用 shadcn registry schema 分发（团队私有 registry）
  • 不用 shadcn 默认组件（设计语言不同）

这是 LinkedIn / Stripe 等公司的模式。

场景 4 · 已有 MUI 项目要不要迁

不建议迁。理由：

  • 迁移成本高（每个组件改）
  • 设计语言差异大（Material Design vs shadcn 极简）
  • 除非有强需求（性能 / RSC 兼容性 / a11y）
  • 渐进迁移：新模块用 shadcn、老模块保留 MUI

场景 5 · 极小团队 (1-3 人)

选 Mantine 或直接用 Radix Primitives。理由：

  • shadcn 工程链 overkill（CVA + cn + registry 等都不需要）
  • Mantine 开箱即用更快
  • 或 Radix 直接用 + 几行 Tailwind

05 常见陷阱

  • "shadcn 一定比 Mantine 好" —— 不一定。不接受 utility-first 团队 Mantine 更合适
  • 已有 MUI 项目强迁 shadcn —— 迁移成本高，不值得。除非有强需求
  • 极小团队用 shadcn 工程链 —— overkill。直接 Mantine 或 Radix + 几行 Tailwind
  • 用 shadcn 但不接受 utility-first 文化 —— 矛盾。要么团队转变，要么换 Mantine
  • 2026 年新项目用 NextUI 而不是 shadcn —— NextUI 关注度下降 + v4 兼容滞后
  • 不写 Tailwind 还想用 shadcn —— shadcn 默认 Tailwind 集成，强行不用 Tailwind 失去主要价值
  • "copy-paste 是技术债务" —— 误解。这是 shadcn 设计哲学。详见 P1.1
  • 商业项目纠结 shadcn vs Catalyst —— 大多数选 shadcn（免费 + 生态大）；可选买 Catalyst 作设计参考
  • 50+ 人企业仍用 shadcn 默认设计 —— 大企业有自己设计语言。基于 Radix 自建 + shadcn registry 模式
  • 2026 用 Base UI 因为"它新" —— 生态资源少。除非有具体理由（性能 / MUI 共存）选 Radix
  • 等 shadcn-vue 完美再用 —— 第三方移植不会 100% 与官方同步。如果不是非 Vue 不可，考虑 React 项目
  • "我们要 vendor-neutral 不用 shadcn" —— shadcn 是 MIT 开源 + 不绑特定平台。已是 vendor-neutral
