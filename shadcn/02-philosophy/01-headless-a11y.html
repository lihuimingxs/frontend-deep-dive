Headless + WAI-ARIA a11y 标准

Radix 的核心赌注：不是发明 a11y、是把 W3C WAI-ARIA APG 标准做到位。这一章把 Dialog / Combobox 两个最复杂 widget 的完整 a11y 标准摊开讲透 —— 让读者看到为什么自己手写组件几乎一定漏 a11y、为什么 Radix 这套投入是工程精华。读完不再觉得 a11y 是"加几个 aria-* 属性"那么简单。

阅读时长 ~30 分钟 前置 P1 项目史 + 写过 React 组件 主要来源 WAI-ARIA APG (W3C) + radix-ui.com docs + Pedro Duarte / a11y 专家公开演讲

01 速览

  1. Headless 定义：组件提供"行为 + 状态 + a11y"，不提供样式。用户用 Tailwind / CSS 写样式。与 styled 库（Mantine / MUI）反义。
  2. WAI-ARIA APG（Authoring Practices Guide）：W3C 维护的"组件 a11y 标准"权威文档。定义每种 widget（Dialog / Combobox / Tab / Menu）的键盘交互 / aria-* 属性 / focus 管理规范。Radix 严格按 APG 实现，不发明。
  3. Dialog 的 a11y 标准有 8 条核心规则：1) 打开 focus 移到 Dialog 内；2) 关闭 focus 回到触发元素；3) ESC 关闭；4) Tab 在 Dialog 内循环（focus trap）；5) aria-modal="true"；6) aria-labelledby 指向 title；7) aria-describedby 指向 description；8) 屏幕阅读器读出 "dialog [title]"。
  4. Combobox 的 a11y 标准更复杂（12 条核心规则）：键盘导航 ↓↑ Enter ESC、aria-expanded / aria-controls / aria-activedescendant（不是真 focus）、屏幕阅读器实时读出可用选项数、选中后清空策略等。
  5. a11y 为什么难手写：标准复杂（每种 widget 几十条规则）/ 不同浏览器 / screen reader 行为差异（NVDA vs JAWS vs VoiceOver）/ 边角情况多（嵌套 dialog / modal 之上 popover / 表单中 combobox）/ 测试难（要真用 screen reader）。
  6. Radix 怎么把这些做到位：每个 primitive 有专门 a11y test suite / 用 NVDA / JAWS / VoiceOver 实测 / 跟踪 WAI-ARIA spec 更新 / 团队成员含 a11y 专家。
  7. aria-activedescendant vs roving tabindex：Combobox / Listbox 用前者（不真 focus 选项，避免输入框失焦）；Menu / Toolbar 用后者（真 focus，Tab 跳到下一控件）。这是 W3C APG 规定的两种 focus 模式。
  8. Focus management是 a11y 最难的部分：Dialog 打开 focus 哪个元素？ESC 关闭后 focus 回哪？嵌套 dialog 怎么处理？Tooltip 不应该抢 focus 但要读出。这些 Radix 都按 APG 处理。
  9. Screen reader 测试是 a11y 验证的金标准。NVDA（Windows，免费）/ JAWS（Windows，商业）/ VoiceOver（macOS/iOS）/ TalkBack（Android）。Radix 用 NVDA + VoiceOver 至少测两套。
  10. 本章建立"Radix 不是发明 a11y、是把 W3C 标准做到位"的心智。让读者看到 a11y 复杂度后自然信任 Radix 不再尝试自己写。

02 底层逻辑

Headless 是什么 · 与 Styled 对比

Headless 库提供：

  • 组件的行为（点击展开 / 键盘导航 / focus 管理）
  • 组件的状态（open / value / focused 等）
  • 组件的a11y 属性（role / aria-* / data-*）

Headless 库不提供：

  • 颜色 / 间距 / 字号 / 阴影 等样式
  • 主题 / 暗色模式系统（用户自己加）
  • 动画（用户自己用 CSS / framer-motion）

对比 styled 库：

tsx 
// Headless（Radix）
<Dialog.Root>
  <Dialog.Trigger>Open</Dialog.Trigger>
  <Dialog.Portal>
    <Dialog.Overlay className="fixed inset-0 bg-black/50" />
    <Dialog.Content className="fixed left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2 bg-white p-6 rounded-lg">
      <Dialog.Title className="text-lg font-semibold">Confirm</Dialog.Title>
      <Dialog.Description>Are you sure?</Dialog.Description>
    </Dialog.Content>
  </Dialog.Portal>
</Dialog.Root>
{/* Radix 提供：行为 + 状态 + a11y；用户提供 className */}

// Styled（Mantine）
<Modal opened={opened} onClose={close} title="Confirm" centered size="md">
  <p>Are you sure?</p>
</Modal>
{/* Mantine 提供：行为 + 状态 + a11y + 样式；用户传 props 调样式 */}

哪个好？看场景：

  • Tailwind 项目 + 想要完整定制 → Headless（Radix）
  • 不要 Tailwind + 想开箱即用 → Styled（Mantine）

WAI-ARIA APG · W3C 权威文档

WAI-ARIA Authoring Practices Guide（APG）是 W3C 维护的组件 a11y 实现指南。完整 URL：w3.org/WAI/ARIA/apg/。

APG 包含的内容：

  • 每种常见 widget 的"设计模式"（design pattern）
  • 每个模式的键盘交互规范
  • 每个模式的 aria-* 属性使用规范
  • 每个模式的 focus 管理规范
  • 每个模式的实际可工作示例（含完整代码）

覆盖的 widgets：

  • Disclosure（折叠面板）
  • Dialog (Modal)（模态框）
  • Alert / Alert Dialog（警告 / 强制确认）
  • Tabs（标签页）
  • Accordion（手风琴）
  • Combobox（输入 + 自动完成）
  • Listbox（列表选择）
  • Menu / Menubar（菜单）
  • Tooltip（提示气泡）
  • Slider / Spin Button
  • Switch / Checkbox / Radio Group
  • Tree（树形列表）
  • Grid / Treegrid（网格 / 树形网格）
  • Date Picker
  • 更多...

Radix 的 30+ primitives 一一对应 APG widgets。这是 Radix "不是发明、是实现标准"的具体表现。

Dialog 的 8 条 a11y 规则（详细）

WAI-ARIA APG 对 Dialog（Modal）规定的核心规则：

规则 1 · Focus 移到 Dialog 内：

  • Dialog 打开时，focus 必须从触发按钮转移到 Dialog 内某个可 focus 元素
  • 默认 focus 第一个可 focus 元素（通常是关闭按钮或第一个 input）
  • 开发者可显式指定（Radix 用 onOpenAutoFocus 自定义）
  • 错误：Dialog 打开但 focus 仍在触发按钮 → 屏幕阅读器用户感知不到 Dialog

规则 2 · ESC 关闭：

  • 用户按 ESC 必须关闭 Dialog
  • Radix 自动监听 ESC keydown
  • 嵌套 Dialog 时只关最上层（一次 ESC 关一层）

规则 3 · Focus Trap：

  • Dialog 打开时，Tab 键必须在 Dialog 内循环
  • 到 Dialog 最后一个可 focus 元素时，Tab 跳回第一个
  • Shift+Tab 反向循环
  • 不能 Tab 出 Dialog 到背景元素
  • Radix 用内部 focus trap 实现

规则 4 · 关闭后 focus 回触发元素：

  • Dialog 关闭后，focus 必须回到打开它的按钮
  • 这让屏幕阅读器用户知道"回到了原页面"
  • Radix 自动处理

规则 5 · aria-modal="true"：

  • Dialog 容器加 aria-modal="true"
  • 告诉屏幕阅读器"这是模态对话框、外面内容暂时不可访问"
  • Radix 自动加

规则 6 · aria-labelledby 指向 title：

  • Dialog 容器加 aria-labelledby="dialog-title-id"
  • title 元素加 id="dialog-title-id"
  • 屏幕阅读器读出 dialog 时同时读 title
  • Radix 用 Compound Components 自动关联（Dialog.Title 自动给 id 并关联到 Dialog.Content 的 aria-labelledby）

规则 7 · aria-describedby 指向 description（可选）：

  • 类似 aria-labelledby 但指向 description（详细说明）
  • Radix Dialog.Description 自动关联

规则 8 · 屏幕阅读器读出 "dialog [title]"：

  • 所有以上规则到位后，屏幕阅读器（NVDA / JAWS / VoiceOver）会自动读出 "dialog [title]" 提示用户
  • 这是 a11y 完整的标志

这 8 条规则 Radix 自动实现，用户写 <Dialog.Root> 不需要操心。但自己手写 Dialog 几乎一定漏其中几条（特别是 focus trap + 关闭后 focus 回触发）。

Combobox 的 12 条 a11y 规则（更复杂）

Combobox 是最复杂的 widget。WAI-ARIA APG 规定的核心规则：

结构：

  • 1) input 元素 + 触发按钮（可选）+ listbox（选项列表）
  • 2) input role="combobox"
  • 3) listbox role="listbox"
  • 4) 选项 role="option"

状态属性：

  • 5) input aria-expanded（listbox 是否展开）
  • 6) input aria-controls（指向 listbox id）
  • 7) input aria-activedescendant（指向当前高亮选项 id）

键盘交互：

  • 8) ↓ ↑ 导航选项（Home 跳第一个、End 跳最后一个）
  • 9) Enter 选中当前高亮选项
  • 10) ESC 关闭 listbox（保留 input 内容）+ ESC 再次清空 input
  • 11) Tab 关闭 listbox + 移动到下一控件

屏幕阅读器：

  • 12) 实时读出 "X options available"（选项数）
  • 读出当前高亮选项的 label + position（"3 of 10"）
  • 读出选中状态（"selected"）

关键设计：aria-activedescendant 而非真 focus：

  • 用户在 input 输入时，焦点应该停在 input（继续输入）
  • 但 ↓ ↑ 应该"高亮"选项
  • 解决方案：用 aria-activedescendant 让屏幕阅读器知道"虚拟 focus"在哪
  • 真 DOM focus 仍在 input
  • 这是 W3C APG 规定的 Combobox / Listbox focus 模式

对比 Menu / Toolbar 用roving tabindex（真 focus 跳）：

  • 菜单项之间用 ↓ ↑ 真 focus 跳转
  • 非当前 focus 项 tabindex="-1"
  • Tab 跳出整个菜单到下一控件

这两种模式选哪个由 widget 类型决定，不是用户选。Radix（和 cmdk）按 APG 用对应模式。

a11y 为什么难手写 · 4 个根因

1. 标准复杂：

  • 每种 widget 几十条规则
  • 规则之间互相依赖（Dialog 的 focus trap 依赖 Tab 监听）
  • 边角情况多（嵌套 dialog 怎么处理 focus？modal 之上 popover 怎么处理 ESC？）

2. 不同浏览器 / screen reader 行为差异：

  • aria-modal 在 NVDA + Chrome 工作完美
  • JAWS + IE11 行为略有差异（IE11 已退场，但 JAWS 自身有 quirks）
  • VoiceOver 在 Safari 与 Chrome 有微妙差异
  • TalkBack（Android）与桌面 screen reader 不一样

3. 测试难：

  • 需要真用 NVDA / JAWS / VoiceOver 听屏幕阅读器
  • 需要键盘 only 测试（拔鼠标）
  • 自动化测试只能 catch aria-* 属性是否存在，不能 catch 行为正确
  • Playwright / Cypress 等 E2E 工具对 a11y 测试有限

4. 工程优先级：

  • 大多数团队"a11y 是最后做的事"
  • 没有专职 a11y 工程师
  • 无 a11y 测试 CI
  • 结果：自己手写组件几乎一定漏 a11y

Radix 的解法："把 a11y 内置到组件，让用户不需要懂"。这是 Radix 最大的工程价值。

Radix 怎么把 a11y 做到位 · 4 个工程实践

1. 每个 primitive 有专门 a11y test suite：

  • 测试 keyboard interaction（按键 + 检查焦点 / aria-* 状态）
  • 测试 aria-* 属性（render 后 DOM 检查）
  • 测试边角情况（嵌套 / 错误状态 / 异步加载）

2. 用真 screen reader 测：

  • NVDA + Chrome（Windows）
  • VoiceOver + Safari（macOS / iOS）
  • JAWS（商业，主要企业用户）
  • 每次大改动手动测一遍

3. 跟踪 WAI-ARIA spec 更新：

  • W3C 偶尔更新 APG（新 widget / 现有 widget 改进）
  • Radix 团队跟踪并更新对应 primitive

4. 团队成员含 a11y 专家：

  • Pedro Duarte 自己是 a11y 专家
  • WorkOS 给 Radix 的全职岗位让团队能投入 a11y

这是为什么 Radix 在 a11y 完整度上断层式领先同代竞品。

03 应用场景

场景 1 · 团队不懂 a11y，要做出 a11y 完整的应用

用 Radix（通过 shadcn）。Radix 把 a11y 内置，用户不需要懂细节就能写出 a11y 完整组件。

但仍要做的：

  • 用 NVDA / VoiceOver 偶尔实测（catch 自己加的内容是否破坏 a11y）
  • 用 axe-core / Lighthouse 自动 a11y audit
  • 表单加 aria-label / 描述（Radix 不能替你写文字）
  • 图片加 alt（Radix 与图片无关）

场景 2 · 法律要求 a11y 合规（ADA / EAA）

美国 ADA / 欧盟 EAA 等法律要求 web 应用 a11y 合规。这种项目：

  • 必须用 a11y 完整的 UI 库（Radix 是首选）
  • 必须做 a11y 审计（雇 a11y 顾问 / 用 axe-core）
  • 必须做 screen reader 测试
  • 不能用 a11y 不全的库（手写组件 + Headless UI 早期版本等）

场景 3 · 为什么不直接 import @radix-ui/react-* 用

可以直接用 Radix Primitives，不通过 shadcn。这是合理选择：

  • 团队有自己设计语言（不接受 shadcn 默认）
  • 大型企业自建 design system
  • 组件库作者基于 Radix 出新库

shadcn 只是"Radix + Tailwind + CVA 的模板"，不是必须。理解这点，shadcn vs 直接用 Radix 的决策才清晰。

04 典型代码

代码 1 · 自己手写 Dialog（漏 a11y）

tsx 
// ❌ 自己手写（漏多条 a11y 规则）
function MyDialog({ open, onClose, children }) {
  if (!open) return null;
  return (
    <div className="fixed inset-0 bg-black/50 flex items-center justify-center">
      <div className="bg-white p-6 rounded-lg">
        <button onClick={onClose}>X</button>
        {children}
      </div>
    </div>
  );
}

// 漏的 a11y：
// ❌ focus 没移到 Dialog（屏幕阅读器读不出 Dialog 打开）
// ❌ 没有 focus trap（Tab 跳到背景）
// ❌ 关闭后 focus 不回触发按钮
// ❌ ESC 不关闭
// ❌ 没有 aria-modal / aria-labelledby / aria-describedby
// ❌ 屏幕阅读器读不出 "dialog"
// ❌ 没有 scroll lock（背景仍可滚）
// ❌ 没有 portal（z-index 战争）

代码 2 · Radix Dialog（a11y 完整）

tsx 
// ✅ Radix（自动处理所有 a11y）
import { Dialog } from 'radix-ui';

function MyDialog() {
  return (
    <Dialog.Root>
      <Dialog.Trigger asChild>
        <button>Open</button>
      </Dialog.Trigger>
      <Dialog.Portal>
        <Dialog.Overlay className="fixed inset-0 bg-black/50" />
        <Dialog.Content className="fixed left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2 bg-white p-6 rounded-lg">
          <Dialog.Title>Confirm</Dialog.Title>
          <Dialog.Description>Are you sure?</Dialog.Description>
          <Dialog.Close>Cancel</Dialog.Close>
        </Dialog.Content>
      </Dialog.Portal>
    </Dialog.Root>
  );
}

// Radix 自动处理：
// ✅ 打开 focus 移到 Dialog
// ✅ Focus trap（Tab 在 Dialog 内循环）
// ✅ 关闭后 focus 回触发按钮
// ✅ ESC 关闭
// ✅ aria-modal="true"
// ✅ aria-labelledby 指向 Dialog.Title
// ✅ aria-describedby 指向 Dialog.Description
// ✅ Portal 渲染到 body 末尾
// ✅ scroll lock（body overflow: hidden）
// ✅ 屏幕阅读器读出 "dialog Confirm"

05 常见陷阱

  • 认为"加几个 aria-* 就是 a11y" —— 错。aria-* 是其中一部分，focus 管理 / 键盘交互 / 屏幕阅读器测试同样重要
  • 自己手写 Dialog / Combobox —— 几乎一定漏 a11y。用 Radix（即使不用 shadcn 也至少用 Radix）
  • 用 axe-core 通过就以为 a11y 完整 —— axe 只能 catch 静态 aria-* 问题，不能 catch focus 管理 / 键盘交互。要真用 NVDA / VoiceOver 测
  • 不真用 screen reader 测 —— 自动化测试有限。a11y 是"用户体验"，必须真听
  • aria-hidden 滥用 —— aria-hidden="true" 让屏幕阅读器跳过元素。用错位置（覆盖整个 main）让整页对屏幕阅读器消失
  • tabindex="-1" 用在错的元素 —— -1 让元素可被 .focus() 但不被 Tab 触达。用错让键盘用户无法到达交互元素
  • Tooltip 在 disabled button 上 —— disabled button 不触发 hover/focus，Tooltip 不会出现。包一层 div 解决
  • 嵌套 Dialog 不处理 focus —— Radix 自动处理（最上层 Dialog 关后 focus 回中层），自己写易错
  • "Combobox 用不到 a11y，反正大家用鼠标" —— 错。键盘用户 + screen reader 用户都需要 Combobox a11y。法律风险
  • aria-label / aria-labelledby 写错文本 —— "Submit" 比 "Click here" 好。要描述动作，不要描述位置
  • Radix Dialog 没用 Dialog.Title 而用 div —— 失去 aria-labelledby 自动关联。必须用 Dialog.Title 或显式设 aria-labelledby
  • "我们项目不需要 a11y，没残障用户" —— 错。键盘用户 / 临时受伤 / 屏幕反光 / 嘈杂环境也需要 a11y。是普世可用性
  • 用 onClick 不监听 keydown —— button 元素 React onClick 自动处理键盘 Enter / Space。但 div 加 onClick 不触发键盘
