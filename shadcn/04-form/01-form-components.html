表单组件全谱

表单 widget 比浮层更难（状态多、a11y 严格、与表单系统集成）。这一章把简单控件（Checkbox / RadioGroup / Switch / Slider）+ 复杂控件（Select / Combobox）讲透，重点 Combobox（最难的 widget）+ shadcn 用 cmdk 而非 Radix 的工程权衡。

阅读时长 ~30 分钟 前置 P2 + P3 + 写过 React 表单 主要来源 radix-ui.com docs + cmdk.paco.me + WAI-ARIA APG Combobox/Listbox

01 速览

  1. 表单控件分两类：简单（Checkbox / RadioGroup / Switch / Slider / Toggle）和 复杂（Select / Combobox / Listbox）。简单控件 a11y 直接（与原生类似）；复杂控件 a11y 严格（特别是 Combobox）。
  2. Checkbox 关键特性：indeterminate state（"半选"，用于"全选"按钮逻辑）、aria-checked。Radix Checkbox 把原生 input + label 包装为更易样式化的组件。
  3. RadioGroup：单选互斥、键盘 ← → 在选项间导航（不用 Tab）、aria-checked。Radix 自动处理 roving tabindex。
  4. Switch：与 Checkbox 类似但视觉是 toggle（拉杆）、role="switch" 而非 "checkbox"。屏幕阅读器读出 "on/off" 而非 "checked/unchecked"。
  5. Slider：单值 / 范围（多 thumb）、键盘 ← → ↑ ↓ Home End、aria-valuemin / aria-valuemax / aria-valuenow。Radix Slider 处理触摸 / 键盘 / 鼠标三种交互。
  6. Select：下拉选择、单选、与原生 <select> 类似 a11y。Radix Select 自定义视觉但保留原生 a11y 行为。
  7. Combobox 是最难的 widget：输入 + 自动完成 + 选项列表 + 选中标签（可清除）。与 Select 区别：Select 是固定选项列表、Combobox 输入框过滤。
  8. Radix 没有 Combobox primitive（!）。shadcn 用 cmdk（Pkmn / Linear 团队开源）作为 Combobox 基础。这是工程权衡 —— Combobox 复杂度高，Radix 团队优先做基础 primitives；cmdk 提供完整 Listbox a11y。
  9. aria-activedescendant 而非真 focus：Combobox 的关键 a11y 设计 —— 用户在 input 输入时焦点应停在 input、↓ ↑ 应"虚拟高亮"选项。aria-activedescendant 指向当前高亮项；真 DOM focus 仍在 input。
  10. 本章实战：完整 Combobox（异步加载 + multi-select + 与 react-hook-form 集成）。读完能写出 a11y 完整的 Combobox 而不是从 0 手写。

02 底层逻辑

简单控件 · Checkbox

Radix Checkbox 把原生 <input type="checkbox"> 包装为可深度样式化的组件。关键能力：

  • indeterminate state："半选"状态（用于"全选"按钮的"部分子项选中"显示）
  • aria-checked：true / false / "mixed"（indeterminate 时）
  • data-state：checked / unchecked / indeterminate（用于 CSS）
  • 受控/非受控：checked + onCheckedChange / defaultChecked
tsx 
import { Checkbox } from 'radix-ui';

// 基础
<Checkbox.Root className="...">
  <Checkbox.Indicator>
    <CheckIcon />
  </Checkbox.Indicator>
</Checkbox.Root>

// indeterminate 状态（"全选"按钮）
const [allChecked, setAllChecked] = useState<boolean | 'indeterminate'>('indeterminate');

<Checkbox.Root checked={allChecked} onCheckedChange={(checked) => setAllChecked(checked)}>
  <Checkbox.Indicator>
    {allChecked === 'indeterminate' ? <MinusIcon /> : <CheckIcon />}
  </Checkbox.Indicator>
</Checkbox.Root>

简单控件 · RadioGroup

互斥单选。a11y 关键：键盘 ← → 在选项间导航（不用 Tab，因为 Tab 跳出整个 group）。

tsx 
import { RadioGroup } from 'radix-ui';

<RadioGroup.Root defaultValue="email" name="contact-method" className="flex flex-col gap-2">
  <label className="flex items-center gap-2">
    <RadioGroup.Item value="email" id="r-email" className="...">
      <RadioGroup.Indicator className="..." />
    </RadioGroup.Item>
    <span>Email</span>
  </label>
  <label className="flex items-center gap-2">
    <RadioGroup.Item value="phone" id="r-phone">
      <RadioGroup.Indicator />
    </RadioGroup.Item>
    <span>Phone</span>
  </label>
</RadioGroup.Root>

{/*
  Radix 自动处理：
  - 键盘 ← → 在 radio 间导航
  - 选中项 tabindex="0"，未选 tabindex="-1"（roving tabindex）
  - aria-checked
  - 互斥（选一个其他自动取消）
*/}

简单控件 · Switch

视觉是拉杆，语义是 on/off。与 Checkbox 的区别：

  • role="switch"（不是 "checkbox"）
  • 屏幕阅读器读 "on/off" 不是 "checked/unchecked"
  • 不支持 indeterminate state
  • 更适合"立即生效的开关"（不需要 submit）

什么时候用 Switch / Checkbox：

  • 表单中"同意条款" → Checkbox（要 submit 才生效）
  • 设置页"启用通知" → Switch（立即生效）
  • 列表"选中要操作的项" → Checkbox

简单控件 · Slider

Radix Slider 是最复杂的简单控件。处理三种交互：

  • 鼠标拖拽：mousedown / mousemove / mouseup
  • 触摸拖拽：touchstart / touchmove / touchend
  • 键盘：← → ↑ ↓（加 Shift 大步）+ Home / End（最小/最大）+ PageUp / PageDown

a11y：

  • aria-valuemin / aria-valuemax / aria-valuenow
  • 屏幕阅读器读出"slider, 50, min 0, max 100"
  • 用户键盘调整值
tsx 
import { Slider } from 'radix-ui';

// 单值
<Slider.Root defaultValue={[50]} max={100} step={1}>
  <Slider.Track>
    <Slider.Range />
  </Slider.Track>
  <Slider.Thumb />
</Slider.Root>

// 范围（两个 thumb）
<Slider.Root defaultValue={[20, 80]} max={100} step={1}>
  <Slider.Track>
    <Slider.Range />  {/* 自动覆盖两 thumb 之间 */}
  </Slider.Track>
  <Slider.Thumb />  {/* thumb 1 */}
  <Slider.Thumb />  {/* thumb 2 */}
</Slider.Root>

复杂控件 · Select

Radix Select 是"自定义视觉的下拉选择"。对比原生 <select>：

能力原生 <select>Radix Select
样式定制有限（浏览器决定）完全自由
搜索过滤无（要用 Combobox）无（同）
选项内自定义内容有限完全自由（图片 / icon / 多行）
键盘导航原生模拟原生
a11y原生模拟原生（用 listbox role）
移动端体验原生（iOS / Android 各自 UI）统一

什么时候用 Select / Combobox：

  • 选项 < 20 + 不需要搜索 → Radix Select（简单）
  • 选项 20+ 或需要搜索 → Combobox（cmdk）
  • 需要原生移动端 UI → 原生 <select>（特别移动端 web）

复杂控件 · Combobox（最难的 widget）

Combobox = 输入框 + 自动完成 + 选项列表。例子：

  • Notion @ 提及（输入用户名筛选）
  • Linear 命令面板（⌘K + 搜索）
  • Stripe 国家选择（输入"China"筛选）
  • GitHub 仓库 / 用户搜索

WAI-ARIA APG 对 Combobox 的规范（详见 P2.1）：

  • input role="combobox"
  • listbox role="listbox"
  • 选项 role="option"
  • input aria-expanded（listbox 是否展开）
  • input aria-controls（指向 listbox id）
  • input aria-activedescendant（指向当前高亮选项 id）
  • 键盘 ↓↑ Enter ESC Tab 全套
  • 实时屏幕阅读器读出选项数 + 当前高亮

shadcn 为什么用 cmdk · Radix 没有 Combobox

2026 年 Radix 仍没有 Combobox primitive。原因：

  • Combobox 复杂度极高（a11y 12+ 条规则、异步加载、multi-select 等多种变体）
  • Radix 团队优先做基础 primitives
  • 市场已有 cmdk 等专门库

cmdk 由 Pkmn / Linear 团队的 emil_kowalski 等开源。原本是 Linear 内部"⌘K 命令面板"的实现，后来开源。

cmdk 的核心特点：

  • 提供 Listbox a11y 完整实现（aria-activedescendant 等）
  • 支持过滤（默认基于 Levenshtein distance + Fuzzy）
  • 支持 group / separator
  • 支持 async items
  • API 类似 Compound Components（与 Radix 风格一致）

shadcn 把 cmdk 包装为 Command 组件 + 在它基础上做 Combobox / 命令面板：

bash 
npx shadcn add command
npx shadcn add combobox

shadcn Combobox 的内部结构（简化）：

tsx 
// shadcn 的 Combobox = Popover（浮层）+ Command（搜索 listbox）
import * as React from 'react';
import { Check, ChevronsUpDown } from 'lucide-react';
import { Button } from '@/components/ui/button';
import {
  Command,
  CommandEmpty,
  CommandGroup,
  CommandInput,
  CommandItem,
  CommandList,
} from '@/components/ui/command';
import { Popover, PopoverContent, PopoverTrigger } from '@/components/ui/popover';
import { cn } from '@/lib/utils';

const frameworks = [
  { value: 'next.js', label: 'Next.js' },
  { value: 'sveltekit', label: 'SvelteKit' },
  { value: 'astro', label: 'Astro' },
  { value: 'remix', label: 'React Router (Remix)' },
];

export function ComboboxFramework() {
  const [open, setOpen] = React.useState(false);
  const [value, setValue] = React.useState('');

  return (
    <Popover open={open} onOpenChange={setOpen}>
      <PopoverTrigger asChild>
        <Button variant="outline" role="combobox" aria-expanded={open} className="w-[200px] justify-between">
          {value
            ? frameworks.find((f) => f.value === value)?.label
            : 'Select framework...'}
          <ChevronsUpDown className="ml-2 h-4 w-4 shrink-0 opacity-50" />
        </Button>
      </PopoverTrigger>
      <PopoverContent className="w-[200px] p-0">
        <Command>
          <CommandInput placeholder="Search framework..." />
          <CommandList>
            <CommandEmpty>No framework found.</CommandEmpty>
            <CommandGroup>
              {frameworks.map((framework) => (
                <CommandItem
                  key={framework.value}
                  value={framework.value}
                  onSelect={(currentValue) => {
                    setValue(currentValue === value ? '' : currentValue);
                    setOpen(false);
                  }}
                >
                  <Check className={cn('mr-2 h-4 w-4', value === framework.value ? 'opacity-100' : 'opacity-0')} />
                  {framework.label}
                </CommandItem>
              ))}
            </CommandGroup>
          </CommandList>
        </Command>
      </PopoverContent>
    </Popover>
  );
}

关键点：

  • Popover 提供浮层 + 智能定位
  • Command 提供 Listbox a11y + 过滤
  • shadcn 用 Tailwind class 添加视觉
  • 整个组件 a11y 完整（基于 cmdk）

aria-activedescendant 而非真 focus · 设计动机

Combobox 的关键 a11y 设计（详见 P2.1）。再深入讲：

问题：用户在 input 输入"j"，listbox 显示 "JavaScript / Java / jQuery"。

  • 用户按 ↓，期望"高亮"第一项
  • 但真 focus 不能跳到选项（input 失焦后用户继续输入字符就到选项里去了）
  • 需要"虚拟 focus"概念

解决：aria-activedescendant：

  • 真 DOM focus 始终在 input
  • input aria-activedescendant 指向当前"虚拟高亮"选项 id
  • 屏幕阅读器读 aria-activedescendant 指向的元素 → 用户感知"高亮在 JavaScript"
  • 用户继续输入 → 字符进 input（不进选项）
html 
<input
  type="text"
  role="combobox"
  aria-expanded="true"
  aria-controls="listbox-1"
  aria-activedescendant="opt-2"
  />

<ul role="listbox" id="listbox-1">
  <li role="option" id="opt-1">JavaScript</li>
  <li role="option" id="opt-2" aria-selected="true">Java</li>  <!-- 高亮 -->
  <li role="option" id="opt-3">jQuery</li>
</ul>

真 DOM focus 在 input。屏幕阅读器读 input aria-activedescendant 指向的 opt-2 → "Java, 2 of 3, list with 3 items"。用户体验上"焦点"在 Java，但物理 focus 仍在 input。

对比 Menu / Toolbar 用 roving tabindex（真 focus 跳，详见 P2.1）。两种模式由 W3C APG 规定，不是用户选。

Multi-select Combobox

选中多个值。常见模式：input 上方显示"标签"（可清除）。

挑战：

  • 选中后 listbox 是否关闭？（通常不关，让用户选下一个）
  • 键盘删除标签？（Backspace 在空 input 时删最后一个）
  • aria-selected 状态
  • 选中项视觉与未选中区分

shadcn 没有官方 Multi-select 组件，但 registry 里有社区实现 + cmdk 自带 multi-select 模式。

异步加载 Combobox

选项数量大或需要 server 搜索（如 GitHub 用户）：

  • 用户输入 → debounce 300ms → 发请求
  • 请求过程显示 loading（"Searching..."）
  • 结果回来更新 listbox
  • 用户继续输入 → 取消上次请求 + 新请求

关键 a11y：屏幕阅读器读出 "Loading..." / 加载完后 "X options available"。

tsx 
function AsyncCombobox() {
  const [search, setSearch] = useState('');
  const [debounced] = useDebounce(search, 300);
  const { data, isLoading } = useQuery({
    queryKey: ['users', debounced],
    queryFn: () => searchUsers(debounced),
    enabled: debounced.length > 0,
  });

  return (
    <Command shouldFilter={false}>  {/* 关闭 cmdk 自带过滤，server 端过滤 */}
      <CommandInput placeholder="Search users..." onValueChange={setSearch} />
      <CommandList>
        {isLoading && <CommandEmpty>Searching...</CommandEmpty>}
        {!isLoading && data?.length === 0 && <CommandEmpty>No users found</CommandEmpty>}
        {data?.map((user) => (
          <CommandItem key={user.id} onSelect={() => selectUser(user)}>
            {user.name}
          </CommandItem>
        ))}
      </CommandList>
    </Command>
  );
}

03 应用场景

场景 1 · "全选" Checkbox 联动子项

tsx 
function TodoList({ todos }) {
  const [selected, setSelected] = useState<string[]>([]);

  const allSelected = selected.length === todos.length;
  const noneSelected = selected.length === 0;
  const someSelected = !allSelected && !noneSelected;

  return (
    <>
      <label>
        <Checkbox.Root
          checked={allSelected ? true : someSelected ? 'indeterminate' : false}
          onCheckedChange={(checked) => {
            setSelected(checked ? todos.map(t => t.id) : []);
          }}
        >
          <Checkbox.Indicator>
            {allSelected ? <CheckIcon /> : someSelected ? <MinusIcon /> : null}
          </Checkbox.Indicator>
        </Checkbox.Root>
        Select all
      </label>
      {todos.map((todo) => (
        <label key={todo.id}>
          <Checkbox.Root
            checked={selected.includes(todo.id)}
            onCheckedChange={(checked) => {
              setSelected(checked
                ? [...selected, todo.id]
                : selected.filter(id => id !== todo.id));
            }}
          >
            <Checkbox.Indicator><CheckIcon /></Checkbox.Indicator>
          </Checkbox.Root>
          {todo.text}
        </label>
      ))}
    </>
  );
}

05 常见陷阱

  • 用 Radix Combobox —— 不存在。Combobox 用 cmdk（详见本章）
  • Combobox 用 真 focus 跳选项 —— 错。用 aria-activedescendant 虚拟焦点（详见 P2.1）
  • Multi-select Combobox 选中后关闭 listbox —— 通常不关（让用户继续选）。选完后用户主动关
  • 异步 Combobox 不 debounce —— 每次输入字符都发请求，server 压力大。debounce 300ms
  • 异步 Combobox 不取消上次请求 —— 慢请求晚到覆盖快请求新结果。用 AbortController / TanStack Query 自动取消
  • cmdk shouldFilter 没关，但用 server 端过滤 —— 双重过滤导致结果错。<Command shouldFilter={false}>
  • Slider value 是 number 不是 number[] —— Radix Slider 始终用数组（即使单值）。value={[50]} 不是 value={50}
  • Checkbox indeterminate state 写错 —— TypeScript 类型是 boolean | 'indeterminate'，三种状态。不是 boolean
  • Switch 用在表单 submit 场景 —— 应该 Checkbox（"立即生效"是 Switch 的语义）
  • Select 选项过多（> 50） —— 用户找不到。改用 Combobox 加搜索
  • RadioGroup 用 Tab 切换选项 —— 错。Tab 跳出 group，← → 在选项间切。这是 W3C APG 规定
  • Combobox label 没绑 input —— 屏幕阅读器读不出"什么 combobox"。用 <label htmlFor="..."> 或包 <label>
  • Combobox aria-expanded 没设 —— 用 shadcn Combobox 自动设；自己写易漏
  • Combobox 选项数 0 时不提示 —— 用 CommandEmpty / 显式"No results"
